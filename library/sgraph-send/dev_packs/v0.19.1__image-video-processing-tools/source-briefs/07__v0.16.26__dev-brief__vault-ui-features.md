# Vault UI: Auto-Commit, Visualisation, Editing, and Sync

**version** v0.16.26
**date** 18 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Architect, Designer
**type** Dev brief

## Context

The vault browser UI (dev.vault.sgraph.ai, eventually vault.sgraph.ai or vault.spki.ai) needs to become the primary interface for working with vaults in the browser. Not just viewing. Editing, committing, syncing, branching, and conflict resolution. This brief defines the features to add.

## Auto-Commit Mode

Every action the user takes in the vault UI should create a commit. The user should never have to think about committing. They edit a file, it is committed. They add a file, it is committed. They delete a file, it is committed.

Commits are local (to the clone branch). They do not push automatically unless the user opts into auto-sync. This means:

- The user never loses work (every change is a commit)
- The commit history shows every action, not just the ones the user remembered to save
- The work can be continued in a different environment (clone the vault elsewhere, the commits are there)
- No data loss even if the browser crashes (the last commit is the recovery point)

### Auto-Sync (Optional)

A mode where push and pull happen automatically in the background. The user's clone branch syncs with the server continuously. Changes from other users appear in near-real-time.

This is opt-in. Default is manual push/pull. Auto-sync is for users who want the "Google Docs" experience where everything is always up to date. Conflicts pause auto-sync and prompt the user to resolve.

Future consideration: WebSocket integration for lower-latency sync. Lambda functions do not support WebSockets natively, but this could work through API Gateway WebSocket endpoints or a lightweight persistent connection service. Not for the initial implementation.

## Visualisation

Users (especially power users and early adopters) need to see what is happening:

- **Branch status:** Which branch am I on? Is it ahead of main? Behind? Diverged?
- **Commit history:** What changes have been made, when, by whom (which branch key signed them)?
- **Server status:** Is the remote ahead of my local? Are there unsynced changes?
- **Sync indicator:** A simple visual (green = in sync, yellow = local changes not pushed, red = conflict or server diverged)

This does not need to be Git-level complexity in the UI. A status bar showing "3 commits ahead, 1 behind" and a sync button is enough for most users. The detailed commit graph is for the power user view.

## File Viewing and Editing

The vault UI should support viewing and editing files directly:

| File Type | View | Edit |
|---|---|---|
| Text / Markdown | Rendered + raw | In-browser editor |
| Images | Thumbnail + full view | Future (crop, redact) |
| PDF | Rendered viewer | Not editable |
| JSON | Formatted view | In-browser editor |
| Code | Syntax highlighted | In-browser editor |
| Audio/Video | Player | Not editable (yet) |

Editing happens in the browser. The edited file is encrypted and committed locally. The vault UI already has code for this from the admin interface experiments. Bring it back as Web Components.

## Web Component Architecture

Every feature is a Web Component. Every component uses sg-layout for positioning:

- `<sg-vault-status>` - branch status, sync indicator, commit count
- `<sg-vault-editor>` - file editor (text, markdown, JSON, code)
- `<sg-vault-viewer>` - file viewer (images, PDFs, audio, video)
- `<sg-vault-history>` - commit history list
- `<sg-vault-branches>` - branch list and switch
- `<sg-vault-conflicts>` - conflict resolution UI

Components are loaded and unloaded dynamically. sg-layout manages the arrangement. Layouts can nest (layouts of layouts), which means a complex vault workspace can have multiple panels: file tree on the left, editor in the centre, history on the right, status bar at the bottom.

By design, elements from one layout cannot move to another layout. This provides isolation: a component in one panel cannot interfere with a component in another panel. This is both a UX decision (panels are stable) and a security decision (components are sandboxed within their layout).

## Conflict Handling

When auto-sync detects a conflict (same file edited by two users on different branches):

1. Auto-sync pauses
2. Conflict indicator turns red
3. User sees both versions side by side (the "two versions exist, pick one" pattern from the version-control-for-everyone brief)
4. User resolves (pick theirs, pick mine, or manually merge)
5. Resolution is committed to the clone branch
6. Auto-sync resumes

This is also a stress test for the vault architecture. Multi-user editing in the browser UI will exercise every part of the branch model: auto-commit, push, pull, merge, conflict detection, conflict resolution.

## Acceptance Criteria

| # | Criterion |
|---|---|
