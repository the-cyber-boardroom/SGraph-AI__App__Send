## Send as Gallery Mode

When a user drops a set of images into the upload UI, offer a "Send as Gallery" option alongside the standard folder/download options. If the upload contains non-image files mixed with images, offer to package just the images as a gallery (making it easy for the user to separate them).

This is a new distribution mode alongside download, folder view, and share link.

## Gallery Editor

Before sending, the user gets a mini editor to annotate the gallery. This is an intermediate step between "upload images" and "share gallery."

### Features

**Comments per image:** Add a title and description to each image. These are stored as additional metadata files in the vault alongside the encrypted image blobs. The gallery viewer renders them below or beside each image.

**Multi-language comments:** If the user writes a comment in English, offer to translate it to other languages (using the LLM pipeline). The gallery viewer then shows comments in the viewer's language preference. This is a genuinely unique feature: share a photo gallery where every caption is available in multiple languages.

**Layout selection:** Choose how the gallery is displayed: grid, masonry, single-column, slideshow. The layout choice is stored as a configuration file in the vault. The gallery viewer reads it and renders accordingly.

**Preview:** Before sending, show a preview of what the gallery will look like to the recipient. WYSIWYG for galleries.

**Basic image editing (future):** Redaction, resize, crop, text overlay. Already explored in previous sessions. Not needed for the initial gallery editor, but the editor UI is the place where these features land when they are ready.

### Technical Implementation

The gallery editor is a Web Component (`<sg-gallery-editor>`). It composes with the existing upload flow:

1. User drops images
2. `<sg-gallery-editor>` opens
3. User adds titles, descriptions, selects layout
4. User clicks "Send as Gallery"
5. Images + metadata files are encrypted and uploaded as a vault
6. Gallery view link is returned

The metadata is stored as JSON files in the vault:

```
_gallery.json       Gallery config (layout, title, description)
_comments/
  image-001.json    { "title": "...", "description": "...", "translations": { "pt": "..." } }
  image-002.json    ...
```

## Security: Iframe Sandboxing

User-provided text (comments, titles, descriptions) introduces XSS risk. If a comment contains JavaScript, it must not execute in the gallery viewer.

Defense in depth:

1. **Input sanitisation:** Strip script tags, event handlers, and dangerous attributes on input
2. **Output encoding:** HTML-encode all user content on render
3. **Iframe isolation:** Render the gallery inside a sandboxed iframe. Even if sanitisation fails, the iframe sandbox prevents script execution from accessing the parent page

The iframe approach is the strongest defense. The gallery viewer should render in an iframe with `sandbox="allow-same-origin"` (no `allow-scripts`). This means even if malicious content gets through sanitisation, it cannot execute.

## Gallery Download as HTML

When a user downloads a gallery, generate a self-contained HTML page that links to (or embeds) the images. The HTML page includes the layout, comments, and navigation. The user gets a portable gallery that works offline, outside of SG/Send.
