# Rich Preview Gallery: Thumbnails for Every File Type, Generated on the Sender's Side

**version** v0.16.26
**date** 18 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Architect, Designer
**type** Dev brief

---

## What We Stumbled Into

The gallery view was built for images, and it works beautifully. But looking at it triggered a bigger idea: why only images? When a user sends a folder with a mix of PDFs, markdown files, images, and other documents, the receiver currently has to click on each file to see what is inside. There is no preview. No sense of what the folder contains at a glance.

Now imagine the gallery view applied to everything. PDFs show their first page as a thumbnail. Markdown renders as a formatted preview. Images show thumbnails (already working). Code files show syntax-highlighted snippets. The receiver opens the folder and immediately sees what is in it, without clicking anything.

This is not just a gallery. It is the next generation of the folder view. A rich, visual, browsable preview of everything in the folder.

## The Core Idea: Sender-Side Processing

All the processing happens on the sender's side, in the browser, before upload. The receiver does not process anything. They receive a rich, pre-built preview experience.

When the sender chooses the rich preview mode (the "gallery view" or whatever we name it), the client:

1. Takes each file in the folder
2. Generates a thumbnail or preview for it
3. Extracts metadata (file type, dimensions, page count, duration, EXIF data from images)
4. Packages all of this into an additional metadata folder alongside the original files
5. Encrypts everything (originals + thumbnails + metadata)
6. Uploads

The receiver opens the folder and the gallery view renders using the pre-generated thumbnails and metadata. No processing on the receiver's side. No waiting. The previews are instant because they are already there.

### Why Sender-Side Matters

- **Performance for the receiver:** Zero processing time. Thumbnails load instantly.
- **Security:** The receiver's browser does not need to render arbitrary PDFs or execute markdown. It displays pre-generated thumbnails, which are just images.
- **Consistency:** The sender controls what the preview looks like. Different browsers, different PDF renderers, different markdown parsers do not produce different results.
- **Time budget:** The sender has already clicked "send." They are willing to wait 5, 10, even 20 extra seconds for the upload to complete. The processing happens during that wait. With parallel file upload (individual files uploading concurrently), the thumbnail generation happens alongside the upload with minimal performance impact.

## Thumbnail Generation by File Type

| File Type | Thumbnail / Preview | How |
|---|---|---|
| Images (jpg, png, webp) | Resized thumbnail (200px wide) | Canvas API in browser |
| PDF | First page rendered as image | pdf.js renders page 1 to canvas |
| Markdown | Rendered HTML screenshot or formatted snippet | Render markdown, capture as image or store rendered HTML |
| Code files | Syntax-highlighted snippet (first 20-30 lines) | Highlight.js or Prism, render to image or store HTML |
| Audio (mp3, wav, etc.) | Waveform thumbnail + duration | Web Audio API for waveform, metadata for duration |
| Video (mp4, webm) | Frame capture (first frame or mid-point) | Video element seeks to frame, canvas captures |
| Text files | First few lines as formatted text | Simple text extraction |
| Spreadsheets | First few rows rendered as a table preview | SheetJS parses, render preview |
| Unknown/binary | File type icon + size + name | Fallback |

All of this runs in the browser. No server processing. No unencrypted data leaves the client.

## The Metadata Folder

The sender's client creates an additional folder structure alongside the original files:

```
_preview/
  _manifest.json          Gallery config, layout, file order
  thumbnails/
    file-001.thumb.jpg    Thumbnail for first file
    file-002.thumb.jpg    Thumbnail for second file
    ...
  metadata/
    file-001.meta.json    { type, size, dimensions, pages, duration, exif, ... }
    file-002.meta.json    ...
```

This folder is encrypted and uploaded alongside the original files. It is part of the same vault/transfer. The gallery view reads `_preview/_manifest.json` and renders accordingly.

If the `_preview/` folder does not exist (the sender did not choose rich preview mode, or it is an older transfer), the folder view falls back to the current behaviour: file list with names and sizes, click to open.

## Metadata Extraction Beyond Thumbnails

Once we are extracting metadata, we can extract more:

- **Image EXIF data:** GPS coordinates, camera model, date taken. Most users do not know how to extract this. We surface it automatically.
- **PDF properties:** Page count, author, creation date, whether it is searchable or scanned.
- **Audio properties:** Duration, bitrate, sample rate, channels.
- **Video properties:** Duration, resolution, codec, frame rate.
- **Semantic graphs (future):** Extract meaning, relationships, entities from text documents. Store as structured metadata. This is the path toward the vault as a knowledge graph, not just a file system.

Each file becomes not just a file but a file with context. The metadata travels with it. The receiver sees not just the content but the content's properties.

## Naming

The current name "gallery" implies images only. This is broader. Options to explore:

- **Rich view** (descriptive, neutral)
- **Preview mode** (what it does)
- **Smart folder** (implies intelligence in the presentation)
- **Showcase** (implies curated presentation)

Or keep "gallery" and let it mean "visual preview of any content type." The Ambassador and Sherpa should weigh in.

## Future: Thumbnail-Only Mode

Eventually, a mode where only the thumbnails and metadata are sent first. The receiver browses the previews and selectively downloads the full files they want. This reduces bandwidth for large folders where the receiver only needs a few files. Not for the initial implementation, but the architecture (separate `_preview/` folder) supports it naturally.

## Connection to Parallel Upload

The upcoming feature for individual file upload (files upload in parallel rather than as a single zip) makes sender-side processing even better. While file 1 is uploading, file 2's thumbnail is being generated. The processing and uploading happen concurrently. The total time increase from thumbnail generation approaches zero.

| # | Criterion |
|---|---|
| 1 | Sender-side thumbnail generation for images (resize to 200px) |
| 2 | Sender-side thumbnail generation for PDFs (first page via pdf.js) |
| 3 | Sender-side thumbnail generation for markdown (rendered preview) |
| 4 | Metadata extraction: file type, size, dimensions, page count, duration |
| 5 | `_preview/` folder structure with manifest, thumbnails, and metadata |
| 6 | Gallery view renders thumbnails from `_preview/` when available |
| 7 | Graceful fallback: if `_preview/` is absent, show standard folder view |
| 8 | All processing happens in the browser (zero server-side processing of unencrypted data) |
| 9 | Processing happens during upload wait time (user has already clicked send) |
| 10 | EXIF metadata extracted and displayed for images |
