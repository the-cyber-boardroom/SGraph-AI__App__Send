## User Personas

### Primary Personas

| Persona | Description | Key Need |
|---------|-------------|----------|
| **Privacy-Conscious Professional** | Lawyer, consultant, financial advisor sharing sensitive documents | Confidence that files are truly private |
| **Security Team Member** | InfoSec professional, pentester, CISO | Technical verification of zero-knowledge claims |
| **Journalist / Activist** | Source protection, whistleblower support | Anonymity, no metadata leakage |
| **Early Adopter** | Tech enthusiast, developer, community member | Vision, self-hosting, API access |

### Emerging Persona

| Persona | Origin | Key Need |
|---------|--------|----------|
| **Community Member** | LinkedIn traffic, GitHub stars | Project vision, contribution pathway, social proof |

This persona emerged from LinkedIn engagement after milestone posts. Interested in the project's open-source potential and privacy-first positioning.

## User Trust Assessment

### Trust Signals (Positive)

| Signal | Version Introduced | Trust Impact |
|--------|-------------------|-------------|
| Hash-fragment URLs | v0.2.24 | High — key never sent to server |
| GA removal (zero cookies) | v0.2.24 | High — no tracking whatsoever |
| i18n (8 languages) | v0.2.24 | Medium — accessibility, inclusivity |
| Transparency panel | v0.2.15 | High — shows what server stores |
| No file names on server | v0.1.0 | High — metadata minimisation |

### Trust Risks (Negative)

| Risk | Impact | Recommendation |
|------|--------|----------------|
| No explanatory content on landing page | High | Users don't understand zero-knowledge |
| localStorage caches key (visible in DevTools) | Medium | Add clear explanation or auto-clear |
| No privacy policy page | Medium | Users expect one, even if minimal |
| No "how it works" visual guide | Medium | 3-step diagram for non-technical users |
| Admin API key visible in network tab | High | Switch to httpOnly cookie |

## Product-Market Fit Indicators

### Positive Signals

- LinkedIn engagement on milestone posts (community interest)
- "Why not WeTransfer?" resonates as positioning
- Zero-cookie posture differentiates from all competitors
- Self-hosting option appeals to enterprise and security teams

### Gaps

- No user interviews conducted yet
- No analytics on usage patterns (GA removed, custom analytics new)
- No NPS or satisfaction measurement
- No competitive user testing
- No pricing/willingness-to-pay research

## Recommendations

### P1 — High Priority

| # | Recommendation | Rationale |
|---|---------------|-----------|
| 1 | Add explanatory content before upload form | Users need to understand zero-knowledge before trusting |
| 2 | Add localStorage transparency | Show users what's cached locally and why |
| 3 | Create "How It Works" visual | 3-step encrypt → share → decrypt diagram |

### P2 — Medium Priority

| # | Recommendation | Rationale |
|---|---------------|-----------|
| 4 | Write privacy policy page | User expectation, compliance |
| 5 | Add file type/size guidance | Users don't know what's supported |
| 6 | Create onboarding tooltip flow | First-time users need hand-holding |
| 7 | Eliminate LaunchList dependency | Direct registration instead |

### P3 — Lower Priority

| # | Recommendation | Rationale |
|---|---------------|-----------|
| 8 | User interviews (5-10) | Validate assumptions with real users |
| 9 | Competitive usability study | Test vs WeTransfer, Tresorit |
| 10 | NPS measurement | Baseline satisfaction score |
| 11 | Usage analytics dashboard | Understand user journey drop-offs |
| 12 | Community contribution guide | Channel early adopter energy |
