# SG/Send Browse — Markdown Authoring Guide

**Version:** v0.20.13 | **Audience:** Claude sessions, human authors creating content for SG/Send encrypted zips

---

## What this is

When a zip archive is shared via SG/Send, the Browse view decrypts and renders its contents in a two-pane layout: a file tree on the left, a tabbed preview on the right. Markdown files are rendered with a custom parser. This guide covers how to write markdown that works well in that environment.

This is **not** standard GitHub Flavored Markdown. It is a safe subset rendered by a custom parser. Features behave slightly differently — read this before authoring.

---

## 1. Links

### Internal file links (open as tabs)

Link to any file in the zip using a relative path from the current file's location:

```markdown
[See the report](report.pdf)
[Go to overview](../README.md)
[Read the spec](docs/spec.md)
[Open the diagram](images/architecture.png)
```

Clicking an internal link opens the target file as a new tab in the preview pane. The resolver:

- Tries an exact path match first
- Falls back to an ends-with match (handles zips with a root wrapper folder)
- Decodes URL-encoded characters automatically (`%20` → space, etc.)
- Tries common extensions if the path has no extension (`.md`, `.pdf`, `.txt`, `.html`, etc.)

### External links (open in browser)

Standard URLs open in a new browser tab:

```markdown
[SG/Send website](https://send.sgraph.ai)
[Email us](mailto:hello@example.com)
```

### Anchor links

Hash anchors (`#section-name`) are **not supported** as cross-file navigation. They are skipped by the link interceptor. Within-page heading anchors have no effect in the current renderer.

### Do NOT link directly to folders

The browse view can navigate to a folder link (it expands the tree and opens the first file), but this is fragile — which file opens depends on sort order and zip structure, and it gives the reader no context about what they are entering.

**Instead: create a `README.md` in every folder that should be navigable, and link to that file.**

```markdown
<!-- Avoid this -->
[See the reports](reports/)

<!-- Do this instead -->
[See the reports](reports/README.md)
```

The `README.md` is the folder's landing page. It describes the folder's contents, links to individual files, and provides context.

## 2. Images

### Basic image embedding

Images are loaded directly from the zip — no external server is contacted:

```markdown
![description](images/photo.jpg)
![diagram](../shared/architecture.png)
```

The image `src` is replaced with a blob URL extracted from the zip. External `http://` / `https://` image URLs are preserved but may be blocked by the browser's Content Security Policy in production.

### Resizing images (BRW-018)

Use the Discourse pipe convention inside the alt text brackets to set image dimensions. The URL stays clean.

| Syntax | Effect |
|--------|--------|
| `![caption](img.png)` | Default — `max-width: 100%` (fills container) |
| `![caption\|400](img.png)` | Fixed width: 400 px |
| `![caption\|50%](img.png)` | 50% of the container width |
| `![caption\|800x600](img.png)` | 800 px wide, 600 px tall (exact crop) |

Examples:

```markdown
![All 14 slides at a glance|100%](images/mosaic.jpg)

![Architecture overview|600](images/architecture.png)

![Logo|120x120](images/logo.png)

![Full-width banner|1200x300](images/banner.jpg)
```

**Degradation in other tools:** If someone opens the same `.md` file in VS Code, GitHub, or another renderer, the image still renders — it just shows the alt text as `caption|400` instead of `caption`. The image is not broken.

**Do not use HTML comment directives.** The parser has no HTML passthrough. This renders as visible escaped text:

```markdown
<!-- img:width=100% -->        ← DO NOT USE — renders as literal text
![image](img.png)
```

## 3. Navigation patterns for multi-file zips

When a zip contains many files, readers need orientation. The browse view does not have built-in breadcrumbs or back/forward buttons. Build navigation into the markdown itself.

### Pattern A — Top-of-page nav bar (for linear sequences)

Put a single line at the very top of each document with prev / index / next links:

```markdown
[← START HERE](../START%20HERE.md) · [Index](../INDEX.md) · [Articles](articles/README.md)
```

Rendered, this appears as a line of clickable links before the document heading. Simple and works well for articles, reports, and step-by-step guides.

### Pattern B — Breadcrumb trail (for deep folder hierarchies)

Put a breadcrumb at the top of each document showing where it sits in the structure. Link each ancestor:

```markdown
[🏠 Vault Index](../../INDEX.md) › [The Cyber Boardroom](../README.md) › [Accountant Pack](README.md) › **Accountant Debrief**
```

The `›` character (U+203A) renders as a clean separator. Bold the current page name to show position. This matches the navigation visible in deep-hierarchy vaults (see pic2 in the authoring brief).

### Pattern C — Hub INDEX.md (for large, complex zips)

Create a root `INDEX.md` that acts as the home page for the entire zip. Structure it as:

1. **Title + one-line description**
2. **Start here** — a single link to the entry-point document
3. **Folder map** — a table describing each top-level folder
4. **Quick links** — direct links to the most important files

```markdown
# 🏠 Vault Index

*Short description of what this zip contains.*

---

## 👉 Start here

[START HERE.md](START%20HERE.md) — Entry point for the reader

---

## 📁 Folder structure

| Folder | Contents |
|--------|----------|
| [business/](business/README.md) | Company documents |
| [personal/](personal/README.md) | Personal files |
| [_internal/](_internal/README.md) | Working files, drafts |

---

## Quick links

- [Main report](business/reports/annual-report.md)
- [Key diagram](business/diagrams/overview.png)
```

Note: folder links in the table go to `folder/README.md`, not `folder/` — following the rule from Section 1.

### Pattern D — README.md per folder

Every folder that a reader might navigate into should have a `README.md` describing:

- What this folder contains
- Which file to read first
- Links to the key files
- A breadcrumb back to the parent

```markdown
[🏠 Index](../INDEX.md) › **Reports**

# Reports

Monthly and annual financial reports.

## Files

- [Annual Report 2024](annual-2024.md)
- [Q4 Summary](q4-2024.md)
- [Q3 Summary](q3-2024.md)
```

## 4. File naming conventions

The browse view auto-opens the **alphabetically first file** in the zip (BRW-003: sorted, not zip iteration order). Take advantage of this:

- Name your entry point `0-START-HERE.md`, `00-INDEX.md`, or `README.md` if you want it to auto-open
- Prefix numbered sections: `01-introduction.md`, `02-overview.md`, `03-details.md`
- Use hyphens not spaces in filenames (spaces work but require `%20` in links)

## 5. Supported markdown syntax

| Feature | Supported | Notes |
|---------|-----------|-------|
| Headings `#` — `######` | ✓ | |
| Bold `**text**` / `__text__` | ✓ | |
| Italic `*text*` / `_text_` | ✓ | |
| Bold+italic `***text***` | ✓ | |
| Strikethrough `~~text~~` | ✓ | |
| Inline code `` `code` `` | ✓ | |
| Fenced code blocks ` ``` ` | ✓ | language label rendered as `data-lang` attribute |
| Tables `\| col \| col \|` | ✓ | Column alignment (`:---:`, `---:`) supported |
| Blockquotes `> text` | ✓ | |
| Unordered lists `- item` | ✓ | |
| Ordered lists `1. item` | ✓ | |
| Horizontal rule `---` | ✓ | |
| Images `![alt](url)` | ✓ | With pipe resize syntax |
| Links `[text](url)` | ✓ | Internal + external |
| HTML tags | ✗ | Stripped / escaped — no passthrough |
| HTML comments `<!-- -->` | ✗ | Rendered as visible text |
| Anchor headings `#id` | ✗ | Not supported |
| Nested lists | ✗ | Not supported |
| Task lists `- [ ]` | ✗ | Not supported |
| Footnotes | ✗ | Not supported |
