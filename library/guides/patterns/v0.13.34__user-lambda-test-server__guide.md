# Guide: Testing Against the SG/Send User Lambda

**version** v0.13.34
**date** 14 Mar 2026
**audience** CLI team (`sg-send-cli`), any project that needs to test against the User Lambda
**source** `sgraph_ai_app_send/lambda__user/testing/Send__User_Lambda__Test_Server.py`

---

## 1. What This Is

The User Lambda test server lets you run the **entire SG/Send User Lambda** — with all routes, middleware, storage, and auth — as either an in-process test client or a real HTTP server on a random port. Both modes use **in-memory storage** (no S3, no AWS credentials, no external dependencies). The server starts in ~100ms.

This is the same stack that runs in production on AWS Lambda, but configured for testing. Every endpoint behaves identically to the deployed Lambda — the only difference is the storage backend (memory instead of S3).

---

## 2. Two Modes

### Mode A: In-Process TestClient (No Real HTTP)

Uses Starlette's `TestClient`, which calls the FastAPI app directly without TCP. Fastest option. Good for unit tests that live inside the same Python process.

```python
from sgraph_ai_app_send.lambda__user.testing.Send__User_Lambda__Test_Server import setup__send_user_lambda__test_client

test_objs = setup__send_user_lambda__test_client()
client    = test_objs.fast_api__client

response  = client.get('/info/health')
assert response.status_code == 200
```

**Characteristics:**
- No TCP, no port, no threading — calls go directly to the ASGI app
- Singleton — first call creates the app, subsequent calls reuse it
- Fastest option (~1ms per request)
- Use `test_objs.fast_api__client` for all requests

### Mode B: Real HTTP Server on Random Port

Starts a uvicorn server on a random port in a background thread. The CLI can make real HTTP requests with `requests`, `httpx`, or any HTTP client. This is what the CLI team needs for integration tests where `sg-send-cli` makes actual HTTP calls.

```python
from sgraph_ai_app_send.lambda__user.testing.Send__User_Lambda__Test_Server import setup__send_user_lambda__test_server

with setup__send_user_lambda__test_server() as test_objs:
    base_url  = test_objs.server_url       # e.g. http://127.0.0.1:54321
    token     = test_objs.access_token
    write_key = test_objs.write_key

    import requests
    response = requests.get(f'{base_url}/info/health')
    assert response.status_code == 200
```

**Characteristics:**
- Real TCP connections on `127.0.0.1:{random_port}`
- Context manager — server starts on `__enter__`, stops on `__exit__`
- Each `with` block creates a fresh server with fresh storage (no state leakage)
- Use standard `requests` or any HTTP library
- Server runs in a background thread (non-blocking)

## 3. What You Get Back

Both modes return a `Send__User_Lambda__Test_Objs` object with these fields:

| Field | Type | Mode A | Mode B | Description |
|-------|------|--------|--------|-------------|
| `fast_api` | `Fast_API__SGraph__App__Send__User` | Yes | Yes | The FastAPI application instance |
| `fast_api__app` | `FastAPI` | Yes | Yes | The underlying FastAPI app object |
| `fast_api__client` | `TestClient` | Yes | No | In-process test client (Mode A only) |
| `fast_api_server` | `Fast_API_Server` | No | Yes | The uvicorn server wrapper (Mode B only) |
| `server_url` | `str` | `''` | `http://127.0.0.1:{port}` | Base URL for HTTP requests (Mode B only) |
| `access_token` | `str` | Yes | Yes | Pre-generated access token (random GUID) |
| `write_key` | `str` | Yes | Yes | Pre-generated write key for vault operations |

## 4. Authentication

The test server configures auth automatically. Two headers are needed for authenticated endpoints:

| Header | Value | Required For |
|--------|-------|-------------|
| `x-sgraph-access-token` | `test_objs.access_token` | Write, delete, batch |
| `x-sgraph-vault-write-key` | `test_objs.write_key` | Write, delete, batch |

Read and list endpoints require **no authentication** (zero-knowledge model).

### Mode A: Headers Are Pre-Set

The `TestClient` has `x-sgraph-access-token` pre-configured in its default headers. You only need to add `x-sgraph-vault-write-key` for vault operations:

```python
test_objs = setup__send_user_lambda__test_client()
client    = test_objs.fast_api__client

# Read — no extra headers needed
client.get('/api/vault/read/my-vault/my-file')

# Write — add write key
client.put('/api/vault/write/my-vault/my-file',
           content = b'encrypted-data',
           headers = {'x-sgraph-vault-write-key': test_objs.write_key,
                      'content-type': 'application/octet-stream'})
```

### Mode B: Set Headers Yourself

```python
with setup__send_user_lambda__test_server() as t:
    auth = {'x-sgraph-access-token'    : t.access_token,
            'x-sgraph-vault-write-key' : t.write_key   }

    requests.put(f'{t.server_url}/api/vault/write/v1/f1',
                 data=b'blob', headers={**auth, 'content-type': 'application/octet-stream'})
```

## 5. Available Endpoints

All User Lambda endpoints are available. The vault-related ones for the branch model:

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| PUT | `/api/vault/write/{vault_id}/{file_id}` | Yes | Write a single blob |
| GET | `/api/vault/read/{vault_id}/{file_id}` | No | Read a single blob (binary) |
| GET | `/api/vault/read-base64/{vault_id}/{file_id}` | No | Read as base64 JSON |
| DELETE | `/api/vault/delete/{vault_id}/{file_id}` | Yes | Delete a blob |
| POST | `/api/vault/batch/{vault_id}` | Yes | Batch operations |
| GET | `/api/vault/list/{vault_id}?prefix=` | No | List files by prefix |
| GET | `/info/health` | No | Health check |

### 6.2 Write and Read a Single File

```python
with setup__send_user_lambda__test_server() as t:
    auth = {'x-sgraph-access-token'   : t.access_token,
            'x-sgraph-vault-write-key': t.write_key,
            'content-type'            : 'application/octet-stream'}

    # Write
    r = requests.put(f'{t.server_url}/api/vault/write/my-vault/my-file',
                     data=b'encrypted-payload', headers=auth)
    assert r.json()['status'] == 'completed'

    # Read (no auth needed)
    r = requests.get(f'{t.server_url}/api/vault/read/my-vault/my-file')
    assert r.content == b'encrypted-payload'
```

### 6.3 Batch Write (Push Simulation)

This is the pattern `sg-send-cli push` should use — write all objects in one request, CAS the ref last:

```python
import base64

with setup__send_user_lambda__test_server() as t:
    auth = {'x-sgraph-access-token'   : t.access_token,
            'x-sgraph-vault-write-key': t.write_key   }
    vault = 'my-vault-id'

    ops = [
        # Write immutable objects (blobs, trees, commits)
        {'op': 'write', 'file_id': 'bare/data/obj-blob1',
         'data': base64.b64encode(b'encrypted-file-content').decode()},
        {'op': 'write', 'file_id': 'bare/data/obj-tree1',
         'data': base64.b64encode(b'encrypted-tree-json').decode()},
        {'op': 'write', 'file_id': 'bare/data/obj-commit1',
         'data': base64.b64encode(b'encrypted-commit-json').decode()},

        # CAS the ref (MUST be last)
        {'op': 'write-if-match', 'file_id': 'bare/refs/ref-current',
         'match': None,                                                # None = file must not exist (first push)
         'data': base64.b64encode(b'obj-commit1').decode()},
    ]

    r = requests.post(f'{t.server_url}/api/vault/batch/{vault}',
                      json={'operations': ops}, headers=auth)

    assert r.status_code == 200
    results = r.json()['results']
    assert all(result['status'] == 'ok' for result in results)
```

### 6.4 Second Push (CAS with Existing Ref)

```python
    # ... continuing from 6.3 ...

    # Second push — match must equal current ref value
    ops2 = [
        {'op': 'write', 'file_id': 'bare/data/obj-commit2',
         'data': base64.b64encode(b'encrypted-commit-2').decode()},
        {'op': 'write-if-match', 'file_id': 'bare/refs/ref-current',
         'match': base64.b64encode(b'obj-commit1').decode(),    # must match what's currently stored
         'data':  base64.b64encode(b'obj-commit2').decode()},
    ]

    r2 = requests.post(f'{t.server_url}/api/vault/batch/{vault}',
                       json={'operations': ops2}, headers=auth)
    assert all(result['status'] == 'ok' for result in r2.json()['results'])
```

### 6.5 Handling a CAS Conflict (Stale Ref)

```python
    # ... continuing from 6.4 ...

    # Push with stale match value — server rejects
    ops3 = [
        {'op': 'write-if-match', 'file_id': 'bare/refs/ref-current',
         'match': base64.b64encode(b'obj-commit1').decode(),    # STALE — server has obj-commit2
         'data':  base64.b64encode(b'obj-commit3').decode()},
    ]

    r3 = requests.post(f'{t.server_url}/api/vault/batch/{vault}',
                       json={'operations': ops3}, headers=auth)

    conflict = r3.json()['results'][0]
    assert conflict['status']  == 'conflict'
    assert conflict['current'] == base64.b64encode(b'obj-commit2').decode()  # server tells you what it has

    # CLI should now: fetch new ref → merge → retry push
```

## 7. How It Works Internally

### Architecture

```
sg-send-cli test process
    │
    │  requests.post('http://127.0.0.1:54321/api/vault/batch/...')
    │
    ▼
┌──────────────────────────────────────────────────────┐
│  uvicorn (background thread, random port)            │
│                                                      │
│  ┌────────────────────────────────────────────────┐  │
│  │  Fast_API__SGraph__App__Send__User             │  │
│  │  (the same class used in production Lambda)    │  │
│  │                                                │  │
│  │  Routes:                                       │  │
│  │    Routes__Transfers                           │  │
│  │    Routes__Presigned                           │  │
│  │    Routes__Early_Access                        │  │
│  │    Routes__Vault__Pointer  ◄── batch, list,    │  │
│  │                                write-if-match  │  │
│  │  Services:                                     │  │
│  │    Transfer__Service                           │  │
│  │    Service__Vault__Pointer                     │  │
│  │    Service__Presigned_Urls                     │  │
│  │                                                │  │
│  │  Storage:                                      │  │
│  │    Storage_FS__Memory  ◄── in-memory, no S3    │  │
│  └────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────┘
```

### Startup Sequence

1. `setup__send_user_lambda__test_server()` returns a context manager
2. `__enter__()` does:
   - Generates random `access_token` and `write_key`
   - Sets `SGRAPH_SEND__ACCESS_TOKEN` env var (the app reads this for auth)
   - Creates `Fast_API__SGraph__App__Send__User().setup()` — this:
     - Auto-detects no AWS credentials → uses `Storage_FS__Memory`
     - Wires up all routes (transfers, presigned, vault, early access)
     - Sets up MCP server mount
   - Creates `Fast_API_Server(app=app)` with a random port
   - Calls `server.start()` — starts uvicorn in a background thread
   - Waits for port to be open (< 100ms typically)
   - Returns `Send__User_Lambda__Test_Objs` with all connection details
3. Tests run against `http://127.0.0.1:{port}/...`
4. `__exit__()` calls `server.stop()` — signals uvicorn shutdown, joins thread, waits for port to close

### Storage Model

All data lives in a Python dictionary inside `Storage_FS__Memory`. The structure:

```
transfers/vault/{vault_id}/manifest.json           ← write_key_hash
transfers/vault/{vault_id}/{file_id}/payload        ← encrypted bytes
```

When the batch endpoint writes `bare/data/obj-abc123`, it stores:

```
transfers/vault/{vault_id}/bare/data/obj-abc123/payload
```

The `bare/` prefix is part of the `file_id` — the server treats it as an opaque string. The server has no knowledge of the branch model, trees, commits, or refs. It stores bytes, serves bytes, and does compare-and-swap.

When the context manager exits, the `Storage_FS__Memory` instance is garbage collected and all data disappears. Each `with` block starts with a clean slate.

### Why No Admin Lambda?

The test server runs only the **User Lambda** (the public-facing one). The Admin Lambda (tokens, analytics, vault lifecycle) is not started. This means:

- Token validation falls back to env-var check (the test helper sets this up)
- No admin API calls (token create/revoke/lookup are stubbed)
- The vault pointer service works fully — it doesn't depend on the admin lambda

If you need admin endpoints too, that would require a separate helper (or extending this one to start both lambdas).

## 9. Troubleshooting

| Symptom | Cause | Fix |
|---------|-------|-----|
| `ModuleNotFoundError: sgraph_ai_app_send` | Package not installed | `pip install -e /path/to/SGraph-AI__App__Send` or add as git dependency |
| `OSError: [Errno 98] Address already in use` | Port collision (very rare with random ports) | Retry — each call picks a new random port |
| `403 on write/batch` | Wrong access token or write key | Use `test_objs.access_token` and `test_objs.write_key` |
| `400 Missing write key` | Header `x-sgraph-vault-write-key` not set | Add the header to your request |
| `404 on read` | File not written yet, or wrote to different vault_id | Check vault_id consistency |
| Write returns `None` (403 via route) | Vault already has a manifest with a different write_key | Each vault_id gets one write_key (set on first write). Use a unique vault_id per test, or use the same write_key. |
| List returns empty | Files written via individual PUT with slashed file_ids | Use batch endpoint for `bare/` paths (see section 10) |

## 10. Important: Slashed File IDs

File IDs in the branch model contain slashes (e.g. `bare/data/obj-abc123`). These work through:

- **Batch endpoint** — file_ids are in the JSON body, slashes are fine
- **List endpoint** — prefix is a query parameter, slashes are fine
- **Individual PUT/GET** — the `{file_id}` URL parameter captures only up to the first slash

**Rule of thumb:** Use the batch endpoint for all `bare/` writes. Use individual PUT/GET only for flat file_ids (no slashes). Use the list endpoint to discover files with slashed paths.

This is the intended pattern — the CLI uses batch for push operations (which involve `bare/` paths) and individual endpoints for simple, flat file_id operations.
