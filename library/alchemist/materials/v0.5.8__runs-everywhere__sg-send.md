## Executive Summary

SG/Send is a single codebase that deploys to **7 targets in 4 patterns.** The same Python application — the same encryption, the same API, the same tests — runs as a serverless Lambda function, a Docker container, a server on bare metal, or a CLI tool on a developer's laptop.

This is not a roadmap aspiration. The architecture supports it today because the application is built on **Memory-FS** — a storage abstraction that means the application code has no idea which backend is active. Swap S3 for local disk, swap Lambda for Docker, and the application doesn't know the difference.

For investors, this means: **no customer is excluded by infrastructure preference.** Cloud, on-premise, air-gapped, multi-cloud — SG/Send runs wherever the customer needs it.

## 1. The Seven Deployment Targets

### Pattern 1: Serverless (Lambda)

| Target | Infrastructure | Use Case |
|---|---|---|
| **AWS Lambda + Lambda URL** | Serverless compute, S3 storage, no API Gateway | SaaS offering, managed instances, lowest cost |

**Current status:** Production-ready. This is how send.sgraph.ai runs today.

**Why it matters:** Scales to zero. No idle costs. New instances provision in minutes. Cost per transfer: fractions of a penny.

### Pattern 2: Container (Docker / Fargate / GCP Cloud Run)

| Target | Infrastructure | Use Case |
|---|---|---|
| **Docker** | Any Docker-compatible host | Self-hosted, on-premise, development |
| **AWS Fargate** | Managed container service, no server management | Enterprise customers wanting container isolation |
| **GCP Cloud Run** | Google's serverless container platform | Multi-cloud customers, GCP-native organisations |

**Current status:** Architecture supports it (FastAPI application, no Lambda-specific dependencies in core). Deployment templates in progress.

**Why it matters:** Enterprise customers who can't use Lambda (policy, multi-cloud, GCP-native) get the same product. Container deployments also support Kubernetes for customers with existing orchestration.

### Pattern 3: Server (EC2 / AMI / Bare Metal)

| Target | Infrastructure | Use Case |
|---|---|---|
| **EC2 / AMI** | Dedicated server, full control | Compliance-sensitive customers, air-gapped deployments |

**Current status:** Architecture supports it. FastAPI runs as a standard ASGI server (Uvicorn) without any modification.

**Why it matters:** Some customers (government, defence, regulated industries) require dedicated infrastructure with no shared compute. Air-gapped deployments are possible because the application has no external dependencies — encryption is browser-native (Web Crypto API), and the server is self-contained.

### Pattern 4: Developer Tools (CLI / PyPI / SDK)

| Target | Infrastructure | Use Case |
|---|---|---|
| **CLI (`sg_send`)** | Terminal command | Developers, automation, CI/CD pipelines |
| **PyPI package** | Python library | Integration into Python applications |

**Current status:** Implementation plan complete. CLI is month 2 priority.

**Why it matters:** The CLI and SDK turn SG/Send from a web application into a **developer platform.** Any application can embed encrypted transfer with a 5-line integration. This is the distribution multiplier.

## 3. How It Works: Memory-FS

### The Architectural Foundation

Memory-FS (`Storage_FS`) is a storage abstraction that provides a consistent interface regardless of the backend:

| Backend | Where It Runs | Use Case |
|---|---|---|
| **In-memory** | Tests, development | Full stack starts in ~100ms, no external dependencies |
| **Local disk** | Docker, server, CLI | Self-hosted deployments |
| **S3** | Lambda, Fargate, Cloud Run | Cloud deployments |

The application code uses `Storage_FS` for all storage operations. It never calls S3 directly, never touches the filesystem directly, never knows which backend is active. Switching backends is a configuration change, not a code change.

### What This Means Practically

```
Same code → Memory-FS → In-memory    → Tests pass in ~100ms
Same code → Memory-FS → Local disk   → Docker runs on any machine
Same code → Memory-FS → S3           → Lambda scales to millions
```

This is the reason SG/Send can claim 7 deployment targets from a single codebase. It's not marketing — it's architecture.

## 4. Air-Gapped Deployment: The Enterprise Premium

### Why Air-Gapped Matters

Some of the highest-value customers operate in environments with no internet access:

- **Government and defence** — classified document handling
- **Financial regulators** — examination materials, enforcement documents
- **Healthcare** — patient records in compliance-restricted networks
- **Legal** — litigation hold materials in secure facilities

These customers cannot use any SaaS product. They cannot use any product that requires external API calls, CDN resources, or certificate authority validation.

### Why SG/Send Can Serve Them

1. **No external dependencies.** Web Crypto API is built into the browser. No CDN. No external JavaScript. No API calls to third-party services.
2. **Self-contained server.** FastAPI application with Memory-FS on local disk. No S3, no Lambda, no internet required.
3. **Browser-native PKI.** Key generation, encryption, and decryption happen entirely in the browser. No certificate authority calls.
4. **Open source.** The customer can audit every line of code before deploying in a classified environment.

### The Commercial Opportunity

Air-gapped deployments command premium pricing because:
- The customer base is small but high-value
- There is no competition (no SaaS tool can serve this market)
- The compliance and support requirements justify enterprise contracts
- Government procurement cycles, once won, produce multi-year revenue

## 6. The Commercial Model: Three Tiers of Deployment

### Tier 1: SG/Send Cloud (SaaS)

- Hosted by SG/Send on AWS Lambda
- Multi-tenant or dedicated instances
- Lowest price, fastest provisioning
- Customer manages nothing

### Tier 2: SG/Send Managed (Container/Server)

- Deployed in customer's cloud or on-premise
- SG/Send manages updates, monitoring, support
- Customer controls infrastructure and data residency
- Premium pricing for isolation and control

### Tier 3: SG/Send Self-Hosted (Licence)

- Customer deploys and manages independently
- Annual licence + optional support contract
- Full control, full responsibility
- Highest price, longest sales cycle, highest retention
