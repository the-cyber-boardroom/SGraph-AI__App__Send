## Executive Summary

SGraph Send (SG/Send) is a zero-knowledge encrypted file and message transfer service. Files are encrypted in the sender's browser before upload. The server never sees the plaintext. The decryption key never leaves the sender's device. No app required, no account required — the recipient just needs a browser.

As of 20 February 2026, SG/Send has achieved browser-based Public Key Infrastructure (PKI): recipients can be addressed by their public key, eliminating key exchange entirely. An investor group called this capability "a killer feature" — before we had built it. We built it in 24 hours.

## 1. The Problem

Every day, confidential documents cross trust boundaries through channels where the server can read them:

- **Email** — plaintext in transit and at rest, forwarded freely, persists indefinitely
- **Slack/Teams** — workspace admins see everything, searchable history, no encryption
- **WhatsApp** — E2E encrypted between users, but cloud backups are unencrypted by default
- **WeTransfer** — no E2E encryption, server can read every file
- **Data rooms (Intralinks, Datasite)** — the provider holds the encryption keys

The documents in question are term sheets, NDAs, merger agreements, financial reports, credentials, medical records, board materials. These are not cat pictures. They are the most sensitive information in any organisation.

No mainstream tool offers the combination of: end-to-end encryption + no app required for the recipient + self-hostable + open source + verifiable. The market has a gap. SG/Send fills it.

## 2. The Solution

SG/Send encrypts everything in the browser before upload using AES-256-GCM via the Web Crypto API. The server stores only encrypted ciphertext. Even with full infrastructure access — database dump, S3 bucket access, Lambda function compromise — the data cannot be read.

**How it works:**

1. Sender opens SG/Send in a browser — no app, no account
2. Drops a file or types a message
3. Content is encrypted in the browser before upload
4. Sender receives a link and a decryption key — shares them via separate channels
5. Recipient opens the link, enters the key, decrypts in their browser

**With PKI (achieved 20 Feb 2026):**

1. Sender selects the recipient's public key
2. Content is encrypted for that specific recipient
3. Recipient opens the link — their browser decrypts automatically using their private key
4. No key to share. No second channel. One link, one click.

The server never sees the plaintext in either flow. Zero-knowledge is preserved.

**Independent validation:** A senior security professional with 25 years of experience described the UX as "foolproof — like Google search" and the cryptography as "the correct implementation." His observation: "I can't tell you how many times I've needed something like this. If you use Signal, Proton, or whatever else, the other party needs the same tech, which is not always the case."

## 3. The Technology Moat

### 3.1 The Encryption Trilemma — Solved

Enterprise encrypted communication faces a fundamental tension between three properties. Every existing solution sacrifices at least one:

| Property | What It Means | Who Sacrifices It |
|---|---|---|
| **Zero-knowledge server** | The server cannot read the data | Intralinks, Datasite, most data rooms |
| **Multi-user access** | Multiple recipients can access the same content | Standard PGP, Signal (one-to-one only) |
| **Revocable access** | Access can be withdrawn after it has been granted | Standard PKI wrapping |

SG/Send's **Secure Pod** architecture resolves this: a dedicated, hardened intermediary that decrypts content, re-encrypts it per-user with a unique per-request key, and then forgets the plaintext. The storage server remains zero-knowledge. The user never holds a key that works on stored content. Access can be revoked at any time.

> *Source: Architect review — `team/roles/architect/reviews/02/21/v0.5.0__review__secure-pod-architecture.md`*

### 3.2 Browser-Based PKI

SG/Send has built what the industry delegates to Verisign and DigiCert — in the browser, with zero external dependencies:

- RSA-OAEP 4096-bit encryption keys + ECDSA P-256 signing keys
- Non-extractable private keys (not even the JavaScript that created them can read them back)
- Hybrid encryption (per-message AES-256-GCM session key wrapped in RSA-OAEP)
- Digital signatures for authentication and non-repudiation
- Zero external dependencies — Web Crypto API only, >97% browser coverage

### 3.3 Tamper-Evident Audit Trails

Every trust relationship and access event is recorded in a hash-chained commit log — the same principle that underpins blockchain, without the overhead. Each entry includes the SHA-256 hash of the previous entry, making retroactive tampering detectable and provable.

### 3.4 Open Source (Apache 2.0)

The entire codebase is open source. Anyone can verify the encryption implementation. For a zero-knowledge tool, a security reviewer told us: "that's not optional." Open source provides verifiable security claims, frictionless enterprise procurement, community trust, and a partner ecosystem.

## 8. Product Roadmap

### Step 1: Key Discovery and Public Registry (Next — ~5 Days)

Users generate key pairs in the browser and publish their public key with a friendly short code (e.g., `A7X9F2`). Senders look up recipients by short code, QR code, or registry page. PKI encryption becomes as easy as entering a phone number.

**Status:** Implementation plan complete. 5-phase build.

### Step 2: LLM Integration — "Claude, Send This Securely"

Compose a document in Claude, say "send this to [founder] via SG/Send," and it is encrypted, uploaded, and delivered. Implementations via CLI tool and MCP Server (Model Context Protocol).

**Status:** Implementation plan complete.

### Step 3: Enterprise Data Rooms with Revocable Access

Zero-knowledge encrypted data rooms where multiple users access shared documents, access can be revoked after granting, and every access event is recorded in a tamper-evident audit trail. Built on the Secure Pod architecture.

**Status:** Architecture designed and reviewed. Implementation gated on customer demand signal.

## 10. Risk Analysis

| Risk | Mitigation |
|---|---|
| **Single founder** | Co-founder search active. Agentic team provides breadth. The model works today. |
| **Browser crypto limitations** | Web Crypto API is W3C standard, supported by >97% of browsers. Non-extractable keys are enforced by the browser, not by our code. |
| **Large file handling** | Current limitation: ~500MB. Architecture for chunked, streaming encryption is designed. Implementation gated on customer demand. |
| **Regulatory changes** | Open source means no single jurisdiction can block the technology. Self-hostable means customers control their deployment. |
| **Competitive response** | WeTransfer adding encryption would validate the market. Our moat is architectural: zero-knowledge + PKI + self-hostable + open source is not a feature toggle. |
| **Revenue model uncertainty** | "We will discover the pricing model." Hypothesis-driven approach. Serverless means near-zero burn while discovering product-market fit. |
