### James is a managing partner at a VC fund

He needs to share term sheets with three portfolio founders. Each founder should see only their own terms.

**Today:** James emails each one separately and hopes nobody forwards. He has no audit trail, no access control, no encryption.

**With SG/Send PKI (achieved 20 February 2026):**

## Slide 8: What PKI Changes

### From two-step to one-step. From friction to flow.

| | Before PKI | With PKI |
|---|---|---|
| **Send** | Link + key via separate channels | One link |
| **Receive** | Paste key, decrypt | Click link, automatic decrypt |
| **Key management** | Sender manages keys manually | Browser manages keys (non-extractable) |
| **Authentication** | None — anyone with the key can decrypt | Cryptographic — only the intended recipient can decrypt |
| **Non-repudiation** | None | Digital signatures prove authorship |

**This is what the investor group called "a killer feature" on 19 February — one day before we built it.**

## Slide 9: The Encryption Trilemma

### Three properties the industry says you can't have simultaneously

| | Zero-knowledge | Multi-user | Revocable |
|---|---|---|---|
| **Intralinks** | No | Yes | Yes |
| **Signal** | Yes | No | No |
| **PGP** | Yes | No | No |
| **SG/Send** | **Yes** | **Yes** | **Yes** |

**The Secure Pod architecture** resolves this: a hardened intermediary that re-encrypts per-user with per-request keys. The storage server remains zero-knowledge. Access can be revoked at any time.

This is the foundation for **encrypted data rooms** — James's term sheets, Sarah's merger documents, board materials, due diligence packages — all with revocable, auditable, zero-knowledge access.
