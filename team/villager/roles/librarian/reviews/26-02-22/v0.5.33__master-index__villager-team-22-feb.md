### Receiver Experience / Short Codes (Brief #3)

| Area | Impact |
|------|--------|
| **API** | **NEW endpoints needed**: short code generation, code validation, code self-destruct |
| **Security** | Self-destruct mechanism needs race condition protection — AppSec review required |
| **Auth model** | Progressive trust tiers (Lobby → Member → Verified → Admin) — major auth expansion |
| **Infrastructure** | Short code storage with TTL, attempt counting, automatic deletion |
| **Testing** | Self-destruct timing, concurrent access, environmental factor validation |
| **Deployment** | New API surface = new Lambda routes, new tests, new monitoring |

**This is the largest new API surface since the project started.** The Villager team must plan for: short code storage, TTL management, self-destruct logic, rate limiting, and environmental factor collection.

## 5. QA Project: Villager Significance

The QA project (Briefs #4–6) is **the Villager team's primary quality gate**. When operational:

- Every SG/Send release triggers QA project test runs
- Test failures block releases
- Generated documentation becomes the user-facing docs
- Screenshot diffs show exactly what changed between releases

**Villager coordination needed:**
1. Provide valid access tokens for test environments
2. Ensure staging environment is available for QA project tests
3. Agree on test coverage thresholds for release gates
4. Integrate QA project CI with SG/Send release pipeline
