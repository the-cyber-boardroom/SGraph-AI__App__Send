## 1. IFD Overlay Infrastructure

### What changed

The v0.3.1 overlay infrastructure was created and the local dev server was updated to support it.

### How the overlay works

v0.3.1 HTML pages load all v0.3.0 base code via `../_common/` paths, then load v0.3.1 patch files as additional `<script>` tags. In the deployed `latest/` folder (and the local dev server's `.local-server-user/`), the structure is flat:

```
latest/
  _common/js/components/send-download/send-browse.js       ← v0.3.0 base
  _common/js/components/send-download/send-browse-v031.js  ← v0.3.1 patch (NEW)
  en-gb/index.html                                          ← v0.3.1 (overwrites v0.3.0)
  en-gb/browse/index.html                                   ← v0.3.1 (overwrites v0.3.0)
```

The v0.3.1 browse page loads `send-browse.js` (base) then `send-browse-v031.js` (patch). The patch uses JavaScript prototype overrides — it replaces specific methods on the `SendBrowse` class after the base class has been defined.

### Files

| File | Type | Purpose |
|---|---|---|
| `v0.3.1/en-gb/index.html` | **NEW** | Upload page — loads v0.3.0 base, sets `uiVersion: 'v0.3.1'` |
| `v0.3.1/en-gb/browse/index.html` | **NEW** | Browse page — loads v0.3.0 base + `send-browse-v031.js` patch |
| `v0.3.1/_common/js/components/send-download/send-browse-v031.js` | **NEW** | Prototype overrides for `SendBrowse` (2 bug fixes) |
| `scripts/user__run-locally.sh` | **MODIFIED** | Now discovers all v0.3.x versions and applies IFD overlays |

### Local dev server changes (`scripts/user__run-locally.sh`)

**Before:** Hardcoded `UI_VERSION="v0.3.0"`, copied only v0.3.0.

**After:** Discovers all `v0.3.x` directories, sorts by patch number, copies v0.3.0 first, then overlays v0.3.1+ on top. Same algorithm as the CI pipeline (`deploy_static_site.py`).

```bash
# Before (hardcoded)
UI_VERSION="v0.3.0"
UI_DIR="$STATIC_DIR/v0/v0.3/$UI_VERSION"

# After (auto-discovery)
VERSIONS=$(ls -d "$STATIC_DIR/v0/v0.3"/v0.3.* | xargs -n1 basename | sort -t. -k3 -n)
LATEST_VERSION=$(echo "$VERSIONS" | tail -1)
# Then copies each version in order into .local-server-user/
```

### CI pipeline

No changes needed — `deploy-ui-user.yml` already discovers `v0.3.x` directories and applies them in overlay order. v0.3.1 is automatically picked up on next deploy to `dev`.

### Path resolution

All v0.3.1 HTML pages use `../_common/` paths (one level up from `en-gb/`). This resolves correctly in:
- **Deployed `latest/`** — `_common/` from v0.3.0 is at `latest/_common/`
- **Local dev server** — `.local-server-user/_common/` (merged from v0.3.0 + v0.3.1)

Note: `../../v0.3.0/_common/` paths would NOT work in production (there is no `v0.3.0/` directory in `latest/`).

## 2. Bug Fix: BRW-001 — Folder Tree Shows Full Zip Path Instead of Filename

### Bug ID: BRW-001

### Description

In the browse/folder view, files inside subfolders displayed the full zip path as their name in the sidebar tree. For example, a file at `4) SG:Send - Integration/1) How_SG_Send/file.pdf` showed the entire path string instead of just `file.pdf`.

### Root cause

JSZip's `entry.name` property contains the **full path** within the zip, not just the filename. When `send-download.js` builds the `zipTree` array (line 277-279), it stores `entry.name` as the `name` property:

```javascript
// v0.3.0 send-download.js line 277-279
const name = entry.name;  // ← full path, e.g. "folder/sub/file.pdf"
this._zipTree.push({ path, name, dir: entry.dir, entry });
```

The `_buildFolderTree()` method correctly distributes files into folder nodes by splitting on `/`, but `_renderFolderNode()` then displays `file.name` which is still the full path.

### Fix

**File:** `v0.3.1/_common/js/components/send-download/send-browse-v031.js`
**Method:** Prototype override of `SendBrowse.prototype._renderFolderNode`

The override extracts the basename before displaying:

```javascript
const basename = file.name.includes('/') ? file.name.split('/').pop() : file.name;
// Then uses basename instead of file.name in the tree HTML
```

### Expected test impact

- **GOOD FAILURE:** Any screenshot that captures the folder tree sidebar will change — file names are now shorter (basename only instead of full path). Update the screenshot baseline.
- **SHOULD STILL PASS:** File opening (clicking a tree item), file content rendering, tab creation — all use `file.path` not `file.name`, so they're unaffected.

## 3. Bug Fix: BRW-002 — PDF Files Missing Present/Fullscreen Mode in Browse View

### Bug ID: BRW-002

### Description

The gallery lightbox has a "Present" button for PDF files that opens the PDF in a new window optimized for presentation (`#toolbar=1&navpanes=0&view=Fit`). The browse/folder view does not have this button, despite being the primary way to view PDFs in multi-file transfers.

### What was added

**File:** `v0.3.1/_common/js/components/send-download/send-browse-v031.js`
**Method:** Prototype override of `SendBrowse.prototype._renderFileContent`

The override wraps the original method, calls it to render the base content, then appends a "Present" button to the action bar for PDF files:

```javascript
SendBrowse.prototype._renderFileContent = (function(original) {
    return function(container, bytes, fileName, type) {
        original.call(this, container, bytes, fileName, type);
        if (type === 'pdf') {
            // Append Present button to action bar
            // Click handler: window.open(pdfUrl + '#toolbar=1&navpanes=0&view=Fit')
            // Fallback: iframe.requestFullscreen() if popup blocked
        }
    };
})(SendBrowse.prototype._renderFileContent);
```

### Behaviour

- **Present button** appears in the file action bar (next to filename, size, and Save) only for PDF files
- **Click:** Opens the PDF blob URL in a new browser window/tab with `#toolbar=1&navpanes=0&view=Fit` (same as gallery lightbox Present mode)
- **Popup blocked:** Falls back to `requestFullscreen()` on the iframe element
- **Non-PDF files:** No change — the button is only added for PDF type

### Expected test impact

- **GOOD FAILURE:** Screenshots of PDF files in browse view will show an additional "Present" button in the action bar. Update the screenshot baseline.
- **SHOULD STILL PASS:** All other file types (image, markdown, text, code, audio, video) are unaffected. Gallery lightbox Present mode is unaffected (separate component).
