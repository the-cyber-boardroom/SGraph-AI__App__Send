## P0 Regression Fix (post-QA)

QA caught a release-blocking regression in the original PLR-006 implementation.

**Bug:** The first deep-link implementation used `/` as the token/path separator:
```
#transferId/encryptionKey/path/to/file
```
`send-download.js` stripped everything after the first `/`, which silently dropped
the encryption key. Every non-friendly-token vault downloaded the ciphertext but
could not decrypt it (`sd.hashKey = undefined`). The browse tree never rendered.

**Fix:** Switch separator to `|`:
```
#transferId/encryptionKey|path/to/file   ← regular token, key preserved
#help-aunt-2780|path/to/file             ← friendly token
```
`|` never appears in transfer IDs, encryption keys, or file paths, so it is always
unambiguous. Three files updated: `send-download.js`, `_plrGetHashPath()`, `_plrSetHashPath()`.

**QA release gate:** the 2 P0 bug tests flip to failing → 13 PLR tests go green → release clear.

## Security Notes

- **Deep-link path never leaks keys** — the hash path (`#token|path`) appends to the full token (including `/key` for raw tokens); both are client-side only and never sent to the server. The server only sees the URL path, not the hash.
- **Blob → data: conversion** (for print window) uses `fetch()` on same-origin blob URLs; no data leaves the browser.
- **CMD+P suppression** is scoped to pages with an active PLR panel; all other pages use the browser default.

---

## Known Limitations

- Edit mode changes are **not persisted** — the textarea is for preview only; there is no save-to-vault path yet.
- Print window images require the blob URLs to still be live (i.e., the page must not have been navigated away from). This is the same constraint as `sg-print.js` markdown printing.
