# Browse Team Brief: Shared Module Extraction

**Version:** v0.20.6 | **Date:** 30 March 2026 | **From:** Vault team (Architect)
**To:** Browse team (Developer)
**Priority:** High — blocks vault-browse unification (Phase 3)
**Architecture plan:** `team/roles/architect/reviews/03/30/v0.20.6__arch-plan__vault-browse-unification.md`

---

## Context

The vault website needs to reuse the Browse module (`send-browse.js` + v0.3.1 overlay) to display vault files. Right now Browse is hardwired to JSZip — every file access goes through `entry.async('arraybuffer')` and the tree is built from a flat zipTree array.

We need Browse to work with **any** data source. The vault team will provide a `VaultDataSource` adapter. Your job is to make Browse accept it.

---

## What We're Asking For

**Make `send-browse.js` data-source-agnostic by introducing a `dataSource` property.**

The data source interface has 3 required methods:

```javascript
dataSource.getTree()
// Returns: { name: '', children: { folderName: TreeNode, ... }, files: [ { path, name, size }, ... ] }
// Same shape that _buildFolderTree() currently returns

dataSource.getFileBytes(path)
// Returns: Promise<ArrayBuffer>
// Given a file path (e.g. 'docs/readme.md'), returns the file content

dataSource.getFileList()
// Returns: [ { path, name, dir, size }, ... ]
// Flat list of all files — used for link resolution, auto-open, gallery filtering
```

Plus optional properties for write support:

```javascript
dataSource.writable        // boolean — if true, Browse shows Edit/Save/Delete buttons
dataSource.saveFile(folderPath, fileName, bytes)   // Promise
dataSource.renameFile(folderPath, oldName, newName) // Promise
dataSource.deleteFile(folderPath, fileName)         // Promise
```

## Specific Changes Needed

### 1. New property: `dataSource`

```javascript
// In constructor:
this.dataSource = null;    // BrowseDataSource instance

// Existing properties stay for backwards compat during transition:
this.zipTree      = null;
this.zipInstance   = null;
this.zipOrigBytes  = null;
```

### 2. Create `ZipDataSource` class

Extract current zip logic into an adapter. This can be a separate file or inline class:

```javascript
class ZipDataSource {
    constructor(zipInstance, zipTree, zipOrigBytes, zipOrigName) {
        this._zip      = zipInstance;
        this._tree     = zipTree;
        this._origBytes = zipOrigBytes;
        this._origName  = zipOrigName;
        this.writable   = false;
    }

    getTree() {
        // Move _buildFolderTree() logic here
        // (including v0.3.1 gallery filtering)
    }

    async getFileBytes(path) {
        const entry = this._tree.find(e => e.path === path && !e.dir);
        if (!entry) throw new Error(`File not found: ${path}`);
        return entry.entry.async('arraybuffer');
    }

    getFileList() {
        return this._tree.filter(e => !e.dir);
    }

    // Zip-specific extras (not part of interface):
    getOrigBytes()  { return this._origBytes; }
    getOrigName()   { return this._origName; }
    getZipInstance() { return this._zip; }
}
```

### 3. Replace direct zip access in `_openFileTab()`

Current (v0.3.0 + v0.3.1):
```javascript
const entry = this.zipTree.find(e => e.path === path && !e.dir);
const bytes = await entry.entry.async('arraybuffer');
```

New:
```javascript
const bytes = await this.dataSource.getFileBytes(path);
```

### 4. Replace `_buildFolderTree()` calls

Current: `this._buildFolderTree()` operates on `this.zipTree`.

New: `this.dataSource.getTree()` — the data source is responsible for building the tree.

### 5. Replace `this.zipTree` references in v0.3.1 overlay

The v0.3.1 overlay references `this.zipTree` in several places:
- `_findZipEntry(zipTree, resolved)` — link resolution
- `_navigateToFolder(browseInstance, zipTree, folderPath)` — folder navigation
- `_autoOpenFirstFile()` — initial file selection

These should use `this.dataSource.getFileList()` instead of `this.zipTree`.

### 6. Update `send-download.js` to create ZipDataSource

```javascript
// In send-download.js, after zip extraction:
const dataSource = new ZipDataSource(this._zipInstance, this._zipTree, this._zipOrigBytes, this._zipOrigName);
browse.dataSource = dataSource;

// Keep setting old properties too for any code that hasn't migrated:
browse.zipTree      = this._zipTree;
browse.zipInstance   = this._zipInstance;
browse.zipOrigBytes  = this._zipOrigBytes;
```

### 7. Emit events for write operations (optional, for vault)

When `dataSource.writable === true`, the file action bar should include Edit/Save/Delete buttons. When clicked, Browse emits events:

```javascript
this.dispatchEvent(new CustomEvent('browse-file-save', {
    detail: { path, bytes }, bubbles: true
}));
```

The vault shell catches these and calls `dataSource.saveFile()`. **This is optional for Phase 1** — the vault team can add these buttons externally in Phase 3.

## What Must NOT Change

- **Zero user-visible difference** — existing send.sgraph.ai/browse works identically
- **All v0.3.1 features preserved** — markdown links, CSV tables, HTML iframes, PDF present, source toggles, keyboard nav, gallery filtering, scrollable tabs
- **No new dependencies** — ZipDataSource wraps existing JSZip, no new libs
- **CSS unchanged** — same `sb-` prefix classes

## Suggested Approach

1. Create `ZipDataSource` class wrapping current logic
2. Add `dataSource` property to SendBrowse
3. In `_build()`, if `this.dataSource` exists use it; else fall back to `this.zipTree` (backwards compat)
4. Migrate `_openFileTab`, `_buildFolderTree`, `_autoOpenFirstFile` to use `dataSource`
5. Migrate v0.3.1 overlay helpers (`_findZipEntry`, `_navigateToFolder`) to use `dataSource.getFileList()`
6. Update `send-download.js` to create `ZipDataSource` and pass it in
7. Test: existing Browse view works exactly as before

The vault team will independently build `VaultDataSource` that implements the same interface. Once both are ready, vault-shell imports `send-browse.js` and passes `VaultDataSource` instead of `ZipDataSource`.
