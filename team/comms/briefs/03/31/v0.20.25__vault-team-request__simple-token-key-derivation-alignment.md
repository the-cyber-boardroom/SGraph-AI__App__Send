**Version:** v0.20.25

**From:** Vault UI team (Explorer)
**To:** CLI team (sg-send-cli / SGit-AI)
**Type:** Technical alignment request
**Priority:** Blocking -- vault UI cannot open simple-token vaults until derivation matches

---

## What We're Trying to Do

The vault UI (v0.2.0) now supports simple tokens (`word-word-NNNN`) as vault keys. When a user enters a simple token in the vault entry page, the UI needs to derive the same `read_key`, `write_key`, and `ref_file_id` that the CLI produced when creating the vault.

**Current status:** The vault_id path in the API URL is correct (the token string itself), but the `ref_file_id` (the HMAC-derived ref pointer) does not match what the CLI computed. This means the UI cannot find the vault's HEAD ref.

---

## How the UI Currently Derives Keys from a Simple Token

When the UI receives a simple token (detected by regex `/^[a-z]+-[a-z]+-\d{4}$/`), it treats the token as both the `passphrase` and the `vault_id`:

```
Input:        <simple-token>
passphrase:   <simple-token>
vault_id:     <simple-token>
```

Then it calls `SGVaultCrypto.deriveKeys(passphrase, vaultId)` which does:

### Step 1: PBKDF2 for read_key
```
read_key = PBKDF2-SHA256(
    password:   passphrase (UTF-8 bytes),
    salt:       "sg-vault-v1:<vault_id>" (UTF-8 bytes),
    iterations: 600,000,
    key_length: 256 bits
)
```

### Step 2: PBKDF2 for write_key
```
write_key = PBKDF2-SHA256(
    password:   passphrase (UTF-8 bytes),
    salt:       "sg-vault-v1:write:<vault_id>" (UTF-8 bytes),
    iterations: 600,000,
    key_length: 256 bits
) -> hex string
```

### Step 3: HMAC for ref_file_id
```
hmac_key  = HMAC-SHA256 key from read_key raw bytes
ref_hex   = HMAC-SHA256(hmac_key, "sg-vault-v1:file-id:ref:<vault_id>")[:12 hex chars]
ref_file_id = "ref-pid-muw-" + ref_hex
```

### Step 4: HMAC for branch_index_file_id
```
branch_hex = HMAC-SHA256(hmac_key, "sg-vault-v1:file-id:branch-index:<vault_id>")[:12 hex chars]
branch_index_file_id = "idx-pid-muw-" + branch_hex
```

```javascript
    branch_index_id:   "idx-pid-muw-<12 hex chars>",

    hmac_ref_domain:   "sg-vault-v1:file-id:ref:<token>",

```

The `ref_file_id` logged by the UI does NOT match the ref file stored on the server for vaults created by the CLI.

## What We Need from the CLI Team

For the same simple token input, the UI and CLI must produce identical `read_key`, `write_key`, and `ref_file_id`. We need to know:

### 1. What inputs does the CLI pass to key derivation?

When the CLI opens a vault with a simple token, what values are used for:
- `passphrase` (is it the token string? something derived from it? empty?)
- `vault_id` (is it the token string? confirmed yes from earlier discussion)

### 2. What PBKDF2 salt does the CLI use for read_key derivation?

The UI uses: `"sg-vault-v1:<vault_id>"` (where vault_id = the token string)

Does the CLI use the same salt format? Or a different salt?

### 3. What HMAC domain string does the CLI use for ref_file_id?

The UI uses: `"sg-vault-v1:file-id:ref:<vault_id>"`

The Python `Vault__Crypto` class (server-side) uses different domains:
- `"sg-vault-v1:file-id:tree:<vault_id>"` for tree_file_id
- `"sg-vault-v1:file-id:settings:<vault_id>"` for settings_file_id

Which domain does the CLI use for the HEAD ref?

## Source Code Reference

The UI's key derivation is in:
```
sgraph_ai_app_send__ui__vault/v0/v0.2/v0.2.0/_common/js/lib/sg-vault/sg-vault-crypto.js
```

Key methods:
- `parseVaultKey(fullVaultKey)` -- detects simple tokens, extracts passphrase + vault_id
- `deriveKeys(passphrase, vaultId)` -- PBKDF2 + HMAC derivation
- `_deriveFileId(hmacKey, input)` -- HMAC-SHA256 -> first 12 hex chars

The Python server-side equivalent is in:
```
sgraph_ai_app_send/utils/Vault__Crypto.py
```

Note: the Python `Vault__Crypto` uses different HMAC domain strings (`tree`/`settings` vs `ref`/`branch-index`), which may explain the mismatch if the CLI is based on the Python implementation rather than the JS.

---

## Possible Root Causes

1. **Different passphrase** -- the CLI may not use the token as the passphrase, or may pre-process it (e.g., PBKDF2 derive a key from the token first, then use that as the passphrase)

2. **Different HMAC domain strings** -- the CLI may use `"sg-vault-v1:file-id:tree:<vault_id>"` (Python style) instead of `"sg-vault-v1:file-id:ref:<vault_id>"` (JS style) for the HEAD ref

3. **Different PBKDF2 salt** -- the CLI may use a different salt format for simple tokens (e.g., `"sgraph-send-v1"` from `FriendlyCrypto` instead of `"sg-vault-v1:<vault_id>"` from `SGVaultCrypto`)

4. **Two-stage derivation** -- the CLI may first derive an intermediate key from the token using FriendlyCrypto (PBKDF2 with salt `"sgraph-send-v1"`), then use that intermediate key as the passphrase for the standard vault key derivation
