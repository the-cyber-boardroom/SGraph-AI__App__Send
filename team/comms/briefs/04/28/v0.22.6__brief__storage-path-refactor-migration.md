# Brief: Storage Path Refactor — Migration & Deployment Notes

**From:** Explorer Dev  
**To:** DevOps / Deployment  
**Date:** 2026-04-28  
**Version:** v0.22.6

---

## What Is Changing

The on-disk storage path structure for all S3 objects is changing from:

```
transfers/{transfer_id}/meta.json
transfers/{transfer_id}/payload
transfers/vault/{vault_id}/manifest.json
transfers/vault/{vault_id}/{file_id}/payload
vault-zips/{vault_id}/{content_hash}.zip
```

To:

```
sg-send__data/sg-send-api__v1.0/{deployment_id}/transfers/{ab}/{transfer_id}/meta.json
sg-send__data/sg-send-api__v1.0/{deployment_id}/transfers/{ab}/{transfer_id}/payload
sg-send__data/sg-send-api__v1.0/{deployment_id}/vault/{ab}/{vault_id}/manifest.json
sg-send__data/sg-send-api__v1.0/{deployment_id}/vault/{ab}/{vault_id}/{file_id}/payload
sg-send__data/sg-send-api__v1.0/{deployment_id}/cache/vault-zips/{vault_id}/{content_hash}.zip
```

Where `{ab}` = first 2 characters of the object ID (256 shards), and `{deployment_id}` defaults to `shared`.

---

## New Environment Variable

| Variable | Default | Example |
|---|---|---|
| `SEND__DEPLOYMENT_ID` | `shared` | `prod`, `staging`, `dev` |

Set this on each Lambda function. If unset, defaults to `shared`.

---

## Breaking Change: Existing Data

**All objects at the old paths become unreachable after deployment.** This is intentional.

No data is deleted. Old objects remain in S3 but the application will not find them. There is no migration script.

---

## Deployment Order

Three environments exist: `dev`, `main`, `prod`. Deploy in this order:

### 1. `dev` environment

- No real user data
- Deploy immediately
- Set `SEND__DEPLOYMENT_ID=dev`
- Verify with smoke tests

### 2. `main` environment

- No real user data
- Deploy after `dev` passes
- Set `SEND__DEPLOYMENT_ID=main`
- Verify with smoke tests

### 3. `prod` environment

- **Real user data exists at old paths — becomes unreachable**
- Deploy last, after `dev` and `main` are validated
- Set `SEND__DEPLOYMENT_ID=shared` (the default, matching the old implicit scope)
- Existing transfers: become unavailable (equivalent to natural expiry)
- Existing vaults: become unavailable on server, but any user who has the vault locally cloned can re-push after deployment

---

## Vault Re-Push After Deployment

Vaults follow the same model as a git remote rebase. If a user has a local clone (`.sg_vault/`), their data is intact on disk. After `prod` deployment:

```bash
sgit push <vault-key>
```

This re-uploads all local vault files to the new path structure. The vault is then accessible again with the same `vault-key`. No data is lost for any user with a local clone.

---

## Smoke Tests After Each Deployment

After deploying to each environment, verify:

1. `GET /health` → 200
2. `POST /api/transfers/create` → 201 with `transfer_id`
3. Upload payload → `POST /api/transfers/upload/{id}` → 200
4. Complete → `POST /api/transfers/complete/{id}` → 200
5. Download → `GET /api/transfers/download/{id}` → 200 with bytes
6. Vault write → `POST /api/vault/write/{vault_id}/{file_id}` → 200
7. Vault read → `GET /api/vault/read/{vault_id}/{file_id}` → 200

These cover both transfer and vault paths through the new structure.

---

## Rollback

If deployment to `prod` fails:

1. Revert to the previous Lambda deployment (previous version in AWS console)
2. Old objects at old paths are still in S3 — immediately accessible again
3. No data recovery needed

---

## IAM Note

If IAM policies scope S3 access to specific key prefixes, update them to include:

```
sg-send__data/*
```

Remove or keep the old prefix policies as needed for rollback safety.
