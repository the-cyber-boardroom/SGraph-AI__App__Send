# CLI Brief: Vault Delete Endpoint

**From:** SGraph Send (Explorer Dev team)
**To:** CLI team (sgit)
**Date:** 2026-04-28
**Implements:** `sgit vault delete`

---

## What Was Shipped

`DELETE /api/vault/destroy/{vault_id}` is live on the User Lambda.
It hard-deletes every server-side file belonging to a vault — blobs, refs, key
material, branch indexes. No recovery, no soft delete.

---

## HTTP Contract

### Request

```
DELETE /api/vault/destroy/{vault_id}
Content-Type: application/json
x-sgraph-vault-write-key: <write_key_hex>

{"vault_id": "<vault_id>"}
```

Both the URL path and the JSON body must carry the `vault_id`. They must
match — a mismatch returns 409. This is an intentional safety guard: it forces
the caller to name the vault explicitly in two independent places, so a
mis-wired request (wrong URL, stale body, copy-paste error) is caught before
any deletion happens.

### Success — `200 OK`

```json
{
  "status": "deleted",
  "vault_id": "abc123de",
  "files_deleted": 142
}
```

`files_deleted` is the count of storage objects removed (blobs + refs + keys +
manifest). It will be `0` if the vault was already deleted or never existed —
both return `200`, not an error.

### Error Responses

| Status | Condition |
|--------|-----------|
| `400` | `x-sgraph-vault-write-key` header missing, or body is not valid JSON, or `vault_id` field absent |
| `403` | `write_key` does not match the vault's stored key material |
| `409` | `vault_id` in body does not match `vault_id` in URL path |

There is no `404` — a vault that does not exist (or was already deleted)
returns `200` with `files_deleted: 0`. This makes the endpoint safe to retry
after a network error.

## Authentication

The server validates the `write_key` against a SHA-256 hash stored in the
vault's `manifest.json` at creation time. The CLI already knows how to derive
`write_key` from `vault_key` (same derivation used by `sgit push` and
`sgit commit`) — no changes to the key derivation logic are needed.

The `vault_key` token (e.g. `apple-river-1234`) encodes both the `vault_id`
and the `write_key`. The CLI can extract both without any extra login step.

**Important:** The `vault_id` sent to the server must be the **opaque
alphanumeric identifier**, not the human-readable vault key name. The server
rejects `vault_id` values containing hyphens, uppercase letters, or fewer than
8 / more than 24 characters. The CLI must pass the extracted opaque ID, not
the full `vault_key` string.

## Suggested CLI Design

```bash
# Delete vault from server only (keep local working copy)
sgit vault delete <vault-key>

# Delete from server AND wipe local .sg_vault/ directory
sgit vault delete <vault-key> --local

# Skip confirmation prompt (for scripting / CI)
sgit vault delete <vault-key> --yes
```

### Confirmation prompt (interactive mode)

Before sending the request, print:

```
About to permanently delete vault <vault_id> from the server.
This cannot be undone. Type the vault ID to confirm: _
```

Only proceed if the user types the `vault_id` exactly. The `--yes` flag skips
this prompt.

### Recommended output on success

```
Deleted vault <vault_id> from server (142 files removed).
```

If `files_deleted` is `0`:

```
Vault <vault_id> was already deleted from the server (nothing to remove).
```

### Retry on network error

The endpoint is idempotent. If the CLI receives a network error (timeout,
connection reset) after sending the request, it is safe to retry — a second
call will succeed with `files_deleted: 0` if the first call completed on the
server.

---

## What the Server Deletes

Everything under the vault's storage prefix. From the CLI's perspective this
means all objects that `sgit push` ever uploaded:

- All committed blobs and trees
- All branch refs (named + clone)
- Vault key material
- Branch indexes
- Vault manifest (the ownership record)

The local working copy (`.sg_vault/`) is **not** touched unless `--local` is
passed.

## Edge Cases

| Scenario | Server response | CLI should |
|----------|----------------|------------|
| Vault already deleted | `200`, `files_deleted: 0` | Print "already deleted" message, exit 0 |
| Vault never existed on server | `200`, `files_deleted: 0` | Same as above |
| Wrong write key | `403` | Print error, suggest checking vault key, exit 1 |
| Network timeout after success | Retry → `200`, `files_deleted: 0` | Treat as success |
| `vault_id` format rejected | `400` | Bug in CLI key parsing — should not happen in production |
