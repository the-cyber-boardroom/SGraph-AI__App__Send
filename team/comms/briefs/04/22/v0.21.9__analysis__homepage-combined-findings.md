# Analysis: Homepage Combined Findings
**Version:** v0.21.9 | **Date:** 2026-04-22
**Sources:** Designer review + Ambassador review of `https://dev.sgraph.ai/en-gb/`
**File under review:** `sgraph_ai__website/v0/v0.2/v0.2.0/en-gb/index.html`

---

## Summary

Both the Designer and Ambassador reviews converge on the same core diagnosis: the homepage has strong raw material — honest copy, a well-structured design system, real differentiators — but fails at composition. It is trying to do too much in one page without adequate visual evidence (screenshots), misprioritises its feature slots (wrong 3rd primitive, SGit buried at the bottom), and does not visualise its most differentiating claim ("for humans and agents").

All findings below are cross-referenced against the reality document (v0.21.8, audited 2026-04-21). No overpromising was found — the issues are structural and compositional, not accuracy problems.

## Finding 2: Encryption badge is visually orphaned

**Type:** CSS fix / Low effort / High impact

The `.enc-badge` sits at the top of the hero with `padding-block-start: var(--sp-24)` (96px) above it. This creates a large void that makes the badge look like a loading artifact rather than an intentional trust signal priming the headline.

**Fix:** Reduce hero top padding to `var(--sp-16)` (64px). Reduce `.enc-badge` bottom margin from `24px` to `12px`.

## Finding 3: Third primitive card is wrong

**Type:** Content change / Low effort / High impact

The three primitive cards represent the product's core identity. Currently:
1. Browser Automation — Available now ✅
2. Encrypted Virtual File System — Available now ✅
3. **Code Execution Sandbox — Coming soon ❌**

Slot 3 actively advertises something that doesn't exist. The slot should be **PKI / Cryptographic Identity** — a genuine differentiator that is either shipped or near-shipped, and that no competitor offers. The Code Execution Sandbox belongs in a "What's coming" section further down the page.

**New card content:**
- Title: `Cryptographic Identity`
- Icon: 🔑
- Badge: `Available now`
- Description: Each vault commit is signed. Agents and humans have verifiable cryptographic identities. Trust without a central authority.
- Features: Per-vault key pair, commit signing, no shared credentials, agent identity scoping

## Finding 4: No visualisation of the three interaction patterns

**Type:** New section / Medium effort / High impact

The headline claim "For humans and agents" is never visualised. The three fundamental interaction patterns of the product are:

| Pattern | Description |
|---|---|
| **Human → Human** | Person sends encrypted vault; recipient opens with token |
| **Agent → Human** | LLM/agent writes output to vault; human accesses via token |
| **Agent → Agent** | Two agents collaborate: one writes, one reads, full audit trail |

A three-panel diagram section (CSS-only, no external libraries) immediately after the hero and before the primitives section would communicate this faster than any amount of copy.

## Finding 5: Vault view modes are invisible

**Type:** Content addition / Low effort / High impact

The vault browse UI has three view modes that exist and are differentiating:
- **Folder view** — hierarchical file navigation
- **Gallery view** — image grid for photo/media vaults
- **Markdown rendering** — articles and documents render formatted in-browser

None are mentioned on the homepage. The "Encrypted Virtual File System" primitive card should list these as features. Screenshots `01-vault-browse.png`, `02-markdown-article.png`, and `03-gallery.png` directly illustrate them.

## Finding 6: SGit and Vaults need dedicated treatment

**Type:** New page + nav change / High effort / Very high impact

SGit + Vaults is the core product primitive. It currently appears as one small card in the "Open Source" section at the bottom of the page:

> **SGit CLI** — Command-line vault operations: init, commit, push, pull, share. → View on GitHub

This is insufficient for a feature that is the central differentiator. A dedicated `/en-gb/vaults/` page is needed covering:
- What a vault is
- Three access modes: Online (browser), CLI (sgit), API (programmatic)
- The commit model — full audit trail
- Key types — vault key vs share token
- View modes — folder, gallery, markdown
- Multi-agent use cases

`Vaults` should be added to the main navigation.

## Finding 7: "No account required to receive" is buried

**Type:** Content move / Low effort / High impact

One of the strongest viral loop enablers is currently in the pricing section:
> "No subscription. No account required for receiving."

Recipients arriving with a share token — the highest-intent visitor — need to know this immediately. It should be in the hero or immediately below the CTAs.

---

## Finding 8: Privacy section needs visual elevation

**Type:** CSS + HTML / Low effort / High impact

The privacy section is the best trust artifact on the page but has weak visual treatment. A `border-left: 4px solid var(--accent)` accent, larger body text, and numbered sentences would elevate it substantially without changing the copy.

Additionally, the privacy section should appear **before** the pricing section — trust must be built before the purchase ask.

---

## Finding 9: Mobile token bar is hidden

**Type:** CSS fix / Low effort / Medium impact

At `max-width: 768px`, the `.token-inline-bar` trigger is hidden (`display: none`). Mobile users — who are most likely to arrive via a shared link — cannot access the paste-token affordance. The trigger button must always be visible; only the expanded input state needs special mobile treatment.

---

## Finding 10: `direction: rtl` layout reversal

**Type:** CSS fix / Low effort / Medium impact

`.use-case-row--reverse` uses `direction: rtl` to flip column order. This is semantically incorrect (it's a language-direction property, not a layout property) and will break when RTL locales (Arabic, Hebrew) are added. The i18n infrastructure already exists; this fix is necessary before RTL expansion.

**Fix:** Use CSS `order` on child elements instead.

---

## Finding 11: Deploy strip redundancy

**Type:** Content simplification / Low effort / Low impact

The page has two elements communicating deployment targets:
1. `deploy-strip` (text badges under hero CTAs): "Runs on: Lambda · Fargate · EC2 / AMI..."
2. `aws-strip` (icon-node diagram below hero): Lambda → S3 → ECR → CloudFront → EC2

These overlap significantly. The `aws-strip` is more informative and visually richer. The `deploy-strip` can be removed or moved to the dedicated Vaults/Architecture page.
