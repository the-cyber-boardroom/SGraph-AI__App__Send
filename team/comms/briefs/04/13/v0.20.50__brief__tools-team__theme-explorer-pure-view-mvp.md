## 1. The One-Line Goal

**Build a standalone tool on `tools.sgraph.ai` where anyone — Claude included — can drop a `_page.json`, see it rendered live in pure view mode, switch themes, tweak components, and copy the result back into a vault.**

This is the MVP. It validates pure view mode end-to-end **without modifying the main Browse component**, gives authors (especially Claude) a fast feedback loop on `_page.json` design, and produces the theme picker / page builder UX that the main app will eventually inherit.

---

## 2. Why a Standalone Tool First (and Not a Mode in Browse)

The pure-view-mode brief proposes adding a view toggle to the existing Browse component. That's the right end state. But shipping it there first is risky:

- The Browse component is already shipped, in production, and is the recipient's primary experience
- Any change to it carries regression risk for live transfers
- The view-mode UX isn't designed yet — toggle placement, default behaviour, mobile breakpoints, fallbacks all need iteration
- Claude needs the feedback loop **today** to author `_page.json` files for PlaybookLM, news infographics, and the examples website

A standalone tool unblocks all of that immediately. It can iterate freely, prove the model, and only then graduate the proven UX into the main Browse component.

## 4. MVP Scope — What This Tool Is

A single page at `tools.sgraph.ai/page-builder/` with three regions:

```
+----------------------------------------------------------+
|  HEADER:  SG/Send Theme Explorer    [Load]  [Copy] [?]   |
+--------------------------+-------------------------------+
|                          |                               |
|  EDITOR (left, ~40%)     |   PREVIEW (right, ~60%)       |
|                          |                               |
|  - Theme picker dropdown |   Live render of _page.json   |
|  - Raw JSON textarea     |   in PURE VIEW MODE           |
|  - Validation errors     |   (no folder chrome)          |
|  - Component palette     |                               |
|    (click to insert)     |   [Mobile] [Tablet] [Desktop] |
|                          |   viewport switcher           |
|                          |                               |
+--------------------------+-------------------------------+
```

### What the user can do

1. **Load** a starting point: empty, the demo JSON, an existing vault folder, or paste from clipboard
2. **Edit** the `_page.json` in the textarea (with debounced live re-render)
3. **Switch theme** via the dropdown — JSON gets updated, preview re-renders
4. **Insert a component** by clicking from the palette — appends a sensible default block to the JSON
5. **Switch viewport** — preview frame resizes to phone / tablet / desktop dimensions
6. **Copy** the final JSON to clipboard, ready to paste into a vault
7. **Save to vault** (Phase 2) — push the `_page.json` into a vault folder via `sgit`

### What pure view mode means in this tool

The preview pane renders the page **exactly as a recipient would see it on a share link with `view=pure`**:

- No left folder tree
- No file metadata
- No upload/edit chrome
- Just the rendered `_page.json` content, full-width, themed
- Mobile viewport switcher proves responsive behaviour before the page is ever shared

This is the **same render path** the production Browse component will eventually call when `view=pure` is active — the tool is the test bench for that mode.

## 5. Architecture

### A new tool, not a fork

```
tools.sgraph.ai/page-builder/
├── index.html
├── page-builder.js          # tool shell (editor + viewport + theme picker)
├── page-builder.css         # tool chrome only
└── (imports from CDN/shared)
    ├── send-browse-v031--page-layout.js     # the renderer (unchanged)
    └── send-browse-v031--page-layout.css    # the renderer styles (unchanged)
```

The tool is a **thin harness** around the existing renderer. It owns:
- The editor textarea + theme dropdown + component palette
- The viewport switcher iframe wrapper
- The load / copy / save buttons
- Tool-specific chrome (header, status bar, error display)

It does **not** own:
- The renderer itself (`send-browse-v031--page-layout.js`)
- The theme schemes (they live in the renderer)
- The component definitions (they live in the renderer)
- Any rendering CSS (lives in the renderer's CSS file)

### Pure view mode = no chrome around the renderer

In the production Browse component, the renderer is wrapped by `sg-layout` panels (folder tree, tabs, file metadata). In this tool, the renderer is wrapped by **nothing** — just an iframe sized to the chosen viewport. That's the entire mechanism for "pure view mode" in the MVP.

When pure view mode eventually graduates to Browse, it does the same thing: collapses the `sg-layout` panels and renders the `_page.json` full-width.

### Iframe for the preview

Use an iframe for the preview pane. Three reasons:
1. **Style isolation** — the renderer's CSS doesn't leak into the tool's chrome and vice versa
2. **Viewport simulation** — resizing the iframe genuinely simulates a phone/tablet/desktop viewport, including media-query breakpoints
3. **Drop-in graduation** — the iframe loads the same renderer file the main app loads, so what you see is what recipients get

## 6. The Theme Explorer Half

This is the part that earns the tool its name and its biggest immediate value: **rapid theme iteration**.

### How it works

The theme picker dropdown in the editor lists the 6 named schemes. When the user selects one:
1. Parse the current JSON
2. Replace (or insert) the `theme` object with the chosen scheme's full configuration
3. Serialise back to the textarea
4. Re-render the preview

The schemes themselves come from the renderer (already shipped). The tool just exposes them.

### Phase 2 additions

Once the picker works:
- **Visual swatches** next to each dropdown option — show the accent colour and background so authors choose by eye, not by name
- **Per-element overrides** — let the user override individual theme properties (accent colour, hero style, font family) without leaving the named scheme
- **Custom theme save** — let the user name a tweaked theme and save the JSON snippet for re-use across pages

## 7. The Component Palette Half

A simple sidebar listing every supported component as a clickable chip:

```
[hero]    [section]    [text]    [title]
[image]   [gallery]    [slides]  [pdf]
[markdown] [bullet-points] [cards] [columns]
[callout] [stats]      [quote]   [author]
[navigation]
```

Click a chip → a default block of that type is appended to the current JSON's `blocks` array, with placeholder content (e.g. `text` chip inserts `{"type": "text", "content": "Lorem ipsum..."}`). Re-render shows the new block immediately.

This is **not** a drag-and-drop visual builder. It's a one-click insert that fast-tracks authoring without needing the user to memorise the schema. The textarea remains the source of truth.

The default blocks should match the patterns shown in `page-layout-demo.json`.

## 8. Phasing

### Phase 1 — Validate the model (target: 1 session)

- Static HTML page with editor textarea on left, iframe on right
- Iframe loads the `_page.json` renderer + the editor's current JSON via `postMessage`
- Theme picker dropdown reused from the existing edit-mode bar
- Load the demo JSON (`page-layout-demo.json`) on first open
- Copy-to-clipboard button
- Three-button viewport switcher (phone 375, tablet 768, desktop 1280)

**Definition of done:** Claude can open the tool, paste a `_page.json`, switch themes, see live mobile preview, and copy the final JSON to drop into a vault.

### Phase 2 — Author quality of life (target: 1–2 sessions)

- Component palette (one-click insert)
- Theme swatches (visual preview of named schemes)
- Validation errors shown inline (parse errors, missing required props)
- Load-from-vault button (paste a vault token, fetch the `_page.json` from it)
- Save-to-vault button (push the current JSON back to the source vault)

### Phase 3 — Graduation (target: 1 session)

- Extract the proven pure-view-mode behaviour (iframe wrapper, viewport switcher, no-chrome render path) into a Web Component the main Browse can import
- Add `view=pure` URL parameter handling to the main Browse — when present, render via the same path as the tool
- Honour `view_mode: "pure"` in `_page.json` as the per-folder default

Phase 3 is **not in this brief's MVP scope** but is the natural next step once the model is validated.

## 9. Out of Scope (Explicitly)

To keep the MVP focused, do **not** build:

- WhatsApp share kit (separate brief, separate component)
- Embeddable interactive components (`sg-file-upload` etc inside `_page.json`) — that's the second 04/13 brief and depends on a security review (whitelist enforcement)
- New theme schemes beyond the 6 already shipped
- A drag-and-drop visual page builder (the textarea + palette is enough for MVP)
- Any modification to the `_page.json` schema itself
- Any modification to the production Browse component
- Account/auth/persistence — the tool is stateless, the JSON is the artefact
