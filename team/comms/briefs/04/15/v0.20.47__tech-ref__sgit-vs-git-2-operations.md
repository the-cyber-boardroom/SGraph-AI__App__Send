## 1. `git init` / `sgit init` / `SGVault.create()`

### Git

```
  git init myrepo
    → create .git/
    → .git/HEAD = "ref: refs/heads/main"
    → .git/config, .git/description, .git/hooks/
    → no objects written
    → no network calls
```

Git `init` is entirely local. The first remote interaction happens on the first `git push`.

### SGit CLI (`sgit init`)

```
  sgit init
    → derive keys from passphrase:vault_id
    → create .sgit/ local working directory
    → write initial commit + root tree to server (like git init + git push in one)
    → write named ref to server
    → total: ~5 PUT requests
```

SGit `init` immediately writes to the server. There is no concept of a local-only vault that has not yet been pushed — the vault exists on the server from the first `init`.

### Web UI (`SGVault.create()`)

```
  SGVault.create(sgSend, passphrase, options)
    → generate 8-char random vault ID
    → derive keys (PBKDF2 ×2, ~600 ms)
    → create empty in-memory tree
    → _commit('Initial vault creation')
        PUT blob (settings)
        PUT tree (empty root)
        PUT commit
        PUT refs/cloneRefFileId
    → PUT refs/refFileId  (named ref — same commit)
    → return vault object, ready to use
    → total: 5 PUT requests
```

```
  Property              git init          sgit init         SGVault.create()
  ─────────────────── ──────────────────  ────────────────  ─────────────────
  Network calls       0                  ~5 PUTs            5 PUTs
  Local state created .git/ directory    .sgit/ directory   in-memory only
  Server state after  none               vault exists       vault exists
  Named ref written   no (push does this) yes               yes
  Clone ref written   no                 yes                yes
  Both refs in sync   n/a                yes (ahead = 0)    yes (ahead = 0)
```

## 2. `git clone` / `sgit clone` / `SGVault.open()`

### Git

```
  git clone https://github.com/org/repo myrepo
    → GET /info/refs?service=git-upload-pack
    → negotiation: client sends "want" list, server sends objects
    → receive-pack: all reachable objects sent as pack file
    → unpack to .git/objects/
    → checkout HEAD → working tree populated
    → network: proportional to full repo size
```

Git clone is a bulk operation. Every object in the repository's history is fetched. After clone, the client can work entirely offline.

### SGit CLI (`sgit clone`)

```
  sgit clone passphrase:vault_id ./mydir
    → derive keys
    → GET named ref → namedCommitId
    → walk commit graph: GET each commit, collect all tree_ids and blob_ids
    → GET all tree objects
    → GET all blob objects (decrypt each)
    → write decrypted files to ./mydir/
    → create .sgit/ working directory
    → network: proportional to full vault size
```

Same bulk model as Git clone. All objects are downloaded and decrypted.

### Web UI (`SGVault.open()`)

```
  SGVault.open(sgSend, fullVaultKey)
    → derive keys (~600 ms, no network)
    → GET branch index (optional, 404 OK)
    → GET named ref
    → GET clone ref (404 OK for fresh/read-only)
    → GET commit
    → GET root tree → decrypt entries → create folder stubs
    → GET settings blob
    → return vault object with lazy in-memory tree
    → network: always 6 GETs regardless of vault size
```

```
  Property              git clone          sgit clone        SGVault.open()
  ─────────────────── ──────────────────  ────────────────  ─────────────────
  Network calls       proportional to     proportional to   always 6 GETs
                      repo size           vault size
  251 MB vault        ~251 MB transfer    ~251 MB transfer  ~2 KB transfer
  Local state         .git/ + working     .sgit/ + files    in-memory tree only
                      tree on disk        on disk
  Offline after?      fully offline       fully offline     cached blocks only
  Sub-folders loaded  all                 all               stubs (on demand)
  File content        all decrypted       all decrypted     on demand
  Time (251 MB vault) minutes             minutes           1–2 s
```

## 3. `git commit` / `sgit commit` / `vault._commit()`

### Git

```
  git add file.txt
  git commit -m "Add file"
    → hash-object: SHA-1(blob content) → write to .git/objects/
    → write-tree: SHA-1(tree) → write to .git/objects/
    → commit-tree: SHA-1(commit) → write to .git/objects/
    → update .git/refs/heads/main → new commit SHA-1
    → network: 0 (purely local)
```

Git commit is entirely local. Nothing touches the network until `git push`.

### SGit CLI (`sgit commit`)

```
  sgit commit -m "Add file"
    → encrypt new blobs → PUT each to server
    → create tree objects → PUT each to server
    → create commit object → PUT to server
    → write clone ref → PUT to server
    → network: N+4 PUTs (same formula as web UI)
```

### Web UI (`vault._commit()`)

```
  vault._commit(message)
    → _buildTreeEntries() (unloaded stubs pass through unchanged)
    → PUT settings blob
    → PUT sub-tree objects (one per loaded folder)
    → PUT root tree
    → PUT commit
    → PUT clone ref
    → network: N+4 PUTs
```

```
  Property              git commit         sgit commit       vault._commit()
  ─────────────────── ──────────────────  ────────────────  ─────────────────
  Network calls       0                  N+4 PUTs           N+4 PUTs
  Local state update  .git/refs/heads/*  .sgit/ + clone ref in-memory only
  Named ref update    no                 no                 no
  Offline capable     yes                no                 no
  History preserved   yes                yes                yes
  Old blobs deleted   no                 no                 no
```

The SGit model has no staging area. Git's `git add` + `git commit` two-step is collapsed into a single operation. The equivalent of `git add` is implicit in the mutation methods (`addFile`, `updateFile`, etc.) which update the in-memory tree. The `_commit()` call serialises whatever state the tree is in.

## 4. `git push` / `sgit push` / `vault.push()`

### Git

```
  git push origin main
    → connect to remote
    → negotiate: what does remote have that we don't?
    → send pack file: all new objects remote doesn't have
    → update remote ref: refs/heads/main → new commit SHA-1
    → network: proportional to new objects since last push
```

Git push sends all new objects (commits, trees, blobs) that the remote doesn't have yet, then updates the ref.

### SGit CLI and Web UI

```
  sgit push  /  vault.push()
    → PUT bare/refs/{refFileId}   body: AES-GCM({ commit_id: headCommitId })
    → network: 1 PUT only
```

SGit push is a **single PUT**. This is because every `_commit()` already writes all objects to the server. There are no new objects to send — the named ref is the only thing that hasn't been updated yet.

```
  Property              git push           sgit push
  ─────────────────── ──────────────────  ────────────────────────────────────
  Network calls       proportional to     1 PUT always
                      new objects
  Objects sent        all new blobs,      none (already on server)
                      trees, commits
  What changes        remote ref +        named ref only
                      remote objects
  Conflict check      yes (fast-forward   no (caller must check getBehindCount)
                      check on server)
  Force push          yes (--force flag)  effectively always possible (no check)
  Pack negotiation    yes                 no
```

The absence of pack negotiation in SGit push is a consequence of the always-on-server commit model. There is nothing to negotiate — the server has everything already.

## 5. `git pull` / `sgit pull` / `vault.pull()`

### Git

```
  git pull origin main
    → git fetch: receive new objects from remote, update refs/remotes/origin/main
    → git merge FETCH_HEAD: merge remote branch into local branch
    → if fast-forward: move local ref forward
    → if diverged: create merge commit (or rebase with --rebase)
    → network: proportional to new objects
```

Git pull handles diverged histories via merge or rebase. Three-way merge is supported.

### SGit

```
  sgit pull  /  vault.pull()
    → GET named ref (live)
    → if headCommitId != namedHeadId: refuse ("push first")
    → fast-forward: _headCommitId = _namedHeadId = serverNamedHead
    → PUT clone ref → serverNamedHead
    → _loadTreeFromCommit(serverNamedHead)
    → network: 1 GET (ref) + 1 PUT (clone ref) + 3 GETs (commit, tree, settings)
```

```
  Property              git pull           sgit pull
  ─────────────────── ──────────────────  ────────────────────────────────────
  Network calls       proportional to     5 requests (ref+commit+tree+settings+cloneRef)
                      new objects
  Merge support       yes (3-way merge)   no (fast-forward only)
  Rebase support      yes (--rebase)      no
  Conflict resolution interactive         refused (must push first)
  Diverged histories  merge commit        error — not supported
  Working tree update yes (file changes)  yes (in-memory tree rebuilt)
```

The fast-forward-only limitation is the most significant operational gap between SGit and Git for multi-user workflows. It reflects the current state of the web UI implementation, not a fundamental design constraint — the commit object supports multiple parents and the object model could support merges.

## 6. `git branch` / SGit branches

### Git

```
  git branch feature/login      → create .git/refs/heads/feature/login
  git checkout feature/login    → update .git/HEAD → new branch
  git branch -d feature/login   → delete ref file
  git branch -a                 → list all local + remote-tracking refs
```

Git branches are free: creating a branch costs one ref file write. Switching branches is a working-tree update. Branches are local by default; `git push -u origin feature/login` publishes them.

### SGit

```
  Web UI:
    One branch only (the named branch). No branch creation in the web UI.
    The clone ref is always named "web-ui" and is a local working branch,
    not a user-visible named branch.

  CLI (sgit):
    Branches exist but are less prominent. Each vault has one named branch
    (created on init). Additional branches via the branch index (idx-pid-muw-*).
    The branch index stores: branch_id, branch_type, head_ref_id, name, created_at.
```

```
  Property              git branch         SGit branch
  ─────────────────── ──────────────────  ────────────────────────────────────
  Create branch       1 ref file          1 new ref-pid-muw-* + update branch index
  Branch names        human-readable      human-readable (stored in branch index)
  Branch ref IDs      path-based          HMAC-derived per-branch
  Switch branch       checkout (fast)     full tree rebuild (loadTreeFromCommit)
  Web UI support      n/a                 single branch only
  Remote branches     refs/remotes/*      no concept of remote
  Branch index        no                  idx-pid-muw-* (sgit only)
```

## 7. `git log` — Commit History

### Git

```
  git log
    → read .git/HEAD → current commit SHA-1
    → load commit object → print author, date, message
    → follow parents[0] → repeat
    → network: 0 (all objects local)
```

### SGit equivalent

```
  No dedicated log command yet (PROPOSED).
  Implementation would be:

  async function log(vault) {
    let cursor = vault._headCommitId
    while (cursor) {
      const commit = await objectStore.load(cursor)    // cache hit likely
      const plain  = await SGSendCrypto.decrypt(commit, readKey)
      const obj    = JSON.parse(new TextDecoder().decode(plain))
      const msg    = await decryptField(obj.message_enc, readKey)
      console.log(cursor, obj.timestamp_ms, msg)
      cursor = obj.parents[0] || null
    }
  }
```

```
  Property              git log            SGit log (proposed)
  ─────────────────── ──────────────────  ────────────────────────────────────
  Network calls       0                  0 (if cached) or N GETs (cold)
  Author shown        yes                no (not stored)
  Timestamp shown     yes                yes (plaintext ms in commit)
  Message shown       yes                yes (requires decrypt)
  Branch graph        yes (--graph)      yes (parents[] supports it)
  Diff per commit     yes (--stat, -p)   possible (compare trees, decrypt entries)
```

## 8. `git diff` / Comparing Commits

### Git

```
  git diff HEAD~1 HEAD
    → load commit HEAD, commit HEAD~1
    → compare their tree objects
    → for changed blobs: compute line-level diff
    → network: 0
```

### SGit equivalent

```
  No diff command yet (PROPOSED).
  But the data to support it exists:

  async function diff(vault, commitIdA, commitIdB) {
    const treeA = await loadTreeFromCommit(commitIdA)
    const treeB = await loadTreeFromCommit(commitIdB)

    // Compare entries: decrypt name_enc and content_hash_enc from both trees
    // Entries with same content_hash_enc → unchanged
    // Entries present in B but not A → added
    // Entries present in A but not B → removed
    // Same name, different content_hash → modified

    // For modified files: load both blobs, decrypt, compute line diff
  }
```

The `content_hash_enc` field (SHA-256(plaintext)[:12]) allows detecting changed files without downloading their content — the same optimisation Git uses (comparing tree SHA-1s before fetching blobs).

## 9. `git cat-file` / Reading Objects Directly

### Git

```
  git cat-file -p HEAD          → print commit object
  git cat-file -p HEAD:file.txt → print file content
  git cat-file blob abc123      → print raw blob content
```

All objects are accessible directly by hash, read from `.git/objects/`. No network.

### SGit equivalent

```
  sgit cat passphrase:vault_id /path/to/file   (PROPOSED — see Dynamic Access Part 3)

  Implementation:
    → derive keys
    → GET named ref → commitId
    → GET commit → tree_id
    → walk path: GET sub-trees as needed
    → GET blob → decrypt → stdout
    → network: 1 ref + 1 commit + depth tree GETs + 1 blob
```

Reading a specific object by ID requires the `read_key` — you cannot read any SGit object without it, unlike Git where any object is readable by hash alone.

## 11. The Fundamental Trade-off

Git's design optimises for offline-first, local-speed operations. Everything that can be done locally is done locally. Network is reserved for explicit synchronisation points (`push`, `fetch`, `pull`).

SGit's design optimises for zero-knowledge security. Everything is encrypted before it leaves the browser. The consequence is that the server becomes a dumb encrypted object store — it cannot participate in any negotiation, merge, or object-selection logic. All intelligence is in the client.

This means:

```
  Git server responsibilities:
    - Store objects
    - Pack negotiation (what does client have / need?)
    - Fast-forward checks
    - Access control (authentication + authorisation)
    - Garbage collection
    - Large file handling (Git LFS)

  SGit server responsibilities:
    - Store encrypted blobs at the path the client specifies
    - Authenticate PUT requests via write_key header
    - Return encrypted blobs by path
    - Nothing else — it cannot do anything else
```

The SGit server is deliberately simpler and more limited than a Git server. It is closer to an authenticated S3 bucket than to a full Git remote. This is the price of zero-knowledge: the server's capability is bounded by what it can do without seeing plaintext.

The dynamic access model and Cache API, described in the vault dynamic access series, are the client-side compensations for this limitation — achieving local-speed access without trusting the server with a local copy.
