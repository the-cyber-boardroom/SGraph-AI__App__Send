## 1. Wire Format — Every Encrypted Blob

Every encrypted value in the vault — file content, tree entries, commit messages, refs, settings — uses the same wire format:

```
[  12 bytes  ][  N bytes  ][  16 bytes  ]
   random IV    ciphertext   GCM auth tag
```

- **Algorithm:** AES-256-GCM
- **IV:** 12 bytes, `crypto.getRandomValues()`, fresh per encryption call
- **Auth tag:** 16 bytes, appended by Web Crypto to the ciphertext output
- **Total overhead:** 28 bytes per encrypted value

Encryption (from `sg-send-crypto.js`):
```javascript
const iv         = crypto.getRandomValues(new Uint8Array(12))
const ciphertext = await crypto.subtle.encrypt({ name: 'AES-GCM', iv }, key, data)
// Output: concat(iv[12], ciphertext+tag[N+16])
const result = new Uint8Array(12 + ciphertext.byteLength)
result.set(iv, 0)
result.set(new Uint8Array(ciphertext), 12)
return result.buffer
```

Decryption:
```javascript
const bytes      = new Uint8Array(data)
const iv         = bytes.slice(0, 12)
const ciphertext = bytes.slice(12)
return crypto.subtle.decrypt({ name: 'AES-GCM', iv }, key, ciphertext)
```

**The server never sees plaintext.** It receives and returns only `[IV | ciphertext | tag]`. It has no knowledge of file names, sizes, directory structure, commit messages, or any metadata.

## 2. Key Derivation — Two Paths

The key derivation algorithm depends on the vault key format.

### 2a. Standard Vault Key — `{passphrase}:{vault_id}`

Format: a human passphrase, colon separator, 4–24 lowercase alphanumeric vault ID.
Example: `correct-horse-battery:a1b2c3d4`

Two **parallel** PBKDF2 derivations from the same passphrase bytes, different salts:

```
passphrase bytes
    │
    ├─ PBKDF2(salt="sg-vault-v1:{vault_id}",        600 000 iter, SHA-256, 256 bit)
    │     → readBits  → AES-GCM CryptoKey  (read_key)
    │
    └─ PBKDF2(salt="sg-vault-v1:write:{vault_id}",   600 000 iter, SHA-256, 256 bit)
          → writeBits → hex string          (write_key)
```

Knowing `read_key` reveals nothing about `write_key` — different salts, independent PBKDF2 runs. This is the zero-knowledge property: a read-only share can be issued without granting write access.

From `read_key` bits, derive an HMAC key for deterministic file ID generation:

```
readBits → HMAC-SHA-256 key  (hmac_key)
    │
    ├─ HMAC(domain="sg-vault-v1:file-id:ref:{vault_id}")[:12 hex]
    │     → "ref-pid-muw-{hex12}"   (named HEAD ref file ID)
    │
    └─ HMAC(domain="sg-vault-v1:file-id:branch-index:{vault_id}")[:12 hex]
          → "idx-pid-muw-{hex12}"   (branch index file ID)
```

Per-branch clone ref (web-UI working branch):
```
HMAC(domain="sg-vault-v1:file-id:branch-ref:{vault_id}:web-ui")[:12 hex]
    → "ref-pid-snw-{hex12}"   (clone branch ref file ID)
```

### 2b. Simple Token — `word-word-NNNN`

Simple tokens use a 4-step derivation. The vault ID is computed from the token, not passed in:

```
token bytes
    │
    PBKDF2(salt="sgraph-send-v1" [FIXED], 600 000 iter, SHA-256, 256 bit)
    → aes_key_bits
    │
    ├─ HKDF(aes_key_bits, info="vault-read-key",  salt=∅, SHA-256, 256 bit)
    │     → read_key (AES-GCM CryptoKey)
    │
    ├─ HKDF(aes_key_bits, info="vault-write-key", salt=∅, SHA-256, 256 bit)
    │     → write_key (hex string)
    │
    └─ SHA-256(token bytes)[:12 hex chars]
          → vault_id (12 hex chars)
```

Then `hmac_key`, `refFileId`, and `branchIndexFileId` are derived identically to the standard path (same HMAC domain strings, same `vault_id`).

**Key difference:** the fixed PBKDF2 salt `"sgraph-send-v1"` means different vaults using simple tokens with different passphrases but the same vault_id would derive the same key material up to the HKDF split. The HKDF info strings and the vault_id-based HMAC domains ensure independence.

### 2c. Summary of Derived Material

| Name | Type | Used for |
|---|---|---|
| `read_key` | AES-256-GCM CryptoKey | Encrypt / decrypt all vault content |
| `write_key` | 256-bit hex string | Server authorisation header on PUT/DELETE |
| `hmac_key` | HMAC-SHA-256 CryptoKey | Derive deterministic file IDs |
| `refFileId` | `ref-pid-muw-{12hex}` | Named branch HEAD ref location |
| `branchIndexFileId` | `idx-pid-muw-{12hex}` | Branch index location |
| `cloneRefFileId` | `ref-pid-snw-{12hex}` | Web-UI working branch ref location |
| `vault_id` | 4–24 alphanumeric chars | Namespace for all server paths |

## 3. The 4-Segment ID Naming Scheme

Every file stored on the server has a self-describing ID in the format:

```
{type}-{derivation}-{mutability}-{hex_id}
```

| Segment | Values | Meaning |
|---|---|---|
| `type` | `obj` `ref` `idx` | Data blob / ref pointer / branch index |
| `derivation` | `cas` `pid` `rnd` | Content-addressed / HMAC-derived / random |
| `mutability` | `imm` `muw` `snw` | Immutable / multi-writer / single-writer |
| `hex_id` | 12 hex chars (6 bytes) | SHA-256 hash or HMAC truncated to 6 bytes |

### Object types in practice

| ID | Server path | Created by | Changes? |
|---|---|---|---|
| `obj-cas-imm-{sha256[:12]}` | `bare/data/{id}` | Every encrypt+store | Never — SHA-256 of ciphertext |
| `ref-pid-muw-{hmac[:12]}` | `bare/refs/{id}` | `init`, `push` | On every `push` (named HEAD advances) |
| `ref-pid-snw-{hmac[:12]}` | `bare/refs/{id}` | First `commit` after `open` | On every `commit` (clone ref advances) |
| `idx-pid-muw-{hmac[:12]}` | `bare/idx/{id}` | `sgit push` (CLI only currently) | On branch operations |

### Immutability guarantee

`obj-cas-imm-*` IDs encode a cryptographic fact: the ID *is* SHA-256(ciphertext)[:12 hex]. If you retrieve a block with the same ID on two different days, the ciphertext is guaranteed to be identical. This makes permanent browser caching safe without any cache-invalidation logic.

## 4. Object Store — Content-Addressed Block Storage

All vault data (file content, tree objects, commit objects, ref payloads, settings) is stored and loaded through `SGVaultObjectStore`. There are two operations:

### `store(ciphertext) → objectId`

```javascript
async store(ciphertext) {
    const objectId = await this.computeObjectId(ciphertext)    // SHA-256(ciphertext)[:12]
    const filePath = `bare/data/${objectId}`
    await this._sgSend.vaultWrite(this._vaultId, filePath, this._writeKey, ciphertext)
    return objectId
}
```

HTTP:
```
PUT /api/vault/write/{vaultId}/bare/data/obj-cas-imm-{hex12}
Headers:
  Content-Type: application/octet-stream
  x-sgraph-vault-write-key: {write_key hex}
Body: [12 IV bytes][ciphertext][16 tag bytes]
```

The object ID is computed before the upload. If the same ciphertext is stored twice (same plaintext + same key → different IV → different ciphertext → different ID), two objects are created. This is intentional: the same file uploaded twice gets two distinct blob IDs, preserving history.

### `load(objectId) → ArrayBuffer`

```javascript
async load(objectId) {
    const filePath = `bare/data/${objectId}`
    // Check Cache API first (imm blocks only)
    const cached = await this._cacheGet(objectId, filePath)
    if (cached) return cached
    // Network fetch
    const data = await this._sgSend.vaultRead(this._vaultId, filePath)
    // Store in cache, return to caller
    await this._cachePut(objectId, filePath, data)
    return data
}
```

HTTP:
```
GET /api/vault/read/{vaultId}/bare/data/obj-cas-imm-{hex12}
→ 200 [12 IV bytes][ciphertext][16 tag bytes]
→ 404 if object does not exist
```

### `loadLarge(objectId) → ArrayBuffer`

For files > 4 MB (Lambda response size limit). Tries presigned S3 URL first, falls back to direct read:

```
GET /api/vault/presigned/read-url/{vaultId}/bare/data/obj-cas-imm-{hex12}
→ 200 { "url": "https://s3.amazonaws.com/...?X-Amz-Signature=..." }
      → GET {presigned_url}   (direct from S3, bypasses Lambda)
→ 400 (memory/local-dev mode) → falls back to vaultRead()
```

### Immutable block cache (Cache API)

All `obj-cas-imm-*` blocks are cached in the browser's Cache API:

```
Cache name:  sg-vault-blocks
Cache key:   https://sgvault/{vaultId}/bare/data/obj-cas-imm-{hex12}
```

- Cache is checked in `_cacheGet()` before any network call
- On miss, fetched ciphertext is stored in `_cachePut()` before returning to caller
- `data.slice(0)` copies the ArrayBuffer so the caller can transfer it without invalidating the cache copy
- `typeof caches === 'undefined'` guard means cache degrades silently in environments without Cache API
- Storage quota errors are silently ignored — vault works correctly without the cache, just slower

**Performance measured:**
- Cold load (no cache): ~6 s for a 251 MB / 155-file vault
- Warm cache: 820 ms

## 5. Object Schemas

### Commit object (`commit_v2`)

Stored encrypted as `obj-cas-imm-*`. Plaintext before encryption:

```json
{
  "schema":       "commit_v2",
  "parents":      ["obj-cas-imm-{parent_commit_hex}"],
  "tree_id":      "obj-cas-imm-{root_tree_hex}",
  "timestamp_ms": 1744754400000,
  "message_enc":  "{base64(AES-GCM(commit_message, read_key))}",
  "branch_id":    null,
  "signature":    null
}
```

Note: `message_enc` is encrypted separately (base64-encoded AES-GCM output). The entire commit object is then encrypted again as the outer layer. So the commit message is double-encrypted — once as `message_enc`, and again as part of the commit JSON blob.

### Tree object (`tree_v1`)

Stored encrypted as `obj-cas-imm-*`. Plaintext before encryption:

```json
{
  "schema": "tree_v1",
  "entries": [
    {
      "name_enc":         "{base64(AES-GCM(filename, read_key))}",
      "blob_id":          "obj-cas-imm-{hex12}",
      "tree_id":          null,
      "size_enc":         "{base64(AES-GCM(size_string, read_key))}",
      "content_hash_enc": "{base64(AES-GCM(sha256_hex[:12], read_key))}"
    },
    {
      "name_enc":         "{base64(AES-GCM(dirname, read_key))}",
      "blob_id":          null,
      "tree_id":          "obj-cas-imm-{sub_tree_hex}",
      "size_enc":         null,
      "content_hash_enc": null
    }
  ]
}
```

Every metadata field (`name`, `size`, `content_hash`) is individually AES-GCM encrypted and base64-encoded. The `blob_id` and `tree_id` pointers are stored **in plaintext** — they are object IDs (SHA-256 hashes), not semantic data. A file entry has `blob_id` set and `tree_id` null. A directory entry has `tree_id` set and `blob_id` null.

The entire tree JSON is then encrypted as the outer AES-GCM layer and stored as `obj-cas-imm-*`.

### Ref object

Stored encrypted at `bare/refs/{refFileId}`. Plaintext before encryption:

```json
{ "commit_id": "obj-cas-imm-{hex12}" }
```

### Branch index (`branch_index_v1`)

Stored encrypted at `bare/idx/{branchIndexFileId}`:

```json
{
  "schema": "branch_index_v1",
  "branches": [
    {
      "branch_id":   "branch-named-...",
      "branch_type": "named",
      "head_ref_id": "ref-pid-muw-{hex12}",
      "name":        "main",
      "created_at":  "2026-04-15T10:00:00Z"
    }
  ]
}
```

### Vault settings (`.vault-settings`)

Stored as a regular blob in the root tree under the special name `.vault-settings`. Plaintext before encryption:

```json
{
  "vault_name":  "My Vault",
  "vault_id":    "a1b2c3d4",
  "created":     "2026-04-15T10:00:00Z",
  "version":     3,
  "description": ""
}
```

## 6. What the Server Knows

| Data | Server has it? | Notes |
|---|---|---|
| Vault ID | Yes | Plaintext in URL path |
| File names | No | Encrypted in tree entries |
| File sizes | No | Encrypted in tree entries |
| Directory structure | No | Encrypted in tree entries |
| File content | No | Encrypted blob |
| Commit messages | No | Double-encrypted |
| Number of objects | Yes | Count of blobs under `bare/data/` |
| When objects were written | Yes | S3 object metadata / server logs |
| Write key (for auth) | Yes | Submitted as `x-sgraph-vault-write-key` header |
| Read key | No | Never sent to server |
| Passphrase | No | Never sent to server |

The write key authorises write operations but cannot be used to read content — it is derived via a separate PBKDF2 run with a different salt, so it is cryptographically independent of the read key.
