## 2. The `BrowseDataSource` Interface — Your Foundation

This is the most important thing in this brief. Any storage provider (vault, zip, IndexedDB, memory) that implements these three methods can be dropped into the browse component, the VFS bridge, and any future viewer component.

```javascript
class MyDataSource {

    // Required: hierarchical tree for the file browser UI
    // Returns: { name, children: { name: TreeNode }, files: [{ path, name, size }] }
    getTree() { ... }

    // Required: fetch a file's bytes by vault-relative path
    // Returns: Promise<ArrayBuffer>
    async getFileBytes(path) { ... }

    // Required: flat file list for search and path resolution
    // Returns: [{ path, name, dir, size }]
    getFileList() { ... }

    // Metadata (optional but useful)
    getOrigName() { return 'my-store'; }
    getOrigSize() { return 0; }

    // Write capability flag (optional)
    writable = false;
}
```

### Reference implementations

| Implementation | File | Storage backend |
|---|---|---|
| `VaultDataSource` | `sgraph_ai_app_send__ui__vault/v0/v0.2/v0.2.0/_common/js/adapters/vault-data-source.js` | SGVault (encrypted, version-controlled) |
| `ZipDataSource` | `sgraph_ai_app_send__ui__user/v0/v0.3/v0.3.2/_common/js/components/send-download/zip-data-source.js` | JSZip in-memory |

**For the Browser VFS:** your `SGFileSystem` providers map directly to this interface. When a tool opens a vault, it hands the browse component a `VaultDataSource`. When it exports a zip, it hands a `ZipDataSource`. The component never changes.

### Lazy sub-tree loading

`VaultDataSource` implements lazy loading — vault sub-trees are only fetched and decrypted when accessed. The `getTree()` call marks lazy folders with `_lazy: true` so the browse UI can show a loading indicator:

```javascript
// In VaultDataSource._convertNode():
if (child._loaded === false && child._tree_id) {
    result.children[name]._lazy      = true;
    result.children[name]._folderPath = '/' + childPath;
}
```

When the user expands a lazy folder, call `dataSource.loadFolder(folderPath)` which triggers `_ensureLoaded()`. This is the pattern your `sg-vault-browse` component should follow — do not pre-load the entire vault tree on open.

## 3. The VFS Bridge — HTML Files Loading Assets From the Vault

This solved a fundamental problem: **HTML files displayed in a sandboxed `<iframe src="blob:...">` have no base URL**. Any relative `fetch('content/manifest.json')` or `img.src = 'images/slide.png'` fails — the blob URL carries no path context.

### The solution: postMessage VFS bridge injected into the HTML

Before creating the blob URL, inject a `<script>` that overrides:

```
window.fetch()                         ← catches XHR/fetch calls for JSON, CSS, etc.
HTMLImageElement.prototype.src setter  ← catches img.src = 'relative/path' assignments
MutationObserver                       ← backup for img.setAttribute('src', ...) calls
```

Each interception sends a `__sgVfsReq` postMessage to the parent window. The parent resolves the relative path against the HTML file's location in the vault, fetches and decrypts via `dataSource.getFileBytes()`, then returns the bytes as an `ArrayBuffer` via `__sgVfsReply`.

```
iframe                              parent
  img.src = 'content/slide.png'
  → __sgVfsReq { id, url }     →
                                    resolve path
                                    dataSource.getFileBytes(path)
                                    ← __sgVfsReply { id, buf, mime }
  blob = new Blob([buf], {type:mime})
  img.src = URL.createObjectURL(blob)
```

### Why `HTMLImageElement.prototype.src` setter, not MutationObserver

The MutationObserver fires when nodes are added to the DOM. But presenter applications (like the one tested) create `new Image()` elements and set `img.src` *before* appending to the DOM:

```javascript
var img = new Image();
img.src = 'content/slides/01/slide.png';   // ← MutationObserver MISSES this
document.body.appendChild(img);
```

Overriding the prototype setter intercepts the assignment at the moment it happens, regardless of DOM state. The MutationObserver is still useful as a backup for `setAttribute('src', ...)` calls.

### The `__sgVfs` flag lifecycle — critical detail

```javascript
set: function(val) {
    if (!val || val.startsWith('http') || val.startsWith('blob:') || ...) {
        _oset.call(this, val); return;           // pass through absolute URLs
    }
    this.__sgVfs = true;    // ← set BEFORE async fetch
    var el = this;
    _vfsReq(val, function(d) {
        el.__sgVfs = false;  // ← CLEAR after blob URL applied
        ...
        _oset.call(el, blobUrl);
    });
}
```

**Do not permanently set `__sgVfs = true`.** The flag prevents the MutationObserver from double-processing *while the async fetch is in flight*. If it is never cleared, the second time a presenter updates `img.src` on the same `<img>` element (navigating between slides), the setter falls through to the native setter — which tries to load the relative path from blob context and fails silently.

The fix: set before the async request, clear in the callback.

### Path resolution

```javascript
// htmlDir = folder of the HTML file within the vault
// e.g. if HTML is at 'content/presenter.html', htmlDir = 'content/'
// if HTML is at root 'presenter.html', htmlDir = ''

function _resolvePath(base, relative) {
    if (!relative) return null;
    if (relative.startsWith('/')) return relative.slice(1);   // absolute vault path
    var parts = (base + relative).split('/');
    var out = [];
    for (var i = 0; i < parts.length; i++) {
        if (parts[i] === '..') out.pop();
        else if (parts[i] && parts[i] !== '.') out.push(parts[i]);
    }
    return out.join('/');
}
```

### Security properties of the sandboxed iframe

```html
<iframe sandbox="allow-scripts" src="blob:..."></iframe>
```

- `allow-scripts` only — no `allow-same-origin`, no `allow-top-navigation`
- The iframe cannot read parent DOM, localStorage, cookies, or IndexedDB
- Even with an XSS in the HTML file, the vault contents are safe
- postMessage is the only channel; `e.source !== iframeEl.contentWindow` check ensures only that specific iframe's messages are handled
- No external URLs are loaded — all assets come from the vault via the bridge

The full bridge script is in `send-browse--v0.3.2.js` around line 564. Lift it directly — do not reinvent it.

### What the bridge currently handles

| Asset type | Mechanism | Status |
|---|---|---|
| `fetch()` calls (JSON, CSS, JS, etc.) | `window.fetch` override | ✅ Working |
| `<img src="...">` assignments | `HTMLImageElement.prototype.src` setter | ✅ Working |
| `img.setAttribute('src', ...)` | MutationObserver | ✅ Working |
| CSS `background-image: url(...)` | Not intercepted | ❌ Not yet |
| `<script src="...">` | Not intercepted | ❌ Not yet |
| `<link rel="stylesheet" href="...">` | Not intercepted | ❌ Not yet |

For Tools team's `sg-vault-browse`, the fetch and img.src overrides cover the vast majority of real-world HTML content. CSS background images are the next gap — if your vault content uses them, a `XMLHttpRequest` override similar to the fetch one would handle it.

## 4. Immutable Block Cache

Every file in a vault is stored as one or more `obj-cas-imm-{SHA256[:12]}` objects. The naming scheme encodes permanent immutability: the object ID *is* a SHA-256 fingerprint of the encrypted ciphertext. Same ID = same bytes, forever.

### The cache

```
Browser Cache API
  Cache name:  sg-vault-blocks
  Cache key:   https://sgvault/{vaultId}/bare/data/{objectId}
               e.g. https://sgvault/jrinl85l/bare/data/obj-cas-imm-a2ad790bd3f1
```

Implemented in `SGVaultObjectStore.load()` and `loadLarge()`. The cache is checked before every network fetch; on a miss, the fetched bytes are stored before returning to the caller.

**What is cached:** encrypted ciphertext — exactly what the server stores. Decryption still happens in the browser (pure CPU, ~ms). The expensive network round-trip is eliminated.

**Guard:** `objectId.includes('-imm-')` — only blocks explicitly marked immutable are cached. `ref-pid-muw-*` (HEAD refs) and `ref-pid-snw-*` (clone branch refs) are never cached because they change with every commit.

### Measured performance

| Load condition | Time |
|---|---|
| Cold load (no cache, 251 MB vault / 155 files) | ~6 s |
| Second load (cache populated) | ~1 s |
| Subsequent loads (browser cache + sg-vault-blocks) | 820 ms |

### Storage management API (for your settings UI)

```javascript
const cache = await caches.open('sg-vault-blocks');

// All cached blocks across all vaults
const keys = await cache.keys();
// → [Request { url: 'https://sgvault/abc123/bare/data/obj-cas-imm-...' }, ...]

// Per-vault breakdown
const vaultBlocks = keys.filter(r => r.url.includes(`/${vaultId}/`));

// Approximate size (response body size, before decompression)
let total = 0;
for (const req of vaultBlocks) {
    const resp = await cache.match(req);
    const buf  = await resp.arrayBuffer();
    total += buf.byteLength;
}

// Clear one vault
for (const req of vaultBlocks) await cache.delete(req);

// Clear all vault blocks
await caches.delete('sg-vault-blocks');
```

The cache is origin-scoped: `https://dev.tools.sgraph.ai` gets its own cache, separate from `https://dev.vault.sgraph.ai`. If you want to share the cache across origins (vault + tools on the same user's browser), you would need a Service Worker with a shared scope — not yet implemented.

## 5. Connecting to the Browser VFS Architecture

From the `v0.20.33__arch-brief__browser-virtual-file-system.md` brief:

> Components do not choose the provider. The tool (or the environment configuration) chooses the provider. The component just uses `fs.readFile()` and `fs.writeFile()`.

The `BrowseDataSource` interface IS the read side of that contract. The mapping:

| VFS method | BrowseDataSource method |
|---|---|
| `fs.readFile(path)` | `dataSource.getFileBytes(path)` → `ArrayBuffer` |
| `fs.listFolder(path)` | `dataSource.getTree()` → hierarchical node |
| `fs.stat(path)` | `dataSource.getFileList()` → flat entry with `size` |

Your `SGFileSystem` vault provider should wrap `VaultDataSource`:

```javascript
class SGFileSystem {
    constructor({ provider, vaultKey }) {
        if (provider === 'vault') {
            this._ds = new VaultDataSource(
                await SGVault.open(sgSend, vaultKey),
                accessKey  // null for read-only
            );
        }
    }

    async readFile(path)     { return this._ds.getFileBytes(path); }
    async listFolder(path)   { /* walk getTree() from path */ }
    async writeFile(path, b) { return this._ds.saveFile(folderOf(path), nameOf(path), b); }
    async deleteFile(path)   { return this._ds.deleteFile(folderOf(path), nameOf(path)); }
}
```

`VaultDataSource` already implements the write methods (`saveFile`, `renameFile`, `deleteFile`, `createFolder`, `deleteFolder`, `moveFile`, `moveFolder`) — you get them for free.

## 6. Expected 404s — Do Not Treat as Errors

When opening any vault, two 404s will always appear in the network tab. These are correct zero-knowledge behaviour:

| Request | Condition | Handled by |
|---|---|---|
| `bare/refs/ref-pid-snw-{hmac}` | Clone branch ref — only exists after the first web-UI commit | `readRef()` returns `null`; vault falls back to named HEAD |
| `bare/idx/idx-pid-muw-{hmac}` | Branch index — only created by `sgit init`/`sgit push` | `readBranchIndex()` returns `null`; vault uses derived `refFileId` |

These are not errors. The server correctly returns 404 for any key it doesn't hold. The vault client handles `null` returns gracefully. Your VFS vault provider should propagate the same pattern: a missing file returns `null`, not a thrown error, unless the file is required.

## 9. The Local Dev Server Pattern

The vault local server (`scripts/vault__run-locally.sh`) solves a problem the Tools repo will also hit: the `index.html` loads shared components from a CDN, so local changes to those components are invisible during development.

The fix is three steps:
1. Copy locale dirs rather than symlinking them (so you can patch `index.html`)
2. Merge all IFD component layers in order (base → overlay → latest) so every required file is present
3. Patch `index.html` to replace CDN URLs with local `/_common/` paths

This pattern applies to any Tools component that is loaded from `dev.tools.sgraph.ai` in an `index.html` that the server generates — same fix, same approach.

## Quick Reference

```
BrowseDataSource interface:
  getTree()              → TreeNode
  getFileBytes(path)     → Promise<ArrayBuffer>
  getFileList()          → [{ path, name, dir, size }]

Cache API key format:
  https://sgvault/{vaultId}/bare/data/obj-cas-imm-{hex12}

VFS bridge postMessage protocol:
  iframe → parent:  { __sgVfsReq: id,  url: 'relative/path' }
  parent → iframe:  { __sgVfsReply: id, buf: ArrayBuffer, mime: 'image/png' }
                 or { __sgVfsReply: id, err: true }

Object ID naming rules:
  obj-cas-imm-*  →  immutable, safe to cache permanently
  ref-pid-muw-*  →  multi-writer, never cache
  ref-pid-snw-*  →  single-writer, never cache
  idx-pid-muw-*  →  branch index, never cache
```
