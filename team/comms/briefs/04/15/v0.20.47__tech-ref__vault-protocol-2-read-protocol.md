## Overview

Read operations never write to the server. They fetch encrypted objects, decrypt in-browser, and build the in-memory tree. The sequence for every read is:

```
fetch ciphertext (GET /api/vault/read/...)
    → check Cache API first (imm blocks only)
    → decrypt with read_key (AES-256-GCM)
    → parse JSON / use ArrayBuffer
```

The only exception is `pull`, which also writes a ref — covered in Part 3.

## 1. `open(sgSend, fullVaultKey)` — Opening a Vault

This is the web equivalent of `git clone` — except nothing is cloned locally. The vault remains on the server; only the current tree state is materialised in memory. Opening a 251 MB vault fetches less than 1 KB of data for the root tree (folder stubs are lazy).

### Step-by-step

**Step 0: Key derivation** (no network calls)

Parse the vault key and run PBKDF2 to derive `read_key`, `write_key`, `hmac_key`. Then derive all deterministic file IDs via HMAC. See Part 1 §2 for full detail. This step takes ~300–600 ms (600 000 PBKDF2 iterations × 2).

```
SGVaultCrypto.deriveKeys(passphrase, vaultId)
    → read_key, write_key, hmac_key,
      refFileId (ref-pid-muw-*),
      branchIndexFileId (idx-pid-muw-*)

SGVaultCrypto.deriveBranchRefFileId(hmacKey, vaultId, 'web-ui')
    → cloneRefFileId (ref-pid-snw-*)
```

**Step 1: Read branch index** (1 GET, optional — 404 is expected and handled)

```
GET /api/vault/read/{vaultId}/bare/idx/{branchIndexFileId}
→ 200: [IV][encrypted branch_index_v1 JSON][tag]
        decrypt → { schema, branches: [{ branch_type, head_ref_id, ... }] }
        find named branch → override refFileId with live head_ref_id
→ 404: use HMAC-derived refFileId (vault predates branch index or has no branches)
```

Purpose: the branch index stores the *live* `head_ref_id` for the named branch. Without it, the client uses the HMAC-derived `refFileId`, which is also the named HEAD for single-branch vaults. For multi-branch vaults (CLI-created), this step is how the client discovers the current branch.

**Step 2: Read named HEAD ref** (1 GET, required)

```
GET /api/vault/read/{vaultId}/bare/refs/{refFileId}
→ 200: [IV][encrypted { "commit_id": "obj-cas-imm-..." }][tag]
        decrypt → { commit_id }
        → namedCommitId
→ 404: throw "Vault not found: HEAD ref missing"
```

This is the authoritative pointer. It is **never** served from the Cache API — it must be fetched live to detect if another client has pushed.

**Step 3: Read clone branch ref** (1 GET, 404 expected for fresh vaults)

```
GET /api/vault/read/{vaultId}/bare/refs/{cloneRefFileId}
→ 200: [IV][encrypted { "commit_id": "..." }][tag]
        decrypt → { commit_id }
        → cloneCommitId
→ 404: cloneCommitId = null (ref created lazily on first commit)
```

`headCommitId = cloneCommitId || namedCommitId`

The clone ref tracks the web-UI working branch — it advances on every `commit()` and is only pushed to the named ref via `push()`. For a read-only vault (no `accessKey` in the UI), this 404 happens every time and is expected.

**Step 4: Load root commit** (1 GET, via object store + cache)

```
GET /api/vault/read/{vaultId}/bare/data/{headCommitId}
→ [IV][encrypted commit_v2 JSON][tag]
   decrypt → { schema, parents, tree_id, timestamp_ms, message_enc, ... }
```

**Step 5: Load root tree** (1 GET, via object store + cache)

```
GET /api/vault/read/{vaultId}/bare/data/{commit.tree_id}
→ [IV][encrypted tree_v1 JSON][tag]
   decrypt → { schema, entries: [...] }
   for each entry: decrypt name_enc, size_enc, content_hash_enc individually
```

**Step 6: Populate in-memory tree (root level only)**

```javascript
this._tree = { '/': { type: 'folder', children: {} } }

for (const entry of tree.entries) {
    if (entry.name === '.vault-settings') {
        // Load settings blob, decrypt, parse JSON → this._settings
    } else if (entry.tree_id) {
        // Sub-tree: create lazy folder stub — DO NOT fetch yet
        this._tree['/'].children[entry.name] = {
            type: 'folder', children: {}, _tree_id: entry.tree_id, _loaded: false
        }
    } else {
        // File: insert directly (blob_id stored, content NOT fetched yet)
        this._tree['/'].children[entry.name] = {
            type: 'file', blob_id: entry.blob_id, size: entry.size, content_hash: entry.content_hash
        }
    }
}
```

Folders in the root tree become lazy stubs. Their content is fetched only when the user expands them (see §3). Files at root are indexed but their ciphertext is not fetched until `getFile()` is called.

### Total HTTP calls on `open()`

| # | Request | Expected result |
|---|---|---|
| 1 | `GET .../bare/idx/{branchIndexFileId}` | 200 or 404 |
| 2 | `GET .../bare/refs/{refFileId}` | 200 (required) |
| 3 | `GET .../bare/refs/{cloneRefFileId}` | 200 or 404 |
| 4 | `GET .../bare/data/{headCommitId}` | 200 |
| 5 | `GET .../bare/data/{commit.tree_id}` | 200 |
| 6 | `GET .../bare/data/{settingsBlobId}` | 200 |

6 requests total on first open. On subsequent opens of the same vault, requests 4–6 hit the Cache API (zero network) if the vault hasn't changed. Requests 1–3 always hit the network (refs can change).

## 2. `getFile(folderPath, fileName)` — Reading a File

```javascript
async getFile(folderPath, fileName) {
    const folder = this._findNode(folderPath)
    const entry  = folder.children[fileName]

    const LARGE_FILE_THRESHOLD = 4 * 1024 * 1024   // 4 MB
    const encrypted = entry.size > LARGE_FILE_THRESHOLD
        ? await this._objectStore.loadLarge(entry.blob_id)
        : await this._objectStore.load(entry.blob_id)

    return SGSendCrypto.decrypt(encrypted, this._readKey)
}
```

### Standard path (≤ 4 MB)

```
1. Check Cache API for https://sgvault/{vaultId}/bare/data/{blob_id}
   → HIT:  return cached ciphertext, decrypt in browser
   → MISS: GET /api/vault/read/{vaultId}/bare/data/{blob_id}
            → [IV][ciphertext][tag]
            store in Cache API
            decrypt → plaintext ArrayBuffer
```

The decryption is pure CPU — Web Crypto `AES-256-GCM.decrypt`. No network involved after the first fetch.

### Large file path (> 4 MB)

```
1. GET /api/vault/presigned/read-url/{vaultId}/bare/data/{blob_id}
   → 200: { "url": "https://s3.amazonaws.com/...?X-Amz-Signature=...&X-Amz-Expires=900" }
          GET {presigned_url}    (direct S3 fetch, bypasses Lambda 6 MB response limit)
          → [IV][ciphertext][tag]
          store in Cache API
          decrypt
   → 400: presigned URL not available (memory mode / local dev)
          fall back to direct GET /api/vault/read/{vaultId}/bare/data/{blob_id}
```

The presigned URL is valid for 900 seconds (15 minutes). The S3 fetch goes directly from browser to S3 — the Lambda function is not in the hot path for large file transfers.

### What is NOT fetched during `getFile()`

- The commit or tree object (already in memory from `open()`)
- Any other file in the vault
- Any folder structure

Each `getFile()` is exactly one network round-trip on cold cache, zero on warm cache.

## 3. `loadSubTreeOnDemand(folderPath)` — Lazy Folder Expansion

Folders are opened as stubs during `open()` — only the root level is materialised. When the user expands a folder in the UI, `loadSubTreeOnDemand()` is called with the folder's vault path.

```javascript
async loadSubTreeOnDemand(folderPath) {
    const node = this._findNode(folderPath)
    if (node._loaded !== false || !node._tree_id) return false  // already loaded

    const subTree = await this._commitManager.loadTree(node._tree_id)
    // loadTree: GET .../bare/data/{tree_id} → decrypt → parse entries → decrypt each field

    for (const entry of subTree.entries) {
        if (entry.tree_id) {
            // Nested folder: create another lazy stub
            node.children[entry.name] = {
                type: 'folder', children: {}, _tree_id: entry.tree_id, _loaded: false
            }
        } else {
            // File: insert with blob_id (content still not fetched)
            node.children[entry.name] = {
                type: 'file', blob_id: entry.blob_id, size: entry.size, content_hash: entry.content_hash
            }
        }
    }

    node._loaded = true
    delete node._tree_id
    return true
}
```

### HTTP calls per folder expansion

```
GET /api/vault/read/{vaultId}/bare/data/{tree_id}
→ [IV][encrypted tree_v1 JSON][tag]
   decrypt tree JSON → decrypt each entry's name_enc, size_enc, content_hash_enc
```

One GET per folder expansion. Sub-folders within that folder get lazy stubs and are NOT fetched until expanded. This means a vault with 155 files in 20 folders only fetches the tree objects for folders the user actually navigates into.

### `needsLoading(folderPath)`

```javascript
needsLoading(folderPath) {
    const node = this._findNode(folderPath)
    return node && node.type === 'folder' && node._loaded === false && !!node._tree_id
}
```

Called by the UI before rendering a folder to decide whether to show a loading spinner. Returns `true` only if the folder has a server-side tree object that hasn't been fetched yet.

### Lazy loading during commit (_buildTreeEntries)

A critical invariant: **unloaded folders preserve their `_tree_id` during commit**. If a user uploads a new file to `/root/` but never expanded `/root/archive/` (which has 100 files), the commit must not destroy the archive's contents.

```javascript
async _buildTreeEntries(node) {
    for (const [name, entry] of Object.entries(node.children)) {
        if (entry.type === 'folder') {
            if (entry._loaded === false && entry._tree_id) {
                // NOT loaded → reuse existing tree_id unchanged
                entries.push({ name, tree_id: entry._tree_id, blob_id: null, ... })
            } else {
                // Loaded → recurse and create new sub-tree object
                const childEntries = await this._buildTreeEntries(entry)
                const childTreeId  = await this._commitManager.createTree(childEntries)
                entries.push({ name, tree_id: childTreeId, blob_id: null, ... })
            }
        }
    }
}
```

Unloaded folders pass through to the new commit with their original `_tree_id` intact. Only folders that were actually loaded (and potentially modified) get new tree objects.

## 4. `getAheadCount()` — Unpushed Commit Count

Walks the commit graph from `headCommitId` back to `namedHeadId`, counting hops.

```javascript
async getAheadCount() {
    if (this._headCommitId === this._namedHeadId) return 0

    let count = 0, cursor = this._headCommitId
    while (cursor && cursor !== this._namedHeadId && count < 100) {
        count++
        const commit = await this._commitManager.loadCommit(cursor)
        cursor = commit.parents[0] || null
    }
    return count
}
```

### HTTP calls

One `GET .../bare/data/{commitId}` per commit in the chain (each fetched from cache if previously loaded). In the common case (1–3 local commits), this is 1–3 requests, all likely cached.

The `max = 100` safety cap prevents runaway traversal on deep chains.

## 5. `getBehindCount()` — Incoming Commit Count

Fetches the current named HEAD ref from the server (live, never cached) and walks backwards:

```javascript
async getBehindCount() {
    const serverNamedHead = await this._refManager.readRef(this._refFileId)
    // GET .../bare/refs/{refFileId} — ALWAYS live, never cached

    if (!serverNamedHead || serverNamedHead === this._namedHeadId) return 0

    let count = 0, cursor = serverNamedHead
    while (cursor && cursor !== this._namedHeadId && count < 100) {
        count++
        const commit = await this._commitManager.loadCommit(cursor)
        cursor = commit.parents[0] || null
    }
    return count
}
```

### HTTP calls

1. `GET .../bare/refs/{refFileId}` — always live (detects remote changes)
2. N × `GET .../bare/data/{commitId}` — one per commit between server HEAD and local known HEAD

If the vault is up to date, only the ref fetch happens (1 request). If behind, commit objects are fetched — these will be new and not in the cache yet.

## 6. The Read-Only Open Pattern

When opened from a share token (no `accessKey` in the vault UI), the behaviour is identical to a writable open with one predictable difference: the clone ref (step 3 in `open()`) always 404s, because the clone ref is created lazily on the first commit. A read-only open never commits, so the clone ref is never created.

This 404 appears in the network tab on every read-only open. It is not an error — `cloneCommitId = null`, `headCommitId = namedCommitId`, vault opens normally.

## 7. Tree Navigation — `_findNode(path)`

The in-memory tree structure:

```javascript
{
  '/': {
    type: 'folder',
    children: {
      'documents': { type: 'folder', children: {}, _tree_id: 'obj-cas-imm-...', _loaded: false },
      'README.md': { type: 'file', blob_id: 'obj-cas-imm-...', size: 1234, content_hash: 'abc...' },
    }
  }
}
```

`_findNode(path)` splits the path on `/`, walks `node.children` at each level:

```javascript
_findNode('/documents/reports') → this._tree['/'].children['documents'].children['reports']
_findNode('/')                  → this._tree['/']
_findNode('/README.md')         → null  (file, not a folder — but _findNode returns the node
                                         regardless; callers check .type)
```

Returns `null` if any segment is missing. Callers must check the returned node's `type` before use.
