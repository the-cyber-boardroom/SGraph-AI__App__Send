## Overview

SGit is a version control system for encrypted content. It shares the same fundamental
design decisions as Git — content-addressed immutable objects, a DAG of commits, named
refs as mutable pointers — but adds a mandatory encryption layer between the client and
the server. Everything the server stores is ciphertext; the server is treated as
untrusted storage.

This document maps SGit's object model and core concepts to their Git equivalents,
highlights where they are identical, where they diverge, and where SGit adds something
Git has no equivalent for.

---

## 1. The Object Model

### Git

Git has four object types, all content-addressed by SHA-1 (or SHA-256 in newer Git):

```
  blob     — file content
  tree     — directory listing (name + mode + object hash per entry)
  commit   — snapshot pointer (tree hash + parent hashes + author + message)
  tag      — annotated tag (points to any object)
```

All objects live in `.git/objects/`. The object ID is `SHA-1(type + " " + size + "\0" + content)`.

```
  .git/objects/
    ab/cdef1234...    ← blob: file content
    12/3456abcd...    ← tree: directory listing
    fe/dcba9876...    ← commit: snapshot
```

Objects are stored as zlib-compressed plaintext. Anyone with filesystem access can read any object.

### SGit

SGit has the same four logical types, but all are stored as AES-256-GCM ciphertext:

```
  blob (obj-cas-imm-*)     — encrypted file content
  tree (obj-cas-imm-*)     — encrypted directory listing (tree_v1 schema)
  commit (obj-cas-imm-*)   — encrypted snapshot pointer (commit_v2 schema)
  [no tag type currently]  — annotated tags not yet implemented
```

All objects share the same ID format: `obj-cas-imm-{SHA-256(ciphertext)[:12 hex]}`. The object type is not encoded in the ID — it is determined by context (a commit's `tree_id` field points to a tree; a tree entry's `blob_id` points to a blob).

```
  server: bare/data/
    obj-cas-imm-a2ad790bd3f1    ← could be blob, tree, or commit
    obj-cas-imm-f81c3b9e0042    ← type determined by who references it
    obj-cas-imm-c3d4e5f60718    ← always encrypted ciphertext
```

### Side-by-side

```
  Property              Git                        SGit
  ─────────────────── ─────────────────────────── ──────────────────────────────
  Object types        blob, tree, commit, tag     blob, tree, commit (no tag yet)
  Content addressing  SHA-1(type+size+content)    SHA-256(ciphertext)
  Hash length         40 hex (SHA-1)              12 hex (truncated SHA-256)
  Storage format      zlib(plaintext)             AES-256-GCM(plaintext)
  Server sees         plaintext (compressed)      ciphertext only
  Type in ID          yes (via header prefix)     no (determined by context)
  Object location     .git/objects/{2}/{38}       bare/data/obj-cas-imm-{12}
  Deduplication       yes (same content = same ID)  yes (same plaintext+key = different ciphertext = different ID*)
```

*This is a key difference: in Git, two identical files produce one object. In SGit, the same file encrypted twice produces two different ciphertexts (different IVs) and therefore two different blob IDs. History is preserved, but byte-level deduplication across commits does not happen automatically.

## 2. The Tree Object

### Git tree entry

```
  mode filename\0{20-byte raw SHA-1}
  e.g.: 100644 README.md\0{raw bytes of blob SHA-1}
        040000 documents\0{raw bytes of sub-tree SHA-1}
```

Filename is plaintext. Mode encodes file type (regular, executable, symlink, submodule). All metadata is in the clear.

### SGit tree entry (`tree_v1`)

```json
{
  "name_enc":         "base64([IV][AES-GCM(filename)][tag])",
  "blob_id":          "obj-cas-imm-{hex12}",
  "tree_id":          null,
  "size_enc":         "base64([IV][AES-GCM(size_string)][tag])",
  "content_hash_enc": "base64([IV][AES-GCM(sha256_hex[:12])][tag])"
}
```

Every metadata field is individually AES-GCM encrypted. `blob_id` and `tree_id` are stored in plaintext — they are SHA-256 hashes of encrypted bytes, not semantic data.

```
  Property              Git tree entry             SGit tree entry
  ─────────────────── ─────────────────────────── ──────────────────────────────
  Filename            plaintext                   AES-GCM encrypted
  File size           not stored in tree          AES-GCM encrypted
  Content hash        implicit (is the blob ID)   SHA-256(plaintext)[:12], encrypted
  File mode           plaintext (100644, 040000)  not stored (web UI only)
  Sub-tree pointer    raw SHA-1 bytes             plaintext obj-cas-imm-* string
  Blob pointer        raw SHA-1 bytes             plaintext obj-cas-imm-* string
  Entire tree object  zlib(plaintext)             AES-GCM(JSON of encrypted entries)
```

The double-encryption of tree entries (fields encrypted individually, then entire tree JSON encrypted again) means the server cannot determine how many files are in a directory, what they are named, or how large they are — even with full access to the encrypted tree object.

## 3. The Commit Object

### Git commit

```
  tree {tree-sha1}
  parent {parent-sha1}
  author {name} <{email}> {timestamp} {tz}
  committer {name} <{email}> {timestamp} {tz}

  {commit message}
```

All fields are plaintext. Author identity, timestamp, and message are readable by anyone with repository access.

### SGit commit (`commit_v2`)

```json
{
  "schema":       "commit_v2",
  "parents":      ["obj-cas-imm-{parent_hex}"],
  "tree_id":      "obj-cas-imm-{root_tree_hex}",
  "timestamp_ms": 1744754400000,
  "message_enc":  "base64([IV][AES-GCM(commit_message)][tag])",
  "branch_id":    null,
  "signature":    null
}
```

The commit message is double-encrypted: once as `message_enc` inside the JSON, and again when the entire commit JSON is encrypted as the outer blob. The timestamp is stored in plaintext (milliseconds since epoch) — this is visible to the server.

```
  Property              Git commit                 SGit commit
  ─────────────────── ─────────────────────────── ──────────────────────────────
  Tree pointer        plaintext SHA-1             plaintext obj-cas-imm-*
  Parent pointers     plaintext SHA-1 (1 per line) array of obj-cas-imm-* strings
  Author identity     plaintext name + email      not stored
  Committer identity  plaintext name + email      not stored
  Timestamp           plaintext                   plaintext (ms since epoch)
  Commit message      plaintext                   double-encrypted
  Branch reference    not in object               branch_id field (null in web UI)
  Signature           GPG signature field         signature field (null currently)
  Outer encoding      zlib(plaintext)             AES-GCM(JSON)
```

The absence of author identity in SGit commits is intentional: the server cannot correlate commits to users. The `write_key` header authenticates writes but is never stored in the commit object itself.

## 4. Refs

### Git refs

```
  .git/refs/heads/main          ← branch ref (plaintext SHA-1)
  .git/refs/remotes/origin/main ← remote-tracking ref
  .git/HEAD                     ← symbolic ref ("ref: refs/heads/main")
  .git/packed-refs              ← packed ref storage
```

Ref files contain a plaintext 40-hex SHA-1 string. `HEAD` is a symbolic ref pointing to a branch name.

### SGit refs

```
  bare/refs/ref-pid-muw-{hmac12}   ← named HEAD (shared/published state)
  bare/refs/ref-pid-snw-{hmac12}   ← clone HEAD (local working state)
  bare/idx/idx-pid-muw-{hmac12}    ← branch index (multi-branch vaults)
```

Each ref file contains an AES-256-GCM encrypted JSON payload: `{ "commit_id": "obj-cas-imm-..." }`. The ref file IDs are derived via HMAC from the vault's `hmac_key` — they are deterministic but not guessable without the key.

```
  Property              Git refs                   SGit refs
  ─────────────────── ─────────────────────────── ──────────────────────────────
  Storage format      plaintext SHA-1 string       AES-GCM({ commit_id })
  Ref file location   .git/refs/...               bare/refs/{derived-id}
  File ID             human-readable branch name   HMAC-derived, opaque to server
  Symbolic refs       yes (HEAD → branch)          no (web UI always resolves)
  Remote tracking     yes (.git/refs/remotes/)     no (two local refs instead)
  Packed refs         yes (.git/packed-refs)       no
  Mutability          mutable (overwritten on push) mutable (same — PUT overwrites)
  Server can read     yes                          no (encrypted)
```

The two-ref model (named + clone) is SGit's most significant structural departure from Git. Git uses a single ref per branch and tracks the remote state separately in `refs/remotes/`. SGit uses two local refs — one for the published state and one for the working state — with no concept of a "remote" as a distinct named entity.

## 5. The Ref Naming Scheme

Git branch names are human-readable strings chosen by the user (`main`, `feature/login`). SGit ref IDs are deterministic HMAC-derived hex strings that are opaque to anyone without the vault key.

```
  Git:
    refs/heads/main         → "abc123def456..."  (SHA-1 of HEAD commit)
    refs/remotes/origin/main → "abc123def456..."

  SGit:
    ref-pid-muw-{hmac12}   → AES-GCM({ "commit_id": "obj-cas-imm-..." })
      ╰──────────────────╯
      derived: HMAC-SHA-256(hmac_key, "sg-vault-v1:file-id:ref:{vault_id}")[:12]
      server sees only: the hex string as a path component
```

The server stores ref content at `bare/refs/{refId}`. It knows a ref exists at that path but cannot read its content or determine which commit it points to.

## 6. Key Derivation — Git Has None

This is the most fundamental difference. Git has no cryptographic key layer at all.

```
  Git:
    No keys. All content is accessible to anyone with repository access.
    Authentication is handled externally (SSH keys, HTTPS credentials)
    but these authenticate the transport, not the content.

  SGit:
    Two keys derived per vault via PBKDF2 (600,000 iterations, SHA-256):
      read_key   — AES-256-GCM CryptoKey, never sent to server
      write_key  — 256-bit hex string, sent as HTTP header for auth
    Plus HMAC key for deterministic file ID derivation.
    Keys never leave the client. Server authenticates writes via write_key
    but cannot decrypt content with it.
```

The separation of `read_key` and `write_key` (different PBKDF2 salts → cryptographically independent) enables a capability that Git cannot express: **read-only sharing without write access**. A sender can share the vault key (read + write) or a share token (read only, derived separately), and the server enforces the distinction without knowing anything about the content.

## 7. Content Addressing: Identical Concept, Different Hash

Both Git and SGit use content addressing — the object ID is derived from the object content.

```
  Git:    ID = SHA-1(type + " " + size + "\0" + plaintext_content)
  SGit:   ID = SHA-256(ciphertext)[:12]
              where ciphertext = [IV][AES-GCM(plaintext, read_key)][tag]
```

The consequence: in Git, the same file content always produces the same blob ID (deduplication). In SGit, the same plaintext encrypted twice produces different ciphertexts (different random IVs) and therefore different IDs. There is no cross-commit blob deduplication in SGit.

This is a deliberate trade-off: deduplication would require deterministic encryption (same plaintext → same ciphertext), which would weaken the encryption (known-plaintext attacks become easier). The security model favours fresh random IVs over storage efficiency.

The `content_hash` field in tree entries (`SHA-256(plaintext)[:12]`) provides deduplication detection at the application level — two entries with the same `content_hash` have the same plaintext — but the server cannot see this field.

## 8. Storage: Remote vs Always-Remote

```
  Git working model:
    1. git clone → full local copy in .git/
    2. Work entirely locally (commit, branch, diff, log — all offline)
    3. git push → send new objects to remote
    4. git pull → fetch new objects from remote

  SGit/web working model:
    1. open() → fetch metadata only (6 objects, ~1 KB)
    2. Work against server (each commit PUTs objects to server)
    3. push() → advance named ref on server (1 PUT)
    4. pull() → fetch new objects from server, rebuild tree

  SGit/CLI working model (sgit clone):
    1. sgit clone → full local copy (same as git clone)
    2–4. Same as git
```

The web vault model has no local working copy on disk. Every `_commit()` writes directly to the server. The Cache API is not a working copy — it stores encrypted ciphertext and is purely a performance layer. The authoritative state is always on the server.

## 10. What SGit Has That Git Does Not

```
  SGit capability                           Git equivalent
  ──────────────────────────────────────── ──────────────────────────────────
  Server-side zero-knowledge encryption    None. Server sees all content.
  Read-only sharing (cryptographic)        None. Access is all-or-nothing
                                           at the transport layer.
  Content-addressed immutable cache        None built-in. External CDN needed.
  Dynamic access without cloning           Partial clone / sparse checkout
                                           (similar goal, different design)
  Deterministic ref IDs (HMAC-derived)     Human-readable branch names only
  Write authorisation via derived key      External auth (SSH/HTTPS credentials)
  Two-branch local/published model         Single branch + remote tracking refs
  Browser-native (Web Crypto API)          CLI only (no browser implementation)
```
