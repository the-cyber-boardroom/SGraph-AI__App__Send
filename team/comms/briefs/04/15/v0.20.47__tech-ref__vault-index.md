## Series A: Vault Protocol (5 parts)

The complete wire-level protocol. Start here if you are building a CLI client, implementing
vault support in a new language, or need to understand exactly what bytes go over the wire
and why.

### [Part 1 — Crypto & Object Model](v0.20.47__tech-ref__vault-protocol-1-crypto-and-object-model.md)

The foundation everything else builds on.

- **AES-256-GCM wire format** — `[12-byte IV][ciphertext][16-byte tag]`, fresh IV per call
- **Key derivation — standard vault key** (`passphrase:vault_id`) — two parallel PBKDF2 runs with different salts producing independent `read_key` and `write_key`
- **Key derivation — simple token** (`word-word-NNNN`) — PBKDF2 fixed salt → HKDF × 2 + SHA-256 for vault_id
- **4-segment ID naming scheme** — `{type}-{derivation}-{mutability}-{hex12}` — how to read any ID and know its caching rules, mutability, and derivation method at a glance
- **Object store** — content-addressed block storage, `store()` and `load()` with Cache API integration
- **All object schemas** — `commit_v2`, `tree_v1`, ref payload, branch index, `.vault-settings`
- **What the server knows** — and crucially, what it does not (file names, sizes, structure, content: nothing)

---

### [Part 2 — Read Protocol](v0.20.47__tech-ref__vault-protocol-2-read-protocol.md)

Every read operation, from `open()` to `getFile()` to ahead/behind counting.

- **`open()` step-by-step** — 6 HTTP calls: branch index, named ref, clone ref, commit, root tree, settings blob. Table of expected results including the two expected 404s.
- **`getFile()`** — standard path (≤4 MB, Cache API) and large file path (>4 MB, presigned S3 URL with Lambda bypass)
- **`loadSubTreeOnDemand()`** — lazy folder expansion (1 GET per folder), the unloaded-folder invariant during `_buildTreeEntries`
- **`getAheadCount()`** — commit graph walk from clone HEAD back to named HEAD, max-100 safety cap
- **`getBehindCount()`** — live named ref fetch (detection step) + commit graph walk
- **Read-only open pattern** — why the clone ref always 404s on read-only opens and why that is correct
- **`_findNode(path)`** — in-memory tree navigation

### [Part 3 — init and commit](v0.20.47__tech-ref__vault-protocol-3-init-and-commit.md)

How a vault is created from scratch and how every local change is persisted.

- **`create()` — 6-step init sequence** — vault ID generation (8 random bytes → 36-char alphabet), key derivation, settings object, empty tree, first commit, named ref write. 5 total PUT requests.
- **`_commit()` internals** — the core write operation called by every mutation:
  - `_buildTreeEntries()` — recursive tree serialisation with the critical unloaded-stub pass-through
  - Settings blob — re-serialised and stored on every commit
  - `createTree()` — field-by-field per-entry encryption then outer AES-GCM on the whole tree JSON
  - `createCommit()` — double-encrypted commit message (once as `message_enc`, again as the outer blob)
  - `writeRef()` — advancing the clone ref only (named ref never moves here)
- **HTTP call count table** — N loaded folders → N+5 PUTs per commit
- **Encryption count per commit** — every AES-GCM call enumerated
- **Object immutability and history preservation** — why old blobs are never deleted and what that enables

---

### [Part 4 — File and Folder Mutations](v0.20.47__tech-ref__vault-protocol-4-file-and-folder-mutations.md)

Every operation that changes vault content, with step-by-step code and HTTP call counts.

- **`addFile()`** — encrypt → store blob → update tree → commit. HTTP calls for root vs sub-folder.
- **`updateFile()`** — new blob, old blob preserved on server (history). Identical HTTP pattern to addFile.
- **`removeFile()`** — in-memory delete only, no server-side blob deletion. Old blob stays for history.
- **`renameFile()`** — same blob_id preserved, only name_enc changes in new tree object. Cheapest file op.
- **`moveFile()`** — both source and destination folders must be loaded; blob_id unchanged.
- **`createFolder()`** — empty `{ type: 'folder', children: {} }` produces empty tree object on server.
- **`removeFolder()`** — works on unloaded stubs (no fetch needed); entire sub-tree removed in one commit.
- **`renameFolder()`** — works on unloaded stubs; `_tree_id` carried to new key, O(1) regardless of folder size.
- **`moveFolder()`** — works on unloaded stubs; both parents get new tree objects.
- **Mutation summary table** — all 9 operations: blob write?, server deletes?, HTTP call count, works on stub?
- **Content hash** — `SHA-256(plaintext)[:12]` vs blob ID `SHA-256(ciphertext)[:12]` — same algorithm, different inputs, different purposes
- **`listFolder()` and `getStats()`** — non-mutating helpers, zero network

### [Part 5 — push and pull](v0.20.47__tech-ref__vault-protocol-5-push-and-pull.md)

The two-branch synchronisation model.

- **Two-branch model** — named ref (`ref-pid-muw-*`) vs clone ref (`ref-pid-snw-*`), what advances each, what the gap between them means
- **`push()`** — 1 PUT only (named ref → clone HEAD). All objects already exist. Pre-conditions (writable, clone ref initialised).
- **`pull()`** — pre-condition (no pending local commits), live named ref fetch, clone ref advance, `_loadTreeFromCommit()` tree rebuild. HTTP call table.
- **`getAheadCount()` and `getBehindCount()`** — implementation detail and HTTP costs
- **Conflict model** — fast-forward only, no merge. Three scenarios: clean fast-forward, diverged (silent overwrite risk), local commits blocking pull.
- **Safe multi-client pattern** — `getBehindCount()` before push, `aheadCount == 0` before pull
- **Full two-client lifecycle example** — session A commits and pushes, session B opens and commits and pushes, session A detects and pulls

## Series B: Dynamic Access (3 parts)

The on-demand access model that makes large vaults fast without cloning. Start here if you
are building a TUI vault browser, implementing `sgit cat`, or want to understand why the
web vault loads a 251 MB vault in 820 ms.

### [Part 1 — Concept and Architecture](v0.20.47__tech-ref__vault-dynamic-access-1-concept.md)

The high-level picture: what dynamic access is and why the Cache API makes it as fast as a local clone.

- **Full clone vs dynamic access** — ASCII diagrams of both models, network cost comparison
- **The game-changer** — dynamic access alone trades clone latency for per-file latency. Cache API eliminates the per-file latency after first access. Result: 251 MB vault in 820 ms.
- **Three levels of on-demand loading** — tree structure (folder expand), file content (blob fetch), refs/commits (navigation) — each independent, each separately cacheable
- **Full system architecture diagram** — browser ↔ Cache API ↔ Server with all component names
- **Dynamic access vs `sgit clone` comparison table** — 9 properties compared
- **Why this matters for the CLI** — `sgit cat docs/report.pdf` should cost 6 GETs regardless of vault size, not clone everything first

### [Part 3 — Cache Layer & CLI Implications](v0.20.47__tech-ref__vault-dynamic-access-3-cache-layer-and-cli.md)

The cache implementation in full, plus a complete specification for CLI implementors.

- **Cache API design** — cache name `sg-vault-blocks`, key format `https://sgvault/{vaultId}/bare/data/{objectId}`, value is raw encrypted ciphertext
- **The `-imm-` guard** — why only `obj-cas-imm-*` blocks are cached, how `muw`/`snw` refs are excluded at two levels
- **`data.slice(0)` semantics** — why the ArrayBuffer is copied before being stored, what breaks without the copy (transferred ArrayBuffers in Web Workers)
- **Graceful degradation** — 5 scenarios where Cache API is unavailable (incognito, quota exceeded, no service worker, etc.) — all silently fall back to network
- **Performance measurements** — cold 6 s → second load 1 s → warm cache 820 ms, with explanation of the 820 ms floor (3 live ref GETs + decrypt CPU + UI render)
- **CLI cache spec** — filesystem layout (`~/.sgit/cache/blocks/{vaultId}/{objectId}`), same `-imm-` guard, same raw ciphertext storage
- **`sgit cat` specification** — implementation flow with network call count (6 GETs for a 3-level-deep file regardless of vault size)
- **`sgit ls` specification** — non-recursive and recursive variants, zero blob fetches
- **`sgit warm` specification** — explicit pre-warming for offline use
- **Complete on-demand protocol spec** — data structures, invariants (6 rules), and all operations in pseudocode, ready for CLI implementation
