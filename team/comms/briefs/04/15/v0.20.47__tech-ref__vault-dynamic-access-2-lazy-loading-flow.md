## 1. State Machine: A Folder Node's Lifecycle

Every folder in the in-memory tree is in one of three states:

```
  ┌─────────────────────────────────────────────────────────────┐
  │                  Folder Node State Machine                  │
  │                                                             │
  │   open() or pull()                                          │
  │        │                                                    │
  │        ▼                                                    │
  │  ┌───────────┐    loadSubTreeOnDemand()    ┌─────────────┐  │
  │  │  STUB     │ ──────────────────────────► │   LOADED    │  │
  │  │           │    (1 GET, decrypt, parse)  │             │  │
  │  │ _loaded:  │                             │ _loaded:    │  │
  │  │  false    │                             │  true       │  │
  │  │ _tree_id: │                             │ _tree_id:   │  │
  │  │  'obj-...'│                             │  (deleted)  │  │
  │  │ children: │                             │ children:   │  │
  │  │  {}       │                             │  {populated}│  │
  │  └───────────┘                             └──────┬──────┘  │
  │        ▲                                          │         │
  │        │         open() or pull()                │         │
  │        │         (vault reloaded)                │         │
  │        └──────────────────────────────────────── ┘         │
  │                                                             │
  │  Note: root '/' is always LOADED — it is materialised       │
  │  directly from the root tree object during open()           │
  └─────────────────────────────────────────────────────────────┘
```

The root folder (`/`) is always loaded immediately. All sub-folders start as stubs and transition to loaded only when the user (or code) expands them. A vault reload (`open()` or `pull()`) resets all sub-folders back to stubs.

## 2. `open()` — What Gets Loaded

```
  open(sgSend, fullVaultKey)
  │
  ├─ deriveKeys()                    (no network)
  │
  ├─ GET bare/idx/{branchIndexId}    (optional — may 404)
  │    └─ override refFileId if branch index present
  │
  ├─ GET bare/refs/{refFileId}       (named HEAD — required)
  │    └─ namedCommitId
  │
  ├─ GET bare/refs/{cloneRefFileId}  (clone HEAD — may 404)
  │    └─ cloneCommitId (null if 404)
  │
  │   headCommitId = cloneCommitId || namedCommitId
  │
  ├─ GET bare/data/{headCommitId}    (commit object)
  │    └─ { tree_id, parents, message_enc, ... }
  │
  ├─ GET bare/data/{commit.tree_id}  (root tree object)
  │    └─ { entries: [ {name_enc, blob_id|tree_id, ...}, ... ] }
  │       decrypt each entry field individually
  │
  └─ for each entry in root tree:
       if entry.name == '.vault-settings'
         └─ GET bare/data/{settingsBlobId}
              └─ decrypt → _settings
       elif entry.tree_id
         └─ _tree['/'].children[name] = STUB { _loaded:false, _tree_id }
       else
         └─ _tree['/'].children[name] = FILE { blob_id, size, content_hash }

  Result: root '/' is LOADED, all sub-folders are STUBs
```

A vault with 155 files in 20 folders fetches exactly **6 objects** on open — 3 refs + 1 commit + 1 tree + 1 settings blob. The 154 other file blobs and 19 sub-tree objects are never touched.

## 3. `loadSubTreeOnDemand()` — Expanding a Folder

```
  User expands /documents/ in the UI
  │
  ├─ needsLoading('/documents')
  │    └─ node._loaded === false && !!node._tree_id  →  true
  │
  └─ loadSubTreeOnDemand('/documents')
       │
       ├─ GET bare/data/{node._tree_id}   (sub-tree object)
       │    └─ { entries: [ ... ] }
       │       decrypt each entry field
       │
       ├─ for each entry:
       │    if entry.tree_id
       │      └─ node.children[name] = STUB { _loaded:false, _tree_id }
       │    else
       │      └─ node.children[name] = FILE { blob_id, size, content_hash }
       │
       ├─ node._loaded = true
       ├─ delete node._tree_id
       └─ return true  (UI re-renders folder contents)
```

One GET per folder expansion, regardless of how deep the tree is or how many files the folder contains. Sub-folders within the expanded folder become stubs — they are not recursively fetched.

### Sequence diagram: expanding two nested folders

```
  UI          SGVault          ObjectStore          Server
  │                │                │                  │
  │ expand /docs   │                │                  │
  │ ─────────────► │                │                  │
  │                │ load(tree_id1) │                  │
  │                │ ─────────────► │  GET /data/tree1 │
  │                │                │ ────────────────►│
  │                │                │ ◄────────────────│
  │                │ ◄────────────  │  [IV][cipher][tag]
  │                │ decrypt, parse │                  │
  │                │ /docs: LOADED  │                  │
  │                │ /docs/archive: │                  │
  │                │   STUB         │                  │
  │ ◄─────────────  │                │                  │
  │ render /docs   │                │                  │
  │ contents       │                │                  │
  │                │                │                  │
  │ expand /docs/  │                │                  │
  │   archive      │                │                  │
  │ ─────────────► │                │                  │
  │                │ load(tree_id2) │                  │
  │                │ ─────────────► │  GET /data/tree2 │
  │                │                │ ────────────────►│
  │                │                │ ◄────────────────│
  │                │ ◄────────────  │                  │
  │                │ decrypt, parse │                  │
  │                │ /docs/archive: │                  │
  │                │   LOADED       │                  │
  │ ◄─────────────  │                │                  │
  │ render archive │                │                  │
  │ contents       │                │                  │
```

Each folder expansion is exactly **one network round-trip**. The user only pays for what they navigate into.

## 4. `getFile()` — Reading File Content

File metadata (name, size, content_hash) is available after `open()` or `loadSubTreeOnDemand()` — the blob content is not. Fetching a file's bytes is a separate on-demand step:

```
  getFile('/documents', 'report.pdf')
  │
  ├─ _findNode('/documents')  →  folder node (must be LOADED)
  ├─ folder.children['report.pdf']  →  { blob_id, size, ... }
  │
  ├─ size > 4 MB?
  │    YES: loadLarge(blob_id)
  │    │     └─ GET /api/vault/presigned/read-url/{vaultId}/bare/data/{blob_id}
  │    │          → { url: "https://s3.amazonaws.com/...?X-Amz-Expires=900" }
  │    │          → GET {presigned_url}  (direct from S3, bypasses Lambda 6 MB limit)
  │    NO: load(blob_id)
  │         └─ _cacheGet(blob_id) → HIT: return cached ArrayBuffer
  │              or
  │            _cacheGet(blob_id) → MISS:
  │              GET /api/vault/read/{vaultId}/bare/data/{blob_id}
  │              _cachePut(blob_id, data)
  │              return data
  │
  └─ SGSendCrypto.decrypt(encrypted, readKey)
       └─ iv = bytes[0:12]
          ciphertext = bytes[12:]
          crypto.subtle.decrypt('AES-GCM', readKey, ciphertext, iv)
          → plaintext ArrayBuffer
```

### Sequence diagram: getFile cold vs warm

```
  COLD (first access):

  Caller       SGVault      ObjectStore    Cache API       Server
    │              │              │             │              │
    │ getFile()    │              │             │              │
    │ ────────────►│              │             │              │
    │              │ load(blobId) │             │              │
    │              │ ────────────►│             │              │
    │              │              │ cacheGet()  │              │
    │              │              │ ────────────►  MISS        │
    │              │              │             │              │
    │              │              │ GET /data/  │              │
    │              │              │ {blobId}    │  ───────────►│
    │              │              │             │  ◄───────────│
    │              │              │ cachePut()  │              │
    │              │              │ ────────────►              │
    │              │ ◄────────────│  stored     │              │
    │              │ decrypt()    │             │              │
    │ ◄────────────│              │             │              │
    │ plaintext    │              │             │              │


  WARM (second access, same session or future session):

  Caller       SGVault      ObjectStore    Cache API       Server
    │              │              │             │              │
    │ getFile()    │              │             │              │
    │ ────────────►│              │             │              │

    │              │              │ cacheGet()  │              │
```

## 5. `_buildTreeEntries()` — The Commit Invariant

When a mutation triggers `_commit()`, the tree serialiser must not lose data for folders the user never opened. The unloaded-stub check is the critical safety gate:

```
  _buildTreeEntries(node)
  │
  └─ for each [name, entry] in node.children:
       │
       ├─ entry.type == 'folder'?
       │    │
       │    ├─ entry._loaded === false && entry._tree_id?
       │    │    └─ STUB: reuse entry._tree_id unchanged
       │    │         entries.push({ name, tree_id: entry._tree_id, ... })
       │    │         ← NO network call, NO new tree object
       │    │
       │    └─ else (LOADED):
       │         └─ recurse: _buildTreeEntries(entry)
       │              → createTree(childEntries)
       │                   → PUT /bare/data/{childTreeId}
       │              entries.push({ name, tree_id: childTreeId, ... })
       │
       └─ entry.type == 'file':
            entries.push({ name, blob_id, size, content_hash, ... })
```

### Why this matters

```
  Before commit (user added README.md, never opened /archive/):

  in-memory tree:
    / (LOADED)
    ├── README.md  { blob_id: 'obj-cas-imm-new' }        ← changed
    └── archive/   { _loaded: false, _tree_id: 'obj-cas-imm-old-tree' }  ← stub

  _buildTreeEntries('/'):
    → README.md entry: use new blob_id
    → archive/ entry: reuse _tree_id 'obj-cas-imm-old-tree'  ← PRESERVED

  New root tree object:
    entries: [
      { name_enc: AES('README.md'), blob_id: 'obj-cas-imm-new', ... },
      { name_enc: AES('archive'),   tree_id: 'obj-cas-imm-old-tree', ... }
    ]

  The archive's contents are untouched on the server — only the pointer is carried forward.
```

Without this invariant, committing any change to root while sub-folders are stubs would silently destroy their contents. The stub's `_tree_id` acts as a frozen reference through any number of commits.

## 6. Navigating a Deep Vault: Cumulative Network Cost

A vault with this structure:

```
  /
  ├── README.md
  ├── documents/
  │   ├── reports/
  │   │   ├── q1.pdf
  │   │   └── q2.pdf
  │   └── contracts/
  │       └── agreement.pdf
  └── images/
      ├── logo.png
      └── banner.jpg
```

Network cost to reach each node from cold start:

```
  Action                        GETs    Cumulative GETs
  ─────────────────────────── ────── ──────────────────
  open()                          6               6
  expand /documents/              1               7
  expand /documents/reports/      1               8
  open q1.pdf                     1               9   (+ store in cache)
  open q2.pdf                     1              10   (+ store in cache)
  expand /documents/contracts/    1              11
  open agreement.pdf              1              12   (+ store in cache)
  expand /images/                 1              13
  open logo.png                   1              14   (+ store in cache)
  open banner.jpg                 1              15   (+ store in cache)

  Total to access all 5 files in 3 folders: 15 GETs

  If vault had 1000 files in 50 folders but user only accessed these 5:
    Still 15 GETs — the other 995 files cost nothing.
```

Compare with `sgit clone`:
```
  clone a vault with 1000 files, 50 folders:
    3 ref GETs + 50 tree GETs + 1000 blob GETs + commit GETs
    Total: ~1053+ GETs, proportional to vault size
    User accessing 5 files still pays for the other 995
```

## 7. `needsLoading()` — The UI Gate

```javascript
needsLoading(folderPath) {
    const node = this._findNode(folderPath)
    return node && node.type === 'folder' && node._loaded === false && !!node._tree_id
}
```

Called by the UI before rendering a folder's contents. Returns `true` only if:
1. The node exists
2. It is a folder (not a file)
3. It has never been loaded (`_loaded === false`)
4. It has a server-side tree object (`_tree_id` is set)

Condition 4 is important: a freshly created folder (`createFolder()`) has `_loaded` undefined (not `false`) and no `_tree_id`. It is an in-memory-only folder that has never been committed. `needsLoading()` returns `false` for it — the UI renders it as already available.

```
  Folder states and needsLoading():

  { type:'folder', _loaded:false, _tree_id:'obj-...' }  → true  (server stub)
  { type:'folder', _loaded:true,  children:{...}      }  → false (loaded)
  { type:'folder', children:{}                         }  → false (new, uncommitted)
```

## 8. Full Flow: Open → Browse → Edit → Commit

```
  Time ──────────────────────────────────────────────────────────────►

  open()             expand /docs     getFile(report)    addFile(new.txt)
    │                     │                │                  │
    ▼                     ▼                ▼                  ▼
  [GET ×6]           [GET ×1]          [GET ×1 or       [PUT ×1 blob]
  root tree          docs/ tree        cache HIT]       [PUT ×N trees]
  all sub-           docs: LOADED      blob decrypted   [PUT ×1 commit]
  folders = STUB     sub-sub = STUBS                    [PUT ×1 ref]

  in-memory tree:     in-memory tree:   in-memory tree:  in-memory tree:
  / (LOADED)          / (LOADED)        / (LOADED)       / (LOADED)
  ├ docs/ (STUB)  →  ├ docs/ (LOADED)  ├ docs/ (LOADED) ├ docs/ (LOADED)
  └ imgs/ (STUB)      │ ├ report        │ ├ report        │ ├ report
                      │ └ archive(STUB) │ └ archive(STUB) │ └ archive(STUB)
                      └ imgs/ (STUB)    └ imgs/ (STUB)    ├ imgs/ (STUB)
                                                          └ new.txt (FILE)

  Network total:      +1 GET           +1 GET (cold)    +4 PUTs
  6 GETs total        7 GETs total     8 GETs total     cloneRef advanced
```

Note: `archive/` is still a STUB throughout. Its contents were never fetched, never re-serialised. The commit correctly preserves its `_tree_id` through `_buildTreeEntries()`.
