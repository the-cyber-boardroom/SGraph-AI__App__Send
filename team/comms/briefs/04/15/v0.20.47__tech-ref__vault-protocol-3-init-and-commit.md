## Overview

Every write operation in the vault follows the same pattern:

```
1. Mutate in-memory tree (or derive new content)
2. Encrypt content → store as obj-cas-imm-* blob(s)
3. _commit():
     a. _buildTreeEntries() → createTree() per folder level
     b. createCommit() → store commit object
     c. writeRef(cloneRefFileId, commitId) → advance clone branch
```

The named branch ref (`ref-pid-muw-*`) is **never** touched by `_commit()`. It only moves during `push()`. This is the core of the two-branch model: the clone ref tracks local working state; the named ref tracks the published/shared state.

## 1. `create(sgSend, passphrase, options)` — Initialising a New Vault

`SGVault.create()` is the web equivalent of `sgit init && sgit push`. It creates a vault from scratch, writes the first commit, and establishes both the named and clone refs so they start in sync (ahead count = 0).

### Step-by-step

**Step 1: Generate vault ID** (no network)

```javascript
const alphabet = 'abcdefghijklmnopqrstuvwxyz0123456789'
const bytes    = crypto.getRandomValues(new Uint8Array(8))
vault._vaultId = Array.from(bytes).map(b => alphabet[b % alphabet.length]).join('')
// e.g. "a1b2c3d4"
```

8 random bytes, each mapped to a character from a 36-char alphabet (a–z, 0–9). The vault ID is not secret — it's part of every server path and is included in the vault key shown to the user.

**Step 2: Derive all keys** (no network, ~600 ms)

```javascript
const keys = await SGVaultCrypto.deriveKeys(passphrase, vault._vaultId)
// → read_key, write_key, hmac_key, refFileId, branchIndexFileId

vault._cloneRefFileId = await SGVaultCrypto.deriveBranchRefFileId(
    keys.hmacKey, vault._vaultId, 'web-ui'
)
// → ref-pid-snw-{hmac[:12]}
```

See Part 1 §2 for full derivation detail.

**Step 3: Create settings object** (no network)

```javascript
vault._settings = {
    vault_name:  options.name || 'Untitled Vault',
    vault_id:    vault._vaultId,
    created:     new Date().toISOString(),
    version:     3,
    description: options.description || ''
}
```

Settings are not stored as a separate object on the server. They live as a special `.vault-settings` entry in every root tree (see `_commit()` §2c below).

**Step 4: Create empty in-memory tree** (no network)

```javascript
vault._tree = { '/': { type: 'folder', children: {} } }
```

**Step 5: First commit** (multiple network calls — see `_commit()` below)

```javascript
await vault._commit('Initial vault creation')
// → writes clone ref: ref-pid-snw-* → headCommitId
```

**Step 6: Write named ref to same commit** (1 PUT)

```javascript
await vault._refManager.writeRef(vault._refFileId, vault._headCommitId)
vault._namedHeadId = vault._headCommitId
```

```
PUT /api/vault/write/{vaultId}/bare/refs/{refFileId}
Headers: x-sgraph-vault-write-key: {write_key}
Body: [IV][encrypted {"commit_id": "obj-cas-imm-..."}][tag]
```

After `create()`, both `refFileId` (named) and `cloneRefFileId` (clone) point to the same initial commit. The ahead count is 0. The vault is immediately usable and shareable.

### Total HTTP calls for `create()`

| # | Request | What it writes |
|---|---|---|
| 1 | `PUT .../bare/data/{settingsBlobId}` | Encrypted settings blob |
| 2 | `PUT .../bare/data/{rootTreeId}` | Encrypted root tree object (empty + .vault-settings entry) |
| 3 | `PUT .../bare/data/{commitId}` | Encrypted initial commit object |
| 4 | `PUT .../bare/refs/{cloneRefFileId}` | Clone ref → commitId |
| 5 | `PUT .../bare/refs/{refFileId}` | Named ref → commitId (step 6) |

5 PUT requests. No GET requests. The vault exists on the server after request 5 completes.

## 2. `_commit(message)` — The Core Write Operation

Every mutation (addFile, updateFile, removeFile, renameFile, createFolder, etc.) ends with `await this._commit(message)`. This method serialises the entire in-memory tree into encrypted objects and advances the clone ref.

### Step 2a: `_buildTreeEntries(node)` — Serialise in-memory tree

Recursively converts the in-memory tree into a flat list of entries per folder level. For each child of a folder node:

**File entry:**
```javascript
entries.push({
    name:         fileName,       // plaintext — will be encrypted in createTree()
    size:         entry.size,
    content_hash: entry.content_hash,
    blob_id:      entry.blob_id,  // obj-cas-imm-* (already stored on server)
    tree_id:      null
})
```

**Unloaded folder (lazy stub — _loaded === false):**
```javascript
// Reuse existing tree_id — DO NOT recurse or create a new tree object
entries.push({
    name:         folderName,
    size:         0,
    content_hash: null,
    blob_id:      null,
    tree_id:      entry._tree_id  // preserved unchanged from open()
})
```

This is critical: an unloaded folder's contents are not fetched or re-serialised. If the user never expanded `/archive/` but commits a change to `/README.md`, the archive's tree object is reused verbatim. The server still holds the old tree object — no data is lost and no unnecessary decryption happens.

**Loaded folder:**
```javascript
// Recurse: build child entries, then create a new sub-tree object
const childEntries = await this._buildTreeEntries(entry)
const childTreeId  = await this._commitManager.createTree(childEntries)
// → PUT .../bare/data/{childTreeId}
entries.push({ name: folderName, tree_id: childTreeId, blob_id: null, ... })
```

A new `obj-cas-imm-*` tree object is created for every loaded folder that has been touched. Unmodified sub-folders within a loaded folder also get new tree objects (because their parent was loaded and re-serialised). This is the same as git's behaviour: touching any file creates new tree objects up the path to root.

### Step 2b: Settings blob

Before creating the root tree, settings are always re-serialised and stored:

```javascript
const settingsPlain     = new TextEncoder().encode(JSON.stringify(this._settings))
const settingsEncrypted = await this._sgSend.encrypt(settingsPlain, this._readKey)
const settingsBlobId    = await this._objectStore.store(settingsEncrypted)
// → PUT .../bare/data/{settingsBlobId}

entries.push({
    name:    '.vault-settings',
    blob_id: settingsBlobId,
    ...
})
```

Settings are stored as a blob (not a special file type) and appear as a regular file entry named `.vault-settings` in the root tree. This makes them transparent to the CLI and compatible with the full object model.

### Step 2c: `createTree(entries)` — Encrypt and store root tree

```javascript
async createTree(entries) {
    const encryptedEntries = await Promise.all(
        entries.map(entry => this._encryptTreeEntry(entry))
    )
    const tree = { schema: 'tree_v1', entries: encryptedEntries }
    const plaintext  = new TextEncoder().encode(JSON.stringify(tree))
    const ciphertext = await SGSendCrypto.encrypt(plaintext, this._readKey)
    return this._objectStore.store(ciphertext)
    // → PUT .../bare/data/{treeId}
}
```

Each tree entry is encrypted field-by-field before the tree JSON is assembled:

```javascript
async _encryptTreeEntry(entry) {
    const enc = async (value) => {
        const plain  = new TextEncoder().encode(String(value))
        const cipher = await SGSendCrypto.encrypt(plain, this._readKey)
        return base64(cipher)   // [IV][ciphertext][tag] → base64 string
    }
    return {
        name_enc:         await enc(entry.name),          // AES-GCM(filename)
        blob_id:          entry.blob_id || null,           // plaintext object pointer
        tree_id:          entry.tree_id || null,           // plaintext object pointer
        size_enc:         entry.blob_id ? await enc(entry.size) : null,
        content_hash_enc: entry.blob_id ? await enc(entry.content_hash) : null
    }
}
```

The final tree object on the wire:
```json
{
  "schema": "tree_v1",
  "entries": [
    {
      "name_enc":         "base64([IV][AES-GCM('README.md')][tag])",
      "blob_id":          "obj-cas-imm-abc123def456",
      "tree_id":          null,
      "size_enc":         "base64([IV][AES-GCM('2048')][tag])",
      "content_hash_enc": "base64([IV][AES-GCM('a1b2c3d4e5f6')][tag])"
    },
    {
      "name_enc":         "base64([IV][AES-GCM('documents')][tag])",
      "blob_id":          null,
      "tree_id":          "obj-cas-imm-789abcdef012",
      "size_enc":         null,
      "content_hash_enc": null
    }
  ]
}
```

This entire JSON is then encrypted as the outer AES-GCM layer and stored:
```
PUT /api/vault/write/{vaultId}/bare/data/{treeId}
Body: [12-byte IV][AES-GCM(tree JSON)][16-byte tag]
```

`blob_id` and `tree_id` are stored in plaintext within the encrypted tree JSON. They are SHA-256 hashes of other encrypted blobs — not semantic data. An attacker with server access can see the object graph topology (which blobs reference which) but cannot determine what any blob contains.

### Step 2d: `createCommit(...)` — Encrypt and store commit

```javascript
async createCommit({ parentIds, treeId, message, branchId }) {
    // Encrypt commit message separately
    const msgPlain  = new TextEncoder().encode(message)
    const msgCipher = await SGSendCrypto.encrypt(msgPlain, this._readKey)
    const msgEnc    = base64(msgCipher)

    const commit = {
        schema:       'commit_v2',
        parents:      parentIds,          // ["obj-cas-imm-{parent}"] or []
        tree_id:      treeId,             // "obj-cas-imm-{rootTree}"
        timestamp_ms: Date.now(),
        message_enc:  msgEnc,             // base64(AES-GCM(message))
        branch_id:    null,               // web UI doesn't use branch IDs
        signature:    null
    }

    // Encrypt the entire commit object
    const plaintext  = new TextEncoder().encode(JSON.stringify(commit))
    const ciphertext = await SGSendCrypto.encrypt(plaintext, this._readKey)
    return this._objectStore.store(ciphertext)
    // → PUT .../bare/data/{commitId}
}
```

The commit message is double-encrypted: once as `message_enc` inside the commit JSON, and again as part of the outer commit blob. This is belt-and-suspenders — even if the outer encryption were somehow broken, the message remains encrypted.

`parents` is an array to support merge commits (not currently used by the web UI — always 0 or 1 parent). The initial commit has `parents: []`.

### Step 2e: `writeRef(cloneRefFileId, commitId)` — Advance clone branch

```javascript
async writeRef(refFileId, commitId) {
    const filePath  = `bare/refs/${refFileId}`
    const payload   = new TextEncoder().encode(JSON.stringify({ commit_id: commitId }))
    const encrypted = await SGSendCrypto.encrypt(payload, this._readKey)
    await this._sgSend.vaultWrite(this._vaultId, filePath, this._writeKey, new Uint8Array(encrypted))
}
```

```
PUT /api/vault/write/{vaultId}/bare/refs/{cloneRefFileId}
Headers: x-sgraph-vault-write-key: {write_key}
Body: [12-byte IV][AES-GCM({"commit_id":"obj-cas-imm-..."})] [16-byte tag]
```

After this write, `this._headCommitId = commitId`.

### HTTP calls for a typical `_commit()` on a vault with one loaded folder

| # | Request | What it writes |
|---|---|---|
| 1 | `PUT .../bare/data/{settingsBlobId}` | Settings blob (re-serialised every commit) |
| 2 | `PUT .../bare/data/{subTreeId}` | Encrypted sub-tree for loaded folder |
| 3 | `PUT .../bare/data/{rootTreeId}` | Encrypted root tree |
| 4 | `PUT .../bare/data/{commitId}` | Encrypted commit object |
| 5 | `PUT .../bare/refs/{cloneRefFileId}` | Clone ref → new commitId |

For N loaded folders: N+2 tree objects + 1 settings blob + 1 commit + 1 ref = N+5 PUT requests per commit.

## 3. Encryption Count Per Commit

For a commit touching one file in one loaded folder:

| Object | Outer encryption | Inner field encryptions |
|---|---|---|
| File blob | 1× AES-GCM | — (already encrypted when stored via addFile) |
| Settings blob | 1× AES-GCM | — |
| Sub-tree (1 folder, M files) | 1× AES-GCM | 3M× AES-GCM (name, size, hash per file) |
| Root tree (root files + 1 folder ptr) | 1× AES-GCM | 3× AES-GCM per root file, 1× for folder name |
| Commit object | 1× AES-GCM | 1× AES-GCM (message_enc) |
| Ref | 1× AES-GCM | — |

Each AES-GCM call is one `crypto.subtle.encrypt()` with a fresh 12-byte IV. Web Crypto is hardware-accelerated on modern devices — the bottleneck is network (PUT latency), not CPU.

## 4. Object Immutability and History Preservation

Because objects are content-addressed and the server never deletes them (currently — GC is a future feature), every previous version of every file is preserved on the server indefinitely. Old `obj-cas-imm-*` blobs remain accessible at their original IDs. The only thing that changes between commits is which tree+commit objects the ref pointer points to.

This means:
- `git log`-equivalent: walk the `parents` chain of commit objects
- `git show HEAD~3:filename`: load commit N-3 → its tree_id → find file entry → load blob
- `git diff`: load tree from commit A and commit B, decrypt entries, compare content_hash_enc values

None of these require any server-side logic. Everything is in the encrypted object graph, readable by anyone with the `read_key`.
