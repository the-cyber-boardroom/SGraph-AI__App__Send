## 1. Cache Design Overview

The cache is built on the browser's [Cache API](https://developer.mozilla.org/en-US/docs/Web/API/Cache). It stores encrypted ciphertext — the same bytes the server stores and returns. Decryption still happens in the browser after each cache hit. The cache eliminates network round-trips, not the decrypt step.

```
  ┌────────────────────────────────────────────────────────────┐
  │                   Cache API: sg-vault-blocks               │
  │                                                            │
  │  Key (URL string)                                          │
  │  https://sgvault/{vaultId}/bare/data/{objectId}            │
  │                                                            │
  │  Value (Response)                                          │
  │  Body: [12-byte IV][N-byte ciphertext][16-byte GCM tag]    │
  │  Headers: Content-Type: application/octet-stream           │
  │  Status: 200                                               │
  │                                                            │
  │  Examples:                                                 │
  │  https://sgvault/jrinl85l/bare/data/obj-cas-imm-a2ad790bd3f1  │
  │  https://sgvault/jrinl85l/bare/data/obj-cas-imm-f81c3b9e0042  │
  └────────────────────────────────────────────────────────────┘
```

## 2. The Guard: Only `-imm-` Blocks Are Cached

```javascript
// _cacheGet and _cachePut both check this before doing anything:
if (!objectId.includes('-imm-') || typeof caches === 'undefined') return null
```

The `-imm-` substring in the object ID is the cache eligibility gate. The four-segment naming scheme encodes mutability directly in the ID:

```
  obj-cas-imm-{sha256[:12]}   ← immutable, content-addressed, SAFE TO CACHE
  ref-pid-muw-{hmac[:12]}     ← mutable (multi-writer), NEVER cached
  ref-pid-snw-{hmac[:12]}     ← mutable (single-writer), NEVER cached
  idx-pid-muw-{hmac[:12]}     ← mutable, NEVER cached
```

Ref files (`muw`, `snw`) never pass through `load()` — they go directly through `SGVaultRefManager.readRef()` which has no cache layer. But the `-imm-` guard is defence-in-depth: even if a ref ID were accidentally passed to `load()`, it would not be cached.

```
  What IS cached:
    obj-cas-imm-*  →  all file blobs, tree objects, commit objects, settings blobs

  What is NOT cached:
    ref-pid-muw-*  →  named HEAD (always live)
    ref-pid-snw-*  →  clone HEAD (always live)
    idx-pid-muw-*  →  branch index (always live)
    presigned S3 URLs  →  per-request, short-lived (900 s)
```

## 3. Cache Key Design

```javascript
_cacheKey(filePath) {
    return SG_VAULT_CACHE_URL + this._vaultId + '/' + filePath
    // → 'https://sgvault/' + 'jrinl85l' + '/' + 'bare/data/obj-cas-imm-a2ad790bd3f1'
    // → 'https://sgvault/jrinl85l/bare/data/obj-cas-imm-a2ad790bd3f1'
}
```

The `https://sgvault/` prefix is a **synthetic URL** — not a real network address. The Cache API requires URL-shaped keys; this scheme satisfies that requirement while keeping the meaningful part of the path (`{vaultId}/bare/data/{objectId}`) verbatim.

### Why `{vaultId}` is in the key

Two different vaults could have blobs whose SHA-256 hashes happen to collide in the first 12 hex characters. Including `{vaultId}` in the cache key scopes entries per vault and prevents cross-vault cache pollution:

```
  Vault A (id=abc12345): https://sgvault/abc12345/bare/data/obj-cas-imm-aaaaaa000000
  Vault B (id=xyz99999): https://sgvault/xyz99999/bare/data/obj-cas-imm-aaaaaa000000
  → Different cache entries despite identical object IDs
```

## 4. `_cachePut()` — The `data.slice(0)` Copy

```javascript
async _cachePut(objectId, filePath, data) {
    // ...
    await cache.put(
        this._cacheKey(filePath),
        new Response(data.slice(0), {   // ← slice(0) is critical
            status:  200,
            headers: { 'Content-Type': 'application/octet-stream' }
        })
    )
}
```

`data.slice(0)` creates a **copy** of the ArrayBuffer before wrapping it in a `Response`. This is necessary because:

1. `ArrayBuffer` objects can be **transferred** (zero-copy ownership transfer) by callers. If `load()` returns `data` and the caller transfers it to a Web Worker via `postMessage([data], [data])`, the original becomes detached and zero-length.
2. `_cachePut` runs concurrently with the return of `data` to the caller. If the caller transfers `data` before `cache.put()` reads the `Response` body, the cache would store an empty blob.
3. `slice(0)` copies before either of these can happen.

```
  load() returns data to caller
       │
       ├─ _cachePut(objectId, filePath, data)
       │      └─ data.slice(0) → NEW ArrayBuffer (independent copy)
       │         cache.put(key, Response(copy))   ← stores the copy
       │
       └─ return data → caller
              └─ caller may transfer data to worker — safe, cache has its own copy
```

## 5. Graceful Degradation

The cache is entirely optional. Both `_cacheGet` and `_cachePut` catch all errors silently:

```javascript
async _cacheGet(objectId, filePath) {
    if (!objectId.includes('-imm-') || typeof caches === 'undefined') return null
    try {
        const cache = await caches.open(SG_VAULT_CACHE_NAME)
        const hit   = await cache.match(this._cacheKey(filePath))
        if (hit) return hit.arrayBuffer()
    } catch (_) { /* storage unavailable — fall through to network */ }
    return null
}

async _cachePut(objectId, filePath, data) {
    if (!objectId.includes('-imm-') || typeof caches === 'undefined') return
    try {
        // ...
    } catch (_) { /* quota exceeded or storage unavailable — ignore */ }
}
```

Scenarios where the cache silently degrades:
- **Service worker not present** — `typeof caches === 'undefined'` is `true`, both methods return immediately
- **Private/incognito mode** — Cache API may be unavailable or quota-zero; `try/catch` handles it
- **Storage quota exceeded** — `_cachePut` throws on `cache.put()`; caught and ignored; vault continues working, just slower
- **Cache corruption** — `cache.match()` throws or returns a bad Response; caught; falls through to network

The vault is **correct** without the cache. It is merely faster with it.

## 6. Cache Lifecycle and Management

The cache persists across page loads and browser sessions until explicitly cleared. There is no TTL — immutable blocks never expire.

### Inspecting the cache (browser DevTools console)

```javascript
// List all cached vaults and block counts
const cache = await caches.open('sg-vault-blocks')
const keys  = await cache.keys()
const byVault = {}
for (const req of keys) {
    const vaultId = req.url.split('/')[3]  // https://sgvault/{vaultId}/...
    byVault[vaultId] = (byVault[vaultId] || 0) + 1
}
console.table(byVault)
// → { jrinl85l: 155, abc12345: 32 }

// Total cached bytes for one vault
let bytes = 0
for (const req of keys.filter(r => r.url.includes('/jrinl85l/'))) {
    const resp = await cache.match(req)
    const buf  = await resp.arrayBuffer()
    bytes += buf.byteLength
}
console.log(`Cached: ${(bytes / 1024 / 1024).toFixed(1)} MB`)
```

### Clearing cache (storage settings UI — future feature)

```javascript
// Clear one vault
const cache = await caches.open('sg-vault-blocks')
const keys  = await cache.keys()
for (const req of keys) {
    if (req.url.includes(`/${vaultId}/`)) await cache.delete(req)
}

// Clear all vault caches
await caches.delete('sg-vault-blocks')
```

## 7. Performance Measurements

Measured on a 251 MB vault with 155 files, desktop browser:

```
  ┌────────────────────────────────────────────────────────────┐
  │  Condition                          Load time              │
  ├────────────────────────────────────────────────────────────┤
  │  Cold — cache disabled              ~6 s                   │
  │  Cold — cache enabled (first load)  ~6 s (same, fills cache│
  │  Second load — cache populated      ~1 s                   │
  │  Warm — browser cache + sg-vault-   820 ms                 │
  │          blocks cache               (limited by decrypt CPU)│
  └────────────────────────────────────────────────────────────┘
```

The 820 ms floor is dominated by:
1. 3 live ref GETs (network, cannot be cached) — ~200–300 ms
2. AES-256-GCM decrypt for all 155 file metadata entries — ~200 ms
3. UI rendering — ~100 ms

The network cost for the 251 MB of ciphertext drops to **zero** after the first full access.

## 8. CLI Implementation Specification

The same dynamic access + cache model can be implemented in `sgit` with a filesystem cache instead of the browser's Cache API. Here is the full specification.

### 8a. Cache location

```
~/.sgit/cache/
  blocks/
    {vaultId}/
      {objectId}        ← raw ciphertext bytes (identical to server response)

  e.g.:
  ~/.sgit/cache/blocks/jrinl85l/obj-cas-imm-a2ad790bd3f1
  ~/.sgit/cache/blocks/jrinl85l/obj-cas-imm-f81c3b9e0042
```

The directory structure mirrors the URL structure of the browser cache key. The content is identical: raw `[IV][ciphertext][tag]` bytes with no additional metadata.

### 8b. Cache guard (same rule as browser)

```python
def is_cacheable(object_id: str) -> bool:
    return '-imm-' in object_id

# Usage:
cached = load_from_cache(vault_id, object_id) if is_cacheable(object_id) else None
if cached:
    return cached
data = fetch_from_server(vault_id, object_id)
if is_cacheable(object_id):
    save_to_cache(vault_id, object_id, data)
return data
```

### 8c. `sgit cat` — Dynamic access without full clone

The primary new CLI command enabled by this model:

```
sgit cat <vault-key> <vault-path>

  # Examples:
  sgit cat correct-horse-battery:a1b2c3d4 /README.md
  sgit cat correct-horse-battery:a1b2c3d4 /documents/report.pdf > report.pdf
  sgit cat word-word-1234 /slides/deck.pptx | open -f -a Preview
```

Implementation flow:

```
  1. deriveKeys(passphrase, vaultId)           (CPU only)
  2. GET bare/refs/{refFileId}                 (live, uncached)
     → namedCommitId
  3. load(namedCommitId)                       (cache check first)
     → commit object → tree_id
  4. load(commit.tree_id)                      (cache check first)
     → root tree → decrypt entries
  5. walk path segments:
       for each path segment:
         find matching entry in current tree
         if entry.tree_id:
           load(entry.tree_id)                 (cache check first)
           → sub-tree → decrypt entries
         else:
           if entry.blob_id: found
  6. load(entry.blob_id)                       (cache check first — likely cold)
     → [IV][ciphertext][tag]
     → decrypt(ciphertext, readKey) → plaintext
  7. write plaintext to stdout / file

  Network calls for cold sgit cat on a 3-level-deep file:
    1 ref GET + 1 commit GET + 3 tree GETs + 1 blob GET = 6 GETs
    (vs sgit clone: potentially thousands of GETs)
```

### 8d. `sgit ls` — Dynamic folder listing

```
sgit ls <vault-key> [vault-path]

  # Examples:
  sgit ls correct-horse-battery:a1b2c3d4
  sgit ls correct-horse-battery:a1b2c3d4 /documents
  sgit ls correct-horse-battery:a1b2c3d4 /documents --recursive
```

For non-recursive `ls`: same flow as `sgit cat` steps 1–5, but output entry names instead of fetching the blob. **Zero blob fetches.**

For recursive `ls` with lazy loading:

```
  walk_tree(tree_id, path_prefix):
    load(tree_id)  →  decrypt entries
    for each entry:
      if entry.tree_id:
        walk_tree(entry.tree_id, path_prefix + '/' + name)  ← recurse
      else:
        print(path_prefix + '/' + name, size)

  Only fetches tree objects, never blobs.
  Network: 1 ref + 1 commit + N tree objects (one per directory level visited)
```

### 8e. Cache pre-warming (`sgit warm`)

For use cases where offline access is needed (e.g. before a flight):

```
sgit warm <vault-key> [vault-path]

  Downloads and caches all blobs under vault-path without decrypting them.
  Equivalent to recursive getFile() for every file.
  After warming, all cached files are available at <10 ms read latency.

  Flow:
    walk_tree recursively → collect all blob_ids
    for each blob_id:
      if not in cache:
        fetch(blob_id) → save to cache
    print: "Warmed N files, M MB"
```

This is the CLI equivalent of a lazy clone — the user explicitly chooses to download everything, but at a time they control (rather than blocking the initial `open()`).

### 8f. Cache invalidation (never needed for imm blocks)

```
  obj-cas-imm-{sha256[:12]}:
    ID = SHA-256(ciphertext)[:12]
    Same ID → same bytes, forever.
    Cache entries never expire, never need revalidation.
    No ETag checks, no If-Modified-Since headers.
    Delete only when user explicitly clears the cache.

  sgit cache clear [vault-id]    ← manual eviction only
  sgit cache stats               ← list vaults and sizes in cache
```

## 9. The On-Demand Sub-Tree Protocol: Full Specification

This is the complete protocol for any client (browser or CLI) implementing lazy sub-tree loading:

```
  DATA STRUCTURES:
  ─────────────────
  FolderNode:
    type:      'folder'
    children:  map[name → FolderNode | FileNode]
    _loaded:   bool | undefined
    _tree_id:  string | undefined   (set only when _loaded === false)

  FileNode:
    type:         'file'
    blob_id:      string            (obj-cas-imm-*)
    size:         int
    content_hash: string            (SHA-256(plaintext)[:12])

  INVARIANTS:
  ────────────
  1. Root '/' is always loaded (_loaded = true or undefined, no _tree_id)
  2. A node with _loaded === false MUST have _tree_id set
  3. A node with _loaded === true MUST NOT have _tree_id
  4. A node without _loaded is a freshly created (uncommitted) folder
  5. _buildTreeEntries MUST reuse _tree_id for nodes where _loaded === false
  6. loadSubTreeOnDemand MUST set _loaded = true and delete _tree_id after load

  OPERATIONS:
  ────────────
  open(key):
    derive keys
    read named HEAD ref (live)
    read clone HEAD ref (live, may 404)
    load commit (cache OK)
    load root tree (cache OK)
    materialise root level: files as FileNode, sub-dirs as STUB FolderNode
    load settings blob (cache OK)

  needsLoading(path):
    node = findNode(path)
    return node.type == 'folder' AND node._loaded === false AND node._tree_id != null

  loadSubTreeOnDemand(path):
    node = findNode(path)
    if not needsLoading(path): return false
    subTree = load(node._tree_id)   ← cache check first
    for entry in subTree.entries:
      if entry.tree_id: node.children[name] = STUB FolderNode(_tree_id=entry.tree_id)
      else:             node.children[name] = FileNode(blob_id, size, content_hash)
    node._loaded = true
    delete node._tree_id
    return true

  getFile(folderPath, fileName):
    folder = findNode(folderPath)
    entry  = folder.children[fileName]
    data   = load(entry.blob_id)    ← cache check first
    return decrypt(data, readKey)

  commit(message):
    entries = buildTreeEntries(root)   ← stubs pass through unchanged
    treeId  = createTree(entries)
    commitId = createCommit(parentIds, treeId, message)
    writeRef(cloneRefFileId, commitId)
```

This single property makes permanent caching safe, makes lazy loading correct, and makes dynamic access equivalent to a local clone — populated gradually, at the speed of user navigation.
