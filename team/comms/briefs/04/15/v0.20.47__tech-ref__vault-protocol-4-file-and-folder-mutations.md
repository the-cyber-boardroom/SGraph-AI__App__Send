## Overview

Every mutation in the vault follows the same three-phase pattern:

```
Phase 1 — Content (file ops only):
    encrypt content → store blob → compute content hash

Phase 2 — Tree:
    mutate in-memory tree (_tree)

Phase 3 — Commit:
    _commit(message)
        → _buildTreeEntries() → createTree() per folder
        → createCommit()
        → writeRef(cloneRefFileId, commitId)
```

Folder operations skip Phase 1 entirely. All operations end with `_commit()`, which is the only code that writes to the server (apart from the blob store in Phase 1). The in-memory tree is the working copy — it is the source of truth between operations.

## 1. File Mutations

### 1a. `addFile(folderPath, fileName, fileData)`

Adds a new file to the vault. `folderPath` is the vault path of the containing folder (e.g. `'/'` for root, `'/documents'` for a sub-folder). `fileData` is an `ArrayBuffer` or `Uint8Array`.

**Step-by-step:**

```javascript
async addFile(folderPath, fileName, fileData) {
    const data = fileData instanceof ArrayBuffer ? new Uint8Array(fileData) : fileData

    // Phase 1: encrypt and store blob
    const encrypted   = await this._sgSend.encrypt(data, this._readKey)
    const blobId      = await this._objectStore.store(encrypted)
    // → PUT /api/vault/write/{vaultId}/bare/data/{blobId}
    const contentHash = await this._commitManager.computeContentHash(data)
    // computeContentHash: SHA-256(plaintext)[:12 hex]  — pure CPU, no network

    // Phase 2: update in-memory tree
    const folder = this._findNode(folderPath)
    folder.children[fileName] = {
        type:         'file',
        blob_id:      blobId,
        size:         data.byteLength || data.length,
        content_hash: contentHash
    }

    // Phase 3: commit
    await this._commit(`Add ${fileName}`)
    return { blobId, fileName, folderPath }
}
```

**Key points:**
- The blob is stored **before** the commit. If `_commit()` fails after the blob store, the blob exists on the server but no commit or tree references it — it is an orphan (future GC will clean it up). This is safe: the vault state is unchanged because the ref hasn't moved.
- `encrypt()` uses the `read_key` (not `write_key`). The `write_key` is the HTTP header; `read_key` is the AES-GCM key for all content.
- `contentHash` is `SHA-256(plaintext)[:12 hex]`. It is stored inside the tree entry's `content_hash_enc` field (AES-GCM encrypted). It is the basis for future `git diff`-equivalent operations.
- The folder named `folderPath` must already exist in the loaded tree. `_findNode()` returns `null` if the path doesn't exist or is a file, causing the method to throw.

**HTTP calls for `addFile()` with one folder in-memory:**

| # | Request | What it writes |
|---|---|---|
| 1 | `PUT .../bare/data/{blobId}` | Encrypted file content |
| 2 | `PUT .../bare/data/{settingsBlobId}` | Re-serialised vault settings |
| 3 | `PUT .../bare/data/{rootTreeId}` | Encrypted root tree (updated entry list) |
| 4 | `PUT .../bare/data/{commitId}` | Encrypted commit object |
| 5 | `PUT .../bare/refs/{cloneRefFileId}` | Clone ref → new commitId |

If the file is in a sub-folder (not root), that sub-folder's tree object is also created:

| # | Request | What it writes |
|---|---|---|
| 1 | `PUT .../bare/data/{blobId}` | Encrypted file content |
| 2 | `PUT .../bare/data/{settingsBlobId}` | Re-serialised vault settings |
| 3 | `PUT .../bare/data/{subTreeId}` | Encrypted sub-folder tree (updated) |
| 4 | `PUT .../bare/data/{rootTreeId}` | Encrypted root tree (sub-folder pointer updated) |
| 5 | `PUT .../bare/data/{commitId}` | Encrypted commit object |
| 6 | `PUT .../bare/refs/{cloneRefFileId}` | Clone ref → new commitId |

General formula: 1 blob + (N loaded folders) tree objects + 1 settings blob + 1 commit + 1 ref = **N + 4** PUT requests.

### 1b. `updateFile(folderPath, fileName, fileData)`

Replaces the content of an existing file.

```javascript
async updateFile(folderPath, fileName, fileData) {
    const data = fileData instanceof ArrayBuffer ? new Uint8Array(fileData) : fileData

    const folder = this._findNode(folderPath)
    if (!folder.children[fileName] || folder.children[fileName].type !== 'file') {
        throw new Error(`File not found: ${fileName}`)
    }

    // Phase 1: encrypt and store new blob
    const encrypted   = await this._sgSend.encrypt(data, this._readKey)
    const blobId      = await this._objectStore.store(encrypted)
    const contentHash = await this._commitManager.computeContentHash(data)

    // Phase 2: overwrite in-memory tree entry
    folder.children[fileName] = {
        type:         'file',
        blob_id:      blobId,       // new blob ID — old blob still exists on server
        size:         data.byteLength || data.length,
        content_hash: contentHash   // new hash
    }

    // Phase 3: commit
    await this._commit(`Edit ${fileName}`)
    return { blobId, fileName, folderPath }
}
```

**Critical history note:** The old `blob_id` is **not deleted** from the server. The previous version of the file remains at its original `obj-cas-imm-*` address, referenced by the previous commit's tree chain. History is preserved automatically by the content-addressed model — nothing is overwritten, only new objects are created and the ref pointer advances.

This means every file update is logically equivalent to creating a new file with a new blob ID. The old content is accessible by walking the commit graph.

**HTTP calls:** identical to `addFile()` — the only difference is the commit message. The old blob is never touched.

### 1c. `removeFile(folderPath, fileName)`

Removes a file from the vault.

```javascript
async removeFile(folderPath, fileName) {
    const folder = this._findNode(folderPath)
    if (!folder) throw new Error(`Folder not found: ${folderPath}`)

    // Phase 2: remove from in-memory tree (Phase 1 skipped — no new blob)
    delete folder.children[fileName]

    // Phase 3: commit
    await this._commit(`Remove ${fileName}`)
}
```

**No blob is deleted from the server.** The `obj-cas-imm-*` blob for the removed file still exists. The new commit's tree simply no longer references it. From a read perspective, the file is gone (no path in the new tree leads to that blob). From a history perspective, the file is recoverable by loading any previous commit that referenced it.

**HTTP calls:** no blob write, so one fewer PUT than `addFile()`:

| # | Request | What it writes |
|---|---|---|
| 1 | `PUT .../bare/data/{settingsBlobId}` | Settings (re-serialised every commit) |
| 2 | `PUT .../bare/data/{rootTreeId}` | Root tree (entry removed) |
| 3 | `PUT .../bare/data/{commitId}` | Commit |
| 4 | `PUT .../bare/refs/{cloneRefFileId}` | Clone ref |

### 1d. `renameFile(folderPath, oldName, newName)`

Renames a file within the same folder.

```javascript
async renameFile(folderPath, oldName, newName) {
    const folder = this._findNode(folderPath)
    if (!folder.children[oldName]) throw new Error(`File not found: ${oldName}`)
    if (folder.children[newName])  throw new Error(`Already exists: ${newName}`)

    // Phase 2: move the in-memory entry to new key (same blob_id, same content_hash)
    folder.children[newName] = folder.children[oldName]
    delete folder.children[oldName]

    // Phase 3: commit
    await this._commit(`Rename ${oldName} to ${newName}`)
}
```

**The blob is not re-uploaded.** The existing `blob_id` is preserved in the tree entry — only the encrypted `name_enc` field changes in the new tree object. The file content is byte-for-byte identical on the server; only the name pointer in the tree is different.

This means a rename is one of the cheapest operations: no blob write, just a new tree + commit + ref.

**HTTP calls:** same as `removeFile()` (no blob write).

### 1e. `moveFile(srcFolderPath, fileName, destFolderPath)`

Moves a file between folders.

```javascript
async moveFile(srcFolderPath, fileName, destFolderPath) {
    if (srcFolderPath === destFolderPath) return   // no-op

    const srcFolder  = this._findNode(srcFolderPath)
    const entry      = srcFolder.children[fileName]
    if (!entry || entry.type !== 'file') throw new Error(`File not found: ${fileName}`)

    const destFolder = this._findNode(destFolderPath)
    if (destFolder.children[fileName]) throw new Error(`Already exists in destination: ${fileName}`)

    // Phase 2: transfer entry between folders in-memory
    destFolder.children[fileName] = entry   // same object reference, same blob_id
    delete srcFolder.children[fileName]

    // Phase 3: commit
    await this._commit(`Move ${fileName}`)
}
```

**Both source and destination folders must be loaded** (not lazy stubs). If either is a stub, `_findNode()` returns the stub node, but its `children` is empty — the operation would silently fail or throw. The UI must call `loadSubTreeOnDemand()` on both folders before calling `moveFile()`.

**HTTP calls:** both the source folder's tree object and the destination folder's tree object are re-created (two sub-tree PUTs if both are sub-folders), plus settings + commit + ref.

## 2. Folder Mutations

All folder mutations are pure in-memory tree operations — no blob is created or deleted. Each ends with `_commit()` which serialises the updated tree.

### 2a. `createFolder(folderPath)`

```javascript
async createFolder(folderPath) {
    const parts  = folderPath.split('/').filter(Boolean)
    const name   = parts.pop()
    const parent = this._findNode('/' + parts.join('/'))
    if (!parent || parent.type !== 'folder') throw new Error(`Parent not found`)
    if (parent.children[name])               throw new Error(`Already exists: ${name}`)

    // Phase 2: insert empty folder into in-memory tree
    parent.children[name] = { type: 'folder', children: {} }

    // Phase 3: commit
    await this._commit(`Create folder ${name}`)
}
```

An empty folder `{ type: 'folder', children: {} }` produces an empty `entries: []` array in `_buildTreeEntries()`, which results in an empty tree object on the server. The folder exists in history even with no files.

**HTTP calls:** same as `removeFile()` (settings + root tree + commit + ref), plus one extra tree PUT for the new empty sub-folder.

---

### 2b. `removeFolder(folderPath)`

```javascript
async removeFolder(folderPath) {
    const parts  = folderPath.split('/').filter(Boolean)
    const name   = parts.pop()
    const parent = this._findNode('/' + parts.join('/'))
    delete parent.children[name]   // entire sub-tree removed in one operation
    await this._commit(`Remove folder ${name}`)
}
```

**The entire sub-tree is removed from the in-memory tree.** All files and nested folders under `folderPath` are deleted in one commit. Their `obj-cas-imm-*` blobs remain on the server (recoverable from history), but the new commit's tree has no reference to them.

If the folder is a lazy stub (`_loaded: false`), deleting it from the parent's `children` is sufficient — the stub's `_tree_id` is discarded. No fetch of the sub-tree contents is needed to remove the folder.

---

### 2c. `renameFolder(folderPath, newName)`

```javascript
async renameFolder(folderPath, newName) {
    const parts   = folderPath.split('/').filter(Boolean)
    const oldName = parts.pop()
    const parent  = this._findNode('/' + parts.join('/'))
    if (!parent.children[oldName]) throw new Error(`Folder not found: ${oldName}`)
    if (parent.children[newName])  throw new Error(`Already exists: ${newName}`)

    // Phase 2: transfer entry under new key (same node object, including _tree_id if unloaded)
    parent.children[newName] = parent.children[oldName]
    delete parent.children[oldName]

    await this._commit(`Rename folder ${oldName} to ${newName}`)
}
```

**Works correctly on unloaded folders.** If the folder being renamed is a lazy stub (`_loaded: false, _tree_id: 'obj-cas-imm-...'`), the entire stub object is moved to the new key. `_buildTreeEntries()` sees the stub and reuses the existing `_tree_id` unchanged. The result is a new root tree object with an encrypted `name_enc` pointing to the same sub-tree object as before — no sub-tree re-serialisation needed.

This is an efficient O(1) rename: one new root tree object, regardless of how many files the folder contains.

---

### 2d. `moveFolder(srcPath, destParentPath)`

```javascript
async moveFolder(srcPath, destParentPath) {
    const srcParts    = srcPath.split('/').filter(Boolean)
    const folderName  = srcParts.pop()
    const srcParentPath = '/' + srcParts.join('/')

    // Guard: cannot move a folder into itself or its descendant
    if (destParentPath === srcPath || destParentPath.startsWith(srcPath + '/')) {
        throw new Error('Cannot move folder into itself')
    }

    const srcParent  = this._findNode(srcParentPath)
    const folderNode = srcParent.children[folderName]  // may be an unloaded stub

    const destParent = this._findNode(destParentPath)
    if (destParent.children[folderName]) throw new Error(`Already exists in destination`)

    // Phase 2: move node reference in-memory
    destParent.children[folderName] = folderNode
    delete srcParent.children[folderName]

    await this._commit(`Move folder ${folderName}`)
}
```

Like `renameFolder()`, this works on unloaded stubs. The stub's `_tree_id` is preserved and reused in the commit. Both source and destination parents get new tree objects (since both were loaded — they're both in the root tree which is always loaded).

## 4. The Content Hash

`computeContentHash(plaintext)` returns `SHA-256(plaintext bytes)[:12 hex]`. This is:

```javascript
async computeContentHash(data) {
    const bytes  = data instanceof Uint8Array ? data : new Uint8Array(data)
    const digest = await crypto.subtle.digest('SHA-256', bytes)
    return Array.from(new Uint8Array(digest)).map(b => b.toString(16).padStart(2,'0')).join('').slice(0, 12)
}
```

It is stored encrypted inside the tree entry's `content_hash_enc` field. The server never sees it.

The content hash is the basis for:
- **Deduplication detection** (two files with the same `content_hash` have the same plaintext content)
- **diff operations** (compare `content_hash` across commits to find changed files without decrypting blobs)
- **Integrity verification** (download blob, decrypt, re-compute hash, compare)

Note: it is NOT used for blob addressing — the blob ID is `SHA-256(ciphertext)[:12]` (of the encrypted blob), while the content hash is `SHA-256(plaintext)[:12]` (of the raw file bytes). Same algorithm, different inputs.

## 5. `listFolder(folderPath)` — Non-Mutating Introspection

```javascript
listFolder(folderPath) {
    const node = this._findNode(folderPath)
    if (!node || node.type !== 'folder') return null
    return Object.entries(node.children).map(([name, entry]) => ({
        name,
        type:         entry.type,
        size:         entry.size || 0,
        content_hash: entry.content_hash || null
    }))
}
```

Returns the immediate children of a folder from in-memory state. No network calls. Unloaded sub-folders appear in the list with `type: 'folder'` and `size: 0, content_hash: null` — the UI uses `needsLoading()` to decide whether to show a loading spinner before expanding.

This is the equivalent of `ls` — it shows names and sizes but does not fetch content.

## 6. `getStats()` — Vault-Wide Totals

```javascript
getStats() {
    let files = 0, folders = 0, totalSize = 0
    const walk = (node) => {
        for (const [, entry] of Object.entries(node.children || {})) {
            if (entry.type === 'folder') { folders++; walk(entry) }
            else { files++; totalSize += entry.size || 0 }
        }
    }
    walk(this._tree['/'])
    return { files, folders, totalSize }
}
```

Walks the **loaded** portion of the in-memory tree only. Unloaded sub-folders are counted as one folder each, but their contents are not reflected in `files` or `totalSize`. If the UI needs accurate counts, it must load all sub-trees first.

No network calls. Pure in-memory traversal.
