## 1. Two Access Models

There are two fundamentally different ways to work with a vault. Understanding the difference is the core insight behind the web UI's performance characteristics.

### Model A: Full Clone

```
┌─────────────┐         ┌──────────────────────────────────────┐
│   Client    │  clone  │             Server                   │
│             │ ──────► │  obj-cas-imm-aaa  (file 1 blob)      │
│  local/     │         │  obj-cas-imm-bbb  (file 2 blob)      │
│  disk copy  │         │  obj-cas-imm-ccc  (file 3 blob)      │
│             │         │  obj-cas-imm-ddd  (tree)             │
│  ALL blobs  │         │  obj-cas-imm-eee  (commit)           │
│  downloaded │         │  ref-pid-muw-xxx  (named HEAD)       │
└─────────────┘         └──────────────────────────────────────┘

Operation:  download everything first, then work locally
Network:    proportional to vault size (251 MB vault = 251 MB download)
Latency:    high upfront, zero after
```

This is how `sgit clone` works. Every blob, tree, and commit object is fetched in full before the client begins work. The client then has a complete local working copy and can operate offline.

### Model B: Dynamic Access

```
┌─────────────┐         ┌──────────────────────────────────────┐
│   Client    │         │             Server                   │
│             │  HEAD   │                                      │
│  in-memory  │ ◄─────  │  ref-pid-muw-xxx  (named HEAD)  ─┐  │
│  tree stub  │         │                                   │  │
│             │  root   │  obj-cas-imm-eee  (commit)    ◄──┘  │
│  /          │ ◄─────  │  obj-cas-imm-ddd  (root tree) ◄──   │
│  ├ docs/▸  │         │                                      │
│  ├ imgs/▸  │         │  obj-cas-imm-aaa  (file 1 blob)      │
│  └ README  │  fetch  │  obj-cas-imm-bbb  (file 2 blob)      │
│            │ on demand│  obj-cas-imm-ccc  (file 3 blob)      │
└─────────────┘         │  obj-cas-imm-fff  (docs/ tree)       │
                        │  obj-cas-imm-ggg  (imgs/ tree)       │
                        └──────────────────────────────────────┘

Operation:  fetch only what is needed, when it is needed
Network:    proportional to what the user actually accesses
Latency:    low upfront, per-item on demand
```

This is how `SGVault.open()` works. Only 5–6 objects are fetched on open (ref, commit, root tree, settings blob). Sub-folders and file contents are fetched only when the user expands or opens them.

## 2. The Game-Changer: Dynamic Access + Immutable Cache

On its own, dynamic access trades clone-time latency for per-request latency. Each file open costs a network round-trip. For a presentation vault accessed on stage, that per-file latency is unacceptable.

The Cache API for immutable blocks changes this completely:

```
First access (cold):

  Browser                    Cache API             Server
    │                            │                    │
    │  load(obj-cas-imm-aaa)     │                    │
    │ ─────────────────────────► │  MISS              │
    │                            │ ──────────────────►│
    │                            │  [IV][cipher][tag] │
    │                            │ ◄──────────────────│
    │◄─────────────────────────  │  store in cache    │
    │  ArrayBuffer               │                    │

Second access (warm, same session or future session):

  Browser                    Cache API             Server
    │                            │                    │
    │  load(obj-cas-imm-aaa)     │                    │
    │ ─────────────────────────► │  HIT               │
    │◄─────────────────────────  │                    │
    │  ArrayBuffer (instant)     │                    │
                                             (no network call)
```

Because `obj-cas-imm-*` IDs encode the SHA-256 of the ciphertext, the same ID **always** maps to the same bytes. The cache never needs invalidation. Once a block is cached, it is cached permanently.

### The combined effect

```
┌──────────────────────────────────────────────────────────────┐
│                  Dynamic Access + Cache                      │
│                                                              │
│  First vault open (cold):                                    │
│    5 network GETs  →  metadata only  →  ~1–2 s              │
│                                                              │
│  First file access (cold):                                   │
│    1 network GET per blob  →  ~200–400 ms per file           │
│    └─► blob stored in Cache API immediately after            │
│                                                              │
│  Subsequent vault opens (same vault, any session):           │
│    5 network GETs (refs)  +  0 network (all blobs cached)    │
│    →  820 ms total for a 251 MB / 155-file vault             │
│                                                              │
│  Subsequent file access (cached):                            │
│    0 network GETs  →  pure CPU decrypt  →  <10 ms            │
└──────────────────────────────────────────────────────────────┘
```

This gives the web UI the best of both worlds:
- **Small initial footprint** — only metadata fetched on open (like dynamic access)
- **Local-speed subsequent access** — cached blobs serve at memory speed (like a local clone)

The vault behaves like a local copy without ever being cloned. The cache *is* the clone — populated lazily, on demand, in the background.

## 3. What "On-Demand" Means at Each Level

There are three levels of on-demand loading, each independent of the others:

```
Level 1: Tree structure (folder contents)
─────────────────────────────────────────
  open()  →  root tree only (1 GET)
  expand folder  →  sub-tree (1 GET per folder, on demand)
  expand nested folder  →  another sub-tree (1 GET, on demand)

  Trigger: user clicks to expand a folder in the UI
  Object type: obj-cas-imm-* (tree object, ~hundreds of bytes)
  Cached: yes — tree objects are immutable blobs

Level 2: File content (blob data)
──────────────────────────────────
  open() / expand()  →  no file content fetched
  getFile()  →  blob for that file only (1 GET, on demand)

  Trigger: user opens/downloads a file, or viewer renders it
  Object type: obj-cas-imm-* (file blob, up to hundreds of MB)
  Cached: yes — file blobs are immutable

Level 3: Refs and commits (navigation)
───────────────────────────────────────
  open()  →  3 ref GETs + 1 commit GET (always network)
  getAheadCount()  →  commit walk (likely all cached)
  getBehindCount()  →  1 live ref GET + commit walk

  Trigger: vault open, push/pull UI, sync checks
  Object type: ref-pid-* (never cached), obj-cas-imm-* (commit, cacheable)
  Cached: refs never; commits yes
```

The pattern is: **navigate for free, pay only for what you read**.

## 4. Contrast with `sgit clone`

| Property | `sgit clone` | `SGVault.open()` + Cache |
|---|---|---|
| Initial network | 100% of vault size | ~1 KB metadata only |
| Time to first interaction | Proportional to vault size | ~1–2 s regardless of size |
| Time to read file (cold) | 0 (already local) | ~200–400 ms |
| Time to read file (warm) | 0 (already local) | <10 ms (Cache API) |
| Offline support | Full | Only cached blobs |
| Storage location | Filesystem (explicit) | Browser Cache API (automatic) |
| Storage management | Manual (`sgit gc`) | Browser handles eviction |
| Concurrent modifications | Tracked via clone ref | Tracked via clone ref (same model) |
| Works on mobile/web | No (CLI dependency) | Yes |
| Vault size limit | Disk space | Browser cache quota (~1–10 GB) |

The web UI dynamic access model is not a degraded clone — it is a different access pattern suited to browse-on-demand use cases. For use cases that need offline-first or script-based access, `sgit clone` remains the right tool.

## 6. Why This Matters for the CLI

The CLI (`sgit`) currently uses a full-clone model: `sgit clone` downloads all objects to a local `.sgit/` directory before any files are accessible. This is correct for scripting, batch operations, and offline use.

But for interactive tools — a TUI file browser, a CLI vault viewer, a `sgit cat <path>` command — dynamic access is superior:

```
sgit cat docs/report.pdf
  → without dynamic access: clone entire vault first (may be 10 GB)
  → with dynamic access: fetch root tree, walk to docs/ tree, fetch report.pdf blob
                          total: 3–4 GETs regardless of vault size
```

The lazy-loading sub-tree protocol and the immutable block cache are both fully portable to the CLI. The CLI has a filesystem to store the cache (instead of the browser Cache API) and can use the same HTTP endpoints. The object ID scheme (`obj-cas-imm-*`) provides the same permanent-cache guarantee on disk.
