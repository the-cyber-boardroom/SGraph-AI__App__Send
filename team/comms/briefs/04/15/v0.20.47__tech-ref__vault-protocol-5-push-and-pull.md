## Overview: The Two-Branch Model

The vault maintains two ref pointers at all times:

```
ref-pid-muw-{hmac}   ← named HEAD  (_namedHeadId)
    The shared/published state. Readable by anyone with the vault key.
    Advances ONLY on push(). Never touched by commit().

ref-pid-snw-{hmac}   ← clone HEAD  (_headCommitId)
    The local working state for this browser session.
    Advances on EVERY commit(). Created lazily on first commit.
```

The gap between these two pointers is the pending work:

```
named HEAD ──── commit A ──── commit B ──── commit C ──── clone HEAD
                                            ↑ ahead = 3 commits
```

`push()` collapses this gap by writing named HEAD → clone HEAD (1 PUT).  
`pull()` collapses it in the other direction: clone HEAD → named HEAD (1 GET + 1 PUT + tree reload).

This two-branch design means:
- Local edits are saved immediately (every `commit()`) without requiring write access to the shared state
- The shared named ref is only updated when the user explicitly chooses to publish
- Multiple browser sessions can work independently and push when ready
- A read-only open never touches either ref (the clone ref is never created, named ref is never written)

## 1. `push()`

Publishes local commits to the shared named branch. Fast-forward only — no merge logic.

```javascript
async push() {
    if (!this._cloneRefFileId) throw new Error('No clone branch initialised')
    if (!this.writable)        throw new Error('Read-only: no write key')

    await this._refManager.writeRef(this._refFileId, this._headCommitId)
    this._namedHeadId = this._headCommitId
}
```

### What `writeRef()` does

```javascript
// sg-vault-ref-manager.js
async writeRef(refFileId, commitId) {
    const filePath  = `bare/refs/${refFileId}`
    const payload   = new TextEncoder().encode(JSON.stringify({ commit_id: commitId }))
    const encrypted = await SGSendCrypto.encrypt(payload, this._readKey)
    await this._sgSend.vaultWrite(this._vaultId, filePath, this._writeKey, new Uint8Array(encrypted))
}
```

HTTP:
```
PUT /api/vault/write/{vaultId}/bare/refs/{refFileId}
Headers: x-sgraph-vault-write-key: {write_key hex}
Body: [12-byte IV][AES-GCM({"commit_id":"obj-cas-imm-..."})] [16-byte tag]
```

### HTTP calls for `push()`

| # | Request | What it writes |
|---|---|---|
| 1 | `PUT .../bare/refs/{refFileId}` | Named ref → clone HEAD commit ID |

**One PUT. That is the entire push.** All the commit and tree objects already exist on the server (they were written by `_commit()` calls). Push only moves the pointer.

### Before and after push

```
Before:
  named HEAD → commit A  (shared with all readers)
  clone HEAD → commit C  (local — 3 ahead)

After push():
  named HEAD → commit C  (same commit C that clone HEAD already points to)
  clone HEAD → commit C  (unchanged)
  _namedHeadId = _headCommitId  (ahead count drops to 0)
```

### Pre-conditions

`push()` throws if:
- `_cloneRefFileId` is null — clone branch was never initialised (vault opened read-only and never committed)
- `!this.writable` — no `write_key` (read-only share token or read-only vault key)

`push()` does **not** check whether the named ref on the server has advanced since the last open. If another client pushed while this session was open, this `push()` will silently overwrite their work. The conflict-detection responsibility lies with `getBehindCount()` — the UI should call that before presenting a push button and warn if behind > 0.

## 2. `pull()`

Fetches new commits pushed by another client and rebuilds the in-memory tree. Fast-forward only.

```javascript
async pull() {
    if (!this.writable) throw new Error('Read-only: no write key')

    // Reject if clone has unpushed commits — would require a merge
    if (this._headCommitId !== this._namedHeadId) {
        throw new Error('Cannot pull: you have unpushed local commits. Push first, then pull.')
    }

    // Fetch the live named HEAD (always bypasses cache — refs can change)
    const serverNamedHead = await this._refManager.readRef(this._refFileId)
    if (!serverNamedHead || serverNamedHead === this._headCommitId) return false  // already up to date

    // Fast-forward in-memory pointers
    this._namedHeadId  = serverNamedHead
    this._headCommitId = serverNamedHead

    // Advance clone ref on server to match
    await this._refManager.writeRef(this._cloneRefFileId, serverNamedHead)

    // Rebuild in-memory tree from new HEAD
    await this._loadTreeFromCommit(serverNamedHead)

    return true  // caller should re-render the file tree
}
```

### Pre-condition: no pending local commits

`pull()` refuses to run if `_headCommitId !== _namedHeadId`. This means the session has committed locally but not yet pushed. A pull in that state would require a merge (the local commit graph and the incoming commit graph have diverged). The web UI does not implement merge — it only supports fast-forward.

The user-facing flow enforced by the UI:
```
if (aheadCount > 0)  → push first, then pull
if (behindCount > 0) → pull is safe (no local commits)
```

### Step-by-step

**Step 1: Read named HEAD live** (1 GET)

```
GET /api/vault/read/{vaultId}/bare/refs/{refFileId}
→ [IV][AES-GCM({"commit_id":"obj-cas-imm-..."})] [tag]
   decrypt → { commit_id: "obj-cas-imm-{newHead}" }
```

This request always hits the network — refs are never served from the Cache API. This is how the client detects that another session has pushed new commits.

If the result equals the current `_headCommitId`, the vault is already up to date → return `false`.

**Step 2: Advance clone ref on server** (1 PUT)

```
PUT /api/vault/write/{vaultId}/bare/refs/{cloneRefFileId}
Body: [IV][AES-GCM({"commit_id":"obj-cas-imm-{serverNamedHead}"})] [tag]
```

The clone ref is advanced to match the new named HEAD. This ensures subsequent `open()` calls start from the correct point and `getAheadCount()` returns 0.

**Step 3: Rebuild in-memory tree** (`_loadTreeFromCommit`)

```javascript
async _loadTreeFromCommit(commitId) {
    const commit = await this._commitManager.loadCommit(commitId)
    // → GET .../bare/data/{commitId}  (likely cached if recently committed by another session)

    const tree = await this._commitManager.loadTree(commit.tree_id)
    // → GET .../bare/data/{commit.tree_id}  (may be cached)

    this._tree     = { '/': { type: 'folder', children: {} } }
    this._settings = null

    for (const entry of tree.entries) {
        if (entry.name === '.vault-settings') {
            const settingsBlob = await this._objectStore.load(entry.blob_id)
            const decrypted    = await SGSendCrypto.decrypt(settingsBlob, this._readKey)
            this._settings     = JSON.parse(new TextDecoder().decode(decrypted))
        } else if (entry.tree_id) {
            // Sub-folder: lazy stub — NOT fetched yet
            this._tree['/'].children[entry.name] = {
                type: 'folder', children: {}, _tree_id: entry.tree_id, _loaded: false
            }
        } else {
            this._insertEntry(entry)  // root-level file
        }
    }
}
```

This is identical to the `open()` step 4–6 sequence. The root tree is loaded and decrypted; sub-folders become lazy stubs. The entire in-memory tree is replaced — any changes the user was browsing are discarded and replaced with the incoming state.

### HTTP calls for a typical `pull()`

| # | Request | Notes |
|---|---|---|
| 1 | `GET .../bare/refs/{refFileId}` | Always live — detects incoming commits |
| 2 | `PUT .../bare/refs/{cloneRefFileId}` | Advance clone ref to new HEAD |
| 3 | `GET .../bare/data/{newCommitId}` | Load incoming commit (may be cached) |
| 4 | `GET .../bare/data/{newTreeId}` | Load incoming root tree (may be cached) |
| 5 | `GET .../bare/data/{settingsBlobId}` | Load vault settings (may be cached) |

**Total: 2 network-required (ref GET + ref PUT) + up to 3 object GETs (likely cached).**

If the incoming commits were made by another web-UI session on the same device (same Cache API), requests 3–5 will hit the Cache API and cost zero network. If coming from the CLI (`sgit push`), the commit and tree objects are new and will require network fetches.

## 3. `getAheadCount()` — Commits Ahead of Named Branch

```javascript
async getAheadCount() {
    if (!this._namedHeadId || this._headCommitId === this._namedHeadId) return 0

    let count  = 0
    let cursor = this._headCommitId
    const max  = 100   // safety cap — avoids runaway on deep chains

    while (cursor && cursor !== this._namedHeadId && count < max) {
        count++
        const commit = await this._commitManager.loadCommit(cursor)
        cursor = commit.parents && commit.parents[0] ? commit.parents[0] : null
    }
    return count
}
```

Walks the commit graph from clone HEAD back toward named HEAD, counting hops. Stops when it reaches `_namedHeadId` or hits the cap.

### HTTP calls

One `GET .../bare/data/{commitId}` per hop in the chain, all likely cached (the session just created these commits). In the typical case (1–5 local commits), this is 1–5 GETs with zero network cost.

The `max = 100` cap means the method returns at most 100 even if the actual ahead count is larger. This prevents infinite loops on corrupted commit chains.

## 4. `getBehindCount()` — Commits Available to Pull

```javascript
async getBehindCount() {
    // Always fetch named HEAD live — this is how we detect remote changes
    const serverNamedHead = await this._refManager.readRef(this._refFileId)
    if (!serverNamedHead || serverNamedHead === this._namedHeadId) return 0

    let count  = 0
    let cursor = serverNamedHead
    const max  = 100

    while (cursor && cursor !== this._namedHeadId && count < max) {
        count++
        const commit = await this._commitManager.loadCommit(cursor)
        cursor = commit.parents && commit.parents[0] ? commit.parents[0] : null
    }
    return count
}
```

### Key difference from `getAheadCount()`

`getAheadCount()` uses `_headCommitId` and `_namedHeadId` — both are in memory, no network needed for the starting point.

`getBehindCount()` starts with a live GET of the named ref — this is the only way to detect that another client has pushed since this session opened. Without this live fetch, the session would never know it was behind.

### HTTP calls

1. `GET .../bare/refs/{refFileId}` — always live (the detection step)
2. N × `GET .../bare/data/{commitId}` — one per incoming commit, these are new and not cached

If behind count = 0 (vault is current), only the single ref GET happens. If behind, each incoming commit object must be fetched from the network (they're new, not in cache). This is acceptable — you need to walk the graph to count the hops.

## 5. Conflict Model: Fast-Forward Only

The web UI implements a strict fast-forward model. There are no three-way merges, no conflict resolution UI, and no rebasing.

### The two scenarios

**Scenario A — Clean fast-forward (push then pull)**
```
Session 1 commits A, B, C → pushes → named HEAD = C
Session 2 opens → named HEAD = C → works from C
Session 2 commits D, E → pushes → named HEAD = E  ✓
```

**Scenario B — Diverged (simultaneous edits)**
```
Session 1: named HEAD = X, makes local commits Y1, Y2
Session 2: named HEAD = X, makes local commits Z1, Z2, pushes → named HEAD = Z2

Session 1 tries to push:
  → named HEAD on server is Z2 (not X)
  → push() overwrites Z2 with Y2  ← SILENT DATA LOSS (current behaviour)
```

The current implementation does not check for divergence before pushing. The UI layer is responsible for calling `getBehindCount()` before showing the push button and blocking or warning if behind > 0.

**Scenario C — Local commits blocking pull**
```
Session has: namedHead = X, cloneHead = Y (ahead by 1)
Another client pushes Z to named HEAD

pull() → throws: "Cannot pull: you have unpushed local commits"
→ user must push Y first, then the next pull gets both Y and Z is a conflict
   (which pull() also can't handle — push first would push Y on top of Z's parent X,
    overwriting Z in the process)
```

The fully correct multi-client flow for diverged sessions would require a merge or rebase — both of which are future features (CLI-side, not web UI).

**For the web UI, the safe multi-client pattern is:**
```
1. Open vault
2. Make changes
3. getBehindCount() == 0 → push()
   OR
   getBehindCount() > 0 → warn user, do not push (or pull first if aheadCount == 0)
```

## 6. Full Lifecycle Example: Two Clients, Sequential Edits

```
── Initial state ──
  Server:   named HEAD → C0  (initial commit)
  Client A: open() → headCommitId = C0, namedHeadId = C0, ahead = 0

── Client A edits ──
  A: addFile('/','hello.txt', ...) → commit C1 → clone HEAD = C1
  A: addFile('/','world.txt', ...) → commit C2 → clone HEAD = C2
  A: getAheadCount() → 2  (C1, C2 walk back to C0 = namedHeadId)
  A: push() → PUT named ref → C2 → namedHeadId = C2

── Client B opens ──
  B: open() → reads named ref → C2, reads clone ref (404) → headCommitId = C2
  B: getBehindCount() → reads named ref live → C2 = namedHeadId → 0
  B: addFile('/','notes.md', ...) → commit C3 → clone HEAD = C3
  B: getAheadCount() → 1  (C3 walks back to C2 = namedHeadId)
  B: push() → PUT named ref → C3

── Client A syncs ──
  A: getBehindCount() → reads named ref live → C3 ≠ namedHeadId (C2) → walk → 1
  A: aheadCount == 0 (A already pushed C2) → safe to pull
  A: pull():
       GET named ref → C3
       PUT clone ref → C3
       loadTreeFromCommit(C3) → GET commit C3, GET tree, GET settings
       rebuild in-memory tree → all 3 files visible
       _namedHeadId = _headCommitId = C3
       return true → UI re-renders file list
```

## 7. Summary of All Push/Pull Network Calls

| Operation | Minimum GETs | Minimum PUTs | Total minimum |
|---|---|---|---|
| `push()` | 0 | 1 (named ref) | 1 |
| `pull()` (up to date) | 1 (named ref check) | 0 | 1 |
| `pull()` (N commits behind) | 1 + N + 2 (commit + tree + settings) | 1 (clone ref) | N + 4 |
| `getAheadCount()` (up to date) | 0 | 0 | 0 |
| `getAheadCount()` (N ahead, cached) | 0 (all cached) | 0 | 0 |
| `getBehindCount()` (up to date) | 1 (named ref live) | 0 | 1 |
| `getBehindCount()` (N behind) | 1 + N | 0 | N + 1 |

`push()` is a single PUT with no preceding GETs. It is the fastest write operation in the entire protocol — faster than any individual `_commit()`.
