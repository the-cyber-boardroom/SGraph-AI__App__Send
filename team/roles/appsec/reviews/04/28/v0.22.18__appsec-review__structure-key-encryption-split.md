# AppSec Review: Structure Key Encryption Split

**Version:** v0.22.18 | **Date:** 28 April 2026 | **Role:** AppSec (Librarian session)
**Source:** `team/humans/dinis_cruz/briefs/04/28/v0.22.17__architect-review__structure-key-encryption-split.md`
**Status:** Design approved — three implementation risks flagged

---

## Security Assessment Summary

The structure key encryption split design is **cryptographically sound** and **correctly specified**. The backwards-compatibility strategy using `InvalidTag` / `OperationError` as a fallback signal is a well-established pattern for AES-GCM migrations.

Three implementation risks are flagged below. None are design flaws — they are correctness risks in the implementation that must be verified before shipping.

---

## 1. Design Approval

### 1.1 Threat Model Alignment

The design correctly separates the information a `structure_key` holder can learn:

| Can learn | Cannot learn |
|-----------|-------------|
| Commit graph (timestamps, parents, tree IDs) | Filenames |
| Branch names, types, HEAD positions | File sizes |
| Directory topology (which object IDs exist) | Content hashes |
| Public PKI keys | MIME types |
| Ref file contents | Commit messages |
| — | File content |
| — | `read_key`, `write_key`, passphrase |

This achieves a clean **structural-only** access tier that is useful for audit log viewers, commit history renderers, and public topology displays without exposing sensitive content.

### 1.2 Key Hierarchy Correctness

The one-way chain `passphrase → read_key → structure_key` is correct:
- `structure_key` cannot be reversed to `read_key` (HKDF is one-way)
- `read_key` cannot be reversed to `passphrase` (PBKDF2 600k iterations)
- Sharing `structure_key` therefore exposes **strictly less** than sharing `read_key`

The `structure_key` derivation info string `b'sg-vault-v1:structure-key'` is distinct from all other derivation info strings, ensuring no accidental key collision.

### 1.3 Backwards Compatibility Mechanism

Using `InvalidTag` (authenticated encryption failure) as the fallback signal is correct because:
1. The wrong key never produces valid-looking plaintext — `InvalidTag` is raised without returning data
2. AES-256-GCM with 96-bit nonce (the standard configuration used here) provides 128-bit authentication strength
3. The probability of a false positive (wrong key accidentally produces a valid tag on real ciphertext) is negligible (2⁻¹²⁸)

**Verdict: The fallback is cryptographically safe. It is not a timing oracle or a padding oracle.**

## 2. Security Risks in Implementation

### Risk 1: `OperationError` Catch Scope Too Broad (Medium)

**Issue:** Web Crypto API raises `OperationError` for multiple failure modes, not just AES-GCM authentication failure. If the catch block is written as:

```javascript
catch (e) {
    if (e.name === 'OperationError') { ... }
}
```

...it may catch and swallow `OperationError` raised by invalid algorithm parameters, key import failures, or other programming errors, silently falling back to `readKey` when the error was not an authentication failure but a bug.

**Mitigation required:**
```javascript
catch (e) {
    if (e instanceof DOMException && e.name === 'OperationError') {
        return await SGVaultCrypto.decrypt(readKey, data)
    }
    throw e
}
```

Add a brief comment explaining that `OperationError` is specifically AES-GCM authentication failure in this context.

**Action:** Verify this guard is in place before merging. Code review checklist item.

### Risk 2: `structureKey` Must Not Be Extractable (Medium)

**Issue:** The `structureKey` CryptoKey object must be created with `extractable: false`:

```javascript
const structureKey = await crypto.subtle.importKey(
    'raw', structureKeyBits,
    { name: 'AES-GCM' },
    false,   // extractable: false — non-negotiable
    ['encrypt', 'decrypt']
)
```

If `extractable: true` is used by mistake, the key can be exported via `crypto.subtle.exportKey()` and exfiltrated from memory. The existing `readKey` and `writeKey` are presumably already non-extractable — `structureKey` must follow the same rule.

**Action:** Verify `extractable: false` in code review.

### Risk 3: Raw Key Bytes Must Not Be Stored in JavaScript State (Medium)

**Issue:** The derivation sequence is:

1. `PBKDF2(passphrase, ...)` → raw `readKeyBits` (ArrayBuffer)
2. `HKDF(readKeyBits, info='sg-vault-v1:structure-key')` → raw `structureKeyBits` (ArrayBuffer)
3. `importKey('raw', structureKeyBits, ...)` → non-extractable `CryptoKey`

The raw `structureKeyBits` ArrayBuffer at step 2 is the actual key material. It must not be stored in any JavaScript object property or closure after `importKey` has consumed it. The `structureKeyBits` variable should be local to the derivation function and should not be kept in `sg-vault.js` state.

Only the non-extractable `CryptoKey` objects (`readKey`, `writeKey`, `structureKey`) should be stored as instance variables.

**Action:** Review `sg-vault.js` state management during implementation to ensure raw bytes are not retained.

## 3. Named Branch Private Key — Related Issue

The architect review identifies a related concern: the named branch private key is currently encrypted with `read_key` and stored server-side. Every clone downloads it. This means sharing `read_key` exposes the private signing key.

**Current impact: low.** Commit signature verification exists but is never enforced. The private key is not currently usable as an access gate.

**AppSec note:** When signature verification is activated, this must be remediated. Re-encrypting with `write_key` (as recommended in the architect review) is the correct fix. The `InvalidTag` fallback applies identically.

**Recommendation:** Create a tracked issue for this remediation. Tag it "blocked on signature verification activation." Do not let it slip through without a tracking record.

---

## 4. What This Change Does NOT Address

The following are **out of scope** for this change and are noted for completeness:

- **Server-side key knowledge**: The server still stores opaque ciphertext and has no key knowledge regardless of this change. This is correct and unchanged.
- **Key exfiltration via XSS**: XSS in the vault UI could exfiltrate `_readKey`, `_writeKey`, or `_structureKey` from `sg-vault.js` state. This is a pre-existing risk not introduced by this change.
- **Nonce reuse**: AES-GCM nonce generation must remain random (not counter-based) to prevent nonce reuse. This change does not affect nonce generation.

## 5. Pre-Ship Security Checklist

Before merging the Vault Web Team implementation:

- [ ] `decryptStructural` catches `DOMException { name: 'OperationError' }` only (not generic `Error`)
- [ ] `structureKey` created with `extractable: false`
- [ ] Raw `structureKeyBits` ArrayBuffer not stored in any instance variable or closure
- [ ] Simple token path (`word-word-NNNN`) also derives `structureKey` (both derivation paths updated)
- [ ] Existing vault tests all pass with old-format vaults (fallback path exercised)
- [ ] Mixed vault fixture test passes (confirms per-object fallback, not per-vault)
- [ ] `openWithStructureKey` (if implemented) verifies attempt to read `name_enc` or blob content throws — not silently returns garbage
