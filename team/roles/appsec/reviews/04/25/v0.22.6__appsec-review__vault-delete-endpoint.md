# AppSec Review: Vault Delete Endpoint

**Version:** v0.22.6 | **Date:** 2026-04-25 | **Role:** AppSec (Explorer Team)
**Branch:** `claude/review-docker-debrief-ZCD2a`
**Brief:** API Brief — Vault Delete Endpoint (CLI team, 25 Apr 2026)
**Risk Rating:** MEDIUM
**Recommendation:** Approve with conditions — three findings must be addressed before implementation; two are advisory

---

## Threat Model

**What this endpoint does:** Destroys all server-side files belonging to a vault — encrypted blobs, refs, key material, branch indexes — permanently and without recovery.

**Who the attackers are:**

| Attacker | Goal | Capability |
|----------|------|------------|
| **Unauthenticated attacker** | Delete arbitrary vaults | Can send requests with guessed vault_ids and random write keys |
| **Write-key holder** (legitimate user) | Accidentally delete own vault | No malicious intent; human error |
| **Write-key holder** (adversary who stole write_key) | Destroy victim's vault | Has valid write_key via theft or leak |
| **Vault-id oracle attacker** | Enumerate existing vault_ids | Sends requests and reads response codes |

**Blast radius of successful attack:** Complete and permanent data destruction for the targeted vault. Since the server stores only encrypted ciphertext, deletion is the maximum possible harm the server can cause to a vault owner (no plaintext is ever at risk). There is no worse outcome for data availability.

**What is NOT at risk:** Confidentiality. The server cannot decrypt any vault content regardless of this endpoint. A successful vault deletion exposes nothing to the attacker; it only destroys the victim's access to their own data.

## Finding 1: Vault Existence Oracle via Response Code

**Severity:** Medium

`Service__Vault__Pointer._check_vault_write_key()` (line 56–59) has two distinct code paths:

```python
def _check_vault_write_key(self, vault_id, submitted_hash):
    manifest = self._load_manifest(vault_id)
    if manifest is None:
        return True          # No manifest yet — first write creates it
    return manifest.get('write_key_hash') == submitted_hash
```

- **Vault exists, wrong write_key** → `_check_vault_write_key` returns `False` → route returns **403**
- **Vault does not exist** → `_check_vault_write_key` returns `True` → delete runs with no files to remove → route returns **200** with `files_deleted: 0`

This means an attacker who sends `DELETE /api/vault/destroy/{vault_id}` with any write_key can determine whether a vault_id exists on the server: a 403 means the vault exists (and the key is wrong); a 200/0 means the vault does not exist or was already deleted.

**Practical impact is limited** because vault_ids are 8–24 lowercase alphanumeric characters drawn from a large space (36^8 = ~2.8 billion minimum). Brute-force enumeration is not feasible in practice. However, the oracle allows a targeted attacker who already knows a candidate vault_id (e.g. via other means) to confirm its existence before attempting other attacks.

**The same oracle already exists** on `PUT /api/vault/write/{vault_id}/{file_id}` — a first write to any vault_id is accepted. The delete endpoint does not make this meaningfully worse.

**Recommendation:** Document the oracle in a code comment. Do not change the auth logic to add a vault-existence pre-check — that would add a storage round-trip and complexity without closing the oracle (an attacker can also probe via `GET /api/vault/health/{vault_id}`). Accept as a known, low-severity limitation of the stateless auth model.

## Finding 2: IP Logging Conflicts with Zero-Knowledge Design

**Severity:** Medium (policy conflict)

The brief specifies: "Log the deletion event (vault_id, timestamp, IP, files_deleted)."

The existing codebase does **not** log raw IPs. In `Routes__Transfers.complete__transfer_id`, IP is handled as:
```python
ip_hash = hashlib.sha256((request.client.host if request.client else '').encode()).hexdigest()
```

A raw IP in any log or audit record conflicts with the zero-knowledge design principle and with GDPR/DPA obligations (IP addresses are personal data under EU law in most interpretations).

**Requirement:** If deletion events are logged, IP must be stored as `ip_hash` (SHA-256 with daily salt matching the existing pattern), not as the raw IP string. The brief must be updated before implementation.

**Secondary issue:** The existing codebase has no structured logging infrastructure. `Routes__Vault__Pointer` and `Service__Vault__Pointer` have no `logger` instance. Implementing deletion logging requires adding a logging layer that does not exist yet. Recommend descoping logging from the initial implementation and treating it as a separate task.

## Finding 3: write_key Adequacy as a Deletion Credential

**Severity:** Low (design consideration, not a flaw)

The brief proposes reusing the existing `write_key` (HKDF-SHA256 derivative of the vault master key) as the deletion credential. This is the same key used for individual file writes and deletes. The question is whether this is appropriate for a **destructive delete-all** operation.

**Arguments for (current proposal):**
- Consistent with the existing write model — no new credential type
- Vault owner already has full destructive power via the existing batch delete endpoint (they can delete all files one-by-one)
- CLI tooling (`sgit`) already manages the write_key; no new key material to distribute

**Arguments against:**
- A write_key compromise (e.g. key logged accidentally, found in CLI history) already grants full write access. Adding delete-vault does not materially expand blast radius — the attacker already has full write capability.
- The brief notes "no admin bypass — even internal tooling must supply a valid write_key", which is correct.

**Verdict:** The write_key as deletion credential is acceptable given the zero-knowledge model. The server cannot enforce a higher-privilege credential without knowing the vault master key, which it should never hold. Accept as designed.

## Finding 4: Timing Side-Channel (Vault Existence)

**Severity:** Low

The response time profile differs across vault states:

| State | Timing | Reason |
|-------|--------|--------|
| Valid access token + vault exists + correct key | Slow | S3 manifest load + file walk + deletes |
| Valid access token + vault exists + wrong key | Medium | S3 manifest load only |
| Valid access token + vault does not exist | Fast | `_check_vault_write_key` returns True without S3 call (cache miss → no manifest → no load) |
| Invalid access token | Fast | Early exit before any vault logic |

An attacker can use timing differences to distinguish "vault exists, wrong key" from "vault does not exist" — confirming the existence oracle in Finding 1 via a second channel.

**Recommendation:** Same as Finding 1 — document as a known limitation. Closing this would require constant-time manifest loading (always hitting storage even for absent vaults), which adds cost and complexity disproportionate to the threat.

## Finding 5: Double Confirmation (body + URL) — Security Value Assessment

**Severity:** Info

The brief requires `vault_id` in both the URL path and the request body, returning 409 on mismatch. This is a **UX safety control, not a security control**:

- Both the path and the body are attacker-controlled; an attacker can trivially supply matching values.
- The 409 indicates a client programming error (mis-wired CLI), not a security boundary.
- It does not prevent accidental deletion (if the caller has the wrong vault_id, they can supply it in both fields).

**What it does protect against:** A CLI or script that constructs the JSON body from a different source than the URL (e.g. a stale config file). This is a legitimate defensive pattern for destructive, irreversible operations.

**Verdict:** Keep it. Document clearly in the implementation that 409 is a client error signal; the CLI must never surface it as a user-facing error.

---

## Finding 6: Partial Deletion Response Does Not Leak Sensitive Information

**Severity:** Info (no action required)

The brief proposes returning `{"files_deleted": N}` in a 500 body after partial deletion. This reveals the count of files in a vault, which is metadata about vault size.

**Assessment:** Since all vault contents are encrypted ciphertext (the server does not know what the files contain or mean), the file count is structural metadata. Leaking it in a partial-deletion 500 response is a negligible risk — the file count is also observable from `GET /api/vault/list/{vault_id}` (which is unauthenticated for read-only access). This is not a new disclosure.

**Verdict:** Accept the proposed response shape.

## Additional Security Requirements Not in the Brief

**Must add:**

1. **`check_access_token` before write_key check** — the Architect review identifies this gap. All existing write-gated vault routes call `check_access_token(request)` before the write-key check. The delete-vault endpoint must do the same. Without this, anyone with a guessed write_key can delete a vault even if the platform's access token system is in effect.

2. **Manifest cache invalidation after deletion** — `_manifest_cache` is a Lambda-lifetime dict. After successful vault deletion, `self._manifest_cache.pop(vault_id, None)` must be called. Failure to do so allows a subsequent write to the same vault_id (within the same Lambda instance) to be accepted using the old write key hash, silently recreating the vault. This is both a correctness and a security bug. (Also identified by Architect — reinforcing it here as security-critical.)

**Advisory:**

3. **No manifest = 200/0 for non-existent vault** — the current `_check_vault_write_key` behaviour means a delete request against a non-existent vault always succeeds (returns True). Implementing a proper "vault does not exist → return 200/files_deleted:0" path requires no code change — it falls out naturally from walking an empty file list. This is acceptable and documented in Finding 1.

---

## Summary

The vault delete endpoint brief is well-conceived and the auth model reuse is sound — the existing write_key mechanism is the correct credential for this operation, requiring no new auth infrastructure. The zero-knowledge design means deletion is the worst outcome (data destruction, no confidentiality breach), which the brief acknowledges correctly. Two findings require action before implementation: (1) raw IP in deletion logs must become `ip_hash` per existing project pattern, and (2) logging should be descoped from the initial implementation since no logging infrastructure exists. Two non-negotiable implementation requirements not in the brief must be added: `check_access_token` before write_key validation, and `_manifest_cache` invalidation after deletion. Rate limiting should be descoped from v1 — it cannot be implemented correctly in the stateless Lambda stack without adding external state, and the endpoint does not expand the existing attack surface materially. The path traversal guard (`VAULT_ID_PATTERN`) is correctly strict and requires no changes.
