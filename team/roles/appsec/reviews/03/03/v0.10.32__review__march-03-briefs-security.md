## Executive Summary

This batch of five briefs represents the most architecturally significant expansion since the PKI and data room briefs. The proposals span two categories: (1) immediate product improvements to the working vault (auth flow change, settings panel, inline editing, Chrome sync, gallery viewer) and (2) a major new architectural layer (file safety orchestration with scanning pods, cross-domain security, cryptographic attestation chains).

**Overall security posture: The file safety architecture is exceptionally well-designed from a trust-boundary perspective. The vault and gallery proposals introduce specific, addressable risks that must be mitigated before implementation.**

The file safety architecture brief is the strongest security-first design document this project has produced. The trust model — sender explicitly untrusted, pod as the only cleartext-bearing component in an isolated network, crossing points that never see plaintext, cryptographic attestation verifiable by the receiver alone — is sound. The attack surface analysis table (Section "What the Sender Cannot Do") correctly identifies and blocks the six most obvious attacks. I have three concerns with the architecture and several more with the other briefs.

**Key findings (18 total):**

| Severity | Count | Summary |
|----------|-------|---------|
| HIGH | 3 | Vault auth flow downgrade, CLI keyfile on disk, pod single-point-of-trust |
| MEDIUM | 8 | Crypto interop divergence, gallery resource exhaustion, Chrome sync key exposure, inline editing data loss, EXIF metadata leakage, gallery XSS vectors, scanning partner data retention, attestation replay window |
| LOW | 4 | Gallery thumbnail timing side-channel, settings panel key display, local storage viewer export, conflict file naming |

## 1. Vault Auth Flow Change — Token Only for Upload

**Brief:** `v0.10.30__dev-brief__vault-fixes-and-ux.md`, Item 1
**Severity: HIGH**

### 1.1 Current vs. Proposed Model

| | Current | Proposed |
|---|---|---|
| Open vault | Requires auth token + vault key | Requires vault key only |
| Browse/read/download | Auth token present | No auth token needed |
| Upload/write | Auth token present | Auth token requested on demand |

### 1.2 Security Analysis

**FINDING VF-01 (HIGH): Auth token removal from browse path creates a two-tier access model where the vault key alone grants full read access, with no server-side visibility into who is reading.**

The auth token currently serves two functions:
1. **Upload gating** — prevents writes without a valid token
2. **Access logging** — every vault operation involves a token, creating a server-side audit signal

Removing the token from the read path eliminates function (2) for all read operations. The security implications:

| Concern | Assessment |
|---|---|
| **Who can read the vault?** | Anyone with the vault key. This is already true today (the vault key is the decryption key), but today the server sees a token validation call on every access. Under the new model, the server sees nothing for read-only access. |
| **Can the server detect unauthorized readers?** | No. A reader with the vault key but no auth token leaves zero server-side footprint. The vault contents are fetched, decrypted client-side, and the server has no way to distinguish legitimate readers from unauthorized key holders. |
| **Does this weaken the zero-knowledge property?** | Paradoxically, it strengthens it. Fewer server-side signals = less metadata. But it trades security visibility for privacy. |
| **Key revocation problem** | If a vault key is shared with someone and later the owner wants to revoke their access, there is no mechanism. The key is symmetric — once shared, it works forever. The auth token was the only revocation lever for read access. Removing it from the read path removes the only revocation mechanism for readers. |
| **Vault key = god key for reading** | The vault key becomes a permanent, irrevocable read credential. Anyone who ever obtains the vault key has permanent read access to the vault and all its contents, past and future, with no server-side detection possible. |

**Recommendation:**

The proposed model is acceptable IF the following conditions are documented and accepted:

1. **Document the trust model explicitly:** "Sharing the vault key grants permanent, irrevocable, undetectable read access to the entire vault."
2. **Consider a vault key rotation mechanism:** When a key holder should lose access, the owner generates a new vault key, re-encrypts the vault, and shares the new key with remaining authorized readers. This is expensive but is the only way to revoke read access in a symmetric-key system.
3. **Consider optional read tokens:** Keep the current model as an option. A vault could be configured as "key-only read" (the proposed mode) or "key + token read" (the current mode). This gives vault owners a choice between convenience and access control.
4. **Add client-side access logging:** Even without server-side token validation, the client could submit an encrypted access log entry when opening a vault. This provides audit signal without requiring a token.

### 1.3 Positive Observations

The brief's instinct is correct: read-only access should be frictionless. For the "share a vault with someone to view" use case, requiring both a vault key AND an auth token is over-engineered. The key IS the authentication for reading — that is how symmetric encryption works. The proposed model aligns the UX with the underlying cryptographic reality.

## 2. Vault CLI and Crypto Interop — Python/JavaScript Compatibility

**Brief:** `v0.10.30__dev-brief__vault-cli-local-sync.md`
**Severity: HIGH (keyfile storage), MEDIUM (crypto interop)**

### 2.1 Crypto Interop Risk

**FINDING VC-01 (MEDIUM): Python/JavaScript PBKDF2 and AES-256-GCM interop has multiple subtle divergence points that can produce silent decryption failures or, worse, silently different derived keys.**

The brief states the Python CLI must produce ciphertext that JavaScript can decrypt, and vice versa. The existing JavaScript implementation (from `sg-send-crypto.js`) uses:

| Parameter | JavaScript Value | Must Match in Python |
|---|---|---|
| Algorithm | AES-GCM | AES-256-GCM (via `AESGCM`) |
| Key length | 256 bits | 32 bytes |
| IV length | 12 bytes | 12 bytes |
| KDF | PBKDF2 | PBKDF2HMAC |
| KDF hash | SHA-256 | hashes.SHA256() |
| KDF iterations | 600,000 | 600,000 |
| Salt | `TextEncoder().encode(salt)` (UTF-8 string) | `vault_key.encode()` (UTF-8 string) |
| Wire format | `[12-byte IV][ciphertext + 16-byte auth tag]` | Must match exactly |

**Specific interop failure modes:**

| Failure Mode | How It Happens | Consequence |
|---|---|---|
| **Salt encoding mismatch** | JS uses `TextEncoder().encode(salt)` which produces UTF-8 bytes. The brief's Python code shows `vault_key.encode()` but the JS passes a separate `salt` parameter to `deriveKey()`. If the CLI uses a different salt value (e.g., the vault ID vs. the vault key itself), different derived keys result. | **Silent failure.** Decryption produces garbage or throws "decryption failed." No data corruption, but files become unreadable across platforms. |
| **Iteration count mismatch** | The brief's example Python code shows `iterations=100000`. The JS implementation uses `600000`. | **Silent failure.** Different derived keys. This is the most likely bug because the brief's code sample is wrong. |
| **Auth tag handling** | Web Crypto API appends the 16-byte GCM auth tag to the ciphertext. Python's `AESGCM.encrypt()` also appends the tag. But some Python libraries separate the tag. If the Python implementation uses a library that returns `(ciphertext, tag)` separately and concatenates them in the wrong order, decryption fails. | **Decryption failure.** Detectable but confusing. |
| **Nonce/IV concatenation order** | JS wire format: `[IV][ciphertext+tag]`. The brief's Python code: `nonce + ciphertext` where `ciphertext` includes the tag. If the Python code puts the tag before the nonce, or uses a different concatenation, wire formats diverge. | **Decryption failure.** Detectable. |
| **String encoding edge cases** | Vault key passphrase containing non-ASCII characters (e.g., accented characters, emoji). JS `TextEncoder` produces UTF-8. Python `str.encode()` defaults to UTF-8 but the behaviour with surrogate pairs differs. | **Silent failure** for affected passphrases. Works fine for ASCII-only keys. |
| **Additional authenticated data (AAD)** | JS code passes no AAD (`null` in the `encrypt` call). The brief's Python code also passes no AAD (`None`). But if either side later adds AAD without updating the other, interop breaks. | **Future regression risk.** |

**CRITICAL NOTE:** The brief's Python code sample uses `iterations=100000` but the JavaScript implementation uses `600000`. This is a 6x difference. If the CLI is implemented with the brief's sample code, it will produce a different derived key from the same passphrase, and cross-platform decryption will silently fail. This must be caught before any code is written.

**Recommendation:**

1. **Create a shared test vector file.** Define 5-10 test cases with known inputs (passphrase, salt, plaintext) and expected outputs (derived key bytes, IV, ciphertext). Both JS and Python implementations must produce identical outputs for the same inputs. This file is the interop contract.
2. **Fix the iteration count in the brief.** The Python code sample must use 600,000 iterations, not 100,000.
3. **Add cross-platform integration tests.** Python encrypts, JS decrypts. JS encrypts, Python decrypts. Automated, run in CI.
4. **Pin the wire format specification.** Document it as a versioned spec: `wire_format_v1: [12-byte-IV][ciphertext][16-byte-GCM-tag]`. Both implementations reference this spec. Any future changes require a version bump.
5. **Test with non-ASCII passphrases.** Include test vectors with accented characters, CJK characters, and emoji in the passphrase.

### 2.2 Keyfile on Disk

**FINDING VC-02 (HIGH): The `.sg_vault/keyfile` stores the vault key in plaintext on disk. Combined with the new auth model (VF-01), this file grants permanent, irrevocable, undetectable read access to the entire vault.**

The brief acknowledges this: "Storing the Vault Key locally is the same security posture as having the key in your browser's local storage." This is true but understates the risk:

| Storage | Threat Model |
|---|---|
| Browser localStorage | Accessible to same-origin JS only. Browser sandboxing protects it from other applications. Cleared when browser data is cleared. |
| `.sg_vault/keyfile` on disk | Accessible to any process running as the user. Readable by backup software, cloud sync, grep, find, malware. Persists indefinitely. Survives browser reinstall. |

The brief further suggests placing the vault folder inside Dropbox/iCloud/Google Drive, which means the keyfile travels to those cloud services. This is explicitly called out as desirable: "the keyfile travels with it -- which is fine, because the key is the access mechanism."

**Risk assessment:**

- The keyfile in Dropbox/iCloud means the vault key is now stored by **three parties**: the user's disk, the cloud sync service, and any other machine syncing that folder.
- A Dropbox/iCloud/Google Drive account compromise exposes the vault key, and with it, permanent read access to the vault contents.
- The vault key in plaintext on disk is searchable. An attacker who compromises any machine that syncs the folder can find all `.sg_vault` directories with a simple `find / -name ".sg_vault" -type d`.

**Recommendation:**

1. **Default to no keyfile.** Require manual entry of the vault key for each CLI session. Store it only in memory for the session duration.
2. **If keyfile is stored, encrypt it.** Use the OS keychain (macOS Keychain, Linux `secret-service`, Windows Credential Manager) via `keyring` Python library. Fall back to a passphrase-encrypted keyfile if no keychain is available.
3. **Add a `.gitignore` pattern.** The `.sg_vault` directory should be in the default `.gitignore` to prevent accidental commits.
4. **Warn on cloud sync detection.** If the CLI detects the vault folder is inside a known cloud sync directory (Dropbox, Google Drive, iCloud, OneDrive), warn the user: "The vault key will be synced to [service]. This means [service] can access your vault contents."

### 2.3 Manifest Security

**FINDING VC-03 (LOW): Conflict file naming with `.conflict` suffix could leak information about sync state.**

The `.conflict` suffix on locally created files is visible to anyone with file system access. In a shared environment, this reveals that two people are editing the same vault simultaneously, which is metadata about collaboration patterns.

**Severity:** LOW — this is a minor information disclosure in a local filesystem context.

**Recommendation:** Use opaque conflict markers (e.g., `.sg_conflict_a1b2c3`) rather than descriptive suffixes.

### 2.4 Positive Observations (INFO)

**FINDING VC-04 (INFO): The CLI brief's wire format matches the JavaScript implementation.**

The brief's Python code shows `nonce + ciphertext` (where ciphertext includes the GCM tag from `AESGCM.encrypt()`), and the JavaScript shows `[IV][ciphertext+tag]`. These are the same format. The Python `cryptography` library's `AESGCM.encrypt()` returns `ciphertext || tag`, and `AESGCM.decrypt()` expects `ciphertext || tag` — matching Web Crypto's behaviour. This is correct.

## 3. Encrypted Gallery Sharing — Resource Exhaustion and XSS

**Brief:** `v0.10.30__dev-brief__encrypted-gallery-sharing.md`
**Severity: MEDIUM (multiple findings)**

### 3.1 Client-Side Resource Exhaustion

**FINDING GS-01 (MEDIUM): A gallery of 100+ files at 5MB each (500MB total) creates a sustained, heavy client-side processing load that could be exploited for denial-of-service or used to mask malicious activity.**

The brief describes the processing pipeline:

```
Download 500MB encrypted zip
  -> Decrypt (AES-256, hardware-accelerated)
  -> Extract zip index
  -> Generate thumbnails on scroll (lazy)
  -> Extract full images on tap
```

**Resource exhaustion vectors:**

| Vector | Attack | Impact |
|---|---|---|
| **Memory exhaustion** | A crafted zip with 1000 entries, each 50MB when decompressed (zip bomb pattern), causes the browser to allocate tens of gigabytes of memory during thumbnail generation. | Browser tab crash, potential OS-level memory pressure. Mobile devices (2-4GB RAM) are particularly vulnerable. |
| **CPU exhaustion** | Thumbnail generation for 100+ high-resolution images (e.g., 8000x6000 from a DSLR) requires canvas downscaling. 100 images at 48 megapixels each = sustained CPU load. | Browser becomes unresponsive. Other tabs affected. Mobile devices heat up, battery drains. |
| **Decompression bomb** | A zip file with high compression ratio (e.g., 1MB compressed -> 10GB decompressed). The zip library extracts entries into memory. | Out-of-memory crash. The brief mentions "extract zip index" as instant, but extracting file contents is not. |
| **Disk exhaustion (via blob URLs)** | Each extracted image/video creates a blob URL. Blob URLs consume memory/disk until revoked. 100 blob URLs for large images = significant memory footprint. | Memory leak if blob URLs are not revoked when thumbnails scroll out of view. |

**Recommendation:**

1. **Validate zip contents before extraction.** Read the zip central directory first. Check: total uncompressed size < configurable limit (e.g., 2GB), no individual file > configurable limit (e.g., 200MB), entry count < configurable limit (e.g., 500 files). Reject zips that exceed limits with a clear error message.
2. **Implement a thumbnail generation budget.** Generate at most N thumbnails concurrently (e.g., 4). Use `requestIdleCallback` or a web worker to avoid blocking the main thread.
3. **Revoke blob URLs aggressively.** When a thumbnail scrolls out of the viewport, revoke its blob URL to free memory. Re-generate when it scrolls back in.
4. **Set a memory ceiling.** Track approximate memory usage (blob sizes + extracted data). Pause extraction when approaching a threshold. Display: "Showing first N of M images. Download remaining?"
5. **Test on low-end devices.** A 2GB RAM Android phone with Chrome is the worst case. Test with 100+ image galleries on such a device.

### 3.2 XSS via Image/Video Rendering

**FINDING GS-02 (MEDIUM): In-browser rendering of decrypted images and videos from untrusted sources creates XSS and code execution vectors.**

The gallery decrypts and renders files from a zip uploaded by a sender. The sender is not fully trusted. Attack vectors:

| Vector | How It Works | Impact |
|---|---|---|
| **SVG with embedded JavaScript** | Sender includes an SVG file in the gallery. SVG files can contain `<script>` tags, `onload` handlers, and `<foreignObject>` with HTML. If rendered as an `<img>` tag, scripts do NOT execute. If rendered via `innerHTML`, `<object>`, or `<embed>`, they DO. | Full XSS in the receiver's browser if SVG is rendered unsafely. |
| **Polyglot image files** | A file that is simultaneously a valid JPEG and a valid HTML/JS file. If the browser sniffs the content type incorrectly, it may execute the JS portion. | Code execution. Mitigated by `X-Content-Type-Options: nosniff` but the gallery generates blob URLs, which bypass server headers. |
| **Video with malicious subtitles** | A video file with embedded VTT/SRT subtitles containing `<script>` tags. If the video player renders subtitles as HTML, scripts execute. | XSS via subtitle injection. |
| **EXIF JavaScript injection** | EXIF metadata in JPEG files can contain arbitrary strings. If the gallery displays EXIF data (e.g., camera model, GPS coordinates) without sanitization, injected HTML/JS in EXIF fields could execute. | XSS via metadata display. |
| **Filename injection** | The gallery.json or zip entry filenames could contain HTML: `<img src=x onerror=alert(1)>.jpg`. If filenames are rendered without escaping, XSS results. | XSS via filename display. |

**Recommendation:**

1. **Render images ONLY via `<img>` tags with blob URLs.** Never use `<object>`, `<embed>`, `<iframe>`, or `innerHTML` for image rendering. The `<img>` tag does not execute scripts in SVGs.
2. **Set Content-Type explicitly on blob URLs.** When creating blob URLs, specify the MIME type: `new Blob([data], { type: 'image/jpeg' })`. Never use a generic type.
3. **Sanitize all displayed text.** Filenames, captions, EXIF data, gallery title, description — all must be HTML-escaped before rendering. Use `textContent` instead of `innerHTML` wherever possible.
4. **Validate file types in the gallery.** Only render known-safe types: JPEG, PNG, GIF, WebP, MP4, WebM. Reject SVG files in gallery context (or render them as `<img>` only, never inline).
5. **Render videos via `<video>` with `disableRemotePlayback` and without subtitle tracks from the zip.** If subtitle support is needed, parse and sanitize VTT/SRT before rendering.

### 3.3 EXIF Metadata Privacy Leakage

**FINDING GS-03 (MEDIUM): Gallery features that read EXIF data (sorting by date, map view by GPS, timeline view) extract and potentially display private metadata from the sender's photos.**

The brief proposes:
- EXIF date sorting (Phase 1)
- Map view from GPS coordinates (Phase 3)
- Timeline view (Phase 3)

This requires the client to read EXIF data from decrypted images. The EXIF data may contain:

| EXIF Field | Privacy Risk |
|---|---|
| GPS coordinates | Reveals where the photo was taken — home address, workplace, travel patterns |
| Camera serial number | Unique device identifier, linkable across galleries |
| Software version | Reveals the sender's device and OS version |
| Date/time | Reveals when the photo was taken (not just uploaded) |
| Thumbnail | Original thumbnail may not have been redacted even if the main image was edited |
| Author/copyright | Reveals the photographer's name |

**This is a client-side operation, so the server never sees EXIF data.** The zero-knowledge property is preserved. However, the RECIPIENT now has access to potentially sensitive metadata that the SENDER may not realize they are sharing.

**Recommendation:**

1. **Strip EXIF data by default before sharing.** When the sender uploads a gallery, offer an option: "Remove location data and camera information from photos before sharing?" Default: yes.
2. **If EXIF data is preserved, warn the sender.** "These photos contain location data. Recipients will be able to see where each photo was taken."
3. **Never display EXIF data without explicit user action.** The gallery grid should not auto-display GPS coordinates or camera info. Require the user to click "Show photo details."

### 3.4 Gallery Thumbnail Timing Side-Channel

**FINDING GS-04 (LOW): Lazy thumbnail generation leaks information about gallery contents via timing.**

When thumbnails are generated lazily as the user scrolls, the time to generate each thumbnail depends on the file size and type. An observer watching network traffic (which is encrypted) cannot see the content, but an observer with access to the recipient's device (e.g., a shared computer, screen-sharing) can infer file sizes from thumbnail generation delays — large files take longer to extract from the zip and render.

**Severity:** LOW — requires physical or screen-sharing access to the recipient's device.

**Recommendation:** No action needed for current threat model. Document as a known side-channel for future consideration.

## 4. File Safety Orchestration Architecture — Pod Trust Model

**Brief:** `v0.10.30__arch-brief__file-safety-orchestration-architecture.md`
**Severity: HIGH (pod as single trust point), MEDIUM (partner data retention, attestation replay)**

### 4.1 Architecture Assessment

The file safety architecture is the most security-rigorous design document in this brief batch. The trust model is well-reasoned:

**FINDING FS-01 (INFO): The architecture correctly identifies the sender as explicitly untrusted and designs the entire scanning chain around this constraint. The attack surface table in the brief is comprehensive and accurate.**

Specific strengths:
- The pod computes the file hash, not the sender — this blocks the "scan clean, upload dirty" attack.
- Scan results are encrypted for the receiver only — the sender cannot learn what defences are in place.
- Attestation includes both `file_hash_in` and `file_hash_out` — covers both original and rebuilt files.
- Crossing points verify policy compliance without seeing file content — clean separation of concerns.
- Compromised crossing points are detected by inner layers — defence in depth with cryptographic verification.

### 4.2 Pod as Single Point of Trust

**FINDING FS-02 (HIGH): The pod sees plaintext. If the pod is compromised, the entire scanning chain is subverted — malicious files pass with forged attestations.**

The architecture acknowledges this: the pod is "the only component that is both trusted and able to see cleartext." This is a correct and unavoidable design trade-off — someone must see plaintext to scan it. But the consequences of pod compromise are total:

| If the pod is compromised, the attacker can: |
|---|
| See all files in plaintext during processing |
| Skip scanning entirely and forge passing attestations |
| Modify files during processing (inject malware into clean files) |
| Exfiltrate file contents |
| Map the receiver's scanning policy (which services are required) |
| Sign forged attestations with the pod's private key |

**The pod private key is the crown jewel of the entire architecture.** Its compromise is equivalent to a CA private key compromise in PKI.

**Recommendation:**

1. **HSM for the pod private key.** The pod's signing key must be stored in a hardware security module (HSM) or AWS CloudHSM/KMS. The key must never be extractable. All signing operations go through the HSM API.
2. **Pod key rotation.** Define a rotation schedule (e.g., quarterly). Old attestations remain valid (verify against the historical key). New attestations use the new key. Publish a key transparency log.
3. **Pod integrity monitoring.** The pod should run in a minimal, immutable container with runtime integrity monitoring (e.g., AWS Nitro Enclaves, or a measured boot chain). Any modification to the pod's code or configuration should be detectable.
4. **Consider multi-pod attestation.** For highest-assurance environments, require attestations from TWO independent pods operating different code in different networks. Both must agree on the scan result. A single compromised pod cannot forge a consensus.
5. **Short-lived pod attestation keys.** Rather than a long-lived pod key, use short-lived (e.g., 24-hour) signing keys derived from a master key. This limits the blast radius of key compromise to the validity period.
6. **The brief's "Future flow" (scanning services sign their own results) significantly reduces pod trust dependency.** Prioritise this. When the scanning service signs directly, a compromised pod can still see plaintext, but it cannot forge scan results that are independently verifiable by the receiver.

### 4.3 Scanning Partner Data Retention

**FINDING FS-03 (MEDIUM): Third-party scanning services receive files in plaintext via their API. If the scanning partner retains files after scanning, the zero-knowledge guarantee is degraded to "zero-knowledge except for the scanning partner."**

The brief acknowledges this risk and proposes contractual mitigation: "Contractually require zero retention after scan completion." However:

| Mitigation | Strength |
|---|---|
| Contractual no-retention clause | Weak — unverifiable. The partner may retain data for their own threat intelligence, model training, or regulatory compliance. |
| Embedded engine (Tier 3) | Strong — no external API call. File never leaves the pod. But requires engineering investment and partner cooperation. |
| WASM browser-side scanning (Tier 4) | Strongest — file never leaves the user's browser. But limited to engines that compile to WASM. |

**Recommendation:**

1. **For the Phase 1 (ClamAV) integration, this is a non-issue.** ClamAV runs locally in the pod. No external API call.
2. **For commercial partner integrations (Phase 2+), require a data processing agreement (DPA) with explicit no-retention clause.** Include the right to audit.
3. **Document the trust model per scanning tier.** "Basic (ClamAV): file never leaves the pod. Standard/CDR/Enterprise: file is submitted to [partner] API. [Partner] data retention policy: [link]." Let receivers make informed decisions.
4. **Prioritise Tier 3 (embedded engines) for sensitive environments.** The embedded engine eliminates the partner retention risk entirely.

### 4.4 Attestation Replay and Recency

**FINDING FS-04 (MEDIUM): The attestation includes a timestamp but the brief does not specify a recency window. A valid attestation from a scan performed days ago could be replayed against the same file.**

The brief mentions: "checks the timestamp is within an acceptable window." But does not define the window. If the window is too large (e.g., 7 days), a sender could:
1. Submit a clean file for scanning on Day 1, receive a valid attestation.
2. Modify the file (without re-scanning) — but wait, the hash would change, so the attestation would not match.

**On further analysis:** The hash binding (`file_hash_out` must match `sha256(decrypted_file)`) prevents this attack. The file cannot be modified without invalidating the attestation. The only viable replay is submitting the exact same file with the exact same attestation.

**Revised risk:** The replay of a valid attestation for the exact same file is low-impact — it is the same file, with the same scan result. The timestamp is useful for detecting stale scans (e.g., a file scanned before a new malware signature was released), not for preventing forgery.

**Recommendation:**

1. **Define the recency window.** 24 hours is reasonable for most use cases. Receivers in high-assurance environments should be able to require < 1 hour recency.
2. **Include a receiver-nonce in the attestation.** The receiver provides a fresh nonce with each delivery request. The pod includes the nonce in the signed attestation. This binds the attestation to a specific delivery, preventing cross-delivery replay entirely.

### 4.5 Cross-Domain Architecture — Specific Strengths

**FINDING FS-05 (INFO): The cross-domain architecture's three-party separation (untrusted sender, isolated pod, semi-trusted crossing points) is architecturally correct and represents a genuine advance over traditional cross-domain solutions.**

The key insight — that crossing points are explicitly not fully trusted, and the architecture assumes some will be compromised — is correct for real-world deployment. Traditional cross-domain solutions assume the crossing point is trusted, which is a structurally weaker assumption.

The multi-classification document pattern (routing document encrypted for the routing service, sensitive document encrypted for the final recipient) is particularly well-designed and maps directly to real government/defence requirements.

## 5. Chrome Sync — Vault Key Storage in Chrome Synced Storage

**Brief:** `v0.10.30__dev-brief__vault-fixes-and-ux.md`, Item 10
**Severity: MEDIUM**

### 5.1 Analysis

**FINDING CS-01 (MEDIUM): Storing vault keys in Chrome synced storage introduces Google as a trust dependency for vault read access.**

This is the same class of finding as the Chrome password sync issue identified in the v0.6.36 review (which concerned private keys in Chrome's credential store). The vault key has the same trust implications as a private key: it grants full read access to all vault contents.

| Storage Mechanism | Trust Dependency | Vault Key Exposure |
|---|---|---|
| `chrome.storage.sync` (extension API) | Requires a Chrome extension. Data synced via Google account. Google encrypts sync data with the user's Google account credentials. | Google infrastructure stores the encrypted vault key. A Google account compromise exposes the vault key. |
| `localStorage` (current) | Same-origin only. Browser sandbox. Not synced. | Local to one device. Lost if browser data is cleared. |
| Credential Management API | Browser-managed. May sync depending on browser settings. | Similar to Chrome sync — browser vendor becomes a trust dependency. |

**Chrome extension security considerations:**

| Concern | Assessment |
|---|---|
| Extension permissions | A Chrome extension that stores vault keys needs `storage` permission. If it also requests `tabs`, `webRequest`, or broad host permissions, it can read vault contents on any site. |
| Extension update attack | A compromised extension update can exfiltrate all stored vault keys. Chrome auto-updates extensions without user confirmation. |
| Chrome sync encryption | Chrome sync data is encrypted with the user's Google passphrase. If the user has not set a custom passphrase, Google holds the encryption key. |
| Cross-device sync lag | Keys synced to a new device may appear before the user expects. If the user shares a Google account with a family member, vault keys sync to all family devices. |

**Recommendation:**

1. **Label this as a convenience feature, not a security feature.** Display: "Syncing vault keys via Chrome uses your Google account for storage. Google's security policies apply."
2. **Default to OFF.** Chrome sync of vault keys should be opt-in, requiring explicit user action.
3. **If implemented via a Chrome extension, minimize permissions.** Request only `storage` — no `tabs`, no host permissions, no `webRequest`.
4. **Do not sync vault keys for vaults marked as "high security."** Allow vault owners to set a flag that prevents key syncing.

### 5.2 Saved Vaults Entry Screen

The brief proposes showing previously accessed vaults on an entry screen (from local storage). This is a usability feature with minor security implications:

**FINDING CS-02 (LOW): The saved vaults list reveals which vaults a user has accessed, stored in browser localStorage.**

If an attacker has access to the browser (physical access, XSS, or browser extension), they can enumerate all vaults the user has accessed, including vault IDs and names. This is metadata leakage.

**Recommendation:** Allow users to clear the saved vaults list. Consider encrypting the saved vault metadata with a browser-derived key (e.g., from `crypto.subtle`), though this adds complexity without strong protection against a determined attacker with same-origin access.

## 6. Inline Editing — Encrypt-on-Save Workflow

**Brief:** `v0.10.30__dev-brief__vault-fixes-and-ux.md`, Item 8
**Severity: MEDIUM**

### 6.1 Data Loss Risk

**FINDING IE-01 (MEDIUM): The encrypt-then-upload-on-save workflow has a multi-step failure path where user edits can be lost silently.**

The save workflow:
```
User edits text -> Click Save -> Encrypt -> Upload to server -> Update vault tree -> Upload vault tree
```

Failure at any step after "Encrypt" can cause data loss or inconsistency:

| Failure Point | Consequence |
|---|---|
| Encryption fails (e.g., key not in memory) | Edit lost. User sees error. **Recoverable** — edit still in the text editor. |
| Upload fails (network error, token expired) | Edit encrypted but not saved to server. **Recoverable** if the encrypted blob is retained in memory. |
| Vault tree update fails | File uploaded but vault tree does not reference it. **Orphaned file.** The vault shows the old version; the new version exists on the server but is unreachable via the tree. |
| Vault tree upload fails | Vault tree updated locally but not on server. **Split brain.** Local tree says new version; server tree says old version. Next sync or page refresh reverts to old version. |
| Browser tab closes during save | Any in-progress step is abandoned. **Edit lost.** |
| Auth token expired between editing and saving | Upload fails. **Edit lost** unless the UI prompts for token re-entry without discarding the edit. |

**Recommendation:**

1. **Implement local draft persistence.** Save edits to browser localStorage/IndexedDB periodically (auto-save every 30 seconds). Encrypt locally before storing. On page reload, detect unsaved changes and offer recovery.
2. **Make save atomic from the user's perspective.** Show a progress indicator during save. If any step fails, retain the edit in the editor and show a clear retry option. Never discard the user's text.
3. **Handle token expiry gracefully.** If the upload requires an auth token and the token has expired, prompt for token re-entry WITHOUT closing the editor or discarding the edit.
4. **Warn on navigation.** Use `beforeunload` to warn the user if they try to close or navigate away with unsaved changes.

### 6.2 Concurrent Edit Conflict

**FINDING IE-02 (LOW): Two users editing the same file in different browsers will overwrite each other's changes. The last save wins.**

The vault has no server-side locking or conflict detection for individual files. If User A and User B both open the same text file, edit it, and save, the second save overwrites the first without warning.

**Recommendation:** This is acceptable for MVP. Document the limitation: "Concurrent editing of the same file from different browsers may cause overwrites. The last save wins." Consider adding a version check on save (compare the server's current version hash with the hash at the time the editor loaded the file; reject the save if they differ).

## 7. Settings Panel — Vault Key Display

**Brief:** `v0.10.30__dev-brief__vault-fixes-and-ux.md`, Item 2
**Severity: LOW**

**FINDING SP-01 (LOW): The settings panel displays the vault key with a copy-to-clipboard button. This is necessary for sharing but creates a shoulder-surfing and clipboard exfiltration risk.**

| Concern | Assessment |
|---|---|
| Shoulder-surfing | A nearby observer can see the vault key. This grants permanent read access (per VF-01). |
| Clipboard persistence | The copied key persists in the clipboard until overwritten. Other applications can read it. |
| Screen recording/sharing | If the user is on a video call, the vault key is captured in the recording. |

**Recommendation:**

1. **Mask the vault key by default.** Show `****...****` with a "Show" button. Require a click to reveal.
2. **Auto-clear clipboard after 60 seconds.** Same pattern as password managers.
3. **Add a "Regenerate Key" option (future).** If a key has been exposed, the user should be able to generate a new key and re-encrypt the vault.

## 9. HTML Preview — Sandboxing Requirement

**Brief:** `v0.10.30__dev-brief__vault-fixes-and-ux.md`, Item 7 (HTML preview listed as "needed")

**FINDING HP-01 (MEDIUM): Rendering decrypted HTML files in the vault without sandboxing is a direct XSS vector.**

The brief lists "HTML: Sandboxed iframe" as needed. This is correct. The sandboxing requirements:

| Attribute | Required Value | Why |
|---|---|---|
| `sandbox` | `"allow-scripts"` | Allow the HTML to run its own scripts (needed for interactive content). |
| `sandbox` | Must NOT include `allow-same-origin` | If `allow-same-origin` is set, the sandboxed iframe can access the parent page's cookies, localStorage, and DOM — defeating the sandbox entirely. |
| `src` | Blob URL with explicit Content-Type | Never set the iframe `src` to a URL on the same origin. Use a blob URL or a `srcdoc` attribute. |
| CSP | `Content-Security-Policy: default-src 'none'; script-src 'unsafe-inline'` | Prevent the sandboxed HTML from loading external resources. |

**Recommendation:** Render HTML previews in a sandboxed iframe with `sandbox="allow-scripts"` (no `allow-same-origin`). Apply a restrictive CSP via the `csp` attribute on the iframe element: `<iframe sandbox="allow-scripts" csp="default-src 'none'; script-src 'unsafe-inline'; style-src 'unsafe-inline'; img-src blob: data:">`.

## 13. Alignment with Zero-Knowledge Architecture

| Feature | Zero-Knowledge Preserved? | Notes |
|---|---|---|
| Vault auth flow change (VF-01) | **Strengthened** | Fewer server-side signals for readers. But trades visibility for privacy. |
| Vault CLI | **Preserved** | Encryption/decryption happens locally in the CLI. Server sees only encrypted blobs. |
| Gallery sharing | **Preserved** | Decryption and rendering are fully client-side. Server stores encrypted zips only. |
| File safety pods | **Partially preserved** | Server remains zero-knowledge. Pods see plaintext transiently. Scanning partners see plaintext. Trust boundary is explicitly documented. |
| Chrome sync | **Degraded** | Vault key stored by Google. Google's infrastructure becomes a trust dependency. |
| Inline editing | **Preserved** | Edit, encrypt, upload — all client-side. Server sees only encrypted blobs. |
| Settings panel | **Preserved** | Client-side display of client-side data. No server interaction for key display. |
| Local storage viewer | **Preserved** | Client-side tool reading client-side data. |

**Overall:** The zero-knowledge property is preserved for the core transfer and vault functionality. The two degradation points are Chrome sync (introduces Google as trust dependency) and file safety pods (introduce scanning partners as plaintext-aware parties). Both are explicitly documented in the briefs and are opt-in features, not defaults. This is an acceptable trade-off.
