# v0.13.30 AppSec Review: March 10-12 Briefs

**version** v0.13.30
**date** 12 Mar 2026
**role** AppSec
**scope** Security review of 9 briefs from 10-12 March 2026

---

## Threat Model Summary

These nine briefs collectively represent the largest expansion of SG/Send's attack surface since the vault system was introduced. The proposals move SG/Send from an encrypted blob store with a thin CLI into a full version-control system with branching, PKI-per-branch, multi-remote support, Lambda-resident key material, a public key registry, nested vaults, a public gallery surface, and a vault download endpoint accessible from AI agent environments.

**Key threat model changes:**

| Change | New Attack Surface | Severity |
|--------|--------------------|----------|
| Lambda identity (key pairs in Lambda) | Key material at rest in compute environment | HIGH |
| Branch key pairs (auto-generated per clone) | Key proliferation, storage diversity (browser, OS keychain, vault-of-keys) | HIGH |
| Vault download endpoint (single-URL bundle) | Unauthenticated or weakly-authenticated bulk data exfiltration | HIGH |
| Public key registry (scraped keys) | Trust model confusion, impersonation via stale/revoked keys | MEDIUM |
| Public gallery (key in URL) | Deliberate ZK bypass for public content; URL leakage exposes everything | MEDIUM |
| Nested vaults (vault-in-vault) | Recursive decryption complexity, confused deputy at vault boundaries | MEDIUM |
| CLI remotes (multiple backends) | Credential management across N remote types | MEDIUM |
| Headless CLI mode (no interactive prompts) | Automation-friendly but removes human confirmation gates | MEDIUM |
| Signed commits (mandatory branch-key signatures) | Signature verification trust anchoring; self-signed vs authority-signed ambiguity | LOW |

## Finding Table

| # | Severity | Brief | Finding | Recommendation |
|---|----------|-------|---------|----------------|
| F-01 | **CRITICAL** | Lambda Vault Logging | Lambda functions hold private keys in memory for the lifetime of the execution context. A Lambda compromise (code injection, dependency supply chain, layer poisoning) exposes the signing key and the ability to forge log entries. | Lambda identity keys MUST be stored in AWS KMS or Secrets Manager, never in environment variables or in-memory from S3. Use KMS Sign/Verify API so the private key never enters Lambda memory. If performance is unacceptable, use an asymmetric KMS key cached only for signing, never for decryption. |
| F-02 | **CRITICAL** | Vault Access from Claude | The proposed `/api/vault/download/{vault_id}` endpoint returns an entire vault in a single HTTP response. If access control is weak (e.g., a guessable download token or no auth at all), this is a bulk data exfiltration vector. The brief explicitly describes a pattern where `web_fetch` bypasses the egress proxy. | The download endpoint MUST require a proof-of-knowledge token (e.g., the `bundle_file_id` scheme from the v0.13.19 architect response). A vault ID alone MUST NOT be sufficient. Rate-limit downloads per vault. Log all download events to the audit trail. Consider time-limited signed URLs rather than persistent endpoints. |
| F-03 | **HIGH** | Branch Model | Key pairs are auto-generated on every clone. The brief lists four storage locations (user vault, browser localStorage, OS keychain, another vault). This creates an unmanaged key sprawl problem. Lost branch keys mean unsigned history; compromised branch keys mean forged commits. | Define a canonical key lifecycle: generation, storage, rotation, revocation, expiry. Branch keys MUST have an expiry. Revoked branch keys MUST be published (CRL or equivalent). Do not allow branch keys in browser localStorage for anything beyond ephemeral sessions -- localStorage is accessible to any JS on the same origin including XSS payloads. |
| F-04 | **HIGH** | Branch Model | Signed commits use the branch key, but the trust anchor is unclear. Self-signed mode only proves "same entity" -- it does not prove identity. Without a trust chain to the vault owner or a CA, signed commits provide integrity but not authentication. An attacker who clones and creates a branch has a valid branch key and can produce valid signed commits. | Require authority-signed mode for any merge to main. Self-signed mode is acceptable for work-in-progress branches but MUST NOT be trusted for merged content without a countersignature from a vault-level or user-level key. Define the trust hierarchy explicitly before implementation. |
| F-05 | **HIGH** | Encrypt for Reader | Multi-level encryption (different readers for different log levels) means multiple public keys are used per log entry. If the "master audit key held in escrow" is compromised, all log entries across all levels are exposed. The escrow key is a single point of failure for the entire logging system. | The master audit key MUST use key splitting (e.g., Shamir's Secret Sharing) or an HSM-backed key with multi-party access. Define escrow recovery procedures before implementation. Never store the escrow key in the same infrastructure as the logs. |
| F-06 | **HIGH** | Bare Vault Unification | The brief asks whether file names in the vault tree are encrypted or plaintext. This is a fundamental ZK question that is currently unanswered. If file names are plaintext in the unified structure, the server (and anyone who accesses S3) can see the directory structure and file names of every vault. | File names MUST be encrypted or replaced with opaque IDs in the server-side tree. The human-readable tree mapping (path -> blob ID) must only be decryptable by the vault key holder. This is a blocking requirement for the unification work. |
| F-07 | **HIGH** | Bare Vault Unification | Nested vaults (vault-in-vault) create a recursive trust boundary. The outer vault key grants access to the inner vault's ciphertext but not its plaintext. The key distribution for inner vaults must be independent. Document the trust boundary clearly: outer vault decryption reveals inner vault ciphertext, not inner vault content. Add automated tests that verify inner vault contents remain opaque after outer vault decryption. |

The SG/Send signature attests "we found this key at this URL on this date" but users may interpret it as "SG/Send vouches for this key's ownership." This is a trust model confusion risk. | The registry UI and API MUST display explicit disclaimers: "This key was found at [URL] on [date]. SG/Send does not verify key ownership." Never use language like "verified" or "trusted." Implement key freshness checks -- re-scrape periodically and mark stale entries. An S3 remote depends on IAM credentials. | Define minimum security requirements per remote type. Local folder remotes MUST warn that data at rest is only as secure as the filesystem. ## Zero-Knowledge Audit

The ZK model states: the server stores only ciphertext and cannot access plaintext, file names, or decryption keys.

| Brief | ZK Impact | Verdict |
|-------|-----------|---------|
| Encrypt for Reader | **Preserves ZK.** Reader-targeted encryption is a stricter form of ZK -- the server is explicitly untrusted. No regression. | PASS |
| Lambda Vault Logging | **Preserves ZK** if implemented correctly. Log entries are encrypted with the reader's public key. The Lambda itself holds a signing key (identity) but the log content is encrypted for the reader, not the Lambda. The server stores ciphertext. However, **the Lambda sees plaintext before encryption** -- this is an inherent requirement for logging. The ZK boundary moves from "server never sees plaintext" to "server sees plaintext during the invocation but encrypts it immediately." | CONDITIONAL PASS -- document this ZK boundary shift explicitly. |
| PKI Document Identity | **Preserves ZK.** Signatures are computed on content hashes, not plaintext. Signature metadata (who signed, when) is intentionally public. No plaintext exposure. | PASS |
| Bare Vault Unification | **RISK: file names.** If the unified vault tree contains plaintext file names, the server can see the directory structure. This would be a ZK regression. The brief explicitly asks this question but does not answer it. | FAIL until file name encryption is confirmed. |
| Branch Model | **Preserves ZK** for content. Branch keys are generated client-side. Commits are signed client-side. The server stores signed, encrypted commits. However, **branch metadata** (branch name, creation time, public key) is necessarily visible to the server for push/pull coordination. This is a minor metadata leak. | CONDITIONAL PASS -- branch names should be opaque IDs, not human-readable names, on the server side. |
| CLI Remotes | **Depends on remote type.** SG/Send remote preserves ZK. Local folder remote has no ZK (plaintext on disk after decryption). S3 remote depends on encryption configuration. | CONDITIONAL PASS -- CLI must warn users about ZK properties per remote type. |
| Vault Access from Claude | **Preserves ZK** if the download bundle is encrypted. The key is provided separately by the human. The server returns ciphertext. Claude decrypts locally. | PASS |
| Gallery (public mode) | **Deliberately breaks ZK** for public galleries by embedding the key in the URL. This is a design choice, not a bug. Private and key-gated galleries preserve ZK. | ACCEPTABLE -- but must be clearly communicated to users. |
| Public Key Registry | **N/A.** Public keys are public by definition. No ZK impact. | PASS |

## Cryptographic Decisions

### 1. Encrypt-for-Reader (Multi-Recipient Encryption)

**Proposed:** Each log entry encrypted with multiple public keys (DevOps, AppSec, compliance, audit).

**Assessment:** This is hybrid encryption (RSA/ECIES wrapping of a symmetric key per entry, with the symmetric key encrypted N times for N readers). Well-established pattern (PGP, age, NaCl sealed boxes). Correct approach.

**Risks:**
- Ciphertext size grows linearly with number of readers (one wrapped key per reader per entry). For high-volume logging, this adds overhead.
- Reader key rotation requires re-encryption of the symmetric key wraps, not re-encryption of content. This is efficient.
- If any reader's private key is compromised, only that reader's access is exposed, not all readers'. This is correct isolation.

**Recommendation:** Use AES-256-GCM for content encryption with a per-entry random key. Wrap the per-entry key with each reader's public key. This is the standard approach and matches SG/Send's existing hybrid encryption model.

### 2. PKI Per Branch (Branch Key Pairs)

**Proposed:** Every clone generates a key pair. Every commit is signed by the branch key.

**Assessment:** This creates a decentralised identity model where branches are first-class cryptographic entities. Similar to SSH key-per-device or Signal's per-device identity keys.

**Risks:**
- Key generation quality depends on the client environment. Browser `crypto.getRandomValues()` is CSPRNG. Python `secrets` module is CSPRNG. Both are acceptable.
- No key revocation mechanism is specified. A compromised branch key can sign commits until someone manually revokes the branch.
- No key rotation mechanism. Long-lived branch keys accumulate risk.
- The "vault-of-keys" pattern (a meta-vault containing branch keys) creates a high-value target. Compromise of the meta-vault exposes all branch keys.

**Recommendation:** Mandate key expiry (e.g., 90 days). Implement branch key revocation as a vault-level operation (vault owner can revoke any branch key). The vault-of-keys pattern needs its own security review before implementation.

### 3. Signed Commits (Mandatory Chain)

**Proposed:** Every commit signed by branch key. Signature covers tree hash, parent hash, message, timestamp, branch identity.

**Assessment:** This is a Merkle chain with signatures -- structurally identical to Git's signed commit model but mandatory rather than optional. Correct approach for integrity.

**Risks:**
- Clock skew in timestamps. Lambda functions and client devices may have different clock sources. Timestamps should be treated as advisory, not authoritative.
- Replay attacks: a valid signed commit from one branch could theoretically be replayed to another branch if the signature does not bind to the branch identity. The proposal includes branch identity in the signed data, which prevents this. Correct.

**Recommendation:** Include a nonce or sequence number in the signed data to prevent replay within the same branch. Ensure timestamp is from a reliable source (prefer server-provided timestamp at push time over client-provided timestamp at commit time).

### 4. Document-Level Signing

**Proposed:** Individual documents, sections, and paragraphs can be signed independently.

**Assessment:** Granular signing is powerful for provenance but creates complexity. Each signature is an independent cryptographic operation. Nested signatures (a signed section within a signed document) require clear precedence rules.

**Recommendation:** Start with document-level signing only. Section and paragraph signing is a research area. Define the signature format (detached vs. embedded) before implementation. Use a standard format (e.g., JWS, CMS) rather than inventing a custom one.

## Key Management Analysis

The March 10-12 briefs introduce at least four new categories of key material:

| Key Type | Where Generated | Where Stored | Lifetime | Revocation |
|----------|----------------|-------------|----------|------------|
| Lambda identity key | Lambda cold start | Lambda memory (proposed) | Lambda execution context lifetime | None specified |
| Branch key pair | Client (clone operation) | Browser localStorage / OS keychain / user vault / meta-vault | Indefinite (no expiry proposed) | None specified |
| Escrow audit key | Offline ceremony (proposed) | Not specified | Indefinite | Not specified |
| Registry signing key | SG/Send infrastructure | Not specified | Indefinite | Not specified |

**Key management gaps identified:**

1. **No key lifecycle management.** None of the briefs specify key expiry, rotation, or revocation procedures.
2. **No key backup/recovery.** If a branch key is lost, the commit history from that branch cannot be verified. There is no recovery mechanism.
3. **No key hierarchy.** The relationship between vault keys, branch keys, Lambda keys, and registry keys is not defined. A clear hierarchy (root -> vault -> branch) with delegation and revocation chains is needed.
4. **Storage diversity creates inconsistency.** Branch keys in browser localStorage have different security properties than keys in OS keychain or in a vault. The system should enforce minimum storage requirements based on key sensitivity.
5. **Lambda identity key is the weakest link.** A Lambda function's execution environment is shared infrastructure (AWS manages the underlying host). Key material in Lambda memory is exposed to any code running in that function, including dependencies. This is the highest-priority key management issue.

**Recommendation:** Before implementing any of the proposed PKI features, produce a Key Management Specification that covers:
- Key hierarchy (root, vault, branch, Lambda, registry)
- Generation requirements (algorithm, key size, entropy source)
- Storage requirements per key type (minimum security level)
- Expiry and rotation schedule
- Revocation mechanism and propagation
- Backup and recovery procedures
- Compromise response playbook

## Recommended Security Gates

These are blocking requirements that should be satisfied before the proposed features ship:

| Gate | Blocks | Requirement |
|------|--------|-------------|
| **SG-01** | Lambda Vault Logging | Lambda identity keys must use KMS or Secrets Manager, not in-memory key material. Demonstrate that the signing key never enters Lambda memory in plaintext. |
| **SG-02** | Vault Download Endpoint | Download endpoint must require proof-of-knowledge (not just vault ID). Rate limiting and audit logging must be in place before the endpoint goes live. |
| **SG-03** | Bare Vault Unification | Confirm file names are encrypted or opaque in the server-side tree. Produce a field-by-field audit showing no plaintext sensitive data in the unified structure. |
| **SG-04** | Branch Model | Key Management Specification must be written and reviewed before branch key generation is implemented. Specification must cover lifecycle, storage, revocation, and hierarchy. |
| **SG-05** | Branch Model | Authority-signed mode must be implemented for merges to main. Self-signed branches must not merge to main without a countersignature. |
| **SG-06** | Public Key Registry | Trust model disclaimer must be visible on every registry entry. No use of the words "verified" or "trusted" in registry UI or API responses. |
| **SG-07** | Gallery (public mode) | Key must be in URL fragment (hash), never in query parameters or path. Gallery creation UI must warn that public galleries are only as private as the URL. |
| **SG-08** | Nested Vaults | Inner vault keys must not be stored in the outer vault. Automated test must verify inner vault contents remain opaque after outer vault decryption. |
| **SG-09** | CLI Remotes | CLI must display security properties of each remote type on `remote add` and `push`. No silent security downgrade when switching remotes. |
| **SG-10** | All PKI features | Key Management Specification (covering all key types introduced in these briefs) must be completed before any PKI feature ships. |

## Summary

The March 10-12 briefs are architecturally ambitious and directionally sound. The "encrypt for reader" paradigm, branch-based PKI, and signed commit chains are strong security primitives that would differentiate SG/Send.

However, the briefs introduce significant key management complexity without specifying key lifecycle procedures. The single highest-risk item is **Lambda identity keys stored in Lambda memory** (F-01) -- this must use KMS. The single most urgent ZK question is **whether file names are encrypted in the vault tree** (F-06) -- this must be answered before the bare vault unification proceeds.

The public gallery and public key registry represent deliberate trade-offs between ZK purity and usability. These are acceptable if communicated clearly to users, but they expand the trust surface and require careful UI language to avoid implying guarantees that do not exist.

No feature in these briefs should ship without the Key Management Specification (SG-10) being completed first. Key management is the foundation that every other proposal depends on.
