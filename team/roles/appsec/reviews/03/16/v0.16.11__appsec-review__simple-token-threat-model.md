## 1. Scope

This review threat-models the proposed "simple token" sharing mode (Mode 1) described in the Architect review. In this mode, a human-readable token (e.g. `coal-render-5512`) is the sole secret. The client derives both the transfer ID and the AES-256 decryption key from this token via PBKDF2 with two different salts.

**PROPOSED -- does not exist yet.** None of the components reviewed here (PBKDF2 derivation, client-set transfer IDs, `sg-pbkdf2.js`, `sg-wordlist.js`, simple-token hash fragment parsing, 409 Conflict handling) exist in the current codebase. The current `Transfer__Service.create_transfer()` generates transfer IDs server-side via `secrets.token_hex(6)`.

## 2. Threat Model

### 2.1 Token Brute-Force

**Threat:** An attacker attempts to guess existing simple tokens to locate and decrypt files.

**Analysis:**

Token format is `{word1}-{word2}-{number}` where the number is 4 digits (0000-9999). The Architect review claims ~39.5 billion combinations. This depends on the wordlist size:

- If the wordlist has ~1,988 words: 1,988 x 1,988 x 10,000 = ~39.5 billion combinations (~35.2 bits of entropy)
- If the wordlist is smaller (e.g. 1,000 words): 1,000 x 1,000 x 10,000 = 10 billion (~33.2 bits of entropy)

**The wordlist size is critical and must be documented.** The Architect review references `sg-wordlist.js` from the Tools repo but does not state the wordlist size.

**PBKDF2 cost per attempt:** 600,000 iterations of SHA-256 per derivation. The attacker needs to derive the transfer ID (one PBKDF2 call) and then check if the transfer exists via the API. At 600K iterations:

- A modern GPU (RTX 4090) can do ~6 billion SHA-256/sec, which translates to ~10,000 PBKDF2-600K derivations/sec
- A consumer CPU does ~500-2,000 derivations/sec
- An ASIC/cloud cluster could reach ~100,000 derivations/sec

At 39.5 billion combinations and 10,000 derivations/sec, exhaustive search takes ~45 days on a single high-end GPU. This is the offline component -- the attacker derives candidate transfer IDs locally.

**However**, the attacker must then check each candidate against the server. This is the critical bottleneck. Without rate limiting, an attacker could probe ~1,000 transfer IDs/sec via `GET /api/transfers/info/{id}` or `GET /api/transfers/download/{id}`. With rate limiting at 10 req/sec per IP, enumeration becomes impractical.

**Key observation:** The attacker does NOT need to hit `/transfers/create` to enumerate. They can probe `GET /transfers/info/{id}` or `GET /transfers/download/{id}` which return 404 for non-existent IDs and 200 for existing ones. **Rate limiting must cover ALL transfer endpoints, not just create.**

| Attribute | Assessment |
|-----------|------------|
| **Severity** | Medium |
| **Likelihood** | Unlikely (for targeted attacks on specific tokens); Rare (for untargeted brute-force across all tokens) |
| **Recommendation** | Mitigate |

**Action items:**

1. **Document the wordlist size** in the PBKDF2 specification. Minimum 2,000 words recommended.
2. **Rate limit all transfer endpoints** (info, download, download-base64), not just create. Recommend: 30 requests/minute per IP for info/download endpoints. **PROPOSED -- rate limiting does not exist yet.**
3. **Monitor for enumeration patterns.** Log 404 responses on transfer endpoints with IP hashes. Alert on >50 misses from a single IP hash within 5 minutes. **PROPOSED -- does not exist yet.**
4. **Consider increasing the number component** to 5 or 6 digits for additional entropy (100K-1M more combinations). This is a one-time decision before launch.

### 2.2 Token Collision

**Threat:** Two users independently choose the same simple token. Both derive the same transfer ID. The second user's `POST /transfers/create` fails with 409 Conflict.

**Analysis:**

This is a correctness issue, not a security issue. The 409 Conflict mechanism is sufficient to prevent overwriting. The client retries with a different token.

**Birthday problem consideration:** With 39.5 billion possible tokens, the probability of any collision reaches 50% at ~250,000 active transfers (sqrt(39.5B) * 1.2). For a service with fewer than 10,000 concurrent transfers, the collision rate is negligible (<0.0001%).

**However:** If users can choose their own words (rather than being assigned random ones), the effective entropy drops dramatically. Common tokens like `test-data-1234` or `hello-world-0000` will collide. The UI should generate random tokens by default and make manual entry optional.

**Edge case:** If the first user's transfer has expired and been deleted, the same token can be reused. This is acceptable -- the ID space is recycled.

| Attribute | Assessment |
|-----------|------------|
| **Severity** | Low |
| **Likelihood** | Unlikely (with random generation); Possible (if users manually choose common words) |
| **Recommendation** | Accept (with UI defaulting to random token generation) |

**Action items:**

1. **Default to randomly generated tokens** in the UI. Manual entry should be an "advanced" option.
2. **Handle 409 gracefully** -- auto-regenerate and retry (up to 3 attempts) without user intervention.
3. **Consider reserving common/dictionary tokens** (e.g. block `test-test-0000` through `test-test-9999`). Low priority.

### 2.3 Enumeration via Transfer-Create

**Threat:** An attacker uses `POST /api/transfers/create` with guessed transfer IDs. A 409 Conflict response confirms that a transfer with that ID exists. This leaks information about which IDs are active.

**Analysis:**

The Architect review states: "A 409 Conflict tells the client 'this ID exists' -- but the client can't do anything with it without also knowing the decryption key. The ID alone is useless without the key."

This is **partially correct but incomplete.** The 409 confirms existence, which has two implications:

1. **For simple-token transfers:** If the attacker is guessing simple tokens, a successful PBKDF2 derivation that produces an ID returning 409 means **the attacker has found a valid token AND already knows the decryption key** (both are derived from the same token). The 409 is not the vulnerability -- the guessable token is. The 409 just confirms the guess was correct.

2. **For server-generated ID transfers (Modes 2/3):** An attacker cannot craft a valid create request with a server-generated ID format. However, if client-set IDs use the same format (12 hex chars) as server-generated IDs, the attacker could probe for server-generated transfers too.

**Critical finding:** The 409 enumeration path via `/transfers/create` is **redundant** with the easier enumeration via `GET /transfers/info/{id}` (which returns 200 vs 404). The create endpoint actually has a higher bar -- it requires a valid access token and a well-formed request body. The info endpoint is the easier enumeration vector.

| Attribute | Assessment |
|-----------|------------|
| **Severity** | Low |
| **Likelihood** | Possible |
| **Recommendation** | Mitigate |

**Action items:**

1. **Rate limit `POST /transfers/create`** to prevent rapid probing. Recommend: 10 creates/minute per access token. **PROPOSED -- does not exist yet.**
2. **Do not distinguish between "ID exists" and "ID format invalid"** in error responses. Return a generic 409 for both cases -- but note this adds complexity. Current design (clear 409 for existence) is acceptable given the low severity.
3. **Log 409 events** with IP hash, access token, and timestamp for anomaly detection.

### 2.4 Token in URL Path vs Hash Fragment

**Threat:** The token or decryption key appears in the URL path or query string, where it could be logged by proxies, CDNs, web servers, browser history, or referrer headers.

**Analysis:**

The Architect review correctly specifies that tokens and keys must **only** appear in the hash fragment (`#`). The hash fragment has well-established security properties:

- **Not sent to server:** The browser never includes the hash fragment in HTTP requests (RFC 3986, Section 3.5).
- **Not logged by proxies/CDNs:** Since it's not in the request, intermediaries never see it.
- **Not in Referrer header:** Per the Fetch spec, the fragment is stripped from the Referrer header.
- **Visible in browser history:** The hash fragment IS stored in browser history and the address bar. This is a known, acceptable trade-off.
- **Visible in shared screenshots:** If a user screenshots the browser with the URL bar visible, the token is exposed. This is a user-education issue.

**Current codebase verification:** The existing download page (`download/index.html`) already uses hash fragments for `#transferId` and `#transferId/key`. The proposed simple-token mode follows the same pattern (`#coal-render-5512`). This is consistent.

**The proposed hash fragment parsing logic** (Section 7 of Architect review) distinguishes token format (`word-word-number`) from ID/key format (`id/key`). This regex-based detection is sound but has an edge case: if a transfer ID happens to match the `^[a-z]+-[a-z]+-\d{4}$` pattern, it would be misinterpreted as a simple token. Since server-generated IDs are 12 hex chars (from `secrets.token_hex(6)`), this collision is impossible -- hex strings contain digits and a-f only, never the `-` separator. **Safe.**

| Attribute | Assessment |
|-----------|------------|
| **Severity** | Informational |
| **Likelihood** | N/A (design is correct) |
| **Recommendation** | Accept |

**Action items:**

1. **Enforce in code review:** Any PR that puts a token or key in a URL path, query parameter, or request body must be rejected.
2. **Add automated test:** Verify that no outbound HTTP request from the UI contains the token or key in the URL path/query. This can be a Playwright test. **PROPOSED -- does not exist yet.**
3. **Add `Referrer-Policy: no-referrer` header** to all HTML pages (if not already present). This provides defence-in-depth against fragment leakage in edge cases.

### 2.5 Client-Set Transfer IDs

**Threat:** Accepting transfer IDs from the client introduces risks: injection attacks, namespace collision with server-generated IDs, and potential for abuse.

**Analysis:**

The current `Transfer__Service.create_transfer()` generates IDs via `secrets.token_hex(6)` -- a 12-character hex string (48 bits of entropy). The proposed change allows the client to provide this ID instead.

**Injection risk:** The transfer ID is used as a path component in `Storage_FS` (`transfers/{transfer_id}/meta.json`). If the client sends a malicious ID like `../../etc/passwd` or `../admin/meta`, it could traverse the storage namespace.

**This is the highest-priority finding in this review.**

Current code in `Transfer__Service` directly interpolates the transfer ID into storage paths:

```python
def meta_path(self, transfer_id):
    return f'transfers/{transfer_id}/meta.json'
```

If `transfer_id` is `../../foo`, the path becomes `transfers/../../foo/meta.json` which resolves to `foo/meta.json` -- potentially outside the transfers namespace.

**Mitigations required:**

1. **Strict format validation:** Server MUST validate that client-provided IDs match `^[0-9a-f]{12}$` (exactly 12 lowercase hex characters). Reject anything else with 400 Bad Request.
2. **Path traversal protection:** Even with format validation, add a defence-in-depth check that the resolved path starts with `transfers/`.
3. **Namespace separation:** Consider prefixing client-set IDs (e.g. `t-{id}`) to distinguish them from server-generated IDs. This prevents a client-set ID from colliding with or overwriting a server-generated transfer. **However**, this would break the simple-token model since the recipient must derive the same ID. Alternative: reserve a format for each. Since PBKDF2-derived IDs are 12 hex chars and server-generated IDs are also 12 hex chars, they share the same namespace. This is acceptable IF the existence check works correctly -- it just means the collision space includes both types.

| Attribute | Assessment |
|-----------|------------|
| **Severity** | High (path traversal without validation); Low (with proper validation) |
| **Likelihood** | Likely (without validation -- any client can send arbitrary strings); Rare (with validation) |
| **Recommendation** | Mitigate (mandatory before implementation) |

**Action items:**

1. **MANDATORY: Add server-side validation** of client-provided transfer IDs. Regex: `^[0-9a-f]{12}$`. Return 400 for invalid format. This must be implemented before the feature ships.
2. **MANDATORY: Add path traversal protection** in `Transfer__Service.meta_path()` and `payload_path()`. Assert that the resolved path starts with `transfers/`. This protects against validation bypass.
3. **Add test cases** for malicious transfer IDs: `../../etc/passwd`, `<script>`, `' OR 1=1--`, empty string, very long strings, strings with null bytes. All must return 400.
4. **Consider using `Safe_Str__Id`** (which already exists in osbot-utils) for the transfer ID parameter in the create endpoint, as it may provide built-in validation.

### 2.6 PBKDF2 Parameters

**Threat:** The PBKDF2 parameters are insufficient to protect simple tokens against offline brute-force attacks.

**Analysis:**

Proposed parameters:
- Algorithm: PBKDF2-SHA256
- Iterations: 600,000
- Salt (ID derivation): `"sgraph-send-v1-id"` (static)
- Salt (key derivation): `"sgraph-send-v1-key"` (static)
- Key length: 256 bits (AES-256)

**Iteration count:** OWASP recommends a minimum of 600,000 iterations for PBKDF2-SHA256 (as of 2023). The proposed 600K meets this minimum. NIST SP 800-63B recommends at least 10,000 iterations but does not specify an upper bound. 600K is within industry best practice.

**Static salt concern:** Traditional password hashing uses a unique random salt per password to prevent rainbow tables and to ensure identical passwords produce different hashes. Here, the salt is static by design -- the same token must always produce the same transfer ID and key (otherwise the recipient can't derive them).

This is a fundamental trade-off of the simple-token architecture. The static salt means:

- **Rainbow tables are possible.** An attacker could precompute `PBKDF2("sgraph-send-v1-id", token, 600K)` for all possible tokens. At 39.5 billion tokens and 32 bytes per output, this table would be ~1.17 TB. This is feasible for a well-funded attacker.
- **Mitigation:** The 600K iteration cost makes precomputation expensive (~125 years of single-GPU time). But a distributed attack with 1,000 GPUs could complete it in ~45 days. This is the fundamental limit of the simple-token approach.
- **Versioned salts** (`v1` in the salt string) allow migration to stronger parameters in the future without breaking existing transfers.

**Dual derivation independence:** Using different salts for ID and key derivation ensures that knowing the transfer ID does not reveal the key. The two PBKDF2 outputs are cryptographically independent. This is correct.

| Attribute | Assessment |
|-----------|------------|
| **Severity** | Medium (static salt enables precomputation); Low (iteration count is adequate) |
| **Likelihood** | Unlikely (requires significant resources for rainbow table construction) |
| **Recommendation** | Accept (with documented limitations and version migration path) |

**Action items:**

1. **Document the static-salt trade-off** in security documentation. Users must understand that simple tokens are "convenient, not maximum security."
2. **Keep the version in the salt string** (`v1`). If PBKDF2 parameters need strengthening, increment to `v2` and support both during migration.
3. **Monitor OWASP recommendations** for PBKDF2 iteration counts. If the recommended minimum increases, update accordingly (this will require a version bump).
4. **Consider Argon2id as a future alternative.** Argon2id is memory-hard, making GPU/ASIC attacks more expensive. However, Web Crypto API does not natively support Argon2id, so this would require a JS library. Low priority -- PBKDF2-SHA256 at 600K is adequate for the threat model.

### 2.7 Simple Token vs Random Key -- Security Trade-Off

**Threat:** Users may use simple tokens for sensitive data where they should use random keys.

**Analysis:**

| Property | Mode 1 (Simple Token) | Mode 2/3 (Random Key) |
|----------|----------------------|----------------------|
| Key entropy | ~35 bits (depends on wordlist) | 256 bits |
| Brute-force cost | Expensive but feasible for motivated attacker | Computationally impossible |
| Can be shared verbally | Yes | No (base64url string) |
| Can be memorised | Yes | No |
| Server knows the key? | No | No |
| Finding ID reveals key? | Yes (same derivation) | No (independent random values) |
| Suitable for sensitive data | **No** | **Yes** |

The 220-bit gap between ~35-bit token entropy and 256-bit random key entropy is enormous. A simple token provides protection against casual observers and opportunistic attackers, but not against targeted attacks with moderate resources.

**The critical UX concern:** If simple token is the default mode (as proposed), users will use it for sensitive data without thinking. The UI must actively discourage this.

| Attribute | Assessment |
|-----------|------------|
| **Severity** | Medium |
| **Likelihood** | Likely (users will use the default for everything) |
| **Recommendation** | Mitigate |

**Action items:**

1. **Show a clear security level indicator** in the UI for each mode. Example: Mode 1 shows "Standard security -- suitable for non-sensitive files." Mode 2/3 shows "Maximum security -- suitable for confidential files."
2. **Do not label simple token as "secure" or "encrypted."** The file IS encrypted, but the key is guessable. Describe it as "convenient sharing" or "easy sharing."
3. **Consider prompting for Mode 2** when the file size is large or the content type suggests sensitivity (e.g. .docx, .xlsx, .pdf with "confidential" in the name). Low priority -- user education is more effective than heuristics.
4. **Document the security trade-off** in user-facing help text on the share mode selection screen.

### 2.8 Incident Response -- Detecting Token Guessing/Brute-Force

**Threat:** An attacker has guessed or brute-forced an existing simple token and downloaded the file. How do we detect this?

**Analysis:**

The current system logs download events with IP hash and user agent in the transfer's `events` array (`Transfer__Service.get_download_payload()`). This provides a forensic trail but no real-time detection.

**Detection signals:**

| Signal | Meaning | Detectability |
|--------|---------|---------------|
| High 404 rate on `/transfers/info/{id}` from one IP | Enumeration attempt | Detectable with endpoint monitoring. **PROPOSED -- does not exist yet.** |
| Download from different IP hash than upload IP hash | Legitimate sharing OR attacker access | Ambiguous -- most downloads will be from different IPs. Not useful as a standalone signal. |
| Many 409s on `/transfers/create` from one IP | Transfer ID enumeration | Detectable. **PROPOSED -- does not exist yet.** |
| Download with no preceding share event | Attacker accessed the file directly | Not detectable -- the system cannot distinguish between a user who received the token via SMS and an attacker who guessed it. |

**Fundamental limitation:** Because the system is zero-knowledge, it cannot distinguish between an authorised recipient (who received the token from the sender) and an attacker (who guessed the token). Both present the same derived transfer ID and decryption key. The server sees identical requests.

**The only detectable signal is enumeration behaviour** (high miss rates), not successful guessing.

| Attribute | Assessment |
|-----------|------------|
| **Severity** | Medium (limited detection capability is inherent to zero-knowledge design) |
| **Likelihood** | Possible (successful guess is undetectable; enumeration attempts are detectable) |
| **Recommendation** | Mitigate (for enumeration); Accept (for successful single-guess attacks) |

**Action items:**

1. **Implement enumeration detection.** Monitor 404 rates on transfer endpoints per IP hash. Alert threshold: >50 misses in 5 minutes from a single IP hash. **PROPOSED -- does not exist yet.**
2. **Implement auto-block.** After N consecutive 404s from one IP hash on transfer endpoints, return 429 Too Many Requests for 15 minutes. **PROPOSED -- does not exist yet.**
3. **Provide transfer activity visibility.** Allow uploaders (if authenticated or if they have the transfer ID) to see download count and IP hash list. This exists partially -- `get_transfer_info()` returns `download_count` but not the per-download IP hashes.
4. **Consider optional download notifications.** If the uploader provides an email, send a notification on first download. **PROPOSED -- does not exist yet.** Low priority for Explorer.
5. **Accept the zero-knowledge trade-off.** A successful single guess is inherently undetectable. This is the cost of the zero-knowledge architecture and is acceptable for Mode 1's intended use case (non-sensitive files).

## 4. Overall Assessment

### Is client-set transfer ID acceptable?

**Yes, with mandatory safeguards.** The approach is architecturally sound -- it follows naturally from the PBKDF2 dual-derivation design and maintains the zero-knowledge property. However, accepting client-provided values as storage path components is inherently dangerous.

**Two controls are non-negotiable before this ships:**

1. **Server-side format validation:** `^[0-9a-f]{12}$` -- reject anything else with 400. This is a one-line regex check but it's the difference between a secure system and a path traversal vulnerability.
2. **Path traversal defence-in-depth:** Assert that constructed paths stay within the `transfers/` namespace. This protects against validation bypass bugs.

### What additional controls are needed?

**Before launch (Explorer scope):**

- [ ] Server-side transfer ID format validation (`^[0-9a-f]{12}$`)
- [ ] Path traversal defence in `Transfer__Service.meta_path()` and `payload_path()`
- [ ] Test cases for malicious transfer IDs
- [ ] Document wordlist size and entropy calculation
- [ ] UI defaults to randomly generated tokens
- [ ] UI communicates security level per sharing mode

**Before production hardening (Villager scope):**

- [ ] Rate limiting on all transfer endpoints (not just create)
- [ ] Enumeration detection and auto-block (429 after N misses)
- [ ] `Referrer-Policy: no-referrer` on all HTML pages
- [ ] Playwright test verifying tokens never appear in HTTP requests
- [ ] Monitoring and alerting for anomalous 404 patterns

**Future roadmap (not blocking):**

- [ ] Optional download notifications
- [ ] Consider Argon2id migration path
- [ ] Consider 5-6 digit number component for additional entropy
- [ ] Transfer activity dashboard for uploaders

### Final Verdict

The simple-token architecture is **acceptable for its intended use case** (convenient sharing of non-sensitive files). The PBKDF2 parameters are adequate, the hash-fragment approach is correct, and the dual-derivation design is cryptographically sound. The main risk is implementation -- specifically the path traversal vector in client-set transfer IDs, which must be addressed before any code is merged.

The broader security posture of SGraph Send remains strong: the zero-knowledge encryption architecture means that even if all the above mitigations fail, an attacker who compromises the server still cannot read file contents. The simple-token mode reduces client-side key entropy but does not weaken the server-side security model.
