# AppSec Review: Vault Architecture — Pre-Launch Security Audit

**Version:** v0.13.34
**Date:** 2026-03-16
**Scope:** Full vault stack — User Lambda (public APIs), Admin Lambda (metadata APIs), service layer, storage, crypto, CLI
**Trigger:** Pre-launch review ahead of official web + CLI vault release
**Prior reviews:** v0.13.30 (vault simulation v6), v0.13.22 (AI agents as vault peers), v0.5.0 (PKI architecture)

---

## Executive Summary

The vault architecture maintains strong zero-knowledge properties — the server never sees plaintext, encryption keys, or original file names. The core threat model is sound. However, this review identifies **3 HIGH**, **4 MEDIUM**, and **3 LOW** severity findings that should be assessed before official launch. The HIGH findings relate to vault squatting, read endpoint enumeration, and unsalted write-key hashing.

## Findings

### F-01: Vault Squatting — First Write Claims Any Vault (HIGH)

**Location:** `Service__Vault__Pointer._check_vault_write_key()` (line 33-38)
**Location:** `Service__Vault__Pointer._ensure_manifest()` (line 40-47)

**Description:** When a vault has no manifest (never been written to), `_check_vault_write_key()` returns `True` for ANY write key. The first write to any `vault_id` creates the manifest and permanently locks the vault to that write key. This means:

1. An attacker who knows (or guesses) a `vault_id` can race the legitimate owner and claim the vault
2. The `vault_id` namespace is flat — there's no pre-registration step that ties a vault_id to an owner before data is written
3. The vault_id is 8 characters (Safe_Str__Id) — enumerable if the attacker has a valid access token

**Impact:** An attacker with any valid access token can squat on vault IDs, preventing legitimate users from using them. The legitimate owner's first write would succeed (creating their manifest), but if the attacker writes first, the owner gets 403 forever.

**Recommendation:**
- **Option A (Recommended):** Require vault pre-registration via the Admin Lambda before any User Lambda write is accepted. The User Lambda should check with the Admin Lambda that the vault_id exists and the write_key matches what was registered.
- **Option B:** Add a vault creation step to the User Lambda that binds vault_id to a specific token, rejecting writes from other tokens.
- **Mitigating factor:** Access tokens are required for writes, limiting the pool of potential attackers to authenticated users.

### F-02: Read and List Endpoints Require No Authentication (HIGH)

**Location:** `Routes__Vault__Pointer.read__vault_id__file_id()` (line 89-99)
**Location:** `Routes__Vault__Pointer.read_base64__vault_id__file_id()` (line 101-116)
**Location:** `Routes__Vault__Pointer.list__vault_id()` (line 160-164)

**Description:** The read, read-base64, and list endpoints require NO authentication — no access token, no write key, nothing. Anyone who knows a `vault_id` can:

1. List all file IDs in the vault (`/api/vault/list/{vault_id}`)
2. Download all encrypted blobs (`/api/vault/read/{vault_id}/{file_id}`)
3. Enumerate vault existence by trying vault IDs

**Security model argument:** The design intent is that the data is AES-256-GCM encrypted, so raw ciphertext is useless without the decryption key. This is the "zero-knowledge" model — the server stores opaque blobs.

**Counter-argument:** File IDs leak metadata. With `{file_id:path}`, IDs like `bare/data/obj-abc123` and `bare/refs/heads/main` reveal the internal structure (Git-like content-addressed store). An attacker can:
- Determine which vaults are active (list returns files)
- Track vault activity over time (file count changes)
- Observe ref names and branch structure (even if content is encrypted)
- Perform targeted attacks on specific file IDs they've observed

**Recommendation:**
- **Short-term:** Add rate limiting on read/list endpoints (IP-based, per vault)
- **Medium-term:** Require access token for list endpoint (file_id metadata is more sensitive than blob content)
- **Long-term:** Consider requiring a `read_key` header derived from the vault passphrase — cheap to validate, prevents casual enumeration
- **NOTE:** This was also flagged in v0.13.22 review (F-02). Remains unaddressed.

### F-03: Write Key Hash Is Unsalted (HIGH)

**Location:** `Service__Vault__Pointer._hash_write_key()` (line 30-31)

```python
def _hash_write_key(self, write_key_hex):
    return hashlib.sha256(write_key_hex.encode()).hexdigest()
```

**Description:** The write key is stored as `SHA256(write_key_hex)` with no salt. If an attacker gains read access to the storage backend (S3 bucket misconfiguration, backup leak, insider threat), they can:

1. Read all manifest.json files to get write_key_hash values
2. Perform offline brute-force or rainbow table attacks against the hashes
3. The write_key is a 64-char hex string (derived from PBKDF2), so entropy is high — but the lack of salt means identical passphrases across vaults produce the same write_key_hash, enabling correlation attacks

**Impact:** If the storage backend is compromised, the attacker can identify vaults that share the same passphrase (same write_key_hash) and correlate ownership.

**Recommendation:**
- Salt the hash with vault_id: `SHA256(vault_id + write_key_hex)` — simple change, breaks correlation between vaults
- Consider HMAC: `HMAC-SHA256(vault_id, write_key_hex)` — more cryptographically correct
- Requires manifest migration for existing vaults

### F-04: TOCTOU Race in Manifest Creation (MEDIUM)

**Location:** `Service__Vault__Pointer._ensure_manifest()` (line 40-47)
**Location:** `Service__Vault__Pointer._check_vault_write_key()` (line 33-38)

**Description:** The check-then-create pattern in `_ensure_manifest()` has a time-of-check-to-time-of-use race:

```
Thread A: _check_vault_write_key() → no manifest → True
Thread B: _check_vault_write_key() → no manifest → True
Thread A: _ensure_manifest() → creates manifest with key_A
Thread B: _ensure_manifest() → file_exists check passes, skips creation
Thread B: write succeeds (using key_A's manifest, but key_B was submitted)
```

Wait — Thread B's write would actually succeed because the manifest was already created by Thread A. But Thread B submitted a different write_key. The `_check_vault_write_key()` returned True (no manifest at that time), and by the time `_ensure_manifest()` runs, A's manifest exists. But B's write data goes through because the auth check already passed.

**Impact:** In a concurrent first-write scenario, the second writer's data can be written to the vault even though their write key doesn't match the manifest that was actually created. This is a one-time race on vault creation only.

**Recommendation:**
- Use atomic create-if-not-exists (S3 `PUT` with `If-None-Match: *` header)
- Or combine check + ensure into a single atomic operation
- Or accept the risk: the window is tiny and both writers must have valid access tokens

### F-05: ACL Operations Have No Owner Authorization Check (MEDIUM)

**Location:** `Routes__Vault.share__vault_cache_key()` (line 186-196)
**Location:** `Routes__Vault._resolve_cache_id()` (line 180-184)

**Description:** The Admin Lambda vault ACL endpoints (share, unshare, permissions) only check that the vault exists — they do NOT verify that the requesting user is the vault owner. The `granted_by` field is hardcoded to `'admin'` (line 193), not derived from the authenticated caller.

Any authenticated Admin Lambda user can:
1. Grant themselves owner/editor/viewer access to any vault
2. Revoke access from other users (except owners)
3. List all permissions for any vault

**Mitigating factor:** The Admin Lambda is API-key gated — only the system itself and authorized administrators can call it. It's not exposed to end users.

**Recommendation:**
- Add caller identity to ACL operations (derive from API key or request context)
- Validate that the caller has `owner` permission before allowing grant/revoke
- This becomes critical if the Admin Lambda API surface is ever exposed to non-admin callers

### F-06: Batch Operation Allows Unlimited Data Volume (MEDIUM)

**Location:** `Routes__Vault__Pointer.batch__vault_id()` (line 136-158)
**Location:** `Service__Vault__Pointer.batch()` (line 131-170)

**Description:** The batch endpoint limits operations to 100 but places NO limit on the size of data in each operation. A single batch request with 100 large base64-encoded payloads could:

1. Exhaust Lambda memory (Lambda has fixed memory allocation)
2. Create very large JSON request bodies (no explicit Content-Length limit)
3. Write 100 large files in a single request, consuming storage rapidly

**Impact:** Potential denial-of-service via resource exhaustion. The Lambda's 6MB payload limit provides some natural throttling, but if the request body is streamed or if the Lambda memory allocation is generous, this could still cause issues.

**Recommendation:**
- Add per-operation data size limit (e.g., 5MB per write operation in batch)
- Add total batch payload size limit
- Consider per-vault storage quotas

### F-07: IP Hash Has No Salt (MEDIUM)

**Location:** `Transfer__Service.hash_ip()` (line 135-138)

```python
def hash_ip(self, ip_address):
    if ip_address is None:
        ip_address = ''
    return hashlib.sha256(ip_address.encode()).hexdigest()
```

**Description:** IP addresses are hashed with SHA-256 but no salt. SHA-256 of an IPv4 address is trivially reversible — there are only ~4.3 billion possible IPv4 addresses. A precomputed rainbow table for all IPv4 addresses takes ~137GB and allows instant lookup.

**Note:** The CLAUDE.md states "IP addresses hashed — SHA-256 with daily salt" but the actual code uses no salt. The documentation is inaccurate.

**Recommendation:**
- Add daily rotating salt as documented: `SHA256(daily_salt + ip_address)`
- Store the salt generation date alongside the hash so old hashes can still be correlated within their day
- This was noted in prior reviews but remains unfixed

### F-08: No Upload Size Limit on Vault Write (LOW)

**Location:** `Routes__Vault__Pointer.write__vault_id__file_id()` (line 66-87)

**Description:** The individual write endpoint reads the full request body with `await request.body()` (line 76) with no size check. While Lambda's 6MB payload limit provides a natural ceiling, the code itself doesn't enforce any limit.

**Recommendation:**
- Add explicit size check before processing: `if len(payload) > MAX_PAYLOAD_SIZE: raise 413`
- Documents the intended limit rather than relying on infrastructure constraints

### F-09: Manifest File Accessible Via Read Endpoint (LOW)

**Location:** `Service__Vault__Pointer.vault_payload_path()` (line 27-28)
**Location:** `Service__Vault__Pointer.vault_manifest_path()` (line 24-25)

**Description:** The manifest is stored at `transfers/vault/{vault_id}/manifest.json`. The payload path is `transfers/vault/{vault_id}/{file_id}/payload`. Because `file_id` uses the `:path` converter, a crafted file_id like `../manifest.json` could potentially traverse to the manifest... but only if the final path component `payload` is somehow bypassed.

**Analysis:** Actually, the path construction is: `f'transfers/vault/{vault_id}/{file_id}/payload'`. A path traversal with `file_id = "../manifest.json"` would produce `transfers/vault/{vault_id}/../manifest.json/payload` — which is NOT the manifest path. And the list endpoint explicitly filters out `manifest.json` from results (line 116).

**Verdict:** **Not exploitable** with current path construction. The `/payload` suffix provides an accidental-but-effective defense. However, the manifest IS directly readable if the attacker constructs the exact S3 key — this is an infrastructure-level concern, not an API-level one.

**Recommendation:** No code change needed. Ensure S3 bucket policies prevent direct object access bypassing the Lambda.

---

### F-10: No Audit Trail for Vault Operations (LOW)

**Location:** `Routes__Vault__Pointer` (all endpoints)
**Location:** `Fast_API__SGraph__App__Send__Admin` (analytics disabled)

**Description:** Vault write, delete, and batch operations are not logged to any audit trail. The analytics middleware is disabled on the Admin Lambda. There is no per-vault activity log.

**Impact:** If a vault is compromised (data overwritten, deleted), there's no record of when it happened, from which IP, or which token was used.

**Recommendation:**
- Log vault write/delete/batch events with: timestamp, vault_id, ip_hash, token_name, operation, file_count
- This is separate from analytics — it's a security audit trail
- Store as append-only (e.g., S3 object per event, or a log file per vault)

## What's Working Well

1. **Zero-knowledge model is sound** — AES-256-GCM with client-side encryption means server compromise doesn't expose plaintext
2. **PBKDF2 with 600K iterations** — strong key derivation, resistant to brute force on the passphrase itself
3. **Write key separation** — read access doesn't grant write access; compromising the read key doesn't allow modification
4. **Safe_Str__Id validation** — input validation on vault_id and file_id parameters prevents injection
5. **Write-if-match (CAS)** — optimistic concurrency control prevents silent data loss in multi-writer scenarios
6. **Batch stop-on-conflict** — prevents cascading errors in push operations
7. **Lambda response limits enforced** — prevents response size exhaustion (5MB direct, 3.75MB base64)
8. **Admin Lambda is API-key gated** — metadata operations aren't exposed to end users
9. **Redirect loop fix** — catch routes prevent infinite 307 loops on missing file_id (fixed today)

## Addendum: Vault UI v0.1.3 Frontend Security Review (post-merge from dev)

After merging latest `dev` (v0.16.4), two new IFD vault UI versions landed: v0.1.2 and v0.1.3 (in `sgraph_ai_app_send__ui__vault/`). These introduce a full git-like object model with content-addressed storage, a vault shell, file browser, and sharing UI. The following additional findings apply to the **frontend code**.

### F-11: Passphrase Stored in URL Hash — Browser History Leak (CRITICAL)

**Location:** `vault-entry.js:38-41,84-85,134`, `vault-share.js:46-51`

The vault key format is `{passphrase}:{vault_id}` and the application stores the full key in `window.location.hash`. This means:
- Browser history contains the plaintext passphrase
- Screenshots, screen recordings, and error reporting services capture it
- Browser extensions with `tabs` permission can read it
- Shared browser sessions (public computers) expose it in history

**Recommendation:** Use `window.history.replaceState()` to clear the hash immediately after parsing. Consider a session-only in-memory store instead of URL-based key passing.

### F-12: Passphrase Retained in Memory Indefinitely (CRITICAL)

**Location:** `sg-vault.js:25,40,82`

`this._passphrase` stores the plaintext passphrase as a class property for the entire page lifetime. After PBKDF2 key derivation completes, the passphrase is no longer needed — but it's never cleared.

An attacker with JavaScript console access (XSS, browser extension, DevTools) can read `vault._passphrase` directly.

**Recommendation:** Null out `this._passphrase` immediately after key derivation. Use `this._passphrase = null` at the end of the `open()` method.

### F-13: Vault Key Displayed in DOM as Plaintext Input (HIGH)

**Location:** `vault-shell.js:154,906`

The vault key (including passphrase) is set as the value of a `<input type="text" readonly>` element. This is visible in:
- Page screenshots and recordings
- DOM inspection by browser extensions
- Error reporting services that capture form state
- Automated accessibility tools that read input values

**Recommendation:** Mask the display (`type="password"`) with a copy button, or show only the vault_id and mask the passphrase.

### F-14: Access Token in sessionStorage — XSS Extractable (HIGH)

**Location:** `vault-entry.js:58`, `vault-shell.js:334,698`

The access token is stored in `sessionStorage`, which is readable by any JavaScript on the same origin. If the vault UI ever loads third-party scripts (analytics, error tracking, CDN resources), those scripts can extract the token.

**Recommendation:** Keep token in a JavaScript closure (not sessionStorage). Clear sessionStorage explicitly on vault lock.

### F-15: Debug Panel Exposes Decrypted Internal State (HIGH)

**Location:** `vault-shell.js:710-716`

The "Raw JSON" debug toggle renders `JSON.stringify(this._vault._tree)` and `this._vault._settings` into a `<pre>` tag. This exposes the decrypted vault structure — folder names, file names, metadata — which the zero-knowledge model is designed to protect.

**Recommendation:** Remove or gate the debug panel behind a developer flag that is disabled in production builds.

### F-16: URL Hash Not Cleared on Vault Lock (MEDIUM)

**Location:** `vault-shell.js:254-255`

When the vault is locked, `this._vaultKey` is cleared from the JavaScript state, but the URL hash still contains the full key. A user can navigate back in browser history or reload to restore the key.

**Recommendation:** Call `window.history.replaceState(null, '', window.location.pathname)` when locking.

### F-17: Vault ID Format Breaking Change (MEDIUM)

**Location:** `sg-vault-crypto.js:38`

Vault ID validation changed from `/^[0-9a-f]{8}$/` (hex only, v0.1.2) to `/^[a-z0-9]{8}$/` (alphanumeric, v0.1.3). Vaults created with v0.1.3 will fail validation on v0.1.2 clients.

**Recommendation:** Document the format change. Ensure CLI and web versions align on the accepted vault ID format.

### F-18: No Passphrase Strength Enforcement (MEDIUM)

**Location:** `vault-entry.js:117-122`

Form validation only checks if passphrase is truthy. No minimum length, entropy, or strength check.

**Recommendation:** Add a minimum length requirement (12+ characters) and optionally a strength meter.
