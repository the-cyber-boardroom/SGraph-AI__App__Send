# AppSec Assessment: 12 March Briefs -- Security Implications

**Version:** v0.13.30
**Date:** 13 March 2026
**Role:** AppSec
**Scope:** Security review of 13 documents in the 12 March daily brief

---

## Executive Summary

The 12 March briefs represent the largest single-day architecture expansion. The vault simulation v6 introduces zero-knowledge branch naming, PKI per branch, signed commit chains, and a unified vault format. The privacy policy brief codifies what the architecture already enforces. Several new attack surfaces need assessment.

---

## HIGH Priority Findings

### H-1: Zero-Knowledge Branch Naming (Simulation v6)

The move from readable paths (`refs/heads/current`) to opaque IDs (`ref-a3f8`, `branch-named-x7k2`) with encrypted index is a major security improvement. Assessment:

- **POSITIVE:** Server sees only folder types and counts. Cannot determine branch names, relationships, or which objects are blobs vs trees vs commits.
- **POSITIVE:** Metadata leakage eliminated. Previous design leaked branch names and key assignments in file paths.
- **RISK:** The encrypted branch index (`idx-{id}`) is eventually consistent. Between GC runs, a stale index could cause a client to miss a branch. This is a correctness issue, not a security issue -- but in a multi-user scenario, a malicious server could serve stale indexes to hide changes.
- **RECOMMENDATION:** Add index freshness validation. Client should compare index branch count against actual file count in `bare/branches/` folder.

### H-2: Named Branch Private Key in Vault Storage

The `current_priv` key (`key-d4a1` in the simulation) is stored encrypted with `read_key` in the vault. This means anyone with `read_key` can sign commits on the named branch.

- **RISK:** Read access = merge authority. A user with only the vault passphrase can merge into the named branch. There is no separate "write" or "merge" permission.
- The simulation notes this explicitly: "anyone with read_key can merge."
- **RECOMMENDATION:** Consider a separate `merge_key` derived from `write_key`, not `read_key`. Or accept that vault passphrase = full authority (simpler, matches current trust model where sharing the passphrase = sharing full access).

### H-3: Change Pack Auth (Scenario 4)

Keyless creators write to `bare/pending/` with a scoped write-only token. The server accepts opaque blobs without validation.

- **RISK:** Denial of service via flooding `pending/` with garbage blobs. The billing-scoped token limits this, but the server cannot verify content integrity until GC drains pending packs.
- **RISK:** A malicious change pack could be crafted to cause issues when drained by a vault owner.
- **RECOMMENDATION:** Size limits on pending writes. Rate limiting per token. GC should validate signatures and schema on drain, reject invalid packs.

### H-4: Bare Vault Unification -- Security Audit Required

The brief explicitly calls for a security audit of the unified vault format. Key questions:

- Are file names encrypted in tree objects? (Brief asks this question explicitly)
- Is any user-identifiable information present in the bare vault beyond signatures?
- Can nested vaults (vault-inside-vault) create confused deputy problems?
- **RECOMMENDATION:** Produce a field-by-field audit document before implementation begins.

## MEDIUM Priority Findings

### M-1: P0 Bug -- Silent Upload Failure

The 5MB upload failure is a UX bug, not a security bug. However, silent failures can lead users to believe their file was transferred when it wasn't.

- **RECOMMENDATION:** Client-side validation with clear error message. Never fail silently.

### M-2: API Key Sharing via Vault (LLM Brief)

Encrypting OpenRouter API keys with user's public key and storing in vault is architecturally sound. The encrypt-for-reader pattern is correct.

- **RISK:** API key spending caps are enforced by OpenRouter, not by vault. If a key leaks (user's device compromised), the spending cap is the only limit.
- **RECOMMENDATION:** Short-lived API keys with automatic rotation. Log all API key usage server-side.

### M-3: Gallery Public Access Mode

Public galleries embed the decryption key in the URL. This is the same pattern as existing SG/Send transfers.

- **RISK:** URL sharing = access sharing. Same as current model. No new risk.
- **NOTE:** The gallery URL includes a human-readable title slug. Ensure the slug does not leak information beyond what the gallery creator intends.

### M-4: Public Key Registry (Portugal Strategy)

Scraping and indexing public keys from GitHub, government sites, and PGP keyservers.

- **RISK:** Key validity. The registry attests "found at URL on date" but does not verify ownership. A scraped key could be revoked, expired, or compromised.
- **RISK:** Legal/GDPR. Indexing public keys associated with names/emails may trigger GDPR obligations even if the keys were publicly posted.
- **RECOMMENDATION:** Clear disclaimers. Opt-out mechanism. Regular re-scraping to detect revocations.

## LOW Priority / Positive Findings

### L-1: Privacy Policy -- Architecture Validates Claims

The six-sentence privacy policy claims "cannot" not "will not." AppSec confirms this is accurate:

- Server stores AES-256-GCM ciphertext only. Verified in code.
- Key in URL fragment, never sent to server. Verified in code.
- Zero cookies, zero tracking. Verified: no analytics scripts in UI code.
- The privacy policy is a factual description of the architecture. No overclaiming detected.

### L-2: Signed Commit Chain Integrity

Every commit signed by branch key. Signature covers tree hash + parent hash + message + timestamp + branch identity. Tampering invalidates all subsequent signatures. This is stronger than Git's optional GPG signing.

- **POSITIVE:** Mandatory signing eliminates the "forgot to sign" class of errors.

### L-3: Uint Timestamps

Switching from ISO datetime strings to uint milliseconds (`Timestamp_Now`). Eliminates timezone leakage. Compact, sortable. Good zero-knowledge hygiene.

---

## Simulation v6 -- Security Verdict

The simulation demonstrates that the vault architecture maintains zero-knowledge properties through all 4 scenarios (solo, two-user, conflict, change pack). The server sees opaque blobs in typed folders with no content knowledge. The key architectural decisions (`bare/` as portable unit, conflicts on clone branch, encrypted index) are all security-positive.

The one architectural gap the simulation explicitly defers (batch endpoint spec) has security implications: atomicity and partial failure handling need to prevent race conditions during concurrent pushes. The write-if-match semantic on refs (optimistic concurrency) is correct but needs formal specification.
