# v0.8.4 AppSec + Architect + Dev: Why Does validate-token Exist?

**Date:** 2026-03-01
**Roles:** AppSec (lead), Architect, Dev
**Version:** v0.8.4
**Trigger:** Human asked — "what is the business reason to have that capability [token consumption] triggered from the client side?"

---

## 1. Short Answer

**There is no valid business reason.** The client-triggered consumption is an artifact of a design that was partially implemented. The original design envisioned server-side page gating; the implementation switched to static HTML + client-side JS but carried the consumption trigger forward without re-evaluation.

**Critical discovery from Dev:** The `validate-token` endpoint is **already dead code** for the current UI. After the INC-003 security fix removed `?token=` from shareable URLs, no current UI version produces download links that contain the token parameter. The download component still parses `?token=` from the URL, but it will always be `null` for any file uploaded through the current UI. The endpoint remains live, public, and destructive — but nothing calls it.

---

## 2. How It Got Here — The Three Design Threads

The Architect traced the design back to three separate threads that converged without critical review:

### Thread A: The Event Taxonomy (v0.2.33)

The Architect's `Schema__Token__Usage_Event` defined three action types: `page_opened`, `upload_initiated`, `upload_completed`. These were meant to **classify** events for analytics — not to define which events should decrement a usage counter.

**Source:** `team/roles/architect/reviews/02/13/v0.2.33__response-to-daily-brief-2__13-feb.md` (line 310)

### Thread B: The Dev Brief's Server-Side Gating (v0.2.41)

The Dev Brief specified a server-side gated route `GET /t/{token_name}/{transfer_id}` that would validate the token **before serving the download HTML page**. The phrasing was:

> "Each page load / download consumes one token use"

This conflated "page load" and "download" as if they were the same event. In the server-gated design, they effectively would be — the server would only serve the page after consuming a use. If consumption failed, the page would never be served.

**Source:** `team/roles/dev/briefs/v0.2.41__dev-brief__ui-token-integration-and-service-registry.md` (lines 125-144)

### Thread C: The Implementation Diverged

The actual implementation (commit `f8c17ab`, 14 Feb) did NOT create the server-side gated route. Instead:

- The download page is served as a **static HTML file**
- The `<send-download>` Web Component calls `POST /transfers/validate-token/{token_name}` from **client-side JavaScript** on page load
- This moved the consumption trigger from server-controlled to client-controlled

**The consumption trigger is a fossil of a design that was partially implemented.** The server-side gating was dropped; the consumption on page load was kept.

### The Sherpa Saw It Early (v0.3.2)

One day after the token system was designed, the Sherpa identified the gap:

> "G-RR-04: No 'page opened' event distinct from 'file downloaded'. A recipient might open the download page, see the transfer info, but not proceed to decrypt."

The Sherpa recommended `page_opened` as an **analytics event** — separate from consumption. The two concerns were never separated.

**Source:** `team/roles/sherpa/reviews/02/14/v0.3.2__action-plan__explorer-next-steps.md` (lines 53, 95)

## 3. The Business Purpose of Tokens (What IS Valid)

The original brief (v0.2.32, Dinis Cruz) defined three purposes for the token system:

| Purpose | How It Works Today | Correct? |
|---------|-------------------|----------|
| **Beta access gating** | `check-token` + `<send-access-gate>` on upload page — non-consuming | YES |
| **Usage metering (billing prototype)** | `upload_completed` consumption in `complete__transfer_id()` — authenticated, tied to real action | YES |
| **Audit trail** | Every consumption event records `ip_hash`, `action`, `transfer_id` | YES |

The token system IS the foundation for billing. From the v0.7.6 pricing brief:

> "SG/Send uses a credit-based pricing model built on the existing access token infrastructure. Price per file: GBP 0.01."

**The problem is specifically the `validate-token` endpoint** — it adds a second, uncontrolled consumption point on top of the correct `upload_completed` one.

---

## 4. The Double-Counting Problem

Each successful file transfer currently consumes **two** token uses:

| Event | Trigger | Action | Authenticated? | Correct? |
|-------|---------|--------|----------------|----------|
| 1st | Recipient opens download page | `page_opened` | **No** | **No** — fires on page view, not download |
| 2nd | Sender completes upload | `upload_completed` | **Yes** | **Yes** — tied to real action |

A token with `usage_limit=50` yields roughly **25 actual file transfers**, not 50. Browser refreshes, bots, accidental navigations, and link previews reduce this further.

---

## 5. Dev Finding: validate-token Is Already Dead Code

The Dev agent traced the git history and found that `validate-token` is **currently orphaned**:

1. **Original design** (commit `f8c17ab`): The upload component's `buildCombinedUrl()` added `?token=X` to shareable download URLs. The `complete` response returned `token_name` so the UI could embed it.

2. **INC-003 security fix**: Both were removed. `token_name` is no longer returned in the `complete` response (line 123: "SECURITY FIX: token_name intentionally NOT returned to client"). The v0.1.4+ `buildCombinedUrl()` does NOT include `?token=`:
   ```javascript
   // v0.1.8 (current)
   buildCombinedUrl(tid, key) {
       return `${window.location.origin}/send/v0/v0.1/v0.1.8/download.html#${tid}/${key}`;
   }
   ```

3. **Result**: The download component still parses `?token=` (line 89: `this.tokenName = params.get('token') || null`), but for any file uploaded through the current UI, `tokenName` will always be `null`. The `if (this.tokenName)` block that calls `validateToken()` never executes.

**The endpoint is dead code for the UI — but remains a live, public, unauthenticated, destructive API endpoint that anyone can call directly.**

---

## 6. Dev Analysis: What Would Need to Change

The Dev agent analysed two options for moving consumption server-side:

### Option A: Move Consumption to Download Endpoint (Recommended)

Move consumption into `download__transfer_id` and `download_base64__transfer_id`. The download endpoint already tracks `download_count` — adding token consumption there is a natural extension. Total change: ~30-40 lines of Python, 1 line of JS, test updates.

**Changes required:**

1. **Store token association in transfer metadata** — in `complete__transfer_id`, save `token_name` into the transfer's `meta.json` (3 lines)
2. **Consume on download** — in `download__transfer_id` and `download_base64__transfer_id`, look up the associated `token_name` from transfer metadata and call `token_use(action='download_completed', transfer_id=...)` (same pattern already used in `complete__transfer_id`)
3. **Remove `validate-token` endpoint** — it is dead code for the current UI. Remove the handler, route registration, and path constant. (Or convert to read-only if backward compat is needed.)
4. **Client-side** — replace `ApiClient.validateToken()` with `ApiClient.checkToken()` in `send-download.js` (one-line change). The page still blocks if the token is exhausted/revoked, but without consuming a use.
5. **Update tests** — remove/adjust `test__validate_token__*` tests, add `test__download_consumes_token_use`, update `test__full_flow_with_token` to verify download triggers consumption

### Option B: Keep validate-token But Rate-Limit

Add per-IP rate limiting to `validate-token` (e.g., max 5 calls per minute per IP). This doesn't fix the design flaw but reduces the attack surface.

**Dev's recommendation: Option A.** It eliminates the public unauthenticated write endpoint entirely and binds consumption to the point where value is actually delivered.

---

## 7. Product Decision Required

Before implementing, the business must decide **what "per file" means** in the pricing model:

| Billing Model | Consume on upload? | Consume on download? | 50-credit token delivers |
|---------------|-------------------|---------------------|-------------------------|
| **Per-send** | Yes (`upload_completed`) | No | 50 sends, unlimited receives |
| **Per-receive** | No | Yes (`download_completed`) | Unlimited sends, 50 receives |
| **Per-transfer** | Yes (`upload_completed`) | No | 50 complete transfers |
| **Both** (current) | Yes | Yes (`page_opened`) | ~25 transfers (double-counted) |

The pricing brief says "GBP 0.01 per file." This most naturally maps to **per-send** — the sender is the customer, the recipient is their audience. Under this model, the `validate-token` consumption can be removed entirely.

---

## 8. Summary of Findings

| Question | Answer |
|----------|--------|
| Is there a business reason for client-triggered consumption? | **No.** It's a fossil of a server-side gating design that was never implemented. |
| Is the consumption logic itself secure? | **Yes.** The increment/status-transition/event-recording all happen server-side in `Service__Tokens.use()`. |
| Is the trigger for consumption secure? | **No.** Any anonymous HTTP client can invoke it at will. |
| Was this design deliberate? | **No.** The Dev Brief specified server-side gating. The implementation diverged. All token features shipped in a single commit (`f8c17ab`) without iterative review. |
| Is validate-token currently used by the UI? | **No.** After INC-003, `?token=` was removed from shareable URLs. The endpoint is dead code for the UI but remains a live public API. |
| What should change? | Remove `validate-token` entirely (it's dead code). Move consumption to the download endpoint (server-side, tied to actual file delivery). |
