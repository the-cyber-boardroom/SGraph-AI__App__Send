## 3. Detailed Validation

### 3.1 PRIORITY: `access_token` as Raw `str` — CONFIRMED REAL

**Severity:** MEDIUM
**Locations (3 methods):**

```
Routes__Transfers.py:86  — create(..., access_token: str = '')
Routes__Transfers.py:97  — upload__transfer_id(..., access_token: str = '')
Routes__Transfers.py:117 — complete__transfer_id(..., access_token: str = '')
```

The `access_token` parameter bypasses `Safe_Str__Id` type validation. It is accepted as raw `str` from query parameters, headers, or MCP tool parameters. Although downstream token lookup validates the value, input sanitisation should happen at the route entry point.

The `check_access_token()` method (line 35) also accepts `str`:
```python
def check_access_token(self, request: Request, access_token: str = ''):
```

**Fix:** Change all occurrences to `Safe_Str__Id`. One-line change per method.

### 3.2 SECONDARY: Route Returns as `dict` — CONFIRMED REAL

**Severity:** LOW
**Locations (6 methods):**

All route methods (`create`, `upload`, `complete`, `info`, `download`, `download_base64`) return raw `dict` instead of `Type_Safe` schema classes.

**Type_Safe schemas already exist but are unused:**
- `Schema__Transfer__Initiated` (transfer_id, upload_url)
- `Schema__Transfer__Complete_Response` (transfer_id, download_url, transparency)
- `Schema__Transfer__Info` (transfer_id, status, file_size_bytes, created_at, download_count)

**TODO comments exist in code** at lines 82, 87, 92, 118, 140 confirming this is known technical debt.

### 3.3 `transfer_id` as `Safe_Str__Id` — CONFIRMED SAFE

All path parameters for `transfer_id` use `Safe_Str__Id`:
```
Routes__Transfers.py:95  — upload__transfer_id(self, transfer_id: Safe_Str__Id, ...)
Routes__Transfers.py:115 — complete__transfer_id(self, transfer_id: Safe_Str__Id, ...)
Routes__Transfers.py:139 — info__transfer_id(self, transfer_id: Safe_Str__Id, ...)
Routes__Transfers.py:154 — download__transfer_id(self, transfer_id: Safe_Str__Id, ...)
Routes__Transfers.py:178 — download_base64__transfer_id(self, transfer_id: Safe_Str__Id, ...)
```

`Safe_Str__Id` strips all characters matching `[^a-zA-Z0-9_-]`, eliminating path traversal vectors (`/`, `.`, `\`, null bytes). The threat model's assessment (Section 2.5) that path traversal is "already mitigated" is **correct**.

### 3.4 Rate Limiting — CONFIRMED MISSING

No rate limiting middleware, decorator, or logic found anywhere in the transfer routes or FastAPI app setup.

**Searched:**
- `Routes__Transfers.py` — no rate limiting on any endpoint
- `Fast_API__SGraph__App__Send__User.py` — only CORS middleware configured
- No rate limiting package in dependencies

**Impact:** An attacker can probe `GET /api/transfers/info/{id}` at high rates to enumerate existing transfer IDs. This is the easiest enumeration vector (not `POST /transfers/create` which requires an access token).

### 3.5 Hash Fragment — CONFIRMED CORRECT

Download component (`send-download.js`) reads `window.location.hash.substring(1)` — the hash fragment only. Tokens/keys are never sent to the server in HTTP requests.

**Missing defence-in-depth:** `Referrer-Policy: no-referrer` header not found on HTML pages. Threat model recommends adding this (Section 2.4, action item 3).

### 3.6 Error Messages — CONFIRMED SAFE (NOT IN THREAT MODEL)

Error responses are generic and do not distinguish between "transfer doesn't exist" and "transfer in wrong state":
```
"Transfer not found or not in pending state"
"Transfer not found or payload not uploaded"
"Transfer not found"
```

No information leakage via error messages. Good practice.

## 4. New Findings (Not in Original Threat Model)

### 4.1 `token_name` Parameters Also Use Raw `str`

**Severity:** LOW
**Locations:**

```
Routes__Transfers.py:199 — check_token__token_name(self, token_name: str)
Routes__Transfers.py:215 — validate_token__token_name(self, token_name: str, ...)
```

These path parameters accept raw `str` and pass unvalidated values to the admin service client. Should be `Safe_Str__Id` for consistency.

### 4.2 CORS Configuration

CORS is configured via `setup_middleware__cors()` in the FastAPI app. This was not reviewed in the threat model but is relevant to the simple-token feature — CORS must allow the download page origins to call transfer APIs.

**Status:** Not blocking, but should be verified during implementation.

## 7. Overall Assessment

The simple-token threat model is **accurate and comprehensive** for the proposed feature. The two priority fixes (`access_token` type, rate limiting) were correctly identified. The `Safe_Str__Id` protection on `transfer_id` is effective and verified.

**The simple-token architecture is acceptable for its intended use case** (convenient sharing of non-sensitive files) provided:
1. `access_token` type is fixed before the feature ships
2. Rate limiting is implemented before production hardening
3. UI clearly communicates the security level difference between Mode 1 (simple token) and Modes 2/3 (random key)
