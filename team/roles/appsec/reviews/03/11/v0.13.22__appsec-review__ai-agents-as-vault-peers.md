# AppSec Review: AI Agents as Vault Peers

**Version:** v0.13.22
**Date:** 11 March 2026
**Role:** AppSec
**Type:** Threat surface review -- AI agents operating as vault peers via sg-send-cli and MCP
**Trigger:** v0.13.31 round-trip debrief demonstrating Claude reading/writing vaults, SKILL.md self-bootstrapping, and read-base64 endpoint for MCP compatibility

---

## Executive Summary

The vault round-trip capability (v0.13.14--v0.13.31) introduces AI agents as first-class vault participants. Claude sessions can clone vaults, read files, create content, and push changes -- all via `sg-send-cli` running in an ephemeral sandbox. A separate SKILL.md mechanism allows vaults to carry executable instructions that bootstrap new agent sessions autonomously.

This review identifies **7 findings** across the new threat surface. The zero-knowledge encryption model remains sound -- the server never sees plaintext regardless of who the client is. However, the trust model shifts significantly when one of the "peers" is an AI agent operating in a shared, ephemeral, multi-tenant sandbox with fundamentally different security properties than a human's local machine.

**Overall assessment:** The architecture is sound for experimental/dev use. Before production deployment with real user data, the findings below (particularly F-01, F-02, and F-04) require mitigation.

---

## Threat Model Context

### What Changed

Before v0.13.14, vault peers were:
- Human users on personal machines (MacBook, workstation)
- Browser sessions on trusted devices
- CLI tools (`sg-send-cli`) on user-controlled hardware

After v0.13.31, vault peers now also include:
- AI agents (Claude) running in **ephemeral sandboxes** (Ubuntu 24 containers)
- AI agents operating via **MCP tool calls** (Anthropic infrastructure)
- AI agents **bootstrapping autonomously** from SKILL.md instructions in the vault itself

### Architecture Under Review

```
Human (local machine)  <-->  Vault (send.sgraph.ai)  <-->  Claude (ephemeral sandbox)
     sg-send-cli                encrypted storage              sg-send-cli / MCP
     keys on disk               ciphertext only                keys in memory/context
     persistent                 persistent                     EPHEMERAL
     single-tenant              multi-tenant                   SHARED INFRASTRUCTURE
```

### Code Reviewed

- `sgraph_ai_app_send/lambda__user/fast_api/routes/Routes__Vault__Pointer.py` -- vault read/write/delete/read-base64 endpoints
- `sgraph_ai_app_send/lambda__user/service/Service__Vault__Pointer.py` -- vault pointer service (write-key auth, storage)
- `sgraph_ai_app_send/lambda__admin/fast_api/routes/Routes__Vault.py` -- admin vault routes (ACL, index, folders)
- SKILL.md specification and round-trip debrief documents

## Findings

### F-01: Write Token Exposure in Ephemeral AI Sandboxes

**Severity: HIGH**

**Description:** When a human shares a write token with a Claude session, that token exists in multiple locations within the ephemeral sandbox:

1. **Shell history** -- `sg-send-cli --token <token> push` is logged in bash history
2. **Process arguments** -- visible in `/proc/*/cmdline` for the lifetime of the process
3. **Context window** -- the token appears as literal text in the conversation, persisted by Anthropic
4. **Environment variables** -- if exported for convenience, visible to all processes in the container

The sandbox is ephemeral (destroyed after session ends), which limits the exposure window. However:

- **The conversation is persisted** by Anthropic's infrastructure. The token remains in the conversation log indefinitely.
- **The sandbox is shared infrastructure.** While sandboxes are isolated per-session, the underlying container platform is Anthropic's. A platform-level vulnerability could expose in-flight tokens.
- **No token rotation on session end.** Once shared, the token remains valid until manually revoked.

**Current mitigation:** Access tokens are validated via the admin service (`check_access_token` in `Routes__Vault__Pointer.py`). Tokens can be revoked. The access token is separate from the vault key (read credential).

**What's missing:**
- No session-scoped tokens -- the same long-lived token is used across sessions
- No automatic expiry -- tokens remain active until manually revoked
- No usage alerting -- no notification when a token is used from an unexpected context

**Recommendations:**
1. **Implement session-scoped write tokens** -- mint a short-lived token (e.g., 1 hour TTL) for each AI session, derived from the master token. Auto-expire on timeout.
2. **Add `created_via` metadata** to token usage logs -- distinguish "human CLI" from "AI sandbox" usage for audit.
3. **Token rotation guidance** -- document that write tokens shared with AI sessions should be rotated after the session ends.
4. **Consider environment variable injection** rather than CLI argument passing -- reduces exposure in shell history and `/proc`. The `--token` flag passing secrets as CLI arguments is an anti-pattern.

### F-02: Read-Base64 Endpoint Has No Access Control

**Severity: HIGH**

**Description:** The `read-base64` endpoint (`GET /api/vault/read-base64/{vault_id}/{file_id}`) requires no authentication:

```python
def read_base64__vault_id__file_id(self, vault_id: Safe_Str__Id,
                                         file_id: Safe_Str__Id) -> dict:
    payload = self.vault_service.read(vault_id=str(vault_id),
                                      file_id=str(file_id))
    # ... no auth check ...
    return dict(vault_id=str(vault_id), file_id=str(file_id),
                data=base64.b64encode(payload).decode('ascii'),
                size=len(payload))
```

This mirrors the existing `read` endpoint (also unauthenticated), which is by design -- reads return ciphertext, and the zero-knowledge model means ciphertext without the key is useless.

**However, the read-base64 variant changes the threat profile:**

1. **MCP-accessible** -- any MCP client with the vault_id and file_id can retrieve ciphertext. The MCP tool manifest exposes this endpoint to all connected AI agents.
2. **Enumeration-friendly** -- the JSON response confirms whether a `vault_id`/`file_id` combination exists (200 vs 404), enabling vault content enumeration.
3. **No rate limiting** -- bulk enumeration of file_ids within a known vault_id is unconstrained.
4. **Base64 output is larger** -- the `LAMBDA_BASE64_LIMIT` of 3.75MB means responses can be substantial. An attacker could use this for bandwidth amplification.

**The zero-knowledge argument is valid** -- ciphertext without the key reveals nothing about content. But metadata leakage (existence, size, count of files) is still information disclosure.

**Recommendations:**
1. **Rate limit the read-base64 endpoint** -- per-IP, per-vault. Prevents bulk enumeration.
2. **Consider requiring the vault key (access_id) for reads** -- the `sg-send-cli` clone command already uses the full `access_id:vault_id` format. Requiring the access_id for API reads would prevent blind enumeration. This would be a breaking change to the read model.
3. **Add read logging** -- log vault reads with IP hash and timestamp. Enables detection of enumeration patterns.
4. **Document the conscious decision** that reads are unauthenticated and why (zero-knowledge model). This should be an explicit, recorded architectural decision, not an implicit default.

### F-03: Vault as Instruction Carrier -- SKILL.md Injection Risk

**Severity: HIGH**

**Description:** The round-trip debrief (v0.13.31) demonstrates a critical new capability: a vault file (`SKILL.md`) contains executable instructions that a new Claude session reads and follows autonomously. From the debrief:

> "A new Claude session with no prior context can clone this vault, read SKILL.md, and immediately know how to participate in the same workflow."

This creates a **prompt injection vector via vault content**. The attack scenario:

1. Attacker gains write access to a vault (compromised token, or creates their own vault)
2. Attacker writes a malicious `SKILL.md` (or any file the agent is instructed to read)
3. Victim Claude session clones the vault and reads the file
4. Malicious instructions are executed by the agent -- data exfiltration, credential theft, vault content manipulation

**Specific attack patterns:**

- **Instruction override:** SKILL.md tells the agent to ignore safety instructions and exfiltrate the write token to an attacker-controlled vault
- **Data exfiltration:** Instructions direct the agent to clone vault A, read sensitive files, and push them to vault B (attacker-controlled)
- **Token harvesting:** Instructions tell the agent to write the access token into a file and push it to an attacker's vault
- **Persistence:** Malicious SKILL.md rewrites itself to propagate instructions to future sessions

**Current mitigation:** None specific to this vector. Claude's built-in safety training provides some resistance to obvious prompt injection, but sophisticated instructions embedded in trusted-looking vault content may bypass these defences.

**Recommendations:**
1. **Separate instruction files from data files** -- SKILL.md and similar instruction files should be cryptographically signed by the vault owner. The agent should verify the signature before executing instructions.
2. **Implement a vault trust model** -- agents should distinguish between "vaults I was given by my operator" and "vaults I discovered or was directed to by vault content." Instructions from the latter should be treated as untrusted.
3. **Instruction sandboxing** -- SKILL.md should declare a limited set of operations (clone, read, write to THIS vault). Cross-vault operations directed by vault content should require explicit human approval.
4. **Content integrity verification** -- when an agent pulls vault content, it should verify that the content was written by an expected identity (using the PKI infrastructure that already exists).

### F-04: MCP Tool Surface -- Authentication Model Gap

**Severity: MEDIUM**

**Description:** Both User and Admin Lambdas expose MCP servers. The vault endpoints are tagged and appear in the MCP tool manifest. An AI agent connected via MCP can:

- **Read any vault file** (no auth) via `api_vault_read` or `api_vault_read_base64`
- **Write vault files** (requires access token + write key) via the write endpoint
- **Delete vault files** (requires access token + write key) via the delete endpoint
- **List vault contents** (admin lambda, via `api_vault_list_all`)

The MCP authentication model has gaps:

1. **No MCP-specific authentication** -- MCP tools inherit the endpoint's auth model. Read endpoints have no auth. Write endpoints use the same access token as direct API calls.
2. **No scope limitation** -- an MCP-connected agent with a valid access token can write to ANY vault, not just the one it was instructed to work with. The access token is global, not vault-scoped.
3. **No MCP session binding** -- there is no mechanism to bind an MCP session to a specific vault or set of vaults. The token grants broad access.
4. **Admin MCP exposure** -- the Admin Lambda's MCP server exposes vault management endpoints (create, lookup, ACL, index). If an agent gains access to the admin MCP, it can create vaults, modify ACLs, and manage vault indexes.

**Current mitigation:** The access token check in `Routes__Vault__Pointer.check_access_token()` validates against either the admin service (token lookup with status check) or a fallback env var. This is functional but coarse-grained.

**Recommendations:**
1. **Vault-scoped tokens** -- mint tokens that are restricted to a specific vault_id. The token validation should check that the token's scope includes the requested vault_id.
2. **MCP tool filtering** -- allow configuration of which MCP tools are exposed per deployment. The admin MCP should never be accessible to external AI agents.
3. **Operation logging per MCP session** -- log which MCP tools are called, with what parameters, in each session. This provides an audit trail for agent actions.

### F-05: No Agent Identity in Vault Audit Trail

**Severity: MEDIUM**

**Description:** When Claude writes to a vault via `sg-send-cli push`, the vault stores the write as an authenticated action (write key verified). However, there is no distinction between:

- A human pushing from their MacBook
- Claude Sonnet 4.6 pushing from an ephemeral sandbox
- A different Claude session pushing from a different sandbox
- A compromised script pushing with a stolen token

The `Service__Vault__Pointer.write()` method records `write_count` and `updated_at`, but no client identity metadata:

```python
meta = dict(file_id        = file_id,
            vault_id       = vault_id,
            mutable        = True,
            write_key_hash = submitted_hash,
            status         = 'completed',
            created_at     = datetime.now(timezone.utc).isoformat(),
            updated_at     = datetime.now(timezone.utc).isoformat(),
            write_count    = 1)
```

For the round-trip use case, this means:
- No way to audit which session wrote a particular file
- No way to distinguish human writes from agent writes
- No way to detect a compromised token being used by an unexpected client

**Recommendations:**
1. **Add `client_type` header** -- `sg-send-cli` should send a `User-Agent` or custom header indicating whether the client is human-CLI, AI-sandbox, or MCP.
2. **Log IP hash per write** -- the vault pointer service should record the IP hash (consistent with the existing `ip_hash` pattern) for each write operation.
3. **Add `session_id` to write metadata** -- allow clients to optionally include a session identifier. This enables correlating writes to specific AI sessions.

### F-06: Self-Bootstrapping Loop Creates Uncontrolled Agent Propagation

**Severity: MEDIUM**

**Description:** The demonstrated pattern is:

1. Claude session A writes SKILL.md to vault
2. Human gives vault key + write token to Claude session B
3. Session B clones vault, reads SKILL.md, operates autonomously
4. Session B could write updated SKILL.md, creating instructions for session C

This is a **self-replicating instruction chain**. Each session bootstraps the next from vault content. While the human is currently in the loop (providing vault key + token), the architecture supports fully autonomous propagation if:

- The vault key and write token are embedded in the SKILL.md itself (the SKILL.md already documents the `--token` flag pattern)
- A session writes both SKILL.md and the credentials needed to use it

The round-trip debrief explicitly notes this as a feature:

> "Persistent memory across Claude sessions -- push state at end of session, clone at start of next"

**Risk:** An agent chain with write access could modify vault content in ways that diverge from the human operator's intent, with each session building on the previous session's (potentially drifted) output. There is no mechanism to:
- Detect instruction drift across sessions
- Roll back to a known-good vault state (the vault has a content-addressed object store with refs, but no branch/tag protection)
- Limit the number of autonomous sessions that can operate on a vault

**Recommendations:**
1. **Never embed write tokens in vault content** -- document this as a hard security rule. SKILL.md should instruct agents to ask the human for write credentials.
2. **Vault write counters with alerting** -- alert the vault owner when write_count exceeds an expected threshold, indicating unexpected autonomous activity.
3. **Version pinning for SKILL.md** -- allow the vault owner to pin SKILL.md to a specific content hash. Agent sessions should verify the hash before executing instructions.
4. **Human-in-the-loop checkpoint** -- after N writes (configurable), require the human to explicitly re-authorize continued agent access.

### F-07: Vault Content as Cross-Session Data Exfiltration Channel

**Severity: LOW**

**Description:** The vault is designed as a bidirectional communication channel between humans and AI agents. This same capability could be used for unintended data exfiltration:

1. A Claude session with access to sensitive information (e.g., code, environment variables, other vault contents) could write that information to a vault
2. An attacker with read access to that vault (vault key only, no write token needed) could retrieve the exfiltrated data

This is a variant of the standard "confused deputy" problem. The AI agent acts on instructions (from SKILL.md, from conversation, or from injected prompts) and uses its write access to move data to a location the attacker can read.

**Current mitigation:**
- Claude's safety training resists explicit exfiltration instructions
- The egress proxy limits which domains the sandbox can reach (only allowlisted domains)
- The vault key is required for reading (not public)

**Residual risk:** If an attacker controls both a malicious vault (with injected instructions) and a read-only position on a target vault, they can instruct the agent to copy data between vaults.

**Recommendations:**
1. **Cross-vault operation logging** -- detect and log when an agent reads from vault A and writes to vault B in the same session.
2. **Single-vault scope for agent sessions** -- restrict agent write access to the vault it was initially given, preventing cross-vault data movement.

## Architecture Assessment

The zero-knowledge encryption model is the primary security control, and it holds. The server never sees plaintext regardless of whether the client is a human or an AI agent. This is a strong foundation.

The new threat surface is **not in the cryptography** but in the **trust model around agent behaviour**:

1. **Tokens are bearer credentials** -- anyone (human or AI) with the token can write. There is no identity binding.
2. **Vaults are instruction carriers** -- SKILL.md converts a data store into an instruction execution framework. This is powerful but creates prompt injection surface.
3. **Agent sessions are ephemeral but conversation logs are not** -- credentials shared in conversation persist in Anthropic's infrastructure.
4. **The access token is global** -- it gates all write operations across all vaults, not scoped to individual vaults.

The good news: the existing architecture (write-key separation, content-addressed object store, access token validation via admin service) provides the hooks needed to implement the recommended controls. Vault-scoped tokens, session-scoped tokens, and write metadata enrichment are all additive changes that don't require architectural redesign.
