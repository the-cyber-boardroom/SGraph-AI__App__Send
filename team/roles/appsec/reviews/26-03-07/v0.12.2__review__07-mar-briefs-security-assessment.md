## 1. sg-layout Event Model Security

### 1.1 Registration Protocol — LOW RISK

The `sg-layout:register` event with synchronous callback is not a meaningful threat — registration only grants layout metadata (panelId, layoutRef, depth, bounds). A component must already be in the DOM to register.

### 1.2 sg-commands — MODERATE RISK

No sender authentication on the broadcast bus. Any component can call `window.sgCommands.send('target-panel-id', ...)`.

**Recommendation:** For sensitive commands (`js-exec`, `state-push`), support optional **signed commands mode** where sender signs with panel private key.

### 1.3 stopPropagation — SUFFICIENT

DOM spec guarantees. Each sg-layout catches its own subtree's events only.

### 1.4 PKI Per Panel — Exportable, but Encrypted

- **YES exportable** — digital twin loses value without persistent identity
- **Encrypt before export** — derive key from user passphrase, encrypt with AES-256-GCM
- Make exportability a config option (ephemeral for low-sensitivity, persistent for vault/data rooms)

### 1.5 js-exec Whitelist — SUFFICIENT with Caveats

- Methods must validate their own arguments
- No `eval()` or `new Function()` — use `component.sgCommands[methodName]?.call()`
- `js-exec` should not return values by default (prevents data exfiltration)

## 2. Identity Infrastructure Security

### 2.1 Web of Trust — SOUND

Multiple independent verification points make impersonation significantly harder.

| Point | Strength |
|---|---|
| DNS TXT record | Strong (requires domain access) |
| Company website | Strong (requires hosting access) |
| GitHub repo | Medium (check canonical repo, not forks) |
| LinkedIn | Weak (social signal only) |
| Companies House | Strong (government-backed) |

### 2.2 DNS TXT Threats

1. DNS hijacking — mitigate with DNSSEC
2. DNS cache poisoning — mitigate with DNSSEC + DoH
3. Registrar compromise — registrar lock + strong MFA

**Recommendation:** Enable DNSSEC on sgraph.ai.

### 2.3 Agent Identity Attack Vectors

1. **Compromised human key** — Store in HSM or vault encrypted storage
2. **Agent key exfiltration** — Keys generated/stored in vault, loaded only for signing
3. **Stale identities** — Revocation list in `.well-known/sg-identity.json`
4. **Cross-repo impersonation** — Identity document references canonical repo URL

### 2.4 Authentication by Decryption — SOUND

Correct principle. Caveats: proves key possession not identity (acceptable with controlled keys), no non-repudiation (pair with signatures), no forward secrecy with static keys (use ephemeral ECDH).

### 2.5 .well-known Protections

CORS: `Access-Control-Allow-Origin: *`. HTTPS only. Version and timestamp. Self-signed integrity. No secrets.

## 3. Temp Token Security

### 3.1 Shared Token — ACCEPTABLE RISK

Limited credits, human monitors, personal token replaces it. Add: per-IP rate limits, weekly rotation, usage alerting.

### 3.2 Hash Fragment — ACCEPTABLE

Never sent to server. Clear from URL after reading (`history.replaceState`).

### 3.3 Client-Side Decryption — SOUND

Same pattern as core SG/Send flow. **Concern:** Token in localStorage is XSS-accessible. Mitigate with strict CSP headers.

### 3.4 Monitoring Requirements

| What | Alert Threshold |
|---|---|
| Shared token usage rate | > 5x expected customer count |
| Geographic distribution | > 10 countries for 10 customers |
| Failed decryption attempts | > 20 per hour |
| Token rotation age | > 7 days |

## 4. Individual File Storage Security

### 4.1 Per-File Encryption — SOUND

Follows established patterns. Key management complexity requires clear UI.

### 4.2 CRITICAL: Tree Structure Metadata Leakage

Tree contains filenames, sizes, types, and **plaintext hashes** (SHA-256 of plaintext).

**Plaintext hashes enable known-document attacks:** attacker checks if a file matches a known document without the decryption key.

**Recommendation:**
- **Remove plaintext hashes from tree.** Use ciphertext hashes for integrity verification.
- Consider encrypting filenames (public layer: count/size/type, encrypted layer: names/paths)

### 4.3 CRITICAL: File Update Authentication

Read access (decryption key) must NOT grant write access. If I share a file, the recipient shouldn't be able to update it.

**Recommendation:** Write auth via signing key (never shared), separate from read auth (decryption key). Aligns with vault architecture (write key + read key).

### 4.4 100-File Limit

No security rationale. Performance boundary only. Document accordingly.

## 5. Web Components Security

### 5.1 Shadow DOM — Encapsulation, NOT Isolation

`mode: 'open'` allows `element.shadowRoot.querySelector()` from parent context.

**Recommendation:** Use `mode: 'closed'` for security-sensitive components (key management, token display).

### 5.2 Event Communication — LOW XSS Risk

Components must treat event data as untrusted. Never use `innerHTML` with event data.

### 5.3 CSS Custom Properties — NEGLIGIBLE Risk

Cosmetic only. **Exception:** Never use CSS custom properties in `url()` functions.

## 6. Critical Security Recommendations

### Priority 1 — Fix Before Shipping

| Item | Risk | Action |
|---|---|---|
| Plaintext hashes in tree | HIGH | Remove from public layer |
| File update authentication | HIGH | Separate write auth (signing key) |
| Token in localStorage (XSS) | MEDIUM | Strict CSP headers |

### Priority 2 — Threat Model Before Implementation

| Item | Action |
|---|---|
| sg-commands sender auth | Design signed commands before implementing |
| Agent identity key storage | Define protection model before creating framework |
| Cross-repo collaboration | Define trust model for repo-to-repo access |

### Priority 3 — Can Proceed Without Review

| Item | Why Safe |
|---|---|
| sg-layout v0.1.0 core | No security-sensitive data flow |
| Translation pipeline | Public, non-sensitive data |
| PWA manifest | Standard browser feature |
| DNS TXT records | Public key publication |
| P0 bug fixes | Token sanitisation improves security |
| .well-known/sg-identity.json | Public information only |
