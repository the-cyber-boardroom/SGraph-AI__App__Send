## 1. sg-layout Event Model Security

### 1.1 Registration Protocol — Event Injection Risk: LOW

The `sg-layout:register` event uses `composed: true, bubbles: true` with a synchronous callback. Registration only grants layout metadata (panelId, bounds, depth). A malicious component gains resize events and commands but can't access other panels' data. **No additional security needed.**

### 1.2 Malicious Component Exploitation: MODERATE RISK

A legitimately registered component can send commands to any panel via `window.sgCommands.send()` — no sender authentication.

**Recommendation:** For sensitive commands (`js-exec`, `state-push`), sg-commands should support optional **signed commands mode** where the sender signs with its panel private key.

### 1.3 stopPropagation: SUFFICIENT

DOM spec guarantees prevent bypass. Each sg-layout claims its own events correctly.

### 1.4 PKI Per Panel — Private Key Exportability

**Recommendation: Exportable, but encrypted.**

- Derive encryption key from user passphrase or device key
- Encrypt private key with AES-256-GCM before export
- Make exportability a configuration option (ephemeral for low-sensitivity, encrypted export for high-sensitivity)

### 1.5 js-exec Whitelist: SUFFICIENT WITH CAVEATS

- Method arguments need validation by each component
- No `eval()` or `new Function()` in the handler
- No automatic return value exposure (prevents data exfiltration)

## 2. Identity Infrastructure Security

### 2.1 Web of Trust Model — SOUND

Multi-point verification (DNS, website, GitHub, LinkedIn) makes impersonation require multi-system compromise.

| Point | Strength |
|---|---|
| DNS TXT record | **Strong** (requires registrar access; DNSSEC adds crypto verification) |
| Company website page | **Strong** (requires hosting access; HTTPS proves domain) |
| GitHub repo (.well-known/) | **Medium** (verify canonical repo, not forks) |
| LinkedIn profile | **Weak** (social signal only) |
| Companies House | **Strong** (government-backed legal document) |

### 2.2 DNS TXT Records — Threat Model

- **DNS hijacking:** Mitigate with DNSSEC
- **Cache poisoning:** Mitigate with DNSSEC + DoH
- **Registrar compromise:** Mitigate with registrar lock, strong MFA

**Recommendation:** Enable DNSSEC on sgraph.ai. Use registrar lock.

### 2.3 Agent Identity Chain of Trust — Attack Vectors

1. **Compromised human key:** Store in HSM or vault encrypted storage. Never plaintext on disk.
2. **Agent key exfiltration:** Store in vault. Load to memory only for signing, then discard.
3. **Stale agent identities:** Include revocation list in `.well-known/sg-identity.json`.
4. **Cross-repo impersonation:** Agent identity must reference canonical repo URL.

### 2.4 "Authentication by Decryption" — SOUND

Cryptographically correct. Caveats: proves key possession (not identity if key is shared), no non-repudiation (pair with signatures), consider ephemeral ECDH for forward secrecy.

### 2.5 .well-known/sg-identity.json Protections

1. CORS: `Access-Control-Allow-Origin: *`
2. HTTPS only
3. Version and timestamp for change detection
4. Self-signed integrity (signature field)
5. Public information only — no secrets

## 3. Temp Token Security

### 3.1 Shared Token Model — ACCEPTABLE RISK

Limited credits bound financial exposure. Human monitors and rotates.

**Additional recommendations:**
- Per-IP rate limits on shared token
- Weekly rotation minimum
- Usage alerting when credits exceed expected levels

### 3.2 Token in URL Hash Fragment — ACCEPTABLE

Hash fragment not sent to server (same pattern as SG/Send decryption keys). **Clear fragment after reading** via `history.replaceState`.

### 3.3 Client-Side Decryption — SOUND

Same pattern as core SG/Send flow. **Concern:** Token in localStorage accessible by any JS on same origin. Ensure strict CSP headers.

### 3.4 Monitoring Needed

| What | Threshold | Why |
|---|---|---|
| Usage rate | > 5x expected customer count | Possible leak |
| Geographic distribution | > 10 countries for 10 customers | Unusual sharing |
| Failed decryptions | > 20/hour | Brute force attempt |
| Token age | > 7 days | Stale token risk |

## 4. Individual File Storage Security

### 4.1 Per-File Encryption — SOUND

Different keys per file follows established patterns. Key management complexity must be reflected in the UI.

### 4.2 Tree Structure — CRITICAL: Metadata Leakage

**CRITICAL: Remove plaintext hashes from tree.** SHA-256 of plaintext enables:
- Known-document attacks (compare suspected document hash)
- Cross-transfer deduplication

**Recommendation:** Include ciphertext hash (integrity after download) and encrypted plaintext hash (verifiable only by key holder). Consider encrypting filenames in the tree.

### 4.3 File Update Authentication — CRITICAL: INSUFFICIENT

Read access (decryption key) must NOT grant write access. Shared decryption key means any recipient can update files.

**Recommendation:** Separate write authentication via signing key (never shared). Aligns with existing vault write key / read key separation.

### 4.4 100-File Limit — No Security Rationale

Performance boundary only. Document as such.

## 5. Web Components Security

### 5.1 Shadow DOM — Encapsulation, Not Isolation

`mode: 'open'` allows `shadowRoot` access from parent context. **Use `mode: 'closed'` for security-sensitive components** (key management, token display).

### 5.2 Event-Driven Communication — XSS: LOW

Custom events carry data in `event.detail`. If rendered as HTML without sanitisation, XSS is possible. **Components must treat event data as untrusted. Never use `innerHTML` with event data.**

### 5.3 CSS Custom Properties — Injection: NEGLIGIBLE

Purely cosmetic impact. **Never use custom properties in `url()` functions** (prevents external URL requests).

## 6. Critical Security Recommendations

### Priority 1 — Fix Before Shipping

| Item | Risk | Action |
|---|---|---|
| **Plaintext hashes in tree** | **HIGH** | Remove from public layer |
| **File update auth** | **HIGH** | Separate write auth (signing key) |
| **Token in localStorage** | **MEDIUM** | Strict CSP headers |

### Priority 2 — Threat Model Before Implementation

| Item | Action |
|---|---|
| sg-commands sender auth | Design signed commands before implementing |
| Agent identity key storage | Define storage and protection before creating framework |
| Cross-repo collaboration | Define trust model for repo-to-repo agent communication |

### Priority 3 — Can Proceed Without Security Review

| Item | Why Safe |
|---|---|
| sg-layout core | No security-sensitive data flow |
| Translation pipeline | Public, non-sensitive data |
| PWA manifest | Standard browser feature |
| DNS TXT, .well-known | Public information only |
| P0 bug fixes | Token sanitisation improves security |
