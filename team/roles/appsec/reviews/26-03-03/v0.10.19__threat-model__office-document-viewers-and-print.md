# Threat Model — Office Document Viewers, Print Support, and CDN Dependencies

**Role:** AppSec
**Version:** v0.10.19
**Date:** 2026-03-03
**Classification:** Internal — Security Sensitive
**Scope:** DOCX viewer (mammoth.js), XLSX viewer (SheetJS), PPTX fallback, markdown/DOCX/XLSX print, CDN script loader, blob URL iframe strategy

---

## 1. Executive Summary

Three new preview capabilities were added to the download component: DOCX rendering (via mammoth.js), XLSX rendering (via SheetJS), and a print button for rendered content. These features introduce **new attack surface** because they process attacker-controlled binary file content and render it as HTML inside the browser. A malicious sender could craft a .docx or .xlsx file specifically to exploit these renderers.

**Overall risk assessment: MEDIUM** — The iframe sandbox provides strong containment, but there are specific findings that need attention. The highest-risk item is the XLSX renderer's `allow-scripts allow-same-origin` sandbox combination.

## 3. Findings

### FINDING OV-01 (HIGH): XLSX iframe has `allow-scripts allow-same-origin` sandbox

**File:** `send-download.js`, `_renderXlsxAsync()`, line ~535
**Sandbox value:** `sandbox="allow-scripts allow-same-origin"`

**Issue:** The combination of `allow-scripts` and `allow-same-origin` in a sandbox effectively **negates the sandbox** — a script running inside the iframe can call `frameElement.removeAttribute('sandbox')` or navigate the parent. This is [documented by MDN](https://developer.mozilla.org/en-US/docs/Web/HTML/Element/iframe#attr-sandbox) as a dangerous combination.

The `allow-scripts` was added because the XLSX renderer includes a `<script>` tag for tab switching. However, this means any HTML injected by SheetJS's `sheet_to_html()` could execute arbitrary JavaScript in the same origin as the parent page.

**Attack scenario:**
1. Attacker crafts an .xlsx file where a cell value contains `<img src=x onerror="...">` or a formula that resolves to HTML
2. SheetJS `sheet_to_html()` renders cell values — if any are not properly escaped, they execute in the iframe
3. Because `allow-same-origin` is also set, the script has full access to `window.parent`, `localStorage`, cookies, and can exfiltrate the decrypted content

**Likelihood:** Medium — depends on whether SheetJS escapes all cell values in `sheet_to_html()`. SheetJS does HTML-escape cell text in most cases, but edge cases (rich text, hyperlinks, embedded objects) may not be fully covered.

**Impact:** High — full XSS in the application's origin. Could steal decryption keys from URL fragments, exfiltrate decrypted file contents, modify the DOM.

**Recommendation:**
- **Option A (Preferred):** Remove `allow-scripts` from the XLSX iframe. Rewrite the tab-switching as a pre-rendered approach (show/hide via CSS `:target` pseudo-class or pre-generate separate iframes per sheet). This eliminates script execution entirely.
- **Option B:** Keep `allow-scripts` but remove `allow-same-origin`. The iframe becomes a unique opaque origin, preventing parent access. Tab switching scripts still work. The tradeoff: `contentWindow.print()` will fail (print button won't work for XLSX via the primary path, but the blob URL fallback will still function).
- **Option C:** Move the XLSX rendering to a Web Worker or use `srcdoc` with a CSP meta tag (`<meta http-equiv="Content-Security-Policy" content="script-src 'unsafe-inline'; frame-ancestors 'none';">`).

**Severity: HIGH**

### FINDING OV-02 (MEDIUM): mammoth.js HTML output injected into iframe without sanitisation

**File:** `send-download.js`, `_renderDocxAsync()`, line ~499-501
**Code:** `const html = result.value;` → injected directly into `<body>${html}</body>`

**Issue:** mammoth.js converts DOCX to HTML. The output HTML is inserted directly into the iframe document without any sanitisation. mammoth.js does escape most text content, but it preserves:
- `<a href="...">` links — potentially `javascript:` URIs
- `<img src="...">` tags — embedded images from the DOCX (base64 data URIs)
- HTML structure from the DOCX's internal XML

If mammoth.js has a bug or an edge case where it emits unescaped content, it would execute in the iframe context.

**Mitigating factors:**
- The iframe uses `sandbox="allow-same-origin"` (no `allow-scripts`), so even if HTML with `<script>` tags or `onerror` handlers is injected, **scripts will not execute**. This is a strong mitigation.
- mammoth.js is a mature, widely-used library (1.8M+ weekly npm downloads).

**Residual risk:** Low, because the sandbox blocks script execution. However, CSS injection or phishing content (e.g., a DOCX that renders as "Enter your password below") is still possible within the iframe.

**Recommendation:**
- Consider adding a DOMPurify pass on the mammoth output before iframe injection: `const clean = DOMPurify.sanitize(html, { ALLOWED_TAGS: [...], ALLOWED_ATTR: [...] })`. This is defence-in-depth.
- The current sandbox without `allow-scripts` is the primary control and is adequate for now.

**Severity: MEDIUM (mitigated to LOW by sandbox)**

### FINDING OV-03 (MEDIUM): Print fallback uses `panel.innerHTML` unsanitised

**File:** `send-download.js`, `_printPreview()`, line ~570
**Code:** `printWindow.document.write(...${panel.innerHTML}...)`

**Issue:** The last-resort print fallback copies the preview panel's `innerHTML` into a new `window.open()` document via `document.write()`. If the preview panel contains attacker-controlled HTML (e.g., from mammoth.js or SheetJS output that was rendered into the panel), this HTML is now written into a **new window with no sandbox restrictions**.

**Attack scenario:**
1. Attacker crafts a DOCX/XLSX with HTML injection payload
2. The content is safely contained in the sandboxed iframe during normal viewing
3. User clicks "Print" → if the primary and secondary print paths fail, the fallback copies the panel's innerHTML (which includes the iframe element and any surrounding HTML) into an unsandboxed window
4. However: the innerHTML of the panel contains `<iframe>` elements, not their rendered content. The iframe's internal document is NOT accessible via innerHTML (cross-origin or opaque). So the actual attacker-controlled HTML inside the iframe does NOT propagate to the print window.

**Reassessment:** After analysis, the risk is **lower than initially appears** because `panel.innerHTML` captures the outer iframe *element* (with a `src` attribute), not the iframe's *document content*. The attacker-controlled HTML stays inside the iframe's blob/data URL, which the print window cannot render.

**Residual risk:** Low. The fallback mostly produces an empty print page for iframe-based content. But if future changes add non-iframe rendered content directly into the panel DOM, this could become a real XSS vector.

**Recommendation:**
- Add a comment documenting why this fallback is safe (only captures outer iframe elements).
- Consider restricting the fallback to only fire for non-iframe content, or sanitise `panel.innerHTML` before injection.

**Severity: MEDIUM (currently mitigated to LOW, but fragile)**

### FINDING OV-04 (MEDIUM): CDN supply-chain risk — loading third-party JavaScript at runtime

**File:** `send-download.js`, `_loadCdnScript()`, lines ~472-483
**CDN URLs:**
- `https://cdnjs.cloudflare.com/ajax/libs/mammoth/1.8.0/mammoth.browser.min.js`
- `https://cdn.sheetjs.com/xlsx-0.20.3/package/dist/xlsx.full.min.js`

**Issue:** The CDN script loader injects third-party `<script>` tags at runtime. These scripts execute with **full access** to the page's origin — they can read localStorage, cookies, DOM, and the decrypted file bytes in memory. If either CDN is compromised, or a supply-chain attack replaces the library, the attacker gains full control of the recipient's browser session.

**Risk factors:**
- **cdnjs.cloudflare.com** — operated by Cloudflare, generally trusted, but has had incidents (e.g., cdnjs RCE in 2021). The URL is version-pinned, which helps.
- **cdn.sheetjs.com** — operated by the SheetJS maintainer (a single developer), not a major CDN provider. This is a higher-risk origin. The version is pinned but there are no SRI (Subresource Integrity) hashes.

**Attack scenarios:**
1. CDN compromise: attacker replaces the JS file at the pinned URL → immediate code execution on all recipients who open DOCX/XLSX files
2. DNS hijacking of cdn.sheetjs.com → attacker serves malicious script
3. BGP hijack targeting the CDN → same effect

**Mitigating factors:**
- Scripts are only loaded on-demand (when a user actually opens a DOCX/XLSX file), reducing the window of exposure
- SGraph Send is a zero-knowledge system, but the CDN scripts execute **after decryption**, so they have access to plaintext content

**Recommendation:**
- **Add Subresource Integrity (SRI) hashes** to both script loads. This is the single most impactful fix. With SRI, even if the CDN is compromised, the browser will refuse to execute a script whose hash doesn't match. Implementation:
  ```javascript
  static _loadCdnScript(url, globalName, integrity) {
      // ...
      script.integrity = integrity;
      script.crossOrigin = 'anonymous';
      // ...
  }
  ```
- **Consider vendoring** the libraries (same as JSZip is vendored today) to eliminate the CDN dependency entirely. This is the safest option for a zero-knowledge encryption product.
- At minimum, add a Content Security Policy (CSP) header that allowlists only the specific CDN domains for `script-src`.

**Severity: MEDIUM (HIGH if targeting a high-value recipient)**

### FINDING OV-05 (LOW): XLSX iframe uses data: URI — inconsistent with DOCX/markdown blob URL fix

**File:** `send-download.js`, `_renderXlsxAsync()`, line ~534-535
**Code:** `const encoded = btoa(unescape(encodeURIComponent(iframeDoc)));` → `src="data:text/html;base64,..."`

**Issue:** The DOCX and markdown iframes were switched from `data:` URIs to blob URLs to fix cross-origin sandbox console errors. The XLSX iframe still uses a `data:` URI. This is inconsistent and will produce the same console warnings.

More importantly, `data:` URIs are treated as **unique opaque origins** in most browsers, which means the `allow-same-origin` in the sandbox doesn't grant parent-page access. This is actually **safer** than a blob URL for the XLSX case (given OV-01), but it's accidental rather than intentional.

**Recommendation:**
- If OV-01 is fixed by removing `allow-scripts`, switch XLSX to a blob URL for consistency
- If `allow-scripts` is kept, **keep the `data:` URI** — the opaque origin partially mitigates OV-01 by preventing parent access despite `allow-same-origin`
- Document the intentional choice either way

**Severity: LOW**

### FINDING OV-06 (LOW): Markdown blob URL + sandbox="allow-same-origin" enables CSS injection

**File:** `send-download.js`, `_renderMarkdownContent()`, line ~424
**Sandbox value:** `sandbox="allow-same-origin"`

**Issue:** The markdown renderer uses `MarkdownParser.parse()` which escapes HTML entities, preventing script injection. However, inline CSS can be injected through certain markdown constructs if future parser changes introduce HTML passthrough. With `allow-same-origin` and blob URLs, the iframe shares the page's origin.

**Current state:** The MarkdownParser is robust — it escapes all text via `_escape()` before rendering. No HTML passthrough exists. This finding is informational.

**Recommendation:** No action needed now. The MarkdownParser's security-first design is correct. Monitor for any future changes that add raw HTML passthrough.

**Severity: LOW (informational)**

---

### FINDING OV-07 (INFO): Blob URL memory leaks

**File:** `send-download.js`, `_renderMarkdownContent()`, `_renderDocxAsync()`
**Code:** `URL.createObjectURL(blob)` — no corresponding `URL.revokeObjectURL()`

**Issue:** Each time markdown or DOCX content is rendered (including when toggling raw/rendered view), a new blob URL is created but never revoked. This causes a memory leak. The existing `_objectUrl` tracking pattern (used for images and PDFs) is not applied to the markdown/DOCX blob URLs.

**Impact:** No security impact. Functional impact only (memory growth during long sessions or repeated view toggles).

**Recommendation:** Track blob URLs and revoke them on re-render, matching the existing pattern used for images/PDFs.

**Severity: INFO**

## 5. Recommended Priority Actions

### Immediate (before next deployment)

1. **Fix OV-01:** Remove `allow-scripts` from the XLSX iframe. Replace the tab-switching `<script>` with a CSS-only solution (e.g., radio buttons + `:checked` sibling selectors, or render one sheet at a time with re-render on tab click from the parent).

### Short-term (within 1-2 sessions)

2. **Fix OV-04:** Add SRI hashes to both CDN scripts. Generate with:
   ```bash
   curl -s https://cdnjs.cloudflare.com/ajax/libs/mammoth/1.8.0/mammoth.browser.min.js | openssl dgst -sha384 -binary | openssl base64 -A
   curl -s https://cdn.sheetjs.com/xlsx-0.20.3/package/dist/xlsx.full.min.js | openssl dgst -sha384 -binary | openssl base64 -A
   ```

3. **Fix OV-05:** Switch XLSX iframe to blob URL (once OV-01 is fixed and `allow-scripts` is removed).

### Medium-term

4. **Vendor the libraries** (like JSZip) to eliminate CDN dependency entirely — strongest stance for a zero-knowledge product.

5. **Fix OV-07:** Add blob URL lifecycle management for markdown and DOCX iframes.
