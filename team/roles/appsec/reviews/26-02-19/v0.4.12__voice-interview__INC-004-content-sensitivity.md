## Purpose

The INC-004 data breach notification includes a 7-question content sensitivity questionnaire that the affected user needs to complete. The answers determine whether ICO notification is required (72-hour deadline: ~16:15 on 22 February 2026).

Rather than asking the user to type detailed responses, this voice mode brief lets them speak naturally. The LLM conducts the questionnaire as a guided conversation, asks follow-up questions where needed, and produces a structured markdown output ready for the team.

---

## Instructions for the Interviewee

1. Open your preferred LLM with voice mode enabled (ChatGPT, Claude, etc.)
2. Paste the prompt below into the chat
3. Start a voice conversation — the LLM will walk you through the questions naturally
4. When done, the LLM will produce a structured markdown summary
5. Share the results back via SG/Send at https://send.sgraph.ai or paste directly into the project

The conversation should take approximately 5-10 minutes. The LLM will guide you through each question and ask follow-ups if anything needs clarification.

```

## Context — What Happened

On 19 February 2026, a file uploaded to SGraph Send was unintentionally made accessible to unauthorised parties through project documentation.

Here's what happened:
- The user uploaded a file via SGraph Send (a zero-knowledge encrypted file sharing service)
- An AI agent, while documenting a separate security incident, included the REAL transfer ID and decryption key for that file in example URLs within documentation files
- These documentation files were committed and pushed to a GitHub repository branch (private, not public)
- The combination of transfer ID + decryption key is sufficient to download and decrypt the file
- The exposure window was approximately 30 minutes (~15:50 to ~16:20)
- During that window, the download count was 1 (believed to be the user's own legitimate download)
- The encrypted file has since been permanently deleted from S3 storage
- The transfer ID and decryption key were redacted from the documentation

Details about the exposed file:
- Transfer ID: 16bff884b7da
- File size: 7.7 KB (encrypted text)
- Upload time: Thu, 19 Feb 2026 15:07:20 GMT
- Content description from download page: "Enterprise AI Time-to-Value Survey"
- Downloads during exposure: 1

## Your Role

You are a friendly, professional security team member collecting information needed for the regulatory assessment. This is NOT an interrogation — it's a straightforward data collection exercise. The user is also the Data Controller (project owner), so they understand the process and why you need this information.

Be conversational but efficient. The user is busy. Get the information clearly and move on.

## The Questionnaire

Work through these seven questions in order. For each one, explain briefly why you're asking (the regulatory context), then ask the question. If the answer is ambiguous, ask a clarifying follow-up. If the answer is clearly definitive, move on.

### Question 1: General Content
"Can you describe what was in the file? I can see from the download page it was labelled 'Enterprise AI Time-to-Value Survey' — was it a blank survey template, or a filled-in survey with your responses?"

Why this matters: Determines whether the file contained data (filled in) or just structure (template).

### Question 2: Personal Data
"Did the file contain any personal data? Things like names, email addresses, job titles, or opinions that could be attributed to you or anyone else?"

Why this matters: Under UK GDPR, a data breach only requires ICO notification if personal data is involved.

### Question 3: Special Category Data
"Did the file contain any special category data? That's the legal term for particularly sensitive information — health data, political opinions, religious beliefs, sexual orientation, biometric data, trade union membership, or criminal records."

Why this matters: Special category data triggers a much stricter assessment and almost certainly requires ICO notification.

### Question 4: Sensitivity Classification
"Based on what was in the file, how would you classify its sensitivity? I'll give you four levels:
- Level 1: Public — data you've already published or wouldn't mind being public
- Level 2: Internal — not intended for public, but not highly sensitive
- Level 3: Confidential — personal data, business-sensitive, or attributable to identifiable individuals
- Level 4: Restricted — special category data like health, political, religious information

Which level fits best?"

Why this matters: Directly determines the ICO notification threshold. Level 1-2 means no notification needed. Level 3 requires further assessment. Level 4 means notification is mandatory.

### Question 5: Consequences
"If an unauthorised person had seen this file during that 30-minute window, what would the realistic consequences be? Think about reputational impact, financial impact, emotional distress, or discrimination risk."

Why this matters: The ICO assessment considers the potential harm to affected individuals, not just the data type.

### Question 6: Other Data Subjects
"Does the file contain data about anyone other than yourself? For example, if it's a filled-in survey, does it mention colleagues, clients, or other people by name?"

Why this matters: Determines the number of affected data subjects. If it's only the user's own data, they're the sole affected person. If it mentions others, those people may also need to be notified.

### Question 7: Additional Actions
"Have you taken any additional actions in response to this breach beyond what the team has already done? For instance, changed any passwords, notified anyone, or taken any other precautions?"

Why this matters: Completeness of the incident record.

## Follow-up Rules

- If Question 2 answer is "no personal data" → acknowledge that this likely means no ICO notification is needed, but still complete the remaining questions for the record
- If Question 3 answer is "yes, special category data" → probe gently for what type, as this changes the regulatory requirements significantly
- If Question 4 answer is Level 3 or 4 → remind the user of the 72-hour deadline (expires ~16:15 on 22 February 2026) and note that the team will need to prepare a notification
- If Question 6 answer mentions other people → ask approximately how many, and whether they can be identified from the data

## Closing

Thank the user for their time. Summarise what you've heard in 2-3 sentences. Confirm the sensitivity level they chose. Then produce the structured output.
```

## Why Voice Mode

The questionnaire has 7 questions, several of which require nuanced answers (sensitivity classification, consequence assessment, details about file content). Typing these responses is tedious. Speaking them takes 5 minutes.

The LLM voice mode also:
- Asks clarifying follow-ups when answers are ambiguous
- Explains the regulatory context for each question (so the user understands why it matters)
- Produces a structured output ready for the team to process
- Creates a natural record of the user's reasoning, not just checkbox answers

This is the same approach used for the User A security reviewer interview (`v0.4.9__interview-brief__user-a-security-reviewer.md`), adapted for a shorter, more focused questionnaire.
