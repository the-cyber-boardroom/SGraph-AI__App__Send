## 1. Executive Summary

This batch of 9 briefs represents the most significant expansion of SGraph Send's feature surface since the initial PKI architecture briefs. The proposals introduce one-time secrets, symmetric-key data rooms, a password vault, browser storage inspection, PKI-friendly URLs, Chrome password sync for private keys, versioned editing, and a complete URL/SPA/CloudFront architecture -- all while proposing a parallel security review framework to govern the expanding codebase.

**Overall security posture: SOUND architecture with specific risks that require mitigation before implementation.**

The briefs demonstrate mature security thinking. The metadata leak analysis in the one-time secret brief is genuinely excellent -- the decision to enforce opaque server-generated IDs in symmetric mode and encrypt metadata separately in PKI mode is the correct design. The per-file security review framework is a well-structured approach to managing the growing attack surface. The zero-knowledge property is preserved across the majority of proposals.

However, this batch introduces several new trust boundaries, new client-side storage of high-value secrets (private keys, vault passwords), and new endpoints that collectively expand the attack surface substantially. Five specific issues require resolution before implementation proceeds:

1. **HIGH: Chrome password sync stores private keys in a third-party credential store** -- introduces Google as a trust dependency for the most sensitive material in the system.
2. **HIGH: Password vault creates a new high-value target on the client side** -- plaintext passwords exist in browser memory during vault operations.
3. **MEDIUM: PKI-mode human-friendly URLs leak customer intent to the server** -- document names like `quarterly-report` or `contract-v3` are stored server-side in the URL path, contradicting the zero-knowledge property.
4. **MEDIUM: Version chain metadata reveals content modification patterns** -- edit frequency, editor identities, and version counts leak information about document importance and collaboration patterns.
5. **MEDIUM: Browser storage viewer tool creates a convenient exfiltration interface** -- the "Export All as JSON" feature exports private keys, vault contents, and contacts in one click.

## 3. Critical Analysis: Metadata Leak Risk

### 3.1 Assessment of the One-Time Secret Brief's Metadata Leak Section

The one-time secret brief's metadata leak analysis (section "Metadata Leak Risk: Human-Friendly Short Codes") is **the strongest security analysis in this entire batch.** The design principle -- "We should not provide easy ways for users to operate unsafely and then blame them for creating security incidents" -- should be elevated to a core project security principle.

**The symmetric-mode decision is correct.** Enforcing server-generated opaque IDs (`xK9mP2qR`) with no user-chosen names eliminates the metadata leak vector entirely. The server stores: an encrypted blob, a random ID, a timestamp, and a hashed IP. A full database dump reveals nothing about customer intent.

**The PKI-mode decision (encrypted metadata file alongside encrypted content) is architecturally elegant.** Two encrypted files per secret -- content and metadata -- means the friendly name exists only in ciphertext the server cannot read. The recipient decrypts both to see the friendly name in their vault. This design preserves zero-knowledge while providing excellent UX.

### 3.2 Metadata Leak Vectors NOT Covered

Despite the strength of the analysis, several metadata leak vectors remain unaddressed:

**METADATA-LEAK-01: PKI-mode URL document names (from Brief #4)**

The PKI-friendly links brief proposes URLs like:
```
send.sgraph.ai/d/b12/quarterly-report
send.sgraph.ai/d/a7f/contract-v3
send.sgraph.ai/d/b12/merger-terms
```

The `{document_name}` segment is in the URL PATH, not the fragment. It IS sent to the server. It IS logged in access logs, CloudFront logs, and Lambda invocation logs. It directly contradicts the one-time secret brief's principle: a server-side data leak now exposes customer intent.

This is the exact vulnerability the one-time secret brief warns about ("q4-layoff-plan" in URLs), but relocated from `/s/` routes to `/d/` routes. The one-time secret brief correctly blocks it for symmetric mode but the PKI-friendly links brief reintroduces it for PKI mode.

**Severity:** HIGH
**Recommendation:** PKI-mode URLs must also use opaque IDs: `send.sgraph.ai/d/b12/xK9mP2qR`. The human-friendly document name belongs in the encrypted metadata file, exactly as the one-time secret brief specifies for its PKI mode. Apply the same principle universally: URL paths are server-visible, therefore URL paths must be opaque.

**METADATA-LEAK-02: Data room names are server-visible**

The symmetric data room brief proposes admin-created data room names: "Project Alpha", "Due Diligence Q1". These names are server-side metadata. A database dump reveals which projects have data rooms, even if file contents remain encrypted.

**Severity:** MEDIUM
**Recommendation:** Data rooms should use server-generated opaque IDs. The human-friendly name should be stored as encrypted metadata, decryptable only by users with the room key. The admin UI resolves the friendly name client-side after decryption.

**METADATA-LEAK-03: Version chain metadata reveals content importance**

The versioned editing brief creates server-visible metadata: version count, editor names, edit timestamps, parent chains. A document with 15 versions edited by 6 people at 2am is clearly more important than a document with 1 version. This is metadata leakage about content significance.

**Severity:** MEDIUM
**Recommendation:** Version metadata (editor name, version label) should be encrypted alongside the content. The server should see only: encrypted blob, parent hash (opaque), timestamp. Editor names should be encrypted or pseudonymised (use opaque user IDs, not names).

**METADATA-LEAK-04: Timing correlation between secret creation and data room user creation**

When an admin creates a data room user and immediately creates a one-time secret link to share the key, the temporal correlation between the two API calls reveals that the secret contains a data room access key. An attacker with server access can correlate: "Secret xK9mP2qR was created 200ms after User B was added to Data Room Y."

**Severity:** LOW
**Recommendation:** Document this as a known metadata correlation risk. Mitigation: the admin UI could batch or delay secret creation to decorrelate timing. This is a low-priority finding but worth tracking.

**METADATA-LEAK-05: Access patterns reveal content type**

Different URL routes (`/s/` for secrets, `/d/` for documents, `/room/` for data rooms) reveal to the server what TYPE of content a user is accessing. The URL architecture brief codifies this: the server knows whether a request is for a secret, a document, or a data room.

**Severity:** LOW
**Recommendation:** Accept for now. Route-based content type inference is inherent to the URL architecture and is low-sensitivity metadata. A future enhancement could unify routes (all content served via a single `/c/{id}` endpoint) at the cost of UX clarity. Not recommended for MVP.

## 4. Zero-Knowledge Preservation Analysis

The core zero-knowledge property: **a full server-side breach exposes zero customer data.** The server stores encrypted blobs, opaque IDs, timestamps, and hashed IPs. The decryption key never touches the server.

### 4.1 URL Structures: Do Any Expose Customer Intent?

| Route | What the server sees | Zero-knowledge impact |
|-------|---------------------|----------------------|
| `/s/{secret_id}#{key}` | `secret_id` only (fragment stripped) | **Preserved** -- opaque ID, key in fragment |
| `/d/{key_prefix}/{doc_name}` | `key_prefix` AND `doc_name` | **VIOLATED** -- document name is customer-chosen content visible to server |
| `/room/{room_name}` | `room_name` | **VIOLATED** -- room name is customer-chosen content visible to server |
| `/room/{name}/{folder}/{file}` | Full path including folder and file names | **VIOLATED** -- folder structure and file names visible to server |
| `/send/{transfer_id}` | `transfer_id` | **Preserved** -- opaque ID |
| `/tools/*` | Tool name | **N/A** -- no customer data |
| `/vault/` | Route only | **Preserved** -- vault contents client-side |
| `/admin/{room_name}` | `room_name` | **VIOLATED** -- admin routes expose room names |

**Verdict:** Four route patterns expose customer-chosen content to the server. The one-time secret route correctly preserves zero-knowledge. The PKI document route, data room route, and admin route do not. These three must be redesigned to use opaque IDs with encrypted metadata for friendly names.

### 4.2 Server-Side Logging: New Data Touching the Server

This brief batch introduces the following new server-side data:

| New server-side data | Source brief | Sensitivity |
|---------------------|-------------|-------------|
| Secret encrypted blobs | One-time secret | Low -- encrypted, time-bounded |
| Secret metadata (expiry, view count, creation time) | One-time secret | Low -- operational metadata |
| Data room user records (names, roles, key mappings) | Symmetric data room | **MEDIUM** -- user names are customer data |
| Data room structure (room names) | Symmetric data room | **MEDIUM** -- room names reveal project names |
| Version chains (parent links, editor names, timestamps) | Versioned editing | **MEDIUM** -- edit patterns reveal content importance |
| Abuse detection signals (IP, rate, timing patterns) | One-time secret | Low -- opaque operational signals |
| Client-side encrypted log blobs | Encrypted logging | Low -- encrypted, readable only by security team |

**Findings:**
- Data room user names ("Bob", "Alice") are stored server-side as plaintext labels. While the brief acknowledges these are "just labels" with no cryptographic identity, they are still customer-chosen data that could reveal real identities. In a compliance or legal context, these names become personally identifiable information (PII).
- Editor names in version metadata are server-visible, creating a record of who edited what and when.

**Recommendation:** User identifiers in symmetric data rooms should be encrypted or replaced with opaque IDs. The admin resolves friendly names client-side. Editor names in version chains should be encrypted within the version metadata blob.

### 4.3 Version Chains: Do Version Metadata Reveal Content Patterns?

Yes. The version linking structure proposed in the versioned editing brief stores:

```json
{
  "file_id": "meeting-notes-v2",
  "name": "meeting-notes.md",
  "version": 2,
  "parent": "meeting-notes-v1",
  "created_by": "Alice",
  "created_at": "2026-02-25T10:30:00Z",
  "hash": "sha256:abc123..."
}
```

Every field except the encrypted content is server-visible:
- `name` -- reveals the file name (customer content)
- `version` -- reveals edit frequency
- `created_by` -- reveals editor identity
- `created_at` -- reveals edit timing
- `parent` -- reveals the edit chain structure
- `hash` -- enables correlation (same hash = same content across rooms)

**This metadata structure violates zero-knowledge.** A server-side breach reveals: which files exist, what they are named, who edits them, how often, and when. This is a significant information disclosure.

**Recommendation:** The version metadata structure should be split:
- **Server-visible (opaque):** `file_id` (random), `parent_id` (random), `created_at`, `encrypted_metadata_blob`
- **Encrypted (client-side):** `name`, `version`, `created_by`, `hash`, all semantic content

The server maintains the chain structure (parent links between opaque IDs) but cannot read what the chain represents.

### 4.4 Password Vault: Risk of Plaintext Exposure?

The password vault stores credentials encrypted with the user's private key. The server stores only encrypted blobs.

**The zero-knowledge property is preserved for password storage.** The server never sees plaintext passwords, usernames, URLs, or labels. The vault entry JSON (containing the password in plaintext) exists only in the browser after decryption.

**Client-side risks:**
- During generation, the plaintext password exists in JavaScript memory. It is accessible to any code running on the same origin (XSS risk).
- The `[Copy]` button places the password in the clipboard. Any application with clipboard access can read it. Clipboard data may persist after the tab is closed.
- The "Export All as JSON" feature from the browser storage viewer would include encrypted vault blobs. If the export file is not itself encrypted, and if the user's private key is also exported, the entire vault is exposed.

**Recommendation:**
1. Implement a short clipboard timeout: clear the clipboard 30 seconds after copying a password.
2. The storage viewer's "Export All as JSON" must NOT export private key material by default. Private keys should require a separate, explicit export with a distinct confirmation flow.
3. Consider `crypto.subtle` for in-memory key handling to reduce exposure window.

| `/view/secret/{id}` | GET | None (link is auth) | Secret ID | **Must be** | Parallel to `/s/{id}` retrieve | PKI links |
| `/view/doc/{id}` | GET | Key required | Doc ID | No | Content rendering (XSS vector) | PKI links |
| `/view/room/{id}` | GET | Key required | Room ID | No | Data room rendering (XSS vector) | PKI links |
| Client-side encrypted log submission | POST | Signed by user key | Encrypted log blob | **Must be** | Log flooding, storage exhaustion | One-time secret |
| Version creation | POST | Room key + editor role | File content + metadata | **Must be** | Storage exhaustion, version flooding | Versioned editing |

**Critical gaps identified:**
1. **No authentication specified for secret creation.** Anyone can create secrets. This enables storage exhaustion attacks. Rate limiting alone is insufficient -- require a lightweight proof-of-work or CAPTCHA for anonymous creation.
2. **No rate limiting specified for any new endpoint.** Every endpoint above that accepts user input must be rate-limited per IP (and per key where applicable).
3. **Data room key brute-force.** Short access keys (~10 digits) have limited entropy. If the key space is `[a-z0-9]` with 10 characters, that is ~52 bits of entropy -- adequate but only if rate-limited. Without rate limiting, offline brute-force against the encrypted master key blob is feasible.

### 5.2 New Client-Side Storage

| Storage Location | Content | Risk | Source Brief |
|-----------------|---------|------|-------------|
| IndexedDB: `sgsend-keys` / `private_keys` | Ed25519/PGP private keys | **CRITICAL** -- key compromise = identity theft | Multiple briefs |
| IndexedDB: `sgsend-keys` / `contacts` | Public keys of contacts | MEDIUM -- reveals social graph | PKI links |
| IndexedDB: `sgsend-vault` / `files` | Encrypted vault entries (passwords, documents) | HIGH -- encrypted but high-value target | Password generator |
| localStorage: `sgsend_locale` | User locale preference | LOW | URL architecture |
| localStorage: `sgsend_ifd_version` | UI version | LOW | URL architecture |

| Clipboard | Plaintext passwords (transient) | HIGH -- readable by any application | Password generator |
| Browser memory | Plaintext passwords, decrypted vault entries | HIGH -- accessible to XSS | Password generator, storage viewer |

**The Chrome password sync proposal is the highest-risk new storage location.** Storing private keys in Chrome's credential manager means:
- Google's infrastructure stores the private key (encrypted with the user's Google account credentials)
- A Google account compromise exposes the private key
- Google's Terms of Service may grant scanning rights over stored credentials
- Chrome extension APIs may be able to access stored credentials
- The private key's security is now bounded by Google account security, not by SGraph Send's architecture

This is a fundamental trust boundary expansion: the user's identity in SGraph Send becomes dependent on Google's security posture.

## 6. Client-Side Encrypted Logging Assessment

### 6.1 Design Strengths

The client-side encrypted logging architecture proposed in the one-time secret brief is well-reasoned. The core insight -- that client-side logs and server-side opaque logs have high interconnectivity, making fabrication detectable at graph level -- is correct and valuable.

**Specific strengths:**

1. **Signed logs provide non-repudiation.** A log entry signed by a user's private key proves the user (or their key holder) performed the action. This is stronger than server-side logging alone, where the server must be trusted to log accurately.

2. **Encrypted logs preserve privacy.** Only the security team can read the logs. An attacker with server access sees encrypted blobs. This maintains the zero-knowledge property for log contents while enabling incident response.

3. **The CSP violation report analogy is apt.** Even partial coverage provides signal. If 90% of clients send logs and one doesn't, the absence is itself a signal. The security team doesn't need 100% coverage to detect anomalies.

4. **Cross-stream consistency checking.** Server-side logs (opaque) plus client-side logs (detailed) together are far more useful than either alone. An attacker who suppresses client logs still generates server-side signals (the API calls still happen). An attacker who manipulates server logs can't coordinate with the signed client logs they can't control.

### 6.2 Design Weaknesses and Risks

**LOGGING-RISK-01: Key distribution for the security team's public key**

The brief states the security team's public key must be "embedded in the client at deploy time." This creates a single point of failure: if the key is compromised, all historical logs become readable by the attacker. If the key is rotated, all new logs must use the new key, and the old key must be securely destroyed.

**Severity:** MEDIUM
**Recommendation:** Use a key hierarchy. Embed a long-term root public key. Derive per-period (e.g., monthly) logging keys. The root key signs the period keys. This limits the blast radius of a key compromise to one period.

**LOGGING-RISK-02: Client-side logs can be replayed**

The brief mentions "timestamps and sequence numbers don't match server-side patterns" as a defence against replay. However, the client generates the timestamp. A sophisticated attacker who captures legitimate logs from one session can replay them in another session with adjusted timestamps.

**Severity:** LOW (requires captured logs AND key compromise)
**Recommendation:** Include a server-issued nonce in each client session. The client includes this nonce in every log entry. The nonce binds the log to a specific server-side session, preventing cross-session replay.

**LOGGING-RISK-03: Log volume as a denial-of-service vector**

If every client action generates an encrypted log blob submitted to the server, a malicious client can flood the logging endpoint with large encrypted blobs. The server cannot distinguish legitimate logs from garbage (because it cannot read them).

**Severity:** MEDIUM
**Recommendation:** Rate-limit log submissions per IP and per session. Set a maximum blob size. The server can verify the log signature without decrypting the content (if the signature is separate from the encryption), which would enable pre-validation filtering.

**LOGGING-RISK-04: Browser modifications can suppress ALL client logs**

A browser extension, tampered JavaScript, or a man-in-the-middle proxy can strip the logging code entirely. The brief acknowledges this ("absence of logs from an active account is itself a signal") but the detection depends on the server correlating "active account" with "no logs received." For anonymous users (symmetric mode, no account), there is no "account" to correlate. Log suppression by anonymous users is undetectable.

**Severity:** MEDIUM
**Recommendation:** For anonymous users, client-side logging should be considered advisory only, not a security control. The server-side opaque signals (creation rate, access timing, IP patterns) remain the primary detection mechanism for anonymous usage. Client-side logging is most valuable for authenticated PKI users where absence can be detected.

### 6.3 Overall Verdict

The client-side encrypted logging architecture is a **net positive** for security posture. It is not a replacement for server-side monitoring but a valuable complement. The design is more sophisticated than most client-side logging proposals (signing, encryption, cross-stream consistency). The weaknesses identified above are manageable through the recommended mitigations.

**Recommended implementation priority:** Implement server-side opaque abuse detection first (rate limiting, pattern analysis). Add client-side encrypted logging as a Phase 2 enhancement once the PKI infrastructure is in place to support signing and encryption.

## 7. Per-File Security Review Framework Assessment

### 7.1 Framework Strengths

The per-file security review framework (Brief #1) is a well-structured approach:

1. **Bottom-up methodology is correct for this codebase.** Top-down reviews miss details. Bottom-up reviews catch every file. The fractal zoom (file -> method -> class -> module -> path -> endpoint) provides natural escalation from detail to architecture.

2. **Parallel security tree is a strong pattern.** A `security/` directory mirroring the `src/` directory makes coverage gaps immediately visible: any file without a `.security.json` is unreviewed. Diffs between the code tree and security tree catch new files.

3. **The security file format is comprehensive.** The proposed JSON schema covers: attack surface (endpoints, auth, input validation), vulnerabilities (ID, severity, attack vector, impact, status, mitigation), secrets exposure, dependencies, and dead code. This is sufficient for initial assessment.

4. **Trail-based threat modelling (via Sherpa) connects files to user journeys.** This is valuable because individual file reviews miss systemic risks that emerge only when files interact along a user flow.

5. **Risk classification per deployment mode is pragmatic.** The same vulnerability has different severity in public cloud vs. IP-restricted vs. private deployment. The three-column risk table enables proportionate response.

### 7.2 Framework Weaknesses and Improvements

**FRAMEWORK-01: Security JSON files may become stale**

If `.security.json` files are not updated when code changes, they become misleading -- worse than having no security review because they imply a review was done. There is no enforcement mechanism proposed.

**Severity:** MEDIUM
**Recommendation:** Add a CI check: when a code file is modified, flag its corresponding `.security.json` as requiring re-review. Use `last_reviewed` date and file modification timestamps to detect staleness. A stale security review should be treated as a missing review.

**FRAMEWORK-02: The framework does not address client-side (JavaScript) code**

The examples and tree structure focus on Python server-side files (`src/api/transfers.py`, `src/storage/s3.py`). The majority of the new attack surface in this brief batch is in client-side JavaScript: encryption logic, vault operations, key management, clipboard handling. The framework must explicitly include the IFD/UI codebase.

**Severity:** HIGH
**Recommendation:** Extend the parallel security tree to cover `sgraph_ai_app_send__ui__user/` and `sgraph_ai_app_send__ui__admin/`. Client-side files need the same `.security.json` treatment, with additional fields for: XSS risk assessment, Content Security Policy compliance, and sensitive data in memory.

**FRAMEWORK-03: No severity ranking for the initial file review order**

The brief says "start with individual files" but does not prioritise which files to review first. With potentially hundreds of files, reviewing them in filesystem order wastes effort on low-risk files while critical-path code remains unreviewed.

**Severity:** MEDIUM
**Recommendation:** Prioritise files for initial review using the following heuristic:
1. Files that handle authentication/authorisation
2. Files that handle encryption/decryption
3. Files that accept external input (API endpoints)
4. Files that manage key material
5. Files that render untrusted content (preview components)
6. Files that interact with storage backends
7. All other files

**FRAMEWORK-04: The security JSON could aid attackers if the repo is public**

The proposed `.security.json` includes vulnerability IDs, descriptions, attack vectors, and status ("open"). If the security tree is checked into the same public repository as the code, it becomes a roadmap for attackers: "VULN-001 is open, here's the attack vector, here's the file."

**Severity:** HIGH
**Recommendation:** The security tree must be in a SEPARATE private repository or a private branch. Never commit open vulnerability details to a public repo. Alternatively, the security JSON can use a "public/private" split: public fields (classification, last_reviewed, reviewer) in the main repo; vulnerability details in a private store.

### 7.3 Overall Verdict

The framework is **sound and should be adopted** with the four improvements above. The bottom-up, file-by-file approach is well-suited to a rapidly growing codebase. The parallel tree structure provides natural coverage tracking. The main risks (staleness, missing JS coverage, aiding attackers) are manageable.

## 8. Security Recommendations

Prioritised list of security actions the team should take before building the features in this brief batch.

### BEFORE ANY IMPLEMENTATION

| Priority | Action | Affects | Rationale |
|----------|--------|---------|-----------|
| **P0** | **Enforce opaque IDs in ALL URL paths** -- remove `{doc_name}` from `/d/{prefix}/{doc_name}` and `{room_name}` from `/room/{room_name}`. Use encrypted metadata files for friendly names universally. | PKI links, data rooms, URL architecture | The zero-knowledge property is the project's foundational security claim. Any server-visible customer content undermines it. This must be resolved in the URL design BEFORE implementation, not patched afterward. |

It accepts arbitrary content from anonymous users and stores it. |

### BEFORE PRODUCTION DEPLOYMENT

| Priority | Action | Affects | Rationale |
|----------|--------|---------|-----------|
| **P1** | **Do NOT store private keys in Chrome password manager without explicit, prominent user warning.** If implemented, display: "Your private key will be stored in your Google account. Google's security policies apply. This is a convenience feature, not a security recommendation." Offer IndexedDB-only as the default. | PKI key portability | Users must understand the trust boundary expansion. Chrome sync should be opt-in, not default. |
| **P1** | **Encrypt data room user metadata.** User names, roles, and room names should be encrypted in the same manner as file content. The server should store only opaque IDs and encrypted blobs. | Symmetric data room | Data room user names are PII. Room names reveal business context. Both violate zero-knowledge if stored in plaintext. |
| **P1** | **Encrypt version chain metadata.** Editor names, file names, and version labels should be stored as encrypted metadata. The server maintains parent links between opaque version IDs but cannot read what the chain represents. | Versioned editing | Edit patterns reveal content importance and collaboration structure. |
| **P1** | **Implement server-side size limits and content-type validation** for all file upload endpoints (secret creation, data room upload, version creation, log submission). | All upload endpoints | Without server-side validation, every upload endpoint is a storage exhaustion target. Markdown rendering must use a sanitiser (DOMPurify or equivalent). | Versioned editing, data room preview | File preview of untrusted content is a direct XSS vector. ### BEFORE SECURITY REVIEW FRAMEWORK ROLLOUT

| Priority | Action | Affects | Rationale |
|----------|--------|---------|-----------|
| **P2** | **Never commit vulnerability details to a public repository.** The `.security.json` files with open vulnerabilities, attack vectors, and mitigation status must be in a private repository or encrypted store. | Per-file security review | Open vulnerability documentation in a public repo is an attacker's roadmap. |
| **P2** | **Extend the security tree to cover client-side code.** The parallel `security/` directory must include JavaScript/Web Component files from the UI codebases, not just Python backend files. | Per-file security review | The majority of new attack surface is client-side. ### ONGOING / PHASE 2

| Priority | Action | Affects | Rationale |
|----------|--------|---------|-----------|
| **P3** | **Implement clipboard timeout.** Auto-clear the clipboard 30 seconds after a password copy operation. | Password generator | Clipboard persistence is a well-known credential leak vector. |
| **P3** | **Separate "Export All" from "Export Private Keys"** in the storage viewer. Private key export must be a distinct action with its own confirmation flow, not bundled into a general export. | Browser storage viewer | A single "Export All" that includes private keys makes key exfiltration too easy. |

## Appendix A: Threat Model Summary for New Features

### A.1 One-Time Secret: Threat Scenarios

| Threat | Likelihood | Impact | Mitigation |
|--------|-----------|--------|------------|
| Anonymous abuse (spam hosting) | HIGH | MEDIUM | Rate limiting + proof-of-work for anonymous creation |
| Storage exhaustion (bulk secret creation) | MEDIUM | MEDIUM | Size limits + rate limits + auto-expiry enforcement |
| Race condition on one-time retrieval | LOW | HIGH (secret viewed twice) | Atomic check-and-delete using database transaction or compare-and-swap |
| Link interception (MitM) | LOW (HTTPS) | HIGH | TLS enforced; fragment never sent to server |
| Secret enumeration (brute-force IDs) | LOW (12-char IDs) | HIGH | Rate limit retrieval attempts; return identical response for "not found" and "already viewed" |
| Timing attack on existence check | MEDIUM | LOW | Ensure constant-time response for "not found" vs "already viewed" |

### A.2 Symmetric Data Room: Threat Scenarios

| Threat | Likelihood | Impact | Mitigation |
|--------|-----------|--------|------------|
| Admin key compromise | MEDIUM | CRITICAL (full room access) | Admin key rotation; MFA for admin operations (future) |
| Cached master key after revocation | MEDIUM | HIGH | Document as known limitation; implement full revocation (re-encrypt) in future |
| Short key brute-force | LOW (with rate limiting) | HIGH | Minimum 60-bit entropy; rate limit key attempts |
| "Login as User" abuse | LOW (admin-only) | MEDIUM | Audit log for "login as" operations; consider requiring re-authentication |

### A.3 Password Vault: Threat Scenarios

| Threat | Likelihood | Impact | Mitigation |
|--------|-----------|--------|------------|
| XSS stealing vault contents | MEDIUM | CRITICAL | Strict CSP; sanitise all rendered content; vault decryption requires user gesture |
| Clipboard exfiltration | MEDIUM | HIGH | Auto-clear clipboard after 30 seconds |
| Browser extension reading IndexedDB | LOW | CRITICAL | Document risk; recommend privacy-focused browser for high-security usage |
| Import CSV containing plaintext | MEDIUM | HIGH | Encrypt immediately on import; never persist plaintext to disk |
