# v0.6.20 AppSec Review: fastapi-mcp Dependency Security

**Date:** 2026-02-25 | **Role:** AppSec | **Severity:** HIGH | **Status:** URGENT REVIEW REQUIRED

---

## Executive Summary

Adding `fastapi-mcp` introduced **17 new supply chain attack vectors** for ~27 lines of application code. Several dependencies carry elevated risk: native code compilation (`cryptography`, `cffi`, `rpds-py`), JWT handling (`pyjwt`), and a policy violation (`pydantic-settings`). The attack surface expansion is disproportionate to the feature value at this stage.

---

## RED FLAG: pydantic-settings Violates Project Policy

**Project rule:** "Never use Pydantic" (use `Type_Safe` from `osbot-utils` instead).

`pydantic-settings` (2.13.1) is a **hard dependency** of both `fastapi-mcp` AND `mcp`. It cannot be removed without forking both packages. While our application code does not import `pydantic-settings` directly, it is loaded at runtime, and:

- It pulls in `python-dotenv`, which reads `.env` files automatically -- **risk of unintended environment variable loading**
- The `mcp` SDK uses it internally for its configuration
- It normalises Pydantic's position in the dependency tree, making accidental future use more likely

**Verdict:** Technical policy violation. Accepted as transitive-only for now, but must be monitored. Our code must NEVER import from `pydantic-settings` directly.

## Supply Chain Risk Assessment

| Metric | Value | Concern Level |
|--------|-------|:---:|
| New packages | 17 | HIGH |
| Packages with native code | 3 (cryptography, cffi, rpds-py) | HIGH |
| Pre-1.0 packages | 2 (fastapi-mcp 0.4.0, python-multipart 0.0.22) | MEDIUM |
| Unique maintainers added | ~10 | MEDIUM |
| New packages needing platform-specific builds | 3 | MEDIUM |

**Key concern:** Each native-code package requires pre-built binary wheels for our target platform (Python 3.12 / arm64 / Amazon Linux 2023). Missing wheels mean compilation at build time, which is fragile and slow.

## Lambda Deployment Impact

| Concern | Detail |
|---------|--------|
| **Cold start** | +17 packages = more imports at startup. `cryptography` and `cffi` are notably slow to import (~200-500ms). |
| **Package size** | Estimated +11-22 MB (native binaries dominate). Lambda has 250 MB limit. |
| **arm64 compatibility** | `cryptography`, `cffi`, and `rpds-py` all need arm64 wheels. Must verify wheels exist for Python 3.12 + Amazon Linux 2023. |
| **Build complexity** | Native deps may need build tools (gcc, cargo) in CI. |

## Recommendations

### Immediate (before next deploy)

1. **Verify pyjwt is not pulling in cryptography in production.** Check if `pyjwt[crypto]` extra is being activated or if it is base `pyjwt` only.
2. **Pin all 17 new dependencies** to exact versions in `pyproject.toml` or `poetry.lock`. Unpinned (`*`) is unacceptable for security-sensitive deps.
3. **Add `.env` to Lambda deployment exclusion list** to prevent `python-dotenv` from loading unexpected files.
4. **Confirm `tomli` necessity** -- Python 3.12 includes `tomllib` in stdlib. This dep may be unnecessary.

### Short-Term

5. **Evaluate if `mcp` SDK can be used without `pyjwt`** -- if we are not using JWT auth, this is dead attack surface.
6. **Measure Lambda cold start impact** -- before/after with the 17 new packages.
7. **Add dependency audit to CI** -- flag any new transitive deps automatically.

### Strategic

8. **Consider building a minimal MCP endpoint without fastapi-mcp.** The MCP HTTP transport is JSON-RPC over POST. A hand-rolled implementation would be ~100-200 lines and zero new dependencies. Trade-off: maintenance burden vs 17 supply chain risks.
9. **Never allow application code to import from pydantic-settings, cffi, pycparser, or cryptography.** Add linting rule.

---

## Verdict

The security cost of 17 new dependencies for 27 lines of code is **disproportionate**. The feature works but the supply chain risk needs active management. Recommend proceeding with caution and implementing the pinning/audit recommendations before production deployment.
