## 1. Security Posture Summary

**Overall assessment: STRONG architecture with critical correctness improvements.**

The 28 briefs represent a significant expansion of the project's security perimeter -- from "encrypted file transfer" to "PKI-based trust infrastructure platform." This expansion introduces substantial new attack surface, but the briefs demonstrate a security-first design philosophy that is rare in early-stage projects.

### What the briefs get right

1. **The Agent Isolation correction (Brief #22) is the single most important document in this batch.** The original claim that "encryption in shared memory is isolation" was dangerously wrong. The correction is thorough, accurate, and technically precise. The distinction between "encryption selects WHO can read" and "process isolation ensures they're the ONLY ones who can" should be quoted in every subsequent architecture document.

2. **Receiver-sets-the-rules model (Brief #7) is architecturally sound.** The receiver defining scanning requirements, not the sender, is the correct trust inversion. The sender is untrusted; the receiver must define what "safe" means. This is a foundational security principle well applied.

3. **Hash-binding of scan results to files (Brief #7) solves a real integrity problem.** Binding scan results to SHA-256 file hashes prevents the swap-clean-for-dirty attack. The cryptographic chain (service signs result, result encrypted for receiver, hash verified on decryption) is well-designed.

4. **Network-level detection (Brief #8) addresses the correct threat layer.** Individual account detection is insufficient against coordinated attackers. The cluster analysis approach is the right abstraction level. The "paradox of randomness" observation is especially valuable.

5. **Defence-in-depth layering (Brief #22) correctly identifies that no single boundary is sufficient.** Container isolation + PKI encryption + signed outputs + time-bounded secrets + anomaly detection + trust scores -- each layer compensates for the others' weaknesses.

### What concerns AppSec

1. **The scope expansion creates a much larger attack surface than the current team can secure.** 28 briefs introduce trust infrastructure, scanning chains, agent orchestration, SaaS integration, cross-domain checkpoints, and network analysis. Each of these is a non-trivial security domain. Prioritisation is essential.

2. **Several briefs make claims that are aspirational rather than proven.** Time-bounded secret decryption, LLM-powered malware analysis, and silent network detection are architecturally described but have significant implementation gaps.

3. **The zero-knowledge model is under tension.** The trust web (Brief #2) explicitly requires the server to know communication patterns (who sends to whom). The environment analysis (Brief #1) collects receiver fingerprinting data. The scanning pipeline (Brief #4) may send file hashes to external services. Each of these is a justified trade-off, but together they erode the zero-knowledge guarantee incrementally. This needs a formal privacy budget.

## 2. Critical Security Findings

### CRITICAL-01: Warm Lambda Reuse Breaks Agent Isolation

**Brief:** #22 (Agent Isolation Reality)
**Severity:** CRITICAL

The brief correctly identifies this but it deserves escalation: AWS Lambda warm starts reuse the execution environment between invocations. Memory, `/tmp` filesystem, and even running processes persist. If Agent A runs in a Lambda, and Agent B subsequently runs in the SAME warm Lambda, Agent B can access Agent A's residual key material.

**The mitigation described (explicitly zero key material before returning) is necessary but insufficient.** Memory zeroing in managed runtimes (Python, Node.js) is unreliable because:
- The garbage collector may have copied the key to a new memory location before the original is zeroed
- The JIT compiler may have optimised away the zeroing operation
- The key may have been written to swap before zeroing

**Recommendation:** For high-security workflows, enforce one Lambda function per agent identity (not one shared function). For the prototype, use hardened containers with separate UIDs as the minimum viable isolation. Document this as a known limitation of serverless isolation.

### CRITICAL-02: Client-Side Scanning Is Adversarial

**Brief:** #4 (File Security Scanning Pipeline)
**Severity:** CRITICAL

The brief proposes scanning files in the sender's browser before encryption. This is fundamentally an adversarial environment. A malicious sender controls the browser and can:
- Bypass client-side scanning entirely (modify JavaScript)
- Report false scan results to the receiver
- Scan a clean decoy, then encrypt the malicious file

The brief acknowledges this ("we don't trust the client") and Brief #7 addresses it with the PKI-verified scanning chain. However, the scanning pipeline brief (Brief #4) presents client-side scanning as a standalone capability without sufficiently emphasising that it is only trustworthy for honest senders trying to avoid accidentally sending malware. It is NOT a defence against malicious senders.

**Recommendation:** The scanning pipeline documentation must clearly state: client-side scanning protects well-intentioned senders from accidental malware. Protection against malicious senders requires the external scanning chain (Brief #7) or the pod intermediary model. All three tiers (client-side advisory, external PKI-verified, pod intermediary) should be presented as a graduated security model, not alternatives.

### CRITICAL-03: File Hash as Privacy-Sensitive Identifier

**Brief:** #7 (External File Security Services)
**Severity:** HIGH

The brief notes that "the file hash is sensitive" but underestimates the risk. Sending a SHA-256 hash to a scanning service allows:
- **Correlation attacks:** The scanning service knows that a specific file exists and is being transferred via SGraph Send
- **Known-file detection:** If the scanning service has the file hash in their database (from a previous scan), they know the exact file content
- **Traffic analysis:** Patterns of hash submissions reveal usage patterns

The hash is functionally equivalent to a unique identifier for the file. Every external scanning service that receives it can link the file to the SGraph Send platform.

**Recommendation:**
1. Document the privacy implications of hash sharing in the threat model
2. Consider a privacy-preserving alternative: send a keyed HMAC of the file rather than the raw SHA-256 hash (the HMAC key is shared only between sender and scanning service, not with the receiver's verification -- but the receiver can verify by recomputing the HMAC with the shared key)
3. For maximum privacy: use the pod model where the scanning service sees the file but doesn't know which platform it came from
4. Offer hash submission as opt-in, not default, with clear user notification

### CRITICAL-04: Scan Result Opacity Creates Trust Asymmetry

**Brief:** #7 (External File Security Services)
**Severity:** HIGH

The brief states scan results are encrypted for the receiver only -- the sender cannot read them. The rationale is that revealing scan findings helps attackers evade future detection. This creates a problematic trust asymmetry:

- A sender who is NOT malicious cannot verify that the scanning service is functioning correctly
- A sender cannot dispute a false positive that blocks their legitimate file
- The sender has no visibility into why their transfer might fail

While the security rationale is sound for malicious senders, it creates a poor experience for legitimate users and makes debugging impossible. A scanning service with a bug or misconfiguration would silently block transfers with no recourse.

**Recommendation:** Implement a two-tier result model:
1. **Summary result to sender:** "File passed" / "File flagged -- macros detected" (enough for the sender to understand and fix issues)
2. **Detailed result to receiver:** Full scan details, service signature, hash binding (the full verification chain)

This preserves the receiver's intelligence advantage while giving honest senders actionable feedback.

## 3. Trust Model Analysis

### Brief #2: Trust Web -- Reputation Through Usage

**Assessment: Architecturally sound, with implementation risks.**

**Strengths:**
- Trust builds slowly, destroys fast -- correct asymmetry for security
- Recipient-weighted trust (sending to trusted recipients is worth more) -- natural Sybil resistance
- Integration with scanning results creates a multi-signal trust model

**Concerns:**

**TRUST-01: Cold Start Problem.** New users have zero trust and can do very little. This is correct for security but creates an adoption barrier. The brief does not address how enterprise users or high-volume senders bootstrap trust without a lengthy warm-up period. A PKI-verified identity (e.g., GitHub key) could provide initial trust, but this is not described.

**TRUST-02: Trust Score Gaming via Automated Receivers.** The trust model rewards sending to high-trust recipients. An attacker could create a service that auto-receives files and never complains, offering "trust boosting" as a service. The counter is that receiving from low-trust senders should not boost the receiver's score, but the interaction between sender and receiver trust needs formal modelling.

**TRUST-03: Privacy Trade-Off Requires Formal Documentation.** The brief acknowledges that the server must know communication patterns (which public keys communicate). This is a departure from pure zero-knowledge. AppSec supports documenting this as an explicit trade-off, but recommends:
- A formal privacy budget documenting exactly what metadata the server collects
- User consent for trust system participation
- An opt-out mechanism (accept reduced functionality in exchange for full zero-knowledge)

### Brief #1: Receiver Environment Analysis

**Assessment: Useful signal, but advisory only.**

**TRUST-04: Environment Reports Are Fundamentally Spoofable.** The brief acknowledges this but it bears repeating: the receiver controls their browser, so they control the environment report. An attacker downloading a file to exfiltrate it will fake the environment report. The anti-gaming measures (server-side IP cross-reference, feature detection) raise the bar but do not eliminate the risk. Browser fingerprinting is an arms race that defenders consistently lose.

**Recommendation:** Label environment analysis as "advisory" in all UI and documentation. Never use it as an enforcement mechanism. It adds value for honest senders verifying honest receivers -- a legitimate use case (e.g., "is my colleague downloading this from their office network?") -- but it is not a security control against determined attackers.

**TRUST-05: Fingerprinting Consent and GDPR.** Collecting IP address, WebGL renderer, screen resolution, and browser details constitutes device fingerprinting under ePrivacy Directive and may constitute processing of personal data under GDPR. The transparency notice in the brief is a good start, but GRC must review whether:
- The fingerprinting data constitutes "personal data" under GDPR Article 4(1)
- Legitimate interest or explicit consent is the appropriate lawful basis
- The data retention period is defined and minimised

## 4. Scanning Pipeline Review

### Brief #4: File Security Scanning Pipeline

**Assessment: Comprehensive approach, but feasibility of client-side execution needs validation.**

**SCAN-01: ClamAV WASM Feasibility is Unproven at Scale.** ClamAV compiled to WebAssembly is technically possible but:
- ClamAV signature databases are ~300MB. Loading this into browser memory is impractical for casual users.
- Signature database updates would need a caching strategy (Service Worker).
- Scanning performance in WASM will be significantly slower than native.
- Memory pressure on mobile devices may be prohibitive.

**Research required:** Prototype ClamAV WASM with a realistic signature database. Measure load time, scan time, and memory usage across Chrome, Firefox, and Safari on desktop and mobile.

**SCAN-02: LLM-Powered Analysis Is Novel and Unvalidated.** Using LLMs to detect malicious macros, scripts, and obfuscation is an intriguing approach, but:
- False positive/negative rates are unknown
- LLM inference latency (2-10 seconds per analysis) may be unacceptable for user workflows
- Sending extracted code to a cloud LLM creates a privacy issue (the scanning pipeline brief acknowledges this)
- Local LLM alternatives (Ollama, Chrome AI) have much weaker detection capabilities
- Adversarial prompt injection of the scanning LLM itself (craft a macro that tricks the LLM into reporting "safe") is an unexplored attack vector

**Research required:** Evaluate LLM-based malware detection accuracy against established benchmarks (EMBER, BODMAS). Test adversarial evasion against the LLM scanner.

**SCAN-03: Archive Inspection Zip Bomb Risk.** The brief mentions "watch for zip bombs" but does not specify a defence. Recursive decompression in the browser could freeze the UI or crash the tab.

**Recommendation:** Implement strict limits: maximum recursion depth (3 levels), maximum decompressed size (100MB or 10x compressed size, whichever is smaller), maximum file count within archive (1000). Abort scanning and flag the file if any limit is exceeded.

### Brief #7: External File Security Services -- PKI-Verified Scanning Chain

**Assessment: Well-designed cryptographic protocol. Implementation complexity is the main risk.**

**SCAN-04: Scanning Service PKI Adoption is the Bottleneck.** No existing scanning service (VirusTotal, MetaDefender, Cuckoo Sandbox) supports PKI-signed results today. The brief acknowledges this ("can we wrap them?"). The wrapping approach works but:
- Adds a trust delegation: the wrapper service signs on behalf of the scanning engine
- The wrapper service must be trusted not to forge results
- This creates a new attack surface (compromise the wrapper, forge any scan result)

**Recommendation:** For MVP, the platform itself can act as the signing wrapper for scan results from external APIs. Document the trust delegation explicitly. Long-term, advocate for scanning services to adopt PKI signing natively (this could become a differentiator for SGraph Send partnerships).

**SCAN-05: Pod Intermediary Model Has the Strongest Security Properties.** The pod model (Section: "The Pod as an Independent Processing Environment") is the highest-assurance option because:
- The pod sees cleartext (accepted -- it is the trusted processing environment)
- The pod calls scanning services on its own behalf (sender cannot interfere)
- The pod re-encrypts for the receiver (provenance chain extends through the pod)
- The pod can apply CDR (strip dangerous content) before re-encryption

**Recommendation:** Prioritise the pod intermediary model for enterprise and high-security use cases. The direct sender-to-service model is acceptable for consumer use cases where the sender is generally trusted.

### Brief #11: File Security Services Landscape

**Assessment: Research scope is appropriate. Key questions are well-framed.**

**SCAN-06: Missing Research Question -- Supply Chain Attacks on Scanning Services.** The research brief should add: "What happens if a scanning service is compromised or produces incorrect results?" Specifically:
- Can a compromised VirusTotal return false negatives?
- What is the SLA for scanning service availability? If the service is down, can the sender still upload?
- Is there a quorum model? (e.g., "file must pass 2 out of 3 scanning services")

**Recommendation:** Add supply chain risk of scanning services to the research questions. Design a quorum/consensus model for multi-service scanning.

## 5. Isolation Assessment

### Brief #22: Agent Isolation Reality -- Process-Level Security

**Assessment: EXCELLENT. This is the most technically rigorous brief in the entire batch.**

The correction of the "encryption in shared memory is isolation" claim is critical and correctly argued. The isolation hierarchy (serverless > dedicated VM > hardened container > default container > separate process different UID > separate process same UID > same process) is accurate.

**ISO-01: The .sgworkflow Solution to State Management is Elegant.** The key insight is that state management across isolated environments is the hard problem that makes isolation impractical. The .sgworkflow file as the coordination medium solves this by making the file itself the message bus, state machine, and checkpoint. This directly addresses the reason most agent frameworks avoid proper isolation.

**ISO-02: Secret Injection Remains the Weakest Link.** Even with per-container isolation, the private key must get INTO the container somehow. The brief mentions "container runtime's secret mechanism (Kubernetes secrets, AWS Secrets Manager, etc.)" but:
- Kubernetes secrets are base64-encoded, not encrypted at rest (by default)
- AWS Secrets Manager is a network-accessible service; the container must have IAM permissions to call it
- The IAM role attached to the container is the effective security boundary

**Recommendation:** Define a specific secret injection protocol for each deployment target. For Lambda: use environment variables encrypted with KMS. For containers: use init containers that fetch secrets and write to tmpfs. For VMs: use instance metadata with IMDSv2. Document the threat model for secret injection separately.

**ISO-03: Orchestrator Compromise is the Catastrophic Failure Mode.** The orchestrator has all agents' public keys, creates workflows, and dispatches tasks. If the orchestrator is compromised:
- The attacker can create new workflows with malicious tasks
- The attacker can route sensitive data to a rogue agent
- The attacker can suppress anomaly detection alerts

The orchestrator must be the most hardened component, but the brief does not address orchestrator security specifically.

**Recommendation:** Define orchestrator hardening requirements: minimal attack surface, no direct internet access, separate security domain from agents, signed workflow manifests (so agents can verify the orchestrator authorised their tasks).

### Brief #28: PKI-Secured Agentic Workflows -- Prompt Injection Containment

**Assessment: Strong vision document. The companion correction (Brief #22) makes the claims in this brief defensible.**

**ISO-04: Time-Bounded Secrets Require a Reliable Clock.** The brief describes secrets that expire after a time window. This requires:
- A clock that the agent cannot tamper with (if the agent can set its own clock, it can extend the window)
- Clock synchronisation across containers (if Container A's clock is 30 seconds ahead, time windows are inconsistent)
- A decision: is the time check enforced by the decryption mechanism itself, or by a policy check after decryption?

**Recommendation:** Time bounds should be enforced by the secret management service (not the container's local clock). The container requests decryption from the service; the service checks the timestamp before returning the cleartext. This requires a network call, which adds latency but provides reliable enforcement.

**ISO-05: Agent Trust Scores Are a Novel Concept with Limited Prior Art.** Applying the trust web model to agents is interesting but differs from user trust in important ways:
- Agents produce outputs that are machine-readable (easier to validate than human behaviour)
- Agents can be restarted from scratch (unlike humans who carry persistent reputation)
- Agent compromise can be sudden and total (unlike gradual trust erosion for humans)

**Recommendation:** The agent trust model should be separate from the user trust model. Agent trust should decay faster when idle (since agents don't "remember" their reputation -- they're stateless). Agent trust should be tied to the agent's key pair, not its deployment identity.

## 8. Recommendations

Ordered by priority. Each recommendation references the findings that motivate it.

### Priority 1: IMMEDIATE (block on these before implementation)

**REC-01: Formalise the three-tier scanning model** (CRITICAL-02, SCAN-05)
Document and enforce the graduated scanning model:
- Tier 1: Client-side advisory scanning (protects honest senders, no security guarantee)
- Tier 2: External PKI-verified scanning chain (receiver-defined requirements)
- Tier 3: Pod intermediary with CDR (highest assurance)
Never present Tier 1 alone as a security feature.

**REC-02: Define agent isolation minimum requirements** (CRITICAL-01, ISO-02)
Before any agentic workflow implementation:
- Minimum: hardened container per agent with separate UIDs, seccomp, AppArmor, no-new-privileges, read-only rootfs, dropped capabilities
- Secret injection protocol defined for each deployment target
- Key material zeroing validated in the chosen runtime
- Lambda: one function per agent identity, not shared

**REC-03: Create a formal privacy budget** (CRITICAL-03, TRUST-03)
Document every piece of metadata the server sees, every external service call, and every piece of data that leaves the browser. Map each against the zero-knowledge guarantee. Define what "zero-knowledge" means for this project precisely (the server never sees file content; the server does see communication patterns).

### Priority 2: HIGH (implement during v0.7.x)

**REC-04: Begin scanning services survey** (R-06, R-18, R-19)
The scanning chain is the most implementation-ready security feature. Start with the services landscape survey to identify which APIs to integrate first.

**REC-05: Design scan result schema with two-tier disclosure** (CRITICAL-04, R-09)
Summary for sender (actionable feedback), full details for receiver (verification chain).

**REC-06: Prototype ClamAV WASM and measure real-world performance** (SCAN-01, R-01)
Validate or invalidate client-side scanning feasibility with real measurements. If ClamAV WASM is impractical, pivot to hash-based lookup + YARA rules (lighter weight).

**REC-07: Add scanning service supply chain risk to threat model** (SCAN-06)
Design a quorum model for multi-service scanning: file must pass N-of-M services. Define fallback behaviour when a service is unavailable.

**REC-08: Define orchestrator hardening requirements** (ISO-03)
The orchestrator is the single point of catastrophic failure in agentic workflows. It needs: minimal attack surface, separate security domain, signed workflow manifests, audit logging, no direct internet access.

### Priority 3: MEDIUM (implement during v0.8.x)

**REC-09: Begin graph-based cluster detection research** (R-11, R-12)
Network-level threat detection is valuable but complex. Start with literature review and algorithm selection before implementation.

**REC-10: Design the trust score formula with formal properties** (TRUST-01, TRUST-02)
The trust model needs formal specification:
- Mathematical properties (monotonicity of positive signals, sharp decay on negative)
- Anti-gaming properties (trust-boosting services ineffective)
- Cold-start solution (PKI-verified identity bootstrap)
- Formal Sybil resistance proof

**REC-11: Implement file hash privacy protection** (CRITICAL-03)
Consider HMAC-based hash submission to scanning services, or offer the pod model as the privacy-preserving alternative. At minimum, document the privacy cost of hash sharing and require user consent.

**REC-12: GRC review of environment fingerprinting under GDPR** (TRUST-05)
The receiver environment analysis collects data that may constitute personal data or require ePrivacy consent. Get a formal legal assessment before implementation.

### Priority 4: LOW (track and plan for v0.9.x+)

**REC-13: Design silent detection workflow** (R-15, R-17)
The intelligence-gathering approach to attacker networks is valuable but operationally complex. Requires legal review (is silent observation of suspected malicious accounts acceptable under applicable law?).

**REC-14: Investigate TEE/Secure Enclave feasibility** (Brief #22)
For high-security agentic workflows, Trusted Execution Environments could provide stronger isolation than containers. Track SGX/SEV/TrustZone developments and side-channel mitigations.

**REC-15: Evaluate SaaS JS library security model** (T-SA-01, T-SA-02)
The drop-in JS library (Brief #15) is a high-value target for supply chain attacks. Design an SRI-based distribution model with self-hosting option. Define a security review checklist for SaaS integration partners.

## Summary

The 28 briefs demonstrate a security architecture that is fundamentally well-designed. The Agent Isolation correction (Brief #22) is the most important single document -- it saves the project from a dangerous misconception. The PKI-verified scanning chain (Brief #7) is the most implementation-ready security feature. The trust web (Brief #2) and network detection (Brief #8) are the most strategically valuable but require significant research before implementation.

The primary risk is not architectural -- it is scope. The briefs describe an ambitious security infrastructure that would take a dedicated security team months to implement fully. AppSec recommends strict prioritisation: scanning chain first (Tier 1-3), agent isolation second, trust model third, network detection fourth.

**Immediate action items:**
1. Formalise the three-tier scanning model in the architecture documents
2. Define agent isolation minimum requirements before any .sgworkflow implementation
3. Create the formal privacy budget documenting all zero-knowledge trade-offs
4. Begin the scanning services landscape survey (R-06, R-18)
