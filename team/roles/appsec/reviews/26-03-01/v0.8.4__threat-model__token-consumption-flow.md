# v0.8.4 AppSec Review: Token Consumption Flow — Threat Model

**Date:** 2026-03-01
**Role:** AppSec
**Version:** v0.8.4
**Trigger:** Human review of Dev review `v0.7.19__review__v2.0-ifd-consolidation-and-code-audit--revised.md` — Section 3 "API Endpoint Inventory" identified `POST /transfers/validate-token/{token_name}` as publicly accessible and consuming token uses.

---

## 1. Executive Summary

The human identified a real architectural concern: **anonymous users can trigger token consumption via a public endpoint**. After tracing the full flow (client → User Lambda → Admin Lambda → cache storage), the finding is **nuanced but important**:

- **The consumption logic IS server-side** — the client cannot directly manipulate usage counts
- **BUT the trigger IS client-side** — any anonymous HTTP client can invoke the consumption
- **This creates a token exhaustion attack vector** — an attacker can drain a token's usage limit without performing any legitimate action

**Severity: MEDIUM** — Denial of service against individual tokens. Does not leak data, does not bypass encryption, does not grant unauthorized access to files.

---

## 2. Data Flow Trace (End-to-End)

### 2.1 The Two Token Endpoints on User Lambda (Public)

```
┌─────────────────────────────────────────────────────────────────┐
│ User Lambda — PUBLIC, NO AUTHENTICATION                        │
│                                                                 │
│ GET  /transfers/check-token/{token_name}                        │
│   → Calls Admin Lambda: GET /tokens/lookup/{token_name}         │
│   → Returns: {valid, status, remaining}                         │
│   → DOES NOT consume a use                                      │
│                                                                 │
│ POST /transfers/validate-token/{token_name}                     │
│   → Calls Admin Lambda: POST /tokens/use/{token_name}           │
│   → Sends: {ip_hash, action='page_opened'}                     │
│   → Returns: {success, usage_count, remaining}                  │
│   → CONSUMES one use                                            │
└─────────────────────────────────────────────────────────────────┘
```

### 2.3 Double Consumption on Upload

Each successful upload triggers **two** consumption events:

| Event | Trigger | Action | Code Location |
|-------|---------|--------|---------------|
| 1st | Download page loads with `?token=X` in URL | `page_opened` | `Routes__Transfers.py:208-221` |
| 2nd | Upload completes (`POST /transfers/complete/{id}`) | `upload_completed` | `Routes__Transfers.py:125-134` |

The 2nd consumption happens inside `complete__transfer_id()` which **does** require the access token (via `check_access_token()`). This is the correct, server-side consumption tied to a real action.

## 3. Findings

### FINDING-1: Token Exhaustion via Public Endpoint (MEDIUM)

**What:** `POST /transfers/validate-token/{token_name}` is publicly accessible with no authentication, rate limiting, or proof-of-work. Any HTTP client can call it repeatedly to exhaust a token.

**Attack:**
```bash
# Exhaust a 50-use token in seconds
for i in $(seq 1 50); do
  curl -X POST https://send.sgraph.ai/transfers/validate-token/community-token
done
```

**Impact:** Denial of service against specific tokens. Legitimate users find their token exhausted. Admin must reactivate or increase limit manually.

**Mitigating factors:**
- Attacker must know the token name (not guessable if using random names)
- Usage events record `ip_hash` — abuse is traceable after the fact
- Admin can reactivate tokens and increase limits via Admin UI
- Does not leak file data or bypass encryption

**Recommendation:**
- **Short-term:** Add per-IP rate limiting on `validate-token` (e.g., max 5 calls per minute per IP)
- **Medium-term:** Separate "validate" (check validity) from "consume" (decrement count). The consumption should only happen when a legitimate action occurs (download started, not just page opened)
- **Long-term:** Consider binding consumption to a server-side action proof — e.g., only consume when `GET /transfers/download/{id}` is called with a valid token, not when an anonymous page load triggers it

### FINDING-2: Token Name Enumeration via check-token (LOW)

**What:** `GET /transfers/check-token/{token_name}` reveals whether a token name exists (returns `{valid: true}` vs `{valid: false, reason: 'not_found'}`). No authentication required.

**Impact:** Attacker can probe for valid token names. Low severity because token names don't grant access to file data (files are E2E encrypted).

**Mitigating factors:**
- Token names alone don't expose encrypted content
- The decryption key is never stored server-side
- Knowing a token exists doesn't reveal what files are behind it

**Recommendation:** Return identical responses for non-existent and invalid tokens (timing-safe comparison, same response shape).

### FINDING-3: Graceful Degradation Bypasses Token Enforcement (LOW)

**What:** Both the `check-token` and `validate-token` client-side calls are wrapped in try-catch with graceful degradation — if the admin service is unreachable, uploads and downloads proceed without token validation.

**Code evidence:**
- `send-download.js` (v0.1.4, lines 106-121): `catch (e) { /* allow through */ }`
- `send-access-gate.js` (v0.1.0, line 107): `catch (e) { /* fall through */ }`

**Impact:** If the Admin Lambda is down or unreachable, token enforcement is silently skipped. Users can upload/download without any token.

**Mitigating factors:**
- This is a **server-side enforcement issue** in `check_access_token()` (line 34-67), not just a client-side issue
- When `admin_service_client is None`, the `check_access_token` method falls back to env-var check, not full bypass
- In production, the admin service should always be available

**Recommendation:** Ensure `check_access_token()` on the server side fails-closed, not fails-open. The server should reject requests when the admin service is unreachable, not allow them through. The client-side degradation is less important since the server is the trust boundary.

### FINDING-4: `page_opened` Consumption Is Semantically Wrong (INFORMATIONAL)

**What:** The `validate-token` endpoint consumes a use with `action='page_opened'`. This means a token use is consumed when someone merely **views** the download page — before they download anything. Browser refreshes, accidental navigations, and bots crawling the URL all consume uses.

**Impact:** Legitimate token usage inflated by non-download page views. A token with 50 uses might only deliver 25 actual downloads if users refresh or navigate back.

**Recommendation:** Token consumption should be tied to the **download action**, not the page view. The server should consume a use when `GET /transfers/download/{id}` or `GET /presigned/download-url/{id}` is called, not when `POST /validate-token` is called from the page load.

### FINDING-5: No Race Condition Protection on Usage Count (LOW)

**What:** `Service__Tokens.use()` performs a non-atomic read-modify-write:
1. Read: `token_data = self.send_cache_client.token__lookup(token_name)` (line 38)
2. Modify: `token_data['usage_count'] = usage_count + 1` (line 57)
3. Write: `self.send_cache_client.token__update(cache_id, token_data)` (line 65)

**Impact:** Concurrent requests could read the same `usage_count` and both increment to the same value, effectively consuming only 1 use instead of 2. With Memory-FS this is unlikely (single-process), but with S3-backed storage under concurrent Lambda invocations, race conditions are possible.

**Mitigating factors:**
- Memory-FS in single-process mode is effectively single-threaded
- The window between read and write is very small
- Worst case is a token gets slightly more uses than its limit — a minor over-consumption, not a security breach

**Recommendation:** Document this as a known limitation. If S3 storage is used in production with concurrent Lambdas, consider using DynamoDB with conditional writes or atomic counters for token state.

## 4. Architecture Assessment

### What's Correct

| Aspect | Assessment |
|--------|-----------|
| Consumption logic is server-side | The client never directly increments `usage_count`. The Admin Lambda's `Service__Tokens.use()` handles all state transitions. |
| Admin Lambda is API-key protected | The client cannot call Admin Lambda endpoints directly. Only the User Lambda (with service-to-service auth) can reach `/tokens/use/`. |
| IP hashing for audit trail | Every consumption event records `ip_hash` — abuse is traceable. |
| Token status transitions | The `active → exhausted` transition is automatic and server-enforced. |
| Token name not leaked in responses | The `complete` endpoint (line 123) explicitly does NOT return `token_name` in shareable URLs. |

### What Needs Attention

| Aspect | Issue |
|--------|-------|
| Public trigger for consumption | The `validate-token` endpoint is the only public endpoint that mutates state (increments `usage_count`). All other User Lambda endpoints are either read-only or require access token. |
| Consumption semantics | Token uses are consumed on **page view**, not on **download**. This is a design choice with real cost. |
| Fail-open on admin unavailability | When admin service is unreachable, the system degrades to no token enforcement. |

## 5. The Human's Core Question — Answered

> "We should never allow the client to 'consume' a token — that can only be done at the server side, since that 'consumption' logic cannot be controlled by the client."

**Answer:** The consumption logic IS server-side. The `usage_count` increment, status transition, and event recording all happen in `Service__Tokens.use()` on the Admin Lambda. The client cannot manipulate these values directly.

**However**, the human's intuition is correct in spirit: **the client can trigger consumption at will**. The `POST /transfers/validate-token/{token_name}` endpoint is a public, unauthenticated trigger for a state-mutating operation. While the mutation itself is safely server-side, the **decision to mutate** is effectively in the client's hands — and that decision currently has no guard rails (no rate limit, no proof of legitimate intent, no binding to a download action).

The fix is not to move the consumption logic — it's already where it should be. The fix is to **gate the trigger**: either rate-limit the public endpoint, or move consumption to an action that proves legitimate intent (e.g., an actual download request with a valid transfer_id).

## 6. Recommended Actions

| Priority | Action | Owner | Effort |
|----------|--------|-------|--------|
| **P1** | Add per-IP rate limiting to `validate-token` endpoint | Dev + DevOps | Low |
| **P2** | Move consumption trigger from `page_opened` to `download_started` — consume on actual download, not page view | Dev + Architect | Medium |
| **P2** | Make `check-token` return identical responses for non-existent and invalid tokens | Dev | Low |
| **P3** | Ensure `check_access_token()` fails-closed when admin service is unreachable | Dev | Low |
| **P3** | Document race condition risk for concurrent Lambda invocations with S3 storage | AppSec + Architect | Low |
