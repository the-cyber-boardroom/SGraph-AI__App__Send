# AppSec Review: Document-Viewer Sandbox Attribute Change

**Version:** v0.11.21
**Date:** 2026-03-06
**Reviewer:** AppSec (Agent)
**Component:** `sgraph_ai_app_send__ui__workspace/v0/v0.1/v0.1.0/components/document-viewer/document-viewer.js`
**Reference:** `sgraph_ai_app_send__ui__workspace/v0/v0.1/v0.1.0/components/js-executor/js-executor.js`

---

## Summary of Change

The HTML preview iframe in `document-viewer.js` (line 543, method `_renderHtml()`) had its sandbox attribute changed:

- **Before:** `sandbox="allow-same-origin"`
- **After:** `sandbox="allow-scripts"`

This aligns the document-viewer with the js-executor component (line 89), which already uses `sandbox="allow-scripts"` for its execution iframe.

---

## Security Analysis

### Previous State: `sandbox="allow-same-origin"`

With `allow-same-origin` alone (no `allow-scripts`):

- **Scripts were blocked.** Any `<script>` tags in the loaded HTML were silently blocked, producing console errors.
- **Same-origin access was granted.** The iframe content was treated as same-origin with the parent, meaning it could theoretically access the parent's DOM, cookies, localStorage, and session storage via JavaScript -- but since scripts were blocked, this access path was inert.
- **Effective security posture:** Strong (no code execution), but the `allow-same-origin` flag was unnecessary and represented a latent risk. If `allow-scripts` had been accidentally added alongside it, the iframe would have had full parent page access.

**Risk assessment (before):** LOW for active exploitation, but the configuration was semantically incorrect -- granting same-origin access to content that cannot run scripts serves no useful purpose and creates a "one flag away from disaster" situation.

### Current State: `sandbox="allow-scripts"`

With `allow-scripts` alone (no `allow-same-origin`):

- **Scripts can execute.** JavaScript in loaded HTML files will run inside the iframe.
- **Same-origin access is denied.** The iframe is treated as a unique, opaque origin. It CANNOT access the parent page's DOM, cookies, localStorage, sessionStorage, or IndexedDB.
- **`postMessage` is available.** The iframe can call `parent.postMessage()`, but the parent must explicitly listen and validate messages. The document-viewer does NOT register any `message` event listeners, so unsolicited messages from previewed HTML are ignored.
- **Network access is available.** Scripts in the iframe can make `fetch()` or `XMLHttpRequest` calls to external servers (subject to CORS). This is the primary residual risk.
- **No form submission.** `allow-forms` is not set, so forms cannot submit.
- **No navigation.** `allow-top-navigation` is not set, so the iframe cannot navigate the parent page.
- **No popups.** `allow-popups` is not set, so `window.open()` is blocked.
- **No pointer lock, no modals.** These are blocked by default in sandboxed iframes.

---

## Is `allow-scripts` Without `allow-same-origin` Safe?

**Yes, this is the correct and recommended configuration** for rendering untrusted HTML content that may contain scripts.

The critical security invariant for sandboxed iframes is:

> **NEVER combine `allow-scripts` and `allow-same-origin` on the same iframe loading untrusted content.**

When both flags are set, the sandboxed content can use JavaScript to remove its own sandbox attribute and escape all restrictions. The W3C HTML specification explicitly warns against this combination.

The current configuration (`allow-scripts` only) ensures:

1. Scripts run in a fully isolated origin (opaque origin, no access to parent resources)
2. The iframe cannot read or write parent DOM, cookies, storage, or credentials
3. The iframe cannot navigate the parent page
4. The iframe cannot open popups or submit forms

This matches the well-established security model used by CodePen, JSFiddle, and similar code preview tools.

---

## Residual Risks

| Risk | Severity | Mitigation |
|------|----------|------------|
| **Outbound network requests** from iframe scripts (data exfiltration of iframe content, beacon to external server) | LOW | The iframe has no access to parent-page data. It can only exfiltrate its own content, which the user already loaded intentionally. No credentials or tokens are accessible. |
| **Crypto-mining / CPU abuse** via long-running scripts | LOW | The browser's own process isolation limits impact. The iframe is hidden (js-executor) or visible (document-viewer) but confined to its panel. Users can navigate away to stop execution. |
| **`postMessage` spam** to parent | LOW | The document-viewer does not listen for `message` events. The js-executor does listen but validates message structure (`msg.type` checks). Malicious messages would be ignored. |
| **Phishing via iframe content** | LOW | The iframe renders inside a workspace panel with clear UI chrome (toolbar, filename). Content cannot escape the iframe boundary or navigate the parent. |
| **Blob URL persistence** | NEGLIGIBLE | Blob URLs are properly revoked in `_revokeBlobUrl()` on component disconnect and before creating new blobs. |

## Comparison with js-executor

The js-executor (line 89) uses the identical sandbox configuration:

```javascript
iframe.sandbox = 'allow-scripts';
```

The js-executor additionally:
- Positions the iframe off-screen (`left:-9999px`)
- Registers a `message` event listener to receive results
- Implements a timeout mechanism to kill long-running scripts
- Cleans up the iframe after execution

The document-viewer's iframe is visible (for HTML preview) and persistent (stays until user navigates away), but the security boundary is identical. The consistency between the two components is appropriate.

---

## Recommendations

1. **No action required on the sandbox change itself.** The new configuration is correct and safer in principle than the old one (eliminates the latent same-origin risk).

2. **Consider adding `Content-Security-Policy` to the blob HTML** (OPTIONAL, low priority). A CSP header or meta tag inside the blob content could restrict `connect-src` to prevent outbound network requests from preview iframes. This is defense-in-depth and not strictly necessary given the opaque origin isolation. Example:
   ```html
   <meta http-equiv="Content-Security-Policy" content="default-src 'unsafe-inline' blob: data:; connect-src 'none'; form-action 'none';">
   ```
   Note: The document-viewer loads raw user HTML via blob URL, so injecting a CSP meta tag would require wrapping the user content, which may affect rendering fidelity. This is a trade-off to evaluate separately.

3. **Consider adding a timeout or size guard** (OPTIONAL, low priority). Unlike the js-executor, the document-viewer does not impose any timeout on script execution within the preview iframe. A runaway script could consume CPU. This is mitigated by browser process isolation but could be addressed with a small watchdog if needed.

---

## Verdict

**APPROVED**

The change from `sandbox="allow-same-origin"` to `sandbox="allow-scripts"` is a security-correct modification that:

- Enables intended functionality (HTML preview with working scripts)
- Maintains strong isolation (opaque origin, no parent access)
- Avoids the dangerous `allow-scripts` + `allow-same-origin` combination
- Aligns with the established security model of the js-executor component
- Follows industry best practice for sandboxed content rendering

No blocking issues identified.
