## Cross-Cutting Themes

Seven strategic threads emerge across all 28 briefs:

### Theme 1: Trust Infrastructure (Briefs 2, 4, 7, 8, 12, 20)

The trust web is the platform's core differentiator. Trust builds slowly through successful transactions and destroys fast on complaints. Network-level detection catches coordinated attackers. The scanning pipeline provides the technical substrate for trust signals. Content trust extends the same model to facts and sources. PKI historical analysis asks the hard question: will this work where PGP failed?

**Key insight:** Trust is a spectrum, not binary. The platform progressively unlocks capabilities based on earned trust.

### Theme 2: File Security and Scanning (Briefs 4, 6, 7, 8, 11)

A comprehensive file security architecture: client-side scanning (ClamAV WASM, YARA, LLM analysis), PKI-verified external scanning chains, CDR for safe delivery, cross-domain checkpoints. The receiver sets the scanning requirements, not the sender. Scanning results are cryptographically signed and bound to file hashes.

**Key insight:** The receiver defines what "safe" means. The sender must comply. Compliance is cryptographically verifiable.

### Theme 3: Data Rooms and Storage (Briefs 6, 14, 15, 17)

Portable data rooms (.sgroom ZIP files), SaaS integration (credit card model for zero-knowledge file storage), data rooms as code (GitHub-managed configuration). MemoryFS abstraction means adding a ZIP backend requires zero changes to application code.

**Key insight:** The data room IS the portable unit. It can live on S3, in a ZIP, on a USB stick, or in an air-gapped environment.

### Theme 4: PKI as Platform (Briefs 1, 3, 13, 16, 23, 24, 26)

GitHub as a pre-existing PKI registry (millions of published SSH/GPG keys). Fractal document signing (per-paragraph, per-clause, per-amendment). Email and messaging integration. Key portability via password managers. Signed Creative Commons as the legal enforcement layer.

**Key insight:** The PKI infrastructure already exists — GitHub users have keys. We don't need to build a registry; we need to use the one that's already there.

### Theme 5: Agentic Security (Briefs 22, 28)

PKI-secured agentic workflows using the same supply-chain pattern as portable data rooms. Critical correction: encryption in shared memory is NOT isolation; container/VM-level separation is required. The .sgworkflow file solves the state management problem across isolated containers.

**Key insight:** Encryption selects WHO can read the data. Process isolation ensures they're the ONLY ones who can.

### Theme 6: Business and Go-to-Market (Briefs 9, 10, 15, 18, 21, 27)

Demo environments as a progressive evaluator journey (Level 1-5 capture-the-flag). Root brand website (sgraph.ai). SaaS integration model. Hyperscaler partnership strategy. Partner programme with reference implementations. DRU/MRU/ARU metrics instead of MRR/ARR.

**Key insight:** Usage is the North Star, not revenue. Every encrypted message — direct or indirect — increases platform value.

### Theme 7: Vision Layer (Briefs 5, 12, 13, 24)

Content trust infrastructure (PKI-signed facts, source attribution, knowledge graphs). Cognitive load protection (information density, reading modes). Fractal document signing (EULA revolution, contract provenance). Signed Creative Commons (legal enforcement for provenance preservation).

**Key insight:** Apply the same patterns from file security to CONTENT security. Malware scanning → fact-checking. Trust web → source reputation. CDR → cognitive load stripping.
