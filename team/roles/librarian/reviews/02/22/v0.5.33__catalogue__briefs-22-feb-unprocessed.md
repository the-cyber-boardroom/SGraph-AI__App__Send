## Brief-by-Brief Analysis

### Brief 1: PKI-Keyed Personal Data Vault (v0.5.15)

**What it introduces:** A personal file system on the server keyed to a user's PKI public key hash. The hash deterministically derives a storage root in S3. Users can create folders, store files, and retrieve them — all addressed by their key identity.

**Key design decisions:**
- Storage root = `sha256(public_key_hash + "/filesystem")` — namespace separation
- Folders are GUID-addressed JSON (children list, no names)
- Files are encrypted blobs stored as cache entries
- Names live ONLY in an encrypted vault index (stored in the vault itself)
- No new API routes — built entirely on existing cache service
- Vault structure is compatible with data room architecture

**Encryption recommendation:** Derived symmetric key (not raw PKI). Consistent with data room model.

**Acceptance criteria:** 10 items — deterministic derivation, no registration, folders/files/index, server sees only GUIDs, no new API routes, data room compatibility.

**Roles affected:** Developer (lead), Architect, AppSec (vault security model), QA (test the flow)

**Cross-references:**
- Builds on PKI infrastructure from 20–21 Feb briefs
- Steps from "send data" to "save and organise data" — beginning of data room
- Personal vault IS a data room with one user

### Brief 2: Reference Implementations — LLM-Consumable Libraries (v0.5.24)

**What it introduces:** A new dependency management pattern for agentic development. Instead of copy-paste or shared libraries (npm/pip), create "reference implementation" repos designed for LLMs to read, understand, and adapt code per project.

**Key concepts:**
- Reference repo = canonical code + over-indexed documentation for LLM comprehension
- Sub-projects pin to reference versions, LLMs adapt features per context
- Required docs: ARCHITECTURE.md, COMPONENTS.md, IMPLEMENTATION_GUIDE.md, CHANGELOG.md, TESTING_GUIDE.md, ADAPTATION_NOTES.md
- Version pinning tracks what each sub-project has synced
- Feedback loop: failed adaptations improve the reference

**Commercial model:** Open source code, paid maintenance/curation/adaptation support.

**First candidate:** Admin UI pattern (IFD web components, message panel, debug panel) used across 4+ projects.

**Acceptance criteria:** 10 items — reference repo, 6 documentation files, version pinning, adaptation documentation, propagation capability, agent persona.

**Roles affected:** Architect (lead), Developer, Librarian (documentation structure), all roles (cross-project pattern)

**Significance:** This is a **paradigm-level brief** — defines how the project manages shared code in an agentic world. Affects all future sub-projects.

### Brief 3: Receiver Experience and the Trust Spectrum (v0.5.24)

**What it introduces:** Complete overhaul of the receiver experience. Replaces gibberish hex URLs with short codes (6 digits). Self-destructing codes. Environmental factors as silent security. Progressive trust from short code → full PKI via two-message key exchange.

**Key design decisions:**
- Short code is NOT the decryption key — it's a one-time-use token that retrieves the real key
- Self-destruct on failure (1–3 attempts max) — even 4-digit codes become secure
- Environmental factors (timezone, country, browser, language) silently augment security
- Two-message key exchange: after one send + one reply, both sides have PKI keys
- Data room access tiers: Lobby → Member → Verified → Admin
- Friendly code formats: numeric, alphanumeric, three words, phrase, personal challenge

**Core principles:**
1. Security is a spectrum, not binary
2. Security should be invisible to users
3. Never compromise server-side encryption
4. Trust builds with usage (TOFU)

**AppSec tasks:** 6 risk assessment items flagged (self-destruct race conditions, TOFU attack window, auto-generated PKI consent, environmental factor security model)

**Designer/Sherpa tasks:** 4 design briefs needed (first-time receiver flow, PKI notification, reply UX, trust indicators)

**Acceptance criteria:** Not explicitly numbered but implied across the brief.

**Roles affected:** Architect (lead), AppSec, Developer, Designer, Sherpa

**Significance:** This is the **most user-facing brief since the project started** — directly addresses the receiver (customer) experience.

### Briefs 4–6: QA Project Bootstrap Pack

These three briefs collectively define a **new separate project** — a browser automation test suite that simultaneously validates SG/Send UI and generates living documentation from screenshots.

**Key characteristics:**
- Separate repo (`sgsend-qa`), separate team (6 roles), separate deployment
- Tests SG/Send as a consumer (read-only), does not modify it
- API-first + CLI-first design
- Runs in: GitHub Actions, CLI, Claude Code, Lambda, Web UI
- Screenshots committed only on visual diff (>1% pixel change)
- Documentation published to GitHub Pages
- Preferred browser automation: Vercel Agent Browser (fallback: Playwright)

**Phase plan:** Pipeline before tests — repo setup → CI pipeline → browser automation research → first smoke test → dogfooding

**Primary test target:** User Lambda at send.sgraph.ai (landing → token → error → upload page flow)
