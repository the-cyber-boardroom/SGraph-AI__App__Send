## Context

The download page on dev.send.sgraph.ai was broken due to a stale file in the S3 `latest/` folder. A previous deployment had placed a v0.2.1 override file at the path `send-download.js`, overwriting the v0.2.0 base class. After the file was renamed to `send-download-v021.js`, the old copy persisted in S3 because the IFD overlay model uses `aws s3 sync` without `--delete`.

This session diagnosed the root cause, fixed the deployment pipeline, and produced a comprehensive architecture review documenting the entire IFD deployment model.

## Code Changes

### `scripts/deploy_static_site.py`

| Function | Purpose |
|----------|---------|
| `clean_latest()` | Delete all files in `latest/` before overlay (recovery tool) |
| `list_releases()` | Parse S3 `releases/` to find all deployed versions, sorted numerically |
| `rebuild_latest_from_releases()` | Delete `latest/`, replay all releases oldest-first (nuclear recovery) |
| `sync_all_types_to_s3()` | Refactored helper ‚Äî eliminates duplicated sync-by-type blocks |
| `deploy_to_s3()` | Updated ‚Äî accepts `do_clean_latest` parameter |

New CLI arguments: `--clean-latest`, `--rebuild-latest`

## Signals

| Signal | Importance | Action |
|--------|-----------|--------|
| Stale files in IFD overlay are a real risk | HIGH | Override files must always use version-suffixed names (e.g. `-v021.js`) from the start |
| Workflow must deploy all versions in the chain | HIGH | Fixed ‚Äî workflow now deploys v0.2.0 + v0.2.1; extend for v0.2.2+ |
| Guard clauses in override files are critical | MEDIUM | Every IFD override should check for its base class before patching |
| CI-generated i18n pages work well with IFD | LOW | Pre-rendered locales generated per version, no runtime cost |
