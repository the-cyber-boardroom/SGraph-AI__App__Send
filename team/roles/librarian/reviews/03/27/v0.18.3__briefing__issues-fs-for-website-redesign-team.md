**Version:** v0.18.3 | **Date:** 27 March 2026 | **Prepared by:** Librarian (Explorer team)
**For:** Website Redesign Team (Designer, Sherpa, Architect, Ambassador)
**Also relevant to:** Conductor, Dev, any agent session touching `sgraph_ai__website/`

---

## Why Issues-FS for This Project

The website redesign involves **4 roles producing content in parallel** (Designer, Sherpa, Architect, Ambassador), a **Dev role assembling it**, and a **Conductor orchestrating**. Work must be handed off between roles — the Sherpa writes copy, the Designer produces layouts, the Dev builds HTML, the Architect reviews claims.

Issues-FS gives this team a **shared task board that lives in Git**. Every agent can see what's assigned to them, what's blocked, and what's done — without needing a separate tool. Tasks are created, updated, and completed by editing text files or running CLI commands. Everything is version-controlled, so handoffs are traceable.

## The Format: Flat `.issues` Files

For this project, use the **lightweight flat file format** — not the full JSON `issue.json` tree. It's simpler, faster to edit, and designed for agent authoring.

### Location

All website redesign issues go in:

```
.issues/website-redesign/
```

### File Format

One issue per line, pipe-delimited:

```
{Type}-{Number} | {status} | {description}
```

Children are tab-indented under their parent. Comments start with `#`.

### Example

```
# .issues/website-redesign/workstream-homepage.issues

Workstream-1 | in-progress | Homepage Redesign (Phase 2)
	Task-1 | todo | Write hero copy (H1 + sub + CTAs) -> Sherpa
	Task-2 | todo | Write "How it works" 3-step copy -> Sherpa
	Task-3 | todo | Write trust section copy (six-sentence policy) -> Sherpa
	Task-4 | todo | Write share modes card copy -> Sherpa
	Task-5 | todo | Design hero two-column layout -> Designer
	Task-6 | todo | Design feature showcase (3 screenshot rows) -> Designer
	Task-7 | todo | Design trust/privacy section -> Designer
	Task-8 | blocked | Build hero HTML/CSS -> Dev
	Task-9 | blocked | Build feature showcase HTML/CSS -> Dev
	Task-10 | blocked | Build trust section HTML/CSS -> Dev
	Task-11 | blocked | Build share modes section HTML/CSS -> Dev
	Task-12 | todo | Review all homepage claims against reality doc -> Architect
	Task-13 | todo | Add Product schema + HowTo schema to head -> Architect
```

### Valid Types and Statuses

| Type | Statuses | Use for |
|------|----------|---------|
| `Workstream-N` | todo, in-progress, done, blocked | A phase or major deliverable |
| `Task-N` | backlog, todo, in-progress, review, done | A unit of work for one role |
| `Bug-N` | backlog, confirmed, in-progress, testing, resolved, closed | Something broken |
| `Feature-N` | proposed, approved, in-progress, released | A capability being built |
| `Question-N` | open, answered, closed | A decision needed from Dinis or another role |

### Cross-References

Use `->` to link to other issues or to indicate which role owns a task:

```
Task-1 | todo | Write hero copy -> Sherpa
Task-8 | blocked | Build hero HTML -> Task-1, Task-5
```

## How Roles Use Issues-FS

### Creating and Updating Tasks

**Option A: Edit the file directly** (simplest for agents)

Open the `.issues` file, change the status:

```diff
- Task-1  | todo | Write hero copy -> Sherpa
+ Task-1  | done | Write hero copy -> Sherpa
```

Commit the change. That's it.

**Option B: Use the CLI**

```bash
# List all tasks assigned to you
issues-fs list --status todo

# Update a task status
issues-fs update Task-1 --status in-progress

# Show a workstream with all children
issues-fs show Workstream-5 --depth 2

# Check all files for format errors
issues-fs check
```

### The Handoff Pattern

This is the key workflow for multi-role collaboration:

**Step 1: Role A completes their task and marks it done**
```
Task-1  | done | Write hero copy -> Sherpa
```

**Step 2: Role A unblocks the dependent task**
```diff
- Task-12 | blocked | Build homepage HTML -> Dev
+ Task-12 | todo | Build homepage HTML -> Dev
```

**Step 3: Role A commits both changes in one commit**
```
git add .issues/website-redesign/workstream-phase2-homepage.issues
git commit -m "Sherpa: hero copy complete, unblock Dev homepage build"
```

**Step 4: Role B (Dev) picks up the unblocked task**
```diff
- Task-12 | todo | Build homepage HTML -> Dev
+ Task-12 | in-progress | Build homepage HTML -> Dev
```

### Team-to-Team Handoffs

When a full workstream is complete and ready for the next phase:

```diff
- Workstream-2 | in-progress | Phase 1: /security/ page
+ Workstream-2 | done | Phase 1: /security/ page
```

The Conductor monitors workstream statuses to know when phases are complete and the next phase can begin.

### Adding New Tasks Mid-Sprint

Any agent can add tasks as they discover work:

```
# Added by Dev during Phase 2 build
	Task-18 | todo | Mobile breakpoint for hero two-column layout -> Designer
	Task-19 | todo | Retina @2x variants for all 8 screenshots -> Task-1 through Task-8
```

Just append to the relevant workstream file, increment the task number, and commit.

### Raising Blockers

If a task is blocked, change its status and add a cross-reference:

```
Task-12 | blocked | Build homepage HTML -> Task-1, Task-6
```

This tells everyone: Task-12 can't start until Task-1 (hero copy) and Task-6 (hero layout design) are done.

### Filing Bugs

Add to `bugs.issues`:

```
# .issues/website-redesign/bugs.issues

Bug-1 | confirmed | Hero screenshot too wide on mobile (crops gallery badges)
Bug-2 | confirmed | /security/ FAQ schema missing closing bracket -> Architect
```

## Role-Specific Instructions

### Designer
- **Your tasks** have `-> Designer` suffix
- When you produce a layout/design, update the task to `done` and unblock any `blocked` Dev tasks that depend on it
- If you need content before designing, the Sherpa task should be `done` first — if it's not, your task stays `blocked`

### Sherpa
- **Your tasks** have `-> Sherpa` suffix
- You produce copy. When done, mark `done` and unblock the downstream Design and Dev tasks
- You are the first mover in most workstreams — your output unblocks everyone else

### Architect
- **Your tasks** have `-> Architect` suffix
- Review tasks: verify claims against reality document, check schema markup
- You can flag issues by adding `Bug-N` entries to `bugs.issues`
- You are the quality gate — nothing ships without your review tasks being `done`

### Ambassador
- **Your tasks** have `-> Ambassador` suffix
- Launch content (announcement, social posts) can be prepared in parallel with Dev build
- Your work has fewer dependencies — most tasks are `todo` not `blocked`

### Dev
- **Your tasks** have `-> Dev` suffix
- Most of your tasks start as `blocked` — they depend on Sherpa copy and Designer layouts
- When all your dependencies are `done`, change your task from `blocked` to `in-progress`
- When you complete a build task, also update relevant `workstream-phase*.issues` status

### Conductor
- Monitor workstream-level statuses across all files
- When all tasks in a workstream are `done`, mark the workstream `done`
- When Phase N is complete, communicate to the team that Phase N+1 can begin
- Use `questions.issues` to track decisions needed from Dinis

## Workflow Summary

```
Sherpa writes copy ──────┐
                         ├──> Dev builds HTML ──> Architect reviews ──> Done
Designer produces layout ┘
                                                  Ambassador prepares launch content (parallel)
```

In `.issues` terms:

```
Task-1 (Sherpa: write copy)     | todo → in-progress → done
Task-5 (Designer: design layout)| todo → in-progress → done
Task-8 (Dev: build HTML)        | blocked → todo → in-progress → review → done
Task-12 (Architect: review)     | blocked → todo → in-progress → done
Task-16 (Ambassador: launch)    | todo → in-progress → done (parallel track)
```

## CLI Quick Reference

```bash
# Validate all .issues files
issues-fs check

# List everything in the website-redesign directory
issues-fs list

# Show a specific workstream with children
issues-fs show Workstream-5 --depth 2

# Filter by status
issues-fs list --status blocked
issues-fs list --status todo

# Update a task
issues-fs update Task-1 --status done

# Export flat files to JSON (canonical format)
issues-fs normalise --dry-run
issues-fs normalise
```

## Getting Started

1. **Conductor** creates the `.issues/website-redesign/` directory and commits the starter files above
2. **Each role** reads their `-> Role` tasks across all workstream files
3. **Sherpa** starts first — most Phase 1 and Phase 2 tasks depend on copy
4. **Designer** starts on layout tasks that don't need copy (e.g., page templates, CSS tokens)
5. **Architect** starts on schema and reality-check tasks (independent of copy/design)
6. **Ambassador** starts on launch content (independent track)
7. **Dev** picks up tasks as they become unblocked
8. **Everyone** commits `.issues` file updates alongside their deliverables
