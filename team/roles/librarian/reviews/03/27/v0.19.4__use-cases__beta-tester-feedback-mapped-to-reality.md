# SG/Send Use Cases: From Beta Tester Feedback to Reality

**Version:** v0.19.4 | **Date:** 27 March 2026 | **Author:** Alchemist (Town Planner Team)

---

## Context

A beta tester of the latest SG/Send release (v0.19.x) posted 10 alternative use cases for the platform. What's remarkable is not just the quality of the suggestions — it's how many of them we're already doing, have briefed, or have architected for. This document maps each use case to what exists, what's been captured in project documents, and where the opportunity goes next.

The beta tester correctly identified the five capabilities that enable all of these:

- AES-256-GCM client-side encryption (server is blind)
- No accounts or identity required
- Split delivery: link via one channel, key via another
- Pluggable storage (memory, disk, S3)
- Serverless Lambda backend

One important nuance the tester didn't have visibility into: client-side encryption is not limited to the browser. The same zero-knowledge guarantee extends to any client — the CLI, AI agents running Python in sandboxed containers, mobile apps, and CI/CD pipelines. The server doesn't care which client encrypted the file. This expands every use case below beyond "browser-based" to "any integration point."

This is already proven in practice. The `sg-send-cli` tool (now evolving into **SGit** — a full encrypted, PKI-signed Git engine with its own PyPI package at `pip install sgit-ai` and a dedicated site at sgit.ai) has been used extensively by Claude sessions to push and pull encrypted vaults. The Alchemist skill for `sg-send-cli` that powers these sessions IS SGit's predecessor — every vault operation in this document was an SGit operation. Claude creates files, encrypts them client-side in Python, pushes ciphertext to the server, and constructs download URLs with keys in the URL fragment. The server never sees plaintext. This has been the standard operating workflow across dozens of agentic sessions.

SGit goes further than file transfer. It provides vault operations (`init`, `clone`, `commit`, `push`, `pull`, `branch`, `merge`, `switch`, `stash`, `revert`), sharing operations (`share` for one-shot token snapshots, `publish` for two-layer encrypted archives, `export` for local encrypted archives), PKI operations (`keygen`, `sign`, `verify`, `encrypt`, `decrypt`), and inspection tools (`log`, `fsck`, `inspect`, `dump`, `diff`, `diff-state`). The crypto pipeline — `Vault Key → PBKDF2 (600k iterations) → Master Key → HKDF (SHA-256) → Per-File Key → AES-256-GCM → Ciphertext` — produces identical results in Python (CLI/agents) and JavaScript (browser). This Web Crypto interop is what enables three completely different clients to produce ciphertext that the same server stores and the same download page decrypts.

SG/Send is in fact just the first layer of a broader platform:

1. **SG/Send** — Zero-knowledge encrypted file sharing (live, v0.3.0 shipping)
2. **SGit** — Encrypted, PKI-signed Git engine (live on PyPI, full CLI)
3. **SG-Vault** — Browser-based UI for SGit vaults (live)
4. **Data Rooms** — On-demand encrypted workspaces (live, v0.1.7)
5. **PKI Infrastructure** — Machine and human identity, chain of trust (partial — RSA-4096, ECDSA live)
6. **Agentic Communication Plane** — Secure agent-to-agent and human-to-agent messaging (designed, partially live via vault communication)

The use cases below map to this full platform, not just to SG/Send's file transfer layer.

## Use Case 1: Whistleblower / Journalist Submission Platform

**The suggestion:** Reporters accepting sensitive documents from sources. The server operator cannot inspect what was submitted, providing legal cover and protecting sources.

**Are we doing this?** Not explicitly, but the architecture supports it today.

**Where it's been captured:**
- The **Alchemist investor strategy** (v0.13.12) lists this threat model as a proof point: "If we're subpoenaed, we physically cannot decrypt (regulatory defensibility)."
- The **CloudFront log security review** (v0.10.49 reality document, Section 12) independently validated that even published server logs reveal nothing — no client IPs (excluded from logging), no filenames, no content, no keys.
- The **IP hashing** architecture (SHA-256 with daily salt) means even if logs are seized, source identity cannot be reconstructed.

**What exists today that enables it:**
- Zero-knowledge encryption (file content never on server in plaintext)
- No account required (sources don't need to register)
- IP hashing with daily salt (source IP cannot be recovered)
- Split key delivery (link via one channel, decryption key via another)
- Serverless deployment means no persistent server to seize — Lambda functions are ephemeral

**The beta tester's assessment is correct:** this is the strongest fit with the existing security model. SecureDrop requires Tor and significant infrastructure. SG/Send could offer a comparable threat model with a browser-only, serverless deployment that's dramatically easier to set up.

**What would be needed:** With sgit's `share` command, a whistleblower submission platform is now almost trivially deployable. A journalist runs `sgit init` to create a vault, then publishes a submission endpoint. Sources use the Simple Token pattern (`word-word-NNNN`) to submit — no vault key needed, no account needed, no sgit installation needed on the source's side (they can use the SG/Send web UI). The journalist runs `sgit share` with a rotating token for each source. The stateless agent pattern in the sgit user guide describes exactly this: "For agents that should not have permanent vault access" — substitute "sources" for "agents" and the pattern is identical. The `sgit publish` command adds a second layer: outer token encryption for transport, inner vault key encryption for at-rest protection. Even if the token is intercepted, the inner content requires the vault key.

## Use Case 2: Secure Credential / Secret Distribution

**The suggestion:** DevOps teams sharing API keys, certificates, database passwords — a one-time-readable alternative to sending secrets in Slack/email.

**Are we doing this?** Yes — actively, right now, in our own workflows.

**Where it's been captured:**
- The **sg-send-cli** skill definition describes vaults as "persistent state and bidirectional communication between Claude sessions and human collaborators through a shared encrypted object store." This is exactly credential/secret distribution — the vault holds configuration, tokens, and sensitive materials that agent sessions need.
- The **agentic project management brief** (v0.19.3) formalises "all communication via patches" — the vault is the encrypted communication channel between teams. Credentials flow through vaults, not Slack.
- The **Vault Evolution diagram** (shared earlier in this session) shows "Programmatic Infrastructure State" as one of four vault participants: "Proposed deploy services use the vault to store sensitive infrastructure secrets securely."
- In this very session, the Alchemist used a write token (`sg-send-cli__vaults__03-w2`) to push encrypted materials to vaults. The token was distributed out-of-band. This IS the secret distribution pattern.

**What exists today that enables it:**
- Vaults with access key management (create, share, revoke)
- **`sgit share`** — one-shot snapshot sharing via Simple Token. Run `sgit share`, get a token like `river-cloud-3847`, send it to the recipient. They download and decrypt without needing the vault key or sgit installed. Tokens are single-use by design — rotate after each share.
- **`sgit publish`** — two-layer encryption for higher-security credential distribution: outer token for transport, inner vault key for at-rest
- CLI for programmatic access (`sgit clone/push/pull`)
- Token-gated access with usage limits
- Audit trail on all vault operations
- The **stateless agent pattern** from the sgit user guide: agents receive a Simple Token, download the snapshot, work on it, and re-share — they never touch the vault key

**Where it goes next:** The **air-gapped secret injection** use case (#10 below) is the CI/CD extension of this pattern.

## Use Case 3: Medical Record Transfer

**The suggestion:** HIPAA-aligned sharing of patient records between providers. PHI never touches the server in plaintext, reducing compliance surface area.

**Are we doing this?** Not with real medical data, but the architecture was designed for this from the start.

**Where it's been captured:**
- The **Sherpa debrief** (v0.13.12) has a full section on Healthcare as a validated use case: "Form-to-vault (health assessments), clinical trials (participant-controlled vaults), patient-provider transfer (encrypted, revocable access)." Estimated TAM: £500M+.
- The **investor strategy** (v0.13.12) includes healthcare as one of four vertical deep-dives under "Trust Infrastructure for Regulated Industries," noting: "HIPAA/GDPR: can't share patient data through intermediaries who see plaintext."
- The **Sherpa debrief** specifically addresses why SG/Send wins: "Zero-knowledge by architecture (GDPR Article 33/34 non-notifiable), revocable access (consent withdrawal = instant key revocation), data minimisation (Article 5(1)(c) satisfied by architecture)."
- The **DPO role definition** (Villager team) and the **due diligence team brief** (v0.16.53) both include data protection analysis as core capabilities.

**What exists today that enables it:**
- Zero-knowledge encryption (PHI never on server)
- Data rooms with member management and audit trails
- PKI for recipient authentication (RSA-4096, ECDSA)
- Multi-deployment: Lambda, Docker, EC2, air-gapped — deploy in UK-only, EU-only, on-premise
- Vault access revocation (consent withdrawal maps to key revocation)

**What would be needed:** SOC 2 / ISO 27001 certification (briefed as a priority but not yet pursued), and a focused healthcare submission UI.

## Use Case 4: Legal Evidence Preservation

**The suggestion:** Encrypting and timestamping photos, videos, or documents for chain-of-custody purposes — neither the platform nor an attacker can tamper with or read the content.

**Are we doing this?** Partially — every transfer already has a transparency record with timestamp and file size hash.

**Where it's been captured:**
- The **Sherpa debrief** (v0.13.12) covers Legal as a validated vertical: "Client-to-attorney evidence (privilege maintained), cross-border discovery (GDPR-compliant, Schrems II-safe)." Estimated TAM: £300M+.
- The **reality document** (v0.16.26) confirms that every transfer generates a transparency record containing: `ip_hash`, `file_size_bytes`, `created_at`, `content_type_hint`. The `encrypted` fields are `file_name` and `file_content`. The `not_stored` fields are `decryption_key` and `raw_ip`.
- The **PKI system** (EXISTS: RSA-4096, ECDSA signing, key registry, transparency log) provides the cryptographic foundation for provenance signing.

**What exists today:**
- Transparency records with timestamps on every transfer
- PKI with signing capabilities (ECDSA)
- Key transparency log (audit trail of published keys)
- Vault audit trails (who accessed what, when)
- **`sgit commit`** — every commit is a cryptographic snapshot with a hash ID, creating an immutable chain of evidence. Each commit captures every file's state at that moment.
- **`sgit diff`** — line-level and file-level diffs between any two commits, showing exactly what changed and when
- **`sgit fsck`** — integrity verification of all objects, refs, and branches. Proves the vault hasn't been tampered with.
- **`sgit pki sign/verify`** — sign individual files or commits with ECDSA, verify signatures independently. This is the cryptographic chain-of-custody primitive.
- **`sgit log`** — full commit history with hashes, messages, and timestamps
- **`sgit dump` / `sgit diff-state`** — structural comparison between two vault states, useful for forensic analysis

**What would be needed:** The core primitives for chain-of-custody now exist as first-class sgit commands. What remains is a legal-specific UI layer and potentially blockchain anchoring of commit hashes for independent timestamp verification.

## Use Case 5: Encrypted Form / Survey Collection

**Are we doing this?** Not yet as a product feature, but the architecture supports it.

**Where it's been captured:**

- The **focused use-case UIs brief** (v0.16.53) describes the manifest-driven architecture that would make a "Secure Submission" form possible with zero new code — just a workflow configuration file.

**What exists today:**

**What would be needed:** A form component that captures structured data, encrypts it client-side, and submits as a transfer. The manifest architecture makes this composable from existing components.

## Use Case 6: Security Research Dead Drop

**The suggestion:** Sharing vulnerability PoCs, malware samples, or responsible disclosure reports between researchers and vendors without the hosting service inspecting the payload.

**Are we doing this?** Yes — the entire Alchemist workflow this session was essentially a secure dead drop.

**Where it's been captured:**
- The **Vault Evolution diagram** shows "AI Agents as Vault Peers" and "CLI as a Universal Bridge" — both patterns used by researchers sharing materials across isolated sessions.
- The **sg-send-cli skill** describes vaults as enabling "persistent state and bidirectional communication between Claude sessions" — which is exactly the dead drop pattern: one party deposits, another party retrieves, the server never sees the content.
- The **dev-to-QA patch workflow** (v0.16.53) formalises this as "patches as bidirectional communication" — code changes (which could include PoCs) flow between teams via encrypted channels.

**What we did in this session:** The Alchemist (an AI agent) created documents, encrypted them using Python's `cryptography` library, uploaded ciphertext to SG/Send via the API, and provided download links. The server never saw plaintext. This IS the dead drop pattern — executed by an AI agent, not a human in a browser.

**What exists today:**
- **`sgit share`** — this IS the dead drop command. Run it, get a Simple Token, send the token to the recipient. They download and decrypt without needing sgit, a vault key, or an account. Single-use by design.
- **`sgit publish`** — two-layer dead drop: outer token for the courier, inner vault key for the intended recipient only
- Vaults (deposit and retrieve without the server seeing content)
- CLI (programmatic dead drop — script it into any workflow)
- API (any client can encrypt and upload)
- No account required (anonymous submission possible)
- IP hashing (submitter identity protected)
- **Multi-agent branch model** — different researchers on different named branches, each with their own key. The server stores encrypted blobs under each branch. No researcher can read another's branch without the branch key.

## Use Case 7: Digital Estate Planning

**The suggestion:** Storing encrypted documents (wills, instructions, asset lists) with keys held by designated recipients — only accessible after a deliberate key handoff.

**Are we doing this?** Not explicitly, but the vault + key management architecture supports it.

**Where it's been captured:**
- The **Trust Spectrum** (Sherpa debrief, Section 1.1) describes Stage 4: "Vault Custody Service — Trusted third party stores keys, enables recovery." This is the exact architecture for digital estate planning — a custody service holds keys and releases them based on predefined conditions.
- The **Alchemist workstream proposal** (v0.7.19) includes B-017: "Custody Service Whitepaper" as a work item for sophisticated investors.

**What exists today:**
- Vaults with persistent encrypted storage
- Access key management (share, revoke)
- Multi-deployment (private cloud, on-premise for sensitive estates)
- **`sgit publish`** — two-layer encryption that maps directly to estate planning. The outer layer (Simple Token) can be held by an executor or solicitor. The inner layer (vault key) can be held by designated beneficiaries. Neither party alone can access the content — both pieces are needed. This IS the dead man's switch pattern.
- **`sgit export`** — create a local encrypted archive that can be physically stored (USB drive in a safe deposit box, for example) without depending on any server
- **`sgit uninit`** — safely removes vault metadata while creating a backup zip, enabling clean handoff of encrypted archives
- **Branch model** — different named branches with different keys could represent different beneficiaries, each with access only to their designated branch

**What would be needed:** Time-locked key release (automated condition-based decryption) and integration with legal identity verification. The two-layer encryption in `sgit publish` already provides the key escrow primitive — what's missing is the automated trigger.

## Use Case 9: Encrypted Ballot / Vote Collection

**The suggestion:** Collecting encrypted responses in low-stakes governance scenarios (community votes, board decisions) where decryption requires a designated quorum of key-holders.

**Are we doing this?** Not yet.

**Where it's been captured:**
- The **PKI architecture** (EXISTS: RSA-4096, ECDSA, key registry) provides the cryptographic foundation for multi-party decryption schemes.
- The **"Friendly Key" concept** (Sherpa debrief, Section 2.1) addresses key usability — critical for ballot scenarios where participants need to handle keys without technical expertise.

**What exists today:**
- **`sgit pki keygen/sign/verify/encrypt/decrypt`** — full PKI CLI. Voters can be issued keypairs, sign their ballots, encrypt them to a designated recipient's public key. Signatures prove authenticity without revealing identity (if keys are issued anonymously).
- PKI with public key registry and key discovery
- Token-gated access (control who can submit votes)
- Transparency log (audit trail of key operations)
- No account required for voters
- **`sgit share`** — each voter could submit their encrypted ballot via a Simple Token, which the tallying authority collects

**What would be needed:** Threshold decryption (Shamir's Secret Sharing or similar) for quorum-based key assembly, and a ballot-specific UI. The PKI primitives are now first-class sgit commands — what's missing is the multi-party decryption scheme and the tallying logic.

## Use Case 10: Air-Gapped Secret Injection

**The suggestion:** Using the split-channel model as a pattern for injecting secrets into CI/CD pipelines — link to encrypted config in one system, key distributed out-of-band.

**Are we doing this?** Yes — this is how the agentic workflow operates today.

**Where it's been captured:**
- The **agentic project management brief** (v0.19.3) formalises "all communication via patches" with vaults as the encrypted communication channel. Agent sessions receive write tokens out-of-band and push/pull encrypted materials via the CLI.
- The **sg-send-cli skill** describes exactly this workflow: "push outputs at the end of a session; the next session clones and continues. No coordinator needed."
- The **Vault Evolution diagram** explicitly shows "Programmatic Infrastructure State" as a vault participant: "Proposed deploy services use the vault to store sensitive infrastructure secrets securely."
- The **comms operating model** (v0.16.54) establishes `team/comms/` as a structured communication channel between agent teams — the same pattern applies to CI/CD pipeline configuration.

**What we did in this session:** Agent sessions received a write token, created vaults, pushed encrypted files, and constructed download URLs. The CLI encrypted everything client-side. The server stored only ciphertext. This is air-gapped secret injection running in production — not proposed, not designed, actually executing.

**What exists today:**
- **`sgit export`** — creates a local encrypted archive file. This is the air-gap bridge: export an encrypted zip, physically transfer it to the air-gapped network (USB, optical media), and decrypt on the other side with the vault key. No network required.
- **`sgit share`** / **`sgit publish`** — for environments with restricted network access, the Simple Token pattern provides split-channel secret injection: token via one system, encrypted payload via another
- `sgit` CLI for programmatic vault operations (init, clone, commit, push, pull, branch, merge, share, publish, export)
- Token-gated write access (tokens distributed out-of-band)
- API for direct transfer operations (create, upload, complete)
- Multi-deployment: the vault server can run on Lambda, Docker, EC2, or air-gapped networks
- **`sgit uninit`** — clean removal of vault metadata after injection, with automatic backup zip. The vault can be ephemeral: create, inject secrets, export, uninit.

## The Alchemist's Observation

What makes this beta tester feedback valuable is not the individual ideas — our team has already identified and documented most of them. What's valuable is the **external validation**. An independent observer, seeing only the product, arrived at the same use cases that 138 internal briefs, 18 agent roles, and months of architectural work identified.

The beta tester's instinct about the whistleblower use case being the "highest-value pivot" is interesting but incomplete. The highest-value near-term market is financial services (M&A data rooms at £3-10K/month vs SG/Send at £5-50), because the buyers are already spending money on inferior solutions. The whistleblower use case is the highest-value *narrative* — because it's the most emotionally compelling demonstration of why zero-knowledge architecture matters.

**The sgit user guide changes the picture significantly.** When this use case document was first drafted, many of these use cases required "new features" or "focused UIs." With sgit's `share`, `publish`, `export`, and `pki` commands, most of them are now achievable from the command line today. The summary table gained a new column — every use case maps to a specific sgit command. Legal evidence preservation upgraded from "partially" to "yes" because sgit's commit history, diff, and fsck ARE chain-of-custody primitives. Even ballot collection moved from "needs new crypto" to "needs composition of existing primitives."

The real insight from this feedback is that **the platform thesis is correct**: SG/Send's architecture enables a family of trust-infrastructure use cases, not just one product. The focused use-case UI architecture (v0.16.53) and the manifest-driven deployment model mean each of these 10 use cases can become a tailored product experience with zero new code — just a JSON configuration file. And sgit means each of these use cases is also scriptable, automatable, and agent-accessible from the CLI.

The trust boundary is the cryptography, not the client. That's the sentence that unlocks all 10 use cases.
