### Theme 1: The `_page.json` Stack — Brief → Proposal → Implementation (Documents 199–201)

These three documents represent the full lifecycle of the `_page.json` feature:

**Document 199 (dev brief):** The original design rationale — why JSON (not HTML), why the manifest-driven pattern, how Claude writes layout files, the fractal folder-as-site structure. Defines 10 component types and the workflow for Claude to produce them.

**Document 200 (v2 proposal):** Schema extensions based on the first rendered output. Five-property `theme` object (`mode`, `accent`, `hero_style`, `font`, `density`). Component-level styling props. Two new components: `columns` (side-by-side layout with ratio) and `cards` (navigable card grid).

**Document 201 (implementation brief):** Production-ready spec for the UI team. Detection logic, TypeScript-style schema definitions, Priority 1/Priority 2 component breakdown, file path resolution rules (`../` traversal), test vault (`lend-folk-9817`), implementation order suggestion, and coexistence rules with markdown.

**Reality cross-check:** All three documents describe features that are now **SHIPPED**. By v0.20.13–v0.20.20 (see reality document), the `_page.json` system, PLR (PageLayoutRenderer), v2 theme schema, `columns`, and `cards` components all exist and work. These briefs are historical record — the inspiration and spec behind what was built.

### Theme 2: Vault-Browse Unification (Document 189)

A strategic architecture decision: strip the vault website's custom Browse implementation and replace it with the shared Browse module from SG/Send, injecting a vault API data source adapter instead of the zip extraction adapter.

**Key insight:** The vault view is actually faster for initial load (tree-only API call vs full zip download) and the architecture naturally extends to fractal vault navigation (vault-in-vault browsing).

**Reality cross-check:** **PROPOSED — does not exist yet.** The vault website and SG/Send browse remain separate implementations.

**Acceptance criteria (7 items):** Browse module extracted as shared component; data source interface defined; vault site imports the module; all recent Browse features available in vault view; vault-specific code stripped to adapter + write ops; fractal vault support; parent vault as vault manager.

### Theme 3: sg-llm Component Family (Document 202)

The Architect (Explorer team) designed a family of 11 independently-versioned IFD components for client-side LLM integration, centred on a **reality constructor** — a UI where users visually build the model's entire context before sending (system prompt + context blocks + constructed history + question + live cost estimate).

**11 components:**

| Component | Role |
|-----------|------|
| `sg-llm-events` | Shared event constants (frozen at v0.1.0) |
| `sg-llm-connection` | Provider/key/model management |
| `sg-llm-reality` | Reality constructor (THE central component) |
| `sg-llm-request` | Fetch engine (headless) |
| `sg-llm-output` | Streaming response display |
| `sg-llm-stats` | Token counts, cost, speed metrics |
| `sg-llm-debug` | Full request inspector |
| `sg-llm-infographic` | SVG extraction + rendering |
| `sg-llm-bundle` | Execution bundle manager |
| `sg-llm-bundle-list` | Bundle browser UI (time travel) |
| `sg-llm-attachments` | File drop, clipboard paste, image cache |

**Design pattern:** Hosted at `tools.sgraph.ai` in `components/llm/` namespace folder. Each component has its own IFD versioning. Version-stamped surgical override files (`sg-llm-reality--v0.1.1.js`) following the convention in Document 197.

**Reality cross-check:** **PROPOSED — does not exist yet.** No sg-llm components are code-verified.

### Theme 4: Client-Side Tools — Transcription, WASM, and Vault Migration (Document 194)

Three related features driven by the offline-first, client-side-capable theme:

1. **Audio transcription** (Whisper WASM): Two-pass architecture — real-time pass (live as audio plays) + clean-up pass (post-recording, timestamps, diarisation). `sg-audio-transcription` component.

2. **`sg-wasm` loader**: Shared lifecycle management for WebAssembly modules. Download → cache in IndexedDB → validate hash → offline use. Serves video editing (FFmpeg), transcription (Whisper), and future tools. `sg-local-storage` integration for storage viewer.

3. **Vault migration**: Multi-remote vault workflow (`sgit-ai remote add/push/pull/sync`). Offline workflow: pull before flight, work offline, push after landing.

**Reality cross-check:** All three are **PROPOSED — do not exist yet.** sgit remote add/list is already in the PROPOSED list (Section 17). The sg-wasm and sg-audio-transcription are new additions.

### Theme 5: One-Shot LLM Development Environment (Document 192)

A visual IDE for Web Component development using managed LLM context. Solves two problems: Claude artifacts cannot import external scripts; standard LLM history grows exponentially.

**Solution:** Explicit context blocks (project background / component library / current code / memory / task), one-shot execution (full context per call, no history growth), two modes (Build → produce code; Update Memory → prune/consolidate).

**Reality cross-check:** **PROPOSED — does not exist yet.** The one-shot LLM environment is distinct from Workspace v0.1.0.

### Theme 7: Folder as Narrative (Document 191)

Folder manifest (`_config.json`) for structured storytelling: title, description, start_file, sections with collapse states and ordering. Portfolio component (`sg-portfolio`) for artist/actor use cases. Per-file component override via manifest JSON.

**Important distinction:** This is separate from `_page.json` (rich component layouts). The folder manifest (`_config.json`) controls the folder navigation experience; `_page.json` controls the page rendering within a folder.

**Reality cross-check:** **PROPOSED — does not exist yet.** No `_config.json` manifest support has been code-verified. The `_page.json` renderer exists but the folder manifest is a separate concept.

### Theme 8: IFD Surgical Overrides (Document 197)

A methodology guide (v1.3.0) documenting the version-stamped filename convention for surgical overrides in repos using a build-time layering system. The double-dash separator (`sg-tree--v0.1.1.js`) prevents name collisions when IFD version folders are merged into a single `_latest/` directory.

**Reality cross-check:** This is a **methodology document** (documentation of a practice), not a feature. The convention is actively used in the sg-tree and sg-llm architecture briefs.
