## Thematic Grouping

### Theme A: Deploy Infrastructure — Local LLM Development Environment

**Document 52** describes a Docker Compose setup for offline LLM chat during travel. Key architecture decisions:

- **Ollama on macOS host** (no GPU passthrough in Docker), container proxies via `host.docker.internal`
- **FastAPI API layer** with 3 route groups: `/api/chat` (streaming proxy), `/api/models`, `/api/sessions` (CRUD)
- **sg-layout Web Component** bundled (not CDN) for offline use — fractal window management, drag-to-dock panels
- **Vanilla Web Components** following the same pattern as `sg_send_deploy__ui__admin` — no framework, no build step, no Shadow DOM
- **Persistent sessions** via volume-mounted JSON files
- **10 Ollama models** available (gemma3:4b default)
- **Relationship to EC2 brief** (`v0.1.5__ec2_llm_arch.md`): same `/api/chat` contract, different Ollama target

**Open questions directed at another LLM session:** sg-layout initialisation protocol, component reuse from SGraph-AI__Tools, design system tokens, versioning conventions.

**Note:** This component lives in `sg_send_deploy__local_llm/`, a separate package within the deploy repo. It is NOT part of SGraph Send's user-facing functionality.

### Theme B: Type_Safe Code Quality — SG_Send__Deploy

**Documents 53–54** form a paired code review + reference guide for the deploy codebase's Type_Safe compliance.

**Code review findings (Document 53):**
- **17 modules reviewed**, 11 are plain classes (no Type_Safe inheritance)
- **2 CRITICAL:** `__init__` overrides that bypass Type_Safe entirely (EC2 Instance schema, Security Group schema)
- **11 HIGH:** Plain classes across services, providers, fleet twin, audit trail
- **6 HIGH:** Untyped `list`/`dict` fields in Type_Safe classes
- **40+ MEDIUM:** Raw `str`/`int`/`float` where Safe_* types required
- **0 methods** use the `@type_safe` decorator (should be on all public methods)
- **11 repeated findings** (RF-1 through RF-11) defined as reusable patterns
- **5-phase remediation plan:** Phase 1 (correctness blockers) → Phase 2 (shared types) → Phase 3 (class migrations) → Phase 4 (primitive hardening) → Phase 5 (decorator coverage)

**Reference guide (Document 54):**
- Assembled from osbot-utils v3.63.4 documentation + project lessons
- **14 core rules** with code examples for each
- **Complete import reference** for all Type_Safe types
- **Testing patterns** (context manager, `.obj()`, `__SKIP__`, `setUpClass`)
- **10 lessons learned** from the SG/Send v0.2.41 audit
- **Full checklist** for schema classes, collection subclasses, service classes, and tests
- **Critical warnings:** wrong Safe_Str type causes silent data corruption (e.g., `Safe_Str__Id` strips `/` from MIME types)

**Significance:** These documents establish the Type_Safe standard for the entire deploy codebase. The reference guide should be read by any agent working on sg_send_deploy code. The code review's 5-phase plan is a concrete remediation roadmap.
