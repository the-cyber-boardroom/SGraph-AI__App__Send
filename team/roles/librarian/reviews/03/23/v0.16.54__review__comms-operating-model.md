## New Structure: `team/comms/`

A structured, in-repo communication channel has been established for cross-team agent communication. This is a **new operating model** that complements the existing role review and debrief system.

### Why It Exists

The QA agentic team clones this repo to run tests against the UI and API. When tests fail, they need to classify failures:
- **Good failure:** The feature was intentionally changed → update the test
- **Bad failure:** The feature wasn't changed → this is a regression, file a bug

The existing debrief/review system is human-facing. `team/comms/` is **agent-facing** — designed for automated consumption by downstream agentic teams.

### Folder Structure

```
team/comms/
  README.md                          # How the system works
  changelog/MM/DD/                   # What changed, version-tagged, with test impact
  qa/briefs/MM/DD/                   # Testing briefs for QA team
  qa/questions/MM/DD/                # Bidirectional Q&A
  plans/MM/DD/                       # Pre-implementation intent
```

### Relationship to Existing Systems

| System | Audience | Purpose |
|---|---|---|
| `team/roles/{role}/reviews/` | Human + Agent | Role-specific analysis (architect, dev, QA, etc.) |
| `team/humans/dinis_cruz/debriefs/` | Human (Dinis) | Session summaries, reading order, decisions |
| `team/comms/` | **Agent (downstream teams)** | Change communication, test impact, Q&A |
| `team/roles/librarian/reality/` | All | Code-verified feature inventory |

### File Naming Convention

Follows project standard: `{version}__{type}__{description}.md`
- `v0.16.54__changelog__qa-change-requests-cr001-cr004.md`
- `v0.16.54__qa-brief__token-counter-zero-state.md`
- `v0.16.54__plan__comms-structure-and-next-steps.md`

### Operating Rules

1. **Every code change that affects UI or API** should have a changelog entry
2. **Every changelog entry must classify expected test impact** (which tests should break vs shouldn't)
3. **QA briefs** go in `qa/briefs/` when the QA team needs specific testing instructions
4. **Questions** go in `qa/questions/` for bidirectional communication
5. **Plans** go in `plans/` before starting implementation work

### Recommended CLAUDE.md Addition

The `team/comms/` operating model should be added to `.claude/CLAUDE.md` as a standard procedure for all sessions that make code changes. Suggested addition to "Key Rules" section:
