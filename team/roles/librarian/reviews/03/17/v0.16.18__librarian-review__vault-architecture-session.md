## Executive Summary

The CLI team completed a major vault architecture redesign session (v0.16.3) that produced 10 documents covering self-describing file IDs, deterministic HMAC entry points, sub-tree model, batch read API, on-demand tree loading, and a four-layer security model. This review cross-references those documents against our codebase reality to identify what already exists, what is new, what issues are resolved, and what work remains for the SGraph-AI__App__Send repo.

**Key finding for our team:** Three of the four "high priority" backend changes proposed by this session already exist in our codebase. The backend is closer to ready than the session documents assume. The remaining work is primarily in the CLI (separate repo) and our browser UIs.

## 4. Issues Resolved by the New Architecture

### Issue: Browser 404s When Opening CLI-Created Vaults

**Root cause identified:** The vault browser's v0.1.2 crypto validates `vault_id` with `/^[0-9a-f]{8}$/` (hex only), but the CLI generates alphanumeric vault IDs (e.g., `12cpxeq9`). The regex rejects valid vault IDs, causing 404-like failures.

**Resolution:** Fix regex to `/^[a-z0-9]{8}$/` (per impl-plan__vault-browser-changes section 2.1).

### Issue: Dual-Write Hack (v0.8.5)

**Root cause identified:** The CLI had to write vault data in two formats -- one for its own ref system (random UUIDs in `bare/refs/`) and one for the browser (HMAC-derived IDs outside `bare/`). This was because the browser can't call `list` to discover random IDs, so it needs deterministic IDs it can compute from the vault key.

**Resolution:** Self-describing IDs with deterministic HMAC derivation for entry points (`ref-pid-muw-{HMAC[:12]}`, `idx-pid-muw-{HMAC[:12]}`). Both CLI and browser compute the same IDs from the vault key. One write per file. The dual-write code in the CLI is deleted entirely.

### Issue: Payload Format Incompatibility Between CLI, Vault Browser, and Vault UI

**Root cause identified:** Three incompatible crypto models:
- **CLI:** AES-256-GCM + PBKDF2 passphrase-derived keys, git-like object model
- **Vault browser v0.1.2:** Same crypto but with flat tree model and hex-only vault ID validation
- **Vault UI v0.1.7:** RSA-4096 hybrid encryption, IndexedDB key storage, flat model with no versioning

**Resolution:** All three clients converge on one data format:
- PBKDF2 key derivation (passphrase model)
- Git-like object model (refs -> commits -> trees -> blobs)
- Sub-tree model (one tree per directory)
- Self-describing IDs (`{type}-{derivation}-{mutability}-{hex_id}`)
- `bare/` prefixed server paths
- Vault key is `{passphrase}:{vault_id}`

### Issue: Flat Tree Performance

**Root cause identified:** Current implementation uses one flat tree listing all files with full relative paths. Every commit rebuilds the entire tree. Merge compares every entry -- O(all files).

**Resolution:** Sub-tree model (one tree per directory). Changed folders get new tree objects; unchanged folders are skipped entirely. Merge becomes O(changed folders), not O(all files). On-demand tree loading means the UI only fetches what the user opens.

### Issue: Plaintext Metadata Leaking to Server

**Root cause identified:** Live vault inspection revealed plaintext `path`, `name`, `size`, and `message` fields alongside encrypted versions. The server could read filenames and folder structure.

**Resolution:** Drop all plaintext metadata fields. Tree entries keep only `name_enc`, `size_enc`, `content_hash_enc`, `blob_id`, `tree_id`. Commits keep only `message_enc`. The server sees opaque encrypted blobs only.

## 5. Open Questions That Need Our Team's Input

### 5.1 Batch Read Auth Model (Backend Team)

The session notes state: "the current batch endpoint requires `write_key` in headers. Read-only batches should work without it."

**Our reality:** This is already solved. `Routes__Vault__Pointer.batch__vault_id()` detects read-only batches and calls `batch_read()` without auth. Mixed/write batches require auth. No change needed.

### 5.2 Caching Layer Priority (Backend Team)

Mutability-based caching (`imm` -> cache forever, `snw` -> short TTL, `muw` -> no cache) is marked as medium priority. Should we implement this before or after the CLI ships the new ID format? It has no value until clients send self-describing IDs.

**Recommendation:** Defer until CLI v0.16.3+ is deployed and producing self-describing IDs.

### 5.3 Vault UI Migration Timing (Frontend Team)

The vault UI migration (RSA -> passphrase) is the biggest user-facing change. Should this happen in a new IFD version (v0.1.8 or v0.2.x) or as an override to v0.1.7?

### 5.4 Test Vectors (Cross-Team)

A known vault key must produce identical self-describing IDs in both Python (CLI) and JavaScript (vault browser + vault UI). Who owns creating and validating these test vectors?

### 5.5 Zip Clone Endpoint (Backend Team)

`GET /api/vault/download/{vault_id}` for one-request full clone is proposed. This does not exist today and was already listed in the reality document as PROPOSED (v0.13.30). Should we build it now or defer?

### 5.6 No Backwards Compatibility

The session explicitly states: "No backwards compatibility. Existing vaults will not be readable." This means existing vaults created by the CLI need to be recreated. How do we communicate this to users?
