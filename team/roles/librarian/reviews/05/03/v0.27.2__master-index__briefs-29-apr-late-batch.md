| Feature | Priority (Dev assessment) | Summary |
|---------|--------------------------|---------|

### Theme 2: Firefox Browser Plugin (Doc 335)

Firefox with MITM proxy becomes the first real-world test of the plugin architecture from 28 Apr. Differs from Playwright in a fundamental way: **interactive vs programmatic** — a human is at the keyboard, not an automation script.

Key design decisions embedded in the brief:
- Firefox plugin lives at `plugins/firefox/`
- Credentials and MITM scripts stored in vault (per-session)
- Browser view rendered inline (iframe preferred; VNC/noVNC as fallback)
- AMI bake captures configured Firefox profiles — saves MITM scripts, bookmarks, extensions

**ALL PROPOSED — does not exist yet.** This is a plugin that runs inside the ephemeral infrastructure — it does not affect SGraph Send's core API or Lambda functions.

### Theme 3: SGit Four-Layer Refactoring (Doc 336)

The most architecturally significant brief in this batch. SGit's codebase must be restructured into 4 clean layers before PKI lands:

| Layer | Contents | Review Standard |
|-------|----------|----------------|
| **Crypto** (Layer 1) | PBKDF2, AES-256-GCM, hashing, PKI ops | Highest — any change = potential security vulnerability |
| **Core** (Layer 2) | Every state-changing action (commit, push, pull) | High — data integrity; typed steps + transaction log |
| **Network** (Layer 3) | All API calls to SG/Send server | Medium-high — API contract changes need coordination |
| **Plugins** (Layer 4) | Read-only operations, formatting, search, extensions | Normal — cannot break state |

**No upward dependencies** is the architectural invariant. Crypto depends on nothing. Core never calls Plugins.

**Transaction logging:** every Core action produces a type-safe JSON transaction record — enabling replay, monitoring, testing, and future rollback support.

**Step-based decomposition:** each CLI command decomposes into discrete typed steps with per-step timing. Performance bottlenecks become visible at step level without requiring premature optimisation.

**Feature flags for plugins:** `config.json` enables/disables individual plugins; disabled plugins do not register CLI commands.

**ALL PROPOSED — does not exist yet.** The existing SGit CLI works but does not use this four-layer structure. This is a refactoring brief, not a new feature brief.

Two new open decisions catalogued:
- **Decision #29:** SGit Crypto layer — when does it move to a separate repo? (After stabilisation)
- **Decision #30:** SGit transaction log storage — alongside vault data or separate log store?
