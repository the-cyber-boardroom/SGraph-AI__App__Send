## Thematic Grouping

### Theme A — Ephemeral Compute Platform Maturity (docs 337–346)

The April 30 batch represents the Ephemeral Compute platform reaching a well-defined architectural state. All nine briefs form a coherent system:

- **Naming** (doc 341): Node, Pod, Spec, Stack — clear taxonomy replaces ad-hoc naming
- **Runtime** (doc 343): Docker + Podman abstraction, control plane API
- **Spec structure** (doc 339): Every spec gets its own FastAPI + CLI via base classes
- **External specs** (doc 342): `sg-compute_spec_{name}` convention, GitHub repos, shallow clone
- **Network isolation** (doc 340): VPC per stack, automated lifecycle
- **S3 compatibility** (doc 338): Full boto3 transparency, call log methodology, four modes
- **First external spec** (doc 345): Cyber Boardroom + S3 Storage Node specs defined
- **Multi-agent collaboration** (doc 346): Email-FS working, vault content rendering proven

All Ephemeral Compute items are **PROPOSED — do not exist yet in the SGraph Send codebase.** This is a companion platform under active development.

### Theme B — Content Library and Agent Pipeline (docs 344, 346)

Vault-rendered content model proven (doc 346). Content library page types defined (doc 344). MVP scope: two agents (@content + @dev). Transition criteria specified before scaling agent count.

### Theme C — Infrastructure Reference (doc 347)

The EC2/Ollama GPU guide is a reference document, not a brief for the SGraph Send codebase. It documents how to set up a GPU node for LLM workloads. References `sp` CLI which is being superseded by `sg-compute`.

### Theme D — SG/Send Backend Performance (doc 348)

**This is the only brief in this batch that directly targets the SGraph Send (`sgraph-ai-app-send`) codebase.** Brief 08b specifies the Clone Pack feature — a binary pack format and two new API endpoints to eliminate the 202-second clone bottleneck (root cause: one HTTP request per BFS wave × ~600 objects). All within the existing zero-knowledge model.
