# Briefing: Individual File Upload & Collaboration Modes

**Date:** 10 April 2026 | **Role:** Librarian (synthesis) | **Purpose:** Single-document briefing on the evolution from monolithic zips to individual-file storage, and the four collaboration modes with graduated resilience

---

## Executive Summary

SGraph Send's storage model is evolving along **two intertwined axes**:

1. **Upload representation:** From monolithic zip blobs → individual encrypted files with directory manifests
2. **Collaboration semantics:** From read-only transfers → editable folders → fully versioned vaults

These are coupled: individual-file storage is the **precondition** for collaboration, incremental updates, progressive browsing, large-folder support, and video streaming. The architecture defines **four modes** (Mode 1 shipped; Modes 2–4 proposed) that progressively add collaboration features and resilience guarantees. Each mode is independently valuable — they are stepping stones, not a single monolithic plan.

---

## 1. The Performance Problem With Monolithic Zips

### The current state (Mode 1 — SHIPPED)

When a user uploads a folder today:
- The entire folder is zipped and encrypted as a **single blob**
- A 1 GB folder with 50 files becomes one 1 GB encrypted object on S3
- **To view any one file**, the recipient must download and decrypt the entire 1 GB
- **To add a file**, the recipient must download everything, re-zip, re-encrypt, re-upload
- **To stream a video inside the zip**, the full zip must be fetched first

### The cost at scale

| Approach | Download to view one file | Download to see file list | Add a file |
|---|---|---|---|
| **Current (single blob)** | Download entire 1 GB, decrypt entire 1 GB | Download everything | Re-zip + re-upload all |
| **Individual files** | Download 5 KB tree + just that file | Download 5 KB tree (instant) | Upload that file + new tree |

**Quotable:**
> "Instant browsing: download the tree structure (5 KB, instant). See all filenames, sizes, types. Browse the folder without downloading anything. On-demand download: click a file → download and decrypt just that file. A 200 KB HTML file from a 1 GB zip downloads in milliseconds, not minutes." — *v0.12.2 dev-brief__temp-token-and-individual-file-storage*

### The zero-knowledge constraint

The server cannot maintain a file listing — it never sees plaintext filenames. The solution: store the file listing as **another encrypted file**, at a **deterministic path** derived from the folder key via PBKDF2 (600k iterations). This is the **deterministic safe paths** primitive — same secret always produces the same ID, no coordination required, brute force infeasible.

## 2. The Four Collaboration Modes

| Mode | Storage | Collaboration | Edit | Version Control | Status |
|---|---|---|---|---|---|
| **1: Read-Only Zip** | Single encrypted blob | None | No | None | **SHIPPED** |
| **2: Read-Only Individual Files** | Per-file blobs + encrypted manifest | None — read-only | No | Manifest tracks files | **PROPOSED — P1** |
| **3: Editable with Versioned Manifest** | Per-file blobs + versioned manifest | Light: multiple uploads, last-write-wins | Yes | Simple version field, compare-and-swap | **PROPOSED — P2** |
| **4: Vault-Backed Folder** | Full vault (SGit-like) | Full: branches, merge, conflict resolution | Yes | Complete Git semantics | **PROPOSED — design only** |

### Mode 1 — Read-Only Zip (shipped)

- Upload folder as zip, encrypt as single blob, download and extract client-side
- **When to use:** Single-time deliveries, accountant packs, one-way sharing
- **Limitations:** Can't add files, can't browse without full download, can't stream

### Mode 2 — Read-Only Individual Files (proposed, P1)

The foundational unlock. Each file is stored separately; a **directory manifest** (encrypted JSON at a deterministic path) tracks the folder structure.

**Upload flow:**
1. Client iterates folder contents
2. Each file is encrypted individually, uploaded as a blob with a content-addressable ID
3. Manifest is built (JSON: version, files list with name/size/type/encrypted_id/IV/timestamp)
4. Manifest is encrypted and stored at the deterministic manifest path

**What Mode 2 alone enables:**
- Progressive loading (browse folder before all files finish uploading — **live upload sharing**)
- Large folder support (>100 MB practical in browser)
- One-file access in milliseconds (not minutes)
- Video playback streaming directly from individual file blobs
- On-demand thumbnail loading

**When to use:** Large folders where recipient wants specific files, photo galleries where thumbnails load incrementally.

### Mode 3 — Editable with Versioned Manifest (proposed, P2)

Mode 2 plus the ability to **add, remove, or replace files**. The manifest gets a version number; updates use **compare-and-swap** (client sends previous manifest version; server rejects if it has advanced).

**Why "light collaboration":**
> "For light collaboration (a few changes per day), versioned manifests with last-write-wins is acceptable." — *v0.20.33 editable-collaborative-folders*

**When to use:** Shared folder growing slowly — a group trip photo gallery, a team shared workspace with low concurrent editing, contributors adding files over days.

**Race condition handling:** For the photo gallery use case (a few changes per day), last-write-wins is acceptable. Include version field so conflicts are detectable but not blocking.

### Mode 4 — Vault-Backed Folder (proposed, design only)

The folder **is** a vault. Every file upload is a commit. The vault tree **is** the directory structure. Full Git-like semantics: branches, merging, conflict resolution.

**When to use:** Heavy collaboration, multiple simultaneous editors, full version history needed, rollback, blame, branching experiments.

**Status:** Design only — depends on vault primitives not yet built. Mode 3 serves most use cases without requiring this level.

## 3. Resilience Features (Across All Modes)

### Immutability + compare-and-swap

All blobs are **immutable** once written. Only the `head` pointer (to the current manifest / tree) is mutable. This gives:
- **No data loss on conflict** — all versions are retained
- **Cheap rollback** — every previous manifest is still on disk
- **Race condition detection** — client must provide previous head; server rejects if head moved
- **Tiny race window** — "two uploads at the exact same millisecond" only

```
folder-root/
  head                  # Mutable: points to current manifest/tree
  manifests/{id}        # Immutable manifest snapshots
  files/{content-id}    # Immutable encrypted file blobs
```

### Chunked upload with resume

From the robust upload/download brief (v0.17.3):
- Break files into chunks; upload in parallel (configurable thread count)
- Track which chunks succeeded; **resume from the last successful chunk** on failure
- Progress visibility: "45 of 120 chunks uploaded"
- Real-time speed measurement drives thread-count tuning (fast connection = more threads)
- Pause on connection dropout; auto-retry when connection returns
- Warn before tab close during in-progress upload

### Error recovery matrix

| Failure Mode | Current | Target Behaviour |
|---|---|---|
| Cloud drive file (not local) | "File not found" | Detect, explain: "Download it first, then upload." |
| Internet dropout mid-upload | Upload lost | Pause, auto-retry when connection returns |
| File exceeds size limit | Error | Detect before upload starts, show limit, suggest alternatives |
| Browser tab closed | Upload lost | Warn before closing, offer resume on return |
| Mobile: iCloud/Google Photos | Fails silently | Detect FileReader error, show: "Download to device first." |

### Content-addressable storage

File IDs are derived from content hashes. This enables:
- **Natural deduplication** — same file content = same blob ID, stored once
- **Delta updates** — new manifest references new blob + existing blobs; unchanged files are not re-uploaded
- **Content integrity** — blob ID mismatch detects tampering

### Live upload sharing

- Tree/manifest is uploaded first
- Files uploaded individually in parallel
- Share link becomes active **before all files finish uploading**
- Recipient sees the tree, sees which files are in progress, downloads completed files immediately
- Combined with chunked resume, a dropped sender connection does not interrupt the recipient's experience

## 4. Access Control — Read vs Write Tokens

Two token types gate folder operations:

| Token | Capabilities | Shareability |
|---|---|---|
| **Read token** | Download files, read manifest | Publicly shareable |
| **Write token** | Upload files, update manifest, update head | Restricted to collaborators |

Both are derived from the folder key using different salts. The server stores nothing about which token holder made which change — the model is still zero-knowledge.

**Publishing model (Architect's Option D, recommended):** A vault can be **published** to a simple-token URL via `POST /api/vault/publish/{vault_id}` with two modes:
- `"link"` — transfer resolves to current vault state at read time (always latest)
- `"snapshot"` — copies current vault state into transfer payload (frozen at publish time)

This cleanly separates **workspace (vault)** from **published artifact (transfer URL)**.

## 5. Storage Architecture (Mode 2/3)

```
POST /api/folder/create                         → returns folder_id, read_token, write_token
PUT  /api/folder/{id}/file                      → upload individual file, returns file_id
GET  /api/folder/{id}/file/{file_id}            → download individual file
PUT  /api/folder/{id}/manifest                  → upload new manifest (compare-and-swap on head)
GET  /api/folder/{id}/head                      → get current manifest pointer
GET  /api/folder/{id}/manifest/{manifest_id}    → get specific manifest snapshot
```

**Manifest shape (encrypted JSON):**
```json
{
  "version": 7,
  "files": [
    {
      "name": "beach-day.jpg",
      "size": 2_847_193,
      "type": "image/jpeg",
      "encrypted_id": "a1b2c3...",
      "iv": "...",
      "created": "2026-04-05T14:23:01Z",
      "contributor_hash": "anon-1"
    }
  ]
}
```

**Metadata files (for Mode 3+):** Votes, tags, descriptions live as additional encrypted JSON files referenced from the manifest.

## 6. The Photo Gallery as Proof-of-Concept

The use case that proves Mode 2 + Mode 3 deliver real collaborative value:

> "A group trip. Five people take photos. Everyone wants to upload their photos to a shared folder. Then the group curates: upvote the best ones, remove duplicates, pick the final set for printing or sharing." — *v0.20.33 editable-collaborative-folders*

**Workflow:**
1. **Upload phase** — Progressive; each photo appears as it completes (not all-or-nothing)
2. **Browse phase** — Grid view with thumbnails, filter by contributor, lightbox
3. **Curation phase** — Upvote buttons; votes stored as JSON metadata file; sort by vote count
4. **Export** — Selected photos as a new folder or zip
5. **Sharing** — Share read-only gallery link, or grant edit access to other contributors

**Voting metadata:**
```json
{
  "version": 1,
  "votes": {
    "{file-id}": { "count": 3, "voted_by": ["anon-1", "anon-2", "anon-3"] }
  }
}
```

## 7. Version Control as Invisible UX

A critical design principle: **non-technical users experience version control without knowing the term.**

> "For most users, version control is invisible. They do not need to know about commits, branches, or merges. They see: an 'undo' button (roll back last change), a 'history' button (see what changed and when), a 'restore' option on deleted files, a conflict message ('Someone else changed this file. Keep yours, keep theirs, or keep both?'). The version control terminology (commit, branch, merge) never appears in the UI." — *v0.20.33 editable-collaborative-folders*

The underlying machinery is there — every change is a commit, every contributor has an implicit branch, merges happen under the hood. The language in the UI is **Undo, History, Restore, Conflict** — consumer-friendly primitives that happen to map perfectly to Git semantics.

This is a key adoption lever: "introducing non-technical users to the power of version control without the intimidation."

## 9. The Underlying Primitive — Deterministic Safe Paths

All of the above depends on one foundational pattern:

```python
derive_deterministic_id(
    secret: str,           # vault key, access token, folder key
    salt: str,             # context-specific: "sg-folder-manifest", "sg-votes"
    iterations: 600_000,   # PBKDF2: brute-force-resistant
    length: 16             # hex output
) -> str                   # deterministic ID
```

**Properties:**
- **Deterministic** — same secret always produces same ID; no lookup needed
- **Server-blind** — server stores data at ID without knowing what ID represents
- **Enumeration-infeasible** — 600k PBKDF2 iterations × billions of token combinations
- **No coordination** — two parties with same secret independently compute same ID

**Uses across the system:** folder manifests, voting metadata, transfer grouping (`by-id-from__user`, `by-id-from__access-token`), vault object IDs, per-user settings.

## 10. Narrative Summary

The journey is:

**Monolithic zip blobs** (current, Mode 1) → **individual encrypted files with directory manifests** (Mode 2, P1) → **editable folders with versioned manifests and compare-and-swap conflict detection** (Mode 3, P2) → **optionally, full vaults for heavy collaboration** (Mode 4, design only).

Each mode is independently shippable and independently valuable. Mode 2 alone unlocks progressive browsing, large folders, video streaming, and deduplication. Mode 3 adds light collaboration — enough for the photo gallery use case. Mode 4 is reserved for workloads that genuinely need Git semantics.

**Upload and download components** (chunked, parallel, resumable, speed-measured) are being designed as reusable primitives that can work with any of the four modes, and are planned for distribution as standalone npm packages for ecosystem adoption.

**Version control stays invisible** to end users — Undo, History, Restore, Conflict — while the underlying machinery (immutable blobs, compare-and-swap on head, content-addressable IDs, deterministic manifest paths) provides the resilience guarantees that make collaboration safe.
