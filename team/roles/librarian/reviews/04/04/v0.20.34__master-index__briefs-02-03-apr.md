## Thematic Grouping

### Theme 1: Revenue Infrastructure (Documents 211, 213, 214, 216)

The business model crystallised around OpenRouter. Infographics cost sub-10 cents to generate; a 25% markup makes them commercially viable. The £5 credit experiment is a designed test: give 50 early users credits, watch whether they buy more when they run out.

**OpenRouter becomes the billing backbone:**
- Programmatic API key creation per user (£5 credit cap)
- Usage stats per key (metadata only — zero-knowledge safe)
- Admin console for provisioning, top-up, revoke
- Component suite: `sg-openrouter-usage`, `sg-openrouter-models`, `sg-openrouter-key-manager`, `sg-openrouter-logs`, `sg-openrouter-balance`

**Key open question from the briefs:** Does the OpenRouter generation history contain prompt/response content, or metadata only? This determines whether usage history can be stored in a user's vault under the zero-knowledge model.

### Theme 2: Per-User Vaults and Data Persistence (Documents 214, 219, 220, 221)

All three infrastructure briefs (03 Apr) converge on the same problem: every tool needs persistent, encrypted storage. The architectural answer is a three-layer stack:

```
Layer 3: Tools (infographic, chat, OpenRouter admin, Browse)
Layer 2: Browser VFS (createFile/readFile/listFolder — provider-agnostic)
Layer 1: SGit Vault Primitives (commit, push, pull, branch, encrypt)
Layer 0: Storage (Memory, IndexedDB, S3, local disk)
```

Each layer hides the one below it. Components at Layer 3 write to a file system and never know whether it is backed by IndexedDB or an encrypted vault on S3.

### Theme 3: Developer Tooling (Documents 212, 215, 220)

Three briefs focus on the development workflow itself:
- **Patch review component** (`sg-patch-review`): visual diffs from vaults, three rendering modes (patch-only, patch+context, patch+live-source), `@agent` comment detection
- **Document-driven analysis**: automated document→infographic pipeline; the `sg-document-input` component unifies how every tool accepts documents
- **SGit browser components**: 11 primitive components as debuggable Phase 1, composed into tools in Phase 2, made invisible in Phase 3

### Theme 4: Institutional Knowledge and Team Methodology (Document 217)

The Library website addresses a real operational need: role definitions, skills, and Claude guidance are scattered across one repo. The proposal is a separate static repo (`library.sgraph.ai`) cloneable into every new session — the "clone-on-start" pattern. This affects every agent session from the moment it ships.
