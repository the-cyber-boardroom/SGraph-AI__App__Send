## Thematic Grouping

### Theme 1: The OWASP London 2026 Production Pipeline (docs 271–273)

All three new human briefs document techniques tested and used during the OWASP London 2026 presentation production session. They represent **battle-tested workflows**, not speculation:

- **Doc 272 (image compression):** The numbers are real — 107-slide PDF went from 116MB to 14.8MB using the browser Canvas JPEG encoder. The `pdf-compressor.html` tool is referenced as already built (though no such file exists in the SGraph/Send repo — see PROPOSED section). The Python PIL pipeline for infographic generation is an immediate integration opportunity.
- **Doc 273 (LinkedIn authoring):** Also tested. The workflow is operational. The Python `build_article()` function and `img_to_b64()` helper are ready-to-use patterns. The four article patterns are derived from actual LinkedIn publications.
- **Doc 271 (video creation):** More speculative relative to the others — the MVP path is documented but requires new WASM dependencies (Kokoro.js 160MB model download, FFmpeg.wasm 30MB). This is the frontier item.

Together these three briefs define a **media production stack** that could become a set of tools on `tools.sgraph.ai`. All PROPOSED — none have SGraph code yet.

**Roles impacted:** Dev (lead for tools), Designer, Journalist (LinkedIn workflow directly relevant), Conductor (prioritisation)

### Theme 2: VFS Bridge Proves the BrowseDataSource Pattern (doc 274)

The inter-team comms brief is a handoff of **working code**, not a proposal. The VFS bridge solves a hard problem (sandboxed iframe with no base URL loading vault assets) using a proven pattern (postMessage protocol + prototype setter override). The `BrowseDataSource` interface (3 methods: `getTree()`, `getFileBytes()`, `getFileList()`) is the key architectural abstraction — any storage backend implementing these three methods works with the viewer.

This brief is the bridge between the Browse component (`send.sgraph.ai`) and the planned vault browser component from the Tools team. The timing is significant: the Tools team now has working reference code before they start building `sg-vault-browse`.

**Roles impacted:** Architect (interface design), Dev (Tools team), QA (VFS bridge needs postMessage security verification)

### Theme 3: Browser Media Tools Expand the Tools Ecosystem

Docs 271 and 272 together propose a browser-native media processing suite with no server costs:
- Video creation ($0/video, client-side only)
- Image/PDF compression (browser Canvas, no upload needed)

Both align with SGraph/Send's zero-knowledge, zero-server-side-processing principle. The image compression tool in particular has an immediate use case: infographic slides from the SGraph API (currently ~850KB PNG each) could be auto-compressed to 113KB JPEG at generation time.

**Business implication:** These tools make `tools.sgraph.ai` a destination for media professionals who need privacy-preserving processing. The LinkedIn article workflow (doc 273) demonstrates a repeatable content production process.
