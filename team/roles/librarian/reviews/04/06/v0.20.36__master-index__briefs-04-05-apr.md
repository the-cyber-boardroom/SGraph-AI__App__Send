## Thematic Grouping

### Theme 1: Fractal IFD for the Tools Site (Document 229)

Document 229 is from the Tools Explorer team — it arrives in SGraph Send's brief stream as an inter-team architecture document. It formalises the current per-component IFD approach and extends it to two new entity types:

**Three entity types, one DAG rule:**
- `COMPONENT` — a web component (JS + CSS + HTML), already per-component IFD
- `TOOL` — a standalone page composing components and/or other tools, **new per-tool IFD**
- `SITE` — a navigable website composing tools and components, **new per-site IFD**

The key rule is: no circular load-time imports. The dependency graph must be a DAG. Any entity can depend on any other entity provided no cycle is introduced. The distinction between load-time imports (cycle-checked) and runtime references (iframe loads, `fetch()` — no cycle risk) is formalised in the manifest `resolution` field.

**What this changes for the tools site:**
- The monolithic `tools/v0/v0.1/` (now frozen) is replaced by per-tool IFD chains: `tools/chat/v0/v0.1/v0.1.0/`, `tools/one-shot-chat/v0/v0.1/v0.1.0/`, etc.
- A new `sites/tools-sgraph-ai/v0/v0.2/v0.2.0/` layer holds `site.json` (the registry + navigation manifest), shared CSS, and grouped sub-pages (LLM Tools section → `tools.sgraph.ai/en-gb/llm/`).
- 14 tools enumerated in `site.json`: chat, one-shot-chat, infographic-gen, llm-dev (LLM group); ssh-keygen, file-hasher, file-encryptor, key-generator, vault-browser, image-tools, video-tools (Utilities); vault-pyodide, pyodide-repl, sg-send-cli (Experimental).
- A DAG verification script validates manifests and rejects cycles at build time.

**Relevance to SG/Send:** The fractal IFD model generalises to the Send UI. The same three-entity decomposition (Component → Tool → Site) applies to the user-facing and admin-facing Send UIs. The `send.sgraph.ai` site could become a first-class site entity in this model.

### Theme 2: URL Resolution for Grouped Storage (Document 230)

Document 230 is a direct follow-on to the 04/04 backend storage restructuring (doc 223). The grouped `by-id-from__*` hierarchy means the transfer ID alone no longer locates the full S3 path. The brief surveys four resolution options:

| Option | Mechanism | Pros | Cons |
|--------|-----------|------|------|
| A | Fallback search (bounded, 3-4 HEAD requests) | No URL changes, no metadata store | Slight latency in worst case |
| B | Numeric config in URL (`/t/2/transfer-id`) | Single lookup | URL changes slightly |
| C | Server-side path registry (DynamoDB/S3) | Instant resolution | Additional write on upload; sync needed |
| D | Derive path from request keys | No URL changes, no store | Only works if enough keys provided |

**Recommendation: A+D combined.** The server first derives what it can from the keys in the request (decryption key + optionally access token). For legacy flat-path transfers, it falls back to the old path. For edge cases, a bounded search (max 3–4 HEAD requests) finds the object. Simple three-word tokens continue to work unchanged. No URL format changes. No new metadata infrastructure.

The acceptance criteria set a latency target: under 50ms additional for the fallback path.

### Theme 3: Editable Collaborative Folders (Document 231)

Document 231 is the largest architectural document in this batch. It defines four progressive folder storage modes:

| Mode | Storage | Collaboration | Status |
|------|---------|---------------|--------|
| 1 | Zip (single object) | None — read-only | **EXISTS (current)** |
| 2 | Individual files + directory manifest | None — read-only | PROPOSED |
| 3 | Individual files + versioned manifest | Light (last-write-wins) | PROPOSED |
| 4 | Vault-backed folder | Full (Git-like) | PROPOSED (designed only) |

**The photo gallery use case** drives Mode 2 and 3: multiple contributors upload photos to a shared folder, browse as a grid, upvote/downvote photos, select a curated set, and export it as a new folder or zip. Votes are stored as a JSON metadata file alongside the photos.

**Version control as invisible UX** is a design principle: undo, history, restore — no Git terminology visible to the user.

**Directory manifest options** for Mode 2/3: store as another encrypted file (discoverable via deterministic ID from folder key), as a vault commit (vault tree IS the directory structure), or client-side with sync. The brief favours the deterministic path approach for Mode 2/3, naturally evolving to vault commits for Mode 4.

**Connection to existing architecture:** Mode 4 depends on the browser VFS + vault primitives from the 03 April briefs. Mode 2/3 can be built independently against the existing presigned upload infrastructure.

**Mobile upload bugs identified (two known issues):**
1. Apple Photos multi-file drop fails on iOS/macOS (DataTransfer API limitation)
2. iCloud / cloud-synced files that are not downloaded locally fail silently

Both require error detection + user guidance + help page.

**Share from inside:** A new sharing pattern — while browsing an editable folder, generate a new share link for a subfolder or the curated set. This is an explicit UX feature requested in the brief.
