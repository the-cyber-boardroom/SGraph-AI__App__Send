## Thematic Summary

### Theme: Vault Encryption Layer — Structure Key Activation

The 04/28 batch is a single architect review covering a significant design gap in the vault encryption layer. The `structure_key` is correctly **derived** (HKDF-SHA256 from `read_key`) but never actually **used** for encryption — all vault objects (structural metadata and file content alike) are currently encrypted with `read_key`. The feature is inert.

The review defines:
1. **What structure key access should reveal**: commit graph, directory topology, branch names, public PKI keys
2. **What it must NOT reveal**: filenames, sizes, content hashes, MIME types, commit messages, file content
3. **Backwards-compatible migration**: AES-GCM `InvalidTag` / Web Crypto `OperationError` as a clean fallback signal — no migration scripts, no flag days
4. **Implementation briefs for four teams**: API Team (no changes needed), CLI Team (primary implementor — `SGit-AI__CLI` repo), Vault Web Team (`sgraph_ai_app_send__ui__vault`), Vault Tools / Components Team

**Verdict from the review**: "The design intent is sound; the implementation is incomplete. This is a four-team change with no flag day required."
