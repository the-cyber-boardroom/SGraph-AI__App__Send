## Thematic Summary

### Theme 1: Email-FS v0.6 — Protocol Finalised at Specification Level

All four documents together constitute the **v0.6 Email-FS protocol release**. This is not incremental polish — v0.5 → v0.6 is a substantive revision driven by Dinis rulings on the session model.

**Key v0.5 → v0.6 changes:**

| Change | Impact |
|--------|--------|
| Sessions now explicitly = unit of agent work (bounded by brief) | Check-in vs session distinction now unambiguous |
| Three-state lifecycle: `active/` / `archived/` / `stale/` | `ls active/` = live answer to "who's in-flight?" |
| Sessions immutable once `end.json` written | Sealed audit artifacts; PKI signing will later enforce |
| Stale threshold is use-case-specific | Spec mandates protocol, not value |
| Stale end carries `ended_by` field | Audit trail for who declared abandonment |
| Resumption keeps same session-id | Session = logical brief, not physical runtime |

**Five Primitives (unchanged from v0.5):** SEND / DELIVER / READ / PROCESS / PURGE

**Core axioms:**
- A1: `.eml` is immutable once written
- A2: Every directory has a clearly defined writer-set; operations disjoint by filename
- A3: No broker — vault is the store, sgit is transport, delivery is recipient-driven

**18 conformance checks** (C1–C18) defined. Future `email-fs fsck` enforces them.

### Theme 2: Implementation Path — Staged

The spec explicitly defers implementation to stages post-manual-stress-test:
- **Stage 1 (current):** Manual use by agents/humans with `cat`/`mv`/`rm`/`sgit`
- **Stage 2:** `pip install email-fs` CLI — mirrors Issues-FS path
- **Stage 3:** Native vault-viewer rendering (inbox, threads, session timelines)
- **Stage 4:** MCP tool exposure for agent runtimes
- **Stage 5:** SMTP bridge via `sg-workmail` FastAPI (already specified in `v0.7.6__dev-brief__sg-workmail-fastapi-service.md`)

**IMPORTANT:** The simulation document (doc 332) notes that it was "actually executed against a scratch vault" — meaning the protocol was validated manually before the spec was published. This is a confidence signal for the design, not an assertion that Email-FS code exists in this repo.
