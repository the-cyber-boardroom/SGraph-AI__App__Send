## Thematic Grouping

### Thread 1: Task Components + Unified Shell (Docs 318, 319, 320)
The tools ecosystem gets its architecture. Seven LLM task components, each task-first with internal model selection. A unified shell wraps them with shared state (document context, cost accumulation, vault context). Eight reusable shared components emerge from the shell. Three standalone tools (image, secrets manager, credits) connect to the shell.

**Key architectural decision:** Tool 2 in doc 320 (localStorage secrets manager) is the immediate implementation; doc 317 (Google app:data) is the upgraded version once OAuth is confirmed cross-domain.

### Thread 2: Ephemeral Infrastructure (Docs 313, 314, 315, 321)
A reusable pattern applied to vaults, observability, and VNC sessions: spin up, use, destroy. Reduces attack surface to two known consumers. Anomaly detection becomes trivial. Cost model favours ephemeral for low-frequency use cases. The pattern is: AMI pre-baked → launch → load data from vault/S3 → work → export to vault → destroy.

**Key architectural decision:** Session manager service is new (not yet outlined in existing code). It orchestrates VNC container lifecycle and billing.

### Thread 3: Identity Resolved (Doc 317)
The identity research thread (opened 17 Apr) closes. Google Drive app:data is the answer for Google users. Client-side OAuth, no server-side code. Encrypted keychain stored in user's own Google Drive. The immediate path (doc 320, localStorage) does not require OAuth. The upgraded path (doc 317) requires confirming cross-domain OAuth client ID behaviour.

**Open decision created:** #22 — does one Google OAuth client ID work across multiple domains (sgraph.ai, send.sgraph.ai, tools.sgraph.ai)?

### Thread 4: Vault Use Cases (Docs 316, 321)
Evidence packs (news stories + risk acceptance decisions) demonstrate vault's value proposition concretely — real content, real tokens, live demos on sgraph.ai. Vault-driven CI makes the Ambassador agent operational — content changes without code changes.

### Thread 5: Villager Hardening (Doc 322)
Video capture reliability is a prerequisite for all video workflows. The Villager brief separates recording from UI, adds health monitoring, and investigates Web Worker to survive tab throttling. This is a Villager team deliverable, not Explorer.
