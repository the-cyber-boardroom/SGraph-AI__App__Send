## What You're Building

A profile page Web Component and a set of OpenRouter integration components. The user activates credits, sees their balance, and uses LLM tools — all from the browser, with the OpenRouter key stored locally and calls going directly to OpenRouter.

### Data Sources

| Data | Source | How to Get It |
|------|--------|---------------|
| Access token | localStorage `sg_access_token` | Direct read |
| OpenRouter key | localStorage `sg_openrouter_key` | Direct read (show masked) |
| Vault keys | localStorage `sg_vault_*` | Count entries |
| Encryption keys | localStorage `sg_enc_*` | Count entries |
| Credit balance | Server `GET /profile/{token}` | Fetch on page load |
| Credit expiry | Server `GET /profile/{token}` | Fetch on page load |
| Usage history | Server `GET /profile/usage` | Fetch on page load |
| Key ID | Server `GET /profile/{token}` | Fetch on page load |

### Activation Flow

When user clicks "Activate £5 Credits":

```
1. Check: does user have a device key pair in localStorage?
   → No: generate EC P-256 key pair (Web Crypto API)
         store private key in localStorage
         send public key to POST /profile/register-device

2. POST /profile/activate
   → Server creates OpenRouter key, encrypts with device public key
   → Returns: { encrypted_key, key_id, credit_balance, credit_expiry }

3. Browser decrypts encrypted_key using device private key
   → Uses ECDH + AES-256-GCM (Web Crypto API)
   → Store decrypted OpenRouter key in localStorage

4. Update profile display: balance, expiry, key status
```

### Security Rules

- **Never send encryption keys, vault keys, or OpenRouter key to the server.** These are read-only from localStorage and displayed locally.
- **Mask sensitive values.** Show `or-***-f8` not the full key. User can click to reveal.
- **The two-column layout is the trust mechanism.** It shows users exactly what stays local vs what the server knows.

A small badge shown across all LLM-powered pages:


Renders as: `£4.23` (green if > £1, amber if < £1, red if £0)

- Header bar (next to user icon) — always visible
- LLM tool pages — prominent before generation
- Profile page — detailed view

### Data
- Reads from server `GET /profile/{token}` on page load
- Caches in sessionStorage (5-minute TTL)
- Updates after each generation (re-fetch)

## 4. OpenRouter Direct Calls (Browser → OpenRouter)

**Critical architecture:** LLM calls go directly from the browser to OpenRouter. SG/Send is NOT a proxy.

```javascript
// The browser calls OpenRouter directly
const response = await fetch('https://openrouter.ai/api/v1/chat/completions', {
    method: 'POST',
    headers: {
        'Authorization': `Bearer ${localStorage.getItem('sg_openrouter_key')}`,
        'Content-Type': 'application/json',
        'HTTP-Referer': 'https://send.sgraph.ai',
        'X-Title': 'SG/Send'
    },
    body: JSON.stringify({
        model: selectedModel,
        messages: [{ role: 'user', content: prompt }]
    })
});
```

**Why this matters:**
- Zero-knowledge: SG/Send server never sees prompts or responses
- No proxy latency
- OpenRouter handles rate limiting, billing, model routing
- Browser CORS: OpenRouter allows browser-origin requests

## 5. Model Selector (`<sg-openrouter-models>`)

Fetches model list from OpenRouter (no auth needed):

```javascript
const models = await fetch('https://openrouter.ai/api/v1/models').then(r => r.json());
```

Displays as a filterable list:

| Filter | Options |
|--------|---------|
| Capability | Text, Vision, Code, Audio |
| Price range | Free, Budget (<£0.01), Standard, Premium |
| Provider | OpenAI, Anthropic, Google, Meta, Mistral |
| Context window | 4K, 32K, 128K, 200K+ |

Each model card shows: name, provider, pricing (input/output per 1M tokens), context window, capabilities.

**Cost calculator:** User enters approximate token count → shows cost per model.

## 6. BYOT Mode ("Bring Your Own Token")

For users with their own OpenRouter/OpenAI/Gemini keys:

```
┌──────────────────────────────────────────────┐
│ How would you like to pay for LLM calls?     │
│                                              │
│ (●) Use SG/Send credits (£4.23 remaining)   │
│ ( ) Use my own API key                       │
│     [Enter your OpenRouter key: _________ ]  │
│                                              │
│ Using your own key: no markup, no credits    │
│ consumed for LLM calls.                      │
└──────────────────────────────────────────────┘
```

When in BYOT mode:
- No cost tracking through SG/Send
- No credit deduction
- User responsible for their own OpenRouter billing
- SG/Send still charges for file/vault operations (just not LLM)

## 8. Key Lifecycle (What the UI Manages)

```
No key        → [Activate Credits] → Key provisioned (encrypted delivery)
                                     ↓
Key active    → Make LLM calls     → Balance decreases
                                     ↓
Key exhausted → [Buy More]         → New key provisioned
                or [BYOT]          → User provides own key
```

### localStorage Keys

| Key | Value | When Set |
|-----|-------|----------|
| `sg_access_token` | Access token string | On first use / token entry |
| `sg_openrouter_key` | Decrypted OpenRouter API key | On credit activation |
| `sg_device_private_key` | EC P-256 private key (JWK) | On first activation |
| `sg_device_public_key` | EC P-256 public key (JWK) | On first activation |
| `sg_credit_balance` | Cached balance (£) | On profile fetch |
| `sg_credit_expiry` | Epoch ms | On profile fetch |
