## What You're Building

A minimal user profile system and OpenRouter key provisioning pipeline. The goal: give 50 early access users £5 of LLM credits each and let them activate and use those credits through a profile page.

**This is the immediate blocker for user outreach.** Nothing fancy — just the plumbing.

---

## 1. The Outreach Flow (End-to-End)

```
Human contacts ~50 users
  → "You have £5 credit. Go to your profile page and activate."
  → User opens profile page
  → Clicks "Activate credits"
  → Server creates OpenRouter API key with £5 cap
  → Server encrypts key with user's device public key
  → Browser decrypts, stores in localStorage
  → User makes LLM calls directly to OpenRouter (browser → OpenRouter, NOT through SG/Send)
  → Credits deplete → user prompted to buy more
```

## 2. New Endpoints Required

### User-Facing (User Lambda)

| Method | Path | What It Does | Auth |
|--------|------|-------------|------|
| GET | `/profile/{access_token}` | Return non-sensitive profile: credit balance, expiration, usage summary, key ID | Access token |
| POST | `/profile/activate` | Activate £5 credits for user, provision OpenRouter key, return encrypted key | Access token + device public key |
| POST | `/profile/register-device` | Register device EC public key against access token | Access token |
| GET | `/profile/usage` | Return usage records (token consumption history) | Access token |

### Admin-Facing (Admin Lambda)

| Method | Path | What It Does | Auth |
|--------|------|-------------|------|
| POST | `/openrouter/keys/create` | Create OpenRouter API key with credit limit | Admin auth |
| GET | `/openrouter/keys/list` | List all provisioned keys + usage | Admin auth |
| POST | `/openrouter/keys/topup/{key_id}` | Add credits to existing key | Admin auth |
| POST | `/openrouter/keys/revoke/{key_id}` | Disable a key | Admin auth |
| GET | `/openrouter/keys/usage/{key_id}` | Per-key usage stats from OpenRouter API | Admin auth |
| GET | `/openrouter/keys/usage-all` | Aggregate usage across all keys | Admin auth |

## 3. Data Model

### Server-Side Storage (per access token)

```python
class User_Profile(Type_Safe):
    access_token  : str              # User identity
    credit_balance: float = 0.0      # Remaining credits (GBP)
    credit_expiry : int   = 0        # Epoch ms (~1 month from activation)
    openrouter_key_id: str = ''      # OpenRouter key ID (NOT the key itself)
    device_keys   : list  = []       # List of registered device public keys
    usage_records : list  = []       # Token consumption history
    activated     : bool  = False    # Whether credits have been activated
    created_at    : int   = 0
    updated_at    : int   = 0
```

### What the Server NEVER Stores

| Data | Where It Lives | Why |
|------|---------------|-----|
| OpenRouter API key | User's browser (localStorage) | Zero-knowledge: server encrypts and forgets |
| Encryption keys | User's browser (localStorage) | Zero-knowledge: server never sees |
| Vault keys | User's browser (localStorage) | Zero-knowledge: server never sees |
| Device private key | User's browser (localStorage/memory) | PKI: only user's device can decrypt |

## 4. OpenRouter API Integration

### Research Required First

These questions MUST be answered before implementation:

| # | Question | Why It Matters |
|---|----------|---------------|
| 1 | Can we create API keys programmatically? | Without this, the entire flow is manual |
| 2 | Can we set per-key credit/spending limits? | £5 cap per user requires this |
| 3 | Can we scope keys to specific models? | Cost control |
| 4 | Can we query per-key usage in real time? | Profile page usage display |
| 5 | Does usage history include prompt content or just metadata? | If content, we CANNOT store it (zero-knowledge violation) |
| 6 | Can we use one master account to provision keys for all users? | Operational simplicity |
| 7 | What are the rate limits on key creation? | Bulk provisioning for 50 users |

**If OpenRouter does NOT support programmatic key creation with caps:** Fall back to a proxy model — SG/Send's master key proxies requests with per-user rate limiting server-side. This is less elegant but functional.

### OpenRouter API Endpoints to Map

```
GET  https://openrouter.ai/api/v1/models          # No auth needed — model list
GET  https://openrouter.ai/api/v1/auth/key         # Key info + credits remaining
POST https://openrouter.ai/api/v1/keys             # Create key (if supported)
GET  https://openrouter.ai/api/v1/generation        # Generation history
```

## 5. PKI Key Exchange Flow (Crypto Detail)

This is how the OpenRouter key gets from server to browser securely:

```
1. Browser generates EC key pair (P-256)
   → Private key stays in browser (localStorage)
   → Public key sent to server via POST /profile/register-device

2. Server creates OpenRouter key via API
   → Server now has the plaintext OpenRouter key

3. Server encrypts OpenRouter key with user's EC public key
   → Uses ECDH + AES-256-GCM (standard Web Crypto pattern)
   → Encrypted blob returned to browser

4. Browser decrypts with private key
   → Plaintext OpenRouter key now in memory
   → Stored in localStorage for subsequent requests

5. Server forgets the plaintext key
   → Only stores the key_id (reference, not the secret)
```

**Why this matters:** Even if the server is compromised AFTER provisioning, the attacker gets key_ids, not actual API keys. The keys only exist in plaintext in the user's browser.

## 6. Credit System

| Field | Value |
|-------|-------|
| Initial credit | £5.00 |
| Expiration | ~1 month from activation |
| Tracking | Server-side, per access token |
| Activation | User-initiated (POST /profile/activate) |
| Depletion | OpenRouter tracks actual usage; server syncs periodically |
| Top-up | Admin-initiated (POST /openrouter/keys/topup) or user self-service (future) |

### Credit Sync

The server periodically polls OpenRouter for per-key usage and updates the local credit balance. This avoids needing a webhook. Polling frequency: every 5 minutes or on-demand when user loads profile.

---

## 7. Security Constraints

1. **GET /profile/{access_token} must return ONLY non-sensitive data.** If someone knows an access token, they see: balance, expiry, usage stats, key_id. NOT: encryption keys, API keys, vault keys.
2. **OpenRouter key never stored in plaintext on server.** Encrypt-then-forget.
3. **Device public keys are not secret** (they're public keys) but should be validated (well-formed EC P-256).
4. **Usage history must be metadata-only.** If OpenRouter returns prompt content, strip it before storing.
5. **Credit balance is approximate.** OpenRouter is the source of truth; server balance is a cache.
