## Thematic Grouping

### Theme 1: The sgit Share Reliability Gap

Doc 275 documents a hard, reproducible failure in the `sgit share` command when operating from an egress-controlled environment (Claude Web / claude.ai). Key facts:

- **What works:** `sgit push` (individual delta object uploads), `sgit pull`, `sgit clone`, `sgit commit`, all vault read/write operations
- **What fails:** `sgit share` — which packages the vault as a single large encrypted blob and uploads via S3 presigned multipart PUT
- **Error signature:** HTTP 503, response body: `DNS cache overflow` (not a standard AWS S3 error — generated by an intermediary)
- **Payload size at failure:** 5,242,880 bytes (5MB part 1 of multipart upload)
- **Works with small vaults:** An earlier share token (`file-hour-6653`) was created when the vault was ~38KB — strongly suggesting payload-size is the discriminator
- **Related failure:** Batch upload during `sgit push` also gets SSL EOF, but falls back gracefully to individual uploads. `sgit share` has no such fallback.

**Architectural implication:** The `sgit share` command is structurally different from `sgit push` — it creates a monolithic snapshot blob rather than delta-uploading individual content-addressed objects. This architectural difference is precisely what makes it fail in environments with egress size limits, while `sgit push` succeeds.

**Roles impacted:** Dev (fix: add fallback path in `sgit share`), Architect (assess blob-vs-delta architectural tension), DevOps (egress allowlist investigation), AppSec (proxy interception of presigned URLs has security implications)
