## Thematic Grouping

### Theme 1: One-Shot Feedback Loops and Managed Context (Documents 204, 205)

The architectural centrepiece of the March 30 batch. The core insight: LLM context should not grow linearly across cycles. Instead of appending every message to history, each cycle extracts findings, prunes stale exploration, and reconstructs a compact, focused bundle.

**The One-Shot Feedback Loop:**
- Cycle N: send bundle → get response → extract facts + decisions → prune exploration → update bundle
- Context stays roughly constant (500–700 tokens vs linear growth to 15,000+)
- SGit branches manage bundle versions: main = confirmed path, experiment-x = exploration, merge or abandon

**Key innovations in Doc 205:**
1. **SGit-branched bundles:** Each bundle (system prompt + context + findings + code + task) is stored as vault files. SGit branches capture explorations. The entire conversation history becomes a diffable commit graph.
2. **Two-LLM pattern:** Primary (Claude, GPT-4) for quality answers; secondary (smaller, possibly local) for the pruning step. Keeps feedback loop cost low.
3. **Token Visualiser component:** Visual breakdown of bundle size by section (system prompt, context, history, reference, images, tool definitions), growth over time, cost estimate per cycle.
4. **Eval tests per bundle:** Deterministic tests validate that pruning preserved essential context.

**Reality status:** The entire one-shot feedback loop system is **PROPOSED — does not exist yet.** The bundle concept is directionally proven by workspace v0.1.1 but the loop infrastructure, token visualiser, and SGit bundle management are all new.

### Theme 2: Offline-First as Deliberate Practice (Document 206)

A plane session proved that offline work is not a limitation but a feature. The brief formalises this as an engineered practice.

**Two innovations:**
1. **Engineered offline time:** Block 2+ hours with no internet. Ollama for LLMs, cloned vaults for data, browser-based tools for execution. The scheduled Librarian and QA agents run online while the human works offline; results wait on return.
2. **`@agent` code comments:** New asynchronous communication channel. While coding offline, leave comments like `# @librarian: this duplicates logic in upload_engine.py`. On next online session, the Librarian (or a scheduled task) reads the codebase, captures comments, routes them to roles, creates issues/tasks, and removes the comment (leaving a reference ID).

**Who processes `@agent` comments?** The brief proposes extending the Librarian role (daily scheduled run), not creating a new role.

**Reality status:** All components of this system are **PROPOSED — does not exist yet.** The Ollama LLM chat tool exists (as a result of the 10-part brief mentioned in Doc 208). The vault clone workflow exists. The `@agent` comment processing, scheduled routing, and code stewardship tasks are all new.

### Theme 3: LLM Chat Tool — Villager-Phase Improvements (Documents 204, 208)

The LLM chat tool (built in Explorer phase from a 10-part brief) is working well enough to use offline on a plane with Ollama. Doc 208 takes it to Villager-quality via architectural discipline and targeted bug fixes.

**Architectural principle:** One Component, One Thing. Shadow DOM enforces isolation. SG Layout wires components together. No component duplicates another's job.

**Component decomposition:**
- Model Chooser — standalone traffic-driving tool (see market opportunity below)
- Chat Window — conversation display only
- Input Area — text entry only
- Attachment Manager — file/image/paste handling (standalone Web Component)
- Response Renderer — markdown only
- Settings Panel — temperature, system prompt

**Model Chooser as market opportunity:** OpenRouter has hundreds of models with poor discovery UIs. A well-built Model Chooser (categories, cost comparison, personal history, localStorage persistence) becomes a tool people visit the website for. Publish model comparison data as a shareable vault.

**Bug fixes (5 identified):**
1. Cannot type while response is streaming (input blocked)
2. Auto-scroll ignores user scroll position (should lock when user scrolls up)
3. Markdown rendering breaks during streaming (unclosed tags cause visual chaos)
4. Image paste from clipboard not working
5. Some LLM components not using SG Layout

**Reality status:** The LLM chat tool EXISTS (from prior Explorer work). The architectural refactoring, Model Chooser as standalone component, Attachment Manager as Web Component, and all 5 bug fixes are **PROPOSED — does not exist yet.**

### Theme 4: Multi-Agent Chat Mode (Document 209)

Talk to multiple agents simultaneously. Pick agents, arrange in SG Layout panels, send the same question to three agents, have a fourth consolidate. This is the social layer on top of the one-shot LLM component family.

**UI Architecture:**
- **Agent Picker (sidebar):** Lists all agents organised by team (Explorer, Villager, Town Planner, Support, Custom). Each entry: name, role, avatar (image model), status.
- **Conversation Panels:** Each agent = one SG Layout panel. Side-by-side, stacked, tabbed, or any SG Layout arrangement.
- **Multi-Ask mode:** Same question → multiple agents simultaneously → independent responses.
- **Debate mode:** Share Agent A's response with Agents B and C for critique. Consolidator synthesises.

**Sync vs Async:**
- Sync: human ↔ agent, real-time
- Async: define task, agent runs in background (minutes), human checks results later. Critical for multi-agent workflows, code reviews, semantic graph generation.

**Cost note:** Async on local models (Ollama) is essentially free. Async on API models shows estimated cost before running.

**Agent context source:** Librarian maintains context pages (role definition, team definition, latest status). Published on website, fetched at conversation start. Updating an agent's context = update the page.

**Reality status:** All of multi-agent chat is **PROPOSED — does not exist yet.** SG Layout exists. Multi-agent coordination, Agent Picker, Debate mode, Consolidator, async task mode are all new.

### Theme 5: SGit Visualisation — `sg-git-graph` (Documents 204, 210)

The visual layer that makes SGit branch structures comprehensible. Without it, branches are invisible (CLI-only). With it, the entire shape of work — LLM feedback loop cycles, multi-agent conversations, IFD version trees, dev-to-QA workflows — becomes navigable.

**Component: `sg-git-graph`**
- Renders nodes (commits), lines (branches), merges, abandoned branches
- Interactive: click commit for details, hover for diff preview, zoom/pan, time slider, filter by branch/author
- Three data sources: live SGit vault (structure key), JSON export (static), programmatic API (for LLM thread visualisation)

**The dual-use insight:** The same component renders both real SGit commit graphs AND LLM conversation branch graphs. An LLM interaction is just a commit. An agent conversation is a branch. A consolidation is a merge. The data model is identical.

**Programmatic API:**
```javascript
const graph = new SGitGraph();
const main = graph.branch('main');
// ... add commits, branches, merges
graph.render(document.getElementById('container'));
```

**Implementation:** Use an existing graph library (D3.js or lighter) for rendering. The component handles data model and interaction. Build at tools.sgraph.ai as IFD-versioned component.

**Reality status:** `sg-git-graph` is **PROPOSED — does not exist yet.** No graph visualisation exists for SGit vaults.
