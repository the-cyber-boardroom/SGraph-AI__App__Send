## Thematic Summary

### Theme 1: Observability Pipeline — Architecture Finalised

The 04/22 arch brief (doc 326) completes the observability system design. The 04/21 brief (doc 315) established ephemeral Elastic; this brief defines the full data pipeline that feeds it.

Six stages, each idempotent, each saving output, each re-runnable independently:

| Stage | Name | Output Storage |
|-------|------|---------------|
| 0 | Raw Capture | `s3://sg-observability-raw/` |
| 1 | Security Cleaning | `s3://sg-observability-clean/` |
| 2 | Structured Extraction | `s3://sg-observability-structured/` (NDJSON) |
| 3 | Signal Reduction | `s3://sg-observability-signals/` (compact JSON) |
| 4 | Elastic Projection | Elastic index (ephemeral, re-creatable) |
| 5 | Kibana Dashboards | Pre-baked in AMI |
| 6 | Screenshot Capture | Vault (via Playwright) |

Key design principle: **Elastic is never the source of truth.** It can be rebuilt from S3/vault at any time.

### Theme 2: LETS Pattern Named + Refactoring Needed

The dev brief (doc 327) names the LETS pattern (Load, Extract, Transform, Save) as the reusable abstraction for all data processing. It also confirms what's working (80s boot, CloudFront connector) and what needs extracting before scale:

- Index management API (create/import/list/delete/consolidate)
- Dashboard management API (create/list/screenshot/export/import)
- Connector API with generic pattern (CloudFront already, Playwright/SG/Send next)

URL segmentation is a key insight: break `/api/v1/vault/status` into `["api", "v1", "vault", "status"]` as an Elastic array field, enabling plain-text search for components.

### Theme 3: WhatsApp Backup — SG/Send as a Platform Play

The WhatsApp brief (doc 328) documents a real incident (Android-to-iPhone migration, phone died, all message history lost, no recovery path). It frames this as evidence for the "primitives for others to build on" philosophy:

- SG/Send provides: client-side encryption, versioned vaults, cross-platform browser access
- Integrator provides: WhatsApp/messaging-platform connector layer
- Revenue: vault storage fees (SG/Send) + subscription (integrator)
- Framing: "I do not want to build this, but somebody should build it" — reference implementation + open source

This is a **Town Planner / Alchemist** brief: it targets investors and business development rather than engineering teams.
