## Thematic Grouping

### Theme 1: Local LLM Backend Selection (Documents 232 + 233)

Documents 232 and 233 are companion documents providing practical guidance for choosing a local LLM inference runtime for the SGraph Send agentic component suite.

**The choice as of April 3, 2026:**

| Runtime | Protocol | Gemma 4 E4B | Context | Audio/Video | HTTP Server | Verdict |
|---------|----------|-------------|---------|-------------|-------------|---------|
| `mlx-openai-server` | OpenAI-compatible | ✅ | 128K | ✅ (multimodal) | ✅ localhost:8000 | **Use now** |
| `mlx_vlm.server` | OpenAI-compatible | ✅ | 128K | ✅ | ✅ (lighter, built-in CORS) | Good alternative |
| LiteRT-LM | Proprietary Python SDK | ❌ (E2B only) | 4K cap | ❌ unclear | ❌ no HTTP server | **Not yet viable** |
| Ollama | Ollama + OpenAI compat | ✅ text/image | 128K | ❌ | ✅ localhost:11434 | Audio blocked |

**Integration path for existing agentic apps:** Since `mlx-openai-server` speaks the OpenAI wire protocol, the existing OpenAI provider in the agentic app can be reused with a base URL change (`http://localhost:8000/v1`) and a model name change. No new serialisation logic is needed.

**CORS note for browser-based apps:** When the browser app runs on a different port, the server may need `--host 0.0.0.0` or a proxy. The lighter `mlx_vlm.server` has built-in CORS support and is a viable alternative if CORS causes friction.

**LiteRT-LM watch list** (not viable today but may become viable):
- Gemma 4 E4B in `.litertlm` format (not yet available)
- Python SDK reaching "Stable" (currently "In Dev")
- Context window beyond 4096 tokens
- Built-in HTTP server mode (not announced)

If Google ships an HTTP server for LiteRT-LM, it becomes the preferred choice for a fully Google-supply-chain setup. Until then, MLX.

### Theme 2: Ollama Audio/Video Limitation — Confirmed Blocker (Document 234)

Document 234 resolves the open question about Ollama audio/video support. The finding is unambiguous: **there is no mechanism to send audio or video bytes to Ollama today**, regardless of which API surface is used.

**What this means for `_buildOllamaBody`:**

The function (which constructs request bodies for the Ollama backend) must NOT attempt to pass audio or video content. When audio/video is present in a request destined for Ollama, the correct behaviour is:

1. Detect audio/video content parts
2. Return a clear error to the caller: "Ollama does not support audio/video input. Use mlx-openai-server or OpenAI API instead."
3. Do NOT silently drop the content parts (that would be a confusing non-error failure)

A TODO comment referencing `ollama/ollama#11798` should be placed at the stub location so it is easy to flip on when upstream lands the `audio` field.

**Audio/video routing implication:** The agentic component suite proposed in doc 224 (Section 21 of reality doc) includes a Model Compatibility Matrix. That matrix should explicitly capture Ollama's audio/video limitation — so the UI can route audio/video requests to a compatible backend automatically.
