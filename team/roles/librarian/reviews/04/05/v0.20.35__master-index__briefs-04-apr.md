## Thematic Grouping

### Theme 1: Backend Storage Architecture (Documents 222, 223, 225)

The backend S3 structure moves from a flat sea of transfer IDs to a typed, hierarchical grouping. The key innovation is the `by-id-from__*` prefix pattern: each grouping level is a static type folder followed by a derived ID folder at the same depth. The path self-documents how data is organised.

**The hierarchy:**
```
transfers/
  {server-provider-region-id}/
  by-id-from__user/{user-derived-id}/
  by-id-from__access-token/{access-token-derived-id}/
  by-id-from__decryption-key/{decryption-key-derived-id}/
  {transfer-id}/
```

Every level is optional; the server calculates defaults when callers don't provide values. This means the simplest upload still gets fully grouped. No client changes required — clients only know transfer IDs.

**What this enables:**
- Per-user cleanup by deleting the access-token-derived folder
- Cost attribution by counting objects per group ID
- Operational visibility: browsing S3 reveals logical groupings
- Vault alignment: decryption-key-derived groups naturally align with vault boundaries

**Security note from the brief:** The grouping is an organisational convenience, not an access control layer. The encryption is unaffected. Seeing a group ID does not reveal the access token or decryption key.

### Theme 2: Deterministic-Safe-Paths as a Platform Primitive (Documents 222, 225)

This is the most strategically significant document in the batch. The pattern was already used in SGit vault object IDs and backend storage grouping. Document 225 elevates it to a named, documented architecture primitive with:
- A shared derivation function (PBKDF2, 600,000 iterations, context-specific salt)
- A salt registry (prevent cross-context collisions)
- An implementation checklist (6 steps)
- Three new application domains proposed: per-user settings, shared secrets communication channels, encrypted search indexes

The pattern is explicitly described as already in use. Formalising it means future features can be built against the pattern with predictable security properties, without each team independently reinventing the approach.

### Theme 3: Agentic Component Suite (Documents 222, 224)

Six components form a complete browser-native agentic loop:

| Component | Role |
|-----------|------|
| `sg-tool-definition` | Visual editor for JSON tool schemas; template library; validation |
| `sg-json-sender` | Structured JSON construction; schema-aware input forms; token count preview |
| `sg-json-receiver` | Auto-detect text/tool_call; JSON tree viewer; diff view; schema validation |
| `sg-tool-runner` | Tool registry + execution; 8 built-in tools (vault search, read, write, run_code, etc.) |
| `sg-sandbox` | JavaScript (sandboxed iframe + Web Worker) + Python (Pyodide WASM); capability opt-in model; zero capabilities by default |
| `sg-agentic-loop` | Full orchestration; max iterations; cost budget; human-in-the-loop gate; trace visualisation |

**Gemma 4 is the primary target.** Released 2 April 2026 under Apache 2.0, it is the first open model with native, trained-in tool calling via special tokens (`<|tool_call>`, `<|tool_response>`). The brief documents the full chat history format and two tool definition methods (JSON schema preferred). The components should be built Gemma-4-first, with broad model compatibility tested and published.

**The sandbox security model** is the most sensitive new component: zero capabilities default, opt-in per tool, no direct VFS access from inside the sandbox, results returned to parent which decides what to save.

### Theme 4: Gemma 4 Local Infrastructure (Documents 226, 227, 228)

Three technical reference documents address running Gemma 4 in an offline-capable local environment:
- **MLX stack** (`mlx-openai-server`) is the practical choice today for desktop use. OpenAI-compatible API, full 128K context, audio/video support on E2B/E4B, works with existing provider code by changing base URL only.
- **LiteRT-LM** (Google official) is the right long-term play for mobile/IoT but is not ready for desktop API server use today (no HTTP server, 4K context limit, E2B only, Python SDK is day-1).
- **Ollama** has no audio/video API surface. The `images[]` field is the only binary data path. Components using `_buildOllamaBody` should return a clear error for audio content and reference upstream issue #11798.

These documents are **technical knowledge, not proposed features**. They inform the agentic components design (doc 224) and provide setup guidance for the development team.
