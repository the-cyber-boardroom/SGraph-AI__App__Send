## Thematic Grouping

### Theme 1: Auth MVP — Revised Architecture (Doc 297)

The original auth brief (doc 291, 04/18) proposed Cognito + Firebase with four Web Components. The subsequent architect review cycle traced through four positions and arrived at a materially simpler design:

- **Drops Cognito, Firebase, master password, KDF, and all server-side user records entirely.**
- **Vault key as credential** — existing `secret:id` format fits in every password manager.
- **Browser password manager** (Credential Management API) as the sync fabric.
- **`<sg-vault-picker>`** — one component replacing the four in the original brief (~150 lines of vanilla JS).
- **Google Identity Services** demoted to optional vault personalisation (a JSON claim stored inside the vault, not an auth mechanism).

This is a significant architectural win: stronger zero-knowledge guarantee, smaller dependency surface, no new server-side code, and no user records on the server by construction.

**Open decisions resolved:** #1 (profile page + credit activation — now "build on `<sg-vault-picker>`"), #10 (identity provider — Cognito dropped entirely), #13 (vault key storage option — browser password manager wins).

**6 questions raised for AppSec + Dev:**
1. Credential Management API coverage (`PasswordCredential` Chromium-only; Safari/FF use form fallback)
2. Shared vault auto-save to password manager on paste
3. Vault name leakage to password manager vendor
4. Multiple accounts sharing a browser profile
5. Recovery UX — MVP does not mandate recovery code display
6. `sg1.` version prefix on stored credential string

**Roles impacted:** Dev (lead: `<sg-vault-picker>` implementation, 4 IFD stages), AppSec (threat-model review requested), Architect (decision recorded — see doc 297), Sherpa (recovery UX design)

### Theme 3: EC2/ECR Deployment — Patterns From Playwright Service (Doc 298)

An inter-team handoff document from the Playwright dev team providing a copy-paste adaptation guide for sg-send's EC2 deployment. Written as a briefing for an agent implementing the sg-send provisioner.

**The key insight:** most of the work is already done. The `sgraph_ai_app_send__docker/provision_ec2.py` committed to `dev` (as found in today's merge) is adapted directly from the Playwright pattern described in this document. The merge confirms doc 298 was acted on.

**Critical patterns documented:**
- AL2023: install `docker` alone, then curl Compose binary separately (dnf fails if combined)
- ECR token: `set +x` before login, `docker logout` + `rm -f ~/.docker/config.json` immediately after pull
- Tag-based metadata store (`sg:service` as immutable filter key, not `Name`)
- SSM over SSH (no port 22, no key pair)
- Health check accepts 401 (auth is working = service is up)
- Auto-select when exactly one instance exists (zero-argument workflow)

**Roles impacted:** DevOps (lead: ECR repository creation, IAM policy for CI, GitHub Actions `docker-build.yml`), Dev (sg-send-ec2 CLI is written; DevOps needs to provision the ECR repo itself)
