## Thematic Grouping

### Theme 1: "Folders with Version Control" — From Architecture to User Experience (docs 281–283, 285)

The vault-as-collaboration-primitive decision from Day 35 (doc 278) now has concrete UI and messaging deliverables.

**Technical side (doc 283):** The main SG/Send upload UI gets a vault-backed mode as a beta toggle. The experience: drop a folder → share link available immediately → files commit in batches → recipient sees files appearing in real time. Five key features: instant vault creation, batched commits, upload status file (`.sg-upload-status.json`), placeholder files for queued large uploads, and persistent upload progress UI. This is the first user-visible feature from the vault convergence.

**Messaging side (doc 285):** "Distributed, encrypted folders with version control" is the one-line description. The brief provides a complete vocabulary mapping (vault→folder, etc.), competitive positioning (SG/Send occupies the intersection Dropbox, Git, and WeTransfer each fail to reach), and explicit guidance on what NOT to say in user-facing copy.

**Proof point (doc 285):** Four simultaneous clients (Claude Code, Perplexity, SGit CLI, vault web UI) all working on the same vault without conflicts — confirmed working in a real session. The collaboration model is no longer theoretical.

All of these features are **PROPOSED — do not exist yet** in the main SG/Send UI.

**Roles impacted:** Dev (vault upload beta), Designer (UX for upload progress + placeholder files), Ambassador (messaging, website copy), Conductor (website positioning update)

### Theme 2: Healthcare Vertical — Three-Brief Thread (docs 284, 287, 288)

A healthcare partner demo (unrecorded, referenced in 17 Apr daily brief) produced three briefs in two days, forming the first coherent vertical use case.

**Video recorder (doc 287):** Client-side camera + screen recording. MediaDevices API + Screen Capture API. AES-256-GCM encryption in browser. Zero GDPR data processing exposure — the recording stays in the browser until encrypted, then only ciphertext leaves. Three output modes: single SG/Send file, folder with thumbnail + metadata, vault with versioned commits. This is also an immediately useful internal tool (screen capture for demos/walkthroughs).

**Healthcare MVP (doc 288):** GP health scan. 15–20 questions, all client-side JavaScript. Vault as data layer (JSON per section, each section = one commit). LLM analysis via OpenRouter (patient sees model + cost before submitting). Clinician review + sign-off via dual branch model. Published summary as read-only share token. Two items already exist: vault creation API and doctor collaboration view (Push/Pull). Six items are new builds.

**Architecture claim:** The healthcare MVP explicitly proves that "all application logic can be client-side." The server stores only encrypted ciphertext. No server-side patient records. This is a demonstrable, inspectable proof of the architecture.

All healthcare MVP features are **PROPOSED — do not exist yet**.

**Roles impacted:** Dev (video recorder + MVP build), Designer (form UX, LLM cost display), AppSec (GDPR review, healthcare data governance), Architect (client-side-only claim verification)

### Theme 3: Identity and Auth — From Research to MVP (docs 286, 292)

**Research brief (doc 286):** Identifies the fundamental gap — vault keys have no home. They live in users' heads or on laptops. The brief requests structured research on 7 technologies (Cognito, GCP, Azure, Google OAuth, GitHub, password managers, Chrome extension). Key concept: master vault key derivation (one stored key derives all vault-specific keys, consistent with existing PBKDF2 pattern). Cognito recommended as starting point.

**Auth MVP (doc 292):** Turns the research into a concrete implementation spec. Social login only (Google + GitHub OAuth). No passwords. Two providers: Cognito (primary) and Firebase (secondary). Vault key storage evaluated across 4 options; MVP uses Option A (user attributes). Four reusable Web Components. Critical constraint: accounts are additive — the platform continues to work without any account.

**Unblocks:** Open decision #1 (profile page + credit activation) is now unblocked by the auth MVP.

All auth MVP features are **PROPOSED — do not exist yet**.

**Roles impacted:** Dev (auth MVP, 4 Web Components), AppSec (zero-knowledge guarantee at identity layer, master key loss recovery), Architect (client-side auth flow, Cognito vs Firebase comparison)

### Theme 4: Playwright Service — From Concept to Product (docs 281–282, 289, 291–295)

The most architecturally dense theme across the three days, evolving from a single April 16 arch brief into a full product + architecture + implementation brief cluster.

**April 16 arch brief (doc 282):** Original design — decoupled Docker image (stable Playwright + Chrome) with runtime-loaded service code. Two API modes. Vault as config + output. 4 deployment targets. 3 open questions on Chrome profiles, timeouts, concurrency.

**April 18 product brief (doc 293):** Elevates the service to a standalone sub-product: "The browser for agents." Adds MITM proxy (100% traffic capture), per-execution billing, AMI + AWS Marketplace experiment, open source + hosted service revenue model. First standalone product outside SG/Send.

**April 18 architecture brief (doc 291):** Three-layer architecture in detail. Layer 1 (core-fastapi) / Layer 2 (adds S3) / Layer 3 (adds Playwright). Dynamic code loading (local, URL, S3). Development mode (fast iteration) vs frozen mode (self-contained). Docker Hub as primary distribution. Separate CI per layer. Admin API with runtime code upload.

**April 18 v4 architecture (doc 294):** Direction-of-travel document. Three axioms: statelessness, least-privilege by declaration, self-description via standard primitives. L1/L2 as PyPI packages, L3 as app images. Network sidecar for hostname policy enforcement. Lockdown layers (subtractive). SKILL files. This is a significantly richer architecture than the same-day layered container brief.

**April 18 implementation brief (doc 295):** Practical first pass for the Playwright dev team. Repo reorganisation into `agentic_fastapi/`, `agentic_fastapi_aws/`, `service/` subpackages. Naming chosen for future mechanical split. Admin API (10 MUST-have endpoints). Env var contract (`AGENTIC_*` prefix). Fast local loop (docker restart ~2s) + fast cloud loop (~30s via `scripts/deploy_code.py`). SKILL files + capabilities.json stub.

All Playwright service features described in docs 282–295 are **PROPOSED — do not exist yet** (the service exists in a separate repo `SGraph-AI__Service__Playwright`, not in this repo).

**Roles impacted:** Architect (v4 direction of travel, SKILL/capabilities standards), Dev (three-layer implementation, deploy scripts, admin API), DevOps (Docker Hub namespace, ECR pipelines, AMI packaging), AppSec (MITM proxy, capabilities.json least-privilege review)
