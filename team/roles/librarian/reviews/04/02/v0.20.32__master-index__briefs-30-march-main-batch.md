## Thematic Grouping

### Theme 1: LLM Conversation Architecture (Documents 205, 207, 208)

The most architecturally significant theme of this batch. Three documents together define how the LLM chat system should be restructured:

- **One-shot feedback loops** (doc 205): Context stays roughly constant across cycles (~500 tokens per cycle) rather than growing exponentially (15,000+ tokens by message 10). Extract facts → prune stale content → update bundle. Use SGit branches for parallel explorations. Each bundle lives in a vault. Two-LLM pattern: frontier model for answers, cheap/local model for pruning.
- **Chat improvements** (doc 207): Decompose the existing monolithic chat tool into single-responsibility components. Model Chooser as a standalone traffic driver. Broadcast communication for loose coupling. Five specific bugs to fix in the existing implementation.
- **Multi-agent chat** (doc 208): Parallel conversation panels in SG Layout. Multi-Ask (same question to N agents simultaneously). Debate Mode (agents critique each other). Consolidator. Async tasks for local Ollama models (zero cost).

**Reality check:** None of this exists yet. The existing workspace v0.1.0 has basic LLM chat components (`llm-chat`, `llm-connection`) but nothing approaching this architecture. The sg-llm component family (from doc 202, already in Section 19) provides the execution layer underneath. All three briefs describe **PROPOSED** functionality.

**Architectural note:** This batch answers a key design question not previously addressed — how do you prevent context window explosion in iterative LLM work? The one-shot feedback loop is the answer, and the bundle-as-vault pattern connects it directly to SGit.

---

### Theme 2: SGit as Universal Data Model (Documents 205, 209)

A significant conceptual leap: SGit is not just a vault CLI. It is a data structure that can model anything with branching history.

- **LLM bundles as vaults** (doc 205): Each cycle is a commit. Each exploration branch is a vault branch. Bundle history is a commit graph.
- **`sg-git-graph` component** (doc 209): The same Web Component renders both real SGit vault histories and LLM conversation threads. The underlying data model is identical: nodes (commits/interactions), edges (branches), merges (consolidations).

**Architectural implication:** This is the moment where SGit expands from "encrypted file storage" to "universal branching history model." The graph visualisation makes this concrete — it renders both domains with the same component because they are structurally identical.

**Reality check:** `sg-git-graph` does not exist. SGit branches exist (in sgit-ai). The visualisation layer is fully **PROPOSED**.

### Theme 3: Offline as a Design Choice (Document 206)

A brief that reframes the relationship between online and offline work. Key pattern: `@agent` comments in code, processed by the Librarian on the next online run. This extends the Librarian role into code — not just documents.

The vault-as-sync-point model (pull before, work offline, push after) is already architected by sgit. What doc 206 adds is:
1. Deliberate scheduling of offline blocks
2. Code annotation as async communication to agents
3. Librarian responsibility for processing `@agent` comments

**Reality check:** The offline sgit operations (commit, branch, diff, log) exist. The `@agent` comment processing pipeline is **PROPOSED**.
