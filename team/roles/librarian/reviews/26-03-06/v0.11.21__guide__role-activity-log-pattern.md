# Role Activity Log Pattern

**Version:** v0.11.21
**Date:** 6 March 2026
**Role:** Librarian
**Commit:** `63323ed` (HEAD of `claude/review-workspace-bugs-jy1ko`)

---

## Problem

When a new session starts, agents have no quick way to find:
1. What the last review was for each role
2. What version/commit that review was pinned to
3. Where to "resume from" when asked to update

The instruction "write an updated version of the last technical debrief" requires manually searching through review folders. This doesn't scale.

## Solution: Activity Log per Role

Each role maintains an `activity-log.md` file in its root directory:

```
team/roles/{role}/activity-log.md
```

### Format

```markdown
# {Role} Activity Log

| Date | Version | Commit | Review | Description |
|------|---------|--------|--------|-------------|
| 2026-03-06 | v0.11.21 | `63323ed` | [link](reviews/26-03-06/file.md) | Brief description |
| 2026-03-05 | v0.11.10 | `60cd2c0` | [link](reviews/26-03-05/file.md) | Brief description |
```

### Rules

1. **Append-only** — new entries go at the top of the table
2. **Version-pinned** — every entry records the `sgraph_ai_app_send/version` at time of writing
3. **Commit-pinned** — every entry records the HEAD commit hash
4. **Relative links** — links to reviews are relative to the activity-log.md file
5. **One row per review** — if a session produces 3 reviews, that's 3 rows
6. **Updated in same commit** — activity log updates ship in the same commit as the review

### Benefits

- **Session start:** Read `activity-log.md` to see last review and its version/commit
- **Gap detection:** Compare activity log version against current version to see what's undocumented
- **Continuity:** New agents know exactly where previous agents left off
- **Audit trail:** Git history of the activity log shows the evolution of each role's work

### Who Maintains It

Each role is responsible for updating its own activity log when producing a review. The Librarian verifies activity logs are current during catalogue reviews.
