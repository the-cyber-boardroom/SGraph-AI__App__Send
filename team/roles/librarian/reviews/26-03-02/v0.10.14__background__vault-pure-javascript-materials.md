## 1. Executive Summary

The v0.10.14 brief proposes building `vault.sgraph.ai` as a pure HTML/JS application sitting on top of send.sgraph.ai's existing encrypted file storage. The key insight is: **the server should have no idea it is hosting a vault**. All vault semantics (folders, navigation, settings, tree structure) live client-side in JavaScript.

This document catalogues everything that already exists in the codebase relevant to this effort:

- **17 vault endpoints** on the Admin Lambda (backend is built but uses a vault-specific cache layer — the new brief wants to use the existing transfer/file storage instead)
- **2 vault UI implementations** (admin vault-manager component + user-facing user-vault component) — both use RSA-4096 hybrid encryption, which the new brief wants to simplify to symmetric-only KDF-based encryption
- **A mature admin UI architecture** with EventBus, Web Components, API client patterns, and dark theme design system — called out as "working really well and we need to reuse"
- **A full i18n system** with 17 locales, build-time page generation, and runtime locale switching — the new brief requires "multilingual from day one"
- **Three distinct encryption patterns** already in use (Send symmetric AES-GCM, Vault hybrid RSA+AES, Room symmetric AES-GCM) — the room pattern is closest to what the new brief proposes
- **Two API client implementations** (user ApiClient and admin AdminAPI) — these are the precursors to the proposed `sg-send.js` library

## 2. Existing Vault Backend

### 2.1 Service Layer

**File:** `sgraph_ai_app_send/lambda__admin/service/Service__Vault.py`

The `Service__Vault` class (extends `Type_Safe`) manages vault lifecycle through a dedicated cache client (`Send__Cache__Client__Vault`). Key operations:

| Method | Description |
|---|---|
| `create(vault_cache_key, key_fingerprint, owner_user_id)` | Creates vault with root folder. Returns `{cache_id, root_folder, owner_user_id, created}` |
| `lookup(vault_cache_key)` | Returns vault manifest |
| `exists(vault_cache_key)` | Boolean check |
| `store_folder / get_folder / update_folder / list_folders / delete_folder` | Folder CRUD by GUID |
| `store_file / get_file / list_files / delete_file` | Encrypted file CRUD by GUID |
| `store_file_chunk / assemble_file` | Chunked upload for files exceeding Lambda 6MB payload limit |
| `store_index / get_index / update_index` | Encrypted index (maps GUIDs to human-readable names) |
| `list_all / delete_all` | Bulk operations |

**Data Model — Vault Manifest:**
```python
manifest = {
    'type':            'vault_root',
    'created':         '2026-03-02T10:00:00+00:00',
    'key_fingerprint': 'sha256:abcdef0123456789',
    'owner_user_id':   '',
    'root_folder':     'a1b2c3d4'   # 8-hex GUID
}
```

**Data Model — Folder:**
```python
folder = {
    'type':     'folder',
    'id':       'a1b2c3d4',
    'children': ['e5f6g7h8', 'i9j0k1l2']  # GUIDs of child items
}
```

**Data Model — Encrypted Index (client-side, after decryption):**
```json
{
    "version": 1,
    "entries": {
        "e5f6g7h8": { "name": "report.pdf", "type": "file", "size": 245000, "parentGuid": "a1b2c3d4", "mime": "application/pdf" },
        "i9j0k1l2": { "name": "documents",  "type": "folder", "parentGuid": "a1b2c3d4" }
    }
}
```

### 2.2 ACL Service

**File:** `sgraph_ai_app_send/lambda__admin/service/Service__Vault__ACL.py`

Phase 1 ACL with permission levels: `viewer`, `editor`, `owner`. Stores permissions as cache entries with grant/revoke/list operations.

### 2.3 Cache Client

**File:** `sgraph_ai_app_send/lambda__admin/service/Send__Cache__Client__Vault.py`

Dedicated cache client using prefixed keys to namespace vault data:
- `vault_root/{vault_cache_key}` — manifest
- `vault_folder/{cache_id}/{folder_guid}` — folder data
- `vault_file/{cache_id}/{file_guid}` — encrypted file bytes
- `vault_index/{cache_id}` — encrypted index

### 2.4 Routes

**File:** `sgraph_ai_app_send/lambda__admin/fast_api/routes/Routes__Vault.py`

17 endpoints registered on the Admin Lambda:

| Method | Path | Description |
|---|---|---|
| POST | `/vault/create` | Create vault |
| GET | `/vault/lookup/{vault_cache_key}` | Get manifest |
| GET | `/vault/exists/{vault_cache_key}` | Check existence |
| POST | `/vault/folder` | Store folder |
| GET | `/vault/folder/{vault_cache_key}/{folder_guid}` | Get folder |
| GET | `/vault/folders/{vault_cache_key}` | List folders |
| POST | `/vault/file` | Store encrypted file |
| GET | `/vault/file/{vault_cache_key}/{file_guid}` | Get encrypted file |
| GET | `/vault/files/{vault_cache_key}` | List files |
| POST | `/vault/file-chunk` | Upload a chunk |
| POST | `/vault/file-assemble` | Assemble chunks |
| POST | `/vault/index` | Store encrypted index |
| GET | `/vault/index/{vault_cache_key}` | Get encrypted index |
| GET | `/vault/list-all/{vault_cache_key}` | List all items |
| POST | `/vault/share/{vault_cache_key}` | Grant access |
| DELETE | `/vault/unshare/{vault_cache_key}/{user_id}` | Revoke access |
| GET | `/vault/permissions/{vault_cache_key}` | List permissions |

### 2.5 Request Schemas

All schemas use `Type_Safe` (not Pydantic):

| Schema | Fields |
|---|---|
| `Schema__Vault__Create__Request` | `vault_cache_key`, `key_fingerprint` |
| `Schema__Vault__Folder__Request` | `vault_cache_key`, `folder_guid`, `folder_data` |
| `Schema__Vault__File__Request` | `vault_cache_key`, `file_guid`, `encrypted_data` (base64) |
| `Schema__Vault__File__Chunk__Request` | `vault_cache_key`, `file_guid`, `chunk_index`, `total_chunks`, `chunk_data` (base64) |
| `Schema__Vault__Index__Request` | `vault_cache_key`, `encrypted_index` (base64) |
| `Schema__Vault__Share__Request` | `user_id`, `permission` |

### 2.6 Relevance to New Brief

The new brief proposes **not using any of these vault-specific endpoints**. Instead, the vault will use the existing **transfer/file storage endpoints** (`/api/transfers/create`, `/api/transfers/upload/{id}`, etc.) — the same ones the Send tool uses. The vault backend becomes the "dumb" encrypted file store it already is.

However, the new brief does require **path-based file organisation** (up to 5 levels of Object IDs separated by `/`), which is a small server-side change to the existing transfer service.

## 3. Existing Vault UI

### 3.1 User-Facing Vault (v0.1.7)

**Location:** `sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.7/`

Three files compose the user vault:

| File | Purpose |
|---|---|
| `vault.html` | Entry page. Loads common CSS + Aurora design system. Contains `<user-vault>` custom element. |
| `js/vault-crypto.js` | RSA-4096 key pair generation/storage (IndexedDB), hybrid AES+RSA encryption, vault cache key derivation |
| `js/vault-api.js` | Fetch-based API client for all 17 vault endpoints on the Admin Lambda |
| `components/user-vault/user-vault.js` | ~600-line Web Component: key generation UI, folder navigation, file upload/download/rename/delete |

**Architecture:**
```
vault.html
  └─ <user-vault> component
       ├─ VaultCrypto (key generation, encryption)
       ├─ VaultAPI (server communication)
       └─ Aurora design system CSS
```

**Key Features:**
- RSA-4096 key pair generated on first use (takes ~2-3 seconds, shows progress)
- Key pair stored in IndexedDB (`sgraph-send-vault-keys`)
- Vault cache key derived from public key hash: `SHA-256(SPKI) → first 32 hex → SHA-256(hash + '/filesystem') → first 32 hex`
- Folder tree with breadcrumb navigation
- Inline folder creation, rename, delete (no browser dialogs)
- Drag-and-drop file upload
- Sortable columns (name, size)
- Status bar showing key fingerprint, vault ID, file/folder counts
- All file names stored in encrypted index, never sent to server

### 3.2 Admin Vault Manager (v0.1.5)

**Location:** `sgraph_ai_app_send__ui__admin/v0/v0.1/v0.1.5/components/vault-manager/vault-manager.js`

A mirror of the user vault but integrated into the admin shell. Registered as `<vault-manager>` with `appId: 'vault'`, `navLabel: 'My Vault'`.

**Key differences from user vault:**
- Uses admin PKI keys from IndexedDB (`sg-send-pki`) rather than dedicated vault keystore
- Uses `adminAPI.vaultXxx()` methods (IIFE extension pattern) rather than standalone VaultAPI
- Imports shared utilities from `window.sgraphAdmin.pki` (escapeHtml, arrayBufToB64, etc.)
- Styled with admin CSS variables (`--admin-*`)

**Admin Vault API Extension:**
**File:** `sgraph_ai_app_send__ui__admin/v0/v0.1/v0.1.5/js/admin-api-vault.js`

Extends the `adminAPI` singleton via IIFE, adding methods:
- `adminAPI.vaultCreate()`, `.vaultLookup()`, `.vaultExists()`
- `.vaultStoreFolder()`, `.vaultGetFolder()`, `.vaultListFolders()`
- `.vaultStoreFile()`, `.vaultGetFile()`, `.vaultListFiles()`
- `.vaultStoreFileChunk()`, `.vaultAssembleFile()`
- `.vaultStoreIndex()`, `.vaultGetIndex()`
- `.vaultListAll()`
- `.vaultShare()`, `.vaultUnshare()`, `.vaultPermissions()`

### 3.3 Relevance to New Brief

The new brief replaces both vault UIs with a new standalone application at `vault.sgraph.ai`. The key architectural change is:

| Aspect | Current Vault | New Brief |
|---|---|---|
| **Encryption model** | RSA-4096 hybrid (per-user key pair) | Symmetric AES-256-GCM from KDF(Vault Key) |
| **Key management** | Key pair in IndexedDB | Vault Key entered by user, derived via PBKDF2/Argon2 |
| **Server endpoints** | 17 vault-specific endpoints | Existing transfer endpoints (upload/download/list) |
| **File organisation** | Flat GUIDs + encrypted index | Path-based (Object IDs, up to 5 levels) |
| **Sharing model** | ACL-based (user IDs) | Share the Vault Key |
| **Metadata** | Encrypted index JSON | vault-settings.json + vault-tree.json (both encrypted) |

**What to reuse from existing vault UI:**
- Folder navigation patterns (breadcrumb, sortable table, inline create/rename/delete)
- Drag-and-drop upload pattern
- Status bar pattern (key info, vault stats)
- File download pattern (fetch, decrypt, blob URL, `<a>` click)
- Error handling patterns (inline messages, loading states)

**File:** `sgraph_ai_app_send__ui__admin/v0/v0.1/v0.1.3/index.html`

```

  ├─ config-manager.js    → window.sgraphAdmin.config
  └─ messages-service.js  → window.sgraphAdmin.messages

Layer 2: API clients + libraries
  ├─ admin-api.js         → window.adminAPI (singleton)
  ├─ admin-api-cache.js   → extends adminAPI
  ├─ admin-api-keys.js    → extends adminAPI
  └─ qr-code.js

Layer 3: Debug panels
  ├─ messages-panel.js
  ├─ events-viewer.js
  └─ api-logger.js

Layer 4: Feature components
  ├─ 4a: PKI (pki-common.js, pki-keys, pki-encrypt, pki-contacts)
  ├─ 4b: Key discovery (key-publish, key-lookup, key-registry, key-log)
  └─ 4c: Legacy (token-manager, analytics-dashboard, system-info, metrics-dashboard, storage-browser)

Layer 5: Registration overlay (adds appId to legacy components)
  └─ register-legacy-apps.js

Layer 6: Shell (reads children, builds layout, auto-registers apps)
  └─ admin-shell.js
```

### 4.2 EventBus Pattern

**File:** `sgraph_ai_app_send__ui__admin/v0/v0.1/v0.1.3/js/event-bus.js`

Central nervous system. **Components never call each other directly.** All communication goes through the EventBus.

```javascript
// API
window.sgraphAdmin.events.on(event, handler)    // Subscribe
window.sgraphAdmin.events.off(event, handler)   // Unsubscribe
window.sgraphAdmin.events.emit(event, data)     // Publish
window.sgraphAdmin.events.getHistory()           // Last 200 events
window.sgraphAdmin.events.clearHistory()

// Key events emitted by the system:
// 'shell-ready'       — admin shell finished loading
// 'app-registered'    — a component registered with the router
// 'navigated'         — user navigated to a different component
// 'message-added'     — a notification was posted
// 'messages-cleared'  — all notifications cleared
```

### 4.3 MessagesService Pattern

**File:** `sgraph_ai_app_send__ui__admin/v0/v0.1/v0.1.3/js/messages-service.js`

User-facing notifications with auto-dismissal and severity-based behaviour:

```javascript
window.sgraphAdmin.messages.success(text)   // Auto-dismiss 5s, no auto-open
window.sgraphAdmin.messages.info(text)      // Auto-dismiss 4s, no auto-open
window.sgraphAdmin.messages.warning(text)   // Auto-dismiss 8s, auto-opens debug sidebar
window.sgraphAdmin.messages.error(text)     // No auto-dismiss, auto-opens debug sidebar
window.sgraphAdmin.messages.getMessages()   // All messages

```

### 4.5 API Client Pattern

**File:** `sgraph_ai_app_send__ui__admin/v0/v0.1/v0.1.0/js/admin-api.js`

Singleton pattern with cookie-based auth:

```javascript
class AdminAPI {
    constructor() { this.baseUrl = ''; }

    async _request(path, options = {}) {
        const config = {
            credentials: 'include',
            headers: { 'Content-Type': 'application/json', ...options.headers },
            ...options
        };
        const response = await fetch(url, config);
        if (!response.ok) { /* extract error detail from JSON body */ }
        return response.json();
    }

    _get(path)        { return this._request(path, { method: 'GET' }); }
    _post(path, body) { return this._request(path, { method: 'POST', body: JSON.stringify(body) }); }
    _delete(path)     { return this._request(path, { method: 'DELETE' }); }
}

const adminAPI = new AdminAPI();  // Singleton
```

**Extension Pattern (IIFE):**
```javascript
// Other files extend the singleton without subclassing:
(function() {
    adminAPI.vaultCreate = function(key, fp) {
        return this._post('/vault/create', { vault_cache_key: key, key_fingerprint: fp });
    };
})();
```

### 5.3 Locale JSON Structure

**Location:** `sgraph_ai__website/i18n/`

Each JSON file is a flat key-value map with dot-separated keys:

```json
{
    "_comment": "SGraph AI Website -- English (United Kingdom)",
    "_agent_ambassador": "=== Agent: Ambassador ===",

    "agent.ambassador.card1.title": "Market Research",
    "agent.ambassador.card1.desc": "Analyses the competitive landscape ...",

    "pricing.main.hero.title": "Transparent Pricing",
    "pricing.main.hero.subtitle": "Zero cost when idle ...",

    "nav.home": "Home",
    "nav.pricing": "Pricing"
}
```

**Conventions:**
- Keys prefixed with `_` are comments/section markers (stripped at load time)
- Keys use dot notation: `{page}.{section}.{element}.{property}`
- Values can contain HTML entities (`&mdash;`, `&rarr;`)
- Missing keys fall back to English (en-gb) automatically

### 5.4 Build-Time Page Generation

**File:** `scripts/generate_i18n_pages.py`

The generator reads English source HTML from `sgraph_ai__website/en-gb/`, applies translations, and outputs locale-specific pages:

```
sgraph_ai__website/
  en-gb/                    ← English source pages
    index.html
    pricing/index.html
    ...
  i18n/                     ← Translation JSON files
    en-gb.json
    pt-pt.json
    ...
  pt-pt/                    ← Generated locale pages
    index.html              ← Pre-rendered with Portuguese content
    pricing/index.html
  de-de/
    index.html
    ...
```

**Translation mechanism in HTML:**
```html
<!-- data-i18n attribute marks translatable content -->
<h1 data-i18n="pricing.main.hero.title">Transparent Pricing</h1>

<!-- data-i18n-placeholder for form inputs -->
<input data-i18n-placeholder="search.placeholder" placeholder="Search...">

<!-- i18n-title-key for page titles -->
<meta name="i18n-title-key" content="pricing.main.meta.title">
```

**Build steps:**
1. Load English master JSON (`en-gb.json`)
2. Load target locale JSON (e.g., `pt-pt.json`)
3. For each HTML file in `en-gb/`:
   - Replace `<html lang="en-GB">` with `<html lang="pt-PT">`
   - Find all `data-i18n="key"` elements, replace content with translated value
   - Find all `data-i18n-placeholder="key"` inputs, replace placeholder
   - Replace `<title>` if `i18n-title-key` meta tag present
   - Inject `<meta name="i18n-prerendered" content="true">`
   - Inject hreflang `<link>` tags for all 17 locales
   - Inject canonical URL
   - Rewrite cross-site links (e.g., `send.sgraph.ai` → `send.sgraph.ai/pt-pt/`)

### 5.5 Runtime i18n Module

**File:** `sgraph_ai__website/_common/js/i18n.js`

The `SgI18n` object handles runtime concerns (no JSON fetching — translations are pre-rendered):

```javascript
const SgI18n = {
    locale: 'en-gb',
    availableLocales: [ /* 17 locales with code, name, flag emoji */ ],

    init()                      // Detect locale from URL path
    _detectLocale()             // Parse URL: /pt-pt/pricing/ → 'pt-pt'
    navigateToLocale(code)      // Navigate to locale version of current page
    renderLocaleSelector(id)    // Render dropdown locale picker
};
```

**Locale detection:** URL path prefix matching. `/pt-pt/pricing/` detects `pt-pt`.

**Locale selector:** Two-column dropdown grid with flag emojis, "Quick switch to English" link on non-English pages.

### 5.6 Send UI i18n (v0.1.6)

The Send UI (v0.1.6) has its own simpler i18n:

**File:** `sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.6/js/i18n.js`

This is a separate, lightweight i18n module for the Send tool. Components call `I18n.t('upload.button.encrypt_send')` for translated strings.

### 5.7 Relevance to New Brief

For `vault.sgraph.ai`, the recommended approach is:

1. **Reuse the locale JSON structure** — same flat key-value format with dot-separated keys
2. **Create vault-specific i18n keys** — `vault.entry.title`, `vault.tree.empty`, etc.
3. **Reuse build-time generation** — extend `generate_i18n_pages.py` or create a vault-specific variant
4. **Reuse `SgI18n` runtime module** — locale detection, selector rendering, navigation
5. **Support all 17 locales from day one** — even if initially only en-gb has full content

## 6. Encryption Patterns

### 6.1 Pattern A: Send Tool — Pure Symmetric AES-256-GCM

**Files:**
- `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/_common/js/crypto.js`
- `sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.3/components/send-upload/send-upload.js`

```javascript
const SendCrypto = {
    ALGORITHM:  'AES-GCM',
    KEY_LENGTH: 256,
    IV_LENGTH:  12,

    async generateKey()           // crypto.subtle.generateKey (AES-GCM, 256, extractable)
    async exportKey(key)          // Export as base64url string
    async importKey(base64Key)    // Import from base64url string
    async encryptFile(key, data)  // [12B IV][ciphertext+tag]
    async decryptFile(key, data)  // Reverse of above
};
```

**Binary format:** `[12 bytes IV][ciphertext with GCM auth tag]`

**Key sharing:** Key is embedded in URL hash fragment (`#transferId/base64urlKey`). Never sent to server.

**SGMETA Envelope:** Wraps file content with metadata before encryption:
```
[7B magic "SGMETA\0"][4B metadata-length (big-endian)][JSON metadata][file content]
```
Metadata contains `{ "filename": "report.pdf" }`. The server never sees the filename.

### 6.2 Pattern B: Vault — RSA-4096 Hybrid (Current)

**Files:**
- `sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.7/js/vault-crypto.js`
- `sgraph_ai_app_send__ui__admin/v0/v0.1/v0.1.3/js/pki-common.js`

```javascript
// Encrypt: generate AES key → encrypt data → wrap AES key with RSA public key
async function encrypt(publicKey, data) {
    const aesKey     = generateKey();
    const iv         = randomBytes(12);
    const encrypted  = AES-GCM-encrypt(aesKey, iv, data);
    const wrappedKey = RSA-OAEP-encrypt(publicKey, rawAesKey);
    // Pack: [4B wrappedKey-length][wrappedKey][12B IV][ciphertext]
    return packed;
}

// Decrypt: unwrap AES key with RSA private key → decrypt data
async function decrypt(privateKey, packed) {
    const wrappedKey = extract(packed);
    const rawAesKey  = RSA-OAEP-decrypt(privateKey, wrappedKey);
    return AES-GCM-decrypt(aesKey, iv, ciphertext);
}
```

**Binary format:** `[4B uint32 wrappedKey-length][wrappedKey (~512B for RSA-4096)][12B IV][ciphertext+tag]`

**Key management:**
- RSA-4096 key pair stored in IndexedDB
- Keys are non-extractable (generated with `extractable: false`)
- Vault cache key derived: `SHA-256(SPKI export) → 32 hex → SHA-256(hex + '/filesystem') → 32 hex`

### 6.3 Pattern C: Data Room — Shared Symmetric Key

**File:** `sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.8/js/room-crypto.js`

```javascript
const RoomCrypto = {
    generateRoomKey()           // 32 random bytes → hex string (64 chars)
    async importKey(hexKey)     // hex → Uint8Array → importKey (AES-GCM, non-extractable)
    async deriveKey(passphrase) // SHA-256(passphrase) → importKey (AES-GCM)
    async encrypt(key, data)    // [12B IV][ciphertext+tag]
    async decrypt(key, data)    // Reverse of above
    arrayBufToB64(buf)          // Standard base64 codec
    b64ToArrayBuf(b64)
};
```

**Binary format:** `[12B IV][ciphertext+tag]` — simpler than vault hybrid format, no RSA wrapping.

**Key sharing:** Room key embedded in URL hash (like Send links). Also supports passphrase-based key derivation via `SHA-256(passphrase)`.

### 6.4 Relevance to New Brief

The new brief proposes encryption closest to **Pattern C (Room Crypto)** but with a proper KDF:

```
Vault Key (user-entered string)
  → KDF (PBKDF2 or Argon2)
  → AES-256 key
  → Encrypt/decrypt all vault files
```

**Key differences from RoomCrypto:**
- Uses PBKDF2/Argon2 instead of plain SHA-256 for key derivation (more secure against brute force)
- Single key for entire vault (same as room pattern)
- Future: per-folder key derivation paths

**What to reuse from existing patterns:**
- `SendCrypto.encryptFile` / `decryptFile` format: `[12B IV][ciphertext]`
- `RoomCrypto.deriveKey` pattern (but upgrade to PBKDF2)
- Base64 codec functions
- Secure context checking

## 7. sg-send.js — Existing Client Library Patterns

The new brief calls for an `sg-send.js` library. No such library exists yet, but there are two API client implementations that serve as precursors:

### 7.1 User API Client (v0.2.0)

**File:** `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/_common/js/api-client.js`

Singleton object (`const ApiClient = {...}`) with:

| Method | Endpoint | Description |
|---|---|---|
| `createTransfer(size, type)` | POST `/api/transfers/create` | Start a transfer |
| `uploadPayload(id, encrypted)` | POST `/api/transfers/upload/{id}` | Upload encrypted bytes |
| `completeTransfer(id)` | POST `/api/transfers/complete/{id}` | Finalise |
| `getTransferInfo(id)` | GET `/api/transfers/info/{id}` | Get metadata |
| `downloadPayload(id)` | GET `/api/transfers/download/{id}` | Download encrypted bytes |
| `checkToken(name)` | GET `/api/transfers/check-token/{name}` | Check token (read-only) |
| `validateToken(name)` | POST `/api/transfers/validate-token/{name}` | Validate + consume use |
| `getCapabilities()` | GET `/api/presigned/capabilities` | Check upload capabilities |
| `initiateMultipart(id, size, parts)` | POST `/api/presigned/initiate` | Start multipart |
| `uploadPart(url, data)` | PUT `{presigned-url}` | Upload part to S3 |
| `completeMultipart(id, uploadId, parts)` | POST `/api/presigned/complete` | Complete multipart |
| `abortMultipart(id, uploadId)` | POST `/api/presigned/abort/{id}/{uploadId}` | Cancel multipart |

**Auth:** `x-sgraph-access-token` header from localStorage.

**URL prefix:** All routes use `/api/*` prefix for CloudFront separation.

### 7.2 Admin API Client (v0.1.0)

**File:** `sgraph_ai_app_send__ui__admin/v0/v0.1/v0.1.0/js/admin-api.js`

Class-based singleton (`class AdminAPI { ... }; const adminAPI = new AdminAPI();`) with:
- Cookie-based auth (`credentials: 'include'`)
- `_request()` / `_get()` / `_post()` / `_delete()` base methods
- Error handling: extract detail from JSON response body
- Content-Type removed for GET requests

### 7.3 Vault API Client (v0.1.7)

**File:** `sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.7/js/vault-api.js`

Singleton object (`const VaultAPI = {...}`) for vault endpoints. Same fetch pattern as ApiClient but talks to the Admin Lambda.

### 7.4 Building sg-send.js

The proposed `sg-send.js` should merge concepts from all three:

```javascript
const sg = new SGSend({
    endpoint: 'https://send.sgraph.ai',
    token: '...'
});

// Core operations (from ApiClient):
await sg.upload(fileData, { path: 'vault-001/settings.json' });
const file = await sg.download('vault-001/settings.json');
await sg.delete('vault-001/old-file.json');
const files = await sg.list('vault-001/');

// Encryption layer (from SendCrypto + RoomCrypto):
const encrypted = await sg.encrypt(plaintext, key);
const decrypted = await sg.decrypt(ciphertext, key);

// Vault-specific (from VaultAPI, reimagined):
const vault = await sg.openVault(vaultKey);
const tree = await vault.getTree();
```

**Key design decisions:**
- Class-based (not singleton object) for multiple instances
- Token-based auth (header, not cookie) for cross-origin use
- Path-based file addressing (new server feature)
- Built-in encryption layer
- Vault logic as a higher-level abstraction on top of core operations

## 8. Key Differences: Current Vault vs New Brief

| Dimension | Current Implementation | New Brief (v0.10.14) |
|---|---|---|
| **Backend** | 17 vault-specific endpoints on Admin Lambda | Use existing transfer endpoints on User Lambda |
| **Storage model** | Vault-specific cache client with prefixed keys | Path-based Object IDs on existing storage |
| **Encryption** | RSA-4096 hybrid (per-user key pair) | Symmetric AES-256-GCM from KDF(Vault Key) |
| **Key management** | IndexedDB key pair, non-extractable | User enters Vault Key, derived via PBKDF2 |
| **Sharing** | ACL-based (user IDs, permissions) | Share the Vault Key (symmetric) |
| **File naming** | GUIDs + encrypted index | Path-based Object IDs + encrypted vault-tree.json |
| **Metadata structure** | Single encrypted index JSON | vault-settings.json + vault-tree.json |
| **Folder model** | Server-side folder objects | Client-side only (tree in vault-tree.json) |
| **Versioning** | None | Built into tree (file entry has versions array) |
| **UI location** | Embedded in admin console + user vault page | Standalone app at vault.sgraph.ai |
| **i18n** | English only | Multilingual from day one (17 locales) |
| **Client library** | Separate API clients per UI | Unified sg-send.js |
