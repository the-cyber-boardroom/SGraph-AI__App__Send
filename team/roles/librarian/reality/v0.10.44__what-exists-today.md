## 4. Infrastructure That Works

### Storage
- **Memory-FS** -- in-memory (dev/test), ~100ms startup
- **S3** -- production storage, auto-bucket creation
- **Auto-detection** via `SEND__STORAGE_MODE` env var or AWS credential presence

### Deployment
- **2 Lambda functions** -- User (public) + Admin (auth-protected)
- **Lambda URL** -- direct HTTPS, no API Gateway
- **3 stages each** -- dev, qa, prod (6 Lambda functions total)
- **CI/CD** -- GitHub Actions: push to `dev` triggers tests + deploy to dev; push to `main` triggers tests + deploy to qa + PyPI publish

### Static Website Deployment (NEW in v0.7.6)
- **deploy-website.yml** -- GitHub Actions workflow: validates HTML files, checks internal links, syncs to S3 with differentiated cache TTLs (HTML 300s, CSS/JS 86400s, images 604800s), invalidates CloudFront, runs smoke test
- **Triggers:** push to `main` on `sgraph_ai__website/**` paths, or manual workflow_dispatch
- **Target:** S3 bucket (via `WEBSITE_S3_BUCKET` secret) + CloudFront distribution (via `WEBSITE_CF_DIST` secret)
- **Region:** eu-west-2

### MCP (Model Context Protocol)
- **Stateless HTTP transport** -- Lambda-compatible (no session persistence)
- **User Lambda MCP tools:** transfers, presigned, vault endpoints
- **Admin Lambda MCP tools:** tokens, keys, vault, users endpoints
- **Verified working:** Claude.ai generated PDF, encrypted it, uploaded via MCP, human decrypted in browser (Milestone M-007)

## 5. Security Properties (Verified in Code)

- Server never sees plaintext (encryption in browser, AES-256-GCM)
- No file names on server (SGMETA envelope stays client-side)
- No decryption keys on server (key in URL hash fragment, never sent)
- IP addresses hashed (SHA-256 with daily salt, stored as `ip_hash`)
- Token-gated uploads (header `x-sgraph-send-access-token` or query param)
- Immutable audit trail (append-only, hash-chained events)
- Vault ACL enforcement (owner/editor/viewer hierarchy)
- Room session tokens (24h expiry, revocable)

**External validation (28 Feb 2026):** An independent security review (ChatGPT, conversation-based assessment) examined CloudFront log data from `send.sgraph.ai`. Verdict: **"Low-risk operational metadata exposure. Not a security incident."** Transfer IDs are opaque without the browser-held decryption key. Client IPs are deliberately excluded from CloudFront logging configuration. No credentials, encryption keys, presigned URLs, session cookies, or personal data present in logs. The review explicitly confirms: "the exposure of operational logs does not materially affect user security." See Section 12 for full details.

- **IFD versioning:** Each UI version patches specific components, inherits others

## 12. External Validation: CloudFront Log Security Review

**Date:** 28 Feb 2026
**Reviewer:** External AI-assisted review (ChatGPT, conversation-based assessment)
**Document:** `team/humans/dinis_cruz/briefs/02/28/security-review/v0.7.6__chat-gpt__review-of-data-stored-in-s3__from-cloudfront-logs.md`

An independent security review examined CloudFront log data from `send.sgraph.ai` captured on 27 Feb 2026 to determine whether the log contents constitute a security or privacy risk.

### Verdict

**Low-risk operational metadata exposure. Does not qualify as a security incident.**

### Key Findings (Verified by Reviewer)

| Finding | Detail |
|---------|--------|
| Transfer IDs are opaque | Without the browser-held decryption key (URL fragment), transfer IDs do not enable data access |
| Client IPs excluded | Deliberately excluded from CloudFront logging configuration |
| No credentials exposed | No encryption keys, presigned URLs, session cookies, or authorization headers in logs |
| No personal data exposed | No email addresses, account identifiers, or PII |
| Tokens are low-risk | Intentionally shareable, usage-capped, monitored, revocable |
| Response sizes do not weaken encryption | Ciphertext length does not compromise properly implemented AES-256-GCM |

### Data Present in Logs

Timestamps, HTTP methods, status codes, hostnames (`send.sgraph.ai`, `admin.send.sgraph.ai`), endpoint paths, user agents, TLS version/cipher, response sizes, CloudFront edge locations, country codes.

### Data NOT Present in Logs

Client IP addresses, email addresses, account identifiers, authorization headers, session cookies, bearer tokens, AWS credentials, presigned S3 URLs, encryption keys, query strings, request bodies.

### Significance

This review validates a core architectural claim: even if all server logs are published, user data remains safe. This is a direct consequence of the zero-knowledge encryption design. The review is cited in the daily brief as a proof point for marketing and educational content.

**Note:** This review was conducted by an external AI assistant, not a human security auditor. AppSec should independently validate the findings.

## 13. Performance Baseline: send.sgraph.ai Lambda (Pre-CDN)

**Date:** 28 Feb 2026
**Document:** `team/humans/dinis_cruz/briefs/02/28/v0.7.6__perf-baseline__send-sgraph-ai-lambda-performance.md`
**Page measured:** `/send/v0/v0.1/v0.1.8/index.html`

This is the **pre-optimization baseline** -- all 18 requests (HTML, CSS, JS, fonts, API calls) are served by the FastAPI Lambda function. No static assets on CDN.

### Three States of Performance

| Scenario | DOMContentLoaded | Page Load | Total Finish | Slowest Resource |
|----------|-----------------|-----------|-------------|-----------------|
| **Cold start** (Lambda reset) | **18.49 s** | **18.50 s** | **19.10 s** | 9.29 s (i18n.js) |
| **Warming** (first request post-deploy) | 3.50 s | 3.68 s | 4.26 s | 3.39 s (design system CSS) |
| **Warm** (subsequent) | 557 ms | 599 ms | 1.10 s | 542 ms (token check API) |

### Key Facts

- 18 requests per page load, all hitting Lambda (including static assets)
- Cold start penalty: ~9 seconds for Lambda initialization
- Warm Lambda: ~170ms per static resource (overhead that CDN would eliminate)
- Two external dependencies: Google Fonts CSS (0.9KB, 31ms) + DM Sans woff2 (63.1KB, 167ms)
- Token check API call: 542-759ms (actual business logic)
- Data transferred: 114KB; Resources loaded: 212KB

### Projected Improvement After CDN Migration

| Scenario | Current | After CDN | After CDN + Lambda Opt |
|----------|---------|-----------|----------------------|
| Cold start | 18.50 s | ~2-3 s | <1.5 s |
| Warm | 599 ms | ~300 ms | ~300 ms |
| First request post-deploy | 3.68 s | ~1 s | <1 s |

**Status:** This is a baseline reference document. Measurements should be repeated after CDN migration to validate projected improvements.
