**version** v0.6.2

**role** Architect (Explorer team)

## Purpose

Review the architectural readiness of SGraph Send for the Data Room product, map what's solid, identify gaps, and recommend the technical path forward for Phases 3–5.

## Architecture Readiness Assessment

### What's Architecturally Sound

| Area | Assessment | Confidence |
|------|-----------|------------|
| **Storage abstraction (Memory-FS)** | Clean. `Storage_FS` with S3 backend. Application code doesn't know which backend is active. | High |
| **Two-Lambda topology** | Correct separation. Public (transfers) vs Admin (tokens, vault, keys, users). | High |
| **Encryption model** | AES-256-GCM client-side, RSA-OAEP key wrapping, server never sees plaintext. V2 multi-recipient format working. | High |
| **Presigned URL flow** | Clean. Browser → presigned S3 → server validates. Multipart for large files. Parallel 5-wide. | High |
| **ACL model** | Simple and correct. owner > editor > viewer hierarchy. Stored alongside vault data. | Medium-High |
| **User identity** | Minimal but functional. PKI fingerprint → user_id binding. Soft delete. | Medium |
| **IFD component architecture** | Strong. 4 vault components (manager, preview, editor, settings) with clean event-based communication. | High |

### What Needs Architecture Work

| Area | Gap | Risk | Phase |
|------|-----|------|-------|
| **Data Room entity** | No room abstraction. Vault = personal storage, not a shared room. | High — this IS the product gap | 3 |
| **Invite & onboarding flow** | No invite mechanism. Users must be manually created. | High — blocks user acquisition | 3 |
| **Folder-level ACL** | ACL is vault-level only. No subtree scoping. | Medium — needed for compartmented access | 3 |
| **Audit trail** | No event logging beyond transfer metadata events. | High — compliance blocker | 3 |
| **Key hierarchy** | Flat: one AES key per file, wrapped per recipient. No room key → folder key → file key derivation. | Medium — works now, doesn't scale | 3 |
| **Session/auth for web UI** | Token-only auth. No JWT sessions, no cookie-based auth for browser flows. | Medium — acceptable for admin, not for data room recipients | 3 |
| **Short code delivery** | No short code generation, validation, or self-destruct. | Medium — blocks receiver UX brief (v0.5.24) | 3 |

## Architectural Decisions Needed for Phase 3

### Decision 1: Data Room = Named Vault or New Entity?

**Option A: Data Room IS a vault with metadata overlay.**
- Pros: Reuses all vault infrastructure (CRUD, encryption, ACL, UI). Minimal new code.
- Cons: Vault was designed for personal use. Room semantics (members, invites, activity) bolted on.
- Assessment: **Recommended for MVP.** A data room starts as a vault owned by the creator, shared via ACL.

**Option B: Data Room is a new first-class entity with its own storage.**
- Pros: Clean separation. Room has its own lifecycle, distinct from vault.
- Cons: Duplicates storage, encryption, and CRUD logic. Significant new code.
- Assessment: Consider for v1.0 if vault overlay proves limiting.

**Recommendation:** Option A for now. `Service__Data_Room.py` wraps `Service__Vault` + `Service__Vault__ACL` + adds room-specific metadata (name, description, status, member list, invite codes).

### Decision 2: Key Hierarchy

**Current:** Flat — one AES key per file, wrapped N times (once per recipient).

**Problem:** Adding a user to a room with 1000 files requires 1000 re-wrapping operations.

**Proposed hierarchy:**
```
room_master_key (AES-256)
  └─ wrapped per-member (RSA-OAEP)
  └─ used to derive folder_key = HKDF(room_key, folder_path)
     └─ used to encrypt file_key = random AES-256
```

**Benefit:** Adding a user = wrap room_master_key with their public key. One operation. All files accessible.

**Risk:** Single room key = if compromised, all files exposed. Mitigated by: key rotation on member removal.

**Recommendation:** Implement for Phase 3 MVP. Current per-file wrapping remains valid for direct transfers (Send flow). Room flow uses key hierarchy.

### Decision 3: Audit Trail Storage

**Options:**
- A. Same cache service (Memory-FS) — consistent with everything else
- B. Dedicated append-only log (S3 object per day, or DynamoDB stream)
- C. Both — cache for recent queries, S3 for immutable archive

**Recommendation:** Option A for MVP (keep it simple). The cache service already stores structured data. Add an `NS_AUDIT` namespace. Migrate to B/C when the audit trail is large enough to need it.

### Decision 4: Session Management for Data Room Recipients

**Current:** Token-based auth (access tokens created via admin). Recipients need a token to download.

**Problem:** Data room recipients shouldn't need admin tokens. They need a lightweight session.

**Proposed:**
1. Recipient enters short code → server validates → returns a session token (JWT or opaque)
2. Session token grants access to the specific room for a time window
3. Session token is scoped: room_id + permission_level + expiry
4. No central user account needed — session binds to PKI key if available

**Recommendation:** Implement as a room-scoped session token. Simpler than full JWT/OAuth. Compatible with progressive trust (short code → session → PKI).

## Phase 4 Architecture Notes (Forward-Looking)

PDF preview is architecturally simple — it's a client-side rendering problem. The vault already decrypts files in the browser; PDF.js just renders the decrypted bytes. No server changes needed.

**Key decision:** Bundle PDF.js (~400KB) or use browser-native `<iframe src="blob:...">` first. Browser-native is lighter but less reliable across browsers and on mobile.

**Recommendation:** Start with browser-native `<iframe>` + blob URL. Fall back to PDF.js only if browser-native fails in testing. This aligns with the "minimum necessary complexity" principle.

## Phase 5 Architecture Notes (Forward-Looking)

Enterprise features (watermarking, expiring access, download restrictions, Q&A, analytics) all depend on Phase 3 infrastructure:

| Feature | Depends On |
|---------|-----------|
| Watermarking | User identity (Phase 1) + preview renderer (Phase 4) |
| Expiring access | ACL with TTL field (Phase 3 extension) |
| Download restrictions | Permission model extension (view-only level) |
| Q&A | Room entity (Phase 3) + encrypted messaging |
| Analytics | Audit trail (Phase 3) + aggregation queries |

No architectural surprises. Phase 3 is the correct prerequisite.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Room key hierarchy adds complexity to encryption | Medium | High | Implement as separate code path; keep per-file wrapping for non-room flows |
| Invite link abuse (spam, scraping) | Medium | Medium | Rate limit, CAPTCHA-optional, self-destruct on N failures (per v0.5.24 brief) |
| Folder-level ACL creates permission explosion | Low | Medium | Max 3 override levels; inherit from parent by default |
| Audit trail grows unbounded | Low-Medium | Low | TTL on old entries; archive to S3 |
| Browser-native PDF fails on some platforms | Medium | Low | PDF.js fallback is well-understood |

---

## Summary

**The architecture is ready for Phase 3.** Phases 1 and 2 provide the foundation — user identity, ACL, multi-recipient encryption, large file transfer. Phase 3 is primarily **assembly** — combining existing building blocks into a new product entity (the data room) with invite flow, audit, and a purpose-built UI.

The 4 architectural decisions above (room as vault overlay, key hierarchy, audit storage, session tokens) should be resolved before implementation begins. All have clear recommended paths.

**Phase 3 is the product.** Everything before it was infrastructure. Everything after it is enhancement.
