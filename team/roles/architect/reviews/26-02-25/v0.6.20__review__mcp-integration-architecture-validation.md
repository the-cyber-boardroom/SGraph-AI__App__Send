## Summary

The MCP (Model Context Protocol) integration is now **validated end-to-end in production**. This review captures what the architecture looks like now that it's proven, what worked, what broke, and what it means for Phase 2.

## What Worked

### 1. Zero New Infrastructure
The MCP server mounts directly onto the existing FastAPI application. No new Lambda functions, no new endpoints (beyond the MCP mount), no new storage backends. The existing 7 API routes became 7 MCP tools automatically via `fastapi-mcp`.

### 2. HTTP Transport Over Lambda
The stateless HTTP transport (JSON-RPC over POST) works perfectly with Lambda's request-response model. No SSE, no WebSockets, no persistent connections needed. This was the key architectural bet and it paid off.

### 3. Zero-Knowledge Property Preserved
The critical security property survived the MCP integration:
- Claude encrypts content in its session context
- Only ciphertext reaches the server via MCP upload
- Decryption key travels in URL fragment (never sent to server)
- Browser decrypts locally with Web Crypto API

The MCP layer added no new attack surface for plaintext exposure.

### 4. 1:1 Route-to-Tool Mapping
Every existing API route maps to exactly one MCP tool. No translation layer, no impedance mismatch, no new abstractions. The MCP server is a thin projection of the existing API.

## What Broke (and How It Was Fixed)

### Issue 1: Tool Name Length (64-char limit)
**Problem:** Claude.ai enforces a 64-character limit on MCP tool names. The auto-generated names from `fastapi-mcp` exceeded this.
**Fix:** Shortened tool names at the MCP configuration level.
**Architectural Impact:** None — cosmetic change only.

### Issue 2: JSON Wrapper Stored Instead of Raw Bytes
**Problem:** MCP tool calls wrap upload content in a JSON envelope. The upload handler stored the entire envelope (including JSON metadata) instead of extracting the raw encrypted bytes.
**Fix:** Modified the upload handler to extract raw binary content from the MCP request before storage.
**Architectural Impact:** Low — the fix is in the request parsing layer, not the storage or encryption layers. This is the expected kind of integration issue when bridging JSON-RPC to binary storage.

### Issue 3: Base64 Encoding Needed
**Problem:** MCP operates over JSON, which can't carry raw binary. The encrypted payload needed base64 encoding for transport.
**Fix:** Added a base64 download endpoint and updated the MCP tools to handle base64-encoded content.
**Architectural Impact:** Minor — adds one encoding/decoding step in the MCP path. The standard HTTP path remains unchanged.

## Architectural Significance

### Validated Thesis: "MCP as Universal Agent Interface"
The 28-brief platform vision (23 Feb) positioned MCP as the interface layer for AI agent participation in the trust infrastructure. Today's working demo proves this is not theoretical. Any MCP-compatible AI can now:

1. Create encrypted transfers
2. Upload encrypted content
3. Generate secure download links
4. Check transfer status

### New Architecture Property: Multi-Client
SGraph Send now has three client types:
1. **Browser** — Human using the web UI directly
2. **API** — Programmatic access via HTTP
3. **MCP** — AI agent access via Model Context Protocol

All three share the same backend, same storage, same security model. The zero-knowledge property holds for all three.

### Lambda Budget Impact
The `fastapi-mcp` dependency adds ~17 packages (~11-22 MB). Current Lambda package is well within the 250 MB limit. No architectural concern, but monitor for Phase 2 additions.

## Phase 2 Recommendations

### Short Term (Next Sprint)
1. **Rate limiting on MCP endpoints** — AI agents can make tool calls much faster than humans click buttons
2. **MCP tool-level auth** — Currently uses the same access_token as HTTP. Consider MCP-specific scoping
3. **Observability** — Add MCP-specific request logging (tool name, latency, payload size)

### Medium Term (Platform)
4. **PGP tools via MCP** — Key lookup, signature verification (Brief B2)
5. **Multi-agent transfers** — Agent A encrypts, Agent B decrypts (requires key exchange protocol)
6. **Transfer metadata via MCP** — Expose metadata queries for agent-driven workflows

### Architectural Decisions Needed
7. **Should MCP tools expose Admin endpoints?** Currently User-only by design (AppSec review recommendation). Revisit if admin automation is requested.
8. **SSE transport for streaming?** Not needed for current tools, but may be useful for large file transfers or progress reporting.

---

## Conclusion

The MCP integration validates the core architectural bet: that SGraph Send's existing API design is clean enough to project directly into MCP tools with minimal adaptation. The zero-knowledge property, Lambda compatibility, and storage abstraction all held up under a new client type. The bugs encountered were integration-level (JSON encoding, name lengths) not architectural.

**Architecture status:** Sound. Ready for Phase 2 expansion.
