# Tech Spec: Deterministic Vault Pointers with Read/Write Key Separation

**Version:** v0.10.35
**Date:** 3 March 2026
**Role:** Architect
**Status:** Tech spec — **human-aligned** (decisions confirmed 3 March 2026)

---

## Executive Summary

The current vault key format `{passphrase}:{vault_id}:{settings_transfer_id}` breaks on every mutation because `_saveTree()` creates new transfers for both tree and settings. This produces a key that changes after every file add, rename, move, or delete — **8 operations trigger key mutation**. Every save also orphans 2 transfers (old tree + old settings), accumulating unbounded server-side waste.

This spec introduces **deterministic file IDs** derived from the vault key, a **write key** for overwrite authorization, and a **vault-prefixed storage namespace**. The vault key simplifies to `{passphrase}:{vault_id}` — stable forever.

---

## Problem Statement

### 1. Key Mutation Chain

```
User renames a file
 └── _saveTree()
      ├── Encrypt tree → upload as NEW transfer    → old tree orphaned
      ├── Update settings._tree_transfer_id
      ├── Encrypt settings → upload as NEW transfer → old settings orphaned
      └── _settingsTransferId changes
           └── vault key changes
                └── anyone with the old key sees a frozen snapshot
```

### 2. Orphan Accumulation

Each `_saveTree()` call creates **2 orphan transfers** (old tree + old settings). A normal session (5 file uploads, 3 renames, 2 folder creates) produces **20 orphan transfers**. The server has no way to identify or clean these up.

### 3. Silent Key Divergence

Old vault keys are silently valid — they decrypt successfully but show stale state. Two users with different key versions can unknowingly branch the vault's tree with no merge mechanism.

## Design Overview

### Key Insight

The Transfer API is append-only — every `upload()` creates a new random transfer ID. There is no update-in-place. To get a stable vault pointer, we need either:

- (a) A new mechanism to **overwrite** an existing transfer's payload, or
- (b) A new endpoint that accepts a **caller-specified transfer ID**

This spec proposes **(a)** — a new write endpoint that overwrites an existing file, authorized by a **write key** derived from the vault passphrase.

### Parallel Key Derivation from Vault Key

From `{passphrase}:{vault_id}`, the client derives **two independent keys** in parallel, plus deterministic file IDs:

```
passphrase + vault_id
    │
    ├──► PBKDF2(passphrase, "sg-vault-v1:{vault_id}",       600K iterations)
    │    └── read_key (32 bytes) ◄── the "encryption key" — encrypts/decrypts all content
    │         │
    │         ├── HMAC-SHA256(read_key, "sg-vault-v1:file-id:tree:{vault_id}")[:12]
    │         │   └── tree_file_id (deterministic, stable)
    │         │
    │         └── HMAC-SHA256(read_key, "sg-vault-v1:file-id:settings:{vault_id}")[:12]
    │             └── settings_file_id (deterministic, stable)
    │
    └──► PBKDF2(passphrase, "sg-vault-v1:write:{vault_id}", 600K iterations)
         └── write_key (32 bytes) ◄── submitted to server for write authorization
```

**Why parallel derivation?** Both keys derive from `passphrase + vault_id` via PBKDF2 but with different salts. This means:
- Knowing the `read_key` does **not** let you compute the `write_key` (and vice versa)
- Both keys have full PBKDF2 key-stretching (600K iterations) against brute-force
- This enables **future read-only sharing**: share the read_key without leaking write access

**Why HMAC for file IDs?** File IDs are derived from the read_key (fast, single HMAC). They are not secret — they're essentially URLs. The security is in the encryption, not in file ID secrecy.

| Derived value | Derivation | Purpose | Who computes it | Stored on server? |
|---|---|---|---|---|
| `read_key` | PBKDF2(passphrase, salt=`"sg-vault-v1:{vault_id}"`) | Encrypt/decrypt all vault content | Client only | Never |
| `write_key` | PBKDF2(passphrase, salt=`"sg-vault-v1:write:{vault_id}"`) | Authorize writes to vault files | Client computes, server verifies | SHA-256 hash only |
| `tree_file_id` | HMAC(read_key, domain string)[:12] | Stable pointer to vault tree | Client only | As the storage key |
| `settings_file_id` | HMAC(read_key, domain string)[:12] | Stable pointer to vault settings | Client only | As the storage key |

## Vault Key Format Change

**Before (v0.1.0):**
```
{passphrase}:{vault_id}:{settings_transfer_id}
```
- Changes on every mutation
- `settings_transfer_id` must be tracked and shared

**After (v0.1.1):**
```
{passphrase}:{vault_id}
```
- Stable forever
- `settings_transfer_id` is derived, not stored

**Parsing:** Split by `:` from the right. Last segment is `vault_id` (8 hex chars). Everything before it is the passphrase (which may contain colons).

## New API Endpoints

### `PUT /api/vault/{vault_id}/write/{file_id}` — Write / Overwrite

Overwrites the payload of a mutable transfer. Creates the transfer on first write.

**Request:**
```http
PUT /api/vault/{vault_id}/write/{file_id}
x-sgraph-send-access-token: {access_token}
x-sgraph-vault-write-key: {write_key_hex}
Content-Type: application/octet-stream

{encrypted_payload_bytes}
```

**Server behaviour:**

```
1. Validate access token (existing auth — required for all writes)

2. Check if mutable transfer exists at vault storage path:
   transfers/vault/{vault_id}/{file_id}/meta.json

3. IF NOT EXISTS (first write):
   a. Store write_key_hash = SHA-256(write_key_hex)
   b. Create meta.json with:
      - transfer_id: {file_id}
      - vault_id: {vault_id}
      - mutable: true
      - write_key_hash: {sha256_of_write_key}
      - status: "completed"
      - created_at: now
      - updated_at: now
      - write_count: 1
   c. Save payload to: transfers/vault/{vault_id}/{file_id}/payload

4. IF EXISTS (overwrite):
   a. Load existing meta.json
   b. Verify: SHA-256(write_key_hex) == stored write_key_hash
   c. If mismatch → 403 Forbidden
   d. Overwrite payload
   e. Update meta: updated_at = now, write_count += 1

5. Return: { "file_id": "{file_id}", "vault_id": "{vault_id}", "status": "completed" }
```

**Response (success):**
```json
{
    "file_id": "a1b2c3d4e5f6",
    "vault_id": "f7e8d9c0",
    "status": "completed",
    "write_count": 3
}
```

**Error responses:**
- `400` — missing write key header or empty body
- `401` — invalid or missing access token
- `403` — write key mismatch (someone without the vault key trying to overwrite)
- `413` — payload too large (Lambda limit)

### `GET /api/vault/{vault_id}/read/{file_id}` — Read

Downloads the payload of a vault transfer. **No auth required** (same as existing transfer download — zero-knowledge model means anyone with the file_id can download the ciphertext).

**Request:**
```http
GET /api/vault/{vault_id}/read/{file_id}
```

**Response:** Raw bytes (application/octet-stream)

**Why a separate read endpoint?** The vault namespace (`transfers/vault/{vault_id}/`) has a different storage path than regular transfers (`transfers/{transfer_id}/`). The existing `/api/transfers/download/{id}` doesn't know to look under the vault prefix. Rather than complicating the existing download route with vault-awareness, we add a clean read endpoint that routes to the correct storage path.

**Alternative (discussed, rejected):** Reuse `/api/transfers/download/{file_id}` with a fallback lookup. Rejected because it couples the vault storage model to the generic transfer model and adds per-request overhead for non-vault downloads.

### `DELETE /api/vault/{vault_id}/file/{file_id}` — Delete File

Deletes a single file from vault storage. Requires the vault write key.

**Request:**
```http
DELETE /api/vault/{vault_id}/file/{file_id}
x-sgraph-send-access-token: {access_token}
x-sgraph-vault-write-key: {write_key_hex}
```

**Server behaviour:**

```
1. Validate access token (existing auth)
2. Load vault meta for file at: transfers/vault/{vault_id}/{file_id}/meta.json
3. Verify: SHA-256(write_key_hex) == stored write_key_hash
4. If mismatch → 403 Forbidden
5. Delete: transfers/vault/{vault_id}/{file_id}/meta.json
6. Delete: transfers/vault/{vault_id}/{file_id}/payload
7. Return: { "file_id": "{file_id}", "vault_id": "{vault_id}", "status": "deleted" }
```

**Note on rename and move:** These are tree-level operations, not storage-level. The server stores opaque encrypted blobs — it has no concept of file names or folder structure. A rename or move is:
1. Client updates the file's path in the in-memory tree JSON
2. Client encrypts and overwrites the tree via `PUT /api/vault/{vid}/write/{tree_file_id}`

No dedicated rename/move endpoint is needed. The tree is the only place file names exist, and it's encrypted.

### Endpoint Summary

| Endpoint | Method | Auth | Purpose |
|---|---|---|---|
| `/api/vault/{vid}/write/{fid}` | PUT | Token + Write Key | Write or overwrite a vault file |
| `/api/vault/{vid}/read/{fid}` | GET | None | Download a vault file (encrypted) |
| `/api/vault/{vid}/file/{fid}` | DELETE | Token + Write Key | Delete a vault file |

**Vault deletion** is reserved for the Admin UI (out of scope for this spec). The admin can delete the entire S3 prefix `transfers/vault/{vault_id}/`.

## Storage Layout

### Current (v0.1.0) — Flat

```
transfers/
  {random_id_1}/meta.json + payload     ← vault settings (orphaned on mutation)
  {random_id_2}/meta.json + payload     ← vault tree (orphaned on mutation)
  {random_id_3}/meta.json + payload     ← file A
  {random_id_4}/meta.json + payload     ← file B
  {random_id_5}/meta.json + payload     ← regular (non-vault) transfer
```

No grouping. No way to identify which transfers belong to a vault. Orphans accumulate.

### New (v0.1.1) — Vault-Prefixed

```
transfers/                               ← regular transfers (unchanged)
  {random_id}/meta.json + payload

transfers/vault/                         ← vault namespace (new)
  {vault_id}/
    {tree_file_id}/                      ← deterministic, overwritten in-place
      meta.json
      payload
    {settings_file_id}/                  ← deterministic, overwritten in-place
      meta.json
      payload
    {file_id_1}/                         ← random (file uploads), append-only
      meta.json
      payload
    {file_id_2}/
      meta.json
      payload
```

**Benefits:**
- All vault data grouped under `transfers/vault/{vault_id}/`
- Delete a vault = delete the S3 prefix
- List vault contents = list the S3 prefix
- Per-vault quota/monitoring becomes trivial
- Clean separation from regular transfers
- No orphans for tree/settings (overwritten in-place)
- File transfer orphans (when files are replaced) are contained within the vault prefix

## Read and Write Flows

### Vault Open (Read Flow)

```
1. Parse vault key → passphrase, vault_id
2. Derive read_key  = PBKDF2(passphrase, "sg-vault-v1:{vault_id}")
3. Derive settings_file_id = HMAC(read_key, "sg-vault-v1:file-id:settings:{vault_id}")[:12]
4. Derive tree_file_id     = HMAC(read_key, "sg-vault-v1:file-id:tree:{vault_id}")[:12]
5. In parallel:
   a. GET /api/vault/{vault_id}/read/{settings_file_id} → decrypt → settings JSON
   b. GET /api/vault/{vault_id}/read/{tree_file_id}     → decrypt → tree JSON
6. Vault is open. All file_ids in the tree are downloadable via the same read endpoint.
```

**Note:** Steps 5a and 5b can be parallelized — both file IDs are deterministic and independent.

### Vault Save (Write Flow)

```
1. After mutation (add file, rename, move, delete, create folder):
2. Derive write_key = PBKDF2(passphrase, "sg-vault-v1:write:{vault_id}")
3. Serialize updated tree JSON → encrypt with read_key
4. PUT /api/vault/{vault_id}/write/{tree_file_id}
   Header: x-sgraph-vault-write-key: {write_key_hex}
   Body: encrypted tree bytes
   └── Server overwrites: transfers/vault/{vault_id}/{tree_file_id}/payload
5. Update settings._tree_file_id (should already match deterministic value)
6. Encrypt settings with read_key
7. PUT /api/vault/{vault_id}/write/{settings_file_id}
   Header: x-sgraph-vault-write-key: {write_key_hex}
   Body: encrypted settings bytes
8. Done. No new transfer IDs. No orphans. Key unchanged.
```

### File Upload (Within Vault)

```
1. Encrypt file data with read_key
2. PUT /api/vault/{vault_id}/write/{new_random_file_id}
   Header: x-sgraph-vault-write-key: {write_key_hex}
   Body: encrypted file bytes
   └── Server creates: transfers/vault/{vault_id}/{file_id}/payload
3. Add file entry to in-memory tree (with new file_id)
4. Save tree (write flow above)
```

### File Delete (Within Vault)

```
1. DELETE /api/vault/{vault_id}/file/{file_id}
   Header: x-sgraph-vault-write-key: {write_key_hex}
   └── Server deletes: transfers/vault/{vault_id}/{file_id}/
2. Remove file entry from in-memory tree
3. Save tree (write flow above)
```

### File Rename / Move (Tree-Only Operation)

```
1. Update file path in in-memory tree JSON (no server call needed)
2. Save tree (write flow above)
```

Rename and move do not touch the file's storage — the server stores opaque blobs. Only the tree (which the client encrypts) knows file names and paths.

**Note:** Individual file transfers use random IDs (not deterministic). The write key is the same for all files within a vault — it authorizes writes to any file under the vault's namespace.

### Comparison: Before vs After

| Aspect | Before (v0.1.0) | After (v0.1.1) |
|---|---|---|
| Vault key | `pass:id:CHANGES` | `pass:id` (stable) |
| Tree save | New transfer (orphans old) | Overwrite in-place |
| Settings save | New transfer (orphans old) | Overwrite in-place |
| Orphans per save | 2 | 0 (for tree/settings) |
| File upload | Creates transfer | Creates transfer (same) |
| Download | `GET /transfers/download/{id}` | `GET /vault/{vid}/read/{fid}` |
| Write auth | Access token only | Access token + write key |
| Storage path | `transfers/{id}/` | `transfers/vault/{vid}/{fid}/` |

## Server-Side Implementation

### New Service: `Service__Vault__Pointer`

```python
class Service__Vault__Pointer(Type_Safe):
    storage_fs : Storage_FS = None

    def vault_meta_path(self, vault_id, file_id):
        return f'transfers/vault/{vault_id}/{file_id}/meta.json'

    def vault_payload_path(self, vault_id, file_id):
        return f'transfers/vault/{vault_id}/{file_id}/payload'

    def write(self, vault_id, file_id, write_key_hex, payload_bytes):
        """Write or overwrite a vault file. Returns success dict or None on auth failure."""
        meta_path    = self.vault_meta_path(vault_id, file_id)
        payload_path = self.vault_payload_path(vault_id, file_id)
        submitted_hash = hashlib.sha256(write_key_hex.encode()).hexdigest()

        if self.storage_fs.file__exists(meta_path):
            # Overwrite — verify write key
            meta = self.storage_fs.file__json(meta_path)
            if meta.get('write_key_hash') != submitted_hash:
                return None  # Auth failure
            meta['updated_at']  = datetime.now(timezone.utc).isoformat()
            meta['write_count'] = meta.get('write_count', 0) + 1
        else:
            # First write — register
            meta = dict(
                file_id        = file_id,
                vault_id       = vault_id,
                mutable        = True,
                write_key_hash = submitted_hash,
                status         = 'completed',
                created_at     = datetime.now(timezone.utc).isoformat(),
                updated_at     = datetime.now(timezone.utc).isoformat(),
                write_count    = 1,
            )

        self.storage_fs.file__save(meta_path, json.dumps(meta).encode())
        self.storage_fs.file__save(payload_path, payload_bytes)
        return dict(file_id=file_id, vault_id=vault_id,
                    status='completed', write_count=meta['write_count'])

    def read(self, vault_id, file_id):
        """Read a vault file's payload. Returns bytes or None."""
        payload_path = self.vault_payload_path(vault_id, file_id)
        if not self.storage_fs.file__exists(payload_path):
            return None
        return self.storage_fs.file__bytes(payload_path)

    def delete(self, vault_id, file_id, write_key_hex):
        """Delete a vault file. Returns success dict or None on auth failure."""
        meta_path    = self.vault_meta_path(vault_id, file_id)
        payload_path = self.vault_payload_path(vault_id, file_id)

        if not self.storage_fs.file__exists(meta_path):
            return None  # File doesn't exist

        meta = self.storage_fs.file__json(meta_path)
        submitted_hash = hashlib.sha256(write_key_hex.encode()).hexdigest()
        if meta.get('write_key_hash') != submitted_hash:
            return None  # Auth failure

        self.storage_fs.file__delete(meta_path)
        self.storage_fs.file__delete(payload_path)
        return dict(file_id=file_id, vault_id=vault_id, status='deleted')
```

### New Routes: `Routes__Vault__Pointer`

Three endpoints added to the **User Lambda** (not Admin Lambda — preserving zero-knowledge model):

```python
@router.put("/api/vault/{vault_id}/write/{file_id}")
async def vault_write(vault_id: str, file_id: str, request: Request):
    write_key = request.headers.get("x-sgraph-vault-write-key")
    if not write_key:
        return JSONResponse(status_code=400, content={"error": "missing write key"})
    payload = await request.body()
    if not payload:
        return JSONResponse(status_code=400, content={"error": "empty payload"})
    result = service.write(vault_id, file_id, write_key, payload)
    if result is None:
        return JSONResponse(status_code=403, content={"error": "write key mismatch"})
    return result

@router.get("/api/vault/{vault_id}/read/{file_id}")
async def vault_read(vault_id: str, file_id: str):
    payload = service.read(vault_id, file_id)
    if payload is None:
        return Response(status_code=404)
    return Response(content=payload, media_type="application/octet-stream")

@router.delete("/api/vault/{vault_id}/file/{file_id}")
async def vault_delete(vault_id: str, file_id: str, request: Request):
    write_key = request.headers.get("x-sgraph-vault-write-key")
    if not write_key:
        return JSONResponse(status_code=400, content={"error": "missing write key"})
    result = service.delete(vault_id, file_id, write_key)
    if result is None:
        return JSONResponse(status_code=403, content={"error": "write key mismatch or file not found"})
    return result
```

## Client-Side Implementation

### JavaScript Key Derivation (sg-vault-crypto.js)

```javascript
class SGVaultCrypto {

    static async deriveKeys(passphrase, vaultId) {
        const encoder = new TextEncoder()
        const passphraseBytes = encoder.encode(passphrase)

        // Import passphrase as PBKDF2 key material (used for both derivations)
        const keyMaterial = await crypto.subtle.importKey(
            'raw', passphraseBytes, 'PBKDF2', false, ['deriveBits']
        )

        // 1. Derive read key (encryption key) — PBKDF2 with read salt
        const readSalt = encoder.encode(`sg-vault-v1:${vaultId}`)
        const readBits = await crypto.subtle.deriveBits(
            { name: 'PBKDF2', salt: readSalt, iterations: 600000, hash: 'SHA-256' },
            keyMaterial, 256
        )
        const readKey = await crypto.subtle.importKey(
            'raw', readBits, { name: 'AES-GCM' }, true, ['encrypt', 'decrypt']
        )

        // 2. Derive write key — PARALLEL PBKDF2 with write salt
        //    Independent from read_key: knowing read_key does NOT leak write_key
        const writeSalt = encoder.encode(`sg-vault-v1:write:${vaultId}`)
        const writeBits = await crypto.subtle.deriveBits(
            { name: 'PBKDF2', salt: writeSalt, iterations: 600000, hash: 'SHA-256' },
            keyMaterial, 256
        )
        const writeKey = Array.from(new Uint8Array(writeBits))
            .map(b => b.toString(16).padStart(2, '0')).join('')

        // 3. Derive deterministic file IDs via HMAC from read_key
        const hmacKey = await crypto.subtle.importKey(
            'raw', readBits, { name: 'HMAC', hash: 'SHA-256' }, false, ['sign']
        )
        const treeFileId = await this._deriveFileId(
            hmacKey, `sg-vault-v1:file-id:tree:${vaultId}`
        )
        const settingsFileId = await this._deriveFileId(
            hmacKey, `sg-vault-v1:file-id:settings:${vaultId}`
        )

        return { readKey, writeKey, treeFileId, settingsFileId }
    }

    static async _deriveFileId(hmacKey, input) {
        const buf = await crypto.subtle.sign(
            'HMAC', hmacKey, new TextEncoder().encode(input)
        )
        return Array.from(new Uint8Array(buf))
            .map(b => b.toString(16).padStart(2, '0')).join('').slice(0, 12)
    }
}
```

**Performance note:** Two PBKDF2 derivations (600K iterations each) run sequentially in Web Crypto. Total derivation time ~1–2 seconds on modern hardware. Both could be parallelized with `Promise.all` since they share the same `keyMaterial` but Web Crypto may not benefit from this (single-threaded). Acceptable for vault open/save operations.

### Updated SGVault Class Changes

**`SGVault.create()`** — instead of `sgSend.upload()`, call `PUT /api/vault/{vid}/write/{fid}` with the write key.

**`SGVault.open()`** — instead of parsing settings_transfer_id from the key, derive it. Then `GET /api/vault/{vid}/read/{fid}`.

**`SGVault._saveTree()`** — overwrite tree and settings in-place via the write endpoint. No new transfer IDs. `_settingsTransferId` and `_treeTransferId` never change.

**`SGVault.getVaultKey()`** — returns `{passphrase}:{vault_id}` (no settings_transfer_id).

**`SGVault.addFile()`** — upload the file via the vault write endpoint (with a random file_id), then save the tree.

## Security Analysis

### Write Key Security

| Property | Analysis |
|---|---|
| **Derivation** | Independent PBKDF2 from passphrase with distinct salt (`"sg-vault-v1:write:{vault_id}"`). Cannot be derived from read_key. |
| **Key stretching** | Full PBKDF2 (600K iterations). Same brute-force resistance as the read key. |
| **Server storage** | SHA-256 hash only. Server never sees the write key in cleartext at rest. |
| **Wire transmission** | Write key sent as hex in HTTP header. **Must use HTTPS** (already required). |
| **Brute force** | Attacker must brute-force the passphrase through 600K PBKDF2 iterations. Server-side rate limiting adds additional protection. |
| **Authorization scope** | One write key per vault. Authorizes writes to (and deletes from) any file under that vault's namespace. |
| **Independence from read_key** | Knowing the read_key does NOT let you compute the write_key. This enables future read-only sharing. |

### Zero-Knowledge Preservation

| Principle | Maintained? | Notes |
|---|---|---|
| Server never sees plaintext | Yes | All payloads are encrypted before upload |
| Server never sees encryption key | Yes | encryption_key never leaves the client |
| Server never sees file names | Yes | Only encrypted tree contains file names |
| Server never sees passphrase | Yes | Only key derivation products reach the server |
| Server can identify vault files | **Partially** | Server knows vault_id and can group files by vault. This is a **controlled trade-off**: the server knows "these files belong together" but cannot read them. |

**New server knowledge:** With vault-prefixed storage, the server knows that files under `transfers/vault/{vault_id}/` belong to the same vault. This is a deliberate trade-off — it enables vault management (deletion, quota, monitoring) at the cost of grouping metadata. The server still has **zero knowledge** of file contents, names, or structure.

### Replay Protection

The write key is static (derived deterministically). An attacker who intercepts a write request could replay it. However:
1. The payload is encrypted — replaying it only restores a previous valid state
2. The access token (existing auth) must also be valid
3. HTTPS prevents interception in transit
4. For stronger protection, a future version could add a nonce/timestamp to the write request

### Read-Only Sharing (Future Extension)

The parallel derivation design enables a future read-only sharing mode:
- Share the `read_key` (exported as base64) + `vault_id`
- Recipient can derive deterministic file IDs (via HMAC from read_key), download, and decrypt
- Recipient **cannot** compute the write_key — it requires the passphrase, not the read_key
- The write_key is derived from `passphrase + vault_id` via PBKDF2 — reversing a read_key back to the passphrase is computationally infeasible

This is out of scope for this spec but the architecture supports it natively because the two keys are derived in parallel, not sequentially.

## Migration Path: v0.1.0 → v0.1.1

**No backwards compatibility required.** v0.1.0 vaults have no active users — the format is superseded entirely by v0.1.1. There is no legacy open flow, no migration command, and no format detection logic. The only supported vault key format is:

```
{passphrase}:{vault_id}
```

## Decisions (Confirmed 3 March 2026)

| # | Question | Decision | Notes |
|---|---|---|---|
| D1 | Vault-scoped vs per-file write keys? | **Vault-scoped.** One write key provides write access to all deterministic files in the vault. | Simplicity wins. Per-file keys add complexity with no immediate benefit. |
| D2 | File uploads use write endpoint? | **Yes.** All vault files (metadata and user files) go through `PUT /api/vault/{vid}/write/{fid}` with the same write key. | Consistent auth model. One endpoint for all vault writes. |
| D3 | Vault deletion endpoint? | **Admin UI only.** No user-facing vault delete endpoint. User-facing operations: file delete, rename, move (all via write key). | Vault delete is a destructive admin operation. File delete is added as `DELETE /api/vault/{vid}/file/{fid}`. |
| D4 | vault_id in storage and API paths? | **Yes.** vault_id is not secret by itself. Brute-force protection via server-side rate limiting. | Defensive measures against web-based brute forcing will be added (rate limiting, IP-based throttling on write endpoints). |
| D5 | Write key derivation method? | **Parallel PBKDF2** from passphrase, NOT sequential HMAC from read_key. | Both keys derive from `passphrase + vault_id` independently. Knowing read_key does not leak write_key. Enables future read-only sharing. |

### Future: Brute-Force Protection

The vault_id appears in the API path and is not secret. To protect against automated key guessing:
- **Rate limiting** on write endpoints (per vault_id and per IP)
- **Exponential backoff** on 403 responses (write key mismatch)
- **IP-based throttling** on repeated failed write attempts
- **Optional: proof-of-work** challenge before write (future, if needed)

Read endpoints remain unprotected (zero-knowledge model — the ciphertext is useless without the key).
