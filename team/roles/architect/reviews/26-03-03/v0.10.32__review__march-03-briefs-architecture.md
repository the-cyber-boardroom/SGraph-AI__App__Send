## Executive Summary

Five briefs. One daily status, two vault-focused dev briefs, one new consumer product concept, and one major architecture brief that proposes a fundamentally new trust layer for the platform.

**The good news:** The vault fixes brief (Brief 2) and the gallery brief (Brief 3) fit cleanly within the existing architecture. They require no backend changes, no new trust models, and no new infrastructure. They are client-side work on proven primitives.

**The hard problems:** The vault CLI brief (Brief 4) introduces the first cross-language cryptographic interoperability requirement. The file safety orchestration brief (Brief 5) proposes the most significant architectural expansion since the pod concept was introduced -- a pluggable scanning chain with cryptographic attestation, cross-domain file security, and a new commercial model. Brief 5 is architecturally sound but represents 6-12 months of engineering across at least three major phases.

**Critical dependency across briefs:** The vault CLI (Brief 4), the gallery viewer (Brief 3), and the vault fixes (Brief 2) all target the same vault data model. Changes to the vault tree structure, encryption format, or metadata schema in any one brief affect the others. The Architect recommends treating these three briefs as a coordinated release, not independent work streams.

## Brief 1: Daily Brief -- Architectural Observations

The daily brief is primarily a status update, but it contains one architectural assertion that needs examination.

### The AWS Primitives Analogy

The brief draws an analogy: SG/Send's encrypted file storage is to SG/Send what S3 + EC2 is to AWS -- the foundation primitive.

**Architect's assessment: The analogy is directionally correct but architecturally incomplete.** AWS primitives are composable because they share a common identity model (IAM), a common networking model (VPC), and a common billing model (metered usage). SG/Send's primitives share an encryption model (AES-256-GCM) and a storage abstraction (Memory-FS), but they do not yet share:

- **A common identity model.** Transfers are anonymous (token-gated). Vaults use a passphrase-derived key. Data rooms use invite codes + session tokens. PKI uses RSA key pairs. These are four different identity mechanisms. For the "everything built on primitives" vision to hold, a unifying identity layer is needed. The vault CLI brief and the file safety brief both implicitly assume this -- the CLI needs to authenticate to the vault API, and the pod needs to verify who is sending and who is receiving.

- **A common metadata model.** Transfers carry SGMETA (filename envelope). Vaults carry vault-tree.json and vault-settings.json. Galleries (proposed) carry gallery.json. These are three different metadata formats for what is conceptually the same thing: "what files are in this collection and how are they organised?" Converging on a shared metadata envelope would reduce the surface area of each new "app on the primitives."

**Recommendation:** Before building more apps on the primitives, define a common metadata envelope and a common authentication model. This does not block any of the current briefs -- it is design work that pays off when the fourth and fifth "apps on primitives" arrive.

## Brief 2: Vault Fixes and UX Improvements -- Architecture Assessment

### What Fits Cleanly

All 10 items in this brief are client-side work. None require backend changes. The vault's architecture -- client-side tree management, server-stores-only-encrypted-blobs -- means UX improvements are entirely within the frontend domain. This is the intended payoff of the zero-knowledge architecture: the server is done; all evolution happens in the client.

| Item | Architecture Impact | Risk |
|------|-------------------|------|
| 1. Auth token only for upload | None -- client-side gate logic | Low |
| 2. Settings panel | None -- reads vault-settings.json already stored | Low |
| 3. Folder upload bug | None -- fix path propagation in vault-upload.js | Low |
| 4. Drag-and-drop | None -- browser APIs only | Low |
| 5. Folder management | Extends vault-tree.json schema (rename metadata) | Low |
| 6. File metadata banner | None -- reads existing file metadata | Low |
| 7. Preview polish | None -- leverages existing send.sgraph.ai modules | Low |
| 8. Inline editing | **Medium** -- introduces write-back to vault-tree.json from a new code path | Medium |
| 9. Local storage viewer | None -- debug tool | Low |
| 10. Chrome sync exploration | **Significant** -- cross-device key sync is a new capability with security implications | High |

### Items Requiring Architectural Attention

**Item 8 (Inline editing):** This is the first case where the vault UI writes content, not just uploads it. The current model is: user selects a file from their machine, the file is encrypted, the ciphertext is uploaded, the tree is updated. Inline editing introduces a second write path: user edits text in a textarea, the edited text is encoded to bytes, encrypted, uploaded as a new transfer, the tree is updated with the new transfer ID. This is architecturally sound -- it follows the existing pattern -- but the implementation must ensure that the old transfer (previous version) is not orphaned. The vault tree must either track version history (transfer IDs for previous versions) or explicitly delete the old transfer. Without this, every edit creates a new encrypted blob on the server and the old one persists indefinitely.

**Recommendation for Item 8:** Add a `previous_file_id` field to the vault tree file node. On edit-save, record the old transfer ID. This enables version history (future) and garbage collection (future) without requiring either to be built now. The cost is one additional field per file node.

**Item 10 (Chrome sync):** The brief correctly identifies this as exploratory. The Architect confirms: `chrome.storage.sync` is only available to Chrome extensions, not to web pages. A web page cannot access Chrome's sync storage without an extension. The options are:

1. **Chrome extension** -- stores vault keys in `chrome.storage.sync`, exposes them to `*.sgraph.ai` via `externally_connectable`. Works, but requires users to install an extension. The extension itself becomes a security-critical component (it holds vault keys).
2. **WebAuthn / Passkey** -- use the platform's credential sync (iCloud Keychain, Google Password Manager, Windows Hello) to store a vault-unlocking credential. This is the standards-based approach and works cross-browser, not just Chrome. But it stores a credential, not an arbitrary key. You would derive the vault key from the passkey assertion, which requires a server-side challenge-response flow -- not currently in the architecture.
3. **Browser local storage + manual export** -- the simplest option. Store vault keys in localStorage, provide an export/import mechanism (QR code, `.sg_vault` file). No cross-device magic, but no extension dependency either.

**Recommendation for Item 10:** Option 3 (export/import) for v0.1.1. Option 2 (WebAuthn/Passkey) as the long-term standard. Option 1 (Chrome extension) only if customer demand specifically requests Chrome-only sync. Do not build the extension speculatively.

### Schema Impact

The vault-tree.json schema should be extended minimally for v0.1.1:

```javascript
// Proposed additions to file nodes:
{
  type: "file",
  file_id: "transfer_id",        // existing
  size: N,                        // existing
  uploaded: "ISO-timestamp",      // existing
  name: "display-name.ext",       // NEW: editable display name (rename support)
  previous_file_id: null,         // NEW: version tracking (inline editing)
  metadata: {}                    // NEW: arbitrary key-value pairs (file metadata banner, future)
}

// Proposed additions to folder nodes:
{
  type: "folder",
  children: { ... },              // existing
  name: "display-name",           // NEW: editable display name (rename support)
  created: "ISO-timestamp"        // NEW: folder creation timestamp
}
```

**Critical constraint:** Any schema changes must be backwards-compatible. A vault created with v0.1.0 must still open in v0.1.1. New fields must be optional with sensible defaults.

## Brief 3: Encrypted Gallery Sharing -- Architecture Assessment

### What Fits Cleanly

The gallery is architecturally elegant because it requires **zero backend changes**. The gallery is a new client-side view of an existing data type (a zip file containing images). The flow is:

1. User uploads a folder of images (existing: folder-to-zip, already shipped)
2. Zip is encrypted and uploaded as a single transfer (existing: transfer pipeline)
3. Recipient downloads and decrypts the zip (existing: download pipeline)
4. **NEW:** Instead of showing a zip file browser, show a gallery view

Steps 1-3 are the existing architecture, unchanged. Step 4 is a new Web Component that processes the decrypted zip in memory and renders a gallery UI. No new endpoints. No new storage. No new encryption. This is exactly the kind of feature that the zero-knowledge architecture was designed to enable.

### Architectural Concerns

**Concern 1: Memory pressure at 100+ files.**

A gallery of 100 images at 5MB each = 500MB zip. The brief correctly identifies lazy loading as the solution, but the implementation details matter:

- The entire encrypted zip must be downloaded before any image can be displayed (encryption prevents random access into the ciphertext). This means a 500MB download before the first thumbnail appears.
- After decryption, JSZip can access the zip index without extracting all files. Thumbnails can be generated on demand. This is correct.
- But generating a thumbnail requires extracting the full-resolution image into memory, rendering it to a canvas, and capturing the thumbnail. For 100 images at 5MB each, doing this for all visible thumbnails simultaneously could consume 500MB+ of browser memory.

**Mitigation strategy:** Generate thumbnails in a worker thread (Web Worker or OffscreenCanvas), one at a time, and cache the thumbnails as blob URLs. Evict thumbnails for off-screen images using an LRU cache. Target: no more than 20 full-resolution images in memory simultaneously. This keeps memory pressure under 100MB for typical photo sizes.

**Concern 2: The gallery.json metadata file creates a format commitment.**

Once gallery.json ships and users create galleries with it, the format is a contract. The brief proposes:

```json
{
  "title": "...",
  "description": "...",
  "created": "ISO-timestamp",
  "items": [{ "filename": "...", "caption": "...", "type": "image|video", "order": N }],
  "sort": "exif_date"
}
```

**Recommendation:** Add a `version` field (start at `1`). This costs nothing and enables backwards-compatible evolution. Also add a `generator` field (e.g., `"sgraph-gallery/0.1.0"`) so the client can detect galleries created by older or newer versions.

```json
{
  "version": 1,
  "generator": "sgraph-gallery/0.1.0",
  "title": "...",
  ...
}
```

**Concern 3: Gallery as a single transfer vs. gallery as a vault.**

The brief describes the gallery as a single encrypted zip uploaded via the transfer pipeline. This works for the sharing use case (sender creates, recipients view). But Phase 4 (collaborative gallery) describes multiple contributors adding photos -- which requires the vault model (a collection of individually addressable encrypted files with a shared key), not the transfer model (a single encrypted blob).

**This is a design fork.** The gallery has two natural evolutionary paths:

- **Path A: Gallery = zip in a transfer.** Simple, works now, great for one-shot sharing. But collaborative editing means re-uploading the entire zip every time someone adds a photo. Does not scale.
- **Path B: Gallery = folder in a vault.** Each photo is an individual encrypted file. The gallery.json is a vault-tree.json extension. Adding a photo is adding a file to the vault. Scales to collaboration. But requires the recipient to have vault access, not just a download link.

**Recommendation:** Build Phase 1 (MVP) as Path A (zip in a transfer). It works, it ships fast, it validates the UX. But design the gallery.json format to be compatible with Path B. Specifically: the `items` array should use `file_id` references (nullable for zip-based galleries) so that when the gallery evolves to vault-backed storage, each item can reference its own encrypted file.

```json
{
  "version": 1,
  "items": [
    {
      "filename": "IMG_4521.jpg",
      "file_id": null,          // null = file is in the zip
      "caption": "...",
      "type": "image",
      "order": 1
    }
  ]
}
```

When `file_id` is null, the gallery reads the file from the zip. When `file_id` is set, the gallery fetches the file individually from the vault. This enables a seamless transition from Phase 1 to Phase 4 without breaking existing galleries.

**Concern 4: Friendly encryption keys.**

The brief proposes human-readable keys like "blue-mountain-77" instead of hex strings. This is a UX improvement, but architecturally it is a key-encoding decision that must be interoperable with the CLI (Brief 4) and the vault (existing).

The current vault key is a passphrase string that is fed into PBKDF2-SHA256 with 600,000 iterations to derive the AES-256 key. A friendly key like "blue-mountain-77" works as a passphrase -- PBKDF2 accepts arbitrary strings.

**Security concern:** "blue-mountain-77" has approximately 32 bits of entropy (assuming a dictionary of 2048 words and 100 numbers). The current vault key is a base64url-encoded 256-bit key (44 characters, 256 bits of entropy). The friendly key is 224 bits weaker. PBKDF2 with 600k iterations adds approximately 20 bits of computational cost, so the effective security is approximately 52 bits -- insufficient for a serious attacker.

**Recommendation:** Offer friendly keys as an **optional alias** for sharing, but the underlying encryption key must remain a full-entropy 256-bit key. The friendly key can be a mnemonic encoding of the full key (like BIP39 for cryptocurrency) -- 24 words from a 2048-word dictionary encodes 264 bits, which is sufficient. Alternatively, the friendly key can be a lookup identifier that maps to the real key (stored in the recipient's browser after first entry), but this adds a server-side mapping that the zero-knowledge architecture currently avoids.

**Recommended approach:** Use a BIP39-style word list. "blue-mountain-sunset-77-river-falcon" (6 tokens) encodes approximately 66 bits -- still low. Better: provide friendly keys for the sharing UX but auto-copy the full base64url key to the clipboard. The human says "blue-mountain-77" verbally; the technical sharing mechanism uses the full key. This separates the UX layer from the security layer.

### Impact on Transfer Model

The gallery at 100+ files per upload does not change the transfer model -- it is still a single transfer (one zip). The credit consumption concern (100 files = 100 credits) is a billing question, not an architecture question. The backend sees one upload of one encrypted blob. Whether the blob contains 1 file or 100 files is invisible to the server.

**If the billing model charges per-file rather than per-transfer:** the gallery zip must be counted differently. Either the upload UI reports the file count to the server (leaking metadata about the encrypted content) or the billing model counts transfers (not files). The Architect strongly recommends billing per-transfer for zero-knowledge consistency. The server should not know what is inside the encrypted blob.

## Brief 4: Vault CLI and Local Sync -- Architecture Assessment

### The Core Challenge: Cryptographic Interoperability

This is the brief's hardest architectural requirement and the one most likely to cause subtle, difficult-to-debug failures.

The vault CLI uses Python's `cryptography` library. The vault browser UI uses Web Crypto API. Both must produce identical ciphertext from the same inputs, and each must decrypt the other's output.

**The interoperability contract has five parameters:**

| Parameter | Browser (Web Crypto) | Python (cryptography) | Match? |
|-----------|---------------------|----------------------|--------|
| Algorithm | AES-256-GCM | AESGCM (256-bit) | Yes |
| IV length | 12 bytes (random) | 12 bytes (random) | Yes |
| Wire format | `[12-byte IV][ciphertext + 16-byte auth tag]` | Must match exactly | **Needs verification** |
| KDF | PBKDF2-SHA256, 600,000 iterations | PBKDF2HMAC(SHA256), must match iteration count | **Critical** |
| Salt | `enc.encode(salt)` -- UTF-8 encode of a string | Must use identical encoding | **Critical** |

**Where it can go wrong:**

1. **Auth tag placement.** Web Crypto API appends the 16-byte GCM authentication tag to the ciphertext. Python's `AESGCM.encrypt()` also appends the tag. This matches. But some Python examples separate the tag from the ciphertext. The CLI must not do this.

2. **KDF iteration count.** The brief's code sample uses 100,000 iterations. The browser uses 600,000. **This is a bug in the brief.** The Python code must use 600,000 iterations to match the browser. If it uses 100,000, the derived key will be different and all decryption will fail silently (wrong key = AES-GCM authentication failure).

3. **Salt encoding.** The browser uses `new TextEncoder().encode(salt)` which produces UTF-8 bytes. The Python code must use `salt.encode('utf-8')`. This will match for ASCII strings. For non-ASCII vault names used as salts, both must agree on the UTF-8 encoding. This is likely fine but must be tested with non-ASCII characters.

4. **Passphrase encoding.** Same concern. The browser encodes the passphrase with `TextEncoder` (UTF-8). Python must encode with `.encode('utf-8')`. Match for ASCII; test for non-ASCII.

5. **AESGCM additional data (AAD).** The browser passes `undefined` as the AAD parameter (no associated data). The Python code passes `None`. Both mean "no AAD." This matches.

**Mandatory test:** Before the CLI ships, create a cross-language test suite:

```
1. Browser encrypts a known file with a known passphrase → produce ciphertext
2. Export the ciphertext as a binary file
3. Python CLI decrypts the same ciphertext with the same passphrase → verify plaintext matches
4. Reverse: Python encrypts → browser decrypts
5. Test with: empty file, 1-byte file, 1MB file, file with non-ASCII content
6. Test with: ASCII passphrase, non-ASCII passphrase (Unicode), passphrase with emoji
```

**This test suite is the acceptance gate for the CLI. Without it, interoperability is assumed, not proven.**

### The Manifest and Sync Architecture

The manifest design is sound. One concern:

**Conflict resolution with "keep both" can corrupt the vault tree.** If User A and User B both modify the vault tree (e.g., both add a file to the same folder), the manifest conflict creates two `.conflict` copies of the local files. But the vault tree itself (vault-tree.json) is also modified by both users. The vault-tree.json is a single encrypted blob on the server. Last-writer-wins. There is no merging of tree structures.

**Scenario:**
1. User A syncs. Vault tree has files [a.txt, b.txt].
2. User B syncs from a different machine. Same tree.
3. User A adds `c.txt` locally and syncs. Tree becomes [a.txt, b.txt, c.txt].
4. User B adds `d.txt` locally and syncs. User B's tree was [a.txt, b.txt], becomes [a.txt, b.txt, d.txt]. User B uploads this tree, overwriting User A's tree. `c.txt` is now an orphan -- the encrypted blob exists on the server but no tree references it.

**This is the classic sync conflict on a shared mutable state.** The manifest tracks file-level changes, but the vault tree is a single document that must be merged, not overwritten.

**Recommendation:** For MVP, document this limitation clearly: "Vault CLI sync is designed for single-user or sequential-access patterns. Concurrent modification from multiple machines may result in files being unreferenced in the vault tree. Unreferenced files are not deleted and can be recovered." This is honest and avoids promising conflict resolution the architecture cannot deliver.

**For Phase 2:** Implement an optimistic concurrency control on the vault tree. The tree upload endpoint should accept a `previous_tree_id` parameter. If the server's current tree ID does not match, the upload is rejected with a conflict error. The CLI then downloads the current tree, merges (three-way merge of tree structures), and retries. This requires a server-side change (a conditional write on the vault tree endpoint) but is a small, well-understood addition.

### CLI as a New Attack Surface

The CLI stores the vault key on disk (in `.sg_vault/keyfile`). This is equivalent to storing a password in plaintext. The brief acknowledges this ("same security posture as browser local storage") but browser local storage is sandboxed per origin, while a file on disk is readable by any process running as the same user.

**Recommendation:** Default to `keyfile` being absent. Require the user to enter the vault key on each `sg-vault sync`. Offer `--save-key` as an explicit opt-in. Document the risk. Consider encrypting the keyfile with the system keychain (macOS Keychain, Linux Secret Service, Windows Credential Manager) via the `keyring` Python library. This provides OS-level protection without inventing a new security model.

### Dependency on Vault API

The CLI calls the vault API endpoints on the Admin Lambda (which requires authentication). The brief does not specify how the CLI authenticates. Options:

1. **Access token** -- the CLI stores an access token (same as the browser). This works but requires manual token management.
2. **API key** -- a new authentication mechanism for programmatic access. This is cleaner for CLI usage but requires a new auth system.
3. **Admin cookie** -- not viable for CLI; cookies are a browser mechanism.

**Recommendation:** Use access tokens for MVP. The user provides their access token during `sg-vault connect`. The token is stored in `.sg_vault/config`. This uses the existing authentication mechanism and requires no backend changes. Consider API keys as a Phase 2 improvement once CLI usage patterns are understood.

## Brief 5: File Safety Orchestration Architecture -- Deep Assessment

This is the most architecturally significant brief in the batch. It proposes a new trust layer (the pod as scanning intermediary), a new commercial model (receiver-defined scanning, sender-pays), and a cross-domain security architecture that positions SG/Send for government and defence markets.

### Architectural Verdict: Sound Design, Major Scope

The architecture is well-designed. The trust model is correct. The separation between the pod (sees cleartext transiently), the server (sees only ciphertext), and the crossing points (see only encrypted blobs and scan results) is clean and defensible. The attestation chain (pod signs scan results, receiver verifies) follows established patterns from code signing and certificate transparency.

**The Architect endorses the core design.** The concerns are about scope, sequencing, and the gap between what exists today and what the architecture requires.

### What Exists vs. What Is Required

| Capability | Status | Gap |
|-----------|--------|-----|
| AES-256-GCM encryption | Exists | None |
| PKI key generation (RSA-4096) | Exists (browser) | Needs server-side equivalent for pod |
| Key registry | Exists | Needs pod key publication |
| Vault with ACL | Exists | None |
| Data rooms with members | Exists | Needs scanning policy configuration |
| Transfer pipeline | Exists | Needs pod interception hook |
| Pod (trusted intermediary) | **PROPOSED -- does not exist** | Entire pod infrastructure |
| Scanning service adapters | **PROPOSED -- does not exist** | ClamAV adapter as first target |
| Attestation chain | **PROPOSED -- does not exist** | Signing, verification, storage |
| Receiver-defined requirements | **PROPOSED -- does not exist** | Requirement publishing, sender notification |
| Cross-domain crossing points | **PROPOSED -- does not exist** | Separate-key encryption for scan results |
| Time-bound keys | **PROPOSED -- does not exist** | Key escrow, rotation, deletion |
| Shamir secret sharing | **PROPOSED -- does not exist** | Quorum unlock mechanism |

**The Architect counts 9 major capabilities that do not exist.** Each is individually well-understood, but collectively they represent a new system layer.

### Trust Model Assessment

The brief's trust model is:

```
Sender: NOT trusted (may be malicious)
Server: Zero-knowledge (sees only ciphertext)
Pod: Trusted intermediary (sees cleartext transiently, signs attestation)
Crossing points: Semi-trusted (see scan results, not file content)
Receiver: Trusted (verifies attestation, decrypts file)
```

**This is correct and the Architect endorses it.** The key insight -- the sender is not trusted, therefore the sender must not compute the hash that binds the scan result to the file -- is the foundational security property. The brief consistently applies this principle throughout the design.

**One caveat:** The pod is a single point of trust. If the pod is compromised, a malicious pod can:
- Attest that scanning passed when it did not (forge attestation)
- Exfiltrate cleartext files (the pod sees them)
- Selectively fail scanning to block specific files

The brief's mitigations (Nitro Enclave for attestation, separate network for pod isolation, rate limiting, audit trail) are the correct response. But the Architect notes that **until Nitro Enclaves are deployed (Phase 3+ in the previous pod architecture review), the pod's trustworthiness is an operational claim, not a cryptographic guarantee.** This is acceptable for Phase 1 (ClamAV baseline) but must be communicated clearly to customers in higher tiers.

### The Pod Architecture: Alignment with Previous Reviews

The Architect reviewed the pod architecture in `v0.5.0__review__secure-pod-architecture.md` (21 Feb 2026). That review covered the pod as a multi-user re-encryption intermediary (for data rooms with revocable access). This brief extends the pod with a second responsibility: file scanning.

**These are compatible.** The pod's two roles can coexist:

1. **Re-encryption role:** Decrypt blob (pod key), re-encrypt for recipient (recipient's public key)
2. **Scanning role:** Decrypt blob (pod key), submit to scanning services, construct attestation, re-encrypt rebuilt file + attestation for recipient

The scanning role is an extension of Step 1 of the re-encryption role (after decryption, before re-encryption). The pod's API can expose both operations on the same infrastructure.

**Concern:** Combining both roles increases the pod's attack surface. The scanning role requires the pod to make outbound HTTP calls to scanning service APIs. The re-encryption role requires only inbound calls. Adding outbound connectivity increases the risk of the pod being used as an exfiltration channel if compromised.

**Mitigation:** Strict egress rules on the pod's network. Allowlist only the specific scanning service API endpoints. No general internet access. The pod's network policy should be: inbound from the main server only, outbound to scanning service endpoints only, no other connectivity. This is straightforward with VPC security groups / network policies.

### Cross-Domain Architecture: Assessment

The cross-domain design (Part 2 of the brief) is the most architecturally ambitious component. It proposes that SG/Send can replace or complement traditional cross-domain solutions (CDGs, data diodes) by ensuring the crossing point never sees plaintext.

**The Architect's assessment: The architecture is correct, but the positioning requires careful scoping.**

What the architecture actually delivers:
- The crossing point verifies scan results without seeing file content. **This is a genuine differentiator.** Traditional CDGs must see plaintext to scan. SG/Send separates scanning (pod) from crossing (infrastructure).
- The attestation chain is verifiable at each crossing point independently. **This is valuable.** Each crossing point can confirm that the pod processed the file, without trusting the previous crossing point.
- Compromise of a crossing point does not expose file content. **This is the strongest property.** The crossing point never has the file's decryption key.

What the architecture does NOT deliver:
- **Protocol-level cross-domain guarantee.** The brief describes the file being "forwarded through normal channels" after scanning. In a government cross-domain context, "normal channels" typically means a specific protocol (e.g., DIIS, CDS-specific protocols) running on accredited hardware. SG/Send's HTTPS transport is not automatically accepted in these environments.
- **Air-gapped operation without a network.** The brief mentions `.sgroom` portable data rooms for air-gapped environments. This is a different architecture from the pod-based scanning chain (which requires network connectivity to the pod). The `.sgroom` model requires embedded scanning engines (Tier 3), which is Phase 5 in the roadmap (weeks 20-30).

**Recommendation:** Position the cross-domain capability as "zero-knowledge crossing point design" rather than "cross-domain solution." The former describes the architectural property accurately. The latter implies accreditation and protocol compliance that does not yet exist. Government and defence customers will expect specific accreditation (e.g., CAPS, CPA in the UK; NIAP in the US). SG/Send's architecture is compatible with these accreditation frameworks but is not itself accredited.

### Scanning Service Integration: Sequencing Assessment

The brief proposes six phases over 30+ weeks. The Architect's sequencing assessment:

| Phase | Brief's Estimate | Architect's Assessment | Dependencies |
|-------|-----------------|----------------------|-------------|
| Phase 1: ClamAV baseline | Weeks 1-4 | **Realistic if pod infrastructure exists.** If the pod must be built from scratch, add 3-4 weeks. Total: 5-8 weeks. | Pod infrastructure (does not exist) |
| Phase 2: First commercial CDR partner | Weeks 4-8 | **Optimistic.** Partner API access negotiation alone can take 2-4 weeks. Adapter development is 3-5 weeks as stated. Realistic: weeks 8-14. | Phase 1 complete, partner agreement |
| Phase 3: Data room policy enforcement | Weeks 8-14 | **Reasonable.** Builds on existing data room infrastructure. The admin UI for policy configuration is the main effort. | Phase 1 complete, data room ACL (exists) |
| Phase 4: Multi-service chaining | Weeks 14-20 | **Reasonable.** Once the adapter pattern is established, adding services is incremental. | Phase 2 complete |
| Phase 5: Embedded engines | Weeks 20-30 | **Underestimated for air-gapped/government.** Embedding a CDR engine requires the engine vendor to provide a deployable binary (not just an API). Licensing and deployment model negotiation for embedded engines is a separate business process. Realistic: 16-24 weeks from Phase 4. | Phase 4 complete, partner SDK access, licensing |

**Total realistic timeline:** 30-50 weeks from the start of pod infrastructure work. This is 7-12 months. The brief's 30-week estimate is the optimistic case.

### Receiver-Defined Scanning: Commercial Model Assessment

The receiver-defined, sender-pays model is commercially novel. The Architect's assessment:

**Strength:** The model creates network effects. Every receiver who sets scanning requirements generates revenue for every file sent to them. As the user base grows, scanning revenue grows superlinearly.

**Weakness:** The model introduces **friction at the sender's decision point.** The sender is about to upload a file. They are told: "This recipient requires CDR + AV scanning. Estimated cost: 0.15 GBP. Proceed?" This is a conversion-killing moment for a product that currently has zero friction beyond "enter your access token." The brief acknowledges this risk (the "scanning creates friction" entry in the risk table) but the mitigation ("scanning is optional") undermines the receiver-defined model.

**Architectural recommendation:** Make the Basic tier (ClamAV) free and automatic for all transfers. The sender is never asked; scanning happens invisibly. The attestation is attached to every transfer. This establishes scanning as infrastructure, not a tollgate. Premium tiers (CDR, DLP) are opt-in by the receiver and disclosed to the sender with cost. This preserves the frictionless default while enabling the premium model.

### Key Architectural Risks in Brief 5

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Pod infrastructure does not exist | High | Must be built before any scanning work begins. Estimated 3-4 weeks for MVP pod (per v0.5.0 review). |
| Scanning adds latency to the upload path | High | Async mode for all files. Immediate upload, scanning in background, attestation attached post-scan. Recipient sees "scanning in progress" until attestation is ready. |
| Partner data retention contradicts zero-knowledge | High | Contractual requirement: zero retention after scan. Verify with test (upload a canary file, check if the partner retains it). |
| Attestation format is a commitment | Medium | Version the attestation schema. Start with v1. Include `schema_version` in every attestation. |
| ClamAV detection rates are low | Medium | ClamAV is the baseline, not the ceiling. Position it as "free included scanning" and upsell commercial engines for higher assurance. |
| Pod sees cleartext | Inherent | This is the fundamental trade-off. Mitigation: Nitro Enclave (Phase 3), audit trail, rate limiting, network isolation. |

## Architectural Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| AD-40 | Vault-tree.json schema extensions must be backwards-compatible | v0.1.0 vaults must open in v0.1.1 |
| AD-41 | Gallery Phase 1 uses zip-in-transfer model, not vault model | Simpler, ships faster, validates UX before committing to vault-backed galleries |
| AD-42 | Gallery.json must include `version` and `generator` fields | Format commitment; versioning enables evolution |
| AD-43 | Gallery.json `items` should include nullable `file_id` | Enables future transition from zip-based to vault-backed galleries |
| AD-44 | Vault CLI encryption must be tested against browser encryption before any feature work | Interoperability is the highest technical risk in Brief 4 |
| AD-45 | Vault CLI PBKDF2 must use 600,000 iterations (not 100,000 as in the brief's code sample) | Brief contains a parameter mismatch; browser uses 600,000 |
| AD-46 | Vault CLI should default to no keyfile on disk; `--save-key` as explicit opt-in | Security by default; reduce attack surface of local key storage |
| AD-47 | File safety orchestration (Brief 5) should not start until pod infrastructure exists | Pod is the prerequisite for all scanning work; building adapters without a pod produces dead code |
| AD-48 | ClamAV (Basic tier) scanning should be free and automatic for all transfers | Removes friction from the sender path; establishes scanning as infrastructure |
| AD-49 | Friendly encryption keys must not reduce key entropy below 256 bits | Friendly names as aliases or BIP39-style mnemonics; never as the actual passphrase |
| AD-50 | Cross-domain capability should be positioned as "zero-knowledge crossing point design," not "cross-domain solution" | Architectural property is proven; accreditation is not yet achieved |

## Open Questions for Human Decision

1. **Gallery billing model.** Does a gallery of 100 photos consume 1 credit (one transfer) or 100 credits (one per photo)? The zero-knowledge architecture means the server does not know the zip contains photos. Billing per-transfer preserves zero-knowledge. Billing per-file leaks metadata. Which is it?

2. **Vault CLI authentication.** The brief does not specify how the CLI authenticates to the vault API. Access tokens (existing mechanism) or API keys (new mechanism)? Access tokens require no backend changes; API keys are more ergonomic for CLI usage but require new infrastructure.

3. **File safety orchestration timing.** Brief 5 is a 7-12 month programme. When does work begin? Is it gated on a specific customer signal (signed contract, LOI), or is it strategically pre-built to enable the government/defence positioning?

4. **Vault tree concurrency model.** Brief 4 (CLI sync) and the vault browser UI can both modify the vault tree simultaneously. Should the server enforce optimistic concurrency control (conditional writes with `If-Match` semantics) now, or accept last-writer-wins for MVP?

5. **Gallery evolution path.** When the gallery needs collaborative editing (Phase 4), does it migrate to a vault-backed model, or does the brief envision a different architecture? This decision affects how gallery.json is designed today.
