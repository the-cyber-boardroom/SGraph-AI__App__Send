## Executive Summary

The data room enhancements brief proposes six areas of work: in-browser file editing, share panel, parallel viewer modes, access analytics, deployment path fix, and shared module extraction. This assessment maps dependencies, identifies what already exists, flags risks, and proposes a phased implementation order.

**Key finding:** More exists than expected. Download analytics are already captured server-side. Deployment path isolation is already implemented. The sg-send.js library exists in the vault UI and needs extraction, not creation from scratch. The i18n system needs architectural improvement to support namespace-based loading for parallel viewers.

## 1. What Already Exists (Code-Verified)

### 1a. Download Analytics — Already Captured

The brief says "start capturing download counts NOW." **This is already done.**

`Transfer__Service.get_download_payload()` (line 119–133) already:
- Increments `download_count` on every download
- Logs a download event with timestamp, IP hash, and user agent
- Persists to storage via `save_meta()`

The `Schema__Transfer` already has `download_count: Safe_UInt`.

`Transfer__Service.get_transfer_info()` (line 105–114) already returns download count to callers.

**What's missing:** A UI to display these analytics to senders, and possibly a dedicated analytics API endpoint that returns the full event history (not just the count).

### 1b. Deployment Path Isolation — Already Implemented

The brief says "deploy to environment-specific paths." **This is already done.**

`deploy-ui-user.yml` (lines 11–14) already deploys to:
```
websites/sgraph-send/dev/    — dev branch
websites/sgraph-send/main/   — main branch
websites/sgraph-send/prod/   — production
```

Three environments are fully configured with separate CloudFront distributions (dev, main, production).

**What's missing:** Path-scoped CI/CD (vault changes triggering send rebuild). The current workflow is already path-scoped to `sgraph_ai_app_send__ui__user/v0/v0.2/**` — this is done. But the Lambda CI pipeline (`ci-pipeline.yml`) may still fire on all changes.

### 1c. sg-send.js — Exists in Vault UI

`sgraph_ai_app_send__ui__vault/v0/v0.1/v0.1.1/_common/js/lib/sg-send/sg-send.js` (108 lines) provides:
- `upload(data, options)` — 3-step transfer flow (create → upload → complete)
- `download(transferId)` — fetch encrypted blob
- `info(transferId)` — transfer metadata
- `encryptAndUpload(plainData, key)` — convenience method
- `downloadAndDecrypt(transferId, key)` — convenience method

`sg-send-crypto.js` (151 lines) provides:
- AES-256-GCM encrypt/decrypt
- Key generation, import, export
- PBKDF2 key derivation
- Base64url encoding/decoding

**What's needed:** Extract these to a shared location. Add zip parsing, file extraction, content type detection, and thumbnail generation (currently embedded in `send-download.js`).

### 1d. i18n System — Exists, Needs Restructuring

The current `i18n.js` has ~200 English string keys inline, with 16 locales supported via URL-based routing. Non-English translations are loaded from JSON files at `i18n/{locale}.json`. All keys are in a single monolithic file.

**Problem for parallel viewers:** Each new viewer will add its own translation keys. A translator agent working on `/gallery/` i18n shouldn't need to load the entire i18n JSON (200+ keys growing to 400+). This becomes worse as viewers multiply.

## 3. Architectural Concerns

### 3.1 Shared Library Location

The brief proposes `shared/sg-send.js` at the deployment root. The vault UI already has `sg-send.js` + `sg-send-crypto.js` at a versioned IFD path.

**Recommendation:** Create a shared library at `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/_common/js/lib/sg-send/` (mirroring the vault structure). Each viewer imports via a relative path to `_common/`. This keeps IFD versioning intact and avoids a separate deployment step.

```
sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/
  _common/
    js/
      lib/
        sg-send/
          sg-send.js          ← API client (from vault, extended)
          sg-send-crypto.js   ← Crypto (from vault, unchanged)
          sg-send-zip.js      ← Zip parse/extract (extracted from send-download.js)
          sg-send-detect.js   ← Content type detection (from file-type-detect.js)
      i18n.js                 ← Core i18n module (keep as-is for backward compat)
      i18n/
        namespaces/           ← NEW: per-feature translation files
  en-gb/
    index.html                ← Current upload page (NEVER TOUCH)
    download.html             ← Current download page (NEVER TOUCH)
  room/
    index.html                ← Data room viewer entry point
  gallery/
    index.html                ← Gallery viewer entry point
  pdf/
    index.html                ← PDF viewer entry point
  video/
    index.html                ← Video viewer entry point
  view/
    index.html                ← Auto-detect router entry point
```

### 3.2 Viewer Entry Points

Each viewer is a standalone HTML page that:
1. Loads `_common/js/lib/sg-send/sg-send.js` and `sg-send-crypto.js`
2. Loads `_common/js/i18n.js` (core module)
3. Loads its own namespace translations
4. Extracts transfer ID from the URL path and decryption key from the hash fragment
5. Fetches, decrypts, and renders content using its own Web Component

**Critical:** The existing `#/download/{token}` hash-based URL scheme in the current UI must never break. New viewers use clean path-based URLs: `/room/{token}#key`.

### 3.3 Feature Toggle MVP

The brief proposes file-presence as the toggle mechanism. This aligns perfectly with the existing deployment model — if `room/index.html` exists in the deployed S3 path, the viewer is active. If not, it 404s.

**For dev:** Deploy all viewers. **For prod:** Only deploy validated viewers.

Implementation: Add a `VIEWERS` matrix variable to `deploy-ui-user.yml` that lists which viewer folders to include in each environment's sync.

### 3.4 URL Routing

New viewer URLs need CloudFront routing. Currently CloudFront routes `/api/*` to Lambda and `/*` to S3. New viewer paths (`/room/`, `/gallery/`, etc.) are already handled by the S3 catch-all.

**Issue:** Each viewer URL like `/room/abc123#key` needs to resolve to `room/index.html` regardless of the token path segment. This requires either:
- **Option A:** CloudFront function to rewrite `/room/*` → `/room/index.html` (recommended)
- **Option B:** Client-side routing in `room/index.html` that reads the URL

**Recommendation:** Option A for clean URLs. A CloudFront Function (not Lambda@Edge — these are free-tier, sub-ms) rewrites `/room/*`, `/gallery/*`, `/pdf/*`, `/video/*`, `/view/*` to their respective `index.html`.

### 3.5 Zip Extraction from send-download.js

The current `send-download.js` (1090 lines) contains embedded zip parsing via JSZip, file type detection, and rendering logic. For parallel viewers, this needs to be split:

| Extract To | Functions | Lines (approx) |
|---|---|---|
| `sg-send-zip.js` | `_decryptAndParseZip()`, zip metadata, file extraction | ~80 |
| `sg-send-detect.js` | `FileTypeDetect.detect()`, MIME mapping, extension mapping | ~60 (already separate file) |
| `sg-send-render.js` (optional) | Shared rendering helpers (escape, blob URLs, sandbox) | ~40 |

The renderers (markdown, image, PDF, code, audio, video, SVG) stay in `send-download.js` for the current UI. New viewers implement their own rendering.

### 3.6 Backend: Analytics API Enhancement

For the analytics UI, we need an endpoint that returns the full event log, not just the count. Currently `get_transfer_info()` returns `download_count` but not the `events` array.

**Recommendation:** Add a `GET /transfers/analytics/{id}` endpoint that returns:
```json
{
  "transfer_id": "abc123",
  "download_count": 5,
  "file_size_bytes": 1048576,
  "created_at": "2026-03-03T10:00:00Z",
  "total_bandwidth_bytes": 5242880,
  "events": [
    { "action": "download", "timestamp": "...", "ip_hash": "...", "user_agent": "..." },
    ...
  ]
}
```

This is a small addition — the data already exists in the meta JSON. The endpoint just exposes it (auth-gated so only the sender sees it).

### 3.7 Security: Viewer Isolation

Per the AppSec post-removal review (`team/roles/appsec/reviews/26-03-03/v0.10.19__review__post-removal-office-viewers-and-print.md`), the current security model is:
- No `allow-scripts` in any iframe sandbox
- All user strings escaped via `escapeHtml()`
- No CDN dependencies
- Blob URLs for same-origin content

**Each new viewer must maintain these properties.** The parallel deployment model naturally achieves blast-radius isolation — a bug in `/gallery/` cannot affect `/room/` or the current UI. But each viewer must independently follow the sandbox and escape patterns.

**Recommendation:** Include security checklist in the viewer template:
- [ ] No `allow-scripts` in iframes
- [ ] All user-controlled strings escaped
- [ ] No CDN script loading
- [ ] Blob URLs with `revokeObjectURL` on cleanup

## 4. i18n Architecture: Namespace-Based Splitting

### Problem

The current `i18n.js` has ~200 English keys in a single inline object. Translation JSON files (`i18n/{locale}.json`) contain all keys for all features. When a translator agent works on gallery viewer translations, it loads the entire file (200+ keys) even though it only needs ~15 gallery-specific keys.

As viewers multiply, this monolithic approach means:
- Translator agents load increasingly large files for small changes
- Risk of accidental changes to unrelated keys
- Merge conflicts when multiple translators work in parallel
- No way to scope translation review to a single feature

### Proposed Architecture

Split translations into **namespaces** — one JSON file per feature per locale:

```
i18n/
  namespaces/
    core/           ← Brand, common, crypto errors (~30 keys)
      en-gb.json
      de-de.json
      ...
    upload/         ← Upload component keys (~42 keys)
      en-gb.json
      ...
    download/       ← Download component keys (~35 keys)
      en-gb.json
      ...
    access-gate/    ← Access gate keys (~12 keys)
      en-gb.json
      ...
    room/           ← Data room viewer keys (~15 keys, NEW)
      en-gb.json
      ...
    gallery/        ← Gallery viewer keys (~15 keys, NEW)
      en-gb.json
      ...
    pdf/            ← PDF viewer keys (~10 keys, NEW)
      en-gb.json
      ...
    video/          ← Video viewer keys (~10 keys, NEW)
      en-gb.json
      ...
```

### Loading Mechanism

Each page declares which namespaces it needs:

```html
<!-- room/index.html -->
<script>
  window.__I18N_NAMESPACES = ['core', 'room'];
</script>
<script src="../_common/js/i18n.js"></script>
```

The `I18n.init()` method loads only the requested namespaces:

```javascript
async init() {
    this.locale = this._detectLocale();
    const namespaces = window.__I18N_NAMESPACES || ['core', 'upload', 'download', 'access-gate'];

    if (this.locale !== 'en-gb') {
        await Promise.all(
            namespaces.map(ns => this._loadNamespaceStrings(ns, this.locale))
        );
    }
    // ... existing DOM update logic
}

async _loadNamespaceStrings(namespace, locale) {
    const basePath = /* ... */;
    const jsonUrl = `${basePath}/i18n/namespaces/${namespace}/${locale}.json`;
    const response = await fetch(jsonUrl);
    if (response.ok) {
        const translations = await response.json();
        this.strings[locale] = { ...this.strings[locale], ...translations };
    }
}
```

### Migration Path

1. **Keep existing monolithic `i18n.js` inline strings** — they are the English fallback and remain working
2. **Split existing `i18n/{locale}.json` files** into namespace files using a build script
3. **Update `_loadLocaleStrings()`** to load from namespaces instead of a single file
4. **Existing pages declare their namespaces** via `window.__I18N_NAMESPACES`
5. **Default (no declaration)** loads all namespaces — backward compatible

### Benefits for Translator Agents

- Translator works on `i18n/namespaces/gallery/en-gb.json` (~15 keys) instead of the full 200+ key file
- Changes are scoped — gallery translations can't accidentally modify upload translations
- PR reviews are focused — "this PR only touches gallery i18n"
- Parallel translation work is conflict-free

## 6. Risk Register

| Risk | Severity | Mitigation |
|---|---|---|
| Breaking existing `#/download/{token}` URLs | **CRITICAL** | Never modify current index.html or download.html. All new work in separate paths. |
| CloudFront routing conflicts | MEDIUM | Test URL rewrite function in dev environment first. Current `/*` → S3 still works as fallback. |
| JSZip version mismatch between current UI and shared lib | MEDIUM | Pin JSZip version. Shared `sg-send-zip.js` uses same version as current `send-download.js`. |
| i18n migration breaks existing translations | MEDIUM | Keep inline English strings as fallback. Namespace loading is additive — missing namespace falls back to inline. |
| Viewer security regression | HIGH | AppSec review for each viewer before production toggle-on. Follow security checklist from section 3.7. |
| Scope creep in file editing | HIGH | Defer file editing to future sprint. Do not start until viewers are stable. |
