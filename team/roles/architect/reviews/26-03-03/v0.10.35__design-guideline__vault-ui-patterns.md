# v0.10.35 Design Guideline — Vault UI Patterns

**Version:** v0.10.35
**Date:** 2026-03-03
**Role:** Architect + Dev
**Scope:** SG/Vault UI (sgraph_ai_app_send__ui__vault)

---

## 1. Avoid Modals Unless Truly Necessary

**Guideline:** Use inline panels, top-level views, or expanding sections instead of modal dialogs.

**When modals ARE acceptable:**
- Destructive confirmations (delete, overwrite, revoke access)
- Authentication gates (access key prompt — one-time per session)
- Critical warnings that require acknowledgement

**When modals are NOT acceptable:**
- Settings panels (use a top-level view)
- File/folder creation (use inline input)
- Rename operations (use inline editing)
- Preview/detail panels (use side panel or main area)
- Configuration or preferences (use a dedicated view)

**Rationale:** Modal dialogs break flow, hide context, and feel heavy on repeated use. Inline patterns keep the user in context and reduce click count.

**Applied in v0.1.3:** Settings moved from modal overlay to a top-level nav view. Folder creation and rename were already inline (v0.1.2). Auth token prompt remains as a modal (gating a destructive action).

---

## 2. Show Visual Loading Feedback for All API Requests

**Guideline:** Every network request must produce visible feedback so the user knows something is happening.

**Implementation:** A thin animated loading bar sits at the bottom of the header. It appears automatically whenever `fetch()` is called and disappears when all pending requests complete.

**Why this matters:**
- Lambda cold starts can take 3-10 seconds
- Users clicking buttons with no feedback will click again (or assume it's broken)
- A loading bar is non-intrusive but clearly communicates "working"

**Technical details:**
- Global `fetch()` wrapper increments/decrements a counter
- Counter-based (not boolean) to handle concurrent requests correctly
- Individual operations can also call `_showLoading`/`_hideLoading` for coverage during non-fetch work (encryption, blob creation)

---

## 3. Navigation Architecture

**Guideline:** The vault UI uses a left-side icon navigation panel (56px wide) for top-level view switching.

**Current views:**
- **Files** (default) — tree sidebar + file browser + preview
- **Settings** — vault configuration, keys, stats, raw JSON

**Future views (planned):**
- **Activity** — audit trail, recent actions
- **Sharing** — access control, shared links
- **Gallery** — media browser for image/video files

**Pattern:** Same as admin console — nav items are icon + label, vertical stack, with active indicator (left border + highlight). The tree sidebar is only visible in the Files view.

---

## 4. Dual Version Display

**Guideline:** The header shows both the vault UI version and the app/server version.

**Format:** `v0.1.3 / v0.10.35` (vault UI / app version)

**Rationale:** During development, it's essential to verify which version of the UI is deployed and which version of the backend it's talking to. The app version is fetched from `/api/health` on load.
