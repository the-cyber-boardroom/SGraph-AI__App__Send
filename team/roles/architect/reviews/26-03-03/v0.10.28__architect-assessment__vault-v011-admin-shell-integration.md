## Executive Summary

The admin shell **can** serve as the base for vault v0.1.1, but through **pattern extraction** rather than code inheritance. The admin shell (v0.1.3) is well-architected with clean separation between layout, routing, debug infrastructure, and business components. The vault should copy its design patterns (3-column grid, EventBus, debug panels) into a new `vault-shell` component. Five modules from send.sgraph.ai can be imported directly; three need forking; two need utility extraction.

**Recommended approach:** Option B from the brief (shared module publishing) with a pragmatic Phase 1 using direct file copying, evolving to CDN-hosted shared modules.

## Question 1: How Cleanly Can the Admin Shell Be Extracted?

### Answer: Clean extraction via pattern copying — NOT inheritance

The admin shell (`v0.1.3/components/admin-shell/admin-shell.js`, 867 lines) is a single Web Component that provides:

| Feature | Lines | Extractable? |
|---------|-------|-------------|
| CSS Grid layout (3-column + header + statusbar) | 490–510 | Yes — copy template |
| Left sidebar navigation with auto-registration | 152–164 | Yes — copy pattern |
| Right debug sidebar (resizable, 3 tabs) | 408–471 | Yes — copy structure |
| Component routing (show/hide + lifecycle hooks) | 85–113 | Yes — copy pattern |
| Health check polling | 168–176 | Adapt — change endpoint |
| Branding/header | 408–430 | Adapt — change title/logo |
| Responsive breakpoints (1024px, 768px) | 550–600 | Yes — copy CSS |
| Preference persistence (localStorage) | 200–220 | Yes — copy pattern |

### Tight Couplings (3 items)

1. **Global namespace:** Shell expects `window.sgraphAdmin.{events, messages, config, router}` — solvable by creating `window.sgraphVault.*`
2. **Hard-coded debug panels:** Shell renders `<messages-panel>`, `<events-viewer>`, `<api-logger>` inline — solvable by copying these 3 components
3. **Admin CSS variables:** Shell uses `var(--admin-*)` tokens — solvable by defining vault equivalents or reusing the same names

### DOM Model Difference

| | Admin Shell | Vault v0.1.0 |
|---|---|---|
| **DOM** | Light DOM | Shadow DOM |
| **Styles** | Global CSS + custom properties | Scoped per-component |
| **Templates** | Inline in JS | External HTML/CSS files |

**These can coexist.** The vault-shell (light DOM) can host vault components (Shadow DOM) inside its main content area. Shadow DOM components render independently within their boundaries.

### Recommendation

**Create a new `vault-shell.js` (~500 lines)** that copies the admin shell's layout patterns:
- Same CSS Grid template (header / sidebar+main+debug / statusbar)
- Same EventBus service pattern
- Same debug panel structure
- Different branding, navigation sections, and health endpoint

**Do NOT subclass or inherit from admin-shell.js.** The admin shell is not designed for inheritance (no exported base class, inline CSS, hard-coded labels). A clean copy-and-adapt is faster and cleaner than refactoring admin-shell for extensibility.

## Question 3: Best Strategy for Shared Module Publishing

### Three Options Assessed

| Option | Description | Pros | Cons |
|--------|-------------|------|------|
| **A: Direct copy** | Copy JS files into vault deployment | Works now, zero infra | Duplication, drift between copies |
| **B: Static CDN** | Publish to `cdn.sgraph.ai/modules/` or similar | Single source of truth, all tools import same files | Requires DevOps work, CDN setup |
| **C: npm package** | Publish `@sgraph/send-modules` to npm | Versioned, dependency management | Requires build step, overkill for current stage |

### Recommendation: Phased Approach

**Phase 1 (vault v0.1.1, now):** Option A — copy files directly into `sgraph_ai_app_send__ui__vault/v0/v0.1/v0.1.1/_common/js/shared/`. This unblocks development immediately.

**Phase 2 (next sprint):** Option B — DevOps creates a shared static path. Suggested structure:

```
sgraph_ai_app_send__ui__shared/
├── js/
│   ├── crypto.js               ← SendCrypto
│   ├── file-type-detect.js     ← FileTypeDetect
│   ├── markdown-parser.js      ← MarkdownParser
│   ├── preview-renderer.js     ← Extracted preview utilities
│   ├── zip-browser.js          ← Extracted ZIP browser
│   └── vendor/
│       └── jszip.min.js        ← JSZip
├── css/
│   └── design-tokens.css       ← Shared Aurora theme tokens
└── services/
    ├── event-bus.js             ← EventBus
    ├── config-manager.js        ← ConfigManager
    └── messages-service.js      ← MessagesService
```

All tools (send.sgraph.ai, vault.sgraph.ai, admin) import from this shared location. Nothing confidential — all open-source JavaScript.

**Phase 3 (future):** Evaluate if Option C (npm) is warranted based on team size and tool count.

## Question 4: Does the Admin UI Need Refactoring First?

### Answer: No — extraction is faster than refactoring

The admin shell (v0.1.3) has been **stable since February**. All subsequent versions (v0.1.4–v0.1.7) only add feature components — none modify the shell itself. This is a sign of good architecture.

However, the admin shell was not designed for extraction:
- 867 lines in a single file
- Inline CSS (860+ lines embedded in the render method)
- Hard-coded branding and labels
- No configuration API

**Refactoring the admin shell into a reusable base class would take longer than copying its patterns.** The refactoring would also risk introducing regressions in the working admin console.

### Recommended Path

1. **Don't touch admin-shell.js** — it works, leave it alone
2. **Create vault-shell.js** by copying the layout structure and adapting it
3. **Copy the 3 service files** (event-bus, config-manager, messages-service) — they're already generic
4. **Copy the 3 debug panel components** — they depend only on the services above
5. **If a third tool needs the shell pattern later**, then consider extracting a shared `app-shell` base class

This follows the "three strikes" rule: don't abstract until you have three consumers.

| **VaultI18n** | `vault-i18n.js` | Good — 17 locales, URL-based detection |

### The Subfolder Bug

**Symptom:** Files uploaded while in a subfolder (e.g., `/docs/`) get saved to root `/`.

**Root cause analysis:** The path propagation chain is:

```
vault-browser._currentPath  →  event 'vault-upload-request' { path }
    →  vault-app._onUploadRequest(e)  →  sets vault-upload._targetPath
        →  vault-upload._processQueue()  →  calls vault.addFile(_targetPath, ...)
            →  sg-vault._findNode(folderPath)  →  traverses tree
```

The most likely failure point is in **vault-upload.js**:

```javascript
async uploadFile(file, path) {
    this._targetPath = path || this._targetPath   // ← fallback to default '/'
    this._addFiles([file])
}
```

If `path` is `undefined` (e.g., the drag-and-drop event doesn't include the path), the fallback is the initial default `'/'`. The fix is to ensure every upload path (button click AND drag-and-drop) passes `vault-browser._currentPath` explicitly.

### Vault Tree Data Model

The tree structure is clean and should be preserved exactly:

```javascript
{
  version: 1,
  updated: "ISO-timestamp",
  tree: {
    "/": {
      type: "folder",
      children: {
        "subfolder": { type: "folder", children: { ... } },
        "file.txt": { type: "file", file_id: "transfer_id", size: N, uploaded: "ISO" }
      }
    }
  }
}
```

**Key property:** Every tree mutation re-encrypts and re-uploads both tree and settings, generating new transfer IDs. The vault key's `settingsTransferId` component changes on every write.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Light DOM shell + Shadow DOM components interaction | Style leakage or event bubbling issues | Shadow DOM isolates styles; use CustomEvent with `composed: true` for cross-boundary events |
| Admin shell pattern drift | Vault shell diverges from admin patterns over time | Document the shared patterns; Phase 2 shared module publishing will re-converge |
| Preview module extraction from send-download.js | Tight coupling makes extraction harder than expected | Start with simple viewers (image, text); extract PDF/ZIP later |
| Vault tree versioning on every write | Performance with many mutations | Acceptable for v0.1.1; consider batching or debouncing saves in future |
