## Part 2: CRITICAL — Vault Key Stability

### The Problem (Code-Verified)

The browser vault (`sg-vault.js` line 241-256) currently changes the vault key on **every file operation**:

```javascript
// sg-vault.js _saveTree() — called on every add/delete/rename
async _saveTree() {
    // 1. Upload new tree → gets NEW transfer ID
    const result = await this._sgSend.upload(new Uint8Array(encrypted))
    this._treeTransferId = result.transferId         // ← CHANGES

    // 2. Update settings to point to new tree → upload → gets NEW transfer ID
    this._settings._tree_transfer_id = this._treeTransferId
    const settingsResult = await this._sgSend.upload(new Uint8Array(encSettings))
    this._settingsTransferId = settingsResult.transferId  // ← CHANGES
}
```

Then `vault-app.js` reconstructs the vault key with the new `settingsTransferId`:

```javascript
// vault-app.js _updateVaultKey() — called after every _saveTree()
_updateVaultKey() {
    const settingsId = this._vault._settingsTransferId  // NEW value
    parts.splice(-2, 2, vaultId, settingsId)
    this._vaultKey = parts.join(':')                    // KEY CHANGED
    this._share.updateKey(this._vaultKey)               // URL hash updated
}
```

**Result:** The vault key (`passphrase:vault_id:settings_transfer_id`) changes every time a file is added, deleted, or renamed. The URL hash in the browser updates silently. Anyone who saved the old key can no longer open the vault.

### The Human's Direction

> "That 'vault key' needs to be super stable. It's the only thing needed to open a vault. It is OK to push breaking changes at this stage."

### Proposed Solution: Transfer Update Endpoint

**Add a `POST /transfers/update/{id}` endpoint** to the User Lambda that allows replacing the content of a completed transfer while keeping its transfer_id.

Currently, transfers are immutable after completion (the upload endpoint checks `status != 'pending'` and rejects re-uploads). The new endpoint:

```
POST /transfers/update/{transfer_id}
Headers: x-sgraph-send-access-token: <token>
Body: <encrypted bytes>

Behaviour:
1. Verify transfer exists
2. Verify access token
3. Replace stored payload with new payload
4. Update metadata (size, modified_at)
5. Keep same transfer_id
6. Return { "status": "updated", "transfer_id": "..." }
```

**Impact on vault flow:**

| Step | Before (broken) | After (stable) |
|---|---|---|
| Create vault | Settings → new transfer_id | Settings → new transfer_id (same) |
| Add file | Settings → NEW transfer_id | Settings content UPDATED, same ID |
| Vault key | `pass:vid:NEW_sid` | `pass:vid:SAME_sid` (never changes) |

**Both browser and CLI** use this endpoint. The browser's `_saveTree()` calls update instead of upload for settings. The CLI's push does the same.

### Vault Key Format (Simplified)

```
Format: {passphrase}:{vault_id}:{settings_transfer_id}
```

The format stays the same, but the `settings_transfer_id` never changes after vault creation. It's assigned once during `init`/`create` and then updated in-place.

**For the `.sg_vault/HEAD` file, we also store the simpler form:**
```
passphrase:vault_id:settings_transfer_id
```

This key is permanent. Share it once, it works forever.

### Implementation: Two Coordinated Changes

1. **Backend (this repo, Explorer session):** Add `POST /transfers/update/{id}` to User Lambda
2. **Browser (this repo):** Modify `_saveTree()` to use update instead of upload for settings and tree
3. **CLI (new repo):** Build against the stable key from day one

**The backend change is a prerequisite.** The CLI team should not start until the update endpoint exists and the browser is aligned.

## Part 3: Type_Safe — Zero Raw Primitives

### The Rule (Non-Negotiable)

From the Type_Safe guidance (`v3.63.4__for_llms__type_safe.md`):

> **NEVER use raw `str`, `int`, or `float` in Type_Safe classes.** Raw primitives enable entire categories of bugs and security vulnerabilities.

This means **every class attribute, every method parameter, and every return type** must use Safe_* types or Type_Safe schemas.

### Domain Types for the CLI

The CLI introduces new domain concepts. Each needs a proper Safe_* type:

| Concept | Raw type (BANNED) | Safe type | Definition needed? |
|---|---|---|---|
| Vault ID | `str` | `Safe_Str__Vault__Id` | Yes — 8 hex chars |
| Vault key | `str` | `Safe_Str__Vault__Key` | Yes — `pass:vid:sid` format |
| Passphrase | `str` | `Safe_Str__Vault__Passphrase` | Yes — user-provided passphrase |
| Transfer ID | `str` | `Safe_Str__Transfer__Id` | Yes — 12 alphanumeric chars |
| File path (local) | `str` | `Safe_Str__File__Path` | Exists in osbot-utils |
| File size | `int` | `Safe_UInt__FileSize` | Exists in osbot-utils |
| File hash (SHA-256) | `str` | `Safe_Str__SHA256` | Yes — 64 hex chars |
| Endpoint URL | `str` | `Safe_Str__Url` | Exists in osbot-utils |
| Access token | `str` | `Safe_Str__Access__Token` | Yes |
| Timestamp | `str` | `Timestamp_Now` | Exists in osbot-utils |
| Sync state | `str` | `Enum__Sync__State` | Yes — enum of 8 states |
| Tree version | `int` | `Safe_UInt` | Exists in osbot-utils |

## Part 4: Class-Based Typer with Type_Safe

### The Problem

Typer promotes module-level function decorators:

```python
# DEFAULT TYPER PATTERN (we do NOT want this)
app = typer.Typer()

@app.command()
def clone(vault_key: str, target_dir: str = "."):    # Static function
    # ... no class, no state, hard to test, hard to reuse
```

This is a **known Typer limitation** (GitHub issues #309, #539, #271). The solution uses bound method registration, which has been **verified working** in Python 3.12.

### The Solution: Typer__Routes Base Class (Mirrors Fast_API__Routes)

The project already has a class-based framework pattern: `Fast_API__Routes`. Routes are instance methods, registered via `setup_routes()`. We apply the identical pattern to Typer.

**Architecture (three layers):**

```
Layer 1: Typer__Routes (cmd_ methods)   — Convention-based command discovery
Layer 2: Core Services                  — Pure logic, no CLI awareness
Layer 3: Schemas + Types                — Pure data, zero raw primitives
```

#### Typer__Routes Base Class

```python
import typer
import inspect
from osbot_utils.type_safe.Type_Safe import Type_Safe

class Typer__Routes(Type_Safe):
    """Base class for Typer CLI command groups.

    Analogous to Fast_API__Routes but for CLI commands.
    Convention:
        cmd_<name>     -> becomes CLI command '<name>'
        cmd_foo_bar    -> becomes CLI command 'foo-bar'
        _methods       -> ignored (private helpers)
        other methods  -> ignored (not commands)
    """

    def typer_app(self, **kwargs) -> typer.Typer:
        app = typer.Typer(**kwargs)
        self.register_commands(app)
        return app

    def register_commands(self, app: typer.Typer):
        """Auto-discover cmd_ methods and register as Typer commands."""
        for name, method in inspect.getmembers(self, predicate=inspect.ismethod):
            if not name.startswith('cmd_'):
                continue
            command_name = name[4:].replace('_', '-')    # cmd_foo_bar -> foo-bar
            app.command(name=command_name)(method)
        return self
```

**Why bound methods work:** When you access `instance.method`, Python returns a bound method whose `inspect.signature()` **excludes `self`**. Typer sees only the real CLI parameters.

**Why this pattern is right:**
- Mirrors existing `Fast_API__Routes` convention (team already understands it)
- `cmd_` prefix = auto-discovery (no manual registration)
- Private `_methods` never become CLI commands
- Bound methods exclude `self` from Typer's signature inspection (verified)
- `Annotated[str, typer.Argument(...)]` works on bound methods (verified)
- `typer.Typer.add_typer()` works for composable sub-groups (verified)

## Part 8: S3 Presigned Upload — 1GB from Day One

### Upload Strategy

| File Size | Method | API |
|---|---|---|
| < 6 MB | Direct upload | `POST /transfers/upload/{id}` |
| 6 MB - 1 GB | Presigned multipart | `POST /presigned/initiate` → PUT parts → `POST /presigned/complete` |

### Presigned Flow (Already Exists)

```
1. POST /presigned/initiate
   Body: { "transfer_id": "...", "file_size": 52428800, "part_size": 5242880 }
   Response: { "upload_id": "...", "presigned_urls": ["https://s3...", ...] }

2. PUT each part to presigned URL (direct to S3, bypasses Lambda)

3. POST /presigned/complete
   Body: { "transfer_id": "...", "upload_id": "...", "parts": [...] }
```

### CLI Implementation

```python
class Vault__Client(Type_Safe):
    DIRECT_UPLOAD_LIMIT : Safe_UInt = 6_000_000       # 6MB

    @type_safe
    def upload_transfer(self, encrypted_data : bytes) -> Safe_Str__Transfer__Id:
        transfer_id = self._create_transfer()
        if len(encrypted_data) < self.DIRECT_UPLOAD_LIMIT:
            self._direct_upload(transfer_id, encrypted_data)
        else:
            self._presigned_upload(transfer_id, encrypted_data)
        self._complete_transfer(transfer_id)
        return transfer_id
```

**Soft limit:** 1GB is communicated to users but not strictly enforced. The presigned URL size is controlled by S3, not by our code.

## Part 9: Encryption Interop Contract (Retained from v1)

Identical to v1 Part 3. Key facts:

- **Algorithm:** AES-256-GCM
- **KDF:** PBKDF2-HMAC-SHA256, 600,000 iterations
- **Salt:** `sg-vault-v1:{vault_id}` (UTF-8)
- **Wire format:** `[12 bytes nonce][ciphertext + 16 bytes tag]`
- **No AAD (Additional Authenticated Data)**
- **First test = interop gate:** Encrypt in Python, decrypt in JavaScript

## Part 11: Key Design Decisions (Updated)

| # | Decision | Rationale |
|---|---|---|
| D1 | `.sg_vault/` as vault identity | Mirrors `.git/`. Single source of local state. |
| D2 | Use Transfer API, not Vault Admin API | Browser interop. Zero-knowledge purity. |
| D3 | Separate core/ from cli/ from commands/ | Three-layer architecture. Testable, reusable. |
| D4 | INI config format | Git convention. Human-readable. |
| D5 | `clone` not `connect` | Git vocabulary. |
| D6 | Stable vault key (requires backend change) | **Human requirement. Non-negotiable.** |
| D7 | Zero raw primitives | **Type_Safe rule. Non-negotiable.** |
| D8 | Class-based Typer (bound methods) | Testable, reusable, follows project patterns. |
| D9 | Pipeline before features | Proven bootstrap pattern (QA project precedent). |
| D10 | Separate repo (SG_Send__CLI) | **Human decision. PyPI name: sg-send-cli.** |
| D11 | 1GB presigned uploads from day one | Confirmed real-world use case (300MB+). |
| D12 | Schemas pure data, logic in services | Type_Safe architectural rule. |
