# Architect Review — workspace.sgraph.ai Architecture

**Version:** v0.10.35
**Date:** 2026-03-03
**Role:** Architect
**Brief:** `team/humans/dinis_cruz/briefs/03/02/v0.10.14__dev-brief__document-transformation-workspace.md`
**Request:** Map out the best architecture for hosting the Document Transformation Workspace at `workspace.sgraph.ai`, maximising reuse of existing code and infrastructure from the other `.sgraph.ai` sites.

---

## 1. Executive Summary

The Document Transformation Workspace can be built as a **pure client-side application** that combines three things we already have — the vault's encrypted file system, the admin shell's component architecture, and the website's deployment pipeline — with one new thing: a browser-based LLM connection layer. No new backend endpoints are needed for Phase 1.

**Key architectural decision:** The workspace is a **new static site** deployed identically to `sgraph.ai`, pointed at the same backend as `send.sgraph.ai`. It reuses existing vault APIs, existing crypto libraries, and existing UI component patterns. The LLM calls go directly from the browser to provider APIs — the server never sees document content or LLM interactions.

### Shared Resource Strategy: Link, Don't Copy

**Critical design principle:** Shared JS/CSS files are **imported at runtime** from the deployed vault site, not copied into the workspace repo. This ensures all `.sgraph.ai` sites run the same version of shared components and updates propagate automatically.

**Import sources (in priority order):**

| Source | URL Pattern | When to Use |
|--------|------------|-------------|
| **vault.sgraph.ai** (latest) | `https://vault.sgraph.ai/_common/js/{file}` | Default — always use latest |
| **dev.vault.sgraph.ai** | `https://dev.vault.sgraph.ai/_common/js/{file}` | Development/testing |
| **Pinned release** | `https://static.sgraph.ai/websites/sgraph-vault/prod/releases/v0/v0.1/v0.1.0/_common/js/{file}` | When pinning to a specific version |

**In `index.html`:**
```html
<!-- Shared infrastructure — imported from vault.sgraph.ai -->
<script src="https://vault.sgraph.ai/_common/js/event-bus.js"></script>
<script src="https://vault.sgraph.ai/_common/js/messages-service.js"></script>
<script src="https://vault.sgraph.ai/_common/js/vault-crypto.js"></script>
<script src="https://vault.sgraph.ai/_common/js/vault-api.js"></script>
<script src="https://vault.sgraph.ai/_common/js/file-type-detect.js"></script>
<script src="https://vault.sgraph.ai/_common/js/markdown-parser.js"></script>

<!-- Workspace-specific components — local -->
<script src="components/workspace-shell/workspace-shell.js"></script>
<!-- ... -->
```

**Benefit:** Zero file duplication across sites. When vault.sgraph.ai ships a fix to `vault-crypto.js`, workspace.sgraph.ai gets it immediately. Can pin to a specific release URL if a breaking change needs to be held back.

## 3. Deployment Architecture

### Option A: Static Site on S3 + CloudFront (RECOMMENDED)

```
workspace.sgraph.ai
  → CloudFront distribution (new)
  → S3 bucket: {account-id}--static-sgraph-ai--eu-west-2
  → Path: websites/workspace-sgraph-ai/latest/
```

**Why this is best:**
- Identical pattern to `sgraph.ai` website deployment
- Reuses existing S3 bucket (just a new prefix)
- Reuses existing `deploy_static_site.py` script
- New CloudFront distribution + DNS record is the only AWS setup needed
- Static assets served at CDN speed (no Lambda cold starts for UI)
- Backend calls go to existing `admin.send.sgraph.ai` Lambda

**DNS:**
```
workspace.sgraph.ai → CNAME → d{xxx}.cloudfront.net
```

**CloudFront configuration:**
- Origin: S3 bucket at `websites/workspace-sgraph-ai/latest/`
- URL rewrite function: reuse existing `cloudfront/url-rewrite.js`
- Cache: HTML 300s, CSS/JS 86400s, images 604800s (same policy)
- TLS: ACM certificate (*.sgraph.ai or workspace.sgraph.ai)

### Option B: Lambda-Served (NOT recommended)

Serving workspace UI through the Lambda function (like send.sgraph.ai currently does) adds cold start latency for no benefit. The workspace has no server-side rendering needs.

### Repo Structure

```
sgraph_ai_app_send__ui__workspace/     # NEW directory
  index.html                            # Workspace shell (imports shared JS from vault.sgraph.ai)
  css/
    workspace.css                       # Workspace-specific styling (extends imported CSS)
  components/
    workspace-shell/
      workspace-shell.js                # Main shell with left-nav + panel routing
    vault-panel/

```

**No `_common/` directory** — shared JS/CSS imported from vault.sgraph.ai at runtime. Debug panels (api-logger, events-viewer, messages-panel) also imported from vault.sgraph.ai. Only workspace-specific components live locally.

## 4. Architectural Decisions

### AD-1: Backend API — Reuse Existing Vault Endpoints

**Decision:** No new backend endpoints for Phase 1. The workspace uses the same vault API that the user vault and admin vault-manager already use.

**Rationale:**
- Vault CRUD (create, file upload/download, index, folders) already exists — 10 endpoints on admin Lambda
- The workspace saves transformations as new vault files — same API
- Version history = saving a new file with a version suffix in the name — the vault index tracks it
- Prompt library = a special folder in the vault containing JSON files — no new API needed

**Trade-off:** This means the workspace requires an admin Lambda connection (same as vault). If we later want unauthenticated workspace access, we'd need to expose vault endpoints on the user Lambda too.

### AD-2: LLM Calls — Direct from Browser, No Proxy

**Decision:** All LLM API calls go directly from the browser to provider APIs. No server-side proxy.

**Rationale:**
- Zero-knowledge principle — server never sees document content or LLM prompts
- No CORS issues — all major LLM providers support browser-direct calls with API keys
- No server cost for LLM usage — the user pays their own API provider
- Simpler architecture — no new Lambda endpoints, no key management on server

**Phase 1 providers (APPROVED):**
| Provider | Browser CORS | Notes |
|----------|-------------|-------|
| **OpenRouter** (openrouter.ai) | YES — designed for browser use | Primary. One API key → Claude, GPT-4, Gemini, Llama, Mistral, 200+ models. Also has a nice set of free models. |
| **Ollama** (localhost:11434) | YES — same-machine | Local inference. Zero cost, maximum privacy. |

**Phase 2 providers (later):**
| Provider | Notes |
|----------|-------|
| Anthropic (api.anthropic.com) | Direct connection — verify CORS when ready |
| OpenAI (api.openai.com) | Direct connection — verify CORS when ready |
| Google (generativelanguage.googleapis.com) | Likely works — API key in query param |

### AD-3: Vault API Origin — Cross-Origin to admin.send.sgraph.ai

**Decision:** The workspace at `workspace.sgraph.ai` calls vault APIs at `admin.send.sgraph.ai`.

**CORS:** Not a concern — the human will handle CORS configuration as part of the CloudFront/DNS setup. Proceed with implementation without blocking on CORS verification.

### AD-4: API Key Storage — localStorage

**Decision:** LLM API keys are stored in the browser's `localStorage`. This persists across sessions without requiring vault infrastructure.

**Design:**
```javascript
// Store to localStorage
const config = { anthropic: 'sk-ant-...', openai: 'sk-...', defaultProvider: 'openrouter' };
localStorage.setItem('workspace-llm-config', JSON.stringify(config));

// Read on page load
const config = JSON.parse(localStorage.getItem('workspace-llm-config') || '{}');
```

**Security:** This is an accepted risk. API keys in localStorage are visible to anyone with physical access to the browser. This is still better than storing keys on a server where the server can read them. The keys are the user's own keys on the user's own machine. We can migrate to SG/Send-based encrypted storage later.

**Future enhancement:** Use SG/Send to encrypt and store keys, adding zero-knowledge protection at rest. Not needed for Phase 1.

│ 📄 │         │                  │                   │  BAR    │
│ 🔧 │  Files  │  Loaded from     │  LLM output or    │  API    │
│ ⚙️ │  Folders│  vault, rendered │  edited version   │  Logs   │
│    │  (src/  │  with type-      │                   │  Events │

### AD-6: Document Viewer — Type-Aware Rendering (Phase 1: Text, HTML, Markdown)

**Decision:** Phase 1 over-indexes on three file types that cover the primary workspace use cases. Each needs a different preview mode.

| Type | Renderer | Preview Mode | Use Case |
|------|----------|-------------|----------|
| **Markdown** | Rendered HTML + source toggle | Rendered view (default) or raw markdown | Documents, briefs, reviews — primary working format |
| **HTML** | Sandboxed iframe (rendered) + source view | Rendered page (default) or raw HTML | Web pages, landing pages, email templates |
| **Plain text** | Monospace `<pre>` | Raw text only | Transcripts, logs, notes, configs |

**Phase 2 types (later):**
| Type | Renderer | Notes |
|------|----------|-------|
| JSON | Pretty-printed + collapsible tree | Translation files, configs |
| CSV | Table rendering | Data files |
| Images | `<img>` element | Send to multimodal LLM for analysis |
| PDF | Embedded viewer (pdf.js) | Read + transform extracted text |

**Why these three first:**
- Markdown is the primary working format for briefs, reviews, and documents
- HTML covers web content and rendered output (the workspace often produces HTML)
- Plain text is the fallback for everything else and requires no rendering logic
- Together these three cover 90%+ of the document transformation use cases

### AD-7: Transformation Save Model — Source/Transform Folder Pairs + Manifest

**Decision:** Each workspace project uses a **folder pair** structure. Every folder has at least two sub-folders: `source/` and `transform/`. Both the source and the transformation documents are independently versionable. A **manifest file** per pair tracks the relationship between source versions and transformation versions.

**Folder structure in vault:**
```
my-project/
  manifest.json                         # Pair manifest (see below)
  source/
    report.md                           # Current source (editable)
    report__v001.md                     # Source version 1
    report__v002.md                     # Source version 2 (edited)
  transform/
    report__transform-001.md            # Transform v1 (from source v1)
    report__transform-002.md            # Transform v2 (from source v2, different prompt)
    report__transform-003__summary.md   # Transform v3 (executive summary variant)
```

**Manifest file (`manifest.json`):**
```json
{
  "project": "my-project",
  "created": "2026-03-03T14:00:00Z",
  "pairs": [
    {
      "id": "pair-001",
      "source": { "guid": "guid-source-v001", "name": "report__v001.md", "version": 1 },
      "transform": { "guid": "guid-transform-001", "name": "report__transform-001.md", "version": 1 },
      "prompt": "Make more concise",
      "model": "anthropic/claude-sonnet-4",
      "timestamp": "2026-03-03T14:30:00Z"
    },
    {
      "id": "pair-002",
      "source": { "guid": "guid-source-v002", "name": "report__v002.md", "version": 2 },
      "transform": { "guid": "guid-transform-002", "name": "report__transform-002.md", "version": 2 },
      "prompt": "Add executive summary",
      "model": "anthropic/claude-sonnet-4",
      "timestamp": "2026-03-03T15:00:00Z"
    }
  ]
}
```

**Key design points:**
- **Both panels are editable and saveable** — the user can modify the source, the transformation, or both
- **Both have independent version histories** — source can have 5 versions while transform has 3
- **The manifest tracks which source version produced which transform** — enables "show me the transform that came from source v2"
- **Version-by-filename** — each save creates a new file with an incremented version suffix (e.g., `report__v003.md`)
- **The vault-panel shows folder pairs** — navigating to a project folder shows the source/transform sub-structure
- **Saving either document** increments its version counter and adds a new entry to the manifest

**Why folder pairs?**
- Clean separation of source vs. output
- Each project is self-contained in one folder
- The manifest is a lightweight alternative to database versioning — it's just another vault file
- Works with the existing vault CRUD API (no new endpoints)

### AD-8: Prompt Library Storage

**Decision:** Prompts are stored as JSON files in a `.workspace/prompts/` folder in the vault.

```json
{
  "id": "prompt-improve-clarity",
  "name": "Improve clarity",
  "category": "document",
  "prompt": "Rewrite the following document for clarity. Preserve all facts and key points. Do not add new information. Output the full revised document.\n\n---\n\n{{document}}",
  "model_preference": null,
  "created": "2026-03-03T14:00:00Z",
  "builtin": true
}
```

**Pre-built prompts** are shipped with the workspace HTML (embedded in JS). **Custom prompts** are saved to vault. On load, the workspace merges built-in + vault-stored prompts.

### AD-9: LLM Response Streaming

**Decision:** Use Server-Sent Events (SSE) streaming where supported by the provider API.

**Why it matters for UX:** Document transformations can take 10-30 seconds. Without streaming, the user stares at a blank right panel. With streaming, text appears progressively.

**Implementation:**
```javascript
// Anthropic streaming
const response = await fetch('https://api.anthropic.com/v1/messages', {
  method: 'POST',
  headers: { 'x-api-key': key, 'anthropic-version': '2023-06-01' },
  body: JSON.stringify({ model, messages, stream: true })
});

const reader = response.body.getReader();
// Read SSE chunks, parse delta events, append to document viewer
```

All major providers support streaming (Anthropic, OpenAI, Google, OpenRouter). Ollama also supports streaming.

### AD-10: No New Lambda Endpoints (Phase 1)

**Decision:** Phase 1 requires ZERO backend changes.

**What the workspace needs from the backend:**
- Create vault → `POST /vault/create` ✅ EXISTS
- Store file → `POST /vault/file` ✅ EXISTS
- Get file → `GET /vault/file/{key}/{guid}` ✅ EXISTS
- Store index → `POST /vault/index` ✅ EXISTS

**Everything else is client-side:** LLM calls, prompt management, document rendering, diff computation, transformation saving.

## 5. Security Considerations

### S-1: LLM API Keys in Browser

**Risk:** API keys are visible in browser DevTools.
**Mitigation:** This is an accepted risk — the user is connecting *their own* keys. The workspace is a tool, not a hosted service. The keys are only as exposed as they would be in any local development environment.
**Enhancement:** Recommend users create scoped/limited API keys for workspace use.

### S-2: Cross-Origin Vault Access

**Risk:** workspace.sgraph.ai calling admin.send.sgraph.ai across origins.
**Mitigation:** CORS configuration will be handled by the human as part of infrastructure setup. Not a blocker for development.

### S-3: HTML Rendering (XSS Vector)

**Risk:** User loads an HTML file from vault and it renders in the document viewer.
**Mitigation:** Render HTML in a sandboxed iframe with restrictive CSP:
```html
<iframe sandbox="allow-same-origin" srcdoc="..."></iframe>
```
Strip `<script>` tags and event handlers before rendering. This aligns with the AppSec findings from the gallery review (5 XSS vectors identified).

### S-4: LLM Response Injection

**Risk:** An LLM could return HTML/script in its response that gets rendered.
**Mitigation:** Always render LLM output as text/markdown, never as raw HTML. Use the existing markdown renderer which doesn't support `<script>` tags.

## 8. Recommendation

**Build it now.** The architectural alignment is strong:
- 70% of the code exists — linked from vault.sgraph.ai, not copied
- Zero new backend endpoints needed for Phase 1
- Zero-knowledge principle is preserved end-to-end
- The workspace naturally drives vault adoption (users need a vault to use the workspace)
- Deployment follows proven patterns (S3 + CloudFront, same as sgraph.ai)
- Human is setting up CloudFront + DNS in parallel

**Start with OpenRouter + Ollama** for LLM connections. OpenRouter gives access to all major models (Claude, GPT-4, Gemini, Llama, Mistral, plus free models for testing) through one browser-friendly API. Ollama provides free local inference for maximum privacy.

**The workspace is the vault's killer feature.** Without it, the vault is just encrypted Dropbox. With it, the vault becomes an encrypted workspace where documents are living, transformable assets.
