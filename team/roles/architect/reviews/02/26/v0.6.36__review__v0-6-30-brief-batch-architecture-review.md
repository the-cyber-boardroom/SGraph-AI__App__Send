# v0.6.36 -- Architect Review: v0.6.30 Brief Batch Architecture Assessment

**Version:** v0.6.36
**Date:** 2026-02-26
**Role:** Architect (Explorer team)
**Type:** Architecture review of 11 briefs from the 25-26 Feb batch

---

## 1. Executive Summary

The 11 briefs delivered on 25-26 February 2026 represent the next evolution of SGraph Send: the transition from a file transfer tool to a **multi-surface encrypted workspace platform**. Where the previous 28-brief batch (23 Feb) established the PKI identity and trust pillars, this batch defines the **operational surfaces** through which users interact with the platform -- data rooms, secret sharing, password vaults, developer tools, and agentic pipelines.

**The architectural vision emerging from this batch:**

SGraph Send is becoming a platform with five distinct interaction surfaces, all sharing a common encryption substrate:

1. **Ephemeral sharing** -- one-time secrets with configurable lifespans (`/s/`)
2. **Persistent encrypted spaces** -- symmetric key data rooms (`/room/`)
3. **Personal encrypted storage** -- password vault and key management (`/vault/`, `/tools/`)
4. **PKI document delivery** -- long-lived encrypted links for identified recipients (`/d/`)
5. **Agentic pipelines** -- MCP-driven encrypted transfer for AI agents and automation

These five surfaces share encryption utilities, preview components, URL routing infrastructure, and a unified static asset pipeline. The architecture challenge is ensuring they compose cleanly without creating a tangled dependency graph.

**Assessment:** The briefs are architecturally coherent. The URL architecture brief provides the routing spine that every other brief hangs from. The encryption patterns across the briefs are consistent and layered correctly (symmetric for usability, PKI for identity, vault for personal storage). The S3/CloudFront migration and Lambda dependency optimization provide the infrastructure foundation for all of this.

**Primary risk:** The briefs define six distinct encryption contexts (symmetric secrets, PKI delivery, data room master keys, data room user keys, vault encryption, client-side encrypted logs) that must interoperate without creating key confusion, management complexity, or accidental cross-contamination. The key hierarchy needs an explicit architecture decision before implementation begins.

## 2. URL Architecture Analysis

**Source:** [`v0.6.30__dev-brief__url-architecture-spa-static-assets-s3-cloudfront.md`](../../../../humans/dinis_cruz/briefs/02/25/v0.6.30__dev-brief__url-architecture-spa-static-assets-s3-cloudfront.md)

### 2.1 The URL Structure: `/{lang}-{country}/{area}/{resource}`

The proposed three-level URL hierarchy is well-designed and provides a stable addressing scheme for all current and planned features.

**Architecture assessment: SOUND.**

The combined locale segment (`en-gb` rather than `en/gb`) is the right call -- it prevents path depth explosion and keeps all functional URLs at a predictable depth. The functional area level (`/send/`, `/room/`, `/d/`, `/s/`, `/tools/`, `/vault/`, `/admin/`) maps cleanly to the five interaction surfaces identified above plus administrative and documentation concerns.

**How the URL structure affects every other brief:**

| Brief | URL Impact | Assessment |
|-------|-----------|------------|
| One-time secrets | `/s/{secret_id}#{key}` -- symmetric secrets use fragment for key, opaque server ID | Clean. Fragment-based key is zero-knowledge compliant. |
| Data rooms | `/room/{room_name}` and `/room/{room_name}/{folder}/{file}` -- hierarchical addressing for room contents | Good. Room names are human-readable because they are not security-sensitive (access is key-based). |
| PKI delivery | `/d/{key_prefix}/{doc_name}` -- key prefix for auto-key-selection | Well-designed. Key prefix in URL path (not fragment) is correct because the URL itself is not a secret in PKI mode. |
| Password generator | `/tools/password-generator` -- static tool, no server logic | Clean. Pure static route served from S3/CloudFront. |
| Browser storage viewer | `/tools/storage-viewer` -- static tool | Same pattern as password generator. |
| Admin | `/admin/*` -- SPA with API calls | Correct separation. Admin is a full SPA backed by the admin Lambda. |

### 2.2 SPA Routing Strategy

The route classification table in the brief is precise and architecturally important. The key insight: **not everything is an SPA**. The brief correctly identifies four routing types:

1. **Server redirect then SPA** -- `/send/{id}`, `/room/{name}` (server validates existence, then redirects to client)
2. **Pure SPA** -- `/d/{prefix}/{name}`, `/admin/*` (entirely client-side routing)
3. **Server + SPA hybrid** -- `/s/{id}#{key}` (server serves encrypted blob, client decrypts)
4. **Pure static** -- `/tools/*`, `/docs/*` (served directly from S3/CloudFront)

**Architecture decision needed:** The SPA fallback pattern (all non-API routes serve `index.html` for client routing) creates a tension with the "server validates existence" pattern for `/send/{id}` and `/room/{name}`. If CloudFront routes everything to `index.html`, how does the server validate that a transfer or room exists before the client loads?

**Recommendation:** Use CloudFront origin request Lambda@Edge (or CloudFront Functions) to handle the validation-then-redirect pattern. Routes like `/send/{id}` hit the edge function, which makes a lightweight API call to check existence, then either serves `index.html` (resource exists) or returns a 404 page (resource does not exist). This keeps the validation server-side without routing all traffic through the Lambda origin.

### 2.3 Locale Detection and Redirects

The locale detection pattern (no locale prefix -> detect from `Accept-Language` -> redirect) is standard and correct. One implementation concern: **redirect loops**. If the client's `Accept-Language` header maps to a locale we do not support, the redirect must fall back to a default (`en-gb`) rather than looping. The redirect should set a `sgsend_locale` cookie or localStorage flag to prevent re-detection on every page load.

## 3. Encryption Architecture

This is the most architecturally significant cross-cutting concern in the batch. Six distinct encryption contexts emerge from the briefs. They use the same underlying primitive (AES-256-GCM via Web Crypto API) but serve different purposes and have different key management models.

### 3.1 The Six Encryption Contexts

| # | Context | Key Type | Key Lifecycle | Key Holder | Brief |
|---|---------|----------|--------------|------------|-------|
| 1 | **Symmetric secrets** | Random AES-256 key per secret | Ephemeral (destroyed with secret) | Sender (in URL fragment) | One-time secrets |
| 2 | **PKI delivery** | Recipient's public key | Long-lived (identity key) | Recipient (private key in browser) | PKI-friendly links |
| 3 | **Data room master key** | Random AES-256 key per room | Room lifetime | Admin (generates), users (derived access) | Symmetric data room |
| 4 | **Data room user keys** | Short friendly keys mapping to master key | Revocable per user | Admin (creates/revokes) | Symmetric data room |
| 5 | **Vault encryption** | User's personal key | Long-lived (identity-bound) | User (in browser IndexedDB) | Password generator |
| 6 | **Client-side encrypted logs** | Security team's public key | Long-lived (team key) | Security team (private key offline) | One-time secrets |

### 3.2 Key Hierarchy Map

```
User's Personal Identity (PKI key pair -- long-lived)
|
+-- Vault Encryption
|   All vault entries encrypted with user's personal key.
|   Password entries, file entries, key backup entries.
|
+-- PKI Delivery (sender side)
|   Encrypts content + metadata with RECIPIENT's public key.
|   Signs with sender's private key (provenance).
|
+-- PKI Delivery (receiver side)
|   Decrypts with own private key.
|   Verifies sender's signature.
|   Imports sender's public key into contacts.
|
+-- Data Room Admin Key
    Admin creates room -> generates AES-256 master key.
    Admin creates user keys -> each is an encrypted copy of master key.
    |
    +-- User A short key -> encrypts/decrypts master key
    +-- User B short key -> encrypts/decrypts master key
    +-- User C short key -> encrypts/decrypts master key

Symmetric Secrets (standalone -- no identity required)
    Random AES-256 key per secret.
    Key embedded in URL fragment.
    No relationship to user identity.

Client-Side Encrypted Logs (standalone -- team key)
    Security team's public key embedded in client.
    Client signs log entries with user's key, encrypts with team key.
```

### 3.3 Interaction Points and Risks

**Interaction 1: One-time secrets sharing data room keys.**
The data room brief explicitly describes using one-time secret links to share user access keys. This means a symmetric secret (context 1) carries a data room user key (context 4). Architecturally, this is clean -- the one-time secret is just an encrypted container; the content happens to be another key. No special handling needed, but the UI should make this flow seamless (the "Share via Secret Link" button in the admin interface).

**Interaction 2: PKI mode on one-time secrets.**
The one-time secret brief describes both symmetric and PKI modes. In PKI mode, the secret is encrypted with the recipient's public key (context 2), and metadata is stored as a second encrypted file. This means the one-time secret path (`/s/`) and the PKI delivery path (`/d/`) share the same encryption model but have different URL structures and different security properties (one-time vs long-lived). The implementation must keep these paths distinct despite their shared substrate.

**Interaction 3: Vault stores passwords AND keys.**
The vault (context 5) stores both password entries (`type: "password"`) and cryptographic keys (private keys, data room keys, imported contacts). This means the vault schema must support heterogeneous entry types under a single encryption envelope. The password generator brief defines the `type: "password"` schema. The browser storage viewer brief shows that keys live in IndexedDB. The vault must unify these storage locations or clearly delineate which store holds what.

**Risk: Key confusion.** A user who has a personal PKI key, three data room keys, and a vault full of passwords is managing five different key contexts. The browser must auto-select the right key based on context (key prefix in PKI URLs, room name for data rooms, personal key for vault). This auto-selection logic is critical UX and must not silently pick the wrong key.

### 3.4 Architecture Recommendation: Unified Key Registry

Implement a browser-side **Key Registry** that:

1. Stores all keys (personal PKI, data room, imported contacts) in a single IndexedDB database
2. Tags each key with its context (`personal`, `data-room:{room_name}`, `contact:{fingerprint}`)
3. Provides a `resolve(context, hint)` method that returns the correct key given a URL context and optional prefix hint
4. Never exposes raw key material to the UI unless explicitly requested (export flow)

This registry is the single source of truth for all client-side key operations. The password generator, data room client, PKI viewer, and secret viewer all call into this registry rather than managing keys independently.

## 4. Shared Component Architecture

Four components are referenced by multiple briefs. These must be designed as reusable, independently versioned Web Components.

### 4.1 Preview Component

**Referenced by:** Data room preview brief, PKI-friendly links brief (minimal viewers), password generator brief (vault view)

**Usage contexts:**
- Receiver/download view -- current home at `send.sgraph.ai`
- Data room file browser -- inline preview of files in a data room
- Admin file inspector -- admin views files through the preview component
- Portable data room viewer -- future `.sgroom` file viewer

**Architecture decision:** The data room brief specifies that the admin side references code published on the user side, never the reverse. This means the preview component is published once (on the user/static side) and imported by all consumers via the compatible-versions manifest.

**Component interface:**
```
<send-preview
  src="{encrypted_blob_url}"
  key="{decryption_key}"
  content-type="{mime_type}"
  mode="inline|fullscreen|minimal"
/>
```

The `mode` attribute controls the viewer style: `inline` for data room file lists, `fullscreen` for document delivery, `minimal` for one-time secret text display.

### 4.2 Editor Component

**Referenced by:** Data room preview brief (versioned editing), admin interface (existing edit feature)

The brief specifies extracting the existing admin edit feature and adapting it for the data room context. The key difference: data room editing creates new versions rather than modifying in place.

**Component interface:**
```
<send-editor
  src="{encrypted_blob_url}"
  key="{decryption_key}"
  content-type="{mime_type}"
  on-save="createVersion"
  parent-version="{parent_file_id}"
/>
```

The `on-save` callback must produce a new file object with version metadata, not overwrite the source. The editor component handles the save-as-new-version semantics internally.

### 4.3 Vault Component

**Referenced by:** Password generator brief (vault entry creation), browser storage viewer brief (vault content inspection), PKI-friendly links brief (imported contacts display)

The vault is evolving from a simple encrypted file store into a **typed encrypted store** with specialised renderers per entry type:

| Entry Type | Schema | Renderer |
|-----------|--------|----------|
| `file` | `{type: "file", name, content, mime_type}` | Preview component |
| `password` | `{type: "password", label, username, password, url, notes, generator_settings}` | Password card (masked, copy, reveal) |
| `key` | `{type: "key", algorithm, fingerprint, public_key, private_key}` | Key card (fingerprint, export) |
| `contact` | `{type: "contact", name, public_key, fingerprint, imported_from}` | Contact card (name, key import source) |

**Architecture decision needed:** Should the vault be a single IndexedDB store with a `type` discriminator, or separate stores per type? The single-store approach is simpler and makes export/import trivial. The separate-store approach allows type-specific indexing and query patterns.

**Recommendation:** Single store with `type` discriminator. The vault will never hold millions of entries -- it is a personal store. Linear scan with type filtering is sufficient. Keep it simple.

### 4.4 Compatible-Versions Manifest

**Referenced by:** Data room preview brief (explicit spec), URL architecture brief (versioned IFD folder)

The manifest is a JSON document that maps component names to specific versions and their static URLs:

```json
{
  "manifest_version": "1",
  "build_version": "v0.6.36",
  "components": {
    "send-preview":   { "version": "v0.1.5", "url": "https://static.send.sgraph.ai/v0/v0.1/v0.1.5/components/send-preview/send-preview.js" },
    "send-editor":    { "version": "v0.1.6", "url": "https://static.send.sgraph.ai/v0/v0.1/v0.1.6/components/send-editor/send-editor.js" },
    "send-vault":     { "version": "v0.1.2", "url": "https://static.send.sgraph.ai/v0/v0.1/v0.1.2/components/send-vault/send-vault.js" },
    "crypto-utils":   { "version": "v0.2.0", "url": "https://static.send.sgraph.ai/v0/v0.2/v0.2.0/lib/crypto-utils.js" }
  }
}
```

**Architecture properties:**
- The manifest is itself a static asset served from S3/CloudFront
- The manifest URL is the only hardcoded reference -- all other component URLs are resolved through it
- Version pinning is explicit: updating a component means updating the manifest
- Multiple manifest versions can coexist (for rollback and A/B testing)
- The admin side and user side can reference different manifest versions if needed during a phased rollout

### 4.5 Encryption Utilities (Shared Library)

**Referenced by:** Every brief that involves encryption (all of them)

The `crypto-utils` library is the most critical shared component. It must provide:

| Function | Used By |
|----------|---------|
| `generateKey()` -- AES-256 random key | Symmetric secrets, data room master keys |
| `encrypt(plaintext, key)` -- AES-256-GCM | All encryption contexts |
| `decrypt(ciphertext, key)` -- AES-256-GCM | All decryption contexts |
| `wrapSGMETA(filename, content)` -- SGMETA envelope | MCP pipeline, file uploads |
| `unwrapSGMETA(payload)` -- extract filename + content | Download/preview |
| `deriveShortKey(masterKey)` -- generate user-friendly key | Data room user keys |
| `resolveShortKey(shortKey, encryptedMaster)` -- map short key to master | Data room access |
| `encryptWithPublicKey(plaintext, publicKey)` -- PKI encryption | PKI delivery, encrypted logs |
| `sign(data, privateKey)` -- digital signature | PKI provenance, encrypted logs |
| `verify(data, signature, publicKey)` -- signature verification | PKI verification |

**Architecture recommendation:** This library must be rigorously versioned. Every encryption/decryption function must include its version in the output format (e.g., a version byte prefix on encrypted blobs) so that future format changes can be handled gracefully. A decryption function must be able to handle any previously-produced format.

## 5. Infrastructure Architecture

### 5.1 S3/CloudFront Migration

**Source:** [`v0.6.30__dev-brief__url-architecture-spa-static-assets-s3-cloudfront.md`](../../../../humans/dinis_cruz/briefs/02/25/v0.6.30__dev-brief__url-architecture-spa-static-assets-s3-cloudfront.md)

The migration from Lambda-served static assets to S3/CloudFront is the most impactful infrastructure change in this batch. It fundamentally changes the deployment topology:

**Before:**
```
Browser -> CloudFront -> API Gateway -> Lambda (FastAPI) -> serves ALL content
```

**After:**
```
Static requests:  Browser -> CloudFront -> S3 (static files)
API requests:     Browser -> CloudFront -> API Gateway -> Lambda (FastAPI)
```

**Architecture implications:**

1. **Two deployment pipelines.** The CI/CD now has two distinct outputs: (a) the Lambda function code (Python, dependencies), and (b) the static asset bundle (HTML, JS, CSS, components). These can be versioned and deployed independently. This is a significant operational improvement -- a UI fix no longer requires a Lambda deployment.

2. **CloudFront behaviour routing becomes the control plane.** The CloudFront distribution configuration defines which paths route to S3 and which route to Lambda. This configuration is itself a deployment artefact that must be versioned and managed.

3. **SPA fallback requires careful configuration.** The brief specifies that all non-API routes should fall back to `index.html` for client-side routing. This is a standard SPA pattern, but the CloudFront custom error response approach (404 -> `index.html`) conflicts with the server-validation pattern for `/send/{id}` and `/room/{name}`. See recommendation in Section 2.2 above.

4. **Multiple sites on one distribution.** The brief proposes consolidating `send.sgraph.ai`, `docs.send.sgraph.ai`, the QA dashboard, and `team.send.sgraph.ai` onto one S3 bucket with one CloudFront distribution. This is cost-effective but requires careful path-based origin routing. Each site should have its own S3 prefix to maintain isolation.

### 5.2 Lambda Dependency Optimization

**Source:** [`v0.6.30__dev-brief__lambda-dependency-optimisation-minimum-viable-code.md`](../../../../humans/dinis_cruz/briefs/02/25/v0.6.30__dev-brief__lambda-dependency-optimisation-minimum-viable-code.md)

The three-phase optimization plan is well-structured:

| Phase | Action | Expected Impact | Risk |
|-------|--------|----------------|------|
| 1 | Consolidate to one zip per Lambda | ~60% size reduction (deduplication) | Low -- straightforward |
| 2 | Strip non-essential files | ~20% additional reduction | Low -- test after each removal |
| 3 | Tree-shaking (minimum viable code) | Variable -- potentially large | Medium -- dynamic imports may break |

**Architecture interaction with S3/CloudFront migration:** These two changes are complementary and should be sequenced carefully. Moving static assets to S3 means the Lambda no longer needs to serve HTML/JS/CSS. This immediately removes a class of dependencies (template engines, static file handlers, MIME type detection) from the Lambda package. The dependency optimization should happen AFTER the S3 migration, because the migration itself will eliminate several dependencies that would otherwise need to be tree-shaken.

**Recommended sequence:**
1. Migrate static assets to S3/CloudFront (removes static-serving code from Lambda)
2. Phase 1: Consolidate dependencies into single zip (now smaller because static-serving deps are gone)
3. Phase 2: Strip non-essential files
4. Phase 3: Tree-shake remaining dependencies

**Architecture recommendation for Phase 3:** Before attempting automated tree-shaking, manually list the import chain from each Lambda handler entry point. The SGraph Send codebase uses `osbot-utils`, `osbot-aws`, `osbot-fast-api`, and `osbot-fast-api-serverless` -- these are the core dependency trees. Profile which submodules of these packages are actually imported. The biggest savings will come from `boto3`/`botocore` (which includes definitions for every AWS service but SGraph Send only uses S3, Lambda, and CloudFront) and `osbot-aws` (which wraps many AWS services but SGraph Send uses a subset).

### 5.3 MCP Pipeline Architecture

**Sources:**
- [`SKILL.md`](../../../../humans/dinis_cruz/briefs/02/26/mcp/SKILL.md) -- MCP skill definition
- [`n8n-workmail-ews-claude-mcp-debrief.md`](../../../../humans/dinis_cruz/briefs/02/26/mcp/n8n-workmail-ews-claude-mcp-debrief.md) -- n8n + WorkMail integration

The MCP pipeline introduces a new client type: **AI agents that create encrypted transfers programmatically.** The pipeline follows the same zero-knowledge model as the browser client:

```
Claude Agent -> create SGMETA envelope -> AES-256-GCM encrypt
  -> upload via MCP (transfers_create, transfers_upload, transfers_complete)
  -> construct download URL (transfer_id + key in fragment)
  -> verify round-trip (download, decrypt, compare)
  -> optionally email link via n8n -> WorkMail
```

**Architecture implications:**

1. **The MCP skill is a CLIENT, not a server extension.** It uses the same API endpoints (`/transfers/create`, `/transfers/upload`, `/transfers/complete`) that the browser client uses. No new server-side code is needed. This is architecturally clean.

2. **The SGMETA envelope format is now a cross-client contract.** Both the browser and the MCP agent produce SGMETA-wrapped encrypted payloads. The format must be formally specified and versioned. The current format (`SGMETA\x00` + 4-byte metadata length + JSON metadata + file bytes) is documented in the SKILL.md but should be promoted to a formal specification.

3. **The n8n integration adds a third infrastructure component.** The stack is now: SGraph Send API (Lambda) + n8n Cloud (workflow orchestration) + AWS WorkMail (email delivery). The n8n workflows are a new deployment surface that must be version-controlled and tested.

4. **Email delivery of encrypted links is a new trust model.** When Claude sends an email containing a download link, the email contains the full URL including the fragment (key). Email is not a secure channel. This means the security of the transfer in the MCP+email flow depends on email security (TLS in transit, server access at rest). The brief should explicitly state that this flow provides convenience, not end-to-end security. For true E2E security, the PKI mode should be used.

**Architecture recommendation:** Formalise the SGMETA envelope as a specification document. Version it (`SGMETA\x01` for the next format revision). Include a version byte in the magic header so that clients can detect and handle format evolution.

## 6. Data Model Implications

The briefs introduce four new data structures that extend the current model.

### 6.1 Version Chains (Data Room Files)

**Source:** [`v0_6_30__dev-brief__data-room-preview-and-versioned-editing.md`](../../../../humans/dinis_cruz/briefs/02/25/v0_6_30__dev-brief__data-room-preview-and-versioned-editing.md)

Each file in a data room can have multiple versions linked in a chain:

```json
{
  "file_id": "meeting-notes-v2",
  "name": "meeting-notes.md",
  "version": 2,
  "parent": "meeting-notes-v1",
  "created_by": "Alice",
  "created_at": "2026-02-25T10:30:00Z",
  "hash": "sha256:abc123..."
}
```

**Architecture considerations:**

- **Version chains are DAGs, not linear.** If Alice and Bob both edit v1 concurrently, you get v2a and v2b -- both with `parent: v1`. The data model must support branching, not just linear chains. The brief acknowledges this ("v2a (Alice's) and v2b (Bob's) -- both preserved") but the schema shows a linear chain. Use `parent` as a single-valued field (each version has one parent) but allow multiple versions to share the same parent. This gives you a tree/DAG without requiring merge logic.

- **Version metadata is server-side.** The version chain metadata (file_id, name, version number, parent, created_by, created_at, hash) is stored on the server. This is a departure from the "server stores only encrypted blobs" model. The server now stores structural metadata about the relationship between encrypted blobs. This is acceptable because the metadata does not contain the file content -- it is the same class of metadata as the current transfer metadata (IDs, timestamps, sizes).

- **The hash field is a content integrity check.** The SHA-256 hash of the decrypted content allows the client to verify that what it decrypted matches what the author originally encrypted. This is defence-in-depth alongside GCM's authentication tag.

### 6.2 Password Vault Entries

**Source:** [`v0_6_30__dev-brief__password-generator-tool.md`](../../../../humans/dinis_cruz/briefs/02/25/v0_6_30__dev-brief__password-generator-tool.md)

The `type: "password"` schema extends the vault's data model:

```json
{
  "type": "password",
  "label": "Gmail -- work account",
  "username": "alice@example.com",
  "password": "kR7#mP2$vX9@nQ4!",
  "url": "https://mail.google.com",
  "notes": "",
  "created": "2026-02-25T11:00:00Z",
  "generator_settings": {
    "length": 16,
    "charset": "full",
    "entropy_bits": 104
  }
}
```

**Architecture considerations:**

- **The vault now has typed entries.** This requires a type discriminator and per-type renderers. The vault UI must dispatch to the correct renderer based on `type`. New types can be added incrementally (e.g., `type: "ssh-key"`, `type: "api-token"`, `type: "totp-secret"`).

- **Import compatibility.** The schema is designed to be compatible with standard password manager export formats (1Password, Bitwarden, LastPass, KeePass, browser CSV). The `label`, `username`, `password`, `url`, `notes` fields map directly to the common denominator across these formats. The `generator_settings` field is an extension that other managers do not produce, so it should be optional on import.

- **The entire vault entry is encrypted as one unit.** The vault does not support querying by field (you cannot search for "all passwords with url containing gmail.com" without decrypting all entries). This is a consequence of the zero-knowledge model. Search and filtering happen client-side after decryption. For small vaults (hundreds of entries) this is fine. For large vaults (thousands) it may need a client-side index.

### 6.3 Encrypted Metadata Files (PKI Mode)

**Source:** [`v0_6_30__dev-brief__one-time-secret-link.md`](../../../../humans/dinis_cruz/briefs/02/25/v0_6_30__dev-brief__one-time-secret-link.md)

In PKI mode, the server stores TWO encrypted files per secret:

```
Server stores:
  1. content.enc     <- the actual secret, encrypted with recipient's public key
  2. metadata.enc    <- friendly name, sender info, description -- also encrypted
```

**Architecture considerations:**

- **This is a new storage pattern.** The current storage model is one encrypted blob per transfer ID. PKI mode introduces a one-to-many relationship: one secret ID maps to multiple encrypted files. The storage layer (Memory-FS / S3) must support this either as a directory (multiple files under one prefix) or as a composite blob (both files concatenated with a separator).

- **Recommendation:** Use a directory-based approach in Memory-FS. Each secret gets a directory: `secrets/{secret_id}/content.enc`, `secrets/{secret_id}/metadata.enc`. This is clean, extensible (future files can be added -- e.g., `signature.sig`), and maps naturally to S3 prefixes.

### 6.4 Security Analysis Tree

**Source:** [`v0.6.30__brief__per-file-security-review-and-threat-modelling.md`](../../../../humans/dinis_cruz/briefs/02/25/v0.6.30__brief__per-file-security-review-and-threat-modelling.md)

The parallel security tree (`.security.json` files mirroring the codebase structure) is not a runtime data structure -- it is a development/review artefact. From an architecture perspective:

- **No runtime impact.** These files are not deployed, not loaded by the application, and do not affect the running system.
- **CI integration potential.** The security tree can be validated in CI: every new file in `src/` must have a corresponding `.security.json` in `security/`. This enforces security review coverage as a gating check.
- **Links to Lambda dependency optimization.** The dead code identification in security reviews directly feeds into the tree-shaking effort. Code flagged as dead by AppSec can be removed by DevOps in the dependency optimization phase.

## 7. Architectural Conflicts and Tensions

### 7.1 Human-Friendly URLs vs Zero Customer Data on Server

**Briefs in tension:** One-time secrets (security principle) vs PKI-friendly links (human-friendly URLs)

The one-time secret brief establishes a critical security principle: "a full server-side data leak should expose zero customer data." This means all identifiers must be opaque.

The PKI-friendly links brief proposes URLs like `/d/b12/quarterly-report` where `quarterly-report` is a human-friendly document name chosen by the sender. This name is stored on the server (as part of the URL path, which is logged by web servers, CloudFront, and API Gateway).

**The resolution is already in the briefs but needs to be made architecturally explicit:** In symmetric mode, identifiers are always opaque. In PKI mode, the URL contains a human-readable name, BUT the brief also states that friendly names should live in encrypted metadata, not in the URL. There is an inconsistency: the PKI-friendly links brief shows `/d/b12/quarterly-report` as a URL pattern, while the one-time secret brief says metadata should be encrypted.

**Architecture decision needed:** Does the PKI URL `/d/{key_prefix}/{doc_name}` store `doc_name` on the server or not? If it does, it violates the zero-customer-data principle. If it does not, the server needs a way to resolve a friendly name to an opaque ID, which means either (a) the sender creates a mapping that is stored encrypted, or (b) the `doc_name` in the URL is itself opaque and the friendly name is only in the encrypted metadata.

**Recommendation:** Make `doc_name` in PKI URLs an opaque, server-generated slug (not user-chosen). The human-friendly name lives exclusively in the encrypted `metadata.enc` file. The URL `/d/b12/xK9mP2qR` is opaque. The recipient sees "Quarterly Report from Alice" only after decryption. This maintains zero-customer-data-on-server consistently across all modes.

### 7.2 Data Room Metadata vs Zero-Knowledge Server

**Brief in tension:** Symmetric data room (server stores room names, user names, role mappings)

The symmetric data room brief describes a model where the server stores:
- Room names ("Project Alpha")
- User names ("Bob", "Alice")
- Role assignments (viewer, editor, admin)
- Key mappings (which short key maps to which encrypted master key copy)

This is a significant departure from the zero-knowledge model. The server now knows the names of rooms and users. In a full server breach, an attacker learns: "There is a data room called 'Project Alpha' with users named Alice, Bob, and Carol."

**The brief acknowledges this explicitly:** "In this mode, the admin has access to all data. The usernames are not PKI-verified -- they're just names. There's no point over-engineering security here."

**Architecture recommendation:** This is acceptable for the symmetric key data room, which is explicitly positioned as the usability-first, lower-security mode. BUT the server should hash room names and user names before storing them (similar to the IP address hashing approach). The server stores `sha256("Project Alpha")` instead of `"Project Alpha"`. The client resolves the hash. This gives the server plausible blindness to room names without adding user friction.

### 7.3 Static Assets Versioning vs Component Versioning

**Briefs in tension:** URL architecture (IFD versioned folders `ifd-v0.6.30/`) vs data room preview (compatible-versions manifest with per-component versions `v0.1.5`)

The URL architecture brief versions the entire frontend as a monolithic folder (`ifd-v0.6.30/`). The data room preview brief versions individual components independently (`vault-preview v0.1.5`, `vault-editor v0.1.6`). These are two different versioning strategies.

**The tension:** If the IFD folder is deployed as `ifd-v0.6.30/` and the compatible-versions manifest points to `components/vault-preview/v0.1.5/`, where does the component physically live? Inside the IFD folder or alongside it?

**Recommendation:** Two-tier versioning.

```
s3://sgsend-static/
  ifd-v0.6.30/             <- the IFD bundle (index.html, app.js, style.css)
  ifd-v0.6.32/             <- next IFD version
  components/               <- shared components, independently versioned
    send-preview/
      v0.1.5/send-preview.js
      v0.1.6/send-preview.js
    send-editor/
      v0.1.6/send-editor.js
  lib/                      <- shared libraries, independently versioned
    crypto-utils/
      v0.2.0/crypto-utils.js
  manifests/
    v0.6.30.json            <- compatible-versions manifest for v0.6.30
    v0.6.32.json            <- compatible-versions manifest for v0.6.32
```

The IFD bundle contains the page shell and routing logic. Shared components are loaded dynamically from the `components/` tree based on the manifest. This allows the IFD version and component versions to evolve independently.

### 7.4 Chrome Key Sync vs Vault Key Storage

**Brief in tension:** PKI-friendly links (Chrome password sync for private key portability) vs password generator (vault encryption with user's private key)

If private keys are stored in Chrome's password manager (synced via Google account), they are in a different storage location from the vault (which is in IndexedDB, encrypted, synced to SGraph Send's server). The user's key identity is split across two storage systems.

**Recommendation:** Chrome password sync is a research track, not a committed architecture. The primary key storage should be IndexedDB (managed by the Key Registry proposed in Section 3.4). Chrome password sync is an optional portability mechanism -- a way to move the master key between devices. The vault is the system of record.

## 8. Recommended Architecture Decisions

The following decisions should be made before implementation begins. They are ordered by dependency (later decisions depend on earlier ones).

### Decision 1: SGMETA Envelope Specification (Blocking)

**Context:** The SGMETA envelope format is used by the browser client, the MCP agent, and any future SDK. It is the wire format for all encrypted content.

**Options:**
- (A) Keep the current informal specification (documented in SKILL.md)
- (B) Formalise as a specification document with a version byte, publish in `library/docs/specs/`

**Recommendation:** (B). Add a version byte (`SGMETA\x01`) to the magic header. Document field types, byte ordering, maximum metadata length, and supported metadata keys. This is the most critical interoperability contract in the system.

### Decision 2: Key Registry Architecture (Blocking)

**Context:** Six encryption contexts, multiple key types, auto-selection needed.

**Options:**
- (A) Each feature manages its own keys in separate IndexedDB stores
- (B) Unified Key Registry with typed entries and context-based resolution

**Recommendation:** (B). A unified Key Registry prevents key management fragmentation and enables the auto-select-by-prefix UX described in the PKI-friendly links brief.

### Decision 3: Component Versioning Strategy (Blocking)

**Context:** IFD monolithic versioning vs per-component independent versioning.

**Options:**
- (A) Everything in the IFD folder, versioned as one unit
- (B) Two-tier: IFD shell + independently versioned shared components via manifest

**Recommendation:** (B). The compatible-versions manifest gives the flexibility needed for multiple consumers (user UI, admin UI, data room viewer, portable viewer) to share components while maintaining version independence.

### Decision 4: PKI URL Naming (Blocking for PKI implementation)

**Context:** Should PKI URLs contain human-readable document names?

**Options:**
- (A) `/d/{key_prefix}/{human_name}` -- human-readable, but name stored on server
- (B) `/d/{key_prefix}/{opaque_id}` -- opaque, friendly name only in encrypted metadata

**Recommendation:** (B). Consistent with the zero-customer-data principle. The friendly name is in `metadata.enc`, decrypted client-side.

### Decision 5: Data Room Metadata Privacy (Non-blocking but recommended)

**Context:** Room names and user names stored in plaintext on server.

**Options:**
- (A) Plaintext room and user names on server (current brief design)
- (B) Hashed room and user names on server
- (C) Encrypted room and user names (only decryptable with master key)

**Recommendation:** (A) for MVP, with a clear migration path to (C) when PKI data rooms are implemented. The symmetric data room is explicitly the usability-first mode. Adding hashing or encryption for room names adds complexity that contradicts the "over-indexes on usability" positioning. Document this as a known trade-off.

### Decision 6: S3/CloudFront Migration Sequencing (Operational)

**Context:** S3 migration and Lambda optimization are complementary but have a dependency.

**Recommended sequence:**
1. Set up S3 bucket and CloudFront distribution
2. Deploy static assets to S3 (keep Lambda serving as fallback)
3. Switch CloudFront behaviours to route static requests to S3
4. Remove static-serving code from Lambda
5. Begin Lambda dependency optimization (Phase 1: single zip)
6. Phase 2: strip non-essential files
7. Phase 3: tree-shake (only after profiling)

### Decision 7: Vault Entry Type System (Non-blocking)

**Context:** The vault now holds files, passwords, keys, and contacts.

**Options:**
- (A) Single IndexedDB store with `type` discriminator field
- (B) Separate IndexedDB object stores per type

**Recommendation:** (A). Simpler export/import, simpler backup, sufficient performance for personal-scale data.

### Decision 8: Client-Side Encrypted Logging Scope (Non-blocking)

**Context:** The one-time secret brief proposes client-side signed, encrypted logs submitted to the server.

**Options:**
- (A) Implement for secrets only
- (B) Implement as a platform-wide logging framework usable by all features

**Recommendation:** (B). Design the logging framework as a shared utility from day one. Data room access, vault operations, key imports, and file previews all benefit from client-side audit logging. The framework should be a composable module that any feature can import.
