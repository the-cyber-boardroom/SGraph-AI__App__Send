## 1. Executive Summary

The 28 briefs delivered on 23-24 February represent a deliberate and ambitious pivot: SGraph Send is evolving from an encrypted file transfer product into a **PKI-based trust infrastructure platform**. Having reviewed the 15 briefs assigned to the Architect role, my assessment is:

**The architectural direction is sound, coherent, and well-grounded.** The briefs form an internally consistent system where each component reinforces the others. The three load-bearing pillars are:

1. **PKI as the universal identity and trust primitive** -- every actor (user, agent, scanning service, data room) is a key pair
2. **MemoryFS as the universal storage abstraction** -- the same code runs against memory, S3, filesystem, ZIP, or SQLite backends
3. **The receiver-defines-the-rules principle** -- security requirements flow from the party at risk, not the party sending

These pillars are already partially implemented (PKI key pairs since v0.4.12, MemoryFS since day one, receiver experience since v0.5.24). The briefs extend rather than replace the existing architecture. This is evolution, not revolution.

**Critical concern:** The scope expansion is massive. The briefs describe a platform that could take years to fully realise. The sequencing and phasing of implementation will determine whether this vision delivers incrementally or collapses under its own weight. The architecture must be built in layers where each layer is independently useful.

**Key architectural insight from the briefs:** The same pattern -- portable encrypted containers with per-participant PKI access control and signed provenance -- applies at every scale: files in a transfer, files in a data room, paragraphs in a document, secrets in an agentic workflow, nodes in a trust graph. This fractal quality is the platform's most powerful architectural property.

## 2. Architecture Impact Assessment -- Core Briefs

### 2.1 Brief #2: Trust Web -- Reputation Through Usage

**Source:** [`v0.6.14__architecture__trust-web-reputation-through-usage.md`](../../../../humans/dinis_cruz/briefs/02/23/part-1/v0.6.14__architecture__trust-web-reputation-through-usage.md)

**Architecture Impact: HIGH**

The trust web introduces a new persistent data layer and a new computation model to the platform.

**What changes architecturally:**

- **New data store requirement.** Trust scores, transaction signals, complaint records, and graph edges need persistent storage. This data is server-side (unlike file content, which is client-side only). The server must store trust scores associated with public key hashes. This is the first time the server stores data that has semantic meaning beyond "encrypted blob at this ID."
- **New computation model.** Trust score computation is a function over transaction history. This is not a request-response API -- it is a graph computation that runs across the full history of a public key's interactions. This implies either batch computation (periodically recompute scores) or event-driven incremental updates.
- **Privacy trade-off is real and accepted.** The server will know which public keys communicate with which other public keys. This is metadata, not content. The brief correctly identifies this as analogous to email server metadata. From an architecture perspective, this means the server's data model expands from `{obj_id: encrypted_blob}` to `{obj_id: encrypted_blob, sender_key_hash: str, receiver_key_hash: str, timestamp: datetime, complaint_flag: bool}`.
- **Rate limiting becomes trust-dependent.** The current architecture has simple rate limits. Trust-based rate limiting requires the request pipeline to query trust scores on every upload/download. This adds latency to the hot path.

**Architecture recommendation:**

- Store trust data in a separate storage backend from file storage. The trust graph is a different data structure (relational/graph) from the file store (key-value blobs).
- Compute trust scores asynchronously. Do not block file transfers on trust computation. Use cached scores that are periodically refreshed.
- Design the trust score model to be pluggable -- the formula should be configurable, not hardcoded.

**Feasibility: Medium-term.** The trust web requires new data infrastructure but does not conflict with existing architecture. Can be added incrementally, starting with simple per-key counters and expanding to full graph computation.

### 2.2 Brief #7: External File Security Services -- PKI-Verified Scanning Chain

**Source:** [`v0.6.14__architecture__external-file-security-services-pki-chain.md`](../../../../humans/dinis_cruz/briefs/02/23/part-2/v0.6.14__architecture__external-file-security-services-pki-chain.md)

**Architecture Impact: HIGH**

This brief defines the mechanism by which untrusted senders prove their files have been scanned by trusted third parties.

**What changes architecturally:**

- **New blob type alongside encrypted files.** The server now stores not just `{encrypted_file}` but `{encrypted_file, [encrypted_scan_result_1, encrypted_scan_result_2, ...]}`. The server does not understand these blobs, but it must store and serve them as a bundle.
- **Client-side verification pipeline.** The receiver's browser must implement a multi-step verification chain: decrypt scan results, verify scanning service signatures, check file hash binding, validate timestamps. This is a non-trivial client-side computation pipeline.
- **Public key discovery for scanning services.** The receiver's browser needs to know the public keys of trusted scanning services. This requires a service registry -- either platform-curated or receiver-defined.
- **Hash-as-binding-primitive creates a new architectural pattern.** The SHA-256 hash of the unencrypted file becomes the universal binding element. This hash must be computed before encryption, included in scan results, and verified after decryption. This is a clean pattern that should be formalised as a platform primitive.

**Architecture recommendation:**

- Define a `ScanResult` schema in `Type_Safe` that captures the signed result structure.
- The file upload API must accept optional scan result blobs alongside the encrypted file.
- Implement scan result verification as a composable client-side module that can be used in the browser, CLI, and SDK.
- Start with platform-curated scanning service list. Receiver-defined lists add UI and key management complexity.

**Feasibility: Medium-term.** The server-side changes are minimal (store additional blobs). The client-side changes are substantial (verification pipeline, scanning service integration). The biggest dependency is on external scanning services supporting the required API pattern -- most current services (VirusTotal, MetaDefender) do not natively sign results with PKI keys, so an adapter layer is needed.

### 2.3 Brief #14: Portable Data Rooms -- ZIP as Encrypted Storage

**Source:** [`v0.6.14__architecture__portable-data-rooms-zip-encrypted-storage.md`](../../../../humans/dinis_cruz/briefs/02/23/part-3/v0.6.14__architecture__portable-data-rooms-zip-encrypted-storage.md)

**Architecture Impact: CRITICAL**

This is the most architecturally significant brief. The .sgroom format is the cornerstone that enables portable data rooms, supply chain use cases, air-gapped environments, and (via .sgworkflow) agentic workflows.

**What changes architecturally:**

- **New MemoryFS backend: ZIP.** The MemoryFS abstraction was designed for exactly this extensibility. Adding a ZIP backend means the entire application layer -- file storage, indexing, access control -- works without modification against a ZIP file. This is the highest-leverage change in the entire batch: one new backend, many new capabilities.
- **New file format: .sgroom.** The ZIP archive structure (manifest, config, files, index, audit, assets) must be formally specified. This is a public format that external tools will need to read and write.
- **Offline-first capability.** Portable data rooms work without a server. This means the viewer (browser or desktop) must include the full application logic, not just a client that calls APIs. The IFD components must be self-contained enough to run from a local file.
- **Multi-participant per-key encryption within a single file.** The encrypted index (`index.json.enc`) must be encrypted differently for each participant -- each sees their own view. This requires either multiple encrypted copies of the index or a hierarchical key scheme where participants decrypt their portions.
- **Sync and merge.** The brief mentions offline-then-sync workflows. Merging changes from two independently modified .sgroom files is a conflict resolution problem. This needs careful design -- the audit log with signatures provides the ordering primitives, but the merge algorithm is non-trivial.

**Architecture recommendation:**

- Prioritise the ZIP MemoryFS backend. This is the single highest-value implementation in the batch.
- Formally specify the .sgroom format version 1.0. Treat it as a public API that must remain backward-compatible.
- For v1, implement per-participant encryption as multiple encrypted copies of the index (simple, correct). Optimise with hierarchical keys later.
- Defer sync/merge to a later phase. For now, .sgroom files are append-only: each participant adds, nobody modifies previous entries.

**Feasibility: Near-term.** The ZIP backend is a well-understood problem. Python's `zipfile` module handles the I/O. The encryption layer is already implemented. The primary work is defining the file format and building the viewer.

### 2.4 Brief #22: Agent Isolation Reality -- Process-Level Security

**Source:** [`v0.6.14__architecture__agent-isolation-reality-process-level-security.md`](../../../../humans/dinis_cruz/briefs/02/23/part-4/v0.6.14__architecture__agent-isolation-reality-process-level-security.md)

**Architecture Impact: CRITICAL (correction)**

This brief is an honest and necessary correction to the original agentic workflows vision. It elevates the architectural requirements from "encryption is enough" to "encryption + container isolation is required."

**What changes architecturally:**

- **Agents must run in separate containers/VMs.** This is a hard infrastructure requirement. The orchestrator must be a container scheduler, not just a task dispatcher. This means Docker, Kubernetes, or serverless functions become part of the agent runtime.
- **The .sgworkflow file becomes the sole inter-agent communication channel.** No shared memory, no shared filesystem, no direct network connections between agent containers. The .sgworkflow file is passed between containers by the orchestrator. This is a clean, auditable, and secure pattern.
- **Secret injection must use container-native mechanisms.** Agent private keys are injected via Kubernetes secrets, AWS Secrets Manager, or equivalent. The .sgworkflow file never contains private keys.
- **Ephemeral containers.** Each agent container is created for a task and destroyed after completion. Key material does not persist. This means container startup time is on the critical path for workflow latency.

**Architecture recommendation:**

- Adopt the recommended architecture exactly as described: orchestrator with per-agent containers, .sgworkflow as the coordination file.
- For development/testing: use the "separate process, different UID" model. It provides moderate isolation at low overhead.
- For production: pre-warmed container pool with hardened Docker (seccomp, AppArmor, read-only rootfs, dropped capabilities).
- Design the .sgworkflow format to be the same .sgroom format with an agent-specific manifest. Do not create a parallel format.

**Feasibility: Medium-term.** Container orchestration is well-understood infrastructure. The novel part is integrating it with the .sgworkflow coordination pattern. The primary risk is latency -- spinning up containers per agent adds seconds to each step.

### 2.5 Brief #23: GitHub as PKI Registry

**Source:** [`v0.6.14__architecture__github-as-pki-registry.md`](../../../../humans/dinis_cruz/briefs/02/23/part-4/v0.6.14__architecture__github-as-pki-registry.md)

**Architecture Impact: HIGH (cold-start breakthrough)**

This brief solves the biggest adoption blocker for any PKI system: the recipient does not have a key pair. By leveraging GitHub's existing key infrastructure, millions of developers already have published public keys.

**What changes architecturally:**

- **New key type support.** The current encryption uses AES-256-GCM via Web Crypto API. Adding Ed25519 (SSH keys) and GPG key support requires new client-side crypto libraries: libsodium (or TweetNaCl) for Ed25519-to-X25519 conversion, and OpenPGP.js for GPG key handling.
- **External key discovery.** The client must fetch public keys from `github.com/{username}.keys` and `github.com/{username}.gpg`. This is a CORS-sensitive cross-origin fetch that may require a proxy or API relay.
- **Multiple encryption schemes coexisting.** Currently all encryption is AES-256-GCM. With GitHub keys, the system must support: AES-256-GCM (existing), NaCl box/sealed box (Ed25519/X25519), and PGP encryption (GPG keys). The file format must indicate which encryption scheme was used.
- **Identity signals from GitHub.** Trust web integration can use GitHub account metadata (age, contribution history, 2FA, org membership) as initial trust signals. This requires GitHub API calls, subject to rate limiting.

**Architecture recommendation:**

- Phase 1: Add GPG encryption/decryption via OpenPGP.js. GPG keys are already designed for encryption -- no conversion needed.
- Phase 2: Add Ed25519-to-X25519 conversion via libsodium/TweetNaCl. This is mathematically sound and widely used.
- Implement a key discovery service as a thin API endpoint that fetches and caches GitHub keys. Do not have the browser fetch directly from GitHub (CORS issues, rate limiting).
- The encrypted file format must include a metadata header indicating the encryption scheme and the recipient's public key fingerprint.

**Feasibility: Near-term.** Libraries exist for all crypto operations. The primary integration work is UI (username lookup, key selection) and the key caching/proxy layer.

## 3. Architecture Impact Assessment -- Secondary Briefs

### 3.1 Brief #1: Receiver Environment Analysis

**Source:** [`v0.6.14__architecture__receiver-environment-analysis.md`](../../../../humans/dinis_cruz/briefs/02/23/part-1/v0.6.14__architecture__receiver-environment-analysis.md)

**Architecture Impact: MEDIUM**

Introduces client-side environment data collection and encrypted reporting to the sender. Architecturally clean -- all collection is client-side, the report is encrypted for the sender, the server sees nothing. The main implication is a new message type (environment report) that must be stored alongside the file transfer.

**Key concern:** Anti-gaming is acknowledged as limited. An attacker with full browser control can fake everything. This should be clearly documented as advisory, not authoritative.

### 3.2 Brief #3: Dual Authorization File Delivery

**Source:** [`v0.6.14__brief__dual-authorization-file-delivery.md`](../../../../humans/dinis_cruz/briefs/02/23/part-1/v0.6.14__brief__dual-authorization-file-delivery.md)

**Architecture Impact: MEDIUM-HIGH**

Changes the file transfer from a one-step to a multi-step state machine: `uploaded -> challenged -> receiver_responded -> sender_reviewed -> released -> downloaded`. This requires server-side state tracking for each transfer and a notification mechanism (polling for MVP, WebSocket/SSE for production).

**Key insight:** Dual authorization is where the PKI key exchange happens transparently. The receiver generates a key pair during the challenge response, and the sender receives the receiver's public key. This bootstraps full PKI without the user knowing.

### 3.3 Brief #4: File Security Scanning Pipeline

**Source:** [`v0.6.14__brief__file-security-scanning-pipeline.md`](../../../../humans/dinis_cruz/briefs/02/23/part-1/v0.6.14__brief__file-security-scanning-pipeline.md)

**Architecture Impact: HIGH**

Defines the client-side scanning strategy. The most architecturally significant elements are:
- ClamAV compiled to WASM for in-browser malware scanning
- LLM-powered analysis of extracted code (macros, scripts)
- The privacy spectrum: local scanning (WASM) vs cloud scanning (LLM API) is a user choice

**Key concern:** WASM-compiled ClamAV will be large (signature database). Need to evaluate bundle size impact and loading strategy (lazy-load, service worker cache).

### 3.4 Brief #6: Cross-Domain File Security

**Source:** [`v0.6.14__architecture__cross-domain-file-security.md`](../../../../humans/dinis_cruz/briefs/02/23/part-2/v0.6.14__architecture__cross-domain-file-security.md)

**Architecture Impact: MEDIUM**

Applies the scanning + PKI pattern to domain boundaries. Architecturally, this is a deployment pattern more than a new component: SG/Send checkpoints at domain boundaries. The key new concept is **per-data-room security policies** -- a configuration layer that defines required scanning, allowed file types, CDR requirements, and minimum trust scores.

### 3.5 Brief #8: Network-Level Threat Detection

**Source:** [`v0.6.14__architecture__network-level-threat-detection-and-attacker-destruction.md`](../../../../humans/dinis_cruz/briefs/02/23/part-2/v0.6.14__architecture__network-level-threat-detection-and-attacker-destruction.md)

**Architecture Impact: MEDIUM-HIGH**

Extends the trust web from individual scores to network-level graph analysis. Architecturally, this requires:
- A graph database or graph computation layer for cluster analysis
- Batch processing pipeline for pattern detection
- A "silent mode" capability where detection does not alter the user experience

**Key concern:** The privacy trade-off deepens here. Network-level detection requires more metadata than individual trust scores. The server needs to compute graph clustering algorithms over communication patterns. This should be explicitly documented in the privacy model.

### 3.6 Brief #15: SaaS Zero-Knowledge File Storage

**Source:** [`v0.6.14__architecture__saas-zero-knowledge-file-storage-credit-card-model.md`](../../../../humans/dinis_cruz/briefs/02/23/part-3/v0.6.14__architecture__saas-zero-knowledge-file-storage-credit-card-model.md)

**Architecture Impact: HIGH (strategic)**

The JS library (`sg-send.js`) for drop-in zero-knowledge file encryption in any SaaS product. Architecturally, this means packaging the client-side encryption/decryption logic as a standalone, dependency-free JavaScript library. This is an extraction of existing browser-side code into a distributable package.

**Key implication:** The library must be independent of the SG/Send server. It should work with any storage backend -- the SaaS company's own S3, their database, whatever. Our server is optional. This reinforces the zero lock-in principle.

### 3.7 Brief #17: Data Room as Code

**Source:** [`v0.6.14__dev-brief__data-room-as-code-github-configuration.md`](../../../../humans/dinis_cruz/briefs/02/23/part-3/v0.6.14__dev-brief__data-room-as-code-github-configuration.md)

**Architecture Impact: MEDIUM**

Data room configuration managed via Git. Architecturally clean separation: admin UI writes to GitHub, CI deploys to runtime, runtime has no GitHub access. This follows standard infrastructure-as-code patterns.

**Key insight:** The admin UI is the blast-radius boundary. If the data room runtime is compromised, the attacker cannot modify configuration (no GitHub token). If the admin UI is compromised, the attacker can modify configuration but changes are auditable via Git history.

### 3.8 Brief #28: PKI-Secured Agentic Workflows

**Source:** [`v0.6.14__vision__pki-secured-agentic-workflows-prompt-injection-containment.md`](../../../../humans/dinis_cruz/briefs/02/23/part-4/v0.6.14__vision__pki-secured-agentic-workflows-prompt-injection-containment.md)

**Architecture Impact: HIGH (vision)**

The vision document for applying portable data room patterns to multi-agent workflows. Read alongside Brief #22 (Agent Isolation Reality), this defines:
- The .sgworkflow format (isomorphic to .sgroom)
- Issues-FS as the agent coordination layer
- Per-agent secret isolation via encryption + containers
- Time-bounded secret tokens

**Architecture assessment:** The isomorphism between supply chain data rooms and agentic workflows is architecturally elegant. If .sgroom works, .sgworkflow works -- they use the same primitives. The risk is that agentic workflows have real-time latency requirements that portable ZIPs may not satisfy. Consider SQLite or shared-object-store backends for latency-sensitive agent workflows.

### 3.9 Brief #13: Fractal Document Signing

**Source:** [`v0.6.14__architecture__fractal-document-signing-pki-paragraphs.md`](../../../../humans/dinis_cruz/briefs/02/23/part-3/v0.6.14__architecture__fractal-document-signing-pki-paragraphs.md)

**Architecture Impact: MEDIUM (strategic/vision)**

Applies the .sgroom model to individual documents. The .sgdoc format is another ZIP-like container with per-section signatures. Architecturally, this is a natural extension of the portable data room -- a document is a data room where each "file" is a paragraph.

**Key concern:** The EULA third-party verification model is compelling but requires a reviewing-organisation ecosystem that does not exist yet. This is a chicken-and-egg problem: users want signed EULAs, but review organisations need demand before they invest in signing clauses.

### 3.10 Brief #27: Zero Lock-In and Usage Metrics (DRU/MRU/ARU)

**Source:** [`v0.6.14__strategy__zero-lock-in-and-usage-metrics-dru-mru-aru.md`](../../../../humans/dinis_cruz/briefs/02/23/part-4/v0.6.14__strategy__zero-lock-in-and-usage-metrics-dru-mru-aru.md)

**Architecture Impact: FOUNDATIONAL (design constraint)**

This is not a feature brief -- it is a constraint that applies to every architectural decision. Zero lock-in means:
- All encryption must use standard algorithms with documented "without SG/Send" paths
- All file formats must be readable by standard tools (GPG, AGE, OpenSSL)
- The JS library must work independently of our server
- Data export must always be possible

**Architecture implication:** The .sgroom and .sgdoc formats should use standard encryption primitives (NaCl, PGP) rather than custom schemes. The "thin wrapper around standard crypto" principle must be enforced in format specification reviews.

## 4. Integration Points with Existing Architecture

The briefs expand the existing SGraph Send architecture. Here is how each major component connects:

### 4.1 Two Lambda Functions (Public + Admin)

| Brief | Impact on Lambdas |
|-------|-------------------|
| Trust Web (#2) | Public Lambda needs trust score lookup on upload/download path. Admin Lambda needs trust management APIs. |
| Dual Authorization (#3) | Public Lambda needs new transfer state machine (challenge/response/release). |
| Scanning Chain (#7) | Public Lambda needs to accept and store scan result blobs alongside files. |
| Data Room as Code (#17) | Admin Lambda potentially serves data room runtime. Admin UI is separate. |

**Assessment:** The two-Lambda model holds. Trust and scanning add APIs to both Lambdas but do not require new Lambda functions. The data room admin is a separate deployment.

### 4.2 Memory-FS (Storage Abstraction)

| Brief | Impact on Memory-FS |
|-------|---------------------|
| Portable Data Rooms (#14) | New ZIP backend. Highest priority. |
| Data Room as Code (#17) | GitHub repo as a backend (via CI deploy to existing backend). |
| Agentic Workflows (#28) | .sgworkflow uses the same MemoryFS abstraction. |
| SaaS Integration (#15) | Library works with ANY storage, not just Memory-FS. |

**Assessment:** Memory-FS is the architectural linchpin. The ZIP backend is the single most important new implementation. If it works, portable data rooms, agentic workflows, and offline-first capabilities all become possible.

### 4.3 Client-Side Encryption (Web Crypto API)

| Brief | Impact on Client Encryption |
|-------|-----------------------------|
| GitHub PKI (#23) | New crypto schemes: NaCl (Ed25519/X25519), PGP (OpenPGP.js). |
| Scanning Chain (#7) | Receiver-side verification of signed scan results. |
| Fractal Signing (#13) | Per-paragraph PKI signing in the browser. |
| SaaS Library (#15) | Extraction of encryption logic into standalone library. |

**Assessment:** The client-side crypto surface area is expanding significantly. Currently it is AES-256-GCM only. The briefs add NaCl, PGP, and signature verification. This needs a modular crypto abstraction layer on the client side -- a `CryptoProvider` interface with implementations for each scheme.

### 4.4 Transfer Model (Upload/Download)

| Brief | Impact on Transfer Flow |
|-------|------------------------|
| Dual Authorization (#3) | Two-step state machine replaces single-step upload/download. |
| Receiver Environment (#1) | Environment report attached to download event. |
| Trust Web (#2) | Trust check on upload and download. |
| Scanning (#4, #7) | Scan results computed before upload, verified on download. |

**Assessment:** The transfer flow is becoming richer. The current model (upload blob, share link, download blob) will coexist with the enhanced model (upload with scans, challenge receiver, verify environment, release, download with verification). Both must be supported -- not all transfers need dual authorization.

## 7. Technical Risks

### 7.1 Scope Creep -- The Biggest Risk

The 28 briefs describe a platform that could occupy a large team for years. The risk is that the Explorer team tries to implement everything in parallel, resulting in many half-built features and no complete ones.

**Mitigation:** Strict phase gating. Each phase must produce independently usable features. The ZIP backend is usable without the trust web. GitHub key fetch is usable without scanning. Dual authorization is usable without environment analysis.

### 7.2 Client-Side Crypto Complexity

Adding NaCl, PGP, and WASM-compiled ClamAV to the browser client increases bundle size and attack surface. Each crypto library is a potential vulnerability.

**Mitigation:** Lazy-load crypto modules. Only load OpenPGP.js when GPG encryption is needed. Only load ClamAV WASM when scanning is triggered. Audit each library before inclusion.

### 7.3 Privacy Model Expansion

The trust web and network detection require the server to store and compute over communication metadata. This is a meaningful departure from the pure zero-knowledge model.

**Mitigation:** Document the privacy model explicitly. Define what metadata the server stores, why, and what it can infer. Provide a clear user-facing disclosure. Consider a "privacy mode" where trust features are disabled and no metadata is stored (at the cost of no trust signals).

### 7.4 External Service Dependencies

The scanning chain depends on external services (VirusTotal, CDR providers) adopting PKI-signed results. Current services do not natively support this.

**Mitigation:** Build an adapter layer that wraps existing service responses with our own signatures. The adapter signs "I fetched this result from VirusTotal at this time" -- not ideal (the scanning service itself should sign) but practical for MVP. Document the trust difference between service-signed and adapter-signed results.

### 7.5 Agentic Workflow Latency

Container-per-agent architecture adds startup latency to each workflow step. For a 5-agent workflow, this could add 10-30 seconds of overhead.

**Mitigation:** Pre-warmed container pools. For development, use process-level isolation (faster but less secure). Accept the latency trade-off for production security.

### 7.6 Format Versioning and Backward Compatibility

The .sgroom, .sgdoc, and .sgworkflow formats will evolve. Once files are created in v1 format, we must be able to read them in v2, v3, and beyond.

**Mitigation:** Include a format version in every manifest. Design the format extensibly from day one -- new fields are additive, old fields are never removed. Follow semantic versioning for the format itself.

### 7.7 GitHub as a Single Point of Failure for PKI

If GitHub is down, key discovery fails. If GitHub is compromised, keys could be replaced.

**Mitigation:** Support multiple registries (GitLab, Keybase, custom). Cache keys locally after first fetch (Trust On First Use). Alert users if a key changes. Never make GitHub the ONLY key source.

## 8. Questions for Dinis

### 8.1 Scope and Priority

1. **What is the target timeline for Phase 1 (ZIP backend + GitHub keys)?** This determines how aggressively we staff the Explorer team on these items.

2. **Should the trust web be designed for SGraph Send users only, or should it be a general-purpose trust infrastructure that other products can use?** The answer affects the data model and API design significantly.

3. **The SaaS JS library (Brief #15) -- is this a near-term product priority or a strategic vision?** If near-term, it should be extracted now while the API is still simple. If it waits, the API surface will grow and extraction becomes harder.

### 8.2 Architecture Decisions Needed

4. **For the .sgroom format: should we use a single encryption scheme (AES-256-GCM for all) or support multiple schemes (AES-256-GCM + NaCl + PGP)?** Multiple schemes maximise interoperability (zero lock-in) but increase format complexity. Single scheme is simpler but locks recipients into one crypto stack.

5. **For the scanning chain: should we build our own scanning adapter service, or invest in getting existing services (VirusTotal) to adopt PKI-signed results?** Building our own is faster. Convincing existing services is harder but more trustworthy.

6. **For agent isolation: should the 19-agent team currently running the project be the first dogfooding target for .sgworkflow?** This would be a powerful demonstration but requires the container infrastructure to be production-ready.

### 8.3 Privacy Trade-offs

7. **The trust web requires the server to know communication patterns (which keys talk to which keys). Is there a threshold beyond which this metadata collection becomes unacceptable?** For example, network-level cluster detection (Brief #8) requires even more metadata. Should there be a "maximum metadata" boundary?

8. **Should users be able to opt out of the trust web entirely?** If yes, what access restrictions apply to zero-trust users?

### 8.4 Business Model

9. **The zero lock-in principle (Brief #27) means our encryption is standard and our formats are open. Competitors could build compatible tools. Is this intentional, or should we retain some proprietary elements?** (I believe from the brief that this is intentional, but want explicit confirmation given the investment implications.)

10. **For the credit card model (Brief #15), who is the first target SaaS partner?** Having a concrete partner in mind would sharpen the JS library design.
