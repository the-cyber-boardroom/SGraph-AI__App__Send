## Executive Summary

The PKI Messaging system is a fully client-side cryptographic communication layer built as a single Web Component (`<pki-manager>`). It provides **confidentiality** (hybrid encryption) and **sender authentication** (digital signatures) with zero server involvement. The server serves the static HTML/JS; every cryptographic operation happens in the browser.

## Cryptographic Algorithm Stack

```
┌────────────────────────────────────────────────────────────┐
│                    Algorithm Layers                         │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  CONFIDENTIALITY (hybrid encryption)                       │
│  ┌──────────────────────┐  ┌────────────────────────────┐ │
│  │  RSA-OAEP 4096-bit   │  │  AES-256-GCM              │ │
│  │  Key encapsulation    │  │  Symmetric encryption     │ │
│  │  Wraps/unwraps AES    │──▶  Encrypts/decrypts data   │ │
│  │  session key (32 B)   │  │  96-bit IV, auth tag      │ │
│  └──────────────────────┘  └────────────────────────────┘ │
│                                                            │
│  AUTHENTICITY (digital signature)                          │
│  ┌──────────────────────────────────────────────────────┐ │
│  │  ECDSA P-256 + SHA-256                               │ │
│  │  Signs ciphertext bytes → proves sender identity     │ │
│  │  64-byte signature                                   │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
│  IDENTIFICATION                                            │
│  ┌──────────────────────────────────────────────────────┐ │
│  │  SHA-256 fingerprint (truncated to 8 bytes / 16 hex) │ │
│  │  Format: sha256:a1b2c3d4e5f6g7h8                     │ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## Public Key Exchange Format

When the user clicks **"Copy Public Key"**, the component builds a JSON bundle:

```
┌─────────────────────────────────────────────────┐
│              Key Bundle (JSON)                    │
│                                                  │
│  {                                               │
│    "v": 1,                                       │
│    "encrypt": "-----BEGIN PUBLIC KEY-----\n..."  │  ◀── RSA-OAEP 4096 SPKI
│    "sign":    "-----BEGIN PUBLIC KEY-----\n..."  │  ◀── ECDSA P-256 SPKI
│  }                                               │
│                                                  │
│  Clipboard ──▶ Paste to recipient                │
│                (Slack, email, etc.)               │
└─────────────────────────────────────────────────┘
```

On import, the component auto-detects the format:

```
Raw text starts with '{'  ──▶  Parse as JSON bundle
                                ├── bundle.encrypt ──▶ import as RSA-OAEP
                                └── bundle.sign    ──▶ import as ECDSA (if present)

Raw text starts with '---' ──▶  Legacy PEM
                                └── try RSA-OAEP, fallback to ECDH
```

## Payload Wire Format

```
Payload v2 (current — signed):

  btoa(JSON.stringify({
    "v": 2,                          ◀── format version
    "w": "base64(RSA-OAEP(AES-key))", ◀── 512 bytes → ~684 base64 chars
    "i": "base64(IV)",                ◀── 12 bytes → 16 base64 chars
    "c": "base64(AES-GCM(plaintext))", ◀── variable length
    "s": "base64(ECDSA-sig)",         ◀── 64 bytes → ~88 base64 chars
    "f": "sha256:a1b2c3d4e5f6g7h8"   ◀── sender's signing fingerprint
  }))

  Final output: single base64 string, ~1.5KB+ overhead for a short message


Payload v1 (legacy — unsigned, still accepted on decrypt):

  btoa(JSON.stringify({
    "v": 1,
    "w": "...",
    "i": "...",
    "c": "..."
  }))

  No "s" or "f" fields.
```

## Security Properties

| Property | Status | Mechanism |
|----------|--------|-----------|
| **Confidentiality** | Yes | RSA-OAEP 4096 + AES-256-GCM hybrid |
| **Integrity** | Yes | AES-GCM authentication tag (implicit) |
| **Sender authentication** | Yes (v2) | ECDSA P-256 signature over ciphertext |
| **Non-repudiation** | Partial | Signature proves sender had the private key; no timestamping |
| **Forward secrecy** | No | Same RSA key pair used for all messages |
| **Key non-extractability** | Yes | Browser-enforced via Web Crypto API |
| **Server zero-knowledge** | Yes | All crypto is client-side; server serves static files only |

### What ECDSA Signs

The signature covers the **ciphertext bytes** (`payload.c` decoded from base64), not the full payload JSON. This means:
- Ciphertext integrity is cryptographically proven by the signature
- The AES-GCM auth tag separately ensures decryption will fail if any ciphertext byte is modified
- The IV (`payload.i`) and wrapped key (`payload.w`) are not directly signed, but tampering with either will cause AES-GCM decryption to fail

### Trust Model

```
┌──────────────────────────────────────────────────────────────┐
│                      Trust Assumptions                        │
│                                                              │
│  1. Key exchange is trusted (manual copy-paste)              │
│     └── "I pasted the key, so I know who it came from"       │
│                                                              │
│  2. Browser is trusted (runs crypto operations)              │
│     └── Web Crypto API enforces non-extractability           │
│                                                              │
│  3. No trusted third party / CA                              │
│     └── Direct key exchange, not PKI hierarchy               │
│                                                              │
│  4. No key revocation                                        │
│     └── Delete contact = local only; no broadcast            │
│                                                              │
│  5. No key expiry                                            │
│     └── Keys persist until browser data is cleared           │
└──────────────────────────────────────────────────────────────┘
```
