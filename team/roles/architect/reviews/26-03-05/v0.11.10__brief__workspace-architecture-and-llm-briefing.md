**Live at:** workspace.sgraph.ai (since 4 March 2026)

## 1. Why SG/Workspace Exists

SGraph Send stores files with zero-knowledge encryption -- the server never sees plaintext. SGraph Vault builds on top of Send to provide an encrypted filesystem (folders, files, metadata) stored as encrypted transfers. But storing encrypted files is only half the story. Users need to **work on** those documents: summarise them, reformat them, extract data, apply transformations.

SG/Workspace bridges the gap between encrypted vault storage and useful document work. It is an in-browser document transformation workbench that:

1. **Decrypts vault files client-side** -- files are fetched from send.sgraph.ai as ciphertext, decrypted in the browser via PBKDF2-derived AES-256-GCM keys, and displayed. The server never sees the plaintext at any point.

2. **Sends LLM requests directly from the browser to the LLM provider** -- API calls to OpenRouter or Ollama go from `fetch()` in the browser straight to the provider's API endpoint. They do not pass through any SGraph server. API keys are stored in `localStorage` only. This maintains the zero-knowledge guarantee: the SGraph infrastructure never sees user content or LLM interactions.

3. **Executes JavaScript transformations in a sandboxed iframe** -- user scripts run against source HTML in an iframe with `sandbox="allow-scripts"` (no `allow-same-origin`), so they cannot access the parent page's DOM, cookies, or localStorage.

4. **Saves results back to the vault** -- transformed content is re-encrypted client-side and written back to the vault as new files.

The core use case is **document transformation**: load an HTML/Markdown/code file from the vault, describe a transformation in natural language or write a JS script, view the result, and save it back. The LLM assists with both generating transformation descriptions and writing the scripts themselves (via the "Gen JS" mode).

### 2.3 LLM Provider Integration

Two providers supported in v0.1.0:

**OpenRouter (cloud):**
- Endpoint: `https://openrouter.ai/api/v1/chat/completions`
- Auth: Bearer token (API key stored in localStorage only)
- Streaming: Server-Sent Events (SSE), parsed via `_readSSE()` in llm-orchestrator
- Non-streaming: Standard JSON response
- Model list: fetched from `/api/v1/models` on connect
- Features: token usage stats, cost calculation from model pricing, finish reason, native request ID
- Multimodal: supports pasted images as base64 `image_url` content parts

**Ollama (local):**
- Endpoint: configurable, default `http://localhost:11434`
- Auth: none (local-only)
- Streaming: NDJSON (newline-delimited JSON), parsed via `_readNDJSON()` in llm-orchestrator
- Non-streaming: Standard JSON via `/api/generate`
- Model list: fetched from `/api/tags` on connect
- Features: token usage from `prompt_eval_count`/`eval_count`
- Multimodal: supports images as base64 strings (Ollama format, without data URL prefix)

### 2.5 Document Transformation Pipeline

The complete data flow from source to saved result:

1. **User opens vault** via vault key (passphrase:vault_id) in `<vault-panel>`
2. **User clicks a file** -- `<vault-panel>` emits `file-selected` event
3. **Source viewer loads the file** -- `<document-viewer data-role="source">` catches the event, calls `vault.getFile()` (downloads encrypted transfer from send.sgraph.ai, decrypts client-side), detects type, renders
4. **User writes a prompt** in `<llm-prompt-input>` (or selects a built-in prompt from `<prompt-library>`)
5. **User clicks Send** (or Ctrl+Enter) -- `<llm-prompt-input>` emits `llm-send` with prompt text + include flags + images
6. **Orchestrator assembles the full prompt** -- `<llm-orchestrator>` gathers artifacts based on include flags (source HTML, data content, script code, result content, console output), appends them to the user prompt with section markers (e.g. `--- SOURCE HTML ---`)
7. **Orchestrator calls the LLM** -- direct `fetch()` from browser to OpenRouter or Ollama API
8. **Response streams into output** -- `<llm-output>` shows streaming text, `<llm-stats>` shows token counts and cost
9. **User routes output** -- Copy-to buttons send LLM response to Script, Data, or Result panel
10. **User runs a script** -- `<script-editor>` sends source HTML + user script to `<js-executor>` which runs in sandboxed iframe
11. **Result appears** -- `<document-viewer data-role="transform">` shows the transformed output
12. **User saves to vault** -- Save button re-encrypts content via SGVault and uploads to send.sgraph.ai

### 2.6 JavaScript Execution Sandbox

The `js-executor` component (`window.sgraphWorkspace.executeJS`) creates a hidden iframe with:

```
sandbox="allow-scripts"
```

This means the script CAN execute JavaScript but CANNOT:
- Access the parent page's DOM (`allow-same-origin` is NOT set)
- Read localStorage, cookies, or session storage
- Navigate the parent frame
- Open popups

The iframe receives source HTML as its body content. Console methods are monkey-patched to send structured data back via `postMessage`. If the user script returns a value, that value becomes the result. If it returns nothing, the modified `document.documentElement.outerHTML` is returned.

### 3.2 Web Components + CustomEvent Pattern

All components are plain `HTMLElement` subclasses. There is no framework. Communication happens via:

1. **EventBus** -- the primary pub/sub mechanism (see Section 3.3)
2. **Direct DOM queries** -- components find each other via `document.querySelector()` when they need to call methods directly (e.g., `document.querySelector('llm-connection').isConnected()`)
3. **Light DOM rendering** -- components write HTML directly to `this.innerHTML` (no Shadow DOM)

Each component follows this lifecycle pattern:

```javascript
class MyComponent extends HTMLElement {
    constructor()         { /* init state, load from localStorage */ }
    connectedCallback()   { /* render, subscribe to EventBus, emit ready event */ }
    disconnectedCallback() { /* unsubscribe from EventBus, cleanup */ }
    _render()             { /* set this.innerHTML, call _bind() */ }
    _bind()               { /* attach DOM event listeners */ }
}
customElements.define('my-component', MyComponent);
```

### 3.5 Script Loading Order

The `index.html` loads scripts in 6 layers. Order matters -- each layer depends on the previous ones being available.

| Layer | What Loads | Why This Order |

| **1: Workspace Init** | `js/workspace-init.js` | Creates `window.sgraphWorkspace`, `EventBus`, `MessagesService`. Everything else depends on these existing. |

### 3.6 LLM Provider Abstraction

The orchestrator abstracts over two provider protocols:

**OpenRouter (SSE streaming):**
```
POST https://openrouter.ai/api/v1/chat/completions
Headers: Authorization: Bearer <key>, Content-Type: application/json
Body: { model, messages: [{role, content}], stream: true/false }

Streaming response: text/event-stream
  data: {"choices":[{"delta":{"content":"token"}}]}
  data: {"choices":[{"finish_reason":"stop"}],"usage":{...}}
  data: [DONE]
```

**Ollama (NDJSON streaming):**
```
POST http://localhost:11434/api/generate
Body: { model, prompt, system, stream: true/false, images?: [base64] }

Streaming response: application/x-ndjson
  {"response":"token"}
  {"response":"token","done":true,"prompt_eval_count":N,"eval_count":N}
```

The orchestrator's `_readSSE()` and `_readNDJSON()` methods handle the streaming protocol differences, both calling `onChunk(text)` for incremental UI updates.

### 3.7 Zero-Knowledge Data Flow

Here is the complete data flow showing what each party sees:

```
USER'S BROWSER                        send.sgraph.ai              LLM Provider
     |                                     |                         |
     |--- Vault key (passphrase:id) ------>|                         |
     |    Derives AES-256-GCM key          |                         |
     |    (PBKDF2, 600k iterations)        |                         |
     |                                     |                         |
     |--- GET /transfers/{id} ------------>|                         |
     |<-- Encrypted ciphertext ------------|                         |
     |                                     |                         |
     |    Decrypt (AES-256-GCM)            |                         |
     |    = vault tree (JSON)              |                         |
     |                                     |                         |
     |--- GET /transfers/{file-id} ------->|                         |
     |<-- Encrypted file ciphertext -------|                         |
     |                                     |                         |
     |    Decrypt = plaintext file         |                         |
     |    Render in document-viewer        |                         |
     |                                     |                         |
     |--- POST /chat/completions ---------|------------------------>|
     |    (prompt + source content)        |  (server never sees    |
     |<-- LLM response -------------------|------------------------|
     |                                     |  this traffic)         |
     |    Display in llm-output            |                         |
     |                                     |                         |
     |    Encrypt result (AES-256-GCM)     |                         |
     |--- POST /transfers/create --------->|                         |
     |    (encrypted ciphertext)           |                         |
     |<-- { transfer_id } ----------------|                         |
     |                                     |                         |
     |    Update vault tree                |                         |
     |    Encrypt vault tree               |                         |
     |--- POST /transfers/update --------->|                         |
     |    (encrypted vault tree)           |                         |
```

**What send.sgraph.ai sees:** encrypted blobs only. No file names, no content, no keys.
**What the LLM provider sees:** the prompt and document content (user's choice to send).
**What the browser has:** everything -- keys, plaintext, prompts, responses.

### 3.8 Vault Integration

The vault integration layer consists of three components working together:

1. **SGSendCrypto + SGSend** (CDN libraries): handle encryption and transfer API calls
2. **SGVaultCrypto + SGVault** (CDN libraries): manage the vault tree (a JSON structure mapping paths to transfer IDs, encrypted and stored as a transfer itself)
3. **VaultPanel** (workspace component): provides the UI for browsing, uploading, and downloading

**Vault key format:** `passphrase:vault_id`

The vault key is parsed into a passphrase (used for PBKDF2 key derivation) and a vault ID (the transfer ID of the encrypted vault tree). The derived key decrypts the vault tree, which contains the metadata for all files and folders. Each file's content is stored as a separate encrypted transfer.

**State machine:** `entry` -> `loading` -> `open` | `error`

**Persistence:** vault key and access token stored in `localStorage` key `sgraph-workspace-vault`. On page load, if a saved vault key exists, auto-opens.

## 4. Key Integration Points for New Features

This section provides the precise hooks for extending SG/Workspace. For each integration point, the exact EventBus topics, owning components, and file locations are specified.

### 4.1 Adding a New File Type Handler

**Relevant components:** `<document-viewer>` (`components/document-viewer/document-viewer.js`)

**How it works today:**
1. `_detectType()` uses `FileTypeDetect.detect(filename)` and falls back to extension-based detection
2. `_renderContent()` switches on `this._renderType` to select the renderer
3. Each render type has a `_render*()` method

**To add a new type (e.g., CSV table view):**
1. Add the new type string (e.g., `'csv-table'`) to the `_detectType()` method's extension matching
2. Add a `_renderCsvTable()` method to `DocumentViewer`
3. Add a `case 'csv-table':` to the switch in `_renderContent()`
4. If the type needs streaming support, add handling in `appendStreamChunk()`
5. Add CSS for the new type in `DocumentViewer.styles`

**EventBus topics to listen:** `file-selected`, `document-loaded`
**Component that owns this concern:** `DocumentViewer`
**Zone/panel:** source zone (top-left), data zone (bottom-left), or transform zone (right column)

### 4.2 Adding a New LLM Provider

**Relevant components:**
- `<llm-connection>` -- provider configuration and model listing
- `<llm-orchestrator>` -- request dispatch and response parsing

**To add a new provider (e.g., Anthropic direct API):**

In `llm-connection.js`:
1. Add the provider to the `PROVIDERS` object with `name`, `baseUrl`, `hint`, `placeholder`, `defaultModels`
2. Add a `_connectNewProvider()` method that fetches models
3. Add the provider button to `_render()` in the provider buttons section
4. Add provider-specific fields to the render (API key input, etc.)

In `llm-orchestrator.js`:
1. Add `else if (provider === 'new-provider')` branches in `_handleSend()` for both streaming and non-streaming
2. Implement `_callNewProvider()` for non-streaming
3. Implement `_streamNewProvider()` for streaming
4. If the streaming protocol differs from SSE or NDJSON, implement a new reader method

**EventBus topics:** `llm-connected`, `llm-disconnected`, `llm-model-changed`
**Components that own this concern:** `LlmConnection` (config), `LlmOrchestrator` (requests)

### 4.3 Extending the Prompt System

**Relevant components:**
- `<prompt-library>` -- built-in and vault-backed prompts
- `<llm-system-prompt>` -- system prompt editing
- `<llm-prompt-input>` -- user prompt composition
- `<llm-orchestrator>` -- prompt assembly

**To add new built-in prompts:**
1. Add entries to `BUILTIN_PROMPTS` array in `prompt-library.js`
2. Use `{{document}}` placeholder for the source document

**To add prompt templates with custom variables:**
1. Extend `llm-orchestrator._handleSend()` to recognize new `{{variable}}` patterns
2. Add corresponding artifact gathering logic (similar to how `incSource` gathers source HTML)

**To add vault-backed prompt folders:**
1. The prompt library already scans `/prompts` in the vault
2. Prompt files use `[system]\n...\n\n[user]\n...` format
3. `<llm-prompt-input>._onFileSelected()` parses this format

**EventBus topics:** `prompt-library-ready`, `prompt-library-updated`, `system-prompt-changed`, `prompt-context-changed`
**Components that own this concern:** `PromptLibrary` (storage), `LlmSystemPrompt` (system prompt), `LlmPromptInput` (composition)

### 4.4 Adding New UI Panels/Views

**Relevant components:** `<workspace-shell>` (`components/workspace-shell/workspace-shell.js`)

**To add a new navigation view (alongside Transform and Settings):**
1. Add an entry to the `NAV_ITEMS` array at the top of workspace-shell.js:
   ```javascript
   { id: 'my-view', label: 'My View', icon: ICONS.myIcon, default: false }
   ```
2. Add the SVG icon to the `ICONS` object
3. Add the view area HTML in the `render()` method (alongside `ws-transform-area` and `ws-settings-area`):
   ```html
   <div class="ws-myview-area" style="display:none">
       <my-component></my-component>
   </div>
   ```
4. Add the view to `_updateViewPanels()` to show/hide based on `this._activeView`
5. Style the new area in `workspace.css` with `grid-area: main`

**To add a panel within the existing transform view:**
- **New column:** Modify `ws-transform-area` grid template in workspace.css to add a 6th column
- **New zone in source column:** Add after `ws-source-bottom` in the shell render, add a new split handle
- **New section in chat zone:** Add inside `ws-chat-body` alongside llm-system-prompt, llm-prompt-input, llm-output, llm-stats

**EventBus topic to emit:** `view-changed` (emitted by shell on navigation)

### 4.5 Adding New Debug Panels

**Relevant components:** `<workspace-shell>` (debug sidebar template)

**To add a new debug tab:**
1. In workspace-shell.js `render()`, add a tab button:
   ```html
   <button class="ws-debug-tab" data-tab="my-debug">MyDbg</button>
   ```
2. Add the panel container:
   ```html
   <div class="ws-debug-panel" data-panel="my-debug" style="display:none">
       <my-debug-panel></my-debug-panel>
   </div>
   ```
3. Create the component file (follow the pattern from `messages-panel.js`)
4. Add the script tag in index.html Layer 3 (debug panels)

The shell's `_switchDebugTab()` method handles showing/hiding based on `data-tab`/`data-panel` matching.

### 4.6 How LLM Output Is Routed to Panels

The `<llm-output>` component has three copy-to buttons. Each emits `llm-output-copy` with a target:

```javascript
// In llm-orchestrator.js _handleCopy():
if (target === 'script') {
    document.querySelector('script-editor').loadScript(content, 'transform.js');
} else if (target === 'data') {
    document.querySelector('document-viewer[data-role="data"]').loadText(content, 'data.json');
} else if (target === 'result') {
    document.querySelector('document-viewer[data-role="transform"]').loadText(content, 'transform.html');
}
```

**To add a new copy target:**
1. Add a button in `llm-output.js` `_render()` with a new `data-target` value
2. Add a handler in `llm-orchestrator.js` `_handleCopy()` for the new target

### 4.7 Adding Components that Need Vault Access

Any component that needs to read/write vault files should:

1. Get the vault panel: `const vaultPanel = document.querySelector('vault-panel');`
2. Check state: `if (!vaultPanel || vaultPanel.getState() !== 'open') return;`
3. Get the vault: `const vault = vaultPanel.getVault();`
4. Read files: `const bytes = await vault.getFile(folderPath, filename);`
5. Write files: `await vault.addFile(folderPath, filename, uint8Array);`
6. List files: `const items = vault.listFolder(path);`
7. Create folders: `vault.createFolder(path);`
8. Save file from viewer: `await vaultPanel.saveFile(uint8Array, filename);`

**EventBus topics to listen:**
- `vault-opened` -- vault is ready for file operations
- `vault-locked` -- vault is no longer available
- `file-selected` -- user clicked a file (with `name`, `folderPath`, `type`, `size`)

## 5. What We Are Working On Next

These are the planned features for future workspace versions. **None of these exist yet** -- they are aspirations, not shipped functionality.

### 5.1 Multi-Turn Chat (Conversation History)
- Replace single-shot prompt/response with a message history array
- The orchestrator currently sends one `system` message and one `user` message per request
- Multi-turn would maintain an array of `{role, content}` messages
- Integration point: `llm-orchestrator._handleSend()` would build a message array instead of a single prompt string
- UI impact: `llm-output` would need to show a conversation thread, not just the latest response

### 5.2 Batch Document Transforms
- Select multiple files from the vault
- Apply the same prompt/script to each file
- Collect results into a folder
- Integration point: new component that iterates over vault files, calls the orchestrator for each

### 5.3 Data Panel Integration (JSON/CSV Viewer with Transformation)
- The data viewer (`<document-viewer data-role="data">`) already loads data files
- Next step: structured rendering (table view for CSV, tree view for JSON)
- Data transformation scripts that output structured data
- Integration point: new render types in `DocumentViewer`, or a dedicated `<data-viewer>` component

### 5.4 Prompt Versioning and Sharing
- Save prompt iterations with version numbers
- Share prompts via vault (already partially works -- vault `/prompts` folder)
- Prompt chains (output of one prompt feeds into the next)

### 5.5 Generate JS Refinement (LLM Writes Transformation Scripts)
- The "Gen JS" mode already generates JavaScript and routes it to the script editor
- Next step: auto-run generated scripts, capture errors, feed errors back to LLM for correction
- Iterative refinement loop: generate -> run -> capture error -> regenerate

### 5.6 Collaborative Features
- Share workspaces via vault sharing
- Multi-user vault access
- Shared prompt libraries
