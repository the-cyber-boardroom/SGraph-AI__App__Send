# Proposal: Script Access to DATA Panel

**Version:** v0.11.21
**Date:** 2026-03-06
**Role:** Architect
**Status:** PROPOSED — does not exist yet
**Companion doc:** `v0.11.21__review__workspace-data-panel-architecture.md`

---

## Problem Statement

The workspace's intended workflow has three phases:

1. **Explore** — Use LLM to iterate on scripts + data models
2. **Stabilise** — Lock the script and data schema
3. **Execute** — Run the proven script against new data, **no LLM needed**

Phase 3 is broken today. Scripts run in a sandboxed iframe that only receives SOURCE HTML. The DATA panel content has no path into the execution sandbox. This forces users back to the LLM every time data changes.

---

## Design Constraints

1. **Security sandbox must stay.** `sandbox="allow-scripts"` with no `allow-same-origin` is correct — scripts must not access parent window, localStorage, or vault encryption keys.
2. **No breaking changes** to existing SOURCE-only scripts.
3. **Bundle compatibility** — bundles already store `context.data.inline`, so any solution should leverage this.
4. **Zero dependencies** — IFD methodology, no libraries.

---

## Proposed Options

### Option A: Inject DATA as a Global Variable (Recommended)

**Change:** Modify `js-executor.js` to accept an optional `dataContent` parameter and inject it as `window.__DATA__` before the user script runs.

**Modified signature:**
```javascript
function executeJS(sourceHtml, userScript, timeoutMs, dataContent)
```

**Modified iframe document:**
```html
<!DOCTYPE html>
<body>
  [sourceHtml]
  <script>
    // Injected data (read-only for the script)
    window.__DATA__ = { raw: "...", parsed: null };
    try { window.__DATA__.parsed = JSON.parse(window.__DATA__.raw); } catch(e) {}

    // Console intercepts...
    // User script IIFE...
  </script>
</body>
```

**Script-editor change:**
```javascript
async _run() {
    const script     = this.getScript()
    const sourceHtml = sourceViewer.getTextContent()
    const dataText   = dataViewer.getTextContent()  // NEW: read DATA panel
    const result     = await window.sgraphWorkspace.executeJS(sourceHtml, script, 10000, dataText)
    // ... rest unchanged
}
```

**Script usage:**
```javascript
// User scripts can now do:
var data = __DATA__.parsed     // Structured (JSON/parsed)
var raw  = __DATA__.raw        // Raw text (CSV, XML, etc.)

// Filter articles by category from data
var categories = data.map(d => d.category)
var articles = document.querySelectorAll('.article')
// ... transform using both DOM and data
```

**Pros:**
- Minimal change (3 files, ~20 lines)
- No new APIs, no new components
- Backwards compatible — `__DATA__` is `{ raw: null, parsed: null }` when no data loaded
- Data stays inside the sandbox — no parent window access needed
- Works with all data formats (JSON parsed automatically, CSV/XML as raw text)

**Cons:**
- Large data files inflate the iframe document size
- No typed parsing for CSV/XML (user must parse in script)

**Effort:** ~1 hour Explorer work

---

### Option B: postMessage Data Channel

**Change:** Instead of injecting data at iframe creation time, use `postMessage` to send data to the iframe after it's ready.

**iframe setup:**
```javascript
window.addEventListener('message', function(e) {
    if (e.data.type === 'inject-data') {
        window.__DATA__ = e.data.payload
    }
})
// Signal ready
parent.postMessage({ type: 'js-exec-ready' }, '*')
```

**Parent (js-executor):**
```javascript
iframe.contentWindow.postMessage({
    type: 'inject-data',
    payload: { raw: dataText, parsed: tryParse(dataText) }
}, '*')
```

**Pros:**
- Data sent asynchronously, doesn't block iframe creation
- Could support streaming data updates mid-execution

**Cons:**
- Race condition: script may run before data arrives (needs ready/ack handshake)
- More complex than Option A for the same result
- Overkill unless we need mid-execution data updates

**Effort:** ~2 hours Explorer work

---

### Option C: Data as Hidden DOM Elements

**Change:** Inject data as `<script type="application/json" id="__data__">` inside the iframe body, alongside the SOURCE HTML.

**iframe document:**
```html
<body>
  [sourceHtml]
  <script type="application/json" id="__data__">[dataContent]</script>
  <script>
    // User script
  </script>
</body>
```

**Script usage:**
```javascript
var raw  = document.getElementById('__data__').textContent
var data = JSON.parse(raw)
```

**Pros:**
- DOM-native, no global variable injection
- `<script type="application/json">` is not executed, just stored
- Discoverable via standard DOM APIs

**Cons:**
- Pollutes the SOURCE DOM — scripts querying `document.querySelectorAll('script')` may pick it up
- DATA and SOURCE are mixed in the same DOM, breaking the clean separation
- Harder to distinguish "real" source elements from injected data

**Effort:** ~1 hour Explorer work

---

### Option D: Dedicated Data Parsing Helpers

**Change:** Extend Option A with built-in parser helpers injected into the sandbox.

**Injected helpers:**
```javascript
window.__DATA__ = {
    raw:       rawText,
    json:      tryParseJSON(rawText),
    csv:       tryParseCSV(rawText),      // Returns array of arrays
    csvObj:    tryParseCSVObjects(rawText), // Returns array of {header: value}
    xml:       tryParseXML(rawText),       // Returns DOM Document
    lines:     rawText.split('\n'),
    filename:  dataFilename,
    extension: ext
}
```

**Pros:**
- Scripts get pre-parsed data for all supported formats
- No user-side parsing code needed for common formats
- Extension field lets scripts branch on data type

**Cons:**
- Larger injected code footprint
- CSV/XML parsers add complexity (edge cases, encoding)
- May not match user expectations for specific CSV dialects

**Effort:** ~3-4 hours Explorer work (mostly CSV/XML parser edge cases)

---

## Recommendation

**Start with Option A** — it's the smallest change that unlocks the full LLM-free workflow:

1. Modify `js-executor.js` to accept `dataContent` parameter
2. Modify `script-editor._run()` to read and pass DATA panel content
3. Inject as `window.__DATA__` with auto-JSON-parse attempt

**Then iterate to Option D** if users need built-in CSV/XML parsing — but that can be driven by actual usage, not upfront.

### Migration Path

| Phase | What | When |
|-------|------|------|
| **Now** | Option A — `__DATA__` global variable | Next Explorer session |
| **If needed** | Option D additions — CSV/XML helpers | When users hit CSV/XML parsing friction |
| **If needed** | Option B — postMessage channel | Only if streaming data updates are needed |

---

## Impact on Bundle Schema

**No changes needed.** Bundles already store `context.data.inline` and `context.data.ref`. The data is already captured — it just needs to flow into the execution sandbox. When a bundle is restored and the user clicks "Run", the script will automatically have access to the restored data via `__DATA__`.

---

## Impact on LLM Workflow

The LLM prompt builder already includes DATA when `incData` is checked. With Option A:

- **LLM-assisted mode:** LLM sees DATA in prompt, generates scripts that use `__DATA__.parsed`
- **LLM-free mode:** User loads new data file → clicks Run → script reads `__DATA__` → result updates

The system prompt should be updated to tell the LLM about `__DATA__` availability:

```
When generating scripts, structured data is available via:
- __DATA__.raw    — raw text content of the DATA panel
- __DATA__.parsed — JSON-parsed version (null if not valid JSON)
```

---

## Security Considerations

- Data is injected as a string literal, JSON-escaped → no script injection risk
- The sandbox boundary is unchanged — `allow-scripts` only, no `allow-same-origin`
- Data stays inside the iframe — cannot leak back to parent except via `postMessage` (which is already the result channel)
- Large data files may slow iframe creation (acceptable trade-off; can add size warning later)
