## 1. What PKI Keys Means

PKI (Public Key Infrastructure) is the framework for managing digital certificates and public/private key pairs. In the context of extending the SSH keygen tool, "adding PKI key generation" means producing some or all of the following artefacts:

### Key Pairs in PEM Format

The foundational output. A PEM-encoded private key and its corresponding public key, suitable for use with TLS, S/MIME, code signing, and general-purpose cryptography. Unlike SSH keys (which use the OpenSSH wire format), PKI keys use standard PKCS#8 (private) and SPKI (public) encodings wrapped in PEM armour:

```
-----BEGIN PRIVATE KEY-----
MIGHAgEAMBMGByqGSM49AgEGCCqGSM49AwEHBG0wawIBAQQg...
-----END PRIVATE KEY-----

-----BEGIN PUBLIC KEY-----
MFkwEwYHKoZIzj0CAQYIKoZIzj0DAQcDQgAE...
-----END PUBLIC KEY-----
```

These use ASN.1/DER encoding inside the base64 payload, but the structures are well-defined and relatively compact for the algorithms we care about.

### Self-Signed X.509 Certificates

A self-signed certificate binds a public key to a subject identity (e.g., a Common Name) with a validity period and signature. It is the standard format used by:
- TLS server and client certificates
- mTLS authentication
- Code signing
- S/MIME email encryption

An X.509 certificate is a complex ASN.1 structure containing: version, serial number, signature algorithm, issuer, validity (notBefore/notAfter), subject, subject public key info, extensions (key usage, basic constraints, subject alternative names), and the issuer's signature over all of the above.

### Certificate Signing Requests (CSRs)

A CSR (PKCS#10) is a signed request asking a Certificate Authority (CA) to issue a certificate for a given public key. It contains the subject, the public key, and a proof-of-possession signature. CSRs are what you submit to Let's Encrypt, internal CAs, or any certificate issuer.

### The PEM Ecosystem

All three artefacts use PEM encoding (base64 of DER with header/footer lines). The key distinction from SSH keys:

| Property | SSH Keys | PKI Keys |
|----------|----------|----------|
| Private key format | OpenSSH proprietary binary | PKCS#8 DER in PEM |
| Public key format | OpenSSH wire format (one-line) | SPKI DER in PEM |
| Certificate format | SSH certificates (rare) | X.509 (universal) |
| Primary use | Server authentication | TLS, mTLS, signing, encryption |
| Encoding | Custom binary + base64 | ASN.1/DER + base64 |

## 2. Web Crypto API Capabilities for PKI

The Web Crypto API provides native support for the key generation and export operations needed for PKI keys. Here is what it can and cannot do.

### What Web Crypto Can Do Natively

| Operation | API | Notes |
|-----------|-----|-------|
| RSA key generation (2048, 4096) | `crypto.subtle.generateKey({name: "RSASSA-PKCS1-v1_5", ...})` | Also RSA-PSS and RSA-OAEP |
| ECDSA key generation (P-256, P-384, P-521) | `crypto.subtle.generateKey({name: "ECDSA", namedCurve: "P-256"}, ...)` | All three NIST curves |
| Ed25519 key generation | `crypto.subtle.generateKey("Ed25519", ...)` | Chrome 113+, Firefox 130+, Safari 17+ |
| Export private key as PKCS#8 DER | `crypto.subtle.exportKey("pkcs8", privateKey)` | Standard ASN.1/DER encoding |
| Export public key as SPKI DER | `crypto.subtle.exportKey("spki", publicKey)` | Standard ASN.1/DER encoding |
| Export as JWK | `crypto.subtle.exportKey("jwk", key)` | JSON Web Key format |
| Sign data | `crypto.subtle.sign(algorithm, key, data)` | Needed for self-signed certs and CSRs |
| SHA-256/384/512 hashing | `crypto.subtle.digest("SHA-256", data)` | Needed for certificate fingerprints |

**Critical insight:** Web Crypto already exports keys in PKCS#8 and SPKI DER format. The existing SSH keygen tool does the *opposite* -- it takes PKCS#8 DER output and converts it to OpenSSH format. For PEM key pairs, we simply need to base64-encode the DER and wrap it in PEM headers. This is dramatically simpler than what we already built.

### What Web Crypto Cannot Do

| Operation | Why Not | Workaround |
|-----------|---------|------------|
| Create X.509 certificate structures | No ASN.1 builder; no certificate-aware API | Must build ASN.1/DER manually or use a library |
| Create CSR (PKCS#10) structures | Same -- no higher-level PKI structures | Must build ASN.1/DER manually or use a library |
| Parse/read existing certificates | No X.509 parser | Must parse ASN.1/DER manually or use a library |
| OID encoding | No ASN.1 utilities at all | Must implement OID encoding |
| Certificate extensions | No extension builder | Must implement as raw ASN.1 |

**The boundary is clear:** Web Crypto handles keys and signing. Everything above that -- certificate structures, ASN.1 encoding, OIDs, extensions -- must be implemented in JavaScript.

## 3. The Hard Parts

### 3a. PEM Key Pairs (Easy)

Generating PEM-format key pairs is straightforward because Web Crypto gives us PKCS#8 and SPKI directly:

```javascript
async function generateECDSAKeyPairPEM() {
    const keyPair = await crypto.subtle.generateKey(
        { name: "ECDSA", namedCurve: "P-256" },
        true,
        ["sign", "verify"]
    )

    const privateDer = await crypto.subtle.exportKey("pkcs8", keyPair.privateKey)
    const publicDer  = await crypto.subtle.exportKey("spki", keyPair.publicKey)

    return {
        privateKey: wrapPEM(base64Encode(new Uint8Array(privateDer)), "PRIVATE KEY"),
        publicKey:  wrapPEM(base64Encode(new Uint8Array(publicDer)),  "PUBLIC KEY")
    }
}

function wrapPEM(base64, label) {
    const lines = []
    for (let i = 0; i < base64.length; i += 64) {
        lines.push(base64.slice(i, i + 64))
    }
    return `-----BEGIN ${label}-----\n${lines.join('\n')}\n-----END ${label}-----\n`
}
```

This is approximately **20 lines of new code** beyond what we already have. The existing `base64Encode` and binary helpers in `ssh-crypto.js` are directly reusable.

**Ed25519 caveat:** Web Crypto exports Ed25519 keys in PKCS#8/SPKI with standard OIDs, which is exactly what `openssl` expects. However, older versions of OpenSSL (pre-1.1.1) may not recognise Ed25519 keys. This is decreasingly a concern in 2026.

### 3b. Self-Signed X.509 Certificates (Hard)

This is where complexity jumps significantly. A self-signed certificate requires building a valid X.509 v3 structure in ASN.1/DER, then signing it with the private key.

**The X.509 TBSCertificate structure (what gets signed):**

```
SEQUENCE {                              -- TBSCertificate
    [0] EXPLICIT INTEGER { 2 }          -- version (v3)
    INTEGER { <serial number> }         -- serialNumber (random, 20 bytes)
    SEQUENCE { OID, NULL }              -- signature algorithm
    SEQUENCE { SET { SEQUENCE {         -- issuer (RDN: CN=...)
        OID (commonName), UTF8String
    }}}
    SEQUENCE {                          -- validity
        UTCTime { notBefore }
        UTCTime { notAfter }
    }
    SEQUENCE { SET { SEQUENCE {         -- subject (same as issuer for self-signed)
        OID (commonName), UTF8String
    }}}
    SEQUENCE { ... }                    -- subjectPublicKeyInfo (from SPKI export)
    [3] EXPLICIT SEQUENCE {             -- extensions
        SEQUENCE { OID, BOOLEAN, OCTET STRING { ... } }   -- basicConstraints
        SEQUENCE { OID, BOOLEAN, OCTET STRING { ... } }   -- keyUsage
        SEQUENCE { OID, OCTET STRING { ... } }            -- subjectKeyIdentifier
    }
}
```

The full certificate is then:

```
SEQUENCE {                              -- Certificate
    <TBSCertificate bytes>
    SEQUENCE { OID, NULL }              -- signatureAlgorithm
    BIT STRING { <signature> }          -- signatureValue
}
```

**Concrete challenges:**

1. **ASN.1 type encoding** -- must implement encoders for: SEQUENCE, SET, INTEGER, BIT STRING, OCTET STRING, OID, UTF8String, UTCTime, GeneralizedTime, BOOLEAN, context-tagged types ([0] EXPLICIT, [3] EXPLICIT)
2. **OID encoding** -- Object Identifiers use a variable-length encoding scheme (base-128 with continuation bits). We need OIDs for: rsaEncryption, ecPublicKey, sha256WithRSAEncryption, ecdsa-with-SHA256, id-Ed25519, commonName, basicConstraints, keyUsage, subjectKeyIdentifier, and the named curves
3. **Time encoding** -- UTCTime for dates before 2050, GeneralizedTime for dates after 2050 (per RFC 5280)
4. **Signature format** -- RSA signatures are straightforward (raw bytes). ECDSA signatures from Web Crypto are in IEEE P1363 format (r || s, fixed-length), but X.509 requires DER-encoded (SEQUENCE { INTEGER r, INTEGER s }). This conversion is required
5. **Subject/Issuer DN encoding** -- Distinguished Names are deeply nested: SEQUENCE of SETs of SEQUENCEs of (OID, value) pairs

**Estimated size:** A minimal ASN.1 encoder (covering only what we need, not a general-purpose library) would be approximately **300-500 lines of JavaScript**. The certificate builder on top of that would be another **200-300 lines**. Total: roughly **500-800 lines** of carefully tested code.

For reference, the current `ssh-crypto.js` is 351 lines and handles a simpler binary format (OpenSSH, which uses length-prefixed fields rather than ASN.1 TLV).

### 3c. CSR Generation (Medium-Hard)

A CSR (PKCS#10 / RFC 2986) is structurally simpler than a certificate:

```
SEQUENCE {                              -- CertificationRequest
    SEQUENCE {                          -- CertificationRequestInfo
        INTEGER { 0 }                  -- version
        SEQUENCE { ... }               -- subject (DN, same as certificate)
        SEQUENCE { ... }               -- subjectPublicKeyInfo (from SPKI)
        [0] IMPLICIT SET { }           -- attributes (usually empty)
    }
    SEQUENCE { OID, NULL }              -- signatureAlgorithm
    BIT STRING { <signature> }          -- signature
}
```

A CSR shares most ASN.1 primitives with the certificate builder. If we build the ASN.1 encoder for certificates, CSRs come almost for free -- approximately **50-100 additional lines** on top of the shared encoder.

### 3d. Library Options

If we chose to use a library instead of hand-coding ASN.1:

| Library | Size | Capability | Trade-off |
|---------|------|------------|-----------|
| **PKI.js** (+ ASN1.js) | ~150KB minified | Full X.509, CSR, CRL, OCSP, CMS | Heavy; depends on asn1js. Overkill for key generation |
| **node-forge** | ~200KB minified | Full PKI, TLS, SSH, PKCS#12 | Very heavy; brings its own crypto (we don't need) |
| **@peculiar/x509** | ~40KB minified | X.509 certs and CSRs on top of Web Crypto | Modern, Web Crypto native. Closest to our needs |
| **Hand-rolled minimal ASN.1** | ~15-25KB (unminified) | Only what we need: cert + CSR + key PEM | Zero dependencies. Full control. Consistent with SSH keygen approach |

**The project's existing approach** -- as demonstrated by `ssh-crypto.js` -- is to hand-roll binary encoding with no external dependencies. The SSH tool implements OpenSSH format encoding in 351 lines of clean, commented JavaScript. The ASN.1 encoder follows the same philosophy but is more complex.

## 4. Scope Options

Four progressively ambitious scopes, from simplest to most feature-complete.

### Scope A: PEM Key Pairs Only

**What it produces:**
- RSA 2048/4096 private key (PKCS#8 PEM) and public key (SPKI PEM)
- ECDSA P-256/P-384 private key (PKCS#8 PEM) and public key (SPKI PEM)
- Ed25519 private key (PKCS#8 PEM) and public key (SPKI PEM)

**What it does not produce:**
- No certificates (no X.509)
- No CSRs

**Implementation complexity:** Very low. Web Crypto exports PKCS#8 and SPKI natively. We only need a `wrapPEM()` function (which is a trivially modified version of the one in `ssh-crypto.js`) and algorithm-specific `generateKey` calls.

**New code:** ~80-120 lines in a new `pki-crypto.js` module. Reuses existing binary helpers from `ssh-crypto.js`.

**Effort:** 1-2 hours. Could be done in a single session.

**Value:** Immediately useful for anyone who needs PEM keys for TLS configuration, API authentication, or code signing key pairs. Covers the most common use case: "I need a key pair and I don't want to install openssl."

### Scope B: PEM Keys + Self-Signed Certificate

**What it produces:**
- Everything from Scope A, plus:
- Self-signed X.509 v3 certificate (PEM) with configurable: Common Name, validity period (default 365 days), key usage (digitalSignature, keyEncipherment)

**What it does not produce:**
- No CSRs
- No CA functionality (cannot sign other certificates)
- No Subject Alternative Names (SANs) -- could be added later

**Implementation complexity:** Medium-high. Requires a minimal ASN.1/DER encoder and certificate structure builder.

**New code:** ~500-700 lines total (ASN.1 encoder + certificate builder + PEM key generation). The ASN.1 encoder is the bulk of the work.

**Effort:** 4-8 hours across 2-3 sessions. The ASN.1 encoding needs careful testing against `openssl x509 -in cert.pem -text -noout` to verify correctness.

**Value:** Self-signed certificates are used for: local development HTTPS, mTLS client authentication, internal services, testing, and Kubernetes secrets. This is the most common reason someone reaches for `openssl req -x509 -newkey ...`.

### Scope C: PEM Keys + Self-Signed Certificate + CSR

**What it produces:**
- Everything from Scope B, plus:
- CSR (PKCS#10 PEM) ready to submit to a CA

**Implementation complexity:** Incremental over Scope B. Once the ASN.1 encoder exists, a CSR is a simpler structure than a certificate.

**New code:** ~50-100 lines on top of Scope B.

**Effort:** 1-2 hours additional on top of Scope B.

**Value:** CSRs are needed whenever you want a CA-signed certificate (Let's Encrypt, internal CA, commercial CA). This completes the "key lifecycle" story: generate key pair, create CSR, submit to CA, receive certificate.

### Scope D: Full PKI Toolkit

**What it produces:**
- Everything from Scope C, plus:
- CA certificate generation (basicConstraints: CA:TRUE)
- Sign a CSR with a CA key to produce a CA-signed certificate
- PKCS#12 (.p12/.pfx) bundle export (certificate + private key)
- Certificate chain validation/display
- Certificate inspector (parse and display existing certificates)

**Implementation complexity:** High. PKCS#12 encoding is its own complex standard (uses password-based encryption, MAC, nested content types). Certificate signing means the tool holds CA key material. Certificate parsing requires a full ASN.1 decoder (not just encoder).

**New code:** ~2000-3000 lines. At this point, we are building a significant crypto library.

**Effort:** 20-40 hours across many sessions. This is a standalone project.

**Value:** A complete browser-based CA. Impressive but niche. The SEO opportunity is interesting ("free online certificate authority") but the complexity and security implications are significant.

## 5. UX Considerations

### Option 1: Tabs Within the Existing Tool (Recommended for Scope A-B)

Add a top-level tab bar above the algorithm selector:

```
[SSH Keys] [PKI Keys]              ← new tab bar
┌─────────────────────────────┐
│  Generate PKI Key Pair      │    ← title changes based on tab
│                             │
│  Algorithm: [ECDSA P-256] [RSA 4096] [Ed25519]
│                             │
│  Output: [Key Pair Only] [+ Self-Signed Cert]  ← scope B toggle
│                             │
│  Common Name: ____________  │    ← shown when cert is selected
│  Validity: [365 days ▾]    │
│                             │
│  [Generate]                 │
└─────────────────────────────┘
```

**Rationale:** Users arriving via SEO for "ssh keygen online" see SSH keys by default. Users looking for PKI keys can switch tabs. The URL can include a query parameter or hash to deep-link to the PKI tab (`/tools/ssh-keygen#pki`), enabling separate SEO landing for "generate ssl certificate online" or "create pem key pair online."

The Web Component state machine extends naturally:

```
Current: idle -> generating -> complete
New:     idle -> generating -> complete    (unchanged per tab)
         + mode: 'ssh' | 'pki'           (new state dimension)
         + pkiOutput: 'keys' | 'cert' | 'csr'   (scope B/C)
```

### Option 2: Separate Page (Recommended for Scope C-D)

If the tool grows to include CSRs, certificate chains, and a certificate inspector, a separate page under `/tools/pki-keygen/` is cleaner:

```
sgraph_ai_app_send__ui__user/
  tools/
    ssh-keygen/    ← existing, focused on SSH
    pki-keygen/    ← new, focused on PKI
      v0/v0.1/v0.1.0/
        index.html
        js/pki-crypto.js
        js/pki-keygen-app.js
        css/pki-keygen.css
```

**SEO benefit:** Separate pages with distinct `<title>`, `<meta description>`, JSON-LD, and canonical URLs perform better for different search intents. "SSH key generator" and "SSL certificate generator" are different searches with different user intent.

**Cross-linking:** Each page links to the other: "Need SSH keys instead?" / "Need TLS certificates instead?"

### Option 3: Unified Crypto Tools Hub (Future, Scope D)

A `/tools/` index page listing all crypto utilities, with each tool on its own page. This is the right end state but premature for now.

### Recommendation

Start with **Option 1 (tabs)** for Scope A/B. If we later build Scope C/D, split into a separate page (Option 2). The crypto module (`pki-crypto.js`) should be a separate file from `ssh-crypto.js` regardless of UX approach -- they share binary helpers but serve different output formats.

## 6. Security Implications

### Self-Signed Certificates Generated in Browser

**Risk: misuse as trust anchors.** Users may add browser-generated self-signed certificates as trusted CAs in their system trust store. This is not inherently wrong -- it is exactly what internal PKI does -- but the tool should include clear guidance about what self-signed means and when it is appropriate.

**Mitigation:** Display a clear warning when generating self-signed certificates: "This certificate is self-signed. It is not trusted by browsers or operating systems by default. It is suitable for development, testing, and internal services. For production websites, use a certificate from a CA like Let's Encrypt."

### CSR Generation Patterns

**Risk: weak subject information.** A CSR with a Common Name that does not match the intended domain will be rejected by most CAs, wasting the user's time.

**Mitigation:** Validate Common Name format when generating CSRs intended for web servers. Suggest "yourdomain.com" format. Show a preview of the CSR contents before generation.

### Private Key Handling

The same security model as the SSH keygen tool applies and is preserved:

| Property | Status |
|----------|--------|
| Keys generated in browser | Yes -- Web Crypto API |
| Keys never sent to server | Yes -- `connect-src 'none'` CSP enforced |
| No persistent storage | Yes -- no IndexedDB, localStorage, or sessionStorage |
| Keys exist only in DOM memory | Yes -- cleared on page unload or reset |
| Hardware-backed RNG | Yes -- `crypto.getRandomValues()` |

**No new attack surface** is introduced by adding PKI key generation. The same CSP (`connect-src 'none'`) that protects SSH keys protects PKI keys. The same trust model ("verify in DevTools Network tab: zero requests") applies.

### Algorithm Considerations

| Algorithm | Key Size | Security Level | Browser Support | Recommendation |
|-----------|----------|---------------|-----------------|----------------|
| RSA 2048 | 2048 bits | 112 bits | Universal | Good for compatibility |
| RSA 4096 | 4096 bits | 128+ bits | Universal | Maximum RSA security |
| ECDSA P-256 | 256 bits | 128 bits | Universal | Recommended default |
| ECDSA P-384 | 384 bits | 192 bits | Universal | High security |
| Ed25519 | 256 bits | 128 bits | Chrome 113+, FF 130+, Safari 17+ | Modern, but limited browser support |

**Recommendation:** Default to ECDSA P-256 for PKI keys. It produces the smallest certificates, is universally supported by all browsers and TLS implementations, and provides 128-bit security. Offer RSA 2048 and RSA 4096 as alternatives for systems that require RSA. Include Ed25519 with the same feature-detection fallback as the SSH tool.

Note that ECDSA P-256 is uncommon for SSH keys (most SSH users prefer Ed25519) but is the standard for TLS certificates. The algorithm recommendation differs between the SSH and PKI tabs, which reinforces the case for presenting them as related-but-distinct tools.

## 8. Recommendation

### Start with Scope A (PEM Key Pairs Only)

**Rationale:**

1. **Minimal effort, maximum immediate value.** PEM key pairs are the single most useful PKI artefact. They work with `openssl`, `curl --cert`, Kubernetes secrets, AWS API Gateway, nginx, Apache, and every TLS implementation. Generating them takes ~100 lines of new code and can ship in a single session.

2. **Validates the UX pattern.** Adding a "PKI Keys" tab to the existing SSH keygen tool tests whether users find and use the PKI functionality. If usage is low, we save the effort of building the full certificate stack. If usage is high, we have clear signal to invest in Scope B.

3. **No new dependencies.** PEM key pairs use only Web Crypto API -- no ASN.1 encoder needed. The output is directly what `crypto.subtle.exportKey("pkcs8")` and `crypto.subtle.exportKey("spki")` produce, wrapped in PEM armour.

4. **SEO opportunity.** "Generate PEM key pair online" and "ECDSA key generator" are viable search terms with lower competition than "SSL certificate generator." Scope A captures this traffic immediately.

### Then Scope B+C Together (Self-Signed Certificate + CSR)

Once Scope A ships and validates the UX:

1. **Build the ASN.1 encoder as a standalone module** (`asn1-encoder.js`). Design it to be testable in isolation -- feed it structures, verify the output bytes against known-good DER.

2. **Build the certificate generator** on top of the encoder. Start with the simplest valid self-signed certificate (v3, basicConstraints, keyUsage, subjectKeyIdentifier). Add extensions incrementally.

3. **Add CSR generation** in the same session -- it reuses the encoder and is structurally simpler than certificates.

4. **If scope grows to warrant it,** split into a separate `/tools/pki-keygen/` page rather than overloading the SSH keygen tab.

### Do Not Pursue Scope D (Full PKI Toolkit) Unless Demand Emerges

Scope D is a different product. A browser-based CA with PKCS#12 export, certificate signing, and chain validation is technically possible but is a significant engineering investment that competes with established tools (step-ca, cfssl, mkcert). The Explorer team's time is better spent on features that differentiate SG/Send -- the chain of trust architecture, key discovery, and secure pod work are higher-leverage investments in the PKI space than a generic certificate tool.

### Architectural Approach

**Module structure:**

```
js/
  ssh-crypto.js           ← existing, unchanged
  pki-crypto.js           ← new: PEM key generation (Scope A)
  asn1-encoder.js         ← new: ASN.1/DER primitives (Scope B, extracted for reuse)
  x509-builder.js         ← new: certificate + CSR structures (Scope B+C)
  ssh-keygen-app.js       ← existing, extended with tab state
```

**Shared code:** The binary helpers in `ssh-crypto.js` (`concatBytes`, `base64Encode`, `encodeUint32`) should be extracted into a shared `crypto-utils.js` module or duplicated in `pki-crypto.js`. For Scope A, duplication is acceptable (only `base64Encode` and `wrapPEM` are needed). For Scope B+, a shared module is cleaner.

**ASN.1 encoder design** (for Scope B):

```javascript
// Target API for the ASN.1 encoder
const ASN1 = {
    SEQUENCE(...children)        // returns DER bytes
    SET(...children)             // returns DER bytes
    INTEGER(value)               // BigInt or Uint8Array
    BIT_STRING(bytes)            // with leading 0x00 for unused bits
    OCTET_STRING(bytes)          // raw bytes
    OID(oidString)               // "2.5.4.3" -> DER-encoded OID
    UTF8_STRING(str)             // UTF-8 encoded
    UTC_TIME(date)               // Date -> UTCTime encoding
    BOOLEAN(val)                 // true/false
    EXPLICIT(tag, child)         // context-tagged: [tag] EXPLICIT child
    IMPLICIT(tag, child)         // context-tagged: [tag] IMPLICIT child
}
```

This API mirrors the ASN.1 types one-to-one and makes certificate construction readable:

```javascript
const tbsCert = ASN1.SEQUENCE(
    ASN1.EXPLICIT(0, ASN1.INTEGER(2)),        // version v3
    ASN1.INTEGER(serialNumber),                // serial
    signatureAlgorithm,                        // SEQUENCE { OID, ... }
    issuerDN,                                  // SEQUENCE { SET { SEQUENCE { OID, UTF8 }}}
    ASN1.SEQUENCE(                             // validity
        ASN1.UTC_TIME(notBefore),
        ASN1.UTC_TIME(notAfter)
    ),
    subjectDN,                                 // same as issuer for self-signed
    subjectPublicKeyInfo,                      // raw SPKI bytes from Web Crypto
    extensions                                 // [3] EXPLICIT SEQUENCE { ... }
)
```

This approach -- hand-rolled, minimal, purpose-built -- is consistent with how we built the SSH format encoder. It produces readable code, ships no dependencies, and is fully auditable.

## 9. Comparison with Existing PKI Work

The project already has significant PKI infrastructure in the admin console (the `<pki-manager>` Web Component). That system uses RSA-OAEP 4096 and ECDSA P-256 for messaging (encryption + signing) with keys stored in IndexedDB.

The SSH keygen / PKI keygen tool is architecturally distinct:

| Property | PKI Manager (Admin) | PKI Keygen Tool |
|----------|-------------------|-----------------|
| Purpose | Encrypt/sign messages between users | Generate downloadable key artefacts |
| Key storage | IndexedDB (persistent, non-extractable) | None (memory only, download to file) |
| Server interaction | Key exchange via copy-paste | Zero (connect-src: none) |
| Output format | JWK (internal use) | PEM (standard interop) |
| Certificate generation | No (keys only) | Yes (Scope B+) |
| Users | Admin console operators | External developers (SEO traffic) |

These are complementary, not competing. The PKI keygen tool serves external SEO traffic with downloadable artefacts. The PKI manager serves internal admin workflows with browser-persistent keys.

However, the ASN.1 encoder built for Scope B could eventually be shared -- the PKI manager could use it to export certificates for admin keys, enabling mTLS client certificate authentication (the path described in the [admin PKI mTLS plan](../26-02-20/v0.4.17__implementation-plan__admin-pki-mtls.md)). This is a future synergy, not an immediate dependency.

## Summary

| Question | Answer |
|----------|--------|
| Is browser-based PKI key generation feasible? | Yes, fully. Web Crypto provides all primitives. |
| What is the simplest useful scope? | Scope A: PEM key pairs only (~100 lines, 1 session) |
| What is the biggest challenge? | ASN.1/DER encoding for X.509 certificates (Scope B) |
| Should we use a library? | No. Hand-roll a minimal ASN.1 encoder, consistent with SSH approach. |
| What UX pattern? | Tabs in the existing SSH keygen tool (Scope A/B), separate page (Scope C/D) |
| Start when? | Scope A can ship immediately. Scope B after validation. |
| Recommendation? | Ship Scope A now. Build Scope B+C when Scope A proves demand. Skip D. |
