# v0.6.2 — Phased Plan: Multi-User Data Vault MVP

**Date:** 2026-02-23
**Role:** Architect
**Status:** Explorer session — plan for review

## Executive Summary

This document maps the path from SGraph Send's current single-user encrypted vault to a **multi-user Data Vault product** — the investment-grade data room that the Alchemist track positions as the primary revenue vehicle. The plan is grounded in:

1. **20 existing building blocks** audited from the codebase (vault manager, storage, encryption, transfers, tokens, PKI)
2. **Human briefs** — data room product brief (v0.5.10), receiver trust spectrum (v0.5.24), chain of trust (v0.4.27), secure pod (v0.4.27), PKI vault (v0.5.15), file transfer engine (v0.3.12)
3. **Alchemist materials** — investor briefing pack (v0.5.8), data room positioning (v0.5.19), vault positioning (v0.5.34)
4. **Technical constraints** — 5MB upload limit (Lambda payload), no presigned URLs yet, no PDF preview

The plan has **5 phases**, each delivering a shippable increment.

### Encryption
- **AES-256-GCM** for file content (Web Crypto API)
- **RSA-OAEP** for key wrapping (vault files encrypted per-key)
- **Vault root** derived from `SHA-256(public_key_hash + "/filesystem")`
- **Share via Send** re-encrypts vault file with fresh AES key, uploads to transfer endpoint

### Constraints
- **5MB upload limit** — `SendUpload.MAX_FILE_SIZE` in Lambda payload (no presigned URLs yet)
- **No PDF preview** — text, markdown, images only
- **Single-user model** — vault keyed by single PKI key, no user_id, no ACLs
- **Token auth only** — no JWT, no sessions, no OAuth

## Phase 1: Multi-User Foundation (Vault Access Control)

**Goal:** Multiple users can access the same vault with different permission levels.
**Delivers:** The core building block everything else depends on.

### 1.1 User Identity Layer

**New file:** `lambda__admin/service/Service__Users.py`

- User record: `user_id` (Random_Guid), `display_name`, `key_fingerprint`, `created_at`
- Bind user identity to PKI key (one key = one user for now)
- Lookup by key fingerprint → user_id

**New schemas:**
```
Schema__User__Create__Request   { display_name, key_fingerprint }
Schema__User__Lookup__Response  { user_id, display_name, key_fingerprint, created_at }
```

### 1.2 Vault Ownership & ACL Model

**Modify:** `Service__Vault.py` — add `owner_user_id` to vault creation, pass `user_id` to all methods.

**New file:** `lambda__admin/service/Service__Vault__ACL.py`

- Permission levels: `owner`, `editor`, `viewer`
- ACL record: `{ vault_cache_key, user_id, permission, granted_by, granted_at }`
- Methods: `grant_access()`, `revoke_access()`, `list_permissions()`, `check_permission()`
- Owner can grant/revoke. Editors can read/write. Viewers can read only.

**New cache client methods** in `Send__Cache__Client__Vault.py`:
- `acl__store()`, `acl__lookup()`, `acl__delete()`, `acl__list()`

### 1.3 Permission-Gated Routes

**Modify:** `Routes__Vault.py` — extract `user_id` from request header, call `check_permission()` before every operation.

**New endpoints:**
```
POST   /vault/{vault_cache_key}/share          { user_id, permission }
DELETE /vault/{vault_cache_key}/unshare/{user_id}
GET    /vault/{vault_cache_key}/permissions
```

### 1.4 Multi-Recipient Encryption

**Modify:** `vault-manager.js` encryption flow:

Current: One wrapped AES key per file (wrapped with vault owner's RSA public key).

New: Store **N wrapped keys** — one per authorized user. When Alice shares a vault with Bob:
1. Alice decrypts the file's AES key with her private key
2. Alice wraps the AES key with Bob's public key (fetched from Key Registry)
3. Both wrapped keys stored alongside the ciphertext

**Packed format change:**
```
[2-byte num_recipients]
  [4-byte wrapped_key_len][wrapped_key][32-byte key_fingerprint]  × N
[12-byte IV][ciphertext]
```

### 1.5 Tests

- Permission matrix: owner × editor × viewer × each operation = pass/deny
- Multi-recipient encrypt/decrypt round-trip
- ACL grant + revoke + re-check
- Concurrent access from two users (no mocks — two in-memory clients)

**Estimated scope:** ~15 files touched, ~8 new files, ~30 new tests.

## Phase 2: Large File Transfer (Presigned URLs)

**Goal:** Break the 5MB barrier. Support files up to 5GB.
**Delivers:** Enterprise-grade file sizes for the data room use case.

### 2.1 Presigned URL Upload Flow

**Reference:** `library/docs/specs/large-file-transfer/` (974 lines of research already exists)

**New file:** `lambda__user/service/Service__Presigned__Upload.py`

Flow:
1. Client calls `POST /transfers/create` with `file_size_bytes`
2. If size > 5MB: server returns presigned S3 PUT URL (single part) or multipart initiation
3. Client uploads directly to S3 (encrypted bytes — server never sees plaintext)
4. Client calls `POST /transfers/complete/{transfer_id}` to finalize
5. Server verifies upload exists in S3, updates transfer status

**Threshold:**
- < 5MB: existing inline upload (no change)
- 5MB–5GB: single presigned PUT
- > 5GB: S3 multipart (initiate → upload parts → complete)

### 2.2 Presigned URL Download Flow

**Modify:** `Routes__Transfers.py` download endpoint.

- If file stored via presigned upload: return presigned GET URL (redirect or JSON)
- Client fetches directly from S3, decrypts in browser
- Download counter still tracked server-side (via S3 event or client callback)

### 2.3 Vault Large File Support

**Modify:** `vault-manager.js` upload flow + `admin-api-vault.js`.

- Detect file size before upload
- If > 5MB: use presigned URL path instead of inline base64
- Progress bar for large uploads (XHR with `upload.onprogress`)

### 2.4 Tests

- Upload 6MB file via presigned URL, download, verify integrity
- Multipart upload for 100MB+ (integration test with LocalStack)
- Fallback to inline for < 5MB
- Token validation still enforced for presigned URL generation

**Estimated scope:** ~6 new files, ~10 modified files, ~20 new tests.

## Phase 3: Data Room Product (Structured Sharing)

**Goal:** Package vault sharing into a "data room" product with invite links, folder-level permissions, and audit trails.
**Delivers:** The product the Alchemist track positions to investors.

### 3.1 Data Room Entity

**New file:** `lambda__admin/service/Service__Data_Room.py`

- Data room = named vault with structured access control
- Fields: `room_id`, `name`, `owner_user_id`, `created_at`, `status` (active/archived)
- Inherits vault storage — a data room IS a vault with extra metadata

### 3.2 Invite System

**New file:** `lambda__admin/service/Service__Invites.py`

- Generate invite link: `https://send.sgraph.ai/room/{room_id}/invite/{invite_code}`
- Invite record: `{ invite_code, room_id, permission, created_by, expires_at, max_uses, used_count }`
- Accepting invite: recipient registers key → gets ACL entry → can decrypt shared content

### 3.3 Folder-Level Permissions

**Extend** `Service__Vault__ACL.py`:

- Permission scoping: vault-level (all folders) OR folder-level (specific subtree)
- ACL record gains optional `folder_guid` — if set, permission applies only to that subtree
- Child folders inherit parent permission unless explicitly overridden

### 3.4 Audit Trail

**New file:** `lambda__admin/service/Service__Audit.py`

- Every vault/room operation logs: `{ event_id, room_id, user_id, action, target_guid, ip_hash, timestamp }`
- Query by room, user, time range, action type
- Immutable — append-only, no delete/update
- Aligns with brief v0.4.27 (chain of trust, hash chaining)

**New endpoint:** `GET /vault/{vault_cache_key}/audit?limit=50&offset=0`

### 3.5 Receiver Experience

**Reference:** Brief v0.5.24 (receiver trust spectrum)

- **Short codes** for vault/room access (human-memorable, e.g., `AB-1234`)
- **Environmental trust signals** — show encryption status, server location, audit availability
- **Progressive disclosure** — anonymous view → register key → full access

### 3.6 Data Room UI

**New component:** `data-room-manager.js` (extends vault-manager pattern)

- Room dashboard: list rooms, create room, archive room
- Member management: invite, remove, change permission
- Activity feed: recent operations across all members
- Audit log viewer: filterable table of all events

### 3.7 Tests

- Full room lifecycle: create → invite → accept → upload → share → audit → archive
- Folder-level permission enforcement
- Invite expiry and max-use limits
- Audit trail completeness and immutability

**Estimated scope:** ~12 new files, ~8 modified files, ~40 new tests.

## Phase 4: PDF Preview & Rich Content

**Goal:** View PDFs and other document types in-browser without downloading.
**Delivers:** Data room usability — investors/lawyers expect to preview documents.

### 4.1 PDF Preview Component

**New component:** `vault-preview-pdf.js`

- Embed PDF.js (Mozilla's open-source PDF renderer)
- Decrypt vault file → render in-browser canvas
- Page navigation, zoom, search
- Never sends plaintext to server — rendering is 100% client-side

### 4.2 Office Document Preview

- DOCX → HTML conversion (client-side library, e.g., mammoth.js)
- XLSX → HTML table (client-side, e.g., SheetJS)
- Lightweight — only load libraries when needed (dynamic import)

### 4.3 Extended MIME Type Detection

**Modify:** `vault-preview.js` — route to correct preview component based on MIME type.

- `application/pdf` → vault-preview-pdf
- `application/vnd.openxmlformats-officedocument.*` → vault-preview-office
- `text/*` → existing text preview
- `image/*` → existing image preview
- Fallback → download button

### 4.4 Tests

- PDF render + page navigation (Playwright E2E)
- DOCX/XLSX preview accuracy
- Large document handling (memory limits)

**Estimated scope:** ~4 new files, ~3 modified files, ~15 new tests.

## Phase 5: Enterprise Features

**Goal:** Features that differentiate the data room for enterprise/investor use cases.
**Delivers:** Revenue-generating capabilities aligned with Alchemist positioning.

### 5.1 Watermarking

- Overlay viewer's identity on previewed documents (canvas-based)
- Discourages unauthorized screenshots/sharing
- Server never sees the watermark — applied client-side from user identity

### 5.2 Expiring Access

- Time-boxed permissions: "access expires in 7 days"
- Auto-revoke on expiry (background job or on-access check)
- Countdown visible to recipient

### 5.3 Download Restrictions

- View-only mode: preview allowed, download blocked
- Download requires separate permission level
- Configurable per folder or per file

### 5.4 Q&A / Two-Way Communication

**Reference:** Brief v0.5.10 (data room product — two-way comms)

- Threaded Q&A per document or folder
- Messages encrypted with room key (all members can read)
- Notifications to room owner when new question posted

### 5.5 Analytics Dashboard

- Per-document: views, unique viewers, time spent, downloads
- Per-user: documents accessed, session duration, last active
- Per-room: activity over time, most viewed documents
- All data derived from audit trail (Phase 3)

**Estimated scope:** ~10 new files, ~5 modified files, ~25 new tests.

## Risk Register

| Risk | Mitigation | Phase |
|---|---|---|
| Multi-recipient key wrapping increases encrypted payload size | Only wrap for active users; lazy re-key on access | 1 |
| Presigned URLs expose S3 bucket structure | Use CloudFront with signed cookies instead of raw S3 URLs | 2 |
| Folder-level ACLs create permission explosion | Inherit from parent; max 3 levels of override | 3 |
| PDF.js bundle size (~400KB) slows initial load | Dynamic import on first PDF open | 4 |
| Lambda cold start with large file metadata | Keep metadata in cache service; S3 for blobs only | 2-3 |
| Key rotation for shared vaults is complex | Defer to Phase 5; document as known limitation | 1 |

1. **Start Phase 1 immediately** — it unblocks everything else

3. **Phase 3 after Phase 1 + 2 merge** — builds on both
4. **Phase 4 can start anytime** after Phase 1 (independent UI work)
5. **Phase 5 after Phase 3** — needs the full data room foundation
