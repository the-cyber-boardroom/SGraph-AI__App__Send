# Architect Review: Vault Component Architecture

**version** v0.5.34
**date** 23 Feb 2026
**role** Architect (Explorer team)
**type** Architecture review

---

## Summary

The vault subsystem has evolved from a single monolithic component (vault-manager v0.1.5, ~600 lines) into a **4-component micro-architecture** (2,543 lines total). This review documents the architectural patterns established and their implications for the broader system.

---

## Component Topology

```
vault-manager (orchestrator)
  ├── vault-settings  (row 2: metadata + actions bar)
  ├── vault-preview   (row 3a: read-only file renderer)
  └── vault-editor    (row 3b: read-write text editor)
```

**Communication pattern:** Event-driven via CustomEvent

```
vault-settings  ──(vault-settings-download)──→  vault-manager
vault-settings  ──(vault-settings-share)────→  vault-manager
vault-settings  ──(vault-settings-edit)─────→  vault-manager
vault-editor    ──(vault-editor-close)──────→  vault-manager
```

vault-manager is the sole orchestrator — it handles encryption/decryption, API calls, and delegates display to the sub-components.

## Layout Architecture: 3-Row Grid

```
┌──────────────────────────────────────────────────────┐
│  Row 1: File Table (tree view + sortable columns)    │
│  ─ flex: 1 (takes available space)                   │
├──────────────────────────────────────────────────────┤
│  Row 2: Settings Bar (vault-settings)                │
│  ─ fixed height (~36px)                              │
├──────────────────────────────────────────────────────┤
│  Row 3: Preview/Editor (vault-preview or vault-editor│
│  ─ flex: 1 (splits space with row 1)                 │
└──────────────────────────────────────────────────────┘
```

This replaced the original 2-column layout (tree | detail panel) which had width-splitting issues at narrower viewports.

## Data Flow: File Operations

### Upload (encrypt + store)
```
User drops file → vault-manager reads File object
  → Web Crypto generates AES-256-GCM key
  → Encrypts file bytes in browser
  → POST /api/vault/{key_fingerprint}/files (encrypted blob)
  → Server stores encrypted ciphertext (never sees plaintext)
```

### Download (fetch + decrypt)
```
User clicks Download → vault-manager calls API
  → GET /api/vault/{key_fingerprint}/files/{guid}
  → Receives encrypted blob
  → Decrypts with PKI private key in browser
  → Creates blob URL → triggers download
```

### Edit (decrypt → edit → re-encrypt → store)
```
User clicks Edit → vault-manager decrypts file
  → Opens vault-editor with plaintext
  → User edits → clicks Save
  → vault-editor calls onSave(newText)
  → vault-manager re-encrypts with same key
  → PUT /api/vault/{key_fingerprint}/files/{guid}
```

### Share (vault → Send transfer)
```
User clicks Share → vault-manager decrypts file
  → Creates Send transfer via API
  → Displays shareable URL with decryption key in fragment
```

## Security Model

The vault inherits the project's zero-knowledge security model:

1. **Server stores only ciphertext** — encrypted with AES-256-GCM
2. **Keys never leave the browser** — PKI keys generated and stored in Web Crypto API (CryptoKey objects, non-extractable by default)
3. **No file names on server** — server stores files by GUID only, encrypted metadata is client-side
4. **Key selection is explicit** — user picks which PKI key to use for the vault session

**New pattern established:** The vault-editor handles plaintext in-memory only. Save re-encrypts before any network call. The browser tab is the only place plaintext exists.

## Patterns Established for Future Components

| Pattern | Example | Reusable |
|---------|---------|----------|
| Orchestrator + sub-component | vault-manager + vault-preview | Any complex feature can split display from logic |
| Event-driven delegation | vault-settings emits, manager handles | Standard for all component communication |
| PKI_SHARED_STYLES import | All vault components use common button/form styles | Any new component can reuse |
| ResizeObserver height tracking | vault-editor dynamically calculates available height | Any full-height embedded component |
| Zero-dependency markdown renderer | vault-editor's _renderMarkdown() | Can extract to shared utility if needed |

## Shell Layout Fix

The admin-shell's `.as-main-content` had `max-width: 1100px` which prevented the main content area from expanding when the debug panel was toggled off. This was changed to `max-width: 100%`. This is architecturally correct — the shell should never constrain the content width; individual components should set their own max-width if needed.

---

## Recommendations for v0.6.x

1. **Extract markdown renderer** — vault-editor's `_renderMarkdown()` is reusable. Consider a shared `sg-markdown.js` utility.
2. **Folder operations** — vault-manager supports flat file list now; folder create/delete would complete the file system metaphor.
3. **Multi-file operations** — select multiple files for bulk download, delete, or share.
4. **Offline vault** — since all crypto is client-side, the vault could work offline with Service Worker caching.
