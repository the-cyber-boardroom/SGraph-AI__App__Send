# Architect Review: Parallel Multipart S3 Upload

**version** v0.6.13
**date** 23 Feb 2026
**role** Architect (Explorer team)
**type** Architecture review

---

## Summary

The multipart upload path has been upgraded from sequential (one part at a time) to a **parallel concurrency pool** with `PARALLEL_UPLOADS = 5`. This review assesses the architecture of the change, its interaction with the existing encryption and security model, scalability characteristics, error handling, and future evolution paths.

**Verdict:** Sound architectural change. The concurrency pool pattern is well-chosen for this problem space, preserves all existing security invariants, and delivers significant upload throughput improvement with minimal code complexity.

---

## 1. Data Flow: End-to-End Upload Path

```
┌───────────────┐     ┌───────────────┐     ┌──────────────────┐
│   Plaintext   │────→│  SGMETA Wrap  │────→│  AES-256-GCM     │
│   (File/Text) │     │  (filename +  │     │  Encrypt (whole   │
│               │     │   metadata)   │     │   buffer, single  │
│               │     │               │     │   IV)             │
└───────────────┘     └───────────────┘     └────────┬─────────┘
                                                     │
                                            encrypted ArrayBuffer
                                            [12-byte IV][ciphertext + auth tag]
                                                     │
                                                     ▼
                                       ┌─────────────────────────┐
                                       │  Size check:            │
                                       │  > 5 MB AND multipart   │
                                       │  capability available?  │
                                       └─────┬──────────┬────────┘
                                             │          │
                                        YES  │          │  NO
                                             ▼          ▼
                                    ┌──────────┐  ┌──────────────┐
                                    │ Presigned │  │ Direct upload│
                                    │ Multipart │  │ (single PUT) │
                                    └─────┬────┘  └──────────────┘
                                          │
                                          ▼
                              ┌───────────────────────┐
                              │ 1. POST /presigned/    │
                              │    initiate            │
                              │    → upload_id         │
                              │    → part_urls[]       │
                              └───────────┬───────────┘
                                          │
                                          ▼
                              ┌───────────────────────┐
                              │ 2. Parallel PUT to S3  │
                              │    (concurrency = 5)   │
                              │    encrypted.slice()   │
                              │    → ETag per part     │
                              └───────────┬───────────┘
                                          │
                                          ▼
                              ┌───────────────────────┐
                              │ 3. POST /presigned/    │
                              │    complete            │
                              │    → parts[] with      │
                              │      {part_number,     │
                              │       etag}            │
                              └───────────────────────┘
```

### Key observations

- **Encryption is whole-buffer, single-IV.** The entire plaintext (with SGMETA envelope) is encrypted as one AES-256-GCM operation producing a single ciphertext + auth tag. Chunking happens _after_ encryption, so individual chunks are meaningless without the complete reassembled ciphertext and the IV.
- **Part ordering is array-indexed.** `completedParts[i]` uses direct index assignment, not `push()`. This guarantees correct part ordering regardless of which parallel upload finishes first.
- **Presigned URLs isolate S3 credentials.** The browser never receives AWS credentials. Each part gets a time-limited presigned PUT URL from the backend.

## 2. Architecture Assessment: Concurrency Pool Pattern

### The implementation

```javascript
const active  = new Set();
const maxPool = SendUpload.PARALLEL_UPLOADS;   // 5
for (let i = 0; i < partUrls.length; i++) {
    const p = uploadOnePart(i).then(() => active.delete(p));
    active.add(p);
    if (active.size >= maxPool) await Promise.race(active);
}
await Promise.all(active);
```

### Assessment

This is the canonical JavaScript concurrency pool pattern. It is correct, minimal, and well-understood.

**Strengths:**

| Property | Assessment |
|----------|-----------|
| Correctness | The `Set + Promise.race` pattern ensures exactly `maxPool` concurrent uploads. No off-by-one: when `active.size >= maxPool`, the loop awaits until one completes and removes itself. |
| Ordering | Parts are dispatched in order (i = 0, 1, 2, ...) but may complete out of order. The indexed array `completedParts[i]` correctly preserves S3's required part ordering. |
| Backpressure | Natural backpressure: if one part is slow, the pool stays full but does not over-commit. New parts are dispatched only as slots free up. |
| Memory | Each `encrypted.slice(start, end)` creates a copy of the part data. With 5 concurrent uploads at 10 MB each, peak additional memory is ~50 MB. Acceptable for the 1 GB file size limit. |
| Simplicity | 7 lines of code. No external dependencies. No worker threads. No complex state machines. |

**No issues found.** The pattern is idiomatic and battle-tested across the JavaScript ecosystem.

## 3. Security Review

### Invariants preserved

| Security property | Status | Detail |
|---|---|---|
| Server never sees plaintext | PRESERVED | Encryption happens before `_uploadViaPresigned` is called. Only ciphertext bytes are uploaded. |
| Decryption key never leaves client | PRESERVED | Key is generated and exported on the client. The parallel upload code never references the key. |
| No file names on server | PRESERVED | SGMETA envelope (containing filename) is encrypted inside the ciphertext. Chunking operates on the encrypted buffer. |
| S3 credentials never reach browser | PRESERVED | Each part upload uses a presigned PUT URL. The `uploadPart()` call is a plain `fetch(presignedUrl, {method: 'PUT', body: partBuf})` with no auth headers. |
| IP hashing | UNAFFECTED | IP hashing occurs server-side during transfer creation, not during part uploads. |
| Authenticated encryption integrity | PRESERVED | AES-256-GCM auth tag is part of the ciphertext buffer. After S3 multipart completion reassembles the object, the full `[IV][ciphertext + tag]` is intact for the downloader to verify. |

### Security note on presigned URL scope

Each presigned PUT URL is scoped to a specific S3 key and part number. A compromised URL allows writing only that specific part of that specific object. The URLs are time-limited (server-configured expiry). This is a sound isolation model.

### Potential concern: presigned URL leakage

Presigned URLs are transmitted over HTTPS (TLS) from the backend to the browser, then used for direct PUT to S3 (also HTTPS). The risk surface is limited to:
- Browser memory (same as all in-memory secrets)
- Browser developer tools / extensions (standard threat model for web applications)

No additional risk is introduced by parallelism -- the URLs were already generated and transmitted in the sequential version.

## 4. Scalability Considerations

### PARALLEL_UPLOADS = 5

| Factor | Analysis |
|--------|----------|
| Browser connection limits | HTTP/2 multiplexes over a single TCP connection per origin. All 5 uploads go to S3 (same origin), so they multiplex efficiently. HTTP/1.1 has a 6-connection-per-origin limit in most browsers; 5 stays safely under that. |
| S3 throughput | S3 can handle thousands of concurrent PUTs per prefix. 5 concurrent uploads from a single client is negligible server-side. |
| Memory pressure | 5 x 10 MB = 50 MB of concurrent part buffers (slices). Combined with the full encrypted buffer (~1 GB max), total peak memory is ~1.05 GB. This is within typical browser tab limits (2-4 GB). |
| Network saturation | On a fast connection (100 Mbps+), 5 parallel streams will saturate upload bandwidth faster than sequential. On slow connections, the pool naturally throttles via backpressure. |

### Tuning guidance

| PARALLEL_UPLOADS | Best for | Trade-off |
|-----------------|----------|-----------|
| 1 | Debugging, very constrained environments | Minimal memory, slowest throughput |
| 3 | Mobile devices, low-memory environments | Conservative memory use |
| **5** (current) | **Desktop browsers, general use** | **Good balance of throughput and memory** |
| 8-10 | High-bandwidth enterprise connections | Higher memory, diminishing returns on throughput |
| >10 | Not recommended | Browser connection limits, memory pressure, no throughput gain |

The current value of 5 is well-chosen for the target user base.

### Part size interaction

With `DEFAULT_PART_SIZE = 10 MB` and `MAX_PARTS = 10,000`, the theoretical maximum file size is 100 GB (far above the 1 GB UI limit). For the current 1 GB limit, a typical upload produces ~100 parts, completed in ~20 pool cycles. The pool drains efficiently.

## 5. Error Handling and Abort Semantics

### Current behaviour

```javascript
try {
    // ... parallel upload pool ...
    await ApiClient.completeMultipart(transferId, uploadId, completedParts);
} catch (err) {
    await ApiClient.abortMultipart(transferId, uploadId);
    throw err;
}
```

### Assessment

| Scenario | Behaviour | Verdict |
|----------|-----------|---------|
| Single part fails | Promise rejects, caught by outer try/catch, abort called | CORRECT |
| Multiple parts fail | First rejection propagates (Promise.race semantics). Other in-flight parts continue to resolve/reject but results are ignored. Abort is called once. | ACCEPTABLE |
| Abort API call fails | Caught silently (`try/catch` in `abortMultipart`). Best-effort is the right policy here. | CORRECT |
| Network disconnect mid-upload | Fetch rejects with TypeError, caught by outer handler, abort attempted (may also fail). Error surfaces to user. | CORRECT |
| Complete API call fails | Caught, abort called to clean up S3 parts. | CORRECT |

### Observation: in-flight parts after failure

When one part fails, the remaining in-flight parts (up to 4) continue running in the background. Their results are discarded since the catch block has already taken over. This is acceptable because:

1. The abort call tells S3 to discard all parts regardless
2. The in-flight PUTs will complete harmlessly (writing to an aborted upload is a no-op in S3)
3. Adding cancellation (via AbortController) would add complexity for minimal benefit

### Recommendation for future consideration

If the project eventually needs per-part retry (see Section 7), an `AbortController` per part would become necessary to cancel in-flight requests on failure. For now, the current approach is correct and simple.

## 6. Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Memory pressure on large files | Low | Medium | 1 GB limit + 50 MB pool overhead is within browser limits. Monitor if limit increases. |
| Stale presigned URLs | Very Low | Medium | URLs expire server-side. If upload takes very long (slow connection + large file), late parts may get 403. Consider URL refresh. |
| S3 orphaned parts from failed aborts | Low | Low | S3 lifecycle rules should clean up incomplete multipart uploads. Verify lifecycle policy is configured. |
| Progress bar non-monotonic display | None | Low | Progress is based on `partsCompleted` count, which only increments. Display is always monotonically increasing. |
| Browser tab close during upload | Medium | Medium | `beforeunload` handler is set. Abort is not called on tab close (no `unload` handler). S3 lifecycle rules handle cleanup. |

## 7. Future Improvements

### Near-term (low effort, high value)

1. **Per-part retry with exponential backoff.** Currently, a single transient failure (e.g., network blip) aborts the entire upload. Wrapping `uploadOnePart` with a retry loop (2-3 attempts, exponential backoff) would significantly improve resilience on unstable connections.

2. **Per-chunk upload progress via XMLHttpRequest.** The current `fetch()` API does not expose upload progress events. Switching to `XMLHttpRequest` for the part PUT calls would enable per-chunk progress reporting, giving users a smoother progress bar instead of the current step-wise updates.

3. **AbortController integration.** Pass an `AbortController.signal` to each `fetch()` call. On user cancellation or first-part failure, abort all in-flight requests immediately rather than letting them complete in the background.

### Medium-term (moderate effort)

4. **Streaming encryption (chunked AES-GCM).** Currently, the entire file is encrypted as one buffer before chunking. For very large files, this means the full plaintext AND the full ciphertext must fit in memory simultaneously (~2x file size). Streaming encryption would encrypt each chunk independently with its own IV, reducing peak memory to `2 x partSize` (~20 MB). This requires a corresponding change to the decryption path and a different ciphertext format (per-chunk IV + tag).

5. **Adaptive concurrency.** Dynamically adjust `PARALLEL_UPLOADS` based on observed throughput and error rates. Start at 5, increase if all parts complete quickly, decrease if parts fail or timeout. This optimises for diverse network conditions.

6. **Resumable uploads.** Track which parts have been successfully uploaded (e.g., in `sessionStorage`). On page refresh or reconnection, query S3 for already-uploaded parts (`ListParts`) and resume from where the upload left off.

### Long-term (significant effort)

7. **Web Worker offloading.** Move encryption and chunking to a Web Worker to avoid blocking the main thread. This would keep the UI responsive during the encryption phase for large files (currently synchronous on main thread via `crypto.subtle.encrypt`).

8. **Parallel download with streaming decryption.** Mirror the upload architecture on the download side: fetch parts in parallel via presigned GET URLs, reassemble, and decrypt. This would provide symmetric performance improvement.

## 8. Architecture Alignment

| Principle | Alignment |
|-----------|-----------|
| Zero-knowledge encryption | Fully aligned. Parallelism operates entirely on post-encryption ciphertext. |
| No frameworks (vanilla JS) | Fully aligned. Pure Promise-based concurrency, no external libraries. |
| IFD methodology | Fully aligned. Change is surgical, self-contained within the send-upload component. |
| Pluggable storage backends | Fully aligned. The parallel upload path is gated behind the `multipart_upload` capability flag. Backends without presigned URL support fall through to direct upload. |
| Graceful degradation | Fully aligned. If capabilities check fails or `multipart_upload` is false, the code path falls back to single-buffer direct upload via Lambda. |

---

## Conclusion

The parallel multipart upload implementation is architecturally sound. It uses a well-established concurrency pattern, preserves all security invariants, handles errors correctly, and provides a clear path for future enhancements. The choice of `PARALLEL_UPLOADS = 5` is well-calibrated for the current deployment context.

The primary recommendations for future iterations are: (1) per-part retry for resilience, (2) XHR-based per-chunk progress for UX, and (3) streaming encryption to reduce memory footprint for very large files. None of these are blocking -- the current implementation is correct and production-ready.
