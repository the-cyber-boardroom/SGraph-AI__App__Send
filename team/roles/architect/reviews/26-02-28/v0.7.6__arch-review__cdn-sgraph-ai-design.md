## 1. Executive Summary

This review defines the architecture for **cdn.sgraph.ai** -- a dedicated CDN origin serving shared, stable assets (fonts, JavaScript libraries, CSS, vendor code, images) to all SGraph Send websites. The CDN is not a third-party service; it is an S3/CloudFront deployment under our control, using the same infrastructure pattern as the website deployments but with its own domain, versioning cadence, and caching policy.

**Key design decisions:**

1. cdn.sgraph.ai maps to the `cdn/latest/` folder in the shared S3 bucket
2. CDN assets are versioned independently from website releases
3. Websites reference CDN assets via absolute URL in online mode, relative paths in offline mode
4. CORS allows cdn.sgraph.ai to serve resources to all `*.sgraph.ai` origins
5. Initial content: Inter and DM Sans fonts, sg-send.js, sg-decrypt.js, logo, favicon, base CSS

**Architecture assessment: SOUND.** The design eliminates third-party CDN dependencies (Google Fonts, cdnjs) while keeping a clean migration path to dedicated CDN providers if needed later.

```
│

    ├── v1.0.0/                      <-- exact copy of latest/ at time of v1.0.0 release
    │   ├── fonts/
    │   ├── js/
    │   ├── css/
    │   ├── vendor/
    │   ├── images/
    │   └── manifest.json
    ├── v1.1.0/
    └── v1.2.0/
```

### 2.1 Folder Structure Rules

| Rule | Detail |
|------|--------|
| **latest/ is mutable** | Updated on every CDN release. CloudFront serves this folder. |
| **releases/{version}/ is immutable** | Once published, never modified. Offline packages reference a specific CDN release. |
| **manifest.json in every release** | Contains: CDN version, file list, SHA-256 checksums, build timestamp. Used for integrity verification and offline packaging. |
| **No nested versioning** | CDN content is flat within a release. No per-file versioning (e.g., no `js/v2/sg-send.js`). The release version covers everything. |
| **woff2 only for fonts** | Modern format with best compression. No woff, ttf, or eot. All target browsers support woff2. |

### 2.2 manifest.json Schema

```json
{
  "cdn_version": "1.0.0",
  "build_timestamp": "2026-02-28T14:30:00Z",
  "files": {
    "fonts/inter/inter-regular.woff2": {
      "size_bytes": 98304,
      "sha256": "abc123..."
    },
    "js/sg-send.js": {
      "size_bytes": 45056,
      "sha256": "def456..."
    }
  }
}
```

This manifest serves two purposes: (1) integrity verification for offline deployments, (2) cache-busting -- websites can check manifest.json to detect CDN updates without loading every file.

## 3. How Websites Reference CDN Assets

### 3.1 Online Mode (Standard Deployment)

In online mode, websites load shared assets from cdn.sgraph.ai using absolute URLs. This is the **only case** where absolute URLs are used -- and only for cross-origin CDN references, never for intra-site navigation.

```html
<!-- Fonts from CDN -->
<link rel="stylesheet" href="https://cdn.sgraph.ai/fonts/inter/inter.css">
<link rel="stylesheet" href="https://cdn.sgraph.ai/fonts/dm-sans/dm-sans.css">

<!-- Shared JS from CDN -->
<script src="https://cdn.sgraph.ai/js/sg-send.min.js"></script>
<script src="https://cdn.sgraph.ai/js/sg-decrypt.min.js"></script>

<!-- Shared CSS from CDN -->
<link rel="stylesheet" href="https://cdn.sgraph.ai/css/sg-send-base.css">

<!-- Images from CDN -->
<link rel="icon" href="https://cdn.sgraph.ai/images/favicon.ico">
```

**Why absolute URLs here are acceptable:** The brief mandates relative paths for all intra-site resources. CDN assets are cross-origin by definition -- they live on a different domain. The browser needs the full URL to fetch them. This is architecturally distinct from hardcoding `https://sgraph.ai/styles/main.css` for a same-site resource.

### 3.2 Offline Mode (Self-Contained Packaging)

In offline mode, CDN assets are copied into the deployment folder and referenced via relative paths. The build/packaging script performs the URL rewrite.

```html
<!-- Online: absolute CDN URL -->
<link rel="stylesheet" href="https://cdn.sgraph.ai/fonts/inter/inter.css">

<!-- Offline: relative path to local copy -->
<link rel="stylesheet" href="./cdn/fonts/inter/inter.css">
```

**Packaging script responsibility:**

1. Copy the entire CDN release folder (e.g., `cdn/releases/v1.0.0/`) into the offline package as `cdn/`
2. Rewrite all `https://cdn.sgraph.ai/` references to `./cdn/` in HTML files
3. Verify checksums against manifest.json after copy

**Folder structure in an offline package:**

```
offline-deployment/
├── index.html           <-- references ./cdn/ paths
├── styles/
├── js/
└── cdn/                 <-- copied from cdn/releases/v1.0.0/
    ├── fonts/
    ├── js/
    ├── css/
    ├── vendor/
    ├── images/
    └── manifest.json
```

### 3.3 URL Rewrite Strategy

The online-to-offline URL rewrite is a simple string replacement, which is a deliberate design choice:

| Pattern | Online | Offline |
|---------|--------|---------|
| **CDN base** | `https://cdn.sgraph.ai/` | `./cdn/` |

No complex template engine or build tool is needed. A Python script with `str.replace()` handles the transformation. This works because:

1. All CDN references use the canonical `https://cdn.sgraph.ai/` prefix (no protocol-relative, no trailing variations)
2. The local `cdn/` folder mirrors the CDN structure exactly
3. Font CSS files use relative paths internally (e.g., `url(./inter-regular.woff2)`) -- they work from any base path

## 4. CDN Versioning Strategy

### 4.1 Independent Versioning

CDN assets have their own version number, independent from website versions. This is essential because:

- Fonts change rarely (new weight added, new font family introduced)
- JavaScript libraries change on their own cadence
- Website content changes frequently

| Artifact | Versioning | Example |
|----------|-----------|---------|
| Website (send.sgraph.ai) | Application version | v0.7.6, v0.7.7, v0.8.0 |
| CDN (cdn.sgraph.ai) | CDN version | v1.0.0, v1.1.0, v1.2.0 |
| Lambda functions | Application version | v0.7.6 |

### 4.2 CDN Version Semantics

| Change Type | Version Bump | Example |
|-------------|-------------|---------|
| Add new font family | Minor (v1.1.0) | Adding "JetBrains Mono" for code blocks |
| Add new JS library | Minor (v1.2.0) | Adding sg-vault.js |
| Update existing library | Patch (v1.0.1) | Bug fix in sg-send.js |
| Remove asset or break API | Major (v2.0.0) | Removing lodash, changing sg-send.js API |
| Add new image or icon | Patch (v1.0.2) | New brand variant |

### 4.3 Website-to-CDN Version Binding

Each website release specifies which CDN version it requires. This binding is recorded in the website's deployment metadata:

```json
{
  "site": "send-sgraph-ai",
  "site_version": "v0.7.6",
  "cdn_version": "v1.0.0",
  "deployed_at": "2026-02-28T14:30:00Z"
}
```

In online mode, this is informational -- the website always loads from `cdn.sgraph.ai/` (which is `cdn/latest/`). In offline mode, the packaging script copies the specific CDN release version.

### 4.4 CDN Release Process

```
1. Make changes in cdn/ working directory
2. Run CDN tests (font loading, JS syntax check, manifest generation)
3. Update manifest.json with new checksums
4. Copy to cdn/releases/v{new_version}/
5. Update cdn/latest/ (copy or sync from the new release)
6. Invalidate CloudFront cache for cdn.sgraph.ai
7. Smoke test: load cdn.sgraph.ai/manifest.json, verify version
```

## 5. S3-to-CloudFront Mapping

### 5.1 CloudFront Distribution Configuration

```
Distribution: cdn.sgraph.ai
├── Origin: S3 bucket ({account-id}--static-sgraph-ai--{region})
│   └── Origin path: /cdn/latest
├── Alternate domain name (CNAME): cdn.sgraph.ai
├── TLS certificate: ACM certificate (*.sgraph.ai or cdn.sgraph.ai)
├── Origin Access Control: enabled (S3 bucket is private)
├── Default root object: (none -- no index.html, CDN serves individual assets)
├── Error pages: 403 → custom 404 response (no directory listing)
└── Cache behaviour:
    ├── Default (/*):
    │   ├── TTL: 3600s (1 hour) -- fonts and images change rarely
    │   ├── Compress: yes (gzip + brotli)
    │   └── Viewer protocol: HTTPS only
    ├── /js/* and /css/*:
    │   ├── TTL: 300s (5 minutes) -- JS/CSS may change more often during dev
    │   └── Compress: yes
    └── /manifest.json:
        ├── TTL: 60s (1 minute) -- used for cache-busting checks
        └── Compress: yes
```

### 5.2 URL Resolution

```
Request:  https://cdn.sgraph.ai/fonts/inter/inter-regular.woff2
          ↓
CloudFront distribution: cdn.sgraph.ai
          ↓
Origin path: /cdn/latest
          ↓
S3 key:   cdn/latest/fonts/inter/inter-regular.woff2
```

### 5.3 Why Origin Path, Not Prefix in Cache Behaviour

Using the CloudFront origin path (`/cdn/latest`) rather than URL prefixing keeps the public URLs clean. Users and websites reference `cdn.sgraph.ai/fonts/inter/inter.css` -- not `cdn.sgraph.ai/cdn/latest/fonts/inter/inter.css`. The origin path handles the S3-to-URL mapping transparently.

## 6. CORS Configuration

### 6.1 Why CORS Is Needed

cdn.sgraph.ai serves assets to websites on different origins:
- `send.sgraph.ai` loads fonts and JS from `cdn.sgraph.ai`
- `admin.sgraph.ai` loads the same shared CSS
- `sgraph.ai` loads the logo and brand assets
- `vault.sgraph.ai` loads sg-decrypt.js

Browsers enforce the Same-Origin Policy. Without CORS headers, font loading (via `@font-face`) and JavaScript `fetch()` calls to the CDN will be blocked.

### 6.2 CORS Headers (CloudFront Response Headers Policy)

```
Access-Control-Allow-Origin: https://*.sgraph.ai
Access-Control-Allow-Methods: GET, HEAD, OPTIONS
Access-Control-Allow-Headers: Range, If-None-Match
Access-Control-Max-Age: 86400
Access-Control-Expose-Headers: Content-Length, Content-Range
```

**Implementation note:** CloudFront response headers policies support wildcard subdomains. Configure this as a CloudFront Response Headers Policy attached to the cdn.sgraph.ai distribution, not as S3 CORS configuration.

### 6.3 CORS Rules

| Rule | Detail |
|------|--------|
| **Allow origin** | `https://*.sgraph.ai` only. No `*` wildcard. No non-sgraph.ai origins. |
| **Methods** | GET and HEAD only. The CDN is read-only. |
| **Credentials** | Not included. CDN assets are public to sgraph.ai origins -- no cookies or auth headers needed. |
| **Preflight cache** | 86400s (24 hours). Preflight requests are expensive; cache aggressively. |
| **Development override** | For local development (`localhost:*`), developers use a browser extension or proxy. The CDN itself does NOT add localhost to CORS. |

### 6.4 Font-Specific CORS Consideration

Fonts loaded via `@font-face` with `url()` are subject to CORS when cross-origin. The browser sends an `Origin` header and expects `Access-Control-Allow-Origin` in the response. Without this, fonts silently fail to load and the browser falls back to system fonts -- a subtle, hard-to-debug rendering difference.

The font CSS files (`inter.css`, `dm-sans.css`) use relative `url()` references:

```css
/* inter.css -- served from cdn.sgraph.ai/fonts/inter/ */
@font-face {
  font-family: 'Inter';
  font-style: normal;
  font-weight: 400;
  font-display: swap;
  src: url(./inter-regular.woff2) format('woff2');
}
```

The relative `url(./inter-regular.woff2)` resolves relative to the CSS file's location (`cdn.sgraph.ai/fonts/inter/`). This means the font file request also goes to `cdn.sgraph.ai` and needs the same CORS headers. Since CORS is configured at the distribution level, all responses carry the headers -- this is correct.

## 8. Migration Path

### 8.1 Phase 1: S3/CloudFront (Current Architecture)

```
cdn.sgraph.ai → CloudFront → S3 bucket (cdn/latest/)
```

Same infrastructure as all other SG/Send websites. No new services, no new providers. The only addition is a new CloudFront distribution and DNS record.

### 8.2 Phase 2: Evaluate Dedicated CDN (If Needed)

If cdn.sgraph.ai traffic grows or latency requirements tighten:

| Provider | Migration Effort | Benefit |
|----------|-----------------|---------|
| **Cloudflare (R2 + CDN)** | Low -- R2 is S3-compatible, change DNS | Zero egress, global edge, DDoS protection |
| **Fastly** | Medium -- different API, need deployment script update | Real-time purging, edge compute |
| **Bunny CDN** | Low -- pull zone from S3 origin, change DNS | Very low cost, good EU coverage |

### 8.3 Migration Is DNS-Only

Because all websites reference `cdn.sgraph.ai` (not `d123456.cloudfront.net`), migrating the CDN backend is a DNS change:

1. Deploy CDN content to new provider
2. Verify content integrity (checksums from manifest.json)
3. Update DNS: `cdn.sgraph.ai CNAME → new-provider-endpoint`
4. Old CloudFront distribution remains as fallback until TTL expires

The content structure, file paths, and CORS configuration stay identical. Websites do not need any changes.

## 9. Security Considerations

| Concern | Mitigation |
|---------|-----------|
| **CDN serves executable JS** | sg-send.js and sg-decrypt.js contain encryption logic. Compromise of the CDN could inject malicious code. Use Subresource Integrity (SRI) hashes in website HTML. |
| **CDN availability** | If cdn.sgraph.ai is down, all sites lose fonts and JS. Mitigation: CloudFront multi-region origins. Long-term: offline-capable sites with Service Workers. |
| **CORS too permissive** | Currently scoped to `*.sgraph.ai`. If we ever serve from non-sgraph.ai domains, CORS needs updating. Do NOT use `*`. |
| **Cache poisoning** | CloudFront caches responses. If a poisoned response enters the cache, it serves to all users until TTL expires. Mitigation: Origin Access Control (no public S3 access), cache invalidation capability. |

### 9.1 Subresource Integrity (SRI)

For JavaScript files loaded from the CDN, websites SHOULD include SRI hashes:

```html
<script src="https://cdn.sgraph.ai/js/sg-send.min.js"
        integrity="sha384-{hash}"
        crossorigin="anonymous"></script>
```

This ensures that even if the CDN is compromised, the browser will refuse to execute modified scripts. The SRI hashes are generated during the CDN release process and recorded in manifest.json.

## 11. Open Questions

| # | Question | Impact |
|---|----------|--------|
| 1 | Should CDN releases be tagged in the same git repo or a separate one? | Affects CI pipeline design. Recommendation: same repo, `cdn/` folder, separate release tags (`cdn-v1.0.0`). |
