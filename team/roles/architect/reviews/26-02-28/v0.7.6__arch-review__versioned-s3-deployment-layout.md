## 1. Executive Summary

This review defines the complete S3 bucket structure for all SGraph Send static deployments -- websites, CDN assets, CI artifacts, and package mirrors. The layout is designed to support immutable versioned releases, instant rollback, offline packaging, multi-region replication, and multi-cloud portability.

**Key design decisions:**

1. Single bucket per region with a well-defined four-section structure: `websites/`, `cdn/`, `packages/`, `ci/`
2. Bucket naming convention: `{account-id}--static-sgraph-ai--{region}`
3. Every site deploys to both `latest/` (mutable, CloudFront-served) and `releases/{version}/` (immutable, archival)
4. Rollback is a copy operation, not a rebuild
5. One CloudFront distribution per site, origin path points to `websites/{site}/latest/`
6. S3 Cross-Region Replication for multi-region redundancy
7. CI artifacts stored by date, not by version -- temporal rather than permanent

**Architecture assessment: SOUND.** The layout is straightforward, scales to many sites, and maps cleanly to CloudFront origin configurations. The separation of mutable (`latest/`) and immutable (`releases/`) content is the critical property that makes rollback reliable.

### 2.3 deploy.json Schema

Every deployment (both `latest/` and `releases/{version}/`) includes a `deploy.json` metadata file:

```json
{
  "site": "send-sgraph-ai",
  "version": "v0.7.6",
  "cdn_version": "v1.0.0",
  "deployed_at": "2026-02-28T14:30:00Z",
  "deployed_by": "ci/github-actions",

  "git_branch": "main",

```

This file enables:
- **Version verification:** any tool can check what version is live by fetching `deploy.json`
- **Rollback audit:** after rollback, `deploy.json` in `latest/` shows which release is active
- **Offline packaging:** the packaging script reads `deploy.json` to confirm version alignment

## 3. Bucket Naming Convention

### 3.1 Format

```
{aws-account-id}--static-sgraph-ai--{region}
```

### 3.2 Examples

| Region | Bucket Name |
|--------|------------|
| Ireland (primary) | `123456789012--static-sgraph-ai--eu-west-1` |
| Virginia (secondary) | `123456789012--static-sgraph-ai--us-east-1` |
| Singapore (future APAC) | `123456789012--static-sgraph-ai--ap-southeast-1` |

### 3.3 Why This Convention

| Property | Benefit |
|----------|---------|
| **Account ID prefix** | Globally unique (S3 names are global). No collision with other AWS accounts. |
| **Double-hyphen separator** | Visually distinct from the name components. Easy to parse programmatically. |
| **Purpose in name** | `static-sgraph-ai` -- immediately clear what the bucket contains. |
| **Region suffix** | Enables multi-region discovery. Scripts can enumerate buckets by pattern. |

### 3.4 Bucket Policy

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowCloudFrontOAC",
      "Effect": "Allow",
      "Principal": {
        "Service": "cloudfront.amazonaws.com"
      },
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::{bucket-name}/*",
      "Condition": {
        "StringEquals": {
          "AWS:SourceArn": "arn:aws:cloudfront::{account-id}:distribution/{distribution-id}"
        }
      }
    }
  ]
}
```

**Rules:**
- Bucket is PRIVATE. No public access.
- Only CloudFront (via Origin Access Control) can read objects.
- Each CloudFront distribution is explicitly listed in the condition.
- No `s3:PutObject` permission for CloudFront -- it is read-only.

## 4. Rollback Procedure

### 4.1 How Rollback Works

Rollback is replacing the contents of `latest/` with the contents of a previous `releases/{version}/` folder. No rebuild, no CI pipeline, no code changes.

```
Current state:
  websites/send-sgraph-ai/latest/     → contains v0.7.6 content
  websites/send-sgraph-ai/releases/
    ├── v0.7.4/
    ├── v0.7.5/
    └── v0.7.6/                       → identical to latest/

Rollback to v0.7.5:
  1. aws s3 rm s3://{bucket}/websites/send-sgraph-ai/latest/ --recursive
  2. aws s3 cp s3://{bucket}/websites/send-sgraph-ai/releases/v0.7.5/ \
               s3://{bucket}/websites/send-sgraph-ai/latest/ --recursive
  3. aws cloudfront create-invalidation --distribution-id {dist-id} --paths "/*"

After rollback:
  websites/send-sgraph-ai/latest/     → now contains v0.7.5 content
  websites/send-sgraph-ai/releases/
    ├── v0.7.4/                       → untouched
    ├── v0.7.5/                       → untouched (source of rollback)
    └── v0.7.6/                       → untouched (still archived)
```

### 4.2 Rollback Python Script

The rollback is a Python script (not YAML, per the CI principle):

```python
# scripts/rollback_static_site.py
# Usage: python scripts/rollback_static_site.py --site send-sgraph-ai --version v0.7.5

def rollback(site: str, version: str, bucket: str, distribution_id: str):
    """Rollback a site to a previous release version."""
    release_prefix = f"websites/{site}/releases/{version}/"
    latest_prefix  = f"websites/{site}/latest/"

    # 1. Verify the target release exists
    verify_release_exists(bucket, release_prefix)

    # 2. Clear latest/
    s3_delete_prefix(bucket, latest_prefix)

    # 3. Copy release to latest/
    s3_copy_prefix(bucket, release_prefix, latest_prefix)

    # 4. Invalidate CloudFront cache
    cloudfront_invalidate(distribution_id, paths=["/*"])

    # 5. Verify rollback
    deployed = fetch_deploy_json(site)
    assert deployed["version"] == version
```

### 4.3 Rollback Rules

| Rule | Detail |
|------|--------|
| **Releases are never modified** | Rollback copies FROM a release TO latest/. The release folder is never touched. |
| **deploy.json confirms the active version** | After rollback, `latest/deploy.json` contains the rolled-back version's metadata. |
| **CloudFront invalidation is mandatory** | Without it, cached content at edge locations will serve the old (pre-rollback) version until TTL expires. |
| **Rollback is idempotent** | Running the same rollback twice produces the same result. |
| **Forward deployment after rollback** | A new release (v0.7.7) follows the normal deployment process -- it does not need to know a rollback happened. |

### 4.4 Emergency Rollback (Under 60 Seconds)

For critical incidents, the entire rollback can complete in under 60 seconds:

| Step | Time |
|------|------|
| S3 delete `latest/` | ~5s (small file count) |
| S3 copy from `releases/{version}/` | ~10s (small file count) |
| CloudFront invalidation request | ~2s (async -- propagation takes 2-5 minutes) |
| Verification | ~3s |
| **Total** | **~20s** (edge propagation: 2-5 minutes) |

## 5. CI Artifacts Storage

### 5.1 CI Folder Structure

```
ci/
└── 2026/
    └── 02/
        └── 28/
            ├── sgraph-ai--v0.7.6--build.log
            ├── sgraph-ai--v0.7.6--git-diff.txt
            ├── sgraph-ai--v0.7.6--git-diff-prev.txt
            ├── sgraph-ai--v0.7.6--git-commit.txt
            ├── sgraph-ai--v0.7.6--test-results.xml
            ├── sgraph-ai--v0.7.6--deps.json
            ├── send-sgraph-ai--v0.7.6--build.log
            ├── send-sgraph-ai--v0.7.6--git-diff.txt
            └── send-sgraph-ai--v0.7.6--git-commit.txt
```

### 5.2 Artifact Naming Convention

```
{site}--{version}--{artifact}.{ext}
```

| Component | Format | Example |
|-----------|--------|---------|
| **site** | Hyphenated domain name | `sgraph-ai`, `send-sgraph-ai` |
| **version** | Release tag | `v0.7.6` |
| **artifact** | Descriptive name | `build`, `git-diff`, `git-commit`, `test-results`, `deps` |
| **ext** | File extension | `.log`, `.txt`, `.xml`, `.json` |

### 5.3 CI Artifacts Inventory

| Artifact | Content | Format | Purpose |
|----------|---------|--------|---------|
| **build.log** | Full build output | Text | Debug deployment failures |
| **git-diff.txt** | `git diff main...HEAD` | Unified diff | See what changed vs. main branch |
| **git-diff-prev.txt** | `git diff v{prev}...v{current}` | Unified diff | See what changed from previous version |
| **git-commit.txt** | Commit messages since last release | Text | Release notes context |
| **test-results.xml** | pytest output in JUnit XML | XML | Evidence that tests passed |
| **deps.json** | `pip freeze` + `npm list` output | JSON | Exact dependency versions |

### 5.4 CI Artifacts Are Temporal, Not Permanent

CI artifacts are stored by date because they represent the build context at a point in time. They are NOT stored inside `releases/{version}/` because:

1. They are not part of the deployed website
2. They would inflate the release size
3. Multiple builds on the same day (different sites) share the same date folder
4. They have a different retention policy (can be archived or deleted after 90 days)

The one exception is `deploy.json` inside each release, which records the deployment metadata permanently.

## 6. CloudFront Origin Configuration

### 6.1 One Distribution Per Site

Each SGraph Send website has its own CloudFront distribution:

| Domain | CloudFront Distribution | S3 Origin Path |
|--------|------------------------|---------------|
| `sgraph.ai` | `d-sgraph-ai` | `/websites/sgraph-ai/latest` |
| `send.sgraph.ai` | `d-send-sgraph-ai` | `/websites/send-sgraph-ai/latest` |
| `admin.sgraph.ai` | `d-admin-sgraph-ai` | `/websites/admin-sgraph-ai/latest` |
| `vault.sgraph.ai` | `d-vault-sgraph-ai` | `/websites/vault-sgraph-ai/latest` |
| `tools.sgraph.ai` | `d-tools-sgraph-ai` | `/websites/tools-sgraph-ai/latest` |
| `docs.sgraph.ai` | `d-docs-sgraph-ai` | `/websites/docs-sgraph-ai/latest` |
| `cdn.sgraph.ai` | `d-cdn-sgraph-ai` | `/cdn/latest` |

### 6.2 Why Separate Distributions (Not Path-Based Routing)

Alternative considered: a single CloudFront distribution with multiple origins and path-based cache behaviours (`/send/*` routes to send-sgraph-ai, `/admin/*` routes to admin-sgraph-ai).

**Rejected because:**

| Concern | Impact |
|---------|--------|
| **Cache invalidation scope** | Invalidating `/*` on a shared distribution purges all sites, not just the one being deployed |
| **Independent TLS certificates** | Each site may need its own certificate (especially for naked domain `sgraph.ai`) |
| **Independent caching policies** | Admin site needs short TTLs; docs site can have long TTLs |
| **Independent error pages** | Each site has its own 404 page |
| **Blast radius** | A misconfiguration on one distribution does not affect others |
| **CORS headers** | cdn.sgraph.ai needs different CORS headers from the website distributions |

### 6.3 CloudFront Distribution Configuration Template

```yaml
# Template for each site's CloudFront distribution
Distribution:
  Origins:
    - Id: S3-primary
      DomainName: {bucket}.s3.{region}.amazonaws.com
      OriginPath: /websites/{site}/latest      # <-- key: maps to latest/
      S3OriginConfig:
        OriginAccessIdentity: ""
      OriginAccessControlId: {oac-id}          # Origin Access Control (not OAI)

  DefaultCacheBehavior:
    TargetOriginId: S3-primary
    ViewerProtocolPolicy: redirect-to-https
    CachePolicyId: {managed-caching-optimized}
    ResponseHeadersPolicyId: {cors-policy-id}   # only for cdn.sgraph.ai
    Compress: true
    DefaultTTL: 300                             # 5 minutes (low during iteration)
    MaxTTL: 3600                                # 1 hour
    MinTTL: 0

  DefaultRootObject: index.html

  CustomErrorResponses:
    - ErrorCode: 403                            # S3 returns 403 for missing objects
      ResponseCode: 404
      ResponsePagePath: /404.html               # custom 404 page
      ErrorCachingMinTTL: 60

  Aliases:
    - {domain}                                  # e.g., send.sgraph.ai

  ViewerCertificate:
    AcmCertificateArn: {acm-cert-arn}           # *.sgraph.ai wildcard or per-site
    SslSupportMethod: sni-only
    MinimumProtocolVersion: TLSv1.2_2021
```

### 6.4 Origin Failover (Multi-Region)

For multi-region deployments, each distribution has an origin group with failover:

```yaml
OriginGroups:
  - Id: S3-failover-group
    FailoverCriteria:
      StatusCodes: [500, 502, 503, 504]
    Members:
      - OriginId: S3-primary           # eu-west-1 bucket
      - OriginId: S3-secondary         # us-east-1 bucket

Origins:
  - Id: S3-primary
    DomainName: {bucket-eu-west-1}.s3.eu-west-1.amazonaws.com
    OriginPath: /websites/{site}/latest

  - Id: S3-secondary
    DomainName: {bucket-us-east-1}.s3.us-east-1.amazonaws.com
    OriginPath: /websites/{site}/latest
```

If the primary origin returns a 5xx error, CloudFront automatically routes to the secondary origin. No DNS change, no manual intervention.

## 7. Multi-Region Replication

### 7.1 S3 Cross-Region Replication (CRR)

S3 CRR automatically replicates objects from the primary bucket to secondary buckets in other regions.

```
Primary:    123456789012--static-sgraph-ai--eu-west-1
                          ↓ (CRR)
Secondary:  123456789012--static-sgraph-ai--us-east-1
                          ↓ (CRR, future)
Tertiary:   123456789012--static-sgraph-ai--ap-southeast-1
```

### 7.2 CRR Configuration

```yaml
ReplicationConfiguration:
  Role: arn:aws:iam::{account}:role/s3-replication-static-sgraph-ai
  Rules:
    - Id: replicate-all-content
      Status: Enabled
      Filter:
        Prefix: ""                    # replicate everything
      Destination:
        Bucket: arn:aws:s3:::{bucket-us-east-1}
        StorageClass: STANDARD
      DeleteMarkerReplication:
        Status: Enabled               # replicate deletions too (for rollback)
```

### 7.3 CRR Properties

| Property | Detail |
|----------|--------|
| **Replication lag** | Typically 15 minutes for most objects (S3 Replication Time Control can guarantee 99.99% within 15 min, at additional cost) |
| **What replicates** | New objects, object metadata, delete markers |
| **What does NOT replicate** | Objects that existed before CRR was enabled (must be manually synced with `aws s3 sync`) |
| **Versioning required** | Both source and destination buckets must have versioning enabled |
| **Cost** | S3 replication requests + storage in destination region. No cross-region data transfer charges for replication itself. |

### 7.4 Replication Monitoring

| Signal | Tool | Alert Threshold |
|--------|------|----------------|
| Replication lag | S3 Replication Metrics (ReplicationLatency) | >30 minutes |
| Failed replications | S3 Replication Metrics (OperationsFailedReplication) | >0 |
| Pending replication bytes | S3 Replication Metrics (BytesPendingReplication) | >10MB for >1 hour |

## 9. Deployment Flow

### 9.1 Standard Deployment (New Release)

```
Step 1: Build static site
  python scripts/build_static_site.py --site send-sgraph-ai

Step 2: Deploy to releases/ (immutable)
  python scripts/deploy_to_s3.py \
    --site send-sgraph-ai \
    --version v0.7.6 \
    --bucket {bucket} \
    --target releases

Step 3: Deploy to latest/ (mutable -- overwrites previous)
  python scripts/deploy_to_s3.py \
    --site send-sgraph-ai \
    --version v0.7.6 \
    --bucket {bucket} \
    --target latest

Step 4: Store CI artifacts
  python scripts/store_ci_artifacts.py \
    --site send-sgraph-ai \
    --version v0.7.6 \
    --bucket {bucket}

Step 5: Invalidate CloudFront cache
  python scripts/invalidate_cloudfront.py \
    --distribution d-send-sgraph-ai

Step 6: Smoke test
  python scripts/smoke_test.py \
    --url https://send.sgraph.ai \
    --expected-version v0.7.6
```

### 9.2 Script Responsibility Separation

| Script | What It Does | What It Does NOT Do |
|--------|-------------|-------------------|
| `build_static_site.py` | Generate HTML/CSS/JS, create deploy.json | Does not upload anything |
| `deploy_to_s3.py` | Upload to S3 (either releases/ or latest/) | Does not build, does not invalidate cache |
| `store_ci_artifacts.py` | Upload git-diff, build.log, test results to ci/ | Does not touch website content |
| `invalidate_cloudfront.py` | Create CloudFront invalidation | Does not upload or modify S3 content |
| `smoke_test.py` | Verify deployed site responds correctly | Read-only -- no modifications |
| `rollback_static_site.py` | Copy a previous release to latest/ | Does not rebuild, does not modify releases/ |

## 10. Offline Packaging Integration

### 10.1 Package Assembly

An offline package combines specific release versions from multiple folders:

```python
# scripts/package_offline.py
def package_offline(version: str, cdn_version: str, output_dir: str):
    """Assemble offline deployment package."""
    sites = ["sgraph-ai", "send-sgraph-ai", "docs-sgraph-ai", "vault-sgraph-ai"]

    for site in sites:
        s3_download(
            bucket=bucket,
            prefix=f"websites/{site}/releases/{version}/",
            local_dir=f"{output_dir}/websites/{site}/releases/{version}/"
        )

    s3_download(
        bucket=bucket,
        prefix=f"cdn/releases/{cdn_version}/",
        local_dir=f"{output_dir}/cdn/releases/{cdn_version}/"
    )

    # Rewrite CDN URLs from absolute to relative
    rewrite_cdn_urls(output_dir, cdn_version)

    # Generate package manifest
    generate_manifest(output_dir, version, cdn_version)
```

## 12. Open Questions

| # | Question | Impact |
|---|----------|--------|
| 1 | Should `latest/` be a real copy or an S3 redirect (x-amz-website-redirect-location)? | Redirect avoids data duplication but adds latency and complicates CloudFront caching. Recommendation: real copy -- storage is cheap, latency matters. |
| 2 | How long should old releases be retained? | Unlimited for now. Add lifecycle policy later (e.g., archive to S3 Glacier after 6 months). |
| 3 | Should CI artifacts be replicated cross-region? | Probably not -- they are diagnostic, not critical. Exclude from CRR rules to save cost. |
| 4 | How does this integrate with the existing QA deployment pipeline? | The QA pipeline uses git-commit-based image deployment. It needs to be retrofitted to use the `websites/qa-sgraph-ai/` folder structure. |
| 5 | Should we enable S3 Object Lock on `releases/` prefixes? | Yes, for at least one region (per the multi-cloud brief's BCP requirements). 30-day retention minimum on immutable releases. |
