## 1. Executive Summary

This review defines the logging architecture for all SGraph Send static site deployments. Three log sources capture different aspects of activity: **CloudFront access logs** (every HTTP request from users), **S3 server access logs** (every storage operation), and **CloudTrail data events** (IAM-level audit trail of who did what). Each log type serves a different audience and a different question.

**Key design decisions:**

1. Dedicated log bucket per region: `{account-id}--logs-sgraph-ai--{region}`
2. Log separation by prefix: `cloudfront/`, `s3-access/`, `cloudtrail/`
3. No logging to the content bucket -- logs go to a separate bucket to avoid recursive logging
4. CloudTrail data events scoped to the content bucket only (not account-wide, to control cost)
5. Cross-provider comparison matrix for AWS, GCP, Azure, and Cloudflare logging

**Architecture assessment: SOUND.** The three-layer logging model provides complete visibility from user request through CDN to storage. The dedicated log bucket avoids the common anti-pattern of logging into the bucket being monitored.

## 2. Log Architecture Overview

```
User request
    │
    ▼
CloudFront (CDN layer)
    │ ──► CloudFront Access Logs ──► s3://{logs-bucket}/cloudfront/{distribution}/
    │
    ▼
S3 Bucket (storage layer)
    │ ──► S3 Server Access Logs ──► s3://{logs-bucket}/s3-access/{content-bucket}/
    │
    ▼
IAM / API layer
    │ ──► CloudTrail Data Events ──► s3://{logs-bucket}/cloudtrail/
    │
```

Each layer captures different information:

| Layer | What It Answers | Who Uses It |
|-------|----------------|-------------|
| **CloudFront access logs** | Who is visiting? From where? Which pages? What status codes? | Analytics, traffic monitoring, abuse detection |
| **S3 server access logs** | What storage operations are happening? What is CloudFront actually fetching? | Debugging, cache miss analysis, data access auditing |
| **CloudTrail data events** | Which IAM principal accessed or modified the bucket? From which IP? | Security auditing, compliance, incident investigation |

## 3. CloudFront Access Logs

### 3.1 What They Capture

CloudFront standard access logs record every viewer request (HTTP or HTTPS) to a distribution. Each log line contains:

| Field | Description | Use Case |
|-------|-------------|----------|
| `date` / `time` | Request timestamp (UTC) | Time-series analysis, peak detection |
| `x-edge-location` | CloudFront edge POP that served the request | Geographic distribution of users |
| `sc-bytes` | Bytes sent to viewer | Bandwidth usage, cost attribution |
| `c-ip` | Client IP address | Geolocation, abuse detection |
| `cs-method` | HTTP method (GET, HEAD) | Request type analysis |
| `cs(Host)` | Requested hostname | Per-site breakdown (send.sgraph.ai vs sgraph.ai) |
| `cs-uri-stem` | Requested path | Page popularity, 404 detection |
| `sc-status` | HTTP status code | Error rate monitoring |
| `cs(Referer)` | Referrer URL | Traffic source analysis |
| `cs(User-Agent)` | User agent string | Browser/bot identification |
| `x-edge-result-type` | Hit, Miss, Error, etc. | Cache hit ratio |
| `time-taken` | Request processing time (seconds) | Latency monitoring |

### 3.2 Log Bucket and Prefix

```
Bucket: {account-id}--logs-sgraph-ai--{region}
Prefix: cloudfront/{distribution-alias}/

Example paths:
  cloudfront/send-sgraph-ai/E1A2B3C4D5.2026-02-28-14.abc123.gz
  cloudfront/cdn-sgraph-ai/E6F7G8H9I0.2026-02-28-14.def456.gz
  cloudfront/sgraph-ai/E1K2L3M4N5.2026-02-28-14.ghi789.gz
```

Each distribution logs to its own prefix. Log files are gzipped and named with the distribution ID, date-hour, and a unique identifier.

### 3.3 Configuration

```yaml
# Per CloudFront distribution
Logging:
  Enabled: true
  IncludeCookies: false            # no cookies to log (static sites)
  Bucket: {account-id}--logs-sgraph-ai--{region}.s3.amazonaws.com
  Prefix: cloudfront/{distribution-alias}/
```

**Note:** CloudFront can also use real-time logs (Kinesis Data Streams) for live analysis. Standard logs have a 5-60 minute delivery delay. Start with standard logs; add real-time if operational monitoring requires it.

### 3.4 Privacy Consideration

CloudFront logs contain client IP addresses. For SGraph Send, which emphasises privacy:

| Option | Trade-off |
|--------|----------|
| **Log IPs as-is** | Full forensic capability, but stores PII (IP addresses) |
| **Hash IPs in post-processing** | Process logs with a script that hashes IPs before long-term storage. Matches the backend's `ip_hash` approach. |
| **Disable IP logging** | Not possible with standard CloudFront logs (IPs are always included) |

**Recommendation:** Post-process logs with a Python script that hashes the `c-ip` field using SHA-256 with a daily salt (same algorithm as the backend `ip_hash`). Store only the processed logs for long-term retention. Keep raw logs for 7 days for incident response, then delete.

## 4. S3 Server Access Logs

### 4.1 What They Capture

S3 server access logs record every API request made to the content bucket. This includes requests from CloudFront (via OAC), CI pipeline uploads, and any direct API access.

| Field | Description | Use Case |
|-------|-------------|----------|
| `Bucket Owner` | Account ID | Ownership verification |
| `Bucket` | Content bucket name | Multi-bucket correlation |
| `Time` | Request timestamp | Timeline analysis |
| `Remote IP` | Requester's IP | Source identification |
| `Requester` | IAM principal ARN | Who made the request (CloudFront, CI, admin) |
| `Request-ID` | S3 request ID | Cross-reference with CloudTrail |
| `Operation` | S3 API operation | GetObject, PutObject, DeleteObject, ListBucket |
| `Key` | Object key (file path) | Which file was accessed |
| `Request-URI` | Full request URI | Complete request details |
| `HTTP Status` | Response code | Error detection |
| `Bytes Sent` | Response size | Bandwidth attribution |
| `Object Size` | Size of the object | Storage analysis |

### 4.2 Log Bucket and Prefix

```
Bucket: {account-id}--logs-sgraph-ai--{region}
Prefix: s3-access/{content-bucket-name}/

Example paths:
  s3-access/123456789012--static-sgraph-ai--eu-west-1/2026-02-28-14-30-00-ABC123
  s3-access/123456789012--static-sgraph-ai--eu-west-1/2026-02-28-14-35-00-DEF456
```

### 4.3 Configuration

S3 server access logging is enabled on the **content bucket** (not the log bucket):

```yaml
# On the content bucket: {account-id}--static-sgraph-ai--{region}
LoggingConfiguration:
  DestinationBucketName: {account-id}--logs-sgraph-ai--{region}
  LogFilePrefix: s3-access/{content-bucket-name}/
```

**Critical rule:** Do NOT enable server access logging on the log bucket itself. This creates a recursive logging loop (each log delivery generates a new log entry, which generates another log delivery...). The log bucket has no access logging.

### 4.4 Key Operations to Monitor

| Operation | Normal Source | Suspicious If |
|-----------|-------------|---------------|
| `REST.GET.OBJECT` | CloudFront (via OAC) | From unknown IAM principals |
| `REST.PUT.OBJECT` | CI pipeline (deploy script) | Outside deployment windows |
| `REST.DELETE.OBJECT` | Rollback script, lifecycle policies | Unexpected bulk deletions |
| `REST.GET.BUCKET` (ListBucket) | CI scripts, admin tools | From unknown sources |
| `REST.HEAD.OBJECT` | CloudFront cache validation | Normal -- frequent |

## 5. CloudTrail Data Events

### 5.1 What They Capture

CloudTrail data events provide IAM-level auditing. Unlike S3 server access logs (which show what happened to objects), CloudTrail shows **who** did it with **which credentials** and **from where**.

| Field | Description | Use Case |
|-------|-------------|----------|
| `eventTime` | UTC timestamp | Timeline reconstruction |
| `eventSource` | AWS service (s3.amazonaws.com) | Service identification |
| `eventName` | API action (GetObject, PutObject) | Operation type |
| `userIdentity` | Full IAM principal details | Who: user, role, federated identity |
| `sourceIPAddress` | Requester IP or service name | Where from: IP or "cloudfront.amazonaws.com" |
| `requestParameters` | Bucket name, object key | What was accessed |
| `responseElements` | Response details | Success/failure details |
| `errorCode` | Error (AccessDenied, NoSuchKey) | Failed access attempts |

### 5.2 Scope and Storage

CloudTrail data events should be scoped to the content bucket only:

```yaml
# CloudTrail trail configuration
DataEvents:
  - Type: AWS::S3::Object
    Values:
      - arn:aws:s3:::{account-id}--static-sgraph-ai--{region}/
```

**Why scoped:** Account-wide S3 data events capture every GetObject/PutObject across all buckets. This is expensive (CloudTrail charges per 100,000 events) and noisy. Scoping to the static content bucket captures only the relevant activity.

### 5.3 Log Delivery

```
Bucket: {account-id}--logs-sgraph-ai--{region}
Prefix: cloudtrail/

Example paths:
  cloudtrail/AWSLogs/{account-id}/CloudTrail/{region}/2026/02/28/{account-id}_CloudTrail_{region}_20260228T1430Z_abc123.json.gz
```

CloudTrail delivers JSON-formatted logs, gzipped, with a standardised path structure. The logs are also optionally available in CloudWatch Logs for real-time querying.

### 5.4 Key Events to Monitor

| Event | Significance |
|-------|-------------|
| `PutObject` from unknown principal | Unauthorized deployment attempt |
| `DeleteObject` on `releases/` prefix | Attempt to modify immutable releases |
| `GetObject` with `AccessDenied` | Failed access attempts (probing) |
| `PutBucketPolicy` | Bucket policy modification (critical) |
| `DeleteBucket` | Catastrophic -- should never happen |
| Any event from unexpected `sourceIPAddress` | Credential compromise indicator |

## 6. Log Bucket Design

### 6.1 Naming Convention

```
{account-id}--logs-sgraph-ai--{region}
```

Mirrors the content bucket naming but with `logs` instead of `static`:

| Purpose | Bucket Name |
|---------|------------|
| Content (eu-west-1) | `123456789012--static-sgraph-ai--eu-west-1` |
| Logs (eu-west-1) | `123456789012--logs-sgraph-ai--eu-west-1` |
| Content (us-east-1) | `123456789012--static-sgraph-ai--us-east-1` |
| Logs (us-east-1) | `123456789012--logs-sgraph-ai--us-east-1` |

### 6.2 Log Bucket Internal Structure

```
{account-id}--logs-sgraph-ai--{region}/
├── cloudfront/
│   ├── sgraph-ai/               <-- logs from sgraph.ai distribution
│   ├── send-sgraph-ai/          <-- logs from send.sgraph.ai distribution
│   ├── admin-sgraph-ai/         <-- logs from admin.sgraph.ai distribution
│   ├── vault-sgraph-ai/
│   ├── cdn-sgraph-ai/
│   ├── tools-sgraph-ai/
│   └── docs-sgraph-ai/
│
├── s3-access/
│   └── {content-bucket-name}/   <-- S3 access logs for the content bucket
│
├── cloudtrail/
│   └── AWSLogs/                 <-- standard CloudTrail path structure
│       └── {account-id}/
│           └── CloudTrail/
│               └── {region}/
│                   └── 2026/
│
└── processed/                   <-- post-processed logs (IP-hashed, aggregated)
    ├── cloudfront/
    │   └── 2026/02/28/
    │       ├── sgraph-ai--hourly-summary.json
    │       └── send-sgraph-ai--hourly-summary.json
    └── alerts/
        └── 2026/02/28/
```

### 6.3 Log Bucket Configuration

```yaml
# Log bucket settings
Versioning: Disabled                  # logs are append-only, versioning not needed
Encryption: SSE-S3 (AES-256)         # encrypt logs at rest
PublicAccess: BlockAll                # no public access
ObjectLock: Disabled                  # not needed for logs (use lifecycle for retention)

LifecycleRules:
  - Id: archive-old-logs
    Status: Enabled
    Transitions:
      - Days: 30
        StorageClass: STANDARD_IA     # infrequent access after 30 days
      - Days: 90
        StorageClass: GLACIER_IR      # deep archive after 90 days
    Expiration:
      Days: 365                       # delete after 1 year

  - Id: delete-raw-cloudfront-logs
    Status: Enabled
    Filter:
      Prefix: cloudfront/
    Expiration:
      Days: 7                         # raw logs kept 7 days (processed logs retained longer)

  - Id: retain-processed-logs
    Status: Enabled
    Filter:
      Prefix: processed/
    Transitions:
      - Days: 90
        StorageClass: GLACIER_IR
    Expiration:
      Days: 730                       # processed logs kept 2 years
```

### 6.4 Log Bucket Access Policy

Only specific principals can write to the log bucket:

| Principal | Permission | Why |
|-----------|-----------|-----|
| CloudFront service | `s3:PutObject` on `cloudfront/` | Deliver access logs |
| S3 logging service | `s3:PutObject` on `s3-access/` | Deliver server access logs |
| CloudTrail service | `s3:PutObject` on `cloudtrail/` | Deliver data event logs |
| Log processing Lambda/script | `s3:GetObject`, `s3:PutObject` on `processed/` | Read raw logs, write processed output |
| Admin IAM role | `s3:GetObject` on `*` | Read logs for investigation |

No `s3:DeleteObject` permission for any automated principal. Log deletion happens only via lifecycle rules.

## 7. Cross-Provider Comparison Matrix

The brief requests a comparison of logging capabilities across AWS, GCP, Azure, and Cloudflare. This matrix covers the key categories.

### 7.1 CDN/Request Logging

| Category | AWS (CloudFront) | GCP (Cloud CDN) | Azure (Azure CDN) | Cloudflare |
|----------|-----------------|-----------------|-------------------|------------|
| **Log type** | Standard access logs (S3) or Real-time logs (Kinesis) | Cloud Logging (integrated) | Diagnostic logs (Azure Monitor) | Logpush (Enterprise) or Analytics API (all plans) |
| **Delivery latency** | Standard: 5-60 min. Real-time: seconds | Near real-time (Cloud Logging) | Near real-time (Azure Monitor) | Logpush: ~60s. Analytics: 1 min aggregation |
| **Fields captured** | 33 fields (IP, path, status, timing, edge location, result type) | ~20 fields (IP, path, status, cache hit/miss, latency) | ~25 fields (IP, path, status, POP, cache behaviour) | ~40 fields (IP, path, status, edge location, WAF actions, bot score) |
| **Storage destination** | S3 bucket (standard) or Kinesis Data Stream (real-time) | Cloud Storage, BigQuery, Pub/Sub | Storage Account, Log Analytics, Event Hub | R2, S3, GCS, Azure Blob, or Datadog/Splunk/Sumo |
| **Cost model** | Standard: free (S3 storage cost only). Real-time: Kinesis charges | Cloud Logging: first 50GB/month free, then $0.50/GB | Free tier varies by CDN tier. Log Analytics: ~$2.76/GB ingested | Logpush: Enterprise only ($). Analytics API: free on all plans |
| **Retention** | S3 lifecycle (configurable) | 30 days default in Cloud Logging, longer in GCS/BigQuery | 30-90 days in Log Analytics, longer in Storage Account | Logpush: destination-dependent. Analytics: 30 days (free), longer (enterprise) |
| **IP privacy** | Must post-process to hash | Can configure data access logging to exclude IPs | Can configure anonymisation in some tiers | Automatic IP anonymisation available |
| **Queryability** | Athena (SQL over S3), CloudWatch Insights | BigQuery (SQL, fast), Cloud Logging query | Kusto (KQL) in Log Analytics | Analytics API (GraphQL), SQL on exported data |

### 7.2 Storage Access Logging

| Category | AWS (S3) | GCP (Cloud Storage) | Azure (Blob Storage) | Cloudflare (R2) |
|----------|----------|--------------------|--------------------|-----------------|
| **Log type** | S3 server access logs | Cloud Audit Logs (data access) | Storage Analytics logs + Azure Monitor | Limited -- R2 is newer, logging capabilities evolving |
| **What's captured** | Every S3 API call: operation, key, requester, status, bytes | Every GCS API call: operation, object, caller, status | Every Blob API call: operation, object, caller | Basic metrics only; full access logs TBD |
| **Delivery** | S3 logs to a target bucket (best-effort, minutes delay) | Cloud Logging (near real-time) | Storage Account or Azure Monitor (minutes delay) | Metrics API; no per-request access logs yet |
| **Cost** | Free (only pay for log storage in S3) | Cloud Audit Logs: free for admin events, data access logs charged as Cloud Logging | Free (storage analytics), Azure Monitor charges apply | Minimal -- R2 metrics are free |
| **Completeness** | Best-effort delivery -- S3 logs do not guarantee 100% of requests are logged | Guaranteed delivery through Cloud Audit Logs | Guaranteed through Azure Monitor | Incomplete -- R2 logging is immature |

### 7.3 Audit Trail / IAM Logging

| Category | AWS (CloudTrail) | GCP (Cloud Audit Logs) | Azure (Activity Log / Monitor) | Cloudflare (Audit Log) |
|----------|-----------------|----------------------|------------------------------|----------------------|
| **Scope** | API-level: who called what, from where, with which credentials | API-level: admin activity + data access (configurable) | Control plane + data plane (configurable) | Account-level actions: login, setting changes, API token usage |
| **Data events cost** | ~$0.10 per 100,000 events (scoped to specific resources) | Admin: free. Data access: charged as Cloud Logging | Activity Log: free. Data events via Monitor: charged | Free (audit log is included) |
| **Retention** | 90 days in CloudTrail console; unlimited in S3 | 400 days in Cloud Logging; unlimited in GCS/BigQuery | 90 days in Activity Log; unlimited in Storage Account | 18 months (Enterprise) |
| **Real-time alerting** | CloudWatch alarms on CloudTrail events via EventBridge | Cloud Monitoring alerts on audit log entries | Azure Monitor alerts | Webhook notifications |
| **Tamper protection** | CloudTrail log file integrity validation (SHA-256 digest chain) | Cloud Logging is append-only; export to GCS with Object Lock for immutability | Immutable storage on Blob for exported logs | No export immutability guarantee |

### 7.4 Summary Assessment

| Provider | CDN Logging | Storage Logging | Audit Trail | Overall |
|----------|------------|----------------|-------------|---------|
| **AWS** | Mature, flexible (standard + real-time), good queryability via Athena | Best-effort but comprehensive fields | Industry-leading (CloudTrail) | Strong across all three layers |
| **GCP** | Good integration with Cloud Logging and BigQuery | Strong (guaranteed delivery via Cloud Audit Logs) | Excellent (integrated Cloud Audit Logs) | Strong, excellent queryability |
| **Azure** | Good (Azure Monitor integration) | Good (Storage Analytics + Monitor) | Good (Activity Log + Monitor) | Solid, but more complex configuration |
| **Cloudflare** | Enterprise-only for full logs; Analytics API covers basics | Immature (R2 is new) | Basic (account-level only, not per-object) | Gaps in storage logging; CDN analytics good on all plans |

### 7.5 Visibility Gaps by Provider

| Gap | Impact | Mitigation |
|-----|--------|-----------|
| **Cloudflare R2 lacks per-request access logs** | Cannot audit which objects were accessed at storage level | Use Cloudflare Analytics API for CDN-level access patterns; accept storage-level gap for now |
| **AWS S3 server access logs are best-effort** | May miss ~1% of log entries | Supplement with CloudTrail data events for critical operations (deletions, policy changes) |
| **Azure CDN logging varies by tier** | Standard tier has limited fields vs Premium | Evaluate which CDN tier is needed based on logging requirements |
| **GCP Cloud CDN logs lack some fields available in CloudFront** | Fewer edge-location and cache-detail fields | Acceptable trade-off; BigQuery queryability compensates |

## 8. Alerting Architecture

### 8.1 Critical Alerts (Immediate)

| Alert | Source | Condition | Action |
|-------|--------|-----------|--------|
| **Bucket policy change** | CloudTrail | `PutBucketPolicy` event | Notify security team immediately |
| **Delete on releases/ prefix** | CloudTrail | `DeleteObject` where key starts with `websites/*/releases/` | Block if possible; alert immediately |
| **Bulk deletion** | CloudTrail | >10 `DeleteObject` events in 5 minutes | Alert -- possible compromise or misconfigured rollback |
| **Unknown IAM principal** | CloudTrail | `userIdentity.arn` not in known-principals list | Alert -- possible credential leak |
| **5xx error spike** | CloudFront logs | >5% of requests returning 5xx in 5-minute window | Alert -- origin failure |

### 8.2 Warning Alerts (Monitor)

| Alert | Source | Condition | Action |
|-------|--------|-----------|--------|
| **Cache hit ratio drop** | CloudFront logs | Cache hit ratio drops below 80% | Investigate -- possible TTL misconfiguration or content churn |
| **Unusual traffic spike** | CloudFront logs | Request rate >3x normal for time of day | Investigate -- possible DDoS or bot activity |
| **Replication lag** | S3 CRR metrics | Replication lag >30 minutes | Check CRR health |
| **Log delivery stopped** | Monitoring | No new log files for >2 hours | Check logging configuration |

## 9. Log Processing Pipeline

### 9.1 Pipeline Architecture

```
Raw logs (CloudFront, S3, CloudTrail)
    │
    ▼
Python processing script (Lambda or scheduled)
    │
    ├── Hash IP addresses (SHA-256 + daily salt)
    ├── Aggregate hourly summaries
    ├── Detect anomalies (spike detection, unknown principals)
    └── Generate alerts
    │
    ▼
Processed logs → s3://{logs-bucket}/processed/
Alerts → SNS topic → email / Slack / PagerDuty
Metrics → CloudWatch custom metrics (optional)
```

### 9.2 Processing Script (Python)

Per the brief's CI principle, log processing logic lives in Python:

```python
# scripts/process_cloudfront_logs.py

def process_cloudfront_logs(bucket: str, prefix: str, date: str):
    """Process raw CloudFront logs: hash IPs, aggregate, detect anomalies."""

    raw_logs = download_logs(bucket, prefix, date)

    for log_entry in parse_cloudfront_log(raw_logs):
        # Hash client IP
        log_entry["c_ip_hash"] = hash_ip(log_entry["c_ip"], daily_salt(date))
        del log_entry["c_ip"]

        # Aggregate
        aggregate_by_hour(log_entry)
        aggregate_by_path(log_entry)
        aggregate_by_status(log_entry)

    # Write processed output
    upload_processed(bucket, f"processed/cloudfront/{date}/", aggregated_data)

    # Check for anomalies
    anomalies = detect_anomalies(aggregated_data)
    if anomalies:
        send_alerts(anomalies)
```

## 11. Open Questions

| # | Question | Impact |
|---|----------|--------|
| 1 | Should we use CloudFront real-time logs (Kinesis) from the start, or standard logs? | Real-time adds cost and complexity. Recommendation: start with standard, add real-time only if operational monitoring requires <1 minute latency. |
| 2 | How long should raw (un-hashed) CloudFront logs be retained? | Recommendation: 7 days for incident response, then delete. Processed (IP-hashed) logs kept for 2 years. |
| 3 | Should log processing run as a Lambda function or a scheduled script? | Lambda is event-driven (triggered by new log file arrival). Scheduled script is simpler. Recommendation: Lambda for CloudFront logs (high volume), scheduled for S3/CloudTrail (lower volume). |
| 4 | Do we need Athena tables for ad-hoc querying of CloudFront logs? | Useful for debugging and analytics. Cost is per-query. Recommendation: set up Athena but only query on demand. |
| 5 | Should processed log summaries be replicated cross-region? | Probably yes -- log summaries are small and valuable for cross-region incident investigation. |
