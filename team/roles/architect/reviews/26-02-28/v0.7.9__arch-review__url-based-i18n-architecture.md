# Architecture Decision: URL-Based i18n Routing for sgraph.ai

**Version:** v0.7.9
**Date:** 2026-02-28
**Role:** Architect
**Status:** Recommended — awaiting human decision on two open questions

---

## Context

The sgraph.ai marketing website currently has a JS-based i18n system (`SgI18n`) that switches translations on the same URL using `data-i18n` attributes. The human requires direct URL-based routing:

- `sgraph.ai/pricing/` → English (default)
- `sgraph.ai/pt-pt/pricing/` → Portuguese (Portugal)
- `sgraph.ai/pt-br/pricing/` or `sgraph.ai/pt/pricing/` → Portuguese (Brazil)
- `sgraph.ai/tlh/pricing/` → Klingon

## Decision: Physical Locale Folders with Pre-Rendered HTML

### Recommended Approach

A **Python build script** (`scripts/generate_i18n_pages.py`) reads English source HTML + locale JSON files and generates complete locale-specific folder trees. Shared assets (CSS, JS, fonts) are NOT duplicated.

### Folder Structure

```
sgraph_ai__website/
  index.html                    # English (source of truth)
  pricing/index.html            # English
  product/index.html            # English
  ...
  pt-pt/                        # Portuguese (Portugal) — GENERATED
    index.html
    pricing/index.html
    product/index.html
    ...
  pt/                           # Portuguese (Brazil) — GENERATED
    ...
  tlh/                          # Klingon — GENERATED
    ...
  css/style.css                 # Shared
  js/i18n.js                    # Shared (modified for URL awareness)
  fonts/                        # Shared
  i18n/*.json                   # Translation source files
```

### Alternatives Considered and Rejected

| Approach | Why Rejected |
|----------|-------------|
| CloudFront rewrite to query param + JS render | FOUC persists; search engines see English at Portuguese URLs |
| Lambda@Edge server-side rendering | Adds SSR to a static site; contradicts architecture |
| Single files with JS URL detection | Same FOUC; no SEO benefit |

### Key Design Details

1. **HTML transformation**: Build script sets `<html lang>`, translates `data-i18n` element content, adjusts asset paths, injects `hreflang` tags, translates `<title>` and `<meta description>`
2. **Relative links work automatically**: Mirrored folder structure means `../product/` resolves correctly within any locale
3. **Asset paths adjusted**: Locale pages are one level deeper; `../css/style.css` becomes `../../css/style.css`
4. **`data-i18n` preserved**: Attributes stay in generated HTML as a runtime fallback layer
5. **Locale switcher navigates**: Instead of JS re-rendering, clicking a locale navigates to the equivalent page in that locale
6. **CloudFront unchanged**: Existing URL rewrite function already handles `/pt-pt/pricing/` → `/pt-pt/pricing/index.html`

### SEO

- `<html lang="pt-PT">` per generated page
- `<link rel="alternate" hreflang="...">` cross-linking all locales
- `<link rel="canonical">` per locale page
- `x-default` hreflang points to English
- Search engines see fully rendered translated content (no JS dependency)

### Impact

- **36 new HTML files** (~360KB total, generated from 12 English pages × 3 locales)
- **Zero changes** to CSS, fonts, CloudFront, or S3 bucket structure
- **Modified**: `js/i18n.js` (URL locale detection, navigate instead of re-render)
- **New**: `scripts/generate_i18n_pages.py`
- **Minor**: Add `meta.description.*` keys to translation JSON files

## Open Questions for Human Decision

1. **Brazilian Portuguese URL slug**: `pt/` (matches current `pt.json` naming) or `pt-br/` (clearer, matches human's original request `sgraph.ai/pt-br/pricing/`)?
2. **Generated files**: Commit to repo (simpler local dev, no build step needed) or generate in CI only (cleaner diffs, no risk of desync)?

## Implementation Phases

1. Build script creation (`scripts/generate_i18n_pages.py`)
2. `i18n.js` modifications (URL detection + navigation)
3. Add `hreflang` and canonical tags to English source pages
4. Generate locale folders, test locally
5. Deploy and validate SEO signals
