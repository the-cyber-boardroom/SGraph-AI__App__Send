## Recommended Batch Plan

### Batch 1: Foundation + Quick Wins (Immediate)

**Goal:** Fix the crypto gap, ship two easy tools, deliver building blocks that unblock everything else.

| # | Element | Layer | Effort | Why First |
|---|---------|-------|--------|-----------|
| 1 | **sg-crypto.js production copy** | Core | Low | Unblocks File Encryptor, sg-public-viewer, sg-key-generator. Current module is unverified — this is a correctness risk. |
| 2 | **sg-upload-dropzone** | Component | Low-Med | Unblocks File Encryptor tool and future sg-file-selector. Pattern needed by multiple tools. |
| 3 | **File Hasher tool** | Tool | Low | Ship immediately. No core module needed. Proves the "add a new tool" checklist works. |
| 4 | **File Encryptor tool** | Tool | Low | Ships once crypto is copied and dropzone exists. High-value — direct product demo. |

| **Dependencies** | Which other elements must exist first |
| **Reference** | Link to the source brief in the SG/Send repo |

### Critical Instruction for Crypto

**The sg-crypto.js production copy is the #1 priority.** The tools team must:
1. Get the exact production file from `sgraph_ai_app_send__ui__user/` (the version that runs on send.sgraph.ai)
2. Copy it byte-for-byte into `core/crypto/v1/v1.0/v1.0.0/sg-crypto.js`
3. Verify wire-format compatibility: encrypt with tools version, decrypt with send version, and vice versa
4. Only THEN build tools that depend on it (File Encryptor, Key Generator)

**Do NOT improve, refactor, or "clean up" the crypto module.** This is the zero-visual-difference principle applied to code: copy the production file, ship it, then improve in v1.1.0.

## Parallelisation Strategy

The tools team can run up to 3 parallel tracks:

| Track | Elements | Blocked By |
|-------|----------|------------|
| **Track A: Crypto + Tools** | Batches 1 + 2 (crypto, dropzone, hasher, encryptor, key generator) | Nothing — start immediately |
| **Track B: Library** | Batch 3 (markdown, library nav, document meta, prompt card) | Nothing — start immediately |
| **Track C: Visualization** | Batch 4 (Drawflow adapter, D3 adapter, workflow editor, tree viewer) | Batch 1 base patterns established |

Tracks A and B can run simultaneously from day one. Track C starts after Track A establishes the core module pattern with the crypto production copy.
