# v0.7.19 Architect Review: BaseComponent Architecture for v0.2.0

**Date:** 2026-03-01
**Role:** Architect
**Version:** v0.7.19
**Context:** Human provided MGraph's `BaseComponent` class as the reference architecture for SGraph Send v0.2.0 UI components

---

## 1. Executive Summary

This review defines the component architecture for SGraph Send v0.2.0, adapted from MGraph's proven `BaseComponent` pattern. The core idea:

- **One base class** (`SendComponent`) that all Web Components extend
- **Separate HTML, CSS, and JS files** per component — fetched at runtime, cached by CloudFront
- **Shadow DOM** for all components — style encapsulation, no global CSS leaks
- **`ComponentPaths` registry** — maps component names to their versioned file paths
- **`SendHelpers` utility class** — shared formatting, escaping, crypto display helpers
- **CDN-friendly** — many small files (10-20ms each from CloudFront) loaded in parallel

### Why This Change

The current v0.1.x architecture has **no base class** — each component duplicates lifecycle boilerplate, event cleanup, and rendering patterns. Templates and CSS are inline (embedded in JS), making surgical IFD overrides require full class redefinition (300-600 lines) instead of patching just the 20-line HTML template or 15-line CSS file that actually changed.

With v0.2.0 moving to S3+CloudFront, the performance concern about many small files disappears. Each file fetches in 10-20ms from edge cache. This unlocks the MGraph pattern: **separate files per concern, loaded in parallel, cached independently**.

## 2. Design Principles

### 2.1. From MGraph: What We Adopt

| MGraph Pattern | SGraph Send Adaptation | Rationale |
|---|---|---|
| `BaseComponent` extends `HTMLElement` | `SendComponent` extends `HTMLElement` | Domain-specific name, same role |
| Shadow DOM + `attachShadow({mode: 'open'})` | Same | Style encapsulation, no CSS conflicts |
| `loadResources()` fetches CSS + HTML in parallel | Same, via `Promise.all()` | CDN makes parallel fetches efficient |
| `ComponentPaths.getComponentPaths(name)` | `SendComponentPaths.resolve(name)` | Maps tag name → `{html, css, js}` paths |
| `Helpers` utility class | `SendHelpers` | Domain-specific helpers (transfer IDs, byte formatting, key display) |
| `$()` / `$$()` Shadow DOM query helpers | Same | Clean shorthand |
| `addTrackedListener()` + `cleanup()` | Same | Prevents memory leaks |
| `emit()` with `composed: true` | Same | Events cross Shadow DOM boundary |
| `whenReady()` promise | Same | Inter-component coordination |

### 2.2. From MGraph: What We Adapt

| MGraph Pattern | SGraph Send Change | Reason |
|---|---|---|
| Global `Helpers` object | `SendHelpers` with domain methods | SGraph needs `formatTransferId()`, `formatKey()`, encryption status helpers |
| `ComponentPaths` with single registry | `SendComponentPaths` with IFD version awareness | Must resolve across IFD versions for surgical overrides |
| Shared CSS loaded once | Shared CSS per component type (user vs admin) | User and admin have different design systems |
| `showError()` inline styles | `showError()` uses shared error CSS | Consistent error display across all components |

### 2.3. What We Don't Adopt

| MGraph Pattern | Reason to Skip |
|---|---|
| Node.js module.exports | SGraph Send is browser-only — no SSR |
| `window.BaseComponent` global | Use ES module or controlled namespace (`window.sgraphSend.BaseComponent`) |

## 3. Architecture: File Structure Per Component

### 3.1. The Separation Principle

Every component is **three files** (or two, if no component-specific CSS):

```
_common/
  js/
    base/
      send-component.js           <- Base class (THE foundation)
      send-component-paths.js     <- Path registry
      send-helpers.js             <- Utility functions
    components/
      send-upload/
        send-upload.js            <- Class definition (extends SendComponent)
        send-upload.html          <- Template (pure HTML, no <script>)
        send-upload.css           <- Component-scoped styles
      send-download/
        send-download.js
        send-download.html
        send-download.css
      send-access-gate/
        send-access-gate.js
        send-access-gate.html
        send-access-gate.css
      send-transparency/
        send-transparency.js
        send-transparency.html
        send-transparency.css
      header-nav/
        header-nav.js
        header-nav.html
        header-nav.css
      footer/
        footer.js
        footer.html
        footer.css
      locale-selector/
        locale-selector.js
        locale-selector.html
        locale-selector.css
    api-client.js                 <- Transfer API calls
    crypto.js                     <- AES-256-GCM (unchanged from v0.1.0)
    i18n.js                       <- Locale detection + translation
  css/
    shared-components.css         <- Shared styles (buttons, cards, status, typography)
    design-tokens.css             <- CSS custom properties (colors, spacing, fonts)
  fonts/
    DMSans-*.woff2
    JetBrainsMono-*.woff2
    fonts.css
```

### 3.2. Why Separate Files Work with CDN

| Concern | With Lambda (v0.1.x) | With CloudFront (v0.2.0) |
|---|---|---|
| **Fetch latency** | 50-200ms per file (Lambda cold start possible) | 10-20ms per file (edge cache) |
| **Parallel loading** | Limited (same Lambda handles all) | Full (CDN serves all in parallel) |
| **Cache granularity** | Coarse (bundle or nothing) | Fine (each file cached independently) |
| **IFD override cost** | Must redefine entire 600-line class | Patch just the 20-line `.html` or `.css` |

**Example: 8 components × 3 files = 24 fetches.** At 15ms each, loaded in parallel across 6 browser connections: **~60ms total**. Acceptable.

### 3.3. Future Consolidation Path

When performance needs tightening (many components, mobile users):

1. **Build step:** Concatenate all component CSS into one file, all JS into one file
2. **HTML inlining:** Embed templates into JS at build time
3. **Tree shaking:** Remove unused components per page

This is a CI/CD pipeline addition — the source architecture stays the same (separate files), only the deployed output changes.

## 4. Architecture: SendComponent Base Class

### 4.1. Class Design

```
SendComponent (extends HTMLElement)
├── Shadow DOM attachment
├── Resource loading (fetchCss, fetchHtml, loadResources)
├── Lifecycle hooks (onReady, bindElements, setupEventListeners)
├── Event helpers (emit, addTrackedListener)
├── Query helpers ($, $$)
├── Utility delegation (escapeHtml, formatBytes, formatNumber)
├── Error display (showError)
├── Cleanup (tracked listeners, ready state)
└── Ready state (isReady, whenReady)
```

### 4.2. Resource Loading Flow

```
1. connectedCallback()
2. └── loadResources()
3.     ├── ComponentPaths.resolve(this.tagName)  →  {html, css}
4.     ├── Promise.all([
5.     │     fetchCss(sharedComponents.css),      ← shared button/card styles
6.     │     fetchCss(component.css),              ← component-specific styles
7.     │     fetchHtml(component.html)             ← template
8.     │   ])
9.     └── shadowRoot.innerHTML = `
10.           <style>${sharedCss}</style>
11.           <style>${componentCss}</style>
12.           ${templateHtml}
13.        `
14. └── bindElements()       ← this.$('#upload-btn') etc.
15. └── setupEventListeners() ← this.addTrackedListener(...)
16. └── onReady()            ← component-specific init
17. └── emit('component-ready')
```

### 4.3. Key Difference from MGraph: CSS Fallback

MGraph's `fetchCss()` warns and returns `''` if CSS not found. SGraph Send should do the same — this enables components that have no component-specific CSS (they inherit everything from shared styles).

```javascript
async fetchCss(url) {
    try {
        const response = await fetch(url);
        if (!response.ok) return '';  // No component CSS — that's fine
        return await response.text();
    } catch {
        return '';  // Network error — degrade gracefully
    }
}
```

### 4.4. Key Difference from MGraph: HTML Required

MGraph's `fetchHtml()` throws on failure. SGraph Send should too — a component without its template is broken and should show an error.

## 5. Architecture: SendComponentPaths Registry

### 5.1. Purpose

Maps a component tag name to its file paths. This is the **single place** that knows where files live — components never hardcode paths.

### 5.2. Design

```javascript
class SendComponentPaths {
    static basePath = '';  // Set at boot: '/_common' or '../_common'

    static sharedCss = {
        components: () => `${SendComponentPaths.basePath}/css/shared-components.css`,
        tokens:     () => `${SendComponentPaths.basePath}/css/design-tokens.css`
    };

    static resolve(tagName) {
        return {
            js:   `${this.basePath}/js/components/${tagName}/${tagName}.js`,
            html: `${this.basePath}/js/components/${tagName}/${tagName}.html`,
            css:  `${this.basePath}/js/components/${tagName}/${tagName}.css`
        };
    }
}
```

### 5.3. IFD Override Awareness

In v0.2.1, if only `send-upload.css` changes, the registry can point to a version-specific override:

```javascript
// v0.2.1 override of SendComponentPaths
SendComponentPaths.resolve = function(tagName) {
    const base = `${this.basePath}/js/components/${tagName}/${tagName}`;

    // Override map: component → { file → version path }
    const overrides = {
        'send-upload': {
            css: '../../v0.2.1/_common/js/components/send-upload/send-upload.css'
        }
    };

    const override = overrides[tagName] || {};
    return {
        js:   override.js   || `${base}.js`,
        html: override.html || `${base}.html`,
        css:  override.css  || `${base}.css`
    };
};
```

This means v0.2.1 only needs to ship the changed `.css` file and a small patch to `SendComponentPaths`. The component JS and HTML are loaded from v0.2.0 via the base path.

## 6. Architecture: SendHelpers

### 6.1. Base Helpers (from MGraph)

```javascript
SendHelpers.escapeHtml(text)        // XSS prevention
SendHelpers.formatNumber(num)       // 1234567 → "1,234,567"
SendHelpers.formatBytes(bytes)      // 1234567 → "1.2 MB"
SendHelpers.isValidUrl(url)         // URL validation
SendHelpers.copyToClipboard(text)   // Async clipboard API
SendHelpers.debounce(fn, wait)      // Event throttling
```

### 6.2. SGraph Send Domain Helpers

```javascript
SendHelpers.formatTransferId(id)        // "abc123def456" → "abc1…f456"
SendHelpers.formatKey(key)              // Base64 key → truncated display
SendHelpers.validateTransferId(id)      // 12 hex chars check
SendHelpers.encryptionStatusLabel(s)    // 'encrypting' → 'Encrypting...'
SendHelpers.encryptionStatusColor(s)    // 'encrypting' → 'var(--color-warning)'
SendHelpers.buildDownloadUrl(tid, key, locale) // Locale-aware download URL
```

### 6.3. Why Not Instance Methods?

MGraph delegates to `Helpers` via instance methods (`this.escapeHtml()`). This is ergonomic — components don't need to know about the `Helpers` class. SGraph Send should do the same:

```javascript
// In SendComponent base class
escapeHtml(text)  { return SendHelpers.escapeHtml(text);  }
formatBytes(b)    { return SendHelpers.formatBytes(b);    }
formatNumber(n)   { return SendHelpers.formatNumber(n);   }
```

## 7. Architecture: Shadow DOM Strategy

### 7.1. Decision: All Components Use Shadow DOM

The current codebase is mixed:
- **User UI:** Light DOM (v0.1.0-v0.1.8) — `this.innerHTML`
- **Admin UI:** Shadow DOM (v0.1.0-v0.1.2), then Light DOM (v0.1.3+)

For v0.2.0, **all components use Shadow DOM** because:

1. **Style encapsulation** — component CSS never leaks to page, page CSS never breaks components
2. **Template loading** — `loadResources()` injects into `this.shadowRoot.innerHTML`, matching MGraph exactly
3. **IFD-friendly** — CSS overrides only affect the targeted component, not the whole page
4. **Consistent pattern** — one way to do things, no Light DOM vs Shadow DOM decisions per component

### 7.2. CSS Custom Properties Cross Shadow DOM

CSS custom properties (design tokens) defined at `:root` or on the `<body>` pass through Shadow DOM boundaries. This means:

- `design-tokens.css` defines `--color-primary`, `--space-4`, `--font-body` on `:root`
- Components inside Shadow DOM can use `var(--color-primary)` in their scoped CSS
- Theming works globally without breaking encapsulation

### 7.3. Events Cross Shadow DOM

MGraph's `emit()` uses `composed: true`, which makes CustomEvents cross Shadow DOM boundaries. SGraph Send adopts the same pattern.

## 9. Architecture: How IFD Overrides Work with This Pattern

### 9.1. v0.2.0 (Base — All Files Present)

```
v0/v0.2/v0.2.0/
  _common/
    js/base/send-component.js
    js/base/send-component-paths.js
    js/base/send-helpers.js
    js/components/send-upload/send-upload.js
    js/components/send-upload/send-upload.html
    js/components/send-upload/send-upload.css
    ...
  en-gb/index.html
  pt-pt/index.html
  ...
```

### 9.2. v0.2.1 (Surgical: Change Only the Upload Template)

```
v0/v0.2/v0.2.1/
  _common/
    js/base/send-component-paths.js    <- Override: point send-upload.html to v0.2.1
    js/components/send-upload/
      send-upload.html                  <- Changed template (the ONLY changed file)
  en-gb/index.html                      <- Updated script loading order
```

The `en-gb/index.html` in v0.2.1 loads:
```html
<!-- Base class from v0.2.0 -->
<script src="../../v0.2.0/_common/js/base/send-component.js"></script>
<script src="../../v0.2.0/_common/js/base/send-helpers.js"></script>

<!-- ComponentPaths: v0.2.1 override (points send-upload.html to v0.2.1) -->
<script src="../_common/js/base/send-component-paths.js"></script>

<!-- Components from v0.2.0 (JS unchanged) -->
<script src="../../v0.2.0/_common/js/components/send-upload/send-upload.js"></script>
<!-- send-upload.js calls loadResources() → ComponentPaths resolves to v0.2.1 HTML -->
```

**Result:** Only 2 new files in v0.2.1 (paths override + HTML template). The JS and CSS come from v0.2.0.

### 9.3. v0.2.2 (Surgical: Change Only the Upload CSS)

```
v0/v0.2/v0.2.2/
  _common/
    js/base/send-component-paths.js    <- Override: point send-upload.css to v0.2.2
    js/components/send-upload/
      send-upload.css                   <- Changed styles only
  en-gb/index.html                      <- Updated script loading order
```

### 9.4. v0.2.3 (Surgical: Change Upload Logic)

```
v0/v0.2/v0.2.3/
  _common/
    js/components/send-upload/
      send-upload.js                    <- New class definition (extends SendComponent)
  en-gb/index.html                      <- Load v0.2.3 JS after v0.2.0 JS
```

Here the override redefines the class entirely (like current v0.1.x pattern), OR patches specific prototype methods (like v0.1.8 does today).

### 9.5. Summary: IFD Override Granularity

| What Changed | Files Needed in Override Version | Old Pattern (v0.1.x) |
|---|---|---|
| Template only | `send-component-paths.js` + `send-upload.html` | Full 600-line class redefinition |
| CSS only | `send-component-paths.js` + `send-upload.css` | Full 600-line class redefinition |
| JS logic only | `send-upload.js` (prototype patch or full redefine) | Full 600-line class redefinition |
| Template + CSS | `send-component-paths.js` + `.html` + `.css` | Full 600-line class redefinition |
| Everything | `.js` + `.html` + `.css` + paths | Full 600-line class redefinition |

**The separate-files pattern makes IFD overrides proportional to the actual change size.**

## 10. Architecture: Page Structure with Locale Folders

### 10.1. How Locale Pages Load Components

Each locale page (`en-gb/index.html`, `pt-pt/index.html`) loads the same set of JS files from `_common/`. The only difference is the pre-rendered text in the HTML.

```html
<!-- en-gb/index.html -->
<!DOCTYPE html>
<html lang="en-gb">
<head>
    <link rel="stylesheet" href="../_common/css/design-tokens.css">
    <link rel="stylesheet" href="../_common/fonts/fonts.css">
</head>
<body>
    <send-header></send-header>
    <main>
        <send-access-gate>
            <send-upload></send-upload>
        </send-access-gate>
        <p class="disclaimer">This is a beta service.</p>
    </main>
    <send-footer></send-footer>

    <!-- Base system -->
    <script src="../_common/js/base/send-helpers.js"></script>
    <script src="../_common/js/base/send-component-paths.js"></script>
    <script src="../_common/js/base/send-component.js"></script>
    <!-- Utilities -->
    <script src="../_common/js/crypto.js"></script>
    <script src="../_common/js/api-client.js"></script>
    <script src="../_common/js/i18n.js"></script>
    <!-- Components -->
    <script src="../_common/js/components/send-header/send-header.js"></script>
    <script src="../_common/js/components/send-footer/send-footer.js"></script>
    <script src="../_common/js/components/send-locale/send-locale.js"></script>
    <script src="../_common/js/components/send-access-gate/send-access-gate.js"></script>
    <script src="../_common/js/components/send-upload/send-upload.js"></script>
    <script src="../_common/js/components/send-transparency/send-transparency.js"></script>
</body>
</html>
```

Note: The page HTML is minimal — just the component tags. All rendering happens inside Shadow DOM via the fetched templates.

### 10.2. i18n Inside Components

Components detect their locale from `<html lang="en-gb">` or from the URL path, then use `I18n.t('key')` in their `onReady()` to translate dynamic text. The static text in the `.html` template uses `data-i18n` attributes.

## 11. ADR Summary

| # | Decision | Rationale |
|---|---|---|
| ADR-1 | `SendComponent` base class for all Web Components | Eliminates boilerplate, enables resource loading pattern |
| ADR-2 | Shadow DOM for all components | Style encapsulation, consistent with MGraph pattern |
| ADR-3 | Separate HTML/CSS/JS files per component | IFD surgical overrides, independent caching, parallel loading |
| ADR-4 | `SendComponentPaths` registry | Single source for file resolution, IFD version-aware |
| ADR-5 | `SendHelpers` utility class | Domain helpers + MGraph base helpers |
| ADR-6 | CDN-first: accept many small files | 10-20ms per file from CloudFront, parallel loading |
| ADR-7 | Future consolidation via build step (not now) | Source stays clean (separate files), CI can bundle later |
| ADR-8 | `shared-components.css` loaded by every component | Buttons, cards, status badges — shared across all components |
| ADR-9 | `design-tokens.css` on page (not in Shadow DOM) | Custom properties cross Shadow DOM — theming works globally |

## 12. Risk Assessment

| Risk | Mitigation |
|---|---|
| Too many HTTP requests on first load | CloudFront caching + HTTP/2 multiplexing. Measure after implementation. |
| Shadow DOM blocks global CSS | Use CSS custom properties for theming (they cross boundaries). Load shared CSS into each Shadow DOM. |
| Template fetch failure | `showError()` with clear message. Graceful degradation. |
| ComponentPaths complexity for IFD | Keep registry simple — just a map of overrides per version. |
| Migration effort from v0.1.x | This is v0.2.0 — a clean consolidation. No incremental migration needed. |
