# v0.7.19 Architect Review: S3+CloudFront Static Migration for send.sgraph.ai

**Date:** 2026-03-01
**Role:** Architect
**Version:** v0.7.19
**Scope:** Architecture for migrating all static pages from send.sgraph.ai to S3+CloudFront, i18n support, v2.0 IFD consolidation

---

## 1. Executive Summary

This review defines the architecture for splitting send.sgraph.ai into two serving layers:

- **S3+CloudFront** serves ALL static content (HTML pages, CSS, JS, fonts, images, locale pages)
- **Lambda** serves ONLY API endpoints (transfers, presigned, health, MCP, Swagger)

The architecture also introduces:
- Full i18n support (15 locales) matching sgraph.ai's pre-rendering approach
- v2.0 IFD consolidation focused on symmetric key encryption features only
- Clean separation of vault/PKI/data rooms (excluded from v2.0, move to another domain)

## 3. Proposed Architecture

### 3.1. CloudFront Origin-Based Routing

```
                    send.sgraph.ai
                         |
                   [CloudFront]
                    Distribution
                   /            \
            Default (*)      API behaviors
                |           (/transfers/*, /presigned/*,
            [S3 Bucket]      /vault/*, /join/*, /info/*,
             /latest/        /mcp*, /docs*, /set-cookie)
                                  |
                           [Lambda Function URL]
                            User Lambda (API only)
```

**Two origins, one domain:**
- **Origin 1 (S3, default):** `{account-id}--static-sgraph-ai--eu-west-2.s3.eu-west-2.amazonaws.com` with origin path `/websites/send-sgraph-ai/latest`. Same bucket as sgraph.ai.
- **Origin 2 (Lambda):** Lambda Function URL as custom origin, HTTPS only, with `X-CloudFront-Secret` header for access control.

### 3.2. CloudFront Behavior Configuration

| Priority | Path Pattern | Origin | Cache Policy | Purpose |
|----------|-------------|--------|-------------|---------|
| 0 | `/transfers/*` | Lambda | CachingDisabled | Transfer API |
| 1 | `/presigned/*` | Lambda | CachingDisabled | Presigned URL API |
| 2 | `/vault/*` | Lambda | CachingDisabled | Vault API (transitional) |
| 3 | `/join/*` | Lambda | CachingDisabled | Join API (transitional) |
| 4 | `/info/*` | Lambda | CachingDisabled | Health/version |
| 5 | `/mcp*` | Lambda | CachingDisabled | MCP protocol |
| 6 | `/set-cookie` | Lambda | CachingDisabled | Cookie route |
| 7 | `/docs*` | Lambda | CachingDisabled | Swagger UI |
| 8 | `/openapi.json` | Lambda | CachingDisabled | OpenAPI spec |
| Default | `*` | S3 | Custom-Static | All static content |

**Why explicit API behaviors (not a single `/api/*`):** The current Lambda routes do NOT use an `/api/` prefix. Routes are `/transfers/create`, `/vault/lookup/{id}`, etc. Changing to `/api/` prefix would break all existing clients, MCP integrations, and download links. The explicit behavior list avoids a breaking change.

**PROPOSED migration path:** In a future version, refactor all Lambda routes under `/api/*` prefix to simplify CloudFront to just two behaviors: `/api/*` -> Lambda, `*` -> S3. PROPOSED -- does not exist yet.

### 3.3. Cache Strategy

| Content Type | Cache-Control Header | CloudFront TTL | Origin |
|-------------|---------------------|----------------|--------|
| HTML | `public, max-age=300` | Default: 300s, Max: 3600s | S3 |
| CSS/JS/JSON | `public, max-age=86400` | Respects header | S3 |
| Fonts/Images | `public, max-age=604800` | Respects header | S3 |
| API responses | No caching | TTL: 0 | Lambda |

### 3.4. Same-Origin Advantage

With CloudFront fronting both S3 and Lambda on the same domain (`send.sgraph.ai`):
- **No CORS needed** for UI->API calls (same origin)
- The UI at `send.sgraph.ai/upload/` calls `send.sgraph.ai/transfers/create` -- same origin
- The existing `ApiClient.baseUrl = ''` (empty string, relative URLs) works without modification
- This eliminates cross-origin complexity entirely

## 4. i18n Strategy

### 4.1. URL Structure

Following sgraph.ai's pattern exactly:

```
send.sgraph.ai/                          -> English (default)
send.sgraph.ai/upload/                   -> English upload
send.sgraph.ai/download/                 -> English download
send.sgraph.ai/pt-pt/                    -> Portuguese landing
send.sgraph.ai/pt-pt/upload/             -> Portuguese upload
send.sgraph.ai/pt-pt/download/           -> Portuguese download
send.sgraph.ai/es-es/                    -> Spanish landing
... (all 15 locales)
```

### 4.2. Pre-rendering Approach (Same as sgraph.ai)

1. **Build time:** `generate_i18n_pages.py` (extended for Send UI) reads English HTML + locale JSONs
2. **Output:** Pre-rendered locale folders (`/pt-pt/`, `/es-es/`, etc.) with translated text baked in
3. **No runtime JSON fetch** for page-level text
4. **Shared assets:** `_common/` directory NOT duplicated per locale -- locale pages use relative paths
5. **hreflang tags** injected for SEO cross-locale linking

### 4.3. Hybrid i18n for Web Components

The Send UI uses Web Components with Shadow DOM. Pre-rendering can't reach into shadow DOM. Solution:

- **Page-level text:** Pre-rendered via `data-i18n` attributes (like sgraph.ai)
- **Component text:** Runtime `I18n.t()` calls with translations embedded in a `<script>` tag during pre-rendering:
  ```html
  <script>I18n.strings['pt-pt'] = { ... }; I18n.locale = 'pt-pt';</script>
  ```
- **No runtime JSON fetch** -- translations are always embedded at build time

### 4.4. Translation Scope

The Send UI has ~130 existing i18n keys (upload, download, access gate, transparency, errors, CTA). Expanding to v2.0 adds ~30 more keys (preview panel, file type badges, new UI elements). Total: ~160 keys per locale.

Compare: sgraph.ai has ~500+ keys. Send UI is simpler.

## 5. v2.0 IFD Consolidation

### 5.1. Scope: Symmetric Key Only

**INCLUDE in v2.0:**
- Upload page (file selection, AES-256-GCM encryption, upload)
- Download page (key input, decryption, file preview, download)
- Transfer history (browser-local storage)
- Access gate (token validation)
- Transparency panel (transfer metadata)
- API client (transfers + presigned endpoints)
- Crypto module (Web Crypto API, AES-256-GCM)
- i18n (15 locales)
- File type detection + markdown parser (preview)

**EXCLUDE from v2.0 (move to another domain):**
- Vault (vault.html, vault-api.js, vault-crypto.js, user-vault component)
- Data Rooms (join.html, room.html, join-api.js, room-crypto.js, room-join/room-view components)
- SSH keygen tool (tools/ssh-keygen/)
- Card design assets

### 5.2. IFD Consolidation

The v2.0 consolidation collapses 9 IFD versions (v0.1.0 through v0.1.8) into a single clean version. This eliminates the prototype monkey-patching override chain.

**Current (v0.1.x):** Each version overrides specific methods via:
```javascript
_orig = SendUpload.prototype.method;
SendUpload.prototype.method = function() { /* new behavior */ _orig.call(this); };
```
HTML loads scripts in chain: v0.1.0 -> v0.1.4 -> v0.1.5 -> v0.1.6 -> v0.1.8

**v2.0:** Single coherent class definition per component. No override chain.

### 5.4. Version Decision

The IFD path is `v2/v2.0/v2.0.0/`. This is a major version because:
- Collapses 9 IFD versions into one clean codebase
- Changes serving architecture (Lambda -> S3+CloudFront)
- Removes vault/PKI/data room features
- Adds full i18n support
- Removes the `/send` path prefix

This is not an incremental v0.1.x override -- it's a new foundation.

### 6.2. Large File Upload/Download Path

Presigned URLs already bypass Lambda entirely -- the client uploads/downloads directly to/from S3. This means large file transfers are NOT affected by the CloudFront migration. Only metadata API calls go through CloudFront -> Lambda.

## 7. Architecture Decision Records

| ADR | Decision | Rationale |
|-----|----------|-----------|
| ADR-1 | Remove `/send` path prefix | Domain IS `send.sgraph.ai` -- prefix is redundant. 301 redirects for backward compat. |
| ADR-2 | Pre-render i18n at build time | Same proven pipeline as sgraph.ai. Zero runtime cost. SEO benefits. |
| ADR-3 | Same-origin API via CloudFront | Eliminates CORS. Simplifies client code. Single SSL cert. |
| ADR-4 | IFD version v2.0.0 | Major architectural shift warrants major version. |
| ADR-5 | Symmetric key features only in v2.0 | Clean separation. Vault/PKI/rooms move to another domain. |
| ADR-6 | Explicit API behaviors (not `/api/*`) | Avoids breaking existing URL scheme. Migrate later. |
| ADR-7 | Same S3 bucket as sgraph.ai | Reuse infrastructure. Different prefix (`websites/send-sgraph-ai/`). |

## 8. Migration Phases

### Phase 1 -- Explorer: Build v2.0 UI
- Consolidate IFD override chain into single components
- Add `data-i18n` attributes to all text
- Create translation JSONs (start with en-gb + pt-pt + es-es)
- Remove `/send` prefix from internal links
- Create page-per-route structure (upload/index.html, download/index.html, etc.)

### Phase 2 -- DevOps: S3 Deployment Pipeline
- Extend `deploy_static_site.py` with `--site send-sgraph-ai`
- Create `generate_send_i18n_pages.py` (or extend existing)
- Create GitHub Actions workflow for Send UI deployment

### Phase 3 -- DevOps: CloudFront Distribution
- Create/update CloudFront distribution with S3 + Lambda origins
- Configure behaviors (API paths -> Lambda, default -> S3)
- Deploy CloudFront Function (URL rewriting + legacy redirects + download ID handling)

### Phase 4 -- Dev: Lambda Cleanup
- Remove StaticFiles mount and HTML page routes
- Remove UI assets from Lambda deployment package
- Add `X-CloudFront-Secret` validation middleware
- Tighten CORS to `["https://send.sgraph.ai"]`

### Phase 5 -- DevOps: DNS Cutover
- Point send.sgraph.ai to CloudFront distribution
- Monitor for 404s or routing issues
- Validate all 15 locales serve correctly

### Phase 6 -- QA: Smoke Tests
- All static pages load from S3 (verify CloudFront X-Cache headers)
- All API calls route to Lambda (verify no S3 headers on API responses)
- All 15 locale pages render correctly
- Encryption/decryption flow works end-to-end
- Legacy `/send/*` paths redirect correctly
- Download with transfer ID resolves correctly
