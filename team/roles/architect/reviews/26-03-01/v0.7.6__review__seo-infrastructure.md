## Summary

This review covers the SEO infrastructure created for sgraph.ai to enable Google Search Console submission: a generated sitemap.xml with full hreflang support, a robots.txt with targeted disallow rules, and a Python generator script for ongoing maintenance. The site serves 18 indexable pages across 17 locales, producing 306 sitemap URLs.

## 1. URL Structure Analysis

### Current Architecture

The site uses a locale-prefixed URL scheme:

```
https://sgraph.ai/{locale}/{page-path}/
```

All 20 pages live under `en-gb/` as the source locale. The `generate_i18n_pages.py` script replicates them into 16 additional locale directories, producing a flat locale structure:

| Locale type | Count | Example |
|---|---|---|
| Source (en-gb) | 1 | `https://sgraph.ai/en-gb/pricing/` |
| Regional variants | 16 | `https://sgraph.ai/pt-br/pricing/` |

### Page Categories

| Category | Pages | Priority | Changefreq | Notes |
|---|---|---|---|---|
| Home | 1 | 1.0 | weekly | Primary landing page |
| Product / Early Access | 2 | 0.8-0.9 | weekly | Core conversion pages |
| Agents hub | 1 | 0.8 | weekly | Agent category page |
| Agent detail | 3 | 0.7 | monthly | Sherpa, Ambassador, Architect |
| Architecture | 1 | 0.7 | monthly | Technical overview |
| Contact | 1 | 0.6 | monthly | Contact form |
| Pricing hub | 1 | 0.8 | weekly | Pricing overview |
| Pricing detail | 7 | 0.5-0.6 | monthly | Individual plan pages |
| Payment callbacks | 2 | -- | -- | Excluded from sitemap |

**Total indexable:** 18 pages x 17 locales = **306 URLs**
**Excluded:** payment/success/ and payment/cancel/ (post-checkout redirects with no indexable content)

### URL Design Strengths

1. **Trailing slashes are consistent.** Every page URL ends with `/`, served by `index.html` files. This avoids duplicate-content issues from `/pricing` vs `/pricing/`.
2. **Locale is always the first path segment.** Clean, predictable structure for both crawlers and CDN routing rules.
3. **No query parameters for locale selection.** Each locale has its own directory, making it fully crawlable without JavaScript.
4. **Directory-based pages.** Every page is `{path}/index.html`, which means the canonical URL is always the directory path with trailing slash.

### URL Design Considerations

1. **No root-level redirect declared.** Visiting `https://sgraph.ai/` should redirect to `/en-gb/` (or show a locale selector). The sitemap assumes en-gb is the x-default destination. The CloudFront or server configuration should handle the root redirect.
2. **`tlh` (Klingon) locale.** This is a non-standard locale code. Search engines will not recognise it as a valid BCP 47 tag. It will be indexed but hreflang matching will be ignored for this locale. This is acceptable as a novelty/Easter egg locale.

## 2. Hreflang Strategy

### Design Decisions

**Bidirectional completeness.** Every URL entry in the sitemap includes `<xhtml:link>` alternates for all 17 locales plus `x-default`. This is the Google-recommended approach: each page must declare the full set of alternates, and those alternates must reciprocate (point back). Since the sitemap is generated, bidirectional consistency is guaranteed by construction.

**x-default points to en-gb.** When a search engine cannot match the user's locale to any declared hreflang, it falls back to x-default. We designate `en-gb` as the default because:
- It is the source locale (content is authored in British English)
- It is the most broadly understood variant for international users
- It avoids needing a separate locale-neutral landing page

**Hreflang values use lowercase.** Although BCP 47 is case-insensitive, Google's documentation and examples use lowercase (`en-gb` not `en-GB`). The sitemap follows this convention for consistency. Note: the existing `generate_i18n_pages.py` uses mixed case (`en-GB`) in HTML `<link>` tags — both are valid, but aligning them to lowercase in a future pass would improve consistency.

### Hreflang Implementation: Sitemap vs HTML `<link>` Tags

The hreflang declarations exist in **two places**:

1. **In the sitemap** (this implementation) — one authoritative source covering all 306 URLs
2. **In HTML `<head>`** (via `generate_i18n_pages.py`) — already injected by the i18n generator

Having both is not a problem. Google merges signals from sitemaps and HTML. The sitemap approach is preferred for large multi-locale sites because:
- A single file declares the complete mapping, reducing the chance of inconsistencies
- No per-page HTML bloat (each page's `<head>` already has 18 `<link>` tags; the sitemap is the canonical source if they ever diverge)
- Easier to audit and maintain via the generator script

## 3. Sitemap Design Decisions

### Generator Script Architecture

The sitemap is produced by `scripts/generate_sitemap.py`, not hand-maintained. This is critical because:

- **306 URLs with 18 hreflang alternates each = 5,508 hreflang links.** Manual maintenance is error-prone.
- **Adding a page or locale** requires changing one list in the script, then re-running. No manual XML editing.
- **The script is the single source of truth** for page inventory, priority, and changefreq. It can be integrated into CI to regenerate on deploy.

### What the Script Does

1. Iterates over 18 pages x 17 locales = 306 URL entries
2. For each entry, emits `<loc>`, `<lastmod>`, `<changefreq>`, `<priority>`
3. For each entry, emits 17 `<xhtml:link rel="alternate">` tags (one per locale) plus 1 x-default
4. Produces well-formed XML with proper namespace declarations

### Sitemap Size

- **306 URL entries**, 7,348 lines, ~571 KB
- Well within Google's limits (50,000 URLs / 50 MB uncompressed)
- If the site grows significantly, a sitemap index can be added later

### Priority and Changefreq Rationale

| Priority | Assigned to | Reasoning |
|---|---|---|
| 1.0 | Home page | Primary entry point, highest crawl importance |
| 0.9 | Product page | Core value proposition |
| 0.8 | Agents hub, Pricing hub, Early Access | High-value navigation/conversion pages |
| 0.7 | Agent detail, Architecture | Supporting content pages |
| 0.6 | Pricing detail, Contact | Detail/utility pages |
| 0.5 | Partners, AWS Marketplace | Niche audience pages |

Changefreq is `weekly` for hub pages (home, product, agents, pricing, early-access, architecture) and `monthly` for detail pages. These are advisory signals; crawlers may adjust based on actual change detection.

### Excluded Pages

`payment/success/` and `payment/cancel/` are excluded because:
- They are post-checkout redirect targets, not navigable content
- They contain no useful information for search users
- They are also blocked in `robots.txt` via `Disallow: /payment/`
- Including them would waste crawl budget

## 4. Canonical URL Recommendations

### Current State

The `generate_i18n_pages.py` script already injects `<link rel="canonical">` into each page's `<head>`, pointing to that page's own locale-specific URL. This is correct behaviour for a multi-locale site where each locale has genuinely different content (translated text).

### Recommendations

1. **Do NOT use cross-locale canonicals.** Each locale page should canonical to itself, not to en-gb. Google uses hreflang (not canonical) to understand locale relationships. Setting canonical to en-gb would tell Google the locale page is a duplicate, suppressing it from locale-specific search results.

2. **Ensure the root domain redirects.** `https://sgraph.ai/` should 301 redirect to `https://sgraph.ai/en-gb/`. Without this, the root URL could be indexed as a separate page with no hreflang, cannibalising the en-gb home page.

3. **HTTPS enforcement.** All canonical URLs and sitemap URLs use `https://`. Ensure HTTP requests 301 redirect to HTTPS (should be handled at CloudFront level).

4. **www vs non-www.** The site uses `sgraph.ai` (no www). Ensure `www.sgraph.ai` 301 redirects to `sgraph.ai` if that subdomain exists.

5. **Trailing slash consistency.** The sitemap uses trailing slashes (`/en-gb/pricing/`). The canonical tags in HTML should match. Verify that requests without trailing slashes redirect to the trailing-slash version (CloudFront function or origin configuration).

## 5. robots.txt Design

The `robots.txt` is minimal and intentional:

```
User-agent: *
Allow: /
Disallow: /payment/
Disallow: /_common/
Sitemap: https://sgraph.ai/sitemap.xml
```

- **`Allow: /`** — explicit allow-all baseline
- **`Disallow: /payment/`** — blocks both `/payment/success/` and `/payment/cancel/` across all locales (`/{locale}/payment/` paths are covered because the payment pages are also excluded from the sitemap and have no inbound links from crawlable pages)
- **`Disallow: /_common/`** — shared CSS/JS/font assets. Not pages, no reason to crawl or index.
- **`Sitemap:` directive** — tells crawlers where to find the sitemap without requiring Search Console submission (though SC submission is still recommended)

### Note on `/payment/` Disallow Scope

The `Disallow: /payment/` rule in robots.txt matches paths starting with `/payment/` at the root level. Locale-prefixed payment pages (`/en-gb/payment/success/`) are NOT blocked by this rule because robots.txt path matching is prefix-based from the root. However, this is acceptable because:
- Payment pages are excluded from the sitemap (no discovery path)
- Payment pages have `noindex` meta tags or no inbound links from crawlable pages
- If explicit blocking of locale-prefixed payment pages is desired, add rules like `Disallow: /*/payment/` (though this is non-standard and not reliably supported)

The more robust protection is the sitemap exclusion and lack of internal links to payment callback pages.
