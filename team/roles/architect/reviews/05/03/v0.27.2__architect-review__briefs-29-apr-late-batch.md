## Assessment: Ephemeral Infra Next Phase (Doc 334)

### What Is Proposed

Seven features that extend the ephemeral infrastructure from "working MVP" to "simulated AWS Marketplace." All build on the 28 Apr plugin architecture brief.

### Architectural Assessment

**Solid design decisions:**

- **Split creation from live instances** — correct separation of concerns. Provisioning and monitoring are distinct contexts; combining them in one view was always a temporary compromise.
- **Three creation modes** — (fresh / bake AMI / from AMI) is the right model. The 80s vs 2min speed difference justifies the AMI bake step. This is a measurable, user-relevant benefit.
- **SG/Send vault server as instance type** — architecturally completing the platform. The platform triangle is now: compute (EC2), browser (Playwright/Firefox), storage (vault server). Each is a pluggable instance type.
- **Option A for remote shell** (API-based via instance FastAPI) — correct MVP choice. Avoids IAM complexity (SSM) and network infrastructure (WebSocket SSH bridge). `POST /shell/execute` is safe enough for the use case. Document the limitation (no interactive terminal) in the acceptance criteria.
- **Prometheus endpoints** — correct approach. Exposing `/metrics` at the instance level lets the observability pipeline (Elastic/Grafana from 22 Apr brief) scrape without a Prometheus push gateway.

**Concern: Stacks scope creep risk**

The brief describes stacks as "simplified CloudFormation — just a JSON config that triggers multiple instance launches." This is the right minimal design. However, the acceptance criteria include "Stack instances can communicate (same VPC/security group)" — which requires VPC and security group management.

VPC management is not trivially simple. It introduces:
- Multi-instance networking configuration
- Security group coordination between instances
- Potential for cross-instance failure states (what happens if Instance B fails after Instance A starts?)

**Recommendation:** Implement stacks in two phases:
- **Phase 1 (MVP):** JSON config triggers sequential instance launches. No networking coordination. Each instance in its default security group. "Stack" = launch shortcut only.
- **Phase 2 (Post-validation):** Same-VPC placement and security group coordination once the instance launch path is stable.

Phase 2 is where the "simplified CloudFormation" framing becomes meaningful. Phase 1 is safe to implement immediately.

**Open question: Docker container management inside instances**

The brief describes managing Docker containers _inside_ EC2 instances via the instance's FastAPI. This requires the FastAPI to have Docker daemon access (via Docker socket mount or Docker-in-Docker). This is an implementation concern rather than an architectural concern, but it should be confirmed before Dev starts.

## Assessment: Firefox Browser Plugin (Doc 335)

### What Is Proposed

Firefox + MITM proxy as the first concrete implementation of the plugin architecture. Demonstrates the plugin model with a real-world use case.

### Architectural Assessment

**Correct plugin application:**

The brief correctly identifies Firefox as a distinct plugin from Playwright: interactive vs programmatic, stateful vs stateless. This validates the plugin architecture's ability to handle heterogeneous instance types. The acceptance criteria (plugin disable hides Firefox from creation UI) confirms the feature-flag model works end-to-end.

**Iframe vs VNC — needs a decision before implementation**

The brief states: "Given that this is a Docker container we control, we should be able to configure it to allow iframe embedding (remove X-Frame-Options headers)."

This is correct as a design principle, but the implementation must verify it before committing to the iframe path. The specific container image needs to be checked for:
- X-Frame-Options headers
- CSP frame-ancestors directives
- Whether the Firefox session supports embedding (some containers specifically disable this)

**Recommendation:** Add an explicit acceptance criterion: "Verify Firefox container allows iframe embedding before starting UI implementation. If not, document which fallback (Neko/noVNC) will be used." Avoid implementing both paths speculatively.

**Vault storage model is correct:**

Storing MITM intercept scripts, credentials, and session metadata in the vault per-session follows the correct security model. Credentials never leave the vault. Scripts are versioned. Session metadata is auditable.

## Assessment: SGit Four-Layer Refactoring (Doc 336)

### What Is Proposed

Comprehensive structural refactoring of the SGit codebase into four clean layers before PKI implementation begins.

### Architectural Assessment — STRONGLY ENDORSED

This is an excellent architectural decision. The timing argument is compelling: "SGit does not yet have wide adoption. Design changes that would be painful later are still cheap now."

**The layering model is correct:**

```
Plugins (Layer 4) — read-only, feature-flaggable
    |
    +-> Core (Layer 2)    — state-changing typed steps, depends on Crypto
    +-> Network (Layer 3) — API client, depends on Crypto
    |
Crypto (Layer 1) — pure functions, no dependencies
```

No upward dependencies is the right invariant. The risk of accidentally introducing one is real — the enforcement mechanism (folder structure + import audit) is lightweight but sufficient for a codebase of this size.

**Transaction logging is the right foundation:**

The transaction record design (`action`, `timestamp`, `vault_id`, `branch`, `duration_ms`, etc.) is minimal and correct. It captures what happened without trying to capture everything. The observation that "this is not the vault's commit history — it is a record of SGit's operations including failed operations" is architecturally important. These are distinct audit artifacts with distinct retention and analysis needs.

**Step-based decomposition is the right performance model:**

Making performance visible before making performance fixes is architecturally disciplined. The example (`sgit push` decomposed into 6 typed steps with per-step timing) shows that "Upload objects" will almost certainly dominate — and the fix (parallelise the upload) can be scoped precisely once the data confirms it.

**PKI readiness is the right framing:**

The brief correctly identifies which layers PKI will touch: Crypto (signing/verification), Core (sign step in commit, verify step in pull), Network (key exchange endpoints). This is a surgical, scoped analysis. If the layers are clean before PKI starts, each PKI addition has a clear home.

### Two Open Decisions — Need Resolution Before Dev Starts

**Decision #29: SGit Crypto layer — when does it move to its own repo?**

The brief defers this to "after stabilisation." However, the implementation team needs to know whether to build the Crypto layer with `pip install sgit-ai-crypto` separation in mind from the start, or to treat it as an internal module for now with migration later.

**Recommendation:** Build as internal module first. The interface will change as PKI lands. Premature extraction creates a dependency management burden without current benefit. Schedule repo extraction as a follow-up task after PKI ships.

**Decision #30: Transaction log storage — alongside vault data or separate?**

Options:
- **Alongside vault data** — simpler, single store, natural co-location with vault state
- **Separate log store** — better for aggregation, monitoring pipelines, and retention policies

**Recommendation:** Alongside vault data for MVP. Transaction logs are per-vault artifacts. Aggregation can be done by the observability layer reading from vault storage. Separate store introduces two-store consistency requirements.

### One Concern: Scope of the Refactoring

The four-layer refactoring touches every file in the SGit codebase. The risk is that it becomes a multi-week project that stalls while PKI waits. The brief acknowledges this risk implicitly but doesn't address it.

**Recommendation:** Structure the refactoring in phases:
1. **Phase 1:** Create the four-layer folder structure. Move existing code to the right layer without changing it. Ensure all tests pass. (2–3 days)
2. **Phase 2:** Add typed step decomposition to the three most-used commands (push, pull, commit). (3–4 days)
3. **Phase 3:** Add transaction logging to Core actions. (2 days)
4. **Phase 4:** Add feature flags to plugins. (1 day)

PKI can start after Phase 1 clears the folder structure. Phases 2–4 can run in parallel with early PKI work.
