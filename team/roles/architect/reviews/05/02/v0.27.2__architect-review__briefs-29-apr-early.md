## 1. Ephemeral Infrastructure Next Phase — Architectural Assessment

### 1.1 UI Separation: Creation vs. Live Instances

The proposed split is architecturally correct. Creation and operational views have fundamentally different data sources, update frequencies, and user intents. Keeping them separate aligns with good component design:

- **Creation panel** — reads static catalogue data (available instance types, AMI list); low update frequency; mostly configuration inputs
- **Live instances panel** — reads real-time state (instance status, uptime, cost); high update frequency; primarily operational controls

**Assessment:** Sound. The fractal frontend (sg-layout) from Day 48 accommodates this split naturally — each panel is its own layout context.

### 1.2 Three AMI Creation Modes

The three-mode model (Fresh → Bake → From-AMI) is a well-understood pattern from production EC2 workflows. The timing claim (80s vs 2 min for Elastic) should be treated as a benchmark direction, not a hard SLA, until measured consistently in the target AWS region.

**Architectural concern:** The AMI library ("simulated AWS Marketplace") introduces a catalogue management problem that grows over time. The brief covers CRUD for AMIs per instance type but does not address:
- AMI ownership / who can bake / who can delete
- AMI retention policies (how many AMIs per type before auto-cleanup)
- Cross-region AMI availability (AMIs are region-scoped in AWS)

These are not blockers, but the Architect should define a `AMI_Catalogue` data model before implementation begins.

### 1.3 SG/Send Vault Server as Instance Type (Storage Primitive)

**This is the architecturally most significant proposal.** Completing the platform triangle (Compute + Browser + Storage) enables qualitatively new workflows. Agents and instances can read/write vaults on ephemeral storage instead of production.

**Design questions the brief leaves open:**

| Question | Impact |
|----------|--------|
| Network topology: how do peer instances discover the vault server IP? | High — needed for Feature 7 (Stacks) |
| Vault server auth: does each session get a new vault key or share one? | High — security boundary |
| Pre-loading vault snapshots: which snapshot format? sgit clone? share token? | Medium |
| Vault server persistence: does data survive instance stop/start? | Medium |

**Recommendation:** The vault server instance type should expose a `VaultServer__Config` type (following Type_Safe conventions) that captures: vault_key (ephemeral, generated at launch), network address, port, auth mode. This config is passed to peer instances at launch time.

### 1.4 Docker Container Management Inside Instances

The proposed UI (list, start, stop, logs, expose ports) is reasonable, but the architectural question is how the instance's FastAPI discovers and manages containers:

- Docker socket (`/var/run/docker.sock`) must be accessible to the FastAPI process
- Security implication: Docker socket access = root equivalent inside the container
- For multi-container Playwright: port allocation strategy needed (sequential? random? registered?)

**Recommendation:** Define a `Container__Registry` model per instance that maps container name → port → status. The FastAPI endpoints read/write this registry. This avoids dynamic port scanning.

### 1.5 Remote Shell — Option A Correctly Chosen

The brief correctly identifies API-based shell as the MVP path. The architectural considerations:

```
POST /shell/execute  — stateless, fire-and-forget commands
POST /shell/stream   — WebSocket for long-running commands
```

**Concern:** WebSocket streaming via Lambda has a hard 29-second timeout on API Gateway (not applicable here since we use Lambda URLs, but Lambda URLs have a 15-minute max). For `tail -f` style streaming, the implementation needs explicit heartbeats and reconnect logic in the browser.

**SSM (Option B) assessment:** SSM StartSession from Lambda is possible but requires additional IAM permissions and SSM Agent on the instance. Worth exploring post-MVP but not blocking.

### 1.6 Prometheus Endpoints

Standard pattern. The brief correctly scopes this to the instance FastAPI (`GET /metrics`) rather than requiring a Prometheus server. The metrics endpoint enables two modes:

1. **Pull mode:** External Prometheus/Grafana scrapes the endpoint (production observability)
2. **Poll mode:** The UI polls the endpoint directly and displays inline (MVP)

**Assessment:** Start with the format (`text/plain; version=0.0.4` Prometheus exposition format). Do not hard-code JSON. Tools like Grafana and Elastic will expect the standard format.

### 1.7 Stacks — Multi-Instance Bundles

**Highest complexity item in this brief.** The JSON-defined stack is a simplification of CloudFormation but introduces real dependency challenges:

```json
{
  "instances": [...],
  "networking": {
    "all_instances_same_vpc": true,
    "vault_server_accessible_from_all": true
  }
}
```

The "all instances in same VPC/security group" networking shortcut works for the MVP but the stack launcher needs to:
1. Launch instances in dependency order (vault server must be running before Playwright can connect to it)
2. Pass connection details between instances at launch time
3. Handle partial failures (2 of 3 instances launched, third fails)

**Architectural pattern recommendation:** The stack launcher should follow a two-phase approach:
- Phase 1: Launch all instances, wait for each to reach "running" state
- Phase 2: For each instance, inject `peer_connections` config (vault server address, API endpoints of other instances)

This avoids orchestration complexity while enabling inter-instance connectivity.

## 2. Firefox Browser Plugin — Architectural Assessment

### 2.1 Plugin Isolation

The brief proposes `plugins/firefox/` following the Day 48 plugin architecture. The Firefox plugin is correctly identified as a distinct product from Playwright: interactive vs. programmatic, stateful vs. stateless.

**The plugin boundary must enforce:**
- Firefox plugin code does not reach into Playwright plugin code (and vice versa)
- Shared concerns (MITM proxy config, vault session storage) use shared library code, not shared plugin code

### 2.2 Vault Storage Schema for Session Data

The brief lists: credentials, MITM scripts, proxy logs, Firefox profile, screenshots, instance metadata. This implies a defined vault directory structure per Firefox session:

```
sessions/<session-id>/
    config.json          — instance metadata, creation mode, AMI used
    credentials.enc      — encrypted username/password
    mitm/
        script.py        — active intercept script
        logs/            — proxy capture logs
    profile/             — Firefox profile (if saved)
    screenshots/         — PNG files
```

**Assessment:** The structure is reasonable. The Architect should specify this schema formally before implementation to avoid per-agent improvisation.

### 2.3 Inline Browser View

The brief proposes iframe embedding as the primary approach (X-Frame-Options removal from the Firefox container config). This is achievable for a Docker container we control.

**Fallback (Neko/noVNC):** If iframe embedding proves unreliable (e.g., Firefox refuses embedding via CSP despite configuration), the remote browser component from the VNC brief (April 21) is the correct fallback. This was documented as PROPOSED in Decision 26.

### 2.4 MITM Script Upload Security

The brief mentions uploading a Python script that mitmproxy executes. This is a code execution primitive:

**Security concern:** The UI allows users to upload arbitrary Python code that runs inside the instance. This is intentional (the use case is security testing / traffic modification), but:
- Script execution must be sandboxed to the instance (no access to host network except through the proxy)
- Script upload should require admin authentication
- Scripts should be version-controlled in the vault (audit trail)

The AppSec role should review this feature before implementation.

## 3. SGit Four-Layer Refactoring — Architectural Assessment

### 3.1 Layer Design Soundness

The four-layer architecture (Crypto → Core → Network → Plugins) is correctly designed. The no-upward-dependencies rule is the key invariant that makes this refactoring meaningful.

**The dependency graph is clean:**
```
Plugins → Core, Network
Core → Crypto, Network
Network → Crypto
Crypto → nothing
```

This enables:
- Crypto to be audited and stabilised independently
- Plugins to iterate without risk of breaking state
- Network to be mocked for offline testing (Core tests)

### 3.2 Transaction Logging Design

The transaction log JSON schema in the brief is well-specified:

```json
{
  "action": "commit",
  "timestamp": "ISO-8601",
  "vault_id": "...",
  "branch": "...",
  "parent_commit": "...",
  "new_commit": "...",
  "files_added": [...],
  "files_modified": [...],
  "files_deleted": [...],
  "tree_hash": "...",
  "duration_ms": 45
}
```

**Open question (Decision 30):** Where is this log stored? Options:
- **Alongside vault data** (e.g., `vault/.sgit/logs/transactions.jsonl`) — vault-native, but pollutes the vault namespace
- **Separate log file** (`~/.sgit/logs/transactions.jsonl`) — local only, not shared
- **Remote log endpoint** — requires server-side changes to this repo's Lambda

**Recommendation:** Start with a local append-only log file (`~/.sgit/logs/transactions.jsonl`). This is the lowest-risk implementation and can be migrated to vault-storage later when Decision 30 is resolved.

### 3.3 Step Decomposition Timing

The step-based decomposition with `duration_ms` per step is a key enabler for performance work. The brief correctly frames this as "make performance visible" rather than "fix performance" — consistent with Wardley methodology (visibility before optimisation).

**Implementation note:** Each step should capture `start_time` at entry and compute `duration_ms` at exit. Use `time.monotonic()` not `time.time()` for wall-clock precision.

### 3.4 PKI Readiness Alignment

The brief explicitly frames the Crypto layer as the PKI insertion point:
- New operations: `sign(data, private_key)`, `verify(data, signature, public_key)`, `key_pair_generate()`
- Core commit action gains a `sign` step
- Core pull action gains a `verify` step
- Network layer gains key exchange endpoints

This alignment is correct. The four-layer separation is the necessary precondition for PKI. **The Villager team should complete this refactoring before any PKI Explorer work begins.**

### 3.5 Separate Repo Timing (Decision 29)

The brief raises the question of extracting Crypto into its own repo post-stabilisation. This is architecturally sound (crypto primitives as a standalone library) but:
- Not needed until the Crypto layer has stabilised through production use
- Separation adds distribution complexity (package versions, dependency pinning)
- Recommend: defer until after PKI is implemented and the API surface is known
