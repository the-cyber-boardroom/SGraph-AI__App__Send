## Executive Summary

This batch contains **four architectural briefs plus one implementation brief** with real architectural decisions. The April 30 briefs collectively define a mature architecture for the **Ephemeral Compute** platform — a companion product to SGraph Send. The May 5 brief (08b) is the only one targeting the SGraph Send codebase directly and carries significant architectural weight: it introduces a binary wire format, a caching subsystem, and new endpoints.

**Architectural quality:** The Ephemeral Compute architecture is coherent and well-reasoned. The SG/Send Clone Pack spec (08b) is exceptionally detailed and implementation-ready.

**Concerns:** Five open questions in brief 08b need architect decisions before any implementation begins. Two design choices have non-trivial implications for the existing storage model.

## Ephemeral Compute Architecture Assessment

### Overall Architecture (docs 337–346)

The Ephemeral Compute platform as specified across the April 30 batch forms a consistent layered architecture:

```
External Users
    ↓
Bastion (public subnet, per-stack VPC)
    ↓
Nodes (private subnet, per-stack VPC)
    ↓
Host FastAPI Control Plane (privileged)
    ↓
Container Runtime Abstraction
    ├── Docker adapter
    ├── Podman adapter
    └── Kubernetes adapter (future)
        ↓
Pods (containers, no host access)
    ├── Long-running (Playwright, Firefox, Elastic, Ollama)
    └── Ephemeral (one-shot commands, QA runners)
```

This is architecturally sound. The separation of concerns is clean:
- VPC provides network isolation (doc 340)
- Control plane provides the only privileged access (doc 343)
- Spec system provides the extension point (doc 339, 342)
- Runtime abstraction hides the container engine (doc 343)

**Assessment:** Approved as architectural direction. No fundamental conflicts with existing SGraph Send design patterns.

### Naming Taxonomy (doc 341)

The Node/Pod/Spec/Stack taxonomy is correct. The old "stack" meaning single instance was confusing; reserving "stack" for multi-node is the right call. The alignment with Kubernetes terminology (Pods) is forward-looking without prematurely committing to Kubernetes.

**One concern:** The taxonomy requires renaming existing code, CLI commands, and API endpoints. This is a breaking change for any existing consumers. Approach with a migration plan: deprecation period with both old and new names, then remove old.

### S3-Compatible API (doc 338)

**Full boto3 compatibility is the right bar.** "Mostly compatible" creates a maintenance nightmare — you never know which edge case will bite in production. The call-log-first methodology (proxy everything, discover what's actually called, implement those) is sound engineering.

**Architectural concerns:**

1. **SigV4 handling is non-trivial.** SigV4 includes a canonical request hash over the full request body. This means the proxy must buffer the full request body before forwarding. For large file uploads, this creates memory pressure. Consider chunked/streaming SigV4 verification for large bodies.

2. **Four operating modes add combinatorial complexity.** Each mode (Local, Proxy, Hybrid, Selective) × two credential modes (Platform, Client) = 8 configurations. Some combinations don't make sense (e.g., Full Proxy + Client Credentials is the same as a transparent proxy, which has no value). Document clearly which combinations are supported.

3. **IAM interception is necessary but risky.** Returning synthetic STS responses will work for simple boto3 usage but may fail for workloads that validate their IAM identity beyond the response structure. Implement the minimum needed and log unknown IAM calls.

**Assessment:** Architecture approved for Phase 1 (call log + full proxy). Caution on implementation complexity of SigV4 + large bodies.

### Spec Internal Structure (doc 339)

The two-layer API (core control plane + spec-specific) is the right pattern. Inheritence from `SpecRoutes` and `SpecCLI` base classes ensures consistency. The `spec.json` as a declarative source of truth for dynamic route registration is elegant.

**Concern:** Dynamic route registration at startup (reading spec.json and registering FastAPI routes) requires care. FastAPI's router is not designed for post-startup route addition. The safe approach: load all spec routes at startup from discovered spec.json files. If a spec is added at runtime, require a restart. Document this constraint clearly.

**Assessment:** Approved. One constraint to document.

### VPC Isolation Per Stack (doc 340)

**Architecturally correct** — VPC-per-stack is the cleanest isolation model. The lifecycle automation (create on stack start, destroy on stack stop) is correctly described.

**Cost concern flagged in the brief:** NAT Gateway at $0.045/hour + data transfer is significant for short-lived stacks. For stacks that don't need internet from private nodes, omit the NAT Gateway. This should be the default (opt-in to internet access, not opt-out).

**Operational concern:** The brief specifies no orphaned resources after destroy. This is hard to guarantee with complex VPC teardown sequences. Recommend: tag every VPC resource with `stack-id` at creation. Implement a reconciliation job that scans for resources with stack-ids that no longer exist.

### External Spec Repos (doc 342)

The `sg-compute_spec_{name}` naming convention is discoverable. Shallow clone for build-run mode is pragmatic.

**Security concern:** Cloning and executing an external repo's `setup.sh` is code execution from a third-party source. This requires:
- Spec signing (or at minimum, repo owner verification)
- Sandboxed execution of setup.sh (the host's control plane shouldn't execute setup scripts with full host privilege)
- Version pinning in stack definitions (not just "latest")

This security concern is **not blocking** for the MVP (first external spec is the Cyber Boardroom, which is our own project), but must be addressed before the spec ecosystem is opened to third parties.

## SGraph Send Architecture Assessment (doc 348: Brief 08b)

This is the only brief in this batch that directly impacts the SGraph Send codebase. It is **implementation-ready** with a detailed wire format, algorithm spec, and testing requirements. The architectural questions below must be resolved before implementation begins.

### Clone Pack: Architectural Fit

The Clone Pack design fits cleanly into the existing SGraph Send architecture:
- Zero-knowledge is preserved: the server assembles ciphertext, never decrypts
- Backward compatibility is maintained: `batch_read` stays unchanged
- The pack is derived from existing per-object storage — no vault format changes

**The fundamental trade-off:** The pack trades server-side CPU + disk at push time for dramatically faster clone time. This is the correct trade-off for a vault service where reads heavily outnumber writes.

### Open Architectural Questions (Must Resolve Before Implementation)

**OQ-08b-2 is the most critical:** Where do packs live in the storage model?

The brief proposes: `<vault storage>/<vault_id>/packs/`. But SGraph Send's storage abstraction (`Storage_FS`) hides the storage backend (memory, disk, S3). Packs should live behind the same abstraction for two reasons:
1. In production (S3 backend), packs should be S3 objects, not raw files — for durability and CDN-ability
2. In tests (memory/disk backend), packs should use the same abstraction for consistency

**Recommendation for OQ-08b-2:** Store packs as regular `Storage_FS` objects under a `packs/` prefix within each vault's storage namespace. This is consistent with how vault objects are stored today and requires no new storage paths.

**OQ-08b-1 (Async runtime):** The SGraph Send Lambda environment does not have a persistent worker queue. `asyncio.create_task` is the right choice for pre-warming — it runs in the same Lambda invocation's event loop after the push response is sent. However, Lambda may kill the process before the pre-warming task completes. Accept this: the pack will be built on-demand by the next clone request.

**OQ-08b-3 (Object-id field width):** Current SGit-AI object-ids are 24 chars (`obj-cas-imm-<12-hex>`). 64-byte max is comfortable. Confirm with SGit team that this won't change.

**OQ-08b-4 (Streaming):** FastAPI's `StreamingResponse` is the correct mechanism. The pack builder should write to a temp file and then stream from the file. Avoid building the pack in memory before streaming — packs can be hundreds of MB.

**OQ-08b-5 (Range pack):** Defer to v2. Implement only `full`, `head`, `bare-full`, `bare-head` for now. Return 404 for `range`.

### Wire Format Assessment

The `SGPK` binary format is well-specified. Notable properties:
- **Index after body:** Correct — allows streaming the body to disk before computing offsets
- **No compression:** Correct for ciphertext (AES-GCM output is already incompressible)
- **Body SHA-256 in footer:** Good integrity check. Covers only body, not header/index — acceptable for v1
- **Sorted index by obj_id:** Makes binary search possible for large packs

**One concern:** The `id_length` field (uint8, max 64) means object-ids longer than 64 bytes are invalid in this format. Verify with SGit team that no current or planned object-id encoding exceeds 64 bytes.

### Impact on Existing Architecture

| Component | Impact |
|-----------|--------|
| `Storage_FS` | Should serve packs — no change needed to the abstraction itself |
| User Lambda (send-api) | Two new routes. Same auth pattern as `batch_read`. |
| Admin Lambda | No change |
| Transfer model | No change |
| Vault model | `packs/` sub-namespace added under each vault's storage |
| CI/CD | No change to pipeline, new endpoints covered by existing test framework |
