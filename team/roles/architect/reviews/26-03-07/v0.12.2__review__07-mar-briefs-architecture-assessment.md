## 1. sg-layout Architecture Assessment

### 1.1 Fractal Composition Model — SOUND

The fractal model (sg-layout instances nesting inside sg-layout instances) is architecturally sound. The key design decisions that make it work:

- **Single responsibility:** Each sg-layout only knows its immediate parent. No grandparent knowledge. This is the right constraint — it prevents coupling across nesting depths.
- **Registration protocol isolation:** `stopPropagation` ensures that each component registers with its nearest sg-layout ancestor only. This is correct.
- **Serialisation recursion:** Inner layout state nests inside the parent tab's `componentState` field. This is clean JSON and deterministically restorable.

**One concern:** The brief says "arbitrarily deep nesting." In practice, nesting beyond 3-4 levels creates UX problems (panels become too small, resize handles pile up). The architecture should support arbitrary depth, but the implementation should include a **practical depth warning** at depth 4-5 — not a hard limit, just a console warning.

### 1.2 Synchronous Registration Protocol — ROBUST WITH CAVEATS

The `dispatchEvent` synchronous protocol is correct. `dispatchEvent` in the DOM spec is synchronous — the event handler executes inline, and the caller can read the result from a callback passed in `event.detail`. This is well-understood and reliable.

**Edge cases to address:**

1. **Late registration:** A component added to a panel after initial layout construction (e.g., dynamically appended via `appendChild`). The `connectedCallback` will fire and dispatch `sg-layout:register`, and the sg-layout listener will still be active — so this works. But the panel won't have been pre-allocated in the tree. **Recommendation:** sg-layout should handle dynamic panel addition gracefully — either auto-create a panel slot or queue the component for next layout update.

2. **Shadow DOM nesting depth:** `composed: true` events bubble through shadow boundaries. If there are many intermediate shadow DOM layers (non-sg-layout components with their own shadow roots), the event still reaches the nearest sg-layout. This is correct per spec. No issue.

3. **Race condition with `customElements.define`:** If sg-layout's script loads after hosted components' scripts, components' `connectedCallback` could fire before sg-layout's listener is set up. **Recommendation:** Document the load order requirement explicitly. Alternatively, sg-layout could retroactively handle already-connected components that missed the registration window.

4. **Infinite loop detection:** The `_registerDepth` counter is good. The `Set` of registered element references is an additional guard. Both are needed. The MAX_REGISTER_DEPTH of 10 is generous — legitimate nesting won't exceed 5. Consider 6 as the limit.

### 1.3 CSS Grid Viability at Scale — NEEDS EMPIRICAL TEST

CSS Grid for a 32x32 grid (1,024 panels) is the right choice to test first. The concern is:

- **DOM node count:** 1,024 panels x panel chrome ~ 5,000-6,000 DOM nodes. Modern browsers handle this without issue.
- **CSS Grid recalculation on resize:** Single `grid-template-columns`/`grid-template-rows` update — browser handles in one pass.
- **The real bottleneck:** Not CSS Grid itself, but the number of **active** components hosting real work (event listeners, DOM mutations).

**Recommendation:** Run the 32x32 test as planned. If CSS Grid handles it, stay with Grid. If performance degrades above ~256 panels, fallback to `position: absolute` is straightforward.

### 1.4 Architectural Risks / Dragons

1. **Tab drag-to-dock** requires coordinating across shadow boundaries — most complex interaction in the framework.
2. **Memory leaks on panel close:** sg-layout should forcibly unsubscribe a panel from sg-commands when closing it, as a safety net.
3. **Layout JSON versioning:** v0.2.0 must handle v0.1.0 JSON. Define migration strategy explicitly.

### 1.5 sg-commands and sg-panel-identity: Same Repo, Separate Modules

**Recommendation: Same repo, separate modules.** (`core/sg-layout/`, `core/sg-commands/`, `core/sg-panel-identity/`). Enables atomic commits, independent versioning, single CI pipeline.

## 2. Identity Infrastructure Assessment

### 2.1 sg-panel-identity to Broader Identity Framework — NATURAL EXTENSION

Layered identity model:
```
Layer 4: Human identity (Dinis's keys, published in DNS/web/.well-known)
Layer 3: Agent identity (Sherpa's keys, signed by Dinis's key)
Layer 2: Vault/transfer identity (write keys, read keys)
Layer 1: Panel identity (ECDH per-panel keys)
```

Each layer uses the same crypto primitives but at different scopes. The chain of trust flows downward.

### 2.2 Chain of Trust Model — SOUND

Recommends **Ed25519** for signing (not ECDH). Include validity period and "superseded by" field for key rotation.

### 2.3 PGP Key Servers — DON'T USE

Use `.well-known/sg-identity.json` + DNS TXT + Keyoxide instead.

### 2.4 DNS TXT — RECOMMENDED FORMAT

```
_sgraph.sgraph.ai  TXT  "v=sg1; k=ed25519; p=base64_key"
```

Follows SPF/DKIM/DMARC pattern. Set CORS headers on `.well-known/sg-identity.json`.

## 3. Individual File Storage Assessment

### 3.1 Tree + Individual Blob Model — SOUND

Mirrors git's content-addressable storage. Tree is the lightweight index; blobs are fetched on demand.

### 3.2 Storage_FS: Compound keys work

`store_content(f"{transfer_id}/tree", tree_data)` — S3 treats `/` as prefix delimiter. Works fine.

### 3.3 S3: 100 PUTs negligible cost

$0.0005 per upload. Use parallel uploads (`Promise.all` client-side).

### 3.4 Immutability — CORRECT

Need orphan blob cleanup strategy (periodic sweep or lazy deletion after retention period).
