## 1. sg-layout Architecture Assessment

### 1.1 Fractal Composition Model — SOUND

The fractal model (sg-layout instances nesting inside sg-layout instances) is architecturally sound. The key design decisions that make it work:

- **Single responsibility:** Each sg-layout only knows its immediate parent. No grandparent knowledge. This is the right constraint — it prevents coupling across nesting depths.
- **Registration protocol isolation:** `stopPropagation` ensures that each component registers with its nearest sg-layout ancestor only. This is correct.
- **Serialisation recursion:** Inner layout state nests inside the parent tab's `componentState` field. This is clean JSON and deterministically restorable.

**One concern:** The brief says "arbitrarily deep nesting." In practice, nesting beyond 3-4 levels creates UX problems (panels become too small, resize handles pile up). The architecture should support arbitrary depth, but the implementation should include a **practical depth warning** at depth 4-5 — not a hard limit, just a console warning.

### 1.2 Synchronous Registration Protocol — ROBUST WITH CAVEATS

The `dispatchEvent` synchronous protocol is correct. `dispatchEvent` in the DOM spec is synchronous — the event handler executes inline, and the caller can read the result from a callback passed in `event.detail`. This is well-understood and reliable.

**Edge cases to address:**

1. **Late registration:** A component added to a panel after initial layout construction (e.g., dynamically appended via `appendChild`). The `connectedCallback` will fire and dispatch `sg-layout:register`, and the sg-layout listener will still be active — so this works. But the panel won't have been pre-allocated in the tree. **Recommendation:** sg-layout should handle dynamic panel addition gracefully — either auto-create a panel slot or queue the component for next layout update.

2. **Shadow DOM nesting depth:** `composed: true` events bubble through shadow boundaries. If there are many intermediate shadow DOM layers (non-sg-layout components with their own shadow roots), the event still reaches the nearest sg-layout. This is correct per spec. No issue.

3. **Race condition with `customElements.define`:** If sg-layout's script loads after hosted components' scripts, components' `connectedCallback` could fire before sg-layout's listener is set up. The brief acknowledges this implicitly by saying "sg-layout connects first, builds tree, then components connect." **Recommendation:** Document the load order requirement explicitly. Components must be defined (script loaded) but not yet in DOM when sg-layout initialises its tree. Alternatively, sg-layout could retroactively handle already-connected components that missed the registration window.

4. **Infinite loop detection:** The `_registerDepth` counter is good. The `Set` of registered element references is an additional guard. Both are needed. The MAX_REGISTER_DEPTH of 10 is generous — legitimate nesting won't exceed 5. Consider 6 as the limit.

### 1.3 CSS Grid Viability at Scale — NEEDS EMPIRICAL TEST

CSS Grid for a 32×32 grid (1,024 panels) is the right choice to test first. CSS Grid handles rectangular layouts efficiently. The concern is:

- **DOM node count:** 1,024 panels × panel chrome (header, tab bar, resize handle, slot wrapper) ≈ 5,000-6,000 DOM nodes for the grid structure alone. Modern browsers handle this without issue.
- **CSS Grid recalculation on resize:** When a resize handle drags, the grid template must be updated. With 1,024 cells, this is a single `grid-template-columns` / `grid-template-rows` update — the browser's layout engine handles this in one pass.
- **The real bottleneck:** Not CSS Grid itself, but the number of **active** components. If each of the 1,024 panels hosts a real component doing real work (event listeners, DOM mutations), that's where performance degrades. The stress test should measure both: (a) 1,024 empty panels (framework overhead only), and (b) 1,024 panels hosting a simple component (e.g., a counter).

**Recommendation:** Run the 32×32 test as planned. If CSS Grid handles it, stay with Grid. If performance degrades above ~256 panels, the fallback to `position: absolute` with calculated positions is straightforward and well-understood.

### 1.4 Architectural Risks / Dragons

1. **Tab rendering:** The brief describes stacks with tabs. Tab rendering inside Shadow DOM is straightforward, but **drag-to-dock** (dragging a tab from one stack to another) requires coordinating across shadow boundaries. The implementation plan needs to detail how cross-shadow-boundary tab drag works. The `composed: true` event pattern should work here, but it's the most complex interaction in the framework.

2. **Memory leaks on panel close:** When a tab is closed and the hosted component is removed from DOM, `disconnectedCallback` fires. The component must clean up its event listeners, especially any subscriptions to `sg-commands`. If it doesn't, we leak listeners. **Recommendation:** sg-layout should forcibly unsubscribe a panel from sg-commands when closing it, as a safety net.

3. **Layout JSON versioning:** The serialised layout JSON needs a schema version field (already present as `sgLayoutVersion`). Good. But what happens when a layout JSON from v0.1.0 is loaded into v0.2.0? **Recommendation:** Define the migration strategy explicitly. v0.2.0 must handle v0.1.0 JSON.

### 1.5 sg-commands and sg-panel-identity: Separate Modules, Same Repo

**Recommendation: Same repo, separate modules.**

These are companion modules that share no code but are frequently co-deployed. Putting them in the same repo (`core/sg-layout/`, `core/sg-commands/`, `core/sg-panel-identity/`) with independent version files enables:
- Atomic commits when they need coordinated changes
- Independent versioning (sg-layout v0.1.3 doesn't force sg-commands to v0.1.3)
- Single CI pipeline with per-module test isolation

Separate repos would add overhead (cross-repo PRs, version coordination) with no benefit at current scale.

## 2. Identity Infrastructure Assessment

### 2.1 sg-panel-identity to Broader Identity Framework — NATURAL EXTENSION

The existing architecture already has:
- **PKI key registry** (EXISTS per reality doc) — server-side keypair management
- **sg-panel-identity** (PROPOSED in sg-layout brief) — client-side per-panel ECDH keys
- **Vault key architecture** (EXISTS) — write key + read key derived from passphrase

The identity infrastructure brief proposes a **human/agent identity layer** that sits above all of these:

```
Layer 4: Human identity (Dinis's keys, published in DNS/web/.well-known)
Layer 3: Agent identity (Sherpa's keys, signed by Dinis's key)
Layer 2: Vault/transfer identity (write keys, read keys)
Layer 1: Panel identity (ECDH per-panel keys)
```

This layered model is architecturally sound. Each layer uses the same crypto primitives (ECDH or Ed25519 for signing) but at different scopes.

### 2.2 Chain of Trust Model — SOUND

Dinis → Agent → Message is the correct hierarchy. The implementation should use:
- **Ed25519** for signing (not ECDH — ECDH is for key agreement, Ed25519 is for digital signatures)
- **X.509-like certificate chain** — Agent identity document signed by Dinis's key, verifiable by anyone with Dinis's public key

**Concern:** Key rotation. What happens when Dinis rotates his key? All agent identity documents need re-signing. **Recommendation:** Include a validity period and a "superseded by" field in identity documents.

### 2.3 PGP Key Servers vs Own Infrastructure

**Recommendation: Don't use PGP key servers.** They are designed for PGP keys, suffer from spam/abuse, and are overkill for our needs. Instead:
- **.well-known/sg-identity.json** in the repo — simple, verifiable, version-controlled
- **DNS TXT records** — standard, widely supported, hard to spoof
- **Keyoxide** — modern alternative, worth investigating as a third-party verification layer

### 2.4 DNS TXT + .well-known — RECOMMENDED APPROACH

Use structured, namespaced TXT records:
```
_sgraph.sgraph.ai  TXT  "v=sg1; k=ed25519; p=base64_key"
```

For `.well-known/sg-identity.json`: ensure CORS headers are set (`Access-Control-Allow-Origin: *`) so any client can fetch and verify.

## 3. Individual File Storage Architecture Assessment

### 3.1 Tree Structure + Individual Blob Model — SOUND

This is the right architecture. The current single-blob model doesn't scale for large uploads.

### 3.2 Memory-FS / Storage_FS Integration

Minimal changes needed. Storage_FS already supports arbitrary key-value storage. Use compound keys: `store_content(f"{transfer_id}/tree", tree_data)` and `store_content(f"{transfer_id}/{blob_id}", file_data)`.

### 3.3 S3 Implications

100 PUTs per upload: $0.0005 cost. Negligible. Recommend parallel uploads with `Promise.all()`. S3 handles concurrent writes without issues. The 100-file limit is pragmatic.

### 3.4 Immutability Model — CORRECT

Git-style immutable content with mutable references is the right model. **Concern:** Orphan blob cleanup needs a garbage collection strategy.

### 4.2 sg-file-manager Scope — WELL-SCOPED
