## 1. Three Primitives Architecture (Doc 306) — Assessment

The arch brief (`v0.21.9__arch-brief__three-primitives-agentic-development.md`) is architecturally sound
and accurately describes what exists vs. what is emerging.

### What is accurate

**Primitive 1 (Browser automation + MITM):** EXISTS. The Playwright service with MITM sidecar is code-
verified from the EC2+ECR deployment guide (doc 298, processed 04/21). Declarative JSON sequences, full
traffic capture, domain restriction, secret injection via proxy — all described correctly.

**Primitive 2 (Encrypted VFS — vaults):** EXISTS substantially. SGit + vault infrastructure (scoped
access, version control, zero-knowledge, dual branch model, push/pull) are code-verified. Multi-agent
collaboration is demonstrated. Two access modes (clone vs tree-only web) both exist.

**Primitive 3 (Code execution sandbox):** PROPOSED — EMERGING. The layered container architecture
(commit `bbaaddb`) provides the foundation: spin up container, load code dynamically from S3/URL/vault,
execute, spin down. But the general-purpose sandbox is not yet implemented. The brief correctly
characterises this as "foundation in place, not fully implemented."

### Architectural concern: sandbox isolation boundaries

The brief describes the code execution sandbox but does not specify the isolation model. Key questions
that need answers before implementation:

1. **Network isolation:** Can sandbox containers reach the internet? The Playwright container uses a MITM
   proxy to restrict egress. The code sandbox should have the same model (explicit allowlist, not default
   allow). This must be a design requirement, not an afterthought.
2. **Resource limits:** CPU/memory limits per container. Without these, a malicious or buggy snippet can
   exhaust the host's resources.
3. **Filesystem isolation:** The layered container loads code from vault/S3/URL. Where does execution
   output land? The brief says "write results to vault" but does not address intermediate ephemeral
   filesystem access.
4. **Secret handling during execution:** The MITM proxy pattern for secret injection (used by Playwright)
   should extend to the code sandbox. Agents passing environment variables or config files directly is a
   credentials exposure risk.

**Recommendation:** Before implementing the sandbox, write a security brief that answers these four
questions. The architecture exists; the trust model does not yet.

## 2. SgSend JS API Design (Doc 303)

The `SgSend` JS API design is clean and follows the "API IS the tool" principle established in April.

### What is strong

- The API surface is minimal: six methods cover the full send/receive primitive with zero leaking of
  implementation details (encryption, chunking, token formats).
- Standard meta API (`getManifest`, `getMethods`, `getSkills`) ensures discoverability across the platform.
- The Web Components (`<sg-send-drop>`, `<sg-send-receive>`, `<sg-send-panel>`) follow the IFD pattern:
  composable, zero-dependency, event-driven.
- The scope is deliberately narrow: send/receive only, not vaults, not SGit. Correct call.

### Architectural question: relationship to existing upload code

The current send flow (upload page, `sg-send.js`) already implements the encryption and upload workflow.
The JS API must wrap this, not duplicate it. The brief does not specify whether `SgSend.sendFile()` is a
thin wrapper over the existing `sg-send.js` or a reimplementation. **It must be a wrapper.** Any
divergence creates two parallel implementations with different security postures — a serious concern given
that the encryption logic is security-critical.

**Recommendation:** The Dev review should confirm that the JS API wraps existing `sg-send.js` primitives.
If reimplementation is planned, require an AppSec review before merging.

### Architectural question: localStorage token history and key leakage

Doc 303 proposes token history stored in localStorage. Tokens are share tokens (read-only), not vault
keys. This is acceptable as an interim measure. However:
- Ensure tokens stored in localStorage are NOT vault keys (write-capable credentials).
- The distinction between vault key and share token (CLAUDE.md rules 32–33) must be enforced in the API.
  `SgSend.sendFile()` returns a share token. It should never return a vault key to localStorage.

## 3. Positioning Decision — Architectural Implications (Doc 309)

The hero line is decided: "Version-controlled, client-encrypted vaults. For humans and agents."

The Architect's job here is to verify that the system actually delivers all four claims.

| Claim | Architectural Status | Gaps |
|-------|---------------------|------|
| "Version-controlled" | **EXISTS** — dual branch model, push/pull, commit history (all code-verified) | Shallow clone (sgit CLI) mentioned as supporting feature — code-verify status unclear |
| "Client-encrypted" | **EXISTS** — AES-256-GCM in browser, key never sent to server (code-verified) | Consistent enforcement across all upload paths? JS API must not introduce a server-side path. |
| "Vaults" | **EXISTS** — vault create/read/write/branch/push/pull, SGit CLI, web UI (code-verified) | |
| "For humans and agents" | **MOSTLY EXISTS** — SGit CLI for agents, Playwright service, vault API | Code execution sandbox is PROPOSED. Agent auth (vault-key-as-credential) is Stage 1 only. |

**Concern:** The hero line is now on the website. The platform must not ship a feature that violates
"client-encrypted" (e.g., a server-side file processing path). Every new endpoint that touches file
content must be reviewed against this invariant.

## 4. AWS Marketplace Strategy — Three AMIs (Doc 307)

### AMI status assessment

| AMI | Backend Status | Packaging Status | Architect Assessment |
|-----|---------------|-----------------|---------------------|
| AMI 1 (Playwright service) | EXISTS (code-verified) | CI pipeline baking — not Marketplace-listed | Ready for listing process |
| AMI 2 (SG/Send Server) | EXISTS (code-verified: `sgraph_ai_app_send__docker/`) | Not packaged as AMI | Needs AMI-specific packaging and UserData script |
| AMI 3 (Tools Suite) | Partially exists (tools.sgraph.ai, client-side) | Not packaged | Needs architecture design — what does an "AMI" for client-side tools mean? |

**AMI 3 architectural question:** Most tools run entirely client-side. An "AMI for tools" is essentially a
static file server + optional API gateway to Playwright/SG Send. This is architecturally straightforward
but the brief does not specify what the AMI _provides_ beyond a web server. The Playwright dependency
(for tools that need browser automation) and the SG/Send API dependency (for saving output) must be
documented as optional vs. required. Without this, the AMI 3 listing will be confusing.

### Inter-AMI communication via vaults

The brief's platform story (three AMIs communicating via vault) is architecturally valid. Playwright
writes to vault on AMI 2. Tools reads from same vault. The vault is the data transfer layer. This is
already the pattern in real sessions (Playwright agent reading briefings from vaults, writing output to
vaults). It just needs documentation.

## 5. Website v0.2.0 — Post-Delivery Assessment

The website shipped (code-verified: commits 1946ca4, fea0238, 72a22fc). Key architectural observations:

**Token bar in header:** The inline token input in the website header directly addresses the "receive
flow broken" issue raised in the website redesign brief (doc 310). One click to open a vault. This
is the correct fix — it does not require the user to navigate to send.sgraph.ai first.

**Component reuse question:** The website brief (doc 310) explicitly required importing components from
tools.sgraph.ai rather than rebuilding them. The delivered v0.2.0 has its own token-bar.js and nav.js.
This should be reviewed: are these duplicates of existing components, or do they interface with the
existing components? **If they duplicate existing send.sgraph.ai components, this creates maintenance debt
and divergence risk.**

**Static file architecture:** Website v0.2.0 follows the IFD pattern (`v0/v0.2/v0.2.0/en-gb/`) and
uses the existing S3+CloudFront deployment. No concerns.

## 6. Playwright Screenshot Simple API (Doc 311)

The `/screenshot` and `/screenshot/batch` endpoints are architecturally clean. The "simple path for 80%
of use cases" principle is correct — the full session API is powerful but heavyweight.

**Concern:** The `javascript` parameter executes arbitrary JS in the browser context. This is correct
for the use case (dismiss modals, scroll, etc.) but the parameter must be constrained on the service side
if the API is publicly accessible. Specifically:
- If the Playwright service is behind API key authentication (as the brief states), this is acceptable.
- If the `/screenshot` endpoint is ever made unauthenticated (e.g., as a free-tier feature), the `javascript`
  parameter becomes an injection risk. Flag for AppSec review before any public exposure.
