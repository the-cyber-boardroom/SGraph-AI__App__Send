## 1. Auth MVP Revised Design — Assessment (Doc 297)

### The Decision Path

The document traces four architectural positions before landing on the current one. Each step is justified and the eliminations are sound:

| Position | Eliminated Because |
|----------|-------------------|
| Cognito + encrypted attributes | AWS admin can read Cognito attributes via `AdminGetUser`; KMS keys are AWS-managed — no true zero-knowledge |
| Cognito + client-side-encrypted attributes | Works, but adds master password (new user-facing secret), KDF layer, recovery phrase UX, and Cognito dependency |
| Google Identity Services directly | Removes Cognito but keeps master-password layer and per-user bundle storage |
| **Vault key as credential (chosen)** | Vault key already has all properties of a credential; password manager is the sync fabric; server stores nothing new |

The progression is correct. Position 4 is what remains after genuinely testing alternatives — not a shortcut.

### Architectural Assessment

**Zero-knowledge guarantee:** This design is strictly stronger than any of the prior positions. The server no longer receives any identity signal whatsoever. Not email, not a user ID, not a wrapping-key ciphertext. There are no user records by construction. The `sg:admin` note in §9 is architecturally correct — "user count" is now literally undefined, which is the right outcome for a zero-knowledge product.

**Dependency surface:** Credential Management API is a browser standard (W3C); Google Identity Services is a CDN-loaded browser script. Neither requires server changes. Neither introduces a new npm or pip dependency. This is the cleanest possible expansion of the platform.

**`<sg-vault-picker>` scope:** ~150 lines of vanilla JS replacing four components. Each of the four components proposed in the original brief (`<sg-login>`, `<sg-user-data>`, `<sg-vault-manager>`, `<sg-admin-panel>`) has a clear disposition in §6 — either "superseded", "N/A", or "trivially satisfied." No dangling requirements.

**IFD staging:** The four-stage delivery is well-structured. Stage 1 (localStorage) lets Dev build and test the full UX contract before touching the Credential Management API. Stage 2 adds the real API. Stages 3–4 are independently shippable. This is correct IFD discipline.

**The `PasswordCredential` coverage gap:** Chrome, Edge, Brave, Opera support it; Safari and Firefox do not. The form fallback is the right solution — it's the pattern password managers already support. The architecture accepts this browser variance correctly and documents it clearly.

### Open Questions Requiring Answers

The document raises 6 questions (§13) that need disposition before Stage 2 can ship:

1. **Shared vault auto-save** — when a recipient pastes a shared vault key, the browser will offer to save it. This is likely desirable but AppSec should confirm. Recommendation: allow auto-save; document in UX copy that the saved credential is the recipient's copy of the vault key.

2. **Vault name leakage** — the vault name chosen by the user is visible to Google/Apple/third-party password managers. This is standard browser trust. Recommendation: note it in the privacy FAQ; not a blocker.

3. **`sg1.` prefix** — Architect recommends prepending `sg1.` to the stored credential value (`sg1.o4ghedh2lghqdvczmr24i797:vtx3iz2h`) to enable future format migration without breaking existing saved credentials. Dev should confirm the vault layer can handle a prefixed input gracefully or strip the prefix before processing.

4. **Recovery UX** — not a blocker for MVP but Sherpa should design the "save your vault key" prompt at creation time. A plain display with a copy button is sufficient for MVP.

5. **`PasswordCredential` form fallback** — confirm the form-submit heuristic works in Safari 17+ and Firefox 124+. This is testable with Playwright (ironic, given the Playwright service context).

6. **Multiple accounts in shared browser profile** — not an architectural issue; document in UX. Out of scope for MVP.

### Verdicts

| Decision | Assessment |
|----------|------------|
| Drop Cognito | **Correct** — confirmed |
| Drop Firebase | **Correct** — confirmed |
| Drop master password / KDF | **Correct** — vault key already does the job |
| Browser password manager as sync fabric | **Sound** — well-understood, cross-vendor |
| Google Identity Services as optional vault personalisation | **Elegant** — separates authentication (not needed) from personalisation (optional) |
| One component (`<sg-vault-picker>`) replacing four | **Correct** — scope is appropriate |
| Zero-knowledge claim "stronger than before" | **Verified** — server has zero identity data by construction |

## 2. Container Deployment — Architecture Validation (from merged code)

The merge brought in the actual Docker container code. This is an architectural review of what landed.

### `Fast_API__SGraph__Send__Container` Design

```
Fast_API__SGraph__Send__Container
  extends Fast_API__SGraph__App__Send__User
    setup()                  → super().setup() + conditional global auth
    should_enable_global_auth()  → True if SEND__ENABLE_AUTH=true OR SGRAPH_SEND__ACCESS_TOKEN set
    enable_global_auth()     → adds Middleware__Check_API_Key with x-sgraph-access-token header
    setup_routes()           → super + static routes + Routes__Set_Cookie
    setup_static_routes()    → mounts sgraph_ai_app_send__ui__user at /send/, adds / redirect
```

**Assessment:** The conditional auth pattern (`should_enable_global_auth`) is correct. When `SGRAPH_SEND__ACCESS_TOKEN` is not set, the container runs open (appropriate for internal/dev use). When it is set, global auth activates automatically. No code change needed to switch modes.

**Auth middleware exclusion of `/auth/set-cookie-form`:** This is essential — the cookie form must be accessible without auth so users can set their auth cookie via the form. Verified by test `test__4__auth_cookie_form_excluded`.

**Redirect from `/` to `/send/v0/v0.x/v0.x.x/index.html`:** Correct. The versioned path is constructed from constants rather than hardcoded, so a version bump in the constants file will update the redirect automatically.

### Storage Mode Architecture

`Enum__Storage__Mode` introduces three modes: MEMORY (default), DISK, and S3. Container deployment adds DISK as a meaningful option (Docker volume mounted at `/data`). This is the correct abstraction — the storage layer remains pluggable and no application code needs to change per deployment target.

**Note:** The S3 mode exists in the enum but the integration tests (LocalStack) remain the only accepted path for testing it. This is consistent with existing project rules.

### `provision_ec2.py` (sg-send-ec2 CLI)

The CLI adapts the Playwright service pattern correctly:
- Service port 8080 (vs 8000 for Playwright)
- IAM role `sg-send-ec2` (vs `playwright-ec2`)
- ECR repo `sg-send`
- Single container (`docker run`) vs Playwright's two-container Compose
- Auth env var `SGRAPH_SEND__ACCESS_TOKEN`

**One structural concern:** the access token is stored in EC2 instance tags (`sg:access-token`). Tags are readable by any IAM principal with `ec2:DescribeTags`. In the Playwright context this was API key for a service, but in sg-send context the access token gates the entire service including vault data. Recommendation: document that the EC2 instance's IAM role should be the only principal with `ec2:DescribeTags` permission on sg-send instances, or use AWS Secrets Manager for the token at production hardening time.

## 3. Nano Banana Prompting Research — Architectural Implications (Doc 296)

This is a research synthesis, not an architecture document. Architectural implications:

**For the infographic generator tool:**

1. **Thinking Mode toggle** — if the tool uses the Gemini API directly, this maps to the `thinking_config` parameter in the API request. This is a server-side parameter change (no UI change required for first pass). Priority: HIGH.

2. **Reference-image parameter** — maps to `inline_data` in the Gemini content parts. Adding this requires the tool to accept an optional image input alongside the text prompt. This is a non-trivial API surface change. Priority: MEDIUM.

3. **Under-8-element heuristic** — a schema validation rule, not an API change. Priority: LOW (easy, but only valuable once the other improvements are in).

**For the prompt templates themselves:** The seven findings (narrative brief, quoted text, positive framing, element count, reference images, targeted edits, Thinking Mode) are prompt engineering changes — no code change required for most of them. They can be applied incrementally by editing the template strings.

**Relationship to PLR:** The PLR (Page Layout Renderer) already exists for structured content delivery. The infographic tool outputs images, not PLR JSON. These are complementary, not competing.
