## Executive Assessment

Email-FS v0.6 is the most architecturally complete protocol to emerge from the Explorer team. The four-document set (Specification + Architecture + SKILL + Simulation) constitutes a real protocol release: it has axioms, conformance checks, a decision log, a worked simulation, and an explicit implementation roadmap. This is not a brief — it is a deliverable.

**Architectural verdict:** Sound. The design choices are principled and consistent. Concerns below are refinements, not blockers.

---

## What's Architecturally Strong

### 1. Single-Writer Invariant is the Right Core

The entire system's safety under concurrent sgit branches flows from one rule: only the named agent writes to `mail/{agent}/`. The mailroom is the only exception, and it achieves safety via a different route — disjoint filenames through ULID uniqueness. This is axiomatic (A2) and the correctness argument is tight.

### 2. No Broker is the Right Choice

Delivery as a recipient-driven `mv` (not a sender-delivered push) means:
- Senders cannot create state in a recipient's mailbox (no unsolicited writes).
- Recipients control their inbox lifecycle completely.
- Delivery receipt is a fact in vault history (mailroom entry absent = delivered), not a protocol message.

This avoids the classic broker failure modes entirely.

### 3. Three-State Sessions Solve a Real Problem

The v0.5 flat `mail/sessions/` layout required every reader to parse `end.json` presence to determine in-flight vs completed. The three-folder split (`active/`, `archived/`, `stale/`) makes this an `ls` operation — O(1), no I/O on completed sessions. This is the right tradeoff and the right level of complexity for the benefit.

### 4. Immutable `.eml` + Mutable Sidecar Folder

The separation between the immutable RFC 2822 message of record and the recipient's mutable structured workspace (sidecar folder) is elegant. The eml can be signed without invalidating workflow state. The sidecar is per-recipient infrastructure that can grow arbitrarily without touching the immutable artifact.

### 5. Implementation Roadmap Mirrors Issues-FS Path

Staged rollout (manual → CLI → viewers → MCP → SMTP) mirrors the proven Issues-FS path. Manual-first stress-testing before committing to a CLI API is disciplined — rough edges surface at zero implementation cost.

## Concerns and Flags

### C1: Cross-Vault Forwarding Deferred but Not Designed

Section 9 of the spec says cross-vault forwarding is "forward-compatible" but gives no design beyond "preserve Message-ID, add Resent- headers." When two agents in different vaults (e.g. Explorer vault and Villager vault) need to communicate, this gap becomes real immediately. The spec says "multi-team vaults" are v0.6+. This should be the first extension scoped in the next spec version.

**Flag for Architect:** Design cross-vault forwarding semantics before Explorer and Villager teams start using Email-FS in production vaults.

### C2: GC of Empty Mailroom Subfolders Creates a Subtle State Issue

The spec says "empty pigeonhole subfolders may be GC'd by sgit and recreated on next SEND." This is architecturally fine but creates a subtle issue: an agent checking `ls mail/mailroom/{me}/` after a GC cannot distinguish "never sent to me" from "mailroom GC'd after delivery." The spec says "sgit history disambiguates" but that requires a log walk. Consider whether the SKILL should recommend checking `ls` existence before treating absent folder as "nothing in transit."

**Flag for Dev (CLI design):** `email-fs deliver` should tolerate missing mailroom subfolder gracefully.

### C3: Session Immutability is Convention-Only Until PKI

The spec correctly notes that session immutability is a convention until PKI signing enforces it cryptographically. The risk is that agents in the current stage can accidentally (or maliciously) append to `archived/` session folders without any enforcement. The `email-fs fsck` conformance check C6 is audit-only, not preventive.

**Flag for AppSec:** Document this as a known limitation in the security posture of Email-FS Stage 1. Treat any post-end.json writes to archived/stale sessions as a security incident.

### C4: Cc/Bcc Deferred but Needed for Broadcast

The spec defers Cc/Bcc to v0.6 (confusingly, it says "Cc/Bcc deferred to v0.6" in the Out of Scope table of v0.6). This is a naming issue — it means the next version. Broadcast patterns (Conductor sending a brief to all Explorer agents) currently require N separate messages with N ULIDs. The simulation's fan-out example (§9 of the Architecture doc) shows this works, but it's verbose.

**Flag for next spec version:** Cc as a single eml with N mailroom writes is a good extension candidate.

### C5: Subject-Line Audit Log Discipline Requires Tooling to Enforce

The spec correctly identifies subject quality as critical — the directory listing IS the audit log. But without `email-fs fsck` and the convention being socially enforced, agents will write poor subjects ("re: task", "update", "fyi"). The SKILL addresses this but tooling is the only reliable enforcement.

**No immediate flag** — acceptable risk for Stage 1. `email-fs log --by-subject` will surface quality issues.

---

## Summary

Email-FS v0.6 is a mature protocol release that should be treated as a stable foundation for agentic communication in the Cyber Boardroom stack. The architecture is sound, the axioms are correct, and the implementation roadmap is realistic. Three concerns worth tracking: cross-vault forwarding design (before multi-team use), PKI enforcement for session immutability (AppSec track), and CLI robustness for GC'd mailroom subfolders.

The four-document set should be promoted to `library/docs/specs/` as a stable reference.
