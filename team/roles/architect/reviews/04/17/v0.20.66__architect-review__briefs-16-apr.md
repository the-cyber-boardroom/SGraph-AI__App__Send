## Architectural Assessment

### The Core Tension: Blob Upload vs Delta Upload

The `sgit share` command follows a fundamentally different upload architecture from `sgit push`:

| Command | Upload model | Payload size | Failure in egress-controlled env |
|---------|-------------|-------------|----------------------------------|
| `sgit push` | Delta upload — individual content-addressed objects, small payloads | Bytes to ~KB per object | No — individual PUTs pass through |
| `sgit push` (batch mode) | Batch upload — multiple objects in one request | Medium | Yes — SSL EOF, graceful fallback to individual |
| `sgit share` | Monolithic blob — entire vault as single AES-GCM encrypted zip | ~38MB (5MB parts via multipart) | Yes — DNS cache overflow, no fallback |

The failure is not a bug in `sgit share`'s HTTP handling. It is a consequence of **the design choice to package the vault as a single blob** for sharing. This is architecturally coherent (a share token is a point-in-time snapshot — a bundle makes sense), but it is incompatible with egress-controlled environments that have per-request payload size limits.

The investigation author's hypothesis is well-reasoned: the "DNS cache overflow" 503 is generated by a network intermediary (transparent proxy/NAT in the Claude Web container), not by S3. The plain-text response body (S3 errors are always XML), the HTTP 503 status code, and the fact that `curl` to the S3 host returns a normal 403 all support this conclusion.

---

### Fallback Architecture Options

**Option A: Delta-share (objects-based share)**

Instead of packaging a single encrypted zip, `sgit share` would upload each content-addressed block individually (same path as `sgit push`) and then write a snapshot manifest pointing to those block IDs. The share token would reference the manifest rather than a blob.

*Pros:*
- No large single upload — shares the push path's known-good behaviour
- Incremental: only new blocks since last share need uploading
- Aligns with the immutable block cache design (blocks are already content-addressed and reusable)

*Cons:*
- Share token no longer self-contained — the recipient's reader must walk a manifest and fetch all referenced blocks from the server
- Increases server read API complexity slightly (manifest + N block fetches vs single blob fetch)
- Share "snapshot" semantics become weaker if blocks are mutable on the server (though the content-addressed design makes this safe)

**Option B: Chunked blob upload via push path**

`sgit share` still packages the vault as a single blob but splits it into small pieces (e.g. ≤512KB) and uploads each piece via the same individual object PUT path used by `sgit push`. A manifest records the sequence.

*Pros:*
- Minimal protocol change — the blob model is preserved; the transport layer changes
- Reader can reconstruct the original blob by concatenating chunks in order
- Compatible with the current share token browsing flow in `send.sgraph.ai`

*Cons:*
- Requires new server-side logic to serve chunked blobs (or the client reassembles before decryption)
- Increases share object count on server (e.g. 38MB / 512KB = 76 chunks)
- More complex read path for the browser browse viewer

**Option C: Smaller-vault detection + user warning**

Before attempting multipart upload, `sgit share` checks vault size. If > threshold (e.g. 4MB), it prints a warning: "Large vault detected. In some network environments, share may fail. Consider using `sgit push` and sharing the vault key directly." Proceeds with the attempt; failure is surfaced as a clear error with the workaround.

*Pros:*
- Zero code change to upload path; low risk
- Honest about the limitation

*Cons:*
- Does not fix the failure — it only improves error messaging
- Only appropriate as a short-term mitigation

---

### Recommended Architecture

**Short term (mitigation):** Option C — add size-aware warning + clear error message with workaround instructions. This is a 1-day Dev task.

**Medium term (structural fix):** Option A — delta-share via objects. This aligns with the existing content-addressed object store design, avoids large-blob transport entirely, and makes the share command as robust as the push command. The `BrowseDataSource` interface (already shipped in `send-browse--v0.3.2.js`) already supports loading individual objects — the viewer can handle a manifest-based share token.

**Option B** is not recommended: chunked blob transport is a mid-path complexity that solves transport without solving the underlying design tension. The object store already does this more cleanly.

### Server-Side Investigation (Required Before Fix)

The investigation document identifies three specific server-side checks that should happen before a fix is attempted:

1. **Egress proxy logs** — confirm whether the 503 is generated by the Claude environment proxy or passed through from S3
2. **`x-deny-reason` header** — the Claude environment egress proxy may return this header; `sgit` v0.10.9 may not be logging it
3. **Presigned URL validity** — the share flow calls `POST /api/transfers/create` to get the presigned URL; verify the returned URL parameters (expiry, content-length restriction, part size) are correct

The third point is notable: if the presigned URL has a `Content-Length` constraint that the 5MB multipart part violates, the failure could be on the SG/Send server side, not the proxy. This should be ruled out before blaming the proxy.

---

### AppSec Note

The investigation exposes a subtle security consideration: the egress proxy in the Claude Web environment is terminating HTTPS connections to S3 (or at least inspecting them at the TCP layer). This means:

- The presigned URL authentication parameters (HMAC signature in query string) are visible to the proxy at the TCP/HTTP layer if TLS is being inspected
- The `curl` connectivity test showed TLS 1.3 completing successfully — but if the proxy is performing SSL inspection, this is expected

This is an environment-specific concern (Claude Web container only) and not a vulnerability in `sgit` or the SG/Send server. However, the DevOps team should document which environments are egress-controlled and whether presigned URL authentication is at risk of MITM.
