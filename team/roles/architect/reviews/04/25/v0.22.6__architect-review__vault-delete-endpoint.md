**Version:** v0.22.6
**Date:** 25 April 2026
**Role:** Architect (Explorer Team)
**Branch:** `claude/review-docker-debrief-ZCD2a`
**Brief author:** CLI team
**Brief priority:** Medium

---

## 1. API Contract Assessment

**Overall fit: Good, with one structural mismatch.**

The existing `Routes__Vault__Pointer` uses a consistent naming convention: `{verb}/{vault_id}/{file_id:path}`. For example:
- `PUT /api/vault/write/{vault_id}/{file_id:path}`
- `DELETE /api/vault/delete/{vault_id}/{file_id:path}`
- `GET /api/vault/list/{vault_id}`
- `GET /api/vault/health/{vault_id}`

The brief proposes `DELETE /api/vault/{vault_id}` — a top-level path without a verb segment. This **breaks the existing naming convention**. The established pattern places a verb between `/api/vault/` and `/{vault_id}`, serving both as a human-readable discriminator and as a routing disambiguation layer.

**Recommendation:** Use `DELETE /api/vault/destroy/{vault_id}` or `DELETE /api/vault/purge/{vault_id}`. The word `delete` is already taken by the file-level endpoint (`DELETE /api/vault/delete/{vault_id}/{file_id:path}`), so a distinct verb is necessary anyway. `destroy` is preferred because it conveys irreversibility and maps to vault lifecycle (not file lifecycle).

The `ROUTES_PATHS__VAULT` list in `Routes__Vault__Pointer.py` will need an entry added, and the path must be registered in `setup_routes()` with `add_route_delete(...)`.

The response shape `{"status": "deleted", "vault_id": "…", "files_deleted": 142}` is clean and consistent with existing dict return patterns.

## 2. Auth Model Fit

**Overall fit: Correct, with one clarification needed.**

The existing write-key mechanism is well understood from the code:

1. On first write to a vault, `_ensure_manifest()` creates `transfers/vault/{vault_id}/manifest.json` containing `write_key_hash` (SHA-256 of the supplied write key hex string).
2. All subsequent write-authenticated operations (`write`, `delete`, `batch`) call `_check_vault_write_key()`, which loads the manifest and compares the hash.
3. The route layer reads `HEADER__SGRAPH_VAULT__WRITE_KEY` (`x-sgraph-vault-write-key`) and passes it down to the service.

This same pattern is the correct one for vault-delete. No new auth mechanism is needed.

**Clarification on `check_access_token`:** The brief does not mention whether vault-delete should also require the general `x-sgraph-access-token`. Looking at existing write-gated routes, both `write__vault_id__file_id` and `delete__vault_id__file_id` call `check_access_token(request)` before checking the write key. The vault-delete endpoint should follow the same two-gate pattern: access token first (to enforce the platform-level allow-list), then write key (to enforce vault ownership). The brief omits this but it is required for consistency.

**Important nuance:** The brief describes write_key as an HKDF-SHA256 derivative of a vault master key. The server, however, stores only `SHA-256(write_key_hex)` in the manifest — it has no knowledge of HKDF. The server-side validation is hash comparison only. This is correct and already live. The HKDF derivation happens client-side in the sgit CLI; the server does not need to change its auth model.

## 3. Body + URL Double Confirmation

**Assessment: Safety value is real but modest; the 409 is the important part.**

The double-confirmation pattern (vault_id in both path and body) is a sensible UX guard for a destructive, irreversible operation. It is the same principle as `rm -rf` confirmation prompts or AWS Console requiring you to type a resource name before deletion.

**What it actually prevents:** A mis-wired CLI that calls the correct path but sends a stale or incorrect JSON body. In practice, the path vault_id is what the routing layer acts on; the body vault_id is the explicit human-level confirmation. Returning 409 when they disagree is the correct signal.

**Tradeoffs:**

| Concern | Assessment |
|---------|-----------|
| Over-engineering? | No — for a hard-delete, irreversible endpoint, a small explicit confirmation cost is justified. |
| Breaks REST conventions? | Slightly. DELETE with a required body is unusual, but not prohibited (see section 5 below). |
| CLI ergonomics? | The sgit CLI already constructs the URL and body from the same local vault_id source. Mismatch risk is theoretical unless the CLI has a bug. |
| Value if both values come from the same variable? | Low mechanical safety, but the pattern forces the CLI author to handle both fields explicitly. |

**Verdict:** Keep it, but document explicitly that the 409 response is a client programming error signal, not a user-facing error. The CLI must never surface a 409 to the end user — it indicates a bug in the CLI.

## 4. HTTP Method Choice: DELETE with a Request Body

**Assessment: Acceptable but requires explicit FastAPI wiring.**

HTTP/1.1 does not prohibit a body on DELETE (RFC 9110 §9.3.5 permits it but notes that bodies have no defined semantics for DELETE). In practice:

- FastAPI supports DELETE with a `request: Request` parameter and `await request.json()` — this is how `delete__vault_id__file_id` already works (it uses `request: Request` to read the write key header).
- AWS Lambda (via Mangum) passes the body through correctly for DELETE.
- However, some HTTP clients and intermediaries (CDNs, proxies) may strip DELETE bodies. This is a **deployment risk** that must be tested in the Lambda URL + CloudFront path.

**Alternative considered:** `POST /api/vault/destroy/{vault_id}` — this sidesteps the DELETE+body issue entirely and is semantically coherent (a destructive action initiated by the client). Many APIs use POST for non-idempotent destructive actions.

**Verdict:** DELETE is preferred for semantic correctness. Accept the body risk and add a smoke test that confirms CloudFront passes the body to Lambda for DELETE requests. If the body is stripped in the CloudFront path, fall back to POST.

## 5. Idempotency Design

**Assessment: The brief is internally inconsistent — pick one behaviour.**

The brief states: "return 200 (or 404) rather than an error" for an already-deleted vault. This is contradictory. The correct answer depends on the client's use case.

**Option A — Return 200 (idempotent success):**
- The CLI's `sgit vault delete` is safe to retry after a network timeout.
- The response `files_deleted: 0` communicates that there was nothing left to delete.
- This is the right choice for a CLI operation that may be retried.

**Option B — Return 404:**
- Signals clearly that the vault was not found.
- Forces the CLI to handle two success states (200 and 404).
- Suitable if the caller needs to distinguish "I deleted it" from "it was already gone".

**Recommendation: Return 200 with `files_deleted: 0`** for an already-deleted vault. This makes the operation safely retryable without special-casing by the CLI. The CLI does not need to distinguish "deleted now" from "already gone" — both are successful outcomes from a lifecycle management perspective.

**One edge case the brief misses:** If the vault_id passes all validation but the manifest does not exist (vault was never written to, or was already deleted), `_check_vault_write_key()` returns `True` (line 58 of `Service__Vault__Pointer`: `"No manifest yet — first write creates it"`). This means a write key check on a non-existent vault always passes. The service-level delete must therefore handle the "no manifest, nothing to delete" case as a graceful 200 with `files_deleted: 0`, not as a 403.

## 6. Partial Deletion Handling

**Assessment: 500 is the correct status code, but the retry story is broken.**

Returning 500 with a partial-deletion body is the right signal that something went wrong mid-operation. The problem is that **a retry after a partial deletion is not safe without knowing which files were deleted**.

Consider the scenario:
1. Vault has 142 files. Server deletes 80, then hits an S3 error.
2. Server returns `500 {"files_deleted": 80}`.
3. CLI retries. Server deletes the remaining 62.
4. Second call returns 200.

This works. **However**, the service must not fail if it encounters a file path that no longer exists mid-walk (e.g. because a concurrent retry already deleted it). The walk-and-delete loop must treat "file already gone" as a non-error.

**What `Storage_FS` provides today:**
- `folder__files__all(parent_folder)` — list all files under a prefix. Returns a `List[Safe_Str__File__Path]`.
- `file__delete(path)` — delete a single file. Returns `bool`.
- There is **no bulk-delete or folder-delete primitive** in `Storage_FS`. The service must walk the file list and call `file__delete` per file.

**Implication:** For an S3 backend, this means N individual delete API calls, not one S3 `DeleteObjects` batch call. For a vault with 10,000 files, this is a Lambda timeout risk. The brief does not address this. See section 9 (Missing Items).

**Verdict:** Accept the partial-deletion + 500 design. Add a recommendation to the Dev team to batch delete calls for the S3 backend (via a `Storage_FS` extension or a bulk delete helper), tracked separately.

## 7. Storage Pattern Fit

**Assessment: Fits, but no bulk-delete primitive exists — must be implemented.**

The storage layout for a vault is:

```
transfers/vault/{vault_id}/manifest.json
transfers/vault/{vault_id}/{file_id}/payload
```

All files are enumerated by `list_files()` via `folder__files__all(f'transfers/vault/{vault_id}/')`. The manifest is explicitly skipped in `list_files()` (line 140-141 of `Service__Vault__Pointer`). For vault-delete, the manifest must also be deleted (it's the ownership record — leaving it behind would mean a future write with any key would be accepted as "first write", reinstating the vault as a clean slate).

**Correct deletion order:**
1. Call `folder__files__all(f'transfers/vault/{vault_id}/')` to get all paths including the manifest.
2. Delete all payload files first.
3. Delete `manifest.json` last — this is the tombstone. Deleting it last means the write-key check passes throughout the deletion walk.

**Path traversal guard:** `_validate_vault_id()` in the route layer (and `validate_vault_id()` in the service) uses `VAULT_ID_PATTERN = re.compile(r'^[a-z0-9]{8,24}$')`. This pattern rejects `/`, `..`, `-`, `_`, uppercase, and any other path traversal characters. The vault prefix is always `transfers/vault/{vault_id}/` — which is fully scoped. This is correct and sufficient. No additional guard is needed beyond what already exists.

**Cache invalidation:** `_manifest_cache` is a Lambda-lifetime dict. After a vault is deleted, the manifest cache must be invalidated (`del self._manifest_cache[vault_id]` or `self._manifest_cache[vault_id] = None`). Otherwise, a write attempt to the same vault_id within the same Lambda instance after deletion would pass the cache hit check and create a new manifest with the old write key hash — silently resurrecting the vault. This is a **correctness bug** that the new `delete_vault` service method must handle.

## 8. Rate Limiting

**Assessment: The brief mentions it but cannot be implemented in the current stack without external state.**

The brief specifies "at most N delete requests per write_key per hour" but gives no N. More importantly, the current stack has no rate-limiting infrastructure:

- The Lambda is stateless (no shared memory between invocations).
- There is no Redis, DynamoDB, or other shared counter store wired in.
- `osbot-fast-api-serverless` has no built-in rate-limiting middleware.
- CloudFront WAF could provide IP-level rate limiting, but not write-key-level limiting.

**Realistic options (in order of implementation cost):**

| Option | Cost | Fit |
|--------|------|-----|
| CloudFront WAF — IP-level rate limit on the delete path | Low | Partial (IP ≠ write key) |
| DynamoDB counter (via `osbot-aws`) — write_key_hash + hour bucket | Medium | Correct but adds a dependency |
| Admin Lambda as rate-limit oracle | High | Overkill for this use case |
| None (accept the risk for now) | Zero | Acceptable for Medium priority |

**Recommendation:** Descope rate limiting from the initial implementation. Vault deletion is an infrequent lifecycle operation. A malicious actor with a valid write_key can already delete files one-by-one via `DELETE /api/vault/delete/{vault_id}/{file_id}`. Vault-level deletion does not meaningfully expand the attack surface. Add a CloudFront WAF rule to rate-limit DELETE requests to `/api/vault/destroy/*` at the IP level as a cheap deterrent, and defer write_key-level rate limiting to a later phase.

## 9. Missing Items and Risks the Brief Does Not Address

### 9.1 Lambda timeout risk for large vaults
For S3 storage, deleting N files requires N `s3:DeleteObject` API calls. A Lambda with a 30-second timeout will fail mid-walk for vaults with thousands of files. The brief does not mention this. The Dev team should use S3's `delete_objects` (batch up to 1,000 keys per call) when the backend is S3. This requires either a `Storage_FS` extension or a backend-aware service method. Track as a separate issue.

### 9.2 Manifest cache invalidation (correctness bug risk)
As noted in section 7: `_manifest_cache` must be cleared for the deleted vault_id. If omitted, the deleted vault_id can be reused within the same Lambda instance.

### 9.3 Zip cache cleanup
`Service__Vault__Zip` stores zip archives in storage (from context of `Fast_API__SGraph__App__Send__User.py` — `vault_zip_service` shares the same `storage_fs`). The vault-delete operation should also remove any cached zip file for the vault. The zip path is not in the `transfers/vault/{vault_id}/` prefix tree (it uses a separate path). The brief does not address this.

### 9.4 `check_access_token` omission
The brief does not mention the general access token (`x-sgraph-access-token`). All existing write-gated endpoints require it. The implementation must call `check_access_token(request)` before the write-key check.

### 9.5 Logging specification
The brief says "log the deletion event (vault_id, timestamp, IP, files_deleted)" but the existing codebase has no structured logging. There is no `logger` instance in `Routes__Vault__Pointer` or `Service__Vault__Pointer`. If logging is required, it must be threaded through; if it is aspirational, descope it from the initial implementation.

### 9.6 Idempotency when vault never existed
A valid write key submitted against a vault_id that has never had any files written: `_check_vault_write_key()` returns `True` (no manifest = first-write permission). The route should return `200 {"status": "deleted", "vault_id": "…", "files_deleted": 0}` rather than 404. This is consistent with the "idempotent success" recommendation in section 5.

### 9.7 No tests yet for vault-level lifecycle
`test_Routes__Vault__Pointer.py` covers file-level write/read/delete/batch/list. There are no tests for vault-level destroy. A complete test suite is required (see section 10).

## 10. Implementation Plan

### 10.1 New service method: `Service__Vault__Pointer.delete_vault()`

**Location:** `sgraph_ai_app_send/lambda__user/service/Service__Vault__Pointer.py`

Signature:
```python
def delete_vault(self, vault_id: str, write_key_hex: str) -> dict | None:
```

Logic:
1. Validate `vault_id` via `VAULT_ID_PATTERN` (already exists as a static method).
2. Hash the write key: `submitted_hash = self._hash_write_key(write_key_hex)`.
3. Check write key: `if not self._check_vault_write_key(vault_id, submitted_hash): return None` (caller maps None → 403).
4. List all files: `paths = self.storage_fs.folder__files__all(f'transfers/vault/{vault_id}/')`.
5. If empty: invalidate manifest cache, return `{"status": "deleted", "vault_id": vault_id, "files_deleted": 0}`.
6. Walk and delete each path, counting successes. On exception mid-walk, return partial count with error flag (caller maps to 500).
7. Invalidate manifest cache: `self._manifest_cache.pop(vault_id, None)`.
8. Return `{"status": "deleted", "vault_id": vault_id, "files_deleted": count}`.

### 10.2 New route handler

**Location:** `sgraph_ai_app_send/lambda__user/fast_api/routes/Routes__Vault__Pointer.py`

Proposed path: `DELETE /api/vault/destroy/{vault_id}`

Handler signature:
```python
@route_path('/destroy/{vault_id}')
async def destroy__vault_id(self, vault_id: Safe_Str__Id, request: Request) -> dict:
```

Logic:
1. `self._validate_vault_id(vault_id)` → 400 if invalid.
2. `self.check_access_token(request)` → 401 if missing/invalid.
3. Read `write_key` from `HEADER__SGRAPH_VAULT__WRITE_KEY` → 400 if missing.
4. Read and validate request body: `body = await request.json()`, confirm `body.get('vault_id') == str(vault_id)` → 409 if mismatch.
5. Call `self.vault_service.delete_vault(vault_id=str(vault_id), write_key_hex=write_key)`.
6. `if result is None: raise HTTPException(status_code=403, detail='Write key mismatch')`.
7. Return `result`.

For partial deletion (mid-walk exception), the service should raise a specific exception that the route catches and translates to 500 with the partial count in the body.

### 10.3 Route registration

Add to `ROUTES_PATHS__VAULT`:
```python
f'/{TAG__ROUTES_VAULT}/destroy/{{vault_id}}',
```

Add to `setup_routes()`:
```python
self.add_route_delete(self.destroy__vault_id)
```

No changes to `Fast_API__SGraph__App__Send__User.py` are needed — `Routes__Vault__Pointer` is already wired in at `setup_routes()`.

## Summary

The brief is well-conceived and the auth model reuse is correct. The primary structural issue is the proposed path `DELETE /api/vault/{vault_id}` — it breaks the existing verb-segment naming convention and collides with the existing file-delete route namespace. The recommended path is `DELETE /api/vault/destroy/{vault_id}`. Four implementation gaps require attention before code delivery: (1) rename the path to use a verb segment, (2) add `check_access_token` to the route, (3) add manifest cache invalidation after deletion, and (4) address Lambda timeout risk for large vaults on S3. Rate limiting should be descoped from the initial implementation. The endpoint is otherwise a natural extension of the existing vault service and requires no new auth infrastructure, no new storage backends, and minimal changes to `Fast_API__SGraph__App__Send__User`.
