# Architect Review: 21 April 2026 Briefs (Docs 312–322)

**Version:** v0.22.17 | **Date:** 25 April 2026
**Scope:** Architectural assessment of 11 briefs from 04/21
**Reviewed by:** Architect (Librarian session)

---

## Executive Summary

The 04/21 batch is architecturally coherent and ambitious. Three threads converge: a task-component ecosystem (well-structured, reusable), an ephemeral infrastructure pattern (sound security reasoning), and identity resolution via Google app:data (credible but carries cross-domain risk). The Villager brief is the most immediately actionable — it fixes reliability that already matters in production.

One significant gap: the **session manager service** for VNC orchestration is described as "new" with no prior blueprint. This is the most complex new service in the batch and deserves a separate architecture document before implementation begins.

## Assessment by Brief

### Doc 313: Ephemeral Vault Infrastructure

**Architectural rating: STRONG**

The core principle (reduce attack surface by reducing infrastructure lifetime and consumer count) is sound and well-articulated. The snapshot-as-zip approach is elegant: no database, no export tool, just a directory zip. The Admin API (upload/download/status) is clean and API-driven rather than SSH-driven.

**Concerns:**

1. **Versioned snapshot storage in S3** — the brief specifies versioned zips but does not define the S3 key naming convention. This matters for rollback scripts and CI integration. Recommend: `snapshots/{vault-name}/v{semver}.zip` + `snapshots/{vault-name}/manifest.json`.

2. **Aggressive timeout (5–10 minutes idle)** — "idle" needs a precise definition. Does a long-running `sgit push` count as idle? Does a CI pipeline waiting for a test count? Recommend: track `bytes transferred` rather than just `requests received` as the idle signal.

3. **VPC-only restriction** — the brief says "can be VPC-only, no public endpoint." This is aspirational but requires the agent session itself to be inside the same VPC, which is not true for Claude Web sessions. The security property is real for CI pipelines; it does not apply to external agent sessions without a VPN or bastion.

4. **PGP signing as future target** — well-flagged. The current vault PKI per-branch design is the foundation. This should be tracked as Open Decision for a future session.

**Verdict:** Implement. Start with the website vault use case (Ambassador + CI pipeline = exactly two consumers). Defer VPC-only until the agent-in-VPC architecture is resolved.

### Doc 314: VNC Streaming Desktop

**Architectural rating: SOUND CONCEPT, CRITICAL GAP**

The use case taxonomy is excellent. Five distinct use cases (isolated browsing, secure document viewing, agentic visual interaction, ephemeral workspaces, collaborative viewing) share one infrastructure pattern. The noVNC approach (WebSocket-to-VNC bridge in browser) is the correct choice for browser-based access.

**Critical gap: Session Manager Service**

The brief identifies "a new FastAPI service that runs with Docker socket access" to manage VNC container lifecycle. This is a non-trivial service:

- Container orchestration (start, assign port, monitor, destroy)
- Session authentication (who is allowed to start a session?)
- Billing metering (session duration tracking)
- Health monitoring (detect stalled/zombie sessions)
- Multi-user routing (unique endpoint per session)

None of this is defined beyond the paragraph in the brief. This service is more complex than the SG/Send Lambda and needs its own architecture document before implementation.

**Concerns:**

1. **Port assignment** — VNC sessions need unique ports or unique WebSocket paths. The brief does not specify whether we use port-per-session (simple, limited, requires firewall rules per session) or path-per-session behind a proxy (complex but scalable).

2. **Performance open decision** — the brief correctly flags #20 (VNC performance acceptable?). Do not implement until performance research answers this. If scroll and mouse response are unacceptable, the entire use case breaks.

3. **noVNC vs Apache Guacamole** — the brief lists Guacamole as an alternative. Guacamole provides both VNC and RDP via a proper HTML5 client, has a maintained server component, and handles session management built-in. Worth evaluating before building a custom session manager.

**Verdict:** Research first (#20), then evaluate Guacamole vs custom session manager. VNC is high-value but do not build until performance is confirmed and session manager is designed.

### Doc 315: Ephemeral Observability (Elastic + Kibana)

**Architectural rating: STRONG**

The separation of data capture (continuous, cheap) from visualisation (ephemeral, expensive) is the right approach. The AMI pre-bake strategy for sub-2-minute startup is consistent with the existing Playwright AMI pattern.

**Concerns:**

1. **Elastic snapshot/restore** — Elastic has a native snapshot API, but configuring it to write to S3 requires an S3 repository plugin. This needs confirming against the stripped-down AMI: if unnecessary plugins are removed, the S3 plugin must explicitly be retained.

2. **Playwright screenshot dependency** — dashboard screenshots via Playwright assume the Playwright service is running when observability is needed. Consider: the observability instance could embed a lightweight Playwright (headless Chrome + Playwright) itself, avoiding the dependency on the external service.

3. **Log data in vaults** — shallow clone of a vault to load only log files is clever but depends on `sgit shallow-clone` being implemented. Confirm this exists before building the import pipeline on it.

**Verdict:** Implement Phase 1 (EC2 + Elastic + Kibana from AMI + import/export). The vault integration (Phase 3) waits on `sgit shallow-clone` confirmation.

### Doc 317: Google app:data + Vault Key Storage

**Architectural rating: SOUND, OPEN QUESTIONS**

The proof of concept is working (client-side OAuth, app:data read/write). The two-tier model (app:data for fast/simple, vault for shared/versioned) is clean.

**Concerns:**

1. **Cross-domain OAuth (Open Decision #22)** — this is the critical question. Google OAuth redirect URIs must be registered per domain. If `sgraph.ai`, `send.sgraph.ai`, and `tools.sgraph.ai` all need to access app:data, each domain needs to be in the authorised redirect URI list for the same OAuth client ID. This is typically supported but must be tested. Failure here means the keychain is domain-scoped, not cross-domain.

2. **Separate infrastructure for keychain service** — the brief correctly identifies this as a stability requirement. The keychain service (keys.sgraph.ai or auth.sgraph.ai) must have its own deployment pipeline. A bug in the main SG/Send CI must not be able to break keychain access.

3. **Revision of Open Decision #10** — the 18 Apr resolution said "vault key IS the credential, drop Cognito entirely." The 04/21 brief introduces Google OAuth as the identity layer for Google users. These are not contradictory (vault key for non-Google users; Google OAuth + app:data for Google users) but the open decision needs amending in the reality document to reflect the two-tier approach.

4. **Secrets manager product scope creep risk** — the brief is architecturally sound, but expanding from "vault key storage" to "general-purpose secrets manager with Chrome extension and CLI" is a significant scope expansion. Recommend: ship the app:data vault key storage (MVP), then evaluate secrets manager as a separate product decision.

**Verdict:** Resolve Open Decision #22 (cross-domain OAuth) first. Then implement app:data vault key storage as MVP. Defer secrets manager product scope.

### Docs 318 + 319: Task Components + Unified Shell

**Architectural rating: WELL-STRUCTURED**

The task-first, model-as-implementation-detail principle is the right abstraction. The shared `sg-llm-request` integration layer means model changes are isolated. The shell-forces-shared-components insight is correct: `<sg-header>`, `<sg-footer-actions>`, `<sg-cost-display>`, `<sg-task-input>`, etc. benefit all tools.

**Concerns:**

1. **`<sg-task-infographic>` wraps existing pipeline** — the infographic generator (sg-llm-infographic, sg-llm-events, sg-llm-request) already exists on tools.sgraph.ai. The component must import and wrap it, not rebuild. This is called out in the brief but needs enforcing at code review.

2. **Perplexity Sonar dependency** — `<sg-task-research>` and `<sg-task-evidence>` depend entirely on Perplexity Sonar source link behaviour. If Perplexity changes its citation format or API, both components break. Recommend: version-pin the Perplexity model (e.g. `sonar-reasoning-pro-2025-04`) and add integration tests that check for source link presence in responses.

3. **Shell routing** — the brief specifies direct task URLs (e.g. `/tasks/research`). This requires either client-side routing or server-side URL-to-component mapping. The existing tools.sgraph.ai architecture (static files, no server-side routing) may need extending. Confirm the routing approach before building the shell.

4. **`<sg-task-input>` as shared input** — document context persistence across task switches (drop once, switch tasks without re-uploading) requires the shell to hold the document in memory and pass it to each component's `execute()` API. This is a non-trivial state management requirement for Web Components with Shadow DOM isolation.

**Verdict:** Build shell first (establishes shared components), then task components in order of user value: `<sg-task-research>` → `<sg-task-evidence>` → `<sg-task-review>` → others.

### Doc 320: Three Tools (Image, Secrets Manager, Credits)

**Architectural rating: STRAIGHTFORWARD**

Tool 1 (image generation) is a thin wrapper over `<sg-task-image>`. Tool 2 (secrets manager) is the immediate predecessor to the app:data approach. Tool 3 (credit sales) requires Stripe integration and OpenRouter key provisioning API.

**Concern:** The credit expiry + auto-refund workflow (Open Decision #23) involves Stripe refunds which have regulatory implications in some jurisdictions. Get legal/finance input before shipping auto-refund.

### Doc 321: Vault-Driven CI

**Architectural rating: ELEGANT**

The Ambassador-writes-vault → CI-reads-vault pattern is a clean separation of concerns. Vault key stored as a GitHub Actions secret is the correct credential management approach. The two-mode architecture (direct rendering vs pipeline deploy) handles both live content and QA-gated releases.

**One concern:** Webhook trigger from vault push to GitHub Actions. How does the vault server send a webhook? The brief says "the webhook URL is stored in the vault itself" — but the vault server (SG/Send) would need to call that URL on every push. This requires a server-side hook in the SG/Send push pipeline. Confirm this is designed (not just a UI-side action).

### Doc 322: Video Capture Hardening (Villager Brief)

**Architectural rating: CORRECT APPROACH**

Decoupling recorder from UI is the right refactoring. The `<sg-recorder>` + three sub-components design is clean. The health-check-every-second approach with visual heartbeat is production-grade. Web Worker investigation for tab throttling is the right technical question.

**Key technical risk:** The Web Worker cannot directly access `MediaRecorder` in all browsers (specifically, MediaStream objects cannot be transferred to Web Workers via `postMessage` in older browser versions). Research must confirm current browser support before committing to this approach. The alternative (visibilitychange warning + Wake Lock) is less elegant but more portable.

## Cross-Cutting Architectural Observations

### The Ephemeral Pattern
Three briefs (docs 313, 314, 315) apply the same ephemeral compute pattern. The tooling to spin up, load, run, and destroy should be a shared CLI module, not three separate implementations. Recommend: a `sgraph-infra` CLI tool (analogous to `sg-send-ec2`) that handles instance lifecycle generically, with mode flags for vault, VNC, or observability.

### Vault as Universal Data Bus
The vault appears in every brief: vault snapshots (doc 313), session data (doc 314), log data (doc 315), evidence packs (doc 316), app:data overflow (doc 317), task output (docs 318-319), CI communication (doc 321). The vault is becoming the universal data bus. This is intentional (it is the project's core proposition) but creates a dependency: all these features depend on vault reliability and performance. The villager hardening of the vault infrastructure becomes increasingly critical.

### Security Architecture Gap: Session Manager
The VNC session manager (doc 314) is the only proposed service with no authentication design. Who can start a VNC session? How does billing authentication work? This needs a security model before implementation.
