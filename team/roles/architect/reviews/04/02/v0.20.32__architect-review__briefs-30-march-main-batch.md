## 1. One-Shot Feedback Loops (Doc 205) — Architectural Assessment

### The Core Pattern

The brief defines a cycle:

```
Bundle = [system_prompt + context + findings + task]
→ Send to LLM
→ Extract facts, decisions, code
→ Prune: remove exploration, stale content, superseded versions
→ Updated Bundle (roughly same size, higher quality)
```

**Architectural verdict: sound.** This directly addresses the exponential growth problem of standard chat. The prune step keeps the context at ~constant size across cycles. The key architectural discipline: never append raw responses to history. Always extract → condense → replace.

### Bundle-as-Vault Pattern

**Verdict: elegant and correct.** Mapping the bundle to vault files:

```
bundle/
  system-prompt.md
  context.md
  findings.md
  code-samples/
  reference/
  task.md
  metadata.json
```

Each cycle produces a commit. Each exploration branch is a vault branch. This means:
- Full provenance without carrying full history in every prompt
- Rollback to any previous context state
- Parallel exploration without contaminating the main line
- Vault tools (sgit diff, sgit log, sgit branch) apply directly to bundle management

**Connection to existing architecture:** The vault presigned endpoints (Phase 1 shipped in v0.19.5) handle the storage side. The sgit CLI handles the versioning side. The missing piece is the orchestration layer that performs the extract+prune step automatically.

### Two-LLM Pattern

**Verdict: practical and cost-effective.** Using a cheap/local model (Ollama) for the pruning step and a frontier model (Claude) for the primary response is the correct decomposition. The pruning task is simpler and does not require frontier model quality. This makes the feedback loop sustainable in cost terms.

**Concern:** The brief does not define the interface between the two LLMs. Who triggers the pruning step? Is it automatic after every response, or manual? The implementation needs a clear decision: when does a cycle boundary occur?

### Token Visualiser

**Verdict: valuable diagnostic tool.** Making token consumption visible per section (system prompt, context, history, reference, images, tool definitions) gives the human real-time feedback on bundle health. The growth-over-time graph makes the difference between standard chat and one-shot loops immediately apparent.

**Architectural fit:** This is a natural extension of `sg-llm-stats` (already PROPOSED in Section 19). It could be implemented as a more detailed mode of that component rather than a separate component.

### Eval / Benchmark Tests

**Verdict: critical and underemphasised.** The brief mentions this briefly, but it deserves more attention. Without deterministic tests for key bundles, you cannot know if pruning degraded quality. The test pattern (same input → full bundle vs pruned bundle → compare output quality) should be formalised before implementing the pruning step. This prevents silent quality regression.

## 2. SGit as Universal Data Model (Docs 205, 209)

### Architectural Insight

The `sg-git-graph` brief (doc 209) makes explicit what was implicit in the one-shot feedback loop brief: SGit's data model (nodes + edges + branches + merges) is not specific to file storage. It is a general-purpose branching history model. The same structure describes:

- File vault evolution over time
- LLM conversation threads with branches
- IFD component version trees
- Dev-to-QA patch workflows

**Architectural verdict: this is the correct generalisation.** The sg-git-graph component should not be built as a "vault viewer with a special LLM mode." It should be built as a pure graph renderer that accepts a common data format, and the vault connector and LLM thread connector each produce that format.

### Recommended Data Model

The programmatic API in the brief is close to correct:

```javascript
const graph = new SGitGraph();
const main = graph.branch('main');
const c1 = main.commit('...', { timestamp, author, files_changed });
```

**Concern:** The brief does not define the JSON schema for the static data source. This needs to be specified before implementation so the schema is stable and both data sources produce the same format. Recommend the Architect produce this schema before the Developer builds the component.

### Rendering Library Choice

The brief suggests D3.js or "a lighter alternative." **Recommendation: prefer a lighter alternative.** D3.js is large (~500KB) and complex. For a git-style linear-branching graph (not a force-directed graph), a lighter renderer (or a purpose-built git graph library) is more appropriate. Evaluate `gitgraph.js` (small, purpose-built) before committing to D3.

## 3. Chat Component Architecture (Doc 207)

### One Component, One Thing

**Verdict: correct principle, clear decomposition.** The proposed breakdown:

```
Model Chooser → Chat Window → Input Area → Attachment Manager → Response Renderer → Settings Panel
```

Each component has a single responsibility. The Model Chooser broadcasts the selected model; other components receive it. This eliminates the current problem of model selection logic scattered across multiple components.

**Concern about the Model Chooser:** The brief positions it as a traffic driver ("every OpenRouter user has a reason to visit"). This is a valid product insight, but it means the Model Chooser needs to be polished to a Villager-quality level before it can attract external users. Explorer builds the functionality; Villager makes it production-ready. Do not ship a rough Explorer-phase Model Chooser as a public-facing tool.

### Broadcast Communication

**Verdict: appropriate for this use case.** A JSON broadcast ("model changed to X", "switch to dark mode") is a lightweight pub/sub mechanism between Shadow DOM-isolated components. This is the correct pattern when components cannot directly access each other's internals.

**Concern:** The broadcast mechanism needs a defined event format. If it is ad-hoc (each component invents its own broadcast format), it becomes chaotic. Recommend a shared event constants file (similar to `sg-llm-events.js` already PROPOSED in Section 19) that defines all broadcast event types with their payload schemas.

### Five Bug Fixes

All five bugs are well-described and tractable:

| Bug | Assessment |
|-----|------------|
| Cannot type while streaming | Fix: decouple input area from response area; they are independent DOM elements |
| Scroll lock on user action | Fix: track `userScrolled` boolean; stop auto-scroll when true, resume at bottom |
| Markdown during streaming | Fix: buffer to complete blocks; or show raw text during stream, render on completion |
| Image paste broken | Fix: Attachment Manager as standalone Web Component handles all input modes |
| SG Layout missing | Fix: straightforward refactor, apply to all LLM components |

These are Villager-appropriate fixes (stabilising existing functionality), not Explorer work. Flag for the Villager team.

## 4. Multi-Agent Chat (Doc 208)

### Architectural Assessment

**Verdict: correct direction, significant complexity.** The Agent Picker + parallel SG Layout panels + Multi-Ask pattern is the right UI model. Each agent conversation is an independent panel. SG Layout handles the arrangement. Broadcast communication passes context between panels.

**Key architectural decision:** The brief offers two modes — One-Shot and Multi-Turn. **Recommendation: default to One-Shot.** Multi-turn conversations accumulate context; with the one-shot feedback loop architecture (doc 205), the same quality is achievable without context explosion. One-Shot should be the primary mode, Multi-Turn an opt-in for exploration.

**Agent context from Librarian pages:** The brief proposes loading agent context (role definition, team definition, project status) from Librarian-maintained pages on the website or QA site. This is architecturally correct — it separates the context data (maintained by Librarian) from the UI (maintained by Dev). **Concern:** those pages do not currently exist in a machine-readable format that the UI can reliably fetch and parse. The Librarian needs to define a stable URL and format for agent context files before the Multi-Agent UI can depend on them.

**Async tasks:** The cost estimate before running async API tasks is critical. Without it, the user has no signal that hitting "run" might cost £5. Implement the cost estimate before enabling async API model workflows.

## 5. Offline Mode and `@agent` Comments (Doc 206)

### Architectural Assessment

**Verdict: low-risk, high-value extension of existing capabilities.** The offline workflow (pull → work → push) already exists via sgit. The `@agent` comment pattern adds a new communication channel: human-to-agent via code annotations.

**Proposed comment syntax:**
```python
# @librarian: [observation or request]
# @qa: [testing concern]
# @architect: [design question]
# @dev: [implementation note]
```

The Librarian's daily scheduled run (9 AM, CONFIRMED OPERATIONAL) is the natural place to process these. After processing, comments are removed from code and replaced with a reference ID (e.g., `# librarian:TASK-042 — resolved`).

**Concern:** Comment removal is a code modification. The agent (Librarian) would need write access to the repo to remove processed comments. This is a different permission model from the current patch-based workflow. Consider leaving processed comments in place (with a `# @librarian[processed]:` prefix) rather than removing them, to avoid needing direct repo write access from the Librarian. The comment then serves as a record that the task was captured.
