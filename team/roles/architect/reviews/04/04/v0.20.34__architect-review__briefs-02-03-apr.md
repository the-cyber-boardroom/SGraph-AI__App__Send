## Executive Summary

The two-day batch from 2–3 April is the most architecturally significant set of briefs since the SGit branch model in early March. Three architecture briefs on 3 April define a clean layered infrastructure for browser-side persistence. Two complementary briefs on 2 April define the revenue model and the billing/user-vault integration. Together they form a coherent architecture for a monetisable, privacy-first, offline-capable tool platform.

**The central architectural decision:** The Browser Virtual File System (VFS) should be the single persistence interface for all browser tools. If built correctly, every subsequent tool — infographic generator, document analyser, OpenRouter admin, patch reviewer — inherits encrypted storage, versioning, offline capability, and sync for free.

## 1. Three-Layer Browser Architecture (04/03 Briefs)

The three 04/03 briefs describe a layered stack. Assessed here from bottom up.

### Layer 0–1: SGit Vault Primitives in the Browser (Doc 220)

**Architectural soundness: High.**

The proposal to expose each SGit operation (`commit`, `push`, `pull`, `branch`, `merge`, `status`, `log`) as an independent Web Component follows the project's established IFD pattern. Each primitive is independently testable. Composition into tools happens at the next layer. This is good decomposition.

**Key design decision:** The `sg-vault-file-drop` component is the most important UX primitive. One drop → full pipeline (stage, commit, push). This must be the standard "save a file" gesture across the platform. Every tool that persists data should use it or the VFS layer that wraps it.

**Phase 1 / Phase 2 / Phase 3 sequencing** (feature-rich → composed tools → invisible) is correct. Do not skip to Phase 3. The debuggability of Phase 1 is how you find vault edge cases before users do.

**Concern: Ephemeral vaults (IndexedDB + zip export).** This is architecturally clean but requires that the VFS correctly scopes vault state per tool/session. Without scoping, two tools using the same IndexedDB key will corrupt each other's data. The component manifest storage declaration (from the VFS brief) is the right answer — implement it before assembling multi-tool pages.

### Layer 2: Browser Virtual File System (Doc 221)

**Architectural soundness: Very high. This is the right abstraction.**

The VFS mirrors the server-side `Storage_FS` / Memory-FS already in the codebase. Components already understand this interface. Extending it to the browser creates a consistent mental model: the same file-system metaphor at server and browser level.

**Provider stack assessment:**

| Provider | Priority | Notes |
|----------|----------|-------|
| `memory` | Immediate (testing) | Already conceptually exists in all components |
| `indexeddb` | High | Needed for all offline work and ephemeral vaults |
| `vault` | High | The production store; wraps SGit primitives |
| `zip` | Medium | Useful for ephemeral workspaces and sharing |
| `local-storage` | Low | 5-10 MB limit makes it unsuitable for most use cases |
| `sqlite` (WASM) | Low | Compelling for structured data but adds complexity and WASM binary weight |
| `external` | Low | Nice-to-have; not needed for v1 |

**Recommended sequence:** Implement `memory` first (tests pass immediately), then `indexeddb`, then `vault`. Skip `sqlite` and `external` for now.

**Component manifest storage declarations** are the right access control mechanism. Enforce path scoping from day one. Do not allow a component to read outside its declared paths — this is the principle of least privilege in the browser.

**Layered write-through / read-through stack** (memory → IndexedDB → vault) is correct and is the standard pattern for offline-first applications. The key implementation detail: writes to memory must be non-blocking, writes to IndexedDB must be async (but guaranteed before page unload), and commits to vault are batched/controlled by the VFS configuration, not the component.

**Service Worker protection** (mentioned as future) is architecturally important but correctly deferred. The VFS abstraction is specifically designed to make this swap possible later.

**Encryption transparency** is architecturally sound. The component never sees ciphertext. The vault provider handles encrypt/decrypt at the boundary. This is consistent with the project's zero-knowledge design for transfers.

### Layer 3: Vault as Universal Data Layer (Doc 219)

**Architectural soundness: High — but requires the VFS to be built first.**

The brief correctly identifies the operational problem: every tool has no persistent home. API keys, infographic bundles, chat history, and LLM transaction IDs are all ephemeral today.

**The deterministic ID access pattern** is key. The brief confirms that vaults support deterministic IDs calculated from the vault key. This enables "vault light" access: fetch specific files without cloning the full vault. This is how browser tools can use vault-as-data-layer without downloading hundreds of megabytes.

**Three access modes (remote/cached/full-offline):**

| Mode | Implementation complexity | Use case |
|------|--------------------------|----------|
| Remote | Low (direct fetch via deterministic IDs) | Browser tools, quick reads |
| Cached (IndexedDB) | Medium (write-through cache layer) | Repeated access, medium vaults |
| Full offline (bare vault clone) | High (requires full SGit in browser or CLI) | The CLI already does this |

Start with Remote mode for browser tools. Cached mode is the IndexedDB provider from the VFS. Full offline remains a CLI concern for now.

**Tool-to-tool communication via vault:** This is the right design. No direct tool-to-tool API. The vault is the shared state. One tool writes, another reads. This avoids tight coupling and supports async workflows (the infographic tool generates, the Browse view renders later).

## 2. OpenRouter Integration Architecture (Doc 216, 214)

**Architectural soundness: Medium — the billing architecture has an open question that must be resolved before building.**

### The Open Question: Is Generation History Metadata-Only?

The briefs identify this themselves. If OpenRouter's generation history contains prompt/response content, we cannot store it in user vaults (zero-knowledge model would be broken). This is a blocker for the "save usage history to vault" feature.

**Recommendation:** Research this first. Do not build vault storage for usage history until the answer is confirmed. The admin-side key management, credit limits, and revocation are safe to build regardless.

### Key Management Architecture

The proposed model:
- Project master OpenRouter account
- One API key per user (created programmatically)
- Credit limit set per key (£5)
- Usage stats retrieved per key
- Keys stored: key ID (not the key itself) in the server; key material only in the user's browser/vault

This is architecturally correct for the zero-knowledge model. The server stores the key ID (for stats retrieval). The key material lives in the user's vault. Even if the server is compromised, the LLM key is not exposed.

**Concern:** What happens when a user's £5 key is exhausted and they top up? Can the credit limit be updated on an existing key, or does a new key need to be issued? This is an OpenRouter API constraint, not an architectural choice — but it affects the user experience design. Research this during the API mapping phase.

### Component Suite (`sg-openrouter-*`)

The five proposed components are well-scoped. Each has a single responsibility. The `sg-openrouter-balance` badge is the right cross-cutting integration point — it should appear in every LLM-powered tool.

**Integration sequence recommendation:**
1. `sg-openrouter-models` first — model listing is useful across all tools immediately
2. `sg-openrouter-balance` — single badge, high visibility, low cost
3. `sg-openrouter-key-manager` — admin tool, enables the £5 credit experiment
4. `sg-openrouter-usage` — depends on having keys and usage data
5. `sg-openrouter-logs` — lowest priority, depends on volume of data

## 3. Per-User Vault Architecture (Doc 214)

**Architectural soundness: High — this is consistent with everything already built.**

The security model is correct:
- Server stores: vault ID, OpenRouter key ID, SG/Send access token
- Server does NOT store: vault decryption key, OpenRouter key material, generated content

This is a direct extension of the existing transfer security model. The user controls the vault key. The server facilitates access without seeing content.

**The "My Workspace" page** (vault viewer + API key stats + access token management + file shares) is the right single-entry-point for user state. This should be a new page in the main user UI, not a separate site.

## 4. Patch Review Component (Doc 212)

**Architectural soundness: Good — the vault-as-communication-channel pattern is consistent.**

The three rendering modes are well-staged:
- Mode 1 (patch-only) is achievable without any server infrastructure
- Mode 2 (patch + manifest) requires agents to write manifest JSON alongside patches
- Mode 3 (patch + live source) requires either a proxy or pre-included source files

**The `@agent` comment detection is architecturally important.** The patch review component must emit structured events when `@agent` comments are found — not just display them. This connects the review workflow to the routing infrastructure the Librarian already maintains.

**Concern:** The brief assumes agents will "push patch + manifest to review vault alongside Git commit." This is a workflow change that every agent must adopt. It needs a clear manifest schema (the brief provides a good starting point) and tooling support in the standard commit workflow. Flag this for the Dev team.

## 5. Infographic Tool v0.1.1 (Doc 213)

**Architectural soundness: Good — simplification is the right direction.**

The Simple/Document/Template/Advanced mode hierarchy is correct UX architecture: progressive disclosure. One text box by default; complexity on demand.

**Bundle saving** (input.json + output.png + metadata.json + source documents + prompt history) is the right unit of reproducibility. The bundle format should be treated as a first-class schema — define it as a `Type_Safe` class on the server side if bundle metadata is ever stored server-side.

**The "each tab is an SG Layout instance" (fractal nesting)** is an interesting design choice. Confirm that SG Layout supports nested instances before committing to this pattern in v0.1.1.

## 6. Library Website (Doc 217)

**Architectural soundness: Medium — the concept is sound but the migration scope is significant.**

Moving role definitions, skills, and CLAUDE.md guidance to a separate repo is operationally correct. The current state (everything in the SG/Send repo) is fine for one project; it breaks down when multiple projects need the same team knowledge.

**The "clone-on-start" pattern** has an architectural implication: every project's CLAUDE.md must declare what to clone and how to use it. This needs a standard format before the library repo exists.

**Recommended phasing:**
1. Create the library repo structure (empty, with README)
2. Copy (not move) one set of role definitions as a pilot
3. Test the clone-on-start pattern in one new session
4. Migrate in bulk once the pattern is validated

Do not migrate everything at once. The risk of breaking existing SG/Send sessions is real.

## Summary

The 04/03 architectural briefs form a coherent, well-designed infrastructure layer. The VFS abstraction is the most important deliverable in this batch — get it right and everything else builds on a solid foundation. The OpenRouter integration is straightforward but has one critical research item (generation history content). The Library website is a strategic asset that needs careful phased migration.

**Priority order for architecture work:**
1. Browser VFS (memory + indexeddb + vault providers, component manifests)
2. SGit browser primitives (Phase 1 — feature-rich/debuggable)
3. Vault as data layer (deterministic ID access, remote mode)
4. OpenRouter API research (answer the generation history question)
5. Per-user vault architecture (depends on OpenRouter research)
6. Library website (standalone effort, lower urgency)
