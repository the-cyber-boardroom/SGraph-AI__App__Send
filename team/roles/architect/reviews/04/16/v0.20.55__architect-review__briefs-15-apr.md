## 1. VFS Bridge Architecture Assessment (doc 274)

**Status: EXISTS — code verified in dev as of 04/15 merge**

### What Was Built

The VFS Bridge solves the sandboxed iframe problem cleanly:
- HTML files opened in a `blob:` URL iframe have no base URL context
- Relative `fetch()` calls and `img.src` assignments fail silently
- The fix: inject a script before blob creation that intercepts both `window.fetch()` and `HTMLImageElement.prototype.src` setter
- Intercepted requests are forwarded to the parent window via `postMessage` and resolved against the vault's storage backend

**Architectural merit: High.** The prototype setter override is the correct approach — a `MutationObserver` would miss `new Image()` elements set before DOM attachment. The `__sgVfs` in-flight flag prevents double-processing. This is a non-trivial problem solved correctly.

### BrowseDataSource Interface

The three-method interface is the right level of abstraction:

```javascript
getTree()           // hierarchical tree for file browser UI
getFileBytes(path)  // Promise<ArrayBuffer> — async by design
getFileList()       // flat array for search/path resolution
```

**Assessment:** Minimal, complete, and decoupled. The `writable` flag as an optional boolean is pragmatic — it allows the same interface to serve both read-only shares and writable vault sessions without requiring two separate interfaces. This is the correct design.

**Concern:** `getFileBytes(path)` returns `ArrayBuffer`. Callers need to know the content type to interpret it. For the current use cases (images, markdown, JSON) the file extension is sufficient. If the interface is extended to serve executables or binary blobs, content-type metadata will need to be added. Low urgency.

### Immutable Block Cache

The cache design is architecturally sound:
- Cache key includes `{vaultId}` prefix — prevents cross-vault collisions
- Only `obj-cas-imm-*` blocks cached — mutable refs (`muw`, `snw`) correctly excluded
- Cache stores encrypted ciphertext — decryption still client-side on each load
- Net result: network eliminated for warm loads, security properties unchanged

**Performance:** 6 s → 820 ms on warm cache is significant. For vaults over 50 MB this is the difference between usable and frustrating. The decision to cache ciphertext (not plaintext) is correct — it doesn't weaken zero-knowledge properties.

**Concern:** No cache eviction policy documented. For large vaults with many versions (content-addressed, immutable — so blocks accumulate), the `sg-vault-blocks` Cache API store could grow unbounded. The storage management API is noted in the debrief as "for a future settings UI" — this is correct sequencing. Flag for Villager track: add a cache size indicator and manual clear option.

## 2. Browser Video Creation Pipeline (doc 271)

**Status: PROPOSED — does not exist yet**

### Stack Assessment

The proposed five-layer stack is technically coherent:

| Layer | Assessment |
|-------|-----------|
| Kokoro.js (82M ONNX TTS) | Mature, Apache 2.0, WebGPU + WASM fallback. 160MB download on first use is the main friction. Acceptable for a tool, not for inline UX. |
| Canvas 2D animation | Native browser capability. No new dependencies. Low risk. |
| TalkingHead (Three.js avatar) | WebGL dependency, MIT licence. Add-on for polish — not core to MVP. |
| MediaRecorder API | Native, mature, WebM output. No risk. |
| FFmpeg.wasm | 30MB WASM download. ~1× realtime conversion. Acceptable for batch export. |

**Architectural verdict:** The stack is sound for a dedicated tool where users opt in to the model download. It would be wrong to embed Kokoro.js (160MB) in the main send.sgraph.ai flow. This belongs on `tools.sgraph.ai` as an opt-in tool.

### MVP Sequence — Architect Assessment

The brief's MVP sequence is well-ordered. The Architect endorses:
- **MVP 1 (slideshow + TTS audio):** Achievable with Canvas + Kokoro.js + MediaRecorder alone. Ship this first.
- **MVP 3 (audio-reactive 2D avatar):** The amplitude-based mouth animation is simple and can be done on the same Canvas as the slide. No Three.js needed.
- **MVP 4 (3D TalkingHead):** Deferred — only after MVP 1-3 are solid.
- **MVP 5 (full automation from manifest):** This is the endgame — takes a `manifest.json` + slides and produces a complete video.

### Integration Point with Existing Infrastructure

The video tool has a natural integration with the existing infographic pipeline:
- SGraph API generates slides as base64 PNG
- Doc 272 proposes JPEG compression at generation time (q82, 1200px)
- Video tool could accept the same slides directly
- One-click: "Generate slides → Narrate slides → Export video"

This pipeline is entirely client-side and aligns with SGraph/Send's zero-server-side-processing principle.

**Concern:** The brief doesn't address what happens when the browser tab is closed during a long video render (20+ minutes for a 30-slide deck). The MediaRecorder stream is lost. For long renders, the pipeline should be broken into per-slide segments that can be assembled, or use a worker thread with periodic saves. This is a resilience concern for MVP 1+.

## 3. Browser Image/PDF Compression (doc 272)

**Status: PROPOSED — does not exist yet in SGraph/Send**

### Technical Assessment

The browser Canvas JPEG encoder is the correct approach. Key parameters validated by the brief's own measurements:
- Q82 at 1200px gives 87% file size reduction (850KB → 113KB)
- Q70 at 800px gives 94% reduction (useful for LinkedIn-sized images)
- Full 107-page PDF: 116MB → 14.8MB — puts it under LinkedIn's 100MB limit

The pdf.js + jsPDF pipeline (render each page to Canvas, re-encode as JPEG, reassemble) is the standard browser-side approach. The memory management note ("process pages sequentially, one Canvas reused") is correct and important.

### Integration Architecture

**Immediate opportunity:** The infographic API already generates base64 PNG slides. Adding JPEG compression at generation time (after the API call, before vault storage) requires only the Canvas API — no new dependencies. This is a **2-day integration**, not a new tool.

**Integration point in existing code:** The infographic tool (currently PROPOSED at v0.19.7) should be designed from the start to produce both PNG (archive) and JPEG-dist variants. The reality document should flag this as a design constraint.

**Tool integration options from brief:**
- **Option A (standalone):** `tools.sgraph.ai/compress/` — drag-drop PDFs and images
- **Option B (inline):** Add JPEG download option to infographic generator after each slide
- **Option C (vault-level):** Detect large images on vault push, offer compression

**Architect recommendation:** Build Option B first (lowest cost, highest value), then Option A as a standalone tool. Option C requires vault pipeline changes that are more complex.

### pdf-compressor.html

The brief mentions "The PDF compressor tool is at pdf-compressor.html" — no such file exists in the SGraph/Send repository. This appears to reference a locally-built tool during the OWASP session that was not committed. This should be tracked as a deliverable to bring into the tools site.

## 4. LinkedIn Article Authoring Guide (doc 273)

**Status:** Reference guide / workflow documentation — not a software feature

### Architectural Note

This is a **process document**, not a feature specification. The Python helpers (`img_to_b64`, `build_article`) are standalone scripts, not SGraph/Send application code.

The architectural relevance is indirect:
- Confirms that the infographic pipeline's JPEG compression (doc 272) should target 800px/q70 for LinkedIn use
- Confirms that the `_page.json` component system is not directly usable in LinkedIn (LinkedIn strips divs and CSS) — so `_page.json` → LinkedIn requires a separate export path
- The four article patterns (slide deck narrative, collage, screenshot walkthrough, multi-series) could inform a future "Export to LinkedIn" button in the vault viewer

**Recommendation:** No immediate architectural decision needed. File as reference. Flag the "Export to LinkedIn" concept for a future Explorer session.
