# Architect Review: Briefs — 30 March 2026

**Version:** v0.20.32 | **Date:** 3 April 2026 | **Role:** Architect (Explorer team)

## Summary Assessment

The 30 March batch introduces three architecturally significant proposals that form a coherent, interconnected system. Individually each is well-specified; together they describe a novel infrastructure for agentic LLM workflows. The key coherence risk is that this infrastructure is ambitious relative to current SG/Send scope, and the briefs do not clearly signal which pieces should be built first.

---

## 1. One-Shot Feedback Loops (Doc 205) — Architectural Assessment

### Soundness: HIGH

The core insight — that standard chat history grows super-linearly while one-shot feedback loops maintain roughly constant context — is architecturally correct and well-demonstrated in the brief with concrete token projections. The prune-extract-update cycle is a clean model with no hidden complexity.

**The SGit-as-bundle-store is the strongest architectural decision in this brief.** It is not an analogy. It is a direct repurposing: `findings.md`, `context.md`, `system-prompt.md` are vault objects. Each pruned bundle is a commit. Each experimental approach is a branch. This gives provenance, rollback, and diff for free, using existing SGit infrastructure.

### Architectural Flags

**Flag 1: Bundle file format needs a contract.**
The brief lists bundle files (`system-prompt.md`, `context.md`, `findings.md`, `code-samples/`, `task.md`, `metadata.json`) but does not specify schemas. Before any implementation, define the bundle schema — especially `metadata.json` (which fields, which are required, how `cycle_number` interacts with SGit commit history). This contract needs to be stable before the first loop runs.

**Flag 2: Token Visualiser requires API-level instrumentation.**
To display token breakdown by section (not just total), the application must assemble the prompt with explicit section boundaries and count tokens per section before sending. This requires the bundle assembly step to be instrumented, not just the API response parsing. The brief does not address this dependency.

**Flag 3: Two-LLM pattern introduces latency and cost topology questions.**
Using a smaller model for pruning sounds cheap, but pruning requires sending the full current bundle AND the new response to the pruning model. For long bundles, the pruning call may be larger (in tokens) than the primary call. Model the cost profile before committing. The brief assumes smaller = cheaper, which is true for per-token cost but may not hold for total cost if the pruning call is large.

**Flag 4: Eval tests are underspecified.**
"Same input → expected output (or expected properties)" needs more precision. What constitutes a "maintained quality" answer when the model is stochastic? The brief acknowledges this is targeted, not full eval infrastructure. Recommend: define evaluations as human-reviewed reference outputs (golden outputs), with structural checks (e.g. "response mentions X", "response does not contradict Y") for automation.

### Recommended Implementation Order

1. Define bundle file schema (`metadata.json` contract)
2. Build bundle assembler (reads vault files, assembles prompt with section boundaries, reports token counts per section)
3. Build extract+prune step (initially manual/human-guided, then LLM-automated)
4. Build SGit commit integration (commit after each cycle, branch API)
5. Build Token Visualiser (requires instrumented assembler from step 2)
6. Explore two-LLM pattern (after step 3 is working with primary model)

## 2. `sg-git-graph` Component (Doc 210) — Architectural Assessment

### Soundness: HIGH with one reservation

The component design is clean. The three data sources (JSON, live SGit, programmatic) are the right abstraction — they all feed into the same internal graph model, and the component renders the model. Keeping rendering behind a library (D3.js or lighter) is correct; do not re-implement graph layout.

**The dual-use pattern (real SGit + LLM thread visualisation) is architecturally elegant.** A commit is a commit whether it is a vault object or an LLM interaction. The same graph model renders both because the underlying structure is identical: directed acyclic graph with nodes, edges, branches, and merges.

### Reservation: Live SGit data source latency

The brief specifies reading from the vault's "structure key" (read-only, metadata only). This is the right constraint — no content access. But for large vaults with many commits, the structure key traversal may be slow. The component needs a loading state and progressive rendering (render commits as they arrive, not after all are fetched). This is a UX design constraint that the Designer should address.

### Architectural Flags

**Flag 1: Graph layout algorithm selection.**
D3.js force-directed layout will not produce a readable git graph — it will produce a physics simulation. Actual git graphs use a dag layout algorithm (Dagre, ELK, or a custom level-based layout). Research Dagre or `d3-dag` before committing to D3 core. This is the single biggest implementation risk.

**Flag 2: Programmatic API stability.**
The API proposed (`graph.branch('main')`, `.commit('message')`, `.merge([branches])`) is clean. But the API must be stable from v0.1.0 — if the multi-agent chat system builds on it, breaking changes in the programmatic API break the chat system. Design the API carefully before shipping v0.1.0.

**Flag 3: Real-time update during multi-agent conversation.**
The brief mentions "real-time update as conversation proceeds." Polling the live data source or receiving push events? For the programmatic case (LLM thread), the caller drives the updates. For the live SGit case, polling is the realistic option unless we add a webhook/SSE endpoint to the vault server. Do not promise real-time for live vaults without infrastructure for it.

### Recommended Implementation Order

1. Research and select dag layout library (Dagre or equivalent)
2. Build static JSON data source (testable without infrastructure)
3. Build programmatic API + integration test with multi-agent mock data
4. Build live SGit data source (structure key traversal)
5. Add interactive features (click, hover, zoom, filter)
6. Add time slider (most complex — requires snapshotting graph state at each commit)

## 3. Multi-Agent Chat Architecture (Doc 209) — Assessment

### Soundness: MEDIUM — vision is clear, implementation dependencies are complex

The UI vision is coherent: Agent Picker + independent SG Layout panels + broadcast coordination. The architectural dependencies, however, create a significant build-order problem.

**Dependencies:**
- Multi-agent chat requires `sg-llm` component family (PROPOSED, Doc 202)
- `sg-llm` requires one-shot feedback loop bundles for context management (PROPOSED, Doc 205)
- Debate mode requires broadcast communication between panels (requires broadcast architecture, Doc 208)
- Consolidator requires multi-turn context management
- Async task mode on Claude requires Claude scheduled tasks infrastructure (EXISTS for Librarian/QA; new for ad-hoc tasks)

This is three layers of PROPOSED components stacked. The brief reads as if these components exist and are being assembled; they do not.

### Architectural Flags

**Flag 1: Agent context loading from Librarian-maintained pages.**
This is the right architecture (decouple context from the chat UI; store it where Librarian can update it). But it requires a URL resolution strategy: which page does "Developer (Explorer team)" map to? The Librarian needs to maintain a registry or convention (`/agents/developer.md`, `/agents/appsec.md`). This should be defined before building Agent Picker.

**Flag 2: One-shot vs multi-turn per agent.**
Offering both modes per agent and per task is correct. The implementation risk is that the multi-turn mode will hit context window limits on long conversations — exactly the problem the one-shot feedback loop solves. The brief acknowledges this. The practical guidance: default to one-shot, make multi-turn an explicit opt-in.

**Flag 3: Async task mode — where do results go?**
The brief says "check results later" but does not specify the mechanism. Options: (a) localStorage with notification on page revisit, (b) vault commit with a webhook/notification, (c) Claude scheduled task that writes to a vault. The mechanism determines the infrastructure required. Option (c) is most aligned with existing architecture.

### Recommended Priority

Do not build multi-agent chat until `sg-llm` component family (Doc 202) is implemented. The natural build order:
1. `sg-llm` family → 2. `sg-git-graph` (for thread visualisation) → 3. Model Chooser (Doc 208) → 4. Agent Picker → 5. Multi-Ask → 6. Debate → 7. Async tasks

## 4. `@agent` Code Comments (Doc 206) — Architectural Assessment

### Soundness: HIGH

Simple, low-risk, high-value pattern. The `@agent` comment is just a comment. The processing step is a text scan. The routing is a case statement. No new infrastructure is needed for the comment-writing side.

**The Librarian as processor is correct.** This extends the Librarian's daily run without creating a new role. The only new capability required is: scan file tree, match pattern `# @{role}:`, extract context (function/class, surrounding code), create issue or task, remove comment, leave reference.

### One architectural concern: reference ID format

After processing, the comment is replaced with a reference ID so the human can find the created task. What format? Recommend: `# @librarian → .issues/04/03/librarian-001.md` — the reference IS the relative path to the created issue. This is consistent with the Issues FS pattern already in use.
