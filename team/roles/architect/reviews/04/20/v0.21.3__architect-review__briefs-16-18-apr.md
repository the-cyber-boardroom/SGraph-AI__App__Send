## Executive Assessment

This is the most architecturally consequential batch of briefs since the vault convergence (04/15). Three major architectural decisions are either made or prompted here:

1. **The layered container architecture** (docs 282, 290, 294) — a fully specified, three-layer build system with dynamic code loading, directly analogous to the existing Lambda deployment pattern. This is buildable now and the architecture is sound.

2. **The auth MVP** (docs 286, 291) — a client-side-only identity layer using social login providers as the credential store. The design is consistent with the zero-knowledge architecture principle, though vault key storage presents a genuine cryptographic design question.

3. **The healthcare MVP** (doc 287) — a vertical proof-of-architecture that is architecturally elegant but depends on identity (doc 291), video recorder (doc 285), and the dual branch model (EXISTS) simultaneously.

The batch also introduces the Playwright-as-product arc (docs 289, 294, 295) which extends the service from internal tooling to a standalone product — architecturally independent from SG/Send but sharing the vault communication pattern.

## 1. Layered Container Architecture (Docs 282, 290, 294)

### Assessment: Sound. Build it.

The three-layer design in doc 290 is a direct application of the separation-of-concerns principle that already governs the osbot-aws Lambda deployment pattern. The insight — separate what changes rarely (runtime environment) from what changes often (application code) — is the right design instinct.

**What is strong:**
- Layer 1 (core FastAPI) is genuinely dependency-free: no AWS SDK, no browser, no application code. This makes it the stable, multi-cloud base.
- The three code loading modes (local volume → URL zip → S3) represent a sensible dev→staging→production progression.
- The admin API (code upload/reload) is architecturally honest — it is remote code execution by design, gated by API key. The brief is clear about this trade-off.
- Frozen mode (code baked in at build time via `COPY`) gives a clean production path that doesn't sacrifice the fast development loop.
- Separate CI per layer is correct. Layer 1 change should not require a full browser image rebuild.

**What requires design attention:**

**Bootstrap script security model.** The bootstrap script is the trust boundary between the stable image and the dynamic code. Before finalising Layer 1, define:
- What code sources are permitted? (S3 bucket allowlist, URL domain allowlist, or unrestricted?)
- How is the downloaded code verified? (Hash check against a manifest, or trust-by-URL?)
- What happens if code download fails? (Fail closed with no service, or fail open with default behaviour?)

The v4 axiom document (doc 294) addresses this via "least privilege by declaration" — capability declarations are JSON files committed to version control. This is the right model. Capability JSON must be defined before the bootstrap is implemented.

**Layer 3 base image dependency.** The Microsoft Playwright image (`mcr.microsoft.com/playwright/python`) is a third-party dependency with its own release cadence. Establish:
- Version pinning strategy (pin to `mcr.microsoft.com/playwright/python:1.X.X-jammy` not `latest`)
- Vulnerability scanning in CI before promoting new Layer 3 builds
- Documented process for Playwright version upgrades (update Layer 3 CI → push to Docker Hub + ECR → no application rebuild needed)

**Multi-cloud neutrality.** Layer 1 is genuinely cloud-neutral. Layer 2 is AWS-specific (osbot-aws, S3). The brief acknowledges GCP/Azure equivalents would need their own Layer 2 variants. This is the right design — don't over-abstract at Layer 2. Build AWS-first, document the pattern for future variants.

### Open Design Question: Lambda Container vs Lambda ZIP

The motivation for the container architecture is Playwright (which needs a system-level browser install). For non-Playwright services, the existing zip-based Lambda deployment pattern (osbot-aws) remains faster and simpler. The architect should specify in the next session which services belong on which path — don't migrate everything to containers just because containers now exist.

## 2. Auth MVP — Identity Layer (Docs 286, 291)

### Assessment: Design is correct; vault key storage option needs AppSec review before build.

The client-side-only constraint (doc 291: "All client-side. No server-side code. This is non-negotiable") is consistent with the zero-knowledge architecture principle. The social login approach (Google + GitHub OAuth, no passwords) is the right MVP scope.

**What is strong:**
- The Web Component approach (`<sg-login>`, `<sg-user-data>`, etc.) follows the IFD methodology — composable, zero framework dependencies, drop-in reuse.
- The "accounts are additive, never a gate" principle correctly preserves the existing user experience. This is the right invariant.
- The admin mode zero-knowledge constraint (admin sees metadata, not user vault keys) is the right security posture.
- Option B (encrypted vault key bundle stored in SG/Send, identity provider stores only a pointer + master key) is architecturally interesting and warrants evaluation alongside Option A.

**What requires AppSec review before build:**

**Vault key storage Option A — user attributes.** Storing vault keys (even encrypted) as Cognito custom attributes means the keys are replicated to and accessible via the Cognito API. If Cognito is compromised, the encrypted keys are exposed. The critical question: are the vault keys encrypted client-side before being written to Cognito attributes? If yes, what key is used for that encryption? If it is derived from the OAuth session token, what is the stability and confidentiality of that derivation input?

**The Google OAuth derivation question.** Doc 291 asks: "Is there a stable, user-specific value from Google OAuth that could serve as encryption key derivation input?" This is a critical cryptographic question. If such a value exists and is consistent across sessions, it enables Option C (true zero-knowledge user data). If it does not exist (or is not stable), Option D (master password) becomes the only strong option. AppSec must answer this before the auth component design is finalised.

**Master vault key derivation (Open Decision #11).** The identity brief (doc 286) proposes PBKDF2/HKDF consistent with the existing 600k-iteration pattern. The Architect endorses this direction, but HKDF is the more appropriate choice for key derivation from a master key (PBKDF2 is for password-to-key derivation; HKDF is for key-to-key derivation). Recommend: PBKDF2 for master key from any password-based input → HKDF for per-vault key derivation from master key. AppSec to confirm this split.

**Cross-provider portability.** Doc 291 asks whether a user who migrates from Google/Cognito to Google/Firebase can bring their data. The answer depends entirely on whether vault keys are derived from a stable Google-user-specific value vs from a Cognito-specific value. Architect recommendation: derive from the user's stable Google sub (subject ID), not from any Cognito-specific attribute. This makes the derivation provider-portable.

### Architecture Decision Recommendation

For the auth MVP, proceed with Option A (user attributes in Cognito) for simplicity, but design the encryption layer so that the derivation input is the Google/GitHub OAuth subject ID — not anything Cognito-specific. This gives MVP simplicity with production portability. Document Option B as the production path for investigation after MVP.

## 3. Healthcare MVP (Doc 287)

### Assessment: Correct architecture, but dependency chain must be explicit.

The healthcare MVP is architecturally elegant: the entire application is client-side JavaScript running against a vault (which is an encrypted object store). The server is a dumb encrypted store. This is exactly what "all app logic can be client-side" means as an architectural claim.

**What is strong:**
- Using the dual branch model for patient-doctor collaboration (patient on named branch, doctor on clone branch) is the right pattern — it is the same model that the vault UI already implements (EXISTS via commits 710c96d + 192978f).
- Using a vault commit log as the audit trail is a first-principles use of the architecture — the audit trail is a free artefact of the data model.
- The published summary as a share token + `_page.json` reuses two existing features (share token EXISTS, `_page.json` PLR rendering EXISTS).

**Dependency chain before build:**

| Dependency | Status | Impact |
|------------|--------|--------|
| Vault creation API (`sgit init`-equivalent from browser) | PROPOSED (doc 278) | Blocker: health questionnaire needs to create a vault on form start |
| Dual branch model browser UI | EXISTS | Unblocked |
| LLM via OpenRouter | PROPOSED (from earlier briefs) | Partial blocker: tools.sgraph.ai components exist; SG/Send integration needs wiring |
| Identity for production (Cognito auth MVP) | PROPOSED (doc 291) | Not a blocker for demo MVP; required for healthcare partner production |
| `_page.json` + PLR for published summary | EXISTS | Unblocked |

**Critical design question:** The questionnaire should be built as a Web Component (consistent with IFD methodology) or as a `_page.json` layout? The brief proposes both as options. Architect recommendation: build as a Web Component registered on tools.sgraph.ai, then embed via `_page.json` whitelisted component list. This separates the data logic (questionnaire component) from the rendering (PLR layout).

## 4. Vault Upload UX (Doc 281)

### Assessment: Implementation-ready. Two decisions need resolution before Dev starts.

The vault-backed upload mode is a direct implementation of the vault collaboration primitive (EXISTS) applied to the main SG/Send upload UI. The architectural changes are minimal — the heavy lifting (vault creation, file commits, tree-only loading) is already done. The new work is the UX layer: beta toggle, instant link, progress banner, placeholder files.

**Decision needed: Batch size.**
Doc 281 proposes 10-20 files per commit or by total size threshold. The architect should confirm: use file count threshold only (simpler to implement, predictable commit log), or size-based batching (more sophisticated, handles mixed large/small files better)? Recommendation: start with file count (e.g., 20 files per commit) for MVP. Add size-based batching only if testing shows problems.

**Decision needed: Upload branch vs direct commit.**
The brief (doc 281) proposes batched commits directly to the main vault. The Section 27 PROPOSED item is "upload branch — dedicated branch for large uploads; fast-forward merge on completion." These are slightly different designs. The upload branch approach is safer (failed upload cannot corrupt main tree) but more complex (requires the merge step). For the beta, either works. Recommend clarifying this before Dev starts: open decision #6 (upload branch strategy) should be resolved at the same time as the vault upload MVP design.

**Placeholder files note.** The placeholder files convention is an optional MVP feature. It is architecturally novel — creating files in the vault that represent "content coming" rather than "content here." This needs a naming convention and UI treatment to be defined before implementation. Recommend: defer placeholder files to beta v2 unless the UX clearly needs them for the initial launch.

## 5. Playwright as Product (Docs 289, 294, 295)

### Assessment: The v4 architecture axioms are correct and should be adopted as the project standard.

The three axioms in doc 294 (statelessness, least privilege by declaration, deterministic behaviour) are not Playwright-specific — they apply to every FastAPI service in the SGraph/Send ecosystem. The architect recommends these be formally adopted as the platform axioms and applied to the admin Lambda and user Lambda, not just the Playwright container.

**MITM proxy.** The MITM proxy for 100% traffic capture (doc 289) is the most architecturally interesting addition. Two options: integrated into the same container image, or as a sidecar container that the browser container routes traffic through. Sidecar is the cleaner separation. Recommend: sidecar approach with mitmproxy as the proxy container; the Playwright container's browser is configured to route via proxy. This keeps Layer 3 images clean.

**AWS Marketplace AMI.** Building an AMI from the Layer 3 image is straightforward (packer + AMI builder). The Marketplace listing process (pricing, metering API integration, AMI publication) is new territory. This is a learning exercise as much as a revenue play — plan for 2-4 weeks of marketplace onboarding overhead.
