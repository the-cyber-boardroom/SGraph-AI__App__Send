## 1. JS API as Universal Tool Primitive (docs 265 + 267)

### The Core Decision

The brief formalises an API-first inversion: every tool on `tools.sgraph.ai` exposes a JavaScript API that all consumers (human UI, Playwright, components, test suites, console, Node.js) call equally. The UI is demoted to "one consumer among many."

This is architecturally sound and mirrors the industry standard pattern (OpenAPI/Swagger for web services applied to in-browser tools). The architectural benefit is significant: brittle selector-based Playwright scripts are replaced by deterministic Promise-based API calls.

### Three-Layer Architecture Assessment

```
Layer 1: JavaScript API  (the contract)
Layer 2: UI             (one consumer)
Layer 3: Content        (SKILL files + manifest)
```

Each layer versioning independently via IFD is **correct and necessary**. The API contract should change infrequently (breaking changes are costly to all consumers). The UI can iterate freely without breaking agents. The SKILL files update when the API changes or when explanations improve.

**Concern 1: `window.__tool` as a global.** Using `window.__tool` works for single-tool pages. When tools are embedded in `sg-layout` frames or `_page.json` component blocks, `window.__tool` from an iframe may conflict with the parent frame's `window.__tool` if the naming is not qualified. **Recommendation:** namespace by tool type — `window.__sg_tool_infographic`, or use a registry pattern: `window.__sg_tools['infographic-generator']`.

**Concern 2: API versioning on breaking changes.** The brief notes that breaking API changes are "costly." A concrete governance rule is needed before tools go to production: who approves breaking API changes? What is the deprecation window? Suggested: add an `api_version` field to the manifest; consumers check this on load; old consumers fail fast with a clear message rather than silently misbehaving.

**Concern 3: `window.__tool_skill` text format.** The SKILL.md text format is described as prose, not schema. This is fine for human consumption but needs a stable structure for agent parsing. Recommend specifying the canonical SKILL.md sections: `## What This Tool Does`, `## Quick Start`, `## Actions`, `## Events`, `## Notes`. Agents should be able to locate the `## Actions` block reliably.

### Manifest Design

The OpenAPI-style JSON manifest is well-designed. Key observations:

- `params` with type annotations and `required` flags are sufficient for v1
- `events` should be in the manifest (not just in SKILL-api.md) so the `<sg-tool-api>` Web Component can display them without parsing prose
- `base_url` in the manifest enables programmatic tool discovery — this is good for future tool registries

**Recommendation:** Add an `events` section to the manifest schema alongside `actions`.

### In-Browser Test Suite

The in-browser contract test + REPL console is a high-value feature. It directly addresses the QA visibility problem: QA can call methods and assert on return values without a Playwright harness. This is also a fast feedback loop for development.

**Concern:** Test suite needs isolation from production state. If `connect()` with a test key is called in the REPL, it should not consume production billing quota. Recommend a `dryRun: true` parameter on sensitive methods, or a test mode on the manifest.

## 2. Pack Delivery Service Business Model (doc 266)

### Architectural Surface Area

The business model brief describes Tier 3 (edit/transform access) as generating per-usage token consumption at 25% markup. This requires backend changes:

| Required Backend Feature | Status | Notes |
|---|---|---|
| Token billing per LLM transformation | PROPOSED | Needs Admin Lambda changes |
| Usage tracking per vault edit action | PROPOSED | Needs Analytics pipeline changes |
| Token expiration enforcement | PROPOSED | Already partially designed (profile page brief, doc 246) |
| Read-only vs write-token separation for vault access | PROPOSED | Server-side zip versioning design (doc 241) covers this partially |

**Concern:** The brief assumes token billing (25% markup) is operational. Per the reality document, the credit/billing infrastructure is PROPOSED, not implemented. Tier 3 billing cannot ship until the credit activation and OpenRouter billing pipeline are built. The brief should have a `[DEPENDS ON: credits, billing]` note.

### Vault-Based Delivery Workflow

The seed → pack → share token → mini-site workflow maps cleanly to existing infrastructure (vault write/share, `_page.json` rendering, folder token). The "sample packs" approach is the right way to validate this before building a wizard.

**Architectural constraint to flag:** Tier 2 (source pack download) requires the full vault/zip to be downloadable by the client. If vault contents are large (multiple infographics + source materials), this may hit the Lambda 6 MB response limit. The presigned S3 download URL path exists for large vault files — the pack assembly workflow should document this constraint and use presigned URLs for Tier 2 delivery.

## 3. Browser Automation API Layer (doc 267)

The dev brief complements doc 265. The `<sg-tool-api>` Web Component design is sound:

- Auto-generates from manifest (no per-tool custom code in the panel)
- [Try It] populates params from form inputs and calls `window.__tool.action(params)`
- [Copy Code] outputs a ready-to-run Playwright snippet
- Real-time event log uses CustomEvent listener (events-over-polling principle confirmed)

**Architecture question for the SKILL.md → manifest relationship:** The brief describes `window.__tool_skill` as plain text and `window.__tool_manifest` as JSON. These should not diverge — the manifest IS the structured form of the skill text. The automation panel reads the manifest (not the skill text). Agents should prefer the manifest for programmatic use and fall back to skill text for contextual understanding. **Recommendation:** Document this precedence order explicitly in the tool pattern definition.

## 4. Theme Explorer/Builder MVP (doc 270)

The comms brief is architecturally clean. Key architectural strengths:

- **Reuse constraint is explicit and correct.** The brief lists every shipped asset and prohibits rebuilding them. This prevents the Explorer team from accidentally re-implementing the renderer.
- **Standalone tool first.** Avoiding modifications to the production Browse component is the right risk-management decision. The Browse component is live and has real users.
- **Phase 2 (vault save) correctly deferred.** Vault save requires sgit integration. Not needed for the authoring feedback loop.

**One concern:** The brief mentions loading "an existing vault folder" as a load option. This requires vault read operations (decryption, blob URL resolution for images). If the preview runs inside a simple iframe, the vault read path may not be available. Clarify: does the preview use the full `send-browse--v0.3.2.js` component (which has vault read), or a lightweight standalone renderer? For MVP, the lightweight renderer (page-layout only, no vault asset loading) is simpler and safer.

## Architecture Principles Confirmed by These Briefs

The following architecture principles are now formally established (previously emergent, now explicit):

1. **API-first tools:** Every tool is a JS API. The UI is one consumer.
2. **Three SKILL files per tool:** SKILL-human.md, SKILL-browser.md, SKILL-api.md — each targeting a different consumer type.
3. **Manifest-driven discoverability:** `window.__tool_manifest` (JSON) + `window.__tool_skill` (prose). Manifest is the canonical structured form.
4. **Events over polling:** All async tool operations use Promises or CustomEvents.
5. **IFD versioning per tool layer:** API (slow/contract), UI (fast), Content (medium).
6. **Reuse before rebuild:** Theme Explorer wraps shipped renderer — does not re-implement it.
