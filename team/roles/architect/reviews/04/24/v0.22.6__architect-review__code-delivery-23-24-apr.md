## Executive Summary

The 04/23–04/24 delivery completes a significant architectural shift: the sgraph.ai website homepage is now entirely composed of Shadow DOM Web Components. The `sg-site-header` component evolved through three rapid iterations (v1.0.4 → v1.0.5 → v1.0.6) to reach a self-configuring, environment-aware design. This is a substantial improvement on the previous inline HTML approach and aligns cleanly with the IFD methodology used throughout the rest of the SGraph stack.

**Verdict: sound architecture. One important pattern to standardise; one open question on deployment.**

## 1. `sg-site-header` Architecture Assessment

### Design Decisions — All Sound

**Shadow DOM encapsulation** is correct for a site header component shared across multiple pages and potentially multiple subsites (Send, Tools, future). The `slot='cta'` and `slot='locale'` design is a well-known idiom for composable layouts. Inline header HTML was fragile (every page needed updating on any nav change); the component makes nav changes O(1) rather than O(n pages).

**Site profile system (v1.0.5)** resolves the tension between convention and configuration correctly. `SITE_CONFIGS` provides complete defaults; explicit attributes override them. This is the "convention over configuration" pattern applied to Web Components. The `HOST_SITE_MAP` is a clean way to auto-detect site identity from hostname without requiring the page author to pass it explicitly.

**Environment-aware routing (v1.0.6)** solves a real problem: cross-site links on `dev.sgraph.ai` should point to `dev.tools.sgraph.ai`, not `tools.sgraph.ai`. The `ENV_PREFIX` approach (derived from hostname at module load) is simple and correct. The alternative (build-time config injection) would require a build step the project deliberately avoids. No concerns here.

**Active link detection via `window.location.pathname`** is correct and avoids the need for page authors to pass `active:true` explicitly. The `HOST_SITE_MAP` extension to cover `dev.*` and `main.*` variants is consistent and complete.

### Versioning Pattern — Approved

Three minor versions (`v1.0.4`, `v1.0.5`, `v1.0.6`) are kept as separate directories rather than overwriting a single version. This is the IFD versioned path pattern (`v1/v1.0/v1.0.{N}/`). Each page's `<script type="module">` imports the specific version it was last tested against. Pages have all been updated to `v1.0.6`. This approach allows rollback and avoids cache invalidation issues.

### Gap: No `manifest.json` on v1.0.6

`v1.0.4` and `v1.0.5` both have a `manifest.json`; `v1.0.6` does not. This should be created for completeness. The manifest is not strictly required for the component to function but is part of the convention established by earlier versions.

## 2. Homepage Web Components Architecture Assessment

### Decision: Shadow DOM for All Sections — Correct for This Context

Isolating each homepage section (hero, patterns, use-cases, etc.) as a Shadow DOM component has several advantages:
- **Independent versioning:** each section can evolve without touching others
- **Isolation:** CSS for one section cannot leak into another
- **SgComponent base class reuse:** the template-loading mechanism (`import.meta.url` relative fetch) is already established in the codebase

### The `jsUrl` Pattern — Critical Lesson

The `d1fb32e` fix reveals a non-obvious requirement of the `SgComponent` base class: every subclass **must** declare `static jsUrl = import.meta.url`. Without this, the base class resolves `.html`/`.css` template files relative to its own module URL (wherever `SgComponent` is hosted), not the subclass's URL. This caused HTTP 403 failures on all 10 new components before the fix.

**Architectural recommendation:** This should be documented as a required boilerplate in any `SgComponent` extension guide. Ideally, `SgComponent` should detect a missing `jsUrl` and warn in the console rather than silently resolving to the wrong base.

### `:not(:defined)` Height Reservations — Correct Pattern

Adding height reservations in `style.css` for all 10 new components prevents cumulative layout shift (CLS) before the Web Components registry upgrades the custom elements. This is the correct pattern for Shadow DOM components that render significant content. It requires that the `style.css` be kept in sync with the component registry — a manual maintenance burden to note.

### Concern: 10 Components × 3 Files = 30 HTTP Requests

The homepage now makes 30 HTTP requests just for component templates (10 components × `.html`, `.css`, `.js`). Plus `sg-site-header` (3 files). Total: 33 component asset requests per page load, plus the page HTML itself and any shared CSS/JS.

This is not a correctness concern — browsers handle concurrent requests well — but it is a performance concern for cold loads. The tradeoffs:
- **In favour:** each component is independently cacheable; the cache hit rate on repeat visits is high
- **Against:** first-paint time on cold loads may be noticeable, especially on mobile connections

**Recommendation:** Monitor Lighthouse CLS and LCP scores for the new homepage. If first-paint time is unacceptable, the path forward is either HTTP/2 server push or a build step that bundles component templates into the JS — but this should only be done if metrics show it is necessary.

### No Back-Pressure on Homepage Complexity

The brief (doc 310) had a hard Wednesday deadline. The delivery shows 10 new components, suggesting the homepage grew significantly in content and structure during implementation. This is architecturally neutral — the IFD methodology supports exactly this kind of incremental addition — but it means the homepage is now more complex to maintain. Each new section type added in future should follow the same component pattern.

## 3. Website v0.2.0 Content Corrections

Several content corrections landed after the initial documentation:

- **Security page TTL copy (`21a7f7c`):** "TTL expires access, not deletes data" — this is architecturally significant. The server does not delete objects on TTL expiry; it stops serving them. This is an important distinction for users evaluating the privacy model.

- **Use-case ordering (`0283982`):** Browser automation moved to last; healthcare rows consolidated — reflects feedback on information hierarchy.

- **CTA hierarchy (`6123f4e`):** 'Send a file' promoted to primary — correct given the product's core use case.

No architectural concerns with these corrections; they are copy and layout fixes.

## 4. CI Fix — Explicit main/dev Dispatch

Replacing `'staging'` with explicit `'main'` and `'dev'` dispatch options (`d91f9b3`) removes an ambiguous environment name. The project uses `dev` and `main` branches; `staging` was never a valid branch target. This fix prevents accidental deployments to a non-existent environment.
