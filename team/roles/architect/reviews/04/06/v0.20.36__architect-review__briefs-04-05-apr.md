## Document 229 — Fractal IFD: Components, Tools, Sites as a DAG

**Source:** Tools Explorer team (`briefs/04/04/from_team__tools/`)
**Type:** Architecture brief + migration plan

### Assessment

The per-tool IFD proposal is the correct evolution of the existing monolithic `tools/v0/v0.1/` approach. The current monolith forces unnecessary version churn — 14 tools increment together even when only one changes. Per-tool IFD eliminates this coupling.

**The DAG rule is sound.** The only constraint (no circular load-time imports) is the minimal constraint needed for correct ES module loading. It does not prevent any useful composition pattern. The distinction between load-time imports (cycle-checked) and runtime references (iframe, fetch — no cycle risk) is architecturally important and correctly identified.

**The three-entity model (Component → Tool → Site) is extensible to SG/Send:**

The `send.sgraph.ai` UI already has the implicit structure:
- Components: `send-upload`, `send-download`, `send-browse`, `sg-vault-*`, `sg-llm-*`, etc.
- Tools: the upload page, the browse page, the admin console — each is a standalone page composing components
- Site: `send.sgraph.ai` — the navigable website with shared chrome, locale routing, 20+ pages

Adopting `site.json` for Send would give the team a registry for all pages, navigation structure, and a canonical list of which components each page uses. This is worth doing when the Send UI reaches a stable component inventory.

**The DAG verification script is mandatory, not optional.** Without it, the `resolution: "load"` / `resolution: "runtime"` distinction is just documentation. The script enforces the rule at build time. It must run in CI alongside the existing test suite.

### Architectural Concerns

**Concern 1 — Versioning granularity for `site.json`.**
The site manifest (`site.json`) is the single source of truth for which version of each tool is "live." A single change to one tool's version (e.g. bumping `chat` from `0.1.1` to `0.1.2`) requires a new site IFD version (`v0.2.1`) even if nothing else in the site changes. This is correct IFD behaviour but creates a high volume of site micro-versions. The team should decide: is a site version bump triggered by any tool version bump, or only by navigation/chrome/grouping changes? The brief implies the latter (navigation changes drive site versions) — the `site.json` tool registry is a pointer, not a requirement to version the site on every tool update.

**Concern 2 — `sg-send-cli` in Experimental section.**
The brief includes `sg-send-cli` as a tool in the Experimental section of the tools site. The reality document records `sg-send-cli` (the PyPI package) as **DEPRECATED**, superseded by `sgit-ai`. The tools team brief may be using the old name. This should be clarified before the tools site v0.2.0 is deployed — the tool page should reference `sgit-ai`, not the deprecated CLI.

### Recommendation

Approve the fractal IFD model. Confirm the site versioning policy (navigation changes only vs. any tool bump). Flag the `sg-send-cli` deprecation to the Tools team. Build the DAG verification script before migrating any tool to per-tool IFD.

## Document 230 — URL Resolution for Grouped Storage Paths

**Source:** `briefs/04/05/` | **Leads:** Architect, Developer
**Type:** Dev brief (architectural decision required)

### Assessment

This is a direct consequence of the backend storage restructuring (doc 223, 04/04). The problem is well-defined: the transfer ID alone no longer maps to a unique S3 path once the `by-id-from__*` hierarchy is in place. The brief surveys four options and recommends A+D.

**The A+D recommendation is correct for the current scale.** Rationale:

1. **No URL format changes** — three-word simple tokens continue to work unchanged. Any token already distributed to users remains valid. This is zero-migration for the client.

2. **No new infrastructure** — Option C (registry) adds a DynamoDB table or a separate S3 metadata file. Both add operational complexity (consistency guarantees, write path latency, new failure mode). For the current user volume, the bounded search in Option A is negligible overhead.

3. **The bounded search is genuinely bounded.** The grouping configurations are defined by the server team, not by users. There are at most 3 configurations today (flat legacy, grouped by access-token only, full hierarchy). Adding a new config requires a code change, which is the right forcing function.

4. **Option D (derive from request keys) handles >95% of cases.** The decryption key is always present in a download request. The access token is always present in an upload. The derivation function produces the correct grouping path in the normal case with zero extra S3 requests.

### Architectural Concerns

**Concern 1 — The fallback search must have a version registry.**
The bounded search works because "we define the grouping configurations." This requires a small, versioned registry of path templates (e.g. `LEGACY_FLAT`, `V1_BY_TOKEN`, `V1_BY_USER_AND_TOKEN`). This registry must live in code (not inferred). As new configurations are added, old ones must be retained until a migration purges all legacy transfers. The brief implies this but does not make it explicit.

**Concern 2 — Latency budget needs measuring before the fallback becomes the hot path.**
The target is "under 50ms additional for the fallback." For a Lambda cold-start + S3 HEAD request from `eu-west-2`, 3 HEAD requests will typically cost 30–50ms. This is within budget at low scale. If the fallback fires on >10% of requests, it becomes a performance risk. The team should instrument the fallback rate and alert if it exceeds 5%.

**Concern 3 — Simple token decryption key derivation.**
Simple tokens encode the transfer ID + decryption key. Option D requires the server to derive the `by-id-from__decryption-key` group ID from the decryption key. This derivation must use the same PBKDF2 function defined in the deterministic-safe-paths pattern (doc 225, 04/04) with the appropriate salt registry entry. Confirm that the salt registry has a dedicated entry for `decryption-key-group-id` derivation.

### Recommendation

Approve Option A+D. Require the path template registry to be explicit in code before implementation. Add fallback rate instrumentation from day one.

## Document 231 — Editable Collaborative Folders

**Source:** `briefs/04/05/` | **Leads:** Developer, Architect, Designer
**Type:** Dev brief

### Assessment

The four-mode progression is architecturally coherent and correctly ordered by implementation dependency:

```
Mode 1 (EXISTS) → Mode 2 → Mode 3 → Mode 4
     zip             individual    versioned     vault-backed
     (current)       files + manifest  manifest      (full Git)
```

**Mode 2 is independent of the vault.** Individual file upload uses the existing presigned endpoint infrastructure. The directory manifest is a new encrypted file stored at a deterministic path derived from the folder key. This is a clean extension of the existing transfer model.

**Mode 3 adds a race condition.** The brief acknowledges this and proposes versioned manifests with last-write-wins for light collaboration. This is acceptable for the photo gallery use case (contributors upload independently, edits are rare). The version number must be included in the manifest file and checked on every update — a manifest update that arrives with a stale version number should be rejected or merged.

**Mode 4 is the vault.** The vault tree IS the directory structure — no separate manifest. Mode 4 is the natural endpoint of the progression. It is correctly marked as "design only" for now — it depends on the browser VFS and vault primitives from the 04/03 briefs, which are themselves PROPOSED.

### Design Decisions Required

**Decision 1 — Manifest storage location.**
The brief proposes three options for where to store the directory manifest:
- (a) As an encrypted file at a deterministic path derived from the folder key
- (b) As a vault commit (vault tree IS the manifest)
- (c) Client-side in localStorage with sync

For Mode 2/3, option (a) is the right choice. It is server-agnostic (works with any storage backend), zero-knowledge (server stores encrypted manifest), and consistent with the deterministic-safe-paths pattern. Option (c) introduces a new sync problem. Option (b) requires vault infrastructure.

**Recommend: store manifest as an encrypted file at `{folder_key_derived_id}/__manifest__`** — a reserved deterministic filename within the folder's key space.

**Decision 2 — Transfer model extension vs. new endpoint.**
Mode 2 requires uploading multiple files to a logical "folder" rather than a single zip. This could be:
- (a) Multiple independent transfers (each file gets its own transfer ID)
- (b) A new "folder transfer" object that groups multiple file objects under one transfer ID
- (c) A "batch upload" endpoint that accepts multiple files and returns a folder transfer ID

Option (a) is the simplest to implement — folders are just a bundle of existing transfer IDs managed by the manifest. But it means there is no single "folder transfer ID" to share via simple token. The share URL would need to reference the manifest's location, not a single transfer.

Option (b) is architecturally cleaner — one folder ID, multiple file objects beneath it. The simple token encodes the folder ID + decryption key. The server stores files at `{folder-id}/{file-derived-path}`. This aligns with the backend storage restructuring (doc 223).

**Recommend: option (b) — folder transfer as a first-class object with multiple file children.** This requires a new folder create endpoint and a file-add endpoint. Design these before coding Mode 2.

**Decision 3 — Voting storage.**
Votes are stored as a JSON metadata file alongside the photos. This file has the same race condition as the manifest in Mode 3. For a low-vote-rate photo gallery, last-write-wins is acceptable. For a scenario with many simultaneous voters, a per-vote append model (each vote is a separate encrypted file, aggregated at read time) avoids conflicts.

For v0.1 of the photo gallery, last-write-wins is fine. Design the vote file as `__votes__.json` alongside `__manifest__.json` in the folder's key space.

### Mobile Upload Bugs

Two mobile bugs are correctly identified:
1. **Apple Photos multi-file drop** — DataTransfer API limitation on iOS/macOS. Architecture has no solution (browser API constraint). The fix is pure UX: detect the failure, show a clear message.
2. **Cloud-synced files not local** — `FileReader` returns an error when the file data is not available locally. Detect via `file.size === 0` or a `FileReader` error event, then show guidance.

Neither bug requires backend changes. Both are pure front-end error handling. They should be fixed in the existing upload component before Mode 2 is implemented (Mode 2 adds more upload surface area; fixing known bugs first reduces regression risk).

### Acceptance Criteria Assessment

The 17 acceptance criteria in the brief are implementation-complete (each maps to a specific feature). However, criteria 1–9 (Mode 2/3 + photo gallery) and criteria 11–17 (file management, sharing, version UX, mobile bugs) are independent implementation streams. They should not be treated as a single deliverable.

**Proposed phasing:**
- **Phase A (Mode 2 MVP):** Criteria 1, 2, 3 — individual file upload + manifest + progressive loading
- **Phase B (Photo gallery):** Criteria 6, 7, 8 — gallery component, voting, sort/filter
- **Phase C (Editing):** Criteria 4, 5, 11, 12, 14 — editable folder, file management, version UX
- **Phase D (Mode 4 design):** Criterion 10 — vault-backed, design document only
- **Deferred (polish):** Criteria 9, 13, 15, 16, 17 — export, share-from-inside, mobile bugs, help page

### Recommendation

Approve the four-mode architecture. Confirm manifest storage (option a) and folder transfer model (option b) before implementation. Fix mobile upload bugs as a prerequisite. Phase A delivers independent value without Phase B–D.

## Cross-Cutting Concerns

### The Deterministic-Safe-Paths Pattern Dependency

Three separate features in this batch depend on the deterministic-safe-paths pattern (doc 225, 04/04):
1. URL resolution (doc 230) — derives group IDs from the decryption key
2. Collaborative folders (doc 231) — derives manifest path from the folder key
3. Vote file storage (doc 231) — same derivation as manifest

The shared derivation function and salt registry (doc 225) must exist before any of these features are implemented. This is a hard dependency. The Architect review for 04/05 (doc 225) is the prerequisite for all three.

### IFD Applicability

Modes 2 and 3 require new UI components (directory manifest browser, progressive file list, photo gallery grid). These components should follow IFD versioning:
- `sg-folder-manifest` — renders directory manifest, progressive file list
- `sg-photo-gallery` — grid view, thumbnails, lightbox, vote buttons
- `sg-folder-edit` — file management (rename, delete, move)

Each as its own component with its own IFD chain. The fractal model from doc 229 is directly applicable here.
