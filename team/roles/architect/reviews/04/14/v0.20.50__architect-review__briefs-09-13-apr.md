## Architectural Assessment

### 1. PlaybookLM Pipeline (docs 254, 262, 263)

**Assessment: Sound. Build-ready once sequencing is resolved.**

The five-step pipeline (sources → presentation doc → per-slide briefs → infographic generation → vault output) is architecturally clean. The three-layer prompt architecture (system + positioning + slide) correctly separates concerns: branding/style is the system layer, narrative context is the positioning layer, and per-slide content is the leaf layer. The JS API first design is proven.

**Architectural observations:**

- The "positioning prompt" (the presentation doc) is a significant design decision: it couples all slides to a shared narrative. This is correct for deck cohesion but means regenerating one slide requires the full positioning context to be in memory. Acceptable at current scale.
- Branding-as-source-document is elegant. No special handling needed; the brand guide becomes the system prompt automatically.
- The vault as output container is the right pattern: every artifact (sources, briefs, prompts, logs, infographics) is traceable. This is the provenance layer.
- `window.__tool.generateAllInfographics(options)` — concurrent generation (validated in v0.1.36 via callId UUIDs) means all 6-7 slides can be generated in parallel, not sequentially. Slide deck generation time could drop from `n × 15s` to `≈15s` total. This is worth prioritising.

**Flag — dependency on tools.sgraph.ai CDN:** `vault-generate.js` already loads sg-llm components from CDN. PlaybookLM extends this pattern further. This creates a hard runtime dependency on `tools.sgraph.ai` availability. For production use (revenue-generating slide decks), we need a fallback or at minimum a health check before pipeline start.

### 2. SG Tools Python Package (doc 255)

**Assessment: Sound design. Package architecture is correct.**

The base class pattern (`ToolBase`, `InfographicGenerator extends ToolBase`, `VideoAssembler extends ToolBase`) is the right abstraction level. Each tool publishes a manifest (`window.__tool_manifest`) and the base class validates expected methods exist on first connection. This is a clean contract.

**Architectural observations:**

- Browser reuse across multiple `create()` calls (not re-launching per image) is critical for performance. The brief specifies this; implementation must guard it. A browser pool or singleton browser lifecycle pattern is appropriate.
- Proxy auto-detection from `HTTPS_PROXY` env var is required for Claude.ai session compatibility. The session briefing v2 (doc 257) confirms this is a real operational requirement, not a nice-to-have.
- **Flag:** The Vault wrapper (`from sg_tools import Vault`) wraps the sgit CLI via subprocess. This introduces subprocess management complexity. Consider whether the vault integration belongs in sg-tools-ai or whether sg-tools-ai should simply declare sgit-ai as a dependency and call its Python API directly (if sgit-ai exposes one). Check sgit-ai's public API surface before deciding.
- `meta.getManifest()` (from the session briefing) gives the full schema including method signatures. The base class should use this to validate tool compatibility, not just count methods.

### 3. Examples Website — Root Vault Pattern (doc 258)

**Assessment: Architecturally elegant. Complexity is in the agentic team coordination, not the infrastructure.**

The root vault pattern is well-designed: one environment variable gives the agentic team access to all tokens and configuration. This is consistent with the deterministic-safe-paths pattern from April 4 (doc 229/230).

**Architectural observations:**

- The IFD approach (website code separate from content) is correct. Content lives in vaults; website fetches and renders at runtime. No deploy needed for content updates.
- The website does not store any secrets. The root vault token is only needed by the agentic Claude Code session, not by the public-facing website. This is correct zero-knowledge design.
- **Flag — cache strategy:** If the website fetches vault content at runtime for every visitor, and vaults are encrypted, decryption happens client-side. This means: (a) every visitor must decrypt, (b) no CDN can cache decrypted content. For a showcase site with potentially high traffic, this is fine for launch but will need a "public preview" cache layer eventually (a pre-rendered copy of the hero page with non-sensitive thumbnails).
- The Webmaster agent role is architecturally lightweight: it reads a `_page.json`, decides placement, and writes an updated `_page.json` to the root vault. The website then reflects the change without any deploy. This is correct.

### 4. Embeddable Components in _page.json (doc 259, Part 1)

**Assessment: High value, moderate risk. Whitelist approach is the right security decision.**

The `type: "component"` extension to the PLR schema is architecturally clean. The whitelist of 7 supported components prevents arbitrary JavaScript injection (correctly flagged as AppSec requirement in acceptance criteria).

**Architectural observations:**

- The fractal architecture (pages containing components containing pages) is a genuine insight. The recursion depth limit must be enforced at the renderer level, not the content level — a `_page.json` author should not be able to trigger unlimited recursion.
- Proposed whitelist: `sg-file-upload`, `sg-file-download`, `sg-markdown-viewer`, `sg-video-player`, `sg-infographic-gen`, `sg-vault-browse`, `sg-image-gallery`. These are all existing Web Components. Implementation effort per component is low (register with PLR, pass props through).
- **Flag — props as data injection surface:** The `props` object in the `component` block is passed to the Web Component. If props include a vault token, that token is now in the shared `_page.json` file. For public `_page.json` files (shareable), this means the vault token is visible in the JSON. This may be acceptable for read-only tokens but must be documented. AppSec review needed before shipping.

### 5. WhatsApp Share Mode (doc 259, Part 3)

**Assessment: Correct analysis. The proposed workaround is the right solution within zero-knowledge constraints.**

The problem is fundamental: WhatsApp's preview fetcher hits the server, and the server cannot see the content (fragment-based zero-knowledge). No server-side solution is possible without breaking the security model.

The client-side workaround (Canvas API screenshot + Web Share API) is the correct approach. The two-step "paste image, then paste link as caption" UX is acceptable given the constraints.

**Architectural flag:** The Canvas screenshot captures the decrypted content client-side. The screenshot is not persistent — it exists only in the browser's memory during the share flow. This is acceptable. Do not store the screenshot server-side.

### 6. News Report Tool — Sonar API Integration (doc 260)

**Assessment: The pipeline is sound. The exploration-first approach is correct.**

Building an exploration UI before the full pipeline is good practice — the Sonar API's citation format is unfamiliar territory. The brief correctly identifies citation extraction as the key technical piece.

**Architectural observations:**

- The 4-stage pipeline (Sonar → analysis → infographic generation → package) maps cleanly onto existing architecture: Sonar is just a new model in OpenRouter; the infographic generator is the existing JS API; packaging is the existing vault + `_page.json` pattern.
- **Flag — Sonar via OpenRouter availability:** The brief mentions checking whether Sonar is available via OpenRouter. This is a prerequisite. If Sonar is not on OpenRouter, we need direct Perplexity API access, which introduces a new API key to manage.
- Citation extraction from streaming responses needs careful handling. The citations appear in response metadata, not in the text stream. The tool must capture the full API response (not just the stream) to get citations.

### 7. Pure View Mode (doc 261)

**Assessment: Architecturally simple. High UX value for low implementation cost.**

The `_page.json` layout and rendering already work. Pure view mode is primarily a CSS/JS change: hide the folder chrome, make the PLR content full-width. The URL parameter approach (`?view=pure`) is the cleanest implementation — no change to vault data structure needed.

The `"view_mode": "pure"` flag in `_page.json` metadata is also clean: the sharing decision is part of the content, not the URL. This is the right design for "share as page" vs "share as folder" — the creator's intent is encoded in the content.

## Architecture Decisions Required

| Decision | Context | Recommendation |
|---|---|---|
| **Browser pool design in sg-tools-ai** | Single browser reuse vs pool for concurrent requests | Start with singleton browser lifecycle; add pool when concurrent use cases emerge |
| **Vault wrapper in sg-tools-ai vs dependency on sgit-ai API** | subprocess vs Python API | Check sgit-ai for Python API surface; prefer direct import over subprocess if available |
| **props-as-tokens in embeddable components** | Security: read-only tokens in _page.json | Acceptable for read-only tokens; document explicitly; block write tokens in props |
| **Sonar via OpenRouter vs direct API** | New API key surface | Prefer OpenRouter if Sonar is available; fall back to direct Perplexity API |
| **Concurrent slide generation in PlaybookLM** | Sequential (safe, current) vs parallel (faster) | Build parallel with callId tracking; sequence is a fallback |
| **CDN health check for tools.sgraph.ai** | PlaybookLM runtime dependency | Add health check (`meta.health()`) before pipeline start; surface error clearly |
