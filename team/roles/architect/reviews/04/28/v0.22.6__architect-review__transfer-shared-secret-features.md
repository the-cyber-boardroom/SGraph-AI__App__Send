# Architect Review: Transfer Shared-Secret Features

**Version:** v0.22.6  
**Date:** 2026-04-28  
**Topic:** Usage limits, expiry, and sender-controlled delete for transfers

---

## Current State

`Transfer__Service` stores transfers with these meta fields:

```
transfer_id, status, file_size_bytes, content_type_hint,
created_at, sender_ip_hash, download_count, events
```

There is no enforcement on `download_count`. There is no expiry. There is no way for the sender to delete a transfer after it is created.

---

## Proposed New Meta Fields

| Field | Type | Default | Purpose |
|---|---|---|---|
| `max_downloads` | `int` | `0` | Max downloads before transfer is exhausted. `0` = unlimited |
| `auto_delete` | `bool` | `False` | Wipe payload bytes after last allowed download |
| `expires_at` | `str \| None` | `None` | ISO-8601 UTC timestamp. `None` = no expiry |
| `delete_auth_hash` | `str \| None` | `None` | SHA-256 of `delete_auth`. `None` = delete not enabled |

All four are optional at creation time. A transfer with all four at their defaults behaves exactly like a current transfer — no breaking change.

---

## Delete Auth Model

The sender can delete a transfer they created by proving they know the decryption key. No separate credential is needed.

**Derivation (client-side):**
```
delete_auth = SHA256(decryption_key_hex + ":" + transfer_id)
```

**Storage (server-side, at creation):**
```
delete_auth_hash = SHA256(delete_auth)
```

**Verification (server-side, on DELETE):**
```
SHA256(submitted_delete_auth) == stored delete_auth_hash
```

This is the same pattern as vault write-key: the server only ever stores a hash of a hash. The original decryption key is never seen by the server. Anyone who knows the decryption key can derive `delete_auth` deterministically — there is no separate secret to manage.

If `delete_auth_hash` is `None` at creation time, the transfer cannot be deleted via this endpoint (the sender opted out).

---

## Download Enforcement

When `get_download_payload()` is called:

1. **Expiry check** — if `expires_at` is set and `now > expires_at`, return `410 Gone`. Do not increment `download_count`.
2. **Limit check** — if `max_downloads > 0` and `download_count >= max_downloads`, return `410 Gone`.
3. **Normal download** — increment `download_count`, append event, return payload bytes.
4. **Auto-delete** — after step 3, if `auto_delete` is `True` and `download_count >= max_downloads`, delete the payload bytes from storage. `meta.json` is kept (for audit/transparency). Status changes to `'exhausted'`.

Order matters: expiry is checked before the limit, so an expired transfer returns `410` even if downloads remain.

---

## New Transfer Status Values

| Status | Meaning |
|---|---|
| `pending` | Created, payload not yet uploaded |
| `completed` | Payload uploaded, available for download |
| `exhausted` | Download limit reached and `auto_delete=True` payload wiped |
| `deleted` | Sender explicitly deleted via DELETE endpoint |

`expired` is NOT a stored status — expiry is computed at request time from `expires_at` vs `now`. No background job needed.

---

## HTTP Status: `410 Gone`

Both expiry and exhaustion return `410 Gone` (not `404`). This is intentional:
- `404` implies the resource never existed
- `410` implies it existed but is permanently gone
- The downloader can distinguish "wrong link" from "too late"

---

## New Endpoint

```
DELETE /api/transfers/delete/{transfer_id}
x-sgraph-delete-auth: <delete_auth_hex>
```

| Response | Condition |
|---|---|
| `200` `{"status": "deleted", "transfer_id": "..."}` | Successfully deleted |
| `200` `{"status": "already_deleted", ...}` | Transfer was already deleted/exhausted — idempotent |
| `400` | Missing `x-sgraph-delete-auth` header |
| `403` | `delete_auth` does not match stored hash |
| `404` | Transfer does not exist |
| `409` | Transfer has no `delete_auth_hash` (delete not enabled at creation) |

There is no `404` for already-deleted transfers — same idempotency pattern as vault destroy.

## Schema Changes

### `Schema__Transfer__Create` — new optional fields

```python
max_downloads    : Safe_UInt                  # 0 = unlimited
auto_delete      : bool                       # wipe payload after last download
expires_at       : str                        # ISO-8601 UTC, empty = no expiry
delete_auth_hash : Safe_Str__Id               # SHA-256 of delete_auth, empty = delete disabled
```

### `Schema__Transfer__Info` — new read fields

```python
max_downloads      : Safe_UInt                # As stored
expires_at         : str                      # As stored (empty = none)
downloads_remaining: Safe_UInt                # max_downloads - download_count (0 if unlimited)
is_expired         : bool                     # Computed: expires_at set and now > expires_at
```

---

## What Does NOT Change

- The decryption key is never sent to the server — the client derives `delete_auth` locally and sends only that
- The `download_count` field remains in meta (used for enforcement and audit)
- All existing transfers continue to work — new fields default to disabled
- Storage paths are unchanged (this is a meta-only change for existing transfers)
