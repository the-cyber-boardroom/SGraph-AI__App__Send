# Architect Review: Storage Path Architecture v1

**Version:** v0.22.6  
**Date:** 2026-04-28  
**Topic:** S3 / Storage_FS path structure redesign

---

## Problem with Current Paths

All storage objects live in a flat prefix:

```
transfers/{transfer_id}/meta.json
transfers/{transfer_id}/payload
transfers/vault/{vault_id}/manifest.json
transfers/vault/{vault_id}/{file_id}/payload
vault-zips/{vault_id}/{content_hash}.zip
```

Issues:
1. **No sharding** — `transfers/` lists 999+ objects at scale; S3 `ListObjectsV2` is capped at 1000 per call and costs scale linearly
2. **No version namespace** — impossible to migrate the layout without breaking live data
3. **No deployment scope** — sharing one bucket across tenants or environments requires external prefix injection via `s3_prefix` field (fragile, duplicated in two services)
4. **Inconsistent root** — `transfers/` and `vault-zips/` are siblings at bucket root; no logical grouping

---

## New Path Structure

```
sg-send__data/
  sg-send-api__v1.0/
    {deployment_id}/              ← env var SEND__DEPLOYMENT_ID, default: 'shared'
      transfers/
        {ab}/                     ← first 2 chars of transfer_id (256 shards)
          {transfer_id}/
            meta.json
            payload
      vault/
        {ab}/                     ← first 2 chars of vault_id (256 shards)
          {vault_id}/
            manifest.json
            {file_id}/payload
      cache/
        vault-zips/
          {vault_id}/
            {content_hash}.zip
```

### Constants

Three constants define the path root, hard-coded in source:

| Constant | Value | Purpose |
|---|---|---|
| `STORAGE__BASE` | `sg-send__data` | Bucket-level namespace |
| `STORAGE__VERSION` | `sg-send-api__v1.0` | On-disk layout version (bumped on breaking changes only) |
| `STORAGE__DEPLOYMENT` | `os.environ.get('SEND__DEPLOYMENT_ID', 'shared')` | Deployment scope (default `shared` = multi-tenant) |

All three combine to form `_ROOT = sg-send__data/sg-send-api__v1.0/{deployment_id}`.

### Why `shared` as the default deployment ID

The default deployment mode is multi-tenant: multiple customers share the same deployment. `shared` names this intent explicitly. Named deployments (e.g. `prod`, `staging`, `customer-acme`) override via env var without code changes.

### Why ID-prefix sharding (`{ab}/`)

S3 key prefix sharding distributes objects across 256 logical buckets (first 2 hex chars = `00`–`ff`). Benefits:
- Deterministic from ID alone — no secondary index
- Works for both `transfer_id` (random hex) and `vault_id` (random alphanumeric) — both distribute uniformly
- Supports efficient scoped `ListObjectsV2` per shard prefix when needed
- Zero migration cost — shard is derived on read and write identically

### Why `cache/` is separate from live data

`cache/vault-zips/` is ephemeral and content-addressable. Separating it from `transfers/` and `vault/` means:
- Cache can be evicted by deleting `{_ROOT}/cache/` without touching live data
- IAM policies can scope writes differently (cache = writable by Lambda; live = write-once by design)
- No shard prefix needed for cache — vault_id already provides fan-out

---

## `s3_prefix` Field (Retirement Plan)

`Service__Presigned_Urls` and `Service__Vault__Presigned` currently have an `s3_prefix` field that prepends an environment scope to S3 keys. This was the old mechanism for what `SEND__DEPLOYMENT_ID` now provides.

After this refactor, `s3_prefix` is removed from both presigned services. The full path is derived from `Storage__Paths` functions, which already include the deployment scope.

---

## Single Source of Truth: `Storage__Paths`

New module: `sgraph_ai_app_send/lambda__user/storage/Storage__Paths.py`

All path construction lives here. No service builds a storage path from string literals anymore.

```
path__transfer_meta(transfer_id)      → .../transfers/{ab}/{transfer_id}/meta.json
path__transfer_payload(transfer_id)   → .../transfers/{ab}/{transfer_id}/payload
path__transfer_prefix(transfer_id)    → .../transfers/{ab}/{transfer_id}/
path__vault_manifest(vault_id)        → .../vault/{ab}/{vault_id}/manifest.json
path__vault_payload(vault_id, file_id)→ .../vault/{ab}/{vault_id}/{file_id}/payload
path__vault_prefix(vault_id)          → .../vault/{ab}/{vault_id}/
path__vault_zip(vault_id, hash)       → .../cache/vault-zips/{vault_id}/{hash}.zip
path__vault_zip_prefix(vault_id)      → .../cache/vault-zips/{vault_id}/
```

## Breaking Change: Existing Data

**All existing S3 objects under the old path structure become unreachable after deployment.**

This is intentional and safe for the following reasons:

1. The project has three environments: `dev`, `main`, `prod`
2. `dev` and `main` can be refactored immediately — no real user data at risk
3. `prod` deployment is the final step; any user who has a vault locally cloned can re-push after deployment (same as a git remote rebase — local data is intact)
4. Transfers (one-time file sends) are ephemeral by design — expiry is an upcoming feature; old transfers becoming unavailable is equivalent to natural expiry

No data migration script is needed. Old objects remain in S3 but are simply unreachable by the application.
