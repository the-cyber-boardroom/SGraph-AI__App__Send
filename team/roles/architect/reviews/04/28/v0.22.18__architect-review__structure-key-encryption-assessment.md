## Assessment Summary

The incoming brief is a well-structured architect review authored by a previous Architect session. This document assesses its correctness, completeness, and actionability from the Architect role's perspective before it is handed to implementing teams.

**Overall verdict:** The design is cryptographically sound, the backwards-compatibility strategy is correct, and the team separation is appropriate. Four clarifications are added below.

---

## 1. Cryptographic Correctness

### 1.1 Key Derivation — Correct

Both derivation paths (standard vault key and simple token) correctly apply PBKDF2 → HKDF in sequence. The one-way properties hold: `structure_key` ← `read_key` ← passphrase is a one-way chain. No weakness in the derivation is introduced by this design.

### 1.2 Encryption Assignment — Correct

The field-level split is well-reasoned:
- **Graph topology** (refs, branch index, commits outer envelope, tree outer envelope, tree entry object IDs, public PKI keys) → `structure_key`
- **Sensitive metadata** (`name_enc`, `size_enc`, `content_hash_enc`, `content_type_enc`, `message_enc`) → `read_key`
- **Content** (blob bytes) → `read_key`

The "opaque field" pattern for `message_enc` inside the commit object is clean: a `structure_key` holder decrypts the outer envelope and sees `message_enc` as an opaque base64 blob it cannot read. This requires no schema change.

### 1.3 Backwards Compatibility Strategy — Correct

Using `InvalidTag` (Python) / `OperationError` (Web Crypto) as the fallback signal is the correct approach. AES-256-GCM authentication failures are:
- **Deterministic**: the wrong key always produces `InvalidTag`
- **Safe**: `InvalidTag` never produces plausible plaintext output
- **Specific**: catching only `InvalidTag` / `OperationError` avoids swallowing genuine errors

The fallback is correctly defined as **permanent** — old objects in cloned vaults will always exist, so the fallback can never be removed.

## 2. Clarifications and Additions

### 2.1 Web Crypto `OperationError` Scope

The brief correctly notes that Web Crypto surfaces AES-GCM authentication failures as `DOMException { name: 'OperationError' }`. **Clarification:** `OperationError` is also raised for some operational failures unrelated to authentication (e.g. invalid algorithm parameters, key not yet imported). The catch clause in `decryptStructural` should therefore be scoped as tightly as possible:

```javascript
catch (e) {
    if (e instanceof DOMException && e.name === 'OperationError') {
        return await SGVaultCrypto.decrypt(readKey, data)
    }
    throw e  // re-throw all other errors including programming mistakes
}
```

This is defensive-correct. The brief's version is functionally equivalent but the `instanceof DOMException` check is worth adding for clarity.

### 2.2 Test Coverage Gap: Mixed Vault Object Verification

The brief specifies testing "old vault fallback" and "new vault correct path" but does not explicitly require testing a **mixed vault** — one where some commits use `read_key` (old) and some use `structure_key` (new). This is the most likely real-world state after the write path switches. The test suite must include:

1. Old vault (all objects `read_key`) → all reads fall back correctly
2. New vault (all objects `structure_key`) → all reads succeed directly
3. Mixed vault (first N commits `read_key`, subsequent M commits `structure_key`) → per-object fallback works without state contamination between reads

### 2.3 `Vault__Components` Blast Radius Underestimated

The brief estimates blast radius as "Vault__Sync (primary orchestrator), Vault__Batch, Vault__Diff, Vault__Revert, Vault__Merge, Vault__Stash, Vault__GC, Vault__Branch_Switch." This is correct for SGit-AI__CLI. For SGraph-AI__App__Send, the blast radius in `sgraph_ai_app_send__ui__vault` is:

- `sg-vault.js` — stores `_readKey`, `_writeKey`; must add `_structureKey`
- `sg-vault-ref-manager.js` — ref read/write operations
- Any Web Component that calls `sg-vault.js` for structural operations (vault-browse-edit, vault-generate, vault-shell, vault-header, sg-vault-picker)

The component-level blast radius in the vault UI should be audited before starting implementation.

### 2.4 API Team Optional Enhancement — Decision Required

The brief categorises `/api/vault/zip/{vault_id}` accepting `structure_key` as "optional enhancement worth considering." This needs a concrete decision from the Conductor before implementing Teams make assumptions. The Architect recommends **deferring** this until the core encryption split is implemented and validated — it is not on the critical path and adds a new auth surface to the API.

## 4. Implementation Priority

For this repo (`SGraph-AI__App__Send`), the Vault Web Team work is the relevant scope:

| Priority | Item | Complexity |
|----------|------|------------|
| 1 | `structureKey` derivation in `SGVaultCrypto.deriveKeys()` | Low — single HKDF call added |
| 2 | `decryptStructural()` helper in vault JS | Low — ~10 lines |
| 3 | `_structureKey` state in `sg-vault.js` | Low — store alongside `_readKey` |
| 4 | Update `sg-vault-ref-manager.js` ref read/write | Medium — propagate both keys through all call sites |
| 5 | Update commit and tree object encryption | Medium-High — requires touching object store, commit creation, tree storage |
| 6 | Integration test: old vault fallback + new vault direct | Medium — requires two vault fixtures |
| 7 | `openWithStructureKey` API surface | Low-Medium — gated interface, done after core split |

The SGit-AI__CLI work is out of scope for this repo but is the primary implementation and should proceed in parallel.

---

## 5. Coordination Requirements

This is a **four-team change** with a dependency ordering:

1. **SGit-AI__CLI** (primary) — implements `decrypt_structural`, updates all structural object operations
2. **Vault Web Team (this repo)** — implements equivalent in `sg-vault.js` ecosystem
3. **Vault Tools Team** — consumer; no crypto changes, but audit needed for `structure_key`-only access patterns
4. **API Team (this repo)** — no changes required for correctness

The CLI and Web teams can work in parallel. Tools team follows after Web team ships. API team enhancement (OQ-2) is independent.

**Cross-team compatibility requirement:** Both CLI and Web implementations must produce interoperable vault objects. A vault written by the CLI must be readable by the Web vault, and vice versa. A regression test that creates a vault via CLI and reads it via Web (or simulates this) is needed before closing this workstream.
