## Executive Summary

The 04/22 arch brief finalises the observability system design that was sketched in the 04/21 ephemeral Elastic brief. The six-stage pipeline is architecturally sound. The core design principle — **Elastic is a projection, not a destination** — is the right call and prevents the classic observability trap where the data lives only in the monitoring system. The LETS pattern (Load, Extract, Transform, Save) provides a clean processing abstraction that maps naturally onto idempotent, re-runnable stages.

Two concerns are worth flagging before implementation begins.

---

## Design Assessment

### Six-Stage Pipeline: Sound

| Stage | Assessment | Risk |
|-------|------------|------|
| 0 — Raw Capture | Simple append-only writes to S3. Correct. | None |
| 1 — Security Cleaning | Hash/remove secrets before all downstream processing. Critical and correct placement. | Cleaning must be exhaustive; incomplete rules will let secrets reach Elastic |
| 2 — Structured Extraction | NDJSON normalisation from source-specific parsers. Correct pattern. | Each source needs a bespoke parser; maintenance cost grows with connector count |
| 3 — Signal Reduction | Aggregation into compact summaries. Correct; prevents Elastic index bloat. | Signal definitions must be agreed before implementation (what gets aggregated, at what granularity?) |
| 4 — Elastic Projection | Load from S3; index is always rebuildable. Correct; fully ephemeral. | None, by design |
| 5 — Kibana Dashboards | Pre-baked into AMI; populate on data load. Correct. | Dashboard definitions must be version-controlled separately from AMI to allow updates without full AMI rebuild |
| 6 — Screenshot via Playwright | Automates the Explorer→Villager→Town Planner hand-off. Correct. | Playwright + Kibana screenshot sequencing requires stable headless timing (known issue from earlier Playwright work) |

### LETS Pattern: Useful Naming, Solid Abstraction

The LETS pattern (Load, Extract, Transform, Save) mirrors established ETL methodology while adding the explicit idempotency and intermediate-save constraints. The naming is specific enough to guide implementation and broad enough to apply across all connectors. Architectural endorsement: adopt LETS as the standard pattern for all data processing in the observability stack.

The URL segmentation insight in the dev brief (doc 327) is architecturally significant: storing `/api/v1/vault/status` as `["api", "v1", "vault", "status"]` in an Elastic array field turns Elastic into a URL-component search index without requiring regex queries. This is a low-cost, high-value feature of the Extract pass.

## Architectural Concerns

### Concern 1: Multi-Repo Boundary

The observability pipeline spans multiple repositories and infrastructure components:
- `SGraph-AI__App__Send` — API endpoints (proposed `/admin/*` and `/api/*` connector endpoints)
- SGit CLI — vault operations (`sgit shallow-clone`, `sgit commit` for screenshot vault)
- Elastic AMI — separate baked image (separate repo/pipeline)
- CloudFront log source — AWS infrastructure outside any SG/Send repo

**Risk:** The arch brief specifies the pipeline at a system level but does not map which repo owns which component. If components are built independently across four repos without a contract document, integration will be fragile.

**Recommendation:** Before implementation, produce a component boundary map: which repo owns each CLI command (`sg-obs capture`, `sg-obs clean`, etc.), each API endpoint, and each storage bucket definition.

### Concern 2: Security Cleaning is the Attack Surface

Stage 1 (security cleaning) sits between raw log ingest and all downstream processing. A missed pattern (a regex that fails to match a token format, a new service's log format that wasn't anticipated) means plaintext secrets in the clean bucket, and from there into Elastic, and from there potentially into a screenshot vault.

**Risk:** This is a higher-severity risk than the brief's description suggests. The raw capture includes MITM proxy logs (full HTTP request/response bodies) which will routinely contain bearer tokens, API keys, and PII.

**Recommendation:**
1. Treat Stage 1 as a security-critical component with its own threat model review (AppSec sign-off before ship)
2. Implement a canary scanner on the clean output: scan cleaned files for known-high-risk patterns before allowing downstream processing
3. The clean bucket must not be globally accessible; IAM policy restricts to the pipeline user only

### Concern 3: Elastic Index Schema Versioning

The arch brief specifies NDJSON structured output (doc 326) but does not address schema versioning. When parsers evolve (new fields added, fields renamed), historical indices will have different shapes from current ones. A Kibana dashboard designed for the current schema will silently drop historical data.

**Risk:** This is manageable but must be designed upfront. Retrofitting schema versioning onto a live index is painful.

**Recommendation:** Include an `_schema_version` field in every NDJSON event from Stage 2. Dashboards filter on this field when schema matters. New parser versions bump the schema version.

## Verdict

The 04/22 observability briefs are architecturally sound. The pipeline design is clean, the ephemeral-by-default constraint is enforced, and the LETS abstraction provides a good implementation guide. Three concerns (multi-repo boundary, security cleaning attack surface, schema versioning) must be resolved before full implementation begins, but none are blockers for starting Stage 0 and Stage 1 work.

**Priority order for implementation:**
1. Multi-repo ownership map (design artefact, no code)
2. Stage 1 security cleaning threat model (AppSec)
3. Refactor existing CloudFront connector to LETS pattern (Dev)
4. Index management API extraction (Dev)
5. Dashboard management API extraction (Dev)
6. Connector API generic interface (Dev)
