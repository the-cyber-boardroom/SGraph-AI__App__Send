## 1. Server-Side Zip Folder Versioning (Doc 241) — ASSESSED

**Brief:** `v0.20.38__arch-brief__server-side-zip-folder-versioning.md`
**Verdict:** Architecturally sound. Well-scoped. Recommend proceeding.

### Assessment

The design is correct. The single-mutable-file invariant (`head` only, everything else immutable) is a proven pattern (content-addressable storage, S3 versioning, Git itself). The compare-and-swap on `head` is the right mechanism for conflict detection at this scale.

**What this solves:** The zip monolith problem. All of Mode 2/3 folder sharing (doc 231 from 04/05) depends on this backend design.

**Structural fit:** This is architecturally between the current zip model and a full vault. The brief correctly identifies it as a stepping stone — folders that outgrow this can be promoted to vault-backed (Mode 4). That promotion path needs to be defined, but is not a blocker.

### Implementation Notes for Dev

| Component | Architectural Constraint |
|-----------|------------------------|
| `head` pointer | S3 conditional writes (If-Match ETag or custom compare-and-swap) must be used — NOT read-then-write (race condition) |
| Tree snapshots | Immutable, content-addressable. Use the existing deterministic-safe-paths pattern (PBKDF2 + salt) for tree IDs |
| File IDs | Same deterministic-safe-paths pattern. `{file_hash}` or `{file_key_derived_id}` — never mutable |
| Read/write token separation | This is new. Current vault pointer model uses a single write_key. The folder model introduces two token types. This needs a new auth scheme — suggest: write token derives from vault_write_key + folder_id; read token derives from folder_id alone |
| API paths | Follow existing pattern: `/api/folders/{id}/...` — separate from `/api/transfers/` and `/api/vault/` |

### Open Questions for Human Decision

1. **Compare-and-swap implementation:** S3 conditional PUT (`If-None-Match: *` for create, `If-Match: {etag}` for update)? Or application-level compare in Lambda with a pessimistic lock? S3 conditional is simpler and correct but has eventual consistency nuances in multi-region.
2. **Tree snapshot ID scheme:** Timestamp-hash (as brief shows) or content-addressable (hash of tree content)? Content-addressable avoids duplicate trees for unchanged states; timestamp-hash is simpler to implement.
3. **Folder promotion path:** How does a Mode 3 folder become a vault? Is promotion atomic (a single operation that converts the head/tree/file structure to vault commits)? This is not needed for v1 but the structure should not prevent it.

## 2. Vault Publishing Convention (Doc 249) — ASSESSED

**Brief:** `v0.20.38__pattern__vault-publishing-convention.md`
**Verdict:** Clean, minimal, recommend adopting. No architectural concerns.

### Assessment

This is a single-commit metadata pattern that adds zero new infrastructure. It uses existing vault primitives (commit, push). The publication record is encrypted at rest (inside the vault). The benefit — discoverable publication history from a single vault ID — is genuine and frequently needed.

**Fit with existing commit-as-communication principle:** Perfect alignment. This is an extension of the same idea already in use for IFD v2.

### One Concern: Timing of Publication Commit

The publication commit must happen **after** the folder share is confirmed (server has accepted the zip upload). If the commit is created before the upload completes and the upload fails, the vault has a phantom publication record. Recommend:

1. Create the folder/share
2. Confirm upload success (S3 ETag or API response)
3. Only then commit the publication metadata to the source vault

This is a sequencing concern for Dev, not an architectural blocker.

## 3. Cross-Linking Tools from Main UI (Doc 240) — ASSESSED

**Brief:** `v0.20.37__dev-brief__cross-linking-tools-manifests.md`
**Verdict:** Strategically correct. Architecturally straightforward. Two parts of different complexity.

### Part A: Deep Links (Simple)

URL parameter passing between send.sgraph.ai and tools.sgraph.ai is straightforward. The challenge is security: the tools site reads `?source=simple-token:apple-banana-cherry` from the URL. This passes through browser history, server logs, and referrer headers.

**Architectural constraint:** The decryption key MUST stay in the URL hash (`#`) and never in query parameters (`?`). Deep links must follow this pattern:
```
https://tools.sgraph.ai/infographic#source=simple-token:apple-banana-cherry
```
Not:
```
https://tools.sgraph.ai/infographic?source=simple-token:apple-banana-cherry  ← WRONG: key in URL query
```
The `?` parameters can carry non-sensitive routing information (tool name, content type hint). The actual token with embedded key must go in the hash.

### Part B: Dependency Manifests (More Complex)

The manifest format proposed is sound. The concern is governance: who owns the manifest? Who updates it when a component version changes on tools.sgraph.ai? This becomes a QA coordination problem if not formalised.

**Recommendation:** The Librarian maintains the master dependency manifest list. Any component update on the tools site triggers a Librarian check of all manifests referencing that component. This is a process decision, not code.

### Part C: Component Retrofitting (Medium Complexity)

Replacing existing folder tree implementations with `sg-tree` (which is still PROPOSED — not yet built) is premature. This is the dependency chain problem: the brief describes consuming components that don't yet exist.

**Sequencing recommendation:** Do not attempt component retrofitting until `sg-tree` and VFS are code-complete and tested on tools.sgraph.ai.

## 4. Chrome Extension + Vault (Doc 238) — ASSESSED

**Brief:** `v0.20.37__dev-brief__chrome-extension-vault.md`
**Verdict:** Architecturally correct direction. One critical concern.

### Assessment

Using the Chrome extension's access to the user's live session (cookies) — without exporting those cookies — is the right security model. The vault as output channel is correct and consistent with existing architecture.

### Critical Concern: Vault Key in Extension

The extension must have a configured vault key. This vault key is sensitive (persistent write access to user data). Where does it live?

- **Option A:** User enters it manually in the extension settings (stored in Chrome's `chrome.storage.sync` — encrypted by the browser, not accessible to page scripts)
- **Option B:** User enters it on the tools site, posts it to the extension via `chrome.runtime.sendMessage`

**Option A is correct.** Chrome's `chrome.storage.sync` provides reasonable security for extension secrets. It is important that the vault key is stored there and NOT in localStorage of any web page.

**AppSec review needed** before implementing this. The extension runs with elevated permissions and the vault key represents persistent write access.

## 5. Serverless Playwright API (Doc 239) — ASSESSED

**Brief:** `v0.20.37__dev-brief__serverless-playwright-api.md`
**Verdict:** Sound design. Important architectural clarification on automation script loading.

### Assessment

The stateless design (clean session per request, vault as only persistence) is correct. Docker over Lambda is the right choice for Playwright (Chrome binary size, no time limits).

### Automation Script Loading

The brief says automation scripts are "loaded at runtime (from vaults, from S3, or passed in the API request)." Passing scripts in the API request is a code injection risk if the API is accessible beyond trusted agents.

**Architectural constraint:** Automation scripts must come from **trusted sources only** (vaults the service has been configured to read, or S3 paths with access control). Never accept scripts in the request body from an unauthenticated caller. The API must be behind auth even in local development mode.

## 6. Agentic Team Creator (Doc 243) — ASSESSED

**Brief:** `v0.20.38__dev-brief__agentic-team-setup-pack-creator.md`
**Verdict:** Architecturally elegant. Phase-gated well. One dependency gap.

### Assessment

The vault-to-vault pattern (template vault + context = output vault) is clean and reusable. Tool calls as phase transition signals is the correct way to implement multi-step LLM workflows (avoids the "UI state machine controlled by the LLM" anti-pattern).

**Dependency gap:** This tool requires the agentic chat UI with tool calling (from Section 21, doc 224 — `sg-agentic-loop`, `sg-tool-runner`) which is still PROPOSED. The team creator cannot be built without that foundation.

**Exception:** A simplified v1 could use vanilla LLM calls (without tool calling) and hardcode the phase transitions in the UI. This would allow the wizard to be built and tested while the full agentic component suite is developed in parallel.

## 7. Profile Page + Credit Activation (Doc 246) — ASSESSED

**Brief:** `v0.20.38__dev-brief__profile-page-credit-activation.md`
**Verdict:** Immediate priority. No architectural concerns. Clear security constraints.

### Assessment

The architecture is correct: server stores only non-sensitive data (balance, expiration, usage), all secrets remain in client localStorage. This is consistent with the zero-knowledge design.

**One clarification needed:** The brief says "If the human (or anyone) knows a user's access token, they can reach the server-side profile but must NOT see any confidential information." This is already guaranteed by the architecture (secrets are in localStorage, not on server). The concern is social engineering — users must understand that if they share their access token, others can see their balance/usage. This is a UX/communications concern, not an architectural one.

**Implementation note:** The credit expiration (~1 month) needs a server-side enforcement mechanism. The server must check expiration date on each token use and reject expired credits. This must be a server-side check, not client-side.

## 8. Audio Capture + Video Generation (Docs 244, 245) — BRIEF ASSESSMENT

**Verdict:** Architecturally sound but both depend on VFS (PROPOSED) for vault storage integration. Can be built with simple direct vault writes in v1.

### Audio Capture (Doc 244)
The segment-based recording approach is the correct solution to the data-loss problem. Each segment saved to localStorage immediately, synced to vault when online. The progressive enhancement pipeline (WASM → OpenRouter → document → infographic) is the right sequencing. Key architectural concern: the local WASM model is experimental — the brief correctly calls this out as "an experiment."

### Video Generation (Doc 245)
The client-side processing constraint (no server-side media processing) is architecturally significant. Canvas + MediaRecorder for assembly is the correct approach for v1. Text-to-speech via OpenRouter is new but straightforward (standard API call pattern). The revenue connection (10–20 LLM calls per video generation) is clear.
