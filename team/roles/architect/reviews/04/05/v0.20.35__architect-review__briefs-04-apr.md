## 1. Backend Storage Restructuring (`by-id-from__*` hierarchy)

**Source:** Doc 223 — `v0.20.33__dev-brief__backend-storage-restructuring.md`

### Assessment: Architecturally Sound

The flat-to-hierarchical migration is a necessary evolution. The `by-id-from__*` typed prefix pattern is elegant: self-documenting, safe to add levels incrementally, and no collision risk between grouping types. The design correctly separates concerns:

- **Organisational hierarchy** (who owns what, where data lives)
- **Access control** (unchanged — encryption remains the mechanism)

The "every level is optional, server fills defaults" principle is correct for a zero-knowledge system. Callers who know nothing about grouping still get fully grouped data. Callers who need explicit control (enterprise, multi-region) can provide values.

### Architectural Concerns

**Concern 1: Salt selection for derivation**
The brief uses `sha256(f"{salt}:{key_material}")` for the group ID derivation example. However, document 225 (deterministic-safe-paths) recommends PBKDF2 with 600,000 iterations for the canonical pattern. There is an inconsistency: the storage restructuring brief uses a single SHA-256 call, while the architectural primitive requires PBKDF2.

**Decision needed:** Which derivation method applies to storage grouping?
- If SHA-256: fast, acceptable for server-side grouping (not adversarially guessable from the outside), but inconsistent with the platform primitive
- If PBKDF2: consistent with the platform, but adds server-side latency on every upload/download (600,000 iterations per request, per grouping level)

**Recommendation:** Define two tiers:
1. **High-security tier** (PBKDF2, 600k): paths where the ID itself is a secret (e.g. per-user settings, shared communication channels)
2. **Organisational tier** (SHA-256 with context salt): paths where the ID is derived from already-known secrets (access tokens, decryption keys already travelling in the request) and the purpose is grouping, not concealment

Storage grouping belongs in tier 2. The security comes from the encryption, not the path obscurity.

**Concern 2: Migration complexity**
The migration brief describes a background migration with dual-path support (check grouped first, fall back to flat). This is correct. However, the migration script must handle:
- Transfers where the access token or decryption key is no longer available (cold metadata only)
- Partial migrations (crash recovery, idempotency)
- Validation that the moved data is intact before deletion of old path

The brief does not address these. Flag for the Dev review.

**Concern 3: PKI identity grouping (future)**
The `by-id-from__public-key/` level is described but not designed. PKI key derivation for paths needs to account for key rotation — if a user rotates their key pair, their old grouped data is under the old key's derived ID. Plan needed before implementing PKI grouping.

### Verdict

Approve the `by-id-from__*` hierarchy design. Flag the SHA-256 vs PBKDF2 derivation question as a decision that must be resolved before implementation. The migration plan needs hardening.

## 2. Deterministic-Safe-Paths Pattern (Architecture Primitive)

**Source:** Doc 225 — `v0.20.33__pattern__deterministic-safe-paths.md`

### Assessment: Excellent — Overdue Formalisation

This is the most architecturally important document in the batch. The pattern was already used informally in at least two places (SGit vault object IDs, backend storage grouping). Formalising it means:
- New features can be built against the pattern with predictable security properties
- The salt registry prevents accidental salt reuse (which would cause cross-context ID collisions)
- The implementation checklist gives developers a 6-step guide that prevents common mistakes

### Design Properties Confirmed

The PBKDF2 + 600,000 iterations specification is correct:
- **Deterministic:** Same inputs → same output. No database lookup required. Two parties sharing a secret can independently reach the same path.
- **Non-reversible:** The path ID cannot be used to recover the secret.
- **Enumeration-resistant:** The input space (vault keys, tokens) is enormous. Scanning for valid IDs is impractical.
- **Server-blind:** The server stores data at the path but cannot determine what the path represents or which user it belongs to.

### Architectural Concerns

**Concern 1: The salt registry must be enforced at code review**
The brief says to document all salts in "one place." This must be a code-level artefact (a constants file or config object), not just documentation. If the salt registry lives only in a markdown file, developers working quickly will invent new salts without checking it.

**Recommendation:** Create `sg_deterministic_ids.py` (or equivalent) with:
```python
class DeterministicPaths:
    SALT_VAULT_OBJECT   = "sg-vault-object"      # SGit vault objects
    SALT_GROUP_ACCESS   = "sg-access"             # Transfer grouping by access token
    SALT_GROUP_KEY      = "sg-key"                # Transfer grouping by decryption key
    SALT_GROUP_USER     = "sg-user"               # Transfer grouping by user identity
    SALT_GROUP_SERVER   = "sg-server"             # Transfer grouping by server/region
    SALT_USER_SETTINGS  = "sg-settings"           # Per-user preferences
    SALT_SHARED_SECRET  = "sg-shared"             # Shared secrets communication
    SALT_SEARCH_INDEX   = "sg-index"              # Encrypted search indexes
```

Any new use of the pattern must add a constant here. This makes salt collisions detectable at code review.

**Concern 2: Output length specification**
The brief recommends 16 hex chars (64 bits) for path segments, 32 hex chars (128 bits) for globally unique IDs. This needs clarification:
- 64-bit path segments are adequate for organisational grouping (collision probability negligible at expected scale)
- For globally unique IDs (transfer IDs, vault IDs), 128 bits is correct

The implementation should document which length applies where. The current Transfer_Id uses 12 chars (cryptographically random, not derived). This is a different type of ID (random, not deterministic) and should not be confused with derived IDs from this pattern.

**Concern 3: Audit logging of derived paths**
If a user disputes access or a security incident occurs, operations teams need to trace which access token or vault key corresponds to which S3 prefix. The pattern is designed to be server-blind, but in practice the admin has the access token (they issued it) and can re-derive the path. This is correct — the design provides operational traceability without storing linkage data.

### Verdict

Approve the pattern formalisation. Strongly recommend creating the constants file as the salt registry. The document should be linked from the architecture index.

## 3. Agentic LLM Component Suite

**Source:** Doc 224 — `v0.20.33__dev-brief__agentic-llm-tool-calling.md`

### Assessment: Well-Designed; Sandbox Security Requires Rigour

The six-component architecture is clean and follows the established SG component patterns (Web Components, event-driven, VFS integration). The choice to run the full agentic loop in the browser is architecturally correct for the zero-knowledge model — no server sees what tools are called or what data is processed.

### Architectural Concerns

**Concern 1: sg-sandbox is the critical path**
The sandbox is the most security-sensitive component in the entire platform. The brief specifies:
- Sandboxed iframe (no same-origin, no parent DOM access, postMessage only)
- Web Worker with timeout and memory limits
- Zero capabilities by default, opt-in per tool
- No network access from inside the sandbox unless explicitly provided

This specification is correct. However, the browser security boundary has known weaknesses:
- postMessage channel can be abused if the parent doesn't validate message origins
- Side-channel attacks (timing, shared memory) remain theoretically possible in Web Workers
- `import_libs` capability (allow numpy, lodash, etc.) introduces supply chain considerations if libraries are fetched from CDN at execution time

**Recommendation:**
1. Specify that `import_libs` capability only permits libraries already available in the component's local bundle (no network imports from sandbox)
2. Specify strict postMessage origin validation (only accept messages from the expected sandbox origin)
3. Add an AppSec review gate before the sandbox component ships to any environment

**Concern 2: Gemma 4 thinking token management**
The brief correctly identifies that thinking tokens must NOT be stripped between tool calls within the same turn, but SHOULD be stripped between turns. This is subtle state management in `sg-agentic-loop`. Incorrect handling degrades model performance.

**Recommendation:** Define a test case in the acceptance criteria: verify that a two-step tool call sequence (list_folder → read_file) preserves thinking tokens between steps and strips them before the second user message.

**Concern 3: Tool definition schema and `response_format` conflict**
The `sg-json-sender` component needs to support both:
1. The `tools` parameter (for tool definitions)
2. The `response_format` parameter (for structured JSON output without tools)

These are not compatible in the same request for all models. The component should enforce mutual exclusivity: one mode selected per request, not both simultaneously.

**Concern 4: VFS integration in tools**
Built-in tools (`search_vault`, `read_file`, `write_file`) operate through the VFS. But the VFS itself (from the 04/03 arch brief) is PROPOSED — none of the providers are implemented. The agentic tools depend on a dependency that does not yet exist. This is a sequencing risk: the tools cannot be fully tested until VFS has at least a memory provider.

**Sequencing recommendation:** Implement the agentic components against a simple memory VFS stub initially, then integrate with the full VFS provider stack as it ships.

### Verdict

Approve the component architecture. Flag the sandbox security model for mandatory AppSec review before production. Clarify the thinking-token strip policy. Document VFS dependency explicitly.
