## 1. Vault as Collaboration Primitive (doc 278)

### Architectural Assessment: SOUND

The brief's core claim is correct. The vault, with tree-only loading and immutable block caching (both EXISTS since `claude/vault-dev-setup-TSfU2`), delivers better collaborative performance than the proposed server-side zip versioning architecture would have. The SGit dual branch model (named + clone branches, EXISTS) provides conflict isolation that the server-side approach would have had to build from scratch.

### Supersession Chain

The brief explicitly closes the design chain that started with the 05 Apr and 08 Apr briefs:

| Brief | Status Change |
|-------|--------------|
| `v0.20.33__dev-brief__editable-collaborative-folders.md` (05 Apr) | Mode 1 EXISTS. Modes 2–4 now resolved by vault. |
| `v0.20.38__arch-brief__server-side-zip-folder-versioning.md` (08 Apr) | **SUPERSEDED.** Vault provides this capability. |
| `v0.20.38__brief__folder-view-vs-vault-explainer.md` (08 Apr) | Simplified: vault = live workspace, share token = snapshot. |

This is a clean architectural decision with clear rationale. No concerns.

### 4 Remaining Vault Additions

| Addition | Architectural Complexity | Notes |
|----------|--------------------------|-------|
| 1. Instant vault creation on upload | Low–Medium | Requires a "create vault" flow early in upload, then incremental commit. Main complexity: UX state management (showing partially-uploaded vault). |
| 2. Upload branches | Medium | A new branch type alongside the existing clone branch model. Design question: should upload branches follow the same HMAC-derived naming convention as clone branches? Or use a fixed name like `upload/in-progress`? This is **open decision #6** per the daily brief. |
| 3. Read-only access control cleanup | Low | `vault._writeKey` guard fix is already in progress per the 04/11 audit. This is tidying rather than new design. |
| 4. Auto-refresh / change detection | Low | A periodic `getBehindCount()` poll. The infrastructure already exists; this is just wiring a timer. Consider: use `setInterval` vs `visibilitychange` event to pause polling when tab is not focused. |

### Open Decision #6: Upload Branch Strategy

**Options:**
- **A. Sequential commits to main:** Simpler. Files appear in the main branch one by one during upload. Risk: partially-uploaded state is visible to all readers.
- **B. Dedicated upload branch, merge on completion:** Cleaner for readers (atomic appearance). Adds upload branch lifecycle management. The brief's acceptance criterion #3 specifies this option.

**Recommendation:** Option B (upload branch), consistent with the brief. The dual branch model already provides the conceptual infrastructure. The key invariant to maintain: upload branch merge should be a fast-forward to avoid creating merge commits in the vault history.

## 2. PlaybookLM Companion Tools (doc 277)

### Shared Architecture: SOUND

All three tools reuse existing components: `sg-llm-infographic`, `sg-llm-events`, `sg-llm-request` from `tools.sgraph.ai`. The brief's instruction to expose a standard JS API (`getManifest`, `getMethods`, `getSkills`) on all three tools aligns with the "API is the tool" principle established in the 08 Apr arch brief (`v0.20.38__arch-brief__javascript-api-tool-primitive.md`).

### Tool 1: Infographic Generator (Simple Mode)

Architecturally straightforward. The brief asks for "integration with the PlaybookLM capture and bundling workflow" — which requires the PlaybookLM pipeline to be buildable. The pipeline spec exists (docs 252–263 territory) but the pipeline itself is PROPOSED. **Dependency:** Tool 1 is blocked on the PlaybookLM pipeline being built first, or it needs to define its own simpler bundling convention.

**Concern:** The brief says "this tool already exists in a basic form" — this appears to refer to the infographic tool at `dev.tools.sgraph.ai/en-gb/infographic-gen/`, which EXISTS. The distinction is the bundling and vault-save integration, which are PROPOSED. The Architect should clarify this boundary before Dev begins work.

### Tool 2: Infographic Editor

The most architecturally novel of the three. Three notable design aspects:

**LLM Analysis Loop:** Analysis → Change Selection → Execution → Verification is a structured multi-step LLM workflow. Each step has distinct state. The tool's architecture needs a state machine, not just a sequence of UI screens. The `sg-llm-events` event-streaming pattern can carry state transitions.

**Browser-Side Tool Execution:** The brief explicitly flags this as "potentially the first experiment with running tools in the browser directly from the LLM prompt." This is a frontier capability. Architecturally, it implies the LLM can emit structured commands (e.g., `{ tool: "redact", region: [x,y,w,h] }`) that the browser interprets and executes on a Canvas. This would be a new pattern — LLM as orchestrator of browser tools. It deserves its own arch brief before implementation.

**Verification by Comparison:** Sending old + new images to an LLM for comparison is straightforward with multimodal models (Gemini). The cost transparency requirement (per-step cost breakdown) maps naturally to the `sg-llm-events` stream if cost data is included in events.

### Tool 3: Simple PlaybookLM

Architecturally this wraps the full PlaybookLM pipeline in a simplified UI shell. The "switch to advanced mode" requirement is a clean capability: preserve intermediate artefacts, expose them in the full PlaybookLM UI. This requires the full pipeline to be built first. **Dependency:** Blocked on PlaybookLM pipeline.

### API Design Concern

The brief specifies `getManifest`, `getMethods`, `getSkills` on all three tools. The infographic generator at `dev.tools.sgraph.ai` already exposes `__tool.generate()`, `__tool.connect()`, `__tool.setTemplate()`, `__tool.setSystemPrompt()`. The new standard API should supersede this, not run in parallel. The Architect should produce an API contract for `window.__tool` that works across all tool pages before these three tools are built.
