## 1. Vault Upload Beta — Architectural Assessment

### What's proposed

- Vault creation triggered synchronously on folder drop
- Batched commits (10–20 files) with `.sg-upload-status.json` as progress state
- Placeholder files for large queued items
- Beta toggle in the existing SG/Send upload UI

### Soundness

The design is architecturally sound. It builds directly on established vault primitives: vault creation, incremental commits, tree-only loading, and the dual branch model. No new server-side capabilities are needed.

### Concerns

**Upload branch strategy (Open Decision #6) is unresolved.** The vault upload beta creates a vault on the first file drop and commits in batches. If the upload fails partway, the vault exists with partial content. The brief does not address cleanup or rollback. This is acceptable for a beta but needs a decision before any production promotion.

**Browser tab close = upload loss.** The brief acknowledges this: "The upload runs client-side, so the user must keep the browser tab open until complete." A service worker or background upload capability would address this, but both are deferred. For beta, the tab close warning is the only mitigation. This is architecturally acceptable for a time-limited beta but is a user experience risk.

**`.sg-upload-status.json` creates a coupling.** The recipient's UI is expected to render this file as a progress banner. This introduces a convention dependency between the uploader and the browse UI. The convention should be documented as a first-class vault convention (alongside the existing `_page.json` convention), not as an informal agreement.

### Recommendation

Proceed with the beta. Resolve Open Decision #6 (upload branch strategy) before any production promotion. Document `.sg-upload-status.json` as a formal vault convention.

## 2. Playwright Service — Three-Layer Architecture Assessment

### The v4 Architecture (doc 294) vs the Implementation Brief (doc 295)

The v4 Agentic FastAPI Runtime architecture and the first-pass implementation brief are well-calibrated. Doc 294 is explicit: "This is direction of travel. It is not what you implement right now." Doc 295 narrows scope appropriately: everything stays in one repo, no PyPI publishing, no Docker Hub publishing of Layer 1 artefacts.

### The Three Axioms

**Axiom 1 (Statelessness):** Architecturally correct and consistent with Lambda deployment. The "no upload-then-reload two-call patterns" rule is well-stated — it prevents a class of integration bugs and makes the admin API safer to expose.

**Axiom 2 (Least privilege by declaration):** `capabilities.json` is a stub in the first pass. That is the right decision. The value is in establishing the format and surfacing it via `/admin/capabilities` from day one. Actual enforcement (network sidecar, lockdown layers) is deferred and correctly labelled as future workstreams.

**Axiom 3 (Self-description via standard primitives):** The SKILL file pattern (`SKILL-human.md`, `SKILL-browser.md`, `SKILL-agent.md`) + OpenAPI + `/admin/manifest` is well-designed. Three reading modes for three audiences (operator, browser, agent) is a clean design. The `/admin/skills/{name}` endpoint serving filesystem reads at request time is simple and correct.

### Layer Dependency Rule

Doc 295 states: "`agentic_fastapi` does NOT import from `agentic_fastapi_aws` or `service`. `agentic_fastapi_aws` does NOT import from `service`. Verify with grep before you merge."

This is the critical rule for making the future repo split mechanical. Dev must enforce this from day one — a single import violation early is much cheaper to fix than a web of accidental dependencies discovered at split time.

### Open Questions 7–9 Still Unresolved

| # | Question | Architectural Impact |
|---|----------|---------------------|
| 7 | Chrome profile persistence | Affects statelessness (Axiom 1). Fresh-per-session is the correct answer for axiom compliance. Vault-stored profile as optional input is acceptable (it's declared input, not in-process state). |
| 8 | Timeout routing (Lambda vs Fargate auto-selection) | If the service auto-routes, it becomes stateful (it has to track which backend handled a session). Recommend: caller specifies target, service does not auto-select. |
| 9 | Concurrent sessions per container | One-per-container is the correct answer for Axiom 1 compliance. Multiple browsers per container share state implicitly (memory, tmp, Chrome user data). Avoid. |

**Recommendation:** Resolve OD #7 as fresh-per-session default (vault-stored profile is optional input). Resolve OD #8 as caller-specified (no auto-routing). Resolve OD #9 as one-per-container.

## 3. Auth MVP — Client-Side Claim Verification

### The Core Claim

Doc 292: "All client-side. No server-side code. This is non-negotiable and consistent with the entire SG/Send architecture."

This claim requires scrutiny. Social login via OAuth involves redirects. The OAuth callback URL must be a real URL, which requires a page to exist at that URL. The page itself can be static (no server-side processing), but the Cognito/Firebase hosted UI does make server-side calls to the identity provider. The claim holds if: (a) SG/Send serves only static assets and (b) no SG/Send server-side code processes tokens.

**AppSec should review:** Specifically, whether Cognito custom user attributes (vault keys) are readable client-side without a server-side intermediary. Cognito's JavaScript SDK does support reading user attributes from the browser via the identity token, but the constraints on attribute size and read permissions need to be confirmed.

### Option A vs Option B

**Option A (user attributes):** Simplest. May hit size limits if a user has many vault keys.

**Option B (encrypted bundle in SG/Send):** More robust for scale. The identity provider stores only a pointer + master decryption key. SG/Send stores the encrypted bundle. Neither can read the other's portion. This is architecturally superior for a zero-knowledge claim.

**Recommendation:** Build Option A for the MVP. Design the `<sg-user-data>` component's interface to be storage-agnostic, so Option B can be substituted without changing callers.

### Cross-Provider Portability (Research Question 5)

If a user logs in via Google on Cognito today and Firebase tomorrow, migration requires transferring encrypted vault key bundles. There is no standard for this. This should be documented as a known limitation in the MVP, not deferred silently.

## 4. Healthcare MVP — Architectural Proof Assessment

### The Claim

"Applications can be entirely client-side, with all logic running in the browser and the only server-side data being encrypted ciphertext."

### Is This Claim Verifiable?

Yes, and the MVP is the right vehicle. Six of the eight listed components are new builds; two already exist (vault creation API, doctor collaboration via Push/Pull). The client-side claim is verifiable by network inspection: if all inter-component calls stay in the browser and only ciphertext transits the network, the claim holds.

**LLM integration is the exception to watch.** The brief says "Send anonymised/structured data to LLM (via OpenRouter)." This is client-side JavaScript calling OpenRouter's API directly. The data sent is "anonymised/structured" — but the brief does not define anonymisation. For a healthcare demo, the questionnaire answers must not include identifying information in the LLM call. AppSec and DPO should review the LLM prompt construction before the MVP is shown to the healthcare partner.

### Patient-Doctor Access Model

The brief proposes:
- Patient: vault key (read-write)
- Doctor: shared vault key (read-write, dual branch model)
- Published summary: read-only share token

This is architecturally clean and uses existing vault primitives. The dual branch model (Phase 2 + Phase 3, shipped in commit `710c96d` and `192978f`) handles concurrent edits. No new server-side capabilities needed.

### Connection to Identity

The brief correctly notes that identity is a precondition for production use. The manual key exchange approach for the MVP (patient sees key on screen, copies it, shares via secure channel) is acceptable for a demo but is not a production pattern. The auth MVP (doc 292) is the production path.

## 5. Cross-Cutting Architectural Observations

### Vault as Universal Primitive

Across all four themes, the vault is used as: upload destination, collaboration layer, config input for Playwright, output store for Playwright, data layer for healthcare, package for agent communication ("dev packs"). This convergence is a strong architectural signal. The vault's dual branch model, tree-only loading, and immutable caching are load-bearing across multiple product surfaces.

**Risk:** As the vault becomes more central, any regression in vault behaviour affects everything. The test suite should reflect this centrality. QA should ensure vault unit tests cover all the use cases now depending on it.

### Docker Hub Namespace — Unresolved (OD #14)

All three container briefs reference `sgraph/` as the Docker Hub namespace. This needs to be resolved before any images are published. Docker Hub namespace registration should be done early to avoid squatting.

### SKILL Files as Agent Interface Standard

The SKILL files pattern (doc 294, 295) is potentially the most reusable architectural decision in this batch. If adopted consistently across all SGraph services, it creates a discoverable interface layer for agents — any agent can call `/admin/skills/agent` on any SGraph service and get self-describing documentation.

**Recommendation:** Adopt the SKILL files pattern as a project-level standard. Document it in the Architect role's guides. Require it for all new services.
