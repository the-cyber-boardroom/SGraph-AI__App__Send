## Architectural Assessment

### What Is Being Moved

The `sgraph_ai__website/` directory contains the sgraph.ai marketing website. This is:
- Static HTML, CSS, JS — no server-side code
- Deployed via GitHub Actions → S3 → CloudFront
- IFD-versioned at `v0/v0.2/v0.2.0/` for Web Components
- Zero Python dependencies on the Send Lambda (the only soft dependency is a version string read from `sgraph_ai_app_send/version` — the dev pack correctly notes this should become independent in the new repo)

This is a textbook extract. There is no coupling to break.

### Decision Analysis: The 10 Default Decisions

The dev pack lists 10 architectural decisions. All are appropriate:

| # | Decision | Assessment |
|---|----------|------------|
| 1 | Flatten directory tree (drop `sgraph_ai__website/` prefix) | Correct — the repo IS the website |
| 2 | Independent `version` file (init at `v0.0.1`) | Correct — website has its own release cadence |
| 3 | Move deploy workflow to new repo | Correct — CI lives with code |

| 10 | Reuse existing GitHub Actions secret names | Correct — avoids coordination overhead |

### Open Question Q2: Component Sharing Strategy

**This is the highest-priority architectural concern.**

The `sg-site-header` Web Component (v1.0.6, at `sgraph_ai__website/v0/v0.2/v0.2.0/_common/js/components/sg-site-header/`) currently lives in the Send repo. After extraction:
- The website will own this component in the new repo
- The Send app (`sgraph_ai_app_send__ui__user/`) does NOT use `sg-site-header` (Send has its own `send-header.js`)
- The Tools site (`tools.sgraph.ai`) currently loads `sg-site-header` from `sgraph.ai` CDN

**Architecture recommendation:** The new repo becomes the canonical source for `sg-site-header` and the 10 homepage section components. Tools and Send load them from the CDN. No package publish needed (they're Web Components loaded via `<script type="module">`). This is the existing pattern and should not change.

**Risk if not addressed:** If the `SGraph-AI__Website` repo is unclear about component ownership, future development sessions may try to put `sg-site-header` in `SGraph-AI__App__Send` again. The dev pack should note this explicitly — though Open Question Q2 invites the human to confirm.

### Open Question Q1: Default Branch

The dev pack asks whether the new repo should use `dev` (matching Send) or `main` (GitHub default).

**Architect recommendation: `dev` + `main`**, matching the Send repo pattern:
- `dev` — active development, CI deploys to `main.sgraph.ai` / `dev.sgraph.ai`
- `main` — production, CI deploys to `sgraph.ai`

Rationale: The website follows the same deploy patterns as Send. Consistency reduces cognitive load for the Explorer/Villager teams.

### CI Architecture

The existing `deploy-website.yml` workflow is well-designed: validate → S3 sync → CloudFront invalidation → smoke test. Moving it to the new repo is low-risk. The 5 required secrets are already provisioned on the Send repo; they must be explicitly added to the new repo.

**Risk:** The workflow currently triggers on `push to main on sgraph_ai__website/**` paths. In the new repo, the trigger should change to `push to main` (all paths). Phase 3, Step 3 (`05_path-rewrites-and-cross-references.md`) correctly identifies this path rewrite.

### Independent Versioning

The dev pack initialises the new repo's `version` file at `v0.0.1`. This is correct. The Send CI pipeline increments `sgraph_ai_app_send/version` independently. Website releases now have their own cadence.

**Soft dependency note:** The old `sgraph_ai__website/` included a `scripts/inject_build_version.py` that read from `sgraph_ai_app_send/version`. After extraction, this becomes unnecessary. The dev pack correctly marks this as "only if needed" and leaves it as a conditional copy. Confirm with Dev that no current CI workflow uses this script in a meaningful way before deleting it.

## Architectural Concerns Not in the Dev Pack

### 1. `sgraph_ai__website/v0.2.0/` vs flat root pages

The website has two sets of pages:
- Flat root: `sgraph_ai__website/en-gb/index.html` etc. (the canonical serving URL)
- IFD versioned: `sgraph_ai__website/v0/v0.2/v0.2.0/en-gb/` (the Web Component dev tree)

Both need to be in the new repo. The dev pack covers this in Phase 2. Confirm that both trees point to the same CloudFront origin path (they do, since both are in the same S3 bucket).

### 2. Journalist Site (`team/roles/journalist/site/`)

Decision #6 correctly excludes this. The journalist Jekyll site is an internal comms tool. However, the journalist site imports some CSS from `sgraph_ai__website/_common/`. After extraction, if the journalist site is ever used, it may need its CSS import paths updated. Flag for the Journalist role.

### 3. Production Traffic Continuity

Phase 5 (cutover) keeps both repos pointing to the same S3 bucket and CloudFront distribution until QA passes. This means both `deploy-website.yml` (Send repo) and the new repo CI could race to deploy simultaneously during the transition window. The cutover plan should disable the Send repo workflow first, then confirm the new repo CI is working, before any traffic is switched.
