## Executive Summary

Three infrastructure research documents from the `ollama__gemma4/` subdirectory clarify the local LLM backend landscape for the SGraph Send agentic component suite. The documents are well-researched and the conclusions are sound. The key architectural takeaways are:

1. **`mlx-openai-server` is the correct local backend choice today** — OpenAI-compatible, supports Gemma 4 E4B with 128K context, handles audio/video, runs at localhost
2. **LiteRT-LM is not viable for desktop API use today** — no HTTP server, 4K context cap, E2B only, Python SDK day-1
3. **Ollama has a confirmed audio/video input blocker** — no mechanism exists in any API surface; this must be handled explicitly in `_buildOllamaBody`

These documents do not change the proposed feature set but provide the implementation-level context needed to correctly design the LLM provider abstraction layer.

## Architectural Assessment

### 1. Provider Abstraction Model

The agentic component suite (Section 21, proposed in doc 224) requires a provider abstraction that insulates the `sg-agentic-loop` and related components from backend-specific differences. Documents 232–234 together define the constraints that abstraction must handle.

**Recommended provider interface shape:**

```
interface LLMProvider {
  id: string                    // "openai" | "ollama" | "mlx" | "litert" | ...
  baseUrl: string               // HTTP endpoint root
  protocol: "openai" | "ollama" // wire format
  capabilities: {
    text: boolean
    image: boolean
    audio: boolean              // Ollama: false; mlx/openai: true
    video: boolean              // Ollama: false; mlx with multimodal: true
    toolCalling: boolean
    streaming: boolean
    maxContextTokens: number    // Ollama/mlx: 128K; LiteRT: 4K
  }
  models: string[]
}
```

**Why capabilities must be explicit:** The `sg-agentic-loop` will receive requests with mixed content types (text + image + audio). It must route to a backend that supports all content types present. If the user selects Ollama and the prompt contains audio, the loop must either reject the request with a clear error or re-route to an audio-capable backend. This must be an architectural decision at the provider abstraction level, not scattered across individual components.

**The routing decision:**
- **Recommend:** Fail-fast with a clear, actionable error message. "This request contains audio content. Ollama does not support audio input. Switch to mlx-openai-server or OpenAI API."
- **Do not:** Silently drop audio content parts and send the rest. This produces confusing model behaviour.
- **Do not:** Auto-reroute without user consent. That changes the privacy/cost model the user chose.

### 2. OpenAI Protocol as Lingua Franca

Document 232 makes explicit what should be the architectural principle for all LLM provider integrations: **the OpenAI wire protocol is the integration surface, not the underlying runtime**. This has a clean architectural implication:

- The `openai` provider and the `mlx` provider share a single request builder (`_buildOpenAIBody`)
- The `ollama` provider uses a separate builder (`_buildOllamaBody`) because the native API uses a different message format
- The `litert` provider (future) would need a translation wrapper around the Python SDK

This is already implicit in the briefs but should be made explicit in the provider abstraction design: **only one message builder per wire protocol, not per backend vendor**.

### 3. LiteRT-LM: Architecture Decision — Not Viable Today

Document 233's assessment is accurate. The key architectural blockers for LiteRT-LM in the SGraph Send context:

| Blocker | Severity | Likely Resolution Timeline |
|---------|----------|--------------------------|
| No HTTP server | Blocking | Unknown (not announced) |
| 4K context cap | Blocking for most use cases | When E4B ships in `.litertlm` format |
| E2B only | Moderate (E2B is capable) | Soon (E4B model conversion is straightforward) |
| Python SDK day-1 | Moderate | 3–6 months for stability |
| No OpenAI compat | Blocking for current architecture | Requires custom FastAPI bridge |

**Architectural decision:** Do not integrate LiteRT-LM as an LLM provider at this time. Add it to the provider roadmap with a gating condition: "when LiteRT-LM ships a built-in HTTP server with OpenAI-compatible endpoints." Until then, the implementation overhead (custom bridge + unstable SDK) is not justified.

**Exception:** If a mobile-native SGraph Send app is ever built (iOS/Android), LiteRT-LM is the correct choice for on-device inference. The provider abstraction should be designed to support this future case without requiring a rearchitecture.

### 4. Ollama Audio/Video — Explicit Error Contract

Document 234 establishes a clear constraint. The architectural contract for `_buildOllamaBody`:

```
PRECONDITION: messages array contains no audio or video content parts
POSTCONDITION: Ollama-format request body
ON VIOLATION: throw ProviderCapabilityError("Ollama does not support audio/video input")
```

This is not a UI concern — it belongs in the provider abstraction layer. The `sg-agentic-loop` must check provider capabilities before dispatching a request and fail before the request is constructed.

**TODO placement:** The code stub for the future `audio` field should be placed in `_buildOllamaBody` at the detection point, referencing `ollama/ollama#11798`. The stub should be a single conditional that can be activated with a feature flag once the upstream PR lands.

### 5. CORS Architecture for Browser-Sourced Requests

Document 232 identifies a CORS concern: when the SGraph Send frontend app (at `send.sgraph.ai` or `localhost:3000`) makes fetch requests to `mlx-openai-server` at `localhost:8000`, CORS headers are needed. Options:

| Option | Pros | Cons |
|--------|------|------|
| `mlx-openai-server --host 0.0.0.0` + server CORS headers | Simple | Exposes server to network |
| Use `mlx_vlm.server` (has built-in CORS) | Built-in | Different binary |
| Proxy through a local FastAPI app | Full control | Extra component |
| Dev: Vite/webpack proxy; Prod: not applicable | Works in dev | N/A for production (server is local) |

**Recommendation:** This is a local development concern only — the mlx server runs on the same machine as the browser. The cleanest solution is `mlx_vlm.server` (lighter, CORS built-in) for development and direct API integration via browser native `fetch()` with appropriate CORS configuration. Document this as a dev setup note, not an architectural decision.

## Concerns and Flags

### Flag 1: Model Compatibility Matrix is Load-Bearing

The model compatibility matrix (proposed in doc 224, Section 21) is more critical than initially scoped. It is not just documentation — it must be machine-readable and consulted at routing time. The architect recommends the matrix be a JSON/JS object in code (not a markdown table) so the `sg-agentic-loop` can query it at runtime.

### Flag 2: Audio Max 30s Constraint

Document 232 notes that Gemma 4 E2B/E4B audio input is capped at 30 seconds. This is a model-level constraint, not a provider-level constraint. The provider abstraction should expose `maxAudioSeconds: number` alongside other capability flags. The `sg-agentic-loop` must enforce this before sending and surface a useful error if exceeded.

### Flag 3: Thinking Token Management

Document 232 notes Gemma 4 supports a `think` parameter and thinking tokens. The proposed policy from doc 224 (Section 21) — "preserve thinking tokens within multi-step tool sequences, strip between turns" — is confirmed as the right approach for Gemma 4. This policy must be encoded in the `mlx` provider's streaming response handler, not in the `sg-agentic-loop` itself.

### Flag 4: LiteRT-LM Supply Chain Advantage

For any future compliance, security audit, or enterprise deployment context, the LiteRT-LM supply chain advantage (100% Google-signed artefacts, no community intermediaries) may be a selling point. The Alchemist/Town Planner team should be aware that this option exists. It is not relevant to the current development cycle but is relevant to the investor materials narrative about on-device privacy guarantees.

## Decision Record

| Decision | Rationale |
|----------|-----------|
| Use `mlx-openai-server` (or `mlx_vlm.server`) as the local LLM backend | Most practical today; OpenAI-compatible; supports Gemma 4 E4B + audio/video |
| Do NOT integrate LiteRT-LM as a provider today | No HTTP server; 4K context cap; day-1 SDK. Revisit when HTTP server ships. |
| Fail-fast on Ollama + audio/video requests | Clear error is better than silent degradation or unauthorised re-routing |
| Provider capabilities must be machine-readable at runtime | Required for `sg-agentic-loop` routing decisions; not just documentation |
| Thinking token strip policy lives in the MLX provider, not the loop | Single responsibility; loop stays protocol-agnostic |
