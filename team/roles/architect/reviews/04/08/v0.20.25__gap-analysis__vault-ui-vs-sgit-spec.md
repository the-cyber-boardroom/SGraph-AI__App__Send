## What's Missing (6 capabilities needed for full SGit parity)

### 1. EDIT FILES (in-browser)

**What:** Edit text/markdown/json files inline and track the modification.

**Current state:** The old v0.1.4 vault-shell had an edit mode in `_renderFileInTab()` with textarea, Ctrl+S save, and `vault.updateFile()`. This was removed when we switched to the shared Browse component.

**What's needed:**
- Add Edit/Save buttons to the Browse file action bar when `dataSource.writable === true`
- On save: call `dataSource.saveFile(folderPath, fileName, bytes)` which calls `vault.updateFile()`
- The VaultDataSource already has `saveFile()`, `renameFile()`, `deleteFile()` methods

**Crypto:** AES-256-GCM encryption of new blob (already in `vault.updateFile()`)

**API:** `POST /api/vault/write/{vault_id}/{path}` (existing pointer API, already used by `vault.addFile()`)

**Effort:** Small - wire edit buttons into Browse action bar, delegate to VaultDataSource

---

### 2. ADD FILES (upload to vault)

**What:** Upload new files into the vault, create folders.

**Current state:** The Upload button exists in the header but shows a placeholder message. The `vault-upload` component from v0.1.4 exists in the codebase but isn't wired to the Browse-unified flow.

**What's needed:**
- Wire the Upload button to open a file picker or drag-and-drop zone
- Call `vault.addFile(folderPath, fileName, fileData)` 
- Call `vault.createFolder(folderPath)` for new folders
- Refresh the Browse tree after upload

**Crypto:** AES-256-GCM encryption of new blob (already in `vault.addFile()`)

**API:** `POST /api/vault/write/{vault_id}/{path}` (existing)

**Effort:** Small - the vault methods exist, just need UI wiring

---

### 3. CREATE CLONE BRANCH (with local private key)

**What:** When opening a vault via simple token or clone, create a local clone branch with its own EC P-256 key pair. The private key stays local (never pushed). This is the cryptographic identity for this browser session.

**Current state:** Not implemented. The vault UI opens in read-only mode on the named branch. No clone branch is created.

**What's needed:**
- Generate EC P-256 key pair in browser (Web Crypto API: `generateKey({name: 'ECDSA', namedCurve: 'P-256'})`)
- Store private key in IndexedDB or localStorage (encrypted with read key)
- Derive clone branch ref file ID: `HMAC(readKey, "sg-vault-v1:file-id:branch-ref:{vault_id}:{branch_name}")`
- Create clone branch ref on server pointing to current HEAD commit
- Track clone branch state: ahead/behind named branch

**Crypto:**
- EC P-256 key pair generation
- HKDF for branch-specific key derivation
- HMAC for deterministic clone ref file ID

**API:** `POST /api/vault/batch/{vault_id}` with write operation for new clone ref

**Effort:** Medium - new crypto, key storage, branch state tracking

---

### 4. CREATE COMMITS

**What:** After editing files, create a commit object (tree + blob + commit) that records the changes.

**Current state:** `vault._commit(message)` already exists and works - it builds tree entries, creates tree objects, creates commit objects, and updates the HEAD ref. It's used by `addFile()`, `updateFile()`, `removeFile()`, `renameFile()`.

**Gap:** Currently commits update the named branch ref directly. With clone branches, commits should update the clone branch ref instead, and push should sync clone -> named.

**What's needed:**
- When clone branch exists, `_commit()` should update clone ref, not named ref
- Track commit history on clone branch (ahead of named)
- Commit message encryption (currently plaintext in v0.1.4 code)

**Crypto:** AES-256-GCM for tree/blob/commit encryption, message_enc for commit messages

**API:** None (commit is local - objects written to server via existing pointer API, but ref update goes to clone ref)

**Effort:** Medium - modify `_commit()` to support clone vs named branch targeting

---

### 5. PUSH (clone branch -> named branch)

**What:** Upload new commits from clone branch to named branch on server. This is how changes become visible to other users.

**Current state:** Not implemented. All writes go directly to the named branch (no clone/push separation).

**What's needed (per SGit spec):**
1. Compute delta: objects in clone branch not in named branch
2. Re-encrypt objects from clone key to named key (if using separate keys)
3. Upload new objects via batch API
4. Update named branch ref with compare-and-swap (CAS)
5. Fast-forward clone ref to match

**Crypto:**
- Object re-encryption (clone key -> named key) if keys differ
- CAS (write-if-match) on named branch ref for concurrent push safety

**API:**
- `POST /api/vault/batch/{vault_id}` with write + write-if-match operations
- `POST /api/vault/presigned/initiate/{vault_id}` for large blobs (>4 MB)

**Effort:** Large - delta computation, re-encryption, batch upload, CAS ref update

---

### 6. PULL (named branch -> clone branch)

**What:** Fetch latest commits from named branch (other users' changes) into clone branch.

**Current state:** The Refresh button re-opens the vault from scratch (`SGVault.open()`), which reads the latest HEAD from named branch. But it discards any local state - there's no merge or fast-forward.

**What's needed:**
1. Fetch named branch ref from server
2. Walk commit chain from named ref to find common ancestor with clone
3. Download missing objects
4. Fast-forward clone ref (if no local commits diverge)
5. Or: merge (create merge commit with two parents) if both sides have commits
6. Re-render Browse tree

**Crypto:** AES-256-GCM decryption of new objects, potential re-encryption to clone key

**API:** `POST /api/vault/batch/{vault_id}` with read operations

**Effort:** Large - commit graph walking, conflict detection, merge strategy

## Implementation Priority

| Priority | Capability | Effort | Unlocks |
|---|---|---|---|
| **P0** | Edit files | Small | Users can modify vault content from the browser |
| **P0** | Add files (upload) | Small | Users can add new files + save LLM infographics |
| **P1** | Create clone branch | Medium | Cryptographic identity for browser sessions |
| **P1** | Create commits | Medium | Version history for browser edits |
| **P2** | Push | Large | Share browser edits with CLI users and other browsers |
| **P2** | Pull | Large | Receive CLI users' changes in the browser |

**Recommended approach:**
- **Phase 1 (now):** Edit + Upload - these work with the existing `vault.updateFile()`/`addFile()` which write directly to the named branch. No clone branch needed yet.
- **Phase 2:** Clone branch + commits - add proper version control. Edits go to clone branch, push syncs to named.
- **Phase 3:** Push + Pull - full bidirectional sync with CLI.

---

## Key Architectural Decision

The SGit spec describes a **two-branch model** where the browser would create a clone branch (like `sgit clone` does) and all edits go to the clone. Push syncs clone -> named.

For **Phase 1**, we can skip this and write directly to the named branch (which is what the current `vault.addFile()`/`updateFile()` already does). This is simpler but means:
- No local commit history before push
- No conflict detection
- Concurrent edits from multiple browsers could collide (last write wins)

This is acceptable for a first version. Phase 2 adds proper branching.
