## 1. Vault-Browse Unification (Document 189)

**Brief:** `v0.19.7__arch-brief__vault-browse-unification.md`

### What's Proposed

Extract the Browse module from SG/Send into a standalone, adapter-based component:

```
Browse Module
├── Core rendering (tree, viewer, tabs, search)
├── Data Source Interface
│   ├── ZipAdapter     — SG/Send transfers
│   └── VaultApiAdapter — vault.sgraph.ai API
└── Feature set (inherited by both consumers)
```

**Fractal vaults:** A folder inside a vault can be another vault (accessed via vault ID + token). The Browse module navigates through the nested vault transparently.

### Architectural Assessment

**Strengths:**
- Adapter pattern is correct. The existing Browse module already has implicit coupling to zip data; extracting this into an explicit interface is low-risk refactoring with high payoff.
- Feature parity between send and vault Browse is genuinely valuable — users should get the same experience regardless of data source.
- Fractal vault concept is elegant: vaults within vaults creates a recursive namespace that matches how people actually organise work (project vaults containing team vaults containing personal vaults).

**Risks:**
- **State management complexity**: The Browse module currently has no concept of "data source state" (auth tokens, vault IDs). The VaultApiAdapter needs to carry credentials. Care needed to avoid leaking auth state across adapter instances.
- **Fractal vault depth**: Unbounded recursion in navigation is a UX and performance risk. Recommend a maximum depth (e.g., 3 levels) with a clear breadcrumb showing path.
- **Adapter contract stability**: Once two consumers depend on the same interface, changing the data source interface becomes a breaking change. Define the interface contract formally before implementation (not after).

**Recommended interface contract (minimal):**
```js
interface DataSourceAdapter {
  getRoot():        Promise<TreeNode>
  getChildren(id):  Promise<TreeNode[]>
  getContent(id):   Promise<Blob | string>
  getMetadata(id):  Promise<FileMetadata>
  canWrite():       boolean
}
```

**Verdict:** ✅ Architecturally sound. Implement after sg-tree v0.1.0 ships (sg-tree is the tree view that Browse uses). The IFD sequence: extract interface → build ZipAdapter → build VaultApiAdapter → release.

## 2. sg-llm Component Family (Document 202)

**Brief:** `v0.19.11__architect-review__sg-llm-reality-constructor.md`

### What's Specified

A 10-component family under `components/llm/`:

```
sg-llm-events.js          — frozen event constants
sg-llm-connection.js      — provider/model/key management
sg-llm-reality.js         — context assembly (THE core)
sg-llm-request.js         — API call engine (headless)
sg-llm-output.js          — streaming display
sg-llm-stats.js           — token/cost tracking
sg-llm-debug.js           — request inspection
sg-llm-infographic.js     — SVG extraction/rendering
sg-llm-bundle.js          — state capture + fork tree
sg-llm-bundle-list.js     — bundle browser / time travel
sg-llm-attachments.js     — file drop + clipboard + cache
```

### Architectural Assessment

**The reality constructor pattern is the most important idea here.** Explicit block assembly (system + context + history + memory + question) is architecturally superior to an accumulated conversation buffer for two reasons:
1. Token budget is visible and controllable at each block
2. Memory block can be rewritten by the LLM itself (Mode 2: Update Memory) — the LLM prunes stale knowledge, preventing context rot

**Strengths:**
- Separation of concerns is clean: `sg-llm-reality.js` (what to send) is decoupled from `sg-llm-request.js` (how to send it) and `sg-llm-output.js` (how to display it). These three can evolve independently.
- `sg-llm-events.js` as a frozen contract prevents tight coupling between components.
- Bundle system with `parent_id` fork tree is already proven in workspace v0.1.1. Extracting it to `sg-llm-bundle.js` is the right call — reuse over re-implementation.
- Provider-agnostic design (connection component handles OpenRouter, Ollama, Anthropic direct) prevents lock-in.

**Concerns:**
- **11 components is a large surface area for v0.1.0.** Suggest sequencing: first implement `sg-llm-events`, `sg-llm-reality`, `sg-llm-request`, `sg-llm-output` as the minimum useful set. Add stats, debug, bundle in v0.1.1. Infographic and attachments in v0.1.2+.
- **`sg-llm-infographic.js`** (SVG extraction) is significantly more complex than the others. It needs to parse LLM output for embedded SVG and render it safely. This is a separate mini-project; flag for separate IFD track.
- **`sg-llm-attachments.js`** raises security questions: clipboard paste of files, file drop from OS. Where is the sandbox boundary? Files from clipboard could contain sensitive data. Needs AppSec review before implementation.

**Block type model (from spec):**
```
system    → { role: 'system', content }
context   → { role: 'user', content: '--- CONTEXT: {label} ---\n...' }
history   → [{ role: 'user'|'assistant', content }, ...]
memory    → { role: 'user', content: '--- SESSION MEMORY ---\n...' }
question  → { role: 'user', content }
```

This is correct. The `context` block pattern (labeled sections in user turn) is a well-established technique that works across all provider APIs regardless of whether they support system messages natively.

**Verdict:** ✅ Architecture is well-specified. Start with the 4-component core (events, reality, request, output). Flag infographic and attachments for separate security + complexity review.

## 3. IFD Surgical Override + Version-Stamped Filenames (Document 198)

**Brief:** `v1.3.0__ifd__surgical-overrides-version-stamped-filenames.md`

### What's Specified

Naming convention for IFD surgical overrides:
- `sg-tree.js` — base (no version stamp)
- `sg-tree--v0.1.1.js` — surgical override (double dash = version stamp)
- `sg-tree-events.js` — part of component name (single dash)

Implementation mechanism: `window.SgTree = SgTree` after `customElements.define()` allows override files to patch prototype methods. ES module deferred execution guarantees base loads before `connectedCallback`.

### Architectural Assessment

**This naming convention should be formalised project-wide.** The double-dash separator is unambiguous and machine-parseable. A build system can easily collect all `*--v*.js` files in loading order.

**Strengths:**
- Avoids filename collisions across IFD versions without runtime script management
- `window` exposure of the class is minimal — only the class constructor, not instances
- Consolidation at major version (v0.2.0 merges all v0.1.x) prevents indefinite patch accumulation
- The pattern works with the existing IFD build-time file layering (no new runtime tooling needed)

**One concern:** `window` exposure leaks the class into global scope permanently. This is acceptable for components used in developer tools (tools.sgraph.ai), but should not be adopted for send.sgraph.ai user-facing components where we want strong encapsulation. Recommend documenting this as a **tools.sgraph.ai pattern only**.

**Verdict:** ✅ Adopt as the canonical IFD surgical override convention. Document in the IFD guide. Apply to sg-tree immediately.

## 5. Cross-Cutting Observation: Adapter Pattern Emerging as a Project-Wide Standard

Across this batch, the **adapter pattern** appears in three independent proposals:
- **Vault-Browse:** `ZipAdapter` and `VaultApiAdapter`
- **sg-llm-connection:** Provider adapters (OpenRouter, Ollama, Anthropic)
- **sg-wasm:** Module adapters (FFmpeg, Whisper, future WASM modules)

This convergence is significant. The project is naturally evolving toward a generic adapter architecture where:
1. Core components define a stable interface
2. Adapters implement the interface for specific data sources/providers
3. Consumers are agnostic about which adapter is active

This is the right direction. Recommend formalising the adapter pattern as a documented architectural standard (add to IFD guide) so future components follow consistently.
