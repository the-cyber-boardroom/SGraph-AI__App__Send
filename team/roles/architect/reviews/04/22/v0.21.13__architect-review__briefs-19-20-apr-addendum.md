## 1. Send/Receive JS API (Doc 302) — Architectural Assessment

### Design

The API is clean and appropriately scoped. `SgSend.{sendFile,sendText,sendFolder,receiveFile,receiveText,receiveFolder}` provides a composable primitive that hides encryption, chunking, and token formats from callers. The standard meta-API (`getManifest`, `getMethods`, `getSkills`) aligns with the established JS API convention from 08 Apr (doc series).

The Web Component layer (`<sg-send-drop>`, `<sg-send-receive>`, `<sg-send-panel>`) is the correct approach for tool integration. Custom events (`token-created`, `file-received`) enable loose coupling between components.

### Concerns

**Namespace collision risk.** `SgSend` as a global namespace conflicts if multiple versions are loaded simultaneously. The convention should adopt a versioned module pattern (`window.SgSend_v1`) or preferably use ES module imports with a CDN path that includes the version. Decision #20 (raised in master index) covers this.

**`sendFolder` chunking strategy.** The brief specifies "10 files" as the acceptance test but does not define what happens when a folder contains 100+ files or total size exceeds Lambda's response limits. The existing chunked upload machinery in `sg-send.js` should handle this, but the API contract needs to document size/count limits.

**Interim localStorage token storage.** The brief explicitly marks this as interim (pending the auth MVP `<sg-vault-picker>`). The architectural contract must prevent this interim solution from becoming permanent. Suggest: localStorage key with a `v1:` prefix and a migration path documented in the same PR.

**`receiveFolder` decryption ordering.** Multi-file receive requires downloading each chunk and decrypting in order. The current `sg-send.js` is single-file oriented. The folder receive path will need to orchestrate multiple concurrent fetches without exceeding Lambda's concurrent invocation limits when called at volume.

### Verdict: Architecturally sound. Ship as described. Flag namespace and folder-size concerns to Dev before implementation.

## 2. Three Agentic Primitives (Doc 306) — Architectural Assessment

### The Claim

The arch brief asserts three primitives that SG/Send provides for agentic workflows:

1. **Browser automation with MITM observability** — EXISTS (Playwright service + EC2+MITM sidecar pattern)
2. **Encrypted VFS (vaults)** — EXISTS (SGit CLI + vault API + browse UI)
3. **Code execution sandbox** — PROPOSED (layered container architecture is the foundation; sandbox itself is not implemented)

### Accuracy Check Against Reality Document

The first two claims are code-verified and accurate:
- Playwright service: `Fast_API__SGraph__Send__Container`, session API, sequence execution — EXISTS
- EC2+MITM sidecar: `provision_ec2.py`, MITM pattern — EXISTS
- Vaults: full vault API, SGit CLI, browse UI, branch model, push/pull — EXISTS

The code execution sandbox claim is explicitly flagged as "not fully implemented yet" in the brief itself. This is correctly marked PROPOSED in this session's master index. The architect must ensure this is not described as EXISTS in any external materials until code is written.

### The Declarative Browser Automation API Gap

The brief proposes a JSON instruction format:
```json
{ "steps": [{ "action": "navigate" }, { "action": "click" }, { "action": "screenshot" }] }
```

The existing Playwright service has a `/sequence/execute` endpoint. Whether this endpoint already accepts the JSON format described in doc 306, or whether the format is aspirational, cannot be determined from briefs alone. **Architect recommendation:** verify `sequence/execute` endpoint schema against the brief's proposed JSON before committing the three-primitives narrative to external materials.

### Positioning Opportunity

The three-primitives framing is the clearest architectural differentiation statement the project has produced. It maps well to the "infrastructure layer under any agentic framework" positioning. Recommend: create an architecture diagram showing how the three primitives compose (agent → browser automation → results written to vault → vault read by another agent). This is also the diagram for the sgraph.ai website's technical section.

### Verdict: Primitives 1 and 2 are architecturally accurate. Primitive 3 must be labelled PROPOSED in all external materials. Priority: verify sequence API format against brief spec.

## 3. QA Three-Mode Architecture (Doc 304) — Architectural Assessment

### Design

The `PlaywrightClient(mode=...)` abstraction is the right pattern. It matches the existing Memory-FS (`Storage_FS`) precedent: the application layer is ignorant of the backend. QA tests become backend-agnostic.

The three modes map cleanly to real deployment targets:
- Mode 1 (local Docker) → development and CI
- Mode 2 (Lambda) → serverless, scalable, no MITM
- Mode 3 (EC2+MITM) → production monitoring with full observability

### Open Architecture Questions

**Where does `PlaywrightClient` live?** The brief shows a Python class (`PlaywrightClient(mode=...)`). This suggests a Python library living in the QA codebase (`tests/`) or as a standalone package. If other teams (e.g., the personalised outreach tool from doc 310) need to invoke Playwright programmatically, a shared package is warranted. If it's QA-only, it stays in `tests/`. The decision affects where the abstraction is placed.

**Session lifecycle in the abstraction layer.** In Mode 1 and Mode 3, sessions are long-lived (container persists across steps). In Mode 2 (Lambda), the function is stateless — a "session" must be either entirely within a single Lambda invocation or persisted to an external store between calls. The abstraction layer needs to expose a session model that works across all three. This is a non-trivial design problem for Mode 2.

**Hourly traffic generation trigger.** CloudWatch Events is the correct trigger for Lambda invocations. But if the traffic generation must run in Mode 3 (EC2+MITM for full observability), the trigger is a cron job on the EC2 instance. The brief does not clarify which mode the hourly heartbeat should use. Recommend: start heartbeat in Mode 2 (Lambda, simplest), with a manual trigger for Mode 3 when deep traffic analysis is needed.

### Verdict: Architecturally sound direction. Three open questions need answers before implementation: library location, Lambda session lifecycle, and heartbeat mode. Flag to QA and Dev.

## 4. AWS Marketplace — Three AMIs (Doc 307) — Architectural Assessment

### AMI 1: Playwright Service

The Playwright service Docker image and CI ECR build job exist (code-verified in v0.21.8 master index). The architecture is complete. The remaining work is:
- Battle hardening (error handling, timeouts, resource limits)
- Marketplace-specific: listing metadata, AMI baking pipeline, pricing configuration
- Security hardening before public listing (ensure no secrets leak via API, rate limiting on screenshot endpoint)

### AMI 2: SG/Send Server

The server code exists (`sgraph_ai_app_send__docker/`, `Fast_API__SGraph__Send__Container`). A Docker image already bakes. The gap is:
- AMI baking from the Docker image (the Playwright CI pattern is the reference)
- Configuring for public-facing use: rate limiting, optional auth, disk vs S3 storage default

The naming decision (#19 in master index) — "SG/Send Server" vs "SGit Server" — is a marketing decision, not an architectural one. **Architect recommendation:** ship both as the same AMI, two Marketplace listings with different descriptions. Single codebase, multiple positioning surfaces.

### AMI 3: Tools Suite

This is the most architecturally ambiguous AMI. The tools mostly run client-side, so the "AMI" is essentially a static file server with an nginx or uvicorn serving the tool HTML/JS. The "AMI" delivers no server-side compute for the tools themselves.

However, some tools call the Playwright service (browser automation tasks) and the SG/Send API (saving output). The Tools Suite AMI needs a clear answer to: does it bundle AMI 1 and AMI 2 as dependencies, or does it require the caller to configure external endpoints?

**Architect recommendation:** Document the dependency model before packaging AMI 3. The simplest approach: AMI 3 is a static file server; callers configure Playwright and SG/Send endpoint URLs via environment variables or a config file served from a protected admin endpoint.

### Verdict: AMI 1 is closest to ready. AMI 2 follows the same pattern. AMI 3 needs dependency model clarification before packaging.

## 5. Positioning Decision — Hero Line (Doc 308)

### Architectural Relevance

The positioning decision (`"Version-controlled, client-encrypted vaults. For humans and agents."`) has one direct architectural implication: **vaults are now the primary front-door concept.** This commits the product to ensuring the vault experience is excellent before everything else. The file transfer (send/receive) flow becomes a secondary experience.

**Architectural consequence:** the send/receive JS API (doc 302) must be integrated into the vault workflow, not the other way around. The "vaults first" commitment means the primary UI surfaces are vault creation, vault browsing, and vault sharing. Send/receive becomes the transfer primitive that vaults build on.

The shallow clone feature mentioned in the positioning brief (already EXISTS per SGit CLI) reinforces the vault-as-primary-interface story. Recommend documenting shallow clone in the architecture overview for the website.

### "For Humans and Agents" — Technical Accuracy

The brief claims agents already use the vaults (Claude Code, Perplexity, SGit CLI collaborating on the same vault). This is demonstrated by the project's own workflow (briefs in vaults, agent sessions reading from vaults). The claim is accurate and code-backed.

### Verdict: Positioning decision is architecturally coherent. "Vaults first" commitment must be reflected in the UX hierarchy going forward.

## 6. Website Redesign — Receive Flow Bug (Doc 309)

The website brief calls out that the receive flow is "currently broken" — token redemption is hard to find on the main site. This is an existing defect, not a new feature.

**Architectural note:** The `send.sgraph.ai#token` → `/en-gb/browse/#token` short-link redirect (EXISTS, code-verified in `v0.16.26__what-exists-today.md`) means that pasting a token directly in the URL bar of `send.sgraph.ai` works. The problem is the UI does not surface this path clearly. The fix is a UI change (add a visible token-paste input to the main page), not a backend change.

## 7. Playwright Screenshot Simple API (Doc 311)

The brief is architecturally straightforward. The screenshot API is a thin wrapper over the existing Playwright service. The execution order (navigate → JS → click → screenshot) is well-defined.

One concern: **`javascript` parameter injection.** Allowing arbitrary JS execution raises XSS/injection concerns if this endpoint is ever exposed publicly without authentication. The existing Playwright service requires an API key; the screenshot endpoint must inherit the same auth requirement. Confirm with AppSec before public AMI listing.

The mini frontend is a standard static page served from the container. No architectural concerns.
