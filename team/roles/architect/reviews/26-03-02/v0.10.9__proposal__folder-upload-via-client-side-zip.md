# Architect Proposal: Folder Upload via Client-Side Zip

**Version:** v0.10.9
**Date:** 2 March 2026
**Role:** Architect (Explorer team)
**Type:** Implementation proposal

---

## Summary

Users currently cannot upload folders — dropping a folder onto the upload zone produces "Failed to read file" because the browser returns a directory entry, not a readable `File` blob. This proposal adds folder upload support by **zipping the folder client-side** before feeding the zip into the existing encrypt → upload pipeline.

**Key property:** The backend needs zero changes. All work is client-side JavaScript in the `send-upload` web component.

---

## 1. Current State: Why Folders Fail

The drop handler in `send-upload.js` (v0.1.8) extracts files from `e.dataTransfer.files`:

```javascript
handleDrop(e) {
    const files = e.dataTransfer && e.dataTransfer.files;
    if (files && files.length > 0) {
        this.selectedFile = files[0];    // ← directory entry, not a File
    }
}
```

When a folder is dropped, `files[0]` is a `File` object with `size: 0` and no readable content. The subsequent `FileReader.readAsArrayBuffer()` call either returns an empty buffer or throws, causing the "Failed to read file" error.

## 2. Proposed Flow

```
┌───────────────┐     ┌───────────────┐     ┌──────────────┐
│  User drops   │────→│  Browser reads │────→│  JSZip builds │
│  folder onto  │     │  folder tree   │     │  .zip in      │
│  drop zone    │     │  recursively   │     │  memory       │
└───────────────┘     └───────────────┘     └──────┬───────┘
                                                    │
                                              zip Blob
                                           (folder-name.zip)
                                                    │
                                                    ▼
                                       ┌─────────────────────────┐
                                       │  Existing pipeline:     │
                                       │  SGMETA wrap            │
                                       │  → AES-256-GCM encrypt  │
                                       │  → upload (direct or    │
                                       │    presigned multipart) │
                                       │  → complete             │
                                       └─────────────────────────┘
                                                    │
                                                    ▼
                                       ┌─────────────────────────┐
                                       │  Recipient downloads    │
                                       │  → decrypts             │
                                       │  → gets folder-name.zip │
                                       │  → unzips locally       │
                                       └─────────────────────────┘
```

**The recipient's experience is unchanged** — they download a file (which happens to be a `.zip`), double-click to unzip. No browser-side decompression needed.

## 3. Technical Design

### 3a. Detecting Folder Drops

The modern `DataTransfer` API provides `items` with `webkitGetAsEntry()` (or the newer `getAsFileSystemHandle()`). When a folder is dropped, the entry's `.isDirectory` is `true`.

```javascript
handleDrop(e) {
    e.preventDefault();
    e.stopPropagation();

    const items = e.dataTransfer.items;
    if (items && items.length > 0) {
        const entry = items[0].webkitGetAsEntry();
        if (entry && entry.isDirectory) {
            this._handleFolderDrop(entry);       // New path
            return;
        }
    }

    // Existing single-file path (unchanged)
    const files = e.dataTransfer.files;
    if (files && files.length > 0) {
        this.selectedFile = files[0];
        this.state = 'idle';
        this.render();
        this.setupEventListeners();
    }
}
```

### 3b. Reading the Folder Tree Recursively

```javascript
async _readDirectoryTree(directoryEntry) {
    const files = [];

    async function readEntries(dirEntry, path) {
        const reader  = dirEntry.createReader();
        const entries = await new Promise((resolve, reject) => {
            reader.readEntries(resolve, reject);
        });

        for (const entry of entries) {
            if (entry.isFile) {
                const file = await new Promise((resolve, reject) => {
                    entry.file(resolve, reject);
                });
                files.push({ path: path + entry.name, file });
            } else if (entry.isDirectory) {
                await readEntries(entry, path + entry.name + '/');
            }
        }
    }

    await readEntries(directoryEntry, '');
    return files;   // [{ path: "subfolder/report.pdf", file: File }, ...]
}
```

**Note:** `readEntries()` may not return all entries in a single call for large directories. Production implementation should loop until an empty array is returned.

### 3c. Folder Scan and Options Panel

When a folder is detected (via drop or browse), scan the tree **before zipping** to show the options panel:

```javascript
async _handleFolderDrop(directoryEntry) {
    this._folderEntry = directoryEntry;
    this._folderName  = directoryEntry.name;

    // Phase 1: Scan (fast — no file content read, just directory walk)
    const allEntries = await this._readDirectoryTree(directoryEntry);
    this._folderScan = {
        entries    : allEntries,
        fileCount  : allEntries.filter(e => !e.isDir).length,
        folderCount: allEntries.filter(e => e.isDir).length,
        totalSize  : allEntries.reduce((sum, e) => sum + (e.file ? e.file.size : 0), 0)
    };

    // Phase 2: Show options panel (user chooses compression, toggles)
    this.state = 'folder-options';
    this.render();
}
```

The options panel renders with the scan summary and user controls (see Section 10.2). When the user clicks "Compress & Upload":

```javascript
async _startFolderZip() {
    const options = this._getFolderOptions();   // { level: 4, includeEmpty: false, includeHidden: false }

    this.state = 'zipping';
    this.render();

    const files = this._folderScan.entries.filter(e => {
        if (e.isDir && !options.includeEmpty) return false;
        if (!options.includeHidden && e.name.startsWith('.')) return false;
        return true;
    });

    const zip = new JSZip();
    for (const { path, file, isDir } of files) {
        if (isDir) {
            zip.folder(path);
        } else {
            const buffer = await file.arrayBuffer();
            zip.file(path, buffer);
        }
    }

    const zipBlob = await zip.generateAsync({
        type:        'blob',
        compression: options.level === 0 ? 'STORE' : 'DEFLATE',
        compressionOptions: { level: options.level }
    });

    // Convert zip Blob to a File object so the existing pipeline works unchanged
    this.selectedFile = new File([zipBlob], `${this._folderName}.zip`, {
        type: 'application/zip'
    });

    this.state = 'idle';
    this.render();
    this.setupEventListeners();
    this.startUpload();   // Auto-start upload after compression
}
```

**After `_startFolderZip` completes**, the existing `startUpload()` flow takes over with no modifications — it sees a `File` named `folder-name.zip` and processes it through SGMETA → encrypt → upload.

### 3d. Browse Buttons (File and Folder)

Two hidden inputs — one for files, one for folders:

```html
<input type="file" id="file-input"   style="display: none;">
<input type="file" id="folder-input" style="display: none;" webkitdirectory>
```

Two visible buttons:

```html
<button id="browse-file-btn">Browse File</button>
<button id="browse-folder-btn">Browse Folder</button>
```

**Browser limitation:** `<input type="file">` and `<input type="file" webkitdirectory>` are mutually exclusive — a single picker can select either files OR folders, not both. Drag-drop has no such limitation (both work on the same drop zone).

The `folder-input` `change` event provides `e.target.files` — a flat list of all files in the folder with `webkitRelativePath` properties containing the relative path from the folder root.

Add two new states to the stage progression:

## 5. Security Invariants

| Invariant | Preserved? | Rationale |
|-----------|------------|-----------|
| Server never sees plaintext | Yes | Zip is built in browser, encrypted before upload |
| Server never sees filenames | Yes | Filenames are inside the zip, which is inside the SGMETA envelope, which is encrypted |
| Server never sees decryption key | Yes | No change to key handling |
| No new server endpoints | Yes | Zip creation is purely client-side |
| Encrypted blob is opaque | Yes | Server sees same `[IV][ciphertext]` format regardless of content |

**Additional security benefit:** Individual filenames within the folder are doubly protected — inside the zip (which the server can't see) and inside the SGMETA envelope (which wraps the zip).

## 6. Size Limits and Memory Considerations

| Scenario | Memory footprint | Mitigation |
|----------|-----------------|------------|
| Small folder (< 50 MB) | ~3x folder size (files + zip + encrypted) | No action needed |
| Medium folder (50-200 MB) | ~150-600 MB browser memory | Show warning before proceeding |
| Large folder (200 MB - 1 GB) | ~600 MB - 3 GB browser memory | Show strong warning; rely on browser GC between stages |
| Folder > 1 GB | Exceeds transfer limit | Block with error: "Folder too large" |

**Recommendation:** Display the total folder size before zipping begins. Warn at 100 MB, hard-block at 900 MB (leaving headroom for zip overhead below the 1 GB transfer limit).

## 7. Browser Compatibility

| API | Chrome | Firefox | Safari | Edge |
|-----|--------|---------|--------|------|
| `webkitGetAsEntry()` | Yes | Yes | Yes | Yes |
| `FileSystemDirectoryEntry.createReader()` | Yes | Yes | Yes | Yes |
| `input[webkitdirectory]` | Yes | Yes | Yes | Yes |
| `File System Access API` (newer) | Yes | No | No | Yes |

**Recommendation:** Use the `webkitGetAsEntry()` API — it has the broadest browser support despite the `webkit` prefix. It is standardised as `getAsFileSystemHandle()` in newer browsers but Firefox support is still incomplete.

## 10. Decisions (Answered by Human)

### 10.1 Drop Zone Text and Browse Buttons

**Decision:** Update the drop zone text to say **"Drop a file or folder"**.

**Browser limitation:** `<input type="file">` and `<input type="file" webkitdirectory>` are mutually exclusive — a single file picker dialog can select either files OR folders, not both. Drag-drop has no such limitation (both work on the same drop zone).

**Implementation:** Two browse buttons below the drop zone:

```
┌─────────────────────────────────┐
│                                 │
│     Drop a file or folder       │
│                                 │
│   [ Browse File ]  [ Browse Folder ]
│                                 │
└─────────────────────────────────┘
```

- `<input type="file" id="file-input">` — triggered by "Browse File"
- `<input type="file" id="folder-input" webkitdirectory>` — triggered by "Browse Folder"
- Drag-drop accepts either — detection via `webkitGetAsEntry().isDirectory`

### 10.2 Compression Level — Options Panel

**Decision:** When a folder is detected, show a **folder options panel** before zipping begins. Default compression level: **4** (fast). The options panel allows experimentation with settings.

```
┌─────────────────────────────────┐
│  Folder: project-docs           │
│  12 files, 3 folders, 4.2 MB   │
│                                 │
│  Compression: [▼ Level 4 ]     │
│    0 = Store (fastest, no compression)
│    4 = Fast (default)
│    6 = Balanced
│    9 = Maximum (slowest, smallest)
│                                 │
│  ☐ Include empty folders        │
│  ☐ Include hidden files         │
│                                 │
│  [ Compress & Upload ]          │
└─────────────────────────────────┘
```

This panel appears **after folder scanning but before zipping**, giving the user control over the trade-offs. The folder scan (counting files, calculating total size) is fast and provides the summary shown at the top.

### 10.3 Empty Folders

**Decision:** Toggle option, **default OFF** (empty folders excluded). Users who need to preserve directory structure can enable it.

Implementation: During the directory tree walk, only add directory entries to the zip when the toggle is enabled:

```javascript
if (entry.isDirectory) {
    if (includeEmptyFolders) {
        zip.folder(path + entry.name);   // Explicit empty folder entry
    }
    await readEntries(entry, path + entry.name + '/');
}
```

### 10.4 Hidden Files

**Decision:** Toggle option, **default OFF** (dotfiles excluded). Filters out files and folders starting with `.` (e.g., `.gitignore`, `.DS_Store`, `.env`).

Implementation: Skip dotfiles during the directory tree walk:

```javascript
for (const entry of entries) {
    if (!includeHidden && entry.name.startsWith('.')) {
        continue;   // Skip hidden files/folders
    }
    // ... process entry
}
```

**Security note:** Excluding dotfiles by default is also a safety measure — `.env` files, `.ssh/` directories, and similar sensitive dotfiles won't accidentally be included in a transfer.

## 11. Recommendation

**Build this as a single IFD layer.** The change is self-contained in the upload web component, requires no backend modifications, preserves all security invariants, and the dependency (JSZip) is small and well-established. The existing encrypt → upload → download pipeline handles the zip blob identically to any other file.
