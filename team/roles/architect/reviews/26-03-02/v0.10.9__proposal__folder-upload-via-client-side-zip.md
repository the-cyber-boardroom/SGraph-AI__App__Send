# Architect Proposal: Folder Upload via Client-Side Zip

**Version:** v0.10.9
**Date:** 2 March 2026
**Role:** Architect (Explorer team)
**Type:** Implementation proposal

---

## Summary

Users currently cannot upload folders — dropping a folder onto the upload zone produces "Failed to read file" because the browser returns a directory entry, not a readable `File` blob. This proposal adds folder upload support by **zipping the folder client-side** before feeding the zip into the existing encrypt → upload pipeline.

**Key property:** The backend needs zero changes. All work is client-side JavaScript in the `send-upload` web component.

---

## 1. Current State: Why Folders Fail

The drop handler in `send-upload.js` (v0.1.8) extracts files from `e.dataTransfer.files`:

```javascript
handleDrop(e) {
    const files = e.dataTransfer && e.dataTransfer.files;
    if (files && files.length > 0) {
        this.selectedFile = files[0];    // ← directory entry, not a File
    }
}
```

When a folder is dropped, `files[0]` is a `File` object with `size: 0` and no readable content. The subsequent `FileReader.readAsArrayBuffer()` call either returns an empty buffer or throws, causing the "Failed to read file" error.

## 2. Proposed Flow

```
┌───────────────┐     ┌───────────────┐     ┌──────────────┐
│  User drops   │────→│  Browser reads │────→│  JSZip builds │
│  folder onto  │     │  folder tree   │     │  .zip in      │
│  drop zone    │     │  recursively   │     │  memory       │
└───────────────┘     └───────────────┘     └──────┬───────┘
                                                    │
                                              zip Blob
                                           (folder-name.zip)
                                                    │
                                                    ▼
                                       ┌─────────────────────────┐
                                       │  Existing pipeline:     │
                                       │  SGMETA wrap            │
                                       │  → AES-256-GCM encrypt  │
                                       │  → upload (direct or    │
                                       │    presigned multipart) │
                                       │  → complete             │
                                       └─────────────────────────┘
                                                    │
                                                    ▼
                                       ┌─────────────────────────┐
                                       │  Recipient downloads    │
                                       │  → decrypts             │
                                       │  → gets folder-name.zip │
                                       │  → unzips locally       │
                                       └─────────────────────────┘
```

**The recipient's experience is unchanged** — they download a file (which happens to be a `.zip`), double-click to unzip. No browser-side decompression needed.

## 3. Technical Design

### 3a. Detecting Folder Drops

The modern `DataTransfer` API provides `items` with `webkitGetAsEntry()` (or the newer `getAsFileSystemHandle()`). When a folder is dropped, the entry's `.isDirectory` is `true`.

```javascript
handleDrop(e) {
    e.preventDefault();
    e.stopPropagation();

    const items = e.dataTransfer.items;
    if (items && items.length > 0) {
        const entry = items[0].webkitGetAsEntry();
        if (entry && entry.isDirectory) {
            this._handleFolderDrop(entry);       // New path
            return;
        }
    }

    // Existing single-file path (unchanged)
    const files = e.dataTransfer.files;
    if (files && files.length > 0) {
        this.selectedFile = files[0];
        this.state = 'idle';
        this.render();
        this.setupEventListeners();
    }
}
```

### 3b. Reading the Folder Tree Recursively

```javascript
async _readDirectoryTree(directoryEntry) {
    const files = [];

    async function readEntries(dirEntry, path) {
        const reader  = dirEntry.createReader();
        const entries = await new Promise((resolve, reject) => {
            reader.readEntries(resolve, reject);
        });

        for (const entry of entries) {
            if (entry.isFile) {
                const file = await new Promise((resolve, reject) => {
                    entry.file(resolve, reject);
                });
                files.push({ path: path + entry.name, file });
            } else if (entry.isDirectory) {
                await readEntries(entry, path + entry.name + '/');
            }
        }
    }

    await readEntries(directoryEntry, '');
    return files;   // [{ path: "subfolder/report.pdf", file: File }, ...]
}
```

**Note:** `readEntries()` may not return all entries in a single call for large directories. Production implementation should loop until an empty array is returned.

### 3c. Building the Zip with JSZip

```javascript
async _handleFolderDrop(directoryEntry) {
    this.state = 'zipping';
    this.render();

    const folderName = directoryEntry.name;
    const files      = await this._readDirectoryTree(directoryEntry);

    const zip = new JSZip();
    for (const { path, file } of files) {
        const buffer = await file.arrayBuffer();
        zip.file(path, buffer);
    }

    const zipBlob = await zip.generateAsync({
        type:        'blob',
        compression: 'DEFLATE',
        compressionOptions: { level: 6 }
    });

    // Convert zip Blob to a File object so the existing pipeline works unchanged
    this.selectedFile = new File([zipBlob], `${folderName}.zip`, {
        type: 'application/zip'
    });

    this.state = 'idle';
    this.render();
    this.setupEventListeners();
}
```

**After `_handleFolderDrop` completes**, the existing `startUpload()` flow takes over with no modifications — it sees a `File` named `folder-name.zip` and processes it through SGMETA → encrypt → upload.

### 3d. Folder Input Button (Browse Alternative)

For users who prefer clicking instead of drag-drop, add a second input:

```html
<input type="file" id="folder-input" webkitdirectory style="display: none;">
```

The `change` event provides `e.target.files` — a flat list of all files in the folder with `webkitRelativePath` properties containing the relative path from the folder root.

### 3e. New UI State: `zipping`

Add one new state to the stage progression:

| State | Progress | Label |
|-------|----------|-------|
| **`zipping`** | 5% | "Compressing folder..." |
| `reading` | 10% | "Reading file..." |
| `encrypting` | 30% | "Encrypting..." |
| `creating` | 50% | "Creating transfer..." |
| `uploading` | 70% | "Uploading..." |
| `completing` | 90% | "Completing..." |
| `complete` | 100% | "Done" |

The `zipping` state only appears for folder uploads. Single-file uploads skip it entirely.

## 5. Security Invariants

| Invariant | Preserved? | Rationale |
|-----------|------------|-----------|
| Server never sees plaintext | Yes | Zip is built in browser, encrypted before upload |
| Server never sees filenames | Yes | Filenames are inside the zip, which is inside the SGMETA envelope, which is encrypted |
| Server never sees decryption key | Yes | No change to key handling |
| No new server endpoints | Yes | Zip creation is purely client-side |
| Encrypted blob is opaque | Yes | Server sees same `[IV][ciphertext]` format regardless of content |

**Additional security benefit:** Individual filenames within the folder are doubly protected — inside the zip (which the server can't see) and inside the SGMETA envelope (which wraps the zip).

## 6. Size Limits and Memory Considerations

| Scenario | Memory footprint | Mitigation |
|----------|-----------------|------------|
| Small folder (< 50 MB) | ~3x folder size (files + zip + encrypted) | No action needed |
| Medium folder (50-200 MB) | ~150-600 MB browser memory | Show warning before proceeding |
| Large folder (200 MB - 1 GB) | ~600 MB - 3 GB browser memory | Show strong warning; rely on browser GC between stages |
| Folder > 1 GB | Exceeds transfer limit | Block with error: "Folder too large" |

**Recommendation:** Display the total folder size before zipping begins. Warn at 100 MB, hard-block at 900 MB (leaving headroom for zip overhead below the 1 GB transfer limit).

## 7. Browser Compatibility

| API | Chrome | Firefox | Safari | Edge |
|-----|--------|---------|--------|------|
| `webkitGetAsEntry()` | Yes | Yes | Yes | Yes |
| `FileSystemDirectoryEntry.createReader()` | Yes | Yes | Yes | Yes |
| `input[webkitdirectory]` | Yes | Yes | Yes | Yes |
| `File System Access API` (newer) | Yes | No | No | Yes |

**Recommendation:** Use the `webkitGetAsEntry()` API — it has the broadest browser support despite the `webkit` prefix. It is standardised as `getAsFileSystemHandle()` in newer browsers but Firefox support is still incomplete.

## 10. Open Questions for Human Decision

1. **Folder button in UI?** Should there be a visible "Upload Folder" button alongside the existing drop zone, or should folder upload be discoverable only via drag-drop?

2. **Compression level:** DEFLATE level 6 (balanced) is the default. Level 0 (store, no compression) would be faster but produce larger zips. For already-compressed content (images, videos), compression adds time with minimal size reduction. Should we offer a toggle, or default to level 6?

3. **Empty folders:** Should empty subdirectories within the folder be preserved in the zip? JSZip supports this but it adds complexity to the tree walk.

4. **Hidden files:** Should dotfiles (`.gitignore`, `.DS_Store`, etc.) be included or filtered out?

---

## 11. Recommendation

**Build this as a single IFD layer.** The change is self-contained in the upload web component, requires no backend modifications, preserves all security invariants, and the dependency (JSZip) is small and well-established. The existing encrypt → upload → download pipeline handles the zip blob identically to any other file.
