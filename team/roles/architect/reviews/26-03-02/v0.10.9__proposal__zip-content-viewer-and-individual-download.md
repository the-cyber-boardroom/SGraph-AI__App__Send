# Architect Proposal: Zip Content Viewer and Individual File Download

**Version:** v0.10.9
**Date:** 2 March 2026
**Role:** Architect (Explorer team)
**Type:** Implementation proposal
**Builds on:** [v0.10.9 Folder Upload via Client-Side Zip](v0.10.9__proposal__folder-upload-via-client-side-zip.md)

---

## Summary

When a recipient downloads a `.zip` file via SG/Send, the current file-view page shows it as a single opaque file with a "Save Locally" button. The user must download the entire zip and extract it locally to see what's inside.

This proposal extends the download/file-view UI to:

1. **Detect zip files** after decryption
2. **Show the zip's file tree** in a browsable list (left panel)
3. **Preview individual files** in the right panel (reusing existing renderers)
4. **Download individual files or folders** without saving the entire zip first

**Key property:** Backend needs zero changes. JSZip runs in the browser to read the zip after decryption.

## 3. Technical Design

### 3a. Detecting Zip Files

Add `zip` to the `FileTypeDetect` render-type map:

```javascript
// In file-type-detect.js
const RENDER_TYPES = {
    // ... existing types ...
    zip: { extensions: ['.zip'], mimeTypes: ['application/zip', 'application/x-zip-compressed'] }
};
```

In `send-download.js`, after decryption:

```javascript
if (this._renderType === 'zip') {
    await this._renderZipContent(this.decryptedBytes);
} else {
    // existing render paths (markdown, image, pdf, code, text, auto-download)
}
```

### 3b. Reading the Zip with JSZip

After decryption, the plaintext bytes (extracted from the SGMETA envelope) are the raw zip content. JSZip reads them directly:

```javascript
async _parseZip(decryptedBytes) {
    const zip   = await JSZip.loadAsync(decryptedBytes);
    const tree  = [];

    zip.forEach((relativePath, zipEntry) => {
        tree.push({
            path  : relativePath,
            name  : relativePath.split('/').filter(Boolean).pop(),
            dir   : zipEntry.dir,
            size  : zipEntry._data ? zipEntry._data.uncompressedSize : 0,
            entry : zipEntry
        });
    });

    return { zip, tree };
}
```

### 3c. Building the File Tree UI

Convert the flat list of zip entries into a nested tree structure for rendering:

```javascript
_buildTreeHtml(tree) {
    // Sort: folders first, then files, alphabetical within each group
    const roots    = [];
    const children = {};

    for (const item of tree) {
        const parts  = item.path.split('/').filter(Boolean);
        const parent = parts.length > 1 ? parts.slice(0, -1).join('/') + '/' : '';

        if (!children[parent]) children[parent] = [];
        children[parent].push(item);
    }

    // Recursive render
    function renderLevel(parentPath, depth) {
        const items = children[parentPath] || [];
        return items.map(item => {
            if (item.dir) {
                const nested = renderLevel(item.path, depth + 1);
                return `<details open>
                    <summary class="zip-folder" data-path="${item.path}">
                        ${item.name}/
                    </summary>
                    <div style="padding-left: var(--space-4);">${nested}</div>
                </details>`;
            }
            return `<div class="zip-file" data-path="${item.path}">
                ${item.name}
                <span class="zip-size">${formatSize(item.size)}</span>
            </div>`;
        }).join('');
    }

    return renderLevel('', 0);
}
```

The tree uses native `<details>`/`<summary>` elements for folder expand/collapse — no JavaScript state management needed for the open/closed state.

### 3d. Previewing Individual Files

When a file entry is clicked, extract it from the zip and render it using the **existing preview renderers**:

```javascript
async _previewZipEntry(zipEntry, filename) {
    const content    = await zipEntry.async('arraybuffer');
    const renderType = FileTypeDetect.detect(filename, null);

    // Store for "Save This File" button
    this._currentEntryBytes    = content;
    this._currentEntryFilename = filename;

    // Reuse existing renderers
    this.decryptedBytes = content;
    this.fileName       = filename;
    this._renderType    = renderType;

    const previewPanel = this.querySelector('#preview-panel');
    if (renderType === 'pdf') {
        previewPanel.innerHTML = this._renderPdfContent();
    } else if (renderType === 'image') {
        previewPanel.innerHTML = this._renderImageContent();
    } else if (renderType === 'markdown') {
        previewPanel.innerHTML = this._renderMarkdownContent();
    } else if (renderType === 'code') {
        previewPanel.innerHTML = this._renderCodeContent();
    } else if (renderType === 'text') {
        previewPanel.innerHTML = this._renderTextContent();
    } else {
        previewPanel.innerHTML = '<p>No preview available. Click "Save This File" to download.</p>';
    }
}
```

**Key insight:** The existing renderers (`_renderPdfContent`, `_renderImageContent`, etc.) all work from `this.decryptedBytes` and `this.fileName`. By swapping those values to the individual file's content and name, the renderers work unchanged.

### 3e. Downloading Individual Files

```javascript
_saveCurrentEntry() {
    if (this._currentEntryBytes) {
        this.saveFile(this._currentEntryBytes, this._currentEntryFilename || 'download');
    }
}
```

This reuses the existing `saveFile()` method (Blob → object URL → anchor click → cleanup).

### 3f. Downloading a Subfolder as Zip

When the user clicks "Save Folder: X", re-zip just that subtree:

```javascript
async _saveFolder(folderPath) {
    const subZip = new JSZip();

    this._zipInstance.forEach((relativePath, zipEntry) => {
        if (relativePath.startsWith(folderPath) && !zipEntry.dir) {
            // Strip the folder prefix so the zip root is the folder itself
            const innerPath = relativePath.slice(folderPath.length);
            subZip.file(innerPath, zipEntry.async('arraybuffer'));
        }
    });

    const folderName = folderPath.replace(/\/$/, '').split('/').pop();
    const blob = await subZip.generateAsync({ type: 'blob' });
    this.saveFile(blob, `${folderName}.zip`);
}
```

## 4. Data Flow

```
Encrypted payload downloaded from server
    │
    ▼
AES-256-GCM decrypt (existing)
    │
    ▼
SGMETA extract → { filename: "project-docs.zip", content: ArrayBuffer }
    │
    ▼
FileTypeDetect.detect("project-docs.zip") → "zip"
    │
    ▼
JSZip.loadAsync(content) → zip instance + file tree
    │
    ├───────────────────────────────────────┐
    │                                       │
    ▼                                       ▼
Left panel: file tree                  Right panel: preview
(built from zip entries)               (click a file to render)
    │                                       │
    ├─ Click file → extract entry      ├─ Reuse existing renderers
    │    from zip → preview                 (PDF, image, markdown,
    │                                        code, text)
    ├─ "Save This File" → saveFile()
    │    (single entry download)
    │
    ├─ "Save All (zip)" → saveFile()
    │    (whole zip download)
    │
    └─ "Save Folder: X" → re-zip
         subtree → saveFile()
```

## 5. Memory Considerations

The entire zip is already in browser memory (it was decrypted there). Parsing it with JSZip adds minimal overhead — JSZip reads the zip's central directory without decompressing all entries. Individual files are only decompressed when previewed or downloaded.

| Operation | Memory cost |
|-----------|------------|
| Parse zip (read directory) | Negligible (~few KB for the directory index) |
| Preview one file | Size of that one decompressed file |
| Download one file | Size of that one decompressed file (temporary) |
| Save subfolder | Sum of files in subfolder (temporary, during re-zip) |

**No additional memory concern** beyond what already exists — the full decrypted zip is in memory regardless.

## 6. Security Invariants

| Invariant | Preserved? | Rationale |
|-----------|------------|-----------|
| Server never sees plaintext | Yes | Zip is parsed client-side after decryption |
| Server never sees filenames | Yes | Filenames are inside the zip, only visible after decryption |
| Server never sees individual files | Yes | File extraction happens in browser |
| No new server endpoints | Yes | All zip operations are client-side |
| Zero-knowledge guarantee | Yes | JSZip operates on already-decrypted bytes |

## 10. Interaction with Folder Upload Proposal

These two proposals are **complementary but independent**:

| Scenario | Folder Upload | Zip Viewer |
|----------|---------------|------------|
| User uploads a folder | Needed | Not needed (recipient gets a .zip, unzips locally) |
| User uploads a .zip file manually | Not needed | Enhances download experience |
| User uploads a folder AND recipient wants to browse | Needed | Enhances download experience |

**Recommended order:**
1. Ship folder upload first (simpler, upload-side only)
2. Ship zip viewer second (enhances all .zip transfers, not just folder uploads)

Both share JSZip as a dependency, so the second to ship gets it for free.

## 11. Open Questions for Human Decision

1. **Auto-expand tree?** Should the file tree be fully expanded by default, or collapsed to top-level folders? For small zips (< 20 entries), fully expanded makes sense. For large zips, collapsed is better.

2. **Preview size limit?** Should we limit which files can be previewed from within the zip? E.g., skip preview for files > 50 MB to avoid memory pressure.

3. **Nested zips?** If a zip contains another `.zip` inside it, should we recursively browse into it? This adds complexity but could be useful for deeply nested archives. Recommendation: defer to a future iteration.

4. **File count warning?** Zips with thousands of files could make the tree UI sluggish. Should we warn or paginate at some threshold (e.g., > 500 files)?

5. **First file auto-preview?** When the zip viewer loads, should it auto-preview the first previewable file, or show an empty right panel until the user clicks?

---

## 12. Recommendation

**Build this as a single IFD layer** after the folder upload feature ships (to share the JSZip dependency). The implementation reuses existing preview renderers for individual files, making the core work the file tree UI and zip parsing — both straightforward with JSZip.

The feature enhances all `.zip` transfers (not just folder uploads) — any user who manually uploads a `.zip` file will benefit from the content viewer on the recipient's side.
