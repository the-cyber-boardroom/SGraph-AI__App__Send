## Executive Summary

The proposal is architecturally sound. The page eliminations, inline form, and direct Stripe links are all achievable within the existing static site infrastructure (CloudFront + S3) and Lambda architecture. There are six technical areas that require attention, two of which are blocking for the Challenge campaign. This review details each, with specific file paths and code references.

## 1. Routing & URL Architecture

### 1.1 Pages Proposed for Elimination

The proposal eliminates three pages. Here is the current state of each:

| Page | Source file | i18n copies | hreflang tags | Canonical URL | Inbound links from other pages |
|------|------------|-------------|---------------|---------------|-------------------------------|
| `/pricing/cloud/` | `sgraph_ai__website/en-gb/pricing/cloud/index.html` | 16 locale copies (generated by `scripts/generate_i18n_pages.py`) | Yes, all 17 locales | `https://sgraph.ai/en-gb/pricing/cloud/` | Homepage `./pricing/cloud/` (line 171), Pricing page `./cloud/` (line 438), `/pricing/free/` page `../cloud/` (line 141) |
| `/pricing/free/` | `sgraph_ai__website/en-gb/pricing/free/index.html` | 16 locale copies | Yes, all 17 locales | `https://sgraph.ai/en-gb/pricing/free/` | Homepage `./pricing/free/` (lines 85, 161), Pricing page `./free/` (line 382) |
| `/early-access/` | `sgraph_ai__website/en-gb/early-access/index.html` | 16 locale copies | Yes, all 17 locales | `https://sgraph.ai/en-gb/early-access/` | Pricing page `../early-access/` (line 322), `/pricing/free/` page `../../early-access/` (line 140) |

**Impact assessment:**

- **51 HTML files to remove** (3 pages x 17 locales). The source files live under `sgraph_ai__website/en-gb/` and copies are generated at deploy time by `scripts/generate_i18n_pages.py` into each locale directory.

- **SEO risk:** All three pages have `<link rel="canonical">` tags and are included in hreflang tag sets across all 17 locales. Google has indexed these URLs. Removing them without redirects will cause 404s for bookmarked links and degrade the hreflang signal chain.

- **Stripe Payment Link:** The only place the Stripe URL (`https://buy.stripe.com/fZu8wHdv23nN1Z85jn5os02`) appears in the codebase is in `/pricing/cloud/index.html` line 298. If this page is eliminated, the Stripe URL must be moved to the homepage hero and pricing page before the cloud page is removed.

### 1.2 Redirect Strategy Required

**Recommendation: CloudFront Function redirects, not S3 redirects.**

The existing CloudFront Function (`sgraph_ai__website/cloudfront/url-rewrite.js`) handles trailing-slash rewrites. It should be extended (or a second function added on the same Viewer Request event) to handle the three redirects:

| Source path pattern | Target | HTTP code | Rationale |
|---|---|---|---|
| `/{locale}/pricing/cloud/` | `/{locale}/pricing/#cloud` | 301 | Permanent. SEO transfers link equity. |
| `/{locale}/pricing/free/` | `/{locale}/pricing/#free` | 301 | Permanent. Matches proposal's anchor suggestion. |
| `/{locale}/early-access/` | `/{locale}/` | 301 | Permanent. Form is now inline on homepage. |

The redirect function must preserve the locale prefix. Example addition to `url-rewrite.js`:

```javascript
// Page elimination redirects (v0.10.9)
var redirects = {
    '/pricing/cloud/': '/pricing/#cloud',
    '/pricing/free/':  '/pricing/#free',
    '/early-access/':  '/'
};
// Strip locale prefix, check, then rebuild with locale
var localeMatch = uri.match(/^\/([a-z]{2}(-[a-z]{2})?)(\/.*)$/);
if (localeMatch) {
    var locale = localeMatch[1];
    var pagePath = localeMatch[3];
    if (redirects[pagePath]) {
        return {
            statusCode: 301,
            statusDescription: 'Moved Permanently',
            headers: { location: { value: '/' + locale + redirects[pagePath] } }
        };
    }
}
```

**Important:** CloudFront Functions execute before the URL rewrite that appends `index.html`. The redirect check must happen first in the handler, before the existing trailing-slash logic.

### 1.3 i18n Page Generation Impact

The `scripts/generate_i18n_pages.py` script (line 88, `find_html_files()`) discovers all `.html` files under `sgraph_ai__website/en-gb/` and replicates them to all 16 other locale directories. When we delete the three source pages from `en-gb/`, the script will automatically stop generating their locale copies. No script changes needed.

However, the hreflang tags embedded in the remaining pages reference the eliminated pages. The `_build_hreflang_tags()` function (line 199) builds hreflang entries based on `rel_path` for all locales, but does not include cross-page references. The hreflang concern is limited to:

1. **The existing en-gb source pages** already have hreflang tags baked in (lines 18-34 of each page). These are static, not generated. When `/pricing/cloud/index.html` is deleted, other pages' hreflang tags are unaffected because hreflang tags only point to the same page in other locales, not to other pages.

2. **The `scripts/generate_sitemap.py`** (referenced in `scripts/`) will need updating to remove the three eliminated pages from the sitemap.

### 1.4 Sequencing Constraint

**The Stripe URL must be live on the homepage and pricing page BEFORE `/pricing/cloud/` is deleted.** Currently the Stripe Payment Link only exists on the cloud detail page. This is a hard ordering dependency.

## 2. Inline Form Architecture

### 2.1 The Form POST Target

The LaunchList replacement brief specifies: `POST /api/early-access/signup` on `send.sgraph.ai`.

**Architectural issue: The website is `sgraph.ai`, the API is `send.sgraph.ai`.**

The inline form on the homepage (`sgraph.ai/en-gb/`) will POST to a different origin (`send.sgraph.ai`). This is a **cross-origin request** and requires:

1. **CORS on the User Lambda** -- already configured. The `setup_middleware__cors()` method in `sgraph_ai_app_send/lambda__user/fast_api/Fast_API__SGraph__App__Send__User.py` (line 65) allows `*` origins. This is sufficient, but the `allow_methods` list (line 71) only includes `GET, POST, HEAD, OPTIONS`. `POST` is already allowed, so CORS is covered.

2. **The form `action` attribute** must use an absolute URL: `action="https://send.sgraph.ai/api/early-access/signup"` (not a relative path, since the form is served from `sgraph.ai`).

### 2.2 Route Structure Fit

The proposed endpoint path `POST /api/early-access/signup` does **not** fit the current User Lambda route structure. Current routes are:

- `/transfers/*` (Routes__Transfers)
- `/presigned/*` (Routes__Presigned)
- `/join/*` (Routes__Join -- currently removed per line 85 comment)
- `/` (redirect)
- `/mcp` (MCP server)

No `/api/` prefix exists on any current route. The User Lambda mounts routes directly at the path prefix level (e.g., `/transfers/create`, not `/api/transfers/create`).

**Two options:**

**Option A: Follow existing conventions.** Use `POST /early-access/signup`. This is consistent with the flat route structure. A new `Routes__Early_Access` class would be created following the same pattern as `Routes__Transfers`.

**Option B: Use the `/api/` prefix as proposed.** This works but creates an inconsistency. All other User Lambda routes are un-prefixed. If we plan to adopt `/api/` as a standard prefix (for CloudFront path-based routing to Lambda), then this is a good time to start -- but it should be a deliberate decision, not an accidental one.

**Recommendation: Option A** (`POST /early-access/signup`) for consistency. The CloudFront routing concern (separating `/api/*` to Lambda from `/*` to S3) is a separate infrastructure decision. The User Lambda at `send.sgraph.ai` does not serve static content from S3 via the same CloudFront distribution -- it has its own Lambda URL.

### 2.3 Where the Endpoint Lives

The early-access signup endpoint belongs on the **User Lambda** (public, unauthenticated), not the Admin Lambda (auth-protected). The form is public-facing and must not require authentication.

Implementation requirements:
- New route class: `sgraph_ai_app_send/lambda__user/fast_api/routes/Routes__Early_Access.py`
- Registration in `Fast_API__SGraph__App__Send__User.setup_routes()` (line 75)
- Email sending: SES via `osbot-aws` (never `boto3` directly per project rules)
- No persistence required (per brief: email notification is the record)

### 2.4 Frontend Form Behaviour

The inline form replaces the LaunchList form. Currently, the early-access page at `sgraph_ai__website/en-gb/early-access/index.html` has:

```html
<form class="eap-form launchlist-form" action="https://getlaunchlist.com/s/zSa8gI" method="POST">
```

The proposed inline form will need:

1. A `<form>` element with `action="https://send.sgraph.ai/early-access/signup"` and `method="POST"`.
2. JavaScript to intercept the submit, send via `fetch()` to avoid a full page navigation, and swap the form UI to the confirmation state.
3. Error handling for network failures, validation errors, and rate limiting.
4. The form must be a reusable HTML snippet since it appears on both the homepage and the pricing page. Options:
   - Duplicate the HTML (simple, but maintenance burden across 2 pages x 17 locales = 34 instances).
   - Extract to a Web Component (`<sg-signup-form>`). This follows the project's IFD methodology and is the architecturally cleaner approach.

**Recommendation: Web Component approach.** Create `<sg-signup-form>` as a small self-contained Web Component in `sgraph_ai__website/_common/js/signup-form.js`. Both the homepage and pricing page include `<sg-signup-form></sg-signup-form>` where the form should appear. The component handles rendering, submission, state transitions, and i18n attributes. This avoids duplicating form HTML across 34 files and keeps the inline form logic in one place.

### 2.5 i18n Impact on the Form

The form text ("Name", "Email", "Join Early Access Programme", "We'll send your token within 48h") needs translation keys in all 17 locale JSON files under `sgraph_ai__website/i18n/`. New keys must be added:

- `signup.form.title`
- `signup.form.name_placeholder`
- `signup.form.email_placeholder`
- `signup.form.submit`
- `signup.form.hint`
- `signup.form.success_title`
- `signup.form.success_message`

These are new keys that must be added to `sgraph_ai__website/i18n/en-gb.json` and all other locale JSON files before the i18n page generator runs.

## 3. Stripe Integration Points

### 3.1 Direct Link Architecture

The "Buy Credits" button going directly to Stripe is architecturally trivial. It is a simple `<a href="https://buy.stripe.com/fZu8wHdv23nN1Z85jn5os02">` element. No server-side integration is needed. The Stripe Payment Link handles everything (payment collection, receipt emails, checkout UI).

**The Stripe URL** currently only appears in one place: `sgraph_ai__website/en-gb/pricing/cloud/index.html` line 298. It must be:

1. Added to the homepage hero CTA (`sgraph_ai__website/en-gb/index.html`)
2. Added to the pricing page Multi-Tenant card (`sgraph_ai__website/en-gb/pricing/index.html`)
3. Added to the homepage "Get Started" section's "Buy Credits" button

**Important: This URL should NOT be a translatable i18n string.** The Stripe Payment Link URL is configuration, not content. It should be hardcoded in the HTML. If the URL changes, it changes in the source HTML files and propagates to all locales via the i18n generator (which copies everything, not just translated strings).

### 3.2 Stripe Success/Cancel Redirect URLs

Stripe Payment Links are configured in the Stripe Dashboard with success and cancel redirect URLs. The existing pages are:

- **Success:** `sgraph_ai__website/en-gb/payment/success/index.html` -- exists and is well-structured. It shows "Payment successful", next steps, and links to `send.sgraph.ai`.
- **Cancel:** `sgraph_ai__website/en-gb/payment/cancel/index.html` -- exists and is well-structured. It shows "Payment cancelled" with a "Back to Pricing" CTA.

**Verification needed:** The Stripe Payment Link must be configured with:
- Success URL: `https://sgraph.ai/en-gb/payment/success/`
- Cancel URL: `https://sgraph.ai/en-gb/payment/cancel/`

**Locale issue:** If a user is on the `fr-fr` version of the site and clicks "Buy Credits", they go to Stripe. On success, they are redirected to `/en-gb/payment/success/` (the hardcoded Stripe config), not `/fr-fr/payment/success/`. This is a minor UX issue but not blocking. The payment success page has i18n support and could be redirected to the correct locale version, but Stripe Payment Links do not support dynamic redirect URLs. Accept this as a known limitation for now.

### 3.3 No Additional Backend Work for Stripe

The Stripe Payment Link is a hosted Stripe product. No endpoints, webhooks, or server-side integration are required on the SGraph Send side. Token creation after payment is currently a manual process (confirmed by the payment success page: "We'll create your access token, typically within a few hours"). This is acceptable for the Challenge campaign volume (~50-100 users).

## 4. Static Site Deployment Impact

### 4.1 Deployment Pipeline

The deployment pipeline is documented in `.github/workflows/deploy-website.yml`:

```
1. Checkout
2. Read version from sgraph_ai_app_send/version
3. Generate i18n locale pages (python scripts/generate_i18n_pages.py)
4. Deploy static site (python scripts/deploy_static_site.py)
5. Store CI artifacts
```

The proposed changes affect steps 3 and 4:

**Step 3 (i18n generation):** If we add or remove pages from `sgraph_ai__website/en-gb/`, the generator automatically picks up the changes. No script modifications needed -- `find_html_files()` dynamically discovers HTML files.

**Step 4 (S3 deploy):** The deploy script uses `aws s3 sync --delete`. When pages are removed from the source directory, `--delete` removes them from S3. This is the correct behaviour.

**However:** The S3 sync happens to both `releases/{version}/` and `latest/`. Old versions (with the eliminated pages) remain in S3 under their release paths. This is by design (IFD immutable releases) and is not a problem.

### 4.2 CloudFront Invalidation

After deployment, the script runs `aws cloudfront create-invalidation --paths "/*"`. This invalidates all cached paths, including the eliminated pages. Combined with the CloudFront Function redirects (Section 1.2), eliminated URLs will serve 301 redirects instead of 404s.

**Timing dependency:** The CloudFront Function update (adding redirects) must be deployed BEFORE the S3 sync that removes the pages. Otherwise there is a window where the pages return 404 instead of 301. The CloudFront Function is deployed separately from the S3 content (it is not part of `deploy_static_site.py`). This needs to be a manual coordinated deployment or added to the CI pipeline.

### 4.3 Deploy Script Validation

The `validate_website_files()` function in `scripts/deploy_static_site.py` (line 137) checks for required files:

```python
required_files = [
    "index.html",
    "product/index.html",
    "agents/index.html",
    "architecture/index.html",
    "contact/index.html",
]
```

The eliminated pages (`pricing/cloud/index.html`, `pricing/free/index.html`, `early-access/index.html`) are NOT in the required files list. No deploy script changes needed.

### 4.4 New Static Assets

If the inline signup form uses a Web Component (`<sg-signup-form>`), a new JavaScript file must be added to `sgraph_ai__website/_common/js/`. The deploy script already syncs all `.js` files from `_common/` with appropriate cache headers (1-day cache for CSS/JS). No deploy script changes needed.

## 5. Navigation Changes

### 5.1 Current Navigation Order

The nav is hardcoded in each HTML page's `<nav class="site-nav">` element:

```html
<a href="./product/">Product</a>
<a href="./agents/">Team</a>
<a href="./architecture/">Architecture</a>
<a href="./pricing/">Pricing</a>
```

### 5.2 Change Scope

Moving "Pricing" to position 2 requires editing the `<nav>` element in every HTML source file under `sgraph_ai__website/en-gb/`. There are currently **21 HTML files** in `en-gb/` (from the glob results). Each has its own nav with depth-adjusted relative paths (e.g., `./pricing/` from homepage, `../pricing/` from first-level pages, `../../pricing/` from second-level pages).

The nav order change is a **text-level change** -- swap the order of the `<a>` elements. The `href` values and `data-i18n` keys remain unchanged.

### 5.3 i18n System Impact

The i18n page generator copies the HTML structure wholesale and only translates `data-i18n` content. Changing the nav order in the en-gb source pages will automatically propagate to all 16 generated locale pages. No i18n JSON changes needed (the translation keys `nav.product`, `nav.team`, `nav.architecture`, `nav.pricing` remain the same).

### 5.4 Component Architecture

There is no shared nav component or template. The nav is duplicated in every HTML file. This is a known architectural weakness but is consistent with the zero-dependency, vanilla-JS approach. The nav reorder must be applied to all 21 source files individually.

**Recommendation:** This is a good candidate for a future Web Component (`<sg-nav>`) or at minimum a build-time include. For now, the manual edit across 21 files is the path of least resistance.

## 7. Additional Architectural Observations

### 7.1 CORS for the Signup Form

The inline form on `sgraph.ai` will POST to `send.sgraph.ai`. The User Lambda's CORS configuration (`allow_origins = ["*"]`) permits this. However, the form submission will trigger a **CORS preflight** (OPTIONS request) because:
- It is a cross-origin POST.
- The `Content-Type` header will be `application/json` (not a CORS-safe content type).

The existing CORS middleware handles OPTIONS via Starlette's `CORSMiddleware`, so this should work without additional configuration. But this must be tested end-to-end before the Challenge launch.

### 7.2 Rate Limiting

The LaunchList replacement brief says "No rate limiting beyond basic sanity" and estimates ~50-100 signups. This is acceptable for the Challenge campaign. However, the endpoint should not be completely unprotected -- a simple in-memory counter (e.g., max 10 signups per IP per hour) would prevent abuse without requiring external infrastructure. This can be a post-launch enhancement.

### 7.3 The Stripe URL as Configuration

The Stripe Payment Link URL (`https://buy.stripe.com/fZu8wHdv23nN1Z85jn5os02`) is currently hardcoded in one HTML file. After the redesign, it will appear in at least 3 places (homepage hero, homepage get-started section, pricing page card). If the Stripe URL changes (e.g., new pricing, new payment link), all instances must be updated.

**Options:**
- Accept the duplication (3 instances in en-gb, auto-propagated to 17 locales). This is simple and matches the current approach.
- Define the URL as a JavaScript constant and render it dynamically. This adds complexity for minimal gain at current scale.

**Recommendation:** Accept the duplication. Document the Stripe URL location in a comment at each instance.

### 7.4 No /payment/success/ Locale Routing

As noted in Section 3.2, Stripe redirects to a fixed URL after payment. If the Stripe Payment Link is configured with `success_url=https://sgraph.ai/en-gb/payment/success/`, users from non-English locales will see the English success page. The i18n system supports locale-specific versions of the payment success page (hreflang tags are present), but Stripe Payment Links do not support dynamic redirect URLs.

This is not a blocking issue. The payment success page content is minimal and the key information (receipt, next steps) is understandable in English for the Challenge campaign's target audience. If needed later, a JavaScript snippet on the success page could detect the user's preferred language and redirect to the correct locale version.

## Verdict

**The proposal is architecturally viable.** The main risks are:

1. **Blocking risk:** The Stripe URL must be added to the homepage BEFORE `/pricing/cloud/` is eliminated. Mitigated by the sequencing in Section 6.3.
2. **CORS risk:** The cross-origin signup form POST from `sgraph.ai` to `send.sgraph.ai` must be tested. Mitigated by existing CORS middleware.
3. **Redirect gap risk:** CloudFront Function redirects must be deployed before pages are removed from S3. Mitigated by making redirect deployment a prerequisite step.

None of these are architectural objections. They are sequencing constraints that the Conductor should enforce in the implementation plan.

**Approved with conditions:**
- Follow the execution order in Section 6.3.
- Test CORS for the cross-origin form POST before Challenge launch.
- Deploy CloudFront Function redirects before removing pages.
- Update the sitemap generator when pages are eliminated.
- Add the Stripe URL to homepage and pricing page before eliminating `/pricing/cloud/`.
