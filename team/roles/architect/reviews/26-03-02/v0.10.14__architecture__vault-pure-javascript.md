**Version:** v0.10.14 | **Date:** 2 Mar 2026 | **Role:** Architect

## Executive Summary

vault.sgraph.ai is a pure HTML/JS static site that provides encrypted vault functionality. It has **no backend of its own**. All file storage, retrieval, and token validation go through the existing send.sgraph.ai service. From send.sgraph.ai's perspective, vault files are just regular encrypted files -- the server knows nothing about vaults, folders, or file relationships. All vault semantics (tree structure, settings, navigation, rendering) live entirely in client-side JavaScript.

This architecture document covers the eight dimensions requested: system topology, component architecture, data flows, server-side changes, encryption architecture, multi-cultural architecture, deployment architecture, and cross-origin considerations.

### Key Topology Decisions

**D-001: vault.sgraph.ai is a separate CloudFront distribution, NOT a path on send.sgraph.ai.**

Rationale: The vault is a distinct product with its own domain, its own deployment cycle, and its own cache invalidation rules. Sharing a distribution with send.sgraph.ai would couple their deployment lifecycles. The existing `deploy_static_site.py` script already supports deploying to any S3 bucket + CloudFront combination.

**D-002: vault.sgraph.ai uses the User Lambda endpoints on send.sgraph.ai, NOT the Admin Lambda.**

Rationale: The Admin Lambda requires cookie-based authentication and is designed for internal/admin operations. The User Lambda is public-facing and already has the transfer lifecycle endpoints (create, upload, complete, download) that vault needs. The User Lambda's CORS middleware already allows `*` origins (see `Fast_API__SGraph__App__Send__User.setup_middleware__cors()`). This means vault.sgraph.ai can call send.sgraph.ai immediately with no CORS changes required.

**D-003: DNS follows the existing pattern.**

vault.sgraph.ai gets a CNAME to its CloudFront distribution, matching the pattern for sgraph.ai. Route 53 hosted zone: sgraph.ai. ACM certificate for `vault.sgraph.ai` in us-east-1 (CloudFront requirement).

### 2.1 sg-send.js -- Core API Client

This is the reusable client library for talking to send.sgraph.ai. It abstracts the transfer lifecycle into simple methods. The existing `ApiClient` in `v0.2.0/_common/js/api-client.js` is the prototype; `sg-send.js` is its evolution.

```javascript
class SGSend {
    constructor({ endpoint, token }) {
        this.endpoint = endpoint;   // 'https://send.sgraph.ai'
        this.token    = token;      // access token for upload auth
    }

    // ── Transfer lifecycle ──────────────────────────────────────────────
    async createTransfer(fileSize, contentType)    { /* POST /api/transfers/create   */ }
    async uploadPayload(transferId, encryptedData) { /* POST /api/transfers/upload   */ }
    async completeTransfer(transferId)             { /* POST /api/transfers/complete */ }
    async getTransferInfo(transferId)               { /* GET  /api/transfers/info     */ }
    async downloadPayload(transferId)              { /* GET  /api/transfers/download  */ }
    async downloadBase64(transferId)               { /* GET  /api/transfers/download-base64 */ }

    // ── Token validation ────────────────────────────────────────────────
    async checkToken(tokenName)                     { /* GET  /api/transfers/check-token */ }

    // ── Presigned URLs (large files) ────────────────────────────────────
    async getCapabilities()                         { /* GET  /api/presigned/capabilities  */ }
    async initiateMultipart(transferId, size, parts) { /* POST /api/presigned/initiate    */ }
    async uploadPart(url, data)                     { /* PUT  to presigned URL             */ }
    async completeMultipart(transferId, uploadId, parts) { /* POST /api/presigned/complete */ }
    async getPresignedDownloadUrl(transferId)       { /* GET  /api/presigned/download-url  */ }

    // ── Convenience ─────────────────────────────────────────────────────
    async upload(plainBytes, key) {
        // 1. Encrypt with sg-crypto
        // 2. Create transfer
        // 3. Upload (direct or multipart based on size)
        // 4. Complete transfer
        // Returns { transfer_id, size }
    }

    async download(transferId, key) {
        // 1. Download encrypted payload
        // 2. Decrypt with sg-crypto
        // Returns ArrayBuffer of plaintext
    }
}
```

**Key design decision:** sg-send.js uses the `/api/` prefix on all routes, matching the v0.2.0 User UI convention. This prefix was introduced for CloudFront behaviour separation (`/api/*` goes to Lambda, `/*` goes to S3). When calling cross-origin from vault.sgraph.ai, the full URL is `https://send.sgraph.ai/api/transfers/create`.

### 2.2 sg-crypto.js -- Encryption Layer

This consolidates and extends the existing `SendCrypto` module from `v0.2.0/_common/js/crypto.js`. The key addition is KDF (Key Derivation Function) support for deriving AES-256 keys from human-readable vault keys.

```javascript
const SGCrypto = {

    // ── Existing (from SendCrypto) ──────────────────────────────────────
    ALGORITHM:  'AES-GCM',
    KEY_LENGTH: 256,
    IV_LENGTH:  12,

    isAvailable()                      { /* check window.crypto.subtle      */ },
    async generateKey()                { /* generate random AES-256 key     */ },
    async exportKey(key)               { /* export to base64url string      */ },
    async importKey(base64Key)         { /* import from base64url string    */ },
    async encrypt(key, plaintext)      { /* AES-256-GCM encrypt            */ },
    async decrypt(key, ciphertext)     { /* AES-256-GCM decrypt            */ },

    // ── New: Key Derivation ─────────────────────────────────────────────
    async deriveKey(vaultKey, salt) {
        // PBKDF2: vaultKey (string) + salt (Uint8Array) -> AES-256 CryptoKey
        //
        // Why PBKDF2 over Argon2:
        //   - PBKDF2 is natively supported by Web Crypto API
        //   - Argon2 requires a WASM library (~100KB), violating zero-dependency rule
        //   - PBKDF2 with 600,000 iterations provides adequate protection
        //     for vault keys that are expected to be reasonably strong
        //
        // Parameters:
        //   - Hash: SHA-256
        //   - Iterations: 600,000 (OWASP 2023 recommendation for PBKDF2-SHA256)
        //   - Key length: 256 bits
        //   - Salt: 16 bytes, generated per vault and stored in vault settings
        //
        const encoder   = new TextEncoder();
        const keyMaterial = await crypto.subtle.importKey(
            'raw', encoder.encode(vaultKey), 'PBKDF2', false, ['deriveKey']
        );
        return crypto.subtle.deriveKey(
            { name: 'PBKDF2', salt, iterations: 600000, hash: 'SHA-256' },
            keyMaterial,
            { name: 'AES-GCM', length: 256 },
            false,
            ['encrypt', 'decrypt']
        );
    },

    generateSalt() {
        return crypto.getRandomValues(new Uint8Array(16));
    },

    // ── Helpers ─────────────────────────────────────────────────────────
    bufferToBase64Url(buffer)          { /* existing */ },
    base64UrlToBuffer(base64url)       { /* existing */ },
    bufferToHex(buffer)                { /* new: for vault IDs              */ },
    generateId(length = 8)             { /* crypto random hex ID            */ }
};
```

### 2.3 sg-vault.js -- Vault Logic

This is the vault-specific layer built on top of sg-send.js and sg-crypto.js. It manages vault creation, opening, tree operations, and file operations. The server never sees any of this structure -- it is purely client-side.

```javascript
class SGVault {
    constructor({ sgSend, vaultKey, vaultId }) {
        this.sgSend   = sgSend;       // SGSend instance
        this.vaultKey = vaultKey;      // Human-readable vault key
        this.vaultId  = vaultId;       // 8-char hex ID (first segment of path)
        this.aesKey   = null;          // Derived AES-256 key (set after open/create)
        this.settings = null;          // Decrypted vault-settings.json
        this.tree     = null;          // Decrypted vault-tree.json
    }

    // ── Vault Lifecycle ─────────────────────────────────────────────────

    static async create(sgSend, { vaultName, description, owner, vaultKey }) {
        // 1. Generate vault ID (8 hex chars)
        // 2. Generate KDF salt (16 bytes)
        // 3. Derive AES key from vaultKey + salt
        // 4. Create settings object, encrypt, upload as {vaultId}/settings
        // 5. Create empty tree object, encrypt, upload as {vaultId}/tree
        // Returns SGVault instance
    }

    async open() {
        // 1. Fetch {vaultId}/settings from server
        // 2. Extract salt from unencrypted prefix (see Encryption Architecture)
        // 3. Derive AES key from vaultKey + salt
        // 4. Decrypt settings
        // 5. Fetch and decrypt {vaultId}/tree
        // 6. Populate this.settings and this.tree
    }

    // ── File Operations ─────────────────────────────────────────────────

    async addFile(virtualPath, fileData, metadata) {
        // 1. Generate file ID (8 hex chars)
        // 2. Encrypt fileData with this.aesKey
        // 3. Upload encrypted file as {vaultId}/{fileId}
        // 4. Update tree: add entry at virtualPath with fileId, size, mime, date
        // 5. Re-encrypt tree, upload as {vaultId}/tree
        // Returns { fileId, virtualPath }
    }

    async getFile(fileId) {
        // 1. Download {vaultId}/{fileId} from server
        // 2. Decrypt with this.aesKey
        // Returns ArrayBuffer of plaintext
    }

    async deleteFile(virtualPath) {
        // 1. Remove entry from tree (do NOT delete from server -- orphaned file)
        // 2. Re-encrypt tree, upload as {vaultId}/tree
    }

    // ── Folder Operations ───────────────────────────────────────────────

    createFolder(virtualPath) {
        // Modify tree in memory (no server call needed)
        // Re-encrypt tree, upload as {vaultId}/tree
    }

    deleteFolder(virtualPath) {
        // Remove folder and all children from tree
        // Re-encrypt tree, upload as {vaultId}/tree
    }

    // ── Tree Operations ─────────────────────────────────────────────────

    getTree()                          { return this.tree; }
    getSettings()                      { return this.settings; }
    listFolder(virtualPath)            { /* navigate tree, return children */ }
    findFile(virtualPath)              { /* navigate tree, return file entry */ }

    // ── Save State ──────────────────────────────────────────────────────

    async saveTree() {
        // Encrypt tree, upload as {vaultId}/tree (overwrite)
    }

    async saveSettings() {
        // Encrypt settings, upload as {vaultId}/settings (overwrite)
    }
}
```

### 2.4 Web Components

**Pattern reuse:** The vault UI reuses the Admin UI patterns that are "working really well":

1. **Light DOM** -- like `admin-shell`, vault-shell uses light DOM for layout (no Shadow DOM encapsulation needed for the shell). Inner components may use Shadow DOM via `SendComponent` base class.
2. **EventBus** -- `event-bus.js` from the Admin UI is reused verbatim. All component communication goes through `window.sgraphVault.events`.
3. **CSS Design System** -- `_common/css/design-tokens.css` provides design tokens. The vault uses the same Aurora theme (dark navy background, teal accent) as send.sgraph.ai.
4. **IFD versioning** -- components are versioned separately. `vault-shell` is at `v0/v0.1/v0.1.0/`. Individual components can be patched without touching the shell.
5. **Section-based navigation** -- components declare `data-nav-section` to group in the sidebar, matching admin-shell's auto-registration pattern.

## 3. Data Flows

### 3.1 Vault Creation Flow

```
User                    vault.sgraph.ai (JS)            send.sgraph.ai (API)
  |                            |                                |
  |  1. Enter vault name,     |                                |
  |     description,          |                                |
  |     vault key             |                                |
  |  -----------------------> |                                |
  |                            |                                |
  |                    2. Generate:                             |
  |                       - vault_id (8 hex chars)             |
  |                       - KDF salt (16 bytes)                |
  |                       - Derive AES key from                |
  |                         vault_key + salt                   |
  |                            |                                |
  |                    3. Build settings.json:                  |
  |                       { vault_name, vault_id,              |
  |                         created, owner, version,           |
  |                         kdf_salt (base64) }                |
  |                            |                                |
  |                    4. Encrypt settings.json                |
  |                       (AES-256-GCM)                        |
  |                            |                                |
  |                    5. Prepend salt to ciphertext:           |
  |                       [salt_length(2 bytes)]               |
  |                       [salt(16 bytes)]                     |
  |                       [IV(12 bytes)]                       |
  |                       [ciphertext]                         |
  |                            |                                |
  |                            | POST /api/transfers/create    |
  |                            | { file_size_bytes, type }     |
  |                            | -----------------------------> |
  |                            |                                |
  |                            | { transfer_id: "settings_tid" }|
  |                            | <----------------------------- |
  |                            |                                |
  |                            | POST /api/transfers/upload/    |
  |                            |   {settings_tid}              |
  |                            | body: [encrypted settings]    |
  |                            | -----------------------------> |
  |                            |                                |
  |                            | POST /api/transfers/complete/  |
  |                            |   {settings_tid}              |
  |                            | -----------------------------> |
  |                            |                                |
  |                    6. Build tree.json:                      |
  |                       { version: 1, updated, tree: { ... }}|
  |                            |                                |
  |                    7. Encrypt tree.json                    |
  |                       (AES-256-GCM, same key)             |
  |                            |                                |
  |                            | POST /api/transfers/create    |
  |                            | POST /api/transfers/upload/   |
  |                            |   {tree_tid}                  |
  |                            | POST /api/transfers/complete/ |
  |                            |   {tree_tid}                  |
  |                            | -----------------------------> |
  |                            |                                |
  |                    8. Build vault manifest (unencrypted):   |
  |                       { vault_id, settings_tid, tree_tid } |
  |                            |                                |
  |                    9. Encode manifest as URL fragment:      |
  |                       vault.sgraph.ai/#vault_id.           |
  |                         settings_tid.tree_tid              |
  |                            |                                |
  |  10. Display:             |                                |
  |      - Vault created      |                                |
  |      - Vault Key (share)  |                                |
  |      - Vault URL (has     |                                |
  |        transfer IDs in    |                                |
  |        fragment, NOT key) |                                |
  |  <----------------------- |                                |
```

**Storage format:** The vault's transfer IDs (for settings file and tree file) plus the vault_id form a "vault address" that can be shared via URL. The vault key is shared separately (out-of-band). This separation means:

- URL fragment: `#vault_id.settings_tid.tree_tid` -- public, shareable, no secret
- Vault key: shared via messaging, email, etc. -- secret, needed to decrypt

**Alternative (simpler MVP):** Store vault_id + settings_tid + tree_tid in localStorage. The vault key is the only thing the user needs to remember. On first access, the vault address is entered alongside the key. This avoids encoding IDs in URLs for the MVP.

## 4. Server-Side Changes

### 4.1 Path-Based File Organisation

The brief proposes path-based file organisation with up to 5 levels of Object IDs. **However, this is NOT required for the vault MVP.** Here is why:

The current transfer system stores files by `transfer_id` (12-char random string). Each vault file gets its own transfer_id. The vault tree (client-side JSON) maps virtual paths to transfer_ids. From the server's perspective, all files are flat -- the "folder structure" exists only in the encrypted tree.json.

**Recommendation:** Defer path-based storage to Phase 2. For MVP, vault files are just regular transfers. This means:

- **Zero server-side changes required for vault MVP**
- The existing transfer lifecycle works unmodified
- All 393 tests continue to pass unchanged
- The vault tree.json handles the hierarchical structure client-side

### 4.2 When Path-Based Storage Makes Sense (Phase 2)

Path-based storage becomes valuable when:

1. **Vault deletion** -- deleting all files under a vault_id prefix in one operation
2. **Vault listing** -- listing all files belonging to a vault without decrypting the tree
3. **Storage quotas** -- counting bytes per vault_id prefix
4. **Sharding** -- distributing files across S3 folders for performance

When implementing, the change is limited to:

- `Transfer__Service.create_transfer()` -- accept an optional `path_prefix` parameter
- `Storage_FS` -- validate path segments (1-5 Object IDs separated by `/`)
- Path traversal prevention -- each segment must match `[a-zA-Z0-9]{8,12}` (Safe_Str__Id)
- No changes to download, info, or any other endpoints

### 4.3 CORS -- No Changes Required

The User Lambda already configures CORS with `allow_origins = ["*"]`:

```python
# From Fast_API__SGraph__App__Send__User.setup_middleware__cors():
self.app().add_middleware(CORSMiddleware,
    allow_origins     = ["*"],
    allow_credentials = True,
    allow_methods     = ["GET", "POST", "HEAD", "OPTIONS"],
    allow_headers     = ["Content-Type", "X-Requested-With", "Origin",
                         "Accept", "Authorization", HEADER__SGRAPH_SEND__ACCESS_TOKEN],
    expose_headers    = ["Content-Type", "X-Requested-With", "Origin",
                         "Accept", "Authorization"])
```

This means vault.sgraph.ai can call send.sgraph.ai immediately. No CORS configuration changes needed.

**Future hardening (Phase 2):** Replace `allow_origins = ["*"]` with an explicit allowlist: `["https://send.sgraph.ai", "https://vault.sgraph.ai", "https://admin.send.sgraph.ai"]`. This is a security improvement but not a blocker for MVP.

### 4.4 Impact on Existing Tests

**Zero impact.** The vault uses the existing transfer lifecycle endpoints with no modifications. All 393 unit tests continue to pass. No new server-side tests are required for the vault MVP.

When path-based storage is added (Phase 2), new tests will be needed:
- Path validation (valid segments, max depth, traversal prevention)
- Path-based listing
- Path-based deletion

## 5. Encryption Architecture

### 5.1 Key Derivation Flow

```
Vault Key (human-readable string, shared between users)
  |
  v
PBKDF2-SHA256
  |  Salt:       16 bytes (random, generated at vault creation)
  |  Iterations: 600,000 (OWASP 2023 minimum for PBKDF2-SHA256)
  |  Hash:       SHA-256
  |
  v
AES-256 CryptoKey (256-bit symmetric key)
  |
  +---> Encrypt/decrypt vault-settings.json
  +---> Encrypt/decrypt vault-tree.json
  +---> Encrypt/decrypt all files in the vault
```

### 5.2 Salt Storage

The KDF salt must be accessible before decryption (chicken-and-egg problem). Solution: **prepend the salt to the encrypted settings file in cleartext.**

```
Encrypted settings file format:
  Bytes 0-1:   Salt length (uint16, big-endian) = 16
  Bytes 2-17:  KDF salt (16 bytes, cleartext)
  Bytes 18-29: IV (12 bytes)
  Bytes 30+:   AES-256-GCM ciphertext

Regular encrypted file format (tree, content files):
  Bytes 0-11:  IV (12 bytes)
  Bytes 12+:   AES-256-GCM ciphertext
```

The salt is not secret -- it prevents rainbow table attacks. Exposing it in cleartext is standard practice (identical to how password hashes store their salt alongside the hash).

### 5.3 How This Differs from the Current Vault

The existing vault (at `/send/v0/v0.1/v0.1.7/vault.html`) uses **RSA-4096 + AES-256-GCM hybrid encryption:**

1. Generate RSA-4096 keypair in browser
2. Export private key (user saves it)
3. For each file: generate random AES key, encrypt file, encrypt AES key with RSA public key
4. Store encrypted file + encrypted AES key

The new vault uses **pure symmetric encryption:**

1. User provides a vault key (passphrase)
2. Derive AES-256 key via PBKDF2
3. All files encrypted with the same derived key

**Why this is simpler:**

| Aspect | RSA-4096 Hybrid (current) | Symmetric KDF (new) |
|--------|---------------------------|----------------------|
| Key management | User must save RSA private key (PEM file) | User remembers a passphrase |
| Key size | 3,392 bytes (RSA-4096 private key) | Human-readable string |
| Sharing | Share the RSA private key file | Share the vault key string |
| Per-file keys | Each file has its own AES key | All files share one derived key |
| Complexity | RSA keygen + AES keygen + RSA encrypt AES key | One PBKDF2 derivation |
| Browser support | Full (Web Crypto API) | Full (Web Crypto API) |

**Trade-off:** The symmetric approach means everyone with the vault key can access everything. No per-file access control in MVP. This is acceptable for the stated use cases (personal vaults, team vaults, project vaults). Per-file key derivation is planned for Phase 3.

### 5.4 Future: Per-Folder Key Derivation (Phase 3)

```
Vault Key
  |
  PBKDF2(salt_root)  --->  Root Key
  |
  HKDF(root_key, "documents/")  --->  Folder Key: /documents/
  HKDF(root_key, "images/")     --->  Folder Key: /images/
  HKDF(folder_key, "report.pdf") ---> File Key: /documents/report.pdf
```

Using HKDF (HMAC-based Key Derivation Function), each folder or file can have its own key derived from the parent. Sharing a folder key grants access to that folder and all its children, without exposing the root key. This is a design-time consideration, not an MVP requirement.

### 5.5 Security Properties Preserved

| Property | Status |
|----------|--------|
| Server never sees plaintext | Preserved -- all encryption in browser |
| No file names on server | Preserved -- file names in encrypted tree.json |
| No decryption keys on server | Preserved -- vault key never sent to server |
| IP addresses hashed | Preserved -- unchanged transfer infrastructure |
| Token-gated uploads | Preserved -- access token required for uploads |

## 8. Cross-Origin Considerations

### 8.1 CORS Policy -- Already Permissive

As documented in Section 4.3, the User Lambda's CORS middleware already allows all origins (`*`). vault.sgraph.ai fetch() calls to send.sgraph.ai will work immediately. The `x-sgraph-access-token` header is already in the `allow_headers` list.

### 8.2 Token Management Across Domains

**Problem:** vault.sgraph.ai and send.sgraph.ai are different origins. localStorage is origin-scoped -- a token stored by send.sgraph.ai is not accessible by vault.sgraph.ai.

**Solution for MVP:** vault.sgraph.ai manages its own token in its own localStorage. Users enter their access token once in the vault UI, and it is stored under the key `sgraph-vault-token` in vault.sgraph.ai's localStorage.

```javascript
// In sg-send.js constructor:
const sg = new SGSend({
    endpoint: 'https://send.sgraph.ai',
    token:    localStorage.getItem('sgraph-vault-token')
});
```

**Future (Phase 2):** Implement cross-origin token sharing via:

1. **Shared token service** -- a `/api/session` endpoint on send.sgraph.ai that returns a short-lived token in a response header
2. **PostMessage** -- an invisible iframe from send.sgraph.ai that communicates the token via `window.postMessage`
3. **Manual entry** -- user pastes their token (current approach, simplest)

For MVP, manual token entry is the right choice. It is explicit, transparent, and requires no server-side changes.

### 8.3 Session/Auth Flow

vault.sgraph.ai has no authentication of its own. There is no login, no user accounts, no session cookies. Authentication is handled entirely by:

1. **Access token** -- validates upload permissions (via `x-sgraph-access-token` header to send.sgraph.ai)
2. **Vault key** -- provides encryption/decryption capability (never sent to server)

A user's "session" with a vault is maintained entirely in browser memory:
- Derived AES key (in memory, CryptoKey object -- cannot be exported to JS)
- Vault tree (in memory, decrypted JSON)
- Vault settings (in memory, decrypted JSON)
- Vault address (localStorage -- transfer IDs for settings/tree)
- Access token (localStorage -- for upload auth)

When the browser tab is closed, the derived key and decrypted data are lost. The user must re-enter the vault key on next visit.

### 8.4 Request Flow Summary

```
vault.sgraph.ai (browser)
  |
```

## 10. Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Tree file race condition (two users upload simultaneously) | Last writer wins -- tree data lost | MVP: accept last-writer-wins. Phase 2: tree versioning with conflict detection (compare tree version before overwrite) |
| KDF performance on mobile | PBKDF2 with 600K iterations may take 1-3 seconds on older phones | Show "Deriving key..." progress indicator. Consider reducing iterations to 300K if UX is unacceptable. |
| Vault key brute force | If attacker has encrypted settings file, they can attempt offline brute force | PBKDF2 600K iterations provides ~$2^{19}$ cost multiplier. Vault keys should be > 20 characters. Consider adding key strength indicator in UI. |
| Transfer ID leakage | Vault address (transfer IDs) could be shared accidentally | Transfer IDs without the vault key are useless -- the server only has encrypted bytes. This is the same security model as send.sgraph.ai links. |
| localStorage cleared | Browser clear or privacy mode loses vault address | Provide "Export vault address" feature. Phase 2: optional cloud backup of vault address (encrypted with vault key). |
| Large vaults (1000+ files) | Tree JSON becomes large, re-encryption on every change is slow | Tree is JSON -- 1000 entries ~= 100KB. Encrypt/upload cycle is ~200ms. Acceptable for MVP. Phase 3: chunked tree or separate tree files per folder. |
