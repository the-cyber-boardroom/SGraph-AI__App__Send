---

## 1. Executive Summary

The upload UX redesign requires **one backend change**: accepting optional client-provided transfer IDs in `POST /api/transfers/create` for PBKDF2 simple-token mode. This change is technically feasible with moderate security implications that are acceptable with proper safeguards (partially in place via `Safe_Str__Id`).

The 10GB file size limit change is **frontend-only**. The state machine redesign (Steps 1-2-3 with deferred encryption) is **100% frontend**. The backend order-of-operations (create -> upload -> complete) does not change.

**Implementation effort:** 4-6 hours (service method + route parameter + validation + tests).

## 2. Current Backend Code Review

### 2.1 Transfer Service: ID Generation

**File:** `sgraph_ai_app_send/lambda__user/service/Transfer__Service.py`

Current `create_transfer()` generates IDs via `secrets.token_hex(6)` (12-char random hex string). No parameter to accept client-provided ID. No existence check for collisions.

**Key fact:** `has_transfer()` method already exists and checks existence via storage. This is the building block for the 409 Conflict mechanism.

### 2.2 Transfer Routes: Create Endpoint

**File:** `sgraph_ai_app_send/lambda__user/fast_api/routes/Routes__Transfers.py`

Current `create()` route receives `Schema__Transfer__Create` request body, validates access token, calls service, returns dict. No optional parameters for client-provided ID.

### 2.3 Transfer Schema

**File:** `sgraph_ai_app_send/lambda__user/schemas/Schema__Transfer.py`

```python
class Schema__Transfer__Create(Type_Safe):
    file_size_bytes   : Safe_UInt__FileSize
    content_type_hint : Safe_Str__Http__Content_Type
```

Needs: optional `transfer_id` field.

### 2.4 Presigned Upload Pipeline

**File:** `sgraph_ai_app_send/lambda__user/service/Service__Presigned_Urls.py`

- **No file size limits enforced** in presigned URL generation
- `MAX_PARTS = 10000` with default 10MB parts = ~100TB theoretical max
- 10GB file = 1024 parts, well within S3 limits
- Single-part presigned upload: S3 accepts up to 5GB per PUT

**Conclusion:** No backend changes needed for 10GB support.

## 3. Required Backend Changes

### Change 1: Schema — Add Optional `transfer_id`

**File:** `sgraph_ai_app_send/lambda__user/schemas/Schema__Transfer.py`

Add optional `transfer_id` field to `Schema__Transfer__Create`. When provided, the server uses it instead of generating a random one.

### Change 2: Service — Accept and Validate Client-Provided ID

**File:** `sgraph_ai_app_send/lambda__user/service/Transfer__Service.py`

Modify `create_transfer()`:
1. Accept optional `transfer_id` parameter
2. If provided, validate format: `^[a-f0-9]{12}$`
3. Check existence via `has_transfer(transfer_id)`
4. If exists, return error dict (route translates to 409)
5. If not provided, generate random ID (current behaviour preserved)

### Change 3: Route — Pass Parameter and Handle Errors

**File:** `sgraph_ai_app_send/lambda__user/fast_api/routes/Routes__Transfers.py`

Update `create()` route:
1. Pass `transfer_id` from request to service
2. Handle `'error': 'transfer_id_exists'` -> 409 Conflict
3. Handle `'error': 'invalid_transfer_id_format'` -> 400 Bad Request

## 5. Security Analysis

### 5.1 409 Conflict Information Leakage

A 409 Conflict on `POST /transfers/create` confirms a transfer ID exists. For simple-token mode, an attacker who derived the ID from a guessed token also has the decryption key (both derived from the same token).

**Mitigations already in code:**
- Access token required: `check_access_token()` — attacker needs a valid token to probe
- IP hashing on create: suspicious patterns logged
- Token validation via admin service

**Additional mitigations needed:**
- Rate limiting on `POST /transfers/create` with client-provided IDs (5-10 per minute)
- IP hash monitoring for high 409 rates

**Assessment:** Acceptable risk. The 409 is necessary for the create-retry workflow.

### 5.2 ID Format Validation

- `Safe_Str__Id` type on route parameters strips `[^a-zA-Z0-9_-]` — path traversal eliminated
- Service-level regex `^[a-f0-9]{12}$` provides additional format validation
- No SQL in the stack — no injection risk
- Storage paths are deterministic — no traversal risk

**Assessment:** Low risk with validation in place.

### 5.3 Namespace Collision

PBKDF2-derived IDs (12 hex chars) share the same format as server-generated IDs. Probability of collision between client-set and server-generated IDs is negligible (48 bits of randomness on server side).

**Assessment:** Very low risk.

## 8. File Size Limit (10GB) — Backend Impact

**Frontend change only.** In `send-upload.js`:
```javascript
static MAX_FILE_SIZE_PRESIGNED = 10 * 1024 * 1024 * 1024;  // 10GB (was 1GB)
```

**Backend verification:**
- `POST /presigned/initiate`: No hardcoded size cap
- `POST /presigned/complete`: No size validation
- `GET /presigned/upload-url/{id}`: No size validation
- Multipart: 10GB / 10MB = 1024 parts < 10,000 limit
- S3: Supports 5TB per object

**No backend changes needed.**

## 11. Conclusion

The backend change for simple-token mode is **minimal, well-scoped, and technically feasible**:

- **One endpoint change:** `POST /transfers/create` accepts optional `transfer_id`
- **One validation rule:** Format check (12 hex chars) + existence check (409 Conflict)
- **No architectural changes:** All existing services work unchanged
- **No storage layer impact:** Works identically in memory and S3 modes
- **Backwards compatible:** Optional parameter, random fallback

**Recommended implementation order:** Schema -> Service -> Route -> Tests. Ship as a single IFD version (likely v0.2.6 per the architect implementation plan).
