## Executive Summary

This review covers four architectural areas arising from the 10 March briefs: (1) the 2-call minimum vault clone design, (2) the `read-base64` endpoint, (3) the ephemeral EC2 Deploy Service, and (4) vault-as-communication-channel. Each receives a formal assessment. The overall picture is architecturally sound, with conditions noted where design decisions need tightening before production.

---

## Area 1: Vault API Contract --- 2-Call Minimum AI Clone Design

**Assessment: Approved**

### What Was Proposed

An AI agent clones a vault in `1 + N` API calls: one call to fetch the encrypted tree (deterministic `tree_file_id` derived from `HMAC-SHA256(read_key, "tree")`), then N calls to fetch each file listed in the tree. A future bundle endpoint reduces this to 1 call.

### Analysis

The v0.13.19 architect response already validated this design. Having now reviewed the actual `Routes__Vault__Pointer.py` code and the session findings, I confirm the design is correct for the following reasons:

1. **The tree is the only manifest.** The current vault format is flat: `vault_id/{file_id}/payload`. The object-store model (refs, commits, content-addressable blobs) from the v0.11.1 architecture brief is PROPOSED -- does not exist yet. The 1 + N call model is the correct model for what exists today.

2. **Deterministic file IDs are the de facto access control.** Individual vault files are gated behind HMAC-derived file IDs (12 hex chars from `HMAC-SHA256(read_key, context_string)`). Without the passphrase, an attacker must brute-force 16^12 possibilities per file. This is a strong enough barrier that reads are safely public (no auth token required).

3. **Settings is optional for read-only clone.** Confirmed: settings contains vault metadata (`vault_name`, `created`, `version`) but no information required for enumeration or decryption. Skipping it saves one call.

4. **The call model is forward-compatible.** When the object-store model lands, the tree content will change (it will contain content-addressable blob IDs instead of flat file IDs), but the access pattern remains the same: derive tree_file_id, fetch tree, iterate files. The bundle endpoint abstracts this further.

### Open Item

The bundle endpoint (`GET /api/vault/bundle/{vault_id}/{bundle_file_id}?tree_id={tree_file_id}&scope={scope}`) from the v0.13.19 response is well-designed. The proof-of-knowledge validation via `tree_file_id` existence check is the correct approach -- it avoids the unauthenticated enumeration problem identified in the original proposal. This should proceed to implementation in the next sprint. No design changes needed.

## Area 2: Base64 Endpoint Design

**Assessment: Approved**

### What Exists

The `read-base64` endpoint is already implemented and deployed to dev. Reviewing the actual code in `Routes__Vault__Pointer.py`:

```python
def read_base64__vault_id__file_id(self, vault_id, file_id):
    payload = self.vault_service.read(vault_id=str(vault_id), file_id=str(file_id))
    if payload is None:
        raise HTTPException(status_code=404, detail='Vault file not found')
    if len(payload) > LAMBDA_BASE64_LIMIT:
        raise HTTPException(status_code=413, detail='...')
    return dict(vault_id=str(vault_id), file_id=str(file_id),
                data=base64.b64encode(payload).decode('ascii'), size=len(payload))
```

### Fit With Existing Patterns

This mirrors the transfers API exactly:

| Pattern | Transfers API | Vault API |
|---------|--------------|-----------|
| Binary read | `GET /api/transfers/download/{id}` | `GET /api/vault/read/{vault_id}/{file_id}` |
| Base64 read | `GET /api/transfers/download-base64/{id}` | `GET /api/vault/read-base64/{vault_id}/{file_id}` |
| Response | `{ data: "<base64>", size: N }` | `{ vault_id, file_id, data: "<base64>", size: N }` |

The consistency is good. The vault variant includes `vault_id` and `file_id` in the response, which is correct -- the transfers endpoint implicitly identifies by transfer ID, while vault files are addressed by a composite key.

### Size Limit

`LAMBDA_BASE64_LIMIT = 3,750,000` bytes (~3.75MB raw, which base64-encodes to ~5MB, hitting the Lambda response limit). This is correctly calculated and the 413 response with a fallback message to use the binary endpoint is the right pattern.

### One Observation

The `read-base64` endpoint is tagged `api/vault` and registered in the MCP manifest. Both were requirements from the dev brief. The endpoint has no auth requirement for reads, consistent with the existing `read` endpoint. This is correct -- the deterministic file ID serves as the access gate.

## Area 3: Deploy Service Architecture

**Assessment: Approved with Conditions**

### What Was Proposed

An ephemeral EC2 instance acting as the infrastructure control plane, with:
- A permanently-running Router Lambda (~pennies/month) as the stable entry point
- Boot-on-demand EC2 that shuts down after idle timeout
- Vault-backed state (not SSM Parameter Store)
- SSH-based appliance provisioning via `osbot-utils` SSH API
- Type__Twins as the default execution mode (AWS digital twin for development)
- Five distinct execution environments with clear capability boundaries

### What I Approve

1. **The Router Lambda + Ephemeral EC2 split is correct.** Lambda is the right choice for the stable entry point (cheap, always on, sub-second cold start). EC2 is the right choice for the control plane (SSH capability, long-running operations, full AWS SDK). The two complement each other.

2. **Vault-as-state-store is architecturally aligned.** Using SG/Send vault for instance state (`deploy-001.json`) instead of SSM is the right decision for this project. It dogfoods the product, provides zero-knowledge encryption at rest, works identically locally and in the cloud, and eliminates an AWS-specific dependency.

3. **The async job model is essential.** Deploy operations take 5-30 minutes. The `POST /stacks -> 202 { job_id } -> GET /jobs/{id}` pattern is the only viable approach. The step-by-step progress reporting (`step: "3/7"`) is valuable for observability.

4. **The dual shutdown mechanism (heartbeat + EventBridge watchdog) is correct.** Belt-and-suspenders. Maximum zombie lifetime of 25 minutes is acceptable. The heartbeat (primary) handles the normal case; EventBridge (secondary) handles hung processes and OOM kills.

5. **IAM scoping is well thought out.** Lambda gets only `ec2:Start/Stop/Describe` and `route53:ChangeResourceRecordSets`. No SSH keys, no create/terminate power. Deploy EC2 holds the high-privilege credentials and never exposes them via API. Created appliances each get their own minimal-privilege IAM role.

6. **Type__Twins as default mode is architecturally sound.** The insight that the digital twin is the *default* execution mode (not a test harness bolted on top) enables development without AWS credentials, proper dry-run workflows, and a clean onboarding story for marketplace distribution.

### Conditions

**Condition 1: Vault write key in Lambda environment variables needs scrutiny.**

The brief states: "The vault write key lives in the Lambda's environment variables and on the EC2 instance." The Router Lambda needs to write vault state transitions (`status: "starting"`, `status: "stopping"`). However, if the Lambda holds the vault write key, a compromised Lambda could overwrite any vault file -- including appliance config, AMI IDs, or other state.

**Recommendation:** The Router Lambda should hold a scoped write key (or a vault pointer to a specific state file) rather than the master vault write key. Alternatively, Lambda writes should be limited to a specific vault path prefix (`deploy/instances/`) using a server-side path validation on the write endpoint. This narrows the blast radius of a Lambda compromise.

**Condition 2: EC2 heartbeat writes to vault create a dependency on SG/Send availability.**

If `send.sgraph.ai` (or `dev.send.sgraph.ai`) is down, the EC2 heartbeat cannot update `last_seen` in the vault. The EventBridge watchdog reads vault state, so it would see a stale `last_seen` and stop the instance -- even though it is healthy and serving requests.

**Recommendation:** The EventBridge watchdog should use a secondary health signal (e.g., `ec2:DescribeInstances` to check instance state + a direct HTTP health check to the EC2 IP) before stopping an instance. Vault `last_seen` should be the *primary* signal, but the watchdog should not stop an instance that is both EC2-running and HTTP-healthy, even if vault writes are failing.

**Condition 3: The five-environment model is well-defined but needs a test-matrix document.**

The capability matrix (ENV1-5) is clear in the brief, but the testing story is implicit. Which tests run in which environment? What is the minimum test coverage required before a merge to main? The brief says "ENV3 runs on every PR" and "ENV4 runs on merge to main," but the exact test suites for each are not specified.

**Recommendation:** Dev or QA should produce a test-matrix document mapping test suites to environments before the first CI pipeline is implemented.

### Interaction with Existing Topology

The current SGraph Send topology (from the reality document) has two Lambda functions: User Lambda (public) and Admin Lambda (admin). The Deploy Service introduces a third Lambda (Router) and an ephemeral EC2. This is a new deployment pattern for the project.

The Router Lambda should follow the same infrastructure patterns as the existing Lambdas:
- Extend `Serverless__Fast_API` from `osbot-fast-api-serverless`
- Use Mangum for Lambda adaptation
- Deploy via GitHub Actions with the same test-tag-deploy flow
- Use the same access token validation model

The EC2 FastAPI service should also extend `Serverless__Fast_API` (or its non-serverless equivalent from `osbot-fast-api`) to maintain consistency.

## Area 4: Vault as Communication Channel

**Assessment: Approved with Conditions**

### What Was Demonstrated

The v0.13.31 round-trip debrief documents a complete bidirectional communication loop:

1. Claude (claude.ai) cloned vault `i0brp94w`, created `claude-web.txt`, pushed
2. Human pulled, received the file
3. Human created `experiments/it-works.md`, pushed
4. Claude pulled, read the message
5. Claude wrote `SKILL.md` (instructions for future sessions), pushed
6. Human pulled, received `SKILL.md`
7. A completely new Claude session cloned the vault, read `SKILL.md`, and operated autonomously

The vault became an asynchronous, persistent, bidirectional communication channel between human and AI, and between AI sessions.

### Architectural Implications

This is a significant emergent capability. The vault was designed as a zero-knowledge encrypted file store. Its use as a communication channel introduces new architectural considerations:

**1. Ordering and Consistency**

The current vault model has no concurrency control. Two writers (human and Claude) can push simultaneously. The last writer wins at the file level (overwrite semantics). There is no merge, no conflict detection, and no locking.

For the communication use case, this is acceptable today because:
- The demonstrated pattern is sequential (one party writes, the other reads)
- The tree is the only source of truth for file enumeration
- Individual file writes are atomic at the storage layer

However, if multiple AI agents write to the same vault concurrently (the "agent-to-agent communication" use case mentioned in the brief), file-level conflicts become likely.

**Recommendation:** For the current use case (human-AI, sequential), no changes needed. For future multi-agent concurrent writes, the vault needs either (a) a compare-and-swap primitive on the tree, or (b) a convention-based partitioning scheme where each agent writes to its own namespace (`agents/{agent_id}/`). The namespace approach is simpler and does not require server-side changes.

**2. Self-Bootstrapping via SKILL.md**

The pattern where Claude wrote `SKILL.md` into the vault, and a future Claude session used it to bootstrap itself, is architecturally significant. It means the vault is not just data storage -- it is an **instruction carrier**. The vault key becomes not just a data access credential but a capability grant: "anyone who holds this key can read the instructions and execute the workflow."

This has security implications. A leaked vault key now provides not just access to data, but a set of executable instructions that an AI agent will follow. The instructions themselves are encrypted (zero-knowledge), but the capability surface is wider than pure data storage.

**Recommendation:** This is an observation, not a blocker. The security model is unchanged -- the vault key is the trust boundary, and everything inside is opaque to the server. But documentation should acknowledge that vaults may contain AI-executable instructions, and that sharing a vault key with an AI agent grants both data access and instruction-following capability.

**3. The vault-as-channel pattern does not need new API surface**

The existing `read`, `read-base64`, `write`, and `delete` endpoints are sufficient for the communication use case. No new endpoints are required. The `sg-send-cli clone/push/pull` commands map directly to the API. This is a good sign -- the API was designed for file storage and the communication pattern is an emergent use, not a forced one.

### Conditions

**Condition 1: Define a convention for AI-authored content.**

When Claude writes a file to a vault, there is no metadata distinguishing it from a human-authored file. For audit and trust purposes, a convention should be established. Options:

- File-level: AI-authored files include a header (e.g., `# Created by: Claude Sonnet 4.6, session: xyz`)
- Tree-level: AI-authored files are placed under a conventional path (`ai/` or `agents/`)
- Metadata-level: A `.vault-meta.json` file lists authorship per file

**Recommendation:** Start with the file-level convention (header in content). It requires zero server changes, works with any vault format, and is human-readable. Formalise later if needed.

**Condition 2: Consider rate limiting for vault writes via MCP.**

The MCP connector gives AI agents write access to vaults (when they hold a token). If an AI agent enters a loop, it could produce a large number of vault writes. The current API has no rate limiting on writes beyond the access token check.

**Recommendation:** Add a simple rate limit on the write endpoint (e.g., 60 writes/minute per vault_id). This protects against runaway loops without affecting normal workflows.
