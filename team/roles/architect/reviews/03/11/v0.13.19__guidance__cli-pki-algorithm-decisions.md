## Context

The CLI implementation plan (v0.13.16) identifies three open questions for PKI (Phase 3). This document provides definitive architectural guidance based on what already exists in the admin lambda and UI.

**Existing PKI implementation (code-verified):**
- Admin UI (`pki-common.js`, `pki-keys.js`, `pki-encrypt.js`) — RSA-4096 + ECDSA P-256 + AES-256-GCM
- Backend key registry (`Service__Keys`, `Routes__Keys`) — PEM key storage, SHA-256 fingerprints, transparency log
- Vault crypto (`Vault__Crypto`) — PBKDF2-SHA256 + AES-256-GCM (symmetric, for vault data)

## Decision 1: Signing Algorithm — ECDSA P-256

**Decision: Use ECDSA P-256. Do NOT use Ed25519.**

### Rationale

The admin UI already generates ECDSA P-256 signing keys alongside RSA-OAEP encryption keys (see `pki-keys.js:99-101`). The entire sign/verify pipeline is built on ECDSA P-256:

```javascript
// pki-keys.js — existing key generation
const signingPair = await crypto.subtle.generateKey(
    { name: 'ECDSA', namedCurve: 'P-256' }, false, ['sign', 'verify']
);

// pki-encrypt.js — existing signing
const signature = await crypto.subtle.sign(
    { name: 'ECDSA', hash: 'SHA-256' }, senderKey.signingKey, dataToSign
);
```

The CLI must produce signatures that the browser can verify and vice versa. ECDSA P-256 with SHA-256 is the only signing algorithm that achieves interop with the existing codebase.

Ed25519 was considered but ruled out because:
1. **The admin UI already uses ECDSA P-256** — changing would break existing key pairs
2. **Web Crypto Ed25519 support is inconsistent** — Chrome 113+, Firefox 130+, Safari 17+ (not universal as of 2026)
3. **No interop path** — Ed25519 signatures cannot be verified by the existing `pki-encrypt.js` verify flow
4. **Key registry accepts ECDSA PEM** — the `signing_key_pem` field on `/keys/publish` expects SPKI-encoded ECDSA keys

### Implementation spec for CLI

| Parameter | Value |
|-----------|-------|
| Algorithm | ECDSA |
| Curve | P-256 (secp256r1) |
| Hash | SHA-256 |
| Python library | `cryptography` (already a dependency via `Vault__Crypto`) |
| Key generation | `ec.generate_private_key(ec.SECP256R1())` |
| Signature format | DER-encoded (default from `cryptography`), convert to raw r‖s (64 bytes) for Web Crypto interop |
| Export format | PEM (SPKI for public, PKCS8 for private) |

### Critical interop note: Signature encoding

Web Crypto ECDSA produces **raw r‖s** signatures (64 bytes for P-256). Python's `cryptography` library produces **DER-encoded** signatures by default. The CLI MUST convert between formats:

```python
# Python sign → browser verify: convert DER to raw
from cryptography.hazmat.primitives.asymmetric.utils import decode_dss_signature
r, s = decode_dss_signature(der_signature)
raw_signature = r.to_bytes(32, 'big') + s.to_bytes(32, 'big')

# Browser sign → Python verify: convert raw to DER
from cryptography.hazmat.primitives.asymmetric.utils import encode_dss_signature
r = int.from_bytes(raw_signature[:32], 'big')
s = int.from_bytes(raw_signature[32:], 'big')
der_signature = encode_dss_signature(r, s)
```

**Test vectors are mandatory.** Generate a signing key pair, sign a known message in Python, verify in browser (and vice versa). This is the highest-risk interop surface.

## Decision 2: Encryption Algorithm — RSA-OAEP 4096 + AES-256-GCM (hybrid)

**Decision: Use RSA-OAEP 4096 with AES-256-GCM hybrid encryption, matching the existing admin UI.**

### Rationale

The admin UI (`pki-common.js:135-150`) implements hybrid encryption:
1. Generate random AES-256-GCM key
2. Encrypt plaintext with AES-256-GCM
3. Wrap the AES key with RSA-OAEP (recipient's public key)
4. Bundle: `{wrappedKey, iv, encrypted}`

The CLI must produce ciphertext the browser can decrypt, and decrypt ciphertext the browser produced.

### Implementation spec for CLI

| Parameter | Value |
|-----------|-------|
| Asymmetric | RSA-OAEP, 4096-bit, SHA-256 |
| Symmetric | AES-256-GCM, 12-byte IV |
| Key wrapping | RSA-OAEP wraps raw AES key (32 bytes) |
| Python library | `cryptography` |
| RSA key gen | `rsa.generate_private_key(public_exponent=65537, key_size=4096)` |
| RSA padding | `padding.OAEP(mgf=padding.MGF1(hashes.SHA256()), algorithm=hashes.SHA256(), label=None)` |
| AES mode | `AESGCM(key)` with 12-byte random nonce |

### Payload format (must match browser)

The browser uses a v2 JSON payload encoded as base64:

```json
{
    "v": 2,
    "w": "<base64 wrapped AES key>",
    "i": "<base64 IV, 12 bytes>",
    "c": "<base64 AES-GCM ciphertext>",
    "s": "<base64 ECDSA signature over ciphertext>",
    "f": "<signing key fingerprint>"
}
```

The CLI must produce and consume this exact format. Fields `s` and `f` are optional (present when signed).

### ECDH P-256 — NOT for CLI Phase 3

The admin UI also supports ECDH P-256 key generation (see `pki-keys.js:87-91`), but:
- ECDH keys use `deriveKey`/`deriveBits` — a different encryption flow (ECIES)
- The encrypt/decrypt component (`pki-encrypt.js`) only supports RSA-OAEP keys for hybrid encrypt
- ECDH is a future track, not needed for CLI v1

**Do not implement ECDH in Phase 3.** RSA-OAEP hybrid is the production encryption path.

## Decision 3: Key Format — PEM

**Decision: Use PEM format for storage and exchange. Support JWK import as a secondary path.**

### Rationale

1. **The key registry (`/keys/publish`) accepts PEM** — `public_key_pem` and `signing_key_pem` fields
2. **The admin UI exports PEM** — `exportPublicKeyPEM()` in `pki-common.js:96-106`
3. **The fingerprint is computed from SPKI DER** — `computeFingerprint()` in `pki-common.js:124-129`
4. **The public key bundle** uses PEM: `{ v: 1, encrypt: "<PEM>", sign: "<PEM>" }`

### Key storage layout for CLI

```
~/.sg-send/
  keys/
    {fingerprint}/
      private_key.pem       # PKCS8-encoded, encrypted with passphrase (PBKDF2 + AES-256-CBC)
      public_key.pem        # SPKI-encoded
      signing_private.pem   # PKCS8-encoded, encrypted
      signing_public.pem    # SPKI-encoded
      metadata.json         # label, algorithm, created, fingerprint, signing_fingerprint
  keyring/
    {fingerprint}.json      # Imported contacts: { label, public_key_pem, signing_key_pem, fingerprint }
```

### Fingerprint computation (must match browser and server)

```python
# SHA-256 of SPKI DER, first 16 hex chars, prefixed with "sha256:"
import hashlib
from cryptography.hazmat.primitives.serialization import Encoding, PublicFormat

der = public_key.public_bytes(Encoding.DER, PublicFormat.SubjectPublicKeyInfo)
digest = hashlib.sha256(der).hexdigest()
fingerprint = f"sha256:{digest[:16]}"
```

This matches:
- Browser: `pki-common.js:124-129` (`computeFingerprint`)
- Server: `Service__Keys.py:22-27` (`_compute_fingerprint_from_pem`)

## Open Question Resolutions

### Q1: Transfer API spec (Phase 1)
Not relevant to PKI. Remains open — needs server-side spec.

### Q2: PKI algorithm choice
**Resolved.** ECDSA P-256 for signing, RSA-OAEP 4096 for encryption. See Decisions 1 and 2 above.

### Q3: Secrets master passphrase UX (Phase 4)
**Recommendation:** Prompt on first use, derive encryption key via PBKDF2 (matching `Vault__Crypto` pattern). Support `SG_SEND_PASSPHRASE` env var for automation. Do NOT derive from vault key — secrets and vaults have independent lifecycles.

### Q4: Backward compatibility period
**Recommendation:** Keep flat command aliases indefinitely (they cost nothing). Deprecation warnings are unnecessary for internal tooling.

## Summary

The CLI PKI implementation must be **interop-first** — every algorithm, format, and encoding choice is dictated by what the admin UI already ships. The existing codebase uses:

- **RSA-OAEP 4096** for encryption (hybrid with AES-256-GCM)
- **ECDSA P-256 with SHA-256** for signing
- **PEM (SPKI/PKCS8)** for key exchange
- **SHA-256 fingerprints** (first 16 hex chars) for key identity
- **v2 JSON payload** (base64-encoded) for encrypted messages

The CLI must match all of these exactly. No new algorithm choices needed — follow the existing implementation.
