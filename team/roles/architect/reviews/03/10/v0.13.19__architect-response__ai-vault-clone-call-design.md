## Q1: Is the tree truly sufficient as the only manifest?

**Yes — for the current format (which is the only format that exists).**

The object-store model (v0.11.1 arch brief) is **PROPOSED — does not exist yet**. There are no refs, commits, or content-addressable blobs. The current vault data model is:

```
vault_id/
  tree_file_id/payload      ← encrypted tree.json (flat map of {path → file_id})
  settings_file_id/payload  ← encrypted settings.json
  {file_id_1}/payload       ← encrypted file blob
  {file_id_2}/payload       ← encrypted file blob
  ...
```

Both `tree_file_id` and `settings_file_id` are **deterministic** — derived from `HMAC-SHA256(read_key, context_string)`, truncated to 12 hex chars. The agent computes them locally from the vault key before any network I/O.

**Current call model (confirmed):**

```
Call 0: (local)  Vault__Crypto.derive_keys(passphrase, vault_id) → tree_file_id, settings_file_id
Call 1: read-base64(vault_id, tree_file_id)  → decrypt → {path: file_id} map
Calls 2..N+1: read-base64(vault_id, file_id) for each of N files
```

**Total: 1 + N calls** (tree + N files). Settings is optional for read-only (see Q4).

**When object-store lands (future):** The ref → commit → tree → blob chain would add 2 extra calls (3 + N). But that model doesn't exist, and when it does, the bundle endpoint (Q2) will make it irrelevant — one call regardless.

## Q2: Can we bundle all blobs into a single response?

**Yes — but the original design had a security gap. Revised design below.**

### Security Assessment — REVISED

The original proposal (`GET /api/vault/bundle-base64/{vault_id}` with no auth) had a meaningful security regression compared to individual reads:

**Current read model (safe):**
- Each read requires `vault_id` (8 hex) + `file_id` (12 hex)
- `file_id` values are HMAC-derived from `read_key` — **effectively secret without the passphrase**
- An attacker who knows only the `vault_id` cannot enumerate files — brute-forcing each `file_id` requires 16^12 (~281 trillion) attempts
- The file_ids act as a **de facto second authentication factor**

**Original bundle proposal (insecure):**
- Only requires `vault_id` (8 hex chars = ~4 billion values)
- Server enumerates all files and returns them — bypasses the file_id knowledge requirement
- A leaked `vault_id` (from URLs, logs, browser history, network traffic) = complete encrypted vault download in one call
- While data is still AES-256-GCM encrypted, the **enumeration barrier is removed**

**Architect assessment — CORRECTED:** The original "no new info leakage" claim was wrong about the access model. The server already *stores* all blobs, but the current API *gates* access behind file_id knowledge. An unauthenticated bundle endpoint removes that gate.

### Revised Design: Deterministic Bundle Path (Proof-of-Knowledge)

The solution follows the same pattern that already works for `tree_file_id` and `settings_file_id`: **make the bundle path itself deterministic, derived from the passphrase.**

```
bundle_file_id = HMAC-SHA256(read_key, "bundle")[:12]   # same derivation as tree/settings
```

The caller must prove they hold the passphrase by presenting the correct `bundle_file_id`. The server verifies this file_id exists as a valid derived pointer before returning any data.

### On-Demand Bundle with Flexible Scope

**Critical design decision:** The bundle is NOT pre-built on every write (doesn't scale). It is generated **on-demand** when requested. The deterministic path serves as both the proof-of-knowledge and the endpoint identifier.

```
GET /api/vault/bundle/{vault_id}/{bundle_file_id}?scope=all
GET /api/vault/bundle/{vault_id}/{bundle_file_id}?scope=tree
GET /api/vault/bundle/{vault_id}/{bundle_file_id}?file_ids=a1b2c3,d4e5f6,789012
GET /api/vault/bundle/{vault_id}/{bundle_file_id}?scope=all&format=zip
```

**Supported scopes:**

| Scope | Behaviour | Use case |
|-------|-----------|----------|
| `all` | Every file in the vault | CLI clone, full backup |
| `tree` | Tree + settings only | Quick vault inspection |
| `file_ids=a,b,c` | Specific files only | Selective sync, partial clone |

**Response (JSON, default):**

```json
{
  "vault_id": "aaaabbp1",
  "files": {
    "a1b2c3d4e5f6": "<base64 encrypted blob>",
    "f7e8d9c0b1a2": "<base64 encrypted blob>"
  },
  "file_count": 2,
  "total_size": 42816
}
```

**Response (zip, when `format=zip`):** Binary zip containing `{file_id}.bin` entries. More efficient for large vaults (no base64 overhead).

**Design notes:**

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Auth | **Proof-of-knowledge via `bundle_file_id`** | Caller must derive from passphrase — same security as individual reads |
| Validation | Server verifies `bundle_file_id == HMAC(read_key, "bundle")[:12]` by checking it matches a known derived pointer pattern | Rejects random guesses |
| Size limit | 5MB total response (JSON) / 6MB (zip) | Lambda response limit. Return 413 with file list + sizes if over limit |
| File map key | `file_id` (not path) | Server doesn't know paths — those are in the encrypted tree |
| Metadata in response | `manifest.json` excluded | Contains `write_key_hash` — must not be in a public read response |
| Generation | **On-demand** — zip/JSON built at request time | Pre-building on every write doesn't scale |
| Scope | Flexible — all, tree-only, or specific file_ids | Supports CLI clone, partial sync, inspection |

**Size limit strategy:** For vaults exceeding the Lambda response limit:

1. **413 with manifest:** Return a `files` array of `{file_id, size}` (no data), letting the agent fall back to individual `read-base64` calls or scoped bundle requests
2. **Scoped requests:** Client can request subsets via `file_ids` parameter to stay under the limit
3. **Zip format:** ~33% smaller than base64 JSON, extends the effective ceiling

### Validation Strategy

The server needs to confirm that `bundle_file_id` is a legitimate derived pointer, not a random guess. Two options:

1. **Convention-based:** The server derives `expected_bundle_file_id = HMAC(read_key, "bundle")[:12]` — but the server doesn't have `read_key` (it's client-derived). **Not viable.**

2. **Existence-based (recommended):** The server checks that the `bundle_file_id` matches a file_id pattern that could only be known by someone who holds the passphrase. Since the server stores a `vault_pointer.json` containing `write_key_hash`, the simplest validation is: **require the client to also send `tree_file_id` as a query parameter** (which the server can verify exists in storage):

```
GET /api/vault/bundle/{vault_id}/{bundle_file_id}?tree_id={tree_file_id}&scope=all
```

Server checks: does `transfers/vault/{vault_id}/{tree_file_id}/payload` exist? If yes, the caller demonstrably holds the passphrase. The `bundle_file_id` in the URL still serves as the deterministic path for caching/logging.

### Implementation path

```python
def bundle(self, vault_id, bundle_file_id, tree_file_id, scope='all', file_ids=None):
    # Step 1: Verify proof-of-knowledge
    tree_path = f'transfers/vault/{vault_id}/{tree_file_id}/payload'
    if not self.storage_fs.file__exists(tree_path):
        return dict(error='unauthorized', status=403)

    # Step 2: Enumerate files based on scope
    prefix = f'transfers/vault/{vault_id}/'
    if scope == 'all':
        all_paths = self.storage_fs.folder__files__all(prefix)
        target_ids = [p.split('/')[-2] for p in all_paths if p.endswith('/payload')]
    elif scope == 'tree':
        target_ids = [tree_file_id]           # caller can add settings_file_id to file_ids if needed
    elif file_ids:
        target_ids = file_ids

    # Step 3: Build response (exclude manifest/meta)
    files = {}
    total_size = 0
    for fid in target_ids:
        payload_path = f'{prefix}{fid}/payload'
        payload = self.storage_fs.file__bytes(payload_path)
        if payload:
            files[fid] = base64.b64encode(payload).decode('ascii')
            total_size += len(payload)

    return dict(vault_id=vault_id, files=files, file_count=len(files), total_size=total_size)
```

### CLI Clone Flow (Single Call)

```
1. (local)  derive read_key, tree_file_id, bundle_file_id from passphrase + vault_id
2. (remote) GET /api/vault/bundle/{vault_id}/{bundle_file_id}?tree_id={tree_file_id}&scope=all
3. (local)  decrypt tree → get {path: file_id} map
4. (local)  decrypt each blob using file_id from tree map
5. (local)  write files to disk
```

**Total: 1 HTTP call** (down from 1 + N).

## Q4: Is settings required for read-only clone?

**No.** Settings contains vault metadata (`vault_name`, `created`, `version`). It is not required to:

- Enumerate files (that's the tree)
- Decrypt files (that uses `read_key_bytes` derived from the vault key)
- Reconstruct the file hierarchy (that's the tree paths)

**Settings is required for:**
- Displaying the vault name in a UI
- Write operations (some clients store the tree_transfer_id in settings for legacy reasons)

**Recommendation:** For an AI agent doing a read-only clone, skip settings. **Saves 1 call in Phase 1 (1 + N → N calls after tree).** In Phase 2 (bundle), it's free — settings is just another blob in the bundle.

## Recommendations

### Phase 1 (done — ship today)

`read-base64` is already implemented and tested. The AI clone path is:

```
1 + N calls:  tree (1 call) + N files (N calls)
```

This works now. No further server changes needed for Phase 1.

### Phase 2 (next sprint)

Build `GET /api/vault/bundle/{vault_id}/{bundle_file_id}?tree_id={tree_file_id}&scope={scope}`:

1. **Key derivation** — add `bundle` context string to `Vault__Crypto` → `bundle_file_id = HMAC(read_key, "bundle")[:12]`
2. **Service method** on `Service__Vault__Pointer` — proof-of-knowledge check + scoped enumerate + bundle
3. **Route** on `Routes__Vault__Pointer` — scope parsing, size check, JSON/zip response
4. **MCP tool** — `api_vault_bundle` (tagged `vault`)
5. **Tests** — bundle round-trip, scope=all, scope=tree, scope=file_ids, empty vault, over-size vault (413), **invalid tree_id rejected (403)**

**Estimated complexity:** Small-medium — the Storage_FS primitives already exist. The proof-of-knowledge check is a single `file__exists` call. Scope parsing is straightforward. ~80 lines of new code + tests.

### Phase 3 (when object-store lands)

The bundle endpoint abstracts away the internal storage model. Whether the vault uses flat files, a commit chain, or a content-addressable store, the bundle response is the same: `{file_id: base64_blob}`. The agent doesn't need to know the internal model.

This means **the bundle API is forward-compatible with the object-store migration** — no breaking changes when the storage model evolves.
