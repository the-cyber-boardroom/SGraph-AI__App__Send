## Q1: Is the tree truly sufficient as the only manifest?

**Yes — for the current format (which is the only format that exists).**

The object-store model (v0.11.1 arch brief) is **PROPOSED — does not exist yet**. There are no refs, commits, or content-addressable blobs. The current vault data model is:

```
vault_id/
  tree_file_id/payload      ← encrypted tree.json (flat map of {path → file_id})
  settings_file_id/payload  ← encrypted settings.json
  {file_id_1}/payload       ← encrypted file blob
  {file_id_2}/payload       ← encrypted file blob
  ...
```

Both `tree_file_id` and `settings_file_id` are **deterministic** — derived from `HMAC-SHA256(read_key, context_string)`, truncated to 12 hex chars. The agent computes them locally from the vault key before any network I/O.

**Current call model (confirmed):**

```
Call 0: (local)  Vault__Crypto.derive_keys(passphrase, vault_id) → tree_file_id, settings_file_id
Call 1: read-base64(vault_id, tree_file_id)  → decrypt → {path: file_id} map
Calls 2..N+1: read-base64(vault_id, file_id) for each of N files
```

**Total: 1 + N calls** (tree + N files). Settings is optional for read-only (see Q4).

**When object-store lands (future):** The ref → commit → tree → blob chain would add 2 extra calls (3 + N). But that model doesn't exist, and when it does, the bundle endpoint (Q2) will make it irrelevant — one call regardless.

## Q2: Can we bundle all blobs into a single response?

**Yes. The server can do this today with existing primitives.**

### Security Assessment

The server stores files under `transfers/vault/{vault_id}/{file_id}/payload`. Using `Storage_FS.folder__files__all('transfers/vault/{vault_id}/')`, the server can enumerate every file_id in a vault without decrypting anything.

**What the server already knows:**
- All file_ids under a vault_id (visible in storage paths)
- Blob sizes (from stored payload length)
- Total file count
- Vault manifest (write_key_hash, created_at)

**What bundling reveals additionally:** Nothing. The bundle response contains exactly the same encrypted blobs the server already stores and could already serve individually. Bundling them into one JSON response changes the transport format, not the information exposure.

**Architect assessment: Bundling is cryptographically clean.** No new information leaks. The zero-knowledge property holds — the server still never sees plaintext.

### Proposed API Contract

```
GET /api/vault/bundle-base64/{vault_id}
```

**Response:**

```json
{
  "vault_id": "i0brp94w",
  "files": {
    "a1b2c3d4e5f6": "<base64 encrypted blob>",
    "f7e8d9c0b1a2": "<base64 encrypted blob>",
    ...
  },
  "file_count": 7,
  "total_size": 42816
}
```

**Design notes:**

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Auth | No auth (reads are public) | Consistent with `read` and `read-base64` — zero-knowledge model |
| Size limit | 5MB total base64 response | Lambda response limit. Return 413 with file list + sizes if over limit |
| File map key | `file_id` (not path) | Server doesn't know paths — those are in the encrypted tree |
| Metadata in response | `manifest.json` excluded | Contains `write_key_hash` — must not be in a public read response |
| Settings/tree files | Included as regular files | They're stored identically; the agent knows their file_ids and can identify them after decryption |

**Size limit strategy:** For vaults exceeding the Lambda 5MB response limit, two options:

1. **413 with manifest:** Return a `files` array of `{file_id, size}` (no data), letting the agent fall back to individual `read-base64` calls for large files
2. **Streaming/pagination:** Out of scope for Phase 2; the Lambda response limit is the natural ceiling

### Implementation path

The service method would be straightforward:

```python
def bundle(self, vault_id):
    prefix = f'transfers/vault/{vault_id}/'
    all_paths = self.storage_fs.folder__files__all(prefix)
    files = {}
    total_size = 0
    for path in all_paths:
        if path.endswith('/payload'):
            file_id = path.split('/')[-2]
            payload = self.storage_fs.file__bytes(path)
            files[file_id] = base64.b64encode(payload).decode('ascii')
            total_size += len(payload)
    return dict(vault_id=vault_id, files=files, file_count=len(files), total_size=total_size)
```

The manifest and meta files are filtered out (only `/payload` paths are included). This is safe and minimal.

## Q4: Is settings required for read-only clone?

**No.** Settings contains vault metadata (`vault_name`, `created`, `version`). It is not required to:

- Enumerate files (that's the tree)
- Decrypt files (that uses `read_key_bytes` derived from the vault key)
- Reconstruct the file hierarchy (that's the tree paths)

**Settings is required for:**
- Displaying the vault name in a UI
- Write operations (some clients store the tree_transfer_id in settings for legacy reasons)

**Recommendation:** For an AI agent doing a read-only clone, skip settings. **Saves 1 call in Phase 1 (1 + N → N calls after tree).** In Phase 2 (bundle), it's free — settings is just another blob in the bundle.

## Recommendations

### Phase 1 (done — ship today)

`read-base64` is already implemented and tested. The AI clone path is:

```
1 + N calls:  tree (1 call) + N files (N calls)
```

This works now. No further server changes needed for Phase 1.

### Phase 2 (next sprint)

Build `GET /api/vault/bundle-base64/{vault_id}`:

1. **Service method** on `Service__Vault__Pointer` — enumerate + bundle
2. **Route** on `Routes__Vault__Pointer` — size check + JSON response
3. **MCP tool** — `api_vault_bundle_base64` (tagged `vault`)
4. **Tests** — bundle round-trip, empty vault, over-size vault (413)

**Estimated complexity:** Small — the Storage_FS primitives already exist (`folder__files__all`, `file__bytes`). The route follows the same pattern as `read-base64`. ~50 lines of new code + tests.

### Phase 3 (when object-store lands)

The bundle endpoint abstracts away the internal storage model. Whether the vault uses flat files, a commit chain, or a content-addressable store, the bundle response is the same: `{file_id: base64_blob}`. The agent doesn't need to know the internal model.

This means **the bundle API is forward-compatible with the object-store migration** — no breaking changes when the storage model evolves.
