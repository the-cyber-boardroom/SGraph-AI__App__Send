## 2. Implementation Strategy — IFD Layering

The brief is large. Implementing everything at once violates IFD principles. I recommend a **layered approach using v0.2.3 as a surgical overlay on v0.2.0**, consistent with how v0.2.1 and v0.2.2 already work.

### Layer Plan

| IFD Layer | Scope | Dependencies |
|-----------|-------|-------------|
| **v0.2.3-a** | Three-step state machine skeleton + step indicator + 10GB limit | None — pure refactor of `send-upload.js` |
| **v0.2.3-b** | Step 2: content detection + delivery cards | v0.2.3-a |
| **v0.2.3-c** | Step 3: PBKDF2 simple token + 3 sharing modes | v0.2.3-a, needs `sg-pbkdf2.js` and `sg-wordlist.js` |
| **v0.2.3-d** | Upload progress trust carousel | v0.2.3-a |
| **v0.2.3-e** | Demo links section on upload page | Independent |
| **v0.2.4** | New route pages (`/d/{token}`, `/view/`, `/browse/`, `/gallery/`) | v0.2.3-c (for token entry page) |

**Recommendation:** Build v0.2.3-a first. It's the foundation everything else plugs into. Each subsequent layer is independently testable.

## 3. State Machine Redesign

### Current States (v0.2.0)

```
idle → reading → encrypting → creating → uploading → completing → complete
                                                                    ↓
                                                                  error
(plus: folder-options, zipping — inserted before reading)
```

### Proposed States (v0.2.3)

```
                    STEP 1                    STEP 2              STEP 3
                    ──────                    ──────              ──────
idle ──▶ folder-options ──▶ zipping ──▶ reading ──▶ encrypting ──▶ encrypted
                                                                     │
                                                               choosing-delivery
                                                                     │
                                                               choosing-share
                                                                     │
                                                          creating ──▶ uploading ──▶ completing ──▶ complete
                                                                                                     │
                                                                                                   error
```

**Key change:** The `encrypted` state is new — it's the pivot point between Step 1 and Step 2. The encrypted blob is held in memory. Nothing has been sent to the server. The user can change their mind about delivery mode without re-encrypting.

**However:** For simple-token mode (Mode 1), the key is derived from the token words via PBKDF2 *before* encryption. This means the token must be chosen/generated at Step 3 *before* encryption happens. The flow becomes:

```
OPTION A (Mode 2/3 — random key):
  drop file → encrypt with random key → choose delivery → show key → upload → complete

OPTION B (Mode 1 — simple token):
  drop file → generate/choose token → derive key via PBKDF2 → encrypt → choose delivery → upload → complete
```

**Architecture decision needed:** Should Step 3 (share mode selection) come *before* encryption, or should we encrypt with a random key and re-encrypt if the user picks Mode 1?

**My recommendation:** Encrypt with a random key first (fast, no user decision needed). If the user picks Mode 1 in Step 3, re-encrypt with the PBKDF2-derived key. Re-encryption is fast (sub-second for files under 100MB, a few seconds for larger files). This keeps Step 1 clean — drop and encrypt immediately with no decisions. The brief's mockups show encryption happening in Step 1 before any choices, which supports this approach.

**Alternative:** Pre-generate a token in the background during Step 1 and always use PBKDF2. But this couples Step 1 to Step 3 logic and means the user can't change their token without re-encrypting.

## 4. File Size Limit Change

Straightforward. In `send-upload.js` line 171:

```javascript
// Current
static MAX_FILE_SIZE_PRESIGNED = 1024 * 1024 * 1024;        // 1GB

// Target
static MAX_FILE_SIZE_PRESIGNED = 10 * 1024 * 1024 * 1024;   // 10GB
```

Also update the `MAX_FILE_SIZE_DIRECT` threshold check in `_checkCapabilities()` — anything over 5MB goes presigned, which already works. The S3 presigned URL pipeline has no enforced cap on the backend (`/presigned/initiate` doesn't check file size).

**Risk:** 10GB files in-browser will consume significant memory during encryption. The Web Crypto API processes the entire ArrayBuffer in one call. For 10GB files, this requires ~20GB of browser memory (plaintext + ciphertext). **Streaming encryption is not yet implemented** — this is a known limitation flagged for Villager hardening.

**Mitigation for now:** Show a warning for files over 2GB: "Large files may take several minutes to encrypt. Keep this tab open."

## 5. Content Detection (Step 2)

The brief's content detection matrix (Section 6) is sound. Implementation approach:

```javascript
function detectDeliveryOptions(file, folderScan) {
    const ext = file.name.split('.').pop().toLowerCase();
    const options = ['download'];  // always available

    if (folderScan) {
        options.push('browse');
        const allImages = folderScan.entries.every(e =>
            ['png','jpg','jpeg','gif','webp','svg'].includes(
                e.name.split('.').pop().toLowerCase()
            )
        );
        if (allImages) options.push('gallery');
    } else if (['pdf','md','txt','html','htm','json','csv','xml'].includes(ext)) {
        options.push('view');
    } else if (['png','jpg','jpeg','gif','webp','svg'].includes(ext)) {
        options.push('view');
    } else if (ext === 'zip') {
        options.push('browse');
        // Could read zip directory via JSZip to check if all-images
    }

    return options;
}
```

**Recommended defaults** (marked with ★ in the brief's matrix):
- Single previewable file → `view`
- Folder with mixed files → `browse`
- Folder with all images → `gallery`
- Non-previewable file → `download` (only option)
- Zip → `browse`

The detection runs after encryption, using the original filename (preserved in the SGMETA envelope and in the component's `selectedFile` reference).

## 6. PBKDF2 Integration (Step 3 — Simple Token)

The Tools repo has the full pipeline. For the Send repo, we need to copy these modules:

| Module | Source (Tools repo) | Target (Send repo) |
|--------|--------------------|--------------------|
| `sg-pbkdf2.js` | `core/pbkdf2/v1/v1.0/v1.0.0/sg-pbkdf2.js` | `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.3/_common/js/` |
| `sg-wordlist.js` | `core/wordlist/v1/v1.0/v1.0.0/sg-wordlist.js` | Same |
| `sg-key-generator.js` | `components/key-generator/v1/v1.0/v1.0.0/` | Same (simplified version) |

**PBKDF2 parameters** (must match Tools repo exactly):
- Algorithm: PBKDF2
- Hash: SHA-256
- Iterations: 600,000
- Salt: `"sgraph-send-v1"` (static, deterministic)
- Key length: 256 bits (AES-256)

**Token format:** `{word1}-{word2}-{number}` e.g. `coal-render-5512`

The conversion is deterministic: same words → same AES-256 key. No server involvement. The recipient enters the token on the `/d/{token}` page, the same PBKDF2 derivation runs, and the derived key decrypts the file.

## 7. New Route Architecture

The brief confirms the upload page URL does **not** change during steps (Decision #1). The three-step wizard is a single-page state machine at `/{lang}/`.

New pages needed (v0.2.4, after the upload flow works):

| Route | Page | Components |
|-------|------|-----------|
| `/{lang}/d/{token}` | Token entry → PBKDF2 → redirect | `sg-key-input`, `sg-pbkdf2` |
| `/{lang}/download/{id}` | Direct download | Existing `send-download.js` (download-only mode) |
| `/{lang}/view/{id}` | PDF/markdown viewer | Existing `send-download.js` (view-only mode) |
| `/{lang}/browse/{id}` | Zip explorer | Existing `send-download.js` (browse-only mode) |
| `/{lang}/gallery/{id}` | Image gallery | New component (PROPOSED — does not exist yet) |

**Key insight:** The `/download/`, `/view/`, `/browse/` pages can reuse the existing `send-download.js` with a mode parameter. The current download page already handles all these preview types — it just needs routing to force a specific mode rather than auto-detecting.

The `/d/{token}` page is genuinely new — it takes a human-friendly token, derives the AES key via PBKDF2, fetches the transfer info to determine the file type, and redirects to the correct viewer page with the key in the URL fragment.

## 10. Risk Assessment

| Risk | Severity | Mitigation |
|------|----------|-----------|
| 10GB files crash browser during encryption | High | Warning message for >2GB; streaming encryption is a Villager task |
| PBKDF2 re-encryption adds user-visible delay | Low | Sub-second for <100MB; show spinner for larger files |
| Simple token brute-force | Low | 600K PBKDF2 iterations + ~39.5B combinations makes brute force impractical |
| Scope creep — too many changes at once | Medium | IFD layering (v0.2.3-a through v0.2.3-e) keeps each change small and testable |
| New pages (v0.2.4) delayed | Low | Upload flow works without new route pages; existing download page handles all modes |
