## 2. Implementation Strategy — IFD Layering

The brief is large. Implementing everything at once violates IFD principles. Each IFD iteration gets its own minor version (v0.2.3, v0.2.4, v0.2.5, etc.), consistent with how v0.2.1 and v0.2.2 already work as overlays on v0.2.0.

### Layer Plan

| IFD Version | Scope | Dependencies |
|-------------|-------|-------------|
| **v0.2.3** | Three-step state machine skeleton + step indicator + 10GB limit | None — pure refactor of `send-upload.js` |
| **v0.2.4** | Step 1: simplified drop zone + file summary display | v0.2.3 |
| **v0.2.5** | Step 2: content detection + delivery cards | v0.2.3 |
| **v0.2.6** | Step 3: PBKDF2 simple token + 3 sharing modes | v0.2.3, needs `sg-pbkdf2.js` and `sg-wordlist.js` |
| **v0.2.7** | Encrypt + upload phase with trust message carousel | v0.2.3 |
| **v0.2.8** | New route pages (`/d/{token}`, `/download/`, `/view/`, `/browse/`, `/gallery/`) — all sharing the same download code | v0.2.6 |
| **v0.2.9** | Demo links section on upload page | Independent |
| **v0.2.10** | localStorage power-user preferences | v0.2.5, v0.2.6 |

Each version is a separate commit, independently deployable, rollback-safe. Versions can go up to v0.3.10 if needed — there's no constraint on how many iterations it takes to get this right.

**Recommendation:** Build v0.2.3 first. It's the foundation everything else plugs into. Each subsequent version is independently testable.

## 3. State Machine Redesign

### Current States (v0.2.0)

```
idle → reading → encrypting → creating → uploading → completing → complete
                                                                    ↓
                                                                  error
(plus: folder-options, zipping — inserted before reading)
```

### Proposed States (v0.2.3)

**Design principle: encryption happens LAST, after all user choices.** This gives the best UX — the user makes all decisions first (fast, interactive), then the time-consuming operations (encryption + upload) run together as a single "processing" phase. No dual encryption, no re-encryption.

```
          STEP 1                  STEP 2                  STEP 3
          ──────                  ──────                  ──────
idle ──▶ folder-options ──▶ file-ready ──▶ choosing-delivery ──▶ choosing-share
                                    │                                  │
                              (file/folder                       (user confirms
                               selected,                         share mode +
                               scanned,                          token/key)
                               NOT encrypted)                         │
                                                                      ▼
                                                              PROCESSING PHASE
                                                              ════════════════
                                                    zipping ──▶ reading ──▶ encrypting ──▶
                                                    creating ──▶ uploading ──▶ completing ──▶ complete
                                                                                              │
                                                                                            error
```

**Key changes from current:**

1. **`file-ready` is the new pivot state.** File is selected and scanned (extension, folder contents analysed for content detection) but NOT encrypted. The plaintext is held in memory.
2. **`choosing-delivery` (Step 2):** User picks how the recipient will experience the file. Content detection runs on the original file metadata.
3. **`choosing-share` (Step 3):** User picks sharing mode (simple token, full link, split key). If Mode 1, the PBKDF2 token is generated/chosen here. If Mode 2/3, a random key is generated.
4. **Processing phase:** Once the user confirms, encryption starts using the correct key (PBKDF2-derived for Mode 1, random for Mode 2/3). Then upload follows immediately. The trust message carousel plays during this combined phase.

**Why this order is better:**

- **No dual encryption.** The key is known before encryption starts — whether it's PBKDF2-derived or random.
- **Faster perceived UX.** Steps 1→2→3 are instant (no waiting for encryption). The user only waits once, at the end.
- **Simple token works cleanly.** The user picks "coal-render-5512" in Step 3. PBKDF2 derives the AES key. Encryption uses that key. No re-encryption needed.
- **Back navigation is free.** Going back from Step 3 to Step 2 or Step 1 costs nothing — no encrypted blob to discard or regenerate.
- **Upload time is useful.** The combined encrypt+upload phase is where the trust carousel plays, giving the user something to read during the only waiting period.

## 4. File Size Limit Change

Straightforward. In `send-upload.js` line 171:

```javascript
// Current
static MAX_FILE_SIZE_PRESIGNED = 1024 * 1024 * 1024;        // 1GB

// Target
static MAX_FILE_SIZE_PRESIGNED = 10 * 1024 * 1024 * 1024;   // 10GB
```

Also update the `MAX_FILE_SIZE_DIRECT` threshold check in `_checkCapabilities()` — anything over 5MB goes presigned, which already works. The S3 presigned URL pipeline has no enforced cap on the backend (`/presigned/initiate` doesn't check file size).

**Risk:** 10GB files in-browser will consume significant memory during encryption. The Web Crypto API processes the entire ArrayBuffer in one call. For 10GB files, this requires ~20GB of browser memory (plaintext + ciphertext). **Streaming encryption is not yet implemented** — this is a known limitation flagged for Villager hardening.

**Mitigation for now:** Show a warning for files over 2GB: "Large files may take several minutes to encrypt. Keep this tab open."

## 5. Content Detection (Step 2)

The brief's content detection matrix (Section 6) is sound. Implementation approach:

```javascript
function detectDeliveryOptions(file, folderScan) {
    const ext = file.name.split('.').pop().toLowerCase();
    const options = ['download'];  // always available

    if (folderScan) {
        options.push('browse');
        const allImages = folderScan.entries.every(e =>
            ['png','jpg','jpeg','gif','webp','svg'].includes(
                e.name.split('.').pop().toLowerCase()
            )
        );
        if (allImages) options.push('gallery');
    } else if (['pdf','md','txt','html','htm','json','csv','xml'].includes(ext)) {
        options.push('view');
    } else if (['png','jpg','jpeg','gif','webp','svg'].includes(ext)) {
        options.push('view');
    } else if (ext === 'zip') {
        options.push('browse');
        // Could read zip directory via JSZip to check if all-images
    }

    return options;
}
```

**Recommended defaults** (marked with ★ in the brief's matrix):
- Single previewable file → `view`
- Folder with mixed files → `browse`
- Folder with all images → `gallery`
- Non-previewable file → `download` (only option)
- Zip → `browse`

The detection runs in the `file-ready` state, before encryption. Since encryption now happens last, we have direct access to the original filename and folder scan data — no need to read it back from the SGMETA envelope.

## 6. PBKDF2 Integration (Step 3 — Simple Token)

The Tools repo has the full pipeline. For the Send repo, we need to copy these modules:

| Module | Source (Tools repo) | Target (Send repo) |
|--------|--------------------|--------------------|
| `sg-pbkdf2.js` | `core/pbkdf2/v1/v1.0/v1.0.0/sg-pbkdf2.js` | `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.3/_common/js/` |
| `sg-wordlist.js` | `core/wordlist/v1/v1.0/v1.0.0/sg-wordlist.js` | Same |
| `sg-key-generator.js` | `components/key-generator/v1/v1.0/v1.0.0/` | Same (simplified version) |

**PBKDF2 parameters** (must match Tools repo exactly):
- Algorithm: PBKDF2
- Hash: SHA-256
- Iterations: 600,000
- Salt: `"sgraph-send-v1"` (static, deterministic)
- Key length: 256 bits (AES-256)

**Token format:** `{word1}-{word2}-{number}` e.g. `coal-render-5512`

The conversion is deterministic: same words → same AES-256 key. No server involvement. The recipient enters the token on the `/d/{token}` page, the same PBKDF2 derivation runs, and the derived key decrypts the file.

## 7. New Route Architecture

The brief confirms the upload page URL does **not** change during steps (Decision #1). The three-step wizard is a single-page state machine at `/{lang}/`.

### All Routes Created Now — Same Code, Different Entry Points

All new viewer pages are created as part of this upload flow work (v0.2.8). **Crucially, they all share the exact same code** — each page's `index.html` imports the same JS/CSS files that `download/index.html` already uses. The only difference is a `<meta>` tag or data attribute that tells the shared `send-download.js` which mode to force.

This means:
1. All routes work immediately (they're just the download page in a specific mode)
2. Each viewer can later be refined independently without affecting the others
3. No code duplication — one set of JS files, multiple HTML entry points

| Route | HTML File | Mode Attribute | Shared Code |
|-------|-----------|---------------|-------------|
| `/{lang}/download/{id}` | `download/index.html` | `data-mode="download"` | `send-download.js`, `crypto.js`, `api-client.js` |
| `/{lang}/view/{id}` | `view/index.html` | `data-mode="view"` | Same imports as download |
| `/{lang}/browse/{id}` | `browse/index.html` | `data-mode="browse"` | Same imports as download |
| `/{lang}/gallery/{id}` | `gallery/index.html` | `data-mode="gallery"` | Same imports as download |
| `/{lang}/d/{token}` | `d/index.html` | `data-mode="token-entry"` | Same imports + `sg-key-input.js`, `sg-pbkdf2.js` |

### Page Template (Example: `view/index.html`)

```html
<!DOCTYPE html>
<html lang="en-gb" data-mode="view">
<head>
    <!-- Same CSS imports as download/index.html -->
    <link rel="stylesheet" href="../_common/css/design-tokens.css">
    <link rel="stylesheet" href="../_common/css/shared-components.css">
    <link rel="stylesheet" href="../_common/js/components/send-download/send-download.css">
</head>
<body>
    <send-header></send-header>
    <main class="page-content">
        <send-download mode="view"></send-download>
    </main>
    <send-footer></send-footer>
    <!-- Same JS imports as download/index.html -->
    <script src="../_common/js/api-client.js"></script>
    <script src="../_common/js/crypto.js"></script>
    <script src="../_common/js/components/send-download/send-download.js"></script>
    <!-- ... other shared scripts ... -->
</body>
</html>
```

The `send-download.js` component reads its `mode` attribute (or the document's `data-mode`) and forces that specific viewer. The current auto-detection becomes the fallback when no mode is specified (backward compatible with existing `download/index.html`).

### Why This Approach

- **Once the upload flow is done, we can focus on each viewer in turn** — improving the PDF viewer, building a proper gallery component, enhancing the zip explorer — all independently
- **No new JS code needed** for the initial route creation — it's just HTML files with the right imports
- **i18n works automatically** — the build script generates locale variants of each page just like it does for the upload and download pages today

### `/d/{token}` — The Token Entry Page

This is the only genuinely new component. It shows `sg-key-input` (word-based token entry), derives the AES key via PBKDF2, fetches transfer info to determine file type, and redirects to the correct viewer page with the key in the URL fragment:

```
User enters "coal-render-5512" on /en-gb/d/coal-render-5512
  → PBKDF2 derives AES key
  → Fetches /api/transfers/info/{id} to get file type
  → Redirects to /en-gb/browse/{id}#derivedKey (if folder)
  → Or /en-gb/view/{id}#derivedKey (if PDF)
  → etc.
```

## 8. Component Architecture — Web Components

Each major UI piece should be its own **Web Component** (custom element). This keeps files small, concerns separated, and enables independent development and testing. The `send-upload.js` orchestrates the state machine but delegates rendering to child components.

### Component Breakdown

| Web Component | Tag | Responsibility | File |
|---------------|-----|---------------|------|
| `<send-upload>` | `send-upload` | State machine orchestrator, coordinates child components | `send-upload-v023.js` |
| `<send-step-indicator>` | `send-step-indicator` | Step progress display (1/2/3) | `send-step-indicator.js` |
| `<send-drop-zone>` | `send-drop-zone` | File/folder drop, browse buttons, file summary | `send-drop-zone.js` |
| `<send-delivery-picker>` | `send-delivery-picker` | Step 2 delivery cards, content detection | `send-delivery-picker.js` |
| `<send-share-mode>` | `send-share-mode` | Step 3 token/link/split mode selection | `send-share-mode.js` |
| `<send-trust-carousel>` | `send-trust-carousel` | Rotating trust messages during processing | `send-trust-carousel.js` |
| `<send-processing>` | `send-processing` | Encrypt + upload progress display | `send-processing.js` |
| `<send-token-editor>` | `send-token-editor` | Word picker for changing simple token | `send-token-editor.js` |

### Communication Pattern

Components communicate via **CustomEvents** (DOM events bubble up) and **attributes/properties** (data flows down):

```
<send-upload>  (orchestrator — listens to events, sets attributes)
  ├── <send-step-indicator step="1" total="3">
  ├── <send-drop-zone max-size="10737418240">
  │     dispatches: 'file-selected', 'folder-selected'
  ├── <send-delivery-picker file-type="pdf" file-name="report.pdf">
  │     dispatches: 'delivery-selected'  (detail: { mode: 'view' })
  ├── <send-share-mode delivery="view">
  │     dispatches: 'share-confirmed'  (detail: { mode: 'token', key: ..., token: ... })
  ├── <send-processing>
  │     dispatches: 'processing-complete', 'processing-error'
  │     └── <send-trust-carousel>
  └── (result display — inline in send-upload)
```

## 9. Backend Impact

**Minimal.** The three-step flow is entirely a frontend change. The backend API doesn't change:

| Endpoint | Change Needed |
|----------|--------------|
| `POST /api/transfers/create` | None |
| `POST /api/transfers/upload/{id}` | None |
| `POST /api/transfers/complete/{id}` | None |
| `POST /api/presigned/initiate` | None (no size cap enforced) |
| `GET /api/presigned/capabilities` | None |

The simple-token mode (PBKDF2) is entirely client-side. No new backend endpoints needed.

The only backend consideration: if we want to store the **delivery mode** chosen by the sender (so the download page knows to show a specific viewer), we'd add a `delivery_mode` field to the transfer metadata. But this can be deferred — the download page already auto-detects file type.

## 10. Risk Assessment

| Risk | Severity | Mitigation |
|------|----------|-----------|
| 10GB files crash browser during encryption | High | Warning message for >2GB; streaming encryption is a Villager task |
| Simple token brute-force | Low | 600K PBKDF2 iterations + ~39.5B combinations makes brute force impractical |
| Scope creep — too many changes at once | Medium | Each IFD version (v0.2.3 through v0.2.10) is small and testable |
| Many web components = many files | Low | Each component is small and self-contained; easier to maintain than one large file |
