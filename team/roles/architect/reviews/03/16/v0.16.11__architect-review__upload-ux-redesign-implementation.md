## 3. State Machine Redesign

### Current States (v0.2.0)

```
idle → reading → encrypting → creating → uploading → completing → complete
                                                                    ↓
                                                                  error
(plus: folder-options, zipping — inserted before reading)
```

### Proposed States (v0.2.3)

**Design principle: encryption happens LAST, after all user choices.** This gives the best UX — the user makes all decisions first (fast, interactive), then the time-consuming operations (encryption + upload) run together as a single "processing" phase. No dual encryption, no re-encryption.

```
          STEP 1                  STEP 2                  STEP 3
          ──────                  ──────                  ──────
idle ──▶ folder-options ──▶ file-ready ──▶ choosing-delivery ──▶ choosing-share
                                    │                                  │
                              (file/folder                       (user confirms
                               selected,                         share mode +
                               scanned,                          token/key)
                               NOT encrypted)                         │
                                                                      ▼
                                                              PROCESSING PHASE
                                                              ════════════════
                                                    zipping ──▶ reading ──▶ encrypting ──▶
                                                    creating ──▶ uploading ──▶ completing ──▶ complete
                                                                                              │
                                                                                            error
```

**Key changes from current:**

1. **`file-ready` is the new pivot state.** File is selected and scanned (extension, folder contents analysed for content detection) but NOT encrypted. The plaintext is held in memory.
2. **`choosing-delivery` (Step 2):** User picks how the recipient will experience the file. Content detection runs on the original file metadata.
3. **`choosing-share` (Step 3):** User picks sharing mode (simple token, full link, split key). If Mode 1, the PBKDF2 token is generated/chosen here. If Mode 2/3, a random key is generated.
4. **Processing phase:** Once the user confirms, encryption starts using the correct key (PBKDF2-derived for Mode 1, random for Mode 2/3). Then upload follows immediately. The trust message carousel plays during this combined phase.

**Why this order is better:**

- **No dual encryption.** The key is known before encryption starts — whether it's PBKDF2-derived or random.
- **Faster perceived UX.** Steps 1→2→3 are instant (no waiting for encryption). The user only waits once, at the end.
- **Simple token works cleanly.** The user picks "coal-render-5512" in Step 3. PBKDF2 derives the AES key. Encryption uses that key. No re-encryption needed.
- **Back navigation is free.** Going back from Step 3 to Step 2 or Step 1 costs nothing — no encrypted blob to discard or regenerate.
- **Upload time is useful.** The combined encrypt+upload phase is where the trust carousel plays, giving the user something to read during the only waiting period.

## 4. File Size Limit Change

Straightforward. In `send-upload.js` line 171:

```javascript
// Current
static MAX_FILE_SIZE_PRESIGNED = 1024 * 1024 * 1024;        // 1GB

// Target
static MAX_FILE_SIZE_PRESIGNED = 10 * 1024 * 1024 * 1024;   // 10GB
```

Also update the `MAX_FILE_SIZE_DIRECT` threshold check in `_checkCapabilities()` — anything over 5MB goes presigned, which already works. The S3 presigned URL pipeline has no enforced cap on the backend (`/presigned/initiate` doesn't check file size).

**Risk:** 10GB files in-browser will consume significant memory during encryption. The Web Crypto API processes the entire ArrayBuffer in one call. For 10GB files, this requires ~20GB of browser memory (plaintext + ciphertext). **Streaming encryption is not yet implemented** — this is a known limitation flagged for Villager hardening.

**Mitigation for now:** Show a warning for files over 2GB: "Large files may take several minutes to encrypt. Keep this tab open."

## 5. Content Detection (Step 2)

The brief's content detection matrix (Section 6) is sound. Implementation approach:

```javascript
function detectDeliveryOptions(file, folderScan) {
    const ext = file.name.split('.').pop().toLowerCase();
    const options = ['download'];  // always available

    if (folderScan) {
        options.push('browse');
        const allImages = folderScan.entries.every(e =>
            ['png','jpg','jpeg','gif','webp','svg'].includes(
                e.name.split('.').pop().toLowerCase()
            )
        );
        if (allImages) options.push('gallery');
    } else if (['pdf','md','txt','html','htm','json','csv','xml'].includes(ext)) {
        options.push('view');
    } else if (['png','jpg','jpeg','gif','webp','svg'].includes(ext)) {
        options.push('view');
    } else if (ext === 'zip') {
        options.push('browse');
        // Could read zip directory via JSZip to check if all-images
    }

    return options;
}
```

**Recommended defaults** (marked with ★ in the brief's matrix):
- Single previewable file → `view`
- Folder with mixed files → `browse`
- Folder with all images → `gallery`
- Non-previewable file → `download` (only option)
- Zip → `browse`

The detection runs in the `file-ready` state, before encryption. Since encryption now happens last, we have direct access to the original filename and folder scan data — no need to read it back from the SGMETA envelope.

## 6. PBKDF2 Integration (Step 3 — Simple Token)

### Dual Derivation: Transfer ID + Decryption Key

The simple token derives **two values** from the same input — not just a decryption key, but also the transfer ID. This means the token alone is enough to locate AND decrypt the file. No server-generated ID needed.

```
"coal-render-5512"
       │
       ├──▶ PBKDF2(salt: "sgraph-send-v1-id",  iterations: 600K) ──▶ Transfer ID (12 chars hex)
       │
       └──▶ PBKDF2(salt: "sgraph-send-v1-key", iterations: 600K) ──▶ AES-256 Decryption Key
```

Both derivations use the same algorithm but **different salts** to produce independent outputs. Same words → same transfer ID AND same key. Always. No server involvement in the mapping.

**This means the client sets the transfer ID** when using simple-token mode. The `POST /api/transfers/create` endpoint needs to accept an optional `transfer_id` parameter. (See Backend Impact section for AppSec considerations.)

### Modules to Copy from Tools Repo

| Module | Source (Tools repo) | Target (Send repo) |
|--------|--------------------|--------------------|
| `sg-pbkdf2.js` | `core/pbkdf2/v1/v1.0/v1.0.0/sg-pbkdf2.js` | `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.3/_common/js/` |
| `sg-wordlist.js` | `core/wordlist/v1/v1.0/v1.0.0/sg-wordlist.js` | Same |
| `sg-key-generator.js` | `components/key-generator/v1/v1.0/v1.0.0/` | Same (simplified version) |

### PBKDF2 Parameters (must match Tools repo exactly)

- Algorithm: PBKDF2
- Hash: SHA-256
- Iterations: 600,000
- **Salt for transfer ID:** `"sgraph-send-v1-id"` (produces the transfer ID)
- **Salt for decryption key:** `"sgraph-send-v1-key"` (produces the AES-256 key)
- Key length: 256 bits (AES-256) for decryption key; transfer ID is truncated to 12 hex chars

**Token format:** `{word1}-{word2}-{number}` e.g. `coal-render-5512`

### Security Note — Simple Tokens Are Not for Sensitive Data

Simple tokens trade security for convenience. The token IS the secret — anyone who guesses or discovers the token can both locate and decrypt the file. This makes simple tokens appropriate for:
- Sharing non-sensitive files with colleagues
- Demo packs, public sharing
- Situations where convenience matters more than confidentiality

**Simple tokens should NOT be the recommended mode for sensitive data.** For sensitive files, users should use Mode 2 (full link with random key in fragment) or Mode 3 (split key). The UI should communicate this clearly.

### Collision Risk and Mitigation

Two users could independently pick the same simple token (e.g. both choose "coal-render-5512"). Since the token determines the transfer ID, this would be a collision.

**Mitigations:**

1. **Existence check on create:** When the client provides a transfer ID, the server checks if it already exists. If it does, the create request fails with `409 Conflict`. The client generates a new token and retries.

2. **Entropy in the number component:** The 4-digit number (0000-9999) plus two words from a curated list gives ~39.5 billion combinations. Accidental collision probability is very low.

3. **Incident response mode (PROPOSED — does not exist yet):** If the server detects that a different IP/user agent is requesting download of a transfer that was uploaded by a different IP, it could:
   - Flag the transfer for review
   - Temporarily block downloads pending human review
   - Notify the uploader (if contact info was provided)
   - Delete the transfer as a precaution
   This is a future feature — not needed for launch but important for the security roadmap.

## 7. New Route Architecture

The brief confirms the upload page URL does **not** change during steps (Decision #1). The three-step wizard is a single-page state machine at `/{lang}/`.

### Critical Security Rule: Tokens and Keys ONLY in Hash Fragments

**Tokens and decryption keys MUST NEVER appear in the URL path or query string.** They must ONLY appear in the hash fragment (`#`), which the browser never sends to the server. This is the same principle the existing download page uses (`/download/#transferId/key`).

All routes use this pattern:

```
/{lang}/download/#token         ← simple token in fragment (Mode 1)
/{lang}/download/#id/key        ← transfer ID + key in fragment (Mode 2/3)
/{lang}/view/#token             ← simple token in fragment
/{lang}/view/#id/key            ← transfer ID + key in fragment
/{lang}/browse/#token           ← simple token in fragment
/{lang}/gallery/#token          ← simple token in fragment
```

The page JS reads the hash fragment, determines whether it's a simple token or an ID/key pair, and acts accordingly:
- **Simple token** (e.g. `#coal-render-5512`): PBKDF2 derives both transfer ID and decryption key client-side
- **ID/key pair** (e.g. `#abc123def456/base64urlKey`): direct lookup + decryption (existing behaviour)

### All Routes Created Now — Same Code, Different Entry Points

All new viewer pages are created as part of this upload flow work (v0.2.8). **Crucially, they all share the exact same code** — each page's `index.html` imports the same JS/CSS files that `download/index.html` already uses. The only difference is a `data-mode` attribute that tells the shared `send-download.js` which viewer to force.

This means:
1. All routes work immediately (they're just the download page in a specific mode)
2. Each viewer can later be refined independently without affecting the others
3. No code duplication — one set of JS files, multiple HTML entry points

| Route | HTML File | Mode Attribute | Shared Code |
|-------|-----------|---------------|-------------|
| `/{lang}/download/` | `download/index.html` | `data-mode="download"` | `send-download.js`, `crypto.js`, `api-client.js`, `sg-pbkdf2.js` |
| `/{lang}/view/` | `view/index.html` | `data-mode="view"` | Same imports as download |
| `/{lang}/browse/` | `browse/index.html` | `data-mode="browse"` | Same imports as download |
| `/{lang}/gallery/` | `gallery/index.html` | `data-mode="gallery"` | Same imports as download |

**Note:** There is no `/d/` page. Every viewer page handles both simple tokens and ID/key pairs via the hash fragment. The page type (download, view, browse, gallery) is determined by the URL path; the access credential is always in the fragment.

### Page Template (Example: `view/index.html`)


The `send-download.js` component reads `data-mode` to force a specific viewer, and reads the hash fragment to determine the access method. The current auto-detection becomes the fallback when no mode is specified (backward compatible with existing `download/index.html`).

### Hash Fragment Parsing Logic

```javascript
function parseHashFragment(hash) {
    // Remove leading #
    const fragment = hash.replace(/^#/, '');

    // Simple token: "coal-render-5512" (word-word-number pattern)
    if (/^[a-z]+-[a-z]+-\d{4}$/.test(fragment)) {
        return {
            type:  'simple-token',
            token: fragment
            // Client derives: transferId via PBKDF2(salt: "sgraph-send-v1-id")
            // Client derives: key via PBKDF2(salt: "sgraph-send-v1-key")
        };
    }

    // ID/key pair: "transferId/base64urlKey"
    const parts = fragment.split('/');
    if (parts.length === 2) {
        return { type: 'id-key', transferId: parts[0], key: parts[1] };
    }

    // ID only (key provided separately): "transferId"
    if (parts.length === 1 && fragment.length > 0) {
        return { type: 'id-only', transferId: fragment };
    }

    return { type: 'unknown' };
}
```

### Simple Token User Flow — "Read It Over the Phone"

The simple token workflow is designed for the simplest possible sharing experience. The sender can literally read the token over the phone:

```
SENDER: "Go to send.sgraph.ai/en-gb/browse and enter coal-render-5512"

RECIPIENT:
  1. Opens send.sgraph.ai/en-gb/browse/
  2. Page shows a token entry field (sg-key-input component)
  3. Types "coal-render-5512"
  4. Browser derives transfer ID + key via PBKDF2
  5. Fetches file using derived transfer ID
  6. Decrypts with derived key
  7. Shows browsable folder view

URL becomes: send.sgraph.ai/en-gb/browse/#coal-render-5512
(token in fragment — never sent to server)
```

If the sender shares the full URL instead, it also works — the recipient just clicks the link and the token is already in the fragment. But the voice/text workflow where you just say three words is the primary use case.

### Why This Approach

- **Once the upload flow is done, we can focus on each viewer in turn** — improving the PDF viewer, building a proper gallery component, enhancing the zip explorer — all independently
- **No new JS code needed** for the initial route creation — it's just HTML files with the right imports
- **i18n works automatically** — the build script generates locale variants of each page just like it does for the upload and download pages today
- **Token never hits the server** — always in the hash fragment, consistent with existing zero-knowledge architecture

## 9. Backend Impact

The three-step flow is mostly a frontend change, but the simple-token mode requires one backend change: **accepting client-provided transfer IDs.**

| Endpoint | Change Needed |
|----------|--------------|
| `POST /api/transfers/create` | **Accept optional `transfer_id` parameter.** If provided, use it instead of generating a random one. Return `409 Conflict` if the ID already exists. |
| `POST /api/transfers/upload/{id}` | None |
| `POST /api/transfers/complete/{id}` | None |
| `POST /api/presigned/initiate` | None (no size cap enforced) |
| `GET /api/presigned/capabilities` | None |

### AppSec Review: Client-Set Transfer IDs

**Why this is needed:** In simple-token mode, the transfer ID is derived from the token via PBKDF2. The client must set it so that the recipient can derive the same ID from the same token.

**Why this is acceptable (pending AppSec confirmation):**

1. **Existence check:** The server rejects the create if the ID already exists (`409 Conflict`). The client picks a new token and retries. No overwrite possible.
2. **ID format validation:** The server validates that client-provided IDs match the expected format (12 hex chars). No injection via malformed IDs.
3. **Rate limiting:** Existing rate limiting on `POST /transfers/create` applies. Brute-force enumeration of existing IDs is rate-limited.
4. **No information leakage:** A `409 Conflict` tells the client "this ID exists" — but the client can't do anything with it without also knowing the decryption key. The ID alone is useless without the key.
5. **IP hash logging:** All create requests log the hashed IP. Suspicious patterns (many 409s from one IP) can be detected.

**Remaining risk:** If an attacker guesses a simple token, they get both the transfer ID AND the decryption key (since both are derived from the token). This is inherent to Mode 1 — the token IS the secret. The mitigation is:
- Mode 1 is clearly labelled as "convenient, not maximum security"
- Modes 2 and 3 use random keys that cannot be guessed
- The incident response mode (see Section 6) can detect and respond to suspicious access patterns

**AppSec action item:** Review and confirm that client-set transfer IDs are acceptable with the above safeguards.
