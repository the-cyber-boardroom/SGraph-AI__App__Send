# Architect Review: Large Blob Support Spec (from CLI Team)

**Version:** v0.19.5
**Date:** 2026-03-28
**Spec reviewed:** `team/humans/dinis_cruz/briefs/03/28/from-team__cli/spec--large-blob-support.md`
**Status:** Review complete — raises questions the CLI team should address before server Phase 1

---

## Executive Summary

The CLI team's spec is well-structured and the core design is sound: use S3 presigned URLs to bypass Lambda payload limits for blobs > ~4 MB. The server already has a complete presigned URL infrastructure for the **transfer** flow (`Routes__Presigned` + `Service__Presigned_Urls`). The vault flow needs 3 new endpoints that mirror this pattern.

This review raises **one critical factual discrepancy** and **several architectural concerns** that should be resolved before implementation.

---

## 1. CRITICAL: Deployment Topology Discrepancy

### What the spec says
> "All vault read/write endpoints route through AWS Lambda (RequestResponse invocation). API Gateway **base64-encodes binary request bodies** before invoking Lambda, adding ~33% overhead."

### What actually exists
Per CLAUDE.md and the codebase: SGraph Send uses **Lambda Function URLs** (direct HTTPS endpoints), **not API Gateway**. The architecture section states:

> "Lambda URL Functions — direct HTTPS endpoints, no API Gateway"

### Impact
Lambda Function URLs have **different payload limits** than API Gateway:
- **API Gateway** synchronous invoke: 6 MB request, 6 MB response (base64 overhead applies)
- **Lambda Function URL** request: **20 MB** streaming request body (no base64 encoding — the body arrives as-is)
- **Lambda Function URL** response: **20 MB** (streaming response, no base64)

**If we are on Lambda Function URLs (not API Gateway), the effective limit is ~20 MB, not ~4.7 MB.** The spec's root cause analysis and threshold calculations may be wrong.

### Action Required
**CLI team must confirm the actual deployment topology.** If the production deployment does use API Gateway (e.g., via CloudFront → API Gateway → Lambda), then the spec's analysis is correct. If it uses Lambda Function URLs directly, the thresholds need recalculating and the "urgency" changes — though presigned URLs are still valuable for truly large files (> 20 MB).

## 2. API Design Consistency — Good Fit

### Proposed endpoints
```
POST  /api/vault/presigned/initiate/{vault_id}
POST  /api/vault/presigned/complete/{vault_id}
GET   /api/vault/presigned/read-url/{vault_id}/{file_id}
```

### Assessment: Follows existing patterns well

The existing presigned routes use:
```
POST  /api/presigned/initiate          (transfer-scoped)
POST  /api/presigned/complete          (transfer-scoped)
GET   /api/presigned/upload-url/{tid}  (transfer-scoped)
GET   /api/presigned/download-url/{tid}(transfer-scoped)
```

The vault equivalents nest under `/api/vault/presigned/` which is consistent with the vault namespace (`/api/vault/write/`, `/api/vault/read/`, etc.). Good.

### Minor naming note
The spec uses `read-url` for download. The existing transfer flow uses `download-url`. Consider aligning:
- Option A: `/api/vault/presigned/download-url/{vault_id}/{file_id}` (matches transfer pattern)
- Option B: `/api/vault/presigned/read-url/{vault_id}/{file_id}` (matches vault `read` nomenclature)

Both are defensible. Option B is slightly better since vault operations use read/write terminology, not upload/download.

## 3. Auth Model — Write-Key on Initiate/Complete

### What the spec says
> "Write key required on all initiate/complete calls (not on the S3 presigned PUT itself — the presigned URL is self-authenticating)"

### Assessment: Correct

This matches the existing vault auth pattern exactly:
- `Routes__Vault__Pointer` validates `x-sgraph-vault-write-key` header on write/delete/batch
- The service layer hashes the key and checks against the vault manifest
- Reads are unauthenticated (data is encrypted, key is useless without the client-side AES key)

The presigned `read-url` endpoint should similarly be **unauthenticated** (same as `/api/vault/read/`). The spec doesn't explicitly state this but the endpoint signature (no write key) implies it. **Confirm with CLI team.**

---

## 4. Storage Layer Bypass — Acceptable but Needs Dual-Mode

### Concern
The presigned flow sends data **directly to S3**, bypassing `Storage_FS` (Memory-FS). This is architecturally correct for production but creates a problem:

- **All tests use in-memory `Storage_FS`** — there's no S3 to generate presigned URLs against
- The existing `Service__Presigned_Urls` already handles this: `is_s3_mode()` returns False in memory mode, and the service returns `presigned_not_available`
- For **transfers**, this works because clients fall back to `/api/transfers/upload` (direct Lambda upload)

### For vaults, the fallback path already exists
The CLI client already uses `/api/vault/write/` for normal blobs. Large blobs would use presigned URLs. In memory mode (tests), there are no truly large blobs — so the existing write path covers it.

### Recommendation
The vault presigned service should follow the same `is_s3_mode()` pattern. Return `presigned_not_available` in memory mode. The CLI client must handle this gracefully (fall back to direct write, accepting the size limit in test environments).

## 5. `Schema__Object_Tree_Entry` — `large: bool` Field

### What exists
`Schema__Object_Tree_Entry` does **not exist in the server codebase**. It only appears in briefs. The server stores vault files as opaque blobs via `Service__Vault__Pointer` — it has no concept of a "tree" or "tree entries".

### Assessment
The `large: bool` flag is a **client-side schema concern**. The server doesn't need to know or care whether a blob is "large" — it just stores bytes at a path. The tree is a client-side data structure that the client writes to the vault as an encrypted blob.

**No server-side schema change is needed for Phase 1.** The CLI team adds `large: bool` to their client-side tree entry schema. The server just stores whatever bytes the client sends.

### Spec Decision #4 is misleading
> "**`large: bool` added to server-side tree entry schema** — server already has `Schema__Object_Tree_Entry`"

The server does **not** have this schema. This is a client-side schema. The spec should clarify this to avoid confusion.

## 6. Presigned URL TTL — Spec Says 15-30 Minutes, Code Uses 1 Hour

The existing `Service__Presigned_Urls` uses:
```python
PRESIGNED_UPLOAD_EXPIRY   = 3600   # 1 hour
PRESIGNED_DOWNLOAD_EXPIRY = 3600   # 1 hour
```

The spec says "default 15–30 minutes (1h upper bound)". The vault endpoints should use the same constants for consistency, or the spec should clarify why vault URLs need shorter TTLs.

## 7. Edge Cases and Risks

### 7a. Orphaned Multipart Uploads
If a client initiates a multipart upload but never completes it, S3 retains the parts indefinitely (costing storage). The existing transfer flow has an `/abort` endpoint. **The vault spec does not include an abort endpoint.**

**Recommendation:** Add `POST /api/vault/presigned/abort/{vault_id}` (mirrors `/api/presigned/abort/{transfer_id}/{upload_id}`). Also consider an S3 lifecycle rule to auto-expire incomplete multipart uploads after 24 hours.

### 7b. Concurrent Uploads to Same `file_id`
Two clients could initiate presigned uploads for the same `file_id` in the same vault simultaneously. The last `complete` call wins (S3 is eventually consistent for overwrites). This is the same semantics as concurrent `/api/vault/write/` calls — acceptable.

### 7c. `file_id` Validation
The spec passes `file_id` in the POST body (initiate/complete) rather than the URL path. The existing vault routes use `{file_id:path}` in the URL. The new endpoints should validate `file_id` follows the same patterns (e.g., `bare/data/{blob_id}`).

### 7d. Download URL for Non-Existent Blobs
`GET /api/vault/presigned/read-url/{vault_id}/{file_id}` should verify the blob exists in S3 before generating a URL. Otherwise the client gets a presigned URL that returns 404 from S3.

## 9. Recommendation

**Proceed with Phase 1 implementation** after resolving Question #1 (deployment topology). The existing `Service__Presigned_Urls` provides a solid template. The new vault presigned service can reuse:
- The same S3 multipart upload/complete/abort logic
- The same `is_s3_mode()` dual-mode pattern
- The same presigned URL generation via `osbot-aws` S3 client

New code needed:
- `Routes__Vault__Presigned` (new route class, ~80 lines, mirrors `Routes__Presigned`)
- `Service__Vault__Presigned` (new service, ~150 lines, adapts `Service__Presigned_Urls` for vault auth)
- Wire into `Fast_API__SGraph__App__Send__User` app setup

Estimated effort: Small-medium. Most logic exists; this is primarily adaptation to vault auth patterns.
