# Architect Review: v0.3.1 Development Strategy

**Version:** v0.19.5
**Date:** 28 March 2026
**Role:** Architect
**Context:** v0.3.0 is now in production. All future UI changes must go into v0.3.1+ as IFD minor versions. This review defines how.

---

## The Constraint

v0.3.0 is a **major IFD version** — fully consolidated, self-contained, deployed to production. Per IFD rules:

- v0.3.0 files are **frozen** — no modifications allowed
- v0.3.1 is a **minor IFD version** — contains **only surgical overrides**, not full file copies
- v0.3.1 depends on v0.3.0 (loads v0.3.0 first, then v0.3.1 patches on top)

This is exactly how v0.2.1 through v0.2.17 worked on top of v0.2.0.

## Recommended Strategy: IFD Surgical Overlays

### Directory Structure

```
sgraph_ai_app_send__ui__user/
  v0/v0.3/
    v0.3.0/                          # FROZEN — production code, do not modify
      _common/js/components/...
      en-gb/index.html
      ...

    v0.3.1/                          # NEW — surgical overrides only
      _common/js/components/
        send-download/
          send-gallery-v031.js       # Prototype overrides for SendGallery
          send-gallery-v031.css      # CSS additions/overrides
          send-browse-v031.js        # Prototype overrides for SendBrowse
        send-upload/
          send-upload-v031.js        # Prototype overrides for SendUpload
          upload-step-done-v031.js   # Overrides for UploadStepDone
      en-gb/
        index.html                   # Extends v0.3.0 — loads base then patches
        browse/index.html            # Extends v0.3.0 browse page
        gallery/index.html           # Extends v0.3.0 gallery page
```

### How It Works

**The HTML pages in v0.3.1 load v0.3.0 base code first, then apply v0.3.1 patches:**

```html
<!-- v0.3.1/en-gb/index.html -->
<head>
    <!-- v0.3.0 base styles -->
    <link rel="stylesheet" href="../../v0.3.0/_common/css/design-tokens.css">
    <link rel="stylesheet" href="../../v0.3.0/_common/js/components/send-upload/upload-step-select/upload-step-select.css">
    <!-- ... all v0.3.0 CSS ... -->

    <!-- v0.3.1 CSS overrides (loaded AFTER base — later rules win) -->
    <link rel="stylesheet" href="../_common/js/components/send-upload/upload-step-select-v031.css">
</head>
<body>
    <!-- v0.3.0 base scripts -->
    <script src="../../v0.3.0/_common/js/crypto.js"></script>
    <script src="../../v0.3.0/_common/js/api-client.js"></script>
    <!-- ... all v0.3.0 JS ... -->

    <!-- v0.3.1 surgical overrides (loaded AFTER — prototype patches win) -->
    <script src="../_common/js/components/send-upload/send-upload-v031.js"></script>
</body>
```

### Override Patterns

**JavaScript — Prototype override (for class methods):**

```javascript
// v0.3.1/_common/js/components/send-upload/send-upload-v031.js
// Override ONLY the _openEmailLink method (U-030 security fix)

SendUpload.prototype._openEmailLink = function() {
    if (!this.result) return;
    var subject = 'Encrypted file via SG/Send';
    var body;
    if (this._shareMode === 'separate') {
        body = "I've sent you an encrypted file via SG/Send.\n\n" +
               "Open this link to download:\n" + (this.result.linkOnlyUrl || '') + "\n\n" +
               "You'll need the decryption key to open it — I'll send that separately.";
    } else if (this._shareMode === 'token' && this.result.friendlyKey) {
        var tokenLink = this.result.tokenLink || this.result.combinedUrl || '';
        body = "I've sent you an encrypted file via SG/Send.\n\n" +
               "Open this link to view it:\n" + tokenLink;
    } else {
        body = "I've sent you an encrypted file via SG/Send.\n\n" +
               "Open this link to view it:\n" + (this.result.combinedUrl || '');
    }
    window.open('mailto:?subject=' + encodeURIComponent(subject) +
                '&body=' + encodeURIComponent(body), '_blank');
};
```

**CSS — Later rule wins (for style changes):**

```css
/* v0.3.1/_common/js/components/send-upload/upload-step-select-v031.css */
/* U-001: File/Text toggle — compact pills instead of chunky buttons */
.mode-toggle__btn {
    padding: 0.375rem 0.75rem;
    border: none;
    font-size: 0.8rem;
}
```

**Shadow DOM consideration:** Components that use Shadow DOM (like `upload-step-done`, `upload-step-confirm`) load their own templates internally. For these, the override pattern is different:

1. **If the component uses `customElements.define()`** — you can't redefine it. Instead, override prototype methods after the base script loads.
2. **If the component renders HTML in a method** — override that render method via prototype to change the HTML output.
3. **If only CSS changes are needed inside Shadow DOM** — the component must adopt external stylesheets or the override must patch the render method to inject `<style>` elements.

## Two Possible Approaches

### Option A: Pure IFD Overlays (Recommended)

Follow the IFD pattern exactly as v0.2.1–v0.2.17 did. v0.3.1 pages reference v0.3.0 base and load patches on top.

**Pros:**
- v0.3.0 is truly untouched — zero risk to production
- Clear audit trail — every v0.3.1 file is a deliberate change
- Can deploy v0.3.1 alongside v0.3.0 (different URL paths)
- Rollback is trivial — just stop serving v0.3.1 paths

**Cons:**
- Some v0.3.0 components use Shadow DOM with internal templates — harder to override CSS
- HTML pages need to be duplicated (with different script loading order)
- Prototype overrides can be fragile if the base method signature changes

### Option B: Copy-and-Patch v0.3.0 into v0.3.1 (Not Recommended)

Copy the entire v0.3.0 folder to v0.3.1, then make changes directly.

**Pros:**
- Simple — just edit files directly
- No overlay complexity

**Cons:**
- **Violates IFD rules** — v0.3.1 minor versions should be surgical, not full copies
- 113 files duplicated, only ~10 changed — massive redundancy
- Unclear what changed — no diff visibility at the file level
- Harder to merge back into v0.4.0 consolidation

---

## Recommendation

**Use Option A (pure IFD overlays)** with one pragmatic modification:

**For Shadow DOM components that need internal template changes** (like `upload-step-done`, `upload-step-confirm`), create the override file with both the prototype patch AND a comment explaining which base method it replaces. This keeps the surgical intent clear while handling the Shadow DOM reality.

### Implementation Steps

1. **Create `v0.3.1/` directory** with only the files that differ from v0.3.0
2. **Create v0.3.1 HTML pages** that load v0.3.0 base + v0.3.1 patches
3. **For each fix from the QA triage**, create the minimal override file
4. **Deploy v0.3.1 at a new URL path** (`/v0/v0.3/v0.3.1/en-gb/`)
5. **Update the routing** to serve v0.3.1 as the default (keep v0.3.0 accessible for rollback)

## How Deployment Works

### Current pipeline

The CI/CD pipeline copies the **latest versioned folder** from `sgraph_ai_app_send__ui__user/v0/v0.3/` into the deployment target. The key behaviour:

1. **All `v0.3.x` folders exist on disk** — `v0.3.0/`, `v0.3.1/`, etc.
2. **The deployment copies all of them** into the S3/static hosting target
3. **The routing** points production URLs (`/en-gb/`) to the latest minor version's HTML pages

### Why it works with IFD overlays

v0.3.1's HTML pages reference v0.3.0 base files via relative paths:

```
v0.3.1/en-gb/index.html
  → loads ../../v0.3.0/_common/js/crypto.js
  → loads ../../v0.3.0/_common/js/api-client.js
  → loads ../../v0.3.0/_common/js/components/send-upload/send-upload.js
  → (then) loads ../v0.3.1 patches
```

Since both `v0.3.0/` and `v0.3.1/` folders are deployed side by side, the relative paths resolve correctly:

```
deployed/
  v0/v0.3/
    v0.3.0/                  ← full base (frozen)
      _common/js/...
      en-gb/index.html       ← v0.3.0 upload page (still accessible)
    v0.3.1/                  ← overlays only
      _common/js/...         ← only patch files (v031.js, v031.css)
      en-gb/index.html       ← loads v0.3.0 base + v0.3.1 patches
```

### No file conflicts

v0.3.1 patch files use a `-v031` suffix (`send-upload-v031.js`, `send-gallery-v031.css`), so they never collide with v0.3.0 filenames. The only file that "replaces" a v0.3.0 counterpart is `en-gb/index.html` — but it lives in `v0.3.1/en-gb/`, not `v0.3.0/en-gb/`, so there's no conflict.

### Routing update

Once v0.3.1 is validated, the production route (`/en-gb/`) is updated to serve from `v0.3.1/en-gb/` instead of `v0.3.0/en-gb/`. Rollback is instant — just point back to `v0.3.0/en-gb/`.

### MVP verification

The v0.3.1 MVP is a single file (`v0.3.1/en-gb/index.html`) that loads the entire v0.3.0 base and sets `uiVersion` to `'v0.3.1'`. If the footer shows "v0.3.1", the overlay pipeline works.
