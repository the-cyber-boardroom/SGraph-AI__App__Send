## Problem Statement

Two codebases do the same thing — display a tree of files with tabbed preview:

| | Browse (`send-browse.js` v0.3.1) | Vault (`vault-shell.js` v0.1.4) |
|---|---|---|
| Tree rendering | `_buildFolderTree()` + `_renderFolderNode()` | `vault-tree-view` component |
| File preview | `_renderFileContent()` — 8 types | `_renderFileInTab()` — 4 types |
| sg-layout | 22/78 split, `s-tree` + `s-preview` | 20/80 split, `s-files` + `s-preview` |
| Tab management | `_openTabs` Map, `_ensurePreviewStack()` | `_openFileTab()`, `_ensurePreviewStack()` |

Browse has richer previews (markdown links/images, CSV tables, HTML iframes, PDF, source toggles). Vault has richer operations (edit/save, rename, delete, SGit). Features added to one don't appear in the other.

## Architecture Decision: Data Source Adapter Pattern

### The Interface

The shared Browse module needs exactly **3 methods** from a data source:

```javascript
/**
 * @typedef {Object} BrowseDataSource
 *
 * @property {function(): TreeNode} getTree
 *   Returns the root tree node: { name, children: {name: TreeNode}, files: [{path, name, size}] }
 *
 * @property {function(string): Promise<ArrayBuffer>} getFileBytes
 *   Given a file path, returns the decrypted file content as ArrayBuffer
 *
 * @property {function(): FileEntry[]} getFileList
 *   Returns flat list of all files: [{path, name, dir, size}]
 *   Used for link resolution, auto-open, gallery filtering
 */
```

Plus optional write operations (vault only):

```javascript
/**
 * @typedef {Object} BrowseDataSourceWritable extends BrowseDataSource
 *
 * @property {boolean} writable — true if edit/delete/rename available
 *
 * @property {function(string, string, ArrayBuffer): Promise} saveFile
 *   Save file content: (folderPath, fileName, bytes) → commit
 *
 * @property {function(string, string, string): Promise} renameFile
 *   Rename: (folderPath, oldName, newName) → commit
 *
 * @property {function(string, string): Promise} deleteFile
 *   Delete: (folderPath, fileName) → commit
 */
```

### Two Adapters

**ZipDataSource** (for send.sgraph.ai/browse):
```javascript
class ZipDataSource {
    constructor(zipInstance, zipTree, zipOrigBytes, zipOrigName) { ... }

    getTree()              // builds hierarchical tree from flat zipTree
    getFileBytes(path)     // calls entry.async('arraybuffer')
    getFileList()          // returns zipTree filtered (!dir, !gallery)
    writable = false
}
```

**VaultDataSource** (for vault.sgraph.ai):
```javascript
class VaultDataSource {
    constructor(vault, accessKey) { ... }

    getTree()              // converts vault._tree to Browse tree format
    getFileBytes(path)     // calls vault.getFile(folderPath, fileName) + decrypt
    getFileList()          // flattens vault tree to flat list
    writable               // true if accessKey present
    saveFile(...)          // calls vault.updateFile()
    renameFile(...)        // calls vault.renameFile()
    deleteFile(...)        // calls vault.removeFile()
}
```

## Implementation Plan

### Phase 1: Extract Browse Module (Browse team work)

**Goal:** Make `send-browse.js` data-source-agnostic without breaking existing zip functionality.

This requires changes to the Browse codebase. **See separate brief for Browse team:** `team/comms/03/30/v0.20.6__browse-team-brief__shared-module-extraction.md`

Summary of Browse team deliverables:
1. Accept a `dataSource` property instead of raw `zipTree`/`zipInstance`/`zipOrigBytes`
2. Replace all `entry.async('arraybuffer')` calls with `dataSource.getFileBytes(path)`
3. Replace `_buildFolderTree()` with `dataSource.getTree()`
4. Replace `this.zipTree.filter(...)` with `dataSource.getFileList()`
5. Create `ZipDataSource` adapter class that wraps current zip logic
6. Existing `send-download.js` creates `ZipDataSource` and passes it in
7. **Zero user-visible change** — existing Browse view works exactly as before

### Phase 2: Create VaultDataSource Adapter (Vault team work)

**Goal:** Bridge the vault API into the Browse data source interface.

```
sgraph_ai_app_send__ui__vault/v0/v0.1/v0.1.5/
  _common/js/adapters/
    vault-data-source.js         ← NEW: VaultDataSource class
```

The adapter translates between vault's tree format and Browse's expected format:

```javascript
// Vault tree format:
{ '/': { type: 'folder', children: {
    'README.md': { type: 'file', blob_id: '...', size: 1234 },
    'docs': { type: 'folder', children: { ... } }
}}}

// Browse tree format:
{ name: '', children: {
    'docs': { name: 'docs', children: {...}, files: [] }
}, files: [
    { path: 'README.md', name: 'README.md', size: 1234 }
]}
```

Key design decisions:
- **Lazy loading preserved** — `getFileBytes()` fetches + decrypts on demand (no pre-download)
- **Write operations exposed** — `writable` flag enables edit/save/rename/delete in Browse UI
- **Gallery metadata** — vault doesn't use `_gallery.` folders, so no filtering needed

### Phase 3: Vault Shell Integration (Vault team work)

**Goal:** Replace vault-shell's custom file rendering with the shared Browse module.

**What vault-shell keeps:**
- Header (brand, vault name, lock, upload button)
- Nav (Files / SGit / Settings)
- Auth banner (access key)
- Status bar
- Settings view
- SGit view (commit history, refs, tree, object inspector)
- `_ensurePreviewStack()` (already in Browse too)

**What vault-shell delegates to Browse:**
- Tree rendering (currently `vault-tree-view`)
- File preview + tabs (currently `_openFileTab` + `_renderFileInTab`)
- File type detection and rendering (markdown, PDF, CSV, HTML, images, etc.)

**Integration approach:**

```
vault-shell.js
  ├── Header / Nav / Auth / Status / Settings / SGit   ← stays
  ├── <sg-layout>                                       ← stays
  │     ├── s-files: shared Browse module               ← NEW (replaces vault-tree-view + custom tabs)
  │     │     └── dataSource: VaultDataSource
  │     └── s-sgit: vault-sgit-view                     ← stays
  └── Action bar extensions (Edit/Save/Rename/Delete)   ← hooked into Browse's action bar
```

The Browse module renders inside vault-shell's sg-layout. Vault-shell listens for Browse events and adds vault-specific action buttons (Edit, Save, Rename, Delete) to the file action bar.

**Event flow:**
```
Browse module emits:   'browse-file-opened'  → vault-shell adds Edit/Delete buttons
Browse module emits:   'browse-file-saved'   → vault-shell calls dataSource.saveFile()
Browse module emits:   'browse-file-renamed'  → vault-shell calls dataSource.renameFile()
```

### Phase 4: Fractal Vaults (Future — after Phase 3 works)

**Goal:** Open vaults inside vaults using sg-layout nesting.

A parent vault contains "vault reference" entries — files with a `.vault` extension or a special `type: 'vault-ref'` in the tree. These contain:
```json
{
    "vault_key": "passphrase:vault_id",
    "name": "Project Alpha Vault"
}
```

When the user clicks a vault reference:
1. Browse module emits `'browse-vault-ref-clicked'`
2. Vault-shell opens a new VaultDataSource for the child vault
3. Child vault Browse instance opens in a new sg-layout tab (or nested panel)
4. Breadcrumb shows: `Master Vault > Project Alpha Vault > docs/`

sg-layout already supports fractal nesting (panels within panels), so the layout infrastructure is ready.

## Migration Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Browse module change breaks existing send.sgraph.ai | Phase 1 creates ZipDataSource as a transparent wrapper — existing behaviour is unchanged |
| Vault write operations don't fit Browse's read-only model | `writable` flag + event emission — Browse only shows buttons when adapter supports it |
| Style conflicts between Browse CSS and vault CSS | Browse CSS uses `sb-` prefix, vault uses `vs-` — no collision. Shared via `<link>` |
| sg-layout version mismatch | Both use same CDN version (v0.1.0). Verify before integration |
| FileTypeDetect differences | Both already use the same `FileTypeDetect` class. Vault's version now includes .txt (this session). Sync when integrating |
