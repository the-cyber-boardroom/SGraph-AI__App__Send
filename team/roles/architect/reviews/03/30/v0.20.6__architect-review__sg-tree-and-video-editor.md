# Architect Review: sg-tree API Contract and Video Editor Architecture

**Version:** v0.20.6 | **Date:** 30 March 2026 | **Role:** Architect
**Briefs reviewed:** Documents 185 (sg-tree API contract), 186 (Simple Token debrief), 187 (video editor)

---

## Executive Summary

Three briefs from 29 March deliver a mix of **code-ready specification** (sg-tree), **shipped implementation** (simple token vault), and **early-stage feature brief** (video editor). Architectural quality is high across all three. The sg-tree brief follows the proven sg-layout pattern and is ready to implement. The simple token implementation introduces a clean PBKDF2+HKDF key derivation model. The video editor brief is ambitious and architecturally sound but needs scoping before implementation begins.

---

## 1. sg-tree: API Contract Assessment

### 1.1 Strategic Alignment

The case for sg-tree is strong. Three production tree implementations exist with zero shared code. The duplication creates maintenance debt and causes bugs to diverge (e.g., the basename fix in v0.3.1 did not reach vault or workspace). The sg-tree brief correctly models this as a `send-browse` / `vault-tree-view` / `vault-panel` unification problem with an adapter layer solution.

The chosen approach — adapter per consumer, canonical data format, Shadow DOM, CSS custom properties — directly mirrors the proven sg-layout pattern. This is a safe architectural bet.

### 1.2 Data Format: Approved

The canonical node format `{id, label, type, children, icon, meta, badge, disabled}` is well-designed:

- `id` uniqueness requirement matches sg-layout's tab id contract
- `meta` pass-through allows consumers to carry arbitrary domain data without coupling sg-tree to their schema
- `type` as an extensible string (not enum) allows custom types without modifying sg-tree
- `badge` as an optional string (not a number) allows display flexibility (count, indicator, status)

**Minor concern:** The brief does not specify behaviour when `id` is not unique. The rule "duplicate ids cause undefined behaviour" should be tightened to either: (a) throw on `setData()` with duplicate ids, or (b) silently deduplicate. The current wording leaves it undefined, which is a maintenance hazard. **Recommendation:** Add runtime validation in `setData()` and throw a descriptive error.

### 1.3 Public API: 12 Methods

The method surface is appropriate. No method is superfluous for v0.1.0.

Specific observations:

| Method | Assessment |
|--------|-----------|
| `setData(root)` | Correct single entry point. Immutable data model. |
| `getData()` | Deep clone is correct — prevents accidental external mutation. |
| `getSelectedId()` → `getSelectedId(): string[] | null` | For `selectable: 'multi'` mode, this returns only one ID. **Gap:** Need `getSelectedIds()` for multi-select or change return type to `string[]` always. |
| `expandPath(id)` | High-value method. Needed for "reveal currently selected file" use case (common in vault/workspace). |
| `filter(fn)` | Correct. Null clears. |
| `setExpandedIds(ids)` | Enables serialisation round-trip. Good for workspace tab restore. |

**Recommendation:** Clarify `getSelectedId()` vs `getSelectedIds()` before implementation. One method that always returns `string[]` (empty, single, or multi) is cleaner than two methods with overlapping semantics.

### 1.4 CSS Contract

The 17 CSS custom properties are complete. The naming convention (`--sgt-*`) is consistent. The shadow boundary is correctly stated to be pierce-able via custom properties.

**One gap:** No `--sgt-font-family` or `--sgt-font-size` is listed. Both sg-layout and sg-tree consumers (send-browse, vault-panel) inherit from their host page's font stack. An explicit `--sgt-font-family: inherit` default (with an override prop) would be cleaner. **Low priority.**

### 1.5 Event Architecture

The separate `sg-tree-events.js` event constants file is the correct pattern — consumers import constants, not magic strings. This is the same approach used in the workspace EventBus.

The set of events (select, expand, collapse, toggle, context-menu) covers the main interaction surface. The `meta` pass-through in events means consumers can act on domain data without querying the tree, which correctly enforces the unidirectional data flow.

**One concern:** The brief does not address keyboard navigation for `selectable: 'multi'` mode (shift+click, ctrl+click). This is important for the vault use case (multi-select for batch operations). **Recommendation:** Document keyboard nav scope in v0.1.0 (single-select only?), with multi-select as v0.1.1 feature.

### 1.6 Implementation Scope: v0.1.0 vs Later

The brief correctly separates the read-only base (v0.1.0) from edit mode (v0.1.1) and virtual scrolling (v0.2.0). This is the right IFD discipline.

**Concern:** The brief lists virtual scrolling as v0.2.0 but does not specify what "large" means. For vault trees (typically <500 nodes), virtual scrolling is premature. For the ontology/dependency graph use case (potentially thousands of nodes), it may be needed early. **Recommendation:** Define a threshold (e.g., 1,000+ nodes triggers virtual scroll) in the design notes, to set expectations for when v0.2.0 is needed.

### 1.7 Hosting Path

```
tools.sgraph.ai/core/sg-tree/v0/v0.1/v0.1.0/sg-tree.js
tools.sgraph.ai/core/sg-tree/v0/v0.1/v0.1.0/sg-tree-events.js
```

This is consistent with the IFD path convention and the sg-layout hosting model.

**Overall verdict:** sg-tree spec is ready for implementation. Resolve the `getSelectedId()` multi-select ambiguity before starting. Estimated v0.1.0 scope: 400–600 lines of JS (3 modules: tree renderer, event manager, state machine).

## 2. Simple Token Vault: Implementation Review

### 2.1 Crypto Model

The three-way HKDF split (read_key, write_key, ec_seed) from a PBKDF2-derived root is cryptographically sound and matches the browser Web Crypto API derivation used by SG/Send transfer AES-GCM. Byte-for-byte compatibility is critical for the `sgit clone <share_token>` import scenario.

The key properties are correct:
- Read, write, and EC seed keys are cryptographically independent (distinct HKDF `info` labels)
- PBKDF2 uses 600,000 iterations (SHA-256, salt `b'sgraph-send-v1'`) — matches current OWASP recommendation
- Token pattern `^[a-z]+-[a-z]+-\d{4}$` is unambiguous and does not conflict with existing vault key formats (hex strings, colon-separated keys)

### 2.2 Clone Resolution Flow

The two-step clone resolution (SGit-AI first, then SG/Send) is pragmatic. The design correctly handles the asymmetry between edit tokens (live on SGit-AI) and share tokens (live on SG/Send). The fallback generates a new edit token from the share token import — a clean separation that prevents share-token holders from having implicit edit access.

**One concern:** The brief notes "EC key generation still uses random" despite `ec_seed` being derived. This means the vault is not fully reconstructible from the edit token alone. The debrief flags this as a future sprint item. **This is a known gap, not a bug.** The current crypto model does not weaken security — it only defers the "reconstructible from token" property.

### 2.3 Config Schema

The `local/config.json` fields (`mode`, `edit_token`, `share_token`, `share_transfer_id`) are minimal and backward-compatible. The `share_transfer_id` caching note ("technically redundant but cached to avoid recomputing") is a reasonable performance trade-off. Document this explicitly to avoid future confusion.

### 2.4 Outstanding Items

From the debrief's "What Remains":

| Item | Priority | Notes |
|------|----------|-------|
| `sgit share --auto` / `auto_share: true` | Medium | Useful for newsletter workflow — every push refreshes the URL |
| Integration tests (live server) | High | 22 new unit tests are good; live round-trip test needed before PyPI release |
| EC key determinism from ec_seed | Low | Aesthetic/completeness; not a security issue |
| Token loss warning at init | High | "Save your edit token — it cannot be recovered" needs to be prominent. Data loss risk. |
| `sgit share --token <t>` | Low | Nice-to-have for first publish |

**Overall verdict:** Solid implementation. The warning at `sgit init` is the most important outstanding item — without it, users will lose access to their vaults.

## 3. Video Editor: Architecture Assessment

### 3.1 JSON Transformation Model

The brief's core architectural insight — "every edit is a JSON transformation, the source file is never modified until export" — is the right approach for this class of application. Non-destructive editing with an append-only transformation log supports undo/redo, replay, remixing, and LLM interoperability.

The transformation schema examples are well-designed:
- `type` + `params` + `applied_at` is minimal and sufficient
- Keyframe format for masks is correct for position interpolation
- "Consolidation" (collapsing incremental movements into one) is the right optimisation for committed vs. history state

**This model is architecturally sound and should be standardised early**, before features diverge. Define `schema/Transform.py` with `Type_Safe` for the backend-portable canonical form, even if the video editor is client-only.

### 3.2 LLM Integration via Screenshots

Using screenshots (not raw video) as the LLM interface is pragmatic and correct. Video frames at 1–2 second intervals are sufficient for captioning and mask suggestion. This also keeps the integration token-efficient.

The brief correctly separates:
- LLM sees screenshots + JSON → produces transformation JSON
- The video editor applies transformations to the actual video
- No video data crosses the LLM API boundary

This maintains the zero-knowledge property for users who process sensitive content (the MRI example).

### 3.3 `sg-local-storage` Component

This is the right abstraction. An IndexedDB-backed component for persisting large files browser-side is independently useful across SG/Send tools (video editor, zip editor, vault browser). The "viewer showing what is stored locally" + capacity management addresses a real UX pain point.

**Architectural note:** `sg-local-storage` should be a first-class IFD-versioned component hosted at `tools.sgraph.ai/core/sg-local-storage/`. It is not video-editor-specific and should be extracted from that scope.

### 3.4 Asset Registry

The SG/Send token integration in the asset registry is clean. Referencing a transfer by its friendly token to pull assets bridges the two product surfaces. This implicitly validates the simple token model from Document 186 — the asset registry becomes a consumer of `sgit clone <token>`.

### 3.5 Scope Concerns

The brief commissions a significant feature set. From an implementation priority perspective:

| Feature | Complexity | Priority |
|---------|-----------|----------|
| `sg-local-storage` component | Low–Medium | **P1** — standalone reusable; unblocks video editor |
| JSON transformation model + undo/redo | Medium | **P1** — core architecture; do first |
| Crop + trim (extends existing) | Low | **P1** — extends "working" foundation |
| Multiple timeline tracks | High | **P2** — UI complexity; implement after core model |
| Caption support | Medium | **P2** — timeline track dependency |
| Masking with keyframes | High | **P2–P3** — keyframe interpolation is complex |
| LLM integration (screenshots) | Medium | **P3** — needs timeline tracks first |
| Asset registry + token integration | Medium | **P3** — depends on LLM + local storage |
| Export (FFmpeg render) | Medium | **P2** — needed before LLM features |

**Recommendation:** Scope v0.1.0 as: JSON transformation model + `sg-local-storage` + crop/trim. Leave timelines, captions, masking, and LLM for subsequent overlays. This gives a working foundation in one IFD cycle.

### 3.6 Reality Check

The brief says "the first version of the video editor is working" with frame extraction, cropping, and basic trimming. This is not listed in the reality document. **Before the Dev team starts implementation, verify what already exists** — inspect the tools.sgraph.ai codebase for any video-editor directory. If nothing exists, v0.1.0 starts from scratch.

## 4. Cross-Document Implications

### sg-tree as Foundation for Video Editor

The timeline in a video editor is a specialised tree structure (tracks → clips → keyframes). Once sg-tree v0.1.0 is shipped, the video editor's timeline navigation could consume sg-tree with a custom `type: 'track'` adapter. This is not critical for v0.1.0 but is worth noting for v0.2.0 planning.

### Simple Token as Asset Reference

The video editor's asset registry uses "friendly tokens" to reference SG/Send transfers. The simple token model (Document 186) provides exactly this: `coral-equal-1234` as a human-readable identifier for a vault or transfer. The asset registry should use the same pattern.

### `sg-local-storage` as Cross-Product Primitive

This component has applications beyond the video editor:
- Workspace: persist active document between sessions
- Vault browser: cache decrypted file list locally
- QA tools: persist test session state

**Recommend:** Treat `sg-local-storage` as a core primitive alongside `sg-tree` and `sg-layout`. Design it for general use from day one.
