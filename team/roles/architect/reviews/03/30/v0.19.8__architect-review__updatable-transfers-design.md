# Architect Review: Updatable Transfers Design

**Version:** v0.19.8
**Date:** 2026-03-30
**Role:** Architect
**Status:** Design Options — awaiting human decision

---

## Problem Statement

The transfer system (`/api/transfers/`) is currently **immutable** — once a file is uploaded, it cannot be updated. Three use cases require mutability:

1. **Folder via simple token + add files later** — create a transfer with a simple token, then push additional files to it over time
2. **Publish vault snapshot to a stable simple token URL** — a vault owner wants a stable, read-only URL (simple token) that always resolves to the latest vault state. The CLI already supports creating vaults from simple tokens and pushing vaults to simple tokens.
3. **Large file zip as separate uploads (future)** — breaking a zip into individual file entries uploaded separately, requiring the transfer to accept multiple payloads

---

## Current Architecture (What Exists)

### Transfer System (`/api/transfers/`)
- **Data model:** `transfers/{transfer_id}/meta.json` + `transfers/{transfer_id}/payload`
- **Lifecycle:** `pending` → (upload payload) → `completed`
- **Auth:** access token only (`x-sgraph-send-access-token`)
- **ID generation:** random 12-hex or client-provided (PBKDF2 simple-token mode)
- **Immutability enforced by:** `upload_payload()` rejects if `status != 'pending'`
- **Single payload file:** one `payload` blob per transfer

### Vault System (`/api/vault/`)
- **Data model:** `transfers/vault/{vault_id}/manifest.json` + `transfers/vault/{vault_id}/{file_id}/payload`
- **Auth:** access token + write key (`x-sgraph-vault-write-key`)
- **Mutability:** full — write, overwrite, delete, batch, CAS, list
- **Multiple files:** arbitrary file tree under a vault_id

### Simple Token Flow
- Client derives a deterministic `transfer_id` from a passphrase using PBKDF2
- The `transfer_id` is passed to `POST /api/transfers/create` as `transfer_id` field
- This gives a stable, human-memorable URL: anyone with the passphrase can derive the transfer_id and download

### Vault ↔ Simple Token (CLI already does this)
- The CLI can **create a vault from a simple token** file and **push a vault to a simple token** file
- This is a client-side operation — the server has no awareness of the relationship

## Design Options

### Option A: Add Update Endpoint to Transfer System

**Concept:** Add `POST /api/transfers/update/{transfer_id}` that overwrites the payload of a completed transfer.

**API Surface:**
```
POST /api/transfers/update/{transfer_id}
  Headers: x-sgraph-send-access-token
  Body: encrypted payload bytes
  → 200 { transfer_id, status: "updated", size }
```

**Data Model Changes:**
- `Transfer__Service.update_payload()` — allows overwrite when `status == 'completed'`
- No new storage paths — still `transfers/{transfer_id}/payload`
- Add `updated_at` and `update_count` to meta.json
- Add `update` event to events list

**Auth Model:**
- Same access token as the original upload — **no additional write key**
- Any valid access token can update any transfer (same as current create/upload)
- This is the simplest model but means transfers are **not owner-protected** — anyone with an access token and the transfer_id can overwrite

**Complexity:** Low — ~30 lines of service code, one new route

**Use Case Fit:**
| Use Case | Fit | Notes |
|---|---|---|
| Folder + add files | Partial | Single payload only — would need to re-upload entire zip/bundle each time |
| Vault snapshot → stable URL | Good | CLI pushes vault as single blob to transfer, overwrites on each publish |
| Zip separate uploads | Poor | Still a single payload slot — can't do individual file entries |

**Pros:**
- Minimal change to existing system
- No new auth model needed
- Simple token URLs remain stable (same transfer_id, updated payload)

**Cons:**
- No write-key protection (any token holder can overwrite any transfer they know the ID of)
- Single payload — can't represent multi-file structures natively
- No versioning — overwrite is destructive

### Option B: Transfer-as-Vault-Alias (Pointer Model)

**Concept:** A simple token transfer becomes a **pointer** to an underlying vault. The transfer stores a vault reference instead of (or in addition to) a raw payload. Downloads resolve through the vault.

**API Surface:**
```
POST /api/transfers/create
  Body: { ..., vault_ref: { vault_id, snapshot: "latest" | "frozen" } }
  → 200 { transfer_id, upload_url }

GET /api/transfers/download/{transfer_id}
  → (internally resolves vault_ref → reads from vault → returns payload)
```

No new endpoints — existing download resolves transparently.

**Data Model Changes:**
- `meta.json` gains optional `vault_ref: { vault_id, file_id?, snapshot_mode }` field
- When `vault_ref` is present, download reads from the vault instead of `transfers/{transfer_id}/payload`
- `snapshot_mode`:
  - `"latest"` — always reads current vault state (live link)
  - `"frozen"` — takes a snapshot at publish time (copies payload into transfer storage)

**Auth Model:**
- Transfer download: no auth (same as today — data is encrypted)
- Vault updates: vault write-key auth (existing)
- Creating the link: access token + vault write-key (proves you own the vault)

**Complexity:** Medium — new resolution logic in `get_download_payload()`, vault_ref in meta.json

**Use Case Fit:**
| Use Case | Fit | Notes |
|---|---|---|
| Folder + add files | Good (with latest mode) | Vault holds multi-file structure; transfer is the stable read-only URL |
| Vault snapshot → stable URL | Excellent | This is exactly the model — vault is source of truth, transfer is the published view |
| Zip separate uploads | Good | Vault already supports multi-file; transfer resolves to vault zip |

**Pros:**
- Clean separation: vault is mutable source, transfer is immutable-looking stable URL
- Leverages existing vault auth (write key) for updates
- Download URL never changes — only the backing data does
- Multi-file support comes free from vaults
- Read-only by design — the simple token URL has no write capability

**Cons:**
- Transfer download now has two code paths (direct payload vs vault resolution)
- Vault must exist as long as the transfer is live
- "Latest" mode means downloads are non-deterministic (may return different data each time)
- More conceptual complexity for the CLI team

### Option C: Versioned Transfers

**Concept:** A transfer_id can hold multiple payload versions. Upload creates v1, update creates v2. Download gets latest by default, or a specific version.

**API Surface:**
```
POST /api/transfers/upload/{transfer_id}
  → works on pending (v1) AND completed (creates v2, v3, ...)

GET /api/transfers/download/{transfer_id}
  → returns latest version

GET /api/transfers/download/{transfer_id}?version=2
  → returns specific version
```

**Data Model Changes:**
- Storage: `transfers/{transfer_id}/payload/v1`, `transfers/{transfer_id}/payload/v2`, ...
- `meta.json` gains `current_version: int` and `versions: [...]`
- Existing download path (`transfers/{transfer_id}/payload`) needs migration or compat shim

**Auth Model:**
- Same as Option A — access token only, no write-key
- Or: add an optional `update_key` to meta.json (set on create, required for subsequent uploads)

**Complexity:** Medium-High — version management, storage path changes, migration for existing transfers

**Use Case Fit:**
| Use Case | Fit | Notes |
|---|---|---|
| Folder + add files | Partial | Still a single payload per version — need to re-upload entire bundle |
| Vault snapshot → stable URL | Good | Each publish creates a new version; download gets latest |
| Zip separate uploads | Poor | Versions are whole payloads, not individual files |

**Pros:**
- Audit trail — previous versions preserved
- Rollback possible
- Download URL stable

**Cons:**
- Storage bloat (every update stores a full copy)
- Breaking change to storage paths (migration needed)
- No multi-file support within a version
- Over-engineered for the primary use case

---

### Option D: Vault Publish Endpoint (Recommended)

**Concept:** Vaults get a **publish** operation that snapshots or links vault content into a transfer. The simple token is the distribution URL; the vault is the source of truth. Server-side operation — no client-side coordination needed.

**API Surface:**
```
POST /api/vault/publish/{vault_id}
  Headers: x-sgraph-send-access-token, x-sgraph-vault-write-key
  Body: { transfer_id?, mode: "link" | "snapshot", file_id? }
  → 200 { transfer_id, download_url, mode }
```

- `transfer_id` optional — if provided, uses that (simple token PBKDF2 ID). If omitted, auto-generates.
- `mode: "link"` — transfer download resolves to vault at read time (always latest)
- `mode: "snapshot"` — copies current vault state into transfer payload (frozen at publish time)
- `file_id` optional — publish a specific file from the vault, or the entire vault (as zip)

**Data Model Changes:**
- `meta.json` gains optional `vault_ref` (same as Option B) when mode is "link"
- For "snapshot" mode: vault content is copied to `transfers/{transfer_id}/payload` — no ongoing dependency
- Transfer service needs resolution logic for `vault_ref` (same as Option B)

**Auth Model:**
- Publish requires: access token + vault write key (proves vault ownership)
- Download: no auth (same as today)
- Updates to the vault automatically update the linked transfer (link mode)
- Re-publish for snapshot mode (explicit action)

**Complexity:** Medium — new publish route, vault_ref resolution in download, but well-bounded

**Use Case Fit:**
| Use Case | Fit | Notes |
|---|---|---|
| Folder + add files | Good | Work in vault (multi-file), publish to transfer when ready |
| Vault snapshot → stable URL | Excellent | Primary design target — publish links vault to stable simple token |
| Zip separate uploads | Good | Vault holds individual files; publish as zip to transfer |

**Pros:**
- Clean mental model: vault = workspace, transfer = published artifact
- Leverages existing vault infrastructure (auth, multi-file, batch)
- Two modes cover different needs: "link" for live, "snapshot" for frozen
- Simple token URL is stable and read-only
- Server-side publish — CLI doesn't need to download-and-reupload
- Existing transfers are completely unaffected (no migration)

**Cons:**
- New endpoint + resolution logic
- "Link" mode has same non-determinism concern as Option B
- Need to handle vault deletion (what happens to linked transfers?)

## Comparison Matrix

| Criteria | A: Update Endpoint | B: Transfer-as-Alias | C: Versioned | D: Vault Publish |
|---|---|---|---|---|
| **Complexity** | Low | Medium | Medium-High | Medium |
| **Multi-file** | No | Yes (via vault) | No | Yes (via vault) |
| **Auth for updates** | Weak (any token) | Strong (write key) | Weak or Medium | Strong (write key) |
| **Existing transfer impact** | None | None | Migration needed | None |
| **Vault snapshot → URL** | Manual (CLI) | Built-in | Manual (CLI) | Built-in |
| **Zip use case** | Poor | Good | Poor | Good |
| **Server-side publish** | No | No | No | Yes |
| **Read-only guarantee** | No (updatable) | Yes | No (updatable) | Yes |

## Recommendation

**Option D (Vault Publish)** — with Option A as a quick tactical addition if needed.

**Rationale:**
1. **Option D directly addresses the primary use case** — vault owner publishes a read-only view to a stable simple token URL. This is what the CLI team is already doing manually (download from vault, upload to transfer).
2. **Server-side publish eliminates round-trips** — instead of CLI downloading vault content and re-uploading to transfer, the server copies or links directly.
3. **Multi-file support is free** — vaults already handle file trees. Publish can produce a zip or link individual files.
4. **Auth is correct** — publish requires write-key proof of vault ownership. The published transfer URL is read-only. No risk of unauthorized overwrites.
5. **No migration** — existing transfers are completely unchanged.

**Option A is worth considering as a quick tactical win** for the simplest case: "I have a transfer and I want to update its payload." It's ~30 lines of code and doesn't conflict with Option D. But it doesn't solve the multi-file or auth problems.

**Options B and C are not recommended.** B is essentially D minus the server-side publish (client has to coordinate). C adds complexity (versioning, migration) without solving multi-file.

## Key Design Questions (For Human Decision)

1. **Link vs Snapshot default?** Should `POST /api/vault/publish/{vault_id}` default to `"link"` (always-latest) or `"snapshot"` (frozen at publish time)? Link is more useful for the "stable URL" use case; snapshot is safer for auditing.

2. **Do we also want Option A?** A simple `POST /api/transfers/update/{transfer_id}` is ~30 lines and useful for non-vault scenarios (e.g., update a standalone simple-token file without creating a vault). Should we add it alongside Option D?

3. **Vault deletion policy:** If a vault is deleted and a transfer links to it, should the download return 404, or should we prevent vault deletion when linked transfers exist?

4. **Zip vs single-file publish:** When publishing a multi-file vault, should the transfer payload be a zip, a specific file_id, or a manifest listing individual download URLs?

5. **Auth for simple update (if Option A):** Should a simple transfer update require just an access token (current model), or should we add an `update_key` (set at creation time, required for updates)?

6. **Should the CLI team consume publish server-side?** Currently the CLI does vault→transfer manually. Do they want a server endpoint, or is this better as a CLI-side operation that calls existing APIs?

### Download Resolution (Link Mode)
```python
def get_download_payload(self, transfer_id, ...):
    meta = self.load_meta(transfer_id)
    if meta.get('vault_ref'):                          # Linked vault
        vault_id = meta['vault_ref']['vault_id']
        file_id  = meta['vault_ref'].get('file_id')
        if file_id:
            return self.vault_service.read(vault_id, file_id)
        else:
            return self.vault_zip_service.build_zip(vault_id)
    # ... existing direct-payload logic
```
