## Executive Summary

12 March is the day the vault architecture locked. Through six revisions of an architecture simulation -- executed entirely on paper, with zero lines of production code -- the branch model was stress-tested across four scenarios (solo, two-user collaboration, conflict, and keyless change packs). Fourteen gaps were found; thirteen were resolved in the same session. The one remaining gap (batch endpoint spec) is a formal API design task, not an architectural uncertainty.

The simulation produced a 1014-line document that predicts exact files, keys, objects, refs, and API calls at every step. This is not a design sketch. It is a test oracle. The QA predictions table at the end is directly translatable into assertions.

Alongside the simulation, three architecture briefs define the next three months of vault work: bare vault unification, branch model with PKI, and CLI with remotes. Five product briefs extend the platform surface (gallery, LLM image generation, vault download endpoint, billing, SPKI brand). Two process briefs (privacy policy and Explorer-to-Villager methodology) require zero architecture changes. One P0 bug (5MB upload failure) requires immediate attention.

The strategic significance: the vault is four features away from "lightweight encrypted Git" (branches, merge, signed commits, remotes). The simulation confirms the architecture supports all four. The build order is clear. The test oracle exists. Implementation can begin on solid ground.

## Simulation v6 Verdict

### Methodology Assessment

The case study (`vault/v0.13.30__case-study__simulation-driven-architecture.md`) proposes simulation-driven architecture as standard practice for all significant architecture changes. **The Architect agrees.** The methodology demonstrated its value concretely:

- **Completeness forcing.** A spec can say "the system merges branches." A simulation must answer: which branch merges into which, who signs the merge commit, what are the parent references, what happens to the refs. Vagueness becomes a runtime error in simulation, surfaced before any code exists.
- **State tracking beyond human capacity.** By Scenario 2, Step 2.4, the simulation was tracking 3 key pairs across 2 devices, 2 diverged branch histories, 17+ objects, and the merge resolution of all of them. No whiteboard could hold this.
- **Gap compounding prevention.** Gap 5 (conflicts on named branch vs. clone branch) would have required a design-level rewrite if discovered during implementation. In simulation, it was a paragraph of conversation.

**Recommendation:** Adopt simulation as a mandatory step for any architecture change that introduces new state transitions or new object types. The threshold: if the change adds a new kind of file to the vault, a new ref type, or a new merge direction, it gets simulated first.

### The Open Gap: Batch Endpoint Spec

Gap 6 remains open. The simulation established that a batch endpoint (`POST /api/vault/batch/{vault_id}`) reduces push from 6+ round trips to 2. What is needed:

- Request payload format (multipart? JSON envelope with base64 blobs? streaming?)
- Atomicity semantics (all-or-nothing? partial success with rollback list?)
- Partial failure handling (which files succeeded, which failed, retry semantics)
- `write-if-match` semantics for ref updates (CAS on the ref value)
- Size limits (max batch size, max individual blob size)
- Error response format

This is a DevOps + Architect collaboration. The simulation assumes the batch endpoint exists. The formal spec must be written before implementation begins.

### Key Architecture Decisions Validated

| Decision | Rationale | Simulation Evidence |
|---|---|---|
| `bare/` as portable vault unit | Same structure on server, local disk, and zip. Delete either copy, the other is a complete backup. | Scenario 1.3: 16 files pushed; server and local `bare/` are identical. Scenario 2.1: clone downloads `bare/`, adds 3 files, both copies remain symmetric. |
| Zero-knowledge branch naming | Opaque IDs (`branch-named-x7k2`, `ref-a3f8`) with encrypted index. Server sees folder counts only. | Server view after Scenario 2: "28 opaque blobs in 6 folders. Zero content knowledge." File paths leak nothing -- no branch names, no key ownership, no object types. |
| Conflicts on clone branch, not named branch | Named branch stays clean. Other users unaffected by one user's conflict. Enables `--branch-only` push and `merge --abort`. | Scenario 3.3: conflict creates `.conflict` file on clone branch. `ref-a3f8` (named branch) is untouched. Three resolution options available without blocking other users. |
| Three-layer model (remote / local-current / clone-branch) | Clean separation of authority (remote), fast-forward tracking (local current), and work (clone branch). | Push workflow Steps 2-7: fetch remote, fast-forward local current, merge current into clone (conflicts here), merge clone into current (always clean), push. Each layer has exactly one responsibility. |
| Uint timestamps everywhere | `Timestamp_Now` from osbot-utils. Compact, sortable, timezone-free. No human-readable temporal metadata leaked to server. | Pending files use `pending-{uint_ts}_{random}`. Commit timestamps are uint milliseconds. No ISO strings anywhere in the vault. |

### QA Predictions as Test Oracle

The predictions table (Simulation v6, final section) provides exact file counts at every step:

| Step | Encrypted files (bare/) | Working dir files | Server files |
|------|------------------------|-------------------|-------------|
| Init | 10 | 0 | 0 |
| Commit | 15 | 2 | 0 |
| Push | 16 | 2 | 16 |
| Clone | 19 | 2 | 19 |
| Both commit | D1:19, D2:23 | 2 each | 19 |
| A pushes | D1:21 | 2 | 21 |
| B pushes | D2:28 | 2 | 28 |
| Conflict | +3 on clone | 3 (.conflict) | unchanged |

**Ready for implementation?** Yes, with one caveat: the predictions assume content-addressed hashing produces unique IDs for every object. The implementation must use the same hashing scheme the simulation assumes (SHA-256 of encrypted content) or the counts will diverge. The simulation uses placeholder hashes (`obj-h_blob_rm01`) -- the implementation must document the actual hash function and confirm the counts still hold.

## Architecture Brief Dependency Chain

The three architecture briefs plus the simulation form a strict dependency chain:

```
  1. BARE VAULT UNIFICATION (Brief #5)
     ├── Unified format for server-side and clone-side
     ├── Security audit: no unencrypted sensitive data
     ├── Clone = zip download (no transformation)
     ├── Version field added to vault format
     └── Foundation for everything below
         │
  2. BRANCH MODEL + PKI (Brief #6)
     │   ├── Auto-branch on clone with key pair
     │   ├── Signed commits (branch key signs)
     │   ├── Merge-only named branch
     │   ├── Conflict resolution on clone branch
     │   └── Builds on unified bare/ structure
     │       │
     │   SIMULATION v6 (test oracle for branch model)
     │       ├── Validates every state transition
     │       ├── QA predictions table = acceptance tests
     │       └── 13/14 gaps resolved
     │           │
  3. CLI REMOTES (Brief #4)
         ├── Pluggable backend abstraction
         ├── Git-model remotes (origin, backup, S3, local, URL, zip)
         ├── Full offline mode
         └── Builds on branch model (push/pull/clone are branch-aware)
```

**Why this order is mandatory:**

- Bare vault unification MUST come first because the branch model assumes `bare/` is the canonical structure. If server-side and clone-side formats differ, every branch operation needs format translation -- exactly the complexity the unification eliminates.
- Branch model MUST precede CLI remotes because push, pull, and clone are branch-aware operations. Implementing remotes without branches means building them twice: once without branches, once with. The branch model defines the push workflow (the 7-step process in the simulation), and remotes implement the transport layer for that workflow.
- The simulation is not a separate deliverable -- it is the test oracle for the branch model. Implementation of Brief #6 should be validated step-by-step against the simulation predictions.

**Estimated timeline alignment:** This matches the 8-9 week assessment from the Explorer team's earlier review. Bare vault unification (2-3 weeks), branch model (3-4 weeks), CLI remotes (2-3 weeks), with overlap possible between branch model testing and remotes scaffolding.

## Product Briefs -- Architecture Implications

### SPKI Brand Strategy (Brief #1)

**Architecture impact: None immediate.**

The sub-domain structure (`send.spki.ai`, `vault.spki.ai`, `tools.spki.ai`, `library.spki.ai`) is validated as technically feasible -- it follows the same pattern as the existing `send.sgraph.ai` / `dev.send.sgraph.ai` / `gallery.sgraph.ai` structure. DNS configuration and CloudFront distribution updates are DevOps tasks, not architecture changes.

The transition is explicitly organic with no deadlines. No code changes required until Phase 2 (when new sub-domains stand up alongside existing ones). At that point, the only architecture consideration is ensuring Lambda URL functions and CORS configurations support the new domain names.

**Action item:** None for Architect. DevOps should note the eventual DNS/CORS update.

### Vault Download Endpoint (Brief #2)

**Architecture impact: New API endpoint required.**

The endpoint `GET /api/vault/download/{vault_id}` (or with download token) returns the entire `bare/` directory as a single downloadable bundle (zip or tar). This ties directly into the bare vault unification work:

- If `bare/` IS the vault (as the simulation establishes), then the download endpoint is: zip `bare/`, serve it. No transformation.
- The download token model needs design: is it a one-time token? Time-limited? Tied to a specific vault version (commit)?
- The endpoint belongs on the User Lambda (public-facing), not the Admin Lambda.

**Dependency:** Bare vault unification must complete first. Once `bare/` is the canonical format, the download endpoint is straightforward. Without unification, the endpoint would need to assemble the download from the server-side format, adding unnecessary complexity.

**Connection to CLI remotes:** The download URL is one of the remote types (`sg-send-cli clone https://send.sgraph.ai/api/vault/download/{vault_id}`). The CLI remotes brief lists URL as a remote type. The download endpoint is the server-side implementation of that remote type.

### Vault Website and Billing (Brief #3)

**Architecture impact: Billing integration, separate repo.**

The vault website moving to a separate Git repo is a deployment decision, not an architecture change. The billing units (vault count, pulls, pushes, commits, branches, storage) align with the branch model -- every billable operation maps to a vault API call that already exists or will exist after the branch model ships.

**Architecture consideration for billing:** Usage tracking via S3 server access logs is the lowest-friction starting point, as the brief recommends. However, the branch model adds new billable operations (branch creation, merge) that S3 logs alone cannot capture. Application-level logging in the vault API will eventually be needed. The Architect recommends designing the billing event model now (a `Type_Safe` schema for billing events) even if the initial implementation only uses S3 logs. This prevents a billing redesign when branch operations launch.

### Gallery UX (Brief #8) and Portugal Strategy (Brief #7)

**Architecture impact: Presentation layer over vault API.**

The gallery is architecturally clean: it is a vault with a `_gallery.json` manifest, displayed through a presentation layer at `gallery.sgraph.ai`. No new storage primitives. No new encryption patterns. The gallery API reads from the existing vault storage.

Key architecture notes:

- **Client-side thumbnail generation.** Thumbnails are generated in the browser and stored as separate encrypted blobs. This follows the existing pattern (encrypt in browser, server stores ciphertext). No server-side image processing. No server access to unencrypted images.
- **`_gallery.json` manifest.** Same pattern as `_pack.json` from the vault UI components work. The manifest is an encrypted blob in the vault that describes the gallery layout, title, and file references. The server does not read the manifest -- the browser decrypts it and renders the gallery.
- **Public gallery URL structure** (`gallery.sgraph.ai/view/{title-slug}/{decryption-hint}`) embeds the key in the URL. This is the same pattern as existing SG/Send transfer links (key in URL fragment). No architecture change.

The Portugal strategy (Brief #7) adds a public key registry (`registry.sgraph.ai`). This is a new service that scrapes and indexes public keys. Architecturally it is independent -- it does not touch the vault or encryption pipeline. It reads from external sources and writes to its own data store. The "Send secure message" button on each registry entry uses the existing SG/Send encryption flow.

**Action item:** Define the `_gallery.json` schema as a `Type_Safe` class. Ensure the gallery API is a read-only presentation layer with no write access to vault internals.

### LLM Image Generation (Brief #11)

**Architecture impact: Encrypt-for-reader pattern for API keys, billing integration.**

The brief proposes encrypting OpenRouter/OpenAI API keys with the user's public key and storing them in the user's vault. This is a concrete application of the "encrypt-for-reader" paradigm identified in v0.13.29. The architecture implications:

- **Key lifecycle:** SG/Send creates an API key programmatically, encrypts it with the user's public key, stores it in their vault. The user's browser decrypts the key and calls the LLM API directly. SG/Send never retains the decrypted API key after the encryption step.
- **Spending cap enforcement:** The spending cap is set on the API key itself (at the OpenRouter/OpenAI level), not enforced by SG/Send. SG/Send's role is key creation and encrypted delivery. This is architecturally clean -- no proxy, no man-in-the-middle.
- **Billing integration:** LLM operations are a new billing event type. The credit model applies: user pays credits, SG/Send pays wholesale API rates, margin is the markup. This requires a billing event for "LLM operation" with cost, model, and user fields.

**Dependency on branch model:** The encrypt-for-reader pattern requires the user to have a public key. In the branch model, every clone creates a key pair. The user's public key is in the vault. This means LLM image generation with secure API key delivery depends on the branch model shipping first (or at minimum, on a simplified key-pair-per-user mechanism being available).

### Privacy Policy (Brief #10)

**Architecture impact: Zero.**

The six-sentence privacy policy describes what the architecture already enforces. "We cannot read your files" is a statement about AES-256-GCM encryption with client-side key management. "Zero cookies" is a statement about the existing frontend implementation. "We do not track you" is a statement about the absence of third-party scripts.

The policy is a marketing document that happens to be an accurate description of the architecture. No code changes, no infrastructure changes, no new endpoints. The Architect's only note: the comparison table ("Most cloud services" vs. "SG/Send") should be reviewed by AppSec to ensure every claim is verifiable.

## P0 Bug: 5MB Upload Failure

The bug: an encrypted payload that exceeds the Lambda function's payload size limit (~6MB) fails silently. The chunked upload path exists but the client-side threshold check does not trigger correctly at the boundary.

**Architecture assessment:**

The fix is client-side validation. Before calling the upload API, the browser must:

1. Calculate the encrypted payload size (plaintext size + AES-GCM overhead + base64 encoding overhead).
2. Compare against the Lambda payload limit (6MB).
3. If exceeded, route through the chunked upload path automatically.

This is not an architecture change. It is a missing boundary check in the existing upload flow. The chunked upload infrastructure already works (adding one more file to the folder pushed it past the threshold and the chunked path handled it correctly).

**Recommended fix:** Add a constant `LAMBDA_PAYLOAD_LIMIT_BYTES = 5_500_000` (with margin below the 6MB hard limit) to the client-side upload logic. Check `encrypted_payload.length > LAMBDA_PAYLOAD_LIMIT_BYTES` before the upload call. If true, use chunked upload. This should be a one-line conditional with zero risk to the rest of the upload flow.

**Priority confirmation:** P0 is correct. Silent failure on a common file size erodes user trust. The fix is small, well-scoped, and independent of all other work streams.

## Explorer-to-Villager: Architecture View

The code quality methodology (Brief #9) defines a four-phase cycle: review, test, refactor, build. The CLI is the first candidate. The Architect's role is Phase 1 (architectural review).

**Key observation:** The simulation effectively IS Phase 1 for the branch model. The simulation reviewed the architecture through execution, found 14 gaps, and resolved 13. This is a more thorough architectural review than a static code review could provide, because the architecture was tested against concrete scenarios.

The implication: for the CLI's branch model work, Phase 1 is already complete. The simulation document is the architectural review. The team can proceed directly to Phase 2 (test coverage for existing CLI code) and Phase 3 (refactoring to clean abstractions for remote/branch/merge additions).

**Phase 2 note for the branch model:** The simulation's QA predictions table provides the test specifications. The test suite for the branch model should include:

- File count assertions at every step (init: 10 encrypted, 0 working dir, 0 server)
- Key pair count assertions (init: 2 pairs; after clone: 3 pairs)
- Ref value assertions (after commit: clone ref advanced, current ref unchanged)
- Signature verification (every commit signed by the correct branch key)
- Conflict location assertions (conflict files appear on clone branch, named branch untouched)
- Object graph assertions (parent references form the expected DAG)

The adversarial testing exercise from Phase 3 maps naturally to the simulation: change a merge direction, change a signing key, change a ref update -- does the test catch it? The simulation specifies exactly what should happen. Any deviation is a bug.
