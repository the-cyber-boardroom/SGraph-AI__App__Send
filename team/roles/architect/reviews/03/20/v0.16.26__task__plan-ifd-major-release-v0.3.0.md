## Context

The IFD (Iterative Feature Development) methodology works in two phases:

1. **Minor versions (v0.2.x):** Incremental overlays — each version patches the previous via prototype mutation. This is fast for exploration but accumulates complexity.
2. **Major version (v0.3.0):** Consolidation — all v0.2.x overlays are merged into a single, clean, self-contained codebase. No more overlay chain.

We are now at v0.2.14 with overlays from v0.2.0 through v0.2.14. It's time to plan the consolidation into v0.3.0.

## What Consolidation Means

- **v0.3.0 is a fresh, standalone version** — one file per component (not 14+ overlay files)
- All behaviour from v0.2.0–v0.2.14 is present, but written as clean, direct code
- No prototype patching chains, no `_v0XXX_` prefixed methods
- No backwards compatibility shims — just the final working behaviour
- The v0.2.x folder remains as-is (frozen, read-only archive)
- v0.3.0 lives in its own version folder: `v0/v0.3/v0.3.0/`

## IFD Cycle

Once v0.3.0 is released:
- v0.3.0 becomes the new base
- Exploration continues with v0.3.1, v0.3.2, etc. (new overlays on v0.3.0)
- When v0.3.x is mature, consolidate again into v0.4.0
- This is the IFD rhythm: explore → consolidate → explore → consolidate
