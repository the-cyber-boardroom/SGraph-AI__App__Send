## 2. Service Layer Analysis

### What's good

- **Single class, clear responsibility.** `Service__Vault__Pointer` is 204 lines doing exactly one thing: opaque blob CRUD with write-key auth. No inheritance hierarchy, no mixins, no surprises.
- **Manifest-as-sole-auth-record** is a clean pattern. One JSON file per vault holds the `write_key_hash` and `created_at`. All write operations check against it. Simple, auditable.
- **Storage abstraction works.** The service talks only to `Storage_FS`. It genuinely doesn't know if it's memory, disk, or S3 behind the interface.
- **CAS (compare-and-swap) via `write_if_match`** is well-designed for the git-like ref-update pattern. Conflict returns current value for retry — correct semantics.
- **Batch stops on conflict** — this is the right call for git push semantics. Blobs are written first, the ref CAS is last, and a conflict means "someone else pushed first."

### Concerns

**A. Dual CAS implementations.** There are two implementations of compare-and-swap:

1. `write_if_match()` (lines 83-104) — standalone method used by the individual route
2. `_cas_write()` (lines 192-203) — internal method used by `batch()`

They implement the same logic but have subtle differences:
- `write_if_match` calls `_check_vault_write_key` and `_ensure_manifest` itself
- `_cas_write` assumes auth is already validated (called from within `batch`)
- `write_if_match` returns `file_id` in result; `_cas_write` returns `op` + `file_id`

**Recommendation:** Extract the core CAS logic into one place. Have both `write_if_match` and `batch` delegate to it. The current duplication risks drift.

**B. `batch()` has auth but `batch_read()` doesn't — and `batch()` also handles reads.** The route layer decides which service method to call (line 153-156 of routes). But the service layer also has `read` ops inside `batch()` (line 143-144), which requires auth. This means:
- A read-only batch goes through `batch_read()` — no auth
- A mixed batch with reads + writes goes through `batch()` — full auth for all ops, including reads

This is correct behaviour, but it's architecturally fragile because the auth decision lives in the **route layer**, while the operation dispatch lives in the **service layer**. If someone adds a new caller of `batch()`, they need to remember the auth split.

**C. `list_files()` scans all paths.** Line 108: `all_paths = self.storage_fs.files__paths()` returns every path in the storage backend, then filters client-side. For a system with thousands of vaults, this is O(total files across all vaults) for every list call.

- Not a problem today (Explorer territory, in-memory storage)
- Will be a problem at scale — this should eventually use a prefix-scoped listing from the storage backend
- **Tag for Villager:** When this moves to S3, use `list_objects_v2(Prefix=...)` instead of full scan

**D. No vault deletion endpoint.** You can delete individual files, but you can't delete a vault (manifest + all files). This is fine for now but worth noting — a vault that has all its files deleted still has a dangling manifest.

## 3. Data Flow

```
Client request
  → Routes__Vault__Pointer (auth check, input validation, HTTP concerns)
    → Service__Vault__Pointer (business logic, write-key verification)
      → Storage_FS (opaque file storage)
        → Storage_FS__Memory | Storage_FS__Disk | Storage_FS__S3
```

**Storage path convention:**
```
transfers/vault/{vault_id}/manifest.json          # Auth record (write_key_hash, created_at)
transfers/vault/{vault_id}/{file_id}/payload       # File content (encrypted ciphertext)
```

**Observations:**

- The `/payload` suffix on every file means a `file_id` of `bare/data/obj-abc` becomes the path `transfers/vault/{vault_id}/bare/data/obj-abc/payload`. That's 5 levels deep. Works with memory-FS but worth checking that S3 doesn't have key-length or prefix-depth issues.
- The `file_id` is fully client-controlled (any string matching `{file_id:path}`). There's no validation regex. The service trusts `Storage_FS` to sandbox paths. This is probably fine but leaves no defence-in-depth against path traversal if Storage_FS has bugs.

## 4. Auth Model

### Two-layer auth

| Layer | What | Where |
|-------|------|-------|
| **Access token** | System-level gate (is this a valid caller?) | Route layer: `check_access_token()` |
| **Write key** | Vault-level ownership (does this caller own this vault?) | Service layer: `_check_vault_write_key()` |

### Auth matrix

| Operation | Access token | Write key | Notes |
|-----------|-------------|-----------|-------|
| Write | Required | Required | Both layers |
| Read | **Not required** | **Not required** | Zero-knowledge: data is encrypted |
| Read-base64 | **Not required** | **Not required** | Same as read |
| Delete | Required | Required | Both layers |
| Batch (read-only) | **Not required** | **Not required** | Route layer decides |
| Batch (mixed/write) | Required | Required | Both layers |
| List | **Not required** | **Not required** | Lists file_ids only (encrypted) |
| Health | **Not required** | **Not required** | Existence check only |

### Discussion points

**Is unauthenticated `list` the right call?** The list endpoint reveals the file_id structure (`bare/data/obj-abc`, `bare/refs/ref-main`). These are opaque IDs, but they do reveal:
- How many files are in a vault
- The namespace structure (data vs refs vs keys)
- When files are added (if IDs contain timestamps)

For a zero-knowledge system, this is metadata leakage. The data is encrypted, but the structure isn't. Worth discussing whether `list` should require at least read-level auth.

**First-write-wins for vault creation.** Anyone can create a vault by writing to a new `vault_id` (line 36: `return True` — no manifest yet means first write succeeds). The vault_id is 8 chars (`Safe_Str__Id`). This means:
- Vault squatting is theoretically possible (pre-create manifests to block legitimate users)
- In practice, vault_ids are generated client-side from crypto-random values, so collisions are near-zero
- But there's no formal "create vault" ceremony — creation is implicit on first write

## 5. Naming & Conventions

**Consistent with codebase patterns:**
- `Type_Safe` base class with aligned column formatting
- `Storage_FS` for storage, no direct filesystem calls
- Dict return values (not Pydantic/dataclass)
- Test naming: `test__{method}__{scenario}`

**Minor inconsistency:** The route class is `Routes__Vault__Pointer` but the ACL test file references `Routes__Vault__ACL` which is a different, skipped test class for an older vault model (share/unshare/permissions). The ACL tests are `@unittest.skip` with a note about re-enabling when vault moves to admin Lambda. This is dead code waiting to be cleaned up or revived.

## 7. Architectural Opinions

### What's working well

1. **The single-service-class approach is appropriate for this stage.** The vault is doing one thing (blob storage with write-key auth). Splitting it prematurely would add indirection without benefit.
2. **The manifest pattern is elegant.** One JSON file per vault, created on first write, checked on every subsequent write. Simple, debuggable, no database needed.
3. **The batch endpoint is the right primitive.** It collapses N round trips into 1 for push/pull. The CAS-on-conflict-stops behaviour matches git semantics.
4. **The auth split (route vs service) mostly works.** Routes handle system-level auth (access tokens), service handles domain-level auth (write keys). Clean separation.

### What needs attention

1. **The health endpoint reaches through the service into storage** (route line 180-181: `self.vault_service.storage_fs.file__exists(manifest_path)`). This breaks the service abstraction — the route should call a service method like `vault_exists(vault_id)`, not directly access storage.
2. **The `check_access_token` method is doing too much.** It has three paths: admin service validation, env-var fallback, and no-token-configured bypass. This belongs in middleware or a shared auth module, not in a vault route class. Every new route class will need to duplicate this.
3. **The ACL test file is dead weight.** `test_Routes__Vault__ACL.py` is entirely skipped. Either delete it or move it to a `_legacy/` folder. Skipped tests rot.

## 8. Questions for Discussion

### Design direction

1. **Should vault become its own Lambda?** Right now vault endpoints live in the User Lambda alongside transfer endpoints. The vault has different auth patterns (write-key vs transfer tokens), different access patterns (many small reads vs single large file), and will likely have different scaling characteristics. Is it time to split?

2. **Should `list` be authenticated?** Currently unauthenticated, consistent with `read`. But `list` leaks structure metadata that `read` doesn't (you need the exact file_id to read, but list gives you all file_ids). For a zero-knowledge system, this metadata exposure might matter.

3. **What's the vault lifecycle?** Currently vaults are created implicitly on first write and can never be fully deleted (manifest persists). Should there be an explicit `create` and `destroy`? What about TTL/expiry?

4. **Is the `/payload` suffix on storage paths necessary?** Every file_id stores data at `{file_id}/payload`. This was useful when there was a `meta.json` sibling, but meta was removed. Now it's an extra path segment with no purpose. Could simplify to `transfers/vault/{vault_id}/{file_id}` directly.

5. **Base64 encoding in batch responses** — batch read returns all data base64-encoded inside JSON. For a vault with many large blobs, this hits Lambda's 6MB response limit quickly (base64 adds 33%). Should there be a streaming/chunked alternative, or is batch read specifically for small objects (refs, trees, commits)?

### Near-term

6. **The `check_access_token` method.** Is there a plan to extract this to shared middleware? It's currently copy-paste-ready for every new route class.

7. **Should the skipped ACL tests be deleted?** They reference a vault model (create/share/unshare/permissions) that no longer exists in the Pointer architecture. Are they aspirational or obsolete?
