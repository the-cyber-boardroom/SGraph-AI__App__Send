## Key Architectural Decisions

### 1. Agent Identity Through Cryptography — SOUND

**Source:** Documents 56, 58, 59

The core innovation: agents get their own cryptographic identity (key pair per clone branch), not user credentials. This was **validated in practice** — Claude built a 62-test Python CLI tool through vault commits.

**Assessment:** This is the right model. It solves:
- Credential leakage (agent has its own key, not yours)
- Over-privileged access (agent can only write to its branch)
- Attribution (commits cryptographically signed)
- Revocation (don't accept that key, not rotate your token)

**Concern:** The case study (document 59) claims "6 layers of security" but layers 5–6 (zero-knowledge server, file-level PKI) are not yet enforced end-to-end. The current implementation has layers 1–4. This is fine for now but should be clearly documented.

### 2. Git-to-SGit Bridge Architecture — SOUND WITH CONCERNS

**Source:** Document 56

**Assessment:** The flow (Git→export→SGit vault→agent→import→Git PR) is architecturally sound. Mode 1 (changed files only) is the right default.

**Concerns:**
- **Manifest format not versioned** — `manifest.json` in `_sync/` has no `schema_version` field. This will break when the format evolves.
- **Lazy clone complexity** — "download on demand" requires the agent to know which files it needs before seeing the codebase. This works for scoped tasks but fails for refactoring or cross-cutting changes.
- **Merge conflict resolution** — the brief doesn't address what happens when the Git branch moves forward while the agent is working. The import step needs a conflict strategy.

### 3. Read-Structure Key — EXCELLENT ARCHITECTURAL ADDITION

**Source:** Document 72

A new key tier derived via HKDF: `vault_key → { read_key (content), structure_key (metadata) }`. Enables diagnostics, monitoring, and auditing without content access.

**Assessment:** This is the right approach. It enables:
- SRE health checks (is the vault consistent?)
- Audit trail inspection (who committed what, when?)
- Table-of-contents sharing (show what's in the vault without revealing content)
- Monitoring dashboards (vault size, commit frequency, branch count)

**No concerns.** This is a clean separation and follows the principle of least privilege.

### 4. Patch-Based Agent Collaboration — SOUND

**Source:** Document 68

SGit sits on top of Git; agents produce Git patches in vaults. Vault contains brief + documentation + patch + metadata.

**Assessment:** Using Git's native patch format is the right choice — portable, reviewable, composable. The vault-as-transport pattern is proven (document 77 demonstrates it working for the upload refactoring).

**Enhancement needed:** Patch verification — the brief mentions verifying patches apply cleanly, but doesn't specify what happens when they don't. Need a clear error path.

### 5. Client-Side Recipient Restrictions — SOUND BUT CLEARLY BOUNDED

**Source:** Document 66

Sender defines policy (country, timezone, browser), encrypted in manifest, evaluated locally by recipient's browser.

**Assessment:** The trust model is correctly described as "soft constraint" — it works against honest users, not adversaries. The architecture is clean: policy encrypted alongside content, recipient collects context locally (SG/Send never sees recipient's IP), third-party geo API called by browser directly.

**The key architectural insight:** this doesn't need to be strong to be useful. "Only open in UK timezone" is a useful UX guard even if technically circumventable.

### 6. Upload Refactoring — IFD WORKING AT SCALE

**Source:** Documents 77, 78

7,221-line monolith decomposed into 7 modules + 6 Shadow DOM components (6,058 lines total). 11 sequential patches delivered via encrypted vault.

**Assessment:** This validates the IFD methodology at meaningful scale. The modular architecture is correct — shared modules for crypto/thumbnails/engine, Shadow DOM components for each step.

**Bug of note:** `[object Object]` in CSS paths caused by sub-components using wrong API. This is the kind of integration bug that typed interfaces would catch. Consider adding a component API contract.
