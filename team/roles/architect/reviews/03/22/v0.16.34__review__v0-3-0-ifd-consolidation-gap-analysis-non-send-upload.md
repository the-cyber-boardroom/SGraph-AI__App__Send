# Architect Review — v0.3.0 IFD Consolidation Gap Analysis (Non-`send-upload`)

**Version:** v0.16.34  
**Date:** 2026-03-22  
**Role:** Architect (Explorer team)  
**Scope:** Compare `sgraph_ai_app_send__ui__user/v0/v0.3/v0.3.0` against the effective `v0.2.x` lineage, starting from `v0.2.13` and reading backwards through the IFD overlay chain. **Explicitly excludes** `send-upload` because that refactor was already reviewed separately.

---

## 1. Why this review exists

The goal of the current IFD refactoring is that `v0.3.0` should be a **consolidated major version** that is functionally equivalent to the latest `v0.2.x` behavior. Under the IFD rules, a major version should merge proven minor-version overlays into a self-contained checkpoint and must not silently drop features or bug fixes that were already present in the active minor stream.

That matters here because the `v0.2.x` family evolved through a long sequence of surgical overlays. If `v0.3.0` consolidates only the happy-path code and misses route pages, locale assets, sample content, or non-upload overlays, then it is no longer a true IFD consolidation — it becomes a partial rewrite with regressions.

---

## 2. IFD framing used for this review

I used the IFD guide as the architectural lens for this review:

- Minor versions are allowed to add focused changes via overlays.
- Major versions must consolidate those overlays into a complete, merged checkpoint.
- The consolidation must preserve proven behavior, not just preserve the existence of the newest files.

In other words: **the reference behavior is the effective runtime behavior of the full v0.2.x chain, not just the files physically present in `v0.2.13/`.**

---

## 3. Comparison method

I compared:

- the current `v0.3.0` tree,
- the `v0.2.13` entrypoints,
- the earlier `v0.2.x` overlay pages and component overrides that still contribute runtime behavior,
- all non-`send-upload` assets and pages that form the effective user UI.

The main review areas were:

1. **Download-side functionality** (`send-download`, route pages, browse/gallery/view behavior)
2. **Access-gate functionality** (`send-access-gate` + v0.2.6 overlay)
3. **Shared UI/runtime assets** (locale infrastructure, test files, welcome flow, root/site files)
4. **Deployment/runtime assumptions** introduced in `v0.3.0`

---

## 4. Executive summary

### High-confidence conclusion

**Yes — outside `send-upload`, the `v0.3.0` consolidation has lost existing functionality and feature coverage from the latest `v0.2.x` line.**

The most important regressions are:

1. **Missing route pages** — `view`, `v`, and `welcome` are present in `v0.2.x` and absent from `v0.3.0`.
2. **Incomplete route handling in `send-download`** — the new consolidated download component only recognises `download`, `browse`, and `gallery`, not `view` or the short `/v/` route.
3. **Missing locale deliverables** — `v0.3.0` still advertises the full locale set in runtime code, but the locale page directories and `i18n/*.json` files are not present in the version tree.
4. **Missing test-file payloads** — the upload page still renders the test-files component, but `v0.3.0` does not include the `test-files/` corpus that the component links to.
5. **Missing site/package assets** — root `index.html`, `manifest.json`, `robots.txt`, and `sitemap.xml` are present in `v0.2.0` but not in the `v0.3.0` tree.
6. **New external dependency** in the download page (`sg-layout` from a remote CDN), which moves away from the repo’s earlier zero-dependency/static-packaging posture.

### Preserved / apparently OK

- `send-access-gate` **does appear to preserve** the `v0.2.6` token-bar redesign and busy-state hiding behavior in consolidated form.
- `send-test-files.js` and `.css` appear unchanged, which is good — but the supporting files they rely on are missing from the `v0.3.0` tree.
- `browse/` and `gallery/` are present in `v0.3.0` as symlinked entrypoints to `download/index.html`, which is architecturally acceptable because the new `send-download` uses route detection.

## 5. Detailed findings

## Finding A — `view` and short `v` routes were lost

### What existed in v0.2.x

By `v0.2.3`, the UI exposed distinct route entrypoints for:

- `/download/`
- `/browse/`
- `/gallery/`
- `/view/`
- `/v/`

These were real HTML entrypoints in the version tree.

### What exists in v0.3.0

`v0.3.0` only contains:

- `en-gb/index.html`
- `en-gb/download/index.html`
- `en-gb/browse/index.html` (symlink to download)
- `en-gb/gallery/index.html` (symlink to download)

There is **no** `en-gb/view/index.html` and **no** `en-gb/v/index.html`.

### Architectural assessment

This is a confirmed consolidation regression. Even if the new unified download component can render many content modes, the route-level contract from `v0.2.x` has not been preserved.

**Severity:** High  
**Type:** Lost entrypoint / route contract regression

---

## Finding B — the new `send-download` route model is narrower than the old one

### What existed in v0.2.x

The `v0.2.x` route set explicitly supported route-specific viewing modes. The HTML entrypoints were part of that product surface, and the later overlays continued to refine route-aware rendering behavior.

### What exists in v0.3.0

The new consolidated `send-download.js` only detects:

- `gallery`
- `browse`
- `download`
- otherwise `auto`

It does **not** detect `view`, and it does not expose any explicit support for the short `/v/` route.

### Architectural assessment

This confirms that the route loss is not just a missing page file problem. The underlying consolidated component has a narrower route contract than the previous family.

**Severity:** High  
**Type:** Functional regression in consolidated route semantics

---

## Finding C — `welcome` token-activation flow was dropped

### What existed in v0.2.x

`v0.2.0` shipped a dedicated `en-gb/welcome/index.html` page backed by a `send-welcome` component and translated strings for activation states.

### What exists in v0.3.0

There is no `welcome/` route and no `send-welcome` component in the `v0.3.0` tree.

### Architectural assessment

This is not an incidental omission. It removes an entire previously shipped user flow from the major-version checkpoint.

If the intention was to de-scope welcome activation in `v0.3.0`, it should have been called out explicitly as a product change. As implemented, it looks like an accidental loss during consolidation.

**Severity:** High  
**Type:** Lost feature / removed page flow

---

## Finding D — locale support is incomplete in the `v0.3.0` tree

### What existed in v0.2.x

`v0.2.0` included:

- concrete locale JSON files under `i18n/`
- locale-aware page structure
- the full multi-locale packaging expected by the locale selector

### What exists in v0.3.0

The new `i18n.js` still declares the same broad locale list and, for non-`en-gb` locales, tries to fetch `i18n/{locale}.json` at runtime.

However, the `v0.3.0` folder does not currently contain an `i18n/` directory or locale JSON files, and it also does not contain generated locale page directories alongside `en-gb/`.

### Architectural assessment

This creates a packaging-level inconsistency:

- the runtime advertises multilingual support,
- but the version folder does not ship the locale assets required to fulfil that contract.

This is especially important for a major IFD consolidation because the result should be a coherent deliverable, not a partially generated intermediate state.

**Severity:** High  
**Type:** Packaging / localization regression

---

## Finding E — test-files feature is present in UI code but not in packaged assets

### What existed in v0.2.x

The upload UI shipped real sample payloads under `test-files/`, including individual files and richer gallery-pack examples.

### What exists in v0.3.0

The page still renders `<send-test-files>` and the component still builds links under `../test-files/`, but the `v0.3.0` tree does not include a `test-files/` directory.

### Architectural assessment

This is a classic consolidation trap: the component survived, but the supporting runtime assets did not.

From a product perspective, this means the UI promises draggable test files that are not actually there in the consolidated tree.

**Severity:** High  
**Type:** Asset-packaging regression

---

## Finding F — root/site assets were not carried into `v0.3.0`

### What existed in v0.2.0

The version root included:

- `index.html`
- `manifest.json`
- `robots.txt`
- `sitemap.xml`
- `404.html`

### What exists in v0.3.0

Those files are not present in the `v0.3.0` tree.

### Architectural assessment

This may not block the upload/download happy path in local development, but it does make the consolidated version materially less complete as a deployable static site package.

At minimum, these omissions should be treated as unresolved consolidation work. In IFD terms, the major version is not yet a complete checkpoint.

**Severity:** Medium  
**Type:** Packaging / deployment completeness regression

---

## Finding G — `browse` and `gallery` are preserved via symlinked entrypoints

### What existed in v0.2.x

Separate `browse/` and `gallery/` HTML pages existed and loaded the download component with route-specific behavior.

### What exists in v0.3.0

`browse/index.html` and `gallery/index.html` are symlinks to `download/index.html`, and the new component reads the pathname to choose route mode.

### Architectural assessment

This is acceptable and, in fact, cleaner than the previous duplicated-entrypoint pattern. It preserves route semantics while reducing duplicated HTML.

**Severity:** None / positive finding  
**Type:** Good consolidation choice

---

## Finding H — `send-access-gate` appears correctly consolidated

### What existed in v0.2.x

`v0.2.0` provided the base gate, and `v0.2.6` overlaid:

- redesigned token bar,
- show/hide current token,
- copy token button,
- share note,
- hide-on-upload-busy behavior.

### What exists in v0.3.0

The consolidated `send-access-gate.js` carries both the base class and the v0.2.6 overlay logic in one file.

### Architectural assessment

This part looks like a proper IFD merge rather than a regression. I did not identify a feature loss here.

**Severity:** None / positive finding  
**Type:** Successful consolidation

---

## Finding I — new external CDN dependency in the download page

### What changed

The `v0.3.0` download page loads `sg-layout` from `https://dev.tools.sgraph.ai/...`.

### Architectural assessment

This is not necessarily a feature regression, but it is an architectural change that cuts against the repo’s IFD guidance around zero external dependencies and self-contained major versions.

If this dependency is intentional, it should be treated as a conscious architecture decision. If not, it is another sign that `v0.3.0` is still in transitional refactor state rather than a finished consolidation.

**Severity:** Medium  
**Type:** Architecture drift / packaging risk

## 7. Recommended next actions

## Priority 1 — restore route and page parity

1. Reintroduce `en-gb/view/index.html`.
2. Reintroduce `en-gb/v/index.html`.
3. Extend `send-download.js` route detection so the route contract matches the old family.
4. Restore `en-gb/welcome/index.html` and the `send-welcome` component if the welcome flow is still part of the product.

## Priority 2 — fix packaging completeness

5. Restore `test-files/` into the `v0.3.0` version tree.
6. Restore locale artifacts (`i18n/*.json` and generated locale directories) or explicitly scope them out if this version is intended to be English-only.
7. Restore the site-level files (`index.html`, `manifest.json`, `robots.txt`, `sitemap.xml`, `404.html`) needed for a coherent static package.

## Priority 3 — close architecture drift

8. Decide whether the external `sg-layout` dependency is acceptable for this major version.
9. If not, vendor it or replace it with a local component before treating `v0.3.0` as a completed IFD consolidation.

---

## 8. Final architect verdict

**`v0.3.0` is not yet functionally equivalent to the effective latest `v0.2.x` lineage outside `send-upload`.**

The biggest issue is not one giant logic bug. It is that the consolidation appears to have focused on the rewritten primary components while leaving behind several surrounding contracts that were still part of the shipped product surface:

- route entrypoints,
- welcome flow,
- locale packaging,
- test asset packaging,
- and static-site root files.

That is exactly the kind of regression pattern IFD major-version reviews are supposed to catch.

So my recommendation is:

> Treat `v0.3.0` as **an in-progress consolidation branch**, not yet as the true successor to `v0.2.17`, until the surrounding non-`send-upload` feature/package parity is restored.
