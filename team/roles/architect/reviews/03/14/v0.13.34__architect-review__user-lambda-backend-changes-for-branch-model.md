## 1. Executive Summary

The arch spec (v0.13.30) defines a git-like branch model with unified `bare/` storage, PKI-signed commits, and three-way merge. This review identifies the **exact backend changes required on the SG/Send User Lambda** to support the `sg-send-cli` workflows described in the spec.

**Bottom line:** The existing 4-endpoint Vault Pointer API (`write`, `read`, `read-base64`, `delete`) is **sufficient as the primitive layer** for the branch model. Two new endpoints are needed: **batch** and **list**. The storage path convention changes from flat to `bare/` sub-folders. No changes to auth model are needed for Phase 1.

## 2. What Exists Today (Code-Verified)

### 2.1 User Lambda Vault Endpoints

| Method | Path | Auth | Purpose |
|--------|------|------|---------|
| PUT | `/api/vault/write/{vault_id}/{file_id}` | access_token + write_key | Write encrypted blob |
| GET | `/api/vault/read/{vault_id}/{file_id}` | None | Read encrypted blob (binary) |
| GET | `/api/vault/read-base64/{vault_id}/{file_id}` | None | Read as base64 JSON (3.75MB limit) |
| DELETE | `/api/vault/delete/{vault_id}/{file_id}` | access_token + write_key | Delete vault file |

**Source:** `sgraph_ai_app_send/lambda__user/fast_api/routes/Routes__Vault__Pointer.py`

### 2.2 Storage Layout (Current)

```
transfers/vault/{vault_id}/manifest.json          write_key_hash, created_at
transfers/vault/{vault_id}/{file_id}/meta.json     per-file metadata
transfers/vault/{vault_id}/{file_id}/payload       encrypted bytes
```

**Source:** `sgraph_ai_app_send/lambda__user/service/Service__Vault__Pointer.py`

### 2.3 Key Observations

1. **`file_id` is already free-form.** The CLI can pass `bare/data/obj-abc123` as `file_id` today — the service treats it as an opaque path segment. No schema validation blocks this.
2. **Write-key auth is vault-scoped.** One write_key per vault, stored as SHA-256 hash in `manifest.json`. This maps directly to the arch spec's `write_key` derived from HKDF.
3. **Read is unauthenticated.** This matches the arch spec — encrypted blobs are opaque, zero-knowledge.
4. **Storage is pluggable.** `Storage_FS` backend (Memory, S3, disk) already abstracts the filesystem.

## 3. New Endpoints Required

The arch spec (sections 12.2 and 12.3) specifies two new endpoints. Here is the Architect's assessment of each.

### 3.1 Batch Endpoint

**Spec reference:** Section 12.2

```
POST /api/vault/batch/{vault_id}
Header: x-sgraph-vault-write-key
Body: {
  "operations": [
    { "op": "write",          "file_id": "bare/data/obj-{hash}", "data": "<base64>" },
    { "op": "write-if-match", "file_id": "bare/refs/ref-{id}",
      "match": "<old_base64>", "data": "<new_base64>" },
    { "op": "delete",         "file_id": "bare/data/obj-{hash}" }
  ]
}
```

**Why it matters:** A push without batch = 6+ round trips (one per object + one per ref). With batch = 1 POST. The CLI is network-bound, not compute-bound. Batch is the single most impactful change.

**Design decisions needed:**

| Question | Recommended Answer | Rationale |
|----------|-------------------|-----------|
| Atomicity | **Best-effort, ordered** — execute operations in array order, stop on first failure, return partial results | Full atomicity requires transactions; S3 doesn't have them. Immutable objects (blobs, trees, commits) are idempotent — writing them twice is harmless. Only the ref `write-if-match` matters for correctness. |
| Max operations per batch | **100** | Keeps Lambda execution under 10s. Each S3 PUT is ~5-50ms. |
| Max payload size | **5MB total** (Lambda request limit) | Base64 encoding adds ~33%, so ~3.75MB of actual data per batch. Individual objects up to ~1MB. |
| Error format | `{ "results": [{ "op": "write", "file_id": "...", "status": "ok" }, { "op": "write-if-match", "file_id": "...", "status": "conflict", "current": "<base64>" }] }` | Client needs to know which operations succeeded and which ref had a conflict. |
| `write-if-match` semantics | **Compare-and-swap on payload bytes.** Read current payload, compare to `match` value. If equal, write `data`. If not, return `conflict` with current value. | This is the optimistic locking primitive the push workflow needs (spec section 6.3, step 7). |
| Auth | **Same as write** — access_token + write_key | Batch is a write operation. Same auth model, validated once at the top. |

**Implementation location:**
- Route: New class `Routes__Vault__Batch` or add method to existing `Routes__Vault__Pointer`
- Service: New method `batch()` on `Service__Vault__Pointer`

**Estimated scope:** ~80-120 LOC (route + service + error handling)

### 3.2 List Endpoint

**Spec reference:** Section 12.3

```
GET /api/vault/list/{vault_id}/{prefix}
Returns: { "files": ["bare/data/obj-abc", "bare/data/obj-def", ...] }
```

**Why it matters:** The CLI needs to enumerate `bare/pending/` during GC drain, and `bare/branches/` for branch discovery. Without list, the CLI would need to maintain a client-side index of every file — fragile and stale.

**Design decisions needed:**

| Question | Recommended Answer | Rationale |
|----------|-------------------|-----------|
| Auth | **Unauthenticated** (same as read) | File IDs are opaque. Listing `bare/data/obj-*` reveals nothing about content or relationships. Consistent with zero-knowledge model. |
| Prefix matching | **Exact prefix on the `file_id` path** | `prefix=bare/refs/` returns all ref files. `prefix=bare/pending/` returns all pending change packs. |
| Response format | `{ "vault_id": "...", "prefix": "...", "files": ["file_id_1", "file_id_2"] }` | Simple, no pagination needed initially (vaults will have <1000 files for a long time). |
| Pagination | **Deferred.** Add `cursor` + `limit` when vaults exceed ~1000 files. | YAGNI for now. Keep it simple. |

**Implementation note:** `Storage_FS` doesn't currently expose a `list(prefix)` method. Options:
1. **Add `files__list(prefix)` to Storage_FS** — clean, but requires upstream change to `memory_fs` package
2. **Use `files__paths()` + filter** — works today, scans all files in vault. Fine for <1000 files.
3. **S3 backend can use `list_objects_v2(Prefix=...)` directly** — optimal for production

**Recommendation:** Start with option 2 (filter on `files__paths()`), add S3-optimised path as a performance win in Villager phase.

**Estimated scope:** ~30-50 LOC

## 4. Storage Path Changes

### 4.1 Current → New Path Mapping

The arch spec introduces the `bare/` folder convention. The server stores files at paths that include the `bare/` prefix:

| Current Pattern | New Pattern (arch spec) |
|----------------|------------------------|
| `transfers/vault/{vault_id}/{file_id}/payload` | `transfers/vault/{vault_id}/{file_id}/payload` |

**Key insight:** The `file_id` parameter already accepts arbitrary strings. The CLI will simply pass `bare/data/obj-abc123` as the `file_id`. The existing storage service will store it at:

```
transfers/vault/{vault_id}/bare/data/obj-abc123/payload
transfers/vault/{vault_id}/bare/data/obj-abc123/meta.json
```

**No storage path code changes needed.** The `bare/` hierarchy is a convention enforced by the CLI, not the server. The server remains dumb — it stores opaque blobs at whatever `file_id` the client specifies.

### 4.2 Meta.json Overhead

Currently every file gets a `meta.json` alongside its `payload`. For the branch model's content-addressed objects, this metadata (write_count, created_at, updated_at) is unnecessary overhead — immutable objects are written once and never updated.

**Options:**
1. **Keep meta.json** — adds ~200 bytes per object but simplifies audit trail. S3 cost is negligible.
2. **Make meta.json optional** — skip it for `bare/data/obj-*` files (immutable, content-addressed).
3. **Defer** — worry about it when vaults exceed 10,000 objects.

**Recommendation:** Option 1 for now (keep meta.json). The overhead is tiny and the audit trail has value. Revisit in Villager phase if S3 list/storage costs become material.

## 5. Auth Model — What Changes, What Doesn't

### 5.1 Phase 1: No Auth Changes

The arch spec's write_key (section 10) is derived via HKDF from the vault passphrase. The server receives this as a hex string and validates it against the SHA-256 hash stored in `manifest.json`. **This is exactly what the current code does.**

| Concern | Current Behaviour | Arch Spec Requirement | Gap? |
|---------|-------------------|----------------------|------|
| Write authentication | SHA-256(write_key_hex) stored in manifest | write_key from HKDF, SHA-256 hashed | **None** — compatible |
| Read authentication | None (zero-knowledge) | None (zero-knowledge) | **None** |
| Branch key signing | N/A | Client-side only (EC P-256) | **None** — server never sees signatures |
| Named branch merge auth | N/A | Named branch private key (encrypted in vault) | **None** — server never sees private keys |

### 5.2 Phase 2: Scoped Write Tokens (Change Packs)

The arch spec (section 12.4) introduces scoped write-only tokens for change packs. This is **not needed in Phase 1** (Foundation + Merge + Push). It's needed in Phase F (Change Packs).

When needed, the change is:
- New header: `x-sgraph-vault-write-token`
- Token validates against a scoped token store (admin lambda creates these)
- Token permits writes ONLY to `bare/pending/*` file_ids
- Service layer adds path-prefix check before write

**Estimated scope:** ~40 LOC in route + service.

## 6. `write-if-match` — The Critical Primitive

This is the single most important new capability. Without it, concurrent pushes can silently overwrite each other's ref updates.

### 6.1 Semantics

```python
def write_if_match(vault_id, file_id, match_value, new_value, write_key_hex):
    current = read(vault_id, file_id)
    if current == match_value:          # Compare raw bytes
        write(vault_id, file_id, new_value, write_key_hex)
        return { "status": "ok" }
    else:
        return { "status": "conflict", "current": base64(current) }
```

### 6.2 Race Condition

In-memory: Single-threaded Lambda — no race. Each Lambda invocation processes one request.

S3: Not atomic. Two concurrent pushes could both read the same ref, both see a match, and both write. **Mitigation:**
1. S3 conditional writes (`If-None-Match` for new objects, but S3 doesn't support `If-Match` for updates)
2. **DynamoDB conditional write** for refs only (adds dependency but solves atomicity)
3. **Accept the race** — it's very rare (two users pushing the same vault within milliseconds). The next push will detect the divergence and require a merge.

**Recommendation:** Option 3 for Phase 1. The push workflow already handles "ref moved while merging" (spec section 6.3, step 7 — retry from step 2). The race window is Lambda invocation duration (~50ms). Acceptable for early adoption. DynamoDB locking is a Villager-phase hardening item.

## 8. What Does NOT Change on the Server

| Item | Why |
|------|-----|
| Encryption/decryption | Client-side only. Server stores opaque bytes. |
| Branch keys (EC P-256) | Client-side only. Server never sees private keys. |
| Commit signing/verification | Client-side only. Signatures are inside encrypted blobs. |
| Three-way merge | Client-side only. CLI merges locally, pushes result. |
| Conflict resolution | Client-side only. `.conflict` files are local working-copy artifacts. |
| Tree traversal | Client-side only. Server doesn't know blob vs tree vs commit. |
| Sub-tree diffing | Client-side only. The server stores immutable content-addressed objects. |
| Branch index | Encrypted blob stored via normal write. Server doesn't parse it. |

**The server's job is simple: store blobs, serve blobs, support batch + list + compare-and-swap.** Everything else is client intelligence.

## 9. Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Batch payload exceeds Lambda 5MB limit | Push fails for large commits | CLI splits into multiple batches; batch endpoint returns partial results |
| S3 write-if-match race condition | Silent ref overwrite on concurrent push | Push workflow retries on conflict (spec step 7); DynamoDB locking in Villager phase |
| `files__paths()` scan for list is slow at scale | List endpoint latency grows with vault size | S3 `list_objects_v2` optimisation in Villager phase; acceptable for <1000 files |
| Scoped tokens bypass write_key auth | Unauthorised writes to `bare/pending/` | Scoped tokens are vault-owner-issued, path-restricted, time-limited |
| Meta.json overhead for immutable objects | 2x S3 PUTs per object (payload + meta) | Acceptable overhead; remove meta.json for `bare/data/` in Villager phase |

## 10. Recommended Build Order

```
1. write-if-match primitive on Service__Vault__Pointer     (~20 LOC)
   └─ Test: write, match succeeds, match fails with current value returned

2. batch endpoint (route + service)                        (~100 LOC)
   └─ Test: batch of writes, batch with write-if-match, partial failure

3. list endpoint (route + service)                         (~35 LOC)
   └─ Test: list with prefix, empty prefix, no matches

4. Route registration in Fast_API__User                    (~5 LOC)

5. Integration test: push workflow simulation              (~50 LOC)
   └─ Write objects via batch, write-if-match on ref, list to verify
```

**Total server-side change for Phase 1: ~210 LOC + ~160 LOC tests = ~370 LOC.**

This is a small, focused delta that unblocks the entire CLI branch model without changing existing endpoints or breaking backward compatibility.
