# IFD Deployment Pipeline — Architecture Review

**Version:** v0.13.34
**Date:** 14 March 2026
**Role:** Architect
**Status:** Implemented and verified on dev.send.sgraph.ai

---

## 1. What Is IFD Deployment?

IFD (Iterative Flow Development) is SGraph Send's methodology for building and deploying UI versions as **surgical overlays** rather than full rewrites. The core idea:

> **The unit of change is the change itself — not the whole file.**

A patch version (e.g. v0.2.1) contains **only the files that differ** from the base (v0.2.0). The deployment pipeline layers these versions on top of each other, producing a `latest/` folder that is the union of all versions.

This is different from conventional deployment where each version is a complete, self-contained build. In IFD, versions are sparse — a patch might contain 3 files out of 50, and the other 47 come from the base.

## 2. How the UI is Structured

### 2.1 Folder Layout

```
sgraph_ai_app_send__ui__user/
  v0/
    v0.2/
      v0.2.0/                      ← Base (complete, self-contained)
        _common/
          css/                     ← Design tokens, shared styles
          fonts/                   ← Vendored fonts (no CDN)
          js/
            base/                  ← Core system (component loader, helpers)
            components/
              send-download/       ← send-download.js (base class)
              send-header/
              send-footer/
              send-locale/
              send-transparency/
            vendor/                ← Third-party (jszip.min.js)
            i18n.js                ← Internationalisation module
            crypto.js              ← Web Crypto API wrapper
            api-client.js          ← Backend API client
        en-gb/                     ← English (source template)
          index.html               ← Upload page
          download/index.html      ← Download page
          welcome/index.html       ← Welcome page
        i18n/                      ← Translation JSON files
          en-gb.json, pt-pt.json, de-de.json, ...
        manifest.json

      v0.2.1/                      ← Patch (sparse — only changed files)
        _common/
          js/
            components/
              send-download/
                send-download-v021.js   ← Override file (NOT send-download.js)
                send-download-v021.css  ← Override styles
        en-gb/
          download/index.html      ← Updated HTML (loads both base + override)
```

**Key observation:** v0.2.1 has ~5 files. v0.2.0 has ~50. The patch is sparse by design.

### 2.2 How Overrides Work in the Browser

The v0.2.1 download page HTML loads scripts in this order:

```html
<!-- v0.2.0 base class — defines class SendDownload, registers <send-download> -->
<script src="../../_common/js/components/send-download/send-download.js"></script>

<!-- v0.2.1 surgical override — mutates SendDownload.prototype -->
<script src="../../_common/js/components/send-download/send-download-v021.js"></script>

<!-- Init — runs against the patched class -->
<script>
    SendComponentPaths.init('../../_common');
    I18n.init();
</script>
```

The override file (`send-download-v021.js`) does **not** re-register the custom element. It saves references to original methods and replaces them on the prototype:

```javascript
// Guard: if base class doesn't exist, skip (don't crash)
if (typeof SendDownload === 'undefined') {
    console.warn('[send-download-v021] SendDownload base class not found — skipping overrides');
    return;
}

// Store originals for super-calling
const _v020_setupEventListeners = SendDownload.prototype.setupEventListeners;
const _v020_cleanup             = SendDownload.prototype.cleanup;

// Override: new two-column zip layout
SendDownload.prototype._renderZipLayout = function() { /* ... */ };
```

CSS follows the same pattern: `send-download-v021.css` loads **after** `send-download.css`, and the cascade ensures override rules win.

### 2.3 Future Versions (v0.2.2, v0.2.3, ...)

This pattern extends naturally:

```
v0.2.2/
  _common/js/components/send-download/send-download-v022.js
  _common/js/components/send-download/send-download-v022.css
  en-gb/download/index.html   ← Adds <script src="send-download-v022.js">
```

The HTML loads all three in order: base → v021 → v022. Each layer patches what it needs. The deploy pipeline overlays v0.2.0 → v0.2.1 → v0.2.2 onto `latest/`, and the browser loads them in sequence.

At v0.3.0 (the next major), all overrides consolidate into a fresh self-contained base — no more layering.

## 3. How the Deployment Pipeline Works

### 3.1 Overview

```
Push to dev branch
        │
        ▼
┌─────────────────────────────────────────────────┐
│  GitHub Actions: deploy-ui-user.yml             │
│                                                 │
│  1. inject_build_version.py (v0.2.0 + v0.2.1)  │
│  2. generate_send_i18n_pages.py (v0.2.0+v0.2.1)│
│  3. deploy_static_site.py --version v0.2.0      │
│  4. deploy_static_site.py --version v0.2.1      │
│  5. CloudFront invalidation                     │
└─────────────────────────────────────────────────┘
        │
        ▼
┌─────────────────────────────────────────────────┐
│  S3 Bucket                                      │
│                                                 │
│  websites/sgraph-send/dev/                      │
│    releases/                                    │
│      v0/v0.2/v0.2.0/  ← archive (complete)     │
│      v0/v0.2/v0.2.1/  ← archive (sparse)       │
│    latest/             ← union (served by CF)   │
└─────────────────────────────────────────────────┘
        │
        ▼
  CloudFront → dev.send.sgraph.ai
```

### 3.2 Step-by-Step

**Step 1: Build version injection** (`inject_build_version.py`)

Generates `_common/js/build-info.js` for each version:

```javascript
window.SGRAPH_BUILD = {
    appVersion : 'v0.13.34',
    uiVersion  : 'v0.2.1',
    buildTime  : '2026-03-14T02:19:00Z'
};
```

This is a CI artifact (`.gitignore`'d) — never committed to git. The footer component reads `window.SGRAPH_BUILD` to display the version.

**Step 2: i18n page generation** (`generate_send_i18n_pages.py`)

Takes `en-gb/` as the source template and generates 15 additional locale folders:

```
en-gb/ (source, committed)  →  en-us/, pt-pt/, pt-br/, de-de/, de-ch/,
                                es-es/, es-ar/, es-mx/, fr-fr/, fr-ca/,
                                it-it/, nl-nl/, pl-pl/, hr-hr/, ro-ro/, tlh/
```

Each locale folder is a copy of `en-gb/` with:
- `<html lang="en-GB">` → `<html lang="pt-PT">`
- `<h1 data-i18n="title">English text</h1>` → `<h1 data-i18n="title">Texto em Português</h1>`
- `<input data-i18n-placeholder="key">` → translated placeholder
- `${I18n.t('key')}` template literals → baked-in translated values
- `<meta name="i18n-prerendered" content="true" />` marker added

Generated locales are `.gitignore`'d — they exist only in CI and S3.

All locale folders share the same `_common/` — no CSS/JS/font duplication.

**Step 3: Deploy v0.2.0 base** (`deploy_static_site.py`)

```bash
python deploy_static_site.py \
    --site sgraph-send \
    --version v0.2.0 \
    --source-dir sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0 \
    --deploy-env dev
```

This does two things:
1. **Archive** — syncs to `releases/v0/v0.2/v0.2.0/` with `--delete` (clean, exact copy)
2. **Overlay** — syncs to `latest/` **without** `--delete` (additive)

The overlay step uses `aws s3 sync` by file type, each with appropriate `Cache-Control`:
- HTML: 5 minutes (`public, max-age=300`)
- CSS/JS/JSON: 1 day (`public, max-age=86400`)
- Images: 1 week (`public, max-age=604800`)

**Step 4: Deploy v0.2.1 overlay**

```bash
python deploy_static_site.py \
    --site sgraph-send \
    --version v0.2.1 \
    --source-dir sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.1 \
    --deploy-env dev \
    --version-file sgraph_ai_app_send/version
```

Same two-step process. v0.2.1's files overlay onto `latest/` on top of v0.2.0. Since v0.2.1 has `send-download-v021.js` (not `send-download.js`), the v0.2.0 base class is **not overwritten**.

The `--version-file` flag uploads `sgraph_ai_app_send/version` to `latest/version`, making it accessible at `https://dev.send.sgraph.ai/version`.

**Step 5: CloudFront invalidation**

`aws cloudfront create-invalidation --paths "/*"` — clears the CDN cache so users see the new version immediately.

### 3.3 What `latest/` Contains After Deployment

```
latest/
  _common/
    js/components/send-download/
      send-download.js          ← from v0.2.0 (base class)
      send-download.css         ← from v0.2.0
      send-download-v021.js     ← from v0.2.1 (override)
      send-download-v021.css    ← from v0.2.1
    js/components/send-header/  ← from v0.2.0 (untouched by v0.2.1)
    js/components/send-footer/  ← from v0.2.0 (untouched by v0.2.1)
    ...
  en-gb/
    index.html                  ← from v0.2.0 (upload page, untouched)
    download/index.html         ← from v0.2.1 (loads both scripts)
  en-us/                        ← CI-generated from v0.2.0 templates
  pt-pt/                        ← CI-generated from v0.2.0 templates
  ...
  version                       ← uploaded separately (plain text: v0.13.34)
```

### 3.4 Environment Isolation

Each branch deploys to its own S3 prefix:

| Branch | S3 Path | Domain |
|--------|---------|--------|
| `dev` | `websites/sgraph-send/dev/` | dev.send.sgraph.ai |
| `main` | `websites/sgraph-send/main/` | main.send.sgraph.ai |
| production | `websites/sgraph-send/prod/` | send.sgraph.ai |

All use the same S3 bucket (`{account-id}--static-sgraph-ai--{region}`), separated by path prefix.

### 3.5 Recovery Tools

The deploy script provides two recovery mechanisms:

**`--clean-latest`** — Delete all files in `latest/` before overlaying the current version. Used when a single stale file needs to be removed.

**`--rebuild-latest`** — Nuclear option. Deletes `latest/`, lists all versions from `releases/`, replays them in order (v0.2.0 → v0.2.1 → ...). Produces a clean IFD union from the release archives.

Both are available as a checkbox in the GitHub Actions workflow dispatch UI.

## 4. Native i18n — How 16 Locales Work Without a Framework

### 4.1 The Problem

Most i18n solutions require either:
- A JavaScript framework with a runtime translation engine (React-intl, vue-i18n)
- A server-side template engine (Jinja, EJS) that renders locale pages
- A JSON-fetching runtime that replaces text after page load (causes flash-of-untranslated-content)

SGraph Send uses **vanilla JS Web Components with zero framework dependencies**. There is no server-side rendering (it's a static site on S3). The challenge: how to serve 16 locales from a static site with no framework and no runtime translation overhead?

### 4.2 The Solution: CI-Generated Pre-rendered Pages

The `en-gb/` folder is the single source of truth, committed to git with `data-i18n` attributes marking translatable elements:

```html
<h1 data-i18n="app.title">Your files, your keys, your privacy</h1>
<input data-i18n-placeholder="upload.dropzone" placeholder="Drop files here">
```

Translation JSON files live alongside in `i18n/`:

```json
// i18n/pt-pt.json
{
    "app.title": "Os seus ficheiros, as suas chaves, a sua privacidade",
    "upload.dropzone": "Largue ficheiros aqui"
}
```

At CI time, `generate_send_i18n_pages.py` reads each JSON and generates a complete locale folder:

```html
<!-- pt-pt/index.html (generated, not committed) -->
<html lang="pt-PT">
<meta name="i18n-prerendered" content="true" />
<h1 data-i18n="app.title">Os seus ficheiros, as suas chaves, a sua privacidade</h1>
<input data-i18n-placeholder="upload.dropzone" placeholder="Largue ficheiros aqui">
```

### 4.3 Why This Works Well

1. **Zero runtime cost** — translations are baked into HTML. No JSON fetch, no DOM manipulation, no flash of untranslated content.
2. **SEO-friendly** — each locale has its own URL (`/pt-pt/download/`), its own `<html lang>`, its own translated `<title>` and `<meta description>`.
3. **No framework dependency** — works with vanilla Web Components.
4. **Shared assets** — all locale folders reference the same `_common/` via relative paths. No CSS/JS/font duplication across 16 locales.
5. **IFD-compatible** — locale generation runs per version. v0.2.0 gets its locales, v0.2.1 gets its locales. The overlay model handles the rest.
6. **Translation workflow** — translators edit JSON files, not HTML. JSON files are small, diff-friendly, and easy to send for review.

### 4.4 Locale Routing

URL structure: `https://send.sgraph.ai/{locale}/{page}/`

```
/en-gb/              → Upload page (English GB)
/en-gb/download/     → Download page
/pt-pt/              → Upload page (Portuguese)
/pt-pt/download/     → Download page (Portuguese)
```

The `send-locale` Web Component provides a language picker dropdown. Switching locale navigates to the equivalent page in the new locale — all pre-rendered, instant load.

## 5. Challenges Faced and Lessons Learned

### 5.1 The Stale File Ghost (Critical — 14 March 2026)

**Problem:** After deploying v0.2.1, the download page crashed with `Uncaught ReferenceError: SendDownload is not defined`.

**Root cause chain:**
1. Early in development, v0.2.1 contained a file named `send-download.js` (the override). This was deployed to `latest/`, **overwriting** v0.2.0's `send-download.js` (the base class).
2. Later, the override was renamed to `send-download-v021.js` to avoid the collision.
3. But the old `send-download.js` was already in S3 `latest/` — and because the IFD overlay uses `aws s3 sync` **without `--delete`**, it was never removed.
4. The stale `send-download.js` in `latest/` was the v0.2.1 override (which expects `SendDownload` to already exist), not the v0.2.0 base class (which defines `SendDownload`).
5. When the browser loaded `send-download.js`, it got the override instead of the base class → crash.

**Why it was hard to diagnose:** The file had the right name (`send-download.js`) and was the right size. Only by inspecting the file contents in Chrome DevTools Sources panel could you see the comment said "v0.2.1 — Surgical overrides on v0.2.0" instead of the v0.2.0 base class definition.

**Fix:** The v0.2.1 override guard (IIFE with `typeof SendDownload === 'undefined'` check) prevented a hard crash and logged a warning, which made the problem visible.

**Lesson:** In an overlay deployment model, **file renames are dangerous**. The old filename persists in `latest/` forever unless explicitly cleaned. Always use new filenames (with version suffixes) for override files from the start.

### 5.2 The Workflow Only Deployed One Version

**Problem:** After identifying the stale file, we deleted `latest/` in S3 and re-ran the deploy. The download page still crashed.

**Root cause:** The GitHub Actions workflow only deployed v0.2.1. It assumed v0.2.0 was already in `latest/` from a previous deployment. When `latest/` was cleaned, v0.2.0's base files were gone, and re-deploying only v0.2.1 gave back the sparse patch without the base.

**Fix:** Updated the workflow to deploy **both v0.2.0 and v0.2.1** on every run. v0.2.0 goes first (base), v0.2.1 overlays on top. This makes every deployment self-contained — it doesn't depend on prior state in S3.

**Lesson:** An IFD deployment workflow must deploy **all versions in the overlay chain**, not just the latest patch. The workflow should be idempotent: running it on an empty `latest/` produces the same result as running it on an existing one.

### 5.3 The Rebuild Checkbox That Didn't Exist Yet

**Problem:** We added a `rebuild_latest` checkbox to the workflow dispatch, deleted `latest/`, and triggered a manual rebuild. It didn't work — the stale file came back.

**Root cause:** The workflow changes were on a feature branch, not `dev`. GitHub Actions runs the workflow definition from the **branch being deployed**, which was `dev` (the old version without the checkbox or the new deploy logic).

**Lesson:** Workflow changes only take effect after merging to the target branch. When debugging deployment issues, you must merge the fix to `dev` before the workflow dispatch will use it.

### 5.4 Release Archives Can Be Stale Too

**Observation:** The `releases/v0/v0.2/v0.2.1/` archive in S3 also contained the old `send-download.js` from the pre-rename deployment. If we had used `--rebuild-latest` (which replays from releases), it would have replayed the stale file.

**Resolution:** The normal deploy path syncs from **local git source** (which is clean), not from S3 releases. So re-deploying from `dev` branch overwrites the stale release archive with the correct files.

**Lesson:** The `releases/` archive is useful for rollback, but it reflects what was deployed at that point in time — including any mistakes. The `--rebuild-latest` flag should only be used when you're confident the release archives are clean. For recovery from stale files, a fresh deploy from git source is safer.

### 5.5 The Guard Clause That Saved Us

The IIFE guard in `send-download-v021.js` was critical for diagnosis:

```javascript
;(function() {
    if (typeof SendDownload === 'undefined') {
        console.warn('[send-download-v021] SendDownload base class not found — skipping overrides');
        return;
    }
    // ... overrides ...
})();
```

Without this guard, the page would have thrown an unhandled error on `SendDownload.prototype.setupEventListeners` (line 23) and potentially broken other components loading after it.

With the guard, the error was:
1. A clear `console.warn` identifying exactly which file had the problem
2. A graceful degradation (page loaded but download component was empty)
3. A second error from the stale `send-download.js` (`SendDownload is not defined`) that pointed directly to the culprit

**Lesson:** Every IFD override file should have a guard clause that checks for its base class. This turns hard crashes into diagnosable warnings.

## 6. Why We Do It This Way

### 6.1 Problems Solved

**Problem 1: Full rebuilds are wasteful and risky.**
Traditional deployment replaces everything on every deploy. A one-line CSS fix redeploys 50 files. IFD deploys only the changed files — a CSS fix is literally one file.

**Problem 2: Version archaeology is hard.**
When every version is a complete copy, understanding what changed between v0.2.0 and v0.2.1 requires diffing two 50-file trees. In IFD, v0.2.1 contains only the 5 files that changed — the diff is the version itself.

**Problem 3: Rollback is all-or-nothing.**
In conventional deployment, rolling back means redeploying the entire previous version. In IFD, you can roll back a single file by removing the override, or roll back an entire patch by removing its sparse folder from the overlay chain.

**Problem 4: i18n without a framework.**
Most i18n solutions require React/Vue/Angular or a server-side renderer. SGraph Send runs as a static site on S3 with vanilla Web Components. CI-generated pre-rendered locale pages solve this cleanly: zero runtime cost, full SEO, no framework lock-in.

**Problem 5: CI is a black box.**
Many projects have CI pipelines that are opaque shell scripts. SGraph Send's CI runs Python scripts that work identically locally and in GitHub Actions. You can debug deployments by running the same scripts on your machine.

### 6.2 Design Principles

1. **The change is the unit** — Patch versions contain only what changed. No copying untouched files.
2. **Overlay, don't replace** — `latest/` accumulates files. Each version adds/overwrites. Previous files persist.
3. **Deploy from source, archive for reference** — Normal deploys sync from git. Release archives are historical snapshots, not the source of truth.
4. **Guard everything** — Override files check for their base class. Deployment scripts validate before syncing.
5. **Idempotent pipeline** — Running the workflow on an empty `latest/` produces the same result as running it on a populated one. Every deploy is self-contained.
6. **CI generates, git stores source** — Locale pages and build-info are CI artifacts. Only `en-gb/` and `i18n/*.json` are in git. Generated output is `.gitignore`'d.

## 8. Adding v0.2.2 — What to Do

1. Create `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.2/` with only the changed files
2. Name override files with `-v022` suffix (e.g. `send-download-v022.js`)
3. Update HTML to load all three: base → v021 → v022
4. In `.github/workflows/deploy-ui-user.yml`:
   - Add `UI_VERSION_PATCH: v0.2.2` and `SOURCE_DIR_PATCH`
   - Add inject + generate steps for v0.2.2
   - Add a third deploy step after v0.2.1
5. The deploy script needs no changes — it already handles any version

At v0.3.0, consolidate all overrides into a fresh self-contained base. The workflow resets to deploying a single version.
