## Executive Summary

Two briefs from 1 Mar 2026 introduce significant new subsystems:

1. **Payment Notification Pipeline** -- Stripe webhook to multi-channel alerts (email, SMS, WhatsApp), with a future two-way agent communication channel via WhatsApp. This extends the Admin Lambda with a new route group and introduces external service dependencies (Stripe, AWS SNS, email service, WhatsApp provider).

2. **Translation Scaling Strategy** -- Three content zones with differentiated translation policies, a decoupled translation workflow, and the Rosetta archive directory. This is primarily a development workflow and repo structure change with deployment implications.

Both briefs build on existing architecture but introduce new integration surfaces. This review analyses placement, data flow, security, and sequencing for each.

### 1.2 Architectural Decision: Where Does the Webhook Endpoint Live?

**Recommendation: Admin Lambda, with a dedicated route group and authentication bypass for Stripe.**

The brief places the webhook at `POST /admin/webhooks/stripe`. This is correct. The Admin Lambda is the right host because:

1. **Token creation is an admin operation.** The webhook's ultimate action (Phase 2+) is creating a token via `Service__Tokens.create()`. This service is already wired into the Admin Lambda.
2. **Audit trail integration.** Payment events should be logged via `Service__Audit.log()` -- the audit service is on the Admin Lambda.
3. **MCP exposure.** The webhook event data and payment status should be queryable via MCP for Sherpa/agent interaction. The Admin Lambda already exposes MCP.
4. **Separation from user traffic.** Webhooks from Stripe are infrastructure traffic, not user traffic. Keeping them on the Admin Lambda avoids polluting User Lambda analytics.

**However, there is an authentication conflict.** The Admin Lambda enforces API key authentication on all routes (via `osbot-fast-api-serverless` default config: `enable_api_key = True`). Stripe cannot provide an `x-api-key` header -- it uses its own `Stripe-Signature` header with HMAC-SHA256 verification.

**Decision needed:** The webhook endpoint must bypass the standard Admin Lambda API key check and instead authenticate via Stripe signature verification. Two approaches:

| Approach | Pros | Cons |
|----------|------|------|
| **A. Route-level auth exclusion** -- add `/webhooks/stripe` to an auth bypass list in `Serverless__Fast_API` config | Clean, explicit, follows existing pattern | Must verify that `osbot-fast-api-serverless` supports route-level auth exclusions |
| **B. Separate router with no auth** -- mount the webhook routes on a separate FastAPI sub-application without API key middleware | Complete isolation, no changes to core auth | Adds complexity, second app instance |

**Architect recommendation:** Approach A. The `Serverless__Fast_API` base class likely supports path-based auth exclusions (the User Lambda already has routes without API key, e.g. `/transfers/download/`). Verify this in `osbot-fast-api-serverless` source before implementation. If not supported, fall back to Approach B.

### 1.3 Data Flow: Stripe Payment to Token Delivery

#### Phase 1 (Semi-Manual)

```
Customer browser
    |
    v
Stripe Checkout (GBP 5.00)
    |
    v
Stripe fires webhook (checkout.session.completed)
    |
    v
POST /webhooks/stripe (Admin Lambda)
    |
    +-- [1] Verify Stripe-Signature (HMAC-SHA256)
    |       |
    |       +-- FAIL --> 400 Bad Request (log attempt)
    |       |
    |       +-- PASS --> continue
    |
    +-- [2] Parse event payload
    |       Extract: customer_email, amount, payment_id, timestamp
    |
    +-- [3] Store payment event (new NS_PAYMENTS namespace in cache)
    |       KEY_BASED strategy, keyed by payment_id
    |       Data: { payment_id, customer_email, amount, currency,
    |               status: 'received', token_name: '', timestamp }
    |
    +-- [4] Log to audit trail
    |       Service__Audit.log(room_id='system', user_id='stripe',
    |                          action='payment_received',
    |                          metadata={payment_id, amount, email_hash})
    |       NOTE: customer_email is PII -- store SHA-256 hash in audit,
    |             full email only in the payment record
    |
    +-- [5] Dispatch notifications (async, best-effort)
    |       |
    |       +-- Email to sherpa@sgraph.ai (via sg-workmail or SES)
    |       +-- Email to human's personal email
    |       +-- SMS to human's phone (via AWS SNS)
    |       +-- WhatsApp to human's phone (via Twilio sandbox)
    |
    +-- [6] Return 200 OK to Stripe
            (Stripe retries on non-2xx, so return 200 even if
             notifications fail -- the payment record is stored)
```

#### Phase 2 (Automated)

```
... Steps [1]-[4] same as Phase 1 ...
    |
    +-- [5] Auto-create token
    |       Service__Tokens.create(
    |           token_name  = generate_token_name(payment_id),
    |           usage_limit = 500,
    |           created_by  = 'stripe-webhook',
    |           metadata    = { payment_id, customer_email, amount })
    |
    +-- [6] Update payment record with token_name
    |
    +-- [7] Send token delivery email
    |       via sg-workmail: send token + instructions to customer_email
    |
    +-- [8] Dispatch admin notifications (same as Phase 1 step 5)
    |       but now include token_name in the notification
    |
    +-- [9] Return 200 OK
```

#### Phase 3 (Two-Way WhatsApp)

```
Human (WhatsApp) --> "Create token for user@example.com"
    |
    v
WhatsApp Business API / Twilio webhook
    |
    v
POST /webhooks/whatsapp (Admin Lambda -- new endpoint)
    |
    +-- [1] Verify webhook signature
    +-- [2] Parse inbound message
    +-- [3] Route to Sherpa agent (MCP tool call)
    |       Sherpa invokes POST /tokens/create via MCP
    +-- [4] Send response back via WhatsApp API
    |       "Token {name} created for user@example.com. 500 credits."
    +-- [5] Audit log
```

### 1.5 Integration with Existing Token Management

The payment pipeline creates tokens programmatically. Key design decisions:

1. **Token naming convention for auto-created tokens.** Suggestion: `pay-{payment_id_short}` where `payment_id_short` is the last 8 characters of the Stripe payment ID. This keeps token names human-readable in the admin UI while being deterministic (idempotent on webhook retry).

2. **Idempotency.** Stripe may retry webhooks. The endpoint MUST be idempotent:
   - Check if a payment record with the same `payment_id` already exists before creating a token.
   - If the payment already has a `token_name`, return 200 without creating a duplicate token.
   - The `Service__Tokens.create()` method already returns `None` for duplicate `token_name`, which provides a safety net.

3. **Token metadata.** The token's `metadata` field should include `{ payment_id, amount, currency, customer_email }`. This allows the admin UI's token-manager component to display payment context alongside token status.

4. **Credit expiry.** The brief mentions 3-month expiry. The current token service has no TTL mechanism -- tokens are active until manually revoked or usage-exhausted. This is flagged as a separate work stream in the v0.7.19 workstream proposal. For Phase 1, tokens have no expiry. Phase 2 should add `expires_at` to the token schema and enforce it in `Service__Tokens.use()`.

### 1.6 Security Analysis

#### Stripe Webhook Signature Verification

**CRITICAL.** Without signature verification, anyone who discovers the webhook URL can forge payment events and create unlimited tokens.

Implementation requirements:

```
1. Store Stripe webhook signing secret in environment variable:
   STRIPE_WEBHOOK_SIGNING_SECRET (never in code)

2. On each request:
   - Read raw request body (bytes, not parsed JSON)
   - Read Stripe-Signature header
   - Compute HMAC-SHA256 of: timestamp + '.' + raw_body
     using the webhook signing secret
   - Compare against signature in header
   - Reject if timestamp is >5 minutes old (replay protection)

3. Use stripe-python library's webhook verification:
   stripe.Webhook.construct_event(payload, sig_header, secret)
```

**Stack rule consideration:** The rule says "never use boto3 directly" and "all AWS calls go through osbot-aws." The Stripe library (`stripe-python`) is a third-party SDK for Stripe, not an AWS service. It is acceptable to use `stripe-python` directly for webhook verification. However, it adds a new dependency to the Admin Lambda (`admin__config.py` `LAMBDA_DEPENDENCIES` list).

**Alternative (no new dependency):** Implement HMAC-SHA256 verification manually using Python's `hmac` and `hashlib` stdlib modules. This avoids adding `stripe-python` to Lambda dependencies. The Stripe webhook signing protocol is well-documented and straightforward.

**Architect recommendation:** Use manual HMAC verification (no `stripe-python` dependency). Reasons:
- Smaller Lambda package (aligns with D6 MVC principle)
- The verification algorithm is simple: split signature, check timestamp, compute HMAC, compare
- Avoids pulling in the full Stripe SDK just for one verification function

#### Notification Channel Security

| Channel | Secret Material | Storage | Risk |
|---------|----------------|---------|------|
| Email (SES/WorkMail) | AWS credentials | IAM role on Lambda | Low -- already in AWS |
| SMS (SNS) | AWS credentials | IAM role on Lambda | Low -- already in AWS |
| WhatsApp (Twilio) | Twilio Account SID + Auth Token | Environment variables | Medium -- new vendor credentials |
| WhatsApp (Meta direct) | App Secret + Access Token | Environment variables | Medium -- new vendor credentials |

**PII in notifications:** The customer email address is transmitted in notification messages. For email and SMS (within AWS), this is acceptable (same trust domain). For WhatsApp, the customer email passes through a third-party provider (Twilio or Meta). The brief accepts this trade-off for Phase 1 (manual human response). For Phase 2+, consider whether the customer email should be replaced with a payment reference code in WhatsApp messages.

#### Webhook Endpoint Exposure

The Admin Lambda is currently auth-protected (API key in headers). The webhook endpoint bypasses this auth. This creates a new attack surface:

- **Mitigation 1:** Stripe signature verification (primary control)
- **Mitigation 2:** Rate limiting on the webhook endpoint (10 requests/minute is generous for a product with low transaction volume)
- **Mitigation 3:** IP allowlisting for Stripe webhook IPs (Stripe publishes their IP ranges; implementable via Lambda@Edge or CloudFront WAF if the Admin Lambda is behind CloudFront)
- **Mitigation 4:** Audit logging of all webhook attempts (successful and failed verification)

**Architect recommendation:** Implement mitigations 1, 4 immediately. Mitigation 2 is low-effort (simple counter in the request handler). Mitigation 3 is deferred -- it requires CloudFront in front of the Admin Lambda, which is a separate infrastructure change.

### 1.7 Future Two-Way WhatsApp Architecture

The brief describes a vision where the human communicates with Sherpa via WhatsApp. This is architecturally significant because it introduces a **second inbound webhook channel** on the Admin Lambda.

Key architectural implications:

1. **Webhook routing.** With both Stripe and WhatsApp webhooks, the route group should be `POST /webhooks/stripe` and `POST /webhooks/whatsapp` (not a single `/webhooks` endpoint with type discrimination). Each has different signature verification, payload format, and processing logic.

2. **Agent invocation pattern.** WhatsApp inbound messages must be routed to Sherpa. Two options:
   - **Direct service call:** The webhook handler directly calls `Service__Tokens.create()` based on parsed intent. Simple but brittle -- every new command requires code changes.
   - **MCP tool invocation:** The webhook handler passes the message to an LLM (via MCP) which decides which tool to invoke. More flexible but introduces LLM latency in the WhatsApp response loop.

   **Architect recommendation:** Start with direct service calls for Phase 3 (limited command set: "create token", "check payments"). Evolve to MCP-mediated invocation when the command vocabulary grows.

3. **SG/Send as the communication channel.** The brief notes that eventually the messaging features of SG/Send itself could replace WhatsApp for agent communication. This is architecturally interesting -- it means the data room messaging feature (PROPOSED, does not exist yet) would need an agent-accessible API. This is a long-term consideration and does not affect the current design.

### 2.2 Three Content Zones -- Architectural Assessment

#### Zone Architecture and Deployment Topology

**Key insight:** Each zone maps to a different deployment target, which is already aligned with the existing infrastructure:

```
Zone 1 (Tools/Product UI)
  +-- send.sgraph.ai (User Lambda -- serves UI + API)
  |     Translation: i18n.js + JSON locale files bundled with UI
  |     Current: 8 languages in v0.2.0
  |     Build: IFD versioned releases
  |

        Translation: Same i18n.js pattern
        Build: Static site deploy script

Zone 2 (Commercial Website)
  +-- sgraph.ai (S3/CloudFront static site)
        Translation: Pre-rendered locale folders via generate_i18n_pages.py
        Current: 18 locale JSON files, en-gb/ folder structure
        Build: Python build script + S3 sync + CloudFront invalidation

Zone 3 (Documentation)
  +-- docs.sgraph.ai (separate static site -- DOES NOT EXIST)
        Translation: English primary, per-demand
        Build: Independent deployment pipeline
```

**Architectural implication:** The zone model does NOT require changes to the deployment topology. Each zone already maps to an existing (or planned) deployment target. The zones formalise what is already implicit in the infrastructure.

**However, the zones do affect the build pipeline.** The `generate_i18n_pages.py` script currently processes all locale JSON files. With zones, the script needs to know which keys belong to which zone, so it can:
- Include all Zone 1 + Zone 2 keys when building the commercial website
- Exclude Zone 3 keys from the website build
- Support Zone 3 builds independently (when a specific doc page gets translated)

### 2.3 Rosetta Archive -- Repo Structure Implications

The brief proposes:

```
rosetta/
  locales/          <-- Translation key-value pairs per locale
  glossary/         <-- Approved terminology per locale
  memory/           <-- Translation memory (corrections over time)
  status.json       <-- Coverage, quality, last updated per locale
```

**Where in the repo does Rosetta live?**

Current translation files:

### 2.4 Decoupled Translation Workflow -- Architectural Impact

The brief's core change: **translation is decoupled from development.**

Current workflow:
```
Developer edits page --> Must update 17 JSON files --> Context window explodes
```

Proposed workflow:
```
Developer edits page --> Adds/updates EN-GB keys in rosetta/locales/en-gb.json only
                     --> Page ships in English immediately
                     --> Translation tasks created asynchronously
                     --> Each task: "translate keys X, Y, Z from EN-GB to PT-PT"
                     --> Translator agent loads only 2 files (en-gb.json + pt-pt.json)
                        or even just the specific keys
```

**Architectural implication for CI/CD:**

The deploy pipeline must handle two scenarios:
1. **English-only deploy:** Developer pushes new page. `generate_i18n_pages.py` runs but the new keys are English-only in non-English locales (fallback to EN-GB). This is acceptable -- the user sees English text for untranslated keys.
2. **Translation deploy:** Translator pushes updated locale JSON. `generate_i18n_pages.py` regenerates all locale folders. Only the translated keys change.

The CI pipeline does not need structural changes. The `generate_i18n_pages.py` script already handles missing keys by falling back to English. The decoupling is a **workflow change**, not an infrastructure change.

**However, the translation task tracking needs a mechanism.** The brief describes decomposed tasks like "translate keys hero.title, hero.subtitle from EN-GB to PT-PT." Where are these tasks tracked?

Options:
1. **Issues FS (`.issues/`)** -- use the existing file-based issue tracker to create per-language translation tasks
2. **`rosetta/status.json`** -- track translation coverage per key per locale, identify missing translations automatically
3. **CI diff detection** -- compare EN-GB keys against each locale, auto-generate translation task list on each build

**Architect recommendation:** Option 2 + Option 3 combined. `rosetta/status.json` is the persistent record of what is translated. A CI step compares the key sets and updates `status.json` on each build, flagging newly added English keys that need translation. No manual task creation needed -- the status file IS the task list.

### 2.6 Cross-Zone Language Navigation

The brief describes visual indicators for cross-zone language transitions (e.g., a Portuguese user clicking a link from sgraph.ai/pt-pt/ to docs.sgraph.ai which is English-only).

**Architectural approach:**

This is a frontend concern, not a backend concern. The implementation belongs in the HTML templates and the `i18n.js` component:

1. **Outbound links from multilingual pages to English-only content:** Add a `data-i18n-zone="3"` attribute to links targeting Zone 3 content. The `i18n.js` runtime appends a language indicator icon when the current locale is not EN-GB.

2. **Inbound banners on English-only pages:** Zone 3 pages include a `<language-notice>` Web Component that shows "This content is available in English only. The main site is available in: [locale flags]."

3. **Language preference persistence:** Already implemented (the i18n.js system stores preference). The brief asks for this to be "persistent and obvious" -- the current implementation should be verified but is not an architectural gap.

### 2.7 Security Considerations for Translation

1. **Translation injection.** Translated strings are inserted into HTML. If a malicious translation contains `<script>` tags or event handlers, it becomes an XSS vector. The `generate_i18n_pages.py` build script MUST sanitise all translated strings (HTML-encode `<`, `>`, `"`, `'`, `&`) before injecting into HTML. The runtime `i18n.js` must use `textContent` (not `innerHTML`) when applying translations.

2. **Key enumeration.** The `rosetta/status.json` file exposes the full key inventory. This is not a security concern -- keys are descriptive labels, not secrets. However, `status.json` should not include PII or confidential business terms in key names.

3. **Translation integrity.** For high-stakes translations (pricing, legal, security descriptions), a human review step is required. The `rosetta/glossary/` files serve as a consistency check but are not a security control. The brief already calls for human review per language.

### 2.9 The Rosetta and Semantic Graph Future

The brief mentions that translations will eventually live in a semantic graph. From the Architect's perspective, the `rosetta/` file structure should be designed for future graph ingestion:

- Each translation key should be a **unique, dot-separated path** (e.g., `hero.title`, `pricing.starter.price`). This maps naturally to graph nodes.
- Each locale file should be a **flat key-value map** (not deeply nested JSON). Flat maps are easier to diff, merge, and ingest into a graph.
- The `glossary/` files establish **semantic relationships** (this term in language A corresponds to this term in language B). These are edges in a future translation graph.

No code changes are needed now -- just maintain the flat key-value convention.
