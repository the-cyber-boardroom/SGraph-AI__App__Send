## Architecture Overview

The vault UI (`vault.sgraph.ai` / `static.sgraph.ai`) is a **pure client-side application** that uses the `send.sgraph.ai` User Lambda as a dumb encrypted storage backend. The server has zero knowledge of vaults, folders, or file names.

```
static.sgraph.ai (S3 + CloudFront)         send.sgraph.ai (User Lambda)
┌─────────────────────────────────┐        ┌──────────────────────────────┐
│  Pure HTML/JS/CSS (no backend)  │        │  Transfer API                │
│                                 │        │  POST /api/transfers/create  │
│  vault-app ← orchestrator      │─fetch──►│  POST /api/transfers/upload  │
│    ├─ vault-entry (auth+open)   │        │  POST /api/transfers/complete│
│    ├─ vault-browser (CRUD)      │◄─fetch──│  GET  /api/transfers/download│
│    ├─ vault-upload (encrypt)    │        │                              │
│    └─ vault-share (key display) │        │  Server sees: encrypted      │
│                                 │        │  blobs with random IDs.      │
│  sg-send-crypto.js (AES+KDF)   │        │  No vault concept. No names. │
│  sg-send.js (API client)       │        │  No folder structure.        │
│  sg-vault.js (vault logic)     │        └──────────────────────────────┘
└─────────────────────────────────┘
```

### Two Separate Vault Systems (Important Context)

There are **two** vault implementations in this codebase:

| | Admin Lambda Vault (v0.1.7) | Pure JS Vault (v0.1.0) — NEW |
|---|---|---|
| **API** | 17 dedicated `/vault/*` endpoints on Admin Lambda | Uses generic `/api/transfers/*` on User Lambda |
| **Encryption** | RSA-4096 + AES-256-GCM hybrid | PBKDF2 → AES-256-GCM (symmetric only) |
| **Server awareness** | Server knows about vaults, folders, files (separate storage entities) | Server has no vault concept — everything is a transfer |
| **Auth** | Cookie-based (Admin Lambda) | Token-based (`x-sgraph-access-token`) |
| **UI** | `sgraph_ai_app_send__ui__admin/` — `vault-manager` component | `sgraph_ai_app_send__ui__vault/` — 5 new Web Components |
| **Hosting** | Served by Admin Lambda | Static site on S3/CloudFront |
| **Status** | Production, tested (71 tests) | Working MVP, live-tested this session |

The new pure JS vault is **intentionally simpler** — it uses the transfer API as a dumb file store and keeps all structure client-side. This is the one being actively developed.

## Data Model: How Vault Data Lives on the Server

```
                          send.sgraph.ai sees:
                          ┌────────────────────────────────┐
   settingsTransferId ──► │ Transfer: random encrypted blob │ ← vault metadata
                          └────────────────────────────────┘
   treeTransferId ──────► │ Transfer: random encrypted blob │ ← folder/file tree
                          └────────────────────────────────┘
   fileTransferId1 ─────► │ Transfer: random encrypted blob │ ← encrypted file
                          └────────────────────────────────┘
   fileTransferId2 ─────► │ Transfer: random encrypted blob │ ← encrypted file
                          └────────────────────────────────┘
```

**Settings** (encrypted JSON):
```json
{
  "vault_name": "My Vault",
  "vault_id": "a3f7b21c",
  "created": "2026-03-02T...",
  "version": 1,
  "_tree_transfer_id": "abc123def456"
}
```

**Tree** (encrypted JSON):
```json
{
  "version": 3,
  "updated": "2026-03-02T...",
  "tree": {
    "/": {
      "type": "folder",
      "children": {
        "Documents": { "type": "folder", "children": { ... } },
        "photo.jpg": { "type": "file", "file_id": "xyz789...", "size": 1234567, "uploaded": "2026-03-02T..." }
      }
    }
  }
}
```

## Cross-Origin Setup (Verified Working)

**User Lambda CORS middleware:**
```python
allow_origins     = ["*"]          # Any origin can call
allow_methods     = ["GET", "POST", "HEAD", "OPTIONS"]
allow_headers     = ["Content-Type", "X-Requested-With", "Origin",
                     "Accept", "Authorization",
                     "x-sgraph-send-access-token"]  # ← vault auth header
allow_credentials = True
```

No CORS changes needed. The vault UI at any origin can call `send.sgraph.ai`.

## Authentication Flow (Added This Session)

```
1. User pastes access key into vault-entry input
2. Key saved to sessionStorage('sg-vault-access-key')
3. On open/create vault:
   a. _requireAccessKey() validates key is present
   b. _getSGSend() creates SGSend({ endpoint, token })
   c. token read from input (fallback: data-token attribute)
4. SGSend._authHeaders() adds: x-sgraph-access-token: {token}
5. Every fetch() call includes the header
```

**Security properties:**
- sessionStorage (not localStorage) — clears when tab closes
- Token never written to URL
- Fallback to `data-token` attribute for embedded/programmatic use

## Open Architecture Questions for Next Session

### 1. Orphaned Transfer Cleanup

Every tree/settings mutation creates new transfers and abandons old ones. The server has no way to know they're orphaned. Options:
- **a)** Accept storage growth (transfers are small, <1KB for tree/settings)
- **b)** Track old transfer IDs in settings and delete them
- **c)** Server-side TTL (but server doesn't know about vaults)
- **Recommendation:** Accept for MVP. Track old IDs in Phase 2.

### 2. Vault Key Stability

The vault key (`passphrase:vaultId:settingsTransferId`) changes after every mutation because `settingsTransferId` changes. Old keys still work (old settings transfer persists), but they show a stale view. Options:
- **a)** Accept this (current behaviour — old links show point-in-time snapshot)
- **b)** Use a stable lookup key (e.g., hash of passphrase+vaultId → always points to latest settings)
- **c)** Add a redirect/alias layer on the server
- **Recommendation:** This is actually a feature — immutable snapshots. Document it. Consider a "latest" pointer in Phase 2.

### 3. Large File Support

Current flow: entire file read into memory → encrypted → uploaded as single transfer. For files >50MB this will fail or be slow. Options:
- **a)** Use presigned multipart uploads (User Lambda has `/api/presigned/initiate`)
- **b)** Client-side chunking with chunk-level encryption
- **c)** Set a file size limit for MVP
- **Recommendation:** Set 50MB limit for MVP. Wire presigned uploads in Phase 2.

### 4. Large Vault Performance

The entire tree is loaded into memory on vault open. For 10,000+ files, this JSON could be several MB. Options:
- **a)** Accept for MVP (most vaults will have <1000 files)
- **b)** Split tree into sub-trees per top-level folder
- **c)** Lazy-load sub-trees on navigation
- **Recommendation:** Accept for MVP. Split tree in Phase 2.

### 5. Migration Path from Admin Vault (v0.1.7) to Pure JS Vault (v0.1.0)

These are fundamentally different architectures. No automatic migration is possible. Options:
- **a)** Manual re-upload (download from admin vault, upload to pure JS vault)
- **b)** Build a migration tool
- **c)** Keep both running in parallel
- **Recommendation:** Keep both running. Build migration tool only if users request it.

    │
    ├── CloudFront distribution

## Summary

The vault architecture is sound and working. All five components function end-to-end. The access key integration is the last piece that was needed to make cross-origin API calls work. The main architectural decisions for the next session are around **scale** (large files, large vaults) and **stability** (vault key immutability vs. mutability). None of these block continued feature work — they're Phase 2 considerations.
