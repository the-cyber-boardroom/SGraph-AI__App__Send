# Workspace DATA Panel — Current Architecture

**Version:** v0.11.21
**Date:** 2026-03-06
**Role:** Architect
**Status:** Code-verified

---

## Overview

The workspace has a four-column layout: **SOURCE | DATA | SCRIPT | RESULT**. The DATA panel is a `<document-viewer data-role="data">` instance that holds structured data files (JSON, CSV, XML, YAML, TSV, NDJSON, JSONL). This document describes how data flows through the system today.

---

## Current Data Flow

```
Vault files (.json/.csv/.xml/...) ──> DATA panel ──> LLM prompt (as context)
                                                          |
                                                          v
                                                    LLM generates script
                                                          |
                                                          v
                                     SOURCE HTML + Script ──> sandboxed iframe ──> RESULT
```

**Key fact:** Scripts cannot access the DATA panel. They only see SOURCE HTML.

## 1. Loading Data INTO the DATA Panel

### From vault (file selection)
When a file is selected in the vault panel, `bundle-manager._trackFileOrigin()` routes it by extension:

```javascript
// bundle-manager.js (lines 542-551)
if (name.endsWith('.js'))           → script-editor
else if (DATA_EXTS.includes(ext))   → document-viewer[data-role="data"]
else                                → document-viewer[data-role="source"]
```

Supported data extensions: `['json', 'csv', 'xml', 'yaml', 'yml', 'tsv', 'ndjson', 'jsonl']`

The data viewer's `_onDataFileSelected()` decrypts from vault and loads:
```javascript
const plaintext = await vault.getFile(data.folderPath, data.name)
this.loadBytes(new Uint8Array(plaintext), data.name, data.type)
```

### From LLM output (copy-to)
LLM orchestrator handles copy-to-data via:
```javascript
if (target === 'data') {
    dataViewer.loadText(content, 'data.json')
}
```

### From bundle restore
When loading a saved bundle, data is restored from `context.data.inline`.

## 2. Script Execution — DATA Is Excluded

### The sandbox boundary
Scripts run in a sandboxed iframe with `sandbox="allow-scripts"` (no `allow-same-origin`):

```javascript
// js-executor.js
iframe.sandbox = 'allow-scripts'
```

### What the iframe receives
Only two inputs:
1. **SOURCE HTML** — injected into `<body>`
2. **User script** — wrapped as IIFE in `<script>`

```javascript
function buildIframeDocument(sourceHtml, userScript) {
    return `<!DOCTYPE html><body>${sourceHtml}<script>...${userScript}</script></body></html>`
}
```

### Script-editor execution flow
```javascript
// script-editor.js (lines 98-162)
async _run() {
    const script    = this.getScript()
    const sourceHtml = sourceViewer.getTextContent()  // Only SOURCE, not DATA
    const result    = await window.sgraphWorkspace.executeJS(sourceHtml, script)
    transformViewer.loadText(result.data, `result.${result.resultType}`)
}
```

**DATA panel content is never passed to `executeJS()`.** The function signature is `executeJS(sourceHtml, userScript, timeoutMs)` — there is no data parameter.

## 3. DATA in LLM Prompts

The LLM orchestrator reads the DATA panel when `incData` flag is checked:

```javascript
// llm-orchestrator.js (lines 86-93)
if (incData) {
    const dataContent = dataViewer.getTextContent()
    if (dataContent) {
        finalPrompt += '\n\n--- DATA ---\n\n' + dataContent
        attachments.data = true
    }
}
```

This means the LLM can "see" the data and generate scripts informed by it, but the script itself runs blind to data content.

---

## 4. Bundle Schema — DATA Persistence

Data is captured and restored as part of the execution bundle:

```javascript
context: {
    source:  { ref: "vault://path/file.html", inline: "<html>..." },
    data:    { ref: "vault://path/data.json",  inline: "{...}" },   // <-- DATA
    script:  { ref: "vault://path/script.js",  inline: "...", filename: "script.js" },
    result:  { inline: "<html>..." },
    console: "...",
    screenshots: []
}
```

Bundles are stored at `/.bundles/{id}.json` in the vault, with `tree.json` as the index.

---

## 5. The Gap — LLM-Free Script Execution

**Current state:** Once a user has iterated with the LLM to build the right script + data model, they should be able to run that script against new data without any LLM calls. But today:

- Scripts can only operate on SOURCE HTML DOM
- DATA panel content has no path into the execution sandbox
- The only consumer of DATA is the LLM prompt builder

This means every time new data arrives, the user must either:
1. Re-invoke the LLM to regenerate a script with embedded data values
2. Manually paste data into the SOURCE HTML
3. Accept that the script cannot use structured data

**This is the architectural gap the proposal document addresses.**
