# Workspace Architecture Evolution: v0.11.10 to v0.11.21

**Version:** v0.11.21
**Date:** 6 March 2026
**Commit:** `63323ed` (Fix bundle restore not applying collapsed panel grid layout)
**Previous Review:** [v0.11.10 — Workspace Architecture and LLM Briefing](../26-03-05/v0.11.10__brief__workspace-architecture-and-llm-briefing.md)
**Scope:** 15 commits across 5 March afternoon and 6 March, covering layout restructuring, execution bundles, and panel collapse

---

## Executive Summary

In 24 hours, SG/Workspace evolved from a 3-column flex/grid hybrid (Source|Script|Result with Source split vertically into Source+Data) into a uniform 4-column CSS grid (Source|Data|Script|Result) with a complete state snapshot system (Execution Bundles) and a centralized panel collapse mechanism. The journey required 9 bug-fix commits to reach stability, each one revealing a deeper architectural lesson about CSS layout authority, state restoration, and the value of event-driven coordination. The final fix -- a single `dispatchEvent` call replacing duplicated grid logic in the bundle manager -- is the cleanest example of the Single Source of Truth principle applied to UI state.

---

## 1. IFD Layer Architecture -- v0.1.0 Frozen, v0.1.1 as Surgical Overlay

The v0.11.10 review documented 19 Web Components and 390 KB of JavaScript in `v0.1.0/`. That code is now frozen. All new behaviour ships in `v0.1.1/`, which contains no copies of v0.1.0 files -- only overrides.

### Loading Strategy

The `v0.1.1/index.html` loads v0.1.0 in full (Layers 0-5), then applies three layers of patches:

```
Layer 6: Bug fixes         (execution-history vault API fix)
Layer 7: Behaviour patches (llm-prompt-input, llm-output, script-editor)
Layer 8: New features      (bundle-manager, bundle-list, workspace-shell override)
```

Every Layer 6-7 script uses prototype patching to modify already-registered custom elements:

```javascript
const ShellClass = customElements.get('workspace-shell');
const _originalRender = ShellClass.prototype.render;

ShellClass.prototype.render = function() {
    _originalRender.call(this);
    injectBundleUI(this);
};
```

This pattern means v0.1.0 is never modified, can be independently tested, and v0.1.1 patches are individually removable. The IIFE scope in each override file prevents variable leakage.

### File Inventory

v0.1.1 adds 8 files (6 overrides + 2 new components + 1 index.html), referencing all 21 v0.1.0 files via relative `../v0.1.0/` paths. Total new code is approximately 45 KB. Combined workspace size is approximately 435 KB.

## 2. Execution Bundles -- Complete State Snapshot System

Commit `a7603f9` introduced the `<bundle-manager>` component -- an invisible orchestrator (no UI of its own) that captures, saves, loads, and tracks workspace state as JSON bundles stored in the vault.

### Bundle Schema

Each bundle is a self-contained JSON document capturing the entire workspace state:

```
{
  id:                "bundle-20260305-143022-a7f2",
  timestamp:         ISO 8601,
  parent_id:         previous bundle ID (enables fork tracking),

  collapsed_sections: { "system-prompt": true, "llm-output": false, ... },
  collapsed_panels:   { "source": false, "data": true, ... },

  config: {
    model, provider, streaming, gen_js_mode,
    include_flags: { src, data, script, result, console }
  },

  context: {
    source:  { ref: "vault:///folder/file.html", inline: "<full content>" },
    data:    { ref: "vault:///data.json",         inline: "..." },
    script:  { ref: "vault:///transform.js",      inline: "...", filename },
    result:  { inline: "..." },
    console: "<console output text>",
    screenshots: [{ type, data: "data:image/png;base64,..." }]
  },

  prompts: { system, user },

  response: {
    text, tokens: { prompt, completion, total },
    cost, duration_ms, finish_reason
  }
}
```

### Key Design Decisions

1. **Inline content, not references only.** Each bundle stores the full text of every panel. This means bundles are self-contained -- they can be loaded even if the original vault files have been modified or deleted. The `ref` field is metadata for provenance tracking, not a load dependency.

2. **Vault-backed storage.** Bundles are stored as encrypted files in `/.bundles/` within the current vault folder. A `tree.json` index maps bundle IDs to metadata (parent_id, timestamp, model, prompt preview, size). This means bundles inherit the vault's zero-knowledge encryption automatically.

3. **Folder-specific bundles.** The `getBundleFolder()` function scopes bundles to the current vault navigation path. Bundles created while browsing `/projects/alpha` are stored in `/projects/alpha/.bundles/`, not a global folder. This was a bug fix in commit `1075f1c`.

4. **Fork tracking via parent_id.** Loading a bundle sets `_activeBundle`. The next save records that ID as `parent_id`, creating an implicit tree of execution states. The `bundle-list` UI could render this as a tree (not yet implemented).

### Time Travel Flow

```
Save:  captureState() -> saveBundle() -> vault.addFile() -> updateTreeIndex()
                                              |
                                        Encrypted & stored
                                         in send.sgraph.ai

Load:  vault.getFile() -> JSON.parse() -> _restoreState() -> panel-by-panel apply
                                               |
                                    dispatches workspace-panels-changed
```

## 3. Four-Column Grid Evolution

### Before: 3-Column with Nested Source Zone (v0.1.0)

```
+--------------------------------------------------+
|                    Header                         |
+------+--------+--+--------+--+--------+--+-------+
| Nav  | Vault  |  | Source |  | Script |  | Debug |
|      |        |  |--------|  |        |  |       |
|      |        |  | Data   |  | Result |  |       |
+------+--------+--+--------+--+--------+--+-------+
|                    Status                         |
+--------------------------------------------------+

Main area grid:
  "source   col-resize  script  col-resize2  transform"
  Source zone internally split via flex into Source (top) + Data (bottom)
```

The Source zone was a flex column containing two `<document-viewer>` instances with a drag handle between them. Script and Result were peer grid cells. This created a **mixed metaphor**: the main area used CSS Grid, but the Source column used flex for its internal split.

### After: 4 Equal Columns (v0.11.21)

```
+------------------------------------------------------------------+
|                           Header                                  |
+------+--------+--+--------+--+--------+--+--------+--+-----------+
| Nav  | Vault  |  | Source |  | Data   |  | Script |  | Result    |
|      |        |  |        |  |        |  |        |  |           |
|      |        |  |        |  |        |  |        |  |           |
+------+--------+--+--------+--+--------+--+--------+--+-----------+
|                           Chat Zone                               |
+------+--------+--+--------+--+--------+--+--------+--+-----------+
|                           Status                                  |
+------------------------------------------------------------------+

grid-template-columns: 1fr 4px 1fr 4px 1fr 4px 1fr
grid-template-areas:
    "source  col-resize1  data  col-resize2  script  col-resize3  result"
    "row-resize ..."
    "chat ..."
```

Commit `1d12d9b` performed the restructuring. The `restructureToFourColumns()` function in the v0.1.1 shell override:

1. Extracts `ws-source-top` and `ws-source-bottom` from their parent `ws-source-zone`
2. Removes the old vertical split handle and 2 column resize handles
3. Creates 3 new column resize handles (`ws-col-resize1`, `ws-col-resize2`, `ws-col-resize3`)
4. Rebuilds the DOM order: source, resize1, data, resize2, script, resize3, result, row-resize, chat
5. Applies the `ws-transform-area--four-col` CSS class

This is a DOM restructuring operation, not a re-render. The v0.1.0 shell renders the original 3-column layout, then the v0.1.1 override immediately restructures it to 4 columns. The original component instances are preserved -- only their DOM parentage changes.

## 4. Panel Collapse System -- The Journey to `applyAllPanelState()`

The panel collapse system went through the most iterations of any feature in this period. The core problem: when a user collapses a panel (e.g., Data), its grid column must shrink to 28px and the remaining panels must absorb the freed space.

### Iteration 1: Individual Inline Styles (commit `df584c3`)

Each panel's click handler directly set its own CSS:

```javascript
panel.style.width = isCollapsed ? '28px' : '';
panel.style.flex = isCollapsed ? '0 0 28px' : '1';
```

**Problem:** Grid columns are controlled by `grid-template-columns` on the parent, not by individual element widths. Setting `width: 28px` on a child of a CSS grid has no effect when the column is defined as `1fr`.

### Iteration 2: Direct Grid Column Manipulation (commit `6c378b2`)

The click handler started setting `grid-template-columns` on the parent area:

```javascript
area.style.gridTemplateColumns = '1fr 4px 28px 4px 1fr 4px 1fr';  // Data collapsed
```

**Problem:** Each panel's handler only knew about its own state. Collapsing Data, then collapsing Source, would overwrite the grid template -- Source's handler didn't know Data was already collapsed.

### Iteration 3: Explicit Pixel Heights / Flex Overrides (commits `9f853a9`, `04ace51`)

Attempted to use `flex: 1` on sibling panels to fill freed space when one collapsed.

**Problem:** The layout was CSS Grid, not flexbox. `flex` properties on grid children are ignored.

### Iteration 4: Centralized `applyAllPanelState()` (commit `9f63076`)

The breakthrough: a single function reads the collapsed state of ALL four panels from localStorage and computes the complete `grid-template-columns` value:

```javascript
function applyAllPanelState(shell) {
    const state = getCollapsedPanels();
    const area  = shell.querySelector('.ws-transform-area');

    // Apply ws-zone--collapsed to all 4 panels uniformly
    for (const def of PANEL_DEFS) {
        const el = shell.querySelector(def.selector);
        if (el) el.classList.toggle('ws-zone--collapsed', !!state[def.id]);
    }

    // Update grid-template-columns based on collapse state
    if (area) {
        const anyCollapsed = PANEL_DEFS.some(d => state[d.id]);
        if (anyCollapsed) {
            const widths = PANEL_DEFS.map(d => state[d.id] ? '28px' : '1fr');
            area.style.gridTemplateColumns = widths.join(' 4px ');
        } else {
            area.style.gridTemplateColumns = '';
        }
    }
}
```

**This works because:**
- It reads ALL panel states, not just the one that changed
- It computes the COMPLETE grid template, not a partial override
- It clears the override entirely when no panels are collapsed (falling back to the CSS class default)
- Every panel click calls the same function

## 5. Event-Driven State Restoration -- The Final Fix

### The Bug (commit `63323ed`)

When loading an execution bundle that had collapsed panels, the bundle manager's `_restoreState()` method would:

1. Write the collapsed panel state to localStorage
2. Apply `ws-zone--collapsed` CSS classes to individual panels
3. But NOT update `grid-template-columns` on the parent area

The panels would show their collapsed visual state (vertical text, hidden content) but remain at `1fr` width because the grid template was unchanged.

### The Wrong Fix (attempted)

Duplicating the grid-template-columns logic from `applyAllPanelState()` into `_restoreState()`:

```javascript
// BAD: duplicated layout logic in bundle-manager
const widths = PANEL_DEFS.map(d => state[d.id] ? '28px' : '1fr');
area.style.gridTemplateColumns = widths.join(' 4px ');
```

This would work but violates the Single Source of Truth principle. Now two places compute grid columns, and they must stay in sync.

### The Right Fix (shipped)

The bundle manager dispatches a `workspace-panels-changed` custom event after writing to localStorage. The shell override already listens for this event:

```javascript
// In bundle-manager _restoreState():
if (bundle.collapsed_panels) {
    try {
        localStorage.setItem('sgraph-workspace-panel-collapsed',
            JSON.stringify(bundle.collapsed_panels));
    } catch (_) {}
    document.dispatchEvent(new CustomEvent('workspace-panels-changed'));
}

// In workspace-shell override setupPanelCollapse():
document.addEventListener('workspace-panels-changed',
    () => applyAllPanelState(shell));
```

The bundle manager's only responsibility is writing state. The shell's only responsibility is reading state and applying layout. The event is the contract between them.

### Why This Pattern Matters

This is the same EventBus pattern used throughout the workspace (Section 3.3 of the v0.11.10 review), but applied to DOM-level layout coordination. It demonstrates that the event-driven architecture scales from component communication (EventBus topics like `file-selected`) down to CSS layout orchestration (`workspace-panels-changed`).

## 6. CSS Architecture Lessons

Six lessons emerged from the 9 bug-fix commits:

### 6.1 Grid Children Cannot Override Grid Columns

Setting `width`, `min-width`, or `flex` on a child of a CSS Grid container does NOT change the column size when the column is defined with `fr` units or explicit values in `grid-template-columns`. The only way to change column size is to change `grid-template-columns` on the parent.

### 6.2 `!important` Creates Invisible Conflicts

Commit `2b8d23c` fixed collapsed panels not shrinking because the v0.1.0 CSS had:

```css
.ws-transform-area {
    grid-template-columns: 2fr 4px 1.5fr 4px 1.5fr !important;
}
```

The `!important` on the base class prevented the v0.1.1 override's inline style from taking effect. The fix was removing `!important` from the base CSS. Lesson: `!important` in base styles prevents all future overrides, including inline styles.

### 6.3 Single Source of Truth for Layout

When multiple components can change layout (panel click handlers, resize drag handlers, bundle restore, chat zone toggle), there must be exactly ONE function that computes the final layout value. All callers must go through that function. `applyAllPanelState()` is that function for the 4-column grid.

### 6.4 State in localStorage, Layout from State

The pattern that emerged:
1. User action (click, drag, bundle load) writes to localStorage
2. A single function reads localStorage and applies CSS
3. The function is idempotent -- calling it twice produces the same result

This means the layout can be reconstructed at any time from localStorage alone, which is exactly what page reload needs.

### 6.5 Flex vs Grid -- Do Not Mix Metaphors

The original Source zone used flex (vertical split) inside a grid cell. This worked but created confusion when collapse behaviour needed to span both systems. The 4-column grid eliminated the flex split entirely, making all four panels peer grid cells with identical collapse behaviour.

### 6.6 Events Beat Direct Coupling

When component A needs component B to update its layout, A should emit an event, not call B's internal methods. This keeps A and B independently testable and means the coupling can be traced by searching for event names rather than method calls.

## 9. Lessons Learned

1. **Prototype patching is a viable IFD strategy.** The v0.1.1 overlay adds significant functionality (bundles, 4-column layout, panel collapse) without modifying a single v0.1.0 file. The cost is careful loading order and IIFE discipline.

2. **CSS Grid demands centralized control.** Unlike flexbox where children can influence their own size, CSS Grid column sizes are dictated entirely by `grid-template-columns` on the parent. Any feature that changes column sizes must go through a single computation function.

3. **Bug-fix sequences reveal architecture.** The 9 commits from `df584c3` to `63323ed` are not wasted work -- they are the exploration that discovered `applyAllPanelState()` and the event dispatch pattern. Each "wrong" approach clarified exactly why the right approach works.

4. **Events are the correct coupling mechanism for cross-component layout.** The bundle manager should not know how panel collapse is implemented. It only knows that writing to localStorage and dispatching `workspace-panels-changed` will cause the layout to update. This is dependency inversion applied to UI layout.

5. **Bundles are more than undo.** With `parent_id` tracking, bundles form a tree of execution states. This is the foundation for experiment branching, A/B prompt comparison, and audit trails -- all stored encrypted in the vault.

## 10. What's Next

1. **Bundle tree visualisation.** The `bundle-list` component currently shows a flat reverse-chronological list. The `parent_id` field enables a tree or timeline view showing fork points.

2. **Auto-save bundles.** The `_autoSave` flag exists but defaults to `false`. Enabling it on `llm-request-complete` would create an automatic execution log.

3. **Bundle diffing.** Comparing two bundles to see what changed (prompt, source, result) between executions.

4. **Column resize persistence.** The 4-column resize handlers update `grid-template-columns` during drag but don't persist the widths to localStorage. Page reload resets to equal widths.

5. **Villager hardening.** The panel collapse system is stable but the CSS could be simplified. The `ws-zone--collapsed` class and the `grid-template-columns` override could potentially be unified into a single CSS mechanism using CSS container queries or `subgrid`.
