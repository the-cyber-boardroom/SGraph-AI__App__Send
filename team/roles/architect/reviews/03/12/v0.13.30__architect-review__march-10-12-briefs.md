# v0.13.30 Architect Review: March 10-12 Briefs

**version** v0.13.30
**date** 12 Mar 2026
**role** Architect
**scope** 13 briefs from 10-12 March 2026

---

## Executive Summary

These 13 briefs represent a significant expansion of the SG/Send vision from encrypted file transfer toward a full key-based encrypted infrastructure platform. The core technical themes are: (1) vault maturation into a Git-like version control system with branches, signed commits, and remotes, (2) infrastructure services (LLM appliances, deploy service, Lambda logging) all built on the vault primitive, and (3) a brand evolution from SG/Send to SPKI reflecting this platform shift. The ambition is high but the dependency chains are deep -- most briefs depend on vault capabilities (branches, signed commits, unified format) that do not exist yet. Sequencing matters enormously.

## Brief-by-Brief Assessment

### 1. Marketplace Appliances (v0.13.12, 10 Mar)

**What it proposes:** Two standalone appliances -- SG/Send Ephemeral Transfer (in-memory file sharing container) and SG/Send Ollama (self-hosted LLM) -- distributed via Docker Hub, PyPI, and AWS Marketplace AMIs.

**Feasibility: HIGH for Ephemeral Transfer, MEDIUM for Ollama.**

The Ephemeral Transfer appliance is largely assembly work. The in-memory storage mode EXISTS (used by all tests). The FastAPI backend EXISTS. The static UI EXISTS. The main engineering is bundling static assets into the container and implementing the two auth modes (open + API key). This is achievable in a focused sprint.

The Ollama appliance requires a FastAPI wrapper around Ollama with OpenRouter-compatible API translation. This is new code but architecturally straightforward. The harder part is the AMI baking pipeline and model pre-loading, which depends on the Deploy Service (brief #2).

AWS Marketplace listing is an operational task (seller account, security scan, review) with 3-5 business day lead times. Should be started early.

**Risks:**
- Docker image size for Ollama bundles (3-20GB) may cause distribution challenges
- GPU instance availability for AMI baking in CI
- AWS Marketplace approval timeline is external and unpredictable

**Dependencies:** Deploy Service brief (for boot-on-demand pattern and AMI baking). LLM Service brief (for the Ollama wrapper).

**Status against reality document:** The in-memory storage mode and FastAPI backend EXIST. Everything else in this brief is PROPOSED -- does not exist yet.

### 2. LLM Service (v0.13.12, 10 Mar)

**What it proposes:** Ollama wrapped in a FastAPI with 100% OpenRouter API compatibility, deployed as an ephemeral EC2 instance via the boot-on-demand pattern.

**Feasibility: MEDIUM.**

The OpenRouter-to-Ollama translation layer is well-specified and technically sound. The SSE format translation is the key engineering challenge but is bounded. The five-environment model (laptop no creds, laptop with Ollama, CI no creds, CI with creds, Lambda+EC2) is well thought out and aligns with existing test patterns.

The boot-on-demand pattern (Router Lambda + ephemeral EC2 + vault state) is PROPOSED -- does not exist yet. This is the highest-risk component. It requires: Router Lambda, EC2 lifecycle management via osbot-aws, Route53 dynamic DNS, vault state read/write, auto-shutdown logic. Each is individually feasible but the integration surface is large.

**Risks:**
- Cold start time target (120s) depends on AMI pre-loading working reliably
- GPU spot instance pricing and availability can be volatile
- The Router Lambda pattern is entirely new infrastructure with no existing implementation

**Dependencies:** osbot-aws EC2 and Route53 capabilities, vault write capabilities for state storage, Deploy Service for AMI baking.

**Status against reality document:** The Workspace LlmOrchestrator EXISTS with OpenRouter and Ollama support. Everything in this brief (FastAPI wrapper, Router Lambda, boot-on-demand, AMI baking) is PROPOSED -- does not exist yet.

### 3. Encrypt for the Reader (v0.13.29, 11 Mar)

**What it proposes:** An architectural paradigm shift from "encrypt for storage" to "encrypt for the reader" -- data encrypted with the intended reader's public key at the point of creation.

**Feasibility: HIGH (as a principle), MEDIUM (as implementation).**

This is primarily a conceptual brief that formalises what SG/Send already does for file transfers (server stores ciphertext, key stays with sender). The extension to multi-level encryption (different log levels encrypted for different teams' public keys) is sound but requires:
- A robust PKI key registry (EXISTS -- Admin Lambda has keys endpoints)
- Multi-recipient encryption in a single operation (PROPOSED)
- Key management for teams/roles (PROPOSED)

The article/marketing angle is straightforward and could proceed independently of implementation.

**Risks:**
- Multi-recipient encryption adds complexity to the encryption pipeline
- Key distribution for teams is an unsolved UX problem

**Dependencies:** PKI key registry (EXISTS), Lambda vault logging (brief #4), document signing (brief #5).

**Status against reality document:** The basic encrypt-for-reader pattern EXISTS in SG/Send file transfers. Multi-level encryption for logs and team-based key targeting is PROPOSED -- does not exist yet.

### 4. Lambda Vault Logging (v0.13.29, 11 Mar)

**What it proposes:** Replace CloudWatch logging with vault-based logging. Lambda functions capture invocation data, encrypt it with the reader's public key, and commit to a vault. Two modes: daily vault (commit per invocation) or execution bundles (one S3 object per invocation).

**Feasibility: MEDIUM.**

Mode 2 (execution bundles) is simpler and more feasible as a first step -- it is essentially encrypt + S3 PUT per invocation, avoiding concurrency concerns entirely. Mode 1 (daily vault with commits) requires the vault to support concurrent commits without conflicts, which depends on the branch model (brief #8).

The headless CLI mode is a prerequisite. Current `sg-send-cli` is designed for interactive use. Programmatic mode (no prompts, exit codes, library-importable) is a significant refactor of the CLI.

The sub-200ms latency target for commit + push is achievable for Mode 2 (single S3 PUT) but questionable for Mode 1 (tree update + blob write + push = at least 2-3 S3 operations).

**Risks:**
- Lambda cold start already adds latency; vault logging adds more
- Concurrent Lambda invocations writing to the same daily vault creates race conditions without branches
- S3 PUT latency in the Lambda execution context varies (p99 can be 200ms+)

**Dependencies:** Headless CLI mode, vault commits (EXIST per reality doc references to "commits"), branch model (for Mode 1 concurrency).

**Status against reality document:** `sg-send-cli` v0.5.0 EXISTS with clone/pull/push/status/init. Headless mode, Lambda identity, vault logging are all PROPOSED -- does not exist yet.

### 5. PKI Document Identity (v0.13.29, 11 Mar)

**What it proposes:** Replace URI-based identity with cryptographic identity -- documents identified by signatures rather than locations. Vault tree entries include signature metadata. "Clue-based resolution" instead of URI resolution.

**Feasibility: LOW-MEDIUM (full vision), HIGH (initial implementation).**

The philosophical framework is compelling but the full vision (paragraph-level identities, signed assertions within documents) is a multi-year research project. The practical starting point is much simpler: add document signing to the CLI (sign a file, verify a signature, store signature metadata in the vault tree).

The existing PKI infrastructure (keys/publish, keys/lookup, keys/list on Admin Lambda) provides the foundation. Adding `sg-send-cli pki sign/verify` and storing signature metadata in the vault tree is feasible.

**Risks:**
- Scope creep -- the brief describes a vision much larger than initial implementation
- Signature metadata in vault trees changes the tree format, which must be coordinated with the bare vault unification (brief #7)

**Dependencies:** CLI PKI features (PROPOSED per reality doc), vault tree format (needs coordination with brief #7).

**Status against reality document:** PKI key endpoints EXIST (publish, lookup, unpublish, list, log on Admin Lambda). CLI PKI features (generate, sign, verify, encrypt) are PROPOSED -- do not exist yet.

### 6. Vault UI Components (v0.13.29, 11 Mar)

**What it proposes:** New `<sg-vault>` Web Component as an abstract data layer, plus `<sg-tree-view>`, `<sg-file-viewer>`, and `<sg-pack-theme>` display components. Pack architecture with `_pack.json` manifests. CRM-through-vaults for contact management.

**Feasibility: MEDIUM.**

The component architecture (vault as data layer, display components consuming events) is sound and follows established Web Component patterns. The IFD methodology already used in the project supports this component-based approach.

The existing `user-vault` component (v0.1.7) handles vault operations but is tightly coupled. Refactoring into the proposed separation of concerns is a non-trivial rewrite.

The CRM-through-vaults concept is interesting but is a separate product built on top of the vault primitive. It should not be conflated with the component refactor.

**Risks:**
- The refactor from monolithic `user-vault` to `<sg-vault>` + display components is a significant UI rewrite
- Pack manifest format (`_pack.json`) needs to be designed before any pack types can be built
- CRM adds product scope that may distract from core vault maturation

**Dependencies:** Vault tree structure (needs to support `_pack.json`), vault UI (EXISTS at v0.1.7 but needs refactoring).

**Status against reality document:** `user-vault` Web Component EXISTS (v0.1.7). `vault-panel` EXISTS in Workspace (v0.1.0). The proposed `<sg-vault>`, `<sg-tree-view>`, `<sg-pack-theme>` components, pack manifest format, and CRM features are all PROPOSED -- do not exist yet.

### 7. Bare Vault Unification (v0.13.30, 12 Mar)

**What it proposes:** Unify server-side and clone-side vault data structures into one format. Security audit of both structures. Nested vaults (vault-inside-vault). SQLite as optional local storage backend. Version field in vault format.

**Feasibility: HIGH (unification and audit), MEDIUM (nested vaults), LOW priority (SQLite).**

This is the most architecturally important brief of the batch. Unifying the two vault representations simplifies everything downstream: clone becomes a zip download, push becomes the reverse, one codebase handles both representations. This should be done before branches, signed commits, or any other vault format changes.

The security audit is essential and should be the first deliverable -- a field-by-field map of both structures identifying any unencrypted sensitive data. This is a bounded, high-value task.

Nested vaults are conceptually clean but add complexity to the CLI and UI. Defer until the base format is stable.

SQLite is correctly identified as a future optimization. Not needed now.

**Risks:**
- Format changes affect every component that reads/writes vaults (CLI, UI, backend)
- Must be done before branching and signing changes to avoid double migration
- Pre-launch status means no backwards compatibility needed (confirmed in brief) -- but this window closes at launch

**Dependencies:** None -- this is foundational. Must be done first.

**Status against reality document:** Vault storage EXISTS on both server (S3 blobs) and client (CLI clone). The unified format, nested vaults, SQLite backend, and version field are all PROPOSED -- do not exist yet.

### 8. Branch Model and Multi-User Collaboration (v0.13.30, 12 Mar)

**What it proposes:** Git-style branching for vaults. Every clone creates a branch with its own key pair. All commits go to branches, never directly to main. Merges are the only way to integrate. Signed commits mandatory. Visualisation tools (CLI graph, Mermaid, Web Component).

**Feasibility: MEDIUM-HIGH (basic branching), LOW (full vision).**

The branch-per-clone model elegantly solves the current concurrency problem (two users pushing simultaneously). The core data model (branch = pointer + key pair, commit = signed tree hash + parent) is sound and well-understood from Git.

The implementation gap is significant. Per the brief's own checklist: content-addressed storage and tree structure EXIST, commits EXIST, but branches, merge, signed commits, remotes, diff, and log all need to be built. That is 6 features, 2 of which (branches + merge) are substantial.

The mandatory signed commits requirement means every branch creation generates a key pair. Key generation performance and key storage (the brief lists 4 possible locations) need to be resolved.

**Risks:**
- Merge conflict resolution is the hardest part of any VCS -- even "simple" automatic merge for non-conflicting changes requires careful tree diffing
- Mandatory PKI per branch means key management becomes a user-facing concern
- The visualisation components (CLI graph, Mermaid, Web Component) are each non-trivial and should be treated as separate features

**Dependencies:** Bare vault unification (brief #7) MUST come first. CLI standalone with remotes (brief #9). PKI document identity (brief #5) for signed commits.

**Status against reality document:** Vault commits are referenced in milestones. Branches, merge, signed commits, diff, log are all PROPOSED -- do not exist yet. The brief itself correctly labels the status of each feature.

### 9. CLI as Standalone with Remotes (v0.13.30, 12 Mar)

**What it proposes:** Invert the CLI mental model -- CLI is the vault tool, SG/Send is one storage backend. Git-style remotes (add/remove/list). Support for local folders, zip files, S3 buckets, URLs as remote types. Full offline mode. Vault export as distributable zip.

**Feasibility: MEDIUM.**

The abstraction of the remote backend is the key architectural challenge. Currently, `sg-send-cli` is tightly coupled to the SG/Send API. Factoring out the API client into a pluggable backend requires a clean interface definition: what operations does a remote need to support? (list, read, write, delete at minimum.)

The offline mode claim is plausible -- vault operations (create, add, commit) are essentially local filesystem + encryption operations. The research task (which operations are server-dependent vs. purely local) should be the first deliverable.

The zip distribution model (vault as a file that travels via email/WhatsApp) is a powerful concept that differentiates from server-dependent alternatives.

**Risks:**
- Remote backend abstraction is a significant refactor of sg-send-cli
- Multiple remote types means multiple integration surfaces to test
- S3-compatible remote requires aws credentials handling in the CLI

**Dependencies:** Bare vault unification (brief #7) -- the remote abstraction is easier when there is one vault format. Branch model (brief #8) -- remotes need to understand branches.

**Status against reality document:** `sg-send-cli` v0.5.0 EXISTS with clone/pull/push/status/init against SG/Send API only. Remotes, offline mode, multiple backend types, zip export are all PROPOSED -- do not exist yet.

### 10. Vault Access from Claude (v0.13.30, 12 Mar)

**What it proposes:** A `/api/vault/download/{vault_id}` endpoint that returns the entire vault as a single downloadable bundle, enabling Claude sessions to fetch vaults via `web_fetch` without network allowlist configuration.

**Feasibility: HIGH.**

This is a well-scoped, practical feature. The endpoint packages all vault blobs into a single downloadable bundle. The CLI already handles vault extraction. The workflow (human pastes URL, Claude fetches, CLI extracts and decrypts) is clean.

This is closely related to the vault bundle endpoint proposed in v0.13.19 (`GET /api/vault/bundle/{vault_id}/{bundle_file_id}`) which is already documented in the reality document as PROPOSED. The two should be unified.

**Risks:**
- Bundle size limits (Lambda response size caps at ~6MB for direct responses; larger vaults need presigned S3 URLs)
- Access control for the download endpoint needs careful design (download tokens vs. public flag)
- Read-only access (no write-back without egress allowlist) limits utility

**Dependencies:** Bare vault unification (brief #7) -- the bundle format should match the unified vault format. Otherwise minimal dependencies.

**Status against reality document:** Vault Pointer endpoints EXIST (read, write, delete, read-base64). The bundle/download endpoint is PROPOSED -- does not exist yet (also noted as PROPOSED from v0.13.19).

- "Launch communication to 40+ users" implies a marketing-ready product, which may not be realistic given how much vault infrastructure is still PROPOSED

**Dependencies:** Vault UI components (brief #6) for demo vaults. Billing automation (PROPOSED). Usage tracking infrastructure.

**Status against reality document:** Vault endpoints EXIST on both Lambdas. Vault UI EXISTS (v0.1.7). The vault website, demo vaults, billable units, usage tracking, and launch communication are all PROPOSED -- do not exist yet. ### 12. Portugal Strategy, Gallery, and Registry (v0.13.30, 12 Mar)

**What it proposes:** gallery.sgraph.ai (vault-backed photo/document galleries with public/private access), Portuguese content strategy, and a public key registry for Portugal (pt.registry.spki.ai).

**Feasibility: MEDIUM (gallery), HIGH (content), LOW (registry).**

The gallery concept is architecturally clean -- a presentation layer over the existing vault API with a `_gallery.json` manifest (parallels `_pack.json` from brief #6). Public galleries with decryption key in URL reuse the existing transfer pattern.

The Portuguese content strategy is marketing work that can proceed independently of technical development.

The public key registry is a new product surface requiring: web scraping infrastructure, key verification workflow, signed attestation pipeline, and a browsable directory UI. This is significant scope that should be treated as its own project.

**Risks:**
- Gallery is yet another product surface (alongside send, vault, workspace, gallery) increasing operational complexity
- The public key registry has legal implications (scraping and publishing third-party public keys, even if publicly available)
- Brand fragmentation risk: gallery.sgraph.ai vs. future gallery.spki.ai

**Dependencies:** Vault API (EXISTS), pack/gallery manifest format (needs design alongside brief #6), self-hosted LLM (brief #2) for Portuguese content generation.

**Status against reality document:** The vault API EXISTS. gallery.sgraph.ai, the public key registry, and all Portuguese-specific content are PROPOSED -- do not exist yet.

## Key Decisions Required

### 1. Vault Format Freeze Point

**Decision needed:** When does the vault format become stable? The brief says "break freely" now, but every component (CLI, UI, backend, Lambda logging) builds on this format. A format freeze date -- even if informal -- prevents cascading rework.

**Recommendation:** Complete brief #7 (unification) and brief #8 (basic branches) before declaring format v1.0.

### 2. Branch Key Storage Default

**Decision needed:** Brief #8 lists four possible key storage locations (vault, browser localStorage, OS keychain, meta-vault). The CLI needs a default before shipping.

**Recommendation:** Start with local filesystem (`.sg-vault/keys/` inside the clone directory). Add OS keychain integration later.

### 3. Billing Granularity

**Decision needed:** Brief #11 proposes metering pulls, pushes, commits, branches, and storage. This is fine-grained metering that requires instrumentation across every endpoint.

**Recommendation:** Start with two billable units only: storage (GB) and API calls (aggregate count). Add granular metering later when usage patterns are understood.

### 4. Gallery vs. Vault Website Priority

**Decision needed:** Brief #11 (vault website) and brief #12 (gallery) both propose new product surfaces. Both use vaults as the backend. Building both simultaneously fragments engineering focus.

**Recommendation:** Build the vault website first. Gallery is a specialisation of vault that can come later.

### 5. CLI Standalone Scope

**Decision needed:** Brief #9 proposes full offline operation including branch and merge. This means the merge algorithm must be implemented in the CLI (Python), not just on the server.

**Recommendation:** Implement merge on the server first (simpler, one codebase). Offline merge is a Phase 3 capability.

## Risks

### 1. Scope Explosion (CRITICAL)

Thirteen briefs in three days, spanning infrastructure (LLM, deploy), vault evolution (branches, signing, unification), new product surfaces (gallery, registry, vault website), billing, branding, and market strategy. The total scope exceeds what can be built in a quarter. Without aggressive prioritisation, nothing ships.

**Mitigation:** Follow the phased implementation order above. Each phase delivers usable value. Resist starting Phase 4 before Phase 2 is solid.

### 2. Vault Format Instability (HIGH)

Five briefs (#5, #7, #8, #9, #12) propose changes to the vault format or tree structure. If these are implemented independently without a unified design, the format will fragment and require multiple migrations.

**Mitigation:** Brief #7 (bare vault unification) must complete first, producing a single vault format specification. All subsequent format changes (branches, signatures, pack manifests, gallery manifests) must be additive extensions to this base format, reviewed by the Architect before implementation.

### 3. LLM/Deploy Service Complexity (HIGH)

Briefs #1 and #2 propose an entirely new infrastructure layer (Router Lambda, boot-on-demand EC2, AMI baking pipeline, vault state management). This is a significant system with its own failure modes, operational concerns, and cost implications. Building it alongside the vault evolution stretches engineering bandwidth.

**Mitigation:** Treat LLM/Deploy as an independent track with its own cadence. Start with the Ephemeral Transfer container (assembly work, high feasibility) as a quick win. The full boot-on-demand pattern can follow.

### 4. Pre-Launch Feature Creep (MEDIUM)

The vault is not yet launched to users (brief #11 proposes a launch). Adding branches, signed commits, galleries, billing, and CRM before launch creates a "launch everything at once" risk where nothing is polished.

**Mitigation:** Launch the vault in its current form (it works, it is tested, Claude uses it). Add features post-launch based on user feedback rather than building everything first.

### 5. Brand Transition Timing (LOW)

Brief #13 correctly says "no timelines." But briefs #12 (pt.registry.spki.ai) and the sub-domain structure (send.spki.ai, vault.spki.ai) imply SPKI domains are needed soon. If the brand transition accelerates, it adds DNS, certificate, and redirect complexity during a high-development period.

**Mitigation:** Keep all new development on sgraph.ai domains. Configure spki.ai domains only when the vault website (brief #11) is ready for public launch.
