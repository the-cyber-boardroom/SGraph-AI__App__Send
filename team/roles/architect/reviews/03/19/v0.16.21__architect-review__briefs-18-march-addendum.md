## Executive Summary

Three documents covering a new deploy tool (Docker LLM chat) and Type_Safe compliance for the deploy codebase. The architectural decisions in the Docker chat brief are sound. The Type_Safe findings expose structural weaknesses in sg_send_deploy that could propagate if patterns are copied into SGraph Send.

---

## Architectural Assessments

### 1. Local LLM Docker Chat — Architecture Review

**Verdict: Sound**

**Key decisions assessed:**

| Decision | Assessment |
|----------|------------|
| Ollama on host, not in Docker | **Correct** — no GPU passthrough on macOS Docker. Container-to-host via `host.docker.internal` is the standard pattern. |
| FastAPI proxy (not direct Ollama access from browser) | **Correct** — CORS handled server-side, session management centralised, API contract abstracted from Ollama specifics. |
| sg-layout bundled for offline | **Correct** — CDN dependency at runtime would break the primary use case (offline during travel). |
| JSON files on volume mount (no database) | **Correct for single-user local tool** — would NOT be correct for multi-user or production. Design is appropriately scoped. |
| Vanilla Web Components | **Correct** — consistent with the rest of the SGraph ecosystem. |
| Same API contract as EC2 LLM brief | **Sound and forward-looking** — the `/api/chat` + `/api/models` contract enables backend swapping between local Ollama and remote EC2 tunnel. |

**Architectural concerns:**

1. **Streaming error propagation** — The brief shows a happy-path streaming implementation. What happens when Ollama returns an error mid-stream (model not loaded, OOM, timeout)? The `ReadableStream` consumer needs error boundaries. Not a blocker — just needs handling during implementation.

2. **Session storage schema evolution** — JSON files have no schema versioning. If the session format changes, old files break silently. Recommendation: add a `schema_version` field to session JSON from day one.

3. **sg-layout dependency coupling** — Bundling sg-layout files into the Docker image creates a version coupling. When sg-layout is updated in SGraph-AI__Tools, the Docker image must be rebuilt with the new files. Consider a `make update-sg-layout` script that copies the latest files from the Tools repo.

### 2. Type_Safe Code Review — Architectural Implications

**Verdict: Critical findings — pattern contamination risk**

The review reveals that 11 of 17 modules in sg_send_deploy are plain classes with no Type_Safe inheritance. Two files have `__init__` overrides that completely bypass Type_Safe. No method in the entire codebase uses the `@type_safe` decorator.

**Why this matters for SGraph Send:**

1. **Pattern copying** — Developers (human or AI) working on sg_send_deploy see plain-class patterns and may replicate them in SGraph Send. The sg_send_deploy codebase should be brought to the same standard as SGraph Send to prevent pattern contamination.

2. **Shared type definitions** — The review proposes types like `EC2__Instance_Id`, `Enum__EC2__Instance_Type`, `Safe_UInt__Port` that could be shared across codebases. These should eventually live in osbot-utils or a shared types package, not be duplicated.

3. **The `__init__` bypass is a data integrity issue** — Fields set in `__init__` before `super().__init__()` are invisible to Type_Safe. They won't appear in `json()` output, won't survive `from_json()` round-trips, and receive no validation. This is not just a style issue — it's a correctness bug.

**Architectural recommendation:** Treat the Type_Safe remediation as **infrastructure debt**, not feature work. Phase 1 + Phase 2 from the code review should be completed before any new feature development in sg_send_deploy.

### 3. Type_Safe Reference Guide — Assessment

**Verdict: Valuable reference, well-structured**

This is not an architecture document — it's a developer reference. The architecture-relevant content:

- **Rule 5 (schemas are pure data)** — aligns with SGraph Send's separation of schemas from services
- **Section 13 Mistake 1 (wrong Safe_Str type)** — critical for anyone choosing types. `Safe_Str__Id` silently strips `/`, `:`, `+` — this would corrupt MIME types, URL paths, and ISO timestamps
- **The three-class architecture** (Type_Safe__Base → Type_Safe__Primitive → Type_Safe) — useful mental model for understanding inheritance

**Recommendation:** Link this document from `library/guides/development/` as the canonical Type_Safe reference for the project ecosystem.
