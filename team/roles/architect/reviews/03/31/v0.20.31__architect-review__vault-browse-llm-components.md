## 2. Vault-Browse Unification Assessment

### 2.1 Architectural Soundness

The unification proposal is **architecturally sound and strategically important**. The core insight — that vault.sgraph.ai and send.sgraph.ai/browse do the same thing with different data sources — is correct, and the proposed solution follows the **dependency injection** pattern cleanly.

**The proposed data source interface:**
```
Browse Module
  ├── Data Source: Zip (download full zip, extract client-side, build tree)
  └── Data Source: Vault API (fetch tree, fetch files on demand)
```

Both sources must implement the same interface: `getTree()`, `getFile(path)`, optionally `writeFile(path, blob)`. This is a well-understood pattern and compatible with the existing `send-browse.js` architecture.

### 2.2 Critical Architectural Questions

**Q1: Where does Browse live after extraction?**

The brief says "Browse module extracted as a shared, importable component." This implies hosting on `tools.sgraph.ai` as an IFD-versioned component (following the `sg-layout`, `sg-tree` pattern). The Browse module is currently embedded in SG/Send v0.3.1. Extraction requires:
- Identifying all Browse-specific logic vs Send-specific logic
- Defining the data source interface formally
- Creating v0.1.0 of `sg-browse` at `tools.sgraph.ai/components/sg-browse/v0/v0.1/v0.1.0/`

This is a significant extraction effort. Send's Browse is ~2,000+ lines across multiple files (`send-browse.js`, `send-browse-v031--page-layout.js`, the PLR module). The extraction must not regress any existing functionality.

**Q2: Fractal vaults require vault-in-vault navigation**

The fractal vault proposal (vault containing references to child vaults, clicking a child opens it in the same Browse interface) is elegant but has a **hidden complexity**: how does the vault tree know that a file is a vault reference, not just an opaque encrypted blob? There needs to be a vault reference format — likely a special file type (`__vault_ref__.json` with a vault token) that the Browse module recognises and treats as a navigable node, not a downloadable file.

**Recommendation:** Define the vault reference format before implementing fractal navigation.

**Q3: Write capabilities in the unified Browse**

The vault data source supports write operations (edit, commit). The current Browse module is read-only. The proposal says vault-specific code adds write operations. This is fine for Phase 1, but the long-term question is: should write operations be a Browse extension point, or always stay in a vault-specific overlay?

**Recommendation:** Keep Browse read-only in v0.1.0. Write capabilities go in the vault adapter as an extension, using the existing IFD surgical override pattern.

### 2.3 Architecture Decision

**APPROVE Phase 1 extraction.** The vault-browse unification is a high-value architectural refactoring that eliminates code divergence and enables fractal vaults. The data source adapter pattern is clean, proven, and aligns with the platform's component philosophy.

**Phased approach recommended:**
1. **Phase 1:** Extract Browse as `sg-browse` component. Define data source interface. Zip adapter only (no regression).
2. **Phase 2:** Vault API adapter. Test against vault website.
3. **Phase 3:** Fractal vault support (requires vault reference format definition).

## 3. sg-llm Component Family Assessment

### 3.1 Architecture Quality

The sg-llm design is **architecturally excellent** and follows the proven patterns established by `sg-layout` and the sg-tree spec. Three design decisions stand out:

**Namespace folder pattern:** Using `components/llm/` as a grouping folder (no version at this level) prevents root `components/` from accumulating too many top-level entries. This is correct — namespacing by domain rather than by component.

**Independent IFD versioning:** Each component evolves independently. `sg-llm-request` can be at v0.1.2 (streaming SSE) while `sg-llm-stats` is still at v0.1.0. This is the right architecture for a family that will evolve at different rates.

**`sg-llm-events.js` as the frozen contract:** Defining a shared event constants file that is frozen at v0.1.0 (additions only, never modifications) solves the inter-component compatibility problem cleanly. Components communicate via events with type-safe constants.

### 3.2 Component Dependency Graph

```
sg-llm-events (v0.1.0, frozen)
  └── all other components import from here

sg-llm-request (headless fetch engine)
  └── emits: llm:request-start, llm:request-chunk, llm:request-complete

sg-llm-connection (provider/key/model)
  └── emits: llm:provider-changed, llm:model-changed
  └── consumed by: sg-llm-reality, sg-llm-request

sg-llm-reality (reality constructor — orchestrator)
  └── imports: sg-llm-events, sg-llm-request, sg-llm-connection
  └── emits: llm:send (the complete messages array)
  └── listens: response events from sg-llm-output

sg-llm-output, sg-llm-stats, sg-llm-debug
  └── consume: llm:request-chunk, llm:request-complete events
  └── no other dependencies

sg-llm-bundle, sg-llm-bundle-list
  └── persist/browse execution history
  └── may use sg-local-storage for IndexedDB

sg-llm-infographic, sg-llm-attachments
  └── specialized rendering and input components
```

**Potential concern:** `sg-llm-reality` is the orchestrator and has the most dependencies. If it grows too large, it becomes a bottleneck for surgical overrides. The brief mitigates this by showing that overrides are small (file-drop on context blocks = ~30 lines). This is acceptable.

### 3.3 Manifest-Driven Dependency Declaration

The `manifest.json` pattern (each component declares its dependencies and `compatibleWith` ranges) is a significant improvement over the current IFD approach where consumers must manually track cross-component compatibility. This should be adopted across the entire tools platform, not just sg-llm.

**Architectural recommendation:** Formalise the manifest schema as a platform standard. The `compatibleWith` field using semver ranges is particularly valuable for the surgical override consumption pattern.

### 3.4 The Reality Constructor Concept

The `sg-llm-reality` component — where users visually construct the model's complete context before sending — is architecturally important beyond just this component. It models a general truth: **every LLM call is a carefully constructed reality, not a conversation**. This is the mental model that makes one-shot engineering economical.

The token count display per block (not just total) is the key UX insight. When users see "system prompt: 180 tokens, context block: 850 tokens" they can make informed decisions about what to include. This is an architectural principle for the whole LLM tooling layer.

### 3.5 Phased Implementation Path

The brief proposes an 11-phase rollout (one component per phase). This is disciplined but may be too sequential. Recommend:

**Stream A (foundation):** sg-llm-events + sg-llm-request + sg-llm-output — these three together form a minimal working unit for streaming LLM calls.

**Stream B (orchestration):** sg-llm-connection + sg-llm-reality — the UI layer.

**Stream C (auxiliary):** sg-llm-stats, sg-llm-debug, sg-llm-attachments — enhancement layer.

**Stream D (advanced):** sg-llm-bundle, sg-llm-bundle-list, sg-llm-infographic — persistence and specialised output.

Streams A and B can proceed in parallel after sg-llm-events ships.

## 4. IFD Surgical Overrides — Version-Stamped Filenames

### 4.1 Technical Assessment

The version-stamped filename convention (`sg-tree--v0.1.1.js`) is **correct and necessary** for build-time layering systems. The document (v1.3.0) fills a genuine gap in the IFD guide: the original guides assumed explicit multi-script loading, not a `_latest/` merge model.

**The collision problem is real:** If v0.1.0 and v0.1.1 both contain `sg-tree.js`, the build-time merger cannot distinguish the base from the override — one destroys the other. Version-stamped filenames solve this without changing the merge algorithm.

**The `window` exposure pattern for ES modules** is the standard approach for surgical overrides in module environments (can't monkey-patch after `import`; must access via `window.*` in the override file). This is correctly documented.

### 4.2 Convention Adoption Status

This convention is actively referenced in:
- sg-tree architect brief (Doc 185, already catalogued)
- sg-llm architect brief (Doc 202, this session)

**Recommendation:** Add this to the project-wide coding standards reference. Any new component family should follow this convention from v0.1.0.

## 5. sg-wasm and Audio Transcription Assessment

### 5.1 sg-wasm Architecture

The `sg-wasm` loader is **the right abstraction**. Without it, each WASM-based tool (FFmpeg for video editing, Whisper for transcription, future image processing) would independently implement download, caching, and lifecycle management. The shared component eliminates this.

**Key design decisions:**
- **IndexedDB caching** (not localStorage) — correct choice; WASM binaries can be 50–100 MB
- **Hash validation** — essential for security; cached binaries must match expected checksums
- **Progress reporting API** — needed for UX; WASM downloads are slow on first use

**Missing design detail:** How does `sg-wasm` handle multiple simultaneous WASM modules? If the video editor and transcription tool are both open, are both modules loaded independently? A module registry (keyed by module name + version) should be specified.

### 5.2 Audio Transcription (Whisper WASM)

The two-pass architecture (real-time fast model + clean-up accurate model) is sound. The timestamp output + SRT/VTT export connecting to the video editor is the right integration point.

**Research-dependent:** The brief correctly notes this requires investigation (which Whisper WASM builds exist, streaming vs batch transcription). This is a pre-implementation research task, not a ready-to-build spec. It should be flagged as requiring a research spike before committing to API contracts.

## 6. One-Shot LLM Dev Environment Assessment

The one-shot LLM dev environment (Document 192) is **architecturally a consumer of sg-llm**, not a separate component family. The UI described is essentially:

```
Context editor (left) → sg-llm-reality
Code editor (center) → standalone HTML/CSS/JS editor
Live preview (right) → iframe with external script loading
LLM panel (bottom) → sg-llm-request + sg-llm-output
```

This is an **application** built from sg-llm components, not a new component itself. The managed memory concept (Update Memory mode) is a usage pattern of the reality constructor, not a new architectural primitive.

**Recommendation:** Design this as an application that consumes sg-llm once those components exist, not as a separate component family.

## 7. Folder Manifest (`_config.json`) vs `_page.json`

The two manifest types serve different layers of the folder experience and must not be conflated:

| Aspect | `_config.json` | `_page.json` |
|--------|---------------|-------------|
| Purpose | Control navigation structure | Control visual rendering |
| Scope | Folder tree: section order, collapse state, start file | Content: hero, slides, gallery, markdown components |
| When detected | Folder load, tree view | Folder load, content area |
| Author | Human or Claude | Primarily Claude |
| Complexity | Simple JSON (5–10 keys) | Rich component tree |

**Architectural concern:** The brief proposes combining these in a single `_config.json`. This risks scope creep — a single file serving navigation AND rendering is harder to version and harder for Claude to generate correctly. Recommend keeping them separate.
