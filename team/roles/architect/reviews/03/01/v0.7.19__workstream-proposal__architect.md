### D1 — Fix Google Fonts in Send UI + SSH KeyGen (9 files)

**ID:** D1
**Title:** Eliminate Google Fonts references from User UI, Admin UI, SSH KeyGen
**Category:** D (Development / Debt Remediation)
**Priority:** P0
**Effort:** S (Small — 1 session)

**Description:**
Nine files in the `sgraph_ai_app_send__ui__user/`, `sgraph_ai_app_send__ui__admin/`, and `tools/ssh-keygen/` directories still load fonts from `https://fonts.googleapis.com`. The `sgraph_ai__website/` is already fixed (vendored fonts in `_common/fonts/`). The same woff2 files (DM Sans + JetBrains Mono) already exist in the repo and must be replicated into the Send UI and SSH KeyGen tool using the same self-hosted `@font-face` approach.

**Rationale:**
This is a critical architectural requirement stated as non-negotiable in the v0.7.6 daily brief: "zero external dependencies." Google Fonts violates both the offline packaging requirement (deployment folder must be self-contained) and the privacy principle (Google tracks font requests). With the website fixed, the 9 remaining files are a known, enumerated debt item with a clear fix pattern already in the codebase.

**Files affected:**
- User UI v0.1.8: `index.html`, `download.html`, `room.html`, `join.html` (4 files)
- User UI v0.1.7: `vault.html` (1 file)
- User UI v0.1.6: `index.html`, `download.html` (2 files)
- SSH KeyGen tool: `tools/ssh-keygen/v0/v0.1/v0.1.0/index.html` (1 file)
- Admin UI v0.1.4: `css/sg-brand.css` (1 file — CSS `@import` must be replaced)

**Dependencies:**
- Blocked by nothing — pattern already established in `sgraph_ai__website/_common/fonts/`
- Unblocks: offline packaging, full compliance with zero-external-dependencies requirement

**Implementation note:**
Do not duplicate the woff2 files into each UI version tree. Instead, the fonts should be served from a shared location — either the Lambda's static asset path or, once D4 is complete, cdn.sgraph.ai. An interim solution is to copy the `_common/fonts/` pattern into the Send UI directories and update the CSS `@font-face` references.

### D2 — Wire deploy-website.yml to Python CI Scripts

**ID:** D2
**Title:** Replace inline YAML deployment logic with Python script invocations
**Category:** D (Development / Debt Remediation)
**Priority:** P0
**Effort:** S (Small — 1 session)

**Description:**
Three Python CI scripts exist (`scripts/deploy_static_site.py`, `scripts/generate_i18n_pages.py`, `scripts/store_ci_artifacts.py`) implementing the deployment logic. However, `.github/workflows/deploy-website.yml` still contains inline `aws s3 sync` commands, cache-control headers, and validation steps. The v0.7.6 arch brief requires GitHub Actions to act only as a trigger — all logic must live in Python. The YAML should be reduced to: checkout, install deps, call Python scripts, notify.

**Rationale:**
CI logic in YAML is untestable locally, non-portable across CI providers, and opaque to the agent workflow. The "Python-first CI" principle means every deployment step can be run locally, inspected, and unit-tested. This is a foundational requirement for the multi-cloud deployment strategy (D9) — the same Python scripts will be the adapters for GCP, Azure, and Cloudflare deployments. YAML-first CI makes multi-cloud harder. Python-first CI makes it straightforward.

**Dependencies:**
- Python scripts already exist (D2 is purely wiring)
- Unblocks: multi-cloud deployment (D9), testable CI pipeline

### D5 — Static Assets for send.sgraph.ai off-Lambda to CDN

**ID:** D5
**Title:** Serve User UI, Admin UI, and SSH KeyGen static assets from S3/CloudFront
**Category:** D (Performance / Infrastructure)
**Priority:** P0
**Effort:** M (Medium — 2-3 sessions)

**Description:**
The performance baseline document (`v0.7.6__perf-baseline__send-sgraph-ai-lambda-performance.md`) shows that all 18 requests for the Send UI upload page — including HTML, CSS, JS, and fonts — are currently served by the FastAPI Lambda function. Cold start penalty is 18.5 seconds; warm Lambda serves static files at ~170ms each. Moving static assets to S3/CloudFront eliminates this overhead: projected warm page load drops from 599ms to ~300ms, and cold start drops from 18.5s to ~2-3s (only API calls wait for Lambda). The deployment pipeline for `send.sgraph.ai` static assets should mirror the `deploy_static_site.py` pattern.

**Rationale:**
The Architect must act on the perf baseline (P0 in the brief action items: "Prioritise moving send.sgraph.ai static assets to S3/CloudFront"). Lambda cold starts of 18.5 seconds are unacceptable for a production product. The static asset serving is pure overhead — every byte of CSS, JS, and font served by Lambda is wasted compute budget. Architecturally, the FastAPI Lambda should only handle API endpoints; all static content should be S3/CloudFront. This is also required for the offline packaging story (D3).

**Projected improvement:**
| Scenario | Current | After CDN |
|----------|---------|-----------|
| Cold start | 18.50 s | ~2-3 s |
| Warm | 599 ms | ~300 ms |
| First request post-deploy | 3.68 s | ~1 s |

**Dependencies:**
- D4 (cdn.sgraph.ai must exist first — fonts served from CDN)
- D1 (Google Fonts must be replaced before CDN migration)
- Unblocks: Lambda cost reduction, D6 (smaller Lambda package without static assets)

### D6 — Lambda Dependency Optimisation (Minimum Viable Code)

**ID:** D6
**Title:** Strip Lambda packages to minimum viable dependencies; reduce cold start
**Category:** D (Performance / Security)
**Priority:** P1
**Effort:** M (Medium — 2 sessions)

**Description:**
Brief B04 from the v0.6.36 backlog describes consolidating each Lambda to a single zip with only the libraries it actually uses, stripping non-essential files (tests, docs, examples from dependencies), and benchmarking cold start before/after. The brief also introduces the "Minimum Viable Code" (MVC) principle: every removed module is one less attack surface. This workstream produces a consolidated requirements file per Lambda, a stripped packaging script, and cold start measurements comparing current vs optimised.

**Rationale:**
Lambda cold start is a compound problem: Python runtime initialisation + dependency loading + application init. Dependency stripping addresses the middle term. With D5 moving static assets off-Lambda, the package itself can be smaller. The MVC principle is also a security property: a smaller dependency surface means fewer transitive CVEs and smaller blast radius on supply chain compromise. The security review (Brief B01 in v0.6.36) explicitly states: "every removed file is one less thing to exploit."

**Deliverables:**
- Per-Lambda `requirements-lambda.txt` (trimmed from full `requirements.txt`)
- Packaging script that strips test dirs, examples, and `__pycache__` from dependency wheels
- Cold start benchmarks: before vs after
- Report on dependency reduction (count, size)

**Dependencies:**
- D5 (static assets should be off-Lambda before benchmarking — removes one variable)
- Unblocks: provisioned concurrency evaluation (smaller package = lower cost)

### P1 — One-Time Secret Link: Architecture + Data Model

**ID:** P1
**Title:** Design and implement ephemeral data sharing (`/secret` endpoint) with configurable lifespans
**Category:** P (Product Feature)
**Priority:** P1
**Effort:** M (Medium — 2-3 sessions)

**Description:**
Brief B11 (canonical version, v0.6.36 backlog) defines one-time secret links: configurable lifespan (one-view, N-views, time-bounded), AES-256-GCM encryption, key in URL fragment (never reaches server), signed action logs, and optional data room key sharing. From the Architect's perspective, this requires: a new route group (`/secret/`) in the User Lambda, a new schema (`Secret_Id`, `Secret_Schema`), storage policy for ephemeral content (auto-delete on view count exhaustion or TTL expiry), and the signed action log architecture defined in B11.

**Rationale:**
The one-time secret is a natural companion to the transfer feature and a high-value differentiator. The AppSec concern (metadata leak via human-friendly URLs) was analysed in B11 — the canonical version concludes that opaque IDs with key-in-fragment are safe. From the Architect's perspective, the most important design decision is the storage expiry model: transfers are persistent until deleted; secrets are ephemeral. The storage abstraction (Memory-FS) supports TTL via the `expires_at` field pattern already used in invite codes.

**Key architecture decisions:**
- Route group: `POST /secret/create`, `GET /secret/{id}` (consumes view count), `GET /secret/status/{id}`
- Secret_Id: 12 chars (matching Transfer_Id pattern), key stays in URL fragment
- TTL enforcement: check `expires_at` on every read; 410 Gone after expiry
- Client-side encrypted logging (B11 section): signed action log using embedded security team public key

**Dependencies:**
- Existing invite code expiry pattern (reuse `expires_at` field logic)
- P2 (symmetric data room) if secrets are used for room key sharing
- B11 AppSec review required before deployment

### P2 — Symmetric Key Data Room: Key Exchange Architecture

**ID:** P2
**Title:** Design and implement the symmetric key data room model (admin-managed keys, per-user short access keys)
**Category:** P (Product Feature)
**Priority:** P1
**Effort:** M (Medium — 2-3 sessions)

**Description:**
Brief B07 (v0.6.36 backlog) defines the symmetric key data room: admin generates a room AES key, issues per-user short access keys, and users derive the room key from their short access key. The critical architectural question is: **where does the key derivation happen?** The zero-knowledge principle requires key material to never reach the server in usable form. Current data rooms share the AES key out-of-band. B07 proposes a "Login as User" mechanism and per-user short access keys — this needs careful architectural analysis to ensure the server never holds the full room key.

**Rationale:**
Data rooms currently require manual out-of-band key distribution. This is a usability blocker for adoption. However, any automated key exchange mechanism that touches the server must be analysed against the zero-knowledge architecture. The Architect must produce a threat model for each proposed key distribution mechanism before any code is written. PROPOSED — does not exist yet.

**Key architecture decisions:**
- Short access key derivation: PBKDF2 or Argon2 (client-side only)?
- Is the admin's "master key" ever sent to the server? (Must be No.)
- Per-user permission model: viewer/editor/admin (B07) vs existing owner/editor/viewer vault ACL — can we reuse?
- "Login as User" feature: admin assumes user's session — requires careful AppSec review

**Dependencies:**
- Existing vault ACL (owner/editor/viewer) as candidate reuse
- P1 (one-time secrets for room key sharing) as companion
- AppSec + DPO review required before implementation

### P8 — Encrypted Website Variants (`/sym/`, `/pki/`)

**ID:** P8
**Title:** Build symmetrically encrypted and PKI-encrypted variants of the sgraph.ai website
**Category:** P (Product / Dogfood)
**Priority:** P3
**Effort:** L (Large — multiple sessions)

**Description:**
Brief 2 (v0.7.6 batch 1) proposes three website modes: plain (current), symmetric (`/sym/` — AES-256-GCM encrypted HTML, key shared out-of-band), and PKI (`/pki/` — RSA+AES hybrid, viewer's public key used to encrypt the symmetric key). The plain variant is the public marketing site. The symmetric variant lets SG/Send share the site content with a trusted audience (e.g., investors, early adopters) using a shared password. The PKI variant lets individuals access personalised content using their RSA-4096 key pair. A `decrypt.js` loader is required for both variants. PROPOSED — does not exist yet.

**Rationale:**
This is the most ambitious and architecturally novel work stream. It dogfoods the encryption primitives — the website itself becomes a demonstration of what SG/Send can do. It is P3 because: (1) it depends on the plain website being fully operational and well-tested; (2) the encrypted variants require a loader (decrypt.js) that is not yet designed; (3) service worker decryption (transparent page-level decryption) is complex and has security implications. The Architect must produce a detailed design document before any code is written.

**Dependencies:**
- D4 (cdn.sgraph.ai for decrypt.js)
- D3 (relative paths — encrypted variants require self-contained loading)
- Full AppSec review of the encryption model and loader

### B3 — DNS Architecture Documentation + Recovery Plan

**ID:** B3
**Title:** Document the full DNS topology (registrar, NS delegation, DNSSEC) and define a recovery plan
**Category:** B (Operational / Infrastructure)
**Priority:** P1
**Effort:** S (Small — 1 documentation session)

**Description:**
The v0.7.6 daily brief lists DNS architecture documentation as P1. No such document exists. Required content: registrar for sgraph.ai, NS record delegation plan for all subdomains (send.sgraph.ai, admin.send.sgraph.ai, cdn.sgraph.ai, tools.sgraph.ai, api.sgraph.ai, etc.), DNSSEC enablement status and plan, recovery procedure if NS records are corrupted or registrar account is compromised. Documentation is a prerequisite for the recovery plan, which is required for the BCP (Business Continuity Plan) that is flagged as P2 in the daily brief.

- Unblocks: BCP/total compromise resilience (P2 in daily brief)
