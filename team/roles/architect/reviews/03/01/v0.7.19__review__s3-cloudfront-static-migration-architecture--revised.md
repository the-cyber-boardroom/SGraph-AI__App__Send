# v0.7.19 Architect Review: S3+CloudFront Static Migration for send.sgraph.ai (REVISED)

**Date:** 2026-03-01
**Role:** Architect
**Version:** v0.7.19
**Revision:** Incorporates human feedback — no backward compat, `/api/*` prefix, v0.2.0 IFD, 3 environments, version manifest, wider layout

---

## 1. Executive Summary

This review defines the architecture for splitting send.sgraph.ai into two serving layers:

- **S3+CloudFront** serves ALL static content (HTML pages, CSS, JS, fonts, images, locale pages)
- **Lambda** serves ONLY API endpoints under the `/api/*` prefix

The architecture also introduces:
- Full i18n support (15 locales) matching sgraph.ai's pre-rendering approach
- v0.2.0 IFD consolidation focused on symmetric key encryption features only
- Clean separation of vault/PKI/data rooms (excluded from v0.2.0, move to another domain)
- Wider page layout matching sgraph.ai for consistent cross-site flow
- Link back to sgraph.ai for documentation, how-tos, and team information
- Three deployment environments (dev, main, production)
- Version manifest file for IFD `latest/` management

### What Changed from v1 of this Review

| Area | v1 Decision | Revised Decision | Reason |
|------|-------------|------------------|--------|
| Backward compat | Keep current API paths, legacy redirects | Not needed — no live customers | Only stable item is the domain `send.sgraph.ai` |
| API routing | 10 explicit CloudFront behaviors per API prefix | Single `/api/*` behavior | All Lambda routes move under `/api/` prefix |
| IFD version | v2/v2.0/v2.0.0 | v0/v0.2/v0.2.0 (major version change from v0.1.x) | Designed for IFD progression — v0.2.0 frozen, next work on v0.2.x |
| Environments | Single production | 3 environments (dev, main, prod) | dev.send, main.send, send.sgraph.ai |
| Design | Independent design | Wider layout matching sgraph.ai | Consistent flow between sites |
| Legacy redirects | 301 from /send/* | Not needed | No live customers using old URLs |
| Version management | Copy to latest/ manually | Version manifest file | Lists exactly which files go to `latest/` |

## 2. Design Principles

### 2.1. Cross-Site Consistency: sgraph.ai → send.sgraph.ai

send.sgraph.ai is a tool/service with its own identity, but the flow from sgraph.ai to send.sgraph.ai must feel **natural and consistent**:

- **Wider page layout** — match the sgraph.ai layout width and structure so the transition feels seamless
- **Shared visual language** — same design tokens, typography (DM Sans, JetBrains Mono), colour palette, component patterns
- **Navigation link back** — send.sgraph.ai includes a clear link back to sgraph.ai (header or footer), where users can find documentation, how-tos, team explanations, pricing, etc.
- **send.sgraph.ai stays focused** — 100% on sending and receiving files. No documentation pages, no team info, no how-tos. That content lives on sgraph.ai.

### 2.2. Speed as a Feature

The UI must be fast. S3+CloudFront edge caching makes this achievable. No framework overhead, no runtime JSON fetching, no unnecessary round-trips.

## 3. Proposed Architecture

### 3.1. CloudFront with Two Origins (Simplified)

```
                    send.sgraph.ai
                         |
                   [CloudFront]
                    Distribution
                   /            \
            Default (*)      /api/* behavior
                |                |
            [S3 Bucket]    [Lambda Function URL]
             /latest/       User Lambda (API only)
```

**Two origins, one domain, two behaviors:**
- **Origin 1 (S3, default):** `{account-id}--static-sgraph-ai--eu-west-2.s3.eu-west-2.amazonaws.com` with origin path `/websites/send-sgraph-ai/latest`. Same bucket as sgraph.ai.
- **Origin 2 (Lambda):** Lambda Function URL as custom origin, HTTPS only, with `X-CloudFront-Secret` header for access control.

### 3.2. CloudFront Behavior Configuration (Simplified)

| Priority | Path Pattern | Origin | Cache Policy | Purpose |
|----------|-------------|--------|-------------|---------|
| 0 | `/api/*` | Lambda | CachingDisabled | ALL API endpoints |
| Default | `*` | S3 | Custom-Static | All static content |

**Why this is now simple:** All Lambda routes move under `/api/` prefix. No backward compatibility needed — there are no live customers. This eliminates the need for 10 separate CloudFront behaviors.

### 3.3. Same-Origin Advantage

With CloudFront fronting both S3 and Lambda on the same domain (`send.sgraph.ai`):
- **No CORS needed** for UI→API calls (same origin)
- The UI at `send.sgraph.ai/en-gb/upload/` calls `send.sgraph.ai/api/transfers/create` — same origin
- The `ApiClient.baseUrl = '/api'` prefix works cleanly
- This eliminates cross-origin complexity entirely

### 3.4. Cache Strategy

| Content Type | Cache-Control Header | CloudFront TTL | Origin |
|-------------|---------------------|----------------|--------|
| HTML | `public, max-age=300` | Default: 300s, Max: 3600s | S3 |
| CSS/JS/JSON | `public, max-age=86400` | Respects header | S3 |
| Fonts/Images | `public, max-age=604800` | Respects header | S3 |
| API responses | No caching | TTL: 0 | Lambda |

## 5. i18n Strategy

### 5.1. URL Structure

Following sgraph.ai's pattern exactly. **en-gb is the base**, and **all content lives inside locale folders** — including en-gb. There are no root-level page routes. The root `/` redirects to the detected locale (default: `/en-gb/`).

```
send.sgraph.ai/                          -> Redirect to /en-gb/ (or detected locale)
send.sgraph.ai/en-gb/                    -> English landing
send.sgraph.ai/en-gb/upload/             -> English upload
send.sgraph.ai/en-gb/download/           -> English download
send.sgraph.ai/en-gb/history/            -> English history
send.sgraph.ai/pt-pt/                    -> Portuguese landing
send.sgraph.ai/pt-pt/upload/             -> Portuguese upload
send.sgraph.ai/pt-pt/download/           -> Portuguese download
send.sgraph.ai/es-es/                    -> Spanish landing
send.sgraph.ai/es-es/upload/             -> Spanish upload
... (all 15 locales, identical structure)
```

**Key principle:** The functionality inside `/en-gb/upload/` is exactly the same as `/pt-pt/upload/` — only the pre-rendered text differs. All shared code (JS, CSS, fonts) lives in `_common/`.

### 5.2. Pre-rendering Approach (Same as sgraph.ai)

1. **Build time:** `generate_i18n_pages.py` (extended for Send UI) reads `en-gb/` HTML pages + locale JSONs from the `i18n/` folder
2. **Output:** Pre-rendered locale folders (`/pt-pt/`, `/es-es/`, `/fr-fr/`, etc.) with translated text baked in — **these non-en-gb locale folders are build artifacts, they do NOT exist in the source repository** (see §6.4 for the full source vs. S3 breakdown)
3. **en-gb is the base language** — its HTML files exist in source inside the `en-gb/` folder and serve as both the actual English pages and the templates for generating all other locales. In S3, all locale files (including en-gb) sit side by side
4. **No runtime JSON fetch** for page-level text
5. **Shared assets:** `_common/` directory NOT duplicated per locale — locale pages use relative paths
6. **hreflang tags** injected for SEO cross-locale linking

### 5.3. Hybrid i18n for Web Components

The Send UI uses Web Components with Shadow DOM. Pre-rendering can't reach into shadow DOM. Solution:

- **Page-level text:** Pre-rendered via `data-i18n` attributes (like sgraph.ai)
- **Component text:** Runtime `I18n.t()` calls with translations embedded in a `<script>` tag during pre-rendering:
  ```html
  <script>I18n.strings['pt-pt'] = { ... }; I18n.locale = 'pt-pt';</script>
  ```
- **No runtime JSON fetch** — translations are always embedded at build time

### 5.4. Translation Scope

~160 keys per locale (130 existing + 30 new for v0.2.0). Compare: sgraph.ai has ~500+ keys. Send UI is simpler.

## 6. v0.2.0 IFD Consolidation

### 6.1. Version: v0/v0.2/v0.2.0

The IFD path is `v0/v0.2/v0.2.0/`. In the IFD versioning scheme `va.b.c`, **a** is the release version, **b** is the major version, and **c** is the minor version. So v0.1.x → v0.2.0 is a **major version change** (the `b` component changed). This is appropriate because:
- It collapses 9 IFD override layers into a clean consolidated codebase
- It changes the serving architecture (Lambda → S3+CloudFront)
- It moves all API routes under `/api/` prefix
- It removes vault/PKI/data room features
- It adds full i18n and wider layout
- It is **designed for IFD progression** — once v0.2.0 is done, that code is frozen. All subsequent features, improvements, bug-fixes and changes go into v0.2.1, v0.2.2, v0.2.3, etc. (minor version increments)
- S3 hosting eliminates any concern about many IFD files — all load quickly from CloudFront edge cache

### 6.2. Scope: Symmetric Key Only

**INCLUDE in v0.2.0:**
- Upload page (file selection, AES-256-GCM encryption, upload)
- Download page (key input, decryption, file preview, download)
- Transfer history (browser-local storage)
- Access gate (token validation)
- Transparency panel (transfer metadata)
- API client (transfers + presigned endpoints, with `/api/` prefix)
- Crypto module (Web Crypto API, AES-256-GCM)
- i18n (15 locales)
- File type detection + markdown parser (preview)
- Wider page layout matching sgraph.ai
- Link back to sgraph.ai in header/footer

**EXCLUDE from v0.2.0:**
- Vault (vault.html, vault-api.js, vault-crypto.js, user-vault component)
- Data Rooms (join.html, room.html, join-api.js, room-crypto.js, room-join/room-view components)
- SSH keygen tool (tools/ssh-keygen/)
- Card design assets

### 6.3. IFD Progression Design

v0.2.0 must be structured so the next set of changes is easy:

```
v0/v0.2/v0.2.0/          <- First version (frozen after completion)
v0/v0.2/v0.2.1/          <- First round of changes (overrides v0.2.0)
v0/v0.2/v0.2.2/          <- Next round
...
```

This means v0.2.0 components must be structured as clean, overridable classes — not monolithic scripts. Each Web Component class should have well-defined method boundaries that v0.2.x can override surgically.

### 6.4. Proposed Directory Structure

> **IMPORTANT — Source vs. Build Output:** In the source code repository, only `en-gb/` (the base language), `_common/`, and `i18n/` translation JSONs exist. All other locale folders (`pt-pt/`, `es-es/`, `fr-fr/`, etc.) **do not exist in source** — they are generated during the CI pipeline by `generate_i18n_pages.py`, which reads the `en-gb/` HTML pages + locale JSON files and produces pre-rendered locale folders. These generated folders only exist in the S3 bucket after deployment.

**What exists in source code (checked into Git):**

**Structure principles:**
- **In source:** Only `en-gb/` exists as a locale folder. The HTML files inside it serve a dual purpose: they are the **actual English pages** (with correct `data-i18n` attributes for the en-gb locale) AND the **templates** that `generate_i18n_pages.py` reads to produce all other locale folders.
- **In S3:** All 15 locale folders exist side by side (`en-gb/`, `pt-pt/`, `es-es/`, etc.). There are no root-level pages — the root `/` redirects to the detected locale (default: `/en-gb/`).
- **Only the non-en-gb locale folders are build artifacts.** The CI pipeline runs `generate_i18n_pages.py` which reads each `en-gb/` HTML page + each locale's JSON translation file and produces pre-rendered HTML with translated text, hreflang tags, and canonical URLs for the remaining 14 locales. The `en-gb/` folder may also be regenerated to inject hreflang/canonical tags.
- The `_common/` folder holds shared JS, CSS, and fonts — referenced by relative paths from the locale HTML pages. It is copied as-is from source to S3.
- The `i18n/` folder holds source translation JSONs — these drive the build-time generation.

## 7. Version Manifest File

### 7.1. Purpose

The version manifest (`manifest.json`) is a file inside each IFD version that lists **every file that should be copied to the `latest/` folder** for this site. This:

1. Makes the CI pipeline simpler — it reads the manifest instead of guessing which files to sync
2. Gives visibility into exactly which files from the IFD are being used in production
3. Prevents stale files from accumulating in `latest/`
4. Works for any IFD version — v0.2.0, v0.2.1, etc.

### 7.2. Format

```json
{
  "version": "v0.2.0",
  "site": "send-sgraph-ai",
  "description": "Symmetric key file sharing — v0.2.0 consolidation",
  "common_files": [
    "_common/css/style.css",
    "_common/css/components.css",
    "_common/js/app.js",
    "_common/js/api-client.js",
    "_common/js/crypto.js",
    "_common/js/i18n.js",
    "_common/js/file-type-detect.js",
    "_common/js/markdown-parser.js",
    "_common/js/components/send-upload.js",
    "_common/js/components/send-download.js",
    "_common/js/components/send-access-gate.js",
    "_common/js/components/send-transparency.js",
    "_common/js/components/header-nav.js",
    "_common/js/components/footer.js",
    "_common/fonts/fonts.css",
    "_common/fonts/DMSans-Regular.woff2",
    "_common/fonts/DMSans-Medium.woff2",
    "_common/fonts/DMSans-Bold.woff2",
    "_common/fonts/JetBrainsMono-Regular.woff2",
    "manifest.json",
    "404.html",
    "500.html",
    "favicon.svg"
  ],
  "locale_folders": [
    "en-gb/",
    "pt-pt/",
    "pt-br/",
    "es-es/",
    "es-ar/",
    "es-mx/",
    "fr-fr/",
    "fr-ca/",
    "de-de/",
    "de-ch/",
    "it-it/",
    "pl-pl/",
    "ro-ro/",
    "nl-nl/",
    "hr-hr/",
    "tlh/"
  ],
  "locale_pages": [
    "index.html",
    "upload/index.html",
    "download/index.html",
    "history/index.html"
  ]
}
```

**Note:** `locale_pages` lists the pages that exist inside every locale folder. Each locale folder (en-gb, pt-pt, etc.) contains the same set of pages with identical functionality — only the pre-rendered text differs. `common_files` lists shared assets that are not duplicated per locale.

### 7.3. CI Pipeline Usage

```python
# In deploy_static_site.py
manifest = json.load(open(f"{source_dir}/manifest.json"))

# 1. Sync common files (shared JS, CSS, fonts, error pages)
for f in manifest["common_files"]:
    s3_upload(f"{source_dir}/{f}", f"releases/{ifd_path}/{f}")

# 2. Sync locale folders (each contains identical page structure with locale-specific text)
for folder in manifest["locale_folders"]:
    s3_sync(f"{build_dir}/{folder}", f"releases/{ifd_path}/{folder}")

# 3. Copy everything from releases/{version}/ to latest/
s3_sync(f"releases/{ifd_path}/", "latest/", delete=True)
```

## 8. Three Deployment Environments

### 8.1. Environment Model

| Environment | Domain | Trigger | Purpose |
|------------|--------|---------|---------|
| **Dev** | `dev.send.sgraph.ai` | CD on push to `dev` branch | Development and experimentation |
| **Main** | `main.send.sgraph.ai` | CD on push/merge to `main` branch | QA and validation |
| **Production** | `send.sgraph.ai` | Manual CI trigger (after QA on main) | Live users |

### 8.2. Infrastructure

Each environment gets its own:
- S3 prefix: `websites/send-sgraph-ai-{env}/latest/` (prod uses `websites/send-sgraph-ai/latest/`)
- CloudFront distribution (or behaviors within a shared distribution)
- Lambda stage (`dev`, `main`, `prod`)

```
s3://745506449035--static-sgraph-ai--eu-west-2/
  websites/
    send-sgraph-ai/           <- Production (send.sgraph.ai)
      releases/v0/v0.2/v0.2.0/
      latest/
    send-sgraph-ai-dev/       <- Dev (dev.send.sgraph.ai)
      releases/v0/v0.2/v0.2.0/
      latest/
    send-sgraph-ai-main/      <- Main (main.send.sgraph.ai)
      releases/v0/v0.2/v0.2.0/
      latest/
```

### 8.3. Flow

```
Developer pushes to dev
    → CD deploys to dev.send.sgraph.ai
    → Test and iterate

PR merged to main
    → CD deploys to main.send.sgraph.ai
    → QA validates on main.send.sgraph.ai

Manual trigger after QA passes
    → Deploys to send.sgraph.ai (production)
```

This gives a safe way to develop and test without impacting live users.
