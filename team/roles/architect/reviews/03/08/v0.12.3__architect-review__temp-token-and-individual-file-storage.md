## Executive Summary

Two features analysed: (1) a post-purchase temporary token delivery page (P0), and (2) individual file storage replacing single-blob zip uploads (P1). Both features build on existing infrastructure with moderate new code. The temp token is a thin UI-only addition with one new static page. Individual file storage requires backend changes to the transfer pipeline, new storage paths, a new tree-structure document format, and substantial UI changes to the download flow.

## Part 1: Post-Purchase Temp Token

### 1.1 What EXISTS That Can Be Reused

| Existing Component | Location | Reuse |
|---|---|---|
| `api-client.js` token management | `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/_common/js/api-client.js` | Direct reuse. Already has `setToken()` (saves to `localStorage` key `sgraph-send-token`), `getToken()`, `hasToken()` |
| `send-access-gate` component | `_common/js/components/send-access-gate/send-access-gate.js` | Reference pattern. The welcome page replaces the manual gate with automatic token injection |
| Transfer create/upload/complete flow | `Routes__Transfers`, `Transfer__Service` | Reuse for uploading the encrypted token to SG/Send (token-as-transfer) |
| AES-256-GCM encryption in browser | `send-upload.js`, `send-download.js` | Direct reuse of the Web Crypto encrypt/decrypt functions |
| User UI versioned folder structure | `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/` | New page goes here |
| `check-token` / `validate-token` endpoints | `Routes__Transfers` | Reuse for validating the decrypted token after injection |
| i18n system | `_common/js/i18n.js` | Reuse for welcome page strings (8 languages) |

### 1.2 What Is PROPOSED (New)

| Component | Type | Description |
|---|---|---|
| `/welcome` static page | UI page | `welcome.html` — thanks user, fetches + decrypts shared token, saves to localStorage |
| Encrypted token transfer | Data | A standard SG/Send transfer containing the plaintext token, encrypted with a known key |

**No new backend endpoints needed.** The welcome page uses the existing transfer download flow:

1. Human creates token via Admin UI (`POST /tokens/create`)
2. Human encrypts token string with a chosen AES key and uploads to SG/Send via the standard transfer flow (`create` -> `upload` -> `complete`)
3. Human configures Stripe redirect URL: `send.sgraph.ai/welcome#{transfer_id}:{aes_key_hex}`
4. Welcome page JS: parse hash -> `GET /transfers/download/{transfer_id}` -> decrypt with AES key -> `api-client.setToken(plaintext_token)` -> done

### 1.3 Data Flow

```
Stripe checkout complete
    |
    v
Redirect: send.sgraph.ai/welcome#<transfer_id>:<aes_key_hex>
    |
    v
welcome.html loads
    |
    +--> Parse URL hash fragment (never sent to server)
    |    Extract: transfer_id, aes_key_hex
    |
    +--> GET /transfers/download/{transfer_id}
    |    Returns: encrypted blob (the token, AES-256-GCM encrypted)
    |
    +--> Web Crypto: decrypt(blob, aes_key)
    |    Returns: plaintext token string
    |
    +--> localStorage.setItem('sgraph-send-token', token)
    |
    +--> GET /transfers/check-token/{token}
    |    Verify token is valid and active
    |
    +--> Display: "Your temporary token is active. Start sending files now!"
         Link to upload page
```

### 1.4 Security Analysis

| Property | Assessment |
|---|---|
| **Key in URL hash** | Hash fragments are never sent to the server (HTTP spec). Safe. Same pattern already used for transfer decryption keys. |
| **Shared token model** | Acceptable risk per brief. Token has limited credits. Human monitors and rotates. |
| **Token leakage via Referer** | Hash fragments are excluded from `Referer` headers (browser spec). No leakage. |
| **Token leakage via browser history** | YES — the full URL including hash is stored in browser history. Acceptable for a shared temp token with limited credits. |
| **Token rotation** | Human uploads new encrypted token to SG/Send, updates Stripe redirect URL with new transfer_id (and optionally new key). Old transfer can be deleted or left to expire. |
| **MITM on redirect** | HTTPS end-to-end (Stripe -> CloudFront -> Lambda). No plaintext in transit. |
| **Server-side visibility** | Server sees the transfer_id (to serve the encrypted blob) but never sees the AES key or plaintext token. Zero-knowledge maintained. |

### 1.5 Hash Fragment Format

**Recommended format:** `#{transfer_id}:{aes_key_hex}`

Example: `#a1b2c3d4e5f6:0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef`

This mirrors the existing download page pattern where the decryption key is in the hash. The colon separator is unambiguous (transfer IDs are hex, AES keys are hex).

**Alternative considered:** Use the existing download page directly (`/d/{transfer_id}#{key}`). Problem: the download page downloads and saves a file. We need different behaviour — extract the token string and save to localStorage. A dedicated welcome page is correct.

### 1.6 Analytics Opportunity

The welcome page load is a conversion event. Track:
- Page load timestamp (purchase-to-activation latency)
- Token fetch success/failure
- Whether user navigates to upload page (activation)

Implementation: fire analytics events via existing `Middleware__Analytics` on the `GET /transfers/download/{transfer_id}` call (already tracked as a download event with `ip_hash` and `user_agent`). No new analytics endpoints needed.

### 1.7 Implementation Cost

| Task | Effort | Dependencies |
|---|---|---|
| Create `welcome.html` | Small (1 page, reuses existing JS) | None |
| Parse hash, fetch, decrypt, save token | Small (30-50 lines JS) | Existing Web Crypto + api-client |
| i18n strings for welcome page | Small | Existing i18n system |
| Stripe redirect configuration | Config change (no code) | welcome.html deployed |

**Total estimate:** Half a session. No backend changes. No new tests beyond static page load test.

## Part 2: Individual File Storage

### 2.1 What EXISTS That Can Be Reused

| Existing Component | Location | Reuse |
|---|---|---|
| `Transfer__Service` | `sgraph_ai_app_send/lambda__user/service/Transfer__Service.py` | Extend. Current storage layout: `transfers/{id}/meta.json` + `transfers/{id}/payload`. New layout adds `transfers/{id}/tree.json` + `transfers/{id}/blobs/{blob_id}` |
| `Storage_FS` abstraction | `memory_fs.storage_fs.Storage_FS` | Direct reuse. `file__save`, `file__bytes`, `file__json`, `file__exists` all work with compound paths (`transfers/{id}/blobs/{blob_id}`) |
| Presigned URL service | `Service__Presigned_Urls` | Extend for per-blob presigned URLs. Current `s3_key()` builds `transfers/{id}/payload`. New method: `s3_key_blob(transfer_id, blob_id)` |
| AES-256-GCM client-side encryption | User UI JS | Reuse. Each file encrypted with the same shared key (P1) or per-file keys (P2 future) |
| Multipart upload flow | `Routes__Presigned` | Reuse for individual files >10MB |
| SGMETA envelope | `send-upload.js`, `send-download.js` | Adapt. Currently wraps filename into the blob. With individual files, filename goes into the tree instead |
| `send-download` component | User UI | Major refactor. Currently: download blob -> decrypt -> save. New: download tree -> show file list -> on-demand per-file download/decrypt |

### 2.2 Storage Layout (Memory-FS / S3)

**Current layout (single blob):**
```
transfers/
  {transfer_id}/
    meta.json          # Transfer metadata (status, size, events, ip_hash)
    payload            # Single encrypted blob (entire upload)
```

**PROPOSED layout (individual files):**
```
transfers/
  {transfer_id}/
    meta.json          # Transfer metadata (extended with storage_mode field)
    tree.json          # Encrypted tree structure (file list, sizes, blob IDs)
    blobs/
      {blob_id_1}      # Encrypted file blob
      {blob_id_2}      # Encrypted file blob
      ...
      {blob_id_n}      # Encrypted file blob
```

**Backward compatibility:** The `meta.json` gets a new field `storage_mode`:
- `"single"` (default, existing transfers) — payload at `transfers/{id}/payload`
- `"individual"` (new) — tree at `transfers/{id}/tree.json`, blobs at `transfers/{id}/blobs/{blob_id}`

Existing code continues to work. The download endpoint checks `storage_mode` to decide which path to use.

### 2.3 Tree Structure Data Model

```json
{
  "version": "1.0",
  "storage_mode": "individual",
  "total_files": 50,
  "total_size_bytes": 1073741824,
  "key_hash": "sha256_of_shared_aes_key",
  "created_at": "2026-03-08T12:00:00Z",
  "upload_status": "in_progress|complete",
  "files": [
    {
      "path": "src/index.html",
      "size_bytes": 3501,
      "content_type": "text/html",
      "extension": "html",
      "plaintext_hash": "sha256_of_original_file",
      "blob_id": "a1b2c3d4",
      "blob_status": "pending|uploaded|verified",
      "public_key": null,
      "encrypted_size_bytes": 3529
    }
  ]
}
```

**Key design decisions:**

1. **`blob_id` format:** 8-char hex (same as `Obj_Id` pattern used elsewhere). Generated client-side to avoid round-trips.
2. **`blob_status`:** Enables live upload sharing. `pending` = not yet uploaded. `uploaded` = blob exists on server. `verified` = server confirmed blob size matches tree declaration.
3. **`key_hash`:** SHA-256 of the AES key. Allows the server to verify that a download request's key matches without seeing the key itself. (The hash is in the tree, which is itself encrypted — so the server never sees this.)
4. **`upload_status`:** `in_progress` while blobs are still being uploaded. `complete` when all blobs are present. Enables live sharing.
5. **`public_key`:** `null` for P1 (shared key). Future P2 populates this with per-file recipient public keys.
6. **Tree is encrypted.** The tree structure contains file names, which are sensitive (same reason SGMETA exists). The tree is encrypted with the same AES key as the blobs.

### 2.4 New Backend Endpoints (PROPOSED)

All on the User Lambda under the existing `transfers` tag:

| Method | Path | Purpose |
|---|---|---|
| POST | `/transfers/upload-tree/{transfer_id}` | Upload encrypted tree structure |
| GET | `/transfers/download-tree/{transfer_id}` | Download encrypted tree structure |
| POST | `/transfers/upload-blob/{transfer_id}/{blob_id}` | Upload individual encrypted file blob |
| GET | `/transfers/download-blob/{transfer_id}/{blob_id}` | Download individual encrypted file blob |
| GET | `/transfers/blob-status/{transfer_id}` | List which blobs exist (for resume/live sharing) |

**Why new endpoints instead of reusing upload/download?**

The existing `upload` endpoint stores a single payload at `transfers/{id}/payload`. The new endpoints need to store at different paths (`tree.json`, `blobs/{blob_id}`). Separate endpoints are cleaner than overloading the existing endpoint with a `type` parameter.

**Why not use the vault endpoints?** The vault system (`POST /vault/file`, `GET /vault/file/{key}/{guid}`) has a similar pattern but is admin-authenticated and uses different storage paths. Transfers are public-facing (token-gated). Keeping them separate maintains the security boundary.

### 2.5 Transfer Service Changes

Extend `Transfer__Service` with new methods:

```python
# New paths
def tree_path(self, transfer_id):
    return f'transfers/{transfer_id}/tree.json'

def blob_path(self, transfer_id, blob_id):
    return f'transfers/{transfer_id}/blobs/{blob_id}'

# New operations
def upload_tree(self, transfer_id, tree_bytes):
    """Store encrypted tree structure. Sets storage_mode='individual' in meta."""

def download_tree(self, transfer_id):
    """Retrieve encrypted tree structure."""

def upload_blob(self, transfer_id, blob_id, blob_bytes):
    """Store individual encrypted file blob."""

def download_blob(self, transfer_id, blob_id):
    """Retrieve individual encrypted file blob."""

def get_blob_status(self, transfer_id):
    """Return list of blob_ids that exist in storage."""
```

**Meta changes:** `create_transfer` gets an optional `storage_mode` parameter. Default remains `"single"`. When `"individual"` is specified, the service expects tree + blobs instead of a single payload. `complete_transfer` for `"individual"` mode verifies that the tree exists and all declared blobs are present.

### 2.6 Presigned URL Extension

For files >6MB, individual blobs need presigned URLs too. Extend `Service__Presigned_Urls`:

```python
def s3_key_blob(self, transfer_id, blob_id):
    key = f'transfers/{transfer_id}/blobs/{blob_id}'
    if self.s3_prefix:
        key = f'{self.s3_prefix}/{key}'
    return key

def create_blob_upload_url(self, transfer_id, blob_id, expiry=3600):
    """Presigned PUT URL for a single blob."""

def create_blob_download_url(self, transfer_id, blob_id, expiry=3600):
    """Presigned GET URL for a single blob."""
```

### 2.7 Client-Side Upload Flow (PROPOSED)

```
User selects zip/folder
    |
    +--> Count files
    |    If > 100: fall back to single-blob mode (existing flow)
    |    If <= 100: use individual file mode
    |
    +--> POST /transfers/create  { storage_mode: "individual", file_size_bytes: total }
    |    Returns: transfer_id
    |
    +--> Build tree structure in memory
    |    For each file: generate blob_id, compute size/type/hash
    |
    +--> Encrypt tree with AES key
    |    POST /transfers/upload-tree/{transfer_id}
    |    (Tree is now downloadable -- share link can be sent immediately)
    |
    +--> For each file (parallel, batch of 5-10):
    |    Encrypt file with AES key
    |    POST /transfers/upload-blob/{transfer_id}/{blob_id}
    |    (Or presigned URL for files > 6MB)
    |
    +--> POST /transfers/complete/{transfer_id}
    |    Server verifies all blobs present
    |
    +--> Return share link: /d/{transfer_id}#{aes_key}
```

### 2.8 Client-Side Download Flow (PROPOSED)

```
Recipient opens /d/{transfer_id}#{aes_key}
    |
    +--> GET /transfers/info/{transfer_id}
    |    Check storage_mode
    |
    +--> If storage_mode == "single": existing flow (download entire blob)
    |
    +--> If storage_mode == "individual":
    |
    +--> GET /transfers/download-tree/{transfer_id}
    |    Decrypt tree with AES key
    |    Display file list (instant -- tree is ~5KB)
    |
    +--> User clicks a file:
    |    GET /transfers/download-blob/{transfer_id}/{blob_id}
    |    Decrypt with AES key
    |    Update file state: remote -> in_browser
    |    Offer save or display
    |
    +--> User clicks "Download All":
    |    Sequential or parallel blob download
    |    Reconstruct zip client-side (JSZip or similar)
    |
    +--> User clicks "Cache locally":
    |    Store decrypted file in IndexedDB
    |    Update file state: in_browser -> cached
```

### 2.9 Three File States in UI

| State | Storage | Survives Refresh | Visual |
|---|---|---|---|
| **Remote** | Server only | N/A | Cloud icon, muted colour |
| **In browser** | Decrypted in JS memory | No | Filled icon, normal colour |
| **Cached** | IndexedDB | Yes | Filled icon + pin/checkmark |

Implementation: the download component maintains a state map `{ [blob_id]: 'remote' | 'in_browser' | 'cached' }`. On page load, check IndexedDB for cached blobs and set initial state accordingly.

**IndexedDB vs localStorage:** IndexedDB is correct for binary file data (no size limits, binary-native). localStorage is limited to ~5MB total and string-only.

### 2.10 Live Upload Sharing

The share link works as soon as the tree is uploaded. The recipient sees the file list immediately. Files that haven't been uploaded yet show as `blob_status: "pending"` in the tree.

**Polling mechanism:** The recipient's page polls `GET /transfers/blob-status/{transfer_id}` every 5-10 seconds to check for newly uploaded blobs. When a blob transitions from pending to uploaded, the UI updates the file state.

**Alternative: no polling.** The tree is uploaded once with all files listed. The recipient tries to download a blob. If it returns 404, the blob hasn't been uploaded yet. UI shows "uploading..." for that file. The recipient can retry. This is simpler but less responsive.

**Recommendation:** Start with the 404-based approach (simpler). Add the polling endpoint in a follow-up if UX feedback demands it.

### 2.11 Security Considerations

| Concern | Analysis |
|---|---|
| **Blob ID predictability** | Blob IDs are generated client-side. Use `crypto.getRandomValues()` (8 hex chars = 32 bits). Low collision risk within a single transfer (max 100 files). Not security-critical because blob download requires knowing the transfer_id. |
| **Tree as metadata oracle** | The tree is encrypted. An attacker with the transfer_id but not the AES key can download the encrypted tree but cannot read file names, sizes, or structure. Zero-knowledge preserved. |
| **Per-blob access control** | Any request with the transfer_id can download any blob in that transfer. No per-blob authentication in P1. This matches the current model (anyone with the transfer_id can download the payload). |
| **Blob count DoS** | Limit enforced client-side (100 files) AND server-side. `upload-blob` should reject if blob count exceeds 100 for the transfer. |
| **Orphan blobs** | If upload is abandoned after some blobs are uploaded, orphan blobs consume storage. Mitigation: transfers have a TTL (existing pattern). Cleanup sweeps delete transfers older than retention period, including all their blobs. |
| **Blob size validation** | Server should validate that each blob's size is within acceptable limits. Reuse the existing per-transfer size limit logic. |

### 2.12 `transfer/info` Response Extension

The existing `GET /transfers/info/{transfer_id}` response needs a new field:

```json
{
  "transfer_id": "a1b2c3d4e5f6",
  "status": "completed",
  "file_size_bytes": 1073741824,
  "storage_mode": "single|individual",
  "file_count": 50,
  "created_at": "2026-03-08T12:00:00Z",
  "download_count": 3
}
```

The download page reads `storage_mode` to decide which flow to use. Backward compatible: old transfers default to `"single"`.

## Risk Areas and Open Questions

### Risks

| # | Risk | Severity | Mitigation |
|---|---|---|---|
| R1 | Zip extraction in browser (large files) | Medium | Use streaming APIs (`ReadableStream`). Test with 1GB+ zips. Fall back to single-blob if extraction fails. |
| R2 | 100 parallel S3 PUTs per upload | Low | S3 handles this trivially. Batch client-side to 5-10 concurrent uploads. |
| R3 | Download page complexity doubles | Medium | Clear separation: single-blob path (existing) vs individual-file path (new). Feature flag if needed. |
| R4 | Tree structure versioning | Low | Version field in tree JSON. Migration logic if format changes. |
| R5 | IndexedDB quota on mobile | Medium | Mobile browsers limit IndexedDB. Show "Cache locally" option only when quota permits. Fallback: in-browser only. |

### Open Questions

| # | Question | Architect Recommendation |
|---|---|---|
| Q1 | Should the tree structure itself be stored encrypted or plaintext on the server? | **Encrypted.** The tree contains file names, which are PII/sensitive metadata. Encrypt with the same AES key as the blobs. This is consistent with zero-knowledge. |
| Q2 | Should blob_ids be content-addressed (hash of encrypted content) or random? | **Random (client-generated).** Content-addressing would require the client to hash the encrypted blob before upload, adding latency. Random IDs are simpler and sufficient. Content-addressing is a P2 optimisation for deduplication. |
| Q3 | Should `complete_transfer` verify all blobs match the tree? | **Yes.** Server reads the encrypted tree... wait, it cannot read the tree (it's encrypted). Server can only verify: (a) tree exists, (b) number of blobs matches `file_count` from meta (set during create). The tree's internal consistency (blob_ids matching actual blobs) must be verified client-side on download. |
| Q4 | Does the download page need a "Download All as Zip" button? | **Yes, eventually.** Use JSZip (MIT, 100KB). But this is a P1b+ feature. MVP shows file list with individual download buttons. |
| Q5 | How does the 100-file limit interact with nested folder structures? | Count all files (not folders). Folders are metadata in the tree path field (`src/components/Button.js`). Only files with content generate blobs. Empty folders exist only in the tree. |
| Q6 | What happens if the welcome page transfer is deleted? | Post-purchase redirect breaks. Mitigation: the human should use a long-lived transfer (or re-upload). The welcome page should show a clear error message if the transfer is not found. |
| Q7 | Should individual-mode transfers support the existing `/transfers/download/{id}` endpoint? | **No.** Return a 400 with message: "This transfer uses individual file storage. Use /transfers/download-tree/{id} and /transfers/download-blob/{id}/{blob_id} instead." This prevents confusion and forces the download page to use the correct flow. |
