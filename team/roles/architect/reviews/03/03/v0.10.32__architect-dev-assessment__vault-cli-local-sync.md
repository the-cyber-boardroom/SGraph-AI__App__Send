## Part 1: Architecture — "Be Inspired by Git"

The human's instruction is clear: **copy git's design patterns as much as possible.** This section maps every git concept to its sg-vault equivalent.

### 1.1 The `.sg_vault/` Directory (Inspired by `.git/`)

The `.git/` folder is the brain of a git repository. Our `.sg_vault/` folder serves the same purpose — it's the local metadata that makes a folder a vault.

```
.git/                           .sg_vault/
├── config          ──────►     ├── config              # Remote, vault_id, endpoint
├── HEAD            ──────►     ├── HEAD                # Current vault key reference
├── index           ──────►     ├── index.json          # File tracking manifest
├── objects/        ──────►     │   (not needed locally — files are the objects)
├── refs/           ──────►     ├── snapshots/          # Future: point-in-time snapshots
├── hooks/          ──────►     ├── hooks/              # Future: pre-push, post-pull hooks
└── FETCH_HEAD      ──────►     └── FETCH_HEAD          # Last known remote tree state
```

**Design decision D1: `.sg_vault/` is the single source of truth for local vault state.** If `.sg_vault/` exists in a directory, that directory is a vault working copy. If it doesn't, it isn't. Just like `.git/`.

#### `.sg_vault/config` (Inspired by `.git/config`)

```ini
[vault]
    id = a1b2c3d4
    name = Project Alpha

[remote "origin"]
    url = https://send.sgraph.ai
    settings-transfer-id = f5e6d7c8b9a0

[auth]
    # Token stored here only if user opts in (like git credential store)
    # If absent, CLI prompts on write operations
    token = <access-token>

[sync]
    auto = false
    ignore = .DS_Store, Thumbs.db, *.tmp
```

**Why INI format (not JSON)?** Because git uses INI. Users who know git will immediately understand this file. It's human-readable and human-editable. Python's `configparser` handles it natively.

#### `.sg_vault/HEAD` (Inspired by `.git/HEAD`)

Contains the vault key in a recoverable format:

```
passphrase:a1b2c3d4:f5e6d7c8b9a0
```

**Security note:** This is equivalent to storing the vault key in the browser's localStorage (which the web UI already does). The HEAD file gives the CLI passwordless access to the vault. If the user doesn't want to store it on disk, they can use `--no-save-key` during clone and the CLI will prompt each time.

**File permissions:** Created with `0600` (owner read/write only), same security posture as SSH keys.

#### `.sg_vault/index.json` (Inspired by `.git/index`)

Git's index is the staging area — it tracks what's in the working directory vs what's committed. Our index.json serves the same purpose: it tracks what's local vs what's remote.

```json
{
  "version": 1,
  "vault_id": "a1b2c3d4",
  "last_sync": "2026-03-03T10:00:00Z",
  "remote_tree_snapshot": "<hash of last known remote tree JSON>",
  "entries": {
    "report.pdf": {
      "local_hash": "sha256:abc123...",
      "local_mtime": "2026-03-03T09:30:00Z",
      "local_size": 245000,
      "remote_transfer_id": "m3n4o5p6",
      "remote_hash": "sha256:abc123...",
      "remote_size": 245000,
      "synced_at": "2026-03-03T10:00:00Z",
      "state": "synced"
    },
    "documents/notes.md": {
      "local_hash": "sha256:def456...",
      "local_mtime": "2026-03-03T11:00:00Z",
      "local_size": 3200,
      "remote_transfer_id": "q7r8s9t0",
      "remote_hash": "sha256:def456...",
      "remote_size": 3200,
      "synced_at": "2026-03-03T10:00:00Z",
      "state": "modified_locally"
    }
  }
}
```

**Entry states** (inspired by `git status`):
- `synced` — local and remote match
- `modified_locally` — local file changed since last sync
- `modified_remotely` — remote file changed since last sync
- `conflict` — changed on both sides
- `added_locally` — new local file, not yet pushed
- `added_remotely` — new remote file, not yet pulled
- `deleted_locally` — local file removed
- `deleted_remotely` — remote file removed

#### `.sg_vault/FETCH_HEAD` (Inspired by `.git/FETCH_HEAD`)

Stores the last fetched remote tree (decrypted JSON), so `sg-vault status` can work offline without re-fetching.

```json
{
  "fetched_at": "2026-03-03T10:00:00Z",
  "tree_transfer_id": "xyz789",
  "tree": { ... }
}
```

### 1.2 `sg-vault clone` (Inspired by `git clone`)

**`git clone` sequence:**
1. Create target directory
2. Create `.git/` with config pointing to remote
3. Fetch all objects from remote
4. Check out the working tree (write files to disk)

**`sg-vault clone` sequence:**
1. Create target directory (or use existing empty directory)
2. Create `.sg_vault/` directory
3. Parse the vault key → derive encryption key
4. Download settings transfer → decrypt → read tree_transfer_id
5. Download tree transfer → decrypt → get full file inventory
6. Create all folders in the tree (including empty ones — lesson from git's `.gitkeep` problem)
7. Download each file → decrypt → write to local filesystem
8. Build `index.json` from the completed state
9. Write `config`, `HEAD`, `FETCH_HEAD`

```bash
# Clone a vault (like git clone)
sg-vault clone "mypassphrase:a1b2c3d4:f5e6d7c8b9a0" ~/vaults/alpha

# Clone with custom name
sg-vault clone "mypassphrase:a1b2c3d4:f5e6d7c8b9a0" ~/vaults/alpha --name "Project Alpha"

# Clone without saving the key to disk
sg-vault clone "mypassphrase:a1b2c3d4:f5e6d7c8b9a0" ~/vaults/alpha --no-save-key
```

**Output (git-style):**
```
Cloning into '/Users/alice/vaults/alpha'...
remote: Vault "Project Alpha" (3 folders, 12 files, 45.2 MB)
Downloading: 100% ████████████████████ 12/12 files (45.2 MB)
Done. Vault cloned to /Users/alice/vaults/alpha
```

**Empty folders:** Unlike git (which cannot track empty directories), `sg-vault clone` explicitly creates all folders from the vault tree, even if they contain no files. This is a deliberate improvement over git's behaviour. The vault tree data structure includes folders as first-class citizens.

### 1.3 `sg-vault init` (Inspired by `git init`)

**`git init` sequence:**
1. Create `.git/` with default structure
2. Ready — no remote, no objects, no commits

**`sg-vault init` sequence:**
1. Create `.sg_vault/` directory
2. Generate a new vault_id (8 hex chars)
3. Generate a passphrase (or accept user-provided one)
4. Create vault on remote (settings + empty tree → upload as transfers)
5. Write `config`, `HEAD`, empty `index.json`
6. Print the vault key (for sharing)

```bash
# Init a new vault (generates passphrase)
sg-vault init ~/vaults/new-project

# Init with custom passphrase
sg-vault init ~/vaults/new-project --passphrase "blue-mountain-77"

# Init with custom name
sg-vault init ~/vaults/new-project --name "New Project" --passphrase "blue-mountain-77"
```

**Output:**
```
Initialised vault in /Users/alice/vaults/new-project
Vault key: blue-mountain-77:e4f5a6b7:k8l9m0n1
Share this key to give others access to this vault.
```

### 1.4 `sg-vault push` / `sg-vault pull` (Inspired by `git push` / `git pull`)

**Push** = upload local changes to remote:
1. Scan local files → compute hashes → compare with index
2. Identify added/modified/deleted files
3. For each new/modified file: encrypt → upload as transfer → get transfer_id
4. For deleted files: remove from tree (transfer data persists — like git, nothing is truly deleted immediately)
5. Update remote tree (encrypt → upload new tree transfer)
6. Update remote settings (with new tree_transfer_id)
7. Update local index

**Pull** = download remote changes to local:
1. Fetch remote settings → decrypt → get tree_transfer_id
2. Fetch remote tree → decrypt
3. Compare remote tree with local index
4. Download new/modified files → decrypt → write to disk
5. Remove locally files that were deleted remotely
6. Create new folders, remove empty folders
7. Update local index and FETCH_HEAD

```bash
sg-vault push                    # Push all local changes
sg-vault push documents/         # Push only files in documents/ (future)
sg-vault pull                    # Pull all remote changes
sg-vault pull --dry-run          # Show what would be pulled without doing it
```

### 1.5 `sg-vault status` (Inspired by `git status`)

```bash
$ sg-vault status
On vault: Project Alpha (a1b2c3d4)
Remote: https://send.sgraph.ai

Changes not yet pushed:
  (use "sg-vault push" to upload)

    new file:   images/logo.png
    modified:   documents/notes.md
    deleted:    old-draft.txt

Changes not yet pulled:
  (use "sg-vault pull" to download)

    new file:   shared/budget.xlsx
    modified:   README.md

2 files to push, 2 files to pull
```

### 1.6 `sg-vault log` (Inspired by `git log` — Future)

Not in MVP, but the architecture should support it. Each sync creates a snapshot reference. The tree's `version` field already increments.

## Part 2: Critical Architecture Decision — Which API?

### The Two Vault Implementations

The codebase has **two distinct vault implementations** that must not be confused:

| | Browser Vault (Transfer-based) | Admin Vault (Cache-based) |
|---|---|---|
| **API** | `POST /transfers/create`, `/transfers/upload/{id}`, `/transfers/download/{id}` | `POST /vault/create`, `/vault/file`, `/vault/index` |
| **Lambda** | User Lambda (public) | Admin Lambda (authenticated) |
| **Storage** | Each file = one transfer (Memory-FS / S3) | Files stored in MGraph-AI Cache Service |
| **Server knowledge** | Server sees unrelated transfers. Zero vault awareness. | Server knows vault structure (cache_id, folders, files) |
| **Encryption** | Client-side AES-256-GCM. Server stores opaque blobs. | Client-side encryption, but server has structured metadata |
| **Used by** | Browser vault UI (sg-vault.js) | Data Rooms, admin operations |

**Design decision D2: The CLI MUST use the Transfer API (User Lambda).**

Rationale:
1. **Interoperability is non-negotiable.** A file uploaded via the CLI must appear in the browser vault, and vice versa. The browser vault uses the Transfer API.
2. **Zero-knowledge purity.** The Transfer API treats every upload as an opaque blob. The server has zero awareness that these transfers form a vault. This is the strongest privacy posture.
3. **No admin auth required for read operations.** The Transfer API is public. Anyone with a transfer_id can download. Only uploads require an access token.
4. **Existing ecosystem.** The vault key format (`passphrase:vault_id:settings_transfer_id`) is already built around transfer IDs.

### API Contract Summary

| Operation | HTTP Method | Endpoint | Auth Required |
|---|---|---|---|
| Create transfer | POST | `/transfers/create` | Yes (access token) |
| Upload encrypted data | POST | `/transfers/upload/{id}` | Yes |
| Mark transfer complete | POST | `/transfers/complete/{id}` | Yes |
| Download encrypted data | GET | `/transfers/download/{id}` | No |
| Get transfer info | GET | `/transfers/info/{id}` | No |

**Auth token:** Header `x-sgraph-send-access-token` or query parameter.

## Part 3: Encryption Interoperability Contract

This is the most critical part of the assessment. **If the CLI produces ciphertext that the browser can't decrypt, or vice versa, nothing works.**

### 3.1 Exact Crypto Parameters

| Parameter | Value | Source |
|---|---|---|
| Algorithm | AES-256-GCM | `sg-send-crypto.js` |
| Key size | 256 bits (32 bytes) | `KEY_LENGTH = 256` |
| Nonce (IV) | 12 bytes, random per encryption | `IV_LENGTH = 12` |
| Auth tag | 16 bytes (appended by GCM) | Web Crypto API default |
| KDF | PBKDF2-HMAC-SHA256 | `deriveKey()` |
| KDF iterations | 600,000 | Hardcoded |
| KDF salt | `sg-vault-v1:{vault_id}` (UTF-8 encoded) | `deriveKey()` salt param |

### 3.2 Wire Format

```
Encrypted blob = [12 bytes nonce][ciphertext + 16 bytes auth tag]
```

- Bytes 0–11: Random nonce (IV)
- Bytes 12+: Ciphertext with GCM authentication tag appended

No additional framing, no length prefix, no version byte. Just nonce + ciphertext.

### 3.3 Python Equivalent

```python
from cryptography.hazmat.primitives.ciphers.aead import AESGCM
from cryptography.hazmat.primitives.kdf.pbkdf2   import PBKDF2HMAC
from cryptography.hazmat.primitives               import hashes
import os

ITERATIONS = 600_000
NONCE_LENGTH = 12

def derive_key(passphrase: str, vault_id: str) -> bytes:
    """Derive AES-256 key from passphrase + vault_id. Must match browser KDF exactly."""
    salt = f"sg-vault-v1:{vault_id}".encode("utf-8")
    kdf  = PBKDF2HMAC(algorithm=hashes.SHA256(), length=32, salt=salt, iterations=ITERATIONS)
    return kdf.derive(passphrase.encode("utf-8"))

def encrypt(data: bytes, key: bytes) -> bytes:
    """Encrypt data with AES-256-GCM. Output: [nonce][ciphertext+tag]."""
    nonce      = os.urandom(NONCE_LENGTH)
    aesgcm     = AESGCM(key)
    ciphertext = aesgcm.encrypt(nonce, data, None)  # No AAD
    return nonce + ciphertext

def decrypt(blob: bytes, key: bytes) -> bytes:
    """Decrypt [nonce][ciphertext+tag] with AES-256-GCM."""
    nonce      = blob[:NONCE_LENGTH]
    ciphertext = blob[NONCE_LENGTH:]
    aesgcm     = AESGCM(key)
    return aesgcm.decrypt(nonce, ciphertext, None)  # No AAD
```

### 3.4 Vault Key Parsing

Format: `{passphrase}:{vault_id}:{settings_transfer_id}`

**Parsing rule:** Split by `:` from the right. The last segment is `settings_transfer_id`, the second-to-last is `vault_id` (8 hex chars), everything before that is the passphrase (which may contain colons).

```python
def parse_vault_key(full_key: str) -> tuple[str, str, str]:
    parts = full_key.split(":")
    settings_transfer_id = parts.pop()
    vault_id             = parts.pop()
    passphrase           = ":".join(parts)
    return passphrase, vault_id, settings_transfer_id
```

### 3.5 Interop Validation (First Test to Write)

The very first test must be: **encrypt in Python → decrypt in JavaScript, and encrypt in JavaScript → decrypt in Python.** This is the interoperability gate. If this test fails, nothing else matters.

Test approach:
1. Hardcode a known passphrase, vault_id, and plaintext
2. Derive key in Python and JavaScript separately → keys must match
3. Encrypt in Python → decrypt result in JavaScript → plaintext must match
4. Encrypt in JavaScript → decrypt result in Python → plaintext must match

## Part 4: Data Structures — Settings and Tree

### 4.1 Settings Object (Encrypted, Stored as Transfer)

```json
{
  "vault_name": "Project Alpha",
  "vault_id": "a1b2c3d4",
  "created": "2026-03-03T12:34:56.789Z",
  "version": 1,
  "description": "Client project files",
  "_tree_transfer_id": "f5e6d7c8b9a0"
}
```

**Update pattern:** When the tree changes, settings is re-encrypted and re-uploaded as a NEW transfer. The settings_transfer_id in the vault key changes. This means the vault key is mutable — the passphrase and vault_id stay the same, but the settings_transfer_id updates with every push.

**Implication for CLI:** After a push, the CLI must update `.sg_vault/config` with the new settings_transfer_id, and `.sg_vault/HEAD` with the updated vault key. Users sharing the vault need the latest key — or more precisely, any key whose settings_transfer_id points to a valid settings object whose tree is reachable.

### 4.2 Tree Object (Encrypted, Stored as Transfer)

```json
{
  "version": 1,
  "updated": "2026-03-03T12:34:56.789Z",
  "tree": {
    "/": {
      "type": "folder",
      "children": {
        "documents": {
          "type": "folder",
          "children": {
            "report.pdf": {
              "type": "file",
              "file_id": "a1b2c3d4e5f6g7h8",
              "size": 102400,
              "uploaded": "2026-03-02T15:45:00.000Z"
            }
          }
        },
        "notes.txt": {
          "type": "file",
          "file_id": "e5f6g7h8i9j0k1l2",
          "size": 1024,
          "uploaded": "2026-03-03T09:15:00.000Z"
        }
      }
    }
  }
}
```

**Path convention:** Unix-style forward slashes. Root is `/`. Folder paths: `/documents/`. File paths: `/documents/report.pdf`.

## Part 8: Sync Algorithm

### 8.1 Change Detection

**Local changes** (detected by comparing filesystem to `index.json`):
- New file: exists on disk, not in index → `added_locally`
- Modified file: hash differs from index entry → `modified_locally`
- Deleted file: in index, not on disk → `deleted_locally`

**Remote changes** (detected by comparing remote tree to `index.json`):
- New file: in remote tree, not in index → `added_remotely`
- Modified file: transfer_id differs from index → `modified_remotely`
- Deleted file: in index, not in remote tree → `deleted_remotely`

### 8.2 Push Algorithm

```
1. Read .sg_vault/index.json
2. Scan local filesystem → compute SHA-256 hashes
3. Diff local state vs index → identify changes
4. For each added/modified file:
   a. Read file bytes
   b. Encrypt with derived key
   c. POST /transfers/create → get transfer_id
   d. POST /transfers/upload/{id} → upload encrypted bytes
   e. POST /transfers/complete/{id}
5. Build updated tree JSON (add new entries, update modified, remove deleted)
6. Encrypt tree → upload as new transfer → get tree_transfer_id
7. Build updated settings JSON (with new tree_transfer_id)
8. Encrypt settings → upload as new transfer → get settings_transfer_id
9. Update .sg_vault/config (new settings_transfer_id)
10. Update .sg_vault/HEAD (new vault key)
11. Update .sg_vault/index.json (all entries now "synced")
```

### 8.3 Pull Algorithm

```
1. Read .sg_vault/HEAD → parse vault key
2. Download settings transfer → decrypt → get tree_transfer_id
3. Download tree transfer → decrypt → get file inventory
4. Diff remote tree vs .sg_vault/index.json → identify remote changes
5. For each added/modified remote file:
   a. GET /transfers/download/{id} → get encrypted bytes
   b. Decrypt with derived key
   c. Write to local filesystem
   d. Compute local hash
6. For each deleted remote file:
   a. Delete from local filesystem
7. Create new folders, remove empty folders
8. Update .sg_vault/index.json
9. Update .sg_vault/FETCH_HEAD
```

### 8.4 Conflict Handling (MVP)

**Strategy: Last writer wins with conflict backup** (simple, predictable, git-inspired)

If the same file was modified both locally and remotely:
1. Rename local version to `filename.conflict.local`
2. Download remote version as `filename`
3. Mark in index as `conflict`
4. Print warning: `CONFLICT: documents/notes.md — local version saved as notes.md.conflict.local`

User resolves manually, then runs `sg-vault push`.

## Part 9: Key Design Decisions Summary

| # | Decision | Rationale |
|---|---|---|
| D1 | `.sg_vault/` as vault identity | Mirrors `.git/`. Single source of local state. |
| D2 | Use Transfer API, not Vault Admin API | Browser interoperability. Zero-knowledge purity. |
| D3 | Separate core/ from cli/ | Core is reusable as a library. CLI is a thin wrapper. |
| D4 | INI config format | Git convention. Human-readable. No extra dependency. |
| D5 | `clone` not `connect` | Git vocabulary. Users already know what `clone` means. |
| D6 | Empty folders explicitly created | Improvement over git. Vault tree has first-class folders. |
| D7 | Vault key stored in HEAD (opt-out) | Convenience by default (like git credential store). Security-conscious users can opt out. |
| D8 | SHA-256 for local change detection | Fast, reliable, same hash algorithm used throughout the project. |
| D9 | httpx not requests | Async-capable, streaming support for large files, modern API. |
| D10 | Type_Safe for models, never Pydantic | Project-wide rule. Non-negotiable. |

## Part 10: Open Questions for Human

1. **Package location:** Should `sg_vault/` live inside this repo (monorepo) or as a separate repo (`SGraph-AI/sg-vault`)? The brief mentions PyPI distribution. Recommend: start in this repo under `sg_vault/`, split out when ready.

2. **Vault key mutability:** Every push changes the settings_transfer_id, which changes the vault key. The browser handles this because it holds the key in memory. For multi-user CLI usage, how do users get the updated key? Options:
   - a) Store a "vault pointer" transfer that always points to the latest settings (adds one extra hop but key never changes)
   - b) Accept that the key changes and users must share the latest key (git model — the remote URL doesn't change, but HEAD moves)
   - c) Use the Admin Vault API's index endpoint as a stable pointer (breaks zero-knowledge purity)

   **Architect recommendation:** Option (b) for MVP. The `.sg_vault/HEAD` auto-updates on push. The "latest key" is always in HEAD. For sharing, we can add option (a) later.

3. **File size limits:** The Transfer API supports direct upload (<6MB) and presigned multipart upload (up to 1GB+). Should the CLI support both from day one, or start with direct-only and add multipart in Phase 2?

   **Architect recommendation:** Support both from day one. The presigned upload endpoints already exist and are tested. Large files are a confirmed use case (300MB+ validated).

4. **Where do tests live?** `tests/unit/sg_vault/` in this repo? Or a separate test directory within `sg_vault/`?

   **Architect recommendation:** `tests/unit/sg_vault/` following the existing repo convention.
