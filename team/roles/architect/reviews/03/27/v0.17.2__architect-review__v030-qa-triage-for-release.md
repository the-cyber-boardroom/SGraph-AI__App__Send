## Risk Assessment for Recommended Changes

### Why these are safe

1. **CSS-only changes** (G-001 completion, U-001, U-003, U-010, D-001, FT-013) — zero functional impact, zero state changes, visually verifiable
2. **Isolated JS changes** (U-030, FT-019, FT-016) — single method or single line, no cross-component side effects
3. **Template changes** (U-021, U-028) — removing duplicate HTML elements, no logic changes
4. **Event handler changes** (U-006, G-002 share, U-017, U-019/U-020, D-006b) — additions or modifications to existing handlers, no state model changes

### What could go wrong

| Risk | Mitigation |
|---|---|
| U-017 hide sections breaks re-display on back | Test: go to Step 2, back to Step 1 — sections must reappear |
| U-030 email fix uses wrong URL | Test: all 3 share modes must produce correct email content |
| FT-016 sort changes visual order unexpectedly | Sort is alphanumeric ascending — matches user expectation |
| FT-019 changing default breaks folder uploads | Gallery is still an option; default changes to browse which is more stable |

## Decision for Dinis

**Recommended:** Ship v0.3.0 with the 17 fixes above (19 minus the 2 already done). This addresses:
- The one security bug (U-030)
- The stability issue (FT-019 — gallery default)
- All 3 regressions from v0.2.x (G-001, G-002, U-001)
- 12 quality-of-life improvements that are all safe

**What we're NOT shipping:** Folder drag-and-drop fix (FT-015), PDF print (FT-011), text loss on back (U-014), loading page improvement (D-009), brand/privacy restoration (D-002). These need more investigation or design work.

**FT-015 (folder drop) note:** This was listed as P1 by QA, but the fix involves debugging async `webkitGetAsEntry()` / `FileSystemDirectoryReader` behaviour — not a quick fix, and a wrong change here could break file drops entirely. Users can work around it by selecting only files (not folders) or using the "Test folder" button.

**U-014 (text loss) note:** Also P1 by QA, but requires state preservation across wizard steps. The safe approach is to implement this as part of the SPA/History API architecture (U-015), not as a point fix that might cause other state issues.
