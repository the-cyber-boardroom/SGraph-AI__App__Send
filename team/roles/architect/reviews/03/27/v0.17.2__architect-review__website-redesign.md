# Architect Review — Website Redesign
**v0.17.2 | 27 March 2026**

Scope: sgraph.ai marketing site structural redesign. This review covers IA changes, content migration, locale architecture, component integration, SEO, and migration risk. The product (send.sgraph.ai) is out of scope except where the `<sg-public-viewer>` component bridges the two.

---

## 1. Current IA Audit

### URL Structure
```
/{locale}/                    Root per locale (16 locales)
/{locale}/product/            Product overview
/{locale}/pricing/            Credits and plans
/{locale}/architecture/       Technical depth (server-side)
/{locale}/agents/             Agent profiles × 3
/{locale}/early-access/       Sign-up capture
/{locale}/payment/            Payment flow
/{locale}/contact/            Contact form
```

Root `/` redirects to `/{locale}/` via JS autodetect. No build step — pure S3 + CloudFront, `_common/` for shared assets.

### What Works
- Flat URL hierarchy is CF-cache-friendly (no query strings, clean paths)
- `/{locale}/` prefix gives search engines unambiguous language signals via hreflang
- `_common/` pattern avoids asset duplication across locales
- Static architecture means zero server cost and predictable cache behaviour

### What Is Broken
- `/product/` is a shallow marketing page for a product already live at send.sgraph.ai — it duplicates without adding depth
- `/architecture/` is technically accurate but positioned wrongly: it reads as an internal design doc, not a trust-building security page
- `/agents/` has no clear audience on the marketing site — it describes the AI team building the product, not the product itself; this creates confusion for visitors expecting product features
- No `/features/` page means shipped functionality (ZK encryption, gallery, folder browser, SgPrint, 3 share modes, 6-step wizard, receipt confirmation, transparency panel) is invisible to search and to first-time visitors
- No `/why/` landing pages means zero presence on workflow-specific search terms (e.g. "send encrypted files to client", "zero knowledge file sharing for legal")
- Navigation hierarchy is flat — all pages carry equal weight, which dilutes priority signals for crawlers

## 2. Content Removal Decisions

### `/agents/` — Move to qa.send.sgraph.ai
**Decision: Remove from main site.**

Rationale: Agent profiles describe the AI development team, not the product. A marketing visitor landing on `/agents/` has no frame of reference; it reads as unexplained content. The QA/docs site (qa.send.sgraph.ai) is the right home — it serves developers and technically curious users who can appreciate process transparency. Redirect `/{locale}/agents/*` → `https://qa.send.sgraph.ai/team/` (302 initially, 301 after QA site is confirmed stable).

### `/architecture/` — Transform into `/security/`
**Decision: Replace with `/security/`, do not redirect to qa.send.sgraph.ai.**

Rationale: The architecture content has genuine trust-building value for security-conscious buyers (CISO audience). The problem is framing, not content. Rename, restructure around CISO concerns (threat model, key handling, server-side guarantees, audit trail, compliance posture), and keep it on the marketing site. Redirect `/{locale}/architecture/` → `/{locale}/security/` (301 permanent).

The QA site can carry a deeper technical architecture document for developers. The marketing `/security/` page is the condensed, trust-first version.

### `/product/` — Merge into homepage
**Decision: Deprecate `/product/`, absorb content into homepage sections.**

Rationale: With a live product at send.sgraph.ai and a new `/features/` page, `/product/` is redundant. The homepage should carry the top-level pitch (what it is, who it's for, primary CTA). Redirect `/{locale}/product/` → `/{locale}/` (301 permanent). Extract any unique content from `/product/` that is not yet on the homepage before removing.

## 4. Locale/i18n Architecture

### Keep `/{locale}/` Prefix
**Recommendation: Retain the prefix architecture unchanged.**

Rationale:
- hreflang signals are unambiguous — search engines receive one clear URL per locale per page
- CloudFront can cache per-path without cookie or Accept-Language variation, keeping edge cache hit rates high
- `_common/` shared assets remain locale-agnostic; only HTML and JSON string files are duplicated
- Removing the prefix to use Accept-Language negotiation would break cache efficiency and add complexity for zero SEO gain

### SEO Impact of Keeping Prefix
Positive: 16 URL-distinct versions of each page can rank independently in local SERPs. Hreflang prevents duplicate content penalties. Canonical URLs per locale are straightforward.

Negative (current): JS-only root redirect (`/` → `/{locale}/`) means crawlers that do not execute JS may not discover locale variants. Mitigation: add a server-side (CloudFront Function) redirect at root to a default locale (en-gb or en-us based on Accept-Language header), with `<link rel="alternate" hreflang="x-default">` pointing to the en-gb variant.

### Maintenance Tradeoff
16 locales × 8 pages = 128 HTML files plus JSON string files. The current `data-i18n` attribute pattern with JSON string files keeps this manageable. The `/why/` workflow pages add 5 × 16 = 80 HTML files — templated generation is advisable but not a blocker given the static architecture.

For the `/security/` page: CISO-depth content in 16 languages is a significant translation burden. Ship English first (en-gb, en-us), then prioritise by market size. Mark untranslated locales as `hreflang="en"` temporarily rather than shipping machine-translated copy.

## 6. `<sg-public-viewer>` Integration

### How It Works in Static S3/CF Context
The component is a vanilla JS Web Component with zero framework dependencies. It can be loaded as a `<script type="module">` from the `_common/` asset path, served directly from S3 via CloudFront. No server-side rendering required.

```html
<!-- Loaded from _common/ -->
<script type="module" src="/_common/components/sg-public-viewer.js"></script>

<!-- Embedded in homepage -->
<sg-public-viewer vault-id="demo-vault-001" locale="en-gb"></sg-public-viewer>
```

The component fetches encrypted content from send.sgraph.ai using a public vault ID, decrypts in-browser using a bundled demo key, and renders the result. From CloudFront's perspective this is a static page fetching a cross-origin API — CORS headers on send.sgraph.ai must allow `sgraph.ai` origin.

### Demo Vault Architecture
Demo vaults are pre-loaded into send.sgraph.ai with known public keys embedded in the component. Three demo vault types are recommended:

| Vault ID | Content Type | Purpose |
|----------|-------------|---------|
| `demo-gallery-001` | 4-image gallery | Demonstrates gallery + ZK encryption visually |
| `demo-doc-001` | Single PDF | Demonstrates SgPrint / document sharing |
| `demo-folder-001` | Folder with 3 files | Demonstrates folder browser |

Demo vaults must be marked read-only and rate-limited at the API level. They should not appear in analytics or count against transfer limits. The demo key is public by design — the security model is "content is intentionally public, key handling is still demonstrated correctly."

### Fallback
If the component fails to load or the vault is unreachable, the element must degrade gracefully to a static screenshot with a "Try it live" link to send.sgraph.ai. This is a hard requirement — a broken embed on the homepage is a conversion killer.

| `<sg-trust-badges>` | Homepage, `/security/` | Zero cookies, ZK encryption, no accounts — composable badge row |

### IFD Alignment
Web Components on the marketing site must follow IFD methodology: zero framework dependencies, surgical versioning (`v0/v0.1/v0.1.0/`), `_common/` shared assets path. Components must not import from npm at runtime. Any component added to the marketing site should be reviewed for potential reuse on send.sgraph.ai before building a parallel implementation.

The `<sg-pricing-calculator>` is the only component that does not yet exist in either codebase — it is PROPOSED. All others are either shipped (sg-public-viewer) or trivial HTML patterns that can be componentised from existing markup.

## 8. SEO and Metadata Architecture

### Current State
Present: Organization schema, Open Graph, Twitter Card, hreflang, canonical URLs.
Missing: Product schema, FAQ schema, BreadcrumbList schema, structured data for `/security/` (no TechArticle or WebPage schema).

### Recommended Additions

**Product schema** — add to homepage and `/features/`:
```json
{
  "@type": "SoftwareApplication",
  "name": "SGraph Send",
  "applicationCategory": "SecurityApplication",
  "operatingSystem": "Web",
  "offers": { "@type": "Offer", "priceCurrency": "GBP" }
}
```

**FAQ schema** — add to `/security/` and `/features/`. Target 5-8 questions per page. FAQs that answer "does the server see my files?" have high-value featured snippet potential.

**BreadcrumbList schema** — add to all non-root pages. Pattern: `sgraph.ai > Features > Gallery`. Low effort, meaningful crawl signal.

**HowTo schema** — add to homepage for the 6-step upload wizard. Step-by-step structured data is eligible for rich results.

**WebPage + Article schema** — add `dateModified` to all pages. CloudFront serves static files; the modification date must be maintained as a meta tag since Last-Modified headers reflect S3 object dates, not content intent.

### Locale-Specific Metadata
Each locale variant needs its own `og:locale` tag and locale-specific `og:title`/`og:description`. Currently these may be using the same string file values — confirm that OG tags are populated from the i18n JSON, not hardcoded in HTML templates.

### robots.txt and Sitemap
A locale-aware sitemap (`sitemap.xml` with `<xhtml:link>` alternate entries) is absent or not verified. This should be generated and submitted to Google Search Console for all 16 locales. Given the static architecture, a build-time or manually maintained sitemap is acceptable.

## 9. tools.sgraph.ai

### What Lives There
PROPOSED — does not exist yet. The brief mentions a speed test tool as the primary use case. Additional candidates: encryption benchmarking tool, transfer size calculator, latency test to eu-west-2 edge.

### Subdomain vs Path Tradeoff

| Factor | Subdomain (tools.sgraph.ai) | Path (sgraph.ai/tools/) |
|--------|---------------------------|------------------------|
| SEO authority | Split from main domain | Inherits main domain authority |
| Deployment isolation | Independent CF distribution | Same distribution, path-based routing |
| CORS | Needs explicit allow on send.sgraph.ai | Same origin as marketing site |
| Cookie/session isolation | Hard isolation | Shared origin (problematic if tools need state) |
| Development speed | Separate repo/deploy | Same static S3 bucket |

**Recommendation: Use `sgraph.ai/tools/` initially.** The subdomain offers deployment isolation but costs SEO authority at a stage when the domain is still building link equity. Tools that need hard isolation (e.g. ones that make API calls to send.sgraph.ai) can be moved to a subdomain later when the authority argument is less asymmetric. If the speed test tool needs to make cross-origin calls to send.sgraph.ai, CORS is equally manageable on either pattern.

Revisit subdomain if: tools require server-side execution, separate CI/CD, or the content is sufficiently distinct to warrant separate brand treatment.

## 10. Migration Risk Register

| Risk | Severity | Likelihood | Mitigation |
|------|----------|-----------|------------|
| 301 redirect from `/product/` breaks inbound links | Medium | Medium | Audit inbound links before redirecting; use 302 for 2 weeks, then 301 |
| 301 redirect from `/architecture/` loses SERP rankings before `/security/` establishes them | High | Medium | Deploy `/security/` before removing `/architecture/`; redirect only after `/security/` is indexed |
| 302 redirect from `/agents/` to qa.send.sgraph.ai — cross-domain permanent redirect risks losing link equity | Low | Low | `/agents/` has negligible inbound links; proceed with 302 then 301 after QA site is stable |
| CF cache stale after page rename | Medium | High | Invalidate CF cache paths explicitly on deploy; do not rely on TTL expiry for renamed paths |
| `<sg-public-viewer>` CORS misconfiguration breaks homepage demo | Critical | Low | Pre-deploy CORS verification step in CI; test `sgraph.ai` as origin against send.sgraph.ai staging |
| Demo vault content expired or deleted | High | Medium | Demo vaults tagged with `type=demo`; exclude from TTL/expiry logic in Storage_FS |
| Demo vault key embedded in component JS — scraped and abused | Low | Low | Content is intentionally public; rate-limit demo vault reads at API level regardless |
| 16-locale expansion of `/why/` pages (80 new HTML files) creates i18n debt | Medium | High | Ship en-gb and en-us first; add locales incrementally; avoid shipping untranslated pages |
| Sitemap missing locale alternates causes crawl budget waste | Medium | Medium | Generate sitemap with `xhtml:link` alternates before submitting to Search Console |
| JS-only root redirect not followed by crawlers | Medium | Low | Add CloudFront Function redirect at root; add `x-default` hreflang |
| `/security/` ships before translation — CISO visitors in non-EN locales see English | Low | High | Acceptable trade-off; mark non-EN locales with `hreflang="en"` temporarily; document as known gap |

## Summary Decisions

1. Remove `/agents/` from main site — redirect to qa.send.sgraph.ai
2. Replace `/architecture/` with `/security/` — same content, CISO framing, 301 redirect
3. Deprecate `/product/` — absorb into homepage, 301 redirect
4. Add `/features/` and `/why/{workflow}/` (×5 pages, en-gb/en-us first)
5. Keep `/{locale}/` prefix architecture — add CloudFront Function for root redirect
6. Integrate `<sg-public-viewer>` on homepage with 3 demo vaults and graceful fallback
7. Add Product, FAQ, BreadcrumbList, and HowTo structured data
8. Use `sgraph.ai/tools/` path initially rather than `tools.sgraph.ai` subdomain
9. Generate locale-aware sitemap before next SEO push
10. Ship `/security/` before removing `/architecture/` to avoid ranking gap
