## Critical Architecture Decisions

### 1. Self-Describing File IDs (v0.16.3 — 03/17)

The vault redesign introduces a new ID format: `{type}-{derivation}-{mutability}-{hex_id}`.

| Component | Values |
|-----------|--------|
| **Type** | `obj`, `ref`, `idx`, `key`, `pkg` |
| **Derivation** | `pid` (deterministic HMAC), `cas` (content-addressed), `rnd` (random) |
| **Mutability** | `imm` (immutable), `snw` (single-writer), `muw` (multi-writer) |

**Impact:** Eliminates dual-write hack, enables storage-agnostic backends, allows mutability-based caching at the server level.

**Verdict:** Sound. The self-describing format carries enough metadata for any storage backend to make intelligent decisions without understanding vault semantics. The 11-phase CLI implementation plan is well-structured.

### 2. Sub-Tree Model (v0.16.3 — 03/17)

Replaces flat tree (one tree per commit listing all files) with hierarchical trees (one tree per directory). Enables on-demand loading — vault open requires only 3 requests (ref + commit + root tree).

**Impact:** Reduces request count from O(total_files) to O(navigated_depth). Critical for large vaults.

**Verdict:** Correct architectural choice. The simulation document (v0.16.3) proves the request budgets with real IDs from a live vault.

### 3. Four-Layer Security Model (v0.16.3 — 03/17)

Reveals the vault protocol has four independently activatable security layers:

| Mode | Layers | Use Case |
|------|--------|----------|
| A (Full zero-knowledge) | 1+2+3+4 | Current SG/Send — everything encrypted |
| B (Structural visibility) | 1+2+4 | Server sees graph structure, not content |
| C (Signed plaintext) | 1+2 | Everything visible, provenance proven |
| D (Pure VCS) | 1 only | Lightweight version control |

**Impact:** The codebase already supports all four modes. Mode B enables server-side search/diff without compromising content confidentiality. Mode C is "signed Git" — a distinct product.

**Verdict:** Architecturally elegant. No code changes needed — just different client-side decisions about what to encrypt. PROPOSED: Only Mode A is currently implemented.

### 4. PKI Operating Modes (v0.13.32 — 03/14)

Four provenance levels from anonymous to countersigned:

| Mode | Keys Required | Provenance |
|------|--------------|------------|
| 1 (Anonymous) | Vault key only | None — current v0.5.x |
| 2 (Device) | Branch key | "This device made this commit" |
| 3 (Author) | + User key | "This person made this commit" |
| 4 (Attested) | + Third-party | "This person, verified by X" |

**Phasing decision:** Mode 2 now, Mode 3 when needed, Mode 4 for regulated use cases.

**Verdict:** Correct phasing. Reserved schema fields (`author_key_id`, `author_signature`, `attestations`) prevent future migration pain.

### 5. Agent-to-Agent Communication (v0.16.17 — 03/17)

Uses vaults as a signed message plane. Agents send EML-format messages signed with private keys and encrypted for specific recipients. Sequential enforcement via cryptographic handoffs.

**Impact:** Prompt injection defense through cryptography — agents literally cannot read messages not encrypted for them. Message graph enables workflow debugging.

**Verdict:** Architecturally sound. Reuses existing vault primitives. EML format is a pragmatic choice (standard, well-tooled, human-readable). PROPOSED — does not exist yet.

## Architecture Concerns

### 1. Three Incompatible Crypto Models

The cross-repo analysis (v0.16.3 debrief) revealed CLI, vault browser, and user vault UI use different encryption models. The vault redesign proposes converging all three on PBKDF2 passphrase + sub-tree. This is the right direction but a breaking change — existing vaults cannot be read by new code.

**Risk:** Migration path for existing vaults is undefined. One-time migration script needed.

### 2. Storage Backend Mapping is Speculative

The 6-backend mapping (FS, S3, DynamoDB, PostgreSQL, Redis, Graph DB) is elegant but only FS and S3 are implemented. DynamoDB is flagged as "ideal fit" but unproven.

**Recommendation:** Keep this as concept documentation only. Do not architect for backends that have no customer demand.

### 3. Request Budget for Large Vaults

The simulation shows 3 requests for vault open, but a vault with 100 directories at root level still requires 100+ requests to fully browse. Batch read helps but doesn't eliminate the fan-out.

**Recommendation:** The `idx-pid-muw-{HMAC}` branch index (proposed) would allow vault overview in 1 request. Prioritise this.
