## 1. Executive Summary

This batch of 13 briefs represents the most ambitious expansion of SGraph Send since the project's inception. The briefs collectively describe a transition from a single-purpose encrypted file transfer tool into a **full-spectrum encrypted collaboration platform** with data rooms, ephemeral secrets, password management, browser transparency tooling, PKI infrastructure, multi-language URL architecture, Lambda optimisation, email outreach workflows, and a security review framework.

The scale is significant: the briefs touch every layer of the stack (frontend, backend, infrastructure, security, marketing, research) and introduce at least 6 new user-facing features, 2 infrastructure overhauls, 1 security framework, and 2 research initiatives. If all were built simultaneously, this would be 3-4 months of focused engineering for a team of 3-5 developers.

**Three dominant themes emerge:**

1. **Platform expansion** -- SG/Send is no longer a single-product story. Data rooms, ephemeral secrets, password vaults, and PKI links are distinct product surfaces that share an encrypted-storage foundation.
2. **Infrastructure maturity** -- URL architecture, S3/CloudFront static assets, Lambda dependency optimisation, and the security review framework are all investments in the platform's ability to scale and harden.
3. **Go-to-market activation** -- The waiting list outreach, MCP email pipeline, and the M-008 milestone (AI-native encrypted delivery, now proven) create urgency to communicate the product story to real users.

The brief batch is strategically coherent but operationally overwhelming. **The primary risk is scope paralysis** -- trying to build everything and finishing nothing. This assessment provides a build order that delivers incremental value at every step.

The original assessment treats B2 as entirely new. The narrative is compelling but the product must deliver on each phase before claiming the next. Announcing "encrypted data rooms" is safe -- they exist. Announcing "ephemeral secrets" or "password management" before they work would damage credibility. **The narrative should follow the product, not lead it.** The current product already supports a strong story; the briefs extend it.

## 7. Risk Assessment

### Risk 1: Scope Creep (Severity: HIGH)

13 briefs is 13 potential scope-creep vectors. Each brief is well-defined individually, but the cumulative ambition is dangerous. The PKI brief (B6) alone contains 4 distinct features (PKI URLs, Chrome sync research, sender key bootstrapping, minimal viewers). The data room brief (B2) has 11 acceptance criteria and a complex key architecture.

**Mitigation:** Strict wave discipline. Only start Wave 2 after Wave 1 deliverables are shipped and tested. No feature jumping between waves.

### Risk 2: Data Room Enhancement Complexity (Severity: MEDIUM -- downgraded from HIGH)

**Correction from original:** The original assessment rated this HIGH because it described B2 as building data rooms from scratch. In reality, data rooms already work -- 12 room endpoints, room-view UI with file browser, invite system, audit trail, all tested. B2 proposes an enhanced key management layer (master key, per-user wrapping, admin CRUD). This is still significant work but builds on a proven foundation. Downgraded to MEDIUM.

**Mitigation:** Build B2 as an incremental enhancement to the existing room system. First version: admin manages symmetric keys per-room, per-user key wrapping. Do not redesign the room system. Leverage existing invite, membership, and audit infrastructure.

### Risk 3: URL Architecture as Bottleneck (Severity: MEDIUM)

B7 (URL architecture) is marked as a dependency for B1, B4, B5, and B6. If taken literally, no feature work starts until the URL architecture is in place. This would serialize everything.

**Mitigation:** The dependency is real for production deployment but not for development. Features can be built with the current Lambda-served URL structure and migrated to S3/CloudFront later. B7 Phase 1 is important but should not block feature development. Build features on the existing path, migrate to the new architecture in Wave 2.

### Risk 4: Security Debt Accumulation (Severity: HIGH)

Every new feature brief (B1, B2, B3, B4, B6) adds attack surface. The security review framework (B10) is correctly identified as P1 for starting, but the review will always trail the feature development. If the gap between feature velocity and security review grows too large, the product accumulates hidden risk.

**Verified attack surface:** The reality document lists 22 user-facing endpoints and 40+ admin endpoints. The security review should start with the reality document's endpoint inventory as its input, not a separate discovery process.

**Mitigation:** Make B10 a continuous parallel track, not a one-time project. Every feature session should include 30 minutes of security analysis for the code produced. The attack surface inventory (B10's first deliverable) should be done before B2 (data rooms) starts, because data rooms introduce the most complex new attack surface. The reality document provides the definitive endpoint list -- use it as the starting point.

### Risk 5: PKI Premature Investment (Severity: MEDIUM)

B6 (PKI links, key portability) is the most technically ambitious brief and has the most open research questions. The Chrome sync approach may not work. The key prefix routing requires a fully functional PKI ecosystem that does not yet exist. Building PKI infrastructure before the symmetric features (B1, B2) are proven and adopted would be premature.

**Reality check:** The PKI key registry already exists (publish, lookup, unpublish, list, transparency log -- 5 endpoints, tested). What B6 proposes is a URL-based discovery and viewer layer on top of this. The foundation is there, but the user-facing layer is entirely PROPOSED.

**Mitigation:** Classify B6 as Research/Wave 4. Extract the minimal viewers (which are independently valuable) as a separate P2 item. Do not invest in PKI URL routing until the symmetric product is being used by real users and they are asking for per-recipient encryption.

### Risk 6: Email Outreach Timing (Severity: MEDIUM)

The waiting list outreach (B9) should happen while users still remember signing up. Every week of delay reduces response rates. But sending outreach before the product has enough to show (beyond basic file transfer) risks disappointing users who expect more.

**Reality check:** The product already has more than basic file transfer. Data rooms, vaults, PKI registry, 8-language localization, AI-native delivery -- these are all working capabilities that can be highlighted in outreach. The product story is already compelling without the proposed features from this brief batch.

**Mitigation:** Wave 1 delivers B1 (one-time secrets) and B5 (storage viewer) alongside B9 (outreach). But even if B1 and B5 are not ready, the outreach email can highlight existing capabilities: encrypted file transfer, data rooms, encrypted vaults, AI-native delivery (M-008), and browser transparency. That is already enough for a compelling first contact.

### Risk 7: Market Risk -- Password Manager Incumbents (Severity: LOW)

B4 (password generator) positions SG/Send against LastPass, 1Password, and Bitwarden. These are mature products with millions of users. Competing directly on password management features would be suicide.

**Mitigation:** B4 is correctly positioned as an on-ramp, not a competitor. The password generator is a tool that leads to vault adoption. It should never be marketed as "SG/Send Password Manager." The differentiation is zero-knowledge vault storage, not feature parity with incumbents.
