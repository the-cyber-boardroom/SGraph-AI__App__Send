## 1. Executive Summary

This batch of 13 briefs represents the most ambitious expansion of SGraph Send since the project's inception. The briefs collectively describe a transition from a single-purpose encrypted file transfer tool into a **full-spectrum encrypted collaboration platform** with data rooms, ephemeral secrets, password management, browser transparency tooling, PKI infrastructure, multi-language URL architecture, Lambda optimisation, email outreach workflows, and a security review framework.

The scale is significant: the briefs touch every layer of the stack (frontend, backend, infrastructure, security, marketing, research) and introduce at least 6 new user-facing features, 2 infrastructure overhauls, 1 security framework, and 2 research initiatives. If all were built simultaneously, this would be 3-4 months of focused engineering for a team of 3-5 developers.

**Three dominant themes emerge:**

1. **Platform expansion** -- SG/Send is no longer a single-product story. Data rooms, ephemeral secrets, password vaults, and PKI links are distinct product surfaces that share an encrypted-storage foundation.
2. **Infrastructure maturity** -- URL architecture, S3/CloudFront static assets, Lambda dependency optimisation, and the security review framework are all investments in the platform's ability to scale and harden.
3. **Go-to-market activation** -- The waiting list outreach, MCP email pipeline, and the M-008 milestone (AI-native encrypted delivery, now proven) create urgency to communicate the product story to real users.

The brief batch is strategically coherent but operationally overwhelming. **The primary risk is scope paralysis** -- trying to build everything and finishing nothing. This assessment provides a build order that delivers incremental value at every step.

## 6. Product Narrative Assessment

### Where We Were: "Encrypted WeTransfer"

Until M-008, SG/Send's story was simple: send encrypted files. The server never sees the plaintext. Better than WeTransfer. End of story.

This is a defensible but limited narrative. It competes on a single axis (privacy) in a crowded market (WeTransfer, Firefox Send, OnionShare, Tresorit). The differentiation is real but the user acquisition path is narrow: you must already care about encryption to use it.

### Where This Brief Batch Takes Us: "Encrypted Everything Platform"

The 13 briefs collectively describe a different product:

| Capability | Brief | What It Means |
|------------|-------|---------------|
| **Ephemeral secrets** | B1 | Competing with OneTimeSecret, PrivNote -- broader utility, bigger audience |
| **Team data rooms** | B2 + B3 | Competing with Tresorit, Virtru, Intralinks -- enterprise collaboration |
| **Password management** | B4 | Competing with LastPass, Bitwarden -- consumer vault, massive TAM |
| **Browser transparency** | B5 | Unique differentiator -- nobody else shows you what they store |
| **PKI infrastructure** | B6 | Long-term moat -- web-of-trust bootstrapped through usage |
| **AI-native delivery** | R12 + M-008 | Category creation -- "AI-native zero-knowledge file delivery" |
| **Agent email** | B9 + R13 | The AI agents are the team -- visible, reachable, operational |

The narrative arc is:

```
Phase 1 (now):     "Send files, nobody can read them"          (single tool)
Phase 2 (Wave 1):  "Share secrets, manage passwords, see       (tool suite)
                     what we store -- all zero-knowledge"
Phase 3 (Wave 2):  "Collaborate in encrypted data rooms        (platform)
                     with AI agents that work for you"
Phase 4 (Wave 4):  "A PKI-native encrypted internet layer      (infrastructure)
                     bootstrapped through usage"
```

### The Risk in the Narrative

The narrative is compelling but the product must deliver on each phase before claiming the next. Announcing "encrypted data rooms" before they work, or "PKI infrastructure" before it is real, damages credibility permanently. **The narrative should follow the product, not lead it.** Each wave delivers a credible story increment:

- Wave 1: "Secure tools for everyone" (secrets, storage viewer, password generator)
- Wave 2: "Encrypted team collaboration" (data rooms)
- Wave 3: "The AI-native collaboration platform" (preview, editing, agent workflows)
- Wave 4: "Your encrypted identity layer" (PKI)

## 7. Risk Assessment

### Risk 1: Scope Creep (Severity: HIGH)

13 briefs is 13 potential scope-creep vectors. Each brief is well-defined individually, but the cumulative ambition is dangerous. The PKI brief (B6) alone contains 4 distinct features (PKI URLs, Chrome sync research, sender key bootstrapping, minimal viewers). The data room brief (B2) has 11 acceptance criteria and a complex key architecture.

**Mitigation:** Strict wave discipline. Only start Wave 2 after Wave 1 deliverables are shipped and tested. No feature jumping between waves.

### Risk 2: Data Room Complexity (Severity: HIGH)

The symmetric key data room (B2) is the most complex single feature in the batch. Master key architecture, per-user key wrapping, role-based permissions, admin CRUD, and integration with B1 (key sharing) and B3 (preview/editing) create a large surface area. If the data room absorbs too many sessions, it could delay everything else.

**Mitigation:** Build B2 as a hard MVP. First version: admin creates a room, creates users with keys, users can access a flat file list. No folders, no editing, no preview. Those come in B3 (Wave 3). The data room MVP should be 5-7 sessions maximum. If it looks like more, cut scope.

### Risk 3: URL Architecture as Bottleneck (Severity: MEDIUM)

B7 (URL architecture) is marked as a dependency for B1, B4, B5, and B6. If taken literally, no feature work starts until the URL architecture is in place. This would serialize everything.

**Mitigation:** The dependency is real for production deployment but not for development. Features can be built with the current Lambda-served URL structure and migrated to S3/CloudFront later. B7 Phase 1 is important but should not block feature development. Build features on the existing path, migrate to the new architecture in Wave 2.

### Risk 4: Security Debt Accumulation (Severity: HIGH)

Every new feature brief (B1, B2, B3, B4, B6) adds attack surface. The security review framework (B10) is correctly identified as P1 for starting, but the review will always trail the feature development. If the gap between feature velocity and security review grows too large, the product accumulates hidden risk.

**Mitigation:** Make B10 a continuous parallel track, not a one-time project. Every feature session should include 30 minutes of security analysis for the code produced. The attack surface inventory (B10's first deliverable) should be done before B2 (data rooms) starts, because data rooms introduce the most complex new attack surface.

### Risk 5: PKI Premature Investment (Severity: MEDIUM)

B6 (PKI links, key portability) is the most technically ambitious brief and has the most open research questions. The Chrome sync approach may not work. The key prefix routing requires a fully functional PKI ecosystem that does not yet exist. Building PKI infrastructure before the symmetric features (B1, B2) are proven and adopted would be premature.

**Mitigation:** Classify B6 as Research/Wave 4. Extract the minimal viewers (which are independently valuable) as a separate P2 item. Do not invest in PKI infrastructure until the symmetric product is being used by real users and they are asking for per-recipient encryption.

### Risk 6: Email Outreach Timing (Severity: MEDIUM)

The waiting list outreach (B9) should happen while users still remember signing up. Every week of delay reduces response rates. But sending outreach before the product has enough to show (beyond basic file transfer) risks disappointing users who expect more.

**Mitigation:** Wave 1 delivers B1 (one-time secrets) and B5 (storage viewer) alongside B9 (outreach). The outreach email can now highlight: encrypted file transfer (existing), one-time secrets (B1), browser transparency (B5), and the AI-native story (M-008). That is enough for a compelling first contact. Do not wait for data rooms.

### Risk 7: Market Risk -- Password Manager Incumbents (Severity: LOW)

B4 (password generator) positions SG/Send against LastPass, 1Password, and Bitwarden. These are mature products with millions of users. Competing directly on password management features would be suicide.

**Mitigation:** B4 is correctly positioned as an on-ramp, not a competitor. The password generator is a tool that leads to vault adoption. It should never be marketed as "SG/Send Password Manager." The differentiation is zero-knowledge vault storage, not feature parity with incumbents.

## 8. Connection to M-008

The M-008 milestone (first AI-generated, encrypted, email-delivered document) fundamentally changes the priority landscape for this brief batch. Here is how:

### M-008 Validates the Email Pipeline (B9 impact: HIGH)

The n8n + WorkMail + MCP integration (R13) is no longer a research question -- it is a proven capability. The waiting list outreach (B9) can proceed with confidence. The Sherpa Agent can compose and send emails autonomously. The GenAI automation research (R11) has already been partially answered: n8n is the working platform, WorkMail is the working backend.

**Priority change:** B9 goes from "needs research" to P1 "execute now."

### M-008 Creates an Urgency to Ship the Story

The LinkedIn announcement reached 22,755 followers. The response was positive ("This is REALLY cool!"). There is now a public expectation that SG/Send is doing interesting things. The waiting list users should hear from the project while this momentum exists.

**Priority change:** Comms and outreach are time-sensitive. The window for maximum impact on waiting list outreach is measured in days, not weeks.

### M-008 Proves MCP as a Distribution Channel

The MCP Skill definition (R12) formalises what M-008 demonstrated: Claude can autonomously create, encrypt, upload, and email files through SG/Send. This is not a feature -- it is a distribution channel. Every Claude.ai user with the MCP connector is a potential SG/Send user.

**Priority change:** R12 (MCP Skill formalisation) becomes P2 -- it should be production-ready so that MCP-connected users have a reliable experience.

### M-008 Validates the Agent Team Model

Six roles produced parallel documentation of the milestone. The agent team model (Conductor, Historian, Sherpa, Journalist, Architect, Cartographer) is not just a process -- it is a demonstrable capability. The outreach email (B9) can honestly showcase this: "This project is built by AI agents. Here's how."

**Priority change:** The Librarian section of the outreach email (B9) becomes more compelling. The M-008 story is the concrete example that makes "built by AI agents" believable.

### M-008 Does NOT Change Feature Priorities

M-008 is about delivery infrastructure, not user-facing features. The one-time secret link (B1), data rooms (B2), password generator (B4), and all other feature briefs are unaffected by M-008 in terms of their product value or build complexity. The build order remains driven by user value and dependencies, not by delivery mechanism.

**The one exception:** If the outreach (B9) generates significant user interest, the priority of B2 (data rooms) could increase because it is the strongest enterprise feature in the batch.

## 10. Conductor's Recommendation

**Build Wave 1 now. Start outreach this week.**

The brief batch is strategically sound and the vision is compelling. The risk is execution paralysis, not strategic direction. The antidote is strict sequencing: Wave 1 delivers immediate value (secrets, transparency, outreach), Wave 2 delivers the flagship feature (data rooms), Wave 3 matures the platform, Wave 4 builds the future.

**Do not start B2 (data rooms) until B1 (secrets) has a working demo.** The data room MVP is the most complex item in the batch and will absorb all available attention. Give it a solid foundation by shipping B1 first.

**Do not start B6 (PKI) until real users are asking for it.** PKI is the right long-term direction but it is premature without a user base. Ship the symmetric products first. Let user feedback drive the PKI timeline.

**Start B10 (security review) in parallel with everything.** Every feature session should include security analysis. The attack surface inventory is non-negotiable before public launch.

**Ship the M-008 story through B9 immediately.** The momentum from the LinkedIn announcement is perishable. The waiting list users should hear from the project while the story is fresh.
