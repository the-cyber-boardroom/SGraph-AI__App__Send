## Gap Analysis

### G1 — CRITICAL: No `LAMBDA_BASE64_LIMIT` (413) test

The `read-base64` endpoint has a size guard at 3.75 MB (`LAMBDA_BASE64_LIMIT = 3750000`). If a file exceeds this limit, the endpoint returns HTTP 413. **This code path has zero test coverage.** This is the only error-handling branch unique to `read-base64` vs `read`, and it protects against Lambda response size violations in production.

**Risk:** If the limit constant or comparison logic is changed, no test will catch the regression.

### G2 — CRITICAL: No full encrypt-write-read_base64-decode-decrypt round-trip test

The dev brief (v0.13.14) explicitly lists this as an acceptance criterion:

> "At least one passing test covering the full encrypt -> write -> read-base64 -> decode -> decrypt round-trip"

This test does not exist. The current `test__read_base64__existing` writes raw bytes and reads them back via base64, but it does not use `Vault__Crypto` to encrypt before writing or decrypt after reading. The round-trip test would validate the exact workflow that AI agents execute.

**Risk:** A subtle change to base64 encoding, key derivation, or the encryption layer could break the AI-native vault access path without any test detecting it.

### G3 — HIGH: No access token validation tests at route level

The `check_access_token` method in `Routes__Vault__Pointer` has 5 distinct code paths:

1. Admin service available + valid active token
2. Admin service available + missing token (401)
3. Admin service available + invalid/expired token (401)
4. Admin service unavailable (503)
5. Env var fallback (local dev mode)

None of these are tested at the route level. The current tests work because the test setup runs without an access token configured (local dev mode — path 5), which implicitly allows all writes. This means the access token enforcement code has **0% coverage**.

**Risk:** Token validation could be completely broken in production and no test would catch it.

### G4 — MEDIUM: No `content-type` assertion on `read-base64` response

The `read-base64` endpoint returns JSON (`application/json`) vs the `read` endpoint which returns `application/octet-stream`. No test verifies the content-type header on `read-base64` responses. This matters for MCP compatibility — MCP transports that check content-type would fail if the response were incorrectly typed.

### G5 — MEDIUM: No concurrent vault access tests

The session findings (v0.13.14) describe the scenario where multiple AI agents access the same vault simultaneously. No test exercises concurrent reads or concurrent writes to the same vault.

**Risk:** Under concurrent access, the in-memory storage might behave differently from S3 (eventual consistency, race conditions on manifest creation).

### G6 — LOW: ACL route tests all skipped

All 9 tests in `test_Routes__Vault__ACL.py` are skipped. ACL is scaffolded but not wired through the User Lambda. This is a known state, not a regression, but it represents 9 tests that provide no value until the feature is wired.

### G7 — LOW: No MCP tool manifest test

The dev brief requires `read-base64` to be tagged `vault` and appear in the MCP tool manifest. No test verifies the MCP tool registration. The MCP manifest is generated from the OpenAPI spec, so this would be an integration-level check.

## Test Plan: AI-Native Vault Access

The following tests are designed to validate the full AI-native vault access workflow as described in the session findings and round-trip debrief.

### Phase 1 — Unit Tests (In-Memory Stack, No Mocks)

All tests use `Service__Vault__Pointer` with in-memory `Storage_FS` and `Vault__Crypto` for key derivation. These can be added to `test_Routes__Vault__Pointer.py`.

#### T1: Full crypto round-trip via read-base64 (P0)

```python
def test__read_base64__crypto_roundtrip():
    """AC4: encrypt -> write -> read-base64 -> decode -> decrypt"""
    from sgraph_ai_app_send.utils.Vault__Crypto import Vault__Crypto
    from cryptography.hazmat.primitives.ciphers.aead import AESGCM
    import base64, os

    crypto = Vault__Crypto()
    keys   = crypto.derive_keys('test-passphrase', 'a1b2c3d4')

    # Encrypt a plaintext payload (simulating browser-side encryption)
    plaintext = b'{"version": 3, "tree": {"/": {"children": {}}}}'
    nonce     = os.urandom(12)
    aesgcm    = AESGCM(keys['read_key_bytes'])
    ciphertext = nonce + aesgcm.encrypt(nonce, plaintext, None)

    # Write ciphertext to vault
    self._write(vault_id  = 'a1b2c3d4',
                file_id   = keys['tree_file_id'],
                write_key = keys['write_key'],
                payload   = ciphertext)

    # Read via base64 endpoint
    response = self._read_base64(vault_id = 'a1b2c3d4',
                                  file_id  = keys['tree_file_id'])
    assert response.status_code == 200
    data = response.json()

    # Decode base64
    recovered_ciphertext = base64.b64decode(data['data'])
    assert recovered_ciphertext == ciphertext

    # Decrypt
    recovered_nonce = recovered_ciphertext[:12]
    recovered_ct    = recovered_ciphertext[12:]
    recovered_plain = aesgcm.decrypt(recovered_nonce, recovered_ct, None)
    assert recovered_plain == plaintext
```

#### T5: Multi-file vault tree walk via read-base64 (P1)

This simulates the AI-native pattern: read tree, parse tree, read individual files.

```python
def test__read_base64__tree_walk_pattern():
    """Simulate AI agent: write tree + files, then read tree via base64, parse, read each file."""
    import base64, json

    tree = json.dumps({"version": 3, "tree": {"/": {"children": {
        "file-a.txt": {"file_id": "aaa111bbb222", "size": 10},
        "file-b.txt": {"file_id": "ccc333ddd444", "size": 20}
    }}}}).encode()

    self._write(file_id='tree-file',    payload=tree)
    self._write(file_id='aaa111bbb222', payload=b'content-a-')
    self._write(file_id='ccc333ddd444', payload=b'content-b-longer-text')

    # Step 1: Read tree via base64
    tree_resp = self._read_base64(file_id='tree-file')
    tree_data = json.loads(base64.b64decode(tree_resp.json()['data']))
    children  = tree_data['tree']['/']['children']

    # Step 2: Read each file via base64
    for name, entry in children.items():
        file_resp = self._read_base64(file_id=entry['file_id'])
        assert file_resp.status_code == 200
        content = base64.b64decode(file_resp.json()['data'])
        assert len(content) == entry['size']
```

## Key Recommendations

1. **Immediately add T1 and T2.** These close the two critical gaps: the untested acceptance criterion (AC4) and the untested 413 error branch. Both are straightforward to implement with the existing test infrastructure.

2. **The `Vault__Crypto` class has no encrypt/decrypt methods.** It only derives keys. The round-trip test (T1) must use `cryptography.hazmat` directly (AESGCM) to simulate what the browser's Web Crypto API does. Consider adding an `encrypt`/`decrypt` convenience method to `Vault__Crypto` — this would benefit both testing and the `sg-send-cli` tool.

3. **Access token validation is a blind spot.** The `check_access_token` method has 5 code paths and 0 tests. In production, token validation routes through the admin service client. A test that wires a minimal admin service mock (or a test-mode admin client) would cover the critical auth enforcement path. This is arguably more important than the AI-native workflow tests for production safety.

4. **The ACL route tests (9 skipped) should either be un-skipped or removed.** Skipped tests provide false comfort in test counts. If the feature is not wired, the tests should be marked with a clear reason and tracked as a known gap.

5. **For Villager team:** When hardening for production, add concurrent access tests (T10-T12). The in-memory storage backend is single-process, but S3 has eventual consistency characteristics that could surface under concurrent AI agent access.
