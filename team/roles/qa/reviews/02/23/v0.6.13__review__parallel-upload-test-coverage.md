## Change Summary

The multipart S3 upload in `send-upload.js` was changed from sequential to parallel execution using a `Set` + `Promise.race` concurrency pool pattern. The static constant `PARALLEL_UPLOADS = 5` limits the number of concurrent part uploads.

**File changed:** `sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.4/components/send-upload/send-upload.js` (lines 481-532)

### Key Characteristics of the Implementation

| Aspect | Detail |
|--------|--------|
| Concurrency model | `Set` of active promises, capped at `PARALLEL_UPLOADS` (5) |
| Scheduling | `Promise.race(active)` — as soon as one slot frees, the next part starts |
| Completion | `Promise.all(active)` drains remaining in-flight after loop ends |
| Part ordering | Parts complete out of order but are stored in a pre-allocated indexed array (`completedParts[i]`) |
| Progress tracking | `partsCompleted` counter incremented per part; progress bar updated on each completion |
| Error handling | Any part failure aborts the entire multipart upload via `ApiClient.abortMultipart` |
| Threshold | Presigned multipart is used only when `encrypted.byteLength > MAX_FILE_SIZE_DIRECT` (5 MB) and capabilities indicate `multipart_upload` is available |

## Gap Analysis

### Critical Gaps (Must Test)

| # | Gap | Risk |
|---|-----|------|
| G1 | Concurrency pool correctness — does it actually limit to 5? | Could overwhelm S3 or the browser's connection limit, causing failures at scale |
| G2 | Part ordering — are parts assembled correctly when completing out of order? | Data corruption: wrong part order produces a corrupted file |
| G3 | Error propagation — does a single part failure abort the whole upload? | Orphaned multipart uploads on S3 (cost and security risk) |
| G4 | Progress bar accuracy — does it reach 90% (70 + 20) at completion? | Minor UX issue, but misleading progress is a trust problem |

### Important Gaps (Should Test)

| # | Gap | Risk |
|---|-----|------|
| G5 | ETag capture from S3 response headers | Silent failure if S3 returns unexpected headers |
| G6 | `completeMultipart` receives correct `completedParts` array | S3 complete call fails if parts array is malformed |
| G7 | Abort path is best-effort (catch-and-ignore inside abort) | If abort itself throws, the original error could be masked |
| G8 | `partSize` fallback when `_capabilities` is null | Defaults to 10 MB; should be tested explicitly |

### Low Priority Gaps (Nice to Test)

| # | Gap | Risk |
|---|-----|------|
| G9 | Progress label i18n interpolation (`current` / `total` params) | Broken label text in non-English locales |
| G10 | DOM query for progress bar during upload (`.progress-bar__fill`) | If DOM structure changes, progress silently stops updating |

## Recommended Test Plan

### 1. Unit Tests (JavaScript — e.g., Vitest or Jest with jsdom)

These tests isolate the `_uploadViaPresigned` method and the concurrency pool logic.

#### 1.1 Happy Path

| Test ID | Scenario | Setup | Assertion |
|---------|----------|-------|-----------|
| U-HP-1 | File with exactly `PARALLEL_UPLOADS` (5) parts | Mock `ApiClient` methods; 50 MB file with 10 MB parts | All 5 `uploadPart` calls made; `completedParts` has 5 entries in correct order; `completeMultipart` called once |
| U-HP-2 | File with fewer parts than pool size (e.g., 2 parts) | 15 MB file, 10 MB part size | Only 2 concurrent uploads; no `Promise.race` needed; completes correctly |
| U-HP-3 | File with 1 part | 8 MB file, 10 MB part size | Single `uploadPart` call; no concurrency pool engaged |
| U-HP-4 | File with many parts (e.g., 20 parts) | 200 MB file, 10 MB part size | At most 5 concurrent at any time; all 20 complete; correct part ordering |
| U-HP-5 | Part size from capabilities | `_capabilities.max_part_size = 20 * 1024 * 1024` | Uses 20 MB parts, not default 10 MB |
| U-HP-6 | Part size fallback | `_capabilities = null` | Falls back to 10 MB default |

#### 1.2 Error Scenarios

| Test ID | Scenario | Setup | Assertion |
|---------|----------|-------|-----------|
| U-ER-1 | One part fails mid-upload (part 3 of 10) | `uploadPart` rejects on index 3 | `abortMultipart` called with correct `transferId` and `uploadId`; error re-thrown |
| U-ER-2 | First part fails immediately | `uploadPart` rejects on index 0 | Abort called; remaining parts not started (or their results ignored) |
| U-ER-3 | Last part fails | `uploadPart` rejects on final index | Abort called; prior completed parts are irrelevant |
| U-ER-4 | `initiateMultipart` fails | `initiateMultipart` rejects | No `uploadPart` calls made; no abort called (no `uploadId` to abort) |
| U-ER-5 | `completeMultipart` fails after all parts succeed | `completeMultipart` rejects | Abort called (because error is inside the try block); error propagated |
| U-ER-6 | Abort itself throws | `abortMultipart` rejects | Original upload error is still propagated (abort is best-effort) |
| U-ER-7 | Network timeout on one part | `uploadPart` hangs then rejects after timeout | Same as U-ER-1; abort called |

#### 1.3 Edge Cases

| Test ID | Scenario | Setup | Assertion |
|---------|----------|-------|-----------|
| U-EC-1 | File size exactly on part boundary | `encrypted.byteLength = partSize * N` exactly | `numParts` equals N (not N+1); last part is full-size, not zero-length |
| U-EC-2 | File size 1 byte over boundary | `encrypted.byteLength = partSize * N + 1` | `numParts` equals N+1; last part is 1 byte |
| U-EC-3 | Very large file (1 GB) | 1 GB buffer (or simulated), 10 MB parts = 100 parts | Concurrency stays at 5; all 100 parts complete; memory not exhausted by queuing |
| U-EC-4 | Encrypted size just above `MAX_FILE_SIZE_DIRECT` (5 MB + 1 byte) | Triggers presigned path | Presigned upload path is taken; single part of ~5 MB |
| U-EC-5 | Zero-byte encrypted payload | `encrypted.byteLength = 0` | `numParts = 0` (from `Math.ceil(0 / partSize)`); no parts uploaded; `completeMultipart` called with empty array — verify S3 accepts this or if it should be blocked earlier |

## Observations

1. **The concurrency pool pattern is sound.** The `Set` + `Promise.race` approach is a well-known pattern for bounded parallelism in JavaScript. The indexed array for `completedParts` correctly handles out-of-order completion.

2. **The abort path has a subtle issue.** If `completeMultipart` fails (line 526), the catch block (line 527-530) calls `abortMultipart`. But the multipart upload is already "complete" from S3's perspective if `completeMultipart` partially succeeded. The abort may fail silently (which is fine — it's best-effort), but this edge case should be documented and tested (U-ER-5).

3. **Zero-byte edge case is potentially problematic.** `Math.ceil(0 / partSize)` returns 0, meaning no parts are uploaded and `completeMultipart` is called with an empty `completedParts` array. S3's `CompleteMultipartUpload` requires at least one part. This would fail at the S3 level, but the error path should be verified (U-EC-5).

4. **Progress bar calculation assumes the 70-90% range.** The formula `70 + (20 * partsCompleted / partUrls.length)` is correct for the multipart segment of overall progress. However, if `partUrls.length` is 1, the jump is 70% to 90% in one step. This is functionally correct but visually abrupt.

5. **No retry logic.** A single part failure aborts the entire upload. For production reliability (Villager territory), adding per-part retry with exponential backoff would be valuable. This is not a bug in the current Explorer implementation, but a gap to flag for the Villager team.

6. **`partsCompleted` is not atomic.** In the current single-threaded JS event loop this is safe, but the increment (`partsCompleted++`) inside an async function that runs concurrently could be confusing to future maintainers. A comment clarifying JS single-threaded safety would help.
