## Brief #9: Read-Only Demo Environments — Test Strategy

**Priority:** HIGH | **QA role:** Supporting (dual-purpose with Sherpa)

### QA Agents as Demo Agents

The brief explicitly states that QA smoke test agents double as demo agents. This means QA infrastructure IS demo infrastructure — quality requirements are higher than typical test environments.

**Test strategy for each demo level:**

| Level | Test Type | What QA Validates |
|---|---|---|
| **Level 1** (single file) | Smoke test | Short code works, file decrypts, renders correctly, loads in under 3 seconds |
| **Level 2** (shared folder) | Integration test | Multiple files decrypt, folder navigation works, all file types render |
| **Level 3** (branded data room) | Visual regression | Branding renders correctly, logo loads, colour scheme applies, responsive layout |
| **Level 4** (use-case scenarios) | E2E workflow | NDA signing flow completes, contract exchange works end-to-end, audit trail populates |
| **Level 5** (live agents) | Continuous smoke test | Agents cycle reliably, timeline updates in real-time, no stale state |

### Capture-the-Flag Validation

The CTF element (link to next level inside current level) requires specific QA coverage:

- Each level's hidden link resolves correctly
- Short codes / access codes rotate as designed
- Forward links survive key rotation
- A broken link at any level blocks the entire evaluator journey — this is a P0 defect

### Test Automation Plan

```
Continuous (every 5 minutes):
  - Level 1 smoke: upload → encrypt → short code → decrypt → verify
  - Level 5 agent cycle: Agent A uploads → Agent B downloads → confirmation

Hourly:
  - Full journey: Level 1 through Level 5 automated walkthrough
  - CTF link validation: every inter-level link resolves

Daily:
  - Visual regression: screenshot comparison for all branded data rooms
  - Performance baseline: response times for each level
  - Key rotation test: rotate demo keys, verify all levels still work
```

### Monitoring Dashboard

QA smoke test results should be visible on the demo site itself as a confidence signal:

- "Last tested: 2 minutes ago"
- "All 5 levels operational"
- "Average response time: 0.4s"

## Brief #22: Agent Isolation — Security Testing Requirements

**Priority:** CRITICAL | **QA role:** Supporting (security testing)

### Architecture Claim Validation

The brief corrects a previous claim about encryption-as-isolation. QA must validate the corrected architecture:

**Security test matrix:**

| Test | What It Proves | Method |
|---|---|---|
| **Memory isolation** | Container A cannot read Container B's memory | Attempt `/proc/<pid>/mem` access across containers — must fail |
| **Key isolation** | Agent A's private key exists only in Container A | After Container A task completes, verify key is zeroed / container destroyed |
| **Network isolation** | Containers cannot reach each other over the network | Attempt TCP/UDP connection between agent containers — must be refused |
| **Filesystem isolation** | No shared filesystem between containers | Write a file in Container A, attempt to read from Container B — must fail |
| **Ephemeral destruction** | Container state does not persist after task | After container destruction, verify no key material in swap, /tmp, or core dumps |
| **Workflow integrity** | .sgworkflow is the only data channel | Monitor all I/O from containers — only .sgworkflow file transfers should occur |
| **Compromise containment** | Compromising one container does not affect others | Deliberately compromise Container A (simulate prompt injection), verify Containers B and C are unaffected |

### Security Test Automation

The isolation tests should run as part of CI for any change to the agent orchestration code:

```
Pre-merge (required):
  - Memory isolation: cross-container /proc access denied
  - Network isolation: inter-container connectivity refused
  - Key isolation: private key absent from non-owner containers

Post-deploy:
  - Ephemeral destruction: key material zeroed after task
  - Full compromise simulation: quarterly (manual + automated)
```

### Acceptance Criteria #8 Test Plan

Brief #22 AC #8 states: "Security test: compromise one container, demonstrate other containers are unaffected." QA's test plan:

1. Deploy 3-agent workflow across 3 containers
2. In Container A, simulate a prompt injection that attempts to: (a) read other containers' memory, (b) connect to other containers' network, (c) access the orchestrator's secrets
3. Verify all attempts fail with appropriate error codes
4. Verify Containers B and C complete their tasks successfully despite Container A's compromise
5. Verify the orchestrator detects Container A's anomalous behaviour

## Brief #4: File Security Scanning Pipeline — Test Vectors

**Priority:** CRITICAL | **QA role:** Supporting (test vector creation)

### Scanning Test Vector Library

QA must create a comprehensive library of test files for validating the scanning pipeline:

| Category | Test Vectors | Expected Result |
|---|---|---|
| **File type validation** | .exe renamed to .pdf, .js renamed to .png, polyglot files | Flagged: magic bytes mismatch |
| **Known malware hashes** | EICAR test file (standard AV test), known hash samples | Flagged: hash match against known malware DB |
| **Macro detection** | .docx with benign macros, .xlsm with Shell() calls, .docm with AutoOpen | Benign: warning. Malicious: flagged with detail |
| **Script detection** | HTML with inline JS, SVG with event handlers, PDF with /JavaScript action | Flagged: embedded script detected |
| **Archive inspection** | ZIP with nested ZIPs (3 levels), ZIP bomb (recursive compression), encrypted ZIP | Nested: scanned recursively. Bomb: detected and blocked. Encrypted: flagged as unscanned. |
| **LLM analysis** | Obfuscated PowerShell in .docx macro, base64-encoded payload in SVG, social engineering document | LLM correctly identifies risk pattern |
| **Clean files** | Standard .pdf, .docx, .xlsx, .png, .txt with no suspicious content | Pass all scans — zero false positives for clean test set |

### False Positive / False Negative Tracking

The scanning pipeline must be measured on both dimensions:

- **False positive rate target:** < 2% on the clean file test set (legitimate files incorrectly flagged)
- **False negative rate target:** 0% on the known-malicious test set (malware that passes scanning)
- **LLM analysis accuracy:** track per-model (Claude, GPT-4, local Ollama) for comparison

### Test Automation Plan

```
Per-commit (scanning pipeline code changes):
  - Run full test vector library against pipeline
  - Verify all expected detections fire
  - Verify no regressions in false positive rate

Weekly:
  - Update known malware hash database
  - Add new test vectors from real-world malware samples (sanitised)
  - LLM analysis comparison: run same vectors through multiple models

Monthly:
  - Full scanning performance benchmark (latency per file type and size)
  - Browser memory impact assessment (client-side scanning adds memory pressure)
```

## Cross-Brief QA Themes

### 1. Dual-Purpose Infrastructure
QA agents = demo agents. This raises the quality bar: test failures in demo environments are visible to evaluators. QA must treat demo environments as production.

### 2. Security Testing as First-Class QA
Two of three briefs (#22, #4) require security testing. QA must expand beyond functional testing to include security validation in the standard test suite.

### 3. No Mocks, No Patches
All tests use real implementations per project rules. Scanning pipeline tests use real files. Isolation tests use real containers. Demo tests use the real platform.
