## 1. QA Architecture Refactoring (Doc 304)

### The Proposal

All QA Playwright usage goes through a single `PlaywrightClient` abstraction with three modes:
- **Mode 1:** Local Docker container (dev/CI, no MITM proxy)
- **Mode 2:** Lambda invocation (scalable, no authenticated upstream proxy)
- **Mode 3:** EC2 instance with MITM sidecar (full capability, production monitoring)

### Assessment

This is the right architecture. The current state (implied by the brief: "no more ad-hoc Playwright setups") means QA tests are scattered and inconsistent in how they invoke Playwright. The `PlaywrightClient` abstraction applies the same pattern as Memory-FS: QA tests are ignorant of infrastructure.

**Acceptance criteria review:**

| Criterion | Assessment |
|-----------|------------|
| AC1: Mode 1 runs tests | Straightforward — local Docker + REST calls |
| AC2: Mode 2 runs tests | Requires Lambda session lifecycle decision (see below) |
| AC3: Mode 3 runs tests + MITM log | Depends on EC2 instance being provisioned via `provision_ec2.py` |
| AC4: Same test code on all modes | This is the design goal; must verify by running the same test suite against all three modes after implementation |
| AC5: Abstraction layer dispatch | Unit test: mock each mode, verify dispatch |
| AC6: MITM captures 100% of traffic | Verify by running a known sequence in Mode 3 and checking the MITM log against expected URLs |
| AC7: Hourly traffic generation running | Smoke test: wait one hour, check vault for new entries |
| AC8: Transfer + vault + screenshot in hourly run | Verify all four steps produce artefacts |
| AC9: Observability data flowing | Check dashboards after one cycle; confirm metrics appear |
| AC10: AMI launchable with service on standard port | Launch from AWS Console or CLI, verify health endpoint returns 200 |
| AC11: No ad-hoc Playwright setups remain | Code scan: `playwright` import outside `PlaywrightClient` should fail lint |

### Lambda Session Lifecycle (Critical Path for AC2)

The QA mode abstraction cannot be tested in Mode 2 until the Lambda session lifecycle design is resolved. Two options:

**Option A (recommended):** Make each Mode 2 invocation self-contained. A single Lambda call takes a complete sequence spec, executes it, and returns results. No session state persists between Lambda calls. This maps directly to the `screenshot_per_step` pattern in the screenshot API brief.

**Option B:** Maintain session state in DynamoDB/S3 across Lambda calls. Enables multi-call sessions but adds infrastructure and latency. Overkill for most QA use cases.

QA testing recommendation: implement and test Option A first. If multi-call sessions are needed, add Option B later without breaking existing tests.

### Mode 3 MITM Traffic Verification

The brief claims MITM captures "100% of Playwright traffic." QA must verify this formally:

1. Define a known test sequence (navigate to 3 known URLs, click 2 known elements)
2. Run in Mode 3
3. Parse the MITM log and verify all 3 URLs appear, plus expected resource requests
4. Flag any requests that appear in network DevTools but not in the MITM log

The "100% capture" claim is strong. Document the known limitations (QUIC/HTTP3 bypasses, pre-flight requests, browser internals) as accepted gaps rather than letting the claim go untested.

### Hourly Traffic Generation — Observability Design

The brief lists desired observability metrics:
- Files accessed per hour
- Vaults created per hour
- Transfer completion rate
- Response times per endpoint
- Error rates
- Screenshot diffs (visual regression)

The QA team needs to agree on the output format before implementation. Recommend: write each heartbeat run's results to a fixed-schema JSON file in a monitoring vault. The schema:

```json
{
  "run_timestamp": "2026-04-22T09:00:00Z",
  "mode": "lambda",
  "results": {
    "transfer_create": { "success": true, "duration_ms": 234 },
    "transfer_receive": { "success": true, "duration_ms": 189 },
    "vault_create": { "success": true, "duration_ms": 312 },
    "browser_verify": { "success": true, "screenshot_hash": "a3f..." }
  },
  "errors": []
}
```

Dashboards read from the monitoring vault. No custom backend needed.

### AMI Dual-Use (QA Infrastructure + Product)

The brief notes the AMI is both our QA infrastructure and a sellable product. From a QA perspective:
- The test suite that validates the QA infrastructure is the same acceptance test that validates the marketplace product
- This is a benefit: marketplace customers and internal QA are testing the same artifact
- Ensure the acceptance test suite (AC1–AC11) can run against a freshly-launched marketplace AMI with no modifications

## 2. Send/Receive JS API — Testability (Doc 302)

### Acceptance Criteria Coverage

The brief's acceptance criteria are testable in the browser via Playwright. The reference tool on tools.sgraph.ai provides the test surface.

Key test cases for QA:

| Test | Method |
|------|--------|
| `sendFile` → token → `receiveFile` round-trip | Create file, call API from console, verify file integrity |
| `sendFolder` (10 files) → `receiveFolder` | Upload 10 files, verify all 10 received, contents match |
| `<sg-send-drop>` emits `token-created` event | Playwright: drag file to component, listen for event |
| `<sg-send-receive>` emits `file-received` event | Playwright: paste token, listen for event |
| Token history persists in localStorage | Send file, refresh page, check localStorage key |

**Browser compatibility matrix:** the brief doesn't specify, but QA should test on Chrome and Firefox as minimum. The Credential Management API (used in the auth MVP `<sg-vault-picker>`) is Chromium-only; the send/receive API itself has no browser-specific dependencies and should work on both.

## 3. Playwright Screenshot API — Testability (Doc 311)

The brief's acceptance criteria are the test spec. All 10 criteria are automatable via API calls:

- AC1–AC7: direct HTTP calls to the Playwright service endpoints
- AC8–AC9: Playwright session navigating to the mini frontend, interacting with the URL input
- AC10: run existing Playwright session tests to confirm nothing regressed

Recommend: add the screenshot API tests to the existing Playwright service test suite (not a new file). The test file is `tests/unit/container/test_Container__App.py` or equivalent for the Playwright service.

## 4. Video Tools — Testability Concerns (Doc 303)

Video tools present testability challenges:
- Video processing (crop export, overlay export) requires visual inspection to verify quality
- "Screenshot diff" for visual regression is possible but requires a baseline to compare against
- Cross-browser: Safari OffscreenCanvas support is limited (no `OffscreenCanvas.transferToImageBitmap` support in some Safari versions)

**QA recommendation before Dev starts the video crop export:**
1. Run a browser compatibility matrix for OffscreenCanvas in Chrome/Firefox/Safari
2. Define acceptable degradation: does export fall back to a synchronous Canvas path in Safari, or is the feature Chrome/Firefox-only for now?
3. Document the decision as a known limitation if Safari is out of scope for MVP

## 5. Website Changes — Smoke Tests (Doc 309)

The existing smoke tests (health, auth, CORS, no-plaintext) cover the backend. The website frontend changes need a lightweight visual smoke test:
- Hero line appears above the fold (text check via Playwright)
- Token input is visible and functional (paste token, verify redirect to `/en-gb/browse/#token`)
- No broken links (automated link checker)

Add these to the deployment smoke test sequence for sgraph.ai.
