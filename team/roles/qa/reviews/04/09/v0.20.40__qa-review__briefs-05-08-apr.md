### Critical Pattern: Playwright Proxy Config

Chromium does NOT inherit `HTTPS_PROXY` env vars. The canonical pattern from doc 251:

```python
import os
from urllib.parse import urlparse

proxy_url = os.environ.get('HTTPS_PROXY')
parsed = urlparse(proxy_url)

browser = playwright.chromium.launch(
    headless=True,
    proxy={
        'server':   f'http://{parsed.hostname}:{parsed.port}',
        'username': parsed.username,
        'password': parsed.password,
    }
)
```

**This pattern must be in the QA framework** (`sg_send_qa/browser_framework.py` or equivalent). Every Playwright session that needs external access should use it.

## 2. QA Workflow for Offline Coding Sessions (Doc 247)

### What This Defines

The brief establishes a formal methodology for extracting value from the human's offline coding sessions. The key insight: **reading the final file state loses the narrative**. The commit sequence tells the story.

### QA Responsibilities Under This Workflow

1. **Identify commit range** (`git log --oneline {last_processed}..HEAD`)
2. **Read commits sequentially** (`git log --reverse -p {range}`)
3. **Extract `@QA` tagged comments** (`git diff {range} | grep -E '@(QA|Dev|Librarian|AppSec|Architect)'`)
4. **Produce test case list** from `@QA` tags
5. **Write session narrative** before per-agent briefs
6. **Track processing state** (last processed commit hash)

Doc 247 was triggered by a specific offline session focused on persisting browser instances across web server restarts. The QA team should:
1. Read the commit history for that session
