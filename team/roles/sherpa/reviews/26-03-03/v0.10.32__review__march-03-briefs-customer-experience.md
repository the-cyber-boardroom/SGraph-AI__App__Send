## Executive Summary

On 2 March 2026, SG/Send crossed the most important threshold in any product's life: strangers gave it money. Ten people completed a purchase flow, received tokens manually, and sent encrypted files. Several declined the Amazon voucher incentive. One customer requested a feature and it shipped in 90 minutes. The core product works. The purchase flow works. The manual onboarding works at this scale.

This review analyses the March 03 briefs through three lenses:

1. **Where we are now** -- mapping every user journey that exists today and identifying the friction, dead zones, and delight moments experienced by the first 10 customers
2. **Where the briefs want to take us** -- assessing gallery sharing as a growth lever, vault UX improvements for retention, and CLI sync for stickiness
3. **What we must do immediately** -- a concrete communication plan for the two customer groups, the signals to watch for product-market fit, and the experience risks that could damage our relationship with these first 10 customers

**The single most important finding:** The first 10 customers represent a fragile asset. They bought because of a personal relationship with the founder. Their continued engagement -- or disengagement -- over the next 14 days will determine whether SG/Send has product-market fit or just social capital. Every brief reviewed here proposes features that serve future users. The Sherpa's concern is that nobody is explicitly serving the users we already have.

## Part 1: User Journey Mapping -- What the First 10 Customers Actually Experienced

### Journey A: Purchase Flow (All 10 Customers)

| Step | Time | What Happened | Experience Quality |
|------|------|--------------|-------------------|
| A1 | T+0 | Received personal WhatsApp/LinkedIn message from founder | **Delight** -- personal, trusted source, clear ask |
| A2 | T+1min | Navigated to sgraph.ai | **Neutral** -- site loads, multilingual detected |
| A3 | T+2min | Found pricing page and Stripe checkout | **Smooth** -- no reported friction |
| A4 | T+3min | Completed payment (Apple Pay for some) | **Delight** -- Apple Pay especially smooth |
| A5 | T+3min | Landed on payment success page | **Neutral** -- page exists, content reviewed in previous Sherpa review |
| A6 | T+3-8min | **Waited for token** | **The Gap** -- but at 5-10 minutes, tolerable |
| A7 | T+8min | Received personal message with token + "try dropping some PDFs" | **Delight** -- human touch, fast response, specific guidance |
| A8 | T+10min | Entered token on send.sgraph.ai, uploaded first file | **Smooth** -- no failures reported |
| A9 | T+12min | Shared link with recipient | **Unknown** -- no data on recipient experience |

**Key observations:**

1. **The post-payment dead zone (A6) was only 5-10 minutes.** My previous review predicted this gap would be the Challenge-killer at scale. At 10 users with a dedicated human, it was manageable. The personal message with the token transformed the gap from "waiting in silence" into "receiving a personal welcome." This is a delight moment that automation will destroy unless carefully designed.

2. **The founder's message at A7 was doing the work of three things simultaneously:** token delivery, onboarding guidance ("try dropping some PDFs"), and relationship building. When this is automated, all three functions need to be preserved. A cold email with a token code is dramatically worse than a personal WhatsApp message.

3. **We have zero data on step A9.** Did recipients successfully decrypt and view the files? Did anyone struggle with the download page? Did anyone not bother clicking the link? The recipient experience is the product's first impression for potential new users, and we know nothing about it.

4. **No customer reported the product not working.** This is a stronger signal than it appears. Zero failures across 10 different users, different devices, different browsers, different file sizes means the engineering is genuinely solid. This is a delight moment that users do not consciously notice -- but they would immediately notice if it failed.

### Journey B: Feature Request to Shipped Feature (1 Customer)

| Step | Time | What Happened | Experience Quality |
|------|------|--------------|-------------------|
| B1 | T+0 | Customer reports: "You can't drop folders" | **Friction identified** -- real use case blocked |
| B2 | T+5min | Founder asks: "What's the workflow you want?" | **Delight** -- listened to, taken seriously |
| B3 | T+10min | Agreement on solution (folder to auto-zip) | **Delight** -- collaborative design |
| B4 | T+25min | Implementation plan shared on GitHub | **Delight** -- transparency, customer reviewed code |
| B5 | T+90min | Feature live in production | **Extraordinary delight** -- 90 minutes from report to shipped |

**This is the single most powerful customer experience story SG/Send has.** A customer reported a need and the product responded in 90 minutes. This is not just fast development -- it is a demonstration that the product listens. For a developer customer who reviewed the GitHub plan, this builds deep trust.

**The risk:** This sets an expectation. If the next feature request takes 3 weeks, the contrast will feel like regression. The Sherpa's recommendation is to be explicit about response times: "We shipped this one in 90 minutes because it was small and aligned with our roadmap. Not every request will be this fast -- but we will always listen and respond quickly."

### Journey C: The Recipient Experience (Unknown -- Critical Gap)

This journey has never been mapped because we have no data on it. But it is the most important journey for growth, because every recipient is a potential new user.

| Step | Time | What Happens | Experience Quality |
|------|------|-------------|-------------------|
| C1 | T+0 | Receives a link (WhatsApp, email, etc.) from the sender | **Unknown** -- depends on how sender frames it |
| C2 | T+1min | Clicks the link | **Unknown** -- does the link inspire trust or suspicion? |
| C3 | T+2min | Sees download page, enters decryption key | **Friction** -- the key is a hex string (or part of the URL hash). Not user-friendly |
| C4 | T+3min | File decrypts and displays/downloads | **Delight** (if it works) -- in-browser preview is impressive |
| C5 | T+3min | Sees SG/Send branding, "powered by" footer | **Potential interest** -- "what is this tool?" |
| C6 | T+4min | Decides whether to explore SG/Send further | **Unknown** -- is there a clear path from download page to "try this yourself"? |

**Critical friction points in the recipient journey:**

**C-F1: The decryption key is unfriendly.** The current system uses a hex string in the URL hash fragment. The gallery sharing brief proposes "friendly keys" (e.g., "blue-mountain-77") but this does not exist yet. For recipients who receive the link via a channel that strips URL fragments (some email clients, some corporate proxies), the key may be lost entirely, and they must enter it manually. A hex string entered manually is error-prone.

**C-F2: The download page has no "try this yourself" CTA.** After successfully decrypting and viewing a file, the recipient has just experienced the product working. This is the highest-intent moment for conversion. If there is no visible path to "send your own encrypted files," the moment passes. PROPOSED -- this does not exist on the current download page.

**C-F3: The sender must explain the product.** When sharing a link via WhatsApp, the sender must provide context: "This is SG/Send, it's encrypted file sharing. Click the link, the key is already in the URL (or here's the key: ABC123)." If the sender does not explain, the recipient sees an unfamiliar URL and may not click. There is no link preview metadata (Open Graph tags) that would explain the link in WhatsApp/iMessage/Slack previews. PROPOSED -- if this does not exist, it should be verified.

These users received free tokens during an earlier phase and may or may not have used the product since.

| Step | Time | What Happens | Experience Quality |
|------|------|-------------|-------------------|
| D1 | T+0 | Receives follow-up message: "Major update, new access code" | **Neutral to positive** -- depends on how long since last contact |

**Critical friction points:**

**D-F1: EAP users may not remember what SG/Send is.** If weeks have passed since their last interaction, the follow-up message needs to re-introduce the product, not just announce features. The proposed message in the brief starts with "quick update on SG/Send" -- this assumes the user remembers what SG/Send is and why they signed up.

**D-F2: No structured feedback collection.** The brief says "collect feedback from first 10 customers" as a Sherpa action. But there is no feedback form, no survey link, no in-product feedback mechanism. Feedback will be collected via WhatsApp/email replies, which is unstructured and hard to track. This is acceptable at 10-15 users but will not scale.

## Part 2: Gallery Sharing -- Growth Lever Assessment

### The Viral Loop Claim

The gallery sharing brief makes a bold claim: "This is the highest-potential viral loop SG/Send has. Every gallery shared is a product demo to every recipient."

**The Sherpa's assessment: This claim is correct, but the loop has three critical gates that will determine whether it actually functions.**

### Gate 1: Does the sender's experience justify the effort?

| Current flow (no gallery) | Proposed gallery flow |
|--------------------------|---------------------|
| Drop files, encrypt, get link | Drop folder of photos, encrypt, get link + friendly key |
| Share link via WhatsApp | Share link + key via WhatsApp |
| Recipient downloads zip, unzips locally | Recipient sees beautiful gallery in browser |

**Assessment:** The sender's effort is roughly equivalent in both flows -- the gallery adds value on the recipient's side, not the sender's side. The sender's motivation to use the gallery depends on: (a) wanting the recipient to have a better experience, and (b) the gallery being the default behaviour for image-heavy uploads, not an opt-in feature.

**Recommendation:** The gallery view should auto-activate when the zip contains predominantly image/video files (>50% by count). The sender should not need to choose "gallery mode" -- it should just happen. If the sender uploads 40 photos and 2 PDFs, the recipient sees a gallery with those 2 PDFs accessible from a "Other files" section.

### Gate 2: Does the recipient experience create desire to use the product?

This is the conversion gate. The recipient sees a beautiful gallery and thinks "I want to do this too."

**What makes this gate open:**
- The gallery looks significantly better than WhatsApp photo sharing
- The gallery loads fast (lazy thumbnails, progressive rendering)
- The gallery works perfectly on mobile (swipe, pinch-zoom, landscape mode)
- The SG/Send branding is visible but not intrusive
- There is a clear, low-friction "Create your own gallery" CTA at the bottom

**What makes this gate close:**
- The gallery takes too long to load (500MB zip download before anything appears)
- The mobile experience is mediocre (no touch gestures, poor layout)
- There is no CTA -- the recipient enjoys the gallery and forgets about SG/Send
- The friendly key is confusing ("what do I do with 'blue-mountain-77'?")
- The gallery looks like a tech demo, not a polished product

**The performance question is critical.** A gallery of 100 photos at 5MB each = 500MB download before the first thumbnail can appear. The brief proposes lazy thumbnail generation from the extracted zip, which is the right approach. But the initial download itself takes time. On a mobile connection at 20Mbps, 500MB takes approximately 3-4 minutes. The recipient will see a progress bar for 3-4 minutes before seeing any photos. Compare with WhatsApp, where compressed thumbnails appear instantly and full images load progressively.

**Recommendation:** The progress bar during download must show meaningful information: "Downloading 42 photos and 3 videos (187 MB remaining)..." followed by "Decrypting..." followed by thumbnails appearing one by one. The worst experience is a generic spinner. The best experience is watching the gallery "fill up" as images are processed.

### Gate 3: Does the "Create your own gallery" path convert?

Even if the recipient thinks "I want this," they must then:
1. Navigate to SG/Send (from the CTA on the gallery page)
2. Understand the pricing (GBP 5 for 500 files)
3. Complete Stripe checkout
4. Wait for token delivery
5. Upload their own photos

**Assessment:** Steps 2-5 are the same purchase funnel that the first 10 customers navigated. The critical difference is motivation: the first 10 customers were asked by a friend. Gallery recipients are self-motivated. Self-motivated users are more likely to complete the flow if it is frictionless, but more likely to abandon at any friction point because they have no social obligation.

**The Sherpa's bottom line on gallery as a growth lever:**

Gallery sharing has genuine viral potential, but only if:
- The gallery MVP is genuinely beautiful (not just functional)
- Mobile experience is native-quality (this is where most photo sharing happens)
- The recipient-to-user conversion path is frictionless
- Token delivery is automated (the post-payment dead zone kills self-motivated users)
- The "Create your own gallery" CTA exists and is prominent

**Priority recommendation:** Build the gallery MVP but do not expect organic viral growth until automated token delivery exists. Without instant access after payment, the viral loop has a fatal gap.

### Credit Consumption Impact

The brief correctly identifies that gallery sharing drives dramatically higher credit consumption: 50-100 files per upload vs. 1 file for a single PDF. At GBP 0.01/file, a 100-photo gallery costs GBP 1 of the GBP 5 pack. A user who shares 3 galleries has consumed 60% of their credits.

**This is both an opportunity and a risk:**
- **Opportunity:** Faster credit depletion means faster re-purchase, higher revenue per user
- **Risk:** A user who shares one gallery and finds 100 credits gone may feel the pricing is too aggressive. "I sent one album and used 20% of my credits?" The per-file pricing was intuitive for single file transfers. For galleries, it may feel punitive.

**Recommendation:** Consider whether gallery uploads should count as 1 credit (for the gallery) rather than N credits (for each photo). Or introduce a "gallery credit" that costs more than 1 credit but less than N. The pricing model needs to align with user expectations. A user who thinks "I'm sharing an album" does not expect to pay for each photo individually.

## Part 3: Vault UX Improvements -- User Perspective Ranking

The vault fixes brief lists 10 improvements. Ranked here by impact on user retention (most important first):

### Tier 1: Must-Fix for Retention (Users Will Leave Without These)

**1. Auth token only required for upload (Fix #1 in brief)**
**Retention impact: Critical**

This is the single highest-impact vault UX change. The current behaviour (token required to open vault) means that sharing a vault with someone requires sharing two secrets: the vault key AND an auth token. This doubles the friction of sharing and doubles the explanation the sharer must provide. For a read-only recipient (e.g., "here, browse my photos"), needing an auth token is baffling.

The proposed change (vault key for reading, auth token only for writing) aligns with how users think about access: "I'm giving you the key to look at my files" vs. "I'm giving you permission to add files."

**2. Folder upload bug (Fix #3 in brief)**
**Retention impact: Critical**

Files going to the wrong folder is a data integrity issue. If a user carefully organises files into subfolders and then finds them dumped in root, they lose trust in the product. This is worse than a missing feature -- it is a feature that works incorrectly. This was flagged as a known bug from v0.1.0. It must be fixed before any vault promotion to customers.

**3. Drag-and-drop (Fix #4 in brief)**
**Retention impact: High**

Every file management application supports drag-and-drop. Its absence is immediately noticeable and makes the vault feel incomplete. Users who have just experienced drag-and-drop on the send page will expect it in the vault. The inconsistency between send (drag-drop works) and vault (drag-drop does not work) signals that the vault is not finished.

### Tier 2: Important for Engagement (Users Will Use the Product More)

**4. Settings panel (Fix #2 in brief)**
**Retention impact: Medium-High**

The settings panel serves multiple needs: it gives users control (editing vault name), it enables sharing (copy vault key), and it provides transparency (raw JSON viewer). The vault key copy button is particularly important -- without it, users cannot easily share vault access with others.

**5. File metadata banner (Fix #6 in brief)**
**Retention impact: Medium**

When a user clicks a file, they want to see what they are looking at: name, size, type, location. Without this, files feel anonymous. The metadata banner makes each file feel "known" and manageable. The future key-value metadata support is the right architectural decision -- build the infrastructure now even if the editing UI comes later.

**6. Folder management -- create and rename (Fix #5 in brief)**
**Retention impact: Medium**

Users organise files into folders. If they cannot create folders or rename them, the vault feels rigid. This is table-stakes functionality for any file management system.

### Tier 3: Nice to Have (Improves Quality of Life)

**7. Preview polish -- markdown, text/JSON, HTML (Fix #7 in brief)**
**Retention impact: Medium-Low**

Previews for markdown, text, and JSON benefit technical users (developers, writers) but are not critical for the general audience. Image, PDF, video, and audio preview already work -- these cover the majority of common file types.

**8. Inline editing for text/markdown/JSON (Fix #8 in brief)**
**Retention impact: Low-Medium**

Editing files directly in the vault transforms it from "encrypted storage" into "encrypted workspace." This is a powerful differentiator but depends on users having text-based files in their vault. For photo-heavy vaults, this adds no value. For document-heavy vaults, it is genuinely useful.

### Tier 4: Exploration (Valuable but Can Wait)

**9. Local storage viewer (Fix #9 in brief)**
**Retention impact: Low**

This is a debugging tool. Useful for power users and developers. Not relevant for retention of typical users.

**10. Chrome sync exploration (Fix #10 in brief)**
**Retention impact: High (if it works) / Zero (if it does not)**

Cross-device vault key persistence would eliminate the single biggest friction point for vault power users: "I set up my vault on my laptop, now I'm on my phone and I don't have the key." But this requires either a Chrome extension or a mechanism that does not exist yet. The exploration is worthwhile but should not block the other 9 improvements.

## Part 4: Customer Communication Plan

### Principle: Personal, Frequent, Specific

The first 10 customers bought because of a personal relationship. Communications must maintain that personal quality. The moment messages feel automated or generic, the relationship shifts from "I'm supporting a friend's product" to "I'm a customer of a company." Keep it personal for as long as possible.

### Immediate Actions (Today -- 3 March)

#### Action 1: Thank-You Messages to Paid Customers

**Channel:** Same channel as original outreach (WhatsApp or LinkedIn DM)
**From:** The founder (not "the team" or "SG/Send")
**Timing:** Today, staggered across the day (not bulk-sent)

**Template (personalise for each customer):**

> Hey [name], just wanted to say a proper thank you for trying SG/Send yesterday. Really appreciate it.
>
> Quick update -- since you signed up, we've already added a few things:
> - Folder upload (drag a folder and it auto-zips and sends)
> - In-browser zip preview (see what's inside without downloading)
> - Video and audio playback right in the browser
>
> Try sending a video to someone -- it plays instantly when they open the link. Pretty smooth.
>
> And [name], if you hit anything that doesn't work or you wish worked differently, just tell me. We shipped the folder upload feature yesterday in 90 minutes after [customer name] asked for it.
>
> Your Amazon voucher is on its way separately.

**Why this works:**
- Opens with gratitude, not a feature list
- Shows the product is actively developing (3 new features in 24 hours)
- Gives a specific action ("try sending a video")
- Reinforces the 90-minute feature request story (social proof within the group)
- Mentions the voucher without making it the focus
- Ends with an invitation to provide feedback

**For customers who declined the voucher, modify the last line:**

> I know you said you don't need the voucher -- that actually means a lot. Let me know if there's anything else I can do.

## Part 5: Signals for Product-Market Fit

### The Signal Hierarchy

Not all signals are equal. Ranked from strongest to weakest:

| Rank | Signal | Current Status | Why It Matters |
|------|--------|---------------|----------------|
| 1 | **Organic return usage** -- a customer comes back without being prompted | Not observed | This is the gold standard. If nobody returns unprompted in 14 days, the product solves a real problem only occasionally, not habitually. |
| 2 | **Referral** -- a customer tells someone else to use SG/Send | Not observed | Word-of-mouth is the only scalable growth channel at this price point. If customers do not tell friends, the product is "nice" but not "remarkable." |
| 3 | **"This saved me time"** -- a customer describes a specific real task it helped with | Not observed | This shifts the product from "curiosity" to "tool." A customer who says "I used SG/Send to send the contracts to my solicitor" has found product-market fit for themselves. |
| 4 | **Repeat purchase** -- a customer buys another credit pack | Not observed | Direct financial validation. Someone spending GBP 5 once might be doing a favour. Spending GBP 10 is choosing the product. |
| 5 | **Feature request beyond folder upload** -- a customer asks for something specific | 1 observed (folder upload) | More feature requests = more engagement = more thinking about real use cases. |
| 6 | **"Can I use this for [specific scenario]?"** -- a customer imagines new use cases | Not observed | This means the product has captured their imagination. They are doing the product-market exploration for us. |
| 7 | **Customers asking about enterprise/team features** | Not observed | This signals willingness to spend significantly more and embed the product in workflows. |
| 8 | **People will pay when asked** | Confirmed (10/10) | Weakest signal because it was driven by social relationship, not product need. Still valuable as a baseline. |
| 9 | **Voucher decline** | Confirmed (several) | Interesting but ambiguous. Could mean "I value the product" or "I value the friendship and don't want to profit from it." |
| 10 | **Purchase flow works** | Confirmed | Infrastructure signal, not product-market signal. |

### What to Watch This Week (3-9 March)

| Signal | How to detect it | What to do if observed |
|--------|-----------------|----------------------|
| Any customer sends a file without being prompted | Check admin analytics/transfer logs | Send a personal note: "I noticed you sent a file -- how did it go?" |
| Any customer mentions SG/Send to someone else | They tell the founder, or an unknown user appears | Document the referral chain. Thank the referrer. |
| Any customer describes a real use case | In their feedback reply or WhatsApp message | Document verbatim. This is marketing gold. |
| No customer has used the product after 7 days | Check transfer logs for the 10 customer tokens | This is a red signal. Send the structured feedback ask earlier. |

### The 14-Day Test

**By 17 March, we should know:**

1. How many of the 10 customers used SG/Send more than once? (Target: 3+)
2. How many described a real use case? (Target: 1+)
3. How many referred someone? (Target: 1+)
4. How many EAP users re-engaged with the new code? (Target: 3+)
5. Did anyone ask "can I use this for [X]?" (Target: 1+)

**If all five targets are met:** There is early product-market fit signal. Proceed with the next outreach batch (10 more contacts).

**If targets 1-3 are all zero:** The product works but does not solve a recurring problem for this audience. Pivot the outreach to a different audience segment (e.g., photographers for gallery sharing, legal professionals for secure document transfer) before spending more on customer acquisition.

## Part 6: User Experience Risks for the First 10 Customers

### Risk 1: The Token Expiry Cliff (Severity: High)

**The problem:** Tokens have a usage limit (500 files). If a customer experiments heavily (especially with gallery-style uploads of 50-100 files), they could hit the limit within a few sessions. There is currently no notification when credits are running low or exhausted. The customer will simply see the access gate reject their token with no explanation.

**Impact:** A customer who was happily using the product and suddenly gets locked out will feel betrayed. "I paid GBP 5 and it stopped working" is the worst possible customer experience story.

**Mitigation:**
- Monitor token usage for all 10 customers (check weekly via admin panel)
- If any customer reaches 80% usage, proactively message them: "You've used 400 of your 500 credits -- that means you're actually using it! Here's a bonus 500 credits on us."
- Add a credit balance indicator to the upload page (PROPOSED -- does not exist, but should be prioritised)

### Risk 2: The Bank Fraud Flag Spreading (Severity: Medium-High)

**The problem:** One customer reported their bank flagged the transaction. The brief correctly identifies this as a new-merchant issue. But the risk is that other customers also received fraud alerts and either (a) did not mention it, or (b) are now wary of SG/Send.

**Impact:** A customer who saw a fraud alert and completed the transaction has a lingering doubt. If they then share a payment link with a friend ("you should try SG/Send"), and the friend's bank also flags it, the referral is poisoned.

**Mitigation:**
- Proactively message all 10 customers: "Quick note -- one user mentioned their bank flagged the purchase as unusual. This sometimes happens with new merchants. If your bank flagged it too, the payment is 100% legitimate. Let me know if you need any confirmation for your bank." This normalises the issue before customers worry in silence.
- Do NOT wait for customers to report this. Proactive transparency builds trust; reactive explanations feel like damage control.

### Risk 3: Recipient Experience Failures (Severity: Medium)

**The problem:** We have no data on what recipients experienced when customers shared files. If even one recipient could not open a file (wrong browser, stripped URL hash, corporate firewall blocking the domain), the customer's experience of the product is "it didn't work."

**Impact:** A customer whose recipient says "I couldn't open it" will not use SG/Send again. They will not report it as a bug -- they will simply stop using the product.

**Mitigation:**
- In the thank-you message, ask: "When you shared a file, did the other person manage to open it OK?"
- If any recipient had issues, investigate immediately and follow up with both the sender and recipient

### Risk 4: Silent Churn (Severity: High)

**The problem:** The most dangerous outcome is that customers politely say "nice product" and then never use it again. At 10 users, each one represents 10% of the customer base. Losing even 3 to silent churn means 30% attrition before any organic growth.

**Impact:** Silent churn produces no feedback, no learning, and no signal. The product team assumes everything is fine because no complaints were received.

**Mitigation:**
- The Week 2 feedback ask is specifically designed to detect silent churn ("I haven't used it yet" is a valid and useful answer)
- Track usage at the token level -- if a token has zero transfers after 7 days, the customer has churned regardless of what they say in messages
- For churned customers, ask a direct question: "I noticed you haven't used SG/Send since the first day. No pressure at all -- but I'm curious: is it that you haven't needed to send a file, or that something about the product put you off?"

### Risk 5: Feature Overload Before Core Stability (Severity: Medium)

**The problem:** The March 03 briefs propose: gallery sharing, 10 vault UX improvements, vault CLI with local sync, and file safety orchestration. These are four major workstreams. Meanwhile, the 10 existing customers need attention, communication, and follow-up.

**Impact:** If the team focuses on building new features and neglects customer communication, the first 10 customers will feel like a transaction, not a relationship. The founder's time is finite -- every hour spent on gallery MVP design is an hour not spent messaging customers.

**Mitigation:**
- Dedicate 30 minutes per day to customer communication (the Sherpa communication schedule)
- Do not announce features to customers until they are shipped and tested
- Do not promise timelines ("gallery viewer coming next week") -- only communicate what is live

### Risk 6: Vault Promotion Before Bug Fixes (Severity: Medium-High)

**The problem:** The vault has a known bug (files saved to wrong folder) and missing core functionality (no drag-and-drop, no settings panel). If customers are directed to the vault before these are fixed, their first impression of the vault will be negative.

**Impact:** First impressions of a new product feature are disproportionately persistent. A customer who encounters a buggy vault will categorise it as "not ready" and may not revisit even after fixes ship.

**Mitigation:**
- Do NOT mention the vault to customers until fixes #1, #3, and #4 from the vault brief are shipped
- When the vault is ready, introduce it as a new feature in the weekly update: "New: encrypted personal vault -- store and organise your files in the browser"
- Frame it as new, not as fixed. Customers should not know the vault existed in a buggy state.

## Part 8: The Vault CLI -- Honest Assessment

The vault CLI brief describes a powerful end state: a distributed encrypted file system with local sync, multi-device access, and backup integration. The Sherpa's assessment is honest about both the appeal and the risks.

### Who This Serves

| User type | Would they use CLI? | Would they pay for CLI? |
|-----------|--------------------|-----------------------|
| Developer | Yes -- CLI is their native environment | Probably -- if sync is reliable |
| Technical power user | Maybe -- if the commands are simple | Maybe -- but they might prefer the browser |
| Business user | No -- they do not use terminals | No |
| Consumer (photo sharer) | No | No |
| Enterprise IT admin | Yes -- for automation and scripting | Yes -- if it integrates with their tooling |

**The CLI serves developers and enterprise IT.** Neither of these groups is well-represented in the first 10 customers (who appear to be friends and contacts of the founder, likely a mix of technical and non-technical). Building the CLI now is building for a future audience, not the current one.

### The Interoperability Risk

The brief correctly identifies the critical requirement: "The Python encryption must produce ciphertext that the JavaScript browser can decrypt, and vice versa." This is a hard engineering problem. If the CLI encrypts a file and the browser cannot decrypt it (or vice versa), the entire vault becomes untrustworthy. Testing this interoperability requires:
- Automated cross-platform encryption/decryption tests
- Testing with various file types and sizes
- Testing across browser versions (Chrome, Firefox, Safari)
- Regression testing on every change to either the JS or Python encryption code

**Recommendation:** The interoperability proof-of-concept should be the first deliverable. Before writing any CLI commands, prove that Python and JavaScript can round-trip encrypt/decrypt a file identically. If this proof fails or is complex, the CLI scope must be reconsidered.
