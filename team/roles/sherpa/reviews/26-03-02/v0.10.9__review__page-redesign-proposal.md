# Sherpa Review: Page Redesign Proposal

**Version:** v0.10.9
**Date:** 2026-03-02
**Role:** Sherpa
**Document reviewed:** `team/humans/dinis_cruz/briefs/03/01/website__sgraph_ai/v0.10.4__sg-send__page-redesign-proposal.md`
**Companion documents:** `v0.10.4__sg-send-homepage-ux-brief.md`, `v0.10.4__sg-send__replace-launchlist.md`
**Review type:** Trail re-walk and friction analysis of proposed page redesign
**Severity:** Supportive with observations -- the proposal fixes the critical friction; this review identifies remaining gaps

---

## Executive Summary

The redesign proposal correctly identifies and eliminates the six friction points documented in the UX brief. The buy path drops from 4 clicks / 4 scrolls / 4 page loads to 1 click / 0 scrolls / 1 page load. The free path drops from 3 clicks / 1 scroll / 3 page loads / 1 brand break to 0 clicks / 1 scroll / 0 page loads / 0 brand breaks. These are transformative improvements for the Challenge campaign.

This review walks every proposed path step by step, scores each one, and documents remaining friction the proposal did not address -- particularly around the post-purchase gap, the "Already have a token?" path, edge cases from eliminated pages, and the inline form's failure modes.

## 1. Trail Re-walk: Buy Path (After Redesign)

### Proposed path

| Step | User action | What happens | Location |
|------|------------|-------------|----------|
| 0 | Arrives at sgraph.ai (from WhatsApp/email) | Homepage loads | `/en-gb/` |
| 1 | Sees hero with "Buy Credits -- GBP 5" as primary CTA (filled, left position) | Button is above the fold, immediately visible | `/en-gb/` hero |
| 2 | Clicks "Buy Credits -- GBP 5" | Navigates directly to Stripe checkout | `buy.stripe.com/...` |
| 3 | Completes Stripe checkout | Stripe redirects to `/payment/success/` | `/en-gb/payment/success/` |

**Friction score:**
- Clicks: 1 (hero CTA to Stripe)
- Scrolls: 0
- Page loads: 1 (Stripe checkout -- external but expected)
- Brand breaks: 0.5 (Stripe is a known, trusted payment processor -- partial brand break is acceptable)
- **Total: 1.5**

### Comparison to current

| Metric | Current | Proposed | Delta |
|--------|---------|----------|-------|
| Clicks | 3-4 | 1 | -3 |
| Scrolls | 3-4 | 0 | -4 |
| Page loads | 3-4 | 1 | -3 |
| Brand breaks | 0 | 0.5 | +0.5 |
| **Total** | **9-12** | **1.5** | **-7.5 to -10.5** |

### Remaining friction the proposal missed

**1. The Stripe URL is opaque.** The current Stripe link on `/pricing/cloud/` is `https://buy.stripe.com/fZu8wHdv23nN1Z85jn5os02`. The proposal says "Buy Credits" goes "DIRECTLY to Stripe." This is correct -- but the user sees a URL with no human-readable content. For a security-conscious audience, this matters. The proposal should specify that the Stripe checkout page includes SG/Send branding (logo, product description, amount) so the user knows they are in the right place. Stripe supports this -- it is a configuration issue, not a code issue.

**2. No confirmation of what they are buying before the click.** The hero CTA says "Buy Credits -- GBP 5" but does not explain what 500 credits means or that each credit = 1 file transfer. The proposal puts "Open source / GBP 0.01/file / 500 transfers" in the sub-hero text, which partially addresses this. However, on mobile, sub-hero text may be below the fold or too small to read before the user clicks. The friction is low but real: the user who clicks "Buy Credits" without reading the sub-text may feel uncertain on the Stripe page about what they are getting.

**Recommendation:** Add "(500 files)" to the button label itself: "Buy Credits -- GBP 5 (500 files)". This eliminates ambiguity without requiring the user to read surrounding text.

**3. The proposal does not address the pricing page buy path.** The pricing page proposal includes a "Buy Credits -- GBP 5" button on the Starter Pack card that goes "DIRECT TO STRIPE." This is good -- but the pricing page also has 8 deployment cards below. A user who arrives at `/pricing/` directly (from nav or bookmark) and scrolls to the SG/Send Cloud section finds a Multi-Tenant card with "[Buy]" that also goes to Stripe. There are now two buy buttons on the pricing page, one in the hero area and one in the card grid. The proposal should confirm they go to the same Stripe URL. Inconsistent destinations for buttons with the same label was one of the original UX brief's criticisms.

## 2. Trail Re-walk: Free Path (After Redesign)

### Proposed path

| Step | User action | What happens | Location |
|------|------------|-------------|----------|
| 0 | Arrives at sgraph.ai | Homepage loads | `/en-gb/` |
| 1 | Sees hero with "Try Free" as secondary CTA (outline, right position) | Button is above the fold | `/en-gb/` hero |
| 2 | Clicks "Try Free" | Page scrolls to inline signup form section below the hero | `/en-gb/#try-free` (same page) |
| 3 | Fills in Name and Email fields | Two form fields, standard input | Same page |
| 4 | Clicks "Join Early Access Programme" | Form submits to internal API endpoint | Same page |
| 5 | Sees inline confirmation: "You're in. We'll send your access token to you@email.com within 48 hours." | No redirect, no page load | Same page |

**Friction score:**
- Clicks: 1 (CTA) + 2 (form fields) + 1 (submit) = 4 interactions but only 1 navigation click
- Scrolls: 1 (smooth scroll to form)
- Page loads: 0
- Brand breaks: 0
- **Total: 1** (counting only navigation friction, not form completion)

### Comparison to current

| Metric | Current | Proposed | Delta |
|--------|---------|----------|-------|
| Clicks (navigation) | 3 | 0 (scroll only) | -3 |
| Scrolls | 1 | 1 | 0 |
| Page loads | 3 | 0 | -3 |
| Brand breaks | 1 (LaunchList) | 0 | -1 |
| **Total** | **8** | **1** | **-7** |

### Remaining friction the proposal missed

**4. What happens if JavaScript is disabled?** The inline form depends on JS for three things: (a) smooth scroll from the "Try Free" button, (b) form submission to the internal API endpoint, and (c) inline confirmation display. If JS is disabled:
- The "Try Free" button will either do nothing (if it is a JS click handler) or navigate to an anchor (if it is `<a href="#try-free">`). The proposal should specify it as an anchor link so it degrades gracefully.
- The form submission will fail silently unless the form has a `<form action="..." method="POST">` fallback. The proposal's LaunchList replacement brief specifies a `POST /api/early-access/signup` endpoint -- but if the form is submitted as a standard HTML POST, the user will navigate away from the page to the API response (a JSON blob). This is a broken experience.
- **Recommendation:** Use progressive enhancement. The form should have a `method="POST"` and `action` pointing to a server-rendered success page as fallback. JS enhances this with fetch() + inline confirmation. The server-rendered fallback page can be a simple page at `/early-access/success/` that shows the same confirmation message.

**5. Form validation error states are unspecified.** The proposal shows "Name" and "Email" fields but does not describe:
- What happens if the user submits with an empty name? (The LaunchList replacement brief says "Basic validation: name not empty, email looks valid.")
- What happens if the email is invalid?
- What happens if the API call fails (network error, 500, rate limiting)?
- What does the error state look like? Inline error text? Border colour change? A toast?
- **Recommendation:** Define three error states: (a) field-level validation errors displayed below each field in red, (b) a general submission error displayed above the submit button, and (c) a "try again" message with a retry button for network failures. All error states should stay inline -- never redirect or open a modal.

**6. Duplicate form submissions.** The proposal does not mention submit button disabling. A user on a slow connection might click "Join Early Access Programme" multiple times. Each click would fire a POST to the API, generating duplicate notification emails. The submit button should be disabled after the first click and show a loading indicator ("Joining...").

**7. The inline form creates a scroll-dependent CTA.** After the redesign, "Try Free" scrolls the user down to the form section. On a tall display (1440p or 4K monitor), the form might be immediately below the fold -- one smooth scroll away. On a phone (375px wide, 667px tall), the hero section may consume most of the viewport, and the form may be further away. The proposal should specify a minimum distance: the form section should begin no more than one viewport height below the hero. This is a CSS layout constraint, not a functionality issue.

**8. Success confirmation does not capture the user's attention strongly enough.** The proposed inline confirmation says "You're in." followed by "We'll send your access token to you@email.com within 48 hours." This is clear but understated. The form card simply swaps content. On a page with other sections above and below, the user might not be certain the submission succeeded -- especially if they were looking away from the screen. **Recommendation:** Add a brief visual transition (fade-out of form, fade-in of confirmation) or a subtle animation on the checkmark to draw the eye. The word "within 48 hours" is the most important takeaway -- it should be visually emphasised.

## 3. "Already Have a Token?" Path

### Proposed path

| Step | User action | What happens | Location |
|------|------------|-------------|----------|
| 0 | User sees sub-hero text: "Already have a token? Go to send.sgraph.ai" | Link is visible below the two CTAs | `/en-gb/` hero |
| 1 | Clicks the link | Opens `https://send.sgraph.ai` | `send.sgraph.ai` |

### Analysis

**9. The gap between "I have a token" and "I'm sending a file."** The link sends the user to `send.sgraph.ai`. This is the correct destination -- it is where they will enter their token and upload a file. However, the proposal does not describe what the user sees when they arrive at `send.sgraph.ai`. Does the app immediately prompt for a token? Is there a landing page? Is there a "paste your token here" input field? The Sherpa has not been asked to review the `send.sgraph.ai` app UX in this brief, but the trail crosses that boundary.

The key question: does the user know what to do when they arrive at `send.sgraph.ai`? If the first thing they see is a token input field with clear instructions ("Enter your access token to start sending encrypted files"), the trail is smooth. If they see a complex app interface with no clear entry point, the trail breaks at the domain boundary.

**Recommendation:** The text "Already have a token? Go to send.sgraph.ai" should link to `https://send.sgraph.ai` -- which it already does in the current homepage (line 91 of the current HTML). This link should open in a new tab (`target="_blank"`) so the user does not lose the sgraph.ai context. The current implementation does NOT have `target="_blank"` on this link. The proposal should specify adding it.

**10. The "Already have a token?" link is easy to miss.** It lives in the sub-hero text, styled as small secondary text (the current CSS shows `font-size: 0.9rem; color: var(--text-secondary);`). For returning users who already have a token, this is the only path from the homepage to the product. It should be more prominent -- perhaps as a third, tertiary link below the two CTAs, or in the nav bar itself. A returning user arriving at sgraph.ai should not have to read fine print to find the product.

**Recommendation:** Consider adding "Go to SG/Send" as a link in the nav bar (rightmost position, styled differently from the other nav items -- perhaps as a button or with an accent colour). This serves returning users who arrive at any page, not just the homepage.

## 4. Post-Purchase Path

### Proposed path (post-Stripe checkout)

| Step | User action | What happens | Location |
|------|------------|-------------|----------|
| 0 | Completes Stripe checkout | Stripe redirects to success page | `buy.stripe.com` |
| 1 | Lands on `/payment/success/` | Sees "Payment successful" with next steps | `/en-gb/payment/success/` |
| 2 | Reads next steps: (1) "We'll create your access token (typically within a few hours)" (2) "You'll receive an email from Sherpa with your token and instructions" (3) "Visit send.sgraph.ai and start sending encrypted files" | User now knows to wait for an email | `/en-gb/payment/success/` |
| 3 | Clicks "Go to SG/Send" or "Back to Pricing" | Navigates away | `send.sgraph.ai` or `/pricing/` |

### Analysis

**11. The post-purchase path is the biggest remaining gap.** The proposal focuses entirely on getting users TO Stripe but does not address what happens AFTER Stripe. The current `/payment/success/` page (which I verified exists) tells the user:

> "We'll create your access token (typically within a few hours)"
> "You'll receive an email from Sherpa with your token and instructions"
> "Visit send.sgraph.ai and start sending encrypted files"

This creates a dead period between "I paid GBP 5" and "I have a token." The user has money invested but cannot use the product. The proposal does not discuss this gap at all.

**Friction points in the post-purchase path:**

- **Waiting time ambiguity.** "Typically within a few hours" is vague. Is it 1 hour? 6 hours? 24 hours? The user has just paid and wants to start using the product. Every hour of waiting is friction. For the Challenge campaign specifically, the participant was told to "buy credits and send a file" -- they cannot complete the challenge until they receive their token. This could be hours or days after the purchase.
- **No immediate gratification.** The user paid GBP 5 and received... a promise. There is no way to start using the product immediately after payment. This is the most significant friction point in the entire post-redesign flow. The buy path is now beautifully efficient (1 click to Stripe) -- but the post-buy path immediately stalls.
- **The "Go to SG/Send" button on the success page is misleading.** It takes the user to `send.sgraph.ai`, but they do not have a token yet. When they arrive, they will not be able to do anything. This button creates a false expectation.

**Recommendation:** This is not a website redesign issue -- it is a product issue. But the Sherpa must flag it because it is the next friction point the Challenge participant will hit after the website friction is eliminated. The ideal fix is instant token delivery (token generated during Stripe webhook processing, displayed on the success page or emailed immediately). The minimum viable fix is: (a) change the "Go to SG/Send" button text to "Go to SG/Send (once you receive your token)" or remove it entirely, (b) make the success page set a clear expectation ("You'll receive your token via email within 2 hours"), and (c) add a "Check your token status" link or mechanism.

**12. The payment cancel path is well-handled but could link to the free option.** The current `/payment/cancel/` page says "No worries -- no charges have been made. You can try again anytime, or join our free Early Access Programme instead." This is good messaging. However, the CTA buttons are "Back to Pricing" and "Ask the Sherpa" -- there is no direct link to the free signup form. Post-redesign, this page should include a "Try Free Instead" link that goes to the homepage inline form (or a direct scroll anchor).

## 6. Edge Cases and Missing Paths

### Users who arrive at `/pricing/cloud/` directly

**15. The proposal says "ELIMINATE" `/pricing/cloud/` but does not specify what happens to direct visitors.** Users might arrive at `/pricing/cloud/` from:
- Browser bookmarks
- Google search results (the page has good SEO metadata and is indexed)
- Cached links in emails, Slack messages, or documents
- The current hreflang alternates across 16 locales (all indexed)

If the page is deleted, these users will see a 404. The proposal mentions: "If we want to preserve a 'learn more about the cloud service' page for SEO or deep-linking, it can exist -- but the buy button on it must be above the fold."

**Recommendation:** Do NOT delete `/pricing/cloud/` immediately. Instead:
1. Move the "Buy Credits -- GBP 5" CTA to the top of the page (above the fold).
2. Add a `<link rel="canonical" href="/en-gb/pricing/">` to consolidate SEO.
3. After 3-6 months of traffic data showing most visitors come through the main pricing page, then consider deletion with a 301 redirect to `/pricing/#cloud`.
4. At minimum, set up a 301 redirect from `/pricing/cloud/` to `/pricing/` on elimination day. Never return a 404 for an indexed URL.

### Users who arrive at `/pricing/free/` directly

**16. Same issue as `/pricing/cloud/`.** The `/pricing/free/` page is indexed across 16 locales. The proposal says redirect to `/pricing/#free`. This is correct -- a 301 redirect to an anchor is the right approach. But the anchor `#free` must exist on the pricing page. The proposal's pricing page layout does not explicitly mention anchor IDs. Each section needs an `id` attribute: `id="free"` on the free tier section, `id="cloud"` on the cloud tier section.

### Users who arrive at `/early-access/` directly

**17. The proposal says "Delete this page. No redirect needed -- there are no external links to it."** This is incorrect. The current `/early-access/` page:
- Has hreflang alternates across 16 locales (all indexed)
- Has a canonical URL
- Is linked from the current `/pricing/free/` page CTA: `<a href="../../early-access/">`
- Is linked from the current homepage "Get Started" section: `<a href="./pricing/free/">`... which then links to `/early-access/`
- May be bookmarked by users who visited previously

Any indexed page should get a 301 redirect, not a silent deletion. Redirect `/early-access/` to `/en-gb/#try-free` (the homepage inline form anchor).

### Error states in the inline form

**18. The proposal does not define error states for the inline form.** The following scenarios need handling:
- Empty name, empty email: field-level validation error
- Invalid email format: field-level validation error
- Network failure (user is offline): "Unable to submit. Check your connection and try again."
- Server error (500): "Something went wrong. Please try again or email sherpa@sgraph.ai."
- Rate limiting: "Too many requests. Please wait a moment and try again."
- Duplicate submission (same email): The LaunchList replacement brief does not mention deduplication. What happens if someone submits twice? Do we send two notification emails? Should the form say "You've already signed up -- check your email for your token"?

**19. The form has no CSRF protection mentioned.** The LaunchList replacement brief describes a simple POST endpoint. For an inline form on a public page, CSRF protection should be considered. A honeypot field (hidden field that bots fill but humans do not) is the minimum viable anti-spam measure. The proposal does not mention this.

## 8. Observations Not Covered Above

**20. The pricing page redesign preserves all 8 deployment cards but adds two primary CTAs at the top.** This is the correct trade-off. The cards communicate the open-source philosophy to browsers and evaluators. The CTAs serve intentful buyers and free-signup users. Neither blocks the other. Good design.

**21. The "Recommended" label on the Starter Pack (pricing page) is correctly placed.** On the current homepage, "Recommended" is on the Early Access card -- the free option. The proposal moves it to the Starter Pack -- the paid option. This aligns the visual hierarchy with the commercial priority. However, the homepage proposal does NOT show a "Recommended" badge on either CTA button. It should -- the "Buy Credits" button should have some visual marker (badge, colour, or weight) that communicates it is the recommended action.

**22. The "Deploy Anywhere" section on the proposed homepage is a good bridge.** It replaces the full pricing grid teaser with a single sentence and a link. This avoids the current problem of showing 8 cards on the homepage. The link "See all deployment & pricing options" goes to `/pricing/` -- this is a clean hand-off.

**23. "Meet the Team" and "View on GitHub" buttons in the "Built Entirely by AI Agents" section are good.** They serve the Technical Evaluator persona without getting in the way of the Intentful Buyer.

**24. The proposal does not mention mobile layout.** The current homepage hero has two CTA buttons side by side. On a 375px-wide mobile screen, these may stack vertically. The proposal should confirm that "Buy Credits" appears ABOVE "Try Free" when stacked, not below -- to preserve the visual hierarchy on the device where most Challenge campaign traffic will arrive (WhatsApp links open in mobile browsers).

## 10. Overall Assessment

The redesign proposal is **structurally sound and should proceed**. It eliminates the six documented friction points from the UX brief. The buy path achieves theoretical minimum friction for a Stripe-based purchase. The free path eliminates the LaunchList brand break and three unnecessary page loads.

The remaining friction is in two categories:

1. **Detail gaps in the proposal itself** (error states, fallbacks, mobile layout, redirect handling) -- these are solvable in implementation and should not block the proposal.
2. **The post-purchase gap** -- this is not a website problem but a product problem. The user pays GBP 5 and then waits hours for a token. No website redesign can fix this. It is the next friction point the Challenge campaign will expose, and it should be on the Explorer's radar as a follow-up.

**The Sherpa supports this proposal proceeding to implementation with the P0 and P1 recommendations incorporated.**
