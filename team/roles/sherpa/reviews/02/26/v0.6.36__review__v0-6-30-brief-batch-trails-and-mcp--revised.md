## 4. Trail-Based Threat Modelling

As assigned in the per-file security review brief, this section maps the attack surface for the three priority trails.

> **Note:** Trail A threat modelling is grounded in **real, working code**. Trails B and C reference real endpoints but also reference proposed features (particularly one-time secret key delivery). Threats against proposed features are marked accordingly.

### Trail A: Send a File (EXISTS -- the only fully working trail)

```
Trail: "Send a file to a recipient"

  Steps:
  1. User opens send.sgraph.ai           -> sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.8/index.html
  2. User drops or selects a file         -> Client-side JS (encryption/keys)
  3. File encrypted (AES-256-GCM)         -> Web Crypto API (client-side)
  4. Encrypted blob uploaded              -> Routes__Transfers.create() / upload() / complete()
  5. Upload to storage (Memory-FS/S3)     -> Transfer__Service.upload_payload()
  6. Sender receives download link        -> Client constructs URL with key in fragment
  7. Sender shares link (out-of-band)     -> External channel (email, WhatsApp, etc.)
  8. Recipient clicks link                -> download.html
  9. Encrypted blob downloaded            -> Routes__Transfers.download__transfer_id()
  10. Recipient decrypts in browser       -> Web Crypto API (client-side)
```

**Status: ALL STEPS EXIST AND WORK.** This is verified by 24 transfer route tests, 12 service tests, and the MCP pipeline validation.

| Step | Attack Surface | Threat | Severity | Current Mitigation |
|------|---------------|--------|----------|-------------------|
| 1 | Landing page served from CDN/Lambda | CDN compromise, script injection | Medium | Subresource integrity (SRI) not yet implemented |
| 2 | File selection in browser | Local malware reads selected file | Low | Out of scope (user device compromise) |
| 3 | Client-side encryption | Weak key generation, side-channel attacks | Low | `crypto.getRandomValues()` + AES-256-GCM = strong |
| 4 | `/transfers/create` endpoint | Unauthenticated transfer creation, resource exhaustion | Medium | Bearer token required; rate limiting not yet confirmed |
| 4 | `/transfers/upload/{id}` endpoint | Oversized file upload (storage exhaustion / cost attack) | Medium | File size limit client-side only; **server-side validation needed** |
| 5 | Storage backend (S3/Memory-FS) | Bucket misconfiguration, direct S3 access | Medium | Storage abstraction via Memory-FS; S3 bucket policy review needed |
| 6 | Download URL with key in fragment | URL logged in browser history, shared unsafely | Medium | Fragment not sent to server; user education on link handling |
| 7 | Out-of-band link sharing | Link intercepted in transit (email, Slack, etc.) | Medium | One-time view would mitigate (PROPOSED -- not built); PKI mode would eliminate (PROPOSED -- not built) |
| 8 | Download page | Script injection on download page | Medium | Static page, no dynamic content from server |
| 9 | `/transfers/download/{id}` endpoint | Transfer ID brute force | Low | 12-char random ID = sufficient entropy (~71 bits) |
| 9 | Download endpoint | Unlimited download rate | Low | Rate limiting not yet confirmed |
| 10 | Client-side decryption | n/a (correct by construction if step 3 is correct) | Low | AES-256-GCM authenticated encryption |

**Priority findings for this trail:**
1. **Server-side file size validation** -- currently client-only. An attacker can bypass the client and upload arbitrarily large files.
2. **Rate limiting on create/upload/download** -- status unconfirmed across all transfer endpoints.
3. **SRI (Subresource Integrity)** -- JS files served from CDN/Lambda are not yet protected against injection.

### Trail B: Access a Data Room (PARTIAL -- endpoints exist, some flows are proposed)

```
Trail: "Access a symmetric key data room"

  Steps:                                                      Status:
  1. User receives access key              -> Out-of-band      EXISTS (manual sharing)
     (via one-time secret)                 -> Trail 1           PROPOSED (not built)
  2. User navigates to join.html           -> join.html         EXISTS
  3. User enters invite code               -> Client-side form  EXISTS
  4. Client validates invite               -> /join/validate    EXISTS
  5. Client accepts invite, gets session   -> /join/accept      EXISTS
  6. Client fetches room contents          -> Vault/Room API    EXISTS
  7. Client decrypts room contents         -> Web Crypto API    EXISTS
  8. User browses/uploads/edits per role   -> room.html         EXISTS
```

| Step | Attack Surface | Threat | Severity | Current Mitigation |
|------|---------------|--------|----------|-------------------|
| 1 | Key delivery channel | Key intercepted in transit | Medium | **Today:** Manual out-of-band sharing. **PROPOSED:** One-time secret link (not built) |
| 2 | Join page | Phishing: fake join page captures keys | Medium | HTTPS + domain verification; user education |
| 3 | Key entry form | Keylogger on user device | Low | Out of scope (user device compromise) |
| 3 | Key entry | Brute force of invite codes | **High** | Invite codes have limited entropy; **rate limiting on invite acceptance is critical** |
| 4-5 | Invite validation/acceptance | Unauthenticated session creation | Medium | Invite must be valid and not expired; usage limits enforced |
| 6 | Room content endpoints | Enumeration of room IDs | Medium | Room IDs should be opaque and high-entropy |
| 7 | Client-side decryption | n/a if key derivation is correct | Low | AES-256-GCM |
| 8 | Role-based operations | Permission bypass (viewer performs editor actions) | Medium | `Service__Vault__ACL.get_permission()` enforces; **needs AppSec review** |

**Priority findings for this trail:**
1. **Invite code brute force** -- invite codes may have insufficient entropy for an internet-facing service. Rate limiting on invite acceptance is essential. Consider lockout after N failed attempts per room.
2. **Session creation authentication** -- the invite endpoint must reject invalid codes before issuing tokens. Verify this is enforced server-side, not just client-side.
3. **Room ID enumeration** -- if room IDs are guessable, an attacker can attempt to brute-force invite codes against known rooms. Room IDs must be high-entropy.

  2. Admin creates/manages rooms            -> Rooms API        EXISTS

| Step | Attack Surface | Threat | Severity | Current Mitigation |
|------|---------------|--------|----------|-------------------|
| 1 | Admin authentication | Credential theft, session hijack | **High** | Separate Lambda with its own auth; token-based access |
| 1 | Admin endpoint exposure | Admin Lambda accessible from public internet | **High** | Lambda URL function; IP restriction recommended but not confirmed |
| 2 | Room creation | Uncontrolled room creation (resource exhaustion) | Medium | Admin auth required; quota not yet implemented |
| 3 | User/key creation | Key generation weakness | Medium | Need to verify crypto source for key generation |
| 4 | Key sharing | Key intercepted during sharing | Medium | **Today:** Manual out-of-band. **PROPOSED:** One-time secret (not built) |

**Priority findings for this trail:**
1. **Admin Lambda exposure** -- the admin Lambda must not be publicly accessible without strong authentication. IP restriction or VPN-gating recommended for production.
2. **Admin session security** -- a compromised admin session gives access to all room data and member management. MFA, session expiry, and audit logging are critical.
3. **Audit immutability** -- the audit service uses an append-only, hash-chained log (verified in reality document: "Immutable audit trail -- append-only, hash-chained events"). This is correctly implemented.

### Gap Assessment

The current MCP pipeline is optimised for one workflow: **encrypt a file, upload it, email the link**. This workflow EXISTS and is VALIDATED. The proposed trails expand the required MCP surface in two directions:

1. **Ephemeral data operations** (one-time secrets -- PROPOSED) -- the transfer pipeline could be adapted since secrets are just small encrypted blobs with expiry metadata. The underlying `transfers_create` / `transfers_upload` / `transfers_complete` tools exist. The missing piece is auto-delete-on-view and secret-specific metadata.

2. **Data room management via MCP** (PROPOSED) -- this requires entirely new MCP tool exposure. Room creation, user management, and room-scoped file operations exist as HTTP endpoints but are NOT exposed as MCP tools. The Admin Lambda MCP currently exposes tokens, keys, vault, and users -- but not rooms or invites.

**Recommended priority order for MCP extension (all PROPOSED):**
1. Expose existing rooms/invites endpoints via Admin Lambda MCP (lowest effort -- endpoints exist, just need MCP tagging)
2. `secrets_create` / `secrets_status` (enables Trail 1 via MCP, unblocks Trail 2 key sharing)
3. Room-scoped file upload (enables Trail 2 file management via MCP)
