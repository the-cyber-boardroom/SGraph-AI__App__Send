## Sherpa Lens: What This Document Is For

The Sherpa owns the user's experience of SGraph Send from first contact through to confident daily use. This proposal surfaces the work streams that matter most from that lens. It answers: **what does a user need to successfully use what has been built, and what guidance infrastructure needs to exist before the first paying customers arrive?**

This is not a feature wishlist. It is an assessment of the gap between what the product can do and what a user can actually accomplish on their own.

## Proposed Work Streams

### WS-P01: Getting Started Guide (First Contact to First File)

**Classification:** P* (Product/Platform)
**ID:** WS-P01
**Title:** Getting Started -- Zero to First Encrypted File in Under 5 Minutes
**Priority:** CRITICAL

**Description:**
Create a user-facing Getting Started guide that walks a new user from the first moment they land on `send.sgraph.ai` through to a completed, verified encrypted file transfer. The guide must work for the non-technical persona (a person who received a Starter token as part of the GBP 5 purchase or the GBP 500 voucher experiment).

The guide consists of:
1. A web page at `sgraph.ai/docs/getting-started/` (or equivalent) with an illustrated, step-by-step flow
2. An inline contextual version embedded on `send.sgraph.ai` (a collapsible "How does this work?" panel visible before the access gate)
3. A short-form version suitable for inclusion in the token delivery email (4-6 steps, plain text)

The guide must address, in plain language:
- What SG/Send does and why to trust it (resolves Friction F001 and F007)
- What the access token is and where it came from
- How to send a file: select, encrypt, copy link, share
- How the recipient decrypts: click link, one button, file appears
- What to do if the hash fragment is stripped from the link (the most common failure mode)
- How to check how many credits remain

**Rationale:**
The Librarian's reality document confirms this page does not exist. The GBP 500 voucher experiment (100 recipients) is the first time SG/Send will reach users who have no relationship with the team. Without a Getting Started guide, all 100 will arrive at a page that does not explain itself (Friction F001), behind an access gate that does not explain what the token is (Friction F012). The experiment will measure user drop-off, but drop-off caused by missing documentation is a waste of the GBP 500 budget. This must exist before the experiment launches.

**Dependencies:**
- Stripe must be configured (Brief 5 action items) before the guide can reference the payment flow
- The token delivery email must be drafted (Ambassador, Brief 5) before the short-form guide can be integrated
- Journalist to write the copy; Sherpa to validate against real users

**Effort:** Medium (one web page + two content variants + email snippet)
**Priority:** CRITICAL -- must be in place before GBP 500 experiment launch

### WS-P02: Post-Payment Onboarding Flow (GBP 5 to First File)

**Classification:** P* (Product/Platform)
**ID:** WS-P02
**Title:** Post-Payment Onboarding -- From Payment Confirmation to First Encrypted File
**Priority:** HIGH

**Description:**
Map and implement the complete user journey from Stripe payment confirmation through to the user's first successful encrypted file transfer. This is currently a broken sequence: the payment success page exists (`sgraph_ai__website/payment/success/index.html`) but delivers no token, no instructions, and no next step.

The onboarding flow consists of:

**Phase 1 (manual, immediate):** The payment success page tells the user: "Your token will be emailed to you within X minutes. Check your inbox at [email address they entered at checkout]." A human (or the sg-workmail service when it exists) sends the token + short Getting Started guide within the promised window.

**Phase 2 (automated, after sg-workmail is live):** Stripe webhook fires -> admin Lambda creates token -> sg-workmail sends token + guide in one automated email. User is on `send.sgraph.ai` within 60 seconds of payment confirmation.

The Sherpa's specific deliverable is the content of each touchpoint:
- Payment success page: what does it say? What does the user do while they wait?
- Token delivery email: what is the subject, the opening line, the 4-step guide?
- What does the user see if they return to `send.sgraph.ai` before their token arrives?
- What error message does the user see if they enter an expired or invalid token?

**Rationale:**
The GBP 5 payment is the highest-commitment action a user takes. The moment after payment is when the user is most engaged and most likely to complete their first use. A gap of hours between payment and first access kills that engagement. The current state -- payment success page with no token, no timeline, no next step -- is the highest-friction moment in the entire product. Resolving it directly increases the probability that paying users become active users.

**Dependencies:**
- Stripe product and Checkout configuration (Brief 5, Dev action P0)
- sg-workmail service (Brief 3) for Phase 2 automation
- Getting Started guide (WS-P01) for the content of the token delivery email

**Effort:** Low (Phase 1 content) to Medium (Phase 2 automation integration)
**Priority:** HIGH -- must be designed before Stripe is wired; Phase 1 can be manual

### WS-P03: Early Access Onboarding Sequence (Post-Signup)

**Classification:** P* (Product/Platform)
**ID:** WS-P03
**Title:** Early Access Onboarding -- What Happens After Someone Signs Up at sgraph.ai/early-access
**Priority:** HIGH

**Description:**
The early-access page exists and collects email addresses. There is no defined sequence of what happens after signup. This work stream defines and implements the EAP onboarding sequence:

**Step 1 -- Confirmation (immediate):** A confirmation email is sent to the signup address. Content: "You are on the list. Here is what SG/Send does. Here is what you will receive. Here is when you might expect access."

**Step 2 -- Welcome (on token issuance):** When the team decides to admit a batch of EAP users, an email is sent with their token + Getting Started guide. This must not feel generic -- it should feel personal and specific.

**Step 3 -- Feedback request (after first use):** 24-48 hours after the token is issued, a short email asks: "Did it work? What was confusing? What would you do differently?" Two questions maximum. A link to a feedback form or simply "reply to this email."

**Step 4 -- Follow-up (if no use within 7 days):** A gentle nudge: "Your token is still valid. Here is a 30-second walkthrough of what to try." Includes the short-form Getting Started guide.

The Sherpa writes all copy for all four touchpoints and specifies the timing and trigger conditions. The Ambassador and sg-workmail execute delivery.

**Rationale:**
The EAP is the current channel for user acquisition. Without a post-signup sequence, sign-ups are warm leads that go cold. The sequence is also the Sherpa's first structured feedback channel -- real users, real friction, real data to feed back into the product.

**Dependencies:**
- sg-workmail service (Brief 3) for automated email delivery
- Getting Started guide (WS-P01) for the welcome email content
- Ambassador coordinates the outreach timing and batch decisions

**Effort:** Low (content drafting) to Medium (sequence automation with sg-workmail)
**Priority:** HIGH -- content can be drafted before sg-workmail exists; execution waits on sg-workmail

### WS-P04: Clean Download URL (Friction F006)

**Classification:** P* (Product/Platform)
**ID:** WS-P04
**Title:** Clean Download URL -- Replace Versioned Path with `/d/{id}` Redirect
**Priority:** HIGH

**Description:**
The current download URL format is:
`send.sgraph.ai/send/v0/v0.1/v0.1.8/download.html#{transferId}/{key}`

This URL is 52+ characters before the hash, exposes the internal IFD versioning structure, contains a `.html` extension, and in several messaging apps (WhatsApp, Telegram, SMS) the hash fragment is stripped or the URL is truncated -- causing the recipient to arrive at the download page without the decryption key.

The target URL format is: `send.sgraph.ai/d/{transferId}#{key}`

Implementation requires:
1. A CloudFront Function (or Lambda URL rewrite) that maps `/d/{id}` to the latest versioned download page
2. The upload component to construct the clean URL rather than the versioned one
3. The download component to extract the transfer ID from the clean URL format
4. Trail validation: after deployment, confirm that the hash fragment is preserved through the rewrite

**Rationale:**
Friction F006 was logged on 13 Feb 2026. It has not been resolved. Every download link currently shared with a recipient exposes the internal version path. Clean URLs reduce the probability of hash stripping by messaging apps, increase user trust (short URL looks legitimate), and decouple the user-visible URL from the internal IFD versioning.

This also resolves a related problem: when the Sherpa sends a recipient a link, the link length and content signals "this is an internal tool" rather than "this is a polished product." For the GBP 500 experiment, that perception matters.

**Dependencies:**
- Architect to confirm clean URL routing approach (CloudFront Function vs Lambda rewrite)
- DevOps to implement the rewrite rule
- Dev to update the URL construction in the upload component

**Effort:** Medium (CloudFront Function + component updates + trail validation)
**Priority:** HIGH -- must be resolved before GBP 500 experiment to avoid hash stripping failures

### WS-P05: MCP Onboarding Guide (Connecting Claude to SG/Send)

**Classification:** P* (Product/Platform)
**ID:** WS-P05
**Title:** MCP Onboarding -- How to Connect Claude to SG/Send in Under 10 Minutes
**Priority:** HIGH

**Description:**
The MCP endpoints exist and are validated in production (Milestone M-007: Claude generated a PDF, encrypted it, uploaded it via MCP, human decrypted in browser). But there is no user-facing documentation on how to connect Claude to SG/Send via MCP. A technical user who wants to use Claude as an encrypted file transfer agent cannot do so without reverse-engineering the setup.

The MCP onboarding guide must cover:
1. Prerequisites: what is MCP, what clients support it (Claude.ai, Claude Desktop, Claude Code)
2. The two endpoints: User Lambda MCP (`/mcp`) and Admin Lambda MCP (`/mcp`) -- when to use each
3. Getting an access token: either EAP or Starter purchase
4. The minimal Claude.ai configuration to point Claude at the User Lambda MCP
5. Verification: "tell Claude to list your available transfer tools and confirm you see these 10 tools"
6. First MCP-assisted transfer: "ask Claude to encrypt a test message and give you a download link"
7. The 7-step pipeline (from SKILL.md) in plain English -- for users who want to use the full pipeline

The guide must include a "What can Claude do via MCP?" summary (create transfer, upload file, download base64, validate token) and be honest about what Claude cannot do yet (create data rooms, create one-time secrets -- all PROPOSED, no code).

**Rationale:**
MCP is the highest-differentiation capability of SG/Send. No other file-sharing service lets Claude autonomously encrypt, upload, and share files. But this capability is invisible to users who do not know it exists, and inaccessible to technical users who do not know the setup steps. The guide converts a validated capability into a discoverable user journey. This is particularly important for the tools.sgraph.ai narrative and for the sgraph.ai/product page where MCP is highlighted.

**Dependencies:**
- MCP endpoints exist (verified in reality document)
- User must have an access token -- connects to WS-P02 and WS-P03
- Content validation: Sherpa must test the guide end-to-end with a fresh Claude.ai account

**Effort:** Medium (technical writing + validation across multiple Claude clients)
**Priority:** HIGH -- should land before or alongside MCP being highlighted in marketing

### WS-P06: Data Room Invite UX Guide (Onboarding for Room Members)

**Classification:** P* (Product/Platform)
**ID:** WS-P06
**Title:** Data Room Invite UX -- What to Tell a New Room Member Before They Click the Link
**Priority:** MEDIUM

**Description:**
The data room join flow works (`join.html`, `room.html`, invite validation, session management all exist and are tested). But a new room member receives an invite code via out-of-band channel (email, WhatsApp) and lands on `join.html` with no context. They do not know:
- What they are joining
- What the invite code is for
- What permissions they will have
- What to do if the code does not work
- What the symmetric room key means for their security model

The Sherpa's deliverable is:
1. A "Data Room Member Onboarding" template -- copy that a room admin can send alongside the invite code to explain what the recipient is joining and how to use the join page
2. Inline help text on `join.html` -- a collapsible "What is this?" section that explains invite codes and the room concept in plain language
3. An error recovery guide: what to tell a member who enters an invalid code (code expired, code already used, wrong code)

This is specifically not the technical invite generation flow (that is for the admin). This is the member-facing experience: what they receive, what they do, what to expect.

**Rationale:**
Trail 3 (Join a Data Room) is partially implemented: the UI and backend work, but the invite key exchange is manual and undocumented. The first data rooms will be used by people who trust the admin who invited them -- but that trust can be broken by a confusing join flow. The inline help and the onboarding template prevent that first impression from being a support request.

**Dependencies:**
- Data room join flow exists (verified)
- Requires access to the join page template to add inline help text (Dev)
- Journalist to write the copy; Sherpa to validate

**Effort:** Low (content templates + inline help text)
**Priority:** MEDIUM -- needed when first data rooms go live with external members

### WS-P07: Self-Hosted Deployment User Guide

**Classification:** P* (Product/Platform)
**ID:** WS-P07
**Title:** Self-Hosted Deployment Guide -- From GitHub Clone to First Encrypted File
**Priority:** MEDIUM

**Description:**
The pricing page lists "Self-Hosted" as a free tier option. The architecture page describes the deployment targets. But there is no user-facing guide for someone who wants to run SG/Send themselves. The pricing matrix brief (Brief 6) specifies a `/pricing/self-hosted` detail page that would include "GitHub repo, quickstart."

The Sherpa's deliverable is the user journey for the self-hosted persona -- specifically a technical user who clones the repo and wants to understand:
1. What they are deploying (User Lambda + Admin Lambda, or just the server mode?)
2. The fastest path to a working local instance for evaluation
3. The fastest path to a production deployment (AWS Lambda, two functions, minimal IAM)
4. What the self-hosted user needs to configure (environment variables, storage mode)
5. Confirmation: "how do I know it is working?" (health endpoint, test upload)

This is a Sherpa-authored user journey map and checklist, not a technical specification. The Dev and DevOps roles produce the technical steps; the Sherpa sequences them into a user-understandable flow.

**Rationale:**
The self-hosted deployment model is a significant differentiator -- no other encrypted file sharing service of this quality is genuinely self-hostable. If the self-hosted experience is opaque, technical evaluators (who are the most influential word-of-mouth advocates) will give up. The Sherpa's role is to ensure they do not.

**Dependencies:**
- `/pricing/self-hosted` detail page (Brief 6, requires Ambassador copy + Dev implementation)
- Dev to confirm the minimum deployment steps for a local instance
- DevOps to confirm the minimum AWS IAM permissions for a Lambda deployment

**Effort:** Medium (user journey mapping + checklist + validation)
**Priority:** MEDIUM -- must exist before the self-hosted tier is promoted

### WS-P08: Privacy-First IP Tool as Sherpa Gateway (tools.sgraph.ai/my-connection)

**Classification:** P* (Product/Platform)
**ID:** WS-P08
**Title:** Privacy-First IP Tool -- Sherpa-Guided Journey from tools.sgraph.ai to SG/Send
**Priority:** MEDIUM

**Description:**
The privacy-first IP tool (`tools.sgraph.ai/my-connection`) is proposed in Brief 1 (batch 2) as a zero-tracking IP/location/ISP display page. From the Sherpa's perspective, this page is a first-contact gateway: a user arrives to check their IP and discovers that this service, unlike every other "what is my IP" site, tracks nothing, calls home to nobody, and does all processing client-side.

The Sherpa's role in this work stream is not to build the tool (that is Dev + DevOps) but to design the user journey that flows from the tool to SG/Send:

1. **The "What this page did NOT do" section:** write the copy for this section. It should be short, verifiable, and non-technical. "No cookie was set. No third-party script loaded. No analytics call was made. Check your browser's developer tools to verify." This is the Sherpa's voice -- guide, do not carry.

2. **The "Want more privacy?" CTA:** after showing the user what zero-tracking looks like, offer SG/Send as the natural next step. The CTA should not be pushy; it should feel like a logical continuation of the same philosophy. Draft copy: "If you want the same standard of privacy for your files, try SG/Send -- encrypted in your browser, server never sees your data."

3. **The journey from CTA to first file:** ensure that a user who clicks through from the IP tool lands on an onboarding path, not a raw upload page. If they are not yet a user, they should reach the early-access page. If they have a token, they should reach the upload page with the access gate already pre-explained.

**Rationale:**
The IP tool attracts users who are already privacy-aware and looking for solutions. These are the Sherpa's best candidates -- they understand why privacy matters and will resonate with SG/Send's zero-knowledge model. The conversion from IP tool visitor to SG/Send user is a short journey if the narrative bridge is well-designed. The Sherpa writes that bridge.

**Dependencies:**
- tools.sgraph.ai infrastructure (does not yet exist -- DNS, static site deployment)
- Privacy-first IP tool implementation (Brief 1, Dev lead)
- Early Access page exists (verified in reality document)

**Effort:** Low (copy and journey design; no code from Sherpa)
**Priority:** MEDIUM -- value multiplied once tools.sgraph.ai is live

### WS-P09: Multi-Language User Guidance (i18n for Onboarding Content)

**Classification:** P* (Product/Platform)
**ID:** WS-P09
**Title:** Multi-Language User Guidance -- Onboarding Content in All 17 Locales
**Priority:** MEDIUM

**Description:**
The sgraph.ai website has 17 locale JSON files and a pre-rendering pipeline. The User UI has 8 locale strings. But the Sherpa's user-facing content (Getting Started guide, error messages, inline help text, token delivery email, onboarding sequence) will be produced in English first. This work stream ensures that the guidance content enters the i18n system from the start, rather than being retroactively translated.

Specific deliverables:
1. All Getting Started guide content is written with i18n strings in mind (short segments, no idioms, no cultural references)
2. Error message rewrites (resolving Friction F009) are added to the i18n string tables, not hardcoded in HTML
3. Inline help text on `join.html` and the upload page is i18n-ready
4. The token delivery email template is written in a way that survives machine translation (simple sentence structure, no metaphors)

The Sherpa does not translate the content (that is the responsibility of the i18n pipeline and the locale JSON files). The Sherpa ensures that the content is designed to be translatable: no language-specific assumptions, no untranslatable phrasing, no cultural context that would confuse a machine-translated rendering.

**Rationale:**
The User UI already supports 8 languages. Users in pt-BR, es-ES, de-DE, fr-FR, and ja can already interact with the upload and download pages in their language. If the Getting Started guide and onboarding emails are English-only, non-English users who follow the UI in their language hit a wall when they look for help. This is a consistency problem that erodes trust precisely at the moment users most need support.

**Dependencies:**
- WS-P01 (Getting Started guide) must be drafted before this work stream can apply
- i18n pipeline (exists: `scripts/generate_i18n_pages.py`, 17 locale JSON files)
- Journalist to confirm i18n-friendly writing standards

**Effort:** Low (writing standards + coordination) to Medium (actually adding strings to locale files)
**Priority:** MEDIUM -- should be part of every content work stream from the start, not a separate phase

### WS-B01: Friendlies Programme Execution (GBP 500 Experiment Sherpa Support)

**Classification:** B* (Business)
**ID:** WS-B01
**Title:** Friendlies Programme -- Sherpa-Guided First Use for GBP 500 Experiment Participants
**Priority:** HIGH

**Description:**
The GBP 500 voucher experiment (Brief 5) sends 100 GBP 5 Amazon vouchers to contacts with "try SG/Send and give feedback" framing. The expected funnel is 100 reached -> 60 pay -> 40 send a file -> 20 give feedback -> 5-10 organic users.

From the Sherpa's perspective, the "40 send a file" step is the critical conversion. Every participant who drops off between "received token" and "sent first file" is a Sherpa failure. This work stream defines and executes the Sherpa's active support for each experiment participant:

1. **Pre-experiment:** Design a structured first-use checklist for experiment participants. Not a generic guide -- a specific "here is your first task" prompt: "Use your token to send yourself an encrypted test file. Here is exactly how. **Rationale:**
The GBP 500 experiment is the first structured test of SG/Send with real external users. Without Sherpa-active support, it is a marketing experiment. With Sherpa-active support, it is a user research exercise that produces actionable data. The investment is the same; the output is orders of magnitude more valuable.

**Dependencies:**

### WS-D01: Trail Telemetry for User Onboarding Flows

**Classification:** D* (Development/Delivery)
**ID:** WS-D01
**Title:** Trail Telemetry -- Instrument Key Onboarding Steps to Enable Sherpa Observation
**Priority:** MEDIUM

**Description:**
The Sherpa's quality gates require that "trail telemetry covers every user-facing flow stage." Today, the analytics dashboard (which exists) captures transfer counts and download counts. But the Sherpa cannot observe:

- What percentage of users who arrive at the upload page successfully complete a transfer
- How long users spend at the access gate (a proxy for confusion)
- Whether users attempt the "Share key separately" toggle (a proxy for advanced use or confusion)
- How many download attempts fail because the hash fragment was stripped
- Whether users return to `send.sgraph.ai` after receiving their token

The Sherpa's specification for required telemetry (what to instrument, not how):

| Event | Signal Value |
|-------|-------------|
| Access gate: token validation attempt (success/fail) | Measures funnel entry |
| Upload page: file mode vs text mode selection | Measures use case split |
| Upload: encrypt-start to complete duration | Measures performance friction |
| Upload: completion rate (file selected but never completed) | Measures abandonment |
| Download: key present on page load (yes/no from hash) | Measures hash stripping rate |
| Download: decrypt success/fail | Measures end-to-end success rate |
| Join page: invite code validation attempt (success/fail/expired) | Measures invite friction |

All events must be privacy-preserving: no personal data, no raw IPs (already hashed), no file content metadata. The DPO must confirm the lawful basis before any new telemetry is collected.

**Rationale:**
The Sherpa can only improve what can be observed. The existing analytics tell us "how many transfers" but not "where do users fail." Without this telemetry, the Friendlies programme feedback is the only signal -- anecdote at a sample size of 100. With this telemetry, every user interaction produces structured improvement data.

**Dependencies:**
- DPO review of new telemetry events before collection
- DevOps to implement the instrumentation
- Architect to confirm that event data can flow through the existing cache service analytics namespace

**Effort:** Medium (telemetry design + DPO review + implementation)
**Priority:** MEDIUM -- should precede the GBP 500 experiment to capture experiment data

### WS-D02: Error Message Rewrite (Friction F009 Resolution)

**Classification:** D* (Development/Delivery)
**ID:** WS-D02
**Title:** Error Message Rewrite -- Every User-Visible Error Follows "What Happened / Why / What To Do"
**Priority:** MEDIUM

**Description:**
Friction F009 was logged on 13 Feb 2026 and remains open. Every error message in the User UI follows a functional but non-guiding pattern ("Upload failed. Please try again."). The Sherpa's quality gate requires: "Error messages explain what happened AND what to do next -- never generic failures."

The deliverable is a complete rewrite of every user-visible error message across all User UI components:
- `send-upload.js`: upload failure, size limit exceeded, token validation failure
- `send-download.js`: transfer not found, wrong decryption key, download failure, expired link
- `room-join.js`: invalid invite code, expired invite, already used invite
- `room-view.js`: upload failure, download failure, permission denied
- `send-access-gate.js`: invalid token, expired token, exhausted token

Format for every rewrite: "[What happened]. [Why this might have happened]. [What to do next]."

Example:
- Before: "Transfer not found. The link may have expired."
- After: "We could not find this file. The link may have expired or been removed by the sender. Ask the sender to share a new link."

All rewrites must be:
- Added to the i18n string table (not hardcoded) -- resolves WS-P09 dependency
- Reviewed by the Journalist for tone
- Validated by the Sherpa against the personas (non-technical users must understand every message)

**Rationale:**
Error messages are the most critical user guidance in any system -- they appear exactly when a user is confused or stuck. A generic error message at this moment signals: "we don't care about you here." A guided error message signals: "we know what went wrong and we are helping you forward." For SG/Send, where the UX is already unfamiliar (encryption, hash fragments, invite codes), error messages must work overtime.

**Dependencies:**
- Journalist to review copy
- Dev to update the i18n string tables and component error handling
- i18n system (exists -- 8 locale strings in User UI)

**Effort:** Low (content rewrite) + Low (code to update i18n strings)
**Priority:** MEDIUM -- quick win; high impact per hour of effort

## What the Sherpa is NOT Proposing

To be explicit about scope:

- **Not proposing user trail implementation (Trails 1-5).** These require new code (one-time secrets, password generator, PKI-friendly links). The Sherpa will design the UX for each trail when the code exists. Not before.
- **Not proposing new features.** Every work stream above works with what EXISTS or what is already in motion (Stripe, sg-workmail).
- **Not proposing marketing content.** That is the Journalist and Ambassador. The Sherpa writes user guidance, not copy.
- **Not proposing security changes.** The IP-hashing, zero-knowledge model, and audit trail are correct. The Sherpa surfaces security UX (error messages, transparency panel) but does not redesign the security model.

## Sherpa's Priority Call: What to Do First

If the team can only start three of these work streams, the Sherpa recommends:

1. **WS-P01** (Getting Started Guide) -- the single document that makes everything else work. Every other onboarding touchpoint links to it or is informed by it. Write it first.

2. **WS-P04** (Clean Download URL) -- the highest-impact technical fix for the recipient experience. Hash stripping in messaging apps is the most common failure mode and it is not user-recoverable without help. Every day this is not fixed, links shared via WhatsApp or SMS fail silently for some recipients.

3. **WS-D02** (Error Message Rewrite) -- the highest return per unit of effort. Every error message in the product is currently a dead end. Turning them into guided recovery paths requires mostly content work, not code changes. It makes every other part of the product more survivable for new users.
