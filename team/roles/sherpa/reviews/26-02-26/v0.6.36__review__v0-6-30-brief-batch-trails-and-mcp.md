## 1. Executive Summary

This brief batch introduces five new user trails, a fully operational MCP email pipeline, and the infrastructure for the first waiting list outreach. The Sherpa's assessment:

**New trails.** The one-time secret, symmetric data room, password generator, and PKI viewer briefs collectively define five distinct user journeys that did not exist before this batch. Each trail has a clear entry point, a small number of steps, and a well-defined security boundary. The Sherpa has mapped all five trails below with step-by-step flows, component dependencies, and entry point URLs.

**MCP infrastructure.** The 7-step encrypted transfer pipeline (SKILL.md) and the n8n + WorkMail integration (debrief) are operational. Claude can autonomously create encrypted files, upload them, verify the upload, and email the download link -- all via MCP tool calls. This is the Sherpa's transport layer for outreach. The gap: the pipeline currently handles file transfer only; it does not yet support one-time secret creation or data room management via MCP.

**Outreach readiness.** We can send emails from `sgraph.ai` today via n8n + WorkMail (confirmed in the debrief). The "Getting Started" section draft is included below. The blocking item for production outreach is not infrastructure -- it is content approval and recipient list preparation.

**Trail-based threat modelling.** As assigned in the per-file security brief, the Sherpa has mapped the attack surface for the three priority trails: send a file, access a data room, and admin operations. The key finding: metadata leakage (human-chosen link names, user-visible filenames, room names) is the most pervasive risk across all trails. The one-time secret brief's decision to enforce opaque server-generated IDs is the correct mitigation.

## 2. New User Trails Defined by This Batch

### Trail 1: Share a One-Time Secret

**Entry point:** `https://send.sgraph.ai/secret` (or "Share Secret" button in admin UI)

#### Sender Journey

| Step | User Action | Component / File | Notes |
|------|-------------|-----------------|-------|
| 1 | Navigate to `/secret` | UI: `sgraph_ai_app_send__ui__user/` (new page, not yet built) | Static page served from S3/CloudFront |
| 2 | Type or paste the secret text | Client-side JS (Web Crypto) | Plaintext never leaves the browser |
| 3 | Optionally set expiry (1h/24h/7d) and passphrase | Client-side form | Defaults: 24h, no passphrase |
| 4 | Click [Create Link] | Client encrypts (AES-256-GCM), POSTs encrypted blob to `/transfers/create` + `/transfers/upload/{id}` + `/transfers/complete/{id}` | Same 3-call transfer pipeline as file send |
| 5 | Receive link: `send.sgraph.ai/s/{secret_id}#{key}` | Client constructs URL with decryption key in fragment | Server-generated opaque ID only -- no user-chosen names |
| 6 | Copy link and share via email/WhatsApp/Slack | Clipboard API | Fragment never sent to server |

#### Recipient Journey

| Step | User Action | Component / File | Notes |
|------|-------------|-----------------|-------|
| 1 | Click the link | Browser navigates to `/s/{secret_id}` | Minimal secret viewer (new page) |
| 2 | If passphrase set: enter passphrase | Client-side prompt | Key derivation from passphrase + URL fragment key |
| 3 | See the decrypted secret | Client decrypts via Web Crypto | Plaintext rendered in minimal viewer |
| 4 | Link self-destructs | Server marks as viewed, deletes encrypted blob | Subsequent access returns "This secret has already been viewed" |
| 5 | Optionally: import sender's public key | [Import Contact] button | Bootstraps PKI relationship for future encrypted replies |

**Sherpa observation:** This is the simplest trail in the system. Two personas, five steps each, zero account creation. This will be the "front door" for new users. The minimal viewer at step 3 (recipient) is critical for first impressions -- it must be beautiful and content-focused.

### Trail 2: Create and Manage a Data Room (Admin Journey)

**Entry point:** Admin UI at `https://send.sgraph.ai/admin/` (existing) or a future dedicated data room creation page

| Step | User Action | Component / File | Notes |
|------|-------------|-----------------|-------|
| 1 | Navigate to admin UI, click "Create Data Room" | `sgraph_ai_app_send__ui__admin/` | Requires admin session |
| 2 | Enter room name and description | Client-side form | Room name stored in admin metadata |
| 3 | Receive admin access key | `Service__Data_Room.create_room()` generates master key, wraps it in admin short key | Key shown once, saved to admin's vault |
| 4 | Log in to the data room with admin key | `Service__Room__Session.create_session()` | Session token scoped to room |
| 5 | Add users: set display name and role (Viewer/Editor/Admin) | `Service__Users.create()` + key generation | Each user gets a unique short access key |
| 6 | Share user keys via one-time secret link | Triggers Trail 1 (sender sub-journey) with the access key as the secret content | Dogfooding: SG/Send shares its own credentials securely |
| 7 | Manage users: show key, login as user, revoke, regenerate | Admin UI user management panel | Admin can view any user's key at any time |
| 8 | Upload files to the data room | Encrypt with master key, upload via transfer pipeline | Same encryption model as file send |

**Sherpa observation:** Step 6 is a trail intersection -- the admin journey spawns a Trail 1 (one-time secret) sub-journey to deliver the access key. This is elegant dogfooding but means Trail 1 must be fully operational before data room user onboarding works. The [Login as User] button (step 7) is a powerful demo/support tool.

### Trail 3: Join a Data Room (User Journey)

**Entry point:** One-time secret link received from admin (Trail 1 output)

| Step | User Action | Component / File | Notes |
|------|-------------|-----------------|-------|
| 1 | Receive one-time secret link from admin (email, WhatsApp, etc.) | External channel | Key delivered via Trail 1 |
| 2 | Click link, view the access key | Trail 1 recipient flow (steps 1-3) | One-time view; user must save the key |
| 3 | Navigate to `send.sgraph.ai/room/{room_id}` (or `/join`) | `sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.8/join.html` | Existing join page |
| 4 | Enter the access key | Client-side form | Key used to decrypt the room's master key |
| 5 | Browse the data room (files, folders) | `sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.8/room.html` | Existing room view |
| 6 | Interact per role: Viewer reads, Editor uploads/edits | Role-based permission enforcement | `Service__Vault__ACL.get_permission()` |

**Sherpa observation:** The critical friction point is step 2-3 transition. The user views a one-time secret (the key), then must navigate to a separate URL and enter the key. If the user loses the key between steps 2 and 3, they cannot recover it (one-time view). Two mitigations needed: (a) the secret viewer should include a "Copy to clipboard" button and clear instructions ("Save this key -- you will need it to access the data room"), and (b) the secret content should include the room URL alongside the key so the user does not need to find it separately.

### Trail 4: Generate and Save a Password

**Entry point:** `https://send.sgraph.ai/tools/password-generator` (new tool page, not yet built)

| Step | User Action | Component / File | Notes |
|------|-------------|-----------------|-------|
| 1 | Navigate to `/tools/password-generator` | New static page in `sgraph_ai_app_send__ui__user/tools/` | Zero account required, zero server involvement |
| 2 | Configure options: length, character sets, passphrase mode | Client-side JS | All generation via `crypto.getRandomValues()` |
| 3 | View generated password with entropy display (bits) | Client-side rendering | Educational: users learn what "strong" means |
| 4 | Click [Copy] or [Again] | Clipboard API / regenerate | Works without vault -- basic utility |
| 5 | Click [Save to Vault] | Prompt for label, username, URL, notes | Encrypted with user's private key |
| 6 | View saved password in vault | Vault renders dedicated password entry UI | Masked by default, copy without reveal |
| 7 | (Optional) Click [Share via Secret Link] | Spawns Trail 1 with the password as secret content | One-time link for sharing credentials securely |

**Sherpa observation:** Steps 1-4 work without any account, vault, or prior interaction. This is the gentlest on-ramp in the product. Step 5 (Save to Vault) is the conversion moment -- the first time a user engages with the vault. Step 7 is another trail intersection (spawns Trail 1). The password generator is independently useful, which makes it a strong acquisition channel.

### Trail 5: Receive a PKI-Encrypted File

**Entry point:** PKI link received from sender: `https://send.sgraph.ai/d/{key_prefix}/{doc_name}`

| Step | User Action | Component / File | Notes |
|------|-------------|-----------------|-------|
| 1 | Click PKI link | Browser navigates to `/d/{key_prefix}/{doc_name}` | New route -- minimal document viewer |
| 2 | Browser auto-selects private key based on `{key_prefix}` | Client-side key matching against stored keys | If multiple keys match prefix, prompt for selection |
| 3 | Browser fetches encrypted blob from server | GET `/transfers/download/{id}` (mapped from doc route) | Server returns opaque encrypted ciphertext |
| 4 | Browser decrypts content + metadata | AES-256-GCM with recipient's private key, then SGMETA extraction | Metadata includes friendly name, sender info |
| 5 | View decrypted content in minimal viewer | Content-type-aware viewer (PDF, image, text, markdown) | Beautiful, full-screen, minimal chrome |
| 6 | See sender identity and public key | Footer: "From: Alice" + [Import Alice's Key] | Viral PKI bootstrap loop |
| 7 | Click [Import Alice's Key] | Sender's public key saved to browser contacts | Enables encrypted replies to Alice |
| 8 | (Optional) Click [Reply to Alice] | Spawns a new send flow with Alice's key pre-selected | Completes the bidirectional PKI relationship |

**Sherpa observation:** This is the most sophisticated trail but should feel effortless. The key prefix auto-selection (step 2) is critical -- without it, the user must manually choose which key to use, which is confusing for non-technical users. The viral loop (steps 6-8) is the primary growth mechanism for PKI adoption. Every file received bootstraps a new sender-recipient relationship. The minimal viewer (step 5) must adapt to content type -- a short secret needs a centred text viewer, a PDF needs a full-screen renderer, a data room needs a folder tree.

## 3. Trail-Based Threat Modelling

As assigned in the per-file security review brief, this section maps the attack surface for the three priority trails.

### Trail A: Send a File (existing trail, enhanced by this batch)

```
Trail: "Send a file to a recipient"

  Steps:
  1. User opens send.sgraph.ai           -> sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.8/index.html
  2. User drops or selects a file         -> Client-side JS (encryption/keys)
  3. File encrypted (AES-256-GCM)         -> Web Crypto API (client-side)
  4. Encrypted blob uploaded              -> Routes__Transfers.create() / upload() / complete()
  5. Upload to storage (Memory-FS/S3)     -> Transfer__Service.upload_payload()
  6. Sender receives download link        -> Client constructs URL with key in fragment
  7. Sender shares link (out-of-band)     -> External channel (email, WhatsApp, etc.)
  8. Recipient clicks link                -> download.html
  9. Encrypted blob downloaded            -> Routes__Transfers.download__transfer_id()
  10. Recipient decrypts in browser       -> Web Crypto API (client-side)
```

| Step | Attack Surface | Threat | Severity | Current Mitigation |
|------|---------------|--------|----------|-------------------|
| 1 | Landing page served from CDN/Lambda | CDN compromise, script injection | Medium | Subresource integrity (SRI) not yet implemented |
| 2 | File selection in browser | Local malware reads selected file | Low | Out of scope (user device compromise) |
| 3 | Client-side encryption | Weak key generation, side-channel attacks | Low | `crypto.getRandomValues()` + AES-256-GCM = strong |
| 4 | `/transfers/create` endpoint | Unauthenticated transfer creation, resource exhaustion | Medium | Bearer token required; rate limiting not yet confirmed |
| 4 | `/transfers/upload/{id}` endpoint | Oversized file upload (storage exhaustion / cost attack) | Medium | File size limit client-side only; **server-side validation needed** |
| 5 | Storage backend (S3/Memory-FS) | Bucket misconfiguration, direct S3 access | Medium | Storage abstraction via Memory-FS; S3 bucket policy review needed |
| 6 | Download URL with key in fragment | URL logged in browser history, shared unsafely | Medium | Fragment not sent to server; user education on link handling |
| 7 | Out-of-band link sharing | Link intercepted in transit (email, Slack, etc.) | Medium | One-time view mitigates; PKI mode eliminates (key not in URL) |
| 8 | Download page | Script injection on download page | Medium | Static page, no dynamic content from server |
| 9 | `/transfers/download/{id}` endpoint | Transfer ID brute force | Low | 12-char random ID = sufficient entropy (~71 bits) |
| 9 | Download endpoint | Unlimited download rate | Low | Rate limiting not yet confirmed |
| 10 | Client-side decryption | n/a (correct by construction if step 3 is correct) | Low | AES-256-GCM authenticated encryption |

**Priority findings for this trail:**
1. **Server-side file size validation** -- currently client-only. An attacker can bypass the client and upload arbitrarily large files.
2. **Rate limiting on create/upload/download** -- status unconfirmed across all transfer endpoints.
3. **SRI (Subresource Integrity)** -- JS files served from CDN/Lambda are not yet protected against injection.

### Trail B: Access a Data Room

```
Trail: "Access a symmetric key data room"

  Steps:
  1. User receives access key              -> Via one-time secret (Trail 1) or direct from admin
  2. User navigates to join.html           -> sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.8/join.html
  3. User enters access key                -> Client-side form
  4. Client derives master key from access key -> Client-side key derivation
  5. Client creates room session           -> Service__Room__Session.create_session()
  6. Client fetches encrypted room index   -> Service__Vault.get_index()
  7. Client decrypts room contents         -> Web Crypto API (client-side)
  8. User browses/uploads/edits per role   -> Role-based operations via Service__Vault / Service__Data_Room
```

| Step | Attack Surface | Threat | Severity | Current Mitigation |
|------|---------------|--------|----------|-------------------|
| 1 | Key delivery channel | Key intercepted in transit | Medium | One-time secret link (self-destructs after view) |
| 2 | Join page | Phishing: fake join page captures keys | Medium | HTTPS + domain verification; user education |
| 3 | Key entry form | Keylogger on user device | Low | Out of scope (user device compromise) |
| 3 | Key entry | Brute force of short access keys | **High** | Short keys (~10 digits) have limited entropy; **rate limiting on session creation is critical** |
| 4 | Key derivation | Weak derivation allows master key recovery from short key | Medium | Need to verify derivation algorithm strength |
| 5 | Session creation endpoint | Session token prediction or hijack | Medium | `Service__Room__Session` uses cryptographic session tokens |
| 5 | Session endpoint | Unauthenticated session creation | **High** | Must require valid access key; **verify enforcement** |
| 6 | Room index endpoint | Enumeration of room IDs | Medium | Room IDs should be opaque and high-entropy |
| 7 | Client-side decryption | n/a if key derivation is correct | Low | AES-256-GCM |
| 8 | Role-based operations | Permission bypass (viewer performs editor actions) | Medium | `Service__Vault__ACL.get_permission()` enforces; **needs AppSec review** |

**Priority findings for this trail:**
1. **Access key brute force** -- short keys (~10 digits) may have insufficient entropy for an internet-facing service. Rate limiting on session creation is essential. Consider lockout after N failed attempts per room.
2. **Session creation authentication** -- the session endpoint must reject invalid access keys before issuing tokens. Verify this is enforced server-side, not just client-side.
3. **Room ID enumeration** -- if room IDs are guessable, an attacker can attempt to brute-force access keys against known rooms. Room IDs must be high-entropy.

### Trail C: Admin Operations

```
Trail: "Admin manages a data room"

  Steps:
  1. Admin authenticates (admin session)    -> Admin Lambda (separate from user Lambda)
  2. Admin creates/manages rooms            -> Service__Data_Room.create_room() / get_room()
  3. Admin creates users and keys           -> Service__Users.create() + key generation
  4. Admin shares keys via one-time link    -> Triggers Trail 1 (sender sub-journey)
  5. Admin views user keys [Show Key]       -> Admin UI
  6. Admin revokes/regenerates keys         -> Admin key management
  7. Admin impersonates user [Login as User]-> Opens room view with user's permissions
  8. Admin views audit trail                -> Service__Audit.get_room_events()
```

| Step | Attack Surface | Threat | Severity | Current Mitigation |
|------|---------------|--------|----------|-------------------|
| 1 | Admin authentication | Credential theft, session hijack | **High** | Separate Lambda with its own auth; token-based access |
| 1 | Admin endpoint exposure | Admin Lambda accessible from public internet | **High** | Lambda URL function; IP restriction recommended but not confirmed |
| 2 | Room creation | Uncontrolled room creation (resource exhaustion) | Medium | Admin auth required; quota not yet implemented |
| 3 | User/key creation | Key generation weakness | Medium | Need to verify crypto source for short key generation |
| 4 | Key sharing | Key intercepted during sharing | Medium | One-time secret mitigates; one-time view enforced |
| 5 | Show Key | Admin session compromise exposes all user keys | **High** | Admin can see all keys by design; **admin session security is paramount** |
| 6 | Key revocation | Revoked key still cached client-side | Medium | Simple revocation deletes server mapping; cached master key still works until re-encryption |
| 7 | Login as User | Admin impersonation logged? | Medium | Must appear in audit trail; **verify audit coverage** |
| 8 | Audit trail | Audit events tampered or deleted | Medium | Append-only audit log recommended; **verify immutability** |

**Priority findings for this trail:**
1. **Admin Lambda exposure** -- the admin Lambda must not be publicly accessible without strong authentication. IP restriction or VPN-gating recommended for production.
2. **Admin session security** -- a compromised admin session gives access to all user keys, all room data, and impersonation capability. MFA, session expiry, and audit logging are critical.
3. **Key revocation gap** -- simple revocation (delete mapping) leaves cached master keys functional. The brief acknowledges this. For production data rooms with sensitive content, full re-encryption must be available.
4. **Audit immutability** -- audit events must be append-only and tamper-resistant. An admin who can delete audit entries can cover their tracks.

### What Works Today

Based on [SKILL.md](../../../../../../team/humans/dinis_cruz/briefs/02/26/mcp/SKILL.md) and the [n8n + WorkMail debrief](../../../../../../team/humans/dinis_cruz/briefs/02/26/mcp/n8n-workmail-ews-claude-mcp-debrief.md):

| Email inbox reading | Operational | `list_inbox`, `get_message` via n8n MCP |
| Email reply (threaded) | Operational | `reply` operation preserves threading |
| SG/Send token validation | Operational | `transfers_check-token` |

| XML-escaped HTML email bodies | Operational | EWS SOAP XML escaping documented and working |

### What the New Trails Require from MCP

| New Capability Needed | Trail(s) | Current Support | Gap |
|----------------------|----------|-----------------|-----|
| One-time secret creation via MCP | 1, 2 (step 6), 4 (step 7) | Not available | New MCP tool needed: `secrets_create(content, expiry, passphrase?)` |
| One-time secret status check | 1 | Not available | New MCP tool: `secrets_status(secret_id)` -> viewed/expired/active |
| Data room creation via MCP | 2 | Not available | New MCP tool: `rooms_create(name, description)` |
| Data room user management via MCP | 2 | Not available | New MCP tools: `rooms_add_user`, `rooms_revoke_user`, `rooms_list_users` |
| Data room file upload via MCP | 2 | Partially available (transfer pipeline works) | Need room-scoped upload: `rooms_upload(room_id, file, access_key)` |
| Password generation via MCP | 4 | Not available | Lower priority -- password generation is inherently a client-side operation |
| PKI key management via MCP | 5 | Not available | Future -- PKI mode not yet implemented |

### Gap Assessment

The current MCP pipeline is optimised for one workflow: **encrypt a file, upload it, email the link**. The new trails expand the required MCP surface in two directions:

1. **Ephemeral data operations** (one-time secrets) -- the transfer pipeline could be adapted for this since secrets are just small encrypted blobs with expiry metadata. The MCP tools `transfers_create` / `transfers_upload` / `transfers_complete` could be extended with expiry parameters, or new dedicated `secrets_*` tools could be created. The Sherpa recommends dedicated tools for clarity.

2. **Data room management** -- this requires entirely new MCP capabilities. Room creation, user management, and room-scoped file operations are not covered by the current transfer pipeline. These will require new admin-side MCP tools.

**Recommended priority order for MCP extension:**
1. `secrets_create` / `secrets_status` (enables Trail 1 via MCP, unblocks Trail 2 key sharing)
2. `rooms_create` / `rooms_add_user` / `rooms_revoke_user` (enables Trail 2 admin operations via MCP)
3. Room-scoped file upload (enables Trail 2 file management via MCP)
