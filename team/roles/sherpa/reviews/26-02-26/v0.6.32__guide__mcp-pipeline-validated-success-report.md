## 1. Validation Summary

The Sherpa Agent MCP workflow documented in [`v0.6.20__guide__mcp-client-session-prompt.md`](../26-02-25/v0.6.20__guide__mcp-client-session-prompt.md) has been **validated end-to-end in production** on 26 February 2026. This is the first confirmed real-world use of the complete pipeline: PDF generation, SGMETA envelope wrapping, AES-256-GCM encryption, MCP upload via the `data` parameter, email delivery via n8n MCP connector, and browser-side decryption with filename preservation.

The workflow was executed by Claude.ai using the Sherpa Agent system prompt with MCP connectivity to both SGraph Send and an n8n email connector backed by AWS WorkMail. Every step completed successfully on the first attempt. The zero-knowledge guarantee held — the server never saw plaintext, the decryption key never left the sender, and the transparency panel in the browser confirmed this.

**Bottom line:** The guide works. The pipeline is proven. MCP clients can follow the documented workflow and achieve correct, secure, end-to-end encrypted file sharing.

## 2. What Was Tested

The following steps were executed in sequence during a single Claude.ai session:

| Step | Action | Tool / Method |
|------|--------|---------------|
| 1 | Generate a 2-page PDF document | Claude.ai internal generation |
| 2 | Build SGMETA envelope around PDF bytes | SGMETA binary format (magic + metadata JSON + file content) |
| 3 | Encrypt the SGMETA-wrapped payload with AES-256-GCM | Web Crypto API equivalent (256-bit key, 96-bit nonce) |
| 4 | Base64-encode the encrypted payload | Standard base64 encoding for MCP transport |
| 5 | `transfers_create` — create a new transfer | SGraph Send MCP (User Lambda) |
| 6 | `transfers_upload` — upload via `data` parameter | SGraph Send MCP (User Lambda), base64 payload in `data` field |
| 7 | `transfers_complete` — mark transfer complete | SGraph Send MCP (User Lambda) |
| 8 | Build download URL with key in fragment | `https://send.sgraph.ai/send/v0/v0.1/v0.1.4/download.html#{id}/{key}` |
| 9 | Send email with download link to AWS WorkMail | n8n MCP connector (email action) |
| 10 | Recipient opens link in browser | Manual browser test |
| 11 | Browser decrypts payload, extracts SGMETA, renders PDF | download.html + SendCrypto.decryptFile |

All 11 steps completed without error or retry.

## 3. Results

### Encryption and Upload
- Transfer created, uploaded, and completed via MCP tools without error

### Email Delivery
- Download URL sent via n8n MCP connector to an AWS WorkMail inbox
- Email arrived within seconds
- The download link contained the transfer ID and base64url-encoded decryption key in the URL fragment

### Browser Decryption
- Recipient clicked the link; the download page fetched the encrypted payload from the server
- **Decryption time: 251ms** — fast, seamless experience
- **File size: 5.0 KB** (encrypted payload)
- **Output: 2-page PDF** rendered inline in the browser

### Filename Preservation
- Original filename `encrypted-send-workflow.pdf` was preserved through the entire pipeline
- The browser detected the SGMETA magic bytes after decryption, parsed the metadata JSON, and applied the filename
- Without SGMETA, the file would have downloaded with no name — this confirms the envelope is essential for MCP clients

### Zero-Knowledge Confirmation
- The transparency panel on the download page confirmed zero-knowledge properties:
  - Server stored only encrypted ciphertext
  - Decryption key was in the URL fragment (never sent to server)
  - No plaintext or filename was visible server-side

## 5. What the Guide Got Right

The MCP client session prompt (`v0.6.20__guide__mcp-client-session-prompt.md`) proved accurate and complete for this workflow. Specific validations:

### SGMETA Envelope Format
The binary format documented in Section 4 of the guide was implemented exactly as specified:
- `SGMETA\0` magic bytes (7 bytes)
- Big-endian 4-byte metadata length
- UTF-8 JSON metadata (`{"filename": "encrypted-send-workflow.pdf"}`)
- Raw file content follows

The browser's SGMETA parser recognized and extracted the metadata without issue. The format is correct and interoperable between MCP clients and the browser download page.

### `data` Parameter for Upload
The guide's emphasis on using the `data` parameter (not the request body) for `transfers_upload` was critical. This is the fix that made MCP uploads reliable — MCP JSON-RPC transport cannot pass raw binary in the request body. The guide's warning about this was prominent and correct.

### Encryption Workflow
The AES-256-GCM encryption with 12-byte nonce prepended to ciphertext matched the browser's `SendCrypto.encryptFile` exactly. The key encoding (base64url without padding) for the URL fragment also worked correctly.

### Authentication via Tool Parameter
The `access_token` parameter approach for Claude.ai web (which cannot inject custom HTTP headers) worked as documented. No authentication failures occurred.

### Download URL Format
The URL pattern `https://send.sgraph.ai/send/v0/v0.1/v0.1.4/download.html#{transfer_id}/{key_b64url}` was correct. The fragment-based key delivery ensures zero-knowledge — the server never receives the decryption key.

## 6. Lessons Learned

### The Pipeline Is Simpler Than Expected
The end-to-end workflow, including SGMETA wrapping and encryption, completed in a single session without debugging or retries. The guide's Python reference implementation and step-by-step documentation provided sufficient information for Claude.ai to execute the workflow autonomously.

### Email as Distribution Channel Works Naturally
Adding email delivery via n8n MCP connector was a natural extension of the upload workflow. Once the download URL is generated, any communication channel (email, Slack, SMS) can distribute it. The guide did not explicitly cover email distribution — this is an enhancement opportunity.

### SGMETA Is Non-Negotiable for MCP Clients
Without the SGMETA envelope, recipients would receive an unnamed blob. The first real-world test confirmed that SGMETA is not optional — it is essential for a usable experience. The guide's documentation of this was thorough and the format was implemented correctly on the first attempt.

### Claude.ai Can Execute the Full Crypto Workflow
Claude.ai successfully performed AES-256-GCM encryption, SGMETA binary construction, and base64 encoding within a single session. This demonstrates that the Sherpa Agent system prompt provides sufficient context for an AI agent to handle the complete zero-knowledge pipeline without human intervention on the crypto side.

### Small File Performance Is Excellent
At 5.0 KB, the entire pipeline (generate, wrap, encrypt, upload, email, decrypt) completed in under 30 seconds. There is no performance concern for documents, small files, or text-based payloads.

## 7. Recommended Enhancements

Based on this first successful run, the following enhancements should be added to the MCP client guide in future versions:

### 7.1 Email Distribution Workflow
Add a new section documenting the email delivery pattern:
- Generate download URL after `transfers_complete`
- Use an email MCP connector (n8n, SendGrid, etc.) to deliver the URL
- Include guidance on email body formatting (the link should be prominent, explain what the recipient will see)
- Note that the decryption key is in the URL fragment — email systems that strip fragments could break the workflow (test with target email providers)

### 7.2 Multi-Connector Composition Pattern
Document the pattern of composing multiple MCP connectors in a single session:
- SGraph Send MCP for encryption and upload
- n8n MCP (or equivalent) for email, Slack, or other delivery
- This is the "Sherpa Agent" pattern — an AI agent orchestrating multiple services via MCP

### 7.3 Larger File Testing
The validated test used a 5.0 KB payload. The guide should include:
- Tested size thresholds (5 KB validated, 5 MB Lambda limit, 3.75 MB base64 limit)
- When to switch from `data` parameter to presigned URL workflow
- Expected performance at 1 MB, 5 MB, and 100 MB+ sizes

### 7.4 Error Recovery Workflow
Add guidance for partial failures:
- What happens if `transfers_upload` succeeds but `transfers_complete` fails? (retry the complete call)
- What happens if the email fails to send? (the transfer is still complete; retry the email with the same URL)
- What happens if the recipient reports decryption failure? (verify key encoding, check SGMETA format)

### 7.5 Batch Send Pattern
Document sending the same encrypted file to multiple recipients:
- Upload once, distribute the same download URL to multiple email addresses
- Each recipient gets the same link; no re-upload needed
- Optionally, create separate transfers per recipient for independent expiry/tracking

### 7.6 Content Type Guidance
Add a table of common `content_type_hint` values and how they affect the download experience:
- `application/pdf` — inline rendering in browser
- `image/png`, `image/jpeg` — inline image display
- `text/plain` — text display
- `application/octet-stream` — generic download prompt
