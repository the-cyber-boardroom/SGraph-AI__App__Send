## Executive Summary

SG/Send today is a browser-only experience. Files go in through a web page, come out through a web page. That's powerful for human-to-human sharing, but it locks out the fastest-growing user of secure infrastructure: **software agents**.

The investor who coined "SigSend" described the workflow explicitly: _"I'm in Claude, composing an NDA. I say 'send this to the founder securely.' Claude encrypts it, uploads it, emails the link. Done."_

That workflow requires three things we don't have yet:
1. **A CLI tool** (`sg_send`) — encrypt and upload from any terminal
2. **An MCP server** — so Claude Desktop, Claude Code, Cursor, and other AI tools can call SG/Send natively
3. **A Python library** — so any script, CI pipeline, or agent can send/receive encrypted files programmatically

All three share the same core: a Python module that replicates the browser's AES-256-GCM encryption and speaks the SG/Send API. Build one, get three.

## The Opportunity: Three Products from One Codebase

### Product 1: The CLI Tool (`sg_send`)

```bash
# Send a file
sg_send upload --file report.pdf --server https://send.sgraph.ai --token my-token
# → {"transfer_id": "abc123", "key": "base64url...", "download_url": "https://...#abc123/key"}

# Receive a file
sg_send download --url "https://send.sgraph.ai/d/abc123#key" --output report.pdf

# Send encrypted to a specific recipient (PKI mode)
sg_send upload --file nda.pdf --recipient DC-7X4F

# Look up someone's public key
sg_send keys lookup DC-7X4F

# Publish your public key
sg_send keys publish --key-file my-key.pub

# Check transfer status
sg_send info --transfer-id abc123

# Health check
sg_send health
```

**Distribution:** `pip install sg_send` or `uvx sg_send` (zero-install, runs immediately).

**Target users:** Developers, scripts, CI pipelines, any LLM with shell access.

### Product 2: The MCP Server

```json
// Claude Desktop configuration (claude_desktop_config.json)
{
  "mcpServers": {
    "sg-send": {
      "type": "stdio",
      "command": "uvx",
      "args": ["sg_send", "mcp"],

        "SGRAPH_SEND_TOKEN": "your-token",
        "SGRAPH_SEND_URL": "https://send.sgraph.ai"
```

Once configured, any MCP-compatible LLM can:

| Tool | What It Does |

| `key_lookup` | Look up a public key by short code |
| `key_publish` | Publish a public key to the registry |
| `health` | Check server status |

**Target users:** Claude Desktop, Claude Code, Cursor, Windsurf, any MCP-compatible AI tool.

**Compatible platforms** (MCP support):
- Claude Desktop / Claude Code / Claude.ai (Anthropic)

### Product 3: The Python Library

```python
from sg_send import SG_Send

client = SG_Send(server="https://send.sgraph.ai", token="my-token")

# Upload
result = client.upload("report.pdf")
print(result.download_url)  # Share this with the recipient

# Download
client.download(transfer_id="abc123", key="base64url...", output="report.pdf")

# PKI: send to a specific recipient
result = client.upload("nda.pdf", recipient_code="DC-7X4F")

# Key management
my_code = client.keys.publish(public_key_pem="...")
their_key = client.keys.lookup("DC-7X4F")
```

**Target users:** Python developers, automation scripts, GitHub Actions, custom agents.

### The Encryption Format (EXISTS — in App__Send)

The browser uses AES-256-GCM via Web Crypto API. The Python CLI must produce **format-compatible** ciphertext:

```
Payload format: [12-byte IV][ciphertext + 16-byte GCM auth tag]
Key encoding:   base64url without padding (44 characters)
Algorithm:      AES-256-GCM, no AAD (Associated Authenticated Data)
```

The Architect already wrote the exact Python implementation (v0.4.17):

```python
from cryptography.hazmat.primitives.ciphers.aead import AESGCM
import os, base64

def encrypt_file(plaintext: bytes) -> tuple[bytes, str]:
    key_bytes  = AESGCM.generate_key(bit_length=256)
    aesgcm     = AESGCM(key_bytes)
    iv         = os.urandom(12)
    ciphertext = aesgcm.encrypt(iv, plaintext, None)
    payload    = iv + ciphertext
    key_b64    = base64.urlsafe_b64encode(key_bytes).rstrip(b'=').decode()
    return payload, key_b64

def decrypt_file(payload: bytes, key_b64url: str) -> bytes:
    key_b64   = key_b64url + '=' * (4 - len(key_b64url) % 4)
    key_bytes = base64.urlsafe_b64decode(key_b64)
    aesgcm    = AESGCM(key_bytes)
    iv        = payload[:12]
    ciphertext = payload[12:]
    return aesgcm.decrypt(iv, ciphertext, None)
```

**Cross-compatibility is critical:** A file encrypted by the CLI must be decryptable in the browser, and vice versa. This is the acceptance test that validates the crypto module.

### The PKI System (EXISTS — in App__Send)

Browser PKI is working:
- Key generation (RSA-OAEP 4096, ECDH P-256, Ed25519)
- Public key registry with short lookup codes (`DC-7X4F` format)
- Hash-chained transparency log (tamper-evident)
- Encrypt/decrypt messages with public/private keys

**For CLI PKI mode**, the CLI needs to:
1. Look up a recipient's public key by code (API call to `/keys/lookup/{code}`)
2. Perform hybrid encryption: ECDH key agreement → AES-256-GCM symmetric encryption
3. Upload the encrypted payload (same as symmetric mode)
4. Include key metadata so the recipient's browser/CLI can identify which private key to use

## Trust Model: Why This Is Safe

### The Trust Shift

```
Browser model:     User → Browser (encrypt) → Server (ciphertext only)
CLI/MCP model:     User → LLM (sees plaintext) → CLI (encrypt) → Server (ciphertext only)
```

When a user asks Claude to "send this securely," Claude has **already read the document** — the user composed it in Claude or shared it for review. The CLI doesn't introduce a new trust boundary. It just automates what the user would do manually in the browser.

### What's Preserved

| Security Property | Browser | CLI/MCP |
|-------------------|---------|---------|
| Server never sees plaintext | Yes | Yes |
| Server never sees decryption key | Yes | Yes |
| File names never sent to server | Yes | Yes |
| Per-transfer ephemeral keys | Yes | Yes |
| IP addresses hashed | Yes | Yes |
| Hash-chained audit trail | Yes | Yes |

### What Changes

| Property | Browser | CLI/MCP |
|----------|---------|---------|
| Who sees plaintext | Only the browser (user's device) | The LLM process (user's device or cloud) |
| Where keys exist | Browser memory only | Python process memory + LLM context window |
| Key persistence | None (browser forgets on page close) | None (CLI process exits, key lost) |

**The Architect documented the formal trust analysis in v0.4.17.** The conclusion: CLI/MCP maintains zero-knowledge for the server. The user's trust boundary extends to include the LLM, which they chose to trust by sharing the document.

4. **Narrative:** "Your AI assistant sends. Your browser decrypts. No key exchange. No friction."

## Summary

The CLI project transforms SG/Send from a browser-only tool into **programmable zero-knowledge infrastructure**. Three products (CLI, MCP, Python library) from one codebase, all preserving the zero-knowledge security model. The investor workflow — "Claude, send this securely" — becomes possible. And every AI platform gets native access to encrypted file transfer.

**Build the crypto module first.** It's the foundation for everything else, and it validates the most important property: Python ↔ JavaScript encryption compatibility.
