## Summary

The Personal Data Vault is a product-defining feature that shifts SG/Send's positioning from "encrypted file transfer" to "encrypted workspace." This review captures the messaging implications and positioning updates.

---

## Product Positioning Update

### Before This Session
> SG/Send: Zero-knowledge encrypted file sharing. Send files that only the recipient can decrypt.

### After This Session
> SG/Send: Zero-knowledge encrypted workspace. Store, edit, preview, and share files — all encrypted, all client-side, all yours.

The vault adds **persistence** to what was previously a **transient** product. Files no longer disappear after transfer. Users now have a reason to come back.

---

## Key Messaging Points

### 1. "Edit encrypted files without decrypting on the server"
This is the headline. No competitor offers in-place editing of encrypted files with zero-knowledge guarantees. Traditional encrypted storage (Tresorit, SpiderOak) decrypts server-side for editing. SG/Send decrypts in the browser, edits in the browser, re-encrypts in the browser.

### 2. "Your vault, your keys"
Users select which PKI key to use for their vault. Different keys = different vaults. This maps to real-world use cases: personal vault, work vault, project vault. All using the same system, different trust boundaries.

### 3. "From storage to sharing in one click"
The Share via Send integration connects vault storage to the Send transfer workflow. No download-then-upload cycle. The vault becomes the outbox.

### 4. "Preview without downloading"
Compliance use case: review a sensitive document without creating a local copy. The file never touches the filesystem — it's decrypted in memory and rendered in the browser.
