# Ambassador Review: Website Homepage
**Version:** v0.21.9 | **Date:** 2026-04-22 | **URL reviewed:** `https://dev.sgraph.ai/en-gb/`
**Role:** Ambassador | **Method:** Live page fetch + reality document cross-check

---

## 1. Executive Summary

The homepage has strong raw material — a technically credible privacy story, honest open-source positioning, and a genuinely differentiated "agents and humans" angle — but it is trying to speak to too many audiences at once and succeeds fully with none of them. The developer/infrastructure framing (Lambda, Fargate, ECR, deploy strips) dominates the above-the-fold experience while the viral loop entry point (token paste) is buried in a secondary CTA. Five use-case sections with no screenshots undermine the credibility the copy is trying to build.

---

## 2. Audience Clarity

**Who the page is currently targeting (in order of emphasis):**
1. Developer/infrastructure buyer — deploy strip, AWS Stack section, open-source cards
2. Security researcher — first use case, zero-knowledge emphasis
3. Multi-agent workflow builder — primitives section, "For humans and agents" headline
4. Healthcare — one use case
5. Anyone with a share token — secondary CTA, header token bar

**Problem:** These audiences have different pains and different vocabulary. Developers care about deploy targets; healthcare teams care about HIPAA-adjacent language; agent builders care about programmatic access. The page doesn't choose a primary audience and pays the price in diluted messaging.

**What works:** "For humans and agents" in the hero headline is genuinely differentiating. No competitor currently claims this clearly, and it establishes a dual-audience framing from the first second.

**Recommendation:** Pick a primary audience for the above-the-fold experience. The agent/developer audience is the most defensible competitive position and the hardest for incumbents to copy. Healthcare and security can live further down the page.

---

## 3. Positioning and Messaging

### Hero headline
> "Version-controlled, client-encrypted vaults. For humans and agents."

**Assessment:** "Version-controlled, client-encrypted vaults" is accurate but abstract. First-time visitors don't yet know what a "vault" means in this context. The "For humans and agents" suffix is the most distinctive phrase on the page.

**What lands:**
- The encryption badge (`AES-256-GCM · Client-side only · Server never sees plaintext`) is strong and specific — more credible than vague "encrypted" claims
- "The server is blind." is excellent micro-copy — concrete, memorable, and technically honest

**What doesn't land:**
- "Version-controlled" may be meaningful to developers but confuses the casual user; it should be explained or replaced earlier
- The lead paragraph ("Every commit, every share, every agent: encrypted in your browser. The server is blind.") is better than the headline — consider elevating it

### Three Primitives section
The "missing primitives for agentic development" framing is strong for a developer/agent builder audience. The problem is it immediately follows the hero section, so before the casual user understands what SG/Send *is*, they are being told about its role in an "agentic development" ecosystem.

### Deploy strip
> "Runs on: Lambda · Fargate · EC2 / AMI · Docker · GCP · CLI · Self-hosted"

This is positioned immediately after the hero CTAs, between the user and the first CTA click. For a non-developer audience it is noise. For a developer audience it is reassuring but premature — they need to understand the product before they care about deploy targets.

## 4. Trust Signals

**Strong trust signals present:**
- AES-256-GCM encryption badge in the hero (specific and verifiable)
- "The server is blind." — memorable, concrete, honest
- "Our privacy model in six sentences" section — exceptional. Six clear, legally-grounded, non-marketing sentences. This is rare and earns trust.
- Open source positioning — source code + CLI on GitHub, self-host option
- "© 2026 SGraph AI Ltd. Released under CC BY 4.0 where applicable." — specific legal entity, specific licence

**Weak trust signals:**
- Five use-case sections all carry "Screenshot coming" placeholders. For a product whose core claim is that it exists and works, the absence of any working screenshot is the single biggest credibility gap on the page.
- No customer testimonials, case studies, or real usage examples
- The privacy section appears at the bottom of the page, after pricing — it should be higher given that trust is the entire value proposition

**Trust anti-signals:**
- "Coming soon" on Code Execution Sandbox is correctly labelled — this is good honesty
- Screenshot placeholders are labelled "Screenshot coming" rather than silently missing — also good honesty, but still damaging

## 6. Viral Loop Analysis

The viral loop for SG/Send is: **file received → token used → vault understood → new vault created**.

**How the page supports it:**

| Loop stage | Page element | Assessment |
|---|---|---|
| Recipient lands with token | Header token bar "Paste a token →" | ✅ Present, but small and secondary |
| Recipient understands what they received | "Already have a token? Paste it here →" in hero | ✅ Present as secondary CTA |
| Recipient decides to send their own | "Send a file — try it →" primary CTA | ✅ Present |
| Recipient understands the product before deciding | Privacy section, how-it-works nav link | ⚠️ Requires scrolling or navigation |

**Key gap:** A recipient arriving with a token is the highest-intent visitor on the site. They already have proof that SG/Send exists and that someone they know uses it. The page does not capitalise on this. There is no moment that says "the person who sent you this used SG/Send — here's how to do the same." The viral loop is supported but not optimised.

**The token bar in the header** is the right instinct — persistent, always available — but it is visually minimal. First-time recipients may not notice it.

## 8. Competitive Differentiation

**What genuinely differentiates SG/Send from alternatives (Bitwarden Send, WeTransfer, Signal, OnionShare, Tresorit):**

1. **Agent-native** — no competitor explicitly addresses agentic workflows or LLM-to-human file passing. This is unique and defensible.
2. **Version-controlled vaults** — git-like commit model for encrypted files is not available elsewhere
3. **Self-hostable with same codebase** — same code runs Lambda, Docker, EC2, GCP
4. **Zero-knowledge + open source** — the combination of client-side encryption AND published source code is rare
5. **No account required to receive** — reduces friction for the viral loop recipient

**What the page communicates about differentiation:**
- Agent positioning: ✅ Clear in hero and primitives section
- Version control: ✅ In hero headline
- Self-host: ✅ In deploy strip and open source section
- Zero-knowledge: ✅ Privacy section, encryption badge
- No-account-to-receive: ✅ Mentioned in pricing section but not in the hero

**Gap:** The "no account required to receive" point is buried in the pricing section. This is one of the strongest viral loop enablers and should be in the hero or immediately after.

## 9. Prioritised Recommendations

### R1 — Add real screenshots to all use-case sections
- **Issue:** All five use-case sections show "Screenshot coming" placeholder. This is the single biggest credibility gap on the page.
- **Growth principle:** Trust signals / Honest positioning
- **Impact:** Removes the strongest visual evidence that the product does not yet exist. Even one real screenshot (e.g. the vault browse UI) would break the pattern.
- **Recommendation:** Prioritise shipping at least one screenshot per use case, starting with the vault browse view (most mature UI). A real UI screenshot in the first use case is more valuable than five polished placeholder messages.

### R2 — Surface "no account required to receive" in the hero
- **Issue:** The strongest viral loop enabler is buried in the pricing section.
- **Growth principle:** Viral loop optimisation
- **Impact:** Recipients arriving with a token need to know immediately that they can open it without creating an account. This reduces drop-off at the most critical conversion moment.
- **Recommendation:** Add a third bullet to the encryption badge strip or a line below the hero lead: "No account needed to receive. Open any vault with a token."

### R3 — Move the privacy section above the pricing section
- **Issue:** For a zero-knowledge product, privacy is the product. The six-sentence privacy model is excellent copy — it should appear before the £5 CTA, not after.
- **Growth principle:** Trust signals / Trust-first positioning
- **Impact:** Users considering a purchase need trust before they need pricing. Reordering increases conversion at the £5 CTA.
- **Recommendation:** Swap the privacy and pricing section order. Privacy → Pricing.

### R4 — Add a "recipient to sender" moment after the token bar
- **Issue:** The viral loop has no explicit bridge from "I received a vault" to "I want to send one."
- **Growth principle:** Viral loop / Recipient-to-sender conversion
- **Impact:** The highest-intent user (someone who just opened a vault) is not shown a clear path to becoming a sender.
- **Recommendation:** When the token bar is used and a vault is opened, show a small contextual prompt: "Liked what you received? Send your own vault in 60 seconds →" This belongs in the browse UI, but the homepage could prime the expectation.

### R5 — Move the deploy strip below the fold
- **Issue:** The deploy strip (`Runs on: Lambda · Fargate · EC2...`) appears between the hero and the first use case, interrupting the narrative before the casual user understands the product.
- **Growth principle:** Audience clarity / Hierarchy
- **Impact:** Non-technical users bounce before reaching the use cases. Developers who care about deploy targets can find this information in the open source section or a dedicated architecture page.
- **Recommendation:** Move the deploy strip to the open source section or create a dedicated "How to deploy" page linked from the footer.

### R6 — Clarify what £5 buys before the CTA
- **Issue:** "One Starter Pack. Enough credits to send files, create vaults, and run your first agentic workflow." — what does a credit correspond to? How many files? How large?
- **Growth principle:** Conversion / Data-driven positioning
- **Impact:** Vague credit descriptions reduce purchase confidence. Users want to know if £5 is enough for their use case before clicking.
- **Recommendation:** Add one concrete example: "£5 = 50 file sends, or 10 vault creates, or 1 month of daily agent workflows." Even approximate numbers are more confidence-building than abstract "credits."

### R7 — Make the competitive moat explicit
- **Issue:** The page lists many features but never states clearly: "Unlike Bitwarden Send / WeTransfer / Signal, SG/Send is the only tool that..." 
- **Growth principle:** Competitive differentiation / Honest positioning
- **Impact:** Users comparing tools cannot easily articulate why SG/Send wins. The differentiation exists but is distributed across the page rather than stated.
- **Recommendation:** Add a single comparison callout box (or a "Why SG/Send?" section) that states the top 3 differentiators in one place. Agent-native, version-controlled, and self-hostable are the three that no competitor can match today.

### R8 — Strengthen the token bar's visual presence
- **Issue:** "Paste a token →" in the header is the primary viral loop entry point but has low visual weight compared to the navigation links.
- **Growth principle:** Viral loop / Recipient conversion
- **Impact:** Recipients arriving with a token may not find the input field on first scan, increasing drop-off.
- **Recommendation:** Give the token bar a distinct visual treatment (coloured background, slightly larger font, or an icon) that makes it stand out from the navigation. It should be the most visible element in the header for a recipient.
