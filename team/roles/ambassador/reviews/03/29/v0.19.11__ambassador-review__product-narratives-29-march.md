## Document 1: "What is SG/Send" — Assessment

### Strengths

**Positioning clarity.** The north-star claim is crisp and verifiable:
> "We cannot read your files — and you never have to download them to view them."

This is a structurally sound competitive differentiator. WeTransfer cannot say the first clause (server-side encryption = they can). Proton cannot say the second (download-to-view model). The claim is supported by technical architecture, not just policy.

**Feature coverage is complete.** All 7 major feature categories (upload, sharing, receiving/viewing, tokens, vaults, rooms, sgit, MCP, Workspace) are covered at the right level of abstraction for an external audience. No missing features; no false claims.

**The "architectural, not policy" framing.** The phrase "This is architectural, not policy. SG/Send **cannot** read your files. Not 'will not.' Cannot." is the strongest sentence in the document. It reframes zero-knowledge encryption from a trust claim (which competitors can also make) to a structural impossibility (which they cannot).

**sgit and MCP positioning.** Including sgit and MCP in the product overview is correct. These are genuine differentiators: zero-knowledge collaboration for AI agents and developer workflows is not offered by WeTransfer, Proton, or similar products.

**Reality-grounded.** The document explicitly states: "Everything below has been code-verified. If it's listed here, it works." This is the correct approach for an Ambassador-produced document — claim only what is verified.

### Weaknesses / Gaps

**"Data Rooms" section is sparse.** Rooms have full audit trail, invite lifecycle, session management, and a file browser. The section lists these but doesn't explain *why* data rooms matter for the target audience. Who uses rooms? What problem do they solve? The upload and sharing sections have strong "why" framing; rooms doesn't.

**"Workspace" section lacks a use case.** The Workspace (v0.1.0) section describes the feature mechanically (five-zone layout, LLM providers, document pipeline) but doesn't show a single concrete use case. Example: "A lawyer sends encrypted contracts to a client via vault; an AI agent on either side summarises, translates, or redlines — all without plaintext ever hitting the server." Without a use case, technical readers will understand it; non-technical readers won't know why to care.

**Pricing model not fully explained.** The document mentions "£0.01 per file" and "Starter Pack — £5 for 500 transfers" but doesn't mention that the token system is the payment mechanism, or that the token is itself delivered via SG/Send (which is a remarkable product-as-delivery story). This is addressed in the tokens section ("Starter Pack: £5 via Stripe → 500 transfer token delivered via encrypted SG/Send transfer") but could be called out more prominently.

**The agent team section** (last third of document) is thorough but may overwhelm a non-technical reader. The "18-agent AI team" model is a differentiator for investors and technically curious readers, but for a product overview aimed at end users, it may be better as an optional section or sidebar.

### Recommended Use Cases

This document is appropriate as:
- The canonical "What is SG/Send" landing page copy (after trimming the agent team section)
- Background reading for journalists, analysts, and potential investors
- The basis for the `/product` page on sgraph.ai

## Document 2: "SG/Send v0.3.0 — The Refactoring" — Assessment

### Strengths

**Honest about the problem.** The "before" description is unflinching: 16-layer prototype mutation chain, 7,328 lines, CSS embedded in JavaScript, 27 symlinks, zero components using the base class. This level of honesty builds credibility. Readers who have experienced similar technical debt will recognise the pattern.

**The "why" is clear.** The document explains that IFD (Iterative Feature Development) is a deliberate methodology — fast for exploration, but requiring eventual consolidation. The v0.3.0 refactoring is not a bug fix or a crisis response; it's the planned IFD lifecycle step: "take everything that works, throw away the scaffolding, and build a clean foundation."

**Quantified before/after.** The code metrics table is compelling:
- Upload workflow: 7,328 lines (16 files, prototype chain) → 5,160 lines (25 files, modular)
- External symlinks: 27 → 0
- Component base class usage: 0 components → 10 components migrated

Concrete numbers make the magnitude of the change legible.

**Critical bugs section.** Calling out the P0 bug (LLM hallucinated a phantom method during the rewrite) is a valuable signal for the AI-assisted development story. It demonstrates that the team has quality controls: the QA session caught it, it was fixed. This is more credible than claiming AI agents are infallible.

**Build workflow section.** The vault-as-patch-transport pattern is explained clearly: Claude Web (no git access) produces `git format-patch` output, pushes it to an encrypted vault, the human pulls and applies with `git am`. This is a technically elegant workflow and positions the product as self-referential (we build with our own tools).

### Weaknesses / Gaps

**Audience is unclear.** This article sits between a technical deep-dive (the architecture section) and a product story (the refactoring narrative). Engineers will want more code — method signatures, before/after diffs. Non-engineers will lose interest at the component architecture. Consider two versions: a 500-word product blog post for general audiences, and a full technical article (this document) for engineers and investors.

**No "what's next" section.** The document ends with "ready for the next phase of growth" but doesn't say what that phase is. Even a single sentence ("v0.3.1 adds PDF present mode and folder tree basenames; the next major version will focus on X") would give readers a forward-looking hook.

**Session chain detail may confuse external readers.** The 5-session chain (Claude Code → Claude Web → Claude Code → Claude Web → Claude Web + QA) is accurate and interesting for investors evaluating the AI-assisted development model, but may confuse readers who aren't familiar with the project structure. A single-sentence summary ("The refactoring was completed across 5 AI agent sessions, with each session producing handoff documents that the next session read to resume work") would work better for general audiences.

### Recommended Use Cases

This document is appropriate as:
- A technical blog post for the sgraph.ai engineering blog (with minor editing for audience)
- Background material for investors evaluating the development methodology
- Reference material for new team members (human or AI) onboarding to the codebase
