## 1. Executive Summary

The 25-26 February 2026 brief batch represents the largest single expansion of SGraph Send's capability surface since the project began. Twelve development briefs and one MCP debrief collectively define **17 new components** (C29-C45), spanning four distinct capability domains:

| Domain | Components | Briefs | Significance |
|--------|-----------|--------|--------------|
| **Ephemeral & Data Room** | C29-C34 | One-Time Secret, Symmetric Key Data Room, Data Room Preview, Versioned Editing | Transforms SG/Send from single-file transfer into a collaborative workspace platform |
| **Tools & Transparency** | C35-C37 | Password Generator, Browser Storage Viewer, PGP Keygen | Creates a standalone utility tool ecosystem that drives vault adoption and DRU |
| **PKI, URL & Infrastructure** | C38-C43 | PKI URL Structure, Key Portability, Minimal Viewers, URL Architecture, S3/CloudFront Hosting, Lambda Dependency Optimization | Establishes the architectural foundations for multi-locale SPA delivery and PKI adoption |
| **Outreach & Security Operations** | C44-C45 | Waiting List Outreach / Agent Email Workflow, Per-File Security Tree, Client-Side Encrypted Logging | Operationalises the MCP pipeline for business use and creates a systematic security review framework |

The scale is significant: the M-008 evolution map (previous document) added 7 components (C22-C28) for the MCP pipeline. This brief batch adds 17 more -- more than doubling the mapped component count in a single planning cycle. The system is transitioning from a focused file-transfer tool to a **zero-knowledge encrypted collaboration platform** with utility tools, data rooms, PKI infrastructure, and agent-operated workflows.

### Wardley Maps Observation

Most of these 17 components enter at **Genesis** or early **Custom-Built**. This is expected: the brief batch is Explorer output -- defining new capabilities for the first time. The Villager team will inherit these components only after they reach stable Custom-Built. The volume of Genesis-stage components signals a deliberate investment in breadth of capability before depth of maturity.

## 3. Evolution Stage Assessment

### C29 -- One-Time Secret Service (Symmetric Mode): Genesis

**Why Genesis:** This is a new capability that does not exist in the codebase today. The brief defines the full workflow (sender creates, recipient views, data self-destructs) but no implementation exists. The concept is well-understood (competitors like OneTimeSecret exist, making the pattern proven in the market), but the SG/Send-specific implementation -- including integration with the zero-knowledge architecture, the metadata leak protections (no user-chosen URL names), and the ephemerality spectrum (one-view, N-views, time-bounded, session-bounded) -- is novel and untested. The brief explicitly flags open design decisions around passphrase protection and PKI mode interaction.

**Key uncertainty:** The ephemerality enforcement mechanism (server-side deletion after view count or expiry) needs to be proven reliable. Race conditions on concurrent access to a one-view secret are a known hard problem.

### C30 -- Ephemeral Data Lifecycle: Genesis

**Why Genesis:** The lifecycle management for ephemeral data -- view counting, time-based expiry, server-side deletion of encrypted blobs -- is a new server-side capability. While individual mechanisms (TTL, counters) are well-understood, combining them with the zero-knowledge storage model (where the server cannot inspect what it is deleting) creates novel requirements. The brief defines five ephemerality levels; no implementation exists for any of them.

**Key uncertainty:** How does the server reliably delete expired data at scale? Does it scan periodically, or does each access check expiry? The brief does not specify the garbage collection mechanism.

### C31 -- Symmetric Key Data Room: Genesis

**Why Genesis:** This is the first data room type for SG/Send. The brief defines a complete architecture -- master key encrypted by per-user short keys, admin-managed key lifecycle, role-based permissions (viewer/editor/admin), "Login as User" impersonation. None of this is implemented. The security model is explicitly acknowledged as trade-off-driven (admin sees everything, usernames are labels not cryptographic identities). The key wrapping architecture (master key encrypted N times, once per user key) is a well-known cryptographic pattern but has not been built in this system.

**Key uncertainty:** The brief distinguishes "simple revocation" (delete mapping) from "full revocation" (re-encrypt everything). Only simple revocation is in scope for MVP, but the architecture must not preclude full revocation later.

### C32 -- Data Room Preview Component: Genesis --> Custom-Built (early)

**Why Genesis --> Custom-Built (early):** The user-side preview component already exists and supports PDFs, HTML, text, and images. The work here is extraction and repackaging -- making it importable from multiple contexts (receiver view, data room, admin, portable viewer). The component itself has proven patterns, but the cross-context deployment (via `static.send.sgraph.ai` and a global compatible-versions manifest) is new. The brief introduces an architectural pattern (shared components served from a single origin, version-pinned via manifest) that is itself a new capability.

**Key uncertainty:** The component versions manifest and the cross-origin import pattern have not been tested. Will CORS and CSP headers allow the admin side to load JS from the user-side static origin?

### C33 -- Data Room Versioned Editing: Genesis

**Why Genesis:** Immutable versioning for data room files is a fundamental design decision: edits create new versions, originals are preserved, versions are linked in a chain. This is inspired by Git but implemented in the browser with encrypted storage. The brief defines the version linking schema, the "Save as New Version" UX pattern, and the conflict model (two editors of v1 produce v2a and v2b). No implementation exists. The editor component must be extracted from the existing admin side and adapted for version-creating saves.

**Key uncertainty:** Version chain integrity -- how are version links stored? Are they in the encrypted payload (invisible to server) or in server-side metadata? The brief shows a JSON structure with `parent` links but does not specify storage location.

### C34 -- Client-Side Encrypted Logging: Genesis

**Why Genesis:** This is a genuinely novel concept: the client creates signed, encrypted action logs and submits them to the server. The server stores blobs it cannot read. The security team decrypts them for audit purposes. The brief draws an analogy to CSP violation reports but acknowledges this is untested. The cross-stream consistency analysis (client logs vs. server-side opaque logs) is a research-level idea. No implementation exists. The security team's public key must be embedded at deploy time and verifiable via a transparency panel.

**Key uncertainty:** The entire trust model is theoretical. Can fabricated client-side logs be reliably detected? The brief argues yes (via graph-level consistency analysis) but this has not been proven. Additionally, the PKI infrastructure required to sign and encrypt these logs is itself at Genesis stage.

### C35 -- Password Generator Tool: Genesis

**Why Genesis:** A standalone browser-based password generator with vault integration. The generation itself is straightforward (Web Crypto API, rejection sampling, EFF wordlist for passphrases), but the vault integration -- saving generated passwords as structured encrypted entries with metadata (label, username, URL, generator settings) -- is new. The brief positions this as a vault adoption driver and DRU mechanism. No implementation exists.

**Key uncertainty:** Low. Password generation is a well-understood problem. The main risk is in the vault entry schema -- it must be compatible with future import from 1Password, Bitwarden, and browser CSV exports. The brief flags this explicitly.

### C36 -- Browser Storage Viewer Tool: Genesis

**Why Genesis:** A transparency tool that displays everything SG/Send stores in the browser (cookies, localStorage, sessionStorage, IndexedDB, Service Workers). The concept is simple but the implementation requires enumerating all browser storage APIs, handling private key material safely (fingerprint only, export with confirmation), and presenting a clear summary. No implementation exists.

**Key uncertainty:** Low. The APIs are well-documented. The main challenge is UX design for presenting technical storage data to non-technical users.

### C37 -- PGP Key Generator Tool: Genesis

**Why Genesis:** Same pattern as the existing SSH keygen tool, but for PGP keys using OpenPGP.js. Entirely client-side. The brief defines it as Part 4 of the URL architecture brief. No implementation exists, but the pattern is established (SSH keygen already exists as a reference implementation).

**Key uncertainty:** Low. OpenPGP.js is a mature library. The main question is key algorithm selection (Ed25519 preferred) and whether the generated keys integrate seamlessly with GitHub's key registry (referenced in other briefs).

### C38 -- PKI URL Structure: Genesis

**Why Genesis:** A fundamental URL redesign for PKI-encrypted delivery: `/d/{key_prefix}/{doc_name}`. The key insight is that in PKI mode, the URL is not a secret -- it is a pointer. Security comes from encryption, not link secrecy. This enables human-friendly URLs (readable, memorable, even printable on business cards). The key prefix in the URL tells the browser which private key to use for decryption. No implementation exists.

**Key uncertainty:** The key prefix collision problem -- what if two keys share the same prefix? The brief mentions "first characters of the public key fingerprint" but does not specify how many characters or how collisions are handled.

### C39 -- Key Portability (Chrome Sync / Password Manager): Genesis (Research)

**Why Genesis (Research):** The brief explicitly frames this as a research task. Seven research questions are listed. The proposed approach -- storing private keys as "passwords" in Chrome's credential manager for sync across devices -- is speculative. The Credential Management API's capabilities for this use case have not been verified. The fallback (dedicated password manager integration) is acknowledged.

**Key uncertainty:** High. This is the most uncertain component in the batch. The approach may not work due to browser security restrictions, API limitations, or platform differences (Chrome vs. Safari vs. Firefox). The brief anticipates this and provides a fallback.

### C40 -- Minimal Viewers: Genesis

**Why Genesis:** Content-type-specific viewer templates that strip away all UI chrome to present content as beautifully as possible. Three modes defined: secret viewer (centred text), document viewer (full-screen PDF/image/markdown), data room viewer (folder tree + preview). The principle is established (remove noise, let content be the star) but no minimal viewer implementations exist.

**Key uncertainty:** Design execution. The concept is clear but achieving "screenshot-worthy" minimal design requires iteration. The brief explicitly references removing things rather than adding them.

### C41 -- URL Architecture (Locale + SPA): Genesis

**Why Genesis:** A comprehensive URL restructuring: `/{lang}-{country}/{area}/{resource}`. The brief defines locale detection, functional areas (/send/, /room/, /d/, /s/, /tools/, /vault/, /admin/, /docs/), SPA vs. server-side routing classification, and redirect patterns. This is architectural planning, not implementation. No URL restructuring has been done yet.

**Key uncertainty:** The locale detection and redirect logic needs careful implementation to avoid infinite redirect loops. The SPA fallback pattern (all non-API routes serve index.html for client routing) must not break direct links to static assets.

### C42 -- S3/CloudFront Static Hosting: Genesis

**Why Genesis:** Moving static assets from Lambda/FastAPI serving to S3/CloudFront. The brief defines the CloudFront behaviour rules (API routes to Lambda, everything else to S3), versioned IFD folder deployment, and rollback capability. This is a well-understood infrastructure pattern (Product/Commodity in the general market) but has not been set up for SG/Send. The SG/Send-specific configuration (multi-site hosting, version switching via localStorage for developers) makes it Custom-Built work within a Commodity technology.

**Key uncertainty:** Low for the technology itself. The risk is in the migration -- ensuring all existing routes continue to work during and after the cutover. The CloudFront behaviour ordering (most specific first) must be correct.

### C43 -- Lambda Dependency Optimization: Genesis

**Why Genesis:** A three-phase optimization plan: (1) consolidate to one zip per Lambda, (2) strip non-essential files, (3) tree-shaking (Minimum Viable Code). The brief reports current state (~19MB of dependency zips with massive duplication) and targets (<3MB, aspirationally <1MB). Phase 1 is straightforward (deduplicated single zip). Phases 2-3 require careful testing to avoid breaking imports. No optimization work has been done yet.

**Key uncertainty:** Phase 3 (tree-shaking) is the hardest. Python's dynamic import system means static analysis cannot guarantee completeness. The brief acknowledges this and proposes an iterative approach.

### C44 -- Waiting List Outreach + Agent Email Workflow: Genesis

**Why Genesis:** This combines two capabilities: (1) a personalised multi-voice email template (human + Sherpa + Ambassador + Librarian sections), and (2) the infrastructure to send those emails (AWS SES for sending, with explicit prohibition on agent inbox access). The MCP pipeline (C24-C28) provides the agent orchestration; this component uses it for a specific business operation. The n8n/WorkMail debrief confirms the send capability works, but the outreach workflow (template generation, personalisation, batch sending, tracking) is new.

**Key uncertainty:** The agent email workflow boundary -- agents can compose and send but explicitly cannot read the human's inbox. This security boundary must be enforced at the infrastructure level, not just by convention.

### C45 -- Per-File Security Tree: Genesis

**Why Genesis:** A bottom-up security review framework: for every file in the codebase, create a parallel `.security.json` analysis file documenting attack surface, vulnerabilities, secrets exposure, dependencies, and dead code. The brief also defines trail-based threat modelling (Sherpa maps user journeys, each journey's files are threat-modelled). No security analysis tree exists yet. This is a process/methodology component, not a code component.

**Key uncertainty:** Coverage and maintenance. Creating security analysis for every file is labour-intensive. The challenge is sustaining the practice as the codebase grows -- the security tree must be updated when code changes, or it becomes stale and misleading.

## 8. Observations and Recommendations

### 8.1 The Platform Inflection Point

This brief batch marks SG/Send's transition from a **single-purpose tool** (encrypted file transfer) to a **platform** with multiple product surfaces:

1. **File transfer** (existing core)
2. **Data rooms** (C31 + C32 + C33)
3. **Ephemeral secrets** (C29 + C30)
4. **Utility tools** (C35 + C36 + C37)
5. **Agent email operations** (C44)

The Wardley Maps implication: the system is deliberately investing in **breadth at Genesis** to discover which capabilities are most valuable before committing to hardening any of them. This is textbook Explorer behaviour.

### 8.2 Infrastructure Components Are the Critical Path

The most impactful components to build first are not the user-facing features but the infrastructure that enables them:

- **C42 (S3/CloudFront)** removes Lambda invocations for static content and enables tool hosting
- **C41 (URL Architecture)** provides the routing framework every new feature needs
- **C43 (Lambda Deps)** reduces cold start times, improving UX for everything

Without these three, every new feature will be deployed through the current Lambda-serves-everything pattern, which is acknowledged as expensive, slow, and wasteful.

### 8.3 The Data Room Is the Most Complex New Capability

C31 (Symmetric Key Data Room) has the deepest dependency chain and the most moving parts: key wrapping, role-based access, admin workflows, user management, key sharing (via C29), file preview (via C32), versioned editing (via C33). It represents 5-7 sessions of work for the core, plus dependencies. This should not be underestimated.

### 8.4 The Security Components (C34, C45) Are Strategically Important

Client-Side Encrypted Logging (C34) and Per-File Security Tree (C45) are not user-facing features, but they address the human's explicitly stated concern: "we've accumulated vulnerabilities." These components enable the project to scale securely. They should be worked on in parallel with feature development, not deferred until "after launch."

### 8.5 Key Portability (C39) Is the Highest-Risk Component

This is the only component in the batch that is explicitly at a "research" sub-stage. The Chrome Credential Management API approach may not work. The brief wisely provides a fallback (password manager integration), but the Architect should plan for both paths and not assume the primary approach will succeed.

### 8.6 Town Planner Signal

This brief batch creates multiple investor-relevant narratives:

- **"Zero-knowledge data rooms"** -- a category-defining term if the data room works as designed
- **"Password manager with zero-knowledge guarantees"** -- directly comparable to LastPass/1Password with an architecturally superior security model
- **"AI agent-operated secure email"** -- the MCP pipeline operationalised for business workflows
- **"Utility tools as acquisition channels"** -- password generator, SSH keygen, PGP keygen, storage viewer as DRU drivers

The Town Planner should be briefed on this expansion. The investable narrative has grown significantly.
