## 7. Impact on Original Map's Recommendations

### 7.1 Critical Path (Revised)

The original map's critical path recommendation remains valid:

1. **C42 (S3/CloudFront Hosting)** -- still blocks C41 and tool hosting
2. **C41 (URL Architecture)** -- still blocks new URL patterns
3. **C29 (One-Time Secret)** -- still a core new feature

However, the revised map reveals that the **existing Data Room + Vault + PKI infrastructure is more mature than the original map suggested**. This means:

- C31 (Symmetric Key Data Room) can leverage the **existing** room system (35 tests) rather than building from scratch. The work is adding the key wrapping layer on top of working infrastructure.
- C32 (Data Room Preview) should be re-estimated downward since the download page preview code exists as a reference, even though extraction is new work.
- The Vault system (71 tests) is production-ready infrastructure that C35 (Password Generator) can directly integrate with.

### 7.2 Session Estimates (Revised)

The original estimates of 44-74 total sessions remain reasonable. The corrections do not significantly change the total because:
- Under-credited existing systems (rooms, vault, PKI) reduce effort for components that depend on them
- Over-credited C32 may need slightly more effort than "Genesis-->Custom-Built early" suggested
- The key observation is that the **foundation is stronger than the original map showed**

### 7.3 The Vault Omission

The most significant finding is that the **Vault system was entirely absent from the evolution map** despite being one of the most feature-rich subsystems (71 tests across 5 test files: vault service, vault cache client, vault ACL, vault multi-user, vault ACL routes). The Vault supports:
- Create vault, folder structure, file operations
- File chunking and assembly (large files)
- Encrypted index
- ACL with owner/editor/viewer hierarchy
- Multi-user sharing scenarios
- 9 admin route tests for share/unshare/permissions

This is a **Custom-Built** component that should be prominently mapped. It is a core building block for C31 (Symmetric Key Data Room) and C35 (Password Generator vault integration).

## 9. Cartographer Notes

### 9.1 Lessons for Future Maps

1. **Always check the reality document first.** The original map was created from briefs, which describe intentions, not implementations. Cross-referencing with `what-exists-today.md` would have caught the omissions.

2. **Map what exists, then map what's proposed.** The original map focused on the new brief batch (correct for its scope) but inherited existing component positions from the M-008 map without re-verifying them against code.

3. **The Vault, Data Room, PKI, Users, and Audit systems are a strong foundation.** These are implemented, tested, and working -- they should be featured prominently in any system-level Wardley Map.

4. **"Sherpa Agent" is an external dependency, not an internal component.** Mapping it at Custom-Built implies engineering effort we control. It should be mapped as an external service (like AWS WorkMail) or at Genesis with an "external" annotation.

5. **OpenPGP.js should not appear until consumed.** Listing unused libraries in the map creates false impressions of capability.
