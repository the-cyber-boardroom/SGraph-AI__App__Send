## Executive Summary

On 26 February 2026, the SGraph Send system achieved a major evolutionary milestone: the first AI-generated document was encrypted, uploaded, delivered by email, and successfully decrypted -- all orchestrated by AI agents through MCP connectors. This proves that SGraph Send is no longer a browser-only tool. It is now a programmable, AI-accessible zero-knowledge delivery system.

This evolution map documents:
1. Updated component evolution positions after the milestone
2. Full dependency map of the proven MCP pipeline
3. Evolution movements -- what moved and why
4. What must happen next for these components to reach Product stage

The significance: **the server never saw the plaintext, the server never saw the filename, and the server never saw the decryption key -- even when the sender was an AI agent, not a human.** The zero-knowledge guarantee held across a fundamentally new sender type.

### 3.2 Why Each Component Moved

**MCP Upload Route: Genesis --> Custom-Built**
- Before: The upload route existed but had only been tested with browser clients. MCP connector usage was theoretical.
- After: The route has been proven with two fundamentally different client types (browser and AI/MCP). The `data` parameter was added and immediately validated. When a component works across multiple client types, it has moved beyond "first experiment" into "repeatable pattern."
- Evidence: Successful upload from Claude.ai via MCP connector. Ciphertext stored correctly. Download link returned correctly.

**SGMETA Envelope: Genesis --> Custom-Built**
- Before: SGMETA was implemented (code existed for wrapping filenames into the encrypted payload) but had not been proven through the full encrypt-upload-download-decrypt cycle in production.
- After: Full round-trip proven. Filename was wrapped by the AI sender, survived encryption, survived upload/storage, survived download, and was correctly unwrapped by the recipient's browser. The envelope format works.
- Evidence: Recipient downloaded file with original filename preserved.

**Sherpa Agent: Concept --> Genesis/Custom-Built (early)**
- Before: "AI-as-sender" was a concept discussed in briefs. No implementation existed. The idea that an AI agent could participate in the zero-knowledge system was unproven.
- After: Working prototype. Claude.ai successfully generated a document, encrypted it, uploaded it, and triggered email delivery. The zero-knowledge guarantee held. However, this required manual orchestration -- the agent needed explicit instructions at each step.
- Reason for "early Custom-Built" rather than full Custom-Built: The pipeline worked, but it is not yet self-contained. The agent required step-by-step guidance. A mature Custom-Built implementation would be a single instruction ("send this document to X") with the agent handling all steps autonomously.

**n8n Email Integration: New at Genesis**
- Before: Not mapped. Email delivery was not part of the system.
- After: First successful execution. One n8n workflow received a trigger via MCP connector and successfully delivered an email via AWS WorkMail. However: no error handling, no retry logic, no email templating, no delivery confirmation, no rate limiting.
- Reason for Genesis: Single successful execution does not establish a pattern. The component needs to work repeatedly, handle failures, and have observable delivery status before it can move rightward.

**`data` Parameter: Born at Custom-Built**
- Before: Did not exist. MCP uploads failed because the connector could not use multipart form encoding reliably.
- After: Added to the upload route and immediately proven in production. A simple but critical fix that made AI-client uploads reliable.
- Reason for "born at Custom-Built": This is not an experiment -- it is a targeted fix to an identified problem, immediately validated. It was Custom-Built from the moment it was deployed.

**Browser Decryption: Stationary at Custom-Built**
- No movement. The existing decryption code handled the AI-generated, SGMETA-wrapped payload without any modifications. This is a positive signal: the decryption layer is client-agnostic. It does not care who the sender was.

## 6. Key Observations

### 6.1 The Zero-Knowledge Guarantee Scales to AI Senders

The most important observation from this milestone is architectural, not functional. The system's zero-knowledge design -- where the server never sees plaintext, filenames, or keys -- was validated against a new category of sender. The AI agent followed the same encryption-before-upload pattern that browsers use. No server-side changes were needed to accommodate AI senders. This means the zero-knowledge architecture is sender-agnostic by design, not by accident.

### 6.2 MCP Connectors Are the New Integration Surface

The browser was the only client until today. Now MCP connectors are a second integration surface. This has implications:
- **API design** must account for non-browser callers (no cookies, no DOM, no multipart forms)
- **Error messages** must be machine-parseable (AI agents need structured errors, not HTML error pages)
- **Documentation** must include MCP tool schemas alongside browser API docs
- **Testing** must include MCP-simulated calls alongside browser tests

### 6.3 The Email Delivery Path Creates a New Trust Surface

The n8n/WorkMail pathway is the first time a decryption key travels through an external system. In the browser flow, the key stays in the URL fragment (never sent to server) or is communicated out-of-band. In the email flow, the key is in the email body, which means:
- AWS WorkMail infrastructure sees the key (encrypted in transit, but visible to AWS)
- The email is stored in the recipient's mailbox (key persists)
- This is a deliberate, documented trust extension -- not a violation of zero-knowledge (the *SGraph Send server* still never sees the key), but it expands the trust perimeter

**Recommendation for AppSec:** Document the email delivery trust model explicitly. Consider a split-delivery model in future (link in one channel, key in another).

### 6.4 The `data` Parameter Is Architecturally Significant

Adding a `data` parameter to the upload route seems small, but it represents a pattern shift: **the API now has two upload modes** (multipart form for browsers, direct data for MCP/programmatic callers). This dual-mode API pattern will likely need to be formalised as more MCP tools are added. The Architect should document the two modes and their contracts.

### 6.5 The Sherpa Agent Opens a New Product Category

SGraph Send was conceived as a tool for humans to send files to humans. With the Sherpa Agent proven, it is now also a tool for AI to send files to humans. This opens entirely new use cases:
- Automated report generation and secure delivery
- AI-assisted document workflows with zero-knowledge guarantees
- Programmatic secure file delivery (CI/CD pipelines, data exports)

The Town Planner should be aware: this milestone creates a new narrative for investors -- "zero-knowledge encrypted delivery, accessible to both humans and AI agents."
