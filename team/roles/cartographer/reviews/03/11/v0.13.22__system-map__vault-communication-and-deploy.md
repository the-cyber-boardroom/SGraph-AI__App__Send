## What Changed Since the Last Map

The v0.1.2 system landscape (Feb 9) described SGraph Send as a two-Lambda system where browsers upload and download encrypted files. The v0.6.32 map (Feb 26) added the MCP pipeline -- AI agents sending files via MCP connectors.

This map captures three architectural shifts that happened on March 10:

1. **The vault became a communication channel.** The vault is no longer just encrypted storage -- it is a bidirectional, asynchronous, persistent communication medium between humans and AI agents, and between AI agents and other AI agents.

2. **The participant model expanded.** The system now has four participant types: browsers, CLI (`sg-send-cli`), AI agents (Claude sessions using CLI or MCP), and programmatic clients (CI/CD, deploy service).

3. **A deploy service was designed.** An ephemeral EC2-based control plane that uses the vault for state storage and SSH for infrastructure management.

## 2. Vault as Communication Channel

This is the central architectural insight from March 10. The vault was designed as encrypted storage. It turns out to also be a communication protocol.

### 2.1 Communication Topology

```
VAULT COMMUNICATION TOPOLOGY
=============================

PATTERN 1: Human <-> Vault <-> Human
(Original design -- file sharing)

  Human A                                              Human B
  +--------+    push     +-----------+    pull     +--------+
  | writes |------------>|           |------------>| reads  |
  | files  |             |   VAULT   |             | files  |
  | pulls  |<------------|           |<------------| writes |
  +--------+    pull     +-----------+    push     +--------+
                         (encrypted,
                          versioned,
                          persistent)


PATTERN 2: Human <-> Vault <-> AI Agent
(Proven on March 10 -- round-trip confirmed)

  Human                                              Claude Session
  +--------+    push     +-----------+    pull     +--------+
  | writes |------------>|           |------------>| reads  |
  | task   |             |   VAULT   |             | task   |
  | or     |             |           |             | +      |
  | message|             |           |             | reasons|
  | pulls  |<------------|           |<------------| writes |
  | result |    pull     |           |    push     | result |
  +--------+             +-----------+             +--------+

  Proven: Human wrote "experiments/it-works.md" -> Claude pulled,
  read it, wrote SKILL.md, pushed -> Human pulled SKILL.md.


PATTERN 3: AI Agent <-> Vault <-> AI Agent
(Proven on March 10 -- self-bootstrapping confirmed)

  Claude Session A                                    Claude Session B
  +------------+  push   +-----------+   clone   +------------+
  | writes     |-------->|           |---------->| reads      |
  | SKILL.md   |         |   VAULT   |           | SKILL.md   |
  | + context  |         |           |           | bootstraps |
  | + results  |         |           |           | itself     |
  |            |         |           |---------->| operates   |
  |            |         |           |   push    | writes     |
  |            |         |           |<----------| own output |
  +------------+         +-----------+           +------------+

  Proven: Session A wrote SKILL.md to vault.
  Session B (no prior context) cloned vault, read SKILL.md,
  executed full workflow autonomously, pushed results.


PATTERN 4: Deploy Service <-> Vault (PROPOSED)
(State storage for ephemeral infrastructure)

  Deploy EC2                                          Vault
  +------------+  push   +-----------+
  | stores     |-------->|           |
  | SSH keys   |         |   VAULT   |
  | AMI IDs    |         |           |
  | config     |         |           |
  |            |<--------|           |
  | reads on   |  pull   |           |
  | boot       |         |           |
  +------------+         +-----------+

  PROPOSED: Deploy Service stores ephemeral secrets
  (SSH keys, IAM tokens, appliance config) in a vault.
  On boot, it clones the vault to recover state.
```

### 2.2 Vault Protocol Properties

| Property | Description | Status |
|---|---|---|
| Bidirectional | Both parties can read and write | EXISTS |
| Asynchronous | Parties do not need to be online simultaneously | EXISTS |
| Persistent | Vault survives session boundaries | EXISTS |
| Versioned | Every push creates a content-addressed commit | EXISTS |
| Encrypted | AES-256-GCM, zero-knowledge, client-side | EXISTS |
| Identity-agnostic | Server does not know who is reading/writing | EXISTS |
| Self-describing | Vault can contain its own instructions (SKILL.md) | EXISTS |
| Access-controlled | Read = vault key only; Write = vault key + token | EXISTS |

### 2.3 What the Server Sees

```
SERVER VIEW OF VAULT COMMUNICATION
====================================

What the server stores:           What the server knows:

  vault_id/                         - vault_id exists
    tree_file_id/payload            - N encrypted blobs exist
    settings_file_id/payload        - blob sizes
    {file_id_1}/payload             - timestamps of writes
    {file_id_2}/payload             - IP hashes of writers
    ...                             - NOTHING ELSE

What the server does NOT know:    Who is communicating:

  - File names                      The server cannot distinguish
  - File contents                   between a human, a CLI user,
  - Directory structure             an AI agent, or a CI pipeline.
  - Who is communicating            All it sees is encrypted blob
  - What language they use          writes and reads.
  - Whether it's Human<->AI
    or AI<->AI
```

## 8. Trust Boundary Update

The vault round-trip introduces a new trust boundary consideration: AI agents as vault participants.

```
TRUST BOUNDARIES (v0.13.22)
=============================

+=====================================================================+
| TRUST BOUNDARY 1: Human's Machine                                    |
|                                                                      |
|  Has: vault key, access token, plaintext files, CLI                  |
|  Risk: standard endpoint security (disk encryption, OS auth)         |
+=====================================================================+

+=====================================================================+
| TRUST BOUNDARY 2: Claude Sandbox (NEW)                               |
|                                                                      |
|  Has: vault key (provided by human), access token (provided by       |
|       human), plaintext files (during session only)                  |
|                                                                      |
|  Properties:                                                         |
|  - Ephemeral: destroyed at session end                               |
|  - Sandboxed: limited network egress (allowlisted domains only)      |
|  - No persistent storage: vault key and token exist only in session  |
|  - Cannot exfiltrate: network restrictions prevent data leakage      |
|                                                                      |
|  Risk: token/key visible in conversation history (claude.ai stores   |
|         the chat). Mitigated by: token scoping, token expiry,        |
|         vault key rotation.                                          |
|                                                                      |
|  NEW CONSIDERATION: SKILL.md in vault teaches future sessions how    |
|  to operate. If a vault key is shared with an untrusted session,     |
|  that session gains full read access. Write requires a separate      |
|  token -- this is the security gate.                                 |
+=====================================================================+

+=====================================================================+
| TRUST BOUNDARY 3: SG/Send Server (unchanged)                         |
|                                                                      |
|  Has: encrypted ciphertext ONLY                                      |
|  Cannot distinguish: human vs AI vs CI/CD participants               |
|  Zero-knowledge guarantee: HOLDS for all participant types           |
```

## 9. Key Observations

### 9.1 The Vault Is Now Infrastructure, Not Just a Feature

In the v0.1.2 map, the vault was one of many features. After March 10, the vault is the foundational communication primitive that connects all participants. File transfers are point-to-point delivery. Vaults are persistent, versioned, multi-party communication channels. The deploy service proposal reinforces this: even infrastructure state flows through the vault.

### 9.2 The Participant Model Is More Important Than the Feature List

Previous maps focused on features (upload, download, tokens, rooms). The March 10 developments show that the system's value is increasingly determined by _who can participate_ and _how they communicate_. The four-participant model (browser, CLI, AI agent, programmatic client) is now the primary axis of capability expansion.

### 9.3 sg-send-cli Is the Bridge

The CLI makes every participant type possible beyond the browser. AI agents use it. The deploy service would use it. CI/CD pipelines can use it. The CLI's capability surface directly determines what non-browser participants can do.

### 9.4 Self-Bootstrapping Is a New Category

SKILL.md is not just documentation -- it is executable knowledge that transfers between AI sessions via the vault. This creates a feedback loop: AI sessions can improve the vault's instructions for future AI sessions. No other component in the system has this property.

### 9.5 The Deploy Service Is a Vault Consumer

The deploy service architecture positions SG/Send's vault as the state store for ephemeral infrastructure. This is significant because it means SG/Send's zero-knowledge guarantee applies to infrastructure secrets -- SSH keys and IAM credentials would be encrypted client-side, and the SG/Send server would never see them.
