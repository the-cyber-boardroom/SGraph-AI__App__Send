# GRC Review: Workspace Document Viewer Sandbox Change

**Version:** v0.11.21
**Date:** 2026-03-06
**Role:** GRC (Governance, Risk, Compliance)
**Status:** CONDITIONALLY ACCEPTED

---

## 1. Change Summary

**File:** `sgraph_ai_app_send__ui__workspace/v0/v0.1/v0.1.0/components/document-viewer/document-viewer.js` (line 543)

**Before:**
```html
<iframe class="dv-iframe" src="${blobUrl}" sandbox="allow-same-origin" title="HTML preview">
```

**After:**
```html
<iframe class="dv-iframe" src="${blobUrl}" sandbox="allow-scripts" title="HTML preview">
```

**Rationale:** User-loaded HTML files containing `<script>` tags were producing console errors when previewed, because `allow-same-origin` blocks script execution. The change enables script execution inside the sandboxed iframe while removing same-origin access.

**Deployment context:** The workspace is live at `dev.workspace.sgraph.ai` with real users. This is the first real-user deployment.

---

## 2. Technical Analysis

### 2.1 What Can Users Load Into the Workspace?

The document-viewer renders files that users have:
- Uploaded to their encrypted vault (decrypted client-side)
- Received as LLM transformation output
- Edited in the workspace's built-in editor

Content sources are **user-controlled**. Users load their own HTML files from their own vault. There is no mechanism for a third party to inject content into another user's document-viewer, because vault contents are end-to-end encrypted.

### 2.2 What Can Scripts Do Inside `sandbox="allow-scripts"` (Without `allow-same-origin`)?

**PERMITTED:**
- Execute JavaScript within the iframe's own DOM
- Manipulate the iframe's own document (read/write elements, styles)
- Use `setTimeout`, `setInterval`, `requestAnimationFrame`
- Create and manipulate in-memory objects (arrays, maps, etc.)
- Use `postMessage()` to send messages to the parent window
- Access Web APIs that do not require same-origin (e.g., `crypto.getRandomValues()`)
- Make network requests to external URLs (fetch/XHR) -- **important risk factor**

**BLOCKED (because `allow-same-origin` is NOT set):**
- Access the parent page's DOM, cookies, localStorage, sessionStorage
- Access the parent page's JavaScript context or variables
- Read the vault encryption key or any workspace state
- Access `window.parent` properties (blocked by opaque origin)
- Use the Storage API (localStorage/sessionStorage)
- Use IndexedDB

**BLOCKED (because other sandbox flags are NOT set):**
- `allow-forms` -- cannot submit forms
- `allow-popups` -- cannot open new windows/tabs
- `allow-top-navigation` -- cannot navigate the parent page
- `allow-modals` -- cannot trigger `alert()`, `confirm()`, `prompt()`

### 2.3 Comparison: Before vs. After

| Capability | Before (`allow-same-origin`) | After (`allow-scripts`) |
|---|---|---|
| Script execution | BLOCKED | ALLOWED |
| Parent DOM access | ALLOWED (shared origin) | BLOCKED (opaque origin) |
| Parent cookies/storage | ALLOWED | BLOCKED |
| Network requests (fetch) | ALLOWED (shared origin) | ALLOWED (opaque origin) |
| Form submission | BLOCKED | BLOCKED |
| Popups / new windows | BLOCKED | BLOCKED |
| Top navigation | BLOCKED | BLOCKED |

**Key insight:** The previous configuration (`allow-same-origin`) was arguably **more dangerous** than the new one. With `allow-same-origin`, any future code path that accidentally added `allow-scripts` would create the well-documented `allow-scripts + allow-same-origin` sandbox escape (the script can remove its own sandbox attribute). The new configuration eliminates this class of risk entirely.

## 3. Risk Assessment

### 3.1 Risk Matrix

| Risk ID | Risk Description | Likelihood | Impact | Rating | Mitigation |
|---|---|---|---|---|---|
| WS-01 | Script in previewed HTML exfiltrates decrypted file content to external server | Low | Medium | **LOW-MEDIUM** | Sandbox opaque origin blocks access to parent DOM and vault state. Script can only access its own iframe content (which the user already has). |
| WS-02 | Script in previewed HTML makes external network requests (data exfiltration of the HTML itself) | Medium | Low | **LOW** | The content being exfiltrated is the user's own file, which they already possess. No vault keys or other user data are accessible. |
| WS-03 | Script causes resource exhaustion (infinite loop, memory bomb) | Medium | Low | **LOW** | The js-executor has a 10-second timeout. For the document-viewer preview iframe, there is no timeout -- a malicious HTML file could consume resources. Impact limited to the user's own browser tab. |
| WS-04 | `allow-scripts + allow-same-origin` combination accidentally introduced | Very Low | Critical | **LOW** | Current code explicitly uses only `allow-scripts`. Code review and AppSec checks should catch any regression. |
| WS-05 | Malicious HTML file shared via vault used as social engineering vector | Low | Medium | **LOW-MEDIUM** | Users load their own vault files. Sharing a vault with a third party's HTML is a user-initiated action. The sandbox prevents the script from accessing anything outside the iframe. |
| WS-06 | Cryptocurrency mining or compute abuse via script execution | Low | Very Low | **VERY LOW** | Limited to user's own browser tab. No persistence mechanism (no storage, no service workers). |

### 3.2 Aggregate Risk Rating: **LOW-MEDIUM**

The change **reduces the most critical risk** (same-origin access to the parent page) while introducing a **lower-severity risk** (script execution in an isolated context). The net security posture improves.

## 4. Compliance Considerations

### 4.1 Content Security Policy (CSP)

**Current state:** The document-viewer iframe does NOT have a `csp` attribute. The AppSec review from 2026-03-03 (`v0.10.32__review__march-03-briefs-security.md`) recommended adding one:

```html
<iframe sandbox="allow-scripts" csp="default-src 'none'; script-src 'unsafe-inline'; style-src 'unsafe-inline'; img-src blob: data:">
```

**Gap:** This recommendation has not been implemented. Adding a `csp` attribute would restrict what scripts inside the iframe can do -- specifically blocking `fetch()` and `XMLHttpRequest` to external origins, which is the primary residual risk (WS-02).

**Recommendation:** Implement the CSP attribute as recommended by AppSec. This is the single most impactful hardening measure remaining.

### 4.2 Iframe Sandboxing Standards

The change aligns with:
- **MDN sandbox guidance:** Using `allow-scripts` without `allow-same-origin` is the recommended pattern for untrusted content execution.
- **OWASP iframe sandboxing:** Principle of least privilege -- grant only the permissions needed.
- **AppSec team recommendation** (v0.10.32): Explicitly recommended `sandbox="allow-scripts"` without `allow-same-origin` for HTML preview.
- **Architect team recommendation** (v0.11.10): Confirmed `allow-scripts` only, no `allow-same-origin` as the correct pattern for workspace iframes.

### 4.3 Consistency With Other Components

| Component | Sandbox Value | Scripts? | Same-Origin? | Notes |
|---|---|---|---|---|
| **js-executor** (workspace) | `allow-scripts` | Yes | No | Consistent with this change |
| **send-download** (user UI) | `allow-same-origin` | No | Yes | Different use case (markdown preview, no scripts needed) |
| **vault-file-preview** (vault UI) | `allow-same-origin` | No | Yes | Different use case (file preview, no scripts needed) |
| **document-viewer** (workspace) | `allow-scripts` | Yes | No | **This change** |

The workspace components are now internally consistent: both the js-executor and document-viewer use `allow-scripts` without `allow-same-origin`.

### 4.4 Data Handling

- **No server-side impact.** This is a client-side rendering change. No data leaves the browser differently.
- **Zero-knowledge property preserved.** The server never sees plaintext. This change does not affect the encryption/decryption pipeline.
- **User data isolation maintained.** The opaque origin prevents the iframe from accessing vault keys, session tokens, or other workspace state.

## 5. Residual Risks After Change

| ID | Residual Risk | Severity | Recommended Action |
|---|---|---|---|
| RR-01 | Scripts in previewed HTML can make outbound network requests | Medium | Add `csp` attribute to iframe to block `connect-src` |
| RR-02 | No execution timeout on document-viewer preview iframe (unlike js-executor's 10s limit) | Low | Consider adding a timeout or resource limit for preview iframes |
| RR-03 | No user-visible indicator that a previewed HTML file is executing scripts | Low | Consider adding a visual badge when scripts are active in preview |

---

## 6. Recommendations

### Immediate (before next release)

1. **Add CSP attribute to the preview iframe** (addresses RR-01):
   ```html
   <iframe class="dv-iframe" src="${blobUrl}"
           sandbox="allow-scripts"
           csp="default-src 'none'; script-src 'unsafe-inline'; style-src 'unsafe-inline'; img-src blob: data:;"
           title="HTML preview">
   ```
   This blocks outbound network requests from scripts inside the preview, eliminating the primary residual risk.

### Short-term (within 2 releases)

2. **Add a "scripts active" indicator** in the document-viewer toolbar when rendering HTML files, so users are aware that the previewed file is executing code.

3. **Verify `allow-scripts` and `allow-same-origin` are never combined** -- add this as an automated check (grep in CI) to prevent regression.

### For tracking

4. **Update the AppSec threat model** to reflect this change and confirm the workspace iframe sandbox posture.

---

## 7. Risk Acceptance Decision

### CONDITIONALLY ACCEPTED

The change from `allow-same-origin` to `allow-scripts` is a **net security improvement**. It removes same-origin access (the more dangerous capability) and adds script execution in a properly isolated sandbox. This aligns with AppSec and Architect recommendations.

**Conditions for full acceptance:**

1. **CSP attribute must be added to the preview iframe** (Recommendation #1) within the next release to block outbound network requests from executed scripts. This is the only material residual risk.

**Rationale:**
- The previous configuration (`allow-same-origin`) gave the iframe access to the parent page's cookies, localStorage, and DOM -- a higher risk than script execution in an opaque origin.
- The workspace's js-executor already uses the same `sandbox="allow-scripts"` pattern, so this brings the document-viewer in line with established practice.
- Real users are actively using the workspace, and HTML preview with working scripts is a legitimate functional requirement.
- The remaining risk (outbound network from scripts) is mitigatable with a CSP attribute and affects only the user's own content.
