## Policy Violations

### VIOLATION 1: pydantic-settings (SEVERITY: MEDIUM)

**Rule:** "All schemas use Type_Safe (from osbot-utils), never Pydantic"

`pydantic-settings` is a hard transitive dependency of both `fastapi-mcp` and `mcp`. It is loaded at runtime. While our application code does not use it directly, its presence:
- Contradicts the stated architectural principle
- Normalises Pydantic in the dependency tree (Pydantic was already present via FastAPI, but `pydantic-settings` extends its footprint)
- Creates risk of developers using it in future code

**Recommendation:** Accept as transitive-only with a documented exception. Add a lint rule blocking direct imports of `pydantic_settings` in application code.

### VIOLATION 2: boto3/botocore Update (SEVERITY: LOW)

boto3/botocore patch bump (1.42.49 to 1.42.56) was a side-effect of dependency resolution, not intentional. Used correctly via `osbot-aws`. Pin if stability is critical.

## Concentration Risk & Alternatives

**Single point of failure:** Removing `fastapi-mcp` eliminates 14 of 17 new packages. One feature owns ~20% of total package count. Key risks: `fastapi-mcp` abandonment (pre-1.0, MEDIUM-HIGH), `mcp` SDK breaking changes (MEDIUM), native build failures on arm64 (MEDIUM).

| Option | New Deps | Trade-off |
|--------|----------|-----------|
| **fastapi-mcp (current)** | 17 | Full auto-generation, pre-1.0, heavy |
| **Hand-rolled MCP endpoint** | 0 | ~100-200 LOC, full control, maintenance burden |
| **MCP via separate Lambda** | 0 in main app | Operational complexity, isolates deps |

**Recommendation:** Hand-rolled or separate-service model better aligns with minimal-dependency philosophy.

## Governance Recommendations

### Immediate Actions

1. **Document the pydantic-settings exception** in project rules with scope limitation (transitive only, no direct use).
2. **Pin `fastapi-mcp` to exact version** (`0.4.0`) -- never use `*` for a pre-1.0 package with 16 transitive deps.
3. **Pin all 17 new packages** in `poetry.lock` and verify lock file is committed.
4. **Add dependency count check to CI** -- alert if `poetry install` adds more than 3 new packages.

### Short-Term (before production)

5. **Measure Lambda impact** -- cold start time and package size before/after.
6. **Create dependency governance policy** -- max transitive ratio (10:1), mandatory review for native/pre-1.0 deps, prohibited direct imports list.
7. **Evaluate isolation** -- can MCP run as a separate Lambda to contain the blast radius?

### Strategic

8. **Revisit at v0.7.0** -- if MCP usage stays limited, hand-rolled implementation eliminates 14+ deps.
9. **Add SBOM generation** to CI for ongoing supply chain visibility.
