## Milestone Summary

SGraph AI now has its own public-facing website at [static.sgraph.ai](https://static.sgraph.ai), deployed through a fully automated CI/CD pipeline built and operated entirely by AI agents. The site runs on the project's own infrastructure -- S3 for storage, CloudFront for global delivery -- with zero external hosting dependencies. Nine pages went live, including three individual agent profile pages, making SGraph one of the first projects to give its AI team members their own public identity. This is not a product launch -- SG/Send has been live at [send.sgraph.ai](https://send.sgraph.ai) since February 2026. This is an infrastructure milestone: the team now owns its web presence end-to-end, from source code to CDN edge.

---

## Key Talking Points

**1. Fully automated deployment pipeline -- brief to live in one day.**
The deployment pipeline is a 47-line GitHub Actions workflow that delegates all logic to a Python deploy script (`deploy_static_site.py`). Push to `main`, and the site deploys: S3 sync with per-file-type cache control, CloudFront invalidation, smoke test verification. Two sessions took the site from brief to production. No manual steps.

**2. Zero external dependencies -- self-hosted everything.**
Fonts (DM Sans, JetBrains Mono) are vendored as local woff2 files. No Google Fonts CDN, no external JavaScript libraries, no third-party analytics. The HTML, CSS, and JS are vanilla -- zero framework dependencies. The site loads from SGraph's own CloudFront distribution. If every third-party service went offline tomorrow, the site would still work.

**3. IFD versioned releases -- every deployment is immutable and traceable.**
Each release is stored at `websites/sgraph-ai/releases/v0/v0.7/v0.7.7/` (IFD nested versioning). The `latest/` path points to the current release. Previous releases are preserved indefinitely. Rolling back is a single S3 copy operation. This is the same versioning model used for the SG/Send application UI.

**4. Built entirely by AI agents -- from design to deployment.**
The website HTML, CSS, deployment scripts, CI workflow, CloudFront URL rewrite function, and cache-control strategy were all created by AI agents operating within the SGraph team structure. The human stakeholder provided the brief and made deployment decisions. The agents did the implementation.

**5. Nine pages including agent profile pages.**
The site launched with: homepage, product page, agents directory, architecture overview, contact page, 404 page, and three individual agent profiles (Sherpa, Ambassador, Architect). The agent profiles are a deliberate choice -- SGraph's AI team members are part of the project's public identity, not hidden behind a generic "AI-powered" label.

## What NOT to Say

These are accuracy guardrails. Do not use the following in any external communication:

| Do NOT say | Why |
|---|---|
| "Self-updating website" | The site is static HTML. Updates require a new deployment through CI/CD. It does not update itself. |
| "Auto-scaling infrastructure" | S3 + CloudFront scales inherently, but calling it "auto-scaling" implies dynamic compute. There is no compute. It is static files on a CDN. |
| "AI-generated content that updates in real-time" | The content was generated by AI agents, but it is static once deployed. There is no real-time generation. |
| "The SGraph AI website launched" | This phrasing implies the company website launched. Be specific: the marketing/information site at static.sgraph.ai launched. The product (SG/Send) was already live. |
| "Fully autonomous deployment" | The human approved the deployment and made configuration decisions (bucket names, CloudFront distribution IDs, secrets). The pipeline is automated; the decisions were human. |
| "Nine AI agents have profile pages" | Three agents have profile pages (Sherpa, Ambassador, Architect). The agents directory page lists more, but individual profile pages exist for three. |
| "No servers" | Technically accurate (it is serverless), but misleading. Say "static site on CDN" instead. |
