## The Headline

**SG/Send loads its entire encrypted file-sharing UI in 150 milliseconds.**

DOMContentLoaded at 112ms. Full page load at 147ms. 27 requests, 129 kB transferred. The two API calls (OWASP check at 399ms, capabilities probe at 63ms) are non-blocking — the UI is fully interactive before either returns.

For context: most modern web applications take 1-3 seconds to become interactive. SG/Send does it in the time it takes to blink.

---

## Why This Matters

### Zero-Framework Payoff

SG/Send uses vanilla JavaScript Web Components with no React, no Vue, no Svelte, no framework bundle of any kind. Every kilobyte transferred is useful code. The largest single JS file is the upload component at ~20kB.

This isn't an aesthetic choice — it's an architectural one. When your application handles encryption in the browser (AES-256-GCM via Web Crypto API), you need every millisecond of browser attention for the actual security work, not framework hydration.

### CloudFront + S3: First Production Deploy

This is the first time SG/Send's UI runs from a CloudFront CDN distribution with two origins:

- **S3** serves all static content (HTML, JS, CSS, fonts, favicons) from edge locations
- **Lambda** handles API calls (transfers, encryption handshake, tokens)

The result: zero Lambda cold-start for the UI. The page loads from the nearest CDN edge, and the API calls happen asynchronously after the user can already see and interact with the interface.

### 16 Languages, Zero Runtime Cost

SG/Send now supports 16 languages (English UK, English US, Portuguese, Spanish, French, German, Italian, Dutch, Polish, Croatian, Romanian, and even Klingon). All translations are pre-rendered at build time — switching languages navigates to a pre-built page, no JSON fetches, no runtime template processing.

The language selector now uses a two-column grid dropdown (matching sgraph.ai's design) instead of a basic `<select>` element.

## The Waterfall Story

Looking at the Chrome DevTools waterfall:

1. **0-40ms**: HTML document arrives, CSS and JS start loading in parallel
2. **40-80ms**: All Web Component scripts loaded (~30kB each chunk)
3. **80-112ms**: DOMContentLoaded — components register and render
4. **112-147ms**: Fonts finish loading (WOFF2, ~30ms each)
5. **147ms**: Complete.

The fonts are the "slowest" part. Everything else is faster than most sites' DNS lookup.
