## 1. Announcement Draft (Blog / Newsletter)

### Your large files, encrypted before they leave your browser

The number one question we heard after launching SG/Send was direct: "This is interesting, but my pain point is large files -- can you do that?"

Today the answer is yes. SG/Send now supports file transfers up to 1GB, with parallel uploads to S3 via presigned URLs. We tested it with a 42MB file: upload completed in approximately 15-20 seconds, a 3-4x improvement over the previous sequential approach. Five concurrent upload streams run in parallel, each chunk encrypted with AES-256-GCM before it leaves your browser. The server still never sees plaintext. The decryption key still never leaves the sender's device. The zero-knowledge guarantee is unchanged.

This is not a separate "large file mode." The same encryption pipeline that protects a 50KB document now protects a 500MB archive. Files are split into chunks, each chunk is encrypted independently with counter-derived initialisation vectors and authenticated additional data (AAD) that prevents chunk reordering or substitution, then uploaded in parallel to S3 via time-limited presigned URLs. The Lambda function generates the URLs and manages state -- it never touches the file data. When the recipient downloads, the process reverses: parallel download, per-chunk decryption, reassembly. All in the browser.

What changed architecturally: the upload path now bypasses the Lambda payload entirely. Instead of pushing encrypted data through the Lambda function (which has a 6MB payload limit), the browser uploads directly to S3 using presigned URLs that the Lambda generates. This removes the file size ceiling while reducing Lambda costs to near zero for the data transfer itself. The server's role is coordination, not transport -- exactly where it should be in a zero-knowledge system.

### What this means for the product

SG/Send started as a proof-of-concept for zero-knowledge file sharing. The first MVP handled small files. The vault added persistence and editing. PKI added identity and signing. Large file support addresses the practical gap that kept SG/Send in the "interesting demo" category. Sending a 200MB due diligence package, a 500MB design archive, or a 1GB database export -- all encrypted, all zero-knowledge, all from a browser tab. The P in MVP moves from "Prototype" to "Product."

## 5. Competitive Comparison

### The zero-knowledge + large file landscape

| Service | Max File Size | Encryption | Zero-Knowledge | Parallel Upload | Open Source | Status |
|---------|--------------|------------|:--------------:|:--------------:|:-----------:|--------|
| **SG/Send** | **1 GB** | **AES-256-GCM, client-side per-chunk** | **Yes** | **Yes (5 concurrent)** | **Yes** | **Active** |
| WeTransfer | 2 GB (free) / 200 GB (Pro) | TLS in transit, server-side at rest | No | Yes | No | Active |
| Firefox Send | 1 GB (was 2.5 GB) | AES-128-GCM, client-side | Yes | No | Yes | **Discontinued (2020)** |
| Tresorit Send | 5 GB | AES-256, client-side | Claimed* | Unknown | No | Active |
| Wormhole (Magic Wormhole) | No limit (P2P) | NaCl / SPAKE2 | Yes (P2P) | No | Yes | Active (niche) |
| OnionShare | No limit (Tor) | Client-side | Yes | No | Yes | Active (niche) |
| Bitwarden Send | 500 MB | AES-256-CBC, client-side | Yes | No | Yes | Active |

*Tresorit claims zero-knowledge but performs some server-side operations for chunked uploads.

### What SG/Send uniquely offers

**1. Zero-knowledge + large files + parallel uploads in one package.**
WeTransfer handles large files with parallel uploads but is not zero-knowledge -- the server can read every file. Firefox Send was zero-knowledge but is discontinued. Tresorit Send claims zero-knowledge but caps at 5GB with unclear chunking architecture. SG/Send is the only active, open-source service that combines all three.

**2. Per-chunk encryption with AAD.**
Each chunk is independently encrypted with a counter-derived IV and authenticated additional data that binds the chunk to its position in the file. This prevents chunk reordering, duplication, or substitution attacks. Most encrypted transfer services either encrypt the entire file as one blob (memory-limited) or encrypt server-side during chunking (breaking zero-knowledge).

**3. Lambda never touches file data.**
The server generates presigned URLs for direct browser-to-S3 upload. This is architecturally distinct from services that proxy uploads through their servers (even if they encrypt at rest afterward). The data path is: browser encrypts, browser uploads to S3, S3 stores ciphertext. The server is a coordinator, not a relay.

**4. Open source and auditable.**
The encryption code, the chunking logic, the presigned URL generation -- all visible in the public repo. "Zero-knowledge" is a claim anyone can verify by reading the code, not a promise buried in a privacy policy.

### The Firefox Send comparison

Firefox Send is the closest historical precedent: browser-side encryption, open source, backed by a major brand. Mozilla shut it down in September 2020 after abuse concerns (malware distribution via encrypted links). SG/Send shares Firefox Send's encryption model but adds:

- Per-chunk encryption (Firefox Send encrypted the entire file as one blob, limiting practical file size by browser memory)
- Parallel uploads (Firefox Send uploaded sequentially)
- Presigned URL architecture (Firefox Send proxied through its server)
- Active development and maintenance (Firefox Send is abandoned)
- Admin controls and token authentication (addressing the abuse vector that killed Firefox Send)

The lesson from Firefox Send is not that zero-knowledge file sharing is a bad idea. It is that zero-knowledge file sharing needs operational controls alongside its cryptographic guarantees. SG/Send has both.

### The WeTransfer comparison

WeTransfer is the market leader in simple file sharing. It is the benchmark for user experience. But WeTransfer is not zero-knowledge -- it encrypts files at rest on its servers using keys it controls. WeTransfer can read every file uploaded to its platform. For most users, this is acceptable. For users handling sensitive documents -- legal, financial, medical, IP -- it is not.

SG/Send does not compete with WeTransfer on file size (WeTransfer Pro supports 200GB). It competes on a different axis: trust model. "We cannot read your files" is a fundamentally different proposition from "we promise not to read your files."

### The Tresorit comparison

Tresorit is the closest active competitor in the zero-knowledge encrypted file sharing space. Tresorit Send allows sharing files up to 5GB with end-to-end encryption. However:

- Tresorit is proprietary and closed-source -- the zero-knowledge claim cannot be independently verified
- Tresorit's chunking and upload architecture is opaque
- Tresorit requires a paid plan (starts at EUR 12/month) for Send functionality
- Tresorit is a Swiss company subject to Swiss data jurisdiction -- relevant for some users, irrelevant for others

SG/Send's advantage: open source, verifiable, self-hostable, and architecturally transparent. The encryption happens in auditable JavaScript in the user's browser.

## AppSec Verification Notes

The following claims in this document require AppSec verification before external publication:

1. **"AES-256-GCM per-chunk encryption with counter-derived IVs"** -- Verify this is the actual implementation, not just the architectural design. Confirm counter-derived IVs are used (not random per-chunk).
2. **"AAD prevents chunk reordering or substitution"** -- Verify AAD includes chunk index, total chunks, and transfer ID. Confirm this is implemented, not just designed.
3. **"Lambda never touches file data"** -- Verify no code path sends file data through the Lambda function for large file uploads. Confirm presigned URLs are the only upload path for chunked transfers.
4. **"Decryption key never leaves the sender's device"** -- Verify no logging, debugging, or telemetry code captures key material during the large file flow.
5. **"5 concurrent upload streams"** -- Verify the actual parallelism level in the current implementation.
6. **"42MB in ~15-20 seconds"** -- This is a reported measurement. Note the conditions (network speed, region, etc.) before using as a marketing claim.

No security claim should be published externally without AppSec sign-off.
