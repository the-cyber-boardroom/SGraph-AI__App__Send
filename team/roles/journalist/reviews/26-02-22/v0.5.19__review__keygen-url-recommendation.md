## Executive Summary

The SSH key generator is the first developer utility in what will become a portfolio of browser-based cryptography tools. The URL it launches under will set the pattern for everything that follows. This review analyses five candidate URL structures across SEO, brand alignment, future-proofing, memorability, and technical complexity.

**Recommendation: `keys.sgraph.ai`** -- with the SSH keygen served at `keys.sgraph.ai/ssh` and future tools at `keys.sgraph.ai/pki`, `keys.sgraph.ai/gpg`, etc.

---

## Context

### What We Are Naming

A browser-based SSH key generator that runs entirely client-side using the Web Crypto API. It generates Ed25519 and RSA key pairs, outputs them in OpenSSH format, and provides copy-paste-ready instructions for GitHub, GitLab, and server SSH access. The server never sees the generated keys. It is the first of several planned developer tools (PKI certificates, potentially password generation, certificate inspection).

### The Brand Question

The SSH keygen is _not_ a file transfer tool. It generates cryptographic keys. It shares SG/Send's core technology (Web Crypto API) and core promise (zero-knowledge, nothing leaves your browser), but it does not share SG/Send's core function (sending files). This is the central tension: does the tool belong to **Send** (the product) or to **SGraph** (the platform)?

### Existing Domain Architecture

The project already uses a `{function}.send.sgraph.ai` subdomain pattern:

| Domain | Purpose |
|--------|---------|
| `send.sgraph.ai` | Production user-facing application |
| `admin.send.sgraph.ai` | Admin console |
| `team.send.sgraph.ai` | Public agent meeting room (planned) |
| `dev.send.sgraph.ai` | Development environment |
| `qa.send.sgraph.ai` | QA environment |

The parent domain `sgraph.ai` currently hosts no subdomains outside the `send` hierarchy.

## Candidate Analysis

### Option 1: `keygen.send.sgraph.ai`

_Follows the existing `{function}.send.sgraph.ai` subdomain pattern._

| Criterion | Assessment | Score |
|-----------|-----------|-------|
| **SEO value** | Poor. Four-level subdomain (`keygen.send.sgraph.ai`) is treated by search engines as deeply nested. Google devalues subdomains of subdomains. The domain authority of `send.sgraph.ai` does not flow strongly to a sub-subdomain. | 2/10 |
| **Brand alignment** | Misleading. "keygen" is not a "send" function. Placing it under `send.sgraph.ai` implies it is part of the file transfer product. Users arriving at `keygen.send.sgraph.ai` will expect to find something related to file sending. | 3/10 |
| **Future-proofing** | Poor. If PKI tools, password generators, or certificate inspectors follow, each gets its own sub-subdomain: `pki.send.sgraph.ai`, `passwords.send.sgraph.ai`. The pattern becomes unwieldy and still misattributes these tools to "Send." | 2/10 |
| **Memorability** | Poor. `keygen.send.sgraph.ai` is 23 characters. Hard to dictate verbally. Hard to remember. The `.send.` segment adds cognitive load without adding meaning for a key generation tool. | 3/10 |
| **Technical complexity** | Low. DNS CNAME to the same infrastructure. No CORS issues since it shares the `send.sgraph.ai` origin hierarchy. SSL wildcard on `*.send.sgraph.ai` covers it. | 8/10 |

**Overall: 3.6/10** -- Easy to deploy but wrong conceptually. Ties a non-Send tool to the Send product permanently.

### Option 2: `keygen.sgraph.ai`

_Elevates the tool to a top-level SGraph subdomain, not tied to Send._

| Criterion | Assessment | Score |
|-----------|-----------|-------|
| **SEO value** | Good. Two-level subdomain directly under `sgraph.ai`. Search engines treat this as a distinct site with its own authority. Clean URL for backlinks. | 7/10 |
| **Brand alignment** | Good. Positions the keygen as an SGraph tool, not a Send feature. Correctly reflects that key generation is a cryptography utility, not a file transfer function. Reinforces SGraph as a platform brand. | 7/10 |
| **Future-proofing** | Poor. "keygen" is specific to key generation. When PKI certificate tools arrive, they need a different subdomain (`pki.sgraph.ai`? `certs.sgraph.ai`?). Each new tool gets its own DNS record, SSL certificate, and deployment. The pattern does not scale. | 3/10 |
| **Memorability** | Moderate. `keygen.sgraph.ai` is 17 characters. Clear what it does. But it burns a top-level subdomain on a single tool. | 6/10 |
| **Technical complexity** | Low-moderate. New DNS record under `sgraph.ai`. Needs its own SSL certificate (or a wildcard on `*.sgraph.ai`). Separate deployment from the Send infrastructure. | 6/10 |

**Overall: 5.8/10** -- Good brand positioning but does not scale. Creates subdomain sprawl as the tool portfolio grows.

### Option 3: `tools.send.sgraph.ai/ssh-keygen`

_Umbrella "tools" subdomain under Send, with path-based tool routing._

| Criterion | Assessment | Score |
|-----------|-----------|-------|
| **SEO value** | Poor. Same sub-subdomain problem as Option 1. The `/ssh-keygen` path helps slightly (keywords in path), but the domain depth hurts. | 3/10 |
| **Brand alignment** | Slightly better than Option 1 -- "tools" is generic, so the connection to Send is looser. But it still lives under `send.sgraph.ai`, implying these are "Send's tools." | 4/10 |
| **Future-proofing** | Good structurally. New tools are just new paths: `tools.send.sgraph.ai/pki`, `tools.send.sgraph.ai/passwords`. One subdomain, one deployment, one SSL cert. But the `/send/` ancestry is still wrong for non-Send tools. | 5/10 |
| **Memorability** | Poor. `tools.send.sgraph.ai/ssh-keygen` is 33 characters. The longest option. Impossible to dictate cleanly. | 2/10 |
| **Technical complexity** | Low. Single DNS record. Path routing handled by the static site or a simple reverse proxy. | 8/10 |

**Overall: 4.4/10** -- Good scaling pattern trapped under the wrong parent domain.

### Option 4: `tools.sgraph.ai/ssh-keygen`

_Umbrella "tools" subdomain at the SGraph level, with path-based tool routing._

| Criterion | Assessment | Score |
|-----------|-----------|-------|
| **SEO value** | Good. Clean two-level subdomain. `/ssh-keygen` puts keywords in the path. `tools.sgraph.ai` can accumulate domain authority as a multi-tool site. Each tool page builds authority for the others. | 7/10 |
| **Brand alignment** | Strong. These are SGraph developer tools -- not Send features. The URL communicates "SGraph makes a suite of developer tools; here is the SSH one." Correctly positions the relationship. | 8/10 |
| **Future-proofing** | Excellent. New tools are new paths: `tools.sgraph.ai/pki`, `tools.sgraph.ai/base64`, `tools.sgraph.ai/cert-inspector`. One subdomain, one deployment, unlimited tools. A tools index page at `tools.sgraph.ai/` ties them together. | 9/10 |
| **Memorability** | Moderate. `tools.sgraph.ai/ssh-keygen` is 27 characters. Not short, but logically structured -- people can reason about the URL. The `tools.sgraph.ai` prefix is memorable once learned; individual tool paths are guessable. | 5/10 |
| **Technical complexity** | Low-moderate. One DNS record for `tools.sgraph.ai`. Static site deployment (GitHub Pages, CloudFront, or similar). Path routing is trivial for a static site. Needs its own SSL cert or wildcard on `*.sgraph.ai`. | 7/10 |

**Overall: 7.2/10** -- Strong contender. Scales well, brand-aligned, good SEO. Slightly verbose for a shareable URL.

### Option 5: `keys.sgraph.ai`

_Short, memorable subdomain scoped to all key/cryptography tools._

| Criterion | Assessment | Score |
|-----------|-----------|-------|
| **SEO value** | Excellent. Short, keyword-rich domain. "keys" is a high-value term for the target audience. Two-level subdomain with strong SEO potential. The word "keys" in the domain signals relevance to search engines for queries like "generate ssh key online," "create ssh key in browser," and "ssh key generator." Individual tools at `/ssh`, `/pki`, `/gpg` keep paths short and keyword-dense. | 9/10 |
| **Brand alignment** | Excellent. SGraph's core brand promise is zero-knowledge cryptography. Keys are the atomic unit of cryptography. `keys.sgraph.ai` says "SGraph does cryptography, and here are the tools to prove it." It reinforces the security narrative without being tied to the Send product. It also creates a natural cross-promotion path: "Your keys, generated at keys.sgraph.ai. Your files, encrypted at send.sgraph.ai." | 9/10 |
| **Future-proofing** | Very good. SSH keys at `/ssh`. PKI certificates at `/pki`. GPG keys at `/gpg`. API keys at `/api-keys`. The scope is bounded but generous -- anything related to cryptographic keys and credentials fits naturally. If a tool does not involve keys (e.g., a base64 encoder), it does not belong here -- which is actually a feature, not a bug. It enforces coherence. Non-key tools can go to `tools.sgraph.ai` later if needed. | 8/10 |
| **Memorability** | Excellent. `keys.sgraph.ai` is 14 characters. The shortest option by a significant margin. Easy to say aloud: "keys dot s-graph dot AI." Easy to type. Easy to remember. ## Recommendation: `keys.sgraph.ai`

### The URL Structure

| URL | Purpose |
|-----|---------|
| `keys.sgraph.ai` | Landing page -- "SGraph Keys: Browser-based cryptography tools" |
| `keys.sgraph.ai/ssh` | SSH key generator (the current tool) |
| `keys.sgraph.ai/pki` | PKI certificate tools (next planned tool) |
| `keys.sgraph.ai/gpg` | GPG key generation (future) |

### Why This Wins

**1. It tells the right story.** The SSH keygen is not a file-sending feature. It is a cryptography tool. Placing it under `send.sgraph.ai` misrepresents what it is. Placing it under `keys.sgraph.ai` correctly positions it: SGraph makes tools for people who care about cryptographic security. Send is one of those tools. Keys is another.

**2. It is the shortest, most shareable option.** At 14 characters for the domain and 19 characters for the full SSH tool URL, `keys.sgraph.ai/ssh` is shorter than any competitor's URL. It fits on a slide. It fits in a tweet. It can be dictated over a phone call without spelling errors.

**3. It creates a natural marketing pair.** "Generate your keys at keys.sgraph.ai. Send your files at send.sgraph.ai." Two products, one brand, one trust model. This is the strongest cross-promotion pattern available.

**4. It builds the SGraph brand, not just the Send brand.** Every backlink to `keys.sgraph.ai` builds domain authority for `sgraph.ai` as a whole. This benefits Send (at `send.sgraph.ai`) through the shared parent domain. The Ambassador's SEO research identified that the SSH key generator space is fragmented with no dominant player -- `keys.sgraph.ai` can realistically rank on the first page for "ssh key generator online" within 3-6 months. That SEO authority flows upward to the SGraph brand.

**5. It scales to the right scope.** The "keys" namespace is broad enough to accommodate SSH, PKI, GPG, and future credential tools, but narrow enough to stay coherent. It will not become a dumping ground for unrelated utilities. If non-key tools are needed later, `tools.sgraph.ai` remains available as a separate domain.

### What About `tools.sgraph.ai`?

`tools.sgraph.ai` is the runner-up and scores well on future-proofing. The case for it is: "We do not know what tools we will build, so pick the broadest namespace." The case against it is: generality is the enemy of memorability. `tools.sgraph.ai/ssh-keygen` is 27 characters. `keys.sgraph.ai/ssh` is 19. In a space where the product is trust and the audience is developers, concision signals competence.

If the tool portfolio later expands well beyond keys (base64 encoding, JSON formatting, regex testing), `tools.sgraph.ai` can be introduced as a separate umbrella that links _to_ `keys.sgraph.ai` for key-related tools. The two are not mutually exclusive.

### Implementation Notes

| Item | Detail |
|------|--------|
| **DNS** | Add CNAME record: `keys.sgraph.ai` pointing to the static site host (GitHub Pages, CloudFront, or S3 static hosting) |
| **SSL** | Either a dedicated certificate for `keys.sgraph.ai` or a wildcard `*.sgraph.ai` certificate (recommended -- enables future subdomains at zero marginal cost) |
| **CORS** | Not applicable. The page sets `connect-src 'none'` in its CSP -- no outbound requests are made. No CORS headers needed. |
| **Deployment** | Independent from Send. Static site only. No Lambda, no backend. Can deploy via GitHub Pages for simplicity. |
| **Redirect** | Set up `send.sgraph.ai/tools/ssh-keygen` as a 301 redirect to `keys.sgraph.ai/ssh` for any existing links or bookmarks |
| **Canonical** | `<link rel="canonical" href="https://keys.sgraph.ai/ssh">` on the SSH keygen page |
| **Cross-linking** | SSH keygen page includes a footer CTA: "Need to send files securely? Try [SG/Send](https://send.sgraph.ai)" |

### Risks

| Risk | Mitigation |
|------|-----------|
| `*.sgraph.ai` wildcard SSL adds cost | ACM (AWS Certificate Manager) provides free wildcard certificates. No cost impact. |
| Separate deployment pipeline from Send | This is a feature, not a risk. The keygen has a different release cadence. Decoupled deployment prevents Send releases from accidentally breaking the key tool. |
| Namespace collision if SGraph later has a product called "Keys" | Unlikely. If it happens, the keygen tools can move to a `/tools` path under the new product. The redirect pattern handles migration gracefully. |
| "keys" could be confused with API keys or license keys | The landing page at `keys.sgraph.ai` should clarify scope immediately: "Cryptographic key generation tools." The context (SGraph's security brand) makes this natural. |

## Rejected Alternatives

### Why Not Keep `/tools/ssh-keygen` on `send.sgraph.ai`?

The current path (`send.sgraph.ai/tools/ssh-keygen`) works for development but should not be the canonical production URL. Reasons:

1. It attributes a non-Send tool to the Send product.
2. It buries the tool three path segments deep, hurting SEO.
3. It ties the tool's deployment to Send's deployment pipeline.
4. It cannot scale to a tool portfolio without the URL becoming unwieldy.

The path should remain as a **redirect** to the canonical URL at `keys.sgraph.ai/ssh`, ensuring existing bookmarks continue to work.

### Why Not a Completely Generic Domain?

Options like `sgraph.tools` or `keygen.dev` were not considered because they break the `sgraph.ai` domain family. The branding strategy document is explicit: all products live under `sgraph.ai`. Introducing a different TLD fragments the brand.
