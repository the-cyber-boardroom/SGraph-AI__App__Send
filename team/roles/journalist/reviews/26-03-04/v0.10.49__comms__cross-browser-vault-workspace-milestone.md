## The Headline

## What Happened

On 4 March 2026, an HTML file was edited inside an encrypted vault in Safari, then opened and transformed by Claude Sonnet 4.6 in a completely separate browser — Chrome — running the SG/Workspace application.

Same file. Same vault. Two different browsers. Two different applications. One encrypted backend that can't read any of it.

The user typed `<h1>SG/Send Test Page!!!!!</h1>` into the vault's inline code editor in Safari, saved it (encrypted, uploaded), switched to Chrome, opened the same vault in the Workspace, saw the file appear, clicked it, watched it decrypt and render in the source viewer, then asked Claude to analyse it. The LLM response streamed in real-time into the transform panel.

The server served identical encrypted blobs to both browsers. It doesn't know which browser is which. It doesn't know what's inside the files. It doesn't know an LLM was involved.

---

## Why This Is a Big Deal

### The Vault Is Now Infrastructure

Until today, the vault was used inside a single application — the Vault UI. Now it's been proven as **shared encrypted storage** across multiple independent applications. The Vault UI writes. The Workspace reads and transforms. Both use the same vault key, the same AES-256-GCM encryption, and the same server — which remains a blind, stateless blob store.

This is the pattern that turns SG/Send from a file-sharing tool into an **encrypted application platform**. Any application that speaks the vault protocol can participate.

### The Edit-Transform Loop

This is the workflow the Workspace was built for:

1. **Edit** a document in the vault (Safari)
2. **Open** it in the workspace (Chrome)
3. **Transform** it with an LLM (Claude Sonnet 4.6 via OpenRouter)
4. **Save** the result back to the vault (encrypted)

Every step is client-side. Every transition crosses an encryption boundary. The server is infrastructure — plumbing, not intelligence.

### Cross-Browser = Cross-Device = Cross-User

If two browsers on the same machine can share a vault, so can two browsers on different machines. So can two different users, each with their own vault key shared out-of-band. The architecture doesn't distinguish between "same person, two browsers" and "two people, two continents." The encrypted blobs don't care who downloads them.

## The Near-Miss

The workspace initially crashed with:

> `Error: Invalid vault key format. Expected passphrase:vaultId:settingsTransferId`

The CDN was serving an old version of the vault library that expected a 3-part key format. The local code had been updated to 2-part format with deterministic key derivation, but the CDN hadn't been redeployed. A quick CDN update fixed it.

Lesson: when multiple applications share crypto libraries via CDN, **deployment is a security-critical step**. A version mismatch in a crypto library isn't a UI glitch — it's a broken encryption pipeline.
