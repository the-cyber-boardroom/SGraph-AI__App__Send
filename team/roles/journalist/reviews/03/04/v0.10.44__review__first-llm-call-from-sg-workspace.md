## The Headline

**SG/Send makes its first LLM call — and the server never saw a single word.**

On 4 March 2026, a document was loaded from an encrypted vault, transformed by an LLM, and the response streamed back into the workspace — all in the browser. The SG/Send server stored nothing but encrypted blobs. It didn't see the document. It didn't see the prompt. It didn't see the answer.

## What Actually Happened

A user opened the SG/Workspace, connected to OpenRouter, loaded a file from their encrypted personal vault, typed a prompt, and hit Send.

The response streamed in real-time into the transform viewer — chunk by chunk, word by word — while the original document remained visible in the source viewer alongside it. The five-zone workspace layout (vault browser on the left, source and transform viewers in the centre, chat panel at the bottom, debug sidebar on the right) worked exactly as designed.

The whole thing runs in vanilla JavaScript. No React. No Vue. No build step. Zero framework dependencies.

## Why This Matters

### Zero-Knowledge + LLM = Possible

The conventional assumption is that if you want AI to process your documents, you have to send them to a server. SG/Send proves otherwise.

The encrypted document is decrypted **in the browser** using keys that never leave the user's device. The decrypted content is sent **from the browser** directly to the LLM provider (OpenRouter or Ollama). The SG/Send backend is never in the loop.

This means:
- **Your server admin can't read your documents** — they're encrypted at rest
- **Your server admin can't read your LLM prompts** — they go browser → provider
- **Your server admin can't read the LLM responses** — they render in the browser and optionally get encrypted back into the vault

The entire intelligence pipeline is client-side. The server is a dumb blob store. By design.

### Two Providers, One Interface — Both Confirmed Working

The workspace supports two LLM providers, and both were verified on the same day:
- **OpenRouter** — cloud-based, hundreds of models, API key authentication
- **Ollama** — local, runs on your machine, no API key needed. Confirmed with **Gemma3:4b** (Google's open-weights model)

Switch between them with two clicks. Same interface, same streaming, same document pipeline. The API logger in the debug sidebar tells the story: `POST https://openrouter.ai/api/v1/chat/completions` and `POST http://localhost:11434/api/generate` sitting side by side, both returning 200s.

For Ollama, the only requirement is launching with CORS enabled for the workspace origin:
```
OLLAMA_ORIGINS="https://dev.workspace.sgraph.ai" ollama serve
```

Want to run completely offline with a local model? Use Ollama. Want GPT-4 or Claude? Use OpenRouter. The zero-knowledge principle holds either way — but with Ollama, even the LLM provider never sees your data.

### The Document Transform Pipeline

The workflow is elegant in its simplicity:

1. **Select** a file from your encrypted vault
2. **View** it in the source panel (markdown rendered, code highlighted, images displayed)
3. **Write** a prompt (or pick one from the built-in library: "Improve Clarity", "Executive Summary", "Extract Actions", "Simplify", "Convert to Markdown")
4. **Watch** the LLM response stream into the transform panel in real-time
5. **Save** the result back to your vault — encrypted, just like the original

Every step happens in the browser. Step 1 decrypts. Step 5 encrypts. Steps 2-4 are pure client-side processing.

## The Bug That Almost Blocked It

The first attempt crashed with `conn.getModels is not a function`.

The cause was subtle: Web Components don't guarantee upgrade order for siblings. When the workspace shell renders its children, `<llm-chat>` tried to call methods on `<llm-connection>` — but the connection element hadn't been upgraded from a bare `HTMLElement` yet. It existed in the DOM but had no custom methods.

Fix: check `typeof conn.getModels === 'function'` before calling. The component renders gracefully with defaults, then re-renders when the connection fires its `llm-connected` event.

Time from bug report to fix: one commit.
