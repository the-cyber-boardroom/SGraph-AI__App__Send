# The Day the Workspace Learned to Remember

**Version:** v0.11.21
**Date:** 5-6 March 2026
**Commit:** `63323ed`
**Commits in scope:** 16 non-merge commits (v0.11.10 to v0.11.21)
**Author:** Journalist

---

## Prologue

Some days in a project's life are ordinary. A feature ships. A test passes. The version number ticks up by one. And then there are days like 5-6 March — twelve hours of relentless iteration that began with a missing feature and ended with an entirely new architecture. This is the story of execution bundles, a stubborn CSS bug, and the most satisfying three-line fix in SGraph Send's history.

---

## Act 1: The Workspace Learns to Remember

By the afternoon of 5 March, the SGraph Send workspace had already crossed a significant threshold. You could load an HTML document into the Source panel, write a JavaScript transformation in the Script panel, execute it against the data, and ask an LLM to explain the results. A complete document-transformation pipeline, visible end to end.

But it had no memory.

Navigate away and everything vanished — the source document, the script you'd been refining, the LLM conversation that helped you understand the output. There was no history, no undo, no way to hand someone a link and say "look what I found." The workspace was powerful but amnesic.

Then, in a single commit (`a7603f9`), execution bundles arrived.

The concept was deceptively simple: capture the workspace's complete state — every panel's content, every prompt, every screenshot — into a single JSON snapshot. Store it in the user's encrypted vault. The implementation touched every layer of the workspace, but the user experience was immediate and intuitive. Click **Save State** and the moment freezes. Click any saved bundle in the sidebar and you are transported back to that exact configuration: the same source document, the same script, the same results, the same conversation.

Time travel for document transformations.

But the bundles also introduced something subtler — fork tracking. Each saved state records which previous state it branched from. Over time, a tree of explorations grows. You can see not just where you ended up, but the path you took to get there, and the roads not taken. For a tool designed to help analysts explore encrypted documents, this is not a convenience feature. It is the feature. The workspace doesn't just transform documents anymore. It remembers how you thought about them.

This was the high point of the afternoon. What came next was considerably less glamorous.

## Act 2: The Bug-Fix Marathon

With execution bundles wired in, the real testing began — and the workspace started falling apart in creative ways.

Thirteen commits landed in rapid succession across the evening of 5 March, each addressing a different failure mode. Taken individually, each was a routine fix. Taken together, they tell the story of a system under stress, revealing every assumption that had been quietly wrong.

**The IIFE scope error.** The v0.1.1 workspace components were wrapped in Immediately Invoked Function Expressions for isolation — a correct instinct, but one that made them invisible to each other at registration time. Components that needed to reference siblings couldn't find them. The fix was surgical: expose the class definitions at module scope while keeping the IIFE's side-effect isolation.

**The vault API mismatch.** Saved bundles were vanishing silently. The bundle manager called `writeFile()` on the vault API; the vault API exposed `addFile()`. No error was thrown — the call simply did nothing. A classic integration seam failure, caught only when someone tried to restore a bundle and found an empty list.

**The shell injection timing bug.** Workspace-shell tried to patch component prototypes before the custom elements had upgraded. The patches landed on the base HTMLElement prototype, not the component class. When the elements finally upgraded, they used their own methods, ignoring the patches entirely. The fix introduced an upgrade-aware initialization sequence.

**The panel collapse cascade.** This one deserves its own act.

## Act 3: The 4-Column Revolution

The collapsible panels should have been straightforward. Each of the four workspace panels — Source, Data, Script, Result — needed to collapse to a thin 28-pixel header bar when clicked, with its siblings expanding to fill the vacated space. The interaction model was obvious. The implementation was not.

The root cause took six commits to fully understand: the workspace layout was a hybrid. Source and Data were stacked vertically inside a flex container, which was itself a cell in a horizontal CSS grid alongside Script and Result. Two different layout algorithms — flex and grid — calculating space by different rules, fighting each other at every resize.

Attempt 1 used flex-based collapse. Panels refused to shrink below their content height. Attempt 2 set explicit pixel heights — this fixed the Data panel but broke Source. Attempt 3 let sibling panels absorb space with `flex: 1` — nearly correct, but the both-collapsed case left a gap. Attempt 4 special-cased the Source zone — fixing one panel's behaviour while introducing a new bug in another.

Each fix was locally correct and globally wrong. The layout was not buggy. The layout was architecturally incompatible with the feature.

On the morning of 6 March, the decision was made: flatten everything.

Commit `1d12d9b` restructured the entire workspace into four equal CSS grid columns. Source | Data | Script | Result. No nesting. No flex-inside-grid. One `grid-template-columns` rule governs the entire horizontal layout. Collapsing a panel sets its column to `28px` and distributes the freed space to the remaining columns with a single rule change.

It was the kind of refactor that feels obvious in retrospect and impossible in prospect. The old layout had grown organically — Source and Data were grouped because they were conceptually related (input data). But the layout engine doesn't care about conceptual relationships. It cares about containment hierarchies. Once the hierarchy was flattened, every panel-collapse bug disappeared simultaneously.

Almost.

Commit `2b8d23c` fixed one last specificity conflict — an `!important` declaration on a panel's width that overrode the grid column assignment. Remove the `!important`, and the grid finally had full authority.

---

## Act 4: The 3-Line Fix

The last bug was the most satisfying to resolve.

When restoring a saved execution bundle, the collapsed-panel states appeared correctly — the headers showed, the chevrons pointed the right direction — but the panels didn't actually resize. The grid columns stayed at their expanded widths. The visual indicators said "collapsed" but the layout said "expanded."

The cause was immediately clear once the 4-column refactor was in place. The bundle-manager component had its own copy of the grid layout logic, written weeks earlier for the original 3-column layout. Fifty lines of code that calculated column widths, accounted for the flex-inside-grid nesting, handled the Source-Data special case. All of it was now wrong — not because it had bugs, but because the layout it targeted no longer existed.

The fix was to delete all fifty lines and replace them with three:

1. Write the collapsed-panel state to `localStorage`.
2. Dispatch a `workspace-panels-changed` custom event.
3. Let workspace-shell's existing `applyAllPanelState()` handler do the rest.

The bundle-manager no longer knows how panels collapse. It doesn't need to. It announces that panel state has changed, and the component that owns the layout — workspace-shell — responds. Less code. Correct behaviour. Zero duplication. Single source of truth.

The user confirmed it worked with a screenshot showing all four panels correctly restored from a saved bundle, collapsed states and all.

## What This Day Means

The workspace entered 5 March as a stateless tool. It left 6 March as a persistent, explorable environment with architectural foundations that will carry it through the next phase of development.

But the deeper lesson is about the relationship between features and architecture. Execution bundles were a feature request. Collapsible panels were a feature request. Neither could be delivered cleanly on the existing layout. The layout wasn't broken — it had simply reached the limit of what its structure could support. The 4-column refactor wasn't on anyone's roadmap. It emerged from the pressure of trying to make two reasonable features work, and discovering that the only path forward was through the foundation.

This is what twelve hours of focused iteration looks like: you start by adding a feature, spend the evening discovering why the feature breaks everything, and finish the next morning with a system that is simpler, more capable, and more correct than what you started with.

The best days in software are the ones where you end up with less code than you began with.
