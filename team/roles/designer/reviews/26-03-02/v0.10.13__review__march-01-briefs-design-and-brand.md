## Brief 1: The SG/Send Challenge — Outreach Campaign Materials

**Source:** `team/humans/dinis_cruz/briefs/03/01/v0.10.1__brief__sg-send-challenge-outreach-campaign.md`

### Design Assessment

The challenge card is the campaign's visual anchor. It will be seen cold — in a WhatsApp message preview, in a LinkedIn feed, in an email inbox — before the recipient knows anything about SG/Send. The card must accomplish three things in under two seconds: establish brand recognition, communicate the challenge concept, and feel personal rather than corporate.

**The brief's wireframe is directionally correct.** The five-step flow, the conversational tone, the absence of hard-sell language — these are sound design intentions. The execution needs specificity.

### Brand Consistency Evaluation

**Colour:** The card must use the Aurora palette. Dark Navy (#1A1A2E) background, Light Grey (#E0E0E0) text, Teal (#4ECDC4) for the SG/ brand mark and key highlight elements, Coral (#E94560) sparingly for the lock icon or encryption marker. The card should not use white backgrounds or bright gradients — these would read as generic marketing and break the Aurora identity.

**Typography:** DM Sans Bold 700 for the headline ("The SG/Send Challenge"), DM Sans Medium 500 for the body steps, DM Sans Regular 400 for the reward line. JetBrains Mono is not appropriate here — the card is conversational, not technical.

**Logo:** The Concept C icon mark should appear in the top-left corner at 32px (or equivalent proportion at the card's output size). The full lockup ("SG/Send" wordmark) is too wide for a card that needs text-dense body copy. The icon alone, at this size, communicates brand without consuming layout space.

### UX Recommendations

**D-CH1: Card dimensions matter for each platform.** WhatsApp renders shared images at approximately 300px wide in the chat. LinkedIn renders inline images at approximately 506px wide in the feed. The card must be legible at both sizes. This means:

- Minimum font size on the card: 16px equivalent at 300px rendering width. At a 1200px source image, that is 64px in the source file.
- The five steps must not be set in a font smaller than the body text. At WhatsApp thumbnail size, small text becomes unreadable blurs.
- The lock icon in the brief wireframe should be the Concept C icon mark, not a generic lock emoji. This reinforces brand recognition in a context where the recipient may never have seen SG/Send before.

**D-CH2: WhatsApp does NOT support SVG.** The brief asks to "verify WhatsApp supports SVG" — it does not. WhatsApp renders JPEG and PNG. The SVG should be produced as the vector source, and PNG exports at 2x resolution (2400x minimum width) should be the delivery format. JPEG is acceptable if file size is a concern, but PNG preserves the sharp edges of the Aurora palette's dark-to-teal transitions better.

**D-CH3: Multilingual card variants need careful text reflow.** The brief requests EN, ES, PT, DE, FR variants at minimum. German text is typically 30-40% longer than English. The card layout must accommodate "Kannst du eine verschlusselte Datei senden, die niemand lesen kann — nicht einmal wir?" without breaking the visual composition. Two approaches:

- **Fixed layout, variable font size:** Keep the card dimensions constant; reduce font size for longer languages. Risk: German card becomes harder to read than English card.
- **Fixed font size, variable card height:** Keep font size constant; allow the card to grow taller for longer languages. Risk: inconsistent visual weight across language variants.

**Recommendation:** Use the fixed-font-size approach. The card height can flex modestly (no more than 15%) without disrupting the visual impression. Set the card width at 1200px (standard OG/social share width) and the height at 630-720px depending on language. Design the English card first as the shortest variant; use the German card as the stress test for maximum expansion.

**D-CH4: The QR code is a missing opportunity.** The brief does not mention a QR code, but for WhatsApp-forwarded cards, a QR code to sgraph.ai in the bottom-right corner allows the recipient to scan-to-visit from a printed or screenshot version. Keep the QR code subtle — 80px square, Slate (#8892A0) on Dark Navy, no frame or branding inside the QR itself.

### Visual Deliverables Needed

| Deliverable | Format | Dimensions | Priority |
|-------------|--------|------------|----------|
| Challenge card — English | SVG (source) + PNG (delivery) | 1200 x 630-720px | P0 |
| Challenge card — ES, PT, DE, FR | SVG + PNG per variant | 1200 x 630-720px | P0 |
| Challenge card — LinkedIn optimised | PNG | 1200 x 628px (LinkedIn spec) | P0 |
| Challenge card — WhatsApp optimised | PNG | 1200 x 1200px (square preview) | P1 |
| Website screenshot per language | PNG | 1200 x 800px (browser frame crop) | P1 |

### Priority Actions

1. **P0:** Produce the English challenge card as the master template with Aurora palette, Concept C icon, DM Sans typography, and the five-step flow.
2. **P0:** Export PNG variants for WhatsApp and LinkedIn at the dimensions specified above.
3. **P0:** Produce DE-DE variant as the expansion stress test, then ES-ES, PT-PT, FR-FR.
4. **P1:** Produce language-specific sgraph.ai screenshots for the LinkedIn post variants.

## Brief 2: Translation Visualisation Tool and Translator Community

**Source:** `team/humans/dinis_cruz/briefs/03/01/v0.10.1__brief__translation-tool-and-translator-community.md`

### Design Assessment

The translation tool has four distinct UI surfaces: side-by-side comparison, culture variant diff view, statistics dashboard, and glossary viewer. Each is a data-dense interface that must remain readable on the Aurora dark theme. The brief's ASCII wireframes are functional but do not address colour coding, information density, or the dark-theme readability challenges specific to data tables.

### Brand Consistency Evaluation

The tool lives at `sgraph.ai/tools/translations/`. It must use the Aurora design system — same tokens, same typography, same header/footer chrome as the sgraph.ai website. The User UI v0.2.0 design tokens (`design-tokens.css`) provide the foundation, but the tool will need additional data-table patterns not currently in the shared components.

### UX Recommendations

**D-TR1: Side-by-side comparison table — column width strategy.** The table has three columns: Key, Language A, Language B. The Key column contains dot-notation strings (`hero.title`, `pricing.per_file`) that are naturally short but variable. The translation columns contain full sentences.

- Key column: fixed width, 180px minimum, truncated with `text-overflow: ellipsis` and tooltip on hover.
- Translation columns: flex-grow, equal width, splitting the remaining space.
- On mobile (<768px): the table must collapse to a stacked card layout — one card per key, showing Language A above Language B. A three-column table on a 375px screen is unreadable.

**D-TR2: Culture variant diff view — visual highlighting is critical.** The brief describes showing "ONLY the differences." This is correct — but the differences must be visually highlighted within the translation string, not just flagged at the row level. A word-level diff (highlighting "ficheiro" vs "arquivo" within the sentence) is dramatically more useful than a row-level "DIFFERENT" marker. Use Coral (#E94560) as the diff highlight colour on a dark background — the same colour used for "what the server never sees" in the brand palette. This is semantically appropriate: the diff view highlights what is different, what you might miss.

**D-TR3: Statistics dashboard — progress bars must use the Aurora palette correctly.** The brief shows progress bars with percentage fill. The fill colour should be:

- 100% coverage: Teal (#4ECDC4)
- 80-99%: Teal at 70% opacity
- Below 80%: Warning orange (#E07C4F)

Do not use green/red as progress indicators — the Aurora palette does not include green, and using traffic-light colours on a dark navy background creates a jarring palette break.

**D-TR4: Quality badges need careful treatment on dark backgrounds.** The brief proposes four badge types: AI, Reviewed, Edited, Native. Emoji badges are acceptable in the brief's context, but the implementation should use styled badges aligned with the existing `.status` component in the shared components CSS:

| Badge | Background | Border | Text |
|-------|-----------|--------|------|
| AI | `rgba(136, 146, 160, 0.1)` | `rgba(136, 146, 160, 0.2)` | Slate #8892A0 |
| Reviewed | `rgba(78, 205, 196, 0.08)` | `rgba(78, 205, 196, 0.15)` | Teal #4ECDC4 |
| Edited | `rgba(224, 124, 79, 0.08)` | `rgba(224, 124, 79, 0.2)` | Warning #E07C4F |
| Native | `rgba(78, 205, 196, 0.15)` | `rgba(78, 205, 196, 0.3)` | Teal #4ECDC4, bold |

These use the existing status badge colour system from `shared-components.css`, which ensures visual coherence with the rest of the design system.

**D-TR5: The "Edit on GitHub" link pattern.** Each row should have a subtle edit icon (pencil) in Slate (#8892A0) that transitions to Teal on hover. This follows the established link-hover pattern in the design tokens (`a { color: var(--accent); }` and `a:hover { color: var(--accent-hover); }`). The icon should be 16px, right-aligned within the key column, and should not compete with the translation text for attention.

**D-TR6: The contributors page needs a design specification.** The brief describes a leaderboard-style display. This should use the card component from shared-components, one card per language, with contributor avatars (GitHub profile pictures), name, and correction count. The sort order should be by correction count descending. The visual treatment should celebrate contribution — a "top contributor" badge using the Native quality badge style for anyone with 10+ corrections.

### Visual Deliverables Needed

| Deliverable | Priority |
|-------------|----------|
| Side-by-side comparison table design (desktop + mobile layouts) | P0 |
| Culture variant diff view with word-level highlighting spec | P0 |
| Statistics dashboard with progress bars and badges | P1 |
| Glossary viewer layout (searchable table) | P1 |
| Contributors section design | P2 |
| Mobile responsive breakpoint specification for all four views | P0 |

### Priority Actions

1. **P0:** Define the table component for data-dense translation views — this is a new component pattern not in the current shared-components CSS. It needs column width strategy, row hover states, cell padding, and mobile collapse behaviour.
2. **P0:** Specify the word-level diff highlighting mechanism for culture variant view.
3. **P1:** Design the statistics dashboard using the existing progress-bar component from shared-components, extended with the translation-specific colour logic.
4. **P1:** Map the quality badges to the existing `.status` component variants.

## Brief 3: Website Improvements and Content Strategy

**Source:** `team/humans/dinis_cruz/briefs/03/01/v0.10.1__brief__website-improvements-and-content-strategy.md`

### Design Assessment

This brief addresses two Designer-relevant items: (a) making the token UX paths more visually distinct, and (b) fixing the pricing table layout for non-English languages. Both are addressed in the page redesign proposal review (`v0.10.9__review__page-redesign-proposal.md`), but this brief adds the multilingual dimension that the redesign proposal did not address.

### Brand Consistency Evaluation

The brief's proposals are consistent with the existing brand. No new visual language is introduced. The risk is in the multilingual execution: when the "Buy Credits" CTA is translated into 17 language variants, the button label width varies dramatically, and the visual hierarchy may break.

### UX Recommendations

**D-WI1: Token UX paths — the "already have a token" vs "get a token" fork.** The brief identifies this as needing visual distinction. The page redesign proposal (Finding D8) already addresses this with two primary cards at the top of the pricing page. From the Designer's perspective, the critical addition is:

- **Path 1 ("I have a token"):** Should be a text link, not a card. It is a bypass route for returning users, not a primary conversion path. Position: below the hero, in the secondary text zone (currently the "Already have a token? Go to send.sgraph.ai" link). Style: `color: var(--accent)`, DM Sans Regular 400, `var(--text-sm)`. This is already in place.
- **Path 2 ("I want a token"):** This is the primary funnel. It branches into Free (form) and Paid (Stripe). The two primary cards on the redesigned pricing page handle this. The homepage hero should have a single CTA pair: "Buy Credits" (primary) and "Try Free" (secondary), which the redesign proposal specifies.

No new design work is needed here beyond what the redesign proposal covers. The brief and the redesign proposal are aligned.

**D-WI2: Pricing table layout for non-English languages.** The brief reports that pricing tables break in DE-DE, HR-HR, and PT-PT. This is the text expansion problem identified in the Designer's work stream proposal P6 (i18n Design). The specific fix is:

- **Pricing card titles:** Must use `overflow-wrap: break-word` and `hyphens: auto` (with `lang` attribute set correctly on the HTML element for each locale) to handle long compound words in German.
- **Pricing card feature lists:** Must use `word-break: normal` (not `break-all`) and allow list items to wrap naturally. The current card design (`padding: var(--space-6)` = 24px) provides sufficient internal space for two-line feature items.
- **Pricing comparison table cells:** Must not use `white-space: nowrap`. Cells must allow wrapping. Column widths should use `min-width` instead of `width` to accommodate variable content.
- **CTA buttons in cards:** The "Buy Credits" button in German ("Credits kaufen") is only slightly longer. But "Jetzt kostenlos starten" (German for "Get Started Free") is 60% longer than the English. Buttons must use `white-space: normal` and `min-height` instead of fixed height, allowing two-line button text on narrow cards.

**Stress test languages for pricing table fixes:** DE-DE (long compound words), HR-HR (diacritics and multi-word translations), PT-PT (slightly longer Romance-language constructions), FR-CA (French with longer preposition constructions than European French).

**D-WI3: The news/blog section design.** The brief proposes a news section. This is a new content type that needs a design template. The template should follow the Aurora system:

- Article card: `.card` component with thumbnail (optional), title (`h3`, DM Sans Bold 700), date (Slate, `var(--text-small)`), excerpt (Light Grey, `var(--text-sm)`), and language badge if not in EN-GB.
- Article listing page: vertical stack of article cards, newest first, with date headers.
- Article detail page: reading-optimised layout (max-width 720px, same as the educational content template from work stream P4).

This is PROPOSED work that does not yet exist. The design template should be queued behind the pricing page redesign and the challenge card.

### Visual Deliverables Needed

| Deliverable | Priority |
|-------------|----------|
| Pricing table CSS fix for text expansion across 17 locales | P0 |
| News/blog article card component design | P1 |
| News/blog article detail page template | P1 |
| Language badge component for content not available in all languages | P2 |

### Priority Actions

1. **P0:** Fix the pricing table CSS for non-English text overflow. This is a live-site bug, not a design proposal. Test against DE-DE, HR-HR, PT-PT, FR-CA.
2. **P1:** Produce the news article card component spec using the existing `.card` base from shared-components.
3. **P2:** Design the language availability indicator ("EN-GB only" badge) for content pages.

## Brief 4: SG/Send Logo Specification (DESIGNER LEAD)

**Source:** `team/humans/dinis_cruz/briefs/03/01/v0.10.1__sg-send-logo-specification.md`

### SVG Implementation Verification

I have inspected both deployed SVG files:

- `sgraph_ai__website/_common/img/logo/sg-send-favicon.svg`
- `sgraph_ai__website/_common/favicon.svg`

**Finding D-LG1: Both SVG files are identical.** Both files contain the same SVG source code. This is correct — there should be a single canonical favicon SVG. Having two copies is acceptable for deployment convenience (one in the logo directory for explicit reference, one at the common root for the `<link rel="icon">` path), but they must remain synchronised. Any update to one must update the other.

**Finding D-LG2: The SVG source matches the specification exactly.** Every element in the deployed SVG matches the specification document:

| Spec Element | Specified | Implemented | Match |
|---|---|---|---|
| Background | `<rect width="32" height="32" rx="7" fill="#1A1A2E"/>` | Identical | Yes |
| S-curve colour | Teal #4ECDC4, stroke-width 3, opacity 0.9 | `stroke="#4ECDC4" stroke-width="3" opacity="0.9"` | Yes |
| S-curve path | Cubic Bezier from (20,9) through four curves to (14,21) | `M20 9 C20 9 22 9 22 11.5 C22 14 18 14.5 16 14.5 C14 14.5 10 15 10 17.5 C10 20 12 21 14 21` | Yes |
| S-curve linecap | `stroke-linecap="round"` | Present | Yes |
| Slash | Coral #E94560, stroke-width 2.5, from (20,5) to (12,27) | `stroke="#E94560" stroke-width="2.5"` | Yes |
| Slash linecap | `stroke-linecap="round"` | Present | Yes |
| Dot 1 (top-left) | cx=7 cy=7 r=1, Slate #8892A0, opacity 0.3 | Identical | Yes |
| Dot 2 (top-left) | cx=7 cy=11 r=1, Slate #8892A0, opacity 0.2 | Identical | Yes |
| Dot 3 (bottom-right) | cx=25 cy=21 r=1, Slate #8892A0, opacity 0.3 | Identical | Yes |
| Dot 4 (bottom-right) | cx=25 cy=25 r=1, Slate #8892A0, opacity 0.2 | Identical | Yes |

**Verdict: The SVG implementation is a faithful rendition of the specification. No discrepancies.**

### Specification Completeness Assessment

The logo specification document is thorough. It covers anatomy, symbolism, design decisions, wordmark composition, colour palette, variants, SVG source code with element breakdown, scalability, and brand guideline alignment. This is a well-structured specification.

**Finding D-LG3: The specification is missing clear-space rules.** The spec defines minimum widths (32px for icon, 280px for full lockup, 16px for monogram) but does not define the minimum clear space around the icon. Clear space prevents other UI elements from crowding the mark and degrading recognition. Recommendation:

- **Icon mark clear space:** 4px (12.5% of the 32px viewBox) on all sides. At scale, this means the icon mark always has a buffer zone equal to the width of one encryption dot.
- **Full lockup clear space:** The height of the "S" character in "SG" on all sides.
- **Absolute minimum:** No element should touch or overlap the logo mark at any size.

**Finding D-LG4: The specification is missing usage restrictions.** A logo specification should state what is not allowed: no rotation, no distortion, no recolouring (with exceptions for monochrome contexts), no placing the icon on busy/patterned backgrounds, no adding drop shadows or outlines. These restrictions prevent well-intentioned but brand-damaging modifications by downstream implementers.

**Finding D-LG5: The specification does not address monochrome and single-colour variants.** There are contexts where the full-colour icon cannot be used: printed on a single-colour surface, used in a low-colour email signature, or rendered as a mask for PWA. The spec should define:

- **Monochrome light variant:** All elements in Light Grey (#E0E0E0) on transparent or Dark Navy background. S-curve at 90% opacity, slash at 100%, dots at 30% and 20%.
- **Monochrome dark variant:** All elements in Dark Navy (#1A1A2E) on white or light background. This is the inverse for light-background contexts (print, light-mode UIs in the future).
- **Single-colour variant:** All elements in Teal (#4ECDC4) on Dark Navy. Use when coral is not available in the rendering context.

**Finding D-LG6: The slash angle specification should include a precise value.** The spec states "approximately 75 degrees from horizontal" and "matching the brand specification." The coordinate geometry of the slash line is `(20, 5)` to `(12, 27)`. The vector is `dx = -8, dy = 22`. The angle from horizontal is `atan(22/8) = 70.0 degrees`, not 75 degrees. This is a discrepancy between the stated angle and the implemented geometry.

Either:
- (a) The SVG should be adjusted so the slash is at 75 degrees. At 75 degrees with the same starting point (20, 5) and the same vertical extent (22px), the x-displacement would be `22 / tan(75) = 5.9px`, making the endpoint `(14.1, 27)` instead of `(12, 27)`.
- (b) The specification should be updated to state the actual angle: 70 degrees.

**Recommendation:** This discrepancy is minor and does not affect visual recognition. The human should decide which is canonical — the spec (75 degrees) or the SVG (70 degrees) — and the other document should be updated to match. From a design perspective, 70 degrees reads well at small sizes; 75 degrees would make the slash slightly more vertical, giving it a slightly different character. Neither is objectively better. Consistency between spec and implementation is the priority.

### Design Token Alignment

The design tokens in User UI v0.2.0 (`design-tokens.css`) are well-aligned with the logo specification:

| Token | Value | Logo Usage | Aligned |
|-------|-------|-----------|---------|
| `--color-bg` / `--bg-primary` | #1A1A2E | Background rectangle | Yes |
| `--color-primary` / `--accent` | #4ECDC4 | S-curve, wordmark slash | Yes |
| `--color-error` / `--danger` | #E94560 | Icon slash (coral) | Yes |
| `--color-text` | #E0E0E0 | Wordmark text | Yes |
| `--color-text-secondary` | #8892A0 | Encryption dots (slate) | Yes |
| `--font-family` | 'DM Sans' | Wordmark typography | Yes |
| `--weight-bold` | 700 | "SG" in wordmark | Yes |
| `--weight-medium` | 500 | "Send" in wordmark | Yes |

**Finding D-LG7: The design tokens do not include a dedicated coral token.** Coral (#E94560) is used in the logo for the slash and in the design system as `--color-error` and `--danger`. This semantic aliasing is problematic: coral in the logo represents the encryption boundary (a positive, identity-defining concept), while `--color-error` and `--danger` represent failure states (a negative, corrective concept). If a designer or developer reads `--danger` as the token for the logo slash, they may instinctively avoid it in brand-positive contexts.

**Recommendation:** Add a dedicated `--color-coral` or `--brand-coral` token to `design-tokens.css` set to `#E94560`. This does not change the value — it adds semantic clarity. The error/danger tokens remain for their purpose; the brand token exists for logo, brand mark, and encryption-boundary contexts. This is a small change with outsized benefit for design system comprehension.

### Scalability Assessment

The specification's scalability table (512px down to 16px) is correct. The SVG's use of stroked paths (not filled shapes) means the S-curve and slash scale gracefully. The encryption dots (1px radius circles at 20-30% opacity) disappear below approximately 24px rendering size, which is intentional and documented.

**Finding D-LG8: The 16px monogram variant ("S/") referenced in the spec has no SVG source.** The spec defines three variants (full lockup, icon mark, monogram) but only provides SVG for the icon mark. The monogram ("S/") is listed for "favicon at very small sizes" (16px). If the monogram is a distinct asset (not just the icon mark scaled down), it needs its own SVG source. If it is simply the icon mark at 16px, the spec should clarify this.

**Recommendation:** Clarify whether the 16px monogram is a distinct asset or the icon mark at 16px. If distinct, produce the SVG. If identical, update the spec to read: "At 16px, the icon mark functions as a monogram: the slash dominates and the S-curve provides shape context."

### Specification Gaps Summary

| # | Gap | Severity | Action |
|---|-----|----------|--------|
| D-LG3 | Missing clear-space rules | Medium | Add clear-space specification to the logo spec |
| D-LG4 | Missing usage restrictions | Medium | Add "don'ts" section to the logo spec |
| D-LG5 | Missing monochrome variants | Medium | Define mono-light, mono-dark, and single-colour variants |
| D-LG6 | Slash angle discrepancy (spec says 75, SVG is 70) | Low | Decide canonical angle; update the non-canonical document |
| D-LG7 | No dedicated coral brand token in design-tokens.css | Low | Add `--color-coral` or `--brand-coral: #E94560` |
| D-LG8 | 16px monogram has no SVG source | Low | Clarify or produce the monogram SVG |

### Priority Actions

1. **P0:** Resolve the slash angle discrepancy (D-LG6). This is a factual inconsistency that should not persist.
2. **P1:** Add clear-space rules and usage restrictions (D-LG3, D-LG4) to the logo specification.
3. **P1:** Define monochrome variants (D-LG5) for print and low-colour contexts.
4. **P1:** Add `--brand-coral` token to `design-tokens.css` (D-LG7).
5. **P2:** Clarify or produce the 16px monogram variant (D-LG8).

## Brief 5: OG Image Social Preview

**Source:** `team/humans/dinis_cruz/briefs/03/01/v0.7.19__task__og-image-social-preview.md`

### Design Assessment

The OG image is the first visual impression of SG/Send on any social platform. When a link to sgraph.ai is shared on WhatsApp, LinkedIn, Slack, or Twitter/X, the OG image is rendered as the thumbnail. This is a high-leverage design asset: every share becomes an ad.

### Brand Consistency Evaluation

The brief suggests "the aurora theme palette" and the SGraph AI logo/brand mark. This is correct. The OG image must be unmistakably SG/Send at a glance.

### UX Recommendations

**D-OG1: The OG image must be legible at 300px wide.** WhatsApp renders OG thumbnails at approximately 300px wide. At 1200x630 source, that is a 4x reduction. Any text on the image must survive this reduction:

- Headline text: minimum 48px at source (renders at ~12px at 300px wide — small but legible in DM Sans Bold).
- Subhead/tagline: minimum 32px at source (renders at ~8px — at the edge of legibility; should be short words, not full sentences).
- Logo mark: the Concept C icon at 80px minimum at source (renders at ~20px — still recognisable).

**D-OG2: The brief's suggested tagline needs updating.** The brief says: "AI Agents That Build, Use, and Secure the Same Platform." This is the SGraph AI parent brand tagline, not the SG/Send product tagline. The SG/Send OG image should use the product-specific tagline: "Send files. Nobody can read them -- not even us." (This is the hero text from the sgraph.ai website.) If this OG image is for the parent sgraph.ai site, the brief's tagline is appropriate; if it is for send.sgraph.ai specifically, use the product tagline.

**Recommendation:** Produce two OG images:
- **sgraph.ai OG image:** SGraph AI logo + "AI Agents That Build, Use, and Secure the Same Platform"
- **send.sgraph.ai OG image:** Concept C icon mark + "SG/Send" wordmark + "Send files. Nobody can read them." (shortened for legibility at thumbnail size)

```
└─────────────────────────────────────────────────────────────┘
```

**D-OG4: Do not use gradients, abstract shapes, or stock imagery.** The Aurora theme is restrained. A dark solid background with typographic elements and the icon mark is more on-brand than aurora-gradient effects or abstract particle animations. The OG image should feel like a screenshot of the brand, not a marketing banner.

**D-OG5: File location and integration.** The brief specifies `sgraph_ai__website/_common/images/og-preview.png`. This is correct. The meta tag should use an absolute URL: `<meta property="og:image" content="https://sgraph.ai/_common/images/og-preview.png">`. Also add `<meta property="og:image:width" content="1200">` and `<meta property="og:image:height" content="630">` for social platform rendering hints. Add `<meta name="twitter:image" content="...">` as well — Twitter/X does fall back to `og:image`, but explicit `twitter:image` is more reliable.

## Cross-Brief Design Findings

### Finding D-XB1: The Challenge Card and OG Image Share a Visual Language

The challenge card (Brief 1) and the OG image (Brief 5) are both 1200px-wide PNG assets using the Aurora palette. They should share a visual system: same background treatment, same typography scale, same icon placement convention. This prevents the scenario where two different team members produce two visually inconsistent assets that both represent the brand.

**Recommendation:** Design the OG image first (it is simpler — fewer text elements), then use it as the template foundation for the challenge card. The challenge card adds the five-step flow as body content within the same visual frame.

### Finding D-XB2: The Translation Tool and User UI v0.2.0 Share a Component Gap

The translation tool (Brief 2) needs data tables. The User UI v0.2.0 shared-components CSS has cards, buttons, inputs, status badges, progress bars, and result panels — but no table component. The Admin UI has tables (token-manager, analytics-dashboard, etc.), but these are defined within individual Web Component shadow DOMs, not in a shared component stylesheet.

**Recommendation:** Before building the translation tool, extract a shared data table component from the Admin UI patterns and add it to the shared-components CSS. This prevents the translation tool from reinventing table styles and ensures consistency if future tools also need tables.

### Finding D-XB3: Font Vendoring is Complete in User UI v0.2.0

The `fonts.css` in User UI v0.2.0 declares `@font-face` rules for DM Sans (400, 500, 600, 700) and JetBrains Mono (400) with local `url()` references. This resolves the Google Fonts violation for the User UI — provided the actual `.woff2` files are present alongside the CSS (the font file declares `src: url('DMSans-Regular.woff2')` with a relative path). The Designer notes that this is a significant improvement from v0.1.8, which still loaded from Google Fonts. The work stream P5 (User UI Aurora uplift) can now focus on token alignment rather than font vendoring.

However, the font declarations include DM Sans SemiBold (600), which is not in the brand palette specification (the spec lists Bold 700, Medium 500, Regular 400). SemiBold 600 is used in the shared-components CSS for `.btn` and `.result-row label`. This is not a violation — SemiBold provides a useful intermediate weight for UI elements — but the brand specification should be updated to acknowledge it.

### Finding D-XB4: The Brand Palette Needs a Formal Coral Semantic

Across all five briefs, Coral (#E94560) appears in three distinct semantic roles:

1. **Logo:** The slash / encryption boundary (positive, identity-defining)
2. **Design tokens:** Error/danger states (negative, corrective)
3. **Translation diff:** Changed content highlighting (neutral, informational)

A single colour carrying three semantic meanings creates confusion. The `--brand-coral` token recommendation from D-LG7 partially addresses this, but the design system documentation (work stream D1) should explicitly list the three semantic roles and when each applies.
