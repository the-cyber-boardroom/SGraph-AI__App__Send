## 1. Design System for vault.sgraph.ai

### 1.1 Relationship to Existing Design Systems

vault.sgraph.ai inherits from and extends two existing design systems:

| System | Source | Relationship |
|--------|--------|-------------|
| Aurora Theme | `sgraph_ai__website/_common/css/style.css` | Brand colours, typography, spacing scale, component patterns |
| Send Design Tokens | `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/_common/css/design-tokens.css` | Application-level tokens, component styles |

vault.sgraph.ai adopts the Aurora theme wholesale and adds vault-specific tokens. It does NOT fork the design system -- it layers on top.

### 1.2 Colour Palette

The vault uses the Aurora palette unchanged, with two additional semantic tokens for vault-specific states.

```
AURORA PALETTE (inherited)
--------------------------
--color-navy:        #1A1A2E   (background primary)
--color-deep-blue:   #16213E   (background secondary)
--color-surface:     #1E2A4A   (card/surface)
--color-teal:        #4ECDC4   (accent / primary action)
--color-teal-hover:  #3DBDB4   (accent hover)
--color-coral:       #E94560   (danger / error)
--color-warm-white:  #F0F0F5   (light background variant)
--color-light-grey:  #E0E0E0   (primary text)
--color-mid-grey:    #8892A0   (secondary text)
--color-slate:       #2C3E6B   (borders)

VAULT-SPECIFIC TOKENS (new)
----------------------------
--vault-locked:      #8892A0   (locked/inaccessible state)
--vault-unlocked:    #4ECDC4   (unlocked/active state -- same as accent)
--vault-encrypting:  #E07C4F   (in-progress encryption -- warm amber)
--vault-version:     #7B93DB   (version indicator -- muted blue)
```

**Cultural colour note:** The palette avoids red as a primary or success colour. Red (`--color-coral`) is reserved exclusively for errors and destructive actions. Green is not used -- teal serves as the positive/success colour, which carries no strong cultural associations across target markets.

### 1.3 Typography

The primary font stack (DM Sans + JetBrains Mono) works well for Latin, Cyrillic, and basic Latin-extended scripts. For multi-cultural support, the font stack must be extended with fallbacks that cover CJK, Arabic, Devanagari, and other scripts.

```css
:root {
    /* Display + Body -- extended with multi-script fallbacks */
    --font-display: 'DM Sans',
                    'Noto Sans',           /* Multi-script fallback   */
                    'Noto Sans CJK SC',    /* Simplified Chinese      */
                    'Noto Sans CJK JP',    /* Japanese                */
                    'Noto Sans CJK KR',    /* Korean                  */
                    'Noto Sans Arabic',     /* Arabic                  */
                    'Noto Sans Devanagari', /* Hindi                   */
                    system-ui,
                    -apple-system,
                    sans-serif;

    --font-body:    var(--font-display);

    /* Monospace -- for vault keys, file IDs, code */
    --font-mono:    'JetBrains Mono',
                    'Noto Sans Mono',
                    'SF Mono',
                    'Cascadia Code',
                    monospace;
}
```

**Strategy:** Load DM Sans (vendored woff2, same as sgraph.ai website). Do NOT bundle Noto Sans variants -- let the system-ui fallback chain handle non-Latin scripts. This keeps the vault lightweight. When a specific locale is heavily used, consider vendoring that Noto variant in a future version.

**Type scale (inherited from Aurora):**

| Token | Size | Usage |
|-------|------|-------|
| `--text-display` | 2.5rem | Vault name on entry screen |
| `--text-h1` | 2rem | Screen titles |
| `--text-h2` | 1.6rem | Section headings |
| `--text-h3` | 1.25rem | Card titles, folder names |
| `--text-body` | 1rem | Body text, file names |
| `--text-sm` | 0.875rem | Metadata, timestamps |
| `--text-small` | 0.8rem | Captions, hints |
| `--text-micro` | 0.625rem | Badges, counters |

### 1.4 Spacing and Sizing

Inherited from Aurora -- 4px base grid:

```css
:root {
    --space-1:   4px;    --space-2:   8px;
    --space-3:  12px;    --space-4:  16px;
    --space-6:  24px;    --space-8:  32px;
    --space-12: 48px;    --space-16: 64px;
}
```

**Vault-specific sizing:**

| Element | Size | Rationale |
|---------|------|-----------|
| Tree item height | 40px | Touch-friendly, clear hit target |
| Tree indent | 24px per level | Visible hierarchy without excessive width |
| File icon | 20x20px | Crisp at 1x and 2x |
| Action button | min 44x44px | WCAG touch target (upgraded from 48px for denser layouts) |
| Breadcrumb segment | min-height 32px | Comfortable tap target |

### 1.5 Icons and Visual Language

**Principle:** Use abstract geometric shapes, not culturally loaded metaphors.

| Concept | Icon | Avoid |
|---------|------|-------|
| Locked vault | Shield with keyhole outline | Padlock (Western-centric), safe (industrial) |
| Unlocked vault | Shield with check mark | Open padlock |
| Folder | Rounded rectangle with subtle fold | Manila folder (US office culture) |
| File | Document rectangle with corner fold | This is universal enough |
| Upload | Upward arrow entering a horizontal line | Cloud with arrow (implies cloud trust) |
| Download | Downward arrow from a horizontal line | Cloud with arrow |
| Encrypt | Concentric circles / ripple | Lock (redundant with vault metaphor) |
| Delete | X mark | Trash can (varies culturally) |
| Version | Clock with reverse arrow | -- |
| Share | Two connected nodes | Chain link (implies restriction) |
| Language | Globe outline | Flags (one flag != one language) |

**Icon implementation:** Inline SVG, no icon font. Each icon is a simple `<svg>` element, 20x20 viewbox, stroke-based at 1.5px weight. Colour inherits from `currentColor`. This ensures icons are:
- Accessible (not reliant on font rendering)
- RTL-safe (SVGs mirror with CSS `transform: scaleX(-1)` where needed)
- Crisp at all densities
- Translatable (no embedded text)

### 1.6 Vault Identity: Distinct but Consistent

vault.sgraph.ai uses the same Aurora theme as sgraph.ai but signals "vault" identity through:

1. **Shield icon** in the header (replacing the SG/ slash animation)
2. **Vault-specific gradient** on the entry screen: subtle radial gradient from `#1A1A2E` (center) to `#0F0F1E` (edges) -- slightly darker, more enclosed feeling
3. **Brand mark:** `SG/Vault` using the established brand pattern (`SG` + teal `/` + `Vault`)
4. **No separate colour scheme** -- same teal/navy/coral. Consistency builds trust.

## 3. Multi-Cultural Design Considerations

### 3.1 Text Expansion Budget

German and French text is 30--40% longer than equivalent English. The UI must accommodate this without breaking layouts.

| English | German | Expansion |
|---------|--------|-----------|
| Upload | Hochladen | +47% |
| Encrypt & Upload | Verschlusseln & Hochladen | +92% |
| Decryption key | Entschlusselungsschlussel | +108% |
| 3 files, 1.5 MB | 3 Dateien, 1,5 MB | +35% |

**Design rules:**

1. **No fixed-width buttons.** All buttons use `min-width` + `padding`, never `width`.
2. **Flexible labels.** File browser column headers truncate with ellipsis if needed.
3. **Test with German strings.** German is the worst case for Latin-script expansion. If the layout works in German, it works for all Latin languages.
4. **Action bar wraps.** The header action buttons (`[Upload] [+Folder] [Share]`) wrap to a second line on narrow screens with long labels.
5. **Breadcrumbs truncate middle segments.** `/documents/reports/2026-Q1/drafts/` becomes `/documents/.../drafts/` when space is tight.

### 3.2 RTL Support

Arabic and Hebrew are right-to-left. The vault must support RTL from day one in its CSS architecture, even if Arabic/Hebrew locale files are not yet available.

**Implementation:**

```css
/* All layout properties use CSS Logical Properties */

/* WRONG (physical, breaks in RTL): */
.tree-item { margin-left: 24px; padding-left: 12px; text-align: left; }

/* CORRECT (logical, works in LTR and RTL): */
.tree-item { margin-inline-start: 24px; padding-inline-start: 12px; text-align: start; }
```

**Logical property mapping:**

| Physical (do not use) | Logical (use this) |
|----------------------|-------------------|
| `margin-left` | `margin-inline-start` |
| `margin-right` | `margin-inline-end` |
| `padding-left` | `padding-inline-start` |
| `padding-right` | `padding-inline-end` |
| `text-align: left` | `text-align: start` |
| `text-align: right` | `text-align: end` |
| `float: left` | `float: inline-start` (or use flexbox) |
| `border-left` | `border-inline-start` |
| `left: 0` | `inset-inline-start: 0` |

**HTML `dir` attribute:** Set `<html dir="rtl" lang="ar">` when an RTL locale is active. The CSS logical properties handle the rest.

**Flexbox direction:** Do NOT set `flex-direction: row-reverse` for RTL. Flexbox already respects `dir="rtl"` on the parent element when using `flex-direction: row`.

**Icons that imply direction** (arrows, chevrons) must mirror in RTL:

```css
[dir="rtl"] .icon-arrow-forward {
    transform: scaleX(-1);
}
```

### 3.3 Number and Date Formatting

**File sizes:**

```javascript
function formatFileSize(bytes, locale) {
    const units = ['B', 'KB', 'MB', 'GB'];
    let i = 0;
    let size = bytes;
    while (size >= 1024 && i < units.length - 1) {
        size /= 1024;
        i++;
    }
    return new Intl.NumberFormat(locale, {
        maximumFractionDigits: i === 0 ? 0 : 1
    }).format(size) + ' ' + units[i];
}

// Examples:
// formatFileSize(245000, 'en-gb')  -> "239.3 KB"
// formatFileSize(245000, 'de-de')  -> "239,3 KB"
// formatFileSize(245000, 'fr-fr')  -> "239,3 KB"
```

**Dates:**

```javascript
function formatDate(isoDate, locale) {
    return new Intl.DateTimeFormat(locale, {
        day: 'numeric',
        month: 'short',
        year: 'numeric'
    }).format(new Date(isoDate));
}

// Examples:
// formatDate('2026-03-02', 'en-gb')  -> "2 Mar 2026"
// formatDate('2026-03-02', 'de-de')  -> "2. Marz 2026"
// formatDate('2026-03-02', 'pt-br')  -> "2 de mar. de 2026"
// formatDate('2026-03-02', 'ja')     -> "2026/3/2"
```

**Relative time (for "modified X ago"):**

```javascript
function formatRelativeTime(isoDate, locale) {
    const rtf = new Intl.RelativeTimeFormat(locale, { numeric: 'auto' });
    const diff = Date.now() - new Date(isoDate).getTime();
    const hours = Math.floor(diff / 3600000);
    if (hours < 24) return rtf.format(-hours, 'hour');
    const days = Math.floor(hours / 24);
    if (days < 30) return rtf.format(-days, 'day');
    return formatDate(isoDate, locale);
}
```

### 3.4 Cultural Colour Associations

| Colour | Western | East Asian | Middle Eastern | Decision |
|--------|---------|-----------|----------------|----------|
| Red | Danger, stop | Good luck, prosperity (China) | Danger | Use only for errors/destructive actions. Never for success. |
| Green | Go, success, eco | Youth, nature (Japan) | Islam, paradise | Avoided as primary. Using teal instead. |
| Teal | Modern, trust | Calm, technology | Neutral | Safe as accent colour across all cultures. |
| White | Purity, clean | Mourning (some contexts) | Purity | Used sparingly; vault is dark-themed. |
| Blue | Trust, corporate | Trust, technology | Truth, heaven | Deep navy background -- universally safe. |

**Design rule:** The vault uses only navy (neutral), teal (neutral), and coral (error-only). These carry minimal cultural baggage.

### 3.5 Iconography Guidelines

**Avoid:**

- **Mailbox** (US rural mailbox is not universal)
- **Thumbs up/down** (offensive in some Middle Eastern cultures)
- **Hand gestures** generally (vary wildly by culture)
- **Religious symbols** (cross, crescent, star) for any functional purpose
- **Animal metaphors** (pigs in Islamic culture, cows in Hindu culture)
- **Country flags** for language selection
- **OK hand gesture** (offensive in Brazil and other cultures)

**Prefer:**

- **Abstract geometric shapes** (shield, circle, square, arrow)
- **Universal symbols** (magnifying glass for search, X for close, + for add)
- **Directional arrows** (mirrored automatically in RTL via CSS)
- **Simple document/folder shapes** (universally understood from digital literacy)

### 3.6 Language Selector Design

**The language selector uses language names in their native script, not flags.**

Rationale:
- One flag does not equal one language (Spanish: Spain, Mexico, Argentina, etc.)
- Some flags are politically sensitive (Taiwan, Palestine, etc.)
- Flags discriminate against diaspora speakers

**Design:**

```
Desktop:                          Mobile:
[Globe icon] English (UK) [v]    [Globe] EN-GB [v]
```

Clicking opens a grid dropdown (same pattern as sgraph.ai and send.sgraph.ai):

```
+--------------------------------------------+
| English (UK)          | English (US)        |
| Deutsch (Deutschland) | Deutsch (Schweiz)   |
| Espanol (Espana)      | Espanol (Mexico)    |
| Francais (France)     | Francais (Canada)   |
| Italiano (Italia)     | Nederlands          |
| Polski (Polska)       | Portugues (Brasil)  |
| Portugues (Portugal)  | Romana (Romania)    |
| Hrvatski (Hrvatska)   | tlhIngan Hol        |
+--------------------------------------------+
```

**Each language name is written in its own script.** Arabic would appear as "العربية", Japanese as "日本語", etc.

The active language is highlighted with the teal accent colour.

## 4. i18n String Architecture for Vault

### 4.1 Locale JSON Structure

Vault locale files follow the same flat key-value pattern used in the existing Send UI (`sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/i18n/`), using dot-separated namespaces.

**Namespace conventions:**

```
vault.{section}.{element}        -- vault-specific strings
```

### 4.3 Pluralisation Rules

Different languages have different plural forms. English has 2 (singular, plural). Arabic has 6.

**Implementation:** Use ICU MessageFormat-style plural rules:

```json
{
    "vault.browser.files_count": "{count, plural, =0 {No files} one {1 file} other {{count} files}}"
}
```

However, for the MVP, we use the simpler approach already established in the Send UI: separate keys per plural form, with the JavaScript code selecting the correct one based on count.

```json
{
    "vault.browser.files_count_zero": "No files",
    "vault.browser.files_count_one":  "1 file",
    "vault.browser.files_count":      "{count} files"
}
```

The JavaScript i18n helper selects the correct key:

```javascript
function pluralise(baseKey, count, locale) {
    if (count === 0) return t(baseKey + '_zero') || t(baseKey).replace('{count}', '0');
    if (count === 1) return t(baseKey + '_one')  || t(baseKey).replace('{count}', '1');
    return t(baseKey).replace('{count}', new Intl.NumberFormat(locale).format(count));
}
```

**Phase 2 enhancement:** When Arabic, Russian, or Polish locales are added (languages with complex plural rules), implement full CLDR plural categories (zero, one, two, few, many, other).

### 4.5 i18n Runtime Architecture

The vault uses the same runtime approach as the v0.2.0 Send UI:

1. **Locale detection:** URL path prefix (`/en-gb/`, `/de-de/`, etc.) or `localStorage` preference
2. **String loading:** JSON file fetched at startup (small -- ~3KB per locale)
3. **DOM application:** `data-i18n` attributes on elements, processed by `I18n.apply()`
4. **Dynamic strings:** JavaScript calls `I18n.t('key', { param: value })` for strings with interpolation
5. **FOUC prevention:** `body.i18n-loading { visibility: hidden }` until strings are applied

```javascript
// Vault i18n initialisation
const VaultI18n = {
    locale: 'en-gb',
    strings: {},

    async init() {
        this.locale = this.detectLocale();
        const response = await fetch(`/i18n/${this.locale}.json`);
        this.strings = await response.json();
        this.apply();
        document.body.classList.remove('i18n-loading');
    },

    t(key, params = {}) {
        let str = this.strings[key] || key;
        for (const [k, v] of Object.entries(params)) {
            str = str.replace(`{${k}}`, v);
        }
        return str;
    },

    apply() {
        document.querySelectorAll('[data-i18n]').forEach(el => {
            const key = el.getAttribute('data-i18n');
            el.textContent = this.t(key);
        });
        document.querySelectorAll('[data-i18n-placeholder]').forEach(el => {
            el.placeholder = this.t(el.getAttribute('data-i18n-placeholder'));
        });
        document.querySelectorAll('[data-i18n-aria-label]').forEach(el => {
            el.setAttribute('aria-label', this.t(el.getAttribute('data-i18n-aria-label')));
        });
    },

    detectLocale() {
        // 1. URL path prefix
        const path = window.location.pathname;
        const match = path.match(/^\/([a-z]{2}(?:-[a-z]{2})?)(?:\/|$)/);
        if (match && this.isSupported(match[1])) return match[1];
        // 2. localStorage preference
        const saved = localStorage.getItem('vault-locale');
        if (saved && this.isSupported(saved)) return saved;
        // 3. Browser preference
        const browserLang = navigator.language.toLowerCase();
        if (this.isSupported(browserLang)) return browserLang;
        // 4. Default
        return 'en-gb';
    }
};
```

## 5. Responsive Design Strategy

### 5.1 Mobile-First Approach

All CSS is written mobile-first. Desktop layouts are added via `min-width` media queries.

### 5.2 Breakpoints

| Breakpoint | Token | Target |
|-----------|-------|--------|
| `< 480px` | `--bp-mobile` | Small phones (iPhone SE, Galaxy S) |
| `480px -- 767px` | `--bp-mobile-lg` | Large phones, small tablets |
| `768px -- 1023px` | `--bp-tablet` | Tablets, small laptops |
| `1024px -- 1279px` | `--bp-desktop` | Laptops |
| `>= 1280px` | `--bp-desktop-lg` | Large screens |

```css
/* Mobile-first: base styles are mobile */

/* Tablet and up */
@media (min-width: 768px) { ... }

/* Desktop and up */
@media (min-width: 1024px) { ... }

/* Large desktop */
@media (min-width: 1280px) { ... }
```

### 5.3 Layout Adaptation by Screen

| Screen | Mobile (< 768px) | Tablet (768--1023px) | Desktop (1024px+) |
|--------|-----------------|---------------------|-------------------|
| **Entry** | Single column, stacked form | Single column, wider card | Single column, max-width 520px centered |
| **Browser** | List only, tree in bottom drawer | Tree sidebar (200px) + list | Tree (220px) + list + detail panel (280px) |
| **Viewer** | Full screen, info in bottom sheet | Full screen, info bar bottom | Full screen, info bar bottom |
| **Upload** | Inline card, full width | Inline card, full width | Inline card, max-width 640px |

### 5.4 Tree Navigation Adaptation

**Desktop (1024px+):** Persistent sidebar, always visible. Tree is collapsible at the folder level.

**Tablet (768--1023px):** Collapsible sidebar. Toggle button in header shows/hides tree. Starts hidden if vault has fewer than 3 folders.

**Mobile (< 768px):** Tree is in a bottom-sheet drawer, triggered by a `[Folders]` button in the bottom action bar. The drawer slides up from the bottom, covering 60% of the screen. Tapping a folder navigates and closes the drawer.

### 5.5 Touch Targets

All interactive elements have a minimum touch target of 44x44px (WCAG 2.5.8 Target Size Enhanced is 44px; the older 48px recommendation is aspirational but 44px is the auditable standard).

```css
/* Minimum touch target */
.touch-target {
    min-height: 44px;
    min-width: 44px;
    display: flex;
    align-items: center;
}

/* File list items on mobile */
.file-item {
    min-height: 48px;   /* slightly larger for comfort */
    padding: var(--space-3) var(--space-4);
}
```

### 5.6 Drag-and-Drop on Mobile

Drag-and-drop is not reliable on mobile browsers. The upload zone on mobile shows a prominent "Tap to browse" button instead. The drag-drop affordance (dashed border, "Drop files here" text) is hidden below 768px.

```css
}

@media (min-width: 768px) {
    .drop-zone__drag-hint { display: block; }
    .drop-zone__tap-hint  { display: none; }
}
```

### 6.4 Colour Contrast

All text and interactive elements must meet WCAG 2.1 AA contrast ratios (4.5:1 for normal text, 3:1 for large text and UI components).

| Element | Foreground | Background | Ratio | Pass |
|---------|-----------|-----------|-------|------|
| Body text | `#E0E0E0` | `#1A1A2E` | 11.3:1 | AA |
| Secondary text | `#8892A0` | `#1A1A2E` | 5.1:1 | AA |
| Teal on navy | `#4ECDC4` | `#1A1A2E` | 8.1:1 | AA |
| Teal on surface | `#4ECDC4` | `#1E2A4A` | 6.7:1 | AA |
| Coral on navy | `#E94560` | `#1A1A2E` | 5.4:1 | AA |
| Button text on teal | `#1A1A2E` | `#4ECDC4` | 8.1:1 | AA |
| Muted text | `#5a6478` | `#1A1A2E` | 2.5:1 | FAIL |

**Action required:** The muted text colour (`#5a6478`) fails contrast on navy background. This colour should only be used for decorative/non-essential content. All functional text must use `#8892A0` or lighter.

### 6.5 Focus Indicators

All interactive elements must have visible focus indicators that meet WCAG 2.4.7 (Focus Visible):

```css
:focus-visible {
    outline: 2px solid var(--accent);
    outline-offset: 2px;
}

/* Remove outline for mouse users */
:focus:not(:focus-visible) {
    outline: none;
}
```

The teal focus ring on dark navy background provides a 8.1:1 contrast ratio, well above the 3:1 minimum for non-text content.

### 6.6 Reduced Motion

Users who prefer reduced motion should not see animations:

```css
@media (prefers-reduced-motion: reduce) {
    *,
    *::before,
    *::after {
        animation-duration: 0.01ms !important;
        animation-iteration-count: 1 !important;
        transition-duration: 0.01ms !important;
    }
}
```

This affects: progress bar animations, modal open/close transitions, tree expand/collapse animations, drag-drop hover effects.

## Summary of Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| Adopt Aurora theme unchanged | Brand consistency across sgraph.ai properties |
| No flags in language selector | Flags exclude diaspora speakers, politically sensitive |
| CSS Logical Properties throughout | RTL support built into the CSS architecture from day one |
| `Intl.NumberFormat` / `Intl.DateTimeFormat` | Locale-aware formatting without bundling locale data |
| System-ui font fallback for non-Latin | Avoid bundling multi-megabyte font files; system fonts handle CJK/Arabic well |
| Three-panel layout collapsing to one | Progressive disclosure: show more on larger screens |
| Inline upload (not modal) | Upload is a core action, not an interruption |
| Four-step progress (read/encrypt/upload/tree) | Makes encryption visible, builds trust |
| Flat JSON key-value locale files | Matches existing Send UI pattern, easy for translators |
| WAI-ARIA TreeView pattern | Standard tree navigation for screen readers |
| 44px minimum touch targets | WCAG 2.5.8 compliance |
| Mobile: tree in bottom-sheet drawer | Saves space, familiar mobile pattern |
| No red for success anywhere | Avoids cultural confusion (red = luck in China) |
