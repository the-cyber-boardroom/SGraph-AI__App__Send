## Executive Summary

The redesign proposal is structurally sound. The information architecture changes -- actions first, explanations second -- align with fundamental design principles (form follows function, hierarchy reflects priority). The proposal preserves the existing visual language and does not require a visual redesign, which is the correct call: the Aurora dark theme, teal accent system, and typographic hierarchy are strong.

This review identifies seven areas where the proposal needs design refinement before implementation. None of them challenge the proposal's direction. All of them address execution details that will determine whether the restructured pages feel as considered as the current ones, or feel hastily rearranged.

## 1. Visual Hierarchy Assessment: CTA Button Swap

### What the proposal changes

The hero CTA pair flips from:

```
Current:   [Get Started Free ->]  (btn-primary)   [See Pricing ->]  (btn-secondary)
Proposed:  [Buy Credits -- L5 ->] (btn-primary)   [Try Free ->]     (btn-secondary)
```

### Design evaluation

**The swap is correct.** The design system's `btn-primary` class uses a filled teal background (`background-color: var(--accent); color: var(--color-navy)`) while `btn-secondary` uses a transparent background with teal outline (`border-color: var(--accent); color: var(--accent)`). In the Aurora dark theme, the filled teal button has substantially more visual weight against the `--bg-primary` (#1A1A2E) background. Giving that visual weight to the commercially primary action (Buy Credits) is the correct hierarchy decision.

**Left position is correct.** In left-to-right reading cultures, the left-positioned element in a horizontal pair is scanned first. The current `.hero-ctas` uses `display: flex; justify-content: center; gap: var(--sp-4)`, which places the first child on the left. Placing "Buy Credits" first in the DOM gives it both positional and visual primacy.

**Label specificity is an improvement.** "Buy Credits -- L5" communicates price, action, and commitment level in three words. The current "Get Started Free" is ambiguous about what happens next (Rams: good design is honest). The proposed label removes the affordance gap identified in the UX brief.

### Findings

**Finding D1: The "Buy Credits -- L5" label may be too long for mobile.** At 480px and below, the `.hero-ctas` stacks vertically (`flex-direction: column; width: 100%`), so both buttons become full-width. The label "Buy Credits -- L5 ->" will render fine. However, if both buttons are full-width, their visual weight becomes equal (same width, same padding). The primary/secondary distinction is preserved only through fill vs. outline treatment. This is sufficient -- the `btn-primary` filled teal will still dominate against the `btn-secondary` outline -- but consider whether the mobile stack order should also place "Buy Credits" first (above). Currently, flex column ordering preserves DOM order, so this will work automatically if "Buy Credits" is the first child.

**Recommendation:** No CSS change needed. Ensure the "Buy Credits" `<a>` element comes first in the DOM, which the proposal already specifies.

**Finding D2: The em dash in "Buy Credits -- L5" should be a proper em dash.** The proposal shows "Buy Credits -- L5" with a double hyphen. The existing codebase uses HTML entities (the cloud page uses `&mdash;` in `Buy Credits &mdash; &pound;5`). Use `Buy Credits &mdash; &pound;5 &rarr;` for consistency with the existing `/pricing/cloud/` CTA and for typographic correctness. The double hyphen is a design detail that communicates carelessness, however small.

## 2. Inline Form Design: Form on the Homepage

### What the proposal adds

A signup form (Name + Email + Submit) appears inline on the homepage, inside a card/panel, between the hero and the Zero-Knowledge Encryption section.

### Design evaluation against the existing design system

The design system already has well-defined form elements:

- `.form-group` for field containers (margin-bottom: `var(--sp-6)`)
- `.form-label` for labels (color: `var(--text-secondary)`, font-size: 0.9rem)
- `.form-input` for text inputs (background: `rgba(22, 33, 62, 0.8)`, border: `1px solid var(--border-color)`)
- `.form-input:focus` for focus state (border-color: `var(--accent)`, box-shadow with teal glow)

These form styles were designed for the dark theme. The input background (`rgba(22, 33, 62, 0.8)`) is slightly darker than `--bg-surface` (`rgba(44, 62, 107, 0.35)`), which creates appropriate contrast. If the form is placed inside a `.card` (which uses `--bg-surface`), the input fields will read as recessed regions within the card -- this is the correct visual hierarchy: card surface > input well.

### Findings

**Finding D3: The form MUST be contained in a card or panel.** The proposal shows the form inside a bordered rectangle, which is correct. Without a containing card, the form fields would sit directly on the page background, and at two fields (Name + Email), the form has insufficient visual weight to constitute a standalone section. A `.card` wrapper provides:

- Visual containment via `background-color: var(--bg-surface)` and `border: 1px solid var(--border-color)`
- The border-radius (`var(--radius-card)`: 12px) that groups the form elements as a unit
- The hover state (`border-color: var(--accent)`) that signals interactivity

**Recommendation:** Use a `.card` wrapper with a max-width constraint. The form card should not stretch to the full `--max-width` (1200px). A two-field form at 1200px width wastes enormous horizontal space. Use `max-width: 520px; margin-inline: auto;` -- this matches the `.modal` max-width (520px) and creates a proportional relationship between the form and the hero text above it (which is constrained to 640px by `.hero .lead`).

**Finding D4: Visual weight competition between the form section and the hero.** The hero section uses `padding-block: var(--sp-24) var(--sp-16)` (96px top, 64px bottom), creating generous breathing room. If the "Try Free" form section immediately follows the hero, the page needs a clear visual break. The proposal shows a dashed separator (`--- -- -- -- --`), but the design system does not have a dashed separator. The design system offers two options for visual breaks:

1. **Section alternation** (`section--alt`), which changes the background from `--bg-primary` (#1A1A2E) to `--bg-secondary` (#16213E). This is the pattern used throughout the current site.
2. **A `.divider`** (horizontal rule with `border-top: 1px solid var(--border-color)`).

**Recommendation:** Place the "Try Free" form section in a `section--alt` container. This provides the background colour shift that visually separates the form from the hero without adding a new design element. The alternating dark navy / deep blue rhythm is already the site's established pattern: on the current homepage, Hero (bg-primary) -> Zero-Knowledge (section--alt) -> Get Started (section) -> Built by AI (section--alt). The proposed structure should follow the same rhythm: Hero (bg-primary) -> Try Free (section--alt) -> Zero-Knowledge (section) -> Built by AI (section--alt) -> Deploy Anywhere (section).

**Finding D5: Form label placement.** The proposal shows labels inline with input fields: `Name: [________]`. The design system's `.form-label` is `display: block`, meaning labels stack above inputs. This is the correct pattern for the dark theme -- side-by-side label+input pairs require careful width management and break unpredictably on mobile. Stacked labels are also more accessible (labels above inputs have higher completion rates in form usability research).

**Recommendation:** Use the design system's stacked label pattern, not the inline label pattern shown in the proposal wireframe. The wireframe's inline layout (`Name: [____]`) is a wireframing shorthand, not a design specification. Implementation should use:

```html
<div class="form-group">
  <label class="form-label" for="name">Name</label>
  <input class="form-input" type="text" id="name" name="name">
</div>
<div class="form-group">
  <label class="form-label" for="email">Email</label>
  <input class="form-input" type="email" id="email" name="email">
</div>
```

## 3. Information Architecture: Section Ordering

### What the proposal changes

```
Current:    Hero -> Encryption -> Get Started -> Built by AI -> Footer
Proposed:   Hero -> Try Free -> Encryption -> Built by AI -> Deploy Anywhere -> Footer
```

### Design evaluation

**The reordering is correct in principle.** Moving the actionable form above the explanatory content follows the "actions first, explanations second" principle established in the UX brief.

### Findings

**Finding D6: The "Try Free" section between hero and encryption creates a natural funnel, but needs careful section header treatment.** The hero establishes the value proposition. The "Try Free" section offers the free path. Below the fold, the encryption section provides evidence. The "Built by AI" section builds credibility. "Deploy Anywhere" provides the expansion options. This is a sound narrative arc: promise -> action -> proof -> credibility -> expansion.

However, the "Try Free" section heading must not compete with the hero. The hero uses `.display` (2.5rem on desktop, 1.85rem on mobile). Section headings use `h2` at `var(--fs-h2)` (1.6rem on desktop, 1.3rem on mobile). This size ratio (2.5:1.6 = 1.56:1) provides adequate hierarchy. The "Try Free" section heading should use the standard `.section-header h2` pattern, not any enlarged treatment.

**Recommendation:** The "Try Free" section should use the standard `section-header` centred heading pattern. The card containing the form should be the visual focal point of the section, not the heading. The heading serves as a label for the section; the card is the affordance.

**Finding D7: The "Deploy Anywhere" teaser is a good addition.** The current homepage jumps from "Built by AI" stats directly to the footer. The proposed "Deploy Anywhere" section provides a bridge to the full pricing page and communicates the open-source philosophy without the 8-card decision paralysis. This section should be lightweight -- a centered `section-header` with a text link, not another card grid.

**Recommendation:** Keep "Deploy Anywhere" to text only: heading, one-paragraph description, and a single `btn-secondary` link to `/pricing/`. No cards. The visual rhythm of the page should be: card section (Try Free) -> text+cards section (Encryption) -> stats section (Built by AI) -> text section (Deploy Anywhere). This alternation of visual density prevents fatigue.

## 4. Pricing Page Redesign: Two Primary Cards + Eight Deployment Cards

### What the proposal changes

The pricing page restructures from:

```
Current:    Philosophy -> Free Cards (4) -> Encryption Callout -> Cloud Cards (4) -> How It Works -> FAQ
Proposed:   Two Primary Cards (Free + L5) -> Philosophy -> Deployment Cards (4+4) -> How Credits Work
```

### Design evaluation

**The two primary cards at the top is the correct move.** The current pricing page buries the two actions that 90%+ of visitors need (try free, buy credits) inside two grids of four cards each, behind a philosophy section. Lifting these two out and placing them at the top creates immediate clarity.

### Findings

**Finding D8: The two primary cards MUST have different visual treatment from the eight deployment cards below.** The proposal implies this by giving the Starter Pack a "RECOMMENDED" badge, but does not specify how the two primary cards differ visually from the eight below. Without visual differentiation, the page presents ten cards that all look the same -- the hierarchy collapses.

The design system already provides the tools:

- `pricing-card--featured` adds `border-color: var(--accent)` and `box-shadow: 0 0 0 1px var(--accent), var(--shadow-card)`. Use this on the Starter Pack card.
- `pricing-card__badge` provides the "Available Now" / "Recommended" pill. Move the badge to the Starter Pack.
- For the Free card, use a standard `.pricing-card` (no `--featured` modifier) but with the inline form, which naturally differentiates it through the presence of interactive form elements.

**Size differentiation:** The two primary cards should be visually larger than the eight deployment cards. The proposal shows them side by side in a 2-column layout, while the eight deployment cards are in a 4-column layout. This size difference (each primary card is ~2x the width of a deployment card) naturally communicates hierarchy through proportion. This is correct.

**Recommendation:** Use `grid--2` (max-width: 720px, centred) for the two primary cards, matching the current homepage "Get Started" section layout. Use the existing `pricing-grid` (4-column) for the eight deployment cards below. The width difference communicates hierarchy through proportion.

**Finding D9: The inline form within the Free card needs careful responsive handling.** At desktop widths, two side-by-side cards at max 720px gives each card ~336px of internal width. After card padding (`var(--sp-6)`: 24px on each side), the content area is ~288px. A form input at 288px width is comfortable. But at tablet widths (1024px breakpoint), the `grid--2` collapses to... actually, no: the custom pricing page uses `pricing-grid` which has its own breakpoints (2-col at 1024px, 1-col at 640px). If the two primary cards use a separate `grid--2` container, they follow the main stylesheet's breakpoints: 1-col at 768px. This means:

- Desktop (>768px): Two cards side by side, form inputs at ~288px width -- adequate.
- Mobile (<768px): Cards stack vertically, form inputs take full card width -- generous.

This works well. No design system changes needed.

**Finding D10: The "RECOMMENDED" badge position needs consideration.** The existing `pricing-card__badge` uses `position: absolute; top: 0; left: 50%; transform: translate(-50%, -50%)`, which centres the badge on the top edge of the card and renders half inside, half outside. This creates a visual marker that draws the eye. Currently, this badge says "Available Now" on the Multi-Tenant card. The proposal moves it to say "Recommended" on the Starter Pack.

**Recommendation:** Keep the existing badge design. Change the label from "Available Now" to "Recommended". The badge treatment on `pricing-card--featured` (teal border + teal box-shadow + badge) creates a three-layer emphasis system: colour (teal border), depth (box-shadow), and label (badge). This is sufficient to differentiate the recommended card from the free card without over-emphasising it.

**Finding D11: The eight deployment cards should be visually subordinated.** The proposal groups them under "FREE & SELF-MANAGED" and "SG/SEND CLOUD" headings. This is correct. Additionally, the eight cards should not use the hover lift effect (`transform: translateY(-2px)`) or should use a more subtle version. This is a minor point, but it helps communicate that these cards are informational (deployment options you might explore later) rather than primary actions (the thing you should do right now).

**Recommendation:** Consider adding a `.pricing-card--secondary` modifier that reduces the hover transform to `translateY(-1px)` or removes it entirely, and reduces the border-hover from `var(--accent)` to a subtler shade. Alternatively, keep the existing hover behaviour -- it is consistent and the size differential (primary cards in 2-col, deployment cards in 4-col) already establishes hierarchy. This is a judgement call; either approach is defensible.

## 5. Consistency and Brand Coherence

### Design evaluation

**The proposal preserves the existing design system.** No new colours, no new typography, no new spatial tokens are introduced. The visual changes are purely structural (section reordering, card rearrangement, form placement). This is a strength of the proposal -- it recognises that the visual language is sound and that the problem is information architecture, not aesthetics.

### Findings

**Finding D12: Navigation reordering is a minor but correct change.** The proposal moves "Pricing" from position 4 to position 2 in the nav:

```
Current:    Product  Team  Architecture  Pricing
Proposed:   Product  Pricing  Architecture  Team
```

The current `.site-nav` uses `display: flex; gap: var(--sp-6)`. Reordering the `<a>` elements is a pure HTML change. On mobile (768px and below), the nav collapses into a vertical menu -- the order becomes top-to-bottom, where "Pricing" at position 2 is still prominent.

**Recommendation:** Implement as proposed. No CSS changes needed.

**Finding D13: Every "Buy Credits" instance must point to the same Stripe URL.** The current `/pricing/cloud/` page contains the actual Stripe link: `https://buy.stripe.com/fZu8wHdv23nN1Z85jn5os02`. The proposal's consistency principle (one label, one destination) requires that every "Buy Credits -- L5 ->" button site-wide links to this exact URL. This is a content decision, not a design decision, but from the design perspective it is a consistency requirement: the same label must produce the same result. Anything else violates affordance expectations and degrades interface trust.

**Finding D14: The "Already have a token?" link should remain in the hero sub-text.** The proposal preserves this: `Already have a token? Go to send.sgraph.ai ->`. The current implementation uses inline styling (`style="color: var(--accent)"`) rather than a class. This works but is inconsistent with the design system approach. Consider using the existing anchor styling (links default to `color: var(--accent)` in the reset) and removing the inline style.

## 6. Responsive Considerations

### Pricing page: Two primary cards

**Finding D15: The two primary cards stack correctly on mobile.** Using `grid--2` (which collapses to `grid-template-columns: 1fr` at 768px), the cards stack vertically on mobile. When stacked, the Free card (with inline form) should appear first, followed by the Starter Pack. This matches the proposal's left-to-right ordering (Free left, Starter Pack right).

However, there is a counter-argument: if "Buy Credits" is the primary CTA (commercially), should it appear first in the mobile stack? On mobile, the first card is the one visible without scrolling. If the Starter Pack card is above the Free card on mobile, the user with purchase intent can act immediately. If the Free card is first, the purchase-intent user must scroll past the form to find the buy button.

**Recommendation:** This is a product decision more than a pure design decision. Both orderings are defensible. If the Challenge campaign is the priority, put the Starter Pack first (in the DOM, which controls both left position on desktop and top position on mobile). If long-term conversion optimisation is the priority, put the Free card first (lower barrier, higher conversion to free, then upgrade later). The proposal should specify this explicitly.

### Homepage: Inline form

**Finding D16: The form card at narrow widths (320-480px) works well with the existing design system.** The `.form-input` is `width: 100%`, and the `.card` padding is `var(--sp-6)` (24px). On a 320px screen, the card content area is 320 - 16 (container padding at 480px breakpoint) - 48 (card padding) = 256px. A text input at 256px is adequate for name and email.

### Navigation on mobile

**Finding D17: The nav reorder has no responsive impact.** The mobile nav is a vertical stack. "Pricing" at position 2 means it appears second in the stack, immediately after "Product". This is good positioning for a user scanning the mobile menu.

## 7. Micro-interactions: Inline Form Confirmation

### What the proposal changes

The form submits and, instead of navigating to a new page, the form card transforms in-place to show a confirmation message:

```
BEFORE: [Name field] [Email field] [Submit button]
AFTER:  "You're in." + confirmation details + next-step links
```

### Design evaluation

This is the correct interaction pattern. No page navigation, no third-party redirect, no brand break. The user stays in context. The question is the transition between states.

### Findings

**Finding D18: Use a crossfade transition, not a slide.** The design system's transitions use `var(--transition-normal)` (250ms ease) and `var(--transition-fast)` (150ms ease). The modal system uses `opacity` and `transform: translateY` for entry/exit. For the form-to-confirmation transition, a slide would imply spatial movement (the confirmation is "somewhere else"), which contradicts the design intention (the confirmation is "right here"). A crossfade communicates replacement-in-place.

**Recommended implementation:**

```css
.form-state {
  transition: opacity var(--transition-normal);
}

.form-state--hidden {
  opacity: 0;
  position: absolute;
  pointer-events: none;
}

.form-state--visible {
  opacity: 1;
  position: relative;
}
```

The card wrapper should have `position: relative; overflow: hidden;` and a smooth `height` transition or use `min-height` to prevent layout shift. The card should not change size abruptly -- the confirmation message should be approximately the same height as the form, or the card should animate its height.

**Finding D19: The confirmation state should maintain the card's visual weight.** After submission, the card should not become visually diminished. The "You're in." heading should use `h3` styling (`var(--fs-h3)`, white, bold). The supporting text should use `--text-secondary`. The next-step links ("How it works", "Source on GitHub", "Buy credits now") should use the standard anchor style (`color: var(--accent)`).

**Finding D20: Consider a brief success indicator.** Before the crossfade to the confirmation state, display a momentary success state: the submit button could transition from its label to a checkmark icon (0.5s), then the full card crossfades (250ms). This two-step animation communicates "action acknowledged" (button feedback) then "here's what's next" (content swap). Total transition time: ~750ms. This follows the design system's existing `.alert--success` pattern (teal background + teal text) but applies it temporally rather than spatially.

**Recommendation:** The implementation should use the design system's existing `--transition-normal` (250ms) for the crossfade. No new animation keyframes are needed. The button's loading/success state can use the existing `.btn-primary` styling with a content swap. Avoid the complexity of slide animations, scale animations, or particle effects -- they would be inconsistent with the Aurora theme's restrained aesthetic.

## Additional Design Findings

**Finding D21: Page elimination is a design improvement.** The proposal eliminates `/pricing/cloud/`, `/pricing/free/`, and `/early-access/`. From a design perspective, each eliminated page removes a context switch, a page load, and a moment where the user must re-orient themselves. The content from these pages (philosophy, how credits work) is absorbed into the main pricing page. This consolidation reduces the user's navigation burden and is consistent with the principle "less, but better."

**Finding D22: The "How Credits Work" section on the pricing page should use a compact horizontal layout.** The proposal shows a single row: "1. Buy via Stripe  2. Each upload = 1 credit  3. Valid for 3 months (downloads are free)". The existing `/pricing/cloud/` page uses `.steps-grid` (3-column grid with counter-styled cards). For the pricing page, a simpler treatment is warranted -- the step cards with their numbered circles are too heavy for what is now a supporting detail rather than a primary content section.

**Recommendation:** Use a `stats-row`-style horizontal layout with three items, or a simple numbered inline list. The visual weight should be less than the deployment cards above. A single line of text with `var(--text-secondary)` colouring, separated by bullets, would be sufficient and proportionate.

**Finding D23: The existing FAQ section on the pricing page is not addressed by the proposal.** The current pricing page has a "Frequently asked questions" section with five questions. The proposal does not mention whether this section is preserved, moved, or eliminated. From a design perspective, the FAQ is valuable for conversion (it addresses objections) and should be preserved, positioned after the deployment cards and "How Credits Work" section. It is low-priority content that benefits from being at the bottom of the page.

**Recommendation:** The proposal should explicitly state that the FAQ section is preserved at the bottom of the pricing page, below "How Credits Work" and above the footer.

## Verdict

**Approve with revisions.** The proposal's structural direction is correct and well-reasoned. The information architecture changes will measurably improve the conversion paths for both the Challenge campaign and general visitors. The proposal correctly preserves the existing visual language rather than introducing a parallel redesign.

The revisions needed are implementation-level design decisions, not directional changes:

1. **High priority:** Visual differentiation between primary and deployment cards (D8), Stripe URL consistency (D13)
2. **Medium priority:** Form containment in `.card` (D3), section alternation pattern (D4), stacked form labels (D5), mobile card ordering decision (D15), crossfade transition specification (D18), FAQ preservation (D23)
3. **Low priority:** Typography entities (D2), hover effect subordination (D11), inline style cleanup (D14), confirmation state styling (D19, D20), "How Credits Work" format (D22)

No new CSS classes are strictly required. The existing design system -- `.card`, `.form-group`, `.form-input`, `.form-label`, `.btn-primary`, `.btn-secondary`, `.pricing-card`, `.pricing-card--featured`, `.pricing-card__badge`, `.grid--2`, `.pricing-grid`, `.section--alt`, `.alert--success` -- provides all the visual components needed. The only new CSS would be the form state transition (D18), which is ~10 lines.

The proposal's strongest contribution is the elimination of intermediate pages. From a design perspective, every page load is a moment where the user can lose context, lose trust, or lose motivation. Reducing the buy path from 4 clicks/4 scrolls to 1 click/0 scrolls is not just a UX improvement -- it is a design improvement. The form of the site will finally follow the function of the user's task.
