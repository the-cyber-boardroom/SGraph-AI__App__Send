# Designer Review — Page Layout Colour Schemes
**Version:** v0.20.17 | **Date:** 2026-03-30 | **Role:** Designer

---

## Brief

The `_page.json` renderer now supports `"theme": { "mode": "light" | "dark", "accent": "...", "font": "...", "density": "...", "background": "..." }`. A real-world vault ("OWASP London: Presentation Materials") revealed two problems:

1. **Light mode text is pale/unreadable** — section titles, body text, captions all render in grey because the Browse shell's dark CSS variables bleed through.
2. **No defined colour system for authors** — authors are guessing `accent` values and `background` values with no palette guidance.

The immediate font-colour bug has been patched (explicit hard-coded overrides on `.plr-page--light`). What remains is a **coherent design system** for the renderer: named colour schemes, typography guidance, print-safe recommendations.

---

## What Needs Designing

### 1. Named colour schemes (themes)

Authors currently write arbitrary hex codes. We need **4–6 named schemes** they can use as starting points. Each scheme must work:
- On-screen (light mode, rendered in the Browse shell)
- On-screen (dark mode, rendered in the Browse shell)
- In **present mode** (white overlay, full-screen)
- In **print / Save as PDF** (no background graphics, forced white)

Proposed named schemes to design:

| Name | Intended feel | Use case |
|------|--------------|----------|
| `default` | Clean, neutral, professional | General content |
| `navy` | Deep navy + white + teal accent | Investor decks, formal reports |
| `slate` | Warm charcoal + off-white | Articles, case studies |
| `minimal` | Pure white, black text, no accent | Print-first content |
| `brand` | SG/Send teal `#4ecdc4` accent | SG/Send internal materials |
| `dark-deck` | Dark hero + white card sections | Presentation-style |

### 3. Typography scale

The renderer supports `"font": "sans" | "serif" | "mono" | "system"`. For each font stack:
- Recommend which content types suit it
- Note whether it prints cleanly (serif is generally better for long text in print)

Current sizes (not to change, just to document):

| Element | Size | Weight |
|---------|------|--------|
| Section title | 1.35rem | 600 |
| Body text | 0.95rem | 400 |
| Card title | 1rem | 600 |
| Card description | 0.875rem | 400 |
| Caption | 0.8rem | 400 |
| Nav link | 0.85rem | 500 |

### 4. Print-specific guidance

The present mode forces `background: #ffffff`. The hero component with a dark `color` prop will print as dark (with `print-color-adjust: exact`). Design guidance needed:

- Should the default `hero.color` be changed from `#0d1117` to something lighter when no image is present?
- Or should `@media print` force hero backgrounds to a light fallback?
- Recommendation: a "print-safe hero" pattern (e.g., teal accent bar instead of full dark block)

### 5. Section `background: "alt"` colour

Currently `"alt"` resolves to `rgba(0,0,0,0.02)` on light, `rgba(255,255,255,0.03)` on dark. This is nearly invisible. Recommend a more distinct alternating shade that still prints cleanly.

## Current Bugs Fixed (context for the Designer)

These are already patched — do not re-introduce them:

- Light mode text now uses hard-coded dark values (`#1a1a2e`, `#2c2c3a`, `#666`) not CSS variables
- Source view always renders on white `#fdfdfd` regardless of page theme
- Text selection uses `#1a73e8` highlight on light pages, `#4ecdc4` on dark
