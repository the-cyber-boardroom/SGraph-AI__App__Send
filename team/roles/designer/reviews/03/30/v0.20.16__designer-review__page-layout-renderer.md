# Designer Review — Page Layout Renderer (`_page.json`)
**Version:** v0.20.16
**Date:** 2026-03-30
**Reviewer:** Designer role
**Scope:** `page-layout-renderer.js`, `send-browse-v031--page-layout.js`, `send-browse-v031--page-layout.css`

---

## Summary

The `_page.json` layout rendering system is a well-structured first implementation that successfully delivers a rich, document-style reading experience inside the Browse component. The hero / sticky-nav / section pattern is immediately legible and mirrors established editorial web conventions. The lightbox, keyboard navigation, and IntersectionObserver-driven active-link highlighting are strong touches that exceed what most MVP implementations provide.

The main design debt falls into three categories: (1) a theme mismatch between the light-on-white page renderer and the dark shell it lives inside, (2) gallery/slide sizing constraints that cap visual impact, and (3) a handful of micro-interaction gaps (thumbnail captions, loading states, scroll root wiring).

---

## What Is Working Well

**1. Hero component.**
The `plr-hero` pattern — dark background, bottom-aligned overlay, gradient scrim (`linear-gradient(to top, rgba(0,0,0,0.72) 0%, rgba(0,0,0,0.15) 100%)`) — is visually effective. The `plr-hero--has-image` modifier correctly increases `min-height` from 240 px to 300 px for photo heroes. The Cryptographic Blueprint page in particular makes good use of the full-bleed photo + gradient text treatment.

**2. Sticky nav with active-link tracking.**
`plr-nav` uses `position: sticky; top: 0; z-index: 100` combined with `backdrop-filter: blur(4px)` — exactly the right implementation for a content panel that scrolls. The `IntersectionObserver` approach in `_renderNav` for active-link highlighting is correct and efficient. `scrollbar-width: none` keeps the overflow-x bar hidden without disrupting scroll.

**3. Lightbox.**
Full keyboard support (`ArrowLeft`, `ArrowRight`, `Escape`), click-outside-to-dismiss, `aria-modal="true"`, and proper blob-URL cleanup via `_objectUrls` tracking. The `plr-lightbox__btn` positioning (`position: fixed; top: 50%; transform: translateY(-50%)`) keeps arrows visible regardless of image aspect ratio.

**4. Deduplication via `_openTabs` tab key.**
The `__page__${folderPath}` key prevents duplicate tabs on repeated folder header clicks — a UX detail that matters heavily in a multi-tab layout.

**5. Clean component dispatch.**
`_renderComponent` with a `try/catch` per component means a broken image or malformed PDF does not crash the whole page render. Silent skip (`return null`) is the right behaviour for a viewer.

**6. Tab naming and 📄 icon.**
Using the folder's last path segment as the tab title (falling back to `this.fileName || 'Home'`) keeps the tab bar readable when multiple pack pages are open simultaneously.

## Improvement Proposals

### P1 — High impact, affects first impression

#### P1-A: Theme mismatch — nav bar renders light on dark shell

**Problem:** `.plr-nav` uses `background: rgba(255, 255, 255, 0.96)` — a near-white bar. The Browse shell (`sb-container`) and the surrounding sg-layout chrome are dark theme (background `#0a0e17` / `#111827`). When the sticky nav appears at the top of the panel it creates a hard white stripe that visually ruptures the chrome.

**Proposal:** Replace the nav background with a dark/translucent variant that matches the shell, or make it conditional on a data attribute the `_page.json` can set:

```css
/* Replace current light rule: */
.plr-nav {
    background: rgba(18, 20, 32, 0.96);   /* dark shell match */
    border-bottom: 1px solid rgba(255, 255, 255, 0.08);
}
.plr-nav__link {
    color: rgba(255, 255, 255, 0.65);
}
.plr-nav__link:hover {
    color: #ffffff;
}
.plr-nav__link--active {
    color: var(--color-primary, #4ecdc4);
    border-bottom-color: var(--color-primary, #4ecdc4);
}
```

Alternatively, expose a `theme` field on the page JSON (`"theme": "dark"`) and add a `.plr-page--dark` modifier class that overrides `.plr-nav`, `.plr-section`, and `.plr-text` for the dark case. This gives the pack author control without coupling the renderer to a single colour scheme.

**Effort:** ~15 lines CSS + 3 lines JS to propagate the modifier class.

---

#### P1-B: `plr-scroll-wrapper` has no explicit height — sticky nav may not stick

**Problem:** `.plr-scroll-wrapper` is defined as `flex: 1; width: 100%` but has no `overflow-y: auto` and no explicit height. Sticky positioning requires the scrolling ancestor to be the element with `overflow: auto/scroll`. The actual scroll is happening on the panel element (`el.style.cssText = 'overflow: auto; height: 100%;'` in `_openFolderPage`), which is the *parent* of `.plr-page` / `.plr-scroll-wrapper`. Because the sticky ancestor is the panel `div` (not `.plr-scroll-wrapper`), the nav *will* stick, but the `IntersectionObserver` in `_renderNav` is constructed with `{ root: scrollRoot }` where `scrollRoot` is the `.plr-scroll-wrapper` — which is *not* the element that actually scrolls. This means section highlighting may never fire.

**Proposal:** Either:
- Pass the panel `el` as the scroll root to `PageLayoutRenderer.render()` (i.e. add a 6th argument `scrollRoot` or detect it from `container.parentElement`), or
- Give `.plr-scroll-wrapper` its own `overflow-y: auto; height: 100%` and remove `overflow: auto` from the panel element in `_openFolderPage`.

The second option is simpler and keeps the renderer self-contained:

```css
.plr-scroll-wrapper {
    flex: 1;
    width: 100%;
    overflow-y: auto;
    height: 100%;
}
```

And in `_openFolderPage`, change the panel style to `overflow: hidden; height: 100%` so the scroll root is the wrapper, not the outer div.

**Effort:** 2-line CSS change + 1-line JS change in `_openFolderPage`.

---

#### P1-C: Gallery thumbnail captions are invisible

**Problem:** `_renderGallery` sets `img.alt` and `img.title` from `captions[i]`, but the CSS for `.plr-gallery__thumb` has no visible caption element. On the Articles hub screenshot the 7×2 grid of slides shows no text under the thumbnails. The `title` tooltip only appears on hover after a delay and is not accessible.

**Proposal:** Add an optional visible caption below each thumb. In `_renderGallery`, after appending the `img`, conditionally append a `<p class="plr-gallery__thumb__caption">` when `captions[i]` is non-empty:

```css
.plr-gallery__thumb__caption {
    font-size: 0.72rem;
    color: var(--color-text-secondary, #888);
    text-align: center;
    padding: 0.25rem 0.25rem 0;
    white-space: nowrap;
    overflow: hidden;
    text-overflow: ellipsis;
    margin: 0;
}
```

Set `aspect-ratio` only on the `img` wrapper inner div rather than the `.plr-gallery__thumb` so the caption sits outside the crop box.

**Effort:** ~8 lines JS + 8 lines CSS.

### P2 — Medium impact, affects sustained use

#### P2-A: Hero min-height is fixed — no responsive scaling

`.plr-hero` has `min-height: 240px` and `.plr-hero--has-image` has `min-height: 300px`. At narrow panel widths (when the file tree takes significant horizontal space, or on a 1280-wide laptop) 300 px is a large fraction of visible height. At wide panel widths the hero feels low relative to its font size.

**Proposal:** Use a `clamp()` or `vw`-based approach:

```css
.plr-hero          { min-height: clamp(180px, 22vw, 320px); }
.plr-hero--has-image { min-height: clamp(220px, 28vw, 420px); }
```

This scales gracefully between compact and expansive without media queries.

---

#### P2-B: `plr-section` max-width (960 px) conflicts with full-bleed gallery intent

**Problem:** `.plr-section { max-width: 960px; margin: 0 auto; }` constrains all section content to a centred column. This works well for text and bullet lists. But when a `gallery` component lives inside a section, the gallery grid is also capped at 960 px — on wide screens this leaves significant whitespace either side. On the Cryptographic Blueprint screenshot the 14-slide 7×2 grid would benefit from filling more of the available panel width.

**Proposal:** Add a `full-bleed` option on the section component and a corresponding CSS modifier:

```css
.plr-section--full-bleed {
    max-width: 100%;
    padding-left: 1rem;
    padding-right: 1rem;
}
```

In `_renderSection`, check for `props.fullBleed` and add the modifier class. This requires no change to existing `_page.json` files — it is opt-in.

---

#### P2-C: Slides component nav is above the image (counter-intuitive)

**Problem:** In `_renderSlides` the DOM order is: `navRow` (‹ 1/14 ›) → `imgEl` → `captionEl`. The navigation bar appears *above* the image. Convention for slide viewers is controls below the image (as in presentation tools, lightboxes, and most carousel implementations). Users moving their eyes to the bottom-right after reading a slide will not find the "next" button where expected.

**Proposal:** Reorder to: `imgEl` → `captionEl` → `navRow`. No CSS changes needed — the flex column order drives the visual result.

**Effort:** 3-line JS reorder in `_renderSlides`.

---

#### P2-D: `plr-gallery__thumb` hover scale does not signal "click opens lightbox"

**Problem:** `.plr-gallery__thumb:hover img { transform: scale(1.04); }` provides feedback on hover but there is no visual cue that clicking opens a fullscreen lightbox (vs. navigating to the file, zooming in-place, or doing nothing). A user who does not experiment will not discover the lightbox.

**Proposal:** Add a subtle overlay hint on hover:

```css
.plr-gallery__thumb::after {
    content: '';
    position: absolute;
    inset: 0;
    background: rgba(0,0,0,0) url("data:image/svg+xml,...") center/24px no-repeat;
    opacity: 0;
    transition: opacity 0.2s;
    pointer-events: none;
}
.plr-gallery__thumb:hover::after {
    opacity: 1;
    background-color: rgba(0,0,0,0.18);
}
```

Alternatively, use a simple CSS `cursor: zoom-in` on `.plr-gallery__thumb` — one line — which signals the expand affordance without any additional HTML. This is the quick win.

**Quick-win version (1 line):**
```css
.plr-gallery__thumb { cursor: zoom-in; }
```

---

#### P2-E: Loading state ("Loading page…") uses inline style, not a class

In `_openFolderPage` the loading message is set with an inline string:

```js
el.innerHTML = '<div style="padding:1rem;color:var(--color-text-secondary,#888);">Loading page…</div>';
```

This is a minor hygiene issue but it means the loading state cannot be restyled without touching JS. A dedicated `.plr-loading` class in the CSS would allow a spinner, a shimmer, or branding to be added later without code changes.

### P3 — Polish, low-effort improvements

#### P3-A: `plr-hero__title` font size does not adapt to title length

At 1.9 rem, a long title (e.g. "The Cryptographic Blueprint") wraps to two lines in a compact panel. At short titles the size feels small. Consider adding a `font-size: clamp(1.4rem, 2.8vw, 2.2rem)` on `.plr-hero__title` for natural scaling.

---

#### P3-B: Lightbox close button uses `position: fixed` — correct, but misaligns when panel is not full-viewport

`.plr-lightbox__close`, `.plr-lightbox__btn--prev`, and `.plr-lightbox__btn--next` all use `position: fixed`. The lightbox itself (`plr-lightbox`) also uses `position: fixed; inset: 0`. This works correctly when the lightbox covers the full viewport — which it does, since it is appended to `document.body`. No action required; noted for completeness.

---

#### P3-C: `plr-bullet-points` uses bare browser `list-style: disc`

The bullet disc colour inherits from the browser and renders as black or dark grey. This can look harsh on very light backgrounds. A quick improvement:

```css
.plr-bullet-points li::marker {
    color: var(--color-primary, #4ecdc4);
}
```

~3 lines of CSS, visible quality uplift on the root page's key-numbers and overview sections.

---

#### P3-D: `plr-nav` padding jumps at small widths

`.plr-nav { padding: 0 1.5rem; }` does not reduce at narrow widths. At panel widths below ~500 px, 1.5 rem left padding wastes horizontal space and may clip the last nav item. Suggest:

```css
.plr-nav { padding: 0 max(0.5rem, 1.5rem - 2vw); }
```

---

#### P3-E: `_page.json` direct-click tab title shows raw filename

When `_openFileTab` intercepts a `_page.json` path, the tab title computation in `_openFolderPage` uses the folder's last segment. For root-level `_page.json` it falls back to `this.fileName || 'Home'`. But `this.fileName` is the full archive filename (e.g. `sgraph-send-strategic-pack-v1.2.zip`) which is long and reads poorly in a narrow tab. A dedicated truncation or a `title` field from the page JSON itself (already parsed at render time) would improve the tab bar readability.

## Recommended Implementation Order

1. **P1-B** (scroll root / IntersectionObserver wiring) — functional correctness before any visual work
2. **P1-A** (dark nav theme) — resolves the most visually jarring inconsistency
3. **P1-C** (visible gallery captions) — highest information-density gain
4. **P2-C** (slide nav below image) — corrects the most counter-intuitive interaction
5. **P2-A + P2-B** (responsive hero + section full-bleed) — visual scaling
6. **P3 quick wins** — can all be batched in a single commit
