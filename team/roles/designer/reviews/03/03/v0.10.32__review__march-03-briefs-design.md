## Part 1: Encrypted Gallery Sharing -- Lead Brief

This is the most design-intensive brief the project has produced. It is not an incremental feature; it is a new product surface that will be the primary viral loop. Every pixel and every interaction matters because the gallery IS the product demo for every recipient who opens it.

### 1.1 Thumbnail Grid Layout

#### Grid Specification

The grid uses CSS Grid with `auto-fill` and `minmax` to produce a fluid, gap-consistent layout that adapts from a single column on narrow phones to four or five columns on wide desktop screens.

```css
.gallery-grid {
    display: grid;
    grid-template-columns: repeat(auto-fill, minmax(var(--gallery-thumb-min), 1fr));
    gap: var(--space-2);              /* 8px between items */
    padding: var(--space-4);
}
```

**Thumbnail minimum sizes by breakpoint:**

| Breakpoint | Viewport | `--gallery-thumb-min` | Columns (approx) | Rationale |
|-----------|----------|----------------------|-------------------|-----------|
| Mobile S | < 375px | 140px | 2 | Minimum viable touch target; two-across fits any phone |
| Mobile L | 375-599px | 160px | 2 | Slightly larger; comfortable two-up on standard phones |
| Tablet | 600-959px | 180px | 3-4 | Three or four columns; room for captions below thumbnails |
| Desktop | 960-1439px | 200px | 4-5 | Four-up with generous spacing in main content column |
| Wide | 1440px+ | 220px | 5-6 | Five or six columns; grid fills wide monitors without stretching |

**Aspect ratio enforcement:** All thumbnail cells use `aspect-ratio: 1` to form a uniform square grid. Images are rendered with `object-fit: cover` so that mixed portrait and landscape photos produce a clean mosaic. This is the same pattern used by Google Photos, iCloud Photos, and Instagram -- users expect a square grid.

**Alternative: masonry layout.** A masonry (Pinterest-style) layout preserves original aspect ratios and avoids cropping, but it creates an uneven bottom edge and makes mental counting harder ("which was photo 12?"). For Phase 1, the square grid is the correct choice. Masonry can be explored in Phase 2 as an optional view toggle.

#### Video Thumbnails in the Grid

Videos need a visual indicator that distinguishes them from photos. The design uses a triangular play icon overlay:

```
+-------------------+
|                   |
|                   |
|       [>]         |   <-- Semi-transparent play icon centred
|                   |
|      0:42         |   <-- Duration badge, bottom-right
+-------------------+
```

**Client-side video thumbnail generation:**
1. Create an off-screen `<video>` element and set `src` to the blob URL of the decrypted video.
2. Seek to `currentTime = 1.0` (one second in -- avoids black frames at 0s).
3. On `seeked` event, draw the video frame onto an off-screen `<canvas>` at 400x400px.
4. Convert to `canvas.toBlob('image/jpeg', 0.7)` for the thumbnail.
5. Cache the thumbnail blob URL in a `Map<string, string>` keyed by file path.

**Performance note:** Video thumbnail generation is more expensive than image thumbnail generation. The component should generate video thumbnails last, after all image thumbnails are visible. A `requestIdleCallback` or `IntersectionObserver` gate prevents video thumb generation from blocking the initial paint.

**Duration extraction:** Read `video.duration` from the same `<video>` element used for thumbnail capture. Format as `M:SS` for videos under 60 minutes, `H:MM:SS` for longer.

#### Gallery of 5 vs Gallery of 100 -- Scaling Behaviour

The grid must feel deliberately designed at both small and large scales.

**5-photo gallery:**

| Concern | Decision |
|---------|----------|
| Grid looks sparse on desktop (5 items in a 5-column grid = one row) | Use `max-width: 720px` centred container when item count < 12 to keep the grid compact |
| Single-row grid feels like a filmstrip | This is fine. The layout is clean, not empty |
| No scrolling needed | Gallery header + grid + footer fit in one viewport. No lazy loading needed |

**100-photo gallery:**

| Concern | Decision |
|---------|----------|
| DOM node count | Do NOT render 100 `<img>` tags on mount. Use `IntersectionObserver` to render thumbnails only when rows scroll into the viewport (virtual scrolling lite). Pre-allocate placeholder grid cells with background colour `var(--bg-secondary)` so the scrollbar height is correct from the start |
| Memory pressure on mobile | Limit concurrent in-memory thumbnails to ~30. Revoke `URL.createObjectURL` for thumbnails that scroll out of view. Re-generate them if the user scrolls back |
| Progress indication | Show a determinate progress bar at top of grid: "Decrypting... 34 of 100 photos". Once all thumbnails are generated, the bar fades out. This gives the user confidence that work is happening |
| Scroll performance | Use `content-visibility: auto` and `contain-intrinsic-size` on grid rows to enable browser paint optimisation |

#### Loading Experience

The loading sequence for a gallery has five distinct visual states. Each must feel intentional, not broken.

```
State 1: KEY ENTRY
   +---------------------------------+
   |  [Friendly key input field]     |
   |  [Open Gallery] button          |
   +---------------------------------+
   Clean, centred. Same layout as current download page.

State 2: DOWNLOADING
   +---------------------------------+
   |  Downloading encrypted gallery  |
   |  ======-------- 42%  3.2 MB/s  |
   +---------------------------------+
   Determinate progress bar. Show speed and percentage.

State 3: DECRYPTING
   +---------------------------------+
   |  Decrypting gallery...          |
   |  ================ instant       |
   +---------------------------------+
   AES-256 hardware acceleration makes this near-instant
   for any size. Brief flash state -- sometimes invisible.

State 4: EXTRACTING THUMBNAILS (progressive reveal)
   +---------------------------------+
   |  Photos from Saturday (42 items)|
   |  +----+ +----+ +----+ +----+   |
   |  |img1| |img2| |img3| |    |   |  <-- Thumbnails pop in left-to-right
   |  +----+ +----+ +----+ +----+   |
   |  +----+ +----+ +----+ +----+   |
   |  |    | |    | |    | |    |   |  <-- Placeholder cells (bg-secondary)
   |  +----+ +----+ +----+ +----+   |
   +---------------------------------+

State 5: GALLERY READY
   Full grid visible. Header shows title + count.
   Footer shows "Encrypted with SG/Send" + Download All.
```

**Transition between states:** Use CSS `opacity` + `transform: translateY(4px)` with 200ms ease for fade-in of each element. Never use layout-shifting animations (no height transitions, no expanding containers).

### 1.2 Lightbox Viewer

The lightbox is the gallery's centrepiece interaction. It must feel native -- comparable to iOS Photos or Google Photos on Android.

#### Desktop Lightbox Layout

```
+-------------------------------------------------------------------+
|                                                                     |
|  [X]                          3 / 42                      [DL]     |
|                                                                     |
|                                                                     |
|                                                                     |
|          +----------------------------------------+                 |
|          |                                        |                 |
|   [<]    |           FULL IMAGE                   |    [>]          |
|          |           max-height: 85vh              |                 |
|          |           max-width: 85vw               |                 |
|          +----------------------------------------+                 |
|                                                                     |
|                                                                     |
|                                                                     |
|                                                                     |
+-------------------------------------------------------------------+
```

**Key design decisions:**

| Decision | Reasoning |
|----------|-----------|
| Black backdrop (`rgba(0, 0, 0, 0.95)`) | The image is the entire focus. No UI chrome should compete |
| Navigation arrows outside the image | Arrows overlaying the image obscure content. Outside-image arrows also provide larger hit areas |
| Counter (3 / 42) top-centre | Minimal, non-intrusive. DM Sans weight-medium, 14px, `rgba(255, 255, 255, 0.7)` |
| Close button top-right | Universal convention. 44x44px touch target minimum |
| Download button top-right after close | Per-image download. Uses the existing download icon pattern |
| Keyboard navigation | Left/Right arrows, Escape to close, Space for video play/pause |
| Image sizing | `object-fit: contain` with `max-height: 85vh; max-width: 85vw` -- image fills available space without cropping |

#### Mobile Lightbox Layout

On mobile, the lightbox is a full-screen overlay with gesture-driven navigation.

```
+-------------------------+
| [X]     3/42            |
|                         |
|                         |
|                         |
|     FULL-SCREEN         |
|     IMAGE               |
|     (object-fit:        |
|      contain)           |
|                         |
|                         |
|                         |
|                         |
|          [Download]     |
+-------------------------+
```

**Mobile-specific interactions:**

| Gesture | Action | Implementation |
|---------|--------|----------------|
| Swipe left | Next image | `touchstart` / `touchmove` / `touchend` with 50px threshold. CSS `transform: translateX()` for smooth tracking. `will-change: transform` on the image container |
| Swipe right | Previous image | Same as above, opposite direction |
| Pinch to zoom | Zoom into image | Track two-touch distance delta. Apply CSS `transform: scale()` centred on the pinch midpoint. Clamp between 1x and 4x |
| Double-tap | Toggle 2x zoom | Zoom to 2x centred on tap point, or reset to 1x if already zoomed |
| Swipe down (while at 1x) | Close lightbox | Vertical swipe > 100px when not zoomed triggers close |
| Pan (while zoomed) | Move zoomed image | Single-touch drag when `scale > 1` translates the image within bounds |

**Preloading strategy:** When viewing image N, preload images N-1 and N+1 into off-screen `<img>` elements. This makes swipe transitions instant. For a 100-photo gallery, only ever hold 3 full-resolution images in memory.

### 1.3 Mobile-First Assessment for Gallery

The gallery is a mobile-first product. Most photo-sharing starts on a phone (WhatsApp message -> tap link -> gallery opens in mobile browser).

**Critical mobile constraints:**

| Constraint | Design Impact |
|-----------|---------------|
| Safe area insets (notch phones) | Gallery header and footer must use `env(safe-area-inset-top)` and `env(safe-area-inset-bottom)`. The lightbox close button must not be hidden behind the notch |
| Orientation changes | Gallery grid re-flows naturally (CSS Grid handles this). Lightbox should support landscape for wider image viewing. Use `screen.orientation` API or `matchMedia('(orientation: landscape)')` |
| Touch target sizes | All interactive elements minimum 44x44px per Apple HIG and Material guidelines. Thumbnail grid cells exceed this. Lightbox arrows must be large tap zones (full left/right half of screen, not just a small arrow icon) |
| Scroll momentum | Grid scrolling must use `-webkit-overflow-scrolling: touch` (or modern `overscroll-behavior`). The grid should feel like a native scroll |
| Video autoplay | Mobile browsers block autoplay with sound. Video in gallery should play muted by default with a tap-to-unmute control |
| Battery and data | On mobile, defer video thumbnail generation until the video cell is visible. Thumbnails should be JPEG quality 0.6 (not 0.9) to reduce memory |

**Responsive breakpoints for the gallery (in `design-tokens.css` additions):**

```css
:root {
    --gallery-bp-sm:    375px;
    --gallery-bp-md:    600px;
    --gallery-bp-lg:    960px;
    --gallery-bp-xl:   1440px;
}
```

These should be documented as CSS custom properties for reference but used in actual `@media` queries as literal values (CSS custom properties cannot be used in `@media` queries).

### 1.4 Gallery Component Architecture (Web Components)

The gallery requires four new Web Components, all following the existing `SendComponent` base class pattern with Shadow DOM encapsulation.

```
<gallery-viewer>                  -- Top-level container. Orchestrates state.
  |
  +-- <gallery-header>            -- Title, photo count, gallery.json metadata
  |
  +-- <gallery-grid>              -- Thumbnail grid with IntersectionObserver
  |     |
  |     +-- <gallery-thumb>       -- Individual thumbnail (image or video)
  |           (repeated N times)
  |
  +-- <gallery-lightbox>          -- Full-screen overlay with gestures
  |
  +-- <gallery-footer>            -- "Encrypted with SG/Send" badge, Download All
```

**Event flow:**

| Event | Emitted by | Handled by |
|-------|-----------|------------|
| `gallery-thumb-tap` | `<gallery-thumb>` | `<gallery-viewer>` opens lightbox at index |
| `gallery-lightbox-close` | `<gallery-lightbox>` | `<gallery-viewer>` returns to grid |
| `gallery-lightbox-navigate` | `<gallery-lightbox>` | `<gallery-viewer>` updates current index |
| `gallery-download-one` | `<gallery-lightbox>` | `<gallery-viewer>` triggers single-file save |
| `gallery-download-all` | `<gallery-footer>` | `<gallery-viewer>` downloads original zip |
| `gallery-thumb-ready` | `<gallery-thumb>` | `<gallery-grid>` tracks loading progress |

All events use `composed: true` to cross Shadow DOM boundaries, matching the existing `SendComponent.emit()` pattern.

**Rationale for separate components:** The lightbox has complex touch gesture state that must be isolated from grid scroll state. The header/footer are simple but benefit from Shadow DOM CSS isolation so gallery styling does not leak into the host page.

### 2.4 Drag-and-Drop

**Design complexity:** Medium (visual feedback states, drop-target highlighting)

The current vault has a basic drop zone (`#uv-dropzone`) with `dragover`/`dragleave`/`drop` handlers. The improvements needed are:

**Four drag-and-drop visual states:**

| State | Visual Treatment |
|-------|-----------------|
| **Idle** | Dashed border (`var(--color-border)`), muted text "Drop files here to encrypt & upload" |
| **Drag over main area** | Solid teal border, teal background glow (`var(--accent-subtle)`), text changes to "Drop to upload to current folder", subtle scale `transform: scale(1.01)` on the drop zone |
| **Drag over specific folder** | That folder row gets a teal left border (4px solid), background becomes `var(--accent-subtle)`. All other rows dim slightly (`opacity: 0.5`). This indicates "drop here to upload into this folder" |
| **Processing** | After drop: border goes solid, pulsing teal glow animation, text changes to "Encrypting and uploading..." with a spinner |

**Folder-targeted drops:** Each `.uv-row[data-action="open-folder"]` element becomes a valid drop target. On `dragenter`, the row highlights. On `drop`, the file is uploaded to that folder's GUID rather than `_currentFolder`.

**Multi-file visual feedback:** When multiple files are dropped, show a stacked count badge: "Uploading 5 files..." with a progress counter that updates as each file completes.

**Component changes:** No new component needed. Extend `<user-vault>` event binding in `_bind()` to attach drag handlers to folder rows. Add CSS classes for the four states.

**New i18n keys (4):**

```

+------------------------------------------------------------------+
|                                                                    |
|  +--------------------------------------------------------------+ |
|  | # Meeting Notes                                               | |
|  |                                                                | |
|  | - Discussed the Q2 roadmap                                    | |
|  | - Agreed on gallery feature priority                          | |
|  |                                                                | |
|  +--------------------------------------------------------------+ |
|                                                                    |
|  Unsaved changes                                                  |
+------------------------------------------------------------------+
```

**Design decisions:**

| Decision | Reasoning |
|----------|-----------|
| `<textarea>` not rich editor | Zero dependencies. IFD principle. A rich editor is Phase 2 |
| Monospace font | Text and JSON editing requires character alignment |
| "Unsaved changes" indicator | Amber dot + text when content differs from saved version |
| Save flow: encrypt -> upload -> update tree | Status toast: "Saving..." -> "Saved" |
| No auto-save | Explicit save avoids accidental writes consuming auth token uses |
| Tab key inserts tab character | Override default tab-to-next-field behaviour inside the textarea |

**Component:** `<vault-inline-editor>` -- new Web Component. Receives file content as a string. Emits `vault-editor-save` with the new content and `vault-editor-cancel`.

**New i18n keys (6):**

```
vault.editor.editing:           "Editing: {filename}"
vault.editor.save:              "Save"
vault.editor.cancel:            "Cancel"
vault.editor.unsaved:           "Unsaved changes"
vault.editor.saving:            "Saving..."
vault.editor.saved:             "Saved"
```

## Part 3: Vault CLI -- Designer Input

The CLI brief (`v0.10.30__dev-brief__vault-cli-local-sync.md`) is Developer-led. The Designer's input is limited to terminal UX conventions.

**Output formatting recommendations:**

| Convention | Rationale |
|-----------|-----------|

| Show progress with a spinner or progress bar for operations > 1s | `sg-vault sync` on 50 files should show per-file progress |
| Use table output for `sg-vault ls` and `sg-vault status` | Aligned columns with fixed-width formatting |
| End every successful operation with a one-line summary | "Sync complete. 3 uploaded, 1 downloaded, 0 conflicts." |
| Error messages must include the fix | "Error: No vault key found. Run `sg-vault connect --key <key>` to set one." |

No i18n is needed for the CLI (English-only for Phase 1).

## Appendix B: Gallery vs Existing Download Page -- Routing

The gallery is NOT a replacement for the download page. It is a new route:

| Route | View | Trigger |
|-------|------|---------|
| `/download?id=xxx#key` | Single file download + preview | Current behaviour, unchanged |
| `/gallery?id=xxx#key` | Gallery grid + lightbox | New route. Server detects that the transfer contains image/video files in a zip and redirects, OR the sender explicitly chooses "Share as Gallery" |

The gallery page is a new HTML entry point: `/send/v0/v0.2/v0.2.0/gallery.html`. It imports the gallery Web Components and the existing crypto module.
