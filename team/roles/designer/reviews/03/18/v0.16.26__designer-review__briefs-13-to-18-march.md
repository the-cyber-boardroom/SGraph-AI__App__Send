## Upload UX Redesign (03/16) — Design Assessment

### What's Right

The three-step flow (upload → distribution → credentials) follows **progressive disclosure** — the user sees only what's relevant at each stage. The team response document (v0.16.10) includes ASCII mockups, state machines, and component breakdowns — this is exactly how design should feed into implementation.

**Key design decisions that align with principles:**
- Step 2 uses separate pages per viewer type (not a single card picker) — respects **hierarchy** and **breathing room**
- Step 3 defaults to simple token (friendliest) — **sensible defaults** pattern
- Upload progress shows rotating trust-building messages — turns waiting time into engagement (uses **delight** principle)
- Demo packs linked from send page itself — **teaches through structure**

### Design Concerns

1. **Three sharing modes naming:** "Simple token", "Full link with key", "Separate key" — these names communicate to engineers, not users. Needs user-facing language that describes the *outcome* not the *mechanism*. Suggestion: "Easy share code", "One-click link", "Maximum security".

2. **Step 2 destination pages:** Risk of cognitive overload if there are 5+ distribution options. Need clear visual hierarchy — primary action prominent, advanced options discoverable but not competing for attention.

3. **i18n scope:** 26 new strings across 17 locales is significant. Design must ensure layouts work for languages with longer strings (German, Portuguese) without breaking.

### Recommendation

The upload UX redesign is the highest-priority design deliverable. Produce high-fidelity mockups for all three steps before implementation begins, with special attention to mobile layout and i18n string length.

## Component Pipeline & Theming (03/18) — Design Assessment

### `<sg-theme>` Component

The theming brief proposes a component that controls: language, colours, terminology, layout, icons, feature visibility, and accessibility settings. This is essentially a **design system runtime** — design tokens applied dynamically via a Web Component.

**Design principles alignment:**
- **Consistency:** One theming mechanism across all components — good
- **Honesty:** Theme changes terminology (e.g., "commit" → "save version" for musicians) — this is communication-appropriate, not misleading
- **Form follows function:** Theme adapts UI to the user's mental model, not just cosmetics

**Design concern:** Theme must not compromise **hierarchy**. If a vertical theme adds custom colours, ensure contrast ratios meet WCAG AA. If terminology changes, ensure help text and error messages update consistently.

### Musicians Vertical (03/18)

First vertical. Maps song production to Explorer/Villager/Town Planner — clever structural metaphor. Audio-specific components (waveform player, multi-track, take comparison) are genuinely new design challenges.

**Design decisions needed:**
- Waveform visualisation style (spectral vs. amplitude)
- Take comparison interaction ("pick this one" button placement)
- Album view layout (artwork prominence vs. track list)
- Terminology mapping throughout: "commit" → "save", "branch" → "version", "merge" → "combine"

## Gallery Features (03/17 + 03/18) — Design Assessment

Gallery editor and rich preview are content-presentation features. Design quality here directly affects how users perceive SG/Send.

**Rich preview (03/18):** Sender-side thumbnail generation is architecturally smart (no server-side processing). Design must ensure thumbnails are high enough quality to be useful (not just tiny placeholders) while small enough for fast upload.

**Gallery editor (03/17):** Per-image comments with multi-language translation is a unique differentiator. Design the comment display to avoid overwhelming the image — consider overlay vs. sidebar vs. below-image placement. The download-as-HTML feature means the gallery design must be self-contained CSS without external dependencies.

## Vault Browser UI (03/18) — Design Assessment

The vault browser brief proposes: auto-commit, auto-sync, branch status, commit history, file editing, and conflict resolution — all as Web Components using sg-layout.

**Design concern:** This is a complex multi-panel application. sg-layout provides VS Code-style docking but the *content* of each panel needs design attention:
- **Conflict resolution:** Side-by-side diff view needs clear visual language (additions green, deletions red, conflicts yellow) with decisive action buttons
- **Branch status:** "Ahead by 3, behind by 2" needs to be understandable by non-Git users. Consider visual metaphors.
- **Commit history:** Linear list vs. graph view. For non-technical users, timeline metaphor works better.
