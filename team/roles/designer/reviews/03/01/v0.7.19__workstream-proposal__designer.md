## Framing

The v0.7.6 briefs mark a significant inflection point for design. The sgraph.ai website is now live with the Aurora design system in place. Pricing and early-access pages exist. 17 locale JSON files are generated. New tools (IP tool, CloudFront log viewer) have been specified. A pricing matrix with 8 options has been proposed. Educational content pages are requested. Revenue collection via Stripe is imminent.

This creates a clear design agenda: **unify, complete, and educate**. Unify the visual language across sgraph.ai, send.sgraph.ai, and tools.sgraph.ai. Complete the missing pricing detail pages and educational content. Educate users through compelling, trustworthy design — the "Compare" section on the IP tool, the encryption explainers, the "What this page did NOT do" verifiable claims pattern.

The work streams below are ordered by their design leverage — how much user trust, revenue, and brand coherence each unlocks.

### P1 — Aurora Design Tokens: Shared Foundation for All Three Sites

**ID:** P1
**Title:** Aurora design token consolidation and cross-site sharing
**Priority:** P0
**Effort:** M

**Description:**
The Aurora design system CSS (`_common/css/style.css`) currently lives only in `sgraph_ai__website/`. The User UI (send.sgraph.ai), Admin UI, and the forthcoming tools.sgraph.ai all use the same fonts and brand language but load them from Google Fonts (still violated in 9 files) and have no shared token foundation. This work stream extracts the Aurora token set into a shareable artifact — either vendored into each repo/directory or served from cdn.sgraph.ai — so that all three surfaces share a single source of truth for colour, typography, spacing, and radius.

**Rationale:**
Until the token layer is shared, every new page or component risks design drift. The Google Fonts violation in 9 files (User UI v0.1.6, v0.1.7, v0.1.8, Admin UI, SSH KeyGen) is a symptom of the same root problem: there is no canonical design foundation that all surfaces draw from. Fixing the font violation AND establishing shared tokens is one compound task. It unblocks every downstream design work stream.

**Dependencies:**
- Depends on: cdn.sgraph.ai infrastructure decision (PROPOSED, does not exist yet) or IFD-style vendoring into each UI directory
- Unblocks: P2, P3, P4, P5, P6, P7 — all downstream UI work assumes shared tokens
- Blocks: Google Fonts violation fix in User UI and Admin UI (9 files) belongs in this work stream

**Deliverables:**
1. Audit document: map every CSS variable and font reference in sgraph_ai__website, User UI, Admin UI, SSH KeyGen tool
2. Decision: cdn.sgraph.ai vs vendored-per-site (coordinate with Architect)
3. Vendor DM Sans and JetBrains Mono woff2 files into User UI, Admin UI, SSH KeyGen (mirrors what was done for sgraph_ai__website)
4. Replace absolute nav links in sgraph_ai__website with relative paths (known violation, 8 HTML files)
5. Document the canonical Aurora token set as a reference for all future design work

### P3 — tools.sgraph.ai: Site Design and "Compare" Section UX

**ID:** P3
**Title:** tools.sgraph.ai site design, IP tool layout, and Compare section pattern
**Priority:** P0
**Effort:** L

**Description:**
tools.sgraph.ai does not exist yet (domain, infrastructure, or design). The privacy-first IP tool (`/my-connection`) and CloudFront log viewer (`/cf-logs`) are the first two tools proposed. Both have demanding design requirements: the IP tool must communicate trustworthiness through restraint (a single page that displays useful information with zero ornamentation from trackers), and the "Compare" section must make a forensic contrast between what this page did and what a typical IP-lookup site does. The CF log viewer must present a data-dense sortable table without overwhelming the user.

**Rationale:**
The IP tool brief contains one of the most compelling user-facing design challenges in the project: a page that proves its privacy claims through its own minimalism. The "What this page did NOT do" section and the "Compare" section are trust devices as much as they are UI components. If they are designed poorly — buried, hard to verify, cluttered — the educational message is lost. These patterns (verifiable claims, side-by-side tracking comparisons) are novel enough to warrant careful design before implementation.

**Dependencies:**
- Depends on: Architect decision on tools.sgraph.ai (separate static site vs sgraph.ai subdirectory) — PROPOSED, not resolved
- Depends on: P1 (shared Aurora tokens, if tools.sgraph.ai joins the same design system)
- Depends on: AppSec peer-review of the whatismyipaddress.com analysis (content for the Compare section)
- Unblocks: Dev implementation of both tools

**Deliverables:**
1. tools.sgraph.ai site structure: navigation, page template, header/footer pattern (or deliberate decision to have none — the IP tool may be more trusted as a standalone page without navigation chrome)
2. IP tool layout: above-the-fold data display (IP, location, ISP, browser details), "What this page did NOT do" verified claims block, "Compare" section layout
3. "Compare" section UX pattern: side-by-side or sequential contrast between tools.sgraph.ai/my-connection and a representative tracking-heavy IP site; data sourced from the brief's forensic analysis (25+ localStorage keys, 15+ sessionStorage keys, 120 requests, 12.3MB)
4. Verifiable claims component: a reusable design pattern for "this page made N requests, loaded X KB, set 0 cookies" — designed to be self-evidently checkable via browser DevTools, not just asserted
5. CF log viewer table design: column layout, filter bar, colour-coded status codes, IATA edge location display, summary stats bar

### P4 — Educational Content Pages: Encryption and PKI Explainers

**ID:** P4
**Title:** Educational content page design (What is PKI, encryption explainer, zero-knowledge primer)
**Priority:** P1
**Effort:** M

**Description:**
The v0.7.6 briefs propose educational crypto pages: "What is PKI?", "Symmetric vs Asymmetric Encryption", and related explainers. None exist. These pages must serve two audiences simultaneously: technically curious non-experts (the primary audience for sgraph.ai) and technical evaluators who will scrutinise the claims. The design challenge is making rigorous technical content readable without dumbing it down. Diagrams, visual metaphors, and progressive disclosure are the primary tools.

**Rationale:**
The zero-knowledge architecture is SG/Send's core differentiator. The security review (Brief 7) confirmed that even publishing the server logs does not compromise user data — this is a remarkable claim that deserves a page explaining why. Investors, evaluators, and new users all need to understand this. The educational pages are not just marketing; they are the technical substance that makes the "privacy-first" positioning credible. Design decisions made here (how to visualise key exchange, how to show what the server does and does not see) will be referenced repeatedly.

**Dependencies:**
- Depends on: P1 (Aurora tokens), Advocate copy (not yet written)
- Depends on: AppSec validation of the CloudFront security review findings (Brief 7) for the "even our logs are safe" claim
- Unblocks: Advocate content, Journalist/Alchemist materials referencing these pages

**Deliverables:**
1. Page template for educational content: reading-optimised layout with sidebar navigation, inline diagrams, call-out boxes for key claims, "Why this matters" summary sections
2. Diagram design: key exchange flow (symmetric), PKI flow (asymmetric), "what the server sees" contrast diagram (encrypted blob vs plaintext), AES-256-GCM encryption pipeline
3. Progressive disclosure pattern: collapsed detail for technical depth, summary for non-technical readers, no scrolljacking
4. "Even our logs are safe" explainer section design: using the security review findings as a concrete, visual proof point

### P5 — User UI: Design System Uplift and Visual Consistency

**ID:** P5
**Title:** User UI design system uplift (fonts, tokens, component visual refresh)
**Priority:** P1
**Effort:** M

**Description:**
The User UI (send.sgraph.ai, v0.1.8) loads Google Fonts from an external CDN (9 files still violated), has no shared token connection to the Aurora design system, and has accumulated visual inconsistencies across Upload, Download, Room, Join, and Vault pages. This work stream brings the User UI into alignment with the Aurora system: vendored fonts, shared colour tokens, consistent component spacing and border radii, and a review of the existing web components against the design system spec.

**Rationale:**

If sgraph.ai (the marketing site) looks polished Aurora and send.sgraph.ai looks visually different, the brand breaks at the moment of actual use. The Google Fonts fix is a P0 security/privacy requirement (a privacy product loading fonts from Google is a credibility problem) and a P1 performance item (per the perf baseline, fonts add 63.1KB over external CDN).

**Dependencies:**
- Depends on: P1 (Aurora token extraction decision — vendored vs cdn.sgraph.ai)
- Depends on: IFD versioning (UI changes go into a new minor version, e.g., v0.1.9 or v0.2.0)
- Unblocks: Any future User UI feature work that assumes a clean design foundation

**Deliverables:**
1. Design audit: side-by-side comparison of current User UI pages against Aurora design system spec (colours, spacing, typography, component patterns)
2. IFD patch plan: which components need updates, which can be inherited, what the new version number should be
3. Font vendoring: apply the same `_common/fonts/` approach from sgraph_ai__website to the User UI
4. Component visual spec: updated visual spec for `send-upload`, `send-download`, `room-view`, `user-vault`, `send-access-gate` aligned to Aurora tokens
5. Relative path migration: audit and fix any absolute path references in User UI pages (parallel to the website violation)

### P6 — i18n Design: Text Expansion, RTL Readiness, and Locale Selector UX

**ID:** P6
**Title:** i18n design considerations: text expansion budgets, RTL readiness, locale selector UX
**Priority:** P1
**Effort:** M

**Description:**
17 locale JSON files exist. Pre-rendered pages will be generated for all locales. The design system has not been tested against translated content. German and Polish text in UI components typically expands 30-40% over English, breaking button labels, card widths, and navigation items. The locale selector dropdown (in `i18n.js`) has no design spec. This work stream stress-tests the Aurora design system against the 17 locales and produces expansion budgets and RTL readiness recommendations.

**Rationale:**
i18n is not just a content problem — it is a layout problem. A button that says "Send encrypted file" in English may say "Enviar archivo cifrado" in Spanish (36% longer). If the design system has no expansion budget for buttons, labels, and cards, the localised pages will break silently. The brief mentions EU 24 official languages as a target; some of these (Arabic, Hebrew, if added later) require RTL support. Building RTL readiness into the design system now is cheaper than retrofitting.

**Dependencies:**
- Depends on: P1 (Aurora tokens), `_common/js/i18n.js` (exists)
- Depends on: locale JSON files (exist for 17 locales)
- Unblocks: i18n page generator output quality, future RTL locale additions

**Deliverables:**
1. Text expansion audit: for each UI component with text content (buttons, labels, navigation items, card titles), measure the maximum expected expansion across the 17 locales and set a budget
2. Component hardening recommendations: which components need `overflow: hidden`, `text-overflow: ellipsis`, or flexible widths to survive expansion
3. Locale selector design: visual spec for the language selector dropdown in `i18n.js` — flag vs language name vs code, mobile behaviour, keyboard accessibility
4. RTL readiness assessment: which CSS properties (margin-left, padding-right, text-align, flexbox direction) need logical property equivalents (margin-inline-start, etc.) for future RTL support
5. Translation status page design: layout spec for the `sgraph.ai/languages/` page (PROPOSED, does not exist yet) showing locale coverage and AI-generated content disclaimer

### B1 — Brand Cohesion Audit Across Three Domains

**ID:** B1
**Title:** Brand cohesion audit: sgraph.ai, send.sgraph.ai, tools.sgraph.ai
**Priority:** P1
**Effort:** S

**Description:**
Three domains are emerging: sgraph.ai (marketing), send.sgraph.ai (product), tools.sgraph.ai (PROPOSED utility tools). Each currently has its own visual approach. This work stream is a structured audit — not a redesign — that identifies where the brand is coherent and where it diverges, and produces a concise set of binding decisions for the Designer to enforce across all three.

**Rationale:**
Brand drift between sgraph.ai (polished Aurora) and send.sgraph.ai (Google Fonts, slightly different component styles) is already detectable. Adding tools.sgraph.ai as a third surface without a coherent brand mandate will compound the drift. An audit document that names the divergences and proposes resolutions is faster to produce than a full redesign and creates alignment across all design work streams.

**Dependencies:**
- Depends on: Reality document (exists), P1 (Aurora token extraction)
- Unblocks: P3 (tools.sgraph.ai design decisions informed by audit), P5 (User UI uplift scope informed by audit)

**Deliverables:**
1. Brand cohesion audit document: visual comparison of all three surfaces (or two existing + one planned) across typography, colour usage, component patterns, navigation chrome, and tone of voice in UI microcopy
2. Binding brand decisions: e.g., "All three sites use Aurora tokens", "tools.sgraph.ai uses the same header/footer as sgraph.ai", "Admin UI is exempt from public brand requirements"
3. Divergence fix backlog: prioritised list of design changes needed to close the audit findings

### D1 — Design System Documentation: Aurora Component Library Reference

**ID:** D1
**Title:** Aurora component library reference documentation
**Priority:** P2
**Effort:** S

**Description:**
The Aurora design system exists in `_common/css/style.css` as a set of CSS custom properties and component classes. No reference documentation exists for it. Developers implementing new pages (pricing detail pages, educational content, tools.sgraph.ai) must reverse-engineer the design system from the CSS. A component library reference — a simple HTML page listing every component with usage examples — eliminates this friction.

**Rationale:**
As the number of pages grows from 13 (current) toward 20+ (pricing detail pages, educational content, About, tools.sgraph.ai), design consistency becomes increasingly dependent on documentation rather than tribal knowledge. The Ambassador writing copy and the Dev implementing HTML both need to know: "what class do I use for a warning callout? what does a ghost button look like?" A reference page answers these questions without requiring a Designer review on every page.

**Dependencies:**
- Depends on: P1 (finalized Aurora token set)
- Unblocks: All future page implementation (pricing details, educational content, tools.sgraph.ai)

**Deliverables:**
1. Aurora reference page (`sgraph_ai__website/design-system/index.html`): live HTML showing every component class (buttons, cards, callouts, stats, forms, modals, badges) with copy-paste code snippets
2. Token reference table: all CSS custom properties with their values, semantic purpose, and example usage
3. Typography specimen: all font sizes, weights, and line-height combinations as rendered on Aurora backgrounds

## Key Design Risks

1. **Font violation in User UI erodes the "privacy-first" brand.** A product that preaches zero-knowledge encryption while loading fonts from Google Fonts will be noticed by technical users. This is not just a performance problem — it is a brand credibility problem. P1 and P5 must close this.

2. **tools.sgraph.ai risks looking like a different company.** If the Architect decides tools.sgraph.ai is a separate static site without shared Aurora assets, and it is built without design oversight, it will look like a different product. The brand cohesion audit (B1) and P3 must pre-empt this.

3. **Pricing matrix complexity can confuse rather than clarify.** Eight options (4 free + 4 commercial) is a lot to process. The design principle "same zero-knowledge encryption, only the infrastructure differs" is the simplifying frame. If P2 fails to make this frame visually dominant, the matrix will overwhelm buyers instead of empowering them.

4. **i18n text expansion will break components silently.** Without expansion budgets (P6), translated pages will have broken button labels and truncated card titles that no one will notice until a native speaker sees them. This is a slow-burn quality problem.

5. **The educational pages are the hardest design challenge.** Making AES-256-GCM encryption comprehensible to a non-technical audience without sacrificing accuracy is genuinely difficult. P4 should begin with the diagram design before any copy is finalised — the diagrams will constrain and clarify the copy, not the other way around.
