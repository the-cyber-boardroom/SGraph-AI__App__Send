# Design Review: Zip/Folder Viewer UX

**Version:** v0.13.0
**Date:** 2026-03-10
**Role:** Designer
**Status:** Active
**Context:** Investor pack delivery via SG/Send zip upload — the recipient experience when opening, decrypting, and browsing a folder of documents.

---

## Hi Dinis — Let's Talk About How This Space Works

I've reviewed the current zip/folder viewer (`send-download.js` v0.2.0, `send-download.css`) and studied the screenshots you shared. The investor pack use case is a perfect lens for this review — it's a real user journey with high stakes (an investor's first impression of SG/Send *is* SG/Send).

The good news: the bones are strong. The decryption flow works, the tree browser exists, the preview pipeline handles markdown/PDF/images/code. What needs attention is **how the space is allocated** and **what the layout communicates** to a first-time recipient.

Here's my assessment, grounded in design principles:

---

## Finding 1: Make It Obvious This Is a Folder View

**What's happening:** The zip header shows the filename, a small `zip` badge, size, and file count. But there's no strong visual signal that says "you're browsing a folder of documents." The tree view exists but it doesn't announce itself.

**Why it matters (hierarchy principle):** The most important thing should be the most visible. The recipient just opened a link, decrypted something, and now they're looking at a complex layout. The first thing they should understand — in under 2 seconds — is: "This is a folder. It has structure. Click around to explore."

**Recommendation:**
- Add a visual folder icon or illustration to the zip header area — not just the text badge
- Consider a brief contextual label: "Folder contents" or "Browsing: leonardo-calisse__sg-send__march-2026.zip" with a folder icon
- The `zip` badge is good but too subtle — make it more prominent, perhaps with a different visual treatment that says "archive" rather than just a mono-font label
- Consider a brief "how to navigate" hint on first load (progressive disclosure): "Click a file to preview it. Use the folder tree to navigate."

---

## Finding 2: Tree View Takes Too Much Horizontal Space

**What's happening:** The zip browser uses a two-column grid (`grid-template-columns: minmax(140px, 200px) 1fr`). The folder tree column at 140–200px is reasonable on wide screens, but on a typical laptop viewport (1280–1440px wide within the container), this is a significant chunk of the browsable area.

**Why it matters (proportion principle):** The tree view is navigation, not content. Navigation should take the minimum space needed to be usable, leaving maximum space for the thing the user came to see — the documents.

**Recommendation:**
- Make the folder tree **collapsible** — allow it to collapse to just icons or a hamburger, expanding on hover/click
- Consider a **breadcrumb** approach at the top of the file list instead of a permanent tree panel. For a typical investor pack with 2–3 folders, breadcrumbs communicate structure without consuming a permanent column
- If keeping the tree: reduce the min-width to 120px, and allow it to be resized (similar to the existing split-resize divider on the non-zip preview layout)

---

## Finding 3: The Preview Cannot Be Maximised

**What's happening:** The zip preview panel has `min-height: 200px` but no mechanism to expand it to full viewport height. The browser section (`max-height: 200px`) sits above the preview, plus the zip header, plus the transparency panel, plus the timing display, plus the "Send another" CTA below. The preview gets whatever space is left after scrolling past all of that.

**Why it matters (breathing room + proportion):** When someone clicks on a markdown document in the investor pack, they want to *read* it. The preview is the primary content — everything else is secondary context. But the current layout treats the preview as one section among equals. It's not. It's the main event.

**Recommendation: Reorganise the Vertical Stack**

Current layout (top to bottom):
1. Success banner
2. Zip header (name, badge, size, actions)
3. Zip browser (folder tree + file list) — `max-height: 200px`
4. Preview panel — `min-height: 200px`
5. Transparency panel ("What we stored about this transfer")
6. Timing display
7. "Send a new File or Text" link
8. "Want to send files too?" CTA

**Proposed layout:**

1. Success banner (collapse after 3 seconds or on scroll)
2. **Compact header bar:** zip name + badge + file/folder count + [Save All] + [Save File] — single line, sticky
3. **Main content area** (takes all remaining viewport height):
   - **Left rail** (collapsible, ~180px): folder tree + file list stacked vertically
   - **Main panel** (flex-grow): document preview — **this should fill the viewport**
4. **Footer section** (below the fold, accessible by scrolling): transparency panel, timing, send another, sign-up CTA

The key insight: **Move the transparency info and timing out of the primary viewport.** They're important for trust but not for reading. A user who has seen the transparency table once doesn't need it taking up space every time they switch files.

---

## Finding 4: PDF Preview Is Broken by Space Constraints

**What's happening (pic5):** When viewing a PDF, the preview is an iframe. The iframe height is constrained by the available space after the browser panel and header. The PDF renders at roughly 400px tall on a 1080p screen — too small to read. The transparency panel below pushes the preview up further.

**Why it matters (affordance):** A PDF preview that's too small to read defeats the purpose of previewing. The user has to download the file anyway, which means the preview provided no value.

**Recommendation:**
- Implement a **maximise button** on the preview panel — a simple toggle that hides the browser section and expands the preview to fill the viewport (minus the sticky header bar)
- For PDFs specifically: set a minimum iframe height of `calc(100vh - 200px)` when the preview is the active view
- Consider a **double-click to maximise** gesture on the preview panel
- Add a keyboard shortcut: `Escape` to exit maximised view

---

## Finding 5: Move Transparency Info to the Header Section

**Your idea is exactly right.** The transparency table ("What we stored about this transfer") is valuable for trust and differentiation — it demonstrates the zero-knowledge architecture. But it belongs in a **collapsible section within the header area**, not below the preview.

**Recommendation:**
- Add a collapsible "Transfer details" or "What we stored" section to the header bar
- Default state: **collapsed** (showing just the title as a clickable link/button)
- Expanded state: shows the full transparency table
- Persist the collapsed state in localStorage — once a user has seen it, they don't need it expanded on every file view
- This frees up all the space below the preview for the document itself

**Alternative:** Put the transparency info in a **slide-out panel** accessible via an info icon in the header bar. This keeps the main area completely clean.

---

## Finding 6: Markdown Preview Should Have a White Background

**What's happening:** Markdown renders in a sandboxed iframe with the dark theme inherited from the page. For an investor pack with long-form documents, the dark background makes reading harder for extended periods.

**Why it matters (honesty + delight):** Markdown documents are meant to be read like documents. Documents have been read on light backgrounds for centuries. The dark theme is perfect for the application chrome, but the content area should serve the content.

**Recommendation:**
- Set the markdown iframe body background to white (`#FFFFFF`) or near-white (`#FAFAFA`)
- Set text colour to dark (`#1a1a1a` or similar)
- Keep the surrounding chrome (header, tree view, borders) in the dark Aurora theme
- This creates a clear visual distinction between "application" and "content" — which itself communicates: "this is *your* document, we're just showing it to you"
- Apply the same treatment to code preview (light editor theme for code blocks)

---

## Finding 7: File Type Visual Indicators

**What's happening:** File types are indicated by emoji icons (`_fileTypeIcon`): markdown gets a memo, PDF gets a page, images get a frame, code gets a laptop, etc. These are functional but not distinctive enough at a glance.

**Recommendation:**
- Replace emoji icons with **SVG icons** from a consistent icon set — this gives more control over size, colour, and style
- Use **colour coding** alongside icons: markdown = blue, PDF = red, images = green, code = amber — following common conventions users already know
- Show a **file type badge** in the preview header area (similar to the `zip` badge but for the currently previewed file type)
- Consider showing a **thumbnail** for image files in the file list itself, not just the preview panel

---

## Finding 8: Additional Ideas

### A. Keyboard Navigation
The zip browser currently requires mouse clicks. For a power user browsing a 10-file investor pack:
- Arrow keys to move between files in the list
- Enter to preview
- Escape to go back to file list focus
- `S` to save current file

### B. Auto-Preview First File
When the zip is decrypted, the first file should auto-preview (you already auto-select the root folder). For the investor pack, this means `0) Start_Here__Reading_Guide.md` is immediately visible — the sender's intended reading order is respected.

### C. Reading Progress
For a folder with numbered files, show a subtle progress indicator: "Viewing 1 of 10 files." This encourages the recipient to explore the full pack.

### D. Breadcrumb Trail
When navigating into subfolders (`supporting-materials/`, `the-three-asks/`), show a breadcrumb at the top of the file list: `/ > supporting-materials >`. This communicates location within the structure.

### E. Full-Screen Mode
Beyond maximising the preview, offer a true full-screen mode (using the Fullscreen API) for PDF and image viewing. This is the nuclear option for "I just want to read this document."

## Sherpa Perspective (Requested)

The Sherpa would focus on the **first-time recipient experience**. From that lens:

- The recipient has never seen SG/Send before. They got a link. They clicked it. They see encrypted data being decrypted (impressive). Then... a complex three-panel layout with a tree view, file list, and preview. **The Sherpa's friction log would flag:** "Where do I click? What am I looking at? What's the tree on the left?"
- **Sherpa recommendation:** Add a brief welcome state when the zip first decrypts: "Your files are ready. Click any file below to preview it." This teaches through structure (Designer principle #1).
- The transparency table is reassuring the first time ("oh, they can't see my data") but becomes noise on subsequent views. The Sherpa would support collapsing it by default after first view.

## Advocate Perspective (Requested)

The Advocate would focus on the **use case** — an investor receiving a curated pack:

- The investor wants to skim the reading guide, scan the architecture doc, and forward the competitive matrix to a colleague. They need: fast navigation, readable previews, and easy per-file download.
- **Advocate recommendation:** The "Save This File" button should be more prominent — it's the second most important action after reading. Consider placing it as a floating button in the preview panel itself, not just in the header bar.
- For the "forward to colleague" use case: a "Copy link to this file" or "Share this file" action would be valuable (PROPOSED — does not exist yet).

## Addendum: Video Single-File View — What's Already Working

**Data point added 2026-03-10** — Screenshots of the video upload experience (`The_Zero-Knowledge_Substrate__Engineering_the_End_of_Shared_Sec.mp4`, 64.9 MB).

### What the Video View Gets Right

The single-file video view already demonstrates the layout principles the zip viewer needs:

1. **Two-column split works well** — Details panel (left) with file name, type badge, size, upload date, Save/Copy buttons, and transparency info. Preview panel (right) with the video player. The video gets the majority of the viewport width. This is exactly the proportion the zip viewer should aim for.

2. **The browser's native fullscreen button is available** — The `<video>` element's native controls include a fullscreen toggle (the `⛶` icon, visible in pic2). This works out of the box. The user can go fullscreen, watch the video, and exit. No extra implementation needed.

3. **Content gets the right proportion** — The video fills the preview panel naturally. The left panel scrolls independently. This is the correct hierarchy: content is primary, metadata is secondary.

### What Could Be Better

**A. Resize behaviour (pic2):** When the browser is narrower, the left details panel compresses but doesn't collapse. The file name wraps awkwardly (`The_Zero-Knowledge_Substrat\ne__Engineering_the_End_of_S\nhared_Sec.mp4`). At narrow widths, the details panel should collapse entirely or switch to an overlay/drawer pattern.

**B. Branded fullscreen experience:** The browser's native fullscreen removes all SG/Send branding. This is a missed opportunity. Instead of (or in addition to) the native fullscreen:

- Add a **custom "Theatre Mode" button** in the SG/Send UI chrome (not relying on the browser's native control)
- Theatre Mode would: hide the details panel, expand the video to full width, keep a minimal SG/Send header bar with the logo and a "Back" button
- This keeps the brand visible while the user watches — important when the video is being shown to someone else ("What tool is this?")
- A subtle **"Played via SG/Send"** watermark or overlay in the bottom-right corner (like YouTube's logo during playback) would reinforce brand association without interfering with content

**C. The "video" type badge** is good — clear, correct, immediately communicates file type. This same pattern should carry over to the zip viewer's file list (Finding #7 in the main review).

### Design Pattern: Apply Single-File Layout to Zip Preview

The key insight from comparing these views: **the zip viewer should use the same two-column split layout as the single-file view** when a file is selected for preview. Currently:

- Single-file view: `details panel (left) | preview panel (right)` — works well
- Zip view: `zip header (top) → zip browser (top, 200px) → preview (bottom)` — stacks everything vertically

**Proposed zip layout using the same split pattern:**

```
┌─────────────────────────────────────────────────────┐
│ Zip header: name + badge + size + [Save All]        │
├────────────┬────────────────────────────────────────┤
│ Folder tree│                                        │
│ ───────────│       Preview panel                    │
│ File list  │       (fills viewport height)          │
│            │                                        │
│ [details]  │                                        │
│ [collapse] │                                        │
├────────────┴────────────────────────────────────────┤
│ Transparency + timing (below the fold)              │
└─────────────────────────────────────────────────────┘
```

The left rail combines the folder tree, file list, and (optionally) collapsed transparency info. The right panel is the full-height preview. This mirrors the single-file layout exactly, which means:
- Consistent mental model for the user across file types
- The same resize/split mechanism already built for single files works here
- Theatre/maximise mode can hide the left rail, just like it would hide the details panel

This reinforces Finding #3 and #4 from the main review — the solution is not inventing a new layout, but **applying the layout that already works for single files** to the zip view.
