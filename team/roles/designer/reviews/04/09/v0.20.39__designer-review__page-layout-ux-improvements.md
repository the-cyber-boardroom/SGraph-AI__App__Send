# Designer Review — Page Layout UX Improvements
**Version:** v0.20.39 | **Date:** 2026-04-09 | **Role:** Designer
**Scope:** `page-layout-renderer.js`, `send-browse-v031--page-layout.js`, `send-browse-v031--page-layout.css`
**Responds to:** User testing with 5 screenshots of the "Document → Infographic Slide Deck" vault

---

## 1. Summary of Findings from User Feedback

User testing across 5 screenshots revealed four categories of friction:

### 1.1 Context problem (pic1)
New users don't understand they're looking at an AI-generated rich-media page rendered from a JSON definition. The rendered page appears without context — no banner, badge, or label explains what they're seeing or that they can interact with it.

**Impact:** High. First-time users may mistake the rendered page for a static preview and miss interactive features (nav, cards, present mode).

### 1.2 Nav visibility (all screenshots)
The sticky navigation bar blends into the action bar above it. Users miss the nav because the two rows have similar styling and insufficient visual separation.

**Impact:** Medium. Users who miss the nav don't get the section-jumping affordance — a key UX advantage of `_page.json` over flat markdown.

### 1.3 Link affordance (pic3, pic4)
Links in `.plr-markdown a` and cards with `plr-card--link` are not obviously clickable. The page looks beautiful but static. Users were not clicking card elements because they didn't appear interactive.

**Impact:** High. This directly suppresses navigation through the vault hub structure.

### 1.4 Bottom cropping (pic5)
The last text block or section at the bottom of the scroll view was cropped — not fully visible even after scrolling to the end. The scroll container did not have enough bottom padding.

**Impact:** Medium. Content cutoff at the bottom creates an impression of incomplete pages.

## 3. New Block Types — Specs and JSON Examples

### 3.1 `callout` (P1 — info/notice boxes)

**Purpose:** Surface important information without interrupting prose flow. Four semantic variants map to distinct visual treatments.

**JSON:**
```json
{
  "type": "callout",
  "props": {
    "variant": "info",
    "title":   "Optional heading",
    "text":    "Body text of the notice."
  }
}
```

**Variants and colours:**

| Variant | Border/text | Background |
|---------|-------------|------------|
| `info` | `#1a8fe0` (blue) | `rgba(26,143,224,0.07)` |
| `warning` | `#d97706` (amber) | `rgba(217,119,6,0.07)` |
| `success` | `#16a34a` (green) | `rgba(22,163,74,0.07)` |
| `tip` | `#7c3aed` (purple) | `rgba(124,58,237,0.07)` |

Dark mode uses 0.13 alpha backgrounds (more visible on dark backgrounds). The icon (ℹ / ⚠ / ✓ / 💡) is set by the renderer based on variant.

**Design rationale:** Border-left-only treatment (not a full border) is less visually heavy than a box. The background tint confirms the semantic meaning without overwhelming text content.

---

### 3.2 `stats` (P1 — metric cards)

**Purpose:** Highlight key numbers at a glance — investor metrics, performance figures, product facts.

**JSON:**
```json
{
  "type": "stats",
  "props": {
    "items": [
      { "value": "256",   "label": "Bit encryption",  "delta": "AES-GCM" },
      { "value": "99.9%", "label": "Uptime SLA",       "delta": "+0.1%" },
      { "value": "0",     "label": "Server plaintext",  "delta": "by design" }
    ]
  }
}
```

**Visual:**
- Cards use `flex: 1` so they distribute evenly across available width
- `min-width: 120px` + `flex-wrap: wrap` means they stack naturally on narrow viewports
- Value uses the `--plr-accent` colour to tie the numbers to the page's accent scheme
- Delta: strings starting with `-` render red (`.plr-stats__delta--down`); all others green (`.plr-stats__delta--up`)
- Dark mode: card border `rgba(255,255,255,0.1)`, background `rgba(255,255,255,0.04)`

---

### 3.3 `quote` (P1 — pull quotes)

**Purpose:** Highlight a key statement, testimonial, or extracted phrase with visual weight.

**JSON:**
```json
{
  "type": "quote",
  "props": {
    "text":   "Zero-knowledge architecture is the only honest answer.",
    "author": "Dinis Cruz",
    "role":   "Founder, SGraph AI"
  }
}
```

**Visual:** Left border using `--plr-accent`, subtle tinted background (`rgba(78,205,196,0.05)`), italic quote text at `1.1rem`, attribution on a separate line in smaller weight. `role` is rendered after `author` with a comma separator.

---

### 3.4 `author` (P1 — attribution block)

**Purpose:** Essential for AI-generated content — clearly labels who/what created the page and when.

**JSON:**
```json
{
  "type": "author",
  "props": {
    "name":   "Dinis Cruz",
    "role":   "Founder, SGraph AI",
    "date":   "2026-04-09",
    "avatar": "images/avatar.jpg"
  }
}
```

**Visual:** Horizontal flex layout — circular avatar (48px, accent-coloured border) on the left, name/role/date stack on the right. Top border separates it from the preceding content. Avatar is optional; the block renders cleanly without one.

## 4. Theme Picker in Edit Mode — Design Spec

### Problem
The edit mode previously showed a raw textarea with no visual assistance for choosing a theme. Users had to know the exact JSON structure and colour values from memory.

### Solution
A compact theme picker bar was added immediately above the textarea (`.plr-edit-theme-bar`). It contains:
- A small "Theme:" label
- A `<select>` dropdown listing all 6 named schemes

**Behaviour:**
1. User opens edit mode (clicks "✎ Edit")
2. Dropdown defaults to "— keep current —" (no-op)
3. User selects e.g. "navy (formal report)"
4. The renderer parses the current textarea JSON, sets `json.theme` to the navy scheme object, re-serialises with `JSON.stringify(parsed, null, 2)`, and writes back to the textarea
5. A 100ms debounce triggers a live re-render in the preview pane
6. The dropdown resets to "— keep current —" so the user can see the result and select another scheme if needed

**Edge cases:** If the textarea contains invalid JSON, the change handler catches the parse error silently and does nothing — the user sees the validation error in the status bar below.

**CSS:** `.plr-edit-theme-bar` uses the same dark background as `.plr-edit-status` (`#181c22`) to visually belong to the editor column, not the preview column.

## 5. Link Affordance Changes

The following changes were already in place from the previous session (v0.20.16–17) and verified as correct:

### 5.1 Markdown links
```css
.plr-markdown a {
    color: var(--plr-accent, #4ecdc4);
    text-decoration: underline;
    text-underline-offset: 2px;
    transition: opacity 0.15s;
}
.plr-markdown a:hover { opacity: 0.75; }
```

Before this fix, markdown links inherited the shell's link styles (often unstyled or barely visible). Now they are clearly blue-green with an underline — the universal affordance for clickability.

### 5.2 Card links
```css
.plr-card--link { cursor: pointer; }
.plr-card--link .plr-card__body::after {
    content: '→';
    color: var(--plr-accent, #4ecdc4);
    opacity: 0;
    transform: translateX(-4px);
    transition: opacity 0.15s, transform 0.15s;
}
.plr-card--link:hover .plr-card__body::after { opacity: 1; transform: translateX(0); }
.plr-card--link:hover {
    box-shadow: 0 4px 20px rgba(0,0,0,0.12);
    transform: translateY(-2px);
    border-color: var(--plr-accent, #4ecdc4);
}
```

The arrow reveal on hover combined with the lift (`translateY(-2px)`) and accent border change provides unambiguous interactive affordance without cluttering the at-rest state.

| Item | Priority | Effort | Notes |

| Context banner / "page identity" block | P1 | Medium | Optional `type: "banner"` for page metadata strip below action bar. Tells new users what they're looking at. |
| `pdf` block documentation in authoring guide | P1 | Small | The `pdf` block exists in the renderer and guide but is missing a demo entry in `page-layout-demo.json` (no real PDF to reference). |
| Theme picker: visual preview swatches | P2 | Medium | The current dropdown is text-only. A small colour swatch next to each option would improve discoverability. |
| `type: "divider"` block | P2 | Small | A simple `<hr>` with theme-aware colour. Currently authors must use `section.divider: "line"` which requires wrapping content in a section. |
| Edit mode: line numbers in textarea | P2 | Medium | Long JSON files are hard to navigate in the current raw textarea. A CodeMirror-lite or simple line-number overlay would help. |
| Keyboard shortcut for edit mode | P3 | Small | `Cmd/Ctrl+E` to toggle edit mode (mirrors `Cmd/Ctrl+P` for print). |
| Animation on callout entrance | P3 | Small | A subtle fade-in when callout scrolls into view (IntersectionObserver, same pattern as nav active state). |

### Known gaps in the demo JSON
The `page-layout-demo.json` file omits `gallery`, `slides`, `pdf`, and `image` blocks because they require real files in a vault to render correctly. The demo is valuable for text/layout block types but needs a real vault to test the media blocks.

### Authoring guide: section on `columns` children limit
The guide notes "`columns` only has direct children (no nesting of `columns` inside `columns`)" but does not explain the workaround (use a `section` with layout `"wide"` to achieve multi-column-within-section layouts). This should be documented.
