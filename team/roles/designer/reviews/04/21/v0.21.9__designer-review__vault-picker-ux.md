**Version:** v0.21.9 | **Date:** 21 April 2026 | **Role:** Designer
**Scope:** UX design for `<sg-vault-picker>` component (all 4 stages) + landing page visual direction
**Responds to:** Doc 297 (revised auth MVP), Dev review 04/21 build priorities

---

## 1. `<sg-vault-picker>` — UX Design

### 1.1 Design Philosophy

The vault picker must feel like a **natural extension of the browser's own credential UI** — not a custom auth screen. The mental model we are supporting:

> "Your vault key is like a Wi-Fi password — you save it once, the browser remembers it, and you never have to think about it again."

Stage 1 uses `localStorage` as a stand-in. The UX must be designed so the same flows work identically in Stage 2 when `PasswordCredential` replaces the storage layer. The user-facing experience does not change between Stage 1 and Stage 2.

---

### 1.2 States and Flows

#### State: No vault (first visit)

```
┌─────────────────────────────────────┐
│  🔒  Your personal vault            │
│                                     │
│  A vault stores your files and      │
│  settings, encrypted in your        │
│  browser. Only you hold the key.    │
│                                     │
│  ┌─────────────────────────────┐    │
│  │  Vault name  [____________] │    │
│  └─────────────────────────────┘    │
│                                     │
│  [ Create vault ]                   │
│                                     │
│  ─── or ───                         │
│                                     │
│  Already have a key?                │
│  [ Paste vault key ] ___________    │
│                       [ Open  ]     │
└─────────────────────────────────────┘
```

**Design notes:**
- Vault name is optional in Stage 1 (defaults to "My vault"). Required only for the password manager entry label in Stage 2.
- "Already have a key?" section is collapsed by default on small screens — expandable.
- No loading spinner; vault creation is instant (key generation is client-side in vault).
- No password or master passphrase. The vault key IS the credential.

#### State: Vault open (key saved in localStorage)

```
┌─────────────────────────────────────┐
│  🟢  My vault                        │
│      apple-river-1234               │
│                                     │
│  [ 📋 Copy key ]  [ Share ]         │
│  [ Switch vault ]                   │
└─────────────────────────────────────┘
```

**Design notes:**
- Show first 4 chars of vault key + ellipsis, full key on hover/focus (accessibility: screen readers read it in full).
- "Share" dispatches `vault-share-requested` event — host page decides what to do (open share modal, copy URL, etc.).
- "Switch vault" returns to the no-vault state WITHOUT clearing localStorage — lets users manage multiple vaults.
- Component is intentionally compact. It should fit in a toolbar or sidebar slot.

#### State: Multiple saved vaults (Stage 2 only — design preview)

```
┌─────────────────────────────────────┐
│  🔒  Choose a vault                 │
│                                     │
│  ● My vault          (last used)    │
│  ○ Work docs                        │
│  ○ Team infographics                │
│                                     │
│  [ Open selected ]                  │
│                                     │
│  [ + Add new vault ]                │
└─────────────────────────────────────┘
```

This state is Stage 2 territory (Credential Management API returns a list). Designed now so Dev can add it without breaking the Stage 1 API contract.

---

### 1.3 Key UX Decisions

| Decision | Recommendation | Reason |
|----------|----------------|--------|
| Show vault key by default? | Show truncated, full on hover | Key anxiety: hiding it makes users nervous they've lost it |
| Copy feedback | 2s "Copied!" label replaces copy icon | Standard clipboard UX pattern |
| Recovery prompt at creation | Optional popover: "Save your key somewhere safe" | Sherpa's ask — low friction, not a blocker |
| Key format display | `apple-river-1234` (human-readable) or `3a8f...b2d4` (raw hex) | Preserve whatever format the vault returns |
| Error state (bad key paste) | Inline red message under paste field: "That doesn't look like a valid vault key" | Never modal, never blocking |

---

### 1.4 Component API (for Dev)

```html
<!-- Basic usage -->
<sg-vault-picker></sg-vault-picker>

<!-- Pre-open a specific vault (deep-link or redirect) -->
<sg-vault-picker vault-key="apple-river-1234"></sg-vault-picker>

<!-- Events the host page should handle -->
vault-opened      → { detail: { vaultKey, vaultName } }
vault-created     → { detail: { vaultKey, vaultName } }
vault-key-copied  → { detail: { vaultKey } }
vault-share-requested → { detail: { vaultKey } }
vault-closed      → {}
```

**Attributes:**
- `vault-key` — pre-load a vault on mount (skips the picker UI)
- `compact` — removes the description text (toolbar mode)
- `label` — override the displayed vault name label

### 1.5 CSS Design Tokens

The component uses CSS custom properties for host-page theming. Defaults are designed for the SG/Send dark browse shell:

```css
sg-vault-picker {
  --vp-bg:           #1e1e2e;
  --vp-border:       #2e2e4a;
  --vp-text:         #d0d0e8;
  --vp-text-muted:   #808099;
  --vp-accent:       #4cc9b0;       /* teal — matches Push button */
  --vp-radius:       8px;
  --vp-font:         inherit;
}
```

The component MUST also render correctly against a light background (infographic editor, PlaybookLM tools). Use `prefers-color-scheme` media queries where the host page hasn't set custom properties.

## 2. Landing Page UX Direction

### 2.1 The "Why not WeTransfer?" Problem

User research from the Explorer CLAUDE.md confirms: **users immediately compare to WeTransfer in the first 10 seconds.** The current landing page doesn't address this. The differentiator must be obvious before the first scroll.

### 2.2 Above-the-fold Layout

```
┌──────────────────────────────────────────────────────┐
│  [SG/SEND logo]                          [Try it →]  │
├──────────────────────────────────────────────────────┤
│                                                      │
│  "Send files that even we can't read."               │
│                                                      │
│  No account. No app. Files encrypted in your         │
│  browser before they leave your device.              │
│                                                      │
│  [ Drop a file to try it ]                           │
│                                                      │
│  ─── Used by teams who can't afford a breach ────    │
│                                                      │
│  Healthcare · Legal · Finance · Journalists          │
│                                                      │
└──────────────────────────────────────────────────────┘
```

**Headline logic:** "even we can't read" is stronger than "zero-knowledge." It is concrete, human, and creates a structural claim that WeTransfer and Google Drive cannot match.

### 2.3 The Comparison Strip

Immediately below the fold — a single table that lands the WeTransfer comparison without naming WeTransfer:

```
┌────────────────┬──────────────┬──────────────┬──────────────┐
│                │  SG/Send     │  Standard    │  Enterprise  │
│                │              │  file share  │  DRM         │
├────────────────┼──────────────┼──────────────┼──────────────┤
│ Server sees    │  Ciphertext  │  Your files  │  Your files  │
│ your files?    │  only        │  ✓           │  ✓           │
├────────────────┼──────────────┼──────────────┼──────────────┤
│ View without   │  ✓ Browser   │  ✓           │  ✗ App       │
│ downloading    │  only        │              │  required    │
├────────────────┼──────────────┼──────────────┼──────────────┤
│ Account needed │  No          │  Often       │  Yes         │
├────────────────┼──────────────┼──────────────┼──────────────┤
│ Key stays with │  You         │  Provider    │  Provider    │
│                │              │              │              │
└────────────────┴──────────────┴──────────────┴──────────────┘
```

**Design guidance:** Keep the table tight — 3 columns, 4 rows. No brand names. Let the reader fill in "standard file share" = WeTransfer/Dropbox/Drive.

### 2.4 Social Proof Placement

Below the comparison strip:
- One sentence from a healthcare user (anonymised, role + sector only)
- One sentence from a journalist/legal professional
- Transfer counter ("X files sent this month — none readable by us")

The counter is technically honest: it shows transfer count, not file contents. The "none readable by us" is architectural fact, not a promise.

## 3. Sherpa Design Request: Recovery UX at Vault Creation

From the auth MVP (doc 297): Sherpa should design a low-friction recovery prompt shown once at vault creation.

**Designer's recommendation:**

Show a popover (not modal, not blocking) immediately after vault creation:

```
┌─────────────────────────────────────────┐
│  ✓ Vault created — apple-river-1234     │
│                                         │
│  📋 [ Copy key ]   📥 [ Download .txt ] │
│                                         │
│  The key above unlocks your vault on    │
│  any device. If you lose it, the vault  │
│  cannot be recovered. There is no       │
│  reset link — by design.                │
│                                         │
│  [ Got it, continue → ]                 │
└─────────────────────────────────────────┘
```

This popover:
- Is NOT a modal — does not block interaction
- Has two save affordances (copy + download .txt)
- Uses honest language: "cannot be recovered" (not "please back up")
- Is shown ONCE and never again (suppressed by a `localStorage` flag)
