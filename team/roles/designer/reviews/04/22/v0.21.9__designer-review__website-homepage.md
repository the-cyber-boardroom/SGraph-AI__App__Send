# Designer Review: Website Homepage
**Version:** v0.21.9 | **Date:** 2026-04-22 | **URL reviewed:** `https://dev.sgraph.ai/en-gb/`
**Role:** Designer | **Method:** Live page fetch, CSS source audit, screenshots from `sgraph_ai__website/_common/images/sg-send/use-cases/`

---

## 1. Executive Summary

The design system (Aurora theme) is well-constructed — consistent token naming, a clear 4px spacing system, good typographic hierarchy. The breakdown happens at the page composition level: the hero section is disconnected from its own opening element (the encryption badge floats in a sea of dark space above the headline), the third primitive card wastes the slot on "Code Execution Sandbox / Coming Soon" instead of a shipped differentiator like PKI, and the privacy section — the strongest trust artifact on the page — is visually indistinguishable from a generic content block. The design system has the vocabulary to fix all three; the page just isn't using it yet.

---

## 2. Hierarchy and Information Architecture

**What works:**
- Clear section rhythm: hero → primitives → use cases → tools → pricing → privacy → open source → footer
- The `section--alt` alternating background creates visual separation without borders
- The use-case alternating layout (normal / `--reverse`) creates a natural reading flow

**What doesn't:**
- **Two deployment target elements on the same page.** The `deploy-strip` (minimal text badges under the hero CTAs) and the `aws-strip` (icon-node diagram below the hero) communicate nearly identical information — deployment targets — in slightly different ways. One communicates to a casual user; neither communicates effectively. Duplication fragments the user's attention without adding information density.
- **The primitives section arrives before the product is explained.** A first-time visitor reads "missing primitives for agentic development" before they understand what SG/Send does. The section assumes knowledge the page hasn't yet provided.
- **Page length vs. content earned.** Five use-case rows × 260px minimum height each = significant scroll depth before reaching pricing. The use cases need screenshots to justify that vertical space. Currently they are text columns next to dashed-border placeholders.

## 3. Visual Design and Consistency

### Design system: strong foundation
The Aurora theme CSS is well-organised:
- Semantic tokens are correctly separated from primitive values (`--accent` not `#4ECDC4`)
- 4px base spacing grid applied consistently throughout (`--sp-1` through `--sp-24`)
- Responsive breakpoints at 1024px, 768px, 480px — correct order, consistent handling
- Button variants (primary, secondary, danger, ghost, sherpa) are coherent and correctly token-referenced

### Issues

**DES-01 — Encryption badge is visually orphaned** *(Principle: Hierarchy / Breathing room)*
The `.enc-badge` appears at the very top of the hero section, preceded only by the section's top padding (`var(--sp-24)` = 96px). On screen this renders as a small pill floating in a large dark void before the headline appears. The badge is meant to be a trust signal that primes the headline — but its placement and the large empty space above it make it look like a loading artifact rather than an intentional design element.

- **Principle violated:** Hierarchy — the most important trust signal should be visually attached to what it qualifies, not floating independently
- **Impact:** Users scanning quickly may not connect the badge to the headline; it reads as decoration rather than certification
- **Recommendation:** Reduce the top padding of the hero to `var(--sp-16)` (64px). Add a subtle visual connector between the badge and the `h1` — the simplest fix is reducing the `margin-bottom` on `.enc-badge` from `24px` to `12px`. The badge should feel like it introduces the headline, not precede it from a distance.

**DES-02 — Third primitive card: wrong content for the slot** *(Principle: Honesty / Hierarchy)*
The three primitives should represent the three most compelling, differentiating capabilities of the platform. Slot 1 (Browser Automation) and Slot 2 (Encrypted Virtual File System) are both available now. Slot 3 (Code Execution Sandbox) is labelled "Coming Soon" — meaning the third most prominent feature card on the page actively tells users something doesn't exist yet. Worse, **PKI (Public Key Infrastructure)** — a shipped or near-shipped differentiator that no competitor offers — is absent entirely.

- **Principle violated:** Honesty — the card accurately says "coming soon", but the choice to put the weakest available capability in the highest-visibility slot misrepresents the product's strength
- **Impact:** The three primitives are the product's core identity on this page. Leading with an unavailable feature in that identity slot undermines confidence in slots 1 and 2
- **Recommendation:** Replace the Code Execution Sandbox card with a **PKI / Identity** primitive card. The Sandbox card can move to a "What's coming" section further down the page. PKI provides a compelling story: cryptographic identity for agents and humans, trust without central authority, verifiable authorship of vault commits.

**DES-03 — Privacy section lacks visual weight for its importance** *(Principle: Hierarchy / Delight)*
The privacy section — "Our privacy model in six sentences" — is SG/Send's most powerful trust artifact. The copy is excellent (six clear, legally-grounded, technically honest sentences). The design buries it. The `.privacy-inline` block is a plain rounded rectangle on a `section--alt` background, with small-caps teal label text and body-weight grey copy. It reads like a footnote.

- **Principle violated:** Hierarchy — the importance of the content is not expressed by its visual treatment. The design says "standard content block"; the content is "our entire value proposition for trust"
- **Impact:** Users who most need to be convinced by the privacy story — security researchers, healthcare, enterprise buyers — will scan past it
- **Recommendation:** Elevate the privacy block with one or more of these treatments:
  - Add a left accent border (4px solid `var(--accent)`) to the entire `.privacy-inline` container, matching the `.callout` pattern already in the design system
  - Increase body font size to `1rem` (currently `0.9rem`) and raise line-height to `2` for breathing room
  - Give each sentence its own numbered block or bullet with a teal check (`✓`) — the six-sentences structure is a feature; make it visible
  - Move this section immediately above the pricing CTA so trust is built right before the ask

**DES-04 — Use-case alternating layout uses `direction: rtl` for reversal** *(Principle: Details / Honesty)*
The `.use-case-row--reverse` pattern reverses column order using `direction: rtl` — a CSS shortcut that works visually but is semantically incorrect (RTL is for right-to-left languages, not layout reversals) and creates issues for i18n and screen readers. The page already has a locale picker and i18n infrastructure — this will cause problems when RTL locales (Arabic, Hebrew) are added.

- **Principle violated:** Good design is thorough — using a language-direction property as a layout hack is a detail that will betray its foundation when the system is under stress (RTL locales)
- **Impact:** Screen readers may announce content in wrong order on reversed rows; actual RTL locale support will be broken for these rows
- **Recommendation:** Replace with CSS grid column ordering:
  ```css
  .use-case-row--reverse { }  /* no direction: rtl */
  .use-case-row--reverse .use-case-row__content { order: 2; }
  .use-case-row--reverse .screenshot-placeholder { order: 1; }
  ```

**DES-05 — Token bar is hidden on mobile** *(Principle: Consistency / Delight)*
At `max-width: 768px`, `.token-inline-bar { display: none; }`. It only reappears when `.is-open` is toggled. But the trigger button that would open it is also hidden. This means mobile users — who are the most likely recipients of a share link — cannot access the primary viral loop entry point from the homepage header without scrolling to find it elsewhere.

- **Principle violated:** Consistency — the token bar is positioned as a persistent, always-available affordance in the desktop layout; mobile breaks that promise
- **Impact:** The highest-intent mobile visitor (recipient with a token) has no obvious entry point
- **Recommendation:** On mobile, show the trigger button ("Paste a token →") persistently in the header even when the input is collapsed. The expanded state can be a full-width dropdown below the header. The trigger must always be visible.

**DES-06 — Emoji icons are inconsistent across sections** *(Principle: Consistency / Details)*
The page uses emoji for all icons but the usage is inconsistent:
- Primitive cards: 🌐, 🔐, ⚙️ — decorative emoji
- AWS strip: λ (text), S3 (text), 📦, ⚡, 🖥 — mixed text and emoji
- Use case tags: none (text only)
- Open source cards: 🐙, ⌨️, 🎭 — decorative emoji

The AWS strip mixes text labels (λ, S3) with emoji in identical-sized icon containers. This breaks visual rhythm. The λ and S3 labels are more precise and professional; the emoji are warmer but less consistent with the overall Aurora theme's technical register.

- **Principle violated:** Consistency — same class of element (icon) rendered with different systems
- **Recommendation:** Standardise the AWS strip to text-based labels throughout (λ, S3, ECR, CF, EC2), or convert all to a single consistent emoji or SVG set. Don't mix systems in the same component.

**DES-07 — `btn-sherpa` is a named internal role in global CSS** *(Principle: Honesty)*
Line 603 of `style.css` defines `.btn-sherpa` — a button variant named after an internal team role. This is an internal naming convention that has leaked into the public design system. If inspected in browser DevTools by an external developer, it reveals internal team structure.

- **Recommendation:** Rename to `.btn-highlight` or `.btn-featured` in the design system. Keep the Sherpa name only in internal documentation.

## 4. Screenshot Placeholders — Structural Issue

The five `.screenshot-placeholder` blocks are built as `<div>` containers with label and description text. When real screenshots are added, the entire element needs to be replaced with an `<img>` (plus caption structure) — this is not a simple `src` attribute update.

A commit on 2026-04-22 (`57a2b0e`) added real screenshots to `sgraph_ai__website/_common/images/sg-send/use-cases/`:
- `01-vault-browse.png`, `02-markdown-article.png`, `03-gallery.png`, `04-upload-flow.png`, `05-push-pull-badge.png`, `07-send-main.png`, `07-vault-landing.png`, `08-token-receive.png`

These screenshots exist in the repo but are **not yet wired into the homepage HTML**. This should be the highest-priority visual fix — the assets exist, the HTML just needs updating.

**Recommendation:** Replace each `.screenshot-placeholder` div with:
```html
<figure class="use-case-screenshot">
  <img src="path/to/screenshot.png" alt="[descriptive alt text]" loading="lazy">
</figure>
```
And add a `.use-case-screenshot` CSS rule with consistent border-radius and shadow matching the card system.

## 5. Use Case Infographics — Missing Content Gap

The current use-case section presents five vertically-stacked text blocks with screenshot placeholders. This works for known-feature illustration but misses the opportunity to communicate the **three interaction patterns** that define SG/Send's value:

| Pattern | Description |
|---|---|
| **Human → Human** | Person sends an encrypted vault; recipient opens it with a token |
| **Agent → Human** | An LLM/agent produces output into a vault; human accesses it with a token |
| **Agent → Agent** | Two agents collaborate via a shared vault: one writes, one reads |

None of these three patterns is visualised anywhere on the page. They are implied by the use-case copy but not shown. A simple three-panel diagram — one row, three columns, each showing an arrow with actors (human icon, agent icon) and a vault in between — would communicate the core proposition faster and more memorably than any amount of copy.

**Recommendation:** Add a "Three ways to use SG/Send" infographic section between the hero/primitives section and the use-case rows. Three side-by-side panels:
- **Human → Vault → Human** (file sharing, send.sgraph.ai flow)
- **Agent → Vault → Human** (agent produces briefing/report; human reads)
- **Agent → Vault → Agent** (multi-agent pipeline; audit trail preserved)

Each panel: actor icon, arrow, vault icon, arrow, recipient icon. Single sentence caption. This is the clearest possible communication of the "For humans and agents" headline claim.

## 6. IFD / Technical Alignment

**Font path is brittle:**
```html
<link rel="stylesheet" href="../../../../_common/fonts/fonts.css">
```
This relative path ascends four directory levels. If the page is ever served from a different depth, fonts silently fail. The CSS asset path (`../_common/css/style.css`) only ascends one level — inconsistent convention within the same `<head>`.

**`body.i18n-loading { visibility: hidden }`** hides the page until JS executes. There is no `<noscript>` fallback and no timeout — if the i18n script errors or is slow, the page is blank with no recovery. A `setTimeout` to remove the class after 500ms would prevent blank-page states.

**Screenshot file naming inconsistency:** The committed screenshots include `07-send-main.png` and `07-vault-landing.png` — two files with the same numeric prefix (`07`). File `06` is missing. This is probably an error in the naming sequence that should be corrected before the files are wired into the HTML.

### Detailed Findings

#### R2 — Encryption badge fix
- **Issue:** The `.enc-badge` floats in ~96px of dark space above the headline, disconnected from what it qualifies
- **Principle:** Hierarchy — the trust signal should visually introduce the headline, not precede it from a distance
- **Impact:** Scannability — fast readers miss the connection between badge and claim
- **Recommendation:** `hero { padding-block-start: var(--sp-16); }` and `.enc-badge { margin-bottom: 12px; }`

#### R3 — Third primitive card: PKI
- **Issue:** "Code Execution Sandbox / Coming Soon" occupies the third most-prominent identity slot on the page
- **Principle:** Honesty — the slot communicates "this is what we are"; a coming-soon feature is not what we are yet
- **Impact:** Reduces confidence in the two available primitives; PKI is a real differentiator going unused
- **Recommendation:** New card: **PKI / Cryptographic Identity** — "Each vault commit is signed. Each agent has a verifiable identity. Trust without central authority." Badge: "Available now." Icon: 🔑 or a key SVG.

#### R4 — Interaction pattern infographic
- **Issue:** "For humans and agents" is the headline claim but is never visualised
- **Principle:** Show, do not just tell — the design should communicate what the copy claims
- **Impact:** The most differentiating claim on the page (agent-native) is invisible to a visual scanner
- **Recommendation:** Three-column diagram section. Simple SVG or CSS-drawn icons suffice — no external library needed. Sits between the hero and the primitives section.

#### R5 — Privacy section elevation
- **Issue:** Best trust copy on the page has weakest visual treatment
- **Principle:** Hierarchy — visual weight should reflect content importance
- **Impact:** The users who need to read this most (enterprise, healthcare, security) are the ones most likely to scan past a grey text block
- **Recommendation:** Add `border-left: 4px solid var(--accent)` to `.privacy-inline`, increase inner font size to `1rem`, consider numbered list format matching the "six sentences" heading.

## 8. Missing Pages and Content — Structural Gaps

### SGit and Vaults need a dedicated page

The SGit CLI currently appears as a single small card in the "Open Source" section at the bottom of the homepage:

> **SGit CLI** — Command-line vault operations: init, commit, push, pull, share. → View on GitHub

This is the equivalent of describing a car by saying "has an engine." SGit + Vaults is the core product primitive — the thing that makes SG/Send different from every other file sharing tool. It deserves a full dedicated page (`/en-gb/vaults/`) covering:

- **What a vault is** — an encrypted, version-controlled, scoped file system
- **Three access modes:** Online (browser UI), API (programmatic), CLI (sgit commands)
- **The commit model** — every file operation is a commit, full audit trail
- **Key types** — vault key (read/write, persistent), share token (read-only snapshot)
- **Multi-agent use** — each agent gets a vault key scoped to one vault only
- **The browse UI** — including folder view, gallery view, markdown rendering

Until this page exists, the homepage is carrying all of this weight inline, which it cannot do well. The three primitive cards, the use-case rows, and the open-source section are all trying to explain vaults in fragments. A dedicated Vaults page lets the homepage say "Encrypted virtual file system →" with a link, and then explain it properly in one place.

**Recommendation:** Add `Vaults` to the main navigation (between `Security` and `Pricing`). Create `/en-gb/vaults/` as a dedicated product page. Move the SGit CLI card off the bottom of the homepage and into this page.

---

### Folder view and gallery view are invisible on the homepage

The vault browse UI has multiple view modes that are genuinely differentiating:
- **Folder view** — hierarchical navigation of vault contents
- **Gallery view** — image grid layout for photo/media vaults
- **Markdown rendering** — articles and documents render formatted in-browser

None of these are mentioned or shown anywhere on the homepage. A recipient opening a vault gets a gallery view of images, a formatted markdown article, or a folder tree — and the homepage never primes them to expect or appreciate this. These view modes are strong visual moments that screenshots alone would communicate instantly.

**Recommendation:**
1. Include folder view and gallery view screenshots in the use-case section (the assets `01-vault-browse.png`, `02-markdown-article.png`, `03-gallery.png` are already committed)
2. Add explicit callout bullets to the "Encrypted Virtual File System" primitive card: "Folder view, gallery view, markdown rendering — all in-browser, all zero-knowledge"
3. On the dedicated Vaults page, lead with these view modes as the first demonstration of what a vault looks like when opened

---

### Three access modes for vaults need explicit treatment

Vaults are accessible in three fundamentally different ways — and this is a significant architectural differentiator:

| Mode | Entry point | Who uses it |
|---|---|---|
| **Online** | Browser at `send.sgraph.ai` | Humans, recipients with tokens |
| **CLI** | `sgit` commands (init, commit, push, pull, share) | Developers, agents with shell access |
| **API** | Programmatic HTTP access | Agents, automation pipelines |

This three-mode story is currently invisible. The homepage mentions sgit CLI (once, in a small card) and the browser UI (implied by the hero CTA). The API mode is not mentioned at all. For a product positioning itself as "for humans and agents," the three access modes are a primary differentiator — agents don't use browsers, they need the CLI or API.

**Recommendation:** Add a "Three ways to access your vault" callout or diagram, either on the homepage or on the dedicated Vaults page. The diagram is simple: three columns (Online / CLI / API), each with its entry point, a brief description, and who uses it. This directly supports the "for humans and agents" headline claim by making the agent access paths explicit.
