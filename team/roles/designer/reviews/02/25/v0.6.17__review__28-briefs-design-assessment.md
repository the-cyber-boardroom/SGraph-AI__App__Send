## 1. UX Flows Required

### Dual Authorization (Brief #3) -- Sender Side

Four-state sender dashboard: Waiting > Receiver Ready > Released / Denied. The sender needs a clear status card showing challenge code, expiry countdown, and delivery requirements. After receiver responds, the environment report replaces the waiting state and presents three actions: Release, Deny, Ask to Change Environment. Each state transition must feel deliberate -- no accidental releases.

### Dual Authorization -- Receiver Side

Two-screen flow: (a) challenge code entry with transparency notice, (b) confirmation screen showing what data will be shared. The transparency notice is a consent gate -- the receiver sees exactly what is collected (approximate location, browser, secure connection status) and what is not shared (exact IP, identity, cookies). Accept or Decline buttons. Declining still notifies the sender.

### Drag-and-Drop Encrypted Email (Brief #25)

Two zones on the main page: Encrypt (drop file + select recipient key = download .sgenc) and Decrypt (drop .sgenc/.age/.gpg = plaintext). File naming options (named, type-only, fully opaque) presented as a simple dropdown during encryption. Template email copy shown as a copiable card after encryption completes. No server round-trip required for either zone.

### Key Import/Export (Brief #26)

Import: drag-and-drop zone that auto-detects format (SSH, GPG, PEM, JWK, native .sgkey). Copy-paste alternative via a text field. Password manager autofill via a standard password-type input. Export: one-click download with a passphrase prompt. QR code generation for device-to-device transfer displayed as a modal.

## 2. Evaluator Journey Design (Brief #9)

Five progressive demo levels, each discoverable from within the previous level (capture-the-flag model):

| Level | UX Goal | Key Design Element |
|-------|---------|-------------------|
| L1 | 30-second win -- decrypt one file | Minimal page, large drop zone, code input |
| L2 | Browse a shared folder | File listing with decrypt-on-click |
| L3 | Branded data room | Logo, colour scheme, folder tree |
| L4 | Business scenario (NDA exchange) | Workflow timeline, multi-party view |
| L5 | Live agent activity | Real-time event feed, status indicators |

The Sherpa's role: each level needs contextual guidance copy ("What you're seeing", "What to try next", "Why this matters"). Pace matching means no forced tutorials -- sophisticated evaluators skip ahead, casual explorers get gentle prompts.

## 3. Transparency Notice Design (Brief #1)

The receiver environment transparency notice is a trust-critical UI moment. Design principles:

- **Plain language** -- no jargon. "Your approximate location" not "IP geolocation."
- **Two columns** -- "We share" vs "We do NOT share" -- immediate clarity.
- **Non-blocking for standard delivery** -- only shown when sender requests environment verification.
- **Posture score** shown to sender as a single word (HIGH / MEDIUM / LOW) with expandable detail. Colour-coded but not red/green (accessibility). Use blue/amber/grey.

## 5. IFD Versioning Plan

All new components follow IFD methodology (zero dependencies, surgical versioning):

- **v0.7.x path**: `v0/v0.7/v0.7.0/` for dual-auth and drag-and-drop components (the "implementable now" briefs #3, #25, #26).
- **v0.8.x path**: `v0/v0.8/v0.8.0/` for demo journey components (brief #9) -- depends on data room features.
- Each component gets its own versioned directory. Breaking changes increment the minor version. Bug fixes increment the patch.
- Components are standalone HTML/JS files with no build step. They register as Custom Elements and communicate via DOM events.
- The encrypt/decrypt zones (#25) are the highest priority -- they enable the drag-and-drop email workflow which the brief marks as "implementable right now."

---

## Recommendations

1. **Start with the encrypt/decrypt zones** -- they deliver immediate user value and have zero dependencies on unbuilt infrastructure.
2. **Design the transparency notice early** -- it sets the trust tone for the entire dual-auth flow.
3. **Demo journey is a marketing asset** -- treat Level 1 as a landing page, not just a demo. It is the first impression for every WhatsApp-shared link.
4. **Key import UX must be forgiving** -- auto-detect format, clear error messages, never ask the user what format their key is in.
