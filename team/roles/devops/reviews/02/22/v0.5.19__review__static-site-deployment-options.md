## Context

The SSH Key Generator tool (`sgraph_ai_app_send__ui__user/tools/ssh-keygen/v0/v0.1/v0.1.0/`) is a browser-only static site with **zero backend dependencies**. Everything runs client-side using the Web Crypto API. The CSP header enforces `connect-src 'none'` -- after the page loads, no network requests are made.

This makes it an ideal candidate for standalone static site deployment, separate from the Lambda-based SG/Send application infrastructure.

### Asset Inventory

| File | Size | Purpose |
|------|------|---------|
| `index.html` | 3.9 KB | Entry point, CSP, structured data, font loading |
| `css/ssh-keygen.css` | 13.3 KB | Page-specific styles |
| `js/ssh-crypto.js` | 13.1 KB | Crypto library (Ed25519 + RSA 4096, pure Web Crypto API) |
| `js/ssh-keygen-app.js` | 17.1 KB | Web Component (`<ssh-keygen-app>`) |
| **Shared:** `v0/v0.1/v0.1.0/css/common.css` | 10.5 KB | Base reset, layout, buttons, forms |
| **Shared:** `v0/v0.1/v0.1.6/css/sg-design-system.css` | 17.0 KB | Aurora theme tokens, design system |
| **Total** | **~75 KB** | Before gzip (~20-25 KB gzipped) |

### External Dependencies

- **Google Fonts** (DM Sans, JetBrains Mono) -- loaded from `fonts.googleapis.com` / `fonts.gstatic.com`
- **No JavaScript dependencies** -- zero npm packages, zero CDN scripts
- **No backend API calls** -- `connect-src 'none'` enforced

### Key Constraint: Shared CSS via Relative Paths

The `index.html` references shared CSS using deep relative paths:

```html
<link rel="stylesheet" href="../../../../../v0/v0.1/v0.1.0/css/common.css">
<link rel="stylesheet" href="../../../../../v0/v0.1/v0.1.6/css/sg-design-system.css">
```

These resolve within the monorepo's IFD versioned directory structure (`tools/ssh-keygen/v0/v0.1/v0.1.0/` climbing 5 levels up to `sgraph_ai_app_send__ui__user/`). For any standalone deployment, this must be resolved.

## 1. GitHub Pages Deployment

GitHub Pages is already in use for this repo (see `.github/workflows/jekyll-pages.yml` deploying the journalist site). There are three approaches:

### Option A: Separate Repository (Recommended)

Create a dedicated repo (e.g., `SGraph-AI/keygen` or `SGraph-AI/ssh-keygen`) containing only the built, self-contained static files.

**Setup:**
1. Create new repo under the `SGraph-AI` (or `the-cyber-boardroom`) GitHub org
2. Enable GitHub Pages on the `main` branch, root directory (`/`)
3. Push the flattened site structure:
   ```
   /
     index.html
     css/
       common.css
       sg-design-system.css
       ssh-keygen.css
     js/
       ssh-crypto.js
       ssh-keygen-app.js
   ```
4. Configure custom domain (see Section 6)

**Pros:**
- Cleanest separation of concerns
- GitHub Pages per-repo is free and has its own deployment pipeline
- No conflict with the existing journalist Jekyll site that already uses this repo's Pages
- Independent release cycle -- ship keygen updates without touching SG/Send
- Simpler CI -- just push to `main` and Pages deploys automatically

**Cons:**
- Requires a build/copy step from this monorepo to the standalone repo
- Two repos to maintain (though the standalone repo is trivially small)

### Option B: Subdirectory in This Repo via GitHub Pages

Use the `docs/` directory or a `gh-pages` branch.

**Problem:** This repo already uses GitHub Pages for the journalist Jekyll site at `team/roles/journalist/site/`. GitHub Pages only supports **one site per repo**. The existing workflow deploys from `team/roles/journalist/site/_site`. Adding a second static site would require merging both into a single Pages deployment, which is messy and couples unrelated concerns.

**Verdict:** Not viable without restructuring the existing Pages setup. Avoid.

### Option C: GitHub Pages via a `gh-pages` Branch

Push the built static files to a `gh-pages` branch. The journalist site currently uses the Actions deploy flow (artifact upload + `actions/deploy-pages@v4`), not a branch-based deployment.

**Problem:** Same single-site-per-repo limitation. The `gh-pages` branch would conflict with the existing Pages deployment.

**Verdict:** Not viable. Use Option A.

## 2. Other Static Hosting Options

### Cloudflare Pages

**How:** Connect a repo (or use direct upload via Wrangler CLI). No build step needed for pre-built static files.

**Pros:**
- Free tier: unlimited bandwidth, 500 deploys/month
- Global CDN with edge caching
- Automatic HTTPS
- Custom domains included
- Preview URLs for every branch/PR
- Built-in Web Analytics (privacy-friendly)

**Cons:**
- Requires Cloudflare account
- Another vendor in the stack

**Fit:** Excellent. The tool's security posture (zero backend, `connect-src 'none'`) aligns perfectly with Cloudflare's edge-first model. Cloudflare Pages would also be the natural choice if SGraph already uses Cloudflare for DNS.

### Netlify

**How:** Connect a GitHub repo, deploy on push. No build command needed.

**Pros:**
- Free tier: 100 GB bandwidth/month, 300 build minutes
- Preview deploys per PR
- Custom headers/redirects via `_headers` / `_redirects` files
- Form handling (not needed here, but available)

**Cons:**
- 100 GB bandwidth limit on free tier (unlikely to be hit for a tool page, but worth noting)
- Another account to manage

**Fit:** Good. Slightly more limited free tier than Cloudflare Pages.

### Vercel

**How:** Connect repo, deploy on push.

**Pros:**
- Free tier, generous bandwidth
- Preview deploys
- Edge network

**Cons:**
- Oriented toward Next.js/React; overkill for a static page
- Commercial use on free tier has restrictions

**Fit:** Adequate but not the best match. Vercel's strengths (serverless functions, ISR, edge middleware) are irrelevant here.

### S3 + CloudFront (AWS)

**How:** Upload static files to an S3 bucket, front with CloudFront distribution.

**Pros:**
- Already within the project's AWS ecosystem (SG/Send uses S3, Lambda, CloudFront)
- Full control over caching, headers, and security policies
- Can use the same CloudFront distribution as `send.sgraph.ai` with a path-based routing rule, or a separate distribution for the subdomain
- Existing CI pipeline already handles AWS credentials and deployments

**Cons:**
- More setup than managed platforms (bucket policy, CloudFront distribution, ACM certificate, Route 53 records)
- Cost: negligible for traffic, but not free-tier-forever like GitHub Pages or Cloudflare Pages
- More moving parts to maintain

**Fit:** Good if you want everything in one AWS account. The existing CI pipeline (`ci-pipeline.yml`) already has AWS credentials configured, so adding an S3 sync step is straightforward.

## 3. Separate Repo Strategy

### Option A: Extract to Dedicated Repo

**Repo name:** `SGraph-AI/keygen` (or `SGraph-AI__Keygen` following the existing naming pattern)

**What goes in it:**
```
/
  index.html          # Modified: CSS paths flattened, canonical URL updated
  css/
    common.css        # Copied from v0/v0.1/v0.1.0/css/common.css
    sg-design-system.css  # Copied from v0/v0.1/v0.1.6/css/sg-design-system.css
    ssh-keygen.css    # Copied as-is
  js/
    ssh-crypto.js     # Copied as-is
    ssh-keygen-app.js # Copied as-is
```

**Pros:**
- Clean, self-contained deployable unit
- Independent versioning and release cycle
- Can be open-sourced independently (the tool already has "open source" as a trust signal)
- Enables GitHub Pages, Cloudflare Pages, or any static host with zero config
- Simplifies the trust story: "view the source at github.com/SGraph-AI/keygen"
- SEO: dedicated repo = dedicated GitHub presence = more discoverability

**Cons:**
- Design system changes in the monorepo must be propagated manually (or via CI)
- Two repos to track
- Risk of CSS drift if the design system evolves and the keygen repo falls behind

### Option B: Keep in Monorepo, Build for Standalone

Deploy directly from this repo using a build script that copies and flattens the assets.

**Pros:**
- Single source of truth for all code
- Design system changes automatically included (if the build script is part of CI)
- No additional repo to manage

**Cons:**
- Build script adds complexity
- Deployment pipeline is coupled to the main SG/Send CI (which handles Lambda, PyPI, etc.)
- GitHub Pages conflict (see Section 1)
- Harder to open-source just the keygen tool

### Recommendation

**Go with the dedicated repo.** The tool is small (6 files, ~75 KB), self-contained, and has no runtime dependency on the monorepo. A dedicated repo makes deployment trivial and decouples the tool's release cycle from the main application. The risk of CSS drift is minimal because the design system is stable (v0.1.6) and the keygen tool only uses design tokens, not complex component CSS.

## 4. Asset Bundling Strategy

The core problem: `index.html` references shared CSS via `../../../../../` relative paths that only resolve within the monorepo's IFD directory structure.

### Option A: Copy at Build Time (Recommended)

A simple shell script (or CI step) copies the shared CSS files into the standalone site's `css/` directory and rewrites the `index.html` paths.

**Build script (`build.sh`):**
```bash
#!/usr/bin/env bash
set -euo pipefail

SRC="sgraph_ai_app_send__ui__user"
OUT="dist"

rm -rf "$OUT"
mkdir -p "$OUT/css" "$OUT/js"

# Copy tool-specific files
cp "$SRC/tools/ssh-keygen/v0/v0.1/v0.1.0/index.html"            "$OUT/index.html"
cp "$SRC/tools/ssh-keygen/v0/v0.1/v0.1.0/css/ssh-keygen.css"    "$OUT/css/ssh-keygen.css"
cp "$SRC/tools/ssh-keygen/v0/v0.1/v0.1.0/js/ssh-crypto.js"      "$OUT/js/ssh-crypto.js"
cp "$SRC/tools/ssh-keygen/v0/v0.1/v0.1.0/js/ssh-keygen-app.js"  "$OUT/js/ssh-keygen-app.js"

# Copy shared CSS
cp "$SRC/v0/v0.1/v0.1.0/css/common.css"                         "$OUT/css/common.css"
cp "$SRC/v0/v0.1/v0.1.6/css/sg-design-system.css"               "$OUT/css/sg-design-system.css"

# Rewrite CSS paths in index.html
sed -i 's|../../../../../v0/v0.1/v0.1.0/css/common.css|css/common.css|g'               "$OUT/index.html"
sed -i 's|../../../../../v0/v0.1/v0.1.6/css/sg-design-system.css|css/sg-design-system.css|g' "$OUT/index.html"

echo "Built standalone site in $OUT/ ($(du -sh "$OUT" | cut -f1))"
```

**Pros:**
- No build tools (npm, webpack, etc.) required -- just `cp` and `sed`
- Output is plain static files, debuggable, inspectable
- Preserves IFD philosophy (each file remains a distinct, readable unit)
- Easy to verify: diff the original and output

**Cons:**
- Must run the script when shared CSS changes
- Fragile if CSS paths change (though they're versioned and stable)

### Option B: Inline CSS into HTML

Use a build step to inline all three CSS files into `<style>` blocks in `index.html`, producing a single self-contained HTML file.

**Pros:**
- Single file deployment (just `index.html` + 2 JS files)
- Eliminates all path issues
- Fewer HTTP requests (though HTTP/2 makes this less relevant)

**Cons:**
- Larger HTML file (~45 KB with all CSS inlined)
- Loses separate CSS cacheability
- Harder to inspect/debug
- Breaks the clean separation of concerns

### Option C: Bundle with a Build Tool (esbuild, Vite, etc.)

Use a modern bundler to produce minified, hashed output.

**Pros:**
- Minification, tree-shaking, content hashing for cache busting
- Production-grade output

**Cons:**
- Introduces build tool dependency (node/npm)
- Overkill for 6 files / 75 KB
- Violates the project's "zero framework dependencies" principle for frontend
- Adds maintenance burden

### Recommendation

**Option A (copy at build time).** It's the simplest approach that solves the problem. No new tooling. The build script is 15 lines of bash and produces a clean, inspectable output. For a 75 KB site, bundling and minification provide negligible benefit.

## 5. CI/CD Pipeline

### For Dedicated Repo (GitHub Pages)

The simplest viable pipeline:

**In the monorepo (`SGraph-AI__App__Send`):**

Add a GitHub Actions workflow that runs the build script and pushes the output to the standalone repo:

```yaml
# .github/workflows/deploy-keygen.yml
name: Deploy SSH Keygen Static Site

on:
  push:
    branches: [dev]
    paths:
      - 'sgraph_ai_app_send__ui__user/tools/ssh-keygen/**'
      - 'sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.0/css/common.css'
      - 'sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.6/css/sg-design-system.css'
  workflow_dispatch:

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build standalone site
        run: bash scripts/build-keygen.sh

      - name: Deploy to standalone repo
        uses: peaceiris/actions-gh-pages@v3
        with:
          deploy_key: ${{ secrets.KEYGEN_DEPLOY_KEY }}
          external_repository: SGraph-AI/keygen
          publish_branch: main
          publish_dir: ./dist
          commit_message: "deploy: keygen from SGraph-AI__App__Send@${{ github.sha }}"
```

**In the standalone repo (`SGraph-AI/keygen`):**

Enable GitHub Pages on the `main` branch. No workflow needed -- GitHub Pages automatically serves the content.

### For S3 + CloudFront

Add a step to the existing CI pipeline:

```yaml
- name: Deploy keygen to S3
  run: |
    bash scripts/build-keygen.sh
    aws s3 sync dist/ s3://${{ secrets.KEYGEN_S3_BUCKET }}/ \
      --delete \
      --cache-control "public, max-age=86400"
    aws cloudfront create-invalidation \
      --distribution-id ${{ secrets.KEYGEN_CF_DISTRIBUTION_ID }} \
      --paths "/*"
```

### For Cloudflare Pages

No CI workflow needed in the monorepo. Connect the standalone repo directly to Cloudflare Pages. Every push to `main` triggers a deploy automatically.

## 6. Custom Domain

### DNS Setup

Assuming the target subdomain is `keygen.sgraph.ai` (shorter and cleaner than `keygen.send.sgraph.ai`):

**For GitHub Pages:**
1. In the standalone repo's Settings > Pages > Custom domain, enter `keygen.sgraph.ai`
2. Add a CNAME record in DNS:
   ```
   keygen.sgraph.ai  CNAME  sgraph-ai.github.io
   ```
3. GitHub auto-provisions a Let's Encrypt TLS certificate
4. Update `index.html` canonical URL: `<link rel="canonical" href="https://keygen.sgraph.ai">`

**For Cloudflare Pages:**
1. In Cloudflare Pages project settings, add custom domain `keygen.sgraph.ai`
2. If DNS is already on Cloudflare, the CNAME is added automatically
3. If DNS is elsewhere, add:
   ```
   keygen.sgraph.ai  CNAME  keygen-site.pages.dev
   ```

**For S3 + CloudFront:**
1. Request ACM certificate for `keygen.sgraph.ai` (in `us-east-1` for CloudFront)
2. Create CloudFront distribution with the S3 bucket as origin
3. Add alternate domain name `keygen.sgraph.ai` to the distribution
4. Add DNS record:
   ```
   keygen.sgraph.ai  CNAME  d1234567890.cloudfront.net
   ```
   (or an A/AAAA alias record if using Route 53)

### Alternative Subdomain Options

| Domain | Pros | Cons |
|--------|------|------|
| `keygen.sgraph.ai` | Short, clean, tool-level branding | Requires DNS on `sgraph.ai` zone |
| `keygen.send.sgraph.ai` | Clearly under SG/Send | Longer, nested subdomain |
| `ssh-keygen.sgraph.ai` | Descriptive | Longer |
| `keys.sgraph.ai` | Short, memorable | Might be too generic if other key tools are added |

**Recommendation:** `keygen.sgraph.ai` -- short, clear, and positions the tool as an SGraph offering (not just an SG/Send sub-feature).

## 7. Recommendation: Simplest Path to Live

### The Plan (3 Steps)

**Step 1: Create the standalone repo and build script** (~30 minutes)

1. Create `SGraph-AI/keygen` repo on GitHub (public)
2. Add the `build-keygen.sh` script to the monorepo at `scripts/build-keygen.sh`
3. Run it locally, verify the output works by opening `dist/index.html` in a browser
4. Push the `dist/` contents to the standalone repo's `main` branch

**Step 2: Enable GitHub Pages** (~5 minutes)

1. Go to `SGraph-AI/keygen` > Settings > Pages
2. Source: Deploy from branch, `main`, root (`/`)
3. The site is immediately live at `sgraph-ai.github.io/keygen`

**Step 3: Configure custom domain** (~15 minutes)

1. Add `keygen.sgraph.ai` CNAME record pointing to `sgraph-ai.github.io`
2. Set custom domain in repo Settings > Pages
3. Wait for TLS certificate provisioning (usually < 10 minutes)
4. Update the `index.html` canonical URL and test

**Step 4 (optional): Add CI workflow** (~15 minutes)

Add `.github/workflows/deploy-keygen.yml` to the monorepo so changes to the keygen tool or shared CSS automatically rebuild and push to the standalone repo.

### Why GitHub Pages Over Alternatives

| Criteria | GitHub Pages | Cloudflare Pages | S3 + CloudFront |
|----------|-------------|-------------------|------------------|
| Cost | Free | Free | ~$0.50/month |
| Setup time | 5 min | 10 min | 30+ min |
| Custom domain | Built-in | Built-in | Manual (ACM + DNS) |
| TLS | Automatic | Automatic | Manual (ACM cert) |
| CDN | GitHub's CDN | Cloudflare edge | CloudFront edge |
| CI integration | Push to branch | Push to branch | AWS CLI in CI |
| Already in use | Yes (journalist site) | No | Yes (SG/Send infra) |
| New accounts needed | None | Cloudflare account | None |
| Maintenance | Near-zero | Near-zero | Low |

GitHub Pages wins on simplicity. The tool is a small static site with no dynamic requirements. GitHub Pages deploys on push, provisions TLS automatically, and requires no additional accounts or infrastructure. If traffic grows significantly or advanced features (custom headers, edge functions, analytics) are needed, migrating to Cloudflare Pages is straightforward.

### Migration Path

If GitHub Pages proves insufficient:

1. **GitHub Pages** (now) -- get it live quickly
2. **Cloudflare Pages** (if needed) -- better CDN, Web Analytics, custom headers
3. **S3 + CloudFront** (if needed) -- full AWS integration, custom caching, WAF

Each migration is a DNS change + deploy target change. The static files are the same everywhere.
