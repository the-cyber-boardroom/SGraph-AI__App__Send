## 1. Executive Summary

The Workspace UI deployment pipeline (`deploy-ui-workspace.yml`) already supports three environments (dev, main, prod) in its logic, but there are gaps in infrastructure and configuration that prevent Main and Prod from actually working. Real users are on `dev.workspace.sgraph.ai` today, which is the Dev environment -- shared with CI and QA. Users need to be migrated to Main or Prod.

**Key finding:** The workflow file is well-structured and already handles branch-based environment resolution. The blockers are infrastructure (CloudFront distributions, DNS records) and GitHub secrets configuration, not code changes.

### 2.3. Workspace Pipeline -- Detailed Analysis

The `deploy-ui-workspace.yml` workflow:

- **Triggers on:** Push to `dev` or `main` branches (when workspace UI files change), plus manual `workflow_dispatch`
- **Environment resolution logic:**
  - `production` (manual dispatch) -> S3 path `websites/sgraph-workspace/prod/`, uses `SGRAPH_WORKSPACE__CLOUDFRONT_DISTRIBUTION_ID`
  - `main` (push to main branch) -> S3 path `websites/sgraph-workspace/main/`, uses `SGRAPH_WORKSPACE__DEV__CLOUDFRONT_DISTRIBUTION_ID`
  - `dev` (push to dev branch) -> S3 path `websites/sgraph-workspace/dev/`, uses `SGRAPH_WORKSPACE__DEV__CLOUDFRONT_DISTRIBUTION_ID`

**Critical observation:** Main and Dev share the same CloudFront distribution ID secret (`SGRAPH_WORKSPACE__DEV__CLOUDFRONT_DISTRIBUTION_ID`). This means either:
1. They share a single CloudFront distribution with different S3 origin paths (likely), or
2. The Main distribution hasn't been created yet and the secret is a placeholder

This mirrors the pattern used by User UI and Vault UI where dev and main share a CF distribution.

## 4. Proposed Pipeline Architecture

### 4.1. Environment Model

| Environment | Domain | Branch | Trigger | Purpose |
|------------|--------|--------|---------|---------|
| **Dev** | dev.workspace.sgraph.ai | `dev` | Auto on push | CI/QA, developer testing |
| **Main** | main.workspace.sgraph.ai | `main` | Auto on merge to main | Staging, pre-production validation |
| **Prod** | workspace.sgraph.ai | -- | Manual dispatch | Real users |

### 4.2. Promotion Flow

```
dev branch push
  -> Tests pass
  -> Deploy UI to S3: websites/sgraph-workspace/dev/
  -> Invalidate dev CF
  -> Available at dev.workspace.sgraph.ai

PR merged to main
  -> Deploy UI to S3: websites/sgraph-workspace/main/
  -> Invalidate main CF
  -> Available at main.workspace.sgraph.ai
  -> QA validates

Manual dispatch (production)
  -> Deploy UI to S3: websites/sgraph-workspace/prod/
  -> Invalidate prod CF
  -> Available at workspace.sgraph.ai
```

### 4.3. S3 Layout

All environments use the same bucket with path isolation:

```
s3://{bucket}/websites/sgraph-workspace/
  dev/
    releases/v0/v0.1/v0.1.0/   (base components)
    releases/v0/v0.1/v0.1.1/   (overlay)
    latest/                     (copy of current release)
  main/
    releases/v0/v0.1/v0.1.0/
    releases/v0/v0.1/v0.1.1/
    latest/
  prod/
    releases/v0/v0.1/v0.1.0/
    releases/v0/v0.1/v0.1.1/
    latest/
```

### 4.4. CloudFront Architecture

Two patterns are possible:

**Option A: Shared CF distribution with origin path isolation (recommended)**
- One CF distribution serves dev + main (via CloudFront behaviors or origin path mapping)
- Separate CF distribution for prod (different caching, WAF, etc.)
- This is what the current secret naming implies (`DEV` distribution shared by dev and main)

**Option B: One CF distribution per environment**
- Three separate distributions, each with its own S3 origin path
- Cleaner separation, independent caching and invalidation
- Higher cost but better isolation

The existing pattern across Send and Vault uses Option A (shared dev/main distribution).

## 5. Steps to Implement

### Phase 1: Main Environment (Immediate -- move users off Dev)

1. **Verify infrastructure exists:**
   - Check if `main.workspace.sgraph.ai` DNS record exists
   - Check if CloudFront distribution handles `main.workspace.sgraph.ai`
   - Check if `SGRAPH_WORKSPACE__DEV__CLOUDFRONT_DISTRIBUTION_ID` secret is set for the `main` GitHub environment

2. **If infrastructure is missing, create:**
   - Add `main.workspace.sgraph.ai` as alternate domain on the dev/main CF distribution
   - Add DNS CNAME pointing to CF distribution
   - Ensure ACM cert covers `main.workspace.sgraph.ai`

3. **Configure GitHub `main` environment secrets:**
   - `SGRAPH_WORKSPACE__UI_S3_BUCKET` (same bucket as dev)
   - `SGRAPH_WORKSPACE__DEV__CLOUDFRONT_DISTRIBUTION_ID` (shared dev/main distribution)
   - `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY`

4. **Trigger deployment:**
   - Either merge dev to main (auto-triggers) or use workflow_dispatch
   - Verify `main.workspace.sgraph.ai` serves the workspace UI

5. **Redirect users from dev.workspace.sgraph.ai to main.workspace.sgraph.ai**

### Phase 2: Production Environment

1. **Create production CloudFront distribution:**
   - Origin: S3 bucket path `websites/sgraph-workspace/prod/`
   - Alternate domain: `workspace.sgraph.ai`
   - Cache policy: production settings (longer TTLs, stricter cache-control)

2. **DNS setup:**
   - Add CNAME/ALIAS for `workspace.sgraph.ai` -> CF distribution

3. **Configure GitHub `production` environment secrets:**
   - `SGRAPH_WORKSPACE__UI_S3_BUCKET`
   - `SGRAPH_WORKSPACE__CLOUDFRONT_DISTRIBUTION_ID` (prod-specific distribution)
   - `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY`

4. **Add environment protection rules:**
   - Require approval for production deployments
   - Optional: restrict to specific branches

5. **Deploy and validate:**
   - Manual dispatch with `production` selected
   - Smoke test `workspace.sgraph.ai`

6. **Redirect users from dev/main to workspace.sgraph.ai**

### Phase 3: Hardening

1. Add post-deployment smoke tests to the workflow
2. Add deployment notifications (Slack/email)
3. Consider WAF rules for production CF distribution
4. Add monitoring/alerting for workspace.sgraph.ai uptime

## 9. Recommendations

1. **Immediate priority:** Verify which GitHub environment secrets are already configured for `main` and `production`. This determines whether this is a 30-minute task or a half-day task.

2. **User migration plan:** Deploy to `workspace.sgraph.ai` (prod) rather than `main.workspace.sgraph.ai`. Users should be on the production URL. Main is for staging.

3. **No workflow changes needed.** The `deploy-ui-workspace.yml` file is correctly structured. Do not modify it.

4. **Follow the established pattern.** Send UI and Vault UI already have this three-environment model working. Replicate their infrastructure setup for workspace.

5. **Backend consideration:** The workspace UI talks to backend APIs. Verify that the backend Lambda for workspace has corresponding main/prod stages, or that the workspace UI is configured to point to the correct API endpoint per environment.
