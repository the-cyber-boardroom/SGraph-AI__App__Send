# DevOps Plan: SG/Send Container Deployment

**Version:** v0.19.8
**Date:** 2026-04-18
**Role:** DevOps
**Status:** Implementation plan — approved decisions, ready to build

---

## Goal

Package SGraph Send as a single Docker container that replicates `send.sgraph.ai`, push to Docker Hub + ECR, and create a Docker-on-EC2 AMI via Packer.

---

## Approved Decisions

| Decision | Choice |
|---|---|
| Container split | **Single combined** (User + Admin in one image) |
| Storage default | **Memory-FS** (configurable to disk volume or S3 via env vars) |
| AMI strategy | **Docker-on-EC2** via Packer |
| TLS handling | **HTTP only** — external proxy handles HTTPS |

---

## Phase 1: Combined FastAPI App

**PROPOSED — does not exist yet.**

Currently, User and Admin are two separate FastAPI apps (`Fast_API__SGraph__App__Send__User` and `Fast_API__SGraph__App__Send__Admin`). For the container, we need a single uvicorn process serving both.

### Approach: Combined App

Create a new entry point that mounts both apps into a single FastAPI instance.

**New file:** `sgraph_ai_app_send/container/app.py`

```python
from fastapi import FastAPI
from sgraph_ai_app_send.lambda__user.fast_api.Fast_API__SGraph__App__Send__User   import Fast_API__SGraph__App__Send__User
from sgraph_ai_app_send.lambda__admin.fast_api.Fast_API__SGraph__App__Send__Admin import Fast_API__SGraph__App__Send__Admin

def create_app() -> FastAPI:
    # Create combined FastAPI app
    app = FastAPI(title="SGraph Send", description="Zero-Knowledge Encrypted File Sharing")

    # Initialize User app (transfers, vault, presigned, UI)
    user_app = Fast_API__SGraph__App__Send__User()
    user_app.setup()

    # Initialize Admin app (tokens, keys, vault mgmt, admin UI)
    admin_app = Fast_API__SGraph__App__Send__Admin()
    admin_app.setup()

    # Mount sub-applications
    app.mount("/", user_app.app())    # User at root (handles /api/*, /send/*)
    # Admin routes added to same app OR mounted at /admin-api/

    return app
```

**Alternative (simpler):** Instead of mounting sub-apps, create a thin wrapper that registers both sets of routes into one FastAPI instance. This avoids the complexity of ASGI sub-application routing. Explore this during implementation — the key constraint is that the combined app must serve:
- All User API routes (`/api/transfers/*`, `/api/vault/*`, `/api/presigned/*`, `/api/vault/presigned/*`)
- All Admin API routes (`/api/tokens/*`, `/api/keys/*`, `/api/metrics/*`)
- User UI static files (`/send/*`)
- Admin UI static files (`/admin/*`)
- Health endpoint (`/info/health`)

### Storage Mode: DISK

**New file:** `sgraph_ai_app_send/lambda__user/storage/Storage_FS__Disk.py`

Add a `DISK` variant to `Enum__Storage__Mode` and a `Storage_FS__Disk` backend that uses `Storage_FS__Memory` but persists to a local directory (or use `memory_fs.storage_fs.providers.Storage_FS__File` if it exists in the `memory-fs` package).

**Env var:** `SEND__STORAGE_MODE=disk` + `SEND__DISK_PATH=/data`

Update `Send__Config.py`:
```python
class Enum__Storage__Mode(str, Enum):
    MEMORY = "memory"
    S3     = "s3"
    DISK   = "disk"     # NEW: local filesystem (Docker volume)
```

### Admin-User Integration

In Lambda, the User Lambda calls the Admin Lambda's HTTP endpoint for token validation (`Admin__Service__Client`). In the container, both run in the same process. Two options:

**Option A (Recommended): Direct service injection**
When both apps share the same process, skip the HTTP call — inject `Service__Tokens` directly into the User app's token validation. This is faster and simpler.

**Option B: Loopback HTTP**
Keep the existing `Admin__Service__Client` pointing at `http://localhost:8080`. Works but adds latency.

## Phase 2: Dockerfile

**New file:** `Dockerfile`

```dockerfile
# ============================================================
# SGraph Send — Combined Container
# ============================================================
FROM python:3.12-slim AS base

WORKDIR /app

# Install system dependencies
RUN apt-get update && apt-get install -y --no-install-recommends \
    && rm -rf /var/lib/apt/lists/*

# Install Python dependencies
COPY pyproject.toml ./
COPY sgraph_ai_app_send/ ./sgraph_ai_app_send/
COPY sgraph_ai_app_send__ui__admin/ ./sgraph_ai_app_send__ui__admin/
COPY sgraph_ai_app_send__ui__user/ ./sgraph_ai_app_send__ui__user/
COPY sgraph_ai_app_send__ui__vault/ ./sgraph_ai_app_send__ui__vault/
COPY sgraph_ai_app_send__ui__workspace/ ./sgraph_ai_app_send__ui__workspace/

RUN pip install --no-cache-dir -e .

# Create data directory for DISK storage mode
RUN mkdir -p /data

# Runtime
EXPOSE 8080
ENV SEND__STORAGE_MODE=memory
ENV SEND__DISK_PATH=/data
ENV PORT=8080

CMD ["uvicorn", "sgraph_ai_app_send.container.app:create_app", \
     "--factory", "--host", "0.0.0.0", "--port", "8080"]
```

## Phase 5: AMI via Packer (Docker-on-EC2)

**New file:** `packer/sg-send.pkr.hcl`

### AMI Spec
- **Base AMI:** Amazon Linux 2023 (arm64)
- **Instance type for build:** `t4g.micro`
- **Software installed:** Docker CE, docker-compose
- **Container config:** Systemd unit pulls and runs `sgraphai/sg-send:latest`
- **Ports:** 8080 exposed
- **Data volume:** `/data` directory mapped to EBS volume

### Systemd Unit

```ini
[Unit]
Description=SGraph Send
After=docker.service
Requires=docker.service

[Service]
Restart=always
ExecStartPre=-/usr/bin/docker pull sgraphai/sg-send:latest
ExecStart=/usr/bin/docker run --rm --name sg-send \
  -p 8080:8080 \
  -v /data:/data \
  -e SEND__STORAGE_MODE=disk \
  -e SEND__DISK_PATH=/data \
  --env-file /etc/sg-send/env \
  sgraphai/sg-send:latest
ExecStop=/usr/bin/docker stop sg-send

[Install]
WantedBy=multi-user.target
```

### Packer Build Flow
1. Launch base AMI
2. Install Docker
3. Pull `sgraphai/sg-send:latest`
4. Install systemd unit
5. Create `/etc/sg-send/env` template
6. Create AMI snapshot
7. Register AMI with tags

## Phase 6: Smoke Tests

**New file:** `tests/deploy/container/test_Container__Smoke.py`

Tests to run after building the container:

1. **Health check:** `GET /info/health` → 200
2. **Transfer create:** `POST /api/transfers/create` → 200, returns `transfer_id`
3. **Vault write/read:** `PUT /api/vault/write/{id}/{file}` → 200, then `GET` returns same bytes
4. **Admin tokens:** `POST /api/tokens/create` → 200 (verifies Admin app is running)
5. **User UI:** `GET /send/` → 200, HTML response
6. **Admin UI:** `GET /admin/` → 200 or redirect
7. **Storage mode:** Container started with `SEND__STORAGE_MODE=disk` persists data across restart

## Usage (End State)

### Quick start (ephemeral, in-memory)
```bash
docker run -p 8080:8080 sgraphai/sg-send
```

### Persistent storage (Docker volume)
```bash
docker run -p 8080:8080 \
  -e SEND__STORAGE_MODE=disk \
  -v sg-send-data:/data \
  sgraphai/sg-send
```

### With S3 backend
```bash
docker run -p 8080:8080 \
  -e SEND__STORAGE_MODE=s3 \
  -e SEND__S3_BUCKET=my-bucket \
  -e AWS_ACCESS_KEY_ID=... \
  -e AWS_SECRET_ACCESS_KEY=... \
  -e AWS_REGION=eu-west-1 \
  sgraphai/sg-send
```

### With access token
```bash
docker run -p 8080:8080 \
  -e SGRAPH_SEND__ACCESS_TOKEN=my-secret-token \
  sgraphai/sg-send
```

## Open Questions

1. **Docker Hub org name:** Is `sgraphai` the correct Docker Hub organization, or should it be `thecyberboardroom` or something else?
2. **ECR account + region:** Which AWS account and region should the ECR repository live in?
3. **AMI regions:** Which AWS regions should the AMI be copied to?
4. **Admin API key for container mode:** In Lambda, the Admin Lambda has its own API key. In the container, should Admin routes be:
   - (a) Open (no API key, suitable for internal/self-hosted use)
   - (b) Protected by a container-level env var (`SGRAPH_SEND__ADMIN_API_KEY`)
   - (c) Same as User routes (use the access token)
5. **memory-fs DISK backend:** Does the `memory-fs` package already have a `Storage_FS__File` or `Storage_FS__Disk` provider? If yes, use it directly. If not, need to implement one.
