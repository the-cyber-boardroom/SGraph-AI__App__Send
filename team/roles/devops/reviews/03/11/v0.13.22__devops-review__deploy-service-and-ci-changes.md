# v0.13.22 DevOps Review: Deploy Service Architecture & CI/CD Workflow Changes

**Date:** 2026-03-11
**Role:** DevOps
**Version:** v0.13.22
**Priority:** HIGH -- deploy service is a foundational infrastructure proposal; CI/CD changes are already shipped

---

## 1. Executive Summary

This review covers two distinct areas:

1. **CI/CD workflow changes (shipped):** All four static site deployment workflows (`deploy-ui-user.yml`, `deploy-ui-vault.yml`, `deploy-ui-workspace.yml`, `deploy-website.yml`) now use a shared `deploy_static_site.py` Python script, replacing inline `aws s3 sync` commands. This is a significant improvement in consistency, testability, and maintainability. Risk: LOW.

2. **Deploy Service architecture (proposed):** An ephemeral EC2 control plane for provisioning cloud appliances, driven by a Router Lambda and vault-stored state. This is a well-designed architecture brief with clear separation of concerns. PROPOSED -- does not exist yet. Risk assessment requires careful attention to IAM, cost controls, and cleanup guarantees.

## 2. CI/CD Workflow Changes -- What Changed

### 2.1. The Shared Deploy Script: `scripts/deploy_static_site.py`

All four UI/website deployment workflows now delegate to a single Python script (`deploy_static_site.py`) instead of inline shell commands. This follows the v0.7.6 architecture brief: "CI pipeline logic lives in Python, GitHub Actions is just the trigger and executor."

**What the script does:**

| Step | Description |
|------|-------------|
| 1. Validation | Check required HTML files, validate `keys.json`, scan for broken internal links (skippable via `--skip-validation`) |
| 2. S3 sync (versioned) | Sync to `websites/{site}/{env}/releases/{ifd-path}/` with per-file-type cache-control headers |
| 3. S3 copy (latest) | Copy the release to `websites/{site}/{env}/latest/` with `--delete` |
| 4. Version file upload | Upload `sgraph_ai_app_send/version` as plain text to `latest/version` |
| 5. CloudFront invalidation | Invalidate `/*` on one or more CloudFront distributions |
| 6. Smoke test | Optional HTTP 200 check against a provided URL |

**Quality assessment:**

| Aspect | Rating | Notes |
|--------|--------|-------|
| Structure | GOOD | Clean separation: constants, helpers, validation, S3, CloudFront, main |
| IFD versioning | GOOD | `version_to_ifd_path()` correctly converts `v0.7.7` to `v0/v0.7/v0.7.7` |
| Cache-control strategy | GOOD | HTML: 5 min, CSS/JS/JSON: 1 day, images: 1 week -- sensible tiering |
| Content-type handling | GOOD | Explicit content-type map for 17 file types including fonts |
| Error handling | ADEQUATE | `sys.exit(returncode)` on failure; validation warnings don't block deploy |
| Dry-run support | GOOD | `--dry-run` flag prints plan without executing |
| Multi-distribution support | GOOD | `--cloudfront-distribution-id` accepts multiple IDs (nargs="+") |
| Environment isolation | GOOD | `--deploy-env` creates isolated S3 paths per environment |

**Concerns:**

| Concern | Severity | Detail |
|---------|----------|--------|
| No rollback mechanism | LOW | If a deploy goes wrong, there is no automated rollback. Previous releases remain in `releases/{version}/` so manual rollback is possible by re-syncing an older release to `latest/`. Acceptable for now. |
| `subprocess.run` for AWS CLI | INFORMATIONAL | The script shells out to `aws s3 sync` and `aws cloudfront create-invalidation`. This is intentional per the architecture brief (performance of `aws s3 sync` vs Python SDK). Acceptable. |
| HTML sync uses `--delete` | LOW | The `--delete` flag on HTML sync removes files from S3 that are not in source. Correct for `latest/` (ensures clean state) and for release paths (ensures exact snapshot). But if a deploy is interrupted mid-sync, `latest/` could be in a partial state. CloudFront cache would serve stale content during the window. |
| Validation is website-specific | INFORMATIONAL | `validate_website_files()` checks for `index.html`, `product/index.html`, `agents/index.html` etc. These are sgraph.ai website pages. Other sites (send, vault, workspace) skip validation via `--skip-validation`. This is fine -- validation is opt-in. |

### 2.2. Workflow-by-Workflow Analysis

#### `deploy-ui-user.yml`

| Aspect | Value |
|--------|-------|
| Triggers | Push to `dev`/`main` (path-filtered), manual dispatch |
| Environments | dev, main, prod (3-way resolution) |
| Pre-deploy steps | Inject build version, generate i18n pages |
| Deploy | `deploy_static_site.py --site sgraph-send --skip-validation` |
| Credential guard | Yes -- checks for AWS creds before deploy steps |
| Concurrency | `cancel-in-progress: false` (safe -- won't cancel running deploys) |

**Assessment:** Well-structured. The credential check step is a good safety pattern -- allows the workflow to run in forks without failing. SAFE.

#### `deploy-ui-vault.yml`

| Aspect | Value |
|--------|-------|
| Triggers | Push to `dev`/`main` (path-filtered), manual dispatch |
| Environments | dev, main, prod |
| Pre-deploy steps | Generate i18n pages (no build version injection) |
| Deploy | `deploy_static_site.py --site sgraph-vault --skip-validation` |
| Credential guard | Yes |

**Assessment:** Mirrors the user UI workflow. No version injection -- vault UI does not embed build-info.js. SAFE.

#### `deploy-ui-workspace.yml`

| Aspect | Value |
|--------|-------|
| Triggers | Push to `dev`/`main` (path-filtered), manual dispatch |
| Environments | dev, main, prod |
| Pre-deploy steps | Build version injection (into v0.1.0 base), CDN path rewriting (v0.1.0 relative to absolute) |
| Deploy | Two-phase: v0.1.0 base to releases, then v0.1.1 overlay to releases + latest |
| Credential guard | Yes |

**Assessment:** Most complex workflow due to IFD overlay model. The CDN path rewriting step (`sed` replacing `../v0.1.0/` with absolute CDN URLs per environment) is the most fragile part -- it depends on exact string patterns in `index.html`. A malformed HTML file could break the rewrite silently. The `grep -n` verification step after rewrite is a good sanity check. SAFE but warrants monitoring.

#### `deploy-website.yml`

| Aspect | Value |
|--------|-------|
| Triggers | Push to `dev`/`main` (path-filtered), manual dispatch |
| Environments | dev, main, prod (defaults to `production` on manual dispatch) |
| Pre-deploy steps | Read version, generate i18n pages |
| Deploy | `deploy_static_site.py --site sgraph-ai` with smoke test URL |
| Credential guard | NO -- unlike the UI workflows, this one does not check for credentials before `configure-aws-credentials` |
| Post-deploy | `store_ci_artifacts.py` |

**Assessment:** The missing credential guard is inconsistent with the other workflows. If this workflow runs in a fork or environment without AWS secrets, it will fail at the `configure-aws-credentials` step rather than skipping gracefully. LOW risk (the workflow only triggers on dev/main branches, which are protected), but should be harmonised with the other workflows for consistency.

| Risk | Impact | Likelihood | Rating |

## 3. Deploy Service Architecture -- Assessment

**STATUS: PROPOSED -- does not exist yet.**

The deploy service brief (`v0.13.12__dev-brief__deploy-service-final.md`) describes an ephemeral EC2 control plane for provisioning and managing cloud appliances. This is a comprehensive architecture document. The following is a DevOps feasibility and risk assessment.

### 3.1. Architecture Summary

```
Router Lambda (permanent, cheap)
    |
    v
Deploy EC2 (ephemeral, boots on demand)
    |
    +-- SSH into target machines
    +-- Create/destroy appliances (LLM, Send Ephemeral, etc.)
    +-- Bake AMIs
    +-- All state in SG/Send vault (zero-knowledge)
    |
    v
EventBridge watchdog (stops stale instances)
```

### 3.2. Feasibility Assessment

| Component | Feasibility | Notes |
|-----------|-------------|-------|
| Router Lambda (start/stop/redirect) | HIGH | Straightforward Lambda function. Minimal IAM. Pattern is well-established. |
| Vault-stored state | HIGH | Already working for SG/Send. Natural extension. |
| Boot-on-demand EC2 | HIGH | `StartInstances` + health polling is a standard pattern. 60-90s cold start is acceptable for a control plane. |
| Auto-shutdown (heartbeat) | HIGH | FastAPI background task + `shutdown -h now` is simple and reliable. |
| EventBridge watchdog | HIGH | Belt-and-suspenders. Good engineering practice. |
| Waiting page with JS poller | HIGH | HTML served from Lambda, JS polls `/status`. Standard pattern. |
| SSH-based provisioning | MEDIUM | Depends on `osbot-utils` SSH API maturity. Sequential SSH commands are inherently fragile (network timeouts, partial execution). |
| AMI baking in CI | MEDIUM | Adds significant complexity to CI pipeline. AMI creation takes 5-15 minutes. Spot instance interruption during baking is a risk. |
| Type__Twins mock layer | MEDIUM | Ambitious. The fractal twin chain described in the brief is a product in its own right. For Phase 1, simple mocks suffice. |
| Ephemeral Route53 DNS | HIGH | `UPSERT`/`DELETE` A records with short TTL is straightforward. 60s TTL propagation is well-understood. |

### 3.3. Risk Assessment

| Risk | Impact | Likelihood | Mitigation (from brief) | DevOps Assessment |
|------|--------|-----------|------------------------|-------------------|
| **Zombie instances** -- EC2 not stopped, ongoing cost | HIGH | MEDIUM | Dual shutdown: heartbeat + EventBridge watchdog. Max zombie: 25 min. | ADEQUATE. The 25-min window is acceptable. Recommend adding CloudWatch alarm for running time > 30 min as a third safety net. |
| **Orphaned resources** -- security groups, IAM roles, DNS left behind | HIGH | MEDIUM | `try/finally` cleanup in all scripts. `sg-deploy down` removes all resources. | FRAGILE. `try/finally` in Python does not survive process kills, OOM, or CI runner timeout. Need a separate cleanup job that scans for tagged resources. |
| **IAM credential exposure** -- Deploy EC2 holds high-privilege creds | CRITICAL | LOW | Creds only on EC2, never in Lambda. EC2 in private network. | ADEQUATE for Phase 1. Recommend: use EC2 instance profile (IAM role) instead of static credentials. Rotate if using static keys. |
| **SSH key management** -- ephemeral keys must be cleaned up | HIGH | MEDIUM | `key_pair_create_to_file` + cleanup in `finally` block. | Same fragility as orphaned resources. CI runner crash = leaked key pair. Recommend: key pair TTL or nightly cleanup job. |
| **Cost runaway** -- multiple appliances left running | HIGH | LOW | Each appliance has its own watchdog. Vault tracks all instances. | ADEQUATE. Recommend: AWS Budget alarm as hard stop. Tag all resources with `Project: sgraph-deploy`. |
| **Cold start latency** -- 60-90s for control plane | LOW | HIGH | Waiting page with progress bar. Expected and communicated. | ACCEPTABLE for a control plane. Not acceptable for user-facing services (those should use Lambda or pre-warmed instances). |
| **AMI baking failure** -- partial AMI, broken deploy | MEDIUM | LOW | Health check before baking. Test AMI after baking. | ADEQUATE. Recommend: keep N-1 AMI as fallback. Never delete the previous AMI until N is verified. |

### 3.4. Integration with Existing Infrastructure

The deploy service proposal fits cleanly alongside the existing Lambda infrastructure:

| Existing | Deploy Service | Interaction |
|----------|---------------|-------------|
| User Lambda (transfers, health, UI) | Deploy EC2 | None -- independent. Deploy does not manage Lambda. |
| Admin Lambda (tokens, stats) | Deploy EC2 | None -- independent. |
| CloudFront + S3 (static sites) | Deploy EC2 | None -- static site deployment stays in GitHub Actions CI. |
| SG/Send vault | Vault state files | Deploy service reads/writes vault for instance state. |
| GitHub Actions CI | AMI baking pipeline | New CI workflow for AMI baking. Additive, does not change existing pipelines. |

**Key observation:** The deploy service is additive. It does not replace or modify any existing infrastructure. Existing Lambda functions, CI pipelines, and S3 deployments are unaffected. This is the lowest-risk integration pattern.

### 3.5. Cost Analysis

| Component | Always-on Cost | Usage Cost |
|-----------|---------------|------------|
| Router Lambda | ~$0.10/month (pennies per invocation) | Negligible |
| Route53 hosted zone | $0.50/month | + $0.40/million queries |
| Deploy EC2 (t3.medium, ephemeral) | $0 when stopped | ~$0.04/hr when running |
| EventBridge rule | Free (within free tier) | Negligible |
| Vault storage | Already paid for | $0 marginal |
| **Total always-on** | **< $1/month** | -- |

The near-zero always-on cost claim in the brief is credible. The key variable is how often the Deploy EC2 is started and how long it runs. At 15-minute idle timeout, a single deploy operation costs approximately $0.01.

## 4. Recommendations

### 4.1. Immediate (CI/CD -- shipped code)

| # | Recommendation | Priority | Effort |
|---|---------------|----------|--------|
| 1 | Add credential guard to `deploy-website.yml` to match other workflows | LOW | 5 min |
| 2 | Add basic tests for `deploy_static_site.py` (argument parsing, IFD path conversion, cache-control mapping) | MEDIUM | 2 hrs |
| 3 | Document the rollback procedure: "re-run workflow with previous version" or "manual s3 sync from releases/" | LOW | 30 min |

### 4.2. Deploy Service -- Before Building

| # | Recommendation | Priority | Effort |
|---|---------------|----------|--------|
| 4 | **Use EC2 instance profiles, not static IAM credentials** on the Deploy EC2. Instance profiles are rotated automatically by AWS and cannot be leaked via process crash. | HIGH | Architecture decision |
| 5 | **Tag all resources** created by the deploy service with `Project: sgraph-deploy`, `ManagedBy: deploy-service`, `CreatedAt: {timestamp}`. This enables cleanup sweeps. | HIGH | Code pattern |
| 6 | **Build a resource cleanup Lambda** that runs daily, scans for tagged resources older than 24 hours, and terminates/deletes them. This is the safety net for orphaned resources beyond `try/finally`. | HIGH | 1 day |
| 7 | **Set up AWS Budget alarm** at $50/month for the deploy service account. Hard stop for cost runaway. | HIGH | 15 min |
| 8 | **Keep N-1 AMI** -- never delete the previous Deploy AMI until the new one is verified in production. | MEDIUM | Policy |
| 9 | **Start with Phase 1 only** -- boot-on-demand with IP redirect. Do not attempt ephemeral DNS, AMI baking pipeline, or Type__Twins in the first iteration. | HIGH | Scope decision |
| 10 | **Add CloudWatch alarm** for Deploy EC2 running time > 30 minutes. Third safety net beyond heartbeat + watchdog. | MEDIUM | 15 min |

### 4.3. Watch Points

| Area | What to Watch |
|------|---------------|
| SSH reliability | Network timeouts during long provisioning sequences. Need idempotent commands and retry logic. |
| Spot instance interruption | If using spot for AMI baking (as proposed), an interruption mid-bake wastes the entire run. Consider on-demand for baking. |
| Vault write latency | If vault writes from Lambda have latency spikes, the state machine could become inconsistent (Lambda thinks "starting" but EC2 thinks "stopped"). |
| Security group cleanup | The `try/finally` pattern for SG cleanup is the weakest link. A CI timeout or OOM will leak security groups with open SSH ports. |
