## Brief #17: Data Room as Code — CI/CD (HIGH, Supporting)

### Infrastructure Components

| Component | What DevOps Builds | Complexity |
|---|---|---|
| **Template repo** | Standard .sgroom folder structure, CI workflows, CLAUDE.md | LOW |
| **Deploy pipeline** | Config change triggers data room redeployment | MEDIUM |
| **QA/docs pipeline** | Headless browser screenshots, markdown gen, GitHub Pages | MEDIUM |
| **Secret management** | GitHub token for admin UI, never exposed to runtime | LOW |

### CI/CD Pipeline

Trigger on push to main: (1) Validate config schemas. (2) Deploy configuration to runtime. (3) QA smoke test with screenshot capture. (4) Generate and publish documentation. (5) Notify.

### Operational Risks

| Risk | Mitigation |
|---|---|
| Large encrypted blobs in Git | Git LFS for files >10MB. Monitor repo size. |
| GitHub rate limits | Batch commits. Use GitHub App token (higher limits). |
| Secret exposure | Token only in admin UI and CI secrets. Runtime has zero GitHub access. |
| Template drift | Versioned template repo. Pull updates via merge. |

### Bootstrap Automation

Script: `gh repo create --template` then configure branch protection, add secrets, run initial pipeline. Output: live data room URL.

## Brief #22: Agent Isolation — Container Orchestration (CRITICAL, Supporting)

### Orchestration Requirements

| Requirement | Implementation |
|---|---|
| One container per agent | Distroless base images, minimal attack surface |
| Ephemeral lifecycle | Created per task, destroyed after completion |
| Network isolation | Own network namespace. Only orchestrator can reach containers. |
| Secret injection | AWS Secrets Manager / K8s secrets. Never in images. |
| No shared filesystem | Ephemeral volumes. .sgworkflow is the only data channel. |
| Resource limits | CPU and memory caps per container |

### Technology Recommendation

| Option | Fit | Notes |
|---|---|---|
| **ECS Fargate** | Production | Serverless containers, per-task isolation, Secrets Manager integration |
| **Kubernetes (EKS)** | Alternative | Fine-grained network policies but higher operational complexity |
| **Lambda** | High-security | Best cold-start isolation but warm-start state leakage risk |
| **Docker Compose** | Local dev | Per-service isolation for testing orchestration patterns |

**Recommendation:** ECS Fargate for production. Docker Compose for dev. Lambda for high-security workflows.

### Container Image Pipeline

Base image (weekly): Python 3.12/arm64, pinned osbot-* deps, distroless, read-only rootfs, non-root user, no SSH/debug tools. Per-agent image (per release): FROM base, agent code only, scanned by Trivy/Snyk, no secrets.

## Brief #14: Portable Data Rooms — ZIP Backend (CRITICAL, Supporting)

### DevOps Responsibilities

| Concern | What DevOps Handles |
|---|---|
| Build pipeline | ZIP backend in CI alongside memory and S3 backends |
| Storage limits | Size limits per deployment target for .sgroom files |
| Browser viewer hosting | Static site or Lambda URL for .sgroom viewer |
| Offline sync | Merge pipeline with conflict resolution for .sgroom re-imports |

### Deployment Target Validation

| Target | Notes |
|---|---|
| **Lambda** | Constrained by /tmp (10GB) and 6MB response. Large rooms need streaming. |
| **Container** | Best for large rooms. No size constraints beyond disk. |
| **EC2/CLI** | Local filesystem. Good for air-gapped environments. |
| **Browser** | Primary use case. JSZip or fflate library. |

### CI Test Matrix Addition

Add ZIP backend integration tests: create .sgroom, write/read files, export from hosted room, import with merge verification, 3-participant supply chain test.
