## Brief #17: Data Room as Code — CI/CD Implications

**Priority:** HIGH | **DevOps role:** Supporting

### Architecture Assessment

The brief proposes GitHub repos as the persistent layer for data rooms, with CI/CD pipelines handling deployment on config changes.

**Key infrastructure components:**

| Component | What DevOps Builds | Complexity |
|---|---|---|
| **Template repo** | GitHub template with standard .sgroom folder structure, CI workflows, CLAUDE.md | LOW — one-time setup |
| **Deploy pipeline** | GitHub Actions workflow: config change triggers data room redeployment | MEDIUM — must handle encrypted blobs safely |
| **QA/docs pipeline** | Headless browser screenshots, markdown generation, GitHub Pages publish | MEDIUM — Playwright in CI, asset management |
| **Secret management** | GitHub token for admin UI, never exposed to data room runtime | LOW — standard GitHub secrets |

### CI/CD Pipeline Design

```
Trigger: push to main branch
  |
  ├── Step 1: Validate config (room.yaml, users.yaml, branding.yaml)
  │   └── Schema validation, public key format check
  |
  ├── Step 2: Deploy data room configuration
  │   └── Copy config to runtime environment (Lambda, container, or static)
  |
  ├── Step 3: QA smoke test
  │   ├── Open data room in headless browser
  │   ├── Verify branding renders
  │   ├── Verify file listing loads
  │   └── Screenshot capture
  |
  ├── Step 4: Generate documentation
  │   ├── Convert screenshots to markdown
  │   └── Publish to GitHub Pages
  |
  └── Step 5: Notify (Slack/webhook)
```

### Operational Risks

| Risk | Mitigation |
|---|---|
| **Large encrypted blobs in Git** | Git LFS for files over 10MB. Monitor repo size. Set per-repo size limits. |
| **GitHub rate limits** | Admin UI batches commits. CI uses GitHub App token (higher limits than PAT). |
| **Secret exposure** | GitHub token exists ONLY in admin UI and CI secrets. Data room runtime has zero GitHub access. Blast radius is limited by design. |
| **Template drift** | Template repo versioned. Existing data rooms can pull template updates via GitHub template sync or manual merge. |

### New Repo Bootstrap

The brief describes a pattern: every new repo gets an agentic team. DevOps should automate the bootstrap:

```
create-data-room.sh:
  1. Create repo from template (gh repo create --template)
  2. Configure branch protection (main: require PR)
  3. Add GitHub Actions secrets (deploy token)
  4. Run initial CI pipeline (validate + deploy empty room)
  5. Output: live data room URL + admin instructions
```

## Brief #22: Agent Isolation — Container Orchestration

**Priority:** CRITICAL | **DevOps role:** Supporting

### Container Architecture

The brief requires one container per agent with strict isolation. For the 19-agent project team, this means orchestrating up to 19 isolated containers per workflow.

**Orchestration requirements:**

| Requirement | Implementation |
|---|---|
| **One container per agent** | Pre-built agent images with minimal attack surface (distroless base) |
| **Ephemeral lifecycle** | Container created per task, destroyed after completion. No state persistence. |
| **Network isolation** | Each container in its own network namespace. No inter-container connectivity. Only orchestrator can reach containers. |
| **Secret injection** | Private keys injected via runtime secret management (AWS Secrets Manager, Kubernetes secrets). Never baked into images. |
| **No shared filesystem** | Each container gets its own ephemeral volume. .sgworkflow file is the only data transfer mechanism. |
| **Resource limits** | CPU and memory caps per container to prevent resource exhaustion attacks. |

### Technology Options

| Option | Fit | Trade-offs |
|---|---|---|
| **AWS ECS Fargate** | Good — serverless containers, per-task isolation, AWS Secrets Manager integration | Cost per container-second. Cold start ~10-30s. |
| **Kubernetes (EKS)** | Good — fine-grained network policies, pod security standards, secrets management | Operational complexity. Cluster management overhead. |
| **AWS Lambda** | Best isolation (cold start) but brief warns about warm start state leakage | Must use one function per agent. Must zero memory before returning. Highest latency. |
| **Docker Compose (local dev)** | Good for development — per-service isolation with network configuration | Not production-grade. Useful for testing the orchestration pattern. |

**DevOps recommendation:** ECS Fargate for production (balance of isolation and operational simplicity). Docker Compose for local development. Lambda reserved for high-security workflows where cold-start latency is acceptable.

### Orchestrator Infrastructure

The orchestrator itself needs:

| Component | Requirement |
|---|---|
| **Compute** | Lightweight — reads .sgworkflow, dispatches to containers, verifies signatures |
| **Has** | All agents' public keys. Orchestrator's own private key. Container runtime credentials. |
| **Does NOT have** | Any agent's private key. |
| **Isolation** | Runs in its own container/Lambda. Separate from all agent containers. |
| **Scaling** | One orchestrator per workflow execution. Stateless — all state in .sgworkflow. |

### Container Image Pipeline

```
Agent base image (built weekly):
  - Python 3.12 / arm64
  - osbot-utils, osbot-aws, osbot-fast-api (pinned versions)
  - Minimal OS (distroless or alpine)
  - No SSH, no debug tools, no package manager
  - Read-only root filesystem
  - Non-root user

Per-agent image (built per release):
  - FROM agent-base
  - COPY agent-specific code
  - No secrets in image
  - Scanned by container security scanner (Trivy/Snyk)
```

## Brief #14: Portable Data Rooms — ZIP Backend for MemoryFS

**Priority:** CRITICAL | **DevOps role:** Supporting

### Infrastructure Implications

The ZIP backend itself is a code change (Developer territory), but DevOps needs to support:

| Concern | What DevOps Handles |
|---|---|
| **Build pipeline** | ZIP backend must be included in CI test suite. Tests run against ZIP backend alongside memory and S3 backends. |
| **Storage limits** | .sgroom files can grow large (many encrypted blobs). Set size limits per deployment target. |
| **Browser viewer hosting** | The .sgroom viewer (opens ZIP files in-browser) needs a hosting target — static site or Lambda URL. |
| **Offline sync** | When .sgroom files sync back to hosted data rooms, the merge pipeline needs conflict resolution infrastructure. |

### Deployment Target Matrix

The brief mentions .sgroom files work across all deployment targets. DevOps must validate:

| Target | ZIP Backend Works? | Notes |
|---|---|---|
| **Lambda** | Yes — read .sgroom from S3, serve via Lambda URL | Constrained by Lambda /tmp (10GB) and response size (6MB payload). Large rooms need streaming. |
| **Container (Docker/Fargate)** | Yes — mount .sgroom as volume or read from S3 | Best for large data rooms. No size constraints beyond disk. |
| **Server (EC2)** | Yes — local filesystem | Simplest. Good for air-gapped environments. |
| **CLI** | Yes — local filesystem | Desktop tool opens .sgroom directly. |
| **Browser (static)** | Yes — JavaScript ZIP library (JSZip or fflate) | The primary use case. User opens .sgroom in browser. |

### CI Integration

Add ZIP backend to the existing test matrix:

```
Current test matrix:
  - Memory backend (unit tests)
  - S3 backend (integration tests via LocalStack)

New addition:
  - ZIP backend (integration tests)
    - Create .sgroom, write files, read back, verify
    - Export hosted room to .sgroom, verify contents
    - Import .sgroom to hosted room, verify merge
    - Supply chain test: 3 participants add data sequentially
```

## Cross-Brief Infrastructure Themes

### 1. Container Strategy Unification

Briefs #22 (agent isolation) and #17 (data room deployment) both need container infrastructure. DevOps should build a unified container platform:

- Shared base images (agent base, data room runtime base)
- Shared CI pipeline for container builds and security scanning
- Shared networking policy (isolation by default, explicit allowlisting)

### 2. Secret Management

All three briefs involve secrets:
- #17: GitHub tokens for admin UI
- #22: Agent private keys injected per-container
- #14: Encryption keys for .sgroom operations

Standardise on AWS Secrets Manager with per-environment isolation. Never store secrets in code, environment variables, or container images.

### 3. The 7 Deployment Targets

The existing 7 deployment targets (Lambda, Docker, Fargate, GCP, EC2, AMI, CLI) now need to support:
- Data room config from GitHub (#17)
- Per-agent container isolation (#22)
- ZIP storage backend (#14)

Not all combinations are required immediately. Priority: Lambda + Fargate + browser.
