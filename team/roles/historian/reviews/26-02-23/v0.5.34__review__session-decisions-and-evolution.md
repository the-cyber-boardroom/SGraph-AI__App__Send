## Summary

This session represents the **vault buildout sprint** — the most code-intensive single session in the v0.5.x series. It took the vault from a stub component (v0.1.5: basic upload/download) to a full encrypted file management system (v0.1.6: tree view, editor, preview, settings, sharing). The session also produced 3 architectural pivots worth recording.

---

## Decision Log

### D070: Split vault-manager into 4 components
- **Context:** The vault detail panel was growing to 300+ lines inside vault-manager. Preview, editing, and settings were all mixed into one render method.
- **Decision:** Extract vault-preview, vault-editor, and vault-settings as standalone Web Components. vault-manager becomes the orchestrator.
- **Rationale:** Single-responsibility principle. Each component can be tested and versioned independently. Follows the pattern established by PKI (pki-keys, pki-encrypt, pki-contacts).
- **Impact:** vault-manager went from ~1,800 lines down to ~1,500 lines. The extracted components total ~750 lines. Net code increase but far better maintainability.

### D071: 3-row layout replaces 2-column layout
- **Context:** Original vault design had a 2-column layout (tree view | detail panel). The detail panel tried to show metadata, actions, preview, and editor all in one column.
- **Decision:** Switch to 3-row layout (file table | settings bar | preview/editor). Each row has a clear purpose.
- **Rationale:** The 2-column layout broke at narrow viewports and couldn't accommodate the editor properly. 3-row is more natural for a file manager (like VS Code's explorer + editor model).
- **Impact:** Required a complete CSS and HTML restructure of vault-manager. 5 commits to complete the migration.

### D072: JS height calculation for vault-editor
- **Context:** The vault-editor's textarea collapsed to minimal height inside nested CSS flex/grid containers. Three CSS-only approaches were attempted and failed.
- **Decision:** Use JavaScript with ResizeObserver to calculate available height and set it explicitly.
- **Rationale:** The nesting depth (shell grid > main area > vault-manager flex > editor flex) exceeds CSS's ability to propagate intrinsic height through multiple levels. JS calculation is the pragmatic fix.
- **Impact:** Editor now reliably fills available space and responds to window resize and debug panel toggle.
- **Lesson learned:** CSS flex/grid height propagation breaks down at 3+ nesting levels. Document this as a known limitation.

### D073: Remove max-width: 1100px from admin-shell main content
- **Context:** When the debug panel was hidden, the main content area didn't expand to fill the space. The grid column expanded correctly (`1fr` with `0 0`), but `.as-main-content` was capped at 1100px.
- **Decision:** Change `max-width: 1100px` to `max-width: 100%`. Let individual components set their own width constraints.
- **Rationale:** The shell should not impose width constraints on its children. Admin panels (data tables, editors, file managers) benefit from full-width layout.
- **Impact:** All admin panels now expand to fill available space when the debug panel is closed.

### D074: Share via Send link integration
- **Context:** The vault stores encrypted files. Users want to share individual files with external recipients.
- **Decision:** Add a "Share via Send" button that decrypts the file, creates a Send transfer, and generates a shareable URL.
- **Rationale:** Connects the vault to the core product (encrypted file sharing). The vault becomes an inbox/outbox, not just storage.
- **Impact:** The vault is no longer a standalone feature — it's integrated into the Send workflow.

## Lessons Learned

1. **CSS height propagation has limits.** At 3+ levels of nesting, CSS flex/grid cannot reliably propagate `height: 100%`. Use JS with ResizeObserver as escape hatch.
2. **Component extraction is cheaper when done early.** vault-preview and vault-settings were easy to extract because they were extracted during the first iteration, before vault-manager became too coupled.
3. **Shell constraints affect all children.** The `max-width: 1100px` on admin-shell affected every component. Shell CSS should be as permissive as possible.
4. **IFD versioning works.** Despite 18 commits of rapid iteration, all changes stayed within v0.1.6. No other versions were affected. The v0.1.3 shell fix was a 1-line change.
