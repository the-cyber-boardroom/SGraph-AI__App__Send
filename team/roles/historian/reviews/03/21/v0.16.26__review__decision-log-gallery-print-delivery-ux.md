## D-014: Content-Hashed `_gallery.{hash}/` Folder Naming

**Context:** The gallery folder needed a stable, meaningful name. Since v0.2.13, a content hash was already being computed from all file hashes in the upload (via `computeFolderHash(fileHashes)`), but the folder was hardcoded as `_preview` then `_gallery`. The hash existed in memory but was discarded.

**Decision:** Name the gallery folder `_gallery.{hash}/` where `{hash}` is derived from the content hashes of all files in the upload.

**Rationale:** Content-addressing provides three properties for free: (1) deduplication — identical uploads produce identical gallery folder names, enabling future cache hits; (2) integrity verification — the folder name is a commitment to its contents; (3) cache-friendliness — CDN and browser caches can treat the folder as immutable. Since the hash was already computed, the implementation cost was changing a string concatenation.

**Alternatives considered:**
- Static `_gallery/` folder name — rejected, loses deduplication and cache-friendliness, and wastes an already-computed value
- UUID-based folder name (`_gallery.{uuid}/`) — rejected, UUIDs have no relationship to content, so identical uploads would produce different folder names
- Timestamp-based folder name — rejected, same problem as UUID — no content relationship, and adds time dependency

**Outcome:** Gallery folder name is now `_gallery.{hash}/`. The `findPreviewFolder()` regex on the download side already handled `_gallery[^/]*` patterns, so no download-side changes were needed.

## D-015: Gallery Folder Only Generated for Gallery Delivery

**Context:** When a user uploads a folder, the system generates thumbnails and stores them in `_gallery.{hash}/` inside the zip. However, users can choose three delivery modes: Browse (folder tree), Gallery (visual thumbnails), or Download (direct zip). The question was whether to always generate the gallery folder or only when the user selects gallery delivery.

**Decision:** Generate the `_gallery.{hash}/` folder only when the user selects "Gallery" delivery. For "Browse" and "Download" delivery, the zip contains the raw uploaded files with zero modifications (apart from existing dot-file and empty-directory cleanup).

**Rationale:** The folder view is a faithful representation of what's in the zip. If a user selects "Browse" delivery, the recipient sees a folder tree and downloads exactly what the sender uploaded. Injecting a `_gallery.{hash}/` folder into a non-gallery zip would break this contract — recipients would find an unexpected folder they did not upload. The principle is: the server stores what the sender intended, nothing more.

**Alternatives considered:**
- Always generate gallery folder regardless of delivery mode — rejected, pollutes non-gallery zips with processing artifacts the recipient did not ask for and the sender did not intend
- Generate but hide via filtering on download side — rejected, creates a hidden layer of processing that violates the "what you see is what you download" principle (and was the previous approach, which led to D-016)

**Outcome:** `_v0213_addPreviewToZip()` is now gated on `delivery === 'gallery'`. Browse and Download zips are pristine.

## D-016: No Filtering in Folder View — Show Everything in the Zip

**Context:** The previous implementation had a `filterGalleryFolders()` function that hid the `_gallery` folder from the folder tree view. This was designed to give recipients a "clean" view, but it meant the folder view did not match what the recipient would actually download.

**Decision:** Remove all filtering from the folder tree view. The folder view shows exactly what exists in the zip, including `_gallery.{hash}/` if present.

**Rationale:** Filtering creates a lie — the user sees one thing but downloads another. If the gallery folder is in the zip, the user should know it is there. With D-015 in place, the gallery folder only appears when the sender chose gallery delivery, so it is intentional content, not an artifact to hide. The folder view is a diagnostic tool: it answers "what will I download?" and must answer truthfully.

**Alternatives considered:**
- Keep filtering to show a "clean" view — rejected, breaks the trust contract between what the UI shows and what the download contains
- Filter visually but show a "hidden items" count — rejected, adds complexity without solving the core problem; D-015 already ensures the gallery folder is only present when intended

**Outcome:** Deleted `filterGalleryFolders()` function and removed file list filtering override from `send-download-v029.js`. Folder view is now a 1:1 representation of zip contents.

## D-017: SgPrint Extracted as Shared Utility Module

**Context:** Print functionality was needed in both the gallery lightbox (markdown viewer) and the folder browse view. The print logic includes A4 CSS styling, a toolbar with SG/Send branding, and a new-window print preview — roughly 194 lines that would need to be duplicated if embedded in each component.

**Decision:** Extract `SgPrint` as a standalone module at `v0.2.0/_common/js/sg-print.js`, symlinked into v0.2.3 and v0.2.13. Any component calls `SgPrint.printMarkdown(html, filename)` to open a print preview window.

**Rationale:** IFD methodology already established `_common/` as the location for shared utilities. The print module has no dependencies on any specific component — it receives HTML and a filename, opens a window, and provides print/close buttons. Placing it at the v0.2.0 level (the common ancestor) and symlinking makes it available to all UI versions without duplication.

**Alternatives considered:**
- Inline print logic in each component that needs it — rejected, duplicates 194 lines and creates divergence risk when one copy is updated but not the other
- Create a Web Component (`<sg-print-preview>`) — rejected as over-engineering; the module opens a new window, which is outside the component tree, so a Web Component wrapper adds no value
- Use the browser's native `window.print()` directly — rejected, native print lacks the branded toolbar, A4 preview styling, and header/footer that make the output professional

**Outcome:** One module, symlinked to all versions that need it. Adding print to a new view is a single `SgPrint.printMarkdown()` call.

## D-018: `@page` Without `size: A4` for Cross-Browser Compatibility

**Context:** The initial print CSS included `@page { size: A4; }` to lock the output to A4 paper dimensions. This worked correctly in Firefox but caused a problem in Chrome: when `size: A4` is specified, Chrome's print dialog disables the paper size dropdown, preventing users from selecting Letter, Legal, or other paper sizes.

**Decision:** Use `@page { margin: 20mm 15mm; }` without specifying `size`. The A4-like appearance comes from the margin and content styling, not from a locked paper size declaration.

**Rationale:** The goal is professional-looking printed output, not enforcement of a specific paper standard. Users in North America typically use Letter size; users in Europe and elsewhere use A4. Locking to A4 forces an inappropriate default on half the user base. The margins and typography produce clean output on any paper size.

**Alternatives considered:**
- Keep `@page { size: A4; }` — rejected, disables Chrome's paper size selector, which is a hostile UX for non-A4 users
- Detect browser and conditionally include `size: A4` for Firefox only — rejected, browser detection is fragile and the styling works well without `size` on all browsers
- Offer a paper size toggle in the print toolbar — rejected as over-engineering for a print preview window; the browser's own print dialog already provides this control

**Outcome:** Clean A4-ish output on all browsers. Chrome users can freely select their preferred paper size. Firefox users see A4 by default (Firefox's default). No browser-specific code.

## D-019: PDF Generation from Markdown Deferred to Future Version

**Context:** After building the print-to-PDF workflow (markdown rendered in browser, printed via browser print dialog), the question arose whether to add direct PDF generation — converting markdown to PDF programmatically without requiring the user to open a print dialog.

**Decision:** Defer PDF generation to a future gallery version. Create a briefing document (`v0.16.26__briefing__pdf-generation-for-markdown-in-gallery.md`) documenting the options and trade-offs, but do not implement.

**Rationale:** The browser print workflow already produces high-quality PDF output. Direct PDF generation would require either a server-side library (violates zero-knowledge architecture — server would see plaintext) or a client-side library like jsPDF or pdfmake (significant bundle size, limited CSS support, and the output would not match the browser's native rendering). The ROI is negative when a working solution already exists. The briefing document preserves the research for when the feature becomes worthwhile — likely when gallery needs batch PDF export or automated report generation.

**Alternatives considered:**
- Implement client-side PDF generation now using jsPDF — rejected, large library (~300KB), limited CSS table/heading support, and the output quality is visibly worse than browser-native print
- Implement server-side PDF generation — rejected outright, would require sending plaintext markdown to the server, breaking the zero-knowledge architecture
- Use a headless browser (Puppeteer) on the server — rejected, same plaintext exposure problem plus significant infrastructure complexity

**Outcome:** Browser print-to-PDF is the shipping solution. Briefing document created for future reference. No code added, no dependencies added.

## Milestones

### M-011: Print-Friendly Markdown Output

**Date:** 21 March 2026
**What:** Complete pipeline from gallery markdown file to professional A4 PDF via browser print. User clicks a markdown file in gallery, clicks Print, sees a branded preview window, clicks Print again, and the browser's native print/save-as-PDF dialog produces clean A4 output with headers, footers, and proper typography.

**Significance:** This closes the loop on document sharing. Senders upload folders containing markdown documentation; recipients view it in the gallery and print it as professional PDFs — all without the server ever seeing the content.

### M-012: Unified Delivery UX

**Date:** 21 March 2026
**What:** Upload Step 2 now always shows three delivery options (Browse, Gallery, Download) as visible cards with smart defaults. Gallery is pre-selected for folders, Download for single files. All three are always available regardless of file type.

**Significance:** Previously, delivery options were conditional and inconsistent — some modes appeared only for certain file types, and the ordering changed. The unified UX eliminates confusion and lets senders make an informed choice about how recipients will experience their files.

### M-013: Content-Addressed Gallery Folders

**Date:** 21 March 2026
**What:** Gallery folders are named `_gallery.{hash}/` using a hash derived from the content hashes of all uploaded files. Combined with gallery-only generation (D-015), the zip contents are deterministic and faithful to the sender's intent.

**Significance:** Content-addressing is the foundation for future deduplication and caching. Two identical uploads will produce identical gallery folder names, enabling server-side or CDN-level deduplication without the server knowing anything about the content.

## Summary

Six decisions across a 3-day Explorer session. The through-line is **fidelity and restraint**: the zip contains exactly what was intended (D-014, D-015), the UI shows exactly what's in the zip (D-016), print output works on any paper size (D-018), shared code lives in one place (D-017), and features that don't yet earn their complexity are documented but not built (D-019).

The most consequential decision was D-015 (gallery-only generation) because it established a clean separation between delivery modes. Before D-015, every zip contained gallery artifacts regardless of intent, and the download side had to filter them out — creating the lie that D-016 then had to address. D-015 solved both problems at the source: if you didn't ask for a gallery, you don't get gallery artifacts. The best filter is the one you don't need.
