# Decision Log — Chat Zone Redesign and Full-Pipeline Visibility

**Version:** v0.11.10
**Date:** 5 March 2026
**Role:** Historian

---

## Decision 1: Decompose `llm-chat` into 5 Event-Driven Components

**Context:** The monolithic `llm-chat.js` (~400 lines) handled system prompts, model selection, prompt input, API calls, streaming, and response display in a single component. Adding features required touching the entire file.

**Decision:** Split into 5 components: `llm-system-prompt`, `llm-prompt-input`, `llm-output`, `llm-orchestrator`, `js-highlighter`. Each communicates through DOM custom events.

**Alternatives considered:**
- Keep monolith, add features incrementally — rejected because coupling was already causing layout problems (LLM response overwriting Result panel)
- Use a state management library — rejected, violates zero-dependency IFD principle

**Outcome:** Each component is under 200 lines. Components can be developed, tested, and replaced independently. The orchestrator pattern keeps coordination logic in one place without coupling the visible components.

---

## Decision 2: Dedicated LLM Output Panel (Not Result)

**Context:** Previously, LLM responses were routed to the Result panel (the `document-viewer` showing transformation output). This meant sending a prompt would overwrite the user's transformation result.

**Decision:** Create a dedicated `llm-output` component in the chat zone's right column. The Result panel is exclusively for script execution output.

**Rationale:** Result and LLM Output serve fundamentally different purposes. Result shows what the script produced. LLM Output shows what the AI thinks. Users need both visible simultaneously to compare the AI's analysis against the actual output.

**Outcome:** The screenshot confirms both are visible: Result shows the filtered HTML, LLM Output shows Claude explaining the transformation. Copy-to buttons allow moving LLM content to Script, Data, or Result when desired.

---

## Decision 3: Always-Visible System Prompt with Auto-Context Preview

**Context:** The system prompt was hidden behind a collapse toggle. Users couldn't see what instructions the LLM was receiving. The Include checkboxes silently appended Source/Script/Result content to the prompt with no visibility.

**Decision:** Make the system prompt always visible in the chat zone's left column. Add a read-only "Auto-appended context" preview below it showing truncated previews of what will be sent based on Include checkbox state.

**Rationale:** "No surprises" principle — if the user checks "Include Source", they should see that the source HTML will be appended. This builds trust in the LLM interaction and helps users understand why the LLM gives certain responses.

**Outcome:** The screenshot shows the system prompt "you are an experience web developer and architect..." always visible, with auto-appended context showing truncated Source and Script content below.

---

## Decision 4: JS Syntax Highlighting via Regex Tokenizer + Overlay

**Context:** The script editor was a plain textarea with no syntax highlighting. Code readability was poor, especially for longer scripts.

**Decision:** Create a lightweight `js-highlighter` component (~100 lines) using regex-based tokenization. The editor uses a transparent `<textarea>` over a highlighted `<pre>` element — both scroll together, the textarea captures input, the pre shows colours.

**Alternatives considered:**
- CodeMirror / Monaco — rejected, massive dependencies violating IFD zero-dependency rule
- ContentEditable with inline styles — rejected, cursor management is a nightmare
- Textarea + overlay pattern — chosen, proven approach used by many lightweight editors

**Token types:** keywords (purple), strings (green), numbers (orange), comments (grey italic), builtins (yellow), operators (cyan), punctuation (light grey).

**Outcome:** The screenshot shows coloured syntax in the script editor. The regex approach handles JS well enough for transformation scripts (typically 5-50 lines). No dependencies added.

---

## Decision 5: Split Source into HTML + Data Sub-Panels

**Context:** The Source panel only showed one document. Upcoming workflows need both an HTML source and a data file (JSON, CSV, XML) visible simultaneously for data-driven transformations.

**Decision:** Split the Source zone into two vertically stacked sub-panels: HTML (top) and DATA (bottom), with a resize handle between them. Both use `document-viewer` instances with different `data-role` attributes.

**Rationale:** Data-driven transformations are a core use case — loading a JSON dataset, writing a script that merges it with HTML, and seeing the result. Having both visible prevents constant tab-switching.

**Outcome:** The screenshot shows the split: HTML source on top ("Tech News Daily"), DATA panel on bottom (empty, showing placeholder for .json/.csv/.xml files). The DATA panel is ready for the next phase of data integration.

---

## Decision 6: Include Checkboxes Control LLM Context

**Context:** The LLM needed to see workspace content to provide useful responses, but users needed control over what was sent.

**Decision:** Provide granular Include checkboxes: Source, Data, Script, Result, and Generate JS. Each checkbox determines whether that panel's content is appended to the LLM prompt. A "Gen JS" toggle instructs the LLM to produce executable JavaScript.

**Rationale:** Different tasks need different context. Asking "explain this script" needs Script checked. Asking "transform this HTML" needs Source + Script. Asking "write me a filter" needs Source + Gen JS. Users should control this explicitly.

**Outcome:** The screenshot shows Src, Script, and Result checked — the LLM received all three and correctly explained the full transformation pipeline. The Gen JS checkbox is unchecked (user was asking for explanation, not code generation).
