## Decision Records

### D-2026-03-04-001: Client-Side LLM Architecture — No Server Proxy

**Context:** SG/Send's core principle is that the server never sees plaintext. When adding LLM capabilities to the Workspace, the team had to decide whether LLM calls should route through the SG/Send backend or go directly from the browser to the LLM provider.

**Decision:** All LLM calls are made directly from the browser to the provider (OpenRouter or Ollama). No SG/Send backend endpoint is involved.

**Rationale:**
- Maintains zero-knowledge architecture — decrypted document content never touches the SG/Send server
- Eliminates an unnecessary proxy hop and its latency
- API keys are stored in the user's browser (localStorage), not on the server
- Users choose their own provider and model — SG/Send doesn't need to manage LLM billing
- Ollama support means fully offline, fully local LLM usage is possible

**Trade-off:** Users must manage their own API keys. The application cannot track or bill for LLM usage. Accepted because the zero-knowledge principle is non-negotiable.

**Status:** Implemented. Both providers confirmed working on 4 March 2026: OpenRouter (cloud) and Ollama (local, Gemma3:4b).

### D-2026-03-04-002: Ollama CORS — OLLAMA_ORIGINS Required

**Context:** When the workspace at `https://dev.workspace.sgraph.ai` attempted to call the local Ollama server at `http://localhost:11434`, the browser blocked the request due to CORS (Cross-Origin Resource Sharing). Ollama's default configuration does not allow cross-origin requests from HTTPS origins.

**Solution:** Launch Ollama with the `OLLAMA_ORIGINS` environment variable:
```
OLLAMA_ORIGINS="https://dev.workspace.sgraph.ai" ollama serve
```

**Rationale:** This is the correct Ollama mechanism for allowing browser-based clients. The alternative (proxying through the server) would violate the zero-knowledge principle. The user controls which origins can access their local Ollama instance.

**Note for users:** This is a per-session setting. Users running the workspace from different origins (e.g., `localhost:8000` for development) will need to adjust the value accordingly. Multiple origins can be comma-separated.

**Status:** Confirmed working. Gemma3:4b model responded successfully from the workspace UI.

### D-2026-03-04-003: Streaming Architecture — SSE for OpenRouter, NDJSON for Ollama

**Context:** Both LLM providers support streaming responses, but use different wire formats.

**Decision:** Implement two stream readers:
- `_readSSE()` for OpenRouter (Server-Sent Events, `data: {"choices":[{"delta":{"content":"..."}}]}`)
- `_readNDJSON()` for Ollama (newline-delimited JSON, `{"response":"...","done":false}`)

**Rationale:** Each provider has a native streaming format. Adapting at the reader level keeps the component logic clean — `_send()` calls the right streamer based on provider, and both feed chunks to the same `onChunk` callback that updates the transform viewer.

**Status:** Implemented. Real-time rendering confirmed — transform viewer updates on each chunk.

---

### D-2026-03-04-004: Document Transform Pipeline — Vault → Source → LLM → Transform → Vault

**Context:** The Workspace needed a workflow for applying LLM transformations to encrypted vault documents.

**Decision:** Implement a five-stage pipeline:
1. **Vault → Source:** User selects file in vault-panel, file is downloaded and decrypted client-side, rendered in source document-viewer
2. **Source → Prompt:** User writes prompt (or selects from prompt-library), `{{document}}` placeholder is replaced with source content
3. **Prompt → LLM:** Browser sends prompt directly to provider, streams response
4. **LLM → Transform:** Each response chunk accumulates and re-renders in transform document-viewer
5. **Transform → Vault:** User clicks "Save to Vault", content is encrypted and uploaded back to vault

**Rationale:** Every stage operates on the client side. Decrypted content exists only in browser memory. The server stores and serves encrypted blobs — it never sees the document, the prompt, or the LLM response.

**Status:** Full pipeline implemented and working. First successful end-to-end run confirmed 4 March 2026.

### D-2026-03-04-005: Custom Element Upgrade Order Fix

**Context:** During first testing, the workspace crashed with `conn.getModels is not a function`. Root cause: when `workspace-shell` sets `innerHTML`, child custom elements are created in DOM order, but `connectedCallback` fires before all siblings are upgraded. `llm-chat` fires before `llm-connection` is upgraded, so `querySelector('llm-connection')` returns a bare `HTMLElement` with no custom methods.

**Decision:** Guard all cross-component method calls with `typeof` checks:
```javascript
const isUpgraded = conn && typeof conn.getModels === 'function';
```

**Rationale:** Web Components spec does not guarantee upgrade order for sibling elements inserted via innerHTML. Defensive checks are the correct pattern. The component re-renders when the `llm-connected` event fires, so initial graceful degradation is acceptable.

**Status:** Fixed. Added try/catch in `connectedCallback` to route errors to the debug messages panel.

## Significance

This milestone marks SG/Send's evolution from a file-sharing tool to a **document intelligence platform**. The architecture proves that zero-knowledge encryption and LLM-powered document transformation are not mutually exclusive — you can have both, with no compromise on either.

The entire LLM pipeline runs in the browser. The server remains a dumb encrypted-blob store. This is, to our knowledge, one of the first implementations of a fully client-side, zero-knowledge LLM document transformation workflow.
