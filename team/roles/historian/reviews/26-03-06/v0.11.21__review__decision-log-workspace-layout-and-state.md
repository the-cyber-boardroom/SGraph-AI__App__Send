# Decision Log — Workspace Layout and State Management

**Version:** v0.11.21
**Date:** 6 March 2026
**Role:** Historian
**Scope:** Decisions made between v0.11.10 (5 March afternoon) and v0.11.21 (6 March morning)

---

## D-007: IFD Overlay Pattern for v0.1.1

**Context:** Execution bundles needed to add significant functionality to the workspace — state capture, restore, fork tracking, and panel layout management. This required changes across multiple components, but v0.1.0 was a stable, working release.

**Decision:** Freeze v0.1.0 completely. Create v0.1.1 as a surgical overlay that uses `customElements.get()` prototype patching to extend existing components without modifying their source files.

**Rationale:** The IFD methodology demands that each version is a self-contained, immutable artifact. Modifying v0.1.0 files would break the versioning contract and make rollback impossible. The overlay pattern lets v0.1.1 add capabilities by patching prototypes at load time — the original component code never changes.

**Alternatives considered:**
- Modify v0.1.0 files directly — rejected, violates IFD immutability principle and makes rollback impossible
- Copy all v0.1.0 files into v0.1.1 and modify the copies — rejected, creates maintenance burden of duplicated code

**Outcome:** v0.1.0 remains untouched. v0.1.1 adds bundles, layout fixes, and state management via 5 overlay files that patch existing components at runtime. Rolling back to v0.1.0 means simply not loading the overlay.

## D-008: Execution Bundles as Complete State Snapshots

**Context:** Users experimenting with LLM-assisted transformations needed to save and restore workspace state — the source document, data, script, execution result, console output, LLM prompts, configuration, and screenshots. Partial saves would lose critical context about what produced a given result.

**Decision:** Capture everything into JSON bundles: source content, data content, script text, execution result, console log, LLM system prompt, LLM conversation history, model configuration, include-checkbox states, panel collapse states, and screenshots. Bundles are stored in the vault's `.bundles/` folder.

**Rationale:** The workspace is an experimentation environment. The value is in reproducing exactly what happened — "I had this source, wrote this script, got this result, asked the LLM this question, and got this response." Partial snapshots would make it impossible to understand why a particular output was produced.

**Alternatives considered:**
- Save only prompts and responses (partial state) — rejected, loses the transformation context that makes the LLM conversation meaningful
- Save to browser localStorage only — rejected, not portable across devices and lost on cache clear

**Outcome:** Full time travel — clicking any bundle restores the exact workspace state, including all panel contents, LLM conversation, and configuration. Users can jump between experiments instantly.

## D-009: Fork Tracking via parent_id

**Context:** LLM experimentation is inherently branching. A user runs a transformation, saves it, then tries a different approach — that is a fork. Later they might return to the original and try a third variation. Understanding these relationships is essential for making sense of experiment history.

**Decision:** Each bundle records a `parent_id` field linking to the bundle it was forked from. When a user clicks "Save as Bundle" while a previous bundle is loaded, the new bundle's `parent_id` is set to the loaded bundle's ID.

**Rationale:** Linear history (ordered by timestamp) cannot represent the actual workflow. Fork tracking captures the causal relationship: "I started from bundle X, changed the script, and saved as bundle Y." This is the minimum metadata needed to reconstruct the exploration tree.

**Alternatives considered:**
- Linear history only (ordered by timestamp) — rejected, loses the causal relationships between experiments
- Full git-style branching with merge — rejected as over-engineering for Phase 2; tree visualization deferred to Phase 3

**Outcome:** Every bundle knows its parent. The data structure supports tree visualization in a future phase. For now, the parent chain is visible in bundle metadata.

## D-010: Collapsible Panels as Toggleable Grid Columns

**Context:** The 4-panel workspace (Source, Data, Script, Result) competes for horizontal space. Users working on a script want the Script panel maximized. Users reviewing LLM output want the Result panel maximized. A fixed layout wastes space on panels the user is not currently focused on.

**Decision:** Each panel header is a toggle button. Clicking it collapses the panel to 28px width, showing only a vertical header label. Clicking again restores it. Panel collapse states are stored in localStorage and included in execution bundles.

**Rationale:** Unlike tabs (which completely hide panels), collapsed panels remain visible as labeled strips. Users can see at a glance which panels are available and restore any panel with a single click. The 28px collapsed width is enough for a rotated label without wasting meaningful space.

**Alternatives considered:**
- Tabs to hide panels completely — rejected, users lose awareness of what panels exist and what content they contain
- Drag-to-resize only — rejected, requires precise mouse interaction and does not provide a quick "maximize this panel" action
- Sidebar drawer pattern — rejected, does not scale to 4 independent panels

**Outcome:** All 4 panels are always visible. Collapsed panels show a vertical header label at 28px width. One-click restore. State persists across page reloads.

## D-011: 4-Column Uniform Grid over Flex/Grid Hybrid

**Context:** The original layout used a nested approach: Source and Data were stacked vertically using flexbox inside the first grid column, while Script and Result occupied the remaining grid columns. When panels collapsed, the flex container did not respond correctly to grid column changes, causing a cascade of 6 CSS bugs — collapsed panels leaving gaps, panels overflowing their boundaries, and inconsistent widths.

**Decision:** Flatten the entire layout to 4 equal CSS grid columns: Source | Data | Script | Result. Each panel is a direct child of the grid container. A single `grid-template-columns` declaration controls all panel widths, with collapsed panels getting `28px` and expanded panels getting `1fr`.

**Rationale:** The flex/grid hybrid created two layout systems that had to agree on sizing. When they disagreed (which happened on every collapse/expand), bugs appeared. A single grid system means one source of truth for column widths. The fix was not to add more CSS — it was to remove the unnecessary layer.

**Alternatives considered:**
- Keep flex/grid hybrid, fix bugs individually — rejected, each fix created new edge cases because the two layout systems could not be reconciled
- Use flexbox for everything — rejected, CSS grid's `grid-template-columns` is the natural fit for a multi-column workspace

**Outcome:** Single `grid-template-columns` expression controls all 4 panels. Eliminated 6 collapse-related CSS bugs. Layout code went from ~40 lines of mixed flex/grid to ~15 lines of pure grid.

## D-012: Centralized `applyAllPanelState()` as Single Source of Truth

**Context:** Panel collapse logic had drifted into two locations: `workspace-shell` had the authoritative version using the 4-column grid, but `bundle-manager` had a stale duplicate that still assumed a 3-column layout (from the pre-D-011 hybrid). When bundle restore triggered the stale version, it overwrote the correct layout with broken 3-column CSS.

**Decision:** One function — `applyAllPanelState()` in `workspace-shell` — owns all layout decisions. No other component is permitted to manipulate `grid-template-columns` or panel visibility. Other components that need to trigger layout recalculation do so by dispatching events, not by running their own layout code.

**Rationale:** Duplicated layout logic is a bug factory. The bundle-manager's copy was already stale and causing regressions. The correct fix is not to update the duplicate — it is to delete it entirely. One function, one location, one truth.

**Alternatives considered:**
- Keep separate implementations in sync via shared constants — rejected, "keep in sync" is a maintenance promise that always breaks
- Move layout logic to a shared utility module — rejected, adds indirection without solving the ownership problem; workspace-shell is the layout owner by definition

**Outcome:** Removed ~50 lines of duplicated layout code from bundle-manager. The best fix is less code. All layout changes now flow through a single function.

## D-013: Event-Driven Component Communication over Direct Coupling

**Context:** After centralizing layout logic in workspace-shell (D-012), bundle-manager still needed a way to trigger layout recalculation after restoring a bundle's panel states. But `applyAllPanelState` is a module-scoped function inside workspace-shell — not exported, not globally accessible. Bundle-manager needed to tell workspace-shell "panel states have changed, recalculate layout" without directly calling its internal function.

**Decision:** Use DOM CustomEvent dispatch. Bundle-manager writes panel states to localStorage, then fires a `workspace-panels-changed` event on the document. Workspace-shell listens for this event and calls `applyAllPanelState()` internally.

**Rationale:** This is the Web Components way — components communicate through events, not through direct function calls or shared globals. The event name is the contract. Bundle-manager does not need to know that workspace-shell exists, only that something will respond to the event. This makes both components independently replaceable.

**Alternatives considered:**
- Export `applyAllPanelState` globally via `window` — rejected, creates global namespace pollution and tight coupling
- Have bundle-manager query the workspace-shell element and call a method on it — rejected, creates a direct dependency on workspace-shell's API surface
- Use a shared event bus / pub-sub library — rejected, a DOM CustomEvent already is an event bus; adding a library violates zero-dependency IFD principle

**Outcome:** Zero coupling between components. Bundle-manager writes state + fires event. Workspace-shell listens + applies layout. Either component can be replaced without affecting the other. The contract is a single event name: `workspace-panels-changed`.

## Summary

Seven decisions spanning one evening and one morning. The through-line is **simplification through ownership**: one overlay mechanism (D-007), one snapshot format (D-008), one parent chain (D-009), one collapse pattern (D-010), one grid system (D-011), one layout function (D-012), one communication pattern (D-013). Each decision reduced the number of things that could disagree with each other.

The most impactful decision was D-011 (4-column grid) because it eliminated an entire class of bugs by removing architectural complexity rather than adding defensive code. D-012 reinforced this by deleting 50 lines of duplicated logic. The best code is the code you delete.
