## The Event

On 26 February 2026, the SGraph Send system achieved a first: an AI agent autonomously generated a document, encrypted it, uploaded it to send.sgraph.ai, and delivered it to a human via email -- all without any human intervention in the pipeline. The server never saw the plaintext. The recipient decrypted the file in their browser. The original filename was preserved through the entire chain.

**The pipeline:**

1. **Claude.ai Sherpa Agent** generated a PDF document describing the zero-knowledge encrypted file transfer workflow
2. The PDF was wrapped in **SGMETA envelope** -- a binary format that preserves the original filename through encryption
3. The SGMETA-wrapped payload was encrypted with **AES-256-GCM** using Web Crypto API patterns
4. The encrypted blob was uploaded to **send.sgraph.ai** via the MCP connector, using the new `data` parameter (introduced in v0.6.32)
5. A download link was generated with the decryption key embedded in the URL fragment (never sent to the server)
6. The Sherpa Agent sent an **email via n8n MCP connector** to an AWS WorkMail account with the encrypted download link
7. Dinis Cruz received the email in his inbox, clicked the link
8. The browser decrypted the file successfully using the key from the URL fragment
9. **Filename was preserved** -- `encrypted-send-workflow.pdf` -- proving the SGMETA envelope survived the full round-trip
10. The PDF rendered inline in the browser -- 5.0 KB, 2 pages

**The transparency panel confirmed zero-knowledge properties:**
- File name: encrypted (only recipient can read it)
- File content: encrypted (only recipient can read it)
- Decryption key: NOT stored on server

## Why This Matters

### AI Agents as First-Class Participants in Zero-Knowledge Systems

M-007 (25 February 2026) proved that an AI agent could encrypt and upload a file. M-008 completes the loop: the AI agent now **delivers** the encrypted file to a specific recipient via email, without human involvement in any step of the pipeline.

This is significant for three reasons:

1. **End-to-end autonomous delivery.** The AI agent did not merely create a link and display it in a chat window. It composed an email, attached the download link, and delivered it to a specific recipient's inbox. The human's first interaction with the document was opening their email.

2. **The SGMETA envelope works in production.** The filename `encrypted-send-workflow.pdf` survived: generation by Claude, SGMETA wrapping, AES-256-GCM encryption, upload to S3, download by the browser, decryption, and SGMETA unwrapping. The server never saw the filename -- it was encrypted inside the SGMETA binary format -- yet the recipient received the file with its correct name and extension.

3. **Multi-MCP-connector orchestration.** The Sherpa Agent used two separate MCP connectors in a single workflow: the send.sgraph.ai MCP connector for encryption and upload, and the n8n MCP connector for email delivery via AWS WorkMail. This demonstrates that AI agents can compose multiple MCP tools into workflows that would traditionally require custom integration code.

### The "Invisible Encryption" Thesis Extends to Email

The original thesis of SGraph Send was that encryption should be invisible -- neither the sender nor the recipient should need to understand cryptography. M-008 extends this to a new scenario: the sender is an AI agent, the delivery mechanism is email, and the recipient simply clicks a link. At no point did anyone -- human or AI -- need to handle raw cryptographic keys, manage certificates, or configure encryption parameters.

|-----------|-----------------|------------|

| SGraph Send MCP Connector | Encrypted the file, uploaded it, returned the download link | MCP tool, AES-256-GCM, SGMETA envelope |

| SGMETA Envelope | Preserved the filename through encryption | Binary format: metadata + encrypted content |

### SGMETA Envelope -- Key Detail

The SGMETA format wraps the original file with its metadata (filename, content type) before encryption. This means the server stores a single encrypted blob that contains both the file content and its metadata. On decryption, the browser parses the SGMETA binary format to extract the original filename and content, then presents the file with its correct name.

Without SGMETA, the server would need to store the filename in plaintext metadata -- breaking the zero-knowledge property. With SGMETA, the filename is encrypted alongside the content.

## What Made This Possible

### The `data` Parameter Fix (v0.6.32, same day)

Earlier on 26 February 2026, a fix was applied to the MCP upload route to accept a `data` parameter for direct binary content. Before this fix, Claude.ai's MCP connector could not pass file content through the MCP tool call because the transport layer expected file uploads via multipart form data -- which MCP does not support. The `data` parameter allowed base64-encoded content to be passed directly in the JSON tool call payload.

This was the critical unblocking fix. Without it, the Sherpa Agent had no way to transmit file content to send.sgraph.ai.

### The SGMETA Documentation in the Sherpa Guide

The Sherpa guide documented the SGMETA binary format and its wrapping/unwrapping logic. This enabled Claude.ai to understand how to construct an SGMETA-wrapped payload before encryption, ensuring the filename would survive the round-trip. The documentation was written for AI agents as readers, not just human developers.

### The MCP Upload Route

The MCP integration (first shipped in v0.6.20 for M-007) provided the foundation. M-008 built on this by using the corrected `data` parameter and combining it with the n8n MCP connector for email delivery.

### AWS WorkMail Integration via n8n

Dinis Cruz configured an n8n workflow that exposes email-sending capability as an MCP tool. The Sherpa Agent was given access to this tool alongside the send.sgraph.ai MCP connector. This is infrastructure configuration, not code -- the n8n workflow was set up by the human, and the AI agent consumed it as a tool.

## Key Decisions That Led Here

| ID | Decision | When | Why It Mattered for M-008 |
|----|----------|------|--------------------------|
| D-M008-001 | Add `data` parameter to MCP upload instead of changing MCP transport | 26 Feb 2026 | Kept the fix simple and backwards-compatible. The MCP spec does not support multipart uploads, so the parameter approach was the only viable path. |
| D-M008-002 | Keep SGMETA wrapping client-side | Pre-M-008 | The server never constructs or parses SGMETA. This preserves zero-knowledge -- the server cannot extract filenames even if it wanted to. The AI agent (client-side) handles SGMETA wrapping before encryption. |
| D-M008-003 | Sherpa Agent architecture -- composable MCP tools | v0.6.20+ | The decision to give the Sherpa Agent access to multiple MCP connectors (send.sgraph.ai + n8n) enabled multi-step workflows. The agent orchestrates, each tool handles one responsibility. |
| D-M008-004 | Document SGMETA format for AI agent consumption | v0.6.20+ | Writing the Sherpa guide with AI agents as the intended audience meant Claude.ai could understand and correctly implement the SGMETA wrapping without human assistance. |
| D-M008-005 | Use n8n as email bridge rather than direct SMTP | Pre-M-008 | n8n provides MCP-compatible tool exposure for email. Building direct SMTP into the Sherpa Agent would have required custom code. n8n made email delivery a simple tool call. |

## What This Unlocks

### 1. AI Agents as Autonomous Senders

AI agents can now be full participants in the zero-knowledge system. They can generate documents, encrypt them, and deliver them -- all without human supervision of the technical steps. The human's role shifts from "operate the tool" to "receive the output."

### 2. Automated Encrypted Document Delivery Pipelines

The combination of MCP connectors (encryption + email) creates a reusable pattern: any AI agent with access to these two tools can build encrypted delivery workflows. Use cases include:

- **Investor materials:** AI-generated due diligence packs, encrypted and emailed to specific investors
- **Legal documents:** Confidential documents generated and delivered with zero-knowledge guarantees
- **Automated reports:** Periodic encrypted reports delivered to a distribution list
- **Compliance deliverables:** Audit documents encrypted and delivered with an immutable trail

### 3. The "Agentic Security" Investment Thesis Becomes Demonstrable

The Alchemist team's narrative -- that AI agents will need zero-knowledge encryption as a primitive -- is no longer theoretical. M-008 is a working demonstration. An AI agent generated a document, encrypted it so the server could not read it, and delivered it to a human who decrypted it in their browser. This is a demo that can be shown to investors.

### 4. Multi-MCP-Connector Orchestration Pattern

The Sherpa Agent composed two MCP connectors into a workflow that neither connector could perform alone. This pattern -- AI agent as workflow orchestrator across multiple MCP tools -- is extensible. Future connectors (Slack, Teams, calendar, CRM) would plug into the same pattern without code changes to the agent.

### 5. SGMETA Envelope Validated in Production

The SGMETA format has now been validated end-to-end in a production pipeline: client-side wrapping, encryption, server storage, download, decryption, unwrapping. The filename survived every step. This means SGMETA is ready for use in all SGraph Send transfers, not just MCP-initiated ones.

## Decision Patterns Observed

### Pattern: Fix the Smallest Thing, Unlock the Largest Capability

The `data` parameter fix was a small, targeted change to the MCP upload route. But it unblocked an entire class of AI-agent-initiated workflows. The pattern: identify the minimal blocker, fix it surgically, and let the existing architecture do the rest.

### Pattern: AI Agents Compose Tools, Not Code

The Sherpa Agent did not write custom integration code. It called two MCP tools in sequence: encrypt-and-upload, then send-email. The orchestration logic was in the AI agent's reasoning, not in a codebase. This is a different model from traditional software integration, where a developer would write a script to connect the encryption service to the email service.

### Pattern: Documentation Enables AI Agent Autonomy

The Sherpa guide documented the SGMETA format, the MCP tool parameters, and the encryption workflow in a way that AI agents could consume. Without this documentation, Claude.ai would not have known how to construct the SGMETA binary envelope or which parameters to pass to the MCP upload tool. Documentation quality directly determines AI agent capability.

### Pattern: Same-Day Bug Fix to Milestone

The `data` parameter bug was discovered, fixed, and the resulting capability was demonstrated and announced on LinkedIn -- all on the same day. This speed is characteristic of the Explorer team's operating model: ship fast, validate immediately, announce publicly.
