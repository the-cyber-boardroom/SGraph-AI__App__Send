## The Event

At approximately 04:34 UTC on 25 February 2026, Claude.ai (Anthropic's web-based AI assistant) successfully generated a PDF document, encrypted it with AES-256-GCM, and transmitted it to a human user through SGraph Send's MCP integration. The human decrypted the PDF in their browser. The server never saw the plaintext.

**The document:** A 1.9 KB PDF titled **"Hello from Claude.ai Web!"** containing the text:
> *"This PDF was generated, encrypted with AES-256-GCM, and uploaded to SG/Send — all from a Claude.ai conversation using the SG/Send MCP connector. The server never saw the plaintext. Only the recipient with the decryption key can read this."*

**The URL:** `send.sgraph.ai/send/v0/v0.1/v0.1.6/download.html#0663aa105a75/3jJsiGjCjKbmlZuKcwRw6Z9jzBAcC5z0d1tKrhJaDN0`

The decryption key (`3jJsiGjCjKbmlZuKcwRw6Z9jzBAcC5z0d1tKrhJaDN0`) was embedded in the URL fragment — never sent to the server — and the browser's Web Crypto API (AES-256-GCM) performed the decryption locally.

### Verified Evidence (Screenshot from SGraph Send Download Page)

The human stakeholder confirmed success with a screenshot showing:
- **Green banner:** "File decrypted and saved successfully."
- **File type:** PDF, 1.9 KB
- **Upload time:** Wed, 25 Feb 2026 04:34:22 GMT
- **Download time:** Wed, 25 Feb 2026 04:41:06 GMT
- **Encryption:** AES-256-GCM
- **File name:** "Encrypted (only you can read it)" — server never received the original filename
- **File content:** "Encrypted (only you can read it)" — server stored only ciphertext
- **PDF rendered in browser:** Full document visible with title and body text, confirming successful decryption
- **Zero-knowledge indicators:** Both file name and file content marked as "only you can read it" — the server's confirmation that it holds no plaintext

This is notable because Claude didn't just send raw text — it **generated a structured PDF document** that self-documents the encryption process, then encrypted and transmitted it. The AI created a rich document format, not just a string.

## Why This Matters

This is the first known instance of an AI agent sending a zero-knowledge encrypted message to a human through a production system where:

1. **The AI initiated the transfer** — Claude chose to create the transfer, upload the encrypted payload, and generate the download link
2. **The server held only ciphertext** — the zero-knowledge property was preserved throughout
3. **The human decrypted in-browser** — no special software required, standard Web Crypto API
4. **No API keys were exposed** — MCP tool calls, not raw API credentials in Claude's context

This crosses the boundary from "humans use encryption tools" to "AI agents participate in encrypted communication channels."

## Historical Parallel: The Evolution of Encrypted Communication

| Era | Who Encrypts | Who Decrypts | Trust Model |
|-----|-------------|-------------|-------------|
| PGP (1991) | Human (manual) | Human (manual) | Web of Trust |
| TLS (1995) | Browser (automatic) | Server (automatic) | Certificate Authorities |
| Signal (2014) | App (automatic) | App (automatic) | Trust on First Use |
| SGraph Send (2026) | AI Agent (MCP) | Human (browser) | Zero-Knowledge + URL Fragment |

**The pattern:** Each generation made encryption more invisible. PGP required expertise. TLS required nothing from the user. Signal required installing an app. SGraph Send requires clicking a link.

**The new dimension:** For the first time, one party in the encrypted exchange is an AI agent. The agent doesn't "know" cryptography — it calls MCP tools that handle encryption. The human doesn't know cryptography — they click a link and the browser handles decryption. Both sides are abstracted away from the complexity.

### The Critical Bug

The final commit (`0f02ec4`) fixed the issue where MCP uploads were storing the JSON request wrapper instead of the raw encrypted bytes. This meant downloads returned `{"content": "base64data..."}` instead of the actual ciphertext, causing decryption to fail with garbage output. The fix extracted the raw binary content before storage.
