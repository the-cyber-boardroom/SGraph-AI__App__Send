## Context

SGraph Send launched as an encrypted file-sharing tool with in-memory storage, later upgraded to S3-backed storage via Memory-FS. The initial architecture proxied files through Lambda, which imposed a hard 6MB payload limit on transfers. Users identified large file support as their primary pain point, making it the highest-priority feature request.

The solution required two innovations:
1. **Presigned S3 multipart upload** -- bypass the Lambda payload limit entirely by having the browser upload encrypted chunks directly to S3
2. **Parallel chunk uploads** -- overcome the sequential bottleneck that made the first working implementation too slow for production use

This decision record captures the technical choices made during implementation and the milestone declaration by the project owner.

## Technical Decisions

### D-PAR-001: Presigned S3 Multipart Upload

| Field | Value |
|-------|-------|
| Date | February 2026 |
| Made By | Explorer team |
| Decision | Use S3 presigned URLs for multipart upload, with the browser uploading encrypted chunks directly to S3 |
| Why | Lambda has a 6MB payload limit (API Gateway) / ~6MB for Lambda Function URLs. Files larger than this cannot be proxied through Lambda. Presigned URLs allow the browser to upload directly to S3, bypassing Lambda entirely for the data path. Lambda only handles the control path (initiate, complete, abort). |
| Alternatives Rejected | Increasing Lambda payload limit (hard AWS limit, not configurable); chunking through Lambda (would require reassembly on server side, adds latency, wastes Lambda compute); external upload service (adds infrastructure complexity) |
| Supersedes | Original proxy-through-Lambda upload path for files exceeding the Lambda payload limit |

### D-PAR-002: Parallel Upload with Concurrency Pool of 5

| Field | Value |
|-------|-------|
| Date | February 2026 |
| Made By | Explorer team |
| Decision | Upload up to 5 chunks concurrently using a Set-based concurrency pool with Promise.race for backpressure |
| Why | Sequential upload of a 42MB file took ~70 seconds. The network waterfall showed each chunk waiting for the previous one to complete, leaving bandwidth unused. With 5 concurrent uploads, the same file is estimated to complete in ~15-20 seconds -- a 3.5-4.7x speedup. The concurrency limit of 5 prevents browser connection exhaustion (browsers typically allow 6 concurrent connections per host). |
| Alternatives Rejected | Unlimited parallelism (would exhaust browser connection pool, cause timeouts); concurrency of 2-3 (insufficient speedup); concurrency of 10+ (diminishing returns, risk of S3 throttling); Web Workers for parallel uploads (adds complexity, not needed for network-bound I/O) |
| Key Constraint | Must work in vanilla browser JS with zero external dependencies (no async pool libraries, no worker threads) |

### D-PAR-003: Set + Promise.race Pattern for Concurrency Control

| Field | Value |
|-------|-------|
| Date | February 2026 |
| Made By | Explorer team |
| Decision | Use a JavaScript Set to track in-flight promises and Promise.race for backpressure, rather than importing a concurrency library |
| Why | The pattern is ~10 lines of code. A Set tracks active upload promises. When the Set reaches size 5, Promise.race waits for any one to complete before adding the next. This is a well-known pattern for bounded concurrency in browser JS. It requires zero dependencies, which aligns with the IFD methodology (zero framework dependencies). |
| Alternatives Rejected | p-limit / p-queue (npm dependency -- violates zero-dependency rule); Promise.all with fixed batches (wastes bandwidth -- if 4 of 5 finish fast, the batch waits for the slowest); async generator with for-await (more complex, harder to reason about error handling) |

## Business Impact: Prototype to Product

Dinis Cruz's declaration -- "this moves the P in MVP from Prototype to Product" -- marks a significant inflection point in the project's history.

### What Changed

| Dimension | Before (Prototype) | After (Product) |
|-----------|-------------------|-----------------|
| File size | Limited by Lambda payload (~6MB) | Unlimited (multipart, direct-to-S3) |
| Upload speed | Sequential chunks (~70s for 42MB) | Parallel chunks (~15-20s for 42MB) |
| User perception | "It works but I can't send real files" | "It handles my actual use case" |
| Market position | Technical demo | Usable tool |

### Why This Matters

The transition from Prototype to Product is not a code milestone -- it is a product milestone. It means:

1. **The core use case works at production scale.** Users asked for large file support. They got it, with acceptable performance.
2. **The architecture scales.** Presigned URLs + parallel upload is the same pattern used by production file-sharing services. The implementation can be tuned (concurrency, chunk size) without architectural changes.
3. **The zero-knowledge model holds.** Files are still encrypted in the browser before upload. Each chunk is a fragment of ciphertext. S3 never sees plaintext. The presigned URL flow does not compromise the security model.

## What Was Considered but Deferred

| # | Feature | Why Deferred | When to Revisit |
|---|---------|-------------|-----------------|
| 1 | Streaming encryption | Currently the entire file is encrypted in memory before chunking begins. Streaming would encrypt and upload simultaneously, reducing memory usage for very large files. | When users report memory issues with files >500MB |
| 2 | Per-chunk retry with exponential backoff | If a single chunk upload fails, the entire upload currently fails. Individual chunk retry would improve resilience on unreliable networks. | When users report upload failures on unstable connections |
| 3 | AbortController integration | Users cannot currently cancel an in-progress upload. AbortController would allow graceful cancellation of all in-flight chunk uploads. | Next UI iteration -- this is a UX feature, not a reliability feature |
| 4 | Dynamic concurrency tuning | The concurrency pool is fixed at 5. Adaptive tuning based on measured throughput could optimise for different network conditions. | When performance data from production usage is available |
| 5 | Download-side parallelism | Downloads are still sequential. The same parallel pattern could be applied to multipart download. | After upload parallelism is validated in production |
| 6 | Progress reporting per chunk | UI could show individual chunk progress rather than overall progress. | Next UI polish pass |

## Decision Patterns Observed

### Pattern: User Feedback Drives Prioritisation

This feature was not on the original roadmap. It was prioritised because users identified it as their primary pain point. The project responded within the same development cycle. This confirms the Explorer team's operating model: build what users need, not what the roadmap says.

### Pattern: Sequential First, Then Optimise

The implementation followed a deliberate two-step pattern:
1. Build it sequentially (correct but slow)
2. Make it parallel (correct and fast)

This is the same pattern seen in the Memory-FS evolution (memory backend first, S3 backend second) and the PKI implementation (encryption first, signatures second). The pattern de-risks delivery: step 1 proves correctness, step 2 proves performance.

### Pattern: Zero-Dependency Discipline Holds Under Pressure

When parallelism was needed, the team could have reached for an npm package. Instead, they implemented the Set + Promise.race pattern in ~10 lines of vanilla JS. This discipline -- no external dependencies in the browser -- has been maintained since v0.1.0 and continues to hold even as the features grow more sophisticated.
