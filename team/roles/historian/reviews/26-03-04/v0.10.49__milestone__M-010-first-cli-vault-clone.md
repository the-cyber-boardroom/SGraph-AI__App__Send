## The Event

On 4 March 2026, within hours of proving cross-browser vault access (M-009), the SG_Send__CLI successfully executed a full vault clone workflow from the terminal:

```bash
sg-send-cli clone "test-vault-1:7f2f5e54"
cd 7f2f5e54
sg-send-cli status
sg-send-cli pull
```

All four commands completed successfully. The vault files were decrypted and written to the local filesystem — creating the **first offline copy** of an encrypted vault stored on send.sgraph.ai.

The deterministic vault key fix that enabled cross-browser access (M-009) is the same fix that enabled CLI access — the 2-part key format (`passphrase:vaultId`) works identically across all three clients.

## What This Proves

### 1. Three Independent Access Methods
The same encrypted vault is now accessible from three completely independent clients:

| Client | Environment | Purpose | Verified |
|--------|-------------|---------|----------|
| **Vault UI** | Safari, `dev.vault.sgraph.ai` | Browse, edit, manage files | M-009 |
| **Workspace** | Chrome, `dev.workspace.sgraph.ai` | Load, transform with LLM, save | M-009 |
| **SG_Send__CLI** | Terminal, local machine | Clone, pull, offline access | **M-010** |

All three derive the same AES keys from the same 2-part vault key. All three independently decrypt. The server has no idea which client is which.

### 2. Git-Like Semantics for Encrypted Vaults
The CLI implements familiar git workflows:
- `clone` — download and decrypt all vault files to local filesystem
- `status` — show vault state and file listing
- `pull` — sync changes from remote vault (returns "Already up to date." when in sync)

Developers and CI/CD pipelines can interact with encrypted vaults using muscle memory from git.

### 3. First Offline Vault Copy
This is the first time encrypted vault content has existed as decrypted files on a local filesystem. This enables:
- **Offline access** — work on vault files without network connectivity
- **IDE integration** — open vault files in any code editor (Cursor, VS Code, etc.)
- **Pipeline integration** — CI/CD can clone vaults, process files, push results back
- **Backup** — local copies of encrypted vault content

### 4. Full Round-Trip Proven
The `transform__transformed__2026-03-04T00-52-58.html` file in the clone proves the complete cycle:

```
Safari (Vault UI)    →  Edit test-page.html, save to vault
Chrome (Workspace)   →  Load test-page.html, transform with Claude Sonnet 4.6
Chrome (Workspace)   →  Save transformed output back to vault
Terminal (CLI)       →  Clone vault, see both original and transformed files locally
```

Four operations, three clients, one vault, zero plaintext on the server.

### 5. Deterministic Key Fix Unlocked Everything
The single technical change — moving from 3-part keys (`passphrase:vaultId:settingsTransferId`) to 2-part keys (`passphrase:vaultId`) with deterministic key derivation via `SGVaultCrypto` — unlocked all three milestones on the same day:
- M-009: Cross-browser vault access
- M-010: CLI vault clone
- Both depend on the same key format, the same crypto library, the same derivation algorithm

## Architectural Significance

```
┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐
│  Safari (Vault)  │  │ Chrome (Workspace)│  │  Terminal (CLI)  │
│  Edit files      │  │ Transform w/ LLM │  │  Clone & pull    │
│  Browser crypto  │  │ Browser crypto   │  │  Node.js crypto  │
└────────┬─────────┘  └────────┬─────────┘  └────────┬─────────┘
         │                     │                      │
         │  Same 2-part key    │  Same 2-part key     │  Same 2-part key
         │  Same AES-256-GCM   │  Same AES-256-GCM    │  Same AES-256-GCM
         │                     │                      │
         ▼                     ▼                      ▼
┌─────────────────────────────────────────────────────────────────┐
│                    SG/Send Backend (Server)                     │
│                                                                 │
│  Stores only encrypted blobs. Serves them to any client.        │
│  Cannot read content. Cannot distinguish clients.               │
│  No browser affinity. No client registration. No sessions.      │
└─────────────────────────────────────────────────────────────────┘
```

The server is now truly **client-agnostic infrastructure**. Browser, terminal, CI pipeline — it doesn't matter. If you have the key, you can access the vault. If you don't, you can't. The server can't help either way.
