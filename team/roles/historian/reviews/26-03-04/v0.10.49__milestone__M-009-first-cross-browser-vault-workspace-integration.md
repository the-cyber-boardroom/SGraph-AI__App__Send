## The Event

On 4 March 2026, a user edited an HTML file (`test-page.html`) inside an encrypted personal vault in **Safari** at `dev.vault.sgraph.ai`, then opened the same vault in the **SG/Workspace** in **Chrome** at `dev.workspace.sgraph.ai` — where the file was decrypted, displayed in the source viewer, and transformed by Claude Sonnet 4.6 via OpenRouter. Both browsers accessed the same zero-knowledge encrypted vault simultaneously.

**The file:** `test-page.html` containing `<h1>SG/Send Test Page!!!!!</h1>`, edited in-place in the Vault UI's inline code editor, then loaded and analysed by an LLM in the Workspace — all without the server ever seeing the plaintext.

## What This Proves

### 1. Vault as Shared Encrypted Storage
The vault is not tied to a single application. Any SG/Send-aware client that has the vault key can open, read, and write to the same encrypted vault. The Vault UI and the Workspace are independent frontends sharing the same backend storage through the same encryption layer.

### 2. Cross-Browser Zero-Knowledge Access
Both Safari and Chrome independently derived the same AES keys from the same vault passphrase, independently decrypted the same encrypted blobs, and independently rendered the same content. The server served identical ciphertext to both — it has no idea what's inside.

### 3. Edit-Transform Loop
The full loop is working: edit a file in the Vault → load it in the Workspace → transform it with an LLM → (save the result back to the vault). This is the core document intelligence workflow that SG/Send's Workspace was designed to enable.

### 4. CDN-Deployed Vault Libraries
The Workspace loads vault libraries (`sg-vault.js`, `sg-vault-crypto.js`, `sg-send.js`, `sg-send-crypto.js`) from `static.sgraph.ai`. The successful cross-browser test confirms the v0.1.1 vault libraries are correctly deployed to the CDN and working with the new 2-part key format (`passphrase:vaultId`).

---

## The Bug That Nearly Blocked It

When the Workspace first attempted to open a vault, it failed with:

```
Error: Invalid vault key format. Expected passphrase:vaultId:settingsTransferId
```

**Root cause:** The CDN at `static.sgraph.ai` was serving the **old v0.1.0** `sg-vault.js` at the v0.1.1 path. The v0.1.0 code expected a 3-part vault key (`passphrase:vaultId:settingsTransferId`), but the v0.1.1 code uses a 2-part format (`passphrase:vaultId`) with deterministic key derivation via `SGVaultCrypto`.

**Fix:** Redeploying the CDN with the correct v0.1.1 `sg-vault.js` that uses `SGVaultCrypto.parseVaultKey()`.

**Lesson:** When multiple applications share libraries via CDN, CDN cache invalidation is a deployment-critical step.

## Architectural Significance

```
┌─────────────────────┐     ┌─────────────────────┐
│  Safari (Vault UI)  │     │  Chrome (Workspace)  │
│  dev.vault.sgraph.ai│     │  dev.workspace.      │
│                     │     │      sgraph.ai        │
│  Edit test-page.html│     │  Load test-page.html  │
│  Save → encrypt     │     │  Decrypt → source     │
│  → upload           │     │  viewer → LLM         │
└────────┬────────────┘     └────────┬──────────────┘
         │                           │
         │  Same vault key           │  Same vault key
         │  Same AES-256-GCM        │  Same AES-256-GCM
         │                           │
         ▼                           ▼
┌─────────────────────────────────────────────────┐
│           SG/Send Backend (Server)              │
│                                                 │
│  Stores only encrypted blobs. Cannot read them. │
│  Serves same ciphertext to both browsers.       │
│  No session coupling. No browser affinity.      │
└─────────────────────────────────────────────────┘
```

The server is stateless with respect to client identity. It doesn't know or care which browser is which. Both browsers independently derive keys, independently encrypt/decrypt, and independently render content. This is the vault-as-infrastructure pattern working exactly as designed.
