---

## Summary

Email-FS v0.6 is a file protocol, not a code module. Stage 1 (manual operation by agents/humans) requires **no implementation work** — the primitives are `cat`, `mv`, `rm`, and `sgit`. Dev effort only begins at Stage 2 (CLI). This review estimates that effort and identifies what's needed to get there.

---

## Stage 1: Manual Use — No Dev Work Required

Every primitive in the spec is executable today by any agent or human with sgit access:

| Primitive | Commands |
|-----------|---------|
| SEND | `cp eml mailroom/{recipient}/`, `cp eml outbox/{recipient}/`, `sgit commit`, `sgit push` |
| DELIVER | `mv mailroom/{me}/...eml inbox/...eml`, `mkdir state/{ULID}/`, write `core.json`, `sgit commit` |
| READ | `cat inbox/{ULID}...eml`, update `core.json` |
| PROCESS | `mv inbox/ processed/`, update `core.json`, `sgit commit` |
| PURGE | Write tombstone, `rm` eml + state, `sgit commit` |
| Session start | `mkdir sessions/active/{id}/`, write `start.json`, `sgit commit` |
| Session end | Write `end.json`, `sgit mv active/ archived/`, `sgit commit` |

**Dev decision needed:** Which team(s) adopt Email-FS in Stage 1 first? A pilot vault would surface rough edges before the CLI is built.

## Stage 2: `email-fs` CLI — Implementation Estimate

### What needs building

The CLI is "a thin wrapper over the file operations in §7." The spec defines the command set:

```
email-fs init
email-fs send --to {recipient} --subject "..." [--body-file | --interactive]
email-fs deliver
email-fs read {ULID}
email-fs process {ULID} [--note "..."]
email-fs purge {ULID} [--reason "..."]
email-fs sessions [--active] [--since DATE]
email-fs threads {ULID}
email-fs log [--by-subject] [--since DATE]
email-fs fsck
```

### Implementation streams

**Stream A — Core file operations (must-have)**

| Command | Effort | Notes |
|---------|--------|-------|
| `email-fs init` | 0.5d | Create `mail/` tree structure |
| `email-fs send` | 1d | ULID generation, RFC 2822 composition, 2-file write |
| `email-fs deliver` | 0.5d | `ls mailroom`, batch `mv` + sidecar creation |
| `email-fs read {ULID}` | 0.5d | `cat` + `core.json` update |
| `email-fs process {ULID}` | 0.5d | `mv` + `core.json` update |
| `email-fs purge {ULID}` | 0.5d | Tombstone write + `rm` |

**Subtotal Stream A:** ~3.5 dev-days

**Stream B — Inspection commands (high value)**

| Command | Effort | Notes |
|---------|--------|-------|
| `email-fs sessions` | 0.5d | `ls` three folders, filter by `start.json` |
| `email-fs log` | 1d | Walk processed + outbox, extract subjects, ULID sort |
| `email-fs threads {ULID}` | 1d | RFC 2822 `In-Reply-To`/`References` traversal |
| `email-fs fsck` | 1.5d | 18 conformance checks; most are `ls`/`grep` level |

**Subtotal Stream B:** ~4 dev-days

**Stream C — Conformance + edge cases**

| Concern | Effort | Notes |
|---------|--------|-------|
| GC-resistant mailroom query | 0.5d | `os.makedirs(exist_ok=True)` before `ls mailroom` |
| Missing `mail/` initialisation detection | 0.25d | Guard in every command |
| Batch DELIVER (N messages) | Included in deliver | |
| Self-addressed reasoning log shorthand | 0.5d | `email-fs log-self --subject "..."` |

**Total CLI estimate:** ~9 dev-days across all streams.

### Dependencies

- `sgit` CLI must be available in the agent's environment
- Python 3.12+ (matches Lambda runtime)
- No osbot-utils/Type_Safe needed — pure stdlib (`email`, `ulid`, `datetime`, `pathlib`)

### ULID generation

Python `ulid-py` (`pip install python-ulid`) or `ulid2`. No decision needed — either works.

## Stage 3: Vault Viewer — What the UI Team Needs

The spec defines what native vault-viewer rendering should provide:
- Per-agent inbox view
- Threaded conversation view (reconstruct from RFC 2822 headers)
- Session timeline view (walk `sessions/{active,archived,stale}/*/start.json` + events)
- Subject-line audit log (directory listing + subject extraction)

The `.eml` viewer already shipped in v0.3.2 (EXISTS, commit `95fc0cf2` + `e7618bbe`) renders individual emails with full email-client UI. Stage 3 extends this to the inbox/thread/session aggregate views.

**Estimate for Stage 3:** 5–8 dev-days in the vault UI (vault-browse-edit.js + new components). Not on the immediate schedule — dependent on Stage 1 producing real Email-FS vaults to test against.

## Risks and Flags

1. **No Email-FS tests in this repo** — Email-FS Stage 1 is manual operation in external vaults. The first test suite for the protocol will be in the `email-fs` CLI package, not here.

2. **SnapStart and x86_64 vs arm64** — Multiple cold-start commits flip between x86_64 and arm64. The final state (commit `eec33f7d`) explicitly sets `SEND__S3_BUCKET` to avoid STS calls during snapshots. The instability in this area is a risk indicator — ensure the deploy test suite runs a full cold-start + SnapStart cycle in CI before next Villager release.

3. **Vault UI v0.2.1 has no automated tests** — The landing page and browse route are tested manually only. Consider adding at minimum a Playwright smoke test that opens a vault key and navigates to the browse route.
