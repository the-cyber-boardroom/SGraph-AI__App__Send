### P1 — Send/Receive JS API (Doc 302)

**Status:** PROPOSED. Core composability work for the tools ecosystem.

#### Component breakdown

| Component | Effort | Notes |
|-----------|--------|-------|
| `SgSend.sendFile(file)` → token | 4h | Wraps existing upload flow in `sg-send.js` |
| `SgSend.sendText(text)` → token | 2h | Text → Blob → sendFile() |
| `SgSend.sendFolder(files)` → token | 8h | Multi-file, chunked; needs size-limit contract |
| `SgSend.receiveFile(token)` → file | 4h | Wraps existing download + decrypt flow |
| `SgSend.receiveText(token)` → text | 1h | receiveFile + text decode |
| `SgSend.receiveFolder(token)` → files | 8h | Multi-file download; parallel fetch orchestration |
| `<sg-send-drop>` Web Component | 4h | Drag-and-drop zone + token-created event |
| `<sg-send-receive>` Web Component | 3h | Token input field + file-received event |
| `<sg-send-panel>` combined component | 3h | Wraps both in a layout |
| Token history (localStorage) | 3h | List, copy, delete; sent/received distinction |
| Standard meta API (`getManifest` etc.) | 1h | Follow 08 Apr convention |
| Reference tool on tools.sgraph.ai | 4h | Deploy send+receive panes as a tool |
| **Total** | **~45h** | ~1 week for one dev |

**Namespace question (from Architect review):** Before writing code, decide: `SgSend` as a global (`window.SgSend`), versioned global (`window.SgSend_v1`), or ES module loaded from a versioned CDN path. The existing `sg-send.js` uses globals. Recommend: `window.SgSend` with a CDN path that includes the version in the file name (`sg-send-api-v1.js`). The version is in the file name, not the object name.

**Suggested stream split:**
- Stream A: Core API (`sendFile`, `receiveFile`) + Web Components → minimum viable for tool integrations
- Stream B: Folder support + token history → adds composability
- Stream C: Reference tool on tools.sgraph.ai → depends on Stream A

Ship Stream A first, get the video recorder and infographic generator using `SgSend.sendFile()`.

### P2 — Video Editing Suite Extension (Doc 303)

**Status:** PROPOSED. Builds on existing video editor.

#### Component breakdown

| Component | Effort | Notes |
|-----------|--------|-------|
| Video cropper: timeline scrubber + crop rectangle overlay | 12h | Canvas API, draggable rect |
| Video cropper: keyframe system | 8h | Set/edit/delete keyframes at timestamps |
| Video cropper: export (per-frame crop + rescale) | 16h | OffscreenCanvas + encode |
| Multi-track overlay: timeline track UI | 12h | 2-3 track layout, drag text elements |
| Multi-track overlay: real-time preview | 8h | Composited Canvas preview |
| Multi-track overlay: export (burn-in) | 12h | Per-frame overlay composition |
| Stream capture: Option A (screen recording + crop) | 2h | Already works via existing recorder; just document the workflow |
| Stream capture: Option B (captureStream()) | 8h | Research + prototype |
| SgSend.sendFile() integration for all tools | 2h | After Stream A of send/receive API ships |
| **Total** | **~80h** |

**Risk:** Video processing in the browser via Canvas/OffscreenCanvas is CPU-intensive and has known compatibility gaps (particularly OffscreenCanvas support in Safari). The crop export path needs a browser compatibility matrix before committing to client-side-only.

**Sequencing dependency:** Video tools need the send/receive API (send/receive Stream A) to be in place before the "save output" integration can ship.

**Recommend:** Start with the multi-track overlay (lower implementation risk) while the Canvas-heavy cropper is researched. Ship Option A of stream capture immediately (it's trivial — just a workflow note for existing screen recorder users).

### P2 — QA Three-Mode Abstraction (Doc 304)

**Status:** PROPOSED. This is QA-owned but Dev implements the Python abstraction library.

#### What to build

| Component | Effort | Notes |
|-----------|--------|-------|
| `PlaywrightClient` base class + mode dispatch | 8h | Python, mode-agnostic session API |
| Mode 1 adapter (local Docker via HTTP) | 6h | `docker run` + REST calls |
| Mode 2 adapter (Lambda invocation) | 12h | Session lifecycle in Lambda is non-trivial |
| Mode 3 adapter (EC2 REST calls) | 6h | Same REST API as Mode 1, different host |
| Hourly traffic generation script | 8h | Upload + download + vault create + screenshot |
| CloudWatch/cron trigger setup | DevOps | Not dev work |
| **Total (dev work)** | **~40h** |

**Blocking question (from Architect review):** Lambda session lifecycle. A "session" in Mode 2 cannot persist across Lambda invocations. The abstraction layer must either (a) make each Mode 2 invocation self-contained (navigate + action + screenshot in one call), or (b) maintain session state in an external store (DynamoDB, S3). Option (a) is simpler and matches the intended use case (one workflow = one Lambda invocation). Recommend Option (a).
