## Stream 1: Layered Container Architecture (Docs 290, 282, 289, 294)

**Priority:** High (foundational for Playwright service + all future containerised services)

### What to Build

Three containers, built in strict dependency order:

**Layer 1 — `sgraph/core-fastapi`**
- Python + FastAPI base
- API key authentication middleware
- Bootstrap/code-loader script (checks `CODE_URL` env var → downloads zip → extracts to `/app/code` → imports)
- Admin API endpoints: `/admin/code/upload`, `/admin/code/upload-zip`, `/admin/code/set-url`, `/admin/code/status`, `/admin/code/reload`
- Health endpoint: `GET /health`
- Dockerfile, docker-compose for local dev
- Unit tests: code loading from each source, admin endpoint auth, health check

**Layer 2 — `sgraph/core-fastapi-aws`**
- Inherits Layer 1
- Adds osbot-aws dependency
- S3 code loading mode: `CODE_S3_BUCKET` + `CODE_S3_KEY` env vars → boto3 S3 download → extract → execute
- Unit tests: S3 code loading (LocalStack)

**Layer 3 — `sgraph/playwright-service`**
- FROM `mcr.microsoft.com/playwright/python:1.X.X-jammy` (pinned version — confirm with DevOps)
- COPY Layer 1/2 code onto Playwright base
- Playwright FastAPI routes:
  - `POST /session/start` → returns `session_id`
  - `POST /session/{id}/navigate` `{url}`
  - `POST /session/{id}/click` `{selector}`
  - `POST /session/{id}/fill` `{selector, value}`
  - `POST /session/{id}/screenshot` → PNG (or vault path)
  - `POST /session/{id}/video/start` / `/stop`
  - `POST /session/{id}/evaluate` `{script}`
  - `POST /session/{id}/close`
  - `POST /sequence/execute` `{config, steps[]}`
- Vault output integration: screenshot/video/HAR → write to configured output vault via `sgit` push
- Unit tests: each endpoint, sequence execution, vault output

### Effort Estimate

| Component | Estimate |
|-----------|----------|
| Layer 1 (bootstrap + admin API + CI) | 3–5 days |
| Layer 2 (S3 loading + CI) | 1–2 days |
| Layer 3 (Playwright routes + vault output + CI) | 4–6 days |
| Docker Hub publishing automation | 1 day |
| **Total** | **9–14 days** |

### Existing Patterns to Reuse

- osbot-fast-api (`Serverless__Fast_API`) — use as the FastAPI base class for all three layers
- osbot-aws S3 methods — use for S3 code loading in Layer 2
- `sg-send.js` vault write methods — adapt for server-side vault output in Layer 3 (or use sgit CLI inside the container)

### Risks

- **Layer 3 image size:** Playwright + Chrome images are typically 1–2 GB. Lambda has a 10 GB container image limit (sufficient) but cold starts will be slow (~30–60s) for Lambda deployment. Document this trade-off; Fargate is the preferred path for performance-sensitive Playwright work.
- **Admin API security:** The code upload endpoints are remote code execution by design. Ensure API key validation is applied before any admin route is accessible. Test that unauthenticated requests to admin endpoints return 401, not 200.

## Stream 2: Auth MVP — Social Login + Vault Key Storage (Doc 291)

**Priority:** High (unblocks profile page [OD #1], healthcare production, video recorder)

### What to Build

Four Web Components, in dependency order:

**`<sg-login>` — Social login**
- Google OAuth flow (PKCE, client-side only, no backend callback)
- GitHub OAuth flow (PKCE or token exchange — confirm with Architect: GitHub OAuth PKCE support)
- Cognito hosted UI as first provider (simplest to wire up)
- Emits `sg-login-success` event with session token and user info
- Handles session persistence (localStorage token refresh)
- Clear documentation: how to add to any page in one line

**`<sg-user-data>` — User attribute storage**
- Read/write Cognito custom user attributes (vault key list, preferences)
- Encrypt vault keys client-side before writing to attributes (ask AppSec for derivation input — see Architect review decision D-3/D-4)
- Emits `sg-user-data-ready` event with loaded data
- Size constraint research: max Cognito custom attribute size (Cognito limit is 2048 chars per attribute)

**`<sg-vault-manager>` — Vault list UI**
- Lists user's stored vaults (from `<sg-user-data>` loaded keys)
- Load / open / create vault actions
- Can be embedded in profile page

**`<sg-admin-panel>` — Admin view**
- User count and registration stats (Cognito Admin SDK, client-side call requires appropriate IAM — needs Architect input on how to call Cognito Admin API safely from browser)
- Protected by Cognito user group membership (`sg-admin` group)

**Firebase equivalent (phase 2):** After Cognito components are working, add Firebase Auth provider. Components should be provider-agnostic internally — only the OAuth flow differs.

### Effort Estimate

| Component | Estimate |
|-----------|----------|
| `<sg-login>` (Cognito + Google OAuth) | 3–4 days |
| `<sg-login>` GitHub OAuth | 1–2 days |
| `<sg-user-data>` | 2–3 days |
| `<sg-vault-manager>` | 2 days |
| `<sg-admin-panel>` | 2 days |
| Firebase equivalents | 2–3 days |
| **Total** | **12–16 days** |

### Dependency

Auth MVP requires AppSec sign-off on vault key encryption approach (Option A vs B, derivation input) before `<sg-user-data>` can be finalised. `<sg-login>` can be built first while that decision is pending.

### Key Rule: No Server-Side Code

Every OAuth callback, token refresh, and attribute read/write must happen entirely in the browser. No API routes added to the admin Lambda or user Lambda for auth. Violation of this rule breaks the zero-knowledge architecture. CI should fail if any server-side auth endpoint is added.

## Stream 3: Vault Upload Beta in Main SG/Send UI (Doc 281)

**Priority:** High (first user-visible feature from vault convergence, ready for beta)

### What to Build

A beta toggle in the main SG/Send upload UI that switches between the existing folder-zip mode and the new vault-backed mode.

**Vault-backed upload flow:**
1. User drops a folder → `sgit init`-equivalent creates vault immediately → share link returned and displayed
2. Upload progress: files queued → batched into groups of 20 (or configurable threshold) → each batch committed to vault
3. Upload status file: `.sg-upload-status.json` written and committed each batch → recipient UI reads and displays progress banner
4. Share link copyable immediately; recipient opens vault and sees files appearing

**UI elements:**
- Beta toggle: checkbox/button in upload area with "New: try folders with version control (beta)" label
- Instant share link: appears within 1 second of folder drop, styled clearly ("Your share link is ready — files uploading in background")
- Progress banner: persistent banner showing "Upload in progress: X of Y files uploaded"
- Tab-close warning: `beforeunload` event listener while upload in progress
- Recipient-side: `.sg-upload-status.json` detected on vault open → renders progress banner (not shown after upload completes)

### What NOT to build in beta

- Placeholder files convention — defer to beta v2
- Service worker for background upload — defer to v2
- Path 2 (user opens vault directly and adds files) — already technically possible; no new code needed

### Effort Estimate

| Component | Estimate |
|-----------|----------|
| Vault creation on folder drop + instant share link | 2–3 days |
| Batched commit loop | 2 days |
| Upload status file convention + writes | 1 day |
| Progress banner (sender + recipient UI) | 1–2 days |
| Tab-close warning + beta toggle | 1 day |
| **Total** | **7–9 days** |

### Dependencies

- **Vault creation API (browser-side):** Requires the "instant vault creation on upload" feature from Section 27 (PROPOSED). This is the first dependency to unblock. Check if vault creation via the vault Web Components is currently possible client-side, or if a new API endpoint is needed.
- **Open Decision #6 (upload branch vs direct commit):** Must be resolved before starting. If upload branch is required (safer), add 2–3 days for branch management logic.

- Health questionnaire Web Component: 15-20 question form (text, multiple choice, scale inputs)
- Vault creation on form start
- Section-by-section commits (each section = one commit, commit message = section name + timestamp)
- Results summary: client-side calculation + display

**Phase 2 — LLM analysis:**
- Anonymised/structured data construction from questionnaire JSON
- OpenRouter API call (model selector, cost display)
- LLM response stored as `llm-analysis.json` in vault (clearly labelled as AI-generated, pending review)

- Vault key sharing flow (patient shows vault key; doctor enters it in their browser)

- Patient creates share token from vault
