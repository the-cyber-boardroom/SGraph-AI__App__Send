## 1. Vault Additions (doc 278)

All four additions build directly on the existing vault architecture. Foundation: SGit dual branch model (Phase 2 Push + Phase 3 Pull), tree-only loading, immutable block cache, browse-edit Phase 1 — all EXISTS.

### Addition 1: Instant Vault Creation on Upload

**What:** Upload flow gains a "Create as Vault" option. Files are committed to the vault incrementally during upload (one file = one commit), rather than bundled as a zip first.

**Estimate:** 2–3 days

**Key implementation points:**
- The upload UI currently funnels to `POST /transfers/create` (zip bundle). The new path needs a vault creation API call first (`sgit init`-equivalent server call), then per-file commit calls.
- A progress indicator showing vault tree growing in real-time would complete this feature nicely, but is not an acceptance criterion.
- Risk: file ordering. Vault commits are linear; if 20 files are uploaded in parallel, commit ordering is non-deterministic. Either commit serially or accept non-deterministic order.

**Acceptance criterion coverage:** Criterion 1 (option during upload), Criterion 2 (individual commits visible in vault log).

### Addition 2: Upload Branches

**What:** Large uploads (many files) go to a dedicated upload branch. Once all files are committed, the upload branch fast-forwards into the main branch.

**Estimate:** 3–4 days

**Key implementation points:**
- Branch creation: the vault ref model (`sg-vault-ref-manager.js`) already handles branch refs. An upload branch needs a well-defined lifecycle: create on upload start → commit files → merge to main → delete.
- The Architect recommends fast-forward merge to avoid merge commits. This means the upload branch must be based on the current main HEAD at start time — any concurrent changes to main during upload would cause a conflict at merge time. Consider a lock during the final merge step.
- The brief's acceptance criterion specifies "Upload 50+ files" — performance testing needed.

**Acceptance criterion coverage:** Criterion 3 (upload branch exists, merges to main on completion).

### Addition 3: Read-Only Access Control Cleanup

**What:** Opening a vault via share token shows no edit/upload/push controls. Write access gated by vault key.

**Estimate:** 1 day

**Key implementation points:**
- The 04/11 audit already fixed the `vault._writeKey` guard to avoid spurious 401 PUT errors on read-only opens (commit `086d00c`, EXISTS). This addition extends that fix to the UI layer: when `dataSource.writable === false`, the Edit button, Upload Files button, New Folder button, Push button, and any other write controls should be hidden (not just disabled).
- `vault-browse-edit.js` is the primary file to modify.

**Acceptance criterion coverage:** Criterion 4 (no edit/upload/push controls in share token view), Criterion 5 (full write access via vault key).

### Addition 4: Auto-Refresh / Change Detection

**What:** A periodic background check calls `getBehindCount()`. When remote changes are detected (count > 0), the Pull badge updates without user interaction.

**Estimate:** 0.5–1 day

**Key implementation points:**
- `getBehindCount()` EXISTS (`sg-vault.js`). This addition is a timer that calls it.
- Implementation: `setInterval(() => vault._refreshBehindCount(), POLL_INTERVAL_MS)` in `vault-shell.js`. Pause when `document.visibilityState === 'hidden'`.
- Poll interval: 15–30 seconds is reasonable. Don't make it configurable yet — just pick a sensible default.

**Acceptance criterion coverage:** Criterion 6 (Push from one session → Pull badge appears in other).

| Addition | Estimate | Risk | Priority |
|----------|----------|------|----------|
| 1. Instant vault creation on upload | 2–3 days | Medium (API flow change) | High |
| 2. Upload branches | 3–4 days | Medium (branch lifecycle) | Medium |

## 2. PlaybookLM Companion Tools (doc 277)

All three tools are standalone pages on `tools.sgraph.ai`. They reuse existing components but require new page-level orchestration.

### Dependency Check

Both Tool 1 (Infographic Generator Simple Mode) and Tool 3 (Simple PlaybookLM) require the **PlaybookLM pipeline** to be built. The pipeline is PROPOSED (from docs 252–263). Before scheduling Tools 1 and 3, confirm with Conductor whether the pipeline is in-progress.

**Tool 2 (Infographic Editor) can be built independently** — it works with existing infographic generation infrastructure.

### Tool 1: Infographic Generator (Simple Mode)

**What:** Drop PDF or text → model select → bundled infographic artefact → vault save.

**Estimate:** 2–3 days (after PlaybookLM pipeline exists)

**Key implementation points:**
- Input handling: PDF → text extraction (via pdf.js) → LLM prompt; Text → direct LLM prompt
- The `sg-llm-infographic` component (EXISTS at tools.sgraph.ai) handles SVG generation from prompt; this tool adds the input-processing wrapper and vault bundling
- Vault save: call `sgit commit` equivalent via the vault API with the generated artefact

**Risk:** Blocked on PlaybookLM pipeline. Cannot estimate more accurately without knowing pipeline API.

### Tool 2: Infographic Editor

**What:** Drop image → LLM analysis → user selects changes → LLM generates edited image → LLM verification → cost breakdown.

**Estimate:** 4–6 days (independent of PlaybookLM pipeline)

**Key implementation points:**

State machine (5 states):
```
IDLE → ANALYSING → AWAITING_SELECTION → EXECUTING_EDIT → VERIFYING → DONE
```

- **Analysis:** Send image to multimodal LLM (Gemini via OpenRouter), prompt for structured analysis output (what to change, text to update, issues)
- **Change execution paths (3 options per brief):**
  - Full LLM image regeneration: send modified prompt to image-gen model
  - Browser-side tool execution: requires architecture design (see Architect note — this is a frontier capability)
  - Hybrid: LLM identifies, browser executes mechanical part, LLM verifies
  - For MVP, implement path 1 (full LLM regeneration) only. Paths 2 and 3 are follow-up sessions.
- **Verification:** Send old image + new image to multimodal LLM, prompt: "Did this change apply correctly without unintended side effects?"
- **Cost tracking:** `sg-llm-events` streams should include cost metadata per request; accumulate and display total
- **PDF image extraction:** `pdf.js` renders page to Canvas, extract as PNG/JPEG; user selects which extracted image to edit

**Risk:** The LLM-triggered browser tool execution path (brief option 2) needs an arch brief before building. For MVP, skip it and implement path 1 only.

### Tool 3: Simple PlaybookLM

**What:** Upload sources + short prompt → full PlaybookLM pipeline → assembled PDF output with stage-by-stage progress.

**Estimate:** 3–4 days (after PlaybookLM pipeline exists)

**Key implementation points:**
- The "switch to advanced mode" requirement means all intermediate artefacts must be persisted and addressable. The tool cannot discard intermediate outputs.
- Stage indicators map 1:1 to pipeline stages (source analysis → brief generation → per-slide briefs → infographic generation → assembly)
- The existing PlaybookLM UI (PROPOSED, not built yet) is what "advanced mode" reveals

**Risk:** Blocked on PlaybookLM pipeline. This tool is effectively a thin shell over the pipeline.

### Shared Architecture: Standard JS API

The brief requires all three tools to expose `getManifest`, `getMethods`, `getSkills`. The Architect flags a concern: the existing infographic tool uses `__tool.generate()`, `__tool.connect()`, etc. Dev should not implement the new API until the Architect produces an API contract that reconciles the two.

### PlaybookLM Tools Summary

| Tool | Estimate | Blocked On | Priority |
|------|----------|-----------|----------|
| Tool 2: Infographic Editor | 4–6 days | Nothing (can start now) | High |
| Tool 1: Infographic Generator (Simple) | 2–3 days | PlaybookLM pipeline | Medium |
| Tool 3: Simple PlaybookLM | 3–4 days | PlaybookLM pipeline | Low (depends on pipeline) |

**Recommended starting point:** Tool 2. It is independent, architecturally novel, and directly reuses the existing infographic generation infrastructure.
