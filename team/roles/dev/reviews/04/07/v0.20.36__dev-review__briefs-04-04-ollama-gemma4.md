## Implementation Impact

### Stream 1: `_buildOllamaBody` — Audio/Video Guard (Doc 234)

**Priority: High — small change, prevents a subtle bug**

The research finding is clear: Ollama has no audio/video input path. Any code that builds Ollama request bodies must handle audio/video content parts explicitly.

**What to implement:**
```javascript
function _buildOllamaBody(model, messages, options = {}) {
  // Guard: detect audio/video content parts
  for (const msg of messages) {
    if (Array.isArray(msg.content)) {
      for (const part of msg.content) {
        if (part.type === 'audio_url' || part.type === 'input_audio' ||
            part.type === 'video_url') {
          // TODO: when ollama/ollama#11798 merges, add audio field here
          throw new ProviderCapabilityError(
            'Ollama does not support audio/video input. ' +
            'Switch to mlx-openai-server or OpenAI API for multimodal requests.'
          );
        }
      }
    }
  }
  // ... existing body construction
}
```

**Effort:** < 1 hour (add guard + test case)
**Risk:** Low — purely additive, no behaviour change for text/image requests

## Dev Setup Notes (From Doc 232)

For any team member who needs to run local Gemma 4 inference:

```bash
# One-time setup
python3.11 -m venv ~/gemma4-env
source ~/gemma4-env/bin/activate
pip install mlx-openai-server huggingface_hub hf_transfer

# Download models (do this while online)
export HF_HUB_ENABLE_HF_TRANSFER=1
huggingface-cli download unsloth/gemma-4-E4B-it-UD-MLX-4bit
huggingface-cli download unsloth/gemma-4-E2B-it-UD-MLX-4bit

# Run (can be fully offline after download)
export HF_HUB_OFFLINE=1
mlx-openai-server launch \
  --model-path unsloth/gemma-4-E4B-it-UD-MLX-4bit \
  --model-type multimodal
# API available at http://localhost:8000/v1
```

For CORS issues (browser app at different port):

```bash
# Use mlx_vlm.server instead — has built-in CORS
pip install mlx-vlm
mlx_vlm.server --model mlx-community/gemma-4-E4B-it-4bit --port 8080
```

## Risks and Flags

**Risk 1: mlx-openai-server API stability**
`mlx-openai-server` is a community project (not Google-official). If its API changes, the MLX provider config needs updating. Mitigation: pin the version in requirements; add an integration smoke test.

**Risk 2: Audio capability depends on `--model-type multimodal`**
The MLX provider's audio capability is only active when the server is started with `--model-type multimodal`. If a user runs the server without this flag, audio requests will fail with an unhelpful error. The provider config or setup docs should make this explicit.

**Risk 3: Thinking tokens not stripped by default**
Gemma 4 with thinking enabled will return `<think>...</think>` blocks. If the response parser doesn't handle these, they'll appear in the UI as raw text. This is a Gemma 4-specific concern that doesn't affect GPT-4o or standard Ollama models. The thinking token strip policy (from doc 224) must be implemented in the MLX provider's response handler.
