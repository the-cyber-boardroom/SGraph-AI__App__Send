## Implementation Streams

### Stream A: Browser VFS (Infrastructure — Do First)

**Priority: Highest. Blocks everything else.**

From doc 221 (`browser-virtual-file-system`).

**Phase A1 — Memory provider (1–2 days)**
- Define the VFS interface: `createFolder`, `writeFile`, `readFile`, `listFolder`, `copyFile`, `deleteFile`, `rename`, `stat`
- Implement `memory` provider (JavaScript objects, no persistence)
- Write unit tests using memory provider — zero external dependencies required
- **Acceptance test:** A component can write and read files via `fs.writeFile()` / `fs.readFile()` without knowing it is using the memory provider

**Phase A2 — IndexedDB provider (2–3 days)**
- Implement `indexeddb` provider (survives page refresh, offline-capable)
- Handle binary data (PNG, PDF) correctly — IndexedDB stores ArrayBuffer natively
- Write tests using IndexedDB provider
- **Acceptance test:** Write a file, refresh the page, read the file back successfully

**Phase A3 — Layered provider stack (1–2 days)**
- Implement the memory → IndexedDB → vault write-through/read-through stack
- Configuration: `{ providers: ['memory', 'indexeddb', 'vault'] }`
- **Acceptance test:** Write goes to memory immediately, is persisted to IndexedDB before page unload, is eventually committed to vault on batch commit

**Phase A4 — Component manifest enforcement (1 day)**
- Parse the `storage` manifest from component declarations
- Enforce path scoping: component at `/openrouter/keys/` cannot write to `/infographic/`
- **Acceptance test:** A component that tries to write outside its declared paths gets a clear error

**Phase A5 — Vault provider (3–4 days)**
- Wrap the existing vault API (deterministic ID access, read/write/delete endpoints) in the VFS interface
- Support the `remote`, `cached`, and `full-offline` access modes
- **Acceptance test:** A component writes a file, it appears as a vault object accessible via the deterministic ID

### Stream B: SGit Browser Primitives (Infrastructure — Parallel with A after A1)

**Priority: High. Feeds into vault provider for Stream A.**

From doc 220 (`sgit-browser-components`).

**Phase 1 — Primitive components (5–7 days)**
Build each component as a standalone Web Component with full visibility (Phase 1 / "feature-rich/debuggable"):
- `sg-vault-init` — create new vault
- `sg-vault-status` — current branch, modified files, remote sync status
- `sg-vault-commit` — stage + commit (file list with checkboxes, commit message)
- `sg-vault-push` / `sg-vault-pull` — sync with remote
- `sg-vault-log` — commit history list
- `sg-vault-branch` — create/switch/list branches
- `sg-vault-browse` — file tree + preview (reuse existing Browse module)
- `sg-vault-file-drop` — the key UX primitive (drop → stage → commit → push)
- `sg-vault-events` — event viewer (real-time log of all vault events)

Each component emits standardised events (see doc 220 for event schema). The event viewer is essential for debugging; build it alongside the first 3 primitives.

**Priority within Phase 1:**
1. `sg-vault-file-drop` (most important UX primitive)
2. `sg-vault-events` (essential for debugging others)
3. `sg-vault-status` (needed to understand state)
4. `sg-vault-commit`, `sg-vault-push`, `sg-vault-pull`
5. `sg-vault-log`, `sg-vault-branch`, `sg-vault-browse`, `sg-vault-init`

**Ephemeral vaults:**
- Create vault in browser (IndexedDB only, no server)
- Export as zip when done
- This is the `indexeddb` VFS provider + a zip export function

## Risks

| Risk | Severity | Mitigation |
|------|----------|------------|
| OpenRouter generation history contains prompt content | High | Research API before building vault storage for it |
| OpenRouter API does not support credit limit updates on existing keys | Medium | Design new-key-on-topup flow as fallback |
| VFS layered stack complexity causes data inconsistency | High | Test write-through and read-through exhaustively with indexeddb provider before adding vault |
| `sg-vault-file-drop` UX too invisible — user does not know their data is persisted | Low | Add visible "saved" / "syncing" / "synced" indicator |
| Library website migration breaks existing SG/Send sessions mid-migration | Medium | Copy-first approach; do not delete from SG/Send until new sessions confirmed working |
| Ephemeral vault isolation — two tools sharing IndexedDB key corrupt each other | High | Implement path scoping from day one; enforce component manifests |
