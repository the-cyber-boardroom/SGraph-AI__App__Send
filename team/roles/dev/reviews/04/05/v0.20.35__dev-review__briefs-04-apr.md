## Stream 1: Backend Storage Restructuring

**Source:** Doc 223 â€” `v0.20.33__dev-brief__backend-storage-restructuring.md`

### What Needs to Be Built

1. **Group ID derivation functions** â€” four functions, each with a type-specific salt:
   ```python
   derive_server_id(region, provider)
   derive_user_id(identity)
   derive_access_id(access_token)
   derive_key_id(decryption_key)
   ```
   See Architect review for the SHA-256 vs PBKDF2 decision that must be made first.

2. **Upload path construction** â€” assemble the full `by-id-from__*` path from provided + defaulted levels

3. **Download path resolution** â€” given a transfer ID, resolve to full path (check grouped structure first, fall back to flat for legacy)

4. **Server-side defaults** â€” if caller does not provide grouping context, server fills:
   - Server/region: from deployment config
   - User ID: derived from access token
   - Access ID: derived from access token (always available in upload requests)
   - Key ID: derived from decryption key (always available in upload requests)

5. **Migration script** â€” for existing flat-structure transfers

### Acceptance Criteria (from brief)

| # | Criterion |
|---|-----------|
| 1 | Group ID derivation function implemented (4 types: user, access-token, decryption-key, public-key) |
| 2 | All new transfers stored under `/transfers/` root |
| 3 | Type prefix folders used (`by-id-from__user`, `by-id-from__access-token`, etc.) |
| 4 | Server calculates defaults when caller doesn't provide grouping IDs |
| 5 | Server/provider/region ID as outermost grouping level (configured per deployment) |
| 6 | User/customer/PKI identity grouping supported |
| 7 | API supports both flat (legacy) and grouped paths during migration |
| 8 | Migration script for existing transfers |
| 9 | Per-user cleanup: delete all transfers for a user by removing their group folder |
| 10 | No client-side changes required |

### Implementation Notes

- The brief uses `sha256(f"{salt}:{key_material}")[:16]` for derivation. The Architect review flags a potential conflict with the PBKDF2 pattern. Get this decision before starting.
- The "check grouped first, fall back to flat" dual-path lookup must be efficient â€” don't S3-scan the flat structure on every request. The transfer ID itself can be used as a direct lookup key in both structures.
- Migration script needs idempotency. If it crashes midway and is re-run, it must not corrupt transferred data.

## Stream 2: Agentic LLM Component Suite

**Source:** Doc 224 â€” `v0.20.33__dev-brief__agentic-llm-tool-calling.md`

### What Needs to Be Built

Six Web Components plus supporting infrastructure:

#### Phase 1: Foundation (unblock everything else)

**1. `sg-tool-definition`**
- Visual editor: form fields for name, description, parameters (type, description, required)
- JSON schema validation before use
- Template library: pre-built definitions for common tools
- Import/export: load from file or VFS

**2. `sg-json-sender`**
- JSON editor with syntax highlighting and validation
- Schema-aware input mode (form fields when schema known)
- Preview: show full payload before sending (system prompt + tools + history + message)
- Token count display

**3. `sg-json-receiver`**
- Auto-detect response type: text, tool_call, or structured JSON
- Tool call display: name + arguments + "Run" button
- JSON tree viewer (collapsible, searchable)
- Schema validation of responses

#### Phase 2: Execution Layer

**4. `sg-tool-runner`**
- Tool registration API: `toolRunner.register(name, async (args) => result)`
- Execute on tool_call, return JSON result to loop
- 8 built-in tools (listed below)
- Error handling: tool throws â†’ LLM receives structured error response

**Built-in tools (Phase 2):**
| Tool | Depends On |
|------|-----------|
| `search_vault` | VFS list + pattern match |
| `read_file` | VFS read |
| `write_file` | VFS write |
| `list_folder` | VFS list |
| `create_infographic` | Infographic engine (PROPOSED) |
| `analyse_document` | sg-llm-request (PROPOSED) |
| `search_web` | Fetch API |
| `run_code` | sg-sandbox |

**Note:** `search_vault`, `read_file`, `write_file`, `list_folder` depend on VFS, which is itself PROPOSED. Start with a memory VFS stub for testing. `create_infographic` and `analyse_document` depend on other PROPOSED components â€” implement as stubs initially.

**5. `sg-sandbox`**
- **JavaScript:** sandboxed iframe + Web Worker
  - `sandbox` attribute set (no same-origin, no parent DOM, no top navigation)
  - Communication via postMessage only (validate origin strictly)
  - Timeout enforcement (kill after N seconds)
  - Memory limit (terminate if exceeded)
- **Python:** Pyodide in Web Worker (WASM)
  - Pyodide loaded and cached via `sg-wasm` component (PROPOSED, use simple download fallback initially)
- Capability model: `compute` + `read_input` + `return_output` by default; `import_libs`, `network`, `filesystem` opt-in
- See Architect review: `import_libs` must only allow locally bundled libraries, not CDN fetches from inside the sandbox

#### Phase 3: Orchestration

**6. `sg-agentic-loop`**
- Assemble prompt: system prompt + tool definitions + context + user task
- Send to LLM (uses `sg-llm-request` from sg-llm component family â€” PROPOSED)
- Parse response: text or tool_call
- Execute tool via `sg-tool-runner` â†’ capture result â†’ send back to LLM
- Repeat until text response or max iterations
- Trace visualisation: clickable timeline of every step

**Safety controls:**
- Max iterations (default: 10, configurable)
- Cost budget (stop if estimated tokens exceed threshold)
- Human-in-the-loop gate (optional approval before each tool execution)
- Full audit log saved to VFS

**Gemma 4 thinking token handling** (see Architect review):
- Within a multi-step tool call sequence: preserve thinking tokens
- Between conversation turns: strip thinking tokens
- Must be configurable per model (not all models have thinking tokens)

## Stream 3: Deterministic-Safe-Paths Pattern Formalisation

**Source:** Doc 225 â€” `v0.20.33__pattern__deterministic-safe-paths.md`

### What Needs to Be Built

1. **Shared derivation function** â€” single canonical implementation:
   ```python
   def derive_deterministic_id(
       secret: str, salt: str,
       iterations: int = 600_000, length: int = 16
   ) -> str
   ```

2. **Salt registry** â€” a constants file (not just documentation)
   - See Architect review for recommended structure

3. **Migrate existing uses** â€” SGit vault object ID derivation and backend storage grouping should both be updated to use the shared function

4. **At least one new feature using the pattern** (per acceptance criteria) â€” per-user settings is the most useful first implementation

### Per-User Settings Implementation (Proposed First Use)

Path pattern: `settings/{derive_id(access_token, "sg-settings")}/preferences.enc`

- Client calculates the path from the access token
- Fetch the settings file, decrypt locally
- Server stores encrypted preferences without knowing whose they are
- No user account, no login, no session

This is a small, well-defined first implementation that exercises the pattern end-to-end.

| 5 | At least one new feature implemented using this pattern |
| 6 | Security review confirming the pattern meets zero-knowledge requirements |

## Technical Reference Notes (for implementation)

### Gemma 4 Integration (Doc 226)

When implementing `sg-agentic-loop` and `sg-json-sender` for local Gemma 4:
- Use `mlx-openai-server` at `localhost:8000/v1` â€” same request format as OpenAI
- Only changes needed: base URL and model name (no new serialization logic)
- Tool calling uses the OpenAI `tools` parameter format (not raw special tokens)
- CORS: server may need `--host 0.0.0.0` if browser app runs on different port
- Recommended generation params: `temperature: 1.0`, `top_p: 0.95`, `top_k: 64`

### LiteRT-LM (Doc 227)

No implementation changes needed today. Watch for:
- Gemma 4 E4B in `.litertlm` format (not yet available)
- Python SDK stability (day-1 software as of 04/02)
- Longer context windows (currently 4096-capped)
