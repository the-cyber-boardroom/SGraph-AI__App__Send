**Version:** v0.22.18 | **Date:** 28 April 2026 | **Role:** Dev (Librarian session)

---

## Overview

The architect review identifies a significant gap: `structure_key` is derived but never used. All vault objects use `read_key` for encryption, making structural read-only access impossible despite the key derivation infrastructure being in place.

This review focuses on the implementation work required **in this repo** — the Vault Web Team items in `sgraph_ai_app_send__ui__vault`. CLI team work (in `SGit-AI__CLI`) is a separate scope.

## 2. Implementation Streams

### Stream A: Crypto Layer (Foundation) — Prerequisite for all other streams

**Files:** `sg-vault-crypto.js`

**Tasks:**
1. Add `structureKey` derivation to `deriveKeys()`:
   - Use `crypto.subtle.deriveBits()` with HKDF-SHA256 from the raw read key bits
   - Import as non-extractable AES-GCM key
   - **Critical:** derive from raw PBKDF2/HKDF output bytes **before** `importKey()` wraps them as non-extractable
2. Add `decryptStructural(structureKey, readKey, data)` helper:
   - Try `structureKey` first
   - Catch `DOMException { name: 'OperationError' }` only
   - Fall back to `readKey`

**Estimated effort:** 2–3 hours. **Risk:** Low — pure additive change, no existing code removed.

### Stream B: Vault State and Propagation — Depends on Stream A

**Files:** `sg-vault.js`

**Tasks:**
1. Store `_structureKey` alongside `_readKey` and `_writeKey`
2. Derive `_structureKey` during vault open/unlock (call `SGVaultCrypto.deriveKeys()` which now returns `structureKey`)
3. Pass `_structureKey` through to all ref manager, commit manager, and object store calls that handle structural objects
4. Ensure simple token path (`word-word-NNNN`) also derives `structureKey` — the architect review shows both paths derive it

**Estimated effort:** 3–5 hours depending on how many call sites. **Risk:** Medium — propagating a new parameter through many call sites risks missing some.

### Stream C: Structural Object Writes — Depends on Stream B

**Files:** `sg-vault-ref-manager.js`, `sg-vault.js` (commit creation), tree storage code

**Tasks:**
1. Ref writes: encrypt with `structureKey`
2. Branch index writes: encrypt with `structureKey`
3. Commit outer envelope: encrypt with `structureKey`; `message_enc` field stays `readKey`
4. Tree outer envelope: encrypt with `structureKey`; entry metadata fields stay `readKey`
5. Public PKI key writes: encrypt with `structureKey`

**Estimated effort:** 4–6 hours. **Risk:** Medium-High — changing the write path affects all new vault objects; old vaults remain backward compatible via fallback.

### Stream D: Structural Object Reads — Depends on Stream A

**Files:** `sg-vault-ref-manager.js`, `sg-vault-object-store.js`

**Tasks:**
1. Ref reads: use `decryptStructural(structureKey, readKey, ...)`
2. Branch index reads: use `decryptStructural`
3. Commit loads: use `decryptStructural` for outer envelope; `readKey` for `message_enc`
4. Tree loads: use `decryptStructural` for outer envelope; `readKey` for `name_enc` etc.
5. Public PKI key reads: use `decryptStructural`

**Estimated effort:** 3–5 hours. **Risk:** Medium — safe to implement with the fallback; regressions immediately visible in vault load failures.

### Stream E: Testing — Depends on Streams A–D

**Tasks:**
1. Old vault fixture test: create vault with all objects encrypted with `readKey`; verify loads still work via fallback
2. New vault fixture test: create vault with all objects encrypted with `structureKey`; verify loads succeed directly
3. Mixed vault fixture test: create vault with first N commits on `readKey`, next M on `structureKey`; verify both read correctly without state contamination
4. Structure-key-only access test: open vault with only `structureKey` (no `readKey`); verify can traverse commit graph; verify cannot decrypt `name_enc` or `message_enc`
5. Regression: all existing vault tests pass unchanged

**Estimated effort:** 4–6 hours. **Risk:** Medium — mixed vault fixture is the hardest to construct and most important to get right.

### Stream F: `openWithStructureKey` API Surface — Optional, after Streams A–E

**Tasks:**
1. Add `SgVault.openWithStructureKey(structureKeyHex, vaultId)` factory method
2. Implements a read-only vault consumer that holds only `structureKey`
3. Throws on any attempt to read sensitive metadata (filenames, content) or write

**Estimated effort:** 3–4 hours. **Risk:** Low — additive new API, existing vault open path unchanged.

## 4. Key Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Missing call sites when propagating `structureKey` | Medium | High — silent failure: old vault object written where new expected | Grep for all `readKey` / `_readKey` usages before starting |
| Web Crypto `OperationError` swallows non-authentication errors | Low | Medium — genuine errors silently caught | Strict `instanceof DOMException` check as specified in Architect review |
| Simple token path misses `structureKey` derivation | Medium | High — simple token vaults not upgraded | Test both derivation paths explicitly in Stream E |
| `structureKey` derived from wrong point in key schedule | Low | Critical — wrong key derived, all objects fail | Derive from raw bits **before** `importKey()` — as specified in brief |
| CLI team ships `structure_key` objects before Web team reads fallback | Low | Medium — Web team fails to read new objects | Coordinate release sequence; fallback on CLI reads handles this |

## 6. Definition of Done

The structure key encryption split is complete for the Vault Web Team scope when:

1. `decryptStructural` exists and is called for all structural object reads
2. All new structural object writes use `structureKey`
3. All existing vault tests pass (backwards compatibility confirmed)
4. Mixed vault fixture test passes (per-object fallback verified)
5. A consumer with only `structureKey` can traverse commit graph but cannot read filenames or file content
6. `sg-vault.js` carries `_structureKey` state and initialises it in both vault open paths (full key + simple token)
