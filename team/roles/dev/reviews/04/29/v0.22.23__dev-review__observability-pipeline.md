## Implementation Streams

### Stream 1: API Extraction Refactoring (Priority: High, Pre-expansion gate)

Extract the three APIs from the entangled current implementation:

**Index Management API**
```
POST /admin/obs/indices/create          — create index with specified mapping
POST /admin/obs/indices/import          — import data from S3/vault/file
GET  /admin/obs/indices/list            — list indices with doc counts and sizes
DELETE /admin/obs/indices/{name}        — delete index
POST /admin/obs/indices/consolidate     — merge multiple small files before import
```

**Dashboard Management API**
```
POST /admin/obs/dashboards/create       — create dashboard from JSON spec
GET  /admin/obs/dashboards/list         — list dashboards
POST /admin/obs/dashboards/screenshot   — screenshot via Playwright
POST /admin/obs/dashboards/export       — export dashboard definition
POST /admin/obs/dashboards/import       — import dashboard definition
```

**Connector API**
```
POST /admin/obs/connectors/cloudfront/import  — import CloudFront logs for date range
POST /admin/obs/connectors/playwright/import  — import Playwright instance logs
POST /admin/obs/connectors/sgsend/import      — import SG/Send server logs
POST /admin/obs/connectors/{type}/import      — generic connector endpoint
```

**Effort estimate:** 3–5 dev sessions. Refactoring existing working code into a clean API layer. Medium complexity. The patterns are already proven; the work is extraction and formalisation.

### Stream 2: LETS Pipeline Stages for CloudFront (Priority: High, Demonstrates the Pattern)

Implement the full LETS pipeline for the existing CloudFront connector:

| Pass | Input | Output | Key Operations |
|------|-------|--------|---------------|
| Load | CloudFront S3 logs (`.gz` files) | Raw in memory | Decompress `.gz` files |
| Extract (Pass 1 — Consolidate) | Multiple `.gz` files for date range | `cloudfront-raw-{date}.ndjson` | Concatenate, normalise |
| Transform (Pass 2 — Enrich) | `cloudfront-raw-{date}.ndjson` | `cloudfront-enriched-{date}.ndjson` | Parse into typed fields; break URL into segments array; tag event types |
| Transform (Pass 3 — Aggregate) | `cloudfront-enriched-{date}.ndjson` | `cloudfront-signals-{date}.json` | Requests/min, error rates, top URLs, unique visitors (IP hash) |
| Save | Enriched + signals files | `s3://sg-observability-*` buckets | Write to correct stage bucket |

URL segmentation is the key value-add of Pass 2: break `/api/v1/vault/status` into `["api", "v1", "vault", "status"]` as an Elastic keyword array. This enables plain `term` queries for URL path components.

**Effort estimate:** 2–3 dev sessions. The enrichment logic (URL parsing, event tagging, aggregation) is the main work. Storage write pattern is straightforward.

### Stream 3: Security Cleaning (Priority: High, Required Before Any Real Log Processing)

Stage 1 of the pipeline (doc 326). Must run before any log leaves the raw capture bucket.

| Operation | Pattern | Risk Level |
|-----------|---------|-----------|
| Hash/remove API keys | Regex + known formats | High — new formats will miss |
| Hash/remove bearer tokens | Auth header patterns | High |
| Hash/remove PII (email, IP) | Well-known patterns | Medium |
| Flag-but-preserve potential secrets | Output to a scan report | Medium |

**Note from Architect review:** Security cleaning is the attack surface. A missed pattern means plaintext secrets in Elastic. Needs AppSec sign-off before production use. Canary scanner on clean output recommended.

**Effort estimate:** 2 dev sessions for core cleaning rules + 1 session for canary scanner. Ongoing maintenance as new log sources are added.

### Stream 4: Dashboard Templates (Priority: Medium)

Decouple dashboard creation from data analysis. Each dashboard becomes a JSON spec:
- Applyable to any date's data (date-parameterised index names)
- Version-controlled (in AMI or vault)
- Applicable to any index following the expected mapping
- Screenshot-capturable via Playwright

Five initial templates (from doc 326 Stage 5):
1. Health Overview (Stage 3 signals)
2. Traffic Analysis (Stage 2 HTTP requests)
3. Vault Activity (Stage 2 vault events)
4. Playwright Sessions (Stage 2 browser actions)
5. System Resources (Stage 2 system stats)

**Blocker:** Lens visualisation experiment needed first (Open Decision #27). Lens is the modern Kibana visualisation type; without Lens support, dashboards use legacy types. Experiment before committing to a template format.

**Effort estimate:** 1 session for experiment; 3 sessions for five templates (post-experiment).

## Risk Assessment

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Security cleaning gaps (secrets leak to Elastic) | High | AppSec review; canary scanner; strict IAM on clean bucket |
| Multi-repo coordination (no clear ownership) | High | Architect to produce component boundary map before any code |
| Lens experiment blocks dashboard templates | Medium | Run experiment early; keep legacy fallback available |
| CloudFront log format changes (AWS side) | Medium | Test against historical logs; versioned parser with schema bump |
| NDJSON schema drift (no versioning) | Medium | Add `_schema_version` field from the start |
