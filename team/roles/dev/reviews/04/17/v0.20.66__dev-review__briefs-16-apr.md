## Failure Summary

`sgit share` (v0.10.9) fails every time on large vaults (~38MB) from Claude Web environments with HTTP 503 "DNS cache overflow". This is a consistent, reproducible failure — not a flaky network issue. The same vault pushes and pulls correctly via `sgit push`.

**Reproduction:** `sgit --token graphs-and-maps --base-url https://dev.send.sgraph.ai share` on a ~38MB vault from a Claude Web container.

---

## Root Cause Analysis

Based on the evidence in the investigation document, the most likely root cause is:

> The Claude Web container's egress proxy rejects large PUT requests (5MB multipart part) to S3 presigned URLs with a plain-text 503 "DNS cache overflow" response. The proxy handles small individual PUTs (used by `sgit push`) without issue. The "DNS cache overflow" message is non-standard — it is the proxy's internal error, not an S3 error.

**Supporting evidence:**
- `curl` GET to S3 host returns 403 (normal — DNS works, TLS works)
- `curl` PUT to S3 host with small unsigned payload returns 501 (expected)
- `sgit push` individual object uploads succeed
- An earlier `sgit share` on the same vault at ~38KB succeeded (share token `file-hour-6653`)
- S3 never returns plain-text 503 — it always returns XML error bodies

**What is NOT the cause:**
- DNS resolution failure (curl confirms DNS works)
- sgit HTTP handling bug (the error comes from S3, not from sgit's request construction)
- S3-side issue (non-standard error body, normal curl connectivity)

## Implementation Streams

### Stream 1: Server-Side Audit (Pre-requisite — 0.5 day)

Before writing any fix, verify the server side is not contributing:

1. Check `POST /api/transfers/create` response — confirm the returned presigned URL parts have no `Content-Length` constraint that the 5MB part violates
2. Check SG/Send server logs for the failed upload attempt — did the create call succeed? Was a presigned URL actually returned?
3. Check if presigned URL expiry is appropriate for large uploads (default is typically 1 hour — should be fine)

**Effort:** 2–4 hours. No code changes needed — just server log inspection + API response audit.

---

### Stream 2: Error Messaging Improvement in sgit (1–2 days — `sgit-ai` repo)

Regardless of the fix chosen, `sgit share` should fail better:

1. **Before upload:** If vault size > 4MB, print: `Warning: large vault detected (~XMB). In egress-controlled environments (e.g. Claude Web), sgit share may fail for vaults > ~4MB. Use 'sgit push' and share the vault key directly as a workaround.`
2. **On 503:** Catch HTTP 503 and check if response body is plain text (not XML). If yes, print: `Error: Upload rejected by network proxy. This is a known issue in egress-controlled environments. Workaround: use 'sgit push' and share the vault key directly. See: https://github.com/SGit-AI/SGit-AI__CLI/issues/XXX`
3. **Log `x-deny-reason` header:** If present in the 503 response, include it in the error output — provides debugging signal for the proxy operator

**Files affected:** `sgit-ai` CLI codebase (not this repo). `sgit-ai` PyPI package release needed.
**Effort:** 1–2 days.

---

### Stream 3: Delta-Share Fallback in sgit (3–5 days — `sgit-ai` repo + SG/Send server)

The structural fix recommended by the Architect review: when the large-blob share upload fails, fall back to delta-share (individual object uploads + manifest).

**Phase 3a: Server-side manifest endpoint (SG/Send repo — 1–2 days)**

New endpoint: `POST /api/transfers/share-manifest` — accepts a JSON manifest of object IDs that constitute a share snapshot. Returns a share token that the browse viewer resolves via the manifest rather than a single blob.

Requirements:
- Auth: access token required
- Manifest format: `{ "vault_id": str, "snapshot_id": str, "objects": [{"id": str, "path": str, "size": int}], "root_ref": str }`
- Server stores manifest in Storage_FS under `transfers/share/{token}/manifest.json`
- Browse endpoint must be updated to serve manifest-based share tokens (walk manifest, serve each object)

**Phase 3b: CLI fallback logic (sgit-ai repo — 1–2 days)**

In `sgit share`:
1. Attempt large-blob upload as today
2. On 503 or SSL EOF: log warning, switch to delta-share mode
3. Delta-share: ensure all vault objects are pushed (call push internally), then call `POST /api/transfers/share-manifest` with the current HEAD ref's object list
4. Return share token as before — user experience unchanged

**Phase 3c: Browse viewer manifest support (User UI — 1 day)**

`send-browse--v0.3.2.js` already supports `BrowseDataSource` — the vault viewer just needs a `ManifestDataSource` implementation that fetches objects via the server API rather than from a single blob. This may already be partially implemented via the `sg-vault.js` object store path.

**Total effort:** 3–5 days across three codebases. Cross-team coordination needed (sgit-ai team + SG/Send Explorer).

## Workaround (Available Now, No Code Change)

The investigation document documents a working workaround:

```
# Instead of sgit share, do:
sgit push                              # Push all objects to server
# Then share the vault key out-of-band (chat, email, etc.)
# Recipient clones with: sgit clone <vault-key>
```

The trade-off: vault key gives write access; share token is read-only. For agent-to-agent handoffs where read-only is sufficient, this workaround is acceptable but requires trusting the recipient with the vault key.
