### Stream D: One-Shot Feedback Loop Orchestration (Doc 205)
**Priority:** Medium — important but complex; needs Stream A complete first  
**Effort:** ~8–12 days  
**IFD version:** New tool / extension of one-shot LLM dev environment (Section 19)

**Tasks:**
1. Define bundle file format (system-prompt.md, context.md, findings.md, code-samples/, task.md, metadata.json)
2. Build bundle viewer/editor (display current bundle state, token count per section)
3. Implement extract step: after LLM response, parse and extract facts/decisions/code
4. Implement prune step: send exchange to smaller/cheaper model → produces condensed context
5. Implement cycle: send bundle → get response → extract → prune → update bundle → commit
6. Wire to sgit: each updated bundle = one commit on current branch
7. Branch creation for exploration (fork current vault at current commit)
8. Wire Token Visualiser (Stream B) to show context size across cycles
9. Build at least one benchmark test: full bundle vs pruned bundle on same task

**Risk:** High. The extract+prune step requires LLM calls and produces non-deterministic output. The benchmark tests are the only way to verify quality is maintained. Build the tests before building the pruning — you need a baseline to compare against.

**Note on two-LLM pattern:** When using Ollama for pruning, the Ollama API must already be working (Stream A validation covers this). Do not implement async two-LLM pattern until the basic synchronous version works.

## Risks and Concerns

1. **Context-extraction quality:** The prune step requires LLM output to be machine-parseable. If the extraction is unreliable (hallucinations, missed items), the feedback loop degrades. The benchmark tests in Stream D are the mitigation.

2. **Ollama compatibility:** The brief assumes Ollama works as a drop-in replacement for cloud APIs. Stream A testing against Ollama will reveal which components have compatibility gaps. Fix them in Stream A before depending on Ollama in Streams D/E.

3. **Agent context pages don't exist yet:** Multi-Agent UI loads agent context from "Librarian-maintained pages." Those pages do not exist in a stable, machine-readable format. The Librarian needs to define and publish these before Stream E can be completed. This is a cross-team dependency.

4. **sg-git-graph read-structure key:** The live SGit data source requires the read-structure key (PROPOSED in Section 17). If this is not available when Stream C is being built, the live data source must be deferred. The JSON data source can be built first without this dependency.

5. **OWASP submission:** The brief asks for white paper + slides + handout. These are Ambassador/Alchemist work. Dev is not the lead role. Flag to the Ambassador team.
