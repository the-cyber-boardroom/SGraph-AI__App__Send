## Stream 1: Vault Upload Beta

### What to build

Five features from doc 283:

1. **Beta toggle** in main SG/Send upload UI (switches folder-zip to vault-backed mode)
2. **Instant vault creation** on folder drop + share link returned before first file uploads
3. **Batched commits** (10–20 files per commit, or by size threshold, whichever fires first)
4. **`.sg-upload-status.json`** written on each batch commit (total files, uploaded, remaining, bytes, ETA, timestamp)
5. **Placeholder files** for queued large files (distinct UI: greyed-out, "uploading..." label; replaced in same commit as the real file lands)
6. **Tab close warning** while upload in progress (browser `beforeunload` event)

### Reuse from existing codebase

- Vault creation API — EXISTS (vault create endpoint)
- Incremental commit (individual file upload) — EXISTS
- Share token generation — EXISTS
- Vault browse UI renders file list — EXISTS
- Large file support (presigned S3) — EXISTS (for files > 4MB)

All six features above are new, except the vault creation call. The new surface is in the upload UI component.

### Effort estimate

| Task | Effort |
|------|--------|
| Beta toggle UI | 0.5 days |
| Instant vault creation + share link display | 1 day |
| Batched commit logic (queue management, batch trigger) | 1.5 days |
| Upload status file (.sg-upload-status.json) | 0.5 days |
| Recipient status banner rendering | 0.5 days |
| Placeholder files (uploader side) | 1 day |
| Placeholder files (recipient UI treatment) | 0.5 days |
| Tab close warning | 0.25 days |
| Tests | 1 day |
| **Total** | **~6.75 days** |

### Dependencies and blockers

- **OD #6 (upload branch strategy)** must be resolved before shipping. An upload that fails partway leaves a partial vault. Recommend: create a dedicated branch per upload session (`upload/{session-id}`), merge to main on completion. If session fails, the branch is not merged.
- **Acceptance criteria 4 (recipient sees status banner)** requires the browse UI to know about `.sg-upload-status.json`. This is a cross-UI dependency — both upload (sender) and browse (recipient) sides need changes in the same release.

### Flags for Architect

- `.sg-upload-status.json` convention should be formally documented before shipping (raised in Architect review)

## Stream 2: Auth MVP (Social Login + Vault Key Storage)

### What to build

From doc 292:

1. **`<sg-login>`** — OAuth flow component (Google + GitHub), Cognito + Firebase
2. **`<sg-user-data>`** — read/write user attributes (vault keys, preferences); storage-agnostic interface
3. **`<sg-vault-manager>`** — list/load/open/create vaults from stored keys
4. **`<sg-admin-panel>`** — user count, registration stats, group management (admin only)
5. **AWS Cognito setup** — Google + GitHub OAuth configured, custom user attributes for vault keys
6. **GCP Firebase Auth setup** — Google OAuth configured
7. **Vault key storage** — encrypted vault keys stored as Cognito user attributes (Option A for MVP)

### Complexity factors

- Two identity providers = two integration paths to maintain
- Client-side OAuth flow requires careful token handling (no server-side processing claim)
- Cognito user attributes have size limits (investigate before committing to Option A at scale)
- Admin mode requires Cognito user groups + conditional rendering in `<sg-admin-panel>`
- Zero-knowledge guarantee at identity layer: admin must not be able to decrypt user vault keys

### Effort estimate

| Task | Effort |
|------|--------|
| Cognito setup (Google + GitHub OAuth, user pool, custom attributes) | 1 day |
| Firebase Auth setup (Google OAuth) | 0.5 days |
| `<sg-login>` component (Cognito path) | 1.5 days |
| `<sg-login>` component (Firebase path) | 1 day |
| `<sg-user-data>` component | 1.5 days |
| Vault key storage (encrypt, store as Cognito attribute, retrieve + decrypt on login) | 2 days |
| `<sg-vault-manager>` component | 1 day |
| `<sg-admin-panel>` component | 1 day |
| Cross-provider research (attribute sizes, token scopes, portability) | 1 day |
| Tests (each component, each provider path) | 2 days |
| **Total** | **~12.5 days** |

### Dependencies and blockers

- **OD #11 (master vault key derivation)** — AppSec must sign off on the PBKDF2/HKDF scheme before vault key encryption is implemented
- **Cognito attribute size limits** — must be confirmed before committing to Option A. If the limit is exceeded, Option B (encrypted bundle in SG/Send) becomes the path, which adds ~2 days
- Profile page (OD #1) is blocked on `<sg-login>` existing — shipping `<sg-login>` first unblocks the profile page as a quick follow-on

### Flags

- Doc 292 recommends the `<sg-user-data>` component interface be storage-agnostic so Option B can be substituted without changing callers. This is a good design requirement — build to it from day one.
- Research question 5 (cross-provider portability) should be documented as a known limitation in the MVP rather than solved.

| Published summary as folder view | EXISTS | Share token + _page.json rendering |
| Audit trail display | **NEW** | Render vault commit log as timeline |

### Client-side-only claim

This MVP must prove: "server stores only encrypted ciphertext." Implementation must be verifiable by network inspection. Every data flow step needs to be auditable. Dev should instrument the demo to show network calls in real time.

### LLM prompt construction review (IMPORTANT)

AppSec and DPO must review the LLM prompt before the MVP is shown to any healthcare partner. The brief says "send anonymised/structured data" but does not define anonymisation. For a GDPR-compliant demo:

- No patient names, dates of birth, or other direct identifiers in the LLM call
- LLM receives only structured health indicators (e.g. "BMI: 28, blood pressure: 130/85")
- LLM response stored in vault labelled as "AI-generated, pending clinician review"

This review must happen before any external demo.
