**Source:** Document 194 (`v0.19.7__dev-brief__folder-as-narrative.md`)
**Effort:** ~2 days | **Status:** PROPOSED

### What to build

**Part A: `_config.json` manifest** (1 day)
```json
{
  "title": "Project Alpha",
  "description": "Q1 2026 deliverables",
  "start_file": "README.md",
  "sections": ["docs/", "assets/", "reports/"],
  "featured": ["reports/Q1-summary.md"]
}
```
Browse view reads `_config.json` if present, uses it to:
- Override folder display title
- Set the default file that opens when folder is first accessed
- Reorder sections in the tree
- Mark files as "featured" (highlighted in UI)

**Part B: Markdown image resolution** (½ day)
Currently, markdown files display `![img](./photo.jpg)` as broken images because relative paths resolve against the request base, not the folder path. Fix: resolve image `src` attributes against the folder's base path before rendering.

**Part C: Portfolio component** (1 day)
Dedicated layout for personal/professional portfolios:
```json
{
  "type": "portfolio",
  "headshot": "photo.jpg",
  "bio": "bio.md",
  "sections": [
    { "title": "Work", "items": ["work/"] },
    { "title": "Videos", "items": ["videos/"] }
  ]
}
```

**Part D: Per-file component overrides** (½ day)
In `_config.json`, allow specifying which component renders a file:
```json
{
  "overrides": {
    "data.csv": "chart-viewer",
    "gallery/": "gallery-layout"
  }
}
```

### Implementation order
1. `_config.json` reading + title/start-file support (easiest win)
2. Markdown image resolution (important bug fix)
3. Portfolio component
4. Per-file overrides

**Source:** Documents 195 + 202

### Minimum viable set

| Component | Effort | Description |
|-----------|--------|-------------|
| `sg-llm-events.js` | ½ day | Frozen event constants — identical pattern to sg-tree-events; copy and adapt |
| `sg-llm-reality.js` | 1 day | Block assembly UI: system, context, history, memory, question inputs; outputs assembled messages array |
| `sg-llm-request.js` | ½ day | Headless fetch to provider API; fires events on token chunks; handles abort |
| `sg-llm-output.js` | 1 day | Streaming text display with markdown rendering; auto-scroll; copy button |

### Key implementation detail for `sg-llm-reality.js`

The memory block is special: it's the only block the LLM rewrites. The UI needs a "Update Memory" button that:
1. Takes the current memory block + last assistant response
2. Sends a meta-prompt: "Update the session memory to incorporate what was just learned, pruning stale entries"
3. Replaces the memory block with the LLM's response

This is Mode 2 in the spec. It should be a distinct code path from Mode 1 (Build).

### `sg-llm-stats.js` (add in v0.1.1)

Token counting per block is important for managing context budget. Each block in `sg-llm-reality.js` should show a token estimate. `sg-llm-stats.js` can be added in v0.1.1 once the core is working.

## Stream 5: sg-wasm WASM Loader Component

**Source:** Document 196
**Effort:** ~2 days | **Status:** PROPOSED | **Prerequisite:** None

### What to build

```html
<sg-wasm module="whisper" src="/wasm/whisper.wasm" cache-key="whisper-v1.0">
```

**Core behaviour:**
1. On connect: check IndexedDB for cached WASM bytes at `cache-key`
2. If not cached: fetch from `src`, store in IndexedDB
3. If cached: load from IndexedDB (offline-capable)
4. Fire `sg-wasm:ready` event when module is initialised
5. Fire `sg-wasm:progress` events during download (for progress bar)
6. Expose `module.instance` for consumers to call WASM functions

**Cache invalidation:** `cache-key` acts as the version. Changing `cache-key` forces re-download. Simple and explicit.

**Size note:** Whisper.wasm and FFmpeg.wasm are large (50MB–200MB). The progress event is important UX — users need to see download progress on first use.

## Stream 6: In-Browser Audio Transcription

**Source:** Document 196
**Effort:** ~3 days | **Status:** PROPOSED | **Prerequisite:** `sg-wasm`

### Two-pass architecture

**Pass 1 (real-time):** Faster Whisper model (e.g., whisper.base). Good enough for immediate display while processing.
**Pass 2 (cleanup):** Larger model (whisper.large or whisper.medium). Higher quality; replaces Pass 1 output.

This gives users fast feedback while the higher-quality result processes in the background.

### Key outputs
- Timestamped transcript (for navigation — click timestamp to seek audio)
- SRT format export
- VTT format export (for video captions)
- Plain text export

### Integration with sg-wasm
The transcription component (`sg-transcription.js` or similar) waits for `sg-wasm:ready` from a Whisper instance, then passes audio chunks via the WASM module's API.

### Research needed
- What Whisper WASM builds are available? (whisper.cpp WASM is most mature)
- What is the browser memory limit for running whisper.large? (May need to cap at whisper.medium for browser use)

## Stream 7: Vault Migration

**Source:** Document 196
**Effort:** ~5 days | **Status:** PROPOSED | **Prerequisite:** Mature vault API

### What's needed

**Server-side:** New vault API endpoints for export/import:
- `GET /vault/{id}/export` — stream all encrypted objects as a zip
- `POST /vault/{id}/import` — receive exported zip, recreate objects

**Client-side:** SGit CLI commands:
- `sgit push --remote <other-server>` — clone vault objects to another server
- `sgit pull --from <other-server>` — pull vault objects from another server

**Key derivation during migration:** The vault encryption keys are derived from the vault key (user's passphrase). Since the key stays with the user (zero-knowledge), migration does not require re-encryption — the same ciphertext is valid on the destination server.

### Risk note
This is the most complex stream in the batch. The edge cases (partial migration, conflict resolution, vault ID collision across servers) need careful design. Flag for a dedicated architecture session before implementation begins.
