## Stream 2: `@agent` Code Comments (Doc 206) — LOW EFFORT, HIGH VALUE

### Implementation Plan

This is a Librarian-side feature but Dev needs to define the comment syntax convention:

```python
# @librarian: observation or request for the Librarian role
# @qa: request for QA (test this edge case, verify this behaviour)
# @architect: architectural concern flagged during implementation
# @dev: technical debt or future optimisation note
```

**Rules:**
- One comment per `@agent` tag
- Tag must be on its own line (not inline after code)
- The comment text is the full task description (no separate issue file needed at authoring time)
- Multi-line: continuation lines start with `#   ` (two spaces, to match tag indentation)

**Librarian processing task (add to daily run):**
1. Glob all Python/JS files
2. Match `# @{role}:` pattern, extract file + line number + context (surrounding 5 lines)
3. Create issue in `.issues/MM/DD/{role}-{sequence}.md`
4. Replace comment with `# [processed → .issues/MM/DD/{role}-{sequence}.md]`
5. Stage modified files (note: this requires the Librarian to have write access to code files — this is acceptable since it is only removing comments, not modifying logic)

**Effort:** 1–2 hours for the Librarian processing script. Zero effort for the comment-writing pattern (just a convention).

## Stream 3: `sg-git-graph` Component (Doc 210) — MEDIUM PRIORITY

This component has high architectural value (visualises LLM threads, SGit vaults, IFD versions) and is one of the cleaner implementations in the batch.

### Implementation Plan

**Phase 1: JSON data source (immediate, testable)**

Build the component with a JSON data source only. Define the JSON schema first:

```json
{
  "branches": [
    { "name": "main", "color": "#4ec9b0" },
    { "name": "experiment-a", "color": "#f0e68c" }
  ],
  "commits": [
    { "id": "c1", "branch": "main", "message": "Initial bundle", "timestamp": "..." },
    { "id": "c2", "branch": "main", "parent": "c1", "message": "Refined context", "timestamp": "..." },
    { "id": "c3", "branch": "experiment-a", "parent": "c1", "message": "Exploring approach A", "timestamp": "..." }
  ],
  "merges": [
    { "from": "experiment-a", "to": "main", "commit": "c4" }
  ]
}
```

This schema serves both real SGit graphs and programmatic LLM thread graphs.

**Phase 2: Programmatic API**

Wrap the JSON builder in a fluent API:
```javascript
const graph = new SGitGraph();
const main = graph.branch('main');
const c1 = main.commit('Initial');
const expA = graph.branch('exp-a', c1);
expA.commit('Explore');
main.merge([expA], 'Merge');
document.getElementById('container').appendChild(graph.element());
```

**Phase 3: Interactive features**

Click, hover diff, zoom/pan via the chosen layout library.

**Phase 4: Live SGit vault data source**

Read structure key, traverse commit graph, build JSON representation, render.

**Library selection (critical decision):**
- **Dagre-d3:** Mature, handles DAG layout correctly, D3 rendering. Heavier than ideal.
- **Cytoscape.js:** Full graph library, dagre plugin for layout. Good option.
- **Custom layout:** Simple git-style lane-based layout (commits per column per branch). Lightweight, predictable. Doable in ~200 lines of JS for basic cases.

Recommend: Try the custom lane layout first (git-style columns, deterministic). Fall back to Dagre if the layout becomes complex.

**Effort:**
- Phase 1 (JSON + render): 4–6 hours
- Phase 2 (programmatic API): 2–3 hours
- Phase 3 (interaction): 3–5 hours
- Phase 4 (live vault): 3–5 hours (depends on structure key traversal)

## Stream 4: One-Shot Feedback Loop (Doc 205) — MEDIUM-LONG TERM

This is the most architecturally ambitious brief. Implementation depends on:
1. Bundle file schema (Architect must define)
2. SGit commit API (already PROPOSED, needs to be built or confirmed)
3. Token Visualiser (requires instrumented bundle assembler)

### Immediate Dev Task

Do not build the full feedback loop. Build the **bundle assembler**:

```python
class BundleAssembler:
    def __init__(self, vault_path):
        self.vault_path = vault_path
    
    def assemble(self) -> BundleResult:
        sections = {
            'system_prompt': self._read('system-prompt.md'),
            'context': self._read('context.md'),
            'findings': self._read('findings.md'),
            'task': self._read('task.md'),
        }
        token_counts = {k: count_tokens(v) for k, v in sections.items()}
        prompt = '\n\n---\n\n'.join(sections.values())
        return BundleResult(prompt=prompt, token_counts=token_counts, total=sum(token_counts.values()))
```

This is low-risk, testable, and provides the instrumentation the Token Visualiser needs. The prune+extract step can be added later.

**Effort for bundle assembler:** 2–3 hours.
**Effort for full loop with SGit integration:** 1–2 weeks.
