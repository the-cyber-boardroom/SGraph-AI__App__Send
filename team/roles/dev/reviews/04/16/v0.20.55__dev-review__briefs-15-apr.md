## Code Review: VFS Bridge (doc 274 — code already in dev)

The comms brief documents implementation that is already merged. Dev review for awareness and handoff.

### VFS Bridge (`send-browse--v0.3.2.js`)

The postMessage protocol is straightforward:
```
iframe → parent: { type: '__sgVfsReq', reqId, path }
parent → iframe: { type: '__sgVfsRes', reqId, buffer: ArrayBuffer } (zero-copy transfer)
```

The `HTMLImageElement.prototype.src` setter override intercepts assignments at the prototype level, catching `new Image()` assignments before DOM insertion. This is the non-obvious part — `MutationObserver` fires after insertion, too late for `img.src` already set. The prototype override is correct.

**`__sgVfs` flag:** Set `true` before async fetch, cleared `false` after. This prevents the `MutationObserver` backup from double-processing the same request. The flag is element-level (on the `<img>` itself), so multiple images in the same iframe work independently.

**Dev note for Tools team:** The VFS bridge is injected into the HTML as an inline script before the blob URL is created. Any HTML viewer tool that needs asset loading from vault must follow this same pattern. Copy the approach, don't reinvent it.

### Immutable Block Cache (`sg-vault-object-store.js`)

Cache key pattern: `https://sgvault/{vaultId}/bare/data/{objectId}` — synthetic URL, never actually fetched. The `https://sgvault/` origin is arbitrary (won't conflict with real network requests). The `{vaultId}` prefix is the critical scoping prefix.

The guard is simple: `objectId.includes('-imm-')`. If an object ID doesn't contain `-imm-`, it goes through the normal fetch path. The mutable ref types (`muw`, `snw`) never match this guard.

**Dev note:** The Cache API is asynchronous and write-through. On first load, blocks are fetched normally and written to cache. On subsequent loads, the cache hit eliminates the network round-trip entirely. The decryption step is CPU-only (~ms) and happens every load regardless of cache — this is correct zero-knowledge behaviour.

## Implementation Streams

### Stream 1: JPEG Compression in Infographic Pipeline — PROPOSED

**Priority: High** | **Effort: 2–3 days** | **Dependency: Infographic tool (itself PROPOSED)**

The infographic API already returns base64 PNG. Adding compression variants at generation time requires only:

```javascript
async function optimizeSlide(b64Png) {
  const img = new Image();
  img.src = 'data:image/png;base64,' + b64Png;
  await new Promise(r => img.onload = r);

  const make = (maxW, q) => {
    const c = document.createElement('canvas');
    const s = Math.min(1, maxW / img.width);
    c.width = Math.round(img.width * s);
    c.height = Math.round(img.height * s);
    c.getContext('2d').drawImage(img, 0, 0, c.width, c.height);
    return new Promise(res => c.toBlob(res, 'image/jpeg', q));
  };

  return {
    dist: await make(1200, 0.82),  // 113KB avg — vault storage
    li:   await make(800,  0.70),  // 50KB avg — LinkedIn
  };
}
```

No new dependencies. Works in all modern browsers. The canvas is reused across slides. This should be built as a utility function alongside the infographic tool, not a separate component.

**Risk:** none. This is pure browser Canvas API.

### Stream 2: PDF Compression Tool (Standalone) — PROPOSED

**Priority: Medium** | **Effort: 3–5 days (including UI)** | **Dependencies: pdf.js, jsPDF**

The brief's implementation example is complete and tested. The tool is:
1. Drop PDF → pdf.js renders each page to Canvas
2. Canvas.toDataURL('image/jpeg', 0.82) re-encodes each page
3. jsPDF assembles a new PDF from the JPEG pages
4. Download compressed PDF

CDN dependencies (can be loaded on demand — user accepts the download):
- pdf.js: `https://cdnjs.cloudflare.com/ajax/libs/pdf.js/4.0.379/pdf.min.mjs` (~200KB)
- jsPDF: `https://cdnjs.cloudflare.com/ajax/libs/jspdf/2.5.1/jspdf.umd.min.js` (~300KB)

Memory management: process pages sequentially, reuse single Canvas. A 107-page PDF (each page ~3.2MB in memory) can be processed in ~15 seconds — acceptable for an explicit user action.

**MVP UI scope:**
- Drop zone for PDF
- Quality slider (0.70–0.95) with live size estimate
- Page count and estimated output size shown before compressing
- Download button after compression

**Note:** `pdf-compressor.html` referenced in brief does not exist in any SGraph repo. This tool needs to be built from scratch or recovered from the OWASP session environment.

### Stream 3: Browser Video Creation Tool — PROPOSED

**Priority: Low (exploratory)** | **Effort: 2–4 weeks for MVP 1** | **Dependencies: Kokoro.js, FFmpeg.wasm**

This is the most ambitious item. Breaking the MVP path into concrete tasks:

**Phase 1 — Slideshow + TTS (MVP 1):** ~1 week
- Load Kokoro.js (82M ONNX model, ~160MB first download, cached after)
- Accept slide images + narration text per slide
- Generate audio per slide via `KokoroTTS.generate()`
- Animate slide on Canvas (crossfade transitions)
- Combine Canvas stream + audio via MediaRecorder
- Output WebM blob

**Phase 2 — Add text animation (MVP 2):** ~3 days
- Animated caption bar at bottom (text fades in over 1s)
- Progress bar across bottom of video
- Lower-third speaker name strip

**Phase 3 — Audio-reactive 2D avatar (MVP 3):** ~2 days
- Web Audio API analyser to get amplitude
- Simple Canvas-drawn face in corner with mouth open proportional to amplitude
- No Three.js needed at this stage

**Resilience concern (flagged by Architect):** Long renders (~15 min for 30 slides) need per-slide output — if the tab closes mid-render, the current approach loses everything. The fix: process one slide at a time, write each slide's WebM blob to IndexedDB, then concatenate at the end (or use a sequential recorder with `requestData()` at slide boundaries). This should be in the MVP 1 design.

**Key constraint:** Kokoro.js uses `import()` (ES module). The tool must be served with proper CORS headers and cannot be a plain file:// URL.
