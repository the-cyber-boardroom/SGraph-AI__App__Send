## Implementation Streams

### Stream 1: JS API Convention (docs 265 + 267) — P1

This is the foundational convention for all future tools. Every tool on `tools.sgraph.ai` must conform to it. The infographic generator is the reference implementation.

**What needs to be built:**

| Deliverable | What It Is | Complexity |
|---|---|---|
| `window.__tool` API on infographic generator | Retrofit existing tool to expose `connect()`, `set_template()`, `generate()`, `export_png()`, `get_cost()`, plus meta methods | Medium — the logic already exists, need to expose it |
| `window.__tool_manifest` JSON | Static manifest JSON embedded in the tool (actions, params, returns, events, version, base_url) | Low — write the JSON |
| `window.__tool_skill` text | SKILL.md-style plain text embedded in the tool (prose description for agent bootstrapping) | Low — write the text |
| Three SKILL files per tool | SKILL-human.md, SKILL-browser.md, SKILL-api.md deployed alongside the tool | Low per file — 3 files to write |
| `<sg-tool-api>` Web Component | Reads manifest, renders Swagger-style panel with [Try It] + [Copy Code] per action, live event log, result viewer | High — new Web Component |
| In-browser test suite + REPL | Test contract assertions + console for ad-hoc calls | Medium |

**Suggested build order:**

1. Retrofit infographic generator with `window.__tool.*` API + manifest JSON + skill text (one session)
2. Validate with existing Playwright pipeline — should work immediately via `window.__tool.generate()`
3. Build `<sg-tool-api>` Web Component (uses manifest for auto-generation)
4. Write three SKILL files for infographic generator
5. Apply the same pattern to the next tool (video editor, LLM chat, etc.)

**Risk:** The Playwright pipeline already works via selector-based automation (as confirmed by QA). The `window.__tool` API needs to be verified not to break the existing pipeline while it is being added. The selector-based approach should continue to work during the transition.

**Dependency:** None — the infographic generator exists on `tools.sgraph.ai` and can be retrofitted immediately.

## Implementation Notes from Briefs

The following decisions from the briefs need to be tracked as implementation constraints:

1. **Events over polling** (doc 265): All async operations must use Promises or CustomEvents. No `setTimeout` polling for DOM changes. Violation of this rule should fail QA.
2. **`window.__tool` registration** (doc 265): Register after `DOMContentLoaded` and fire a `tool:ready` CustomEvent so consumers know when to start calling.
3. **SKILL-api.md format** (doc 265): Must include working code examples in every actions section. The brief shows a Playwright snippet. Include one example per action.
4. **Manifest `events` field** (Architect review recommendation): Add `events` to the manifest schema so `<sg-tool-api>` can display them.
5. **`dryRun` mode** (Architect review recommendation): Add to billing-sensitive methods before in-browser test suite ships.
