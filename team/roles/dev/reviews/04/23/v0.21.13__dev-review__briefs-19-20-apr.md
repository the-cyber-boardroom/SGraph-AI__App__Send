## Stream 1: SgSend JS API + Web Components (Doc 303) — HIGH PRIORITY

### Status: PROPOSED — does not exist yet

### What the brief asks for

1. `SgSend` module with 6 methods: `sendFile`, `sendText`, `sendFolder`, `receiveFile`, `receiveText`, `receiveFolder`
2. 3 Web Components: `<sg-send-drop>`, `<sg-send-receive>`, `<sg-send-panel>`
3. Reference tool on tools.sgraph.ai
4. Token history in localStorage (interim)
5. Standard JS meta API: `getManifest`, `getMethods`, `getSkills`

### Effort estimate

| Task | Estimate | Notes |
|------|----------|-------|
| `SgSend` module wrapping `sg-send.js` | 2–3h | Thin wrapper. Must NOT reimport encryption logic — see Architect concern. |
| `<sg-send-drop>` component | 2h | Drag-and-drop zone → emits `token-created`. |
| `<sg-send-receive>` component | 2h | Token input → emits `file-received`. |
| `<sg-send-panel>` combined component | 1h | Wrapper over drop + receive. |
| Token history (localStorage) | 1h | List view: recent tokens, copy, delete, sent vs received distinction. |
| Meta API (`getManifest` etc.) | 0.5h | Convention already established. |
| Reference tool on tools.sgraph.ai | 2–3h | New page/tool entry. Left pane: send. Right pane: receive. |
| IFD versioning and test scaffolding | 1h | Static file tests for each component. |
| **Total** | **~12h** | |

### Key risk

The brief says: "The API handles all of it. The caller does not need to know about encryption, chunking, upload endpoints, or token formats." This is only possible if the JS API wraps the existing `sg-send.js` cryptographic primitives. If `sendFile()` is written from scratch, it must undergo AppSec review before merging. Confirm the implementation approach before starting.

### Sequencing note

This stream MUST complete before the video tools and other tool integrations, because doc 304 explicitly requires `SgSend.sendFile()` as the output path for all three video tools.

## Stream 2: Video Editing Tools — Crop, Overlay, Capture (Doc 304)

### Status: PROPOSED — does not exist yet (existing video editor component is the base)

### What the brief asks for

**Tool 1:** Dynamic crop with keyframe regions
**Tool 2:** Multi-track overlay (titles, lower-thirds)
**Tool 3:** Video stream capture (MVP: screen-record + crop via Tool 1)

### Effort estimate

| Task | Estimate | Notes |
|------|----------|-------|
| Tool 1: keyframe crop editor | 8–12h | Timeline scrubber, draggable rectangle, keyframe store, interpolation/hard-cut, Canvas-based export |
| Tool 2: multi-track overlay | 6–8h | Track timeline UI, text element placement, preview render, burn-in export |
| Tool 3: stream capture (MVP only — screen record + crop) | 2h | Screen recorder already exists per 17 Apr brief. This is "record + hand off to Tool 1" |
| Integration: all tools → `SgSend.sendFile()` | 1h | After Stream 1 completes |
| Vault save option | 1h | Optional vault commit path |
| IFD versioning, static tests | 2h | |
| **Total** | **~20–24h** | |

### Key risk: Canvas API performance

Tool 1 (dynamic crop) uses the Canvas API for per-frame processing. For long videos (5+ minutes) or high
resolution, client-side frame processing can be extremely slow. The brief acknowledges this implicitly
("OffscreenCanvas" mentioned) but does not set performance constraints. **Recommendation:** Define a max
video duration (e.g. 30 seconds) and resolution cap for the MVP. Avoid a design that becomes unusable
for real-world content.

### Sequencing

Stream 2 depends on Stream 1 for the output path. Can build Tools 1 and 2 in parallel with Stream 1 up
to the point of the final export step.

## Stream 3: QA Refactoring — Playwright Modes + Traffic Generation (Doc 302)

### Status: PROPOSED — does not exist yet

### What the brief asks for

1. `PlaywrightClient(mode="local"|"lambda"|"ec2")` abstraction layer
2. All existing QA Playwright code refactored to use the abstraction
3. Hourly traffic generation: transfer + vault + browser verification cycle
4. Observability data pipeline from traffic generation

### Effort estimate

| Task | Estimate | Notes |
|------|----------|-------|
| `PlaywrightClient` abstraction class | 3–4h | Three mode dispatch, same API surface |
| Refactor existing QA to use abstraction | 4–6h | Depends on volume of existing ad-hoc Playwright code |
| Mode 1 (local Docker) implementation | 2h | `docker run` + session API calls |
| Mode 2 (Lambda invocation) | 2h | Lambda client, container image invocation |
| Mode 3 (EC2/AMI + MITM) | 3–4h | SSH/SSM session, proxy config, traffic log retrieval |
| Hourly traffic generation task | 3h | CloudWatch/cron, transfer + vault + screenshot cycle |
| Observability pipeline (metrics to dashboard) | 4–6h | Depends on monitoring stack; scope TBD |
| AMI packaging (Playwright + MITM sidecar) | 2–3h | EC2 provision script already exists (doc 298) |
| **Total** | **~24–30h** | |

### Key risk: observability scope

The brief describes "dashboards show files accessed, vaults created, response times." The monitoring stack
is not specified. This could mean CloudWatch, a custom dashboard, or simply writing to a vault. Before
implementing, decide: what is the monitoring target? CloudWatch adds AWS dependency. A vault-based log is
simpler and consistent with the platform model.
