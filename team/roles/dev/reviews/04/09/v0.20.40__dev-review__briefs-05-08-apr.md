## Implementation Streams

### Stream 1 (P0) — Profile Page + Credit Activation (Doc 246)

**Effort estimate:** Medium (2–3 sessions)
**Unblocks:** £5 credit outreach to ~50 users
**Risk:** Low — well-defined scope, no new infrastructure

**Implementation plan:**

1. **UI component:** Add profile icon (top-right) to send header. Clicking it opens a profile panel/page.
2. **Local storage viewer:** Display `access_token`, `encryption_keys`, `openrouter_api_key`, `vault_keys` from localStorage. Show values with copy buttons. Do NOT send these to server.
3. **Server-side data:** Add admin endpoint `GET /api/users/profile/{access_token}` returning credit balance, expiration, usage records. Or reuse existing token endpoints.
4. **Credit activation:** Button that calls a new endpoint `POST /api/users/credits/activate/{access_token}` — server marks the pre-allocated credit as active, sets expiration 30 days from now.
5. **Usage display:** Query existing analytics/token endpoints for usage tied to this access token.

**Key constraint (from Architect review):** Credit expiration check must be server-side. The server must reject expired credits on each token use, not rely on the client to check.

**IFD versioning:** Land in `v0/v0.3/v0.3.2/` as a new minor overlay — profile page is additive and independent.

**Acceptance criteria (per brief):**
- Profile accessible via user icon (top-right) ✓
- Local storage contents shown transparently ✓
- Credits activatable from the profile page ✓
- Credits have expiration (configurable, default ~1 month) ✓
- Usage/token consumption visible ✓
- No confidential data in server-side profile ✓
- OpenRouter key in local storage only, never sent to server ✓
- Credit balance shown consistently across platform ✓

### Stream 2 (P0) — Server-Side Zip Folder Versioning (Doc 241)

**Effort estimate:** Large (3–5 sessions)
**Unblocks:** Mode 2/3 folder sharing (docs 230-231), video playback component (doc 248)
**Risk:** Medium — new storage structure, conflict detection logic

**Implementation plan:**

**Phase 1 — Upload (write path):**
1. New route `POST /api/folders/create` — creates folder manifest (empty head)
2. New route `PUT /api/folders/{id}/file` — upload individual file, get file-id back (uses existing presigned upload infrastructure)
3. New route `PUT /api/folders/{id}/tree` — atomically update tree + head (compare-and-swap)

**Phase 2 — Download (read path):**
1. New route `GET /api/folders/{id}/head` — get current tree reference
2. New route `GET /api/folders/{id}/tree/{tree-id}` — get specific tree snapshot
3. New route `GET /api/folders/{id}/file/{file-id}` — download individual encrypted file

**Storage structure** (in Memory-FS / S3):
```
folders/{folder_id}/
  head                    # Mutable — JSON: {"tree_id": "..."}
  trees/{tree_id}         # Immutable — JSON tree snapshot
  files/{file_id}         # Immutable — encrypted file content
```

**Conflict detection:** Use S3 conditional write for `head` (`If-Match: {current_etag}`). In Memory-FS (for tests), implement a simple compare-and-swap with an in-memory lock. Tests MUST cover the conflict rejection path.

**Acceptance criteria (per brief):**
- Uploaded zip files stored as individual files ✓
- Only `head` is mutable; trees are immutable ✓
- CAS on tree updates with previous head reference ✓
- Conflict detected and rejected when head has changed ✓
- Individual files downloadable without full folder ✓
- Read and write tokens are separate ✓
- All tree versions retained for recovery ✓
- File IDs use deterministic-safe-paths pattern ✓

### Stream 5 (P2) — Serverless Playwright API (Doc 239)

**Effort estimate:** Large (3–5 sessions)
**Dependencies:** Docker setup, ECR, ECS/Fargate config
**Risk:** Medium — new infrastructure (Docker)

**Architect constraint (from review):** Automation scripts must come from trusted sources only (configured vaults or S3 paths). The `/session/{id}/execute` endpoint with an inline `script` field must require auth and must not accept scripts from unauthenticated callers.

**Implementation plan:**
1. Dockerfile: `playwright` + Chromium + FastAPI server
2. Routes: `/session/create`, `/session/{id}/execute`, `/session/{id}/save-state`, `/session/{id}/destroy`
3. Vault integration: load cookies via `GET /api/vault/read/{vault_id}/{file_id}`, save via `PUT /api/vault/write/{vault_id}/{file_id}`
4. Local testing: `docker run` with same API
5. First use case: QA smoke test for send.sgraph.ai
