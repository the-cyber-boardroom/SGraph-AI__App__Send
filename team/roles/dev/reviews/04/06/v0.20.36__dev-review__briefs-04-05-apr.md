## Implementation Streams

### Stream 1: URL Resolution for Grouped Storage Paths (Doc 230)

**Priority:** High — this is a blocking prerequisite for the backend storage restructuring (doc 223) to go to production.
**Effort:** Small (1–2 days)
**Dependencies:** Deterministic-safe-paths derivation function (doc 225), backend storage restructuring merged

**Tasks:**

1. **Path template registry** — Define a small enum/dict in code:
   ```python
   PATH_CONFIGS = {
       "LEGACY_FLAT":      "{transfer_id}/",
       "V1_BY_TOKEN":      "by-id-from__access-token/{at_id}/{transfer_id}/",
       "V1_FULL":          "by-id-from__user/{user_id}/by-id-from__access-token/{at_id}/by-id-from__decryption-key/{dk_id}/{transfer_id}/"
   }
   ```
   This registry is the contract for the fallback search. It must not grow without a code review.

2. **Derivation function integration** — Wire the PBKDF2 group ID derivation from doc 225 into the transfer read path. When a download request arrives, derive `dk_id` from the decryption key (if provided) and `at_id` from the access token (if provided).

3. **Option D — derive path** — New method `resolve_transfer_path(transfer_id, decryption_key=None, access_token=None)`:
   - If both keys available → try `V1_FULL` first
   - If only decryption key → try `V1_BY_TOKEN` variant
   - If neither → skip to fallback

4. **Option A — bounded fallback search** — If derivation doesn't find the object, iterate through `PATH_CONFIGS` in order (most recent first, `LEGACY_FLAT` last). At most 3–4 S3 HEAD requests.

5. **Instrumentation** — Log which config path resolved the transfer. Track fallback rate. Alert at >5%.

6. **Tests** — Unit tests for path derivation, integration tests for legacy flat path resolution, integration tests for V1 grouped path resolution. Target: all 6 acceptance criteria verified.

**Risk:** Low. No API changes, no client changes. Worst case: slightly elevated latency on legacy transfers until migration completes.

### Stream 2: Collaborative Folders — Mode 2 MVP (Doc 231, Phase A)

**Priority:** Medium — enables the photo gallery product and large folder support
**Effort:** Large (1–2 weeks for Mode 2 alone)
**Dependencies:** URL resolution (Stream 1), folder transfer model design decision, deterministic-safe-paths function

**Prerequisite decisions needed before coding:**
- Architect to confirm: folder transfer as first-class object (option b) vs. bundle of independent transfers (option a)
- Architect to confirm: manifest stored at deterministic path `{folder_key_derived_id}/__manifest__`
- Designer to confirm: "Make this folder editable" toggle UX on creation screen

**Backend tasks (Mode 2):**

1. **New endpoint: `POST /api/folders/create`** — Creates a folder transfer object, returns folder ID + decryption key
2. **New endpoint: `POST /api/folders/{folder_id}/files`** — Adds a file to an existing folder (presigned S3 URL for the individual file)
3. **New endpoint: `GET /api/folders/{folder_id}/manifest`** — Returns the encrypted directory manifest
4. **New endpoint: `PUT /api/folders/{folder_id}/manifest`** — Updates the manifest (with version check for Mode 3)
5. **Storage layout** — Individual files at `folders/{folder_id}/{file_derived_id}`; manifest at `folders/{folder_id}/__manifest__`

**Frontend tasks (Mode 2):**

1. **Upload path split** — Detect "folder upload" vs. "single file" at drop time. If folder → Mode 2 upload path.
2. **Individual file encryptor** — Each file encrypted separately with its own IV. Same key as the folder.
3. **Manifest builder** — Client-side JSON: `{ version, files: [{ name, size, type, encrypted_id, iv, created }] }`
4. **Progressive upload progress** — Show individual file progress bars as files upload. Not all-or-nothing.
5. **Mode toggle** — "Make this folder editable" checkbox on upload creation. Saves mode to manifest metadata.

**Frontend tasks (Browse, Mode 2):**

1. **Manifest fetcher** — When a folder transfer is opened, fetch manifest first, then load files on demand.
2. **File list renderer** — Simple list view (not zip viewer). Files load individually as the user clicks.
3. **Thumbnail prefetch** — For image files, preload thumbnails progressively.

**Effort estimate:** Backend ~3 days, Frontend upload path ~3 days, Browse path ~2 days = ~8 days total for Mode 2 MVP.

### Stream 3: Photo Gallery Component (Doc 231, Phase B)

**Priority:** Medium-Low — dependent on Mode 2 working
**Effort:** Medium (4–5 days)
**Dependencies:** Mode 2 MVP working, IFD component scaffold for `sg-photo-gallery`

**Tasks:**

1. **`sg-photo-gallery` component** — Grid of thumbnails (CSS grid, aspect ratio 1:1). Responsive.
2. **Lightbox** — Full-screen single-photo view, keyboard navigation (← → ESC).
3. **Upvote button** — Per-photo vote toggle (filled/unfilled heart). Writes to `__votes__.json`.
4. **Vote aggregation** — On load, fetch `__votes__.json`, merge into photo metadata. Sort by vote count.
5. **Filter by contributor** — Show/hide photos by uploader identity (from manifest metadata).
6. **Sort controls** — Date, name, vote count.

**`__votes__.json` format:**
```json
{ "version": 1, "votes": { "{file_derived_id}": { "count": 3, "voted_by": ["anon-1", "anon-2"] } } }
```

**Race condition handling:** For v0.1, last-write-wins. Include `version` field in the votes file. Log conflicts but do not block.

### Stream 4: Mobile Upload Bugs (Doc 231, prerequisite)

**Priority:** High — these are user-reported bugs, independent of Mode 2
**Effort:** Small (1 day)
**Dependencies:** None

**Tasks:**

1. **Apple Photos multi-file drop detection** — Wrap the `drop` handler to detect `DataTransferItem` entries that have `type: ""` (empty MIME type, common for Photos app items on iOS). Show: *"Multi-file drop from Apple Photos isn't supported in this browser. Please use the file picker button instead."*
2. **Cloud-synced file detection** — After reading a `FileList`, attempt `FileReader.readAsArrayBuffer()` on each file. If a `FileReader` error fires (or `file.size === 0` for an undownloaded iCloud file), show: *"This file isn't downloaded to your device yet. Please download it from iCloud first."*
3. **File picker fallback** — Ensure the fallback "select files" button is always visible and clearly labelled when a drag-and-drop failure occurs.
4. **Help page content** — Create a simple help page in the user UI: `en-gb/help/upload-from-photos/index.html`. Step-by-step for Apple Photos, Google Photos, iCloud Drive. Linked from the error messages above.

### Stream 5: Fractal IFD Adoption for New Send Components (Doc 229)

**Priority:** Low (process/architecture, not a feature)
**Effort:** Negligible (follow-on to existing IFD practice)

The fractal IFD model from doc 229 is already applied to existing Send components. New components built for Modes 2/3 should follow it explicitly:

- `sg-photo-gallery` — own IFD, own manifest.json
