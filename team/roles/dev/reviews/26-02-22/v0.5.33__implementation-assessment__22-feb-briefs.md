## Brief 1: PKI-Keyed Personal Data Vault

### Implementation Verdict: Ready to Build

The vault brief is well-specified and builds entirely on existing infrastructure. No new API routes. No new backend services. The cache service already supports the CRUD operations needed.

### What Exists Today

| Component | Status |
|-----------|--------|
| Cache service (`POST/GET/PUT/DELETE /api/cache/{id}`) | Deployed, tested |
| PKI key pair generation (Web Crypto API) | Deployed in admin UI v0.1.2 |
| AES-256-GCM encryption (client-side) | Deployed in transfer flow |
| IndexedDB storage for keys | Deployed in pki-manager.js |
| Admin UI v0.1.6 with vault components | Merged to dev (vault-manager.js, vault-editor.js, vault-preview.js) |

### What Needs Building

| Component | Effort | Notes |
|-----------|--------|-------|
| **Vault root derivation** (client JS) | Small | `sha256(publicKeyHash + "/filesystem")` — straightforward crypto |
| **Folder JSON CRUD** (client JS) | Medium | Create/read/update folder JSON via cache API. Children array management. |
| **Encrypted index** (client JS) | Medium | JSON index of names/metadata, encrypted with derived symmetric key, stored at deterministic cache ID |
| **Index sync** (client JS) | Medium | Fetch → decrypt → cache in IndexedDB → re-encrypt → upload on change |
| **Vault UI page** (`/admin/vault.html`) | Medium | File manager UI — partially exists in v0.1.6 vault-manager.js (1838 lines) |
| **Symmetric key derivation** | Small | Derive AES key from PKI key pair for vault encryption |

### Architecture Observations

1. **No backend changes needed.** The vault is a client-side abstraction over the cache service. The server doesn't know it's serving a file system. This is elegant and low-risk.

2. **Vault-manager.js already exists** (1838 lines in v0.1.6). Need to assess how much of the brief's spec is already implemented vs. what needs to be added.

3. **Concurrency risk:** If the user opens the vault from two browsers simultaneously, the index could conflict. The brief doesn't address this. For MVP, last-write-wins is acceptable. Future: add version counter to index.

4. **Cache entry TTL:** Current cache entries may have TTL. Vault entries must be persistent. Verify the cache service supports indefinite storage or add a `permanent: true` flag.

5. **Storage quotas:** No mention of per-user storage limits. For MVP, trust the cache service quotas. Future: vault manifest should track total storage used.

### Estimated Effort

| Phase | Work |
|-------|------|
| Phase 1: Vault CRUD (client-side) | 2–3 sessions |
| Phase 2: Encrypted index | 1–2 sessions |
| Phase 3: Vault UI integration | 1–2 sessions |
| Phase 4: Tests (in-memory cache) | 1 session |
| **Total** | **5–8 sessions** |

## Brief 2: Reference Implementations — LLM-Consumable Libraries

### Implementation Verdict: Documentation-First, Minimal Code

This is primarily a process and documentation pattern, not a code implementation. The "implementation" is:

1. Create a reference repo (e.g., `ref-admin-ui`)
2. Move canonical admin UI code there
3. Write 6 documentation files (ARCHITECTURE, COMPONENTS, IMPLEMENTATION_GUIDE, CHANGELOG, TESTING_GUIDE, ADAPTATION_NOTES)
4. Add version pinning config to sub-projects

### What Needs Building

| Component | Effort | Notes |
|-----------|--------|-------|
| Reference repo structure | Small | Standard repo with docs/ and src/ |
| Documentation (6 files) | Medium | Needs deep knowledge of admin UI internals |
| Version pinning config | Small | JSON in sub-project configs |
| Agent persona (context.md) | Small | For reference implementation agent |

### Developer Observations

1. **The hard part is documentation quality**, not code. The brief explicitly says docs are "over-indexed for LLM comprehension." This means the Librarian/Sherpa should lead, with Dev providing technical accuracy.

2. **First candidate (admin UI) is the right choice.** The admin panel pattern is shared across 4+ projects. IFD web components, message panel, debug panel, event bus — all well-defined, all duplicated.

3. **Risk: moving code to a reference repo could break existing projects.** The code stays in each project — the reference is additive, not extractive. No migration needed.

### Estimated Effort

| Phase 2: Write 6 documentation files | 2–3 sessions |

## Brief 3: Receiver Experience and Trust Spectrum

### Implementation Verdict: Significant New Capability — Phased Approach Required

This is the most complex brief of the three. It introduces new API endpoints, a new auth flow, self-destruct logic, environmental factor collection, and a progressive trust model. This is NOT a single-session implementation.

### What Needs Building

#### Server-Side (New)

| Component | Effort | Notes |
|-----------|--------|-------|
| **Short code generation** | Medium | 6-digit/alphanumeric code mapped to decryption key. Storage with TTL. |
| **Short code validation** | Medium | One-use check, attempt counting, self-destruct trigger. |
| **Self-destruct mechanism** | High | Delete code AND associated key on failure. **Must be atomic** — race condition risk flagged in brief. |
| **Attempt tracking** | Medium | Per-code attempt counter with configurable limit (1–3). |
| **TTL management** | Medium | Configurable expiry (1h, 24h, 7d). Background cleanup or lazy deletion. |
| **Environmental factor endpoints** | Medium | Receive timezone/country/browser/language from client, validate against sender constraints. |

#### Client-Side (New)

| Component | Effort | Notes |
|-----------|--------|-------|
| **Short code entry UI** | Medium | Clean, friendly, confidence-inspiring. Multiple format support (numeric, words, phrase). |
| **Environmental factor collection** | Small | Browser timezone, Accept-Language, User-Agent — already available in JS. |
| **PKI auto-generation for recipients** | Medium | Transparent key pair creation on first visit. Notification UI. |
| **Reply mechanism** | Medium | "Reply" button in data room, encrypts with sender's public key, attaches recipient's public key. |
| **Trust tier indicators** | Small | Visual cues for Lobby/Member/Verified/Admin. |

#### API Surface

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/api/codes` | POST | Generate short code for a transfer/data room |
| `/api/codes/{code}/validate` | POST | Validate code, return decryption key if valid |
| `/api/codes/{code}` | DELETE | Manually revoke a code |
| `/api/codes/{code}/status` | GET | Check code status (active, expired, destroyed) |

### Architecture Observations

1. **Self-destruct atomicity is the hardest problem.** Two concurrent requests to `/api/codes/{code}/validate` could both pass the attempt check before either increments the counter. Solutions: DynamoDB conditional writes, or Redis INCR (atomic), or optimistic locking on S3. Given the project uses Memory-FS, this needs careful design.

2. **Environmental factors as KDF input is elegant but adds complexity.** If timezone/country/browser are part of the key derivation, the sender must specify expected values at creation time. Mismatch = wrong key = decryption failure (not an API error). This is secure but could confuse users. Recommend: environmental factors as **validation gates** (reject before returning key) rather than **KDF inputs** for MVP.

3. **Two-message key exchange is achievable with existing PKI infrastructure.** The PKI key pair generation already exists. Attaching public keys to messages is a metadata addition. The reply mechanism is the main new UI work.

4. **Data room access tiers are future work.** The brief describes Lobby/Member/Verified/Admin but the immediate implementation is just the short code (Lobby tier). The architecture should accommodate tiers but only Lobby needs building now.

5. **Storage backend for short codes.** Short codes are ephemeral (self-destruct, TTL). They don't fit the same storage pattern as transfers. Consider a separate storage namespace or a lightweight in-memory store with TTL.

### Recommended Phasing

| Phase | What | Priority |
|-------|------|----------|
| **Phase 1: Short codes** | Generate 6-digit code, validate, single-use, TTL expiry | P1 — immediate |
| **Phase 2: Self-destruct** | Attempt counting, atomic self-destruct on failure | P1 — with Phase 1 |
| **Phase 3: Receiver UI** | Short code entry page, clean landing experience | P1 — user-facing |
| **Phase 4: Environmental factors** | Timezone/country validation gates (not KDF) | P2 — enhances security |
| **Phase 5: PKI auto-generation** | Transparent key pair for new recipients | P2 — establishes trust |
| **Phase 6: Reply + key exchange** | Two-message PKI key exchange | P2 — completes trust cycle |
| **Phase 7: Trust tiers** | Lobby/Member/Verified/Admin model | P3 — future |
| **Phase 8: TOTP** | Authenticator app integration | P3 — future |

### Estimated Effort

| Phase | Work |
|-------|------|
| Phases 1–3 (MVP short codes + UI) | 4–6 sessions |
| Phases 4–6 (environmental + PKI exchange) | 3–5 sessions |
| Phases 7–8 (tiers + TOTP) | Future — scope TBD |
| **Total for MVP (Phases 1–3)** | **4–6 sessions** |
