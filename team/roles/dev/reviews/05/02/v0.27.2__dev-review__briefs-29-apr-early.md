## Stream 1: Ephemeral Infrastructure Next Phase (Dev brief, Doc 334)

**Recipient team:** Explorer (Dev + DevOps)
**Depends on:** Day 48 deliverables — serverless provisioning, plugin architecture, fractal frontend

### Implementation Streams (Priority Order)

**P1 — UI Split: Creation vs. Live Instances**

Effort: ~2 days. This is the foundation everything else builds on. Until the UI is split, adding new creation modes and instance types creates layout conflicts.

Implementation:
- Create two top-level sg-layout panels in the infra UI
- Move existing creation controls to "Creation" panel
- Move existing running instance display to "Live Instances" panel
- Acceptance: AC-1 from brief (two distinct panels)

**P2 — Three Creation Modes**

Effort: ~3 days. The CLI and API already support three modes (per brief). This is a UI wiring task:
- Add radio group: Fresh build / From AMI / (Bake AMI — separate action)
- When "From AMI" selected, show AMI picker dropdown
- Wire to existing API
- Acceptance: AC-2, AC-3

**P3 — AMI Management**

Effort: ~4 days. This is the "simulated marketplace catalogue":
- API: `GET /amis/{instance-type}` — list with metadata (date, source, size)
- API: `POST /amis/bake` — create AMI from running instance (async, progress indicator)
- API: `DELETE /amis/{ami-id}` — delete an AMI
- API: `PUT /amis/{ami-id}/default` — set as default for instance type
- UI: AMI management sub-panel within Creation section
- Note: baking takes 5–10 minutes — needs async status polling
- Acceptance: AC-4

**P4 — Remote Shell (Option A)**

Effort: ~2 days. Simpler than it looks because each instance FastAPI already exists:
- Add `POST /shell/execute` endpoint to instance FastAPI — runs command, returns stdout/stderr/exit_code
- Add `POST /shell/stream` endpoint — WebSocket, streams command output
- UI: terminal component (textarea + input field, basic ANSI rendering)
- Note: do NOT implement SSH or SSM — Option A only for MVP
- Acceptance: AC-8

**P5 — Prometheus Endpoints**

Effort: ~1 day. Standard instrumentation:
- Add `GET /metrics` to instance FastAPI
- Collect: CPU, memory, disk, uptime, request count, container count (if Docker available)
- Use `prometheus_client` Python library (exposition format, not JSON)
- UI can poll this endpoint with `setInterval` and display inline (no Prometheus server needed)
- Acceptance: AC-9

**P6 — SG/Send Vault Server Instance Type**

Effort: ~5 days (requires coordination with DevOps for AMI creation):
- New plugin under `plugins/vault-server/`
- Plugin config: `{ "type": "sg-vault", "from_ami": true }`
- Instance FastAPI exposes vault endpoints (proxying to the running sgraph-send service)
- Other instances can target this server via `VAULT_SERVER_URL` env var at launch
- Need a vault-server AMI with sgraph-send pre-installed
- Acceptance: AC-5, AC-6

**P7 — Docker Container Management**

Effort: ~5 days:
- Docker socket must be accessible to instance FastAPI (check `/var/run/docker.sock`)
- `GET /containers` — list running containers with status
- `POST /containers/start` — start from image catalogue
- `POST /containers/{id}/stop`
- `GET /containers/{id}/logs` — tail recent logs
- `POST /containers/{id}/ports` — expose port (update security group or route through FastAPI)
- For Playwright multi-container: sequential port assignment from a registered range
- Acceptance: AC-7

**P8 — Stacks**

Effort: ~8 days (highest complexity):
- Define `Stack__Config` Type_Safe model
- Stack launcher: parse config, launch instances in dependency order, inject peer connections
- API: `POST /stacks/launch`, `GET /stacks/{id}`, `DELETE /stacks/{id}`
- UI: stack catalogue + launch confirmation + progress view
- Phase 1: all instances in same security group (no explicit orchestration)
- Partial failure handling: if any instance fails, report which failed and which succeeded
- Acceptance: AC-10, AC-11

**Total ephemeral infra stream estimate:** ~30 developer-days if all features implemented in sequence. Recommend phasing across multiple sessions: P1–P3 first, then P4–P6, then P7–P8.

**Step 4 — Browser view** (~2 days):

- Fallback: determine if Neko/noVNC is needed (test iframe first)

**Step 5 — Health checks** (~1 day):
- Container running check (Docker)

**Step 6 — Vault session storage** (~1 day):

## Stream 3: SGit Four-Layer Refactoring (Villager brief, Doc 336)

**Recipient team:** Villager (Dev — SGit-AI/SGit-AI__CLI repo)
**Note:** This is a separate repo. These estimates are for the SGit Villager team, not for this repo.
**Goal:** No new features. Restructure codebase, add transaction logging, decompose actions into steps.

### Phase 1 — Folder Structure Migration (~3 days)

Create the four-layer folder structure:
```
sgit/crypto/, sgit/core/, sgit/network/, sgit/plugins/, sgit/cli/
```

Move existing code into the correct layer. For each moved module:
- Verify no upward imports introduced
- Run full test suite after each module move
- Fix any circular imports

**No logic changes in Phase 1.** Only move code. This keeps the diff reviewable.

### Phase 2 — Core Action Step Decomposition (~5 days)

Decompose each Core action (commit, push, pull, init, branch, merge) into typed steps:
- Define `Step__Input` and `Step__Output` type-safe models per step
- Each step function: `(input: StepInput) -> StepOutput`
- Step functions are pure (no side effects except the disk writes they're responsible for)
- Step timing: wrap each call with `time.monotonic()` before/after

Start with `push` (as shown in the brief example — 6 steps). Then `pull`, then `commit`.

### Phase 3 — Transaction Logging (~2 days)

After Phase 2, the step outputs are all typed. Transaction logging is composition:
- After each Core action completes, assemble the transaction record from step outputs
- Write to local append-only log: `~/.sgit/logs/transactions.jsonl`
- Log both successful and failed transactions (include `error` field for failures)

### Phase 4 — Plugin Feature Flags (~1 day)

- Add `plugins` block to sgit config (`.sgit/config.json` or `~/.sgitrc`)
- Each plugin entry: `{ "enabled": true/false, "stability": "stable|experimental" }`
- CLI command registration: skip registration if `enabled: false`
- Test: disable a plugin, verify its CLI command returns "not found" or is absent

### Phase 5 — Verification and Integration Tests (~3 days)

- Full integration test suite passes after all phases
- Crypto tests pass with Core, Network, Plugins removed from sys.path
- Replay test: execute a commit transaction record against fresh vault, verify identical state
- Performance baseline: capture per-step timing for push/pull, establish benchmark

**Total SGit refactoring estimate:** ~14 developer-days. This is Villager work — no shortcuts on test coverage. Every Core action needs typed step tests.

## Implementation Recommendations

1. **Before starting infra next-phase:** Audit what actually EXISTS from Day 48 (serverless provisioning, plugin architecture, fractal frontend). Check the code, not the brief. The Librarian reality document should be updated once verified.

2. **Start with P1 (UI split):** Every other infra feature needs the split UI to slot into. This is the mandatory first step.

3. **Firefox plugin can proceed in parallel** with infra once the plugin architecture interface is stable (Day 48 base verified).

4. **SGit refactoring is Villager-only** — Explorer should not touch the SGit codebase during this period. Concurrent Explorer additions would increase regression risk.

5. **Use Type_Safe throughout** — every new model (AMI_Config, Stack_Config, TransactionRecord, StepInput/Output) must extend `Type_Safe` from osbot-utils. No Pydantic, no dataclasses.
