## Stream A: SGit Four-Layer Refactoring (Doc 336)

### Summary

Structural refactoring of the SGit CLI codebase into four clean layers. No new user-facing features. All existing tests must pass after each phase.

### Implementation Plan (Phased)

**Phase 1: Folder structure + code migration (2–3 dev-days)**

Create `crypto/`, `core/`, `network/`, `plugins/` directories. Move existing code to the correct layer without logic changes. Run full test suite after each move. This phase is mechanical but must be thorough — every import must be updated.

Key check: Crypto layer must have zero imports from Core, Network, or Plugins after this phase.

**Phase 2: Typed step decomposition for push, pull, commit (3–4 dev-days)**

Decompose the three highest-value commands into typed steps. Each step: input object in, output object out, independently testable.

Priority: `sgit push` first (highest user-visible latency), then `sgit pull`, then `sgit commit`.

**Phase 3: Transaction logging (2 dev-days)**

Each Core action produces a JSON transaction record. Log to a configurable location (default: alongside vault data — see Decision #30).

Fields: `action`, `timestamp`, `vault_id`, `branch`, `duration_ms`, plus action-specific fields (files added/modified, new commit hash, etc.).

**Phase 4: Plugin feature flags (1 dev-day)**

`config.json` enables/disables plugins. Disabled plugins: CLI commands not registered, code not loaded.

### Total Estimate

8–10 dev-days for all four phases. Can be parallelised with PKI work after Phase 1.

### Risks

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Import graph is more complex than expected | Medium | Run static import analysis before starting Phase 1 |
| Test suite exercises cross-layer boundaries that will change | Medium | Identify boundary tests before refactoring; update them explicitly |
| Transaction log introduces performance regression | Low | Add per-step timing; measure baseline before Phase 3 |

## Stream B: Ephemeral Infra Next Phase (Doc 334)

### Summary

7 features extending the ephemeral infrastructure MVP. All build on the plugin architecture from 28 Apr.

### Prioritised Feature Breakdown

**P1 — Split creation from live instances (1–2 dev-days)**

Two separate UI panels. Creation panel: instance type selector, creation mode (fresh/AMI/bake), size, timeout, launch button. Live instances panel: running instance cards with status, cost, action buttons.

This is a UI refactoring of an existing view, not a new feature. Frontend-only.

**P1 — AMI management (2–3 dev-days)**

List AMIs per instance type, bake from running instance, delete old AMIs, set default per type. Backend: API endpoints on the control plane Lambda. Frontend: management table in creation panel.

Acceptance criteria includes "Create from AMI boots Elastic in under 90 seconds" — this requires an existing AMI to be available in the test environment.

**P2 — SG/Send vault server instance type (2–3 dev-days)**

New plugin under `plugins/vault-server/`. Launches an EC2 instance running the SGraph Send Docker container (from ECR). Other instances in the same network can read/write vaults on this instance.

Requires confirming the SGraph Send container image is available in ECR and the security group rules allow inter-instance traffic on the vault API port.

**P2 — Docker container management inside instances (2–3 dev-days)**

Instance FastAPI gains Docker management endpoints: list containers, start/stop/restart, view logs, expose ports. Requires Docker socket access from the FastAPI container.

Implementation note: FastAPI must run with `--privileged` or docker socket mount (`-v /var/run/docker.sock:/var/run/docker.sock`). Confirm this is acceptable in the instance AMI setup.

**P2 — Remote shell — Option A (1–2 dev-days)**

`POST /shell/execute` and `POST /shell/stream` (WebSocket) on instance FastAPI. UI renders a terminal component that sends commands and displays output.

Option A is deliberately minimal. No tab completion, no curses. Document this limitation clearly in the UI. A future Option B (SSM proxy) can replace it if users need interactive terminals.

**P3 — Prometheus endpoints (1 dev-day)**

`GET /metrics` on instance FastAPI returns Prometheus text format. Metrics: CPU, memory, disk, uptime, container count, request count. Use `prometheus_client` Python library if available; otherwise hand-roll the text format (it's simple).

**P3 — Stacks (3–4 dev-days, Architect recommends Phase 1 only)**

Phase 1 (MVP): JSON stack definition triggers sequential instance launches. No VPC coordination. Stack launch = loop over instance definitions + call individual launch endpoint for each.

Skip Phase 2 (VPC/security group coordination) until Phase 1 is validated with real use.

### Total Estimate

10–14 dev-days for all 7 features at the recommended priority.

### Recommended Sequence

1. Split creation/live UI (P1, UI only, low risk)
2. AMI management (P1, unblocks fast-boot testing)
3. Remote shell Option A (P2, high user value, low complexity)
4. Docker container management (P2, depends on instance FastAPI extension)
5. SG/Send vault server (P2, depends on ECR confirmation)
6. Prometheus endpoints (P3, standalone, any time)
7. Stacks Phase 1 (P3, build last, low risk after other features stable)

## Stream C: Firefox Browser Plugin (Doc 335)

### Summary

Firefox + MITM proxy as the first concrete plugin. Plugin lives at `plugins/firefox/`. Follows the same plugin structure as other instance types.

### Implementation Plan

**Step 1: Plugin scaffold (0.5 dev-days)**

Create `plugins/firefox/` with: `routes.py`, `cli.py`, `config.json`, `tests/`. Follow exact same structure as existing plugins.

**Step 2: Instance launch and health checks (1–2 dev-days)**

Firefox Docker container launch. Health check endpoints (container running, Firefox alive, MITM proxy connected, network connectivity, login page accessible). Status display in UI panel (green/amber/red).

**Step 3: Firefox UI panel (2 dev-days)**

The Firefox-specific detail view: left side (credentials, MITM proxy controls, security settings, Firefox settings, instance controls), right side (browser view). Uses sg-layout as per fractal frontend architecture.

**Step 4: Iframe embedding (0.5–1 dev-day)**

Verify Firefox container allows iframe. If X-Frame-Options headers are present, configure the container to remove them. If not possible, integrate Neko or noVNC as fallback.

**Step 5: Vault storage integration (1 dev-day)**

Store credentials, MITM scripts, logs, and metadata in vault per-session. On reconnect, restore from vault.

**Step 6: AMI bake support (0.5 dev-day)**

The three-mode creation (fresh/bake AMI/from AMI) is inherited from the plugin architecture. Firefox-specific: ensure MITM script paths and Firefox profile data are included in the AMI image.

### Total Estimate

5–7 dev-days for complete Firefox plugin.

### Dependency

Firefox plugin depends on the ephemeral infra platform being stable (at minimum: instance launch works, Docker management works). Start Stream C after Stream B P1 and P2 features are stable.
