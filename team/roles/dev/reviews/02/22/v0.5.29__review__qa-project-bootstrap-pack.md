## Executive Summary

The QA project briefs describe a browser-automation test suite in a separate repo that (a) validates SG/Send UI flows and (b) generates living documentation from screenshots. This review assesses technical feasibility, identifies reusable patterns from the main codebase, and flags decisions that need to be made early.

**Assessment: Feasible and well-scoped.** The brief's "pipeline before tests" phasing is correct. The main risk is browser automation environment support (Lambda + Claude Code), which the brief already acknowledges as P2/P3.

## Technical Decisions Flagged

### Decision 1: Browser Automation Tool

The brief recommends evaluating Vercel Agent Browser first, Playwright as fallback. From a Dev perspective:

| Factor | Vercel Agent Browser | Playwright |
|---|---|---|
| Maturity | Newer (agent-focused) | Industry standard |
| Python support | TBD (research pending) | Excellent (`playwright` pip package) |
| Screenshot API | TBD | Built-in (`page.screenshot()`) |
| Visual comparison | TBD | Built-in (`expect(page).to_have_screenshot()`) |
| GitHub Actions | TBD | First-class support |
| Lambda support | TBD | Possible with `playwright-aws-lambda` layer |
| CI integration | TBD | `npx playwright install --with-deps` |

**Dev recommendation:** Start with Playwright for the first smoke test (guaranteed to work in GitHub Actions and locally). Evaluate Vercel Agent Browser in parallel. The test abstraction layer should allow swapping the browser backend later.

### Decision 2: Screenshot Diff Strategy

The brief proposes pixel-level comparison with a 1% threshold. Two approaches:

| Approach | Pros | Cons |
|---|---|---|
| **Pixel diff (Pillow/ImageMagick)** | Simple, no dependencies beyond Pillow | Sensitive to font rendering, anti-aliasing across OS/versions |
| **Playwright built-in** | Handles anti-aliasing, threshold configurable, generates diff images | Ties you to Playwright |

**Dev recommendation:** Use Playwright's built-in `toHaveScreenshot()` for comparison. It generates diff images automatically and handles anti-aliasing. If the team later switches to Vercel Agent Browser for capture, the comparison can still use Playwright's image comparator as a standalone utility.

### Decision 3: Where to Put the Bootstrap Pack

The meta-brief asks for a single uploadable directory. Options:

| Location | Pros | Cons |
|---|---|---|
| `library/sgraph-send/dev_packs/v0.5.29__qa-project-bootstrap/` | Follows existing dev_pack pattern | Slightly deep path |
| `team/roles/dev/reviews/02/22/qa-bootstrap-pack/` | Close to this review | Not the standard dev_pack location |

**Dev recommendation:** `library/sgraph-send/dev_packs/v0.5.29__qa-project-bootstrap/` — consistent with the CloudFront and PKI dev packs.

### Decision 4: Test Target URLs

The brief mentions testing against `https://send.sgraph.ai/` (production) and `localhost`. The QA project needs a clear target strategy:

| Target | When | Token Source |
|---|---|---|
| `localhost:10062` (user) + `localhost:10061` (admin) | Local dev, CI with local server | Env var or test token |
| `https://send.sgraph.ai/` | Smoke tests against production | Pre-provisioned test token |
| `https://dev.send.sgraph.ai/` | Post-deploy validation | Dev environment token |

**Dev recommendation:** Default to localhost in CI (start SG/Send as a subprocess, run tests, tear down). Production smoke tests as a separate workflow triggered on release.

### Decision 5: Token for Testing

The QA project needs a valid access token to test the authenticated flows. Options:

| Approach | Security | Practicality |
|---|---|---|
| Hardcoded test token in config | Low — token in repo | Simple |
| GitHub Actions secret | Medium — token in CI secrets | Good for CI |
| Admin API creates token per test run | High — ephemeral tokens | Best, but requires admin access |

**Dev recommendation:** GitHub Actions secret for CI. For local dev, `.env` file (gitignored). For production smoke tests, a dedicated long-lived test token with limited uses.

## Architecture Notes for Bootstrap Pack

### User Lambda Token Flow (for Test Design)

From codebase analysis, the user lambda token validation works as follows:

1. User enters token in `<send-access-gate>` component
2. Component stores token in `localStorage['sgraph-send-access-token']`
3. `api-client.js` reads from localStorage, sends as `x-sgraph-access-token` header
4. `Routes__Transfers.check_access_token()` validates via HTTP call to admin lambda's `/tokens/lookup/{token_name}`
5. Invalid token → 401 response → UI shows red error banner
6. Valid token → 200 response → UI transitions to upload view

The smoke test must handle this flow: enter token in input field, click Go, wait for either error or page transition.

### Admin Lambda Auth (for Test Design)

Admin uses global API key authentication:
- Header name from env var (typically `X-API-Key`)
- Value from env var
- Missing/invalid → 401
- Cookie auth also supported (set via `/set-cookie` endpoint)

The admin tests need to either:
- Set the cookie before navigating (via direct API call to `/set-cookie`)
- Or inject the token into the browser's cookie jar

### Key UI Selectors (from Reference Screenshots)

From the 4 reference screenshots, the QA test should target:

| Element | Selector (estimated) | State |

| Access token input | `input[placeholder="Paste your access token"]` | Landing page |
| Go button | Button with text "Go" | Landing page |

| File/Text tabs | Tab elements "File" / "Text" | After valid token |
| Drop zone | Element containing "Drop your file here" | After valid token |
| Usage counter | Element containing "uses remaining" | After valid token |
