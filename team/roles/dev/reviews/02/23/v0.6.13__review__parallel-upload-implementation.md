## 1. Code Quality Assessment

### The Set + Promise.race Pool Pattern

```js
const active  = new Set();
const maxPool = SendUpload.PARALLEL_UPLOADS;
for (let i = 0; i < partUrls.length; i++) {
    const p = uploadOnePart(i).then(() => active.delete(p));
    active.add(p);
    if (active.size >= maxPool) await Promise.race(active);
}
await Promise.all(active);
```

This is the canonical concurrency-limited pool pattern in modern JavaScript. It is widely used in production codebases (Node.js `p-limit`, Deno standard library, etc.) and has well-understood behaviour.

**Strengths:**

- Minimal code (7 lines for the pool logic)
- No external dependency needed — the pool is self-contained
- `PARALLEL_UPLOADS` is a static class constant, easily tunable
- The `.then(() => active.delete(p))` cleanup is correct — the promise removes itself from the active set upon resolution, freeing a slot for the next iteration
- `Promise.all(active)` at the end correctly drains any remaining in-flight uploads (up to `maxPool - 1` residual promises)

**Observation:** The `p` variable captures correctly because `const` in the loop body creates a fresh binding per iteration. This is a subtle but important detail — if `var` were used, the `.then()` closure would capture a shared mutable reference and the pool would break.

## 2. Correctness: `completedParts[i]` Indexed Assignment

```js
const completedParts = new Array(partUrls.length);
// ...
completedParts[i] = { part_number: partUrls[i].part_number, etag };
```

**Correct.** Parts complete in arbitrary order (part 3 may finish before part 1), but each writes to its own index `i`. Since `i` is the loop variable captured by the async closure, every part writes to a unique slot. There is no interleaving risk.

After the pool drains, `completedParts` is a dense array with entries at indices `0..numParts-1`, each containing the correct `part_number` and `etag`. The `completeMultipart` call receives them in part-number order, which S3 requires.

**No issue here.**

## 3. Thread Safety: `partsCompleted++`

```js
partsCompleted++;
const partPct = Math.round(70 + (20 * partsCompleted / partUrls.length));
```

**Safe.** JavaScript is single-threaded with a cooperative event loop. The `await` inside `uploadOnePart` yields to the microtask queue, but the `++` and subsequent DOM update run as a synchronous block after the `await` resumes. Two `uploadOnePart` calls cannot interleave within the `partsCompleted++; ... label.textContent = ...` sequence.

There is no race condition. The counter will reach exactly `partUrls.length` when all parts complete.

## 4. Memory: `encrypted.slice(start, end)` Creates Copies

```js
const partBuf = encrypted.slice(start, end);
```

`ArrayBuffer.prototype.slice()` creates a **new copy** of the byte range. With `PARALLEL_UPLOADS = 5` and a default part size of 10 MB, the peak additional memory from sliced buffers is approximately:

| Scenario | encrypted size | Extra memory from slices | Total peak |
|----------|---------------|------------------------|------------|
| 50 MB file (5 parts) | 50 MB | 50 MB (all 5 in flight) | ~100 MB |
| 200 MB file (20 parts) | 200 MB | 50 MB (5 in flight) | ~250 MB |
| 1 GB file (100 parts) | 1 GB | 50 MB (5 in flight) | ~1.05 GB |

For the 1 GB `MAX_FILE_SIZE_PRESIGNED` limit, the browser already holds the full `encrypted` ArrayBuffer (~1 GB). The 50 MB of concurrent slices is a ~5% overhead — acceptable.

**Risk surfaces at extreme sizes:** If the encrypted buffer is already near browser memory limits (mobile Safari caps around 1.5-2 GB total heap), the slices push closer to OOM. This is not an issue today given the 1 GB cap, but is worth noting for future size-limit increases.

**Future iteration:** Use `new Uint8Array(encrypted, start, end - start)` to create a **view** rather than a copy. Views share the underlying ArrayBuffer memory. However, this requires confirming that the `fetch()` PUT body correctly reads from the view without issues on all target browsers. The Fetch spec accepts `ArrayBufferView` as a body, so this should work.

## 5. Progress Bar Accuracy with Parallel Completions

```js
partsCompleted++;
const partPct = Math.round(70 + (20 * partsCompleted / partUrls.length));
const bar     = this.querySelector('.progress-bar__fill');
if (bar) bar.style.width = `${partPct}%`;
```

**Assessment:**

- The progress bar covers the 70%-90% range during part uploads (20% of the total bar). This is sensible — earlier stages (reading, encrypting, creating transfer) consume 0%-70%, and completion consumes 90%-100%.
- Progress is **part-granular**, not byte-granular. A 1 GB file with 100 parts updates the bar 100 times during upload. A 15 MB file with 2 parts updates only twice — a visible stall during each part upload.
- The bar may appear to "jump" because parallel completions can cluster. If 3 parts finish within the same 100ms, the user sees a single jump of 3 increments rather than 3 smooth steps.

**Acceptable for current implementation.** The UX is functional and honest (it only advances when real progress occurs).

**Future iteration:** Use `XMLHttpRequest` with `upload.onprogress` instead of `fetch()` to get byte-level progress within each part. This would allow smooth, continuous progress reporting regardless of part count.

## 6. Error Propagation: One Failed Part Aborts All

```js
// In the pool loop:
const p = uploadOnePart(i).then(() => active.delete(p));
active.add(p);
if (active.size >= maxPool) await Promise.race(active);

// ...

} catch (err) {
    await ApiClient.abortMultipart(transferId, uploadId);
    throw err;
}
```

**How it works:**

1. If `uploadOnePart(i)` rejects, the `.then()` chain also rejects (`.then()` does not catch).
2. The rejected promise stays in the `active` Set (`.delete(p)` is in the `.then()` success path, which never runs on rejection).
3. Either `Promise.race(active)` or `Promise.all(active)` will reject with the error.
4. The `catch` block calls `abortMultipart` (best-effort, swallows errors) then re-throws.
5. The outer `startUpload()` catch handler displays the error to the user.

**This is correct.** One failed part causes the entire upload to abort.

**Nuance:** When one part fails, other in-flight parts are **not cancelled** — their `fetch()` calls continue to completion (or failure) in the background. The browser will finish sending bytes for those parts even though the multipart upload has been aborted server-side. This is harmless (S3 ignores parts for an aborted upload) but wastes bandwidth.

**Residual promise in `active` Set:** The rejected promise remains in the `active` Set. Since the `catch` block throws, control exits `_uploadViaPresigned` entirely, so the Set is garbage-collected. No leak.

## 7. Edge Cases

### 7a. Only 1 Part (file slightly over 5 MB, under 10 MB)

- `numParts = 1`, `partUrls.length = 1`
- Pool loop runs once: `i = 0`, one promise added to `active`, `active.size = 1 < 5`, so `Promise.race` is **not** awaited inside the loop
- `Promise.all(active)` waits for the single part
- **Works correctly.** The pool degrades gracefully to a single upload.

### 7b. Parts < PARALLEL_UPLOADS (e.g., 3 parts, pool size 5)

- Pool loop runs 3 times, `active` never reaches size 5
- `Promise.race` is never called inside the loop
- `Promise.all(active)` waits for all 3
- **Works correctly.** All 3 parts launch immediately and run in parallel. This is optimal — no artificial serialisation.

### 7c. Parts Exactly Equal to PARALLEL_UPLOADS (5 parts, pool size 5)

- All 5 launch. On iteration `i=4`, `active.size = 5 >= 5`, so `Promise.race` is awaited.
- As soon as one completes, the loop ends (no more iterations). `Promise.all` drains the rest.
- **Works correctly.**

### 7d. 0 Parts (degenerate — `encrypted.byteLength = 0`)

- `Math.ceil(0 / partSize) = 0`, so `numParts = 0`
- `initResult.part_urls` would be an empty array
- `completedParts = new Array(0)` (empty)
- Pool loop body never executes
- `Promise.all(active)` resolves immediately (empty Set)
- `completeMultipart` called with empty parts array
- **Depends on server behaviour.** The server should reject a 0-part completion. In practice, a 0-byte encrypted buffer is impossible (AES-GCM adds at least IV + auth tag), so this path is unreachable. No action needed.

### 7e. Network Disconnect Mid-Upload

- One or more `fetch()` calls reject with `TypeError: Failed to fetch`
- Error propagates as described in section 6
- `abortMultipart` also fails (network is down) but is swallowed by the `catch(e) {}` in `ApiClient.abortMultipart`
- User sees the error message
- **Correct behaviour.** The orphaned S3 multipart upload will be cleaned up by the server's lifecycle policy.

## 8. Additional Observations

### 8a. `uploadPart` Returns Empty String on Missing ETag

```js
return response.headers.get('ETag') || '';
```

S3 always returns an ETag header for successful PUT requests. If the header were somehow missing, an empty string would be passed to `completeMultipart`. S3 would reject the completion call with an `InvalidPart` error. This is a safe fallback — it surfaces the problem rather than silently corrupting data.

### 8b. `abortMultipart` Best-Effort Pattern

```js
async abortMultipart(transferId, uploadId) {
    try {
        await fetch(...);
    } catch (e) { /* best-effort cleanup */ }
}
```

Correct for a cleanup path. The abort is a courtesy to free S3 resources early. If it fails, S3 lifecycle rules clean up incomplete multipart uploads (typically after 7 days). The original error from the failed part is preserved and re-thrown.

### 8c. No Abort Signal for In-Flight Fetches

The `fetch()` calls in `uploadPart` do not use `AbortController`. When one part fails, other in-flight parts continue until they naturally complete or fail. This means:

- Wasted bandwidth on the remaining parts (up to 4 parts x 10 MB = 40 MB)
- The user sees the error immediately (good), but network activity continues in the background (cosmetic issue)

## 9. Suggestions for Future Iterations

| # | Suggestion | Effort | Impact |
|---|-----------|--------|--------|
| 1 | **Per-part retry with exponential backoff** — Wrap `uploadOnePart` in a retry loop (2-3 attempts) before failing the entire upload. Transient S3 errors (503 SlowDown) are common under load. | Medium | High — reduces user-visible failures |
| 2 | **XHR for byte-level progress** — Replace `fetch()` with `XMLHttpRequest` in `uploadPart` to access `upload.onprogress`. Pass a callback to report bytes-uploaded-so-far per part. Aggregate across all active parts for smooth progress. | Medium | Medium — UX polish for large files |
| 3 | **AbortController support** — Create one `AbortController` per part, store in a Map. On first failure, abort all in-flight controllers. Add a user-facing "Cancel" button wired to the same controllers. | Medium | Medium — saves bandwidth, enables cancel |
| 4 | **ArrayBuffer views instead of copies** — Replace `encrypted.slice(start, end)` with `new Uint8Array(encrypted, start, end - start)` to avoid copying. Reduces peak memory by up to 50 MB. | Low | Low-Medium — matters on mobile/low-memory devices |
| 5 | **Adaptive concurrency** — Start with `PARALLEL_UPLOADS = 2`, measure per-part throughput, ramp up to 5 if latency is low. Avoids saturating slow connections. | High | Medium — better behaviour on varied networks |
| 6 | **Structured error types** — Instead of `throw new Error('Part upload failed: 503')`, throw typed errors that distinguish retryable (503, 429, network) from permanent (403, 404) failures. | Low | Medium — enables smart retry logic |
