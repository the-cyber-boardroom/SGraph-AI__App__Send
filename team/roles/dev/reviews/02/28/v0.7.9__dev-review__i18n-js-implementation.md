## What Was Built

### 1. SgI18n Module (`sgraph_ai__website/js/i18n.js`)

A vanilla JS i18n system adapted from send.sgraph.ai's i18n module, providing:

- **Async locale loading**: Fetches JSON translation files on demand
- **`data-i18n` attribute system**: Automatically translates element `textContent` or `innerHTML`
- **`data-i18n-placeholder`**: Translates input placeholder text
- **`data-i18n-title`** and **`data-i18n-aria`**: Translates title and aria-label attributes
- **Page title translation**: Via `<meta name="i18n-title-key">` in `<head>`
- **localStorage persistence**: Saves locale preference as `sgraph-locale`
- **Locale dropdown UI**: `renderLocaleSelector()` dynamically builds a dropdown with flag + locale name
- **FOUC prevention**: `body.i18n-loading { visibility: hidden }` removed after translations apply

## Design Decisions

1. **Named `SgI18n`** (not `I18n`) to avoid collision with send.sgraph.ai's module
2. **English loaded async via fetch** (not inline) — keeps all locales uniform
3. **`_basePath()` auto-detects depth** from URL to resolve `/i18n/*.json` paths
4. **HTML-aware rendering**: Keys containing `<` are set via `innerHTML`, others via `textContent`

## What Changes for URL-Based Routing

Per the Architect's decision (see `team/roles/architect/reviews/02/28/v0.7.9__arch-review__url-based-i18n-architecture.md`):

1. **`_detectLocale()`**: New method reads locale from URL path prefix
2. **Locale switcher**: Changes from `setLocale()` (JS re-render) to `window.location.href` navigation
3. **`_basePath()`**: Must account for locale prefix adding one extra directory level
4. **`init()`**: Simplified — content is already translated in pre-rendered HTML
5. **`data-i18n` preserved**: Runtime system remains as fallback layer

## Known Issues

- Architecture, agents index, and agent profile pages have incomplete `data-i18n` coverage on content elements (nav/footer only)
- Missing `meta.description.*` translation keys for SEO (needed for build script)
