## 7. Architecture Decisions Requiring Resolution

### D1. sg-layout external CDN dependency

v0.3.0 download page loads `sg-layout` from `https://dev.tools.sgraph.ai/core/sg-layout/v0.1.0/sg-layout.js`.

**Options:**
- **A) Vendor it** — download and include in `_common/js/vendor/`. Preserves zero-dependency posture.
- **B) Keep it** — accept the external dependency. Simpler but breaks IFD self-contained principle.
- **C) Remove it** — rewrite send-browse to not use sg-layout (significant work).

**Recommendation:** Option A (vendor) for the v0.3.0 consolidation. Can switch to CDN later as a conscious decision.

### D2. Symlinks vs copies for shared components

v0.3.0 uses symlinks for shared components (header, footer, locale, transparency, etc) pointing back to `v0.2.0/_common/`. This means v0.3.0 is NOT self-contained — it requires v0.2.0 to be deployed alongside it.

**Options:**
- **A) Copy** — make v0.3.0 truly self-contained by copying (not symlinking) shared components.
- **B) Keep symlinks** — acceptable during development, but deployment pipeline must resolve them.

**Recommendation:** For development, symlinks are fine. The CI/CD pipeline already resolves symlinks during S3 sync. But this should be documented in manifest.json.
