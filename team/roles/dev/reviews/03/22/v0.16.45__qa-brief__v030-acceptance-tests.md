## Objective

Build an automated acceptance test suite for v0.3.0 User UI that validates every user-facing flow end-to-end. These tests are the gate for promoting code from dev → main → prod. They run against a local stack: in-memory API backend + static file server for the UI.

## 1. Test Infrastructure Setup

### 1a. API Backend (in-memory, no external dependencies)

The User Lambda has a test server helper that starts the full API stack in-memory in ~100ms. No S3, no AWS, no network. Use the real HTTP mode (not TestClient) because Playwright needs real URLs.

```python
from sgraph_ai_app_send.lambda__user.testing.Send__User_Lambda__Test_Server import (
    setup__send_user_lambda__test_server
)

# Context manager — starts on a random port, stops on exit
with setup__send_user_lambda__test_server() as test_objs:
    api_url      = test_objs.server_url       # e.g. http://127.0.0.1:54321
    access_token = test_objs.access_token      # for token-authenticated requests
    write_key    = test_objs.write_key         # for vault write operations
```

This gives you the full transfer API: create → upload → complete → info → download, plus vault endpoints, early-access signup, and token management.

### 1b. UI Static Server

Use the same script the human runs locally:

```bash
./scripts/user__run-locally.sh
```

This serves v0.3.0 on `http://localhost:10062`. It creates a `.local-server-user/` directory mirroring the S3 structure, injects `build-info.js` with `apiEndpoint` pointing at the remote dev API.

**For QA tests, override `SGRAPH_API_ENDPOINT` to point at the local in-memory API:**

```bash
SGRAPH_API_ENDPOINT=http://127.0.0.1:{api_port} ./scripts/user__run-locally.sh
```

This means the full stack is local — no network calls, deterministic, fast.

**Alternative: programmatic static server.** If you need more control (e.g. starting/stopping per test class), you can replicate what the script does in Python:

```python
import subprocess, os

def start_ui_server(api_port, ui_port=10062):
    env = os.environ.copy()
    env['SGRAPH_API_ENDPOINT'] = f'http://127.0.0.1:{api_port}'
    proc = subprocess.Popen(
        ['bash', 'scripts/user__run-locally.sh'],
        env=env, stdout=subprocess.PIPE, stderr=subprocess.PIPE
    )
    # Wait for server to be ready (poll http://localhost:{ui_port}/en-gb/)
    return proc
```

## 3. Use Cases to Test

### UC-01: Single File Upload + Download (Happy Path)

**What the user does:** Drops a text file onto the upload zone, goes through the 6-step wizard, gets a download link, opens it, sees the decrypted content.

**Test flow:**
1. Navigate to `/en-gb/`
2. Verify upload zone is visible (drop zone, browse buttons)
3. Create a test file programmatically and feed it to the input
4. Verify step indicator advances through: Upload → Delivery → Share mode → Confirm → Encrypt & Upload → Done
5. On step 6 (Done): capture the download link
6. Navigate to the download link
7. Verify content decrypts and displays correctly

**Key assertions:**
- Step indicator shows correct step at each stage
- Progress bar appears during encryption/upload
- Download link contains the transfer ID and key in the hash
- Decrypted content matches the original file content
- Footer shows `v0.3.0`

### UC-02: Folder Upload → Gallery + Browse

**What the user does:** Uploads a folder containing images and documents, sees gallery view, switches to folder view.

**Test flow:**
1. Upload a folder with 3+ images and a markdown file
2. Verify gallery mode auto-selected (3+ images triggers gallery)
3. Verify thumbnail grid renders with correct file count
4. Click a thumbnail → verify lightbox opens with full image
5. Arrow keys navigate between images in lightbox
6. Close lightbox, click "Folder view" link
7. Verify browse view loads with folder tree on left, preview on right
8. Click a file in the tree → verify it opens in a preview tab
9. Click "Gallery view" → verify it switches back

**Key assertions:**
- Gallery header shows archive name, size, file count
- Extension badges (PNG, JPG, etc.) appear on thumbnails
- Mode switcher URLs preserve the hash fragment (key/token not lost)
- Browse folder tree shows correct hierarchy
- Browse tabs work (multiple files can be open)

### UC-03: Friendly Token (Simple Token Mode)

**This was a P0 bug — critical to test.**

**What the user does:** Uploads a file with "Simple token" share mode, copies the friendly token (e.g. `gleam-flint-6001`), pastes it in a new browser tab.

**Test flow:**
1. Upload a file, select Simple Token share mode on step 3
2. On step 6: capture the friendly token (word-word-NNNN format)
3. Open a new tab to `/en-gb/browse/#<friendly-token>`
4. Verify the token resolves: transfer info loads, content decrypts

**Key assertions:**
- Friendly token matches pattern `^[a-z]+-[a-z]+-\d{4}$`
- Token resolves to a valid transfer ID (12 hex chars)
- Decryption succeeds with PBKDF2-derived key
- No "Transfer not found" error (the P0 bug we fixed)
- Hash is cleared from URL after decrypt (key not visible in address bar)

### UC-12: Zero-Knowledge Security Verification

**This is the most important test from a product perspective.**

**Test flow:**
1. Upload a file with known plaintext content (e.g. "CANARY_STRING_12345")
2. Capture the transfer ID
3. Hit the API directly: `GET /api/transfers/download/{id}`
4. Verify the downloaded bytes do NOT contain "CANARY_STRING_12345"
5. Verify the downloaded bytes start with an IV (12 bytes) followed by AES-GCM ciphertext
6. Hit `GET /api/transfers/info/{id}` — verify no filename in the response
7. Verify the SGMETA envelope (filename) is inside the encrypted blob, not alongside it

**Key assertions:**
- Server storage contains only encrypted ciphertext
- No original filename in API responses
- No plaintext content in API responses
- Encryption is AES-256-GCM with 12-byte IV prepended

## 4. Test API Layer

For tests that need to set up transfers programmatically (without going through the UI), use the API directly:

```python
class TransferHelper:
    """Helper for creating transfers via the API (for download-side tests)."""

    def __init__(self, api_url, access_token=None):
        self.api_url = api_url
        self.headers = {}
        if access_token:
            self.headers['x-sgraph-send-access-token'] = access_token

    def create_and_complete(self, payload: bytes, content_type='application/octet-stream'):
        """Create a transfer, upload payload, complete it. Returns transfer_id."""
        # 1. Create
        create = httpx.post(f'{self.api_url}/api/transfers/create',
                            json={'file_size_bytes': len(payload),
                                  'content_type_hint': content_type},
                            headers=self.headers)
        tid = create.json()['transfer_id']

        # 2. Upload
        httpx.post(f'{self.api_url}/api/transfers/upload/{tid}',
                   content=payload,
                   headers={**self.headers, 'content-type': 'application/octet-stream'})

        # 3. Complete
        httpx.post(f'{self.api_url}/api/transfers/complete/{tid}',
                   headers=self.headers)

        return tid

    def upload_encrypted(self, plaintext: bytes, filename='test.txt'):
        """Encrypt client-side (matching the browser's Web Crypto), upload,
        return (transfer_id, base64_key) for constructing download URLs.

        NOTE: This requires pycryptodome or a Python AES-GCM implementation
        to match the browser's encryption exactly:
          - AES-256-GCM
          - 12-byte random IV prepended to ciphertext
          - SGMETA envelope wrapping (magic + JSON + content)
        """
        import os, base64, json
        from cryptography.hazmat.primitives.ciphers.aead import AESGCM

        # Generate key + IV
        key_bytes = os.urandom(32)
        iv        = os.urandom(12)
        aesgcm    = AESGCM(key_bytes)

        # SGMETA envelope
        meta_json = json.dumps({'filename': filename}).encode()
        sgmeta_magic = bytes([0x53, 0x47, 0x4D, 0x45, 0x54, 0x41, 0x00])
        meta_len  = len(meta_json).to_bytes(4, 'big')
        envelope  = sgmeta_magic + meta_len + meta_json + plaintext

        # Encrypt (IV prepended)
        ciphertext = iv + aesgcm.encrypt(iv, envelope, None)

        # Upload to API
        tid = self.create_and_complete(ciphertext)

        # Export key as base64url (matches browser's SendCrypto.exportKey)
        key_b64 = base64.urlsafe_b64encode(key_bytes).rstrip(b'=').decode()

        return tid, key_b64
```

This lets download-side tests create real encrypted transfers without the browser.

## 6. Key Technical Notes for the QA Team

**Web Crypto requires `localhost`, not `127.0.0.1`.** The AES-256-GCM encryption uses the Web Crypto API which requires a "secure context" (HTTPS or localhost). Playwright must navigate to `http://localhost:10062/`, not `http://127.0.0.1:10062/`.

**Hash fragments are client-only.** The `#transferId/key` hash is never sent to the server. Playwright can read `page.url` to inspect it, but the API server never sees it.

**Friendly token derivation is deterministic.** Given the same token string, the same transfer ID and key are always derived. You can compute expected values in Python using `hashlib.sha256` and `hashlib.pbkdf2_hmac` to cross-validate.

**SGMETA envelope format.** The encrypted blob starts with a 7-byte magic (`SGMETA\0`), 4-byte big-endian length, JSON metadata, then the actual file content. This is all inside the encrypted envelope — the server never sees it.

**The `_preview/` and `_gallery/` folders are filtered.** When testing zip uploads, any files in these folders are hidden from the user-facing tree/gallery but used for thumbnails.

**No mocks, no patches.** Per project rules, all tests use real implementations. The in-memory backend is the real code with the storage layer swapped to memory. No stubbing API responses.
