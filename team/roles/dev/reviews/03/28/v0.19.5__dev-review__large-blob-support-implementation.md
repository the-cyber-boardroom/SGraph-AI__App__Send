## Executive Summary

Phase 1 asks the server team to build 3 new endpoints for vault-scoped presigned URLs. The existing transfer presigned infrastructure (`Service__Presigned_Urls` + `Routes__Presigned`) provides an excellent template. This is primarily an adaptation task — the S3 multipart logic already exists; we need to wire it to vault auth patterns instead of transfer auth patterns.

## 4. S3 Key Scheme for Vaults

### Current vault storage path (Memory-FS / S3)
```
transfers/vault/{vault_id}/{file_id}/payload
```
(From `Service__Vault__Pointer.vault_payload_path()`)

### Presigned upload S3 key
Must write to the **same path** that `Service__Vault__Pointer` reads from. This is critical — after presigned upload completes, `/api/vault/read/{vault_id}/{file_id}` must be able to find the blob.

For S3 mode, `Storage_FS__S3` maps Memory-FS paths to S3 keys. The vault presigned service needs to produce the same S3 key that the storage backend would use:
```python
def s3_key(self, vault_id, file_id):
    path = f'transfers/vault/{vault_id}/{file_id}/payload'
    if self.s3_prefix:
        prefix = self.s3_prefix.rstrip('/') + '/'
        return f'{prefix}{path}'
    return path
```

**This is the trickiest part of the implementation** — if the S3 key doesn't exactly match what `Storage_FS__S3` uses, the normal read path won't find presigned-uploaded blobs.

### Action: Verify S3 key mapping
Before implementing, verify how `Storage_FS__S3.file__save()` maps `transfers/vault/{vault_id}/{file_id}/payload` to an S3 key. The presigned service must produce the identical key.

## 6. Risk Areas

### 6a. S3 Key Alignment (HIGH RISK)
The biggest risk is the presigned upload writing to a different S3 key than what `Service__Vault__Pointer.read()` expects. Must verify `Storage_FS__S3` key mapping before coding.

### 6b. `file_id` with Path Characters
Vault `file_id` values contain slashes (e.g., `bare/data/abc123`). The `initiate` endpoint receives `file_id` in the POST body, not the URL path — so URL encoding isn't an issue. But the S3 key must handle these correctly.

The `read-url` endpoint uses `{file_id:path}` in the URL (matching existing vault routes). This handles slashes correctly.

### 6c. Race Condition: Presigned Upload vs. Normal Write
A client could do a presigned upload to the same `file_id` that another client is writing via `/api/vault/write/`. Last write wins (S3 semantics). This is acceptable — same behavior as concurrent normal writes.

### 6d. Memory Mode Client Experience
In memory mode (local dev, tests), presigned endpoints return `presigned_not_available`. The CLI client must handle this gracefully. The spec's fallback logic (try normal path first, use presigned only when needed) naturally handles this — but the CLI team should test against a memory-mode server.
