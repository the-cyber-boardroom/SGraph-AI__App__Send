# Briefing: PDF Generation for Markdown Files in Gallery Workflow

**Version:** v0.16.26 | **Date:** 21 March 2026 | **Role:** Dev | **Status:** PROPOSED — does not exist yet

---

## Context

The gallery workflow (v0.2.13) already generates thumbnails for markdown files during upload — rendering markdown to a Canvas element and capturing a JPEG snapshot. Users can also print markdown to PDF from the gallery/lightbox view using `SgPrint` (v0.2.9), which opens a styled A4 print preview window.

**The question:** Can we generate a full PDF from markdown files *programmatically* during gallery creation, so the PDF is included in the `_gallery.{hash}/` folder alongside the thumbnails?

---

## What We Already Have

| Component | Location | Purpose |
|-----------|----------|---------|
| `MarkdownParser` | `v0.2.0/_common/js/markdown-parser.js` | Converts markdown → HTML |
| `SgPrint` | `v0.2.0/_common/js/sg-print.js` | A4-styled print preview with SG/Send branding |
| `PRINT_STYLES` | `send-download-v029.js` (also in `sg-print.js`) | Full A4 CSS: `@page`, headings, code blocks, tables |
| PDF.js (v3.11.174) | CDN-loaded in v0.2.13 | **Reading** PDFs — cannot write |
| Canvas markdown renderer | `send-upload-v0213.js` lines 168–361 | Direct canvas text rendering for thumbnails |

---

## Browser Limitation

Browsers intentionally block programmatic `window.print()` — it requires user interaction. So we **cannot** silently capture `SgPrint` output as a PDF. We need a dedicated PDF generation library.

---

## Recommended Approach: jsPDF + `.html()` Method

**Library:** [jsPDF](https://github.com/parallax/jsPDF) (~90KB, MIT license)

**How it works:**
1. Parse markdown via `MarkdownParser.parse(mdText)` → HTML string
2. Apply the same `PRINT_STYLES` CSS used by `SgPrint`
3. Call `jsPDF.html(htmlElement, options)` to render HTML → PDF pages
4. Extract as `Blob` and add to the zip as `_gallery.{hash}/pdfs/file_001.pdf`

**Would it look the same as the print preview?** Yes — same HTML from `MarkdownParser`, same CSS from `PRINT_STYLES`. Same fonts, same A4 layout, same heading hierarchy, same code block styling.

### Performance Estimate

| Metric | Per markdown file | Notes |
|--------|------------------|-------|
| Parse time | ~5ms | MarkdownParser already fast |
| PDF render | ~200–500ms | jsPDF `.html()` with CSS |
| Output size | ~50–150KB | Text-based PDF, selectable text |
| Total gallery overhead | +1–3s for 10 markdown files | Parallelisable |

### Comparison with Current Thumbnail

| Aspect | Thumbnail (current) | Full PDF (proposed) |
|--------|---------------------|---------------------|
| Format | JPEG, 400×520px | PDF, A4 multi-page |
| Text selectable | No | Yes |
| Full content | First ~40 lines | All content |
| Size | 15–25KB | 50–150KB |
| Generation time | 50–100ms | 200–500ms |

---

## Integration Point

In `_v0213_addPreviewToZip()`, after generating the JPEG thumbnail for a markdown file, also generate the PDF:

```
_gallery.{hash}/
  _manifest.json
  thumbnails/file_001.thumb.jpg    ← existing
  metadata/file_001.meta.json      ← existing
  pdfs/file_001.pdf                ← NEW: full rendered PDF
```

The `_manifest.json` would gain a new field per file: `"pdf": "pdfs/file_001.pdf"`.

---

## When to Implement

**Not now.** This belongs in a more evolved version of the gallery workflow that includes:

- User-controlled options (quality, page size, what to generate)
- More metadata extraction capabilities (EXIF, document properties, etc.)
- Configurable `_gallery` contents (thumbnails only, thumbnails + PDFs, full metadata)
- Per-file-type processing options

The current gallery workflow is intentionally simple — it generates thumbnails automatically with zero configuration. PDF generation adds decision points (which files? what styling? what page format?) that belong in a version with user controls.

---

## Dependencies to Add (When Ready)

| Library | Size | CDN | Purpose |
|---------|------|-----|---------|
| jsPDF | ~90KB | `cdnjs.cloudflare.com/ajax/libs/jspdf/2.5.1/jspdf.umd.min.js` | PDF generation |
| html2canvas | ~60KB | Optional — jsPDF `.html()` uses it internally | HTML → canvas for PDF |

**Note:** jsPDF's `.html()` method bundles html2canvas internally in recent versions, so only one CDN load is needed. Load pattern would mirror pdf.js — lazy-loaded only when markdown files are present and PDF generation is enabled.

---

## Alternative Approaches Considered

| Approach | Verdict |
|----------|---------|
| **Server-side (pdfkit/puppeteer)** | Breaks zero-knowledge — server would see plaintext markdown |
| **Canvas → multi-page images** | Works but produces image-based PDFs (no text selection) |
| **SVG foreignObject → PDF** | Browser security restrictions taint the canvas |
| **Print window automation** | Browsers block — requires user click |

**jsPDF is the only viable browser-side approach** that produces real, text-selectable PDFs without breaking zero-knowledge architecture.
