# Dev Review: IFD Minor Version Workflow — Why It Matters

**Version:** v0.13.22
**Date:** 2026-03-11
**Role:** Dev (Explorer team)
**Type:** Workflow lesson and guidance

---

## The Moment That Proved It

During the zip viewer layout redesign (Architect spec v0.13.22), the Dev's first instinct was to edit v0.2.0 directly. The changes were significant — 399 insertions across 2 files — touching `_renderZipLayout`, `_renderMarkdownContent`, `_fileTypeIcon`, `setupEventListeners`, `cleanup`, `_selectZipFolder`, and `_previewZipEntry`.

After the changes landed, the human immediately asked to revert and use a proper v0.2.1 IFD layer instead.

**The revert was trivial** — `git checkout HEAD~1 -- <files>` restored v0.2.0 to pristine. But if this had been merged, deployed, and then discovered to break something two weeks later? The blast radius would have been the entire 1,500-line component. No clean revert. No way to isolate the regression. Just a diff to read and hope.

With v0.2.1 as a separate layer, the revert is: **delete the v0.2.1 folder and point back to v0.2.0's HTML.** Nothing else changes. Nothing else can break.

---

## The Rule

**After a major version is stable (merged to main or deployed to prod), ALWAYS create a new minor version for changes. Never modify the major version in-place.**

This applies doubly when:

1. **You haven't touched the code recently.** Coming back to a component after days or weeks means your mental model is stale. You will miss interactions. A surgical override limits the damage.

2. **The change is large.** The zip viewer redesign touched 7 methods and added 5 new ones. That's not a "quick fix" — it's a structural change. Structural changes in a separate layer can be cleanly removed.

3. **The change is experimental.** Explorer work is, by definition, experimental. You don't know if it will survive contact with real users. A minor version makes it trivially revertable.

4. **Multiple people (or agents) are working on the component.** Two agents editing v0.2.0 simultaneously creates merge hell. Two agents creating v0.2.1 and v0.2.2 creates zero conflicts — the HTML just loads them in order.

---

## What IFD Minor Versions Give You

| Without minor version | With minor version |
|---|---|
| Changes mixed into base file | Changes isolated in separate file |
| Revert = read diff, undo selectively | Revert = delete folder |
| Regression source unclear | Regression source = the minor version |
| Can't A/B test | Swap HTML to load v0.2.0 or v0.2.1 |
| Merge conflicts with parallel work | Zero conflicts — additive only |
| One broken method breaks everything | One broken override, base still works |
| Must understand entire file to review | Only review the override file |

## The Surgical Override Pattern

v0.2.1's JS file is entirely self-contained:

```javascript
// Save v0.2.0's original method
const _v020_setupEventListeners = SendDownload.prototype.setupEventListeners;

// Override with v0.2.1 version
SendDownload.prototype.setupEventListeners = function() {
    _v020_setupEventListeners.call(this);  // Call v0.2.0 first
    // ... add v0.2.1 listeners
};
```

**Key properties:**
- **No `customElements.define()`** — reuses v0.2.0's registration
- **Calls v0.2.0 via saved reference** — doesn't break v0.2.0's behaviour
- **Loaded after v0.2.0 via `<script>` order** — prototype mutation is deterministic
- **CSS uses cascade** — v0.2.1 CSS loaded after v0.2.0, later selectors win

---

## When to Consolidate (Create v0.3.0)

Minor versions accumulate. When you have v0.2.0 + v0.2.1 + v0.2.2 + v0.2.3 and the feature set is stable, create v0.3.0 by merging everything into a single consolidated file. This is the Villager's job — not the Explorer's.

**Signs it's time to consolidate:**
- More than 5 minor versions on a major
- The minor versions collectively override >50% of the base methods
- The feature is shipping to production and needs performance (fewer script loads)
- The override chain is getting hard to reason about

---

## Practical Checklist for Devs

Before editing any UI component:

1. **Check the version.** Is it a major (v0.2.0) or minor (v0.2.3)?
2. **Is the major version on main/prod?** If yes → create a new minor version. Always.
3. **Is your change >20 lines or touches >2 methods?** If yes → new minor version.
4. **Are you returning to code you haven't touched in >3 days?** If yes → new minor version.
5. **Are you experimenting?** If yes → new minor version.
6. **Is it a genuine 1-line bugfix in a method you fully understand?** OK to edit in-place — but even then, a minor version is safer.

**Default to minor version. The cost is one extra `<script>` tag. The benefit is clean revertability.**

## File Structure Reference

```
v0.2.0/                          ← Major: consolidated, stable, untouchable
  _common/js/components/...      ← Full implementation
  en-gb/download/index.html      ← Loads only v0.2.0

v0.2.1/                          ← Minor: surgical overrides only
  _common/js/components/...      ← Prototype overrides, no customElements.define()
  en-gb/download/index.html      ← Loads v0.2.0 base + v0.2.1 overrides

v0.2.2/                          ← Next minor: more overrides
  _common/js/components/...      ← Can override v0.2.0 OR v0.2.1 methods
  en-gb/download/index.html      ← Loads v0.2.0 + v0.2.1 + v0.2.2

v0.3.0/                          ← Next major: consolidation (Villager job)
  _common/js/components/...      ← Everything merged into single files
```
