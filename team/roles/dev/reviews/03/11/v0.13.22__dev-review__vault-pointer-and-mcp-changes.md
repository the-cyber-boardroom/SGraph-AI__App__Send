## 1. Routes__Vault__Pointer.py Changes

**File:** `sgraph_ai_app_send/lambda__user/fast_api/routes/Routes__Vault__Pointer.py`
**Commit:** `73c41cb`

### What Changed

- Added `import base64`
- Added `read-base64/{vault_id}/{file_id}` to `ROUTES_PATHS__VAULT` list
- Added module-level constant `LAMBDA_BASE64_LIMIT = 3750000`
- Added `read_base64__vault_id__file_id()` method
- Registered the new route in `setup_routes()`

### Code Quality Observations

**Positive:**

- The new method exactly mirrors the existing `read__vault_id__file_id()` pattern, sharing the same `vault_service.read()` call. No service layer changes required -- route-level concern only.
- Response schema matches the brief's specification: `{vault_id, file_id, data, size}`.
- The 413 error message includes the fallback path (`Use /api/vault/read/{vault_id}/{file_id} instead`), which is good UX for MCP consumers.
- No auth required on reads -- consistent with the zero-knowledge model where reads are public and write-key gates mutations.
- Method is synchronous (no `async`) -- correct, since `vault_service.read()` is synchronous.

**Observations (non-blocking):**

1. **`LAMBDA_BASE64_LIMIT` is a module-level constant here but an instance-level constant in `Routes__Transfers`.** In `Routes__Transfers.py` (line 169), the same value is defined as `LAMBDA_BASE64_LIMIT = 3750000` inside the class body (`self.LAMBDA_BASE64_LIMIT`). In `Routes__Vault__Pointer.py` it is a module-level constant. Both work, but the inconsistency is worth noting. A shared constant (e.g., in `user__config.py`) would be cleaner, especially if the Lambda response limit ever changes.

2. **Size check uses `len(payload)` (raw bytes), not the base64-encoded size.** The raw payload at 3.75MB encodes to ~5MB base64, which is the Lambda response limit. This is correct -- the check is on raw size before encoding. The Transfers variant does the check differently: it reads `file_size_bytes` from metadata before fetching the payload, avoiding loading a too-large payload into memory. The Vault variant loads the full payload first, then checks size. For vault pointer files (typically small encrypted objects like tree nodes, settings, refs), this is acceptable. Vault objects are unlikely to approach 3.75MB. If large vault objects become common, consider adding a size check before payload read.

3. **No download event recording.** The `read__vault_id__file_id` and `read_base64__vault_id__file_id` methods do not record analytics events (IP hash, user agent, timestamp). The Transfers download endpoints do record download events via `get_download_payload()`. This is a deliberate design difference -- vault reads are high-frequency (CLI clone/pull reads dozens of objects), and recording each would create noise. However, if vault analytics become desired, this is where to add them.

### Alignment with Existing Patterns

The implementation follows the established base64 endpoint pattern from `Routes__Transfers.download_base64__transfer_id()` line-for-line:

| Aspect | Transfers | Vault Pointer |
|--------|-----------|---------------|
| Size guard | Pre-check via metadata | Post-load via `len(payload)` |
| Response fields | `transfer_id`, `data`, `file_size_bytes` | `vault_id`, `file_id`, `data`, `size` |
| Error on too-large | 413 with presigned URL fallback | 413 with raw-read fallback |
| Auth required | No | No |
| Base64 encoding | `base64.b64encode(payload).decode('ascii')` | Same |

Good alignment. The field name difference (`file_size_bytes` vs. `size`) is a minor inconsistency but follows the vault pointer's simpler naming convention.

## 2. MCP__Setup.py Changes

**File:** `sgraph_ai_app_send/utils/MCP__Setup.py`
**Commit:** `e47ebd2`

### What Changed

The `_set_short_operation_ids()` method was fixed to handle multi-segment tags and produce valid MCP tool names.

**Before (broken):**
```python
path_parts = [p for p in route.path.split('/') if p and not p.startswith('{')]
if path_parts and path_parts[0] == tag:
    path_parts = path_parts[1:]
name = f"{tag}_{action}"
```

This produced names like `api/transfers_create` -- the slash violates MCP's tool name pattern `^[a-zA-Z0-9_-]{1,64}$`.

**After (fixed):**
```python
tag_segments = [p for p in tag.split('/') if p]
path_parts = [p for p in route.path.split('/') if p and not p.startswith('{')]
for seg in tag_segments:
    if path_parts and path_parts[0] == seg:
        path_parts = path_parts[1:]
safe_tag = tag.replace('/', '_')
name = f"{safe_tag}_{action}"
```

### Code Quality Observations

**Positive:**

- The fix correctly handles multi-segment tags (`api/transfers`, `api/vault`) by splitting the tag into segments and stripping each one from the path parts.
- `tag.replace('/', '_')` is a simple, predictable transformation.
- The duplicate-name detection logic (appending HTTP method for collisions) is unchanged and still works.

**Observation (non-blocking):**

1. **The tag-stripping loop is greedy and order-dependent.** The `for seg in tag_segments` loop strips matching path segments sequentially. If a route path happened to contain the same segment names in a different position (e.g., `/vault/api/transfers/...`), this could mis-strip. In practice, all routes in this codebase follow the pattern `/{tag_segments}/action/...`, so this is not a real risk. Just noting the assumption.

2. **Tool name length.** The method's docstring mentions Claude.ai's 64-char tool name limit. The current names (e.g., `api_vault_read_base64`, `api_transfers_download_base64`) are well under 64 chars. No issue.

## 4. sg-send-cli Integration Assessment

### How the CLI Interacts with Vault API

Based on the round-trip debrief (`v0.13.31`) and SKILL.md (`v0.13.21`):

```
sg-send-cli clone <vault-key>     -->  GET /api/vault/read/{vault_id}/{file_id}     (multiple calls)
sg-send-cli pull                   -->  GET /api/vault/read/{vault_id}/{file_id}     (delta)
sg-send-cli push --token <token>   -->  PUT /api/vault/write/{vault_id}/{file_id}    (multiple calls)
sg-send-cli init --token <token>   -->  PUT /api/vault/write/{vault_id}/{file_id}    (manifest + initial objects)
```

The CLI uses the **raw binary** read endpoint (not read-base64), which is correct. The CLI handles binary data natively -- it doesn't need base64 wrapping. The read-base64 endpoint is for MCP transport compatibility (the MCP JSON-RPC layer cannot handle raw bytes).

### Architecture Observations

1. **CLI and MCP are complementary, not competing.** The CLI (`sg-send-cli`) runs as a subprocess with full binary I/O capability. MCP runs within the Claude.ai sandbox where JSON-RPC is the transport. Both consume the same vault API but through different endpoints:
   - CLI: `read` (binary) + `write` (binary)
   - MCP: `read-base64` (JSON) + `write` (binary via headers, but this is the request direction -- the server receives binary)

2. **Write path is not base64-ified.** The v0.13.14 brief explicitly scopes this out: "Write-base64 variant -- out of scope (writes require binary payload; different problem)." This is pragmatic. MCP tool calls can send binary data in request bodies. The problem was response bodies, not request bodies.

3. **The SKILL.md self-bootstrapping pattern is notable.** A Claude session writes `SKILL.md` into a vault, and a future session clones the vault and reads it to learn the workflow. This is an emergent usage pattern -- the vault API was designed for human-to-human file sharing, but it works equally well for agent-to-agent knowledge transfer. No code changes are needed to support this pattern.

### Concerns

1. **No rate limiting on vault reads.** A CLI `clone` operation reads every object in the vault sequentially. For a vault with hundreds of objects, this is hundreds of rapid GET requests. There is no per-IP or per-vault rate limit. For the current usage (small vaults, trusted users with vault keys), this is fine. If vaults grow large or become publicly shared, rate limiting should be considered.

2. **No vault size limit.** There is no server-side enforcement of how many objects a vault can contain or how much total storage it can consume. The write endpoint accepts any payload up to the Lambda request body limit. The only guard is the access token requirement for writes.

3. **Vault key scope.** The vault key (`access_id:vault_id`) grants read access to all objects in the vault. There is no per-object access control for reads. This is by design (zero-knowledge model -- the key is the access), but worth noting when the CLI usage patterns expand.

## 5. Recommendations

### P0 -- Do Now

1. **Add test for 413 size guard on read-base64.** Write a payload exceeding `LAMBDA_BASE64_LIMIT` and assert the endpoint returns 413. This is a missing coverage gap for the new code path. Estimated effort: 10 minutes.

### P1 -- Do Soon

2. **Extract `LAMBDA_BASE64_LIMIT` to a shared constant.** Both `Routes__Transfers` and `Routes__Vault__Pointer` define it independently. Move to `user__config.py` to ensure they stay in sync.

3. **Update reality document.** The v0.10.49 reality document does not list the `read-base64` vault endpoint. Per project rules, the reality document should be updated in the same commit as code changes. This was missed in commit `73c41cb`.

### P2 -- Consider

4. **Pre-check payload size before loading.** The Transfers variant checks `file_size_bytes` from metadata before fetching the payload. The Vault Pointer variant loads the full payload into memory, then checks size. For small vault objects this is fine, but if large vault objects become common, consider storing payload size in the vault file metadata and checking before `storage_fs.file__bytes()`.

5. **Add MCP-specific test.** Verify that the `read-base64` endpoint appears in the MCP tool manifest with the correct short name (e.g., `api_vault_read_base64`). This would catch regressions in the `_set_short_operation_ids()` logic for vault routes specifically.
