## 3. Inline Form Implementation

### Architecture Decision: Plain HTML + Vanilla JS

The existing Early Access form at `/early-access/index.html` is plain HTML:
```html
<form class="eap-form launchlist-form" action="https://getlaunchlist.com/s/zSa8gI" method="POST">
```

The inline form should follow the same pattern -- no Web Component needed. The site uses vanilla JS throughout. A simple `<form>` with a `fetch()` call to the new endpoint is sufficient.

### Endpoint Location

The brief specifies `POST /api/early-access/signup` on `send.sgraph.ai`. This means the form on `sgraph.ai` (the marketing site) will POST cross-origin to `send.sgraph.ai` (the application backend).

### CORS: Already Solved

The User Lambda CORS configuration at `sgraph_ai_app_send/lambda__user/fast_api/Fast_API__SGraph__App__Send__User.py` (lines 65-73) already allows:
- `allow_origins = ["*"]` -- any origin, including `sgraph.ai`
- `allow_methods = ["GET", "POST", "HEAD", "OPTIONS"]` -- POST is allowed
- `allow_headers` includes `Content-Type` -- sufficient for JSON POST

**No CORS changes needed.** The form can POST JSON to `https://send.sgraph.ai/api/early-access/signup` from `sgraph.ai` without any backend modification.

### Frontend Implementation (~40 lines JS)

```javascript
// Inline in the homepage, no separate file needed
document.getElementById('eap-form').addEventListener('submit', async (e) => {
    e.preventDefault();
    const name  = e.target.elements.name.value;
    const email = e.target.elements.email.value;

    try {
        const res = await fetch('https://send.sgraph.ai/api/early-access/signup', {
            method: 'POST',
            headers: { 'Content-Type': 'application/json' },
            body: JSON.stringify({ name, email })
        });
        if (res.ok) {
            // Replace form with confirmation
            document.getElementById('eap-form-card').innerHTML = `
                <h3>You're in.</h3>
                <p>We'll send your access token to <strong>${email}</strong> within 48 hours.</p>
                <p><a href="https://buy.stripe.com/fZu8wHdv23nN1Z85jn5os02">Buy credits now and start today</a></p>
            `;
        }
    } catch (err) {
        // Show error state
    }
});
```

### Form HTML (~25 lines)

The form card styles already exist in `_common/css/style.css` (`.form-input` class, lines 631-682). The early-access page has custom styles for `.eap-form-card` that can be inlined or extracted to `_common/css/style.css`.

### Implementation Recommendation

1. Add the form section with `id="try-free"` to the homepage HTML (for anchor scrolling).
2. Inline the JS in a `<script>` block at the bottom of the page (consistent with how `i18n.js` is loaded).
3. Use the existing `.form-input` and `.btn` CSS classes.
4. Copy the `.eap-form-card` styles from `early-access/index.html` into the homepage `<style>` block (or better, move to `_common/css/style.css`).

## 4. i18n Impact

### Scale of Impact

The website has **18 locale JSON files** (en-gb + 17 locales) and **20 HTML pages** under `en-gb/`. The i18n generation script (`scripts/generate_i18n_pages.py`) reads all HTML files from `en-gb/` and produces locale copies for all 16 non-English locales (plus Klingon).

### Keys Affected

| Category | Key Count | Action |
|----------|-----------|--------|
| `eap.*` (Early Access page) | 32 keys | Remove if page eliminated, or keep if form is inlined |
| `pricing.whyfree.*` (Why Free page) | 16 keys | Remove when `/pricing/free/` eliminated |
| `pricing.cloudpage.*` (Cloud detail page) | 33 keys | Remove when `/pricing/cloud/` eliminated |
| `home.hero.cta_free`, `home.hero.cta_pricing` | 2 keys | Replace with new CTA text keys |
| `home.getstarted.*` | 6 keys | Modify (swap featured status) |
| New inline form keys | ~10 keys | Add new keys for form labels, confirmation, etc. |

**Total i18n work:** ~81 keys removed, ~12 keys added/modified, across 18 JSON files.

### Generation Script Impact

The generation script (`scripts/generate_i18n_pages.py`) discovers HTML files dynamically via `find_html_files()` (line 86). When pages are deleted from `en-gb/`, they are automatically excluded from generation. **No script changes needed for page elimination.**

However, the script also needs to handle the new inline form. The `data-i18n-placeholder` attributes on form inputs (e.g., `data-i18n-placeholder="eap.form.name_placeholder"`) are already supported by the script (lines 138-155). **No script changes needed for the inline form.**

### Locale Translation Propagation

For the P0 changes (CTA text), the new English keys will be used as fallbacks for all locales that haven't been translated yet. The generation script (line 127) uses `translations.get(key, en_translations.get(key, old_content))` -- if a locale doesn't have a key, it falls back to English. This means P0 changes can ship without updating all 17 locale files.

However, the removed keys should be cleaned from all locale JSON files to avoid dead key accumulation. This is a mechanical task -- about 81 keys x 17 files = 1,377 line deletions.

### Key Consideration: Nav Order

The nav order change (P3) is embedded in HTML, not i18n. The nav links use `data-i18n` for their labels but the link order is determined by HTML structure. Reordering the nav in `en-gb/` will automatically propagate to all locales via the generation script.

## 5. Page Elimination Strategy

### Pages to Eliminate

| Page | Path | Current Inbound Links | Elimination Strategy |
|------|------|----------------------|---------------------|
| Cloud detail | `/pricing/cloud/` | From `/pricing/` card CTA (line 438) | Delete file, update CTA to Stripe link |
| Why Free | `/pricing/free/` | From homepage hero CTA (line 85), from `/pricing/` footer link (line 382) | Delete file, update links |
| Early Access | `/early-access/` | From `/pricing/` EAP card (line 322), from `/pricing/free/` CTA (line 140) | Delete file after inline form ships |

### File Deletion Plan

**Step 1 (with P0):** Update all inbound links away from these pages. Do NOT delete files yet.

**Step 2 (with P1/P2):** Delete the files:
- `sgraph_ai__website/en-gb/pricing/cloud/index.html` (372 lines)
- `sgraph_ai__website/en-gb/pricing/free/index.html` (193 lines)
- `sgraph_ai__website/en-gb/early-access/index.html` (409 lines)

**Step 3:** The i18n generation script will stop generating locale copies automatically (it discovers HTML files from `en-gb/`).

### Redirect Handling

The sgraph.ai website is served from S3 + CloudFront. The CloudFront Function (`sgraph_ai__website/cloudfront/url-rewrite.js`) handles trailing-slash rewrites but does NOT support redirect rules.

**Options for redirects:**

1. **CloudFront Function update** -- Add redirect rules to `url-rewrite.js` for `/pricing/cloud/` -> `/pricing/` and `/pricing/free/` -> `/pricing/`. This adds ~10 lines to the function. Recommended approach.

2. **S3 redirect objects** -- Create S3 objects with `x-amz-website-redirect-location` headers. Requires the S3 bucket to be configured for website hosting (which it is, via CloudFront OAC).

3. **Leave 404s** -- The proposal says `/early-access/` has "no external links." For `/pricing/cloud/` and `/pricing/free/`, search engines may have indexed these. Leaving 404s will cause temporary SEO impact.

**Recommendation:** Use CloudFront Function redirects for `/pricing/cloud/` and `/pricing/free/`. Skip redirect for `/early-access/` (low external link risk). Update `url-rewrite.js` to include:

```javascript
// Redirect eliminated pages
var redirects = {
    '/en-gb/pricing/cloud/': '/en-gb/pricing/',
    '/en-gb/pricing/free/':  '/en-gb/pricing/'
};
// Add locale variants too
```

**Risk:** The redirect map needs entries for all 17 locales, not just `en-gb`. This is ~34 redirect entries. Consider a pattern-based redirect instead:

```javascript
if (uri.match(/\/pricing\/cloud\/?$/)) {
    return { statusCode: 301, headers: { location: { value: uri.replace(/\/cloud\/?$/, '/') } } };
}
```

### Impact on CloudFront URL Rewrite Function

The existing `url-rewrite.js` (49 lines) is clean and simple. Adding redirect logic is straightforward but should be done carefully:
- Redirects should return 301 (permanent), not 302 (temporary), since these pages are being permanently eliminated.
- Pattern matching handles all locale prefixes at once.
- The function runs on every viewer request, so it must remain fast.

## 7. Risk Assessment

### Low Risk

| Risk | Mitigation |
|------|-----------|
| **Wrong Stripe URL** | The URL `https://buy.stripe.com/fZu8wHdv23nN1Z85jn5os02` is already live on `/pricing/cloud/`. Copy it exactly. |
| **Nav reorder breaks nothing** | Nav links are self-contained per page. Reordering is purely cosmetic. |
| **i18n generation breaks** | The script auto-discovers HTML files. Deleting files automatically excludes them. |

### Medium Risk

| Risk | Mitigation |
|------|-----------|
| **CORS preflight caching** | Browsers cache CORS preflight responses. If the new endpoint has different headers than existing ones, some browsers may reject requests temporarily. Mitigation: the User Lambda already allows `*` origins and `POST` method, so the new endpoint inherits this. |
| **Broken external links to `/pricing/cloud/`** | Google may have indexed this page. Add 301 redirects in CloudFront Function. Monitor Search Console for 404 spikes. |
| **Email delivery failure** | If SES is not configured, the early-access signup endpoint works but emails don't arrive. Mitigation: the brief suggests n8n webhook as fallback. Test email delivery before Challenge launch. |
| **Form JS errors on old browsers** | `fetch()` API is used for form submission. Not supported in IE11. Mitigation: IE11 usage is negligible for the target audience (security-conscious users). |
| **Locale JSON desync** | Removing 81 keys from 17 JSON files is error-prone. Mitigation: write a script to batch-remove keys, or accept that dead keys in JSON are harmless. |

### High Risk

| Risk | Mitigation |
|------|-----------|
| **P0 depends on P1 for "Try Free" CTA destination** | The P0 CTA swap changes "Get Started Free" to "Try Free". If P1 (inline form) isn't ready, the "Try Free" button has no destination. **Mitigation:** Use `href="./early-access/"` as temporary target until P1 ships. Update to `href="#try-free"` when the inline form is ready. |
| **LaunchList removal before replacement is ready** | If the LaunchList form is removed (in `early-access/index.html`) before the internal endpoint is operational, signups break entirely. **Mitigation:** Ship the new endpoint first, verify email delivery, THEN update the form action. Keep LaunchList as fallback until the new endpoint is proven. |
| **Challenge campaign launches before all P0 changes deploy** | If the WhatsApp messages go out before the Stripe link is on the homepage, the campaign generates the exact friction the UX brief documents. **Mitigation:** P0 changes must deploy before any Challenge messages are sent. This is a hard dependency. |

## 10. Verdict

The proposal is well-structured and the phased approach is correct. The P0 changes are genuinely small -- the homepage CTA swap and Stripe link can ship in under an hour. The main risk is the dependency between P0's "Try Free" CTA destination and P1's inline form. The mitigation (temporary link to `/early-access/`) is sound.

The LaunchList replacement (P0) is the most technically complex item due to email infrastructure, but the scope is intentionally minimal -- one endpoint, one email notification, no database.

**Recommendation:** Ship Phase 1 (CTA swap + Stripe link) immediately as a standalone commit. This alone reduces the Challenge purchase path from 4 clicks to 1 click. Phase 2 (LaunchList replacement) can follow in a separate session.
