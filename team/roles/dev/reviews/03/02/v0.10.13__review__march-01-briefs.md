### The n8n Webhook Question

The brief says: "use SES if already configured, or any available email sending capability. If SES isn't viable quickly, even a webhook to an n8n workflow that sends the email would work as an interim solution."

The implementation chose the n8n webhook path. This is a **reasonable interim choice**:

**Strengths:**
- n8n infrastructure already exists and is configured.
- Uses stdlib `urllib.request` -- no new dependency.
- 10-second timeout prevents hanging.
- Per-recipient success/failure tracking allows partial delivery.
- `X-Webhook-Secret` header provides webhook authentication.

**Risks:**
- If `N8N_WEBHOOK_URL` env var is empty, notifications fail silently. The user gets a 200 response with `success: true` even when no notification was sent. The `notifications` array in the response body contains the failure details, but the top-level `success` field is always `true` if validation passed.
- n8n is a single point of failure for all notification flows (Early Access + future payment notifications). If n8n is unreachable, signups are silently lost.
- No retry mechanism on webhook failure.

**Verdict:** Acceptable for the current ~50-100 signup scale. When the SG WorkMail service is available, `_call_webhook()` should be replaced with direct SES/WorkMail sending.
