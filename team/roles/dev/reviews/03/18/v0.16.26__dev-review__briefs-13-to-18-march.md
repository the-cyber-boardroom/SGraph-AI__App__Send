## Implementation Streams (by Priority)

### P0 — Vault Redesign (CLI)

**Source:** v0.16.3 vault-redesign documents (03/17)

11 phases, estimated ~300 LOC added, ~370 LOC removed (net smaller codebase):

| Phase | Description | Effort |
|-------|-------------|--------|
| 1 | Schema cleanup (remove legacy fields) | Small |
| 2 | Crypto layer (new derivations, remove legacy) | Medium |
| 3 | Object store & ref manager (self-describing IDs) | Medium |
| 4 | Branch manager (deterministic index IDs) | Medium |
| 5 | **Sub-tree model** (new commit algorithm) | **Large** |
| 6 | Init (deterministic IDs) | Small |
| 7 | Push (remove dual-write, add merge commit) | Medium |
| 8 | Clone (graph-walking, lite clone mode) | Medium |
| 9 | Pull (sub-tree comparison with shortcut) | Medium |
| 10 | API client (batch read) | Small |
| 11 | Cleanup (delete all legacy code) | Small |

**Critical detail:** Phase 5 (sub-tree model) is the big one — new bottom-up tree building algorithm. The detailed implementation guide (v0.16.3) provides exact code signatures and test vectors.

## Risks

1. **Vault redesign scope:** 11 phases is ambitious. Phase 5 (sub-tree) is the critical path. If it slips, everything downstream (push, clone, pull) is blocked.
2. **Breaking change:** New ID format and sub-tree model are not backward-compatible. Existing vaults must be recreated.
3. **Three crypto convergence:** Merging CLI, browser, and UI onto one crypto model touches every file in the vault stack.
4. **Upload UX component imports:** Dependencies on Tools repo components that may need adaptation for SG/Send context.
