## Executive Summary

The CLI team's vault architecture redesign (v0.16.3) introduces self-describing file IDs, sub-tree model, batch read API, deterministic HMAC addressing, and encrypted-only metadata. This review assesses the impact on our browser codebase across four areas: backend (FastAPI routes/service), vault browser UI (v0.1.2), user vault UI (v0.1.7), and the shared JS library.

**Key finding:** Our backend already supports batch read with unauthenticated read-only mode and a health check endpoint. The two highest-priority backend items from the redesign are already shipped. The remaining work is concentrated in the JS layer: updating ID formats, fixing the vault_id regex, adding sub-tree rendering, and eventually replacing the user vault's RSA model.

**Verdict:** Most changes are additive and well-contained. The v0.1.2 vault browser is architecturally aligned with the redesign (git-like model, PBKDF2, HMAC-derived IDs). The v0.1.7 user vault (RSA model) requires a full rewrite but that is a separate phase, blocked on the shared library updates.

## 2. Vault Browser Assessment (v0.1.2)

The v0.1.2 vault browser (`sgraph_ai_app_send__ui__vault/v0/v0.1/v0.1.2/`) uses the git-like model and is the correct base, as the redesign confirms. However, several specific changes are needed.

### 2.1 Bug Fix: vault_id Regex (Can Start Now)

**File:** `v0/v0.1/v0.1.2/_common/js/lib/sg-vault/sg-vault-crypto.js` line 35

**Current:** `/^[0-9a-f]{8}$/` (hex only)
**Required:** `/^[a-z0-9]{8}$/` (alphanumeric)

The CLI generates vault IDs with the full alphanumeric charset (`secrets.token_urlsafe()` produces base62). The current regex rejects valid vault IDs like `12cpxeq9` because `c`, `p`, `x`, `q` are valid in `[a-z0-9]` but the `p`, `q`, `x` pass `[0-9a-f]` only if they happened to be hex. This is an **existing interop bug** that the redesign documents identified. The v0.1.3 crypto already has the correct regex (`/^[a-z0-9]{8}$/`).

**Impact:** This single-character regex fix unblocks CLI-created vaults from being opened in the v0.1.2 vault browser. **Do this first.**

### 2.2 Self-Describing ID Prefixes in `deriveKeys()` (Can Start Now)

**File:** `sg-vault-crypto.js`

The current `deriveKeys()` returns bare 12-char hex IDs:
```javascript
return { readKey, writeKey, refFileId, branchIndexFileId }
// refFileId = "b1d1771f9fb5"
```

The redesign requires prefixed IDs:
```javascript
return {
    readKey, writeKey,
    refFileId:         'ref-pid-muw-' + refFileId,
    branchIndexFileId: 'idx-pid-muw-' + branchIndexFileId
}
```

All callers of `deriveKeys()` that build `bare/refs/{refFileId}` paths will automatically get the correct new format. This is a small change with wide impact — it affects every vault operation.

**Compatibility note:** This is a **breaking change**. Vaults created with the old format will not be openable with the new format (different file_id paths). The redesign explicitly states "no backwards compatibility."

### 2.3 Per-Branch Ref Derivation (New Method — Can Start Now)

**File:** `sg-vault-crypto.js`

Add:
```javascript
static async deriveBranchRefFileId(hmacKey, vaultId, branchName) {
    const hex = await this._deriveFileId(hmacKey,
        `sg-vault-v1:file-id:branch-ref:${vaultId}:${branchName}`)
    return 'ref-pid-snw-' + hex
}
```

Not immediately used by the vault browser (it only reads named HEAD), but needed for full interop with CLI-created clone branches.

### 2.4 Object ID Prefix Update (Can Start Now)

**File:** `sg-vault-object-store.js` line 45

**Current:** `return 'obj-' + hex.slice(0, 12)`
**Required:** `return 'obj-cas-imm-' + hex.slice(0, 12)`

This changes the content-addressed object ID format. Combined with 2.2, this means all API calls use the new self-describing paths:
- `bare/data/obj-cas-imm-aeaa531d8450` instead of `bare/data/obj-aeaa531d8450`
- `bare/refs/ref-pid-muw-b1d1771f9fb5` instead of `bare/refs/b1d1771f9fb5`

### 2.5 Commit Schema Update (Blocked on CLI)

**File:** `sg-vault-commit.js`

**Current:** Commits store `message` as plaintext.
**Required:** Commits store `message_enc` as encrypted.

The `createCommit()` method (line 29) currently writes:
```javascript
message: message
```

The redesign requires:
```javascript
message_enc: await enc(message)
```

And `loadCommit()` must decrypt `message_enc` to display the message.

**Dependency:** The CLI must produce the new commit format first. If we update `loadCommit()` now, it will break reading existing vaults. **Wait for CLI phase 5 completion.**

### 2.6 Sub-Tree Rendering (Blocked on CLI Phase 5)

**Files:** `sg-vault.js`, `sg-vault-commit.js`, `vault-tree-view.js`, `vault-browser.js`

The current vault browser uses a **flat tree model** — `SGVault.open()` (line 113) iterates `tree.entries` and calls `_insertEntry()` which parses `entry.name` (a full path like `an-folder/file-b.txt`) to build the in-memory nested tree.

The redesign replaces this with **sub-tree objects**: each tree entry references either a `blob_id` (file) or a `tree_id` (sub-folder). The root tree only contains direct children.

**Changes needed:**

1. **`SGVault.open()`** — After loading root tree, entries with `tree_id` are folders. Do NOT recursively load sub-trees upfront. Instead, render root level and load sub-trees on demand when the user expands a folder.

2. **`SGVaultCommit.loadTree()`** — Already decrypts `name_enc`, `size_enc`, `content_hash_enc`. The decrypted entries will have `tree_id` (for folders) instead of `name` containing a path separator.

3. **`SGVault._flattenTree()` and `SGVault._commit()`** — Must build sub-tree objects bottom-up (one tree per directory), not a single flat tree. The `_commit()` method currently creates one tree with all flat path entries.

4. **`VaultTreeView._renderNode()`** — Currently reads from `this._vault._tree['/']` (the in-memory nested tree). This needs to be replaced with an async folder-expand that fetches sub-tree objects on demand.

5. **`VaultBrowser._renderFileList()`** — Same dependency on the in-memory nested tree. Needs async folder loading.

**This is the biggest change in the browser codebase.** The entire open flow and tree rendering model changes from "load everything upfront, render from in-memory tree" to "load root, render root, fetch sub-trees on user interaction."

**Blocked on:** CLI producing vaults with sub-tree format. Can start the architecture/scaffolding now, but cannot test until CLI Phase 5 ships.

### 2.7 Encrypted Field Decryption (Partially Done)

**File:** `sg-vault-commit.js`

The v0.1.2 `_decryptTreeEntry()` already decrypts `name_enc`, `size_enc`, `content_hash_enc`. This is correct and aligned with the redesign. However:

- It currently also expects a plaintext `name` field to exist (used in `_insertEntry()`). After the redesign, there is no plaintext `name` — only `name_enc`.
- The `_encryptTreeEntry()` encrypts `entry.name` into `name_enc`. After the redesign, the source of the name is the in-memory tree node name, not a flat path.

### 2.8 Timestamp Display Fix (Can Start Now)

The redesign notes that the vault browser renders `created_at` timestamps from the branch index as year 58176. The value is uint milliseconds but is being multiplied by 1000 somewhere. Fix: pass directly to `new Date(timestamp_ms)` without multiplication. This is likely in the storage-viewer or branch rendering component.

## 3. User Vault Assessment (v0.1.7)

### 3.1 Current State: RSA-4096 Hybrid Encryption

The user vault (`sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.7/`) uses a completely different crypto model:

| Aspect | User Vault (v0.1.7) | Vault Browser (v0.1.2) | CLI |
|---|---|---|---|
| Encryption | RSA-4096 + AES-256-GCM hybrid | PBKDF2 + AES-256-GCM | PBKDF2 + AES-256-GCM |
| Key storage | IndexedDB (`sgraph-send-vault-keys`) | None (derived from vault key) | None (derived from vault key) |
| API | Admin API (`/vault/create`, `/vault/file`, etc.) | User API (`/api/vault/write`, etc.) | User API (same as browser) |
| Data model | Flat (folder/file GUIDs) | Git-like (refs, commits, trees, blobs) | Git-like (same as browser) |

**The user vault is fundamentally incompatible** with both the vault browser and the CLI. The redesign proposes replacing it entirely.

### 3.2 Required Changes

1. **Delete** `vault-crypto.js` (RSA model) and `vault-api.js` (admin API client)
2. **Import** the shared `sg-vault/` library from the vault browser's `_common/js/lib/`
3. **Rewrite** `user-vault.js` to use `SGVault.open()` / `SGVault.create()` with passphrase model
4. **New UX:** Vault key display, copy, save warning (replaces IndexedDB key management)

### 3.3 Blocked On

- Vault browser shared library updates (sections 2.2-2.6 above) must be complete first
- CLI must produce the new format for testing
- Decision on shared library location: currently in `sgraph_ai_app_send__ui__vault/_common/js/lib/`. The user vault is in a different UI package (`sgraph_ai_app_send__ui__user/`). Need a strategy for cross-package JS sharing.

## 4. What Solves Existing Problems

### 4.1 Browser 404s on CLI-Created Vaults

**Problem:** The v0.1.2 vault browser rejects vault IDs containing non-hex alphanumeric characters (regex `/^[0-9a-f]{8}$/`). CLI uses full `[a-z0-9]` character set.

**Fix in redesign:** Section 2.1 above — change regex to `/^[a-z0-9]{8}$/`. **One-line fix.**

### 4.2 Dual-Write Hack

**Problem:** The CLI (v0.8.5) writes entry-point files twice — once at the random ID and once at the HMAC-derived ID — because the browser can only discover files via HMAC, but the CLI stores them at random IDs.

**Fix in redesign:** Deterministic HMAC-derived IDs for all entry points. Both CLI and browser compute the same file_id from the vault key. No dual-write. The CLI team deletes all dual-write code in their Phase 7.

**Browser impact:** None — the browser already uses HMAC-derived IDs (via `SGVaultCrypto.deriveKeys()`). The fix is entirely on the CLI side.

### 4.3 Flat Tree Performance

**Problem:** The flat tree model stores every file as a flat entry with full path. Large vaults become slow — every commit rebuilds the entire tree, every merge compares every entry.

**Fix in redesign:** Sub-tree model. One tree per directory. On-demand loading. O(changed) merge instead of O(all files).

**Browser impact:** Significant — requires the sub-tree rendering changes in section 2.6. But the performance benefit is real: vault open goes from "load everything" to "load root level in 3 requests."

### 4.4 Plaintext Metadata Leak

**Problem:** Tree entries contain both plaintext fields (`name`, `size`, `content_hash`) and encrypted fields (`name_enc`, `size_enc`, `content_hash_enc`). The plaintext fields leak metadata to the server.

**Fix in redesign:** Drop all plaintext fields. Only `_enc` versions exist. The v0.1.2 browser already encrypts/decrypts `name_enc` — it just also writes the plaintext version. Removing the plaintext fields is straightforward in `_encryptTreeEntry()`.

## 7. Risks and Concerns

### 7.1 Breaking Change Coordination

Phase A changes (ID prefixes) make the vault browser incompatible with existing vaults. We need:
- Clear communication that this is intentional
- The CLI team to ship their changes in close sequence
- Test vectors to validate Python-JS interop before deploying

### 7.2 Flat-to-Sub-Tree Migration in Browser

The current `SGVault` class maintains an in-memory nested tree (`this._tree = { '/': { type: 'folder', children: {} } }`) that is fully populated at open time. The sub-tree model changes this to lazy loading. Every component that reads `vault._tree` directly (tree view, browser, stats) needs to become async-aware. This touches:

- `VaultTreeView._renderNode()` — accesses `this._vault._tree['/']` directly
- `VaultBrowser._renderFileList()` — calls `this._vault.listFolder()` (synchronous)
- `VaultBrowser._onDownload()` — calls `this._vault.getFile()` (already async)
- `SGVault.getStats()` — walks entire in-memory tree synchronously

**Recommendation:** Keep the in-memory tree as a cache. When `open()` loads only the root, populate the root level. When the user expands a folder, fetch the sub-tree, decrypt it, and insert it into the in-memory tree. All synchronous tree-reading code continues to work, but only for expanded folders. Components that need to show unexpanded folders display a "loading" state.

### 7.3 Batch Read Not Used by Browser Yet

The `SGSend` class (`sg-send.js`) currently uses individual `vaultRead()` calls (one HTTP request per read). The redesign's batch read API is already supported by the backend, but the browser JS client has no `batchRead()` method. Adding it would reduce vault open from 3 sequential requests to 2-3 (the ref-to-commit-to-tree dependency means we cannot batch all 3 into one request). This is an optimisation, not a blocker.

### 7.4 No `batchRead()` in SGSend JS Client

**File:** `sg-send.js`

Currently has `vaultWrite()`, `vaultRead()`, `vaultDelete()` — but no `vaultBatch()` or `batchRead()`. Needs adding:

```javascript
async vaultBatchRead(vaultId, fileIds) {
    const response = await this._fetch('POST', `/api/vault/batch/${vaultId}`, {
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({
            operations: fileIds.map(fid => ({ op: 'read', file_id: fid }))
        })
    })
    return response.json()
}
```

This is Phase B work — it helps but is not required for Phase A.

### 7.5 Cross-Package JS Sharing

The vault browser and user vault are in separate UI packages:
- `sgraph_ai_app_send__ui__vault/v0/v0.1/v0.1.2/_common/js/lib/sg-vault/`
- `sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.7/js/`

The redesign proposes the user vault imports from the vault browser's shared library. IFD methodology suggests the vault browser's `_common/` is the canonical location. The user vault would need a build step or a symlink/copy strategy to access those files. **Decision needed from Architect on cross-package sharing pattern.**

## 9. Open Questions for Team

1. **When does the CLI team expect Phase 5 (sub-tree model) to ship?** This gates our Phase B.

2. **Cross-validation test vectors:** Who creates the known vault key that produces known IDs in both Python and JS? This is critical for verifying interop before deployment.

3. **Cross-package JS sharing strategy:** How should `sgraph_ai_app_send__ui__user` import from `sgraph_ai_app_send__ui__vault/_common/js/lib/sg-vault/`? Options: symlink, copy-on-build, shared package, or move the shared lib to a third location.

5. **v0.1.3 disposition:** The v0.1.3 vault browser uses the flat model (incompatible with sub-tree). Per IFD, v0.1.2 overrides it. Should we document v0.1.3 as superseded?
