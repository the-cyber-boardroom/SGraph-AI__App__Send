## 3. Implementation Priority: Vault-Browse Unification (Doc 189)

**Complexity:** HIGH | **Value:** HIGH | **Risk:** MEDIUM

### What Needs to Be Built

Phase 1 (extraction):
- Extract `send-browse.js` and related files into a standalone `sg-browse` component
- Define the data source interface (`getTree()`, `getFile(path)`, `writeFile(path, blob)`)
- Implement zip data source adapter (existing logic, just wrapped)
- Verify no regression against existing send.sgraph.ai browse functionality

Phase 2 (vault adapter):
- Implement vault API data source adapter (tree from API, files on demand)
- Wire up to vault.sgraph.ai
- All recent Browse features (clickable links, inline images, CSV tables, `_page.json`) work automatically

Phase 3 (fractal vaults):
- Vault reference format (`__vault_ref__.json` or similar)
- Click-to-open-child-vault navigation in the tree

### Effort Estimate

| Phase | Effort | Risk |
|-------|--------|------|
| Phase 1: Browse extraction | 4–6 sessions | MEDIUM — regression risk on existing functionality |
| Phase 2: Vault adapter | 2–3 sessions | LOW — adapter is thin, Browse does the work |
| Phase 3: Fractal vaults | 1–2 sessions | LOW — once format defined |

### Risk: Regression

The Browse module is large and deeply integrated. Extraction must be surgical. Recommend:
1. Write comprehensive Browse tests before extraction (snapshot tests of existing functionality)
2. Extract incrementally — zip adapter first, verify parity with original, then publish
3. Do NOT change vault.sgraph.ai until Phase 1 is stable

## 4. Implementation Priority: sg-llm Component Family (Doc 202)

**Complexity:** MEDIUM per component | **Value:** HIGH | **Risk:** LOW

### Stream-Based Implementation Plan

**Stream A — Foundation (Prerequisite for everything else):**
| Component | Effort | Output |
|-----------|--------|--------|
| `sg-llm-events.js` | 0.5 session | Frozen constants file; simple to write, must get right |
| `sg-llm-request` v0.1.0 | 1 session | Fetch engine: POST to LLM API, stream response chunks |
| `sg-llm-output` v0.1.0 | 1 session | Streaming text display, handles chunks from event bus |

Stream A delivers: make a working LLM API call with streaming output. Can be tested independently.

**Stream B — Orchestration (Once Stream A exists):**
| Component | Effort | Output |
|-----------|--------|--------|
| `sg-llm-connection` v0.1.0 | 1 session | Provider/API key/model selector; no code dependency on Stream A |
| `sg-llm-reality` v0.1.0 | 2 sessions | Reality constructor; block editor; token counting; integrates A+B |

Stream B delivers: the full one-shot workflow with visible context construction.

**Stream C — Auxiliary (Can be added to any shipping tool):**
| Component | Effort | Output |
|-----------|--------|--------|
| `sg-llm-stats` v0.1.0 | 0.5 session | Metrics: tokens/s, cost, request time |
| `sg-llm-debug` v0.1.0 | 1 session | Full request inspector (request JSON, response JSON, timing) |
| `sg-llm-attachments` v0.1.0 | 1 session | File drop + clipboard paste + image cache |

**Stream D — Persistence and Specialised Output:**
| Component | Effort | Output |
|-----------|--------|--------|
| `sg-llm-bundle` v0.1.0 | 1.5 sessions | Save/load/replay execution bundles; uses sg-local-storage |
| `sg-llm-bundle-list` v0.1.0 | 1 session | Browse saved bundles (time travel); depends on sg-tree v0.1.0 |
| `sg-llm-infographic` v0.1.0 | 1 session | Extract SVG from LLM output, render inline |

**Total estimate:** 12–15 sessions for all 11 components, but Stream A alone (3 sessions) delivers 80% of the practical value.

### Key Technical Decisions

1. **`sg-llm-request` streaming**: The SSE/streaming API needs careful implementation. The `llm:request-chunk` event must fire for each token/chunk. Buffering strategy matters for performance.

2. **Token counting**: The brief mentions "token count per block visible". This requires a client-side tokeniser or a server-side estimation call. The simplest approach: rough estimate using character count × 0.25 (typical tokens-per-char ratio) with a note that it's approximate.

3. **Provider abstraction**: The connection component must support OpenRouter, direct Anthropic, direct OpenAI, and Ollama (local). The request component needs to format differently for each. A provider adapter pattern (same as the data source adapter for Browse) is the right approach.

## 5. Implementation Priority: sg-wasm + Audio Transcription (Doc 194)

**Complexity:** MEDIUM–HIGH | **Value:** HIGH | **Risk:** MEDIUM (research-dependent)

### sg-wasm Loader

**Pre-requisite research (before any code):**
- What Whisper WASM builds exist and what size are they? (whisper.cpp → WASM is likely)
- Does whisper.cpp support streaming transcription or batch only?
- What language support is available out-of-the-box?
- FFmpeg WASM binary size and loading time baseline

**Once research is done, sg-wasm implementation is straightforward:**
| Task | Effort |
|------|--------|
| Download manager (fetch WASM binary, progress callbacks) | 1 session |
| IndexedDB caching (hash-validated) | 1 session |
| Module instantiation (load WASM, set up memory, init) | 0.5 session (module-specific) |
| `sg-local-storage` integration for viewer | 0.5 session |

**Risk:** WASM binary size. If Whisper base model is 75–150 MB, the first download is slow. UX strategy: "Download once, use forever" messaging + persistent progress.

### Audio Transcription Component

Once sg-wasm and Whisper research are complete:
| Task | Effort |
|------|--------|
| Real-time pass (small model, live transcription) | 1–2 sessions |
| Clean-up pass (large model, timestamps, diarisation) | 1 session |
| SRT/VTT export | 0.5 session |
| Clickable timestamps (jump to audio position) | 0.5 session |

### Vault Migration

The brief describes `sgit-ai remote add/push/pull/sync`. Looking at the existing sgit-ai codebase, `sgit remote add/list` is already in the PROPOSED list (Section 17 of reality doc). This is:
- A CLI-side feature (sgit-ai repo, not SG/Send repo)
- Server-side: no new endpoints needed if vault read/write already works across servers
- Main technical challenge: sync conflict resolution when both local and remote have changes

**Effort estimate:** 2–3 sessions (CLI-side only; no SG/Send code changes needed)
