## Implementation Summary

Eight briefs landed between 10-12 March 2026. They span five themes: marketplace packaging, vault infrastructure hardening (bare vault unification, branching, remotes), vault UI components, Claude integration, and vault commercialisation. The briefs are heavily interdependent -- the branch model, CLI remotes, and bare vault unification form a tight cluster that should be built together. The marketplace and vault website/billing briefs are more independent but depend on the core vault work being stable.

**Existing foundation (from reality doc v0.13.22):**
- 22 User API endpoints, 51 Admin API endpoints (73 total)
- `sg-send-cli` v0.5.0 with clone/pull/push/status/init
- Vault Pointer model (write/read/read-base64/delete) on User Lambda
- Vault CRUD (create/lookup/folders/files/index/sharing) on Admin Lambda
- 584 passing tests, in-memory stack
- Milestones M-011 through M-013: vault round-trip with Claude, self-bootstrapping SKILL.md, vault init from CLI

**What does NOT exist yet (relevant to these briefs):**
- Branches, merge, signed commits
- Remotes abstraction in CLI
- Vault bundle/download endpoint
- Headless CLI mode
- Docker/AMI packaging
- Vault UI Web Components (`<sg-vault>`, pack themes)
- Vault billing/metering
- Vault website (separate repo)

### 5. Branch Model and Multi-User Collaboration (v0.13.30, 12 Mar)

**What exists:**
- Content-addressed storage (blob hashes)
- Tree structure (directory hierarchy)
- Commits (already shipped, per the brief's checklist)
- Vault Pointer model (single-writer via write_key)

**What needs building:**

| Work Item | Scope | Notes |
|-----------|-------|-------|
| Branch data structure (branch = identity + key pair + pointer to HEAD) | Large | Core vault format extension |
| Auto-create branch on clone | Medium | CLI generates key pair, registers branch with vault |
| Per-branch key pair generation and storage | Medium | Pluggable key storage (vault, localStorage, OS keychain) |
| Signed commits (branch key signs tree hash + parent + message + timestamp) | Large | Cryptographic signing in CLI, verification on server/CLI |
| Merge: automatic (non-conflicting) | Large | Three-way merge of vault trees |
| Merge: conflict detection and reporting | Large | Diff trees, identify conflicting files |
| `sg-send-cli log --graph` (ASCII branch/commit history) | Medium | Parse commit chain, render ASCII graph |
| Mermaid diagram generation | Small | Export commit history as Mermaid `gitGraph` |
| `<sg-vault-graph>` Web Component | Large | Branch/commit/merge visualisation in browser |
| Multi-vault view | Large | Multiple vaults side-by-side with independent branch views |
| Diff (show changes between commits) | Medium | Compare two tree states, output per-file diffs |

**Effort: XL**

This is the single largest brief in the batch. It transforms the vault from a blob store into a version control system. Branches + merge + signed commits is a substantial cryptographic and data structure project.

**Dependencies:**
- **Depends on Brief 4** (bare vault unification) -- need stable format before adding branches
- **Blocks Brief 2** (Lambda vault logging) -- signed commits needed for log entry signing
- **Tight coupling with Brief 6** (CLI remotes) -- branches need remote push/pull semantics

**Risks:**
- Merge is hard. Three-way merge of encrypted trees where file contents are ciphertext is particularly hard. The merge logic operates on tree structure (which files changed) not file contents (which are encrypted).
- Key storage across multiple backends (vault, localStorage, OS keychain) is a large surface area. Start with one backend.
- The "lightweight encrypted Git" target is ambitious. Git took years to mature its merge semantics.

**Testing strategy:**
- Unit: branch creation, commit signing, signature verification
- Integration: two independent clones, both commit, both push, merge into main
- Conflict: modify same file in two branches, verify conflict detection
- Visualisation: generate ASCII graph and Mermaid from known commit history, verify correctness
- Security: verify branch key isolation (branch A cannot sign as branch B)

## Recommended Build Order

The briefs have strong interdependencies. The recommended sequence:

```
Phase 1: Foundation (Weeks 1-2)
  [4] Bare Vault Unification  -- defines the format everything else builds on
  [2] Lambda Vault Logging     -- headless CLI mode (prerequisite for automation)
      (headless CLI only, defer signed logs to Phase 2)

Phase 2: Core Vault Features (Weeks 3-5)
  [5] Branch Model             -- branches + signed commits + merge
  [6] CLI Remotes              -- remote abstraction + local/S3/zip backends

Phase 3: Integration (Weeks 5-6)
  [7] Vault Access from Claude -- download endpoint + bundle format
  [2] Lambda Vault Logging     -- complete: signed logs using branch model
  [3] Vault UI Components      -- <sg-vault>, <sg-tree-view>, pack manifest

Phase 4: Commercialisation (Weeks 7-8)
  [8] Vault Website + Billing  -- separate repo, billing units, demo vaults
  [1] Marketplace Appliances   -- Docker containers, AMI pipeline
      (SG/Send ephemeral first, Ollama later)
```

**Rationale:**
- Brief 4 (unification) is the foundation. Every other vault brief assumes a stable format.
- Briefs 5 + 6 (branches + remotes) are tightly coupled and form the core of the "lightweight encrypted Git" target.
- Brief 7 (Claude access) is a quick win once the bundle format exists.
- Briefs 3 + 8 (UI + billing) depend on the data layer being stable.
- Brief 1 (marketplace) is the most independent but benefits from everything being stable.

## Open Questions for Architect

1. **File name encryption (Brief 4):** Are file names in the vault tree encrypted or plaintext? This is flagged in the brief as an open question. The answer affects the entire vault model, all UI components, and the merge logic (you cannot merge what you cannot read).

2. **Merge semantics on encrypted content (Brief 5):** The merge operates on tree structure (which files changed), not file contents (which are ciphertext). Confirm: is merge purely structural (file-level add/modify/delete), or do we need content-level merge for any scenarios?

3. **Branch key storage default (Brief 5):** The brief lists four key storage backends (vault, localStorage, keychain, meta-vault). Which is the default for the CLI? Which for the browser? We need a decision before implementation.

4. **Vault bundle size limits (Brief 7):** Lambda URL function responses cap at ~6MB. What is the strategy for vaults larger than 6MB? Presigned S3 URL redirect? Chunked download? This affects the endpoint design.

5. **Billing granularity (Brief 8):** The brief lists 6 billable units (vault count, pulls, pushes, commits, branches, storage). Is every unit metered from day one, or do we start with a subset (e.g., vault count + storage only)?

6. **Ollama appliance scope (Brief 1):** The Ollama appliance is a separate product surface with its own API (OpenRouter-compatible), model management, and bundling. Should this live in this repo or a separate repo? The SG/Send ephemeral appliance is clearly in-repo; the Ollama appliance may not be.

7. **CLI repo boundary (Briefs 2, 4, 5, 6):** Four of eight briefs require changes to `sg-send-cli`, which is an external PyPI package not in this repo. What is the coordination model? Do we need a mono-repo or at least synchronised releases?

8. **Nested vault recursion depth (Brief 4):** Vaults inside vaults. Is there a maximum nesting depth? Unlimited recursion with encrypted zip files could become a resource exhaustion vector.
