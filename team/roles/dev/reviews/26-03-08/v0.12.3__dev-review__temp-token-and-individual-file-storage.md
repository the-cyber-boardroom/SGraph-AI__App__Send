## Executive Summary

This plan covers two features from the 7 March brief:

1. **Part 1 (P0): Post-Purchase Temp Token** -- A `/welcome` page that auto-applies a shared temporary token from an encrypted SG/Send transfer. No new backend endpoints needed; uses existing upload/download infrastructure.

2. **Part 2 (P1): Individual File Storage** -- Store zip/folder uploads as a tree structure + individual encrypted blobs instead of a single blob. Requires coordinated backend and frontend changes.

Both features are **PROPOSED -- do not exist yet.**

## Part 1: Post-Purchase Temp Token (P0)

### What Exists Today (from reality document)

- **Upload/download endpoints:** `Routes__Transfers` with create, upload, complete, download, download-base64 (EXISTS)
- **Presigned download URLs:** `Routes__Presigned` with download-url (EXISTS)
- **Token validation:** `check-token/{name}` and `validate-token/{name}` endpoints (EXISTS)
- **Access gate component:** `send-access-gate.js` -- gates pages behind localStorage token (EXISTS)
- **API client:** `api-client.js` with `setAccessToken()`, `getAccessToken()`, `checkToken()` (EXISTS)
- **Static route setup:** `Fast_API__SGraph__App__Send__User.setup_static_routes()` with redirect routes (EXISTS)
- **Download page:** `/send/v0/v0.2/v0.2.0/en-gb/download/index.html` (EXISTS)
- **Welcome page:** DOES NOT EXIST

### Architecture Decision: No New Backend Code

The welcome page uses the **existing** SG/Send transfer infrastructure:

1. Human creates an access token (admin panel)
2. Human uploads the token string as an encrypted transfer via SG/Send
3. Transfer ID + decryption key form the welcome URL: `send.sgraph.ai/welcome#{transferId}/{decryptionKey}`
4. Welcome page JS: fetch encrypted blob -> decrypt -> extract token -> save to localStorage

The download endpoint (`/api/transfers/download/{id}`) does **not** require an access token -- only uploads do. So the welcome page can fetch the encrypted token transfer without authentication.

## Part 2: Individual File Storage (P1)

### What Exists Today (from reality document)

- **Single-blob storage:** `Transfer__Service.upload_payload()` stores one blob at `transfers/{id}/payload` (EXISTS)
- **SGMETA envelope:** Client packs filename + content into a binary envelope before encrypting (EXISTS)
- **Folder/zip upload:** `send-upload.js` has `_startFolderZip()` using JSZip to create a zip in-browser (EXISTS)
- **Zip browsing on download:** `send-download.js` has `_zipInstance`, `_zipTree`, zip file navigation (EXISTS)
- **Storage_FS:** `Storage_FS__Memory` and `Storage_FS__S3` backends via `file__save(path, bytes)` and `file__bytes(path)` (EXISTS)
- **Presigned multipart upload:** For files > direct upload limit (EXISTS)
- **Tree structure file storage:** DOES NOT EXIST
- **Individual file blob storage:** DOES NOT EXIST
- **Per-file download endpoint:** DOES NOT EXIST

### Architecture

#### Storage Layout (PROPOSED)

Current single-blob layout:
```
transfers/{transfer_id}/meta.json
transfers/{transfer_id}/payload
```

New individual-file layout (coexists with single-blob):
```
transfers/{transfer_id}/meta.json          # Extended with storage_mode field
transfers/{transfer_id}/tree.json.enc      # Encrypted tree structure
transfers/{transfer_id}/files/{blob_id}    # Individual encrypted file blobs
```

The `meta.json` gains a `storage_mode` field:
- `single_blob` (default, backward compatible) -- existing behaviour
- `individual_files` -- new tree + individual files mode

#### Tree Structure Schema (PROPOSED)

The tree structure is itself encrypted (the recipient needs the decryption key to see filenames). The plaintext tree structure before encryption:

```json
{
    "version": "1.0",
    "storage_mode": "individual_files",
    "total_files": 50,
    "total_size": 1073741824,
    "files": [
        {
            "path": "src/index.html",
            "size": 3501,
            "type": "text/html",
            "blob_id": "a1b2c3d4e5f6",
            "status": "uploaded"
        }
    ]
}
```

Fields `public_key`, `hash`, `extension` from the brief are deferred to P2 (per-file PKI). Keeping the initial schema minimal.

The `status` field per file enables live upload sharing: `pending` while uploading, `uploaded` when done. The tree is updated as each file completes.

#### Upload Flow (PROPOSED)

Browser-side (new):
1. User drops folder / selects zip
2. If file count <= 100: use individual file mode
3. If file count > 100: fall back to single-blob (existing zip flow)
4. Individual file mode:
   a. Generate one AES-256-GCM key (shared across all files in this transfer)
   b. Build tree structure JSON
   c. Encrypt tree structure
   d. Call `POST /api/transfers/create` with `storage_mode: "individual_files"` and `file_count: N`
   e. Upload encrypted tree: `POST /api/transfers/upload-tree/{transfer_id}`
   f. For each file (parallel, pool of 5):
      - Encrypt file with shared key
      - `POST /api/transfers/upload-file/{transfer_id}/{blob_id}`
   g. After all files uploaded: `POST /api/transfers/complete/{transfer_id}`

Server-side (new):
- `upload-tree` stores to `transfers/{id}/tree.json.enc`
- `upload-file` stores to `transfers/{id}/files/{blob_id}`
- `complete` verifies tree + all files present, marks transfer as completed
- `download-tree` returns just the encrypted tree (5KB, instant)
- `download-file` returns a single encrypted file blob

#### Download Flow (PROPOSED)

Browser-side (new):
1. Fetch transfer info -> check `storage_mode`
2. If `individual_files`:
   a. Fetch encrypted tree: `GET /api/transfers/download-tree/{transfer_id}`
   b. Decrypt tree with key from URL hash
   c. Render file list with three states: remote / in-browser / cached
   d. User clicks file -> fetch `GET /api/transfers/download-file/{transfer_id}/{blob_id}`
   e. Decrypt file -> display/preview/download
3. If `single_blob`: existing download flow (unchanged)

### Part 2 Migration Considerations

1. **Backward compatibility is automatic.** All existing transfers have no `storage_mode` field in meta.json. The service treats missing/null `storage_mode` as `single_blob`. No migration needed.

4. **Incremental rollout.** The upload UI can be feature-flagged: only show individual-file option when file count <= 100. Above 100, the existing zip path is used. This threshold can be raised later.

## Risk Assessment

| Risk | Mitigation |
|------|------------|
| Large file individual upload could hit Lambda 6MB response limit | Presigned URLs for files > 5MB (Task 2.4) |
| File count explosion (user uploads 10,000 files) | Hard limit at 100 files, fallback to single-blob above |
| Parallel upload failures (network errors on file 37 of 50) | Retry logic per file; if > 3 retries, mark file as failed; show partial upload state |
| Tree structure grows too large | At 100 files, tree JSON is ~10-20KB. Well within limits |
| IndexedDB storage limits | Browser-dependent (typically 50MB-unlimited). Start without caching, add in v0.2.1 |
| Welcome page token leakage | Accepted risk per brief. Shared temp token with limited credits. Human monitors + rotates |

## Open Questions for Architect

1. **Blob ID format:** The brief doesn't specify. Recommendation: use the same `secrets.token_hex(6)` (12-char random hex) as Transfer_Id. This is generated client-side and sent to the server. Is there a risk of collision? With 100 files per transfer, collision probability is negligible.

2. **Tree update endpoint:** The brief mentions live upload sharing (tree updated as files upload). Should the tree be re-uploaded after each file, or should the server maintain upload progress separately? Recommendation: server tracks `files_uploaded` count in meta; tree is uploaded once with all files listed as `status: "pending"`, and the download page polls transfer info to see progress.

3. **File size per blob:** Should there be a per-file size limit? Currently the total transfer size is limited. With individual files, a single 5GB file within a 50-file upload would need presigned URLs.

4. **Deferred to P2:** Per-file PKI keys, file update/versioning, hash verification. These are mentioned in the brief but explicitly marked as P2/future. This plan does not include them.
