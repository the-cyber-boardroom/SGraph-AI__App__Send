### Dependencies and Blockers

| Dependency | Status | Blocker? |
|------------|--------|----------|
| Decision on where Rosetta lives (this repo vs separate) | Not decided | **Soft blocker** -- can start in this repo and move later |
| Existing i18n JSON file locations | Known (in website assets) | No |
| Agent workflow changes (smaller tasks) | Process change, not code | No |

### Implementation Recommendation

The Rosetta archive is fundamentally a **directory structure + tooling** problem, not a server-side feature. The Dev role should:

1. Create the `rosetta/` structure in the repo root
2. Build the diff/status scripts as Python CLI tools (consistent with Python-first CI philosophy)
3. Integrate `status.json` generation into the CI pipeline (run on every push that touches locale files)
4. Leave the actual translation work to the Translator role and agents

The three-zone classification (Zone 1: tools, Zone 2: commercial, Zone 3: docs) is a policy decision, not a code decision. The Dev role implements it by ensuring the build pipeline treats zones differently -- Zone 3 content does not trigger translation tasks.

#### Implementation Quality Assessment

**Strengths:**
- Clean separation: Routes, Schema, Service layers follow project conventions.
- `Type_Safe` schema (not Pydantic) -- correct per project rules.
- `Service__Early_Access` extends `Type_Safe` -- correct.
- n8n webhook integration uses `urllib.request` (stdlib, no new dependency).

**No code changes recommended** for the current scope. When the SG WorkMail service (from v0.7.19 workstream D5) is available, `_call_webhook()` should be replaced with a direct email send.

## Architectural Notes for the Architect Role

1. **Stripe webhook on Admin Lambda** -- The brief specifies Admin Lambda. This is correct since the webhook will eventually trigger token creation, which is an Admin operation. The endpoint should be accessible without the standard Admin auth cookie, since Stripe authenticates via signature. The route should be explicitly excluded from cookie auth middleware.

2. **n8n as notification transport** -- Currently used for Early Access and proposed for payment notifications. This creates a single point of failure in n8n for all notification flows. The Architect should plan for the SG WorkMail FastAPI service (v0.7.19 D5) to replace n8n as the email transport.

3. **Rosetta archive location** -- The brief proposes `rosetta/` at repo root. This is reasonable for now. If the translation archive grows large (many languages, many keys), it could be extracted to a separate repo. The Dev role recommends starting in-repo and extracting later.

4. **Translation tool hosting** -- `sgraph.ai/tools/translations/` is consistent with the existing `sgraph.ai/tools/ssh-keygen/` pattern. No new domain or CloudFront distribution needed.
