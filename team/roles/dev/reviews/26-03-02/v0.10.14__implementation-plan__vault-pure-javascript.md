## Executive Summary

This plan implements a pure client-side encrypted vault (`vault.sgraph.ai`) that uses the existing `send.sgraph.ai` transfer API as its storage backend. The server has no concept of vaults -- all vault logic (folder trees, metadata, encryption key derivation) runs in the browser via JavaScript.

The plan breaks into six sections: (1) server-side path support, (2) `sg-send.js` client library, (3) `sg-vault.js` vault logic, (4) vault UI Web Components, (5) static site structure, and (6) sprint task ordering.

### Key Architectural Decisions

| Decision | Rationale |
|----------|-----------|
| Use User Lambda transfer API, not Admin Lambda vault API | The new vault deliberately avoids the admin-authenticated vault endpoints. It uses the public transfer API (with access tokens) as a dumb file store. This eliminates the PKI/RSA dependency and cookie-based auth. |
| PBKDF2 via Web Crypto for MVP KDF | PBKDF2 is natively available in Web Crypto API. Argon2 via WASM adds complexity and bundle size. PBKDF2 with 600,000 iterations is sufficient for MVP; Argon2 is a Phase 3 upgrade. |
| Reuse v0.2.0 UI architecture | The v0.2.0 User UI has the mature `SendComponent` base class (Shadow DOM, async resource loading, CSS/HTML/JS separation), `_common/` shared assets, `SendComponentPaths`, and 17-locale i18n. Build on this, not the older v0.1.x patterns. |
| Vault tree stored as a single encrypted transfer | The vault tree JSON is encrypted client-side and stored as a regular transfer on `send.sgraph.ai`. No special server support needed -- it is just another encrypted blob. |

## 1. Server-Side Changes

### 1.1. Path-Based File Organisation

**Current state:** Transfer IDs are 12-char hex strings (via `secrets.token_hex(6)`). The transfer service stores files at `transfers/{transfer_id}/meta.json` and `transfers/{transfer_id}/payload`. IDs are validated by FastAPI's `Safe_Str__Id` type, which accepts alphanumeric characters plus hyphens.

**Problem:** The brief requires file keys to be hierarchical paths (up to 5 levels of Object IDs separated by `/`), e.g. `a1b2c3d4/e5f6g7h8`. Currently `Safe_Str__Id` does not allow `/` characters.

**Proposed approach:** Rather than modifying the existing transfer ID validation (which would have security implications for the current transfer workflow), introduce path-based organisation as **metadata on the client side only**. The server continues to see flat transfer IDs. The vault tree JSON maps logical paths to transfer IDs.

**Why this is better:** The brief says "the server should have no idea it's hosting a vault." If we add server-side path support, the server IS aware of structure. Keeping everything flat maintains the zero-knowledge principle and requires **zero server-side changes** for MVP.

**If server-side paths are required in Phase 2** (e.g. for S3 prefix-based sharding), the change would be:

### 1.2. CORS Changes for vault.sgraph.ai

**Current state:** CORS is configured in `Fast_API__SGraph__App__Send__User.setup_middleware__cors()` at line 81-89 of `/home/user/SGraph-AI__App__Send/sgraph_ai_app_send/lambda__user/fast_api/Fast_API__SGraph__App__Send__User.py`. Currently uses `allow_origins = ["*"]`.

## 2. sg-send.js Client Library

### 2.1. Purpose

A standalone JavaScript module that abstracts the `send.sgraph.ai` API. Reusable across the vault, future data rooms, document workspaces, and any tool that needs to store/retrieve encrypted files.

### 2.2. API Design

```javascript
// ─── Construction ──────────────────────────────────────────────────
const sg = new SGSend({
    endpoint: 'https://send.sgraph.ai',
    token:    'my-access-token'
});

// ─── File Operations (transfer lifecycle) ─────────────────────────
const { transferId } = await sg.upload(encryptedData, {
    contentType: 'application/octet-stream',
    sizeBytes:   encryptedData.byteLength
});
const data           = await sg.download(transferId);
const info           = await sg.info(transferId);

// ─── Encryption ─────────────────────────────────────────────────
const key            = await sg.generateKey();
const encrypted      = await sg.encrypt(plaintext, key);
const decrypted      = await sg.decrypt(ciphertext, key);
const keyString      = await sg.exportKey(key);
const importedKey    = await sg.importKey(keyString);

// ─── Key Derivation (vault key -> AES key) ─────────────────────
const derivedKey     = await sg.deriveKey(vaultKeyString, salt);

// ─── Convenience: encrypt + upload in one call ──────────────────
const { transferId, key } = await sg.encryptAndUpload(plainData);
const plainData           = await sg.downloadAndDecrypt(transferId, key);
```

### 2.4. How It Talks to send.sgraph.ai

The library uses the standard `fetch` API. Key patterns:

| Concern | Implementation |
|---------|---------------|
| **Base URL** | Configurable: `new SGSend({ endpoint: 'https://send.sgraph.ai' })` |
| **Auth** | Header: `x-sgraph-access-token: {token}`. Matches existing `HEADER__SGRAPH_SEND__ACCESS_TOKEN` constant. |
| **Upload flow** | Three-step: `POST /api/transfers/create` -> `POST /api/transfers/upload/{id}` (raw bytes) -> `POST /api/transfers/complete/{id}` |
| **Download flow** | `GET /api/transfers/download/{id}` for files under 5MB; `GET /api/presigned/download-url/{id}` for larger files |
| **Error handling** | Check `response.ok`. On 401, emit `'auth-error'` event. On 413, auto-switch to presigned upload. On 404, return null. On network error, throw with descriptive message. |
| **Large files** | Auto-detect via `GET /api/presigned/capabilities`. If `multipart_upload` is true and file > 5MB, use `POST /api/presigned/initiate` -> PUT parts -> `POST /api/presigned/complete`. |
| **CORS** | All requests include `mode: 'cors'`. No cookies needed (header-based auth). |

### 2.5. Encryption Layer

Reuses the proven `SendCrypto` pattern from `/home/user/SGraph-AI__App__Send/sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.0/js/crypto.js`:

| Property | Value |
|----------|-------|
| **Algorithm** | AES-256-GCM |
| **Key length** | 256 bits |
| **IV length** | 12 bytes (random per encryption) |
| **Wire format** | `[12B IV][ciphertext + 16B auth tag]` |
| **Key export** | base64url (44 chars for 256-bit key) |
| **API** | Web Crypto (`crypto.subtle`) |

### 2.6. KDF Implementation

The vault key (a human-shareable string) must derive a deterministic AES-256 key:

```javascript
// PBKDF2 via Web Crypto API
async deriveKey(passphrase, salt) {
    const enc         = new TextEncoder();
    const keyMaterial = await crypto.subtle.importKey(
        'raw', enc.encode(passphrase), 'PBKDF2', false, ['deriveKey']
    );
    return crypto.subtle.deriveKey(
        { name: 'PBKDF2', salt: enc.encode(salt), iterations: 600000, hash: 'SHA-256' },
        keyMaterial,
        { name: 'AES-GCM', length: 256 },
        true,    // extractable -- needed for export
        ['encrypt', 'decrypt']
    );
}
```

**Salt strategy:** The salt is `sg-vault-v1:{vault_id}` where `vault_id` is the 8-hex-char Object ID of the vault container. This ensures the same passphrase produces different keys for different vaults.

**Why PBKDF2 over Argon2:**
- PBKDF2 is native Web Crypto -- zero dependencies, zero WASM
- 600,000 iterations with SHA-256 provides adequate protection for MVP
- Argon2 (memory-hard) is better against GPU attacks but requires a ~200KB WASM binary
- Phase 3 can add Argon2 as an optional upgrade without breaking existing vaults

### 3.2. Vault Creation

```javascript
static async create(sgSend, vaultKey, options = {}) {
    const vault = new SGVault(sgSend, vaultKey);

    // 1. Generate vault ID (8 hex chars)
    vault._vaultId = crypto.getRandomValues(new Uint8Array(4))
        .reduce((s, b) => s + b.toString(16).padStart(2, '0'), '');

    // 2. Derive AES key from vault key
    vault._derivedKey = await sgSend.deriveKey(vaultKey, `sg-vault-v1:${vault._vaultId}`);

    // 3. Create settings JSON
    vault._settings = {
        vault_name:  options.name || 'Untitled Vault',
        vault_id:    vault._vaultId,
        created:     new Date().toISOString(),
        owner:       options.owner || '',
        version:     1,
        description: options.description || '',
        // Store transfer IDs for tree and settings (self-referential after first save)
        _tree_transfer_id: null
    };

    // 4. Create initial empty tree
    vault._tree = {
        version: 1,
        updated: new Date().toISOString(),
        tree: {
            '/': { type: 'folder', children: {} }
        }
    };

    // 5. Encrypt and upload tree
    const treeData      = new TextEncoder().encode(JSON.stringify(vault._tree));
    const encryptedTree = await sgSend.encrypt(treeData, vault._derivedKey);
    const treeResult    = await sgSend.upload(encryptedTree);
    vault._treeTransferId = treeResult.transferId;

    // 6. Store tree transfer ID in settings, then encrypt and upload settings
    vault._settings._tree_transfer_id = vault._treeTransferId;
    const settingsData      = new TextEncoder().encode(JSON.stringify(vault._settings));
    const encryptedSettings = await sgSend.encrypt(settingsData, vault._derivedKey);
    const settingsResult    = await sgSend.upload(encryptedSettings);
    vault._settingsTransferId = settingsResult.transferId;

    // 7. The "vault key" the user shares is: vaultKey + ':' + settingsTransferId
    //    Or more elegantly: the settingsTransferId is encoded in the URL hash
    return vault;
}
```

### 3.3. Vault Opening

The vault key URL format: `https://vault.sgraph.ai/#<vaultKey>:<settingsTransferId>`

```javascript
static async open(sgSend, fullVaultKey) {
    // 1. Parse vault key: "passphrase:settingsTransferId"
    const colonIdx      = fullVaultKey.lastIndexOf(':');
    const vaultKey      = fullVaultKey.substring(0, colonIdx);
    const settingsId    = fullVaultKey.substring(colonIdx + 1);

    // 2. Download encrypted settings
    const encSettings   = await sgSend.download(settingsId);
    if (!encSettings) throw new Error('Vault not found');

    // 3. We need the vault ID for the KDF salt, but it is inside the encrypted
    //    settings. Solution: derive a "bootstrap key" using settingsTransferId as salt.
    //    The settings file itself stores the vault_id for subsequent KDF operations.
    //    Alternative: encode vault_id in the URL hash too.
    //
    //    Simplest approach: include vault_id in the shareable key:
    //    fullVaultKey = "passphrase:vaultId:settingsTransferId"

    const vault       = new SGVault(sgSend, vaultKey);
    vault._vaultId    = /* parsed from fullVaultKey */;
    vault._derivedKey = await sgSend.deriveKey(vaultKey, `sg-vault-v1:${vault._vaultId}`);

    // 4. Decrypt settings
    const settingsPlain = await sgSend.decrypt(encSettings, vault._derivedKey);
    vault._settings     = JSON.parse(new TextDecoder().decode(settingsPlain));
    vault._settingsTransferId = settingsId;

    // 5. Download and decrypt tree
    vault._treeTransferId = vault._settings._tree_transfer_id;
    await vault._loadTree();

    return vault;
}
```

**Vault Key Format (shareable):**
```
{passphrase}:{vault_id}:{settings_transfer_id}
```
Example: `my-secret-phrase:a1b2c3d4:7f8e9d0c1b2a`

This is the single string that grants access to the entire vault. The passphrase provides the entropy; the vault_id and settings_transfer_id are the locators.

### 3.4. File Operations

```javascript
async addFile(folderPath, fileName, fileData) {
    // 1. Encrypt file data
    const encrypted = await this._sgSend.encrypt(
        fileData instanceof ArrayBuffer ? new Uint8Array(fileData) : fileData,
        this._derivedKey
    );

    // 2. Upload encrypted file as a transfer
    const result    = await this._sgSend.upload(encrypted);
    const fileId    = result.transferId;

    // 3. Update tree: add file entry at folderPath
    const folder = this._findNode(folderPath);
    if (!folder || folder.type !== 'folder') throw new Error('Folder not found');

    folder.children[fileName] = {
        type:     'file',
        file_id:  fileId,
        size:     fileData.byteLength || fileData.length,
        mime:     '',     // caller can provide via options
        uploaded: new Date().toISOString(),
        versions: [{ file_id: fileId, date: new Date().toISOString() }]
    };

    // 4. Re-encrypt and re-upload tree
    this._tree.version += 1;
    this._tree.updated  = new Date().toISOString();
    await this._saveTree();

    return { fileId, fileName, folderPath };
}

async getFile(folderPath, fileName) {
    const folder = this._findNode(folderPath);
    if (!folder) throw new Error('Folder not found');
    const entry  = folder.children[fileName];
    if (!entry || entry.type !== 'file') throw new Error('File not found');

    // Download and decrypt
    const encrypted = await this._sgSend.download(entry.file_id);
    return this._sgSend.decrypt(encrypted, this._derivedKey);
}

async removeFile(folderPath, fileName) {
    const folder = this._findNode(folderPath);
    if (!folder) throw new Error('Folder not found');
    delete folder.children[fileName];
    // Note: the encrypted transfer is NOT deleted from the server (orphaned).
    // Cleanup is a Phase 3 concern.
    this._tree.version += 1;
    this._tree.updated  = new Date().toISOString();
    await this._saveTree();
}
```

### 3.5. Folder Operations

```javascript
createFolder(folderPath) {
    const parts  = folderPath.split('/').filter(Boolean);
    const name   = parts.pop();
    const parent = this._findNode('/' + parts.join('/'));
    if (!parent || parent.type !== 'folder') throw new Error('Parent folder not found');
    if (parent.children[name]) throw new Error('Already exists');

    parent.children[name] = { type: 'folder', children: {} };
    this._tree.version += 1;
    this._tree.updated  = new Date().toISOString();
    return this._saveTree();
}

listFolder(folderPath) {
    const node = this._findNode(folderPath);
    if (!node || node.type !== 'folder') return null;
    return Object.entries(node.children).map(([name, entry]) => ({
        name,
        type:     entry.type,
        size:     entry.size || 0,
        uploaded: entry.uploaded || null,
        versions: entry.versions ? entry.versions.length : 0
    }));
}
```

### 3.6. Tree Management

```javascript
_findNode(path) {
    const parts = path.split('/').filter(Boolean);
    let node    = this._tree.tree['/'];
    for (const part of parts) {
        if (!node || node.type !== 'folder' || !node.children[part]) return null;
        node = node.children[part];
    }
    return node;
}

async _loadTree() {
    const encrypted = await this._sgSend.download(this._treeTransferId);
    const plain     = await this._sgSend.decrypt(encrypted, this._derivedKey);
    this._tree      = JSON.parse(new TextDecoder().decode(plain));
}

async _saveTree() {
    const treeData      = new TextEncoder().encode(JSON.stringify(this._tree));
    const encrypted     = await this._sgSend.encrypt(treeData, this._derivedKey);
    const result        = await this._sgSend.upload(encrypted);

    // Update settings with new tree transfer ID
    const oldTreeId     = this._treeTransferId;
    this._treeTransferId = result.transferId;
    this._settings._tree_transfer_id = this._treeTransferId;

    // Re-upload settings (pointing to new tree)
    const settingsData  = new TextEncoder().encode(JSON.stringify(this._settings));
    const encSettings   = await this._sgSend.encrypt(settingsData, this._derivedKey);
    const settingsResult = await this._sgSend.upload(encSettings);
    this._settingsTransferId = settingsResult.transferId;

    // Note: the vault key URL must be updated to point to the new settings transfer ID.
    // This is a fundamental trade-off: immutable transfers mean every tree update changes
    // the "entry point" transfer ID.
    //
    // SOLUTION: Use a "pointer" transfer that always points to the latest settings.
    // The pointer transfer ID is stable and stored in the vault key.
    // Phase 1 workaround: the vault key in the URL hash is updated after each write.
    // The user's bookmarked URL always works because the hash is updated via replaceState.
}
```

### 3.7. The Pointer Problem

Every tree update creates a new transfer (transfers are immutable). This means the settings transfer ID changes, which means the vault key changes. This is a problem for sharing.

**Solution: Stable Pointer Transfer**

Create a special "pointer" transfer on vault creation. This transfer's content is the encrypted current settings transfer ID. On every tree update, the pointer is updated (uploaded as a new transfer, but the pointer ID remains the concept).

Wait -- transfers are immutable. We cannot update in place.

**Revised solution:** Use the existing Admin Lambda vault endpoints for the pointer ONLY. The Admin vault API supports `POST /vault/index` which can be updated in place. Store a single encrypted "pointer" in the vault index, containing the latest settings transfer ID.

**Simplest Phase 1 solution:** Accept that the vault key includes the settings transfer ID and changes on every write. The vault UI updates `window.location.hash` via `history.replaceState()` after every write. The user always sees the current key. Sharing requires copying the current key.

**Phase 2:** Introduce a stable "locator" endpoint (a new route on User Lambda: `POST /pointers/update` and `GET /pointers/{id}`) that stores a mutable 64-byte payload. This is a minimal addition -- just a KV store for transfer ID pointers.

#### `<vault-upload>` -- File Upload

**Behaviour flow:**
1. User selects file(s) or drops them on the drop zone
2. For each file:
   a. Read file as ArrayBuffer
   b. Encrypt with vault's derived AES key (`sg-send.js` encrypt)
   c. Upload encrypted blob via `sg-send.js` upload
   d. Add entry to vault tree
   e. Re-encrypt and re-upload tree
3. Show progress (encrypting... uploading... done)
4. Refresh folder view

**Size handling:**
- Under 5MB: direct upload via `POST /transfers/upload/{id}`
- 5MB-1GB: multipart via presigned URLs (auto-detected by `sg-send.js`)
- Over 1GB: reject with error message

#### `<vault-preview>` -- File Preview (Phase 2)

Not in Phase 1 scope. Phase 2 will add inline preview for:
- Images (blob URL + `<img>`)
- PDF (blob URL + `<iframe>` or `<embed>`)
- Markdown (parse + render)
- HTML (sandboxed `<iframe>`)
- Text (plain `<pre>`)

#### `<vault-share>` -- Share Vault Key

Simple component that displays the current vault key and provides:
- Copy to clipboard button
- QR code rendering (same `qr-code.js` pattern from admin v0.1.3)
- Security warning: "Anyone with this key can access all files in this vault"

## Appendix B: Security Considerations

| Concern | Mitigation |
|---------|-----------|
| **Vault key strength** | PBKDF2 with 600k iterations provides ~87 bits of brute-force resistance for a 4-word passphrase. UI should encourage 6+ word passphrases or 128-bit random keys. |
| **KDF salt predictability** | Salt is `sg-vault-v1:{vault_id}`. The vault_id is random (8 hex chars = 32 bits). This prevents rainbow tables but is predictable per-vault. Acceptable for MVP. |
| **Tree update atomicity** | Every tree update creates new transfers for tree + settings. If the browser crashes mid-update, the old tree (via old settings transfer) remains valid. No data loss. |
| **Orphaned transfers** | Deleting a file from the tree does NOT delete the encrypted transfer from the server. The server stores orphans indefinitely. Phase 3 should add a garbage collection mechanism. |
| **Replay attacks** | An attacker with a previous vault key (before a passphrase change) can access files encrypted with the old key. Phase 3 should add key rotation with re-encryption. |
| **Server-side correlation** | The server can see that multiple transfers are uploaded by the same IP/token in succession (suggesting a vault). This is acceptable given the zero-knowledge principle -- content remains encrypted. |
| **Vault key in URL hash** | The hash fragment is never sent to the server (HTTP spec). It IS visible in browser history, bookmarks, and screen sharing. This matches the existing send.sgraph.ai key-in-hash pattern. |
