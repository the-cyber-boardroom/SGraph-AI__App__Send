### Encryption Architecture

```
Passphrase + VaultID → PBKDF2 (600k iterations, SHA-256) → AES-256-GCM key
                                                             │
                                     ┌───────────────────────┼───────────────────────┐
                                     ▼                       ▼                       ▼
                              encrypt(settings)       encrypt(tree)           encrypt(files)
                                     │                       │                       │
                                     ▼                       ▼                       ▼
                              upload as transfer      upload as transfer      upload as transfer
```

**Vault key format:** `passphrase:vaultId:settingsTransferId`
**KDF salt:** `sg-vault-v1:{vaultId}`

### API Integration

Uses `send.sgraph.ai` User Lambda transfer endpoints:
- `POST /api/transfers/create` → get transfer_id
- `POST /api/transfers/upload/{id}` → upload encrypted bytes
- `POST /api/transfers/complete/{id}` → finalise
- `GET /api/transfers/download/{id}` → get encrypted bytes

Auth: `x-sgraph-access-token` header (sent on every request via `SGSend._authHeaders()`)

## What to Work On Next

### Priority 1: i18n Translations
The 4 new access key strings (`access_label`, `access_placeholder`, `access_hint`, `error.no_access_key`) only exist in `en-gb.json`. They need translating to the other 16 locales. The i18n system uses `generate_vault_i18n_pages.py` to pre-render locale pages.

### Priority 2: Token Validation UX
Currently no feedback after pasting the access key. Options:
- Call `/api/health` or a lightweight endpoint to verify the token works
- Show green/red indicator next to the input

### Priority 3: Token via URL Parameter
Support `?token=...` in the URL for deep-link scenarios where the access token is provided externally (e.g., from an admin panel link).

### Priority 4: Error Handling & Edge Cases
- Network failure recovery (retry logic in `SGSend._fetch()`)
- Large file handling (presigned uploads not yet wired)
- Very large vaults (1000+ files) — tree rendering performance
- Browser compatibility checks (Web Crypto availability fallback)

### Phase 2 (Deferred)
- Server-side path-based storage (avoid tree bloat)
- Presigned multipart uploads for files >5MB
- Multi-user vault sharing

## Known Issues / Quirks

1. **Tree re-upload on every mutation** — Every file add/delete re-encrypts and re-uploads both the tree and settings. This creates orphaned transfers (old versions). The server doesn't know they're orphaned. This is fine for MVP but needs a cleanup mechanism later.

2. **Vault key changes after every mutation** — Because `settingsTransferId` changes, the vault key in the URL hash updates. If the user shared the old key, it still works because the old settings transfer still exists and points to the correct tree at that point in time. But any files added after the share won't be visible.

3. **No offline support** — Every operation hits the API. No service worker or local cache.

4. **`data-endpoint` attribute defaults to `https://send.sgraph.ai`** — For local development, set `data-endpoint="http://localhost:8000"` on the `<vault-entry>` component.
