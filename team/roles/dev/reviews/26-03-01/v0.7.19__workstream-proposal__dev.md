## D2 — Static Assets to CDN: Move send.sgraph.ai Static Files off Lambda

**Priority:** P0
**Effort:** M

### Description

All 18 requests per page load of the send.sgraph.ai UI currently go to the FastAPI Lambda function, including static HTML, CSS, JS, and font files. The performance baseline (Brief 7) shows 18.5 s cold start because Lambda must initialise before the first byte of HTML arrives. After CDN migration, static assets would be served directly from CloudFront with near-zero cold start penalty. The Lambda would only serve API calls (transfers, tokens, presigned URLs).

The deployment infrastructure (`scripts/deploy_static_site.py`) and the S3/CloudFront pipeline already exist for sgraph.ai. The task is to apply the same pipeline to the User UI (`sgraph_ai_app_send__ui__user/`) and Admin UI (`sgraph_ai_app_send__ui__admin/`) and re-point CloudFront to serve those paths from S3 instead of proxying to Lambda.

### Rationale

This is the single largest performance improvement available. The baseline shows warm-state static resource overhead of ~170 ms per file — overhead that disappears when files are on CDN. Cold start drops from 18.5 s to ~2–3 s (only the first API call waits for Lambda). This also reduces Lambda invocation cost and eliminates the entire class of "Lambda timeout on static file" failure.

From a code perspective, this is a configuration and deployment script change, not a code change. The deploy_static_site.py script already implements the correct S3 folder layout and CloudFront cache control headers.

### Rationale for P0

The performance baseline brief exists specifically to document the "before" state. The improvement must happen soon or the baseline loses relevance. Additionally, D1 (Google Fonts fix) only fully works if fonts are served locally — once on CDN, the font files travel with the UI assets, not through Lambda.

### Dependencies

- Depends on: D1 (Google Fonts fix should be in place first — don't CDN a version that still loads from Google)
- Depends on: deploy-website.yml workflow pattern being callable for send.sgraph.ai assets
- Unblocks: P1 (One-Time Secrets UI), P2 (Password Generator UI), any future User UI work — all will benefit from CDN delivery

## D3 — Wire deploy-website.yml to Call Python CI Scripts

**Priority:** P0
**Effort:** S

### Description

The three Python CI scripts (`scripts/deploy_static_site.py`, `scripts/generate_i18n_pages.py`, `scripts/store_ci_artifacts.py`) implement the target "Python-first CI" architecture from the arch brief. However, the reality document notes that the `deploy-website.yml` GitHub Actions workflow may still contain inline YAML logic (aws s3 sync commands, cache control headers) rather than calling these Python scripts.

This work stream verifies the current state of `deploy-website.yml`, identifies any remaining inline logic, and refactors the workflow to use the Python scripts as the single source of deployment logic. GitHub Actions becomes a trigger only — it sets environment variables and calls the scripts.

### Rationale

The "Python-first CI" rule is a core architectural requirement from the v0.7.6 arch brief. Python code is testable, version-controlled, readable, and reproducible outside GitHub. YAML logic is not. Any deployment logic that lives only in YAML is a maintenance risk: it cannot be unit-tested, cannot be run locally without the GitHub runner, and duplicates logic that the Python scripts already implement. This work stream closes the gap.

Additionally, if we run the same Python scripts locally and in CI, the deployment is auditable end-to-end. This matters for multi-cloud: the same script can target AWS, GCP, Azure, or Cloudflare by switching provider config.

### Dependencies

- Depends on: Python scripts already existing (they do — v0.7.6 delivered them)
- Unblocks: D4 (multi-cloud deployment), testable CI pipeline

## D4 — Absolute Path Audit and Fix in sgraph.ai Website

**Priority:** P1
**Effort:** S

### Description

The sgraph.ai website uses absolute paths for navigation links (`href="/"`, `href="/product/"`, etc.) in all 8 HTML pages. The arch brief requires all links to be relative for versioned deployments and offline packaging. Absolute paths break when the site is served from a path prefix (e.g., `releases/v0.7.19/`) or packaged for offline use (e.g., a USB drive or a docker image without a web server).

The fix is mechanical: traverse all HTML files in `sgraph_ai__website/` and replace absolute navigation hrefs with relative equivalents appropriate to each page's depth in the directory tree.

### Rationale

This is a prerequisite for offline packaging (D6). If navigation links are absolute, an offline bundle cannot function as a self-contained unit — clicking "Home" will try to hit the absolute path, which either resolves to the user's file system root or fails. The arch brief is explicit: all internal links must be relative.

The fix is also testable: the `deploy_static_site.py` script already validates internal links. We should extend its validation to reject absolute navigation hrefs as part of the CI gate.

### Dependencies

- Depends on: nothing
- Unblocks: D6 (offline packaging), D7 (multi-cloud distribution)

The main new work is: a cleaned-up API surface (POST `/secrets/create`, GET `/secrets/read/{id}`), a minimal dedicated UI, and time-based expiry.

## P4 — CloudFront Log Viewer Tool: `tools.sgraph.ai/cf-logs`

**Priority:** P2
**Effort:** M

### Description

A client-side CloudFront log viewer. User drags a `.gz` file (or pastes plaintext) from the CloudFront real-time logging pipeline into the browser. The tool detects whether the file is actually gzip-compressed (magic byte check — because the Kinesis Firehose pipeline has a known bug writing plaintext with `.gz` extension), parses the CloudFront tab-separated log format into labelled columns, and displays a sortable/filterable table with:

- Human-readable timestamps (not epoch)
- Decoded URL paths
- Colour-coded status codes (2xx green, 4xx amber, 5xx red)
- Edge location IATA code → city name lookup
- Summary stats: total requests, cache hit ratio, error rate, average TTFB

All processing is client-side. Log data never leaves the browser.

**Reality check:** This feature does NOT exist. No code.

### Rationale

The brief documents a real operational pain point: the AWS Console log viewer is limited (LIMIT 5 truncation, silent), and downloading logs fails because `.gz` files are not actually gzip-compressed. This is a genuine productivity problem that affects DevOps and Developer roles every time they need to investigate a production issue.

Because this is a pure client-side static tool, the implementation risk is low: no backend, no auth, no AWS credentials. The gzip magic byte detection is a few lines of JavaScript. The CloudFront log format is documented and stable. The IATA-to-city lookup is a static JSON file. This is achievable in 2–3 days.

Note: the Kinesis Firehose pipeline bug (`.gz` extension on uncompressed files) is a separate DevOps action (not a Dev work stream) and should be fixed independently of this tool.

### Dependencies

- Depends on: P3 (tools.sgraph.ai domain/path decision — same infrastructure)
- Unblocks: better operational visibility for the team, directly supports the logging requirements from the arch brief

## D8 — Offline Packaging: Self-Contained Deployment Folder

**Priority:** P2
**Effort:** M

The arch brief describes this as foundational to the multi-cloud strategy. Until offline packaging works, multi-cloud distribution is just "sync to multiple S3-compatible buckets" — valuable, but not the full vision.

## D9 — Credit Expiry: Time-Based Token Invalidation

**Priority:** P2
**Effort:** M

### Description

The current token schema has `max_uses` (usage-capped) and `is_active` (revocable), but no time-based expiry. The pricing brief specifies 3-month credit expiry. Implement a `expires_at` field on the token schema and enforce it during token validation: if `now() > expires_at`, the token is treated as exhausted.

This requires:
1. Schema addition (`expires_at: datetime | None`) on the Token schema
2. `Token_Service.validate_token()` to check expiry as a second condition alongside usage limit
3. `Token_Service.create_token()` accepts optional `ttl_days` parameter to set expiry at creation
4. Update tests to cover expiry paths (token created with 0-day TTL is immediately invalid, etc.)

### Rationale

The 3-month expiry is not optional — it is part of the stated pricing model. Without it, a GBP 5 Starter Pack never forces a re-purchase decision, defeating the stated purpose of the credit model ("the most important question is 'will they buy more?'"). Expiry creates the re-purchase moment.

The implementation is self-contained: it is a schema and service layer change with no external dependencies. The existing test suite can be extended in-place.

### Dependencies

- Depends on: nothing (standalone backend change)
- Unblocks: B2 (Stripe webhook creates tokens with expiry), correct commercial pricing model
