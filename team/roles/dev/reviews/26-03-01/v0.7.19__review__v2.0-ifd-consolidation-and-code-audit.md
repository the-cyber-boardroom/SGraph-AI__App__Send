## 1. Executive Summary

The v2.0 IFD consolidation is a clean operation. The symmetric key features (upload, download, preview) are architecturally distinct from vault/PKI/data rooms. The current v0.1.8 UI has 9 IFD layers that will be collapsed into a single coherent codebase. The hardest engineering task is merging the send-upload (5 override layers) and send-download (5 override layers, 740 lines of patches on a 400-line base) components.

The API split is equally clean: 14 transfer/presigned endpoints are purely symmetric-key. 17 vault and 3 join endpoints are clearly scoped and conditionally mounted.

## 4. Static vs Dynamic Analysis

### 4.1. Everything is Static

The entire UI is purely static content:
- **Zero server-side rendering** -- Lambda serves files as-is via Starlette StaticFiles
- **No template engine** -- no Jinja2, no server-side substitution
- **All dynamic behavior is client-side** -- Web Components, fetch() calls, Web Crypto API

This means moving to S3 changes nothing about how the UI works. The UI doesn't care where its files come from.

### 4.2. API Call Pattern

All API calls use **same-origin relative paths**:

```javascript
// api-client.js (v0.1.0)
const ApiClient = {
    baseUrl: '',  // Same origin
    async createTransfer(fileSizeBytes, contentTypeHint) {
        const response = await fetch(`${this.baseUrl}/transfers/create`, { ... });
    }
};
```

**With CloudFront fronting both S3 and Lambda on the same domain, this works unchanged.** The empty `baseUrl` means `fetch('/transfers/create')` goes to `send.sgraph.ai/transfers/create`, which CloudFront routes to Lambda.

### 4.3. Download URL Construction

The send-upload component constructs download URLs:

```javascript
// v0.1.8 send-upload.js
`${window.location.origin}/send/v0/v0.1/v0.1.8/download.html#${tid}/${key}`
```

**For v2.0, this changes to:**
```javascript
`${window.location.origin}/download/#${tid}/${key}`
```

The CloudFront Function rewrites `/download/` to `/download/index.html`.

### 4.4. i18n JSON Loading

Currently fetches at runtime: `fetch('../v0.1.4/i18n/{code}.json')`

**For v2.0, two approaches (both work):**
1. Keep runtime fetch but change path to `/_common/i18n/{code}.json`
2. Embed translations in `<script>` tag during pre-rendering (recommended for performance)

## 7. Lambda Code Changes

### 7.1. Remove Static Serving

In `Fast_API__SGraph__App__Send__User.py`:
- Remove `setup_static_routes()` method (lines 170-196)
- Remove `StaticFiles` import and mount
- Remove redirect routes (`/`, `/send`, `/tools/ssh-keygen`)

### 7.2. Conditional Vault/Join Routes

The vault and join routes are already conditionally mounted. For v2.0, they can remain on the Lambda (other clients may still use them) but are not exposed via the Send UI. In a later phase, they move to their own Lambda on a different domain.

### 7.3. CORS Tightening

Current: `allow_origins = ["*"]`
v2.0: `allow_origins = ["https://send.sgraph.ai"]`

### 7.4. CloudFront Secret Validation

Add middleware to validate `X-CloudFront-Secret` header:

```python
@app.middleware("http")
async def validate_cloudfront_origin(request, call_next):
    if request.url.path == "/info/health":
        return await call_next(request)  # Health checks bypass
    expected = os.environ.get("CLOUDFRONT_SECRET")
    if expected and request.headers.get("x-cloudfront-secret") != expected:
        return JSONResponse(status_code=403, content={"error": "Direct access not allowed"})
    return await call_next(request)
```
