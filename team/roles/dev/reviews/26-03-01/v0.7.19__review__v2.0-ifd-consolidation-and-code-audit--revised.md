**Date:** 2026-03-01

## 1. Executive Summary

The v0.2.0 IFD consolidation is a clean operation. The symmetric key features (upload, download, preview) are architecturally distinct from vault/PKI/data rooms. The current v0.1.8 UI has 9 IFD layers that will be collapsed into a single coherent codebase at v0.2.0.

**Critical design constraint:** v0.2.0 must be structured for easy IFD progression. Once v0.2.0 is complete, that code is frozen. All subsequent features, improvements, bug-fixes and changes go into v0.2.1, v0.2.2, v0.2.3 etc. This means components must have clean, overridable method boundaries — not monolithic scripts.

The API split is equally clean: all routes move under `/api/` prefix. No backward compatibility needed — there are no live customers.

### What Changed from v1 of this Review

| Area | v1 | Revised | Reason |
|------|-----|---------|--------|
| IFD version | v2/v2.0/v2.0.0 | v0/v0.2/v0.2.0 | Designed for IFD progression (v0.2.x) |
| API prefix | Keep current paths | Move all under `/api/*` | No live customers, cleaner architecture |
| Page layout | Independent | Match sgraph.ai wider layout | Consistent cross-site flow |
| Component design | Single clean class | Clean class + overridable methods | Must support v0.2.x IFD progression |
| Backward compat | Legacy redirects, old URL support | Not needed | No live customers |
| IFD file concern | Worry about many files | Not a concern | S3 hosting loads everything quickly |

## 4. IFD Consolidation Detail

### 4.1. Design for v0.2.x Progression

**This is the most important design constraint.** v0.2.0 code will be frozen. All future changes happen in v0.2.1+, using IFD override patterns. This means:

1. **Each Web Component class must have clear, named methods** that v0.2.x can override
2. **No anonymous closures or IIFEs** — everything must be accessible via `ClassName.prototype.methodName`
3. **Template/render methods must be separate** from logic methods — so UI changes don't require re-implementing business logic
4. **CSS must use design tokens** — so v0.2.x can restyle without copying all styles
5. **Event handlers must be named methods** — not inline lambdas — so they can be overridden

**Example of good v0.2.0 structure (designed for override):**

```javascript
class SendUpload extends HTMLElement {
    // Template — overridable in v0.2.x for layout changes
    get template() { return `<div class="upload-zone">...</div>`; }

    // Lifecycle — clear entry points
    connectedCallback()    { this.render(); this.setupListeners(); }

    // Render — overridable for UI changes
    render()               { this.innerHTML = this.template; }

    // Logic — overridable for behavior changes
    async handleFileSelect(file)   { ... }
    async encryptFile(file)        { ... }
    async uploadEncrypted(data)    { ... }
    updateProgress(percent)        { ... }
    showResult(transferId, key)    { ... }

    // Event setup — overridable for new interactions
    setupListeners() { ... }
}
```

**v0.2.1 can then surgically override:**
```javascript
// v0.2.1 — only changes what's needed
const _origTemplate = Object.getOwnPropertyDescriptor(SendUpload.prototype, 'template');
Object.defineProperty(SendUpload.prototype, 'template', {
    get() { return `<div class="upload-zone-v2">...</div>`; }
});
```

## 5. Design: Wider Layout Matching sgraph.ai

### 5.1. Layout Changes

The v0.1.x layout uses a narrow, centered card. The v0.2.0 layout matches sgraph.ai's wider page structure:

- **Full-width header** with navigation (logo, link back to sgraph.ai)
- **Wider content area** — matches sgraph.ai's max-width and padding
- **Consistent typography** — same font sizes, weights, and spacing as sgraph.ai
- **Same design tokens** — colors, shadows, border-radius from sg-design-system.css
- **Footer** — with link back to sgraph.ai for documentation, how-tos, etc.

### 5.2. send.sgraph.ai Stays Focused

The UI on send.sgraph.ai is 100% focused on:
1. **Sending files** — upload, encrypt, get link
2. **Receiving files** — enter link, decrypt, download/preview

No documentation pages. No how-to guides. No team pages. No pricing. Those live on sgraph.ai. The send UI links back to sgraph.ai for all that content.

### 6.4. CloudFront Secret Validation

```python
@app.middleware("http")
async def validate_cloudfront_origin(request, call_next):
    if request.url.path == "/api/info/health":
        return await call_next(request)  # Health checks bypass
    expected = os.environ.get("CLOUDFRONT_SECRET")
    if expected and request.headers.get("x-cloudfront-secret") != expected:
        return JSONResponse(status_code=403, content={"error": "Direct access not allowed"})
    return await call_next(request)
```

## 7. Version Manifest

Each IFD version includes a `manifest.json` that lists every file that should be copied to `latest/`. See Architect review for full specification.

This is important for:
1. **CI pipeline simplicity** — read manifest, sync listed files, done
