## 2. Quick Wins -- Implementation Sketches

### Brief #25: Drag-and-Drop Encrypted Email Workflow

**What it is:** A local encrypt/decrypt workflow where the server is never involved. The user encrypts a file in the browser using a recipient's public key, downloads the `.sgenc` file, and sends it via any channel (email, Slack, WhatsApp). The recipient drops the file into the browser to decrypt it.

**What exists today:**
- `SendCrypto` (`v0/v0.1/v0.1.0/js/crypto.js`) -- AES-256-GCM symmetric encryption
- `VaultCrypto` (`v0/v0.1/v0.1.7/js/vault-crypto.js`) -- RSA-OAEP hybrid encryption with IndexedDB key storage
- `SSHCrypto` (`tools/ssh-keygen/v0/v0.1/v0.1.0/js/ssh-crypto.js`) -- Ed25519 and RSA-4096 key generation in OpenSSH format
- `send-upload` and `send-download` Web Components across multiple versions

**Implementation sketch:**

```
New UI component: <send-encrypt-local>
  Location: sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/components/send-encrypt-local/

  1. Encrypt mode:
     - User selects recipient public key (from contacts, by lookup code, or paste)
     - User drops a file
     - Browser encrypts file using VaultCrypto.encrypt(recipientPublicKey, fileBytes)
     - Browser packages: { format: 'sgenc-v1', encrypted_data, sender_fingerprint }
     - User chooses file naming level: named/type-only/opaque
     - Downloads the .sgenc file

  2. Decrypt mode:
     - User drops a .sgenc / .age / .gpg file
     - Browser detects format from magic bytes / extension
     - For .sgenc: VaultCrypto.decrypt(privateKey, packedData)
     - For .gpg: OpenPGP.js (new dependency)
     - For .age: age-encryption (new dependency, or TweetNaCl for X25519)
     - Decrypted file offered for download

  3. Template emails:
     - Generate copy-paste templates (pure JS string templates)
     - Include the recipient's lookup code in the template
```

**New files needed:**
- `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/components/send-encrypt-local/send-encrypt-local.js`
- `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/encrypt.html`
- `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/js/sgenc-format.js` (format definition for `.sgenc`)

**Backend changes:** None. This is entirely client-side. The server never sees the file.

**Key decision needed:** The `.sgenc` file format specification. Proposal:
```
Bytes 0-5:    Magic bytes "SGENC\x01" (format identifier + version)
Bytes 6-9:    uint32 LE -- metadata length
Bytes 10-N:   JSON metadata (sender fingerprint, timestamp, naming level)
Bytes N+1-M:  VaultCrypto packed blob [4B wrapLen][wrappedKey][12B IV][ciphertext]
```

**Third-party libraries (browser):**
- OpenPGP.js (~200KB) -- for .gpg format support (defer to Phase 2)
- TweetNaCl.js (~20KB) -- for .age/X25519 support (defer to Phase 2)

**Effort: M (Medium).** Core `.sgenc` encrypt/decrypt is 2-3 days. Multi-format support (.age, .gpg) adds 3-5 days. Template emails are ~1 day.

### Brief #26: Key Portability and Password Manager Integration

**What it is:** Allow users to import/export their private key across devices. Support drag-and-drop import, clipboard paste, password manager autofill, and passphrase-encrypted export.

**What exists today:**
- `VaultCrypto` stores keys in IndexedDB (non-extractable RSA-OAEP 4096)
- `SSHCrypto` generates Ed25519 and RSA keys and exports them as OpenSSH PEM strings
- `Service__Keys` manages public key registry (server-side)
- Admin UI has `pki-keys.js` component for key management

**Implementation sketch:**

```
Key portability module: vault-key-portability.js
  Location: sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/js/vault-key-portability.js

  1. Export flow:
     a. User clicks "Export Key"
     b. Extract private key from IndexedDB (change key generation to extractable=true)
     c. Export as JWK (Web Crypto native) or PEM (via ASN.1 encoding)
     d. Prompt user for passphrase
     e. Encrypt the exported key with AES-256-GCM derived from passphrase (PBKDF2)
     f. Package as .sgkey file: { format: 'sgkey-v1', kdf: 'pbkdf2', ... }
     g. Download the .sgkey file

  2. Import flow:
     a. Drag-and-drop a key file onto import area
     b. Auto-detect format by extension and content:
        - .sgkey: our native format (decrypt with passphrase, import JWK)
        - id_ed25519: OpenSSH Ed25519 (parse PEM, extract seed, convert to X25519)
        - .asc/.gpg: GPG private key (parse with OpenPGP.js)
        - .pem: PKCS#8 PEM (parse ASN.1, import via Web Crypto)
        - .jwk: JSON Web Key (direct import via Web Crypto)
     c. Store imported key in IndexedDB via VaultCrypto
     d. Clear clipboard / warn to delete file

  3. Password manager integration:
     a. Render a hidden <input type="password"> field with autocomplete="current-password"
     b. On key generation: trigger the browser's "Save password?" dialog
        - Website: send.sgraph.ai
        - Username: key fingerprint
        - Password: base64-encoded private key (JWK)
     c. On import: password manager autofills the field
     d. Read the autofilled value and import
```

**Critical change to existing code:** `VaultCrypto.generateKeyPair()` currently generates keys with `extractable: false`. This must change to `extractable: true` for export to work. This is a breaking change for existing keys -- users who generated keys before this change cannot export them. Need a migration path: regenerate key pair, re-encrypt vault contents.

**Files modified:**
- `sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.7/js/vault-crypto.js` -- change `extractable: false` to `true` (or create v0.2.0 with this change)

**New files needed:**
- `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/js/vault-key-portability.js`
- `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/js/key-format-detector.js` (auto-detect logic)
- `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/components/key-import-export/key-import-export.js`

**Backend changes:** None. Key import/export is entirely client-side.

**Effort: M (Medium).** Native `.sgkey` export/import is 2-3 days. Multi-format auto-detect is 3-5 days. Password manager integration is 1-2 days (it leverages standard browser credential APIs).

### Brief #23: GitHub as PKI Registry

**What it is:** Fetch public keys from GitHub/GitLab (`https://github.com/{username}.keys` and `.gpg`), convert Ed25519 to X25519 for encryption, and enable "send to any GitHub user" in the UI.

**What exists today:**
- `Service__Keys` and `Routes__Keys` -- public key registry (our own)
- `SSHCrypto` -- Ed25519 key generation (but NOT Ed25519-to-X25519 conversion)
- `VaultCrypto` -- RSA-OAEP encryption (would need X25519/NaCl box alternative)

**Implementation sketch:**

```
Phase 1: GPG support (browser)
  New file: sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/js/openpgp-adapter.js
  - Import OpenPGP.js
  - Encrypt file with PGP public key
  - Decrypt file with PGP private key
  - Accept .gpg files in decrypt drop zone

Phase 2: SSH Ed25519 -> X25519 conversion (browser)
  New file: sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/js/ed25519-to-x25519.js
  - Use TweetNaCl.js for crypto_sign_ed25519_pk_to_curve25519()
  - Parse SSH public key wire format to extract raw Ed25519 bytes
  - Convert to X25519 public key
  - Encrypt using NaCl sealed_box (or box)

Phase 3: GitHub username lookup (browser + backend proxy)
  Frontend: New component <github-key-lookup>
  - Input field for GitHub username
  - Fetch https://github.com/{username}.keys (CORS issue -- need proxy)
  - Fetch https://github.com/{username}.gpg
  - Display identity signals (requires GitHub API)

  Backend: New route for proxying GitHub key fetches
  - Route: GET /keys/github/{username}
  - Fetches keys from GitHub, returns to client
  - Caches results briefly (5 min TTL)
  - Location: sgraph_ai_app_send/lambda__admin/fast_api/routes/Routes__Keys__GitHub.py
  - Service: sgraph_ai_app_send/lambda__admin/service/Service__Keys__GitHub.py

Phase 4: CLI tools (future -- not in this sprint)
Phase 5: Marketing campaign (Ambassador/Journalist -- not Dev)
```

**Backend changes needed:**
- New route: `Routes__Keys__GitHub` -- proxy for GitHub key fetches (avoids CORS)
- New service: `Service__Keys__GitHub` -- fetch, parse, cache GitHub keys
- Extend `Service__Keys` to support linking GitHub identities to key entries

**Frontend changes needed:**
- New component: `<github-key-lookup>` Web Component
- New crypto modules: `openpgp-adapter.js`, `ed25519-to-x25519.js`
- Extend encrypt drop zone to accept GitHub username as recipient

**CORS issue:** Browser JavaScript cannot directly fetch `https://github.com/{username}.keys` due to GitHub's CORS policy. Two options:
1. Backend proxy (recommended) -- add a route to the Admin Lambda
2. Use a CORS proxy service (not recommended for production)

**Third-party libraries:**
- OpenPGP.js (~200KB) -- PGP encryption/decryption
- TweetNaCl.js (~20KB) -- Ed25519-to-X25519 conversion + NaCl box
- Optionally: libsodium.js (~200KB) -- more complete, but heavier

**Effort: L (Large).** Phase 1 (GPG): 3-4 days. Phase 2 (SSH/X25519): 3-4 days. Phase 3 (GitHub lookup + proxy): 3-4 days. Total: ~10-12 days across phases.

### Compatibility Notes

**Type_Safe alignment:** Briefs #3, #14, #22, and #28 introduce new domain objects that must be implemented as `Type_Safe` classes, not Pydantic models or raw dicts. The existing codebase pattern (`Schema__Transfer`, `Schema__Token__Metadata`, etc.) provides the template.

**Memory-FS alignment:** Brief #14 (Portable Data Rooms) is the most significant Memory-FS change -- it requires a new `Storage_FS__ZIP` backend that implements the same `Storage_FS` interface as `Storage_FS__Memory` and `Storage_FS__S3`. This is architecturally clean because the abstraction layer already exists; the ZIP backend simply provides a new implementation.

**No-Pydantic rule:** All new schemas must use `Type_Safe`. This is straightforward for briefs #25/#26/#23 (minimal backend). Briefs #14 and #28 will need careful schema design with `Type_Safe`.

**No-boto3 rule:** Brief #22 (Agent Isolation) will need `osbot-aws` for container orchestration (ECS tasks, Secrets Manager). The `osbot-aws` package already wraps these services.

**IFD compliance:** Briefs #25, #26, #23, #1, and #3 all produce new UI components. Per IFD methodology, each gets its own versioned directory under `sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/`.

### Why This Order

1. **#26 (Key Portability) first** because every subsequent brief assumes users can move keys between devices. Without this, #25 and #23 work only on a single browser.

2. **#25 (.sgenc) second** because it delivers immediate user value with zero backend changes. Users can encrypt and email files today.

3. **#23 (GitHub PKI) in Sprint 2** because it requires OpenPGP.js and TweetNaCl, which also serve #25 Phase 2 (multi-format support). Building #23 first means #25 Phase 2 is nearly free.

4. **#3 (Dual Auth) in Sprint 3** because it depends on #1 (environment analysis) being available and builds on the transfer state machine that already exists.

5. **#22 and #14 in Sprint 4** because they are infrastructure-heavy and benefit from the crypto libraries and patterns established in Sprints 1-3.

6. **#7, #17, #28 deferred** because they have significant external dependencies (scanning service APIs, new GitHub repos, container orchestration) that require research phases first.
