## Implementation Priorities

### Sprint 1: Minimum Viable Data Room (Backend)

**Goal:** Create a room, invite a user, user joins and sees files.

```
1. Service__Data_Room.py
   - create_room(name, owner_user_id) → room_id
   - get_room(room_id) → room metadata
   - list_rooms(user_id) → rooms user has access to
   - archive_room(room_id) → soft archive
   - Internally: creates a vault, sets owner ACL

2. Routes__Data_Room.py
   - POST   /rooms/create         { name }
   - GET    /rooms/{room_id}
   - GET    /rooms/list
   - POST   /rooms/{room_id}/archive
   - GET    /rooms/{room_id}/members
   - POST   /rooms/{room_id}/invite

3. Service__Invites.py
   - create_invite(room_id, permission, created_by, expires_at, max_uses)
   - validate_invite(invite_code) → room_id + permission
   - accept_invite(invite_code, user_id) → ACL entry created
   - expire_invite(invite_code)

4. Service__Room__Session.py
   - create_session(room_id, user_id, permission) → session_token
   - validate_session(session_token) → room_id + user_id + permission
   - revoke_session(session_token)
```

**Tests for Sprint 1:**
- Room lifecycle: create → get → list → archive
- Invite lifecycle: create → validate → accept → user gets access
- Invite constraints: expire, max uses, duplicate acceptance
- Session: create → validate → revoke
- Permission enforcement: viewer can't edit, editor can't manage members

### Sprint 2: Minimum Viable Data Room (Frontend)

**Goal:** UI for room management and the receiver landing page.

```
1. data-room-manager.js
   - Room list (cards or table)
   - Create room dialog
   - Room detail view (wraps vault-manager for file browsing)
   - Member list with permission badges
   - Invite generation with copy-to-clipboard

2. receiver-landing.js
   - Clean landing page: "Enter your access code"
   - Short code input (6 chars, auto-format)
   - Error handling (expired, invalid, max attempts)
   - Success: redirect to room view

3. data-room-invite.js
   - Accept flow: show room name, register key if needed, enter room
   - PKI key auto-generation for new recipients (transparent)

4. Room API client
   - Wraps all /rooms/* and /invites/* endpoints
   - Session token management
```

### Sprint 3: Audit & Polish

```
1. Service__Audit.py + Routes__Audit.py
   - Log every room operation
   - Query by room, user, time range, action type
   - Immutable append-only

2. data-room-activity.js
   - Activity feed (recent events)
   - Audit log viewer (filterable table)

3. Room key hierarchy (if approved by Architect)
   - HKDF room_key → folder_key → file_key
   - Migration path for existing per-file wrapped keys

4. Folder-level ACL
   - Extend ACL with optional folder_guid
   - Inheritance rules
```

## Reuse Opportunities

### Service__Tokens as Invite Code Template

`Service__Tokens` already handles:
- Token creation with metadata
- Token lookup by name
- Token use (single-use or quota)
- Token revocation

`Service__Invites` follows the same pattern. Consider whether `Service__Invites` should extend or compose `Service__Tokens`. Recommendation: **compose** — invites have room-scoped semantics that tokens don't.

### Vault Manager as Data Room Manager Template

`vault-manager.js` (1,800 lines) handles:
- File tree view
- Upload/download with encryption
- Share via Send
- Component orchestration (preview, editor, settings)

`data-room-manager.js` starts as a copy that adds:
- Room selector (which room am I in?)
- Member panel (who's in this room?)
- Activity feed (what happened recently?)
- Invite management (generate/copy/revoke links)

**IFD approach:** Build room manager as a new `v0.1.7` component. Don't modify `v0.1.6` vault components — they remain independently functional.

## Known Technical Debt to Address

| Item | Current State | Impact on Data Room | Fix |
|------|--------------|---------------------|-----|
| No per-part retry in multipart upload | Whole upload fails on one chunk | Large file upload UX in rooms | Add retry with exponential backoff per part |
| Vault ACL not enforced on all routes | ACL service exists but not all routes check it | Must enforce for rooms | Middleware or decorator pattern |
| No rate limiting on API routes | Open endpoints | Invite link abuse | Add rate limit middleware |
| Frontend test coverage | 0% for upload concurrency | Can't validate room upload behaviour | Playwright E2E for room flows |

## Recommendation

**Start with Sprint 1 (backend).** The backend services are the foundation — they can be built, tested, and validated independently of the UI. Once the room/invite/session APIs are solid, the frontend sprints can move fast by following established vault UI patterns.

**The minimum viable demo is Sprint 1 + Sprint 2:** Create a room → generate invite → recipient enters code → sees files → downloads. That's the investor demo described in the data room product brief (v0.5.10).
