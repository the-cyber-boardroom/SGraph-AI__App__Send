## Overview

Phase 4 implements the `sg-vault` CLI tool that syncs a local directory with a remote encrypted vault on `send.sgraph.ai`. The CLI uses the Python `Vault__Crypto` class (from this repo) for key derivation and the vault pointer API endpoints for read/write operations.

The CLI lives in a **separate repository** but depends on the key derivation and API contracts defined here. This document provides all the information needed to implement it.

---

## API Endpoints (from Phase 1)

All endpoints are on the User Lambda at `send.sgraph.ai`:

| Method | Path | Auth | Purpose |
|---|---|---|---|
| `PUT` | `/api/vault/write/{vault_id}/{file_id}` | Access token + Write key | Write or overwrite a vault file |
| `GET` | `/api/vault/read/{vault_id}/{file_id}` | None | Download a vault file (encrypted) |
| `DELETE` | `/api/vault/delete/{vault_id}/{file_id}` | Access token + Write key | Delete a vault file |

### Headers

| Header | Required for | Value |
|---|---|---|
| `x-sgraph-access-token` | PUT, DELETE | Access token string |
| `x-sgraph-vault-write-key` | PUT, DELETE | Write key hex string (64 chars) |
| `Content-Type` | PUT | `application/octet-stream` |

### Security: Vault Manifest

The server enforces vault-level ownership via a manifest file. The first write to a vault establishes the `write_key_hash`. All subsequent writes and deletes to that vault must use the same write key. An attacker who knows the vault_id but not the write key cannot create or delete files.

## Key Derivation (from Phase 2)

### Python Implementation

The `Vault__Crypto` class in `sgraph_ai_app_send/utils/Vault__Crypto.py` provides:

```python
from sgraph_ai_app_send.utils.Vault__Crypto import Vault__Crypto, parse_vault_key

# Parse vault key
passphrase, vault_id = parse_vault_key('my-passphrase:a1b2c3d4')

# Derive all keys
crypto = Vault__Crypto()
keys   = crypto.derive_keys(passphrase, vault_id)

keys['read_key_bytes']    # bytes (32) — AES-256-GCM key for encrypt/decrypt
keys['write_key']         # str (64 hex chars) — for x-sgraph-vault-write-key header
keys['tree_file_id']      # str (12 hex chars) — deterministic file ID for vault tree
keys['settings_file_id']  # str (12 hex chars) — deterministic file ID for vault settings

# Or in one call:
keys = crypto.derive_keys_from_vault_key('my-passphrase:a1b2c3d4')
keys['passphrase']        # str
keys['vault_id']          # str
```

### Derivation Parameters

| Parameter | Value |
|---|---|
| Algorithm | PBKDF2-HMAC-SHA256 |
| Iterations | 600,000 |
| Read key salt | `sg-vault-v1:{vault_id}` |
| Write key salt | `sg-vault-v1:write:{vault_id}` |
| Key length | 256 bits (32 bytes) |
| File ID derivation | HMAC-SHA256(read_key, domain_string)[:12] |
| Tree domain string | `sg-vault-v1:file-id:tree:{vault_id}` |
| Settings domain string | `sg-vault-v1:file-id:settings:{vault_id}` |

## Encryption Format

All payloads are encrypted with AES-256-GCM using the `read_key`:

| Property | Value |
|---|---|
| Algorithm | AES-256-GCM |
| Key | `read_key_bytes` (32 bytes from PBKDF2) |
| IV | 12 random bytes (per encryption) |
| Wire format | `[12 bytes IV][ciphertext + 16-byte auth tag]` |

### Python Encryption (using `cryptography` package)

```python
import os
from cryptography.hazmat.primitives.ciphers.aead import AESGCM

def encrypt(plaintext: bytes, key: bytes) -> bytes:
    iv = os.urandom(12)
    aesgcm = AESGCM(key)
    ciphertext = aesgcm.encrypt(iv, plaintext, None)
    return iv + ciphertext                              # [12B IV][ciphertext + tag]

def decrypt(data: bytes, key: bytes) -> bytes:
    iv, ciphertext = data[:12], data[12:]
    aesgcm = AESGCM(key)
    return aesgcm.decrypt(iv, ciphertext, None)
```

This format is byte-compatible with the JavaScript `SGSendCrypto.encrypt()` / `decrypt()`.

## CLI Commands

### `sg-vault clone <vault-key> [directory]`

1. Parse vault key → passphrase, vault_id
2. Derive keys: read_key, write_key, tree_file_id, settings_file_id
3. `GET /api/vault/read/{vault_id}/{settings_file_id}` → decrypt → settings JSON
4. `GET /api/vault/read/{vault_id}/{tree_file_id}` → decrypt → tree JSON
5. Walk the tree, download each file: `GET /api/vault/read/{vault_id}/{file_id}` → decrypt
6. Write files to local directory
7. Create `.sg_vault/HEAD` with vault key
8. Create `.sg_vault/tree.json` with decrypted tree (for diffing on push)

### `sg-vault push`

1. Read `.sg_vault/HEAD` → vault key → derive keys
2. Read `.sg_vault/tree.json` → old tree
3. Scan local directory → build new tree
4. Diff old tree vs new tree → identify added, modified, deleted files
5. For each added/modified file: encrypt → `PUT /api/vault/write/{vault_id}/{file_id}`
6. For each deleted file: `DELETE /api/vault/delete/{vault_id}/{file_id}`
7. Encrypt updated tree → `PUT /api/vault/write/{vault_id}/{tree_file_id}`
8. Encrypt updated settings → `PUT /api/vault/write/{vault_id}/{settings_file_id}`
9. Update `.sg_vault/tree.json`

### `sg-vault status`

1. Read `.sg_vault/HEAD` and `.sg_vault/tree.json`
2. Scan local directory → compare against tree
3. Report: added, modified, deleted files

## Dependencies

The CLI needs:
- `cryptography` (Python package) — for PBKDF2 and AES-256-GCM
- `httpx` or `requests` — for HTTP calls to the vault pointer API
- `typer` — for CLI framework (as specified in the brief)
- `Vault__Crypto` class — can be copied from this repo or imported as a dependency

---

## Important Notes

1. **Read endpoint requires no auth** — the zero-knowledge model means anyone with the file_id can download ciphertext. Security is in the encryption, not URL secrecy.
2. **Write/delete require both access token AND write key** — two layers of auth.
3. **Tree and settings are overwritten in-place** — no orphan transfers. The vault key is stable forever.
4. **User file IDs are random** — only tree_file_id and settings_file_id are deterministic.
5. **`.sg_vault/HEAD` must be in `.gitignore`** — it contains the vault passphrase.
6. **Vault manifest on server** — the first write establishes ownership. The CLI cannot accidentally create files in someone else's vault.
