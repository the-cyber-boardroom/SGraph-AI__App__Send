### Item-by-Item Assessment

#### 1. Auth Token Only Required for Upload

**Feasibility:** High. The change is in `vault-entry.js` and `sg-vault.js`. Currently the access token is stored in sessionStorage on entry and sent with every API request via `SGSend._authHeaders()`. The change splits the flow: vault-key-only opens the vault (settings + tree downloads use a different auth path or public download), access token is prompted only on first write operation.

**Technical concern:** The current transfer download endpoint (`GET /transfers/download/{id}`) requires token validation via `validate-token`. If downloads also require auth, then the "read-only with vault key only" flow requires either: (a) a separate public download mechanism, or (b) the transfer downloads are public by ID (security through obscurity -- the transfer ID is unguessable). Need to check whether downloads currently require the `x-sgraph-access-token` header.

**Effort:** M (1-2 days). The logic change is straightforward, but testing the auth boundary between read-only and write flows needs care.

**Risk:** Medium. If the download endpoint requires auth, this item needs backend changes or an architectural decision about public-by-ID downloads.

#### 2. Settings Panel

**Feasibility:** High. A new `vault-settings` Web Component. The vault-tree.json and vault-settings.json are already decrypted in `sg-vault.js` -- they just need to be surfaced in a panel. The raw JSON viewer is a `<pre>` element with `JSON.stringify(data, null, 2)`. Vault statistics (file count, total size, folder count) can be computed from the in-memory tree.

**Effort:** M (1-2 days). New component, but no new data fetching required -- all data is already in memory.

**Risk:** Low.

#### 3. Folder Upload Bug

**Feasibility:** High. The v0.10.28 debrief notes this was already partially fixed: "Fixed subfolder upload path propagation in vault-upload.js (`path || this._targetPath` -> explicit `undefined` check)." Need to verify the fix is complete.

**Effort:** S (few hours). Likely already fixed in v0.1.1; needs verification and regression test.

**Risk:** Low.

#### 4. Drag-and-Drop

**Effort:** S-M (half day to 1 day). Most of this is already built. Folder-to-zip on drop is the only new work.

**Risk:** Low. JSZip is already vendored.

#### 5. Folder Management

#### 6. File Metadata Banner

**Feasibility:** High. `vault-file-properties` already exists in v0.1.1 with a download button. The brief asks for expanded metadata display (size, type, upload date, folder path) and action buttons (download, delete, rename). Key-value metadata support for future extensibility is a data model question -- adding an `attributes: {}` map to each file entry in the vault tree.

**Effort:** S-M (half day to 1 day). Expanding an existing component.

**Risk:** Low.

#### 7. Preview (Confirm and Polish)

**Feasibility:** High. The v0.10.28 debrief confirms 8 format renderers are working in vault-file-preview. The brief asks to add: Markdown (already working), Text/JSON syntax highlighting (text already working, syntax highlighting is cosmetic), HTML in sandboxed iframe (new but straightforward -- create a blob URL, load in `<iframe sandbox="allow-scripts">`).

**Effort:** S (few hours). Text/JSON highlighting and HTML iframe are small additions.

**Risk:** Low. HTML sandboxing needs `sandbox` attribute to prevent script escape.

#### 8. Inline Editing (Text, Markdown, JSON)

**Feasibility:** Medium-High. Requires: (a) a `<textarea>` or code editor in the preview panel, (b) Save button that encrypts the edited content + uploads as a new transfer + updates vault-tree, (c) handles the vault-key mutation (since the settings transfer ID changes on every mutation). No syntax highlighting library is specified -- a plain `<textarea>` with monospace font is the MVP.

**Effort:** M (1-2 days). The encrypt-upload-update-tree pipeline is already established for file uploads; inline editing reuses it with in-memory content instead of a File object.

**Risk:** Medium. The vault-key mutation on every save (noted in the v0.10.19 handover as a known issue: "Vault key changes after every mutation") means the user's vault key in the URL hash will change on every edit. This is confusing UX. Not a blocker for the feature, but the user needs to understand that their key changes.

#### 9. Local Storage Viewer (Debug Tool)

**Feasibility:** High. Pure client-side. `Object.keys(localStorage)`, display in a table, clear buttons. No server interaction.

**Effort:** S (few hours). New component, trivial implementation.

**Risk:** None.

#### 10. Chrome Sync Exploration

**Feasibility:** Low-Medium. Web apps cannot access `chrome.storage.sync` -- that API is Chrome extension-only. The brief correctly identifies this: "Can a web app access this without being an extension?" The answer is no. The MVP path is a minimal Chrome extension that: (a) stores vault keys in `chrome.storage.sync`, (b) exposes them to sgraph.ai domains via `content_scripts` or `externally_connectable`. This is a meaningful departure from "zero framework dependencies, vanilla JS" -- it introduces a browser extension.

**Effort:** L (3-5 days). Chrome extension development, manifest v3, content security policy, Chrome Web Store submission, and the bidirectional communication between the extension and the web page.

**Risk:** High. Chrome extension development is a separate toolchain, separate review process, separate deployment pipeline. The Chrome Web Store review process can take days to weeks. This is exploration-appropriate but should not be on the critical path.

**Alternative approach:** Store vault metadata (name, last-opened date, vault key hint) in `localStorage` or `IndexedDB` per domain. This gives the "saved vaults" entry screen experience without requiring a Chrome extension. It does not sync across devices, but it solves the "which vaults have I opened before?" problem immediately.

## Brief 3: Encrypted Gallery Sharing (v0.10.30)

**Source:** `team/humans/dinis_cruz/briefs/03/03/v0.10.30__dev-brief__encrypted-gallery-sharing.md`

### What Exists Today

The brief correctly identifies the existing infrastructure:

| Component | Status | Code Location |
|-----------|--------|---------------|
| Folder upload -> auto-zip | Shipped | Send UI v0.2.0 |
| In-browser zip processing | Shipped | JSZip vendored in both send and vault UIs |
| Image display | Shipped | Send download page + vault preview |
| Video playback | Shipped | HTML5 `<video>` in vault-file-preview |
| Audio playback | Shipped | HTML5 `<audio>` in vault-file-preview |
| Encryption (AES-256-GCM) | Shipped | sg-send-crypto.js |
| Friendly key concept | PROPOSED -- does not exist yet | No code generates word-based keys |

### Technical Assessment

#### Phase 1: Beautiful Gallery MVP

The gallery is a **new view component** on the download page. When a user downloads and decrypts a zip file, the system detects it contains images/videos and switches from the generic zip browser to a gallery view.

**What needs building:**

| Component | Effort | Notes |
|-----------|--------|-------|
| Gallery detection | S (few hours) | After zip extraction, check if >50% of files are images/videos. If yes, offer gallery view. |
| Thumbnail grid layout | M (1-2 days) | CSS Grid with `object-fit: cover`. Lazy loading via `IntersectionObserver`. Thumbnails generated client-side by rendering images to a `<canvas>` at reduced resolution. |
| Lightbox viewer | M (2-3 days) | Full-screen overlay, left/right navigation, swipe gestures (Touch API), keyboard navigation (arrow keys), pinch-to-zoom (multi-touch). This is the most complex new component. |
| Mobile optimisation | M (1-2 days) | Touch gestures, orientation handling (`screen.orientation` API), safe-area insets (`env(safe-area-inset-*)` CSS), viewport meta. |
| Gallery metadata (gallery.json) | S (few hours) | Parse optional JSON from zip root. |
| Progress indicator | S (few hours) | Progress bar during decrypt + extract. |
| Download options | S (few hours) | Individual image download (create blob URL, trigger `<a download>`), download all (serve original encrypted zip). |
| Friendly keys | M (1-2 days) | Word list (BIP39 or custom), mapping function `hex_key -> word-word-number`. Must be deterministic and reversible. Needs both JS (browser) and Python (CLI) implementations. |

**Total Phase 1: 6-10 days**

#### Key Technical Decisions

1. **Thumbnail generation strategy.** Drawing 100 images to `<canvas>` on the main thread will freeze the UI. Options: (a) Web Workers with `OffscreenCanvas` (Chrome 69+, Firefox 105+, Safari 16.4+ -- good coverage), (b) sequential generation with `requestIdleCallback` to avoid blocking, (c) generate only visible thumbnails via `IntersectionObserver`. Recommendation: (c) is the simplest and sufficient for MVP.

2. **Memory management.** 100 images at 5MB each = 500MB in memory if all held simultaneously. The zip is already in memory (downloaded + decrypted). Thumbnails should be generated from the zip entry stream, not from fully-extracted images. Revoke blob URLs when thumbnails scroll out of view.

3. **Swipe gesture implementation.** Options: (a) Hammer.js (12KB, well-tested), (b) custom Touch API handlers. Recommendation: custom implementation for zero-dependency consistency with the project's "zero framework dependencies" rule. Swipe detection is ~50 lines of Touch API code.

4. **Pinch-to-zoom.** More complex than swipe. Options: (a) CSS `touch-action: manipulation` + native browser zoom, (b) custom multi-touch handler with transform matrix. Recommendation: native CSS approach for MVP. Custom pinch-to-zoom adds 2-3 days and is error-prone on iOS Safari.

#### Phases 2-4

PROPOSED -- does not exist yet. Phase 2 (photo editing, annotation, redaction) and Phases 3-4 (LLM, collaboration) are substantial features. Not assessed in detail here. Phase 2 alone is L-XL (2-4 weeks). Phase 3 requires client-side LLM integration infrastructure that does not exist.

### Dependencies

| Dependency | Status | Blocker? |
|------------|--------|----------|
| JSZip in send UI | Shipped (vendored) | No |
| Image/video rendering | Shipped | No |
| Friendly key generation | Does not exist | Yes for the branded key experience, No for gallery functionality |
| Gallery.json specification | Needs agreement | No for MVP (auto-detect) |

### Risks

1. **Performance on mobile.** Decrypting a 500MB zip + generating thumbnails on a mobile device is memory-intensive. Mobile Safari on older iPhones (2GB RAM) may crash. Mitigation: test on low-spec devices early, implement progressive loading with strict memory budgets.

2. **iOS Safari quirks.** Pinch-to-zoom conflicts with Safari's page zoom. `<meta name="viewport" content="user-scalable=no">` is required but may interfere with accessibility. The lightbox needs to handle this carefully.

3. **Scope creep.** The brief describes a full product with 4 phases, LLM integration, collaborative editing, map views, and timeline views. Phase 1 alone is 6-10 days. The brief's vision is compelling but must be built incrementally. The "it's a product, not a feature" framing is correct for strategy but dangerous for sprint planning.

### Recommendation

Gallery Phase 1 is a high-impact deliverable that reuses existing infrastructure. It directly supports the 50-customer goal: every gallery shared is a product demo to every recipient. However, it is **not** on the critical path for this week. Vault fixes (Brief 2) should come first because they improve the experience for existing customers. Gallery is a P1 sprint item.

**Proposed sequence:** Vault fixes (week 1) -> Gallery Phase 1 (week 2-3).

## Brief 4: Vault CLI and Local Sync (v0.10.30)

**Source:** `team/humans/dinis_cruz/briefs/03/03/v0.10.30__dev-brief__vault-cli-local-sync.md`

### What Exists Today

- **No CLI tool exists.** No `sg-vault` CLI, no Python package for vault operations, no local sync infrastructure.
- **Python crypto library available.** The brief uses `cryptography` package (AESGCM, PBKDF2HMAC). This is a standard, well-audited library.
- **Browser-side vault code exists.** `sg-vault.js` (JavaScript) implements vault CRUD. The CLI must produce interoperable ciphertext.
- **Vault API endpoints exist.** The vault uses the transfer API (`/transfers/create`, `/upload`, `/complete`, `/download`) plus vault-specific endpoints on Admin Lambda (`/vault/create`, `/vault/lookup`, etc.).

### Technical Assessment

#### The Interoperability Challenge

This is the highest-risk item in the entire brief. The Python CLI must produce ciphertext that the JavaScript browser can decrypt, and vice versa. This requires exact alignment on:

| Parameter | JavaScript (Browser) | Python (CLI) | Match Required |
|-----------|---------------------|---------------|----------------|
| Algorithm | AES-256-GCM (Web Crypto API) | AES-256-GCM (`cryptography.hazmat`) | Exact |
| Key derivation | PBKDF2 (SHA-256, 600k iterations) | PBKDF2HMAC (SHA-256, 600k iterations) | Exact |
| Salt format | `sg-vault-v1:{vaultId}` (UTF-8 encoded) | Same | Exact |
| Nonce | 12 bytes random, prepended to ciphertext | Same | Exact |
| Ciphertext format | `nonce (12 bytes) + ciphertext + tag (16 bytes)` | Same | Exact |
| Associated data (AAD) | Need to check: `null` or specific value | Same | Exact |

**The AAD question is critical.** In `sg-send-crypto.js`, the `encrypt()` call uses `crypto.subtle.encrypt({ name: 'AES-GCM', iv: nonce }, key, data)` -- no `additionalData` parameter, which means AAD is empty. The Python side must also use `aesgcm.encrypt(nonce, data, None)` -- where `None` means no AAD. The brief's code snippet shows `aesgcm.encrypt(nonce, data, None)`, which is correct.

**The tag handling matters.** Web Crypto API's AES-GCM appends the 16-byte authentication tag to the ciphertext by default (`tagLength: 128`). The `cryptography` library also appends the tag. Both should be compatible, but this must be verified with cross-language round-trip tests before anything else is built.

**Recommended first task:** Write a single test that encrypts a known plaintext with a known key/nonce in JavaScript, produces hex output, then decrypts that exact hex in Python (and vice versa). If this test passes, everything else is buildable. If it fails, the CLI is blocked until the format mismatch is resolved.

#### Effort Estimate

| Task | Effort | Notes |
|------|--------|-------|
| **Cross-language crypto interop test** | M (1-2 days) | The gating task. Must pass before anything else. |
| **`sg-vault connect`** | M (1-2 days) | Parse vault key, derive encryption key, download + decrypt settings, create `.sg_vault/` directory with config + manifest. |
| **`sg-vault pull`** | M (1-2 days) | Download + decrypt vault tree, iterate files, download + decrypt each, write to local filesystem. |
| **`sg-vault push`** | M (1-2 days) | Read local file, encrypt, upload via transfer API, update vault tree, encrypt + upload tree. |
| **`sg-vault sync`** | L (2-3 days) | Bidirectional: compute local hashes, fetch remote tree, diff, upload new local files, download new remote files, update manifest. Conflict detection. |
| **`sg-vault ls` / `status` / `info`** | S (half day each, 1.5 days total) | Read-only commands, mostly display logic. |
| **Typer CLI framework** | S (few hours) | Typer setup, command registration, help text. |
| **PyPI packaging** | S (few hours) | `pyproject.toml`, entry point, `pip install sg-vault`. |

**Total: 10-14 days (2-3 weeks)**

#### Architecture Concerns

1. **Auth token management in CLI.** The vault uses `x-sgraph-access-token` for API auth. The CLI needs to store this somewhere. Options: (a) `.sg_vault/config` file, (b) environment variable, (c) system keyring (`keyring` Python package). Recommendation: `.sg_vault/config` for MVP, with env var override.

2. **Vault tree mutation concurrency.** If User A syncs from CLI while User B edits in the browser, both modify the vault tree. The tree is a single encrypted JSON file. Last-write-wins will cause data loss. The brief's conflict handling ("keep both, rename with `.conflict`") only applies to individual files, not to the tree itself. This is a harder problem.

   **Mitigation for MVP:** The sync command should: (a) fetch the remote tree, (b) merge changes (new files from both sides), (c) detect true conflicts (same file modified on both sides), (d) upload the merged tree. This is essentially a three-way merge of JSON structures. Doable but non-trivial.

3. **Vault key format.** The vault key is `passphrase:vaultId:settingsTransferId`. The `settingsTransferId` changes on every mutation (noted in v0.10.19 handover). The CLI needs to handle this: after a sync that modifies the tree, the vault key changes. The CLI must update its stored key.

4. **Large file handling.** The current transfer API has a 6MB direct upload limit. Files >6MB require presigned multipart upload. The CLI must implement the multipart path for real-world use. The browser UI already handles this (`SGSend.encryptAndUpload()` with multipart fallback).

### Dependencies

| Dependency | Status | Blocker? |
|------------|--------|----------|
| `cryptography` Python package | Standard, pip-installable | No |
| `typer` Python package | Standard, pip-installable | No |
| Cross-language crypto interop | MUST BE VERIFIED | **Yes -- gating** |
| Vault transfer API auth | Works but needs token | No |
| Presigned upload endpoints | Exist on User Lambda | No |

### Risks

1. **Crypto interoperability.** If the JavaScript and Python implementations produce different ciphertext for the same inputs, the entire CLI is blocked. This is the highest-risk item.

2. **Tree mutation conflicts.** Two concurrent writers (browser + CLI, or CLI + CLI) can corrupt the vault tree. No locking mechanism exists.

3. **Scope.** 10-14 days for a full CLI is a significant investment. The MVP should be `connect` + `pull` + `push` only (no bidirectional sync). This reduces effort to 5-7 days and delivers the core value: "I can access my vault files locally."

### Recommendation

Start with the cross-language crypto interop test. If that passes, build `connect` + `pull` + `push` (one-way commands). Defer `sync` (bidirectional) to a later iteration. The one-way commands are immediately useful and avoid the tree concurrency problem entirely.

**Priority: P1 sprint, not P1 immediate.** The vault web UI improvements (Brief 2) serve all users. The CLI serves power users. Fix the web UI first.

## Brief 5: File Safety Orchestration Architecture (v0.10.30)

**Source:** `team/humans/dinis_cruz/briefs/03/03/v0.10.30__arch-brief__file-safety-orchestration-architecture.md`

### Assessment

This is an **architecture brief**, not a dev brief. It describes a long-horizon platform capability: pluggable scanning services with cryptographic attestation, cross-domain file security, and a receiver-defined scanning model. The document is comprehensive (400+ lines), covering strategic case, technical depth, integration tiers, cross-domain architecture, and government/defence positioning.

### What Exists Today That Supports This

| Foundation | Status | Relevance |
|------------|--------|-----------|
| PKI key infrastructure | Shipped (Admin Lambda) | Keys, key registry, key log -- foundation for attestation |
| Vault ACL | Shipped | Owner/editor/viewer hierarchy -- foundation for policy enforcement |
| Data rooms | Shipped | Room creation, members, invites, audit trail -- scanning policy target |
| Audit service | Shipped | Append-only hash-chained events -- foundation for attestation chain |
| Transfer encryption (AES-256-GCM) | Shipped | Client-side encryption pipeline |
| Immutable audit trail | Shipped | Hash-chained, append-only |

### What Does Not Exist

- **No scanning service integration.** No CDR, AV, DLP, or any third-party scanning API call.
- **No pod architecture.** No trusted intermediary component that decrypts, processes, and re-encrypts.
- **No attestation chain.** No signed scanning results bound to file hashes.
- **No receiver-defined scanning policy.** No mechanism for a receiver to declare required scanning services.
- **No scanning cost model.** No per-scan billing or sender-pays mechanism.

PROPOSED -- none of this exists yet.

### Technical Feasibility

The architecture is technically sound. The cryptographic primitives (PKI signing, hash binding, encrypted attestation) are well-understood and the project already has the PKI infrastructure. The four integration tiers (REST API, data room policy, embedded engine, WASM browser-side) are correctly ordered by complexity and value.

However, this is a **multi-quarter initiative**, not a sprint deliverable:

| Tier | Effort | Timeline |
|------|--------|----------|
| Tier 1 (REST API per service) | 3-5 weeks per service | Q2 2026 earliest |
| Tier 2 (Data room policy) | 6-10 weeks | Q2-Q3 2026 |
| Tier 3 (Embedded engine) | 12-16 weeks | Q3-Q4 2026 |
| Tier 4 (WASM browser-side) | Research phase | 2027 |

### Risks

1. **Partner dependency.** Tier 1 requires a scanning partner's API access, documentation, and test environment. No partners are named as committed.

2. **Pod architecture is a new component.** The trusted intermediary pod is not a modification of existing infrastructure -- it is a new isolated service with its own deployment, network isolation, PKI key, and audit requirements. This is DevOps and AppSec territory as much as Dev.

3. **Cross-domain positioning requires government/defence certification.** The document's most compelling market segment (government, defence) requires security certification (e.g., NCSC, Common Criteria). This is a years-long process.

### Recommendation

**This brief requires no Dev action this sprint.** It is a strategic architecture document for the Architect and Town Planner roles. The Dev role should:

1. Acknowledge the brief and note the PKI foundation is in place.
2. Flag that the cross-language crypto interop test (from Brief 4) is also relevant here -- the pod will need Python crypto that matches the browser's JavaScript crypto.
3. Note that if the project moves toward Tier 1, the first Dev task will be: build a scanning service adapter interface (a Python abstract class that the pod calls for each scanning service, returning a standardised result structure).

**Priority: P2+ (next quarter).** No code to write now.

## Technical Concerns and Missing Pieces

### 1. Vault Key Mutation Problem

Flagged in the v0.10.19 handover and relevant to Briefs 2, 3, and 4: the vault key (`passphrase:vaultId:settingsTransferId`) changes on every mutation because the settings transfer ID changes. This means:

- A shared vault key becomes stale after any modification
- The CLI must track the current key and update it after every push/sync
- Users who bookmarked or saved the key will find it works but shows a snapshot, not the latest state

This is not a blocker but is a UX friction point that will generate support requests. The Architect should consider a stable vault addressing scheme (e.g., the vault ID is permanent, and the latest settings transfer ID is discoverable via an API call using the vault ID + passphrase).

### 2. Download Auth Requirement

Brief 2 item 1 (auth for upload only) depends on whether the transfer download endpoint requires auth. The reality document shows `GET /transfers/download/{id}` exists and is tested, but does not specify whether it requires the `x-sgraph-access-token` header. If it does, the "view-only with vault key" flow requires either: (a) making downloads public-by-ID, or (b) a separate read-only token type.

**Action needed:** Check the transfer download route code to verify auth requirements.

### 3. Reality Document is Stale

The reality document is at v0.6.36 (audit date: 2026-02-26). We are now at v0.10.32 with 93 new commits. The following items exist in code but are not reflected in the reality document:

- Vault v0.1.1 UI (42 files, 5,575 lines)
- Folder-to-zip upload (send UI)
- Zip viewer/browser (send UI)
- Video/audio preview (send + vault UIs)
- i18n runtime with 17 locales (v0.2.0)
- S3 deployment isolation (branch-based paths)
- Homepage CTA redesign
- Send UI v0.2.0 consolidation

**The reality document needs updating to v0.10.32.** This is a Librarian task but affects Dev: every dev review that references existing state must manually verify code instead of trusting the reality document.

### 4. Friendly Keys Are Not Trivial

The gallery brief mentions "blue-mountain-77" style friendly keys. This requires: a word list (2048+ words for security), a deterministic mapping from hex key to words, and implementations in both JavaScript and Python (for the CLI). BIP39 is the standard word list (2048 English words) but it generates 12-24 words for cryptographic keys. A 3-word + number format like "blue-mountain-77" is memorable but has much lower entropy (~34 bits for 2048 words + 2 digits). This needs an Architect decision on acceptable entropy for the use case.

### 5. No E2E Tests for Vault

The vault has 38 static file tests (file existence and content patterns) but no Playwright E2E tests. As vault complexity increases (v0.1.2 with all Brief 2 improvements), the risk of regression increases. Vault E2E tests should be planned for this sprint.

## Notes for Other Roles

**Architect:** The vault key mutation problem (settingsTransferId changes on every write) is becoming a cross-cutting concern.
