### 3.2 OpenRouter Implementation (Phase 1 Primary)

```javascript
class OpenRouterProvider extends LLMProvider {
  constructor(apiKey, model = 'anthropic/claude-sonnet-4') {
    super('openrouter', { apiKey, baseUrl: 'https://openrouter.ai/api/v1', model });
  }

  async *streamCompletion(messages, options) {
    const response = await fetch(`${this.config.baseUrl}/chat/completions`, {
      method: 'POST',
      headers: {
        'Authorization': `Bearer ${this.config.apiKey}`,
        'Content-Type': 'application/json',
        'HTTP-Referer': 'https://workspace.sgraph.ai',
        'X-Title': 'SGraph Workspace'
      },
      body: JSON.stringify({
        model: this.config.model,
        messages,
        stream: true
      })
    });

    // Read SSE stream (OpenAI-compatible format)
    const reader = response.body.getReader();
    const decoder = new TextDecoder();
    // ... parse SSE chunks, yield deltas
  }
}
```

**Why OpenRouter first:**
- Explicitly supports browser CORS (designed for web apps)
- One API key → access to Claude, GPT-4, Gemini, Llama, Mistral, and 200+ models
- OpenAI-compatible API (well-documented, battle-tested)
- Avoids the CORS uncertainty of calling Anthropic/OpenAI directly from a browser

### 3.3 Ollama Implementation (Phase 1 Local)

```javascript
class OllamaProvider extends LLMProvider {
  constructor(model = 'llama3.2') {
    super('ollama', { baseUrl: 'http://localhost:11434', model });
  }

  async *streamCompletion(messages, options) {
    const response = await fetch(`${this.config.baseUrl}/api/chat`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ model: this.config.model, messages, stream: true })
    });
    // ... parse NDJSON stream
  }
}
```

**Why Ollama:**
- Runs locally — zero network, zero cost, maximum privacy
- No CORS issues (localhost)
- Good for testing and development
- Appeals to the privacy-conscious audience (SGraph's core market)

## 4. Document Viewer — Technical Detail

### 4.1 Component Structure

```javascript
// document-viewer.js
class DocumentViewer extends HTMLElement {
  constructor() {
    super();
    this.attachShadow({ mode: 'open' });
    this._content = null;       // raw content (decrypted)
    this._filename = null;      // for type detection
    this._renderType = null;    // 'markdown' | 'text' | 'json' | 'image' | 'html'
    this._mode = 'rendered';    // 'rendered' | 'source' | 'diff'
  }

  // Public API
  load(content, filename) { /* detect type, render */ }
  clear() { /* empty state */ }
  getContent() { /* return current content */ }
  showDiff(original, transformed) { /* Phase 2 */ }
}
```

### 4.2 Render Pipeline

```
1. User clicks file in vault-panel
   ↓
2. vault-panel emits 'file-selected' event via EventBus
   ↓
3. workspace-shell catches event, calls:
     vaultAPI.getFile(vaultKey, fileGuid)
   ↓
4. Encrypted blob received
   ↓
5. vaultCrypto.decrypt(privateKey, blob) → plaintext
   ↓
6. FileTypeDetect.detect(filename, mimeType) → renderType
   ↓
7. source-viewer.load(plaintext, filename)
     → renders in left panel based on renderType
```

### 4.3 Transformation Flow

```
1. User types prompt in llm-chat panel
   ↓
2. llm-chat reads source content from source-viewer.getContent()
   ↓
3. Constructs LLM messages:
     [{ role: 'system', content: 'You are a document transformation assistant...' },
      { role: 'user', content: `${prompt}\n\n---\n\n${documentContent}` }]
   ↓
4. Calls provider.streamCompletion(messages)
   ↓
5. Streams chunks into transform-viewer (right panel)
   ↓
6. On completion, enables "Save to Vault" button
   ↓
7. User clicks save → encrypt → store as new vault file → update index
```

## 8. Cross-Origin Vault Access — The CORS Detail

The workspace at `workspace.sgraph.ai` needs to call vault APIs at `admin.send.sgraph.ai`. This is a cross-origin request.

### What Needs to Change on the Backend

The admin Lambda's CORS configuration must add `workspace.sgraph.ai` to allowed origins.

**Current state:** I need to verify exactly where CORS origins are configured. Expected location: the FastAPI app's CORS middleware setup in `Serverless__Fast_API` or the Lambda handler.

**Specific change needed:**
```python
# In the admin Lambda's FastAPI app configuration
allowed_origins = [
    "https://admin.send.sgraph.ai",        # existing
    "https://send.sgraph.ai",              # existing
    "https://workspace.sgraph.ai",         # NEW
    "https://vault.sgraph.ai",             # NEW (if not already)
]
```

**Credentials:** The vault API currently uses `credentials: 'include'` for same-origin cookie auth. For cross-origin, we need:
1. Backend: `Access-Control-Allow-Credentials: true`
2. Backend: Explicit origin (not wildcard `*`)
3. Frontend: `fetch(..., { credentials: 'include' })`

### Alternative: Token-Based Auth

If cookie-based cross-origin auth is complex, an alternative is vault-token-based auth:
- Workspace generates a session token on vault open
- Passes token in `x-sgraph-vault-token` header
- No cookies needed, simpler CORS

**Recommendation:** Verify current CORS config first. If cookies work cross-origin with the existing setup, use that. If not, implement token-based auth as a single new endpoint.

## 12. Technical Concerns

### C-1: CORS Verification (HIGH — blocks vault access)

**Risk:** If the admin Lambda's CORS config doesn't allow `workspace.sgraph.ai`, the vault panel can't load any data.
**Action:** Verify CORS config before any frontend work. Test with a simple fetch from a `workspace.sgraph.ai` origin.
**Mitigation:** If CORS changes require a backend deploy, this should be done in parallel with Days 1-2 frontend work.

### C-2: OpenRouter CORS Verification (MEDIUM — blocks LLM calls)

**Risk:** While OpenRouter claims browser support, we need to verify streaming works from a non-localhost origin.
**Action:** Day 1 test: make a streaming API call from a browser page hosted on a real domain.
**Mitigation:** If OpenRouter streaming has issues, fall back to non-streaming (wait for full response) as a temporary measure.

### C-3: Large Document Handling (LOW — Phase 1 scope)

**Risk:** LLM context windows limit how large a document can be transformed.
**Observation:** Claude 3.5 Sonnet via OpenRouter supports 200K tokens (~150K words). Most documents fit easily.
**Action:** Show estimated token count before sending. Warn if document exceeds model's context window.

### C-4: Vault API Base URL Configuration (LOW)

**Risk:** Hard-coding `admin.send.sgraph.ai` makes the workspace inflexible.
**Solution:** Read API base URL from a config constant at the top of `vault-api.js`. Default to `admin.send.sgraph.ai` but allow override via URL parameter or config file.

### C-5: This Competes with Vault Sprint Priorities (MEDIUM — scheduling)

**Observation:** The v0.10.32 debrief Sprint 1 (this week) focuses on vault fixes — folder upload bug, auth token separation, drag-and-drop, settings panel. The workspace build should probably be Sprint 2 or 3.
**Recommendation:** Don't start workspace build until vault fixes are shipped. The workspace depends on a stable vault.
