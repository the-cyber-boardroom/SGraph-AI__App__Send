## Components Built

### 1. vault-manager.js (1,800 lines)

The primary vault component — a full-featured encrypted file manager.

**Features delivered:**
- Tree-view file browser with folder hierarchy
- PKI key selector (generated keys from the PKI subsystem)
- File upload with client-side AES-256-GCM encryption
- File download with client-side decryption
- Drag-and-drop file upload (including drag-to-move between folders)
- File table with sortable columns (name, size, type, date)
- Detail panel with file metadata and raw encrypted data viewer
- Inline rename (double-click filename)
- File delete with confirmation
- File statistics (count, total size)
- Responsive resize handling

**Architecture evolution (3 phases in this session):**
1. **v1 — Detail panel layout** (2-column: tree | detail) — initial implementation
2. **v2 — Share integration** — added "Share via Send link" to vault files
3. **v3 — 3-row layout** (table | settings bar | preview/editor) — major refactor splitting the monolithic detail panel into three dedicated components

### 2. vault-editor.js (305 lines)

In-vault text and markdown editor.

**Features:**
- Full-height textarea with monospace font
- Live markdown preview toggle (custom renderer, zero dependencies)
- Dirty indicator (modified/unmodified)
- Save back to vault (re-encrypts via onSave callback)
- Keyboard shortcuts: Ctrl+S to save, Tab inserts spaces
- Line/char/byte statistics in footer
- Unsaved changes confirmation on close
- Dynamic height calculation with ResizeObserver tracking

### 3. vault-preview.js (200 lines)

Multi-format file preview renderer.

**Supported formats:**
- Text/code (syntax-highlighted monospace)
- Markdown (rendered HTML with headings, lists, code blocks, links)
- Images (blob URL rendering with max-size constraints)
- Loading state with spinner
- Empty state with icon
- Binary/unsupported file type indicator

### 4. vault-settings.js (238 lines)

Metadata and actions bar for selected vault items.

**Features:**
- File metadata display (name, type, size, MIME, upload date)
- Action buttons: Download, Share, Edit, Rename, Delete, Open folder
- Event-driven architecture (emits `vault-settings-*` custom events)
- Folder vs file context (different actions for each)
- Inline rename with input field

## Key Technical Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Split vault-manager into 4 components | Single-responsibility. The detail panel was growing to 300+ lines inside vault-manager. Splitting to vault-preview, vault-editor, vault-settings keeps each component focused. |
| 2 | 3-row layout (table, settings, preview) | Better UX than 2-column. The settings bar gives always-visible file actions. Preview fills remaining space below. |
| 3 | Event-driven component communication | vault-settings emits events, vault-manager listens. No tight coupling. Components can be tested independently. |
| 4 | ResizeObserver for editor height | CSS flex/grid couldn't reliably handle the nested height constraint (shell grid > main area > vault-manager > editor). JS calculation with ResizeObserver is the pragmatic fix. |
| 5 | Custom markdown renderer (zero dependencies) | Follows project principle of zero framework dependencies. Handles headers, lists, code blocks, bold, italic, links, blockquotes, horizontal rules. |
| 6 | Removed max-width: 1100px from admin-shell | Was preventing main content from expanding when debug panel closed. Changed to max-width: 100% so content fills available space. |

## IFD Compliance

All work follows IFD methodology:
- **Independence:** Each component is a standalone Web Component with its own styles
- **Versioned paths:** All files live under `v0/v0.1/v0.1.6/`
- **Layered loading:** index.html loads components in dependency order (Layer 4b)
- **No framework dependencies:** vanilla JS + Web Crypto API + CSS custom properties
- **Surgical versioning:** Only vault components are v0.1.6; shell stays at v0.1.3
