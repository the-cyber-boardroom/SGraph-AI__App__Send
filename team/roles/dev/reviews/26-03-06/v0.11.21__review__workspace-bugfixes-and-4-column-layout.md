## 2. Panel Collapse Evolution

The panel collapse system required 6 commits to get right. This section documents each attempt and why it failed, because the progression reveals a real lesson about CSS layout and the cost of incremental fixes on a flawed foundation.

### Attempt 1: Flex-based collapse (`df584c3`)

The first approach made panel headers into toggle buttons and used `flex` shorthand to collapse:

```js
panel.style.flex = collapsed ? '0 0 28px' : '1 1 auto';
```

**Result:** Data panel did not visually shrink. The parent container's `min-height` and the flex item's intrinsic content size prevented the panel from collapsing below its content height.

### Attempt 2: Explicit pixel height (`6c378b2`)

Added explicit inline styles to force the collapsed height:

```js
if (collapsed) {
    panel.style.flex      = '0 0 28px';
    panel.style.maxHeight = '28px';
    panel.style.overflow  = 'hidden';
} else {
    panel.style.flex      = '1 1 auto';
    panel.style.maxHeight = '';
    panel.style.overflow  = '';
}
```

**Result:** Data panel now collapsed correctly. But Source panel still did not collapse -- it was in a different flex context (horizontal, not vertical) and these rules only handled the vertical case.

### Attempt 3: Explicit pixel height, take 2 (`9f853a9`)

Replaced `flex: auto` with explicit pixel height for expanded state:

```js
panel.style.height = collapsed ? '28px' : '';
```

**Result:** Worked for individual panel collapse but exposed a new problem -- the sibling panel did not expand to fill the vacated space.

### Attempt 4: Sibling fill (`04ace51`)

When one panel collapses, the sibling gets `flex: 1`:

```js
const sibling = getSiblingPanel(panel);
if (collapsed) {
    sibling.style.flex = '1';
}
```

**Result:** Single-panel collapse now worked correctly. But collapsing both Source AND Data simultaneously did not cause the source zone container itself to shrink -- the outer grid column stayed at its original width.

### Attempt 5: Source zone shrink (`9f63076`)

Added logic to detect when all children of the source zone are collapsed and shrink the zone itself:

**Result:** Worked, but the code was becoming a maze of special cases. Each panel type had different collapse logic. The root cause was clear: the layout was a flex/grid hybrid where panels lived in different layout contexts.

### The Solution: 4-column CSS grid (`1d12d9b`, `2b8d23c`)

Replaced the entire nested flex/grid hybrid with a uniform 4-column CSS grid:

```
Source | Data | Script | Result
```

All four columns live in a single `grid-template-columns` declaration. Collapse is handled by one function that reads the state of all four panels and sets the grid template accordingly:

```js
function applyAllPanelState() {
    const states = ['source', 'data', 'script', 'result'].map(name => {
        return localStorage.getItem(`panel-${name}-collapsed`) === 'true';
    });
    const columns = states.map(collapsed => collapsed ? '28px' : '1fr').join(' ');
    workspace.style.gridTemplateColumns = columns;
}
```

One function. One layout declaration. All four panels handled uniformly. The 5 previous attempts -- flex hacks, sibling detection, zone shrinking -- all became unnecessary.

**Lesson:** The incremental fixes were treating symptoms of a structural problem. The layout was a hybrid of nested flex containers and grid, which meant each panel existed in a different layout context with different collapse semantics. The fix was not "better collapse logic" but "simpler layout."

## 3. The 4-Column Restructure

### Why the hybrid layout was replaced

The workspace layout before `1d12d9b` looked like this:

```
[Source Zone (flex column)]     [Script (flex)]  [Result (flex)]
  [Source Panel (flex item)]
  [Data Panel (flex item)]
```

Source and Data were stacked vertically inside a flex column. Script and Result were siblings at the grid level. This meant:

- Collapsing Source/Data required vertical flex logic
- Collapsing Script/Result required horizontal grid logic
- Collapsing the Source Zone when both children were collapsed required a third kind of logic
- None of these could share code

The 4-column grid makes all panels peers:

```css
.workspace {
    display: grid;
    grid-template-columns: 1fr 1fr 1fr 1fr;
}
```

Now collapsing any panel is the same operation: change its column from `1fr` to `28px`. No special cases, no sibling detection, no zone shrinking.

## 4. The `!important` Bug

**Commit:** `2b8d23c`

After the 4-column restructure, panel collapse still did not work. The grid columns stayed fixed regardless of JavaScript style changes.

**Root cause:** The CSS stylesheet contained:

```css
.workspace {
    grid-template-columns: 1fr 1fr 1fr 1fr !important;
}
```

The `!important` declaration in the stylesheet has higher specificity than inline styles set via JavaScript (`element.style.gridTemplateColumns = ...`). This is a well-known CSS specificity rule but easy to miss when the `!important` was added in an earlier commit for a different reason (likely to override some other conflicting rule).

**Fix:** Removed `!important` from the stylesheet and centralized all grid-template-columns changes through `applyAllPanelState()`. This function is the single authority for the grid layout -- no competing declarations.

```js
// Before: CSS had !important, JS style changes were silently ignored
// After:  CSS sets the default, applyAllPanelState() overrides via inline style

function applyAllPanelState() {
    const states = ['source', 'data', 'script', 'result'].map(name =>
        localStorage.getItem(`panel-${name}-collapsed`) === 'true'
    );
    const columns = states.map(c => c ? '28px' : '1fr').join(' ');
    document.querySelector('.workspace').style.gridTemplateColumns = columns;
}
```

The `!important` keyword should be treated as a code smell in any codebase where JavaScript dynamically modifies styles. It silently defeats programmatic control with no runtime error.

## 5. The Bundle Restore Fix

**Commit:** `63323ed`

This was the best fix of the entire sprint. Not because it was clever, but because it was the opposite of clever.

### The problem

When loading a saved bundle, the restore logic had to set panel collapse states. The bundle restore code contained its own 50-line implementation of panel collapse:

```js
// BEFORE: 50 lines of stale 3-column grid logic in bundle-manager.js
function restorePanelStates(bundle) {
    const source = bundle.panels?.source?.collapsed;
    const data   = bundle.panels?.data?.collapsed;
    const script = bundle.panels?.script?.collapsed;
    const result = bundle.panels?.result?.collapsed;

    // ... 40+ lines of manual DOM manipulation ...
    // ... setting flex, maxHeight, overflow on individual panels ...
    // ... handling sibling fill logic ...
    // ... source zone shrink logic ...
    // ... all using the OLD 3-column layout assumptions ...
}
```

This code was written during the flex/grid hybrid era. After the 4-column restructure, it was not just wrong -- it was actively fighting the new layout by setting stale CSS properties.

### The fix

Replaced 50 lines with 3:

```js
// AFTER: 3 lines -- write state, notify the layout system
function restorePanelStates(bundle) {
    ['source', 'data', 'script', 'result'].forEach(name => {
        localStorage.setItem(`panel-${name}-collapsed`, bundle.panels?.[name]?.collapsed ?? false);
    });
    window.dispatchEvent(new CustomEvent('panel-state-changed'));
}
```

The `panel-state-changed` event triggers `applyAllPanelState()`, which is the single source of truth for layout. The bundle restore code no longer needs to know anything about CSS grid, flex, pixel heights, or column counts. It writes the desired state and tells the system to apply it.

### Why this matters

The 50 lines were not just dead weight. They were a **second implementation** of panel collapse that:

1. Could not stay in sync with the primary implementation
2. Used assumptions from a layout that no longer existed
3. Would break again every time the layout changed
4. Made the bundle restore a source of bugs rather than a consumer of a working system

Deleting code that duplicates logic elsewhere is the most reliable kind of fix. It cannot introduce new bugs. It cannot regress. It reduces the surface area for future errors.

## 6. Code Quality Pattern: Removal as the Best Fix

Across this sprint, the most effective fixes were deletions:

| Fix | Lines removed | Lines added | Effect |
|-----|---------------|-------------|--------|
| Bundle restore (`63323ed`) | ~50 | 3 | Eliminated stale layout duplication |
| `!important` removal (`2b8d23c`) | 1 | 0 | Unblocked all JS-driven layout changes |
| 4-column restructure (`1d12d9b`) | ~80 (flex hacks) | ~20 (grid) | Eliminated 5 special-case collapse paths |

The pattern: when a fix requires understanding multiple interacting systems, check whether the interaction itself is the problem. Often the right fix is not "handle the interaction better" but "remove the interaction by consolidating to one system."

## Recommendations

1. **Audit for other `!important` declarations** in workspace CSS. Any `!important` that governs a property also set by JavaScript is a latent bug.
2. **Apply the event dispatch pattern** to other bundle restore operations. If restore code manually sets DOM state, it should instead write to the state store and dispatch an event.
3. **Consider persisting panel state in the bundle format** rather than localStorage, so different bundles can have different default layouts.
