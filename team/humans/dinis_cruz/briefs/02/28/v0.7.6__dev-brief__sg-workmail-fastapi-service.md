# Dev Brief: SG WorkMail — FastAPI Service Replacing n8n Email Workflows

**version** v0.7.6  
**date** 28 Feb 2026  
**from** Human (project lead)  
**to** Developer (lead), Architect, Sherpa, AppSec  
**type** Dev brief — new service, immediate priority  

---

## Objective

Replace the current n8n-based email workflows with a new FastAPI service that exposes WorkMail operations as both REST endpoints and MCP tools. This directly addresses two problems identified in v0.7.4:

1. **n8n email workflows are slow, inflexible, and add complexity** (v0.7.4 n8n assessment)
2. **Email sending is token-expensive** — Claude generates full HTML per email instead of using templates (v0.7.4 email workflow optimisation)

The FastAPI service gives us a clean, testable, type-safe API that Claude can call via MCP with minimal token usage. The template and structured message patterns also extend beyond email — the same approach works for WhatsApp messages and other messaging channels in the future.

---

## Why Not n8n?

The v0.7.4 n8n assessment identified that n8n is good for triggers and routing but bad for: email body construction, data transformation, template rendering, and business logic. The current email workflow through n8n is:

- **Slow** — multiple n8n nodes per email, visual debugging only
- **Limited** — no type safety, no template engine, tiny text boxes for configuration
- **Complex** — more moving parts than necessary for what is fundamentally: read inbox, send email
- **Token-expensive** — Claude must describe the full email through MCP calls to n8n, which then assembles and sends it

The FastAPI service replaces n8n for email operations. n8n can remain for triggers (e.g., "new email arrived" → trigger a Claude session) but the actual email read/write/send operations move to the FastAPI service.

## Phase 1: Core Email Primitives

Four fundamental operations — the raw building blocks:

### 1. List Inbox

```
GET /inbox
→ Returns: [{id, from, to, subject, date, snippet, read}]
```

Lists recent messages in the Sherpa's inbox. Returns metadata only (not full bodies). Supports pagination and filtering (unread, date range, sender).

### 2. Read Email

```
GET /inbox/{message_id}
→ Returns: {id, from, to, cc, subject, date, body_text, body_html, attachments: [{name, size, content_type}]}
```

Returns the full email content for a specific message. Body in both plaintext and HTML.

### 3. Send Email

```
POST /send
Body: {to, cc?, bcc?, subject, body_html, body_text?, attachments?}
→ Returns: {message_id, status}
```

Sends a raw email. The caller provides the full subject and body. This is the low-level primitive — Claude uses this when it needs full control over the email content.

### 4. Reply to Email

```
POST /inbox/{message_id}/reply
Body: {body_html, body_text?}
→ Returns: {message_id, status}
```

Replies to a specific email. The service handles `In-Reply-To` headers, quoting, and threading.

### 5. List Sent / Outbox

```
GET /sent
→ Returns: [{id, to, subject, date, status}]
```

Lists recently sent emails. Essential for visibility — "what has the Sherpa actually sent?" Without this, sent emails disappear into the void. Supports the same pagination and filtering as `/inbox`.

### MCP Exposure

All five operations are exposed as MCP tools via fastapi-mcp:

```python
@app.get("/inbox")
async def list_inbox(limit: int = 20, unread_only: bool = False):
    """List recent emails in the Sherpa inbox."""
    ...

@app.get("/inbox/{message_id}")
async def read_email(message_id: str):
    """Read a specific email by ID."""
    ...

@app.post("/send")
async def send_email(email: EmailSendRequest):
    """Send an email."""
    ...

@app.post("/inbox/{message_id}/reply")
async def reply_to_email(message_id: str, reply: EmailReplyRequest):
    """Reply to a specific email."""
    ...

@app.get("/sent")
async def list_sent(limit: int = 20):
    """List recently sent emails — what has the Sherpa actually sent?"""
    ...
```

From Claude, this becomes:

```
Human: "Check the Sherpa inbox for new emails"
Claude: [calls list_inbox MCP tool] → sees 3 new messages
Claude: [calls read_email for message_id X] → reads the content
Claude: "There are 3 new emails. The first is from..."
```

This is dramatically simpler than the current n8n workflow, uses fewer tokens, and gives Claude direct access to the inbox.

## Phase 2: Template-Based Sending

The v0.7.4 email workflow brief identified that Claude regenerates ~14KB of HTML per email. The fix: templates.

### Template Definition

Templates are type-safe schemas with named placeholders:

```python
class EarlyAccessTemplate(BaseModel):
    """Early Access Programme welcome email."""
    recipient_name: str
    recipient_email: str
    access_code: str
    access_link: str
    language: str = "en"

@app.post("/send/template/early-access")
async def send_early_access(data: EarlyAccessTemplate):
    """Send the Early Access Programme welcome email.
    The HTML template is stored server-side. Only the placeholder 
    values are provided by the caller."""
    html = render_template("early-access", data)
    return await send_email_internal(
        to=data.recipient_email,
        subject=f"Welcome to SG/Send Early Access, {data.recipient_name}",
        body_html=html
    )
```

### What Claude Sends (Phase 2)

Instead of generating 14KB of HTML:

```
Claude: [calls send_early_access MCP tool]
  Input: {
    recipient_name: "Alice",
    recipient_email: "alice@example.com",
    access_code: "EAP-March-Week-1",
    access_link: "https://send.sgraph.ai/eap/...",
    language: "en"
  }
```

~200 bytes instead of ~14KB. For 50 emails: 10KB instead of 700KB. The template HTML lives server-side, versioned, reviewed, and rendered on send.

### Batch Sending

```
POST /send/template/early-access/batch
Body: {
  recipients: [
    {name: "Alice", email: "alice@example.com", access_code: "...", access_link: "..."},
    {name: "Bob",   email: "bob@example.com",   access_code: "...", access_link: "..."},
    ...
  ],
  language: "en"
}
→ Returns: {
    batch_id: "...",
    total: 50,
    sent: 48,
    failed: 2,
    failures: [{email: "...", error: "..."}]
  }
```

Claude provides the list. The service sends them all. One MCP call, not 50. No context window exhaustion.

### Send Manifest

Per the v0.7.4 brief, every batch creates a manifest — a JSON record of what was sent, to whom, when, with what status. The manifest is stored in the data room (and optionally as an EML copy per the Email-FS brief).

## Phase 3: Type-Safe Structured Messages

Beyond email templates, introduce a structured message format that Claude can use to compose messages in natural text that get rendered into properly formatted emails:

```python
class StructuredMessage(BaseModel):
    """A message composed in natural text, rendered to HTML by the service."""
    to: str
    subject: str
    greeting: str          # "Hi Alice"
    body_paragraphs: list[str]  # Each paragraph as plain text
    sign_off: str          # "Best regards"
    sender_name: str       # "Sherpa"
    sender_role: str       # "SG/Send Team"
    ps: str | None = None  # Optional P.S.

@app.post("/send/structured")
async def send_structured(msg: StructuredMessage):
    """Compose and send a structured message.
    The service renders the plain text into a properly formatted 
    HTML email using the SG/Send email design system."""
    ...
```

Claude writes naturally ("Hi Alice, I wanted to follow up on..."). The service handles HTML rendering, styling, headers, footers, and branding. This is the "type the message in normal text and submit it using a type-safe schema" from the voice memo.

## Phase 4: EML Archival and Security

### EML Copy in Data Room

Every sent email is saved as an EML file in the data room (per the v0.7.4 Email-FS brief):

```
.emails/sent/2026-02-28/
├── 001__alice@example.com__welcome-early-access.eml
├── 002__bob@example.com__welcome-early-access.eml
└── manifest.json
```

Once EML viewer support lands in the data room, these become browsable, searchable, and verifiable.

### Security: AppSec in the Loop

As the prompt injection brief (v0.7.4) identified, the inbox is an attack surface. Future phases add:

| Feature | Description |
|---|---|
| **AppSec pre-read** | Before Claude reads an email, AppSec agent classifies it (safe/suspicious/malicious) |
| **Encrypted inbox** | Emails stored encrypted in the data room — read access requires AppSec signature or decryption |
| **Indirect read** | Claude doesn't read the raw email — it reads AppSec's sanitised summary |
| **Send approval** | For batch sends, the manifest is reviewed before send is triggered |

These build on the v0.7.4 prompt injection testing results. Start with Phase 1 (raw access), add security layers based on test findings.

## Authentication: MCP → FastAPI

### The Problem

Claude's MCP connector needs to authenticate with the FastAPI service. The current pattern is API-key-based auth. The question is: how does the MCP configuration in Claude pass the API key?

### The Solution

The MCP server URL includes auth configuration. The fastapi-mcp library supports header-based auth:

```json
// Claude MCP configuration
{
  "name": "sg-workmail",
  "type": "url",
  "url": "https://workmail.sgraph.ai/mcp",
  "headers": {
    "Authorization": "Bearer {api-key}"
  }
}
```

The API key is stored in the Claude MCP connector settings. The FastAPI service validates it on every request.

**Action:** Confirm the exact mechanism for passing auth headers in Claude's MCP connector configuration. This is the one integration question that needs answering before implementation.
