## Core Idea

The same website — identical content, identical structure — is published in three delivery modes:

```
https://sgraph.ai/                          ← Standard (plaintext HTML)
https://sgraph.ai/sym/                      ← Symmetric encrypted (key required)
https://sgraph.ai/pki/                      ← PKI encrypted (private key required)
```

This is not a gimmick. It serves four purposes simultaneously:

1. **Dogfooding:** We test our own encryption on real content, in production, with real users
2. **Data vault proving ground:** The encrypted site IS a data vault — pages that load only if you have the key. This is the exact same concept, just applied to a website instead of a document.
3. **User education:** Visitors see how PKI and symmetric encryption work by experiencing it — not by reading about it
4. **Performance benchmarking:** We measure decryption overhead, browser compatibility, UX friction, and SEO impact with real traffic

## Part 1: Three Delivery Modes

### Mode 1: Standard (Plaintext)

Standard static HTML. No encryption. This is the public-facing website that search engines index, that visitors find, that the exec from the breakfast meeting sends to his team.

URL: `https://sgraph.ai/` (or `https://sgraph.ai/en-gb/` with language prefix)

### Mode 2: Symmetric Encrypted

The entire site is encrypted with a symmetric key. To view any page, the user needs the key. The key is shared publicly (because this is public content — the point is the demonstration, not the secrecy).

URL: `https://sgraph.ai/sym/` (or a better name — see naming discussion below)

How it works:
1. User navigates to `/sym/`
2. Browser loads a small decryption loader script
3. User is prompted for the key (or it's provided via URL fragment, or pre-loaded from a previous visit)
4. Each page is decrypted client-side before rendering
5. The experience is identical to the plaintext site — just with a decryption step

### Mode 3: PKI Encrypted

The site is encrypted with the site's public key. To view it, the user needs the corresponding private key.

URL: `https://sgraph.ai/pki/`

How it works:
1. User navigates to `/pki/`
2. Browser loads the decryption loader
3. User provides their private key (from their vault, from a downloaded key file, or from browser storage)
4. Each page is decrypted with the private key
5. Same content, same experience — different trust model

### Naming

The URL segments need to be intuitive for non-technical users:

| Option | Symmetric | PKI | Notes |
|---|---|---|---|
| `sym` / `pki` | Technical, short | Technical, short | Too jargony for users |
| `shared-key` / `private-key` | Descriptive | Descriptive | Longer but educational |
| `key` / `your-key` | Simple | Personal | Interesting — "key" = you need A key, "your-key" = you need YOUR key |
| `encrypted` / `personal` | Clear | Clear | Might work |

**Recommendation:** Experiment with multiple options. The naming itself is a UX research question — which framing helps users understand the difference?

## Part 2: Decryption Approaches to Evaluate

### Approach A: Inline Script + Encrypted Blob

Each HTML page contains a decryption script and the encrypted content as a blob within the page:

```html
<html>
<head><script src="./decrypt.js"></script></head>
<body>
  <div id="content"></div>
  <script>
    const encrypted = "base64-encoded-encrypted-blob...";
    const key = getKey(); // prompt, URL fragment, or stored
    document.getElementById("content").innerHTML = decrypt(encrypted, key);
  </script>
</body>
</html>
```

Pros: Simple, self-contained, works offline.  
Cons: Encrypted blob is visible in source, decryption script is downloadable (attackable), every page carries the decryption overhead.

### Approach B: Service Worker / Web Worker Decryption

A service worker intercepts page requests and decrypts content transparently:

```
1. User loads /pki/ → service worker installs
2. Service worker intercepts all subsequent requests
3. For each page request:
   a. Fetch the encrypted page from the server
   b. Decrypt using the key held in the worker
   c. Return the decrypted HTML to the browser
4. From the user's perspective: pages just load (after initial key entry)
```

Pros: Transparent after setup, feels like a normal website, key stays in the worker (not in page JS), enables the "pod" / "enclave" concept for secure browsing.  
Cons: Service worker support varies, more complex to implement, first-load experience is different.

### Approach C: Hybrid

Approach A for the landing/entry page (self-contained, works everywhere). Approach B activated after the first page load for subsequent navigation. Best of both worlds.

**Recommendation:** Start with Approach A (ship fast, learn). Develop Approach B in parallel as the path to the data vault's page-level encryption. Approach C is the production target.

### Performance Questions to Answer

| Question | How We Measure |
|---|---|
| Decryption latency per page | Client-side timing: fetch → decrypt → render |
| First meaningful paint difference (plain vs encrypted) | Lighthouse / Web Vitals comparison across modes |
| Browser compatibility | Test across Chrome, Firefox, Safari, Edge, mobile browsers |
| Key entry UX friction | User testing: how many abandon at the key prompt? |
| SEO impact of encrypted pages | Are encrypted pages indexed? (They shouldn't be, but verify) |
| Offline performance | Does the encrypted site work from a downloaded folder? |

## Part 3: Public Keys and the Private Key Paradox

The encrypted versions of the site use public content — everything on the plaintext site is already public. So sharing the private key is safe in this context. But how and where we share it is itself an educational demonstration.

### Scenarios for Key Distribution

| Scenario | Where the key lives | What it demonstrates |
|---|---|---|
| **Key in the URL fragment** | `sgraph.ai/sym/#key=abc123` | Symmetric mode — anyone with the link can view (like SG/Send's standard share links) |
| **Key on the GitHub repo** | `github.com/sgraph-ai/keys/` | Public key distribution — verifiable, version-controlled |
| **Key in an S3 bucket** | Downloadable from a known URL | Key distribution via infrastructure |
| **Key in the user's vault** | Stored locally from a previous session | Persistent access — the user "owns" the key |
| **Key delivered via SG/Send** | One-time link → key download | Dogfooding SG/Send to distribute the keys to view the SG/Send website (recursive!) |

Each scenario is a real-world key distribution pattern that we're demonstrating with our own website. The educational value is built into the experience.

### Dogfooding Encrypted Storage

A side exploration: can we start storing some of our own artifacts (CI outputs, release metadata, dependency snapshots) in encrypted format in S3? This is public data encrypted with a published key — so anyone with the key can read it. But it means:

- We exercise the encryption pipeline on real data
- We learn the performance implications of encrypted-at-rest storage
- We move toward a world where "encrypted by default" is the norm, not the exception
- We demonstrate that "S3 encrypted at rest" (AWS-managed keys) is fundamentally different from "encrypted with OUR keys that WE control"

The distinction matters: AWS "encryption at rest" protects against someone stealing a physical hard drive from an AWS data center. It does not protect against anyone with AWS API access to your account. Our encryption protects against anyone without the key — including AWS, including a compromised account.

### Multi-Cultural, Not Just Multi-Lingual

Translation is not just language. Each locale should consider:

| Dimension | Example |
|---|---|
| **Language** | English → Portuguese |
| **Formality** | UK formal vs US casual vs Japanese honorific |
| **Metaphors** | "Lock and key" works in English; may not translate directly |
| **Date/number formats** | DD/MM/YYYY vs MM/DD/YYYY, 1.000 vs 1,000 |
| **Reading direction** | LTR vs RTL (Arabic) |
| **Colour associations** | Red = danger (Western) vs red = prosperity (Chinese) |
| **Legal/regulatory context** | GDPR (EU), LGPD (Brazil), PIPL (China) |

### Translation Workflow

```
Phase 1 (now):
  AI-generated translations with clear disclaimer:
  "This translation was generated by AI. Help us improve it: [GitHub PR link]"

Phase 2 (community):
  - Source markdown stored in GitHub
  - Community members submit translation PRs
  - Agent team (Librarian + Ambassador) reviews PRs
  - Human translators validate AI output
  
Phase 3 (agent-assisted):
  - Translation agents generate initial versions
  - Cultural agents review for locale-specific issues
  - Human translators do final pass
  - Agentic pipeline: source change → auto-translate → PR → review → merge → deploy
```

### Educational Pages

Add a section of educational content to the website — interactive, JavaScript-powered pages that explain:

| Topic | Format | SEO Value |
|---|---|---|
| **What is PKI?** | Interactive diagram: generate a key pair, encrypt a message, decrypt it | High — people search for this |
| **Symmetric vs asymmetric encryption** | Side-by-side comparison with live demo | High |
| **What is hashing?** | Enter text, see the hash, change one character, see the hash change completely | High |
| **Weak ciphers and brute force** | Visual: how long to crack DES vs AES at different key lengths | Medium |
| **What does "zero-knowledge" mean?** | Step-by-step walkthrough of an SG/Send file transfer | High — directly ties to product |
| **TLS vs end-to-end encryption** | Diagram showing where TLS stops and E2E continues | High — from v0.7.1 security explainer |

These pages are educational first, marketing second. Brand connection is soft — they live on the sgraph.ai domain, but the content is genuinely useful regardless of whether the reader uses SG/Send.

Multi-lingual educational content is particularly valuable: crypto education in Portuguese, Spanish, Arabic, Hindi reaches audiences massively underserved by English-only resources.

## Part 5: Agent Dashboard and IssuesFS on the Website

### Agent Status Pages

The static website becomes a coordination channel. For each agent, publish a status page:

```
sgraph.ai/team/{role}/
├── index.html          ← Current status card: what the agent is working on
├── tasks/              ← Current task list (from IssuesFS)
├── briefs/             ← Latest briefs assigned to this role
├── debriefs/           ← Latest debriefs from this role
└── history/            ← Past work, linked to GitHub commits
```

The status card is a compact summary:


### IssuesFS Publishing

The IssuesFS flat file (already working in the repo) is published to the static site so it can be browsed in a web UI:

```
sgraph.ai/issues/
├── index.html          ← Filterable, sortable view of all issues
├── open/               ← Open issues
├── in-progress/        ← In-progress issues
└── closed/             ← Closed issues
```

This is fractal: every task, every work stream, every iteration can have its own IssuesFS folder. The top-level view consolidates across all of them.

### Workflow: Website → Claude Session

The status page isn't just for reading. It defines what happens when a Claude session starts:

```
1. Human opens sgraph.ai/team/architect/ on their phone
2. Sees: "Current: Static site deployment — waiting on S3 bucket"
3. Opens Claude Code Web
4. Says: "Start a session for the Architect — execute what's on the status page"
5. Claude reads the status page, loads the CLAUDE.md, picks up the task
```

The website is the briefing. The agent reads the website. The loop closes.

## Part 6: Micropayment Experiment (Future)

An interesting side exploration: if every encrypted page requires decryption, and decryption has a computational cost (borne by the client's browser), can we model micropayments around page access?

```
User opens encrypted page → decryption occurs → page renders
                          ↓
                    micro-charge logged
```

This is a thought experiment for now — not a P0 implementation. But the encrypted website variants give us a natural place to experiment with:

- Per-page access logging (how many pages, how often)
- Token-based access (consume a token to decrypt a page)
- Pay-per-view models for premium content
- The philosophical question: should we charge for computation done on the client?

Pin this for the Accountant and Advocate to explore.
