# Part 1: What whatismyipaddress.com Is Doing

## Context

The user visited whatismyipaddress.com, clicked **"Disagree"** on the consent banner, and then inspected the browser's Network tab, Local Storage, Session Storage, IndexedDB, and Cookies. The screenshots below show what the site captured — or had already captured — despite the explicit refusal of consent.

## What the User Sees

The visible page is simple: your IPv4 address (203.0.113.42), whether IPv6 is detected, your ISP (Example ISP Ltd), city (London), region (England), country (UK), and a map pinpointing your approximate location. There's a banner ad (Three mobile), a push notification prompt, and a scare-tactic CTA: "Your location may be exposed!" with a bright red "HIDE MY IP ADDRESS NOW" button selling VPN services.

The functional content — the IP address and basic geolocation — could be delivered in under 5KB of HTML with zero JavaScript.

## The Irony

A website called "What Is My IP Address" — positioned as a privacy awareness tool ("Your location may be exposed!") — is itself running an aggressive tracking ecosystem. The site's business model is:

1. Scare you about your IP being visible
2. Track you while you're reading the warning (even with "Disagree" — and likely far more extensively with "Agree")
3. Sell you a VPN to "protect your privacy"
4. Share your data with up to 1,570 advertising partners

The "We value your privacy" banner is a compliance checkbox, not a privacy feature. The "Disagree" button reduces but does not eliminate tracking. The data captured with "Reject" selected is likely a subset of what would be captured with "Agree" — which makes the volume documented above even more striking, as it represents the **minimum** tracking state.

## Objective

Build a "What's My Connection" page at `tools.sgraph.ai/my-connection` that provides the same useful information as whatismyipaddress.com — your IP address, approximate location, ISP, browser details — but built entirely on SG/Send's security principles. No tracking. No cookies. No third-party requests. No consent banners (because there's nothing to consent to).

This page serves three purposes:

1. **Useful tool** — people genuinely need to check their IP address, ISP, and connection details
2. **Educational contrast** — a live, side-by-side comparison with how the traditional web works (and what SG/Send does differently)
3. **Trust builder** — a visitor's first experience with SG/Send's tools demonstrates the privacy principles before they ever upload a file

---

## Design Principles

### 1. Zero Tracking

| Rule | Detail |
|---|---|
| **No cookies** | None. Not first-party, not third-party. The response headers include no `Set-Cookie`. |
| **No localStorage** | Nothing written to localStorage. Nothing read from it. |
| **No sessionStorage** | Nothing. |
| **No IndexedDB** | Nothing. |
| **No fingerprinting** | No canvas fingerprinting, no WebGL fingerprinting, no audio fingerprinting, no font enumeration. |
| **No analytics** | No Google Analytics, no Clarity, no FullStory, no Amplitude, no Mixpanel, no Plausible, no Fathom, nothing. |
| **No third-party requests** | Zero. Every resource (HTML, CSS, JS, fonts) loaded from our domain. See: zero external dependencies (v0.7.6 arch brief). |
| **No push notification SDK** | No OneSignal, no Firebase Cloud Messaging. |
| **No ad network** | No ads. No ad exchange scripts. No bid requests. |

### 2. Minimal Server-Side Data

The server necessarily sees the request. This is how TCP/IP works — to respond, the server must know the IP address. But we can minimise what we do with it:

| Data Point | Server Sees It? | Server Stores It? | Server Sends to Third Party? |
|---|---|---|---|
| IP address | Yes (TCP requirement) | CloudFront access logs only (with retention policy) | No |
| User agent | Yes (HTTP header) | CloudFront access logs only | No |
| Requested URL | Yes (HTTP header) | CloudFront access logs only | No |
| GeoIP lookup | Done server-side or via CloudFront headers | Not stored beyond the response | No |
| ISP/ASN lookup | Done server-side or via CloudFront headers | Not stored beyond the response | No |

CloudFront provides GeoIP headers (`CloudFront-Viewer-Country`, `CloudFront-Viewer-City`, etc.) without needing a separate lookup service. The Lambda@Edge or CloudFront function reads these headers, includes them in the response, and discards them. No database of visitor IPs. No analytics pipeline.

## Technical Implementation

### Architecture

```
User → CloudFront → Lambda@Edge (or CloudFront Function)
                         ↓
                    Reads CloudFront-Viewer-* headers:
                      - CloudFront-Viewer-Address (IP)
                      - CloudFront-Viewer-Country
                      - CloudFront-Viewer-City
                      - CloudFront-Viewer-Country-Region-Name
                    Reads HTTP headers:
                      - User-Agent (for browser info display only)
                      - Accept-Language (for language detection)
                         ↓
                    Injects into static HTML template
                         ↓
                    Returns HTML (no Set-Cookie, no tracking headers)
```

**Alternative (fully static + client-side):** The page could be fully static with a tiny API endpoint (`api.sgraph.ai/my-ip`) that returns the viewer's IP and CloudFront geo headers as JSON. The static page fetches this single endpoint and renders it. This keeps the page in the static deployment pipeline while still providing server-side IP detection.

```
Static HTML (from S3/CloudFront) → JS fetches api.sgraph.ai/my-ip
                                          ↓
                                    Returns JSON:
                                    {
                                      "ip": "203.0.113.42",
                                      "country": "GB",
                                      "city": "London",
                                      "region": "England",
                                      "isp": "Example ISP Ltd"
                                    }
                                          ↓
                                    Client renders (no storage written)
```

### Response Headers

```http
HTTP/2 200
Content-Type: text/html; charset=utf-8
Cache-Control: no-store
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self'; img-src 'self'; font-src 'self'; connect-src 'self' api.sgraph.ai; frame-ancestors 'none'
Referrer-Policy: no-referrer
Permissions-Policy: camera=(), microphone=(), geolocation=(), interest-cohort=()
```

Note:
- **No `Set-Cookie` header** — nothing is set
- **Strict CSP** — only loads resources from `'self'` (and `api.sgraph.ai` for the IP endpoint). Third-party scripts cannot load even if injected.
- **`interest-cohort=()`** — explicitly opts out of Google's FLoC/Topics API
- **`Referrer-Policy: no-referrer`** — if the user clicks a link, the destination doesn't learn they came from our page
- **`Cache-Control: no-store`** — the response contains the user's IP, so it should not be cached

## Additional Information to Display

Beyond basic IP/location, the page can show additional connection details that are useful for technical users — all derived from HTTP headers or client-side JavaScript with zero tracking:

| Section | Data | Source |
|---|---|---|
| **Connection** | IPv4, IPv6, ISP/ASN | CloudFront headers |
| **Location** | Country, region, city (approximate) | CloudFront GeoIP headers |
| **Browser** | Browser name/version, OS, platform | `navigator.userAgent` (displayed to user, not stored) |
| **TLS** | Protocol version (TLS 1.2/1.3), cipher suite | CloudFront headers or `SecurityPolicyProtocol` |
| **HTTP** | HTTP version (HTTP/2, HTTP/3), compression | Response headers |
| **DNS** | DNS resolver detected (if possible via timing) | Client-side timing heuristic |
| **Headers sent** | Full list of HTTP headers the user's browser sent | Displayed back to the user — educational ("this is what every website you visit can see") |

The "Headers sent" section is particularly educational — users don't realise their browser sends `Accept-Language`, `User-Agent`, `Accept-Encoding`, and other headers that reveal information about them on every request.
