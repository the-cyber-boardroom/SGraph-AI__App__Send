## The Problem: Three UX Failures to See Your Own Logs

The current SG/Send infrastructure already captures CloudFront real-time logs via the pipeline:

```
CloudFront → Kinesis Data Firehose → S3
```

The logs land in:
```
s3://{aws-account-id}--sgraph-send-cf-logs--eu-west-2/cloudfront-realtime/2026/02/27/
```

Each log file is a gzipped text file (~2.5KB compressed). The filename looks like:
```
sgraph-send-cf-logs-to-s3-2-2026-02-27-07-58-29-{uuid}.gz
```

To actually **read** this file, the human tried two paths — both painful.

### Path A: Download and Open Locally

1. Click "Download" in the S3 console → file downloads (11KB)
2. Double-click to unzip → **macOS Archive Utility fails**: "Unable to expand — it is in an unsupported format"
3. Try `gunzip` in the terminal → **also fails**: `gunzip: not in gzip format`
4. Try `cat` on the file to inspect what's actually inside → **the content is plaintext.** The file is not compressed at all. It's a plain text log file with a `.gz` extension.

**The file was never gzipped.** The Kinesis Firehose → S3 pipeline is writing plaintext CloudFront logs but naming them with a `.gz` extension. The S3 console shows "Type: gz" because it's reading the extension, not the content. Every tool that trusts the extension (macOS Archive Utility, `gunzip`) fails. Only `cat` — which ignores the extension and reads raw bytes — reveals the truth.

**This is a pipeline configuration bug.** Either:
- Firehose should be configured to actually gzip-compress the output (saves storage, matches the extension), or
- Firehose should not append `.gz` to the filename if compression is disabled

**DevOps action:** Check the Kinesis Data Firehose delivery stream configuration. The compression setting is likely set to "disabled" or "none" while the S3 prefix/suffix still appends `.gz`. Fix either the compression setting or the file naming.

**Friction:** 3 failed attempts (Archive Utility, `gunzip`, confusion) before `cat` revealed the content was plaintext all along. A developer who didn't think to try `cat` on a `.gz` file might have given up entirely.

### Path B: S3 Select (Query in Place)

1. Click "Object actions" → "Query with S3 Select"
2. Read the input settings (format: CSV, delimiter: Comma, compression: GZIP) — the compression was set to GZIP, but the file isn't actually gzipped. S3 Select apparently handled this gracefully (it likely detected the content wasn't compressed and read it as plaintext anyway — silently papering over the pipeline bug).
3. The default SQL query is `SELECT * FROM s3object s LIMIT 5`
4. Click "Run SQL query" → results appear (5 records, 2597 bytes, 711ms)
5. View in "Raw" format — a wall of space-separated fields, URL-encoded values, wrapping text. Barely readable.
6. View in "Formatted" format — marginally better, but the columns are unlabelled and the horizontal scrolling makes it hard to correlate fields.

**The hidden UX trap:** The `LIMIT 5` in the default query means you're only seeing 5 records. There's **no indication in the UI that the file contains more records.** The human's experience: "Surely there are more requests here" — confirmed only after downloading and viewing the full file locally, which showed 25+ records.

This is a subtle but dangerous UX issue. If you're debugging a production problem and you trust the S3 Select results, you might conclude there were only 5 requests when there were actually 25, 250, or 2500. The AWS console gives no record count, no "showing 5 of N" indicator, no warning.

### Path C: What Should Have Happened

```
1. Open tools.sgraph.ai/cf-logs
2. Paste the S3 URI (or browse the log bucket)
3. See a formatted, searchable, filterable table of all log entries
4. With labelled columns, human-readable timestamps, decoded URLs
5. In 2 clicks, under 3 seconds
```

## The Brief: CloudFront Log Viewer Tool

### What It Does

A web-based tool at `tools.sgraph.ai/cf-logs` (or a component within the admin dashboard) that:

1. **Loads CloudFront log files** — from S3 URI, file upload, or paste raw log content
2. **Parses the CloudFront log format** — maps fields to labelled columns
3. **Displays as a sortable, filterable table** — with human-readable timestamps, decoded URLs, colour-coded status codes
4. **Shows all records** — no silent truncation
5. **Handles .gz transparently** — decompresses gzipped files in the browser (using pako or the Compression Streams API)
6. **Runs entirely client-side** — the log data never leaves the browser (same zero-tracking principle as the IP tool)

### Client-Side Architecture

```
User pastes S3 URI or drops .gz/.txt file
       ↓
Browser fetches / reads the file
       ↓
Check gzip magic bytes (0x1f 0x8b):
  ├── Yes → decompress (Compression Streams API or pako.js)
  └── No  → treat as plaintext (handles the Firehose misconfig)
       ↓
Parse CloudFront log format (space-separated, known field order)
       ↓
Render as interactive table
       ↓
No data sent anywhere. No cookies. No tracking.
```

**Critical:** Log files contain sensitive data — IP addresses, user agents, request paths, transfer IDs, token lookups. This tool **must** process everything client-side. The server serves the static HTML/JS. The log data stays in the browser.

### Table Columns (Parsed and Labelled)

| Column | Raw Field | Transformation |
|---|---|---|
| **Time** | `1772179106.892` | → `2026-02-27 07:58:26.892 UTC` (human-readable) |
| **TTFB** | `0.073` | → `73ms` (with colour: green <100ms, yellow <500ms, red >500ms) |
| **Status** | `307` | → `307` with colour badge (2xx green, 3xx blue, 4xx yellow, 5xx red) |
| **Size** | `4312` | → `4.2 KB` (human-readable) |
| **Method** | `GET` | As-is |
| **Host** | `send.sgraph.ai` | As-is, filterable |
| **Path** | `/send/v0/v0.1/v0.1.5/js/i18n.js` | URL-decoded, with filter/search |
| **Edge** | `LHR5-P7` | → `London (LHR5-P7)` (map edge code to city name) |
| **Cache** | `Miss` | → `Miss` / `Hit` with colour (Hit=green, Miss=yellow) |
| **TLS** | `TLSv1.3 TLS_AES_128_GCM_SHA256` | → `TLS 1.3 / AES-128-GCM` (shortened) |
| **HTTP** | `HTTP/2.0` | → `H2` (shortened) |
| **Browser** | `Mozilla/5.0 (Macintosh...)` | → `Chrome 145 / macOS` (parsed UA string) |
| **Request ID** | `{cf-request-id}...` | Truncated with copy button |

### Features

| Feature | Description |
|---|---|
| **Filter by host** | Show only `send.sgraph.ai` or `admin.send.sgraph.ai` |
| **Filter by status** | Show only errors (4xx/5xx), redirects (3xx), successes (2xx) |
| **Filter by cache status** | Show only cache misses (useful for tuning cache config) |
| **Filter by edge location** | Show only London, only Chicago, etc. |
| **Search by path** | Find all requests to `/transfers/complete/*` or `/tokens/lookup/*` |
| **Sort by any column** | Click column header to sort |
| **Record count** | Always visible: "Showing 25 of 25 records" — never hide this |
| **Time range** | Show the time span covered by the log file |
| **Summary stats** | Top of page: total requests, unique IPs (count only), cache hit ratio, error rate, avg TTFB |
| **Multiple files** | Load multiple .gz files from the same time period — merge and sort by timestamp |

### Handling the .gz Problem

The `.gz` files from the Kinesis Firehose pipeline are **not actually gzipped** — they're plaintext with a `.gz` extension (a pipeline configuration bug). The tool must handle both cases gracefully:

```
User drops a file
       ↓
Check: does it start with gzip magic bytes (0x1f 0x8b)?
       ├── Yes → decompress, then parse
       └── No  → it's already plaintext, parse directly
```

The tool should never fail on a `.gz` file regardless of whether it's actually compressed. Display a note when a mismatched extension is detected:

```
Drop a CloudFront log file here
  Supports: .gz (gzipped or plaintext), .txt, .log, .csv
  
  ℹ️ Note: Some CloudFront/Kinesis pipelines produce .gz files that 
  are actually plaintext (a common misconfiguration). This tool 
  handles both transparently.
```

**DevOps action (separate from this tool):** Fix the Firehose delivery stream to either enable gzip compression or stop naming files `.gz`. The current state wastes storage (uncompressed) and confuses every tool that trusts the extension.

## UX Lessons from This Experience

### Lesson 1: "LIMIT 5" with No Indication is Dangerous

The S3 Select default query silently truncates to 5 records with no "showing 5 of N" indicator. In a debugging scenario, this can lead to wrong conclusions. Our tools must **always** show record counts and never silently truncate.

**Principle for all SG/Send tools:** Always show "Showing X of Y records." If truncation happens, make it loud and obvious.

### Lesson 2: File Extensions Can Lie

The `.gz` extension means "this file is gzip-compressed" — except when it isn't. The Kinesis Firehose pipeline writes plaintext files with `.gz` extensions, which causes every tool that trusts the extension to fail: macOS Archive Utility, `gunzip`, and any script that attempts decompression. Only `cat` — which ignores the extension — reveals the truth.

**Principle:** Never trust file extensions. Always check magic bytes / file signatures. SG/Send tools should detect actual content type regardless of what the filename claims. And our own pipelines must not produce misleading extensions.

### Lesson 3: AWS Console UX Is Enterprise-Grade Pain

The path from "I want to see my CloudFront logs" to actually reading a log entry requires: navigate to S3 → find the bucket → navigate the date hierarchy → pick a file → choose between download (broken on macOS) or S3 Select (truncated, unlabelled columns, URL-encoded values). This is 6-8 clicks and 2-3 minutes for a power user, with two failure modes.

**Principle:** Our admin tools should get from intent to answer in 2 clicks. "Show me the logs" → logs are on screen. The data room viewer, the log viewer, the token manager — all should follow this pattern.

## AppSec Note

This tool processes CloudFront logs which contain: request paths (which may include transfer IDs, token names, presigned URL parameters), user agents, edge locations, and timestamps. Client IPs are deliberately excluded from the CloudFront logging configuration.

An independent security review (ChatGPT, 28 Feb 2026) confirmed that the log contents constitute **low-risk operational metadata** — not a security incident even if published publicly. Transfer IDs are opaque without the browser-held decryption key. Tokens are intentionally shareable and revocable. No credentials, encryption keys, or personal data are present. The review is stored at: `team/humans/dinis_cruz/briefs/02/28/security-review/chat-gpt__review-of-data-stored-in-s3__from-cloudfront-logs.md`

Nonetheless, the tool must:
- Process everything client-side (no server upload)
- Not persist any data (no localStorage, no sessionStorage)
- Include a clear notice: "Your log data is processed entirely in your browser. Nothing is sent to our servers."
- CSP headers must prevent any exfiltration even if the page were compromised
