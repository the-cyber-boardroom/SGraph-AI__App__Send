## Pricing Model

### Credit-Based System

SG/Send uses a credit-based pricing model built on the existing access token infrastructure:

| Parameter | Value |
|---|---|
| **Price per file** | £0.01 (one penny) |
| **Starter credit pack** | £5.00 |
| **Files per £5 pack** | 500 files |
| **Credit expiry** | 3 months from purchase |
| **Currency** | GBP (British Pounds) |

This pricing is designed to be a good deal (one penny per encrypted file transfer is clearly valuable) while ensuring users realistically run out of credits within the 3-month window. This is deliberate: **the most important question we need to answer is "will they buy more?"** The answer to that question determines whether we have a viable business with the features currently shipped.

**The strategic framing:**

- If users run out and buy more → we have product-market fit signal, even at this early stage
- If users run out and don't buy more → we learn why (is it price? features? frequency of use? friction in the top-up flow?) and we've still built the full commercial workflow
- If users don't run out → the pricing is too generous and we're not learning fast enough

Either way, we learn. And we will have implemented the entire commercial provisioning pipeline (Stripe → token → delivery → consumption → expiry → re-purchase) which is needed regardless.

**Important context:** This initial launch is happening with only our lowest-value service offering live — the **symmetric key file send**. The PKI mode, key management, and data vaults are not yet live. If we can get paying users and re-purchases with just the symmetric mode, the higher-value features (when they ship) have a strong foundation to build on.

## Payment Flow: Stripe Integration

### Current Flow (MVP — Partially Manual)

```
Customer                    Stripe                     Human/Sherpa
   │                          │                            │
   │  Clicks "Buy Credits"    │                            │
   │─────────────────────────>│                            │
   │                          │                            │
   │  Stripe Checkout page    │                            │
   │  (£5 payment)            │                            │
   │─────────────────────────>│                            │
   │                          │                            │
   │  Payment confirmed       │  Webhook / notification    │
   │<─────────────────────────│───────────────────────────>│
   │                          │                            │
   │                          │     Creates access token   │
   │                          │     (via admin panel)      │
   │                          │                            │
   │  Email: "Your access     │                            │
   │  token is: {token}"      │<───────────────────────────│
   │<─────────────────────────│                            │
   │                          │                            │
   │  Uses token on           │                            │
   │  send.sgraph.ai          │                            │
```

**This is intentionally semi-manual for the MVP.** The payment is automated (Stripe handles it), but token creation is manual (human creates it via admin panel and sends via email). This is fine for the first 100 customers. Automate later.

### Target Flow (Automated)

```
Customer → Stripe Checkout → Stripe webhook → 
  FastAPI endpoint → auto-create access token → 
  email token to customer → customer uses token
```

This automation is a Phase 2 goal. For Phase 1, the manual step is acceptable and gives us visibility into every customer.

### Billing Research

## Customer Acquisition Strategy: The £500 Experiment

### The Approach

Rather than asking friends to buy the product (which feels transactional and awkward), the strategy is:

**"Here's a £5 Amazon voucher as a thank you. Would you mind trying SG/Send and giving me feedback?"**

This reframes the ask from "buy my product" to "help me test this and I'll thank you for your time." The £5 voucher covers the cost of the £5 credit pack — the customer is net-zero financially but has experienced the full purchase flow.

### Budget

| Item | Amount |
|---|---|
| **Total budget** | £500 |
| **Voucher value** | £5 each |
| **Number of vouchers** | 100 |
| **Target conversions** | 100 people go through the full flow |

### The Funnel

```
100 people reached out to (friends, contacts, friendlies)
  ↓
~80 accept the voucher and agree to try
  ↓
~60 actually go through Stripe and buy credits
  ↓
~40 actually send a file using the service
  ↓
~20 give meaningful feedback
  ↓
~5-10 become organic users who continue using it
```

Even if only 5-10 become real users, we get:
- 60+ people through the full payment flow (testing Stripe integration)
- 40+ real file transfers (testing the product end-to-end)
- 20+ feedback data points (product insights)
- 100 people who now know what SG/Send is (awareness)
- Revenue data in Stripe (£300-500 in transactions, even if funded by vouchers)

### Voucher Logistics

**Action for Ambassador:** Research the best way to distribute £5 vouchers:

| Option | Pros | Cons |
|---|---|---|
| **Amazon vouchers** | Universal, everyone uses Amazon, easy to redeem | Need to figure out bulk purchase, Amazon-specific |
| **Visa/Mastercard gift cards** | Works for Stripe payment directly | Higher fees, more friction |
| **PayPal transfer** | Direct, simple | Not everyone has PayPal |
| **Bank transfer** | Most direct | Too personal, friction |
| **SG/Send credits (free)** | Skip the payment step entirely | Doesn't test the billing flow |

**Recommendation:** Amazon vouchers are the simplest. The goal is to compensate people for their time, not to make the payment flow frictionless. They redeem the Amazon voucher separately, then spend £5 on SG/Send through Stripe. Two separate transactions — that's fine.

**Action:** Figure out how to bulk-purchase 100 × £5 Amazon vouchers. Amazon Business or Amazon Incentives may have a programme for this.

### Outreach Message

The Ambassador should draft an outreach message (multiple variants for different channels):

**Tone:** Personal, not salesy. "I've been building something and I'd love your honest feedback."

**Key elements:**
- Brief explanation of what SG/Send does (one sentence)
- "Would you try it and tell me what you think?"
- "Here's a £5 Amazon voucher as a thank you for your time"
- "It takes about 5 minutes to try"
- Link to the pricing page
- No pressure, no follow-up guilt

**Channels:** Direct message (WhatsApp, LinkedIn, email), small group posts, 1:1 conversations.

## What the Customer Experiences

### End-to-End Flow

```
1. Receives personal message: "I've been building SG/Send — 
   would you try it? Here's a £5 Amazon voucher for your time."

2. Visits sgraph.ai → sees the pricing page

3. Clicks "Buy Credits" (£5) → Stripe checkout
   → Pays with card / Apple Pay / Google Pay
   → Gets Stripe receipt

4. Receives email from Sherpa:
   "Welcome! Your access token is: {token}
    Here's how to send your first encrypted file: [link]"

5. Visits send.sgraph.ai → enters token → uploads a file
   → File encrypted in browser → gets a download link

6. Shares the link with someone → they download and decrypt

7. (Optional) Provides feedback via reply email or form
```

### What We Learn

For each customer in the funnel, we can measure:

| Stage | What We Learn |
|---|---|
| **Outreach** | Response rate, which channels work, which message variants work |
| **Pricing page** | Do they understand the tiers? Do they click "Buy Credits"? |
| **Stripe checkout** | Conversion rate, drop-off points, payment method preferences |
| **Token delivery** | Time from payment to token delivery (manual for now — measure it) |
| **First file send** | Do they complete it? Where do they get stuck? How long does it take? |
| **Sharing** | Do they share the download link? Does the recipient successfully decrypt? |
| **Feedback** | What do they like? What's confusing? What's missing? |

## Cost Analysis

### Worst-Case Scenario Check

| Parameter | Value |
|---|---|
| Price per file | £0.01 |
| Files per £5 pack | 500 |
| Max file size (current) | 100 MB (assumed current limit) |
| Worst case: 500 × 100 MB | 50 GB of S3 storage + transfer |
| S3 storage cost (50 GB / 3 months) | ~£0.35 |
| S3 + CloudFront egress (50 GB) | ~£4.00 |
| Lambda invocations (500 × 3 calls each) | ~£0.005 |
| **Total AWS cost for worst case** | **~£4.35** |
| **Revenue** | **£5** |

**At £0.01/file with 500 files, even the worst case is marginally profitable.** If every user maxes out at 100MB per file (unlikely), the AWS cost is ~£4.35 against £5 revenue. Realistic usage (average file size much smaller than 100MB) will be comfortably profitable per pack.

The Accountant should monitor actual file sizes and transfer volumes to validate this model as real usage comes in.

### Marketing Budget

| Item | Cost |
|---|---|
| 100 × £5 Amazon vouchers | £500 |
| Stripe fees (2.9% + 20p per transaction, ~60 transactions) | ~£21 |
| **Total marketing spend** | **~£521** |

For £521 we get: 100 people through the awareness funnel, 60+ through the payment flow, 40+ real product users, and 20+ feedback data points. This is exceptionally cheap customer acquisition for a B2B/prosumer product.
