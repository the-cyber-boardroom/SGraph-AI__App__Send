## Principle

**If we can't do multi-cloud for a static website, we can't do it for anything else.**

Static sites are the simplest deployment unit: HTML files in a storage bucket behind a CDN. No compute, no databases, no state. If we prove multi-cloud, multi-region deployment here — with all the automation, monitoring, and failover — the patterns scale to serverless compute, data stores, and encrypted environments.

Our first multi-cloud, multi-region deployment should be the static websites.

---

## Part 1: Multi-Region Within AWS

### Current State

Everything runs in a single AWS region. If that region has an outage, everything is down.

### Target State

The static site S3 buckets are replicated across at least two AWS regions:

```
AWS:
├── eu-west-1  (Ireland — primary)
│   └── {account}--static-sgraph-ai--eu-west-1
│
├── us-east-1  (Virginia — secondary)
│   └── {account}--static-sgraph-ai--us-east-1
│
└── ap-southeast-1  (Singapore — future, for APAC latency)
    └── {account}--static-sgraph-ai--ap-southeast-1
```

CloudFront already handles edge caching globally, but the origin bucket is a single point of failure. Multi-region origins with failover eliminate this.

## Part 2: Multi-Cloud Deployment

### Target Architecture

The same static content deployed to AWS, GCP, Azure, Cloudflare, and potentially cheaper or EU-sovereign storage providers.

```
AWS:        S3 + CloudFront               → sgraph.ai (primary)
GCP:        Cloud Storage + Cloud CDN      → gcp.sgraph.ai
Azure:      Blob Storage + Azure CDN       → azure.sgraph.ai
Cloudflare: R2 + Cloudflare CDN/Pages      → cloudflare.sgraph.ai
EU:         EU-sovereign provider (TBD)    → eu.sgraph.ai
Cheap:      Backblaze B2 / Wasabi          → backup / archive
```

### Cloudflare as a Strategic Deployment Target

Cloudflare is explicitly included as a first-class deployment target — not just a CDN layer, but a full hosting partner:

```
cloudflare.sgraph.ai                         ← Cloudflare-hosted main site
eu-west-1.cloudflare.send.sgraph.ai          ← Cloudflare-hosted send app (EU)
```

**Why Cloudflare specifically:**
- Massive innovation in edge computing, R2 (S3-compatible, zero egress), Workers, Pages
- Free egress on R2 — significant cost advantage for static content
- Strong security reputation — DDoS protection, WAF, bot management included
- Potential partner relationship — their developer platform aligns with our goals
- Cloudflare Pages can deploy static sites directly from git — an alternative CI path worth evaluating

**DevOps action:** Set up Cloudflare R2 + Pages as the first non-AWS deployment target. This tests the cloud-agnostic storage abstraction with a provider that has an S3-compatible API (R2) but a different CDN and DNS model.

### URL Structure for Demonstrating Multi-Cloud

To show — visibly — that SG/Send runs across multiple clouds:

```
Primary (production):
  https://sgraph.ai/                        ← served from whichever cloud is primary

Cloud-specific (demonstration/verification):
  https://eu-west-1.aws.sgraph.ai/          ← AWS EU West 1
  https://us-east-1.aws.sgraph.ai/          ← AWS US East 1
  https://europe-west1.gcp.sgraph.ai/       ← GCP Europe West 1
  https://westeurope.azure.sgraph.ai/       ← Azure West Europe
  https://cloudflare.sgraph.ai/             ← Cloudflare (global edge)
  https://eu-west-1.cloudflare.send.sgraph.ai/ ← Cloudflare send app (EU)
  https://eu.sgraph.ai/                     ← EU-sovereign provider (TBD)
```

These cloud-specific URLs serve the same content. They exist to:
- **Demonstrate** multi-cloud capability to customers ("look — same site, three clouds")
- **Verify** that deployments are in sync
- **Test** failover by hitting specific endpoints
- **Measure** latency from different origins

### Deployment Pipeline

```
GitHub release triggered
    ↓
Python CI script builds static site
    ↓
Deploy in parallel:
    ├── AWS S3 (eu-west-1)       ✅
    ├── AWS S3 (us-east-1)       ✅
    ├── GCP Cloud Storage         ✅
    ├── Azure Blob Storage        ✅
    ├── Cloudflare R2 / Pages     ✅
    ├── EU-sovereign provider     ✅ (when selected)
    └── Backblaze B2 / Wasabi     ✅
    ↓
Verification: curl each endpoint, compare checksums
    ↓
Update DNS / CDN if all pass
```

### Cloud-Agnostic Storage Abstraction

The Python deployment scripts should use an abstraction layer for storage:

```python
class StaticSiteDeployer:
    """Deploy static site to any cloud storage provider."""
    
    def deploy(self, site_path: str, version: str, provider: StorageProvider):
        provider.upload_folder(site_path, f"websites/{site_name}/releases/{version}/")
        provider.copy_folder(
            f"websites/{site_name}/releases/{version}/",
            f"websites/{site_name}/latest/"
        )

# Implementations
class S3Provider(StorageProvider): ...
class GCSProvider(StorageProvider): ...
class AzureBlobProvider(StorageProvider): ...
class BackblazeB2Provider(StorageProvider): ...
```

Same interface, different backends. The CI pipeline deploys to all configured providers in parallel.

## Part 3: Cheap Storage Providers

There are storage providers that are an order of magnitude cheaper than S3 for static content hosting. These are worth exploring as additional deployment targets:

| Provider | Cost (storage) | Cost (egress) | Notes |
|---|---|---|---|
| **AWS S3** | ~$0.023/GB/month | ~$0.09/GB | Baseline |
| **Backblaze B2** | ~$0.006/GB/month | Free with Cloudflare | 4x cheaper storage, free egress via Cloudflare |
| **Wasabi** | ~$0.007/GB/month | Free | No egress charges, minimum 1TB |
| **Cloudflare R2** | ~$0.015/GB/month | Free | Zero egress, S3-compatible API |
| **Hetzner Storage Box** | ~€3.50/TB/month | Included | European, very cheap |

**Recommendation:** Add Cloudflare R2 or Backblaze B2 as a secondary deployment target. S3-compatible APIs make integration straightforward. The cost savings are significant for static content, and the additional provider increases resilience.

## Part 4: Resilience and BCP

### Data Distribution as Security

Every release is stored in multiple places:

```
Release v0.7.6 exists in:
  ✅ GitHub (source code)
  ✅ AWS S3 eu-west-1 (static site)
  ✅ AWS S3 us-east-1 (static site)
  ✅ GCP Cloud Storage (static site)
  ✅ Azure Blob Storage (static site)
  ✅ Backblaze B2 (archive)
  ✅ CI artifacts in S3 (build logs, git diffs)
```

An attacker would need to compromise ALL of these simultaneously to destroy a release. Even a complete AWS account compromise — all keys, all access — leaves copies in GCP, Azure, and Backblaze.

### Immutability and Delete Protection

For archive copies, enable delete protection:

| Provider | Mechanism |
|---|---|
| **AWS S3** | Object Lock (WORM — Write Once Read Many), versioning with MFA delete |
| **GCP** | Retention policies, bucket lock |
| **Azure** | Immutable blob storage, legal hold |
| **Backblaze B2** | Object Lock (S3-compatible) |

With Object Lock enabled, even the account owner cannot delete objects until the retention period expires. A ransomware attack that compromises all credentials cannot destroy the archived data.

**Recommendation:** Enable Object Lock on at least one storage provider per release. 30-day retention minimum. This guarantees that even a total compromise cannot destroy recent releases.

### GitHub as a Single Point of Failure

Currently, GitHub holds:
- Source code
- CI pipeline definitions
- Git history (commits, diffs, blame)
- Issues and PR discussions
- Release artifacts

If GitHub goes down or our account is compromised:

| What we lose | Mitigation |
|---|---|
| Source code | Git mirrors to GCP/Azure/self-hosted (standard) |
| CI pipelines | Python scripts in the repo — just the YAML triggers are GitHub-specific |
| Git history | Mirrored with full history |
| CI artifacts (build logs, diffs) | Stored in S3 per the v0.7.6 deployment architecture |
| Issues/PRs | IssuesFS — the filesystem-based issue tracker already exists in the repo |
| Release artifacts | Multi-cloud deployment means releases exist everywhere |

**Action:** Set up automated git mirroring to at least one non-GitHub provider. The CI artifact storage in S3 (from the static site deployment brief) handles the rest.

## Part 5: AppSec — Total Key Compromise Resilience

### Threat Scenario

**What happens if the human's entire desktop is compromised?** All SSH keys, all AWS credentials, all API tokens, all browser sessions — everything is in the attacker's hands.

The AppSec team should map:

| Question | Analysis Required |
|---|---|
| What can the attacker access? | Map every credential on the desktop → what it unlocks |
| What can the attacker delete? | Which storage providers allow deletion with these credentials? |
| What can the attacker modify? | Can they push malicious code? Modify deployed sites? |
| What can't the attacker touch? | Object-locked archives, credentials they don't have (other cloud providers with separate auth) |
| How do we detect the compromise? | Anomalous API calls, unusual deployments, credential usage from unexpected IPs |
| How do we recover? | Revoke all credentials, deploy from a clean machine using archived releases |
| What is the blast radius? | One region? All of AWS? All clouds? |
| Can we survive it? | Is there a path from "everything compromised" to "back in production" using only archived, immutable data? |

### The Static Site as BCP Test

The static website multi-cloud deployment is the simplest BCP test:

```
Scenario: All AWS credentials compromised. AWS account locked.

Can we:
  1. ✅ Access the static site from GCP?
  2. ✅ Access the static site from Azure?
  3. ✅ Point the DNS to a non-AWS CDN?
  4. ✅ Recover the source code from a git mirror?
  5. ✅ Rebuild and deploy from a clean machine using only non-AWS providers?
  6. ✅ Restore the full release from an Object-Locked archive?

If yes → our BCP works for static content.
Next → extend the same resilience to serverless compute and data stores.
```
