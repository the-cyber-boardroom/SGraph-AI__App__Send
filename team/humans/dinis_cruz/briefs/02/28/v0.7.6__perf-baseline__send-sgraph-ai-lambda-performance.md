# Performance Baseline: send.sgraph.ai — Pre-Static/CDN Optimization

**version** v0.7.6  
**date** 28 Feb 2026  
**from** Human (project lead)  
**to** DevOps, Developer, Architect  
**type** Performance baseline — pre-optimization reference  

---

## Context

These measurements were taken on 28 Feb 2026 from the production `send.sgraph.ai` site, specifically the page at `/send/v0/v0.1/v0.1.8/index.html`.

**Critical context — this is the BEFORE state:**

1. **No static/CDN publishing yet.** Every request — HTML, CSS, JS, fonts — is served by the FastAPI Lambda function. None of these resources are on S3/CloudFront or cdn.sgraph.ai (which doesn't exist yet).
2. **All 18 requests hit Lambda.** The FastAPI server processes every resource request, including static assets that should be served from a CDN.
3. **Lambda dependency optimisation not yet implemented.** The cold start includes loading all FastAPI dependencies, which we have a full plan to refactor but haven't executed yet.
4. **Two external dependencies still present.** Google Fonts (`fonts.googleapis.com`) and Google Static Fonts (`fonts.gstatic.com`) are loaded — these will be eliminated per the zero-external-dependencies requirement.

These numbers are the baseline against which all future optimisations will be measured.

| 13 | send-upload.js (v0.1.5) | 169 ms | 84 ms | Fast |
| 14 | send-upload.js (v0.1.6) | 108 ms | 229 ms | Fast |
| 15 | send-upload.js (v0.1.8) | 119 ms | 227 ms | Fast |

### Key Observations

- **9+ second response times for most resources.** The first ~10 requests all queue behind the Lambda cold start. Once the Lambda is initialized, subsequent requests in the same page load are fast (100-200ms).
- **The waterfall shows a clear "wall" at ~9 seconds** — everything blocks on the Lambda cold start, then resources flood through.
- **index.html takes 9.17 seconds.** The HTML document — the very first thing the browser needs — is blocked by the cold start. Nothing can render until this arrives.
- **The UI is completely unusable during cold start.** 18.5 seconds to page load. A user would have long abandoned the page.
- **The pattern is clear:** ~9s cold start penalty, then ~170ms per resource once warm. This is entirely the Lambda initialization time loading all FastAPI dependencies.

## What This Means for the v0.7.6 Architecture

### Impact of Static Site / CDN Publishing

When we move static assets to S3/CloudFront (per the v0.7.6 static site deployment architecture brief):

| Resource Type | Current (Lambda) | Target (CDN) | Expected Improvement |
|---|---|---|---|
| HTML | 9.17s (cold) / 170ms (warm) | <50ms | **No cold start dependency for static content** |
| CSS | 9.03s (cold) / 230ms (warm) | <50ms | Same |
| JS | 9.03s (cold) / 170ms (warm) | <50ms | Same |
| Fonts | External (Google) | cdn.sgraph.ai, <50ms | **Eliminates external dependency** |
| API calls | 673ms (cold) / 542ms (warm) | Same (still Lambda) | **Only API calls remain on Lambda** |

**Expected result after CDN migration:**
- **Cold start:** ~1-2s (only API calls wait for Lambda, static assets load instantly from CDN)
- **Warm:** ~500ms (static from CDN + API calls from warm Lambda)
- **The UI renders immediately** — HTML/CSS/JS from CDN, API calls happen after initial render

### Impact of Lambda Dependency Optimisation

The 9-second cold start is the Lambda loading all FastAPI dependencies. The planned optimisation (already briefed, not yet implemented) would:

- Strip unnecessary dependencies from the Lambda package
- Use Lambda layers for shared dependencies
- Implement provisioned concurrency for critical paths
- Target: cold start <2 seconds

### Combined Improvement Projection

| Scenario | Current | After CDN | After CDN + Lambda Opt |
|---|---|---|---|
| Cold start, page load | **18.50 s** | ~2-3 s | **<1.5 s** |
| Warm, page load | 599 ms | ~300 ms | **~300 ms** |
| First request after deploy | 3.68 s | ~1 s | **<1 s** |
