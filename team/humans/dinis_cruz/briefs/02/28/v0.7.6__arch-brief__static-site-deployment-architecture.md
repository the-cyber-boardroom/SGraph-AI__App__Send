## Principle

Every SG/Send website — sgraph.ai, send.sgraph.ai, admin.sgraph.ai, vault.sgraph.ai, tools.sgraph.ai, docs, library, and any future sites — runs as static content served from S3 behind CloudFront. No Lambda for page serving. No server-side rendering. Static HTML, CSS, JS, and assets.

This is not a limitation. It is a design choice that simplifies deployment, enables versioning, makes offline packaging trivial, and prepares the infrastructure for multi-cloud distribution.

---

### Cache TTL: Keep It Low For Now

While we're iterating rapidly, set CloudFront cache TTL low across all static assets:

```
Default TTL:    300 seconds  (5 minutes)
Max TTL:        3600 seconds (1 hour)
Min TTL:        0 seconds
```

This avoids stale content during frequent deployments. We can increase TTLs later when release cadence stabilises. cdn.sgraph.ai assets (fonts, vendor libs) can have longer TTLs since they change rarely — but still start conservative.

### Logging: CloudFront Request Logs and S3 Data Access Logs

**DevOps action required:** Configure full logging for all request and data access activity across every deployment.

**AWS — what to enable:**

| Log Type | What It Captures | Where to Store |
|---|---|---|
| **CloudFront access logs** | Every HTTP request: IP, path, status, bytes, user-agent, referer | Dedicated logging S3 bucket (not the content bucket) |
| **S3 server access logs** | Every S3 API call: GetObject, PutObject, ListBucket, etc. | Same logging bucket, different prefix |
| **S3 CloudTrail data events** | API-level audit: who accessed what, from which IAM identity | CloudTrail → S3 or CloudWatch |

**Multi-cloud logging — DevOps to map the equivalent for each provider:**

| Provider | Request logs (CDN) | Storage access logs | Audit trail |
|---|---|---|---|
| **AWS** | CloudFront access logs | S3 server access logs | CloudTrail data events |
| **GCP** | Cloud CDN logs (via Cloud Logging) | GCS access logs / audit logs | Cloud Audit Logs |
| **Azure** | Azure CDN diagnostic logs | Blob Storage analytics logs | Azure Monitor / Activity Log |
| **Cloudflare** | Logpush (Enterprise) or Analytics API | R2 — TBD (newer service) | Audit Log API |

**DevOps deliverable:** A comparison matrix of logging capabilities, performance impact, cost, and retention options across all four providers. This will be valuable as we scale multi-cloud — we need to understand where we have visibility gaps.

### DNS Infrastructure

**Current state:**

```
Domain registrar:  Network Solutions (networksolutions.com)
                       ↓
sgraph.ai NS records → AWS Route 53 Hosted Zone
                       ↓
Route 53 manages:
  ├── sgraph.ai
  ├── dev.sgraph.ai
  ├── admin.dev.sgraph.ai
  ├── send.sgraph.ai
  └── (TLS certificates via ACM, validated against this hosted zone)
```

**What needs mapping:**

As we add multi-cloud and multi-region deployments, DNS becomes a control plane. The team needs to document:

| Question | Why It Matters |
|---|---|
| Who controls the registrar (Network Solutions)? | Single point of failure for the entire domain. If compromised, all DNS can be redirected. |
| Who controls the Route 53 hosted zone? | Controls all subdomain routing and TLS cert validation. |
| Can we add NS delegations for cloud-specific subdomains? | e.g., `cloudflare.sgraph.ai` delegated to Cloudflare's DNS, `gcp.sgraph.ai` to GCP Cloud DNS |
| What's the TTL on the NS records at the registrar? | Determines how fast we can failover if Route 53 goes down |
| Do we have DNSSEC enabled? | Prevents DNS spoofing — important for a security-focused product |
| What's the recovery plan if the registrar account is compromised? | Transfer lock, registry lock, 2FA status |

**Architecture direction:** Route 53 remains the primary DNS authority. Cloud-specific subdomains (cloudflare.sgraph.ai, gcp.sgraph.ai) can use NS delegation to the respective cloud's DNS for their subtree. This keeps the root zone in our control while letting each cloud manage its own endpoints.

## Core Architecture

### Serving Stack

```
User → CloudFront (CDN + TLS + domain mapping)
         ↓
       S3 Bucket (private — not publicly accessible)
         ↓
       Specific folder per site, per version
```

CloudFront serves specific folders from S3. The S3 bucket is never public — CloudFront has an Origin Access Identity (or Origin Access Control) granting it read access. This is the only access path.

### Relative Paths Only

All links, assets, scripts, and stylesheets use relative paths. No absolute URLs. No hardcoded domains. A website must work correctly regardless of the base path it's served from — whether that's `/`, `/latest/`, `/releases/v0.7.6/`, or an offline folder on someone's desktop.

```
✅  <link href="./styles/main.css">
✅  <script src="./js/app.js">
✅  <a href="./use-cases/passwords/">
✅  <img src="./assets/logo.svg">

❌  <link href="https://sgraph.ai/styles/main.css">
❌  <script src="/js/app.js">
❌  <a href="/use-cases/passwords/">
```

This is non-negotiable. Absolute paths break versioned deployments, offline packaging, and multi-domain hosting.

### ZERO External Dependencies

**CRITICAL:** Self-contained and offline deployments must have absolutely zero external dependencies. Nothing loaded from Google, Cloudflare CDN, unpkg, cdnjs, Google Fonts, or any other third-party origin. Every font, every JavaScript library, every CSS framework, every icon set — everything the page needs to render and function — must exist inside the deployment folder.

```
❌  <link href="https://fonts.googleapis.com/css2?family=Inter">
❌  <script src="https://cdnjs.cloudflare.com/ajax/libs/lodash/4.17.21/lodash.min.js">
❌  <link href="https://cdn.jsdelivr.net/npm/bootstrap@5/dist/css/bootstrap.min.css">

✅  <link href="./fonts/inter/inter.css">
✅  <script src="./vendor/lodash.min.js">
✅  <link href="./vendor/bootstrap.min.css">
```

**Known issue:** The current send.sgraph.ai loads fonts from Google Fonts. This breaks offline deployments and creates a third-party dependency. The Dev and Architect must audit all current sites and eliminate every external resource load.

**Action required — External Dependency Audit:**

| Site | What to audit |
|---|---|
| send.sgraph.ai | Google Fonts (confirmed), any CDN-loaded JS/CSS |
| admin.sgraph.ai | Same |
| All future sites | Must pass zero-external-dependency check before first deployment |

Using common, well-known fonts (Inter, system fonts, etc.) is fine and encouraged for cross-browser compatibility — but they must be downloaded, stored in our deployment, and served from our own infrastructure.

### cdn.sgraph.ai — Our Own CDN Origin

To manage shared, stable, commonly-used artifacts across all SG/Send sites, introduce **cdn.sgraph.ai** as a dedicated domain:

```
cdn.sgraph.ai/
├── fonts/
│   ├── inter/
│   │   ├── inter-regular.woff2
│   │   ├── inter-bold.woff2
│   │   └── inter.css
│   └── system-fonts.css
│
├── js/
│   ├── sg-send.js              ← the SG/Send helper JS file (referenced in multiple briefs)
│   ├── sg-send.min.js
│   └── sg-decrypt.js           ← client-side decryption library
│
├── css/
│   ├── sg-send-base.css        ← shared base styles
│   └── sg-send-components.css
│
├── vendor/
│   ├── lodash.min.js           ← vendored third-party libraries
│   └── papaparse.min.js
│
├── images/
│   ├── logo.svg
│   ├── favicon.ico
│   └── brand/
│
└── releases/                   ← versioned snapshots of the full CDN
    ├── v1.0.0/
    └── v1.1.0/
```

**cdn.sgraph.ai serves two purposes:**

1. **Online mode:** All SG/Send sites load shared resources from `cdn.sgraph.ai` — a single, fast, cacheable origin. This avoids duplicating fonts/JS across every site deployment while keeping everything on our own infrastructure.

2. **Offline mode / self-contained packaging:** The CDN contents are copied into the deployment folder. The relative paths switch from `cdn.sgraph.ai/fonts/inter/` to `./cdn/fonts/inter/`. Because it's our CDN with our versioning, this is a simple copy operation — no third-party license or availability concerns.

**Future migration path:** cdn.sgraph.ai starts as an S3/CloudFront deployment (same infrastructure as the websites). If traffic or performance demands it later, it can be moved to a dedicated CDN provider (Cloudflare, Fastly, etc.) by simply changing the DNS — the content and folder structure don't change. Starting with our own domain now means the migration is seamless.

**Relationship to the S3 bucket structure:**

```
S3 Bucket: {account}--static-sgraph-ai--{region}
│
├── websites/           ← per-site deployments (sgraph-ai, send, docs, etc.)
├── cdn/                ← cdn.sgraph.ai content (fonts, JS, CSS, vendor, images)
│   ├── latest/
│   └── releases/
├── packages/           ← pypi/npm mirrors
└── ci/                 ← build artifacts
```

The `cdn/` folder in the existing S3 structure IS cdn.sgraph.ai. CloudFront maps `cdn.sgraph.ai` to the `cdn/latest/` folder. Same infrastructure, different domain, different caching and versioning cadence.

## Versioned Deployment Model

Every deployment publishes to two locations:

```
S3 Bucket: {account-id}--static-sgraph-ai--{region}
│
├── websites/
│   ├── sgraph-ai/
│   │   ├── latest/              ← always points to the most recent release
│   │   │   ├── index.html
│   │   │   ├── styles/
│   │   │   ├── js/
│   │   │   └── assets/
│   │   └── releases/
│   │       ├── v0.7.4/          ← immutable snapshot
│   │       ├── v0.7.5/
│   │       └── v0.7.6/          ← current release
│   │
│   ├── send-sgraph-ai/
│   │   ├── latest/
│   │   └── releases/
│   │       └── v0.7.6/
│   │
│   ├── docs-sgraph-ai/
│   │   ├── latest/
│   │   └── releases/
│   │
│   └── vault-sgraph-ai/
│       ├── latest/
│       └── releases/
│
├── packages/                     ← future: pypi/npm mirror
│   ├── pypi/
│   └── npm/
│
├── cdn/                          ← shared assets, design system
│   └── assets/
│       ├── latest/
│       └── releases/
│
└── ci/                           ← CI pipeline artifacts (temporal, by date)
    └── 2026/
        └── 02/
            └── 28/
                ├── sgraph-ai--v0.7.6--build.log
                ├── sgraph-ai--v0.7.6--git-diff.txt
                └── sgraph-ai--v0.7.6--git-commit.txt
```

### Key Properties

| Property | Detail |
|---|---|
| **latest/** always exists | Updated on every release. This is what CloudFront serves to the public domain. |
| **releases/{version}/** is immutable | Once published, never modified. A release is a complete snapshot. |
| **Version = release tag** | Not a date, not a git hash. The release tag (v0.7.6) controls the folder name. |
| **CI artifacts are temporal** | Stored by date, not by version. Build logs, git diffs, git commit messages. |
| **Rollback = point latest to a previous release** | No rebuild needed. Copy or symlink the previous version folder to latest/. |

### Why This Matters

A specific release of SG/Send now has a tight lock on every artifact: the code, the Lambda deployment, the static website for that version, the docs for that version, the dependency versions, the CI output. Everything is versioned together. When packaging an offline deployment for a customer, copy the version folder — it's self-contained.

## Bucket Naming Convention

```
{aws-account-id}--static-sgraph-ai--{region}
```

Examples:
- `123456789012--static-sgraph-ai--eu-west-1`
- `123456789012--static-sgraph-ai--us-east-1`

This convention prepares for multi-region deployment (each region gets its own bucket) and is unambiguous about ownership and purpose.

## CI Pipeline: Python First, GitHub Actions Minimal

### Principle

The CI pipeline logic lives in Python, not in GitHub Actions YAML. GitHub Actions is the trigger and executor — it calls Python scripts from the command line. The Python scripts contain all the logic.

```
GitHub Actions workflow:

  steps:
    - checkout
    - setup-python
    - run: python scripts/build_static_site.py --site sgraph-ai
    - run: python scripts/deploy_to_s3.py --site sgraph-ai --version ${{ github.ref_name }}
    - run: python scripts/store_ci_artifacts.py --version ${{ github.ref_name }}
```

### Why

| GitHub Actions YAML | Python scripts |
|---|---|
| Hard to test locally | Easy to test locally |
| Hard to debug | Standard debugging tools |
| No IDE support | Full IDE support, linting, type checking |
| Vendor lock-in | Portable — runs anywhere |
| Complex conditional logic is painful | Natural language constructs |
| Sharing logic between workflows requires complex action composition | Import a module |

If a step requires invoking a CLI tool (like `aws s3 sync`) because it's faster than a Python equivalent, invoke it via `subprocess` from Python and capture the result. The Python script remains the orchestrator.

## Offline Packaging

A deployment for a customer who runs SG/Send in an air-gapped or offline environment is a copy of the version folder:

```
Customer package for "MedCorp" running v0.7.6:

medcorp-deployment-v0.7.6/
├── websites/
│   ├── sgraph-ai/releases/v0.7.6/       ← main site
│   ├── send-sgraph-ai/releases/v0.7.6/  ← send app
│   ├── docs-sgraph-ai/releases/v0.7.6/  ← documentation
│   └── vault-sgraph-ai/releases/v0.7.6/ ← vault UI
├── cdn/releases/v1.0.0/                  ← shared assets (fonts, sg-send.js, vendor libs)
│   ├── fonts/
│   ├── js/
│   ├── css/
│   ├── vendor/
│   └── images/
├── packages/
│   ├── pypi/                              ← Python dependencies
│   └── npm/                               ← JS dependencies
├── keys/
│   ├── public/                            ← public keys for this deployment
│   └── private/                           ← private keys (customer-managed)
└── config/
    └── deployment.json                    ← region, domain, feature flags
```

Because every path is relative, every version is a complete snapshot, and every dependency (including fonts, JS libraries, and vendor code) is included — this package runs from any web server with zero internet connectivity. No Google Fonts call, no cdnjs fetch, no npm install. Everything is in the folder.

## Package Mirroring (Future)

In addition to publishing to PyPI and npm, we should also publish package releases to this S3 bucket. This solves two problems:

1. **PyPI availability:** PyPI is occasionally slow or unavailable. A local mirror ensures CI and offline deployments aren't blocked.
2. **Release completeness:** A version's S3 folder contains everything — the website, the packages, the docs, the CI artifacts. Nothing external is required.

This is a future item but the folder structure (`packages/pypi/`, `packages/npm/`) is reserved now.

## CI Artifacts to Store

For every release, store alongside the website:

| Artifact | Location | Purpose |
|---|---|---|
| **Git diff** (from main, from previous version) | `ci/{date}/` | See exactly what changed in this release |
| **Git commit messages** | `ci/{date}/` | Release notes, context |
| **Build log** | `ci/{date}/` | Debugging failed deployments |
| **Dependency snapshot** | `releases/{version}/deps.json` | Exact versions of every dependency |
| **Test results** | `ci/{date}/` | Evidence that tests passed |

This data currently lives only in GitHub. If GitHub goes down, we lose visibility into what happened. Storing it in S3 (and eventually in multiple cloud providers — see multi-cloud brief) removes that single point of failure.

---

## Shared / Slow-Changing Assets

Not everything changes on every release. Design assets, brand images, fonts — these change rarely. Store them separately:

```
cdn/
├── assets/
│   ├── latest/
│   │   ├── logo.svg
│   │   ├── fonts/
│   │   └── brand-colours.css
│   └── releases/
│       └── v1.0.0/
```

The CDN assets have their own versioning cadence — they don't need a new release every time the website content changes. Website releases reference a specific CDN asset version.
