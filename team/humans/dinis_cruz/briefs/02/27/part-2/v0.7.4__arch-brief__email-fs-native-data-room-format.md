# Arch Brief: Email-FS — Email as Native Data Room Format and Agent Communication Primitive

**version** v0.7.4  
**date** 27 Feb 2026  
**from** Human (project lead)  
**to** Architect (lead), Developer, Librarian  
**type** Arch brief — new primitive, design phase  

---

## The Insight

We already have 90% of what's needed to build a mini email client. The data room supports file storage, preview, versioning, and encryption. An EML file is just a file. If the data room can render EML files the way it renders PDFs and CSVs, we have an inbox.

The further insight: email is not just a user feature — it is a communication primitive for agents. The email format (to, from, subject, body, attachments, signatures) maps cleanly to the kind of structured messages agents already exchange. If agents communicate via email primitives stored in data rooms, we get traceability, verifiability, and a familiar interface for humans to inspect agent communication.

## Part 1: EML as a Native Data Room Format

### What EML Gives Us

The EML format (RFC 5322 + MIME) is a well-defined, widely understood envelope:

```
From: sherpa@sgraph.ai
To: human@sgraph.ai
Subject: Morning Standup — 27 Feb 2026
Date: Thu, 27 Feb 2026 09:00:00 +0000
Content-Type: multipart/mixed; boundary="boundary123"

--boundary123
Content-Type: text/html; charset=utf-8

<html>
  <p>Good morning. 4 new emails, 12 pending, 7 resolved.</p>
  ...
</html>

--boundary123
Content-Type: application/json; name="inbox-summary.json"
Content-Disposition: attachment; filename="inbox-summary.json"

{ "new": 4, "pending": 12, "resolved": 7 }

--boundary123--
```

This is a self-contained message with metadata (from, to, date, subject), a body (HTML or plaintext), and attachments. The data room already stores files. An EML file is just another file — but one that the preview component can render as a message.

### The Rendering

When a `.eml` file is opened in the data room, the preview component renders it as an email:

```
📧 Morning Standup — 27 Feb 2026

From:    Sherpa <sherpa@sgraph.ai>
To:      Human <human@sgraph.ai>
Date:    27 Feb 2026, 09:00

Good morning. 4 new emails, 12 pending, 7 resolved.
[... formatted body ...]

📎 Attachments:
  📄 inbox-summary.json    [Preview] [Download]
```

This is not a full email client. It is a file viewer that understands the EML format — the same way the CSV viewer understands CSV and the PDF viewer understands PDF.

### From Viewer to Inbox

A folder of EML files, sorted by date, IS an inbox:

```
📁 Data Room: "Sherpa Communications"
└── 📁 inbox/
    ├── 📧 2026-02-27__morning-standup.eml
    ├── 📧 2026-02-27__alice-reply.eml
    ├── 📧 2026-02-26__outreach-batch-001-confirmation.eml
    └── 📧 2026-02-26__bob-enquiry.eml

    [Sort: newest first]  [Filter: unread]  [Search]
```

The data room's existing folder navigation, sorting, and filtering — plus the EML renderer — produces a functional inbox view with zero new backend infrastructure. The messages are files. The inbox is a folder. The encryption is the data room's existing encryption.

### Reply, Forward, CC

The step from "view emails" to "compose emails" is primarily a UI addition:

| Action | Implementation |
|---|---|
| **Reply** | Create a new EML file with `In-Reply-To` header referencing the original. Pre-fill `To:` from the original's `From:`. Threading is header-based (standard email threading). |
| **Forward** | Create a new EML with the original's body and attachments included. New `To:` field. |
| **CC** | Standard EML header. No special handling needed. |
| **Compose** | Create a new EML from scratch. To, Subject, Body, optional attachments from the data room. |

The composed EML is saved to the data room. If it needs to be sent externally (via SES), a send workflow picks it up — the same pattern as the outreach workflow, but triggered by the user composing in the data room.

## Part 2: Email-FS — The Fractal Email System

### The Concept

Issues-FS uses the file system as a database for issue tracking. Email-FS uses the same principle for email: the file system IS the email store.

```
Issues-FS:                          Email-FS:
  .issues/                            .emails/
  ├── open/                           ├── inbox/
  │   ├── issue-001/                  │   ├── msg-001.eml
  │   │   ├── issue.json              │   ├── msg-002.eml
  │   │   └── attachments/            │   └── msg-003.eml
  │   └── issue-002/                  ├── sent/
  └── closed/                         │   ├── msg-004.eml
      └── issue-003/                  │   └── msg-005.eml
                                      ├── drafts/
                                      │   └── msg-006.eml
                                      └── threads/
                                          └── thread-001/
                                              ├── msg-001.eml
                                              └── msg-002.eml (reply)
```

### The Fractal Property

Every data room can have its own email system. Data rooms within data rooms can have their own email systems. An EML attachment can itself contain an Email-FS structure (if the attachment is a zip of EML files). This is the same fractal property as Issues-FS:

```
Data Room: "Project Alpha"
├── 📁 .emails/                    ← Project Alpha's email system
│   ├── 📁 inbox/
│   └── 📁 sent/
├── 📁 sub-rooms/
│   └── 📁 "Due Diligence"/
│       ├── 📁 .emails/            ← Due Diligence sub-room has its own email
│       │   ├── 📁 inbox/
│       │   └── 📁 sent/
│       └── 📁 documents/
└── 📁 documents/
```

Each level is independent. The parent room's email system does not see the child's, unless explicitly shared. This enables compartmentalised communication — the due diligence team's internal messages are separate from the project's main communication.

## Part 3: Agent-to-Agent Communication via Email

### Why Email Works for Agents

The email format maps naturally to agent communication:

| Email field | Agent meaning |
|---|---|
| **From** | Which agent/role sent this |
| **To** | Which agent/role should process this |
| **Subject** | Task or topic identifier |
| **Body** | The reasoning, instructions, or report |
| **Attachments** | Supporting data, files, analysis outputs |
| **In-Reply-To** | Links this to the previous message in a conversation |
| **Signature** | PKI signature — verifiable authorship |

When agents communicate via email primitives:

- **Traceability**: every message is a file, stored, versioned, auditable
- **Reasoning capture**: an agent can leave its reasoning in the email body — this becomes a permanent, inspectable record of why a decision was made
- **Threading**: the `In-Reply-To` chain shows the conversation flow — which agent asked what, who responded, what was decided
- **Human inspection**: the human opens the agent communication folder and reads emails — the most natural interface for understanding a conversation

### Agent Email Pods

An agent execution environment ("pod") has its own email context:

```
Agent Pod: "Sherpa — Outreach Session 2026-02-27"

  Inputs (received emails):
    📧 From: Human → "Here is the user list for today's batch"
       📎 Attachment: users.csv
    📧 From: Ambassador → "Use case messaging for this batch"
       📎 Attachment: messaging-guide.md

  Outputs (sent emails):
    📧 To: Human → "Batch draft ready for review"
       📎 Attachment: send-manifest.json
    📧 To: GRC → "Sensitivity classification for 3 new recipients"

  Internal reasoning (self-addressed):
    📧 To: self → "Decision log: chose FOLLOWUP-01 for carol@corp.com because..."
```

The pod receives its inputs as email, produces its outputs as email, and logs its reasoning as email. Everything is a file. Everything is inspectable. Everything is signed when PKI is active.

### Connection to the Supply Chain Model

This connects to the execution environment idea: give a pod the code, the data, and the agent context — all as an email envelope. The agent processes it and returns results as another email envelope. The envelopes are encrypted, signed, and stored in the data room.

```
Execution request (email to agent):
  From: conductor@sgraph.ai
  To: sherpa@sgraph.ai
  Subject: Execute: Outreach Batch 2026-02-27
  Attachments:
    - batch-manifest.json (encrypted)
    - template.html (encrypted)
    - users.csv (encrypted, scoped — only this batch's rows)

Execution result (email from agent):
  From: sherpa@sgraph.ai
  To: conductor@sgraph.ai
  Subject: Re: Execute: Outreach Batch 2026-02-27
  Body: "Batch complete. 10 sent, 0 failed."
  Attachments:
    - updated-manifest.json (encrypted, signed by Sherpa)
    - execution-log.json (encrypted)
```

## Part 4: Email Systems Stuck in Time

The human's observation from using AWS WorkMail: enterprise email systems are frozen in time. They are maintained but not improved. The people who built them are long gone. The UIs are from 2010. The APIs are limited. There is no innovation happening.

This is an opportunity — not to build a full email replacement (not the starting point), but to demonstrate that a modern, encrypted, agent-native email system is possible by building it incrementally inside the data room.

The path:
1. **EML viewer** in the data room (this brief — Phase 1)
2. **Compose/reply** in the data room (Phase 2)
3. **Send via SES** from the data room (Phase 3 — connect to existing email infrastructure)
4. **Agent communication** via Email-FS (Phase 4)
5. **Full email client** as a data room feature (Phase 5 — long-term)

Each phase is independently useful. Phase 1 alone makes the Sherpa inbox visible in the data room. Phase 2 lets the human reply without leaving the browser. Phase 4 gives agents a traceable, verifiable communication channel.
