# Brief: n8n Platform Assessment and Encrypted Workflow Data

**version** v0.7.4  
**date** 27 Feb 2026  
**from** Human (project lead)  
**to** Architect (lead), AppSec, DevOps, Developer  
**type** Brief — platform assessment, architecture direction, immediate review  

---

## Context

n8n is currently the orchestration layer for email sending, MCP bridging, and workflow triggers. After extensive hands-on use during the first real outreach batch, the human has a detailed assessment of the platform's strengths, weaknesses, and — critically — its security posture.

The conclusion is not "replace n8n" but "contain n8n" — treat it as a routing/triggering layer and move all sensitive data handling to SG/Send's encrypted infrastructure.

---

## Part 1: Platform Assessment — What Works and What Doesn't

### What Works

| Strength | Detail |
|---|---|
| **Visual flow builder** | The block-connection model is genuinely good for seeing workflow structure at a glance |
| **Debuggability** | Execution logs show exactly what happened at each step — invaluable for troubleshooting |
| **MCP integration** | Connecting to SG/Send and other services via MCP worked cleanly |
| **Trigger ecosystem** | Webhooks, schedules, email triggers — broad coverage for event sources |

### What Doesn't Work

| Problem | Detail | Severity |
|---|---|---|
| **No real version control** | Workflows are versioned as snapshots, not diffs. You can revert to a previous version, but you cannot see what changed between versions. No branching, no merging, no diffs. This is version control like Word's — a backup mechanism, not a development tool. | High |
| **Complexity scales badly** | Every new node adds visual and cognitive complexity. There is no refactoring — no way to extract a pattern into a reusable abstraction. Sub-workflows exist but don't compose cleanly. The more you build, the harder it gets, instead of easier. | High |
| **Editing environment is poor** | No concept of scripts within scripts. JavaScript nodes are edited in a tiny text box with no syntax highlighting, no linting, no multi-file support. For anything beyond trivial logic, it's faster to write code. | Medium |
| **Brittle XML/JSON transformations** | Data mapping between nodes uses hard-coded field references. Change a field name upstream and every downstream reference breaks silently. No type checking, no refactoring tools. | High |
| **No secrets management** | The version tested had no built-in secrets vault. API keys, tokens, and credentials are stored in node configuration — visible in the workflow JSON, visible in execution logs. | Critical |
| **No undo** | A misclick that deletes a node or connection has no undo. The only recovery is reverting the entire workflow to a previous snapshot version. | Medium |

### The Core Problem: Abstraction Failure

The fundamental issue is that n8n's visual programming model lacks the abstraction mechanisms that make text-based programming manageable at scale. In Python, you manage complexity through functions, classes, modules, packages — each level hiding detail. In n8n, every detail is visible all the time. There is no equivalent of "extract function" or "create module."

This means n8n workflows that start simple become unmanageable as they grow. The human's assessment: **"After a while I was like, it's faster for me to code this."** And the coded version would be less brittle, more testable, more version-controllable.

### Recommendation: Contain, Don't Replace

n8n is useful for what it does well: visual triggers, webhook routing, simple integrations. The architecture should contain n8n to this role and move everything else out:

```
n8n's role (keep):              Move out of n8n:
  - Webhook triggers             - Email body construction
  - Schedule triggers             - Data transformation
  - Simple routing                - Template rendering
  - MCP bridge calls              - Sensitive data handling
  - Event notifications           - Business logic
                                  - Anything that touches secrets
```

## Part 2: n8n as a Data Leak Surface

This is the critical security finding.

### The Problem

n8n's execution logs capture the full request and response bodies for every node. In the current email workflow, this means the logs contain:

- Full email bodies (including download URLs with decryption keys)
- Recipient email addresses
- Access tokens
- Template content
- Any data passed through the workflow

**A full n8n compromise — database access, log access — exposes all of this.** This directly violates the zero-customer-data-on-server-leak principle established for SG/Send.

### Current State

| Data in n8n logs | Sensitivity | Should it be there? |
|---|---|---|
| Recipient email addresses | Personal data (GDPR) | No |
| Email body with download URL + key | Contains decryption key | **Absolutely not** |
| Access tokens | Credential | No |
| Template HTML | Low sensitivity | Acceptable |
| Workflow execution metadata | Low sensitivity | Acceptable |

### The Fix: Zero Sensitive Data in n8n

Apply the same zero-knowledge principle to n8n that applies to SG/Send's server:

**Principle: A full n8n compromise — all databases, all logs, all workflow definitions — should expose zero customer data and zero decryption keys.**

### Architecture: Encrypted Envelopes

Instead of passing plaintext data through n8n nodes, pass encrypted references that are resolved only at the point of use:

```
Current (broken):
  Claude → [email body + key in plaintext] → n8n → SES
  n8n logs contain: everything

Target:
  Claude → [encrypted envelope in SG/Send data room] → reference ID → n8n
  n8n receives: an opaque reference ID
  n8n calls SES with: the reference ID resolved at send time via scoped MCP call
  n8n logs contain: opaque IDs, timestamps, status codes — nothing readable

  The email body is never in n8n's memory in plaintext.
  The decryption key is never in n8n's memory at all.
```

### How This Works with the Data Room

The outreach data room already holds drafts, templates, and user data. The workflow becomes:

```
1. Claude generates: template + personalisation → saved to data room as encrypted files
2. Claude creates: send manifest (JSON) in data room → lists recipients + references to their personalised content
3. n8n trigger: reads manifest reference (opaque ID only)
4. n8n per-recipient: calls a send endpoint that:
   a. Reads the encrypted email content from the data room (scoped access)
   b. Decrypts at the point of SES submission
   c. Sends via SES
   d. Updates the manifest with status
5. n8n logs contain: manifest ID, per-recipient status, timestamps — no content
```

The decryption and SES submission happen in a Lambda function or dedicated service — not in n8n. n8n is the trigger and status tracker. The sensitive data never passes through n8n's execution context.

### Scoped Access Per Workflow Step

Each n8n workflow step gets access only to what it needs:

| Step | Access | Does NOT have access to |
|---|---|---|
| Trigger (webhook/schedule) | Manifest reference ID | Email content, recipient data |
| Send step | Scoped read: one recipient's encrypted content | Other recipients, decryption keys for other emails |
| Status update | Write: manifest status field | Email content |
| Notification | Summary stats (count, success/fail) | Any recipient data |

This is the same capability minimisation principle from the agentic email security brief (v0.7.1) — applied to the infrastructure layer instead of the agent layer.

## Part 3: n8n Log Retention Policy

Even with encrypted envelopes, n8n's default logging should be hardened:

| Setting | Current | Target |
|---|---|---|
| Execution log retention | Default (indefinite?) | 7 days maximum |
| Log full request bodies | Yes (default) | No — log status codes and metadata only |
| Log full response bodies | Yes (default) | No |
| Workflow definition backup | Manual | Automated, version-controlled in git |
| Credentials storage | In workflow JSON | External secrets manager (AWS Secrets Manager) |

**DevOps action required:** Configure n8n log retention and body logging before the next outreach batch.
