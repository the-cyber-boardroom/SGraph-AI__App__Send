## Context

External validation has confirmed the messaging direction: **use cases beat features**. A senior executive's reaction to SG/Send was indifferent during the technical explanation but immediately engaged when shown the Claude MCP workflow — "go to Claude, say send this secret, it just works." That specific demo was the moment it clicked.

We now need marketing materials that lead with use cases, not encryption specs. And we need to expand outreach beyond the waiting list to the human's direct contacts.

---

## Part 1: The Messaging Shift — Use Cases First

### The Principle

Stop leading with "zero-knowledge AES-256-GCM encryption." Start leading with "don't email it — Send it."

Every marketing material starts with a workflow the user already does, then shows how SG/Send makes it better. The encryption is the reason it's better — but it's not the headline.

### Per-Use-Case Comparison Campaign

For each common workflow, produce a clear, validated comparison: what happens today vs what happens with SG/Send.

| Use Case | Today's Workflow | SG/Send Workflow | Key Advantage |
|---|---|---|---|
| **Share a password** | Paste it in Slack/WhatsApp/email — persisted in plaintext, searchable, visible to admins | Send via one-time secret link — viewed once, then gone | Password never stored in a chat history |
| **Send a contract to a client** | Email attachment — your email provider, their email provider, every mail server in between can read it | Send via SG/Send — encrypted in your browser, only the recipient can open it, server stores nothing readable | Neither email provider can read the document |
| **Share an API key with a colleague** | Paste in Slack DM — Slack retains it, admins can search it, it appears in compliance exports | One-time secret link — self-destructs after first view | Zero persistence in any third-party system |
| **Send ID documents to a bank/solicitor** | Email attachment or upload to their portal (which they control and you can't verify) | SG/Send link — you control the encryption, the recipient decrypts in their browser, nothing stored server-side after download | You don't have to trust their infrastructure |
| **Send files from Claude** | Claude generates a file → you download it → you email it → recipient downloads it | "Claude, send this to alice@example.com" → done. Encrypted, delivered, one step. | The AI integration IS the workflow |

**AppSec and GRC must validate each comparison.** The claims must be accurate and defensible. Specifically:
- Each "Key Advantage" must be a verifiable architectural property, not a marketing aspiration
- The comparison must be fair — acknowledge what SG/Send does NOT provide (e.g., symmetric mode key-in-email is not E2E in the purest sense)
- Reference the symmetric vs PKI explainer (v0.7.1) for nuance

### The Pragmatic Security Position

Yesterday's AppSec and GRC reviews established the key framing: **as long as we are not reducing the security of any workflow, we should have the confidence to say "this is better."**

The comparison is not "SG/Send vs theoretical perfect security." It is "SG/Send vs what people actually do today." Against that baseline, every use case above is a clear improvement. The GRC risk register (v0.7.1) documents this position with accept/mitigate decisions.

## Part 2: Marketing Materials — What to Produce

### Material 1: Shareable Images for WhatsApp and LinkedIn

The human needs to be able to drop an image into a WhatsApp chat or LinkedIn post that immediately communicates the value proposition for a specific use case.

**Format:** SVG (scalable, editable, high-quality on all screens) + PNG export (for platforms that don't support SVG)

**Dimensions:**
- WhatsApp: 1200×630px (link preview) or square 1080×1080px (image share)
- LinkedIn: 1200×627px (link post) or 1080×1080px (image post)

**Per use case, produce:**

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│   Don't WhatsApp your passwords.                     │
│   Send them.                                         │
│                                                      │
│   [Simple visual: password in a chat bubble with     │
│    a red X, vs a lock icon with a green check]       │
│                                                      │
│   Viewed once. Then gone. Nobody else can read it.   │
│                                                      │
│   send.sgraph.ai                                     │
│   Your code: EAP-MARCH-W1                            │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### Variants to Produce

| Variant | Headline | Visual Concept |
|---|---|---|
| **Password sharing** | "Don't WhatsApp your passwords. Send them." | Chat bubble → lock icon |
| **Document sharing** | "Don't email contracts. Send them." | Email attachment → encrypted envelope |
| **Claude integration** | "Ask Claude to send it securely. It just works." | Claude chat → SG/Send delivery |
| **General** | "Your files. Only the recipient can open them." | File → lock → recipient |
| **PDF viewer** | "Send a PDF. They see it in the browser. Encrypted." | PDF rendering in SG/Send UI |

### Two Production Paths (in parallel)

**Path A — Designer produces polished SVGs:** The Designer creates the final materials following the SG/Send brand. These are the production versions.

**Path B — Detailed descriptions for image LLMs:** For each variant, produce a detailed text description that the human can take to NotebookLM, Midjourney, DALL·E, or other image generation tools and say "make me one of these." This enables rapid iteration on visual concepts without waiting for the Designer.

The description should include:
- Exact text on the image
- Layout description (what goes where)
- Colour palette (SG/Send brand colours)
- Visual metaphor being used
- Dimensions and format
- What to avoid (no stock photos, no generic "security shield" clichés)

## Part 4: Early Adopter Programme Expansion

### EAP Code Management

Current code: `EAP-February-Week-4` (used for waiting list batch).  
Next code: `EAP-March-Week-1` (for personal contacts outreach).

The human is deliberately using guessable, human-friendly codes. This is intentional:

| Concern | Response |
|---|---|
| "The codes are easy to guess" | Good. We want usage. If someone guesses the code and tries the platform, that's a success. |
| "Someone could abuse the code" | We monitor usage. The audience is small. The risk is low and the upside of frictionless sharing outweighs it. |
| "What about rate limiting?" | Monitor. If abuse appears, tighten. Don't pre-optimise for a problem that doesn't exist yet. |

The codes should be trackable — each code maps to a batch, so we can see which outreach channel drove which sign-ups.

### Outreach Channels

| Channel | Material | Code |
|---|---|---|
| **Waiting list** (website sign-ups) | Multi-agent email (already sent) | EAP-February-Week-4 |
| **Personal contacts** (email) | Personal email template (this brief) | EAP-March-Week-1 |
| **WhatsApp** (direct messages) | Shareable image + short message | EAP-March-Week-1 |
| **LinkedIn** (posts + DMs) | LinkedIn post variant + DM with code | DM for code (or EAP-March-Week-1) |

### Website Examples

The website must show what SG/Send looks like in action. Specifically:
- Screenshot of the upload flow (drop a file → encrypt → get a link)
- Screenshot of the download/viewer (recipient opens PDF inline, sees transparency panel)
- Screenshot of the Claude integration ("send this secret" → delivery confirmation)

These screenshots should be real — not mockups. The product already has good viewers (PDF rendering, text preview). Show them.
