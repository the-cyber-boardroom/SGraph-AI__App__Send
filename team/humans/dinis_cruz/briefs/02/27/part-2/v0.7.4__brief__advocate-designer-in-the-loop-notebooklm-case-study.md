# Brief: The Advocate and Designer in the Loop — Why Small Bugs Reveal Big Pipeline Problems

**version** v0.7.4  
**date** 27 Feb 2026  
**from** Human (project lead)  
**to** Advocate (lead), Designer, Accountant, Librarian, all roles  
**type** Brief — role activation, design philosophy, change economics  

---

## Purpose

This brief uses NotebookLM as a case study to make two arguments:

1. **The Advocate and Designer must be involved at every decision, every step** — the same way AppSec is involved at every layer. Small bugs that persist in production are not P3s. They are P1 signals that the development pipeline cannot respond to user feedback.

2. **Every change has a cost, and the cost must be budgeted** — not just in money, but in cognitive load, clicks, and user trust. The Accountant manages this budget. Changes that reduce friction earn credits. Changes that add friction spend them.

---

## The Case Study: NotebookLM

NotebookLM is one of the best AI tools available today. The human uses it daily, pays for the upgraded Google Workspace tier specifically for it, and considers it brilliant. When a document needs to be understood, the first step is to put it through NotebookLM.

**What NotebookLM does spectacularly well:**
- Creates an intermediate representation of source material — an internal analysis that captures the essence of the content with remarkable accuracy
- Produces infographics and slide decks that feel like they genuinely understood the material — creative, witty, sometimes with real common sense
- The visualisations bring documents to life in ways that text alone cannot

**And yet it has three bugs that reveal something deeper.**

---

## Bug 1: Garbled Text in Beautiful Infographics

NotebookLM generates stunning visual layouts — colour, composition, hierarchy, all excellent. Then it leaves garbled text in the output. Font rendering artifacts, gibberish characters, text that clearly shouldn't be there.

The fix is trivial: run OCR on the generated image, compare the detected text against the source content, flag anything that doesn't match. A post-processing step that any intern could specify.

**What this tells us:** There is no quality gate between generation and delivery. The model produces the image, and the image ships. Nobody (and no automated check) looks at the output before it reaches the user. This suggests the team either lacks the pipeline to add post-processing checks, or the check was deprioritised. Either way: a gap in the engine that a Designer or Advocate reviewing outputs would catch instantly.

**Severity reframe:** This looks like a P3 (cosmetic). It's actually a P1 signal: the output quality pipeline has no validation step.

---

## Bug 2: The Upload Workflow — Death by a Thousand Clicks

The human's core workflow is simple: drop one file, get an infographic and a slide deck. That's it. One input, two outputs. This should be drag-and-drop plus one click.

Here is what actually happens:

```
Step 1:  Go to NotebookLM                                    (page load: ~2s)
Step 2:  Click "Upload Sources" (can't drag-drop to main UI)  (click + load: ~2s)
Step 3:  Drag and drop the markdown file                      (action)
Step 4:  Spinner keeps spinning (bug: 9/10 times it hangs)    (wait... wait...)
Step 5:  Refresh the entire page                              (workaround: ~3s)
Step 6:  Files are there — but ALL sources are selected        
Step 7:  Click "Unselect All"                                 (click)
Step 8:  Select the ONE file you just uploaded                 (click)
Step 9:  Find the Infographic button                          (scan UI)
Step 10: Click it, wait for it to activate                    (click + wait)
Step 11: Click it again                                       (click)
Step 12: Wait for generation                                  (wait)
Step 13: Open the infographic to read the title               (click)
Step 14: Manually copy the title                              (select + copy)
Step 15: Click "Download"                                     (click)
Step 16: File downloads as "Untitled.png"                     (not the title)
Step 17: Rename the file on local disk                        (manual rename)
Step 18: Drag to destination folder                           (file management)
```

**18 steps for a workflow that should be 2-3.**

The slide deck, by contrast, downloads with the correct filename. So the infrastructure to name downloads correctly exists — it's just not applied to infographics. This is not a technical limitation. It's a gap in the workflow that nobody with authority is fixing.

**What this tells us:** The team either doesn't have telemetry showing this is a dominant user pattern, or they do and can't prioritise the fix. Either way: the Advocate (the role that maps user trails and identifies friction) is either absent or powerless.

---

## Bug 3: The Buttons That Moved

In a recent update, NotebookLM rearranged the location of UI buttons. The same buttons, the same functions, just in different places. No buttons were added or removed. No workflow was simplified. The user still sees the same eight options — they're just somewhere else now.

**The cost of this change:**
- Every existing user's muscle memory is broken
- Cognitive load increases ("where did it go?")
- No steps were removed, no clicks were saved
- The workflow is not faster, not simpler, not better

**The benefit of this change:** Unknown. Possibly none for the user.

**What this tells us:** Someone in the pipeline made this change. It was reviewed, approved, tested, and deployed. At no point did someone with authority ask: "Is this better for the user? Does this remove a step? Does this reduce cognitive load?" Or if they asked, the answer was irrelevant — the change shipped anyway.

**This is the most important bug of the three.** Not because it's the worst user experience, but because it reveals the most about the pipeline.

## The Jonathan Ive Principle: Good Design Starts with the User

The Apple design philosophy under Jonathan Ive was not "make it look different." It was "make it better." The starting question was always: **how can we make this workflow simpler, more natural, more invisible?**

The canonical example: Steve Jobs was shown a proposed workflow for burning MP3s to a CD. He walked to the whiteboard, drew a big rectangle ("this is the screen"), drew a small circle ("this is the CD icon"), drew a box in the middle ("this is the folder with your MP3s"), and said: "You select these, you drag and drop them there, and they get burned. That's it."

Compare that to the alternative: open a special tool, navigate to your files, select them, add them to a staging area, configure burn settings, click OK, wait, confirm. The Jobs version: drag and drop. Done.

**The principle:** It's okay for design to change. It's okay for things to move, to look different, to be added or removed. But you don't start there. You start with: **is it better?** You start with the user's intent, and you work backward to the simplest possible interaction.

Good design is invisible. Good design feels natural. When a change is well-designed, the only way to notice it is to go back to the previous version and think: "This is way worse." You don't notice the improvement — you notice its absence.

The button rearrangement in NotebookLM fails this test. Going back to the previous layout would feel... the same. Or better, because muscle memory still works.

## The Midjourney Counterpoint: Product Before Polish

One nuance: Midjourney has shown that you can ship with a genuinely bad user experience if the product itself is extraordinary. Discord-only interface, no direct UI, cryptic commands — and millions of users anyway.

This is not an excuse for bad UX. It's a prioritisation insight: **when you're still discovering how the product is used, focus on making the output incredible.** The UX can be rough in the early phase because you're learning what users actually do. It's only when you have usage data that you can iterate on the experience.

NotebookLM gets away with its bugs for the same reason: the output is insanely good and there's no real alternative. But that grace period doesn't last forever. And the small bugs are the canary.

## The Pipeline Inference: What Persistent Small Bugs Really Mean

Here is the key insight:

**When you see small, obvious bugs that persist across multiple releases — bugs that any user, QA tester, or power user would notice immediately — you are not seeing a QA problem. You are seeing a pipeline problem.**

The bug was noticed. It's impossible that nobody at Google saw garbled text in infographics, or experienced the spinner hang, or noticed the buttons moved. Someone noticed. Multiple people noticed.

**The question is not "did someone see it?" The question is: "did someone with authority to fix it have a pipeline that could respond?"**

If changing the button locations back takes a week of change control, sprint planning, QA cycles, and release management — it won't happen. The cost of the fix exceeds the perceived severity. The bug ships. The bug stays.

This is how companies become bloated. Not by making bad decisions, but by making it expensive to make small decisions. Change control that was designed for safety becomes a tax on quality. Agile that was designed for responsiveness becomes waterfall with shorter cycles.

**Bret Victor's "Inventing on Principle":** The direct connection between creator and creation is what produces great work. When a craftsman can make a hundred small iterations — each slightly different, slightly better — they converge on excellence. But if there's a week between each iteration, or even an hour, or sometimes even a minute — you break the mojo. The feedback loop is severed. The craftsman can't craft.

**Amazon's two-pizza teams:** Put a highly concentrated group of multidisciplinary talent in the same place. They achieve extraordinary results because they can make decisions fast, iterate together, and connect directly with users. The Advocate, the Designer, the Developer, the QA — all in one room (or one agent team), all with authority to act.

## The SG/Send Counter-Example: Yesterday's Access Token Features

Yesterday, the human was working with SG/Send's access token management and wanted four features: revoke tokens, change allocation numbers, sort by columns, and filter. Small features. Each one individually trivial.

```
09:00  Human identifies the need (using the product)
09:15  Brief written
09:30  Architect maps it
10:00  Developer implements
10:30  QA tests
11:00  Shipped to production

Four features. One morning. Now in prod.
```

These four small features individually seem minor. Together they transform the access token experience from "functional" to "useful." And they were possible because the pipeline supports it: brief → architect → dev → QA → ship, with enough test coverage that pushing to production is safe.

**The principle:** If you can't comfortably change button locations the day before release, your pipeline is broken. If all your tests pass and you still don't trust the deployment, your test infrastructure has gaps — and it's better to discover those gaps in peacetime than during an incident.

## What This Means for SG/Send: The Advocate's Mandate

### The Advocate Must Be Involved at Every Layer

The same way AppSec reviews every change for security implications, the Advocate reviews every change for user impact. Not as a gatekeeper — as a voice.

| Layer | Advocate's Question |
|---|---|
| **Explorer** (new feature) | "What user problem does this solve? How many clicks?" |
| **Villager** (productisation) | "Is the workflow simpler than before? Show me the before/after click count." |
| **Town Planner** (commodity) | "Does this change break muscle memory? What's the migration path?" |
| **Every PR** | "What changed from the user's perspective? Is the change log updated?" |

### The Designer Must Validate Every UI Change

No UI change ships without the Designer confirming it meets the Jonathan Ive test: **does it start with the user? Is it simpler? Would going back to the old version feel worse?**

If the answer to the last question is "no" or "the same," the change should not ship unless there's a documented technical reason (and the Accountant approves the cognitive load cost).

### Per-User Changelogs

When we change something, users should know. Not buried in release notes — directly in their experience:

```
"You were on v0.17. You're now on v0.25. Here's what changed for you:
  ✅ PDF viewer now loads 3x faster
  ✅ You can now sort access tokens by column
  ⚠️ The share button moved to the top bar (here's why: [link])"
```

This is the Advocate's deliverable: a per-user understanding of what changed and why. Eventually, this should be generated automatically from the change log and the user's usage patterns.

## The Change Budget: A New Economic Model

### The Concept

Every change to the product has a cost. Not just engineering time — **cognitive cost to the user.** The Accountant manages this budget.

| Change Type | Budget Effect |
|---|---|
| Remove a click from a workflow | **+1 credit** (earned) |
| Remove a field from a form | **+1 credit** (earned) |
| Simplify a function (less code, same result) | **+1 credit** (earned) |
| Add a button to the UI | **-1 credit** (spent) |
| Move a button to a different location | **-2 credits** (spent — broke muscle memory, no benefit) |
| Add complexity to a function | **-1 credit** (spent) |
| Add a new feature that adds steps | **-N credits** (spent, proportional to steps added) |

### How It Works

```
Developer's change credit balance: +12

Proposed change: Rearrange navigation buttons
  Cost: -2 credits (UI change, no step reduction)
  Advocate review: "No user benefit identified"
  Accountant decision: Rejected — spend credits on changes that earn credits instead

Alternative: Add keyboard shortcut for infographic generation
  Cost: -1 credit (new feature)
  Benefit: +3 credits (removes 3 clicks from the workflow)
  Net: +2 credits
  Advocate review: "Core workflow improved"
  Accountant decision: Approved — net positive
```

### Applied to Code

The same model applies to code changes:

| Code Change | Budget Effect |
|---|---|
| Refactor: extract function, reduce complexity | **+credit** |
| Refactor: remove dead code | **+credit** |
| New code: adds complexity to existing function | **-credit** |
| New code: new function, clean interface | **neutral** (necessary growth) |

The Accountant doesn't control this to spend less. The Accountant controls this to **spend efficiently** — to make sure that the team is building user value, not accumulating cognitive debt.

## The Agentic Scaling Argument

The traditional objection: "We can't have someone review every small change — that doesn't scale."

The agentic response: **yes it does.**

Every bug can have a customised agent team: an Advocate instance that evaluates user impact, a Designer instance that validates UI changes, a QA instance that tests the fix, an Accountant instance that approves the budget. These are not heavyweight processes — they're role MDs with scoped context, running in parallel.

```
Bug: "Infographic downloads as Untitled.png instead of the title"

  Agent team (spawned automatically):
    🤖 Advocate:   "High-frequency workflow. Users rename every download. -3 clicks if fixed."
    🤖 Designer:   "Filename should match the generated title. Slide deck already does this."
    🤖 Developer:  "One-line fix: pass title to download handler."
    🤖 QA:         "Test: download infographic → filename matches title."
    🤖 Accountant: "Fix earns +3 credits. Cost: 1 dev-hour. Approved."

  Result: Fix shipped same day.
```

This is where we're heading. Not a world where every bug requires a meeting — a world where every bug gets an instant, multi-perspective evaluation and a fast path to resolution.

## Lessons for the SG/Send Team

| # | Lesson | Action |
|---|---|---|
| 1 | Small persistent bugs signal pipeline problems, not QA problems | Track time-to-fix for user-reported issues. If it's measured in weeks, the pipeline needs work. |
| 2 | The Advocate must have authority, not just visibility | Advocate can block a release if user impact is negative and unjustified. |
| 3 | The Designer validates every UI change against the Ive test | "Is it simpler? Would reverting feel worse?" |
| 4 | Changes cost cognitive credits — the Accountant tracks them | Net-positive changes (fewer clicks) are always approved. Net-negative changes need justification. |
| 5 | Per-user changelogs — tell people what changed and why | Advocate produces the changelog. Eventually automated from PR metadata + usage patterns. |
| 6 | If you can't push a one-line fix to prod today, your pipeline is broken | CI confidence = deployment confidence. Gaps in tests are better found in peacetime. |
| 7 | Every bug can have its own agent team | Scale review through agents, not meetings. Advocate + Designer + Dev + QA + Accountant per issue. |
