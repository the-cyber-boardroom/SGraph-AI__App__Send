# Dev Brief: Email Sending Workflow Optimisation

**version** v0.7.4  
**date** 27 Feb 2026  
**from** Human (project lead)  
**to** Developer (lead), Architect, Sherpa  
**type** Dev brief — workflow optimisation, immediate priority  

---

## Context

Last night we sent the first 10 Early Access Programme emails through the full agentic workflow. It worked — all 10 emails were delivered. But the process exposed three serious problems that must be fixed before the next batch.

---

## Problem 1: 14KB of HTML Regenerated Per Email

The email template is approximately 14KB of HTML with inline styles (branded to match the SG/Send website). The current workflow asks Claude to generate the full HTML for every single email. The only thing that changes between emails is a small placeholder at the top — "Hi [Name]" if we have a name.

```
Current (broken):
  For each of 10 recipients:
    Claude generates 14KB of HTML from scratch
    = 140KB of LLM output for 10 emails
    = most of it identical every time
    
Required:
  Claude generates the HTML template ONCE
  For each recipient:
    Claude fills in the placeholder (name, access code link)
    = 14KB + (10 × ~200 bytes of personalisation)
```

### The Fix: Template-Then-Fill

```
Step 1: Claude generates the full HTML email template with {{placeholders}}
  - {{recipient_name}} — "Hi Alice" or "Hi there" if no name
  - {{access_link}} — the one-time secret link for this recipient
  - {{sherpa_message}} — Sherpa's section (same for all in a batch, or per-user)
  - {{ambassador_message}} — Ambassador's section

Step 2: For each recipient, Claude (or a simple script) does string replacement
  - No LLM inference needed for the replacement
  - Just template.replace("{{recipient_name}}", user.name)

Step 3: Send the filled template via n8n/SES
```

This should reduce the per-email cost from ~14KB of LLM output to ~200 bytes of personalisation data. The template is generated once per batch, not once per email.

### Implementation Options

| Option | Approach | LLM Usage |
|---|---|---|
| **A: Claude does the replacement** | Claude generates template, then does find-replace per user | Minimal — but still uses context window |
| **B: n8n does the replacement** | Claude generates template → n8n workflow receives template + user list → n8n does string replacement per user → n8n sends | Zero LLM usage for send loop |
| **C: Python script in the repo** | Template stored as file, script reads user list from data room, fills placeholders, calls SES | Zero LLM usage, version-controlled, repeatable |

**Recommendation:** Option B for immediate use (n8n already wired), Option C as the production path. Option A is the current state and should be eliminated.

## Problem 2: Context Window Exhaustion

Sending 10 emails consumed enough context that Claude had to compress the conversation mid-send. This created two risks:

1. **Loss of state** — Claude's compression is lossy. Details about which emails were sent, which are pending, and what personalisation was applied can be dropped.
2. **Duplicate sends** — if Claude loses track of which recipients have already been emailed, it might resend. This did not happen last night (probably), but the risk is real.

### Why This Happens

Each email round-trip adds to the context:
- The HTML template (~14KB) is repeated in each send message
- The MCP tool call and response for each send
- The Sherpa's reasoning about personalisation per user
- n8n's response confirmation per send

At 10 emails, the accumulated context is significant. At 50 or 100 emails (the next batch sizes), it will be unmanageable.

### The Fix: Stateless Send Loop

The send loop should not accumulate context. Each email send should be an independent operation:

```
Before (stateful, context-accumulating):
  Claude session holds:
    - Template
    - Full user list
    - Per-email: draft → send → confirm → next
    - Context grows linearly with each email

After (stateless, context-minimal):
  1. Claude generates: template + personalised data per user (JSON array)
  2. Send loop is external (n8n workflow or script):
     For each user in array:
       Fill template
       Send via SES
       Log result
  3. Results returned to Claude as a summary: "10 sent, 0 failed"
```

The Claude session's job is authoring and approval. The send loop's job is mechanical execution. These should not be the same process.

---

## Problem 3: Lack of State Capture and Replayability

The send workflow has no durable state record. If the process fails midway (context exhaustion, network error, n8n timeout), there is no way to know:

- Which emails were sent successfully
- Which failed
- Which were never attempted
- What the exact content of each sent email was

n8n captures partial state (execution logs), but not the full picture.

### The Fix: Send Manifest

Before any emails are sent, create a **send manifest** — a JSON file in the outreach data room that records the intended batch:

```json
{
  "batch_id": "eap-2026-02-27-001",
  "template_version": "v0.7.4",
  "created_at": "2026-02-27T09:00:00Z",
  "created_by": "sherpa",
  "approved_by": "human",
  "recipients": [
    {
      "email": "alice@example.com",
      "name": "Alice",
      "language": "en-gb",
      "access_link": "https://send.sgraph.ai/s/xK9...",
      "status": "sent",
      "sent_at": "2026-02-27T09:05:12Z",
      "ses_message_id": "abc123..."
    },
    {
      "email": "bob@example.com",
      "name": "Bob",
      "language": "en-us",
      "access_link": "https://send.sgraph.ai/s/yL8...",
      "status": "failed",
      "error": "SES rate limit exceeded",
      "retry_count": 0
    }
  ]
}
```

This manifest:
- **Before send:** records what was approved (the human can review it)
- **During send:** updated per-email with status (sent/failed/pending)
- **After send:** complete audit trail of the batch
- **On failure:** enables retry of only the failed/pending emails
- **For replayability:** the exact content and recipients are recorded, not just in Claude's conversation context

The manifest lives in the data room at `sent/batch-eap-2026-02-27-001.json`. The Sherpa reads it for the morning standup. The human can inspect it in the browser via the CSV/JSON viewer.

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | Email template generated once per batch with {{placeholders}} — not regenerated per email |
| 2 | Send loop is external to Claude's context (n8n workflow or script) |
| 3 | Claude session produces: template + personalisation JSON array. Send loop consumes it. |
| 4 | Send manifest created in data room before any emails are sent |
| 5 | Manifest updated per-email with status (sent/failed/pending), timestamp, SES message ID |
| 6 | Failed sends can be retried from the manifest without re-running the full workflow |
| 7 | Context window usage for a 50-email batch is no larger than for a 10-email batch |
| 8 | Human can inspect the manifest in the data room before approving the send |
