## Scope

The workflow defined in `SKILL.md`: a 7-step pipeline where Claude (running in a sandbox) creates a file, wraps it in an SGMETA envelope, encrypts it with AES-256-GCM, uploads the encrypted blob to SG/Send via MCP, constructs a download URL with the decryption key in the URL fragment, verifies the round-trip, and optionally emails the link via n8n WorkMail.

This review covers two distinct modes — **symmetric (key-in-URL)** and **PKI** — because their threat profiles are fundamentally different and must not be conflated.

---

## The Two Modes: A Critical Distinction

This is the most important section of this document. The two modes look similar from the outside (both produce a download link) but have radically different security properties.

### Mode A: Symmetric (Decryption Key in URL Fragment)

```
URL: https://send.sgraph.ai/send/v0/v0.1/v0.1.8/download.html#<transfer_id>/<key_b64url>

The key IS in the URL.
Anyone who has the full URL can decrypt.
Security = secrecy of the URL.
```

### Mode B: PKI (Public Key Encryption)

```
URL: https://send.sgraph.ai/d/<key_prefix>/<doc_name>

The key is NOT in the URL.
The content is encrypted with the recipient's public key.
Security = possession of the recipient's private key.
The URL can be public. Only the private key holder can decrypt.
```

| Property | Symmetric (Mode A) | PKI (Mode B) |
|---|---|---|
| **Who can decrypt** | Anyone with the full URL | Only the private key holder |
| **URL sensitivity** | HIGH — URL is a secret | LOW — URL is a reference |
| **Sender verification** | None — anyone could have sent it | Yes — sender signs with their private key |
| **Server knowledge** | Encrypted blob + opaque ID | Encrypted blob + opaque ID (same) |
| **Key transmission risk** | Key travels with every URL share | Key never transmitted (already at recipient) |
| **Revocation** | Delete the transfer | N/A — key never transmitted |
| **Replay attack** | Possible until expiry/deletion | Not applicable |

## Mode A: Symmetric — Full Entity Analysis

This is the section the human specifically flagged. **Who can actually read the content in symmetric mode, in the current workflow?**

### The Entity List

When Claude creates and sends a file using the SKILL.md workflow with email delivery:

| Entity | What They See | When | Can they decrypt? | Notes |
|---|---|---|---|---|
| **Claude session context** | Plaintext file content, key, full URL | During generation | Yes — key is in context | Ephemeral — context window clears after session |
| **Anthropic (Claude logs)** | Potentially: plaintext content and/or key, depending on logging policy | Post-session | Depends on log retention | Anthropic's data retention policy applies |
| **n8n (in transit)** | Full URL including fragment (key) | During email workflow execution | Yes — URL passes through n8n | TLS-protected in transit |
| **n8n execution logs** | Full URL including fragment | During and after execution | Yes — if logs retained | Depends on n8n log retention policy |
| **AWS SES (in transit)** | Email body containing the URL | During send | Yes — URL in email body | TLS-protected in transit (mailed-by: eu-west-1.amazonses.com confirmed) |
| **AWS SES logs** | Email metadata (to, from, subject, size) | Post-send | No — body not in delivery logs typically | Metadata only |
| **Gmail (recipient's provider)** | Full email body including URL | On delivery | Yes — Gmail can read email | Signed by sgraph.ai, encrypted TLS in transit |
| **Google (Gmail)** | Full email body | Persisted in Gmail servers | Yes | Google's data access policies apply |
| **Recipient's Gmail account holder** | Full URL in email | On receipt | Yes — this is the intended recipient | Intended |
| **Anyone with access to recipient's Gmail** | Full URL | Ongoing | Yes | Shared accounts, device theft, account compromise |
| **SG/Send server** | Encrypted blob + opaque transfer ID | On upload | **No** — zero-knowledge preserved | Key never reaches server |

### The Honest Assessment

The entity list looks long. In context, it is still dramatically better than the alternative (cleartext transmission), but it must be presented honestly.

**What this is:** encrypted file storage with key transmitted via email, protected by TLS in transit and Gmail's account-level security.

**What this is not:** end-to-end encryption where only the recipient holds the key.

**The correct analogy:** a locked safe (the SG/Send blob), where the combination (the key) is written on a piece of paper and posted in a sealed envelope (TLS-protected email). The paper is safe from eavesdroppers on the street (TLS). But the post office (email provider), the delivery system (n8n), and the letter-writer (Claude) all handled the paper.

### Mitigating Factors

| Risk | Mitigation Available |
|---|---|
| n8n logs contain key | Configure n8n to truncate/not log full URL body. Flag for DevOps. |
| Claude session logs | One-time use: key is useless after first download. Short expiry window. |
| Gmail persistence | One-time use link: even if Gmail retains the email, the link stops working after download |
| Account compromise | One-time use + short expiry reduces the window of exposure |
| Anthropic log retention | Out of our control — acknowledge in DPO documentation |

**Key mitigation: one-time use is the primary defence in symmetric mode.** Once the recipient downloads and the transfer is deleted, the key decrypts nothing. The email in Gmail becomes a dead link. The n8n log entry becomes worthless. This converts the threat from "persistent exposure" to "narrow time window."

## Mode B: PKI — Threat Model

PKI mode has a fundamentally different threat profile. The comparison is stark.

### What Changes

In PKI mode, the content is encrypted with the **recipient's public key** before it ever leaves Claude's context. The private key is:
- Never generated by this workflow
- Never transmitted
- Held only by the recipient's browser (or hardware token)
- Not accessible to Claude, n8n, Gmail, or SG/Send

### Revised Entity Analysis (PKI Mode)

| Entity | What They See | Can they decrypt? |
|---|---|---|
| **Claude session context** | Plaintext content (before encryption), recipient's public key | No — encrypts but cannot decrypt |
| **Anthropic (Claude logs)** | Possibly the plaintext content (before encryption) | No — they never see the private key |
| **n8n** | The download URL (opaque reference, no key) | No |
| **Gmail** | The download URL (opaque reference) | No |
| **SG/Send server** | Encrypted blob | No — zero-knowledge preserved |
| **Recipient** | Decrypted content (in their browser, with their private key) | Yes — intended |
| **Anyone else** | An opaque URL that references an encrypted blob | No |

### The Remaining Risk in PKI Mode

The only entity that sees plaintext content in PKI mode is the Claude session that generates it. This is an unavoidable property of the current architecture: Claude creates the content, so Claude sees the plaintext. 

If the content is sensitive enough that Claude's session should not see it, the workflow must change: the human provides pre-encrypted content, or the content is generated externally and only the transmission uses this workflow.

For the current use cases (sending summaries, credentials, documents generated by Claude), Claude seeing the content it just generated is not an additional risk.

### Sender Authentication in PKI Mode

PKI mode enables something symmetric mode cannot: **proof of authorship**. If Claude/the Sherpa signs the content with the sender's private key before encrypting with the recipient's public key, the recipient can verify:
1. The content was encrypted for them specifically (cannot be read by anyone else)
2. The content was signed by the stated sender (cannot be forged)

This is the upgrade path the platform should push users toward. See the user positioning brief.

## Threat Scenarios

### T-001: URL Interception (Symmetric)

**Attack:** Attacker intercepts the email containing the download URL.

**Likelihood:** Low (TLS in transit), Medium (post-delivery Gmail compromise)

**Impact:** Full decryption of the transferred file.

**Mitigations:** One-time use (delete after download), short expiry, monitor for unexpected access.

**Residual risk:** Low with mitigations. Medium without.

---

### T-002: n8n Log Exfiltration (Symmetric)

**Attack:** Attacker gains read access to n8n execution logs, extracts URLs containing decryption keys.

**Likelihood:** Low (n8n is internal infrastructure) but non-zero.

**Impact:** Decryption of any transfer whose URL was logged and not yet expired.

**Mitigations:** Configure n8n to not log full request bodies. Rotate n8n API credentials. One-time use transfers.

**Residual risk:** Low with log configuration in place.

---

### T-003: Claude Session Context Exfiltration

**Attack:** Attacker accesses Claude session logs at Anthropic, recovers plaintext content and/or decryption keys.

**Likelihood:** Very Low (requires Anthropic infrastructure compromise).

**Impact:** Full decryption of content generated in that session.

**Mitigations:** Out of our control. Acknowledge in DPO documentation. Short-lived transfers reduce the value of stale keys.

**Residual risk:** Very Low. Accepted.

---

### T-004: SG/Send Server Breach (Both Modes)

**Attack:** Attacker gains full read access to SG/Send database and storage.

**Likelihood:** Non-zero (all servers are vulnerable to some degree).

**Impact (Symmetric):** Attacker has encrypted blobs. Without the keys (which are not on the server), they cannot decrypt. **Zero-knowledge is preserved.**

**Impact (PKI):** Same. Attacker has encrypted blobs, no keys. Zero-knowledge is preserved.

**Mitigations:** Zero-knowledge architecture — no keys on the server by design. Transfer IDs are opaque (no meaningful metadata exposed).

**Residual risk:** Very Low. This is a core design property.

---

### T-005: Prompt Injection via File Content (Both Modes)

**Attack:** A file being processed by the Sherpa contains injected instructions that cause the Sherpa to exfiltrate data or take unintended actions.

**Likelihood:** Low for current workflows (human reviews content before sending). Medium if automation increases.

**Impact:** Potential data exfiltration, unintended actions, reputational damage.

**Mitigations:** Security Analyst gate (see agentic email security brief v0.7.1). Human approval for all sends. Capability scoping per workflow step.

**Residual risk:** Medium without mitigations. Low with.

---

### T-006: Transfer ID Enumeration

**Attack:** Attacker guesses valid transfer IDs by brute force and downloads encrypted blobs.

**Likelihood:** Very Low — transfer IDs have sufficient entropy.

**Impact (Symmetric):** Attacker has encrypted blobs. Without the key in the URL fragment, they cannot decrypt.

**Impact (PKI):** Same. Encrypted for a specific private key.

**Residual risk:** Very Low.
