# Brief: Advocate Role Activation — Use-Case-First Development

**version** v0.7.1  
**date** 26 Feb 2026  
**from** Human (project lead)  
**to** Advocate (lead), Conductor, Architect, Developer, Sherpa  
**type** Brief — role activation, development philosophy, immediate effect  

---

## The Problem

We have an enormous backlog of features, ideas, and directions. The briefs from the last two weeks alone represent months of development work. Everything is valid. Not everything is equally urgent. We are making prioritisation decisions implicitly — whoever writes the next brief shapes what gets built next.

This needs to change. The Advocate role must now become an active, opinionated voice in what gets built and in what order.

---

## The Advocate's Job

The Advocate owns the user. Not the technology. Not the architecture. The user.

The questions the Advocate asks about every proposed feature:

```
1. Who is the user who needs this?
2. What are they trying to do?
3. Is this the simplest path to that outcome?
4. What do they need to do BEFORE this is useful?
5. Does this make the platform easier or harder to understand?
6. Would a non-technical user be able to use this without help?
7. What happens if we don't build this? Can the user do without it?
```

If a feature cannot be justified against a specific, named user need — it goes to the backlog. Not because it isn't good, but because there will always be more features than time.

---

## The Human Is the Power User

The human (project lead) is not just a stakeholder — they are an active user of the platform. This is unusual and valuable. Most software is built by people who don't use what they build. The human uses what they build, which means their workflow frustrations are real product gaps, not hypothetical ones.

This means: **features that unblock the human's current workflow are high-priority by definition.** They represent real, immediate needs — not speculative future ones. The Advocate should treat the human's workflow blockers as the first tier of the backlog.

---

## Use Cases First, Features Second

The shift being requested: stop thinking in terms of features and start thinking in terms of **use cases** — specific, end-to-end workflows that a real user needs to complete.

A feature is: "CSV editor in the data room."

A use case is: "As the human, I want to give the Sherpa a list of email addresses to contact, so that the Sherpa can send the outreach batch without me having to paste content into a chat window."

The use case is why the feature exists. The use case determines whether the feature is urgent. The use case defines what "done" looks like.

### The Use Case Template

Every proposed feature should be mapped to a use case:

```
USE CASE: [Short name]

Who:       [The specific user — human, Sherpa, external user, investor]
Goal:      [What they want to achieve]
Today:     [How they do it today — manual, workaround, can't do it]
Blocker:   [What's missing — the gap between today and the goal]
Feature:   [The specific capability that closes the gap]
Done when: [The observable outcome that means this is complete]
Priority:  [P0 / P1 / P2 — with rationale]
```

## The Parallel Development Mandate

As real users arrive on the platform, the cost of coupled changes increases. A bug in one feature that blocks another feature means downtime for users who depend on both. We need to decouple.

The Architect and Developer must now actively structure work so that:

1. **Each feature has an independent deployment path.** Shipping the CSV editor should not require a full platform deployment.
2. **Each feature has an independent rollback path.** Breaking the CSV editor should not break file transfer.
3. **APIs between features are versioned and stable.** The Sherpa reading from the data room should use a documented interface, not an internal implementation detail.
4. **Features can be developed in parallel by different agents/sessions** without stepping on each other.

This is not about microservices (yet). It's about discipline in how changes are scoped, tested, and deployed.

### Blast Radius Principle

Every change should be assessed: if this breaks, what else breaks? The answer should be "only this feature" — not "half the platform."

The Architect should maintain a **dependency map**: which features depend on which other features, and flag any proposed change that creates a new coupling.

## Advocate Deliverable: Use Case Backlog

The Advocate should maintain a living document: the **Use Case Backlog**. This is distinct from the feature backlog. It is ordered by user impact, not development convenience.

Format:

```
USE CASE BACKLOG — SG/Send — v0.7.1

[P0] Give Sherpa email list via data room table
[P0] Sherpa symmetric key vault access
[P0] Waiting list outreach campaign
[P1] Agent public key directory (sgraph.ai website)
[P1] Website contact → Sherpa email integration
[P1] Parallel deployment of independent features
[P2] Support screenshot via email
[P2] Vault MCP / Send MCP separation
...
```

This backlog is the Advocate's primary output. Every sprint, the Advocate reviews it, reorders it based on new user signals (feedback, usage, the human's current pain points), and presents the top 5 to the Conductor for scheduling.

---

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | Advocate produces initial Use Case Backlog covering all current known use cases |
| 2 | Every feature in active development is mapped to a use case (template above) |
| 3 | Architect produces initial dependency map — what depends on what |
| 4 | Development sprint scoping uses use cases as the unit, not features |
| 5 | Every new brief includes a "use case" section before the feature spec |
