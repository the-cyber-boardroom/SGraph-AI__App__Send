## Purpose

This document maps every data flow in the SG/Send skill workflow, identifies personal data processing, maps data controllers and processors, and documents what must be disclosed to users under GDPR transparency obligations (Articles 13/14).

## Data Flow Map: Symmetric Mode (Full Pipeline)

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  STEP 1-2: Claude Sandbox                                                       │
│                                                                                 │
│  INPUT: File content (may include personal data)                                │
│  PROCESSING: SGMETA envelope + AES-256-GCM encryption                          │
│  CONTROLLER: Anthropic (Claude platform) + Human (project lead as data user)   │
│  DATA IN MEMORY: Plaintext content, encryption key, encrypted blob              │
│  LOGGED BY: Anthropic (session logs — retention policy: see below)             │
└──────────────────────────┬──────────────────────────────────────────────────────┘
                           │ Encrypted blob (base64)
                           ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  STEP 3: SG/Send MCP → SG/Send Server (AWS, eu-west-1)                         │
│                                                                                 │
│  DATA UPLOADED: Encrypted blob only                                             │
│  SERVER SEES: Encrypted ciphertext + opaque transfer ID                        │
│  SERVER DOES NOT SEE: Plaintext content, filename, decryption key               │
│  CONTROLLER: SGraph AI Ltd (operating send.sgraph.ai)                          │
│  RETENTION: Until transfer downloaded (one-time use) or expiry                 │
│  LOGS: Transfer ID, timestamp, encrypted file size, IP address of uploader     │
└──────────────────────────┬──────────────────────────────────────────────────────┘
                           │ Transfer ID (returned)
                           ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  STEP 4: Claude Sandbox — URL Construction                                      │
│                                                                                 │
│  CONSTRUCTED: Full URL including decryption key in fragment                     │
│  DATA IN MEMORY: Full URL = transfer_id + key_b64url                           │
│  LOGGED BY: Anthropic (session logs — key is in the constructed URL)           │
└──────────────────────────┬──────────────────────────────────────────────────────┘
                           │ Full URL (including key)
                           ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  STEP 6: n8n Workflow Execution (n8n Cloud or self-hosted)                      │
│                                                                                 │
│  DATA PROCESSED: Email body containing full URL (including key)                 │
│  IN TRANSIT: TLS-encrypted (HTTPS to n8n API, then n8n → SES)                 │
│  LOGGED BY: n8n execution logs (⚠️ may include full request body — see RISK-004)│
│  CONTROLLER: Human (project lead) as n8n operator                              │
└──────────────────────────┬──────────────────────────────────────────────────────┘
                           │ Email (TLS-encrypted in transit)
                           ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  AWS SES (eu-west-1)                                                            │
│                                                                                 │
│  DATA PROCESSED: Full email (headers + body including URL with key)             │
│  IN TRANSIT: TLS (confirmed: mailed-by eu-west-1.amazonses.com, signed sgraph.ai)│
│  LOGGED BY: SES delivery logs (metadata only — to/from/subject/size)           │
│  CONTROLLER: Human (project lead) as SES operator                              │
│  RETENTION: SES delivery receipts — 14 days default                            │
└──────────────────────────┬──────────────────────────────────────────────────────┘
                           │ Email delivered
                           ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  Recipient's Email Provider (Gmail in current workflow)                         │
│                                                                                 │
│  DATA STORED: Full email body including URL with decryption key                 │
│  CONTROLLER: Google LLC (Gmail) / recipient                                     │
│  ACCESS: Google may access email content per their Terms of Service             │
│  RETENTION: Until recipient deletes — potentially indefinite                   │
│  MITIGATION: One-time use link — key becomes worthless after first download    │
└──────────────────────────┬──────────────────────────────────────────────────────┘
                           │ Recipient opens email, clicks link
                           ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  Recipient's Browser (decryption)                                               │
│                                                                                 │
│  REQUESTS: Encrypted blob from SG/Send (transfer_id only — key not sent)       │
│  DECRYPTS: AES-256-GCM locally using key from URL fragment                     │
│  SERVER SEES: Transfer ID lookup request — not the key, not the content        │
│  RESULT: Plaintext file in recipient's browser — never re-uploaded anywhere    │
└─────────────────────────────────────────────────────────────────────────────────┘
```

## Data Flow Map: PKI Mode

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│  STEP 1-2: Claude Sandbox                                                       │
│                                                                                 │
│  INPUT: File content + recipient's PUBLIC key                                   │
│  PROCESSING: Encrypt with recipient's public key                                │
│  LOGGED BY: Anthropic (session logs — plaintext content only; no key risk)     │
└──────────────────────────┬──────────────────────────────────────────────────────┘
                           │ Encrypted blob
                           ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  SG/Send Server                                                                 │
│                                                                                 │
│  STORES: Encrypted blob (only decryptable by recipient's private key)           │
│  NEVER SEES: Plaintext content, decryption key, private key                    │
└──────────────────────────┬──────────────────────────────────────────────────────┘
                           │ Opaque reference URL (no key)
                           ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  Email / Any Channel                                                            │
│                                                                                 │
│  TRANSMITTED: Opaque URL — a reference, not a key                               │
│  RISK IF INTERCEPTED: None — URL is a pointer, not a decryption key            │
│  EMAIL PROVIDER SEES: A URL. Cannot decrypt. Cannot access content.            │
└──────────────────────────┬──────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────────────────────────────┐
│  Recipient's Browser                                                            │
│                                                                                 │
│  FETCHES: Encrypted blob                                                        │
│  DECRYPTS: Using recipient's PRIVATE KEY (never leaves browser)                │
│  VERIFIES: Sender's signature (if signed) — proof of authorship                │
└─────────────────────────────────────────────────────────────────────────────────┘
```

**PKI data flow contrast:** In symmetric mode, the key travels through n8n, SES, and Gmail. In PKI mode, the key never exists in the transmission chain — the recipient already has the only key that matters (their private key, generated and held in their browser). This is the fundamental architectural difference.

## Transparency Obligations

Under GDPR Articles 13/14, users must be informed at the point of data collection about how their data is processed. For SG/Send users, the following must be disclosed in the privacy policy and ideally in the product UI:

### What must be disclosed for Symmetric Mode

> When you send a file using SG/Send with email delivery:
> - Your file content is processed in the Claude AI session that creates the transfer. Anthropic may retain session logs per their data handling policy.
> - The encrypted file is stored on SG/Send servers (operated by SGraph AI on AWS eu-west-1). The server cannot read the file content.
> - A download link containing the decryption key is emailed via AWS SES. The email is encrypted in transit (TLS). The email provider (e.g. Gmail) can read the email body, which includes the decryption link.
> - The download link is single-use. After the recipient downloads the file, the link is invalidated.
> - SG/Send stores: an encrypted blob, a transfer ID, upload/download timestamps, and IP addresses of uploader and downloader. Nothing else.

### What must be disclosed for PKI Mode

> When you send a file using SG/Send in PKI mode:
> - Your file is encrypted with the recipient's public key before it leaves the generating session. Only the holder of the corresponding private key can decrypt it.
> - The decryption key is never transmitted. It exists only in the recipient's browser.
> - The email (or other channel) used to share the download link contains only an opaque reference — not a key. Interception of the link cannot expose the file content.
> - SG/Send stores an encrypted blob it cannot read. Nothing else.
