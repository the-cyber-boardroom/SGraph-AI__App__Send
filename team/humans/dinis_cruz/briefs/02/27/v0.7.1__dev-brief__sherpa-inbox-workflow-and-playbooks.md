## Context

We now have two inboxes:
- **Sherpa inbox** — the agent-operated inbox for user-facing communication
- **Human inbox** — the project lead's personal inbox, which agents do NOT access

The MCP + n8n integration is live. The Sherpa already has four capabilities: list inbox, read email, send email, reply to email. This is everything needed to run a functional user communication workflow. This brief defines how to use those capabilities in a structured, daily workflow.

---

## The Morning Standup

The primary workflow is a human-initiated morning check. The human goes to Claude and asks:

```
"Sherpa — what's the current status of our inbox?  
 What emails have we received? What's pending? What needs action?"
```

The Sherpa:
1. **Lists the inbox** — calls `list_inbox` on the Sherpa's email account via MCP
2. **Reads new messages** — calls `read_email` for each unread message
3. **Categorises** — classifies each message: user reply, new enquiry, bounce, out-of-office, unknown
4. **Reports status** — presents a structured summary to the human
5. **Proposes actions** — for each message, suggests the appropriate next step from the playbook

The human reviews, approves (or modifies), and the Sherpa executes.

**This is not fully automated — yet.** The human remains in the loop for approval. Automation increases as playbooks prove reliable and as prompt injection risks are addressed (see companion security brief).

### Example Morning Report

```
Sherpa Inbox — 26 Feb 2026, 09:00

  📬 New messages: 4
  📭 Pending (no reply yet): 12
  ✅ Resolved: 7

  New messages:
  
  1. Alice Smith <alice@example.com>
     Subject: Re: Your SG/Send access code
     Category: USER_REPLY — positive response
     Suggested action: Send onboarding follow-up (Playbook: ONBOARD-02)
  
  2. bob.dev@gmail.com
     Subject: Question about data rooms
     Category: USER_ENQUIRY — feature question
     Suggested action: Reply with data room explainer (Playbook: FAQ-DATAROOM)
  
  3. mailer@linkedin.com
     Subject: [LinkedIn] Someone viewed your profile
     Category: NOISE — not user-related
     Suggested action: Archive
  
  4. noreply@mailchimp.com
     Subject: Delivery failure: 3 recipients
     Category: BOUNCE — 3 failed deliveries
     Suggested action: Mark as bounced in CRM data room (Playbook: BOUNCE-01)

  Awaiting reply (oldest first):
    carol@corp.com — sent 3 days ago — no reply
    Suggested action: Send gentle follow-up (Playbook: FOLLOWUP-01)
```

## Playbooks

Playbooks are named, versioned response templates. The Sherpa selects the appropriate playbook for each message type. Playbooks live in the outreach data room and can be edited without code changes.

### Playbook Structure

```
📁 Data Room: "User Outreach"
└── 📁 playbooks/
    ├── ONBOARD-01__initial-access-email.md
    ├── ONBOARD-02__onboarding-follow-up.md
    ├── FAQ-DATAROOM__data-room-explainer.md
    ├── FAQ-ENCRYPTION__encryption-explainer.md
    ├── FOLLOWUP-01__gentle-nudge.md
    ├── FOLLOWUP-02__final-follow-up.md
    ├── BOUNCE-01__handle-bounced-email.md
    └── UNSUBSCRIBE-01__remove-from-list.md
```

Each playbook is a markdown file containing:
- The response template (with `{{placeholders}}` for personalisation)
- The trigger conditions (when to use it)
- Which agent sections to include (Sherpa only? + Ambassador? + DPO note?)
- The follow-up schedule (when to check again if no reply)

### Core Playbooks (MVP)

| Playbook | Trigger | Agent Sections |
|---|---|---|
| **ONBOARD-01** | Initial welcome email to waiting list user | Sherpa (getting started) + Ambassador (use cases) + DPO (unsubscribe note) |
| **ONBOARD-02** | User replied positively to welcome | Sherpa (next steps) + Ambassador (specific use case for their context) |
| **FAQ-DATAROOM** | User asks about data rooms | Sherpa (direct answer) |
| **FOLLOWUP-01** | No reply after 5 days | Sherpa (gentle check-in) |
| **BOUNCE-01** | Delivery failure | Mark in CRM, no reply |
| **UNSUBSCRIBE-01** | User requests removal | Remove from CRM, confirm deletion |

## CRM State in the Data Room

Every user interaction is tracked in the outreach data room. The Sherpa reads and writes to this data room as part of every workflow step.

```
📁 Data Room: "User Outreach"
├── 📄 users.csv              ← Master list: email, name, source, status, last_contact
├── 📁 contacts/
│   ├── 📁 alice-smith/
│   │   ├── 📄 profile.json   ← Name, email, source (LinkedIn/waiting list), language, notes
│   │   ├── 📄 timeline.json  ← Every action: sent, received, status changes
│   │   └── 📄 emails/
│   │       ├── 📄 2026-02-25__onboard-01-sent.md
│   │       └── 📄 2026-02-26__reply-received.md
│   └── 📁 bob-dev/
│       └── ...
└── 📁 playbooks/
    └── ...
```

The `users.csv` is the master view (see companion brief: Data Room CSV Viewer) — the human can open it in the browser, see all users, their status, and add/edit entries in a spreadsheet-like interface.

The `contacts/` folder is the per-user detail — a mini-CRM with full interaction history. When PKI is available, sensitive notes and correspondence can be stored encrypted in the user's vault section.

## n8n Triggers (Future Automation)

Initially, the Sherpa operates on-demand (human asks → Sherpa checks → human approves → Sherpa acts). As playbooks mature and prompt injection risks are mitigated, n8n can trigger the workflow automatically:

| Trigger | n8n Workflow | Human Approval |
|---|---|---|
| New email arrives | Check category → prepare response → notify human | Required (for now) |
| User reply detected | Pull playbook → draft response → notify human | Required |
| Follow-up timer fires | Draft follow-up → notify human | Required |
| Bounce detected | Mark in CRM → archive | Auto (low risk) |
| Unsubscribe request | Remove from CRM + confirm | Auto (legal requirement) |

The approval step is always a human decision for now. The Sherpa prepares; the human executes. Automation of the approval step is a future milestone, gated on security review.

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | Morning standup: Sherpa lists inbox, reads new messages, categorises, reports status |
| 2 | Sherpa proposes playbook action for each message, human approves before execution |
| 3 | Playbook library in data room: at least ONBOARD-01, FAQ-DATAROOM, FOLLOWUP-01, BOUNCE-01, UNSUBSCRIBE-01 |
| 4 | CRM state written to data room after every action (users.csv + contact folder) |
| 5 | Per-user contact folder created on first interaction |
| 6 | Unsubscribe requests handled immediately and automatically (legal requirement) |
| 7 | Prompt injection mitigations in place before any automated execution (see security brief) |
