# Arch Brief: MCP Segmentation — SG/Send MCP vs Vault MCP

**version** v0.7.1  
**date** 26 Feb 2026  
**from** Human (project lead)  
**to** Architect (lead), Developer, AppSec  
**type** Architecture brief — MCP segmentation, immediate design decision  

---

## The Problem

During a Claude session on 26 Feb 2026, an agent accessed via the SG/Send MCP attempted to interact with the vault. This is not the intended behaviour — the MCP connector for file transfer should not expose vault operations. The agent was confused because the current boundary between "send" and "vault" is not explicit in the MCP layer.

This is a capability creep problem. If agents can reach the vault through the send MCP, then any agent with send permissions has vault permissions. That is not the intended access model.

---

## The Intended Segmentation

```
SG/Send MCP                     Vault MCP
──────────────────────────────   ──────────────────────────────
File transfer operations:        Vault operations:
  transfers_create               vault_list
  transfers_upload               vault_read
  transfers_complete             vault_write
  transfers_download             vault_delete
  transfers_check-token          vault_share
  info_*                         vault_permissions

Used by: any agent or user        Used by: admin-driven, specific
that needs to send/receive files  agents only, not publicly exposed
```

These are two different MCP connectors with two different authentication models, two different capability sets, and two different trust levels.

---

## Why This Matters

### Security

The vault stores private keys, personal data, sensitive documents, and CRM data. An agent that can read vault contents has access to all of that. The principle of least privilege requires that agents only have the capabilities they need for their specific task.

An agent running the outreach workflow needs: `send_email`, `read_data_room`. It does not need `vault_read`. If the vault is reachable via the send MCP, that agent implicitly has vault access — which is wrong.

### Clarity for Agents

When an agent sees a vault operation in its available tool list, it will try to use it. This is not a bug in the agent — it's a configuration problem. If vault operations should not be available, they should not be in the tool list. The cleanest solution is a separate connector that is only granted to agents that explicitly need it.

### Admin-Driven Access

The vault MCP is not something users create. It is provisioned by an admin for specific agents. This mirrors the pattern used for admin API access in the current architecture: admin-driven, scoped, audited.

## Architecture Decision

### Two Connectors, Two Auth Models

```
Connector 1: SG/Send Transfer MCP
  URL: mcp.send.sgraph.ai/transfer (or equivalent)
  Auth: Transfer token (current model — user-created, scoped to transfers)
  Capabilities: File transfer only
  Who gets it: Any user, any agent that needs to send/receive files
  Provisioned by: User self-service

Connector 2: SG/Send Vault MCP  
  URL: mcp.send.sgraph.ai/vault (separate endpoint)
  Auth: Vault access key (admin-provisioned, scoped to specific vault)
  Capabilities: Vault read/write/manage only
  Who gets it: Admin-provisioned agents only, not public
  Provisioned by: Admin only
```

The current `SKILL.md` connector (`send.sgraph.ai` MCP) maps to Connector 1. It stays exactly as it is. Nothing breaks.

Connector 2 is new. It does not exist yet. It is what the Sherpa needs to access the data room / vault to read the email list (the workflow described in today's brief).

### Connector 2: Vault MCP for Sherpa

The immediate use case: the Sherpa needs to read `users.csv` from the outreach data room. Today, the human copies and pastes data into the chat window. With Vault MCP, the Sherpa can:

```
vault_read(vault_id="outreach-data-room", path="users.csv")
→ Returns the CSV content

vault_write(vault_id="outreach-data-room", path="users.csv", content="...")
→ Updates the CRM after a send
```

The Sherpa is provisioned with a **symmetric vault key** for the outreach data room. This key is:
- Generated by the admin (human)
- Stored in the Sherpa's configuration (not in the chat window)
- Scoped to the outreach data room only — not all vaults
- Revocable: admin deletes the key mapping

This is exactly the symmetric data room model from the symmetric-key-data-room-mvp brief (v0.6.30) — the Sherpa is a "user" of the data room with an Editor role.

## Migration Path

### Phase 1 (now): Explicit separation in documentation and agent configuration

- Document that the SG/Send MCP is for transfer only
- Remove vault-related tools from the SG/Send MCP tool list if they are currently exposed
- Update `SKILL.md` to explicitly state: "This connector does not provide vault access"
- Update agent system prompts to clarify the boundary

### Phase 2 (next sprint): Vault MCP endpoint

- Create `mcp.send.sgraph.ai/vault` (or equivalent) as a separate FastAPI router / Lambda function
- Implement vault read/write/list operations behind vault-key auth
- Provision the Sherpa with a symmetric vault key for the outreach data room
- Test: Sherpa reads `users.csv` via vault MCP, updates status after send

### Phase 3 (future): PKI-auth vault MCP

- Replace symmetric key auth with PKI: agent proves identity with its private key
- The vault MCP checks the agent's public key against the vault's ACL
- Per-agent, per-vault, cryptographically enforced access control
- Consistent with the PKI roadmap across the platform

## Capability Map: What Each Connector Exposes

### SG/Send Transfer MCP (Connector 1) — current

| Tool | Description | Available to |
|---|---|---|
| `transfers_create` | Create a new encrypted transfer | Any authenticated user |
| `transfers_upload` | Upload encrypted payload | Any authenticated user |
| `transfers_complete` | Mark transfer complete | Any authenticated user |
| `transfers_download` | Download encrypted payload | Any authenticated user |
| `transfers_check-token` | Validate access token | Any authenticated user |
| `info_health` | Platform health check | Any |

### Vault MCP (Connector 2) — to be built

| Tool | Description | Available to |
|---|---|---|
| `vault_list` | List files in a vault/data room | Provisioned agents only |
| `vault_read` | Read a file from the vault | Provisioned agents only |
| `vault_write` | Write/update a file in the vault | Provisioned agents with Editor+ role |
| `vault_delete` | Delete a file | Provisioned agents with Admin role |
| `vault_share` | Share a vault with a new agent | Admin only |
| `vault_revoke` | Revoke access | Admin only |
