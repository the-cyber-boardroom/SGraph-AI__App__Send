# Dev Brief: Data Room CSV and Spreadsheet Viewer

**version** v0.7.1  
**date** 26 Feb 2026  
**from** Human (project lead)  
**to** Developer (lead), Designer  
**type** Dev brief — immediate implementation  

---

## Context

The data room is becoming a two-way communication channel between the human (working in the browser) and agents (running in the cloud environment via MCP). For this to work practically, the human needs rich ways to read and edit the data the agents produce — and the most natural format for the kind of tabular data we're working with (user CRM, email status tracking, outreach lists) is a spreadsheet.

This is not a full spreadsheet application. It is a focused, practical CSV viewer and editor built into the data room that makes managing tabular data as natural as opening a file in Excel or Google Sheets.

---

## The Problem

Currently, a CSV file in a data room renders as raw text or a basic code block. To edit it, a user would need to download it, edit it externally, and re-upload. This is a blocking friction for the outreach workflow: the human needs to be able to open `users.csv`, add five email addresses, tick a checkbox, and save — all inside the browser, without leaving the data room.

Agents write CSV files to the data room. Humans need to edit them. The data room is the coordination layer. The viewer makes it functional.

---

## The Viewer: Spreadsheet Mode

When a `.csv` file is opened in the data room, the preview component detects the file type and renders it in spreadsheet mode — not as raw text.

### Basic View

```
📄 users.csv   [Spreadsheet View ▼]   [Edit]   [Export]   [↺ Refresh]

┌──────────────────┬──────────────────────┬────────────┬──────────┬──────────────────────┐
│ Name             │ Email                │ Language   │ Status   │ Notes                │
├──────────────────┼──────────────────────┼────────────┼──────────┼──────────────────────┤
│ Alice Smith      │ alice@example.com    │ en-gb      │ ✅ sent  │ Met at conf          │
│ Bob Dev          │ bob@gmail.com        │ en-us      │ 🕐 ready │                      │
│ Carol Santos     │ carol@company.pt     │ pt-pt      │ 🕐 ready │ Knows the Arch       │
│ Dave Müller      │ dave@corp.de         │ de-de      │ ❌ bounce│                      │
│ Eve Chen         │ eve@startup.com      │ en-gb      │ ✅ sent  │ Followed up 24 Feb   │
└──────────────────┴──────────────────────┴────────────┴──────────┴──────────────────────┘

  Showing 5 of 23 rows   [← Prev]  Page 1 of 5  [Next →]

  [+ Add Row]
```

### Column Types

Not all columns are plain text. The viewer should support typed columns that render appropriate UI elements:

| Column Type | Detection | UI |
|---|---|---|
| **Text** | Default | Plain text input |
| **Email** | Column header contains "email" / values contain `@` | Text input + validation |
| **URL** | Values start with `http` | Clickable link + text input |
| **Status / Enum** | Small set of repeated values | Dropdown selector |
| **Boolean / Flag** | Values are `true/false`, `yes/no`, `0/1` | Checkbox |
| **Date** | Values match ISO date patterns | Date picker |
| **Number** | All values are numeric | Number input, right-aligned |

Column type is inferred from the data — no schema definition required. The user can override if the inference is wrong.

### Editing In-Line

Click any cell to edit it in place:

```
│ Carol Santos     │ carol@company.pt     │ pt-pt      │ [🕐 ready ▼]│ Knows the Arch │
                                                            ↑
                                              Dropdown opens:
                                              ○ ready
                                              ○ sent
                                              ○ bounce
                                              ○ unsubscribed
                                              ○ + Add value...
```

Changes are saved immediately on blur / Enter. The underlying CSV file in the data room is updated. A new version is created (consistent with the versioned editing model from the data room brief v0.6.30 — original is never overwritten, each save creates v2, v3, etc.).

### Adding Rows

```
[+ Add Row] at the bottom of the table → appends a new blank row with all columns empty.
User fills in the fields inline.
Save: creates a new version of the CSV.
```

For the outreach workflow, this is the core interaction: the human opens `users.csv`, clicks `[+ Add Row]`, types in a name and email address, sets the language and status, and saves. The agent picks it up on the next check.

## Filtering and Sorting

For larger lists (100+ users), filtering and sorting become essential:

```
[🔍 Filter...]   [↕ Sort by: Status ▼]   [👁️ Columns ▼]

Filter panel:
  Status:   [All ▼]  → filter to: ready / sent / bounce / etc.
  Language: [All ▼]  → filter to: en-gb / pt-pt / de-de / etc.
  Search:   [          ]  → text search across all columns
```

Filtering is client-side (the full CSV is loaded). For files under ~5,000 rows this is fast and responsive. Larger files need pagination + server-side filtering — flag as a future enhancement.

### Quick Filters for the Outreach Workflow

The most common filtering actions should be one click:

```
Quick filters:  [📬 Ready to contact]  [🕐 Awaiting reply]  [❌ Bounced]  [✅ Sent]
```

These are essentially saved filter presets for the status column — the four states the Sherpa uses in its morning report.

---

## Column Management

Users can show/hide columns and reorder them. This is especially useful when the agent adds columns (e.g., `last_contact`, `reply_received`, `follow_up_sent`) that the human doesn't need visible at all times.

```
[👁️ Columns ▼]
  ✅ Name
  ✅ Email
  ✅ Status
  ✅ Notes
  ☐ language       (hidden — inferred from email domain, not needed in view)
  ☐ last_contact   (hidden — shown in contact folder, not in the table)
  ☐ source         (hidden — all from waiting list in this batch)
```

Column visibility is stored in localStorage per file — a persistent preference, not part of the CSV itself.

## Issues-FS Visualisation (Future Enhancement)

The transcript notes that the mini-CRM inside the data room might eventually use Issues-FS as the underlying data model, with the spreadsheet viewer adding visualisation capabilities. This is flagged here but not in scope for the current implementation.

When it arrives, the spreadsheet viewer becomes a visual front-end for Issues-FS: each row is an issue, columns are fields, status is the issue state. The same viewer component works for both flat CSV files and structured Issues-FS data — the rendering layer is the same; only the data source changes.

---

## Export

```
[Export ▼]
  → Download as CSV
  → Copy as Markdown table
  → Copy as JSON array
```

Export is always from the current view (with active filters applied) or the full file. User chooses.

## Connection to Existing Architecture

| Document | Connection |
|---|---|
| **Data Room Preview and Versioned Editing** (v0.6.30) | The CSV viewer is the spreadsheet mode of the preview component. Every save creates a new version — same immutable versioning model. |
| **Sherpa Inbox Workflow** (v0.7.1) | The `users.csv` file is the human's interface to the CRM the Sherpa reads and writes. This viewer is what makes that coordination practical. |
| **User Outreach** (v0.7.1) | Adding users to the outreach list = adding rows to `users.csv` in spreadsheet mode. Checking status = filtering by status column. |
| **Static Assets / CloudFront** (v0.6.30) | The spreadsheet viewer is a client-side component loaded from `static.send.sgraph.ai`. No Lambda invocations for the rendering. |
| **Zero Lock-In** (v0.6.14) | The underlying data is always a plain CSV. The viewer is a convenience layer — users can always download the CSV and open it in Excel. |

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | CSV files in the data room render in spreadsheet mode (table with rows and columns) |
| 2 | Column types inferred: text, email, URL, enum/status, boolean, date, number |
| 3 | Enum/status columns render as dropdowns with existing values as options |
| 4 | Boolean columns render as checkboxes |
| 5 | In-line cell editing: click cell → edit → blur/Enter → save |
| 6 | Each save creates a new version (original CSV never overwritten) |
| 7 | [+ Add Row] appends a new blank row, saves new version on confirm |
| 8 | Column show/hide persisted in localStorage per file |
| 9 | Quick filter presets for status column |
| 10 | Free-text search across all columns |
| 11 | Sort by any column (asc/desc) |
| 12 | Export: CSV download, Markdown table copy, JSON array copy |
| 13 | Works on mobile (touch-friendly cell editing) |
| 14 | Performance: files up to 1,000 rows render without lag |
