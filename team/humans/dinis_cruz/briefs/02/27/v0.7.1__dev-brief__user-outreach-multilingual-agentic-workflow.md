## The Email: Multi-Agent, Multi-Voice

Each outreach email is written by multiple agents. The human provides the opening context. Each agent contributes their section. The result is an email that is personal, substantive, and demonstrates the agentic nature of the project directly.

### Email Structure

```
From: Sherpa <sherpa@sgraph.ai>
Reply-To: sherpa@sgraph.ai
Subject: Your SG/Send access — and a bit about how we built this

Hi [Name],

[HUMAN SECTION — 2-3 sentences]
  Personal note from the project lead. Thanks for being on the list.
  What's been happening since they signed up (brief — 1-2 milestones).
  Why now is the right time to try it.

---

🧭 From the Sherpa — Getting Started

[SHERPA SECTION — 3-4 sentences]
  Here is your access code: [one-time secret link]
  Here is where to start: [direct link to send.sgraph.ai + getting started trail]
  Here is where to ask questions: [reply to this email]
  Here is where to give feedback: [GitHub / feedback form]

---

🌍 From the Ambassador — What You Can Do Today

[AMBASSADOR SECTION — 3-4 sentences]
  Specific use case relevant to this user's context (if known — human provides angle)
  OR the top 2-3 use cases most relevant to our audience.
  The zero-lock-in principle: you're not dependent on us. Standard tools work too.

---

📋 From the GRC — Your Data

[GRC/DPO SECTION — 2 sentences, always included]
  What we store about you (nothing meaningful — the transparency panel proves it).
  If you don't want further emails: reply with "unsubscribe" and we'll remove you immediately.

---

[Footer]
  SGraph Send — zero-knowledge encrypted file sharing
  send.sgraph.ai | github.com/sgraph-ai/sg-send
  Built by one human and 18 AI agents. The code is public.
```

### Why the GRC/DPO Section Is Non-Negotiable

Every email must include the unsubscribe mechanism in plain language. This is both a legal requirement (GDPR, CAN-SPAM) and an ethical one. The user signed up for a waiting list — not for an ongoing newsletter. The DPO section makes the unsubscribe path obvious.

## Personalisation

### Per-User Context

The human maintains a CSV in the outreach data room with a `notes` column per user. This is where the human captures the personal angle — what they know about this person, why they signed up, what use case is most relevant.

The Sherpa reads this context when generating the email. The Ambassador section adapts to the user's probable use case. The human section opener is drafted with the specific user in mind.

### Language Detection

For each user, attempt to detect their preferred language:
1. Check CRM `language` field if set by human
2. Infer from email domain (`.pt` → Portuguese, `.de` → German, `.jp` → Japanese)
3. Infer from LinkedIn profile language if available
4. Default: English (`en-gb`)

Multilingual emails use the same multi-agent structure, translated per agent section. The Sherpa section is in the user's language. The footer links to language-specific docs where available.

**Priority languages for first batch** (based on expected user base): English, Portuguese (pt-br + pt-pt), Spanish, German. Other languages follow based on actual sign-up distribution.

## Access Code Delivery

The access code (data room key or platform invite) is NOT pasted into the email body. It is delivered via a one-time secret link:

```
Sherpa section:
  "Here is your personal access link. It will work exactly once — 
   click it, save the code, and it's yours:
   https://send.sgraph.ai/s/xK9mP2qR#..."
```

This demonstrates SG/Send in the act of onboarding. The first thing a new user does is experience zero-knowledge one-time secret delivery. The platform markets itself through the onboarding flow.

## The Data Room as Coordination Layer

The outreach data room is the shared workspace between the human and all agents involved in this campaign. It contains:

```
📁 Data Room: "User Outreach"
├── 📄 users.csv              ← Master list (edited by human in browser)
├── 📁 contacts/              ← Per-user interaction history
├── 📁 playbooks/             ← Response templates per scenario
├── 📁 drafts/                ← Emails drafted but not yet approved
├── 📁 sent/                  ← Approved and sent emails (for reference)
└── 📁 analytics/
    └── 📄 metrics.json       ← Opens (estimated), replies, conversions
```

The human accesses this via the browser. The Sherpa (running in Claude's cloud environment) also accesses it via MCP — reading user context, writing drafts, updating status. The data room is the coordination layer between human and agent.

## Workflow Step-by-Step

### Batch Preparation (Human)

1. Human opens `users.csv` in the data room browser view (spreadsheet mode)
2. Human selects users for this batch (e.g., 10 users — first wave)
3. Human adds personal notes in the `notes` column for users they know
4. Human sets the `language` field where known
5. Human marks selected rows as `status: ready_to_contact`

### Email Generation (Sherpa + Agents)

6. Sherpa reads `users.csv`, filters `status: ready_to_contact`
7. For each user:
   a. Sherpa generates the Sherpa section (in user's language)
   b. Ambassador generates the Ambassador section (tailored to user context)
   c. GRC section is fixed template (translated)
   d. Human section opener is drafted (human reviews and personalises)
8. Full draft saved to `drafts/` folder in the data room

### Human Review

9. Human opens each draft in the data room
10. Human edits the human section opener for each user
11. Human generates the one-time secret link for the access code
12. Human marks each draft as `approved`

### Send

13. Sherpa sends approved emails via AWS SES
14. Updates CRM: `status: contacted`, `last_contact: [date]`
15. Moves email from `drafts/` to `sent/`

### Follow-Up

16. After 5 days with no reply: Sherpa proposes a follow-up (Playbook: FOLLOWUP-01)
17. Human approves → Sherpa sends
