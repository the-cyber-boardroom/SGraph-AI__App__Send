# Design Brief: Physical Card Redesign — Non-Technical User Acquisition

**version** v0.7.1  
**date** 26 Feb 2026  
**from** Human (project lead)  
**to** Designer (lead), Ambassador  
**type** Design brief — print-ready SVG cards, batch production  

---

## Context and Problem

Physical cards were produced in a previous sprint (see current cards photo). They work well as artefacts and have been used successfully. But two fundamental problems prevent them from being effective user acquisition tools at scale:

**Problem 1 — Wrong audience.** The current back reads: *"Zero-knowledge encrypted file sharing. Your files are encrypted in your browser before they leave your device."* This is correct and accurate. It is also completely meaningless to the person you want to hand these to. A security professional already knows what zero-knowledge means. The people we need to acquire don't — and that language makes them feel like this isn't for them.

**Problem 2 — Handwritten access token.** At 50–100 cards per batch, handwriting each token is not viable. The token needs to be printed. The design must accommodate a pre-printed token with an optional per-batch postfix.

The objective is simple: hand someone this card, say nothing, and they become a new user. Ideally they share it with someone else. If the card requires any explanation from the person handing it out, the card has failed.

---

## Target Audience

**Not:** security professionals, developers, privacy researchers. They're already in.

**Yes:** anyone who has ever sent a file and been vaguely uncomfortable about it. Anyone who has emailed a document to a solicitor, a CV to a recruiter, a scan of their ID to a bank, a photo to a family member. Everyone.

The mental model to target: *"This is like WeTransfer, but nobody can see what you're sending — not even us."* That sentence should be on the card. It is accurate, relatable, and explains the core value in terms people already understand.

## The Token System

### Printed Token

The access token is pre-printed on the card. No handwriting.

The token format is designed to be:
- Memorable (a real English word or short phrase)
- Easy to type (no ambiguous characters: no O/0, no l/1/|)
- Appropriate for sharing (safe to say out loud, post on social media, tell a friend)
- Batch-specific (optional postfix to distinguish print runs)

```
Base token:    CHISWICK
With postfix:  CHISWICK-A   (first print run)
               CHISWICK-B   (second print run, different event or location)
               CHISWICK-26  (date-coded)
```

The token format on the card: printed large, clean, in a monospace font. Same visual style as the current handwritten version but set in type. The postfix (if used) is printed smaller alongside.

### Why Tokens Are Safe to Share

The card explicitly tells recipients to share the token. This needs to be true — the token must be a symmetric data room key or access code that is designed for sharing. Multiple people using the same token = multiple DRU events = exactly what we want.

The card messaging: *"Share this code with someone you trust."*

The security model: the token gives access to a shared data room or allows creating an account. It is not a private key. Sharing it is the intended use.

## Card Specifications

**Format:** Standard business card — 85mm × 55mm (UK/EU standard)  
**Sides:** Two (front and back)  
**Production:** SVG source files, print-ready, bleed and crop marks included  
**Colours:** Two-colour preferred (reduces print cost for batch runs) — black + one brand accent. Review SG/Send brand colour (current teal/blue) for suitability.  
**Typography:** Consistent with existing SG/Send brand — clean, modern, readable at small sizes  
**Batch size:** Designed for 50–100 unit print runs at a time

## Front of Card

**Objective:** Immediate recognition, curiosity, a clear call to action. Five seconds to capture attention.

### Hierarchy

```
[Logo mark — SG/Send, small, top corner]

[HEADLINE — large, dominant]
"Send anything.
 Privately."

[SUBHEAD — one line]
Like WeTransfer, but only you can see what's inside.

[CTA — prominent]
Try it free →
send.sgraph.ai

[Token area — bottom, clean]
YOUR CODE
CHISWICK
```

### Design Direction

- Minimal. White or very light background.
- The headline is the largest element. Nothing competes with it.
- The URL and code are easy to read and easy to remember.
- No technical language anywhere on the front.
- No mention of encryption, zero-knowledge, AES, PKI, or any technical term.
- The logo mark anchors the brand without dominating.

### Alternative Headlines to Test

The Designer should prepare 2–3 variants for the headline. Suggestions to evaluate:

| Variant | Tone |
|---|---|
| "Send anything. Privately." | Direct, confident |
| "Your files. Your keys. Nobody else." | Ownership-focused |
| "Files that only the recipient can open." | Outcome-focused |
| "Like WeTransfer. But encrypted end-to-end." | Comparison-based |

The winning variant should be tested by showing it to someone non-technical and asking: "What do you think this does?" If they get it in 10 seconds, it works.

## Back of Card

**Objective:** One person who received this card passes it to someone else, or tells them about it. The back is the "why should I bother" and the "tell a friend" section.

### Hierarchy

```
[SECTION 1 — What it does, in plain language]

"Send files to anyone. 
 We can't read them. 
 They can't be intercepted. 
 No app needed."

[SECTION 2 — Simple analogies, very short]

"Think of it like a locked box.
 You put a file in. 
 Only the person you send it to has the key.
 Not even us."

[SECTION 3 — Share the card / viral mechanic]

"Share this code with a friend →
 They get their own private space, no sign-up."

YOUR CODE
CHISWICK

[FOOTER — brand anchor]
send.sgraph.ai   SG/Send
```

### Tone

- Short sentences. No jargon.
- Every sentence should pass the "would my mum understand this" test.
- The analogy section is important — it gives people a mental model they can repeat when recommending the product. *"It's like a locked box"* is repeatable. *"Zero-knowledge AES-256-GCM encryption"* is not.
- The share mechanic is explicit: the card tells you to give it to someone. This is not an afterthought — it is on the card because it is the intended behaviour.

### What to Avoid on the Back

| Avoid | Why |
|---|---|
| "Zero-knowledge" | Meaningless to target audience |
| "AES-256-GCM" | Causes anxiety, not confidence |
| "End-to-end encrypted" | Better than zero-knowledge but still jargon for many |
| Long paragraphs | Won't be read |
| QR code (optional) | Adds cost, may not add conversion — test in a later batch |

## Viral Mechanic: The Shared Token

The card's viral mechanic relies on the token being shareable. The back of the card should make this explicit:

```
"This code works for you AND your friends.
 Pass the card on, or just share the code: CHISWICK"
```

This creates a network effect: one card handed out → recipient uses the code → recipient tells a friend → friend uses the same code → two users from one card.

The token tracking on the platform side: each batch token is distinct (CHISWICK, VICTORIA, CAMDEN, etc. — all London neighbourhoods, memorable and printable) so the platform can track which batch a user came from. This is DRU attribution.

## Success Criteria

The card works if:
1. A non-technical person picks it up, reads it, and goes to send.sgraph.ai within 24 hours
2. At least one person who received a card passes it to someone else
3. Batch tokens are trackable — we can see which cards drove sign-ups

The card has failed if:
1. Someone needs to ask "what does zero-knowledge mean?" to understand what it does
2. The token can't be printed cleanly without design changes between batches
3. The printing cost per card makes batches of 50–100 uneconomical
