## Context

We have a product (`send.sgraph.ai`) and a project (the agentic development model). We don't yet have a home for the parent brand — the place we point investors, new users, curious developers, and people who want to understand what SGraph AI is and how it works.

`sgraph.ai` needs to exist. It is currently the missing front door.

## Architecture

### Static Site, Published to S3/CloudFront

`sgraph.ai` is a pure static website. No backend. No Lambda. No database.

```
Source: /sgraph-ai-website/ (new folder in the current repo)
Technology: Jekyll (or equivalent static site generator — see below)
Build: CI pipeline generates static HTML/CSS/JS
Deploy: Published to S3 bucket → served via CloudFront
URL: sgraph.ai (root domain)

Consistent with the static asset architecture from the URL brief (v0.6.30):
  - S3 for storage
  - CloudFront for distribution
  - No Lambda for page serving
```

### Technology Choice

Jekyll is the suggested starting point because:
- Zero JavaScript required for basic rendering
- Markdown content files — easy for agents to update
- GitHub Pages compatible (familiar pattern)
- Large template ecosystem

Alternatives to evaluate: Hugo (faster build), Astro (better component support), or plain HTML (simplest possible). The Architect makes the final call. Whatever is chosen must: (a) produce pure static output, (b) be deployable to S3 without a build server long-term, (c) be editable by agents writing markdown files.

### Separation from send.sgraph.ai

`sgraph.ai` and `send.sgraph.ai` are separate sites, separate repos (or separate folders in the same repo), separate deployments. This is intentional:

| Site | Purpose | Technology | MCP Access |
|---|---|---|---|
| `send.sgraph.ai` | The SG/Send product | FastAPI + Lambda + static assets | SG/Send MCP (file transfer only) |
| `sgraph.ai` | The brand/project site | Pure static | None needed |

No feature from the product bleeds into the brand site and vice versa.

## Agent Directory: Public Key Discovery

Every agent has a page. Every agent page has a machine-readable public key. This enables:
- Users to encrypt messages for a specific agent
- Other agents to look up a peer's key for encrypted coordination
- Investors/auditors to verify agent identity

### Per-Agent Page Structure

```
sgraph.ai/agents/sherpa/

  SHERPA
  ──────────────────────────────────────────────
  Role:        Guide and user advocate
  Inbox:       sherpa.explorer@sgraph.ai
  Version:     Active since v0.5.x

  Public Key (Ed25519):
  ┌──────────────────────────────────────────┐
  │  ssh-ed25519 AAAA...                     │
  │  [Copy]  [Download .pub]                 │
  └──────────────────────────────────────────┘

  PGP Public Key:
  ┌──────────────────────────────────────────┐
  │  -----BEGIN PGP PUBLIC KEY BLOCK-----    │
  │  ...                                     │
  │  [Copy]  [Download .asc]                 │
  └──────────────────────────────────────────┘

  Send an encrypted message:
  [Send via SG/Send →]  (links to send.sgraph.ai with key pre-populated)
```

### Machine-Readable Key Directory

A JSON endpoint for programmatic key discovery:

```
sgraph.ai/agents/keys.json

{
  "agents": [
    {
      "role": "sherpa",
      "name": "Sherpa Explorer",
      "email": "sherpa.explorer@sgraph.ai",
      "ed25519_public_key": "ssh-ed25519 AAAA...",
      "pgp_fingerprint": "b12a...",
      "pgp_public_key_url": "https://sgraph.ai/agents/sherpa/key.asc",
      "active_since": "v0.5.x",
      "last_updated": "2026-02-26"
    },
    ...
  ]
}
```

This is the agent equivalent of `github.com/{username}.keys` — a predictable, public location for key discovery. Any agent can `GET sgraph.ai/agents/keys.json` to find keys for all other agents.

## Contact Integration: Email Sherpa from Every Page

Every page on `sgraph.ai` has a persistent, low-friction way to send a message to the Sherpa. Not a full contact form — a minimal interaction.

### The Pattern

```
[Footer of every page, always visible]

  Got a question?  [Email the Sherpa ↗]

Clicking opens a lightweight modal:

  ┌─────────────────────────────────────────────┐
  │  Send a message to the Sherpa               │
  │                                             │
  │  Message: [                              ]  │
  │  Your email (optional): [              ]    │
  │  [📎 Attach screenshot]                     │
  │                                             │
  │  [Send]                                     │
  │                                             │
  │  ⓘ Your message goes to sherpa@sgraph.ai   │
  │    Reply will come from the Sherpa.         │
  └─────────────────────────────────────────────┘
```

### Email field is optional but important

If the user provides their email, the Sherpa can reply. If they don't, we receive the feedback but cannot follow up. The UI should gently encourage it:

> "Add your email if you'd like a reply — we read every message."

### Screenshot Attachment

A user reporting a UI issue can attach a screenshot. This is a simple HTML feature: `<input type="file" accept="image/*">`. The screenshot is sent as an attachment in the email to the Sherpa.

**Future enhancement (flagged in transcript):** A "send screenshot of current page" button that uses the browser's `html2canvas` or `MediaDevices.getDisplayMedia()` API to capture a screenshot automatically — the user doesn't need to take a screenshot manually. This is a P2 feature; the manual upload is the MVP.

### Backend

The contact form submits to a lightweight serverless endpoint (Lambda or a simple n8n webhook) that:
1. Receives the form data (message, optional email, optional screenshot)
2. Sends an email to `sherpa@sgraph.ai` via AWS SES
3. CC's the user's email if provided

This is the same SES infrastructure already used for outreach. One additional trigger type.

## Email and Contact Metrics

Every email received via the contact form is a signal. Track:

| Metric | What It Tells Us |
|---|---|
| Total contact form submissions | Volume of user interest / confusion |
| Submissions with user email | Conversion intent (they want follow-up) |
| Submissions with screenshot | UI/UX issues (users struggling with something specific) |
| Submissions by page (referrer) | Which pages generate the most questions |
| Reply rate | Sherpa responsiveness |

These metrics go into the same data room used for user outreach tracking. The Librarian includes contact volume in the weekly ecosystem health report.

## Product Segmentation: What Lives Where

The transcript flags confusion: one Claude agent tried to use the vault via the SG/Send MCP because the boundaries weren't clear. The website is an opportunity to make segmentation explicit — both for users and for agents.

```
sgraph.ai              ← Brand, project, agent directory, contact
send.sgraph.ai         ← SG/Send product (file transfer, data rooms, tools)
static.send.sgraph.ai  ← Static assets for the product (CDN)
docs.send.sgraph.ai    ← Product documentation
team.send.sgraph.ai    ← Team/agent site (internal)
```

Each subdomain has a clear purpose. The agent directory is on `sgraph.ai` because it belongs to the project, not the product. The SG/Send product is on `send.sgraph.ai`. The vault is part of `send.sgraph.ai` but is distinct from the transfer product (see MCP segmentation brief).
