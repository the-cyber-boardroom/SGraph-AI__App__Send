## Purpose

This register documents GRC's risk-based decisions for each identified risk in the SG/Send skill workflow (see AppSec threat model v0.7.1). For each risk, GRC records: the deployment mode it applies to, the decision (Accept / Mitigate / Transfer / Avoid), the rationale, and any conditions attached to the decision.

---

## Deployment Mode Definitions

| Mode | Description |
|---|---|
| **Current (Symmetric, email delivery)** | Claude generates file → AES-256-GCM encrypted → uploaded to SG/Send → link emailed via n8n WorkMail → recipient downloads |
| **Future (PKI)** | Content encrypted with recipient's public key → link is opaque reference → only private key holder can decrypt |
| **Future (PKI + signed)** | PKI encryption + sender signs with their private key → non-repudiation, verified authorship |

## Risk Register

### RISK-001: Decryption Key Transmitted via Email (Symmetric Mode)

**Description:** In symmetric mode, the decryption key is embedded in the URL fragment, which is included in the email body. The email traverses n8n, AWS SES, and the recipient's email provider (Gmail in current workflow). Each is a potential interception point.

**Deployment mode:** Symmetric only  
**GRC Decision:** **ACCEPT** (with conditions)  
**Rationale:**  
The transmission chain is TLS-encrypted throughout. The entities that handle the key (n8n, SES, Gmail) are all operating under their own security and compliance frameworks. The risk is materially lower than alternatives:

| Alternative | Risk level vs current |
|---|---|
| Send file as email attachment (cleartext) | Higher — file fully readable by all mail intermediaries |
| Send via WeTransfer | Similar — WeTransfer has access to all files |
| Send via Dropbox link | Similar — Dropbox has access to all files |
| Send via SG/Send + phone call for key | Lower — key transmitted out-of-band |
| Send via SG/Send + PKI | Lowest — key never transmitted |

Current mode is not the most secure option. It is a significant improvement over common alternatives and acceptable for non-highly-sensitive content.

**Conditions for acceptance:**
1. One-time use enforced — link expires after first download
2. Transfer expiry: 24 hours default
3. n8n execution logs must not retain full request bodies (see DevOps action)
4. Content classification: only use symmetric mode for content that would be acceptable to send via a shared Dropbox link. For higher sensitivity, use PKI.

### RISK-002: Claude/Anthropic Session Log Access

**Description:** Anthropic may log Claude session content, including plaintext file content generated during the workflow and the decryption key constructed in Step 4.

**Deployment mode:** Both (plaintext content); Symmetric only (key in logs)  
**GRC Decision:** **ACCEPT** (with disclosure conditions)  
**Rationale:**  
Anthropic's data handling policies govern this risk. It is outside our control. The risk is acknowledged and disclosed to users.

For the decryption key specifically: one-time use transfers convert this from a persistent risk to a narrow time window. By the time any log review could occur, the transfer may already be consumed.

**Conditions:**
1. DPO to document Anthropic data retention policy in user-facing privacy documentation
2. User guidance: do not use this workflow for content you would not be comfortable disclosing to an AI provider under their standard data handling terms
3. PKI upgrade path: for content where AI provider log exposure is unacceptable, require PKI mode

---

### RISK-003: SG/Send Server Breach

**Description:** A breach of SG/Send infrastructure could expose stored data.

**Deployment mode:** Both  
**GRC Decision:** **ACCEPT** (unconditionally)  
**Rationale:**  
This risk is architecturally neutralised. The server stores only AES-256-GCM encrypted blobs. The decryption key is never transmitted to the server (it travels in the URL fragment, which browsers do not send to servers). A complete server breach — including all databases, all S3 storage, all logs — exposes encrypted blobs and opaque transfer IDs. Without keys, these are computationally indecipherable.

This is not a policy claim. It is a verifiable architectural property. The code is public.

**No conditions required.**

---

### RISK-004: n8n Log Retention of Decryption Keys

**Description:** n8n execution logs may record the full email body, including the download URL containing the decryption key.

**Deployment mode:** Symmetric with email delivery  
**GRC Decision:** **MITIGATE**  
**Rationale:**  
Unlike the SES/Gmail risk (which is inherent to email delivery), n8n log retention is entirely within our control. This risk has a simple, low-cost mitigation.

**Required mitigations (deployment gate — must be in place before this workflow is used for sensitive content):**
1. Configure n8n workflow to not log full webhook request/response bodies
2. Set log retention to minimum viable (7 days or less)
3. Confirm with DevOps that mitigation is in place before this workflow processes content above LOW sensitivity classification

### RISK-005: Recipient Gmail Account Compromise

**Description:** If the recipient's Gmail account is compromised, the attacker has the download URL and can decrypt the file.

**Deployment mode:** Symmetric with email delivery  
**GRC Decision:** **ACCEPT** (with controls)  
**Rationale:**  
This risk is inherent to any email-based key delivery. It is equivalent to the risk of a password reset link, a shared document link, or any other credential delivered via email. The industry standard is to accept this risk with compensating controls.

**Controls:**
1. One-time use: link is invalid after first download — account compromise post-download is a dead risk
2. Short expiry: 24-hour default limits the window during which a compromise is exploitable
3. Content classification: email delivery of decryption key is appropriate for NORMAL sensitivity. For HIGH sensitivity, require PKI or out-of-band key delivery
4. Future: anomalous access detection (multiple attempts, unexpected geography — see AppSec brief)

---

### RISK-006: Non-Repudiation Gap (Symmetric)

**Description:** In symmetric mode, the recipient cannot verify that the file was sent by the stated sender. Anyone with the sender's SG/Send access token could have created the transfer.

**Deployment mode:** Symmetric only  
**GRC Decision:** **ACCEPT** (with user disclosure)  
**Rationale:**  
Non-repudiation is not a requirement for the current use cases (Sherpa sending summaries, credentials, documents to the human). It becomes important for legal and contractual contexts.

**Conditions:**
1. User guidance: symmetric mode transfers do not provide proof of authorship
2. PKI + signing is the required mode for any transfer where sender verification matters
3. GRC to flag use cases that legally require non-repudiation (contracts, regulated disclosures) as requiring PKI mode

---

### RISK-007: Prompt Injection via Transferred File Content

**Description:** A file received via SG/Send and processed by an agent could contain injected instructions.

**Deployment mode:** Both (any agentic receive workflow)  
**GRC Decision:** **MITIGATE**  
**Rationale:**  
Prompt injection is a high-impact risk for agentic workflows. It is addressed in the separate agentic email security brief (v0.7.1). The controls defined there apply equally to file content processing.

**Required mitigations:** See v0.7.1 agentic email security brief. Security Analyst gate is a deployment prerequisite for any automated receive-and-process workflow.

## Sensitivity Classification Matrix

GRC defines the following content sensitivity levels and maps them to permitted transmission modes:

| Sensitivity | Examples | Permitted Modes |
|---|---|---|
| **LOW** | Public documents, marketing materials, anything already public | Symmetric (email delivery) ✅, PKI ✅ |
| **NORMAL** | Internal documents, meeting summaries, non-confidential credentials | Symmetric (email delivery) ✅ with one-time use enforced, PKI ✅ |
| **HIGH** | Personal data, legal documents, financial information, access credentials | Symmetric — only with out-of-band key delivery, PKI ✅ preferred |
| **CRITICAL** | Regulated data (health, financial), legal privilege, M&A | PKI ✅ required, Symmetric ❌ |

The Sherpa and other agents must be configured to classify content before selecting transmission mode.
