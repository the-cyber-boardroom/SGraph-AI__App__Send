# Dev Brief: Lambda Dependency Optimisation — Minimum Viable Code

**version** v0.6.30  
**date** 25 Feb 2026  
**from** Human (project lead)  
**to** DevOps (lead), Developer, Architect  
**type** Dev brief — infrastructure optimisation  

---

## The Problem

The current Lambda deployment pipeline installs each dependency in isolation, zips it individually, and uploads to S3. At runtime, the Lambda downloads these zips from S3 and adds them to the Python path. This worked well initially but is now causing problems:

- **Duplication**: common libraries (MemoryFS, FastAPI, boto3) are included in EVERY individual dependency zip. The same code is downloaded multiple times.
- **Cold start slowdown**: downloading and extracting 5-19MB of dependency zips on every cold start is increasingly expensive.
- **Every megabyte compounds**: at scale (hundreds of thousands of Lambda invocations/month), every byte saved in the dependency payload reduces bandwidth cost AND boot time.

---

## Current Architecture

```
CI Pipeline:
  For each dependency:
    1. Check if dependency zip exists in S3
    2. If not: pip install → zip → upload to S3
    
Lambda Cold Start:
  1. Download each dependency zip from S3
  2. Extract to /tmp/
  3. Add /tmp/ to Python path
  4. Execute handler
```

This was faster than Lambda Layers because: (a) layers also download from S3 (a zip file), (b) layers are harder to publish and change, (c) the human's direct-to-AWS approach minimises overhead. But now the per-dependency isolation is creating bloat.

---

## Phase 1: Consolidate to One Zip Per Lambda

Instead of one zip per dependency, create ONE zip containing ALL dependencies for each Lambda function. This immediately deduplicates shared libraries.

```
Before (current):
  s3://bucket/deps/fastapi-0.115.zip        (3MB — includes uvicorn, starlette, pydantic...)
  s3://bucket/deps/memory-fs-1.2.zip        (2MB — includes boto3, botocore...)
  s3://bucket/deps/fastapi-mcp-0.2.zip      (4MB — includes fastapi AGAIN, mcp, httpx...)
  s3://bucket/deps/openpgp-js-wrapper.zip    (1MB)
  ...
  Total: ~19MB with massive duplication

After (phase 1):
  s3://bucket/deps/lambda-user-v0.6.30.zip  (8MB — all deps, deduplicated)
  s3://bucket/deps/lambda-admin-v0.6.30.zip (7MB — all deps, deduplicated)
  
  Single download. Single extract. No duplication.
```

### Implementation

```python
# Build script (runs in CI or locally)

# 1. Read the combined dependency list for this Lambda
deps = read_config("lambda-user/dependencies.yaml")

# 2. Install ALL dependencies into one directory
pip install --target /tmp/lambda-user-deps/ {all deps}

# 3. Zip the entire directory
zip -r lambda-user-v0.6.30.zip /tmp/lambda-user-deps/

# 4. Upload to S3
s3.upload("lambda-user-v0.6.30.zip", f"deps/lambda-user-v{version}.zip")
```

### Versioning

Dependencies don't change on every deployment. Version the dependency zip separately from the code:

```yaml
# lambda-user/config.yaml
dependency_version: "v0.6.30"  # Only bump when dependencies change
```

The CI pipeline checks: has `dependency_version` changed? If no, reuse the existing zip. If yes, rebuild. This avoids rebuilding dependencies on every code deployment.

The version string also creates a history: `lambda-user-v0.6.28.zip`, `lambda-user-v0.6.30.zip` — a nice record of dependency evolution.

## Phase 2: Remove Non-Essential Files

Python packages include a lot of files that aren't needed at runtime. After installing deps, strip them:

| Remove | Why | Typical Savings |
|---|---|---|
| `*.dist-info/` | Package metadata, not needed at runtime | 5-15% |
| `__pycache__/` | Bytecode cache, regenerated on first import | 5-10% |
| `*.pyc` | Compiled bytecode (regenerated) | 5-10% |
| `tests/` | Test suites shipped with packages | 5-20% |
| `docs/` | Documentation shipped with packages | Variable |
| `*.md`, `*.txt`, `*.rst` | README files | Small |
| `*.pyi` | Type stubs (not needed at runtime) | Small |
| `*.c`, `*.h` | C source files (sometimes shipped with compiled extensions) | Variable |

```bash
# Strip non-essential files after pip install
find /tmp/deps/ -type d -name "__pycache__" -exec rm -rf {} +
find /tmp/deps/ -type d -name "*.dist-info" -exec rm -rf {} +
find /tmp/deps/ -type d -name "tests" -exec rm -rf {} +
find /tmp/deps/ -type d -name "docs" -exec rm -rf {} +
find /tmp/deps/ -name "*.pyc" -delete
find /tmp/deps/ -name "*.pyi" -delete
find /tmp/deps/ -name "*.c" -delete
find /tmp/deps/ -name "*.h" -delete
```

**Test after every removal.** Some packages break if dist-info is removed (rare but possible). Iterate carefully.

## Phase 3: Tree-Shaking — Minimum Viable Code (MVC)

The concept borrowed from C compilation: if a library has 500 functions but we call 5, why ship the other 495?

### The C Linker Analogy

In C compilation, the linker reads the import table (which functions are actually called), then strips all code that's unreachable from those entry points. The final binary contains only the code that's needed.

Python doesn't have a linker. But we can approximate this:

### Approach: Static Analysis + Selective Inclusion

```
1. Identify our entry points (Lambda handlers)
2. Trace all imports from the handlers (recursively)
3. Build a list of every module actually imported
4. Remove all modules NOT in the import tree
```

Tools that can help:
- **modulegraph** / **py2app** — traces Python import graphs
- **pydeps** — visualises dependency trees
- **importlib** — runtime import tracing
- Custom AST analysis of our codebase

### Why This Is Hard (and Why We Iterate)

Python has dynamic imports (`importlib.import_module()`), conditional imports (`try: import X except: import Y`), and plugin systems that break static analysis. This is why we don't attempt a perfect solution — we iterate:

```
Phase 3a: Remove obviously unused top-level packages
  (e.g., if we install boto3 but only use the S3 client, 
   we might not need ALL of boto3's service definitions)

Phase 3b: Remove unused submodules within packages
  (e.g., boto3 includes definitions for every AWS service. 
   We only use S3 and Lambda. Remove the rest.)

Phase 3c: Custom slimming for the biggest offenders
  (Profile the zip: which packages take the most space? 
   Slim those first for maximum impact.)
```

## Measurement: Cold Start Benchmarking

We can't optimise what we don't measure. Set up cold start benchmarking:

### How to Trigger a Cold Start

```python
# Changing any environment variable resets all Lambda instances
# Even a no-op config apply forces cold starts

lambda_client.update_function_configuration(
    FunctionName='sgsend-user',
    Environment={'Variables': {**current_vars, 'COLD_START_TRIGGER': str(time.time())}}
)

# Wait for update to propagate
# Then invoke the function — this WILL be a cold start
```

### What to Measure

| Metric | How | Baseline (before) | Target |
|---|---|---|---|
| **Cold start total** | Time from invocation to first response | Measure now | 50% reduction |
| **Dependency download time** | Time to download zips from S3 | Measure now | 70% reduction (fewer, smaller zips) |
| **Dependency extraction time** | Time to unzip and add to path | Measure now | 50% reduction |
| **Handler execution time** | Time for actual business logic | Measure now | Unchanged (this optimisation doesn't affect logic) |
| **Zip file size** | Total bytes downloaded | Measure now | 50%+ reduction |

### Data Sources

- **Lambda execution logs**: start/end timestamps, billed duration
- **Custom timing**: add timing instrumentation to the boot sequence
- **AWS X-Ray**: if enabled, provides detailed cold start breakdown
- **CloudWatch metrics**: Duration, Init Duration (cold start specific)
- **OSBot AWS**: already has Lambda creation/measurement capabilities — use for controlled testing

### The Benchmark Loop

```
For each phase:
  1. Measure current state (cold start times, zip sizes)
  2. Apply optimisation
  3. Measure again
  4. Record the delta
  5. Ship if improvement confirmed
  6. Roll back if regression
```

## The Goal: Minimum Viable Code (MVC)

Every single byte in the production Lambda should justify its existence. If a file isn't needed for the Lambda to serve requests, it shouldn't be there.

```
Ideal state:
  lambda-user-v1.0.zip: 500KB - 1MB
    Contains: only the Python modules actually imported by the handler
    Excludes: tests, docs, type stubs, unused submodules, metadata
    
Current state:
  Multiple zips totalling: ~19MB
  
Target after all phases:
  Single zip: < 3MB (aspirational: < 1MB)
```

Every megabyte removed = faster cold starts × hundreds of thousands of invocations = real cost savings and real UX improvement.
