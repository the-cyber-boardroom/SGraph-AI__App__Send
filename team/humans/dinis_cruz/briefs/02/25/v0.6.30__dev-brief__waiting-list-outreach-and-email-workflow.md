## The Email: Personalised and Multi-Voice

Each email comes from the human (founder/lead developer) but includes messages from three agents, each covering their angle:

| Agent | Their Section in the Email | What They Cover |
|---|---|---|
| **Sherpa** | "Getting Started" | Here's how to use the platform. Here's where to find guidance. Here's where to give feedback (GitHub repo, feedback form). Links to current documentation and trails. |
| **Ambassador** | "Use Cases and Opportunities" | Example use cases for secure file sharing. Partnership opportunities. How the platform fits into their workflows. |
| **Librarian** | "The Agentic World" | Curious about how this project is built by AI agents? Here's where to learn more. Links to the team site, architecture docs, the agentic development model. |

## Email Infrastructure (MVP)

We don't need a full email platform. We need to send ~10-50 emails and maybe track responses.

### Options Researched

| Option | Pros | Cons | Recommendation |
|---|---|---|---|
| **AWS SES (Simple Email Service)** | Already in our AWS stack. Programmatic. Cheap. Can send from custom domain. | No inbox — send only. Need separate solution for replies. | **Use for sending** |
| **AWS WorkMail** | Full inbox + sending. Web UI. Integrates with SES. | May have been discontinued or reduced. Research current status. | Research availability |
| **SES + S3 for receiving** | SES can route incoming emails to S3 as files. We can process them. | Complex setup. Emails as files in S3 is clunky. | Possible but not MVP |
| **Human sends manually** | Simple. Personal. | Doesn't scale. Can't automate. | Fallback option |
| **Gmail via OAuth** | Full inbox. Familiar. | Human doesn't want to give agent full inbox access (dangerous, even read-only). | **Not recommended** — security risk |

### Recommended MVP

1. **Send via AWS SES** from `hello@sgraph.ai` or `human@sgraph.ai`
2. **Reply-to** set to a monitored address
3. **Templates** generated by Sherpa/Ambassador/Librarian, personalised per recipient
4. **Tracking**: store sent emails metadata in a data room (see below)
5. **Responses**: for MVP, human monitors replies manually. Automate later.

### Security: No Full Inbox Access for Agents

The human explicitly does not want to give agents read access to their personal email inbox. This is wise — an agent with email read access is a major attack surface. The solution:

- Agents compose email content (templates, personalisation)
- Agents can SEND via SES (write-only, specific address)
- Agents CANNOT read incoming email
- Incoming responses go to a monitored address that the human reviews
- Relevant responses can be copied into a data room for agent collaboration

## Data Rooms for Agent Collaboration

The outreach workflow creates a need: where do agents store and share the data they produce (email lists, templates, response tracking)?

**Answer: a data room.**

```
Data Room: "User Outreach"
├── 📁 waiting-list/
│   ├── 📄 users.json           ← Waiting list signups (from website)
│   └── 📄 outreach-status.json ← Who's been emailed, who responded
├── 📁 templates/
│   ├── 📄 welcome-email.md     ← Email template
│   ├── 📄 sherpa-section.md    ← Sherpa's contribution
│   ├── 📄 ambassador-section.md
│   └── 📄 librarian-section.md
├── 📁 responses/
│   ├── 📄 response-alice.md    ← Copied from email
│   └── 📄 response-bob.md
└── 📁 analytics/
    └── 📄 metrics.json         ← Open rates, responses, conversions
```

This is dogfooding: using a data room for the exact kind of team collaboration it's designed for. It also demonstrates the workflow to future users — "this is how WE use data rooms."

---

## Personal Vault as Agent Working Space

The personal vault (already implemented in the browser) could serve as the agent's local working space. Files that need to be processed privately before sharing go in the vault. Files ready for collaboration go in the data room.

**Enhancement needed** (from the PKI workflows memo): the ability to easily move files in and out of the personal vault. A simple UI to:
- See all files in the vault
- Move a file from vault → data room
- Move a file from data room → vault
- Upload a file directly to the vault

This is a small UI enhancement that makes the vault practical for daily use.
