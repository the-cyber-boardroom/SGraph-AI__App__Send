# Dev Brief: URL Architecture, SPA Strategy, and Static Asset Deployment

**version** v0.6.30  
**date** 25 Feb 2026  
**from** Human (project lead)  
**to** Architect (lead), Developer, DevOps  
**type** Dev brief — architecture sprint  

---

## Context

We're moving into a world where URLs matter. Data room links, PKI-friendly URLs, one-time secrets, multi-language support — all need predictable, stable, well-structured URLs. Currently, static assets are served through FastAPI/Lambda, which wastes Lambda invocations and is slow. This needs a dedicated architecture sprint to map out the URL strategy, implement SPA routing, and move static assets to S3/CloudFront.

---

## Part 1: URL Structure

### The Hierarchy

```
https://send.sgraph.ai/{lang}-{country}/{area}/{resource}

Examples:
  send.sgraph.ai/en-gb/send/abc123
  send.sgraph.ai/pt-br/room/project-alpha
  send.sgraph.ai/en-us/tools/ssh-keygen
  send.sgraph.ai/de-de/secret/xK9mP2qR
  send.sgraph.ai/ja-jp/d/b12/quarterly-report
```

### Level 1: Language and Country

The first two path segments encode language and country:

```
/{lang}-{country}/...

  lang:    ISO 639-1 language code (en, pt, de, ja, fr, es, ar, zh...)
  country: ISO 3166-1 alpha-2 country code (gb, us, br, de, pt, ao, cv...)

Examples:
  /en-gb/  → English, United Kingdom
  /pt-br/  → Portuguese, Brazil
  /pt-pt/  → Portuguese, Portugal
  /pt-ao/  → Portuguese, Angola
  /de-de/  → German, Germany
  /de-at/  → German, Austria
  /ja-jp/  → Japanese, Japan
```

**Why both language AND country?** Portuguese in Brazil (pt-br) is different from Portuguese in Portugal (pt-pt) — different terminology, different cultural references, different date formats. The country code gives us room to localise not just language but culture, legal context, and regional conventions.

**Combined as one segment** (`en-gb` not `en/gb`) to keep URLs shorter and avoid an explosion of path depth. One segment for locale, then straight into the functional area.

**Default**: if no locale prefix, detect from browser (`Accept-Language` header) and redirect. `send.sgraph.ai/send/abc123` → `send.sgraph.ai/en-gb/send/abc123` (based on browser locale). Direct links with locale always work without redirect.

### Level 2: Functional Area

```
/{locale}/{area}/...

Areas:
  /send/        → Send/receive files (core product)
  /room/        → Data rooms
  /d/           → PKI-encrypted document delivery (from PKI-friendly links brief)
  /s/           → One-time secrets
  /secret/      → Create a one-time secret (standalone tool)
  /tools/       → Utility tools (ssh-keygen, pgp-keygen, storage-viewer)
  /vault/       → Personal vault
  /admin/       → Admin interface
  /docs/        → Documentation
```

### Level 3: Resource

The resource level is area-specific:

```
/send/{transfer_id}              → View/download a transfer
/room/{room_name}                → Enter a data room
/room/{room_name}/{folder}/{file} → Direct link to a file in a data room
/d/{key_prefix}/{doc_name}       → PKI-encrypted document (from PKI links brief)
/s/{secret_id}#{decryption_key}  → One-time secret
/tools/ssh-keygen                → SSH key generator
/tools/pgp-keygen                → PGP key generator (new)
/tools/storage-viewer            → Browser storage viewer (new)
/admin/{room_name}               → Admin interface for a data room
```

## Part 2: SPA vs Server-Side Routing

Not everything should be a single-page application. Some routes are server-side redirects. Some are static pages. Some are client-side SPA routes.

### Route Classification

| Route Pattern | Routing Type | Why |
|---|---|---|
| `/{locale}/send/{id}` | **Server redirect → SPA** | Server validates transfer ID exists, then redirects to the SPA client which handles decryption/display |
| `/{locale}/room/{name}` | **Server redirect → SPA** | Server validates room exists, redirects to client SPA for the data room experience |
| `/{locale}/d/{prefix}/{name}` | **SPA** | Entirely client-side. The key prefix and doc name are resolved in the browser. |
| `/{locale}/s/{id}#{key}` | **Server + SPA** | Server serves the encrypted blob. Client (SPA) decrypts using the fragment key. |
| `/{locale}/tools/*` | **Static** | Pure static HTML/JS. No server logic needed. Served directly from S3/CloudFront. |
| `/{locale}/docs/*` | **Static** | Documentation pages. Static HTML. S3/CloudFront. |
| `/{locale}/admin/*` | **SPA** | Client-side admin interface. SPA with API calls to the backend. |
| `/api/*` | **Server (FastAPI)** | API endpoints. Always server-side. Never served from S3. |

### Server-Side: Thin Redirect Layer

The Lambda/FastAPI server handles:
1. API endpoints (`/api/*`, `/transfers/*`, `/info/*`)
2. Redirects from old URLs to new SPA URLs
3. Locale detection and redirect (no locale → detect → redirect)
4. Transfer/room existence validation (returns 404 early if resource doesn't exist)

The server does NOT serve static assets (HTML, JS, CSS, images). That's CloudFront's job.

## Part 3: Static Assets to S3 / CloudFront

### The Problem

Currently, all static assets (IFD folder — the frontend code) are served through FastAPI on Lambda. Every CSS file, every JavaScript bundle, every image = a Lambda invocation. This is:
- **Expensive**: Lambda charges per invocation and duration
- **Slow**: Lambda cold starts affect static file serving
- **Wasteful**: Lambda should handle business logic, not file serving

### The Solution

```
Before:
  Browser → CloudFront → API Gateway → Lambda (FastAPI) → serves static file
  Cost: Lambda invocation + API Gateway request + Lambda duration
  Latency: Cold start possible

After:
  Browser → CloudFront → S3 (static files)
  Cost: S3 GET request (fraction of Lambda cost) + CloudFront cache
  Latency: CloudFront edge cache (milliseconds)

  API requests still go through Lambda:
  Browser → CloudFront → API Gateway → Lambda (FastAPI) → API response
```

### Deployment: Versioned IFD Folder to S3

```
CI Pipeline:
  1. Build frontend (IFD folder)
  2. Version the build: ifd-v0.6.30/
  3. Upload to S3: s3://sgsend-static/ifd-v0.6.30/
  4. Update CloudFront to point to new version
  5. Keep previous versions (rollback capability)

S3 bucket structure:
  s3://sgsend-static/
  ├── ifd-v0.6.28/
  │   ├── index.html
  │   ├── app.js
  │   ├── style.css
  │   └── assets/
  ├── ifd-v0.6.30/    ← current
  │   ├── index.html
  │   ├── app.js
  │   ├── style.css
  │   └── assets/
  └── latest/          ← symlink or redirect to current version
```

### This Also Covers Other Sites

We already have multiple sites that should be on S3/CloudFront instead of GitHub Pages:

| Site | Current Hosting | Move to S3/CloudFront |
|---|---|---|
| **send.sgraph.ai (frontend)** | Lambda/FastAPI | ✅ This brief |
| **docs.send.sgraph.ai** | GitHub Pages | ✅ S3/CloudFront |
| **QA dashboard** | GitHub Pages | ✅ S3/CloudFront |
| **team.send.sgraph.ai** | GitHub Pages | ✅ S3/CloudFront |
| **sgraph.ai (root brand site)** | TBD | ✅ S3/CloudFront |

One S3 bucket, one CloudFront distribution, multiple origins or path patterns. All static sites under one infrastructure.

### CloudFront Configuration

```
CloudFront distribution: send.sgraph.ai

  Behaviour 1: /api/*     → Origin: API Gateway (Lambda)
  Behaviour 2: /transfers/* → Origin: API Gateway (Lambda)
  Behaviour 3: /info/*    → Origin: API Gateway (Lambda)
  Behaviour 4: /*         → Origin: S3 bucket (static files)
                             Default: /ifd-v0.6.30/index.html
                             (SPA fallback: all routes → index.html → client routing)
```

### Version Switching

The human needs to control which version of the frontend is loaded. Options:

| Method | How It Works | UX |
|---|---|---|
| **CloudFront default** | CloudFront points to `latest/` which is the current version. Changing the version = updating the CloudFront origin path. | Zero UX — users always get the current version |
| **Local storage flag** | Store `ifd_version` in localStorage. The bootloader checks this and loads the specified version. | Developer/admin tool — set in console or settings panel |
| **Feature flags** (future) | A lightweight feature flag system that controls which version and which features are loaded. | Most flexible. Build later. |

For now: CloudFront default for production, localStorage flag for developers and debugging. Feature flags are the future evolution.

## Part 4: PGP Key Generator Tool

Same pattern as the existing SSH keygen at `/tools/ssh-keygen`. Add a PGP key generator:

```
send.sgraph.ai/tools/pgp-keygen

  Generate PGP Key Pair
  
  Name:  [Alice Smith            ]
  Email: [alice@example.com      ]
  
  Key Type: [Ed25519 ▼]
  Passphrase: [••••••••••        ] (optional)
  
  [Generate]
  
  ── Your Keys ──────────────────────
  
  Public Key:   [Copy] [Download .asc]
  -----BEGIN PGP PUBLIC KEY BLOCK-----
  ...
  
  Private Key:  [Copy] [Download .asc]
  -----BEGIN PGP PRIVATE KEY BLOCK-----
  ...
  
  ⚠️ Your private key was generated in your browser.
  It has NOT been sent to any server. Save it securely.
```

Uses OpenPGP.js. Entirely client-side. Nothing leaves the browser. Same trust model as SSH keygen. Connects to the GitHub-as-registry brief: users generate a PGP key here, publish it on GitHub, and we can use it for encryption.
