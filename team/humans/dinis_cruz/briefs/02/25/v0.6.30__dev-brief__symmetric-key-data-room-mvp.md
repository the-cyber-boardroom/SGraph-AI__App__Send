# Dev Brief: Symmetric Key Data Room MVP — Admin-Managed Keys

**version** v0.6.30  
**date** 25 Feb 2026  
**from** Human (project lead)  
**to** Developer (lead), Architect  
**type** Dev brief — immediate implementation (first data room type)  

---

## Context

We now have a first MVP of the data room. This brief readjusts the implementation to align with the target use case. There will be several types of data rooms (symmetric key, PKI-based, etc.). This brief covers the FIRST type: symmetric key data rooms.

---

## Data Room Types (Roadmap)

| Type | Security Model | Use Case | Status |
|---|---|---|---|
| **Symmetric key** (this brief) | Admin manages keys. Shared decryption key. Admin can access all data. | Team collaboration. Safe group workspace. Not for highly confidential per-user data — for a group that needs a shared secure space. | **Now — MVP** |
| **PKI-based** (future) | Each user has their own key pair. Per-user encryption. Users can't read each other's private data unless explicitly shared. | High-security data rooms. Confidential per-party data. Legal, financial, due diligence. | Future |

The symmetric key data room over-indexes on USABILITY while keeping the existing flow. It's not about per-user confidentiality within the group — it's about a safe, easy way for a group to collaborate.

---

## The Security Model (Honest Assessment)

In this mode, the admin has access to all data. The usernames are not PKI-verified — they're just names. There's no point over-engineering security here because identity is not cryptographically enforced. This is acknowledged and acceptable for the use case.

What we DO provide:
- Encrypted storage (data at rest is encrypted)
- Access control via short decryption keys (admin creates/revokes per user)
- Zero-knowledge server (server stores encrypted blobs, can't read them)
- Audit trail (who accessed when)

What we DON'T provide in this mode:
- Per-user encryption isolation (all users share the same underlying encryption key)
- Verified identity (usernames are labels, not cryptographic identities)
- Protection from the admin (admin can see everything — by design)

## Key Architecture

The underlying encryption uses a long master key. Each user gets a short access key (the ~10-digit friendly key) that decrypts the master key. Revoking a user = deleting their short key's encrypted mapping to the master key.

```
Master Key: AES-256 (the real encryption key for all data room content)
    │
    ├── Admin Key (short, friendly) → encrypts/decrypts the Master Key
    ├── User A Key (short, friendly) → encrypts/decrypts the Master Key
    ├── User B Key (short, friendly) → encrypts/decrypts the Master Key
    └── User C Key (short, friendly) → encrypts/decrypts the Master Key

Each short key is an independent encrypted copy of the Master Key.
Revoking User B = delete User B's encrypted copy.
User B's short key now decrypts nothing.
```

### Key Revocation

**Simple revocation** (delete the mapping): User B's short key no longer resolves to the master key. User B can't access anything going forward. BUT — if User B previously downloaded or cached the master key, they still have it. This is acceptable for the symmetric model.

**Full revocation** (re-encrypt everything): Generate a new master key. Re-encrypt all data room content with the new key. Issue new short keys to remaining users. Old keys (including User B's cached master key) now decrypt nothing. More secure, more expensive. Implement later when needed.

For MVP: simple revocation (delete the mapping). Flag full revocation as a future enhancement.

## Admin Workflow

### 1. Create Data Room

```
Admin goes to admin interface → "Create Data Room"
  ├── Name: "Project Alpha"
  ├── Description: "Shared workspace for the Alpha team"
  └── [Create]

Result:
  ├── Data room created
  ├── Admin's access key generated automatically (first key)
  ├── Admin can immediately log in to the data room with this key
  └── Admin key has admin permissions (create users, manage keys, edit content)
```

### 2. Create Users and Keys

```
Admin goes to data room settings → "Users"
  ├── [+ Add User]
  │   ├── Name: "Bob" (just a label)
  │   ├── Role: Viewer / Editor / Admin
  │   └── [Create]
  │
  │   Result:
  │   ├── Short access key generated for Bob
  │   ├── Key displayed to admin (one-time view, or copy)
  │   └── Admin shares this key with Bob
  │       (via one-time secret link — see companion brief)
  │
  ├── Existing users:
  │   ├── Alice (Editor) — Key: active     [Revoke] [Regenerate]
  │   ├── Bob (Viewer)   — Key: active     [Revoke] [Regenerate]
  │   └── Carol (Editor) — Key: revoked    [Re-enable]
```

### 3. Admin Can Log In As Any User

Since the admin manages all keys and there's no PKI identity enforcement, the admin can log in using any user's key. This is by design — the admin is the data room administrator. For demos and management, the admin needs to verify what each user sees.

### 4. Key Sharing via Secure Send

When the admin creates a user key, they need a secure way to share it. This should use SG/Send's own infrastructure — specifically the one-time secret link feature (see companion brief). The workflow:

```
Admin creates Bob's key → clicks [Share Securely]
  → SG/Send generates a one-time link containing the key
  → Admin sends the link to Bob (via email, WhatsApp, etc.)
  → Bob clicks the link → sees the key → link self-destructs
  → Bob uses the key to access the data room
```

This is dogfooding: using SG/Send to share SG/Send access credentials.

## Permissions Model

| Role | View Files | Upload Files | Edit Files | Manage Users | Manage Keys | Delete Files |
|---|---|---|---|---|---|---|
| **Viewer** | ✅ | ❌ | ❌ | ❌ | ❌ | ❌ |
| **Editor** | ✅ | ✅ | ✅ (creates new version) | ❌ | ❌ | ❌ |
| **Admin** | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |

These map to the roles already implemented in the admin interface (visitors, editors, admins).

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | Admin creates a data room and receives an admin access key |
| 2 | Admin can log in to the data room with the admin key |
| 3 | Admin can create users with roles (viewer, editor, admin) |
| 4 | Each user gets a unique short access key |
| 5 | Admin can revoke a user's key (simple revocation — delete mapping) |
| 6 | Admin can regenerate a user's key (old key stops working, new key issued) |
| 7 | Admin can share keys via one-time secret link (companion brief) |
| 8 | Users can log in with their short key and see the data room |
| 9 | Permissions enforced per role (viewer can't upload, editor can't manage users) |
| 10 | Admin can log in as any user to verify their view |
