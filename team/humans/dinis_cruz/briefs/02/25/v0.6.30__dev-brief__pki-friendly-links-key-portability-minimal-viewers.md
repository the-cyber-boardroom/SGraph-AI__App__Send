## 1. Human-Friendly PKI URLs

### The Problem with Hashes

Current file sharing links look like: `send.sgraph.ai/s/xK9mP2qR#a7f3b2c1d4e5...`

This is functional but unfriendly. You can't read it out loud. You can't remember it. You can't tell from the URL what it contains. It looks intimidating to non-technical users.

### The PKI-Enabled URL Structure

When the recipient has a public key, the URL structure changes fundamentally. The URL is no longer the security mechanism — the encryption is. This means the URL can be human-friendly because it's just a REFERENCE, not a secret.

```
https://send.sgraph.ai/d/{key_prefix}/{document_name}

Examples:
  send.sgraph.ai/d/b12/quarterly-report
  send.sgraph.ai/d/a7f/contract-v3
  send.sgraph.ai/d/b12/hello-world
  send.sgraph.ai/d/b12/meeting-notes-feb-25

Where:
  /d/              ← route for encrypted document delivery
  {key_prefix}     ← first characters of recipient's public key fingerprint
                     (enough to identify which key to use)
  {document_name}  ← human-friendly name chosen by the sender
```

### Why the Key Prefix Matters

Users will have multiple keys (personal, work, project-specific). The key prefix in the URL tells the recipient's browser WHICH private key to use for decryption. Without it, the browser would have to try every key — slow and error-prone.

```
Alice has three keys in her browser:
  🔑 b12... (personal)
  🔑 a7f... (work)  
  🔑 c93... (Project Alpha data room)

Alice receives: send.sgraph.ai/d/a7f/expense-report

Browser sees "a7f" → automatically selects Alice's work key → decrypts.
No guessing, no prompting, instant.
```

### The Security Properties

| Mode | URL is secret? | Who can decrypt? | Link lifespan |
|---|---|---|---|
| **Symmetric** (hash in fragment) | YES — anyone with the full URL can decrypt | Anyone with the link | One-time or short-lived |
| **PKI** (key prefix + doc name) | NO — URL is safe to share publicly | Only the private key holder | Long-lived, permanent |

In PKI mode, you could post the link on a website, print it on a business card, or shout it in a crowded room. Only the intended recipient can decrypt. The link is a pointer, not a key.

## 2. Always Include Sender's Public Key

Every message and file sent through SG/Send should include the sender's public key by default. This creates a viral bootstrapping loop:

```
Bob receives a file from Alice (first contact):

  📄 Quarterly Report
  Encrypted for: Bob (key: b12...)
  
  ────────────────────────────────────
  From: Alice
  🔑 Alice's public key: a7f3b2c1...
  
  [Import Alice's Key]  [Reply to Alice]
  ────────────────────────────────────

Bob clicks [Import Alice's Key]:
  → Alice's public key saved in Bob's browser contacts
  → Bob can now encrypt files for Alice
  → The PKI relationship is established in ONE interaction
```

### The Viral Loop

```
Alice sends Bob a file
  → Bob imports Alice's key
  → Bob sends Carol a file (attaching Bob's key)
  → Carol imports Bob's key
  → Carol sends Dave a file (attaching Carol's key)
  → ...

Every file sent bootstraps a new PKI relationship.
The trust web grows with every message.
```

This should be default-on, with an option to disable for anonymous sends. If the sender doesn't want to reveal their identity, they can turn it off.

## 3. Key Portability via Chrome Password Sync

### The Problem

Private keys are trapped in a single browser on a single device. Moving to a new laptop or browser means losing your identity. The key portability brief (v0.6.14) identified this and proposed password manager integration.

### The Chrome Sync Approach

Chrome's password manager syncs credentials across devices via the user's Google account. The research question: can we use this for private key storage?

**What we know:**
- Chrome password manager stores credentials per-website
- It syncs across all Chrome installations logged into the same Google account
- It supports the Credential Management API for programmatic access
- The limitation: only accessible from the specific origin (send.sgraph.ai) — which is exactly what we want

**The proposed approach:**

```
Key storage:
  Website: send.sgraph.ai
  Username: [public key fingerprint, e.g., "b12-personal"]
  Password: [private key, base64-encoded]

Chrome syncs this across all devices.
When user opens send.sgraph.ai on any device,
Chrome offers to autofill the "password" (which is the private key).
```

**The critical requirement**: the private key must NEVER be in the clipboard. The autofill mechanism passes the value directly to the JavaScript field without clipboard involvement. This avoids the clipboard exfiltration attack vector.

### Research Tasks

| # | Research Question |
|---|---|
| 1 | Can the Credential Management API (`navigator.credentials.get()`) retrieve stored passwords programmatically on send.sgraph.ai? |
| 2 | Does this work across Chrome on desktop AND mobile (Android, iOS)? |
| 3 | Does Safari's Keychain offer equivalent programmatic access? |
| 4 | Does Firefox's password sync offer equivalent access? |
| 5 | Can we store a base64-encoded private key in the password field without Chrome flagging it as suspicious? |
| 6 | Is there a user action required (click, biometric) to retrieve the credential? (This is acceptable — even desirable as a security confirmation.) |
| 7 | If none of the above work seamlessly, do we need a minimal Chrome extension? (Last resort.) |

### Fallback: Password Manager Integration

If Chrome sync doesn't work programmatically, fall back to the password manager integration from the key portability brief (v0.6.14): 1Password, Bitwarden, etc. These have more mature APIs for credential retrieval and are designed for exactly this kind of secure secret management.

## 4. Minimal, Beautiful Viewers

### The Principle

When someone receives a simple secret, a text message, or a single document, the viewer should be BEAUTIFUL. Screenshot-worthy. Minimal. The content is the star — not the UI chrome.

```
┌──────────────────────────────────────────────┐
│                                              │
│                                              │
│      "The staging API key is:                │
│       sk-live-abc123def456"                  │
│                                              │
│      From: Alice                             │
│      🔑 Import Alice's key                   │
│                                              │
│                                              │
└──────────────────────────────────────────────┘

That's it. Nothing else. No navigation bars, no sidebars, 
no menus. Just the content, the sender, and the key import.
```

### The Cognitive Load Principle

Think about the recipient's experience. They clicked a link. They want to see the content. Everything else is noise. The viewer should adapt to the content:

| Content Type | Viewer | What's Shown |
|---|---|---|
| **Short text / secret** | Centred text on clean background | The text. Sender info. Key import. |
| **PDF** | Full-screen PDF renderer | The PDF fills the viewport. Sender info at top. |
| **Image** | Full-screen image | The image fills the viewport. Sender info at top. |
| **Markdown document** | Rendered markdown, clean typography | The document. Sender info at top. |
| **Data room** | Folder tree + preview pane | The data room navigation. |

The current viewers are getting good (PDF rendering is great, new previews are strong). The work here is about REMOVING things — stripping away UI elements that aren't needed for the specific content type. Less is more.

### Multiple Viewer Modes

Implement as separate view templates that the system selects based on content:

```
/view/secret/{id}    ← Minimal text viewer (for secrets, passwords, short text)
/view/doc/{id}       ← Document viewer (for PDFs, markdown, images)
/view/room/{id}      ← Data room viewer (folder tree + preview)
```

Start with the secret viewer — it's the simplest and the most impactful for first impressions. A user's first interaction with SG/Send is often receiving a shared secret or a single file. That first experience must be beautiful.
