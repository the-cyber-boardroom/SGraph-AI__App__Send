# Brief: Per-File Security Review Framework and Threat Modelling

**version** v0.6.30  
**date** 25 Feb 2026  
**from** Human (project lead)  
**to** AppSec (lead), GRC, Architect, Developer, Sherpa  
**type** Brief — security review process, immediate priority  

---

## Context

We've shipped a lot of code in a short time: data rooms, vaults, file viewers, API endpoints, admin interfaces. The human is worried — rightly — that we've accumulated vulnerabilities. Some access tokens have already been leaked in posts (not yet abused, but only because usage is low and assets aren't high-value yet). This won't stay true as the platform grows.

We need to map every vulnerability, every attack surface, every risk — not to fix everything immediately, but to make informed, risk-based decisions about what to address and when.

---

## The Approach: Bottom-Up, File-by-File

Don't try to do a top-down architecture review. Do it bottom-up. Start with individual files and build up to the full picture. The fractal nature of the codebase means each level of zoom reveals more detail:

```
File → Method → Class → Module → Path/Journey → Endpoint → Full Attack Surface
```

### Parallel Security Tree

For every file in the codebase, create a corresponding security analysis file in a parallel tree structure:

```
Main codebase:                    Security analysis tree:
src/                              security/
├── api/                          ├── api/
│   ├── transfers.py              │   ├── transfers.security.json
│   ├── auth.py                   │   ├── auth.security.json
│   └── info.py                   │   └── info.security.json
├── encryption/                   ├── encryption/
│   ├── keys.py                   │   ├── keys.security.json
│   └── symmetric.py              │   └── symmetric.security.json
├── storage/                      ├── storage/
│   ├── s3.py                     │   ├── s3.security.json
│   └── memory_fs.py              │   └── memory_fs.security.json
└── ui/                           └── ui/
    ├── viewer.py                     ├── viewer.security.json
    └── admin.py                      └── admin.security.json
```

### Security File Format

Each `.security.json` (or `.security.yaml` or Issues-FS folder) contains:

```json
{
  "file": "src/api/transfers.py",
  "last_reviewed": "2026-02-25",
  "reviewer": "AppSec",
  "classification": "critical-path",
  
  "attack_surface": {
    "endpoints": [
      {
        "path": "/transfers/create",
        "method": "POST",
        "auth_required": true,
        "input_validation": "partial",
        "rate_limited": false,
        "notes": "Accepts file upload. File size not validated server-side."
      }
    ]
  },
  
  "vulnerabilities": [
    {
      "id": "VULN-001",
      "severity": "medium",
      "description": "File size limit not enforced server-side, only client-side",
      "attack_vector": "Attacker bypasses client, sends oversized file via API",
      "impact": "S3 storage exhaustion, potential cost attack",
      "status": "open",
      "mitigation": "Add server-side file size validation",
      "risk_decision": "Accept for now — low usage, low impact. Fix before public launch."
    }
  ],
  
  "secrets_exposure": {
    "handles_secrets": true,
    "secrets_types": ["encryption_keys", "api_tokens"],
    "secrets_in_memory": true,
    "secrets_logged": false,
    "notes": "Keys are in memory during request processing. Cleared after response."
  },
  
  "dependencies": {
    "external_calls": ["s3", "lambda"],
    "trust_boundary_crossings": ["client → server", "server → S3"]
  },
  
  "redundant_code": {
    "dead_functions": ["legacy_upload_handler"],
    "unused_imports": ["old_crypto_lib"],
    "notes": "legacy_upload_handler was replaced in v0.6.17 but not removed"
  }
}
```

### Why This Approach

| Benefit | Explanation |
|---|---|
| **Incremental** | Don't try to review everything at once. Do one file at a time. Each file reviewed is progress. |
| **Fractal** | A file's security analysis can be expanded: per-method, per-line if needed. Start coarse, zoom in on critical code. |
| **Diffable** | When code changes, diff the security tree against the code tree. New files without security analysis = gap. Changed files = re-review. |
| **Foundation for threat modelling** | Once every file is analysed, the threat model emerges from the bottom up: attack surfaces, trust boundaries, data flows. |
| **Finds dead code** | Mapping every file reveals code that's no longer called. Dead code = hidden attack surface. Remove it. |

## Team Responsibilities

| Role | Responsibility |
|---|---|
| **AppSec** | Lead reviewer. Creates security analysis files. Identifies vulnerabilities. Classifies severity. |
| **GRC** | Risk-based decisions. Accepts or rejects risks. Maps risks to deployment modes. Ensures compliance posture. |
| **Architect** | Helps map dependencies, trust boundaries, data flows. Understands the design intent behind each file. |
| **Developer** | Helps clarify code behaviour. Fixes identified vulnerabilities. Removes dead code. |
| **Sherpa** | Maps the trails (user journeys) and identifies which files are on which trails. Threat model per trail. |

### The Sherpa's Role: Trail-Based Threat Modelling

The Sherpa maps user journeys (trails). Each trail passes through specific files. A threat model per trail shows the attack surface for each user flow:

```
Trail: "Send a file to a new recipient"

  Steps:
  1. User opens send.sgraph.ai         → ui/sender.js
  2. User enters recipient identifier   → api/keys.py (lookup)
  3. User drops a file                  → encryption/keys.py (encrypt)
  4. File uploaded                      → api/transfers.py (create)
  5. Upload to S3                       → storage/s3.py (upload)
  6. Recipient gets link                → api/transfers.py (info)
  7. Recipient downloads                → api/transfers.py (download)
  8. Recipient decrypts                 → ui/receiver.js (decrypt)

  Attack surface for this trail:
  - Input validation at step 2 (recipient identifier)
  - File handling at step 3 (malicious file?)
  - Auth at step 4 (who can create transfers?)
  - S3 permissions at step 5 (bucket policy)
  - Link guessability at step 6 (transfer ID entropy)
  - Rate limiting at step 7 (download abuse)
```

## Risk-Based Decisions by Deployment Mode

The GRC team should map risks to deployment modes, because risk profiles differ:

| Risk | Public Cloud (current) | IP-Restricted Data Room | Private Deployment |
|---|---|---|---|
| Leaked access tokens | Medium — tokens work from anywhere | Low — IP restriction limits exploitation | Very Low — private network |
| Unauthenticated API access | High — anyone can try | Medium — only from allowed IPs | Low — internal only |
| S3 bucket misconfiguration | Medium — public bucket = data leak | Same | Lower — private infrastructure |
| Transfer ID brute force | Low — IDs have sufficient entropy | Very Low — IP restricted | Very Low |

This allows the GRC team to say: "This vulnerability is acceptable in the IP-restricted deployment but must be fixed before public launch."

Before deep analysis, create a simple inventory of every externally-accessible endpoint:

| Endpoint | Method | Auth Required | Input | Rate Limited | Notes |
|---|---|---|---|---|---|
| `/` | GET | No | None | No | Redirect |
| `/send` | GET | No | None | No | Redirect |
| `/tools/ssh-keygen` | GET | No | None | No | Static tool |
| `/info/health` | GET | No | None | No | Health check |
| `/info/server` | GET | No | None | No | Server info — what does this expose? |
| `/transfers/create` | POST | ? | File + metadata | ? | Core endpoint. Review thoroughly. |
| `/transfers/upload/{id}` | POST | ? | File bytes | ? | File upload. Size limits? |
| `/transfers/download/{id}` | GET | ? | Transfer ID | ? | File download. Auth check? |
| ... | ... | ... | ... | ... | ... |

This inventory comes directly from the OpenAPI spec (the Swagger screenshot from v0.6.19). Every endpoint must have its auth, input validation, and rate limiting documented.

## Zero Dead Code in Production

The human's principle: **zero lines of code in production that are not being used.** Dead code is hidden attack surface. It's code that nobody reviews because nobody thinks it runs. But it CAN run if an attacker finds a way to reach it.

The per-file security review should flag dead code. The developer should remove it. The CI pipeline should eventually enforce it (no unused imports, no unreachable functions).
