**version** v0.5.10  
**date** 21 Feb 2026  
**from** Human (project lead)  
**to** Librarian (lead), Developer, all roles  
**type** Operational brief — content structure for agent context loading

## What to Build

Expand the existing GitHub Pages site (currently used by the Journalist) into a comprehensive agent context hub. The site serves two purposes:

1. **Human-readable catalogue**: a browsable site showing all catalogued data, hyperlinked, navigable
2. **LLM-readable context**: structured pages per role that can be fetched via GET and loaded directly into an LLM conversation (for the admin agent chat feature)

## The Key File: `context.md` Per Role

Each role gets a `context.md` — a single consolidated page designed to be loaded into an LLM context window. This is the file the admin agent chat fetches.

### Structure of `context.md`

```markdown
# {Role Name} — Agent Context

## Project Introduction
{Brief project overview — what is SG/Send, current version, key principles.
 Pulled from project/introduction.md. Same across all roles.}

## Role Definition
{The full ROLE.md content for this specific role.}

## Current Focus
{What this role is currently working on. Updated regularly.
 Includes current priorities (P/B/D ratings) for this role's tasks.}

## Recent Briefs and Deliverables
{The last 3-5 briefs/documents produced by or assigned to this role.
 Full content or substantive summaries, depending on context window budget.}

## Key Decisions and Rationale
{Recent decisions this role has made or been involved in,
 with the reasoning behind them.}

## Open Questions and Pending Tasks
{What's unresolved. What's blocked. What needs input from other roles.}

## Relationships
{Which other roles this role works closely with on current tasks.
 Cross-references to their context pages.}
```

### Example: Architect `context.md`

```markdown
# Architect — Agent Context

## Project Introduction
SG/Send (v0.5.10) is an encrypted communication platform providing 
zero-knowledge file transfer and data rooms. Built on a serverless 
AWS stack (Lambda, S3, CloudFront, API Gateway) with browser-based 
PKI (Web Crypto API, non-extractable keys in IndexedDB). The project 
uses a 19-role AI agent team structure...

## Role Definition
The Architect is responsible for the technical architecture of the 
SG/Send platform. This includes system design, infrastructure decisions, 
security architecture, and technical direction...

## Current Focus
- Data room architecture: GitHub-as-store + ephemeral EC2 compute
- Chain of trust and key graphs
- Browser pod security model
- Identity spectrum design

## Recent Briefs and Deliverables
### v0.5.10 — GitHub Store and Ephemeral Compute
[full brief content or substantive summary]

### v0.5.10 — Browser Pod Architecture
[full brief content or substantive summary]

### v0.4.12 — Chain of Trust and Key Graphs
[full brief content or substantive summary]

## Key Decisions
- PKI uses Web Crypto API with non-extractable keys (not WASM libraries)
- Two storage classes: persistent (GitHub) vs ephemeral (S3 with TTL)
- EC2 instances are sealed boxes: port 443 inbound only, zero egress
- Browser pod uses Web Worker + Service Worker layered model
- Pod compromise is containable, not catastrophic (bounded by detection + budget)

## Open Questions
- Trust graph data model in Issues-FS: schema design pending
- Challenge-response protocol for layered auth: needs AppSec review
- ECDH vs RSA for PKI: RSA for MVP, ECDH evaluation pending

## Relationships
- Works closely with: Developer (implementation), AppSec (security review), 
  DevOps (infrastructure), GRC (risk acceptance)
```

## Maintenance

### Who Updates What

| Content | Who Updates It | When |
|---|---|---|
| `ROLE.md` | Each role maintains their own | When role definition changes |
| `context.md` | **Librarian** compiles from latest briefs and status | After each briefing session, or daily |
| Recent briefs in `recent/` | **Librarian** adds new briefs as they're produced | When briefs are finalised |
| `project/introduction.md` | **Librarian** or **Sherpa** | When version changes or major milestones |
| `project/priorities.md` | **Sherpa** | Daily or after priority changes |

### Automation (GitHub Actions)

A GitHub Action can automate parts of this:

```yaml
# On push to briefs directory:
# 1. Copy new briefs to the appropriate role's recent/ folder
# 2. Regenerate the role's index.html (list of briefs)
# 3. Rebuild the catalogue index

# On manual trigger or schedule (daily):
# 1. Regenerate context.md for each role
#    (pull latest ROLE.md + last N briefs + priorities)
```

For now, the Librarian maintains this manually. Automation comes as the cadence stabilises.

## LLM Context Budget

The `context.md` file needs to fit within LLM context windows. Target sizes:

| Provider | Context Window | Budget for Agent Context | Leaves for Conversation |
|---|---|---|---|
| Claude Sonnet | 200K tokens | ~20K tokens (~15,000 words) | ~180K tokens |
| GPT-4o | 128K tokens | ~20K tokens | ~108K tokens |
| Ollama (Llama 3) | 8-32K tokens | ~4K tokens (need a shorter version) | 4-28K tokens |
| Chrome AI (Gemini Nano) | ~4K tokens | ~1K tokens (minimal version) | ~3K tokens |

For smaller context windows, the `context.md` should have a **summary mode** at the top (1,000 tokens) followed by the full detail. The agent chat page can truncate based on the selected provider's context window.

## How This Connects to Agent Chat

The admin agent chat page (`/admin/agent-chat.html`) fetches context from this site:

```javascript
async function loadAgentContext(roleName) {
  const baseUrl = "https://{org}.github.io/{repo}";
  
  // Fetch the consolidated context for this role
  const contextUrl = `${baseUrl}/roles/${roleName}/context.md`;
  const response = await fetch(contextUrl);
  const contextMd = await response.text();
  
  return contextMd;
}
```

One GET request per agent. Returns everything the LLM needs. No authentication (it's GitHub Pages — public). Fast (CDN-cached).
