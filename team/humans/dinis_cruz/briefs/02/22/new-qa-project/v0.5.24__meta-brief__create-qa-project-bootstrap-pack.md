# Brief: Create the QA Project Bootstrap Pack

**version** v0.5.24  
**date** 22 Feb 2026  
**from** Human (project lead)  
**to** Librarian (lead), Sherpa, Architect, Developer, DevOps  
**type** Meta-brief — create a deliverable for another project  
**context** This brief is for the CURRENT Claude Code session / main SG/Send repo. The output is a document pack that will be uploaded to a NEW Claude Code session to bootstrap the QA Villager project.  

---

## What to Produce

A **bootstrap pack** — a set of documents that gives a fresh Claude Code session everything it needs to set up and run the QA Villager project. The new session will have zero context about SG/Send. The pack must be self-contained enough for a cold start, but reference back to this repo for anything it doesn't need to copy.

---

## The Pack Contents

### 1. Project Context Document

A concise overview of what SG/Send is, what has been built, and where to find things. NOT a copy of every brief — a summary with pointers:

| Section | What to Include |
|---|---|
| **What is SG/Send** | One-paragraph description. Encrypted communication platform. Serverless AWS. Browser-based PKI. |
| **Current version** | v0.5.24. What's shipped, what's in progress. |
| **Architecture summary** | Key components: Lambda functions, S3, FastAPI, admin panel, PKI, data rooms. Just enough to understand what's being tested. |
| **The admin panel** | What pages exist, what they do, how authentication works (access tokens). This is the primary test target. |
| **Repo location** | Where to clone the main repo (read-only). How to run SG/Send locally for testing. |
| **Key principles** | Zero-knowledge encryption, API-first, IFD web components, event-driven architecture. |

### 2. QA Project Mission Brief

The purpose, scope, and acceptance criteria for the QA project. Draw from `v0.5.24__brief__browser-automation-testing-and-living-docs.md` but formatted as a standalone mission brief for the new team:

| Section | What to Include |
|---|---|
| **Mission** | Browser automation tests + living documentation from screenshots |
| **Scope** | Test the SG/Send admin panel and public pages. NOT modify SG/Send code. |
| **Deliverables** | Test suite, screenshots, markdown docs, CI pipeline, FastAPI test runner, static docs site |
| **What success looks like** | Run tests → get pass/fail + documentation pages with screenshots, all automated |

### 3. Role Definitions for the QA Team

The new project doesn't need 19 roles. It needs a focused subset:

| Role | Why Needed |
|---|---|
| **QA Lead** | Designs tests, defines quality gates, curates the test suite |
| **Developer** | Implements browser automation, builds FastAPI test runner, writes CLI |
| **Architect** | Evaluates browser automation tools (Vercel Agent Browser, Playwright), designs execution environments |
| **DevOps** | CI pipeline (GitHub Actions), Lambda deployment, GitHub Pages for docs |
| **Sherpa** | Turns test descriptions into user-facing documentation text |
| **Librarian** | Organises generated docs, maintains the static site structure |

Six roles, not nineteen. Each role definition should be written as a ROLE.md following the same format used in the main project but scoped to QA work.

  Summary: We classify incidents by B/P/D priority...
  
- **Brief format**: /docs/practices/briefs.md
  Summary: Every brief has version, date, from, to, type...

The principle: **reference, don't copy.** The QA project reads from the main repo for practices. It only copies what it needs to modify.

### 5. Technical Bootstrap Guide

Step-by-step instructions for the new Claude Code session to set up the QA project infrastructure:

```
Phase 1: Repo Setup
  ├── Create the QA repo structure (tests/, screenshots/, docs/, server/, cli/)
  ├── Clone the main SG/Send repo as a read-only reference
  ├── Set up Python environment
  └── Create initial CLAUDE.md with role definitions and project context

Phase 2: Pipeline (Before Any Tests)
  ├── Create FastAPI server skeleton (routes for running tests, viewing results)
  ├── Create CLI skeleton (same actions as API)
  ├── Set up GitHub Actions workflow (trigger on push, run tests, deploy docs)
  ├── Deploy FastAPI to Lambda (same serverless pattern as main project)
  ├── Set up GitHub Pages for generated documentation
  └── Verify: can push to prod, can trigger from API, can trigger from CLI

Phase 3: Browser Automation Research
  ├── Evaluate Vercel Agent Browser (preferred — check capabilities, CLI, replay)
  ├── Evaluate Playwright as fallback
  ├── Can we run headless browser in Lambda? (probably needs container-based Lambda)
  ├── Can we run headless browser in Claude Code?
  ├── Can we run headless browser in GitHub Actions? (yes — standard)
  ├── Document: which environments support which execution modes
  └── Check Librarian archives: human previously wrote about meta-language for 
      browser commands — find and reference this

Phase 4: First Test (The Smoke Test)
  ├── Open the SG/Send admin home page in a headless browser
  ├── Take a screenshot
  ├── Enter an invalid access token → screenshot → assert error
  ├── Enter a valid access token → screenshot → assert dashboard loads
  ├── Generate markdown page from screenshots
  ├── Commit screenshots (only if visually changed)
  └── Push docs to GitHub Pages

Phase 5: Dogfooding
  ├── Use the QA project's own GitHub Pages as a test target
  ├── Open the QA docs site → screenshot → verify it renders
  └── This validates both the test infrastructure AND the docs pipeline
```

## Project Maturity Path

The QA project starts as a **Villager** (focused, bounded, one job) but will mature:

| Stage | What It Does | When |
|---|---|---|
| **Explorer** | Research browser automation tools, evaluate execution environments | First week |
| **Villager** | Stable test suite, running in CI, generating docs | First month |
| **Town Planner** | Owns the release process. Every release goes through this project's tests before shipping. | When test coverage is sufficient |

## Deliverable Format

The bootstrap pack should be a single directory that can be uploaded to a new Claude Code session:

```
qa-bootstrap-pack/
├── 01_project-context.md           ← What is SG/Send, what's been built
├── 02_mission-brief.md             ← The QA project's purpose and scope
├── 03_role-definitions/
│   ├── qa-lead.md
│   ├── developer.md
│   ├── architect.md
│   ├── devops.md
│   ├── sherpa.md
│   └── librarian.md
├── 04_practices-reference.md       ← Pointers back to main repo
├── 05_technical-bootstrap-guide.md ← Step-by-step setup instructions
├── 06_what-to-clone.md             ← Read-only access list
└── 07_first-session-brief.md       ← "Read this first" — the brief for the new 
                                       Claude Code's first session
```

File `07_first-session-brief.md` is the document described in the companion brief (`v0.5.24__brief__qa-project-first-session.md`). It's what the new Claude Code reads first.
