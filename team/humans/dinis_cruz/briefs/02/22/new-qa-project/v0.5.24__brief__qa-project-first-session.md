## Welcome

You are starting a new project. This is a **QA and documentation automation project** for SG/Send, an encrypted communication platform. You are NOT working on SG/Send itself — you are building the testing and documentation infrastructure that validates SG/Send works correctly.

This is a separate repo, a separate team, a separate concern. You consume SG/Send as a release. You test it. You document it. You don't modify it.

---

## Your Mission

Build a browser automation test suite that:

1. **Runs automated browser tests** against the SG/Send admin panel and public pages
2. **Captures screenshots** at each step of every test
3. **Generates markdown documentation** from those screenshots — living docs that are always up to date
4. **Publishes documentation** to a GitHub Pages site
5. **Runs everywhere** — CLI, FastAPI API, GitHub Actions, Claude Code, and eventually Lambda

Every test run produces two outputs: a pass/fail result AND a documentation page with screenshots.

## Key Technical Decisions

### API-First

Everything must be triggerable via API AND CLI. The web UI is a convenience layer on top of the API. Claude Code should be able to run the same tests that the web UI runs.

```bash
# CLI
python cli/run_tests.py --test admin.test_home_page --target https://localhost:8000

# API (same action)
curl -X POST https://qa-api.send.sgraph.ai/api/tests/run/admin.test_home_page
```

### Screenshot Commit Strategy

Screenshots are committed to the repo. To avoid unnecessary commits from rendering noise:

- Compare new screenshot with existing one using visual diff (pixel comparison with threshold)
- Only commit if the difference exceeds the threshold (e.g., >1% pixel change)
- If no code changes were made → test run → all pass → screenshots identical → **zero commits**
- If UI changed → screenshots differ → **new commit showing exactly what changed**

### Execution Environments

The same test must eventually run in:

| Environment | Priority | Notes |
|---|---|---|
| **GitHub Actions** | P1 — first | Standard, well-supported, headless browsers work natively |
| **CLI (local)** | P1 — first | Developer runs locally during development |
| **Claude Code** | P2 | Needs headless browser support in Claude Code environment |
| **FastAPI (Lambda)** | P3 | Needs container-based Lambda with headless browser |
| **Web UI** | P3 | Triggers tests via API, displays results |

Start with GitHub Actions and local CLI. Add Lambda and web UI later.

### External Services to Evaluate

| Service | What It Does | Why It's Relevant |
|---|---|---|
| **Browserless.io** | Headless browser as a service | Solves the "headless browser in Lambda" problem |
| **Vercel Agent Browser** | Agent-native browser automation with CLI and replay | Human's preferred tool. Can potentially record sessions for YouTube demos. |
