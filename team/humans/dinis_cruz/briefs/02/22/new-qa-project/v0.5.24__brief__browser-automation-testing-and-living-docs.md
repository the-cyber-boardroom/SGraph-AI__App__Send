## What to Build

A browser automation test suite that does two things simultaneously:

1. **Validates the UI works** — automated browser tests that catch bugs only visible in a real browser
2. **Generates user documentation** — screenshots taken during test runs become the images in markdown documentation pages

Every test run produces both a pass/fail result AND a set of screenshots that visually document the functionality being tested. The documentation is always up to date because it's generated from the tests, not written manually.

---

## Why This Matters

The admin panel now has complex workflows (PKI key management, key discovery, vault, fleet management, agent chat). Manual testing doesn't scale. But the usual problem with automated tests is they validate code without producing anything humans can read.

By combining testing with documentation, every test is a business asset: it proves the feature works AND explains how it works, with screenshots showing exactly what the user sees at each step.

## Project Structure: Separate Repo (Villager Project)

This is a **separate Git repository**, not part of the main SG/Send codebase. Reasons:

| Reason | Why |
|---|---|
| **Formal QA process** | The QA project consumes SG/Send as a release, not as source. It tests what users get, not what developers push. |
| **Clean separation** | Test infrastructure, screenshots, and documentation don't bloat the main repo. |
| **Independent cadence** | Tests can run on any release without being coupled to development branches. |
| **Image storage** | Screenshots are committed to this repo. That's a lot of binary data — keep it out of the main repo. |
| **Villager project** | Follows the project's pattern of independent, consumable sub-projects. Eventually brought in as a git submodule if needed. |

## The Execution Flow

### Step 1: Download and Start SG/Send

The test suite starts by downloading the latest release of SG/Send and running it locally:

```
1. Pull latest SG/Send release (or point to a running instance)
2. Start the server locally (or connect to target URL)
3. Verify the server is healthy (GET /api/health)
4. Run the test suite
```

### Step 2: Run a Test (Example: Access Token Validation)

```python
async def test_access_token_flow(browser, screenshots):
    # 1. Navigate to home page
    await browser.goto("https://localhost:8000/admin/")
    await screenshots.capture("01_navigate_to_home", 
        description="Navigate to the admin panel home page")
    
    # 2. Enter an invalid token
    await browser.fill("#access-token", "invalid-token-12345")
    await browser.click("#submit-token")
    await screenshots.capture("02_invalid_token_rejected",
        description="An invalid access token is rejected with an error message")
    
    # Assert: error message is shown
    assert await browser.is_visible(".error-message")
    
    # 3. Enter a valid token
    await browser.fill("#access-token", config.valid_token)
    await browser.click("#submit-token")
    await screenshots.capture("03_token_accepted",
        description="A valid access token grants access to the admin panel")
    
    # Assert: admin panel is loaded
    assert await browser.is_visible("#admin-dashboard")
    
    # 4. Verify dashboard content
    await screenshots.capture("04_admin_dashboard",
        description="The admin dashboard shows available management pages")
    
    assert await browser.is_visible("#link-pki")
    assert await browser.is_visible("#link-vault")
```

### Step 3: Generate Documentation

After the test run, the screenshots and descriptions are assembled into a markdown page:

```markdown

With a valid token, the admin dashboard loads showing all available management pages.
```

### Step 4: Commit Changes (Only If Visual Diff)

This is critical. Screenshots should only generate a new commit if the visual content actually changed:

```python
def should_commit_screenshot(new_image_path, existing_image_path):
    if not os.path.exists(existing_image_path):
        return True  # new screenshot, always commit
    
    # Compare images — allow for minor rendering differences
    # (anti-aliasing, font rendering, sub-pixel variations)
    diff = compute_visual_diff(new_image_path, existing_image_path)
    
    # Only commit if the difference exceeds a threshold
    # (e.g., >1% pixel difference — catches real UI changes,
    #  ignores rendering noise)
    return diff > VISUAL_DIFF_THRESHOLD
```

**If no code changes were made** → run tests → all pass → screenshots are visually identical → **no new commit to the docs repo.** Clean.

**If a UI change was made** → run tests → screenshots differ → **new commit with updated screenshots.** The git diff shows exactly what changed visually.

## Multiple Execution Environments

The same tests must run in multiple environments. API-first + CLI-first means one test suite, many runners:

| Environment | How | When |
|---|---|---|
| **Local (developer machine)** | `python cli/run_tests.py --target localhost:8000` | During development |
| **Claude Code** | Same CLI, run from Claude Code terminal | During agentic development sessions |
| **GitHub Actions** | `.github/workflows/run-tests.yml` triggers on push or schedule | CI/CD pipeline |
| **Web UI (FastAPI)** | POST `/api/tests/run` or click [Run Tests] in the admin UI | On-demand from browser |
| **Serverless (Lambda)** | Same FastAPI deployed as Lambda | Remote execution |

### API-First Design

```
POST /api/tests/run              ← Run all tests (or filtered by tag/suite)
POST /api/tests/run/{test_name}  ← Run a specific test
GET  /api/tests/results          ← List all test results
GET  /api/tests/results/{run_id} ← Get results for a specific run
GET  /api/docs                   ← View generated documentation
GET  /api/docs/{page}            ← View a specific documentation page
POST /api/docs/generate          ← Regenerate docs from latest screenshots
```

### CLI Interface

```bash
# Run all tests
python cli/run_tests.py --target https://localhost:8000

# Run specific test
python cli/run_tests.py --test admin.test_home_page

# Run and generate docs
python cli/run_tests.py --target https://localhost:8000 --generate-docs

# Just regenerate docs from existing screenshots
python cli/run_tests.py --docs-only
```

## Browser Automation: Research Needed

The Architect and Developer should evaluate browser automation options:

| Option | What It Is | Pros | Cons |
|---|---|---|---|
| **Vercel Agent Browser** (Vercel Labs) | Browser automation designed for agents. CLI with replay. Popular project. | Agent-friendly, good CLI, can record sessions for video. Human's preferred option. | Newer project, may have gaps. |
| **Playwright** (Microsoft) | Industry-standard browser automation. Python and Node bindings. | Mature, well-documented, excellent screenshot support, built-in visual comparison. | Not agent-specific. |
| **Puppeteer** (Google) | Chrome/Chromium automation. Node.js. | Lightweight, good for Chrome-specific testing. | Chrome only. |

**Recommendation**: evaluate Vercel Agent Browser first (the human specifically suggested it). It has the advantage of being agent-native and supporting session recording, which could eventually generate YouTube demo videos directly from test runs. Fall back to Playwright if the Vercel tool has gaps.

## Documentation Output: GitHub Pages

The generated documentation is published via GitHub Pages:

```
GitHub Actions runs tests
    │
    ├── Tests pass
    ├── Screenshots captured
    ├── Markdown docs generated
    ├── Visual diff check: only commit changed screenshots
    ├── Commit to docs repo (if changes)
    │
    ▼
GitHub Pages publishes updated docs
    │
    ▼
qa.send.sgraph.ai (or docs.send.sgraph.ai/qa)
```

The DevOps sets up the GH Pages deployment. The docs site shows every feature with screenshots from the latest passing test run.

## The Documentation Lifecycle

```
Feature implemented → Test written → Test runs → Screenshots captured 
    → Markdown generated → Docs published → Feature is documented

Bug fixed → Test updated → Regression confirmed → Screenshots updated
    → Docs updated → Fix is documented

UI changed → Test runs → Screenshots differ → New commit
    → Docs show the new UI → Change is documented

Version released → Full test suite runs → All screenshots regenerated
    → Complete docs for this version → Release is documented
```

Every version has a complete, accurate, screenshot-verified set of documentation. No manual documentation effort. No stale screenshots. No documentation drift.
