# Architecture Brief: Receiver Experience and the Trust Spectrum

**version** v0.5.24  
**date** 22 Feb 2026  
**from** Human (project lead)  
**to** Architect (lead), AppSec, Developer, Designer, Sherpa  
**type** Architecture brief — receiver UX and key delivery

## The Problem

The receiver experience is bad. Today, a recipient gets a link that looks like gibberish — a URL containing a document ID and a 32-character hexadecimal decryption key. Nobody can type this. Nobody can read it over the phone. Nobody can send it via WhatsApp without it looking suspicious.

We've been focused on the sender experience (upload, encrypt, manage keys). The receiver experience has been neglected. This needs to change because the receiver IS the customer experience — if the recipient can't easily access the file, the sender's experience doesn't matter.

---

## Core Principles

### 1. Security Is a Spectrum, Not a Binary

The level of security should match the sensitivity of the content and the threat model of the situation. Sending an NDA to an investor is different from sending nuclear launch codes. The system should support both, and everything in between.

There's no point protecting the data room entrance with Fort Knox if the user is receiving the access code via unencrypted SMS. Security is only as strong as the weakest link in the entire chain.

### 2. Security Is Not a Spectator Sport

The security properties should increase without adding friction for the user. The goal is to make security invisible — the user types a short code, the system silently layers on additional verification factors (browser fingerprint, timezone, IP range, etc.) to compensate for the simplicity of the user-facing code.

### 3. Never Compromise Server-Side Security

If we make the user-facing key simpler (e.g., a 5-digit code instead of a 32-character hex string), the data at rest must NOT become less secure. The short code is a delivery mechanism for the real key, not a replacement for it. The actual encryption key remains full-strength. The short code is a one-time-use token that retrieves the real key under controlled conditions.

### 4. Trust Builds with Usage (TOFU)

The first interaction is the most vulnerable and — paradoxically — the safest, because both parties are paying attention. After the first exchange, PKI keys are established and all subsequent communication is fully authenticated. The system should leverage this: acknowledge the TOFU vulnerability, minimise it, and then rapidly escalate to full PKI trust.

## The Short Code Model

### How It Works

Instead of giving the recipient a 32-character hex key, give them a short code — 5-6 alphanumeric characters, or even just a number:

```
Current:    "Open send.sgraph.ai/d/a3f7c891?k=8b2ef443d55bc229a1b3..."
Proposed:   "Open send.sgraph.ai and enter: 847293"
```

The short code is NOT the decryption key. It's a **one-time-use token** that retrieves the decryption key from the server under controlled conditions:

```
1. Sender uploads file → encrypted with full-strength key → stored on server
2. Server generates a short code (e.g., 847293) mapped to the decryption key
3. Sender sends the short code to recipient (WhatsApp, SMS, phone call, in person)
4. Recipient visits send.sgraph.ai, enters the short code
5. Server validates the code (one-use, not expired, correct environment)
   ├── Valid → returns the decryption key to the browser
   └── Invalid → self-destructs (see below)
6. Browser uses the decryption key to decrypt the file
7. Short code is invalidated — can never be used again
```

**The server-side encryption is unaffected.** The file is still encrypted with a full-strength key (256-bit AES, or PKI). The short code is just a controlled delivery mechanism for that key.

### Self-Destructing Codes

The short code self-destructs on failure. This is the critical security property:

| Property | Behaviour |
|---|---|
| **One-use** | The code can be used exactly once. After a successful retrieval, it's deleted. |
| **Failed attempts** | After N failed attempts (e.g., 1-3), the code AND the associated decryption key are destroyed. The file becomes permanently inaccessible via this code. The sender must generate a new code. |
| **Expiry** | The code expires after a configurable time (1 hour, 24 hours, 7 days). After expiry, it's deleted. |
| **Monitoring** | Every attempt (successful or failed) is logged. Anomalous patterns (rapid attempts, different IPs) trigger alerts. |

**The analogy**: a Da Vinci Code-style puzzle box with acid inside. Get it wrong and the contents are destroyed. The attacker doesn't get unlimited guesses — they get one shot. Even a 2-digit code (99 possibilities) becomes secure if the attacker only gets one attempt.

### The Math

| Code Length | Possibilities | Attacker Chance (1 attempt) | Attacker Chance (3 attempts) |
|---|---|---|---|
| 2 digits | 100 | 1% | 3% |
| 4 digits | 10,000 | 0.01% | 0.03% |
| 6 digits | 1,000,000 | 0.0001% | 0.0003% |
| 6 alphanumeric | 2,176,782,336 | negligible | negligible |

With self-destruction on failure, even a 4-digit code provides practical security for most use cases. A 6-digit code is more than sufficient.

## Environmental Factors as Silent Security

The system can silently use environmental data from the recipient's browser to augment the short code. The recipient doesn't need to do anything — these factors are collected automatically:

| Factor | What It Is | How It Helps |
|---|---|---|
| **Timezone** | The browser's timezone | Sender can specify "recipient is in GMT+0". If the request comes from GMT+8, reject. |
| **Country / IP range** | GeoIP lookup | Sender can specify "recipient is in the UK". Request from Nigeria → reject. |
| **Browser type** | User-Agent string | Sender knows recipient uses Chrome. Request from curl → reject. |
| **Language** | `Accept-Language` header | Sender knows recipient's browser is in English. Request with Chinese → flag. |
| **Time of day** | When the request arrives | Sender can specify "only accessible between 9am-6pm". |

### How This Works Cryptographically

These environmental factors can be used as **additional key material** in deriving the decryption key:

```
actual_key = KDF(short_code + timezone + country + browser_hint)
```

The server doesn't need to know or store these factors. The recipient's browser provides them automatically. If the attacker doesn't have the correct combination (right code + right timezone + right country + right browser), the derived key is wrong and decryption fails.

The sender specifies which factors to include when creating the short code. More factors = shorter code needed. Fewer factors = longer code needed. The system balances automatically.

### Important: Invisible to the User

The recipient types a 6-digit code. That's all they do. Behind the scenes, the browser collects timezone, language, approximate location (if permitted), and browser type. These are added to the key derivation. The user experience is "type 6 digits" — the security experience is "6 digits + timezone + country + browser."

## Progressive Trust: From Short Code to Full PKI

The goal is to get both parties to full PKI (public/private key pairs) as quickly and invisibly as possible.

### The Two-Message Key Exchange

```
Message 1: Sender → Recipient
  │
  ├── Sender has a PKI key pair (or creates one)
  ├── Sender encrypts the message
  ├── Sender ATTACHES their public key to the message
  ├── Recipient receives via short code (first-time flow)
  ├── Recipient's browser auto-generates a PKI key pair
  │   (transparent — "We've created a secure identity for you" notification)
  │
  ▼
Message 2: Recipient → Sender (Direct Reply)
  │
  ├── Recipient replies via the data room / reply link
  ├── Recipient's browser ATTACHES their public key to the reply
  ├── Sender receives the reply + recipient's public key
  │
  ▼
Both sides now have each other's public keys.
All subsequent messages are fully PKI-authenticated.
No more short codes needed.
The trust is established.
```

**After two messages, the system has silently upgraded from short-code-based access to full PKI.** The users didn't do anything special. They sent a message and got a reply. The PKI infrastructure was established transparently.

### The Direct Reply Feature

This is a separate dev brief (flagged here for the Developer). The "reply" button in the data room / message viewer sends a response back to the sender. The reply:

- Is encrypted with the sender's public key (received in Message 1)
- Contains the recipient's public key (auto-generated on first visit)
- Goes through the same encrypted channel
- Completes the PKI key exchange

This is the simplest, most powerful feature for establishing trust: reply to a message and both sides are now cryptographically authenticated.

## Data Room Access Tiers

Data rooms can have multiple access levels, matching the trust spectrum:

| Tier | How You Access | Trust Level | Use Case |
|---|---|---|---|
| **Lobby** | Short code (6 digits) | Low — first-time access, TOFU | Initial document delivery, first contact |
| **Member** | PKI key pair (established after first exchange) | Medium — identity verified by key exchange | Regular data room access, messaging |
| **Verified** | PKI + out-of-band verification (TOTP, phone call, video) | High — multi-factor verified | Sensitive documents, board communications |
| **Admin** | PKI + admin key (installed out-of-band, high ceremony) | Highest — physically verified | Data room management, key rotation, member management |

Users automatically progress through tiers as trust builds. First visit = Lobby. After key exchange = Member. After admin verification = Verified. The data room owner decides which tiers can access which content.

### Friendly Code Options for Data Room Entry

| Format | Example | Use Case |
|---|---|---|
| **Numeric** | `847293` | Easy to type, say over the phone |
| **Alphanumeric** | `K7X-4F2` | Shorter but harder to dictate |
| **Three words** | `coral-summit-breeze` | Easy to remember, easy to say (What3Words-style) |
| **Phrase** | `the quick brown fox` | Natural language, very memorable |
| **Personal challenge** | Image selection (pick the one that's meaningful to you) | Out-of-band knowledge — attacker can't guess |

The sender chooses the format when creating the code. All formats map to the same underlying mechanism (one-use token → retrieve real key).

## TOTP Integration (Future)

For verified-tier access, integrate with TOTP (Time-based One-Time Password) — the same protocol used by Google Authenticator, Authy, etc.:

- Admin generates a QR code for the data room
- User scans with their authenticator app
- Each access requires a 6-digit TOTP code (changes every 30 seconds)
- This is standard 2FA, applied to data room access

Not for immediate implementation, but the architecture should accommodate it. The short code mechanism already supports the pattern — TOTP is just a different way to generate and validate the code.

## Risk Assessment Tasks for AppSec

| Question | Context |
|---|---|
| What is the effective security of a 6-digit self-destructing code with 1 attempt? | Compare to current standards (SMS 2FA, banking verification codes). |
| How does environmental factor injection affect key derivation security? | Model the attacker who knows some but not all factors. |
| What are the risks of auto-generating PKI keys for new recipients? | The recipient didn't explicitly request a key pair. Is this a consent issue? |
| Map the TOFU attack window for the two-message key exchange | What can an attacker achieve between Message 1 and Message 2? |
| Review the self-destruct mechanism for race conditions | Can an attacker make multiple simultaneous requests before the first failure triggers self-destruct? |
| How does this interact with quantum-safe encryption goals? | The short code is a delivery mechanism, not the encryption. Server-side encryption can be upgraded independently. |

## For the Designer and Sherpa

The receiver experience needs to be designed from scratch. Current state: a gibberish URL. Target state: a clean, friendly, confidence-inspiring flow.

Design briefs needed:

| Brief | Focus |
|---|---|
| **First-time receiver flow** | What does the recipient see when they get a WhatsApp message with a code? What does the landing page look like? How do they enter the code? What happens if it fails? |
| **PKI key generation notification** | The browser auto-generates a key pair. How do we explain this to the user? What's the notification? How does the user know they now have a secure identity? |
| **Direct reply UX** | What does the reply button look like? What does the reply composer look like? How is it different from a new message? |
| **Trust indicator** | How does the UI show the current trust level? (Lobby → Member → Verified → Admin) What visual cues indicate "this is your first exchange" vs "this is a trusted contact"? |
