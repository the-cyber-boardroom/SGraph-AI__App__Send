# Architecture Brief: Reference Implementations — LLM-Consumable Libraries

**version** v0.5.24  
**date** 22 Feb 2026  
**from** Human (project lead)  
**to** Architect (lead), Developer, all roles  
**type** Architecture brief — dependency management pattern for agentic development  

---

## The Problem

The project now has multiple sub-projects sharing similar code: the admin UI pattern, the IFD web components, the FastAPI structure, the messaging panel, the debug panels. The classical approaches both fail:

**Copy-paste**: each project gets its own copy. A bug fix in one project doesn't propagate to the others. Implementations drift apart. Features are reimplemented slightly differently everywhere.

**Shared library (npm package, pip install)**: one fix applies everywhere. But now a change for one project can break another. Dependency trees explode. Importing the library brings in 100% of the code when you need 10%. Version conflicts. Regression cascades.

The agentic world offers a third option.

---

## The Solution: Reference Implementations

A **reference implementation** is a standalone repo that contains the canonical, well-documented version of a shared capability. It is designed specifically to be consumed by LLMs, not imported as a dependency.

```
Reference Implementation Repo              Sub-Project (e.g., SG/Send admin)
┌────────────────────────────┐              ┌────────────────────────────────┐
│                            │              │                                │
│  Canonical code            │   LLM reads  │  Project-specific code         │
│  Detailed documentation    │ ────────────►│  Adapted from reference        │
│  Implementation guides     │              │  Customised for this context   │
│  Test examples             │              │  Pinned to reference version   │
│  Changelog per feature     │              │                                │
│                            │              │  No import. No dependency.     │
│  Designed for LLMs to      │              │  The code lives HERE, adapted  │
│  understand and adapt      │              │  from the reference.           │
│                            │              │                                │
└────────────────────────────┘              └────────────────────────────────┘
```

### The Key Difference from Traditional Libraries

| Traditional Library | Reference Implementation |
|---|---|
| You import it: `npm install admin-ui` | You don't import it. The LLM reads it and writes adapted code in your project. |
| Bug fix in library → automatic update in all consumers | Bug fix in reference → LLM reviews the fix and adapts it per project |
| 100% of the code arrives in your project | LLM takes 2-10% of the code — only what's needed |
| Version conflicts between consumers | No conflicts — each project has its own adapted copy |
| Breaking change breaks all consumers | Breaking change in reference → each project's LLM evaluates and adapts independently |
| Designed for runtime import | Designed for LLM comprehension and adaptation |

## How It Works in Practice

### Adding a New Feature

The human wants to add user confirmation dialogs to the messaging panel (event-driven, web component, works via the existing message centre):

```
Step 1: Implement in the reference repo
  │
  ├── Write the feature in the canonical codebase
  ├── Write detailed documentation:
  │   - What was added
  │   - Why it was designed this way
  │   - How to integrate it
  │   - How to test it
  │   - Code samples for common variations
  ├── Write tests that demonstrate the feature
  ├── Tag the release (e.g., v1.9 → v1.10)
  │
  ▼
Step 2: Update each sub-project
  │
  ├── Go to sub-project (e.g., SG/Send admin)
  ├── Tell the LLM: "The reference implementation at [repo] has a new
  │   feature as of v1.10. We are currently pinned to v1.9. Read the
  │   changelog, understand the new feature, and implement it in our
  │   project adapted to our context."
  ├── LLM reads the reference docs + code
  ├── LLM reads the sub-project's existing code
  ├── LLM writes an adapted implementation
  ├── Developer/Architect reviews
  ├── Pin sub-project to v1.10 of the reference
  │
  ▼
Step 3: Document the adaptation
  │
  ├── Record: "Sub-project X adapted feature Y from reference v1.10"
  ├── Record: any deviations from the reference and why
  ├── Record: any parts of the feature NOT adopted and why
  │
  ▼
Three assets now exist for review:
  1. The reference implementation (what was built)
  2. The adaptation (what was implemented in this project)
  3. The deviation log (what differs and why)
```

### Fixing a Bug

```
Bug found in sub-project X
  │
  ├── Where is the bug?
  │   ├── In the sub-project's adaptation → fix locally, document deviation
  │   ├── In the reference implementation → fix in reference, propagate
  │   └── In the guidance/docs → fix the docs, re-evaluate adaptations
  │
  ▼
If bug is in the reference:
  ├── Fix in reference repo
  ├── Update reference version (v1.10 → v1.10.1)
  ├── For each sub-project:
  │   ├── LLM reads the fix
  │   ├── LLM applies an adapted version of the fix
  │   ├── Review and test
  │   └── Pin to v1.10.1
```

## What Makes a Reference Implementation LLM-Consumable

The reference repo is **over-indexed on documentation for LLM comprehension.** This is the crucial design principle. The code is good, but the documentation is what makes the pattern work.

### Required Documentation

| Document | Purpose | Example |
|---|---|---|
| **ARCHITECTURE.md** | High-level design: what this library does, key principles, component relationships | "This is an event-driven web component library for admin panels. Components communicate via CustomEvents on a shared message bus." |
| **COMPONENTS.md** | Catalogue of all components with descriptions and interfaces | "MessagePanel: displays messages, responds to `sg-message` events, supports `type: info/warning/error/question`" |
| **IMPLEMENTATION_GUIDE.md** | Step-by-step guide for LLMs adapting the code | "To add the message panel to your project: 1. Copy message-panel.js, 2. Register the custom element, 3. Connect to your event bus..." |
| **CHANGELOG.md** | Per-version, per-feature descriptions of what changed and why | "v1.10: Added user confirmation dialogs. MessagePanel now supports `type: question` with `onConfirm` and `onCancel` callbacks." |
| **TESTING_GUIDE.md** | How to test each component, expected inputs/outputs | "To test the confirmation dialog: dispatch a `sg-message` event with `type: question`, verify the dialog renders, simulate user clicking Confirm..." |
| **ADAPTATION_NOTES.md** | Known variations across sub-projects, common customisation points | "SG/Send uses a dark theme — override `--msg-bg` CSS variable. The QA project uses a simplified message format — omit the metadata fields." |
| **Per-feature READMEs** | Detailed docs per feature in subdirectories | `features/confirmation-dialog/README.md` with full explanation, code samples, screenshots |

### The Reference Agent

The reference implementation can have its own agent persona (like the project's 19 roles). A "reference implementation agent" loaded with the repo's documentation can:

- Answer questions about the implementation
- Explain why design decisions were made
- Help an LLM in a sub-project adapt a feature
- Review an adaptation and flag deviations from the reference

## Version Pinning

Each sub-project tracks which version of the reference it's synced with:

```json
// In the sub-project's config or README
{
  "reference_implementations": {
    "admin-ui": {
      "repo": "github.com/sgraph/ref-admin-ui",
      "pinned_version": "v1.10",
      "last_synced": "2026-02-22",
      "adaptations": [
        "Dark theme override",
        "Custom message format for encrypted content",
        "PKI key display component (not in reference)"
      ],
      "skipped_features": [
        "v1.10 analytics panel — not needed in this project"
      ]
    }
  }
}
```

This makes it trivial to see: "this project is 3 versions behind the reference. Here's what features it's missing. Here's what it's customised."

## The Feedback Loop

When a sub-project's adaptation reveals a problem in the reference, the feedback flows back:

```
Sub-project adaptation fails or is awkward
    │
    ├── Problem in the adaptation → fix in sub-project, note in deviation log
    │
    ├── Problem in the reference guidance → fix ADAPTATION_NOTES.md or
    │   IMPLEMENTATION_GUIDE.md in the reference repo
    │
    ├── Problem in the reference code → fix the code, bump version,
    │   propagate to other sub-projects
    │
    └── Ambiguity in the reference docs → clarify docs, add examples,
        prevent the same confusion in other sub-projects
```

This is self-improving. Every failed adaptation makes the reference better. Every clarification prevents future confusion. The reference implementation gets better every time an LLM tries to use it.

## Commercial Open Source Model

This pattern has a natural commercialisation angle for open source:

- **The reference implementation is open source** — anyone can read it, fork it, adapt it
- **The value is in the maintenance** — keeping the reference up to date, fixing bugs, adding features, writing documentation for LLM consumption
- **The billable service** is the ongoing curation and adaptation support — not the code itself but the guidance, the changelog, the implementation notes, the agent that helps you adapt it

Open source companies can charge for the creation and — more importantly — the **maintenance** of reference implementations that are designed for agentic consumption. The code is free. The intelligence around the code (how to use it, how to adapt it, what changed, what to watch out for) is the product.

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | Reference implementation repo created with canonical admin UI code |
| 2 | ARCHITECTURE.md explains the design principles and component model |
| 3 | COMPONENTS.md catalogues all components with interfaces |
| 4 | IMPLEMENTATION_GUIDE.md provides step-by-step guidance for LLM adaptation |
| 5 | CHANGELOG.md documents each feature with enough detail for an LLM to adapt it |
| 6 | At least one sub-project is pinned to a version of the reference |
| 7 | Adaptation is documented: what was taken, what was customised, what was skipped |
| 8 | A new feature added to the reference can be propagated to a sub-project by an LLM agent reading the changelog |
| 9 | A bug fix in the reference can be propagated to sub-projects independently |
| 10 | The reference repo has a persona/agent definition (context.md) usable in agent chat |
