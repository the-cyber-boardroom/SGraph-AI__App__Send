**version** v0.4.4

**from** Human (project lead)  
**to** Librarian, Historian, Journalist, DPO, AppSec, GRC

---

## The Mission

Create a body of evidence — based entirely on real, documented examples from this project — that demonstrates the value added by having a DPO, AppSec, and GRC role embedded in the agentic team from the start. Not theoretical benefits. Not "it could do this." Real examples. Real influence. Real impact on real decisions.

The focus is **not** on what these roles have done as a function (e.g., "the DPO wrote a privacy policy"). The focus is on **how these roles influenced other roles' decisions and actions** — moments where a developer, architect, DevOps, or other role made a better decision because the DPO, AppSec, or GRC was in the mix.

---

## Why This Matters

One of the most powerful claims about the agentic team model is that every role — including security and privacy — is present from the very first conversation, not bolted on at the end. This means:

- The Architect designs with privacy constraints baked in, not retrofitted
- The Developer writes code that respects security boundaries by default, not after a review
- The DevOps sets up infrastructure that's compliant from day one, not patched after an audit
- Every brief, every decision, every design choice has been influenced by having these roles present

But claims need evidence. This document commissions the collection of that evidence and the production of articles that present it.

## Phase 1: Evidence Collection (Librarian + Historian)

### The Task

Go through **every document, brief, memo, role definition, and conversation record** in this project and extract every instance where:

1. **DPO** influenced a decision, flagged a concern, or shaped an outcome
2. **AppSec** influenced a decision, flagged a concern, or shaped an outcome
3. **GRC** influenced a decision, flagged a concern, or shaped an outcome

For each instance, capture:

| Field | Description |
|---|---|
| **Source** | Exact document name, version, date, and section (with link if possible) |
| **Quote** | The actual text — what was said, by which role, in what context |
| **What happened** | What decision or action was influenced |
| **What would have happened without this role** | The counterfactual — what would the outcome have been if this role wasn't present? |
| **Impacted role(s)** | Which other roles were influenced (Developer, Architect, DevOps, etc.) |
| **Category** | Privacy, security, compliance, architecture, deployment, data handling, etc. |

### Known Evidence to Start From

The Librarian and Historian should start with these known examples (confirmed from project documents), then search for more:

#### DPO Evidence

**1. IP Addresses Not Stored in CloudFront Logs**
- **Source**: `v0.3.10__daily-brief__explorer-team-15-feb-2026.md` — Progress section
- **What happened**: When configuring AWS CloudFront logging via Firehose to S3, the DPO role ensured that IP addresses were NOT captured in the log pipeline. The brief explicitly states: "Privacy by design worked — DPO role ensured no IP addresses captured in logs."
- **Impact**: This influenced the DevOps configuration (what fields to capture), the Developer's log parser (what fields to expect), and the entire SA pipeline design (no PII in the data we process). Without the DPO, the default CloudFront log format includes IP addresses — we would have been storing PII by default without realising it.
- **Impacted roles**: DevOps, Developer, Architect
- **Category**: Privacy by design, GDPR compliance

**2. Empirical Verification of IP Address Exclusion**
- **Source**: `v0.4.3__briefs__briefing-packs-for-agents.md` — CloudFront pipeline pack, AppSec/DPO addendum
- **What happened**: The DPO (together with AppSec) required not just configuration-level assurance but **empirical verification** — examine actual log data to confirm IP addresses are truly absent. "This needs to be confirmed empirically, not assumed."
- **Impact**: Elevated the standard from "we configured it correctly" to "we have evidence it's working correctly." This is a compliance artefact — auditable proof.
- **Impacted roles**: QA (testing approach), DevOps (verification procedure)
- **Category**: Compliance evidence, audit trail

**3. Privacy Implications of Log Storage and Archival**
- **Source**: `v0.4.3__daily-brief__villager-team-16-feb-2026.md` — SA pipeline section
- **What happened**: The log pipeline includes an archive step (move processed logs to Glacier). The DPO's presence ensures that data retention considerations are built into the archive design — how long do we keep logs? What's the legal basis for storage? When must they be deleted?
- **Impact**: The archive pipeline was designed with retention awareness, not just as a cost optimisation.
- **Impacted roles**: DevOps (archive policy), Architect (pipeline design)
- **Category**: Data retention, GDPR compliance

**4. Localised Privacy Policies**
- **Source**: `v0.4.3__role-definition__translator.md` — Cross-cutting nature section
- **What happened**: When defining the Translator role, the DPO was explicitly noted as producing privacy policies that need legally-appropriate localisation per market. "DPO produces privacy policies → Translator produces legally-appropriate localised versions."
- **Impact**: Privacy policies aren't an afterthought translated by a generic translator — they're a DPO deliverable that gets specialist legal-aware translation.
- **Impacted roles**: Translator, Advocate (user-facing documentation)
- **Category**: Regulatory compliance, internationalisation

**5. Zero-Knowledge Architecture as a DPO-Influenced Decision**
- **Source**: Project architecture (referenced across multiple briefs)
- **What happened**: The entire encryption model — client-side encryption, server never sees plaintext, zero-knowledge design — is fundamentally a privacy-first architecture. The DPO's presence from the start means this wasn't a feature bolted on for marketing — it's the foundational architectural decision.
- **Impact**: Every technical decision flows from this — the transfer engine design, the manifest structure, the storage model, the key management approach. The DPO didn't "add" privacy — the DPO made privacy the starting point.
- **Impacted roles**: Architect (entire system design), Developer (encryption pipeline), DevOps (infrastructure never handles plaintext)
- **Category**: Privacy by design, architecture

#### AppSec Evidence

**6. Three-Word Keys: Security Review Before Implementation**
- **Source**: `v0.3.10__daily-brief__explorer-team-15-feb-2026.md` — UX section
- **What happened**: A UX idea emerged to use three-word keys ("sunset-bicycle-ocean") instead of complex cryptographic keys for better usability. AppSec was immediately flagged: "Needs security review before implementation." The idea was parked as a research item, not rushed into code.
- **Impact**: Prevented a potential security downgrade from being shipped in the name of UX. The security review will determine if the entropy is sufficient before any implementation.
- **Impacted roles**: Developer (blocked from implementing before review), Designer (must wait for security assessment)
- **Category**: Security review gate, usability vs. security trade-off

**7. Schema Injection Risk in Interactive Encrypted Workflows**
- **Source**: `v0.4.3__explorer-brief__interactive-encrypted-data-workflows.md` — Security model section
- **What happened**: When designing the JSON-schema-driven interactive forms (the encrypted workflow concept), AppSec was flagged to review a specific risk: "how do we ensure the schema can't be used to exfiltrate decrypted data? The renderer must be carefully sandboxed." The brief explicitly states schemas must be declarative, never executable.
- **Impact**: Established a security principle (no executable code in schemas) before a single line of implementation was written. The "declarative, not executable" rule is now a design constraint.
- **Impacted roles**: Architect (schema design constrained by security), Developer (renderer must be sandboxed)
- **Category**: Secure design, injection prevention

**8. NPM Publishing with Provenance (Supply Chain Security)**
- **Source**: `v0.3.2__briefs__js-transfer-engine-project-bootstrap.md` — GitHub Actions publish workflow
- **What happened**: The CD pipeline for publishing the `sgraph-send` NPM package uses the `--provenance` flag, which cryptographically links the published package to the exact GitHub commit and build. This is a supply chain security measure.
- **Impact**: Every published version of the package can be verified as coming from our repo, not tampered with. This is AppSec influence on the DevOps pipeline.
- **Impacted roles**: DevOps (pipeline configuration)
- **Category**: Supply chain security

**9. Security Review of Code and Publishing Pipeline**
- **Source**: `v0.3.2__briefs__js-transfer-engine-project-bootstrap.md` — Team setup
- **What happened**: AppSec is listed as a medium-priority role in the new JS project team, specifically for "security review of the code and publishing pipeline." This means security review is built into the team from the start, not added before launch.
- **Impacted roles**: DevOps (pipeline), Developer (code review process)
- **Category**: Security by default

#### GRC Evidence

**10. Compliance Context in Corporate Translation Profiles**
- **Source**: `v0.4.3__role-definition__translator.md` — Corporate translation section
- **What happened**: When designing the corporate translation feature (per-entity translation profiles), GRC's compliance perspective was embedded: "different markets have different regulatory language requirements." Each translation profile includes a compliance context dimension.
- **Impact**: The product feature is designed to be compliance-aware per market, not just linguistically accurate.
- **Impacted roles**: Translator (profile design), Architect (data model includes compliance context)
- **Category**: Regulatory compliance, market entry

**11. Production Change Discipline**
- **Source**: `v0.4.3__daily-brief__villager-team-16-feb-2026.md` — Production discipline section
- **What happened**: The production change workflow (identify → test first → plan → risk assess → migrate → deploy → verify) includes explicit risk assessment and rollback planning. "CISO + AppSec: for every change that touches production, assess the risk."
- **Impact**: Every production deployment includes a security and risk assessment step — not as an optional review, but as a mandatory stage in the workflow.
- **Impacted roles**: DevOps (deployment process), Developer (change management), Conductor (workflow enforcement)
- **Category**: Operational security, change management

## Phase 2: Quick Interviews (Journalist)

Once the evidence is collected, the Journalist conducts brief interviews with the impacted roles. Not long interviews — focused questions based on the evidence:

**For each evidence item, ask the impacted role:**
1. Was this the right call? In hindsight, was the DPO/AppSec/GRC influence valuable?
2. Would you have made the same decision without this role present? Be honest.
3. How much effort did it add to your work? (A lot? A little? None — it was just guidance?)
4. Did it change how you think about your own role's responsibilities?

**For the DPO, AppSec, and GRC themselves:**
1. What do you see as your most impactful intervention in this project so far?
2. What's the biggest risk you've prevented (or reduced)?
3. What's different about being embedded in the team from the start vs. reviewing at the end?
4. What concerns do you still have that haven't been addressed yet?

The interviews should be short and factual. We're collecting data, not writing feature profiles.

## Important Principles

### Evidence Only — No Assumptions

Every claim in the evidence pack must link to a specific document, with a specific quote, from a specific version. If we can't point to it, we don't include it.

This is NOT a document about what DPO/AppSec/GRC COULD do. This is a document about what they DID do, with receipts.

### Near-Misses Don't Count (Yet)

We're focused on actual influence, not potential influence. "The DPO would have caught this if..." is not evidence. "The DPO did catch this, and here's the document where it happened" IS evidence.

As the project matures and we have more examples — including near-misses we can document — we'll produce updated versions of these articles. This is v1, focused on what we have now.

### This Is a Living Document

As the project continues, more evidence will accumulate. The Librarian should maintain the evidence pack as a living document — adding new examples as they occur. Each new example gets the same structured treatment (source, quote, impact, counterfactual, impacted roles, category).

The articles can be updated periodically (quarterly?) with new evidence. The cumulative case gets stronger over time.
