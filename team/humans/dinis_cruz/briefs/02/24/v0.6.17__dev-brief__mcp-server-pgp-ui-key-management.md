# Dev Brief: MCP Server, PGP Key Support in UI, and User Key Management

**version** v0.6.17  
**date** 24 Feb 2026  
**from** Human (project lead)  
**to** Developer (lead), Architect  
**type** Dev brief — investor-requested priority + key management UX  

---

## Context

Investor feedback: they want an MCP server. This is a priority. MCP (Model Context Protocol) is the standard for connecting LLMs to tools and data. Adding MCP support means Claude, Cursor, and any MCP-compatible agent can interact with SG/Send directly — read messages, look up public keys, send encrypted files, browse data rooms — all through natural language.

This brief covers three connected priorities:

1. **MCP server** — expose our API to LLM agents
2. **PGP key support in the UI** — currently UI only works with shared keys
3. **User key management in the UI** — create, import, export, manage keys from the main user journey

All three feed into each other: the MCP server exposes whatever the UI supports, and key management is the foundation for both.

---

## Priority 1: MCP Server

### What We Want

```
User in Claude: "Hey Claude, can you read me this message on SG/Send?"
Claude: [calls SG/Send MCP tool: read_message(id="msg-abc123")]
Claude: "The message from Bob says: 'Q4 report attached. 
         Please review by Friday.' There's a PDF attachment."

User: "Can you find the public key for github user octocat?"
Claude: [calls SG/Send MCP tool: lookup_key(github_username="octocat")]
Claude: "Found their Ed25519 SSH key. Want me to encrypt a file for them?"

User: "Send this report to Alice's data room"
Claude: [calls SG/Send MCP tool: upload_to_dataroom(room="project-alpha", file=...)]
Claude: "Done. The file has been encrypted with Alice's public key 
         and uploaded to the project-alpha data room."
```

### Implementation Options

Research found two mature approaches for adding MCP to our existing FastAPI:

**Option A: `fastapi-mcp` (tadata-org)**

```python
from fastapi import FastAPI
from fastapi_mcp import FastApiMCP

app = FastAPI()  # our existing app

# That's it — auto-generates MCP from existing endpoints
mcp = FastApiMCP(app, name="sgsend-mcp", base_url="https://api.send.sgraph.ai")
mcp.mount()

# MCP server now available at /mcp
```

Two lines of code. Auto-discovers all FastAPI endpoints and exposes them as MCP tools. Preserves schemas, Pydantic models, and documentation. Supports authentication via FastAPI's `Depends()`. Mounts directly to the existing app — no separate deployment needed.

**Option B: `FastMCP` (Prefect/jlowin — the standard framework, powers 70% of MCP servers)**

```python
from fastmcp import FastMCP

mcp = FastMCP.from_fastapi(app=app)

# Or mount as a sub-application
mcp_app = mcp.http_app(path='/mcp')
app.mount("/mcp", mcp_app)
```

Also auto-generates from FastAPI. More control over how endpoints are mapped (RouteMap for fine-grained mapping of GET → resources, POST → tools). FastMCP 1.0 was incorporated into the official MCP Python SDK. More configurable but slightly more setup.

**Option C: Add MCP endpoint to existing Lambda**

Add an MCP-compatible SSE endpoint to the existing Lambda function. Minimal infrastructure change — just a new route. However, Lambda's execution model (request/response) may not align well with MCP's SSE streaming.

### Recommendation

**Start with Option A (`fastapi-mcp`).** It's the lowest-effort path: two lines added to our existing FastAPI app, auto-discovery of endpoints, immediate Claude compatibility. If we need more control later, migrate to FastMCP. Don't over-engineer the first version.

### Does MCP Give Us Immediate Claude Support?

**Yes** — with a caveat. Claude Desktop connects to MCP servers via SSE (Server-Sent Events) or stdio. Our FastAPI + `fastapi-mcp` exposes an SSE endpoint at `/mcp`. Claude Desktop (and Claude.ai with connectors) can connect to this directly. Claude Code connects via stdio, which needs `mcp-proxy` to bridge to our SSE endpoint.

The workflow:
1. Add `fastapi-mcp` to our FastAPI app
2. Deploy (same Lambda or a lightweight always-on endpoint)
3. Users configure Claude Desktop or Claude.ai to connect to `https://api.send.sgraph.ai/mcp`
4. Claude can now call SG/Send tools

### MCP Tools to Expose (MVP)

| Tool | What It Does | Maps to Existing API |
|---|---|---|
| `read_message` | Read a decrypted message by ID | GET /messages/{id} |
| `list_messages` | List messages in inbox | GET /messages |
| `send_message` | Send an encrypted message | POST /messages |
| `send_file` | Encrypt and send a file | POST /files |
| `lookup_public_key` | Find a user's public key (our registry, GitHub) | GET /keys/{identifier} |
| `list_dataroom_files` | List files in a data room | GET /datarooms/{id}/files |
| `upload_to_dataroom` | Upload encrypted file to a data room | POST /datarooms/{id}/files |
| `get_trust_score` | Look up a user's trust score | GET /trust/{identifier} |
| `verify_signature` | Verify a PKI signature | POST /verify |

Start with `read_message`, `send_message`, `lookup_public_key`, and `list_dataroom_files`. Add the rest based on usage.

### Security Consideration

MCP tools execute on behalf of the user. The MCP server must authenticate the user (API key or OAuth token) and only expose data/actions the user is authorised for. The zero-knowledge principle still applies: the MCP server handles encrypted blobs. Decryption happens in the client (Claude's tool execution environment or the user's browser).

**Critical**: the MCP server must NOT have access to private keys. It orchestrates encrypted operations. Decryption is always client-side.

## Priority 2: PGP Key Support in the UI

### Current State

The UI currently only works with shared keys (symmetric encryption or platform-managed key pairs). It does NOT support user-provided PGP keys.

### What Needs to Change

The UI should support PGP encryption as a first-class option. If a user has a PGP key (from GPG, from GitHub, from a key server), they should be able to use it in the SG/Send UI for sending and receiving.

| Feature | What It Does |
|---|---|
| **Encrypt with PGP public key** | User provides a recipient's PGP public key → file encrypted with PGP → downloadable as `.gpg` or sendable via platform |
| **Decrypt with PGP private key** | User imports their PGP private key → decrypts `.gpg` files in the browser |
| **Fetch PGP key from GitHub** | User enters a GitHub username → UI fetches their `.gpg` key → uses it for encryption (from the GitHub-as-registry brief, v0.6.14) |
| **Set encryption key in UI** | User can configure WHICH key to use for encryption: platform key, PGP key, SSH key. This was a feature request from users. |

### Technical Approach

**OpenPGP.js** is the browser-compatible PGP implementation. It handles key parsing, encryption, decryption, signing, and verification — all client-side.

```javascript
import * as openpgp from 'openpgp';

// Encrypt with recipient's PGP public key
const publicKey = await openpgp.readKey({ armoredKey: recipientPGPKey });
const encrypted = await openpgp.encrypt({
    message: await openpgp.createMessage({ binary: fileBytes }),
    encryptionKeys: publicKey,
    format: 'binary'
});

// Decrypt with user's PGP private key
const privateKey = await openpgp.decryptKey({
    privateKey: await openpgp.readPrivateKey({ armoredKey: userPGPKey }),
    passphrase: userPassphrase
});
const decrypted = await openpgp.decrypt({
    message: await openpgp.readMessage({ binaryMessage: encryptedBytes }),
    decryptionKeys: privateKey
});
```

The backend code for PGP already exists in the admin/backend. This is about exposing it in the main user-facing UI.

## Priority 3: User Key Management in the UI

### Current State

Key generation and management exists in the backend / admin side. The main user-facing UI does NOT expose it. Users cannot create, import, export, or manage their own keys from the standard user journey.

### What the UI Needs

A key management section accessible from the user's profile or settings:

```
🔐 My Keys

  Active Key: Ed25519 (generated 2026-02-10)
  Fingerprint: sha256:abc123...
  
  [+ Generate New Key]    [↑ Import Key]    [↓ Export Key]

  ── Key Ring ──────────────────────────────────────────
  
  🔑 Ed25519 (platform)     ★ Active    [Export] [Delete]
     Generated: 2026-02-10
     Fingerprint: sha256:abc123...
  
  🔑 PGP/GPG                              [Set Active] [Delete]
     Imported: 2026-02-15
     Fingerprint: sha256:def456...
     Source: GitHub (username: alice)
  
  🔑 SSH Ed25519                           [Set Active] [Delete]
     Imported: 2026-02-18
     Fingerprint: sha256:ghi789...
     Source: Manual import
```

### Key Actions

| Action | What It Does |
|---|---|
| **Generate new key** | Create a new Ed25519 or PGP key pair in the browser. Store in browser secure storage. |
| **Import key** | Import an existing key: drag-and-drop file, paste from clipboard, or password manager autofill. Supports PGP, SSH Ed25519, PEM, JWK. Auto-detect format. (From key portability brief, v0.6.14) |
| **Export key** | Download the key pair (passphrase-encrypted). For backup or transfer to another device. |
| **Set active key** | Choose which key to use by default for encryption/decryption. |
| **Delete key** | Remove a key from the browser. Warning: if this is your only copy, you lose access to anything encrypted with it. |
| **Set encryption key for sending** | When sending a message, choose which key to encrypt WITH (the recipient's key) and which key to SIGN with (your key). This was a feature request. |

### Connection to Key Portability Brief (v0.6.14)

The key portability brief specced out import/export methods (drag-and-drop, clipboard, password manager, QR code, one-time link) and format auto-detection (SSH, PGP, PEM, JWK, native). This UI work is the implementation of that brief's acceptance criteria.

## What Gets Exposed Through MCP

The MCP server exposes whatever the UI supports. Once PGP keys and key management are in the UI, the MCP tools can also:

```
User: "Import my PGP key from GitHub into SG/Send"
Claude: [calls lookup_key(github_username="alice", key_type="pgp")]
Claude: [calls import_key(key=..., source="github")]
Claude: "Done. Your PGP key from GitHub has been imported. 
         Fingerprint: sha256:def456..."

User: "Encrypt this file for Bob using his PGP key"
Claude: [calls lookup_key(identifier="bob", key_type="pgp")]  
Claude: [calls send_file(file=..., recipient_key=..., encryption="pgp")]
Claude: "Encrypted with Bob's PGP key and sent."
```

The MCP server + PGP support + key management = full agentic workflow for encrypted communication through Claude.
