## Context

Solid (Social Linked Data) is a web decentralisation project led by Tim Berners-Lee, the inventor of the World Wide Web. Developed originally at MIT and now driven commercially by Inrupt (with Bruce Schneier as chief of security architecture), Solid aims to give users full control of their data by decoupling applications from data storage.

Solid and SG/Send are solving the same problem from different angles. Solid solved interoperability and data ownership but left privacy as server-enforced access control. SG/Send solved zero-knowledge encryption but hasn't yet addressed interoperability and app decoupling. Together, they form the complete picture: user-owned data that's both interoperable AND truly private.

## Part A: How Solid Works

### The Four Decoupled Components

**1. Pods (Personal Online Data Stores)**

A Pod is a RESTful file server that stores your data in containers (folders). The Solid specification defines the HTTP API; the storage backend is implementation-specific (filesystem, database, cloud storage — the spec doesn't care). You choose where your Pod is hosted: self-hosted on your own server, Inrupt's Enterprise Solid Server, community servers like solidcommunity.net, or any compliant provider.

Data inside Pods is structured using Linked Data (RDF — Resource Description Framework), making it machine-readable and interoperable across applications. A Pod exposes data as HTTP resources: each file or container has a URL, can be read with GET, written with PUT/POST, and deleted with DELETE. Standard web operations on standard web URLs.

```
Alice's Pod (https://alice.pod.example/)
├── profile/
│   └── card              ← WebID Profile (RDF)
├── contacts/
│   ├── bob.ttl           ← Contact info in Turtle/RDF
│   └── carol.ttl
├── photos/
│   ├── vacation/
│   │   ├── photo1.jpg
│   │   └── photo2.jpg
│   └── .acl              ← Access control for photos/
├── health/
│   ├── records.ttl
│   └── .acl              ← Restricted access
└── inbox/                ← Linked Data Notifications inbox
```

**2. WebID (Identity)**

A WebID is a URI — like `https://alice.example/profile/card#me` — that resolves to an RDF profile document. The profile links to your Pods, your identity provider, your inbox, your public keys, and any other information you choose to publish. It's the decentralised equivalent of a username, but it's a URL you control.

WebIDs are the universal identifier across the Solid ecosystem. They can identify people, organisations, applications, or any agent. A WebID doesn't require a specific provider — any URL that resolves to a valid WebID Profile document is a WebID.

**3. Authentication (Solid-OIDC)**

Solid uses OpenID Connect adapted for decentralised use (Solid-OIDC). You authenticate with your identity provider (which is separate from your Pod provider), receive a Proof of Possession (PoP) token, and use that token to access resources across any Pod.

The flow is similar to "Sign in with Google" but decentralised: you pick your identity provider. The identity provider verifies your identity; the Pod server checks your token against its access control rules. Importantly, the identity provider and the Pod provider are independent services — you can mix and match.

Earlier versions of Solid used WebID-TLS (client certificates), but browser vendors dropped support for the key generation element, pushing the ecosystem toward Solid-OIDC.

**4. Authorization (WAC / ACP)**

Access control in Solid is handled by Web Access Control (WAC) or Access Control Policies (ACP). These are ACL-based systems: you define who (which WebID) can do what (Read, Write, Append, Control) to which resources.

```turtle
# Alice's access control for her photos folder
<#authorization1> a acl:Authorization;
    acl:agent <https://bob.example/profile/card#me>;
    acl:accessTo <https://alice.pod.example/photos/>;
    acl:mode acl:Read.
```

Access control is per-resource and fine-grained. The Pod server enforces the rules — when Bob requests Alice's photos, the server checks whether Bob's WebID has Read access before serving the response.

### The Core Design Principle

**Applications are decoupled from data.** An app doesn't store your data — it reads from and writes to YOUR Pod with YOUR permission. Switch apps, keep your data. Switch Pod providers, keep your apps. The data is yours, stored where you choose, structured in standard formats (RDF) that any compliant app can read.

### Current State

Solid has W3C Community Group standardisation in progress, several server implementations (Node Solid Server, Inrupt Enterprise Solid Server), early adopters including the BBC (media recommendations) and the NHS (patient data access), and a growing but still small developer community. Adoption has been slower than hoped — a parallel worth noting given our PKI historical analysis research (v0.6.14).

## Part B: Similarities and Differences with SG/Send

### The Mapping

| Concept | Solid | SG/Send | Analysis |
|---|---|---|---|
| **Where data lives** | Server-side Pod. Provider hosts the data. RESTful API. | Client-side encrypted blobs. Server stores ciphertext. Storage backend abstracted via MemoryFS (S3, ZIP, SQLite, memory). | Both decouple storage from application. Solid exposes a standard API. We abstract the backend internally. |
| **Identity** | WebID — a URL resolving to an RDF profile document | PKI key pair — the public key IS the identity | Both are decentralised. WebID is human-readable and discoverable. PKI is cryptographically verifiable. They complement each other. |
| **Authentication** | Solid-OIDC — delegated to an identity provider, PoP tokens | PKI signatures — cryptographic proof, no third party needed | Solid requires a trusted identity provider. SG/Send requires only the key pair. Different trust models, both valid. |
| **Authorization** | WAC/ACP — server-enforced ACLs. The server checks your WebID against access control rules before serving data. | Encryption-based — if you don't have the key, you can't read it. Enforcement is mathematical, not server-side. | **This is the critical difference.** Solid trusts the server to enforce rules. SG/Send doesn't trust the server at all. |
| **Data ownership** | User chooses Pod provider, can migrate between providers | User holds private key. Server can't read data even if they wanted to. | Solid gives you portability. SG/Send gives you privacy. Both are needed. |
| **App decoupling** | Core feature — any Solid app can read/write to any Pod via standard APIs | Not yet — but MemoryFS abstraction + open formats (GPG, AGE) + zero lock-in principle make this architecturally possible | Solid is ahead here. We should learn from their interoperability model. |
| **Interoperability** | Linked Data / RDF / W3C standards. Machine-readable, semantically rich. | Open encryption standards (GPG, AGE, Web Crypto API). File formats are open but not semantically structured. | Solid is ahead on interoperability. RDF makes data meaningful across apps. Our encrypted blobs are opaque without the key. |
| **"Pod" concept** | A hosted server you access via HTTP — persistent, always-on | Our browser pod (ephemeral compute for scanning/processing) — short-lived, task-specific | Confusingly similar name, completely different concept. Our "pod" is an ephemeral compute environment. Solid's "Pod" is a persistent data store. |
| **Discovery** | WebID Profile links to Pods, keys, inbox. Standard Linked Data discovery. | Public key registry (GitHub, our registry). No standardised discovery yet. | Solid has mature discovery. We should integrate with it rather than reinvent. |
| **Notifications** | Linked Data Notifications (LDN), WebSockets | Polling-based (MVP), WebSocket/SSE planned | Both heading toward real-time. Compatible approaches. |

### The Critical Difference: Server Trust

Solid's Pod provider CAN read your data. Access control is enforced by the server checking ACLs before serving responses. If the server is compromised, or if the provider is malicious, or if a government issues a subpoena — the data is readable.

SG/Send's server CANNOT read your data. The server stores encrypted blobs. Even a compromised server, a malicious provider, or a government subpoena yields only ciphertext. The data is only readable by holders of the appropriate private keys.

This is not a criticism of Solid — it's a different design choice with different tradeoffs:

| | Solid (Server-Enforced) | SG/Send (Encryption-Enforced) |
|---|---|---|
| **Server can read data** | Yes | No |
| **Server can enforce complex access rules** | Yes (fine-grained ACLs, conditional access, time-based rules) | Limited (you either have the key or you don't) |
| **Server can index and search data** | Yes (data is in cleartext, server-side search works) | No (server sees only encrypted blobs) |
| **Survives server compromise** | No (compromised server = exposed data) | Yes (compromised server = exposed ciphertext) |
| **Survives provider turning malicious** | No | Yes |
| **Supports Linked Data / RDF** | Yes (data is structured, machine-readable) | Not yet (encrypted blobs are opaque) |
| **App interoperability** | High (any Solid app can read any Pod) | Low currently (apps need our encryption layer) |

**The combination is more powerful than either alone.** A Solid Pod where the data is encrypted client-side (SG/Send-style) before storage gives you BOTH: Solid's interoperability, discovery, and app-decoupling AND SG/Send's zero-knowledge privacy.

## Part C: Short-Term Compatibility (Easy Wins)

### 1. WebID as Identity in Our Registry

A user's WebID could link to their SG/Send public key. The WebID Profile is an RDF document — adding a public key triple is straightforward and already supported by the Solid spec:

```turtle
# Alice's WebID Profile (at https://alice.example/profile/card)

@prefix foaf: <http://xmlns.com/foaf/0.1/>.
@prefix sec: <https://w3id.org/security#>.
@prefix sgsend: <https://sgraph.ai/ns/send#>.

<#me> a foaf:Person;
    foaf:name "Alice";
    sec:publicKeyPem "-----BEGIN PUBLIC KEY-----\nMIIB...";
    sgsend:sgsendPublicKey "ssh-ed25519 AAAAC3NzaC1...";
    sgsend:keyFingerprint "sha256:abc123...".
```

Now Solid's discovery mechanism finds our encryption keys. Any app that knows how to read a WebID Profile can discover Alice's public key and send her an encrypted message. This is essentially free — it's adding triples to an existing RDF document.

**Effort**: Low. Spec out the RDF predicates, document the integration, create a helper tool.

### 2. Solid Pod as a MemoryFS Storage Backend

Our MemoryFS already abstracts storage (S3, ZIP, file system, memory). Adding a Solid Pod backend means implementing the Linked Data Platform (LDP) REST API as a storage adapter:

```
Existing:  fs = MemoryFS(backend='s3', bucket='data-room-abc123')
New:       fs = MemoryFS(backend='solid-pod', pod_url='https://alice.pod.example/sgsend/')
```

SG/Send data rooms could store encrypted blobs in a user's Solid Pod. The Pod sees encrypted blobs in standard containers. Solid handles discovery, pod-to-pod federation, and the HTTP API. We handle encryption. The user's data lives in their Pod — not on our servers.

**Effort**: Medium. LDP REST adapter for MemoryFS. The API is well-documented.

### 3. Encrypted Files Stored in a Solid Pod

Even without deep integration, a user could store their `.sgenc` / `.age` / `.gpg` files in their Solid Pod as opaque resources. Solid apps see encrypted blobs. Our app (or any compatible tool) decrypts them. Standard Solid access control manages who can see the encrypted blobs; our encryption manages who can read the contents.

```
Alice's Pod
└── sgsend/
    ├── inbox/
    │   ├── message-001.sgenc    ← Encrypted message from Bob
    │   └── message-002.sgenc    ← Encrypted message from Carol
    ├── sent/
    │   └── message-003.sgenc    ← Alice's sent message to Dave
    └── datarooms/
        └── project-alpha/
            ├── spec.pdf.sgenc
            └── contract.docx.sgenc
```

**Effort**: Low. This requires no protocol changes — just using a Solid Pod as a file store for encrypted files.

### 4. WebID-OIDC for Data Room Authentication

Instead of building our own identity system for data room access, we could accept WebID-OIDC. Users authenticate with their existing Solid identity, then our PKI layer handles encryption and decryption. This gives data rooms federated authentication out of the box — any user with a WebID from any provider can be given access.

```
Data room access flow:
1. User authenticates via WebID-OIDC (their chosen identity provider)
2. Data room verifies their WebID
3. Data room looks up their public key (from WebID Profile or our registry)
4. Data encrypted for their public key is now accessible
5. User decrypts in their browser with their private key
```

**Effort**: Medium. Solid-OIDC client library integration + WebID-to-public-key discovery.

## Part D: Future and Tighter Integrations

### 1. The Encrypted Solid Pod (The Big One)

A Solid-compliant server where all data is encrypted client-side before storage. The server exposes the standard Solid API but stores only encrypted blobs. This is what Solid is missing — their Pod providers can read your data. We fix that.

```
Standard Solid Pod:
  App → writes cleartext RDF → Pod stores cleartext → Server can read it

Encrypted Solid Pod (SG/Send + Solid):
  App → encrypts with user's key → writes encrypted blob → Pod stores ciphertext
  → Server CANNOT read it
  → Other apps with the key CAN read it (interoperability preserved)
  → Solid's discovery, ACLs, and federation still work (on encrypted resources)
```

**The challenge**: Solid's power comes partly from server-side operations on structured data (search, indexing, SPARQL queries). Encrypting everything means the server can't do these operations. Solutions exist (searchable encryption, encrypted indexes, client-side search) but they're complex.

**The pragmatic approach**: support BOTH encrypted and unencrypted resources in the same Pod. Public profile data stays in cleartext (it's public anyway). Private data (health records, financial data, confidential files) is encrypted. The user chooses per-resource. Solid's ACLs provide the first layer; encryption provides the second.

### 2. Linked Data + Content Trust

Solid uses RDF for structured data. Our content trust infrastructure (PKI-signed facts, provenance chains from v0.6.14) maps naturally to RDF:

```turtle
# A signed claim as RDF

<#claim-001> a sgsend:SignedClaim;
    sgsend:content "Infection rates reduced by 23% with early intervention";
    sgsend:author <https://drsmith.example/profile/card#me>;
    sgsend:signature "sha256:abc123...";
    sgsend:timestamp "2026-02-24T10:00:00Z";
    sgsend:evidence <https://dataset.example/study-2026>;
    sgsend:licence <https://creativecommons.org/licenses/by-s/4.0/>.
```

This is the bridge between our PKI provenance and Berners-Lee's Linked Data vision. The semantic web parallel identified in the PKI historical analysis brief (v0.6.14) — the idea that knowledge graphs and LLMs have created the conditions for linked data to finally succeed — applies directly here. Solid provides the interoperability framework (RDF, SPARQL, Linked Data). We provide the trust layer (PKI signatures, provenance chains, trust web).

Signed claims as RDF triples can be queried, traversed, and verified. "Show me all claims by Dr. Smith" is a SPARQL query. "Verify this claim's signature" is a cryptographic operation. Both work together.

### 3. Solid WebID for Agent Identity

The agentic workflows brief (v0.6.14) gives each agent a PKI key pair. Solid's WebID gives agents a discoverable, standardised identity. Combine them:

```turtle
# An agent's WebID Profile

<#research-agent> a sgsend:Agent;
    foaf:name "Research Agent";
    sgsend:agentType "research";
    sec:publicKeyPem "-----BEGIN PUBLIC KEY-----\n...";
    sgsend:trustScore "0.87";
    sgsend:capabilities "web-search", "data-analysis";
    sgsend:workflow <https://workflow.example/pricing-analysis>;
    pim:storage <https://agent-pod.example/research-agent/>.
```

Any platform that understands WebID can discover the agent's identity, public key, trust score, and capabilities. This is a standardised way to identify and authenticate agents across platforms — not just within our ecosystem.

### 4. Notifications as Encrypted Messages

Solid's notification spec (Linked Data Notifications via LDN, plus WebSockets for real-time) could serve as an encrypted message delivery channel:

```
1. Bob sends Alice an encrypted message
2. The encrypted blob is written to Alice's Pod inbox
3. Alice's Pod sends a notification (via WebSocket) to Alice's browser
4. Alice's browser fetches the encrypted blob from her Pod
5. Alice's browser decrypts with her private key
6. Alice reads the message
```

The notification tells you "something changed." The encryption ensures only you can read what changed. Solid's infrastructure handles the delivery; our PKI layer handles the privacy.

### 5. Inrupt as a Strategic Partner

Inrupt is the commercial company behind Solid. Their CTO is Tim Berners-Lee. Their chief of security architecture is Bruce Schneier. They've built the Enterprise Solid Server (ESS) for performance, scale, and compliance.

Our zero-knowledge encryption layer is exactly what Solid needs but doesn't have. The conversation:

> "Solid gives users data ownership. But Pod providers can still read the data. What if we add a layer where even the Pod provider can't read it? We have the technology — browser-based PKI, zero-knowledge encryption, client-side key management. Your infrastructure, our encryption. Together: user-owned data that's truly private."

This aligns with both organisations' missions. It's not competitive — Solid doesn't do encryption; we don't do interoperability. It's complementary.

**Ambassador action**: initiate conversation with Inrupt. The technical integration points (WebID public keys, Pod as MemoryFS backend, Solid-OIDC for data rooms) are concrete enough to propose a collaboration.

## Architecture: Where the Pieces Fit

```
┌─────────────────────────────────────────────────────────────────┐
│                        User's Browser                           │
│                                                                 │
│  ┌─────────────┐  ┌──────────────┐  ┌────────────────────────┐ │
│  │  SG/Send    │  │  Solid App   │  │  Any Solid-Compatible  │ │
│  │  UI         │  │  (3rd party) │  │  App                   │ │
│  └──────┬──────┘  └──────┬───────┘  └──────────┬─────────────┘ │
│         │                │                      │               │
│  ┌──────▼──────────────────────────────────────▼─────────────┐ │
│  │              SG/Send Encryption Layer                      │ │
│  │  PKI key management │ Encrypt/Decrypt │ Sign/Verify       │ │
│  │  (Web Crypto API, Service Worker trust anchor)             │ │
│  └──────┬────────────────────────────────────────────────────┘ │
│         │                                                       │
│  ┌──────▼────────────────────────────────────────────────────┐ │
│  │              Solid Client Library                          │ │
│  │  WebID-OIDC auth │ LDP read/write │ ACL management        │ │
│  └──────┬────────────────────────────────────────────────────┘ │
└─────────┼───────────────────────────────────────────────────────┘
          │ HTTPS
          ▼
┌─────────────────────────────────────────────────────────────────┐
│                     User's Solid Pod                            │
│                                                                 │
│  Standard Solid Pod server (Inrupt ESS, NSS, self-hosted)      │
│                                                                 │
│  Stores: encrypted blobs (.sgenc)                               │
│  Exposes: standard Solid/LDP REST API                           │
│  Can read: NOTHING (encrypted blobs are opaque)                 │
│  Provides: discovery, federation, notifications                 │
│                                                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  /profile/card          ← WebID + public key (cleartext) │  │
│  │  /sgsend/inbox/         ← Encrypted messages             │  │
│  │  /sgsend/datarooms/     ← Encrypted data room files      │  │
│  │  /sgsend/vault/         ← Personal encrypted vault       │  │
│  │  /public/               ← Public data (cleartext, user's │  │
│  │                            choice)                        │  │
│  └──────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

The encryption layer sits between the application and the Solid client library. Apps write through the encryption layer (data is encrypted before it reaches the Pod). Apps read through the encryption layer (data is decrypted after it's fetched from the Pod). The Pod itself is a standard Solid Pod — no modifications needed. It just happens to store encrypted blobs.
