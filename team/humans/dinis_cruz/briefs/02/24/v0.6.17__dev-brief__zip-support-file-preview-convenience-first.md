# Dev Brief: ZIP Support, In-Browser File Preview, and the Convenience-First Metric

**version** v0.6.17  
**date** 24 Feb 2026  
**from** Human (project lead)  
**to** Developer (lead), Architect, Designer  
**type** Dev brief — immediate implementation priority  

---

## The Trigger

An early user sent a file via SG/Send — a ZIP containing a markdown file and an issues JSON file. The recipient opened it on an Android phone. What happened:

1. Downloaded the file ✅ (but the Chrome pop-up disappeared, had to hunt for it in the Files app)
2. Unzipped it ✅ (Android could extract)
3. Read the markdown file ❌ (no way to open `.md` on the phone)

**The recipient literally could not read the file they were sent.** On their phone, the workflow broke at the last step. This is unacceptable attrition.

---

## The Strategic Principle: Convenience-First

> **Can we make it so that people share files securely via our platform not because they are secure, but because it's the most convenient and easiest way to share a file?**

This is a key metric. Security is the foundation, but convenience is what drives adoption. If SG/Send is the easiest way to send a ZIP file to someone and have them actually be able to open and read it — on any device, in any browser, with no extra apps — then security comes for free. Users choose convenience. We make convenience secure.

This connects directly to the DRU/MRU/ARU metric (v0.6.14): usage is our North Star, and usage is driven by convenience. Every time someone chooses SG/Send because "it's just easier" — even if they don't care about encryption — that's a win.

## Feature 1: ZIP File Support in the Browser

### Open ZIPs Without Downloading

When a user receives a ZIP file (encrypted or not), they should be able to browse its contents directly in the SG/Send browser — the same folder-and-file UI we already have for data rooms.

```
User receives: project-files.zip.sgenc

In the browser:
  1. Decrypt (automatic — private key in browser)
  2. ZIP contents displayed as a folder tree:
     
     📁 project-files/
     ├── 📄 README.md          [preview] [download]
     ├── 📄 issues.json        [preview] [download]
     ├── 🖼️ screenshot.png     [preview] [download]
     ├── 📄 report.pdf         [preview] [download]
     └── 📁 src/
         ├── 📄 index.html     [preview] [download]
         └── 📄 style.css      [preview] [download]

  3. Click any file → preview inline
  4. Download individual files or the whole ZIP
```

### Technical Approach

ZIP parsing in the browser is well-established:

| Library | What It Does | Size |
|---|---|---|
| **JSZip** | Read and create ZIP files in JS. Mature, widely used. | ~45KB gzipped |
| **fflate** | Faster, smaller ZIP/DEFLATE. Modern, performant. | ~8KB gzipped |
| **zip.js** | Full-featured, supports streaming, large files, encryption. | ~30KB gzipped |

One import away. The ZIP is already decrypted in memory (we decrypted the `.sgenc` wrapper). We just need to parse the ZIP contents and render the folder tree.

### Connection to Portable Data Rooms

The `.sgroom` format (v0.6.14) IS a ZIP file. ZIP browser support is not just a nice-to-have — it's infrastructure for portable data rooms. The same viewer that opens a user's ZIP attachment opens a portable data room. Same UI, same code path, same experience.

## Feature 2: In-Browser File Preview

### The Core Requirement

When a user clicks on a file — whether inside a ZIP, in a data room, or as a standalone decrypted file — they should see a preview without downloading. The browser already knows how to render most common formats. We just need to present them.

### File Types and Preview Strategy

| File Type | Extensions | Preview Method | Library / Approach |
|---|---|---|---|
| **Images** | .png, .jpg, .gif, .webp, .svg | Native `<img>` tag | None needed — browser-native |
| **PDF** | .pdf | PDF.js (Mozilla's PDF renderer) or native browser PDF viewer via `<iframe>` / `<object>` | PDF.js (~300KB) or browser-native |
| **Markdown** | .md | Parse to HTML, render | marked (~7KB) or markdown-it (~30KB) |
| **Plain text** | .txt, .csv, .log, .json, .yaml | Render in `<pre>` with syntax highlighting | Prism.js or highlight.js |
| **Code** | .js, .py, .html, .css, .ts, .sh | Syntax-highlighted `<pre>` | Prism.js (~5KB core) |
| **HTML** | .html | Sandboxed `<iframe>` | Browser-native (sandbox for security) |
| **Word** | .docx | Extract text/structure, render as HTML | mammoth.js (~25KB) — already available in our artifact environment |
| **Spreadsheet** | .xlsx, .csv | Render as HTML table | SheetJS (~200KB) — already available |
| **Audio** | .mp3, .wav, .ogg | Native `<audio>` element | Browser-native |
| **Video** | .mp4, .webm | Native `<video>` element | Browser-native |
| **Archives** | .zip, .tar.gz | Folder tree (see Feature 1) | JSZip / fflate |

### The Preview UI

```
┌──────────────────────────────────────────────────┐
│  📄 README.md                          [↓ Download] │
├──────────────────────────────────────────────────┤
│                                                  │
│  # Project Overview                              │
│                                                  │
│  This project implements the PKI-secured         │
│  agentic workflow described in the v0.6.14       │
│  architecture brief.                             │
│                                                  │
│  ## Getting Started                              │
│                                                  │
│  1. Clone the repo                               │
│  2. Run `npm install`                            │
│  3. ...                                          │
│                                                  │
└──────────────────────────────────────────────────┘
```

Simple, clean, mobile-friendly. The file name at the top, a download button, and the preview filling the rest of the viewport. On mobile, this must be full-screen and touch-friendly.

### Priority Order

Based on what users actually send and what caused the trigger incident:

| Priority | Format | Why |
|---|---|---|
| **P0** | Markdown (.md) | The trigger incident. Also: our own docs, READMEs, briefs are all markdown. |
| **P0** | Images (.png, .jpg, .gif) | Most commonly shared file type. Browser-native, zero library cost. |
| **P0** | PDF (.pdf) | Second most commonly shared document type. High business value. |
| **P0** | Plain text / JSON (.txt, .json, .csv) | Developer-friendly. Zero library cost. |
| **P1** | HTML (.html) | Mini apps, demos, reference implementations — sandboxed iframe. |
| **P1** | Code (.js, .py, .html, .css) | Developer audience shares code constantly. |
| **P2** | Word (.docx) | Business users. mammoth.js already available. |
| **P2** | Spreadsheet (.xlsx) | Business users. SheetJS already available. |
| **P3** | Audio / Video | Browser-native but bandwidth-heavy. Lower priority. |

## Feature 3: Mobile Experience

The trigger incident was on an Android phone. Mobile is where the biggest friction exists today:

| Problem | Fix |
|---|---|
| Chrome download pop-up disappears, user can't find the file | Don't download to disk at all — decrypt and preview in the browser. The file never needs to leave the browser. |
| No app to open .md files on phone | We ARE the app. Preview in our browser UI. |
| Unzipping on phone is clunky | Parse ZIP in the browser. Show folder tree. No unzipping needed. |
| Small screen, complex UI | Preview UI must be mobile-first. Full-screen preview. Large touch targets. Swipe between files in a ZIP. |

### The Mobile Flow (After Fix)

```
1. User receives WhatsApp message: "Here's the project files: [link]"
2. Taps link → opens send.sgraph.ai in mobile Chrome
3. Enters short code (or auto-decrypts if key is in browser)
4. Sees: folder tree of ZIP contents
5. Taps README.md → full-screen rendered markdown
6. Swipes → next file (screenshot.png) → full-screen image
7. Taps download icon if they want to save any file locally

No app switching. No file hunting. No "can't open this format."
```

## Feature 4: Simple File Utilities (Future — User-Demand-Driven)

The memo mentions image compression, image manipulation, and other "JavaScript-away" utilities. These are future convenience features that reinforce the "easiest way to share a file" positioning:

| Utility | What It Does | Why |
|---|---|---|
| **Image resize / compress** | Before sending, resize or compress an image | "I want to send this photo but it's 12MB" |
| **Image format conversion** | Convert PNG → JPG, HEIC → JPG | "My phone takes HEIC photos and nobody can open them" |
| **PDF merge** | Combine multiple PDFs into one | "I need to send three invoices as one file" |
| **Markdown → PDF** | Render markdown as a clean PDF | "I wrote this doc in markdown but the recipient needs a PDF" |

These should be user-demand-driven — don't build until users ask. But each one is a reason to use SG/Send even when you don't care about encryption. Every utility adds to the convenience moat.

**Implementation principle**: all processing happens in the browser. No server involvement. The file never leaves the client. This is both a privacy feature and a performance feature.
