## The Problem SaaS Companies Have

SaaS platforms accumulate files they don't need to read. HR systems hold passports. Ticketing systems hold confidential attachments. Marketing platforms hold customer data. Workflow tools hold contracts. Every one of these is a breach waiting to happen.

As a CISO: governance was a nightmare. GDPR erasure requests across dozens of SaaS platforms, each holding PII in cleartext. Supply chain data flowing through systems where the intermediary doesn't need to see the content — just pass it along.

**The industry has gotten away with this because there hasn't been a breach catastrophic enough to break the economic model.** But deepfakes, AI-powered attacks, and regulatory pressure are closing that window.

---

## The Credit Card Model

The payment card industry solved this years ago:

```
Old model:
  Customer → enters card number → SaaS stores it in cleartext → processes payment
  Problem: every SaaS company is a breach target holding card numbers

New model (tokenisation):
  Customer → enters card number → SaaS encrypts with payment provider's public key
  → SaaS stores encrypted blob (useless without provider's private key)
  → When payment needed: sends blob to provider → provider decrypts and processes
  
  Result: SaaS company never sees the card number. Breach exposes nothing useful.
```

**Apply the same pattern to files:**

```
Old model:
  User → uploads passport → SaaS stores it in cleartext → anyone with DB access can see it
  Problem: every SaaS company is holding sensitive documents in cleartext

New model (SG/Send pattern):
  User → uploads passport → browser encrypts with recipient's public key
  → SaaS stores encrypted blob (useless without private key)
  → When file needed: authorised reader decrypts in their browser
  
  Result: SaaS company never sees the file contents. Breach exposes encrypted blobs.
```

## What SaaS Companies Need

A drop-in library that encrypts files at the point of upload and decrypts only for authorised readers:

```html
<!-- SaaS company adds our JS library -->
<script src="https://cdn.sgraph.ai/sg-send.js"></script>

<!-- Their existing file upload component -->
<input type="file" id="upload">

<script>
  // Encrypt before upload — file never reaches server in cleartext
  const encrypted = await sgSend.encrypt(file, recipientPublicKey);
  
  // Upload encrypted blob through their existing API
  await fetch('/api/upload', { body: encrypted.blob });
  
  // Later, authorised reader decrypts
  const decrypted = await sgSend.decrypt(blob, readerPrivateKey);
</script>
```

The SaaS company's server only ever sees encrypted blobs. Their existing storage, their existing API, their existing UI — just with encryption wrapped around the file handling.

## Use Cases by SaaS Category

| SaaS Category | What Gets Encrypted | Who Can Decrypt |
|---|---|---|
| **HR / People** | Passports, ID documents, offer letters, medical records | HR team members with appropriate role keys |
| **Ticketing / Support** | Confidential attachments, screenshots with PII, log files | Assigned agent + customer |
| **Legal / Contract** | Contracts, NDAs, legal opinions, court documents | Named parties to the contract |
| **Financial** | Invoices, bank statements, tax documents | Finance team + external auditor (when authorised) |
| **Marketing** | Customer lists, contact databases, analytics exports | Marketing team leads |
| **Supply Chain** | Purchase orders, shipping manifests, certificates of origin | Specific supply chain participants (not the platform) |
| **Healthcare** | Patient records, lab results, imaging | Patient + treating clinician |

### The Supply Chain Example

A supply chain platform routes data between suppliers:

```
Old model:
  Supplier A → uploads order data → Platform sees everything → Supplier B downloads

New model:
  Supplier A → encrypts with Supplier B's public key → Platform stores blob
  → Supplier B decrypts → Platform never saw the contents
  
  Supplier A → encrypts component spec for Supplier C → Platform routes
  → Supplier C decrypts only the spec relevant to them
  → Platform facilitated the exchange without seeing the data
```

Each supplier sees only what's encrypted for their key. The platform knows WHO is exchanging with WHOM (routing metadata) but not WHAT (content).

## The GDPR Advantage

| GDPR Requirement | How Zero-Knowledge Helps |
|---|---|
| **Right to erasure** | Delete the encrypted blob. Even if backups exist, they're useless without the key. Rotate the key and the data is cryptographically erased. |
| **Data minimisation** | The SaaS company literally cannot access the data. They store the minimum: an encrypted blob they can't read. |
| **Purpose limitation** | Processing requires the key. No key = no processing. Access is cryptographically enforced, not just policy-enforced. |
| **Data breach notification** | If breached, encrypted blobs are exposed. Without keys, the data is not "personal data" in the GDPR sense. The breach severity is dramatically reduced. |
| **Cross-border transfer** | The data crosses borders encrypted. The receiving jurisdiction sees blobs. Reduces Schrems II concerns. |
