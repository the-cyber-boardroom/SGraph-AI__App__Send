## The Question

What would it look like if AWS, Azure, GCP, or a European cloud provider offered SG/Send-style encrypted file storage as a native service? How would they build it? What would the billing model look like? And critically — could they build it on top of our open source platform?

---

## Why This Matters: The Commoditisation Strategy

This is deliberate ILC strategy. We WANT encrypted file storage to become a commodity:

```
More environments running encrypted storage
    → More places our platform can deploy
    → More recipients in our network
    → More value in our trust web
    → More demand for the services we build ON TOP (scanning, trust, content analysis)
```

The infrastructure is commoditised. The intelligence (trust web, scanning chain, content trust) is the value layer. Every cloud provider that adds compatible encrypted file storage adds value to our network.

## Research: What Each Provider Would Build

### For Each Provider, Map Out:

| Dimension | What to Research |
|---|---|
| **Existing building blocks** | What storage, KMS, PKI, and encryption services does the provider already have? |
| **What they'd need to add** | Client-side encryption workflow, zero-knowledge storage, PKI key management in the browser |
| **Architecture** | How would the service be structured using their existing primitives? |
| **Billing model** | Per-file? Per-GB? Per-operation? How does scanning fit into billing? |
| **Integration points** | Where would SG/Send fit? As the file ingestion point? As a compatible API? |
| **Competitive advantage** | The provider can do BOTH storage AND the scanning/verification pod — we can't (zero-knowledge constraint). They could offer the full stack. |

### AWS Blueprint

```
Client (browser with SG/Send JS library)
    │
    ├── Encrypt file client-side (Web Crypto API)
    ├── Upload encrypted blob to S3 (standard S3 API)
    ├── PKI keys managed in browser (or AWS KMS for enterprise)
    │
    ▼
AWS Services:
    ├── S3: Encrypted blob storage (the provider already does server-side 
    │        encryption — adding client-side encryption is the shift)
    ├── Lambda: Scanning pod (receives file, scans, signs result, re-encrypts)
    ├── KMS: Key management (for enterprise customers who want AWS-managed keys)
    ├── CloudFront: File delivery
    ├── SNS/SQS: Notifications (file ready, scan complete)
    └── IAM: Access control (maps to our trust tiers)

Billing:
    ├── S3 storage: per-GB/month (existing)
    ├── Scanning Lambda: per-invocation (new)
    ├── KMS operations: per-key/per-operation (existing)
    └── Data transfer: per-GB (existing)
```

### Research Each Provider

| Provider | Key Questions |
|---|---|
| **AWS** | How does this compare to S3 + KMS + client-side encryption? What's missing? Could they use our open source platform on Lambda? |
| **Azure** | Azure Blob Storage + Key Vault + Confidential Computing. Does Azure's TEE (Trusted Execution Environment) solve the scanning pod problem? |
| **GCP** | Cloud Storage + Cloud KMS + Confidential VM. What about GCP's BeyondCorp model for access control? |
| **European providers** (OVH, Hetzner, Scaleway) | Sovereignty matters. GDPR-native. What would a European-sovereign encrypted file service look like? |
| **Cloudflare** | R2 storage + Workers. Workers as scanning pods? Cloudflare already does browser isolation. Natural fit? |

## The Integration Model

How SG/Send plugs into each provider:

```
Option A: SG/Send as the entry point
    User → SG/Send UI → encrypts → uploads to [provider storage]
    (Our UI, their storage)

Option B: Provider-native UI, our protocol
    User → Provider's UI → uses SG/Send JS library → encrypts → provider storage
    (Their UI, our encryption protocol)

Option C: Full provider service built on our open source
    Provider builds their own service using our open source platform
    Compatible APIs → files can transfer between providers
    (Their everything, our protocol and network)
```

**Option C is the commoditisation goal.** Every provider that adopts the protocol becomes a node in our network. Files can flow between AWS, Azure, GCP — all using the same encryption and trust model.

---

## Network Effect

Every provider that implements a compatible service becomes:

| Role | What It Adds |
|---|---|
| **Another storage backend** | More places to store encrypted files |
| **Another scanning pod** | Provider can run scanning services (they have the compute) |
| **Another trust node** | Provider's users join the trust web |
| **Another recipient endpoint** | "Send encrypted files to anyone on AWS/Azure/GCP" |

The more providers, the more valuable the network. This is the moat: not the code (which is open source) but the trust web and the interoperability.
