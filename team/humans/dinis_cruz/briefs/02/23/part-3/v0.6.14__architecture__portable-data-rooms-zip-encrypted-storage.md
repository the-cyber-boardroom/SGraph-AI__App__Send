## The Foundation: MemoryFS Abstraction

The entire SG/Send platform is built on MemoryFS, which abstracts storage from logic. The cache system doesn't care WHERE the bytes live — it just needs a storage backend that implements the interface. This is already proven:

| Backend | Status | Use Case |
|---|---|---|
| **Memory** | ✅ Working (tests) | Unit tests, ephemeral environments |
| **S3** | ✅ Working (production) | Production storage |
| **File system** | ✅ Working | Local development |
| **SQLite** | Possible | Single-file database backend |
| **ZIP file** | **NEW — this brief** | Portable, shippable encrypted storage |

Because MemoryFS abstracts the storage, adding a ZIP backend means the entire data room — files, configuration, permissions, index — can live inside a single ZIP file. Same encryption. Same PKI. Same access control. Just a different storage backend.

## The Concept: Ship the Data Room

Instead of giving someone access to a hosted data room, SEND them the data room as a file:

```
Traditional:
  "Here's a link to the data room: send.sgraph.ai/room/abc123"
  → Requires internet connection
  → Requires the server to be running
  → Data lives on our infrastructure

Portable:
  "Here's the data room: room-abc123.sgroom"
  → Self-contained ZIP file with all encrypted data
  → Open locally with SG/Send viewer (or in-browser)
  → No server needed
  → Data travels with the recipient
```

The `.sgroom` file IS the data room. Everything is inside: encrypted files, folder structure, permissions, branding, encryption metadata. The recipient opens it with their private key and sees the data room exactly as it was.

## The Supply Chain Use Case

This is where portable data rooms become powerful:

```
Manufacturer → Assembler → Distributor → Retailer

Step 1: Manufacturer creates a data room (ZIP)
  Contains: product specs, certificates of origin, test results
  Encrypted for: Assembler's public key
  Signs: with Manufacturer's private key

Step 2: Manufacturer ships the .sgroom file to Assembler
  (email, USB drive, any transfer method)

Step 3: Assembler opens the data room
  Decrypts with their private key
  Reads the specs, certificates
  ADDS: assembly reports, quality checks
  Signs additions with Assembler's private key
  Re-exports the .sgroom (now with both parties' data)

Step 4: Assembler ships updated .sgroom to Distributor
  Distributor can see:
    - Everything Manufacturer encrypted for them (if any)
    - Everything Assembler encrypted for them
    - Cannot see data encrypted for other parties only

Step 5: Each supply chain participant adds their data
  The .sgroom file accumulates signed, encrypted data
  Full provenance chain: who added what, when, signed by whom
```

**The ZIP file is the single source of truth that travels through the supply chain.** Each participant adds their piece. Each piece is signed. The full history is verifiable.

## How It Works Technically

### The .sgroom File Format

A `.sgroom` file is a ZIP archive with a defined structure:

```
room-abc123.sgroom (ZIP archive)
├── manifest.json              ← Room metadata (name, version, created, format version)
├── config/
│   ├── room.json              ← Room settings, access tiers
│   ├── keys.json              ← Public keys of all participants
│   └── branding.json          ← Logo reference, colours, title
├── files/
│   ├── {guid1}.enc            ← Encrypted file blobs
│   ├── {guid2}.enc
│   └── ...
├── index.json.enc             ← Encrypted file index (names, folders, metadata)
│                                 (encrypted per-participant: each sees their view)
├── audit/
│   ├── log.json.enc           ← Signed audit trail (who did what, when)
│   └── signatures/
│       ├── {participant1}.sig ← Participant's signature over their contributions
│       └── {participant2}.sig
└── assets/
    └── logo.png               ← Room branding (unencrypted — it's cosmetic)
```

### Opening a .sgroom File

```
1. User opens .sgroom in SG/Send viewer (browser-based or desktop)
2. Viewer reads manifest.json → identifies the data room format
3. Viewer reads config/keys.json → checks if user's public key is listed
4. If listed: user decrypts index.json.enc with their private key
   → Shows only the files/folders they have access to
5. User browses, reads files (decrypting each on demand)
6. User can ADD files (encrypted, signed with their key)
7. User EXPORTS the updated .sgroom
   → New ZIP with additions, updated index, updated audit log
```

### The MemoryFS Integration

```python
# Existing MemoryFS backend pattern
from memory_fs import MemoryFS

# Current: S3 backend
fs = MemoryFS(backend='s3', bucket='data-room-abc123')

# New: ZIP backend
fs = MemoryFS(backend='zip', path='room-abc123.sgroom')

# The rest of the code is IDENTICAL
# Same cache operations, same file operations, same encryption
file = fs.get(guid)
fs.put(guid, encrypted_data)
```

Because MemoryFS abstracts the backend, the entire data room codebase works with ZIP storage without modification. The ZIP backend just reads from / writes to the ZIP archive instead of S3.

## Use Cases

| Use Case | Why Portable? |
|---|---|
| **Supply chain** | Data room travels with the goods. Each participant adds their part. Full provenance. |
| **Legal discovery** | Ship a data room to opposing counsel. They open it offline. Access-controlled — they see only what's been disclosed. |
| **Regulatory submission** | Package all required documents into a signed .sgroom. Submit as a single file. Regulator opens with their key. |
| **Air-gapped environments** | Defence, government, critical infrastructure — no internet access. The data room is a file on a USB stick. |
| **Archival** | Archive a completed data room as a single file. All encryption keys, all files, all audit logs. Self-contained for decades. |
| **Backup** | Export any hosted data room as a .sgroom for offline backup. Restore by uploading the ZIP to a new server. |

## Offline-First, Sync Later

A portable data room can work offline and sync when connectivity returns:

```
1. Export .sgroom from hosted data room
2. Work offline — add files, make changes
3. Reconnect — upload modified .sgroom
4. Server merges changes (conflict resolution based on timestamps and signatures)
5. Other participants see the updates
```

This makes SG/Send work in low-connectivity environments: field offices, construction sites, ships, aircraft.

## Security Properties

| Property | How It Works in a ZIP |
|---|---|
| **Encryption at rest** | All files are encrypted blobs inside the ZIP. Opening the ZIP reveals nothing useful. |
| **Access control** | Each participant's view (index) is encrypted with their key. They see only their files. |
| **Provenance** | Every file and every addition is signed by the contributor's private key. |
| **Tamper detection** | The audit log is signed. Modifying any file without the correct private key is detectable. |
| **Key rotation** | Re-export the .sgroom with new keys. Old versions with old keys are still encrypted — forward secrecy depends on key management. |
