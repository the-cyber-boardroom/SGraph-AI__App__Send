**version** v0.6.14  
**date** 23 Feb 2026  
**from** Human (project lead)  
**to** Ambassador (lead), Architect, Alchemist  
**type** Research + strategy brief — email and messaging integration

## The Problem

Why can't I import my public keys into my email system? Why can't I have confidence that a message from my daughter is actually from my daughter? Why can PGP encrypt emails but nobody uses it? Why are deepfakes and voice impersonation getting easier while identity verification in messaging stays stuck in 2005?

**The trust web we're building has no way to reach people through the channels they actually use: email, WhatsApp, SMS, voice calls.**

If a recipient has a public key registered in our system, we should be able to send them an encrypted message through ANY channel — and they should be able to verify it came from us.

---

## The Gap

| What We Have | What We Don't Have |
|---|---|
| PKI key pairs in the browser | Keys that work in email clients |
| Encrypted file transfer via send.sgraph.ai | Encrypted file transfer via email attachment |
| Trust web between SG/Send users | Trust signals in email, WhatsApp, SMS |
| Public key registry | Integration with email/messaging providers |

**We can send encrypted files through our platform. We cannot send encrypted files through email.** This limits our reach to people who visit send.sgraph.ai. Email reaches everyone.

## Integration Approaches

### Approach 1: SG/Send as Email Bridge

We don't change email. We use email as the NOTIFICATION channel and SG/Send as the DELIVERY channel:

```
Sender:
  1. Uploads file to SG/Send
  2. SG/Send sends an email to recipient:
     "You have a secure file waiting. Open send.sgraph.ai and enter code: 847293"
  3. Recipient opens SG/Send, enters code, gets file

This is what we can do TODAY. No email provider integration needed.
```

### Approach 2: Encrypted Email Attachments

Send the encrypted file AS an email attachment, with a link to decrypt:

```
Sender:
  1. Uploads file to SG/Send
  2. SG/Send creates an email with:
     - Encrypted file as attachment (.sgenc file)
     - HTML body explaining how to decrypt
     - Link to send.sgraph.ai/decrypt
  3. Recipient downloads attachment, drags it onto send.sgraph.ai/decrypt
  4. Browser decrypts using recipient's private key

Advantage: file travels via email (familiar channel)
Challenge: email providers may strip unfamiliar attachments
```

### Approach 3: Email Provider Plugin / Extension

A browser extension or email plugin that adds PKI to the email experience:

```
Gmail with SG/Send extension:
  - [🔐 Send Secure] button next to Send
  - Encrypt email body + attachments with recipient's public key
  - Recipient's extension auto-decrypts when opened
  - Public keys discovered via our registry

Outlook with SG/Send add-in:
  - Same functionality via Outlook add-in API
  - Enterprise deployment via admin policy
```

### Approach 4: Lobby Email Providers to Add Native PKI

The strategic play: convince email providers that native PKI support is a competitive advantage.

```
Pitch to Gmail / Outlook / ProtonMail:
  - "Your users want to verify who sent them a message"
  - "Deepfakes make impersonation trivial — PKI makes it verifiable"
  - "Import public keys → verify sender identity → trust indicator in the UI"
  - "We provide the open source PKI infrastructure. You integrate it."
  
  What they'd add:
  - Public key import (from our registry or manual)
  - Signature verification (✅ verified sender)
  - Optional: encrypt email body with recipient's public key
  - Trust indicator: "This sender's identity has been verified by [SG/Send trust web]"
```

## Messaging Platforms

The same logic applies to WhatsApp, Signal, Telegram, and SMS:

| Platform | What PKI Would Enable |
|---|---|
| **WhatsApp** | Verify the message is from who it claims to be (beyond phone number). Signed messages. |
| **Signal** | Already has encryption — add public key registry integration for cross-platform identity. |
| **SMS** | The most vulnerable channel. PKI signatures would eliminate spoofing. |
| **Voice/Video calls** | Verify the caller's identity before answering. Counter deepfake impersonation. |

The deepfake threat makes this urgent: if I can clone your daughter's voice and call you from a spoofed number, the only defence is cryptographic identity verification. "This call is signed by your daughter's private key" is unforgeable.

---

## Corporate Strategy: Network Effect

Every email provider, messaging app, or SaaS company that integrates PKI support adds value to the trust web:

```
More providers support PKI
    → More users have public keys
    → More public keys in the registry
    → Stronger trust web
    → More reason for the NEXT provider to support PKI
    → Network effect compounds
```

**The Ambassador and Alchemist should be promoting this.** Every external adoption of PKI infrastructure increases the value of our network.
