## The Concept

Data room configuration is managed through GitHub. The data room itself has no idea GitHub exists — it just runs. A separate admin UI is the only thing that talks to GitHub, committing changes via pull requests. GitHub is the persistent layer: files, users, permissions, branding — all stored as code in a repo.

**Data Room as Code**: the same way infrastructure-as-code manages servers, we manage data rooms through version-controlled configuration.

---

## What GitHub Stores

| What | Format | Example |
|---|---|---|
| **Files** | Encrypted blobs committed to the repo | `/files/{guid}.enc` |
| **Users / permissions** | JSON or YAML configuration | `/config/users.yaml` — who has access, what roles, what keys |
| **Registry** | Public key registry for the data room | `/config/registry.yaml` — member public keys |
| **Branding** | Logos, titles, colour schemes | `/config/branding.yaml` + `/assets/logo.png` |
| **Room configuration** | Room settings, scanning requirements, access tiers | `/config/room.yaml` |
| **Documentation** | Auto-generated docs (from QA pipeline) | `/docs/` — generated by CI |

## Architecture: The Data Room Doesn't Know

```
┌──────────────────┐     ┌──────────────────┐     ┌──────────────┐
│  Data Room Admin  │     │  GitHub Repo      │     │  Data Room    │
│  UI (separate)    │     │  (persistent      │     │  (runtime)    │
│                   │     │   layer)          │     │               │
│  • Manage users   │────►│  • config/        │────►│  • Serves     │
│  • Upload files   │     │  • files/         │     │    files      │
│  • Set branding   │     │  • assets/        │     │  • Enforces   │
│  • Set permissions│     │  • docs/          │     │    permissions│
│                   │     │                   │     │  • Shows      │
│  Has: GitHub token│     │  Has: Git history │     │    branding   │
│  Makes: commits / │     │  CI: auto-deploy  │     │               │
│  pull requests    │     │  on push          │     │  Has: NO      │
│                   │     │                   │     │  GitHub access│
└──────────────────┘     └──────────────────┘     └──────────────┘
```

Key separation: the admin UI is the ONLY component with GitHub access. The data room runtime reads configuration but never writes to GitHub. This limits the blast radius — if the data room is compromised, the attacker doesn't get the GitHub token.

## The Admin UI

A separate, simple web interface for managing data room configuration:

| Action | What Happens in GitHub |
|---|---|
| Add a user | Commits updated `users.yaml` with new public key and permissions |
| Upload a file | Commits encrypted blob to `/files/`, updates file index |
| Change branding | Commits new logo/colours to `/assets/` and `/config/branding.yaml` |
| Update permissions | Commits updated `users.yaml` |
| Change room settings | Commits updated `room.yaml` |

Every change is a Git commit. Full history. Full auditability. Rollback is `git revert`.

### Where the Admin UI Runs

| Environment | Use Case |
|---|---|
| **Locally** | Human runs it on their machine. Has GitHub token locally. Most secure — token never leaves the machine. |
| **CI pipeline** | GitHub Actions has the token. Automated changes (e.g., key rotation, doc generation) run in CI. |
| **Deployed (future)** | Hosted admin UI with authentication. Token stored as secret. For when multiple admins need access. |

For now: local + CI. Deployed admin comes later.

## GitHub Folder Structure

```
data-room-{name}/
├── config/
│   ├── room.yaml           ← Room name, description, scanning requirements, access tiers
│   ├── users.yaml           ← User public keys, roles, permissions
│   ├── registry.yaml        ← Public key registry for this room
│   └── branding.yaml        ← Logo path, colours, title, description
├── assets/
│   ├── logo.png             ← Room logo
│   └── favicon.ico
├── files/
│   ├── {guid1}.enc          ← Encrypted file blobs
│   ├── {guid2}.enc
│   └── index.json.enc       ← Encrypted file index (names, folders, metadata)
├── docs/
│   ├── index.md             ← Auto-generated documentation
│   └── screenshots/         ← Auto-generated by QA pipeline
├── .github/
│   └── workflows/
│       ├── deploy.yaml      ← Deploy data room on config change
│       └── docs.yaml        ← Regenerate docs on change
└── CLAUDE.md                ← Agentic team configuration
```

## New Repo, New Agentic Team

This data room admin gets its own repo with its own agentic team — following the same bootstrap pattern established for the QA Villager project:

| What | Detail |
|---|---|
| **Repo** | New repo: `sgraph/data-room-admin` (or similar) |
| **Agentic team** | Bootstrapped from the same pattern: clone the bootstrap pack approach from the QA project |
| **Roles needed** | Developer, Architect, DevOps, Designer, Sherpa (5 roles — focused team) |
| **Learns from** | Main SG/Send repo (practices, IFD components, admin UI patterns) and QA repo (documentation generation, CI pipeline) |

### The Pattern: Every New Repo Gets an Agentic Team

This is now the third repo following this pattern (main SG/Send, QA Villager, data room admin). The bootstrap process should be getting easier each time. Eventually: create a new repo → run the bootstrap → agentic team is ready in minutes.

## Auto-Generated Documentation

The data room admin repo uses the same QA documentation pipeline:

```
Config change committed to GitHub
    │
    ▼
CI pipeline triggers
    │
    ├── Deploy updated data room configuration
    │
    ├── Run QA smoke test against the data room
    │   ├── Open data room in headless browser
    │   ├── Screenshot: landing page with branding
    │   ├── Screenshot: file listing
    │   ├── Screenshot: user permissions page
    │   └── Generate markdown docs from screenshots
    │
    └── Publish docs to GitHub Pages
```

The data room's documentation is always up to date because it's generated from the actual running data room, not written by hand. Same principle as the QA living documentation brief.

## Workflow: Creating a New Data Room

```
1. Create a new GitHub repo from the data-room template
2. Edit config/room.yaml (name, description, scanning requirements)
3. Edit config/branding.yaml (logo, colours)
4. Add initial users to config/users.yaml (public keys + roles)
5. Commit and push
6. CI pipeline deploys the data room
7. QA pipeline takes screenshots and generates docs
8. Data room is live, documented, and version-controlled
```

Adding a file:
```
1. Open admin UI (local or web)
2. Select file to upload
3. File is encrypted in the browser (admin's key or room key)
4. Admin UI commits encrypted blob to /files/ and updates index
5. Push to GitHub
6. CI deploys updated file listing
```
