## What to Research

A comprehensive survey of which communication platforms and identity providers currently support PKI, public key registries, and encrypted messaging. The goal: find platforms we can integrate with TODAY, and understand the gaps where a Chrome extension could bridge.

## Research Matrix

### Communication Platforms

For each platform, answer:

| Question | Why It Matters |
|---|---|
| Does it support PGP / S/MIME / any PKI? | Can we send PKI-encrypted messages natively? |
| Does it have a public key registry? | Can we look up a recipient's public key through the platform? |
| Can external systems send messages via API? | Can SG/Send push notifications / encrypted messages into the platform? |
| Does it support rich messages (HTML, attachments, buttons)? | Can we embed a "decrypt this" link in a message? |

## The Integration Opportunity

If a platform supports PKI or has an API for sending messages, we can integrate:

```
Tier 1: Native PKI support
  → We send PKI-encrypted messages directly through their platform
  → Recipient decrypts natively
  → Best experience, no extra tools needed

Tier 2: API for sending messages, no PKI
  → We send a notification through their API: "You have an encrypted file"
  → Recipient clicks link to send.sgraph.ai to decrypt
  → Good experience, one extra click

Tier 3: No API, but has browser extension support
  → We build a Chrome extension that adds PKI encryption
  → Extension encrypts/decrypts in the browser
  → Works but requires extension installation

Tier 4: No integration possible
  → Copy-paste a link. Current state for most platforms.
```

### Chrome Extension Architecture

For Tier 3 platforms (e.g., Gmail without native PKI):

```
Chrome Extension: SG/Send Secure Mail
├── Content script: detects email compose / read views
├── Compose: [🔐 Encrypt] button added next to Send
│   └── Encrypts subject + body + attachments with recipient's public key
│       (looked up from SG/Send registry)
├── Read: detects encrypted content, offers [🔓 Decrypt] button
│   └── Decrypts using user's private key (stored in browser / pod)
├── Key management: links to SG/Send PKI (key generation, import, registry)
└── Settings: which platforms to enable, key preferences

Platforms to target first:
  1. Gmail (largest market, good extension APIs)
  2. Outlook Web (second largest, Office add-in API)
  3. LinkedIn messaging (professional context — high-value messages)
```

**Trade-off**: encrypted email can't be searched by the email provider. This is a feature (privacy) AND a limitation (no search). The extension could maintain a local encrypted search index.

---

## The Core Principle

Every integration must maintain: **the server (email provider, messaging platform) never sees cleartext.** If the platform's architecture requires server-side access to message content (e.g., for search, compliance scanning), we encrypt anyway and accept the trade-off (no server-side search of encrypted content).
