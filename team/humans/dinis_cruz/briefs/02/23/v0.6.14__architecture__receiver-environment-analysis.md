# Architecture Brief: Receiver Environment Analysis

**version** v0.6.14  
**date** 23 Feb 2026  
**from** Human (project lead)  
**to** Architect (lead), AppSec, Developer  
**type** Architecture brief — client-side security posture collection  

---

## The Concept

When a receiver opens a file, their browser reveals information about their environment: IP address, timezone, browser type, language, network properties. Today, this information is ignored. Instead, we should **collect it on the client side and share it with the sender** so the sender can make an informed decision about whether this is a safe delivery.

### The Reversed Security Model

| Server-Side | Client-Side |
|---|---|
| Knows as little as possible about the user | Knows everything the browser can reveal |
| Cannot see IP, location, browser details | Can see IP, location, browser, network quality |
| Zero-knowledge — by design | Full-knowledge — by design |
| Cannot make trust decisions about users | CAN make trust decisions and share them with the sender |

The server stays zero-knowledge. All environment analysis happens in the receiver's browser and is shared with the sender via encrypted messages. The server never sees the analysis.

---

## What the Browser Can Reveal

| Data Point | How to Collect | What It Tells You |
|---|---|---|
| **IP address** | `fetch("https://api.ipify.org")` or server-side header (but we use client-side) | Geographic location, ISP, VPN detection, known malicious IP ranges |
| **Timezone** | `Intl.DateTimeFormat().resolvedOptions().timeZone` | Expected geography vs actual IP geography mismatch |
| **Language** | `navigator.language` / `navigator.languages` | Expected language for this recipient |
| **Browser** | `navigator.userAgent` | Browser type, version, platform. Is it a real browser or a headless bot? |
| **Platform** | `navigator.platform` | OS (Windows, Mac, Linux, Android, iOS) |
| **Screen resolution** | `window.screen.width/height` | Real device vs VM/headless |
| **WebGL renderer** | `WebGLRenderingContext.getParameter(RENDERER)` | Hardware fingerprint. VMs have distinctive renderers. |
| **Connection type** | `navigator.connection.effectiveType` | 4G, WiFi, etc. Unusual for expected context? |
| **Do Not Track** | `navigator.doNotTrack` | Privacy preference |
| **Cookies enabled** | `navigator.cookieEnabled` | Basic browser configuration |
| **Secure context** | `window.isSecureContext` | Is HTTPS enforced? |
| **Service Worker support** | `'serviceWorker' in navigator` | Can the trust anchor / pod be installed? |

### Enriched Data (External Lookups)

With the receiver's consent, the browser can make API calls to enrich the environment data:

| Data Point | API | What It Tells You |
|---|---|---|
| **IP geolocation** | ipinfo.io, ip-api.com | City, country, ISP, ASN |
| **IP reputation** | AbuseIPDB, VirusTotal | Is this IP known for malicious activity? |
| **VPN/proxy detection** | Various IP intel APIs | Is the receiver hiding behind a VPN? |
| **ISP classification** | IP intel | Residential, business, datacenter, mobile |

## How It Works

### The Environment Report

The receiver's browser generates an environment report:

```json
{
  "timestamp": "2026-02-23T10:00:00Z",
  "ip": {
    "address": "203.0.113.42",
    "country": "GB",
    "city": "London",
    "isp": "BT Group",
    "type": "residential",
    "vpn_detected": false,
    "reputation": "clean"
  },
  "browser": {
    "name": "Chrome",
    "version": "122.0",
    "platform": "macOS",
    "language": "en-GB",
    "secure_context": true,
    "service_worker": true
  },
  "device": {
    "screen": "2560x1440",
    "timezone": "Europe/London",
    "webgl_renderer": "Apple M2 Pro"
  },
  "security_posture": {
    "https": true,
    "secure_browser": true,
    "known_malicious_ip": false,
    "vpn": false,
    "datacenter_ip": false
  }
}
```

### The Security Posture Score

From the environment report, derive a simple posture score:

| Factor | Positive | Negative |
|---|---|---|
| HTTPS | ✅ Secure context | ❌ Not HTTPS |
| IP reputation | ✅ Clean IP | ❌ Known malicious IP |
| VPN | Neutral (privacy choice) | ⚠️ Flagged if unexpected |
| Datacenter IP | ⚠️ Could be a bot | — |
| Timezone match | ✅ Timezone matches expected country | ⚠️ Mismatch |
| Real device | ✅ Real hardware (WebGL renderer) | ⚠️ VM or headless browser |
| Secure browser | ✅ (e.g., Surf Security, up-to-date Chrome) | ⚠️ Outdated browser |

### Sharing with the Sender

The environment report is **encrypted with the sender's public key** and attached to the download event:

```
1. Receiver visits the download link
2. Receiver's browser collects environment data
3. Browser generates the environment report
4. Report is encrypted with the SENDER's public key
   (receiver can see the report too — it's about them)
5. Encrypted report is stored as a message to the sender
6. Sender views the report in their dashboard
7. Sender decides: safe to deliver? Or ask receiver to use a different environment?
```

The server sees an encrypted blob. It has no idea what's in it.

## Transparency and Consent

The receiver must know what data is being collected and shared:

```
┌─────────────────────────────────────────────────────────────┐
│  The sender has requested environment verification.         │
│                                                             │
│  To help verify this is a secure delivery, we'll share      │
│  the following with the sender:                             │
│                                                             │
│  • Your approximate location (from IP address)              │
│  • Your browser type and version                            │
│  • Whether you're on a secure connection                    │
│                                                             │
│  We do NOT share:                                           │
│  • Your exact IP address (only city-level location)         │
│  • Your name or identity                                    │
│  • Any cookies or browsing history                          │
│                                                             │
│  [Accept and Continue]  [Decline]                           │
└─────────────────────────────────────────────────────────────┘
```

If the receiver declines, the sender is notified ("receiver declined environment verification") and can decide whether to proceed anyway.

## Anti-Gaming

The environment report is generated on the RECEIVER's browser. Could an attacker fake it?

| Attack | Defence |
|---|---|
| **Fake IP** | Use server-side IP detection (the server sees the request IP) combined with client-side reporting. Mismatch = flag. |
| **Fake browser details** | Cross-reference `navigator.userAgent` with actual browser behaviour (feature detection). Headless browsers have detectable differences. |
| **Fake WebGL renderer** | Hard to spoof convincingly. VM renderers are distinctive. |
| **Fake timezone** | Compare `Intl.DateTimeFormat` timezone with IP-derived timezone. Mismatch = flag. |

The human noted that we don't control the receiver's browser — an attacker with full browser control can fake anything. This is acknowledged. The environment report is **advisory, not authoritative.** It raises the bar for attackers but doesn't eliminate the risk. Combined with the trust web and scanning pipeline, it's one layer of many.

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | Client-side environment collection implemented (IP, browser, timezone, device) |
| 2 | Environment report generated as JSON |
| 3 | Report encrypted with sender's public key and stored as a message |
| 4 | Sender can view the environment report in their dashboard |
| 5 | Transparency: receiver is shown what data is collected before consenting |
| 6 | Receiver can decline environment sharing |
| 7 | Security posture score derived from environment factors |
| 8 | Anti-gaming measures documented (server-side IP cross-reference, feature detection) |
