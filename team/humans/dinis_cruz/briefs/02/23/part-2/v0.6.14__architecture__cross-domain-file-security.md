## The Concept

Files cross boundaries: between companies, between departments, between internal systems and external publishing services, between law firms and their clients. Every crossing is a risk surface. Every boundary is a place where a file's safety should be verified.

The scanning, trust, and PKI infrastructure we've built for sender → receiver applies directly to these cross-domain scenarios. SG/Send becomes the secure checkpoint that files pass through when crossing boundaries.

---

## Cross-Domain Scenarios

### Files Leaving an Organisation

```
Internal system → [SG/Send checkpoint] → External recipient

Example: A company publishes reports to a third-party publishing service.
Problem: How do you know the files being published are the correct ones, 
         unmodified, and free of accidental data leaks or malware?

SG/Send's role:
  - Files are scanned before leaving (CDR, malware scanning)
  - Files are signed by the organisation's PKI key (provenance)
  - Recipient can verify: this file came from this organisation, unmodified
  - Audit trail: who sent what, when, to whom
```

### Files Entering an Organisation

```
External sender → [SG/Send checkpoint] → Internal system

Example: A client sends legal documents to a law firm.
Problem: How do you know the incoming files are safe?

SG/Send's role:
  - External sender must comply with the organisation's scanning requirements
  - Files pass through third-party scanning services (PKI-verified results)
  - CDR strips dangerous content before delivery
  - Organisation's data room enforces its own security policy
```

### Files Within a Data Room

```
Data room member A → [Data room policy] → Data room member B

Example: Lawyers sharing case files in a secure data room.
Problem: Even within a trusted data room, how do you know every file is safe?
         A compromised member could upload malware.

SG/Send's role:
  - Every file uploaded to the data room is scanned per room policy
  - Scan results are attached (PKI-signed, per the scanning chain brief)
  - Room admin sets minimum scanning requirements
  - Members see scan results before opening any file
  - Trust scores apply: new members' files get extra scrutiny
```

### Files Crossing Between Data Rooms

```
Data Room A (Company) → [Cross-domain checkpoint] → Data Room B (Regulator)

Example: A company submits financial reports to a regulatory data room.
Problem: The regulator needs assurance the files are authentic, unmodified,
         and safe. The company needs assurance the regulator actually received them.

SG/Send's role:
  - Files signed by the company's PKI key (provenance)
  - Files scanned per the regulator's requirements (compliance)
  - Delivery receipt signed by the regulator's system (non-repudiation)
  - Complete audit trail: filed, scanned, delivered, received
```

## The Checkpoint Model

Every domain boundary gets a checkpoint — a point where files are verified before crossing:

```
┌─────────────┐     ┌───────────────────────┐     ┌─────────────┐
│   Domain A  │     │     SG/Send           │     │   Domain B  │
│             │     │     Checkpoint         │     │             │
│  Internal   │────►│                       │────►│  Internal   │
│  systems    │     │  • Scan (per B's      │     │  systems    │
│             │     │    requirements)       │     │             │
│  PKI key: A │     │  • CDR (strip danger) │     │  PKI key: B │
│             │     │  • Sign (A signs out,  │     │             │
│             │◄────│    B signs receipt)    │◄────│             │
│             │     │  • Audit (every step   │     │             │
│             │     │    logged and signed)  │     │             │
└─────────────┘     └───────────────────────┘     └─────────────┘
```

The checkpoint is configurable per domain:
- Domain B sets what scanning is required for incoming files
- Domain A's outbound policy defines what checks happen before files leave
- The checkpoint enforces both policies
- Both domains get signed audit trails

## Data Room Security Policies

A data room admin can set file security policies:

| Policy | What It Enforces |
|---|---|
| **Required scanning services** | "All files must be scanned by [VirusTotal + Votiro CDR]" |
| **Allowed file types** | "Only PDF, DOCX, XLSX, and images. No executables. No archives." |
| **CDR required** | "All Office documents must be CDR-processed (macros stripped) before delivery" |
| **Minimum trust score** | "Only members with trust score > X can upload files" |
| **Dual authorization** | "Files from new members require admin approval before other members can download" |
| **Maximum file size** | "No files over 50MB" |
| **Expiry** | "Files auto-delete after 90 days" |

These policies are enforced at the data room level. Individual members can add their own additional requirements on top.

## Commercial Positioning

Cross-domain file security is a well-understood enterprise need with existing market language:

| Market Term | What SG/Send Provides |
|---|---|
| **Secure file transfer (MFT)** | Encrypted transfer with PKI provenance and scanning — but zero-knowledge on the server |
| **Data Loss Prevention (DLP)** | Outbound checkpoint scanning — but the scanning is PKI-verified and the results are cryptographically bound to the file |
| **Content Disarm and Reconstruction (CDR)** | Integrated via third-party services in the scanning chain — but with full provenance and receiver-defined requirements |
| **Secure data rooms** | Already our core product — now with per-room security policies and cross-room transfer |
| **Regulatory compliance** | Signed audit trails, provenance chains, non-repudiation — all PKI-based |

The Ambassador should map these to market language for positioning. SG/Send is not "another MFT tool" — it's the first zero-knowledge platform where the receiver defines the security requirements and every step is cryptographically verified.

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | Data room security policies can be set by admin (scanning, file types, CDR, trust score) |
| 2 | Files uploaded to a data room are checked against the room's policy |
| 3 | Files that don't meet policy are rejected with explanation |
| 4 | Cross-domain transfer: file signed by sender's domain, scanned per receiver's requirements |
| 5 | Audit trail for every cross-domain transfer (signed by both parties) |
| 6 | Ambassador has market positioning document mapping SG/Send to MFT/DLP/CDR language |
