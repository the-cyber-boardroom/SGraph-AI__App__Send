## The Concept

The same way we protect receivers from malicious files, we can protect them from cognitive overload. A document that's 80% padding and 20% insight is wasting the reader's time. A document that restates existing information without adding new evidence is noise, not signal. During incidents or time-critical decisions, the ability to cut through noise and find the essential content is critical.

**Cognitive load protection means: help the receiver focus on what matters, strip what doesn't, and measure the actual information density of content.**

---

## The Analogy to File Security

| File Security | Cognitive Load Protection |
|---|---|
| Malware = harmful content that damages your system | Noise = low-value content that wastes your time |
| CDR strips dangerous elements from files | Cognitive CDR strips low-value sections from documents |
| Scanning tells you "this file is dangerous" | Analysis tells you "this document is 80% padding" |
| Receiver sets security requirements | Receiver sets cognitive load requirements ("I only have 5 minutes — give me the essentials") |
| Trust score for senders | Information density score for content |

## What to Measure

### Document-Level Metrics

| Metric | What It Measures | Example |
|---|---|---|
| **Information density** | Ratio of novel claims/data to total content | "This 5,000-word report contains 3 novel data points. Density: low." |
| **Redundancy** | How much of the content restates things said elsewhere (in this document or in cited sources) | "60% of this document paraphrases the source material without adding analysis." |
| **Source-to-opinion ratio** | Proportion of sourced claims vs unsourced assertions | "12 claims are sourced, 8 are unsourced opinions." |
| **Novelty** | What does this document add that wasn't in its cited sources? | "This article adds a new interview quote and one original analysis. Everything else is from Reuters." |
| **Decision relevance** | For a specific question/decision, how much of this document is directly relevant? | "You asked about Q4 revenue. 2 paragraphs of this 20-page report are relevant." |

### Section-Level Analysis

Within a document, identify:

- **Essential sections**: contain novel data, primary sources, key conclusions
- **Context sections**: provide background (valuable for new readers, skippable for experts)
- **Padding sections**: restate information available elsewhere, add no new insight
- **Opinion sections**: clearly labelled opinion or analysis (valuable, but different from fact)

### During an Incident

When a decision needs to be made under time pressure:

```
Incident: Critical security vulnerability reported

Available documents: 12 reports, 4 advisories, 23 news articles

Cognitive load analysis:
  "3 of 12 reports contain original vulnerability analysis.
   The rest cite these 3. Here they are, ranked by information density.
   For immediate action, read the vendor advisory (page 2, 
   'affected versions and mitigation'). Everything else is context."
```

## How It Connects to the Trust Web

Content with high information density and strong source attribution earns higher trust. Content that's mostly padding and unsourced claims earns lower trust. This feeds directly into the trust web:

| Content Quality | Trust Impact |
|---|---|
| High density, well-sourced, novel analysis | Source's trust score increases |
| Restatement of others' work without attribution | Neutral (or negative if presented as original) |
| Opinions stated as facts | Trust score decreases |
| Content later shown to be incorrect | Trust score decreases, blast radius applies |

## The Fractal Nature of Content

Content analysis operates at multiple zoom levels — the fractal nature of documents:

```
Document level:  "This report has 5 key findings"
    │
    ├── Section level:  "This section introduces 1 new dataset"
    │       │
    │       ├── Paragraph level:  "This paragraph cites 3 sources"
    │       │       │
    │       │       └── Claim level:  "This claim is backed by [signed source]"
    │       │
    │       └── Paragraph level:  "This paragraph is opinion (no sources)"
    │
    └── Section level:  "This section restates the introduction (redundant)"
```

Eventually, every claim within a document can be individually scored, sourced, and connected to the knowledge graph. Zoom in to verify a single fact. Zoom out to see the document's overall quality. Connect across documents to see how claims relate.

This connects to the semantic knowledge graph capability mentioned in the project roadmap — the ability to create analysis across documents, within paragraphs, connecting titles and claims into massive graphs.

## Technology Approach

### Phase 1: LLM-Powered Analysis

Use LLMs (via the browser pod / scanning chain) to analyse documents:

- Extract claims and classify them (sourced fact, unsourced assertion, opinion, data point)
- Identify redundancy (sections that restate other sections or external sources)
- Calculate information density metrics
- Generate reading mode summaries (key findings, executive summary, decision brief)

This is client-side — the document is analysed in the reader's browser (or pod), not on the server.

### Phase 2: Cross-Document Analysis

Compare a document against its cited sources:

- What does this document add beyond its sources?
- Are the citations accurate (does the source actually say what the document claims)?
- Are there contradictions between this document and its sources?

### Phase 3: Knowledge Graph Integration

Connect claims across documents into a graph:

- Claim A (in document 1) supports Claim B (in document 2)
- Claim C (in document 3) contradicts Claim A
- Dataset D (in document 4) is the primary evidence for Claims A, B, and C

The reader navigates the graph, not individual documents. The cognitive load is managed by the graph structure — follow the connections that matter, skip the noise.

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | Document-level metrics calculated (information density, redundancy, source ratio, novelty) |
| 2 | Section-level classification (essential, context, padding, opinion) |
| 3 | Multiple reading modes generated (full, key findings, executive summary, decision brief) |
| 4 | Analysis runs client-side (browser/pod, LLM-powered) |
| 5 | Reader can specify time constraint ("I have 5 minutes") and get appropriate summary |
| 6 | Content quality feeds into source trust scores |
