# Architecture Brief: Network-Level Threat Detection — Destroying Attacker Networks

**version** v0.6.14  
**date** 23 Feb 2026  
**from** Human (project lead)  
**to** AppSec (lead), Architect, GRC  
**type** Architecture brief — threat detection and incident response at scale  

---

## The Key Insight

It's easy to create a single fake account that looks authentic. It's nearly impossible to create thousands that don't share patterns.

The Trust Web brief (v0.6.14) describes how individual trust builds and destroys. This brief is about what happens AFTER a trust violation: not just hitting the individual account, but **mapping and destroying the entire attacker network** — and potentially doing it silently to capture more of the operation before acting.

---

## Why Individual Detection Isn't Enough

An attacker who knows the trust model can game it at the individual level:

- Create an account that behaves normally
- Send clean files for weeks to build trust
- Eventually send a malicious payload
- Get caught, account destroyed, create a new one, repeat

This is a race to the bottom. The attacker treats accounts as disposable. The cost of losing one account is low. Especially in an agentic world, creating convincing individual accounts with realistic behaviour is trivially cheap — full conversations, varied activity patterns, plausible usage.

**The defence is not at the individual level. It's at the network level.**

---

## The Network Pattern Problem

Although single-account behaviour can be made to look authentic, operating at scale always produces patterns:

### What's Hard to Hide

| Signal | Why It's Hard to Fake at Scale |
|---|---|
| **IP ranges** | 5,000 accounts need 5,000 different IPs. Residential proxies help but cluster around certain ASNs. Datacenter IPs are detectable. |
| **Timing patterns** | Accounts created in batches share temporal patterns. Activity bursts correlate. Sleep/wake cycles cluster. |
| **Behavioural similarity** | Even "randomised" behaviour follows templates. 5,000 "random" accounts share the randomisation algorithm's fingerprint. |
| **Network topology** | Fake accounts mostly interact with each other (to build trust). This creates dense clusters with few connections to the genuine user graph. |
| **Device/browser fingerprints** | Running thousands of browser instances produces fingerprint clusters (screen resolution, WebGL renderer, timezone, language configurations). |
| **Transaction patterns** | File sizes, file types, sending frequency, recipient selection — all follow statistical distributions that differ from genuine usage. |
| **Registration patterns** | Email domains, username patterns, registration timing, onboarding behaviour. |

### The Paradox of Randomness

If an attacker makes their accounts behave randomly (to avoid pattern detection), the randomness itself becomes the pattern. Genuine users don't behave randomly — they have habits, preferences, routines. A cluster of accounts with high-entropy behaviour is suspicious precisely because it's too random.

## The Detection Model: Cluster Analysis

Instead of asking "is this individual account malicious?", ask: **"does this group of accounts share patterns that suggest coordinated operation?"**

### Detection Layers

| Layer | What It Detects | How |
|---|---|---|
| **Registration clustering** | Batch-created accounts | Temporal analysis: N accounts created within T minutes from similar IP ranges or with similar registration patterns |
| **Activity correlation** | Coordinated behaviour | Statistical correlation of activity timing across accounts. Do these accounts all go active/inactive at similar times? |
| **Network topology** | Insular clusters | Graph analysis: groups of accounts that primarily interact with each other and have few connections to the broader trust graph |
| **Fingerprint similarity** | Shared infrastructure | Accounts that share device characteristics, browser configurations, or network properties |
| **Behavioural templates** | Scripted behaviour | Sequence analysis: accounts that follow similar action sequences (even with timing variation) |
| **Trust-building patterns** | Farming strategies | Accounts that follow suspiciously efficient trust-building paths (optimal recipient selection, optimal file types) |

### The Graph Approach

```
Genuine user graph:               Attacker network:

    A ─── B                        X₁ ─── X₂
   / \   / \                      / | \ / | \
  C   D E   F                   X₃  X₄  X₅  X₆
  |   |     |                    \ | / \ | /
  G   H     I                     X₇ ─── X₈
      |
      J ─── K                  Dense cluster, few
                               external connections.
Sparse, organic,               Internal connections
many cross-connections         >> external connections.
to wider network.              Detectable.
```

The attacker network appears as a **dense subgraph with disproportionately few connections to the genuine user population.** Standard community detection algorithms (Louvain, Label Propagation, spectral clustering) can identify these.

## Incident Response: Cascade Destruction

When a trust violation is detected (malware report, scanning failure, complaint), treat it as a **major incident** regardless of severity. The goal is not just to remove the offending account but to map and destroy the entire operation.

### The Incident Cascade

```
Step 1: DETECTION
  Account X₅ sends malware. Reported by recipient.
  Trust score drops to zero.

Step 2: IMMEDIATE CONTAINMENT
  X₅ is suspended.
  All files sent by X₅ are flagged for review.
  All recipients of X₅'s files are notified.

Step 3: NETWORK MAPPING
  Who has X₅ interacted with?
  Which accounts share patterns with X₅?
    - Similar IP ranges?
    - Similar registration timing?
    - Similar behavioural patterns?
    - Similar device fingerprints?
    - Part of the same dense cluster in the trust graph?
  
  Result: candidates X₁, X₂, X₃, X₄, X₆, X₇, X₈

Step 4: CASCADING TRUST REDUCTION
  All candidate accounts get trust reduced.
  Not destroyed immediately — flagged for investigation.
  "You're from a dangerous neighbourhood — less trust until 
   we understand more."
  
  Their activity is now under increased scrutiny.
  Any further violations → immediate destruction.

Step 5: HISTORICAL REVIEW
  Every file ever sent by X₅ and the candidate network:
    - Re-scan if possible (if cleartext is available via pod)
    - Notify recipients of past files
    - "A file you received from [key fingerprint] on [date] 
       is under review. We recommend not opening it."

Step 6: PATTERN EXTRACTION
  What made this network detectable?
  Extract the pattern signatures:
    - IP ranges used
    - Registration patterns
    - Behavioural templates
    - Network topology characteristics
  
  Feed these patterns back into the detection system
  to catch future operations earlier.

Step 7: NETWORK DESTRUCTION (when ready)
  Once the full network is mapped:
    - Destroy all accounts simultaneously
    - Notify all affected recipients
    - Block the identified infrastructure (IP ranges, etc.)
    - The attacker loses not just one account but their 
      entire operation
```

### The Cost to the Attacker

| What the Attacker Loses | Impact |
|---|---|
| **All accounts in the network** | Weeks or months of trust-building destroyed |
| **All infrastructure** | IP ranges, browser configurations, registration patterns now flagged |
| **Future effectiveness** | Pattern signatures extracted — similar future operations detected earlier |
| **Recipients' trust** | Any genuine users who interacted with the network are now warned |
| **Operational secrecy** | If we use silent detection (Step 7 delayed), we may have mapped the command and control infrastructure |

## Silent Detection: The Intelligence Option

Sometimes the best response to detecting an attacker is **not to let them know.**

### Why Wait?

| Reason | Benefit |
|---|---|
| **Map the full network** | The longer we observe, the more accounts and infrastructure we identify |
| **Find the command and control** | Delayed response lets us trace back to the orchestration layer |
| **Catch operational patterns** | We learn HOW the attacker operates, not just THAT they operate |
| **Bigger destruction** | When we finally act, we destroy more of the operation at once |

### When to Wait, When to Act Immediately

| Situation | Response |
|---|---|
| **Active malware being delivered to real users** | Act immediately. Protect users first. |
| **Suspected network but no active harm** | Wait. Observe. Map. Build the case. |
| **Low-trust accounts with suspicious patterns** | Monitor. Reduce trust. Flag for investigation. Don't tip off. |
| **High-confidence network identification** | Prepare coordinated takedown. Destroy all at once. |

### How to Observe Without Tipping Off

- Don't change the detected accounts' experience (no visible trust reduction)
- Don't block their files (they'd notice and create new accounts)
- Increase logging and monitoring silently
- Track their infrastructure and registration patterns
- Wait until the network is fully mapped
- Then destroy everything simultaneously

The attacker's first signal that they've been detected is when their entire operation goes dark at once.

## Feeding Back into the Trust Web

The detection patterns become inputs to the trust system:

| Detection Output | Trust System Input |
|---|---|
| Identified IP ranges | Accounts from these ranges start with lower trust |
| Registration pattern signatures | New accounts matching the pattern get extra scrutiny |
| Behavioural templates | Activity matching known attack templates flags the account |
| Network topology rules | New clusters matching the dense-subgraph pattern are flagged |

This creates a **learning system**: each attacker operation that's detected and destroyed makes the platform harder for the next attacker.

## Connection to Incident Practice

This maps directly to the project's existing incident management framework:

| Incident Element | Application Here |
|---|---|
| **Detection** | Automated: cluster analysis, pattern matching, trust score drops. Manual: user reports. |
| **Classification** | Severity based on: is malware actively being delivered? How many users affected? How large is the suspected network? |
| **Response** | Immediate containment → network mapping → cascading trust reduction → historical review → pattern extraction → network destruction |
| **Post-incident** | Pattern signatures fed back into detection. Lessons learned documented. Detection rules updated. |
| **Communication** | Affected recipients notified. If the network was customer-facing (enterprise data rooms), the enterprise admin is briefed. |

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | Trust violation triggers network mapping (not just individual account action) |
| 2 | Cluster detection identifies accounts sharing patterns with the flagged account |
| 3 | Cascading trust reduction applied to suspected network accounts |
| 4 | Historical file review triggered for all accounts in the suspected network |
| 5 | Affected recipients notified about potentially compromised files |
| 6 | Pattern signatures extracted and fed back into detection system |
| 7 | Silent detection mode documented and available (observe without tipping off) |
