## The Strategic Pattern: ILC

The project follows the Wardley Map ILC strategy: Innovate, Leverage, Commoditise.

1. **Innovate**: PKI-based zero-knowledge encryption, browser-based key management, trust webs
2. **Leverage**: Use that PKI infrastructure for file scanning chains, cross-domain security, data rooms
3. **Commoditise**: Make file security a commodity — then build the NEXT layer on top

This document describes that next layer: **applying the same trust web, PKI provenance, and network destruction patterns to content and facts instead of files and malware.**

## The Reframing

Every concept in the file security architecture has a direct analogue in content trust:

| File Security | Content Trust |
|---|---|
| A malicious file | A false fact, an opinion stated as fact, an unsourced claim |
| Malware scanning | Fact-checking, source verification, evidence validation |
| File hash (binding) | Claim hash — binding a specific claim to specific evidence |
| PKI-signed scan result | PKI-signed source attribution — "this entity said this, verifiably" |
| Trust web (sender reputation) | Source reputation — organisations and individuals who have track records of accuracy |
| Network destruction (blast radius) | When a source is discredited, every claim that depends on it is flagged |
| CDR (strip dangerous content) | Strip unsourced claims, flag opinion-as-fact, highlight what's evidenced vs what's asserted |
| Receiver sets requirements | Reader sets requirements — "only show me content with verified sources" |

## How It Works

### PKI-Signed Sources

A source (journalist, researcher, government body, company) has a PKI key pair. When they publish a claim, they sign it:

```
Claim: "GDP grew 2.3% in Q4 2025"
Source: Office for National Statistics
Signed by: sha256:ONS_public_key
Timestamp: 2026-01-15T09:00:00Z
Evidence: [link to dataset, also signed by ONS]
```

A journalist who cites this in an article includes the signed claim. The reader can:
1. Verify the journalist signed the article (the journalist stands behind it)
2. Verify the claim was signed by ONS (the original source stands behind the data)
3. Follow the chain to the original dataset
4. Check if ONS has subsequently updated or retracted the claim

### The Provenance Chain for Content

```
Original research (signed by researcher)
    │
    ├── Cited by journalist (signed by journalist)
    │       │
    │       ├── Quoted by news outlet A (signed by outlet A)
    │       │       │
    │       │       └── Shared by social media user (unsigned — low trust)
    │       │
    │       └── Cited by news outlet B (signed by outlet B)
    │
    └── Cited by policy paper (signed by think tank)
            │
            └── Referenced in legislation (signed by government)
```

Every link in the chain is verifiable. The reader can trace any claim back to its origin and check every intermediate step.

## The Trust Web Applied to Content

### Sources Build Trust the Same Way Senders Do

| Signal | Positive | Negative |
|---|---|---|
| **Claims verified over time** | Source makes claims that are later confirmed | Source makes claims that are later disproven |
| **Corrections issued** | Source corrects errors promptly (builds trust — shows integrity) | Source never corrects (destroys trust — shows carelessness or intent) |
| **Cited by trusted sources** | Trusted outlets cite this source | No trusted outlets reference this source |
| **Evidence provided** | Claims are backed by signed, verifiable evidence | Claims are unsourced or self-referential |
| **Opinion clearly labelled** | Source distinguishes fact from opinion | Source presents opinion as fact |

### Network Destruction Applied to Content

The graph approach from the attacker network brief applies directly:

```
When a foundational claim is disproven:

    Claim A (disproven)
    ├── Article 1 (cited Claim A as fact) → FLAGGED
    │   ├── Policy paper (cited Article 1) → FLAGGED
    │   └── Social media post (quoted Article 1) → FLAGGED
    ├── Article 2 (cited Claim A with caveats) → REVIEWED
    └── Article 3 (cited Claim A, later corrected) → ACKNOWLEDGED

Every downstream consumer of the bad claim is notified.
The blast radius is proportional to how deeply the claim was embedded.
```

This is the INVERSE of how misinformation works today. Currently, a false claim propagates virally and the correction barely travels. In this model, the correction propagates with the same force as the original claim — because every consumer of the claim is cryptographically linked to it and can be notified.

## Facts Have Weight

The human's core insight: **facts have evidence, opinions don't.** A fact is backed by other facts, which are backed by other facts, all the way down to primary sources. An opinion is backed by the authority of the person stating it — which is valid, but different.

### Measuring Content Value

```
Claim with:
  - 3 primary sources (all PKI-signed)
  - 2 corroborating independent analyses
  - 1 dataset (publicly verifiable)
  - 0 retractions or corrections
  → HIGH evidentiary weight

Claim with:
  - 0 primary sources
  - Cites 2 other articles (which cite each other)
  - No dataset
  - 1 partial correction issued
  → LOW evidentiary weight
```

The system doesn't decide what's TRUE — it measures what's EVIDENCED. The reader sees the evidence chain and decides.

### The Graph Shape of Facts vs. Opinions

Applying the graph insight from the network detection brief:

```
Well-evidenced fact:              Circular opinion echo:

    Source A ─── Dataset 1           Blog A ─── Blog B
   / \           │                    / \       / \
  B   C     Dataset 2               C   D ── E   F
  |   |          │                    \  |  /
  D   E     Primary research           G ── H
      |
      F ─── Independent replication   Dense cluster citing each other.
                                      No connection to primary sources.
Broad, deep, connected to            Detectable — same pattern as
primary sources and data.            attacker networks.
```

The shape of the citation graph reveals whether a claim is well-evidenced or an echo chamber. Same detection algorithms, different domain.

## Financial Flows Follow Value to the Source

Today, content economics are broken. The last-mile publisher (the outlet that rephrases someone else's research) captures most of the revenue. The original researcher or investigative journalist who produced the insight gets little.

If the provenance chain is visible and verifiable:

```
Reader pays for: Article in Outlet X
Article's actual value comes from:
  - 60% Original research by Researcher Y
  - 25% Data from Organisation Z
  - 10% Journalist's synthesis and writing
  -  5% Outlet X's distribution

Financial flow with provenance:
  Reader → Outlet X → Journalist → Researcher Y + Organisation Z
                                    (proportional to contribution)
```

The PKI provenance chain makes this trackable. Micro-payments can flow upstream to the actual sources of value. This creates economic incentive to invest in original research and investigative journalism, because the value is no longer captured only by the last-mile distributor.

This maps to the financial flow value analysis the project has done previously — the value is in the dependencies, not just the final service.

## What This Means for SG/Send

This is NOT something SG/Send builds directly. This is something that can be **built on top of SG/Send's open source PKI infrastructure:**

| SG/Send Provides | Content Trust Uses |
|---|---|
| PKI key generation and management (browser-based) | Sources generate and manage their signing keys |
| Public key registry and discovery | Source public keys are discoverable and verifiable |
| Trust web and reputation scoring | Source reputation builds through verified claims over time |
| Network detection and blast radius | Disproven claims propagate corrections through the citation graph |
| Signed, encrypted messaging | Sources communicate signed claims to journalists/publishers |
| Data rooms | Secure collaboration spaces for investigative journalism |
| Cross-domain file security | Content crossing between publishers, fact-checkers, and archives |

SG/Send's role: provide the infrastructure. The content trust layer is built by others on top of it — which drives adoption and value back to the platform.
