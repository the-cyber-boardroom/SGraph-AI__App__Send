# Architecture Brief: External File Security Services — PKI-Verified Scanning Chain

**version** v0.6.14  
**date** 23 Feb 2026  
**from** Human (project lead)  
**to** Architect (lead), AppSec, Developer  
**type** Architecture brief — zoomed-in from file scanning pipeline brief  

---

## The Problem (Zoomed In)

The server can't scan files because it never sees cleartext. The client CAN scan files — but we don't trust the client (the sender might be malicious). We need a way to get **trustworthy, verifiable** scan results from third-party services, where the scan results are:

- **Cryptographically signed** by the scanning service (provenance)
- **Readable only by the receiver** (not the sender, not the server)
- **Verifiable against the actual file** (hash binding — the scan result is for THIS file, not some clean decoy)

│          │     │ encrypted│     │  in cleartext     │     │ Receives │

Every party has a PKI key pair. Every message between them is signed and (where needed) encrypted.

## The Core Insight: The Receiver Sets the Rules

The receiver decides what scanning is required — not the sender, not the server. The receiver says: "Don't send me any file that hasn't been validated by these services." The sender must comply, and the compliance is cryptographically verifiable.

This matters because the sender is untrusted. If the sender chose the scanning requirements, they'd choose "no scanning required."

---

## The Hash as a Binding Primitive

The SHA-256 hash of the unencrypted file is the binding element across the entire chain. Every scan result is bound to a specific file hash. When the receiver decrypts the file and computes its hash, they can verify that the scan results they received are about THIS file, not some decoy.

```
file_hash = sha256(unencrypted_file_bytes)

Scan result = {
  file_hash: "a3f7...",           ← binds result to specific file
  service_id: "filescan.example",  
  result: { ... },
  timestamp: "2026-02-23T...",
  signature: sign(service_private_key, file_hash + result + timestamp)
}
```

**Privacy note**: the file hash is sensitive. Knowing the hash of a file reveals that this file exists and is being transferred. The hash should only be shared with the scanning service and the receiver — not the server (unless the receiver explicitly allows it).

## Scanning Workflow: Client Calls Service, Result Goes to Receiver

### Step by Step

```
1. SENDER's browser has the file in cleartext (before encryption)

2. SENDER computes file_hash = sha256(file_bytes)

3. SENDER uploads the file to the SCANNING SERVICE
   - The scanning service has its own PKI key pair
   - The upload is encrypted with the scanning service's public key
   - The sender's identity (public key) is attached

4. SCANNING SERVICE analyses the file:
   - File type classification (magic bytes, structure analysis, polyglot detection)
   - Malware signature scanning
   - Macro/script extraction and analysis
   - LLM-powered content analysis (optional, premium tier)

5. SCANNING SERVICE creates a SCAN RESULT:
   {
     file_hash: "a3f7...",
     classification: {
       declared_type: "application/pdf",
       actual_type: "application/pdf",
       polyglot: false,
       embedded_types: ["image/jpeg", "font/ttf"]
     },
     threats: {
       malware_signatures: [],
       suspicious_scripts: [],
       macros: { count: 0 },
       risk_level: "low"
     },
     service_id: "filescan.example",
     service_public_key: "...",
     timestamp: "2026-02-23T10:05:00Z"
   }

6. SCANNING SERVICE signs the result with its PRIVATE KEY
   - This proves the scanning service produced this result
   - Nobody can forge a scan result — it requires the service's private key

7. SCANNING SERVICE encrypts the signed result with the RECEIVER's PUBLIC KEY
   - The SENDER cannot read this (encrypted for receiver only)
   - The SERVER cannot read this
   - Only the RECEIVER can decrypt and verify it

8. The encrypted scan result blob is returned to the SENDER

9. SENDER attaches the encrypted scan result blob to the file upload
   - SENDER uploads: encrypted_file + encrypted_scan_result
   - SENDER cannot read the scan result (it's encrypted for the receiver)
   - SENDER cannot forge it (it's signed by the scanning service)

10. SERVER stores both blobs. Server sees nothing meaningful.

11. RECEIVER downloads both blobs
    - Decrypts the scan result with their private key
    - Verifies the scanning service's signature (authenticity)
    - Decrypts the file
    - Computes sha256(decrypted_file_bytes) 
    - Verifies file_hash in scan result matches actual file hash (binding)
    - If everything checks out: file is safe to open
```

### The Verification Chain

```
Receiver checks:

1. Is the scan result signed by a trusted scanning service?
   → Verify signature against known service public key
   → YES: the scanning service actually produced this result

2. Is the scan result about THIS file?
   → Compare file_hash in result with sha256(decrypted_file)
   → YES: the scan was performed on exactly this file

3. Is the scan result recent?
   → Check timestamp is within acceptable window
   → YES: this isn't a stale result from a different upload

4. Did the scan pass?
   → Read the classification, threats, risk_level
   → YES: file is classified as safe by the service
```

If any check fails, the receiver's browser shows a warning and blocks (or warns about) opening the file.

## What the Sender Cannot Do

| Attack | Why It Fails |
|---|---|
| **Skip scanning** | Receiver requires scan results. No scan result = file rejected by receiver's browser. |
| **Forge scan results** | Scan results are signed by the scanning service's private key. Sender doesn't have it. |
| **Submit a clean file for scanning, then swap in a malicious file** | The scan result is bound to a specific file hash. If the file hash doesn't match, verification fails. |
| **Read the scan results** | Scan results are encrypted with the receiver's public key. Sender can't decrypt. |
| **Modify scan results in transit** | Results are signed. Any modification invalidates the signature. |

### Why Can't the Sender Read Scan Results?

The receiver might not want the sender to know what scanning revealed. For example: a scanning service might detect that a file contains hidden metadata or steganographic content. If the sender is malicious, knowing that the scan detected this would help them evade detection next time.

By encrypting results for the receiver only, we protect the receiver's intelligence about what defences are in place.

## File Type Classification: A Harder Problem Than It Looks

The memo specifically called out file classification as non-trivial. A dedicated service for this is valuable:

| Challenge | Why It's Hard |
|---|---|
| **Magic bytes can be spoofed** | A file can have JPEG magic bytes but actually be an executable |
| **Polyglot files** | A single file that is simultaneously valid as multiple formats (e.g., a file that is both a valid PDF and a valid ZIP) |
| **Embedded content** | A PDF containing JavaScript, an Office doc containing macros, an SVG containing scripts |
| **Corrupted files** | Files that crash parsers or exploit parsing vulnerabilities |
| **Archive contents** | A ZIP file containing an EXE renamed to .jpg |

A proper file classification service would:
1. Check magic bytes (first pass)
2. Attempt to parse as declared type (structural validation)
3. Check for polyglot patterns (cross-format magic byte sequences)
4. Extract and classify embedded content recursively
5. Flag mismatches between declared extension and actual structure

This alone could be a standalone service / business model.

## Multiple Services in a Chain

The receiver can require multiple scanning services:

```
Receiver's requirements:
  1. File classification service (filescan.example)
  2. Malware scanning service (malware.example)
  3. LLM content analysis service (llmscan.example)

Sender must:
  1. Send file to filescan.example → get signed result A (encrypted for receiver)
  2. Send file to malware.example → get signed result B (encrypted for receiver)
  3. Send file to llmscan.example → get signed result C (encrypted for receiver)
  4. Upload: encrypted_file + result_A + result_B + result_C

Receiver verifies all three results before opening.
```

Each service signs independently. Each result is bound to the same file hash. The receiver verifies all of them.

### Who Pays?

The sender calls the services. The sender pays. This is by design:

- It adds cost to sending files (anti-spam / anti-abuse)
- It means the receiver doesn't pay for scanning files sent to them
- If the sender is an attacker, they pay for scanning every malicious file attempt (cost of attack increases)

## The Pod as an Independent Processing Environment

For higher assurance, a pod (from v0.4.12/v0.5.10 architecture) can act as an independent intermediary:

```
Sender → uploads file to POD (not directly to storage)
         │
         ▼
     POD decrypts file
     POD calls scanning services
     POD receives scan results
     POD re-encrypts file with its own key
     POD uploads encrypted file to storage
         │
         ▼
     Receiver downloads file encrypted by the POD
     (not by the sender)
```

In this model, the file in storage was encrypted by the pod, not the sender. This means:
- The pod verified the scan results before accepting the file
- The pod can strip malicious content (sanitise the file)
- The pod can transform the file (e.g., re-render a PDF to flatten JavaScript)
- The receiver knows the file was processed by a trusted intermediary

This is the highest-assurance mode but requires the pod to see cleartext (accepted — it's the pod's job, same as the server-side secure pod model).

## Provenance Chain: Signed at Every Step

Every operation in the chain is signed:

```
1. Sender signs: "I uploaded file with hash X at time T"
2. Scanning service signs: "I scanned file with hash X, result is Y, at time T"
3. Pod signs (if used): "I received file X, ran scans, re-encrypted as Z at time T"
4. Server records: "Encrypted blob stored at time T" (no content knowledge)
5. Receiver verifies: all signatures, all hashes, all timestamps
```

This creates a complete, cryptographically verifiable audit trail of who did what, when, and to which file. No party can deny their participation (non-repudiation). No party can forge another's signature.

## Service Discovery and Trust

How does the receiver know which scanning services to trust? How does the sender find them?

| Approach | How It Works |
|---|---|
| **Platform-curated list** | SG/Send maintains a list of approved scanning services with their public keys. Simplest. Centralised trust decision. |
| **Receiver's personal list** | Receiver specifies which services they trust (by public key). Decentralised. Receiver has full control. |
| **Trust web integration** | Scanning services have trust scores like users. Services that produce accurate results build trust. Services that miss malware lose trust. |

For MVP: platform-curated list. For production: receiver can add their own trusted services.

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | Scanning service has its own PKI key pair |
| 2 | Sender uploads file to scanning service (encrypted with service's public key) |
| 3 | Scanning service signs results with its private key |
| 4 | Results encrypted with receiver's public key (sender cannot read) |
| 5 | Results bound to file hash (receiver verifies hash match after decryption) |
| 6 | Receiver's browser verifies signature + hash binding before displaying results |
| 7 | Receiver can require specific scanning services (requirements enforced client-side) |
| 8 | Multiple scanning services supported in a chain |
| 9 | Sender pays for scanning (cost is on the sender, not receiver) |
| 10 | Full provenance chain: every step signed, timestamps verified |
