## The Elephant in the Room

An encrypted file transfer service can be used to send malware. We encrypt files so the server can't read them — which means the server can't scan them either. This is a tension that must be addressed, especially as we move toward commercial service and free tiers.

The platform must be **hostile to malicious senders** while preserving the zero-knowledge encryption model.

---

## When Scanning Can Happen

There are specific moments when file content is available in cleartext:

| Moment | Where | What's Available |
|---|---|---|
| **Before encryption (sender's browser)** | Client-side | Full file content. The sender's browser has the file before encrypting it. |
| **After decryption (receiver's browser)** | Client-side | Full file content. The receiver's browser has the file after decrypting it. |
| **Server-side (never)** | Server | Only encrypted blobs. Cannot scan. |

**All scanning happens on the client side.** The server never sees cleartext. This is non-negotiable.

## Scanning Technologies: Research Required

The AppSec team should research and evaluate the following approaches. This is a spectrum — we may use several in combination:

### Traditional Approaches

| Technology | What It Detects | How It Works in Our Model |
|---|---|---|
| **File type validation** | Files masquerading as other types (e.g., .exe renamed to .pdf) | Check magic bytes against declared file type. Client-side, instant. |
| **Hash-based lookup** | Known malware (by file hash) | Hash the file before encryption. Check against known malware hash databases (VirusTotal API, etc.). Requires network call — privacy trade-off (the hash is sent externally). |
| **Signature scanning** | Known malware patterns | Run YARA rules or ClamAV signatures against file content. Can run client-side via WASM (ClamAV compiled to WebAssembly). |
| **Macro detection** | Malicious macros in Office documents | Parse document structure, flag macros. Libraries exist for client-side Office file parsing. |
| **Script detection** | JavaScript, VBScript, PowerShell in unexpected file types | Scan for script tags, embedded code, obfuscated payloads in HTML, PDF, SVG, etc. |
| **Archive inspection** | Malware hidden in ZIP, RAR, tar.gz | Decompress and scan contents. Watch for zip bombs (recursive compression). |

### LLM-Powered Analysis

This is the novel approach. Modern LLMs are capable of detecting suspicious patterns that signature-based scanners miss:

| Capability | What the LLM Analyses | Example |
|---|---|---|
| **Macro analysis** | Full macro source code from Office documents | "This VBA macro uses Shell() to execute a downloaded payload from a suspicious URL" |
| **JavaScript analysis** | Scripts embedded in HTML, SVG, PDF | "This JavaScript uses eval() on a base64-decoded string that appears to be obfuscated shellcode" |
| **PDF structure analysis** | PDF object streams, JavaScript actions, embedded files | "This PDF contains a /JavaScript action that triggers on open and loads an external resource" |
| **Obfuscation detection** | Encoded/obfuscated payloads in any file type | "This file contains base64-encoded content that, when decoded, appears to be a PowerShell downloader" |
| **Social engineering detection** | Document content designed to trick users | "This document instructs the user to 'Enable Content' and contains a macro that runs on enable" |

### How LLM Scanning Works in Our Model

```
1. File is in cleartext in the sender's browser (before encryption)
2. Extract analysable content:
   - For Office files: extract macro source code
   - For HTML/SVG: extract script blocks
   - For PDF: extract JavaScript actions, embedded objects
   - For archives: list contents, extract any scripts
3. Send the extracted content to an LLM (via the browser pod)
   - NOT the full file — just the suspicious parts
   - The LLM analyses and returns a risk assessment
4. Display the risk assessment to the sender AND attach it to the file metadata
5. The receiver sees the risk assessment when they download
```

**Privacy consideration**: the LLM call sends extracted code snippets to a cloud LLM. For maximum privacy, this can run on a local LLM (Ollama, Chrome AI). For maximum detection quality, use a cloud LLM (Claude, GPT-4). The user chooses — it's a spectrum.

## Integration with the Trust Spectrum

The scanning results feed into the broader trust system (see Trust Web brief):

- Files that pass all scans contribute positively to sender's trust score
- Files flagged as suspicious are tracked
- If a recipient reports a file as malicious, the sender's trust score drops dramatically
- Repeat offenders get restricted (limited recipients, limited file types, lower rate limits)

---

## Free Tier and Commercial Implications

| Tier | Scanning Level |
|---|---|
| **Free / anonymous** | Full scanning required. Cannot skip. Suspicious files blocked. |
| **Authenticated (low trust)** | Full scanning. Warnings shown but user can proceed. |
| **Authenticated (high trust)** | Scanning runs but is advisory. User has earned the right to send freely. |
| **Enterprise / data room** | Configurable per data room. Admin sets scanning policy. |

This makes the platform progressively more permissive as trust is earned — hostile to new/anonymous users, flexible for established users.
