# Brief: Dual Authorization File Delivery

**version** v0.6.14  
**date** 23 Feb 2026  
**from** Human (project lead)  
**to** Architect (lead), Developer, Designer  
**type** Architecture + dev brief — two-step secure delivery  

---

## The Concept

Borrowed from financial services where sensitive transactions require "two eyes" — dual authorization before execution. Applied to file delivery: the sender doesn't release the file until the receiver has confirmed their identity, environment, and readiness.

**Normal flow**: upload → encrypt → send link → receiver downloads. One step. No verification.

**Dual authorization flow**: upload → encrypt → send challenge → receiver responds with environment + identity → sender reviews → sender releases file. Two steps. Verified delivery.

---

## The Flow

```
Step 1: SENDER initiates
┌─────────────────────────────────────────────────────────────┐
│  Sender uploads a file and marks it as "secure delivery"    │
│  File is encrypted and stored on the server.                │
│  File is NOT yet downloadable.                              │
│  Sender receives a challenge code to give the receiver.     │
└─────────────────────────────────────────────────────────────┘
        │
        │  Sender sends challenge code to receiver
        │  (WhatsApp, SMS, email, phone call, in person)
        ▼
Step 2: RECEIVER responds
┌─────────────────────────────────────────────────────────────┐
│  Receiver visits send.sgraph.ai and enters the code.        │
│  Receiver's browser:                                        │
│    - Collects environment data (IP, browser, timezone)      │
│    - Auto-generates a PKI key pair (if first time)          │
│    - Shows transparency notice (what data is shared)        │
│  Receiver confirms: "I'm ready to receive this file"        │
│  Receiver's environment report + public key are encrypted   │
│  with sender's public key and stored as a message.          │
│                                                             │
│  File is STILL not downloadable. Sender must approve.       │
└─────────────────────────────────────────────────────────────┘
        │
        │  Notification to sender: "Receiver is ready"
        ▼
Step 3: SENDER reviews and releases
┌─────────────────────────────────────────────────────────────┐
│  Sender sees:                                               │
│    - Receiver's environment report (posture score)          │
│    - Receiver's public key (for future PKI communication)   │
│    - Any flags (unusual IP, VPN, timezone mismatch)         │
│                                                             │
│  Sender decides:                                            │
│    [Release File] → file becomes downloadable for receiver  │
│    [Deny]         → file stays locked, receiver notified    │
│    [Ask to Retry] → sender requests different environment   │
└─────────────────────────────────────────────────────────────┘
        │
        │  If released:
        ▼
Step 4: RECEIVER downloads
┌─────────────────────────────────────────────────────────────┐
│  Receiver is notified: "File has been released"             │
│  Receiver downloads and decrypts with their key.            │
│  PKI key exchange is complete (both sides have public keys) │
│  All future communication is fully authenticated.           │
└─────────────────────────────────────────────────────────────┘
```

## What This Achieves

| Property | How |
|---|---|
| **Sender control** | The sender decides IF and WHEN to release the file, based on actual evidence about the receiver's environment. |
| **Transparent PKI bootstrap** | The receiver gets a PKI key pair automatically. The sender gets the receiver's public key in the response. After one dual-auth exchange, both sides have full PKI. |
| **Environmental verification** | The sender sees WHERE the receiver will open the file before releasing it. |
| **Out-of-band confirmation** | The challenge code is sent via a different channel (WhatsApp, phone) than the file delivery (SG/Send). Two channels = harder to intercept both. |
| **Audit trail** | Every step is logged: challenge sent, receiver responded, sender reviewed, file released. |

## Secure Browser Enforcement

The sender can require that the receiver uses a specific browser or environment:

```
┌─────────────────────────────────────────────────────────────┐
│  Delivery Requirements (set by sender)                      │
│                                                             │
│  ☐ Any browser (default)                                   │
│  ☐ Secure browser only (e.g., Surf Security)               │
│  ☐ Specific browser: [Chrome 120+           ]              │
│  ☐ Country restriction: [United Kingdom     ]              │
│  ☐ No VPN / proxy                                          │
│  ☐ Business hours only (9am-6pm recipient timezone)         │
│                                                             │
│  If requirements not met:                                   │
│  ● Block delivery and notify receiver of requirements       │
│  ● Allow delivery with warning to sender                    │
└─────────────────────────────────────────────────────────────┘
```

The receiver sees: "The sender requires you to use [Chrome on macOS] from [United Kingdom]. Your current environment: Chrome on macOS, London, UK. ✅ Requirements met."

## The Challenge Code

The challenge code follows the same model from the Receiver Experience and Trust Spectrum brief (v0.5.24):

| Property | Value |
|---|---|
| **Format** | 6-digit numeric, alphanumeric, or three-word phrase (sender's choice) |
| **One-use** | Self-destructs after first successful use |
| **Attempt limit** | 1-3 failed attempts before self-destruct |
| **Expiry** | Configurable (1 hour to 7 days) |
| **Scope** | Retrieves the challenge (NOT the file). The file is released separately by the sender. |

This is doubly secure compared to single-step delivery: even if the challenge code is intercepted, the attacker only gets to the "environment check" stage. The sender still has to approve the release, and the attacker's environment will look different from the expected receiver's.

## Data Room Integration

For data rooms, dual authorization can be applied at multiple levels:

| Level | Trigger | Who Authorises |
|---|---|---|
| **Room entry** | First-time access to a data room | Data room admin reviews environment before granting access |
| **Sensitive folder** | Accessing a restricted section within a data room | Folder owner reviews and releases |
| **Individual file** | Downloading a high-sensitivity document | File uploader reviews and releases |

This maps naturally to the data room access tiers (Lobby → Member → Verified → Admin from v0.5.24). Dual authorization is the mechanism for transitioning between tiers.

## The Notification System

Dual authorization requires real-time notifications:

| Event | Who Gets Notified | How |
|---|---|---|
| Receiver responded to challenge | Sender | In-app notification + optional email/webhook |
| Sender released the file | Receiver | The download page shows "File available" |
| Sender denied the delivery | Receiver | "The sender has not approved delivery at this time" |
| Challenge expired | Both | "The delivery request has expired" |
| Sender requested environment change | Receiver | "The sender requests you use a different environment: [details]" |

For MVP, notifications are polling-based (receiver refreshes the page). For production, WebSocket or SSE for real-time updates.

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | Sender can mark a file for "secure delivery" (dual authorization mode) |
| 2 | Challenge code generated and displayed to sender |
| 3 | Receiver enters challenge code and sees transparency notice |
| 4 | Receiver's environment report + public key sent to sender (encrypted) |
| 5 | Sender sees environment report and can release, deny, or request change |
| 6 | File is NOT downloadable until sender explicitly releases it |
| 7 | PKI key exchange completes transparently through the dual-auth flow |
| 8 | Challenge code is one-use and self-destructs on failure/expiry |
| 9 | Sender can set delivery requirements (browser, country, VPN policy) |
| 10 | Notifications inform both parties of state changes |
