# Architecture Brief: Trust Web — Reputation Through Usage

**version** v0.6.14  
**date** 23 Feb 2026  
**from** Human (project lead)  
**to** Architect (lead), AppSec, GRC  
**type** Architecture brief — trust and reputation model

## The Concept

Every successful file transfer builds trust. Every complaint destroys it. Over time, a web of trust emerges where senders, receivers, and their relationships form a graph. Attackers must invest heavily in building fake trust networks — and those networks are destroyed the moment one node is flagged.

This is the "access is a spectrum" principle applied to reputation: what you can do on the platform depends on how much trust you've earned.

---

## How Trust Builds

### Per-Transaction Trust Signals

Every file sent generates trust signals:

| Signal | Positive | Negative |
|---|---|---|
| **Delivery** | File delivered, no issues | Delivery failed, recipient couldn't open |
| **No complaint** | Recipient didn't report the file | Recipient reported file as malicious/spam |
| **Scanning passed** | File passed all security scans | File flagged by scanning pipeline |
| **Recipient trust** | Recipient has a high trust score themselves | Recipient is new/untrusted (neutral) |
| **Reply received** | Recipient replied (confirms engagement) | No engagement (neutral) |
| **Time without complaint** | Days/weeks pass with no issues | Late complaint (still counts, reduced weight) |

### Trust Score Model

```
sender_trust = f(
  successful_deliveries,       # files sent without complaints
  recipient_trust_weighted,    # weighted by recipients' own trust scores
  scan_pass_rate,              # % of files that pass security scanning
  account_age,                 # time on platform
  complaint_count,             # negative — each complaint heavily weighted
  reported_malware_count       # severely negative — near-instant trust destruction
)
```

The exact formula is for the Architect and AppSec to design, but the key properties are:

- **Trust builds slowly** — many successful transactions over time
- **Trust destroys fast** — one malware report drops trust dramatically
- **Recipient trust matters** — sending to trusted recipients is worth more than sending to new/anonymous users (prevents self-referential trust farming)
- **The web is the defence** — an attacker needs to build relationships with genuinely trusted users, who would then need to NOT report the malicious file

## What Trust Unlocks

| Trust Level | What You Can Do | Rate Limits |
|---|---|---|
| **New / Anonymous** | Send files (scanned, warnings shown). Limited recipients. Short code access only. | 5 files/day, 1 MB max |
| **Low Trust** | Send files with scan results shown. More recipients. | 20 files/day, 5 MB max |
| **Medium Trust** | Send files with advisory scanning. Create simple data rooms. | 100 files/day, 50 MB max |
| **High Trust** | Full platform access. Data rooms with multiple tiers. Scanning is advisory. | Generous limits |
| **Verified Trust** | Enterprise features. Custom data rooms. API access. | Per-agreement |

These are illustrative — the specific thresholds are for the product team to define. The principle is: **earn access through behaviour, not just payment.**

## The Graph Defence

The trust web forms a graph:

```
     Alice (high trust)
    /    \
   /      \
  Bob      Carol (high trust)
  |    \
  |     \
  Dave    Eve (new)
```

If Eve sends malware to Dave, and Dave reports it:
1. Eve's trust is destroyed
2. Any files Eve sent to others are flagged for review
3. Eve's future access is severely restricted
4. Eve needs to rebuild trust from zero — with new recipients who are themselves trusted

**The graph makes attacks expensive.** An attacker can't just create a new account and start sending malware. They need to:
- Build trust over time (slow)
- Send clean files to trusted users who won't report (requires real relationships)
- Maintain the facade across multiple recipients (fragile — one report destroys it all)

### Sybil Resistance

The trust model is naturally Sybil-resistant because:
- New accounts have zero trust (can't do much)
- Trust from interactions with other new accounts is worth very little (low recipient trust weight)
- Only interactions with genuinely trusted users build meaningful trust
- An attacker creating many fake accounts that only interact with each other builds no real trust

## Where Trust Data Lives

**Critical design decision**: trust data must not violate the zero-knowledge model.

| Data | Where Stored | Who Can See It |
|---|---|---|
| **Aggregate trust score** | Server-side (as a number associated with a public key hash) | Anyone can query a public key's trust score. The server knows the score but not what the score represents in human terms. |
| **Transaction history** | Client-side (sender's and receiver's browsers) | Only the parties involved. The server sees encrypted blobs and Obj_Ids, not who sent what to whom. |
| **Complaint records** | Server-side (a flag associated with a file's Obj_Id) | The server knows "this Obj_Id was reported" but not what the file contained. |
| **Trust graph edges** | Derived from transaction patterns (server sees "key A uploaded, key B downloaded" but not file contents) | The server can compute graph relationships from access patterns. This is an accepted trade-off — the server knows who communicates with whom (at the public key level) but not what they communicate. |

### Privacy Trade-Off

The trust system requires the server to know SOMETHING about usage patterns. Specifically: which public keys are sending to which public keys, and whether complaints were filed. This is a trade-off against full zero-knowledge. 

**AppSec and GRC**: document this trade-off. The server knows communication patterns (metadata) but not communication content. This is similar to how email servers know who emails whom but not what the emails say. Is this acceptable? Under what conditions?

## Abuse Scenarios and Responses

| Scenario | Detection | Response |
|---|---|---|
| **Single malware send** | Recipient reports file. Scanning pipeline flags file. | Sender trust drops. File quarantined. Sender may be restricted. |
| **Trust farming (slow buildup)** | Attacker sends clean files for weeks, then sends malware. | Malware report destroys accumulated trust. All previous files flagged for review. The investment is lost. |
| **Sybil attack (many fake accounts)** | Fake accounts only interact with each other. Low recipient trust weights mean no real trust accumulates. | Cluster detection: groups of accounts that only interact with each other are flagged. |
| **Compromised trusted account** | Trusted account starts sending malware (account takeover). | Sudden behaviour change detected (clean history → malware). Account frozen pending review. PKI key rotation required. |
| **Bot / agent gone rogue** | Automated agent sends files at high volume. | Budget controls (from pod architecture) cap activity. Anomalous volume triggers circuit breaker. |

## Connection to Existing Architecture

| Component | How It Connects |
|---|---|
| **File scanning pipeline** (v0.6.14) | Scanning results feed into trust signals. Clean scans = positive. Flagged scans = negative. |
| **PKI key pairs** (v0.4.12+) | Trust is associated with public key hashes, not accounts or emails. Identity IS the key. |
| **Data rooms** (v0.5.10) | Data room access tiers can reference sender/receiver trust scores. |
| **Budget controls** (v0.4.12 pod architecture) | Budget limits are a hard cap. Trust is a soft cap. Both work together. |
| **Receiver environment analysis** (v0.6.14) | Environmental data adds context to trust decisions. |
