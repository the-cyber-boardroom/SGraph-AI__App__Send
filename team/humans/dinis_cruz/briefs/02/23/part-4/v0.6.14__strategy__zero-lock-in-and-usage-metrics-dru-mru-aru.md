## The Core Metric: Encrypted Messages Facilitated

SG/Send does not measure success by revenue. SG/Send measures success by:

> **How many encrypted messages do we facilitate and enable — directly or indirectly — per unit of time?**

This is the North Star. Every product decision, every architecture choice, every partnership, every feature is evaluated against this metric.

---

## DRU / MRU / ARU — Not MRR / ARR

Traditional SaaS measures **Monthly Recurring Revenue (MRR)** and **Annual Recurring Revenue (ARR)**. These metrics optimise for extracting money from users. They incentivise lock-in, premium gates, and artificial scarcity.

SG/Send measures **Recurring Usage**:

| Metric | Definition | What It Captures |
|---|---|---|
| **DRU** (Daily Recurring Usage) | Number of encrypted messages facilitated per day | Daily engagement. Is the platform being used actively? |
| **MRU** (Monthly Recurring Usage) | Number of encrypted messages facilitated per month | Growth trend. Are more people encrypting more messages? |
| **ARU** (Annual Recurring Usage) | Number of encrypted messages facilitated per year | Long-term trajectory. Is the encrypted ecosystem growing? |

### What Counts as "Facilitated"

| Scenario | Counts? | Why |
|---|---|---|
| File encrypted and sent through send.sgraph.ai | ✅ | Direct usage |
| File encrypted via our CLI tool | ✅ | Direct usage |
| File encrypted via our JS library embedded in a SaaS product | ✅ | Indirect — our library, their product |
| File encrypted with GPG using a key pair generated on our platform | ✅ | Indirect — we bootstrapped the key, they used their own tools |
| File encrypted with GPG using a key discovered via our GitHub registry integration | ✅ | Indirect — we provided the discovery, they did the rest |
| User learned about PKI from our docs, generates keys with ssh-keygen, encrypts with AGE, never visits our site again | ✅ | Indirect — we educated, they act independently |
| Cloud provider builds compatible service on our open source | ✅ | Indirect — our protocol, their infrastructure |
| Partner integrates our scanning chain into their product | ✅ | Indirect — our trust model, their users |

**Every encrypted message in the ecosystem increases our value — regardless of whether SG/Send directly processed it.**

### Why This Works Economically

```
More encrypted messages in the ecosystem
    → More people with PKI keys
    → More need for trust services (scanning, verification, trust web)
    → More need for data rooms (collaboration spaces)
    → More need for enterprise features (audit trails, compliance)
    → Revenue follows usage naturally

Traditional SaaS:
    Gate features → extract revenue → hope for growth

Usage-first model:
    Maximise usage → build the ecosystem → revenue is a side effect of ubiquity
```

Revenue is a lagging indicator of usage. Optimise for usage; revenue follows.

## The Zero Lock-In Principle

### The Principle

> SG/Send always defaults to what is better for the user, and never introduces lock-in. We want to be the easiest platform to use, but we always give the user the ability — and the tools and guidance — to use their own technologies, tools, and practices.

This applies at **BOTH ends** of every process:

| End | What It Means |
|---|---|
| **Creation** (encryption, key generation, signing) | User can use our tools OR GPG, AGE, OpenSSL, ssh-keygen, hardware tokens, or any compatible tool |
| **Consumption** (decryption, verification, reading) | User can use our browser OR GPG, AGE, OpenSSL, any PGP-compatible tool, or any compatible library |

### Why Zero Lock-In Increases Value

| Lock-in approach | Zero lock-in approach |
|---|---|
| Forces users to use our platform for every operation | Users CAN use our platform or anything compatible |
| Users who leave = lost value | Users who leave our platform but keep encrypting = gained value |
| Growth limited to our direct user base | Growth includes everyone in the encrypted ecosystem |
| Network effect limited to our platform | Network effect extends to the entire PKI ecosystem |
| Competitors are threats | Competitors who use compatible encryption are allies — they grow the ecosystem |

**A user who generates keys on SG/Send and then exclusively uses GPG for the rest of their life is a success story.** They are one more person in the encrypted ecosystem. Their public key is one more key in the registry. Their encrypted messages are one more data point proving the market exists.

### How to Implement Zero Lock-In

| Area | Lock-in-free Design |
|---|---|
| **File formats** | Support standard formats (.gpg, .age, .pem) — not just proprietary .sgenc. .sgenc should itself be a thin wrapper around standard crypto. |
| **Key formats** | Accept and export SSH Ed25519, PGP/GPG, PEM, JWK, PKCS#8 — any standard key format |
| **Protocols** | Use standard encryption (NaCl, AES-256-GCM, RSA-OAEP) — not custom crypto |
| **Documentation** | Always document the "without SG/Send" path. Every feature page should show how to do the same thing with open tools. |
| **CLI tools** | Our CLI is a convenience wrapper around standard tools. If our CLI disappears, users can still decrypt with GPG. |
| **Data export** | Any data room can be exported as standard encrypted files. No data is trapped in our platform. |
| **Key export** | Users can always export their private key. We never hold keys hostage. |
| **APIs** | Standard REST APIs. No proprietary protocols. Anyone can build a compatible client. |

### The Documentation Test

For every feature, ask: **"Can a user do this without SG/Send?"** If yes, document how. If no, redesign until they can.

```
Feature: Send an encrypted file to a GitHub user

With SG/Send:
  1. Go to send.sgraph.ai
  2. Enter GitHub username
  3. Drop file
  4. Done

Without SG/Send:
  1. curl https://github.com/username.keys > recipient.pub
  2. Convert Ed25519 to X25519 (libsodium or age)  
  3. age -r "ssh-ed25519 AAAA..." file.pdf > file.pdf.age
  4. Email file.pdf.age to recipient
  5. Recipient: age -d -i ~/.ssh/id_ed25519 file.pdf.age > file.pdf

Both paths documented. SG/Send is easier. Neither path is required.
```

## How This Applies to Every Document in the Project

This principle retroactively applies to every architecture decision:

| Document | Zero Lock-In Implication |
|---|---|
| **Portable Data Rooms (ZIP)** | Data rooms export to standard ZIP files with standard encryption. Any tool can extract. |
| **Fractal Document Signing** | Signed documents use standard PGP signatures. Any PGP tool can verify. |
| **GitHub as PKI Registry** | We use GitHub's existing infrastructure. We don't require our own registry. |
| **External File Scanning** | Scanning services are third-party. Users choose which services. We don't lock them into our scanners. |
| **Trust Web** | Trust data is exportable. Users can take their reputation to a compatible platform. |
| **Cross-Domain File Security** | The checkpoint model uses standard encryption. Any compatible tool can create files that pass through. |
| **Email Integration** | Users can encrypt with GPG and send via email. No SG/Send involvement required. |
| **SaaS Integration (Credit Card Model)** | The JS library uses standard Web Crypto API. The encryption is standard. If the library disappears, the encrypted data is still decryptable with standard tools. |

## The Competitive Moat (That Isn't Lock-In)

If we don't lock users in, what's the moat?

| Moat | Why It Works |
|---|---|
| **The trust web** | Trust scores, reputation, scanning history — this is data that builds over time. Switching platforms means rebuilding trust. But it's EXPORTABLE if they want to leave. |
| **Network density** | The more users in our ecosystem (direct + indirect), the easier it is to find someone's public key and send them an encrypted message. |
| **UX superiority** | We're the easiest way to do it. Users CAN use GPG. They CHOOSE to use SG/Send because it's simpler. |
| **Partner integrations** | Scanning chains, identity providers, cloud storage — these integrations create convenience. |
| **Community and documentation** | The best guides, the best examples, the most helpful community. People come for the docs, stay for the platform. |

**The moat is being the best, not being the only option.**
