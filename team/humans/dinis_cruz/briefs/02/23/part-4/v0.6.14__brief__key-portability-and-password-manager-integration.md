# Brief: Key Portability and Password Manager Integration

**version** v0.6.14  
**date** 23 Feb 2026  
**from** Human (project lead)  
**to** Developer (lead), Architect, Ambassador (partner)  
**type** Dev brief + partner brief — key management across devices  

---

## The Problem

The human already has multiple desktops and laptops. PKI keys generated in one browser don't exist in another. Without portability, you end up with one identity per device — which is not what anyone wants.

**Current state**: SG/Send generates a key pair in the browser and stores it in the browser's local storage / Service Worker. That key is trapped in that one browser on that one device.

**Required state**: the user can move their key pair to any browser on any device, securely, and ideally store it in a tool they already use for sensitive data — their password manager.

---

## Key Portability Features

### Import Private Key into SG/Send Browser

```
User has their private key from:
  - Password manager (1Password, Bitwarden, LastPass, etc.)
  - ~/.ssh/ directory (SSH keys)
  - GPG keyring
  - A previous SG/Send browser session (exported)
  - A .key file on a USB drive

User goes to send.sgraph.ai:
  1. Clicks "Import Key" (or drags and drops a key file)
  2. Key is loaded into the browser's secure storage
  3. Browser now has the private key — all encrypted files can be decrypted
  4. The imported key file can be deleted from the temp location
```

### Export Private Key from SG/Send Browser

```
User wants to use their key on another device:
  1. Goes to send.sgraph.ai on the original device
  2. Clicks "Export Key"
  3. Downloads a key file (encrypted with a passphrase the user chooses)
  4. Transfers the file to the new device (password manager, USB, AirDrop, etc.)
  5. Imports on the new device
```

### Import Methods

| Method | How It Works | Security |
|---|---|---|
| **File drag-and-drop** | Drop a .key / .pem / .pgp file onto the import area | Key file exists on disk temporarily |
| **Copy-paste** | Paste the key text into a text field | Key is in clipboard temporarily |
| **Password manager autofill** | Password manager fills a "key" field (like a password field) | Key handled by the password manager's secure pipeline |
| **QR code scan** | Display QR code on one device, scan with another | Key is in the QR code briefly — use for device-to-device transfer |
| **One-time link** | Generate a time-limited, self-destructing link containing the encrypted key | Uses SG/Send's own infrastructure — eat your own dogfood |

## Password Manager Integration

Password managers are the natural home for private keys. Users already trust them with passwords. Private keys are just another secret.

### How It Works with Password Managers

```
1. User generates key pair in SG/Send browser
2. SG/Send prompts: "Save your private key to your password manager?"
3. User clicks "Save" → browser triggers the password manager's save dialog
   (same mechanism as saving a password)
4. Password manager stores:
   - Website: send.sgraph.ai
   - Username: [user's public key fingerprint]
   - Password: [private key, base64-encoded]
   - Notes: "SG/Send private key. Public key: [fingerprint]"

5. On a new device:
   - User goes to send.sgraph.ai
   - Password manager autofills the "key" field
   - SG/Send imports the private key
   - Done — same identity on the new device
```

### Password Managers as Partners

| Manager | Integration Path | Notes |
|---|---|---|
| **1Password** | SSH key agent already built-in. 1Password can store and serve SSH keys. Natural fit. | Premium partner — they already understand PKI |
| **Bitwarden** | Open source. Custom fields for key storage. API for programmatic access. | Good alignment with our open source philosophy |
| **Apple Keychain** | iCloud Keychain syncs across Apple devices. Passkey support built in. | Covers the Apple ecosystem automatically |
| **Google Password Manager** | Built into Chrome. Syncs across Chrome browsers. | Widest reach for Chrome users |
| **LastPass** | Custom fields. Browser extension API. | Large user base |
| **KeePass** | Open source, local storage. No cloud sync (by design). | For privacy-focused users |

The Ambassador should approach 1Password and Bitwarden first — both understand PKI, both have developer-friendly APIs, both align with our values.

## Key Formats to Support

| Format | Extension | Source | Notes |
|---|---|---|---|
| **SG/Send native** | `.sgkey` | Our key generation | JSON containing public + private key, optionally passphrase-encrypted |
| **SSH Ed25519** | `id_ed25519` | `~/.ssh/` directory | Most developers already have this |
| **PGP/GPG** | `.asc`, `.gpg` | GPG keyring | Standard PGP private key export |
| **PEM** | `.pem` | OpenSSL | Standard format, widely supported |
| **JWK** | `.jwk` | Web Crypto API | JSON Web Key — native to browser crypto |
| **PKCS#8** | `.p8` | Various tools | Standard key serialisation |

SG/Send should auto-detect the format on import and convert as needed. The user shouldn't have to know or care which format their key is in.

## Security Considerations

| Concern | Mitigation |
|---|---|
| **Key file on disk** | Warn user to delete after import. Offer to import from memory (clipboard) only. |
| **Key in clipboard** | Clear clipboard after import (browser API: `navigator.clipboard.writeText('')`). |
| **Key in password manager** | This is a feature, not a bug. Password managers are designed for this. The key is encrypted at rest by the manager. |
| **Key in transit between devices** | Always passphrase-encrypt the exported key file. Use a strong passphrase. |
| **Key in browser storage** | Already handled by the Service Worker trust anchor (v0.4.12 architecture). Key is in IndexedDB, accessible only to our origin. |
| **Multiple devices = multiple attack surfaces** | True. But the alternative (one key per device) means lost keys and fragmented identity. Portability is the practical choice. |

## The Practical Reality

The human's insight: "there's a lot of security properties that we provide. But let's also have a sense of real world practicality."

Users WILL:
- Use multiple devices
- Lose browsers / reinstall OSes
- Need to recover their keys
- Want to set up a new laptop quickly

If we don't make key portability easy, users will:
- Screenshot their keys (terrible)
- Email keys to themselves (terrible)
- Give up on PKI entirely (worst case)

Making it easy to do the RIGHT thing (password manager, secure export/import) prevents users from doing the WRONG thing.
