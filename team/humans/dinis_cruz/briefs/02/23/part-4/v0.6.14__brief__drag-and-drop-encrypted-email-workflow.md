## The Concept

Decouple sending from decryption. Send encrypted files through ANY channel — email, WhatsApp, USB, Slack, carrier pigeon — and decrypt by dragging the file into the SG/Send browser. No server involvement. No platform dependency. The encrypted file IS the delivery mechanism.

**This is something we can implement right now.** It's simple, practical, and useful today.

---

## The Workflow

### Sending

```
1. Sender goes to send.sgraph.ai
2. Enters recipient's public key (or GitHub username, or selects from contacts)
3. Drops a file onto the encrypt area
4. Gets back: confidential.pdf.sgenc (or .age, or .gpg)
5. Attaches that file to an email / WhatsApp / Slack / anything
6. Sends it through whatever channel they normally use

The email body:
  Subject: "Encrypted file for you"
  Body: "Here's an encrypted file. To decrypt:
         1. Go to send.sgraph.ai
         2. Drag and drop the attached file
         Your private key in your browser will decrypt it automatically."
```

### Receiving

```
1. Recipient gets the email (or WhatsApp message, or Slack message)
2. Downloads the .sgenc attachment
3. Goes to send.sgraph.ai (or already has it open)
4. Drags and drops the file onto the decrypt area
5. Browser has their private key (from setup, or imported from password manager)
6. File decrypts instantly in the browser
7. Recipient sees/downloads the original file
```

### What Just Happened

The server never saw the file. The email provider saw an encrypted blob. The decryption happened entirely in the browser. The only thing that travelled over the insecure channel (email) was an encrypted file that's useless without the private key.

## File Naming Options

The sender can choose how much metadata to expose:

| Option | File Name | What's Visible |
|---|---|---|
| **Named** | `quarterly-report.pdf.sgenc` | File name visible, contents encrypted |
| **Type only** | `document.pdf.sgenc` | File type visible, name hidden |
| **Fully opaque** | `message.sgenc` | Nothing visible — name, type, size all hidden |

The receiver's preference (from their profile or data room settings) determines the default. Sender can override towards MORE privacy but not less.

---

## Encrypted File Formats

Support multiple formats — meet users where they are:

| Format | Extension | Library | Who Uses It |
|---|---|---|---|
| **SG/Send native** | `.sgenc` | Web Crypto API (our implementation) | Our users — default |
| **AGE** | `.age` | age-encryption.org | Developers familiar with AGE — works with SSH keys |
| **PGP/GPG** | `.gpg` or `.asc` | OpenPGP.js | Anyone with GPG — the widest compatibility |

The SG/Send browser should accept ALL THREE formats for decryption. For encryption, default to `.sgenc` but allow the sender to choose `.age` or `.gpg` if the recipient prefers those.

## The Zero-Lock-In Scenario: No SG/Send Required After First Use

**This is a core principle, not just a feature.** After the initial key exchange, the user should be able to encrypt and decrypt files WITHOUT EVER using SG/Send again — not the website, not the CLI, not any of our tools. Standard, open tools that already exist on their machine should work.

### Scenario: Pure Open-Source Workflow (No SG/Send)

```
SETUP (one-time, can use SG/Send or not):
  Alice and Bob exchange public keys (by any means)

ALICE ENCRYPTS (no SG/Send):
  # Using GPG (installed on most Linux/macOS systems)
  $ gpg --encrypt --recipient bob@example.com report.pdf
  → produces report.pdf.gpg

  # OR using AGE (one binary, no config)
  $ age -r "ssh-ed25519 AAAA..." report.pdf > report.pdf.age

  # OR using OpenSSL
  $ openssl pkeyutl -encrypt -pubin -inkey bob-public.pem -in report.pdf

ALICE SENDS (no SG/Send):
  Attaches report.pdf.gpg to a normal email. Sends via Gmail/Outlook/anything.

BOB DECRYPTS (no SG/Send):
  # Using GPG
  $ gpg --decrypt report.pdf.gpg > report.pdf

  # OR using AGE
  $ age -d -i ~/.ssh/id_ed25519 report.pdf.age > report.pdf

  # OR using any PGP-compatible tool on any platform
```

**SG/Send was never involved.** The encryption happened with standard tools. The decryption happened with standard tools. The keys were exchanged once. The email provider carried the encrypted blob. Every tool in this flow is open source, widely available, and controlled by the user.

### Why This Matters

SG/Send's platform value is measured by how many encrypted messages are facilitated — **directly OR indirectly.** A user who learned about PKI through our platform, got their first key pair from our browser tool, and then switched to GPG on the command line is a SUCCESS, not a lost customer. They are now part of the encrypted ecosystem. Every encrypted message they send increases the value of the network.

### The Spectrum of SG/Send Involvement

| Level | SG/Send Involvement | User Experience |
|---|---|---|
| **Full SG/Send** | Encrypt + store + deliver + decrypt all through send.sgraph.ai | Easiest. Zero-knowledge server. Full audit trail. |
| **SG/Send encrypt, open delivery** | Encrypt on send.sgraph.ai, send via email, decrypt on send.sgraph.ai | Our encryption, their delivery channel. |
| **SG/Send key exchange only** | Use send.sgraph.ai to generate keys and exchange public keys. After that, use any tool. | We bootstrap the relationship. They own it after. |
| **No SG/Send at all** | User already has GPG/SSH keys. Uses our documentation to learn the workflow. Never visits our site. | We provided guidance. They did everything themselves. |

**All four levels count as SG/Send-facilitated encrypted communication.** All four increase the value of the network. We optimise for Level 1 (best UX) but we CELEBRATE Level 4 (maximum user autonomy).

### Both Ends: Creation AND Consumption

The zero-lock-in principle applies symmetrically:

| End | Lock-in-free Approach |
|---|---|
| **Creation (encryption)** | User can encrypt with GPG, AGE, OpenSSL, our CLI, our browser, or any PGP-compatible tool |
| **Consumption (decryption)** | User can decrypt with GPG, AGE, OpenSSL, our CLI, our browser, or any PGP-compatible tool |
| **Key generation** | User can generate keys with our browser, ssh-keygen, gpg --gen-key, or any key generation tool |
| **Key storage** | User can store keys in our browser, their password manager, their SSH directory, their GPG keyring, or a hardware token |
| **Key exchange** | User can exchange keys via our platform, GitHub, email, QR code, in person, or any other method |

We provide the EASIEST path. We never make it the ONLY path.

## No Chrome Extension Needed

This workflow works WITHOUT any browser extension:

| With Extension | Without Extension |
|---|---|
| Encrypt/decrypt inside Gmail UI | Encrypt at send.sgraph.ai, attach to Gmail manually |
| Seamless, integrated experience | Two tabs — still simple, no installation needed |
| Requires extension installation | Works in any browser immediately |

The extension is a UX improvement. The drag-and-drop workflow is the foundation that works everywhere, today, with zero installation.

## Connection to First Use (TOFU)

The first message requires exchanging public keys. After that, everything is smooth:

```
First message (one-time setup):
  Alice: "Here's my public key" (sends via email, website, QR code, GitHub)
  Bob:   "Here's mine" (same)
  
  Both import each other's public key into SG/Send browser.

Every subsequent message:
  Alice: encrypt file with Bob's public key → email .sgenc → done
  Bob:   drag and drop → decrypted → done
```

The first exchange is the TOFU moment — pay attention, verify. After that, the keys are established and every exchange is frictionless.
