**date** 24 Feb 2026

**to** AppSec (lead), Architect, Developer

## The Blind Spot

Originally, the PKI-Secured Agentic Workflows document (v0.6.14) made this claim:

> "No agent ever sees a secret that isn't encrypted for their key. Even if all agents run on the same machine, in the same process, with shared memory — the cryptographic boundary enforces isolation."

**This statement is wrong.** It confuses cryptographic boundaries (mathematical) with process isolation (OS-enforced). Encryption in memory is not isolation. This document corrects that claim, explains why, and describes the architecture that makes the isolation real.

---

## Why Encryption in Shared Memory Is Not Isolation

### The Fundamental Problem

At the moment of decryption, the private key and the decrypted data MUST exist in cleartext in memory. There is no way around this. The CPU operates on cleartext. The key must be cleartext for the decryption operation. The result is cleartext so the agent can use it.

```
Agent A's memory (simplified):

  0x7f00_0000: Agent A's private key (cleartext — needed for decryption)
  0x7f00_1000: Decrypted secret: DATABASE_API_KEY = "sk-abc123..."
  0x7f00_2000: Agent A's task context
  0x7f00_3000: Agent A's LLM prompt buffer
  ...

If Agent B runs in the same process:

  Agent B can read 0x7f00_0000 → Agent A's private key
  Agent B can read 0x7f00_1000 → Agent A's decrypted secret
  Agent B can read ANY address in the shared process memory
```

This is not theoretical. This is how operating systems work.

### The Only Real Isolation Boundaries in an OS

Modern operating systems provide exactly two meaningful isolation boundaries:

| Boundary | What It Isolates | Enforced By |
|---|---|---|
| **Kernel vs Userspace** | User code cannot access kernel memory. Hardware-enforced via CPU privilege rings (ring 0 vs ring 3) and page table permissions. | CPU hardware + OS kernel |
| **Process A vs Process B** (different users) | Processes running under different user accounts cannot read each other's memory. Enforced by the kernel via virtual memory and permission checks. | OS kernel (page tables, UID checks) |

Everything else is convention, not enforcement:

| "Boundary" | Why It's Not Real Isolation |
|---|---|
| **Language-level access control** (private fields, encapsulation) | Convention. Any code in the same process can use reflection, unsafe pointers, or FFI to bypass. |
| **Managed code / sandboxes** (.NET managed, JVM SecurityManager) | The runtime enforces restrictions, but unmanaged code / native calls bypass the runtime entirely. In .NET, `unsafe` blocks access raw memory. JVM SecurityManager is deprecated. |
| **Thread isolation** | Threads share the process address space. Thread A can read Thread B's stack and heap. No OS enforcement between threads. |
| **Variable scoping** | A language feature, not a security boundary. A prompt-injected agent that can execute arbitrary code doesn't respect variable scope. |
| **"Encryption in memory"** | The key is in memory too. Find the key → decrypt anything. Or just find the already-decrypted data. |

### How an Attacker (or Prompt-Injected Agent) Reads Other Agents' Memory

If a prompt injection gives an agent the ability to execute code (which is the threat model for code-executing agents), the attack surface includes:

| Technique | How It Works | Platform |
|---|---|---|
| **Direct memory read** | In languages with unsafe/unmanaged code (C, C++, Rust unsafe, .NET P/Invoke, Python ctypes), read arbitrary memory addresses | All |
| **`/proc/self/mem`** | On Linux, a process can read its own memory via the proc filesystem. If agents share a process, they share `/proc/self/mem`. | Linux |
| **`/proc/<pid>/mem`** | If agents run as separate processes under the SAME user, one can read the other's memory via procfs (requires ptrace permissions, but same-user usually has them). | Linux |
| **Debugging APIs** | `ptrace()` on Linux, `ReadProcessMemory()` on Windows. Same-user processes can attach to and read each other. | Linux, Windows |
| **Shared memory segments** | If agents use shared memory (common in multi-agent frameworks for performance), all data is directly accessible. | All |
| **Memory scanning** | Scan the heap for patterns (key material has recognisable structure). Tools like Volatility, or custom scanners. | All |
| **Core dumps** | Trigger or access a core dump. All memory, including keys and decrypted data, is in the dump file. | All |
| **Swap / page file** | Memory pages containing keys may be written to swap. Readable by any same-user process. | All |
| **Environment variables** | Many frameworks pass secrets via env vars. `environ` is readable by any code in the process, and by same-user processes via `/proc/<pid>/environ`. | All |

**A prompt-injected agent that can execute code has the same privileges as the process it runs in.** If that process has access to other agents' memory, the prompt injection has access to other agents' memory.

## The Correct Architecture: One Agent, One Isolated Environment

The only way to achieve the isolation described in the agentic workflows brief is to run each agent in a **separate, isolated execution environment** where:

1. **Only that agent's private key exists** — no other agent's key material is present
2. **The environment has minimal privileges** — no network access to other agents' environments, no shared filesystem, no shared memory
3. **The environment is ephemeral** — created for the task, destroyed after completion. Key material doesn't persist.

### Isolation Options (Ordered by Strength)

| Isolation Level | Technology | Isolation Strength | Overhead |
|---|---|---|---|
| **Serverless function** | AWS Lambda, Cloudflare Workers, GCP Cloud Functions | **Cold start**: strongest — fresh VM, no residual state. **Warm start**: weaker than assumed — the execution environment is frozen between invocations and thawed on the next request. File system, memory, and even running processes persist across invocations. A process started in one request that doesn't exit before the response is returned will be frozen and restored on the next invocation. This means Agent A's secrets could still be in memory or on the `/tmp` filesystem when Agent B's task runs in the same warm environment. AWS has introduced per-user isolation features that reduce this risk, but the default behaviour is shared state across warm invocations. **Mitigation**: explicitly zero key material from memory and `/tmp` before returning. Use one Lambda function per agent (not one shared function for all agents). Request cold starts for high-security workflows (accept the latency). | Highest latency (cold start) or moderate (warm) |
| **Dedicated VM** | EC2 instance, GCP Compute, Azure VM | Strong — hardware-level isolation via hypervisor | High resource cost |
| **Container (hardened)** | Docker with seccomp, AppArmor/SELinux, no-new-privileges, read-only rootfs, dropped capabilities | Strong if properly configured — kernel namespace isolation | Moderate |
| **Container (default)** | Docker with default settings | Moderate — namespace isolation but shared kernel, default capabilities may be too broad | Low |
| **Separate process, different user** | `su agent-a -c "run-agent"` | Moderate — kernel-enforced UID isolation. Ptrace restrictions help but aren't bulletproof. | Low |
| **Separate process, same user** | `fork()` or subprocess | Weak — same UID, ptrace possible, `/proc/<pid>/mem` readable | Lowest |
| **Same process, different threads** | Thread pool | None — shared address space. No isolation. | N/A |

### The Recommended Architecture

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Orchestrator                                 │
│                                                                     │
│  Has: all agents' PUBLIC keys (for encrypting tasks)               │
│  Has: orchestrator's PRIVATE key (for signing task assignments)    │
│  Does NOT have: any agent's private key                            │
│                                                                     │
│  Runs in: its own isolated container / serverless function         │
└───────┬──────────────────┬──────────────────┬──────────────────────┘
        │                  │                  │
        │ .sgworkflow      │ .sgworkflow      │ .sgworkflow
        │ (encrypted       │ (encrypted       │ (encrypted
        │  for Agent A)    │  for Agent B)    │  for Agent C)
        ▼                  ▼                  ▼
┌───────────────┐  ┌───────────────┐  ┌───────────────┐

│               │  │               │  │               │
│  Agent A      │  │  Agent B      │  │  Agent C      │
│               │  │               │  │               │
│  Has: Agent   │  │  Has: Agent   │  │  Has: Agent   │

│  key ONLY     │  │  key ONLY     │  │  key ONLY     │
│               │  │               │  │               │

│  encrypted    │  │  encrypted    │  │  encrypted    │

```

Now the claim from the original document IS true — but only because the isolation is enforced by the container/VM boundary (kernel-level), not by encryption alone.

**Encryption selects WHO can read the data. Process isolation ensures they're the ONLY ones who can.**

## The Traditional Problem: State Management Across Isolation

This architecture — one agent per isolated environment — is well understood in security. It's not new. The reason most agent frameworks don't do it is NOT because they don't know about process isolation. It's because **managing workflow state across isolated environments is genuinely hard.**

### The Hard Problems

| Problem | Why It's Hard |
|---|---|
| **Passing intermediate results** | Agent A produces output that Agent B needs. How does it get from Container A to Container B without a shared filesystem or shared memory? |
| **Workflow ordering** | Agent B must run after Agent A. Who coordinates? How does the orchestrator know Agent A is done? |
| **Shared context** | Some context is common to all agents (e.g., "the goal of this workflow is X"). How is it distributed without giving all agents access to everything? |
| **Error handling and retry** | If Agent B fails, how does the orchestrator retry? Where is the state that allows a clean retry? |
| **Performance** | Spinning up and tearing down containers for every agent invocation adds latency. Cold starts are expensive. |
| **Secret injection** | How does Agent A's private key get into Container A without passing through a shared secret store that other containers can also access? |

These are real problems. They're why most frameworks take the easy path: dump everything into one process, share memory, and hope for the best.

### Why .sgworkflow Solves This

The `.sgworkflow` file (ZIP / SQLite / shared object store) is the coordination medium that makes isolation practical:

| Problem | .sgworkflow Solution |
|---|---|
| **Passing intermediate results** | Agent A writes its signed output to the .sgworkflow. The orchestrator passes the .sgworkflow to Container B. Agent B reads Agent A's output (if it was encrypted for Agent B's key, or if it was public). **The file is the message bus.** |
| **Workflow ordering** | The orchestrator reads the .sgworkflow, checks which issues are completed (by verifying signatures), and dispatches the next issue to the next container. **The file is the state machine.** |
| **Shared context** | Common context is encrypted for ALL agents' keys (multi-recipient encryption). Per-agent secrets are encrypted for ONE agent's key. **Encryption IS the access control.** |
| **Error handling and retry** | The .sgworkflow contains the full state. If Agent B fails, the orchestrator re-dispatches the same .sgworkflow to a new Container B. The state is in the file, not in the container. **The file is the checkpoint.** |
| **Performance** | Containers can be pre-warmed. The .sgworkflow is small (metadata + encrypted blobs, not the full context). Latency is dominated by LLM inference, not container startup. |
| **Secret injection** | Agent A's private key is injected into Container A via the container runtime's secret mechanism (Kubernetes secrets, AWS Secrets Manager, etc.). The key never exists in any other container. The .sgworkflow itself doesn't contain private keys — only public keys and encrypted data. |

```
The flow:

1. Orchestrator creates .sgworkflow
   (issues, encrypted secrets per agent, agent public keys)

2. Orchestrator spins up Container A
   Injects: Agent A's private key (via secret management)
   Passes: .sgworkflow file

3. Container A:
   - Reads .sgworkflow
   - Decrypts Issue #1 (encrypted for Agent A)
   - Decrypts Agent A's secret (encrypted for Agent A)
   - Executes the task
   - Signs output with Agent A's private key
   - Writes signed output back to .sgworkflow
   - Container is destroyed (key material gone)

4. Orchestrator receives updated .sgworkflow
   Verifies Agent A's signature on the output
   Spins up Container B
   Injects: Agent B's private key
   Passes: updated .sgworkflow

5. Container B:
   - Reads .sgworkflow
   - Decrypts Issue #2 (encrypted for Agent B)
   - Reads Agent A's output (if encrypted for Agent B, or if public)
   - CANNOT decrypt Agent A's secret (encrypted for Agent A, not Agent B)
   - CANNOT access Agent A's private key (it was in Container A, which is destroyed)
   - Executes the task
   - Signs output
   - Container is destroyed

6. Repeat for each agent in the workflow
```

**The .sgworkflow file is the supply chain shipment.** It travels between isolated environments the same way it travels between supply chain participants. Each environment opens it, processes its portion, signs its contribution, and the environment is destroyed. The state management problem that makes isolation impractical is exactly what the portable data room architecture was designed to solve.

## Defence in Depth: Layered Security Model

The correct mental model is layers, not a single boundary:

| Layer | What It Provides | What It Doesn't Provide |
|---|---|---|
| **Container / VM isolation** | Process-level isolation. Agent A's code cannot read Agent B's memory. Private keys are physically separated. | Doesn't prevent a compromised agent from misusing its OWN secrets or producing malicious output. |
| **PKI encryption** | Data-level access control. Even if an attacker gains access to the .sgworkflow file, they can only decrypt sections encrypted for keys they possess. | Doesn't prevent memory access within the same process (hence the need for container isolation). |
| **Signed outputs** | Provenance and tamper detection. Every output is signed. Compromised agent's output is identifiable. | Doesn't prevent the compromised agent from producing convincing-looking malicious output. |
| **Time-bounded secrets** | Temporal limitation. Secrets expire. Even if exfiltrated, the window of exploitation is limited. | Doesn't prevent use within the valid window. |
| **Anomaly detection** | Behavioural monitoring. Unusual outputs or access patterns trigger alerts. | Doesn't prevent the first malicious action — only detects it. |
| **Trust scores** | Historical reputation. Agents with consistent good behaviour are trusted more. Sudden anomalies are flagged. | Doesn't prevent a first-time compromise. |

**No single layer is sufficient. All layers together create defence in depth.** Container isolation makes the cryptographic boundaries real. PKI encryption makes access control survive data-at-rest exposure. Signed outputs make provenance unforgeable. Time bounds limit damage. Anomaly detection catches what prevention misses.

## Practical Considerations

### Performance vs Security Tradeoff

| Model | Security | Performance | When to Use |
|---|---|---|---|
| **Serverless per agent (cold start enforced)** | Highest (if cold start) | Coldest starts, highest latency | High-security workflows: financial, medical, legal, defence. Must use separate function per agent and force cold starts or explicitly zero memory before returning. |
| **Pre-warmed container pool** | High | Warm starts, moderate latency | Production enterprise workflows |
| **Container per agent, shared host** | Good | Fast, low overhead | Standard workflows with moderate sensitivity |
| **Process per agent, different UIDs** | Moderate | Fastest | Development, testing, low-sensitivity workflows |

For our own project (19 agents): container per agent with pre-warmed pool is the practical sweet spot. Serverless for production-critical workflows where secrets are high-value.

### What About Secure Enclaves (TEE)?

Trusted Execution Environments (Intel SGX, ARM TrustZone, AMD SEV) provide hardware-enforced memory encryption. Code running in an enclave has memory that even the OS kernel cannot read.

If agents run inside TEEs, the in-process isolation claim becomes closer to true — the hardware prevents other code from reading the enclave's memory. However, TEE adoption is limited, side-channel attacks exist (Spectre, Meltdown, SGX-specific attacks), and the programming model is restrictive.

**For now**: container isolation is the practical answer. TEEs are worth tracking for high-security environments in the future.

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | Each agent runs in a separate container with only its own private key |
| 2 | Containers are ephemeral — destroyed after task completion |
| 3 | Private keys injected via secret management (not in the .sgworkflow file) |
| 4 | .sgworkflow file is the only data exchanged between containers |
| 5 | Orchestrator verifies agent signatures before dispatching to next agent |
