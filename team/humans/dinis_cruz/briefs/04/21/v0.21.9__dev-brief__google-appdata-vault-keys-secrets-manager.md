## What We Proved

The proof of concept is working. We have established:

1. **Client-side Google OAuth works.** Pure JavaScript, no server-side code. Returns the user's email, image, username.
2. **Google Drive app:data permission works.** We can request access to the user's application data folder on Google Drive. This is one of the least sensitive permissions Google offers.
3. **We can store and retrieve data in app:data.** The storage is per-application, per-user. Google cannot read it. Other apps cannot read it. It persists across sessions and devices as long as the user has their Google account.

This is the answer to the identity research brief (17 Apr): **Google Drive app:data is where the master vault key lives.**

## Part 1: Vault Key Storage via app:data

### How It Works

```
User visits sgraph.ai
    |
    v
Clicks "Sign in with Google"
    |
    +--> Google OAuth (client-side, no server)
    +--> Request scope: openid + email + drive.appdata
    |
    v
User authenticated
    |
    +--> Email, name, image available
    +--> app:data folder accessible
    |
    v
Read master keychain from app:data
    |
    +--> If exists: decrypt with master password, load vault keys
    +--> If not exists: create new keychain, set master password, save to app:data
    |
    v
User's vault keys available for the session
```

### What Gets Stored in app:data

A single encrypted JSON file (or a small set of files) containing:

```json
{
  "version": "1.0",
  "encryption": "AES-256-GCM",
  "keyDerivation": "PBKDF2-600k",
  "vaults": [
    {
      "name": "Website ops",
      "key": "<encrypted-vault-key>",
      "server": "send.sgraph.ai",
      "lastAccessed": "2026-04-21T10:30:00Z"
    },
    {
      "name": "Healthcare MVP",
      "key": "<encrypted-vault-key>",
      "server": "send.sgraph.ai",
      "lastAccessed": "2026-04-20T15:00:00Z"
    }
  ],
  "secrets": [
    {
      "name": "OpenRouter API key",
      "value": "<encrypted-value>",
      "category": "api-key"
    }
  ]
}
```

The entire file is encrypted with a key derived from the user's master password (PBKDF2, 600k iterations, consistent with existing pattern). Google stores the ciphertext. Only the user's master password can decrypt it.

### Storage Capacity

Google Drive app:data allows sufficient storage for what we need. Even hundreds of vault keys and secrets is a tiny JSON file. This is not a bulk storage solution. It is a keychain.

### Deployment: Separate Infrastructure

This should deploy on completely separate infrastructure from the main SG/Send service:

- Separate AWS account (or at minimum separate S3 bucket, separate Lambda)
- Separate domain (e.g. keys.sgraph.ai or auth.sgraph.ai)
- Separate deployment pipeline
- Different region if desired

**Why:** If a code push breaks the main SG/Send UI, it must not break the master key storage. The keychain service should be feature-complete and stable. It does not need new features. It needs to not break.

This also reduces the attack surface: the keychain service has minimal functionality (Google OAuth + app:data read/write), minimal traffic (one access per session), and can be monitored trivially.

### Cross-Domain Access

The vault keys stored in app:data need to be accessible from multiple SG/Send domains (sgraph.ai, send.sgraph.ai, tools.sgraph.ai). Since the app:data is tied to the Google OAuth application (not to a specific domain), the same OAuth client ID can be used across domains. The user authenticates once, and their keychain is available from any SG/Send property.

Investigate: confirm that the same Google OAuth client ID works across multiple domains, or whether we need a specific redirect URI configuration per domain.

### Access Token for Vaults

Once the keychain is unlocked, the user has vault keys. But they also need an access token for the vault server. Two options:

**Option A: Multiple tokens.** Each vault server has its own access token, stored alongside the vault key in the keychain. The user needs both the vault key (for encryption) and the access token (for server access).

**Option B: Vault key IS the access.** The vault key itself provides server access (the current model for most vault operations). No separate token needed.

For the MVP, Option B is sufficient. For production (especially with separate infrastructure), we may need Option A to scope access more tightly.

## Part 2: Secrets Manager Product

### The Opportunity

Now that we have Google app:data working, we can build something bigger than just vault key storage: a full secrets manager powered by Google Drive.

**The pitch:** "If you have a Google account, you have a password manager. Free. Open source. Your secrets stored in YOUR Google Drive, encrypted with YOUR password. We cannot read them. Google cannot read them."

### Why This Is Interesting

Current password managers have problems:

- **LastPass, 1Password, Bitwarden:** Subscription fees. Your secrets stored on their servers. You trust their encryption and their security practices. Breaches have happened.
- **Google Password Manager (built-in):** Terrible search. Bad UI. No secrets beyond username/password. No API. No programmatic access.
- **AWS Secrets Manager, GCP Secret Manager:** Per-secret pricing that gets expensive at scale (10, 20, 50, 100 secrets). Enterprise-focused. Overkill for individual use.

Our approach: your secrets live in your Google Drive app:data. Encrypted client-side. Open source code. You can audit exactly what happens. No subscription. No third-party server storing your secrets.

### What It Manages

Not just vault keys. A general-purpose secrets manager:

| Category | Examples |
|----------|---------|
| Vault keys | SG/Send vault keys, access tokens |
| API keys | OpenRouter, OpenAI, GitHub, AWS |
| Passwords | Website logins (username + password) |
| SSH keys | Private keys for server access |
| Environment variables | Development secrets, config values |
| Notes | Secure notes, recovery codes, PINs |

### The Two Storage Tiers

| Tier | Storage Location | Needs Vault Credits | Syncs Across Devices |
|------|-----------------|--------------------|--------------------|
| **app:data tier** | Google Drive app:data | No | Yes (via Google account) |
| **Vault tier** | SG/Send vault | Yes (vault storage) | Yes (via vault) |

Secrets in the app:data tier are stored directly in Google Drive. No SG/Send server involved. No credits needed. No vault needed. Just Google OAuth + client-side encryption.

Secrets in the vault tier are stored in an SG/Send vault. This allows sharing secrets with other people or agents (via vault collaboration), version control of secrets, and larger storage.

The user can move secrets between tiers. The UI treats both as one unified keychain.

### Distribution Options

**Web app:** A page on sgraph.ai (or a dedicated domain) where you log in with Google and manage your secrets. Works in any browser.

**Chrome extension:** A browser extension that auto-fills passwords and provides quick access to secrets. The extension reads from app:data via the same Google OAuth flow.

**CLI:** A command-line tool for developers who want to access secrets from scripts and CI pipelines.

All three share the same underlying data (app:data + optionally vault). The web app is the MVP. The Chrome extension and CLI are future iterations.

### Open Source Advantage

The entire secrets manager is open source. The user can read every line of code that touches their secrets. This is a trust proposition that LastPass and 1Password cannot match. "Do not trust us. Read the code. Your secrets never leave your browser unencrypted."
