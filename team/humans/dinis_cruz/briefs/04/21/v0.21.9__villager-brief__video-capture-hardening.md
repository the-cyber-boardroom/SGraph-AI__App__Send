## The Situation

The video capture tool works. The UI and workflow are good. But it is an Explorer-grade component being used in production, and it shows: video freezes, visualisation freezes, audio desync. These are not edge cases. They happen during normal use.

The capture has one job: record data. Anything less than recording correctly is a catastrophic failure. The health checks and quality controls are not add-ons. They are the core of what this component does.

Time to make it Villager-grade.

## The Approach: Decouple and Rebuild the Recorder

The current implementation mixes recording logic with UI logic. The Villager refactoring separates them completely.

### The Recorder Component (The One That Must Never Fail)

A standalone Web Component (or set of components) whose only job is capturing media streams reliably. It does not care about UI, layout, file management, or vault integration. It receives instructions (start, stop, pause, resume) and produces recorded data. That is it.

**Three sub-components, each handling one stream:**

| Component | Responsibility |
|-----------|---------------|
| `<sg-recorder-audio>` | Capture audio from microphone via `getUserMedia` |
| `<sg-recorder-video>` | Capture video from camera via `getUserMedia` |
| `<sg-recorder-screen>` | Capture screen/window/tab via `getDisplayMedia` |

**One orchestrator:**

| Component | Responsibility |
|-----------|---------------|
| `<sg-recorder>` | Manages the sub-components. Handles combined vs individual stream recording. Exposes the JS API. |

Each sub-component is independently testable. Each can be used alone. The orchestrator composes them.

### The JS API

```javascript
const recorder = document.querySelector('sg-recorder')

// Configuration
recorder.configure({
  audio: true,
  video: true,
  screen: false,
  format: 'webm',          // or 'mp4' if supported
  combinedStream: true,     // record all streams as one file, or separate files
  healthCheckInterval: 1000 // ms between health checks
})

// Control
await recorder.start()
recorder.pause()
recorder.resume()
const blob = await recorder.stop()  // returns the recorded data

// Health / Status
recorder.getStatus()    // { recording: true, duration: 45.2, audioActive: true, videoActive: true, ... }
recorder.getMetrics()   // { bytesRecorded: 12400000, framesRecorded: 1356, droppedFrames: 2, ... }

// Events
recorder.addEventListener('health-warning', (e) => { ... })
recorder.addEventListener('recording-error', (e) => { ... })
recorder.addEventListener('stream-lost', (e) => { ... })
```

The component receives instructions via this API. It does not know about the UI that wraps it. Any tool (the video capture tool, the screen recorder, a future meeting recorder) can use this component.

## The Bugs to Fix (Root Causes)

### Video Freeze

Likely cause: the browser throttles inactive tabs. When the user switches to another tab, the recording tab loses priority and the MediaRecorder receives fewer frames or pauses entirely.

**Solutions to investigate:**

- Run the MediaRecorder in a Web Worker (decoupled from the main thread and tab visibility). This is the most promising approach: Web Workers are not throttled when the tab is inactive.
- Use the `visibilitychange` event to detect tab switches and warn the user ("Recording may be affected while this tab is in the background").
- Request a Wake Lock (via the Screen Wake Lock API) to prevent the device from sleeping.
- Use `requestAnimationFrame` alternatives that are less susceptible to throttling.

### Audio Desync

Likely cause: separate MediaRecorder instances for audio and video with slightly different clock sources, or the combined stream losing sync under load.

**Solutions to investigate:**

- Record audio and video as a single combined MediaStream (one MediaRecorder, not two). This is the most reliable sync approach.
- If recording separately (for flexibility), use timestamps on each chunk and re-sync during assembly.
- Monitor clock drift between streams and warn if it exceeds a threshold.

### Visualisation Freeze

Likely cause: the visualisation (waveform, preview, metrics display) is on the main thread and competes with the recording for resources.

**Solution:** The recording must never compete with the visualisation. If the main thread is busy rendering the UI, the recording continues in its Web Worker unaffected. The visualisation can drop frames or lag. The recording cannot.

## Health Checks and Fail-Safes

This is the core of the Villager work. The recorder's reliability monitoring is not a feature. It is the product.

### Continuous Health Monitoring

Every N milliseconds (configurable, default 1 second), the recorder checks:

| Check | What It Detects | User Feedback |
|-------|----------------|---------------|
| MediaRecorder state | Has the recorder stopped unexpectedly? | "Recording stopped unexpectedly. Last N seconds may be lost." |
| Bytes recorded delta | Is data still being produced? (Zero bytes in last interval = frozen) | "No data recorded in the last 3 seconds. Recording may have stalled." |
| Frame count delta | Are frames still arriving? (Video-specific) | "Video frames stopped arriving. Camera may have disconnected." |
| Audio level | Is audio signal present? (Silence detection) | "No audio detected. Check microphone." |
| Dropped frames | Are frames being dropped? (Quality degradation) | "Frames being dropped. Recording quality may be reduced." |
| Tab visibility | Is the tab in the background? | "Tab is in background. Switch back to ensure recording quality." |
| Memory usage | Is the recording consuming excessive memory? | "Recording is using significant memory. Consider stopping and saving." |
| Stream active state | Is the MediaStream track still active? | "Camera/microphone disconnected." |

### Visual Metrics Dashboard

Display recording metrics in real-time alongside the preview:

- Duration (elapsed time)
- Data size (bytes recorded, updating live)
- Data rate (KB/s, provides a visual heartbeat that something is happening)
- Frame rate (actual vs target, video only)
- Audio level (waveform or VU meter)
- Dropped frames count
- Health status indicator (green/amber/red)

The data rate visualisation is particularly important: a live chart of KB/s over time gives an instant visual cue when something goes wrong. A sudden drop to zero is unmissable. A gradual decline signals degradation.

### Error Recovery

When the recorder detects a problem:

1. **Warn immediately.** Do not silently fail. A banner, a colour change, a sound. The user must know something is wrong.
2. **Save what we have.** If the recording is irrecoverable, save the data recorded up to the failure point. Do not lose everything because the last 5 seconds failed.
3. **Attempt recovery.** If a stream drops (camera disconnected), attempt to re-acquire. If the MediaRecorder stops, attempt to restart with the existing streams.
4. **Log everything.** Every health check result, every warning, every error, every recovery attempt. This log is available via `recorder.getHealthLog()` for debugging.

## Testing Strategy

### Simulation Tests

Create test harnesses that simulate failure conditions:

- Tab switch during recording (visibility change)
- Camera disconnect mid-recording (stream track ended)
- Memory pressure (large recordings, limited memory)
- Long recordings (1 hour+, looking for drift and memory leaks)
- Multiple simultaneous streams (audio + video + screen)
- Rapid start/stop cycles

### Cross-Browser Testing

MediaRecorder behaviour varies across browsers. Test on:

- Chrome (primary target)
- Firefox (different MediaRecorder implementation)
- Safari (historically weakest MediaRecorder support)
- Mobile Chrome and Safari

Document known limitations per browser. The component should degrade gracefully (warn and disable unsupported features) rather than fail silently.

## What the UI Tool Does (Separately)

The video capture tool on tools.sgraph.ai wraps the recorder component with:

- Source selection UI (which camera, which microphone, which screen)
- Preview display
- Start/stop/pause controls
- Metrics dashboard (fed by `recorder.getMetrics()`)
- Health warnings (fed by `recorder` events)
- Save/share/vault actions (using shared footer components)

The tool is the UI. The recorder is the engine. If the recorder is solid, the tool is solid. If the recorder is flaky, no amount of UI polish helps.
