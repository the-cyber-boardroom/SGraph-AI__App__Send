# Task-Focused LLM Components: Use Case First, Model Second

**version** v0.21.9
**date** 21 Apr 2026
**from** Human (project lead)
**to** Developer (lead), Designer
**type** Dev brief

---

## The Principle

Do not build "a Claude component" or "a Gemini component." Build a "review document" component, a "research topic" component, a "create infographic" component. The model is an implementation detail inside the component, chosen because it is the best fit for that specific task.

The user thinks in tasks, not models. "I want to spell-check this." "I want evidence for this claim." "I want an infographic from this brief." The component handles model selection, prompt construction, output formatting, and all the model-specific quirks internally. The caller gets a clean task-oriented API.

## Why Components, Not a Chat Interface

A general-purpose LLM chat lets you do anything but optimises for nothing. These components optimise for specific, high-frequency tasks. Each component can:

- Use the right model for the task (not whichever model the user last selected)
- Construct prompts that account for model-specific behaviours, requirements, and bugs
- Format output for the specific task (not generic markdown)
- Handle known model quirks internally (Gemini's image generation constraints, Perplexity's citation format, Claude's structured output strengths)
- Display cost transparently per task
- Be embedded in any tool or page as a Web Component

## The Seven Components

### 1. `<sg-task-review>` : Review Document

**Task:** Drop a document (PDF, markdown, text), get a structured review with findings, suggestions, and an overall assessment.

**Model:** Claude (strongest at structured analysis and nuanced feedback)

**Input:** Document (file drop or text paste)
**Output:** Structured review: summary, strengths, weaknesses, specific suggestions with line/section references, overall assessment score or rating

**Why Claude:** Produces the most consistently structured and actionable reviews. Handles long documents well. Good at maintaining the author's voice in suggestions.

---

### 2. `<sg-task-spellcheck>` : Spell Check Document

**Task:** Drop a document, get it back with spelling, grammar, and style corrections highlighted.

**Model:** Claude (most reliable for preserving meaning while fixing language)

**Input:** Document (file drop or text paste)
**Output:** Corrected document with changes highlighted (diff view or track-changes style). Summary of changes made. Option to accept/reject individual changes.

**Why Claude:** Least likely to silently change meaning while correcting language. Good at understanding technical terminology and not "correcting" domain-specific terms.

**Implementation note:** The diff view is critical. The user must see exactly what changed. Do not just return a corrected document with no indication of changes.

---

### 3. `<sg-task-discuss>` : Discuss Ideas

**Task:** Paste an idea, a brief, a strategy, or a question. Get a structured discussion: arguments for and against, alternative approaches, questions to consider, potential blind spots.

**Model:** Claude (strongest at multi-perspective analysis and intellectual honesty about trade-offs)

**Input:** Text (paste or type)
**Output:** Structured discussion: the idea restated, arguments in favour, arguments against, alternative approaches, questions the author should consider, blind spots or assumptions identified

**Why Claude:** Best at "thinking about thinking." Produces genuinely useful counter-arguments rather than agreeable summaries.

---

### 4. `<sg-task-infographic>` : Create Infographic

**Task:** Drop a document or brief, get an infographic (SVG) summarising the key points visually.

**Model:** Gemini (for image generation) or Claude (for structured content extraction that feeds the existing sg-llm-infographic pipeline)

**Input:** Document (file drop or text paste), optional style/theme selection
**Output:** SVG infographic rendered inline, downloadable, saveable to vault

**Why this model routing:** The infographic pipeline already exists (sg-llm-infographic, sub-10-cent cost, 15 seconds). This component wraps it with a task-focused UI: drop document, get infographic. The model choice depends on whether we are generating the visual directly (Gemini) or extracting structured content then rendering via the existing SVG pipeline (Claude to extract, then the renderer).

**Reuse:** This component should import and wrap the existing sg-llm-infographic, sg-llm-events, and sg-llm-request components from tools.sgraph.ai. Not rebuild them.

---

### 5. `<sg-task-image>` : Create Image

**Task:** Describe an image or provide a document/context, get a generated image.

**Model:** Gemini (native image generation via OpenRouter)

**Input:** Text prompt, or document + instruction ("create a header image for this article")
**Output:** Generated image rendered inline, downloadable, saveable to vault

**Why Gemini:** Currently the most accessible image generation model via OpenRouter. Known quirks to handle internally: aspect ratio constraints, content policy boundaries, consistency of style across multiple generations.

**Implementation note:** If Gemini's image generation has a content policy rejection, the component should surface a clear message (not a generic error) and suggest prompt adjustments.

---

### 6. `<sg-task-research>` : Research Topic

**Task:** Provide a topic, question, or brief. Get a research summary with sourced evidence and links.

**Model:** Perplexity Sonar (the only model that provides real, verifiable source links)

**Input:** Topic, question, or document to research around
**Output:** Research summary with inline citations. Each claim linked to its source URL. Sources listed at the bottom with titles and URLs. Confidence indicators where possible.

**Why Perplexity Sonar:** This is the critical differentiator. As far as we can tell, Perplexity (via its Sonar models on OpenRouter) is the only provider that returns genuine source links with its responses. Claude, Gemini, and others generate plausible-sounding claims but do not provide verifiable evidence URLs. For any task where the user needs to trust and verify the output, Perplexity Sonar is the only credible option.

**Implementation note:** The source links are the product here, not the summary text. The component should render links prominently, not as footnotes. Each claim should be visually connected to its source.

---

### 7. `<sg-task-evidence>` : Find Evidence for Text

**Task:** Paste a document, an article, or a set of claims. Get evidence (with source links) supporting or contradicting each major claim.

**Model:** Perplexity Sonar

**Input:** Document or text with claims to verify
**Output:** Each major claim extracted, with: supporting evidence (with source URLs), contradicting evidence (with source URLs), confidence assessment, and "evidence gap" flags for claims that cannot be verified.

**Why Perplexity Sonar:** Same reasoning as Research Topic. The source links are the entire value. Without verifiable URLs, this tool is just another LLM rephrasing claims without grounding.

**Implementation note:** This is the most valuable of the seven components for professional use cases (investor materials, security reviews, compliance documentation, articles). The ability to say "every claim in this document has a source link" is a differentiator for any deliverable.

## Shared Architecture

All seven components should:

### Share the OpenRouter integration layer

A single `sg-llm-request` module handles all OpenRouter API calls. Each component specifies its model and prompt. The integration layer handles authentication, streaming, error handling, cost tracking, and rate limiting.

```
<sg-task-review>  ─┐
<sg-task-spellcheck> ─┤
<sg-task-discuss>  ─┤
<sg-task-infographic> ─┼──> sg-llm-request ──> OpenRouter API
<sg-task-image>   ─┤
<sg-task-research>  ─┤
<sg-task-evidence>  ─┘
```

### Share the input component

All seven accept document input (file drop, text paste, or clipboard). Use a shared `<sg-task-input>` component for the drop zone and text area. Do not build seven different input UIs.

### Share the output/save component

All seven produce output that the user may want to save (as SG/Send file, to vault, or download). Use the `<sg-send-drop>` component from the send/receive reference implementation brief (19 Apr) for saving output. One "Save" button, one integration.

### Display cost per task

Every component shows the cost of the LLM call after completion. "This review cost 3.2 cents." Transparency is a product principle.

### Expose the standard JS API

Each component exposes `getManifest()`, `getMethods()`, `getSkills()`. Each is callable from console, Playwright, other components, and agents.

```javascript
// Programmatic usage
const review = document.querySelector('sg-task-review')
const result = await review.execute({ document: myFile })
```

## Model-Specific Quirk Handling

This is the hidden value of task-focused components. Each component absorbs the model's quirks so the user does not have to deal with them:

| Model | Known Quirks to Handle |
|-------|----------------------|
| **Claude** | Tends to be overly cautious with content warnings. Can refuse benign tasks if prompts are not framed carefully. Excellent at structured output but needs explicit formatting instructions. |
| **Gemini** | Image generation has strict content policies with unhelpful error messages. Aspect ratio constraints. Inconsistent style across generations. Output format varies between versions. |
| **Perplexity Sonar** | Source link format may change between API versions. Some sources may be paywalled. Citation numbering can be inconsistent. Response structure differs from standard chat models. |

Each component's prompt engineering and output parsing accounts for these. When Gemini updates and breaks the image format, we fix it in `<sg-task-image>`, not in every tool that generates images. When Perplexity changes its citation format, we fix it in `<sg-task-research>`, not everywhere.

## Deployment

All components deploy on tools.sgraph.ai as both:

- Standalone tools (a page per task, usable directly)
- Embeddable components (droppable into any page, any tool, any vault page layout)

The standalone page for each task should be minimal: the component, a brief description, and nothing else. The power is in the embedding: any vault page, any tool, any workflow can include `<sg-task-evidence>` and get evidence-finding capability without building anything.

## Acceptance Criteria

| # | Criterion | Verification |
|---|-----------|-------------|
| 1 | `<sg-task-review>` accepts a document and returns structured review via Claude | Drop a PDF, receive review with summary, strengths, weaknesses, suggestions |
| 2 | `<sg-task-spellcheck>` returns corrected document with diff view | Drop text with errors, see corrections highlighted, accept/reject individually |
| 3 | `<sg-task-discuss>` returns multi-perspective analysis | Paste an idea, receive arguments for, against, alternatives, blind spots |
| 4 | `<sg-task-infographic>` produces SVG from document input | Drop a brief, receive rendered infographic |
| 5 | `<sg-task-image>` generates image via Gemini | Provide prompt, receive generated image |
| 6 | `<sg-task-research>` returns summary with verifiable source links via Perplexity | Provide topic, receive summary where every claim has a clickable source URL |
| 7 | `<sg-task-evidence>` extracts claims and finds evidence with source links | Paste a document, receive per-claim evidence with supporting/contradicting sources |
| 8 | All components share sg-llm-request for OpenRouter calls | Single integration layer, not seven separate API implementations |
| 9 | All components share input component (file drop, text paste) | Consistent input UX across all seven |
| 10 | All components display cost after completion | Cost visible in UI after each task |
| 11 | All components expose standard JS API | getManifest, getMethods, getSkills callable from console |
| 12 | All components saveable via SgSend.sendFile() or vault | Output saveable with one click |
| 13 | Model-specific quirks handled internally (not exposed to user) | Gemini content policy rejection shows clear message, not raw error |
| 14 | Perplexity source links rendered prominently (not footnotes) | Links visually connected to claims, not buried at bottom |
