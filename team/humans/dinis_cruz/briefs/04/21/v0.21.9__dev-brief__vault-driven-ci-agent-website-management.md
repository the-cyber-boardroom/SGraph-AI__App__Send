# Vault-Driven CI: Agent-to-Pipeline Communication for Website Management

**version** v0.21.9
**date** 21 Apr 2026
**from** Human (project lead)
**to** DevOps (lead), Developer, Ambassador
**type** Dev brief

## The Concept

Use vaults as the communication layer between agents and CI pipelines. The Ambassador agent writes instructions, content, and configuration to a vault. The CI pipeline reads from that vault and deploys. The vault is the contract between "what should the website look like" and "make it so."

This is the hero line in practice: version-controlled, client-encrypted vaults for humans and agents. The agent is the Ambassador. The "human" is the CI pipeline (or the human reviewing the changes before deploy). The vault carries the intent.

## The Problem Today

Updating the website requires a developer to make changes, commit to GitHub, and trigger a deploy. This is fine for code changes, but overkill for content changes: updating a promotion, changing the free tier budget, rotating an OpenRouter key, adding a daily infographic, publishing new text.

These are operational decisions, not engineering decisions. An agent should be able to make them without touching the codebase.

## The Architecture

```
Ambassador Agent (Claude session)
    |
    +--> Reads the repo, understands current state
    +--> Given a vault key for the "website-ops" vault
    +--> Writes content, config, and instructions to the vault
    |       |
    |       +--> daily-infographic.svg
    |       +--> promotion-config.json
    |       +--> free-tier-budget.json
    |       +--> new-article.md
    |       +--> feature-toggles.json
    |
    v
Website-Ops Vault (the contract)
    |
    +--> Version controlled (every change is a commit)
    +--> Encrypted (only the agent and CI have the key)
    +--> Auditable (full history of who changed what)
    |
    v
CI Pipeline (GitHub Actions)
    |
    +--> Triggered by webhook or schedule
    +--> Clones the vault (shallow, only what it needs)
    +--> Reads the latest content and config
    +--> Builds the website with the new content
    +--> Deploys to S3 + CloudFront
    +--> Done
```

## Two Delivery Modes

### Mode 1: Direct Vault Rendering (Live)

The website opens the vault directly and renders content from it. We already have this capability: HTML pages in vaults can load data from other files in the vault. Markdown pages render natively. Infographics display inline.

In this mode, there is no CI pipeline step for content. The Ambassador updates the vault. The website reads from the vault on the next page load. Changes are live as soon as they are pushed.

**Best for:** Daily infographics, rotating content, promotional text, dynamic sections.

**Limitation:** Depends on the vault server being available. Content is not baked into the static site.

### Mode 2: CI Pipeline Deploy (Static)

The CI pipeline reads from the vault and bakes the content into the static website build. Changes go live when the pipeline runs and deploys.

**Best for:** Feature toggles, HTML page updates, structural changes, anything that should be QA'd before going live.

**How it works:**
1. Ambassador commits changes to the vault
2. Vault sends a webhook to GitHub (the webhook URL is stored in the vault itself)
3. GitHub Actions triggers the build pipeline
4. Pipeline clones the vault (using vault key stored as a GitHub secret or environment variable)
5. Pipeline reads config and content from the vault
6. Pipeline builds the static site incorporating the vault content
7. Pipeline deploys to S3

Both modes can coexist. Some content renders directly from the vault (Mode 1). Some content is baked in during deploy (Mode 2).

## What the Ambassador Agent Manages

| Content | Vault File | Mode |
|---------|-----------|------|
| Daily infographic | `content/daily-infographic.svg` | Mode 1 (live) |
| Promotional text | `config/promotion.json` | Mode 1 or 2 |
| Free tier budget (how many free transfers per day) | `config/free-tier-budget.json` | Mode 2 (needs deploy) |
| Shared OpenRouter key for demos | `secrets/demo-openrouter-key.enc` | Mode 2 (encrypted, CI decrypts) |
| Feature toggles | `config/feature-toggles.json` | Mode 2 (needs deploy) |
| New articles or blog posts | `content/articles/` | Mode 1 or 2 |
| Seasonal campaigns | `config/campaign.json` | Mode 2 |

## The Agent Session Flow

```
Human starts Ambassador session:
    "Here's the repo. Act as the Ambassador. Here's your vault key."
    |
    v
Ambassador reads the repo, understands current state
    |
    v
Ambassador clones the vault (shallow)
    |
    v
Ambassador receives instructions:
    "Set the free tier to 50 transfers per day for the next week"
    "Create a daily infographic about our AWS Marketplace launch"
    "Enable the video recorder tool on the main site"
    |
    v
Ambassador writes changes to vault, commits with descriptive messages:
    "Set free-tier-budget to 50/day until 28 Apr"
    "Added daily infographic: AWS Marketplace launch"
    "Enabled feature toggle: video-recorder"
    |
    v
Ambassador pushes to vault
    |
    v
Webhook triggers CI pipeline (or content goes live via Mode 1)
```

## The Budget Concept

The Ambassador agent can be given a budget: a specific OpenRouter credit allocation for demo purposes, a specific number of free transfers to offer, a specific spend limit on infographic generation. The budget is stored in the vault as a config file. The CI pipeline reads it and enforces it.

This allows a daily or weekly agent run: "Ambassador, review the current state, create today's content, manage the budget, push to the vault." Automated, audited, version-controlled.

## Vault Management at Scale

As more vaults come into play (website-ops, investor materials, demo packs, customer review deliverables, agent collaboration, Playwright configs), we need a meta-view. One of the agents is already tracking vault creation and maintaining a registry in a dedicated vault. This is the beginning of the "graph of graphs" concept: a vault that indexes other vaults.

For now, this is sufficient. As the number grows, we may need a proper vault registry with search, tagging, and access control metadata. But the current approach (an agent maintains a vault listing in a vault) is pragmatic and works.

## Security Properties

| Property | How Vault Achieves It |
|----------|----------------------|
| Least privilege | Agent gets one vault key, not repo access or AWS credentials |
| Audit trail | Every change is a commit with timestamp and message |
| Separation of duties | Agent writes intent. CI pipeline executes deploy. Human reviews if needed. |
| Rollback | Revert to any previous vault snapshot to undo changes |
| Encryption | Content encrypted at rest. Only agent and CI have the key. |
| Monitoring | Two consumers (agent + CI). Any other access is anomalous. |
