## Tool 1: OpenAI Image Generation via OpenRouter

### The Opportunity

OpenAI released their latest image model (Image 2) and it is genuinely good. This is a concrete example of the pattern from the task-focused components brief: create a dedicated tool for a specific model capability, wrap it in the SG/Send ecosystem, and sell access via OpenRouter credits.

### What It Does

A client-side tool on tools.sgraph.ai where the user types a prompt (or drops a document for context), selects OpenAI's image model via OpenRouter, and gets generated images back. Images are saved to a vault, not stored on our servers.

### Why This Matters Beyond the Tool

This is a template for every new model capability that ships. When a new model drops (image, video, audio, reasoning), we should be able to spin up a dedicated tool for it within hours:

1. Create the tool page (reusing shared components: `<sg-task-input>`, `<sg-header>`, `<sg-footer-actions>`, `<sg-cost-display>`)
2. Wire it to OpenRouter with the model ID
3. Add vault save for the output
4. Deploy

The privacy angle is real: via OpenRouter, the API call is connected to a key, not to an individual. The images are encrypted in the user's vault. Compare this to using OpenAI directly (account required, images stored on their servers, linked to your identity) or the web UI (subscription required).

### Implementation

Use the `<sg-task-image>` component from the task-focused components brief. This is just a deployment of that component with OpenAI's model pre-selected (while still allowing model switching for comparison).

## Tool 2: Secrets Manager (Local Storage + Vault)

### The Problem (Clarified)

The identity research brief (17 Apr) and the auth MVP brief (18 Apr) explored where to store vault keys via identity providers (Cognito, Firebase, Google OAuth). But there is a simpler, more immediate need: a local secrets manager that runs in the browser.

The key insight from this memo: **the identity provider only needs to store the master secret.** Everything else can be managed locally. The master password (or master key) unlocks a local keychain. The keychain stores all individual vault keys. The master password is the only thing that needs to live in LastPass, or Google app:data, or wherever the user keeps their one critical secret.

This decouples two problems:
- **Where to store the master secret:** LastPass, 1Password, Google app:data, Cognito, brain (separate problem, separate brief)
- **How to manage the keychain once unlocked:** this tool

### What It Does

A tool (and reusable Web Component) that manages vault keys and other secrets in the browser.

**Mode A: localStorage-based (immediate)**

Vault keys stored in the browser's localStorage, encrypted with a master password. The user sets a master password on first use. On subsequent visits, they enter the master password to unlock the keychain. All vault keys are decrypted locally.

```
User enters master password
    |
    v
Master password derives encryption key (PBKDF2, 600k iterations)
    |
    v
Decrypt keychain from localStorage
    |
    v
Individual vault keys available for use
```

Limitations: browser-local only (does not sync across devices), cleared if user clears browser data. But it works right now, requires no server infrastructure, and solves the immediate problem of "I have 15 vault keys and they are all in a text file on my desktop."

**Mode B: Vault-backed (portable)**

The keychain itself is stored as a vault. The master password encrypts the keychain vault. The encrypted keychain can be synced across devices by sharing the vault.

This is recursive (a vault storing keys to other vaults) but it works cleanly: the keychain vault is just another vault with a known structure. The master password is the one secret that is not stored in the system.

### The Component

`<sg-secrets-manager>`: a Web Component that any tool can embed to provide "remember my vault keys" functionality. The task launcher shell, the send/receive tool, the video recorder, any tool that needs a vault key can use this component instead of asking the user to paste a key every time.

### What This Does NOT Solve

Where to store the master secret itself. That is the domain of LastPass, 1Password, Google app:data, or the user's memory. This tool manages everything that the master secret unlocks.

## Tool 3: Credit Sales Workflow (Simplified OpenRouter Key Distribution)

### The Problem

Right now, if someone wants to use the LLM-powered tools, they need their own OpenRouter key or OpenAI key. Getting one requires creating an account, adding a payment method, understanding API keys. This friction kills casual users.

### What It Does

A simple shopping cart workflow on sgraph.ai where the user:

1. Selects a credit amount (1 pound, 5 pounds, 10 pounds)
2. Pays via Stripe
3. Receives an OpenRouter API key with that credit balance pre-loaded
4. Uses the key in any tool on the platform

The user does not need an OpenRouter account. They do not need to understand API keys. They buy credits, they get a key, they use the tools.

### Pricing and Refund Model

**Pricing:** We sell at a markup on OpenRouter rates (25% as per the existing model). The user pays for credits. We provision the key on OpenRouter with the equivalent amount minus our margin.

**Refund / expiry model:** Credits expire after a period (start with 1 month, extend to 3 months if usage patterns support it). Unused credits are refunded automatically. This avoids the "unused credits as revenue" problem, which feels wrong and creates regulatory complexity. The business makes money on the margin of credits that are actually used, not on breakage.

This is a philosophical choice: we do not want credit sales to be a money-making operation based on people forgetting to use what they bought. We want it to be a frictionless access mechanism where unused value returns to the customer.

### Privacy Angle

Interesting side effect: OpenRouter keys are not tied to an individual identity. The key tracks usage, but the key itself is not linked to a name, email, or account. The user gets more privacy than using OpenAI, Google, or Anthropic directly (all of which require accounts with personal information).

This is worth mentioning in the tool's description: "More private than using the model provider directly."

### Implementation

The backend needs:
- Stripe checkout flow (already partially built for the existing credits system)
- OpenRouter key provisioning API (create a key with a specific credit balance)
- Key delivery (display the key to the user, optionally save to their secrets manager)
- Expiry tracking (cron or Lambda function to check for expired keys, trigger refunds)

The frontend needs:
- Credit amount selector
- Stripe checkout integration
- Key display and copy
- Optional: save to `<sg-secrets-manager>` component

## How These Three Connect

```
User buys credits (Tool 3)
    |
    +--> Gets OpenRouter key
    +--> Saves key in secrets manager (Tool 2)
    |
    v
User opens image generator (Tool 1)
    |
    +--> Secrets manager provides OpenRouter key automatically
    +--> User generates images
    +--> Images saved to vault
    +--> Cost displayed per image
```

Each tool is independently useful. Together, they create a frictionless flow from "I want to try this" to "I am using it and my data is encrypted."
