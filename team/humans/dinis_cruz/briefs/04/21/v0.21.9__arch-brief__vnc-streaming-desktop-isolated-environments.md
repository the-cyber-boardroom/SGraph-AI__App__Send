# Streaming Desktop: VNC-Based Isolated Environments for Browsing, Viewing, and Agent Interaction

**version** v0.21.9
**date** 21 Apr 2026
**from** Human (project lead)
**to** Architect (lead), Developer, DevOps
**type** Arch brief

## The Idea

Run a browser (or any application) inside a container with a desktop environment, stream it to the user via VNC (or similar), and give the user keyboard and mouse access only. The user sees and interacts with the application but never has direct access to the underlying environment, the files, the secrets, or the network.

This is remote desktop as a security primitive, not as a convenience feature.

## The Origin: Accessing the MITM Proxy

The immediate need: we have a Playwright container with a MITM proxy sidecar. The MITM proxy has a web UI for browsing captured traffic, debugging requests, and inspecting sessions. Right now, accessing that UI requires SSH into the EC2 instance. We need a way for users to browse the MITM proxy UI directly, without giving them SSH access to the machine.

VNC solves this. Run a lightweight desktop with a browser inside the container, point the browser at the MITM proxy UI, stream the desktop to the user. The user sees the proxy dashboard. The user cannot access anything else on the machine.

But once you have this capability, the use cases multiply fast.

## Use Case 1: Isolated Browsing (Secure Browser)

The user browses the internet through a container. The browser runs remotely. The user sees the screen via VNC. Malware, tracking scripts, malicious downloads: all contained inside the ephemeral container. The user's local machine is never exposed.

This is what "browser isolation" products do, but we can build it on top of infrastructure we already have (the Playwright container + MITM proxy). The MITM proxy adds full traffic logging. The vault provides encrypted storage for anything the user saves.

## Use Case 2: Secure Document Viewing (Read Without Download)

This is where vaults and VNC combine powerfully.

A vault contains sensitive documents. The recipient should be able to read them but not download them. With VNC, we render the document inside the container and stream the view to the user. The user sees the document. The user can scroll, navigate, zoom. But the actual file never leaves the container. The user cannot copy-paste the content. The user cannot save the file locally. The worst they can do is take a screenshot of their screen, which you can never fully prevent, but it is a far cry from having the original file.

This is a lightweight secure document viewer. No DRM, no special client software. Just a VNC stream of a rendered document inside an ephemeral container.

**Audit capability:** Because the container controls the rendering, we can log exactly which pages the user viewed, how long they spent on each page, which sections they scrolled to. Full auditability of document access, not just "file was downloaded."

## Use Case 3: Agentic Visual Interaction

Two interesting modes:

**User watches the agent.** The Playwright container is already automating a browser (headless). With VNC, we can make it non-headless: the user sees the browser as the agent controls it. Pages loading, buttons being clicked, forms being filled. The user watches the agent work in real time. This is powerful for debugging, for demos, and for trust-building ("see what the agent is doing, step by step").

**User and agent collaborate.** The user has mouse and keyboard. The agent has programmatic control. Both interact with the same browser session. The user can intervene, correct, or guide the agent. The agent can automate repetitive parts while the user handles the judgment calls. This is a genuinely novel interaction pattern: human-agent pair browsing.

**Computer vision integration.** Modern vision models (Gemini, Claude) can understand screenshots well. An agent could take VNC screenshots, analyse what is on screen, and decide what to do next. This opens up automation of applications that have no API, only a GUI. The agent "sees" the screen and acts on it.

## Use Case 4: Ephemeral Workspaces (Billable Sessions)

Spin up a container with specific tools, configurations, or applications. The user accesses it via VNC. When the session ends, the container is destroyed. Nothing persists except what was explicitly saved to a vault.

This is a billable unit: "user session, 30 minutes, with Playwright + MITM + Chrome, billed at X per hour." The infrastructure is ephemeral. The data governance is clean (nothing persists outside the vault).

**Applications you could deliver this way:**

- Claude Code (if it runs on Linux) in an isolated, pre-configured environment
- Terminal tools with specific configurations and secrets injected at runtime
- Custom browsers with specific extensions, profiles, or proxy configurations
- Any Linux application that needs to be accessible but isolated

## Use Case 5: Collaborative Viewing

Multiple users connect to the same VNC session. They all see the same screen. One person controls the mouse (or they take turns). This is screen sharing without a screen sharing app, running inside a controlled, logged, encrypted environment.

Use case: a consultant walks a client through findings in a vault. Both see the same view. The consultant controls navigation. Everything is recorded. The underlying documents never leave the container.

## Architecture

```
User's Browser
    |
    v
VNC Client (noVNC in browser, or native VNC client)
    |
    v (VNC stream over WebSocket)
    |
EC2 Instance / Container
    |
    +--> Lightweight Desktop (e.g. Xvfb + fluxbox or similar)
    |       |
    |       +--> Chromium (non-headless, visible on desktop)
    |       |       |
    |       |       +--> MITM proxy UI
    |       |       +--> Vault content rendered in browser
    |       |       +--> Web applications
    |       |       +--> Agent-controlled automation (visible)
    |       |
    |       +--> Terminal (optional)
    |       +--> Other applications (optional)
    |
    +--> MITM Proxy (sidecar, captures all traffic)
    |
    +--> Vault Access (encrypted storage for session data)
    |
    +--> VNC Server (e.g. TigerVNC, x11vnc)
    |
    +--> noVNC (WebSocket-to-VNC bridge for browser access)
```

**noVNC** is the key component for browser-based access: it bridges WebSocket connections to the VNC protocol, so the user accesses the remote desktop from any browser without installing a VNC client.

## Secret Injection (Enhanced)

This architecture amplifies the MITM proxy secret injection pattern from the Playwright brief. In a VNC session:

- The user browses a website that requires authentication
- The MITM proxy injects the credentials into the request
- The user sees the authenticated page
- The user never sees the credentials (they are injected at the proxy layer)
- The user never has the credentials in their browser, clipboard, or local storage

This is "capability without knowledge of the secret" taken to the extreme. The user can use authenticated services without ever possessing the authentication credentials. The credentials exist only inside the ephemeral container, injected by the proxy, destroyed when the session ends.

## Container Orchestration Requirement

To make this work as a multi-user service, we need a container orchestrator that can:

- Start a new container per user session (with VNC, desktop, browser, MITM proxy)
- Assign a unique VNC port or WebSocket endpoint per session
- Monitor session health and activity
- Destroy containers when sessions end (or after timeout)
- Track session duration for billing

This is a new FastAPI service (a "session manager") that runs with Docker socket access. It manages the lifecycle of VNC session containers. This is the service that makes the whole thing billable and scalable.

## Research Questions

1. **VNC performance in 2026:** How smooth is browser-via-VNC? Does it lag? Is mouse tracking responsive? Is scrolling fluid? This determines whether the UX is acceptable for document viewing and browsing.

2. **noVNC maturity:** Is noVNC the right WebSocket-to-VNC bridge? Are there better alternatives? What about Apache Guacamole (which provides both VNC and RDP in browser)?

3. **Container startup time:** How fast can we spin up a VNC-enabled container? If it takes 30 seconds, that is acceptable for a session. If it takes 5 minutes, it is not.

4. **Alternative streaming protocols:** Is VNC the best option, or should we look at RDP (via xrdp), SPICE, or browser-native solutions like WebRTC screen sharing from inside the container?

5. **Does Claude Code / Cursor run on Linux?** If yes, delivering a pre-configured AI coding environment via VNC is a compelling product.

6. **noVNC + sg-layout:** Can we embed the noVNC canvas inside an sg-layout panel alongside our own UI (vault browser, chat, controls)?

## Relationship to the Existing Stack

This is not a separate product. It is another layer on top of the existing infrastructure:

- The **Playwright container** (already exists) gains a visible desktop mode
- The **MITM proxy sidecar** (already exists) becomes directly browsable
- The **vault** (already exists) provides encrypted storage for session data
- The **session manager** (new) orchestrates container lifecycle
- The **AMI** (already being packaged) gains VNC as an optional mode

| 1 | VNC-enabled container starts with lightweight desktop + Chromium | Container launches, VNC server running, browser visible on desktop |

| 3 | MITM proxy UI accessible via VNC browser session | Open MITM web UI inside the VNC desktop, browse captured traffic |
| 4 | Document rendered inside container, viewable but not downloadable | Open vault document in VNC browser, user can read but not save locally |

| 7 | Session manager can start/stop VNC containers on demand | API call starts a session, returns VNC URL, session is isolated |
