# Briefing: Browser-Based Image and PDF Compression

*For implementation as an SG/Send tool*

## The Problem

Infographic slides generated by the SGraph API come back as PNGs at ~1400px wide, averaging 850KB each. When assembled into PDFs, a 107-slide deck hits 116MB — over LinkedIn's 100MB upload limit and slow to download from vaults.

Converting PNG→JPEG at quality 82 and resizing to 1200px drops each slide from ~850KB to ~113KB (87% reduction). A 107-slide PDF goes from 116MB to 14.8MB.

This compression should happen at two points:
1. **At generation time** — when infographic slides are created, store both PNG (archive) and JPEG (distribution)
2. **At export time** — when assembling PDFs or preparing content for LinkedIn/sharing

## The Technique: Canvas JPEG Encoding

Every browser has a built-in JPEG encoder via the Canvas API. No WASM, no external libraries needed for the core compression:

```javascript
// Load an image (PNG, any format)
const img = new Image();
img.src = sourceUrl; // or data URI, or blob URL

img.onload = () => {
  // Create canvas at target size
  const canvas = document.createElement('canvas');
  const maxWidth = 1200;
  const scale = Math.min(1, maxWidth / img.width);
  canvas.width = Math.round(img.width * scale);
  canvas.height = Math.round(img.height * scale);

  // Draw and encode as JPEG
  const ctx = canvas.getContext('2d');
  ctx.drawImage(img, 0, 0, canvas.width, canvas.height);

  // Quality: 0.0 to 1.0
  const jpegDataUrl = canvas.toDataURL('image/jpeg', 0.82);

  // Or as a Blob (more efficient for large files)
  canvas.toBlob(blob => {
    // blob is a JPEG Blob, ready to use
  }, 'image/jpeg', 0.82);
};
```

### Key Parameters

| Parameter | Recommended | Range | Effect |
|-----------|------------|-------|--------|
| Quality | 0.80-0.85 | 0.30-0.95 | Below 0.70 gets visibly muddy. Above 0.90 diminishing returns. |
| Max width | 1200px | 600-2000px | 1200px is sharp for slides. 800px for LinkedIn inline. |
| Format | JPEG | JPEG/WebP/PNG | JPEG for photos/infographics. WebP 30% smaller but less compatible. PNG only for text-heavy screenshots. |

### Quality vs Size (for a typical infographic slide)

| Quality | Avg size | Reduction from PNG | Visual quality |
|---------|----------|-------------------|----------------|
| 0.95 | ~350KB | 59% | Indistinguishable |
| 0.85 | ~150KB | 82% | Excellent |
| 0.80 | ~113KB | 87% | Very good |
| 0.70 | ~75KB | 91% | Good, slight softness |
| 0.50 | ~40KB | 95% | Noticeable artifacts |

## PDF Compression in the Browser

For compressing existing PDFs (render each page, re-encode, reassemble):

### Libraries Required

1. **pdf.js** (Mozilla) — renders PDF pages to Canvas
   - CDN: `https://cdnjs.cloudflare.com/ajax/libs/pdf.js/4.0.379/pdf.min.mjs`
   - Worker: `https://cdnjs.cloudflare.com/ajax/libs/pdf.js/4.0.379/pdf.worker.min.mjs`

2. **jsPDF** — creates new PDFs from images
   - CDN: `https://cdnjs.cloudflare.com/ajax/libs/jspdf/2.5.1/jspdf.umd.min.js`

### Pipeline

```
Input PDF
  → pdf.js renders each page to Canvas
    → Canvas.toDataURL('image/jpeg', quality) encodes as JPEG
      → jsPDF.addImage() adds to new PDF
        → Output compressed PDF as Blob
```

### Implementation

```javascript
import * as pdfjsLib from 'pdf.js';

async function compressPDF(arrayBuffer, { quality = 0.82, maxWidth = 1200 } = {}) {
  const pdf = await pdfjsLib.getDocument({ data: arrayBuffer }).promise;
  const numPages = pdf.numPages;

  // Get first page dimensions
  const firstPage = await pdf.getPage(1);
  const vp = firstPage.getViewport({ scale: 1 });
  const scale = Math.min(1, maxWidth / vp.width);
  const w = Math.round(vp.width * scale);
  const h = Math.round(vp.height * scale);

  // jsPDF page size in points (72 dpi)
  const ptW = w * 72 / 96;
  const ptH = h * 72 / 96;
  const doc = new jsPDF({ unit: 'pt', format: [ptW, ptH], compress: true });

  const canvas = document.createElement('canvas');
  const ctx = canvas.getContext('2d');

  for (let i = 1; i <= numPages; i++) {
    const page = await pdf.getPage(i);
    const viewport = page.getViewport({ scale });

    canvas.width = Math.round(viewport.width);
    canvas.height = Math.round(viewport.height);

    await page.render({ canvasContext: ctx, viewport }).promise;

    const jpegUrl = canvas.toDataURL('image/jpeg', quality);

    if (i > 1) doc.addPage([ptW, ptH]);
    doc.addImage(jpegUrl, 'JPEG', 0, 0, ptW, ptH);
  }

  return doc.output('blob');
}
```

### Memory Considerations

- Each page render creates a Canvas bitmap in memory (~4 bytes per pixel)
- A 1200×670 slide = ~3.2MB in memory per page
- Processing 100+ pages sequentially is fine (one Canvas reused)
- Do NOT render all pages in parallel — memory will spike

## Infographic Generation Pipeline Optimization

Currently the SGraph infographic generator returns base64 PNG data. The optimization should happen right after generation:

### Current Pipeline

```
Prompt → Gemini API → base64 PNG (~850KB) → save as PNG file
```

### Optimized Pipeline

```
Prompt → Gemini API → base64 PNG → decode → resize/compress → save as:
  - PNG (archive, full quality)
  - JPEG (distribution, 1200px, quality 82, ~113KB)
  - JPEG (LinkedIn, 800px, quality 70, ~50KB)
```

### Python Implementation (for Claude sessions)

```python
import base64
from PIL import Image
from io import BytesIO

def optimize_slide(b64_png, output_prefix):
    """Save optimized versions of a generated slide."""
    img_data = base64.b64decode(b64_png)
    img = Image.open(BytesIO(img_data))

    # 1. Archive PNG (original)
    img.save(f'{output_prefix}.png')

    # 2. Distribution JPEG (1200px, q82)
    dist = img.copy()
    if dist.width > 1200:
        ratio = 1200 / dist.width
        dist = dist.resize((1200, int(dist.height * ratio)), Image.LANCZOS)
    dist.save(f'{output_prefix}-dist.jpg', quality=82)

    # 3. LinkedIn JPEG (800px, q70)
    li = img.copy()
    if li.width > 800:
        ratio = 800 / li.width
        li = li.resize((800, int(li.height * ratio)), Image.LANCZOS)
    li.save(f'{output_prefix}-li.jpg', quality=70)
```

### JavaScript Implementation (for browser/tool)

```javascript
async function optimizeSlide(imageSrc) {
  const img = new Image();
  await new Promise((resolve, reject) => {
    img.onload = resolve;
    img.onerror = reject;
    img.src = imageSrc;
  });

  const variants = [
    { suffix: 'dist', maxWidth: 1200, quality: 0.82 },
    { suffix: 'li',   maxWidth: 800,  quality: 0.70 },
  ];

  const results = {};
  for (const v of variants) {
    const canvas = document.createElement('canvas');
    const scale = Math.min(1, v.maxWidth / img.width);
    canvas.width = Math.round(img.width * scale);
    canvas.height = Math.round(img.height * scale);
    canvas.getContext('2d').drawImage(img, 0, 0, canvas.width, canvas.height);

    results[v.suffix] = await new Promise(resolve =>
      canvas.toBlob(resolve, 'image/jpeg', v.quality)
    );
  }
  return results;
}
```

## Tool Integration Plan

### Option A: Standalone Compression Tool

A page on tools.sgraph.ai that accepts:
- Drag-and-drop PDFs → outputs compressed PDF
- Drag-and-drop PNGs → outputs optimized JPEGs
- Batch processing for multiple files
- Quality/size slider with live preview

### Option B: Integrated into Infographic Generator

After each slide is generated:
- Automatically create JPEG variants alongside the PNG
- Show file sizes in the UI (PNG: 850KB, JPEG: 113KB)
- Offer "Download optimized" alongside "Download original"
- When building deck PDFs, use JPEG variants automatically

### Option C: Vault-Level Optimization

When pushing files to a vault via sgit:
- Detect image files over a size threshold
- Offer to create optimized variants
- Store both versions, reference the optimized one in manifests
- Download optimization: serve JPEG to browsers, PNG for archival

### Recommended: Start with Option A, evolve to B

Option A is a standalone HTML file (already built as pdf-compressor.html). Ship it as a tool. Then integrate the JPEG generation into the infographic pipeline (Option B) so new content is optimized at creation time.

## WebP as an Alternative

WebP offers ~30% smaller files than JPEG at equivalent quality. Browser support is now universal (including Safari 16+). However:

- LinkedIn does not accept WebP in articles (base64 img tags)
- Some PDF viewers don't render WebP
- jsPDF doesn't support WebP natively

Recommendation: stick with JPEG for compatibility. Consider WebP for vault browsing and web display only.
