# Vault as the Collaboration Primitive: We Already Built It

**version** v0.24.55
**date** 15 Apr 2026
**from** Human (project lead)
**to** Developer (lead), Architect
**type** Dev brief

## The Realisation

With the latest vault UI implementation, we have effectively solved the collaboration workflow that the folder sharing modes (Modes 2, 3, 4 from the 05 Apr and 08 Apr briefs) were designed to address. The vault already does it, and does it better than the proposed server-side zip versioning approach.

We do not need the other modes. We just use the vault.

## What Changed

The vault UI implementation made two architectural choices that turned out to be decisive:

**1. Tree-only loading.** The browser vault does not download all files on open. It loads only the tree structure needed to render the file list. Individual files are fetched on demand. This took vault open time from approximately 6 seconds to approximately 1 second, and dramatically reduced server-side load (even though the client was already making parallel requests). Large files (images, PDFs, videos) are never fetched until explicitly accessed.

**2. Immutable file caching.** Many vault objects are immutable by design (single-write). The browser caches these aggressively. On subsequent visits, only the tree pointers need to be refreshed, not the file contents. This gives quasi-real-time access to the vault without re-downloading content.

Combined with the features that already shipped (browse-edit Phase 1, Push/Pull via the dual branch model, large file support via presigned S3 URLs), the vault now supports the full collaboration workflow: multiple users uploading files, making changes, and seeing each other's updates, all without conflicts.

## Why This Supersedes the Folder Versioning Approach

The 08 Apr arch brief (`v0.20.38__arch-brief__server-side-zip-folder-versioning.md`) proposed server-side per-file storage with mutable head pointers and compare-and-swap conflict detection as the "simple version without vault." The rationale was that a full vault was overkill for some collaboration scenarios.

That rationale no longer holds. The vault UI now provides:

- The same tree-based browsing (load tree first, files on demand)
- Better conflict handling (SGit's dual branch model with Push/Pull, designed for exactly this)
- Version history (every change is a commit)
- Encryption (zero-knowledge, client-side)
- All the browse components (gallery, page layout renderer, markdown rendering, present mode)

The vault is not overkill. It is the right primitive, and it is already built.

## What We Need to Finish

A small number of additions to make this the complete collaboration experience:

### 1. Instant vault creation on upload

When a user uploads a folder, offer a mode that creates a vault immediately and adds files incrementally. Instead of bundling everything into a zip first, create the vault, then add files one by one (or in batches) with individual commits. The user (or other collaborators) can start browsing immediately while the upload is still in progress.

### 2. Upload branches

For large uploads (many files), create a dedicated upload branch. Files are committed to the upload branch one by one, allowing the upload to proceed incrementally without blocking the main branch. Once complete, the upload branch merges into the main branch. This also means a failed upload does not corrupt the main tree.

### 3. Read-only vs read-write access controls

Ensure the read-only access path works cleanly (no spurious write attempts, which the 04/11 audit found and fixed). The read-write mode should be gated by vault key ownership. Read-only share tokens should give browse access to the latest snapshot without write capability.

### 4. Auto-refresh / change detection

Add a periodic check (or a lightweight poll) for vault updates. When another user pushes changes, the Pull button badge updates automatically. This gives the multi-user experience of seeing new content appear without manual refresh. The Refresh button already exists; this just automates the trigger.

## The Architecture Now

```
                   Vault (the core primitive)
                   /                          \
         Web UI (browser)                   SGit CLI
         |                                   |
         +-- Tree-only loading               +-- Full clone (downloads everything)
         +-- Immutable file caching           +-- Offline editing
         +-- Browse-edit (Phase 1)            +-- Branch, commit, push, pull
         +-- Push/Pull (Phase 2+3)            +-- Conflict resolution
         +-- Gallery, PLR, Present mode       +-- Script automation
         +-- Upload Files, New Folder
         +-- Large file support (presigned S3)
         +-- Refresh button
         |
         v
    Collaboration Mode:
    - Multiple users open same vault
    - Each gets a clone branch (HMAC-derived)
    - Commits land on clone branch
    - Push to advance named branch
    - Pull to get others' changes
    - No conflicts unless editing same file
    - SGit handles conflict resolution
```

## Why This Matters

The vault is now the universal primitive for both sharing and collaboration. The distinction the 08 Apr brief raised (Folder View vs Vault) simplifies: there is the vault (the live workspace) and the share token (a read-only snapshot). The vault handles everything from a single file share to a multi-user collaborative workspace with version control.

This also means the server-side zip folder versioning sprint (open decision #5 from the Librarian's master index) can be deprioritised or scoped down. The vault already provides the architecture that spec was trying to build from scratch.
