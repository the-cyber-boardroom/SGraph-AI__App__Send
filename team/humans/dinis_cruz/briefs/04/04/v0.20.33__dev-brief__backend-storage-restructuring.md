# Backend Storage Restructuring: Key-Specific Grouping for Transfers

**version** v0.20.33
**date** 04 Apr 2026
**from** Human (project lead)
**to** Architect (lead), Developer, DevOps
**type** Dev brief

## The Problem

The backend stores all transfers in a flat S3 structure. Every transfer ID sits at the same level. As the number of transfers grows, this creates problems:

- **Hard to identify who owns what.** All transfers are in one flat list. No way to tell which access token or user created which transfers without querying metadata.
- **Hard to clean up.** Want to delete all transfers for a specific user or access token? You have to scan the entire list.
- **Hard to migrate.** Want to move one user's data to a different server or bucket? You cannot isolate their files.
- **Hard to maintain.** A flat list of thousands (eventually millions) of transfer IDs becomes unmanageable for any operational task.

This is not a security problem. The encryption is fine. This is a logistics and organisation problem that gets worse with every new transfer.

## The Solution: Key-Derived Grouping

Add a grouping level derived from the access token and/or the decryption key. The grouping ID is deterministic (calculated from the key), so it does not need to be stored separately.

### Two Grouping Options

**Option A: Derived from the access token**

The access token determines who uploaded. Grouping by access token means: all transfers from the same user (same access token) are in the same folder.

```
s3://bucket/
  {access-token-derived-id}/
    {transfer-id-1}/
      encrypted-data...
    {transfer-id-2}/
      encrypted-data...
  {another-access-token-derived-id}/
    {transfer-id-3}/
      encrypted-data...
```

**Use case:** "Show me all transfers from this access token." "Delete everything uploaded by this user." "Move this user's data to another bucket."

**Option B: Derived from the decryption key**

The decryption key determines who can read. Grouping by decryption key means: all transfers that share the same decryption key are in the same folder.

```
s3://bucket/
  {decryption-key-derived-id}/
    {transfer-id-1}/
      encrypted-data...
```

**Use case:** "All files that this key can decrypt are in one place." Useful for vault-like grouping where a key represents a workspace.

### Recommendation: Full Hierarchy with Optional Levels

Use a multi-level hierarchy where each level is optional and the server calculates defaults when the caller does not provide a value. All transfers live under the `/transfers/` folder. Each grouping level is prefixed with a type indicator and all grouping levels sit at the same depth (sequential path segments, not nested):

```
s3://bucket/transfers/
  {server-provider-region-id}/
  by-id-from__user/{user-derived-id}/
  by-id-from__access-token/{access-token-derived-id}/
  by-id-from__decryption-key/{decryption-key-derived-id}/
  {transfer-id}/
    encrypted-data...
```

Full path example:

```
transfers/eu-west-2/by-id-from__user/usr_4a8b/by-id-from__access-token/at_7f3a/by-id-from__decryption-key/dk_9c2e/transfer-001/data.enc
```

The `by-id-from__*` segments are all peers. None nests inside another. Each is a type/value pair that adds one level of grouping to the path. The path reads left to right as a series of filters: region, then user, then access token, then decryption key, then the transfer itself.

### The Type Prefix Pattern

Each grouping segment is a pair: a static type folder followed by a derived ID folder. This makes the S3 structure self-documenting. Anyone browsing the bucket can immediately see how the data is organised:

| Type Prefix | What The Next Folder Contains |
|---|---|
| `by-id-from__user` | ID derived from username, customer ID, or user identifier |
| `by-id-from__access-token` | ID derived from the SG/Send access token |
| `by-id-from__decryption-key` | ID derived from the transfer decryption key |
| `by-id-from__public-key` | ID derived from a PKI public key (future, strongest identity) |

These type folders are literal, static folder names. The derived ID folder sits immediately after them at the same level. This means you can have the same derived ID under different type prefixes without collision, and the path itself documents the derivation method.

### Flexible Grouping: Not All Levels Required

The flat structure means levels can be included or omitted. A simple upload might produce:

```
transfers/by-id-from__access-token/at_7f3a9c/{transfer-id}/
```

A more detailed grouping:

```
transfers/eu-west-2/by-id-from__user/usr_4a8b/by-id-from__access-token/at_7f3a9c/{transfer-id}/
```

A full enterprise path with PKI:

```
transfers/eu-west-2/by-id-from__public-key/pk_abc123/by-id-from__access-token/at_7f3a9c/by-id-from__decryption-key/dk_def456/{transfer-id}/
```

The number of grouping segments depends on how much context the caller provides. The server fills in defaults for anything missing. The structure never nests deeper than one level per grouping type.

### Level 1: Server / Provider / Host / Region

The outermost grouping identifies the deployment context:

- Which server or region hosts this data
- Which provider (AWS, self-hosted, partner deployment)
- Which environment (production, dev, staging)

This enables multi-region deployment, data sovereignty (keep EU data in EU), and environment isolation. When moving an entire deployment to a new provider, this level is the boundary.

**Default:** If not provided by the caller, the server uses its own configured region/provider/host ID.

### Level 2: User / Customer / PKI Identity

Groups by the identity of the owner:

- Derived from a username or user ID
- Derived from a customer ID (for enterprise/B2B)
- Derived from a PKI public key (the strongest identity, ties to cryptographic identity)

This is the "whose data is this?" level. Essential for multi-tenant deployments, enterprise customers with their own namespace, and the future PKI model where every user has a key pair.

**Default:** If not provided by the caller, the server can derive from the access token (since the access token is already tied to a user/customer).

### Level 3: Access Token Derived

Groups by who uploaded. All transfers from the same access token are together.

**Default:** Always calculable from the access token provided with the upload request.

### Level 4: Decryption Key Derived

Groups by who can read. All transfers sharing the same decryption key are together.

**Default:** Always calculable from the decryption key provided during upload.

### Level 5: Transfer ID

The individual transfer. Unchanged from the current system.

### The Key Principle: Optional Levels with Server-Side Defaults

Every level above the transfer ID is optional. The caller can provide explicit values for any level, or the server calculates a default:

| Level | Caller Provides | Server Default |
|---|---|---|
| Server/Region | Explicit region or provider ID | Server's own configured region ID |
| User/Customer | Username, customer ID, or PKI public key | Derived from access token |
| Access Token | (Always present in the request) | Derived from the access token |
| Decryption Key | (Always present in the request) | Derived from the decryption key |

This means the simplest upload (just an access token and a file) still gets fully grouped. The server fills in every level. But an enterprise customer can provide their own customer ID, a specific region, and a PKI identity for explicit control.

### PKI Identity as a Grouping Level

When PKI is implemented, the user's public key becomes the strongest grouping identity. All transfers signed by the same key pair are naturally grouped. This aligns with the vault model where a vault key pair defines the access boundary.

```
transfers/eu-west-2/by-id-from__public-key/pk_abc123/by-id-from__access-token/at_789ghi/by-id-from__decryption-key/dk_jkl012/transfer-001/data.enc
```

### Backward Compatibility: Transfer ID Always Works

The current transfer-ID-based access remains fully functional. A transfer ID uniquely identifies a transfer regardless of where it sits in the hierarchy. The API resolves the transfer ID to its full path internally.

The worst case with the new hierarchy: someone could place a file in a group belonging to someone else (by providing a different user/customer ID). This is an organisational inconvenience, not a security breach: the file is still encrypted, and only someone with the decryption key can read it. The grouping is for logistics, not for access control. The encryption handles access control.

### Derivation Method

Use the same deterministic ID approach as the vault system:

```python
import hashlib

def derive_group_id(key_material: str, salt: str = "sg-group") -> str:
    return hashlib.sha256(f"{salt}:{key_material}".encode()).hexdigest()[:16]

def derive_server_id(region: str, provider: str) -> str:
    return derive_group_id(f"{provider}:{region}", salt="sg-server")

def derive_user_id(identity: str) -> str:
    return derive_group_id(identity, salt="sg-user")

def derive_access_id(access_token: str) -> str:
    return derive_group_id(access_token, salt="sg-access")

def derive_key_id(decryption_key: str) -> str:
    return derive_group_id(decryption_key, salt="sg-key")
```

Each level uses a different salt to prevent collisions between levels. All IDs are deterministic, truncated hashes. Not reversible. The server can always recalculate them.

## Migration

### For Existing Transfers

Existing transfers in the flat structure need migration:

1. For each existing transfer, calculate the group ID from its access token
2. Move the transfer into the grouped structure
3. Update any references (if transfer IDs are stored in metadata with paths)

This can be done as a background task. The API can support both the old flat path and the new grouped path during migration (check grouped first, fall back to flat).

### For New Transfers

New transfers go directly into the grouped structure. No code change needed on the client side (the client only knows the transfer ID, not the S3 path).

## What This Enables

| Capability | How Grouping Helps |
|---|---|
| Per-user cleanup | Delete the access-token-derived folder, all their transfers are gone |
| Per-user migration | Copy the access-token-derived folder to a new bucket/server |
| Usage tracking | Count objects per group ID = transfers per user |
| Cost attribution | S3 storage per group ID = storage cost per user |
| Vault alignment | The decryption-key-derived group naturally aligns with vault boundaries |
| Operational visibility | Browse S3 and see logical groupings instead of a flat sea of IDs |

## Security Considerations

- The group IDs are derived hashes, not the keys themselves. Seeing a group ID does not reveal the access token or decryption key.
- The grouping is an organisational convenience. The actual security (encryption, access control) remains unchanged.
- If someone has S3 access, they could already see transfer IDs. The grouping just adds structure. It does not add or remove any security property.
