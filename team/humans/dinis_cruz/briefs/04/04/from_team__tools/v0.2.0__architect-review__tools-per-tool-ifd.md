# Tools v0.2.0: Fractal IFD — Components, Tools, Sites as a DAG

**Version:** v0.2.0 (consolidation of tools/v0/v0.1/v0.1.0–v0.1.8)
**Date:** 1 April 2026
**Role:** Architect (Explorer team)
**Type:** Architecture brief + migration plan
**Status:** Pre-implementation — ready for review
**Follows:** IFD v1.1.0 (core), v1.2.1 (intro), v1.3.0 (surgical overrides)

---

## 1. The Problem

The tools site currently has **one IFD chain for everything** — 14 tools, the landing page, shared CSS, SEO files, and i18n all version together as a single monolith under `tools/v0/v0.1/`. When one tool updates, a new site version is created that carries every other tool along with it.

Additionally, with 14 tools already shipped and more coming, the flat listing on the landing page needs to evolve into **grouped sub-pages** — the 4 LLM tools as their own section with its own landing page, the utility tools as another, experimental tools as a third. This navigation structure is its own concern that changes independently of both the tool implementations and the site chrome.

## 2. Three Entity Types, One Rule

There are three types of entity, each with its own IFD versioning and manifest:

```
COMPONENT        a web component (JS + CSS + HTML)
                 own IFD, own manifest
                 registers a custom element

TOOL             a standalone page that composes entities
                 own IFD, own manifest
                 an HTML page with imports + layout

SITE             a navigable website that composes entities
                 own IFD, own manifest (site.json)
                 registry, navigation, grouping, sub-pages
```

```
sgraph_ai_tools__static/
  components/                ← components (already per-component IFD)
  tools/                     ← tools (per-tool IFD, new)
  sites/                     ← sites (per-site IFD, new)
```

### 2.1 The One Rule: No Circular Load-Time Imports

These are NOT strict layers. Any entity can depend on any other entity — a component can embed a tool, a tool can embed a site, a component can depend on another component. The only constraint:

**The dependency graph must be a DAG (directed acyclic graph).** No circular load-time imports. If A imports B, B cannot import A (directly or transitively).

This is enforced at the manifest level. Each manifest declares its dependencies. A verification script walks the graph and rejects cycles.

### 2.2 Common Patterns

The **typical** dependency flow is components → tools → sites. But the system supports any acyclic composition:

```
TYPICAL:
  site → tool → component → component

ALSO VALID:
  component → tool              (a test harness component that loads tools)
  component → site              (a site preview component)
  tool → site                   (an admin tool that manages sites)
  tool → tool                   (a comparison tool that loads two tools side-by-side)
  site → site                   (a meta-site that aggregates other sites)
```

### 2.3 Fractal Examples

**A component that tests tools:**

A `sg-tool-tester` component that takes a tool name, reads its manifest, loads the tool page in an iframe, and runs assertions against it. This component depends on tools (reads their manifests, renders their pages) — but tools don't depend on it. No cycle.

```json
{
    "name": "sg-tool-tester",
    "version": "0.1.0",
    "dependencies": {
        "tools": {
            "_comment": "Reads tool manifests at runtime, loads tool pages in iframes",
            "resolution": "runtime",
            "type": "tool"
        }
    }
}
```

**A component that visualises a site:**

A `sg-site-preview` component that renders a site's landing page, navigation structure, and tool registry as an interactive map. It reads `site.json`, renders the navigation tree, and shows tool cards with their component dependency counts. A developer uses this to understand the site's structure.

**A tool that compares tools:**

A "Tool Comparison" tool page that loads two tools side-by-side in iframes, lets the user send the same prompt to both, and compares the outputs. The tool's manifest declares both tools as dependencies:

```json
{
    "name": "tool-comparison",
    "version": "0.1.0",
    "dependencies": {
        "chat":           { "type": "tool", "version": "0.1.1" },
        "one-shot-chat":  { "type": "tool", "version": "0.1.0" }
    }
}
```

**A tool that embeds a mini-site:**

An "LLM Dashboard" tool that shows all LLM tools in a tabbed interface. It reads the site's `site.json`, filters for the LLM section, and loads each tool's page. Effectively, it's a tool that presents a subset of a site.

### 2.4 The Manifest is the Contract

The manifest system makes this work because every entity declares what it depends on and what it provides. The dependency `type` field distinguishes between component, tool, and site dependencies:

```json
"dependencies": {
    "sg-llm-events":  { "type": "component", "path": "/components/llm/...", "version": "0.1.0" },
    "chat":            { "type": "tool",      "path": "/tools/chat/...",     "version": "0.1.1" },
    "tools-sgraph-ai": { "type": "site",      "path": "/sites/tools-sgraph-ai/...", "version": "0.2.0" }
}
```

The verification script doesn't care about entity types — it just walks the dependency graph and checks for cycles.

### 2.5 Runtime vs Load-Time Dependencies

The no-cycles rule applies to **load-time imports** (ES module `import` statements, `<script>` tags). **Runtime references** (fetching a manifest via `fetch()`, loading a tool in an iframe, reading `site.json`) don't create import cycles because they're lazy and isolated:

| Dependency type | Creates import cycle risk? | Example |
|----------------|--------------------------|---------|
| `import '/components/...'` | Yes — load-time | A component importing another component |
| `<script src="/tools/...">`| Yes — load-time | A tool loading a component |
| `fetch('/tools/.../manifest.json')` | No — runtime | A component reading a tool's manifest |
| `<iframe src="/en-gb/chat/">` | No — isolated | A tool embedding another tool's page |
| `fetch('/sites/.../site.json')` | No — runtime | A component reading a site's registry |

The manifest captures both types. Load-time dependencies go in `components` (the current structure). Runtime/iframe dependencies go in a separate `references` or `dependencies` section with `"resolution": "runtime"`.

### 5.2 site.json Enables Grouped Sub-Pages

The `navigation.sections` array defines how tools are grouped. Each section can have its own landing page (`page` field) and URL path (`path` field).

The LLM section gets `"path": "llm/"` which means `tools.sgraph.ai/en-gb/llm/` is a landing page showing just the 4 LLM tools. The top-level landing page shows all sections with a few featured tools per section and a "See all →" link to the section page.

**Adding a new section later** (e.g. "Security Tools") is a site version bump:

```
sites/tools-sgraph-ai/v0/v0.2/
  v0.2.1/
    site.json                              ← adds "security" section
    pages/en-gb/security/index.html        ← new section landing page
```

No tools change. No components change. Just the site layer evolves.

## 6. How Entities Compose (DAG, not Layers)

### 6.1 Typical Dependency Flow

The common case is site → tool → component:

```
site.json
  │
  │  declares tools (name + version + ifd_base)
  │
  ├─→ tools/chat/ manifest.json
  │     │
  │     │  declares dependencies (components, resolution: load)
  │     │
  │     ├─→ components/llm/sg-llm-events/v0/v0.1/v0.1.0/
  │     ├─→ components/llm/sg-llm-request/v0/v0.1/v0.1.0/
  │     ├─→ components/llm/sg-llm-chat-history/v0/v0.1/v0.1.1/
  │     └─→ ...
  │
  ├─→ tools/one-shot-chat/ manifest.json
  │     ├─→ components/llm/sg-llm-reality/v0/v0.1/v0.1.0/
  │     └─→ ...
  │
  └─→ tools/ssh-keygen/ manifest.json
        ├─→ components/key-generator/v1/v1.0/v1.0.0/
        └─→ ...
```

### 6.2 Fractal Dependency Flow

But the DAG allows any direction as long as there are no cycles:

```
components/llm/sg-tool-tester/          ← a COMPONENT that references TOOLS
  manifest.json
    dependencies:
      chat:           { type: "tool", resolution: "runtime" }   ← loads in iframe
      one-shot-chat:  { type: "tool", resolution: "runtime" }
      sg-llm-events:  { type: "component", resolution: "load" }  ← normal import

tools/tool-comparison/                   ← a TOOL that references other TOOLS
  manifest.json
    dependencies:
      chat:           { type: "tool", resolution: "runtime" }
      one-shot-chat:  { type: "tool", resolution: "runtime" }
      site-header:    { type: "component", resolution: "load" }

tools/site-admin/                        ← a TOOL that references a SITE
  manifest.json
    dependencies:
      tools-sgraph-ai: { type: "site", resolution: "runtime" }  ← reads site.json
      site-header:      { type: "component", resolution: "load" }
```

The verification script traverses all `resolution: "load"` edges and rejects cycles. `resolution: "runtime"` edges are recorded for documentation and offline-copy but don't participate in cycle detection.

### 6.3 Independence at Each Entity

| Entity type | What triggers a version bump | What does NOT trigger a version bump |
|-------------|-------|-------|
| **Component** | Bug fix, new feature, API change | A tool updating its manifest to use this component |
| **Tool** | New dependency, layout change, feature addition | A site adding this tool to a new section |
| **Site** | New tool added, navigation restructured, section sub-page added, shared CSS updated | A tool's internal dependency bumps (unless the tool version also changes in the registry) |

### 6.3 Example: Chat Tool Gets a New Feature

1. `sg-llm-chat-history` gets a v0.1.2 (component layer — new feature)
2. `tools/chat/` gets a v0.1.2 (tool layer — bumps the component import)
3. `sites/tools-sgraph-ai/` gets a v0.2.x (site layer — updates chat version in site.json)

Each layer versions independently. Steps 1, 2, and 3 could be separate commits or the same commit — the structure doesn't dictate the workflow.

### 6.4 Example: New Tool Added

1. Create `tools/markdown-preview/v0/v0.1/v0.1.0/` with manifest + index.html
2. `sites/tools-sgraph-ai/` gets a v0.2.x — adds tool to site.json + navigation section
3. Done. No other tool touched. Landing page renders it from site.json.

### 6.5 Example: LLM Tools Get Their Own Sub-Page

1. `sites/tools-sgraph-ai/` gets a v0.2.x — adds `"path": "llm/"` to the LLM section in site.json + creates `pages/en-gb/llm/index.html`

## 7. IFD Minor Versions Across Entity Types

Each entity type has different file types, and each file type has its own minor-version mechanic. The v1.3.0 IFD guide (`v1.3.0__ifd__surgical-overrides-version-stamped-filenames.md`) defines the version-stamped filename convention for JS/CSS. This section extends that to cover JSON manifests and HTML pages used in tools and sites.

### 7.1 The Rule by File Type

| File type | Minor version mechanic | Why |
|-----------|----------------------|-----|
| **JS (code)** | **Surgical override** — version-stamped filename (`sg-tree--v0.1.1.js`). Both base and override coexist in `_latest/`. Prototype patching at load time. | Code can be patched. The base class is still needed. |
| **CSS** | **Surgical override** — version-stamped filename (`sg-tree--v0.1.1.css`). Later CSS loaded after base, specificity wins. | CSS cascades. Both files needed. |
| **HTML (pages)** | **Full replacement** — same filename. Later version overwrites base in `_latest/`. | An HTML page is a complete composition root. Can't "patch" a page. |
| **JSON (manifest.json, site.json)** | **Full replacement** — same filename. Later version overwrites base in `_latest/`. | JSON is data, not code. Each version needs the complete state. |
| **New files** | **Normal IFD** — unique name, no stamp needed. Appears in `_latest/` without collision. | Nothing to collide with. |

### 7.2 Component Minor Versions (already covered by v1.3.0)

```
components/llm/sg-llm-chat-history/
  v0/v0.1/
    v0.1.0/
      sg-llm-chat-history.js          ← base (canonical name, ~400 lines)
      manifest.json                    ← full manifest for v0.1.0

    v0.1.1/
      sg-llm-chat-history.js          ← COMPLETE REWRITE (overwrites base in _latest/)
      manifest.json                    ← full manifest for v0.1.1

After _latest/:
  sg-llm-chat-history.js              ← from v0.1.1 (overwrites v0.1.0 — complete rewrite)
  manifest.json                        ← from v0.1.1 (overwrites v0.1.0)
```

Note: `sg-llm-chat-history` v0.1.1 is a complete rewrite, not a surgical override. It uses the same canonical filename so it overwrites the base in `_latest/`. This is correct per v1.3.0: "Complete rewrite of a component → Version-stamped rewrite" OR same-name replacement.

If it were a **surgical patch** instead (e.g. just fixing the scroll logic):

```
    v0.1.1/
      sg-llm-chat-history--v0.1.1.js  ← surgical override (version-stamped, ~40 lines)
      manifest.json                    ← full manifest (overwrites — JSON always full replacement)

After _latest/:
  sg-llm-chat-history.js              ← from v0.1.0 (PRESERVED — no collision)
  sg-llm-chat-history--v0.1.1.js      ← from v0.1.1 (unique filename — coexists)
  manifest.json                        ← from v0.1.1 (overwrites v0.1.0)
```

The consumer (tool page) loads both:
```html
<script type="module" src="/components/llm/sg-llm-chat-history/sg-llm-chat-history.js"></script>
<script type="module" src="/components/llm/sg-llm-chat-history/sg-llm-chat-history--v0.1.1.js"></script>
```

### 7.3 Tool Minor Versions

A tool's files are mostly HTML pages (full replacement) and a manifest (full replacement). But a tool CAN contain its own JS files that use surgical overrides.

**Simple case — tool page update (most common):**

```
tools/chat/
  v0/v0.1/
    v0.1.0/
      manifest.json                    ← complete manifest
      en-gb/index.html                 ← complete tool page

    v0.1.1/
      manifest.json                    ← complete manifest (updated component versions)
      en-gb/index.html                 ← complete tool page (updated imports)

After _latest/:
  manifest.json                        ← from v0.1.1 (overwrites v0.1.0)
  en-gb/index.html                     ← from v0.1.1 (overwrites v0.1.0)
```

**Complex case — tool with its own JS (vault-browser):**

```
tools/vault-browser/
  v1/v1.0/
    v1.0.0/
      manifest.json                    ← complete
      en-gb/index.html                 ← complete
      en-gb/vault-manager.js           ← base (canonical name)
      en-gb/vault-file-list.js         ← base
      en-gb/vault-file-preview.js      ← base

    v1.0.1/
      manifest.json                    ← complete (full replacement)
      en-gb/vault-manager--v1.0.1.js   ← SURGICAL OVERRIDE (version-stamped)
      en-gb/vault-graph-commits.js     ← NEW FILE (unique name, no stamp needed)

After _latest/:
  manifest.json                        ← from v1.0.1
  en-gb/index.html                     ← from v1.0.0 (not overwritten — v1.0.1 didn't include it)
  en-gb/vault-manager.js               ← from v1.0.0 (PRESERVED)
  en-gb/vault-manager--v1.0.1.js       ← from v1.0.1 (coexists)
  en-gb/vault-file-list.js             ← from v1.0.0
  en-gb/vault-file-preview.js          ← from v1.0.0
  en-gb/vault-graph-commits.js         ← from v1.0.1 (new file)
```

The tool's `index.html` (from v1.0.0, still in `_latest/`) loads the scripts. But it needs to know about the v1.0.1 override. Two approaches:

**Approach A: Tool page lists overrides explicitly.** v1.0.1 includes an updated `en-gb/index.html` that adds the override script tag. Simple, explicit. This means `en-gb/index.html` is in BOTH versions (full replacement).

**Approach B: Tool page reads manifest at runtime.** The page reads `manifest.json`, finds the version-stamped overrides, and loads them dynamically. More automated, but adds complexity.

**Recommendation: Approach A.** Keep it explicit. The tool page is the composition root — it decides what to load, just like the v1.3.0 guide says ("HTML page is the composition root"). If a tool version needs a new script, the tool page is updated to reference it. This means most tool minor versions include both `manifest.json` AND `en-gb/index.html`.

### 7.4 Site Minor Versions

The site's `site.json` is always a full replacement. Pages are always full replacement. Shared CSS/JS assets CAN use surgical overrides.

**Adding a new tool to the site:**

```
sites/tools-sgraph-ai/
  v0/v0.2/
    v0.2.0/
      site.json                        ← 14 tools, 3 sections
      _common/css/tools.css
      pages/en-gb/index.html
      pages/en-gb/llm/index.html

    v0.2.1/
      site.json                        ← 15 tools, 3 sections (full replacement — adds markdown-preview)

After _latest/:
  site.json                            ← from v0.2.1 (overwrites v0.2.0)
  _common/css/tools.css                ← from v0.2.0 (preserved)
  pages/en-gb/index.html               ← from v0.2.0 (preserved — reads site.json dynamically)
  pages/en-gb/llm/index.html           ← from v0.2.0 (preserved)
```

Note: the landing page does NOT need updating when a tool is added — it reads `site.json` at runtime. Only `site.json` changes.

**Adding a new section sub-page:**

```
    v0.2.2/
      site.json                        ← adds security section with path (full replacement)
      pages/en-gb/security/index.html  ← new file (unique path, no collision)

After _latest/:
  site.json                            ← from v0.2.2
  pages/en-gb/security/index.html      ← from v0.2.2 (new)
  pages/en-gb/index.html               ← from v0.2.0 (preserved)
  pages/en-gb/llm/index.html           ← from v0.2.0 (preserved)
```

**Surgical CSS update to shared assets:**

```
    v0.2.3/
      _common/css/tools--v0.2.3.css    ← SURGICAL CSS OVERRIDE (version-stamped)

After _latest/:
  _common/css/tools.css                ← from v0.2.0 (PRESERVED)
  _common/css/tools--v0.2.3.css        ← from v0.2.3 (coexists)
  site.json                            ← from v0.2.2
  ...
```

The landing page loads both:
```html
<link rel="stylesheet" href="../_common/css/tools.css">
<link rel="stylesheet" href="../_common/css/tools--v0.2.3.css">
```

Or — since the landing page is a full file in v0.2.0 and hasn't been updated since — a new landing page version is needed to add the CSS link. Pragmatically, if you're changing CSS, you bump the landing page too.

### 7.5 Summary: What Goes Where

| Entity | manifest/site.json | HTML pages | JS files | CSS files |
|--------|-------------------|------------|----------|-----------|
| **Component** | Full replacement | N/A | Surgical (version-stamped) OR full rewrite (same name) | Surgical (version-stamped) |
| **Tool** | Full replacement | Full replacement | Surgical (version-stamped) for tool-specific JS | N/A (inline in HTML or via components) |
| **Site** | Full replacement | Full replacement | Normal IFD | Surgical (version-stamped) for `_common/` CSS |

The pattern is consistent: **data files (JSON, HTML) are always full replacement. Code files (JS, CSS) can be surgical or full replacement per v1.3.0 rules.**

### 7.6 Manifest Always Tells the Truth

Regardless of whether a minor version uses surgical overrides or full replacement for its code files, the `manifest.json` in that version is **always the complete, current manifest** — not a delta.

This is critical for the offline copy and verification scripts: they read the latest manifest and get the full picture. They don't need to merge manifests from multiple versions.

```
v0.1.0/manifest.json    → { dependencies: { "sg-llm-events": "0.1.0", "sg-llm-request": "0.1.0" } }
v0.1.1/manifest.json    → { dependencies: { "sg-llm-events": "0.1.0", "sg-llm-request": "0.1.0", "sg-llm-stats": "0.1.0" } }
                                                                          ← added sg-llm-stats
```

After `_latest/`: `manifest.json` is from v0.1.1 — complete, self-describing, no merging needed.

## 8. Build System

### 7.1 `build_tools_site.py`

New script that composes the three layers into a deployable site:

```python
def build_site(site_name):
    # 1. Build site layer _latest
    build_ifd_latest(f'sites/{site_name}/v0/v0.2')
    site_json = load(f'sites/{site_name}/v0/v0.2/_latest/site.json')

    # 2. Start with site shell (shared assets, pages, 404, etc.)
    site_output = copy(f'sites/{site_name}/v0/v0.2/_latest/')

    # 3. For each tool in site.json, build tool _latest and copy into site
    for tool_name, tool_info in site_json['tools'].items():
        build_ifd_latest(tool_info['ifd_base'])
        tool_latest = f'{tool_info["ifd_base"]}/_latest/'

        # Read tool manifest for verification
        tool_manifest = load(f'{tool_latest}/manifest.json')

        # Copy tool files into site at the tool's URL path
        for locale_dir in glob(f'{tool_latest}/en-gb/'):
            copy(locale_dir, f'{site_output}/en-gb/{tool_name}/')

        # Verify component paths exist
        for comp_name, comp_info in tool_manifest['components'].items():
            assert exists(comp_info['path']), f'{tool_name} needs {comp_name} at {comp_info["path"]}'

    # 4. Build section landing pages (from site.json navigation)
    for section in site_json['navigation']['sections']:
        if section.get('path') and section.get('page'):
            # Section page already in site _latest (from step 1)
            pass  # it was copied with the site shell

    # 5. Write _site/ output
    write(site_output, f'sites/{site_name}/_site/')
```

### 7.2 Verification Script

`verify_deps.py` — walks the full dependency DAG, verifies all paths exist, detects cycles in load-time dependencies:

```python
def verify(site_name):
    site_json = load(f'sites/{site_name}/.../site.json')
    load_graph = {}  # adjacency list for cycle detection

    for tool_name, tool_info in site_json['tools'].items():
        manifest = load(f'{tool_info["ifd_base"]}/.../manifest.json')

        for dep_name, dep_info in manifest.get('dependencies', {}).items():
            # Verify path exists
            if not exists(dep_info['path']):
                error(f'{tool_name} → {dep_name} at {dep_info["path"]} NOT FOUND')

            # Build load-time dependency graph (for cycle detection)
            if dep_info.get('resolution', 'load') == 'load':
                load_graph.setdefault(tool_name, []).append(dep_name)

            # Recursively check dependencies of dependencies
            dep_manifest = load(f'{dep_info["path"]}/manifest.json')
            for sub_dep, sub_info in dep_manifest.get('dependencies', {}).items():
                if sub_info.get('resolution', 'load') == 'load':
                    load_graph.setdefault(dep_name, []).append(sub_dep)

        # Check tool version matches site.json
        if manifest['version'] != tool_info['version']:
            warn(f'{tool_name}: site.json says {tool_info["version"]}, manifest says {manifest["version"]}')

    # Detect cycles in load-time graph
    if has_cycle(load_graph):
        error('CIRCULAR LOAD-TIME DEPENDENCY DETECTED')
        print_cycle(load_graph)
```

### 7.3 Offline Entity Copier

`copy_offline.py` — creates a self-contained directory for any entity (component, tool, or site) by walking its dependency manifest recursively:

```python
def copy_offline(entity_path):
    manifest = load(f'{entity_path}/manifest.json')
    output = f'offline/{manifest["name"]}/'

    # Copy this entity's files
    for f in manifest['files']:
        copy(f'{entity_path}/{f}', f'{output}/{f}')

    # Recursively copy all dependencies
    for dep_name, dep_info in manifest.get('dependencies', {}).items():
        dep_manifest = load(f'{dep_info["path"]}/manifest.json')
        for f in dep_manifest.get('files', []):
            copy(f'{dep_info["path"]}/{f}', f'{output}/deps/{dep_name}/{f}')

        # If a dependency has its own dependencies, recurse
        if dep_manifest.get('dependencies'):
            copy_offline_deps(dep_info['path'], output)

    # Rewrite import paths to relative
    rewrite_imports(output)
```

This works for any entity type — copy a component offline (gets its sub-component dependencies), copy a tool offline (gets its component dependencies), copy a site offline (gets its tools and their components). The fractal structure means one script handles all cases.

## 12. Open Questions

| # | Question | Recommendation |
|---|----------|----------------|
| 1 | Should section sub-pages be their own IFD entities or just files in the site? | Files in the site entity. A section page is thin (reads site.json, renders tools from one section). It doesn't need its own versioning. |
| 2 | Should tool manifests include the tool's `title`, `description`, `icon` — duplicating what's in site.json? | Yes. The tool manifest describes the tool standalone. The site.json can override these for site-specific presentation. Source of truth for the tool's identity is its own manifest. |
| 3 | Should `build_ifd_latest.py` be extended or should `build_tools_site.py` be a new script? | New script. `build_ifd_latest.py` is a clean single-purpose tool (layer one IFD chain). `build_tools_site.py` orchestrates multiple IFD chains. Different concerns. |
| 4 | URL structure for section sub-pages: `/en-gb/llm/` or `/en-gb/tools/llm/`? | `/en-gb/llm/` — keep it short. The domain is already `tools.sgraph.ai`. Individual tools stay at `/en-gb/chat/`, `/en-gb/ssh-keygen/` etc. |
| 5 | Should the top-level landing page fetch site.json or should the build system pre-render it? | Both. Runtime fetch for the dynamic experience. Build step can pre-render for SEO/noscript. |
| 6 | Can a tool appear in multiple sections? | Yes — site.json's `navigation.sections[].tools` is just an array of tool IDs. A tool could appear in "LLM Tools" and "Featured". The tool itself doesn't know or care. |
| 7 | Name: `sites/` vs `websites/` vs something else? | `sites/`. Short, maps to domain names, extensible to `sites/send-sgraph-ai/` later. |
| 8 | Should `resolution: "runtime"` dependencies be verified differently from `resolution: "load"`? | Yes. Load-time deps must exist at build time (hard failure). Runtime deps should exist but could be optional (warning, not failure) — the entity might work in degraded mode without them. |
| 9 | Do we need a fourth entity type, or are component/tool/site sufficient? | Three types cover the current needs. The fractal model means new patterns (test harness, dashboard, admin panel) are just components or tools that happen to reference other entities — not a new entity type. If a pattern emerges that doesn't fit any of the three, we can add a type then. |
| 10 | Should the verification script generate a visual dependency graph (e.g. Mermaid)? | Yes — and it's a great dog-fooding opportunity: use the infographic generator or a new `sg-dep-graph` component to render the DAG. The verification script outputs a `deps.mermaid` file, a tool renders it. |
| 11 | Can a component's test page be a tool? | Yes, and this is already happening informally. Each component's test page (`tests/sg-llm/reality.test.html`) loads the component + mock data + assertions — that's a tool in everything but name. Formalising test pages as tools (with manifests) means they get the same IFD, offline copy, and dependency tracking benefits. Defer this to v0.3.0. |
