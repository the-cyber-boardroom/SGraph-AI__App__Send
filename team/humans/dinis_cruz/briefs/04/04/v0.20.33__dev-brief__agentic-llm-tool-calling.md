## Context

Models like Gemma, Mistral, and others now support structured tool calling and JSON output modes. This is a different paradigm from the standard "send text, receive text" chat. The LLM receives a set of tool definitions, decides which tool to call, returns structured JSON with the tool name and arguments, and expects the tool result back as structured JSON before continuing.

We need a set of specialised components to support this workflow, including the ability to execute tool code (JavaScript and Python) in a sandboxed browser environment.

## The Agentic Loop

```
User provides a task
  → LLM receives: task + available tool definitions (JSON schema)
  → LLM responds: tool_call { name: "search_files", args: { query: "*.md" } }
  → Browser executes the tool in a sandbox
  → Tool returns result as JSON
  → LLM receives the result
  → LLM either calls another tool or produces a final answer
  → Repeat until done
```

This loop runs entirely in the browser. The LLM is called via API (OpenRouter, Ollama, direct). The tool execution happens locally. No server-side code execution needed.

## Component 1: sg-tool-definition

A component for defining, editing, and managing tool definitions (the JSON schema that tells the LLM what tools are available).

```json
{
  "name": "search_vault",
  "description": "Search for files in the current vault matching a pattern",
  "parameters": {
    "type": "object",
    "properties": {
      "pattern": { "type": "string", "description": "File name pattern (glob)" },
      "content_search": { "type": "string", "description": "Search within file contents" }
    },
    "required": ["pattern"]
  }
}
```

The component provides:

- Visual editor for tool definitions (name, description, parameters with types)
- JSON schema validation (ensure the definition is valid before sending to the LLM)
- Tool library: save and reuse tool definitions across sessions
- Import/export: load tool definitions from a file or vault
- Template library: pre-built tool definitions for common operations (file search, data analysis, web fetch, vault operations)

## Component 2: sg-json-sender

A component for constructing and sending structured JSON to the LLM. Not free-text chat. Structured data exchange.

Features:

- JSON editor with syntax highlighting and validation
- Schema-aware input: if the expected JSON shape is known, provide form fields instead of raw JSON editing
- Template support: pre-fill from saved templates
- History: previous JSON payloads sent (stored in VFS)
- Preview: show exactly what will be sent to the LLM (system prompt + tool definitions + conversation history + current message)
- Token count: show the size of the payload before sending

## Component 3: sg-json-receiver

A component for receiving, parsing, and displaying structured JSON responses from the LLM.

Features:

- Auto-detect response type: text response, tool_call, or structured JSON output
- Tool call visualisation: when the LLM returns a tool_call, display the tool name, arguments, and a "Run" button
- JSON tree viewer: collapsible, searchable view of the response JSON
- Diff view: compare the current response to a previous one
- Copy/export: copy the JSON or save to VFS
- Schema validation: validate the response against an expected schema (did the LLM return what we asked for?)

## Component 4: sg-tool-runner

The execution environment. When the LLM returns a tool_call, this component executes the tool code and returns the result.

### Tool Registration

Tools are registered with their implementation:

```javascript
toolRunner.register('search_vault', async (args) => {
  const files = await vfs.listFolder('/');
  return files.filter(f => f.name.match(args.pattern));
});

toolRunner.register('read_file', async (args) => {
  return await vfs.readFile(args.path);
});

toolRunner.register('create_infographic', async (args) => {
  // Calls the infographic tool's API internally
  return await infographicEngine.generate(args.prompt, args.model);
});
```

### Built-In Tools

Pre-register tools that leverage existing SG/Send capabilities:

| Tool | What It Does | Uses |
|---|---|---|
| `search_vault` | Find files by name or content | VFS |
| `read_file` | Read a file from the vault | VFS |
| `write_file` | Write a file to the vault | VFS |
| `list_folder` | List folder contents | VFS |
| `create_infographic` | Generate an infographic | Infographic engine |
| `analyse_document` | Summarise or analyse a document | LLM (nested call) |
| `search_web` | Search the web (if online) | Fetch API |
| `run_code` | Execute JavaScript or Python in sandbox | Sandbox (see below) |

## Component 5: sg-sandbox (Sandboxed Code Execution)

The most critical component. Allows the LLM to write and execute code in the browser without security risk.

### JavaScript Sandbox

Use an iframe with `sandbox` attribute (no same-origin, no scripts from parent, no top navigation) combined with a Web Worker:

```javascript
// The sandbox creates an isolated iframe
const sandbox = new SGSandbox({ language: 'javascript' });

// Execute code from the LLM
const result = await sandbox.execute(`
  const data = [3, 1, 4, 1, 5, 9, 2, 6];
  const sorted = data.sort((a, b) => a - b);
  const mean = data.reduce((a, b) => a + b) / data.length;
  return { sorted, mean, count: data.length };
`);
// result: { sorted: [1, 1, 2, 3, 4, 5, 6, 9], mean: 3.875, count: 8 }
```

Security properties:

- Runs in a sandboxed iframe (no access to parent DOM, cookies, localStorage)
- Communication only via postMessage (structured clone, no object references)
- Timeout enforcement (kill execution after N seconds)
- Memory limits (terminate if exceeding threshold)
- No network access from within the sandbox (unless explicitly provided via a tool)
- No access to the VFS directly (results are returned to the parent, which decides what to save)

### Python Sandbox (Pyodide)

Use Pyodide (Python compiled to WebAssembly) for Python execution:

```javascript
const sandbox = new SGSandbox({ language: 'python' });

const result = await sandbox.execute(`
import json
data = [3, 1, 4, 1, 5, 9, 2, 6]
result = {
    "sorted": sorted(data),
    "mean": sum(data) / len(data),
    "count": len(data)
}
json.dumps(result)
`);
```

Pyodide runs in a Web Worker. Same security properties as the JavaScript sandbox. The WASM binary is managed by the sg-wasm component (cached in IndexedDB, works offline after first download).

### Sandbox Capabilities (Opt-In)

The sandbox starts with zero capabilities. Tools can opt in to specific capabilities:

| Capability | What It Allows | Risk Level |
|---|---|---|
| `compute` | Pure computation, no I/O | Low |
| `read_input` | Read data passed in by the caller | Low |
| `return_output` | Return structured results | Low |
| `import_libs` | Import allowed libraries (numpy, pandas in Python; lodash in JS) | Medium |
| `network` | Make HTTP requests (via proxy) | High (opt-in only) |
| `filesystem` | Read/write to a scoped VFS area | High (opt-in only) |

Default: `compute` + `read_input` + `return_output`. The LLM can do calculations and return results. Nothing else unless explicitly enabled.

## Component 6: sg-agentic-loop

The orchestrator that ties everything together. Manages the full agentic cycle:

1. Receives user task
2. Assembles the prompt (system prompt + tool definitions + context)
3. Sends to LLM
4. Parses response (text or tool_call)
5. If tool_call: executes via sg-tool-runner, captures result, sends back to LLM
6. Repeats until LLM produces a final text response (or max iterations reached)
7. Displays the full trace (every tool call, every result, every LLM response)

### Safety Controls

- **Max iterations:** Stop after N tool calls (default: 10, configurable)
- **Cost budget:** Stop if estimated token cost exceeds a threshold
- **Human-in-the-loop:** Optional approval step before each tool execution ("The LLM wants to call `write_file`. Allow?")
- **Audit log:** Every step recorded (saved to VFS for review)

### Trace Visualisation

Show the entire agentic loop as a timeline:

```
[User] "Analyse the files in my vault and create a summary"
  → [LLM] tool_call: list_folder({ path: "/" })
  → [Tool] returned: ["report.pdf", "data.csv", "notes.md"]
  → [LLM] tool_call: read_file({ path: "/data.csv" })
  → [Tool] returned: "name,value\nA,10\nB,20..."
  → [LLM] tool_call: run_code({ language: "python", code: "import pandas..." })
  → [Sandbox] returned: { mean: 15, rows: 2 }
  → [LLM] "Based on the analysis, your vault contains..."
```

Each step is clickable (see the full JSON payload). The trace is saveable to the VFS as a bundle.

## Gemma 4: The Primary Target Model

Gemma 4 was released on 2 April 2026 under Apache 2.0 (a significant change from Gemma 3's restrictive licence). It is the most capable open model family available and has native, trained-in function calling support. Our components should be built Gemma-4-first.

### Gemma 4 Model Family

| Model | Parameters | Active Params | Context | Best For |
|---|---|---|---|---|
| gemma-4-E2B-it | ~2B effective | ~2B | 128K | On-device, mobile, Raspberry Pi, browser (WASM potential) |
| gemma-4-E4B-it | ~4B effective | ~4B | 128K | Edge devices, laptops, local Ollama |
| gemma-4-26B-A4B-it | 26B total (MoE) | 3.8B active | 256K | Fast inference (low latency), good balance |
| gemma-4-31B-it | 31B dense | 31B | 256K | Maximum quality, fine-tuning base, offline code assistant |

The 31B model is #3 on the Arena AI text leaderboard, outcompeting models 20x its size. The 26B MoE activates only 3.8B parameters per inference, making it exceptionally fast.

### Gemma 4 Tool Calling Protocol

Gemma 4 uses dedicated special tokens for function calling (unlike earlier Gemma models which relied on prompt engineering):

**Tool Declaration (in the prompt):**

```
<|tool>declaration:get_current_weather{
  description:<|"|>Gets the current weather for a location.<|"|>
  parameters:{
    location:{type:<|"|>string<|"|>}
  }
}<tool|>
```

**Tool Call (model output):**

```
<|tool_call>call:get_current_weather{location:<|"|>Tokyo, JP<|"|>}<tool_call|>
```

**Tool Response (fed back to model):**

```
<|tool_response>response:get_current_weather{temperature:15,weather:<|"|>sunny<|"|>}<tool_response|>
```

The special tokens (`<|tool_call>`, `<tool_call|>`, `<|tool_response>`, etc.) are trained into the model. This is not prompt engineering. The model was trained to produce and consume these tokens natively, making tool calling significantly more reliable than with older models.

### Two Ways to Define Tools for Gemma 4

**1. JSON Schema (recommended for our components):**

```json
{
  "type": "function",
  "function": {
    "name": "get_current_weather",
    "description": "Gets the current weather for a given location.",
    "parameters": {
      "type": "object",
      "properties": {
        "location": {
          "type": "string",
          "description": "The city name, e.g. San Francisco"
        }
      },
      "required": ["location"]
    }
  }
}
```

Pass this via the `tools` argument in the chat template. The Hugging Face `apply_chat_template()` function handles converting JSON schema to Gemma 4's internal token format.

**2. Raw Python Functions:**

Pass actual Python functions with type hints and Google-style docstrings. The system auto-generates the JSON schema from the function signature. Useful for rapid prototyping but less portable than JSON schema.

### Gemma 4 Chat History with Tool Calls

The complete conversation history format when tool calls are involved:

```json
[
  { "role": "system", "content": "You are a helpful assistant." },
  { "role": "user", "content": "What's the temperature in London?" },
  {
    "role": "assistant",
    "tool_calls": [
      { "function": { "name": "get_current_weather", "arguments": { "location": "London" } } }
    ],
    "tool_responses": [
      { "name": "get_current_weather", "response": { "temperature": 15, "weather": "sunny" } }
    ],
    "content": "The temperature in London is 15 degrees and it is sunny."
  }
]
```

The `tool_calls` and `tool_responses` are part of the assistant message. Parse `tool_calls` from the model output, execute the functions, and append the `tool_responses` before sending back for the final response.

### Gemma 4 Structured JSON Output (No Tool Calling)

For cases where we just need structured JSON back (not a tool call), Gemma 4 supports constrained decoding via JSON schema:

```json
{
  "response_format": {
    "type": "json_schema",
    "json_schema": {
      "name": "city-info",
      "schema": {
        "type": "object",
        "properties": {
          "city": { "type": "string" },
          "population": { "type": "integer" },
          "landmarks": { "type": "array", "items": { "type": "string" } }
        },
        "required": ["city", "population", "landmarks"]
      }
    }
  }
}
```

The model is constrained to produce valid JSON matching the schema. This is useful for the sg-json-receiver component: when the expected output shape is known, validate against the schema automatically.

### Gemma 4 + Thinking Mode + Tool Calling

Gemma 4 combines thinking (step-by-step reasoning) with tool calling. When both are enabled:

- The model thinks first (reasoning about which tool to call and why)
- Then produces the tool call
- Between tool calls in the same turn, thoughts must NOT be stripped (the model needs them for multi-step planning)
- Between separate conversation turns, thoughts should be stripped (to prevent context pollution)

This is important for our sg-agentic-loop: manage the thinking tokens correctly. Strip between turns, preserve within a tool-calling sequence.

### Gemma 4 Availability

| Platform | Models Available | Notes |
|---|---|---|
| Ollama | All four sizes | Primary local target. `ollama run gemma4:31b` |
| Hugging Face | All four sizes | Requires transformers 5.5.0+ |
| vLLM | All four sizes | Use `--tool-call-parser gemma4` flag |
| OpenRouter | 31B and 26B | API access, usage-based billing |
| Google AI Studio | 31B and 26B | Free experimentation |
| LiteRT-LM | E2B and E4B | On-device, mobile, IoT |
| llama.cpp / MLX | All four sizes | Community quantised versions |

For our browser tools, the primary paths are: Ollama (local, offline) and OpenRouter (API, online). The E2B model (2B effective parameters) is small enough that browser-based WASM execution may eventually be feasible.

### What This Means for Our Components

1. **sg-tool-definition** should output JSON schema compatible with Gemma 4's `tools` parameter format
2. **sg-json-receiver** should parse Gemma 4's `<|tool_call>` special tokens (when using raw model output) or the structured `tool_calls` array (when using the OpenAI-compatible API via vLLM/OpenRouter)
3. **sg-agentic-loop** should handle the thinking + tool calling interaction correctly (preserve thoughts within multi-step tool sequences, strip between turns)
4. **sg-json-sender** should support both the `tools` parameter (for tool definitions) and `response_format` (for structured JSON output without tools)
5. The **sandbox** security model aligns with Google's own guidance: "Always validate function names and arguments before execution"

Research and document which models work best with our tool definitions. Not all models handle tool calling equally well. Some hallucinate tool names, some produce malformed JSON, some ignore the schema. Build a compatibility matrix and publish it.

### Testing Framework

For each model, run a standard test suite:

- Can it correctly call a simple tool? (search_vault with a pattern)
- Can it chain two tool calls? (list then read)
- Does it respect the JSON schema? (correct types, required fields)
- Does it know when to stop? (produce a final answer instead of infinite tool calls)
- How does it handle tool errors? (tool returns an error, does the LLM recover?)

Publish results on the website. This is another traffic driver: anyone evaluating models for agentic use cases benefits from our testing.

## Connection to Existing Architecture

- **sg-llm component:** The agentic loop uses sg-llm for API calls, adding tool definitions to the request
- **VFS:** Tools read/write through the virtual file system
- **sg-wasm:** Pyodide loaded and cached via the WASM loader
- **SG Layout:** Each component (tool definitions, JSON sender, JSON receiver, trace) is a panel in SG Layout
- **Model Chooser:** Select which model to use for the agentic loop
- **One-shot feedback loops:** The agentic loop IS a feedback loop (task, tool calls, results, refinement)
- **Multi-agent chat:** Multiple agentic loops running in parallel, each with different tool sets
