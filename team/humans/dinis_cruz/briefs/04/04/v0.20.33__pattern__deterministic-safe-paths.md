## The Pattern

Create server-side paths and identifiers that are deterministic (the same input always produces the same ID) but practically impossible to discover without knowing the secret that generated them. The server stores data at these paths but has no way to determine what they represent or who they belong to.

This pattern has been used in multiple places and is becoming a strategic primitive across the platform. Time to formalise it.

## How It Works

```
Strong Secret (vault key, access token, simple token, etc.)
  │
  ▼
PBKDF2 (600,000 iterations, with salt)
  │
  ▼
Deterministic ID (hex string, used as path/filename/index)
  │
  ▼
Server stores data at this ID
```

### The Properties

| Property | What It Means |
|---|---|
| **Deterministic** | Same secret always produces the same ID. No lookup table needed. Anyone with the secret can calculate the path. |
| **Computationally expensive to reverse** | PBKDF2 with 600,000 iterations makes brute-force discovery impractical. Even knowing the ID, you cannot work backward to the secret. |
| **Practically impossible to enumerate** | The input space is enormous (vault keys, tokens with billions of combinations). You cannot scan for valid IDs. |
| **Server-blind** | The server stores data at the ID but does not know what the ID represents. It might know "this is a user ID" (from the path prefix) but not which user. |
| **No coordination needed** | Two parties who share the same secret will independently calculate the same ID. No server-side registration, no handshake, no ID assignment. |

### The Formula

```python
import hashlib

def derive_deterministic_id(
    secret: str,
    salt: str,
    iterations: int = 600_000,
    length: int = 16
) -> str:
    """
    Derive a deterministic ID from a secret.

    - secret: The strong secret (vault key, access token, etc.)
    - salt: A context-specific salt (prevents cross-context collisions)
    - iterations: PBKDF2 iterations (600k = computationally expensive)
    - length: Output length in hex characters (16 = 64 bits)
    """
    derived = hashlib.pbkdf2_hmac(
        'sha256',
        secret.encode(),
        salt.encode(),
        iterations
    )
    return derived.hex()[:length]
```

The salt is critical: it prevents the same secret from producing the same ID in different contexts. A vault key used as a transfer group ID should produce a different result than the same vault key used as a user ID. The salt encodes the context.

## Where It Is Already Used

### 1. SGit Vault Object IDs

Vault objects (commits, trees, blobs) are stored at deterministic paths derived from the vault key + content hash. Anyone with the vault key can calculate where objects live. The server stores them without knowing what vault they belong to.

### 2. Backend Storage Grouping (New)

The `by-id-from__*` path segments in the transfer storage restructuring (from today's earlier brief). Group IDs derived from access tokens, decryption keys, user identities, and PKI public keys. The server routes transfers into grouped paths without knowing who the groups belong to.

### 3. Vault Browse Without Cloning

The vault viewer at vault.sgraph.ai accesses specific files via deterministic IDs calculated from the vault key. No need to clone the entire vault. Calculate the ID, fetch the file. The server serves the file without knowing what it is or who it is for.

## Where It Can Be Used Next

### Per-User Settings and Preferences

Store user settings at a path derived from their access token:

```
settings/{derive_id(access_token, "sg-settings")}/preferences.enc
```

The user's browser calculates the path, fetches the settings, decrypts locally. The server stores encrypted preferences without knowing whose they are. No user account, no login, no session. Just a deterministic path and a decryption key.

### Shared Secrets Between Parties

Two parties who share a secret (a simple token, a vault key) can independently calculate a meeting point:

```
shared/{derive_id(shared_secret, "sg-shared")}/messages/
```

Both parties write to and read from the same path. The server does not know they are communicating. No server-side channel creation needed.

### Encrypted Search Indexes

Store search terms at deterministic paths derived from the search term + a vault key:

```
index/{derive_id(vault_key + ":" + search_term, "sg-index")}/results.enc
```

The client calculates the index path for a search term, fetches the encrypted result set, decrypts locally. The server cannot see what was searched for or what was found.

### Double Encryption for Sensitive Data

For particularly sensitive data, use the pattern twice:

1. First layer: data encrypted with the standard key, stored at the standard path
2. Second layer: a subset of data re-encrypted with a stronger key, stored at a path derived from the stronger key

The server sees two sets of encrypted files. It cannot tell that they are related. Even if the first layer is compromised, the second layer remains protected.

### Ephemeral Communication Channels

Create a temporary path derived from a one-time secret:

```
ephemeral/{derive_id(one_time_secret, "sg-ephemeral")}/
```

Both parties calculate the path from the shared one-time secret. Exchange data. Delete when done. The server never knew who was communicating or what was exchanged.

## Security Analysis

### What the Server Knows

- That data exists at a particular path
- The size and timestamp of the data
- The path prefix (if typed, e.g. `by-id-from__user/`)
- That someone with the right secret can access it

### What the Server Does Not Know

- Which user, which vault, which transfer the data belongs to
- What the data contains (encrypted)
- The secret that generated the path
- Whether two paths are related to the same user (different salts produce different IDs from the same secret)

### What Would Be Needed to Compromise

- Knowledge of the secret (vault key, access token, etc.)
- OR brute-force PBKDF2 with 600,000 iterations across a search space of billions of combinations
- The computational cost of the second option makes it impractical

## Implementation Checklist

When using this pattern for a new feature:

1. **Identify the secret.** What strong secret does the user already have? (Vault key, access token, simple token, PKI key)
2. **Choose a unique salt.** The salt must be unique to this feature/context. Document it. Never reuse a salt across features.
3. **Use PBKDF2 with 600,000 iterations.** Do not reduce iterations for performance. The computational cost is the security.
4. **Choose the output length.** 16 hex chars (64 bits) for path segments. 32 hex chars (128 bits) for globally unique IDs.
5. **Encrypt the data at rest.** The deterministic path is access control (you can only find the path if you know the secret). But the data should also be encrypted (defence in depth).
6. **Document the derivation.** In the codebase, document exactly how the ID is derived (secret source, salt, iterations, length). Future developers must be able to recalculate.
