# Browser Automation API Layer: Making Tools Machine-Readable for Playwright Agents and Human-Explorable via Swagger-Style UI

**version** v0.20.38
**date** 08 Apr 2026
**from** Human (project lead)
**to** Developer (lead), Architect, QA, Designer
**type** Dev brief

---

## The Problem

We have powerful browser-based tools (infographic generator, vault browser, video editor, LLM chat) that work well for humans. We also have agents (Claude with computer use, automated Playwright sessions) that can drive these tools via browser automation. But the current automation approach is brittle and non-deterministic: hand-crafted Python scripts that poll for CSS selectors, guess at timing, and break when the UI changes.

The infographic generation pipeline demonstrates both the power and the fragility. It works: Claude can set up Chromium, navigate to `dev.tools.sgraph.ai`, fill in fields, wait for generation, export PNGs. But the script is 80+ lines of fragile selector targeting, timing hacks, and manual state detection.

This does not scale. Every new tool needs a bespoke script. Every UI change breaks existing scripts.

## The Core Principle: One Interface, Two Audiences

The solution is NOT a separate mode for agents. It is a single interface that is equally usable by humans (via a Swagger-style interactive UI) and by agents (via `window.__tool` API and a SKILL.md-like text description). The same actions, the same parameters, the same results. Two ways to invoke them.

Think Swagger/OpenAPI: developers use the interactive docs to explore and test APIs. Automated systems use the same endpoint definitions programmatically. Same API, two consumers.

```
|                                                                  |

|  |                    SHARED TOOL CORE                         |  |

|  |  window.__tool_manifest   (JSON: actions, params, events)   |  |

```

Both the human UI and the automation panel call the same `window.__tool` core. The human clicks buttons that call `window.__tool.generate()`. The automation panel lets you type JSON and call the same function. An agent via Playwright calls it directly. Same function, three invocation paths.

## The Three Layers

### Layer 1: SKILL.md (Agent Reads This First)

Just like Claude reads a SKILL.md file before using a skill, an agent visiting a tool reads `window.__tool_skill` (or fetches `?format=skill` as plain text). This is the agent's instruction manual.

```
window.__tool_skill = `
# Infographic Generator

## What This Tool Does
Generates visual infographics from text prompts using LLM image
generation models via OpenRouter. Outputs PNG images.

## Quick Start
1. Call connect() with your OpenRouter API key
2. Call set_template() to choose a visual style
3. Call generate() with your prompt text
4. Wait for the 'generation:complete' event
5. Call export_png() to get the image

## Actions

### connect({ api_key: string }) -> { connected: bool, model: string }
Connects to OpenRouter and validates the API key.
Must be called before any generation.

### set_template({ template: string }) -> { applied: bool }
Sets the infographic style template.
Options: executive-summary, architecture, timeline,
         comparison, process-flow, stats-dashboard, mind-map

### generate({ prompt: string, model?: string }) -> { generation_id: string }
Starts infographic generation. This is async.
Listen for events:
  - 'generation:started'   -> generation has begun
  - 'generation:complete'  -> infographic ready, call export_png()
  - 'generation:error'     -> { error: string, code: string }
Typical generation time: 12-15 seconds.

### export_png() -> { url: string, size_bytes: number }
Exports the current infographic as PNG.
Only valid after 'generation:complete'.

### get_cost({ generation_id: string }) -> { total_cost, model, ... }
Fetches cost data from OpenRouter for the given generation.

## Example: Full Pipeline in Playwright

    const m = await page.evaluate(() => window.__tool_manifest);
    await page.evaluate(k => window.__tool.connect({api_key:k}), key);
    await page.evaluate(() =>
      window.__tool.set_template({template:'process-flow'}));
    const gen = await page.evaluate(p => {
      return new Promise((resolve, reject) => {
        addEventListener('generation:complete',
          e => resolve(e.detail), {once:true});
        addEventListener('generation:error',
          e => reject(e.detail), {once:true});
        window.__tool.generate({prompt: p});
      });
    }, prompt);
    const png = await page.evaluate(() => window.__tool.export_png());

## Notes
- Generation typically costs ~$0.07 with gemini flash
- Wait for 'tool:ready' event before calling actions
- All actions return Promises
- Errors throw with { error: string, code: string }
`;
```

This is prose, not just JSON schema. It explains the flow, the sequencing, the gotchas, and includes a working code example. Exactly like a SKILL.md.

### Layer 2: Manifest (Machine-Readable Schema)

The structured JSON schema that both the Swagger UI and programmatic consumers use:

```json
{
  "tool": "infographic-generator",
  "version": "0.20.38",
  "base_url": "https://dev.tools.sgraph.ai/en-gb/infographic-gen/",
  "actions": {
    "connect": {
      "description": "Connect to OpenRouter with API key",
      "params": {
        "api_key": { "type": "string", "required": true }
      },
      "returns": {
        "connected": { "type": "boolean" },
        "model": { "type": "string" }
      }
    },
    "set_template": {
      "description": "Choose infographic visual style",
      "params": {
        "template": {
          "type": "string", "required": true,
          "enum": ["executive-summary", "architecture", "timeline",
                   "comparison", "process-flow", "stats-dashboard",
                   "mind-map"]
        }
      },
      "returns": { "applied": { "type": "boolean" } }
    },
    "generate": {
      "description": "Generate infographic from prompt (async)",
      "params": {
        "prompt": { "type": "string", "required": true },
        "model":  { "type": "string", "required": false }
      },
      "returns": { "generation_id": { "type": "string" } },
      "events": ["generation:started", "generation:complete",
                 "generation:error"],
      "async": true
    },
    "export_png": {
      "description": "Export current infographic as PNG",
      "params": {},
      "returns": {
        "url": { "type": "string" },
        "size_bytes": { "type": "number" }
      },
      "precondition": "generation:complete"
    },
    "get_cost": {
      "description": "Fetch generation cost from OpenRouter",
      "params": {
        "generation_id": { "type": "string", "required": true }
      },
      "returns": {
        "total_cost": { "type": "number" },
        "model": { "type": "string" },
        "tokens_prompt": { "type": "number" },
        "tokens_completion": { "type": "number" },
        "latency": { "type": "number" }
      }
    }
  },
  "events": {
    "tool:ready":           "Tool fully loaded and ready",
    "generation:started":   "Generation has begun",
    "generation:complete":  "Infographic ready for export",
    "generation:error":     "Generation failed"
  }
}
```

**Key UI features:**

- **[Try It]** executes the action with the filled params, shows result below
- **[Copy Code]** copies the Playwright JavaScript snippet for that action with current params
- **Event Log** shows all events in real time (same events an agent would listen for)
- **Result panel** shows the JSON response from the last action
- **[Copy SKILL.md]** copies the full skill text for pasting into a Claude session or agent config
- **[Copy Manifest]** copies the JSON schema

## Component Architecture

```
+-------------------------------------------------------------------+
|                    Web Component: <sg-tool-api>                    |
+-------------------------------------------------------------------+
|                                                                   |
|  Inputs:                                                          |
|    manifest  = JSON schema of actions                             |
|    skill     = SKILL.md text                                      |
|    tool_ref  = reference to window.__tool implementation          |
|                                                                   |
|  Renders:                                                         |
|    +-----------------------------+                                |
|    | Automation Panel UI         |  (generated from manifest)     |
|    | - Action forms              |                                |
|    | - [Try It] buttons          |                                |
|    | - [Copy Code] buttons       |                                |
|    | - Event log                 |                                |
|    | - Result viewer             |                                |
|    | - [Copy SKILL.md]           |                                |
|    | - [Copy Manifest]           |                                |
|    +-----------------------------+                                |
|                                                                   |
|  Exposes:                                                         |
|    window.__tool_manifest  (JSON)                                 |
|    window.__tool_skill     (string)                               |
|    window.__tool.action()  (Promise-based API)                    |
|    CustomEvents            (tool:ready, action:*, etc.)           |
|                                                                   |
+-------------------------------------------------------------------+
```

The `<sg-tool-api>` Web Component is reusable. Drop it into any tool, pass it a manifest and a skill text, and it auto-generates the full automation panel plus the `window.__tool` API. The tool developer only needs to:

1. Define the manifest (what actions exist)
2. Write the skill text (how to use them)
3. Implement each action as a function
4. Fire events at state transitions

The component handles everything else: UI rendering, code generation, event logging, the Swagger-style explorer.

## The SKILL.md Convention

Every tool publishes a SKILL.md at a predictable URL:

```
https://dev.tools.sgraph.ai/en-gb/infographic-gen/SKILL.md
https://dev.tools.sgraph.ai/en-gb/video-editor/SKILL.md
https://dev.tools.sgraph.ai/en-gb/vault-browse/SKILL.md
```

An agent setting up a workflow can fetch these before opening the tool:

```python
# Agent planning phase
skill = requests.get(f'{tool_url}/SKILL.md').text
# Read the skill, understand the tool, plan the workflow
# Then open the tool in Playwright and execute
```

This also means the tools index page can list all available tools with their SKILL.md summaries, creating a discoverable catalogue.

## How It Connects to MCP

The manifest format is deliberately close to MCP tool definitions. A future bridge could expose browser tools as MCP servers: the MCP client calls a tool, the bridge translates to `window.__tool.action()`, and results flow back. This means browser tools become available to any MCP-capable agent without Playwright at all.

```
Agent (Claude, etc.)
  |
  +--[MCP]---> MCP Bridge ---> Playwright ---> window.__tool.action()
  |
  +--[Direct Playwright]-----> window.__tool.action()
  |
  +--[Fetch SKILL.md]-------> Plan, then Playwright
```

## Current State vs Target

| Aspect | Current (Playwright scripting) | Target (Tool API + Swagger UI) |
|---|---|---|
| Lines of code per tool | 80+ per tool | ~10 per tool |
| Brittleness | High (CSS selectors, timing) | Low (named actions, events) |
| Discoverability | Read the DOM, guess | SKILL.md + manifest + catalogue |
| State detection | Poll for element absence | Listen for events |
| Human exploration | Use the tool, hope | Swagger panel shows everything |
| Code generation | Write from scratch | [Copy Code] gives exact snippet |
| Reusability | None (bespoke per tool) | Universal `<sg-tool-api>` component |
| Agent setup | Custom script per workflow | Fetch SKILL.md, plan, execute |
| Tool chaining | Manual data passing | Folder/vault shuttle between tools |
