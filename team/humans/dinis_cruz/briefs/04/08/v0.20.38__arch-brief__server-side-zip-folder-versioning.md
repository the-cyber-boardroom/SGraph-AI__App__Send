# Server-Side Zip Folder Versioning: Per-File Storage, Mutable Head, and Conflict Detection

**date** 08 Apr 2026

## The Problem

Currently, zip files on SG/Send are monolithic: upload the whole thing, download the whole thing. This blocks several workflows: video playback from large zips (download one file, not all), multi-user editing of shared folders, and incremental updates. We need server-side support for individual file access within a zip, with versioning and conflict detection.

This is the "simple version without the vault" for cases where a full vault is overkill but multi-file, multi-user access is needed.

## Storage Structure

When a zip is uploaded, the server stores each file individually under deterministic encrypted paths (using the existing deterministic-safe-paths pattern). The tree structure is stored separately.

```
<folder-root>/
  head                    # Mutable: points to latest tree version
  trees/
    <timestamp-hash-1>   # Immutable: tree snapshot at time of upload
    <timestamp-hash-2>   # Immutable: tree snapshot after edit
    ...
  files/
    <file-id-1>          # Immutable: individual encrypted file
    <file-id-2>          # Immutable: individual encrypted file
    ...
```

### Key Invariant: One Mutable File

Only `head` is mutable. It contains a reference (ID) to the latest tree snapshot. Everything else is immutable and append-only.

- `head` contains the ID of the current tree
- Each tree snapshot is a JSON listing of every file in the folder (name, file-id, size, timestamp)
- Files are stored individually and never overwritten (content-addressable or timestamp-keyed)

## Conflict Detection

### The Race Condition

Two users have the same folder open. Both make changes. The second write would overwrite the first user's tree.

### The Solution: Compare-and-Swap on Head

When updating the tree, the client sends:

1. The new tree data
2. The previous head reference (the tree ID the client believes is current)

The server checks: does the current `head` match the client's previous reference? If yes, update. If no, reject with a conflict error.

```
Client A reads head -> tree-v1
Client B reads head -> tree-v1
Client A writes tree-v2, previous=tree-v1 -> SUCCESS, head now points to tree-v2
Client B writes tree-v3, previous=tree-v1 -> REJECTED, head is tree-v2 not tree-v1
Client B re-reads head -> tree-v2, resolves conflict, retries
```

The race condition window is tiny (two uploads at the exact same millisecond), and even then, the compare-and-swap catches it.

### Recovery

Because all tree versions are immutable and stored with timestamps, no data is ever lost. If a conflict produces a bad state, rebuild the tree from the stored versions. Every file ever uploaded remains accessible by its ID.

## Access Control: Read-Only vs Write Tokens

The simple token system needs two modes:

| Token Type | Permissions | Use Case |
|---|---|---|
| Read token | Download files, read tree | Sharing, viewing, playback |
| Write token | Upload files, update tree, update head | Collaboration, editing |

The read token is shareable publicly. The write token is restricted to collaborators. Both are derived from the same folder ID but with different encryption keys or access scopes.

## API Operations

| Operation | Method | What It Does |
|---|---|---|
| `GET /folder/{id}/head` | Read | Get current tree reference |
| `GET /folder/{id}/tree/{tree-id}` | Read | Get specific tree snapshot |
| `GET /folder/{id}/file/{file-id}` | Read | Download individual file |
| `PUT /folder/{id}/file` | Write | Upload individual file, get file-id back |
| `PUT /folder/{id}/tree` | Write | Upload new tree, provide previous head for conflict check |

## Connection to Existing Systems

This is architecturally between a raw zip upload (no versioning, no partial access) and a full vault (SGit, branches, commits, encryption). It uses the same deterministic-safe-paths pattern (PBKDF2, 600k iterations) for file and tree IDs. It could serve as a stepping stone: folders that outgrow this model can be promoted to full vaults.
