## The Insight

The previous brief (browser-automation-api-layer) described adding an automation panel to tools. This brief goes deeper. The insight is: **the JavaScript API is not an automation layer added to a tool. It IS the tool. The UI is just one consumer of the API.**

This is the classic MVC/API-first inversion. Today, our tools are UIs that happen to do things. The target: every tool is a JavaScript API that happens to have a UI. The API is the contract. The UI, Playwright scripts, other components, test suites, and Node.js consumers are all equal clients of the same API.

## The Primitive: JavaScript Execution Within the Tool's Domain

The key concept is: **any entity that can invoke JavaScript within the domain of the tool is a first-class consumer.** This includes:

- A human clicking buttons (the UI calls the API internally)
- Playwright driving a browser (calls `window.__tool.action()` via `page.evaluate()`)
- Another component in the same page (calls the API directly or via events)
- An iframe embedding the tool within SG Layout (scripting access across frames)
- A test suite running assertions against the API
- An in-browser console typing commands manually
- (Future) Node.js running the same code without a browser

The primitive is not "browser automation." The primitive is "JavaScript invocation."

## The Three Layers of Every Tool

Every tool ships with three things:

```
+------------------------------------------------------------------+
|                           TOOL                                   |
|                                                                  |
|  1. JavaScript API    (the contract, the primitive)              |
|     - Public methods: connect(), generate(), export(), etc.      |
|     - Event emitters: generation:started, generation:complete    |
|     - Meta methods: getManifest(), getSkills(), listMethods()    |
|     - Registered on window.__tool or via custom element API      |
|                                                                  |
|  2. UI Layer          (one consumer of the API)                  |
|     - Buttons, forms, panels that call the JS API               |
|     - Purely declarative over time, made of shared components    |
|     - Version changes independently from the API                |
|                                                                  |
|  3. Content Layer     (markdown, skills, definitions)            |
|     - SKILL files (three variants, see below)                    |
|     - Manifest (OpenAPI-style JSON definition)                   |
|     - Markdown documentation (human-readable explanation)        |
|     - Shipped with the tool, versioned separately                |
+------------------------------------------------------------------+
```

### IFD Versioning Applies

The API and UI version independently, following IFD principles:

| Layer | Version Cadence | Stability |
|---|---|---|
| JavaScript API | Slow, deliberate changes. This is the contract. | High (breaking changes are costly) |
| UI | Fast, iterative. Can change freely as long as it calls the same API. | Low (expected to evolve) |
| Content (skills, docs, markdown) | Updated when API changes or when better explanations emerge. | Medium |

## Three SKILL Files Per Tool

Each tool ships three skill files, each targeting a different consumer:

```
tool/
  SKILL-human.md          # For a human using the UI
  SKILL-browser.md        # For Playwright / browser automation
  SKILL-api.md            # For JavaScript API consumers
  manifest.json           # OpenAPI-style structured definition
  README.md               # General explanation of the tool
```

### SKILL-human.md (User Manual)

How a human uses the tool. Click here, type there, what to expect.

### SKILL-browser.md (Browser Automation)

For agents simulating a human via Playwright. Documents: element IDs, button labels, text input selectors, events to listen for, stable `data-automation-*` attributes. This is the "UI driving" mode.

### SKILL-api.md (JavaScript API)

The most important one. Documents every public method, its parameters, return values, events emitted, and async behaviour. Includes working code examples. This is the mode where you skip the UI entirely and call the API directly.

```
# Infographic Generator - JavaScript API

## Meta Methods (every tool has these)

### __tool.meta.getManifest() -> object
Returns the full manifest (OpenAPI-style definition).

### __tool.meta.getMethods() -> string[]
Lists all available API methods.

### __tool.meta.getSkills() -> { human: string, browser: string, api: string }
Returns all three skill files as text.

### __tool.meta.getVersion() -> { api: string, ui: string, content: string }
Returns version numbers for each layer.

## Tool-Specific Methods

### __tool.connect({ api_key: string }) -> { connected: bool, model: string }
### __tool.setTemplate({ template: string }) -> { applied: bool }
### __tool.generate({ prompt: string, model?: string }) -> { generation_id: string }
  Events: generation:started, generation:complete, generation:error
### __tool.exportPng() -> { url: string, size_bytes: number }
### __tool.getCost({ generation_id: string }) -> { total_cost, model, tokens_prompt, tokens_completion, latency }
```

## The Meta API: Standard Methods Every Tool Exposes

Every tool, every component, exposes a base set of meta methods. This is the equivalent of calling MCP to get the tool list, or calling OpenAPI to get the schema.

```
+-------------------------------------------------------------------+
|                  Base API (every tool inherits this)               |
+-------------------------------------------------------------------+
|                                                                   |
|  __tool.meta.getManifest()    -> JSON schema of all methods       |
|  __tool.meta.getMethods()     -> list of method names             |
|  __tool.meta.getSkills()      -> three skill files as text        |
|  __tool.meta.getVersion()     -> { api, ui, content } versions    |
|  __tool.meta.getEvents()      -> list of events the tool emits   |
|  __tool.meta.health()         -> { status, dependencies, ready }  |
|                                                                   |
+-------------------------------------------------------------------+
                              |
                    (tool extends this)
                              |
+-------------------------------------------------------------------+
|              Tool-Specific API (unique per tool)                   |
+-------------------------------------------------------------------+
|                                                                   |
|  Infographic:  connect, setTemplate, generate, exportPng, getCost |
|  Vault:        clone, commit, push, pull, share, status           |
|  Encryption:   encrypt, decrypt, generateKey                      |
|  Video:        extractAudio, captureFrames, assemble, exportMp4   |
|  Audio:        record, transcribe, enhance                        |
|                                                                   |
+-------------------------------------------------------------------+
```

An agent arriving at any tool calls `__tool.meta.getManifest()` first. It gets back a structured definition of every method available. It then reads `__tool.meta.getSkills().api` for the prose explanation. Then it starts calling methods.

## Events vs Direct API Calls

Two invocation patterns, each with a clear use case:

```
Same component (internal)        Cross-component (external)
        |                                |
        v                                v
  Direct API call                  Event dispatch
  __tool.generate()                dispatchEvent('tool:generate')
        |                                |
        v                                v
  Returns Promise                  Listener calls __tool.generate()
  (sync-style await)               (decoupled, any listener)
```

**Rule:** Events are the wrapper for cross-component communication. Within the same component, call the API directly. An event should just be a thin wrapper that invokes an API method.

This means: if you are Playwright or a test suite, call the API directly. If you are another component on the page, dispatch/listen for events. Both end up calling the same code.

## Async Model: Events Over Polling

The current Playwright scripts poll for DOM changes (is the Stop button gone? is the Generating text gone?). This is fragile. The target:

```
CURRENT (polling):
  for i in range(90):
      page.wait_for_timeout(2000)
      stop = page.query_selector('button:has-text("Stop")')
      cancel = page.query_selector('button:has-text("Cancel")')
      if not stop and not cancel:
          break

TARGET (event-driven):
  result = await page.evaluate(() => {
    return new Promise((resolve, reject) => {
      window.addEventListener('generation:complete',
        e => resolve(e.detail), { once: true });
      window.addEventListener('generation:error',
        e => reject(e.detail), { once: true });
      window.__tool.generate({ prompt: '...' });
    });
  });
```

Or even simpler, the API method itself can be synchronous (returns a Promise that resolves when done):

```
TARGET (async/await):
  result = await page.evaluate(async (p) => {
    return await window.__tool.generate({ prompt: p, await: true });
  }, prompt);
```

**Preference order:** async/await call > event listener > polling. Polling is the last resort, never the design.

## The In-Browser Tester UI

Every tool gets a companion tester component. This is separate from the Swagger-style automation panel (which is for exploration). The tester is for verification.

```
+-----------------------------------------------------------------------+
| TOOL TESTER: Infographic Generator                        [Run All]   |
+-----------------------------------------------------------------------+
|                                                                       |
|  TEST SUITE                                                           |
|  +---------------------------------------------------------------+   |
|  | [x] connect_with_valid_key        PASS  (1.2s)                |   |
|  | [x] connect_with_invalid_key      PASS  (0.8s)  expects error |   |
|  | [x] set_template_process_flow     PASS  (0.1s)                |   |

|  | [x] get_cost_returns_data         PASS  (1.1s)  $0.07         |   |
|  | [ ] generate_without_connect      FAIL  expected error thrown  |   |
|  | [x] export_before_generate        PASS  (0.1s)  expects error |   |
|  +---------------------------------------------------------------+   |
|                                                                       |
|  CONSOLE                                                [Clear]       |
|  +---------------------------------------------------------------+   |
|  | > __tool.meta.getMethods()                                     |   |

|  | > __tool.connect({api_key: "sk-or-..."})                       |   |
```

**Two sections:**

1. **Test suite:** Pre-written scenarios that exercise the API. Click [Run All] or run individually. These are the contract tests. If a UI change breaks a test, the API contract is violated.
2. **Console:** An in-browser REPL where you type `__tool.connect({...})` and see the result. Same as the browser DevTools console but embedded in the tool page itself, with autocomplete from the manifest.

## Parallel Execution Becomes Trivial

With a proper API, running five infographics from five models in parallel is straightforward:

```javascript
const models = [
  'google/gemini-3.1-flash-image-preview',
  'google/gemini-3-pro-image-preview',
  'anthropic/claude-sonnet-4',
  'openai/gpt-4o',
  'meta/llama-3-70b'
];

const results = await Promise.all(
  models.map(model =>
    window.__tool.generate({
      prompt: 'Architecture diagram of a vault system',
      model: model,
      await: true
    })
  )
);

// Five infographics, five models, parallel execution
// Each result contains generation_id, cost, image data
```

This is impossible with the current UI-driving approach (one generation at a time, sequential polling). With the API, it is five lines.

## Node.js Compatibility (Future)

The JavaScript API should be designed with Node.js compatibility in mind. If the API methods do not depend on browser-specific APIs (DOM, Canvas, WebAudio), they can run in Node.js directly. For browser-dependent operations, provide a shim or mark them as browser-only in the manifest.

```json
{
  "actions": {
    "generate": {
      "environment": ["browser", "node"],
      "description": "Generate infographic (uses fetch, works in both)"
    },
    "exportPng": {
      "environment": ["browser"],
      "description": "Export as PNG (requires Canvas API)"
    }
  }
}
```

This is not a priority now, but designing the API boundary cleanly makes it achievable later. The key principle: the code base is the same. We do not maintain separate browser and Node versions. We maintain one API that works in both where possible.

## The Agent Workflow (Revised)

With all three layers in place, the agent workflow becomes:

```
Agent (Claude + Playwright)
        |
        v
   Open tool URL in Chromium
        |
        v
   page.evaluate(() => __tool.meta.getManifest())
        |
        v
   Agent now knows every method, parameter, return type, event
        |
        v
   page.evaluate(() => __tool.meta.getSkills().api)
        |
        v
   Agent reads prose explanation with examples
        |
        v
   Agent plans: which methods in what order
        |
        v
   Direct API calls via page.evaluate()
   (no CSS selectors, no timing hacks, no DOM inspection)
        |
        v
   All async operations return Promises or fire events
   (no polling, no sleep loops)
        |
        v
   Results come back as structured JSON
   (no screenshot parsing, no text extraction from DOM)
```

Compare to today: open page, screenshot it, guess at selectors, write 80 lines of polling code, hope the timing works. The API-first approach eliminates every fragile step.

## Component Registration

When a tool or component loads on a page, it registers its API:

```javascript
// Tool registers itself
window.__tools = window.__tools || {};
window.__tools['infographic-generator'] = {
  meta: { getManifest, getMethods, getSkills, getVersion, getEvents, health },
  connect, setTemplate, generate, exportPng, getCost
};

// Also register as the primary tool if there is only one
window.__tool = window.__tools['infographic-generator'];

// Fire ready event
window.dispatchEvent(new CustomEvent('tool:ready', {
  detail: { tool: 'infographic-generator', version: '0.20.38' }
}));
```

When multiple components are on the same page (e.g. SG Layout with multiple panels), each registers under `window.__tools[name]`. The agent can enumerate all available tools via `Object.keys(window.__tools)`.

## What This Means for QA

The test suite is no longer "click buttons and check screenshots." It is: call API methods, assert on return values and events. This is unit testing for browser tools.

```javascript
// QA test: generate should fail without connect
try {
  await __tool.generate({ prompt: 'test' });
  assert.fail('Should have thrown');
} catch (e) {
  assert.equal(e.code, 'NOT_CONNECTED');
}

// QA test: export should fail without generation
try {
  await __tool.exportPng();
  assert.fail('Should have thrown');
} catch (e) {
  assert.equal(e.code, 'NO_GENERATION');
}

// QA test: full pipeline
await __tool.connect({ api_key: TEST_KEY });
await __tool.setTemplate({ template: 'process-flow' });
const gen = await __tool.generate({ prompt: 'test', await: true });
assert.ok(gen.generation_id);
const png = await __tool.exportPng();
assert.ok(png.size_bytes > 0);
const cost = await __tool.getCost({ generation_id: gen.generation_id });
assert.ok(cost.total_cost > 0);
```

These tests run in the browser, against the real tool, using the real API. No mocks, no Playwright overhead for simple contract tests.
