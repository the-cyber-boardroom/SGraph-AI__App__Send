# Profile Page Component: Token Management, Credit Activation, and Usage Visibility

**version** v0.20.38
**date** 08 Apr 2026
**from** Human (project lead)
**to** Developer (lead), Designer, Architect
**type** Dev brief

---

## The Goal

Add a profile page (top-right user icon, standard pattern) that lets users manage their tokens, activate credits, view usage, and see exactly what data is stored where. This is the immediate blocker for the early access user outreach: the human needs to tell users "go to this page and activate your credits."

## The Outreach Flow

1. Human contacts early access users and customers (~50 users)
2. Message: "You have been given five pounds credit. Go to your profile page and activate it."
3. User visits profile page, activates credits, gets an OpenRouter key provisioned
4. Credits have an expiration date (roughly one month) for tracking and urgency
5. User starts consuming tokens through the platform

## What the Profile Page Shows

### Client-Side Data (Local Storage)

The profile page displays what is stored in the browser's local storage. The user should see this transparently:

| Data | Stored Where | Why |
|---|---|---|
| Access token | Local storage | User identity (the access token IS the user ID for now) |
| Encryption keys | Local storage | Zero-knowledge: server never sees these |
| OpenRouter API key | Local storage | User's key for LLM access |
| Vault keys | Local storage | Per-vault decryption keys |

**Critical privacy point:** If the human (or anyone) knows a user's access token, they can reach the server-side profile but must NOT see any confidential information. Encryption keys, API keys, and vault keys are local storage only. The server stores no secrets.

### Server-Side Data

Minimal. The server tracks:

- Credit balance and expiration date
- Usage records (token consumption)
- Key ID (reference only, not the key itself)

### Usage Display

- Current credit balance
- Token consumption history
- Link to OpenRouter stats for that user (if available)
- Show credits prominently and consistently across pages to demonstrate that costs are minimal

## Credit System

| Field | Value | Notes |
|---|---|---|
| Initial credit | Five pounds | For existing users and early access |
| Expiration | ~1 month from activation | Creates urgency, allows tracking |
| Tracking | Server-side per user | Tied to access token |
| Activation | User-initiated via profile page | User clicks "activate" to claim credits |

Credits are the sales mechanism. Give people credits, let them experience the platform, then convert to paid usage.

## Data Architecture: What Lives Where

```
Browser (local storage)           Server (per access token)
  |                                 |
  +-- access token                  +-- credit balance
  +-- encryption keys               +-- credit expiration
  +-- OpenRouter API key            +-- usage records
  +-- vault keys                    +-- key ID (reference)
  +-- preferences
```

The profile page shows both columns side by side. The user sees exactly what data exists and where it lives.

## Token Portability

If the user moves to a new browser, they lose their local storage (keys, preferences). For now, they reactivate manually. Future: use a personal vault to back up and restore this data across browsers. This is the entry point for the personal vault concept, but not required for v1.

## Security Constraint

Even with knowledge of someone's access token, an observer must not gain access to any confidential information. The server-side profile contains only non-sensitive data (balance, usage, key IDs). All secrets remain in local storage under the user's control.
