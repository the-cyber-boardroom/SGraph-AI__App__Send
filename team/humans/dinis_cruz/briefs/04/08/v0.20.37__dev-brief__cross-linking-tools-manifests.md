# Cross-Linking Tools from the Main UI: Deep Links, Dependency Manifests, and Component Retrofitting

**version** v0.20.37
**date** 07 Apr 2026
**from** Human (project lead)
**to** Developer (lead), Architect, Designer
**type** Dev brief

## The Moment

The tools are maturing. The infographic generator works. The vault integration is solid. The LLM chat is functional. The tree view, the virtual file system, the SG Layout: all production-ready components. But they are islands. The main SG/Send UI does not link to them. A user browsing a folder cannot click a button to create an infographic from it. A user viewing a file cannot open it in the chat tool for analysis. The components exist. The connections do not.

Time to wire everything together.

## Feature 1: Deep Links from Main UI to Tools

### The Pattern

Wherever a user is viewing content (a file, a folder, a vault), add action buttons that link directly to the relevant tool with the content pre-loaded.

| Context | Action Button | Links To | Passes |
|---|---|---|---|
| Viewing a folder | "Create Infographic" | Infographic tool | Folder ID / simple token |
| Viewing a document (markdown, PDF) | "Analyse Document" | Document analysis tool | File reference |
| Viewing a folder | "Open in Chat" | LLM chat tool | Folder / vault reference |
| Viewing any file | "Open in Editor" | Appropriate editor (video, image, text) | File reference |
| Viewing a vault | "Browse in Vault Explorer" | Vault Explorer tool | Vault key |
| Viewing a gallery | "Create Slideshow" | Slideshow / presentation tool | Gallery reference |

### How Deep Links Work

The tools need to accept URL parameters that pre-load content:

```
https://tools.sgraph.ai/infographic?source=simple-token:apple-banana-cherry
https://tools.sgraph.ai/infographic?vault=vault-key&path=/reports/summary.md
https://tools.sgraph.ai/chat?source=transfer-id:abc123&key=decryption-key
```

The tool reads the URL parameters on load:
1. Detect the source type (simple token, transfer ID + key, vault + path)
2. Fetch the content
3. Pre-populate the input (the infographic tool gets the document text, the chat gets the context)
4. The user is one click away from the result

### Bidirectional

The tools should also link back. After creating an infographic, the user should be able to:
- Save it to their vault
- Share it via SG/Send (create a new transfer)
- Open it in the Browse view

The flow is circular: Browse → Tool → Result → Browse.

## Feature 2: Retrofit Existing Features with New Components

Several powerful components have been built recently but are not yet integrated into the main website features:

| Component | What It Does | Where It Should Be Used |
|---|---|---|
| `sg-tree` | Tree view for files and folders | Replace the current folder tree in Browse and Vault views |
| VFS (Virtual File System) | Unified file access across providers | Underlying layer for all file operations in the main UI |
| `sg-layout` | Panel management, resizing, fractal nesting | Replace fixed layouts in the main website with flexible panels |
| `sg-vault-*` primitives | Commit, push, pull, branch components | Add vault operations to the Browse view for vault-backed folders |
| Model Chooser | LLM model selection with categories and cost | Any feature that calls an LLM should use this |

### Retrofitting Plan

1. **Identify:** List every feature on the main website that could benefit from a tools-site component
2. **Version check:** Which version of each component is the tools site using? Which would the main site use?
3. **Integrate:** Replace the old implementation with the new component (IFD-versioned, independently deployable)
4. **Test:** QA verifies the integration does not break existing functionality

## Feature 3: Dependency Manifests for Website Features

As the main website starts consuming components from the tools site, we need to track dependencies. Each feature on the main website should have a manifest declaring:

```json
{
  "feature": "folder-browse",
  "version": "0.3.2",
  "dependencies": [
    { "component": "sg-tree", "version": "0.2.1", "source": "tools.sgraph.ai" },
    { "component": "sg-layout", "version": "0.4.0", "source": "tools.sgraph.ai" },
    { "component": "sg-markdown-viewer", "version": "0.1.5", "source": "tools.sgraph.ai" }
  ],
  "last_tested": "2026-04-07",
  "tested_by": "QA"
}
```

This manifest enables:

- **Version tracking:** Know exactly which component versions the main site uses
- **Update management:** When a component updates on the tools site, check which main site features are affected
- **Packaging:** Eventually bundle the right versions for offline deployment or partner distribution
- **Testing:** QA knows which combinations to test
- **Dependency graph:** Visualise which features depend on which components (the sg-git-graph component could render this)

This is the IFD manifest pattern applied to the main website. The tools site already has IFD versioning per component. Now the main site declares which versions it consumes.

## The Marketing Connection

Every cross-link is a marketing opportunity:

- A user creates an infographic from a document they shared: they see the tools in action
- A user opens a vault in the explorer from the main UI: they discover the vault capabilities
- A user chats with an LLM about a file they uploaded: they experience the AI features

The tools are not a separate product. They are features of SG/Send that happen to be accessible at tools.sgraph.ai. The cross-links make this visible. The user's journey goes: upload a file → browse it → create something from it → share the result. Every step happens within the SG/Send ecosystem.
