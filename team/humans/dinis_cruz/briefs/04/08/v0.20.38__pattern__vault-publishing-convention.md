# Pattern: Vault Publishing Convention with Commit-Tracked Snapshots

**version** v0.20.38
**date** 08 Apr 2026
**from** Human (project lead)
**to** Architect (lead), Developer
**type** Architecture pattern

---

## The Problem

When a vault is published as a folder (shared via simple token), there is no record inside the vault of what was published, when, or where. A publisher managing multiple snapshots of the same vault has no way to discover all published versions from the vault itself.

## The Pattern

Every time a vault is published as a folder, record the publication as a new commit in the source vault.

### Publication Metadata

```json
{
  "type": "publication",
  "published_at": "2026-04-08T14:30:00Z",
  "commit_hash": "abc123def",
  "folder_token": "apple-banana-cherry",
  "vault_id": "vault-xyz",
  "description": "Team setup pack for Partner X"
}
```

Store this in a conventional location, e.g. `_publications/2026-04-08T14-30-00Z.json` or appended to a `_publications/index.json`.

### Consequences

- A publisher manages **one vault ID** locally
- All published folder snapshots are discoverable from the vault history
- SGit already supports the primitives (commit, push) needed to implement this
- The publication log is itself versioned and encrypted

### Relationship to Existing Primitives

This uses the same commit-as-communication principle established in the IFD v2 methodology. The commit is not a code change; it is a metadata event recorded in the version history.

## Acceptance Criteria

| # | Criterion | Validated By |
|---|---|---|
| 1 | Publishing a folder creates a metadata commit in the source vault | Developer, Architect |
| 2 | All published snapshots are discoverable from a single vault | QA |
| 3 | Metadata includes commit hash, token, timestamp, and description | Developer |
