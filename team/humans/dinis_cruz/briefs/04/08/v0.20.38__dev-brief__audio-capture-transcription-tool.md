## The Goal

Replace the Otter.ai dependency with an in-house audio capture tool on tools.sgraph.ai (or the main site). Record voice memos directly from a phone, transcribe locally in real time, then progressively enhance through a multi-model pipeline. The output: a structured document, infographic, and slide deck, all generated from a single voice memo.

## Key Requirement: Zero Data Loss

The tool records on a phone. Phones lose battery, lose signal, crash. The architecture must guarantee no data loss:

- Record audio in multiple segments (not one continuous stream)
- Each segment is saved to local storage immediately on completion
- If the phone dies mid-segment, only the current segment is at risk
- All previous segments are already persisted
- Sync to a vault when connectivity allows

## The Progressive Enhancement Pipeline

Four stages, each adding quality. Stage 1 works fully offline. Stages 2+ require connectivity.

| Stage | Where | Model | Output | Latency |
|---|---|---|---|---|
| 1. Capture + live captions | Phone browser, offline | Local WASM model (Whisper.cpp or similar) | Raw audio segments + rough transcript with real-time captions | Instant |
| 2. Improved transcription | OpenRouter | Larger speech-to-text model | Cleaned transcript, better punctuation, speaker labels | Minutes |
| 3. Document generation | OpenRouter | LLM (text model) | Structured brief/article from transcript (like current workflow) | Minutes |
| 4. Visual generation | OpenRouter + client | LLM + infographic tool | Infographic, slide deck, summary | Minutes |

### Stage 1: Local Model Experiment

This is an experiment. Questions to answer:

- Which local speech-to-text model runs on a phone browser via WASM?
- Whisper.cpp (via whisper.cpp WASM), Whisper Web, or alternatives?
- What is the latency for real-time captions?
- What is the transcript quality compared to Otter.ai?
- Try multiple models, benchmark accuracy and speed

The key feature: captions appear in real time as the user speaks. Even if the local model is imperfect, the user gets immediate visual feedback that their audio is being captured.

### Stage 2: Server-Side Cleanup

Send the audio (or the rough transcript) to a more capable model via OpenRouter. Compare multiple models for transcription quality. This could be:

- A dedicated speech-to-text model
- An LLM with audio input capability
- A cleanup pass: send the rough local transcript + audio to an LLM and ask it to correct errors

### Stages 3-4: Reuse Existing Tools

Document generation is exactly what this briefing team does today: take a transcript, produce a structured brief. The infographic and slide generation tools already exist. This pipeline automates the full voice-memo-to-published-content workflow.

## Vault Integration

The audio tool is a prime use case for personal vaults:

```
Vault: voice-memos/
  |
  +-- 2026-04-08/
  |     +-- segment-001.webm
  |     +-- segment-002.webm
  |     +-- transcript-local.md      (Stage 1)
  |     +-- transcript-enhanced.md   (Stage 2)
  |     +-- brief.md                 (Stage 3)
  |     +-- infographic.png          (Stage 4)
  |     +-- slides.pdf               (Stage 4)
  |
  +-- 2026-04-09/
        +-- ...
```

Each stage writes its output to the vault. The vault becomes the persistent, versioned, encrypted record of every voice memo and everything derived from it.

## From Voice Memo to Published Article: The Full Loop

This tool completes a loop that has been manual since 11 March:

```
Current:  Phone -> Otter.ai -> Copy/paste -> Claude -> Briefs
Proposed: Phone -> Audio tool -> Local transcript -> OpenRouter -> Briefs + Infographics + Slides
```

Everything after "Phone" is automated and runs through the platform, consuming tokens and producing content.
