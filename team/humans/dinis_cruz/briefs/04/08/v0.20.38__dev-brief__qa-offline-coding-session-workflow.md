# QA Workflow for Processing Offline Coding Sessions: Commit-by-Commit Narrative Extraction

**version** v0.20.38
**date** 08 Apr 2026
**from** Human (project lead)
**to** QA (lead), Librarian, Developer
**type** Dev brief

---

## The Problem

When the human does a deep offline coding session (no agentic team, just raw experimentation), the result is a branch with dozens of commits containing inline comments tagged `@Dev`, `@QA`, `@Librarian`, plus experimental code, performance data, and temporary scaffolding. The knowledge is trapped in the commit sequence. Reading the final file state alone loses the narrative: experiments that were tried and discarded, performance numbers gathered mid-flow, and the reasoning behind breaking changes.

This brief defines the workflow for the QA team to extract, process, and structure that knowledge.

## Context: The Browser Persistence Session

The triggering session focused on persisting browser instances across web server restarts. The key achievement: reducing initialisation cost from ~1.5 seconds to ~50 milliseconds by persisting browser state. The session introduced breaking changes and produced a rich commit history that tells the story of the experiment.

## The Workflow

### Step 1: Identify the Commit Range

Find the delta between the last processed commit and the current branch head.

```bash
# Find the last commit that was processed (tracked by QA)
LAST_PROCESSED="<commit-hash>"

# Get the full range of new commits
git log --oneline $LAST_PROCESSED..HEAD

# Generate a combined patch for the full range
git diff $LAST_PROCESSED..HEAD > full-delta.patch
```

The full delta patch gives the overall picture, but Step 2 is where the real value is.

### Step 2: Read Commits Sequentially (Not the Final File State)

This is the critical insight. The commits tell a story. Each commit captures a moment in the experiment: what was tried, what was measured, what was kept, what was discarded.

```bash
# List commits in chronological order (oldest first)
git log --reverse --oneline $LAST_PROCESSED..HEAD

# Show each commit's changes individually
git log --reverse -p $LAST_PROCESSED..HEAD
```

**Why sequential reading matters:**

| Reading Method | What You Get | What You Miss |
|---|---|---|
| Final file state only | Current code, surviving comments | Discarded experiments, performance data from intermediate states, reasoning chain |
| Full delta patch | Net changes from A to B | Intermediate steps, commits that added then removed code |
| Commit-by-commit (chronological) | The full narrative: hypothesis, experiment, measurement, decision | Nothing (this is the complete record) |

### Step 3: Extract Tagged Comments

Scan the commit range for inline comments addressed to specific agents.

```bash
# Extract all tagged comments from the diff
git diff $LAST_PROCESSED..HEAD | grep -E '@(Dev|QA|Librarian|AppSec|Architect)'
```

Each tag type maps to a different output:

| Tag | Owner | Action |
|---|---|---|
| `@Dev` | Developer | Refactoring tasks, code improvements, TODOs |
| `@QA` | QA | Test cases needed, validation requirements, edge cases found |
| `@Librarian` | Librarian | Knowledge to capture, patterns to document, methodology notes |
| `@AppSec` | AppSec | Security considerations, threat model updates |
| `@Architect` | Architect | Design decisions, structural changes, breaking change rationale |

### Step 4: Separate Temporary from Permanent

The human deliberately leaves experimental code and comments in commits knowing they will be cleaned up. The QA team must distinguish:

- **Permanent additions:** New functionality, configuration, API changes
- **Temporary scaffolding:** Performance measurement code, debug logging, experimental branches
- **Comments to process then delete:** Tagged comments that exist to transfer knowledge, not to remain in code

### Step 5: Produce the Outputs

From a single offline session, produce multiple documents in sequence:

1. **Session narrative** (debrief): What was attempted, what was discovered, what was achieved. Written from the commit history.
2. **Per-agent briefs:** One dev brief per major technical topic. One QA brief for test requirements. Librarian entries for patterns and methodology.
3. **Refactoring brief:** Breaking changes identified, migration path, what experimental code should be promoted to production.
4. **Lessons learned:** Methodology and principles discovered during the session, captured for reuse.

**Order matters.** The narrative must be written first because it provides the context that makes the per-agent briefs coherent.

### Step 6: Track Processing State

After processing, record the last processed commit hash so the next session has a clean starting point.

```bash
# Record in QA tracking (could be a file in the repo, a vault entry, or a tag)
git tag qa-processed-$(date +%Y%m%d) HEAD
```

## The Principle: Commits as Communication

This workflow treats the git history as a first-class communication channel. The human writes commits not just to save code state, but to tell a story to the agentic team. This aligns with the existing "patches as communication" principle but extends it: the commit message and the commit diff together form a narrative unit. The sequence of commits forms the complete story.

## Key Requirement: The File Will Grow Then Shrink

During offline sessions, files grow as experiments accumulate. The human adds data, measurements, and comments. Later commits clean up: deleting failed experiments, removing processed comments, consolidating findings. This is by design. The QA team should expect the full delta to be smaller than the sum of individual commit diffs.

## Acceptance Criteria

| # | Criterion | Validated By |
|---|---|---|
| 1 | QA can identify the commit range for an unprocessed offline session | QA |
| 2 | Commits are read in chronological order, not just final file state | QA |
| 3 | All tagged comments (`@Dev`, `@QA`, `@Librarian`, etc.) are extracted and routed to correct agents | QA, Librarian |
| 4 | Session narrative (debrief) is produced before per-agent briefs | QA, Librarian |
| 5 | Temporary scaffolding is identified and separated from permanent changes | QA, Developer |
| 6 | Processing state is tracked (last processed commit hash) | QA |
| 7 | Breaking changes are flagged and documented in a refactoring brief | Developer, Architect |
