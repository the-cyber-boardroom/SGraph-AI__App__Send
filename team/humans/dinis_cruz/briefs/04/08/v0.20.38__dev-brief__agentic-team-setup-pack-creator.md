# Agentic Team Setup Pack Creator: A Multi-Step Wizard from Template Vault to Running Team

**date** 08 Apr 2026
**from** Human (project lead)
**to** Developer (lead), Architect, Designer
**type** Dev brief

## Context: The Partner Call That Validated the Flow

A partner call today demonstrated the full loop working end to end: briefing pack created in Claude, vault created, share document produced, `_page.json` with readme shared via link. The partner was impressed but the real excitement came when the human spun up an agentic team for them, live in the same session. The Librarian (with access to previous briefing packs) created a new vault containing the full agentic team setup. That vault was shared as a folder, and the partner could see the team definitions immediately.

Key feedback: "This is great, but can we make it even simpler?" The partner does not use the command line. The online vault view solved that, but the flow from "I want a team" to "here is your team running" needs to be a single guided experience.

Also surfaced: the website needs a clear explanation of the difference between the Folder View (browser, read-only published snapshot) and the Vault (live, versioned, encrypted store).

## The Tool: Agentic Team Creator

A new tool (component on tools.sgraph.ai) that walks a user through creating a customised agentic team in three to four steps. Each step is an LLM interaction with constrained inputs and outputs. Tool calls control phase transitions.

### The Vault-to-Vault Pattern

The core pattern: **input vault (template) + user context = output vault (customised team)**

```
Template Vault          User Input           Output Vault
(role definitions,  +  (project brief,  =>  (customised team,
 examples, skills,      objectives,          tailored roles,
 Claude.md samples)     attachments)         infographics,
                                             ready to run)
```

This is the "vault to vault" or "folder to folder" model. Each stage consumes a folder and produces a folder. Zips and vaults are interchangeable containers.

### Step-by-Step Workflow

| Step | What Happens | LLM Role | Tool Calls Available |
|---|---|---|---|
| 1. Select template | User provides a simple token or picks from published templates. UI downloads the folder containing role definitions, skills, examples. | None (UI only) | `download_folder(token)` |
| 2. Describe the project | Conversational: what is the project, what kind of team, objectives, background info, file attachments. System prompt guides the elicitation. | Elicitation | `mark_complete()` to advance |
| 3. Generate materials | LLM creates customised files: Claude.md, role definitions per agent, team objectives, workflow instructions. Multiple LLM calls, one per file or group. | Generation | `create_file(name, content)` |
| 4. Enrich (optional) | Create infographics per document, slide decks, summaries. Reuses existing infographic generation tooling. | Generation | `create_infographic(source)`, `create_slides(source)` |
| 5. Package and share | Bundle into a folder. Share as zip, vault, or published folder link. | None (UI only) | `create_vault(folder)`, `share(token)` |

### Phase Transitions via Tool Calls

Each step has a dedicated system prompt. The LLM's available tools determine what it can do in that phase. When the LLM calls a transition tool (e.g. `mark_complete()`), the UI advances to the next step with a new system prompt and new tool set. The user can navigate back and forward between steps.

This is a concrete implementation of the "customised one-shot prompt" pattern: a series of constrained LLM interactions where each step's output feeds the next step's context. Token usage is minimised because each step only sends what it needs.

```
Step 1          Step 2              Step 3            Step 4
[Select] -----> [Describe] -------> [Generate] -----> [Enrich]
  |               |                    |                  |
  | download      | mark_complete()    | create_file()    | create_infographic()
  | folder        | (advances step)    | (per material)   | create_slides()
  |               |                    |                  |
  v               v                    v                  v
 Template        Project brief        Team files         Enriched pack
 loaded          captured             generated          ready to share
```

### Template Vault Structure

The template vault contains the "best of" from the main repo, condensed:

- Example `Claude.md` files
- Explorer team role definitions
- Villager team role definitions
- Town Planner team role definitions
- Support team role definitions (Librarian, Sherpa, Conductor, etc.)
- All 20+ role definitions
- Skills definitions
- Example workflows and briefs

This can be topic-specific: a security-focused template, a development-focused template, a research-focused template.

### Convention: Private Folders with Underscore Prefix

Since dot-prefixed folders get stripped during folder operations, use underscore prefix for private/intermediate folders:

- `_claude/` or `_agentic/` contains team instructions, system prompts, configuration
- Claude (and other LLMs) can find these by convention
- The folder view can hide or show these based on user preference

### Publishing Convention for Vaults

When a vault is published as a folder, record the publication metadata as a new commit in the vault:

```json
{
  "published_at": "2026-04-08T14:30:00Z",
  "commit_hash": "abc123",
  "folder_token": "apple-banana-cherry",
  "vault_id": "vault-xyz"
}
```

This means a publisher manages one vault ID locally and can discover all published folder snapshots from the vault history. SGit already supports the primitives for this.

### "Talk to the Team" Mode

After the team pack is created (Step 5), offer a "Talk to the Team" option. This loads the generated team definitions into the existing agentic chat UI. The user drops the folder (local or from SG/Send) and the UI reads the `_agentic/` convention folder to bootstrap the team. Avatars per agent would improve the experience.

This reuses the existing multi-agent chat components but makes them folder-driven rather than hardcoded.

## Connection to Existing Components

| Existing Component | Role in This Workflow |
|---|---|
| Folder download/browse | Step 1: load template |
| LLM chat with tool calling | Steps 2-4: elicitation, generation, enrichment |
| Infographic generator | Step 4: per-document visuals |
| Vault create/share | Step 5: package and distribute |
| Agentic chat UI | Post-creation: talk to the team |
| `_page.json` layout | Published folder presentation |

## What This Unlocks

This is a user acquisition flow. Users who have never experienced agentic workflows get a guided path from "I have a project" to "I have a running team with defined roles, objectives, and materials." The vault-to-vault pattern means every interaction produces a shareable, versionable artifact.
