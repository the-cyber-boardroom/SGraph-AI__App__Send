## Context

Following the healthcare partner demo (which went well: they were impressed by the encryption workflow, the GDPR story, and the vault UI), we need an MVP that demonstrates the full platform capability in a concrete healthcare scenario. This is both a sales tool and a proof of architecture.

The scenario: a GP-style health checkup. 15-20 questions, client-side data capture, vault-based storage, doctor collaboration, LLM-assisted analysis, and the full data governance loop.

## Why This Matters Architecturally

The healthcare MVP is not just a demo. It proves a fundamental architectural claim: **applications can be entirely client-side, with all logic running in the browser and the only server-side data being encrypted ciphertext.**

No server-side application logic. No server-side database with patient records. No backend API handling sensitive health data. The vault stores encrypted data. The client-side JavaScript handles all the workflow, forms, analysis, and display. The server is a dumb encrypted store.

This is the data governance story that the healthcare partner responded to. This MVP makes it concrete and inspectable.

## The Workflow

```
Step 1: Patient Arrives
    |
    v
Create a vault for this patient/session
    |
    v
Step 2: Health Questionnaire (15-20 questions)
    |
    +--> All client-side JavaScript
    +--> Answers stored as structured JSON in the vault
    +--> Each section can be a separate commit (progress saved)
    |
    v
Step 3: Results Summary
    |
    +--> Client-side rendering of answers
    +--> Basic health indicators calculated locally
    |
    v
Step 4: LLM Analysis (optional)
    |
    +--> Send anonymised/structured data to LLM (via OpenRouter)
    +--> LLM provides guidance, flags, recommendations
    +--> Results stored back in vault as a new commit
    +--> Cost displayed to user (usage-based)
    |
    v
Step 5: Share with Clinician
    |
    +--> Patient shares vault with doctor (vault key or share token)
    +--> Doctor opens vault, sees questionnaire + LLM analysis
    +--> Doctor adds notes, validation, sign-off
    +--> Doctor's additions are new commits on their clone branch
    |
    v
Step 6: Clinician Review and Sign-Off
    |
    +--> Doctor reviews LLM suggestions
    +--> Doctor adds clinical notes, corrections, prescriptions
    +--> Doctor pushes changes (dual branch model)
    +--> Patient pulls to see doctor's notes
    |
    v
Step 7: Published Summary
    |
    +--> Final report published as a folder view (read-only share token)
    +--> Patient can share with other doctors, family, insurance
    +--> Folder view uses _page.json for clean rendering
```

## Key Technical Points

### All Logic is Client-Side

The questionnaire form, the validation, the results calculation, the LLM prompt construction, the rendering: all JavaScript running in the browser. The server never sees unencrypted health data. This is the architectural proof point.

This could be built as a Web Component (consistent with the IFD methodology) or as a `_page.json` layout with embedded components (using the whitelisted component embedding from the 10 Apr brief).

### Vault as the Data Layer

Each patient session is a vault. The structured data (questionnaire answers, LLM analysis, doctor notes) is stored as JSON files committed to the vault. The version history provides a complete audit trail: when each piece of data was captured, when the LLM analysis was added, when the doctor reviewed and signed off.

### Two Access Modes

**Patient view (vault key):** Full read-write access. The patient owns the vault and the key. They control who else gets access.

**Doctor view (shared vault key):** Read-write access for collaboration. The doctor gets a vault key (not a share token) so they can add notes and push changes. The dual branch model handles concurrent edits.

**Published summary (share token):** Read-only. The patient can create a share token to give a published snapshot to other parties (specialist referral, insurance, family).

### LLM Integration

The LLM step is optional and explicit. The patient sees what data is being sent, which model is being used, and what it costs. The LLM response is stored in the vault as a separate file, clearly labelled as AI-generated and pending clinician review.

The clinician's sign-off is a separate commit that validates (or corrects) the LLM output. This creates a clean audit trail: AI suggestion, then human validation.

## Connection to Identity Research

This MVP immediately hits the identity gap described in today's identity research brief. The patient needs a way to create a vault and store the key. The doctor needs a way to receive and store a shared vault key.

For the MVP, this can be handled simply: the patient gets a vault key displayed on screen (copy/save it), the doctor receives it via a secure channel (in-person, encrypted message). For the production version, this is where the Cognito or OAuth integration from the identity brief becomes essential.
