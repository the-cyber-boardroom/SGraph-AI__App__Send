## The Gap

SG/Send does everything except store vault keys.

We have encryption. We have vaults. We have collaboration. We have the dual branch model. We have file sharing, version control, agentic workflows. But when a partner asks "how do I integrate this into my application?", there is always this moment: where do the vault keys go?

Right now, vault keys live in the user's head, in their clipboard, or on their laptop. That works for individual use, but it breaks down for enterprise integration, for multi-user applications, and frankly even for day-to-day use when you are collaborating on a growing number of vaults.

This brief requests a research deliverable: map the identity and key storage landscape, produce reference implementations for the top options, and give partners a clear answer to "how do I store vault keys in my existing infrastructure?"

## What We Need

A structured research document covering 5-6 identity/key storage technologies, each with:

- How it works (architecture overview)
- How vault keys would be stored (as user attributes, secrets, or derived credentials)
- Integration pattern (OAuth flow, API calls, SDK)
- Pros and cons for our use case
- Level of effort to build a reference implementation
- Whether it supports a master vault key pattern (one stored key that derives all others)

## The Master Vault Key Pattern

The most interesting architecture is the master vault key: the user authenticates via their chosen identity provider, and that authentication unlocks (or derives) a single master key. All individual vault keys are then derived from the master key using deterministic derivation (PBKDF2, HKDF, or similar, consistent with the existing deterministic-safe-paths pattern using PBKDF2 with 600k iterations and unique salts per context).

This means:

- The user remembers one credential (their identity provider login)
- One master key is stored in the identity provider's user attributes or secret store
- All vault keys are derived deterministically from that master key
- Losing the master key means losing access to all vaults (this is the trade-off of zero-knowledge: we cannot recover it)

## Recommended Starting Point: Cognito Reference Implementation

Build a reference implementation with AWS Cognito as the first proof of concept:

1. User logs in via Cognito (Google OAuth or email/password)
2. On first login, generate a master vault key and store it as an encrypted Cognito user attribute
3. When the user accesses a vault, derive the vault-specific key from the master key
4. Wrap the existing SG/Send UI so that vault access is seamless after login

This gives us a concrete, working example that we can demo to partners. The same pattern can then be replicated for GCP, Azure, and others.

## The Real-World Problem

This is not abstract. I am already collaborating on multiple vaults, and the keys mainly exist on my laptop. If I lose that laptop, I lose access. If I want to switch devices, I have to manually transfer keys. Partners evaluating SG/Send for enterprise use will hit this wall immediately.

Solving this is a precondition for enterprise adoption.
