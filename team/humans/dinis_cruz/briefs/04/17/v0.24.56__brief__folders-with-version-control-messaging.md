## Context

Yesterday's dev brief (`v0.24.56__dev-brief__folders-with-version-control-vault-upload.md`) specified the technical implementation: vault-backed upload mode in the main SG/Send UI, batched commits, instant share links, beta toggle. This brief captures the positioning and messaging side of the same concept: how we talk about this to users who have never used version control.

## The Proof Point

The vault collaboration model is now working in practice. In a recent session, four different clients were making changes to the same vault simultaneously: a Claude Code session, a Perplexity session, a local CLI (SGit), and the vault web UI accessing the vault directly without cloning. They all worked. Commits landed on clone branches. Push/Pull synchronised changes. No conflicts.

This is not theoretical anymore. It works.

## The One-Line Description

**Distributed, encrypted folders with version control.**

That is the product. That is the message. Every word earns its place:

- **Distributed:** accessible from anywhere, any client, any device. Not locked to one server.
- **Encrypted:** zero-knowledge, client-side AES-256-GCM. The server never sees plaintext.
- **Folders:** the concept everyone already understands. Not "vaults," not "repositories," not "encrypted object stores." Folders.
- **Version control:** every change is tracked. Go back, go forward, see the history, create branches. The power of Git, without requiring anyone to learn Git.

## The Audience

This messaging is aimed at the vast majority of professionals who have never used version control. They use shared drives. They use Dropbox. They use WeTransfer. They email files to themselves. They name files `report_final_v3_FINAL_actually_final.docx`.

For this audience, "version control" is not an intimidating technical concept. It is a relief. It means: you cannot lose your work. You can always go back. You can see who changed what. You can have multiple versions without multiple files.

The key is to never mention Git, branches, commits, or any developer terminology in the user-facing messaging. Behind the scenes, that is exactly what powers it. In front of the user, it is just folders that remember everything.

## Website Positioning

The next version of the website should feature this prominently. Suggested framing:

**Hero section:** "Share folders. Track changes. Stay encrypted."

**Three-column feature block:**

| Feature | Message |
|---------|---------|
| Encrypted | Your files are encrypted before they leave your device. We cannot read them. Nobody can. |
| Version controlled | Every change is tracked. Go back to any version. See the full history. Never lose work. |
| Shareable | Share a link. That is it. No accounts, no sign-ups, no apps to install. |

**Comparison section:** "Like Dropbox, but encrypted. Like Git, but for everyone. Like WeTransfer, but with history."

## What NOT to Say

- Do not say "vault" in user-facing copy. Say "folder" or "shared folder."
- Do not say "commit." Say "version" or "saved change."
- Do not say "branch." Say "your copy" or "your workspace."
- Do not say "push/pull." Say "sync" or "share your changes."
- Do not say "clone." Say "open" or "access."
- Do not say "zero-knowledge encryption." Say "we cannot read your files."

The developer vocabulary stays in the documentation, the API docs, and the CLI. The user vocabulary stays in the UI, the website, and the marketing.

## Technical Vocabulary Mapping

For internal reference, here is how the technical concepts map to user-facing language:

| Technical (internal) | User-facing |
|---------------------|-------------|
| Vault | Folder / Shared folder |
| Commit | Saved version |
| Branch | Your copy |
| Push | Share your changes |
| Pull | Get latest changes |
| Clone | Open / Access |
| Vault key | Access link / Share link |
| Share token | View-only link |
| SGit | (not exposed in UI) |
| Merge conflict | "Someone else changed this file too" |

## The Competitive Positioning

This is genuinely differentiated. No other product in the market offers all three together:

- WeTransfer: sharing, no version control, no encryption
- Dropbox: sharing, basic version history, not zero-knowledge encrypted
- Git/GitHub: version control, not encrypted, not for non-developers
- Tresorit/Boxcryptor: encrypted, no meaningful version control
- SG/Send: encrypted + version control + sharing, accessible to non-developers

"Folders with version control" is the positioning that captures this unique intersection.
