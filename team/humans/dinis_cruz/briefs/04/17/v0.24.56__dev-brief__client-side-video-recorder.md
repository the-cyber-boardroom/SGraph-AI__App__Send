## Origin

During a demo to a potential healthcare partner, they asked: "How can we record the clinician's video analysis without GDPR problems?" They were considering Loom and similar tools, but every one of them sends the video to third-party servers, which immediately triggers GDPR data processing obligations.

That got the thinking going: we already have the plumbing for this. Browser APIs support camera access and screen capture. We have client-side encryption. We have SG/Send for file storage and vaults for versioned storage. The gap is just the recording UI.

## What It Does

A new tool on tools.sgraph.ai that records video (camera and/or screen), encrypts it client-side, and saves it directly to an SG/Send file, folder, or vault. The video data never touches any server in unencrypted form. Zero GDPR data processing exposure.

## Two Recording Modes

### Mode 1: Camera Recording

Uses the browser MediaDevices API (`navigator.mediaDevices.getUserMedia()`) to access the camera and microphone. Records video locally. On stop, encrypts the recording and uploads to SG/Send.

### Mode 2: Screen Recording

Uses the browser Screen Capture API (`navigator.mediaDevices.getDisplayMedia()`) to capture the screen (full screen, application window, or browser tab). Records locally. On stop, encrypts and uploads.

Both modes can be combined (camera + screen picture-in-picture) if the browser supports it.

## Output Packaging

The tool should offer three output options:

**Simple file:** Just the video, encrypted and uploaded as a single SG/Send transfer. Returns a three-word token. Recipient downloads and plays.

**Folder with extras:** The video plus optional extras: a thumbnail image, a set of periodic screenshots (e.g. one every 30 seconds), a metadata JSON file (duration, resolution, recording date, file size). This gives the recipient a browsable summary before downloading the full video.

**Vault:** The video committed to a vault. This enables the versioned workflow: record multiple takes, keep the history, share the latest. The healthcare use case benefits from this: each consultation recording is a commit, the full history is preserved, and the vault's encryption means the data governance is clean.

## Technical Notes

**Recording format:** Use MediaRecorder API. Default to WebM (VP8/VP9 + Opus), which is natively supported by Chrome. MP4 (H.264 + AAC) would require transcoding, which is heavier but more universally playable. Start with WebM. Consider offering MP4 as an option if there is demand.

**File size and chunking:** Video files can be large. For recordings over 4MB, use the existing large file support (presigned S3 URLs, already shipped). For very long recordings, consider chunking the MediaRecorder output into segments and uploading progressively (similar to the batched commit pattern from the vault upload brief).

**Client-side only:** At no point does unencrypted video data leave the browser. The recording happens locally via MediaRecorder. Encryption happens locally via Web Crypto API (AES-256-GCM). Only ciphertext is uploaded. This is the entire GDPR value proposition.

**Existing components to reuse:** `sg-send.js` for file upload and encryption, vault components for vault save, the gallery component for thumbnail display, the video playback component (from 08 Apr brief `v0.20.38__dev-brief__video-playback-component.md`) for preview and playback.

## The GDPR Angle

This is not just a feature. It is a compliance argument.

For healthcare, legal, financial services, and any sector handling sensitive personal data, the question "where does the recording go?" has significant regulatory implications. With Loom, Zoom cloud recordings, or any SaaS recorder, the answer is: to someone else's servers, in a jurisdiction you may not control, processed by a company whose data practices you must audit.

With this tool, the answer is: nowhere. The recording stays in the browser until it is encrypted. The encrypted data goes to SG/Send's storage. Nobody (including us) can decrypt it. The data processing footprint is zero.

This is a concrete, demonstrable selling point for the healthcare partner and any similar regulated-industry prospect.

## The Screen Recorder for Internal Use

The screen recording mode is also immediately useful internally: recording demos, capturing workflow walkthroughs, creating tutorial content. Currently these require Loom or OBS, both of which are overkill for quick captures. A built-in screen recorder that saves directly to a shareable SG/Send link is a tool we would use daily.
