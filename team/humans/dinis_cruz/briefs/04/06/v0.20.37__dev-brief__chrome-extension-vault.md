# Chrome Extension with Vault Communication: Capturing Data from Websites

**version** v0.20.37
**date** 06 Apr 2026
**from** Human (project lead)
**to** Developer (lead), Architect
**type** Dev brief

---

## The Problem

Some data we need is only available in website UIs, not via APIs. The immediate case: OpenRouter has a really good transaction history on their website, but the API does not seem to expose the same level of detail. We need that data to understand usage patterns, debug billing, and feed into our admin dashboard.

The old approach (sharing cookies, passing session tokens between scripts) never felt correct. It was fragile, hard to secure, and broke when sessions expired.

## The Solution: Chrome Extension + Vault

A Chrome extension that:

1. Runs on a specific website (e.g. OpenRouter)
2. Executes specific commands on that site (API calls, DOM scraping, or both)
3. Captures the data
4. Writes the data to a vault

The vault is the communication channel between the extension and everything else (agents, tools, dashboards). The extension writes. The agents read. No cookies shared. No sessions passed around. The vault handles the encrypted transport.

## How It Works

### Step 1: User Installs the Extension

The extension is site-specific (or has per-site configurations). For OpenRouter, it knows which API endpoints to call and which page elements contain the data we need.

### Step 2: User Navigates to the Site

The user is already logged into OpenRouter (or whatever site). The extension detects it is on the target site and activates.

### Step 3: Extension Captures Data

The extension calls the site's internal APIs (using the user's existing session cookies, which the extension has access to as a browser extension on that domain). It can also read DOM elements if needed.

Example for OpenRouter:
- Fetch the full transaction history (including details not available via the public API)
- Fetch per-key usage breakdowns
- Capture model pricing data
- Read any dashboard data visible to the logged-in user

### Step 4: Extension Writes to Vault

The captured data is pushed to a vault using the SGit/vault browser primitives (from the 3 April brief). The extension has a configured vault key. It writes the data as JSON files:

```
vault/openrouter/
  transactions/2026-04-06.json
  usage/per-key-breakdown.json
  models/pricing-snapshot.json
```

### Step 5: Agents Read from Vault

The Librarian, the admin dashboard, the OpenRouter integration tools all read from the vault. They do not know (or care) that the data came from a Chrome extension. They just see files in a vault.

## Architecture

```
Chrome Extension
  │
  ├── Site-specific scripts (per website)
  │     ├── openrouter.js (calls OpenRouter internal APIs)
  │     ├── github.js (future: capture repo stats)
  │     └── cloudwatch.js (future: capture AWS metrics)
  │
  ├── Vault writer (shared)
  │     └── Uses sg-vault browser primitives to commit and push
  │
  └── Configuration
        ├── Vault key (configured once by the user)
        ├── Site list (which sites to activate on)
        └── Schedule (manual trigger, or periodic capture)
```

## Why This Is Better Than Cookie Sharing

| Approach | Cookie Sharing | Chrome Extension + Vault |
|---|---|---|
| Security | Cookies exported and passed around (risky) | Cookies stay in the browser, never leave |
| Session management | Sessions expire, need manual refresh | Extension uses the live session |
| Data transport | Custom scripts, manual copy, fragile | Vault (encrypted, version controlled) |
| Automation | Hard to automate reliably | Can be triggered manually or on schedule |
| Multi-site | Separate scripts per site | One extension, per-site configurations |

## Execution Modes

| Mode | How It Works | When To Use |
|---|---|---|
| Manual | User clicks the extension button, data is captured | One-off data grabs, debugging |
| Periodic | Extension captures data every N minutes while the page is open | Monitoring, continuous data collection |
| Event-driven | Extension captures data when specific page events occur (e.g. new transaction) | Real-time capture |

## Future: Beyond OpenRouter

Once the extension framework exists, adding new sites is just a new site-specific script:

- **GitHub:** Capture repo stats, PR data, commit history not available via public API
- **AWS Console:** Capture CloudWatch metrics, S3 usage, Lambda invocation data
- **LinkedIn:** Capture post analytics, connection stats
- **Any internal tool:** Capture data from corporate dashboards that have no API

Each site script writes to the vault. The agents read from the vault. The pattern is always the same.
