# Multiple Levels of Intelligence: From Frontier Models to Browser-Based AI

I am now working with at least five distinct tiers of AI capability simultaneously. This is new. Six months ago, the question was "which model should I use?" Now the question is "which model should I use for this specific task, at this specific cost, at this specific moment?"

This is not a technology problem. It is a management problem. And it maps surprisingly well to how you manage teams of people with different skill levels, different costs, and different availability.

## The Five Tiers

### Tier 1: Frontier Models (Claude, GPT-4, Gemini)

The most capable. The most expensive. The slowest (in terms of cost per token, not necessarily latency). These are the models that can reason about complex architecture, write nuanced articles, and handle ambiguous instructions.

Use for: complex analysis, strategic decisions, nuanced writing, tasks where quality matters more than cost. The equivalent of your most senior team member: you do not ask them to format spreadsheets.

### Tier 2: Strong API Models (Claude Sonnet, Gemma 4 31B via OpenRouter)

Very capable, significantly cheaper. Good at structured tasks, code generation, document analysis. These can handle most of what Tier 1 does, with occasional quality drops on the most subtle tasks.

Use for: code generation, document summarisation, infographic prompts, structured JSON output. The equivalent of a strong mid-level team member who handles 80% of the work.

### Tier 3: Local Large Models (Gemma 4 31B/26B via Ollama)

Same capability as Tier 2, but running locally. Zero cost per token. Slower inference (depends on hardware). Requires a decent GPU or Apple Silicon.

Use for: anything Tier 2 does, when cost matters more than speed. Offline work. Privacy-sensitive tasks. Experimentation and iteration (run it 50 times without worrying about the bill). The equivalent of an in-house team: slower to respond, but no invoicing.

### Tier 4: Local Small Models (Gemma 4 E4B, Phi, small Mistral via Ollama)

Less capable but fast and free. Good at focused tasks: classification, extraction, formatting, simple Q&A. Not good at complex reasoning or nuanced generation.

Use for: the Librarian's daily indexing, code comment processing, metadata extraction, search ranking, simple transformations. The equivalent of a reliable junior team member: give them clear instructions and they deliver consistently.

### Tier 5: Browser-Based Models (WASM, WebGPU, tiny models)

The smallest, fastest, and most constrained. Running entirely in the browser. No network needed. Useful for real-time tasks that need instant response.

Use for: auto-complete, spell checking, simple classification (is this a photo or a document?), real-time search ranking of local results, basic text processing. The equivalent of a script or a rule engine: fast, predictable, limited.

## The Refactoring Principle

Start with Frontier. Refactor down.

When building a new capability:

1. **Prototype with Tier 1** (Claude Opus, GPT-4). Validate the approach. Get the quality right. Do not worry about cost.
2. **Validate with Tier 2** (Sonnet, Gemma 4 31B API). Does the same prompt produce acceptable results at lower cost? If yes, move down. If no, refine the prompt until it does.
3. **Test with Tier 3** (local Ollama). Does it work offline? Is the quality acceptable for this task? If yes, this becomes the default for this task.
4. **Optimise with Tier 4** (small local models). Can a simpler model handle this task with good-enough quality? If yes, use the small model for the recurring version and save the larger models for edge cases.
5. **Push to Tier 5** (browser) if the task is simple enough and needs to be instant.

The result: Frontier models handle only the tasks that genuinely need them. Everything else runs at the appropriate tier. Cost drops dramatically. Offline capability increases. The system becomes more resilient (if the API is down, Tiers 3-5 still work).

## The People Analogy

This is exactly how you manage a team:

- You do not ask your CTO to write unit tests
- You do not ask your junior developer to design the system architecture
- You have senior people review work, not produce all of it
- You have junior people handle the volume, with seniors handling the exceptions

The same principle applies to LLM tiers:

- Frontier models review and refine (like a senior reviewing a junior's work)
- Mid-tier models produce the bulk of the output
- Small models handle the recurring, predictable tasks
- Browser models handle the instant, real-time tasks

## What This Means for Our Architecture

### Every Tool Should Support Multiple Tiers

The Model Chooser component (from the LLM chat brief) becomes critical. Every tool should:

- Default to the most cost-effective tier for its task type
- Allow the user to upgrade to a higher tier if needed
- Show the cost difference ("Tier 2: ~£0.02, Tier 1: ~£0.15 for this task")
- Remember which tier works best for each task (learn from usage)

### Context Management Changes

With Frontier models via Claude Web, the context is managed by the platform (conversation history, system prompt, everything). Moving to API changes this:

- We manage the context (the one-shot feedback loops brief)
- We control what goes in each request (the bundle architecture)
- We can optimise: send less context to cheaper models, full context to Frontier
- We can chain: small model extracts facts, large model reasons about them

This is the direction of travel for Claude anyway (limits on context windows, push toward API usage). Building the infrastructure now means we are ready.

### Scheduled Tasks by Tier

The Claude scheduled tasks (Librarian, QA) currently use Frontier models. Many of these tasks could run on Tier 3 or Tier 4:

| Task | Current Tier | Could Be | Savings |
|---|---|---|---|
| Librarian daily index | Tier 1 (Claude) | Tier 3 (local Gemma 4 31B) | 100% (free) |
| QA smoke test analysis | Tier 1 (Claude) | Tier 3 (local Gemma 4 31B) | 100% (free) |
| Code comment processing | Tier 1 (Claude) | Tier 4 (local small model) | 100% (free) |
| Metadata extraction | Tier 1 (Claude) | Tier 4 (local small model) | 100% (free) |
| Daily infographic | Tier 2 (API) | Tier 2 (API) | Already optimised |
| Strategic analysis | Tier 1 (Claude) | Tier 1 (stays) | N/A (needs the quality) |

### The Two-LLM Pattern Everywhere

The two-LLM pattern from the one-shot feedback loops brief (Frontier for answers, cheap model for pruning) generalises:

- **Generate + Review:** Tier 3 generates, Tier 1 reviews
- **Extract + Reason:** Tier 4 extracts facts, Tier 2 reasons about them
- **Draft + Polish:** Tier 3 drafts, Tier 1 polishes
- **Index + Query:** Tier 4 indexes continuously, Tier 1 handles complex queries

This splits every expensive task into a cheap part and an expensive part. The cheap part runs at volume. The expensive part runs only when needed.

## The Business Implication

For our users with £5 of OpenRouter credits:

- If every task uses Tier 1, £5 lasts a few dozen tasks
- If we route intelligently (Tier 2 for most tasks, Tier 1 only when needed), £5 lasts hundreds of tasks
- If we add local model support (Tier 3-4), many tasks are free

Intelligent routing extends the user's credits. Longer credits mean more usage. More usage means more engagement. More engagement means higher conversion when credits run out.

This is not just a cost optimisation. It is a user experience optimisation. The user does not care which tier produced the result. They care that it was good and that their credits lasted.
