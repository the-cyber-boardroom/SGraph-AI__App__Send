## The Concept

A Docker-based Playwright service, deployable as an API, that agents and scripts can invoke to run browser automations. Vaults solve the session management problem: encrypted cookies and credentials stored in vaults, loaded per-session, destroyed after use.

## Why Docker, Not Lambda

Playwright in Lambda is problematic (Chrome binary size, cold start times, execution limits). A Docker container (deployed via ECR to ECS/Fargate, or run locally) is more practical:

- Chrome runs naturally in Docker
- No size or time limits
- Can batch multiple requests per session
- Same container runs locally for development and testing

The container should be **built once** because the primitives (Playwright, Chrome, the API server) are stable. What changes is the automation scripts, which are loaded at runtime (from vaults, from S3, or passed in the API request).

## The API

A simple HTTP API exposed by the Docker container:

### POST /session/create

Create a new browser session. Optionally load cookies/credentials from a vault.

```json
{
  "vault_key": "optional-vault-key-for-credentials",
  "credential_path": "/sessions/openrouter/cookies.json",
  "headless": true
}
```

Returns a session ID. The container launches a Chrome instance with the loaded credentials.

### POST /session/{id}/execute

Run an automation script in the session.

```json
{
  "script": "await page.goto('https://openrouter.ai/activity'); ...",
  "timeout_seconds": 30
}
```

Returns the script output (scraped data, screenshots, etc.).

### POST /session/{id}/save-state

Save the current session state (cookies, localStorage) back to the vault.

```json
{
  "vault_key": "vault-key",
  "save_path": "/sessions/openrouter/cookies.json"
}
```

This captures login state after a manual or automated login, so the next session can resume without re-authenticating.

### DELETE /session/{id}

Destroy the session. Close Chrome. Delete all temporary files. Clean execution environment.

## The Session Lifecycle

```
1. Create session (load credentials from vault if available)
2. Execute one or more automation scripts
3. Capture results (data, screenshots, state)
4. Save state back to vault (if needed)
5. Destroy session (clean everything)
```

Every session starts clean. Every session ends clean. No persistent state on the container between sessions. The vault is the only persistent storage.

## Vault Solves the Credential Problem

The long-standing challenge with browser automation: how to handle login credentials and session cookies securely.

| Old Approach | New Approach (Vault) |
|---|---|
| Cookies stored in plain text files | Cookies encrypted in vault |
| Credentials in environment variables | Credentials encrypted in vault |
| Session state lost between runs | Session state saved to vault after each run |
| Hard to share sessions between agents | Any agent with the vault key can load the session |
| No audit trail | Vault commit history shows when credentials were used/updated |

### The Login Flow

For sites that require authentication:

1. First time: run a login automation (or manual login) that captures the session cookies
2. Save cookies to vault: `vault/sessions/{site}/cookies.json`
3. Subsequent runs: load cookies from vault, skip login
4. If cookies expire: re-run the login automation, update the vault

The cookies are encrypted at rest in the vault. Only agents with the vault key can load them. The vault commit history shows when the cookies were last updated.

## Use Cases

### QA Smoke Tests

The daily QA scheduled task invokes the Playwright API to:
- Load the SG/Send website
- Run smoke tests (upload, download, browse, share)
- Capture screenshots of each step
- Save results to the QA vault

### Data Capture from External Sites

Agents invoke the Playwright API to:
- Log into OpenRouter (using vault-stored cookies)
- Capture transaction history
- Save to the data vault

This complements the Chrome extension: the extension captures data when the user is browsing. The Playwright API captures data on a schedule without the user being present.

### Content Generation

Agents invoke the Playwright API to:
- Take screenshots of the SG/Send website for documentation
- Capture visual diffs between versions
- Generate screenshot-based reports

### Integration Testing

Test the full user flow across multiple services:
- Upload a file via the SG/Send website
- Verify it appears in the vault
- Share it via a simple token
- Access the share link in a new browser session
- Verify the file is downloadable

## Local and Production Parity

The same Docker container runs:
- **Locally:** `docker run sg-playwright-api` for development and testing
- **On CI:** As part of the QA pipeline
- **On AWS:** Deployed via ECR to ECS/Fargate for production automation
- **In Claude Code:** The scheduled tasks can invoke the API (we already solved Chromium in Claude)

The API is the same everywhere. The automation scripts are the same. Only the endpoint URL changes.

## Clean Execution Per Session

Every session:
- Gets a fresh Chrome profile (no cross-session data leakage)
- Runs in an isolated temporary directory
- Has all processes terminated on session destroy
- Has all temporary files deleted on session destroy

For Lambda-like isolation without Lambda's limitations. The container can batch multiple sessions sequentially but each session is fully isolated.
