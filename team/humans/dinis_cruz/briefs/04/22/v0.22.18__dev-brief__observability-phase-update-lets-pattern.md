## Refactoring: Extract Core Components into an API

The current implementation works but the components are entangled. Before adding more connectors and dashboards, extract the reusable pieces into a clean API layer.

### What Needs Extracting

**Index management API:**
```
POST /indices/create          -> Create an index with a specified mapping
POST /indices/import          -> Import data from S3/vault/file into an index
GET  /indices/list            -> List all indices with doc counts and sizes
DELETE /indices/{name}        -> Delete an index
POST /indices/consolidate     -> Merge multiple small files into one before import
```

**Dashboard management API:**
```
POST /dashboards/create       -> Create a dashboard programmatically (from JSON spec)
GET  /dashboards/list         -> List all dashboards
POST /dashboards/screenshot   -> Take screenshot of a specific dashboard (via Playwright)
POST /dashboards/export       -> Export dashboard definition for re-import
POST /dashboards/import       -> Import a dashboard definition
```

**Connector API:**
```
POST /connectors/cloudfront/import   -> Import CloudFront logs for a date range
POST /connectors/playwright/import   -> Import Playwright instance logs
POST /connectors/sgsend/import       -> Import SG/Send server logs
POST /connectors/{type}/import       -> Generic connector pattern
```

Each connector follows the same pattern: point it at a data source, it reads, transforms, and loads into Elastic. The connector abstracts the source-specific parsing.

### Why Refactor Before Expanding

The CloudFront connector works. But if we build the Playwright connector, the SG/Send connector, and the vault activity connector with the same ad-hoc approach, we will have four separate implementations with duplicated logic. Extract the pattern now. Every future connector becomes simpler.

## The LETS Pattern: Load, Extract, Transform, Save

Every data processing operation follows this pattern:

**Load:** Read data from a source (S3, vault, CloudFront logs, Playwright logs, wherever).

**Extract:** Parse the raw data into structured events. Pull out metadata. Break URLs into segments. Identify event types. Normalise timestamps.

**Transform:** Enrich, aggregate, filter, deduplicate. Multiple passes, each reducing complexity:
- Pass 1: Consolidate (300 separate log files into one NDJSON file)
- Pass 2: Enrich (break URLs into path segments, add geo data to IPs, tag event types)
- Pass 3: Aggregate (requests per minute, error rates, unique URLs)

**Save:** Write the output to storage (S3, vault, or Elastic index). Every intermediate result is saved. Nothing is lost.

Each LETS pass is idempotent: re-running it on the same input produces the same output. Each pass's output is stored alongside the input. You can re-run any pass without re-running the ones before it.

### Concrete Example: CloudFront Logs

```
LOAD: Read CloudFront access logs from S3 (separate .gz files, one per distribution per hour)
    |
    v
EXTRACT (Pass 1 - Consolidate):
    - Decompress all .gz files for the date range
    - Concatenate into one NDJSON file
    - Save as cloudfront-raw-{date}.ndjson
    |
    v
TRANSFORM (Pass 2 - Enrich):
    - Parse each line into structured fields (timestamp, URL, status, bytes, etc.)
    - Break URL into segments: /api/v1/vault/status -> ["api", "v1", "vault", "status"]
    - Add URL segment array as a field (enables searching for "api" to find all API calls)
    - Tag event types (page view, API call, asset load, etc.)
    - Save as cloudfront-enriched-{date}.ndjson
    |
    v
TRANSFORM (Pass 3 - Aggregate):
    - Requests per minute by URL pattern
    - Status code distribution
    - Top URLs by request count
    - Unique visitors (by IP hash)
    - Bytes transferred by content type
    - Save as cloudfront-signals-{date}.json
    |
    v
SAVE to Elastic:
    - Import enriched data as "cloudfront-enriched-{date}" index (detailed analysis)
    - Import signals as "cloudfront-signals-{date}" index (dashboard summaries)
```

The URL segmentation example is important: by breaking `/api/v1/vault/status` into an array `["api", "v1", "vault", "status"]` and storing it as an Elastic array field, you can search for "vault" and immediately get all vault-related API calls without writing complex URL pattern queries.

## Dashboard Abstraction

Decouple dashboard creation from data analysis. The dashboards should be templates that work with any date's data, not hand-built for a specific import.

### Dashboard Templates

Each dashboard is a JSON specification that can be:

- Created programmatically (via the dashboard API)
- Saved and version-controlled (in a vault or in the AMI)
- Applied to any index that follows the expected mapping
- Screenshot-captured via Playwright for reporting

### Experiments Needed

Before building production dashboards, run experiments:

1. **Lens visualisations:** Kibana Lens is the modern visualisation builder. Add Lens support to the programmatic dashboard creation. Lens visualisations are more flexible and performant than the legacy visualisation types.

2. **Dashboard-as-code:** Can we define a complete dashboard (with all visualisations, filters, and layout) as a single JSON file that an agent can generate? This is the path to "Claude creates a dashboard from a natural language description."

3. **Cross-index dashboards:** A single dashboard that pulls from multiple indices (CloudFront + Playwright + SG/Send). This is the "unified health overview" dashboard.

4. **Time-range templates:** Dashboards that automatically scope to "last 24 hours" or "last 7 days" based on what data is loaded.

## What Comes Next for CloudFront

The CloudFront connector is working at the basic level. Next steps:

1. **Consolidate log files** (Pass 1): currently loading hundreds of small files per analysis. Consolidate into one file per date first.

2. **URL breakdown and enrichment** (Pass 2): segment URLs, tag event types, identify API calls vs page loads vs asset fetches.

3. **Content type analysis:** Break down by what is being served: HTML pages, API responses, vault data, static assets, infographics.

4. **Vault activity extraction:** From the URL patterns, extract vault-specific events: vault created, vault opened, file uploaded, file downloaded, share token used.

5. **Create dedicated dashboards:** CloudFront traffic overview, API usage, vault activity, error analysis. Each as a reusable template.
