## The Philosophy

Elastic is a projection of data, not a destination for data.

Data flows through multiple stages, each reducing complexity and extracting signal. At every stage, the raw input is preserved. Nothing is lost. Elastic and Kibana sit at the end as a visualisation layer: interactive for discovery (Explorer phase), then automated screenshots for reporting (Villager/Town Planner phase).

The ephemeral infrastructure brief (21 Apr) established that Elastic spins up on demand and is destroyed after use. This brief defines what happens between "data is generated" and "data appears in a Kibana dashboard."

## The Core Principle: Elastic Is Never the Data Store

At no point should Elastic be the source of truth for any data. It is always a projection: a transformed, indexed, visualisable copy of data that lives somewhere else.

If Elastic is destroyed (which it will be, regularly, because the infrastructure is ephemeral), no data is lost. The raw data is in S3 or a vault. The transformed data is in S3 or a vault. Elastic can be rebuilt from any of these at any time.

In the edge case where data is sent directly to Elastic (e.g. a real-time log shipper), we must extract it back out and store it externally before the instance is destroyed. But the preferred pattern is: data goes to storage first, Elastic loads from storage.

## The Pipeline Stages

```
Stage 0: Raw Capture
    |  (logs generated by services, stored as-is)
    v
Stage 1: Security Cleaning
    |  (remove/hash secrets, API keys, PII)
    v
Stage 2: Structured Extraction
    |  (parse logs into structured events, extract meaning)
    v
Stage 3: Signal Reduction
    |  (aggregate, deduplicate, compute metrics)
    v
Stage 4: Elastic Projection
    |  (load into Elastic, create indices)
    v
Stage 5: Kibana Visualisation
    |  (dashboards, charts, tables)
    v
Stage 6: Screenshot Capture
       (Playwright takes screenshots, saves to vault)
```

Each stage reads from the previous stage's output and writes its own output. Every stage's output is stored (S3 or vault). You can re-run any stage without re-running the ones before it. You can project Stage 2 output into Elastic for detailed exploration, or Stage 3 output for high-level dashboards. Multiple views of the same data, different levels of detail.

## Stage 0: Raw Capture

### What Gets Captured

Taking the Playwright EC2 instance as the concrete example:

| Log Source | What It Contains |
|------------|-----------------|
| Instance boot logs | EC2 startup, Docker container launch, service initialisation |
| MITM proxy logs | Every HTTP request/response through the proxy (full traffic) |
| Playwright execution logs | Browser actions, screenshots taken, sequences executed |
| FastAPI access logs | API calls to the Playwright service |
| System stats | CPU, memory, disk, network (Prometheus-style metrics) |

### How to Capture (Keep It Simple)

The current approach (four sidecars capturing logs) is over-engineered. The capture mechanism should be as simple as possible:

- Tail the log files
- Send them to S3 (or write to a vault)
- Done

No complex log shipping infrastructure. No Fluentd, no Logstash, no Filebeat at this stage. Just: read log, write to storage. A simple Python script or shell command that runs alongside the service.

If CloudWatch is the path of least resistance (the EC2 instance already has CloudWatch agent capabilities), use it. But the data must also land in S3 (or a vault) in raw form, because CloudWatch is not our long-term storage.

### Storage

Raw logs go to S3 in a structured path:

```
s3://sg-observability-raw/
    playwright/
        {instance-id}/
            {date}/
                boot.log
                mitm-proxy.log
                playwright.log
                fastapi-access.log
                stats.json
```

Or to a vault (which provides encryption and version control but adds slight complexity). For high-volume raw logs, S3 is probably the pragmatic choice. For sensitive logs (containing request/response bodies), a vault is better.

## Stage 1: Security Cleaning

Before any log leaves the raw capture bucket for further processing:

- **Hash or remove API keys** that appear in log lines
- **Hash or remove bearer tokens** and session identifiers
- **Remove or mask PII** (email addresses, IP addresses if not needed)
- **Flag but preserve** anything that looks like a secret (do not silently delete; flag it so we know there is a leak to fix)

This stage produces a "clean" copy of the logs in a separate location. The raw logs are preserved (they may be needed for incident investigation where the secret or PII is relevant), but all downstream processing uses the cleaned version.

```
s3://sg-observability-clean/
    playwright/
        {instance-id}/
            {date}/
                boot.clean.log
                mitm-proxy.clean.log
                ...
```

## Stage 2: Structured Extraction

Parse the cleaned logs into structured events. Each log line becomes a JSON event with typed fields:

```json
{
  "timestamp": "2026-04-22T10:15:32.456Z",
  "source": "mitm-proxy",
  "event_type": "http_request",
  "method": "GET",
  "url": "https://sgraph.ai/api/v1/vault/status",
  "status_code": 200,
  "duration_ms": 45,
  "response_size_bytes": 1234,
  "instance_id": "i-abc123"
}
```

This is where we extract all the meaning we can from the raw logs. Each log source has its own parser. The output is a set of structured JSON files (or NDJSON for bulk loading into Elastic).

```
s3://sg-observability-structured/
    playwright/
        {instance-id}/
            {date}/
                http-requests.ndjson
                browser-actions.ndjson
                api-calls.ndjson
                system-stats.ndjson
```

## Stage 3: Signal Reduction

Aggregate and compute metrics from the structured events:

- Requests per minute by endpoint
- Error rates by service
- P50/P95/P99 response times
- Unique URLs accessed per session
- Files created/accessed per hour
- Vaults created/opened per hour
- Playwright sequences executed per day

This produces compact summary data that can drive high-level dashboards. The structured events from Stage 2 are still available for drill-down.

```
s3://sg-observability-signals/
    {date}/
        playwright-summary.json
        sg-send-summary.json
        vault-activity-summary.json
```

## Stage 4: Elastic Projection

Load data into Elastic. Which stage's data gets loaded depends on the use case:

| Use Case | Load From | Why |
|----------|-----------|-----|
| Debugging a specific issue | Stage 2 (structured events) | Need full detail, specific timeframes |
| Daily health dashboard | Stage 3 (signals) | Compact, fast to load, key metrics only |
| Traffic analysis | Stage 2 (HTTP requests only) | Full request/response detail |
| Discovery / exploration | Stage 1 or 2 | Maximum detail for unknown-unknowns |

The Elastic instance is ephemeral. Import the data you need, explore, export any new insights, destroy the instance.

## Stage 5: Kibana Dashboards

Pre-configured dashboards baked into the AMI:

| Dashboard | Data Source | Purpose |
|-----------|-----------|---------|
| Health Overview | Stage 3 signals | Green/amber/red status for each service |
| Traffic Analysis | Stage 2 HTTP requests | URLs, status codes, response times, error rates |
| Vault Activity | Stage 2 vault events | Vaults created, files added, access patterns |
| Playwright Sessions | Stage 2 browser actions | Sequences executed, screenshots taken, errors |
| System Resources | Stage 2 system stats | CPU, memory, disk, network over time |

These dashboards are pre-built and saved as part of the Elastic AMI. When the instance starts and data is loaded, the dashboards populate immediately.

## Stage 6: Screenshot Capture

The final stage: Playwright takes screenshots of Kibana dashboards and saves them to a vault.

```
Playwright -> http://localhost:5601/dashboard/health-overview -> screenshot.png
Playwright -> http://localhost:5601/dashboard/traffic-analysis -> screenshot.png
...
All screenshots committed to a vault: "observability-report-{date}"
```

This is the Villager/Town Planner output. The interactive Kibana exploration is the Explorer phase (human poking around, discovering patterns). The automated screenshots are the production output: a daily observability report as a vault of dashboard images, shareable via token, browsable via page layout.

The progression:

- **Explorer:** Spin up Elastic, load data, click around Kibana interactively
- **Villager:** Same data, same dashboards, but screenshots are automated and reports are generated
- **Town Planner:** Daily automated pipeline: capture logs, transform, project, screenshot, publish report vault. No human in the loop.

## Every Stage Has an API

Every stage is driven by a CLI command and an API endpoint:

```bash
sg-obs capture --source playwright --instance i-abc123
sg-obs clean --input raw/playwright/i-abc123/2026-04-22
sg-obs extract --input clean/playwright/i-abc123/2026-04-22
sg-obs reduce --input structured/playwright/i-abc123/2026-04-22
sg-obs project --input signals/2026-04-22 --target elastic
sg-obs screenshot --dashboards health,traffic,vaults --output vault:observability-report
```

Each command is also a FastAPI endpoint, callable from other services, agents, or CI pipelines. The CLI wraps the API. Both exist.

## Acceptance Criteria

| # | Criterion | Verification |
|---|-----------|-------------|
| 1 | Raw logs captured from Playwright instance to S3 | Logs from boot, MITM, Playwright, FastAPI all present in S3 bucket |
| 2 | Security cleaning removes/hashes API keys and tokens | Scan cleaned logs, verify no plaintext secrets |
| 3 | Structured extraction produces typed JSON events | Parse cleaned logs, output valid NDJSON with typed fields |
| 4 | Signal reduction produces compact summaries | Summaries contain requests/min, error rates, response time percentiles |
| 5 | Elastic projection loads data and populates dashboards | Start Elastic, import signals, dashboards show data |
| 6 | Pre-configured dashboards available on AMI launch | Launch AMI, load data, all five dashboards render correctly |
| 7 | Playwright screenshots of Kibana dashboards saved to vault | Run screenshot command, images committed to vault |
| 8 | Every stage has a CLI command and API endpoint | All six commands callable from terminal and from HTTP |
| 9 | Any stage re-runnable without re-running prior stages | Delete Stage 3 output, re-run from Stage 2 output, same result |
| 10 | Elastic destruction loses no data (all data in S3/vault) | Destroy Elastic instance, verify all data recoverable from storage |
| 11 | Raw logs preserved even after cleaning and transformation | Raw bucket untouched by downstream processing |
