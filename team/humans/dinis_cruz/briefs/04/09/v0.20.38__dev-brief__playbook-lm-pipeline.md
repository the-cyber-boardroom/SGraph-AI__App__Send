## The Concept

A Notebook LM competitor built on SG/Send. The name is **PlaybookLM**. It works on multiple levels: a playbook is a strategic document (which is what the tool generates), it implies action and execution (not just reading), it subtly nods to Playwright (the automation layer underneath), and it does not limit the scope to slides. A playbook can contain briefs, infographics, analysis, slide decks, the whole pack. It fits the three-tier delivery model: the playbook IS the deliverable.

The key differentiator versus Notebook LM: you get everything. Not just the final slides, but the source materials, the presentation document, the per-slide briefs, the prompts, the logs, and the ability to edit and regenerate any part of it.

## What Notebook LM Does Well (and What It Misses)

Notebook LM takes source materials and produces polished outputs (audio discussions, slide decks). The secret sauce is an intermediate step: whatever prompt Google uses to extract "the essence" of the source materials is genuinely good. It creates a condensed interpretation that drives all downstream generation.

**What Notebook LM does not give you:**

- The intermediate "essence" document (the presentation brief)
- The per-slide descriptions/prompts that drove each slide
- Vector graphics (slides are screenshots/rasters)
- The ability to edit and regenerate individual slides
- The source materials alongside the output
- The prompts and logs from generation
- Branding support (custom templates, corporate identity)

**What we now have that Notebook LM does not:**

- The JS API layer for deterministic infographic generation
- Vault-based delivery with full provenance (every request, every invocation logged)
- Editable outputs (change a slide brief, regenerate just that slide)
- Multiple infographic styles per slide (run the same brief through different models/templates)
- All intermediate artifacts preserved and shareable

## The Pipeline

```
SOURCE DOCUMENTS
(text, PDFs, images, any format)
        |
        v
STEP 1: Presentation Document
(LLM extracts the narrative, key points, structure)
(This is the "essence" that Notebook LM keeps hidden)
        |
        v
STEP 2: Per-Slide Briefs
(One document per slide: system prompt + positioning context + slide-specific text)
        |
        v
STEP 3: Infographic Generation
(Each slide brief -> infographic via JS API)
(Can generate multiple variants per slide: different models, different templates)
        |
        v
STEP 4: Assembly
(Combine infographics into slide deck, add metadata, create _page.json)
        |
        v
OUTPUT VAULT
(Everything: sources, presentation doc, slide briefs, prompts,
 infographics, logs, costs, assembled deck)
```

### The Prompt Architecture

For consistent infographic generation across a deck, the prompt structure needs three layers:

```
+-----------------------------------------------+
| SYSTEM PROMPT (same for all slides in a deck)  |
| - Style, tone, branding, visual guidelines     |
| - Template selection                            |
+-----------------------------------------------+
                    |
+-----------------------------------------------+
| POSITIONING PROMPT (same for all slides)        |
| - The presentation document ("essence")         |
| - Overall narrative context                     |
| - Audience, purpose, key messages               |
+-----------------------------------------------+
                    |
+-----------------------------------------------+
| SLIDE PROMPT (unique per slide)                 |
| - This specific slide's content                 |
| - What to visualise, what data to include       |
+-----------------------------------------------+
```

Sending all three to the infographic generator for each slide ensures visual and narrative consistency. The system prompt handles branding (which is just another source document in the input set). The positioning prompt ensures each slide knows where it sits in the overall story.

## Branding Support

Branding becomes a source document. Provide a brand guide (colours, fonts, logo, tone of voice) as one of the inputs, and it becomes part of the system prompt for all infographic generation. This means corporate slide decks are a natural output: same pipeline, different branding document.

## JS API First

The entire pipeline is JS API first (per yesterday's architecture). The UI above is one consumer. Claude calling the API directly is another. The functions:

```javascript
window.__tool.loadSources([file1, file2, ...])
window.__tool.generatePresentation()     // Step 2
window.__tool.generateSlideBriefs()      // Step 3
window.__tool.generateInfographic(slideIndex, options)  // Step 4
window.__tool.generateAllInfographics(options)
window.__tool.export({ format: 'pdf', includeSource: true })  // Step 5
```

## Connection to Business Model

This is the pack delivery service (yesterday's brief) in product form. The three tiers map directly:

- **Tier 1 (final deliverables):** The exported PDF slide deck
- **Tier 2 (source pack):** The full vault with all intermediate artifacts
- **Tier 3 (edit access):** The tool itself, with the vault loaded, allowing the customer to edit briefs and regenerate slides

Every regeneration, every "Try Different Model" click, every edit-and-regen consumes tokens via OpenRouter at the 25% markup.
