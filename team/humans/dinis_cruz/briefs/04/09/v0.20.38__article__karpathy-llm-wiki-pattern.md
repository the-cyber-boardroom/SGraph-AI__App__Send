# We've Been Building Karpathy's LLM Wiki for a Month. Here's What We Learned.

Last week Andrej Karpathy published a gist called "LLM Wiki" that went viral: 19 million impressions, 5,000+ stars, 3,600+ forks. The core idea: stop using LLMs to re-derive knowledge from raw documents on every query. Instead, have the LLM build and maintain a persistent, structured wiki that compounds over time. Raw sources in, compiled knowledge out, with a schema that tells the LLM how to maintain it all.

Aaron Fulkerson (who co-built MindTouch, one of the original enterprise wikis) responded with a post showing he'd independently built a production version: 26 custom skills, 14 MCP servers, learning loops that graduate observations into permanent system rules. His point: Karpathy's gist is the starting point. Production is where it gets interesting.

I read both and thought: we've been doing this since 11 March. Not as an experiment. As a daily production workflow. And the infrastructure we built to support it is exactly what this pattern needs to scale.

## The Workflow We Actually Run

Every day for the past month, I record voice memos throughout the day: technical architecture decisions, feature ideas, business model thinking, bug reports, strategic positioning. These get transcribed and pasted into a Claude session. The session processes each memo into structured documents: dev briefs, architecture briefs, articles, debriefs, patterns.

In Karpathy's terminology:

- **Raw sources** = my voice memo transcripts (immutable, never modified)
- **Wiki** = the briefs, patterns, and articles the LLM generates and maintains (170+ documents and counting)
- **Schema** = the handover pack that tells each new session how to work: document types, naming conventions, hard rules, project context, the full arc of what has been built

The schema is the key. Every new Claude session starts by reading a handover document that contains the complete methodology: what document types exist, how to name files, what header format to use, what the project looks like, what decisions have been made. The LLM does not start from zero. It starts from a compiled understanding of the entire project.

This is exactly Karpathy's insight. The knowledge compounds. Session 28 is dramatically more productive than session 1, not because the LLM got smarter, but because the accumulated context got richer.

## What Karpathy's Gist Misses (and What Production Taught Us)

Karpathy describes the pattern beautifully but keeps it intentionally abstract. After a month of running this in production, here is what we found matters:

**1. The wiki needs to be encrypted and shareable.**

Karpathy says "the wiki is just a git repo of markdown files." That works for personal use. It breaks the moment you want to share a subset with a partner, deliver a pack to a client, or collaborate across sessions without exposing everything. We built SGit (git for encrypted files) and zero-knowledge vaults specifically for this. The server never sees plaintext. You share via three-word tokens. The wiki is not just version-controlled, it is encrypted, shareable, and access-controllable.

**2. The schema is not one file. It is a team.**

Karpathy describes a single schema document (CLAUDE.md). In practice, we found that a single file becomes a bottleneck. Our schema evolved into a full agentic team methodology: 21+ defined roles (Developer, Architect, QA, Librarian, AppSec, Sherpa, Conductor, and more), each with specific responsibilities. When a voice memo comes in about a security concern, it gets routed to an AppSec brief. When it is about architecture, it goes to the Architect. The schema is not a document. It is an organisation.

**3. The compilation step needs structure, not just summarisation.**

Karpathy's "ingest" operation reads a source and updates wiki pages. Our equivalent is more structured: each voice memo gets classified by type (dev brief, architecture brief, article, debrief, pattern), given a standard header, assigned to specific team roles, and checked against hard formatting rules. The LLM does not just summarise. It produces a professional deliverable with acceptance criteria, code examples, and specific technical recommendations. The output is not a wiki page. It is a brief that someone can act on.

**4. You need a daily index, not just a log.**

Karpathy describes a log.md (chronological record of what happened). We found that a daily index is essential: a document that links every brief produced that day, tracks signals (what is new, what changed status), maintains the full project arc, and summarises previous days. This is the entry point for any new session. Without it, the LLM has to reconstruct context from scattered files. With it, the LLM reads one document and knows exactly where the project stands.

**5. The handover is the hardest problem.**

LLM sessions end. Context is lost. Karpathy mentions Obsidian as the persistent layer. We found that the handover between sessions is the single most important document in the system. It contains: how the workflow operates, what the project is, every key decision ever made, what is live, what is designed, what articles exist, and how to start the next session. Writing a good handover is an art. It is the compiled schema that makes the next session productive from minute one.

**6. Commits are communication.**

This is something Karpathy does not address at all. When I do an offline coding session (no agentic team, just raw experimentation), the git commits become the communication channel. I tag comments in the code with @Dev, @QA, @Librarian. The QA team later reads the commits chronologically (not just the final file state) and extracts the narrative: what was tried, what was measured, what was kept, what was discarded. The commit sequence is a story. This extends Karpathy's "raw sources" concept: the raw source is not just a document you drop in. It is a versioned history with embedded metadata.

## The Infrastructure Gap

Here is what struck me about the responses to Karpathy's gist. Dozens of people built implementations within days. Obsidian plugins, Claude Code skills, MCP servers, npm packages. All of them hit the same walls:

- How do you share the wiki without exposing everything?
- How do you collaborate when multiple agents access the same files?
- How do you version the wiki so you can see what changed and when?
- How do you deliver a subset of the wiki to someone who does not have your setup?

These are infrastructure problems. The pattern is clear. The plumbing is what is missing.

That is what we have been building with SG/Send. Zero-knowledge encrypted vaults with git-like versioning. A CLI (SGit) that does clone, commit, push, pull, branch, share. Rich page layouts that turn a vault folder into a browsable mini-site. Browser-based tools for infographic generation, video processing, and document analysis, all with JavaScript APIs that agents can call directly. And a daily workflow that has produced 170+ structured documents in 28 days.

## The Numbers

28 daily sessions. 170+ documents. 11 LinkedIn articles. 21+ defined agent roles. One handover pack that lets any new session pick up exactly where the last one left off.

The raw output is not the point. The point is that the knowledge compounds. Session 1 produced basic encryption specs. Session 28 produces architecture briefs for JavaScript API layers with ASCII art mockups, Swagger-style automation panels, and acceptance criteria tables, because the LLM has 27 sessions of accumulated context telling it exactly what level of detail and technical depth is expected.

That is what Karpathy is describing. That is what Fulkerson built with MindTouch patterns and Claude Code skills. And that is what we have been running in production, with the infrastructure to make it encrypted, shareable, versioned, and deliverable.

## What Comes Next

We are building PlaybookLM: a tool that takes source documents, generates a presentation brief, creates per-slide prompts, generates infographics via our browser-based tools, and packages everything into a vault. The full Notebook LM workflow, but you get the source materials, the intermediate briefs, the prompts, the logs, and the ability to edit and regenerate any part of it.

The JavaScript API refactoring we did this week means an agent can now generate a full slide deck (6-7 infographics) in about 15 seconds, deterministically, via direct API calls instead of flaky browser automation. That is the kind of infrastructure that makes Karpathy's pattern practical at scale.

The wiki is not enough. You need the wiki, the encryption, the versioning, the sharing, the tools, and the delivery pipeline. That is what we are building.

If you want to see it in action: sgraph.ai. The vaults, the tools, the workflow, it is all live. And every document I described in this article was produced by the exact workflow I am describing.
