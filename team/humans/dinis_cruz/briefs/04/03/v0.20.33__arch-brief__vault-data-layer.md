## Where We Are

The tools are maturing. The LLM chat works. The OpenRouter integration is live. The infographic generator produces results. But every tool has the same problem: **where do I save things?**

API keys created by the OpenRouter admin tool have no persistent home. Transaction IDs from LLM calls are lost if not captured immediately (OpenRouter has no API to retrieve them after the fact). Infographic bundles exist in memory until the page closes. Chat history evaporates on refresh.

The vault is the obvious answer. It provides encrypted storage, version control, branching, and offline access. Every piece of the puzzle exists. They just need to be connected.

### The Key Insight: Deterministic IDs

Vaults support deterministic IDs calculated from the vault key. This means you can access specific files in a vault without cloning the entire vault. You calculate the ID, make an HTTP request, get the file. This is "vault light": fast, targeted access from the browser without downloading everything.

This is what makes vault-as-data-layer feasible for browser tools. Each tool does not need to clone a 500MB vault. It fetches the specific files it needs.

## What Needs to Happen

### The Vault Becomes the Data Layer

Every tool should use a vault for persistence. The only questions are:

- **Which vault?** (The user's personal vault, a tool-specific vault, a shared vault)
- **What key?** (The user's vault key, configured once, used everywhere)

When the OpenRouter admin creates an API key, it saves to the vault. When the infographic tool generates an image, bundle goes to the vault. When the chat records a conversation, it goes to the vault. When LLM calls return transaction IDs, they go to the vault.

The vault is not just storage. It is version-controlled storage. Every save is a commit. Every experiment is a branch. Rollback is trivial. History is preserved.

### The Vault Becomes the Communication Layer

Tools communicate through vaults:

- The infographic tool saves a bundle to the vault
- The Browse view renders it
- The document analysis tool reads it as input
- The Sherpa references it in an email

No tool-to-tool API needed. The vault is the shared state. Each tool reads and writes independently.

## Three Access Modes

| Mode | How It Works | Best For | Offline? |
|---|---|---|---|
| **Remote** | Fetch files via deterministic IDs from the server. No local storage. | Quick access, small reads, browser tools | No |
| **Cached** | Fetch files, cache locally (IndexedDB). Subsequent reads are instant. | Repeated access, medium vaults | Partial (cached files available offline) |
| **Full offline** | Clone entire bare vault locally. Full SGit operations (commit, branch, push, pull). | Heavy work, offline sessions, creating new vaults | Yes |

Browser tools default to Remote or Cached mode. The CLI uses Full Offline mode. The user does not choose: the tool picks the appropriate mode based on what it needs.
