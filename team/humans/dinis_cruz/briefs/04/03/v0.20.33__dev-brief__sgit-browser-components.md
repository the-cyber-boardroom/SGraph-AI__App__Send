# SGit in the Browser: Component Primitives, Drag-and-Drop, and Event Visualisation

**version** v0.20.33
**date** 03 Apr 2026
**from** Human (project lead)

## The Goal

Implement the full SGit experience in the browser as a set of independent Web Components. Every vault primitive (commit, push, pull, branch, merge, init) becomes a separate component. Wire them together into tools. Make vaults as invisible and natural as a file system.

## Component Primitives

Each SGit operation gets its own Web Component. Each component is independent, testable in isolation, and composable with others.

| Component | What It Does | Key Visual |
|---|---|---|
| `sg-vault-init` | Create a new vault (local or remote) | "Create vault" button, key display |
| `sg-vault-commit` | Stage files, create a commit | File list with checkboxes, commit message input |
| `sg-vault-push` | Push local commits to remote | Progress bar, remote status, conflict indicator |
| `sg-vault-pull` | Pull remote changes to local | Incoming changes list, merge status |
| `sg-vault-branch` | Create, switch, list branches | Branch dropdown, create branch dialog |
| `sg-vault-merge` | Merge branches | Diff view, conflict resolution, merge button |
| `sg-vault-status` | Show current vault state | Current branch, modified files, remote sync status |
| `sg-vault-log` | Show commit history | Commit list with messages, timestamps, authors |
| `sg-vault-remote` | Configure remotes (add, remove, list) | Remote list with URLs, push/pull controls |
| `sg-vault-file-drop` | Drop a file, auto-commit and push | Drop zone that handles the full pipeline |
| `sg-vault-browse` | Browse vault contents (integrated with Browse module) | File tree + preview panel |

### The File Drop Component

The most important for user experience. A single drop zone that:

1. Accepts a file (drag and drop, paste, or file picker)
2. Adds it to the local vault branch
3. Commits with an auto-generated message
4. Pushes to the configured remote (if configured)
5. Shows status throughout

One action from the user ("drop a file"). Everything else is automatic. This is how vaults become invisible: the user thinks they are saving a file. They are actually committing to a version-controlled encrypted vault.

## Three Phases of Development

### Phase 1: Primitives (Feature-Rich, Debuggable)

Every component shows everything. Every file being created, loaded, sent, pulled. Every branch visible. Every event logged. The traffic, the connections, the implementations: all visible.

This is the Explorer phase for vault components. Ugly but functional. Every button exposed. Every state visible. The point is to understand how everything flows and to catch bugs.

### Phase 2: Composition (Tools)

Assemble the primitives into tools:

| Tool | Components Used | User Experience |
|---|---|---|
| Vault Manager | init + status + branch + log + browse | Full vault management (like a Git GUI) |
| Quick Save | file-drop + commit + push | Drop a file, it is saved and synced |
| Vault Explorer | browse + log + status | Read-only vault browsing with history |
| Sync Dashboard | pull + push + status + remote | See all vaults, sync status, push/pull controls |

### Phase 3: Invisibility (Consumer Mode)

The user never sees vault primitives. They see:

- A file system (drag files, create folders, rename, delete)
- An auto-save indicator ("saved" / "syncing..." / "synced")
- A history button (view past versions if they want)

The vault, the commits, the branches, the encryption: all invisible. The user works with files. The system handles everything underneath.

## Ephemeral Vaults (Local-Only, Zip Export)

Support vaults that exist only locally (in browser storage / IndexedDB):

1. Create a vault in the browser (no server)
2. Work with files, commit changes
3. When done: export as a zip (a zip of the bare vault folder)
4. Upload the zip to SG/Send, or download locally, or open in the CLI

This enables:

- Offline vault creation (no internet needed)
- Temporary workspaces (create, use, export, discard)
- Privacy-first workflows (the vault never touches any server until the user explicitly exports)
- Testing and experimentation (create vaults freely without consuming server resources)

## Event System and Viewer

Every vault component emits events:

```javascript
// Events emitted by components
'sg-vault:file-added'     { file, branch, timestamp }
'sg-vault:commit-created'  { hash, message, files, branch }
'sg-vault:push-started'    { remote, branch }
'sg-vault:push-complete'   { remote, branch, commits }
'sg-vault:pull-received'   { remote, branch, commits }
'sg-vault:branch-created'  { name, from }
'sg-vault:conflict'        { file, branches }
```

Build an Event Viewer component (`sg-vault-events`) that shows a real-time log of all vault events. This is essential for debugging during Phase 1 and useful for monitoring in production.

The event system also enables tool integration: when the OpenRouter tool makes an API call, it emits an event. The vault listener catches it and auto-saves the transaction data. No explicit "save" step needed.

## Vault-to-Vault Operations

Moving files between vaults:

- Open two vault browsers side by side (SG Layout)
- Drag a file from vault A to vault B
- The file is committed to vault B automatically

This is the file manager experience applied to encrypted, version-controlled vaults. Power users manage multiple vaults. Casual users just see folders.

## Performance and Stress Testing

Dog-food aggressively:

- Every tool saves every piece of data to vaults
- Log everything: LLM calls, API transactions, user actions
- Push the limits: how many files in one vault? How many concurrent commits? How large a file?
- Use the QA infrastructure (Playwright) to simulate heavy vault usage

This is the moment where the vault system either proves it can handle production workloads or reveals its bottlenecks. Better to find them now with our own tools than when external users hit them.
