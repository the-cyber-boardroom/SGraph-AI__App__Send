## The Insight

The vault is a version-controlled, encrypted file system. But components should not need to know that. Components should see a file system: create file, read file, edit file, delete file, create folder, list folder. Where the files are stored (memory, localStorage, IndexedDB, vault, S3) and whether they are encrypted should be invisible to the component.

This is ridiculously powerful once you think it through.

## The Virtual File System (VFS)

A set of primitives that any browser component can use:

```javascript
const fs = new SGFileSystem({ provider: 'vault', id: 'my-workspace' });

// Core primitives
await fs.createFolder('/reports/q1');
await fs.writeFile('/reports/q1/summary.md', markdownContent);
await fs.writeFile('/reports/q1/data.json', jsonData);
await fs.writeFile('/reports/q1/chart.png', binaryData);

const content = await fs.readFile('/reports/q1/summary.md');
const files = await fs.listFolder('/reports/q1');
await fs.copyFile('/reports/q1/summary.md', '/archive/summary-v1.md');
await fs.deleteFile('/reports/q1/draft.md');
await fs.rename('/reports/q1/data.json', '/reports/q1/financials.json');

// Metadata
const info = await fs.stat('/reports/q1/summary.md');
// { size, created, modified, type: 'text/markdown' }
```

The API mirrors what we already have in memory-FS (the server-side virtual file system). Learn from that structure: it works, it is tested, and components already know how to use it.

## Storage Providers

The VFS is an abstraction. The actual storage is pluggable:

| Provider | Storage | Persistence | Offline | Speed | Best For |
|---|---|---|---|---|---|
| `memory` | JavaScript objects | None (lost on refresh) | Yes | Fastest | Testing, temporary data |
| `local-storage` | Browser localStorage | Survives refresh | Yes | Fast (5-10MB limit) | Small config, settings |
| `indexeddb` | Browser IndexedDB | Survives refresh | Yes | Fast (large capacity) | Files, images, WASM modules |
| `vault` | SGit vault (local or remote) | Version controlled | Depends on mode | Medium | Production data, collaboration |
| `sqlite` | SQLite in browser (WASM) | Survives refresh | Yes | Fast | Structured data, queries |
| `zip` | In-memory zip file | Export as download | Yes | Fast | Ephemeral workspaces, sharing |
| `external` | HTTP API to any backend | Depends | No | Slow | Integration with other systems |

Components do not choose the provider. The tool (or the environment configuration) chooses the provider. The component just uses `fs.readFile()` and `fs.writeFile()`.

### Default Provider Stack

For most tools, use a layered approach:

1. **Memory** for active data (fastest)
2. **IndexedDB** as persistent cache (survives refresh)
3. **Vault** as the authoritative store (version controlled, shareable)

Write operations go to memory immediately, then async to IndexedDB, then batch-commit to vault. Read operations check memory first, then IndexedDB, then vault. This gives the speed of in-memory operations with the persistence of vault storage.

## Component Manifest: Declaring Storage Needs

Each component declares what it needs from the file system in its manifest:

```json
{
  "component": "sg-openrouter-admin",
  "storage": {
    "needs": ["read", "write", "list"],
    "paths": ["/openrouter/keys/", "/openrouter/stats/"],
    "auto_commit": false,
    "max_size": "10MB"
  }
}
```

The VFS uses this to:
- Grant or deny access (if the vault does not support writes, the component knows)
- Scope access (the component can only read/write within its declared paths)
- Optimise (if auto_commit is false, batch commits for efficiency)
- Report (if the component cannot get what it needs, it shows a clear message instead of failing silently)

## The Vault as an Extension of the VFS

The vault adds three things on top of the basic file system:

| Capability | What It Adds |
|---|---|
| **Version control** | Commit, branch, log, diff, rollback |
| **Sync** | Push, pull, remotes (sync with server or other vaults) |
| **Encryption** | Files encrypted at rest, PKI-based access control |

These are opt-in. A component that just needs to save a file does not need to know about commits. The VFS handles the commit automatically (based on the `auto_commit` setting). A component that needs branching or explicit version control can access vault-specific APIs through an extended interface.

```javascript
// Basic VFS (all components use this)
await fs.writeFile('/data/report.md', content);

// Vault extension (only components that need it)
await fs.vault.commit('Updated Q1 report');
await fs.vault.branch('experiment-new-format');
await fs.vault.push('origin');
```

## Encryption Layer

The encryption is transparent. When a vault provider is used, files are encrypted before storage and decrypted on read. The component never sees encrypted data. The VFS handles it.

Future PKI support adds access control: different keys for different paths. A component with the "reports" key can read `/reports/` but not `/admin/`. This maps directly to the read-structure key concept from the SGit tree dump brief.

## Performance: Minimising Commits

When capturing telemetry, LLM transaction data, or other high-frequency writes, committing every write is wasteful (each commit becomes a file in S3). Strategies:

- **Batch commits:** Accumulate writes, commit at intervals (every 30 seconds, every 10 writes)
- **Consolidation:** Multiple writes to the same file produce one commit with the final state
- **Telemetry buffer:** High-frequency data goes to a buffer file, committed periodically
- **Consumer does not control this:** The VFS configuration decides the commit strategy, not the component

## Service Worker Protection (Future)

The VFS could run inside a Service Worker, adding a layer of protection:

- The main page cannot directly access the file system
- All access goes through the Service Worker API
- The Service Worker can enforce access control, rate limiting, and encryption
- Even a compromised page (XSS) cannot directly read vault contents

This is a future enhancement, not immediate. But the VFS abstraction makes it possible: swap the provider from "direct IndexedDB" to "Service Worker mediated IndexedDB" without changing any component code.

## Connection to Everything

This VFS is the infrastructure layer that every other brief has been building toward:

- **Per-user vaults (£5 credits brief):** The user's workspace IS a VFS with a vault provider
- **Infographic tool:** Saves bundles to the VFS
- **OpenRouter integration:** Saves keys and stats to the VFS
- **LLM chat:** Saves conversations to the VFS
- **Multi-agent chat:** Each agent's context lives in the VFS
- **Offline mode:** The VFS with IndexedDB provider works offline
- **Browse view:** Renders the VFS contents (same tree, different provider)

One abstraction. Every tool. Every storage mode. Every access pattern.
