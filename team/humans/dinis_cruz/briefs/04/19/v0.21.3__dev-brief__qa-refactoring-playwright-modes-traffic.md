## What Happened

A highly productive session yesterday produced an unexpected but powerful outcome. The starting point was a practical problem: we could not get the Lambda function or Docker container with Playwright to use upstream proxies with authentication. The solution that emerged was an EC2 instance running Playwright with a MITM proxy (mitmproxy) as a sidecar container.

The workflow itself was notable: a Claude agent providing files and instructions, executed inside a Docker container, inside an EC2 instance, with SSH access, connected to a vault. Multiple layers of tooling all composing cleanly.

But the real outcome is bigger than the proxy fix. The MITM sidecar pattern gives us full monitoring, logging, and management of all Playwright traffic. This changes the QA architecture.

## The Architecture That Emerged

```
EC2 Instance (or AMI)
    |
    +--> Playwright Container
    |       |
    |       +--> FastAPI service (browser control API)
    |       +--> Chrome browser
    |       +--> All traffic routed through localhost proxy
    |
    +--> MITM Proxy (sidecar)
            |
            +--> Intercepts all Playwright HTTP/HTTPS traffic
            +--> Full request/response logging
            +--> Traffic analysis and monitoring
            +--> Upstream proxy support (with authentication)
```

This gives us 100% traffic visibility for every browser action. Every page load, every API call, every asset fetch, every redirect: all logged and inspectable.

## The QA Refactoring

The QA codebase and tools need a significant refactoring to run exclusively on two modes. No more ad-hoc Playwright setups. Every QA execution goes through the Playwright service.

### Mode 1: Local Docker

QA code starts a local Docker container with the Playwright service. Tests call the FastAPI endpoints. No Lambda, no EC2, no cloud infrastructure needed. This is the development and CI mode.

```
QA Test Runner
    |
    v
docker run sgraph/playwright-service
    |
    v
POST /session/start
POST /session/{id}/navigate -> sgraph.ai
POST /session/{id}/screenshot
POST /session/{id}/close
    |
    v
Results back to test runner
```

**Limitation:** No MITM proxy in this mode (the sidecar is not present). Acceptable for basic QA, not for traffic analysis.

### Mode 2: Lambda Function (Container)

QA code invokes the Playwright Lambda function. The Lambda runs the container image, executes the browser commands, and returns results. This is the serverless, scalable mode for regular traffic generation.

```
QA Test Runner (or scheduled CloudWatch event)
    |
    v
Lambda invocation (Playwright container image)
    |
    v
Browser commands execute
    |
    v
Results returned (or written to vault)
```

**Limitation:** No upstream proxy with authentication (the original problem that led us to the EC2 solution). Basic proxy works, authenticated proxy does not in Lambda.

### Mode 3: EC2/AMI (Full Capability)

The EC2 instance with Playwright + MITM sidecar. Full traffic capture, authenticated proxy support, long-running sessions. This is the production monitoring mode.

```
QA Test Runner (or cron/scheduled task)
    |
    v
EC2 instance (AMI)
    +--> Playwright container
    +--> MITM proxy sidecar
    |
    v
Full traffic logged
Results written to vault
```

### Abstraction Layer

The QA code should not know which mode it is running in. A single abstraction layer handles the dispatch:

```python
# QA code calls this
playwright_client = PlaywrightClient(mode="local")   # or "lambda" or "ec2"

# Same API regardless of mode
session = await playwright_client.start_session()
await session.navigate("https://sgraph.ai")
screenshot = await session.screenshot()
await session.close()
```

The mode is configured via environment variable or config. The QA tests are identical across all three modes.

## Regular Traffic Generation (The Heartbeat)

This is the next milestone after the refactoring. We need automated, regular traffic hitting the production environment at consistent intervals.

### What It Does

Every hour (configurable), a scheduled task triggers a QA workflow that:

1. Creates a transfer (upload a test file via SG/Send)
2. Downloads the transfer (verify the token works)
3. Creates a vault (via SGit or API)
4. Adds files to the vault
5. Opens the vault in the browser (via Playwright)
6. Takes screenshots (visual verification)
7. Checks key pages and endpoints
8. Logs all results

This is not just QA. It is:

- **Smoke testing:** does the core workflow still work?
- **Observability data:** how many files are being accessed, how many vaults created, what is the response time?
- **Traffic baseline:** regular synthetic traffic creates a baseline for anomaly detection
- **Monitoring validation:** does our logging and monitoring actually capture what it should?

### Logging and Observability Goals

The regular traffic generation should feed directly into the observability pipeline. We want to see:

- Files accessed per hour
- Vaults created per hour
- Transfer completion rate
- Response times per endpoint
- Error rates
- Screenshot diffs (visual regression)

This is the "live observability" that we need before pushing to production more frequently. Regular synthetic traffic plus real monitoring gives us confidence that deployments are safe.

## Deployment: AMI

Package the EC2 setup (Playwright + MITM sidecar) as an AMI. This connects to yesterday's Playwright-as-a-service product brief (`v0.21.2__dev-brief__playwright-as-a-service-product.md`). The AMI is both:

- Our own QA infrastructure (we run it for monitoring)
- A sellable product (customers run it for their own browser automation)

Same image, two markets.

## Acceptance Criteria

| # | Criterion | Verification |
|---|-----------|-------------|
| 1 | QA tests run in Mode 1 (local Docker) | `docker run` + test suite passes against local Playwright container |
| 2 | QA tests run in Mode 2 (Lambda) | Lambda invocation + test suite passes |
| 3 | QA tests run in Mode 3 (EC2 with MITM sidecar) | EC2 instance + test suite passes + full traffic log captured |
| 4 | Same QA test code runs on all three modes without modification | Switch mode via config, same tests execute |
| 5 | Abstraction layer dispatches to correct mode based on config | `PlaywrightClient(mode=X)` routes correctly |
| 6 | MITM sidecar captures 100% of Playwright traffic on EC2 | Run test sequence, verify every HTTP request logged |
| 7 | Scheduled hourly traffic generation running against production | CloudWatch/cron triggers QA workflow every hour |
| 8 | Traffic generation creates transfer, vault, and browser verification | Hourly run produces: upload, download, vault create, screenshot |
| 9 | Observability data flowing from traffic generation to monitoring | Dashboards show files accessed, vaults created, response times |
| 10 | AMI packaged with Playwright + MITM sidecar | AMI launchable, service available on standard port |
| 11 | No ad-hoc Playwright setups remain in QA codebase | All QA Playwright usage goes through the abstraction layer |
