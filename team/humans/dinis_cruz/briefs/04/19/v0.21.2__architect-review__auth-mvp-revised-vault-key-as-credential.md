# Architect Review: Auth MVP — Revised Design (Drop Cognito, Use Vault Key + Browser Password Manager)

**Version:** v0.21.2 | **Date:** 19 April 2026 | **Role:** Architect
**Supersedes:** `team/roles/architect/reviews/04/19/v0.21.2__architect-review__auth-mvp-aws-cognito-research.md`
**Responds to:** `team/humans/dinis_cruz/briefs/04/18/v0.21.2__dev-brief__auth-mvp-social-login-vault-keys.md`
**Status:** Decision recorded. Awaiting AppSec threat-model review.

---

## Executive Summary

The original dev brief proposed social login + vault key storage backed by AWS Cognito and Firebase. After researching the Cognito path in detail and tracing through the architectural implications, we have arrived at a materially simpler design that:

1. **Drops Cognito entirely.** No identity provider is used as an authentication primitive.
2. **Uses existing vault keys as the sole credential.** No master password, no KDF, no wrapping key, no user bundle storage layer.
3. **Stores the vault key in the user's browser password manager** (Google Password Manager, iCloud Keychain, 1Password, Bitwarden, etc.) via the Credential Management API, with a form-based fallback for browsers that don't support it.
4. **Treats Google sign-in (via Google Identity Services) as optional content inside the vault** for UI personalisation only — not as an authentication mechanism.

The result: the server stores no user records, no emails, no identity claims, no wrapping keys, nothing tied to any person. Just encrypted vaults at opaque addresses. The zero-knowledge guarantee is stronger than it would have been with Cognito, and the dependency surface is smaller than any of the options in the original brief.

---

## 1. The Path We Travelled

The conversation that led here moved through four architectural positions. Each is worth recording because it shows why the simpler design is not a shortcut — it is what's left after eliminating genuine candidates.

### Position 1 — Original brief (Cognito + encrypted attributes)

Use Cognito for social login, store vault keys as custom user attributes. Research revealed: the AWS admin can read every custom attribute in plaintext via `AdminGetUser`. Cognito uses AWS-managed KMS keys with no customer-managed key option, so there is no architectural way to hide attribute values from AWS operators. This means Option A in the original brief violates the zero-knowledge guarantee at the identity layer.

### Position 2 — Cognito + client-side-encrypted attributes

Keep Cognito, but encrypt the vault bundle client-side with a key the server never sees (master password + PBKDF2 wrapping key). Cognito then holds ciphertext only.

Researched and documented as viable in the prior review. Works. But introduced a master password (a new user-facing secret), a KDF layer, a recovery-phrase UX, and still kept Cognito as a dependency.

### Position 3 — Drop Cognito, use Google Identity Services directly

Recognised that Cognito's real jobs were (a) identity and (b) encrypted-blob storage. Both can be replaced: identity by Google Identity Services (`accounts.google.com/gsi/client`) in the browser with zero backend, and storage by SG/Send's own Memory-FS keyed by the Google `sub` claim. This removed Cognito but kept the master-password layer and the per-user bundle storage.

### Position 4 — The design in this document

Recognised that the SG/Send **vault key already does everything** the master password + wrapping key + bundle storage was meant to do. The vault key is already:

- A pointer to an existing encrypted vault on SG/Send's existing storage
- The decryption material for that vault's contents
- User-shareable and cross-device portable by design
- Already in a format (`<key>:<id>`) that a password manager can store as a credential

Once that was seen, the auth layer collapsed. No master password. No KDF. No bundle storage. No user records on the server. Google sign-in becomes optional enrichment data inside the vault, not an auth step.

This is the design recorded below.

## 2. Decision: Drop Cognito

**We will not use AWS Cognito, AWS Identity Centre, Firebase Auth, or any identity provider as an authentication primitive for SG/Send.**

Rationale:
- The architectural job Cognito was hired to do (identify the user and hold their vault keys) can be done by the user's browser password manager, which the user already has, which is already cross-device-synced, which is already end-to-end encrypted by the vendor, and which requires no changes to SG/Send's server or storage layer.
- The zero-knowledge guarantee gets stronger, not weaker, because the server no longer has even an identifier tying storage to a person.
- The dependency surface shrinks: no AWS service on the critical path, no Google Cloud service either. The architecture stays vendor-neutral and fully client-side.
- The "no account" compatibility property from the original brief (SG/Send works without accounts today, and must continue to) is satisfied trivially, because there *are* no accounts in this design.

The original brief's acceptance criteria that relate to Cognito specifically (criteria 1–4, 7, 11) are no longer applicable. Criteria 5, 6, 8–10 are preserved and satisfied by this design, with criteria 8–9 (admin panel / admin cannot decrypt) reframed below.

## 3. The Core Observation: A Vault Key Is Already a Credential

A vault key in SG/Send has this form:

```
o4ghedh2lghqdvczmr24i797:vtx3iz2h
└─────── secret ────────┘ └ id ─┘
```

Two halves separated by a colon.

| Half | Name | Role | Sensitivity |
|------|------|------|-------------|
| `o4ghedh2lghqdvczmr24i797` | Vault **key** (decryption/write key) | Encrypts and authenticates vault operations | **SECRET** — must never leave the user's control |
| `vtx3iz2h` | Vault **id** | Addresses the vault on SG/Send's storage | **PUBLIC** — designed to be exposed |

The full string is the credential the user needs to access their vault. It is 33 characters. It comfortably fits in every known browser password manager (Chrome caps at ~1024 characters, 1Password and Bitwarden go far higher).

This matters because **a credential is exactly the shape of object that browser password managers are built to store, sync, autofill, and share across devices**. SG/Send didn't design the vault key to be a password-manager credential, but it happens to have every property required of one.

The secret-first ordering is consistent with common crypto URI conventions (e.g. Fragment identifiers carrying decryption material) where the sensitive portion is structurally first and separable from the public locator.

## 4. The Model

```
┌──────────────────────────────────────────────────────────────────┐
│  Browser                                                         │
│                                                                  │
│  ┌────────────────────────────┐    ┌───────────────────────────┐│
│  │ Browser password manager   │    │ <sg-vault-picker>         ││
│  │ (Google / iCloud / 1Pass…) │    │                           ││
│  │                            │◄───┤ credentials.get/store     ││
│  │ stored: vault key          │    │                           ││
│  │ synced cross-device        │    │ form fallback (Safari/FF) ││
│  └────────────────────────────┘    └─────────────┬─────────────┘│
│                                                   │              │
│                                     ┌─────────────▼─────────────┐│
│                                     │ Vault layer (existing)    ││
│                                     │ — decrypts with <key>     ││
│                                     │ — reads/writes vault      ││
│                                     │   contents at <id>        ││
│                                     └─────────────┬─────────────┘│
└───────────────────────────────────────────────────┼──────────────┘
                                                    │ HTTPS
                                       ┌────────────▼────────────┐
                                       │ SG/Send (unchanged)     │
                                       │ Memory-FS at vtx3iz2h   │
                                       │ — ciphertext only       │
                                       │ — never sees <key>      │
                                       └─────────────────────────┘
```

No identity provider in the diagram. No user records on the server. No KDF, no master password, no wrapping key, no bundle storage. The client is the password manager's client; the password manager is the sync fabric; the server is the existing SG/Send vault storage.

### What the password manager stores

```
Origin:   https://send.sgraph.ai
Username: "My personal vault"            ← user-supplied at creation (default: "Vault 1")
Password: o4ghedh2lghqdvczmr24i797:vtx3iz2h
```

One credential per vault. The `username` is a user-facing vault name, not an identity — chosen by the user at vault creation time so the password manager's chooser UI shows something meaningful.

### Why username = vault name (not Google email)

Considered during design; rejected:
- **Vault id as username** — unreadable (`vtx3iz2h` means nothing to a human browsing their password manager list).
- **Google email as username** — reasonable when the user signs in with Google, but creates an inconsistency if the same vault is used without Google identity, or if the user attaches a different Google account later. The username in a password manager should be stable across the vault's lifetime. A vault name is.
- **User-supplied vault name** — default suggested ("Vault 1", "Vault 2"), user can rename at any time. Renaming updates the password manager entry on the next save. Stable, meaningful, user-controlled.

## 5. User Experience by Browser

### New user, first vault, Chromium browsers (Chrome, Edge, Brave, Opera)

1. User visits `send.sgraph.ai`, clicks "Create vault."
2. Optional step: name the vault (default "Vault 1").
3. Client generates the vault key (existing SG/Send mechanism — unchanged).
4. Client calls `navigator.credentials.store(new PasswordCredential({id: name, password: vaultKey}))`.
5. Chrome shows its native "Save password for send.sgraph.ai?" bubble.
6. User clicks Save. One click total past the "Create vault" button.
7. User is now inside an empty vault.

### New user, first vault, Safari / Firefox

Same flow, but `PasswordCredential` is not implemented. Fallback:

1. Client renders a hidden form pre-filled with the vault name and vault key.
2. Client programmatically submits the form.
3. The browser's native password-save heuristic detects a form with `autocomplete="new-password"` being submitted with a visible result (DOM change / navigation), and offers to save.
4. User clicks Save. One click.

The fallback is the same form-based pattern every website uses; it just happens to be invisibly driven by our own code.

### Returning user, same device

- **Chromium:** `navigator.credentials.get({password: true, mediation: 'optional'})` shows the account chooser with the user's saved vaults. One click picks a vault. Zero typing.
- **Safari / Firefox:** Render the unlock form; the native password manager autofills both fields. Click submit (or auto-submit after detection).
- **1Password / Bitwarden / LastPass users:** Extension offers the credential on the form; one click fills.

### Returning user, new device

- **Chrome signed in with sync:** Credentials are already on the new device. Identical to "same device" flow.
- **iCloud Keychain (Safari on another Apple device):** Credentials sync via iCloud. Identical.
- **Third-party manager (1Password/Bitwarden):** The user's vault of credentials is already on their other device. Identical.
- **Fresh device with no sync:** Password manager is empty for this origin. User needs their vault key (by whatever out-of-band means) and pastes it into the form. The browser then offers to save it. Subsequent visits are transparent.

### Sharing a vault

The vault key is already shareable — this is existing SG/Send behaviour and does not change. The new thing is: when a recipient pastes a shared vault key into the form on `send.sgraph.ai`, their browser offers to save it. They click Save, give it a name, and the shared vault is now one of their synced vaults with the same UX as their own.

This means SG/Send's existing sharing model gains cross-device persistence for the recipient *at no additional architectural cost.*

## 7. Google Identity as Optional Vault Content

The original brief proposed Google OAuth as the authentication mechanism. In this design, Google sign-in plays a different role: **optional personalisation content stored inside the vault.**

### Flow

1. User is already inside their vault (having opened it with their vault key, as per §4–5).
2. User can optionally click "Sign in with Google to personalise."
3. Google Identity Services (`accounts.google.com/gsi/client`) runs entirely client-side and returns a JWT with: `sub`, `email`, `email_verified`, `name`, `picture`, `given_name`, `family_name`.
4. The client stores the relevant claims **inside the vault** (alongside other vault contents, encrypted under the vault key).
5. UI personalises: "Welcome back, Alice" with avatar.

### Properties

- The server sees none of this. It only ever sees the encrypted vault blob.
- The Google JWT is never sent to SG/Send's server. It never needs to be — there is nothing server-side to authenticate.
- The personalisation is per-vault. If a user has two vaults and personalises both with the same Google account, they see "Welcome Alice" in both; if with different accounts, they see different identities per vault. This is correct behaviour — vaults are independent.
- Signing out of Google does not affect vault access. The vault key is what unlocks the vault. Google is decoration.

### Client-side JWT verification

Google's ID token is an RS256 JWT signed by Google. In a traditional server-side integration, the server would verify the signature against `https://www.googleapis.com/oauth2/v3/certs`. In this design, verification is optional because:

- The token is consumed by the same client that requested it from `accounts.google.com/gsi/client`.
- The client trusts the script it loaded from `accounts.google.com` over HTTPS (that is the trust root).
- A compromised client can bypass any JS-level verification anyway.
- Nothing server-side relies on the JWT's authenticity.

We may still verify the signature for defensive purposes (`jose` library, ~20KB gzipped, works in the browser), but it is not load-bearing.

## 8. Recovery

Because the vault key is the only thing that unlocks the vault, losing it means losing the vault. This is the same property SG/Send already has, and is not introduced by this design.

The password manager is the primary recovery mechanism: if the user is signed into Chrome, Safari, or their third-party manager on any device, the vault key is recoverable by opening the password manager.

Optional strengthening (not required for MVP):
- At vault creation, offer to display the vault key with a "save this somewhere safe" prompt.
- Allow the user to email themselves the key (encrypted with a short PIN).
- Allow the user to print a recovery card.

None of these change the architecture — they are UX affordances around the same single secret.

## 9. Zero-Knowledge and Admin Visibility — Stronger Than Before

The original brief specified: "the admin should ideally not be able to read user vault keys ... The admin can see metadata (user count, registration dates, activity) but not decrypt user data."

In this design:

| What the admin (operator of SG/Send) can see | Status |
|---|---|
| Encrypted vault ciphertext at `<vault-id>` addresses | Yes, by design |
| Vault keys (decryption material) | **No — never touches the server** |
| Email addresses, names, or any identity claim | **No — Google JWTs never leave the browser** |
| User count | **Architecturally undefined** — there are no users, only vaults |
| Registration dates | **Architecturally undefined** — there is no registration event |
| Activity per user | **Architecturally undefined** — activity is per vault, not per user |
| Which vaults belong to the same person | **Architecturally undefined** — the server has no link between vaults |

This is a strictly stronger zero-knowledge guarantee than the Cognito-based design, which had AWS holding emails and registration timestamps. Here, even those facts don't exist on the server.

Admin operations that *do* make sense (disk usage, ciphertext size per vault, rate-limiting by source IP) are architecturally orthogonal to identity and remain available.

The original brief's acceptance criterion 8 ("Admin mode shows user count and registration stats") is **not satisfied** — but we argue this is the correct outcome. A product whose tagline is zero-knowledge should not, by construction, know how many users it has. If business metrics are needed, they come from outside the zero-knowledge boundary (web analytics on the landing page, opt-in telemetry, etc.), not from inside the vault system.

## 11. Component Specification: `<sg-vault-picker>`

One new web component replaces the four proposed in the original brief.

```
<sg-vault-picker>
├── Methods
│   ├── createVault(name?)           → { vaultKey, vaultId }
│   ├── openVault()                  → { vaultKey, vaultId } | null
│   ├── saveVault(name, vaultKey)    → saves to browser password manager
│   └── shareVault(vaultKey, name)   → UI affordance for out-of-band sharing
├── Events
│   ├── vault-opened                 → vault is ready for contents
│   ├── vault-created                → new vault, show onboarding
│   └── vault-cancelled              → user dismissed chooser
└── Internal
    ├── Credential Management API path (Chromium)
    └── Form fallback path (Safari/Firefox, with autocomplete attributes)
```

The component is reusable: any tool that wants to let the user open or create a SG/Send vault drops this component in. The infographic editor, the PlaybookLM tools, the video recorder — all get vault persistence for free.

Approximately 150 lines of vanilla JS plus a small HTML template. No dependencies beyond what SG/Send already uses.

## 13. Open Questions for AppSec and Dev

1. **Credential Management API browser coverage.** `PasswordCredential` is Chromium-only (Chrome, Edge, Brave, Opera). Safari and Firefox require the form fallback. AppSec: review whether the form fallback's reliance on `autocomplete="new-password"` heuristics is acceptable, or whether we should prefer form-only for all browsers to reduce code-path variance.

2. **Shared vault auto-save.** When a recipient pastes a shared vault key, their browser offers to save it. This is generally desirable, but: a vault shared for one-time read access gets persisted to the recipient's password manager. AppSec: is this behaviour acceptable, or should shared-vault flows set a flag to suppress the save prompt?

3. **Vault name leakage.** The vault name chosen by the user is visible to the password manager (Google / Apple / third party). If the user names a vault "Tax return 2026", that string lives in their Google Password Manager. This is standard browser-trust territory, but worth flagging.

4. **Multiple accounts in the same browser profile.** If two users share a device and a browser profile, they'll see each other's vault credentials in the password manager chooser. Same as for any shared site. UX copy should advise single-user browser profiles for privacy-sensitive use.

5. **Recovery UX.** MVP does not mandate a recovery-code display. Sherpa should design whether to offer it at vault creation and how forcefully.

6. **`sg1.` format prefix.** Earlier discussion suggested a `sg1.` version prefix on the stored value to enable future format migration. Vault keys today do not carry such a prefix. Decision: do we prepend one when saving to the password manager (`sg1.o4ghedh2lghqdvczmr24i797:vtx3iz2h`) to enable future evolution, or rely on the vault layer's existing format discipline? Architect recommends prefix; Dev to confirm.

## 14. Implementation Order (IFD-Friendly)

Stage 1 — Minor version: `<sg-vault-picker>` built against `localStorage` as stand-in password manager. Full vault create/open/save flow working, no Credential Management API calls yet. Unit tests cover the UI logic.

Stage 2 — Minor version: Replace `localStorage` with Credential Management API for Chromium browsers. Form fallback implemented and tested in Safari and Firefox. No code path changes in the vault layer.

Stage 3 — Minor version: Google Identity Services integration as optional vault personalisation. Identity claims written into vault contents, not used for authentication.

Stage 4 — Minor version: `<sg-vault-picker>` rolled into at least one tool (candidate: infographic editor, PlaybookLM tools, or existing `send.sgraph.ai` UI) as the first adoption site.

Each stage is independently shippable, independently testable, and leaves the system in a working state.
