## The Opportunity

On Wednesday, I am attending the AWS Conference in London. This is a concrete opportunity to reach out to AWS contacts, demonstrate the technology stack, and start finding a strategic partner for SG/Send.

## What We Can Show

The timing is good. We now have a stack that is deeply integrated with AWS and demonstrates real, working patterns:

- **Lambda deployment** (user-facing and admin): two Lambda functions behind Lambda URL endpoints
- **S3 storage** via Memory-FS abstraction (application code is backend-agnostic)
- **ECR container deployment** for the Playwright service
- **AMI packaging** for the Playwright + MITM proxy setup (EC2 with sidecar)
- **Serverless architecture** with near-zero idle cost
- **Zero-knowledge encryption** (client-side AES-256-GCM, server stores only ciphertext)
- **The layered container architecture** (dynamic code loading from S3, separating runtime from application code)
- **Vault infrastructure** with full version control, collaboration, and encrypted storage

This is not a pitch deck. This is working infrastructure that people can inspect and use.

## Key Objectives

### 1. Get AMIs on AWS Marketplace

The Playwright-as-a-service AMI (browser container + MITM sidecar) is the simplest product to list first. It is self-contained, immediately useful, and demonstrates the "run everywhere" principle in an AWS-native context.

**Action before Wednesday:** Push to get the AMI listing process started. Even if it is not live by Wednesday, having the submission in progress is a conversation starter.

### 2. Find a Strategic Partner

The time has come to find a partner for SG/Send within the AWS ecosystem. Target conversations:

- AWS Solutions Architects who work with security-focused customers
- AWS Partner Network contacts
- Teams working on encryption, data governance, or zero-trust architectures
- Marketplace team (for the AMI listing)

The pitch: "We built a zero-knowledge encrypted platform entirely on AWS serverless. Lambda, S3, ECR, CloudFront. The server never sees plaintext. Here is the architecture. Here is the AMI. Here is the open source repo."

### 3. Showcase Architecture Patterns

Prepare materials showing the AWS-specific architecture patterns:

- Lambda URL functions (no API Gateway) for cost-effective HTTPS endpoints
- Memory-FS storage abstraction (S3 backend, but application code is portable)
- Container-based Lambda with dynamic code loading from S3
- MITM proxy sidecar pattern for full traffic observability
- Deterministic safe paths (PBKDF2, 600k iterations) for encrypted object storage on S3

These are reusable patterns that AWS teams can reference and recommend to other customers.
