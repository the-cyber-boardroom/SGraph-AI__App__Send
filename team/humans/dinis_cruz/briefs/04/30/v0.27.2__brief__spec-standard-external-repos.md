# Spec Standard: External Repos and the sg-compute_spec Convention

**version** v0.27.2
**date** 30 Apr 2026
**from** Human (project lead)
**to** Architect (lead), Developer
**type** Brief

---

## The Need

Specs need to live outside the main codebase. The Cyber Boardroom is a clear example: its spec should be in its own GitHub repo (`sg-compute_spec_cyber-boardroom`), not inside the SG Compute repository. This enables third parties to create specs, keeps the core lean, and allows independent versioning.

For this to work, we need: a naming convention, a defined structure that every spec repo must follow, and a mechanism to load and execute specs from external repos.

## Naming Convention

```
sg-compute_spec_{name}
```

Examples:
- `sg-compute_spec_cyber-boardroom`
- `sg-compute_spec_firefox`
- `sg-compute_spec_elastic`
- `sg-compute_spec_ollama`
- `sg-compute_spec_s3-storage`

This naming makes specs discoverable (search GitHub for `sg-compute_spec_`) and clearly identifies them as part of the SG Compute ecosystem.

## Loading Specs from GitHub

### The Workflow

```
User provides a GitHub repo URL (or just a spec name)
    |
    v
SG Compute resolves the repo: github.com/the-cyber-boardroom/sg-compute_spec_cyber-boardroom
    |
    v
Clone the repo onto the node (EC2 instance)
    |
    v
Read the spec definition (spec.json or similar)
    |
    v
Execute the setup script (install dependencies, configure containers, start services)
    |
    v
Node is running with the spec's containers and API
```

### Where Does the Clone Happen?

Two options, depending on the deployment mode:

**Build-run mode (from EC2 instance):** The node EC2 instance clones the repo, runs the setup, and starts the containers. This requires the node to have internet access and Git installed. Slower (clone + setup + start), but always gives the latest version.

**AMI mode (pre-baked):** The spec repo is cloned and set up during the AMI baking process. The resulting AMI has everything pre-installed. Launching from the AMI is fast (no clone needed) but runs whatever version was baked in.

For the MVP, build-run mode is sufficient. AMI baking is the production optimisation.

### Can We Avoid Cloning the Whole Repo?

Ideally, we would download only the spec definition and the setup script without cloning the full repo. Options:

- **GitHub API:** Fetch specific files via the GitHub API (no clone needed, but requires API access)
- **Shallow clone:** `git clone --depth 1` (fast, minimal download)
- **Release artifacts:** Download a zip from the GitHub release (no Git needed)

Start with shallow clone. It is the simplest and works without GitHub API tokens.

### Lambda Function Limitation

The current Lambda function that provisions nodes cannot clone repos (it is stateless, short-lived, and may not have Git). For specs loaded from GitHub, the Lambda function starts a base EC2 instance with Git pre-installed, and the instance itself does the clone and setup on boot. This is the build-run mode.

For AMI-based specs, the Lambda function just launches the AMI. No clone needed.

## The Spec Repo Structure

Every `sg-compute_spec_*` repo must follow this structure:

```
sg-compute_spec_{name}/
    |
    +--> spec.json              (spec definition: metadata, containers, ports, requirements)
    +--> setup.sh               (setup script: install deps, configure, start containers)
    +--> README.md              (documentation: what this spec does, how to use it)
    |
    +--> api/                   (optional: spec-specific FastAPI routes)
    |       +--> routes.py
    |       +--> cli.py
    |
    +--> config/                (optional: default configuration files)
    |       +--> defaults.json
    |
    +--> tests/                 (optional: spec-specific tests)
            +--> test_spec.py
```

The minimum viable spec repo is: `spec.json` + `setup.sh` + `README.md`.

## AMI Registration in Vault

Once a spec has been baked into an AMI, the AMI ID should be registered in the vault:

```json
{
  "spec": "cyber-boardroom",
  "repo": "github.com/the-cyber-boardroom/sg-compute_spec_cyber-boardroom",
  "amis": [
    {
      "ami_id": "ami-abc123",
      "region": "eu-west-2",
      "baked_at": "2026-04-30T10:00:00Z",
      "spec_version": "v1.2",
      "commit": "abc123def"
    }
  ]
}
```

This means the vault becomes the AMI registry. To launch a spec, check the vault for a baked AMI first. If one exists, use it (fast). If not, fall back to build-run mode (slower).

## Fractal Note

The Lambda function that provisions SG Compute nodes is itself a spec in principle. It is just that running it from Lambda is convenient because the initial state is "no state." But if we need to deploy SG Compute to a different AWS account, a different region, or run it from local Docker, we need the same spec-based approach. The provisioning function becomes a spec that can be loaded and run anywhere.

## Acceptance Criteria

| # | Criterion | Verification |
|---|-----------|-------------|
| 1 | Naming convention documented: `sg-compute_spec_{name}` | All existing specs renamed to follow convention |
| 2 | At least one spec lives in an external GitHub repo | `sg-compute_spec_cyber-boardroom` exists on GitHub |
| 3 | SG Compute can load a spec from a GitHub URL | Provide URL, node starts with the spec's containers running |
| 4 | Shallow clone used for build-run mode | Verify `--depth 1` on clone, minimal download |
| 5 | AMI baking works from an external spec repo | Clone spec repo, set up, bake AMI, launch from AMI |
| 6 | AMI registered in vault after baking | Vault contains AMI ID, region, bake date, spec version |
| 7 | Minimum spec repo (spec.json + setup.sh + README) is sufficient | Create a minimal spec repo, load it, verify it works |
