## Major Milestone: Three Agents Collaborating via Vaults

This brief captures a significant achievement and defines the next phase building on it. Three agents are now collaborating in production using vaults as the communication and content delivery layer.

### The Three Agents

| Agent | Name | Model | Role |
|-------|------|-------|------|
| @dev | Dev Agent | Sonnet | Code changes. Has repo access. Can push to QA branch. Can run Playwright tests and take screenshots. |
| @content | Content Agent | Sonnet | Content creation. Creates and manages the content vault. Writes markdown, creates navigation structures. No repo access. |
| @dinis | Inbox Manager | Haiku | Reads the email traffic between agents. Monitors what is happening. Reports status to the human. No code or content access. |

### The Communication Layer: Email-FS

The agents communicate via a file-based email system running on top of vaults. This is a pure file-system email solution: each "email" is a file in a shared vault. Agents write messages, other agents read them. No SMTP, no external services, no API integrations. Just files in a vault.

This works because vaults already provide everything email needs: write access (commit a message file), read access (clone or shallow-read the vault), notification (check for new files), history (every message is a commit), and encryption (zero-knowledge).

### The Two Vaults in Play

| Vault | Purpose | Who Writes | Who Reads |
|-------|---------|-----------|-----------|
| Collaboration vault | Agent-to-agent communication (email-FS). Also stores screenshots and working artefacts. | @dev, @content, @dinis | All three agents |
| Content vault | Content that renders on the live website. Markdown files, images, navigation structures. | @content (primary), @dev (screenshots) | The website (via read key + object ID) |

## Major Milestone: Vault Content Rendered on Live Web Page

This is the breakthrough. The dev agent:

1. Received an email from @content containing a **read key** and an **object ID** for a markdown file in the content vault
2. Created a web page in the QA branch that embeds the read key and object ID
3. Pushed to QA, which auto-deployed
4. The web page successfully renders the markdown content directly from the vault
5. Load time for the vault dependencies: approximately 120 milliseconds (acceptable, comparable to normal backend rendering)

**This means the website can now render content from vaults without any server-side content processing.** The content lives in the vault. The web page references it by read key + object ID. The client-side JavaScript fetches, decrypts, and renders. The web server serves only static HTML with vault references.

### The Full Workflow That Just Happened

```
@content creates a vault
    |
    v
@content writes a markdown file to the vault
    |
    v
@content generates a read key for the vault
    |
    v
@content sends email to @dev via collaboration vault:
    "Here is the read key: XXX, object ID: YYY"
    |
    v
@dev receives email, creates a web page with the read key + object ID embedded
    |
    v
@dev pushes to QA branch
    |
    v
CI pipeline deploys QA branch (1-2 minutes)
    |
    v
@dev writes a Playwright test against the QA deployment
    |
    v
@dev takes a screenshot of the rendered page
    |
    v
@dev pushes the screenshot to the collaboration vault
    |
    v
@dev sends email to @content: "Page is live, here is the screenshot"
    |
    v
@dinis reads the email traffic, reports status to the human
```

Two agents collaborating, producing a live web page, with automated verification and evidence capture, communicating entirely through vaults. The human was not in the loop for the execution, only for the briefing.

## What Comes Next: The Content Library

### The Concept

A "library" section on the website (like knowledge base articles, how-to guides, technical documentation) where ALL content comes from vaults. Not from static HTML files in the repo. Not from a CMS. From vaults.

This means:

- @content creates and updates content by writing to the content vault
- The website renders it by referencing read keys and object IDs
- Content updates do not require a code deployment (update the vault, the page updates)
- Every piece of content has version history (vault commits)
- Content can be previewed via vault share tokens before going live

### The First Library Entry: Meta (How This System Works)

The first knowledge base article should explain the system that creates it. This is deliberately meta: the content describes the agent collaboration workflow, the vault-rendered pages, and the email-FS communication. Screenshots taken by @dev during the workflow appear in the article alongside the explanation.

This serves two purposes: it creates real content to test the system with, and it documents the system for future agents and humans who need to understand it.

### Navigation Structure from Vault

The navigation (table of contents, category structure, ordering) should also come from the content vault, not from the repo. A JSON file in the vault (inspired by the `_page.json` convention from vault page layouts) defines:

```json
{
  "library": {
    "title": "Knowledge Base",
    "sections": [
      {
        "title": "How It Works",
        "articles": [
          {
            "title": "Vault-Rendered Web Pages",
            "content_object_id": "abc123",
            "read_key": "xxx",
            "icon": "vault"
          },
          {
            "title": "Agent Communication via Email-FS",
            "content_object_id": "def456",
            "read_key": "xxx",
            "icon": "email"
          }
        ]
      }
    ]
  }
}
```

The website reads this JSON from the vault, renders the navigation, and each article link loads content from the vault by object ID. @content manages this JSON file. @dev does not need to touch the website code to add new articles.

### Content Types Supported

The vault content renderer should handle:

| Type | How It Renders |
|------|---------------|
| Markdown (.md) | Rendered as HTML via existing markdown component |
| Images (.png, .jpg, .svg) | Displayed inline, loaded from vault by object ID |
| Screenshots | Same as images, but taken by @dev via Playwright |
| Mermaid diagrams | Rendered from mermaid code blocks in markdown |
| ASCII art / code blocks | Rendered in monospace from markdown code blocks |
| JSON data | Rendered as formatted, navigable data |
| Videos | Played via existing video component |

### The Screenshot Loop

@dev can take screenshots of the QA deployment (or production) and push them to the collaboration vault. @content can then copy those screenshots into the content vault and reference them in articles. This creates a self-documenting loop:

1. @content writes an article about a feature
2. @dev takes a screenshot of the feature running
3. @content includes the screenshot in the article
4. The article renders on the website with the live screenshot
5. If the feature changes, @dev takes a new screenshot, @content updates the reference

The screenshots are always current because they are always taken from the live system.

### Creating Content at Scale

Once the first article works end-to-end, the same workflow produces articles "by the dozens and hundreds":

- @content receives a topic (from a human, from another agent, from a schedule)
- @content writes the markdown, generates or requests screenshots
- @content updates the navigation JSON
- The library page updates automatically (no deployment needed)

The human's role shifts from "write content" to "brief the content agent" to "review what the content agent produced."
