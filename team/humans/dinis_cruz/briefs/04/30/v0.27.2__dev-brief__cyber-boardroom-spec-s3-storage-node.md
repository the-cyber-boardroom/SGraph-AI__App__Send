## The Full Circle

The Cyber Boardroom is the legal entity that powers sgraph.ai. It was the first project where I implemented the "run from anywhere" principle. It already runs in containers. It already supports Ollama for local LLMs. It was built to be portable.

But the Cyber Boardroom was missing pieces: confidential execution, ephemeral sessions that disappear, encrypted storage, agentic workflows. Over the past two months, I have been building exactly those pieces in SG/Send and the Ephemeral Compute platform.

Now we bring it full circle: run the Cyber Boardroom as a spec inside Ephemeral Compute. The missing pieces are no longer missing. Storage is vaults or ephemeral disk. Compute is nodes. LLMs are Ollama pods. The framework is complete.

## Spec 1: Cyber Boardroom Node

### What It Is

A spec that runs the Cyber Boardroom application as a node in the Ephemeral Compute platform. Self-contained, ephemeral, with all conversations and data disappearing when the node is destroyed (unless explicitly saved to a vault).

### Why This Matters

- **Confidential sessions:** Spin up a Cyber Boardroom instance for a specific meeting or analysis. When done, destroy the node. No data persists anywhere.
- **Ephemeral by default:** No database, no persistent storage. The session lives and dies with the node.
- **Optional persistence:** If you want to keep the output, save to a vault before destroying. The user chooses what survives.
- **Run anywhere:** The same spec runs on EC2, on a laptop via Docker, on a customer's infrastructure. The Cyber Boardroom's original "run from anywhere" promise, now delivered through the Ephemeral Compute framework.

### The Spec

```json
{
  "name": "cyber-boardroom",
  "display_name": "Cyber Boardroom",
  "description": "Confidential, ephemeral Cyber Boardroom sessions",
  "pods": [
    {
      "name": "cyber-boardroom",
      "image": "sgraph/cyber-boardroom:latest",
      "ports": { "5000": 5000 },
      "storage": "ephemeral"
    }
  ],
  "requires": {
    "llm": true,
    "storage": "optional"
  },
  "stability": "experimental"
}
```

### As a Stack (with Ollama)

The Cyber Boardroom needs an LLM. Running it as a stack combines two specs:

```json
{
  "name": "cyber-boardroom-full",
  "display_name": "Cyber Boardroom (Full Stack)",
  "nodes": [
    { "spec": "cyber-boardroom" },
    { "spec": "ollama", "model": "gemma-4" }
  ],
  "networking": {
    "cyber-boardroom-connects-to": "ollama"
  }
}
```

One click: "Launch Cyber Boardroom stack." Two nodes start. The Cyber Boardroom connects to the Ollama instance for LLM inference. Everything runs locally on the customer's infrastructure. No data leaves their environment. When the session ends, both nodes are destroyed.

### Separate Repo

This spec should live in its own repo (e.g. `the-cyber-boardroom/sg-compute-spec-cyber-boardroom`). This tests the pattern of third-party specs: can someone outside the core team create a spec that runs in Ephemeral Compute? The Cyber Boardroom is the first external spec, even though it is our own project.

### Migration Notes

The Cyber Boardroom currently has some serverless functions. For the spec to be self-contained in Docker, these need to be moved to run inside the container (FastAPI endpoints replacing Lambda functions). This should not be hard since the business logic is already containerised. The Lambda functions are thin wrappers.

## Spec 2: S3-Compatible Storage Node

### What It Is

An EC2 node that provides S3-compatible object storage via an API that is 100% compatible with the S3 API. Clients (including our own code that uses osbot-aws/boto3) can point at this node instead of AWS S3 and everything works.

This is the storage primitive for the Ephemeral Compute platform. MinIO started here but never implemented a good UI. OpenStack Swift attempted it but was overly complex. We can do it simply because we only need the core S3 APIs.

### Three Storage Backends

| Mode | Where Data Goes | When to Use |
|------|----------------|-------------|
| **Disk** | Local EBS volume on the EC2 instance | Ephemeral storage that survives pod restarts but not node termination |
| **S3 relay** | Proxied to a real S3 bucket | When you want S3 compatibility locally but data in real S3. Acts as a caching proxy. |
| **Vault** | Stored in an SG/Send vault | Encrypted, versioned, persistent. The most powerful mode. |

The S3 API facade is the same regardless of backend. The client does not know (or care) whether data lands on disk, in S3, or in a vault.

### The S3 APIs to Implement

Start with the core operations that cover 90% of usage:

```
PUT    /{bucket}/{key}           -> Upload object
GET    /{bucket}/{key}           -> Download object
DELETE /{bucket}/{key}           -> Delete object
HEAD   /{bucket}/{key}           -> Object metadata
GET    /{bucket}?list-type=2     -> List objects in bucket
PUT    /{bucket}                 -> Create bucket
DELETE /{bucket}                 -> Delete bucket
GET    /                         -> List buckets
```

Presigned URLs (for large file upload/download) can come in the second iteration. Multipart upload in the third. The core CRUD operations are enough to make osbot-aws and boto3 work.

### The S3 Browser UI

A web-based file browser for the S3-compatible storage, served from the same node:

```
+-----------------------------------------------------------+
|  S3 Browser                                                |
+-----------------------------------------------------------+
|                                                           |
|  Buckets:                                                 |
|  [sg-vault-data]  [sg-logs]  [sg-backups]  [+ Create]    |
|                                                           |
|  sg-vault-data/                                           |
|  +-- vaults/                                              |
|  |   +-- vault-abc123/                                    |
|  |   +-- vault-def456/                                    |
|  +-- sessions/                                            |
|  +-- config/                                              |
|                                                           |
|  [Upload] [Download] [Delete]                             |
|                                                           |
+-----------------------------------------------------------+
```

This is what MinIO should have built. A simple, clean, browsable UI for object storage. Navigate buckets, browse keys, upload/download files, see metadata. This becomes a core tool in the Ephemeral Compute platform.

### Why This Matters

With an S3-compatible storage node, the Ephemeral Compute platform becomes self-contained:

- **Compute:** Docker/Podman nodes with container orchestration
- **Browser:** Playwright, Firefox, Neko for web interaction
- **Storage:** S3-compatible node for object storage
- **LLM:** Ollama for local inference
- **Vaults:** SG/Send for encrypted, versioned storage

No external AWS services required (except EC2 itself for the compute). A customer could run the entire platform in an air-gapped environment with local storage.

### The Spec

```json
{
  "name": "s3-storage",
  "display_name": "S3-Compatible Storage",
  "description": "S3-compatible object storage with browser UI",
  "pods": [
    {
      "name": "s3-server",
      "image": "sgraph/s3-storage:latest",
      "ports": { "9000": 9000 },
      "env": {
        "STORAGE_BACKEND": "disk",
        "STORAGE_PATH": "/data/s3"
      },
      "volumes": { "/data/s3": "persistent" }
    }
  ],
  "stability": "experimental"
}
```

### Relationship to Memory-FS

SG/Send already uses Memory-FS as a storage abstraction (in-memory, disk, S3). The S3-compatible storage node is the inverse: it provides the S3 API on top of local storage (or vault storage). Memory-FS consumes S3. This node produces S3. Together they create a fully portable storage layer.

## How These Two Specs Connect

```
Stack: "Cyber Boardroom (Self-Contained)"
    |
    +--> Node 1: Cyber Boardroom (application)
    |       +--> Pod: cyber-boardroom app
    |       +--> Connects to Node 2 for LLM
    |       +--> Connects to Node 3 for storage
    |
    +--> Node 2: Ollama (LLM inference)
    |       +--> Pod: ollama with gemma-4
    |
    +--> Node 3: S3 Storage (data)
            +--> Pod: s3-server (disk-backed or vault-backed)
            +--> Pod: s3-browser UI
```

Three nodes. One stack. Completely self-contained. No external dependencies. No data leaving the environment. Launch with one click. Destroy with one click. Everything disappears.

This is the "run everywhere" principle taken to its logical conclusion.
