# Spec Internal Structure: Every Spec Gets Its Own API and CLI

**version** v0.27.2
**date** 30 Apr 2026
**from** Human (project lead)
**to** Architect (lead), Developer
**type** Arch brief

---

## The Principle

Every spec defines behaviour. That behaviour is exposed by a FastAPI service and a CLI. Always. The API and CLI are thin wrappers around core functionality. The same base classes power every spec's API and CLI.

This is not optional. If a spec runs containers, it has an API that controls those containers. If it has configuration, the API exposes it. If it has actions (start a browser, run a query, import data), the API exposes them.

## Two Layers of API Per Node

Every running node has two APIs:

### Layer 1: Core Control Plane (Same for Every Node)

This is the host FastAPI that exists on every node regardless of spec type:

```
GET    /health                    -> Node health
GET    /host/status               -> CPU, memory, disk
GET    /containers                -> List running containers
POST   /containers                -> Start a container
POST   /host/shell/execute        -> Run a shell command
GET    /metrics                   -> Prometheus metrics
```

This comes from the base image. Every node has it. It never changes per spec.

### Layer 2: Spec API (Unique per Spec)

This is the spec-specific behaviour:

**Docker spec:**
```
POST   /docker/run                -> Run a container with specific image
GET    /docker/images             -> List available images
POST   /docker/pull               -> Pull an image from registry
```

**Firefox spec:**
```
POST   /firefox/session           -> Start a new Firefox session
POST   /firefox/navigate          -> Open a URL
POST   /firefox/proxy/script      -> Upload MITM intercept script
GET    /firefox/proxy/traffic     -> Get captured traffic
```

**Elastic spec:**
```
POST   /elastic/import            -> Import data from vault/S3
GET    /elastic/indices            -> List indices
POST   /elastic/dashboards/screenshot -> Screenshot a dashboard
GET    /elastic/status             -> Elastic cluster health
```

**Cyber Boardroom spec:**
```
POST   /boardroom/session         -> Start a new boardroom session
POST   /boardroom/prompt          -> Send a prompt to the session
GET    /boardroom/history         -> Session history
POST   /boardroom/export          -> Export session to vault
```

Each spec defines only the endpoints that make sense for its behaviour. The core control plane handles everything generic.

## Base Classes

All spec APIs inherit from the same base classes:

```python
# Base class provides: auth, error handling, logging, vault access
from sg_compute.spec_base import SpecRoutes, SpecCLI

class FirefoxRoutes(SpecRoutes):
    """Firefox-specific API routes."""
    
    prefix = "/firefox"
    
    async def session_start(self, request: SessionStartRequest) -> SessionStartResponse:
        # Firefox-specific logic
        ...

class FirefoxCLI(SpecCLI):
    """Firefox-specific CLI commands."""
    
    prefix = "firefox"
    
    def navigate(self, url: str):
        # Calls self.api.post("/firefox/navigate", {"url": url})
        ...
```

The base classes provide:
- API key authentication (inherited, not reimplemented)
- Error handling and error response format (consistent across all specs)
- Vault access (read/write to the node's vault)
- Logging and metrics (every request logged, timing captured)
- CLI-to-API bridging (CLI commands call the API, never implement logic directly)

The spec-specific code should be **as small as possible.** A minimal spec has 50-100 lines of route definitions and a thin CLI wrapper. The base classes do the heavy lifting.

## CLI Mirrors API (Always)

The CLI for every spec is auto-generated or trivially thin:

```bash
# CLI
sg-compute firefox session-start
sg-compute firefox navigate --url https://example.com
sg-compute firefox proxy-script --upload script.py

# These are identical to:
POST /firefox/session     {}
POST /firefox/navigate    {"url": "https://example.com"}
POST /firefox/proxy/script  (file upload)
```

The CLI calls the API. The API does the work. No logic in the CLI that is not in the API. No API-only features. No CLI-only features. They are the same thing with different interfaces.

## What Goes in spec.json

The spec definition includes not just container configuration but also the API and CLI metadata:

```json
{
  "name": "firefox",
  "display_name": "Firefox + MITM Proxy",
  "version": "1.0",
  
  "containers": [
    {
      "name": "firefox",
      "image": "jlesage/firefox:latest",
      "ports": {"5800": 5800, "5900": 5900}
    },
    {
      "name": "mitm-proxy",
      "image": "mitmproxy/mitmproxy:latest",
      "ports": {"8080": 8080}
    }
  ],
  
  "api": {
    "module": "api.routes",
    "prefix": "/firefox",
    "endpoints": [
      "session_start", "session_stop", "navigate",
      "proxy_script_upload", "proxy_traffic"
    ]
  },
  
  "cli": {
    "module": "api.cli",
    "prefix": "firefox",
    "commands": [
      "session-start", "session-stop", "navigate",
      "proxy-script", "proxy-traffic"
    ]
  },
  
  "build": {
    "setup_script": "setup.sh",
    "ami_boot_time_target_seconds": 60
  }
}
```

The spec.json is the single source of truth. The core SG Compute reads it to: register the API routes (dynamically), register the CLI commands (dynamically), start the containers, and display the spec in the UI.

## The Pattern Applied to Every Current Spec

| Spec | Spec-Specific API Endpoints | Notes |
|------|---------------------------|-------|
| Docker | run, images, pull, build | Generic container management |
| Podman | Same as Docker (same abstraction) | Runtime adapter handles differences |
| Firefox | session, navigate, proxy config, proxy traffic | Interactive browser with MITM |
| Neko | session, resolution, audio, multi-viewer | WebRTC-based remote browser |
| Playwright | screenshot, sequence, session management | Programmatic browser automation |
| Elastic | import, export, indices, dashboards, screenshot | Data visualisation |
| SG/Send Vault | vault CRUD, token management, backup/restore | Encrypted storage |
| Ollama | models, generate, pull, status | Local LLM inference |
| S3 Storage | Full S3 API (see S3 compatibility brief) | S3-compatible storage |
| Cyber Boardroom | session, prompt, history, export | Confidential analysis environment |
| Linux Terminal | (shell only, covered by core control plane) | Minimal spec |

## Acceptance Criteria

| # | Criterion | Verification |
|---|-----------|-------------|
| 1 | Base classes exist: SpecRoutes, SpecCLI | Import and use in a new spec with under 50 lines of spec-specific code |
| 2 | Every running node has both core control plane and spec API | Hit `/health` (core) and `/{spec}/...` (spec-specific) on same node |
| 3 | CLI mirrors API for every spec | Every CLI command maps to exactly one API endpoint |
| 4 | spec.json defines API and CLI metadata | Core reads spec.json, dynamically registers routes and CLI commands |
| 5 | Adding a new spec API endpoint requires no changes to core | Add a route to a spec's routes.py, restart, new endpoint available |
| 6 | At least 3 specs have their own API defined | Firefox, Elastic, and Playwright each have spec-specific endpoints |
| 7 | API authentication inherited from base class (not reimplemented) | Spec route has no auth code, auth works via base class |
| 8 | All API requests logged with timing | Check call log, see spec-specific requests with response times |
