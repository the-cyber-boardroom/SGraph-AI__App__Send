## The Content Pipeline: MVP First, Then Scale

This brief describes both where we are going (a multi-agent content pipeline with specialised roles) and where we start (two agents: @dev and @content). The future vision informs the architecture, but the MVP is deliberately simple.

**Until we have a good first MVP of the site and the workflows proven, we are only using two agents: @content and @dev.** Adding more agents before the pipeline works end-to-end would add complexity without adding value. Get the two-agent workflow solid first. Scale the agent count later.

### MVP: Two Agents Only (@content + @dev)

In the MVP phase, @content does everything on the content side: creates content, reviews it, manages the content vault, updates navigation, and coordinates with @dev when structural changes are needed. @dev handles all code changes, Playwright screenshots, QA branch deployments, and structural additions.

```
MVP Pipeline (now):

@content
    |
    +--> Creates markdown content in the content vault
    +--> Updates navigation JSON
    +--> Requests screenshots from @dev (via email-FS)
    +--> Reviews rendered output on QA branch
    |
    v
@dev
    |
    +--> Makes code changes when new page types or components are needed
    +--> Takes Playwright screenshots, pushes to collaboration vault
    +--> Pushes to QA branch, verifies deployment
    +--> Responds to @content requests via email-FS
    |
    v
Human (@dinis)
    |
    +--> Reviews significant changes
    +--> Approves promotion from QA to dev to main
```

This is sufficient to build the first version of the content library, prove the vault-rendered content model, and establish the email-FS communication patterns. Only once this MVP is stable do we introduce additional agents.

### Future: Multi-Agent Content Pipeline (After MVP)

Once the two-agent workflow is proven and the site has its first set of pages, the pipeline expands:

### Future Pipeline (Target State)

```
Content Creator (Sherpa, Ambassador, or specialised content agent)
    |
    v  writes content to content vault
    |
Content Reviewer (@content agent, or QA agent)
    |
    v  reviews, suggests changes, signs off
    |
Optional: AppSec Review (for sensitive content)
    |
    v  security review if needed
    |
Approval (human review, or automated if confidence is high)
    |
    v  content goes live (vault content renders on website automatically)
```

In the future state, the @dev agent is only involved when:
- A new page type needs to be created (structural change)
- A new Web Component needs to be embedded
- The rendering pipeline needs modification
- A bug in the content rendering needs fixing

For all other content changes (new articles, updated text, new screenshots, navigation updates), the pipeline runs without @dev.

### Agent Roles (Future State)

The following roles are the target for when the pipeline scales beyond two agents. In the MVP, @content covers all content roles (creator + reviewer) and @dev covers all technical roles.

| Agent | Role in Content Pipeline |
|-------|------------------------|
| **Sherpa** | Creates initial content drafts. Research-driven. Writes articles, how-to guides, technical explanations. |
| **Ambassador** | Creates marketing and positioning content. Investor materials. Case studies. |
| **@content** | Reviews and signs off content. Does NOT create content (repositioned from creator to reviewer). Quality gate. |
| **AppSec** | Reviews content for security implications (accidental key exposure, sensitive information, correct encryption claims). |
| **@dev** | Only involved for code/structural changes. Not in the content loop. |
| **@dinis** | Monitors the pipeline. Final approval for significant content. |

### When to Add More Agents (Transition Criteria)

Stay with two agents until ALL of the following are true:

1. The content vault to website rendering pipeline works reliably (vault content appears on the live site consistently)
2. At least 10 library pages have been published through the two-agent workflow
3. The email-FS communication between @content and @dev is stable and well-understood
4. The QA branch deployment is automated and fast (under 2 minutes)
5. The navigation JSON structure is established and proven

Once these are met, introduce one new agent at a time (e.g. @ambassador for marketing content), observe the three-agent dynamic, stabilise, then consider adding more. Do not jump from two agents to six.

### Section 2: How It Works (Technical)

| Page | Content |
|------|---------|
| **Vault-rendered content (meta page)** | Explains how this very page works. Shows the read key and object ID being used. Shows the encrypted content. Shows the decryption in the browser. Interactive: the user can see the actual keys and objects that power the page they are reading. |
| How encryption works | Client-side AES-256-GCM. Web Crypto API. Key derivation. What the server sees (ciphertext only). |
| How version control works | Commits, branches, push/pull. How SGit maps to Git concepts. |
| How agents collaborate | Email-FS, vault communication, the three-agent workflow. |

The vault-rendered content page is the most important: it is self-documenting. It shows the read keys being used on the page, the object IDs, even the encrypted content before decryption. A technical user can inspect exactly what is happening. This is trust through transparency.

## Interactive Component Embedding

Since all frontend elements are Web Components, library pages can embed live components directly:

```html
<!-- In a library page's vault content -->
<p>Here is the send component in action:</p>
<sg-send-drop></sg-send-drop>

<p>Here is a vault browser showing this article's source vault:</p>
<sg-vault-browser vault-key="xxx" object-id="yyy"></sg-vault-browser>

<p>Here is the encrypted content before decryption:</p>
<sg-encrypted-view object-id="yyy"></sg-encrypted-view>
```

The components render inline in the article. The user can interact with them. This turns documentation into a live demo. "Here is how the send component works" followed by an actual working send component right on the page.

A central page tracking every public vault and every read key:

```
+-----------------------------------------------------------+

|                                                           |

|                                                           |
|  OWASP Presentation Pack                                  |

|  [Browse] [Clone]                                         |
|                                                           |
|  Content Library Source                                    |
|  Read Key: yyy  |  Objects: 12  |  Last Updated: 30 Apr   |
|  [Browse] [Clone]                                         |
|                                                           |
|  Demo: Healthcare MVP                                     |
|  Read Key: zzz  |  Objects: 8   |  Last Updated: 17 Apr   |
|  [Browse] [Clone]                                         |
|                                                           |
+-----------------------------------------------------------+
```

This page is important for three reasons:
1. **Management:** We need to track all public vaults and read keys for backup and deployment
2. **Transparency:** Visitors can see and access everything we have published
3. **Offline deployment:** If we want to deploy the website offline, we need to know which vaults to include

## Offline Website Capability (Emerging)

If all content comes from vaults, and we have a registry of all vaults, then an offline deployment is: download all vault content, bundle it into the static site, deploy without any vault server dependency. The read keys and object IDs still work, but they resolve to local files instead of the vault API.

This is not immediate priority, but the vault registry makes it possible. Worth noting in the architecture.

## Infographics for Library Content

Each library page should eventually have an infographic version (via the PlaybookLM pipeline). The infographic acts as both:
- A visual summary for visitors who prefer visual learning
- A shareable asset for social media (drives traffic back to the full article)

This can be automated: when a new article is published to the content vault, a pipeline generates an infographic and adds it to the same vault.
