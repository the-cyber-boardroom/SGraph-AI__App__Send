# VPC Isolation Per Stack: Network Boundaries for Multi-Node Deployments

**version** v0.27.2
**date** 30 Apr 2026
**from** Human (project lead)
**to** Architect (lead), DevOps
**type** Arch brief

## The Concept

Every stack (a combination of multiple nodes deployed together) gets its own VPC. Full network isolation. Each stack is its own island. Nodes within a stack can talk to each other freely. Nodes in different stacks cannot see each other.

This is the infrastructure equivalent of "every vault has its own key." Every stack has its own network.

## Why VPC Per Stack

### Security Isolation

Some containers in a stack may run without authentication (because adding auth is complex or breaks functionality). In a shared VPC, unauthenticated services are exposed to every other node. In an isolated VPC, they are only accessible to nodes in the same stack. The VPC boundary IS the access control.

### Client Separation

Different customers pay for different stacks. Customer A's Elastic instance must not be on the same network as Customer B's Playwright instance. VPC isolation makes this automatic. No firewall rules to manage. No security groups to audit. The networks simply do not connect.

### Clean Lifecycle

When a stack is destroyed, its VPC is destroyed. All network resources, subnets, routing tables, security groups: all gone. No leftover configuration. No orphaned rules. No "we forgot to clean up the VPC from three weeks ago."

### Billing Attribution

Each VPC can be tagged with the stack ID and the customer ID. AWS cost allocation tags on the VPC propagate to all resources within it. Cost per stack becomes trivially calculable.

## The Architecture

```
Stack: "cyber-boardroom-full" (Customer A)
    |
    +--> VPC: 10.0.0.0/16
    |       |
    |       +--> Subnet: 10.0.1.0/24 (private)
    |       |       |
    |       |       +--> Node 1: Cyber Boardroom (10.0.1.10)
    |       |       +--> Node 2: Ollama (10.0.1.11)
    |       |       +--> Node 3: S3 Storage (10.0.1.12)
    |       |
    |       +--> Subnet: 10.0.2.0/24 (public, bastion only)
    |               |
    |               +--> Bastion: VNC/SSH jump host (10.0.2.10)
    |
    (No connection to any other VPC)

Stack: "observability" (Customer B)
    |
    +--> VPC: 10.1.0.0/16
    |       |
    |       +--> Node 1: Elastic + Kibana (10.1.1.10)
    |       +--> Node 2: Playwright (10.1.1.11)
    |
    (No connection to any other VPC)
```

### Access Pattern

External users access the stack through a bastion node in the public subnet:

1. User authenticates to the bastion (VNC, SSH, or API)
2. Bastion provides access to internal nodes (Kibana, Cyber Boardroom UI, etc.)
3. Internal nodes have no public IP. They are only reachable through the bastion.

For API-only access (no interactive browsing), the bastion can expose a reverse proxy that routes API calls to the appropriate internal node.

## The Lifecycle

```
Create Stack:
    1. Create VPC with CIDR block
    2. Create private subnet(s)
    3. Create public subnet (for bastion)
    4. Create internet gateway (for bastion's public access)
    5. Create route tables
    6. Create security groups (internal: allow all within VPC, bastion: allow user IP)
    7. Launch nodes into private subnet
    8. Launch bastion into public subnet
    9. Configure bastion to proxy to internal nodes

Destroy Stack:
    1. Terminate all nodes
    2. Terminate bastion
    3. Delete security groups
    4. Delete subnets
    5. Delete route tables
    6. Detach and delete internet gateway
    7. Delete VPC
    
    Everything is gone. No residue.
```

### Automation

The entire lifecycle must be automated: one API call to create, one to destroy. The LLMs can generate the CloudFormation or boto3 code for this. The important thing is that the create/destroy cycle is tested, reliable, and complete (no orphaned resources after destroy).

## Cost Considerations

VPCs themselves are free in AWS. The costs come from:

- NAT Gateway (if private nodes need internet access): ~$0.045/hour + data transfer
- Internet Gateway: free
- Elastic IPs: free if attached to a running instance, $0.005/hour if idle

For ephemeral stacks (live for 30-60 minutes), the VPC overhead is negligible. A NAT Gateway for 1 hour costs ~$0.05. That is noise compared to the EC2 instance costs.

For stacks that do not need internet access from private nodes (e.g. air-gapped analysis), skip the NAT Gateway. Nodes communicate internally and with the bastion only.

### Cost Research Needed

Document the actual cost per stack creation/destruction cycle. Include: VPC creation time, NAT Gateway cost, data transfer within VPC, and any hidden charges. This informs the pricing model for commercial stacks.

## Stack Definition Enhancement

The stack spec.json gains a networking section:

```json
{
  "name": "cyber-boardroom-full",
  "nodes": [
    { "spec": "cyber-boardroom" },
    { "spec": "ollama", "model": "gemma-4" },
    { "spec": "s3-storage", "backend": "disk" }
  ],
  "networking": {
    "vpc_isolation": true,
    "cidr": "10.0.0.0/16",
    "bastion": {
      "enabled": true,
      "type": "vnc",
      "access": "user-ip-only"
    },
    "internet_access": {
      "bastion": true,
      "private_nodes": false
    }
  }
}
```

## Acceptance Criteria

| # | Criterion | Verification |
|---|-----------|-------------|
| 1 | Stack creation produces a new VPC | Create stack, verify VPC exists with correct CIDR |
| 2 | Nodes in stack are in private subnet (no public IP) | Inspect nodes, confirm no public IP assigned |
| 3 | Bastion in public subnet with user-IP-only access | Bastion accessible from user's IP, not from any other |
| 4 | Internal nodes reachable from bastion only | SSH/API from bastion to internal node works. Direct access from outside fails. |
| 5 | Nodes within same stack can communicate freely | Node 1 can reach Node 2 on any port within the VPC |
| 6 | Nodes in different stacks cannot communicate | Node in Stack A cannot reach any node in Stack B |
| 7 | Stack destruction removes all VPC resources | Destroy stack, verify VPC, subnets, security groups, gateways all deleted |
| 8 | No orphaned resources after stack destruction | AWS resource audit shows nothing remaining from destroyed stack |
| 9 | Cost per stack creation/destruction documented | Cost analysis with breakdown per resource type |
| 10 | Stack spec.json includes networking section | Networking config in spec, honoured during creation |
