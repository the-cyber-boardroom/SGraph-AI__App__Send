## Part 1: Components Inside _page.json

### The Insight

If `_page.json` can render hero sections, text blocks, images, and navigation, why can it not also render interactive components? A file upload drop zone, an infographic generator, a markdown viewer, a video player. Each of these is already a Web Component. The `_page.json` layout system just needs to know how to include them.

This turns `_page.json` from a document layout format into a page builder. Users can create custom UIs by composing components in JSON, without writing HTML.

### How It Works

Add a `component` type to the `_page.json` schema:

```json
{
  "type": "component",
  "component": "sg-file-upload",
  "props": {
    "mode": "drag-and-drop",
    "accept": "image/*,application/pdf",
    "vault_token": "apple-banana-cherry"
  }
}
```

The page renderer sees `type: "component"`, loads the named Web Component, and passes the props. Any registered component becomes embeddable.

### Supported Components (Initial Set)

Do not allow arbitrary JavaScript injection. Instead, maintain a whitelist of supported components:

| Component | What It Does | Use Case in _page.json |
|---|---|---|
| `sg-file-upload` | Drag-and-drop file upload | Custom upload page for a client |
| `sg-file-download` | Download button/panel | Delivery page for a pack |
| `sg-markdown-viewer` | Render markdown from a file | Documentation pages |
| `sg-video-player` | Play video from vault/folder | Media showcase |
| `sg-infographic-gen` | Infographic generator tool | Interactive demo page |
| `sg-vault-browse` | Vault file browser | Technical showcase |
| `sg-image-gallery` | Image gallery with lightbox | Portfolio/examples page |

### Fractal Architecture

Pages can contain components that contain pages. A `sg-markdown-viewer` can render a markdown file that itself references a `_page.json`. A `sg-vault-browse` can display a vault that has its own `_page.json`. This is fractal by design, with a recursion depth limit to prevent infinite loops.

### Why This Matters

This means the examples website (previous brief) can be built entirely via `_page.json` files in vaults. No custom HTML. No separate codebase for layouts. Just JSON that composes existing components. And critically, this allows exploration and iteration outside the main codebase: try different layouts, different component combinations, different themes, all by editing JSON in a vault.

This is also the path to user-created UIs: a customer could build their own upload page, their own documentation site, their own dashboard, all by composing `_page.json` files with embedded components.

## Part 2: Pure View Mode (Reinforcement)

User feedback confirms the previous brief: the folder UI feels "geeky and dark" for non-technical recipients. The solution is the same: a pure view mode that shows only the `_page.json` rendered content. The view button should be visible alongside the existing "Gallery" and "Folder" buttons at the top of the browse UI.

With embeddable components, view mode becomes even more powerful: the shared link is not just a static page. It can include interactive elements (upload forms, infographic tools, video players) while still hiding all the folder/vault infrastructure.

## Part 3: WhatsApp Share Mode

### The Problem

When you share an SG/Send link via WhatsApp, the link preview cannot show a thumbnail. This is by design: the server never sees the decrypted content (it is after the `#` fragment). WhatsApp's link preview fetcher hits the server, which cannot return a meaningful preview because the content is encrypted.

### The Workaround: Copy-Paste Share Kit

Since we cannot solve this server-side without breaking zero-knowledge, provide a client-side "Share to WhatsApp" feature:

1. Generate a screenshot/thumbnail of the current view (client-side, using Canvas API)
2. Provide the share link as copyable text
3. Show instructions: "Paste this image, then paste this link as the caption"
4. Optionally: try to trigger WhatsApp's share intent with both image and text (may work on mobile via Web Share API)

```
+-----------------------------------------------+
| Share to WhatsApp                    [X]      |
+-----------------------------------------------+
|                                               |
|  [Preview screenshot of the page]             |
|                                               |
|  1. Copy image: [Copy Image]                  |
|  2. Copy link:  [Copy Link]                   |
|                                               |
|  Paste the image in WhatsApp, then            |
|  add the link as caption.                     |
|                                               |
|  [Share via WhatsApp] (mobile only)           |
+-----------------------------------------------+
```

This is not perfect (two-step paste), but it is the best possible within zero-knowledge constraints. The alternative (server-side preview) would require the server to see the content, which violates the core security model.
