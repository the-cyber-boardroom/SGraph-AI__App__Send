# Layered Container Architecture: Dynamic Code Loading for Serverless Containers

**date** 18 Apr 2026
**from** Human (project lead)

## The Problem

When you move from zip-based Lambda deployment to container-based Lambda deployment, you lose the fast development loop.

With zip-based Lambda (the osbot-aws pattern), deployment is: upload a zip to S3, point the Lambda at it, done. It takes seconds. The Lambda loads the zip and runs.

With container-based Lambda, deployment is: rebuild the Docker image, push to ECR, update the Lambda to use the new image, wait for it to pull. This takes minutes, sometimes longer. And it happens every time you change a single line of code.

The container gives us things the zip cannot (Playwright, Chrome, system-level dependencies), but the development pipeline becomes a bottleneck. We need the benefits of containers without the deployment penalty.

## The Solution: Two-Step Container Architecture

Separate what changes rarely (the runtime environment) from what changes often (the application code).

**The container** provides the stable base: OS, Python, system libraries, and (in the Playwright case) browser binaries. This changes rarely (weeks or months between updates).

**The application code** is loaded dynamically at startup. It lives outside the container image. When the container starts, a small bootstrap script checks for application code, fetches it if needed, and executes it. Updating the application means updating the code source, not rebuilding the image.

This is the same principle as the osbot-aws Lambda deployment pattern, adapted for containers.

## The Bootstrap Logic

Every container runs one piece of bootstrap code at startup:

```
Container starts
    |
    v
Check: is there application code in the code folder?
    |
    +--> YES: load and execute it
    |
    +--> NO: check for a code source (env var, config)
              |
              +--> Local folder (mounted volume)
              +--> URL (download zip, extract, execute)
              +--> S3 bucket + path (download, extract, execute)
              |
              v
         Load code, execute it
```

The bootstrap does not know or care that it is running in a container. It would work the same way on a bare EC2 instance, a Fargate task, a Kubernetes pod, or a local laptop. The container just provides the environment.

## Code Loading Modes

### Mode 1: Local Folder

The code folder inside the container already contains the application code. This is the "frozen" or "baked-in" mode for production: the container is fully self-contained.

Also used for local development: mount a local directory into the container as the code folder, edit files on your host, they are immediately available inside the container.

```bash
# Development: mount local code
docker run -v ./my-app:/app/code -p 8000:8000 base-image

# Production: code baked into image
docker run -p 8000:8000 production-image
```

### Mode 2: URL (Zip Download)

The container receives a URL (via environment variable or config) pointing to a zip file. At startup, it downloads the zip, extracts it to the code folder, and executes.

```bash
docker run -e CODE_URL=https://example.com/app-v1.2.zip -p 8000:8000 base-image
```

This is the simplest remote loading mode. Works with any HTTP server, S3 presigned URLs, GitHub release assets, or even a vault.

### Mode 3: S3 Bucket

The container receives an S3 bucket, path, and version. At startup, it pulls the code from S3 using boto3 (via osbot-aws). This is the AWS-native mode, directly analogous to how osbot-aws deploys Lambda functions.

```bash
docker run -e CODE_S3_BUCKET=my-bucket -e CODE_S3_KEY=apps/playwright/v1.2.zip -p 8000:8000 aws-image
```

This requires the AWS layer (see Layer 2 below).

## The Three Layers

Each layer builds on the previous one. Each is a separate folder, a separate Dockerfile, a separate CI pipeline, a separate Docker Hub image, and a separate ECR deployment.

### Layer 1: Core (FastAPI Base)

**Contains:**
- Python runtime
- osbot-fast-api / osbot-fast-api-serverless (FastAPI with API key auth)
- AWS Lambda Web Adapter (for Lambda compatibility)
- The bootstrap/code-loader script
- Debug/admin routes for testing code loading

**Capabilities:**
- Spin up a FastAPI service, protected by API key
- Load application code from local folder or URL
- Debug endpoints: upload code, upload zip, set code URL, inspect loaded modules
- Runs locally, on Docker, on Lambda (via web adapter), on Fargate, anywhere

**What it does NOT contain:**
- No AWS SDK (no boto3, no S3 access)
- No browser (no Playwright, no Chrome)
- No application-specific code

**Image:** `sgraph/core-fastapi:latest` on Docker Hub

### Layer 2: Core + AWS (S3 Code Loading)

**Builds on:** Layer 1

**Adds:**
- osbot-aws (boto3 wrapper, never use boto3 directly)
- S3 code loading mode

**Capabilities:**
- Everything from Layer 1
- Load application code from S3 bucket
- Access AWS services (S3, Secrets Manager, etc.)

**Image:** `sgraph/core-fastapi-aws:latest` on Docker Hub

### Layer 3: Playwright (Browser Execution)

**Builds on:** Layer 2 (or Layer 1 if AWS is not needed)

**Important:** This layer uses the official Playwright base image (e.g. `mcr.microsoft.com/playwright/python`) as its FROM image, then adds the Layer 1/2 code on top. This is because the Playwright image already contains Chrome, Firefox, and the system-level dependencies that are painful to install manually.

**Adds:**
- Playwright library + browser binaries (Chrome, Firefox)
- The Playwright FastAPI routes (browser control, screenshots, PDFs, sequences)

**Capabilities:**
- Everything from Layer 1 (and optionally Layer 2)
- Browser control via API (navigate, click, fill, screenshot, PDF, video)
- Sequence execution (JSON instruction list)
- Proxy configuration support

**Image:** `sgraph/playwright-service:latest` on Docker Hub

## The Debug/Admin API (Layer 1)

The core container exposes admin routes (protected by API key) that allow runtime code management. These are intentionally "dangerous" (remote code execution by design) because the entire purpose of this architecture is to load and execute code dynamically. The API key authentication makes this safe for controlled environments.

```
POST /admin/code/upload          -> Upload a code file
POST /admin/code/upload-zip      -> Upload a zip of code
POST /admin/code/set-url         -> Set URL to fetch code from
GET  /admin/code/status          -> What code is loaded, from where
GET  /admin/code/modules         -> List loaded modules
POST /admin/code/reload          -> Re-fetch and reload code
```

This is the beginning of an agentic setup pattern. An agent can provision a container, upload its code, and start executing. The container is the runtime. The code is the agent's logic.

## Development vs Frozen Mode

**Development mode:** Code loaded dynamically at startup (from URL, S3, or mounted volume). Every restart picks up the latest code. No image rebuild needed.

**Frozen mode:** Code baked into the container image. A Dockerfile that copies the application code into the code folder at build time. The result is a fully self-contained image that runs anywhere without any external dependencies.

```dockerfile
# Frozen mode Dockerfile
FROM sgraph/playwright-service:latest
COPY ./my-playwright-app /app/code
```

The workflow is: develop in dynamic mode (fast iteration), then freeze when stable (self-contained deployment). The two modes use the same bootstrap logic. The only difference is whether the code is already present or needs to be fetched.

## Docker Hub as Distribution

Docker Hub is the primary distribution medium (not just ECR) because:

- Multi-cloud: GCP, Azure, and other providers can pull from Docker Hub natively
- Public availability: anyone can `docker pull sgraph/playwright-service`
- Not locked into AWS
- ECR is used in addition to Docker Hub for AWS-specific deployments (Lambda, Fargate)

## Cloud Portability

The Layer 2 (AWS) variant is AWS-specific because of S3 access. For GCP or Azure, equivalent layers would be needed with their respective storage APIs. But Layer 1 (core) and the dynamic code loading via URL work on any cloud or on-premises environment. The Playwright layer works anywhere Docker runs.

This is a concrete example of the "run everywhere" principle: the same application code runs on Layer 1 (any cloud), Layer 2 (AWS), or Layer 3 (with browser). The layers add capabilities without changing the application.

## CI/CD Pipeline Structure

Each layer has its own pipeline:

```
Layer 1 (core-fastapi)
  |-- Build and test
  |-- Push to Docker Hub: sgraph/core-fastapi:{version}
  |-- Push to ECR: {account}.dkr.ecr.{region}.amazonaws.com/core-fastapi:{version}

Layer 2 (core-fastapi-aws)
  |-- FROM sgraph/core-fastapi:{version}
  |-- Build and test
  |-- Push to Docker Hub + ECR

Layer 3 (playwright-service)
  |-- FROM mcr.microsoft.com/playwright/python:{version}
  |-- COPY Layer 1/2 code
  |-- Build and test
  |-- Push to Docker Hub + ECR
```

Layer 1 changes trigger Layer 2 and Layer 3 rebuilds. Layer 2 changes trigger Layer 3 rebuild. Playwright base image updates trigger Layer 3 rebuild only.
