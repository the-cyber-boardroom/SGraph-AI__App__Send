# Secure Folders with Version Control: How I Solved Agent-to-Agent Communication

Here is a problem nobody talks about enough: when you want two AI agents to collaborate, you have to give them far too many privileges.

Want an agent to share files with another agent? Give it access to your GitHub repo (full access, because you cannot easily scope it). Want it to use the file system? Again, usually unrestricted access. And the worst part: you usually have to give the agent your identity. Your credentials. Your API keys. All because you want two agents to pass some files back and forth.

That felt wrong to me. So I built something different.

## What I Actually Use

I built SG/Send, a zero-knowledge encrypted platform at sgraph.ai, and as part of it, I developed SGit (pip install sgit-ai), which is essentially a signed, encrypted version of Git. It lets you create vaults: secure folders with version control, where the data is encrypted client-side and the server never sees plaintext.

The Git-like workflow (clone, commit, push, pull, branch) turns out to be exactly what agent communication needs. Agents already understand Git. They know how to commit files, organise folder structures, and push changes. So when I give an agent access to a vault via SGit, it just works. No special training, no custom protocols.

Here is what that looks like in practice.

## The Basic Workflow

I start a session with an agent (Claude, Perplexity, whatever the task needs). The agent installs SGit (`pip install sgit-ai`), runs `sgit init`, and creates a vault. From there, the agent creates files, organises them into folders, commits, and pushes.

When the agent is done (or when I want to check progress), I have two options:

**Clone locally.** The agent gives me the vault key, I run `sgit clone <key>` on my laptop, and I have all the files. Full offline access, version history, the lot.

**View online.** I open the vault in the web viewer, which loads only the tree structure (not all the files), so it is fast. I can browse, edit text files, drag and drop new files in, create folders, rename, delete. It is like GitHub but for encrypted content.

Both modes are powerful. The local clone gives me full control. The web viewer gives me speed and convenience.

## Agent-to-Agent: Where It Gets Interesting

The real power shows up when multiple agents collaborate on the same vault.

Here is a workflow I use regularly: I have a Perplexity session doing research on a topic. Perplexity has access to a vault, and it puts its findings in there (reports, data, source links). Then I have a Claude session that clones the same vault, reads the research, and creates infographics and processed outputs from it.

Two agents. One shared vault. No direct connection between them. No shared credentials. No shared file system. The vault is the communication channel.

I have also done a more layered version: I use Claude's web interface (which is better for conversation and iterative thinking) to do analysis and create guidance. That session writes its findings to a vault. Then I point a Claude Code session (which has access to a source repo) at the same vault, and it reads the briefs and starts implementing.

The web session does not have access to code. The code session does not have to re-discover the thinking. The vault carries the context between them.

## Packaging Memory Across Sessions

One of my most powerful use cases is what I call "packaging memory."

When I start a new project, I used to create dev packs: zip files with code standards, role definitions, briefings, background materials. It was messy. Files got lost. Structure was inconsistent.

Now I do it differently. I run a comprehensive session with Claude where I give it access to all my code standards, agentic role definitions, architecture patterns, and project context. The agent creates a detailed folder structure with briefings, analysis, guidance, and background materials. Everything you need to recreate my agentic workflow in a new repo.

I package all of that into a vault. Then, when I start a new Claude Code session, my opening prompt is: "Install sgit-ai, clone this vault, and start reading."

The agent does. It reads the briefings, copies the relevant files to the repo, commits to GitHub, and starts working with the full context. No dragging and dropping files. No pasting long documents into the chat. No repeating instructions. The vault carries the memory.

And it gets even better: I now have a two-way conversation between environments. The Claude Code session can put questions and findings into the vault. I can go back to my original Claude web session, have it clone the repo read-only, review the work, and add comments. Multiple agents, multiple perspectives, one shared vault.

## Why This Works Better Than the Alternatives

The traditional options for agent communication all have significant drawbacks:

**GitHub repos:** Version controlled, but you give the agent full repo access. No encryption. The agent operates under your identity. Cannot scope access to just the communication channel.

**File system:** No version control (unless you add Git, which brings back the GitHub problems). Cannot restrict access easily. No encryption. No remote access.

**Direct API calls between agents:** Requires both agents to be online simultaneously. No persistent state. No audit trail. Complex to set up.

**Shared databases:** Server-side data processing. No encryption (or encryption managed by the server, not the client). Schema rigidity.

The vault approach gives you: version control (every change is a commit), encryption (zero-knowledge, client-side), scoped access (each vault has its own key), audit trail (full commit history), and no identity sharing (the agent gets a vault key, not your credentials).

## The Communication Layer, Not Just File Storage

One thing I have learned is that vaults are not just about storing files. They are about communicating the contents of those files.

I spent time building features specifically for this: a JSON-based page layout system (`_page.json`) that lets you define navigation, sections, and content structure for a vault. HTML pages that can load and render content from other files in the vault. A presenter mode for walking through PDFs, markdown pages, and infographics in sequence.

The point is to create a narrative around the materials, not just dump files in a folder. When I ship a company review, for example, I do not just send the findings. I create a vault with the deck, the report, the data, the evidence, and a page layout that guides the reader through the materials in a logical order. Everything bundled together. Everything encrypted. Everything versioned.

## What Comes Next

All of this works today without the more advanced features that are possible. We have not even added PKI-based access control (where specific agents can only read specific files, and the workflow enforces a sequence of events). We have not added real-time sync (currently you push and pull manually). We have not added automated agent-to-agent handoff protocols.

But even without those features, the basic pattern (encrypted, versioned, scoped vault as the communication channel between agents) has already transformed how I work. I have more continuity between sessions, better context transfer, cleaner collaboration, and a full audit trail of every decision.

If you are working with multiple agents and struggling with how they share information, try this: give them a shared folder with version control. Not a repo (too much access). Not a file system (no encryption, no versioning). A vault.

It is simpler than you think, and it solves a problem that is only going to get bigger as agentic workflows scale.
