# Auth MVP: Social Login and Vault Key Storage on AWS Cognito and GCP

**version** v0.21.2
**date** 18 Apr 2026
**from** Human (project lead)
**to** Developer (lead), Architect, AppSec
**type** Dev brief

## Context

Yesterday's identity research brief (`v0.24.56__brief__identity-research-vault-key-storage.md`) identified the gap: SG/Send does everything except store vault keys. This brief specifies the MVP implementation: build a working social login with vault key storage on two cloud providers (AWS and GCP), all client-side, as reusable Web Components.

## The Core Requirement

**All client-side.** No server-side code. This is non-negotiable and consistent with the entire SG/Send architecture. The user logs in via social login, stores vault keys and preferences in the identity provider's user storage, and manages their vaults. The server never processes authentication tokens or user data in plaintext.

## What To Build

### Two Cloud Providers

| Provider | Service | Why |
|----------|---------|-----|
| **AWS** | Cognito (primary). Also evaluate Identity Centre. | Existing experience. Supports social login. User attributes for key storage. |
| **GCP** | Firebase Auth / Identity Platform | Broad adoption. Good social login support. Explore what storage options exist for user data. |

### Social Login Only

No usernames. No passwords. No email/password registration. Only social login providers:

- Google OAuth (must have)
- GitHub OAuth (must have, developer audience)
- Others as available (Apple, Microsoft, etc., but not required for MVP)

The user authenticates through their existing identity (Google account, GitHub account). We never touch their password. The identity provider handles MFA, session management, and credential security.

### Two Modes

**User mode:** Normal login. The user authenticates, accesses their stored vault keys and preferences, manages their vaults.

**Admin mode:** Elevated access for platform administration. Ability to see how many users are registered, usage statistics, and manage user groups/permissions. Implemented via identity provider groups (Cognito user groups, Firebase custom claims).

Important constraint: the admin should ideally not be able to read user vault keys (they are encrypted). The admin can see metadata (user count, registration dates, activity) but not decrypt user data. This is the zero-knowledge principle applied to the identity layer.

## Architecture

```
Browser (client-side only)
    |
    +--> Social Login Component
    |       |
    |       +--> Google OAuth
    |       +--> GitHub OAuth
    |       |
    |       v
    |    Identity Provider (Cognito / Firebase)
    |       |
    |       +--> User authenticated
    |       +--> Session token returned to browser
    |
    +--> User Data Component
    |       |
    |       +--> Store vault keys (encrypted) in user attributes
    |       +--> Store preferences
    |       +--> Retrieve on login
    |
    +--> Vault Component
            |
            +--> Load vault keys from user data
            +--> Open vaults, collaborate, etc.
```

No server-side code in this flow. The browser talks directly to the identity provider's API (Cognito SDK, Firebase SDK). User data is stored in the identity provider's user attribute storage.

## Vault Key Storage: Where Exactly?

This is the key research question for each provider: how much user-controllable data can we store in the identity provider's user space?

### Option A: Identity Provider User Attributes

Cognito supports custom user attributes. Firebase supports custom claims (limited to 1000 bytes) and Firestore (unlimited). Can we store encrypted vault keys as custom attributes?

Investigate: maximum attribute size, number of attributes, read/write permissions, whether the user (via client-side SDK) can read/write their own attributes without server involvement.

### Option B: Encrypted Data in SG/Send

Store the encrypted vault key bundle as a file in SG/Send itself. The identity provider stores only a pointer (vault ID) and the master decryption key. On login, the client fetches the vault key bundle from SG/Send and decrypts it locally.

This is interesting because it means the identity provider stores minimal data (one master key) and SG/Send stores the bulk (encrypted vault key bundle). The identity provider cannot read the bundle. SG/Send cannot read it either. Only the authenticated client can.

### Option C: Store Encrypted Data in the Identity Provider's Storage

Store encrypted blobs directly in Cognito's user storage or Firebase's Firestore, encrypted with a key derived from the user's social login session. The identity provider has the ciphertext but not the key. The client decrypts after authentication.

Investigate: is there a piece of data that only the authenticated user can access via the social login? For example, a stable user-specific secret from Google OAuth that could be used as a derivation input for encryption. If such a value exists, it enables a true zero-knowledge user data store where even SG/Send never sees the data.

### Option D: Master Password

The user sets a master password (separate from their social login) that encrypts all vault keys. The encrypted bundle is stored anywhere (Cognito attributes, Firebase, SG/Send). Without the master password, nobody (including us, including the identity provider) can decrypt.

Trade-off: this adds a password to a "no passwords" system. But it provides the strongest zero-knowledge guarantee.

**Recommendation for MVP:** Start with Option A (user attributes) for simplicity. Evaluate Option B as the production path. Document Option C and D as future options.

## Reusable Web Components

This must be built as modular, reusable Web Components, not a monolithic auth page. Each component is independent and can be dropped into any tool or page:

| Component | What It Does |
|-----------|-------------|
| `<sg-login>` | Social login button(s). Handles OAuth flow. Returns session token. |
| `<sg-user-data>` | Read/write user attributes (vault keys, preferences). Requires active session. |
| `<sg-vault-manager>` | Lists user's vaults, load/open/create vaults using stored keys. |
| `<sg-admin-panel>` | Admin view: user count, registration stats, group management. Requires admin group. |

The composability requirement is critical. If the video recorder tool (from today's earlier brief) wants to add "save to your account," it should just drop in `<sg-login>` and `<sg-user-data>`. No custom auth code. No server-side integration.

## The "No Account" Compatibility

SG/Send works without accounts today. That must remain true. The account system is additive:

- Without account: use SG/Send exactly as today. Tokens, file sharing, vaults with manual key management.
- With account: same features, plus persistent vault key storage, preferences, and the convenience of not losing keys.

The account is an enhancement, not a gate.

## Research Questions

1. **Cognito user attributes:** What is the maximum size? Can client-side SDK read/write without server? How many custom attributes?
2. **Cognito vs Identity Centre:** Is Identity Centre suitable for consumer-facing social login, or is it enterprise-only?
3. **Firebase custom claims vs Firestore:** What is the right storage layer for user data in GCP?
4. **Google OAuth user-specific data:** Is there a stable, user-specific value from Google OAuth that could serve as encryption key derivation input?
5. **Cross-provider portability:** If a user logs in via Google on Cognito today and Firebase tomorrow, can we migrate their data?
