## Context

The Playwright container from the 16 Apr arch brief (`v0.24.56__arch-brief__playwright-service-decoupled-image.md`) has been taking shape and is solving real problems. This brief elevates it from an internal service to a standalone sub-product worth launching on its own.

The core insight: a self-contained, API-driven browser execution service that takes instructions (not code) and returns screenshots, PDFs, and captured data does not exist as a simple, ready-to-run product today. That gap is the opportunity.

## What It Is

A self-contained container that gives any agent (or human) the ability to control a browser via API. No setup. No state. No code required from the caller.

You spin it up (as a container, a Lambda function, or an AMI), you issue instructions (go to page, click, fill form, take screenshot, capture PDF), and you get results back. That is it.

**The one-line pitch: "The browser for agents."**

## Why This Does Not Exist

Incredibly, there is still no simple container that gives you a FastAPI out of the box with:

- Programmatic browser control via HTTP endpoints (not code, instructions)
- Screenshot, image, and PDF capture
- Sequence execution (run a list of steps, get results)
- Self-contained deployment (no external dependencies, no setup, no state)
- API-driven (not a library you import, a service you call)

Playwright itself exists. Headless Chrome exists. But the "spin it up, call the API, get results" wrapper does not exist as a clean, deployable product.

## Product Packaging

### Deployment Targets (run everywhere)

| Target | How | Use Case |
|--------|-----|----------|
| Local Docker | `docker run -p 8000:8000 ...` | Developer laptop, local agent browsing |
| AWS ECR | Already building this | Private deployments, CI/CD integration |
| AWS Lambda (container) | Container image deployment | Serverless, per-execution billing, auto-scaling |
| AWS AMI | New: package as a marketplace AMI | One-click launch, AWS Marketplace listing |
| Behind load balancer | Multiple containers behind ALB/NLB | High-throughput, multi-agent scenarios |

The AMI path is specifically interesting for learning the AWS Marketplace flow: package it, list it, set pricing, handle billing. This is a self-contained product that is simple enough to be the first Marketplace experiment.

### Revenue Model

**Per-execution billing.** Each API call (or sequence of calls) is a billable unit. This aligns with the usage-based pricing principle from the OWASP presentation pack:

- Screenshot: X pence
- PDF capture: X pence
- Sequence execution (N steps): X pence per step
- Video recording: X pence per minute

Alternatively, for AMI/container deployment, the customer pays for their own compute and the product is free (or has a small licence fee). This is the "run everywhere" model: we provide the software, they provide the infrastructure.

### Open Source

The product itself is open source (consistent with the core primitives). Revenue comes from:

- Hosted service (we run it, you pay per execution)
- AWS Marketplace AMI (one-click deploy to their account)
- Support and customisation

## Key Features

### Already Working

- Container with Playwright + Chrome
- FastAPI service with browser control endpoints
- Screenshot capture
- Deploying to ECR
- Sequence execution (issue instructions, get results)

### Needed for Product Launch

**Man-in-the-middle proxy support.** Native support for MITM proxy (e.g. mitmproxy) to capture 100% of traffic generated by the browser. This enables:

- Full traffic logging for audit and compliance
- Request/response replay and analysis
- Recording everything an agent does online (the "100% logs" capability)

This could be a second container (the proxy) wired to the browser container, or integrated into the same image. The combination is powerful: you can record every HTTP request, every response, every redirect, every asset load. For agent safety and observability, this is essential.

**Logging and capture.** Comprehensive logging of every action taken, every page visited, every element interacted with. This is the observability layer that makes the difference between "an agent browsed the internet" and "here is exactly what the agent did, step by step, with evidence."

**Simple website.** A landing page for the product: what it does, how to deploy it, pricing, API docs. Could be a single page on sgraph.ai or a dedicated subdomain.

**Billing integration.** Wire up Stripe for the hosted service. AWS Marketplace billing for the AMI. This is a learning exercise as much as a revenue play.

## The Agent Safety Angle

This product provides something that does not exist cleanly today: a safe, controlled, observable way for agents to browse the internet.

When you give an agent direct browser access, you have no control over what it does and no logs of what it did. With this service:

- Every action goes through the API (controllable)
- Every action is logged (observable)
- The browser runs in an isolated container (contained)
- Traffic can be routed through a MITM proxy (auditable)
- The container is stateless (no persistent access)

For enterprise agent deployments, this is not a nice-to-have. It is a compliance requirement.

## The "Run Everywhere" Proof

This product is also a clean example of the "run everywhere" principle from the OWASP presentation pack:

- Runs on a laptop (local Docker)
- Runs serverless (Lambda)
- Runs in a customer's cloud (AMI in their AWS account)
- Runs behind a load balancer (scalable)
- Runs offline (local container with no internet, for testing)
- No database, no state, no setup

It is the simplest possible end-to-end product that demonstrates the full deployment model. What we learn shipping this (AMI packaging, Marketplace listing, per-execution billing, multi-cloud deployment) applies directly to SG/Send and every future product.
