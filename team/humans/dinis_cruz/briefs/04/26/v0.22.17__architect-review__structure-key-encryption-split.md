# Architect Review: Structure Key Encryption Split

**Version:** v0.22.17 | **Date:** 26 April 2026 | **Scope:** Vault encryption layer — structure key / read key separation
**Basis:** Code inspection of `SGit-AI__CLI` (`Vault__Crypto`, `Vault__Sub_Tree`, `Vault__Ref_Manager`, `Vault__Branch_Manager`, `Vault__Commit`, `Vault__Dump`) and `SGraph-AI__App__Send` (`Service__Vault__Pointer`, `Routes__Vault__Pointer`)
**Status:** DESIGN DECISION — action required across four teams

---

## Executive Summary

The vault currently derives a `structure_key` from the `read_key` via HKDF-SHA256, but encrypts **all** vault objects — structural metadata and file content alike — with the `read_key`. The `structure_key` is therefore effectively unused: sharing it with a third party grants no additional decryption capability, but also provides none of the intended read-only structural access.

This review defines the correct encryption split, proposes a backwards-compatible migration strategy using AES-GCM's `InvalidTag` exception as a clean fallback signal, and issues implementation briefs to the four teams responsible for applying the change.

**Verdict: the design intent is sound; the implementation is incomplete. This is a four-team change with no flag day required.**

---

## 1. Current State

### 1.1 Key Derivation (Correct)

Two derivation paths exist, both producing three keys:

**Standard vault key (`passphrase:vault_id`):**
```
read_key      = PBKDF2(passphrase, salt='sg-vault-v1:{vault_id}',        600k iterations)
write_key     = PBKDF2(passphrase, salt='sg-vault-v1:write:{vault_id}',  600k iterations)
structure_key = HKDF(read_key,     info=b'sg-vault-v1:structure-key')
```

**Simple token (`word-word-NNNN`):**
```
aes_key       = PBKDF2(token,      salt=b'sgraph-send-v1',               600k iterations)
read_key      = HKDF(aes_key,      info=b'vault-read-key')
write_key     = HKDF(aes_key,      info=b'vault-write-key')
structure_key = HKDF(read_key,     info=b'sg-vault-v1:structure-key')
```

The one-way properties hold in both paths: `structure_key` cannot be reversed to `read_key`; `read_key` cannot be reversed to the passphrase or token.

### 1.2 Encryption (Incorrect — Everything Uses read_key)

All vault objects are currently encrypted with `read_key`:

| Object | Location | Fields | Should use |
|--------|----------|--------|------------|
| Refs | `bare/refs/` | `commit_id` | `structure_key` |
| Branch index | `bare/indexes/` | `branch_id`, `name`, `type`, `head_ref_id`, `created_at`, `key_ids` | `structure_key` |
| Commits | `bare/data/obj-cas-imm-*` | `tree_id`, `parents`, `timestamp_ms`, `branch_id`, `signature` | `structure_key` |
| Commits | `bare/data/obj-cas-imm-*` | `message_enc` | `read_key` — commit messages may contain sensitive intent |
| Trees | `bare/data/obj-cas-imm-*` | tree object wrapper | `structure_key` |
| Tree entries | inside tree objects | `blob_id`, `tree_id`, `large` | `structure_key` |
| Tree entries | inside tree objects | `name_enc`, `size_enc`, `content_hash_enc`, `content_type_enc` | `read_key` — filenames and hashes are sensitive |
| Blob content | `bare/data/obj-cas-imm-*` | raw file bytes | `read_key` |
| Public PKI keys | `bare/keys/key-rnd-imm-*` | public key PEM | `structure_key` |
| Named branch private key | `bare/keys/key-rnd-imm-*` | private key PEM | `write_key` (separate issue — see Section 5) |

### 1.3 The Consequence

Sharing the `structure_key` today grants zero decryption capability — everything fails with `InvalidTag` because nothing is encrypted with it. The `dump_with_structure_key` path in `Vault__Dump` passes the structure_key as the `read_key` argument to `dump_local`, which then fails on every decryption attempt and records errors throughout the result. The feature is inert.

## 2. Target Design

### 2.1 What Structure Key Access Should Reveal

A holder of the `structure_key` (and nothing else) should be able to reconstruct:

- The full commit graph: all commits, their parents, timestamps, branch IDs, tree IDs
- The complete directory topology at each commit: which folders and files exist, their object IDs
- Branch names, types, creation times, HEAD positions
- All ref file contents (which commit each ref points to)
- Public PKI keys for all branches

A holder of the `structure_key` should NOT be able to learn:

- Filenames, file sizes, content hashes, or MIME types
- Commit messages
- File content
- The `read_key`, `write_key`, or original passphrase

### 2.2 Encryption Assignment

| Category | Encrypt with | Rationale |
|----------|-------------|-----------|
| Refs | `structure_key` | Graph topology only |
| Branch index | `structure_key` | Graph topology only |
| Commit object (excl. message_enc) | `structure_key` | Graph topology only |
| `message_enc` field | `read_key` | Commit messages often contain sensitive intent |
| Tree object wrapper | `structure_key` | Graph topology only |
| Tree entry `blob_id`, `tree_id`, `large` | `structure_key` | Object IDs are opaque; graph topology |
| Tree entry `name_enc` | `read_key` | Filenames are sensitive |
| Tree entry `size_enc` | `read_key` | Sizes can identify known files |
| Tree entry `content_hash_enc` | `read_key` | Hashes enable rainbow-table attacks on known content |
| Tree entry `content_type_enc` | `read_key` | MIME types partially reveal content nature |
| Blob content | `read_key` | The actual file payload |
| Public PKI keys | `structure_key` | Public by nature; needed for signature verification |
| Named branch private key | `write_key` | See Section 5 |

### 2.3 Revised Commit Object Encoding

Because `message_enc` must use `read_key` while the rest of the commit uses `structure_key`, commits require a two-layer approach:

```
commit_core = {
    schema, tree_id, parents, timestamp_ms, branch_id, signature
}
encrypted_core   = AES-GCM(structure_key, commit_core_json)   → stored as obj-cas-imm-*

message_enc      = AES-GCM(read_key, commit_message)          → stored as field inside encrypted_core
                   OR: stored as a separate companion object alongside the commit
```

**Recommended approach:** Keep `message_enc` as an opaque encrypted field inside the commit JSON — the structure_key decrypts the commit object and sees `message_enc` as an opaque base64 string it cannot read. This requires no schema change; the field is already `Safe_Str__Encrypted_Value`. The structure_key holder sees the field exists but cannot decrypt it.

Similarly for tree entries: the structure_key decrypts the tree object and sees `blob_id`, `tree_id`, `large` in plaintext (within the encrypted tree envelope), while `name_enc`, `size_enc`, `content_hash_enc`, `content_type_enc` remain as opaque base64 blobs it cannot read.

## 3. Backwards Compatibility Strategy

### 3.1 The Fallback Signal

AES-256-GCM provides a clean, cryptographically reliable failure signal. Decrypting with the wrong key raises `InvalidTag` — not corrupt output, not a generic exception. This is deterministic and safe to catch:

```python
from cryptography.exceptions import InvalidTag

def decrypt_structural(self, structure_key: bytes, read_key: bytes, data: bytes) -> bytes:
    """Try structure_key first (new vaults); fall back to read_key (old vaults)."""
    try:
        return self.decrypt(structure_key, data)
    except InvalidTag:
        return self.decrypt(read_key, data)
```

This is the correct pattern for all structural object reads (refs, branch index, commits, trees, public keys). It is NOT used for blob content or tree entry metadata reads — those always use `read_key` only.

### 3.2 Migration Behaviour

| Vault age | Write path | Read path | Result |
|-----------|-----------|-----------|--------|
| Old vault | read_key (unchanged) | try structure_key → InvalidTag → fall back to read_key | Works transparently |
| New vault | structure_key for structural objects | try structure_key → succeeds | Works correctly |
| Mixed vault | New commits use structure_key; old commits use read_key | per-object fallback | Works per-object |

No migration scripts. No flag days. No version checks. Old and new objects coexist naturally — each object is decrypted correctly regardless of when it was written.

### 3.3 Write Path Change

New structural objects are written with `structure_key` immediately. The fallback on read is a permanent feature — not a temporary migration shim — because old objects will always exist in cloned vaults.

## 4. Implementation Briefs by Team

---

### 4.1 API Team (SGraph-AI__App__Send)

**Scope:** Server-side — `Service__Vault__Pointer`, `Routes__Vault__Pointer`

**Impact: None required for correctness.**

The server stores and serves opaque encrypted blobs. It has no knowledge of which key was used to encrypt any object, and the `write_key_hash` comparison is key-agnostic. The encryption split is entirely a client-side concern.

**One optional enhancement worth considering:**

The `/api/vault/zip/{vault_id}` endpoint currently requires the `write_key`. Since the zip is a raw dump of all ciphertext (still encrypted, not decrypted server-side), this endpoint could optionally accept a `structure_key` or `read_key` header instead — allowing read-only clients to pull a full vault snapshot without holding the write_key. This is a separate design decision, not a blocker for this change.

**Action:** No changes required. Acknowledge this review.

### 4.2 CLI Team (SGit-AI__CLI)

**Scope:** `Vault__Crypto`, `Vault__Ref_Manager`, `Vault__Branch_Manager`, `Vault__Commit`, `Vault__Sub_Tree`, `Vault__Dump`, `Vault__Batch`, `Vault__Sync`

**This is the primary implementation team for this change.**

#### 4.2.1 Add `decrypt_structural` to `Vault__Crypto`

```python
from cryptography.exceptions import InvalidTag

def decrypt_structural(self, structure_key: bytes, read_key: bytes, data: bytes) -> bytes:
    """Decrypt a structural object: try structure_key (new), fall back to read_key (old)."""
    try:
        return self.decrypt(structure_key, data)
    except InvalidTag:
        return self.decrypt(read_key, data)

def decrypt_metadata_structural(self, structure_key: bytes, read_key: bytes,
                                 b64_ciphertext: str) -> str:
    """decrypt_metadata variant with structural fallback."""
    from base64 import b64decode
    data = b64decode(b64_ciphertext)
    return self.decrypt_structural(structure_key, read_key, data).decode('utf-8')
```

#### 4.2.2 Update `Vault__Ref_Manager`

- `write_ref`: encrypt with `structure_key`
- `read_ref`: use `decrypt_structural(structure_key, read_key, ...)`
- `encrypt_ref_value`: encrypt with `structure_key`
- All call sites must pass both `structure_key` and `read_key`

#### 4.2.3 Update `Vault__Branch_Manager`

- `save_branch_index`: encrypt with `structure_key`
- `load_branch_index`: use `decrypt_structural`
- `store_public_key` (in `Vault__Key_Manager`): encrypt with `structure_key`
- `load_public_key`: use `decrypt_structural`
- All call sites must pass both keys

#### 4.2.4 Update `Vault__Commit`

The commit object has a mixed encryption requirement:

```python
def create_commit(self, read_key, structure_key, tree_id, ...):
    # message_enc stays encrypted with read_key (sensitive)
    if message:
        message_enc = self.crypto.encrypt_metadata(read_key, message)

    commit = Schema__Object_Commit(
        tree_id=tree_id, parents=parents,
        timestamp_ms=timestamp_ms, branch_id=branch_id,
        message_enc=message_enc,   # opaque to structure_key holder
        ...
    )
    commit_data = json.dumps(commit.json()).encode()

    if signing_key:
        # sign before outer encryption
        sig_raw = self.pki.sign(signing_key, commit_data)
        commit.signature = base64.b64encode(sig_raw).decode()
        commit_data = json.dumps(commit.json()).encode()

    # Outer envelope uses structure_key
    encrypted_commit = self.crypto.encrypt(structure_key, commit_data)
    return self.object_store.store(encrypted_commit)

def load_commit(self, commit_id, read_key, structure_key):
    ciphertext = self.object_store.load(commit_id)
    # Try structure_key first (new), fall back to read_key (old)
    commit_data = self.crypto.decrypt_structural(structure_key, read_key, ciphertext)
    return Schema__Object_Commit.from_json(json.loads(commit_data))
    # Note: message_enc field remains opaque to structure_key holder — caller
    # must use read_key to decrypt it separately if needed
```

#### 4.2.5 Update `Vault__Sub_Tree`

The tree object and tree entry structural fields use `structure_key`; metadata fields stay on `read_key`:

```python
def _store_tree(self, tree, read_key, structure_key):
    # Tree entries: split encryption by field sensitivity
    # blob_id, tree_id, large — already plaintext within the JSON
    # name_enc, size_enc, content_hash_enc, content_type_enc — use read_key (unchanged)
    tree_json = json.dumps(tree.json()).encode()
    encrypted_tree = self.crypto.encrypt(structure_key, tree_json)  # outer uses structure_key
    return self.obj_store.store(encrypted_tree)

def _load_tree(self, tree_id, read_key, structure_key):
    ciphertext = self.obj_store.load(tree_id)
    tree_data  = self.crypto.decrypt_structural(structure_key, read_key, ciphertext)
    return Schema__Object_Tree.from_json(json.loads(tree_data))
    # name_enc, size_enc etc. remain opaque — caller uses read_key to decrypt them
```

Blob content encryption is **unchanged** — always `read_key`:
```python
encrypted = self.crypto.encrypt(read_key, content)   # unchanged
```

#### 4.2.6 Update `Vault__Dump`

`dump_with_structure_key` now works correctly without special handling — the fallback in `decrypt_structural` handles old vaults; new vaults decrypt directly with `structure_key`. The dump intentionally never decrypts `name_enc`, `size_enc`, `content_hash_enc`, `content_type_enc`, or `message_enc` — so structure_key access reveals commit graph and directory topology but not filenames, sizes, or messages.

#### 4.2.7 Propagate Both Keys Through `Vault__Components` and `Vault__Sync`

`Vault__Components` must carry `structure_key` alongside `read_key`. `_init_components` derives it once from `read_key` via `crypto.derive_structure_key(read_key)` and stores it. All downstream call sites receive both keys.

**Estimated blast radius:** `Vault__Sync` (primary orchestrator), `Vault__Batch`, `Vault__Diff`, `Vault__Revert`, `Vault__Merge`, `Vault__Stash`, `Vault__GC`, `Vault__Branch_Switch` — any file currently passing `read_key` to structural operations needs updating.

**Testing requirement:** All existing unit tests must pass with both old-format vaults (read_key fallback path) and new-format vaults (structure_key path). Add a specific test that creates an old vault, upgrades the write path, and verifies both old and new objects read correctly.

### 4.3 Vault Web Team (sgraph_ai_app_send__ui__vault)

**Scope:** `sg-vault-crypto.js`, `sg-vault.js`, `sg-vault-ref-manager.js`, `sg-vault-object-store.js`, and all Web Components that call into these

**This team implements the same split as the CLI team, using Web Crypto API equivalents.**

#### 4.3.1 Add `structure_key` Derivation to `SGVaultCrypto`

```javascript
static async deriveKeys(passphrase, vaultId) {
    // ... existing PBKDF2 derivation of readKey and writeKey ...

    // Add structure_key derivation via HKDF
    const structureKeyBits = await crypto.subtle.deriveBits(
        { name: 'HKDF', hash: 'SHA-256', salt: new Uint8Array(0),
          info: new TextEncoder().encode('sg-vault-v1:structure-key') },
        readKeyRaw,   // the raw read key material before importKey
        256
    )
    const structureKey = await crypto.subtle.importKey(
        'raw', structureKeyBits, { name: 'AES-GCM' }, false, ['encrypt', 'decrypt']
    )

    return { readKey, writeKey, structureKey, ... }
}
```

Note: `structureKey` must be derived from the raw PBKDF2/HKDF output bytes **before** `importKey` wraps them as a non-extractable CryptoKey.

#### 4.3.2 Add `decryptStructural` Helper

```javascript
static async decryptStructural(structureKey, readKey, data) {
    try {
        return await SGVaultCrypto.decrypt(structureKey, data)
    } catch (e) {
        // AES-GCM authentication failure → old vault, fall back to readKey
        if (e.name === 'OperationError') {
            return await SGVaultCrypto.decrypt(readKey, data)
        }
        throw e
    }
}
```

Note: Web Crypto API surfaces AES-GCM authentication failures as `DOMException` with `name === 'OperationError'`, not `InvalidTag`. Catch specifically on `OperationError` to avoid swallowing genuine errors.

#### 4.3.3 Update `sg-vault.js` and `sg-vault-ref-manager.js`

- Ref reads/writes: use `structureKey` for encryption, `decryptStructural` for decryption
- Branch index reads/writes: same
- Commit object outer envelope: use `structureKey`; `message_enc` field stays `readKey`
- Tree object outer envelope: use `structureKey`; `name_enc`, `size_enc`, `content_hash_enc`, `content_type_enc` stay `readKey`
- Blob content: unchanged, always `readKey`

#### 4.3.4 Store `structureKey` in `sg-vault.js` State

`sg-vault.js` already stores `_readKey` and `_writeKey`. Add `_structureKey` alongside these. Pass it through to ref manager, commit manager, and object store where structural operations occur.

**Testing requirement:** Verify that an old vault (all objects encrypted with `read_key`) still loads correctly via the `decryptStructural` fallback. Verify that a new vault (structural objects encrypted with `structure_key`) loads correctly and that a read-only consumer holding only `structure_key` can traverse the commit graph and directory topology but cannot decrypt filenames or file content.

### 4.4 Vault Tools / Components Team (tools.sgraph.ai)

**Scope:** Any tool or component that reads from or writes to vault structural objects — `sg-vault-picker`, vault browse components, infographic generator vault integration, PlaybookLM vault output, any component using `sg-vault.js` or the vault API directly

**This team is a consumer of the Vault Web Team's updated `sg-vault.js`.**

#### 4.4.1 No Direct Crypto Changes Required

If the Tools team uses `sg-vault.js` as a library (the expected pattern), all crypto changes are encapsulated there. The `structure_key` derivation and `decryptStructural` fallback are invisible to component consumers.

#### 4.4.2 New Capability: Read-Only Vault Access

Once the Vault Web Team ships the split, the Tools team can implement genuine read-only vault consumers. A component that receives only a `structure_key` (not the full vault key) can:

- Traverse the commit graph
- List directories and files by object ID
- Render vault topology (branch graph, commit history)

It **cannot** decrypt filenames, file sizes, or file content — and therefore cannot accidentally expose sensitive content even if the component has a bug.

**Recommended pattern for read-only tool components:**

```javascript
// Read-only consumer: receives structure_key only, never the full vault key
const vaultView = await SgVault.openWithStructureKey(structureKeyHex, vaultId)
// vaultView.listFiles() → returns blob_ids and tree_ids (opaque)
// vaultView.getFileName() → throws — no read_key available
// vaultView.readFile()   → throws — no read_key available
```

This is the correct API surface for components like `sg-vault-browser` when used in a context where the consumer should not have content access (e.g. a shared team dashboard, a public commit history viewer, an audit log tool).

#### 4.4.3 Action Items

- Audit all tools that currently require a full vault key — identify which ones actually need content access vs. structural access only
- For structural-only consumers: update to accept `structure_key` instead of full vault key once Vault Web Team ships
- For content consumers: no change required — they continue to use full vault key as today
- Update tool documentation to distinguish the two access modes

## 5. Related Issue: Named Branch Private Key Encryption

During the code review that produced this document, a separate concern was identified: the named branch private key is currently encrypted with `read_key` and stored in `bare/keys/` on the server. Every clone downloads it. This means sharing the `read_key` also exposes the named branch private signing key.

**Current impact: low.** Commit signature verification (`verify_commit_signature`) exists in the code but is never called during push, pull, or clone. The signing key is not currently enforced as an access gate.

**Future impact: significant.** If signature verification is enforced server-side or client-side, the named branch private key should be gated behind `write_key`, not `read_key`. A read-only consumer should not be able to forge commits.

**Recommendation:** When signature verification is activated, re-encrypt the named branch private key with `write_key` in the same migration pass as this change. The `InvalidTag` fallback strategy applies identically.

This is noted here for the Historian to track. It is not a blocker for the current change.

## 6. Summary of Decisions

| Decision | Rationale |
|----------|-----------|
| Structural objects encrypted with `structure_key` | Enables read-only structural access without content exposure |
| `message_enc` stays on `read_key` | Commit messages may contain sensitive intent |
| Tree entry metadata (`name_enc` etc.) stays on `read_key` | Filenames and hashes are sensitive |
| Blob content stays on `read_key` | The actual file payload |
| `InvalidTag` / `OperationError` used as fallback signal | AES-GCM authentication failure is cryptographically reliable and distinct |
| Fallback is permanent, not a migration shim | Old objects always exist in cloned vaults — fallback must work indefinitely |
| No migration scripts | Per-object fallback makes them unnecessary |
| Named branch private key re-keying deferred | Low current impact; re-key when signature verification is activated |
