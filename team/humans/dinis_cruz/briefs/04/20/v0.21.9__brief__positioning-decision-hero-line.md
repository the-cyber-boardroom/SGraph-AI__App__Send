# Positioning Decision: "Version-Controlled, Client-Encrypted Vaults. For Humans and Agents."

**from** Human (project lead)

## The Decision

The hero positioning line for sgraph.ai, send.sgraph.ai, and sgit.ai is:

**Version-controlled, client-encrypted vaults. For humans and agents.**

Nine words. This is the tagline. This ships to the website. This goes on every pitch deck. This is the opening statement.

## Why This Works

The previous positioning ("zero-knowledge encrypted file sharing") had a fundamental problem: not enough people have the practical, felt pain of "I need to send encrypted data." It does not ring. It does not hit a problem space that makes people stop and pay attention.

This new line solves that by naming four things that are genuinely different about what we do, each word earning its place:

### Version-controlled

This is the differentiator that leads. Every encrypted storage product claims "encrypted" first. Leading with "version-controlled" earns the technical reader's attention because it signals git-semantics in the first word. It immediately distinguishes from Dropbox and Google Drive (snapshot semantics, no real history). It names the feature that the last two weeks of vault development have made real: folders with full commit history, branches, push/pull.

### Client-encrypted

Not "encrypted." Not "zero-knowledge." "Client-encrypted."

Every product in the category claims "encrypted." The technical buyer's next thought is always "encrypted where?" At rest? In transit? End-to-end? "Client-encrypted" answers the question before it is asked. It means: the encryption happens on the client, not the server. The server cannot read the data. That is logically equivalent to "zero-knowledge" for any technical reader, but without the jargon that causes anxiety in non-technical audiences.

This is the load-bearing word in the line. It forecloses the biggest competitive ambiguity in one word.

### Vaults

Not "folders" (Dropbox territory). Not "password vaults" (1Password territory). "Vaults" carves a third category: encrypted, versioned object stores with git-like semantics. Plural signals platform (many vaults, not one product). The word has the right connotation: a place where you put sensitive, valuable things.

### For humans and agents

Names agents as first-class citizens of the same infrastructure. This is 18-24 months ahead of mainstream acceptance, but for the early audience (technical, security-savvy, AI-adjacent), it is a sophistication signal that self-selects correctly.

The proof is real: Claude Code, Perplexity, SGit CLI, and the vault web UI all collaborating on the same vault simultaneously. Agents creating dev packs in vaults. Agents reading briefings from vaults. The Playwright container reading configuration from vaults and writing output to vaults. This is not aspirational. This is what we do.

## What This Replaces

| Prior positioning | Status |
|-------------------|--------|
| "Zero-knowledge encrypted file sharing" | **Replaced.** "Zero-knowledge" removed (jargon). "File" expanded to "vaults." Version control added. |
| "The trust boundary is the cryptography, not the client" | **Kept.** Different job. This is the thesis for decks. The hero line is the opening statement on the site. |
| "Like WeTransfer, but nobody can see what you're sending" | **Different audience.** Consumer framing for future user acquisition. Different door of the same building. |

These do not compete. They serve different audiences on different surfaces.

## The Website: Show It In Action

The hero line establishes the claim. The rest of the page must prove it. The website should immediately show the product working, not explain the architecture.

### Use Cases to Feature (Drawn from Real Workflows)

**1. The OWASP presentation.** Created a pack of 8 articles as a vault. Shared via token at a live event. Attendees browsed the materials online. The vault was the delivery mechanism and the demo.

**2. The startup security review.** Codified a review standard, ran it via agentic workflow, delivered findings as a vault: deck, report, data, evidence, all bundled with a page layout that guides the reader through the narrative. Shipped as a single share token.

**3. The healthcare partner demo.** GP health scan questionnaire, all client-side JavaScript, answers stored in encrypted vault, LLM analysis, doctor collaboration via push/pull, published summary via share token. The server never saw the health data.

**4. Multi-agent collaboration.** Perplexity researching a topic, writing findings to a vault. Claude reading the findings and creating infographics. Two agents, one shared vault, no shared credentials.

**5. The Playwright + vault workflow.** Agent browsing the web via Playwright container, configuration loaded from vault (including secrets injected via MITM proxy without the agent knowing them), output screenshots and traffic logs written to vault. Full observability, full encryption.

Each use case should be a section on the website: one-line description, a screenshot or short video, and a "try it" link (share token to a real vault with the materials).

### Subhead Candidates (To Be Tested)

The hero line is decided. The subhead is not. Candidates:

1. "Every commit, every share, every agent: encrypted in your browser. The server is blind."
2. "Git semantics, browser-grade encryption, multi-agent ready."
3. "Encrypt locally. Version globally. Share safely. Across humans, CLIs, and AI agents."

Test these with real visitors. The subhead decision is test-driven, not committee-driven.

## Known Trade-offs

### Audience self-selection

"For humans and agents" will land perfectly with engineering founders, AI-adjacent investors, and security-savvy early customers. It will confuse everyone else. That is a feature at this stage: the line self-selects for the right early audience and filters out the wrong one. The mainstream audience gets a different surface later.

### "Vaults" as the primary frame

Using "vaults" as the hero word means vaults are the front-door concept. The other layers (SG/Send file transfer, data rooms, SGit CLI, PKI) get progressively disclosed inside the site. This is a positioning commitment: vaults first, everything else second.

### The "no accounts" compatibility

The site still needs to communicate that you can use SG/Send without an account. The vault positioning does not conflict with this: "Version-controlled, client-encrypted vaults" describes what the technology does. "No account required to receive" describes how it works. Both are true.

## Shallow Clone: A Supporting Feature

SGit now supports shallow clones: access a vault's file listing and metadata without downloading all the objects. This is the CLI equivalent of the web UI's tree-only loading. It reinforces the "vaults that work at scale" message: even from the command line, you can browse a vault with 200MB of files without cloning everything.

This should be documented in the SGit CLI documentation and mentioned in the "how it works" section of the website.

## Hyphenation Convention

Both compound modifiers get hyphens: "version-controlled" and "client-encrypted." This is standard English for compound adjectives preceding a noun. The canonical form is hyphenated. No-hyphen forms can be used in logo-adjacent contexts where visual cleanliness matters.

Rule: pick one and be consistent. Do not mix hyphenated and non-hyphenated in the same document.
