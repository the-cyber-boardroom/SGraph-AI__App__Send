## The Strategy

Ship three AMIs to the AWS Marketplace (and potentially other container marketplaces). Each AMI is one of the three agentic primitives from today's architecture brief. Together, they form a platform that anyone can deploy to build secure agentic workflows.

This is not just a distribution channel. It is the product strategy: package the primitives, make them one-click deployable, charge per use or per instance, and learn the end-to-end marketplace flow (packaging, pricing, billing, marketing, partner discovery).

## The Three AMIs

### AMI 1: Playwright Service (Browser Automation)

**Status:** CI pipeline already baking this AMI. Already launchable. Needs further testing and battle hardening.

**What it ships with:**
- Playwright + Chrome container
- FastAPI service with browser control API (direct actions + sequence execution + screenshot simple API)
- MITM proxy baked in as sidecar
- Full traffic logging and observability
- API key authentication

**What the customer gets:** Spin up the AMI, get a browser automation API on a known port. Send JSON instructions, get screenshots, PDFs, network dumps. MITM proxy captures 100% of traffic. No setup, no state, no dependencies.

**Use cases:** Agent browser access, QA automation, web scraping with full audit trail, screenshot services, security testing.

### AMI 2: SG/Send Server (Encrypted Virtual File System)

**Status:** Needs to be packaged as an AMI. The server code already exists and supports Memory-FS with in-memory, disk, and S3 backends.

**What it ships with:**
- SG/Send server (FastAPI, the same code that runs sgraph.ai)
- Vault support (create, read, write, branch, push, pull)
- The send/receive UI (upload files, get tokens, paste tokens, download files)
- Storage backends: in-memory (ephemeral), disk (persistent on EBS), S3 (scalable)

**What the customer gets:** Spin up the AMI, get a zero-knowledge encrypted file sharing and vault server. Send files, create vaults, collaborate. All client-side encryption. The server never sees plaintext. Runs inside their AWS account, their VPC, their security controls.

This is also the SGit server: the backend that SGit CLI talks to. With this AMI, a customer has a complete encrypted vault infrastructure inside their own environment. No data leaves their account.

**Use cases:** Encrypted file transfer, secure data rooms, agent-to-agent communication, vault-based collaboration, development pack distribution, agentic workflow state management.

**Naming options:**
- "SG/Send Server" (file sharing focus)
- "SGit Server" (vault/version control focus)
- Or ship both names pointing to the same AMI with different front-end websites. Same backend, different positioning.

### AMI 3: Tools Suite (Client-Side AI Tools)

**Status:** Most tools already work client-side. Needs packaging as a self-contained AMI with a landing page.

**What it ships with:**
- All tools from tools.sgraph.ai: infographic generator, LLM chat, page layout renderer, video editor, and others as they ship
- Most tools run entirely client-side (JavaScript in the browser)
- Some tools need the Playwright API (browser automation tasks) or the SG/Send API (saving output)
- LLM access via OpenRouter or local Ollama

**What the customer gets:** Spin up the AMI, get a tools portal. Infographic generation, document processing, LLM-powered workflows. Client-side execution means sensitive data stays in the browser. Connect to local LLMs (Ollama) for fully air-gapped operation.

**Use cases:** AI-powered document processing, infographic generation, content creation, presentation workflows, all deployable inside a customer's environment.

## The Platform Story

Individually, each AMI is useful. Together, they are a platform for secure agentic development:

```
AMI 1: Playwright          AMI 2: SG/Send Server       AMI 3: Tools Suite
(Browser for Agents)       (Encrypted VFS)             (AI Tools)
    |                          |                           |
    +--> Browse web            +--> Store files             +--> Generate content
    +--> Take screenshots      +--> Create vaults           +--> Process documents
    +--> Execute sequences     +--> Agent communication     +--> LLM workflows
    +--> Full traffic audit    +--> Version control         +--> Infographics
    |                          |                           |
    +----------+---------------+---------------------------+
               |
               v
        Vault (data transfer layer between all components)
```

An LLM agent can: issue browser automation calls to AMI 1 (browse, screenshot, interact), store results and communicate with other agents via AMI 2 (vaults), and generate content using AMI 3 (infographics, documents, presentations). All within the customer's environment. All encrypted. All auditable.

## Marketplace Strategy

### AWS Marketplace (First)

Start here. We already have the CI pipeline baking AMIs. AWS Marketplace is the natural first listing.

**Pricing model options:**
- Free tier (open source, customer pays only for EC2 compute)
- Bring-your-own-license (customer deploys, we charge a small monthly fee)
- Usage-based (per API call, per screenshot, per vault operation)
- Annual subscription (for enterprise customers who want predictable costs)

Start with free tier to get traction and usage data. Add paid tiers once we understand the demand.

### Docker Hub (Parallel)

All three are also available as Docker images on Docker Hub. This is the multi-cloud play. GCP, Azure, on-premises, Kubernetes: anywhere Docker runs, these run.

### Other Marketplaces (Future)

| Marketplace | Priority | Notes |
|-------------|----------|-------|
| GCP Marketplace | Medium | Already have a GCP account. Container-native. |
| Azure Marketplace | Medium | Enterprise customers on Microsoft stack. |
| Docker Hub Verified Publisher | Medium | Credibility and discoverability. |
| Third-party agent platforms | Explore | Partner with agentic framework providers who need these primitives. |

## What We Learn

Shipping these AMIs is as much a learning exercise as a revenue play:

- How to package and list on AWS Marketplace
- How to handle marketplace billing and metering
- How to price per-execution vs per-instance
- How to market developer infrastructure
- How to manage updates and versioning across marketplace listings
- How to find and engage partners through the marketplace

Every lesson applies directly to SG/Send's commercial model.
