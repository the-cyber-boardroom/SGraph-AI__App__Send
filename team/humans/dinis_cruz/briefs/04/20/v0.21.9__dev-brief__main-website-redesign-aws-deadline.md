# Main Website Redesign: sgraph.ai Before the AWS Conference

**version** v0.21.9
**date** 20 Apr 2026
**from** Human (project lead)
**to** Developer (lead), Designer, Librarian, Ambassador
**type** Dev brief

---

## Urgency

The AWS Conference in London is on Wednesday. The main corporate website (sgraph.ai) needs to be updated before then to reflect the current capabilities, the new positioning, and the product in action. This is the marketing surface I will be pointing partners at.

## What the Website Must Show

### Hero Section

**"Version-controlled, client-encrypted vaults. For humans and agents."**

This is the decided positioning from today's hero line brief (`v0.21.9__brief__positioning-decision-hero-line.md`). Below the hero: a subhead candidate (to be tested) and a single "Try it" call-to-action that takes visitors to the send/receive flow or a demo vault.

### The Receive Problem (Fix Immediately)

Right now it is very hard to find the link to receive a file. If someone has a token, they should be able to go to the website, find the receive input, paste the token, and get the file. One click away. This is broken today and must be fixed before the conference, because the entire demo flow depends on sharing tokens with people who then need to redeem them.

### Key Sections

**1. What It Does (Use Cases in Action)**

Not feature descriptions. Real workflows with screenshots or short videos:

- Encrypted file transfer (send a file, get a token, paste the token)
- Vault collaboration (multiple users editing, push/pull, version history)
- Agent-to-agent communication (two agents sharing via vault)
- Content delivery packs (review reports, presentation bundles, infographic sets)
- The healthcare workflow (client-side data capture, doctor collaboration)

Each use case: one screenshot or video, two sentences, a "try it" link to a real vault.

**2. How It Works (Architecture for Technical Visitors)**

The AWS stack story: Lambda, S3, ECR, CloudFront. Client-side AES-256-GCM encryption. Memory-FS storage abstraction. Seven deployment targets. The Playwright service. The three primitives (browser automation, encrypted VFS, code execution sandbox).

This is the section for the AWS Conference audience. Show the architecture diagram. Link to the open source repo.

**3. Tools**

Link to tools.sgraph.ai with previews of what is available: infographic generator, LLM chat, video editor, page layout renderer. These should be one-click accessible from the main site, not buried behind navigation.

**4. Buy Credits / OpenRouter Keys**

The Stripe integration for purchasing credits and receiving matched OpenRouter keys. This needs to be visible and easy to find. It is a revenue surface and a demo talking point.

**5. AWS Marketplace**

If the AMI listings are submitted by Wednesday, feature them: "Deploy in your AWS account." Link to the Marketplace listings. Even if not yet approved, describe what is coming.

**6. Open Source**

Link to the GitHub repo. "251+ briefs. 22+ agent roles. Every decision documented." The transparency story is a differentiator.

## The Bootstrap Pattern (How to Build This)

Use the same pattern that worked for the Playwright project:

1. Create a vault containing all the research, materials, content, component references, and design guidance needed to bootstrap the website
2. Create a clean repo (standard dev/main/projects structure, deployed from S3)
3. First prompt to a Claude Code session: "Install sgit-ai, clone this vault, start reading"
4. The agent reads the vault, sets up the repo, copies materials, and builds the site

This is the vault-as-dev-pack pattern from the agent communication article. Use it here.

### What Goes in the Bootstrap Vault

- The hero line brief and positioning decision
- The use case descriptions and screenshots
- Existing component references from tools.sgraph.ai (to import, not rebuild)
- The architecture diagrams
- The current site structure (as reference for what to keep)
- Design guidance (brand colours, typography, layout principles)
- Content from the OWASP presentation pack (reusable material)
- The Playwright vault as a reference for how this pattern works

## Component Reuse

The website must import components from tools.sgraph.ai, not rebuild them. This establishes a unified component layer across the ecosystem:

- `sg-send` components for the upload/receive flow
- `sg-vault` components for vault browsing demos
- Page layout renderer for content display
- Video player for demo videos
- Any shared UI components (buttons, cards, navigation)

This means the transition between the website, the tools site, and the send/receive flow is seamless. The user does not feel like they are moving between different products.

## The Smooth Flow

A visitor's journey should be:

```
sgraph.ai (main site)
    |
    +--> "Try it" -> send.sgraph.ai (send/receive flow)
    |                    |
    |                    +--> Paste token -> receive file (ONE CLICK)
    |                    +--> Upload file -> get token
    |
    +--> "Tools" -> tools.sgraph.ai (infographic, LLM, video)
    |
    +--> "Browse a demo vault" -> send.sgraph.ai/browse (vault viewer)
    |
    +--> "Buy credits" -> Stripe checkout
    |
    +--> "Deploy it yourself" -> AWS Marketplace / GitHub repo
```

Every transition should feel like one product, not five websites. Shared navigation, shared components, shared design language.

## Infrastructure

Standard deployment: S3 + CloudFront, with the existing deploy-website GitHub Action. Dev/main branching. SSL via CloudFront. Same pattern as the existing sites.
