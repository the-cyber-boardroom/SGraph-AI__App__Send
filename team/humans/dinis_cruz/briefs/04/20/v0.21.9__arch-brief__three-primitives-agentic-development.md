## The Landscape Today

The agentic development ecosystem has solved some pieces well. It is now easy to create a chat interface. It is easy to use OpenRouter, Ollama, or the OpenAI API to make LLM calls. There is growing tooling around monitoring and intercepting LLM traffic (though visualisation is still weak). The "talk to an LLM" primitive is essentially solved.

But there are gaps. Three specific capabilities that agents need, that the current ecosystem does not provide cleanly, and that we have now built working solutions for.

## Primitive 1: Browser Automation with Full Traffic Observability

### What Exists Today

Agents can use Playwright or Puppeteer to control a browser. But there is no clean, API-driven, declarative way to do it. You have to write code that directly instruments the browser. The agent needs Playwright installed, needs to manage browser lifecycle, and the operations are imperative (function calls), not declarative (descriptions of intent).

### What We Built

A service where you send JSON instructions and get results back. Not "call this Playwright function." Instead: "here is a JSON object describing what I want to happen: open page, click here, enter text there, take screenshot." The service handles the execution. The caller handles the intent.

```json
{
  "steps": [
    { "action": "navigate", "url": "https://example.com" },
    { "action": "click", "selector": "#login" },
    { "action": "fill", "selector": "#email", "value": "test@example.com" },
    { "action": "screenshot" }
  ]
}
```

This is declarative browser automation. The caller does not care what runs underneath (Playwright, Puppeteer, Selenium, whatever). They care that the instructions execute and they get results.

### The MITM Proxy Addition

When running on compute (EC2, Fargate, local Docker with sidecar), the Playwright container is paired with a MITM proxy container. All browser traffic routes through the proxy. This gives you:

**100% traffic traceability.** Every HTTP request, every response, every redirect, every asset load. You can literally dump the entire session traffic. For an agent that browsed 50 pages and clicked 200 elements, you have a complete record of every network interaction.

**Access restrictions.** Via the proxy, you can restrict which domains the agent can access. DNS-level or URL-level filtering. The agent cannot browse outside the allowed list.

**Secret injection.** The proxy can inject authentication headers, cookies, or credentials into requests without the agent ever knowing the secrets. The agent says "go to this page." The proxy adds the auth header. The agent never sees the credential. This is a genuinely powerful security pattern: the agent operates with capability but not with knowledge of the secret.

**Traffic modification.** The proxy can modify requests and responses (add headers, strip tracking, inject scripts). Full MITM capability for testing, debugging, and security analysis.

### Why This Matters for Agents

Without this, giving an agent browser access is a trust-and-pray situation. You do not know what it accessed. You cannot restrict where it goes. You give it your credentials directly. With this, you have full control, full observability, and secret isolation. The agent gets capability without excessive privilege.

## Primitive 2: Encrypted Virtual File System (Vaults)

### What Exists Today

Agents use the local file system or Git repos for storage. Both have problems. The file system gives too much access (the agent can read anything the process can read) and has no version control. Git repos provide version control but require the agent to have your Git credentials and typically give full repo access, not scoped access to just the communication channel.

### What We Built

SG/Send vaults via SGit. A virtual file system that provides:

**Scoped access.** Each vault has its own key. The agent gets a vault key, not your GitHub credentials. Access is scoped to exactly one vault.

**Version control.** Every file operation is a commit. Full history. Branches. Push/pull. The agent understands Git workflows already, so it operates naturally.

**Encryption.** Zero-knowledge, client-side. The server stores only ciphertext. The agent's files are encrypted at rest and in transit. No server-side data processing.

**Multi-agent collaboration.** Multiple agents (and humans) can work on the same vault simultaneously. The dual branch model (clone branches, Push/Pull) handles concurrent access without conflicts (as long as different agents work on different files, which is the normal case).

**Secret sharing.** Vaults are a natural place to store configuration, secrets, and credentials for agent sessions. The Playwright container reads its proxy configuration from a vault. The agent reads its briefing from a vault. Output artefacts go into a vault. The vault is the universal communication channel.

**Two access modes.** Full clone (SGit CLI, downloads everything, offline access) or tree-only web access (fast, loads only the structure, files on demand). The agent picks whichever mode suits the workflow.

### Why This Matters for Agents

Without this, agent-to-agent communication requires shared file systems (insecure), shared repos (over-privileged), or custom protocols (fragile). With vaults, you have a versioned, encrypted, scoped file system purpose-built for agent communication. Files in, files out, full audit trail, no credential sharing.

## Primitive 3: Isolated Code Execution (Emerging)

### What Exists Today

When an agent needs to execute code (run a script, process data, transform a file), it typically runs in the same environment as everything else. No isolation. No resource limits. No audit trail of what code ran.

### What We Are Building

The foundation is already in place: the layered container architecture (from the 18 Apr arch brief) supports spinning up a container, loading code dynamically, executing it, and spinning down. The same pattern that powers the Playwright service can power a general-purpose code execution sandbox:

- Spin up a container (Layer 1: core FastAPI)
- Load the code to execute (from URL, S3, or vault)
- Execute in an isolated environment
- Capture all output (stdout, stderr, files produced)
- Write results to a vault
- Spin down the container

This is not fully implemented yet, but the architecture supports it. The container is stateless. The code is loaded at runtime. The execution is isolated. The results are captured.

### Why This Matters for Agents

Without this, code execution is "the agent runs whatever it wants in your environment." With this, code execution is scoped, isolated, auditable, and disposable. The agent gets a sandbox, not your machine.

## The Three Primitives Together

```
Agent Ecosystem (existing)
    |
    +--> LLM Access (solved: OpenRouter, Ollama, OpenAI API)
    +--> LLM Monitoring (partially solved: various tools)
    |
    +--> Browser Automation (OUR PRIMITIVE)
    |       +--> Declarative JSON instructions
    |       +--> MITM proxy for 100% traffic observability
    |       +--> Domain restrictions, secret injection
    |       +--> Screenshots, PDFs, network dumps
    |
    +--> Virtual File System (OUR PRIMITIVE)
    |       +--> SG/Send vaults via SGit
    |       +--> Encrypted, versioned, scoped
    |       +--> Multi-agent collaboration
    |       +--> Secret and config storage
    |       +--> Two access modes (clone or web)
    |
    +--> Code Execution Sandbox (OUR PRIMITIVE - emerging)
            +--> Isolated container per execution
            +--> Code loaded at runtime from vault/S3/URL
            +--> Output captured to vault
            +--> Stateless, disposable
```

These three primitives, combined with the existing LLM access layer, give you a complete agentic runtime: the agent can think (LLM), browse (browser automation), store and communicate (vaults), and execute code (sandbox). Each primitive is observable, restrictable, and auditable.

## Positioning

This should be a prominent part of the website and the investor materials. The messaging:

"We provide the missing primitives for agentic development: a browser the agent can use safely, a file system the agent can use securely, and a sandbox the agent can execute code in. Each one is observable, restrictable, and encrypted."

This is not a framework. It is not a chat UI. It is the infrastructure layer that sits underneath any agentic framework, providing the capabilities that agents need but that nobody has packaged cleanly.
