# SKILL: Email-FS — Operating Email-FS Manually

## What This Is

Email-FS is the file-based email system inside an sgit-encrypted vault. Agents and humans communicate by writing RFC 2822 `.eml` files in a defined folder structure. There is no broker, no API, no daemon — the vault is the message store, sgit is the transport, every operation is a filesystem operation you can do by hand.

This skill is everything you need to *use* Email-FS without any CLI tooling. (The Email-FS CLI does not yet exist; once the protocol is stress-tested manually, it will be implemented as `pip install email-fs`.)

For deeper background, see the companion documents — **Email-FS Specification v0.6** (the contract) and **Email-FS Architecture v0.6** (the diagrams). You do not need them to use Email-FS; this skill is sufficient.

**v0.5 → v0.6 changes:** Session model substantially refined. Sessions are now explicitly the *unit of agent work* (a coherent brief, possibly spanning days and many check-ins), distinct from per-check-in cycles. Three-state lifecycle: `active/` → `archived/` (graceful end) or `stale/` (declared abandoned). Sessions become immutable once `end.json` is written. PROCESS-vs-replied semantics clarified.

## Core Concept: Session vs Check-in

This is the most important distinction in the whole skill. Get it right and everything else follows.

| | **Session** | **Check-in** |
|---|---|---|
| What it is | One coherent unit of agent work, bounded by a brief | One iteration of the work loop |
| Lifespan | Hours to days; many rounds; survives runtime restarts | Minutes; one pull/process/push cycle |
| Bookended by | `start.json` ... `end.json` | nothing persistent — just a commit |
| Example | "Refine v0.2 design into Email-FS v0.6 spec set" — runs from when human hands over the v0.2 doc until the four-doc set is locked | "Pull, see Conductor's pushback, refine §3, push" — happens 5 times over 3 days as part of one session |
| Identity | Same `session-id` throughout | (no identity — events log a `check-in-start`/`check-in-end` if you want to mark them) |

A session contains many check-ins. The runtime can stop and start; the session continues. You start a *new* session only when the brief itself is new — not just because a new chat window opened.

## When You're Working in an Email-FS Vault

### A. Starting a session (do once at the beginning)

Run these checks before doing any work:

1. **Does `mail/` exist at the vault root?** If yes → Email-FS is active here, follow this skill. If no → either it's not an Email-FS vault, or you may need to initialise it (see §10).
2. **Are you continuing existing work or starting fresh?** Check `mail/sessions/active/` — if a session there matches the brief you've been given, you may be resuming it (see §3.6). Otherwise, you're starting a new session.
3. **For a new session: pick your agent name.** It is `{role}.{location}.{session-id}`. Generate a fresh session-id (see §2.3). Write `mail/sessions/active/{your-session-id}/start.json` (see §3.4).
4. **Write your first reasoning log.** Address it to yourself, summarising the brief in your own voice and what you understand the objectives to be. This becomes the first node in your reasoning trail. (See §7.)

### B. Each check-in (do many times during a session)

A check-in is one full work loop. Run this every time you sit down to do work in the vault:

```
sgit pull                           ← get latest vault state
ls mail/mailroom/{your-name}/       ← any new mail?
DELIVER  (§5.2)                     ← move waiting mail into your inbox
READ + PROCESS  (§5.3, §5.4)        ← work through your inbox
SEND  (§5.1) any replies / new messages
write reasoning logs  (§7) for any meaningful decision
sgit commit + push                  ← publish your changes
```

Optionally, log a `kind: check-in-end` event in your session's `events/` folder if the check-in produced a meaningful milestone. Most check-ins are routine — don't log them.

### C. Ending a session (do once at the end, when the work is done)

When the brief is satisfied (or abandoned, or handed off), you **end the session explicitly**:

1. Write a final reasoning log summarising what was accomplished and why (mirrors `end_summary`).
2. Write `mail/sessions/active/{your-session-id}/end.json` (see §3.7).
3. **Move the entire session folder** from `active/` to `archived/`.
4. One commit covering steps 2–3.

**Once `end.json` exists, the session is immutable.** No further events, no further updates. If you want to keep working on the topic, start a new session with `parent_session` pointing at the archived one.

## 1. Vault Layout (What You Will See)

```
{vault-root}/
└── mail/
    ├── sessions/                                    cast of agents past and present
    │   ├── active/                                  in-flight sessions (mutable)
    │   │   └── {session-id}/
    │   │       ├── start.json
    │   │       └── events/
    │   ├── archived/                                gracefully-ended sessions (immutable)
    │   │   └── {session-id}/
    │   │       ├── start.json
    │   │       ├── events/
    │   │       └── end.json
    │   └── stale/                                   abandoned sessions (immutable)
    │       └── {session-id}/
    │           ├── start.json
    │           ├── events/
    │           └── end.json    (with end_status: "stale", ended_by: ...)
    ├── mailroom/                                    transit zone
    │   └── {recipient-name}/
    │       └── {ULID}__{sender}__{slug}.eml         in-flight messages
    └── {agent-name}/                                one folder per agent
        ├── outbox/{recipient}/                      archive of messages you sent
        ├── outbox/self/                             your reasoning logs
        ├── inbox/                                   delivered, not yet processed
        ├── processed/                               actioned
        ├── purged/                                  tombstones for deleted messages
        └── state/{ULID}/                            your sidecar workspace per message
            └── core.json
```

**Writer rule — do not violate this:** only the agent named in the path writes anywhere under `mail/{agent-name}/`. The mailroom is the only exception — it has a producer-consumer pattern (anyone writes new files; only the named recipient deletes them).

**Immutability rule:** anything under `mail/sessions/archived/` or `mail/sessions/stale/` is read-only. Never modify those folders after they've reached their final state.

## 2. Naming

### 2.1 Agent name

Format: `{role}.{location}.{session-id}`, lowercase, dot-separated.

| Part | Examples |
|---|---|
| `role` | `conductor`, `architect`, `dev`, `librarian`, `appsec`, `dpo`, `sherpa`, `tools`, `cartographer`, `ambassador` |
| `location` | `claude-ai`, `claude-code`, `claude-code-web`, `claude-code-tools` |
| `session-id` | `s-01HXY8K3M2` (12 chars total, see §2.3) |

Examples:

```
conductor.claude-ai.s-01HXY8K9PR
architect.claude-ai.s-01HXY8K3M2
dev.claude-code-web.s-01HXY8KAQB
```

Humans use `human.{name}` with no session-id (e.g. `human.dinis`). Humans are addressable peers.

### 2.2 Why session-id is required

Without it, two parallel sessions of the same role/location collide. The session-id disambiguates and makes every session's mailbox separately addressable in vault history.

### 2.3 Generating your session-id

1. Generate a fresh ULID. Easiest: `python -c "import ulid; print(ulid.new())"` or any online ULID generator.
2. Take the **last 10 characters** of the ULID's random portion.
3. Prefix with `s-`. Total: 12 chars.

What matters is uniqueness within the vault; 10 chars of Crockford-base-32 (32^10 ≈ 10^15) gives you that trivially.

**If you can't generate a ULID** (no library handy): pick 10 random Crockford-base-32 chars (`0-9` and `A-Z` minus `I`, `L`, `O`, `U`).

### 2.4 Message ULID (different from session-id)

Each message gets its own full 26-char ULID. This ULID is used twice and they MUST match:

- In the `Message-ID:` header: `<{ULID}@vault.sgraph.ai>`
- In the filename: `{ULID}__{sender}__{slug}.eml`

Generate a fresh ULID for every new message. ULIDs sort lexicographically by creation time, so directory listings naturally show oldest-first.

### 2.5 Filename slug

Kebab-case, ≤ 6 words, derived from the subject. Examples:

```
01HXY8K3M2P4QRS5TVWZ7C9DAB__architect.claude-ai.s-01HXY8K3M2__email-fs-v0.5-spec.eml
01HXYBAC...__conductor.claude-ai.s-01HXY8K9PR__build-the-dev-page.eml
```

## 3. Sessions

A session is your **unit of work** — bounded by a coherent brief, possibly spanning many check-ins, hours, or days. See "Core Concept: Session vs Check-in" near the top of this skill.

### 3.1 Are you starting a session or resuming one?

Before writing anything: check if you're continuing existing work.

```bash
$ ls mail/sessions/active/
```

If you see a session folder there whose `start.json` matches the brief you've been given (same agent name, same brief topic, work clearly unfinished), **you are resuming that session**. Skip ahead to §3.6.

Otherwise — and this is the common case for fresh work — proceed to §3.2.

### 3.2 Start a new session

```bash
$ mkdir -p mail/sessions/active/{your-session-id}/events

$ cat > mail/sessions/active/{your-session-id}/start.json << 'EOF'
{
  "schema_version": "0.6",
  "session_id": "{your-session-id}",
  "team": "explorer",
  "role": "architect",
  "location": "claude-ai",
  "agent_name": "architect.claude-ai.{your-session-id}",
  "runtime_model": "claude-opus-4-7",
  "human_driver": "human.dinis",
  "parent_session": "{the session that brought you in, or null}",
  "brief": "One-line statement of what this session is here to do",
  "objectives": [
    "First concrete objective",
    "Second",
    "..."
  ],
  "started_at": "{now-utc}"
}
EOF

$ sgit commit -m "session-start: {your-agent-name}"
$ sgit push
```

**Required fields:** `schema_version`, `session_id`, `role`, `location`, `agent_name`, `started_at`, `brief`.
**Recommended:** `team`, `runtime_model`, `human_driver`, `parent_session`, `objectives`.

`parent_session` is the session that brought you in — useful for hierarchical orchestration (Conductor invokes Architect; Architect's `start.json` names Conductor's session as parent). Or, if you're a continuation of an archived session on the same topic, point at the archived one.

### 3.3 Write your first reasoning log

Right after `start.json`, write a self-addressed reasoning log capturing the brief in your own voice:

```
From: {your-name} <{your-name}@vault.sgraph.ai>
To: {your-name} <{your-name}@vault.sgraph.ai>
Subject: Session brief — {short summary}
X-EmailFS-Kind: reasoning-log

I'm starting this session to {brief in your own words}.

Objectives as I understand them:
  1. {objective}
  2. {objective}

Constraints / context I'm aware of:
  - {anything relevant}

First steps I'll take:
  - {opening moves}
```

This is the first node in your reasoning trail. When you (or a future agent) resume this session, the first thing they read is this message — it's how the session "remembers" what it's here to do, in agent-voice. Complements the structured `start.json`.

### 3.4 Log events during the session (optional, for milestones)

For meaningful milestones during your work, write a small JSON to `mail/sessions/active/{your-session-id}/events/`:

```
mail/sessions/active/s-01HXY8K3M2/events/01HXY8K3MA__delivered-v0.3.json
```

```json
{
  "schema_version": "0.6",
  "event_id": "01HXY8K3MA",
  "session_id": "s-01HXY8K3M2",
  "at": "2026-04-28T17:42:11Z",
  "kind": "milestone",
  "summary": "Delivered Email-FS v0.3 spec",
  "refs": {
    "messages": [],
    "issues": [],
    "vault_paths": ["docs/email-fs-v0.3-spec.md"]
  }
}
```

`kind` is open: `milestone`, `decision`, `pushback-received`, `handoff`, `error-recovered`, `paused`, `resumed`, `check-in-start`, `check-in-end`. Use what fits.

Events vs reasoning logs:
- **Events** are coarser, structured, queryable. "Delivered v0.3 spec." "Resumed after 2 days." "Handed off to Librarian."
- **Reasoning logs** are finer, prose, decision-focused. "Chose PKCE over implicit because..."

Use both. The simulation in v0.6 had ~3-5 events per session and ~10-20 reasoning logs.

### 3.5 End the session (graceful close)

When the work is done — brief satisfied, objectives met, ready to wrap up:

**Step 1.** Write a final reasoning log summarising what was done and why.

**Step 2.** Write `end.json` and move the folder, in one commit:

```bash
$ cat > mail/sessions/active/{your-session-id}/end.json << 'EOF'
{
  "schema_version": "0.6",
  "session_id": "{your-session-id}",
  "ended_at": "{now-utc}",
  "end_status": "delivered",
  "end_summary": "What was accomplished. One paragraph.",
  "messages_sent": 7,
  "messages_received": 3,
  "events_logged": 5
}
EOF

$ sgit mv mail/sessions/active/{your-session-id}/ mail/sessions/archived/{your-session-id}/

$ sgit commit -m "session-end: {your-agent-name} — delivered"
$ sgit push
```

**`end_status` values:**

| Value | Meaning |
|---|---|
| `delivered` | Work complete; brief satisfied |
| `incomplete` | Work stopped before brief was fully satisfied; explain in `end_summary` |
| `abandoned` | Decided not to continue; explain why |
| `handed-off` | Work passed to another session; include `handed_off_to: "{successor-session-id}"` |
| `stale` | (Special — see §3.7; not used by the working agent on themselves) |

**Once `end.json` is written, the session is immutable.** Do not write further events, do not modify `start.json` or `end.json`, do not move the folder back to `active/`. If you want to keep working on the topic, start a new session with `parent_session` pointing at the now-archived one.

### 3.6 Resuming an active session

When a new runtime instance picks up an existing brief (next day; after a context reset; etc.), it resumes the session rather than starting a new one:

```bash
# 1. Identify the session you're resuming
$ ls mail/sessions/active/
{your-session-id}/

# 2. Read the start.json to recover the brief
$ cat mail/sessions/active/{your-session-id}/start.json

# 3. Walk events/ chronologically — the milestone trail
$ ls mail/sessions/active/{your-session-id}/events/

# 4. Read recent reasoning logs in your outbox/self/ — finer-grained context
$ ls -t mail/{your-agent-name}/outbox/self/ | head

# 5. Optionally, log a "resumed" event
$ # ...write {event-ULID}__resumed-after-N-days.json...

# 6. Optionally, write a reasoning log capturing what you re-read and what you plan to do
```

The agent name (and therefore the session-id) **stays the same across resumptions**. You're not a new agent — you're the same logical agent picking up where the previous runtime left off.

### 3.7 Declaring someone else's session stale

This is the case where you (or a human, or a future GC tool) have decided a session in `active/` is abandoned — its agent has stopped working but never wrote `end.json`. The threshold is use-case-specific (could be 1 hour, could be 30 days). Whoever decides is responsible for closing it out.

**Step 1.** Identify the candidate session in `mail/sessions/active/{session-id}/`.

**Step 2.** Write an `end.json` *into the active folder* with `end_status: "stale"`:

```json
{
  "schema_version": "0.6",
  "session_id": "{session-id}",
  "ended_at": "{now-utc}",
  "end_status": "stale",
  "end_summary": "No commits in 14 days; brief presumed abandoned. Last event: {summary of last event}.",
  "ended_by": "{your-agent-name, or human.{name}}",
  "stale_threshold_used": "14 days inactivity"
}
```

The `ended_by` field is **mandatory** for stale ends (so it's clear who made the call). The `stale_threshold_used` field documents the policy you applied.

**Step 3.** Move the folder to `stale/`:

```bash
$ sgit mv mail/sessions/active/{session-id}/ mail/sessions/stale/{session-id}/
$ sgit commit -m "session-stale: {session-id} — {short reason}"
$ sgit push
```

Once moved, the session is immutable. If the original agent later returns and finds their session in stale/, they cannot revive it. They start a new session with `parent_session` pointing at the stale one.

### 3.8 Initialising your mailbox

If your `mail/{agent-name}/` folder doesn't exist yet, create it on first SEND or DELIVER. The subfolders (`outbox/`, `inbox/`, `processed/`, `purged/`, `state/`) are created on first use — empty parents are not needed up front.

## 4. Message Format

### 4.1 Mandatory headers

```
From: architect.claude-ai.s-01HXY8K3M2 <architect.claude-ai.s-01HXY8K3M2@vault.sgraph.ai>
To: conductor.claude-ai.s-01HXY8K9PR <conductor.claude-ai.s-01HXY8K9PR@vault.sgraph.ai>
Date: Tue, 28 Apr 2026 17:42:00 +0000
Subject: v0.5 spec ready — 4 open questions inside
Message-ID: <01HXY8K3MZP4QRS5TVWZ7C9DAB@vault.sgraph.ai>
Content-Type: text/plain; charset=utf-8
```

### 4.2 Conditional headers

For replies:

```
In-Reply-To: <01HXY7ABCDEF...@vault.sgraph.ai>
References: <01HXY7ABCDEF...@vault.sgraph.ai>
```

For replies-to-replies, append the new parent's ID to `References` (oldest first).

### 4.3 Email-FS extension headers

```
X-EmailFS-Schema-Version: 0.5
X-EmailFS-Vault-ID: mist-drip-9145
X-EmailFS-Issues-Ref: issues/workstream-dev-section.issues
X-EmailFS-Asset-Ref: docs/email-fs-specification-v0.5.md
X-EmailFS-Priority: normal
X-EmailFS-Kind: task
X-EmailFS-Sender-Session: s-01HXY8K3M2
```

| Header | Purpose |
|---|---|
| `X-EmailFS-Schema-Version` | Always `0.5` for messages following this spec |
| `X-EmailFS-Vault-ID` | Helpful for cross-vault traceability |
| `X-EmailFS-Issues-Ref` | Path(s) to related Issues-FS files |
| `X-EmailFS-Asset-Ref` | Path(s) to vault assets the message references |
| `X-EmailFS-Priority` | `low` \| `normal` \| `high` \| `urgent` (default `normal`) |
| `X-EmailFS-Kind` | `task` \| `reply` \| `notification` \| `reasoning-log` \| `forward` |
| `X-EmailFS-Sender-Session` | Your session-id (redundant with From, useful for filtering) |

### 4.4 Forbidden in the eml

**Never put a `Status:` header in the eml.** Status is sidecar-only (§6). The eml is immutable once written.

### 4.5 Subject lines matter — they are the visible audit log

Because directory listings are how agents and humans scan activity, **subject quality is part of the contract**. Conventions:

- **Tasks:** imperative — "Build the dev page", "Review the spec"
- **Completions:** past-tense — "PR opened", "Spec delivered"
- **Replies:** prefix `Re:` — "Re: Build the dev page"
- **Questions:** end with `?` — "Should we use ULID or NNN?"

A vault scan reading `build-dev-page → page-shells-done → review-page-shell → spec-feedback → spec-v0.5-delivered` should tell the story without anyone opening a single message.

### 4.6 Vault assets — reference, do not attach

If you're referencing a file that already lives in the vault, use `X-EmailFS-Asset-Ref` and mention the path in the body. **Do not MIME-attach it.** The vault is the canonical store; attaching duplicates content.

MIME attachments are reserved for content that does NOT already live in the vault (external uploads, content arriving from real SMTP).

### 4.7 Worked example

```
From: architect.claude-ai.s-01HXY8K3M2 <architect.claude-ai.s-01HXY8K3M2@vault.sgraph.ai>
To: conductor.claude-ai.s-01HXY8K9PR <conductor.claude-ai.s-01HXY8K9PR@vault.sgraph.ai>
Date: Tue, 28 Apr 2026 17:42:00 +0000
Subject: Email-FS v0.5 spec ready — 4 open questions inside
Message-ID: <01HXY8K3MZP4QRS5TVWZ7C9DAB@vault.sgraph.ai>
In-Reply-To: <01HXY7ABCDEFG@vault.sgraph.ai>
References: <01HXY7ABCDEFG@vault.sgraph.ai>
Content-Type: text/plain; charset=utf-8
X-EmailFS-Schema-Version: 0.5
X-EmailFS-Vault-ID: mist-drip-9145
```

## 5. The Five Primitives

Every Email-FS operation is one of these five. Anything else is a workflow built from them.

### 5.1 SEND

You're sending a message to another agent.

```
PRECONDITION: sgit pull is current.

1. Generate a fresh 26-char ULID.
2. Compose the eml (§4). Make sure Message-ID's ULID == filename's ULID.
3. Write to the recipient's pigeonhole in the mailroom:
     mail/mailroom/{recipient}/{ULID}__{your-name}__{slug}.eml
4. Write a copy to your own outbox (your permanent send archive):
     mail/{your-name}/outbox/{recipient}/{ULID}__{your-name}__{slug}.eml
5. (Optional) Log a session event in mail/sessions/{your-session-id}/events/
6. sgit commit -m "send: {ULID-short} → {recipient} — {slug}"
7. sgit push
```

**Result:** the message is now in transit (in the mailroom for the recipient to pick up) and archived in your outbox.

### 5.2 DELIVER

You're picking up messages waiting for you.

```
PRECONDITION: sgit pull is current.

1. ls mail/mailroom/{your-name}/
   (returns 0 to N files; 0 means nothing to deliver)

2. For each .eml file found:
     a. mv mail/mailroom/{your-name}/{ULID}__...eml \
           mail/{your-name}/inbox/{ULID}__...eml

     b. Create your sidecar folder mail/{your-name}/state/{ULID}/
        Write core.json with:
          {
            "schema_version": "0.5",
            "message_id": "{ULID}",
            "status": "unread",
            "delivered_at": "<now-utc>",
            "read_at": null,
            "processed_at": null,
            "purged_at": null,
            "thread_root": "{from In-Reply-To/References, or this Message-ID}"
          }

3. sgit commit -m "deliver: {N} new message(s) to {your-name}"
4. sgit push
```

**Idempotent.** Running DELIVER on an empty mailroom is a no-op.

**Sender's view:** once you've delivered, the sender's pull will show their mailroom for you is empty — that is their delivery confirmation. No "read receipt" header is needed; file absence is the receipt.

**Determining `thread_root`:** if the eml has no `In-Reply-To`, this is a new thread — use the message's own Message-ID. If it has `References`, the thread root is the first (oldest) ID in that header. If only `In-Reply-To` is present, use its value as a best-effort root.

### 5.3 READ

You're reading a message in your inbox.

```
1. cat mail/{your-name}/inbox/{ULID}__...eml
2. Update mail/{your-name}/state/{ULID}/core.json:
     status: "read"
     read_at: "<now-utc>"
3. (Commit may be batched with PROCESS or session end — see §8.)
```

READ is allowed to be silent (no commit) for ephemeral session reads. Convention: commit at session end. If your session crashes mid-read, the read flag is lost; that's OK — the eml itself is intact and the next agent can re-read.

### 5.4 PROCESS

You've **completed the work the message asked for**. Note this carefully:

> **PROCESS = work done.** Not "I sent a reply." A reply asking for clarification is a SEND — the original message stays in your `inbox/` until the work is genuinely complete. PROCESS only when the task the sender asked for is actually finished.

This makes `ls mail/{any-agent}/inbox/` an accurate "what's still open?" view across the whole team.

```
1. mv mail/{your-name}/inbox/{ULID}__...eml \
      mail/{your-name}/processed/{ULID}__...eml

2. Update mail/{your-name}/state/{ULID}/core.json:
     status: "processed"
     processed_at: "<now-utc>"
     processing_notes: "{free-text summary of what you did — optional}"

3. (Optionally) write extension files in mail/{your-name}/state/{ULID}/
   for any structured workspace the recipient has built around this message.
   Examples by role:
     dev.claude-code-web      → pr.json   {pr-url, status, last-checked-at}
     appsec.claude-code       → risk.json {risk-rating, mitigations}
     librarian.claude-code    → cataloged.json {master-index-refs}
     dpo.claude-ai            → gdpr.json {lawful-basis, retention-class}
   See §6 for sidecar extension patterns.

4. sgit commit -m "process: {ULID-short}"
5. sgit push
```

**Examples of when to PROCESS vs leave in inbox:**

| Situation | PROCESS now? |
|---|---|
| Task: "build dashboard shell" — you've opened the PR | ✅ Yes — work done |
| Task: "build dashboard shell" — you replied with a clarifying question | ❌ No — work still open, leave in inbox |
| Task: "review my doc" — you sent the review back | ✅ Yes — work done |
| Task: "review my doc" — you read it but haven't reviewed yet | ❌ No — set status=read, leave in inbox |
| Question: "what's the route prefix?" — you answered | ✅ Yes — question resolved by your answer |
| Brief: "implement X" — X has dependency you've now blocked on | ❌ No — leave in inbox; consider sending an `X-EmailFS-Kind: blocked` notification |

**Sender verification.** After processing, the sender (on their next pull) can confirm three facts in vault history:

- Mailroom for them empty for this ULID → delivered
- Eml in your `processed/` → actioned
- Your `state/{ULID}/core.json` shows `processed_at` → with timestamp

This is the entire round-trip audit. No protocol messages needed.

### 5.5 PURGE

You're deleting a message from your mailbox. **Precondition: it must be in `processed/` first** (you cannot purge directly from inbox).

```
1. Read the eml's Message-ID, From, Date, Subject (you'll need them for the tombstone).

2. Write the tombstone:
     mail/{your-name}/purged/{ULID}.tombstone.json
     {
       "schema_version": "0.5",
       "message_id": "{ULID}",
       "from": "{sender-name}",
       "date": "{original Date header}",
       "subject": "{original Subject}",
       "purged_at": "<now-utc>",
       "reason": "{free text — optional}"
     }

3. rm mail/{your-name}/processed/{ULID}__...eml
4. rm -rf mail/{your-name}/state/{ULID}/
5. sgit commit -m "purge: {ULID-short} — {reason or 'retention'}"
6. sgit push
```

**Important:** PURGE removes the message from YOUR mailbox. The sender's outbox copy is untouched (they own that record). The tombstone preserves thread integrity — anyone walking `In-Reply-To` chains lands on the stub, not a 404.

For coordinated multi-mailbox erasure (GDPR right-to-erasure shape), each mailbox holding the message executes its own PURGE. That coordination is a workflow on top of the PURGE primitive, not a primitive itself.

## 6. The Sidecar — Your Per-Message Workspace

The sidecar is **a folder, not a single file**. It is your workspace for everything you do with this message.

```
mail/{your-name}/state/{ULID}/
├── core.json              REQUIRED — status + lifecycle timestamps
├── labels.json            optional — your tags/categories
├── attachments/           optional — extracted MIME parts
├── extracted/             optional — derived artifacts (summaries, OCR, etc.)
├── workflow/              optional — your processing-pipeline state
└── notes/                 optional — your reasoning trace about this message
```

### 6.1 `core.json` — required minimum

The minimum any conformant recipient must write. Schema in §5.2.

### 6.2 Extension files — recipient-defined

Beyond `core.json`, you can write any additional files in your sidecar folder. Conventions:

- Use lowercase, descriptive names (`labels.json`, `pr.json`, `risk.json`)
- File formats are your choice (JSON for structured data, MD for prose, raw bytes for assets)
- Other agents do not need to understand your extensions

This is where role-specific behaviour lives. Two recipients can handle the same incoming message differently — one extracts attachments, another raises an issue, another forwards. All workflow state belongs here.

## 7. Self-Addressed Reasoning Logs

You email yourself on every meaningful decision. These accumulate into your inspectable reasoning trail.

### 7.1 What counts as meaningful

- Choosing one option over another with non-trivial consequences
- Departing from a brief or instruction
- Resolving an ambiguity
- Catching and recovering from a mistake
- Completing a stage of work

The bar is "a future reviewer would want to see this." Not every thought; not every tool call.

### 7.2 How to write one

```
1. Generate a fresh ULID.
2. Compose an eml with From: yourself, To: yourself.
3. Set X-EmailFS-Kind: reasoning-log.
4. Write to mail/{your-name}/outbox/self/{ULID}__{your-name}__{slug}.eml
5. (No DELIVER step — addressed to self, stored once.)
6. Commit (may be batched with other actions — see §8).
```

Example:

```
From: architect.claude-ai.s-01HXY8K3M2 <architect.claude-ai.s-01HXY8K3M2@vault.sgraph.ai>
To: architect.claude-ai.s-01HXY8K3M2 <architect.claude-ai.s-01HXY8K3M2@vault.sgraph.ai>
Date: Tue, 28 Apr 2026 17:42:00 +0000
Subject: Decision — sessions index becomes a folder, not a file
Message-ID: <01HXY8KREASON0001@vault.sgraph.ai>
X-EmailFS-Schema-Version: 0.5
X-EmailFS-Kind: reasoning-log

Decision: sessions index = mail/sessions/{session-id}/ folder, not a .session file.

Rationale: A growing log file forces re-read on every event. Folder-of-small-
immutable-files lets agents read only what they need. Same reasoning as the
sidecar-as-folder pattern elsewhere in the design.
```

### 7.3 Reasoning logs vs session events

| | Reasoning log | Session event |
|---|---|---|
| Format | Full eml, addressed self → self | Small JSON in `sessions/{id}/events/` |
| Granularity | Fine — any meaningful decision | Coarse — milestones, handoffs |
| Threadable | Yes (RFC 2822 In-Reply-To) | Within events folder, by event_id |
| Audience | Future agents acquiring full context | Anyone scanning who-did-what at milestone level |

Use both.

## 8. Commit Discipline

The atomicity unit is **one logical agent action, one commit**. Batch where natural.

| Logical action | What you wrote | Commit |
|---|---|---|
| Send 1 message | mailroom + own outbox | one commit |
| Send 3 messages in a batch (fan-out) | 3 × (mailroom + outbox) | one commit |
| Deliver 5 incoming messages | 5 inbox moves + 5 sidecar folders | one commit |
| Read a message | core.json updated | may batch with PROCESS or session end |
| Process a message | move to processed/ + sidecar update + extension files | one commit |
| Purge | tombstone + delete eml + delete sidecar | one commit |
| Reasoning log | one eml in outbox/self/ | one commit (or batched into next action) |

The principle: read `sgit log` after-the-fact and it should read as the story of what happened.

Commit message format (suggested, not enforced):

```
{verb}: {short-subject}
```

Verbs: `send`, `deliver`, `read`, `process`, `purge`, `reason`, `session-start`, `session-end`.

Examples:

```
send: 01HXY8 → dev.claude-code-web — build dev page
deliver: 3 new messages to architect.claude-ai.s-01HXY8K3M2
process: 01HXY7 — review complete, PR feedback in extension file
reason: 01HXYR — decision on session-id format
session-start: architect.claude-ai.s-01HXY8K3M2
session-end: architect.claude-ai.s-01HXY8K3M2 — delivered, 7 sent, 3 received
```

## 9. Verification — Did It Work?

After actions, verify by inspecting files. Email-FS gives you three independent facts per message:

| Question | Where to look |
|---|---|
| Did my message get sent? | `ls mail/{your-name}/outbox/{recipient}/` should list your ULID |
| Did the recipient deliver it? | `ls mail/mailroom/{recipient}/` — your ULID should be **absent** |
| Did the recipient action it? | `ls mail/{recipient}/processed/` for your ULID, then `cat mail/{recipient}/state/{ULID}/core.json` for `processed_at` |

If you suspect something's off, `sgit log` shows every commit. Filter to the mail folder:

```
sgit log -- mail/
```

Each commit message tells you what happened, who did it, when.

## 10. Initialising Email-FS in a Vault

If `mail/` doesn't exist at the vault root, create it on first use. There is no "init" command; the structure self-creates as agents start writing into it.

Minimal first-time bootstrap (for the very first agent in a vault):

```bash
mkdir -p mail/sessions
mkdir -p mail/mailroom
# Your own mailbox folders are created when you first SEND or DELIVER.
# The mailroom subfolder for a recipient is created when someone first SENDs to them.
```

Then write your `mail/sessions/{your-session-id}/start.json` per §3.1.

If you're the first agent ever, also commit a brief README at `mail/README.md` explaining this is an Email-FS namespace and pointing at the spec/architecture/skill docs. Useful for humans browsing the vault who don't yet know about Email-FS.

## 11. Gotchas

A short list of things that will trip you up. Read these.

1. **Forgetting to pull before SEND.** If you SEND on a stale clone, your push may need a merge. Always `sgit pull` first.

2. **Forgetting to pull before DELIVER.** Same. The mailroom contents you see locally may be stale.

3. **ULID mismatch between filename and Message-ID header.** They MUST be the same. Conformance check C9 fails if not.

4. **Editing a delivered eml.** Don't. The eml is immutable (axiom A1). Change your sidecar instead, or send a new message.

5. **Writing into another agent's mailbox folder.** Don't. The only place you write to "someone else's" space is the mailroom — and even there you only create new files, never modify or delete existing ones.

6. **Mixing up `outbox/{recipient}/` with `outbox/self/`.** The first is messages you sent to others. The second is your reasoning logs to yourself.

7. **Not writing a sidecar `core.json` on DELIVER.** Conformance check C12 fails. The eml without a sidecar is "delivered but un-tracked", which breaks the lifecycle.

8. **Putting `Status:` in the eml header.** Forbidden. Conformance check C13 fails. Status lives in the sidecar.

9. **Purging a message from `inbox/` directly.** Not allowed — must PROCESS first, then PURGE from `processed/`. The constraint is intentional: a message that wasn't actioned shouldn't be deletable without leaving a processed-state record.

10. **PROCESSing a message because you sent a reply.** PROCESS = work done, not "I responded". A reply asking for clarification is a SEND; the original stays in `inbox/` until the work is genuinely complete. See §5.4.

11. **Confusing a session with a check-in.** A session is the unit of *work* (could span days). A check-in is one iteration of pull/process/push. You start one session per coherent brief, but you do many check-ins within it. Don't open a new session every time you sit down to work.

12. **Modifying an archived or stale session.** Once `end.json` is written, the session is immutable. No new events. No edits to `start.json` or `end.json`. If you want to keep working on the topic, start a new session with `parent_session` pointing at the old one.

13. **Putting your session in the wrong state folder.**
    - Active session → `mail/sessions/active/`
    - Gracefully ended → `mail/sessions/archived/` (you moved it there yourself)
    - Declared abandoned by someone else → `mail/sessions/stale/` (they moved it there with `end_status: "stale"`)
    Don't put your own session in stale/. That's a state someone else assigns to you.

14. **Forgetting to move the session folder when ending.** Writing `end.json` in `active/{session-id}/` is half the action. The folder must also move to `archived/{session-id}/` (or `stale/` if it's a stale declaration). Single commit covers both.

15. **Generating a session-id that collides.** Vanishingly unlikely with 32^10 namespace, but if you somehow do (e.g. you tested with a hardcoded one), pick another.

## 12. Quick Reference Card

**Lifecycle:**

```
START SESSION (once):
  generate session-id
  mkdir mail/sessions/active/{session-id}/events
  write start.json
  write first reasoning log to outbox/self/
  sgit commit + push

CHECK-IN (many times during a session):
  sgit pull
  ls mail/mailroom/{my-name}/    # any new mail?
  DELIVER  any waiting messages
  READ + PROCESS  inbox messages (PROCESS only when work is done)
  SEND  any new messages or replies
  write reasoning logs for meaningful decisions
  sgit commit + push

END SESSION (once, when work is done):
  write final reasoning log
  write end.json
  sgit mv active/{session-id}/ archived/{session-id}/
  sgit commit + push
```

**The five primitives:**

| Op | Action | Sidecar effect |
|---|---|---|
| SEND | write to `mailroom/{recipient}/` + `outbox/{recipient}/` | none |
| DELIVER | `mv` from mailroom to inbox | create `state/{ULID}/core.json` |
| READ | `cat` the eml | update `core.json`: status=read, read_at |
| PROCESS | `mv` from inbox to processed (only when work done) | update `core.json`: status=processed, processed_at |
| PURGE | tombstone + `rm` eml + `rm -rf` state | sidecar deleted |

**Naming:** `{role}.{location}.{session-id}`
**Filename:** `{ULID-26-chars}__{full-agent-name}__{kebab-slug}.eml`
**ULID in filename MUST equal ULID in Message-ID header.**
**Session immutability:** `active/` is mutable; `archived/` and `stale/` are read-only.

## 13. Long-Running Sessions — Tips

This is the case the v0.5 SKILL didn't address well. A session can run for days, with the agent's runtime stopping and restarting. Some practical guidance:

**Leave breadcrumbs for the next runtime.** When you wrap up a check-in, write a `kind: check-in-end` event summarising what state the work is in. The next runtime that picks up the session reads this first.

```json
{
  "schema_version": "0.6",
  "event_id": "...",
  "session_id": "...",
  "at": "...",
  "kind": "check-in-end",
  "summary": "Drafted §3 of spec. Pending: human ruling on Q2 (cross-vault forwarding). Next steps: incorporate ruling, regenerate sim.",
  "refs": {
    "messages": ["{ULIDs of messages in flight}"],
    "vault_paths": ["{paths to drafts being worked on}"]
  }
}
```

**Use objectives in `start.json` as a checklist.** When you wrote `start.json` you listed objectives. Each check-in, optionally write a `kind: objective-progress` event noting which are done and which are open.

**Resumption read order.** When a new runtime takes over the session:
1. `start.json` — what was the brief?
2. `events/` walked oldest-first — what milestones happened?
3. `outbox/self/` recent reasoning logs — what was I thinking?
4. `inbox/` and recent `processed/` — what's the conversation state?
5. Then, only then, start writing.

**The session as the agent's external memory.** This is the architectural punchline: in an Email-FS-compliant agentic system, an agent's "memory" across runtime restarts is the session folder + their outbox/self/. There is no need for a separate memory/persistence system. Read the trail, pick up where you left off.
