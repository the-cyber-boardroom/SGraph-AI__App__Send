## 1. Mental Model

> **Email-FS is a contract for laying out RFC 2822 messages on a filesystem so that any reader — human, agent, or SMTP gateway — can determine sender, recipient, delivery state, conversation thread, and the cast of participating sessions, without consulting any external system.**

### 1.1 What Email-FS Is For

Email-FS does for agentic communications what sgit does for distributed encrypted version control. It is the **provenance and explainability substrate** for multi-agent systems:

| Capability | sgit | Email-FS |
|---|---|---|
| Primary concern | State of files over time | State of communication and decisions over time |
| What gets recorded | Every change to content | Every message, every delivery, every action taken on a message |
| What gets reconstructed | "What did the file look like at point X?" | "Who said what, who heard it, who did what about it, and why?" |
| Key property | Distributed, encrypted, content-addressed | Distributed (rides on sgit), single-writer-per-directory, message-immutable |
| Native artifact | Commits | RFC 2822 messages |

In an agentic system where multiple Claude (or other) instances collaborate, Email-FS gives you:

- **Provenance** — every instruction, every reply, every decision is a file. Who sent, who received, when delivered, when actioned. Cryptographically attestable (once PKI signing is wired).
- **Explainability** — self-addressed reasoning logs (§6) capture every meaningful decision an agent made, in its own voice, threaded into the larger conversation.
- **Replayability** — vault history is the complete record. Future agents joining a project read the mail and the reasoning logs to acquire context.
- **Cross-system interop** — because messages are real RFC 2822, any email client opens them, any SMTP gateway forwards them, any indexer that grok'd "email" for the last forty years grok's this.

The framing is intentional: **Email-FS is the messaging-comms-tracing layer of the Cyber Boardroom stack**, sitting on top of sgit (transport) and alongside Issues-FS (workflow tracking).

### 1.2 Three Axioms

The design follows from three axioms. Anything that violates one is not Email-FS.

| # | Axiom | Consequence |
|---|---|---|
| **A1** | The `.eml` is the message of record. **Immutable** once written. | All mutable per-recipient state lives in sidecar files. Signing remains valid. The eml itself is never modified after creation. |
| **A2** | Every directory has a clearly defined writer-set, and writers' operations are **disjoint by filename**. | Producer-consumer patterns are allowed (mailroom). True multi-writer races on the same file are forbidden. |
| **A3** | The vault is the message store. sgit is the transport. **There is no broker.** | Delivery is a recipient-driven move, not a sender-driven push. Senders publish to a shared transit zone; recipients pull. |

### 1.3 Vault Scope = Team Scope

A single Email-FS vault belongs to exactly one team (`explorer`, `villager`, `town-planner`, …). Team is therefore not encoded in agent IDs — it is implied by the vault. Cross-team communication (a v0.7+ concern, not specified here) requires either a separate vault per team-pair or a deliberately multi-team vault that re-introduces team to the ID.

## 2. Namespace and Layout

### 2.1 The Folder

The Email-FS namespace is **`mail/`** at the vault root. Non-dot, because sgit excludes `.`-prefixed directories.

### 2.2 Fractal Property

Every vault — including sub-vaults — has its own independent `mail/` namespace. They do not interact unless explicitly bridged by a forward.

```
{vault-root}/
├── mail/                              ← Email-FS for THIS vault (one team)
└── sub-rooms/
    └── due-diligence/
        └── mail/                      ← Email-FS for the sub-room (independent)
```

### 2.3 Top-Level Layout Inside `mail/`

```
mail/
├── sessions/                          ← three-state session lifecycle
│   ├── active/                        ← in-flight sessions (mutable)
│   ├── archived/                      ← gracefully-ended sessions (immutable)
│   └── stale/                         ← abandoned sessions (immutable)
├── mailroom/                          ← transit zone; per-recipient pigeonholes
└── {agent-name}/                      ← per-agent mailbox; only that agent writes here
```

### 2.4 Per-Agent Mailbox

```
mail/{agent-name}/
├── outbox/                            ← permanent record of messages this agent sent
│   ├── {recipient-1}/
│   ├── {recipient-2}/
│   └── self/                          ← reasoning logs (§6)
├── inbox/                             ← messages received and not yet processed
├── processed/                         ← messages this agent has actioned
├── purged/                            ← tombstones for messages this agent deleted
└── state/                             ← sidecar state per inbox/processed message
    └── {ULID}/                        ← one folder per message; recipient's structured workspace
```

**Writer rule:** *only the agent named in the path writes to that path*. Mailbox folders are strictly single-writer.

### 2.5 The Mailroom

```
mail/mailroom/
├── {recipient-1}/
│   ├── 01HXY8K...__from-conductor__build-dev-page.eml   ← waiting to be picked up
│   └── 01HXY9A...__from-architect__review-dev-page.eml
└── {recipient-2}/
    └── (empty — nothing in transit)
```

**Writer set** for `mail/mailroom/{recipient}/`:
- *Senders* — write new `.eml` files (every filename ULID is globally unique, so no name collision).
- *The named recipient* — deletes files via `mv` to their own inbox, once delivered.

These operations are disjoint by filename: senders never touch existing files; recipients never create files. Axiom A2 is satisfied.

**State semantics** of mailroom contents:
- File present → message in transit, not yet delivered.
- File absent → never sent, or already delivered (sgit history disambiguates).

Empty mailroom subfolders may be GC'd by sgit and recreated on next SEND.

### 2.6 Agent Naming

Agents are named **`{role}.{location}.{session-id}`**, lowercase, dot-separated.

| Position | Purpose | Example values |
|---|---|---|
| `role` | What the agent does | `conductor`, `architect`, `dev`, `librarian`, `appsec`, `dpo`, `sherpa`, `tools` |
| `location` | Where the agent runs | `claude-ai`, `claude-code`, `claude-code-web`, `claude-code-tools` |
| `session-id` | This specific instance | `s-01HXY8K3M2` (10-char Crockford-base-32 ULID-suffix, prefixed `s-`) |

Examples:

```
conductor.claude-ai.s-01HXY8K9PR
architect.claude-ai.s-01HXY8K3M2
dev.claude-code-web.s-01HXY8KAQB
librarian.claude-code.s-01HXY8KEFG
```

Humans participating in the system use `human.{name}` with **no session-id** (humans are continuous):

```
human.dinis
```

### 2.7 Session-id Generation

Each session generates its own session-id once at session start:
- Take a fresh ULID (26 chars).
- Take the last 10 chars (the random portion).
- Prefix `s-`. Result is 12 chars total.

Why 10: only needs to disambiguate within one vault's lifetime. 32^10 ≈ 10^15 of namespace.

### 2.8 RFC 2822 Address Form

The agent name is the local-part of an email address:

```
From: architect.claude-ai.s-01HXY8K3M2 <architect.claude-ai.s-01HXY8K3M2@vault.sgraph.ai>
```

Stays under RFC 5321's 64-char local-part limit. Team is not encoded — vault context implies it.

## 3. Sessions

### 3.1 What a Session Is

A **session** is one agent's coherent unit of work. It is bounded by a brief at the start and an explicit end at the finish. Within those bounds it may span any number of check-ins (pull / deliver / process / send / push cycles), any number of message rounds, any number of hours or days. The Architect session that wrote this specification, for example, is one session — it began when human.dinis handed over a v0.2 design document and asked for an Architect review, and will end when the spec set is locked.

A session is **not**:

- a single message exchange (that's one round inside a session),
- a single check-in (that's one iteration of the work loop inside a session),
- a runtime invocation (the runtime can stop and restart while the session continues — see §3.6 on resumption),
- "every time the agent talks to the vault" (those are check-ins).

### 3.2 The Three-State Lifecycle

Sessions live in one of three folders, reflecting their lifecycle state:

```
mail/sessions/
├── active/             ← in-flight work; mutable; no end.json
├── archived/           ← completed work; immutable; ended gracefully
└── stale/              ← abandoned work; immutable; ended by external decision
```

| State | Has start.json | Has end.json | Mutable? | Who moved it here |
|---|---|---|---|---|
| **active** | yes | no | yes | the agent itself, on session start |
| **archived** | yes | yes | **no** | the agent itself, on graceful end |
| **stale** | yes | yes (with `end_status: stale`) | **no** | another actor (human, peer agent, future GC), on declaring it stale |

### 3.3 Layout (within each state folder)

```
mail/sessions/active/
├── s-01HXY8K3M2/                          ← one folder per session
│   ├── start.json                          ← written once at session start; immutable thereafter
│   ├── events/                             ← append-only; one small file per event
│   │   ├── 01HXY8K3MA__delivered-v0.3.json
│   │   └── 01HXY8K3MZ__delivered-v0.4.json
│   └── (no end.json yet — still active)
├── s-01HXY8K9PR/
│   └── ...
└── s-01HXY8KAQB/
    └── ...

mail/sessions/archived/
├── s-01HW3KXM4P/                          ← session that ended gracefully
│   ├── start.json
│   ├── events/
│   │   └── ...
│   └── end.json                            ← written by the agent themselves
└── s-01HW7BC2N9/
    └── ...

mail/sessions/stale/
└── s-01HV2TXKQ8/                          ← session declared stale by an outside actor
    ├── start.json
    ├── events/
    │   └── ...
    └── end.json                            ← written by whoever moved it here
```

**Why the folder split.** With v0.5's flat `mail/sessions/` layout, distinguishing active from completed work required parsing every session folder for end.json presence. With three folders, the *answer to "what's in flight?"* is just `ls mail/sessions/active/`. Cheap, scannable, no I/O on completed sessions.

**Why folder-not-file (preserved from v0.5).** A single growing log file forces re-reading the whole file on every event. A folder of small immutable event files lets agents read only what they need, gives trivial concurrency, and stays performant as sessions grow long.

### 3.4 `start.json` — the brief

Written **once** at session start, in `mail/sessions/active/{session-id}/start.json`. Immutable from the moment it is written.

```json
{
  "schema_version": "0.6",
  "session_id": "s-01HXY8K3M2",
  "team": "explorer",
  "role": "architect",
  "location": "claude-ai",
  "agent_name": "architect.claude-ai.s-01HXY8K3M2",
  "runtime_model": "claude-opus-4-7",
  "human_driver": "human.dinis",
  "parent_session": "s-01HXY8K9PR",
  "brief": "Refine vault-mail design proposal into canonical Email-FS spec",
  "objectives": [
    "Produce v0.6+ specification",
    "Surface architecturally significant decisions for human ruling",
    "Hand off to vault-viewer team"
  ],
  "started_at": "2026-04-28T17:00:00Z"
}
```

**Mandatory fields:** `schema_version`, `session_id`, `role`, `location`, `agent_name`, `started_at`, `brief`.
**Optional but recommended:** `team`, `runtime_model`, `human_driver`, `parent_session`, `objectives`.

`brief` is a one-line statement of what the session is here to do. `objectives` (added in v0.6) is the structured form — what success looks like, in chunks.

### 3.5 Event files

Any number of append-only events, written into `events/` during the active life of the session. Filename: `{event-ULID}__{slug}.json`.

```json
{
  "schema_version": "0.6",
  "event_id": "01HXY8K3MA",
  "session_id": "s-01HXY8K3M2",
  "at": "2026-04-28T17:42:11Z",
  "kind": "milestone",
  "summary": "Delivered Email-FS v0.3 spec",
  "refs": {
    "messages": [],
    "issues": [],
    "vault_paths": ["docs/email-fs-v0.6-spec.md"]
  }
}
```

`kind` is open-set; suggested values: `milestone`, `decision`, `pushback-received`, `handoff`, `error-recovered`, `paused`, `resumed`, `check-in-start`, `check-in-end`. Events are coarser-grained than reasoning logs (§6); use both.

### 3.6 Check-ins and resumption

Within an active session, the agent does one or more **check-ins** — each check-in being one full pull/deliver/process/send/push cycle. A check-in is *not* a session; the session contains many.

When the agent's runtime stops and a new instance picks up the same brief later (next day, after a context reset, etc.), the new runtime **resumes the existing session** by:

1. Reading `mail/sessions/active/{session-id}/start.json` to recover the brief.
2. Walking `events/` chronologically to reconstruct what's been done.
3. Reading recent reasoning logs in `mail/{agent-name}/outbox/self/` for finer-grained context.
4. Optionally writing a `kind: resumed` event noting the resumption.

The agent name (and therefore the session-id) **stays the same across resumptions**. A new session-id is only generated for genuinely new work, not for runtime continuity.

### 3.7 `end.json` — closing a session

When work is complete (or abandoned, or handed off), the agent writes `end.json` and **moves the session folder** from `active/` to `archived/` in a single commit.

```json
{
  "schema_version": "0.6",
  "session_id": "s-01HXY8K3M2",
  "ended_at": "2026-04-29T19:35:00Z",
  "end_status": "delivered",
  "end_summary": "Email-FS v0.6 spec delivered. Session model refined per dinis pushback. 4 documents updated.",
  "messages_sent": 7,
  "messages_received": 3,
  "events_logged": 5
}
```

`end_status` ∈ `{delivered, incomplete, abandoned, handed-off, stale}`.

For `handed-off`, include `handed_off_to: "{successor-session-id}"`.
For `stale`, see §3.8.

The same commit performs both writes:

```
sgit mv mail/sessions/active/s-01HXY8K3M2/ mail/sessions/archived/s-01HXY8K3M2/
# (write end.json into the new location)
sgit commit -m "session-end: architect.claude-ai.s-01HXY8K3M2 — delivered"
```

### 3.8 Immutability after end

**Once `end.json` exists, the entire session folder is read-only.** This is a hard contract:

- No new events may be appended to `events/`.
- `start.json` and `end.json` may not be modified.
- The folder may not be moved out of `archived/` or `stale/` back to `active/`.

If an agent wants to continue work on the same topic after archiving a session, **they must start a new session**, with `parent_session` pointing at the archived one. The new session's `start.json` carries the continuity reference; the old session remains a sealed audit artifact.

This invariant is currently a convention. Once Email-FS adopts PKI signing, it will be enforced cryptographically — the signature on `end.json` seals the session-folder hash, and any subsequent modification breaks verification. Agents and tools should already behave as if it were enforced.

### 3.9 Stale: declaring a session abandoned

A session is **stale** when its agent has stopped working but never wrote `end.json`. There is no automatic timer in the spec — staleness is always declared by an explicit actor. Whoever declares it is responsible for moving the folder and writing `end.json`.

The threshold for staleness is **use-case-specific**. A real-time orchestration scenario might consider 1 hour stale; a weekly review cadence might be 30 days. The spec does not mandate a value; it mandates the *protocol* for declaring it.

**The stale-declaration protocol:**

```
1. Identify the candidate session in mail/sessions/active/{session-id}/
2. Write end.json into the active folder, with:
     {
       "schema_version": "0.6",
       "session_id": "{session-id}",
       "ended_at": "<now-utc>",
       "end_status": "stale",
       "end_summary": "{why this was declared stale — e.g. 'no commits in 14 days, brief presumed abandoned'}",
       "ended_by": "{actor-name — the session declaring it stale, or human.{name}}",
       "stale_threshold_used": "{e.g. '14 days inactivity'}"
     }
3. sgit mv mail/sessions/active/{session-id}/ mail/sessions/stale/{session-id}/
4. sgit commit -m "session-stale: {session-id} — {short-reason}"
5. sgit push
```

The `ended_by` field is new in stale ends and identifies the actor who declared it (because they are not the session's own agent). The `stale_threshold_used` field documents the policy that was applied.

Once moved to `stale/`, the session is immutable like any archived session.

If the original agent later returns and finds their session in stale/, they cannot revive it. They start a new session, with `parent_session` pointing at the stale one — same continuity pattern as archived/ → new.

### 3.10 Read protocol

Common queries against the sessions index:

| Question | Operation |
|---|---|
| What sessions are active? | `ls mail/sessions/active/` |
| What was completed recently? | `ls -t mail/sessions/archived/ \| head` |
| Who's been here ever? | `ls mail/sessions/{active,archived,stale}/` |
| What's the brief for session X? | `cat mail/sessions/{active\|archived\|stale}/{X}/start.json` |
| What did session X do? | Read `start.json`, `end.json` (if present), walk `events/` |
| What sessions in this vault stalled? | `ls mail/sessions/stale/` |
| What did anyone do today? | Walk `events/*.json` across all three state folders, filter by `at` |

The future `email-fs sessions` CLI tabulates and filters this.

### 3.11 No central index file

By design. A central `index.json` would be a multi-writer file with merge conflicts on every parallel session start. The directory listings *are* the indexes — three of them now (active/, archived/, stale/), each independently scannable, no coordination needed.

## 4. Message Format

### 4.1 The eml

Strict RFC 2822 with MIME (RFC 2045–2049) for attachments. No proprietary format. Any RFC-compliant tool reads and writes the same bytes.

### 4.2 Mandatory Headers

| Header | Value |
|---|---|
| `From` | `{agent-name} <{agent-name}@vault.sgraph.ai>` |
| `To` | One recipient per message in v0.6 (Cc/Bcc deferred) |
| `Date` | UTC, RFC 2822 format |
| `Subject` | Human-readable summary. **Treat subjects as first-class** — they form the visible audit log when an agent or human scans a directory listing |
| `Message-ID` | `<{ULID}@vault.sgraph.ai>` — same ULID as filename |
| `Content-Type` | `text/plain; charset=utf-8` or `multipart/mixed; boundary=…` |

**Note on subject lines.** Because the directory listing is the at-a-glance view of activity in the vault, subjects function as the visible audit timeline. A vault scan reading "build-dev-page → page-shells-done → review-dev-page → pr-opened-dev-section → spec-feedback" tells the story without opening a single message. Conventions are encouraged: imperative verbs for tasks, past-tense for completions, `Re:` for replies. The future `email-fs log` CLI will rely on subject quality.

### 4.3 Conditional Headers

| Header | When | Purpose |
|---|---|---|
| `In-Reply-To` | Reply | Parent's `Message-ID` |
| `References` | Thread of ≥ 2 prior messages | Space-separated chain, oldest first |

### 4.4 X-Headers (Email-FS Extensions)

| Header | Purpose |
|---|---|
| `X-EmailFS-Schema-Version` | `0.6` for messages following this spec; `0.5` accepted for legacy compat |
| `X-EmailFS-Vault-ID` | Vault this message lives in |
| `X-EmailFS-Issues-Ref` | Path(s) to related Issues-FS files |
| `X-EmailFS-Asset-Ref` | Path(s) to vault assets referenced by this message (use **instead of** MIME attachment when the asset already lives in the vault) |
| `X-EmailFS-Priority` | `low` \| `normal` \| `high` \| `urgent` (default `normal`) |
| `X-EmailFS-Kind` | `task` \| `reply` \| `notification` \| `reasoning-log` \| `forward` |
| `X-EmailFS-Sender-Session` | The sender's session-id (redundant with `From:` but useful for filtering) |
| `X-EmailFS-Recipient-Session` | The recipient's session-id when known at send time (often omitted) |

**Forbidden:** any `Status:` header in the eml. Status is sidecar-only.

### 4.5 Vault Assets vs MIME Attachments

If an agent is referencing a file that already lives in the vault, **do not attach it as MIME** — reference it via `X-EmailFS-Asset-Ref` and a path in the body. Reasons:

1. The vault is the canonical store. Attaching duplicates content and breaks single-source-of-truth.
2. Vault paths survive forward/reply chains; copies of attachments do not.
3. Reading agents can fetch with a path open, not by parsing MIME.

MIME attachments are reserved for content that **does not** already live in the vault — external uploads, attachments arriving from real SMTP gateways, etc.

### 4.6 Filename Convention

```
{ULID}__{sender}__{slug}.eml
```

- `ULID` — 26-char Crockford-base-32. Lex-sortable, collision-free without coordination.
- `sender` — full agent name including session-id.
- `slug` — kebab-case, ≤ 6 words, derived from subject.

Example:

```
01HXY8K3M2P4QRS5TVWZ7C9DAB__architect.claude-ai.s-01HXY8K3M2__email-fs-v0.6-spec.eml
```

The filename ULID **must equal** the `Message-ID` ULID. Mismatch = malformed.

## 5. Sidecar State (Per-Message Workspace)

### 5.1 The Sidecar Is Not Just a Status Flag

In earlier drafts, the sidecar stored "status: read/processed". This understates it. **The sidecar is the recipient's structured workspace for everything they do with the message:**

- Required: status, timestamps (delivered/read/processed/purged), thread root.
- Optional and extensible: extracted attachments, derived assets, links to issues raised, scratch reasoning, processing pipeline state, agent-specific labels, retry counters, escalations, custom metadata for downstream tooling.

This is where agent-specific behaviour lives. Two recipients can handle the same incoming message in entirely different ways — one extracts attachments and runs OCR, another raises an Issues-FS task, another forwards to a peer. All of that workflow state belongs in the sidecar, not the eml.

### 5.2 Sidecar as Folder

Per-message sidecar is **a folder, not a single file** — same scaling argument as sessions. One small immutable file per concern beats one growing file.

```
mail/{agent}/state/{ULID}/
├── core.json                          ← REQUIRED. Status + lifecycle timestamps.
├── labels.json                        ← optional. Recipient's tags/categories.
├── attachments/                       ← optional. Extracted MIME parts (if any).
│   ├── manifest.json
│   └── 01__report.pdf
├── extracted/                         ← optional. Derived artifacts.
│   ├── ocr.txt
│   └── summary.md
├── workflow/                          ← optional. Per-agent processing state.
│   └── retries.json
└── notes/                             ← optional. Recipient's reasoning trace.
    └── 01HXY8...__why-i-prioritised-this.md
```

### 5.3 `core.json` (required)

The minimum any conformant recipient must write.

```json
{
  "schema_version": "0.6",
  "message_id": "01HXY8K3M2P4QRS5TVWZ7C9DAB",
  "status": "read",
  "delivered_at": "2026-04-28T17:05:11Z",
  "read_at": "2026-04-28T17:07:42Z",
  "processed_at": null,
  "purged_at": null,
  "processing_notes": null,
  "thread_root": "01HXY8KFOO...",
  "priority_actual": "high"
}
```

**Required fields:** `schema_version`, `message_id`, `status`, `delivered_at`, `thread_root`.
**Lifecycle fields** (set when transition occurs): `read_at`, `processed_at`, `purged_at`.
**Optional fields** (recipient may use): `processing_notes`, `priority_actual` (recipient's evaluation; may differ from `X-EmailFS-Priority`), and any extension fields.

`status` lifecycle: `unread` → `read` → `processed` → optionally `purged`.

### 5.4 Extension Files (open schema)

Beyond `core.json`, the recipient is free to write any additional files in their sidecar folder. Conventions:

- Names should be descriptive and lowercase (e.g. `labels.json`, `attachments/manifest.json`).
- File formats are recipient's choice (JSON for structured data, MD for prose, raw extracted bytes for assets).
- `email-fs fsck` only validates `core.json`. Extensions are agent-specific freedom.

**Examples of recipient-defined extensions in real use:**

| Agent | Extension | Purpose |
|---|---|---|
| `dev.claude-code-web` | `pr.json` | The PR opened in response, with status, URL, last-checked-at |
| `appsec.claude-code` | `risk.json` | Risk assessment of the request, risk-rating, mitigations applied |
| `librarian.claude-code` | `cataloged.json` | Catalog refs added to master index |
| `dpo.claude-ai` | `gdpr.json` | Lawful basis assessment, retention class, purge-eligibility-date |

This is what your "tons of uses" comment was getting at — the sidecar is **per-recipient infrastructure** that turns Email-FS into a substrate for arbitrarily-complex agent behaviour while keeping the eml itself a clean immutable record.

## 6. Self-Addressed Reasoning Logs

Per ruling: every agent emails itself on every meaningful decision.

### 6.1 What Counts as Meaningful

- Choosing one option over another with non-trivial consequences.
- Departing from a brief or instruction.
- Resolving an ambiguity.
- Catching and recovering from a mistake.
- Completing a stage of work.

The bar is "a future reviewer would want to see this", not "every thought".

### 6.2 Format

`From: self, To: self`. Set `X-EmailFS-Kind: reasoning-log`.

```
From: architect.claude-ai.s-01HXY8K3M2 <architect.claude-ai.s-01HXY8K3M2@vault.sgraph.ai>
To: architect.claude-ai.s-01HXY8K3M2 <architect.claude-ai.s-01HXY8K3M2@vault.sgraph.ai>
Date: Tue, 28 Apr 2026 17:42:00 +0000
Subject: Decision — sessions split into active/archived/stale
Message-ID: <01HXY8KREASON0001@vault.sgraph.ai>
X-EmailFS-Schema-Version: 0.6
X-EmailFS-Kind: reasoning-log

Decision: mail/sessions/ becomes three folders — active/, archived/, stale/.

Rationale: human.dinis flagged that v0.5's flat sessions/ folder forces every
reader to parse end.json presence to distinguish in-flight from completed work.
Three folders make that free: ls active/ is the in-flight index; ls archived/
is completed; ls stale/ is abandoned. Same scaling principle as folder-not-file
elsewhere in the design.
```

### 6.3 Storage

`mail/{self}/outbox/self/`. No DELIVER step (addressed to self).

### 6.4 Inspection

Read `mail/{agent}/outbox/self/` chronologically (alphabetical = chronological by ULID).

### 6.5 Reasoning Logs vs. Session Events

| | Reasoning log | Session event (§3.3) |
|---|---|---|
| Address | self ↔ self, full eml | session-scoped JSON |
| Granularity | Fine. Any meaningful decision. | Coarse. Milestones, handoffs, status changes. |
| Threadable | Yes (RFC 2822 In-Reply-To) | Within events/ folder, by event_id |
| Audience | Future agents acquiring full context | Anyone scanning "who's done what at the milestone level" |

Both are append-only. Both contribute to provenance. Use both.

## 7. The Five Primitive Operations

Five primitives. Anything else composes from them. Each is specified as file-level operations executable by hand.

### 7.1 SEND

**Actor:** Sender. **Precondition:** `sgit pull` is current.

```
1. Generate ULID.
2. Compose eml. Ensure Message-ID ULID == filename ULID.
3. Write to mailroom (transit):
     mail/mailroom/{recipient}/{ULID}__{sender}__{slug}.eml
4. Write copy to own outbox (permanent record):
     mail/{sender}/outbox/{recipient}/{ULID}__{sender}__{slug}.eml
5. Optionally write a session event in mail/sessions/{sender-session}/events/.
6. sgit commit "send: {ULID} → {recipient} — {slug}"
7. sgit push
```

### 7.2 DELIVER

**Actor:** Recipient. **Precondition:** `sgit pull` is current.

```
1. ls mail/mailroom/{self}/
   (returns 0 to N files)

2. For each .eml found:
     a. mv mail/mailroom/{self}/{ULID}__...eml \
           mail/{self}/inbox/{ULID}__...eml
     b. Create sidecar folder mail/{self}/state/{ULID}/
        Write core.json with:
          status: "unread"
          delivered_at: <now-utc>
          thread_root: <derived from In-Reply-To/References, or own Message-ID>

3. sgit commit "deliver: {N} new message(s) to {self}"
4. sgit push
```

**Empty mailroom == delivered.** This is the sender's delivery receipt — a fact in vault history, not a header in a reply.

**Idempotency:** DELIVER on empty mailroom is a no-op.

### 7.3 READ

**Actor:** Recipient.

```
1. cat mail/{self}/inbox/{ULID}__...eml
2. Update mail/{self}/state/{ULID}/core.json:
     status: "read"
     read_at: <now-utc>
3. (commit may be batched with PROCESS or session end)
```

READ may be silent in-session. Convention: commit at session end.

### 7.4 PROCESS

**Actor:** Recipient. **Precondition:** the agent has actioned the message.

```
1. mv mail/{self}/inbox/{ULID}__...eml \
      mail/{self}/processed/{ULID}__...eml
2. Update mail/{self}/state/{ULID}/core.json:
     status: "processed"
     processed_at: <now-utc>
     processing_notes: "{free text — optional}"
3. (Optionally write extension files in mail/{self}/state/{ULID}/ —
    e.g. attachments/, extracted/, workflow/, etc.)
4. sgit commit "process: {ULID}"
5. sgit push
```

**Sender verification.** On next pull, sender can confirm round-trip with three observations:
- Mailroom empty for that ULID → delivered.
- Eml in recipient's `processed/` → actioned.
- `core.json` shows `processed_at` → with timestamp.

### 7.5 PURGE

**Actor:** Recipient. **Precondition:** message is in `processed/` (cannot purge directly from inbox).

```
1. Read eml's Message-ID, From, Date, Subject.
2. Write tombstone:
     mail/{self}/purged/{ULID}.tombstone.json
     {
       "schema_version": "0.6",
       "message_id": "{ULID}",
       "from": "{sender}",
       "date": "{original Date}",
       "subject": "{original Subject}",
       "purged_at": "<now-utc>",
       "reason": "{free text — optional}"
     }
3. rm mail/{self}/processed/{ULID}__...eml
4. rm -rf mail/{self}/state/{ULID}/
5. sgit commit "purge: {ULID} — {reason or 'retention'}"
6. sgit push
```

**Postcondition:** eml gone from recipient's mailbox. Sender's outbox copy unaffected. Tombstone preserves thread integrity (readers walking In-Reply-To chains land on a stub, not a 404).

### 7.6 Commit Discipline

The atomicity unit is **one logical agent action, one commit**. Some examples:

| Logical action | Files written/changed | Commit |
|---|---|---|
| Send 1 message | mailroom + outbox | one commit |
| Send 3 messages in session-end batch | 3 × (mailroom + outbox) | one commit |
| Deliver 5 incoming messages | 5 inbox moves + 5 sidecar folders | one commit |
| Read a message | core.json updated | may batch with PROCESS or session end |
| Process a message | move to processed/ + sidecar update + extension files | one commit |
| Purge | tombstone + delete eml + delete sidecar | one commit |

This is the right granularity for two reasons. Operationally, commits are the natural progress checkpoint. Audit-wise, one-action-one-commit makes `sgit log` readable as the story of what happened.

## 8. Threading

Standard RFC 2822 — `In-Reply-To` and `References`. No custom thread store; threads are a view, reconstructed in O(N) by walking headers.

```
SEND #1:        Message-ID: <01ABC@vault.sgraph.ai>
                (no In-Reply-To)

REPLY:          Message-ID: <01DEF@vault.sgraph.ai>
                In-Reply-To: <01ABC@vault.sgraph.ai>
                References: <01ABC@vault.sgraph.ai>

REPLY-TO-REPLY: Message-ID: <01GHI@vault.sgraph.ai>
                In-Reply-To: <01DEF@vault.sgraph.ai>
                References: <01ABC@vault.sgraph.ai> <01DEF@vault.sgraph.ai>
```

Reconstruction: walk `In-Reply-To` backward; grep `In-Reply-To: <{this-id}>` to walk forward.

## 9. Cross-Vault Forwarding (Forward-Compatible Stub)

**Not implemented in v0.6.** Documented to ensure the v0.6 design is compatible with later cross-vault forward.

When implemented, cross-vault forwards will:
- **Preserve original `Message-ID`** (RFC convention).
- Add `Resent-From:`, `Resent-To:`, `Resent-Date:` headers per RFC 2822 §3.6.6.
- Set `X-EmailFS-Forwarded-From-Vault: {origin-vault-id}`.
- The forwarded copy lives in the destination vault's `mail/` and is immutable as usual.

The v0.6 spec is compatible with this without change. Implementations should not assume `Message-ID` is unique within a single vault; it is unique within the *originating* vault, and may appear in multiple vaults via forwarding.

## 10. Conformance

A vault is **Email-FS-conformant at v0.6** if and only if:

| # | Check | Pass criterion |
|---|---|---|
| C1 | Namespace folder | `mail/` exists at vault root |
| C2 | Sessions three-state layout | `mail/sessions/active/`, `mail/sessions/archived/`, `mail/sessions/stale/` all exist |
| C3 | Active session integrity | Every folder under `mail/sessions/active/` has a `start.json` and **no** `end.json` |
| C4 | Archived session integrity | Every folder under `mail/sessions/archived/` has both `start.json` and `end.json`; `end_status ≠ "stale"` |
| C5 | Stale session integrity | Every folder under `mail/sessions/stale/` has both `start.json` and `end.json`; `end_status = "stale"`; `end.json` has `ended_by` field |
| C6 | Session immutability | (Audit-only) For any session in `archived/` or `stale/`, no commit has modified its contents after its `end.json` was written |
| C7 | Per-agent layout | Each `mail/{agent}/` has `outbox/`, `inbox/`, `processed/`, `purged/`, `state/` |
| C8 | Mailroom folder | `mail/mailroom/` exists (subfolders may be GC'd when empty) |
| C9 | Filename binding | Every `.eml` filename's ULID equals its `Message-ID` ULID |
| C10 | Schema declaration | Every `.eml` has `X-EmailFS-Schema-Version: 0.6` (or `0.5` for legacy messages) |
| C11 | RFC 2822 validity | Every `.eml` parses with stdlib RFC 2822 parser |
| C12 | Sidecar core | Every file in `inbox/` and `processed/` has a matching `state/{ULID}/core.json` |
| C13 | No status header | No `.eml` contains a `Status:` header |
| C14 | Outbox archive | Every delivered message has a copy in some `{sender}/outbox/{recipient}/` |
| C15 | Tombstone integrity | Every file in `purged/` is `{ULID}.tombstone.json`; no corresponding `.eml` in same mailbox |
| C16 | Single-writer rule | (Audit-only) sgit history shows all writes to `mail/{agent}/` were authored by `{agent}` |
| C17 | Session reference integrity | Every agent name appearing in any eml header has a corresponding `start.json` somewhere under `mail/sessions/{active,archived,stale}/{session-id}/` |
| C18 | No eml mutation | No commit has ever modified an existing `.eml`'s bytes (mv allowed; rewrite not) |

Future `email-fs fsck` enforces these.

## 11. Implementation Path

This document is the **specification**. Implementations will follow.

### 11.1 Stage 1 — Manual Use (current)

Email-FS is operated by hand using `cat`, `cp`, `mv`, `rm`, `sgit`. This forces the protocol to be simple enough that any agent or human can execute it without tooling. Edge cases surface during this stage.

### 11.2 Stage 2 — `email-fs` CLI (next)

Once the protocol has been stress-tested manually across a few vaults — and the rough edges have been smoothed — Email-FS will be implemented as a CLI tool, likely `pip install email-fs`. Mirrors the path Issues-FS took. Commands will include at minimum:

```
email-fs init
email-fs send --to {recipient} --subject "..." [--body-file ... | --interactive]
email-fs deliver
email-fs read {ULID}
email-fs process {ULID} [--note "..."]
email-fs purge {ULID} [--reason "..."]
email-fs sessions [--active] [--since DATE]
email-fs threads {ULID}
email-fs log [--by-subject] [--since DATE]
email-fs fsck
```

The CLI will be a thin wrapper over the file operations defined in §7. The protocol is the contract; the CLI is convenience.

### 11.3 Stage 3 — Native Vault-Viewer Support

Vault-rendering tools (the Cyber Boardroom data-room viewer, `send.sgraph.ai`, etc.) gain native rendering of `mail/` namespaces:
- Inbox views per agent.
- Threaded conversation views.
- Session timelines.
- Subject-line audit log views.
- Quasi-email-client UI on top of the on-disk format.

Because the wire format is real RFC 2822, third-party email clients (Thunderbird, Apple Mail) can also open `.eml` files directly with no adaptation.

### 11.4 Stage 4 — MCP Exposure

Once the CLI exists, expose its commands as MCP tools so agent runtimes can SEND, DELIVER, READ, PROCESS, PURGE through the tool interface natively, without shelling out.

### 11.5 Stage 5 — Bridge to Real SMTP

`v0.7.6__dev-brief__sg-workmail-fastapi-service.md` already specifies the FastAPI service that bridges to AWS WorkMail. Email-FS is byte-compatible with what that service produces and consumes — the bridge is essentially `cp` in both directions plus header rewriting for routing.
