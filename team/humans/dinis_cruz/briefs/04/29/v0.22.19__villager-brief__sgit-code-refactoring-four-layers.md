## Why Now

SGit is maturing. Real users are hitting real challenges. The innovation phase must continue, but the risk of new features breaking existing functionality is growing. The next major feature set (PKI native support) will touch the deepest layers of the codebase. If those layers are not clean, well-tested, and well-separated before PKI lands, the complexity will compound.

There is also a window: SGit does not yet have wide adoption. Design changes that would be painful later are still cheap now. This is the moment to make them.

This is not about adding features. This is about making the codebase ready to safely receive the next wave of features.

## The Four-Layer Architecture

All SGit code should be organised into four layers, each in its own folder, each with different properties, different review requirements, and different testing strategies.

### Layer 1: Crypto

**What it contains:** All cryptographic operations required for SGit. Key derivation (PBKDF2, 600k iterations), encryption/decryption (AES-256-GCM), hashing, signature verification, and eventually PKI operations.

**Properties:**
- Changes should be extremely rare after stabilisation
- Every change requires heavy review, testing, and justification
- Eventually moves to its own separate repo
- No side effects: pure functions, input in, output out
- Zero dependencies on other layers (no network calls, no disk access, no vault operations)

**Review standard:** Highest. Any change to crypto is a potential security vulnerability. Every PR touching this layer gets dedicated security review.

### Layer 2: Core

**What it contains:** Every action that changes state on disk. Commit, push, pull, init, branch, merge, add, delete. These are the operations that modify the vault's file structure, tree pointers, and object store.

**Properties:**
- Each action is a discrete, type-safe step: typed input, typed output
- Each action should be expressible as a JSON record (for transaction logging and replay)
- Side effects are explicit and captured (which files changed, which pointers moved)
- Depends on Crypto layer (for encryption/decryption of objects)
- Does not depend on Network or Plugin layers

**Review standard:** High. Changes here affect data integrity. Every PR needs tests that verify state transitions.

### Layer 3: Network

**What it contains:** Every call to interact with the SG/Send server API. Upload objects, download objects, fetch tree, push branch, pull branch, authenticate.

**Properties:**
- The API client layer
- Depends on Crypto (for encrypting/decrypting data in transit)
- Does not depend on Core (Core calls Network, not the other way around)
- Should be mockable for offline testing
- Every request/response pair is type-safe and loggable

**Review standard:** Medium-high. Changes here affect connectivity and data transfer. API contract changes need coordination with the server team.

### Layer 4: Plugins (Read Layer + Extensions)

**What it contains:** Everything else. Listing files, viewing logs, displaying status, formatting output, searching, filtering, diffing, and any future extensions.

**Properties:**
- Read-only operations (no state changes)
- Can be iterated independently of the other three layers
- Can eventually live in separate repos
- Can be feature-flagged (not all plugins enabled for all users)
- Depends on Core (for reading state) and Network (for fetching remote state)
- Adding or removing a plugin never affects Core, Crypto, or Network

**Review standard:** Normal. Plugins can iterate fast because they cannot break state.

### The Layer Dependency Graph

```
Plugins (Layer 4)
    |
    +--> depends on Core (Layer 2) for reading state
    +--> depends on Network (Layer 3) for remote reads
    |
Core (Layer 2)
    |
    +--> depends on Crypto (Layer 1) for encryption
    +--> depends on Network (Layer 3) for remote operations
    |
Network (Layer 3)
    |
    +--> depends on Crypto (Layer 1) for in-transit encryption
    |
Crypto (Layer 1)
    |
    +--> depends on nothing (pure functions)
```

No upward dependencies. Crypto never calls Core. Core never calls Plugins. This is enforced by folder structure and imports.

## Transaction Logging

Every Core action (every state change) should produce a transaction record: a type-safe JSON object describing what happened.

```json
{
  "action": "commit",
  "timestamp": "2026-04-29T10:15:32.456Z",
  "vault_id": "abc123",
  "branch": "main",
  "parent_commit": "def456",
  "new_commit": "ghi789",
  "files_added": ["report.md"],
  "files_modified": [],
  "files_deleted": [],
  "tree_hash": "jkl012",
  "duration_ms": 45
}
```

These transaction records enable:

- **Replay:** Re-execute a sequence of actions against a known starting state to reproduce a bug
- **Monitoring:** Track what operations are happening, how long they take, where bottlenecks are
- **Testing:** Assert that a specific sequence of actions produces a specific end state
- **Audit:** Full history of every state change, independent of the vault's own commit history
- **Transaction support (future):** Rollback a partially completed multi-step operation

The transaction log is not the vault's commit history. It is a record of SGit's operations, including failed operations, retries, and intermediate states that the vault's commit history does not capture.

## Step-Based Action Decomposition

Every CLI command (commit, push, pull, init, etc.) should decompose into discrete steps. Each step is a typed function: input object in, output object out.

Example: `sgit push`

```
Step 1: Read local branch state
    Input:  { vault_id, branch_name }
    Output: { local_head, local_commits_ahead }

Step 2: Read remote branch state
    Input:  { vault_id, branch_name, server_url }
    Output: { remote_head, remote_commits }

Step 3: Calculate diff
    Input:  { local_head, remote_head, local_commits_ahead }
    Output: { objects_to_upload, new_tree_hash }

Step 4: Upload objects
    Input:  { objects_to_upload, server_url }
    Output: { uploaded_count, failed_count, errors }

Step 5: Update remote branch pointer
    Input:  { branch_name, new_head, server_url }
    Output: { success, previous_head }

Step 6: Log transaction
    Input:  { all step outputs }
    Output: { transaction_record }
```

Each step is independently testable. Each step's input and output is a typed object. The sequence of steps can be interrupted, resumed, or rolled back. Performance bottlenecks are visible at the step level (Step 4 took 3 seconds, everything else was under 50ms).

## Folder Structure

```
sgit/
    |
    +--> crypto/
    |       +--> aes_gcm.py
    |       +--> key_derivation.py
    |       +--> hashing.py
    |       +--> tests/
    |
    +--> core/
    |       +--> actions/
    |       |       +--> commit.py      (steps for commit)
    |       |       +--> push.py        (steps for push)
    |       |       +--> pull.py        (steps for pull)
    |       |       +--> init.py
    |       |       +--> branch.py
    |       |       +--> merge.py
    |       +--> transaction_log.py
    |       +--> tests/
    |
    +--> network/
    |       +--> api_client.py
    |       +--> upload.py
    |       +--> download.py
    |       +--> auth.py
    |       +--> tests/
    |
    +--> plugins/
    |       +--> list_files.py
    |       +--> log_viewer.py
    |       +--> status.py
    |       +--> diff.py
    |       +--> search.py
    |       +--> tests/
    |
    +--> cli/
            +--> commands/          (CLI wrappers that call core actions)
            +--> formatters/        (output formatting for terminal)
```

The CLI is a thin wrapper. It parses arguments, calls Core actions (for writes) or Plugins (for reads), and formats output. It contains no business logic.

## Performance Visibility

The refactoring is not about fixing performance directly. It is about making performance visible. Once every action is decomposed into steps with timing, the bottlenecks become obvious:

- "Push takes 5 seconds" becomes "Push Step 4 (upload objects) takes 4.8 seconds, everything else is under 200ms"
- "Clone is slow" becomes "Clone Step 2 (download objects) downloads 150 objects sequentially instead of in parallel"

The fix then targets the specific step, not the whole operation. And the fix can be validated by comparing step timings before and after.

## Preparing for PKI

The PKI feature set (signed commits, per-branch access control, key distribution) will primarily touch:

- **Crypto layer:** New operations (signing, verification, key management)
- **Core layer:** New steps in commit (sign) and pull (verify)
- **Network layer:** Key exchange endpoints

If these three layers are clean and well-separated before PKI work begins, the PKI implementation can be scoped to specific files in specific layers. If the layers are entangled, PKI will touch everything and the risk of regression is high.

## Feature Flags for Plugins

Plugins should be enableable/disableable via a config file:

```json
{
  "plugins": {
    "list_files": { "enabled": true },
    "log_viewer": { "enabled": true },
    "diff": { "enabled": true },
    "search": { "enabled": false, "stability": "experimental" }
  }
}
```

Disabled plugins: their CLI commands do not register, their code is not loaded. This is the same pattern as the ephemeral infrastructure backend plugins, applied to SGit itself.

## Acceptance Criteria

| # | Criterion | Verification |
|---|-----------|-------------|
| 1 | Four-layer folder structure implemented | crypto/, core/, network/, plugins/ each exist with clear boundaries |
| 2 | No upward dependencies (Crypto depends on nothing, Core does not call Plugins) | Static analysis or import audit confirms no upward imports |
| 3 | Crypto layer has zero dependencies on other layers | Crypto tests pass with Core, Network, and Plugins removed |
| 4 | Every Core action decomposed into typed steps | commit, push, pull, init each have step functions with typed input/output |
| 5 | Transaction log produced for every Core action | Run a commit, verify JSON transaction record written |
| 6 | Transaction records are replayable | Replay a transaction sequence against a fresh vault, verify identical end state |
| 7 | Step-level timing captured | Run push, verify per-step duration_ms in transaction record |
| 8 | Plugins are feature-flaggable | Disable a plugin in config, verify its CLI command is not registered |
| 9 | CLI is a thin wrapper with no business logic | Audit CLI code, confirm it only parses args and calls Core/Plugins |
| 10 | Core tests pass independently of Plugin tests | Run core tests with plugins disabled, all pass |
| 11 | Performance bottleneck visible at step level after refactoring | Identify slowest step in push/pull by reading transaction log |
| 12 | No regressions: all existing SGit functionality works after refactoring | Full integration test suite passes |
