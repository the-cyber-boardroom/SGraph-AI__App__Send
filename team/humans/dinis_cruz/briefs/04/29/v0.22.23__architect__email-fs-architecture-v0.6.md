## 4. The Mailroom

The only producer-consumer zone in Email-FS. Everything else is single-writer.

```
                        mail/mailroom/
                              │
       ┌──────────────────────┼──────────────────────┐
       ▼                      ▼                      ▼
  ┌──────────┐           ┌──────────┐           ┌──────────┐
  │ alice/   │           │ bob/     │           │ carol/   │
  └────┬─────┘           └────┬─────┘           └─────┬────┘
       │                      │                       │
       │                      │                       │
   ┌───┴───┐             ┌────┴────┐             ┌────┴────┐
   │       │             │         │             │         │
   ▼       ▼             ▼         ▼             ▼         ▼
 (file) (file)         (file)   (file)        (empty)
   ▲      ▲              ▲        ▲
   │      │              │        │
 BOB    DAVE          CAROL    ALICE
 sent   sent          sent     sent
        ↑                                            ↑
        │                                            │
        └──── any sender writes new files ──────────┘
              (filename ULIDs are globally unique → no name collision)

  ALICE deletes ALICE's pigeonhole files (via mv to her inbox).
  BOB deletes BOB's pigeonhole files (via mv to his inbox).
  No agent ever touches another agent's pigeonhole.

  Disjoint operations:
  - Senders: only CREATE files
  - Recipients: only DELETE (move) files
  → Single filename never has two writers.

  Empty pigeonhole = "delivered or never sent" (sgit history disambiguates)
```

## 5. The Five Primitives — Visual Reference

### 5.1 SEND

```
SENDER (alice.claude-ai.s-XX)              vault state after SEND

  1. Generate ULID                         mail/
  2. Compose eml                           ├── mailroom/
  3. Write to mailroom ──────────────►     │   └── bob.../
  4. Write copy to own outbox ────────►    │       └── 01HXY...eml   ← in transit
  5. (optional) session event              ├── alice.../
  6. sgit commit "send: ..."               │   └── outbox/
  7. sgit push                             │       └── bob.../
                                           │           └── 01HXY...eml  ← archived
                                           └── ...
```

### 5.2 DELIVER

```
RECIPIENT (bob.claude-code.s-YY)           vault state after DELIVER

  1. ls mail/mailroom/bob.../              mail/
     → 01HXY...eml found                   ├── mailroom/
                                           │   └── bob.../          ← now empty
  2. mv mailroom/bob/01HXY...eml           │       (file moved)
        bob/inbox/01HXY...eml              ├── bob.../
                                           │   ├── inbox/
  3. mkdir bob/state/01HXY/                │   │   └── 01HXY...eml  ← delivered
     write core.json:                      │   └── state/
       status: "unread"                    │       └── 01HXY/
       delivered_at: ...                   │           └── core.json
       thread_root: ...                    └── ...
  4. sgit commit "deliver: 1 message"
  5. sgit push

  Sender's view on next pull:
  - mailroom/bob/01HXY...eml is GONE  ← delivery confirmed by file absence
  - Commit message tells WHO delivered and WHEN
```

### 5.3 READ

```
RECIPIENT                                  state file evolution

  1. cat bob/inbox/01HXY...eml             core.json BEFORE:
                                             { status: "unread",
  2. update bob/state/01HXY/core.json:       delivered_at: T0,
       status: "read"                        read_at: null,
       read_at: <now>                        processed_at: null }

  3. (commit may be batched)               core.json AFTER:
                                             { status: "read",
                                               delivered_at: T0,
                                               read_at: T1,
                                               processed_at: null }
```

### 5.4 PROCESS

```
RECIPIENT                                  vault state after PROCESS

  1. mv bob/inbox/01HXY...eml              mail/
        bob/processed/01HXY...eml          └── bob.../
                                               ├── inbox/         ← empty (or N-1)
  2. update bob/state/01HXY/core.json:         ├── processed/
       status: "processed"                     │   └── 01HXY...eml  ← actioned
       processed_at: <now>                     └── state/
                                                   └── 01HXY/
  3. (optionally) write extension files:           ├── core.json (updated)
     bob/state/01HXY/                              ├── pr.json     (optional)
       ├── pr.json                                 ├── extracted/  (optional)
       ├── extracted/                              └── notes/      (optional)
       └── notes/

  4. sgit commit "process: 01HXY"
  5. sgit push

  Sender's verification (3 facts in vault history):
  ┌──────────────────────────────────────────────────────────┐
  │ ✓ mailroom/bob/01HXY...eml          → ABSENT  (delivered)│
  │ ✓ bob/processed/01HXY...eml         → PRESENT (actioned) │
  │ ✓ bob/state/01HXY/core.json         → processed_at set   │
  └──────────────────────────────────────────────────────────┘
```

### 5.5 PURGE

```
RECIPIENT                                  vault state after PURGE

  PRECONDITION: message is in processed/   mail/
                                           └── bob.../
  1. Read eml's metadata                       ├── processed/   ← eml gone
  2. Write tombstone:                          ├── purged/
     bob/purged/01HXY.tombstone.json           │   └── 01HXY.tombstone.json
       {                                       │       (200-byte stub)
         message_id, from, date,               └── state/
         subject, purged_at, reason                     ← state/01HXY/ removed
       }
  3. rm bob/processed/01HXY...eml          Sender's outbox copy is UNTOUCHED.
  4. rm -rf bob/state/01HXY/               Threads still resolve via tombstone.
  5. sgit commit "purge: 01HXY — ..."
  6. sgit push
```

## 9. Multi-Agent Fan-Out (One Sender, Many Recipients)

Conductor pushes a brief to three agents in one session.

                            VAULT STATE
   ─────────────────────────────────────────────────────────────────
                                  ▲
                                  │  sgit push
                                  │
   ┌──────────────────────────────┴──────────────────────────────┐
   │                                                             │
   │  mailroom/                                                  │

                    BOB ──┐
                          │
                          │  SEND

                          │                     ULIDs are globally unique)

                          │

                          │

   Each sender is on a different sgit branch (or pushes asynchronously).

     ONE commit: "deliver: 3 new messages"

## 11. Sessions — Three-State Lifecycle

How `mail/sessions/` answers "who's working, who's done, who walked away?"

```
                              mail/sessions/
                                     │
              ┌──────────────────────┼──────────────────────┐
              ▼                      ▼                      ▼
        ┌──────────┐           ┌──────────┐           ┌──────────┐
        │ active/  │           │archived/ │           │  stale/  │
        │ MUTABLE  │           │IMMUTABLE │           │IMMUTABLE │
        └────┬─────┘           └────┬─────┘           └────┬─────┘
             │                      │                      │
   ┌─────────┴─────────┐    ┌───────┴───────┐    ┌─────────┴─────────┐
   ▼                   ▼    ▼               ▼    ▼                   ▼
┌──────────┐     ┌──────────┐ ┌──────────┐    ┌──────────┐
│s-01HXY8K3│     │s-01HXY8KA│ │s-01HW3KX │    │s-01HV2TX │
│ M2/      │     │ QB/      │ │ M4P/     │    │ KQ8/     │
│          │     │          │ │          │    │          │
│start.json│     │start.json│ │start.json│    │start.json│
│events/   │     │events/   │ │events/   │    │events/   │
│          │     │          │ │end.json  │    │end.json  │
│ (no      │     │ (no      │ │ status:  │    │ status:  │
│  end.json)│    │  end.json)│ │ delivered│    │ stale    │
│          │     │          │ │          │    │ ended_by:│
│IN-FLIGHT │     │IN-FLIGHT │ │GRACEFULLY│    │ external │
│          │     │          │ │ ENDED    │    │ actor    │
└──────────┘     └──────────┘ └──────────┘    └──────────┘


   Read patterns:

   "who's active right now?"
     → ls mail/sessions/active/
     → returns: s-01HXY8K3M2, s-01HXY8KAQB

   "what was completed recently?"
     → ls -t mail/sessions/archived/ | head

   "what's the brief for session X?"
     → cat mail/sessions/{active|archived|stale}/{X}/start.json

   "what walked away without ending properly?"
     → ls mail/sessions/stale/

   "everything that happened today across all sessions"
     → walk sessions/*/events/*.json across all three folders, filter .at
```

## 11.1 Session-as-Unit-of-Work State Machine

A session moves through these states. The transitions are explicit actions, never automatic.

```
                  ┌─────────────────────────────┐
                  │     Session does not yet    │
                  │           exist             │
                  └───────────────┬─────────────┘
                                  │
                                  │ Agent receives a brief.
                                  │ Generates session-id.
                                  │ Writes start.json.
                                  │ (single commit)
                                  ▼
                  ┌─────────────────────────────┐
                  │     ACTIVE                  │
                  │ mail/sessions/active/{id}/  │◄────┐
                  │                             │     │ Each check-in:
                  │ - mutable                   │     │  pull, deliver,
                  │ - has start.json            │ ────┘  process, send,
                  │ - no end.json               │        push, log events.
                  │ - events/ append-only       │        Many iterations
                  │                             │        possible.
                  └─────────────┬───────────────┘
                                │
                                │
            ┌───────────────────┴──────────────────┐
            │                                      │
            │ Agent decides work                   │ Some other actor
            │ is complete.                         │ (human, peer, GC)
            │ Writes end.json.                     │ decides session is
            │ Moves folder.                        │ stale.
            │                                      │ Writes end.json
            │                                      │ with end_status: stale,
            │                                      │ ended_by: ...
            │                                      │ Moves folder.
            ▼                                      ▼
┌─────────────────────────────┐         ┌─────────────────────────────┐
│        ARCHIVED             │         │           STALE             │
│  mail/sessions/archived/{id}/│         │   mail/sessions/stale/{id}/ │
│                             │         │                             │
│ - IMMUTABLE                 │         │ - IMMUTABLE                 │
│ - has start.json            │         │ - has start.json            │
│ - has end.json              │         │ - has end.json              │
│ - end_status:               │         │ - end_status: "stale"       │
│     delivered |             │         │ - ended_by: {actor}         │
│     incomplete |            │         │ - stale_threshold_used: ... │
│     abandoned |             │         │                             │
│     handed-off              │         │                             │
└─────────────────────────────┘         └─────────────────────────────┘
            │                                      │
            └──────────────┬───────────────────────┘
                           │
                           │ Agent wants to keep working on
                           │ the topic? START A NEW SESSION
                           │ with parent_session pointing here.
                           │ DO NOT modify this folder.
                           ▼
                ┌─────────────────────────────┐
                │  New ACTIVE session,        │
                │  parent_session = old.id    │
                └─────────────────────────────┘


   Hard rules:

   1. ARCHIVED ↛ ACTIVE.   Once ended, no resurrection.
   2. STALE ↛ ACTIVE.      Same.
   3. STALE ↛ ARCHIVED.    Once labelled stale, stays stale.
   4. ACTIVE → ARCHIVED:   the agent themselves makes the move.
   5. ACTIVE → STALE:      someone OTHER than the agent makes the move.

   Continuation = new session. Always. The old one is a sealed audit artifact.
```

## 14. The Fractal Property

Vaults nest. Each vault's `mail/` is independent.

```
   {parent-vault}/
   ├── mail/                              ← THIS VAULT's Email-FS
   │   ├── sessions/
   │   ├── mailroom/
   │   ├── conductor.claude-ai.s-AAA/
   │   └── architect.claude-ai.s-BBB/
   │
   └── sub-rooms/
       ├── due-diligence/                 ← a sub-vault
       │   ├── mail/                      ← INDEPENDENT Email-FS
       │   │   ├── sessions/
       │   │   ├── mailroom/
       │   │   ├── lawyer.claude-code.s-CCC/
       │   │   └── analyst.claude-ai.s-DDD/
       │   └── docs/
       │
       └── product-launch/                ← another sub-vault
           ├── mail/                      ← INDEPENDENT Email-FS
           │   └── ...
           └── briefs/

   Crossing the boundary requires an EXPLICIT FORWARD (not in v0.6).
   Email-FS in the parent vault has no awareness of Email-FS in sub-vaults.
   They are completely independent namespaces.

   Same agent name in two sub-vaults = two different mailboxes.
   (analyst.claude-ai.s-DDD in due-diligence ≠ analyst.claude-ai.s-DDD
    in product-launch — vault context disambiguates.)
```

## 15. Single-Writer Principle (The Core Invariant)

What makes the whole system safe under concurrent sgit branches.

```
   ┌───────────────────────────────────────────────────────────────┐
   │                   STRICTLY SINGLE-WRITER                      │
   │                                                               │
   │   mail/{agent}/inbox/                                         │
   │   mail/{agent}/processed/                                     │
   │   mail/{agent}/purged/                                        │
   │   mail/{agent}/outbox/   (and all subfolders, including self/)│
   │   mail/{agent}/state/                                         │
   │   mail/sessions/{my-session}/                                 │
   │                                                               │
   │   ───► Only the named agent ever writes here.                 │
   │   ───► Last-write-wins is moot: there's only one writer.      │
   │   ───► Merge conflicts physically impossible.                 │
   └───────────────────────────────────────────────────────────────┘

   ┌───────────────────────────────────────────────────────────────┐
   │           PRODUCER-CONSUMER (DISJOINT BY FILENAME)            │
   │                                                               │
   │   mail/mailroom/{recipient}/                                  │
   │                                                               │
   │   ───► Senders ONLY CREATE files (with globally-unique ULIDs) │
   │   ───► Recipient ONLY DELETES files (via mv)                  │
   │   ───► No filename ever has two writers.                      │
   │   ───► Sgit content-addressed merge handles cleanly.          │
   └───────────────────────────────────────────────────────────────┘

                               That's it.
                There are no other write patterns in Email-FS.
```

## 17. One-Slide Summary

If a vault-viewer dev reads only one section, this is it.

```
   ┌─────────────────────────────────────────────────────────────┐
   │                                                             │
   │           EMAIL-FS in 30 seconds                            │
   │                                                             │
   │   LAYOUT                                                    │
   │   • mail/{agent}/             per-agent mailbox             │
   │   • mail/mailroom/{me}/       transit zone                  │
   │   • mail/sessions/active/     in-flight work (mutable)      │
   │   • mail/sessions/archived/   completed work (immutable)    │
   │   • mail/sessions/stale/      abandoned work (immutable)    │
   │                                                             │
   │   FORMAT                                                    │
   │   • .eml = RFC 2822, IMMUTABLE                              │
   │   • Sidecar = recipient's workspace folder per message      │
   │   • ULID filenames (globally unique, time-sortable)         │
   │                                                             │
   │   PRIMITIVES                                                │
   │   • SEND     → write to mailroom + own outbox               │
   │   • DELIVER  → mv mailroom → my inbox + write sidecar       │
   │   • READ     → cat + sidecar update                         │
   │   • PROCESS  → mv inbox → processed/ (only when work done)  │
   │   • PURGE    → tombstone + delete                           │
   │                                                             │
   │   SESSION LIFECYCLE                                         │
   │   • Session = unit of work, bounded by start.json/end.json  │
   │   • Many check-ins per session (pull/process/push cycles)   │
   │   • Active → Archived: agent ends gracefully                │
   │   • Active → Stale:    external actor declares abandoned    │
   │   • Once ended, IMMUTABLE. Continuation = new session.      │
   │                                                             │
   │   PRINCIPLES                                                │
   │   • Subjects first-class — directory listing IS audit log   │
   │   • Vault assets referenced by path, not MIME-attached      │
   │   • Agents email themselves (reasoning logs) on decisions   │
   │   • PROCESS = work done, NOT "I replied"                    │
   │                                                             │
   │   Built on sgit. Compatible with real SMTP. Provenance      │
   │   and explainability for agentic systems.                   │
   │                                                             │
   └─────────────────────────────────────────────────────────────┘
```
