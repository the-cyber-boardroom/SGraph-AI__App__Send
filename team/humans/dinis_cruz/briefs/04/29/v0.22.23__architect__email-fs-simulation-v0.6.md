# Email-FS Simulation — v0.6

**Document type:** Simulation / Worked Example
**Spec version:** 0.6 (companion to *Email-FS Specification v0.6*, *Email-FS Architecture v0.6*, and *Email-FS Skill v0.6*)
**Date:** 29 April 2026
**Author:** Architect (Claude Opus, claude.ai session, s-01HXY8K3M2)

This document walks through complete, realistic Email-FS scenarios from start to finish, with the vault state shown at every step. Every operation in this document was **actually executed** against a scratch vault; the file contents and commit log shown are not invented examples, they are the real output of running the protocol by hand.

The point is two-fold: (a) it gives agents a worked reference for what every step looks like in practice, and (b) it stress-tests the spec against real-shaped problems to surface anything that would only show up in execution.

**v0.5 → v0.6 changes:** The original three-agent banner-launch scenario (sections below) is preserved as the canonical short-cycle simulation, but its file paths reference `mail/sessions/{id}/` — the v0.5 flat layout. Conceptually it remains correct. A **new §14 — Session Lifecycle Walkthrough** has been added at the end demonstrating the v0.6 three-state session lifecycle (active/archived/stale) with real captured filesystem state.

When reading the original sections, mentally substitute `mail/sessions/active/{id}/` for `mail/sessions/{id}/`, and `mail/sessions/archived/{id}/` for "the same folder after end.json was written." The protocol is otherwise identical.

## The Scenario

**Vault:** `cb-launch-vault` — a Cyber Boardroom team's working vault.

**Brief from human.dinis:** add a GDPR-compliant privacy banner to the CB landing page before Friday's launch.

**Cast:**

| Agent | Role | Brought in by | When |
|---|---|---|---|
| `human.dinis` | Human driver | — | Origin |
| `pm.claude-ai.s-VE3Z73KV3A` | Project manager — picks up brief, dispatches, coordinates | dinis | T0 |
| `dev.claude-code-web.s-39N5F1V95F` | Dev — makes the code change | pm | T0 + 30min |
| `reviewer.claude-ai.s-49F4VXR4D7` | Reviewer — writes/approves GDPR copy | pm | T0 + 75min (mid-flow) |

**What we'll see exercised:**
- SEND, DELIVER, READ, PROCESS (all five primitives except PURGE — which has no natural place in a same-week scenario; covered in §13)
- Reply with `In-Reply-To` and `References`
- Multi-recipient pattern (pm sends to two different agents)
- Mid-flow agent spawning (reviewer brought in after work has started)
- Reasoning logs (pm decides to bring in reviewer; dev decides to use placeholder copy)
- Session events (pm logs a milestone)
- Sidecar extension files (dev writes `pr.json`, reviewer writes `review.json`)
- Mailroom-as-delivery-receipt (sender sees their pigeonhole emptied)
- Subject-line audit log (the directory listing tells the story)

## Step 2 — pm SENDs the task to dev

pm composes the brief and sends to dev. Per the spec, this means writing the eml in **two places** — the recipient's mailroom (transit) and pm's own outbox (permanent record).

**pm's actions:**

```bash
DEV=dev.claude-code-web.s-39N5F1V95F
MSG_1=01KQCQBE52ZVGZEG5BKQE1VPE1
SLUG=add-privacy-banner

mkdir -p mail/mailroom/${DEV}
mkdir -p mail/${PM}/outbox/${DEV}

cat > mail/mailroom/${DEV}/${MSG_1}__${PM}__${SLUG}.eml <<'EOF'
From: pm.claude-ai.s-VE3Z73KV3A <pm.claude-ai.s-VE3Z73KV3A@vault.sgraph.ai>
To: dev.claude-code-web.s-39N5F1V95F <dev.claude-code-web.s-39N5F1V95F@vault.sgraph.ai>
Date: Tue, 28 Apr 2026 14:05:00 +0000
Subject: Add GDPR-compliant privacy banner to landing page
Message-ID: <01KQCQBE52ZVGZEG5BKQE1VPE1@vault.sgraph.ai>
X-EmailFS-Schema-Version: 0.5
X-EmailFS-Vault-ID: cb-launch-vault
X-EmailFS-Sender-Session: s-VE3Z73KV3A
X-EmailFS-Kind: task
X-EmailFS-Priority: high
X-EmailFS-Asset-Ref: site/pages/landing.html, site/components/banner.html
Content-Type: text/plain; charset=utf-8

Brief from human.dinis: add a GDPR-compliant cookie/privacy banner to the
Cyber Boardroom landing page before Friday launch.

Files to update (already in vault):
  site/pages/landing.html
  site/components/banner.html  (new)

Requirements:
  - Three options: Accept all / Reject all / Customise
  - First-party cookie, 12-month expiry
  - No tracking scripts load until choice made
  - GDPR-compliant copy - leave placeholder + flag if uncertain

Reply when PR opened.
--
pm.claude-ai.s-VE3Z73KV3A
EOF

cp mail/mailroom/${DEV}/${MSG_1}__${PM}__${SLUG}.eml \
   mail/${PM}/outbox/${DEV}/${MSG_1}__${PM}__${SLUG}.eml

# pm also logs a session event — milestone-level record
cat > mail/sessions/${PM_SID}/events/01KQCQEFT3PHKHA0TRFWD4HP9T__dispatched-banner-task.json <<'EOF'
{
  "schema_version": "0.5",
  "event_id": "01KQCQEFT3PHKHA0TRFWD4HP9T",
  "session_id": "s-VE3Z73KV3A",
  "at": "2026-04-28T14:05:30Z",
  "kind": "milestone",
  "summary": "Dispatched privacy-banner task to dev",
  "refs": { "messages": ["01KQCQBE52ZVGZEG5BKQE1VPE1"], "issues": [], "vault_paths": [] }
}
EOF

sgit commit "send: 01KQCQBE -> dev - add-privacy-banner"
sgit push
```

**Notice three things:**
1. **`X-EmailFS-Asset-Ref` lists the vault files**, body refers to them by path. pm did not MIME-attach the files — they live in the vault, the path is enough.
2. **The eml is in two places** — `mail/mailroom/{dev}/...` (in transit) and `mail/{pm}/outbox/{dev}/...` (pm's record).
3. **pm logged a session event** — coarse-grained "milestone" record. Not every send needs an event; this one does because it's the start of a workstream.

**Vault state after step 2:**

```

├── mailroom/
│   └── dev.claude-code-web.s-39N5F1V95F/
│       └── 01KQCQBE...__pm.claude-ai.s-VE3Z73KV3A__add-privacy-banner.eml   ← in transit
├── pm.claude-ai.s-VE3Z73KV3A/
│   ├── outbox/
│   │   ├── dev.claude-code-web.s-39N5F1V95F/

│   │   └── self/

│   └── state/
└── sessions/
```

```bash
  "message_id": "01KQCQBE52ZVGZEG5BKQE1VPE1",
  "status": "unread",
  "delivered_at": "2026-04-28T14:30:15Z",
  "read_at": null,
  "processed_at": null,
  "purged_at": null,
  "thread_root": "01KQCQBE52ZVGZEG5BKQE1VPE1",
  "processing_notes": null

sgit push
```

**Notice:** dev batched two logical operations into one commit (session-start AND deliver). ## Step 4 — dev READs and writes a reasoning log

dev reads the brief, decides on an approach, and **records the decision as a reasoning log**.

**dev's actions:**

```bash
cat mail/${DEV}/inbox/01KQCQBE*.eml
# (reads the brief — sees the "leave placeholder, flag for review" instruction)

# Update sidecar — status read, timestamp
# (in real life this is JSON edit; shown as Python here for clarity)
python3 -c "
import json, datetime
p = 'mail/${DEV}/state/01KQCQBE52ZVGZEG5BKQE1VPE1/core.json'
d = json.load(open(p))
d['status'] = 'read'
d['read_at'] = '2026-04-28T14:33:00Z'
json.dump(d, open(p, 'w'), indent=2)
"

# Reasoning log — meaningful decision worth recording
REASON=01KQCR3VD3WD561Q3RZ0X9CQ7B
mkdir -p mail/${DEV}/outbox/self
cat > mail/${DEV}/outbox/self/${REASON}__${DEV}__placeholder-legal-copy.eml <<EOF
From: dev.claude-code-web.s-39N5F1V95F <...>
To: dev.claude-code-web.s-39N5F1V95F <...>
Date: Tue, 28 Apr 2026 14:45:00 +0000
Subject: Decision - using placeholder for GDPR copy, flagging for review
Message-ID: <01KQCR3VD3WD561Q3RZ0X9CQ7B@vault.sgraph.ai>
X-EmailFS-Schema-Version: 0.5
X-EmailFS-Kind: reasoning-log
Content-Type: text/plain; charset=utf-8

Decision: Build the banner with placeholder legal copy and explicitly flag
in the PR + reply that the copy needs review by a domain expert.

Reasoning: Brief said "when in doubt, leave a placeholder and flag it".
Better to ship the mechanism with placeholder text now than to either
(a) block on copy or (b) ship final-looking copy that quietly turns
out to be non-compliant.

Going to use the literal string "[GDPR-COPY-PLACEHOLDER]" so it can't
be missed in review.
EOF

sgit commit "read: 01KQCQBE + reasoning: 01KQCR3V"
sgit push
```

**Notice:** the reasoning log went straight to `outbox/self/`. **No mailroom step, no DELIVER step** — addressed to self, stored once.

dev writes the code. (The actual code change to `site/components/banner.html` and `site/pages/landing.html` happens in the wider repo — not shown here. The Email-FS perspective is: dev has done the work and is now archiving the message and recording what was done.)

**dev's actions:**

```bash
# Move the original task from inbox to processed
mv mail/${DEV}/inbox/01KQCQBE*.eml mail/${DEV}/processed/

# Update core.json — status = processed, with notes
python3 -c "
import json

d = json.load(open(p))
d['status'] = 'processed'
d['processed_at'] = '2026-04-28T15:08:00Z'

cat > mail/${DEV}/state/01KQCQBE52ZVGZEG5BKQE1VPE1/pr.json <<'EOF'

  "pr_number": 142,

# SEND the reply (a SEND with In-Reply-To)

Date: Tue, 28 Apr 2026 15:10:00 +0000

Message-ID: <${MSG_2}@vault.sgraph.ai>
In-Reply-To: <01KQCQBE52ZVGZEG5BKQE1VPE1@vault.sgraph.ai>

PR #142 opened: claude/privacy-banner-28apr

Mechanism done. ONE THING NEEDS YOUR ATTENTION:
  Legal copy is placeholder. Visible string is literally
  '[GDPR-COPY-PLACEHOLDER]' so it can't ship by mistake.
  Need a reviewer to write the real copy.
  Reasoning log: mail/${DEV}/outbox/self/01KQCR3V...placeholder-legal-copy.eml

CI passing. Ready for review of mechanism. Copy review blocking ship.
--
${DEV}
EOF

cp mail/mailroom/${PM}/${MSG_2}__${DEV}__${SLUG}.eml \
   mail/${DEV}/outbox/${PM}/${MSG_2}__${DEV}__${SLUG}.eml

sgit commit "process: 01KQCQBE + send reply 01KQCQMK"
sgit push
```

**Notice:**
- **One commit** for "process the task + write extension files + send reply." This is the right granularity — these are one logical action ("I'm done with this task and reporting back").
- The reply has **`In-Reply-To`** (the parent's Message-ID) and **`References`** (the full chain — for a direct reply, this equals In-Reply-To).
- The **subject is "Re: ..."** — convention. Subject-line audit log will read "Add privacy banner → Re: Add privacy banner" — the conversation shape is visible from `ls`.
- The reply **explicitly references the reasoning log** by path. This is how an agent invites readers to dig deeper without bloating the reply.

```bash
  "delivered_at": "2026-04-28T15:14:00Z",
  "read_at": "2026-04-28T15:14:15Z",
```

**Notice the `thread_root`** — when pm wrote the sidecar, it set `thread_root` to the ULID at the *root* of the thread (the original task message), not to the immediate parent. This is computed from the eml's `References` header (first entry). Future agents pulling threads use this to group conversations.

## Step 7 — pm writes a reasoning log: bring in a reviewer

pm has now read dev's reply. The PR is up but the legal copy is blocked. pm has to decide: wait for dinis (slow), have dev write the copy (dev explicitly asked not to), or spawn a reviewer agent. **This is a meaningful decision — write a reasoning log.**

**pm's actions:**

```bash
mkdir -p mail/${PM}/outbox/self

cat > mail/${PM}/outbox/self/01KQCR3VD3WD561Q3RZ0X9CQ7F__${PM}__bring-in-reviewer.eml <<EOF
From: pm.claude-ai.s-VE3Z73KV3A <...>
To: pm.claude-ai.s-VE3Z73KV3A <...>
Date: Tue, 28 Apr 2026 15:15:00 +0000
Subject: Decision - bring in reviewer.claude-ai for GDPR copy review
Message-ID: <01KQCR3VD3WD561Q3RZ0X9CQ7F@vault.sgraph.ai>
X-EmailFS-Schema-Version: 0.5
X-EmailFS-Kind: reasoning-log
Content-Type: text/plain; charset=utf-8

Decision: Spawn a reviewer session to write/approve the GDPR copy.

Reasoning: dev did the right thing flagging it. I could escalate to
human.dinis but he's busy until evening, and the launch is Friday.
A reviewer agent with the GDPR brief context can produce compliant
copy now and we lose no time.
```

## What This Simulation Surfaced

Things observed during execution that should feed back to the spec for v0.6:

1. **The "session-start + deliver" pattern is canonical, should be blessed in spec.** Both `dev` (step 3) and `reviewer` (step 9) opened a session AND delivered messages in the same commit. That is the natural shape of "an agent comes online to do a thing." Spec should explicitly call this out as the canonical first commit of any session that's responding to existing mail.

2. **Subject convention emerged as: imperative for tasks, "Re: ..." for replies, "Decision - ..." for reasoning logs.** Held throughout the simulation, made the directory listings extremely readable. Worth adding to SKILL.md as recommended convention.

3. **`X-EmailFS-Asset-Ref` header proved its value.** Three messages used it; in each case the body referenced the path and the header made the asset list machine-discoverable. No MIME attachment needed — the vault is the canonical store, exactly as spec §4.5 says.

4. **Sidecar extension files exceeded expectations as a pattern.** dev's `pr.json` and reviewer's `review.json` carry domain-specific metadata that would have been awkward in either the eml body or `core.json`. The "open extension schema" decision (D18 in v0.5) was the right call.

5. **The thread reconstructs perfectly via `References`** — the chain `01KQCQBE → 01KQCQMK → 01KQCRS7 → 01KQCS2B` (reviewer side branch) and `01KQCQBE → 01KQCQMK → 01KQCSBG → 01KQCSXV` (main pm-dev branch) is unambiguous from headers alone. Two branches share a common ancestor; thread reconstruction code can render this as a Y-shape.

6. **Multi-day note:** in this simulation everything happens in one day. PURGE was not exercised because nothing aged out. A v0.6 simulation could cover a 3-month-later PURGE pass to validate the tombstone mechanism.

7. **One real ergonomic friction:** updating `core.json` in place is the most awkward step in every PROCESS / READ. In manual operation it requires either a Python one-liner or a small JSON-edit dance. **The eventual `email-fs` CLI will hide this** — but it's worth noting that in the manual era, agents will be writing a lot of inline Python to update sidecars, and SKILL.md should suggest a small helper recipe for it.

## Closing Observations

**The protocol holds.** Three agents, six messages, two reasoning logs, one event, two extension files, twelve commits — all coordinated through nothing but file moves and `sgit`. No broker. No server. No race. No ambiguity about who said what.

**The friction is in the manual JSON editing.** This is the single biggest case for the eventual CLI. Once `email-fs send` and `email-fs deliver` exist, the protocol becomes invisible — agents call commands, the CLI manages all the file dance. But the protocol *could* be — and just was — operated entirely by hand.

**The audit log is free.** `sgit log --oneline` reads as the story. `ls processed/` reads as the timeline. `cat outbox/self/*` reads as the reasoning. Nothing was instrumented, nothing was logged separately — these all fall out of the file layout.

**The simulation didn't surface a single architectural problem.** Some ergonomic feedback (item 7), a couple of conventions to bless in spec (items 1, 2). No "ah, this whole approach won't work" moment. That's the validation this exercise was meant to produce.

### What This Demonstrates About the v0.6 Session Model

| Property | How it showed up |
|---|---|
| **Session = unit of work, not unit of runtime** | Episode 3: the runtime stopped overnight; the session continued, same session-id |
| **Three-state lifecycle** | All three states present in the final vault — `ls active/`, `ls archived/`, `ls stale/` answer different questions |
| **Self-end vs external-end discriminator** | Episode 1's `end.json` has `end_status: delivered`, no `ended_by`. Episode 2's has `end_status: stale`, `ended_by: human.dinis`. Different actors, different shapes. |
| **Immutability after end** | Once Librarian's session moved to `archived/`, neither Librarian nor anyone else writes to it. The `events/` folder is sealed. |
| **Mailbox stays where it is on session-end** | Librarian's `mail/librarian.../outbox/` did not move when the session was archived. Mailbox is per-agent, lifecycle-independent. |
| **First reasoning log = brief in own voice** | Each episode's first artifact in `outbox/self/` is a reasoning log restating the brief. This is the agent's "memory anchor" for resumption. |
| **`kind: check-in-end` as breadcrumb** | Episode 3's day-1 wrap was a single event whose `summary` was sufficient to orient day 2's runtime. No external coordination needed. |
| **Same session-id across resumptions** | Episode 3 used s-V0RESUMABLE on both days. The session-id is logical, not physical. |

### Discoveries from This Walkthrough

1. **The mailbox doesn't move with the session.** Worth being explicit in the Specification — `mail/{agent-name}/` and `mail/sessions/{state}/{session-id}/` have *independent* lifecycles. The agent's mailbox folder remains in place even after the session is archived or stale. (The archived session-id appears in the agent name as a permanent label, but the mailbox's contents are not touched by the session-end.)

2. **`ended_by` is a quiet but important field.** It's the only thing distinguishing self-declared from externally-declared session ends. If signing is later added, this is the field whose signer must match `ended_by`.

3. **The `kind: check-in-end` event with a summary is the most important breadcrumb.** Without it, a resuming runtime has to walk the entire reasoning-log trail to figure out where it is. With it, one read tells the story.

4. **Stale sweep is genuinely use-case-specific.** In Episode 2 the threshold was 14 days. For an orchestration scenario it could be 1 hour. Spec correctness here is the *protocol* (who writes what), not the *threshold* (when to apply it). v0.6 gets this right.
