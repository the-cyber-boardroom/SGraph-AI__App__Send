# Investigation: S3 "DNS cache overflow" Error During sgit Share

**Date:** 2026-04-16
**Environment:** Claude Web (claude.ai) container
**sgit version:** v0.10.9
**Vault size:** ~38MB, 56 files
**Status:** Unresolved — needs server-side investigation

---

## Summary

`sgit share` consistently fails with an HTTP 503 response containing the body `DNS cache overflow` when uploading the encrypted vault snapshot to S3 via presigned multipart upload. `sgit push` (which uploads individual small objects) works fine. The error is not from sgit, not from the SG/Send API server, and not a standard AWS S3 error.

---

## Reproduction

```bash
cd /home/claude/vault/33n-framework-analysis   # ~38MB vault, 56 files
sgit --token graphs-and-maps --base-url https://dev.send.sgraph.ai share
```

**Result:** Fails every time. Tested 5+ attempts with 3-15 second delays between retries. All produce the same error.

---

## Error Details

### What sgit reports

```
error: API Error: HTTP 503 Service Unavailable
  Request:  PUT https://745506449035--sgraph-send-transfers--eu-west-2.s3.amazonaws.com/
            transfers/<transfer-id>/payload?uploadId=<multipart-id>&partNumber=1&...
  Headers:  {}
  Payload:  5242880 bytes
  Response: DNS cache overflow
```

### Key observations

| Attribute | Value |
|-----------|-------|
| HTTP method | PUT |
| Target | S3 presigned URL (multipart upload, part 1) |
| Payload size | 5,242,880 bytes (5MB — single part of multipart) |
| HTTP status | 503 |
| Response body | `DNS cache overflow` (literal string) |
| S3 bucket host | `745506449035--sgraph-send-transfers--eu-west-2.s3.amazonaws.com` |
| AWS region | eu-west-2 |
| Presigned URL auth | AWS4-HMAC-SHA256 with temporary security token |

## What Works vs What Fails

| Operation | Mechanism | Result |
|-----------|-----------|--------|
| `sgit push` | Individual object uploads (delta, small payloads) | ✅ Works |
| `sgit push` (batch) | Batch upload of multiple objects | ⚠️ Fails (SSL EOF), falls back to individual |
| `sgit share` | Single large blob via S3 multipart presigned URL | ❌ Fails consistently (DNS cache overflow) |
| `curl` GET to S3 host | DNS + TLS connectivity test | ✅ Works (403 expected) |
| `curl` PUT to S3 host | Small unsigned payload test | ✅ Gets 501 (expected — unsigned) |

## Diagnostic Evidence

### 1. S3 domain resolves and connects fine

```bash
$ curl -s -o /dev/null -w "HTTP %{http_code} | DNS: %{time_namelookup}s" \
    https://745506449035--sgraph-send-transfers--eu-west-2.s3.amazonaws.com/

HTTP 403 | DNS: 0.015709s | Connect: 0.016102s | Total: 0.638879s
```

DNS resolves to multiple IPs (3.5.246.26, 3.5.244.50, etc.), TLS 1.3 handshake completes, S3 responds with a normal 403 for unsigned requests.

### 2. No proxy environment variables set

```bash
$ echo "HTTPS_PROXY: $HTTPS_PROXY"
HTTPS_PROXY:

$ echo "HTTP_PROXY: $HTTP_PROXY"
HTTP_PROXY:

$ python3 -c "import os; print(os.environ.get('HTTPS_PROXY', 'none'))"
none
```

### 3. The push batch upload also fails (different error, same infrastructure)

```
Pushing to default...
  ▸ Re-syncing vault structure to server
Warning: batch upload failed (<urlopen error EOF occurred in violation of protocol (_ssl.c:2406)>),
  falling back to individual uploads
Vault structure re-synced to server.
```

The batch upload gets an SSL EOF error but gracefully falls back to individual uploads. The share command has no such fallback.

### 4. The response body is non-standard

"DNS cache overflow" is NOT a standard AWS S3 error code. Standard S3 503 errors return XML bodies like:

```xml
<Error>
  <Code>SlowDown</Code>
  <Message>Please reduce your request rate.</Message>
</Error>
```

or `ServiceUnavailable` / `InternalError`. The plain-text `DNS cache overflow` response suggests the 503 is being generated by an **intermediary** between the Claude environment and S3, not by S3 itself.

---

## Hypothesis

### Most likely: Egress proxy or network layer in the Claude environment

The Claude container environment has network egress controls (see `network_configuration` in the system prompt). The allowed domains list includes `745506449035--sgraph-send-transfers--eu-west-2.s3.amazonaws.com`. However:

1. The egress proxy may handle small requests (individual object PUTs during `sgit push`) differently from large multipart uploads during `sgit share`
2. The proxy may have a DNS cache with a connection or entry limit that gets exhausted during multipart upload session setup
3. The presigned URL is extremely long (~2KB+ with the security token) — the proxy may have URL length limits that trigger the error
4. The `x-deny-reason` header that the proxy can return might contain more information, but sgit may not be logging it

### Less likely: AWS-side issue

S3 presigned URLs are just S3 with authentication in the URL. If the URL is valid (which it is — the API server generates it), S3 should accept the upload. The non-standard error body strongly suggests the response is not from S3.

### Less likely: sgit bug

sgit is using Python's `urllib` for the upload. The SSL EOF error during batch upload and the DNS cache overflow during share may share a root cause (the intermediary network layer), but sgit's HTTP handling is standard.

---

## Suggested Investigation Steps

1. **Check egress proxy logs** — look for requests to `745506449035--sgraph-send-transfers--eu-west-2.s3.amazonaws.com` and see if the proxy is generating the 503 response or passing it through

2. **Check the `x-deny-reason` header** — the Claude environment documentation mentions the egress proxy returns this header on failures. sgit should log this if present.

3. **Test with a smaller vault** — create a vault with only 1-2 small files and try `sgit share`. If it works, the issue is payload-size-related. If it fails, the issue is with the S3 multipart flow itself.

4. **Test the presigned URL directly** — take the presigned URL from the error output and try the PUT with `curl` directly from the same environment:
   ```bash
   dd if=/dev/zero bs=5242880 count=1 | curl -X PUT -T - "<presigned-url>"
   ```

5. **Check if sgit share can use the same delta-upload path as push** — since push works via individual object uploads, could share use the same mechanism instead of a single-blob S3 multipart?

6. **Check the SG/Send transfer creation API** — the share flow starts with `POST https://dev.send.sgraph.ai/api/transfers/create` which returns the presigned URL. Verify the presigned URL parameters (expiry, content-length, part size) are correct.

---

## Workaround

The vault is fully pushed via `sgit push` and accessible via the vault key:

```
sgit clone rsabx8v4p50dm2l1zwosibwf:t63y7gee
```

The share command (which creates a separate read-only snapshot for distribution) is the only operation that fails. All vault read/write operations via push/pull work correctly.

An earlier share token (`file-hour-6653`) was created successfully when the vault was smaller (~38KB at initial commit). This further supports the hypothesis that the issue is payload-size-related.
