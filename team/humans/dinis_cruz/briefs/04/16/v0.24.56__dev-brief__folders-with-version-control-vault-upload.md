# Folders with Version Control: Vault-Backed Upload Mode in Main SG/Send UI

**version** v0.24.56
**date** 16 Apr 2026
**from** Human (project lead)
**to** Developer (lead), Designer, Architect
**type** Dev brief

---

## Context

Yesterday's brief (`v0.24.55__dev-brief__vault-as-collaboration-primitive.md`) established that the vault UI, with tree-only loading and immutable file caching, already solves the multi-user collaboration workflow that folder Modes 2-4 were designed to address. The server-side zip versioning sprint was deprioritised.

This brief makes that concrete: add vault as an upload option in the main SG/Send UI, alongside the existing folder drop, and expose it as a beta feature. The killer framing is "folders with version control": the user drops a folder, gets an instantly shareable link, and can keep adding files while the recipient is already browsing.

## The Shift: Passive Upload to Active Workflow

The current folder upload is passive. The user drops a folder, waits for the full zip to upload, then gets a share link. Nothing is visible until it is all done.

The vault-backed upload inverts this. The moment the user drops a folder, SG/Send creates a vault and returns a share link immediately. Files are committed to the vault in batches (10-20 at a time, or by file count) as they upload. The recipient can open the link while the upload is still in progress and see files appearing in real time.

This is what "active" means here: the upload is itself a usable, shareable workflow, not a precondition for one.

## Workflow

```
User drops folder
    |
    v
Vault created immediately
    |
    v
Share link returned (copyable, openable NOW)
    |
    +--> Files uploaded in batches, committed to vault
    |       (10-20 files per commit, or batch by size)
    |
    +--> Optional: placeholder files marked as "uploading"
    |       so the other side has visual cues before content arrives
    |
    v
Recipient opens share link
    |
    +--> Sees files appearing as they are committed
    +--> Can refresh (auto or manual) to see latest state
    +--> Can download individual files as they become available
```

## Key Features to Implement

### 1. Instant Vault Creation on Drop

When a folder is dropped onto the upload zone, create a vault immediately. Return the vault's share link before the first file has finished uploading. The UI should emphasise this: "Your share link is ready" appears within a second, alongside a progress indicator for the ongoing upload.

### 2. Batched Commits

Files upload in batches. Each batch produces one commit. Batch size can be either by file count (10-20 files) or by total size (whichever threshold hits first). This keeps the commit log readable without flooding it, and ensures large files get their own commits rather than blocking smaller ones.

### 3. Upload Status File

Use the same convention pattern already established for galleries: a special file in the folder root (e.g. `.sg-upload-status.json` or similar) that contains metadata about the upload progress. Fields: total files, files uploaded, files remaining, total bytes, bytes uploaded, estimated time remaining, last update timestamp.

This file is updated as a commit itself (or as part of each batch commit). The recipient's UI can render this as a status banner at the top of the vault view: "Upload in progress: 47 of 112 files uploaded, approximately 3 minutes remaining."

### 4. Placeholder Files (New Convention)

Optional feature, inspired by `.gitkeep` and similar Git conventions: when a large file is queued for upload but not yet committed, write a small placeholder file in its place with a reserved naming convention (e.g. filename followed by a specific suffix, or a sidecar file). The recipient sees that something is coming without having the content yet.

This gives a visual cue on the receiving side before the content arrives. Especially useful for large files (videos, archives) where the upload might take several minutes.

Implementation note: the placeholder should be clearly distinguishable from real files in the UI (greyed out, different icon, "uploading..." label). When the real file lands, the placeholder is replaced in the same commit.

### 5. Keep the Browser Open Problem

The upload runs client-side, so the user must keep the browser tab open until complete. This is not ideal, but it is acceptable for the beta. Add clear UI indication: a persistent banner or progress bar that makes it obvious the upload is in progress, and a warning on tab close. For a future iteration, consider a service worker or small helper to continue the upload in the background.

## Two UX Paths (Keep Simple for Beta)

**Path 1: Upload adds files.** User drops a folder, new files can be dragged and dropped onto the same upload area to be added to the same vault. Simple continuation of the upload workflow.

**Path 2: User opens the vault directly.** Once the vault is created, the user can open the vault UI in a new tab and add files there. This is the power user mode.

For the beta, focus on Path 1 (extension of the upload workflow). Path 2 is technically already possible (the vault exists, the user has the key) and can be promoted later.

## The Invisible Vault

The most powerful framing here is that the user does not need to know they are using a vault. From their perspective, they dropped a folder, they got a share link, it works like SG/Send always has. Behind the scenes, it is a vault with full version control, incremental commits, and multi-user collaboration potential.

This is the "folders with version control" concept. The user gets all the benefits of a versioned, incremental, collaborative store without learning any new concepts. The vault is the engine. The folder is the experience.

## Beta Positioning

Ship this as a beta feature, toggle-visible in the UI. "Try our new vault-backed upload mode (beta)." This:

- Gives users something visibly new to notice
- Creates a clean feedback channel (beta users expect rough edges)
- Lets us gather usage signal before making it the default
- Aligns with the shipping-first principle from the OWASP presentation pack

The default upload remains the current folder-as-zip flow. The beta toggle switches to vault-backed mode.
