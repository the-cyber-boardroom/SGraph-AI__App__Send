## The Opportunity

OpenRouter is becoming the LLM billing and routing platform for our users. Before building custom infrastructure, maximise what OpenRouter already provides. The more we get from the platform, the less we implement ourselves.

Additionally, OpenRouter-specific tools (model comparison, usage visualisation, key management) are a traffic driver: anyone using OpenRouter benefits from better tooling, and they find us through the tools.

## Research First: What Does OpenRouter's API Provide?

Before building, fully map the OpenRouter API capabilities:

### User API (Per Key)

| Capability | What We Get | How We Use It |
|---|---|---|
| Usage stats per key | Tokens consumed, cost, model breakdown | User dashboard: "you have spent £2.30 of your £5" |
| Model listing | Available models, pricing, capabilities | Model Chooser component data source |
| Rate limits | Requests per minute, tokens per minute | Display to user, manage expectations |
| Generation history | Past requests (metadata, not content?) | Usage timeline, spending patterns |

**Critical question:** Does the generation history include prompt/response content, or only metadata (model, tokens, cost, timestamp)? If content is included, we cannot store it (zero-knowledge). If it is metadata only, we can safely display it and save to the user's vault.

### Admin API

| Capability | What We Get | How We Use It |
|---|---|---|
| Create API keys | Programmatic key provisioning | Onboarding: create key for each new user |
| Set credit limits | Per-key spending caps | £5 per user, top up as needed |
| Revoke keys | Disable a key | User offboarding, abuse prevention |
| View all keys | List provisioned keys and their usage | Admin dashboard |
| Usage across keys | Aggregate stats | Business metrics: total spend, revenue projection |
| Logs per key | Detailed request logs | Debugging, support |

Map every endpoint. Document what is available. Identify gaps (things we need to build ourselves vs things OpenRouter already handles).

## Two Modes: User and Admin

### User Mode: "My OpenRouter"

A tool (or section of tools) for users who have an OpenRouter token (either provisioned by us or their own):

**Usage Dashboard**
- Credits remaining (£5 balance minus usage)
- Spending chart over time (daily, weekly)
- Breakdown by model ("you spent £1.20 on Claude, £0.80 on Mistral")
- Breakdown by tool ("£1.50 on infographics, £0.50 on document analysis")
- Cost per generation (average, min, max)

**Model Explorer**
- All available models with pricing
- Filter by capability (text, vision, code, fast, cheap, best quality)
- Compare models side by side (the Model Chooser component, expanded)
- Try a model: one-shot test prompt to see quality before committing
- Personal rankings: mark favourites, add notes

**Key Management**
- View your key status (active, credits remaining)
- Usage history (timeline of requests, anonymised)
- Export usage data to vault

### Admin Mode: "OpenRouter Admin"

For the project lead (and eventually for enterprise customers managing their teams):

**Key Provisioning**
- Create new API key with credit limit
- View all provisioned keys (user, creation date, usage, remaining credits)
- Top up credits for a key
- Revoke a key
- Bulk operations (create 10 keys for a new cohort of users)

**Usage Analytics**
- Total spend across all keys
- Top users by consumption
- Model popularity (which models are users choosing)
- Tool popularity (which tools generate the most usage)
- Cost projection (at current burn rate, how long until £100 is depleted)

**Logs and Debugging**
- Per-key request log (timestamp, model, tokens, cost)
- Filter by date range, model, cost threshold
- Identify anomalies (a key suddenly consuming 10x normal)
- Support tool: when a user reports an issue, look up their key's recent activity

## Component Architecture

### Standalone Components (Reusable)

| Component | What It Does | Used In |
|---|---|---|
| `sg-openrouter-usage` | Displays usage stats for a key (chart, breakdown) | User dashboard, Admin per-key view |
| `sg-openrouter-models` | Lists and filters available models with pricing | Model Chooser, Model Explorer |
| `sg-openrouter-key-manager` | Create, view, top up, revoke keys | Admin tool |
| `sg-openrouter-logs` | Displays request logs with filtering | Admin tool, User history |
| `sg-openrouter-balance` | Simple badge showing credits remaining | Header bar, User page |

### Tools (Assemblies of Components)

| Tool | Components Used | Audience |
|---|---|---|
| OpenRouter User Dashboard | usage + models + balance + logs | All users with a key |
| OpenRouter Admin Console | key-manager + usage (all keys) + logs + models | Project lead / admin |
| Model Comparison Tool | models + one-shot test | Anyone (traffic driver, no key needed for browsing) |

### Integration with Existing Tools

Every LLM-powered tool (infographic generator, document analyser, chat, multi-agent) should show:

- The `sg-openrouter-balance` badge (credits remaining)
- Cost estimate before generation ("this will cost approximately £0.05")
- Actual cost after generation ("this cost £0.04")
- Model selection via `sg-openrouter-models`

## Marketing Angle: Tools for the OpenRouter Community

The Model Comparison Tool does not require an SG/Send account. Anyone can browse models, compare pricing, see example outputs. This drives traffic from the entire OpenRouter user base:

- "Best models for infographics" (comparison page with examples)
- "Cheapest models for document summarisation" (cost-per-task ranking)
- "Model quality evolution" (same prompt across model versions over time)
- "My model picks" (curated recommendations, shareable as a vault)

Publish these comparisons as content on the website and LinkedIn. Every comparison links to the tool. Every tool user sees SG/Send.

## Existing Admin UI

We already have an admin UI. Add the OpenRouter admin features there. Do not create a separate admin site. The existing admin page gains a new section: "OpenRouter Keys" with the key manager, usage analytics, and logs.
