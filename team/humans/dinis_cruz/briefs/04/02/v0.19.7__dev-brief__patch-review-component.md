# Patch Review Component: Visual Code Diffs from Vaults

**version** v0.19.7
**date** 31 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Architect, Designer
**type** Dev brief

---

## The Need

Agents push patches to repos and vaults. The human needs to review them. Currently, this means reading raw diff files or going to GitHub to see the changes. There is no easy, visual, offline-capable way to review patches as part of the daily workflow.

The `@agent` comment pattern from the offline mode brief is already working well. Now the review side needs matching tooling: a series of Web Components that visualise patches, diffs, and code changes, readable from vaults, usable offline.

## The Vault as the Patch Review Channel

Currently, agents push patches to the Git repo. Add a parallel step: agents also push the patch (plus supporting materials) to a review vault.

```
Agent commits to branch
  │
  ├── Push to Git repo (as today)
  │
  └── Push to review vault:
        ├── patch file (the git format-patch output)
        ├── manifest.json (commit hash, branch, timestamp, agent, description)
        ├── context/ (relevant source files, or just the affected lines)
        └── notes.md (agent's explanation of what changed and why)
```

The review vault accumulates patches from all agents. The human opens the vault in a review component and works through them.

## The Component: sg-patch-review

### Three Rendering Modes

**Mode 1: Patch Only (Minimal)**

Just the diff, rendered as a visual side-by-side or unified diff view. No source code access needed. Works offline. Works from any vault.

```
- old line of code
+ new line of code
  unchanged context line
- another removed line
+ replacement line
```

Syntax highlighted. Line numbers. Colour-coded (red for removed, green for added). This is the minimum viable review experience.

**Mode 2: Patch + Agent Context (Recommended)**

The agent includes a manifest with additional context:

```json
{
  "commit": "abc123",
  "branch": "claude/upload-refactor",
  "agent": "Developer",
  "timestamp": "2026-03-31T09:00:00Z",
  "description": "Refactored chunk upload to use retry decorator",
  "files_changed": [
    {
      "path": "src/upload-engine.js",
      "lines_before": [40, 80],
      "lines_after": [40, 75],
      "source_snippet_before": "... full function before ...",
      "source_snippet_after": "... full function after ..."
    }
  ]
}
```

The component renders:
- The agent's description of what changed
- The diff with surrounding context (from the included source snippets)
- The full affected functions (not just the changed lines)

This is the mode that makes reviews effective: you see not just what changed, but the function it changed in, the agent's reasoning, and the commit metadata.

**Mode 3: Patch + Live Source (Full)**

If the source code is accessible (public GitHub repo, or a proxy service), the component fetches the full file and overlays the diff on it. You can scroll through the entire file with the changes highlighted in context.

This may require:
- A proxy API that fetches files from GitHub (since the browser cannot directly access raw GitHub content cross-origin in all cases)
- Or a pre-fetch step where the source files are included in the vault

Start with Mode 1 and Mode 2. Mode 3 is an enhancement.

## The @agent Comment Integration

When the human adds `@agent` comments during offline code review:

```javascript
// @dev: this retry logic should use the shared decorator from utils
// @qa: add a test for the case where all retries fail
// @architect: this function is doing too much, consider splitting
```

The patch review component should:
1. Detect `@agent` comments in the diff
2. Highlight them visually (different colour, icon per agent)
3. When processing the review, extract the comments and route them (same as the Librarian code comment processing)

This means the review component is not just a viewer. It is a review tool: read the patch, add comments, submit the review. The comments go to the appropriate agents.

## Multi-Patch Review

A review session typically involves multiple patches from multiple agents. The component needs:

- **Patch list panel:** All patches in the vault, sorted by date/agent/branch
- **Patch detail panel:** The currently selected patch with diff view
- **Status tracking:** Mark patches as reviewed, approved, needs-changes, rejected
- **Batch operations:** Approve all from a trusted agent, flag all from a new branch

Use SG Layout for the multi-panel arrangement.

## Offline Review Workflow

The complete offline review cycle:

1. Before going offline: agents have been working, patches accumulate in the review vault
2. Clone the review vault locally
3. Open in the patch review component (browser-based, offline)
4. Review each patch: read the diff, read the agent's notes, add `@agent` comments
5. Mark patches as approved/needs-changes
6. When back online: push the review vault (with status updates and comments)
7. The Librarian processes the `@agent` comments and routes them
8. Approved patches get merged. Needs-changes patches go back to the agent.

This is the Dev-to-QA patch workflow brief applied to code review. The vault is the communication channel. The patches are the language. The review component is the UI.
