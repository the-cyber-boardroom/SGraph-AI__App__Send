# £5 OpenRouter Credits: User Vaults, Billing Validation, and the Revenue Experiment

**from** Human (project lead)

## The Experiment

Give every existing user (Early Access Programme + paying customers, approximately 40-50 users) £5 of OpenRouter credits. Then watch what happens.

The questions this answers:

- What happens when someone has free credits to create infographics, analyse documents, and generate text?
- What do they use the credits for?
- How fast do they burn through £5?
- **What happens when they run out?** Do they buy more?

That last question is the one that matters. If users spend their £5, enjoy the experience, and then buy more credits, we have a business. If they spend the £5 and leave, we need to improve the tools or the onboarding.

## OpenRouter as the Billing Solution

### The Hypothesis

Can OpenRouter serve as our billing and cost control solution? Instead of building our own token metering, credit management, and payment processing for LLM usage, use OpenRouter's existing infrastructure:

- OpenRouter handles API routing to multiple models
- OpenRouter tracks usage per API key
- OpenRouter handles billing
- We provision API keys for users with a credit balance
- We top up the balance as needed (or users buy their own credits)

### What We Need to Validate

1. Can we programmatically create API keys? (Via the OpenRouter API)
2. Can we set per-key credit limits? (So a £5 key has exactly £5 of usage)
3. Can we retrieve per-key usage stats? (To show the user their consumption)
4. Is the usage data confidential? (Does it contain prompt content, or just stats?)
5. Can we use a single master account (Cyber Boardroom email) to provision keys for all users?

If the usage data is stats-only (no prompt content), we can safely store it in the user's vault for them to review. This is important for the zero-knowledge model: we should never see what the user generated, only how much they consumed.

## Per-User Vaults

### The Feature

Each user gets a vault. The vault stores:

- Files they have created (infographics, documents, analysis outputs)
- Usage stats (OpenRouter consumption data)
- Settings and preferences
- History of generations (bundles from the infographic tool)

### The Security Model

The user creates their own vault. They choose (or are given) a simple token. Without this token, nobody (including SG/Send) can access the vault contents. The UI writes to the vault automatically (saving generated infographics, stats updates). The server does not need to read the vault.

```
User visits site
  → No vault yet → "Create your vault" button
  → User enters/generates a simple token
  → "Save this token! Without it, you cannot access your data."
  → Vault created, tied to user's access token
  → UI writes generated materials to vault automatically
```

### What the Server Stores

The server stores:

- The vault ID (so the UI knows which vault belongs to this user)
- The OpenRouter API key ID (not the key itself, just the ID for stats retrieval)
- The user's SG/Send access token (already stored)

The server does NOT store:

- The vault decryption key (only the user has this)
- The OpenRouter API key itself (only the user's browser uses it)
- Any generated content (only in the encrypted vault)

### The User Page

A new page on the website: "My Account" or "My Workspace"

- View your vault (files, infographics, documents)
- View your API key stats (credits remaining, usage history)
- Manage your SG/Send access token
- See files you have shared
- Download your data

## The Outreach

### The Message to Users

Replace the Amazon voucher offer with:

> "Thank you for being an early user of SG/Send. Here is £5 of credits to use our new AI tools: create infographics, analyse documents, generate summaries. Your credits work across all our tools. When you run out, you can buy more from your account page."

This is better than an Amazon voucher because:

- It keeps users in the platform (not sending them to Amazon)
- It validates the billing model (do they buy more?)
- It generates usage data (what do they use credits for?)
- It creates content (every infographic, every analysis is a potential showcase)

### Funding the Credits

50 users at £5 each = £250 total exposure. But not all will use their credits immediately. Start with £100 in the OpenRouter account. Top up as usage grows. Monitor the burn rate.

## Admin Interface

Add to the existing admin UI:

- Create new OpenRouter API key for a user (with credit limit)
- View all provisioned keys and their usage
- Top up credits for a user
- Revoke a key
- Export usage data (for accounting and analysis)

This should be a simple CRUD interface. The OpenRouter API handles the heavy lifting.
