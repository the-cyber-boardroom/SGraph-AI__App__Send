# The Library Website: Centralised Roles, Skills, and Team Definitions

**date** 02 Apr 2026
**from** Human (project lead)
**to** Librarian (lead), Designer, Developer, Conductor

## The Need

We have a real, immediate need for a central place where all role definitions, team structures, skills, and Claude guidance live. Currently, this content is scattered across the SG/Send repo (team/roles/, CLAUDE.md files, skill definitions). It needs to be:

- Accessible from any project (not locked inside one repo)
- Runnable offline (a static website from a local folder)
- Cloneable into every agent session
- The single source of truth for "how does this team work?"

## The Library Website

A static website at library.sgraph.ai (or similar). Same architecture as all other sites: static files, S3 + CloudFront, runs locally from a folder. Markdown content. Auto-built.

### What It Contains

| Section | Content |
|---|---|
| **Roles** | Full role definition for every agent (identity, objective, approach, outputs, interactions). All 21+ roles. |
| **Teams** | Team structures: Explorer team, Villager team, Town Planner team. Which roles belong to which team. How teams collaborate. |
| **Skills** | Claude skills files. The prompts and guidance that configure Claude for specific tasks. |
| **Claude Guidance** | CLAUDE.md files, reality documents, project conventions (no em dashes, version naming, file naming, etc.) |
| **Workflows** | How briefings work. How patches flow. How the Dev-to-QA handoff works. The IFD methodology. |
| **Project Index** | Links to all repos, all websites, all vaults. The map of the entire project. |

### What It Does NOT Contain

- Status updates (these change constantly, live in the repos)
- Daily briefs (these are per-project, live in the project repo)
- Code (this is a documentation site, not a code repo)
- Secrets (no vault keys, no API keys, no credentials)

The library contains things that change infrequently: role definitions, team structures, skills, conventions. It is the stable foundation that every project session starts from.

## The Clone-on-Start Pattern

Every new Claude Code session should clone two repos:

1. **The project repo** (SG/Send, QA, Tools, etc.) - the code being worked on
2. **The library repo** - the roles, skills, and guidance

This means every session starts with full context: the code and the team knowledge. No "let me read the role definition from somewhere" at the start of each session. It is already there.

### For Claude Web Sessions

The library content can be:
- Uploaded as a zip at the start of a session
- Referenced via a vault (clone the library vault)
- Or simply linked (the website is public, Claude Web can read it)

### For Offline Work

Clone the library repo before going offline. All role definitions, skills, and guidance are available locally. The library website runs from the local folder (just open the HTML files).

## Migration from Current Locations

Move content from the SG/Send repo to the library:

| Current Location | New Location |
|---|---|
| `team/roles/*/role.md` | `library/roles/` |
| `team/teams/*.md` | `library/teams/` |
| `CLAUDE.md` (per-project parts remain) | `library/claude-guidance/` (shared parts move) |
| Skill files | `library/skills/` |
| Workflow documentation | `library/workflows/` |

The SG/Send repo keeps its project-specific CLAUDE.md (project conventions, repo structure). The library gets the shared content (role definitions, general skills, methodology).

## Cross-Project Reuse

The library is not SG/Send-specific. It is the agentic team methodology. When spinning up a new project (the due diligence team, the partner research vault, any future project):

1. Clone the library
2. The roles, skills, and guidance are immediately available
3. The new project adds project-specific context on top

This is the two-tier team model in practice: the meta-team's knowledge lives in the library. Project teams inherit it.

## The Designer's Brief

The Designer agent should create the website structure:

- Clean, simple, navigable
- One page per role (with the full definition)
- Team overview pages (showing which roles belong to which team)
- Skill browser (list skills, click to see the full skill definition)
- Search (find a role, a skill, a workflow by keyword)
- Print-friendly (for offline reference)

Start simple. Markdown files rendered as HTML. Same Jekyll or static site generator as the QA site. Navigation sidebar. No complexity.
