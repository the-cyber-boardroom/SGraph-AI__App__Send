# URL Resolution for Grouped Storage Paths

**version** v0.20.33
**date** 05 Apr 2026
**from** Human (project lead)
**to** Architect (lead), Developer
**type** Dev brief

---

## The Problem

Yesterday's backend storage restructuring introduced grouped paths:

```
transfers/eu-west-2/by-id-from__user/usr_4a8b/by-id-from__access-token/at_7f3a/by-id-from__decryption-key/dk_9c2e/transfer-001/
```

The transfer ID alone no longer tells you the full S3 path. The grouping segments are derived from keys the server can calculate, but the share URL that the user copies and sends to someone needs to resolve to the right location. We cannot put the full `by-id-from__*` structure in the URL (ugly, exposes structure). But without it, the server does not know where to find the transfer.

## Options

### Option A: Server-Side Lookup (Fallback Search)

The server receives a transfer ID and checks a small number of known locations:

1. Check the default grouped path (calculate from the access token and decryption key provided in the request)
2. Check the flat legacy path (for pre-migration transfers)
3. If not found, check 2-3 other possible grouping locations

Since the number of possible grouping configurations is small (we define them, not users), this is a bounded search. Not a scan of the entire bucket. At most 3-4 S3 HEAD requests to check if the transfer exists at each location.

**Pros:** No URL changes. Simple tokens still work. No new metadata storage.
**Cons:** Slight latency (multiple checks). Assumes a small number of grouping configurations.

### Option B: Numeric Convention in the URL

Encode the grouping configuration as a short numeric prefix or suffix in the share URL:

```
https://send.sgraph.ai/t/1/transfer-001         (config 1: grouped by access token only)
https://send.sgraph.ai/t/2/transfer-001         (config 2: grouped by user + access token)
https://send.sgraph.ai/t/3/transfer-001         (config 3: full hierarchy)
```

The number maps to a hard-coded path template in the server code. The server knows "config 2 means `by-id-from__user/{user-id}/by-id-from__access-token/{at-id}/`" and can calculate the full path.

**Pros:** One lookup, no fallback search. Path template is version-specific (fine to hard-code).
**Cons:** URL changes slightly. Need to manage the config number registry.

### Option C: Server-Side Path Registry

When a transfer is created, the server stores a small metadata record mapping transfer ID to its full grouped path:

```json
{ "transfer-001": "eu-west-2/by-id-from__user/usr_4a8b/by-id-from__access-token/at_7f3a/..." }
```

This could be a DynamoDB entry, a small JSON file in a known S3 location, or even an entry in the transfer's own metadata.

**Pros:** Instant resolution. Works for any grouping depth.
**Cons:** Adds a metadata store. Additional write on upload. Need to keep in sync.

### Option D: Derive from the Request Context

When someone accesses a transfer, they provide:
- The transfer ID (always)
- The decryption key (always, needed to read the content)
- Optionally: the access token (if they are the uploader)

The server can derive the grouping path from whatever keys are provided in the request. Since the grouping IDs are deterministic (same key always produces the same group ID), the server calculates the path and looks up the transfer directly.

**Pros:** No URL changes. No metadata store. Uses the keys the user already has.
**Cons:** Only works if the request includes enough keys to reconstruct the path. The decryption key alone may not be enough if the grouping used the access token.

### Recommendation: Option A + D Combined

Default behaviour: the server derives what it can from the keys in the request (Option D) and falls back to checking a few known locations (Option A). This handles:

- New transfers: derivable from the request keys
- Legacy transfers: found via fallback to flat path
- Edge cases: found via bounded search (3-4 checks maximum)

No URL changes. No metadata store. Simple tokens continue to work unchanged. The cost is a few extra S3 HEAD requests in the worst case, which is negligible.

### Simple Token Consideration

Simple tokens encode the transfer ID and decryption key in three words. They do not encode grouping information. This is fine with Options A and D: the server derives the grouping from the decryption key and falls back if needed. No change to the simple token format required.

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | Server resolves transfer ID to grouped path using derivation from request keys |
| 2 | Fallback to flat legacy path for pre-migration transfers |
| 3 | Bounded search (max 3-4 checks) for edge cases |
| 4 | Simple tokens continue to work without changes |
| 5 | No new metadata storage required |
| 6 | Resolution latency measured (target: under 50ms additional for fallback) |
