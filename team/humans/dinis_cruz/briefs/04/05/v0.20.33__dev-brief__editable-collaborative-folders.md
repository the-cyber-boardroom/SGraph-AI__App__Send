## The Use Case

A group trip. Five people take photos. Everyone wants to upload their photos to a shared folder. Then the group curates: upvote the best ones, remove duplicates, pick the final set for printing or sharing. This requires:

- Multiple people uploading to the same folder
- Browsing all photos from all contributors
- Voting/upvoting individual photos
- The folder growing over time (not a single upload)

This is a collaboration pattern that the current single-upload-as-zip model does not support.

## The Underlying Challenge: Individual File Storage

Currently, when a user uploads a folder, it becomes a zip file stored as a single S3 object. This works for read-only sharing but breaks down for:

- **Large folders:** A 2GB folder means a 2GB zip. Adding one photo means re-uploading 2GB.
- **Collaboration:** Multiple people cannot add to the same zip simultaneously.
- **Progressive access:** The recipient must download the entire zip before seeing any file.
- **Editing:** Changing one file means replacing the entire zip.

The solution: store each file individually on the server side (still encrypted), with a directory structure that tracks what files belong to the folder.

## Four Modes of Folder Storage

### Mode 1: Read-Only Zip (Current Behaviour)

Upload a folder as a zip. Download the zip to browse. No editing. No collaboration. Simple, fast, works today.

**When to use:** Sending a folder to someone who just needs to download it. Accountant packs. One-time deliveries.

### Mode 2: Read-Only Individual Files

Each file in the folder is stored as a separate encrypted object on the server. A directory manifest tracks the structure. Files are downloadable individually (no need to fetch the entire folder).

**When to use:** Large folders where the recipient may only need specific files. Photo galleries where you want thumbnails to load one by one. Any folder larger than ~100MB.

**Upload flow:**

1. User selects a folder (or multiple files)
2. Each file is uploaded and encrypted individually
3. A directory manifest is created (JSON file listing all files, paths, sizes, types)
4. The manifest is stored alongside the files
5. The recipient browses via the manifest, downloads individual files on demand

**Directory manifest stored where?** This is the challenge. If the server is zero-knowledge, it cannot maintain a file listing. Options:

- Store the manifest as another encrypted file alongside the data (discoverable via deterministic ID from the folder key)
- Store the manifest as a vault commit (the vault tree IS the directory structure)
- Store the manifest in the client (localStorage) and sync it

### Mode 3: Editable Individual Files (Light Collaboration)

Same as Mode 2, but files can be added, removed, or replaced. The directory manifest is updated on each change.

**When to use:** A shared folder that grows slowly (a few changes per day). Photo galleries where contributors add photos over time.

**The race condition problem:** If two people update the manifest simultaneously, one update overwrites the other. Solutions:

- Version the manifest (each update includes a version number, reject if stale)
- Use optimistic locking (try to update, retry if conflict detected)
- Accept eventual consistency (last write wins, but keep all previous versions)

For light collaboration (a few changes per day), versioned manifests with last-write-wins is acceptable. For heavy collaboration, use Mode 4.

### Mode 4: Vault-Backed Folder (Full Collaboration)

The folder is a vault. The vault tree IS the directory structure. Adding a file is a vault commit. The vault handles version control, branching, merging, and conflict resolution.

**When to use:** Heavy collaboration. Multiple people editing simultaneously. Version history needed. Full Git-like workflow on the folder.

**How it works:**

- The folder is created as a vault (using the vault primitives from the 3 April brief)
- Each file upload is a commit to the vault
- The vault tree serves as the directory manifest (no separate manifest needed)
- Contributors clone the vault (or access via deterministic IDs for light browsing)
- Conflicts handled by vault merge logic

**For the photo gallery:** The vault stores the photos as blobs. Votes/upvotes are stored as metadata commits (a JSON file tracking votes per photo). The vault tree provides the browsing structure.

## The Photo Gallery: Specific Features

### Upload Phase

- Drag and drop multiple photos (or entire folders)
- Each contributor uploads independently (no coordination needed)
- Photos appear in the gallery as they upload (progressive, not all-at-once)
- Automatic thumbnail generation (client-side, from the uploaded image)

### Browse Phase

- Grid view of all photos (thumbnails)
- Filter by contributor (who uploaded this?)
- Sort by date, name, or vote count
- Full-screen lightbox for individual photos

### Curation Phase

- Upvote/downvote individual photos (click a star, heart, or thumbs up)
- Votes stored as a JSON metadata file in the folder/vault
- View by "most voted" to find the best photos
- Mark photos as "selected" for the final set
- Export the selected set as a new folder or zip

### Sharing

- Share the gallery via simple token (read-only for viewers)
- Share with edit access (contributors can add photos)
- The gallery renders in the Browse view with the photo-specific components

## Implementation: Start Simple, Evolve

1. **First:** Mode 2 (read-only individual files). Break the zip into individual files during upload. Serve them individually. This alone enables progressive loading and large folder support.

2. **Second:** Mode 3 (editable with versioned manifest). Allow adding files to an existing folder. Handle the manifest update. This enables the basic photo gallery.

3. **Third:** Add voting/curation as a metadata JSON file alongside the photos. The Browse view gets a photo gallery component with upvote buttons.

4. **Fourth:** Mode 4 (vault-backed) for full collaboration. This gives version history, branching, and proper conflict resolution.

Each step is independently valuable. Mode 2 alone is a significant improvement for large folders.

## Connection to Existing Architecture

- **VFS:** The folder modes map to VFS providers (zip provider for Mode 1, individual-file provider for Modes 2-3, vault provider for Mode 4)
- **sg-vault-file-drop:** The drag-and-drop upload component from the SGit browser brief
- **Browse view:** Renders the folder regardless of mode (the data source adapter pattern from the vault-browse unification brief)
- **Deterministic paths:** Each file stored at a deterministic path derived from the folder key + file name
- **SG Layout:** Gallery view as a component (grid of thumbnails, lightbox, filter/sort controls)

## File Management Inside the Folder

Beyond uploading, users need to manage files within the folder:

| Action | What It Does |
|---|---|
| Drag and drop files | Add individual files to the folder |
| Drag and drop folders | Add entire folder structures (preserving hierarchy) |
| Rename files | Change file name in place (manifest update + commit) |
| Rename folders | Change folder name (all child paths update) |
| Delete files | Remove from the folder (soft delete with version history, hard delete optional) |
| Move files | Drag between folders within the same shared folder |
| Per-file metadata | Add tags, classification, description, notes to individual files |
| Per-folder metadata | Add description, cover image, sort order to folders |

Each of these operations is a manifest update (Mode 3) or a vault commit (Mode 4). The version history means every operation is reversible.

## Share From Inside

A powerful feature: while browsing a shared folder, create a new share link from within it. Use cases:

- "I want to share just this subfolder with someone else"
- "I want to share the selected/upvoted photos as a separate link"
- "I want to create a read-only view of this editable folder"

The share button lives inside the folder browser, not just on the original share page. This means collaboration flows naturally: someone receives a folder, browses it, selects the best photos, and shares the curated set with a new link.

## Version Control as a Transparent Experience

For most users, version control is invisible. They do not need to know about commits, branches, or merges. They see:

- An "undo" button (roll back the last change)
- A "history" button (see what changed and when)
- A "restore" option on deleted files (bring back something that was removed)
- A conflict message ("Someone else changed this file. Keep yours, keep theirs, or keep both?")

The version control terminology (commit, branch, merge) never appears in the UI. The concepts are there (every change is a commit, every contributor is on their own branch, viewing the combined result is a merge), but the language is consumer-friendly.

This is a key opportunity: introducing non-technical users to the power of version control without the intimidation. If the photo gallery "just works" with undo, history, and conflict resolution, users experience version control without knowing the term. That is how it should spread.

## Share URL Considerations

The share URL for an editable folder needs to encode the folder identity. The user/access-token derived group IDs do not need to be enormous. The collision space is manageable (we are not serving millions of users yet). Options:

- Include a short group identifier in the URL (4-6 characters adds minimal length)
- Use the simple token format (three words already encode the transfer ID + decryption key)
- For editable folders, the URL might include a "mode" indicator so the client knows to expect individual files rather than a zip

Map out what the URLs look like for each mode and ensure they are clean enough for messaging and email.

## Known Bugs: Mobile Upload

### Apple Photos Gallery

Reported by users: dropping individual files from the Apple Photos app works, but dropping multiple files from the image gallery does not. This is likely a browser API limitation (the DataTransfer API handles multi-file drops differently from native photo galleries on iOS/macOS).

### Files Not Available Locally

When a user tries to drop a file that is stored in iCloud (or Google Photos, or any cloud-synced gallery) but not downloaded locally, the drop fails silently. The file reference exists but the data is not available.

### Required Fixes

1. **Detect the failure:** When a drop event contains file references that cannot be read, show a clear message: "These files are stored in the cloud. Please download them to your device first, then try again."
2. **Provide alternatives:** "You can also use the file picker button to select files" (the file picker may handle cloud-synced files better than drag-and-drop on some platforms).
3. **Document the workarounds:** On the website, create a help page: "How to upload photos from Apple Photos / Google Photos / iCloud" with step-by-step instructions for each platform.
4. **Test on all platforms:** iOS Safari, macOS Safari, macOS Chrome, Android Chrome, Windows Chrome. Each handles multi-file drops and cloud-synced files differently.
