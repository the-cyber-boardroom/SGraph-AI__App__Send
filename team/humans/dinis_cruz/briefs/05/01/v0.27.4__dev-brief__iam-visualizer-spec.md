# IAM Visualizer Spec: Graph-Based Permission Analysis and Audit

**version** v0.27.4
**date** 01 May 2026
**from** Human (project lead)
**to** Developer (lead), AppSec
**type** Dev brief

---

## The Problem

AWS IAM is a mess in any account older than a few months. Roles accumulate. Policies attach and are never removed. Permissions are granted for a one-time task and left in place forever. Nobody fully understands which role has which permissions, which resources they grant access to, or which roles are actually in use.

The AWS IAM console is not designed for this. It shows lists, not relationships. You cannot see the intersection of a role's policies with the resources they affect. You cannot see which roles are unused. You cannot confidently delete anything because you do not know what depends on it.

## What It Does

A container that downloads the full IAM state as JSON, builds a graph of roles, policies, permissions, and resources, visualises the graph interactively, and provides the confidence to clean up.

### Core Capability: Digital Twin of IAM

Capture the complete IAM state:
- Roles (name, ARN, trust policy, attached policies, inline policies)
- Policies (name, ARN, policy document with all statements)
- Users (name, attached policies, groups)
- Groups (name, attached policies, members)
- Permission boundaries
- Service-linked roles
- Last-used timestamps (which roles were actually used recently)

Store as JSON. This is the digital twin: a complete, offline-queryable snapshot of the IAM state.

### Graph Visualisation

Render the IAM data as an interactive graph:

```
Role: sg-compute-provisioner
    |
    +--> Policy: EC2FullAccess
    |       |
    |       +--> Permission: ec2:RunInstances (Resource: *)
    |       +--> Permission: ec2:TerminateInstances (Resource: *)
    |       +--> Permission: ec2:DescribeInstances (Resource: *)
    |
    +--> Policy: S3ReadOnly
    |       |
    |       +--> Permission: s3:GetObject (Resource: arn:aws:s3:::sg-*)
    |       +--> Permission: s3:ListBucket (Resource: arn:aws:s3:::sg-*)
    |
    +--> Trust: Lambda service (can be assumed by Lambda)
    |
    +--> Last used: 2026-04-30T14:22:00Z (yesterday)
```

Navigate the graph: click a role, see its policies. Click a policy, see which roles use it. Click a permission, see which resources it affects. Search for a specific permission (`s3:DeleteBucket`) and see every role that has it.

### Cleanup Confidence

The tool provides the data needed to delete safely:

| Signal | What It Means |
|--------|--------------|
| Role last used > 90 days ago | Likely unused, candidate for deletion |
| Policy attached to zero roles | Orphaned policy, safe to delete |
| Role with AdministratorAccess | Over-privileged, needs scoping |
| Inline policy (not managed) | Harder to audit, should migrate to managed |
| Permission with Resource: * | Over-broad, should be scoped to specific resources |
| Trust policy allows any account | Potential cross-account risk |

Each signal links to the specific roles, policies, and permissions involved. The user can review and decide, not just see a summary.

### Time-Series IAM Audit

Run the capture regularly (daily or after TerraForm/CloudFormation changes). Compare snapshots:

- "Three new roles were created since yesterday"
- "This policy was modified: two new permissions added"
- "This role was used for the first time in 60 days"
- "Five roles have not been used since the last snapshot"

This is the IAM audit log that AWS does not provide natively. CloudTrail captures API calls, but it does not capture "the state of IAM at this point in time." The digital twin snapshots do.

Post-TerraForm visibility: run a TerraForm apply, then run the IAM analyzer. The diff between the before and after snapshots shows exactly what TerraForm changed in IAM. This is the "state of production after this change" visibility.

### Delete Mode (Write Capability)

With write permissions, the tool can execute cleanups:

- Delete unused roles (with confirmation)
- Detach orphaned policies
- Remove inline policies and replace with managed equivalents
- Scope over-broad permissions

Every delete action is logged in the transaction log and saved to the vault. Full audit trail of what was deleted, when, and why.

**Safety:** Delete actions require explicit confirmation. The UI shows exactly what will be deleted and what depends on it before executing. A "dry run" mode shows what would be deleted without executing.

## The Container

Same pattern as EC2 Analyzer: one container, runs anywhere, LETS pipeline for data, Elastic for visualisation.

```json
{
  "name": "iam-visualizer",
  "display_name": "IAM Visualizer and Auditor",
  "description": "Graph-based IAM analysis, audit snapshots, cleanup tools",
  "image": "sgraph/iam-visualizer:latest",
  "ports": { "8000": 8000 },
  "requires": {
    "aws_credentials": true,
    "storage": "optional",
    "elastic": "optional"
  }
}
```

## Acceptance Criteria

| # | Criterion | Verification |
|---|-----------|-------------|
| 1 | Full IAM state captured as JSON (roles, policies, users, groups) | Run collect, verify JSON files contain complete IAM data |
| 2 | Graph visualisation renders roles, policies, permissions interactively | Click a role, see its policies and permissions. Click a permission, see all roles that have it. |
| 3 | Unused roles identified (last used > 90 days) | Flag roles not used in 90 days, verify against CloudTrail |
| 4 | Orphaned policies identified (attached to zero roles) | Orphaned policies listed, confirmed no attachments |
| 5 | Over-privileged roles flagged (AdministratorAccess, Resource: *) | Over-broad permissions highlighted in the graph |
| 6 | Time-series diff between two snapshots shows changes | Capture before and after a change, diff shows new/modified/deleted items |
| 7 | Dry-run delete mode shows what would be removed | Select unused roles for deletion, dry-run shows impact without executing |
| 8 | Delete actions logged in vault with audit trail | Execute a deletion, verify audit record in vault |
| 9 | Runs standalone and as SG Compute pod | Both modes functional |
| 10 | Post-TerraForm snapshot shows IAM changes | Run TerraForm apply, capture snapshot, diff shows exactly what changed |
