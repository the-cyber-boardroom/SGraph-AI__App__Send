# SGit Research: Commit Object Bundling for Performance Optimisation

**version** v0.27.4
**date** 01 May 2026
**from** Human (project lead)
**to** Villager (lead), Developer (SGit team)
**type** Research brief

---

## The Problem

SGit's performance bottleneck is not bandwidth. It is request count.

A single commit may involve dozens of objects: the commit object itself, one or more tree objects (recursive), and the content blobs. Each object is a separate encrypted file on S3. Downloading a commit means making dozens of individual HTTP requests, each with connection overhead, TLS handshake, S3 latency, and decryption.

The total bytes transferred would be the same whether you download 50 small files or one bundled file. But the wall-clock time is dramatically different because the latency per request dominates when objects are small.

## The Hypothesis

If we bundle all objects created by a commit into a single zip file and store that zip alongside the individual objects, download and upload performance should improve significantly. One request instead of dozens. Same bytes. Much less latency.

## What to Research

### 1. Commit Bundle on Save

When a commit is created, in addition to storing each object individually (the current behaviour), also create a zip containing all objects associated with that commit:

```
commit-{hash}.zip
    |
    +--> commit.json          (the commit object)
    +--> tree-{hash-1}.json   (root tree)
    +--> tree-{hash-2}.json   (subtree)
    +--> blob-{hash-1}.enc    (content blob, encrypted)
    +--> blob-{hash-2}.enc    (content blob, encrypted)
    +--> manifest.json        (list of all objects in this bundle)
```

The individual objects are still stored separately (for backward compatibility and for cases where you need just one object). The zip is an additional optimisation layer.

### 2. Commit Bundle on Load

When cloning or pulling, instead of fetching each object individually:

```
Current:
    Fetch commit object          (1 request)
    Fetch root tree              (1 request)
    Fetch subtree 1              (1 request)
    Fetch subtree 2              (1 request)
    Fetch blob 1                 (1 request)
    Fetch blob 2                 (1 request)
    ...
    Total: N requests for N objects

Proposed:
    Fetch commit-{hash}.zip      (1 request)
    Extract all objects locally
    Total: 1 request
```

### 3. Server-Side vs Client-Side Bundling

**Phase 1 (client-side only, no server changes):** The client creates the zip locally after encrypting all objects, uploads the zip as one additional S3 object. On download, the client checks for the zip first. If it exists, download the zip. If not, fall back to individual object fetches.

This works without any server-side changes. The server sees the zip as just another object.

**Phase 2 (server-aware):** The server receives the zip and extracts the individual objects for storage. On download, the server bundles requested objects into a zip on the fly. This is more efficient but requires server changes.

Start with Phase 1. It proves the concept without touching the server.

### 4. Storage Mode Options

Two possible modes per vault:

| Mode | Storage | Best For |
|------|---------|----------|
| **Object mode** (current) | Individual objects only | Large files, random access to specific objects |
| **Bundle mode** (new) | Individual objects + commit zip bundles | Many small files, clone/pull performance |
| **Bundle-only mode** (experimental) | Only commit zip bundles, no individual objects | Maximum storage efficiency, but no random object access |

Bundle mode is additive: it stores the zip in addition to individual objects. Storage cost increases slightly (the zip is redundant data), but download performance improves dramatically.

Bundle-only mode is aggressive: it stores only the zip. Individual objects are not separately accessible. This saves storage but means you cannot fetch a single object without downloading the entire commit bundle. This might be acceptable for vaults where the primary access pattern is "clone everything" rather than "fetch one file."

### 5. Measurements Needed

To validate the hypothesis, measure:

| Metric | How to Measure |
|--------|---------------|
| Current clone time (N individual requests) | Clone a vault with 100 objects, time it |
| Bundled clone time (1 zip request) | Same vault, download the commit bundle, time it |
| Request count reduction | Count HTTP requests for both approaches |
| Upload time comparison | Commit with 20 new files: individual uploads vs one zip upload |
| Storage overhead of bundles | Size of individual objects vs size of individual objects + zips |
| Latency breakdown | Per-request latency (TLS, S3 response, decryption) vs per-bundle |

### 6. Compatibility

The bundle is an optimisation. Clients that do not understand bundles fall back to individual object fetches. The server stores both. Old clients work. New clients are faster.

This means we can roll out gradually: new SGit versions use bundles when available, old versions ignore them. No breaking change.

## Connection to the Shallow Clone Feature

The recent shallow clone feature (tree-only access without downloading all objects) is complementary. Shallow clone fetches the tree structure. Bundle download fetches the objects for a specific commit. Together:

```
Shallow clone: fetch only the trees (fast, gives you the file listing)
    |
    v
User selects a commit or date range to fully materialise
    |
    v
Bundle download: fetch the commit bundle (one request, all objects for that commit)
```

This is a powerful two-step access pattern: browse quickly (shallow), then download fully (bundled) only what you need.

## Connection to the Four-Layer Refactoring

This research falls cleanly into the SGit four-layer architecture from the 29 Apr Villager brief:

- **Core layer:** New "bundle" step in the commit action (create zip after encrypting objects) and new "bundle fetch" step in the pull/clone action (check for zip, download if available)
- **Network layer:** New upload/download methods for bundle files
- **Crypto layer:** Unchanged (objects are encrypted individually before bundling, same as today)
- **Plugin layer:** New `sgit bundle` CLI commands for manual bundle management

The transaction log (also from the Villager brief) will show exactly how much time is spent on individual object fetches vs bundle downloads, making the performance comparison visible at the step level.
