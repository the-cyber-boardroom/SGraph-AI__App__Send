## What It Does

A container that captures, visualises, and monitors EC2 infrastructure. Depending on the IAM privileges it is given, it provides read-only visibility (what exists, what is running, what is exposed) or read-write control (start, stop, modify instances).

This is the "digital twin" concept applied to AWS EC2: capture the state as JSON, visualise it, track changes over time, and flag security misconfigurations.

## Privilege-Adaptive UI

The spec detects what IAM privileges it has and adapts accordingly:

| Privilege Level | What It Can Do | UI Shows |
|----------------|---------------|----------|
| **Minimal (describe-only)** | List instances, read metadata, list security groups, list IP addresses | Read-only dashboards, configuration viewer |
| **Read-enhanced** | Above + CloudWatch metrics, VPC details, EBS volumes | Performance data, network topology |
| **Write-capable** | Above + start/stop instances, modify security groups | Action buttons (with confirmation gates) |

The UI disables controls the spec does not have permission to execute. No "Start Instance" button if the role does not include `ec2:StartInstances`. The privilege check runs at startup and is displayed as a health check: "This node has read-only access to EC2. To enable actions, provide a role with write permissions."

Optionally, the caller can provide a JSON file declaring the expected privileges. The spec compares expected vs actual and flags discrepancies: "Expected ec2:DescribeInstances but role does not have it."

## What It Captures (LETS Pipeline)

### Load: AWS EC2 API

Capture as JSON:
- Running instances (ID, type, state, launch time, tags, public/private IP, security groups)
- Security groups (rules, ports, source/destination CIDRs, attached instances)
- Elastic IPs (allocated, associated, unassociated)
- EBS volumes (attached, unattached, snapshots)
- VPC configuration (subnets, route tables, NAT gateways, internet gateways)
- Key pairs
- AMIs (owned, in use)

### Extract: Structured Events

Convert raw API responses into normalised JSON events. Key enrichments:
- Map security group rules to "this port is open to the internet" flags
- Identify instances with public IPs that should not have them
- Identify unattached EBS volumes (cost waste)
- Identify unassociated Elastic IPs (cost: $0.005/hour each)
- Tag-based grouping (which instances belong to which SG Compute stack/node)

### Transform: Security Signals

- Open ports summary (which ports are open to 0.0.0.0/0, which instances)
- Orphaned resources (security groups with no instances, unattached volumes, unused IPs)
- Cost waste (resources running but potentially unnecessary)
- Configuration drift (changes from one snapshot to the next)

### Save: Time-Series Snapshots

```
ec2-analyzer/
    data/
        2026/05/01/
            08/
                instances.json
                security-groups.json
                elastic-ips.json
                ebs-volumes.json
                vpcs.json
                signals/
                    open-ports.json
                    orphaned-resources.json
                    drift-from-previous.json
                manifest.json { "complete": true }
            14/
                ...
            20/
                ...
```

Run 2-3 times per day. Each snapshot captures the full state. Diff between snapshots reveals what changed: new instances, modified security groups, new open ports.

### Visualise: Elastic/Kibana Dashboards

| Dashboard | What It Shows |
|-----------|-------------|
| EC2 Overview | All instances, by state, by type, by region, by tag |
| Security Posture | Open ports, internet-facing instances, security group analysis |
| Resource Waste | Unattached volumes, unused IPs, stopped instances still incurring costs |
| Change History | What changed between snapshots (new instances, modified rules, deleted resources) |
| Network Topology | VPC, subnets, route tables, NAT gateways as a graph |

## The Container

This is one container. It contains: the CLI, the FastAPI, the data collection code, the transformation logic, and the visualisation export. It runs anywhere: inside SG Compute as a pod, standalone on Docker, on an EC2 instance, or from an AMI.

```json
{
  "name": "ec2-analyzer",
  "display_name": "EC2 Analyzer",
  "description": "EC2 visibility, security analysis, and digital twin snapshots",
  "image": "sgraph/ec2-analyzer:latest",
  "ports": { "8000": 8000 },
  "requires": {
    "aws_credentials": true,
    "storage": "optional",
    "elastic": "optional"
  }
}
```

## Acceptance Criteria

| # | Criterion | Verification |
|---|-----------|-------------|
| 1 | Privilege detection at startup shows available permissions | Start with read-only role, UI shows read-only capabilities |
| 2 | EC2 instance data captured as JSON | Run collect, instances.json contains all running instances |
| 3 | Security group analysis flags internet-open ports | Open port to 0.0.0.0/0 flagged in open-ports.json |
| 4 | Time-series snapshots stored with time-based directory structure | Two snapshots taken hours apart, both preserved |
| 5 | Diff between snapshots shows changes | Modify a security group, next snapshot flags the change |
| 6 | Orphaned resources identified (unattached volumes, unused IPs) | Unattached EBS volume appears in orphaned-resources.json |
| 7 | Runs standalone (Docker container with AWS credentials) | `docker run` with credentials, data collected |
| 8 | Runs as SG Compute pod | Start as pod inside a node, same functionality |
