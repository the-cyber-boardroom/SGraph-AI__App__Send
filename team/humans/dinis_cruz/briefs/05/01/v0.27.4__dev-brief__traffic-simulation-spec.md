# Traffic Simulation Spec: Synthetic User Behaviour for Performance Testing

**date** 01 May 2026
**from** Human (project lead)
**to** Developer (lead), QA

## What It Does

A building block that simulates real user behaviour on a website: navigating pages, clicking links, waiting between actions (at human speed, not machine speed), and capturing the results. This is not a load testing tool that hammers a server with thousands of concurrent requests. It is a behaviour simulator that replicates what actual users do, at the pace they do it.

## Why Human-Speed Matters

The point is not to stress test the server with maximum throughput. The point is to generate realistic traffic patterns that exercise the full user experience: page loads, JavaScript rendering, vault content decryption, SPA routing, deep linking, and asset loading. Running at human speed means:

- Click a page, wait 2-5 seconds (simulating reading)
- Click the next link, wait 3-8 seconds
- Scroll down, wait 1-3 seconds
- Navigate back, click something else

This produces CloudFront logs, Lambda invocations, and vault access patterns that look like real usage. The LETS observability pipeline can then analyse this traffic as if it were production traffic.

## The Two-Container Architecture

### Container 1: Coordination Engine

The brain. Manages test sessions, defines user journeys, starts/stops execution containers, collects results.

```
Coordination Engine (FastAPI + UI)
    |
    +--> Define user journeys (which pages, in what order, with what timing)
    +--> Start execution containers (one per simulated user)
    +--> Monitor execution (which step each user is on)
    +--> Collect results (screenshots, timing, errors)
    +--> Stop execution when done
    |
    +--> Needs: Docker/Podman socket access (to start execution containers)
    +--> Provides: API + Web UI for managing simulations
```

### Container 2: Execution Engine

The hands. Runs a Playwright browser, follows the journey script, captures output.

```
Execution Engine (Playwright browser)
    |
    +--> Receives a journey script (list of URLs + actions + wait times)
    +--> Executes each step:
    |       Navigate to URL
    |       Wait N seconds (human-speed delay)
    |       Optionally: click element, scroll, fill form
    |       Take screenshot
    |       Capture timing (page load, content render, vault decrypt)
    +--> Reports results back to coordination engine
    |
    +--> Needs: network access to the target website
    +--> Provides: nothing externally (talks only to coordination engine)
```

Multiple execution containers can run simultaneously, each simulating a different user on a different journey.

## Two Execution Backends

### Phase 1: Lambda-Based (Immediate)

We already have a Lambda function with Playwright fully configured. The coordination engine invokes this Lambda for each user journey. The Lambda runs the journey, returns results.

```
Coordination Engine (on EC2)
    |
    +--> Invokes Lambda function per journey
    |       Lambda has Playwright pre-installed
    |       Lambda executes journey steps
    |       Lambda returns screenshots + timing
    |
    +--> Collects results
```

Advantage: no EC2 cost for execution, Lambda scales automatically. Limitation: Lambda has a 15-minute timeout (fine for most journeys) and cold start latency.

### Phase 2: EC2-Based (Later)

The execution engine runs as a pod inside an SG Compute node. The coordination engine starts/stops pods using the node's control plane API.

```
Coordination Engine (pod on EC2 node)
    |
    +--> Starts execution pod via control plane API
    |       Execution pod runs Playwright
    |       Execution pod executes journey
    |       Results stored locally or sent to coordination engine
    |
    +--> Can start multiple execution pods (parallel users)
```

Advantage: no Lambda timeout limit, full control over the browser environment, can run alongside MITM proxy for traffic capture.

## The Journey Script

A JSON definition of what the simulated user does:

```json
{
  "name": "library-browser",
  "description": "User browses the content library",
  "target": "https://sgraph.ai",
  "steps": [
    { "action": "navigate", "url": "/", "wait_seconds": 3 },
    { "action": "navigate", "url": "/library", "wait_seconds": 5 },
    { "action": "click", "selector": "a[href*='how-vault-pages']", "wait_seconds": 4 },
    { "action": "scroll", "direction": "down", "amount": 500, "wait_seconds": 2 },
    { "action": "navigate", "url": "/library/encryption-explained", "wait_seconds": 6 },
    { "action": "screenshot", "name": "encryption-page" },
    { "action": "navigate", "url": "/dev", "wait_seconds": 3 }
  ],
  "timing": {
    "min_wait": 1,
    "max_wait": 8,
    "randomize": true
  }
}
```

The `timing.randomize` flag adds variance to wait times so traffic patterns are not perfectly regular (which would look artificial in logs).

## What It Captures

Per step:
- Page load time (navigation start to load complete)
- Content render time (for vault-rendered content: vault fetch + decrypt + render)
- Screenshot (PNG of the page at that step)
- Errors (JavaScript errors, network failures, timeouts)
- HTTP status codes for all requests

Per journey:
- Total duration
- Step-by-step timing breakdown
- All screenshots in sequence
- Error summary

Per simulation:
- All journeys completed
- Aggregate timing statistics
- CloudFront log entries generated (visible in the LETS observability pipeline)

## Storage (Phase 1: On the Coordination Engine)

For the first phase, results are stored on the coordination engine's local filesystem and viewable via the UI. Screenshots, timing data, and error logs are all accessible from the web interface.

Later phases add vault storage (persist results across sessions), S3 storage (long-term archival), and Elastic projection (dashboards for timing trends over time).

## Acceptance Criteria

| # | Criterion | Verification |
|---|-----------|-------------|
| 1 | Journey script definable as JSON | Create a 5-step journey, script parses correctly |
| 2 | Coordination engine starts and monitors execution | Start a simulation, see step progress in UI |
| 3 | Execution engine runs journey at human speed (1-8 second delays between steps) | Observe execution timing, not instant, varies per step |
| 4 | Screenshots captured per step | Browse results, see PNG for each step |
| 5 | Page load timing captured per step | Results show load time for each navigation |
| 6 | Lambda-based execution works (Phase 1) | Coordination engine invokes Lambda, journey completes, results returned |
| 7 | Multiple concurrent simulated users supported | Start 3 users on different journeys simultaneously |
| 8 | Randomized timing produces non-uniform traffic patterns | Check CloudFront logs, traffic is not perfectly periodic |
| 9 | Results viewable in UI without external storage | Browse screenshots and timing on the coordination engine's web page |
| 10 | CloudFront logs show the simulated traffic | After a simulation, LETS pipeline picks up the generated CloudFront entries |
