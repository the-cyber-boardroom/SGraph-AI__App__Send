## The Bigger Picture

We are not building a monolithic platform. We are building a library of self-contained building blocks. Each block is a container that does one thing well, has its own API and CLI, follows the LETS data pipeline pattern, and runs anywhere: inside SG Compute, on any Docker host, from an AMI on the AWS Marketplace, or on a customer's Kubernetes cluster.

This is the core product strategy expressed as architecture.

## The Properties Every Building Block Shares

Every building block (spec/container) in our library has these properties:

### 1. Self-Contained

One container. Everything it needs to function is inside the container (or declared as an explicit dependency). No implicit requirements. No "also install this library on the host." No "also run this sidecar that is not mentioned anywhere."

A building block can be started with `docker run` and a set of environment variables. That is the completeness test.

### 2. API-First (FastAPI + CLI)

Every building block exposes a FastAPI service with typed routes. Every API endpoint has a matching CLI command. The CLI calls the API. No logic in the CLI that is not in the API. No API-only features.

The base classes (SpecRoutes, SpecCLI from the 30 Apr spec structure brief) provide auth, error handling, logging, and vault access. The building block's code is only the domain-specific logic.

### 3. LETS Pipeline Native

Every building block that processes data follows the Load, Extract, Transform, Save pattern:

- **Load:** Fetch data from a source (AWS API, log files, user upload, another building block's output)
- **Extract:** Parse into structured, typed events
- **Transform:** Enrich, aggregate, signal
- **Save:** Store to local disk, vault, S3, or Elastic

Each LETS stage is idempotent, independently re-runnable, and produces stored output. The "only fetch what you do not have" principle applies to every Load stage.

### 4. Runs Anywhere (Five Deployment Modes)

| Mode | How | When |
|------|-----|------|
| **Local Docker** | `docker run sgraph/{block}` | Development, testing, quick analysis |
| **SG Compute Pod** | Pod inside a node, managed by the control plane | Part of a larger workflow, integrated with other blocks |
| **Standalone EC2** | EC2 instance running just this container | Dedicated, always-on deployment |
| **AMI** | Pre-baked AMI, launch from AWS console or API | Fastest start, AWS Marketplace listing |
| **Any Container Host** | Kubernetes, Fargate, GCP Cloud Run, Azure Container Instances | Multi-cloud, customer's infrastructure |

The container does not know which mode it is running in. It talks to its API, reads from its data sources, writes to its storage. The deployment mode is invisible to the code.

### 5. Composable (Stacks)

Building blocks compose into stacks. A stack is a set of building blocks deployed together with shared networking:

```
Stack: "AWS Audit Suite"
    +--> EC2 Analyzer (read-only visibility)
    +--> IAM Visualizer (permission audit)
    +--> AWS Billing (cost analysis)
    +--> Elastic + Kibana (visualisation)
    +--> S3 Storage (local data persistence)
```

Each block in the stack is independent. It can be started, stopped, or replaced without affecting the others. The stack definition (JSON) declares which blocks to include and how they connect.

### 6. Storage-Agnostic

Every building block supports multiple storage destinations:

| Storage | When | Properties |
|---------|------|-----------|
| Local disk | Ephemeral analysis, quick work | Fast, lost on container/instance termination |
| Vault | Persistent, encrypted, versioned | Survives instance termination, full audit trail |
| S3 | Long-term archival, large datasets | Cheap, durable, accessible from any block |
| Elastic | Visualisation and exploration | Ephemeral projection, not a data store |

The building block does not hard-code a storage destination. It accepts a storage configuration (environment variable or vault config) and writes to whatever is configured.

### 7. Privilege-Adaptive

Building blocks that interact with cloud APIs (EC2, IAM, S3, billing) adapt their functionality to the privileges they are given:

- Detect available permissions at startup
- Enable only the features the current role supports
- Display a health check showing what is available and what is not
- Never fail with a permission error on a feature the user can see in the UI

### 8. Time-Series by Default

Every building block that captures data stores it in a time-based directory structure:

```
{block}/data/{year}/{month}/{day}/{hour}/
    +--> raw data
    +--> extracted events
    +--> signals
    +--> manifest.json (complete/partial flag)
```

This enables: historical comparison (what changed since yesterday?), trend analysis (is this metric getting worse?), anomaly detection (this looks different from the last 7 days), and regulatory compliance (prove the state of the system at a specific point in time).

### 9. Visualisation-Ready

Every building block can export its data to Elastic and comes with pre-configured Kibana dashboards. The dashboards are baked into the block's container or spec definition. Start Elastic, import data, dashboards render immediately.

Screenshots of dashboards are capturable via Playwright and saveable to vaults for reporting.

### 10. Marketplace-Ready

Every building block is a potential AWS Marketplace product:

- Self-contained (no external dependencies beyond AWS credentials)
- Documented (README, API docs, dashboard descriptions)
- Priced (per-hour for AMI, per-execution for Lambda, free for open source)
- Versioned (AMI versions track spec versions)

The Marketplace listing is the distribution channel. The building block is the product.

## The Pattern for New Building Blocks

Creating a new building block follows a repeatable pattern:

1. **Define the spec** (spec.json: metadata, containers, ports, requirements)
2. **Implement the API** (FastAPI routes inheriting from SpecRoutes)
3. **Implement the CLI** (CLI commands inheriting from SpecCLI, wrapping the API)
4. **Implement the LETS pipeline** (Load from source, Extract events, Transform signals, Save to storage)
5. **Create Elastic dashboards** (pre-configured, baked into the spec)
6. **Write tests** (unit, integration, synthetic)
7. **Publish to Docker Hub** (container image)
8. **Optionally bake AMI** (for Marketplace listing)
9. **Optionally create external repo** (sg-compute_spec_{name})

Steps 1-6 should take a day or two for a focused building block. The LETS pipeline pattern, the base classes, and the spec structure eliminate most of the boilerplate. The domain-specific code is all that remains.

## Why This Matters Commercially

The building block library is the moat. Not because the code is proprietary (it is open source). But because:

- **Each block is useful alone** (buy the IAM Visualizer, use it standalone)
- **Blocks are more useful together** (the AWS Audit Suite stack combines billing + EC2 + IAM)
- **SG Compute is the orchestration layer** (manages the blocks, handles storage, provides vaults)
- **The Marketplace is the distribution** (one-click deploy to any AWS account)
- **The expertise is the service** (we built these, we understand them, we can customise them)

This is the "deployment is the product" model from the OWASP presentation pack, realised as a growing library of building blocks.
