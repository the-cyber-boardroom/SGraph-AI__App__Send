## The SPA Architecture (Already Working)

The sgraph.ai website is now a Single Page Application with deep linking:

- CloudFront redirect sends all traffic to the main HTML page
- JavaScript router parses the URL path and loads the correct content
- Deep links work: `sgraph.ai/library/how-vault-pages-work` loads directly
- Content comes from vaults (read key + object ID)

This is the foundation for everything below.

## The Object Viewer Page

### What It Needs

A page at a route like `sgraph.ai/view/{vault-id}/{object-id}` that:

1. Receives a vault ID and an object ID from the URL
2. Looks up the read key for that vault (from a local registry)
3. Fetches the object from the vault
4. Decrypts client-side
5. Renders the content (markdown, image, JSON, whatever the object is)

### The Read Key Problem

The object viewer needs the vault's read key to decrypt. Two approaches:

**Option A: Hash fragment (never reaches the server)**

```
sgraph.ai/view/{vault-id}/{object-id}#read-key={key}
```

The hash fragment (`#read-key=...`) is never sent to the server. The JavaScript reads it from `window.location.hash`. This is cryptographically clean: the server never sees the decryption key. It also demonstrates a real security property to the user.

Good for: sharing specific objects externally, one-off links.

**Option B: Local vault registry (cached in browser)**

A JSON file (on the site or fetched once) lists all vaults currently exposed and their read keys:

```json
{
  "vaults": [
    { "vault_id": "abc123", "name": "Content Library", "read_key": "xxx" },
    { "vault_id": "def456", "name": "OWASP Pack", "read_key": "yyy" }
  ]
}
```

The page JavaScript checks: do I have this registry cached in localStorage? If not, fetch it. Then look up the vault ID, get the read key, and decrypt.

Good for: navigating within the site (all library content uses known vaults with known read keys).

**Recommendation:** Use both. Option B for site-internal navigation (the registry is cached, every vault content page resolves instantly). Option A for external sharing (the link is self-contained, no registry needed).

### The Rendering

The object viewer renders based on content type:

| Type | How It Renders |
|------|---------------|
| Markdown (.md) | Rendered as HTML via markdown component |
| Image (.png, .jpg, .svg) | Displayed inline |
| JSON | Formatted, navigable tree view |
| HTML | Rendered in a sandboxed iframe or inline |
| PDF | PDF viewer component |
| Unknown | Download link |

### These Are Existing Components

The vault browser, tree viewer, commit history, and file renderer already exist as Web Components on tools.sgraph.ai. The object viewer page does not need to rebuild them. It imports and embeds them:

```html
<sg-vault-tree vault-id="{id}" read-key="{key}"></sg-vault-tree>
<sg-vault-history vault-id="{id}" object-id="{oid}"></sg-vault-history>
<sg-vault-metadata vault-id="{id}"></sg-vault-metadata>
```

The new work is: the SPA routing, the vault registry, the read key resolution, and the layout that composes these components into the library's design.

## Content Provenance

Every piece of content on the library should be traceable to its source:

- "This article is from vault `abc123`, object `def456`"
- "Last updated: 30 Apr 2026, commit `ghi789`"
- "View source vault" (link to the tree view)
- "View history" (link to the commit history for this file)

This is the transparency layer. The user is not reading a static page. They are reading a vault object with a full version history, a known source, and a verifiable decryption chain. The provenance information should be subtle (a small footer or info icon) but always present.

## The Vault Registry JSON

A JSON file maintained by @content (or automatically generated from the content vault), listing all public vaults and their read keys:

```json
{
  "version": "2026-05-01",
  "vaults": [
    {
      "vault_id": "abc123",
      "name": "Content Library",
      "read_key": "xxx",
      "description": "Library articles, guides, and documentation",
      "object_count": 45,
      "last_updated": "2026-05-01T10:00:00Z"
    },
    {
      "vault_id": "def456",
      "name": "OWASP Presentation Pack",
      "read_key": "yyy",
      "description": "8 articles on building companies in the AI era",
      "object_count": 12,
      "last_updated": "2026-04-15T14:00:00Z"
    }
  ]
}
```

This file is:
- Hosted on the site (fetched once, cached in localStorage)
- Updated by @content when vaults are added or read keys change
- The single source of truth for "which vaults power this website"
- The backup manifest for offline deployment (download these vaults to deploy offline)

## Stress Testing the Platform

Every feature described here is also a stress test:

| Feature | What It Tests |
|---------|-------------|
| SPA deep linking | CloudFront redirect, JavaScript routing, content loading |
| Object viewer | Vault fetch, client-side decryption, content rendering |

| Version history | Commit log fetch, diff rendering |

If the website works well, the platform works well. The website is the most visible test of the technology.
