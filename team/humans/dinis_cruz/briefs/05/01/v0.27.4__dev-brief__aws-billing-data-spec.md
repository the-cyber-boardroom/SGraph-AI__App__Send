## Why This Spec

AWS billing data is a mess. The dashboard is inconsistent. The data is not available in real time (updated a couple of times per day). AWS charges you every time you make a Cost Explorer API call, even if the data has not changed since the last call. And the data does not correlate well across services.

This is a perfect candidate for the LETS pipeline and a good candidate for the first spec built and maintained from an external repo, potentially even by an open source model.

## The Spec: sg-compute_spec_aws-billing

A self-contained spec that downloads AWS billing data, stores it, transforms it, and visualises it. Runs as a container, an EC2 instance, or a pod inside SG Compute. Also runs standalone.

### What It Does

1. Downloads AWS billing data (Cost Explorer API, or CUR reports from S3)
2. Stores the raw data locally (never re-downloads data we already have)
3. Transforms into structured, queryable format
4. Visualises via Elastic/Kibana dashboards (or exports for Grafana)
5. Provides a better billing dashboard than AWS does

## The LETS Pipeline for Billing

### Load: Smart Data Collection

**The "only fetch what you do not have" principle:**

```
Check: do we have data for 2026-04-30?
    |
    +--> YES, and period is complete (day is over): skip
    +--> YES, but period is partial (today): re-fetch (data may have updated)
    +--> NO: fetch from AWS, save
```

This minimises Cost Explorer API calls (which AWS charges for). We only call when:
- We are missing a day entirely
- The current day's data may have updated (fetch at most 2-3 times per day)
- We never re-fetch completed past days

The file system is the cache: if `billing/raw/2026/04/30/cost-data.json` exists and the manifest says `"complete": true`, we do not fetch again.

### Extract: Parse AWS Billing Format

AWS billing data comes in a messy format. Extract into clean, typed events:

```json
{
  "date": "2026-04-30",
  "service": "Amazon EC2",
  "usage_type": "EU-BoxUsage:t3.small",
  "region": "eu-west-2",
  "cost_usd": 0.0208,
  "usage_quantity": 1.0,
  "usage_unit": "Hrs",
  "resource_id": "i-abc123",
  "tags": {
    "sg-compute-stack": "observability",
    "sg-compute-node": "elastic-1"
  }
}
```

Key enrichments:
- Break down by service, region, resource
- Correlate with SG Compute tags (which stack, which node, which spec)
- Calculate daily totals, service totals, resource totals
- Identify cost anomalies (this resource costs 3x more than yesterday)

### Transform: Aggregate and Signal

```
Daily summary:
    Total cost: £X.XX
    Top 5 services by cost
    Top 5 resources by cost
    Cost per SG Compute stack
    Cost per SG Compute spec type
    Cost trend (vs yesterday, vs 7-day average)
    Anomalies flagged

Monthly projection:
    Current month spend so far
    Projected month-end spend (based on daily average)
    Comparison to previous month
```

### Save: Multiple Destinations

| Destination | When | Why |
|-------------|------|-----|
| Local disk (ephemeral) | Always | Fast access during the session |
| Vault | If persistent storage wanted | Encrypted, versioned cost history |
| S3 | If long-term archival wanted | Cheap, durable |
| Elastic | For visualisation | Dashboards, drill-down, time series |

### Visualise: Kibana Dashboards

Pre-configured dashboards baked into the spec:

| Dashboard | What It Shows |
|-----------|-------------|
| Daily Cost Overview | Total spend, by service, by region, trend line |
| SG Compute Costs | Cost per stack, per node, per spec type (using tags) |
| Resource Deep Dive | Individual resource costs, sorted by spend |
| Anomaly Detection | Resources or services with unusual cost spikes |
| Monthly Projection | Current spend vs projected month-end, vs previous month |
| Cost per Feature | Map costs to product features (how much does observability cost? how much does Playwright cost?) |

Screenshot these dashboards via Playwright and save to a vault for daily cost reports.

## Why This Is a Good First External Spec

### Simple but Not Easy

The concept is simple (download billing data, visualise it). The implementation has real complexity (AWS API quirks, data format inconsistencies, cost allocation tags, time zone handling, partial data for the current day). This makes it a good test of the spec system.

### Standalone and Composable

The billing spec works standalone (just point it at AWS credentials, it fetches and visualises). It also composes with other SG Compute infrastructure (it reads tags from nodes and stacks to correlate costs).

### Marketplace Candidate

"Better AWS billing dashboard, runs in your account, ephemeral, no data leaves your environment" is a real product. List it on the marketplace. Even as a free tool, it drives awareness of SG Compute.

### External Repo Test

Build this in `sg-compute_spec_aws-billing` as a separate GitHub repo. This tests the spec-from-external-repo workflow. Bonus: see if an open source model (Gemma, Llama, Qwen) can create and maintain this spec, which would validate the "specs built by agents" pattern.

## Credential Handling

The spec needs AWS credentials to call Cost Explorer. Two modes:

**Environment variable mode:** Credentials provided via environment variables when the container starts. The SG Compute provisioning system injects them from the vault.

**IAM role mode:** The EC2 instance has an IAM role that permits Cost Explorer access. No credentials to manage. This is the preferred mode for AWS-hosted deployments.

The spec should support both and detect which is available automatically.

## Data Collection Frequency

AWS billing data is not real-time. The recommended collection schedule:

| Frequency | What It Fetches | Cost |
|-----------|----------------|------|
| 3x per day (08:00, 14:00, 20:00) | Current day's data (partial, will change) | 3 API calls/day |
| 1x per day (after midnight) | Previous day's final data (mark as complete) | 1 API call/day |
| Never | Completed past days (already have the data) | 0 API calls |

Total: approximately 4 Cost Explorer API calls per day. AWS charges $0.01 per call. Monthly cost: approximately $1.20. Trivial.
