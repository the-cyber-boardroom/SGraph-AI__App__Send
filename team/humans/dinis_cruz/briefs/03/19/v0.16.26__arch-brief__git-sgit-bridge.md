# Git to SGit to Git: Agent Identity, Branch Isolation, and Delta Sync

**from** Human (project lead)

## The Problem

When an AI agent (Claude Web, Claude Code, or any LLM session) needs to work with a Git repo, the current options are bad:

1. **Give the agent your identity token.** The agent acts as you. It can push to main. It can touch any branch. There is no isolation. The only protection is convention ("please only push to your branch"), not enforcement.

2. **Clone publicly, produce patches.** The agent clones a public repo, makes changes, and produces a patch file. This works but is messy: patch management, manual application, no structured workflow for tracking what the agent changed.

Neither option provides what we actually need: **an agent with its own identity, limited to a specific branch, that can make changes and hand them back in a structured, verifiable way.**

## The Idea: SGit as the Bridge

Use an SGit vault as an intermediary between Git and the agent. The vault provides the identity (PKI per branch), the isolation (agent can only write to its branch), and the structured handoff (the vault contains exactly what changed).

### The Flow

```
Git repo (source of truth)
  ↓
Export: branch delta → SGit vault
  ↓
Agent clones SGit vault (with its own branch, its own key pair)
  ↓
Agent makes changes, commits to its branch (signed)
  ↓
SGit vault contains: only the changed files (or patches)
  ↓
Import: SGit vault → Git branch → pull request
  ↓
Human reviews PR, merges if appropriate
```

The agent never has your Git identity. The agent never has access to main. The agent operates in an SGit vault where it has its own cryptographic identity and can only touch its own branch. The changes flow back to Git through a structured import that produces a PR.

## Two Modes for the Vault Content

### Mode 1: Changed Files Only

The SGit vault contains only the files that were modified on the branch. When importing back to Git, these files are applied on top of the current branch state. This is lightweight: the vault is small, containing only the delta.

The import process:
1. Check out the target Git branch
2. For each file in the SGit vault, overwrite the corresponding file in the Git working directory
3. Git commit (the Git diff shows exactly what the agent changed)

### Mode 2: Patch/Diff Format

The SGit vault contains diff patches rather than full files. Even more lightweight. The import process applies the patches using standard Git tooling (`git apply`).

This mode is more fragile (patches can fail to apply if the base has changed) but more efficient for large files where only a few lines changed.

### Recommendation

Start with Mode 1 (changed files only). It is simpler, more robust, and easier to debug. Mode 2 is an optimisation for later.

## Manifest and Control Files

The vault includes manifest files that track the sync state:

```
_sync/
  manifest.json     {
                      "source_repo": "github.com/org/repo",
                      "source_branch": "main",
                      "source_commit": "abc123",
                      "tracked_files": ["src/vault.py", "src/cli.py", ...],
                      "agent_branch": "agent-explorer-session-42",
                      "created_at": 1710500000000
                    }
  changes.json      [
                      { "file": "src/vault.py", "action": "modified" },
                      { "file": "src/new_file.py", "action": "added" }
                    ]
```

The manifest ensures that when importing back to Git:
- We know which commit the changes are based on (detect if the base has moved)
- We know which files were changed (apply only those)
- We can verify the agent's branch and identity (signed commits in the vault)

## Why This Solves the Agent Identity Problem

Today with Claude Code:

- The agent uses your personal token
- The agent acts as you
- You cannot restrict the agent to a branch (convention only)
- You cannot prevent pushes to main (convention only)
- You cannot cryptographically verify which agent made which change

With the SGit bridge:

- The agent gets its own SGit vault with its own key pair
- The agent can only commit to its branch (enforced by the vault architecture)
- Every commit is signed by the agent's key (provenance)
- The human imports changes back to Git through a PR (review gate)
- The agent never has your Git identity

This is the difference between "please don't push to main" and "you physically cannot push to main because you don't have the key."

## Git Compatibility Question

An important open question: if you take a Git repo, export changes to SGit, modify them, and import back to Git, is the resulting Git repo structurally identical to one where the changes were made directly?

Specifically: if you apply the same sequence of file changes to a Git repo, do you get the same commit hashes? Likely not, because Git commits include timestamps, author info, and parent hashes that would differ. But the file content should be identical, and the Git diff should be clean.

This means Git-to-SGit-to-Git does not produce byte-identical Git histories. It produces content-identical results. The PR diff on the Git side will show exactly what the agent changed, which is what matters for review.

**Research needed:** Verify this. Create a test case: make the same file changes directly in Git and via the SGit bridge. Compare the resulting file trees. They should be identical. The commit metadata will differ (different timestamps, different author), but the content diff should be the same.

## Lazy Clone (Download on Demand)

The memo mentions "clone light" or "organic cloning" where only the needed files are downloaded. This is already supported by the vault architecture: the tree structure tells you what files exist, and you can read individual blobs without downloading the entire vault.

For the Git bridge, this means the agent does not need to clone the entire repo into the vault. Only the files relevant to the task are exported. The agent works on those files. The manifest tracks which files are from the source repo.
