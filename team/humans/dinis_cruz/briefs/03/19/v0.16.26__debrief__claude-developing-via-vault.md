## What Happened

A real working session where Claude was used for development with a vault as the collaboration medium instead of Git. The workflow:

1. Human asked Claude to create a Node.js CLI tool
2. Claude confirmed it knows how to use an SGit vault
3. Claude made changes and committed to the vault
4. Human pulled the changes to their laptop
5. Changes were also visible on the web and on phone
6. Claude continued iterating: write code, commit, push
7. Human pulled updates as needed

This is working. With a couple of minor bugs discovered, the core workflow is solid: **the LLM develops, the vault is the collaboration channel, the human reviews on any device.**

## Why This Is Better Than Git for Agents

The traditional approach: give Claude your Git credentials. The problems:

- Claude acts as you (your identity, your permissions)
- Credentials are overpowered (full repo access, push to any branch)
- Sharing credentials is a security risk
- Git authentication from inside Claude sessions is fragile (submodules, SSH keys, token management)

The vault approach: give Claude a vault key. Claude commits to the vault. The human pulls.

**The key insight from this session: the only secret the agent needs is its own private key.** Not the repo identity. Not your credentials. Not an overpowered access token. Just its own identity.

This is a fundamentally different authorization model:

| Traditional (Git) | Vault-Based |
|---|---|
| Agent gets your credentials | Agent gets its own identity (private key) |
| Agent has your full permissions | Agent has only the permissions its identity grants |
| Agent can push to any branch | Agent can only commit to branches its key can sign |
| You trust the agent by convention | You constrain the agent by cryptography |
| Revoking access = rotating your token | Revoking access = not accepting that key's signatures |

Instead of giving the agent the privileges and the potential access to all assets, you give the agent credentials for its own identity. The vault controls what that identity can do.

## Bugs and Improvements Found

### .sgitignore (or equivalent)

Need a mechanism to prevent committing:

- `node_modules/` (can be enormous, should never be in the vault)
- `.git/` directories (if the working directory is also a Git repo)
- `.env` files (secrets that should not be committed)
- Other language-specific artifacts: `__pycache__/`, `.pyc`, `*.o`, `build/`, `dist/`

Implement as a `.sgitignore` file (or `_vault_ignore`) with glob patterns, similar to `.gitignore`. Hard-code a default set of patterns that are always ignored regardless of the ignore file.

### .keep files

Need support for `.keep` files (or equivalent) to preserve empty directories in the vault. Git has this convention. SGit should too.

## Agent Identity as a First-Class Concept

This session validates the agent identity model from the branch model spec and the Git-to-SGit bridge brief. The practical experience confirms:

- The agent should receive its private key as a credential (like a startup secret)
- The vault defines what that identity can do (which branches it can commit to, which files it can read)
- The human controls the vault's authorization rules
- The agent's commits are signed, so every change has provenance

### Future: Server-Side Enforcement

For additional security against a rogue or compromised agent:

- Server-side signature verification: reject unsigned commits or commits signed by unrecognised keys
- Rate limiting per identity: an agent cannot make thousands of commits per minute
- Branch restrictions enforced server-side: certain branches only accept commits signed by specific keys
- Read restrictions: certain files or folders encrypted with keys the agent does not have (the agent physically cannot read them, as already designed in the branch model)

Most of this is already possible with the current architecture. The server-side enforcement adds defense in depth for production deployments.

## The Workflow Going Forward

This is the development workflow for agents:

```
1. Human creates a vault for the task
2. Human gives the agent the vault key (or the agent's own key pair)
3. Agent clones, develops, commits, pushes
4. Human pulls and reviews (on laptop, web, or phone)
5. Human merges to the main branch when satisfied
6. Agent's branch is the audit trail of everything it did
```

This replaces: "give Claude your GitHub token and hope it only pushes to the right branch."
