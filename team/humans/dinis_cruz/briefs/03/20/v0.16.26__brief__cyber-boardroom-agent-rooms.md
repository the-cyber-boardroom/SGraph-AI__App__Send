## Context

Cyber Boardroom is the legal entity behind SG/Send. The original Cyber Boardroom product had use cases that people genuinely liked, but it lacked a good way to share data securely and hold confidential information. SG/Send now solves exactly that. The use cases that were promising before can now be implemented properly, using vaults as the data layer and LLMs as the analysis layer.

This also connects to the broader strategy: the same way we created a musician vertical, we can create a cybersecurity board governance vertical. And the same agentic chat room pattern applies to any domain where multiple specialist agents need to advise a human.

## Private Agentic Chat Rooms

### The Pattern

A vault-backed environment where a user can talk to one or more specialist agents. The vault stores the conversation, the uploaded documents, the analysis outputs, and the context. The LLM (local via Ollama, or via OpenRouter/OpenAI with user-provided or purchased tokens) processes the queries.

This is a private, encrypted chat room where:

- The user chooses which agent(s) to speak to
- Multiple agents can be active simultaneously (each with its own role definition)
- All conversation and data is stored in the vault (encrypted, signed, versioned)
- The user can bring their own LLM tokens or buy through SG/Send

### What the Web Components Need

- Agent selector (choose which role to talk to: CEO advisor, CFO advisor, security strategist, etc.)
- Chat interface (message input, response display, conversation history from vault)
- Document panel (upload documents to the vault, reference them in conversation)
- Multi-agent view (see responses from multiple agents side by side)

This is the general pattern. The Cyber Boardroom is one instantiation of it. Musicians could have a "talk to a producer" agent. Accountants could have a "talk to a tax advisor" agent. The infrastructure is the same.

## Cyber Boardroom Use Cases

### Use Case 1: Security Executive Talking to Multiple Board Personas (Way Up)

A CISO or security leader preparing for a board presentation. They upload their materials (budget proposals, strategy documents, risk assessments) and talk to multiple board member personas:

| Agent Persona | What They Challenge |
|---|---|
| CEO advisor | Strategic alignment, business impact, competitive positioning |
| CFO advisor | Budget justification, ROI, cost efficiency |
| Risk committee member | Risk appetite, residual risk, compliance gaps |
| Strategic board member | Long-term vision, industry trends, competitive threats |

The security executive asks: "How will the board react to this budget proposal?" and gets responses from each persona's perspective. They refine their pitch before the real board meeting.

This is the "one-to-many" pattern that early Cyber Boardroom users found valuable. The vault stores everything: the uploaded materials, the questions, the responses, the iterations. The executive walks into the board meeting having already stress-tested their arguments.

### Use Case 2: Customised Security Advisor (Athena)

The existing Athena concept, upgraded. Instead of a generic "helpful cybersecurity agent," Athena becomes a role-defined agent with specific expertise, personality, and context. The agentic workflow methodology (role definitions in markdown, context in vaults) makes customisation straightforward.

Multiple Athena variants:

- Athena (Strategy): focused on security strategy, programme maturity, roadmap advice
- Athena (Technical): focused on architecture review, vulnerability analysis, technical controls
- Athena (Compliance): focused on regulatory requirements, audit preparation, policy gaps
- Athena (Incident): focused on incident response, forensics guidance, communication during crises

Each variant is a role definition. The LLM is the same. The context is different. The user picks the Athena they need for the current task.

### Use Case 3: Board Member Receiving and Analysing Reports (Way Down)

The reverse flow. A board member receives a cybersecurity report (in a vault) and wants to understand it. They upload the report, ask questions, get analysis.

"What does this risk score mean? How does our spending compare to industry benchmarks? What are the top three things I should ask the CISO about?"

The vault captures the board member's questions and the analysis. This becomes a record of the board's engagement with cybersecurity, which is valuable for governance documentation.

### Use Case 4: Multi-Level Analysis Workflow

A more structured workflow where documents go through multiple stages of analysis:

```
1. Upload invoices, contracts, or reports to the vault
2. First pass: LLM extracts key data points, summarises each document
3. Second pass: LLM cross-references, finds patterns, flags anomalies
4. Third pass: LLM generates a board-ready summary
5. All intermediate outputs stored in the vault
6. Final report shared as a vault or downloaded as a zip
```

Each pass is a one-shot LLM call. The vault captures every stage. The user can see the work at any point. Nothing is lost. The analysis is reproducible (re-run with different parameters, the vault shows both versions).

## Objective-Driven Vaults

An important concept from this memo: **the point of a vault is to have an objective.** A vault is not raw storage. It is not a dumping ground. It is created to achieve a specific outcome.

Examples:

| Vault Objective | When It's Done |
|---|---|
| Get a board report ready | Report is approved, shared with the board |
| Analyse Q3 cybersecurity spend | Analysis is complete, recommendations documented |
| Prepare for a board presentation | Presentation is rehearsed, stress-tested, finalised |
| Respond to an incident | Incident is resolved, lessons learned documented |
| Complete due diligence on an acquisition | Due diligence report is delivered |

When the objective is met, the vault becomes an archive: a zip or a folder that captures everything that happened. Upload it to long-term storage, delete it from SG/Send, or keep it as reference. The vault had a purpose. It served that purpose. It is done.

This framing matters for marketing: "Create a vault for your board report" is more compelling than "Store files in an encrypted vault." The vault is a tool for getting something done, not a place to put things.

## PKI for Need-to-Know Access

Within a Cyber Boardroom vault, not everyone should see everything:

- The CISO sees all documents and all analysis
- A board member sees the summary report and the recommendations, not the raw vulnerability data
- An external auditor sees the compliance evidence, not the strategic planning documents

Per-document or per-folder encryption with different keys enforces this. The vault contains everything, but each participant only decrypts the portions encrypted for their key. This is the same pattern as the supply chain (per-supplier encryption) applied to board governance.
