## Context

We are preparing to launch features to the MISC (cybersecurity) community. A few additions will make a significant difference: file expiry enforcement, client-side recipient restrictions, and a PKI teaser in the UI. These require both server-side and client-side changes.

## File Expiry and Cleanup

### Expiry Options

Already discussed in the retention brief, but now confirmed for implementation:

- 1 day, 3 days (default), 7 days, 30 days
- Maximum expiry: 30 days hard cap
- Default for most transfers: 3 days
- The expiry timestamp is stored in the file metadata (already shared alongside download count)

### Server-Side Cleanup

Implement automated S3 cleanup based on expiry:

- S3 lifecycle rules or a scheduled Lambda that deletes expired files
- This is important now: as test files accumulate and users grow, storage needs active pruning
- The $1,000 AWS credits make this less urgent cost-wise, but good hygiene matters

### Display

Show the expiry information to the recipient: "This file expires in 2 days" or "Available until 23 March." Alongside the existing download count display.

## Client-Side Recipient Restrictions

### The Idea

The sender defines requirements that the recipient must satisfy before they can access the file. The server does not know what the requirements are. The server stores an encrypted policy blob. The recipient's browser evaluates the policy locally.

Examples of restrictions the sender can set:

- Country: "only accessible from GB or PT"
- Timezone: "only accessible from European timezones"
- Browser: "not accessible from automated browsers (webdriver flag)"
- IP address: "only accessible from this specific IP"
- Custom: any client-side property the sender wants to check

### Architecture: Server Stays Blind

```
Sender                          Server                    Recipient
  │                               │                          │
  │ Define policy                 │                          │
  │ Encrypt policy with           │                          │
  │   vault/access key            │                          │
  │ Upload file + encrypted       │                          │
  │   policy blob                 │                          │
  │──────────────────────────────►│                          │
  │                               │ Store opaque blobs       │
  │                               │ (cannot read policy)     │
  │                               │                          │
  │                               │◄─────────────────────────│
  │                               │ Recipient requests file  │
  │                               │──────────────────────────►│
  │                               │ Returns file +           │
  │                               │   encrypted policy blob  │
  │                               │                          │
  │                               │          Recipient:      │
  │                               │          1. Decrypt policy│
  │                               │          2. Collect local │
  │                               │             context      │
  │                               │          3. Evaluate      │
  │                               │             policy(ctx)  │
  │                               │          4. If pass:      │
  │                               │             decrypt file  │
  │                               │          5. If fail:      │
  │                               │             show error    │
```

The server never sees the policy in plaintext. The server never sees the recipient's IP, country, or browser properties. All evaluation happens in the recipient's browser.

### Context Collection: Two Sources

**Local browser signals (no network calls):**

Collected purely from standard web APIs:

- User agent, platform, vendor
- Language, locale
- Hardware hints (CPU cores, memory, touch points)
- Privacy flags (cookies enabled, do-not-track, webdriver detection)
- Screen properties (resolution, colour depth, pixel ratio)
- Timezone

These are available instantly with no external requests.

**Remote IP/geo signals (browser calls third-party, not SG/Send):**

The recipient's browser calls a third-party IP/geo API directly:

- IP address (from ipify.org, seeip.org, or similar)
- Country, city (from country.is, seeip.org, or similar)

The request goes from the recipient's browser to the third-party service. SG/Send is not in the path. SG/Send never sees the recipient's IP.

### Policy Format

A simple JSON policy DSL stored in the encrypted manifest:

```json
{
  "version": 1,
  "rules": [
    { "field": "remote.country", "op": "in", "value": ["GB", "PT"] },
    { "field": "local.webdriver", "op": "==", "value": false }
  ],
  "logic": "AND"
}
```

The client evaluates: for each rule, extract the field from the collected context, apply the operator, combine results with AND/OR logic. Simple, extensible, no server involvement.

### Tampering and Trust Model

Important caveat: this is a **soft constraint**, not a hard security guarantee. The recipient controls their browser. They can:

- Fake IP/geo responses
- Patch the evaluation logic
- Hard-code a passing context
- Replace the JavaScript bundle

Client-side context checks are excellent for honest-user controls and UX restrictions. They are not sufficient against a determined adversary who controls the client.

For stronger guarantees, future options include:

- Signed IP attestation services (a third party that signs the IP/geo response with their key, so SG/Send can verify it was not forged). This service does not seem to exist today. There is a market gap here, and potentially a billable service SG/Send could offer.
- Server-side checks (but this breaks the zero-knowledge model)
- Trusted runtimes or hardware tokens (for enterprise deployments)
- Native app enforcement (desktop/mobile clients are harder to tamper with than browser JavaScript)

For now: document that these are soft constraints, implement them as UX-level restrictions, and design the architecture so stronger enforcement can be added later without changing the policy format.

### Implementation: Component First

Build the restriction system as a standalone Web Component first:

1. Create `<sg-policy-editor>` (sender defines restrictions)
2. Create `<sg-policy-evaluator>` (recipient context collection + evaluation)
3. Test both components side by side in a mini tool on tools.sgraph.ai
4. Verify with both passing and failing contexts
5. Once mature, integrate into the main upload and download flows

This follows the component-first principle: build it as a tool, stress test it, then integrate.

## PKI Teaser in UI

Add a greyed-out "PKI" option in the sharing UI. Label it "Coming soon." This signals to the security community that PKI-based sharing is on the roadmap. It costs nothing to implement (it is a disabled button) and generates conversations about what it will do.
