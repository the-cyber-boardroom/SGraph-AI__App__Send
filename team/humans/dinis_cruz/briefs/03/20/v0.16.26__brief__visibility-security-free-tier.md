# Visibility-First Security: Monitoring Stack, Abuse Prevention, and Free Tier Confidence

**date** 20 Mar 2026
**from** Human (project lead)

## Where We Are

The product is approaching a launch moment. The folder view is working. The gallery with thumbnails for PDFs and markdown is looking great. Features exist that could dramatically drive usage. But usage requires a free tier, and a free tier requires confidence that we can see what is happening and prevent abuse.

The blocker is not the features. It is visibility. Without monitoring, opening up free usage is a security risk. With monitoring, it becomes a growth strategy.

Additionally, AWS Startup credits ($1,000) have been secured, which removes the cost concern for spinning up monitoring infrastructure (Grafana, Elastic, etc.).

## The Principle: Security Enables Business

This is a case where security is not a blocker. Security is the enabler. The business wants to:

- Allow free usage (fingerprint-based, 5 credits per anonymous user)
- Share access tokens publicly (for demos and onboarding)
- Let anyone try the product without signing up
- Provide daily free credits for experimentation

Security makes this possible by providing the visibility to detect abuse, the controls to limit it, and the confidence to open the doors. Without security, the business cannot afford to be generous. With security, generosity becomes safe.

## What We Need to See

### Every Request

Every HTTP request hitting any SG/Send property should be visible:

- Who (IP, fingerprint, user agent, access token if present)
- What (path, method, parameters, file sizes)
- When (timestamp, request duration)
- Where (which endpoint: send, vault, gallery, tools, website)
- Result (status code, response size, errors)

### Every Bot

Identify and categorise automated traffic:

- Search engine crawlers (allow, track)
- AI crawlers (monitor, potentially restrict per robots.txt)
- Scrapers (detect, rate limit)
- Abuse bots (detect, block)
- Legitimate API clients (sg-send-cli, vault operations)

### Every User Journey

Correlate requests into sessions and journeys:

- Anonymous user visits website → browses → uploads a file → shares link
- Fingerprinted user returns → uses free credits → uploads → downloads
- Paying user uploads folder → gallery view → recipient downloads

Understanding journeys tells us where users drop off, what features they use, and where the product needs improvement.

### Cost Correlation

Map every request to a cost:

- Lambda invocation cost per request
- S3 storage cost per upload
- S3 bandwidth cost per download
- CloudFront bandwidth per page load

The goal: know exactly what each user session costs. Right now traffic is minimal (spending less than daily coffee). That is the right time to instrument, not when traffic is high and costs are surprising.

## The Monitoring Stack

Leverage the $1,000 AWS credits:

| Component | What It Provides | Source |
|---|---|---|
| CloudFront access logs | Every HTTP request to the CDN (all website traffic) | AWS native, free |
| S3 server access logs | Every read/write to storage buckets | AWS native, free |
| CloudWatch | Lambda metrics, API Gateway metrics, custom metrics | AWS native, included |
| CloudWatch Logs | Lambda function logs, application logs | AWS native, included |
| X-Ray | Request tracing across services | AWS native, free tier generous |
| Amazon Managed Grafana | Dashboard visualisation | AWS credits |
| Prometheus | Custom metrics from Lambda endpoints | Self-hosted or managed |
| Elastic (OpenSearch) | Log aggregation, search, analysis | AWS credits (Managed OpenSearch) |

### Phase 1: Turn Everything On

Before writing any custom code:

1. Enable CloudFront access logging (every request to every site)
2. Enable S3 server access logging (every storage operation)
3. Enable CloudWatch detailed monitoring on all Lambda functions
4. Enable X-Ray tracing on API Gateway and Lambda
5. Set up Amazon Managed Grafana, connect to CloudWatch
6. Create initial dashboards: requests/minute, errors/minute, bandwidth, cost estimate

This gives us visibility into every request hitting the infrastructure. No custom code. Just configuration.

### Phase 2: Correlation and Intelligence

1. Feed CloudFront logs into Elastic/OpenSearch for searchable log analysis
2. Add user fingerprinting (browser fingerprint for anonymous users)
3. Correlate fingerprints to sessions to journeys
4. Build abuse detection rules: too many uploads from one fingerprint, unusual download patterns, suspicious user agents
5. Add Prometheus custom endpoints (vault metadata, traffic stats, as specified in the Grafana brief)

### Phase 3: Automated Response

1. Rate limiting per fingerprint (X requests per day for free tier)
2. Automatic blocking of detected abuse patterns
3. Alerting: Grafana alerts when anomalies are detected
4. AppSec agent and SRE agent review alerts and adjust rules

## Free Tier Model

Once visibility is in place, the free tier becomes safe to launch:

| User Type | How Identified | What They Get |
|---|---|---|
| Anonymous (no token) | Browser fingerprint | 5 free credits per day (upload, download, view) |
| Guest (leaked demo token) | Demo access token | Access to demo vaults, galleries, folder views |
| Free tier (signed up) | Access token | Generous daily allowance, all features |
| Paying customer | Access token + credits | Full access, purchased credits |

The fingerprint-based free tier is the key unlock. Users can try the product without signing up. They get enough credits to upload a file, view a gallery, open a folder. Just enough to experience the product. If they want more, they sign up or buy credits.

### Why Fingerprint, Not Just IP

IP addresses are shared (NAT, VPN, corporate proxies). Fingerprinting (a combination of browser properties, screen size, timezone, installed fonts, etc.) provides a more stable identity for anonymous users. It is not perfect, but it is good enough to prevent trivial abuse while keeping the friction near zero.

### Why Practice Now

Traffic is minimal. This is the time to:

- Tune fingerprinting accuracy
- Calibrate rate limits
- Build abuse detection rules
- Understand cost per session
- Train the AppSec and SRE agents on real (if sparse) traffic patterns

When traffic increases, the systems are already in place and calibrated.

## Protecting the Sites

Harden all SG/Send properties against common attacks:

- **DDoS:** CloudFront provides baseline DDoS protection. AWS Shield Standard is free. Consider Shield Advanced if traffic warrants it.
- **Injection:** All user input sanitised. Gallery viewer in sandboxed iframe (already specified). No server-side rendering of user content.
- **Credential stuffing:** Access tokens are not passwords. But rate limit token validation attempts.
- **Scraping:** Detect and rate limit automated scraping of public galleries or folder views.
- **Abuse of free tier:** Rate limit per fingerprint. Flag anomalous usage patterns. Block after warning.

The goal: when we have a lot of traffic, we are already practiced at protecting it. Build the muscle now while the stakes are low.
