## Context

A meeting with a potential partner agreed on a next step: set up a team, build an MVP/POC, and test how the technology works in practice. The approach: create a generic, open source example that the partner can evaluate, test, and then add their commercial and proprietary layer on top.

This is the first instance of the "Build Your Company on SG/Send" ecosystem strategy being executed with a real partner.

## The Dev Pack

Create a dev pack (the same pattern used for internal projects) for the repo `sgit-ai__vault__research-documents`. This is a vault specifically designed for the research documents use case.

### What the Dev Pack Contains

The pack is a folder (or vault) that contains everything needed to start a new Claude Code session and begin building:

| Contents | Purpose |
|---|---|
| Agentic team role definitions | Explorer, Villager, Town Planner teams with roles tailored to the research domain |
| CLAUDE.md files per agent | Conductor, Architect, Developer, Researcher, QA, Sherpa, AppSec |
| Repo structure template | Folder structure for the project with README, docs, src, tests |
| CI pipeline setup | GitHub Actions workflow for testing and deployment |
| Vault integration guide | How to use SGit for storing and sharing research documents |
| Example workflows | Sample research document workflows (upload, analyse, share, version) |

### How It Works

1. Partner starts a Claude Code session pointed at the new repo
2. Claude Code reads the CLAUDE.md and role definitions
3. The agentic team is operational: Architect designs, Developer implements, QA tests
4. The team iterates on the research documents vault MVP
5. The vault itself is used for communication between human and agents during development

### Self-Documenting

The dev pack is both the tool and the demonstration. The partner sees:

- How the agentic team works in practice (because they are using it)
- How the vault works for collaboration (because the dev pack communicates via vault)
- How the SGit workflow operates (because the code is managed through SGit)

This is the strongest possible demo: not a slideshow about the technology, but the technology being used to build itself.

## The MVP Target

The research documents vault MVP should demonstrate:

- Create a vault for a research project
- Upload research documents (PDFs, papers, notes)
- Version control on the documents (edits, additions tracked with commits)
- Share the vault with collaborators (each gets their own branch)
- LLM analysis of the documents (summarise, extract key findings, cross-reference)
- Export as a zip or share via access key
- Branded UI (the partner's branding, not SG/Send branding)

The only thing the end user should need is an access key. From there, everything works.

## Using a Vault to Communicate

An interesting experiment: use a vault as the communication channel for this partnership. Instead of emails and shared Google Docs, the briefing materials, the MVP iterations, the feedback, and the decisions all live in a vault. The partner pulls updates. The human pushes briefs. Claude commits code.

This is another self-documenting move: the partnership itself runs on the technology being evaluated.

## Branding

The MVP should be brandable: the partner can apply their own branding (logo, colours, domain) without forking the code. This validates the `<sg-theme>` component and the theming pipeline from the component strategy brief.

For the initial demo, create a fictitious research brand to avoid using the partner's real identity before the agreement is finalised.
