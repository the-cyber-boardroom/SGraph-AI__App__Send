# SGit Visualisation: Interactive Branch and Commit Explorer

**version** v0.19.7
**date** 30 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Architect, Designer
**type** Dev brief

---

## What We Need

A Web Component that visualises SGit vaults: branches, commits, merges, and the relationships between them. Interactive, clickable, zoomable. This is the visual layer that makes the vault's internal structure comprehensible.

## Why Now

SGit branches are about to be used everywhere:

- **LLM feedback loops:** Each cycle is a commit, each exploration is a branch (from the one-shot feedback loops brief)
- **Multi-agent chat:** Each agent conversation could be a branch, with a merge when they consolidate
- **IFD versioning:** Each component version is a commit, the version tree is a branch graph
- **Dev-to-QA workflow:** Patches are commits on branches, reviews are merges
- **Vault collaboration:** Human and Claude each work on branches, merge results

Without a visualisation, these branch structures are invisible. You can query them via CLI, but you cannot see the shape of the work. The visualisation makes branches usable, not just available.

## The Component: sg-git-graph

### What It Renders

A git-style commit graph showing:

Each node is a commit. Each line is a branch. Merges show where branches rejoin. Abandoned branches show they were not merged.

### Interactive Features

| Feature | What It Does |
|---|---|
| Click a commit | Show commit details: message, timestamp, files changed, author/agent |
| Click a branch | Highlight the branch path, show branch metadata |
| Hover a commit | Preview the diff (what changed in this commit) |
| Zoom in/out | Navigate large graphs with many commits |
| Time slider | Scrub through the history, see the vault at any point in time |
| Filter by branch | Show only specific branches (e.g. only the LLM experiment branches) |
| Filter by author | Show only commits from a specific agent |

### Data Source

The component reads from:

- **SGit vault (live):** Connect to a vault, read the commit graph via the structure key (from the read-structure key brief). No content access needed, just metadata.
- **JSON export:** A static JSON file containing the commit graph (for documentation, offline viewing, or embedding in the QA site).
- **Programmatic:** JavaScript API to build the graph programmatically (for LLM thread visualisation where the "commits" are LLM interactions, not real SGit commits).

## Use Case: LLM Thread Visualisation

The same component visualises LLM conversation branches:

```
main thread    в”Җв”Җ[Q1]в”Җв”Җ[A1]в”Җв”Җ[Q2]в”Җв”Җ[A2]в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ[Consolidated]в”Җв”Җ
                              в”Ӯ                  вҶ—
agent-architect               в””в”Җв”Җ[Q2a]в”Җв”Җ[A2a]в”Җв”Җв”ҳ
                              в”Ӯ                вҶ—
agent-appsec                  в””в”Җв”Җ[Q2b]в”Җв”Җ[A2b]в”Җв”ҳ
```

The main thread asks a question. Two agents explore it in parallel (on branches). A consolidation step merges the results. The visualisation shows the structure of the multi-agent conversation.

Each "commit" in this view is an LLM interaction (prompt + response). The branch is the agent. The merge is the consolidation. The same sg-git-graph component renders both real SGit vaults and LLM conversation structures, because the underlying data model is the same: nodes, edges, branches, merges.

## Programmatic Generation

The component should support building graphs from code:

```javascript
const graph = new SGitGraph();

const main = graph.branch('main');
const c1 = main.commit('Initial question about Python patterns');
const c2 = main.commit('Refined question with context');

const archBranch = graph.branch('architect', c2);
archBranch.commit('Architecture perspective on the problem');
archBranch.commit('Proposed solution with diagrams');

const secBranch = graph.branch('appsec', c2);
secBranch.commit('Security analysis of the approach');

const merge = main.merge([archBranch, secBranch], 'Consolidated recommendation');

graph.render(document.getElementById('graph-container'));
```

This makes the component usable from LLM tools: the multi-agent chat system programmatically builds the graph as conversations proceed. The visualisation updates in real time.

## Implementation

Build on tools.sgraph.ai as a standalone IFD-versioned component. Use an existing graph rendering library (D3.js, or a lighter alternative) for the actual drawing. The component handles the data model and interaction; the library handles the rendering.

Start with the JSON data source (easiest to test with static data). Add the live SGit connection second. Add the programmatic API third.
