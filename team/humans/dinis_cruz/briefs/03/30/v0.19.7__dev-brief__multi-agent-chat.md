# Multi-Agent Chat Mode: Parallel Conversations, Async Workflows, and Offline Agents

**version** v0.19.7
**date** 30 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Architect, Designer
**type** Dev brief

---

## The Vision

Talk to multiple agents at the same time. Pick which agents you want, arrange them in the UI, ask the same question to three of them, have a fourth consolidate. Debate ideas. Compare answers. Run async tasks in the background. All powered by local models when offline, API models when online.

## The UI

### Agent Picker (Sidebar)

A panel listing all available agents, organised by team:

- Explorer team: Developer, Designer, Researcher
- Villager team: QA, Architect, AppSec
- Town Planner team: SRE, DevOps
- Support team: Librarian, Sherpa, Ambassador, Alchemist
- Custom: any agent the user defines

Each agent entry shows: name, role, avatar (generated by an image model if available), status (idle, busy, offline).

Click an agent to open a conversation panel. Click multiple agents to open multiple panels.

### Conversation Panels (SG Layout)

Each agent conversation is a panel in SG Layout. Panels can be:

- Side by side (compare two agent responses)
- Stacked (multiple agents in a column)
- Tabbed (switch between agents)
- Any SG Layout arrangement (fractal nesting, resizing, drag-and-drop)

Each panel is an independent chat window using the Chat component. Each panel has its own model selection (one agent might use Ollama locally, another might use Claude via API).

### Multi-Ask Mode

Send the same question to multiple agents simultaneously:

```
User types: "How should we structure the new website?"
Sends to: Ambassador, Architect, Designer (3 panels)
Each agent responds independently with their perspective
```

Then optionally: a Consolidator agent reads all three responses and produces a synthesis.

### Debate Mode

Agents can review each other's responses:

1. Ask all three agents the same question
2. Share Agent A's response with Agents B and C: "What do you think of this?"
3. Each agent critiques or builds on the others
4. The Consolidator summarises agreements and disagreements

This is where role-playing quality matters. Research which models do well at staying in character, maintaining a perspective, and constructively disagreeing.

## Sync vs Async Workflows

### Synchronous (Interactive)

The human asks, the agent responds, the human follows up. This is the standard chat flow. Good for exploration, quick questions, and interactive work.

### Asynchronous (Background)

The human defines a task, the agent runs in the background, the human checks results later. This is critical for:

- Multi-agent workflows (three agents analyse, fourth consolidates: takes minutes, not seconds)
- Code review tasks (agent reads an entire repo, produces a report)
- Semantic graph generation (agent processes the codebase, builds the ontology)
- Diff analysis (agent compares two versions, documents changes)

The UI needs to support this: show that a task is running, show progress if available, notify when complete. The human does other work while agents process in the background.

### Cost Implications

Async workflows on local models (Ollama) are essentially free. The same workflows on API models (Claude, GPT) cost tokens. The UI should show estimated cost before running an async task on an API model.

Local models make async experimentation sustainable. Run five agents overnight on Ollama: zero cost. Run the same on Claude API: significant cost. Use local for experimentation, API for production-quality outputs.

## Agent Context: What Gets Loaded

When an agent is selected, the system loads:

| Context | Source | Purpose |
|---|---|---|
| Role definition | From Librarian-maintained pages (website, QA site, or library site) | Who the agent is, what they do, how they think |
| Team definition | Same source | Which team they belong to, how they collaborate |
| Latest status | From scheduled Librarian run | What is currently happening in the project |
| Relevant files | From vaults or repos | Materials the agent needs for the current task |

The Librarian maintains these context pages. They are published on the website (or the library site). The UI fetches them when loading an agent. This means updating an agent's context is: update the page, the next conversation uses the new context.

### One-Shot vs Multi-Turn

For each agent conversation, the user can choose:

- **One-shot:** Full context assembled, sent in one request, response received. No history. Clean and focused. (The pattern from the one-shot LLM dev environment brief.)
- **Multi-turn:** Standard chat with growing history. Better for exploration and follow-up. Risk of context window overflow on long conversations.

The default should be configurable per agent and per task.

## Saving and Publishing Conversations

All conversations should be saveable:

- Save to local storage (persist between sessions)
- Save to a vault (share with others, archive)
- Publish as a folder on SG/Send (viewable via Browse, with the JSON page layout)

Conversations with no secrets become valuable content: "here is how the Architect and the AppSec agent debated the vault security model." Publishable. Referenceable. Part of the project history.

## Model Research Needed

### Which Models Work Best For:

| Task | What We Need | Models to Test |
|---|---|---|
| Role playing / staying in character | Agents that maintain a perspective | Test various Ollama models |
| Conversational quality | Natural back-and-forth | Mistral, Llama, Qwen |
| Code analysis | Understanding and reviewing code | DeepSeek Coder, CodeLlama |
| Consolidation | Synthesising multiple inputs | Larger models (Mixtral, Llama 70B) |
| Fast responses | Quick one-shot answers | Smaller models (Phi, Gemma) |
| Browser-only (WASM) | Runs entirely in the browser | Explore available WASM models |

### Publish Model Rankings

Create a page (on the website or as a shared vault) with model rankings per task. Update regularly. This is useful internally and externally: anyone exploring local models benefits from curated recommendations.

## Connection to Existing Architecture

- **SG Layout:** Each agent is a panel. Arrangement is flexible. Fractal nesting for complex multi-agent setups.
- **Model Chooser:** The component from the chat improvements brief. Each panel uses it independently.
- **Broadcast communication:** When one agent produces output, broadcast it to other agents for review.
- **sg-llm component:** The one-shot execution layer underneath each conversation.
- **Scheduled tasks:** Async agent workflows can run on Claude scheduled tasks, with results pushed to vaults.
- **Vault communication:** Results saved to vaults for persistence and sharing.
