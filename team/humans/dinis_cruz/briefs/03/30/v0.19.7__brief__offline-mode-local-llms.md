# Offline Mode: Local LLMs, Code Comments as Communication, and Engineered Focus

**date** 30 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Architect, Conductor
**type** Brief

## The Discovery

A productive plane session revealed something important: being offline is not a limitation. It is a feature. The constant interactivity of online work (delegating to agents, checking results, triggering new tasks, reviewing outputs) is operationally effective but strategically expensive. Sometimes you need two hours of uninterrupted focus.

The technology stack already supports offline work (Ollama for local LLMs, vault clones for local data, browser-based tools). What needs engineering is the deliberate practice of going offline.

## Engineered Offline Time

This is not about being forced offline on a plane. It is about choosing to go offline:

- Two hours of focused work with no internet, no delegating, no checking
- Local LLMs for questions and coding assistance
- Vaults cloned locally for all needed materials
- Browser-based tools running entirely client-side

The plane session proved this works. Now make it a regular practice: engineer offline blocks into the workday. The scheduled tasks (Librarian, QA) run while you are offline. When you come back online, the results are waiting.

## Code Comments as Agent Communication

A pattern that emerged during offline coding: leaving comments in the code that are addressed to an agent, not to a human reader.

### How It Works

While coding offline, add comments that capture observations, requests, or concerns:

```python
# @librarian: this function duplicates logic from upload_engine.py line 45,
#   consider refactoring into a shared utility
def process_chunk(data):
    ...

# @qa: add a test for this edge case - what happens when chunk_size > file_size?
if chunk_size > remaining:
    ...

# @architect: this pattern of retrying with exponential backoff appears in 4 places,
#   should we create a shared retry decorator?
for attempt in range(max_retries):
    ...

# @dev: the performance of this loop is O(n^2), fine for small files but
#   will be a problem for large vaults. Flag for optimization in v0.3.2
for file in files:
    for other in files:
        ...
```

### Processing the Comments

When back online (or as a scheduled task), an agent reads the codebase and processes the `@agent` comments:

1. **Captures** each comment with its file location and context
2. **Routes** to the appropriate agent (Librarian, QA, Architect, Developer)
3. **Creates** tasks, briefs, or issues from the comments
4. **Optionally removes** the comment from the code (replacing with a reference ID so it can be found later)
5. **Updates** the project management structure

This is a new communication channel between the human and the agents. The human writes a comment while deep in the code. The agent processes it later. The code stays clean (comments are removed after processing). The tasks are captured in the proper structure.

### What to Call This Role

The agent that processes code comments is a "Librarian for code." It reads the codebase, finds the annotations, routes them, and maintains the index. It could be:

- An extension of the existing Librarian role (code is just another type of document to catalogue)
- A new role: the **Code Steward** (maintains code quality through comment processing, pattern detection, and cross-referencing)
- An extension of the Ontologist (the comments feed into the semantic graph of the codebase)

For now, it does not need a new role. It is a task the Librarian can perform during the daily scheduled run.

## Local LLM Stack

### What Worked on the Plane

- Ollama running locally with the models available
- The LLM chat tool we built recently
- Coding questions answered locally (slower than Claude, but functional)
- Sonnet-level models produce decent code, sometimes needing cleanup

### What Needs Improving

- **Model quality:** Explore which open source models perform best for coding tasks. Mistral, DeepSeek Coder, CodeLlama, Qwen. Test each and document quality levels.
- **Model freshness:** Upgrade to the latest model versions before going offline
- **Async tasks:** Some LLM operations (long analysis, code review of large files) can run in the background. The user does not need to wait. Start the task, do other work, check results later.
- **Performance benchmarking:** For each model, measure: time to first token, tokens per second, quality of code output, quality of analysis. Document so the human knows which model to use for what.

### The Security Model: Browser-Based Everything

Even offline, the security model remains: everything runs in the browser. The local LLM is called from the browser (Ollama exposes an HTTP API on localhost). The tools are Web Components. The vaults are decrypted client-side.

Future exploration: can we run isolated code execution (Python, Node) in the browser too? WASM-based Python (Pyodide) is already proven. This would mean the offline environment is fully self-contained in a browser tab: LLM, code execution, vault access, tool suite.

## Vault as the Offline Sync Point

The pattern for offline work:

1. **Before going offline:** Pull all needed vaults locally. Clone repos. Download WASM modules (sg-wasm caches them).
2. **While offline:** Work in vaults. Make changes. Commit locally. Leave `@agent` comments in code. Use local LLMs for assistance.
3. **When back online:** Push vaults. The Librarian processes `@agent` comments. The QA smoke tests verify nothing broke. The scheduled tasks catch up on whatever accumulated.

The vault is the sync point. Everything flows through it. Online or offline, the workflow is the same. The only difference is whether the push/pull happens immediately or when connectivity returns.
