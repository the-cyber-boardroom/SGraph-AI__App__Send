## The Core Idea

Our one-shot prompt approach is different from what most people mean by "one-shot." Most people mean: send one prompt, get one answer. Our approach is: curate reality, send it all at once, get the answer, then use the answer to improve the reality for the next prompt.

This is not a chat with growing history. It is a series of one-shot prompts where each prompt is constructed from the refined output of all previous prompts. The context does not grow exponentially. It stays constant (or grows slowly) because each cycle prunes, consolidates, and focuses.

## The Problem with Standard Chat

In a standard LLM chat:

```
Message 1: 500 tokens sent, 500 tokens received
Message 2: 1,500 tokens sent (500 + 500 + 500), 400 tokens received
Message 3: 2,900 tokens sent, 600 tokens received
Message 4: 4,500 tokens sent, 500 tokens received
...
Message 10: ~15,000 tokens sent, response quality degrading
```

Every message sends the entire history. The token count grows linearly (at best) or exponentially (when responses are long). The LLM has to process increasingly large contexts, much of which is stale (early exploration, wrong paths, superseded decisions).

## The One-Shot Feedback Loop

```
Cycle 1:
  Bundle = [system prompt + context + task]
  → Send to LLM → Get response
  → Extract: facts, decisions, code samples, key findings
  → Prune: remove exploration, wrong paths, superseded info
  → Updated Bundle = [system prompt + pruned context + extracted findings + new task]

Cycle 2:
  Bundle = Updated Bundle from Cycle 1
  → Send to LLM → Get response
  → Extract + Prune → Updated Bundle

Cycle 3:
  Bundle stays roughly constant size
  But contains increasingly refined, focused information
```

The key difference: instead of appending every message to the history, we use the LLM (or a second LLM) to analyse each exchange and produce a condensed, improved context for the next cycle.

### What Gets Extracted

After each LLM response, extract:

- **Facts discovered:** "The upload component uses AES-256-GCM encryption"
- **Decisions made:** "We will use chunked upload with 1MB chunks"
- **Code produced:** The final working version (not the three failed attempts)
- **Questions resolved:** Remove the question, keep the answer
- **New questions raised:** Add to the task for the next cycle

### What Gets Pruned

- Exploration paths that did not lead anywhere
- Earlier versions of code that were superseded
- Questions that were answered (the answer stays, the question goes)
- Conversational back-and-forth that added no information
- Duplicate or redundant context

### The Size Profile

Instead of linear growth:

```
Standard chat:  500 → 1500 → 2900 → 4500 → 6500 → ...
One-shot loops: 500 → 600  → 550  → 620  → 580  → ...
```

The context stays roughly constant. It grows a little when new information is added. It drops when a rabbit hole resolves and the exploration can be condensed to just the conclusion. The graph has occasional drops (moments of understanding where large chunks of context become a single sentence).

## Token Visualiser Component

Build a component that shows token consumption visually:

- **Total tokens per request:** How big is the bundle?
- **Breakdown by section:** System prompt (X tokens), context (Y tokens), history (Z tokens), reference materials (W tokens), images (V tokens), tool definitions (U tokens)
- **Growth over time:** Graph showing how the bundle size changes across cycles
- **Comparison:** Standard chat token growth vs one-shot loop token growth
- **Cost estimate:** Based on the model's pricing, how much does each cycle cost?

This component makes the invisible visible. Users (and the human managing agents) can see when context is bloating and when pruning is effective.

## SGit Branches for Bundle Management

This is the breakthrough connection: **use SGit branches to manage bundle versions.**

### The Bundle as a Vault

Each bundle (system prompt + context + extracted findings + task) is a set of files in a vault:

```
bundle/
  system-prompt.md      The system prompt
  context.md            Background information, project description
  findings.md           Extracted facts and decisions from past cycles
  code-samples/         Working code from past cycles
  reference/            Documents, specs, images needed
  task.md               Current task
  metadata.json         Cycle number, token count, model used
```

### Branches for Exploration

When the conversation branches (exploring two different approaches), create an SGit branch:

```
main           → The primary conversation line
experiment-a   → Exploring approach A
experiment-b   → Exploring approach B
```

If experiment-a works, merge it back to main. If experiment-b fails, abandon the branch. The full history of the exploration is preserved in the branch, but the main line only has the successful result.

This gives:

- **Full provenance:** Every version of the bundle is a commit. Every branch is an exploration. The entire history is traceable.
- **Rollback:** If a pruning step removed something important, roll back to the previous commit.
- **Parallel exploration:** Two approaches tested simultaneously on different branches.
- **Comparison:** Diff two bundle versions to see what changed between cycles.

### Branches Replace History

In a standard chat, the history is a linear sequence. In SGit-branched bundles, the "history" is a graph. Each cycle is a commit. Each exploration is a branch. The bundle at any point in time is the state of the vault at that commit.

This is more powerful than linear history because it captures the structure of the exploration, not just its sequence.

## Eval / Benchmark Tests

For critical prompts (the ones used repeatedly, like agent role definitions or analysis templates), create deterministic tests:

- Same input → expected output (or expected properties of the output)
- Refactor the input → verify the output quality is maintained
- Reduce the input (fewer tokens) → verify the output quality is maintained

These tests validate that pruning did not lose essential context. If a pruned bundle produces worse output than the full bundle on the same test, the pruning was too aggressive.

This is not full eval infrastructure. It is targeted tests for the bundles that matter most. Keep the test cases alongside the bundle in the vault.

## The Two-LLM Pattern

For the pruning step, consider using a different (cheaper, faster) LLM:

- **Primary LLM** (Claude, GPT-4): Answers the actual question, produces the response
- **Pruning LLM** (smaller model, even local Ollama): Analyses the exchange, extracts facts, prunes context, produces the updated bundle

The pruning task is simpler than the primary task. It does not need frontier-model quality. A smaller model can do it faster and cheaper. This keeps the cost of the feedback loop low while maintaining quality on the primary output.

## Visual Workflow Designer

As these feedback loops become more complex (branch here, prune here, send to three models, consolidate), a visual workflow designer becomes useful. Research existing visual workflow components (node-and-edge editors, flow builders) that we can integrate. Do not build from scratch.
