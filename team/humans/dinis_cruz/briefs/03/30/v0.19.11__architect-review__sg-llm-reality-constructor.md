**Version:** v0.19.11
**Date:** 29 March 2026
**Role:** Architect (Explorer team)

**Status:** Pre-implementation — ready for review

## 1. The Core Idea: Controlling the Reality

From the article "One-Shot LLM: Engineering AI Requests from Dollars to Fractions of a Cent":

> When I send a one-shot request via API, I control the model's entire universe. [...] Everything it knows about the task, the context, the constraints, and the expected output is because I deliberately included it. Everything the model doesn't need? Simply absent.

The central component is not a prompt textarea — it is a **reality constructor**. The user visually builds the model's entire universe for each request:

- A **system prompt** block that defines the agent's role
- **Context blocks** — documents, code, schemas, data (each showing its token count)
- **Constructed history** — hand-crafted user/assistant message pairs that prime the model's understanding ("fake" conversation that designs the model's memory)
- The **actual question**
- **Live cost estimate** updating as blocks are added/removed

Each block is explicitly chosen. The total token cost is visible before sending. The user is engineering a reality, not chatting.

## 2. The One-Shot Flow

```
┌─────────────────────────────────────────────────────────────────────┐
│                        THE CONSTRUCTED REALITY                      │
│                                                                     │
│  ┌─ System Prompt ──────────────────────────────────── 180 tokens ─┐│
│  │  You are an expert infographic designer...                      ││
│  └─────────────────────────────────────────────────────────────────┘│
│                                                                     │
│  ┌─ Context Block: "Architecture Doc" ──────────────── 850 tokens ─┐│
│  │  <attached: architecture-overview.md>                           ││
│  └─────────────────────────────────────────────────────────────────┘│
│                                                                     │
│  ┌─ Constructed History ────────────────────────────── 420 tokens ─┐│
│  │  👤 User: "The page has a header with nav, a hero section..."   ││
│  │  🤖 Asst: "I understand — a standard marketing page with..."    ││
│  │  👤 User: "The key data points are revenue ($2.3M), growth..."  ││
│  │  🤖 Asst: "Got it. Three primary metrics with a timeline..."    ││
│  └─────────────────────────────────────────────────────────────────┘│
│                                                                     │
│  ┌─ Question ───────────────────────────────────────── 120 tokens ─┐│
│  │  Create an executive infographic from this data.                ││
│  └─────────────────────────────────────────────────────────────────┘│
│                                                                     │
│  Total: 1,570 tokens in ≈ $0.003         [Send]  [Save as Bundle] │
└─────────────────────────────────────────────────────────────────────┘
         │
         │  llm:send  (the complete messages array)
         ▼
┌─────────────────┐     llm:request-start      ┌──────────────┐
│  sg-llm-request │ ─────────────────────────▶  │ sg-llm-stats │
│  (fetch engine) │     llm:request-chunk       │ (metrics)    │
│                 │ ─────────────────────────▶  └──────────────┘
│                 │     llm:request-chunk       ┌──────────────┐
│                 │ ─────────────────────────▶  │ sg-llm-output│
│                 │                             │ (streaming)  │
│                 │     llm:request-complete    └──────────────┘
│                 │ ─────────────────────────▶  ┌──────────────┐
└─────────────────┘                             │ sg-llm-debug │
                                                │ (inspector)  │
                                                └──────────────┘
```

**The messages array** is not `[{role:'user', content:'hello'}]`. It is:

```js
messages: [
    { role: 'system',    content: systemPrompt },           // Block 1
    { role: 'user',      content: contextDoc },             // Block 2 (context)
    { role: 'user',      content: 'The page has a...' },    // Block 3 (constructed)
    { role: 'assistant', content: 'I understand...' },      // Block 3 (constructed)
    { role: 'user',      content: 'The key data...' },      // Block 3 (constructed)
    { role: 'assistant', content: 'Got it. Three...' },     // Block 3 (constructed)
    { role: 'user',      content: actualQuestion },          // Block 4 (the real ask)
]
```

The user designs this array. The reality constructor makes each block visible, draggable, measurable (token count), and editable. The constructed history pairs are the part that "surprises people" — fabricated exchanges where the user wrote both sides, priming the model's comprehension before the actual question.

## 3. The Component Family

Grouped under `components/llm/` as a **namespace folder** — this prevents the root `components/` directory from growing unmanageably large. But each component within `llm/` has its **own independent IFD versioning**. They do NOT share a version. `sg-llm-connection` can be at v0.2.0 while `sg-llm-debug` is still at v0.1.0.

This is critical for offline/pinned tools: a tool that only needs connection + request + output copies those three components at their specific major versions. It does not copy the debug inspector or the infographic renderer.

Surgical override files use **version-stamped filenames** with a double-dash separator (`sg-llm-reality--v0.1.1.js`), following the convention in `v1.3.0__ifd__surgical-overrides-version-stamped-filenames.md`.

### 3.1 Each Component's manifest.json Declares Its Own Dependencies

```json
{
    "name":        "sg-llm-reality",
    "version":     "0.1.0",
    "description": "One-shot LLM reality constructor — build the model's complete universe visually",
    "files": [
        "sg-llm-reality.js",
        "sg-llm-reality.css"
    ],
    "exports": {
        "sg-llm-reality.js": ["SgLlmReality"]
    },
    "dependencies": {
        "sg-llm-events": "/components/llm/sg-llm-events/v0/v0.1/v0.1.0/sg-llm-events.js"
    },
    "compatibleWith": {
        "sg-llm-events":     ">=0.1.0",
        "sg-llm-request":    ">=0.1.0",
        "sg-llm-connection":  ">=0.1.0",
        "sg-layout":          ">=0.1.0"
    }
}
```

Each component declares:
- **`dependencies`** — what it imports (always includes sg-llm-events)
- **`compatibleWith`** — which versions of sibling components it has been tested with

A tool copying `sg-llm-reality/v0/v0.2/v0.2.0/` for offline use reads the manifest, sees it depends on `sg-llm-events >= 0.1.0`, and copies that too. It does NOT need to copy `sg-llm-debug` or `sg-llm-infographic` unless the tool uses them.

### 4.1 sg-llm-events.js — Shared Event Constants

The contract between all components. Frozen at v0.1.0. Additions only.

```js
export const SGL_LLM = Object.freeze({
    // --- Connection ---
    CONNECTED:           'llm:connected',           // { provider, model, baseUrl }
    DISCONNECTED:        'llm:disconnected',         // {}
    MODEL_CHANGED:       'llm:model-changed',        // { model }
    MODELS_LOADED:       'llm:models-loaded',        // { models[] }

    // --- Reality construction ---
    REALITY_CHANGED:     'llm:reality-changed',      // { messages[], tokenCount, estimatedCost }
    BLOCK_ADDED:         'llm:block-added',           // { block }
    BLOCK_REMOVED:       'llm:block-removed',         // { blockId }
    BLOCK_REORDERED:     'llm:block-reordered',       // { blockIds[] }

    // --- Request lifecycle ---
    SEND:                'llm:send',                  // { messages[], model, provider }
    CANCEL:              'llm:cancel',                // {}
    REQUEST_START:       'llm:request-start',         // { provider, model, streaming, tokenEstimate }
    REQUEST_CHUNK:       'llm:request-chunk',         // { chunk, accumulated }
    REQUEST_COMPLETE:    'llm:request-complete',       // { content, promptTokens, completionTokens, cost, latencyMs, model, finishReason }
    REQUEST_ERROR:       'llm:request-error',          // { error, status, provider }
    REQUEST_CANCEL:      'llm:request-cancel',         // {}

    // --- Output ---
    OUTPUT_READY:        'llm:output-ready',           // { content, format }
    OUTPUT_COPIED:       'llm:output-copied',          // {}

    // --- Stats ---
    STREAMING_CHANGED:   'llm:streaming-changed',      // { streaming: bool }

    // --- Infographic ---
    INFOGRAPHIC_READY:   'llm:infographic-ready',      // { svg, width, height }
    INFOGRAPHIC_EXPORTED:'llm:infographic-exported',   // { format }

    // --- Memory ---
    MEMORY_UPDATED:      'llm:memory-updated',          // { content, tokenCount }

    // --- Attachments ---
    ATTACHMENT_ADDED:    'llm:attachment-added',          // { attachment: { id, type, name, size, dataUrl?, textContent? } }
    ATTACHMENT_REMOVED:  'llm:attachment-removed',        // { id }
    ATTACHMENTS_CLEARED: 'llm:attachments-cleared',       // {}

    // --- Bundles ---
    BUNDLE_SAVE_REQUESTED: 'llm:bundle-save-requested', // {}
    BUNDLE_SAVED:          'llm:bundle-saved',           // { bundle }

    BUNDLE_LOADED:         'llm:bundle-loaded',          // { bundle }
    BUNDLE_DELETED:        'llm:bundle-deleted',         // { bundleId }
    BUNDLE_LIST_CHANGED:   'llm:bundle-list-changed',    // {}
```

### 4.2 sg-llm-connection (.js + .css)

**Tag:** `<sg-llm-connection>`

**Responsibility:** Provider selection, API key entry (masked), model listing/favourites, connection testing, key persistence in localStorage.

**UI:** Settings panel — provider tabs (OpenRouter, Ollama, Direct API), key input, model dropdown with star-to-favourite, "Test" button with ✓/✗ indicator.

**Emits:** `CONNECTED`, `DISCONNECTED`, `MODEL_CHANGED`, `MODELS_LOADED`

**Consumes:** Nothing — pure source (user input only).

**State persisted:** Provider, API key, model, favourites, base URL — all in localStorage under `sg-llm-config`. Key NEVER sent to any `*.sgraph.ai` domain.

### 4.3 sg-llm-reality (.js + .css) — THE REALITY CONSTRUCTOR

**Tag:** `<sg-llm-reality>`

**Responsibility:** The user builds the model's complete universe for each one-shot request. This is the central component.

**UI:** A vertical list of **blocks** that the user adds, removes, reorders, and edits. Each block has a type badge, a collapsible editor, and a token count:

```
┌─ ⚙️ System ──────────────────────────────────── 180 tok ─┐
│  You are an expert infographic designer who produces...   │
│  [Edit] [Remove]                                          │
└───────────────────────────────────────────────────────────┘
┌─ 📎 Context: "architecture.md" ──────────────── 850 tok ─┐
│  (attached text, collapsed by default, click to expand)   │
│  [Edit] [Remove]                                          │
└───────────────────────────────────────────────────────────┘
┌─ 💬 Constructed History ─────────────────────── 420 tok ─┐
│  👤 "The page has a header with nav, a hero section..."   │
│  🤖 "I understand — a standard marketing page with..."    │
│  👤 "The key data points are revenue ($2.3M)..."          │
│  🤖 "Got it. Three primary metrics with a timeline..."    │
│  [Add pair] [Edit] [Remove]                               │
└───────────────────────────────────────────────────────────┘
┌─ 🧠 Memory ─────────────────────────────────── 380 tok ─┐
│  - Building folder viewer using sg-tree                   │
│  - Using Shadow DOM + CSS custom properties               │
│  - Decided on flat row rendering for scroll performance   │
│  - Added expand/collapse with aria-expanded               │
│  - Next: multi-select (Shift+click range)                 │
│  [Update Memory ↻] [Edit] [Remove]                        │
└───────────────────────────────────────────────────────────┘
┌─ ❓ Question ────────────────────────────────── 120 tok ─┐
│  Create an executive infographic from this data.          │
│  📷 2 images attached (from sg-llm-attachments)           │
│                                                           │
└───────────────────────────────────────────────────────────┘

[+ System] [+ Context] [+ History] [+ Memory] [+ Question]

Total: 1,950 tokens + 2 images ≈ $0.006 (claude-sonnet-4)    [▶ Send] [↻ Update Memory]
```

**Block types:**

| Type | Role in messages array | UI |
|------|----------------------|-----|
| **System** | `{ role: 'system', content: ... }` | Single textarea. One per reality (first block). |
| **Context** | `{ role: 'user', content: '--- CONTEXT: {label} ---\n' + content }` | Text area. Label + content. Collapsible. sg-llm-attachments creates Context blocks automatically when text files are dropped/pasted. |
| **History** | Array of `{ role: 'user', content }, { role: 'assistant', content }` pairs | Paired textareas (user says / assistant says). Add/remove pairs. |
| **Memory** | `{ role: 'user', content: '--- SESSION MEMORY ---\n' + content }` | LLM-managed text block. "Update Memory" sends Mode 2 request; response replaces this block's content. User CAN edit manually. |
| **Question** | `{ role: 'user', content: ... }` or multimodal `[{ type: 'text', text }, { type: 'image_url', image_url: { url } }, ...]` | Textarea. When sg-llm-attachments has images, the Question shows a badge ("📷 2 images attached") and the messages array content becomes a multimodal array. |

**Relationship with sg-llm-attachments:**

Within a single sg-layout instance, sg-llm-reality assumes there is at most one `<sg-llm-attachments>` component. When assembling the messages array:

1. **Text attachments** — sg-llm-reality listens for `ATTACHMENT_ADDED` events. When a text file is attached, reality automatically creates a Context block with the file name as label and the text content as body. The user sees it appear as a block they can edit, reorder, toggle, or remove.

2. **Image attachments** — sg-llm-reality tracks image attachments from `ATTACHMENT_ADDED` events. When building the messages array for `SEND`, if images are present, the Question block's content becomes a multimodal content array instead of a plain string. The Question block shows a badge ("📷 2 images attached") so the user sees the images are included.

3. **Token count** — reality's total includes text attachment tokens (from the Context blocks they created) and notes image count separately ("1,950 tokens + 2 images").

This means attachments is the **cache and input surface** (owns file data, handles UX for drop/paste/browse), and reality is the **assembler** (reads from attachments, builds the messages array, shows the user how everything fits together).

**Key features:**

- **Drag to reorder** blocks (context before/after history changes the model's comprehension)
- **Token count per block** — estimated via `text.length / 4` for instant feedback, refined via tiktoken if loaded
- **Toggle blocks on/off** — disable a block without deleting it (it's excluded from the messages array but stays in the reality for later re-enabling)
- **Attachment-aware** — text attachments auto-create Context blocks; image attachments inject into Question as multimodal content; token totals reflect attached content
- **Live cost estimate** — total tokens × model pricing + image count (from connection's selected model)
- **Two send modes** — "Send" (Mode 1: Build, full reality) and "Update Memory" (Mode 2: condense memory block via LLM). See Section 8.
- **Save/load reality** — serialise all blocks to JSON for reuse. Bundles (Section 9) extend this with full state capture + fork tree.
- **Templates** — pre-built realities for common tasks (infographic, summarise, code review, component development). User selects a template, reality populates with placeholder blocks.

**Emits:** `REALITY_CHANGED` (on every edit — the messages array + token count + cost + image count), `SEND` (when user clicks Send — the fully assembled messages array, which may contain multimodal content arrays + mode), `CANCEL`, `BLOCK_ADDED`, `BLOCK_REMOVED`, `BLOCK_REORDERED`, `MEMORY_UPDATED` (after Mode 2 response replaces memory block)

**Consumes:** `CONNECTED` / `MODEL_CHANGED` (to show model name, update cost estimate, check image support), `REQUEST_START` / `REQUEST_COMPLETE` / `REQUEST_ERROR` (to toggle send/cancel state), `BUNDLE_LOADED` (restores all blocks from a loaded bundle), `ATTACHMENT_ADDED` / `ATTACHMENT_REMOVED` / `ATTACHMENTS_CLEARED` (text → Context blocks, images → Question multimodal content)

### 4.4 sg-llm-request (.js only — headless)

**Tag:** `<sg-llm-request>`

**Responsibility:** The only component that makes fetch calls to external APIs. Receives a messages array, calls the provider, emits response events. Supports streaming (SSE) and non-streaming. Abort support.

**UI:** None (headless). Can optionally show a minimal status dot (idle/sending/error) when placed in a visible panel.

**Provider protocols supported:**

| Provider | Endpoint | Auth | Message format |
|----------|----------|------|----------------|
| OpenRouter | `/v1/chat/completions` | `Bearer sk-or-...` | OpenAI-compatible messages array |
| Ollama | `/api/chat` | None | Ollama messages format |
| Direct Anthropic | `/v1/messages` | `x-api-key` | Anthropic messages format |
| Direct OpenAI | `/v1/chat/completions` | `Bearer sk-...` | OpenAI messages array |

**Critical:** sg-llm-request receives the messages array exactly as constructed by sg-llm-reality. It does NOT modify, reorder, or inject into the array. The reality constructor owns the model's universe. The request engine is a dumb pipe.

**Multimodal handling:** When a message's `content` field is an array (not a string), sg-llm-request treats it as multimodal content and formats it per provider:

| Provider | Text-only message | Multimodal message (text + images) |
|----------|------------------|-----------------------------------|
| OpenRouter / OpenAI | `{ role: 'user', content: 'text' }` | `{ role: 'user', content: [{ type: 'text', text: '...' }, { type: 'image_url', image_url: { url: 'data:...' } }] }` |
| Ollama | `prompt: 'text'` | `prompt: 'text', images: ['base64...']` (strips data URL prefix) |
| Anthropic | `{ role: 'user', content: 'text' }` | `{ role: 'user', content: [{ type: 'text', text: '...' }, { type: 'image', source: { type: 'base64', media_type: '...', data: '...' } }] }` |

sg-llm-request detects `Array.isArray(message.content)` and transforms accordingly. The reality constructor always builds the canonical format (OpenAI-style content array); the request engine adapts it per provider. This is the same pattern as the workspace's `_buildUserContent()`.

**Emits:** `REQUEST_START`, `REQUEST_CHUNK`, `REQUEST_COMPLETE`, `REQUEST_ERROR`, `REQUEST_CANCEL`

**Consumes:** `SEND` (from reality constructor — includes the messages array, which may contain multimodal content arrays), `CANCEL`, `CONNECTED` (stores provider config)

### 4.5 sg-llm-output (.js + .css)

**Tag:** `<sg-llm-output>`

**Responsibility:** Render the LLM response. Streaming display (chunks appear live). Markdown rendering. Code block syntax highlighting. Copy to clipboard. Download as file.

**UI:** Response area with streaming text, markdown toggle, copy/download buttons.

## 8. The Memory Block + Two-Mode Execution

From the dev environment brief (`v0.19.7__dev-brief__oneshot-llm-dev-environment.md`), the one-shot pattern has a problem: how do you carry learning forward across iterations without growing an unbounded conversation history?

The answer is a **Memory block** — a new block type in sg-llm-reality that the LLM itself rewrites between iterations.

### 8.1 The Memory Block Type

Added to sg-llm-reality's block types:

| Type | Role in messages array | UI | Persistence |
|------|----------------------|-----|-------------|
| **System** | `{ role: 'system', content }` | Single textarea. First block. | Saved in bundle |
| **Context** | `{ role: 'user', content: '--- CONTEXT: {label} ---\n...' }` | Text/file drop. Collapsible. | Saved in bundle (with `vault://` ref if from vault) |
| **History** | Array of user/assistant pairs | Paired textareas. | Saved in bundle |
| **Memory** | `{ role: 'user', content: '--- SESSION MEMORY ---\n...' }` | **LLM-managed** text block. Read-only between iterations (user can edit manually but normally doesn't). | Saved in bundle. Evolves across iterations. |
| **Question** | `{ role: 'user', content }` | Textarea. Last block. | Saved in bundle |

The Memory block contains accumulated decisions, preferences, and patterns from the current session. Example:

```
--- SESSION MEMORY ---
- Building a folder viewer component using sg-tree
- Using Shadow DOM with CSS custom properties for theming
- Decided on flat row rendering (not nested divs) for scroll performance
- Added expand/collapse, using aria-expanded attribute
- Keyboard nav: arrows + j/k working, Enter selects
- Next: need to add multi-select (Shift+click range)
- User prefers compact token counts, not verbose explanations
```

### 8.2 Two Execution Modes

sg-llm-reality supports two send modes, selectable via the Send button:

**Mode 1: Build** (default) — assembles the full reality (system + context + memory + history + question) and sends it as a one-shot request. The response appears in sg-llm-output. The memory is NOT automatically changed.

**Mode 2: Update Memory** — sends the current memory block + the latest question/response pair to the LLM with a meta-prompt: "Consolidate this memory. Remove stale decisions. Add new learnings from the latest interaction. Keep it under 500 tokens." The response replaces the memory block content.

This prevents exponential growth. The memory is actively managed, not passively accumulated. After 20 iterations, the memory is still ~500 tokens — not 20× the original context.

**Events for two-mode execution:**

```js
// Mode 1: normal send (existing SEND event)
llm:send  { messages[], model, provider, mode: 'build' }

// Mode 2: memory update
llm:send  { messages[], model, provider, mode: 'memory-update' }
```

sg-llm-request doesn't care about modes — it receives a messages array and calls the provider. The mode distinction is in sg-llm-reality (which assembles different messages arrays for each mode) and sg-llm-output (which routes the response: Mode 1 renders in the output panel, Mode 2 replaces the memory block).

### 9.2 What Made Bundles Work Well

From the debrief (`v0.11.21__debrief__workspace-layout-bundles-and-team-docs.md`) — "The Day the Workspace Learned to Remember":

1. **Complete state capture.** Not just the prompt — everything: all panel contents, all include flags, all UI state, screenshots. When you load a bundle, you're back to EXACTLY where you were.

2. **`parent_id` enables fork trees.** Every bundle records which bundle was active when it was saved. This creates a tree structure: branch from any point, try a different approach, compare results. This is the compression path made visible — you can see your Genesis bundle ($0.50) next to your Product bundle ($0.003) and trace the evolution.

3. **`vault://` references.** Context blocks record which vault file they came from (`vault:///docs/architecture.md`). When you load a bundle, the system can re-fetch fresh content from the vault instead of using stale inline content.

4. **Auto-save option.** The workspace can automatically save a bundle after every LLM response. This creates a full audit trail without any user action.

5. **`tree.json` index.** A lightweight index file listing all bundles with parent_id, timestamp, model, and prompt preview. The bundle-list component reads this for fast rendering without loading every full bundle.

6. **Event-driven save/load.** `bundle-save-requested` → `bundle-saved`. `bundle-load-requested` → `bundle-loaded`. The bundle-manager is an invisible orchestrator (like sg-llm-request) — it listens for events and does the work.

### 9.3 How This Maps to the sg-llm Component Family

The bundle system needs two new components under `components/llm/`:

**`sg-llm-bundle/`** — the invisible orchestrator (headless). Captures the full reality (all blocks from sg-llm-reality + config from sg-llm-connection + response from sg-llm-output + metrics from sg-llm-stats). Saves/loads bundles to localStorage (MVP) or vault (when vault integration exists). Maintains the `parent_id` tree. Emits `llm:bundle-saved`, `llm:bundle-loaded`, `llm:bundle-list-changed`.

**`sg-llm-bundle-list/`** — the visible panel. Shows saved bundles in reverse chronological order. Click to load (time travel). Shows parent_id relationships as indentation or tree lines. Rename, delete, fork actions.

Events to add to `sg-llm-events.js`:

```js
// --- Bundles ---
BUNDLE_SAVE_REQUESTED: 'llm:bundle-save-requested',   // {}
BUNDLE_SAVED:          'llm:bundle-saved',              // { bundle }
BUNDLE_LOAD_REQUESTED: 'llm:bundle-load-requested',    // { bundleId }
BUNDLE_LOADED:         'llm:bundle-loaded',             // { bundle }
BUNDLE_DELETED:        'llm:bundle-deleted',             // { bundleId }
BUNDLE_LIST_CHANGED:   'llm:bundle-list-changed',       // {}
```

### 9.4 The Bundle + Memory + One-Shot Loop

This is how all the concepts converge in practice:

```
1. User builds reality: System + Context + Memory (empty) + Question
2. Send (Mode 1: Build) → LLM response appears in output
3. Auto-save bundle (captures reality + response + metrics)
4. User clicks "Update Memory" (Mode 2) → LLM condenses learnings into Memory block
5. Memory block updates. Token count shows it's 400 tokens.
6. User edits the Question block for the next iteration
7. Send (Mode 1: Build) → new response, but Memory carries forward learnings
8. Auto-save bundle (parent_id points to step 3's bundle)
9. Repeat. Memory stays ~500 tokens. Bundle tree grows. History is implicit in the tree, not in the context window.
```

After 20 iterations: context window is ~2,000 tokens (system + context + 500-token memory + question). Not 40,000 tokens of accumulated chat. Cost per iteration: ~$0.003. The compression path from the article, automated.

## 12. The Compression Path (Built Into the UX)

The seven principles from the LLM efficiency brief are not just team doctrine — they're visible in the UI:

| Principle | How the Components Make It Visible |
|-----------|-------------------------------------|
| **One-shot, no history** | No "conversation" accumulates. Each send is the reality you constructed. Memory block carries forward only what matters. |
| **Minimum viable context** | Token count per block. User sees exactly what they're paying for. Remove a block, cost drops instantly. Toggle blocks on/off. |
| **Constructed history** | Explicit fake user/assistant pairs — not hidden, not automatic. The user designs the model's memory. |
| **Managed memory** | The LLM itself prunes the memory via Mode 2. Old decisions that no longer matter are dropped. The memory stays focused. |
| **Track externalities** | sg-llm-stats shows cost per request, session totals, tokens/sec. The accountant sees everything. |
| **Wardley evolution** | sg-llm-bundle-list shows the fork tree. Click any ancestor bundle to see what the context looked like at Genesis ($0.50) vs Product ($0.003). The compression path is visible in the bundle history. |
| **Use LLMs to not use LLMs** | The dev environment's code editor + iframe preview: once the LLM writes the code, the code runs locally at zero cost forever. The bundle records the moment the LLM became unnecessary. |

## 13. Security Invariants

1. **API keys never leave the browser.** sg-llm-request makes fetch calls directly to provider APIs. No `*.sgraph.ai` proxy.
2. **sg-llm-connection stores keys in localStorage only.** Never cookies, never server-side.
3. **sg-llm-debug shows full reality.** This is transparency, not a leak — the user can see exactly what was sent.
4. **No telemetry.** No component sends prompts, responses, or realities to any SG server.
5. **Event scoping via sg-layout.** Two LLM instances on one page don't leak events between each other.
6. **Bundles are local-first.** sg-llm-bundle stores bundles in localStorage (MVP) or encrypted vault (when available). Bundle data never passes through SG servers unencrypted.
