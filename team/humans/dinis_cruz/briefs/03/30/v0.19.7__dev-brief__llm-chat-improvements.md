# LLM Chat Tool: Component Architecture, Model Chooser, and Next Version

**version** v0.19.7
**date** 30 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Architect, Designer
**type** Dev brief

## Context

The LLM chat tool was developed from a detailed 10-part brief, implemented almost in one go, and worked well enough to use offline on a plane with Ollama. The implementation leveraged existing components and IFD guidance. It is a solid Explorer-phase output. Now it needs Villager-phase improvements.

## Architecture Principle: One Component, One Thing

Every component should do one thing well and not care about what other components are doing. Shadow DOM isolation enforces this. The SG Layout wires them together. The tool is the assembly of components.

```
Tool (assembles components in SG Layout)
  │
  ├── Model Chooser (select which LLM to use)
  ├── Chat Window (display conversation)
  ├── Input Area (type messages, attach files)
  ├── Attachment Manager (handle files, images, paste)
  ├── Response Renderer (markdown, code, images)
  └── Settings Panel (temperature, system prompt, etc.)
```

No component should duplicate another's job. If you need to choose a model, use the Model Chooser component. No other component should have model selection logic. If you need to render markdown, use the Response Renderer. No other component should do markdown rendering.

### Component Discovery

A meta-component that lists all available components in a section. When working in SG Layout, the user should be able to click and select which components to open. Each component provides capabilities. The tool assembles them.

Components can have different views (simple vs advanced). The Model Chooser as a simple dropdown, or the Model Chooser as a full panel with categories, costs, and capabilities. Same component, different rendering mode.

### Broadcast Communication

Components should be able to receive broadcasts from the environment. A JSON message sent to all components in a panel: "the model has changed to X" or "switch to dark mode" or "the vault key is Y." Components that care about the message react. Components that do not, ignore it.

This is how configuration flows without tight coupling: the Model Chooser broadcasts the selected model, the Chat Window receives it and uses it for the next request.

## The Model Chooser Component

This deserves special attention because it is a tool people will come to the website for.

### What It Does

- Lists available models (from OpenRouter, Ollama, direct APIs)
- Categorises models: fast, cheap, good for coding, good for vision, good for analysis, expensive but best
- Shows cost per token (from OpenRouter pricing)
- Keeps a history of models used (in localStorage)
- Allows favouriting models
- Persists selection in localStorage (remembered between sessions)

### Model Categories

| Category | Examples | Use Case |
|---|---|---|
| Fast / cheap | Haiku, small Mistral | Quick questions, simple tasks |
| Good for coding | DeepSeek Coder, CodeLlama, Sonnet | Code generation, debugging |
| Good for vision | Models with multimodal support | Image analysis, screenshot review |
| Best quality | Opus, large models | Complex analysis, important documents |
| Local / offline | Ollama models | Offline work, privacy-sensitive tasks |
| Multi-model | Models supporting text + image + code | Versatile, general purpose |

### The Market Opportunity

OpenRouter has hundreds of models. The UIs for exploring and comparing them are not great. If we build a model chooser that is genuinely better (clearer categories, cost comparison, quality indicators, personal history), every OpenRouter user has a reason to visit our tools. This is a traffic driver.

We could even publish model comparison data as a shared vault/folder: "here are the models I recommend for different tasks, with benchmarks." That becomes shareable content.

## Bug Fixes for the Chat Window

### 1. Cannot Type While Response Is Streaming

Currently, the input area is blocked while the LLM response is streaming in. The user should be able to type the next message while the current response is still arriving. The input area and the response area are independent. One should not block the other.

### 2. Scroll Lock on User Action

When the response is streaming and text is being added at the bottom, the window auto-scrolls to follow. But if the user manually scrolls up (to re-read something earlier in the response), the auto-scroll should stop. The text keeps being added at the bottom, but the viewport stays where the user put it.

Resume auto-scroll when the user scrolls back to the bottom (or clicks a "jump to bottom" button).

### 3. Markdown Rendering During Streaming

Sometimes the streaming response produces incomplete markdown (a code block that has not been closed yet, a bold tag that is not yet terminated). This causes the rendered output to go "all over the place" visually. Solutions to explore:

- Buffer the rendering: only render complete markdown blocks (paragraphs, code blocks, lists)
- Render incrementally but handle unclosed tags gracefully
- Show raw text during streaming, render markdown on completion
- Look at how other chat UIs solve this (Claude's own UI handles it well)

### 4. Copy and Paste Images Not Working

Image paste from clipboard into the chat input is broken. The Attachment Manager component should handle this. Make it a standalone Web Component (as with all components) that manages:

- File drag and drop
- Clipboard paste (images, text)
- File picker dialog
- Preview of attached files
- Remove attachment

### 5. All LLM Components Should Use SG Layout

Several of the LLM tool components were built without SG Layout. They should be refactored to use it. SG Layout provides the panel management, resizing, and fractal nesting that makes the tool feel professional.

## Ollama Compatibility

Some components work with the cloud LLM providers but not with Ollama. This is likely a model compatibility issue (different response formats, different capabilities). Test each component against Ollama specifically and document which models work with which components.
