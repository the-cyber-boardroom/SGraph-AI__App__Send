# Architecture + API Contract: sg-tree — Generic Tree View Web Component

**Version:** v0.19.11
**Date:** 29 March 2026
**Role:** Architect (Explorer team)
**Type:** Architecture brief + API contract + implementation plan
**Status:** Pre-implementation — ready for review
**Pattern:** Follows `sg-layout` (proven, shipped, consumed by send-browse v0.3.0)

---

## 1. Strategic Intent

`sg-tree` is a standalone, zero-dependency, vanilla JS + Web Components tree view. Its single responsibility is rendering and navigating hierarchical data — folders/files, ontologies, dependency graphs, org charts, or any node-and-children structure.

It has no opinion about what the tree data represents. A zip file's folder structure, a vault's pointer tree, a workspace's file index, and an ontology graph all look the same to `sg-tree` — they are all `{id, label, children, icon, type}`.

**What sg-tree IS NOT:** It is not a file manager, not a zip editor, not a vault browser. Those are consumers of sg-tree. The editing operations (rename, move, delete, add) are a separate mode (v0.1.1+), not part of the read-only base.

## 2. The Problem: Three Trees, Zero Shared Code

Audited from codebase 28 March 2026:

| Product | Component | File | Lines | Rendering | Keyboard | Selection | Expand/Collapse | Drag |
|---------|-----------|------|-------|-----------|----------|-----------|-----------------|------|
| **SG/Send** browse | `send-browse.js` | `v0.3.0/_common/js/components/send-download/send-browse.js` | ~200 (tree portion) | innerHTML, nested `<div>` | j/k, arrows (files only) | Single file highlight | Click toggle, ▸/▾ | No |
| **SG/Vault** | `vault-tree-view.js` | `v0.1.3/_common/js/components/vault-tree-view/vault-tree-view.js` | ~493 | innerHTML, nested `<div>` | None | Single file, emits `tree-file-selected` | Click toggle, `-`/`+` | Yes (native HTML drag) |
| **SG/Workspace** | `vault-panel.js` | `v0.1.0/components/vault-panel/vault-panel.js` | ~300 (tree portion) | innerHTML, nested `<div>` | None | Single file, emits `file-selected` | Click toggle | No |

**Duplication:** All three build a `{name, children, files}` tree from their data source, render it as nested divs, attach click listeners for expand/collapse, and emit selection events. A basename display bug fixed in SG/Send v0.3.1 has not propagated to the vault or workspace.

**After sg-tree:** Each consumer writes a 10–20 line adapter that transforms their source data into sg-tree's canonical format. Tree rendering, expand/collapse, keyboard nav, selection, and styling are shared.

## 3. Core Design Principles

These mirror sg-layout's principles — proven patterns, not experiments.

### 3.1 Single Responsibility

`sg-tree` renders a tree and reports what the user does with it (select, expand, collapse). It does not:

- Fetch data (the consumer provides it)
- Know about zips, vaults, or workspaces
- Handle file preview or content display
- Manage state beyond its own tree rendering
- Own application routing or navigation

### 3.2 Shadow DOM

True CSS isolation. The consumer themes via CSS custom properties that pierce the shadow boundary. No class-based overrides. Same model as sg-layout.

### 3.3 Data-Driven Rendering

The tree is a JSON structure. The DOM is a projection of it. The component never queries the DOM to understand tree state — it queries `this._data`. Same rule as sg-layout's layout tree.

### 3.4 Zero Coupling to Consumers

Consumers do not import sg-tree internals. The shared surface is:

- The `<sg-tree>` custom element tag
- Event name constants from `sg-tree-events.js`
- CSS custom properties for theming
- The canonical data format `{id, label, children, icon, type, meta}`

### 3.5 Adapter Pattern for Data Sources

Each consumer writes a thin adapter:

```js
// SG/Send adapter — transforms JSZip tree to sg-tree data
function zipToTree(zipEntries) {
    const root = { id: 'root', label: 'Archive', type: 'folder', children: [] };
    // ... build tree from zip entries
    return root;
}

// SG/Vault adapter — transforms vault pointer tree to sg-tree data
function vaultToTree(vaultTree) {
    const root = { id: '/', label: vaultName, type: 'folder', children: [] };
    // ... build tree from vault._tree['/']
    return root;
}
```

The adapter is 10–30 lines. sg-tree never changes when a new consumer appears.

## 4. Canonical Data Format

```js
{
    id:       'unique-string',           // Required. Unique within the tree.
    label:    'Display Name',            // Required. Rendered text.
    type:     'folder' | 'file' | string,// Required. Controls icon + expand behaviour.
                                         //   'folder' = expandable. 'file' = leaf.
                                         //   Custom types ('entity', 'component', 'test') allowed.
    children: [ /* child nodes */ ],     // Optional. Only meaningful for expandable types.
    icon:     '📁' | 'svg-string' | null,// Optional. Overrides default icon for type.
    meta:     { size: 1024, ... },       // Optional. Arbitrary data passed through to events.
    badge:    '3' | null,                // Optional. Count/badge rendered right-aligned.
    disabled: false,                     // Optional. Greyed out, not selectable.
}
```

**Rules:**

- `id` must be unique across the entire tree. Duplicate ids cause undefined behaviour (same rule as sg-layout tab ids).
- `children` on a non-expandable type is ignored.
- `type: 'folder'` is expandable by default. Custom types can be registered as expandable via `expandableTypes`.
- `meta` is opaque to sg-tree — it is passed through in events so consumers can act on it without querying the tree.

## 5. Component API

### 5.1 HTML Usage

```html
<sg-tree id="file-tree"></sg-tree>

<script type="module">
import { SgTree }      from '/core/sg-tree/v0/v0.2/v0.2.0/sg-tree.js';
import { SGT_EVENTS }  from '/core/sg-tree/v0/v0.2/v0.2.0/sg-tree-events.js';

const tree = document.getElementById('file-tree');
tree.setData(myTreeData);
</script>
```

### 5.2 Public Methods

| Method | Returns | Description |
|--------|---------|-------------|
| `setData(root)` | void | Set and render the tree from a canonical data object |
| `getData()` | object \| null | Get the current tree data (deep clone, no DOM refs) |
| `getSelectedId()` | string \| null | Get the currently selected node id |
| `select(id)` | void | Programmatically select a node (scrolls into view) |
| `expand(id)` | void | Expand a folder node |
| `collapse(id)` | void | Collapse a folder node |
| `expandAll()` | void | Expand all expandable nodes |
| `collapseAll()` | void | Collapse all expandable nodes |
| `expandPath(id)` | void | Expand all ancestors of a node (useful for "reveal file") |
| `scrollTo(id)` | void | Scroll the tree to make a node visible |
| `filter(fn)` | void | Show only nodes where `fn(node)` returns true. `null` clears. |
| `getExpandedIds()` | string[] | List of currently expanded node ids |
| `setExpandedIds(ids)` | void | Restore expansion state (for serialisation round-trip) |

### 5.3 Configuration Properties

Set via attributes or properties on the `<sg-tree>` element:

| Property | Attribute | Type | Default | Description |
|----------|-----------|------|---------|-------------|
| `selectable` | `selectable` | `'single'`\|`'multi'`\|`'none'` | `'single'` | Selection mode |
| `expandableTypes` | — | `string[]` | `['folder']` | Which `type` values have expandable children |
| `showRoot` | `show-root` | boolean | `true` | Whether to render the root node or start from its children |
| `indentPx` | `indent-px` | number | `16` | Pixels of indentation per depth level |
| `sortFn` | — | function \| null | `null` | Custom sort function for siblings. Default: folders first, then alpha. |
| `iconMap` | — | object | built-in | Maps `type` string to icon (emoji, SVG string, or CSS class) |

## 6. Event Architecture

### 6.1 sg-tree-events.js (frozen at v0.1.0)

```js
/**
 * sg-tree-events.js
 * Event name constants for the sg-tree component.
 * Import this file. Never hardcode event name strings.
 * v0.1.0 — these names are frozen. Adding is ok. Changing is a breaking change.
 */
export const SGT_EVENTS = Object.freeze({

    // --- Selection ---
    NODE_SELECTED:      'sg-tree:node-selected',     // { id, node, path }
    NODE_DESELECTED:    'sg-tree:node-deselected',   // { id }
    SELECTION_CHANGED:  'sg-tree:selection-changed',  // { selectedIds }

    // --- Expand / Collapse ---
    NODE_EXPANDED:      'sg-tree:node-expanded',     // { id, node }
    NODE_COLLAPSED:     'sg-tree:node-collapsed',    // { id, node }

    // --- Activation (double-click or Enter on a leaf) ---
    NODE_ACTIVATED:     'sg-tree:node-activated',    // { id, node, path }

    // --- Context ---
    NODE_CONTEXT:       'sg-tree:node-context',      // { id, node, x, y }

    // --- Tree state ---
    DATA_CHANGED:       'sg-tree:data-changed',      // { root }
    READY:              'sg-tree:ready',              // {}
});
```

### 6.2 Event Dispatch

All events are `CustomEvent` with `bubbles: true, composed: true`. Consumers listen on the `<sg-tree>` element or any ancestor:

```js
tree.addEventListener('sg-tree:node-selected', (e) => {
    const { id, node, path } = e.detail;
    // path is the array of ancestor ids from root to this node
    openFilePreview(node.meta);
});

tree.addEventListener('sg-tree:node-activated', (e) => {
    // Double-click or Enter on a file
    openFileInEditor(e.detail.node.meta);
});
```

### 6.3 Internal EventBus

For internal coordination (keyboard navigation state, scroll position), sg-tree uses a private `EventBus` identical to sg-layout's. Not exposed publicly.

## 7. Keyboard Navigation

Active when the `<sg-tree>` element has focus (it sets `tabindex="0"`).

| Key | Action |
|-----|--------|
| `ArrowDown` / `j` | Move focus to next visible node |
| `ArrowUp` / `k` | Move focus to previous visible node |
| `ArrowRight` | If folder and collapsed → expand. If folder and expanded → move to first child. If file → no-op. |
| `ArrowLeft` | If folder and expanded → collapse. If collapsed or file → move to parent. |
| `Enter` | Select focused node. On leaf, also emits `NODE_ACTIVATED`. |
| `Space` | Toggle selection on focused node (multi-select mode). |
| `Home` | Move focus to first node. |
| `End` | Move focus to last visible node. |
| `*` | Expand all siblings of focused node. |

Focus is tracked as a separate concept from selection (you can focus a node without selecting it, allowing keyboard preview before committing).

## 8. Rendering Architecture

### 8.1 Flat List Virtualisation (not nested DOM)

Unlike the current implementations that nest `<div>` inside `<div>` matching the tree depth, sg-tree renders a **flat list of rows** with indentation via `padding-left`. Each row is a single `<div>` at the same DOM depth.

Why: Nested DOM trees of 500+ nodes cause layout thrashing. A flat list with virtual scrolling (v0.1.2+) scales to tens of thousands of nodes.

```
Shadow DOM:
.sgt-root
  .sgt-viewport (overflow-y: auto)
    .sgt-row[data-id="root"]         padding-left: 0
    .sgt-row[data-id="folder-a"]     padding-left: 16px
    .sgt-row[data-id="file-1"]       padding-left: 32px
    .sgt-row[data-id="file-2"]       padding-left: 32px
    .sgt-row[data-id="folder-b"]     padding-left: 16px
    ...
```

The visible rows are computed from `this._data` by walking the tree and skipping collapsed subtrees. This "flattened visible list" is the rendering source.

### 8.2 Connector Lines

Optional tree connector lines (vertical + horizontal) drawn via CSS `::before` pseudo-elements on each row, using `--sgt-connector-color`. Controlled by `connectors` attribute (default: `false`).

### 8.3 Icons

Default icon map:

| Type | Icon |
|------|------|
| `folder` (collapsed) | `▸ 📁` |
| `folder` (expanded) | `▾ 📂` |
| `file` | `📄` |
| Custom types | `◆` (or consumer-provided via `iconMap`) |

The toggle chevron (▸/▾) is part of the row for expandable types. Non-expandable types get the icon directly with equivalent left padding.

## 9. Serialisation

```js
// Save state
const state = {
    expandedIds: tree.getExpandedIds(),
    selectedId:  tree.getSelectedId(),
    scrollTop:   tree.getScrollTop(),
};
localStorage.setItem('tree-state', JSON.stringify(state));

// Restore state
const saved = JSON.parse(localStorage.getItem('tree-state'));
tree.setData(data);
tree.setExpandedIds(saved.expandedIds);
tree.select(saved.selectedId);
tree.setScrollTop(saved.scrollTop);
```

**For sg-layout integration:** If sg-tree is hosted inside an sg-layout panel, it implements `getLayoutState()` / `setLayoutState()` so the layout's serialisation captures tree state automatically:

```js
// Automatic — sg-layout calls this during getLayout()
getLayoutState() {
    return {
        expandedIds: this.getExpandedIds(),
        selectedId:  this.getSelectedId(),
        scrollTop:   this._viewport?.scrollTop || 0,
    };
}

setLayoutState(state) {
    if (state.expandedIds) this.setExpandedIds(state.expandedIds);
    if (state.selectedId)  this.select(state.selectedId);
    if (state.scrollTop != null) this.setScrollTop(state.scrollTop);
}
```

### 10.2 Placeholder Folders from Day One

All v0.1.x folders are created at the start of v0.1.0 development, each containing a `README.md` that specifies:

- **Scope:** Exactly which methods/behaviours this version adds or overrides

## 11. IFD Version Sequence — Surgical Override Plan

All v0.1.x versions through v0.1.6 must ship before sg-tree replaces any current consumer. The v0.2.0 consolidation is the integration point.

Each minor version is a surgical prototype override — not a copy of the full file. The v0.3.1 Send UI pattern (where `send-browse-v031.js` wraps original methods via IIFE and overrides `SendBrowse.prototype._renderFolderNode`) is the proven technique.

### v0.1.0 — Core (the base class, ~800-1000 lines)

**File:** `sg-tree.js` (canonical name, no version stamp)

**Delivers:** `SgTree` class, Shadow DOM, setData/getData, flat row rendering, expand/collapse, single selection, keyboard nav (arrows + j/k + Enter + Home/End), all SGT_EVENTS, CSS custom properties, default icons, `getLayoutState`/`setLayoutState` for sg-layout hosting, ARIA roles.

**Critical:** The base class exposes itself for surgical overrides: `window.SgTree = SgTree;` after `customElements.define()`. This is required by the v1.3.0 IFD convention — override modules access the class via `const { SgTree } = window;`.

**Test file:** `v0.1.0.test.html` — 30 programmatic assertions + interactive demo.

**This is the only version that contains the full class definition and calls `customElements.define()`.** All subsequent versions are surgical overrides with version-stamped filenames.

### v0.1.1 — Editor Mode (~80-120 lines surgical override)

**File:** `sg-tree--v0.1.1.js` + `sg-tree--v0.1.1.css`

**Overrides:** `_renderRow` (adds edit affordances when `editable="true"`), `_onKeydown` (adds F2 for rename, Delete for delete).

**Adds:** `_startRename(id)`, `_commitRename(id, newLabel)`, `_deleteNode(id)`, `_createFolder(parentId)`, `_addFiles(parentId, files)`. New events: none (uses `DATA_CHANGED` from v0.1.0).

**Pattern:**
```js
const { SgTree } = window;
const _originalRenderRow = SgTree.prototype._renderRow;
SgTree.prototype._renderRow = function(node, depth) {
    const el = _originalRenderRow.call(this, node, depth);
    if (this._editable) { /* add edit button, inline input */ }
    return el;
};
```

**Test file:** `v0.1.1-editor-demo.html` — rename inline edit, delete confirmation, create folder, drop files onto tree. MUST also run all v0.1.0 assertions to verify no regression.

### v0.1.2 — Virtual Scrolling (~60-80 lines surgical override)

**File:** `sg-tree--v0.1.2.js`

**Overrides:** `_renderVisibleRows` (replaces full render with windowed render), `_onScroll` (new, calculates visible range).

**Adds:** Row recycling pool, `_visibleRange` state, `_rowHeight` measurement.

**Test file:** `v0.1.2-virtual-scroll-stress.html` — 5,000-node tree, measure mount time (<200ms), scroll FPS (>30fps), verify selection/expand still work at depth. MUST run all v0.1.0 + v0.1.1 assertions.

## 14. Consumer Integration Example

How SG/Send browse would use sg-tree (the target state after Phase 3):

```js
// In send-browse.js — _initLayout() after sg-layout mounts

import { SGT_EVENTS } from 'https://tools.sgraph.ai/core/sg-tree/v0/v0.2/v0.2.0/sg-tree-events.js';

_populateTree() {
    const treeEl = document.createElement('sg-tree');
    treeEl.setAttribute('show-root', 'false');
    treeEl.setData(this._zipToSgTree());

    // File selected → open preview tab
    treeEl.addEventListener(SGT_EVENTS.NODE_SELECTED, (e) => {
        if (e.detail.node.type === 'file') {
            this._openFileTab(e.detail.node.meta.path);
        }
    });

    // File activated (double-click) → open in present mode
    treeEl.addEventListener(SGT_EVENTS.NODE_ACTIVATED, (e) => {
        if (e.detail.node.type === 'file') {
            this._openFilePresentMode(e.detail.node.meta.path);
        }
    });

    const panel = this._sgLayout.getPanelElement('t-tree');
    panel.innerHTML = '';
    panel.appendChild(treeEl);
}

_zipToSgTree() {
    const root = { id: 'root', label: this.fileName || 'Archive', type: 'folder', children: [] };

    const addNode = (parentNode, zipChildren) => {
        const folders = zipChildren.filter(e => e.dir).sort((a, b) => a.name.localeCompare(b.name));
        const files   = zipChildren.filter(e => !e.dir).sort((a, b) => a.name.localeCompare(b.name));

        for (const folder of folders) {
            const node = {
                id:       folder.path,
                label:    folder.name,
                type:     'folder',
                badge:    String(this._countFiles(folder)),
                children: [],
            };
            addNode(node, folder.children || []);
            parentNode.children.push(node);
        }
        for (const file of files) {
            parentNode.children.push({
                id:    file.path,
                label: file.name,
                type:  'file',
                icon:  this._getFileIcon(file.name),
                meta:  { path: file.path, size: file.size },
            });
        }
    };

    addNode(root, this.zipTree);
    return root;
}
```

That replaces ~200 lines of `_renderFolderNode`, `_setupTreeListeners`, `_onKeydown`, expand/collapse all, and active file highlighting. The keyboard nav, selection highlighting, expand/collapse, and event dispatch all come from sg-tree.

## 16. What sg-layout Taught Us

Patterns to repeat:

1. **Shadow DOM + CSS custom properties** — proven isolation model, themeable by consumers
2. **Frozen event constants file** — never modified after v0.1.0, only added to
3. **Dual-purpose test page** — programmatic assertions + interactive demo in one HTML file
4. **Synchronous registration protocol** — `getLayoutState`/`setLayoutState` for sg-layout hosting
5. **Single consolidated file at major version** — develop via IFD surgical overrides, consolidate at v0.2.0
6. **Surgical overrides via prototype replacement** — the v0.3.1 Send UI pattern (IIFE wrapping original method) is clean and proven

Patterns to fix:

1. **IFD deep versioning** — sg-layout used flat `v0.1.0/` but should have been `v0/v0.1/v0.1.0/`. sg-tree uses the correct deep path from the start.
2. **Placeholder folders for future versions** — sg-layout didn't do this. sg-tree creates all v0.1.x folders with README scope documents at the start, making the IFD sequence visible and guiding future sessions.
3. **Version-stamped filenames** — sg-layout put the consolidated file at `v0.1.0/sg-layout.js` and didn't use surgical override files at all (it shipped one big consolidated file). sg-tree uses the `v1.3.0__ifd__surgical-overrides-version-stamped-filenames.md` convention: base is `sg-tree.js` (no stamp), overrides are `sg-tree--v0.1.1.js` (double-dash + full semver). The base exposes itself via `window.SgTree = SgTree` for overrides to access.

Patterns to improve:

1. **Flat row rendering** — sg-layout uses CSS Grid for its tree; sg-tree uses a flat list for better scroll performance with large trees. This is an architectural improvement, not a contradiction.
2. **ARIA from v0.1.0** — sg-layout deferred accessibility. sg-tree includes it from the start because tree views have well-established ARIA patterns (`role="tree"`, `aria-expanded`, `aria-selected`).
3. **Test accumulation** — sg-layout's test page tested only the current version. sg-tree's test pages accumulate — every new version's test page runs ALL prior assertions first, guaranteeing no regressions before the v0.2.0 consolidation.
