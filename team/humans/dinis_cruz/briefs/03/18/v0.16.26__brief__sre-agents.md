## New Team Members

Two SRE (Site Reliability Engineering) agents are added to the project, one per team tier:

| Agent | Team | Responsibility |
|---|---|---|
| SRE (Villager) | Villager | Non-production environments: dev, QA, staging. CI pipeline health. Test infrastructure. |
| SRE (Town Planner) | Town Planner | Production environments. Live monitoring. Cost tracking. Uptime. Grafana ownership. |

This brings the agent count to 20+.

## SRE (Villager Team)

Owns the non-production infrastructure:

- Dev, QA, and staging environments: keeping them running, keeping them consistent with production
- CI pipeline health: are builds passing, are tests running, are deployments succeeding
- Test infrastructure: the local server instances used by the QA environment (see companion QA brief)
- Environment parity: catching drift between dev and production before it causes surprises

This SRE ensures that the Explorer and Villager teams have reliable environments to work in. When a developer says "it works on my machine," this SRE has the data to confirm or deny.

## SRE (Town Planner Team)

Owns production. This is the agent that:

- Fires at least once a day (ideally more frequently) to collect and process production data
- Owns Grafana: the dashboards, the alerts, the data sources (see companion Grafana brief)
- Owns monitoring: CloudWatch, Prometheus endpoints, X-Ray traces
- Tracks cost: Lambda spend, S3 storage, bandwidth, projected monthly cost
- Owns the migration path to OpenTelemetry as the observability standard
- Produces regular reports on production health, pushed to a vault

### The Daily Vault Workflow

The production SRE's daily routine as a vault-based workflow:

```
1. Lambda function triggers (scheduled, daily or more frequent)
2. Opens a vault for this execution run
3. Queries AWS metrics (CloudWatch, S3, Lambda stats)
4. Queries custom Prometheus endpoints (vault metadata, traffic stats)
5. Runs health checks against production endpoints
6. Collects all data into the vault
7. Generates a summary report
8. Pushes the vault (or zips and stores it)
9. Optionally pushes the zip to another vault as a historical archive
```

Every day produces a vault. Every vault is a snapshot of production health at that point in time. The historical archive of daily vaults becomes the production health timeline.

## Vault-Based Monitoring and Encrypted Dashboards

An interesting property: monitoring data collected into a vault can be published to a website where only authorised users can read it.

The SRE can create a public-facing status page where the data is encrypted or signed. The page is technically public, but the sensitive metrics (cost data, detailed error rates, infrastructure specifics) are only visible to someone with the decryption key. The general health status (up/down, response time ranges) can be shown in plaintext.

This solves a long-standing problem with monitoring dashboards: they contain sensitive operational data, so they need to be behind authentication. With vault-based encrypted monitoring, the dashboard can be public. The sensitive data is encrypted. Only the team sees the details. Everyone else sees the health summary.

## CI Pipeline Data as Vaults

Extend the vault capture pattern to CI pipelines:

- Every CI run produces a vault (or a folder within a shared vault) containing: build logs, test results, deployment status, performance metrics, code coverage
- These vaults are browsable, searchable, and analysable by agents
- Historical CI data becomes a structured archive, not a pile of CloudWatch logs

## OpenTelemetry Path

The Town Planner SRE should already be thinking about the migration from CloudWatch-native metrics to OpenTelemetry. OpenTelemetry provides vendor-neutral instrumentation, which means:

- Metrics, traces, and logs in a standard format
- Portable across monitoring backends (Grafana, Datadog, whatever comes next)
- Instrumentation code does not change when the monitoring backend changes

This is not urgent (CloudWatch + Grafana covers the immediate need), but the SRE should be evaluating the path and documenting what the migration would look like.

## SRE Website

Both SREs should publish materials to a website (similar to how the Librarian maintains library.sgraph.ai and the Sherpa will maintain guides). The SRE website contains:

- Production health status (public, with encrypted detail layer)
- Infrastructure documentation
- Runbooks for common incidents
- Cost reports (encrypted, team-only access)
- Performance trend reports
