## The Observation

A musician friend saw the gallery view and said: "This feels quite geeky." He is right. The technology works. The presentation does not speak to him.

This is not a bug. It is the opportunity. The same underlying technology, re-skinned and re-structured for a specific audience, becomes a completely different product. Every industry has workflows that have never been served with good tooling. A few JavaScript components and a theme change transforms "encrypted vault with version control" into "music collaboration workspace" or "medical document sharing" or "legal evidence room."

## The Competitive Advantage

The competitive advantage is not any single feature. It is the speed at which we can compose new UIs from existing components.

What we have:

- A solid CI pipeline (getting stronger with the QA environment brief)
- A testing and QA workflow (agentic, vault-based execution capture)
- sg-layout for fractal window management (layouts of layouts)
- Web Components that are event-driven, standalone, and composable
- IFD methodology for rapid iteration
- Pyodide working (sg-send-cli running inside the browser via WebAssembly)
- Agentic teams that can produce and iterate on components fast

This means we can take a use case (musicians collaborating on a song, a lawyer managing evidence, a doctor sharing patient records), identify the 3-5 components that make it work, compose them in sg-layout, apply a theme, and ship it. The underlying vault, encryption, branching, and sync infrastructure does not change. Only the surface changes.

Most of these transformations are surprisingly small. The same file viewer with a waveform visualisation becomes a music tool. The same folder view with a document preview becomes a legal workspace. Sometimes it is the exact same functionality with a couple of twists, but it looks dramatically different and feels dramatically different to the target audience.

## Component Lifecycle

Every component follows the same lifecycle:

```
1. Created as a standalone tool (tools.sgraph.ai)
   - No server-side dependency
   - File in, file out (the core primitive)
   - Event-driven, composable
   - Can be used independently in a plain HTML page

2. Stress-tested as a tool
   - Used in isolation to find bugs
   - Managed with IFD methodology
   - Version controlled independently
   - Eventually may get its own repo

3. Composed into workflows
   - Dropped into sg-layout alongside other components
   - Used as a viewer in the vault UI
   - Used in gallery, folder view, or custom vertical UIs

4. Themed for verticals
   - Same component, different visual presentation
   - Cultural/industry theming applied
   - Language, colours, layout adjusted

5. Eventually: plugins and integrations
   - Components that connect to external tools
   - Management systems, DAWs, medical systems
   - The component becomes an integration point
```

### The File-In, File-Out Principle

Every component's core contract: it receives a file (or files) and outputs a file (or files). The component does not call servers. It does not call APIs. It processes data locally. The vault (or SG/Send) handles storage and transport.

This means:

- Components work offline
- Components work in any context (tools page, vault UI, gallery, custom vertical)
- Components are testable in isolation (give it a file, check the output)
- The server never sees unencrypted data (the component processes plaintext in the browser, encrypts before saving)

## Theme and Culture Component

A new type of component: `<sg-theme>`. This is not just colours and CSS. It is a configuration component that controls:

| Property | What It Changes |
|---|---|
| Language | All UI labels, instructions, tooltips switch to the selected language |
| Colour scheme | Brand colours, dark/light mode, high contrast accessibility |
| Layout preferences | Which components are visible, how they are arranged |
| Terminology | "Branch" becomes "Your workspace" for non-technical audiences, "Take" for musicians, "Version" for lawyers |
| Iconography | Industry-specific icons and visual metaphors |
| Feature visibility | Hide advanced features for simple workflows, show them for power users |

When the theme is set to "musician," the entire UI adapts: the colour scheme changes, the terminology changes, the default viewers change (audio player instead of text editor), the layout rearranges to prioritise the components musicians need.

This is the same pattern as i18n (switching languages) extended to switching audiences. Language is one dimension. Industry, skill level, and aesthetic preference are others.

### Accessibility as a Theme

High contrast, large text, screen reader optimisation, reduced motion: these are theme settings, not separate implementations. The `<sg-theme>` component handles them alongside cultural and industry theming.

## Marketing Connection

Every component is a marketing asset:

- It is a tool on tools.sgraph.ai (users discover it, use it, see the brand)
- It is a viewer in the vault UI (users experience it when they open a vault)
- It is a demo on the Sherpa website (interactive walkthrough)
- It is a vertical-specific landing page ("Are you a musician? Here is your workspace")

The component pipeline feeds the marketing pipeline. Every new component is a new reason for a new audience to try the platform.
