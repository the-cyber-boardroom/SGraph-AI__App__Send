# Local LLM Docker Chat — Technical Briefing
**SG_Send__Deploy · Local Development Environment**
Version v0.1.13 · 18 March 2026 · SGraph Send Team

---

## 1. What We're Building and Why

A Docker-based local development environment that provides:
- A **chat UI** for interacting with local LLM models via Ollama
- An **API layer** (FastAPI) that proxies to Ollama on the host machine
- **Persistent chat sessions** that survive container rebuilds
- A UI built with the **sg-layout** web component from SGraph-AI__Tools

**Primary use case:** Offline LLM chat during travel (flight on 19 March 2026). Must work completely disconnected from the internet once the Docker image is built and Ollama models are pulled.

**Secondary use case:** Foundation for the QA runner panel described in the EC2 LLM architecture brief (`v0.1.5__ec2_llm_arch.md`). The same `/api/chat` proxy pattern works against both local Ollama and the EC2 SSH tunnel described in that brief.

## 2. Architecture Overview

```
MacOS Host
├── Ollama (native, no Docker)
│   └── localhost:11434
│   └── Models: gemma3:4b, deepseek-coder, gemma2, llama3.2,
│               llama3.2-vision, phi3, llama3, codellama, mistral, gemma
│
└── Docker Compose
    └── sg-llm service
        ├── FastAPI (uvicorn, port 8080)
        │   ├── /api/chat         → streaming proxy to host Ollama
        │   ├── /api/models       → proxy to Ollama /api/tags
        │   ├── /api/sessions     → CRUD for chat sessions
        │   └── /static           → serves the UI files
        │
        ├── Static UI (sg-layout + web components)
        │   └── Loaded from filesystem, no CDN dependency
        │
        └── Volume mount: ./data → /app/data
            └── sessions/*.json   → persistent chat history
```

### Network: Docker → Host Ollama

The container reaches the host's Ollama via `host.docker.internal:host-gateway` (Docker's standard mechanism for container-to-host communication on macOS). The `OLLAMA_HOST` environment variable configures the proxy target.

## 4. API Design

### 4.1 POST /api/chat — Streaming Proxy

Forwards the request to Ollama's `/api/chat` endpoint with `stream: true`. Returns a `StreamingResponse` so the browser can render tokens as they arrive.

**Request body** (matches Ollama's native format):
```json
{
  "model": "gemma3:4b",
  "messages": [
    {"role": "user", "content": "explain audit trails"}
  ],
  "stream": true
}
```

**Response:** Server-Sent Events / newline-delimited JSON, proxied directly from Ollama. Each chunk:
```json
{"model":"gemma3:4b","message":{"role":"assistant","content":"An"},"done":false}
```

Final chunk has `"done": true` with token counts and timing.

**Implementation:** Uses `httpx.AsyncClient` to stream from Ollama, yielding chunks as they arrive via FastAPI's `StreamingResponse`.

### 4.2 GET /api/models

Proxies to Ollama `/api/tags`. Returns the list of locally available models.

**Response:**
```json
{
  "models": [
    {"name": "gemma3:4b", "size": 3300000000, "modified_at": "..."},
    {"name": "deepseek-coder:latest", "size": 776000000, "modified_at": "..."}
  ]
}
```

### 4.3 Session CRUD

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/sessions` | List all sessions (id, title, model, created, message_count) |
| `POST` | `/api/sessions` | Create/save a session |
| `GET` | `/api/sessions/{id}` | Load a session with full message history |
| `PUT` | `/api/sessions/{id}` | Update a session (append messages, rename) |
| `DELETE` | `/api/sessions/{id}` | Delete a session |

**Session JSON format** (stored in `data/sessions/{id}.json`):
```json
{
  "id": "a1b2c3d4",
  "title": "Audit trail discussion",
  "model": "gemma3:4b",
  "created": "2026-03-18T10:00:00Z",
  "updated": "2026-03-18T10:15:00Z",
  "messages": [
    {"role": "user", "content": "explain audit trails"},
    {"role": "assistant", "content": "An audit trail is a chronological record..."}
  ]
}
```

Title is auto-generated from the first user message (first ~50 chars) unless explicitly set.

## 5. UI Architecture

### 5.1 sg-layout — The Layout Engine

`sg-layout` is a Web Component from SGraph-AI__Tools (`sgraph_ai_tools__static/core/sg-layout/v0.1.0/`). It provides:

- **Fractal window management** — nested row/column/stack layout nodes
- **Drag-to-dock** — drag tabs between panels (top/left/center/right/bottom zones)
- **Resize handles** — drag between adjacent panels
- **Collapse/expand** — panels can minimize to strips
- **Tab system** — multiple tabs per stack, click to switch

**Public API we use:**
```javascript
getLayout()                    // serialize current layout to JSON
setLayout(json)                // restore a saved layout
addPanel(config)               // add a new panel
getPanelElement(id)            // get the DOM element for a panel
```

**Source files (bundled into Docker image for offline use):**
- `sg-layout.js` — the custom element definition
- `sg-layout.css` — embedded styles (depth-aware theming)
- `sg-layout-events.js` — internal event bus

**Production URL (for reference, NOT used at runtime):**
`https://dev.tools.sgraph.ai/core/sg-layout/v0.1.0/sg-layout.js`

### 5.2 Web Components (Vanilla, No Framework)

Following the exact pattern from `sg_send_deploy__ui__admin`:

- **No build step** — plain ES6 modules, served directly
- **No framework** — vanilla `HTMLElement` subclasses
- **No Shadow DOM** — light DOM only, global CSS applies
- **`customElements.define()`** for registration

Each component:
```javascript
class ChatPanel extends HTMLElement {
    connectedCallback()  { this._render(); }
    _render()            { this.innerHTML = `...`; /* bind events */ }
    async _loadData()    { /* fetch from API */ }
}
customElements.define('chat-panel', ChatPanel);
```

### 5.4 Chat Streaming in the Browser

The chat panel uses `fetch` with `ReadableStream` to render tokens incrementally:

```javascript
const response = await fetch('/api/chat', {
    method: 'POST',
    headers: {'Content-Type': 'application/json'},
    body: JSON.stringify({model, messages, stream: true})
});

const reader = response.body.getReader();
const decoder = new TextDecoder();

while (true) {
    const {done, value} = await reader.read();
    if (done) break;
    const chunk = JSON.parse(decoder.decode(value));
    appendToken(chunk.message.content);  // update DOM incrementally
}
```

### 5.5 llm-api.js — API Client

Thin fetch wrapper, same pattern as `admin-api.js` in the admin UI:

```javascript
class LlmAPI {
    constructor(baseUrl = '') { this.baseUrl = baseUrl; }

    async chat(model, messages)     { /* POST /api/chat, returns ReadableStream */ }
    async listModels()              { /* GET /api/models */ }
    async listSessions()            { /* GET /api/sessions */ }
    async getSession(id)            { /* GET /api/sessions/{id} */ }
    async saveSession(session)      { /* POST /api/sessions */ }
    async updateSession(id, data)   { /* PUT /api/sessions/{id} */ }
    async deleteSession(id)         { /* DELETE /api/sessions/{id} */ }
}

const llmAPI = new LlmAPI();  // global instance
```

## 8. Constraints and Decisions Already Made

These are locked in — not up for discussion:

| Decision | Rationale |
|----------|-----------|
| Ollama stays on macOS host, not in Docker | No GPU passthrough on macOS Docker. CPU inference works fine for gemma3:4b. |
| sg-layout files bundled into Docker image | Must work offline (flight). No CDN dependency at runtime. |
| No Open WebUI | Too heavy (Svelte, SQLite, auth). Doesn't use sg-layout. Fights the stack. |
| No database | JSON files on volume mount. Simple, inspectable, sufficient for single-user. |
| No auth | Local single-user tool. |
| FastAPI for the API layer | Consistent with the rest of SG_Send__Deploy. |
| `httpx` for Ollama proxy | Async streaming support, clean API. |
| Vanilla Web Components | Same pattern as `sg_send_deploy__ui__admin`. No framework, no build step. |
| Chat sessions persist via volume mount | `./data/sessions/` survives container rebuilds. |
| Default model: gemma3:4b | Best balance of quality and speed for the available models. |
| Directory: `sg_send_deploy__local_llm/` | Lives in this repo for now. May move later. |

## 10. Relationship to EC2 LLM Brief (v0.1.5)

This local Docker setup and the EC2 ephemeral LLM architecture share the same API contract:

| Concept | EC2 Brief | Local Docker |
|---------|-----------|--------------|
| Ollama access | SSH tunnel to EC2 (`localhost:11435`) | Docker → host (`host.docker.internal:11434`) |
| API proxy | Deploy Lambda routes | FastAPI in container |
| Chat endpoint | `/api/chat` → tunnel → Ollama | `/api/chat` → host → Ollama |
| Models endpoint | `/api/models` → tunnel → Ollama | `/api/models` → host → Ollama |
| QA checks | `Schema__QA__Check` / `Schema__QA__Result` | Same schemas, phase 2 |

When the QA runner panel is added (phase 2), it will work against both backends — just change the Ollama target URL.
