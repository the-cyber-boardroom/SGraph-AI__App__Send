# Agentic QA Environment: Performance Testing, Traffic Simulation, and Vault-Based Execution Capture

**version** v0.16.17
**date** 18 Mar 2026
**from** Human (project lead)
**to** QA (lead), Developer, Architect, DevOps, AppSec, SRE
**type** Dev brief

---

## What We Need

A QA and performance testing environment that can:

- Simulate traffic (HTTP requests and browser automation) against multiple SG/Send endpoints
- Run against dev, staging, and production environments
- Measure performance baselines and detect regressions
- Capture full execution data in vaults for post-run analysis
- Be driven by an agentic team (QA, Sherpa, Architect, AppSec, SRE agents) that analyses results and generates new test scenarios

## Architecture

### Execution Layer

Two execution contexts:

**CI pipeline:** Runs against a local server instance (baseline data, no network variability) and then against live servers (real-world performance). Part of the regular CI flow. Every push gets performance-checked, not just functionally tested.

**Lambda functions:** On-demand execution for load testing and scheduled canary runs. A conductor Lambda orchestrates the test run, spinning up worker Lambdas that generate traffic in parallel.

### Multi-Environment Execution

Every test suite runs against three environments:

| Environment | Purpose |
|---|---|
| Dev | Baseline. Catch regressions before they reach staging. |
| Staging/Main | Pre-production validation. Confirms dev changes do not degrade performance. |
| Production | Real-world measurement. Canary requests running continuously. |

A critical bellwether: the performance profile of dev should match staging should match production. If dev introduces a regression (not just functional, but performance), it is caught before it reaches users.

### Two Types of Traffic

**Canary/baseline requests:** Control requests that run continuously at fixed intervals (every 20 seconds, every minute). They execute a small, specific set of actions simulating one user hitting the website independently. These provide:

- Heartbeat (is the service up?)
- Baseline performance metrics (response time, throughput)
- Cost baseline (what does steady-state traffic cost in Lambda invocations, bandwidth, S3 operations?)

The canary requests are the constant. Everything else is measured against them.

**Load test requests:** Scripted scenarios that increase traffic volume. Multiple simulated users executing workflows in parallel. Upload a file, wait, download it, upload five more, create a vault, clone it, push to it. Realistic user journeys at scale.

### Script-Driven, Not Agent-Inline

The agents do not generate traffic in real time. The agents write the scripts. The scripts execute the traffic.

This is deliberate:

- **No prompt injection risk.** The agents never process live data from the website inline. They process captured data after the run.
- **Efficiency.** A script executes a thousand HTTP requests in seconds. An agent generating each request inline would take minutes.
- **Reproducibility.** A script is deterministic. Run it again, get the same traffic pattern. An agent inline would produce different patterns each time.
- **Separation of concerns.** The agents think (analyse, hypothesise, design new scenarios). The scripts execute. The vault stores results. The agents analyse the results after the run.

The agent workflow:

```
1. Agents design test scenarios and write scripts
2. Scripts execute against target environment
3. Execution data captured in a vault (one vault per run)
4. Agents analyse the vault contents post-run
5. Agents generate reports, flag regressions, propose new scenarios
6. Reports pushed to vault
7. Cycle repeats
```

## What We Want to Measure

### Performance Baselines

| Metric | What It Tells Us |
|---|---|
| Upload latency (by file size) | How fast can a user upload? Where is the threshold? |
| Download latency (by file size) | How fast can a user download? |
| Vault clone time (by vault size) | How fast can a CLI clone? |
| Vault push time (by commit size) | How fast can a CLI push? |
| Folder view render time | How fast does the folder UI load? |
| Gallery render time | How fast does the gallery load? |

### Infrastructure Questions

| Question | Why It Matters |
|---|---|
| Does Lambda function memory size affect performance? | AWS charges by memory-ms. Finding the sweet spot between cost and speed. |
| Are there AWS settings that improve throughput? | Provisioned concurrency, memory tuning, regional placement. |
| At what traffic level is Lambda no longer economically viable? | The crossover point to Fargate, Firecracker, or EC2. |
| What does steady-state traffic cost per month? | Baseline cost model for pricing decisions. |
| What does a load spike cost? | Burst pricing model. Are there surprises? |

### Regression Detection

Every CI run compares its performance results against the previous baseline. If upload latency increases by more than 10%, flag it. If vault clone time doubles, block the deploy. Performance regressions are treated like functional test failures.

## Vault-Based Execution Capture

Every test run produces a vault containing:

- The scripts that were executed
- Raw HTTP response data (status codes, latencies, headers)
- Telemetry data (Lambda duration, memory usage, cold start frequency)
- Screenshots (for browser automation tests)
- Error logs
- Summary report (generated by the agents post-run)

The vault is the execution record. After the run, the vault is zipped and stored. Any agent (QA, Architect, AppSec) can clone the vault and analyse the data independently, at any time, without needing access to the live environment.

This means:

- No real-time log infrastructure needed for the initial version (post-run analysis from the vault is sufficient)
- Every run is reproducible (the vault contains everything)
- Historical comparison is easy (compare vault from run N to vault from run N-1)
- Grafana and real-time monitoring can be added later as an enhancement, not a prerequisite

## Agentic Team for QA

The agents involved:

| Agent | Role in QA |
|---|---|
| QA | Designs test scenarios, reviews coverage, flags gaps |
| Sherpa | Validates user journey accuracy (are we testing what real users do?) |
| Architect | Reviews performance implications of architecture changes |
| AppSec | Adversarial testing (what happens under malicious traffic patterns?) |
| SRE | Infrastructure analysis (cost, scaling, Lambda vs Fargate crossover) |
| Developer | Writes and maintains test scripts |

The agents process execution vaults asynchronously. They are not in the traffic path. They receive a vault, analyse it, produce reports, and push the reports back to the vault.
