## Context

The agentic QA environment (see companion brief) captures execution data in vaults for post-run analysis. That solves the batch case. But we also need real-time visibility: what is happening right now across the platform. Grafana and Prometheus provide that.

## Research Needed: AWS Native Monitoring

Before building anything custom, map what AWS already provides out of the box. Specifically:

| AWS Service | What to Monitor | What We Get for Free |
|---|---|---|
| Lambda | Invocations, duration, errors, cold starts, memory usage, throttles | CloudWatch metrics, native Grafana integration |
| S3 | Bucket size, request count, bandwidth, 4xx/5xx errors | CloudWatch metrics, S3 server access logs |
| CloudFront | Requests, bandwidth, cache hit ratio, error rates | CloudWatch metrics, access logs |
| API Gateway | Request count, latency, 4xx/5xx, integration latency | CloudWatch metrics |
| EC2 (when running) | CPU, memory, network, disk | CloudWatch metrics, detailed monitoring |
| X-Ray | Request tracing across Lambda, API Gateway, S3 | Trace maps, service graphs, latency distributions |
| Firehose/Data Streams | Throughput, delivery latency, error rates | CloudWatch metrics |

**Key question:** How much of this can we wire directly into Grafana without writing any custom code? AWS has a managed Grafana service (Amazon Managed Grafana). If it connects natively to CloudWatch, X-Ray, and S3 metrics, we get a significant dashboard for near-zero effort.

**Research deliverable:** A document listing every AWS metric available for our infrastructure, which ones are free vs. paid, and which ones connect to managed Grafana natively.

## Prometheus Integration

Prometheus is a pull-based monitoring system: it pings endpoints at regular intervals and collects metrics. This model fits our architecture well.

### Why Prometheus Works Here

The Prometheus model (scrape an endpoint, get metrics at a point in time) maps naturally to what we can expose. Because S3 data and vault contents are always encrypted, the metadata endpoints are not confidential. We can expose metrics endpoints without admin authentication because we are literally not exposing any confidential data. File IDs, counts, sizes, timestamps: all of this is non-sensitive metadata.

### What We Can Expose as Prometheus Endpoints

**Lambda function metrics endpoints:**

Each Lambda function exposes a `/metrics` endpoint (or a dedicated metrics Lambda) that returns Prometheus-formatted data:

- Vault count per server
- Total blob count, total storage size
- Request rates (reads, writes, deletes) per time window
- Active vault operations (clones, pushes, pulls in the last N minutes)
- Transfer statistics (uploads, downloads, bandwidth)

**Vault metadata endpoints:**

Non-confidential metadata about vault contents:

- Number of branches per vault
- Number of commits per branch
- Object count per vault (blobs, trees, commits)
- Vault age, last modified timestamp
- Pending change packs count

None of this is confidential. The server already knows file IDs, sizes, and folder structure (the opaque metadata). Exposing it in Prometheus format does not reveal any encrypted content.

### Prometheus-Native Execution Data

The QA execution runs (from the companion brief) should output metrics in Prometheus format. This means:

- Each test run produces Prometheus-formatted metric files alongside the raw data in the vault
- After the run, these files can be loaded into Prometheus (or Grafana directly) without transformation
- Historical performance data is both in the vault (for agent analysis) and in Prometheus format (for dashboard visualisation)

The Prometheus exposition format is simple:

```
# HELP sg_upload_latency_seconds Upload latency in seconds
# TYPE sg_upload_latency_seconds histogram
sg_upload_latency_seconds_bucket{file_size="1mb",le="0.5"} 45
sg_upload_latency_seconds_bucket{file_size="1mb",le="1.0"} 92
sg_upload_latency_seconds_bucket{file_size="1mb",le="2.0"} 98
sg_upload_latency_seconds_count{file_size="1mb"} 100
sg_upload_latency_seconds_sum{file_size="1mb"} 78.5
```

Adopt this as a standard output format for all performance measurement, both real-time and batch.

## Dashboard Design

### Real-Time Dashboard (Grafana)

Panels:

- **Platform health:** Lambda invocations, error rates, cold starts (from CloudWatch)
- **Traffic:** Uploads and downloads per minute, bandwidth usage (from CloudWatch + custom metrics)
- **Vault activity:** Clones, pushes, pulls per time window (from custom Prometheus endpoints)
- **Storage:** Total S3 usage, growth rate, cost projection (from CloudWatch S3 metrics)
- **Cost:** Estimated Lambda cost, S3 cost, bandwidth cost per hour/day (derived metrics)
- **Performance:** Upload/download latency percentiles (P50, P95, P99) from custom endpoints

### QA Comparison Dashboard

Side-by-side comparison of metrics from different test runs or different environments:

- Dev vs staging vs production latency profiles
- Before/after comparison for a specific code change
- Historical trend: is performance improving or degrading over time?

## Implementation Path

1. **Phase 1:** Set up Amazon Managed Grafana. Connect to CloudWatch. Get the free AWS metrics on a dashboard. Zero custom code.
2. **Phase 2:** Add custom Prometheus endpoints to Lambda functions (vault metadata, traffic stats). Connect to Grafana.
3. **Phase 3:** Make QA execution runs output Prometheus-native metric files. Load into Grafana for visual comparison.
4. **Phase 4:** Add alerting (Grafana alerts when error rates spike, latency exceeds threshold, or costs exceed budget).
