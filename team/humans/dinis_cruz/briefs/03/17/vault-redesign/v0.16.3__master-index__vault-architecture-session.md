## Executive Summary

This session audited the vault implementation across two repos (`SGraph-AI__App__Send`
and `SG_Send__CLI`), four UI codebases, and a live S3 deployment. It produced
architectural decisions that unify the CLI, browser, and web vault into one data
format with self-describing file IDs, batch read API, sub-tree model, and
on-demand tree loading.

**Key outcomes:**
- Self-describing file ID schema: `{type}-{derivation}-{mutability}-{hex_id}`
- Deterministic HMAC-derived entry points (eliminates dual-write hack)
- Sub-tree model (one tree per directory, not flat listing)
- Batch read API (multiple files in one request)
- On-demand tree loading as primary UX pattern
- Lite clone concept (fetch objects lazily, not upfront)
- Four-layer security model (encryption and signing are optional layers)
- Storage-agnostic backend architecture

## Key Decisions Made in This Session

| Decision | Rationale |
|----------|-----------|
| Self-describing IDs: `{type}-{derivation}-{mutability}-{hex_id}` | Server can cache/enforce without vault logic. Humans can read IDs at a glance. |
| Keep `bare/` prefixes on server | Preserves local = server = zip invariant |
| Deterministic HMAC IDs for entry points | Eliminates dual-write, enables browser interop |
| Sub-tree model (one tree per folder) | O(changed) merge, not O(all files). Natural delta push. |
| On-demand tree loading as primary UX | Load root, expand folders on demand. Most users touch 1-2 folders. |
| Encrypted-only metadata fields | Drop plaintext `path`, `name`, `size`, `message`. Keep only `_enc` versions. |
| Batch read API | Multiple reads in one HTTP call. Vault open in 3 requests, not 6+. |
| Lite clone | Same `bare/` structure, lazy fetch. Full clone only for offline/backup. |
| Named branch merge commit on push | Provenance boundary — named branch has own signature chain. |
| Indexes are future optimisation | Head snapshot, commit log, object manifest — not needed for first impl. |
| No backwards compatibility | Clean break. Keeps code simple. Existing vaults can be recreated. |

## Open Questions for Team Review

1. **Batch read auth model** — the current batch endpoint requires `write_key`
   in headers. Read-only batches should work without it. Backend team to confirm.

2. **Lite clone timing** — the architecture supports it now. When do we implement?
   CLI team to assess priority vs other Phase 5-9 work.

3. **Zip clone endpoint** — `GET /api/vault/download/{vault_id}` for one-request
   full clone. Backend team to assess effort. Low priority but high impact for
   large vaults.

4. **Test vectors** — need a known vault key that produces known self-describing
   IDs in both Python AND JavaScript. CLI team + vault browser team to create
   and cross-validate.
