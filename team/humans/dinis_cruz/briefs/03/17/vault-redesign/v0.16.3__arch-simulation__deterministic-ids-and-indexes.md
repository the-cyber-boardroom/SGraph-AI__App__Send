# Vault Access Simulation: Deterministic IDs, Batch Read, and Self-Describing Filenames

**Version:** v0.16.3  
**Date:** 2026-03-17  
**Role:** Architect  
**Status:** Working simulation — exploring minimal-request access patterns  
**Ref:** v0.13.30 simulation v6, architecture addendum v0.16.3  
**Based on:** Live vault `12cpxeq9` traced via vault browser (2026-03-16)

---

## Design Challenge

The v0.16.3 architecture addendum proposed HMAC-derived entry points as SEPARATE
files outside `bare/`. This breaks the v0.13.30 principle:

> "Bare vault is the portable unit. `bare/` syncs to the server, packs into a
> zip, and survives deletion of either side."

**Revised approach:** Make the IDs of files WITHIN `bare/` deterministic for
entry-point files. The `bare/` structure is preserved. The server mirrors it
exactly. But clients can COMPUTE the IDs of key files instead of listing
directories.

Additionally: add **batch read** to the API (alongside existing batch write),
and make every filename **self-describing** — encoding how the ID was derived,
what the file is, and whether concurrent writes need protection.

## Conventions

```
  📁  file on disk (plaintext working copy)
  🔒  encrypted object in bare/ or on server
  🔑  cryptographic key (local only)
  ☁️   on server
  →   "points to" / "contains"
```

## Unified ID Naming Schema

Every file ID follows the pattern: `{type}-{derivation}-{mutability}-{hex_id}`

Four segments, each 3 letters, dash-separated. From any filename you instantly
know three things: what it is, how to find it, and whether concurrent writes
need protection.

### Segment 1: Type (what the file is)

| Code | What it is |
|------|-----------|
| `obj` | Data object (blob, tree, or commit — type discovered after decryption) |
| `ref` | Ref pointer (mutable, points to a commit) |
| `idx` | Index file (denormalized cache) |
| `key` | Cryptographic key |
| `pkg` | Change pack / pending bundle |

### Segment 2: Derivation (how the ID was computed)

| Code | Meaning | Generation method |
|------|---------|------------------|
| `pid` | Pseudo-deterministic | HMAC from vault key + domain — computable by any client with the secret |
| `cas` | Content-addressed | SHA256 of ciphertext — deterministic from content |
| `rnd` | Random | `secrets.token_hex()` at creation time — discoverable only via index |

### Segment 3: Mutability (lifecycle and concurrency model)

| Code | Meaning | Server behaviour |
|------|---------|-----------------|
| `imm` | Immutable | Written once, never changes. Cache forever. Idempotent writes. |
| `snw` | Single-writer mutable | Only one entity updates. Cache with short TTL. No compare-and-swap needed. |
| `muw` | Multi-writer mutable | Multiple entities may update. No cache. Require compare-and-swap on writes. |

**Compare-and-swap (CAS):** The server's `write-if-match` operation — "only
write this new value if the current value matches what I expect." Prevents lost
updates when two clients push simultaneously. Only needed for `muw` files.

**Every file in `bare/` is encrypted.** No exceptions. No plaintext metadata.

### Full ID Examples

```
  obj-cas-imm-4e0340b66947      Content-addressed, immutable object (tree/commit/blob)
  obj-cas-imm-aeaa531d8450      Cache forever, no concurrency issues
  ref-pid-muw-b1d1771f9fb5      Pseudo-deterministic, multi-writer ref (named HEAD)
  ref-pid-snw-{clone_derived}   Pseudo-deterministic, single-writer ref (clone branch)
  idx-pid-muw-48c9319a269c      Pseudo-deterministic, multi-writer index (branch index)
  key-rnd-imm-4a6daf74fc4a1fe0  Random, immutable key
  pkg-rnd-imm-{ts}_{random}     Random, immutable change pack
```

### How This Reads Inside Encrypted Data

Inside a commit object:
```json
{
  "tree_id": "obj-cas-imm-867f9d7b56e1",
  "parents": ["obj-cas-imm-ce9c5c111aa2"]
}
```

Inside a branch index:
```json
{
  "head_ref_id": "ref-pid-muw-b1d1771f9fb5",
  "public_key_id": "key-rnd-imm-4a6daf74fc4a1fe0"
}
```

The type leads — you know WHAT it is before you know HOW it was derived.

## Batch Read API

### The Problem

Currently every vault read is a separate HTTP request. Opening a vault with
2 files requires 6 requests (ref + commit + root tree + sub-tree + 2 blobs).
Each request is a full internet round trip (~50-200ms depending on latency).

### The Solution

Extend the existing `POST /api/vault/batch/{vault_id}` endpoint to support
`read` operations alongside `write`, `write-if-match`, and `delete`:

```json
POST /api/vault/batch/12cpxeq9
{
  "operations": [
    { "op": "read", "file_id": "bare/refs/ref-pid-muw-b1d1771f9fb5" },
    { "op": "read", "file_id": "bare/data/obj-cas-imm-4602b76dd1ad" },
    { "op": "read", "file_id": "bare/data/obj-cas-imm-867f9d7b56e1" }
  ]
}

Response:
{
  "results": [
    { "file_id": "bare/refs/ref-pid-muw-b1d1771f9fb5", "status": "ok",
      "data": "<base64 encoded ciphertext>" },
    { "file_id": "bare/data/obj-cas-imm-4602b76dd1ad", "status": "ok",
      "data": "<base64 encoded ciphertext>" },
    { "file_id": "bare/data/obj-cas-imm-867f9d7b56e1", "status": "ok",
      "data": "<base64 encoded ciphertext>" }
  ]
}
```

**Server-side:** Lambda reads all requested files from S3 (AWS-to-AWS,
~1-5ms each), assembles the base64-encoded responses, returns in one payload.
One internet round trip instead of N.

**No authentication on batch read** — same as individual reads. The data is
encrypted; knowing the file_id doesn't help without the vault key.

### Server Caching by Mutability

The server can use the mutability segment for caching decisions:

| Mutability | Cache behaviour |
|-----------|----------------|
| `imm` (e.g. `obj-cas-imm-*`) | Cache aggressively — content never changes. Same ID = same bytes forever. |
| `snw` (e.g. `ref-pid-snw-*`) | Cache with short TTL (~5s). Only one writer, so staleness is brief. |
| `muw` (e.g. `ref-pid-muw-*`) | No cache. Must always read fresh — multiple writers may have updated. |

For a batch read of 6 files where 4 are `imm` and 2 are `muw`, the server
reads 2 from S3 and 4 from cache. Sub-millisecond for cached objects.

## Three Tiers of File IDs

| Tier | Derivation | Naming | Examples | Discovery |
|------|-----------|--------|---------|-----------|
| **1. Vault-key entry points** | `HMAC(read_key, domain)[:12]` | `*-pid-*` | Branch index, named HEAD ref | Compute directly — no listing, no index |
| **2. Discovered entry points** | `HMAC(read_key, domain + branch_name)[:12]` | `*-pid-*` | Per-branch refs | Compute after reading branch index |
| **3. Content-addressed** | `SHA256(ciphertext)[:12]` | `*-cas-*` | Blobs, trees, commits | Follow object graph from refs |
| **4. Random** | `secrets.token_hex()` | `*-rnd-*` | Keys, change packs | Discoverable via index or branch metadata |

**Tier 1 IDs are the entry points.** A client with only the vault key computes
them and starts reading. Tier 2 requires data from tier 1 (branch names).
Tier 3 requires data from tier 2 (commit IDs). Tier 4 requires data from
tier 1 (key IDs from the branch index).

**All tiers live inside `bare/`.** The `bare/` structure is identical on disk,
server, and zip.

## Scenario A: Opening a Vault (With Batch Read)

**Goal:** A client with only the vault key opens the vault and browses files.  
**Vault state:** 2 files (`adding file-1.txt`, `an-folder/file-b.txt`),
3 commits, sub-tree model.

**Important:** This scenario shows the **primary flow** — tree-based on-demand
loading. No head snapshot index is needed. This is the first implementation.
The snapshot index (Scenario F.1) is a future optimisation for CLI operations
that need the full file listing upfront.

### Step A.1: Derive Keys (0 requests)

```
  Input:  vault key "t92cse8g3cb0di4v8htvmd32:12cpxeq9"

  Derive:
    read_key           = PBKDF2(passphrase, "sg-vault-v1:12cpxeq9")
    write_key          = PBKDF2(passphrase, "sg-vault-v1:write:12cpxeq9")
    refFileId          = "ref-pid-muw-" + HMAC(read_key, "sg-vault-v1:file-id:ref:12cpxeq9")[:12]
                       = "ref-pid-muw-b1d1771f9fb5"
    branchIndexFileId  = "idx-pid-muw-" + HMAC(...)[:12] = "idx-pid-muw-48c9319a269c"

  Requests: 0  (pure computation, ~100ms for PBKDF2 600k iterations)
```

### Step A.2: Read HEAD Ref (1 request)

The only ID the client can compute upfront is the ref. The commit ID and
tree ID are discovered sequentially — each depends on the previous response.

```
  POST /api/vault/batch/12cpxeq9
  {
    "operations": [
      { "op": "read", "file_id": "bare/refs/ref-pid-muw-b1d1771f9fb5" }
    ]
  }

  Response:
    ref → decrypt → { "commit_id": "obj-cas-imm-4602b76dd1ad" }

  Requests so far: 1
```

### Step A.3: Read Commit (1 request)

Now the client knows the commit ID:

```
  POST /api/vault/batch/12cpxeq9
  {
    "operations": [
      { "op": "read", "file_id": "bare/data/obj-cas-imm-4602b76dd1ad" }
    ]
  }

  Response:
    commit → decrypt → {
      "tree_id": "obj-cas-imm-867f9d7b56e1",
      "parents": ["obj-cas-imm-ce9c5c111aa2"],
      "message_enc": "<encrypted: added file inside folder>",
      ...
    }

  Requests so far: 2
```

### Step A.4: Read Root Tree (1 request)

Now the client knows the root tree ID:

```
  POST /api/vault/batch/12cpxeq9
  {
    "operations": [
      { "op": "read", "file_id": "bare/data/obj-cas-imm-867f9d7b56e1" }
    ]
  }

  Response:
    root tree → decrypt → {
      "schema": "tree_v1",
      "entries": [
        { "name_enc": "<encrypted: adding file-1.txt>",
          "blob_id": "obj-cas-imm-aeaa531d8450",
          "size_enc": "<encrypted: 11>" },
        { "name_enc": "<encrypted: an-folder>",
          "tree_id": "obj-cas-imm-{sub_tree_hash}" }
      ]
    }

  Client now knows the ROOT LEVEL contents:
    ✓ "adding file-1.txt" — file, 11 bytes, downloadable
    ✓ "an-folder/" — folder, expandable (sub-tree ID known)

  Requests so far: 3
```

**Three requests to see the root folder.** The UI renders immediately:
one file and one folder. The user can click to download the file or expand
the folder. This is the correct UX pattern — load the level the user can
see, fetch deeper levels on demand.

Note: request 2 and 3 could be optimised into a single batch if the client
caches the commit→tree_id mapping from a previous session. On a fresh open,
the sequential dependency (ref → commit_id → tree_id) forces 3 round trips.

### Step A.5: User Expands Folder (1 request)

User clicks `an-folder/`:

```
  POST /api/vault/batch/12cpxeq9
  {
    "operations": [
      { "op": "read", "file_id": "bare/data/obj-cas-imm-{sub_tree_hash}" }
    ]
  }

  Response:
    sub-tree → decrypt → {
      "schema": "tree_v1",
      "entries": [
        { "name_enc": "<encrypted: file-b.txt>",
          "blob_id": "obj-cas-imm-7c8b22f12f92",
          "size_enc": "<encrypted: 32>" }
      ]
    }

  Requests so far: 4
```

Each folder expand is one request. The UI stays responsive because it only
loads what the user is looking at.

### Step A.6: User Downloads a File (1 request)

User clicks `adding file-1.txt`:

```
  POST /api/vault/batch/12cpxeq9
  {
    "operations": [
      { "op": "read", "file_id": "bare/data/obj-cas-imm-aeaa531d8450" }
    ]
  }

  Decrypt: "file-1.txt\n"

  Requests so far: 5
```

### Step A.7: User Downloads All Files (1 request per unexpanded folder)

If the user wants everything, the client fetches any unexpanded sub-trees
then batch-reads all blobs:

```
  POST /api/vault/batch/12cpxeq9
  {
    "operations": [
      { "op": "read", "file_id": "bare/data/obj-cas-imm-aeaa531d8450" },
      { "op": "read", "file_id": "bare/data/obj-cas-imm-7c8b22f12f92" }
    ]
  }

  Total for open + expand all + download all: 3 + depth + 1 blob-batch
```

### Summary: Vault Open Request Counts (Tree-Based, No Indexes)

```
  See root folder:                          3 requests (ref + commit + root tree)
  Expand one sub-folder:                    +1 request per folder
  Download one file:                        +1 request
  Download all root files:                  +1 request (batch)
  See everything + download everything:     3 + depth + 1 requests

  For a vault with 50 top-level files, 0 folders:
    3 + 0 + 1 = 4 requests (see root + download all)

  For a vault with 50 files across 5 folders:
    3 + 5 + 1 = 9 requests (see everything + download all)

  But a typical browse session:
    3 to see root, +1 per folder the user actually opens, +1 per file downloaded
    Most users open 1-2 folders and download 1-3 files = 5-8 requests total
```

## Scenario B: Full Vault Overview With Branch Discovery

**Without indexes (first implementation):**

```
  Request 1: batch read ref
  Request 2: batch read commit
  Request 3: batch read root tree + branch index
    → root-level files and folders + all branch metadata

  Total: 3 requests for root level + branches
  Each additional folder level: +1 request
```

**With indexes (future optimisation):**

```
  POST /api/vault/batch/12cpxeq9
  {
    "operations": [
      { "op": "read", "file_id": "bare/refs/ref-pid-muw-b1d1771f9fb5" },
      { "op": "read", "file_id": "bare/indexes/idx-pid-muw-{headSnapshotFileId}" },
      { "op": "read", "file_id": "bare/indexes/idx-pid-muw-{commitLogFileId}" },
      { "op": "read", "file_id": "bare/indexes/idx-pid-muw-48c9319a269c" }
    ]
  }

  One request returns:
    ✓ HEAD commit ID (from ref)
    ✓ Complete FLAT file listing — ALL files at ALL depths (from snapshot)
    ✓ Full commit history (from commit log)
    ✓ All branch metadata (from branch index)

  Total: 1 request for a complete vault overview
```

The snapshot index is the key difference: it contains a pre-computed flat
listing of every file in the vault (all depths), not just the root tree's
direct children. This is a denormalized cache built during push. It enables
the vault browser to render the full file tree, commit history, and branch
listing from a single API call.

**But for the first implementation,** tree-based on-demand loading is the
correct approach. It works without any indexes, it matches how UIs naturally
render (show root, expand on demand), and it's the most bandwidth-efficient
pattern for typical browse sessions where the user only opens a few folders.

## Scenario C: Efficient Pull

### Step C.1: Check for Remote Changes (1 request)

```
  POST /api/vault/batch/12cpxeq9
  {
    "operations": [
      { "op": "read", "file_id": "bare/refs/ref-pid-muw-b1d1771f9fb5" }
    ]
  }

  Decrypt: { "commit_id": "obj-cas-imm-NEWCOMMIT" }

  Compare with local tracking.json:
    Local HEAD was "obj-cas-imm-4602b76dd1ad"
    Remote HEAD is "obj-cas-imm-NEWCOMMIT"
    → Remote has new commits

  Requests so far: 1
```

### Step C.2: Download New Commit + Root Tree (1 request)

```
  POST /api/vault/batch/12cpxeq9
  {
    "operations": [
      { "op": "read", "file_id": "bare/data/obj-cas-imm-NEWCOMMIT" }
    ]
  }

  Response:
    commit → decrypt → get tree_id, parents, message_enc

  Requests so far: 2
```

### Step C.3: Compare Trees (1 request + depth)

```
  POST /api/vault/batch/12cpxeq9
  {
    "operations": [
      { "op": "read", "file_id": "bare/data/obj-cas-imm-{NEW_ROOT_TREE}" }
    ]
  }

  Compare root tree entries against local root tree:
    - Sub-tree with same obj-cas-imm ID → unchanged folder, skip entirely
    - Sub-tree with different ID → folder has changes, fetch sub-tree
    - Blob with same ID → unchanged file, skip
    - Blob with different ID → changed file, need to fetch
    - New entry → added file/folder
    - Missing entry → deleted file/folder

  This is where the sub-tree model pays off: if an-folder/ has the same
  tree_id in both versions, skip it entirely — no need to check individual
  files inside it. Only recurse into changed sub-trees.

  Requests so far: 3 + changed_depth_levels
```

### Step C.4: Download Changed Blobs (1 request)

```
  POST /api/vault/batch/12cpxeq9
  {
    "operations": [
      { "op": "read", "file_id": "bare/data/obj-cas-imm-{changed_blob_1}" },
      { "op": "read", "file_id": "bare/data/obj-cas-imm-{changed_blob_2}" }
    ]
  }

  Total pull: 3 + changed_depth + 1 requests
    1 batch: check ref
    1 batch: new commit
    1+ batch: compare trees (root + changed sub-trees only)
    1 batch: download changed blobs

  For a vault with 50 files across 5 folders, 1 file changed:
    1 (ref) + 1 (commit) + 1 (root tree) + 1 (changed folder tree) + 1 (blob) = 5 requests
    Not 50+ requests — the sub-tree shortcut skips 4 unchanged folders entirely.
```

**With snapshot index (future optimisation):** Pull reduces to 3 requests
(ref + commit + snapshot gives the full file listing with blob IDs, then
1 batch for changed blobs). The snapshot eliminates the tree walk entirely.

## Scenario E: Clone

### Understanding Clone Modes

With batch read and deterministic indexes, the scenarios where a full clone is
needed become less common. A full clone means "download the entire bare/ structure
for offline access." This matters for:

- Working offline (airplane, no connectivity)
- Creating a backup zip
- Transferring a vault to another storage backend

For most other operations — browsing, editing, committing, pushing, merging —
the client can work with a **lite clone** that fetches objects on demand. The
important insight: the same `bare/` directory structure and the same code paths
work in both modes. The only difference is whether objects are fetched eagerly
(full clone) or lazily (lite clone).

### E.1: Full Clone (Offline Copy)

The fastest full clone is a **zip download** — one request for the entire
`bare/` folder as a compressed archive:

```
  GET /api/vault/download/12cpxeq9
  Header: x-sgraph-access-token: {token}

  Response: ZIP file containing entire bare/ structure
  Client unpacks, creates local/, extracts working copy.

  Total: 1 request
```

This is the ideal for large vaults. The server zips `bare/` on the fly (or
caches a pre-built zip). The client gets everything in one round trip.

When a zip endpoint is not available, the client falls back to batch reads:

```
  Step 1: Batch read entry points (1 request)
    ref + snapshot + branch index
    → Now knows: all file blob IDs, all key IDs, HEAD commit ID

  Step 2: Batch read commit + keys (1 request)
    HEAD commit + all keys from branch index
    → Now knows: root tree ID

  Step 3: Batch read root tree (1 request)
    → Now knows: sub-tree IDs for depth 1

  Step 4: Batch read sub-trees at depth 1 (1 request per depth level)
    → Now knows: sub-tree IDs for depth 2
    → Continue until all tree IDs discovered

  Step N: Batch read all blobs (1 request)
    All blob IDs were known from Step 1 (snapshot). This step
    downloads the actual content.

  Step N+1: Batch write to register new clone branch (1 request)

  Total: 4 + max_folder_depth requests
```

For a vault with `a/b/c/d/file.txt` (4 levels deep), this is 8 requests.
For a flat vault (all files at root), this is 4 requests.

The tree walk is needed because tree object IDs are content-addressed — you
can't know a sub-tree's ID until you've read its parent tree. The snapshot
gives you blob IDs (for the working copy) but not the intermediate tree IDs
(needed for the bare/ object store).

**Future optimisation — object manifest index:**

An additional deterministic index at `idx-pid-muw-{object_manifest_id}`
could list ALL object IDs reachable from HEAD:

```json
{
  "schema": "object_manifest_v1",
  "commit_id": "obj-cas-imm-4602b76dd1ad",
  "objects": [
    "obj-cas-imm-4602b76dd1ad",
    "obj-cas-imm-867f9d7b56e1",
    "obj-cas-imm-{sub_tree_hash}",
    "obj-cas-imm-aeaa531d8450",
    "obj-cas-imm-7c8b22f12f92"
  ]
}
```

With this manifest, clone becomes constant-time:

```
  Step 1: Batch read ref + snapshot + branch index + manifest (1 request)
  Step 2: Batch read ALL objects from manifest + keys (1 request)
  Step 3: Batch write to register branch (1 request)
  Total: 3 requests (constant, regardless of depth or file count)
```

This manifest could be computed:
- By the CLI during push (add to the batch write)
- By the server as a background GC/maintenance task
- By the cloning client after a full clone (and pushed back for future cloners)

This is a **future optimisation** — the tree-walking approach works correctly
without it, and the depth penalty is small for most real vaults.

### E.2: Lite Clone (On-Demand Fetch)

A lite clone creates the `bare/` directory structure locally but does NOT
download all objects upfront. Instead, it fetches objects on demand when they
are needed by a command (status, commit, push, merge, log).

```
  Step 1: Batch read entry points (1 request)
    ref + snapshot + branch index
    → Create local bare/refs/, bare/indexes/ with downloaded files
    → Create empty bare/data/, bare/keys/

  Step 2: Batch write to register branch (1 request)

  Step 3: Extract working copy from snapshot (1 request)
    Batch read all blobs listed in the snapshot
    → Write to working directory
    → Store downloaded blobs in bare/data/

  Total: 3 requests
```

The critical insight: **the `bare/` structure and all CLI code paths are
identical for full and lite clones.** The only difference is a transparent
interception layer in the object store:

```python
class Vault__Object_Store:
    def load(self, object_id):
        # Try local first
        if self._exists_locally(object_id):
            return self._read_local(object_id)

        # Not local — fetch from server, cache locally, then return
        data = self.api.read(self.vault_id, f'bare/data/{object_id}')
        self._write_local(object_id, data)
        return data
```

This means:
- `sgit status` — works immediately (compares working dir against committed tree)
- `sgit commit` — works immediately (only needs to read blobs for changed files,
  which were downloaded during working copy extraction)
- `sgit push` — works (fetches any missing objects needed for delta computation)
- `sgit log` — fetches commit objects on demand as it walks the chain (or uses
  the commit log index, which was downloaded at clone time)
- `sgit merge` — fetches tree objects on demand as it performs three-way merge

Over time, frequently accessed objects accumulate locally. A lite clone
gradually becomes a full clone through normal usage. The user never notices
the difference — the same commands work, they just trigger background fetches
when needed.

**When lite clone is sufficient:**
- Working on a vault, making changes, pushing them
- Browsing vault contents from the CLI
- Participating in a collaboration workflow
- Any scenario with network access

**When full clone is needed:**
- Working offline (airplane mode)
- Creating a backup
- Exporting as a zip for transfer
- Regulatory requirement for local data retention

## Scenario F: Deterministic Indexes (Future Optimisation)

**These indexes are NOT part of the first implementation.** They are documented
here to show the optimisation path. The tree-based on-demand flow (Scenarios

batch write. They are denormalized caches — the object graph (tree objects,
commit objects) remains the source of truth. If an index is missing or stale,

## Revised Request Budgets (Final)

| Operation | Old (individual reads) | New: tree-based (first impl) | Future: with indexes |
|-----------|----------------------|-----------------------------|--------------------|
| **See root folder** | 6+ individual requests | **3 requests** (ref + commit + root tree) | **1 request** (ref + snapshot) |
| **Expand one sub-folder** | N/A (flat tree) | **+1 request** per folder | **0** (snapshot has all files) |
| **Download one file** | +1 request | **+1 request** | **+1 request** |
| **See everything + download all** | 6 + N_files requests | **3 + depth + 1** requests | **2 requests** (snapshot + blob batch) |
| **Full overview (+ history + branches)** | 6 + N_commits + N_branches | **3 + depth** (+ commit chain walk) | **1 request** (ref + snapshot + log + index) |
| **Pull (check for changes)** | 1 request | **1 request** | **1 request** |
| **Pull (download delta)** | 1 + N_changed | **3 + changed_depth + 1** | **3 requests** (ref + commit + snapshot + blobs) |
| **Push** | 2 requests | **2 requests** | **2 requests** (+ index updates in batch) |
| **Full clone (zip)** | 1 list + N_all_objects | **1 request** | **1 request** |
| **Full clone (batch read)** | 1 list + N_all_objects | **4 + depth** requests | **3 requests** (with manifest) |
| **Lite clone** | N/A (not supported) | **3 requests** | **3 requests** |
| **Typical browse session** | 6+ requests | **5-8 requests** (root + 1-2 folders + 1-3 files) | **2-3 requests** |

listing is needed programmatically.

## Key Insight: On-Demand Tree Loading is the Primary Design

The sub-tree model naturally supports on-demand loading: read the root tree,
show the user what's there, fetch sub-trees only when the user expands a folder.
This is not a limitation — it is the **correct UX and performance pattern.**

| Operation | Needs full clone? | What it actually needs (first impl) |
|-----------|------------------|-------------------------------------|
| Browse root folder | No | Ref + commit + root tree (3 requests) |
| Expand a sub-folder | No | 1 sub-tree read per folder |
| Download a file | No | 1 blob read |
| View commit history | No | Walk commit chain (N requests, or 1 with commit log index later) |
| Commit changes | No | Working copy + changed blobs (local) |
| Push | No | Delta objects + refs (already local from commit) |
| Pull | No | Ref + commit + tree comparison + changed blobs |
| Merge | No | Tree objects on demand (lite clone fetch) |
| Work offline | **Yes** | Full bare/ needed |
| Create backup zip | **Yes** | Full bare/ needed |
| Transfer vault | **Yes** | Full bare/ needed |

Most users interact with 1-2 folders and a handful of files per session.
Loading the entire vault upfront wastes bandwidth and time. The tree-based
approach loads exactly what the user asks to see.

The deterministic indexes (snapshot, commit log, object manifest) are **future
optimisations** for specific use cases: CLI `status` (needs the full file list
to compare), CLI `clone` (needs all blob IDs), `log --all` (needs full history),
and the vault browser's "show everything" view. They don't change the primary
flow — they accelerate the cases where the full listing is genuinely needed.
