# Architecture Note: Storage Backend Mapping

**Version:** v0.16.3  
**Date:** 2026-03-17  
**Role:** Architect  
**Status:** Design exploration — mapping storage backends to the four-layer model  
**Ref:** v0.16.3 four-layer model, v0.16.3 simulation, v0.13.30 arch spec

> **Note:** This document captures ideas and concepts for future reference.
> It is NOT a specification for immediate implementation. The current backends
> (filesystem for CLI, S3 + Lambda for server) are the only ones being built
> today. The DynamoDB, PostgreSQL, Redis, and graph database mappings are
> documented here to show that the vault protocol is storage-agnostic by
> design — the self-describing ID schema and batch API work with any backend.
> These alternative backends may be implemented in the future if there is
> customer demand (e.g., DynamoDB for sub-5ms vault opens, Neo4j for
> server-side graph traversal), or by others since the project is open source.

---

## Overview

The vault protocol (layer 1) requires exactly three operations from its storage
backend:

```
  put(vault_id, file_id, bytes)                        → store
  get(vault_id, file_id) → bytes                       → retrieve
  conditional_put(vault_id, file_id, expected, new)    → compare-and-swap
```

Plus two optional operations for efficiency:

```
  batch_get(vault_id, [file_id, ...]) → [bytes, ...]   → batch retrieve
  batch_put(vault_id, [(file_id, bytes), ...])          → batch store
```

And one for administration:

```
  list(vault_id, prefix) → [file_id, ...]              → enumerate
```

**That's it.** Any system that implements these operations can host a vault.
The self-describing ID schema (`{type}-{derivation}-{mutability}-{hex_id}`)
gives the storage backend enough information to optimise without reading
any encrypted content.

This document maps specific storage technologies to the vault protocol,
showing how each technology's strengths align with different operational
modes (A through D from the four-layer model).

## The Self-Describing ID as a Storage Hint

The storage backend can parse the filename to make intelligent decisions
without any vault-specific logic:

```
  ref-pid-muw-b1d1771f9fb5
  │   │   │
  │   │   └── muw = multi-writer mutable
  │   │       → no caching, require compare-and-swap on writes
  │   │       → store in hot/fast tier
  │   │
  │   └── pid = pseudo-deterministic
  │       → ID can be computed by clients (entry point)
  │       → may be read on every operation (hot path)
  │
  └── ref = reference pointer
      → small payload (< 100 bytes typically)
      → read-heavy, write-occasional

  obj-cas-imm-aeaa531d8450
  │   │   │
  │   │   └── imm = immutable
  │   │       → cache forever, write-once
  │   │       → can be replicated aggressively
  │   │       → idempotent writes (safe to retry)
  │   │
  │   └── cas = content-addressed
  │       → same ID = same bytes, always
  │       → deduplication is free
  │
  └── obj = data object
      → variable payload size (bytes to megabytes)
      → read frequency varies (hot blobs vs old commits)
```

A generic storage backend could implement a simple rules engine:

```python
  def storage_policy(file_id):
      parts = file_id.split('-')
      mutability = parts[2]  # imm, snw, muw

      if mutability == 'imm':
          return dict(cache=True, cache_ttl='forever', cas_required=False)
      elif mutability == 'snw':
          return dict(cache=True, cache_ttl='5s', cas_required=False)
      elif mutability == 'muw':
          return dict(cache=False, cas_required=True)
```

## Backend 1: Filesystem (Current CLI)

### How It Maps

```
  put(vault_id, file_id, bytes)  →  write file to .sg_vault/bare/{folder}/{file_id}
  get(vault_id, file_id)         →  read file from .sg_vault/bare/{folder}/{file_id}
  conditional_put(...)           →  read + compare + write (not atomic, but single-user)
  batch_get(...)                 →  N individual file reads
  list(vault_id, prefix)         →  os.listdir() with filter
```

### Strengths and Limitations

| Property | Assessment |
|----------|-----------|
| Simplicity | Excellent — just files on disk |
| Batch read | Poor — one syscall per file, no parallelism |
| Compare-and-swap | Not atomic — but acceptable for single-user CLI |
| Large vaults | Poor — thousands of small files strain inodes |
| Portability | Excellent — zip the folder, done |
| Layer support | All modes (A-D) — the filesystem doesn't care about content |

### Applicable Modes

All modes (A-D). The filesystem is content-agnostic. The CLI currently uses this
backend and it works for all scenarios up to moderate vault sizes.

### Optimisation Path

Replace the bare filesystem with **SQLite** — one `.sg_vault/vault.db` file
instead of hundreds of small files in `bare/`:

```sql
  CREATE TABLE vault_objects (
      file_id    TEXT PRIMARY KEY,
      payload    BLOB NOT NULL,
      created_at INTEGER NOT NULL
  );

  -- Batch read: single query
  SELECT file_id, payload FROM vault_objects
  WHERE file_id IN ('obj-cas-imm-abc123', 'obj-cas-imm-def456', ...);

  -- Compare-and-swap: atomic with WHERE clause
  UPDATE vault_objects
  SET payload = :new_payload
  WHERE file_id = :file_id AND payload = :expected_payload;

  -- Immutable write: safe to retry
  INSERT OR IGNORE INTO vault_objects (file_id, payload, created_at)
  VALUES (:file_id, :payload, :now);
```

Same API surface as the filesystem backend. Faster bulk operations. Single
file for backup/copy. No inode limits. Transaction support for atomic
multi-file writes. The `bare/` folder structure becomes a virtual view
over the database rows (or the CLI reconstructs it on export/zip).

## Backend 2: Amazon S3 (Current Server)

### How It Maps

```
  put(vault_id, file_id, bytes)  →  S3 PutObject to
                                     s3://{bucket}/transfers/vault/{vault_id}/{file_id}/payload
  get(vault_id, file_id)         →  S3 GetObject
  conditional_put(...)           →  Emulated in Lambda (read + compare + write)
  batch_get(...)                 →  Multiple S3 GetObject calls (AWS-to-AWS, parallel)
  list(vault_id, prefix)         →  S3 ListObjectsV2 with prefix
```

### Strengths and Limitations

| Property | Assessment |
|----------|-----------|
| Scalability | Excellent — unlimited objects, automatic sharding |
| Batch read | Good — Lambda can parallel-fetch from S3 (~1-5ms each) |
| Compare-and-swap | Poor — not native, must emulate (race condition window) |
| Cost | Moderate — per-request pricing, storage is cheap |
| Latency | Moderate — 5-50ms per S3 call within AWS |
| Layer support | All modes (A-D) — S3 stores opaque bytes |

### Applicable Modes

All modes (A-D). S3 is content-agnostic. The `bare/` folder structure maps
directly to S3 key prefixes.

### Caching with Mutability Hints

A Lambda-side cache layer can use the ID schema:

```python
  class S3_With_Cache:
      def get(self, vault_id, file_id):
          mutability = file_id.split('-')[2]

          if mutability == 'imm':
              cached = self.cache.get(f'{vault_id}/{file_id}')
              if cached:
                  return cached  # immutable — cache hit is always valid

          result = self.s3.get_object(...)

          if mutability == 'imm':
              self.cache.set(f'{vault_id}/{file_id}', result, ttl=None)  # forever

          return result
```

For batch reads, immutable objects (the majority of requests) come from cache.
Only `muw` objects hit S3.

## Backend 3: Amazon DynamoDB

### How It Maps

```
  Table: sg_vault_objects
  Partition key: vault_id (String)
  Sort key:      file_id (String)
  Attributes:    payload (Binary), created_at (Number)

  put(vault_id, file_id, bytes)
    → PutItem { vault_id, file_id, payload, created_at }

  get(vault_id, file_id)
    → GetItem { vault_id, file_id } → payload

  conditional_put(vault_id, file_id, expected, new)
    → PutItem with ConditionExpression:
      "payload = :expected"
    → Native atomic compare-and-swap — no race conditions

  batch_get(vault_id, [file_ids])
    → BatchGetItem — up to 100 items per call, native
    → Single-digit millisecond latency for the entire batch

  list(vault_id, prefix)
    → Query with begins_with(file_id, :prefix)
```

### Strengths and Limitations

| Property | Assessment |
|----------|-----------|
| Batch read | **Excellent** — `BatchGetItem` is native, up to 100 items, ~5ms total |
| Compare-and-swap | **Excellent** — `ConditionExpression` is native, atomic, no race conditions |
| Latency | **Excellent** — single-digit millisecond reads and writes |
| Scalability | Excellent — auto-scaling, no provisioning needed (on-demand mode) |
| Cost | Moderate — per-request pricing, can be expensive for high-volume |
| Payload size | Limited — 400KB max item size (blobs > 400KB need S3 overflow) |
| Layer support | All modes (A-D) |

### Why DynamoDB is a Strong Fit

The vault protocol maps almost perfectly to DynamoDB's data model:

1. **Partition key = vault_id** — all objects for a vault are co-located
2. **Sort key = file_id** — efficient prefix queries for listing
3. **BatchGetItem** — native batch read, exactly what we need
4. **ConditionExpression** — native compare-and-swap, exactly what `muw` files need
5. **On-demand pricing** — matches vault usage (bursty, not constant)

The self-describing IDs enable DynamoDB-native optimisations:

```python
  # Immutable write — safe to retry, no overwrite
  table.put_item(
      Item={'vault_id': vid, 'file_id': fid, 'payload': data},
      ConditionExpression='attribute_not_exists(file_id)'
  )

  # Multi-writer compare-and-swap — atomic
  table.put_item(
      Item={'vault_id': vid, 'file_id': fid, 'payload': new_data},
      ConditionExpression='payload = :expected',
      ExpressionAttributeValues={':expected': expected_data}
  )
```

### DynamoDB + S3 Hybrid

For blobs larger than 400KB (DynamoDB's item size limit), use S3 as overflow:

```
  obj-cas-imm-{hash} with payload < 400KB   → DynamoDB (fast, native batch)
  obj-cas-imm-{hash} with payload >= 400KB  → S3 (large object storage)
                                                DynamoDB stores S3 pointer
```

The `imm` flag means the overflow decision is made once at write time
and never changes. The batch read handler checks DynamoDB first; if the
payload is an S3 pointer, it fetches from S3 transparently.

### Performance Estimate

```
  Vault open (1 batch read: ref + snapshot):
    BatchGetItem with 2 items → ~2ms server-side
    + network latency (~50ms from browser, ~5ms from Lambda)
    = ~52ms from browser, ~7ms from Lambda

  Full clone (3 batch reads):
    3 × BatchGetItem → ~6ms server-side
    + network → ~56ms from browser, ~11ms from Lambda

  Push (1 batch write):
    BatchWriteItem → ~5ms server-side
    + ConditionExpression on ref → ~2ms
    + network → ~57ms from browser
```

### Applicable Modes

All modes (A-D). DynamoDB stores binary payloads without interpretation.

## Backend 4: PostgreSQL

### How It Maps

```sql
  CREATE TABLE vault_objects (
      vault_id    VARCHAR(16)   NOT NULL,
      file_id     VARCHAR(64)   NOT NULL,
      payload     BYTEA         NOT NULL,
      created_at  BIGINT        NOT NULL,
      PRIMARY KEY (vault_id, file_id)
  );

  -- Single read
  SELECT payload FROM vault_objects
  WHERE vault_id = $1 AND file_id = $2;

  -- Batch read
  SELECT file_id, payload FROM vault_objects
  WHERE vault_id = $1 AND file_id = ANY($2::text[]);

  -- Immutable write (insert-only)
  INSERT INTO vault_objects (vault_id, file_id, payload, created_at)
  VALUES ($1, $2, $3, $4)
  ON CONFLICT (vault_id, file_id) DO NOTHING;

  -- Compare-and-swap (atomic)
  UPDATE vault_objects
  SET payload = $3
  WHERE vault_id = $1 AND file_id = $2
  AND payload = $4
  RETURNING file_id;
  -- Returns 0 rows if CAS failed (payload didn't match)

  -- List
  SELECT file_id FROM vault_objects
  WHERE vault_id = $1 AND file_id LIKE $2 || '%';
```

### Strengths and Limitations

| Property | Assessment |
|----------|-----------|
| Batch read | Good — `ANY(array)` query, single round trip |
| Compare-and-swap | Good — `UPDATE ... WHERE payload = $expected` is atomic |
| Scalability | Moderate — needs connection pooling, sharding for very large deployments |
| Cost | Low — self-hosted or managed (RDS), predictable pricing |
| Payload size | Good — `BYTEA` supports up to 1GB per row |
| Ecosystem | Excellent — widely deployed, well-understood, rich tooling |
| Layer support | All modes (A-D) |

### Mode B/C/D Advantage: Queryable Metadata

In modes B, C, and D where tree/commit metadata is plaintext or structurally
visible, PostgreSQL can index the decrypted fields:

```sql
  -- Mode C/D: commits table with extracted metadata
  CREATE TABLE vault_commits (
      vault_id    VARCHAR(16)   NOT NULL,
      file_id     VARCHAR(64)   NOT NULL,
      payload     BYTEA         NOT NULL,
      -- Extracted metadata (mode C/D: plaintext; mode B: decrypted by server)
      message     TEXT,
      timestamp_ms BIGINT,
      branch_id   VARCHAR(64),
      tree_id     VARCHAR(64),
      parent_ids  TEXT[],
      PRIMARY KEY (vault_id, file_id)
  );

  CREATE INDEX idx_commits_time ON vault_commits (vault_id, timestamp_ms DESC);
  CREATE INDEX idx_commits_branch ON vault_commits (vault_id, branch_id);

  -- "Show me all commits on this branch in the last hour"
  SELECT * FROM vault_commits
  WHERE vault_id = $1 AND branch_id = $2 AND timestamp_ms > $3
  ORDER BY timestamp_ms DESC;
```

In mode A (full encryption), these indexes don't exist — the server stores
only opaque `BYTEA` payloads. In modes B-D, the server can extract and index
metadata, enabling server-side queries that eliminate the need for client-side
indexes (head snapshot, commit log).

### Applicable Modes

| Mode | How PostgreSQL is used |
|------|----------------------|
| A (full encryption) | Pure key-value store — `BYTEA` payloads, no indexing |
| B (structural visibility) | Key-value + metadata indexes on tree/commit fields |
| C (signed plaintext) | Full-text search on commit messages, file path queries |
| D (pure version control) | Full relational model — joins, aggregates, reports |

## Backend 5: Redis

### How It Maps

```
  put(vault_id, file_id, bytes)
    → SET vault:{vault_id}:{file_id} <bytes>

  get(vault_id, file_id)
    → GET vault:{vault_id}:{file_id}

  batch_get(vault_id, [file_ids])
    → MGET vault:{vault_id}:{fid1} vault:{vault_id}:{fid2} ...
    → Sub-millisecond, native

  conditional_put(vault_id, file_id, expected, new)
    → WATCH vault:{vault_id}:{file_id}
      MULTI
      SET vault:{vault_id}:{file_id} <new>
      EXEC
    → Native optimistic locking

  list(vault_id, prefix)
    → SCAN with MATCH vault:{vault_id}:{prefix}*
```

### Strengths and Limitations

| Property | Assessment |
|----------|-----------|
| Batch read | **Excellent** — `MGET` is native, sub-millisecond |
| Latency | **Excellent** — sub-millisecond for cached data |
| Compare-and-swap | Good — `WATCH`/`MULTI` is native optimistic locking |
| Persistence | Moderate — requires AOF/RDB configuration for durability |
| Cost | High — everything in memory, expensive for large vaults |
| Payload size | Limited — practical limit ~512MB per key, but memory cost |
| Layer support | All modes (A-D) |

### Best Use: Caching Layer

Redis excels as a **caching tier in front of S3 or DynamoDB**, not as
primary storage:

```
  Client → Lambda → Redis (cache) → S3/DynamoDB (primary)

  Batch read flow:
    1. MGET all requested keys from Redis
    2. For cache misses, batch-fetch from primary storage
    3. Write misses back to Redis with TTL based on mutability:
       imm → no TTL (cache forever)
       snw → TTL 5s
       muw → don't cache (always fetch from primary)
    4. Return assembled results
```

With this architecture, a vault open (1 batch read of ref + snapshot) hits
Redis on the second request and returns in < 1ms server-side.

## Backend 6: Graph Database (Neo4j / Amazon Neptune)

### How It Maps

This is where the mapping gets interesting, because a graph DB can operate
at two levels depending on the mode.

### Mode A: Graph DB as Key-Value Store (Zero-Knowledge Preserved)

```cypher
  // Store: node per object, payload as property
  CREATE (o:VaultObject {
      vault_id: 'aaaabbp1',
      file_id: 'obj-cas-imm-aeaa531d8450',
      payload: <encrypted bytes>
  })

  // Retrieve
  MATCH (o:VaultObject {vault_id: 'aaaabbp1', file_id: $fileId})
  RETURN o.payload

  // Batch retrieve
  MATCH (o:VaultObject {vault_id: 'aaaabbp1'})
  WHERE o.file_id IN $fileIds
  RETURN o.file_id, o.payload
```

No edges. No graph structure. Just using the graph DB as a key-value store
with good batch read support. The server sees only opaque nodes.

**Value over DynamoDB/PostgreSQL:** minimal. Graph DB as pure KV store
is not where it shines.

### Mode B: Structural Edges, Encrypted Content

In mode B, the server can see graph structure (decrypted tree/commit
metadata) but not file content (blobs remain encrypted). The graph DB
stores explicit edges:

```cypher
  // Ref points to commit
  CREATE (r:Ref {file_id: 'ref-pid-muw-b1d1771f9fb5', vault_id: 'aaaabbp1'})
  CREATE (c:Commit {file_id: 'obj-cas-imm-4602b76dd1ad', vault_id: 'aaaabbp1',
                    message: 'added file inside folder',
                    timestamp_ms: 1773703777318,
                    branch_id: 'branch-clone-8909460c441869a6'})
  CREATE (r)-[:POINTS_TO]->(c)

  // Commit has tree
  CREATE (t:Tree {file_id: 'obj-cas-imm-867f9d7b56e1', vault_id: 'aaaabbp1'})
  CREATE (c)-[:HAS_TREE]->(t)

  // Tree contains entries
  CREATE (b:Blob {file_id: 'obj-cas-imm-aeaa531d8450', vault_id: 'aaaabbp1',
                  name: 'adding file-1.txt', size: 11,
                  payload: <ENCRYPTED CONTENT>})
  CREATE (t)-[:CONTAINS {name: 'adding file-1.txt'}]->(b)

  // Sub-tree
  CREATE (st:Tree {file_id: 'obj-cas-imm-{sub_hash}', vault_id: 'aaaabbp1'})
  CREATE (t)-[:CONTAINS {name: 'an-folder'}]->(st)

  // Commit parent
  CREATE (c)-[:PARENT]->(parent_commit)
```

**Now the server can answer graph queries natively:**

```cypher
  // Clone: all objects reachable from HEAD (replaces object manifest index)
  MATCH (r:Ref {file_id: 'ref-pid-muw-b1d1771f9fb5'})-[:POINTS_TO]->(c:Commit)
  MATCH path = (c)-[*]->(obj)
  RETURN DISTINCT obj.file_id, obj.payload

  // LCA: find common ancestor (replaces client-side chain walking)
  MATCH (c1:Commit {file_id: $commit1}),
        (c2:Commit {file_id: $commit2}),
        path1 = (c1)-[:PARENT*]->(ancestor),
        path2 = (c2)-[:PARENT*]->(ancestor)
  RETURN ancestor.file_id
  ORDER BY length(path1) + length(path2)
  LIMIT 1

  // File history: all commits that changed a specific file
  MATCH (c:Commit)-[:HAS_TREE]->()-[:CONTAINS*]->(b:Blob {name: 'README.md'})
  RETURN c.message, c.timestamp_ms, b.file_id
  ORDER BY c.timestamp_ms DESC

  // Diff between two commits: find changed objects
  MATCH (c1:Commit {file_id: $commit1})-[:HAS_TREE]->(t1),
        (c2:Commit {file_id: $commit2})-[:HAS_TREE]->(t2)
  MATCH (t1)-[:CONTAINS]->(obj1)
  MATCH (t2)-[:CONTAINS]->(obj2)
  WHERE obj1.name = obj2.name AND obj1.file_id <> obj2.file_id
  RETURN obj1.name, obj1.file_id AS old, obj2.file_id AS new
```

### Indexes Made Redundant by Graph Queries

| Client-side index | Graph query that replaces it |
|-------------------|----------------------------|
| HEAD snapshot (`idx-pid-muw-*`) | `MATCH (r:Ref)-[:POINTS_TO]->(c)-[:HAS_TREE]->()-[:CONTAINS*]->(obj) RETURN obj` |
| Commit log (`idx-pid-muw-*`) | `MATCH (c:Commit) WHERE c.vault_id = $v RETURN c ORDER BY c.timestamp_ms DESC` |
| Object manifest (`idx-pid-muw-*`) | `MATCH (r:Ref)-[*]->(obj) RETURN DISTINCT obj.file_id` |
| LCA computation | `shortestPath` between two commits via `:PARENT` edges |
| Branch listing | `MATCH (b:Branch) WHERE b.vault_id = $v RETURN b` |

In mode B with a graph DB, **all five deterministic indexes become optional.**
The graph DB computes them on the fly. The `bare/indexes/` folder could be
empty. The client still creates them (for portability to non-graph backends),
but the server never reads them.

### Mode C/D: Full Graph Visibility

In modes C and D (plaintext content), the graph DB also stores and indexes
file content. Full-text search, content-based queries, and cross-vault
analytics become possible:

```cypher
  // Search across all vaults for files containing "database"
  MATCH (b:Blob)
  WHERE b.content CONTAINS 'database'
  RETURN b.vault_id, b.name, b.file_id

  // Find all vaults with a README.md
  MATCH (t:Tree)-[:CONTAINS]->(b:Blob {name: 'README.md'})
  RETURN DISTINCT t.vault_id

  // Deduplication: find identical blobs across vaults
  MATCH (b1:Blob), (b2:Blob)
  WHERE b1.file_id = b2.file_id AND b1.vault_id <> b2.vault_id
  RETURN b1.vault_id, b2.vault_id, b1.name
```

## Backend Selection Guide

| Deployment scenario | Recommended backend | Mode | Reasoning |
|--------------------|--------------------|----|-----------|
| **CLI (local)** | Filesystem → SQLite | A | Simplest. Single-user. No server. |
| **SG/Send (current)** | S3 + Lambda | A | Already deployed. Scales. Cost-effective. |
| **SG/Send (fast)** | DynamoDB + Lambda | A | Native batch read + CAS. Sub-5ms vault open. |
| **SG/Send (cached)** | Redis + S3 + Lambda | A | Sub-ms for repeat reads. S3 for durability. |
| **Enterprise (private)** | PostgreSQL | B | Queryable metadata. Self-hosted. Familiar. |
| **Enterprise (graph)** | Neo4j / Neptune | B | Server-side traversal. Instant clone/LCA. No indexes needed. |
| **Open-source project** | PostgreSQL or SQLite | C/D | Full visibility. Searchable. Relational. |
| **Embedded in app** | SQLite | D | Single file. No server. Zero dependencies. |

### Hybrid Architectures

Multiple backends can serve different roles in the same deployment:

```
  Hot path (refs, indexes):       Redis or DynamoDB
  Bulk storage (blobs):           S3
  Metadata queries (mode B):      PostgreSQL or Neptune
  Local cache:                    SQLite or filesystem

  The batch read API abstracts all of this — the client sends one request,
  the server routes to the appropriate backend per file_id.
```

The mutability segment in the file ID tells the routing layer where to look:

```python
  def route_read(self, vault_id, file_id):
      mutability = file_id.split('-')[2]
      if mutability == 'imm':
          # Try Redis cache first, fall back to S3
          return self.redis.get(file_id) or self.s3.get(file_id)
      elif mutability in ('snw', 'muw'):
          # Always read from primary (DynamoDB) for mutable files
          return self.dynamodb.get(vault_id, file_id)
```

## Impact on the Vault Protocol

**None.** The vault protocol is unchanged regardless of backend. The client
sends the same batch read/write requests. The `bare/` folder structure is
the same. The self-describing IDs are the same. The encryption layers are
the same.

The storage backend is a **server-side implementation detail** that the client
never sees. A vault created on S3 can be cloned to DynamoDB, exported to
SQLite, queried via PostgreSQL, and traversed via Neo4j — all using the
same `bare/` structure and the same encrypted (or plaintext) payloads.

This is the power of the layer 1 protocol: **storage independence by design.**
