# Debrief & Case Study: Cross-Repo Analysis + Live Vault Inspection

**Version:** v0.16.3  
**Date:** 2026-03-17  
**Role:** Conductor / Architect  
**Type:** Workflow case study — reusable pattern for the agentic team

---

## Summary

A single Conductor/Architect session performed a comprehensive audit of the
vault implementation across two repos, four UI codebases, and a live S3
deployment. The session evolved from debugging a 404 error into designing a
complete architectural revision: self-describing file IDs, batch read API,
sub-tree model, on-demand tree loading, lite clone, a four-layer security
model, and a storage-agnostic backend architecture.

**Duration:** ~3 hours  
**Deliverables:** 9 documents (1 addendum, 4 impl plans, 1 simulation,
2 arch notes, this debrief)  
**Key finding:** The vault system is not just an encryption tool — it is a
general-purpose version control protocol with optional security layers

---

## The Pattern (Reusable)

### Phase 1: Multi-Repo Code Orientation

**What we did:**
1. Cloned both repos (`SGraph-AI__App__Send` + `SG_Send__CLI`)
2. Read master guidance, reality document, architecture specification
3. Read the same logical component across 4 codebases (vault crypto in
   Python CLI, JS vault browser v0.1.2, JS vault browser v0.1.3, JS user vault)
4. Identified three incompatible crypto models

**Key technique:** Cross-repo comparison of the same component reveals
incompatibilities invisible to single-repo review.

### Phase 2: Live Data Inspection via Vault Browser

**What we did:**
1. Human created a vault using CLI: `init`, `commit`, `push`
2. Human opened vault browser, entered vault key, shared screenshots
3. Architect annotated each screenshot with code paths

**What we saw:**
- Flat tree structure confirmed (no sub-trees)
- Dual-write artifacts on S3 (unexpected folders outside `bare/`)
- Plaintext `path` field alongside encrypted `name_enc`
- Branch index with real IDs traceable to code
- Commit signatures, parent chain, tree→blob links
- Timestamp display bug (ms treated as seconds → year 58176)

**The vault browser as architectural audit tool:** A tool built for user
transparency became the most powerful analysis tool in the session.

### Phase 3: Gap Analysis → Architecture → Challenge → Revise

1. **Gap analysis:** spec vs implementation (6 major departures found)
2. **Architecture proposal:** deterministic HMAC IDs + sub-tree model
3. **Human challenge:** "keep `bare/` identical everywhere" → corrected
   the proposal to put HMAC IDs inside `bare/`, not outside
4. **Simulation:** traced every API call for vault open, pull, push, clone
5. **Human challenge:** "root tree shows one folder level, not everything"
   → corrected the simulation to show on-demand tree loading as primary
6. **ID naming evolved:** through 3 iterations from bare hex → prefixed →
   four-segment `{type}-{derivation}-{mutability}-{id}`

### Phase 4: Emergent Insights

The iterative design process surfaced insights beyond the original scope:

- **Batch read API** — the server already had batch write; batch read
  transforms every operation's request count
- **Self-describing IDs** — encoding derivation + type + mutability in the
  filename enables server-side caching and compare-and-swap enforcement
  without any vault-specific logic
- **Lite clone** — on-demand fetch makes full clone the exception, not the rule
- **Four-layer model** — encryption and signing are optional layers on a
  general-purpose version control protocol
- **Storage-agnostic backends** — the protocol maps to filesystem, S3,
  DynamoDB, PostgreSQL, Redis, and graph databases
- **Graph DB mode** — with visible edges (mode B), all five deterministic
  indexes become redundant because the DB computes them natively

## What Went Right

1. **Reading the spec before the code** — precise gap identification
2. **Cross-repo comparison** — found three incompatible crypto models
3. **Live data inspection** — confirmed flat tree, found dual-write artifacts
4. **Human challenges to architecture** — prevented two structural mistakes
   (dropping `bare/` prefixes, and claiming "1 request for full vault")
5. **Simulation before code** — explored request budgets and discovered
   optimisations before writing any implementation
6. **Iterative naming design** — three rounds of feedback produced a schema
   that encodes derivation, type, AND mutability in every filename

## Implementation Order

1. **Backend: batch read + health check** — minimal, unblocks everything else
2. **CLI** — master implementation (schema cleanup → crypto → IDs → sub-tree → push merge → graph-walk clone → tracking → delete legacy)
3. **Vault browser** — consolidate on v0.1.2, update for sub-tree + encrypted fields + new IDs
4. **Vault UI** — migrate from RSA to passphrase model using shared library

## The Reusable Workflow

**Orient → Compare → Inspect live → Gap-analyse → Propose → Challenge → Revise → Document.**

Use this pattern when:
- Multiple implementations of the same concept exist
- A tool can inspect live data (use it as an audit tool)
- There's a spec AND an implementation (the gap is where findings live)
- An architectural decision affects multiple components

The key discipline: **let the human challenge every assumption.** The first
proposal is rarely the final one. Two corrections in this session (bare/
prefix preservation, on-demand tree loading as primary) prevented structural
mistakes from propagating into code.
