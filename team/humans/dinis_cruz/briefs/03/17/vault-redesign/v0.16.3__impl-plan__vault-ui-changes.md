# Vault UI Changes — Migrate to Git-Like Object Model

**Version:** v0.16.3  
**Date:** 2026-03-17  
**Role:** Architect → Dev  
**Ref:** v0.16.3 arch addendum, v0.16.3 simulation

---

## Scope

The user-facing vault page at `send.sgraph.ai/send/.../vault.html` uses RSA-4096
hybrid encryption — fundamentally incompatible with the CLI's passphrase model.

| | Vault UI (v0.1.7) | CLI (v0.8.5) |
|---|---|---|
| Encryption | RSA-4096 hybrid | AES-256-GCM + PBKDF2 |
| Key storage | Browser IndexedDB | Vault key (passphrase:vault_id) |
| Data model | Flat: settings + tree | Git-like: refs → commits → trees → blobs |
| Versioning | None (overwrites in-place) | Commit chain with parents |

**Decision:** Replace the RSA model with the CLI's passphrase model.
All three clients (CLI, vault browser, vault UI) converge on one data format.

---

## 1. Replace RSA with Passphrase Model

- Vault key is `{passphrase}:{vault_id}` (same as CLI)
- Key derivation uses PBKDF2 (same as CLI)
- Data model uses refs → commits → trees → blobs (same as CLI)
- No browser-side RSA key generation
- No IndexedDB key storage

---

## 2. Vault Open Flow

```
User enters vault key
  → derive keys (PBKDF2 + HMAC)
  → read ref-pid-muw-{refFileId} from bare/refs/
  → decrypt → commit_id
  → read obj-cas-imm-{commit_id} from bare/data/
  → decrypt → tree_id
  → read obj-cas-imm-{tree_id} from bare/data/
  → decrypt → root tree entries
  → render root level (files + expandable folders)
  → sub-trees loaded on demand when user clicks a folder
```

On-demand tree loading is the primary UX — same as the vault browser.

---

## 3. Vault Create Flow

```
User clicks "Create Vault"
  → generate passphrase + vault_id
  → derive keys (PBKDF2)
  → generate EC P-256 key pairs (named + clone branch)
  → create empty tree → encrypt → obj-cas-imm-{hash}
  → create init commit (signed by clone key) → obj-cas-imm-{hash}
  → write HEAD ref at ref-pid-muw-{hmac} in bare/refs/
  → create branch index → encrypt → idx-pid-muw-{hmac} in bare/indexes/
  → store keys at key-rnd-imm-{random} in bare/keys/
  → display vault key to user
```

**Vault Key UX:**
- Show key prominently after creation
- "Copy to clipboard" button
- Warning: "This is your only way to access this vault. We cannot recover it."
- Optional: save to browser localStorage (with clear warning about data loss)

---

## 4. File Operations (Sub-Tree Model)

### Upload

```
User selects file → read → encrypt blob → obj-cas-imm-{hash}
  → upload to bare/data/
  → build new sub-tree (encrypted name_enc, blob_id)
  → build new root tree (updated sub-tree_id for changed folder)
  → create commit (signed by clone branch key)
  → compare-and-swap update ref-pid-muw-{refFileId}
```

### Download

```
User clicks file in tree → blob_id already known from tree entry
  → read bare/data/obj-cas-imm-{blob_id}
  → decrypt → trigger browser download
  → filename from decrypted name_enc
```

### Folders

Folders are implicit — a tree entry with `tree_id` instead of `blob_id`.
"Create folder" in the UI sets a path prefix for the next upload.
The sub-tree object is created when the first file is added.

## 7. End State

| Client | Key derivation | Data model | Server addressing | ID format |
|--------|---------------|-----------|-------------------|----|
| CLI | PBKDF2 | Git-like (sub-tree) | `bare/` prefixed | `{type}-{derivation}-{mutability}-{id}` |
| Vault browser | PBKDF2 | Git-like (sub-tree) | `bare/` prefixed | `{type}-{derivation}-{mutability}-{id}` |
| Vault UI | PBKDF2 | Git-like (sub-tree) | `bare/` prefixed | `{type}-{derivation}-{mutability}-{id}` |

One vault, one format, any client.
