# Architecture Note: The Four-Layer Model

**Version:** v0.16.3  
**Date:** 2026-03-17  
**Role:** Architect  
**Status:** Architectural insight — captures the realisation that sg-vault is a  
general-purpose version control protocol with optional security layers  
**Ref:** v0.16.3 simulation, v0.13.30 arch spec

> **Note:** This document captures ideas and concepts that emerged during the
> architecture session. It is NOT a specification for immediate implementation.
> The current priority is Mode A (full zero-knowledge) — the only mode being
> built today. Modes B, C, and D are documented here to show the architectural
> potential of the layer 1 protocol. They may be implemented in the future if
> there is customer demand, or by others since the project is open source.
> The value of this document is in recognising that the architecture already
> supports these modes — no redesign would be needed to add them.

---

## The Insight

Through designing the self-describing ID schema (`{type}-{derivation}-{mutability}-{id}`),
batch read operations, deterministic indexes, and the lite clone model, we
discovered that what we've built is not fundamentally an encryption system.
It is a **content-addressed, portable, storage-agnostic version control protocol.**
Encryption and signing are features layered on top — not foundational requirements.

The protocol works at every combination of encryption and signing, producing
four distinct operational modes from the same codebase, the same `bare/`
structure, the same ID schema, and the same batch API.

---

## The Four Layers

```
  ┌─────────────────────────────────────────────────────────────┐
  │  Layer 4: Content Encryption                                │
  │  AES-256-GCM on blob payloads (file contents)              │
  │  Server cannot read files. Requires read_key to decrypt.    │
  ├─────────────────────────────────────────────────────────────┤
  │  Layer 3: Metadata Encryption                               │
  │  AES-256-GCM on tree entries (name_enc, size_enc) and      │
  │  commit messages (message_enc). Server cannot see filenames, │
  │  folder structure, or commit descriptions.                   │
  ├─────────────────────────────────────────────────────────────┤
  │  Layer 2: Commit Signing                                    │
  │  ECDSA P-256 signatures on every commit. Proves provenance  │
  │  — which branch/device/user created each commit.            │
  │  Immutable audit trail. Non-repudiation.                    │
  ├─────────────────────────────────────────────────────────────┤
  │  Layer 1: Content-Addressed Object Graph (THE PROTOCOL)     │
  │  refs → commits → trees → blobs                             │
  │  Sub-tree model (one tree per directory level)              │
  │  Self-describing IDs: {type}-{derivation}-{mutability}-{id} │
  │  Batch read/write API                                       │
  │  Deterministic indexes (snapshot, commit log, branch index) │
  │  bare/ as portable unit (local = server = zip)             │
  │  Lite clone with on-demand fetch                            │
  │  Compare-and-swap on multi-writer refs                      │
  └─────────────────────────────────────────────────────────────┘
```

Layers are independent and stack upward. Each layer adds capability without
changing anything below it. A client that only implements layer 1 can interoperate
with a vault created by a client that implements all four layers — it just
can't decrypt the content or verify signatures.

## The Four Modes

| Mode | Layers | Encryption | Signing | What server sees |
|------|--------|-----------|---------|-----------------|
| **A: Full zero-knowledge** | 1 + 2 + 3 + 4 | Everything | Yes | Opaque blobs only |
| **B: Structural visibility** | 1 + 2 + 4 | File content only | Yes | Graph structure + metadata, not content |
| **C: Signed plaintext** | 1 + 2 | None | Yes | Everything visible, provenance proven |
| **D: Pure version control** | 1 | None | No | Everything visible, no provenance |

### Mode A: Full Zero-Knowledge (Current SG/Send)

Every layer is active. The server stores only opaque encrypted blobs. It cannot
see filenames, folder structure, commit messages, file contents, or any
relationships between objects. Discovery requires the vault key. Content
requires the read key. Provenance is verified via ECDSA signatures.

```
  Server sees:
    obj-cas-imm-867f9d7b56e1    (opaque blob, 739 bytes)
    obj-cas-imm-aeaa531d8450    (opaque blob, 11 bytes)
    ref-pid-muw-b1d1771f9fb5    (opaque blob, 33 bytes)

  Server knows: file sizes and ID prefixes. Nothing else.
```

**Use cases:** Sensitive documents, legal files, healthcare data, financial
records, whistleblower protection, any scenario where the server operator
must not have access to content or metadata.

### Mode B: Structural Visibility

Layer 3 (metadata encryption) is removed. Tree entries and commit messages
are stored in plaintext JSON (still within the encrypted object envelope,
but readable after the object-level decryption). File content (layer 4)
remains encrypted.

```
  Server sees (if using graph DB with visible edges):
    Commit "added file inside folder" → tree with 2 entries
    Tree entry: "adding file-1.txt" → blob obj-cas-imm-aeaa531d8450
    Tree entry: "an-folder/" → sub-tree obj-cas-imm-{hash}

  Server cannot see: actual file contents (blobs are still AES-256-GCM)
```

In mode B with a graph database, the server can perform:
- Server-side clone (traverse graph, return all reachable objects)
- Server-side LCA computation (find common ancestor without client round-trips)
- Server-side file listing (read tree entries directly)
- Server-side search by filename (index tree entry names)
- Server-side commit history (walk parent chain)

The HEAD snapshot index, commit log index, and object manifest index become
**unnecessary** — the server computes them on the fly from the graph.

**Use cases:** Enterprise deployments where the server operator is trusted
with metadata but not content. Internal document management. Collaboration
platforms where search and discovery are server-side features.

### Mode C: Signed Plaintext

Layers 3 and 4 are removed. All content is stored in plaintext. Layer 2
(signing) remains — every commit is ECDSA-signed, providing an immutable,
cryptographically verifiable audit trail.

```
  Server sees:
    Commit "added file inside folder" (signed by branch-clone-890946...)
    Tree: adding file-1.txt → blob containing "file-1.txt\n"
    Blob: plaintext file content

  Server can verify: who committed what, when, and that no content was tampered with.
```

The signing layer provides:
- **Provenance:** every change traces to a specific branch/key/device
- **Integrity:** any modification to historical commits breaks the signature chain
- **Non-repudiation:** the committer cannot deny creating the commit
- **Tamper detection:** even the server operator cannot silently alter history

**Use cases:** Open-source projects (code signing), regulatory compliance
(audit trails), supply chain integrity (signed manifests), any scenario where
content is public but provenance must be proven.

### Mode D: Pure Version Control

Only layer 1. No encryption, no signing. The system is a lightweight,
content-addressed, portable version control system.

```
  Server sees: everything — plaintext content, full structure, full history.
  Server cannot do: silently tamper with content (content-addressed IDs
  would change if content changes, breaking all references).
```

Even without signing, the content-addressed model provides basic integrity:
changing the content of any object changes its hash, which changes its ID,
which breaks every reference to it. An attacker would need to rewrite the
entire graph from the point of modification forward — similar to git.

**What mode D has that git doesn't:**
- **Simpler object model:** JSON objects, not binary pack format
- **HTTP-native:** batch read/write over standard REST API
- **Storage-agnostic:** same `bare/` structure works with files, S3, DynamoDB, graph DB
- **Self-describing IDs:** `obj-cas-imm-*` tells you everything about the file
- **Lite clone:** on-demand fetch, no full download required
- **No staging area:** commit always snapshots the working directory
- **No pack files:** each object is a separate file (storage backend handles compression)
- **Portable:** `bare/` is a zip, a folder, a database — same structure everywhere
- **Batch operations:** one HTTP call for N reads or writes

**What mode D doesn't have that git does:**
- No branching at the ref level (branches are in the encrypted index — but in
  mode D this could be a plaintext index)
- No interactive rebase, cherry-pick, or complex history rewriting
- No hooks
- No submodules
- Smaller ecosystem (no GitHub/GitLab integration)

**Use cases:** Lightweight document versioning, configuration management,
embedded version control in applications, any scenario where you want
version history without the complexity of git.

## Mode Mixing Within a Vault

The layered architecture supports **per-file encryption levels** within a
single vault. A tree entry could reference:

- A blob encrypted with the vault's read_key (mode A — everyone with vault key can read)
- A blob encrypted with a specific user's public key (selective disclosure)
- A blob encrypted with a per-folder key (folder-level access control)
- A plaintext blob (mode C/D — public content)

The tree entry's `blob_id` doesn't change — it's still `obj-cas-imm-{hash}`.
The encryption level is an attribute of the blob, not the reference. A client
that can't decrypt a specific blob still sees the tree structure and knows
the file exists — it just can't read the content.

This enables scenarios like:
- A vault with public documentation and private contracts in the same tree
- A vault where file names are visible but content requires additional keys
- A vault where some files are readable by everyone and some by specific users
- Gradual declassification — re-encrypt a file with a weaker key (or no key)
  without changing the tree structure

## Layer Independence

Each layer can be added or removed independently:

| Change | Impact on other layers |
|--------|----------------------|
| Add layer 4 (content encryption) | No change to layers 1-3. Blobs become opaque. |
| Remove layer 3 (metadata encryption) | No change to layers 1, 2, 4. Tree/commit fields become readable. |
| Add layer 2 (signing) | No change to layers 1, 3, 4. Commits gain signatures. |
| Remove layer 4 (content encryption) | No change to layers 1-3. Blobs become readable. |

A vault can even **transition between modes** over its lifetime:
- Start in mode D (prototyping, no encryption)
- Add signing (mode C) when the project goes public
- Add content encryption (mode A) when sensitive data is added
- Selectively decrypt historical blobs when they're declassified

The `bare/` structure, the ID schema, the batch API, the indexes — all unchanged
across transitions. Only the blob payload format changes (plaintext vs ciphertext).

## Implications for the Codebase

The CLI and browser code should be structured to reflect the layers:

```
  Layer 1 (always present):
    Vault__Object_Store     — store/load by content-addressed ID
    Vault__Ref_Manager      — read/write refs by deterministic ID
    Vault__Commit           — create/load commits with parent chain
    Vault__Sync             — push/pull/clone/status
    Vault__Batch            — batch read/write operations
    Vault__Branch_Manager   — branch index management

  Layer 2 (optional):
    PKI__Crypto             — ECDSA key generation, signing, verification
    Vault__Key_Manager      — branch key pair storage

  Layer 3 (optional):
    Tree entry name_enc/size_enc/content_hash_enc encryption
    Commit message_enc encryption
    Branch index field encryption

  Layer 4 (optional):
    Vault__Crypto           — AES-256-GCM blob encryption/decryption
    PBKDF2 key derivation
    HMAC-based ID derivation (pid- IDs)
```

Note: in mode D (no encryption), `pid-` IDs could still be used — they just
derive from a publicly known vault name instead of a secret passphrase. Or
they could be replaced with `rnd-` IDs discovered via a plaintext index.
The ID schema accommodates both.

---

## Summary

What started as a zero-knowledge encrypted file sharing tool has revealed
itself to be a **general-purpose version control protocol** with four
independently toggleable security layers. The core protocol (layer 1) is
a clean, HTTP-native, storage-agnostic, content-addressed object graph
with self-describing IDs and batch operations. Encryption and signing are
features — powerful, important features — but they sit on top of a foundation
that works without them.

This is not a redesign. This is a recognition of what the architecture
already is. Every decision we made today — deterministic IDs, batch read,
the naming schema, lite clone, the head snapshot index — lives at layer 1.
They benefit all four modes equally.
