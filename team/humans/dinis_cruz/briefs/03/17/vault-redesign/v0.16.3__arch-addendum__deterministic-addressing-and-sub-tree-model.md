## Context

This addendum captures architectural decisions made during implementation of the
v0.13.30 branch model specification. Through building the CLI (v0.8.5), the vault
browser, and tracing live vault data on S3, we identified gaps between the spec
and what's needed for CLI ↔ browser interop, efficient sync, and the
zero-knowledge security model.

The v0.13.30 spec remains the canonical reference. This document specifies
changes and refinements discovered through implementation.

## 1. Self-Describing File ID Schema

### 1.1 The Problem

The v0.13.30 spec and the current implementation use inconsistent ID formats:
`ref-{random}`, `idx-{random}`, `obj-{hash}`, `key-{random}`. Looking at a
filename tells you the type but nothing about how the ID was derived, whether
the file can change, or what concurrency controls are needed. The dual-write
fix (v0.8.5) introduced bare 12-char hex IDs for HMAC-derived files, which are
visually indistinguishable from random noise.

### 1.2 The Solution: Four-Segment IDs

Every file ID follows the pattern: `{type}-{derivation}-{mutability}-{hex_id}`

**Segment 1 — Type (what the file is):**

| Code | What it is |
|------|-----------|
| `obj` | Data object (blob, tree, or commit) |
| `ref` | Ref pointer (mutable, points to a commit) |
| `idx` | Index file (denormalized cache) |
| `key` | Cryptographic key |
| `pkg` | Change pack / pending bundle |

**Segment 2 — Derivation (how the ID was computed):**

| Code | Meaning | Method |
|------|---------|--------|
| `pid` | Pseudo-deterministic | HMAC from vault key + domain — computable by any client with the secret |
| `cas` | Content-addressed | SHA256 of ciphertext — deterministic from content |
| `rnd` | Random | `secrets.token_hex()` — discoverable only via index |

**Segment 3 — Mutability (lifecycle and concurrency):**

| Code | Meaning | Server behaviour |
|------|---------|-----------------|
| `imm` | Immutable | Written once, never changes. Cache forever. Idempotent writes. |
| `snw` | Single-writer | Only one entity updates. Short TTL cache. No compare-and-swap needed. |
| `muw` | Multi-writer | Multiple entities may update. No cache. Require compare-and-swap on writes. |

**Compare-and-swap:** The server's `write-if-match` operation — "only write
this new value if the current value matches what I expect." Prevents lost
updates when two clients push simultaneously. Only needed for `muw` files.

### 1.3 Complete ID Mapping

| File | Full ID | Location |
|------|---------|----------|
| Content-addressed object | `obj-cas-imm-{SHA256[:12]}` | `bare/data/` |
| Named HEAD ref | `ref-pid-muw-{HMAC[:12]}` | `bare/refs/` |
| Clone branch ref | `ref-pid-snw-{HMAC[:12]}` | `bare/refs/` |
| Branch index | `idx-pid-muw-{HMAC[:12]}` | `bare/indexes/` |
| Public/private key | `key-rnd-imm-{random}` | `bare/keys/` |
| Change pack | `pkg-rnd-imm-{ts}_{random}` | `bare/pending/` |

### 1.4 bare/ Structure Preserved

The `bare/` folder structure from v0.13.30 is unchanged. Only the IDs within
folders change. The portable-unit invariant holds:

```
bare/ on server = bare/ on disk = bare/ in zip
```

The server API receives `bare/refs/ref-pid-muw-b1d1771f9fb5` as the file_id.  
The storage path is `transfers/vault/{vault_id}/bare/refs/ref-pid-muw-b1d1771f9fb5/payload`.  
The local path is `.sg_vault/bare/refs/ref-pid-muw-b1d1771f9fb5`.

Identical everywhere.

## 2. Deterministic HMAC-Derived Entry Points

### 2.1 The Problem

The v0.13.30 spec uses random UUIDs for refs and indexes, discoverable only by
listing directories. This prevents browser interop (browsers can't call `list`)
and caused the dual-write compatibility hack in v0.8.5.

### 2.2 The Solution

Entry-point files use HMAC-derived IDs within `bare/`. Clients with the vault
key can compute the IDs directly — no listing, no dual-write.

| Entry point | HMAC domain | Derivation key |
|-------------|-------------|---------------|
| Named HEAD ref | `sg-vault-v1:file-id:ref:{vault_id}` | `read_key` (from vault key) |
| Branch index | `sg-vault-v1:file-id:branch-index:{vault_id}` | `read_key` |
| Per-branch ref | `sg-vault-v1:file-id:branch-ref:{vault_id}:{branch_name}` | `read_key` + branch name (from index) |

The dual-write bridge from v0.8.5 is eliminated. One write per file.

### 2.3 Three Tiers of Discovery

| Tier | ID type | Discovery method |
|------|---------|-----------------|
| 1. Vault-key entry points | `*-pid-*` | Compute from vault key alone |
| 2. Discovered entry points | `*-pid-*` | Compute from vault key + data from tier 1 |
| 3. Content-addressed objects | `*-cas-*` | Follow object graph from tier 1/2 refs |
| 4. Random IDs | `*-rnd-*` | Read from tier 1 branch index |

## 3. Sub-Tree Model

### 3.1 The Problem

The current implementation uses a flat tree model: one tree object listing
every file with full relative paths. Confirmed by inspecting live vault data:
`"path": "an-folder/file-b.txt"` as a flat entry. This causes:

- Tree object grows linearly with file count
- Every commit rebuilds the entire tree
- Merge compares every entry — O(all files)
- No sub-tree shortcut for unchanged folders

### 3.2 The Solution

Implement the sub-tree model from v0.13.30 section 4.1. One tree object per
directory level. Tree entries reference `blob_id` (file) or `tree_id` (folder).

This enables on-demand tree loading — the correct UX pattern for browsing.
Read the root tree, show direct children, fetch sub-trees only when the user
expands a folder. Each folder expand is one request.

### 3.3 Tree Entry Schema (Encrypted-Only)

Drop all plaintext metadata fields:

```json
{
  "name_enc": "<AES-256-GCM encrypted name>",
  "blob_id": "obj-cas-imm-{hash}",
  "tree_id": "obj-cas-imm-{hash}",
  "size_enc": "<AES-256-GCM encrypted size>",
  "content_hash_enc": "<AES-256-GCM encrypted hash>"
}
```

**Removed:** `path`, `name`, `size`, `content_hash` (plaintext versions).

## 4. Batch Read API

### 4.1 The Problem

Every vault read is a separate HTTP request. Opening a vault requires multiple
sequential round trips.

### 4.2 The Solution

Extend `POST /api/vault/batch/{vault_id}` to support `read` operations:

```json
{
  "operations": [
    { "op": "read", "file_id": "bare/refs/ref-pid-muw-b1d1771f9fb5" },
    { "op": "read", "file_id": "bare/data/obj-cas-imm-4602b76dd1ad" }
  ]
}
```

Returns all payloads in one round trip. The server reads from S3 (AWS-to-AWS),
assembles the response. Unauthenticated (same as individual reads).

### 4.3 Server Caching by Mutability

The mutability segment enables automatic caching:
- `*-imm-*` → cache forever (content never changes)
- `*-snw-*` → cache with short TTL
- `*-muw-*` → never cache (multiple writers)

## 5. Encrypted Commit Messages

Add `message_enc` field. Drop plaintext `message`.

### Commit Schema Cleanup

| Field | Action | Reason |
|-------|--------|--------|
| `parent` (single) | **Remove** | Replaced by `parents[]` |
| `timestamp` (ISO) | **Remove** | Replaced by `timestamp_ms` (uint ms) |
| `message` (plaintext) | **Remove** | Replaced by `message_enc` |
| `version` | **Remove** | Redundant with `schema` field |

## 7. Clone: Graph-Walking, Not Full Download

### 7.1 Primary Flow (Tree-Based)

Clone follows the object graph from the deterministic entry point:

```
1. Derive refFileId from vault key
2. Read HEAD ref → commit_id
3. Read commit → tree_id
4. Read root tree → discover direct children (blob_ids + sub-tree_ids)
5. Recursively fetch sub-trees (1 request per depth level)
6. Fetch all blobs
7. Read branch index + keys
8. Register new clone branch
```

Request count: 4 + depth + 1 (for blobs batch) + 1 (register branch).

### 7.2 Zip Clone (Fastest)

One request: `GET /api/vault/download/{vault_id}` returns the entire `bare/`
as a zip archive. The ideal for large vaults and offline use.

### 7.3 Lite Clone (Minimal)

Download only entry points + working copy blobs. Tree objects and historical
commits fetched on demand through a transparent interception layer in the
object store. Same `bare/` structure, same code paths — just lazy fetching.

```
1. Batch read: ref + root tree + branch index (1 request)
2. Batch read: all blobs from root tree (1 request per depth level + 1 for blobs)
3. Batch write: register branch (1 request)
```

Over time, frequently accessed objects accumulate locally. A lite clone
gradually becomes a full clone through normal usage.

## 8. Future Optimisation: Deterministic Indexes

**Not in first implementation.** Documented as the optimisation path.

| Index | Purpose | Priority |
|-------|---------|----------|
| HEAD snapshot (`idx-pid-muw-*`) | Flat file listing (all depths) | Future |
| Commit log (`idx-pid-muw-*`) | Full commit history | Future |
| Object manifest (`idx-pid-muw-*`) | All reachable object IDs | Future |

These are denormalized caches rebuilt on every push. The object graph remains
the source of truth. They become valuable for CLI operations (`status`, `clone`,
`log`) where the full listing is needed programmatically.

| File ID format | `ref-{random}`, `idx-{random}`, `obj-{hash}` | `{type}-{derivation}-{mutability}-{hex_id}` |

| Batch API | Write-only | Read + write (batch read added) |

| Clone | Graph-walking (specified) | Via HMAC entry point + tree-based on-demand |
| Dual-write bridge | Not in spec (v0.8.5 hack) | Eliminated |
| Indexes | Not in spec | Future optimisation (snapshot, commit log, manifest) |
