# Agent-to-Agent Communication: Vaults as a Signed Message Plane

**from** Human (project lead)

## Context

The latest vault architecture has high-performance reads and writes with deterministic pseudo-random IDs derived from the vault key. This means agents can read, write, and use branches on a vault without cloning the entire thing. This unlocks a new capability: using vaults as a communication plane between agents.

## The Problem Vaults Solve for Agent Communication

Agent-to-agent communication today lacks:

- **Authentication:** How does Agent B know a message actually came from Agent A?
- **Confidentiality:** How do you ensure only the intended agent reads a message?
- **Sequencing:** How do you enforce that Agent B can only act after Agent A has completed?
- **Provenance:** How do you trace what happened, when, and by whom?
- **Replay and debugging:** How do you rewind and replay an agent workflow?
- **Prompt injection defense:** How do you prevent an agent from reading instructions meant for another agent?

Vaults with PKI solve all of these through the same primitives used for human workflows.

## The Architecture

### Message Format: EML-Like Files

Use EML (email) as the message format between agents. EML has useful built-in properties:

- To / From / CC headers (routing)
- Subject (topic)
- Timestamps
- Reply-To (threading)
- Attachments (data payloads)
- Body (instructions, results, status updates)

An agent sending a message to another agent literally creates an EML file, signs it with its private key, encrypts it with the recipient's public key, and writes it to the vault.

### PKI Per Agent

Each agent has its own key pair. This is already the pattern from the branch model (every clone gets a key pair). Applied to agent communication:

| Property | What It Gives Us |
|---|---|
| Signed messages | Agent B can verify the message came from Agent A (not hallucinated, not injected) |
| Encrypted messages | Only Agent B can read messages encrypted with Agent B's public key |
| Sequential enforcement | Agent B can only act if Agent A has signed and encrypted the handoff message |
| Prompt injection defense | An agent physically cannot read instructions meant for another agent (no private key) |

The prompt injection defense is the sleeper feature here. If Agent C tries to read Agent B's instructions, it fails. Not because of a policy. Because of cryptography. The agent does not have the private key. This is the "cannot" vs "will not" principle applied to agent security.

### Workflow Control via Encryption

Sequential workflows are enforced by the encryption chain:

```
Controller creates order, signs instructions for Supplier 1
  → encrypts with Supplier 1's public key
  → only Supplier 1 can read and act

Supplier 1 completes, signs completion message
  → encrypts handoff for Supplier 2 with Supplier 2's public key
  → Supplier 2 can verify Supplier 1 signed the handoff
  → Supplier 2 can now act

Supplier 2 completes, signs completion message
  → encrypts handoff for Supplier 3...
```

If Supplier 2 tries to act before Supplier 1 has signed the handoff, it cannot. The message does not exist yet. If Supplier 2 tries to forge a message from Supplier 1, it cannot. It does not have Supplier 1's private key. The sequence is enforced by cryptography, not by policy or access controls.

### Branching for Parallel Work

When agents work in parallel (not sequentially), each agent operates on its own branch. Branches merge when the parallel work converges. The vault branch model handles this natively. Conflicts are resolved at merge time, not during execution.

### Start, Stop, Replay

Because the vault stores the full message history as signed, timestamped files:

- **Start/Stop:** Pause an agent workflow by stopping message delivery. Resume by allowing the next message to be read.
- **Replay:** Clone the vault at any point in time. Replay the message sequence to reproduce the execution.
- **Fork:** Branch the vault and try a different approach from any point in the history.
- **Audit:** The entire execution flow is a graph of signed messages. Visualise it. Trace any decision back to its source.

### Visualisation

The message graph (who sent what to whom, when) can be visualised as a directed graph. Each message is an edge. Each agent is a node. Timestamps provide ordering. Signatures provide authentication. This is both a debugging tool and an explainability tool.

## Connection to Existing Architecture

This builds entirely on existing primitives:

- Vault storage (existing)
- PKI key pairs per clone/branch (branch model spec)
- Signed commits (branch model spec)
- High-performance read/write without full clone (current CLI)
- Change packs (branch model spec, for high-frequency message writes)

No new primitives needed. Only a message format convention (EML) and a routing pattern (encrypt-for-recipient).
