# Vault Browser Changes — Self-Describing IDs, Sub-Tree Rendering, Batch Read

**Version:** v0.16.18 (updated post-implementation)  
**Date:** 2026-03-17  
**Role:** Architect → Dev  
**Ref:** v0.16.3 arch addendum, v0.16.3 simulation, v0.16.3 implementation debrief

---

## Scope

The vault browser (`sgraph_ai_app_send__ui__vault/`) is the debug/admin tool
that decrypts and renders vault contents.

**Decision:** v0.1.2 (git-like model) is the correct base. Override v0.1.0,
v0.1.1, v0.1.3 (flat model) where appropriate.

**Important:** The CLI implementation has removed ALL plaintext fields from
stored objects. Tree entries have no `path`, `name`, `size`, or `content_hash`.
Commits have no `parent`, `timestamp`, `message`, or `version`. The browser
must decrypt `_enc` fields for everything — there is no plaintext fallback.

---

## 1. Override Previous Versions (IFD)

Following the IFD architecture, the new vault browser version overrides
components from previous minor versions where the flat model is incompatible.
Code from v0.1.0, v0.1.1, and v0.1.3 that remains useful (shared components,
CSS, utility functions) should be reused — only the vault-specific logic
(crypto model, open flow, tree rendering) is overridden.

| Directory | Action |
|-----------|--------|
| `v0/v0.1/v0.1.0/` | Reuse shared components where applicable, override vault logic |
| `v0/v0.1/v0.1.1/` | Reuse shared components where applicable, override vault logic |
| `v0/v0.1/v0.1.3/` | Override — flat vault model incompatible with CLI's git-like model |

## 2. Crypto Updates (`sg-vault-crypto.js`)

### 2.1 vault_id Validation Fix

v0.1.2 validates `/^[0-9a-f]{8}$/` (hex only). The CLI generates alphanumeric.
Fix: `/^[a-z0-9]{8}$/`.

### 2.2 ID Format in `deriveKeys()`

Return IDs with the new prefixes:

```javascript
return {
    readKey,
    writeKey,
    refFileId:         'ref-pid-muw-' + await this._deriveFileId(hmacKey, `sg-vault-v1:file-id:ref:${vaultId}`),
    branchIndexFileId: 'idx-pid-muw-' + await this._deriveFileId(hmacKey, `sg-vault-v1:file-id:branch-index:${vaultId}`)
}
```

### 2.3 Add Per-Branch Ref Derivation

```javascript
static async deriveBranchRefFileId(hmacKey, vaultId, branchName) {
    const hex = await this._deriveFileId(hmacKey, `sg-vault-v1:file-id:branch-ref:${vaultId}:${branchName}`)
    return 'ref-pid-snw-' + hex
}
```

### 2.4 Add Metadata Encryption/Decryption

Tree entry names, sizes, content hashes, and commit messages are AES-GCM
encrypted and stored as base64. The browser needs helpers for these:

```javascript
static async encryptMetadata(readKey, plaintext) {
    const data       = new TextEncoder().encode(plaintext)
    const ciphertext = await SGSendCrypto.encrypt(readKey, data)
    return btoa(String.fromCharCode(...ciphertext))
}

static async decryptMetadata(readKey, b64Ciphertext) {
    const ciphertext = Uint8Array.from(atob(b64Ciphertext), c => c.charCodeAt(0))
    const data       = await SGSendCrypto.decrypt(ciphertext, readKey)
    return new TextDecoder().decode(data)
}
```

## 3. Vault Open Flow — On-Demand Tree Loading

### 3.1 Primary Flow (No Indexes Needed)

```javascript
static async open(sgSend, fullVaultKey) {
    const { passphrase, vaultId } = SGVaultCrypto.parseVaultKey(fullVaultKey)
    const keys = await SGVaultCrypto.deriveKeys(passphrase, vaultId)

    // Step 1: Read HEAD ref (deterministic ID, 1 request)
    // Ref value is encrypted JSON: {"commit_id": "obj-cas-imm-..."}
    const refData    = await sgSend.vaultRead(vaultId, `bare/refs/${keys.refFileId}`)
    const refJson    = JSON.parse(await SGSendCrypto.decrypt(refData, keys.readKey))
    const commit_id  = refJson.commit_id

    // Step 2: Read commit (1 request)
    const commitData = await sgSend.vaultRead(vaultId, `bare/data/${commit_id}`)
    const commit     = JSON.parse(await SGSendCrypto.decrypt(commitData, keys.readKey))
    // commit has: schema, tree_id, parents[], timestamp_ms, message_enc, branch_id, signature

    // Step 3: Read root tree (1 request)
    const treeData   = await sgSend.vaultRead(vaultId, `bare/data/${commit.tree_id}`)
    const rootTree   = JSON.parse(await SGSendCrypto.decrypt(treeData, keys.readKey))
    // rootTree has: entries[] where each entry has blob_id/tree_id + name_enc + size_enc + content_hash_enc

    // Render root level — sub-trees loaded on demand when user expands folders
    return { commit, rootTree, keys }
}
```

### 3.2 Batch Read Optimisation

The commit_id is known from request 1, but tree_id is only known after
decrypting the commit. So the open flow is always 3 sequential requests.

Batch read helps when:
- Loading sub-trees (batch multiple folder expands in one request)
- Loading blobs (batch all files in a folder)

```javascript
// Batch load all blobs in a folder after expanding it
const blobIds = entries.filter(e => e.blob_id).map(e => e.blob_id)
const blobPaths = blobIds.map(id => `bare/data/${id}`)
const results = await sgSend.batchRead(vaultId, blobPaths)
```

### 3.3 `bare/` Prefix — Unchanged

The `bare/` prefixes remain in API calls. The only change is the ID format
within the path: e.g. `bare/data/obj-cas-imm-xxx` instead of `bare/data/obj-xxx`.

## 4. Sub-Tree Rendering

### 4.1 Folder Expand

Tree entries with `tree_id` (not `blob_id`) represent folders. The UI renders
them as expandable nodes. On click:

```javascript
async expandFolder(vaultId, treeId, readKey) {
    const data = await sgSend.vaultRead(vaultId, `bare/data/${treeId}`)
    const tree = JSON.parse(await SGSendCrypto.decrypt(data, readKey))
    // Decrypt names for display, then render
    const entries = await Promise.all(tree.entries.map(async entry => ({
        ...entry,
        name:         await SGVaultCrypto.decryptMetadata(readKey, entry.name_enc),
        size:         entry.size_enc ? parseInt(await SGVaultCrypto.decryptMetadata(readKey, entry.size_enc)) : 0,
        contentHash:  entry.content_hash_enc ? await SGVaultCrypto.decryptMetadata(readKey, entry.content_hash_enc) : null,
        isFolder:     !!entry.tree_id,
        isFile:       !!entry.blob_id,
    })))
    return entries
}
```

### 4.2 Decrypt ALL `_enc` Fields — No Plaintext Fallback

**Critical:** The stored tree entries contain ONLY encrypted fields. There are
no plaintext `path`, `name`, `size`, or `content_hash` fields. The browser
MUST decrypt to display anything:

```javascript
// Entry schema (as stored on disk):
{
    "blob_id": "obj-cas-imm-4f5365b013a0",     // OR tree_id for folders
    "name_enc": "base64...",                     // AES-GCM(readKey, filename)
    "size_enc": "base64...",                     // AES-GCM(readKey, "1234")
    "content_hash_enc": "base64..."              // AES-GCM(readKey, sha256_prefix)
}
```

Decryption for each field:

```javascript
async decryptEntry(entry, readKey) {
    const name = entry.name_enc
        ? await SGVaultCrypto.decryptMetadata(readKey, entry.name_enc)
        : '(unnamed)'

    const size = entry.size_enc
        ? parseInt(await SGVaultCrypto.decryptMetadata(readKey, entry.size_enc))
        : 0

    const contentHash = entry.content_hash_enc
        ? await SGVaultCrypto.decryptMetadata(readKey, entry.content_hash_enc)
        : null

    return { name, size, contentHash, blobId: entry.blob_id, treeId: entry.tree_id }
}
```

### 4.3 Decrypt Commit Fields — No Plaintext Fallback

**Critical:** The stored commit also has NO plaintext fields for message,
timestamp, or parent. The commit schema is now:

```javascript
// Commit schema (as stored on disk):
{
    "schema":       "commit_v1",
    "tree_id":      "obj-cas-imm-...",
    "parents":      ["obj-cas-imm-...", ...],    // list, NOT single "parent"
    "timestamp_ms": 1710412800000,                // uint ms, NOT ISO string "timestamp"
    "message_enc":  "base64...",                  // encrypted, NOT plaintext "message"
    "branch_id":    "branch-named-...",
    "signature":    "base64..."                   // optional ECDSA signature
}
```

**Removed fields** (will be `undefined` — do NOT read these):
- `parent` — use `parents[0]` instead
- `timestamp` — use `timestamp_ms` instead (`new Date(commit.timestamp_ms)`)
- `message` — use `decryptMetadata(readKey, commit.message_enc)` instead
- `version` — use `schema` string (e.g. `'commit_v1'`) instead

Decryption:

```javascript
async decryptCommit(commit, readKey) {
    const message = commit.message_enc
        ? await SGVaultCrypto.decryptMetadata(readKey, commit.message_enc)
        : '(no message)'

    const date = commit.timestamp_ms
        ? new Date(commit.timestamp_ms)           // NOT new Date(commit.timestamp_ms * 1000)
        : null

    const parents = commit.parents || []           // NOT commit.parent

    return { message, date, parents, treeId: commit.tree_id, schema: commit.schema }
}
```

## 5. Fix Timestamp Display

Branch index `created_at` and commit `timestamp_ms` are both uint milliseconds.
Pass directly to `new Date()` — do NOT multiply by 1000.

```javascript
// CORRECT
const date = new Date(commit.timestamp_ms)     // 1710412800000 → 2024-03-14T16:00:00Z
const branchDate = new Date(branch.created_at)

// WRONG — produces year 58176
const date = new Date(commit.timestamp_ms * 1000)
```

## 6. List View Updates

The left panel groups files by prefix. With self-describing IDs, the grouping
is more informative:

```
bare/data/
  obj-cas-imm-4e0340b66947
  obj-cas-imm-aeaa531d8450
  ...

bare/indexes/
  idx-pid-muw-48c9319a269c    <- deterministic, multi-writer

bare/keys/
  key-rnd-imm-4a6daf74fc4a1fe0    <- random, immutable
  key-rnd-imm-07e41bef0c27fd03

bare/refs/
  ref-pid-muw-b1d1771f9fb5    <- named HEAD (deterministic, multi-writer)
  ref-pid-snw-{clone_ref}     <- clone branch (deterministic, single-writer)
```

**Note:** Branch metadata's `public_key_id` and `private_key_id` fields now
carry the `key-rnd-imm-` prefix. When loading keys from branch metadata, use
the full prefixed ID: `bare/keys/key-rnd-imm-{hex}`.

**Recommendation:** Default to the graph view (start from HEAD ref, navigate
the object graph). The raw file listing is a secondary debug mode.

## 8. Summary of Breaking Changes from Previous Browser Versions

| What changed | Old (pre-v0.16.3) | New (v0.16.3) |
|---|---|---|
| Object IDs | `obj-{hash}` or bare `{hash}` | `obj-cas-imm-{hash}` |
| Ref IDs | `ref-{random}` | `ref-pid-muw-{hmac}` / `ref-pid-snw-{hmac}` |
| Index IDs | `idx-{random}` | `idx-pid-muw-{hmac}` |
| Key IDs | `key-{random}` | `key-rnd-imm-{random}` |
| Ref value format | Plaintext commit ID | Encrypted JSON `{"commit_id": "..."}` |
| Tree entry `path` | Plaintext string | **REMOVED** — use `name_enc` |
| Tree entry `name` | Plaintext string | **REMOVED** — use `name_enc` |
| Tree entry `size` | Plaintext integer | **REMOVED** — use `size_enc` |
| Tree entry `content_hash` | Plaintext string | **REMOVED** — use `content_hash_enc` |
| Tree entry `tree_id` | Not present | Sub-tree reference (folder) |
| Commit `parent` | Single string | **REMOVED** — use `parents[]` |
| Commit `timestamp` | ISO string | **REMOVED** — use `timestamp_ms` |
| Commit `message` | Plaintext string | **REMOVED** — use `message_enc` |
| Commit `version` | Integer counter | **REMOVED** — use `schema` string |
| Tree model | Flat (all paths in one tree) | Sub-tree (one tree per directory) |
