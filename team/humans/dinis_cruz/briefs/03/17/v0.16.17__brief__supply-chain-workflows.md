# Supply Chain Workflows: Per-Supplier Encryption and Sequential Control

## The Use Case

A company sells products online. Fulfilling an order requires three suppliers and a carrier:

- **Supplier 1:** Manufactures or sources the product
- **Supplier 2:** Packages and assembles the order
- **Supplier 3:** Prepares for shipment (labelling, customs, documentation)
- **Carrier:** Delivers to the customer (the only entity that needs the delivery address)

Each supplier needs some information but not all of it. The carrier needs the address but not the order details. Today, this is managed through emails, ERP systems, and shared spreadsheets where everyone can see everything. Data minimisation is aspirational, not enforced.

## How Vaults Solve This

### Per-Supplier Encryption

Each supplier is onboarded with a key pair (public key registered, private key stays with the supplier). When an order is created, the vault contains instructions encrypted per supplier:

- Supplier 1 receives instructions encrypted with Supplier 1's public key. Only Supplier 1 can read them.
- Supplier 2 receives instructions encrypted with Supplier 2's public key.
- Supplier 3 receives instructions encrypted with Supplier 3's public key.
- The carrier receives the delivery address encrypted with the carrier's public key. The carrier cannot see the order details, product information, or pricing.

If the vault (or the zip file containing it) is intercepted, compromised, or accidentally shared, each supplier's data is only readable by that supplier. Data minimisation is enforced cryptographically, not by access control lists.

### Sequential Enforcement

The supply chain is sequential: Supplier 1 must finish before Supplier 2 can start. This is enforced through signed handoffs:

```
Order created → Controller signs instructions for Supplier 1

Supplier 1 completes → signs completion message
  → creates handoff encrypted for Supplier 2
  → Supplier 2 can verify Supplier 1's signature
  → Supplier 2 can now read their instructions

Supplier 2 completes → signs completion message
  → creates handoff encrypted for Supplier 3

Supplier 3 completes → signs completion message
  → creates handoff encrypted for Carrier
  → Carrier can now read the delivery address
```

Supplier 2 cannot act before Supplier 1 has signed the handoff. The handoff message does not exist until Supplier 1 creates and signs it. This is not a policy check. It is a cryptographic constraint. The message literally does not exist for Supplier 2 to read.

### Compromise Containment

If Supplier 2 is compromised, the attacker can only access:

- The instructions encrypted for Supplier 2
- The handoff messages Supplier 2 has received
- Nothing else: not Supplier 1's instructions, not Supplier 3's data, not the delivery address

This is data minimisation by design. Each entity in the chain holds only the data it needs. A breach at any single point exposes only that point's data.

### GDPR Alignment

The vault model is naturally GDPR-friendly for supply chains:

- **Data minimisation:** each supplier only has access to their relevant data
- **Purpose limitation:** the encryption scope matches the purpose (Supplier 2 can only see packaging instructions, not manufacturing details)
- **Storage location independence:** the encrypted data can be stored anywhere. The geographic location of the server does not matter because the server cannot read the data
- **Provenance:** every action is signed. The audit trail is cryptographic.
- **Right to erasure:** delete the vault key. The data becomes permanently unreadable.

### Distribution Model

Two options, matching different supply chain maturity levels:

**Zip-based (simple):** The vault is exported as a zip and sent by email, WhatsApp, or any channel. Each supplier opens the zip, enters their private key, and sees only their data. No SG/Send account needed. No server dependency. Works with existing email-based workflows.

**Server-based (real-time):** The vault lives on SG/Send. Each supplier accesses it via a transaction ID and their private key. Real-time updates. The controller can see the status of each stage. Suppliers push their completion messages directly to the vault.

The zip model matches today's reality (most supply chains run on email). The server model is the upgrade path.

## Business Opportunity

This is a standalone company waiting to be built (see the ecosystem strategy brief). The primitives are: encrypted file transfer, per-entity encryption, signed sequential handoffs, vault-based state management. SG/Send provides all of these. The company built on top provides the supply chain domain expertise, the industry relationships, and the specific workflow UI.

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | Supply chain demo: 3 suppliers + carrier, per-entity encryption, sequential handoff |
| 2 | Each supplier can only read their own instructions (verified by attempting cross-read) |
| 3 | Sequential enforcement: Supplier 2 cannot act before Supplier 1 signs handoff |
| 4 | Carrier can only read the delivery address, not order details |
| 5 | Zip-based distribution working (email the vault, open with private key) |
| 6 | Compromise simulation: breach of one supplier exposes only their data |
| 7 | GDPR alignment documented for the supply chain pattern |
