## Where We Are

A massive piece of QA work just landed. Two QA teams working in parallel (Claude Code implementing, Claude Web reviewing and guiding). The page objects, browser automation, and screenshot capture are working. The QA site is being published. The foundation layer is proven.

Now: the next phase. Move from "QA tests that verify functionality" to "a unified API that serves QA, deployment gating, traffic simulation, and monitoring."

## The Unified API

Right now the QA code has detailed tests that cover every edge case, every component state, every scenario. That is the full QA suite. But we also need a different mode: fast, focused user story replays that verify the happy path works end to end.

### Two Execution Modes, One Codebase

| Mode | Purpose | Speed | Screenshots | Detail Level |
|---|---|---|---|---|
| QA mode | Verify every edge case, capture screenshots, document behaviour | Slower (screenshots, assertions per step) | Yes | High (every state, every variant) |
| Smoke mode | Verify the happy path works, confirm deployment is safe | Fast (skip screenshots, minimal assertions) | No (unless failure) | Low (success path only) |

Same underlying page objects. Same workflow code. The mode determines whether screenshots are captured, how many assertions run, and whether edge cases are exercised. The smoke mode chases the success story. The QA mode chases every corner.

## User Personas and Stories

The smoke tests should model real users with real objectives:

**Persona A: File Sender**
1. Goes to upload page
2. Drops a file
3. Selects delivery mode
4. Clicks send
5. Gets a link

**Persona B: File Recipient**
1. Receives the link from Persona A
2. Opens the link
3. Sees the file
4. Downloads it

**Persona C: Gallery Sender**
1. Drops a folder of images
2. Gets a gallery link

**Persona D: Gallery Viewer**
1. Opens the gallery link
2. Browses thumbnails
3. Opens lightbox
4. Saves a file

Each persona is a user story. Each user story is a path through the flow graph. The smoke tests replay these stories. When you want performance testing, run 10, 20, 30 personas in parallel.

## FastAPI Lambda Function

Deploy the QA API as a Lambda function (FastAPI, same technology as the existing backend). This Lambda:

- Accepts an endpoint call specifying which workflow to run and against which server
- Runs the workflow (browser-based via Playwright, or HTTP-only for raw traffic)
- Returns results as JSON containing:
  - Success/failure status
  - Timing data (per step, total)
  - Prometheus-formatted metrics
  - Logs and traces (for Elastic/OpenSearch)
  - Screenshots (optional, on failure or when requested)

### Target Servers

The same Lambda function can be pointed at:

| Target | Use Case |
|---|---|
| Local dev server | Development testing |
| dev.send.sgraph.ai | Pre-production verification |
| send.sgraph.ai | Production smoke tests |

This means the deployment workflow becomes:

1. Push code to dev branch
2. CI deploys to dev server
3. Lambda runs smoke tests against dev
4. If all pass: merge to main
5. Lambda runs smoke tests against main
6. If all pass: push to production is safe

**Any test failing on main is an incident.** Main should always be deployable. The smoke tests are the gate.

## Two Traffic Modes

### Browser Mode (Playwright)

Real browser sessions executing real user workflows. This is the authoritative test: if the browser can complete the workflow, a user can complete the workflow. Slower but comprehensive. This is the mode for:

- QA testing
- Smoke tests (deployment gate)
- User story replays
- Screenshots and documentation

### HTTP Mode (Raw Requests)

Direct HTTP requests to the API endpoints. No browser. No JavaScript rendering. This is faster and cheaper. This is the mode for:

- High-volume traffic simulation (generate thousands of requests)
- Performance testing (measure API response times under load)
- Security testing (verify that raw HTTP access is blocked where it should be)
- Canary checks (lightweight, frequent, high-volume)

The HTTP mode is also a security smoke test: requests that should require browser-side decryption should fail when sent as raw HTTP. If raw HTTP can access decrypted content, that is a security bug. The HTTP mode becomes a permanent canary that verifies the zero-knowledge model holds.

## Scheduling

| Frequency | What Runs | Against |
|---|---|---|
| Every push to dev | Full QA suite (browser mode) | Local + dev server |
| Every merge to main | Smoke tests (browser mode) | Dev server |
| Every 20-60 minutes | Smoke tests (browser mode) | Production |
| Continuously | HTTP canary (raw requests) | Production |
| On demand | Performance tests (browser + HTTP, parallel) | Dev or production |

The 20-60 minute production smoke tests provide continuous confidence that the live site works. The HTTP canary provides continuous security verification.

## Prometheus and Observability Output

Every Lambda execution produces Prometheus metrics:

```
sg_smoke_test_duration_seconds{workflow="upload_file",target="prod"} 4.2
sg_smoke_test_success{workflow="upload_file",target="prod"} 1
sg_smoke_test_success{workflow="download_file",target="prod"} 1
sg_smoke_test_duration_seconds{workflow="gallery_view",target="prod"} 3.1
```

These feed into the Grafana dashboards alongside the production metrics. A failing smoke test shows up as a Grafana alert. The SRE agent sees it immediately.

Logs and traces go to Elastic/OpenSearch for searchable history: when did a test last fail? What was the error? How long has the issue existed?

## Connection to the Sherpa

The user personas and stories that drive the smoke tests are the same user journeys the Sherpa demonstrates. The Sherpa's interactive walkthrough of "how to share a file" is the same sequence of steps as the smoke test for "file sender persona." The Sherpa and the QA system share the same workflow definitions.

When the Sherpa is updated with a new workflow, a corresponding smoke test is automatically available. When a smoke test fails, the Sherpa knows that workflow is broken.
