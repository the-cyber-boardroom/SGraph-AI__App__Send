## What We Need

A Lambda function that runs Playwright browser automation. This serves the smoke tests, synthetic traffic, and scheduled QA runs described in the unified QA API brief. The key constraint: **changing the test code should not require rebuilding and deploying a new container.**

## Research Required

Before building, research what people are doing in March 2026 for serverless Playwright execution. This has been done before (the project lead has done it in the past), but the ecosystem may have improved. Look at:

- AWS Lambda container image support for Playwright/Chromium
- Existing open source solutions for serverless browser automation
- Lambda Web Adapter compatibility with Playwright
- Cold start times and memory requirements for headless Chromium in Lambda
- Any managed services that have emerged for this use case

## The Architecture: Stable Container, Hot-Swap Code

### The Problem with Normal Lambda Deployment

Every code change requires either a new zip upload (for zip-based Lambda) or a new container push (for container-based Lambda). Container pushes are slow: build, push to ECR, update Lambda configuration, wait for propagation. This kills iteration speed for QA tests that change frequently.

### The Solution: Code-as-Zip, Loaded at Boot

The pattern (already proven on the existing OS-bot AWS deployment):

1. **Build a container once.** The container contains Python, Playwright, Chromium, and the boot loader. This container is pushed to ECR and configured as the Lambda image. It changes rarely (only when dependencies change).

2. **Store the test code as a zip on S3.** The actual QA test code (page objects, workflows, FastAPI endpoints) is zipped and uploaded to S3. This changes frequently.

3. **At Lambda boot, the container loads the zip.** Environment variables tell the container:
   - `CODE_ZIP_S3_PATH`: where the zip lives on S3
   - `HANDLER`: which function to invoke (equivalent of the Lambda handler)
   - `CODE_VERSION`: optional version tag for tracking

4. **The boot loader downloads the zip, extracts to /tmp, adds to Python path, and invokes the handler.**

```
Container (stable, rarely changes)
  │
  ├── Python runtime
  ├── Playwright + Chromium
  ├── Boot loader script
  │     │
  │     ├── Read CODE_ZIP_S3_PATH from env
  │     ├── Download zip from S3
  │     ├── Extract to /tmp/code/
  │     ├── Add /tmp/code/ to sys.path
  │     └── Import and invoke HANDLER
  │
  └── (Optional) Lambda Web Adapter

S3 (changes frequently)
  │
  └── qa-code-v0.16.61.zip
        ├── main.py (FastAPI app or handler)
        ├── page_objects/
        ├── workflows/
        ├── schemas/
        └── requirements.txt (if extra deps needed)
```

### Deploying a Code Change

To update the QA tests:

1. Zip the code folder
2. Upload to S3 (`aws s3 cp qa-code.zip s3://bucket/qa-code/latest.zip`)
3. Next Lambda invocation loads the new code

No container rebuild. No ECR push. No Lambda configuration change. The zip upload takes seconds. The next cold start picks up the new code. For warm instances, either wait for natural recycling or trigger a function update to force a cold start.

### Dependencies

If the code zip needs Python dependencies beyond what is in the container:

1. A helper script installs dependencies locally into a folder
2. The folder is zipped and uploaded to S3 alongside the code zip
3. The boot loader downloads both zips, extracts the deps to /tmp, adds to Python path

This replicates the Lambda layers pattern but without the Lambda layers overhead and size limits.

## Lambda Web Adapter

The Lambda Web Adapter is worth including in the base container. It provides:

- HTTP request/response buffering (important for long-running Playwright executions)
- Standard HTTP interface (the FastAPI app works natively)
- Streaming support for large responses (screenshots, traces)

### Two Modes: Direct Invoke vs HTTP

| Mode | How It Works | Best For |
|---|---|---|
| Direct invoke | Lambda handler receives event, runs workflow, returns result | Scheduled smoke tests, CI-triggered runs |
| HTTP via Lambda Web Adapter | FastAPI app receives HTTP request, runs workflow, returns response | On-demand API calls, interactive testing |

**Ideal:** One container that supports both modes. The Lambda Web Adapter can be a configuration toggle (environment variable). If `USE_WEB_ADAPTER=true`, boot the FastAPI app behind the adapter. If `USE_WEB_ADAPTER=false`, boot as a direct Lambda handler.

If making one image work for both modes is not feasible, create two images from the same Dockerfile with a build arg that controls the entrypoint. The base layers (Python, Playwright, Chromium, boot loader) are shared. Only the entrypoint differs.

## Container Contents

The stable container needs:

| Component | Purpose | Changes |
|---|---|---|
| Python 3.12+ | Runtime | Rarely |
| Playwright | Browser automation library | Occasionally (version bumps) |
| Chromium (headless) | The browser | Occasionally (bundled with Playwright) |
| Boot loader script | Downloads zip, sets up paths, invokes handler | Rarely |
| Lambda Web Adapter | HTTP buffering (optional) | Rarely |
| boto3 | S3 access for downloading code zip | Rarely |

Everything else (the actual test code, page objects, workflows, schemas) lives in the S3 zip and is loaded at boot.

## Version Control

Version the S3 zips:

```
s3://bucket/qa-code/latest.zip          (always the current version)
s3://bucket/qa-code/v0.16.61.zip        (version-tagged for rollback)
s3://bucket/qa-code/v0.16.53.zip        (previous version)
```

The Lambda environment variable can point to `latest.zip` for normal operation or a specific version for debugging/rollback.

## Deployment Flow

```
Developer updates QA tests
  │
  ├── zip the code
  ├── upload to S3 (latest.zip + versioned copy)
  └── done (next Lambda invocation uses new code)

DevOps updates container (rare)
  │
  ├── rebuild Dockerfile (Playwright version bump, etc.)
  ├── push to ECR
  ├── update Lambda to use new image
  └── done (all future invocations use new container)
```

Two separate deployment paths. Code changes are fast (seconds). Container changes are slow but rare.
