## Part 1: Event-Driven Browser Tests

### The Problem with `wait_for_timeout()`

The P0 browser tests were littered with hardcoded sleeps like:

```python
page.wait_for_timeout(500)   # hope the gate vanished
page.wait_for_timeout(4000)  # hope the upload finished
page.wait_for_timeout(4000)  # hope the download decrypted
```

These are fragile. They fail on slow CI runners and pass spuriously on fast local machines. They also make tests slow by design — every sleep is dead time.

### The Fix: JS_Query__Execute Predicates

`JS_Query__Execute` (from osbot-utils) exposes three predicate builders that return JS strings Playwright can poll with `wait_for_function()`:

| Predicate | What it checks |
|-----------|---------------|
| `predicate__light_not_exists(selector)` | Element gone from DOM |
| `predicate__light_property_equals(selector, attr, value)` | DOM attribute equals value |
| `predicate__light_property_in(selector, attr, values)` | DOM attribute in a set of values |

These hook directly into the SG/Send UI's state machine attributes (`send-upload._state`, `send-download.state`) — the same attributes the UI uses internally to drive its own rendering.

### browser_helpers.py — Rewritten

The shared test helper module was fully rewritten:

**`goto(page, url)`** — replaces bare `page.goto()`:
```python
page.goto(url, wait_until="commit")
page.wait_for_selector("body[data-ready]", timeout=10_000)
```
`wait_until="commit"` fires as soon as the HTTP response is committed (not after full load). The `body[data-ready]` selector waits for SG/Send's own signal that it's ready. Together they're faster and more accurate than `wait_until="networkidle"`.

**`handle_access_gate(page, token)`** — replaces `fill + timeout`:
```python
pred = _js(page).predicate__light_not_exists("#access-token-input")
page.wait_for_function(pred, timeout=5000)
```
Waits until the input is gone — not a fixed 500ms sleep.

**`wait_for_upload_state(page, state)`** and **`wait_for_download_state(page, state)`** — new state machine helpers:
```python
# Wait for one state
wait_for_upload_state(page, "complete")

# Wait for any of several states (e.g. success or error)
wait_for_download_states(page, ["complete", "error"])
```

### Tests Updated

All three P0 test files had their `wait_for_timeout()` calls replaced:

| Test file | Old approach | New approach |
|-----------|-------------|-------------|
| `p0__combined_link` | 3× `timeout(500)` + 2× `timeout(4000)` | upload state predicates + download state predicates |
| `p0__friendly_token` | `timeout(500)` + 3× download waits | same pattern |
| `p0__upload__single_file` | `timeout(500)` + upload waits | `wait_for_upload_states(["file-ready", "choosing-delivery"])` |

Result: **zero `wait_for_timeout()` calls** remain in any P0 test.

## Part 2: Schema Page Objects

### Why Schema Objects?

Browser tests interact with the UI and need to capture its state (for assertions and documentation). Without a schema, this state lives in ad-hoc local variables scattered across the test. With schemas, state is structured, typed, and serialisable.

### Five Page Schemas Created

All five follow the same pattern: `Type_Safe` base class, `str` fields default to `""`, `bool` fields default to `False`.

**`Schema__Upload_Page`** — captures upload wizard state:
```python
class Schema__Upload_Page(Type_Safe):
    state           : str  = ''   # idle|file-ready|choosing-delivery|choosing-share|confirming|complete
    file_name       : str  = ''
    share_link      : str  = ''
    friendly_token  : str  = ''
    is_gate_visible : bool = False
```

**`Schema__Download_Page`** — captures download/decrypt state:
```python
class Schema__Download_Page(Type_Safe):
    state               : str  = ''   # loading|entry|ready|decrypting|complete|error
    is_key_input_visible: bool = False
    content_text        : str  = ''
    error_message       : str  = ''
    transfer_id         : str  = ''
```

**`Schema__Browse_Page`**, **`Schema__Gallery_Page`**, **`Schema__Viewer_Page`** — same pattern for the file browsing, gallery, and viewer pages.

These live in `sg_send_qa/browser/` alongside the existing harness code.

### Unit Tests

11 unit tests cover all five schemas — `tests/unit/browser/test_Schema__Page_Objects.py`. All green. Tests verify: default field values, kwarg construction, type enforcement, `json()` serialisation, and field assignment.

## Part 3: The Screenshot → Docs → GitHub Pages Pipeline

### The Architecture

```
CI push to dev
    │
    ├── run-unit-tests ──────────────────────┐
    ├── run-integration-tests ───────────────┤  artifacts:
    ├── run-v030-browser-tests ──────────────┤  screenshots-integration
    │                                        │  screenshots-v030
    │                                        ▼
    ├── generate-docs ◄────── downloads both artifacts
    │       │                 runs QA_Generate_Docs
    │       │                 commits sg_send_qa__site/ (retry push loop)
    │       ▼
    ├── deploy-docs ◄──────── Jekyll build
    │       │                 GitHub Pages upload + deploy
    │       ▼
    └── https://qa.send.sgraph.ai/
```

### Key Components

**`QA_Generate_Docs`** — CLI class that:
1. Walks `sg_send_qa__site/pages/use-cases/` grouped directories
2. Reads `_groups.json` for group metadata
3. Scaffolds missing `.md` pages with front matter, description, and screenshot table
4. Regenerates `_data/qa_sidebar.json` and `_data/qa_summary.json`

**Jekyll site** (`sg_send_qa__site/`) — structured as:
```
sg_send_qa__site/
├── _config.yml              ← Jekyll config, base URL, plugins
├── _layouts/default.html    ← sidebar + content layout
├── _data/
│   ├── qa_sidebar.json      ← auto-generated navigation
│   └── qa_summary.json      ← auto-generated summary
└── pages/use-cases/
    ├── 01-access-gate/
    ├── 02-upload-share/
    └── ...
```

**Screenshot naming convention:**
```
sg_send_qa__site/pages/use-cases/{group}/{test_name}/
    screenshots/
        01_landing.png
        02_file_selected.png
        ...
```

### Bugs Fixed Along the Way

**1. 404 on sidebar links**

Sidebar links were `/pages/use-cases/02-upload-share/combined_link/` but pages were at `/pages/use-cases/combined_link/` (no group in URL). Root cause: the `FRONT_MATTER` template in `QA_Generate_Docs` omitted `{group}` from the permalink.

Fix: added `{group}` to `FRONT_MATTER`, passed `group_name` through `process_use_case()` → `scaffold_page()`.

**2. Duplicate `github-pages` artifact**

When two pushes hit CI simultaneously, both runs reached `deploy-docs` and tried to upload a `github-pages` artifact — GitHub Pages allows only one per run.

Fix: added job-level concurrency control:
```yaml
concurrency:
  group: github-pages
  cancel-in-progress: true
```

**3. Non-fast-forward push in `generate-docs`**

The `generate-docs` job checked out the branch, ran tests, generated docs, then tried to push — but another job (e.g. `increment-tag`) had already pushed to the branch in the meantime.

Fix: replaced `stefanzweifel/git-auto-commit-action` with a manual retry loop:
```bash
for attempt in 1 2 3 4; do
    git pull --rebase origin $BRANCH \
        && git push origin HEAD:$BRANCH \
        && exit 0
    sleep $((attempt * 2))
done
exit 1
```
Stage first, commit, then rebase + push up to 4 times with exponential backoff (2s, 4s, 8s, 16s).

**4. `generate-docs` running on test failures**

Policy: failing tests must break the build. Screenshots from a broken test run should not be published as documentation.

Fix: removed `if: always()` from `generate-docs`. It now only runs when all test jobs succeed. `deploy-docs` only runs when `generate-docs` succeeds.

## What Makes This Workflow Powerful

### Tests as the Source of Truth for Documentation

There is no separate documentation effort. Every test run:
1. Exercises the real UI
2. Captures screenshots at each step
3. Generates a human-readable page showing what happened

If the UI changes, the screenshots change. If the screenshots change, the docs change. The documentation is always current because it is generated from the tests.

### State Machine Predicates Instead of Sleeps

The tests don't guess how long the UI takes. They ask the UI itself: "are you in state X yet?" This is the same question the UI's own rendering loop asks. The tests are now as fast as the UI allows, and as reliable as the state machine guarantees.

### Push-to-Deploy

A developer pushes a commit → CI runs → tests pass → docs update → site deploys. No manual steps. The QA site at `qa.send.sgraph.ai` is a live, continuously-updated mirror of test coverage.
