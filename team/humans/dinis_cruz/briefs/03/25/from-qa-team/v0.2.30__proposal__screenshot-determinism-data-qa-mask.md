**Version:** v0.2.30

**From:** QA Lead (Claude Web session)

## The Problem

The QA pipeline captures screenshots at every step of every P0 test. These screenshots are committed to the repo and published to `qa.send.sgraph.ai`. The pipeline includes a `diff_screenshots` step that compares each new screenshot against the previous version and reverts changes below a 1% pixel threshold — this filters out rendering noise (font hinting, anti-aliasing).

**The problem: legitimate CI runs produce functionally identical screenshots that differ in non-deterministic text content.** Every run generates fresh transfer IDs, encryption keys, timestamps, and friendly tokens. The screenshot captures these values in the rendered UI. The pixel diff sees >1% change and commits a new screenshot every time — even though the UI layout, styling, and functionality are unchanged.

### Evidence

The auto-commit `ci: update QA screenshots and docs [skip ci]` (25 March 2026, commit `2758d9ec`) modified `06_link_captured.png` in the combined link test. Side-by-side comparison shows:

| Element | Before | After | Changed? |
|---|---|---|---|
| Layout | Step 6 of 6, upload wizard | Same | No |
| Status | "Encrypted and uploaded successfully" | Same | No |
| File name | combined-link-test.txt | Same | No |
| Delivery mode | File viewer mode | Same | No |
| Share mode | Combined link | Same | No |
| **Combined link URL** | `localhost:10062/en-gb/browse/#5bb2a4227bbc/KtGIsLN7XYjh...` | `localhost:10062/en-gb/browse/#3bf01e6a1109/ImsutAq0uZRDe...` | **Yes** |
| **Timing** | Upload completed in 0.04s | Upload completed in 0.03s | **Yes** |
| **Progress bar widths** | Reading 14ms, Encrypting 1ms, ... | Different ms values | **Yes** |

The layout is identical. The behaviour is identical. But the screenshot has changed because the transfer ID, key, and timing values are different.

### Where This Happens

Every screenshot that shows any of these elements will trigger a false commit:

| Dynamic element | Where it appears | Why it changes |
|---|---|---|
| Transfer ID | Combined link URL, download URL, browse URL | Fresh UUID per test run |
| Encryption key | Combined link URL (base64 after `#tid/`) | Fresh key per encryption |
| Friendly token | Token display (`apple-mango-1234`) | Random words per run |
| Timing values | "Upload completed in 0.04s", step durations | Varies with CI load |
| Progress bar widths | Timing breakdown (Reading, Encrypting, etc.) | Proportional to timing |
| Timestamps | "Uploaded: Wed, 25 Mar 2026 18:26:19 GMT" | Current time |
| Port numbers | `localhost:10062` in URLs | Potentially changes |

This affects at minimum: `06_link_captured.png`, `05_token_captured.png`, `07_auto_decrypted.png`, and any screenshot of the download/browse/gallery pages that show transfer metadata.

### Why This Matters

1. **Git noise.** Every CI run commits modified PNGs even when nothing meaningful changed. The repo history fills with binary diffs that have zero information value.

2. **Review fatigue.** When reviewing the `sg_send_qa__site/` changes, every PR shows "12 files changed" with screenshot diffs that are all false positives. Real visual regressions get lost in the noise.

3. **Incorrect documentation updates.** The QA site at `qa.send.sgraph.ai` re-publishes with every commit. Readers see screenshots with different transfer IDs between page loads — confusing for anyone using the site as reference documentation.

4. **Cache invalidation.** GitHub Pages CDN and browser caches treat each new PNG as changed content. Users re-download identical-looking images.

## Current Solution: Pixel Diff Threshold

`sg_send_qa/ci/diff_screenshots.py` uses Pillow to compare each screenshot pixel-by-pixel. If less than 1% of pixels changed, the file is reverted (`git checkout`). If more than 1% changed, the file is kept and committed.

### Why the threshold isn't enough

The 1% threshold catches rendering noise (sub-pixel anti-aliasing, font hinting variations between CI runs). But the dynamic text in URLs, tokens, and timing values occupies more than 1% of the screenshot area — especially the combined link URL which spans the full width of the page. The threshold would need to be raised to ~5-8% to catch these, but at that level it would also suppress real visual regressions (a missing button, a broken layout shift).

**The threshold cannot distinguish between "the URL text changed" (noise) and "the button moved 20px" (real bug).** Both produce similar pixel-change percentages.

### Options considered

| Approach | Pros | Cons |
|---|---|---|
| Raise threshold to 5-8% | Simple | Hides real regressions |
| Coordinate-based exclusion zones | No UI changes | Brittle — breaks when layout shifts |
| Per-screenshot mask selectors | Flexible | Test code carries mask knowledge |
| **`data-qa-mask` attributes on UI** | **Declarative, maintainable, self-documenting** | **Requires UI team change** |
| Replace dynamic text in post-processing | No UI changes | Complex image manipulation, fragile |

## Proposed Solution: `data-qa-mask`

### The Concept

The SG/Send UI adds a `data-qa-mask` HTML attribute to any element whose rendered content is non-deterministic. The QA screenshot system reads these attributes before capturing and paints those elements with a solid color matching the app background.

The app declares: "this element's text content changes between runs." The QA system acts on that declaration. Neither side needs to know the other's internals.

### How It Works

**Step 1: UI team adds `data-qa-mask` to dynamic elements**

```html
<!-- upload-step-done.html (inside shadow DOM) -->
<div class="share-link-box">
    <input readonly data-qa-mask="transfer-url"
           value="http://localhost:10062/en-gb/browse/#5bb2a4227bbc/KtGIsLN7XYjh..." />
    <button>Copy link</button>
</div>

<div class="timing-summary" data-qa-mask="timing">
    Upload completed in 0.04s
</div>

<div class="timing-breakdown" data-qa-mask="timing-bars">
    <div class="timing-row">Reading file <span class="bar"></span> 14ms</div>
    <div class="timing-row">Encrypting <span class="bar"></span> 1ms</div>
    ...
</div>
```

```html
<!-- upload-step-done.html — token display -->
<span id="simple-token" data-qa-mask="friendly-token">apple-mango-1234</span>
```

```html
<!-- send-download — after decrypt -->
<div class="decrypt-timing" data-qa-mask="timing">
    Decrypted in 0.02s
</div>
<div class="upload-meta" data-qa-mask="timestamp">
    Uploaded: Wed, 25 Mar 2026 18:26:19 GMT
</div>
```

The `data-qa-mask` value is a human-readable category (for debugging and documentation). The QA system doesn't use the value — it just looks for the attribute's presence.

**Step 2: QA screenshot capture masks these elements**

Playwright's native `page.screenshot(mask=[...])` parameter accepts locator arrays. Before capturing, the QA system queries for all `[data-qa-mask]` elements and passes them as masks:

```python
# In ScreenshotCapture.capture():
def capture(self, page, name, description):
    # Find all elements the UI has marked as non-deterministic
    mask_elements = page.locator("[data-qa-mask]").all()

    # Playwright paints masked areas with a solid color
    page.screenshot(
        path    = self._path_for(name),
        mask    = mask_elements,
        mask_color = "#1a2332"        # matches SG/Send dark background
    )
```

The mask color (`#1a2332`) matches the app's dark theme background. Masked areas appear as solid dark rectangles — visually unobtrusive, clearly intentional, and deterministic across runs.

**Step 3: Result**

The saved screenshot shows:
- Full layout, all UI elements, real styling
- Dynamic text areas replaced with solid-color blocks
- Identical PNGs across runs (assuming no layout change)
- `diff_screenshots` sees 0% change → reverts → no commit noise

### Shadow DOM Consideration

SG/Send's upload wizard components use Shadow DOM. Playwright's `page.locator("[data-qa-mask]")` pierces open shadow roots automatically (CSS descendant selector), so `[data-qa-mask]` attributes inside shadow roots are found without special handling.

However, if any component uses closed shadow roots (unlikely but possible), the QA system falls back to the JS_Query layer:

```python
# Fallback for closed shadow roots (if ever needed):
mask_count = sg_send.js().evaluate("""
    document.querySelectorAll('[data-qa-mask]').length +
    Array.from(document.querySelectorAll('*'))
        .filter(el => el.shadowRoot)
        .reduce((n, el) => n + el.shadowRoot.querySelectorAll('[data-qa-mask]').length, 0)
""")
```

For now, all SG/Send shadow roots are `mode: "open"`, so the simple `page.locator("[data-qa-mask]")` approach works.

## Brief to SG/Send UI Team

### Request

Add `data-qa-mask` attributes to HTML elements in the SG/Send v0.3.0 UI whose text content is non-deterministic (changes between sessions/runs).

### Why

The QA automation pipeline captures screenshots at each test step and commits them as living documentation at `qa.send.sgraph.ai`. Screenshots containing dynamic values (transfer IDs, encryption keys, timestamps, timing data) produce false diffs on every CI run, causing unnecessary commits, review noise, and incorrect documentation updates.

### What to change

Add `data-qa-mask` (no value required, but a descriptive category is helpful) to the following elements:

**upload-step-done component:**

| Element | Selector | Content | Mask category |
|---|---|---|---|
| Combined link URL | `input[readonly]` containing the `#tid/key` URL | Transfer ID + encryption key | `transfer-url` |
| Simple token display | `#simple-token` | Friendly token (`word-word-NNNN`) | `friendly-token` |
| Full link display | `#full-link` | Full URL with key | `transfer-url` |
| Link-only display | `#link-only` | URL without key | `transfer-url` |
| Decryption key display | `#decryption-key` | Base64 key | `encryption-key` |
| Timing summary | `.timing-summary` or equivalent | "Upload completed in 0.04s" | `timing` |
| Timing breakdown rows | `.timing-row` or equivalent | "Reading file 14ms" etc. | `timing` |
| Progress bar widths | `.bar` or equivalent | Variable-width bars | `timing-bars` |

**send-download component:**

| Element | Selector | Content | Mask category |
|---|---|---|---|
| Decrypt timing | `.decrypt-timing` or equivalent | "Decrypted in 0.02s" | `timing` |

**General (any page):**

| Element | Selector | Content | Mask category |
|---|---|---|---|
| Upload timestamp | Element showing "Uploaded: Wed, 25 Mar..." | Date/time | `timestamp` |
| Download link display | `input[readonly]` in download view | URL with transfer ID | `transfer-url` |

### What NOT to mask

Do not add `data-qa-mask` to:
- Static UI text (labels, headings, descriptions)
- File names (these are test-controlled and deterministic)
- File sizes (deterministic for the same test file)
- UI state indicators (step numbers, progress dots)
- Buttons, icons, navigation elements
- Error messages (these should be captured for debugging)

The rule: **mask only content that changes between runs with the same test input.** If the test always uploads `test.txt` with the same content, the file name and size are deterministic. But the transfer ID, encryption key, and timing are not.

### Impact

- Zero functional impact — `data-qa-mask` is a purely declarative attribute with no JS behaviour
- No CSS changes needed — the attribute is consumed only by the QA system
- No runtime cost — Playwright reads the attribute only during screenshot capture
- Forward-compatible — if new dynamic elements are added, just add `data-qa-mask` to them

### Example

```html
<!-- Before -->
<input readonly value="http://localhost:10062/en-gb/browse/#5bb2a4227bbc/KtGIsLN7XYjh..." />

<!-- After -->
<input readonly data-qa-mask="transfer-url"
       value="http://localhost:10062/en-gb/browse/#5bb2a4227bbc/KtGIsLN7XYjh..." />
```

That's it. One attribute. The QA pipeline handles the rest.

## Implementation Plan

### The Two-Screenshot Approach

Every `capture()` call saves **two files**: the real screenshot (full UI with actual data) and a deterministic version (dynamic content masked). Both are committed. Both are published to the Jekyll site. The deterministic version is used for diffing.

```
screenshots/
├── 06_link_captured.png                    ← real screenshot (actual URLs, timing, tokens)
└── 06_link_captured__deterministic.png     ← masked screenshot (solid blocks over dynamic content)
```

**Why two files:**
- The real screenshot shows the actual UI — essential for documentation, debugging, and visual review
- The deterministic screenshot is stable across runs — enables accurate pixel diffing
- `diff_screenshots` compares the `__deterministic` versions; if unchanged, it reverts **both** files
- Both are published to the QA site — power users can inspect the deterministic version to understand exactly what's being masked and verified

**The flow:**

```
ScreenshotCapture.capture()
    │
    ├─ Save real screenshot:          06_link_captured.png
    │   (full UI, actual dynamic data)
    │
    ├─ Query [data-qa-mask] elements
    │
    └─ Save deterministic screenshot: 06_link_captured__deterministic.png
        (masked with solid color over dynamic elements)

diff_screenshots (CI step)
    │
    ├─ Compare __deterministic PNGs against previous commit
    │
    ├─ < 1% change in deterministic? → revert BOTH files (no commit noise)
    │
    └─ > 1% change in deterministic? → keep BOTH files (real layout change)
```

### ScreenshotCapture Implementation

```python
def capture(self, page, name, description):
    real_path = self._path_for(name)
    det_path  = self._deterministic_path_for(name)     # name__deterministic.png

    # 1. Real screenshot — full UI, actual data, for documentation
    page.screenshot(path=real_path)

    # 2. Deterministic screenshot — masked, for diffing
    mask_elements = page.locator("[data-qa-mask]").all()
    if mask_elements:
        page.screenshot(
            path       = det_path,
            mask       = mask_elements,
            mask_color = self.mask_color                # "#1a2332" dark / "#f8f9fa" light
        )
    else:
        # No maskable elements on this page — deterministic = real
        import shutil
        shutil.copy2(real_path, det_path)

    self._record_metadata(name, description, has_masks=bool(mask_elements))
```

### diff_screenshots Update

```python
def diff_screenshots(site_dir):
    for det_path in glob(f"{site_dir}/**/screenshots/*__deterministic.png", recursive=True):
        real_path = det_path.replace("__deterministic.png", ".png")

        # Compare deterministic version against git HEAD
        change_pct = pixel_diff_percentage(det_path, git_head_version(det_path))

        if change_pct < THRESHOLD:                      # < 1% — noise, revert both
            git_checkout(det_path)
            git_checkout(real_path)
        # else: > 1% — real change, keep both for commit
```

### Jekyll Site — Showing Both Versions

Both screenshots are published. The generated markdown shows the real screenshot by default, with the deterministic version available for power users:

```markdown

Combined link shown after successful upload

![06 Link Captured](screenshots/06_link_captured.png)

<details>
<summary>View deterministic version (dynamic content masked)</summary>

![06 Link Captured — deterministic](screenshots/06_link_captured__deterministic.png)

*Elements with `data-qa-mask` are painted over for stable visual comparison across CI runs.*
</details>
```

This uses HTML `<details>` — collapsed by default, one click to expand. Casual readers see the real screenshot. Power users (QA engineers, developers debugging diffs) can expand to see exactly what's masked and what's being compared.

### QA_Generate_Docs Update

The doc generator needs to:
1. Pair each `__deterministic.png` with its real counterpart
2. Generate the `<details>` block only for screenshots that have a deterministic companion
3. Skip screenshots without a companion (no dynamic content on that page)

```python
def _screenshot_markdown(self, name, description, has_deterministic):
    md = f"### {name}\n\n{description}\n\n"
    md += f"![{name}](screenshots/{name}.png)\n\n"

    if has_deterministic:
        md += f"<details>\n"
        md += f"<summary>View deterministic version (dynamic content masked)</summary>\n\n"
        md += f"![{name} — deterministic](screenshots/{name}__deterministic.png)\n\n"
        md += f"*Elements with `data-qa-mask` are painted over for stable visual comparison across CI runs.*\n"
        md += f"</details>\n\n"

    return md
```
