# Playwright Optimization: Page Classes, State Capture, and Scalable Test Architecture

## Context

The Playwright automation is working. Now it needs to be optimised and made scalable. Several techniques need to be applied to move from Explorer-phase automation to Villager-phase test infrastructure.

## Technique 1: Page Object Classes

Each page with significant functionality gets its own class that encapsulates the interaction logic for that page:

```python
class UploadPage:
    def __init__(self, page):
        self.page = page

    def drop_file(self, path): ...
    def select_delivery_mode(self, mode): ...
    def get_step_indicator_state(self): ...
    def extract_page_model(self): ...

class GalleryPage:
    def __init__(self, page):
        self.page = page

    def get_card_count(self): ...
    def open_lightbox(self, index): ...
    def switch_to_folder(self): ...
    def extract_page_model(self): ...
```

These classes leverage the custom Playwright wrappers we have already built (no timeouts, synchronous-feeling calls) and become the building blocks for the higher-level workflow commands. The test never touches raw Playwright selectors. The page class handles that.

## Technique 2: Page State as JSON

For each page, capture the state as a structured JSON file. This is not a screenshot (visual). This is the data model of what is on the page at a point in time:

```json
{
  "page": "upload",
  "step": "select",
  "test_files": ["test-text.txt", "test-image.png", "test-document.pdf"],
  "mode": "file",
  "access_token_visible": false,
  "credits_remaining": 125,
  "upload_zone_active": true,
  "captured_at": "2026-03-25T10:30:00Z"
}
```

This page state JSON can be:

- Used across multiple test executions (skip setup steps by loading saved state)
- Compared between versions (diff the v0.2.x page model against the v0.3.0 page model)
- Used for multi-step analysis and review without re-running the browser
- Versioned alongside the tests (state snapshots per release)

### Cache Hygiene

The state files should be scoped to a test session (tied to the port configuration file we already store). They should not persist across sessions by default, to avoid testing against stale state. In debug mode, reuse state between runs to avoid slow setup. In CI mode, always start fresh.

## Technique 3: Schema Classes for Page Data

For pages with rich data (upload with its test files and delivery options, gallery with its cards and lightbox), create typed schema classes that represent the data on the page:

```python
@dataclass
class UploadPageModel:
    step: str
    mode: str  # "file" or "text"
    test_files: list[TestFile]
    credits_remaining: int
    upload_zone_active: bool

@dataclass
class GalleryPageModel:
    cards: list[GalleryCard]
    total_files: int
    view_mode: str  # "gallery" or "folder"
    share_link: str | None
```

These are serialisable (to JSON, to disk). They represent the page at a moment in time. The `extract_page_model()` method on each page class produces one of these. Tests assert against the model, not against HTML elements.

This means tests read like:

```python
model = upload_page.extract_page_model()
assert model.step == "select"
assert len(model.test_files) == 9
assert model.credits_remaining > 0
```

Not like:

```python
step_text = page.locator('.step-indicator .active').text_content()
assert "Upload" in step_text
```

The schema is the contract. The extraction logic handles the DOM. The test checks the contract.

## Technique 4: localStorage as Inter-Test Communication

Use the browser's localStorage as a lightweight communication channel between tests:

- Test A uploads a file, stores the transfer ID and key in localStorage
- Test B reads the transfer ID and key from localStorage, tests the download
- No need to pass state through Python variables or external files

localStorage is easy to read and write from Playwright (`page.evaluate`), persists across navigations within the same browser context, and is naturally scoped to the test session.

This is especially useful for upload-then-download workflows where the upload test produces data that the download test needs.

## Technique 5: Stable Port Configuration

The port configuration file (already implemented) ensures the API server and static site use consistent ports across sessions. This means:

- Bookmarked URLs still work between debug sessions
- State files that reference localhost:PORT remain valid
- CI runs use the same ports every time (deterministic)

Small detail, big impact on developer experience during interactive test development.

## How These Connect

```
Page Object Class
  │
  ├── extract_page_model() → Schema Class → JSON state file
  │
  ├── action methods (drop_file, click_send, etc.)
  │
  └── leverages custom Playwright wrappers (sync, no timeouts)

Tests assert against Schema Classes (not DOM)
State passes between tests via localStorage or JSON files
Port config keeps everything stable across sessions
```
