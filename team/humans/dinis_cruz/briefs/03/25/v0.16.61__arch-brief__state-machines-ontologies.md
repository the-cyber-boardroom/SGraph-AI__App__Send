## Why Now

This is a good example of waiting for the right time to develop something. The need has arrived from the QA team: browser automation requires state capture, state handoff between workflows, and a way to describe what is on a page and what should happen next. The QA team needs this to do its job.

But the same infrastructure serves security (threat models are graphs of the architecture), design (user flows are graphs of interactions), documentation (feature maps are graphs of capabilities), and version comparison (is v0.3.0 equivalent to v0.2.x? compare the graphs).

The complexity already exists. We usually just do not deal with it, or deal with it inefficiently through hardcoded tests and manual descriptions. Now we deal with it properly.

## The Three Layers

### Layer 1: Page Object Models (Schemas)

Every page has a structured representation of its data and state. This is the schema layer from the Playwright optimization brief, but it goes further than testing.

The page model for the upload page captures:
- Which step the user is on (select, delivery, share mode, confirm, progress, done)
- What files are loaded
- What delivery mode is selected
- What the access token state is
- What is clickable, what is disabled, what is visible

This is not HTML. This is the semantic meaning of what is on the page. It is serialisable to JSON. It is diffable. It is versionable.

### Layer 2: Flow Graphs (State Machines)

The workflows through the application are state machines:

```
Upload Flow:
  select → delivery → share_mode → confirm → progress → done
  (with branches: text mode skips delivery, single file skips folder options)

Download Flow:
  access_gate → decrypt → route_detection → [gallery | browse | viewer | auto_download]

Gallery Flow:
  grid → lightbox → [navigate | save | print | share | switch_to_folder]
```

Each state has:
- Entry conditions (what must be true to enter this state)
- Available actions (what the user can do)
- Transitions (which states can follow)
- Data produced (what information is generated at this state)

These flow graphs are the formal description of how the application works. They serve QA (test every transition), design (visualise the user journey), and development (the state machine is the spec).

### Layer 3: Ontology / Taxonomy

The semantic structure of the entire application:

- **Entities:** Users, transfers, files, folders, vaults, access tokens, encryption keys
- **Relationships:** User creates transfer. Transfer contains files. File has encryption key. Transfer has access token. Access token resolves to encryption key.
- **Taxonomies:** File types (image, document, code, audio, video). Delivery modes (link with key, friendly token, direct link). View modes (gallery, folder, viewer).
- **Use cases:** Share a video. Send a document securely. Collaborate on a vault. Each use case is a path through the flow graph involving specific entities and relationships.

This is the knowledge graph of the application. It connects the page models (what is on the screen) to the flow graphs (what happens in what order) to the domain model (what the entities are and how they relate).

## What This Enables

### Version Comparison as Graph Diff

The question "is v0.3.0 equivalent to v0.2.x?" becomes: are the graphs the same?

- Extract the page models from v0.2.x (run the QA suite against v0.2.x, capture schemas)
- Extract the page models from v0.3.0 (run the QA suite against v0.3.0, capture schemas)
- Diff the graphs

Missing nodes = missing features. Extra nodes = new features. Changed edges = changed workflows. This is a formal, automated answer to "did the refactoring preserve all functionality?"

### Bugs as Graph Anomalies

A bug is something that we have mapped in the graph that is not happening in reality. The graph says "from gallery, clicking 'Folder view' transitions to the folder view." The test says "clicking 'Folder view' navigates away from the page." That is a graph anomaly. The expected transition does not match the actual transition.

This reframes bugs from "something is broken" to "the reality diverges from the model." The model is the source of truth. The code is supposed to implement the model. When they diverge, either the code has a bug or the model needs updating.

### Threat Modelling from the Graph

The security team builds threat models on top of the same graph:

- The ontology shows which entities handle sensitive data (encryption keys, file content)
- The flow graph shows where data transitions between states (plaintext to encrypted, client to server)
- Threat modelling asks: at each transition, what could go wrong? What if the encryption step is skipped? What if the access token is leaked?

The graph IS the architecture diagram. The threat model IS the annotated graph. No separate documentation needed.

### Feature Requests as Graph Extensions

A new feature is a new node or edge in the graph. "Add PDF present mode to the gallery" is: add a transition from gallery lightbox to fullscreen PDF viewer. The graph shows where it connects, what states it affects, and what tests need updating.

## Implementation: Start from QA, Zoom Out

The pragmatic path:

1. **Start with page schemas** (already needed for Playwright optimization)
2. **Add flow graphs for the upload and download workflows** (already implicit in the test structure)
3. **Serialise to JSON files in the QA repo** (version-controlled, diffable)
4. **Visualise with Mermaid or a graph tool** (for human review and documentation)
5. **Build the ontology incrementally** (add entities and relationships as they are tested)
6. **Compare v0.2.x and v0.3.0 graphs** (the first real validation of the approach)

Do not try to model the entire application at once. Build the graph from the tests. Every test that passes adds a confirmed edge to the graph. Every bug is a missing or incorrect edge.

## Type Safety

The schemas and state machines should be type-safe:

- Python dataclasses for page models (serialisable to JSON)
- Enum types for states, transitions, entity types
- JSON Schema for the serialised format (validatable)

This means the graph is not just documentation. It is executable. The test framework can validate that the graph matches reality on every test run.
