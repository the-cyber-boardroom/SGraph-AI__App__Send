# QA Site Improvements: GitHub Integration, Code Visibility, and Release Documentation

**version** v0.16.61
**date** 25 Mar 2026
**from** Human (project lead)
**to** Architect (lead), Sherpa, Designer, Developer, QA
**type** Dev brief

---

## Context

The QA site at qa.send.sgraph.ai is live and working. Tests run, screenshots are captured, pages are generated, and the site auto-deploys on every green CI push. This is already a significant achievement. Now it needs to become more than a test report. It needs to become the living documentation of the product.

The v0.2.30 debrief (from the QA team) describes the foundation: event-driven browser tests with zero `wait_for_timeout()` calls, five schema page objects, and the full screenshot-to-docs-to-GitHub-Pages pipeline. That foundation is solid. This brief describes the next phase.

## Review Required First

Before making changes, the Architect, Sherpa, and Designer need to do a thorough review of what exists now:

- Homepage structure and content
- Navigation and information architecture
- How flows are explained (or not)
- What is missing for someone trying to understand the product
- How the current one-to-one test-to-page mapping serves (or does not serve) different audiences

## Improvement 1: Code Alongside Screenshots

Currently, each page shows the test description and the screenshots. What is missing: the actual code that produced each screenshot.

### What to Add

For each test page:

1. **The full test code** displayed on the page (syntax highlighted, collapsible). The user can see exactly what the test does, step by step.

2. **Per-screenshot code snippets.** Next to each screenshot, show the specific line(s) of code that triggered that screenshot. The reader sees: "this action was performed" and "this is what the screen looked like after."

This means a screenshot entry becomes:

```
Action: page.drop_file("test-photo.png")
Code:   upload_page.drop_file(test_files["photo"])
        upload_page.wait_for_state("file-ready")
        screenshot("02_file_selected")

[Screenshot: 02_file_selected.png]
```

The reader can trace every visual state to the exact code that produced it. This is documentation, QA evidence, and developer reference simultaneously.

## Improvement 2: Bidirectional GitHub Links

### From QA Site to GitHub

Every test page should link to:
- The test source file on GitHub (direct link to the file in the repo)
- The specific commit that generated the current screenshots
- The CI run that produced this version of the page

### From GitHub to QA Site

Every test file in the repo should include (in a comment or companion file):
- The URL of the corresponding QA site page
- A badge or link in the README pointing to the live documentation

The goal: from any test on GitHub, one click to see the visual documentation. From any QA page, one click to see the source code on GitHub.

## Improvement 3: Commit and Version Provenance

### Capture the Hash

For every test execution, capture and display:
- The Git commit hash that the test ran against
- The version number (v0.16.61, etc.)
- The test file hash (to detect when the test itself has changed)

This provides provenance: "this screenshot was captured at commit abc123, version v0.16.61, using test code with hash xyz789." If a screenshot changes, you can trace whether the test changed or the application changed.

### Version Number Ordering

The version number increment should happen before the documentation generation step in the CI pipeline. This ensures the generated pages reference the correct version. Currently the increment tag may happen after docs are generated, which creates a mismatch.

## Improvement 4: User Stories and Feature Documentation

The QA site currently maps one-to-one to tests. But users do not think in tests. They think in features and workflows. Add a layer that connects:

- **Use cases** (what the user wants to do: "share a file," "view a gallery")
- **Tests** (what verifies the use case works)
- **Screenshots** (what the user sees at each step)

This turns the QA site into user-facing documentation. The same screenshots that verify the product also explain the product. The Sherpa's walkthrough content and the QA evidence are the same material.

## Improvement 5: LLM-Assisted Review

Add prompts and workflows for regular LLM review of the QA site:

1. **Screenshot analysis.** Give the screenshots to Claude and ask: "What bugs do you see? What looks wrong? What could be improved?" Many visual issues are immediately obvious from a screenshot.

2. **Test coverage review.** Give the test list to Claude and ask: "What scenarios are missing? What edge cases are not covered?"

3. **Documentation quality review.** Give the generated pages to Claude and ask: "Is this clear? What would a new user not understand?"

These reviews should happen regularly (weekly or on each major release) and produce tasks that feed back into the QA and development workflow.

## Improvement 6: Release Notes from Tests

Every release is a set of changes. Every change should have associated tests. The release notes should be generated from:

- Issue/task files created for each piece of work
- The tests that verify each fix or feature
- Before/after screenshots where the UI changed
- The commit messages and version numbers

The release note for "v0.16.61" becomes a page listing every resolved issue with links to the test page, the screenshots, the code, and the GitHub commit. This is not manually written. It is assembled from the artifacts that already exist in the CI pipeline.

### Issue Files

Start creating issue files (markdown, one per task/bug/feature) that live in the repo. Each issue file references:
- The version it was reported in
- The version it was fixed in
- The test that verifies it
- The QA page that documents it

These issue files are the atoms. The release note is the molecule.

## Improvement 7: Static Enrichment During Build

The Jekyll site build step can do more than scaffold pages. During the build, it can:

- Pull commit messages from Git and include them on version pages
- Pull PR descriptions and include them in release notes
- Pull file change statistics and include them in coverage reports
- Generate dependency graphs or architecture diagrams from the codebase

All of this runs during the build step, produces static HTML, and is served from GitHub Pages. No dynamic backend needed. The build step is the intelligence layer.
