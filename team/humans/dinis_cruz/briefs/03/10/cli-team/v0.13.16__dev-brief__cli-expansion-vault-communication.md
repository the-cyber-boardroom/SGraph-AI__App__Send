## The Breakthrough: Vault Init from CLI

We already had vault clone (read a vault locally). What we just shipped is vault init (create a vault from the CLI on the live server). Combined with the existing clone and push, this means the full lifecycle works from the command line:

```
sg-send-cli init    -> creates a new vault on the server
sg-send-cli clone   -> clones an existing vault locally
sg-send-cli push    -> pushes local changes back to the server
sg-send-cli pull    -> pulls remote changes to local
```

The server has no idea it's storing a vault. It sees a folder of encrypted files. The tree structure, file metadata, and content are all encrypted blobs that only make sense with the vault key. The server is a dumb blob store, exactly as designed.

## Part 1: Claude as a Vault Peer (Confirmed Working)

### The Idea

If Claude (running in claude.ai or Claude Code) can `pip install sg-send-cli`, and the target domain is on the egress allowlist, then Claude can:

1. Clone a vault (read-only, needs only the vault key)
2. Read and reason about the vault contents
3. Create new files in the vault
4. Push changes back (needs an access token for writes)

This makes Claude a peer in a version-controlled encrypted vault workflow. Not a code generator that produces files for the human to place. An actual participant that reads from and writes to the same vault the human uses.

### Confirmed End-to-End

The session findings (v0.13.14) and debrief (v0.13.15) document the full round-trip:

1. Claude installed `sg-send-cli` via pip in the bash sandbox
2. Claude cloned vault `i0brp94w` from `dev.send.sgraph.ai`
3. Claude read the vault contents (demo transforms, HTML pages, experiments)
4. Claude created `claude-web.txt` inside the vault
5. Claude pushed the file back to the server
6. The human ran `sg-send-cli pull` locally and received the file

**Zero-knowledge preserved throughout.** The server never saw plaintext. Claude encrypted client-side using the vault key. The human decrypted client-side using the same key.

### What This Enables

| Workflow | Before | After |
|---|---|---|
| **Claude produces a brief** | Claude writes text in conversation. Human copies to a file. | Claude writes directly to the vault. Human pulls. |
| **Claude reviews code** | Human pastes code into conversation. Claude comments. | Claude clones the vault, reads the code, writes review notes, pushes. |
| **Deploy secrets** | Secrets passed via environment variables, pasted in chat, or stored insecurely. | Deploy Service stores ephemeral secrets in a vault. Claude (or CI/CD) reads them via CLI. |
| **Cross-team handover** | Human copies briefs between repos manually. | Team A writes brief to shared vault. Team B clones the vault and reads it. |
| **Agent-to-agent communication** | Agents communicate through conversation history (expensive, lossy). | Agents communicate through a shared vault (persistent, versioned, encrypted). |

## Part 2: Expand the CLI Beyond Vault Operations

The CLI is currently vault-focused. But it's called `sg-send-cli`, not `sg-vault-cli`. It should cover the full SG/Send API surface.

### File Transfer Primitives

| Command | What It Does |
|---|---|
| `sg-send-cli upload <file>` | Upload a file, get a download link |
| `sg-send-cli upload <folder>` | Upload a folder (zip or individual files) |
| `sg-send-cli upload <file> --zip` | Upload as a zip archive |
| `sg-send-cli download <transfer-id>` | Download a file by transfer ID |
| `sg-send-cli download <transfer-id> --zip` | Download as zip |
| `sg-send-cli download <transfer-id> --partial <files>` | Download specific files from a folder transfer |

### Vault Operations (Already Partially Implemented)

| Command | Status |
|---|---|
| `sg-send-cli vault init` | Just shipped |
| `sg-send-cli vault clone <key>` | Working |
| `sg-send-cli vault push` | Working |
| `sg-send-cli vault pull` | Working |
| `sg-send-cli vault status` | Needed (show local vs remote diff) |
| `sg-send-cli vault log` | Needed (show commit history) |

### PKI Operations

| Command | What It Does |
|---|---|
| `sg-send-cli pki generate` | Generate a new key pair |
| `sg-send-cli pki add <public-key>` | Add a public key to the local keyring |
| `sg-send-cli pki list` | List known keys |
| `sg-send-cli pki sign <file>` | Sign a file with your private key |
| `sg-send-cli pki verify <file> <signature>` | Verify a signature |
| `sg-send-cli pki encrypt <file> <public-key>` | Encrypt for a specific recipient |

### Secrets Management

The CLI runs on the user's machine, which means it has access to the OS keychain:

| Command | What It Does |
|---|---|
| `sg-send-cli secrets store <key> <value>` | Store a secret in the OS keychain (macOS Keychain, Linux secret-service, Windows Credential Manager) |
| `sg-send-cli secrets get <key>` | Retrieve a secret |
| `sg-send-cli secrets list` | List stored secret keys (not values) |
| `sg-send-cli secrets delete <key>` | Remove a secret |

This is significant because it means the CLI can store access tokens, vault keys, and PKI private keys in the OS keychain instead of in plaintext config files. The integration with existing password managers (1Password, Bitwarden, etc.) comes for free if they integrate with the OS keychain.

### Why This Matters for Claude Code

Claude Code can `pip install sg-send-cli`. With the CLI covering the full API surface:

```bash
# Claude Code can upload a file
sg-send-cli upload ./report.pdf --token $SG_TOKEN

# Claude Code can download a file
sg-send-cli download abc123 --key apple-mango-56

# Claude Code can manage vault contents
sg-send-cli vault clone $VAULT_KEY --base-url https://dev.send.sgraph.ai
sg-send-cli vault push --token $SG_TOKEN

# Claude Code can manage keys
sg-send-cli pki generate --store  # generates and stores in OS keychain
```

This means Claude Code becomes a first-class SG/Send client. Not via MCP (which has context window constraints for binary data). Not via web_fetch (which has URL allowlist constraints). Via the actual CLI, the same tool the human uses.
