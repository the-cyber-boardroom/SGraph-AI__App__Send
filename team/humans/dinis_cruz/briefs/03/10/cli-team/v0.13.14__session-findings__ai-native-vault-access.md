# Session Findings: AI-Native Vault Access via MCP

**Version:** v0.13.14  
**Date:** 10 March 2026  
**Session:** Conductor + Architect (claude.ai web)  
**Vault tested:** `i0brp94w` on `dev.send.sgraph.ai`

---

## What We Were Trying To Do

Enable a Claude session (claude.ai web) to clone and read an SG/Send encrypted vault entirely from within the AI session — no human running CLI commands, no local terminal. The goal: Claude receives a vault key, clones the vault, reads the files, and can reason about the contents.

The motivating use case is an AI agent that can access a user's vault as part of an agentic workflow — treating the vault as a zero-knowledge document store the agent can read from (and eventually write to).

---

## The Environment Constraints

Claude sessions (claude.ai web) operate with two distinct execution contexts that cannot directly share data:

| Context | What it can do | Network access |
|---------|---------------|----------------|
| **MCP layer** | Call MCP tools, receive JSON responses | Reaches `dev.send.sgraph.ai` via Anthropic's MCP infrastructure |
| **`bash_tool`** | Run Python, write files, run CLI tools | Egress proxy — only allows `github.com`, `pypi.org`, `api.anthropic.com` etc. `send.sgraph.ai` is **blocked** |

These two contexts cannot directly share memory. Data can only move between them by passing through Claude's **context window** — i.e. as visible text in the conversation.

Additionally:
- `web_fetch` respects `robots.txt` — `send.sgraph.ai` disallows `/api/*` for AI crawlers (Cloudflare-managed)
- `sg-send-cli` installed fine from PyPI but its direct HTTP calls are blocked by the egress proxy

## What We Tried

### Attempt 1: Run `sg-send-cli clone` directly
```bash
sg-send-cli clone aaaaaaaai3fn72ppuoksteht:bbbbbb94w
```
**Result:** `OSError: Tunnel connection failed: 403 Forbidden`  
Egress proxy blocks outbound HTTPS to `send.sgraph.ai`. Dead end.

---

### Attempt 2: `web_fetch` the vault API directly
```
web_fetch: https://send.sgraph.ai/api/vault/read/i0brp94w/5c59a41b5322
```
**Result:** `ROBOTS_DISALLOWED`  
`send.sgraph.ai` has a robots.txt that disallows `/api/*`. `web_fetch` respects this.

---

### Attempt 3: MCP `api_vault_read`
```
MCP: api_vault_read(vault_id="i0brp94w", file_id="5c59a41b5322")
```
**Result:** `'utf-8' codec can't decode byte 0xd1 in position 5`  
The endpoint IS reachable and data IS there — but it returns raw AES-256-GCM binary ciphertext. The MCP transport chokes on non-UTF-8 bytes. Data confirmed present, but unreadable.

---

### Attempt 4: Add `api_vault_read-base64` endpoint ✅

**Action taken:** Filed a dev brief. The Send team shipped `GET /api/vault/read-base64/{vault_id}/{file_id}` to `dev.send.sgraph.ai` and added it to the MCP manifest.

**Result:** Works. Returns:
```json
{
  "vault_id": "i0brp94w",
  "file_id": "5c59a41b5322",
  "data": "<base64 encoded ciphertext>",
  "size": 239
}
```

---

### Attempt 5: Full end-to-end vault read for small vault ✅

With `api_vault_read-base64` available, and `Vault__Crypto` from `sg-send-cli` installed in `bash_tool`:

**Call 1:** `api_vault_read-base64(vault_id, tree_file_id)`  
→ base64 ciphertext (239 bytes)  
→ Python: `Vault__Crypto.decrypt(read_key, base64.decode(...))` → tree JSON:
```json
{
  "version": 3,
  "tree": {"/": {"children": {
    "test.txt": {"file_id": "a58388877fb7", "size": 38}
  }}}
}
```

**Call 2:** `api_vault_read-base64(vault_id, "a58388877fb7")`  
→ base64 ciphertext (66 bytes)  
→ Python decrypt → `"this is a test file!!!!!\n\nChanged here\n"`

**Total: 2 MCP calls. Full vault cloned. Zero-knowledge preserved throughout.**

This worked because the vault was small — base64 strings were short enough to pass through the context window without friction.

---

### Attempt 6: Same vault after it grew ❌ (partially)

The vault tree grew to 3,709 bytes. The base64 encoding of the tree ciphertext became a ~5,000 character string. Every approach to get this from the MCP layer into `bash_tool` hit the same wall:

- **Python heredoc** — streams the entire base64 as visible text through the UI. Takes minutes.
- **`create_file`** — same problem, the file content has to be written as text through the UI rendering pipeline.
- **`cat` with heredoc** — same.

The base64 string must travel as visible text from the MCP context into the bash subprocess. There is no IPC, no shared memory, no filesystem path accessible to both contexts. Everything goes through the conversation as tokens.

## Additional Experiments (Same Session)

### Experiment: Open robots.txt + web_fetch as alternative to MCP

**Hypothesis:** if `web_fetch` can reach the vault API directly, the response arrives as a collapsed document in the UI rather than streaming through a bash command box — potentially cleaner than MCP for getting data into context.

**Step 1: robots.txt was blocking `/api/*`**

Original `dev.send.sgraph.ai/robots.txt` had no explicit `Allow: /`, causing Cloudflare to block AI crawlers from `/api/*`. Changed to `Allow: /` with only `/_common/` and `/i18n/` disallowed.

**Step 2: web_fetch URL allowlist is an independent gate**

Even with robots.txt open, `web_fetch` refused API URLs that hadn't been explicitly provided by the user or appeared in prior search results:
```
PERMISSIONS_ERROR: This URL cannot be fetched because it was not provided 
by the user nor did it appear in any search/fetch results
```
This is a separate, harder constraint — it sits in front of robots.txt entirely.

**Step 3: User-provided URL unlocks web_fetch ✅**

Once `https://dev.send.sgraph.ai/api/vault/read-base64/i0brp94w/5c59a41b5322` was pasted directly into the conversation, `web_fetch` succeeded and returned the full JSON response. Crucially, the response arrived as a **collapsed document** in the UI — not streaming as a wall of text.

Also confirmed via this path: `https://dev.send.sgraph.ai/api/openapi.json` — full OpenAPI spec (~6.2KB, 28 endpoints) returned cleanly.

**Step 4: Decrypt still hits the same wall**

The base64 data is in Claude's context window regardless of whether it arrived via MCP or `web_fetch`. To decrypt it in `bash_tool`, it still has to travel as a literal string into a Python subprocess. Same fundamental constraint, different delivery mechanism.

**Conclusion on web_fetch:**

Not a scalable solution for automated vault access — every URL must be user-provided or appear in search results, so it can't be used programmatically. Useful for ad-hoc human-in-the-loop exploration (reading openapi.json, fetching a known file URL), but not for agentic vault clone workflows.

---

### Corrected constraint ordering

The original findings incorrectly framed robots.txt as a primary constraint. Correct ordering:

```
1. web_fetch URL allowlist   ← hardest gate; URL must come from user or search results
2. bash_tool egress proxy    ← blocks all direct HTTP to send.sgraph.ai from subprocess  
3. robots.txt                ← now open on dev; was never the fundamental problem
```

Fixing robots.txt on `dev.send.sgraph.ai` was the right call regardless — it enables ad-hoc `web_fetch` access when URLs are user-provided. But it doesn't unblock agentic workflows.

## What We Found

### Finding 1: The context window IS the data pipe

MCP responses are tokens in Claude's context window. `bash_tool` is a separate subprocess. The only way to move data between them is as text rendered in the conversation. For small payloads this is invisible. For large payloads it becomes the bottleneck.

This is a known MCP constraint — Claude Code documents a 25,000 token default limit on MCP tool output, configurable via `MAX_MCP_OUTPUT_TOKENS`. In claude.ai web there is no equivalent setting.

### Finding 2: Binary data is the worst case

Base64-encoded binary is:
- ~33% larger than the raw bytes
- Completely opaque (no natural token boundaries, no compression)
- Has to be rendered as a wall of characters in the UI before it can be used

A 3,709 byte encrypted blob becomes ~4,950 base64 characters, which is ~1,240 tokens of pure noise in the context window.

### Finding 3: robots.txt blocks direct `web_fetch` to `/api/*`

`send.sgraph.ai` uses Cloudflare-managed robots.txt that disallows all `/api/*` paths for AI crawlers. This is independent of the egress proxy — even if we could reach the domain via `web_fetch`, the robots.txt would block API endpoints. This is actually correct security posture for the production domain.

### Finding 4: The egress proxy is the hard wall

`dev.send.sgraph.ai` is not on the egress allowlist for `bash_tool`. There is no way for the installed `sg-send-cli` to make direct HTTP calls to the vault API. The MCP connector works because it routes through Anthropic's own infrastructure, not through the bash subprocess network.

### Finding 5: The pattern that works (for small files)

```
1. Derive keys locally: sg-send-cli derive-keys {vault_key}  →  tree_file_id, read_key
2. MCP: api_vault_read-base64(vault_id, tree_file_id)         →  base64 tree
3. Write to /tmp via create_file                              →  /tmp/tree.json
4. bash: python3 vault_clone.py tree {vault_key} /tmp/tree.json  →  file manifest
5. For each file_id in manifest:
   MCP: api_vault_read-base64(vault_id, file_id)              →  base64 blob
   Write to /tmp/file_{id}.json
   bash: python3 vault_clone.py file {vault_key} /tmp/file_{id}.json {output_path}
```

**Scales to:** Vaults with small individual files (< ~1KB each). Breaks down when tree or file blobs produce large base64 strings.

---

## The Core Limitation

**All data returned by MCP tools must fit comfortably in Claude's context window AND be transferable to `bash_tool` as rendered text.**

For vault operations this means: **the sum of all base64-encoded blobs in a single MCP response must be small enough to write to a file without streaming visibly through the UI.**

Practically: individual files up to ~500 bytes work fine. A tree with many entries or files > 1KB becomes painful.

## What Would Actually Fix This

### Option A: Add `dev.send.sgraph.ai` to bash egress allowlist ✅ SOLVED

The cleanest fix. `sg-send-cli` already has the correct implementation. The entire problem goes away.

**This was implemented during the session.** Claude.ai Settings → Code execution → Domain allowlist → Added `dev.send.sgraph.ai`.

**Important caveat discovered:** The egress proxy JWT is baked into the container at session start. The allowlist change only takes effect in the **next conversation** — the running container keeps the old JWT and returns `x-deny-reason: host_not_allowed` until a fresh session is started.

**Result in new session: Full round-trip confirmed working.**

`sg-send-cli clone`, file creation, and `sg-send-cli push` all executed natively from `bash_tool` with zero context window involvement for data transfer. Confirmed by screenshot: `claude-web.txt` visible in vault alongside existing files, containing:

```
Hello from Claude (claude-web.txt)
================================
This file was created by Claude Sonnet 4.6 running inside claude.ai on 2026-03-10.
The sg-send-cli vault was cloned, this file was added, and then pushed back —
all from within a Claude conversation using the bash_tool.
Vault: 630v8qcri3fn72ppuoksteht
Key:   i0brp94w
```

This is the target state: Claude acting as a zero-knowledge vault agent, with no human running CLI commands.

### Option B: MCP server writes to a shared location
No longer the primary path — Option A solved the problem cleanly. Still relevant for AI platforms where custom egress allowlists aren't available.

### Option C: `api_vault_bundle` endpoint
Still worth building as a fallback for contexts where egress can't be opened. A single MCP call returning the full vault bundle is cleaner than N individual `read-base64` calls through the context window.

### Option D: Accept the constraint and design for it
No longer the primary recommendation. Superseded by Option A.

## What Works Right Now

With `dev.send.sgraph.ai` added to the claude.ai domain allowlist (Settings → Code execution → Domain allowlist), a Claude session can:

- ✅ `sg-send-cli clone {vault_key}` — full vault clone directly from `bash_tool`
- ✅ Read, create, modify files in the cloned vault locally
- ✅ `sg-send-cli push` — push changes back to the vault
- ✅ Full zero-knowledge preserved throughout — server never sees plaintext
- ✅ No context window involvement for data transfer — binary blobs never touch the conversation
- ✅ Works for vaults of any size — no 3KB limit, no streaming problem

**Proven end-to-end:** Claude cloned vault `i0brp94w`, created `claude-web.txt`, and pushed it back — all within a single `bash_tool` session, confirmed visible in the vault.

**Prerequisite:** The domain allowlist change only takes effect in a new conversation (JWT is baked into the container at session start).
