# Memo: AI-Native Vault Clone — Minimum Call Design

**To:** CLI Explorer Team — Architect  
**From:** Conductor (on behalf of Send team session, 10 March 2026)  
**Version:** v0.13.14  
**Subject:** What is the minimum number of API calls to clone a vault for an AI agent?

---

## Context

We are working on enabling AI agents (specifically Claude, operating via an MCP connector) to clone and read an SG/Send vault from inside a session where:

- Direct outbound HTTP to `send.sgraph.ai` is **blocked** by an egress proxy
- The **SG/Send MCP connector** IS reachable (it goes through Claude's infrastructure, not the egress proxy)
- The MCP connector currently exposes: `api_vault_read`, `api_vault_write`, `api_vault_delete`

The agent has the vault key (`{passphrase}:{vault_id}`) and can run local Python via `bash_tool` — so `Vault__Crypto` for decryption is available. The constraint is purely on the **network I/O side**: every file fetch must go through an MCP tool call, not direct HTTP.

The question: **what is the minimum number of MCP calls to fully clone a vault?**

---

## Our Current Understanding

From reading the `Vault__Sync.clone()` source and the architecture brief (`v0.11.1`), our instinct is:

### Minimum: 2 calls (plus N for files)

**Call 1 — Get the tree** (deterministic file_id, derived from vault key):

```
api_vault_read(vault_id, tree_file_id)
→ encrypted tree blob
→ decrypt locally → flat map of {file_path: file_id}
→ now we know exactly which file_ids exist in the vault
```

**Call 2..N — Get each file** (file_ids learned from the tree):

```
for each (file_path, file_id) in tree:
    api_vault_read(vault_id, file_id)
    → decrypt locally → write to disk
```

So for a vault with F files: **2 + F calls total** (tree + settings + F files, or just tree + F files if settings is optional for a read-only clone).

The tree gives us the complete manifest. There is no discovery step — the tree_file_id is deterministic from the vault key, so call 1 is always known before any network I/O happens.

---

## The Questions for the Architect

### Q1: Is the tree truly sufficient as the only manifest?

Our reading of the current `Vault__Sync` is that `tree.json` contains every file_id needed to reconstruct the vault. Is that still true in the object-store model (v0.11.1)? Or does the object-store format require walking a commit chain first (ref → commit → tree → blobs), meaning additional calls before we even know the file_ids?

Specifically: for the current format (not yet migrated to object-store), is it:

```
Call 1: tree_file_id  → tree  → {file_path: file_id}   # done, have all IDs
```

Or for object-store format:

```
Call 1: ref_file_id   → {commit_id}
Call 2: commit_id     → {tree_id}
Call 3: tree_id       → {file_path: blob_id}
Calls 4..N: each blob_id → file content
```

That's 3 + N calls for the object-store model, vs 1 + N for legacy. Worth knowing before we commit to a call-count assumption.

### Q2: Is there a way to bundle the tree + all files into a single encrypted payload?

The instinct from the Send team session: could the server offer a `GET /api/vault/bundle/{vault_id}` endpoint that returns a single encrypted archive containing:

```json
{
  "tree":  "<base64 encrypted tree>",
  "files": {
    "<file_id>": "<base64 encrypted blob>",
    "<file_id>": "<base64 encrypted blob>",
    ...
  }
}
```

The server can do this because it already knows all the file_ids stored under a vault_id — it doesn't need to decrypt anything. It just bundles what it has.

This reduces the AI agent clone to **a single MCP call**, regardless of vault size. Decrypt everything locally after.

**Security question for Architect + AppSec:** Does bundling multiple encrypted blobs in one response leak anything the server doesn't already know? Our assessment is no — the server already knows the file_ids and blob sizes. Bundling them doesn't reveal additional information. But we want the Architect's formal assessment before proposing this to Dev.

### Q3: What is the current vault size for `i0brp94w`?

We know the vault exists (MCP `api_vault_read` returns data for `tree_file_id` and `settings_file_id`, confirming the vault is live on `dev.send.sgraph.ai`). We don't know how many files are in it. Can you check, or is there a way for the agent to determine file count before committing to N sequential MCP reads?

### Q4: Is `settings` required for a read-only clone?

`Vault__Sync.clone()` currently downloads both `settings` and `tree`. For an AI agent doing a read-only clone (no push), is `settings` actually needed? If not, that saves one call.

---

## Our Proposal (Subject to Architect Confirmation)

For the immediate AI-native clone path, we propose:

**Phase 1 (minimal server change):**
- Add `GET /api/vault/read-base64/{vault_id}/{file_id}` (already briefed)
- Agent does 1 + N MCP calls (tree + N files)
- Agent uses `Vault__Crypto` locally for all decryption

**Phase 2 (optimal):**
- Add `GET /api/vault/bundle-base64/{vault_id}` returning all blobs in one JSON response
- Agent does 1 MCP call, decrypts everything locally
- Requires Architect + AppSec sign-off on the bundling model

The bundle endpoint is the real prize. If the Architect confirms it's clean, it makes the AI-native vault clone as efficient as it can possibly be — and it's useful beyond AI agents, for any client that wants a full vault snapshot in one round trip.
