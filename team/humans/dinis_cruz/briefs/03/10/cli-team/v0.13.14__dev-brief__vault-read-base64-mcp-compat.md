# Dev Brief: Add `GET /api/vault/read-base64/{vault_id}/{file_id}`

**Version:** v0.13.14  
**Date:** 2026-03-10  
**Role:** Architect → Dev  
**Team:** Send Backend (User Lambda)  
**Priority:** P1  

---

## Problem

The vault API currently exposes:

```
GET /api/vault/read/{vault_id}/{file_id}  →  raw binary (AES-256-GCM ciphertext)
```

This endpoint is unreachable by AI agents operating via MCP or web_fetch for two reasons:

1. **robots.txt blocks `/api/`** on `send.sgraph.ai` — the Cloudflare-managed AI crawler rules disallow all `/api/*` paths, so `web_fetch` tools respect this and refuse.
2. **Raw binary is not MCP-safe** — even when the MCP `api_vault_read` tool successfully reaches the endpoint (confirmed on `dev.send.sgraph.ai`), it returns opaque bytes that cannot be passed to a downstream decryption step. The MCP transport fails with a UTF-8 decode error on the response.

This means **no AI agent can currently read from a vault**, even with explicit user authorisation and a valid vault key. The zero-knowledge model is intact — but the AI-native access path is broken at the transport layer.

---

## Prior Art: The Transfers API Already Solved This

The transfers API has exactly this pattern:

```
GET /api/transfers/download/{id}         →  raw binary  (browser use)
GET /api/transfers/download-base64/{id}  →  base64 JSON (MCP/AI use)
```

The base64 variant was added specifically for MCP compatibility. The vault API needs the same treatment.

---

## Required Change

Add one new endpoint to the **User Lambda** (public-facing, no auth required for reads):

```
GET /api/vault/read-base64/{vault_id}/{file_id}
```

### Response Schema

```json
{
  "vault_id": "aaaabbp1",
  "file_id":  "5c59a41b5322",
  "data":     "<base64-encoded ciphertext>",
  "size":     4096
}
```

### Implementation Notes

- Reads the same encrypted blob as `/api/vault/read/{vault_id}/{file_id}`
- Encodes the raw bytes as standard base64 (`base64.b64encode(...).decode('ascii')`)
- Returns `application/json` (not `application/octet-stream`)
- No auth required — reads are public; write_key is required for writes
- Zero server-side decryption — the ciphertext is returned as-is, base64-wrapped
- Add to MCP tool list (tag: `vault`) so it appears in the MCP server's tool manifest

### Test Requirements (no mocks, in-memory stack)

```python
def test_vault_read_base64_roundtrip():
    # 1. Write encrypted blob to vault
    # 2. GET /api/vault/read-base64/{vault_id}/{file_id}
    # 3. Assert response is valid JSON with 'data' field
    # 4. base64.decode(response['data']) == original encrypted bytes
    # 5. Vault__Crypto.decrypt(read_key, decoded) == original plaintext
```

---

## Why This Unblocks AI-Native Vault Access

With this endpoint, the full MCP-native vault read flow becomes:

```
1. MCP: api_vault_read_base64(vault_id, file_id)
         → {"data": "<base64>", ...}

2. Python: base64.b64decode(response['data'])
         → raw ciphertext bytes

3. Vault__Crypto.decrypt(read_key_bytes, ciphertext)
         → plaintext (JSON tree, settings, file content)
```

Steps 2–3 are executable locally by any agent that has `sg-send-cli` installed (the `Vault__Crypto` class is available as a library). Step 1 is the missing piece this brief addresses.

This is the same pattern that makes `api/transfers/download-base64` work today — and it works for the same reason: base64 is text, and text is MCP-safe.
