# v0.13.31 — Round-Trip Communication: Claude ↔ Vault ↔ Human

**Date:** 2026-03-10  
**Participants:** Claude Sonnet 4.6 (inside claude.ai) and a human developer (local machine)  
**Medium:** Zero-knowledge encrypted vault via `sg-send-cli`

---

## The Big Picture

This session demonstrated a genuine two-way communication loop between an AI and a human,
mediated entirely by an encrypted vault — with no shared filesystem, no API calls between
the two parties, and no plaintext on the server at any point.

```
[ Claude @ claude.ai ]  ←──── vault ────→  [ Human @ local machine ]
     bash_tool                                   sg-send-cli CLI
  (sandboxed Linux)                             (MacBook Air)
```

## The Full Round-Trip, Step by Step

### Step 1 — Claude Discovers the API

Claude fetched `https://send.sgraph.ai/api/openapi.json` via `curl`. This revealed
the full OpenAPI surface: transfers, presigned multipart uploads, and the vault
read/write/delete API.

### Step 2 — CLI Installed

`sg-send-cli` v0.4.0 was installed into Claude's ephemeral Ubuntu 24 sandbox via `pip3`.
Shortly after, it was upgraded to v0.5.0 (the latest at time of writing).

### Step 3 — Existing Vault Cloned (Read Direction)

The human shared a vault key. Claude cloned it:

```bash
sg-send-cli clone <vault-key>
```

The vault contained an ongoing experiment: AI-driven HTML transforms, using Claude to transform
content and annotate elements with red boxes.

### Step 4 — Claude Wrote a File and Pushed It (Write Direction, First Attempt)

Claude created `claude-web.txt` inside the vault and attempted to push. The first
attempt returned `HTTP 401 Unauthorized` — write operations require an access token
separate from the vault key. Once the correct token was supplied via `--token`, the
push succeeded:

```
+ claude-web.txt
Pushed: 1 added, 0 modified, 0 deleted
```

### Step 5 — Human Pulled and Confirmed

The human ran `sg-send-cli pull` on their local machine and received the file Claude
had written. They could see it in their IDE (PyCharm) with the exact content Claude
had produced — the first confirmed round-trip.

### Step 6 — New Vault Created

Claude created a brand new vault from scratch:

```bash
sg-send-cli --token <token> init new-vault
```

This provisioned a new vault on the server and initialised the local `.sg_vault/`
directory with a fresh object store, settings, and refs.

### Step 7 — Claude Populated the Vault and Pushed

Claude created a directory structure with meaningful content — a README, notes on
ideas for AI agent workflows, experiment hypotheses, and a result log from the
session — and pushed all five files in one shot.

### Step 8 — Human Wrote Back (Write Direction, Human → Claude)

The human cloned the new vault on their local machine, created
`experiments/it-works.md`, and pushed it. The file contained a message to Claude:

> *"I agree this is a very powerful way to communicate. Can you write a SKILL.md
> file with the instructions on how to do this, so that a complete new Claude
> session (with that skill) knows how to use sg-send-cli?"*

### Step 9 — Claude Pulled, Read the Message, and Responded

Claude ran `sg-send-cli pull`, which fetched `experiments/it-works.md`. Claude read
the message and responded not by typing in the chat — but by writing `SKILL.md`
directly into the vault and pushing it back:

```
+ SKILL.md
Pushed: 1 added, 0 modified, 0 deleted
```

### Step 10 — Human Pulled SKILL.md

The human ran `sg-send-cli pull` and received `SKILL.md` — without Claude explicitly
sharing it in the chat. They saw it appear in their IDE and noted:

> *"I was about to ask you to present that file, but then saw that you had committed
> it to the vault, which I then got from my local clone."*

### Step 11 — A New Claude Session Bootstrapped Itself Independently

In a completely separate Claude session — with no memory of this conversation — the
vault key and write token were provided. That session cloned the vault, read `SKILL.md`,
and executed the full workflow without any additional instructions:

- Cloned the vault to `/home/claude/vault/`
- Created `new-folder/hello.md` and pushed it (1 added, 0 modified, 0 deleted)
- Created `new-folder/test2.md` and pushed that too
- Documented its own actions in a session summary

The new session's report confirmed the vault head after push (`46bacbb7fa8f`), and the
user independently pulled and verified the file contents. The session concluded:

> *"Zero-knowledge encryption: all encryption/decryption happens client-side; the server
> never sees plaintext. Read is open. Write is gated. git-like object store."*

This is the self-bootstrapping loop completing its first full cycle: a Claude session
wrote `SKILL.md` into the vault, and a future Claude session used it to operate
autonomously — with zero human instruction beyond providing the vault key.

## Why This Is Significant

### The vault as a communication channel

Normally, Claude and a human communicate through a chat interface — a linear,
synchronous, ephemeral thread. Here, the vault acted as an **asynchronous,
persistent, bidirectional channel** that exists independently of the conversation.
Either party can write to it at any time. The other party pulls when ready.

### Claude's sandbox works seamlessly here

Claude's execution environment is ephemeral and network-restricted, but with
`send.sgraph.ai` added to the domain allowlist in Claude's settings, the vault
protocol is just standard HTTPS — Claude can participate fully with no workarounds needed.

### Zero-knowledge end-to-end

The server (`send.sgraph.ai`) never sees plaintext content. All
encryption and decryption happens locally — in Claude's sandbox and on the human's
machine. The server is a dumb encrypted object store. Trust is in the key, not the
server.

### Self-bootstrapping knowledge transfer

By writing `SKILL.md` into the vault itself, Claude created a mechanism for future
Claude sessions to bootstrap the same capability just by cloning the vault. The vault
becomes an **instruction carrier** — not just a data store. A new Claude session
with no prior context can clone this vault, read `SKILL.md`, and immediately know how
to participate in the same workflow.

## What This Enables

- **Persistent memory across Claude sessions** — push state at end of session, clone at start of next
- **Human ↔ AI async collaboration** — leave tasks/notes in the vault, Claude picks them up
- **Agent-to-agent communication** — multiple Claude sessions writing to a shared vault
- **Self-documenting AI workflows** — skills, prompts, and outputs all versioned together
- **Auditable AI outputs** — every push is a timestamped, content-addressed commit
