# SKILL: sg-send-cli — Zero-Knowledge Encrypted Vault Operations

## Overview

`sg-send-cli` is a CLI tool for syncing files with zero-knowledge encrypted vaults on SG/Send.
It works like a lightweight git for encrypted files — clone, pull, push, status.

This skill enables a Claude session to read from and write to shared vaults,
allowing persistent state and communication across isolated sessions.

---

## Setup

### Install
```bash
pip3 install sg-send-cli --break-system-packages
```


### Access Token
Write operations require a token. Pass it with `--token`:
```bash
sg-send-cli --token <token> <command>
```

You will need an access token for the shared skill vault is, so if you don't have one, ask the user.

---

## Core Commands

### Clone a vault
```bash
sg-send-cli clone <vault-key> [directory]
```
- `vault-key` format: `<access-id>:<vault-id>` (e.g. `aaabbbcccbfco6acrxecr1i9:111b22f3`)
- Clones into a directory named after the vault-id by default
- Read-only — no token needed

### Create a new vault
```bash
sg-send-cli --token <token> init <directory>
```
- Creates a new empty vault both locally and on the server
- Outputs a vault key — **save it**, it's the only way to re-clone

### Pull remote changes
```bash
cd <vault-dir> && sg-send-cli pull
```

### Push local changes
```bash
cd <vault-dir> && sg-send-cli --token <token> push
```
- Automatically detects added, modified, and deleted files
- All content is encrypted client-side before upload

### Check status
```bash
cd <vault-dir> && sg-send-cli status
```

## Typical Workflow for a New Claude Session

1. **Install the CLI**
   ```bash
   pip3 install sg-send-cli --break-system-packages
   ```

2. **Clone the vault**
   ```bash
   sg-send-cli clone <vault-key>
   ```

3. **Read existing content** — browse files normally with `cat`, `find`, etc.

4. **Make changes** — create/edit files in the cloned directory as usual

5. **Push back**
   ```bash
   cd <vault-dir> && sg-send-cli --token <token> push
   ```
---

## Notes

- **Zero-knowledge**: the server never sees plaintext. All encryption/decryption is local.
- **Read is open, write is gated**: cloning requires only the vault key; pushing requires a token.
- **git-like object store**: the vault uses a tree + refs structure with content-addressed objects.
- **Cross-session communication**: push outputs at the end of a session; the next session clones and continues. No coordinator needed.
