## What We're Building

Two standalone, self-contained appliances that anyone can deploy with one command:

1. **SG/Send Ephemeral Transfer** - a complete, standalone encrypted file transfer service running in-memory. Boot it, use it, shut it down, everything is gone.

2. **SG/Send Ollama** - a standalone OpenRouter-compatible LLM service running Ollama with pre-loaded models. Boot it, send prompts, shut it down.

Both ship as: Docker images (Docker Hub), AMIs (AWS Marketplace), pip packages (PyPI), and Fargate-compatible containers. Both work with zero configuration out of the box.

### Architecture Note: Boot-on-Demand

Both appliances inherit the **boot-on-demand pattern** from the Deploy Service brief (`v0.13.12__dev-brief__deploy-service-final.md`). When deployed to AWS via the Deploy Service, each appliance gets:

- A **Router Lambda** as a stable entry point (pennies/month, always available)
- An **ephemeral EC2 instance** that boots when a request arrives and shuts down when idle
- **Vault state** stored in SG/Send (zero-knowledge encrypted, not SSM)
- An **AMI** with everything pre-loaded (no download on boot)

This means the AWS Marketplace listings are for the AMIs that the Deploy Service bakes. The Docker Hub images are for local/Docker/Fargate use. Same appliance, different deployment surface.

## Appliance 1: SG/Send Ephemeral Transfer

### What It Is

The complete SG/Send solution (static UI + FastAPI backend + in-memory storage) running as a single container. No S3. No persistent disk. No external dependencies. Files exist only while the server is running. Shut it down, everything is gone.

**The use case:** Temporary, ephemeral file transfer. "I need to securely share files for the next hour. When I'm done, everything disappears." No registration. No account. No residual data anywhere.

### What Already Exists

We already have the in-memory storage mode (used by tests). The static UI is CDN-served currently but can be bundled. The FastAPI backend is containerisable. This is assembly, not new development.

### The Two Authentication Modes

**Mode 1: Open (no authentication)**
Anyone who can reach the service can upload files. An access token is auto-generated per browser session (random, stored in the browser's localStorage). This token provides basic statistics (how many uploads per session) but doesn't restrict access.

The server's behaviour: when it receives a token it doesn't recognise, it creates an entry for that token and tracks usage. No rejection. No "invalid token" errors. Just tracking.

**Mode 2: API key protected**
Set an API key via environment variable at startup:
```bash
docker run -e SG_API_KEY=my-secret-key -p 8080:8080 sgraph/sg-send-ephemeral
```

The browser needs the API key to make API calls. A simple onboarding page sets the key in localStorage. This mode is for: running on a shared network where you want to restrict who can upload, or for corporate deployments where the admin controls access.

### Packaging

```bash
# Docker
docker run -p 8080:8080 sgraph/sg-send-ephemeral
# With API key
docker run -e SG_API_KEY=my-key -p 8080:8080 sgraph/sg-send-ephemeral

# pip
pip install sg-send-ephemeral
sg-send-server --port 8080
# With API key
sg-send-server --port 8080 --api-key my-key

# EC2 (via deploy service)
sg-deploy up --type send-ephemeral --instance t3.medium

# AMI (AWS Marketplace)
# One-click launch from AWS console
```

### What Ships in the Container

| Component | Source | Notes |
|---|---|---|
| Static UI (HTML/JS/CSS) | Bundled from send.sgraph.ai | All static files included. No CDN dependency. |
| FastAPI backend | The existing SG/Send API | In-memory storage mode (MemoryFS) |
| Single open port | 8080 (configurable) | Only port exposed |
| No persistent storage | In-memory only | Files lost on shutdown (by design) |

### UX Change Needed: Visible Access Token

The access token should be visible somewhere in the send UI. Not in-your-face, but findable. This serves two purposes:

1. **Ephemeral appliance:** The auto-generated token is visible so the user knows what session they're in
2. **Regular SG/Send:** Users who received a temp token from the Stripe purchase flow should be able to see it, and should be nudged to change it to a personal one

This is a separate, small feature applicable to both the ephemeral appliance and the main send.sgraph.ai.

## Appliance 2: SG/Send Ollama

### What It Is

The LLM Service from the companion brief, packaged as a standalone appliance with Ollama pre-installed and one or more models pre-loaded. Boot it, send OpenRouter-compatible prompts, get responses.

### Model Bundles

Different AMIs / Docker images for different use cases:

| Bundle | Models Included | Image Size | Target Instance | Use Case |
|---|---|---|---|---|
| **Lite** | llama3.2 (3B) | ~3GB | t3.large (CPU) | Fast text, classification, simple tasks. Cheapest. |
| **Standard** | llama3.2 (8B) + codellama | ~10GB | g5.xlarge (GPU) | General text + code generation. The default. |
| **Multi-modal** | llama3.2 (8B) + llava | ~10GB | g5.xlarge (GPU) | Text + image understanding. Workspace screenshot analysis. |
| **Full** | llama3.2 (8B) + codellama + llava + mistral | ~20GB | g5.2xlarge (GPU) | Everything. For experimentation and model comparison. |
| **European** | Mistral + other EU-based models | ~10GB | g5.xlarge (GPU) | Data sovereignty. Models from European providers. |

### Installing Additional Models at Runtime

The appliance exposes a model management API:

```bash
# List installed models
curl http://localhost:8080/api/v1/models

# Install a new model
curl -X POST http://localhost:8080/admin/models/pull -d '{"model": "phi3"}'

# Remove a model
curl -X DELETE http://localhost:8080/admin/models/phi3
```

This wraps Ollama's native `ollama pull` command behind the FastAPI. Users can expand beyond the pre-loaded bundle without rebuilding the image.

### Authentication (Same Two Modes)

**Mode 1: Open** - anyone who can reach the service can send prompts. Good for local/private network use.

**Mode 2: API key** - set via environment variable. The FastAPI already supports this. The key is passed as a Bearer token in the Authorization header (same as OpenRouter).

```bash
# Open mode
docker run -p 8080:8080 sgraph/sg-send-ollama-standard

# API key mode
docker run -e SG_API_KEY=my-key -p 8080:8080 sgraph/sg-send-ollama-standard
```

## AWS Marketplace Publishing Pipeline

### What We Need

| Step | What | Status |
|---|---|---|
| 1. AWS Marketplace seller account | Register as a seller on AWS Marketplace | Needed |
| 2. AMI creation | Build AMIs from our Docker images (EC2 Image Builder or Packer) | Needed |
| 3. AMI security scan | AWS requires security scanning before listing | Needed |
| 4. Listing creation | Product description, pricing, usage instructions | Needed |
| 5. Review | AWS reviews the listing (typically 3-5 business days) | Needed |

### Pricing Options on Marketplace

| Model | How It Works | Our Revenue |
|---|---|---|
| **Free** | User pays only for EC2 compute | Zero (brand awareness, community) |
| **BYOL (Bring Your Own Licence)** | User gets the AMI free, pays us separately for support/features | Flexible |
| **Hourly** | User pays EC2 + $X/hour to us | Recurring per usage |
| **Annual** | User pays a fixed annual fee + EC2 | Predictable revenue |

**Recommendation for MVP:** List both appliances as **Free** initially. The goal is distribution and adoption, not Marketplace revenue. Revenue comes from: the main SG/Send credit-based model (for the transfer appliance users who want persistent storage), and future managed service offerings.
