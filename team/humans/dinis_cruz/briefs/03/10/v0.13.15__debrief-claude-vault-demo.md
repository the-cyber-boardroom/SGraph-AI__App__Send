# Debrief: Claude Writing to a Zero-Knowledge Encrypted Vault

**Date:** 2026-03-10  
**Summary:** A live demonstration of Claude (running inside claude.ai) interacting end-to-end with a zero-knowledge encrypted vault via `sg-send-cli`, successfully writing a file that was then pulled down on the user's local machine.

---

## What Happened

### 1. API Exploration

The session started by fetching the OpenAPI spec from the SG/Send dev server directly via `curl` from Claude's bash environment. The spec revealed three main API groups: `api/transfers` (core file transfer lifecycle), `api/presigned` (S3-style multipart uploads), and `api/vault` (persistent encrypted key-value store).

### 2. Installing the CLI

`sg-send-cli` (v0.4.0) was installed via `pip3` inside Claude's sandboxed Linux environment, pulling in `osbot-utils` as a dependency alongside the already-present `cryptography` and `cffi` packages.

### 3. Cloning the Vault

Running `sg-send-cli clone <vault-key>` failed initially with a `403 Forbidden` tunnel error — the CLI defaults to `https://send.sgraph.ai`, which is outside Claude's allowed egress domains. Inspecting the source revealed a `--base-url` flag. Re-running with `--base-url https://dev.send.sgraph.ai` (which is in the allowed domain list) succeeded, cloning the vault to `/i0brp94w/`.

### 4. Inspecting the Vault Contents

The cloned vault contained ~1.7MB of files across two main directories:

- **`demo-transforms/`** — reusable JavaScript snippets for DOM manipulation (ad removal, article extraction, AI-only filtering)
- **`html-pages/`** — experiments applying AI-driven content transforms to real news pages from `text.npr.org` and `theregister.com`, testing red-box annotation, negative content removal, and blurring — run against both Claude Haiku and Claude Sonnet 4.6

The prompt pattern used in the experiments was: instruct the model to return modified HTML with negative or unwanted content removed, without breaking the page layout.

### 5. Writing a New File

A new file `claude-web.txt` was created inside the vault directory with a short note documenting its origin. The first push attempt returned `HTTP 401 Unauthorized` — the vault ID alone is not sufficient for writes, an access token is also required. Once the correct write token was supplied via the `--token` flag, the push succeeded:

```
+ claude-web.txt
Pushed: 1 added, 0 modified, 0 deleted
```

### 6. Pull Confirmed on the User's Side

On the user's local machine, running `sg-send-cli pull` immediately picked up the new file:

```
+ claude-web.txt
Pulled: 1 added, 0 modified, 0 deleted
```

The file appeared in the IDE (PyCharm) with the exact content Claude had written, confirming the full round-trip.

---

## Key Observations

**Egress constraints are real.** Claude's network sandbox only permits connections to a specific allowlist of domains. The default production endpoint was blocked; only the dev endpoint worked. Being able to inspect the CLI source and identify the `--base-url` override was essential.

**Zero-knowledge means read is open, write is gated.** Cloning (read) required only the vault key embedded in the transfer ID. Writing required a separate access token — the server never sees plaintext content, but it still enforces write authorization.

**Claude can participate in real developer workflows.** This wasn't a simulated demo — the file was genuinely encrypted, stored remotely, and retrieved by an independent client. Claude acted as a peer in a version-controlled encrypted vault workflow, not just as a code generator.
