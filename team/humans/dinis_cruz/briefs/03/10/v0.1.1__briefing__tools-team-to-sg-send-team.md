## 3. Architecture That's Now Established

### Three-Tier Structure (Working)

```
sgraph_ai_tools__static/
  core/                     ← Pure JS, no UI
    crypto/v1/v1.0/v1.0.0/    sg-crypto.js
    ssh/v1/v1.0/v1.0.0/       sg-ssh.js

  components/               ← JS + CSS + HTML (Shadow DOM Web Components)
    base/v1/v1.0/v1.0.0/      sg-component.js          ← Base class (vault pattern)
    tokens/v1/v1.0/v1.0.0/    sg-tokens.css             ← Canonical brand tokens
    site-header/v1/v1.0/v1.0.1/  sg-site-header.{js,html,css}  ← Current version
    site-footer/v1/v1.0/v1.0.0/  sg-site-footer.{js,html,css}
    locale-picker/v1/v1.0/v1.0.0/ sg-locale-picker.{js,html,css}

  tools/                    ← Standalone pages (compose core + components)
    v0/v0.1/v0.1.1/           Landing page, SSH Keygen, 404, i18n, CSS
```

### Component Composition Pattern (New in v1.0.1)

The site-header now uses **`<slot>`-based composition** so any page can compose a locale picker (or anything else) into the header:

```html
<sg-site-header site="Tools" nav-items='[...]'>
    <sg-locale-picker slot="locale" locale="en-gb" base-path="/tools/v0/v0.1/v0.1.1/"></sg-locale-picker>
</sg-site-header>
```

This is important for the Send team because **sgraph.ai can import these same components** from the tools.sgraph.ai CDN, replacing the hand-coded header + `renderLocaleSelector()` in i18n.js. See section 7 for details.

### What a New Tool Page Looks Like

Every tool page follows this template:

```html
<!-- 1. Load shared components -->
<script type="module" src="/components/site-header/v1/v1.0/v1.0.1/sg-site-header.js"></script>
<script type="module" src="/components/locale-picker/v1/v1.0/v1.0.0/sg-locale-picker.js"></script>

<!-- 2. Compose header with locale picker -->
<sg-site-header site="Tools" nav-items='[...]'>
    <sg-locale-picker slot="locale"></sg-locale-picker>
</sg-site-header>

<!-- 3. Tool-specific content -->
<div class="main">...</div>

<!-- 4. Import core module and wire up -->
<script type="module">
    import { someFunction } from '/core/module-name/v1/v1.0/v1.0.0/sg-module.js';
    // ... tool logic
</script>

<!-- 5. Shared footer -->
<sg-site-footer brand-desc="..." columns='[...]'></sg-site-footer>
```

Any new tool brief should follow this exact pattern.

## 4. What Each Component Does (API Reference)

### sg-component.js (Base Class) — v1.0.0

The foundation that all UI components extend. Provides:
- Shadow DOM isolation
- Automatic fetch of sibling `.html` template + `.css` styles
- Shared CSS injection (design tokens)
- Event tracking with auto-cleanup on disconnect
- `onReady()` / `bindElements()` / `setupEventListeners()` lifecycle

**Any new component should extend `SgComponent`.** It handles all the plumbing.

### sg-site-header — v1.0.1 (current)

| Attribute | Type | Description |
|-----------|------|-------------|
| `site` | string | Product name after "SG/" (e.g. "Tools", "Send", empty for root sgraph.ai) |
| `home-url` | string | Logo click target |
| `nav-items` | JSON | Array of `{label, href}` for navigation links |

| Slot | Description |
|------|-------------|
| `locale` | Accepts any element (designed for `<sg-locale-picker>`) |

### sg-locale-picker — v1.0.0

| Attribute | Type | Description |
|-----------|------|-------------|
| `locale` | string | Current locale code (e.g. `en-gb`) |
| `base-path` | string | URL prefix before the locale slug |

| Export | Description |
|--------|-------------|
| `SG_LOCALES` | Array of 17 locale objects: `{code, name, flag}` |

Renders a dropdown trigger button + 2-column grid of 17 locales. Handles navigation on locale change.

### sg-site-footer — v1.0.0

| Attribute | Type | Description |
|-----------|------|-------------|
| `brand-desc` | string | Description under SG/ logo |
| `columns` | JSON | Array of `{title, links: [{label, href}]}` |
| `copyright` | string | Override copyright text |
| `sherpa-text` | string | CTA button text |
| `sherpa-href` | string | CTA button link |

### sg-tokens.css — v1.0.0

Canonical brand tokens as CSS custom properties (`:host` scoped for Shadow DOM). Includes:
- Brand colours (`--sg-color-teal`, `--sg-color-navy`, etc.)
- Semantic tokens (`--sg-bg`, `--sg-text`, `--sg-accent`, `--sg-border`)
- Typography (`--sg-font-display`, `--sg-font-mono`)
- Spacing (`--sg-sp-1` through `--sg-sp-8`)

These map 1:1 to the sgraph.ai website's CSS custom properties but use the `--sg-` prefix and `:host` instead of `:root`.

### sg-crypto.js — v1.0.0 (UNVERIFIED)

AES-256-GCM encryption via Web Crypto API. Exports: `generateKey()`, `encryptFile()`, `decryptFile()`, `importKey()`, `exportKey()`, etc.

**Warning:** Written from scratch, not copied from Send production crypto. Wire-format compatibility not tested.

### sg-ssh.js — v1.0.0

Ed25519 SSH key pair generation via Web Crypto API. Exports: `generateSSHKeyPair({comment})`.

**Limitation:** Only Ed25519. Send's version also supports RSA-4096.

## 5. CI/CD Pipeline

### What's Deployed

| Workflow | Trigger | What It Does |
|----------|---------|-------------|
| `deploy-tools.yml` | Push to `dev` or `main`, or manual `workflow_dispatch` | Generates i18n pages, injects build version, syncs to S3, creates `latest/` alias, invalidates CloudFront |
| `deploy-tools__dev.yml` | Push to `dev` | Calls consolidated workflow |
| `deploy-tools__main.yml` | Push to `main` | Calls consolidated workflow |

### S3 Path Structure

```
s3://{bucket}/websites/sgraph-tools/dev/releases/v0/v0.1/v0.1.1/   ← versioned
s3://{bucket}/websites/sgraph-tools/dev/latest/                      ← CI alias
s3://{bucket}/websites/sgraph-tools/main/releases/v0/v0.1/v0.1.1/
s3://{bucket}/websites/sgraph-tools/main/latest/
```

### What's Blocked

AWS secrets are not yet configured in GitHub. CI workflows will run but skip the S3 sync step. Once Dinis configures secrets and creates the CloudFront distribution, deployment is automatic.

**Current access:** `dev.tools.sgraph.ai` is live and serving from S3 (v0.1.0). The v0.1.1 version needs a push to `dev` to deploy.

## 8. What the Next Tools Brief Should Know

### Adding a New Tool — The Checklist

For any tool the Send team briefs us to build, here's what's needed:

1. **Core module** (if the tool needs one) — pure JS file at `core/{name}/v1/v1.0/v1.0.0/sg-{name}.js` with `manifest.json`
2. **Tool page** — HTML file at `tools/v0/v0.1/v0.1.1/en-gb/{tool-name}/index.html` following the template in section 3
3. **i18n keys** — add translation keys to all 5 JSON files in `i18n/`
4. **Landing page** — add a tool card to `en-gb/index.html`
5. **No build step** — every file must be deployable as-is (no webpack, no bundler)
6. **No frameworks** — vanilla JS + ES modules only
7. **Client-side only** — no server calls for processing

### Things That Are Ready for New Tools

| Capability | Ready? | Notes |
|-----------|--------|-------|
| Shadow DOM component system | Yes | Extend `SgComponent`, get templating + styling for free |
| Shared header/footer/locale picker | Yes | Just add `<script>` tags and elements |
| Design tokens (colours, spacing, fonts) | Yes | `sg-tokens.css` loaded automatically by all components |
| i18n pipeline | Yes | Add keys to JSON, CI generates locale pages |
| File download from browser | Yes | Pattern proven in SSH keygen (Blob + ObjectURL) |
| Clipboard API | Yes | Pattern proven in SSH keygen |
| Web Crypto API | Yes | Pattern proven in SSH keygen (Ed25519), crypto module has AES-256-GCM |
| Upload / drag-and-drop | No | Upload dropzone component listed as proposed — not built |
| FFmpeg WASM | No | Video splitter needs this — not yet integrated |
| WebSocket / streaming | No | LLM client would need this for streaming responses |

### Suggested Priority for Next Tools

Based on what's already in the codebase and what would be easiest to ship:

| Tool | Effort | Why |
|------|--------|-----|
| **File Encryptor** | Low | `sg-crypto.js` already has `encryptFile()` / `decryptFile()`. Just needs a page with file picker + download. |
| **File Hasher** | Low | Web Crypto `digest()` is trivial. No core module needed — inline in the page. |
| **Video Splitter** | High | Requires FFmpeg WASM integration (~3MB download), worker thread, progress UI |
| **LLM Client** | High | Requires extraction from workspace, API key management, streaming UI |
| **Audio Transcription** | High | Requires Whisper WASM — very large binary, complex integration |
