## What This Service Does

The LLM Service runs Ollama and exposes it via a FastAPI wrapper that is **100% compatible with OpenRouter**. Any code that calls OpenRouter works with this service by changing only the endpoint URL. Same request format, same response format, same streaming protocol.

This is the data plane. It receives LLM requests, executes them on Ollama, and returns results. It has no infrastructure management capabilities. It does not know how it was deployed. It just serves prompts.

```
┌──────────────────────────────────────────────────────────────────────┐
│  LLM APPLIANCE — what it is and what creates it                      │
│                                                                      │
│  CREATED BY:  Deploy Service (via SSH + osbot-utils)                 │
│  MANAGED BY:  Router Lambda (llm.sgraph.ai → boot-on-demand)         │
│  STATE IN:    SG/Send vault (vault://deploy/instances/llm-001.json)  │
│  SHUTS DOWN:  Automatically after idle timeout (default 15 min)      │
│                                                                      │
│  WHAT THE LLM EC2 ITSELF DOES:                                       │
│    ✓ Runs Ollama with pre-loaded models (baked into AMI)             │
│    ✓ Exposes OpenRouter-compatible FastAPI                            │
│    ✓ Writes heartbeat to vault every 60s                             │
│    ✓ Shuts itself down when idle                                     │
│    ✗ No SSH into other machines                                      │
│    ✗ No AWS management capability                                    │
│    ✗ No conversation state                                           │
└──────────────────────────────────────────────────────────────────────┘
```

## The 100% OpenRouter Compatibility Principle

This is the single most important design requirement. The workspace's `LlmOrchestrator` already supports two providers:

| Provider | Endpoint | Streaming | Auth |
|---|---|---|---|
| **OpenRouter** | `https://openrouter.ai/api/v1/chat/completions` | SSE | Bearer token |
| **Ollama (local)** | `http://localhost:11434/api/chat` | NDJSON | None |

The problem: Ollama's native API uses a different format (NDJSON streaming, different request schema). The workspace handles this with separate code paths (`_readSSE()` for OpenRouter, `_readNDJSON()` for Ollama).

The LLM Service eliminates this split. It wraps Ollama behind an OpenRouter-compatible API:

```
POST /api/v1/chat/completions     ← OpenRouter-format request in
     │
     ▼
  FastAPI translates to Ollama format
     │
     ▼
  Ollama executes (models pre-loaded in AMI — no download wait)
     │
     ▼
  FastAPI translates response to OpenRouter format
     │
     ▼
  SSE stream out                  ← OpenRouter-format response out
```

From the consumer's perspective, `llm.sgraph.ai/api/v1/chat/completions` is identical to `openrouter.ai/api/v1/chat/completions`. The workspace uses the OpenRouter code path with a different URL. No third code path. No special cases.

### API Contract

**Request (identical to OpenRouter):**
```json
{
  "model": "llama3.2",
  "messages": [
    { "role": "system", "content": "You are a helpful assistant." },
    { "role": "user", "content": "Hello" }
  ],
  "stream": true,
  "max_tokens": 1000,
  "temperature": 0.7
}
```

**Streaming response (SSE, identical to OpenRouter):**
```
data: {"id":"gen-abc","choices":[{"delta":{"content":"Hello"},"index":0}]}
data: {"id":"gen-abc","choices":[{"delta":{"content":" there"},"index":0}]}
data: {"id":"gen-abc","choices":[{"delta":{},"finish_reason":"stop","index":0}],"usage":{"prompt_tokens":25,"completion_tokens":12}}
data: [DONE]
```

**Non-streaming response (JSON, identical to OpenRouter):**
```json
{
  "id": "gen-abc",
  "choices": [{
    "message": { "role": "assistant", "content": "Hello there! How can I help?" },
    "finish_reason": "stop",
    "index": 0
  }],
  "usage": { "prompt_tokens": 25, "completion_tokens": 12, "total_tokens": 37 },
  "model": "llama3.2"
}
```

### Endpoints

| Endpoint | Method | Purpose | OpenRouter equivalent |
|---|---|---|---|
| `/api/v1/chat/completions` | POST | Chat completion (streaming + non-streaming) | Identical |
| `/api/v1/models` | GET | List available models | Identical format |
| `/health` | GET | Ollama status, models loaded, readiness | Custom |
| `/metrics` | GET | Token counts, request counts, latency | Custom |

## The Boot-on-Demand Pattern (inherited from Deploy Service)

The LLM appliance uses the same Router Lambda pattern as the Deploy Service. The LLM EC2 is not running 24/7. It boots when a request arrives, serves requests, and shuts itself down when idle.

```
STABLE ENTRY POINT
llm.sgraph.ai  →  API Gateway  →  Router Lambda (llm)
(permanent)                        (same pattern as deploy.sgraph.ai)

  ┌──────────────┐   ┌──────────────┐   ┌────────────────┐
  │   RUNNING    │   │   STOPPED    │   │   STARTING     │
  │              │   │              │   │                │
  │ Ollama up,   │   │ no instance  │   │ StartInstances │
  │ model loaded,│   │ or stopped   │   │ called,        │
  │ health pass  │   │              │   │ waiting...     │
  └──────┬───────┘   └──────┬───────┘   └───────┬────────┘
         │                  │                   │
         ▼                  ▼                   ▼
  307 Redirect        Start EC2 from       Waiting page
  to llm-001          LLM AMI, show        "Loading model..."
  .sgraph.ai          waiting page         polls /status
```

### The LLM-Specific Waiting Page

The LLM waiting page has a longer expected wait than the Deploy Service (model warm-up after EC2 boot), and should communicate this honestly:

```
┌──────────────────────────────────────────────────────────┐
│  SG/Send LLM                          llm-001            │
│  ──────────────────────────────────────────────────────  │
│                                                          │
│   🔄  Starting LLM instance...                           │
│                                                          │
│   EC2 booting (models pre-loaded — no download needed).  │
│   Usually ready in 60-90 seconds.                        │
│                                                          │
│   [████████████░░░░░░░░]  Waiting for Ollama...         │
│                                                          │
│   You'll be redirected automatically.                    │
│   Alternatively: set endpoint in your workspace now,    │
│   requests will queue until ready.                       │
└──────────────────────────────────────────────────────────┘
```

**Why "no download needed":** models are pre-loaded into the AMI at bake time. The instance does not pull models on boot. This is the critical difference between a usable cold start (60-90 seconds) and an unusable one (5-30 minutes for a 40GB model download).

## The AMI Baking Pipeline — Models Go In At Bake Time

This is the most important operational decision in this brief. **Models are never downloaded at runtime.** They are baked into the AMI by the Deploy Service in CI.

```
ENV4: CI WITH CREDS — LLM AMI BAKING
════════════════════════════════════════════════════════════════════════

  Deploy Service (running in CI) does this:

  Step 1: Boot a clean EC2 (GPU instance)
  ─────────────────────────────────────────
  ec2_instance.create_kwargs = dict(
      image_id      = UBUNTU_BASE_AMI,
      instance_type = 'g5.xlarge',        # GPU needed to pull large models
      spot_instance = True,
      tags          = {'Purpose': 'llm-ami-bake'}
  )
  ec2_instance.create()

  Step 2: SSH in and provision via osbot-utils
  ─────────────────────────────────────────────
  with ec2_instance.ssh(path_key_pair, 'ubuntu') as ssh:
      linux = ssh.ssh_linux()
      linux.apt_update()
      linux.apt_install('curl')

      # Install Ollama
      ssh.exec('curl -fsSL https://ollama.ai/install.sh | sh')
      ssh.exec('sudo systemctl enable ollama')
      ssh.exec('sudo systemctl start ollama')

      # Pull models INTO the instance (this is the slow part — happens once at bake)
      ssh.exec('ollama pull llama3.2')          # ~2GB
      ssh.exec('ollama pull codellama')         # ~4GB
      # 'ollama pull llama3.3:70b' for the full bundle (~40GB, longer bake)

      # Install FastAPI wrapper
      py = ssh.ssh_python()
      py.pip_install('sg-llm')
      py.pip_install('fastapi uvicorn')

      # Configure and enable the service
      ssh.scp().copy_file_to_host('/tmp/sg-llm.service', '/etc/systemd/system/')
      ssh.exec('sudo systemctl enable sg-llm')
      ssh.exec('sudo systemctl start sg-llm')

      # Verify before baking
      def verify_llm_ready():
          import requests, time
          for _ in range(30):
              try:
                  r = requests.get('http://localhost:8080/health')
                  if r.json().get('status') == 'ready':
                      return r.json()
              except:
                  pass
              time.sleep(2)
          raise Exception("LLM service did not become ready")

      ssh.ssh_python().execute_python__function(verify_llm_ready)

  Step 3: Bake the AMI with models already inside
  ─────────────────────────────────────────────────
  ami_id = ec2.create_image(
      instance_id = ec2_instance.instance_id,
      name        = f'sgraph-llm-standard-{version}',
      no_reboot   = True
  )

  Step 4: Write AMI ID to vault, terminate build instance
  ─────────────────────────────────────────────────────────
  vault.write('deploy/config/llm_standard_ami', ami_id)
  ec2_instance.delete()
  # clean up key pair, security group ingress rule


RESULT: When the Router Lambda starts this AMI, boot time is 60-90 seconds.
        Ollama starts, finds models already on disk, begins serving immediately.
        No download. No wait. Just boot.
```

### Model Bundles → Separate AMIs

Different AMIs for different use cases. The CI pipeline bakes all of them:

| Bundle | AMI name | Models inside | Instance type | Boot time |
|---|---|---|---|---|
| **Lite** | `sgraph-llm-lite-{v}` | llama3.2 (3B) | t3.large (CPU) | ~60s |
| **Standard** | `sgraph-llm-standard-{v}` | llama3.2 (8B) + codellama | g5.xlarge (GPU) | ~75s |
| **Multimodal** | `sgraph-llm-multimodal-{v}` | llama3.2 (8B) + llava | g5.xlarge (GPU) | ~75s |
| **Full** | `sgraph-llm-full-{v}` | llama3.2 + codellama + llava + mistral | g5.2xlarge (GPU) | ~90s |
| **European** | `sgraph-llm-eu-{v}` | Mistral + EU-origin models | g5.xlarge (GPU) | ~75s |

All AMI IDs written to vault on bake. The Router Lambda reads the correct AMI ID from vault for the requested bundle.

## Statelessness

The LLM Service is stateless by design. This is the one-shot principle applied to infrastructure:

| Property | Value |
|---|---|
| **Conversation history** | None. Every request is self-contained. |
| **Session state** | None. No sticky sessions. |
| **Persistent storage** | None. Models are on disk in the AMI, not modified at runtime. |
| **Destruction safety** | 100% safe. Nothing is lost when the instance is destroyed. |

This means:
- Any instance can handle any request — load balancer distributes freely
- Auto-scaling adds/removes instances without migration
- Failed instances are replaced, not recovered
- The instance is disposable by architecture, not by accident

## The REPL Loop for LLM Debugging

When the LLM EC2 is behaving unexpectedly, the osbot-utils SSH pattern from the Deploy Service applies directly. Boot a dev instance from the LLM AMI, SSH in, inspect interactively:

```python
# Boot from the LLM AMI for interactive debugging
ec2_instance = EC2_Instance()
ec2_instance.create_kwargs = dict(
    image_id      = LLM_STANDARD_AMI_ID,   # pre-baked AMI
    instance_type = 'g5.xlarge',
    key_name      = f'sgraph-dev-{run_id}',
    spot_instance = True
)
ec2_instance.create()
ec2_instance.wait_for_ssh()

with ec2_instance.ssh(path_key_pair, 'ubuntu') as ssh:
    ssh.ssh_execute().print_after_exec = True

    linux = ssh.ssh_linux()
    linux.running_processes()                  # is Ollama running?
    linux.disk_space()                         # do models fit?
    ssh.exec('systemctl status sg-llm')        # is FastAPI up?
    ssh.exec('ollama list')                    # which models are loaded?

    # Test a real inference from inside the instance
    def test_inference():
        import requests
        r = requests.post('http://localhost:8080/api/v1/chat/completions',
            json={"model": "llama3.2", "messages": [{"role": "user", "content": "Hello"}],
                  "stream": False})
        return r.json()
    ssh.ssh_python().execute_python__function(test_inference)

    # Port-forward Ollama to localhost for local testing
    tunnel = ssh.ssh_execute().start_port_forward(local_port=11435, remote_port=8080)
    # Now: curl http://localhost:11435/api/v1/models
    tunnel.terminate()

# Always clean up
try:
    ...
finally:
    ec2_instance.delete()
    # clean up key pair + security group
```

## Model Management at Runtime

The AMI has models pre-loaded. Additional models can be installed at runtime without rebuilding the AMI — but they will not persist across instance restarts (the instance is ephemeral and stateless by design). For persistent availability, add the model to the CI bake pipeline.

```
# Via admin API (from Deploy Service or directly)
curl -X POST http://llm-001.sgraph.ai/admin/models/pull \
     -H "Authorization: Bearer $SG_API_KEY" \
     -d '{"model": "phi3"}'

# Check loading status
curl http://llm-001.sgraph.ai/health

# Remove a model from this instance (not from the AMI)
curl -X DELETE http://llm-001.sgraph.ai/admin/models/phi3 \
     -H "Authorization: Bearer $SG_API_KEY"
```

## Cost Economics: Self-Hosted vs OpenRouter

The per-token cost approaches zero when self-hosted — you pay for compute time, not tokens.

| Scenario | OpenRouter cost | Self-hosted (g5.xlarge, ~$1.50/hr) |
|---|---|---|
| 10 req/hr, 10K tokens each | ~$0.30-1.00 | $1.50 — **OpenRouter cheaper** |
| 100 req/hr, 10K tokens each | ~$3-10 | $1.50 — **self-hosted cheaper** |
| 1,000 req/hr, 10K tokens each | ~$30-100 | $1.50 — **self-hosted dramatically cheaper** |
| Batch: 500 translation jobs | ~$5-20 | $1.50 for the hour — **self-hosted wins** |

**Crossover point:** self-hosted wins when the fixed hourly cost is less than the per-token cost at your request volume. For high-volume batch operations (translating 17 language files, processing 50 HTML pages, running evals), self-hosted is the right choice. For occasional one-off requests, OpenRouter is cheaper.

**The Accountant's routing rule:** route high-volume batch work to self-hosted; route occasional requests to OpenRouter. This is model routing at the infrastructure level, and is exactly what the boot-on-demand pattern enables — the self-hosted instance only costs money when it is actually being used.

**Spot instances reduce the self-hosted cost further.** A g5.xlarge spot instance is typically 60-70% cheaper than on-demand. At ~$0.50/hr spot, the crossover point drops to ~30 requests/hr.

| # | Criterion |
