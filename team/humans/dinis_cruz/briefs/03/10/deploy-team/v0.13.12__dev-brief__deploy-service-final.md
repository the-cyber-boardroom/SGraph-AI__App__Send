## What This Service Does

The Deploy Service is the infrastructure control plane. It provisions, configures, and manages cloud resources programmatically. Its core capability is SSH: using the `osbot-utils` SSH API to interact with Linux servers, execute commands sequentially, copy files, and build infrastructure step by step.

This is the service that **creates** other services (LLM Appliance, Send Ephemeral, future appliances). It is the builder, not the product.

Critically: **the Deploy Service is itself an appliance** — it follows the same boot-on-demand pattern it will later use to create other appliances. It runs ephemerally on EC2, shuts down when idle, and costs nothing when not in use. The only permanently running component is a lightweight Router Lambda that costs pennies per month.

## The Core Insight: One Primitive, Many Appliances

```
┌────────────────────────────────────────────────────────────────┐
│  EPHEMERAL DEPLOY EC2  — the one primitive                     │
│                                                                │
│  • Boots on demand, triggered by its Router Lambda             │
│  • Has SSH capability (osbot-utils SSH API)                    │
│  • Has boto3 + Type_Safe / Type_Twins AWS wrappers             │
│  • Exposes the Deploy Service FastAPI                          │
│  • Shuts itself down when idle                                 │
│  • All state stored in SG/Send vault (zero-knowledge)          │
└────────────────────────────────────────────────────────────────┘
```

The Deploy EC2 is the only entity in the system that can SSH into other machines. It is the only entity that can create and configure other EC2 instances and bake AMIs. Everything else either has no SSH capability (Lambda, the browser, CI pipelines) or is a product of the Deploy EC2's work.

## Router Lambda — The Stable Entry Point

The Router Lambda is the only permanently-running compute in this system. Its job is routing, not proxying. It answers one question: is the Deploy EC2 up? If yes, redirect. If no, start it, show a waiting page.

```
STABLE ENTRY POINT
deploy.sgraph.ai  →  API Gateway  →  Router Lambda
Never changes. This is what users, CI pipelines, and other
services point at. The EC2 behind it is ephemeral.
```

### Lambda IAM — Minimal Blast Radius

```
Lambda IAM role needs ONLY:
  ec2:StartInstances
  ec2:StopInstances
  ec2:DescribeInstances
  route53:ChangeResourceRecordSets   ← only needed for ephemeral DNS (Phase 2)

Nothing else. If this Lambda is compromised, damage is bounded.
Private keys, vault write keys, and SSH keys are never in Lambda.
```

## State Storage: SG/Send Vault (not SSM)

All instance state is stored in the SG/Send vault, not AWS SSM Parameter Store.

```
SSM PARAMETER STORE              SG/SEND VAULT
─────────────────────────────    ─────────────────────────────────────
AWS-only                         Portable — works locally, any cloud
Plaintext at rest (by default)   Zero-knowledge encrypted at rest
Visible in AWS console           Only readable by key holder
Another AWS dependency           Dogfoods our own product
Costs per API call               Already paid for
No structured audit trail        Full vault history = audit log
No local equivalent              Works identically on laptop
```

### Vault File Schema

```
vault://deploy/instances/deploy-001.json

{
  "status":       "running",      // stopped | starting | running | stopping
  "instance_id":  "i-0abc123",
  "ip":           "52.18.x.x",
  "dns":          "deploy-001.sgraph.ai",   // null until Phase 2
  "started_at":   "2026-03-10T14:00:00Z",
  "last_seen":    "2026-03-10T14:22:00Z",   // EC2 heartbeat writes this
  "idle_timeout": 900,                       // seconds
  "build":        "v0.13.12"
}

// One vault file per appliance:
vault://deploy/instances/llm-001.json
vault://deploy/instances/send-eph-001.json
```

The vault write key lives in the Lambda's environment variables and on the EC2 instance. Server-side, nobody can read these files — including us. Genuinely zero-knowledge infrastructure state.

## Auto-Shutdown — Two Independent Mechanisms

```
MECHANISM 1: Heartbeat watcher on the EC2 (primary)
─────────────────────────────────────────────────────
  Deploy Service FastAPI tracks last_request_at in memory.
  Background asyncio task checks every 60 seconds:

  if (now - last_request_at) > idle_timeout:
      vault.write(status="stopping")
      subprocess.run(["sudo", "shutdown", "-h", "now"])

  Fires reliably when the FastAPI process is healthy.


MECHANISM 2: EventBridge watchdog Lambda (belt)
────────────────────────────────────────────────
  EventBridge rule fires every 10 minutes.
  Lambda reads vault last_seen.
  If (now - last_seen) > idle_timeout + buffer AND instance running:
      ec2.stop_instances(instance_id)
      vault.write(status="stopped")

  Catches: hung processes, OOM kills, crashed FastAPI.

  ┌───────────────────────────────────────────────────────────┐
  │  Maximum zombie lifetime:                                 │
  │  idle_timeout (15 min) + watchdog window (10 min) = 25m  │
  │  After 25 minutes of no use, the instance is guaranteed  │
  │  to be stopped. Zero ongoing cost.                       │
  └───────────────────────────────────────────────────────────┘
```

## Ephemeral DNS

### Phase 1 — MVP: Raw IP Redirect

Lambda redirects to `http://52.18.x.x:8080`. Works immediately. IP changes on every start unless Elastic IP is assigned (optional, small cost).

### Phase 2 — Route53 Ephemeral DNS

```
ON EC2 START — Router Lambda writes Route53 A record:

  Name:  "deploy-001.sgraph.ai"
  Type:  "A"
  TTL:   60              ← short TTL, propagates in ~60s
  Value: "52.18.x.x"    ← current EC2 IP

ON EC2 STOP — watchdog Lambda removes the record.

NAMING CONVENTION:
  deploy.sgraph.ai          ← stable entry point (permanent → Lambda)
  deploy-001.sgraph.ai      ← ephemeral, points to running Deploy EC2
  llm.sgraph.ai             ← stable entry point for LLM appliance
  llm-001.sgraph.ai         ← ephemeral, points to running LLM EC2
  send-eph.sgraph.ai        ← stable entry point for Send Ephemeral
  send-eph-001.sgraph.ai    ← ephemeral, points to running Send container
```

## Deploy Service FastAPI (on the EC2)

This is the powerful API — the one with SSH and full AWS management capability. It only exists while the EC2 is running.

| Endpoint | Method | Purpose |
|---|---|---|
| `/status` | GET | All running stacks, instances, costs |
| `/stacks` | GET | List all managed stacks |
| `/stacks` | POST | Create a new appliance stack (async job, returns job_id) |
| `/stacks/{id}` | DELETE | Destroy a stack (async job) |
| `/stacks/{id}/ssh` | POST | Execute a command via SSH on a stack's instance |
| `/stacks/{id}/logs` | GET | Stream logs from a stack |
| `/stacks/{id}/health` | GET | Health check for a stack |
| `/jobs/{id}` | GET | Status and logs for an async deploy/destroy job |

**All endpoints protected by the existing FastAPI access token system.**

### Async Job Pattern

All stack operations are async — they return a `job_id` immediately (202 Accepted). The caller polls `/jobs/{id}` for progress. This is essential because deploy operations take 5-30 minutes and cannot be held open synchronously.

```
POST /stacks { type: "llm", instance: "g5.xlarge" }
◄── 202 { job_id: "job-abc" }    ← returns in milliseconds

GET /jobs/job-abc
◄── { status: "running", step: "3/7", log: ["EC2 launched", "SSH connected", "Installing Ollama..."] }

GET /jobs/job-abc
◄── { status: "complete", result: { endpoint: "https://llm-001.sgraph.ai" } }
```

## SSH-Based Infrastructure Building

The `osbot-utils` library provides the SSH API for Linux server interaction:

| Capability | What It Does |
|---|---|
| Execute commands | Run shell commands sequentially on remote servers |
| Copy files | Upload/download files to/from remote servers |
| Interactive debugging | Step through commands, inspect output at each stage |
| Sequential scripting | Build complex deployment scripts as Python code |

### Stack Lifecycle (Deploy EC2 executing via SSH)

```
Deploy EC2 receives: POST /stacks { type: "llm", instance: "g5.xlarge" }

  1. Create security group  (boto3)
  2. Create IAM role        (boto3, minimal permissions)
  3. Launch EC2 instance    (boto3)
  4. SSH in: install Ollama, configure FastAPI wrapper  (osbot-utils SSH)
  5. Create target group + NLB  (boto3)
  6. Write vault state file for new appliance
  7. Map DNS: llm-001.sgraph.ai → NLB IP  (Route53)
  8. Health check: verify LLM responds on /health
  9. Report: job complete, endpoint live

Deploy EC2 receives: DELETE /stacks/llm-001

  1. Remove Route53 DNS entry
  2. Delete vault state file for llm-001
  3. Delete NLB + target group
  4. Terminate EC2 instance
  5. Delete security group + IAM role
  6. Report: stack destroyed, zero resources remaining
```

## CLI and pip Install

```bash
pip install sg-deploy

sg-deploy status                              # Show all running infrastructure
sg-deploy up --type llm --instance g5.xlarge  # Create an LLM appliance
sg-deploy down --stack llm-001               # Destroy a stack
sg-deploy ssh deploy-001                     # SSH into the Deploy EC2
sg-deploy logs llm-001                       # Stream logs from an appliance
```

The CLI wraps Python classes that can also be imported directly:

```python
from sg_deploy import DeployService

deploy = DeployService(region='eu-west-1')
stack = deploy.create_llm_stack(instance_type='g5.xlarge')
print(stack.endpoint)  # https://llm-001.sgraph.ai
stack.destroy()
```

**Local mode:** On a developer laptop, the CLI calls the Deploy Service directly (no Lambda, no API Gateway). The web UI and stable DNS entry are only needed when the Deploy Service is running remotely.

## Execution Environments

There are two completely distinct operating modes, and five distinct execution environments. These must be kept separate — they have different credentials, different capabilities, and different costs.

---

### The Two Operating Modes

```
MODE A: BUILD                          MODE B: USE
─────────────────────────────────      ────────────────────────────────────
Create the Deploy EC2 AMI              Start a pre-baked Deploy EC2 AMI
Provision + configure from scratch     Use it (call FastAPI, SSH into targets)
SSH in, install, verify, bake AMI      Stop it when done
                                        
Only needs to happen:                  Happens constantly:
  - When the base AMI changes            - Every deploy operation
  - When dependencies change             - Every LLM/Send appliance creation
  - On new releases                      - Every developer session

Runs in:                               Runs in:
  CI with creds (production)             Laptop (dev), CI with creds, Lambda
  Laptop with creds (dev only)

Does NOT need to run:                  Does NOT need:
  - In Lambda (never)                    - SSH (Lambda can't SSH anyway)
  - In CI without creds                  - AMI baking capability
  - On every PR                          - Full AWS IAM privileges
```

### The REPL Development Loop (osbot-aws pattern)

The `osbot-utils` SSH API and `Type_Safe` classes are designed for an interactive development loop that is as fast in ENV1 (mock) as in ENV2 (real). This is the workflow that makes infrastructure code actually debuggable.

```
ENV1 — LAPTOP, NO CREDS  (Type__Twins mock)
════════════════════════════════════════════════════════════════════════

  Developer iterates on business logic without touching AWS.
  All AWS calls are intercepted by Type__Twins, which returns
  recorded or mock responses.

  from sg_deploy import DeployService
  from type_twins import mock_aws        # Type__Twins intercept layer

  with mock_aws():
      deploy = DeployService()
      job = deploy.create_appliance(type='llm', instance='g5.xlarge')
      assert job.status == 'queued'
      job.run()                          # executes against mock responses
      assert job.status == 'complete'
      assert job.result.endpoint == 'https://llm-001.sgraph.ai'

  ► No AWS account needed. No cost. Runs in 200ms.
  ► Type_Safe catches wrong field names at import time, not at runtime.
  ► Full logic coverage before a single real API call is made.
```

```
ENV2 — LAPTOP, WITH CREDS  (real AWS, interactive)
════════════════════════════════════════════════════════════════════════

## Type__Twins — The AWS Digital Twin Layer

### What It Is

Every AWS API call in the Deploy Service goes through a `Type__Twin` boundary. By default — when no AWS credentials are present — that boundary intercepts the call and handles it in-process: tracking state, returning realistic responses, never touching the network.

This is not a test mock bolted on from outside. It is the **default execution mode**. Real AWS is the special case, activated only by wiring in credentials.

```
┌─────────────────────────────────────────────────────────────────────────┐
│  DEPLOY SERVICE                                                         │
│                                                                         │
│  deploy.create_appliance(type='llm')                                    │
│       │                                                                 │
│       ▼                                                                 │
│  EC2_Instance.create()   ← Type_Safe class, all fields validated        │
│       │                                                                 │
│       ▼                                                                 │
│  ┌──────────────────────────────────────────────────────────────────┐  │
│  │  Type__Twin boundary                                             │  │
│  │                                                                  │  │
│  │  if credentials:   → forward to real AWS (or next Twin layer)   │  │
│  │  if no credentials: → handle in-process, track state            │  │
│  └──────────────────────────────────────────────────────────────────┘  │
│       │                                                                 │
│       ▼                                                                 │
│  Returns realistic response either way.                                 │
│  Calling code cannot tell the difference.                               │
└─────────────────────────────────────────────────────────────────────────┘
```

### The Fractal Chain

The Type__Twin boundary is chainable. Each layer does not know what is below it — real AWS, another Twin, or pure in-memory. This makes the chain composable to any depth:

```
CHAIN DEPTH 1 — Pure in-memory (unit tests, local dev, no network)
══════════════════════════════════════════════════════════════════════════

  Deploy Service
      └── Type__Twin (in-memory state)
               └── [no network, no AWS]

  ► Instant. Free. Works with no AWS account.
  ► State is Python dicts. Fully inspectable.
  ► The Deploy Service thinks it talked to AWS.


CHAIN DEPTH 2 — State-persisted Twin (integration tests, QA workflows)
══════════════════════════════════════════════════════════════════════════

  Deploy Service
      └── Type__Twin (state persisted to vault / disk)
               └── [no network, no AWS]

  ► State survives process restarts. Can replay sequences.
  ► Can diff "before" and "after" state for any operation.
  ► Enables: "show me what AWS would look like after this deployment script runs"
  ► Enables: authorisation workflows — inspect pending state before committing


CHAIN DEPTH 3 — Record-then-forward (policy + audit layer)
══════════════════════════════════════════════════════════════════════════

  Deploy Service
      └── Type__Twin A  (records all calls, enforces policy rules)
               └── Type__Twin B  (simulates realistic state transitions)
                        └── real AWS  (or another Twin — fractal)

  ► Twin A sees every API call before it reaches AWS.
  ► Twin A can: log it, check policy, approve/reject, rate-limit.
  ► Twin B gives Twin A realistic state to reason about.
  ► Real AWS is only reached after Twin A approves.


CHAIN DEPTH N — Arbitrary composition
══════════════════════════════════════════════════════════════════════════

  Service A (Deploy)                    Service B (LLM creation)
      └── Twin (thinks it has creds)        └── Twin (same)
               └── Twin (policy)                     └── Twin (state)
                        └── Twin (audit log)                   └── real AWS

  ► Each service is independently testable at its own boundary.
  ► The chain can be serialised and replayed.
  ► You can run the entire multi-service deployment in simulation
    before committing a single real API call.
```

### Why This Is the Default Mode (Not Just a Test Feature)

```
SCENARIO: Deploy Service on AWS Marketplace, first boot
────────────────────────────────────────────────────────

  New customer downloads the Deploy Service AMI.
  They have not yet wired in AWS credentials.

  Without Type__Twins as default:
  ✗ Every API call fails with credential errors
  ✗ Service is unusable until credentials are set up
  ✗ Customer cannot explore, learn, or plan

  With Type__Twins as default:
  ✓ Service boots and works immediately
  ✓ Customer can run deployment scripts against the Twin
  ✓ Customer sees exactly what AWS state their script would create
  ✓ Customer can inspect, modify, re-run — zero cost, zero risk
  ✓ When ready: wire in credentials → same scripts run against real AWS

  The Twin mode is the onboarding experience.
  Real AWS is the graduation.
```

### The State Inspection Superpower

The persisted Twin can answer questions no other tool can answer cheaply:

```
QUESTION: "What is my AWS state after I run this deployment script?"

  # Before running anything
  twin = Type__Twin(persist=True)
  twin.snapshot('before')

  # Run the full deployment script against the Twin
  deploy = DeployService(twin=twin)

    nlbs:              ['nlb-sim-llm', 'nlb-sim-send']

  ► No CloudFormation. No Terraform plan. Just Python.

| Concern | How It's Handled |
|---|---|
| **IAM credentials** | Deploy EC2 holds high-privilege IAM credentials. Never leave its environment. Never in Lambda. |
| **SSH keys** | SSH private keys stored in Deploy EC2's environment (or vault). Never exposed via API. |
| **Lambda IAM** | Minimal: start/stop EC2, update Route53. Nothing else. Bounded blast radius. |
| **Admin access** | Access token required for all Deploy FastAPI endpoints. Rate-limited. |
| **Created instances** | Each appliance gets its own minimal-privilege IAM role. No shared credentials. |
| **Network** | Created instances in private subnets. Only NLB is public. SSH only from Deploy EC2 IP. |
| **State** | Vault-encrypted. Server cannot read instance state. Zero-knowledge throughout. |
| **Destruction** | `sg-deploy down` removes ALL resources including IAM roles, security groups, DNS entries. Zero orphans. |

## Build Phases

### Phase 1 — Boot-on-demand with IP redirect (start here)

- [ ] Router Lambda reads vault, calls `StartInstances`, writes `STARTING`
- [ ] Lambda serves waiting page HTML with JS poller hitting `/status`
- [ ] Lambda `/status`: calls `DescribeInstances` + health ping to EC2
- [ ] Lambda redirects to `http://{ip}:8080` when health check passes
- [ ] EC2 FastAPI heartbeat: updates vault `last_seen` every 60s
- [ ] EC2 background task: `shutdown` after `idle_timeout` of no requests
- [ ] EventBridge watchdog Lambda: stops stale instances every 10 min

**Deliverable:** Deploy Appliance boots on demand, shuts down when idle. All state in vault. Zero always-on EC2 cost.

### Phase 2 — Ephemeral DNS

- [ ] Router Lambda calls Route53 `UPSERT` on EC2 start
- [ ] Watchdog Lambda calls Route53 `DELETE` on EC2 stop
- [ ] Redirect goes to `https://deploy-001.sgraph.ai` instead of raw IP
- [ ] TLS via ACM wildcard `*.sgraph.ai`

**Deliverable:** Clean URLs for every running appliance. Bookmarkable. TLS.

### Phase 3 — Deploy EC2 creates and manages other appliances

- [ ] Deploy FastAPI exposes `/provision` endpoint
- [ ] Deploy EC2 uses SSH to boot and configure target appliances
- [ ] Deploy EC2 creates target vault state files and Route53 entries
- [ ] LLM Appliance and Send Ephemeral created through this path

**Deliverable:** Full appliance lifecycle management. LLM brief becomes executable.
