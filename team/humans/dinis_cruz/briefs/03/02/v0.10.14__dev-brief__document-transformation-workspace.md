## Overview

Once the vault exists, the next step is a workspace where you can have LLM-powered conversations about vault documents — transform them, compare them side by side, generate new content from them, and save every transformation back to the vault with full version history.

This is a variation of the vault, using the same infrastructure, deployed at a different URL. It's the beginning of a data room that combines file storage with document analysis and transformation.

---

## Key Principle: All Client-Side

Like the vault itself, the document workspace has **no server-side LLM components**. The user connects their own LLM API keys. The LLM calls happen from the browser. The server never sees the document content or the LLM interactions. This is zero-knowledge all the way through.

```
┌─────────────────────────────────────────────────────┐
│  BROWSER (client-side)                               │
│                                                      │
│  ┌──────────────┐  ┌───────────────┐  ┌──────────┐ │
│  │ Vault Files   │  │ LLM Connection │  │ Workspace│ │
│  │ (sg-send.js)  │  │ (user's keys)  │  │ UI       │ │
│  └──────┬───────┘  └───────┬───────┘  └────┬─────┘ │
│         │                  │                │        │
│         ▼                  ▼                ▼        │
│  ┌─────────────────────────────────────────────────┐ │
│  │         Document Transformation Engine           │ │
│  │                                                  │ │
│  │  Load file from vault → Send to LLM with prompt  │ │
│  │  → Render result → Save transformation to vault  │ │
│  └─────────────────────────────────────────────────┘ │
│         │                  │                         │
└─────────┼──────────────────┼─────────────────────────┘
          ▼                  ▼
   SG/Send Backend      LLM APIs (direct)
   (encrypted storage)  (Anthropic, OpenAI,
                         Google, OpenRouter,
                         Ollama local)
```

### Supported Providers

| Provider | Connection | Notes |
|---|---|---|

## Core Feature: Side-by-Side Document Comparison

The primary UI pattern: source document on the left, transformation on the right.

```
┌────────────────────────┬────────────────────────┐
│  SOURCE                │  TRANSFORMATION         │
│                        │                         │
│  report-v2.md          │  report-v3.md (draft)   │
│                        │                         │
│  # Quarterly Report    │  # Quarterly Report     │
│                        │                         │
│  Revenue grew by 15%   │  Revenue grew by 15%    │
│  in Q4, driven by...   │  in Q4, primarily due   │
│                        │  to expansion in the    │
│                        │  EMEA region...         │
│                        │                         │
│                        │  [Diff highlighted]     │
│                        │                         │
├────────────────────────┴────────────────────────┤
│  💬 Chat with LLM                                │
│  ┌──────────────────────────────────────────────┐│
│  │ "Make the executive summary more concise     ││
│  │  and add a bullet-point recommendations      ││
│  │  section"                                    ││
│  └──────────────────────────────────────────────┘│
│  [Send to Claude ▾]                [Save to Vault]│
└──────────────────────────────────────────────────┘
```

### The Workflow

1. **Load a file** from the vault (left panel)
2. **Send to LLM** with a prompt: "Improve this document" / "Translate to Portuguese" / "Create an executive summary" / "Convert to slide deck outline"
3. **See the result** in the right panel
4. **Compare** — diff highlighting shows what changed
5. **Iterate** — refine the prompt, send again, compare again
6. **Save** — the transformation becomes a new file (or new version) in the vault

Every transformation is saved. The vault keeps the full history: original → v1 → v2 → v3. You can always go back.

## Document Types Supported

| Type | View | Transform | Notes |
|---|---|---|---|
| **Markdown** | Rendered + source | Full LLM transformation | Primary working format |
| **HTML** | Rendered in iframe | Full LLM transformation | For web content, landing pages |
| **PDF** | Rendered viewer | Send extracted text to LLM | Read + transform, not edit in place |
| **Plain text** | Raw display | Full LLM transformation | Transcripts, logs, notes |
| **JSON** | Pretty-printed | LLM can restructure | Translation files, configs |
| **Images** | Displayed | Send to multimodal LLM for analysis | Gemini particularly strong here |
| **PPTX** | Slide preview | LLM generates content, tool creates deck | Via document creation service |

## Prompt Library

A set of pre-built prompts to make common transformations easy:

### Document Prompts
| Prompt | What It Does |
|---|---|
| "Improve clarity" | Rewrites for readability without changing meaning |
| "Make more concise" | Reduces length while preserving key points |
| "Add executive summary" | Prepends a 3-paragraph summary |
| "Convert to bullet points" | Restructures prose into actionable bullets |
| "Identify gaps" | Analyses what's missing or unclear |

### Translation Prompts
| Prompt | What It Does |
|---|---|
| "Translate to {language}" | Full translation with cultural adaptation |
| "Compare translations" | Side-by-side source and translated version |
| "Review translation quality" | LLM analyses naturalness and cultural fit |

### Creation Prompts
| Prompt | What It Does |
|---|---|
| "Create slide deck outline" | From document → PPTX structure |
| "Create infographic brief" | From document → visual summary spec |
| "Create audio briefing script" | From document → NotebookLM-style script |
| "Extract key decisions" | From meeting transcript → decision log |

### Analysis Prompts
| Prompt | What It Does |
|---|---|
| "What are the risks?" | Risk analysis of a plan or proposal |
| "What's missing?" | Gap analysis |
| "Who needs to action what?" | Action item extraction |
| "Compare with {other document}" | Cross-document analysis |

Users can create and save their own prompts to the vault.

## Infographic Generation (Google Gemini)

The human specifically called out Gemini's visual capabilities. The workflow:

```
1. Select a PDF or document in the vault
2. Choose "Create infographic" prompt
3. Select Gemini as the model
4. Gemini analyses the document and generates a visual summary
5. Result displayed in the right panel
6. Save to vault as an image or HTML
```

**Why Gemini specifically:** Google's models (especially via the Generative AI API) have strong capabilities for visual content generation and document understanding. The multi-LLM architecture means we can route specific tasks to the best model.

The existing admin shell already has three capabilities that this workspace needs:

| Admin Shell Feature | Workspace Usage |
|---|---|
| **Communication panel** | Chat with LLMs about documents |
| **API viewer** | See all API calls (to SG/Send backend and to LLM providers) — transparency and debugging |

The workspace is built on the same web component architecture, with the same UI patterns, just pointed at vault files instead of admin functions.

## Deployment

| URL | What Lives There |
|---|---|
| `vault.sgraph.ai` | The core vault (file management, upload/download, sharing) |
| `workspace.sgraph.ai` | The document transformation workspace (vault + LLM integration) |

Both share the same `sg-send.js` library and the same vault infrastructure. The workspace is a superset of the vault — it has everything the vault has, plus the LLM panel and transformation UI.

**Or:** Both could be at `sgraph.ai/vault/` and `sgraph.ai/workspace/` as part of the main site.

## What This Becomes: The Data Room

The vault + workspace together are the foundation of the data room concept that has been discussed across multiple briefs:

```
Data Room = Vault (encrypted files)
           + Workspace (LLM-powered transformation)
           + Messaging (encrypted inbox, agent communication)
           + Email integration (send/receive from within)
           + Activity log (who accessed what, who changed what)
           + Permissions (per-folder, per-file access control)
```

Each component is built independently, on the same `sg-send.js` core, and combined into the data room experience. This brief covers the first two components.
