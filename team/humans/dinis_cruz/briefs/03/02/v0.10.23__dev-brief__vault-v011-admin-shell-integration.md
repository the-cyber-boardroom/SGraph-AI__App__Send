## Status: v0.1.0 Works

The first version of the vault is implemented and working. The workflow is correct — the core concept of a pure JavaScript encrypted vault on top of the SG/Send file infrastructure is validated. This is a significant milestone.

**What works in v0.1.0:**
- Vault Key entry and decryption
- File upload and download (encrypted)
- Multiple folder support (display)
- Basic vault UI

**What needs fixing:**
- Files created in subfolders don't get saved to the correct folder (bug)
- No drag-and-drop upload
- No file/folder renaming
- No file preview
- No inline editing
- Not using the admin shell (missing debug features, API viewer, system messages)
- Not leveraging the shared code from the main send.sgraph.ai

## Architecture Decision: Move to Admin Shell

**Lock the current vault as v0.1.0.** Build v0.1.1 from the admin shell as the base, not from the current vault code.

The admin shell has capabilities the vault needs:
- Debug features and API viewer
- System message display
- Communication panel
- Established component patterns

The main send.sgraph.ai has recently gained modules the vault should reuse:
- ZIP file support
- PDF rendering
- Image display
- Media file handling

### The Dependency Chain

```
send.sgraph.ai (main product)
  └── JS modules (zip, pdf, images, media, encryption)
        │
        ├── vault.sgraph.ai (imports these modules)
        └── admin (also uses these modules)
```

**Goal:** The vault should import and reuse the same JavaScript modules that the main SG/Send UI uses. One codebase for file handling, shared across all tools.

### Static File Publishing (DevOps Action)

The admin UI's JavaScript modules are not currently published as static files. Two options:

**Option A (quick):** Copy the necessary JS files into the vault's deployment. Works now, creates duplication.

**Option B (better):** DevOps publishes the shared modules as static files that any SG/Send tool can consume. Nothing confidential in these files — they're all open source.

**Recommendation:** Option B. Ask DevOps to create a static path for shared modules (e.g., `cdn.sgraph.ai/modules/` or `static.sgraph.ai/js/`). All tools import from there. This is the beginning of the unified code base.

**Note:** The admin UI itself may need refactoring before its shell can be cleanly extracted. The Architect should assess this before starting v0.1.1 — it may be faster to extract the shell patterns into a clean base than to inherit the admin's current structure.

## v0.1.1 Feature Specification

### 1. Admin Shell as Base

Replace the vault's current UI with the admin shell:
- Left panel: folder tree view
- Right panel: file display area
- Top banner: file properties (name, size, type, dates)
- Bottom/overlay: debug panel, API viewer, system messages

This is the "explorer" pattern — familiar, functional, proven.

```

├──────────────┬───────────────────────────────────────┤
│              │                                       │
```

### 2. Fix: File Creation in Correct Folder

**Bug:** When creating a file while inside a subfolder, the file gets created at the root level instead of in the current folder.

**Fix:** The upload/create handler must use the current folder path when updating the vault tree. This is likely a simple state management issue — the current folder path isn't being passed to the file creation function.

### 3. Drag-and-Drop Upload

The previous admin vault had drag-and-drop. Bring this back:

- Drop zone covers the right panel
- Visual indicator when dragging over (highlight, "drop files here" overlay)
- Multiple files supported
- Files encrypted and uploaded to current folder
- Vault tree updated after upload

### 4. File and Folder Renaming

- Click on file/folder name to rename (inline edit)
- Updates the vault tree (the server doesn't know about names — they're in the encrypted tree file)
- No server-side change needed — renaming is purely a vault-tree.json update

### 5. File Preview

Leverage the modules already built for send.sgraph.ai:

| File Type | Preview | Module Source |
|---|---|---|
| **PDF** | Rendered in viewer | send.sgraph.ai PDF module |
| **Images** (PNG, JPG, GIF, WebP) | Displayed inline | send.sgraph.ai image module |
| **Markdown** | Rendered HTML | New or existing markdown renderer |
| **HTML** | Rendered in sandboxed iframe | New |
| **Text / JSON** | Syntax-highlighted source | New |
| **ZIP** | Contents listing | send.sgraph.ai ZIP module |
| **Media** (audio, video) | Browser native player | send.sgraph.ai media module |
| **Other** | Download button + file properties | Fallback |

### 6. Inline Editing

For text-based files, allow direct editing in the vault:

| File Type | Edit Mode |
|---|---|
| **Text files** (.txt, .csv, .log) | Plain text editor |
| **Markdown** (.md) | Markdown editor with preview |
| **JSON** (.json) | JSON editor with formatting |
| **HTML** (.html) | Source editor |

**Save workflow:** Edit → encrypt → upload as new version → update vault tree. Previous version preserved in version history.

**Keep it simple:** Each editor should be a self-contained web component. No complex rich-text editing for v0.1.1 — just clean source editing with syntax highlighting.

### 7. Web Component Architecture

Every feature should be an individual web component with minimal coupling:

```
<vault-app>
  ├── <vault-tree-view>         ← Left panel folder tree
  ├── <vault-file-properties>   ← Top banner
  ├── <vault-file-preview>      ← Right panel display
  │   ├── <vault-pdf-viewer>
  │   ├── <vault-image-viewer>
  │   ├── <vault-markdown-viewer>
  │   ├── <vault-text-editor>
  │   └── <vault-media-player>
  ├── <vault-upload-dropzone>   ← Drag-and-drop overlay
  └── <vault-debug-panel>       ← Admin shell debug tools
```

**Blast radius:** Each component is isolated. A bug in the PDF viewer doesn't affect the markdown editor. A change to the tree view doesn't break file preview. This is critical for iteration speed.

## Implementation Plan

### Step 1: Architect Assessment (before coding)

The Architect needs to answer:
1. How cleanly can the admin shell be extracted as a reusable base?
2. Which modules from send.sgraph.ai can be directly imported vs. need adaptation?
3. What's the best strategy for shared module publishing (static files, CDN, npm package)?
4. Does the admin UI need refactoring first, or can we extract what we need cleanly?

**Output:** Architecture document with dependency map and recommended approach.

### Step 2: Shell + Tree View

- Set up vault v0.1.1 project using admin shell as base
- Implement folder tree view (left panel)
- Fix folder-creation bug
- Vault Key entry and decryption working

### Step 3: File Operations

- Drag-and-drop upload (to current folder)
- File/folder renaming
- File deletion
- File download

### Step 4: Preview + Edit

- Import preview modules from send.sgraph.ai (PDF, images, media, ZIP)
- Add markdown rendering
- Add text/markdown/JSON inline editing
- Save edits as new versions

### Step 5: Polish

- Multilingual (leverage existing i18n infrastructure)
- Keyboard shortcuts (navigate tree, rename, delete)
- Responsive layout
- Debug panel integration
