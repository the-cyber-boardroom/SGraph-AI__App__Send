## The Rethink

The previous vault code was over-engineered. This brief goes back to first principles: **if we already have an encrypted file storage service, why do we need a separate backend for a vault?**

A vault is just a collection of encrypted files with a shared access key and some metadata about structure. The server doesn't need to know it's a vault. From the server's perspective, it's just files — the same files the Send tool already handles. All the vault logic (structure, navigation, rendering, editing) lives in JavaScript on the client side.

---

## The Primitive

We already have:
- ✅ Encrypted file upload (AES-256-GCM, client-side encryption)
- ✅ Encrypted file download and decryption (client-side)
- ✅ Access tokens controlling who can read/write
- ✅ S3 storage backend
- ✅ FastAPI service managing file metadata

We need to add:
- ⬜ `sg-send.js` — a JavaScript library that knows how to talk to the SG/Send backend (save files, load files, manage encryption)
- ⬜ Path-based file organisation (not just flat IDs)
- ⬜ A vault metadata file (settings, structure, tree)
- ⬜ Client-side vault UI (multilingual from day one)

**That's it.** No new backend service. No new database. No new infrastructure. Just JavaScript on top of what exists.

---

## Architecture: The Server Knows Nothing

The design principle: **the server should have no idea it's hosting a vault.** From the server side, it's just encrypted files with IDs. All vault semantics are client-side.

```
CLIENT (vault.sgraph.ai)                    SERVER (send.sgraph.ai)
┌────────────────────────┐                  ┌─────────────────────┐
│                        │                  │                     │
│  Vault UI (JS)         │                  │  Encrypted file     │
│  ├─ sg-send.js         │───  upload  ───► │  storage service    │
│  ├─ vault-settings.json│◄── download ───  │  (existing FastAPI)  │
│  ├─ vault-tree.json    │                  │                     │
│  ├─ file renderer      │                  │  Has NO concept of: │
│  ├─ file editor        │                  │  - vaults           │
│  └─ encryption layer   │                  │  - folders          │
│                        │                  │  - file types       │
│  All vault logic HERE  │                  │  - relationships    │
│                        │                  │                     │
└────────────────────────┘                  └─────────────────────┘
```

### How It Works

1. **User enters the Vault Key** — a reasonably cryptographically strong string (not just the access token)
2. **The Vault Key derives the decryption key** — using a KDF (Key Derivation Function), the Vault Key generates the actual AES-256 key used to encrypt/decrypt all vault content
3. **Client fetches the vault settings file** — this is just another encrypted file stored on the server, decrypted client-side. It contains: vault name, description, creation date, owner
4. **Client fetches the vault tree file** — another encrypted file. Contains: the folder/file structure, file IDs, file metadata (names, types, sizes, dates)
5. **User navigates the tree** — all client-side rendering
6. **User opens a file** — client fetches the encrypted file by ID, decrypts with the derived key, renders in browser
7. **User adds/edits a file** — client encrypts, uploads to server. Updates the tree file (encrypted, re-uploaded)

**Sharing a vault = sharing the Vault Key.** That's it. Anyone with the key can access everything. Future: per-file or per-folder keys for granular access.

## sg-send.js — The Client Library

This is the missing piece that multiple features need. A JavaScript library that abstracts the SG/Send backend:

```javascript
// Core API
const sg = new SGSend({ endpoint: 'https://send.sgraph.ai', token: '...' });

// File operations
await sg.upload(fileData, { path: 'vault-001/settings.json' });
const file = await sg.download('vault-001/settings.json');
await sg.delete('vault-001/old-file.json');
const files = await sg.list('vault-001/');

// Encryption (client-side)
const encrypted = await sg.encrypt(plaintext, key);
const decrypted = await sg.decrypt(ciphertext, key);

// Vault-specific (built on top of core)
const vault = await sg.openVault(vaultKey);
const tree = await vault.getTree();
await vault.addFile('documents/report.pdf', fileData);
await vault.createFolder('documents/2026-03/');
```

This library is reusable across all SG/Send tools — the Send tool, the vault, the future data room, the document transformation workspace.

## Server-Side Enhancement: Path-Based File Organisation

### The Problem

Currently, files are stored with flat IDs. As the number of files grows, having everything in one "folder" on S3 becomes unwieldy. We also need a way to logically group files (all vault files together, all send files together).

### The Solution

Allow the file key to be a path, up to 5 levels deep, using Object IDs:

```
{object_id}/{object_id}                         ← 2 levels
{object_id}/{object_id}/{object_id}              ← 3 levels
{object_id}/{object_id}/{object_id}/{object_id}/{object_id}  ← 5 levels max
```

**Example:**
```
a1b2c3d4/                          ← vault container
a1b2c3d4/e5f6g7h8                  ← vault settings file
a1b2c3d4/i9j0k1l2                  ← vault tree file
a1b2c3d4/m3n4o5p6                  ← a file in the vault
a1b2c3d4/q7r8s9t0                  ← another file in the vault
```

**Rules:**
- Each segment is a valid Object ID (sanitised, no path traversal)
- Maximum depth: 5
- The server treats these as opaque keys — it doesn't interpret the structure
- Enables basic sharding: first-level Object ID distributes files across logical partitions

**This is a small change to the existing FastAPI service** — instead of validating a single ID, validate a path of 1-5 IDs separated by forward slashes.

## Vault Metadata Files

Two special files per vault, stored as regular encrypted files:

### vault-settings.json (encrypted)
```json
{
  "vault_name": "Project Alpha",
  "vault_id": "a1b2c3d4",
  "created": "2026-03-02T10:00:00Z",
  "owner": "user@example.com",
  "version": 1,
  "description": "Shared project vault for Alpha team"
}
```

### vault-tree.json (encrypted)
```json
{
  "version": 3,
  "updated": "2026-03-02T14:30:00Z",
  "tree": {
    "/": {
      "type": "folder",
      "children": {
        "documents": {
          "type": "folder",
          "children": {
            "report.pdf": {
              "type": "file",
              "file_id": "m3n4o5p6",
              "size": 245000,
              "mime": "application/pdf",
              "uploaded": "2026-03-02T11:00:00Z",
              "versions": [
                { "file_id": "m3n4o5p6", "date": "2026-03-02T11:00:00Z" },
                { "file_id": "x1y2z3w4", "date": "2026-03-01T09:00:00Z" }
              ]
            }
          }
        },
        "images": {
          "type": "folder",
          "children": {}
        }
      }
    }
  }
}
```

**Version history is built-in:** Each file entry can track previous versions (previous file IDs). Old versions aren't deleted — they're just older entries in the tree. The user can view any version.

## The Vault Key and Key Derivation

```
Vault Key (shared with users)
  │
  ▼
KDF (e.g., PBKDF2 or Argon2)
  │
  ▼
AES-256 Encryption Key
  │
  ├──► Encrypt/decrypt vault-settings.json
  ├──► Encrypt/decrypt vault-tree.json
  └──► Encrypt/decrypt all files in the vault
```

**For MVP:** A single symmetric key for the entire vault. Everyone with the Vault Key can access everything.

**Future:** Per-folder or per-file key derivation, allowing granular access control without changing the architecture.

### File Operations
- **Upload:** Drag-and-drop or file picker → encrypt client-side → upload → update tree
- **Download:** Fetch encrypted file → decrypt client-side → save locally
- **Preview:** PDF, images, markdown, HTML rendered inline in the browser
- **Delete:** Remove from tree (optionally delete from server)
- **Version history:** View previous versions of any file
- **Share:** Display the Vault Key for sharing
