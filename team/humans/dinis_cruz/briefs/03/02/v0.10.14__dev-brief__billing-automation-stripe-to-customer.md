## Overview

This brief covers the full automation pipeline from Stripe payment to customer receiving their access token. It introduces **agent-to-agent email communication** as the mechanism — specifically, the Accountant agent emailing the Sherpa agent, which becomes the foundation for all inter-agent messaging.

This supersedes and extends the v0.10.14 Payment Notification Pipeline brief from 1 March (which focused on notifying the human). This brief adds the automated provisioning flow.

---

## The Full Pipeline

```
Customer pays £5 on Stripe
  │
  ▼
Stripe webhook → FastAPI endpoint (with Stripe auth key verification)
  │
  ▼
ACCOUNTANT receives and processes the payment
  ├── Records the transaction
  ├── Creates access token (via admin API)
  ├── Updates accounting records
  ├── (Future: updates QuickBooks or accounting package)
  │
  ▼
ACCOUNTANT emails SHERPA (agent-to-agent email)
  │  From: accountant@sgraph.ai
  │  To: sherpa@sgraph.ai
  │  CC: human's personal email
  │  Content: transaction details + token details
  │
  ▼
SHERPA receives the email and triggers customer communication
  │
  ▼
SHERPA emails CUSTOMER
  │  From: sherpa@sgraph.ai
  │  To: {customer_email}
  │  Content: branded welcome message + access token + getting started guide
  │
  ▼
Customer uses their token on send.sgraph.ai
```

## Part 1: Accountant Email on Every Token Action

**Before wiring up Stripe**, first implement email notifications for every token management action that already exists in the admin console. This gives us the agent-to-agent email infrastructure that Stripe automation will use.

### Actions That Trigger Accountant → Sherpa Email

| Action | Email Subject | Content |
|---|---|---|
| **Create token** | "Token created: {token_name}" | Token name, credit amount, expiry date, created by whom |
| **Update token** | "Token updated: {token_name}" | What changed, old value → new value |
| **Revoke token** | "Token revoked: {token_name}" | Reason (if provided), who revoked |
| **Restore token** | "Token restored: {token_name}" | Who restored, new status |
| **Increase quota** | "Quota increased: {token_name}" | Old quota → new quota, reason |

**All emails from:** accountant@sgraph.ai  
**All emails to:** sherpa@sgraph.ai  
**All emails CC:** human's personal email  

### Implementation Options (Parallel Tracks)

**Track A: Add to existing admin console (fast, today)**

The admin console at send.sgraph.ai already has token management. Add email sending to each action handler:

```python
# On token creation
token = create_access_token(name, credits, expiry)
send_email(
    from_addr="accountant@sgraph.ai",
    to_addr="sherpa@sgraph.ai",
    cc=HUMAN_EMAIL,
    subject=f"Token created: {token.name}",
    body=format_token_email(token, action="created")
)
```

This is the fastest path. Works today with the existing admin infrastructure.

**Track B: Build as part of new billing service (parallel)**

Start building the separate billing service that will eventually handle all commercial operations. The same email logic lives here but with proper separation of concerns.

**Recommendation:** Do both. Track A gets us working emails today. Track B builds the right architecture for scale.

### Who Is the Accountant?

The Accountant is a new agent role whose inbox (accountant@sgraph.ai) is managed by the billing web service. The Accountant:

- Receives Stripe webhooks
- Manages transaction records
- Sends notifications on token lifecycle events
- (Future) Integrates with accounting software (QuickBooks)
- (Future) Manages invoicing and receipts

The Accountant does NOT interact with customers directly — that's Sherpa's job. The Accountant tells Sherpa what happened; Sherpa communicates with the customer.

## Part 2: Stripe Webhook Integration

### Stripe Webhook Endpoint

```
POST /admin/webhooks/stripe
Headers: Stripe-Signature (MUST verify)
```

On `checkout.session.completed`:

1. Verify Stripe signature using webhook signing secret
2. Extract: customer email, amount, payment ID, timestamp
3. Create access token (500 credits, 3-month expiry)
4. Accountant emails Sherpa with transaction + token details
5. Accountant emails customer email as CC (or Sherpa handles customer email separately)

### Stripe Test Cards

**Action for Developer:** Set up Stripe test mode for validating the full pipeline:

| Card Number | Behaviour |
|---|---|
| `4242 4242 4242 4242` | Successful payment |
| `4000 0000 0000 0002` | Card declined |
| `4000 0000 0000 3220` | 3D Secure authentication required |

Use test mode to:
- Validate the webhook fires correctly
- Validate token creation happens
- Validate emails are sent
- Validate the full end-to-end flow without real money

**Important:** Test transactions should also create real test tokens — this validates the entire pipeline. Use a naming convention like `test-{timestamp}` so they're easy to identify and clean up.

### From Test to Automated Provisioning

The test card flow is the same as the real flow — the only difference is which Stripe environment (test vs live) fires the webhook. Once the test flow works, switching to live is just pointing the webhook at the production endpoint.

## Part 3: Agent-to-Agent Email — Data Format

### Why Email Between Agents?

Email is the first inter-agent communication channel. It's:
- Already working (we have WorkMail)
- Human-readable (the CC to the human's email means full visibility)
- Auditable (every email is a record)
- Extensible (can add structured data as JSON attachments or in email body)

### Email Content: Structured Data

The Accountant's email to Sherpa should include structured data that Sherpa can parse and act on:

```
Subject: 💰 New payment: £5.00 from customer@example.com

--- Transaction Details ---
Payment ID: {stripe_payment_id}
Amount: £5.00
Customer: customer@example.com
Timestamp: 2026-03-02T10:30:00Z

--- Token Created ---
Token Name: {auto-generated-name}
Credits: 500
Expiry: 2026-06-02
Token Value: {token}

--- Action Required ---
Sherpa: Send welcome email to customer@example.com with token details.
```

**Future enhancement:** Include a JSON block in the email body or as an attachment. This allows automated processing while keeping the email human-readable:

```json
{
  "event": "payment.completed",
  "stripe_payment_id": "pi_xxx",
  "amount": 500,
  "currency": "gbp",
  "customer_email": "customer@example.com",
  "token": {
    "name": "...",
    "credits": 500,
    "expiry": "2026-06-02"
  },
  "action": "send_welcome_email"
}
```

This is the beginning of thinking about how agents communicate structured data via email — encrypted JSON payloads, classification headers, action types. The infrastructure built here scales to the full agentic email vision from the v0.7.4 briefs.

## Part 5: Accounting Software Integration

### The Problem

Transaction data currently lives only in Stripe and in our internal records. For proper accounting (especially as The Cyber Boardroom Ltd is the legal entity with existing revenue), transactions should be in an accounting package.

### Research Required

**Action for Developer/Accountant:** Research integration with:

| Package | Pros | Cons |
|---|---|---|
| **QuickBooks** | Industry standard, good API, human may already use it | Subscription cost, complexity |
| **Xero** | Popular in UK, good API, MTD-compatible | Subscription cost |
| **FreeAgent** | UK-focused, simple, MTD-compatible | Less feature-rich |

**MVP:** For now, Stripe's built-in reporting + the Accountant's email records are sufficient. Accounting package integration is a next-sprint item.

**Backup value:** An accounting package also provides a backup of transaction data outside of our AWS infrastructure. The human noted: "I don't trust any data in S3 because you're one delete away from losing everything." This is a valid concern addressed in Part 6.

## Part 6: S3 Hardening

**Separate from billing but flagged in this memo:**

The human expressed concern about S3 data durability. Two immediate actions:

### S3 Versioning

Enable versioning on all S3 buckets that contain:
- Encrypted file transfers
- Access token data
- Transaction records
- Website content

With versioning enabled, deleted objects are not permanently removed — they become "delete markers" that can be restored.

### S3 Object Lock / Delete Protection

Enable MFA Delete on critical buckets. This requires MFA authentication to permanently delete objects or change versioning state.

**Action for DevOps:** 
1. Audit all S3 buckets — which contain critical data?
2. Enable versioning on all critical buckets
3. Enable MFA Delete on the most critical buckets (encrypted transfers, token data)
4. Document the recovery procedure ("how to restore a deleted object")

This connects to the v0.7.6 multi-cloud brief's recommendation for Object Lock on at least one storage provider.
