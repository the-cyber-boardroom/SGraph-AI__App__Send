## Why Now

We're about to send real people to the site. Before that, we need automated tests that validate the entire user journey works — not just individual endpoints, but the full flow: visit site → find pricing → pay → receive token → send file → recipient downloads.

We also need synthetic users running these flows daily (eventually hourly) to catch regressions before real users hit them.

---

## The Three API Layers

The human described three distinct layers of test automation, each serving a different purpose:

### Layer 1: Bare Metal HTTP

Direct HTTP requests to the FastAPI endpoints. No browser, no JavaScript, no UI.

```python
# Direct class invocation (fastest, in-memory)
token = TokenService().create_token("test-user", credits=500)
assert token.credits == 500

# FastAPI test client (tests routing + serialization)
response = test_client.post("/admin/tokens/create", json={...})
assert response.status_code == 200

# Live HTTP request (tests the deployed service)
response = requests.post("https://send.sgraph.ai/admin/tokens/create", ...)
assert response.status_code == 200
```

**What it tests:** API contracts, business logic, authentication, error handling.  
**What it doesn't test:** UI rendering, JavaScript execution, browser-specific behaviour.  
**Speed:** ~100ms per test. Can run hundreds per minute.

### Layer 2: Browser Automation

Headless browser (Playwright, Puppeteer, or Selenium) driving the actual UI.

```python
# Navigate to site, find pricing, click buy
page.goto("https://sgraph.ai")
page.click("text=See Pricing")
page.click("text=Buy Credits")
# ... complete Stripe checkout
# ... enter token on send.sgraph.ai
# ... upload file
# ... verify download link works
```

**What it tests:** The full user experience — navigation, button visibility, form flows, JavaScript execution, file upload/download, Stripe checkout integration.  
**What it doesn't test:** API performance under load.  
**Speed:** ~5-30 seconds per flow. Suitable for daily/hourly runs, not continuous.  
**Maintainer:** QA team owns and maintains these tests.

### Layer 3: JavaScript DSL (In-Browser)

JavaScript functions that can be executed from the browser console to trigger specific flows. These are both test tools and demo tools.

```javascript
// From browser console on send.sgraph.ai:
SGSend.uploadFile("test-file.txt", "Hello World")
SGSend.getDownloadLink()
SGSend.verifyEncryption()

// From browser console on admin:
SGAdmin.createToken("demo-user", 500)
SGAdmin.checkTokenUsage("demo-user")
```

**What it tests:** Client-side functionality in a real browser environment.  
**What it enables:** Live demos, user support ("run SGSend.diagnostics() and tell me what you see"), and developer debugging.  
**Speed:** Real-time, interactive.

## User Story APIs

The human described a key insight: test APIs should be structured as **user stories**, not as technical endpoints. The test client should speak in domain language:

```python
# Instead of:
response = requests.post("/api/tokens", json={"name": "test", "credits": 500})
response = requests.post("/api/transfers/initiate", json={"file": "..."})

# The test API should offer:
user = SyntheticUser("test-march-02")
user.visit_pricing_page()
user.purchase_credits(amount=5)
user.enter_token()
user.upload_file("test-document.pdf")
link = user.get_share_link()
recipient = SyntheticUser("recipient-march-02")
recipient.open_link(link)
recipient.download_and_verify()
```

**These user story APIs become the acceptance criteria.** If the sequence above passes, the product works. If it fails at any step, we know exactly where the user journey breaks.

### FastAPI Endpoints for User Stories

Some of these user story steps should have FastAPI endpoints so they can be triggered programmatically:

```
POST /test/flows/purchase    → Simulates a purchase flow
POST /test/flows/send-file   → Simulates uploading and sharing a file
POST /test/flows/receive-file → Simulates receiving and downloading a file
POST /test/flows/full-cycle  → Runs the complete user journey
```

These endpoints are **test-only** (behind authentication, not exposed to production users). They return structured results:

```json
{
  "flow": "full-cycle",
  "steps": [
    {"step": "visit_pricing", "status": "pass", "time_ms": 120},
    {"step": "purchase_credits", "status": "pass", "time_ms": 3400},
    {"step": "enter_token", "status": "pass", "time_ms": 200},
    {"step": "upload_file", "status": "pass", "time_ms": 1800},
    {"step": "share_link", "status": "pass", "time_ms": 50},
    {"step": "recipient_download", "status": "pass", "time_ms": 900}
  ],
  "total_time_ms": 6470,
  "result": "pass"
}
```

## Synthetic Users (Bots)

### Daily Automated Runs

Synthetic users should run the full user journey at least daily:

```
06:00 UTC — Synthetic user runs full purchase + send + receive flow
12:00 UTC — Synthetic user runs full flow (catches daytime deployments)
18:00 UTC — Synthetic user runs full flow
```

Each run:
1. Creates a test Stripe payment (using test cards)
2. Verifies token is provisioned
3. Uploads a test file
4. Generates a share link
5. Downloads and verifies the file
6. Reports results to a monitoring dashboard or Slack/email alert

### Stripe Test Mode for Synthetic Users

Synthetic users use Stripe's test mode. Stripe test cards (`4242 4242 4242 4242`) trigger real webhooks in test mode, so the full pipeline is exercised without real money.

**Naming convention for synthetic tokens:** `synth-{date}-{run_number}` (e.g., `synth-2026-03-02-001`). Easy to identify and clean up.

### Synthetic User as Acceptance Criteria

**The maturity model:**

| Level | What Runs | Frequency | What It Validates |
|---|---|---|---|
| **L0** (now) | Manual testing by human | Ad hoc | "It works when I try it" |
| **L1** (this sprint) | HTTP API tests | On deploy | "The API endpoints work" |
| **L2** (next sprint) | Browser automation flows | Daily | "The full UI journey works" |
| **L3** (future) | Synthetic users + Stripe test mode | Hourly | "A real user would succeed right now" |
| **L4** (future) | Multi-language synthetic users | Daily | "The experience works in all 17 variants" |
