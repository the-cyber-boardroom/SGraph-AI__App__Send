# The Villager Phase: What Happens When You Actually Read the Code Your Agents Wrote

I have been writing about agentic development workflows for a few weeks now. How experts should use agents. How I manage 20+ AI agents. How primitives compound. How to work at two altitudes. Who should be reading the code.

This week I lived all of it at once. And I want to share what actually happened, because the theory is nice but the practice is messier, more interesting, and more instructive.

## The Setup

SG/Send, the encrypted file sharing platform I am building, has a web UI that evolved through what I call IFD (Iterative Flow Development). The Explorer phase produced 17 overlay versions (v0.2.0 through v0.2.17), each adding a feature on top of the previous one. Fast iteration, rapid experimentation, features shipping every day. Classic Explorer work.

But 17 layers of patches-on-patches had accumulated to the point where the code, while functional, was becoming hard for both humans and LLMs to reason about. A 7,221-line monolithic file. Thirty chained method overrides. The browser handled it fine. The agents were starting to struggle with it. And I could feel the technical debt building.

Time for the Villager phase. Time to read the code.

## Three Agents, Three Outcomes

I tried three different AI coding tools for this refactoring. The results were instructive.

**OpenAI Codex (on the web):** I gave up quickly. The experience was, to put it diplomatically, not ready for this kind of work. Moving on.

**Claude Code:** Got further. Made real progress on the upload workflow refactoring. But after 232 tool calls, the session timed out. Claude Code is excellent at focused tasks within a repo, executing one change at a time. It is less suited to large-scale refactoring that requires holding an entire codebase in context and making architectural decisions across dozens of files simultaneously. Two separate sessions hit the same wall.

**Claude Web (regular Claude, not Code):** This is where it got interesting. Claude Web does not have direct repo access. It cannot push to Git. It cannot run the code. But it can hold a much larger context, compress information better, and have an interactive architectural conversation. And crucially, it can produce patches.

## The Vault Workflow That Changed Everything

Here is the workflow that emerged:

1. I gave Claude Web the full codebase context (the v0.2.x overlay chain, the v0.3.0 target, the gap analysis)
2. Claude Web created an encrypted SG/Send vault using `sg-send-cli`
3. Claude Web produced `git format-patch` patches and pushed them to the vault
4. I pulled the patches from the vault on my laptop
5. I reviewed the diffs, applied with `git am`, tested in the browser
6. I pushed to GitHub
7. Repeat for the next patch

Eleven patches for the upload workflow. Five more for the download workflow and non-upload parity. A P0 bug found and fixed (friendly tokens were completely broken due to a hallucinated API call). The CI pipeline rewritten. All 27 external symlinks replaced with actual file copies. The full v0.2.x overlay chain audited feature by feature.

The agent never had write access to the Git repo. The agent communicated exclusively through an encrypted vault. I controlled what landed on the codebase. Every patch was reviewable in standard diff format. Nothing proprietary in the pipeline.

## Why This Matters: Patches as the Development Pattern

I think what we stumbled into here is going to become a major development pattern. Let me explain why.

Today, when you use AI coding tools, the agent typically has full access to your repo. It commits directly. It can push to any branch. It acts with your identity and your permissions. The only protection is convention: "please only modify files in this directory."

The patch-based vault workflow inverts this entirely. The agent produces patches. The patches travel through an encrypted vault. The human reviews and applies. The agent never touches the repo.

This gives you:

**Provenance.** Every code change is a patch file in a vault. The patch is signed. The vault has version history. You can trace every change back to which agent produced it, in response to which brief, reviewed by whom.

**Blast radius control.** A patch is a bounded diff. You can see exactly what it changes before you apply it. No surprise modifications to files you did not expect. No accidental commits to the wrong branch.

**Multi-agent review.** Before I applied a patch, I could give it to a different agent for review. A QA agent could check the patch for issues. An AppSec agent could scan it. All without any code change having been made to the actual codebase.

**Agent identity separation.** The agent does not need my Git credentials. It does not need SSH keys. It does not need repo write access. It needs the vault key and its own signing identity. That is it.

**Offline capability.** The patches are files. They can be emailed, stored, archived, compared, replayed. The vault can be a zip file transferred by any channel. The workflow does not depend on any specific CI/CD integration.

## Working at Two Altitudes (For Real This Time)

I wrote last week about working at two altitudes: strategic and hands-on. This refactoring was the hands-on altitude in practice.

I was not just reviewing agent output. I was reading every line of the v0.2.x overlay chain. I was comparing method signatures between the old and new code. I found the P0 bug (a hallucinated API call that broke all friendly token URLs) because I was reading the actual code, not just the test results. I was making architectural decisions about component boundaries, manifest structures, and deployment pipelines.

The agents did the heavy lifting of producing the patches. But I was the one who decided which patches to apply, in what order, and whether the architecture was right. The Explorer built it. The Villager (me, working with agents) cleaned it up. Both altitudes, same day.

## The Numbers

The v0.3.0 refactoring took the upload workflow from 7,221 lines in one monolithic file to 6,058 lines across 25 files (7 modules, 6 Shadow DOM components, support files). Every feature from v0.2.13 was preserved. The download workflow was audited and brought to parity. The codebase went from 27 external symlinks to zero. The CI pipeline was halved in size.

And the new UI is starting to look really good. The step indicator (Upload, Delivery, Share mode, Confirm, Encrypt and Upload, Done) gives a clean guided flow. The "Zero cookies, Zero tracking, We cannot read your files" line sits right on the upload page. The 10GB upload limit is displayed. The test files are there for anyone to try.

There are still bugs to fix. There always are after a refactoring of this scale. But the foundation is solid, and fixing bugs in a well-structured codebase is a fundamentally different experience from fixing bugs in a 17-layer overlay chain.

## What I Learned

**Claude Code and Claude Web are complementary, not competing.** Claude Code is for focused execution within a repo. Claude Web is for architectural reasoning and large-scale context. Knowing when to switch is a skill.

**The vault-as-patch-transport pattern is production-ready.** We used it for 16 patches across a major refactoring. It worked. The encrypted vault provides security, the patches provide reviewability, and the workflow provides control.

**Reading the code is not optional.** The P0 bug (hallucinated API call) would not have been caught by tests alone, because the tests would have been written against the same hallucinated API. It was caught because a human read the code and noticed that `FriendlyCrypto.resolve()` is not a real method. This is exactly the "Who Should Read the Code" argument in practice.

**The Explorer/Villager cycle works.** Seventeen Explorer overlays in rapid succession, then one Villager consolidation that cleaned everything up. The Explorer discovered what to build. The Villager built it properly. Neither phase was wasted work.

**IFD needs to evolve.** The experience revealed that method-level patching (v1) should give way to file-level replacement with manifests (v2). Smaller files, explicit dependency declarations, and a loader that reads from a manifest rather than hardcoded script tags. That architecture brief is written and ready for the next phase.

## What Comes Next

Fix the remaining bugs. Release v0.3.0 to production. Then the Explorer phase restarts on v0.3.x, building on a clean foundation. The gallery features, the vault integration, the LLM workflows: all the features from the last two weeks of briefs land on top of a codebase that is modular, well-tested, and architecturally sound.

The velocity should increase, not decrease. That is the promise of the Villager phase: you invest in quality now so that the next Explorer phase moves even faster. Every refactoring cycle raises the floor.

And every patch will travel through an encrypted vault, reviewable before it lands, signed by the agent that produced it. Not because we have to. Because it is a better way to work.
