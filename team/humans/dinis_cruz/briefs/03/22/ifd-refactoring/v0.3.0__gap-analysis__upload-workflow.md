# v0.3.0 Upload Workflow — Gap Analysis & Action Plan

**Date:** 2026-03-22
**Branch:** `ifd-v0.3.0__upload_refactoring`
**Base commit:** `a3751b13` (refactored send-upload)
**Analyst:** Dev agent
**Method:** Line-by-line comparison of v0.2.13 overlay chain (v0.2.0 → v0.2.17) against v0.3.0 modules

---

## 1. Scope

This analysis compares the **upload workflow** only. It covers:
- The v0.2.13 `en-gb/index.html` script chain (16 overlays + base)
- The v0.3.0 decomposed modules (7 shared modules + 6 Shadow DOM sub-components)

It does NOT cover: download page, welcome page, room/vault UIs, admin UI, or shared infrastructure (i18n, api-client, crypto).

### ❌ Features MISSING from v0.3.0

#### P1 — Core UX (visible to every user)

| # | Feature | Source | Impact | Where to implement |
|---|---------|--------|--------|-------------------|
| M1 | **Inline Next button** — persistent button at top-right of card, same row as step indicator. Advances with current defaults. Disabled during processing. | v0.2.7 + v0.2.8 | Every user sees this button on steps 2-4. Without it, users must click individual cards to advance. Power users lose the quick-advance path. | Orchestrator `_render()` — inject button after step indicator. Wire click to advance logic. |
| M2 | **Always show 3 delivery options** — File viewer / Gallery / Download for every upload. Smart defaults: single viewable → View, single other → Download, folder/multi → Gallery. | v0.2.14 + v0.2.15 | Currently `upload-file-utils.js` `detectDeliveryOptions()` conditionally shows 1-3 options. v0.2.15 overrides to always show all 3. | `upload-file-utils.js` — rewrite `detectDeliveryOptions()` to always return 3 options. Update `getRecommendedDelivery()` for smart defaults. |
| M3 | **Consistent delivery card naming and order** — "Folder view mode" / "Gallery mode" / "Download zip mode" (or "Download mode" for single files). Gallery always DEFAULT for folders. | v0.2.14 | Users see different card names and orders depending on file type. Confusing. | `upload-file-utils.js` — update option labels and ordering. |
| M4 | **Multi-file drop** — dropping multiple files treats them as a folder bundle (auto-zips). | v0.2.5 | Users who select multiple files get nothing. Common real-world use case. | `upload-step-select` + orchestrator — detect multi-file, call `UploadFolder.buildFolderScan()`. |
| M5 | **Multi-file input picker** — file browse dialog accepts multiple files. | v0.2.5 | Single-file only currently. | `upload-step-select` — add `multiple` attribute to file input. |
| M6 | **Multi-file paste** — Ctrl+V with multiple clipboard items. | v0.2.5 | Edge case but consistent with multi-file support. | `upload-step-select` — handle multiple paste items. |
| M7 | **Smart skip** — folders and multi-file skip file-ready state, go straight to delivery. | v0.2.5 | Folder users see an unnecessary intermediate step. | Orchestrator — after folder/multi-file set, call `_advanceToDelivery()` directly. |
| M8 | **Single file gallery** — when user selects Gallery delivery for a single file, wrap it in a zip with _gallery/ thumbnail. | v0.2.17 | Without this, single-file gallery URLs point to raw content with no thumbnails. Gallery view breaks. | `upload-engine.js` or orchestrator — before upload, if delivery=gallery and no folderScan, wrap file in zip via UploadFolder. |

## 4. Action Plan — Sequential Patches

Each patch is self-contained and testable. Apply in order.

| Patch | Features | Complexity | Files Changed |
|-------|----------|------------|---------------|
| `0002__delivery-options-always-3` | M2, M3 | Small | `upload-file-utils.js` |
| `0003__multi-file-and-smart-skip` | M4, M5, M6, M7 | Medium | `upload-step-select/*.js`, orchestrator |
| `0004__inline-next-button` | M1, M11 | Medium | orchestrator, `send-upload.css` |
| `0005__single-file-gallery` | M8 | Small | orchestrator or `upload-engine.js` |
| `0006__clickable-steps` | M9, M10 | Medium | `send-step-indicator.js`, orchestrator |
| `0007__drag-feedback` | M12 | Small | `upload-step-select/*.js`, `upload-step-select/*.css` |
| `0008__text-mode` | M13 | Large | New sub-component or modify select, orchestrator, engine |
| `0009__test-file-enhancements` | M14, M15, M16 | Medium | `send-test-files.js` (separate component) |

### Patch dependency graph
```
0002 ─┐
0003 ─┤
0004 ─┼─ (all independent of each other, apply in any order)
0005 ─┤
0006 ─┤
0007 ─┘
0008 ─── (depends on stable orchestrator — apply after 0002-0007)
0009 ─── (independent — separate component)
```

### Recommended implementation order

**Phase 1 — Match v0.2.13 feature parity (patches 0002-0007):**
These close every gap that a user switching from v0.2.13 to v0.3.0 would notice. After Phase 1, the upload workflow is functionally equivalent but with clean architecture.

**Phase 2 — Enhancements (patches 0008-0009):**
Text mode and test file improvements. Lower priority — the v0.2.13 production site has these but they're not critical path.
