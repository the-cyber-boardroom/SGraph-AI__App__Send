## What We Did

Decomposed the 7,221-line monolithic `send-upload.js` into a clean modular architecture and brought it to feature parity with the v0.2.13 production UI — all in a single session, delivered as 11 sequential patches via an encrypted SG/Send vault.

## Approach

### Phase 1: Understand the monolith

The upload UI had accumulated 16 IFD overlay files (v0.2.0 through v0.2.17), each applying prototype mutations on top of the previous. The consolidated monolith preserved all this code but was unreadable — 30+ chained overrides, duplicated functions across layers, and no clear ownership boundaries.

I started by reading the v0.2.13 `index.html` script loading order. This was the key insight: **let the HTML tell you what to read**, not the filesystem. The HTML showed exactly which overlays loaded and in what order, revealing the dependency chain without needing to read all the v0.2.x directories.

### Phase 2: Gap analysis (systematic, overlay-by-overlay)

Rather than guessing what was missing, I walked through each overlay's header comments (which describe their changes) and checked whether v0.3.0 implemented each feature. This produced a structured gap analysis with 16 specific missing features, each tagged with:
- Source overlay version
- User impact
- Exact file to implement in

This document (`v0.3.0__gap-analysis__upload-workflow.md`) became the work plan for every subsequent patch.

### Phase 3: Module extraction (patch 0001)

Before adding features, I refactored what existed. The 1,167-line orchestrator was split into 7 focused modules:

| Module | Responsibility |
|--------|---------------|
| `upload-constants.js` | Configuration — step labels, state mapping, carousel messages, size limits |
| `upload-crypto.js` | Friendly key generation, PBKDF2 derivation |
| `upload-file-utils.js` | File type detection, delivery option computation |
| `upload-thumbnails.js` | Image/PDF/markdown/video/audio thumbnail generators |
| `upload-folder.js` | Directory tree scanning, JSZip compression, gallery preview |
| `upload-engine.js` | The upload pipeline (read → encrypt → create → upload → complete) |
| `send-upload.js` | Orchestrator — just state machine, rendering, and event wiring |

Each module is a revealing-module-pattern IIFE that exposes a clean public API. The orchestrator calls into these modules but contains zero business logic itself.

### Phase 4: Feature parity (patches 0002–0010)

With the architecture clean, each missing feature had an obvious home. Patches were small and independent:

| Patch | Feature | Lines changed |
|-------|---------|--------------|
| 0002 | Always 3 delivery options + smart defaults | `upload-file-utils.js` rewrite |
| 0003 | Multi-file drop/paste/input + smart skip | orchestrator + select component |
| 0004 | Inline Next button with step indicator | orchestrator + CSS |
| 0005 | Single file gallery wrapping | 13 lines in orchestrator |
| 0006 | Clickable completed steps | step indicator + orchestrator |
| 0007 | Fix `[object Object]` path bug | 5 sub-components |
| 0008 | Drag feedback | select component CSS + JS |
| 0009 | Text mode toggle | select component + orchestrator |

| 0011 | Locale detection fix (en → en-gb) | done component |

### Phase 5: Bug fixes found during integration

Two bugs were discovered and fixed during the process:

1. **`[object Object]` in CSS paths** (0007) — All sub-components except delivery referenced `SendComponentPaths.base` or `.resolve()` or `.forComponent()`, none of which exist. The actual API is `.basePath`. This was a bug in the original sub-component extraction (pre-session).

2. **Locale detection in done step** (0011) — The done component had its own `_detectLocalePrefix()` that checked `parts[0].length === 2`, which matches `en` but not `en-gb`. Fixed to use the same regex as the engine module.

## How We Worked Together

### The vault-as-patch-transport pattern

Since I can't `git push` directly, we used an SG/Send encrypted vault (`2y0tyre74nw684v09hf9ucx0:4xw8kugl`) as a bidirectional patch channel:

1. I generate a `git diff` patch against the current branch HEAD
2. Push the `.patch` file to the vault via `sg-send-cli`
3. Dinis pulls from the vault, applies with `git apply`, commits, pushes to GitHub
4. I pull from GitHub, verify parity, and start the next patch

This worked well for 10 out of 11 patches. Patch 0003 failed to apply because it was generated against my local state (which had 0002 applied differently than Dinis's commit). The fix was to regenerate 0003 from Dinis's exact commit (`271ff2fc`).

**Lesson learned:** Always generate patches from the remote branch HEAD, not from local commits. I should have been pulling Dinis's commits before generating each subsequent patch.

### The branch discipline issue

I initially committed directly on `ifd-v0.3.0__upload_refactoring` instead of working on a separate branch. Dinis flagged this, and I created `claude/upload-refactor-session` as a local reference branch. In future sessions, I should:

1. Create a working branch at session start: `claude/{description}-{session-id}`
2. Work on that branch
3. Generate patches as diffs against the target branch
4. Never modify the target branch directly

### Sequential naming

Dinis requested sequentially-named patches (`0001__description.patch`) partway through. This was a good call — it made the apply order unambiguous and the vault browsable.
