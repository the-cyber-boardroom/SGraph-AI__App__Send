# v0.3.0 IFD Refactoring: Explorer-to-Villager in Practice

**version** v0.16.38
**date** 22 Mar 2026
**from** Human (project lead)
**to** All teams
**type** Debrief

---

## What Happened

The SG/Send UI has been evolving through IFD (Iterative Flow Development). The v0.2.x series reached v0.2.17 with 17 overlay patches, each adding a feature on top of the previous version. This is classic Explorer work: fast, independent features, rapid iteration, each version building on the last.

The v0.2.x overlays started showing the expected Explorer-phase symptoms: patches becoming less stable, bugs harder to isolate, the cumulative weight of 17 layers making each new change riskier. This is not a failure. This is the signal that it is time for the Villager phase.

The v0.3.0 refactoring is the Villager transition. Take all 17 overlay experiments, consolidate into a clean rewrite. The upload workflow went from a 7,221-line monolithic file to 7 modules and 6 Shadow DOM sub-components. All 16 features preserved. Code reduced dramatically (in some cases 7,000 lines to 2,000 lines). The opportunity was used to break things into better components, improve the architecture, and make the next Explorer phase easier.

## The SGit Patch Workflow in Action

This refactoring was a real-world test of the SGit patch workflow described in earlier briefs. Here is how it played out:

### Claude Code Hit Its Limits

Claude Code was used for the initial refactoring attempts. It struggled with the full picture: 17 overlay files, thousands of lines, complex interdependencies. Two sessions literally blew up (froze, could not continue). Claude Code is good at executing focused tasks within a repo. It is not good at consuming an entire codebase for a large-scale refactoring.

### Claude Web + SGit Patches Worked

Switched to Claude Web (normal Claude, not Code). Claude Web does not have direct repo access, only a read-only copy. But it does a better job at compressing information, understanding the full picture, and doing interactive, iterative work.

The workflow:

1. Claude Web received the v0.2.x code and the v0.3.0 target
2. Claude Web created a vault
3. Claude Web produced patches (10 sequential patches: 0001 through 0010+)
4. Claude Web pushed patches to the vault
5. Human pulled patches from the vault
6. Human applied patches to the Git repo using `git am`
7. Human reviewed, tested, committed

This is exactly the SGit-on-top-of-Git workflow from the earlier brief, working in production. The vault contained the patches. The agent never had write access to the repo. The human applied and reviewed. Provenance was maintained.

### Why This Worked Better

Claude Web succeeded where Claude Code failed because:

- **Better at compression.** Claude Web could hold the full context of 17 overlay files and the 3.0 target simultaneously. Claude Code tried to operate on the files directly and ran out of context.
- **Interactive iteration.** The human could have a conversation about the refactoring strategy: "what about this component?" "should we split this?" "compare the v0.2.7 gallery with the v0.3.0 gallery." Claude Code's agentic loop is less suited to this kind of architectural debate.
- **Patches as the output.** Instead of Claude Code committing directly (requiring repo write access), Claude Web produced patches that the human controlled. Cleaner separation.

## The Gap Analysis

After the upload workflow was refactored (patches 0001-0011, merged to dev), the Dev agent did a file-by-file comparison of v0.2.x against v0.3.0 for everything outside upload. Found 21 actions across 4 phases:

- **Phase 1 (routes):** Missing `/view/` and `/v/` routes, missing manual token entry
- **Phase 2 (packaging):** Missing i18n files, test files, root site files, welcome component
- **Phase 3 (download features):** Missing gallery/folder mode switcher, keyboard navigation, share/info tabs, print buttons, stale response guard
- **Phase 4 (architecture):** Decision needed on sg-layout CDN dependency (vendor vs keep vs remove)

The gap analysis and the session handoff brief (both from the Claude Code session) are included as-is in today's pack. They provide the detailed action plan for completing the v0.3.0 consolidation.

## What This Demonstrates

This refactoring is a textbook example of several principles working together:

**IFD methodology.** Version 0.2.x was the Explorer phase: fast iteration, overlay patches, feature accumulation. Version 0.3.0 is the Villager phase: consolidation, rewrite, quality. The version numbering makes the transition explicit and manageable.

**Explorer-to-Villager handover.** The Explorer produced 17 overlays with all the features. The Villager produced a clean rewrite preserving those features in better architecture. Neither phase is wasted work. The Explorer discovered what to build. The Villager built it properly.

**SGit patch workflow.** The vault-based patch workflow solved a real problem (Claude Code could not handle the refactoring, Claude Web could but needed a way to deliver code changes). This is exactly the use case SGit was designed for.

**The power and the limits of agents.** Claude Code is exceptional for focused implementation tasks. It is not yet good at large-scale refactoring that requires holding an entire codebase in context. Recognising when to switch tools (from Code to Web, from direct commits to patches) is part of the workflow.

## Next Steps

Complete the v0.3.0 non-upload parity:

1. Phase 1 + 2 (routes, packaging): mostly done (patches 0013-0014)
2. Phase 3 (download feature gaps): 4 remaining items (N13, N16, N17, N20)
3. Phase 4 (architecture): vendor sg-layout decision
4. Go live with v0.3.0 across the full Send UI

Then the Explorer phase restarts on v0.3.x with the next wave of features, built on a clean foundation.
