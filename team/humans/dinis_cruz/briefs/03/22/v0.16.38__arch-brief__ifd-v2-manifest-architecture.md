# IFD v2: Manifest-Driven Component Loading and File-Level Replacement

**version** v0.16.38
**date** 22 Mar 2026
**from** Human (project lead)
**to** Architect (lead), Developer
**type** Architecture brief

## Lessons from IFD v1

The v0.2.x series proved that IFD works. 17 overlay versions, tonnes of changes, tonnes of releases, and remarkably few side effects. The Web Components and the overlay architecture provided genuine isolation. The browser could handle the complexity. Changes in one overlay rarely broke another.

What is remarkable is how far the Explorer phase went before needing the Villager refactoring. The technical debt that accumulated was not about side effects or breakage. It was about complexity of the override chain: 17 overlays loading in sequence, patching methods on top of each other, becoming hard for LLMs (and humans) to follow. Claude Code struggled to refactor the 7,000-line monolith not because the code was broken, but because the layering was too deep to reason about in one pass.

The fixes themselves were straightforward once identified. The LLMs liked the IFD structure because it was easy to see where a fix belonged. The problem was navigating the layers to find where to look.

## What Changes in IFD v2

### Core Principle: Replace Files, Not Methods

IFD v1 patched methods. Each overlay loaded a JavaScript file that overrode specific methods on the component from the previous version. This avoided copying entire files, but it meant loading 3, 5, 10 versions of the same component in sequence, each patching the previous. The browser executed them fine. But the dependency chain was implicit and hard to trace.

IFD v2 replaces entire files. Each new version contains a manifest listing which files are replaced. The new file completely replaces the old one. No patching. No overrides. No chain of method mutations.

This changes the incentive structure: instead of patching one method in a large file, you are motivated to make files as small as possible. The smaller the file, the smaller the blast radius when it is replaced. This rewards the right behaviour: more components, more modules, smaller files, cleaner separation.

### Every Component Gets IFD Versioning

In v1, IFD versioning was applied at the page/app level (v0.2.0, v0.2.1, etc.). Individual components inside the page were not independently versioned.

In v2, every component has its own IFD version tree:

```
components/
  send-upload/
    v0/
      v0.1/
        v0.1.0/    (release.major.minor)
        v0.1.1/
      v0.2/
        v0.2.0/
  send-gallery/
    v0/
      v0.1/
        v0.1.0/
  send-browse/
    v0/
      v0.1/
        v0.1.0/
```

The numbering: v{release}.{major}.{minor}. Each component evolves independently. A new version of send-gallery does not require a new version of send-upload.

This creates an explosion of small, independently versioned components. That is the point. Each one is atomic. Each one can be tested independently. Each one can be replaced independently.

### The Manifest File

Every version of every component has a manifest file that lists:

- The files this version contains
- The dependencies (other components and their required versions)
- The files being replaced from the previous version (the diff from the previous manifest)

The manifest is the single source of truth for what gets loaded. When you push a new version (v0.3.1, v0.3.2, v0.3.3), you update the manifest with the changed files and dependencies. Nothing else.

```json
{
  "component": "send-upload",
  "version": "0.3.1",
  "files": [
    "send-upload.js",
    "upload-engine.js",
    "upload-crypto.js",
    "upload-thumbnails.js"
  ],
  "dependencies": {
    "send-header": "0.1.0",
    "send-footer": "0.1.0",
    "send-access-gate": "0.2.0",
    "sg-layout": "0.1.0"
  },
  "replaces_from_previous": [
    "upload-engine.js"
  ]
}
```

### The Loader

A small, stable loader component (versioned in core/common, rarely changes) that:

1. Reads the manifest for the current page/release
2. Resolves all dependencies (component manifests referencing other component manifests)
3. Loads the required JavaScript files in the correct order
4. Registers Web Components as they load

The index.html for any page becomes minimal:

```html
<script src="/_common/core/loader.js"></script>
<script>
  SGLoader.init({ manifest: './manifest.json' });
</script>
<body>
  <send-upload></send-upload>
</body>
```

No inline scripts. No hardcoded script tags. No CSS imports. No version-specific paths in the HTML. The manifest decides everything. Changing the manifest changes what loads.

### Version the Commons Too

In v1, the `_common/` directory was unversioned. Shared components (header, footer, locale, transparency) lived in one place and were symlinked. Changes to a shared component affected all versions simultaneously.

In v2, even common/core components are versioned. This means:

- A change to the header does not accidentally break an older page version
- Different pages can use different versions of the same shared component
- Rollback is trivial: point the manifest at the previous version

### The `latest` Folder

When deploying to production, the build script reads the manifest for the current release and produces a `latest/` folder that contains all the resolved files in a flat structure. This is what gets served.

The difference between development and production:

| | Development | Production |
|---|---|---|
| Structure | Full IFD tree (v0.1.0, v0.1.1, etc.) | Flat `latest/` folder |
| Loading | Manifest resolves from versioned paths | All files in one directory |
| Switching versions | Change the manifest | Rebuild `latest/` |
| Side-by-side comparison | Load two versions in adjacent windows | Not needed (use dev for this) |

### Local Storage for Version Selection

An interesting option: store the current manifest version in localStorage. The loader reads from localStorage to determine which manifest to load. This means:

- Version switching happens in the browser without changing any files
- A UI toggle could switch between v0.3.0 and v0.3.1 live
- Testing a new version is: change one localStorage value, reload
- No need to update index.html for each new release

This removes the current pain of having to modify the index page every time a new version ships.

## How This Solves the Current Problems

| Problem in v1 | How v2 Fixes It |
|---|---|
| Method patching chains become unreadable | File replacement, no chains |
| Large monolithic files | Incentive to make files small (smaller file = smaller replacement) |
| Index.html changes on every release | Manifest-driven loading, index stays static |
| Deployment copies files on top of each other | Manifest resolves to flat `latest/` folder cleanly |
| Components not independently versioned | Every component has its own IFD tree |
| Commons not versioned (accidental breakage) | Commons versioned like everything else |
| Web Component re-registration errors | Component registered once, dependencies loaded dynamically |
| Hard to test specific versions | localStorage toggle, side-by-side in dev |

## How This Connects to the Broader Architecture

**Smaller files reward the component pipeline.** The `<sg-theme>`, the gallery editor, the audio player, every tool from the component pipeline brief becomes a small, independently versioned, manifest-declared dependency. Drop it in by adding one line to the manifest.

**LLM-friendly.** Small files are easier for LLMs to read, understand, and modify. A patch that replaces one small file is easier to review than a patch that modifies three methods across a 2,000-line file. The SGit patch workflow produces cleaner, more atomic patches.

**The Explorer/Villager cycle gets faster.** Explorer adds a new file (small, focused). Villager reviews and refactors. The blast radius of each change is one file. The manifest tracks what changed. The version history is explicit.

## Implementation Plan

1. **Define the manifest schema** (JSON format, dependency resolution rules)
2. **Build the loader** (read manifest, resolve dependencies, load scripts, register components)
3. **Version the v0.3.0 components individually** (each gets its own IFD tree)
4. **Version the commons** (header, footer, locale, etc.)
5. **Convert index.html to manifest-driven** (remove hardcoded script tags)
6. **Add localStorage version selection** (loader reads version from localStorage)
7. **Update the build/deploy script** to produce `latest/` from manifest resolution
8. **Document the IFD v2 conventions** for all teams
