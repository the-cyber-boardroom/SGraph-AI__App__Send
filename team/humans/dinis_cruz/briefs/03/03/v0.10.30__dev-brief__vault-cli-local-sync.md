## Why This Is a Game-Changer

The vault currently exists only in the browser. That means:
- Files only exist on the server (encrypted) and in the browser (temporarily, decrypted)
- There's no local copy — if you want to work with a file, you download it, work on it, then re-upload
- There's no way to programmatically interact with the vault
- There's anxiety about file durability ("what happens to my files?")

A CLI that syncs a local folder to a vault changes the equation completely:

1. **Files exist locally** — in a folder on your machine, synced to the vault
2. **Backup for free** — if the local folder is in Dropbox/iCloud/Google Drive, you get backup automatically
3. **No browser required** — a user could use the vault entirely from the terminal
4. **Multi-device sync** — User A syncs from their machine, User B syncs from theirs, both have the same files
5. **Programmatic access** — scripts and automation can interact with the vault

**The killer use case:** I go to the vault, add a file via the CLI. Someone else, on a different computer, with the same Vault Key, syncs and sees that file. Send someone the key on WhatsApp, they sync locally, done.

## Architecture

```
Local Machine                          SG/Send Server
┌────────────────────┐                 ┌──────────────────┐
│                    │                 │                  │
│ ~/vaults/alpha/    │                 │  Encrypted files │
│ ├── report.pdf     │ ◄── sync ──► │  (S3 backend)    │
│ ├── notes.md       │                 │                  │
│ ├── images/        │                 │  No knowledge of │
│ │   └── logo.png   │                 │  vault structure │
│ └── .sg_vault      │                 │                  │
│     ├── config     │                 │                  │
│     ├── manifest   │                 │                  │
│     └── keyfile    │                 │                  │
│                    │                 │                  │
└────────────────────┘                 └──────────────────┘

        sg-vault CLI
        (Python / Typer)
        - Encrypts/decrypts locally
        - Uploads/downloads via API
        - Manages manifest for sync
        - Handles conflict detection
```

### The .sg_vault Directory

Each synced vault folder contains a hidden `.sg_vault` directory with:

| File | Purpose |
|---|---|
| `config` | Vault ID, server endpoint, sync settings |
| `manifest.json` | File inventory: paths, hashes, sizes, server file IDs, last sync timestamps |
| `keyfile` | The Vault Key (optional — can require manual entry instead of storing on disk) |

**Security note on keyfile:** Storing the Vault Key locally is the same security posture as having the key in your browser's local storage. The local machine already has access to the decrypted files. Use a `.sg_vault` extension so it's recognisable. If the sync folder is inside Dropbox/iCloud/Google Drive, the keyfile travels with it — which is fine, because the key is the access mechanism.

**File extension:** `.sg_vault` for the keyfile (e.g., `project-alpha.sg_vault`). This gives us a recognisable file type that could eventually have OS-level associations.

## CLI Design (Python / Typer)

Install from PyPI:
```bash
pip install sg-vault
```

### Core Commands

```bash
# Initialise a new vault (creates local folder + vault on server)
sg-vault init ~/vaults/my-project --name "My Project"

# Connect to an existing vault
sg-vault connect ~/vaults/my-project --key "vault-key-here"

# Sync: upload local changes + download remote changes
sg-vault sync ~/vaults/my-project

# List files in a vault (remote)
sg-vault ls ~/vaults/my-project
sg-vault ls ~/vaults/my-project/documents/

# Upload a specific file
sg-vault push ~/vaults/my-project/report.pdf

# Download a specific file
sg-vault pull ~/vaults/my-project/report.pdf

# Show vault status (what's changed locally, what's changed remotely)
sg-vault status ~/vaults/my-project

# Show vault info (name, file count, size, last sync)
sg-vault info ~/vaults/my-project
```

### Sync Workflow

```
sg-vault sync ~/vaults/my-project
  │
  ├── 1. Read local manifest (.sg_vault/manifest.json)
  ├── 2. Scan local files (compute hashes)
  ├── 3. Compare with manifest → identify local changes
  ├── 4. Fetch remote vault-tree.json (decrypt)
  ├── 5. Compare with manifest → identify remote changes
  ├── 6. Upload local-only / modified files (encrypt → upload)
  ├── 7. Download remote-only / modified files (download → decrypt)
  ├── 8. Handle conflicts (if any)
  ├── 9. Update vault-tree.json (encrypt → upload)
  └── 10. Update local manifest
```

### Conflict Handling (MVP)

For MVP, keep it simple:
- **No conflict:** File changed on one side only → sync normally
- **Conflict:** File changed on both sides → keep both, rename the local version with a `.conflict` suffix, flag it
- **Future:** Proper merge strategies, three-way diff for text files

## The Encryption Layer

The CLI uses the same encryption as the browser:
- **AES-256-GCM** for file encryption
- **KDF** to derive the encryption key from the Vault Key
- All encryption/decryption happens locally — never on the server

```python
from cryptography.hazmat.primitives.ciphers.aead import AESGCM
from cryptography.hazmat.primitives.kdf.pbkdf2 import PBKDF2HMAC

def derive_key(vault_key: str) -> bytes:
    """Same KDF as the browser implementation."""
    kdf = PBKDF2HMAC(algorithm=hashes.SHA256(), length=32, salt=SALT, iterations=100000)
    return kdf.derive(vault_key.encode())

def encrypt_file(path: Path, key: bytes) -> bytes:
    """Encrypt a file with AES-256-GCM."""
    aesgcm = AESGCM(key)
    nonce = os.urandom(12)
    data = path.read_bytes()
    ciphertext = aesgcm.encrypt(nonce, data, None)
    return nonce + ciphertext

def decrypt_file(ciphertext: bytes, key: bytes) -> bytes:
    """Decrypt a file with AES-256-GCM."""
    aesgcm = AESGCM(key)
    nonce = ciphertext[:12]
    return aesgcm.decrypt(nonce, ciphertext[12:], None)
```

**Critical:** The Python encryption must produce ciphertext that the JavaScript browser can decrypt, and vice versa. Same algorithm, same KDF parameters, same nonce handling. This is the interoperability contract.

## MVP Scope

### What to Build First

| Priority | Feature | Why |
|---|---|---|
| **1** | `sg-vault connect` — connect to existing vault with key | Must have — links local folder to remote vault |
| **2** | `sg-vault pull` — download all files from vault | Must have — get files locally |
| **3** | `sg-vault push` — upload a file to vault | Must have — add files from CLI |
| **4** | `sg-vault sync` — bidirectional sync | The killer feature |
| **5** | `sg-vault ls` — list remote files | Useful for verification |
| **6** | `sg-vault status` — show what's changed | Useful for understanding state before sync |
| **7** | `sg-vault init` — create new vault from CLI | Can wait — create vaults via web for now |

### What to Skip for MVP

- Watch mode (auto-sync on file change) — manual `sg-vault sync` is fine
- Conflict resolution beyond "keep both" — flag conflicts, let user resolve
- Partial sync (sync specific folders only) — sync everything
- Multi-vault management — one vault per local folder
- Version history via CLI — use web UI for this

## The Backup Insight

If the local sync folder lives inside an existing cloud sync service:

```
~/Dropbox/vaults/project-alpha/     ← Synced to Dropbox
~/Google Drive/vaults/client-data/  ← Synced to Google Drive
~/iCloud/vaults/personal/           ← Synced to iCloud
```

Then the user gets automatic backup of all vault files (decrypted locally, encrypted in cloud sync + encrypted on SG/Send). This solves the "what if I lose my files" anxiety immediately, with zero additional infrastructure.

**Even better:** The `.sg_vault/keyfile` travels with the synced folder. On a new machine, the user sets up their cloud sync, the vault folder appears, and `sg-vault sync` just works.

## The End-to-End Use Case (Acceptance Test)

This sequence must work end-to-end:

```bash
# User A: Connect to vault and add a file
$ sg-vault connect ~/vaults/alpha --key "my-vault-key"
Connected to vault "Project Alpha"

$ cp ~/Documents/report.pdf ~/vaults/alpha/
$ sg-vault sync ~/vaults/alpha
Uploading: report.pdf (245 KB)... done
Sync complete. 1 file uploaded, 0 downloaded.

# User A: Open browser, go to vault.sgraph.ai, enter same vault key
# → See report.pdf in the vault UI
# → Preview works

# User B: On a different machine, connect to same vault
$ sg-vault connect ~/vaults/alpha --key "my-vault-key"
Connected to vault "Project Alpha"

$ sg-vault sync ~/vaults/alpha
Downloading: report.pdf (245 KB)... done
Sync complete. 0 uploaded, 1 downloaded.

$ ls ~/vaults/alpha/
report.pdf
```

**That's it.** When this works, we have a proper distributed encrypted file system.
