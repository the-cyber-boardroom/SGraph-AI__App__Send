## Critical Constraint: Feature Toggles and Parallel Viewers

### The Problem

The current send.sgraph.ai serves one UI to all users. If we modify the viewer or add new modes, we risk breaking the experience for existing users who are actively sharing links. A link shared yesterday must still work tomorrow, exactly as the recipient expects.

### The Approach: Parallel Viewers with Feature Toggles

**New viewer modes are deployed alongside the current UI, not as replacements.** The current production viewer remains untouched. New viewers are developed, tested, and deployed at separate URL paths. Only when a new viewer is validated does it become available — and even then, the old path continues to work.

```
CURRENT (unchanged, always works):
send.sgraph.ai/#/download/{token}     <- Existing UI, never breaks

NEW VIEWERS (parallel, opt-in):
send.sgraph.ai/view/{token}           <- New default viewer (once ready)
send.sgraph.ai/gallery/{token}        <- Gallery viewer for photo sets
send.sgraph.ai/pdf/{token}            <- PDF reader viewer
send.sgraph.ai/video/{token}          <- Video player viewer
send.sgraph.ai/room/{token}           <- Data room viewer (folder/zip)
```

**The rules:**

1. **The current URL scheme never breaks.** Every link ever shared continues to work. The existing hash-based download path is permanent.
2. **New viewers are separate HTML entry points.** Each viewer is its own page/app that loads the same encrypted blob from the same API, but renders it differently. They share the decryption logic (from sg-send.js) but have their own UI.
3. **Feature toggles control which viewers are available.** A server-side or client-side toggle determines whether new viewer URLs are active. Toggle off means 404 or redirect to the current viewer. Toggle on means the new experience.
4. **The sender chooses the viewer at share time** (future). When sharing a link, the sender selects which viewer to use. The default is the current viewer until new viewers are validated.
5. **Auto-detection suggests a viewer** (future). Based on the file type (single PDF, folder of images, video, mixed folder), the system suggests the appropriate viewer. The sender can override.

### Implementation Architecture

```
send.sgraph.ai/
  index.html                    <- Current production app (NEVER TOUCH)
  view/
    index.html                  <- New default viewer entry point
  gallery/
    index.html                  <- Gallery viewer entry point
  pdf/
    index.html                  <- PDF viewer entry point
  video/
    index.html                  <- Video viewer entry point
  room/
    index.html                  <- Data room viewer entry point
  shared/
    sg-send.js                  <- Shared decryption + API library
    decrypt.js                  <- Decryption logic
    api.js                      <- Backend API calls
```

Each viewer imports from shared/ for the common functionality (fetch encrypted blob, decrypt, parse zip) and implements its own rendering. This is the web component blast-radius principle from the vault brief — a bug in the gallery viewer cannot affect the PDF viewer or the current production UI.

### Feature Toggle Mechanism

For MVP, use a simple approach: the toggle is which viewer paths exist in the deployed static files.

```
Deploy to dev:   all viewers active (for testing)
Deploy to prod:  only validated viewers active
```

In future, this becomes a proper feature flag system:

```json
{
  "viewers": {
    "gallery": { "enabled": true, "beta": false },
    "pdf":     { "enabled": true, "beta": false },
    "video":   { "enabled": true, "beta": true },
    "room":    { "enabled": false }
  }
}
```

Beta true means the viewer is accessible but not auto-suggested. Enabled false means the path returns 404 or redirects.

## 1. In-Browser File Editing (Zip Package Editing)

### The Problem

The human had to rename files locally and re-drop the folder multiple times to get the order and names right. This should happen in the UI.

### What is Needed

| Feature | Description | Notes |
|---|---|---|
| **Rename files** | Click filename to rename inline | Updates the zip metadata, not the server, until save |
| **Reorder files** | Drag and drop to reorder | Ordering stored as a manifest inside the zip |
| **Add files** | Drop additional files onto the open zip view | Appended to the existing set |
| **Remove files** | Delete a file from the package | Removed from the set before re-upload |
| **Unsaved changes indicator** | Show when edits exist that have not been uploaded | Prevent accidental navigation away |
| **Save** | Re-zip, re-encrypt, re-upload | Consumes a new credit (new upload) |

### Editor vs. Viewer

For now: **anyone with an access token is an editor.** This is fine at current scale. The distinction matters later (when we add view-only sharing), but not now.

### Decryption Key Options on Save

When saving an edited package:

| Option | What Happens | Use Case |
|---|---|---|
| **Keep same key** | New encrypted version, same decryption key. Previous link plus key still works, points to new version. | Updating materials for the same audience |
| **New key** | New encrypted version, new decryption key. Previous key no longer decrypts the new version. | Sharing with a new audience, or revoking previous access |

Both options create a new upload (new credit). The difference is whether the decryption key is regenerated.

## 2. Share Button and Link Management

### The Problem

After creating a data room, the share link and decryption key are not easily accessible. The URL disappears from the browser bar. The sender has to manually track what they shared.

### What is Needed

A **Share panel** accessible from the viewer/editor:

- Link display with Copy Link button
- Key display with Copy Key button
- Copy Link plus Key (combined, for pasting into a message)
- Email This button: opens the default email client via mailto: link with pre-populated subject and body containing the link and key
- QR Code generation for the link (useful for in-person sharing)

The Email This button uses mailto: — no server involvement, opens the user's native email app.

## 3. Viewer Modes — Parallel Development

As described in the critical constraint section above, each viewer is a **separate, parallel deployment** that shares the decryption/API layer but has its own rendering logic.

### 3a. Data Room / Folder Viewer (/room/{token})

The current zip preview, elevated to a purpose-built data room experience:

- Left sidebar: file tree with folders
- Right panel: file preview (PDF, image, markdown, video, text)
- Top banner: file metadata (name, size, type)
- Numbered file ordering respected
- Clean, professional typography
- Subtle SG/Send branding

This is the viewer for the investor review use case — a structured set of documents presented professionally.

### 3b. Gallery Viewer (/gallery/{token})

See the separate encrypted gallery sharing brief for full specification. Summary:

- Thumbnail grid layout
- Full-screen lightbox with swipe navigation
- Videos play inline
- Pinch to zoom on mobile
- Mobile-optimised (90% of screen is content)

### 3c. PDF Viewer (/pdf/{token})

For single PDF files:

- Full-page PDF reader
- Page navigation (previous/next, page number input, thumbnail sidebar)
- Zoom controls
- Search within PDF
- Download button
- Feels like a PDF reader, not a file transfer tool

### 3d. Video Viewer (/video/{token})

For single video files:

- Full-screen video player
- Play/pause/scrub controls
- Auto-rotate on mobile
- Subtitle support (if .srt or .vtt file is included alongside)
- Download button
- Feels like a video player, not a file transfer tool

### 3e. Default Viewer (/view/{token})

Auto-detects the file type and routes to the appropriate viewer:

| Content | Routes To |
|---|---|
| Single PDF | PDF viewer |
| Single video | Video viewer |
| Single image | Image viewer (full-screen display) |
| Zip with mostly images/videos | Gallery viewer |
| Zip with mixed files | Data room viewer |
| Other single file | Current download UI |

The auto-detection logic examines the decrypted file type (or zip contents) and redirects. If detection fails or is ambiguous, falls back to the current UI.

### Development Priority

| Viewer | Priority | Why |
|---|---|---|
| **Data Room (/room/)** | P1, first | Highest immediate business value (investor use case is live) |
| **Gallery (/gallery/)** | P1, parallel | Highest consumer potential, covered in separate brief |
| **Video (/video/)** | P2 | Quick win, mostly CSS/layout work on existing video playback |
| **PDF (/pdf/)** | P2 | Quick win, PDF.js already works, just needs full-screen layout |
| **Default auto-detect (/view/)** | P2 | Depends on other viewers existing first |

## 4. Access Analytics

### What Data We Have (Server-Side)

The server already knows some things about each file. We need to expose this data, not collect new data.

| Metric | How We Know | Implementation |
|---|---|---|
| **Download count** | Count API requests for this file ID | Counter on the download endpoint |
| **Download timestamps** | Timestamp of each request | Log or lightweight database entry per download |
| **Bandwidth consumed** | File size multiplied by download count | Calculated, not measured |
| **File size** | Stored at upload time | Already available |
| **Upload timestamp** | Stored at upload time | Already available |

### What We Do Not Have (Client-Side)

| Metric | Why We Do Not Have It | Could We Get It? |
|---|---|---|
| Whether decryption succeeded | Happens in the browser | Optional callback (privacy-sensitive, discuss first) |
| Which files in a zip were viewed | Happens in the browser | Optional callback |
| Time spent viewing | Happens in the browser | Optional callback |
| Geographic region | Available from request IP | Basic geo-lookup on download request |

**For now: expose server-side metrics only.** Download count plus timestamps plus bandwidth is already valuable. Client-side analytics are a privacy discussion for later.

### Where to Show Analytics

In the sender's view (when they open their own shared link with their token): show download count, last accessed time, total bandwidth, and a timeline of download events.

In the admin panel (for the human to see across all files).

**Start capturing this data now** even if the UI to display it comes later. Every download should be logged. Retroactively adding analytics is impossible — we can only show data from the point we start capturing it.

## 5. DevOps: Deployment Path Fix

### The Problem

Currently, pushing code to dev overwrites the production files because all environments deploy to the same S3 path (root /). The only reason this has not caused visible issues is CloudFront cache invalidation timing. With real users, this is unacceptable.

### The Fix

Deploy to environment-specific paths:

```
send.sgraph.ai/prod       <- Production (CloudFront serves this as default)
send.sgraph.ai/dev/       <- Development builds (accessible for testing)
send.sgraph.ai/main/      <- Main branch / staging
```

The path (/prod) serves production. Dev and main are accessible at their specific paths for testing. CloudFront routing rules handle this.

**Critical:** The new viewer paths (/view/, /gallery/, /pdf/, /video/, /room/) must also respect environment isolation:

```
Production:  send.sgraph.ai/prod/gallery/{token}
Dev:         send.sgraph.ai/dev/gallery/{token}
Main:        send.sgraph.ai/main/gallery/{token}
```

### CI/CD Pipeline Scoping

**The problem:** Any change to any file in the repo fires the full CI pipeline. Uploading vault code triggers a send.sgraph.ai rebuild.

**The fix:** Path-scoped GitHub Actions. Changes to send/ only rebuild send. Changes to vault/ only rebuild vault. Changes to shared/ trigger rebuilds for all consumers.

### Versioning

Every deployment must be traceable:

- Tag each deploy with the git SHA
- The deployed app should expose its version (e.g., a version.json at the root, or a meta tag)
- The admin panel should show the deployed version for each environment

## 6. Shared Module Strategy: sg-send.js

All viewers share the same core functionality. This must be a single, shared library, not duplicated code.

```
shared/sg-send.js
  fetch(token)           -> Download encrypted blob from API
  decrypt(blob, key)     -> AES-256-GCM decryption
  parseZip(decrypted)    -> Extract zip contents (file list, metadata)
  extractFile(zip, path) -> Extract a single file from the zip
  generateThumbnail(img) -> Create thumbnail from image data
  detectContentType(zip) -> Analyse zip contents, suggest viewer
  analytics.logDownload()-> Record download event (server callback)
```

Each viewer imports sg-send.js and builds its UI on top. The library handles the "get the files, decrypt them, make them available" layer. The viewer handles the "show them beautifully" layer.

**This is the same sg-send.js from the vault brief.** One library, used by: send UI, vault UI, gallery viewer, PDF viewer, video viewer, data room viewer, and eventually the CLI (Python equivalent).
