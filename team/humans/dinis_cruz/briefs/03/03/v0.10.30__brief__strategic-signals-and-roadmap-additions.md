## 1. Translation Quality — External Validation

### What Happened

Real customers confirmed that the translations work in **Argentinian Spanish** and **Croatian**. No one said "this is weird" or "WTF does this mean?" — the requests were for typo-level corrections, not comprehension failures.

### Why This Matters

This is the signal we were waiting for. The fear with machine translation has always been the allergic reaction: users hitting a page in their language and immediately bouncing because the translation is obviously broken, robotic, or nonsensical. That reaction did not happen. Two languages, two completely different language families, two real users — and neither flagged comprehension issues.

**What this does NOT mean:** The translations are perfect. The human knows from reading the Portuguese and Brazilian Portuguese variants that there are plenty of "little things" that need improvement — phrasing that's technically correct but not how a native speaker would say it, awkward constructions, tone mismatches. The Translator role and human reviewer workflow remain essential.

**What this DOES mean:** The LLM translation baseline is good enough that users can navigate, understand, and use the product in their language without friction. The 10% human refinement that the Translator role is designed to deliver is the difference between "good enough" and "feels native" — but "good enough" is already dramatically better than what machine translation produced even two years ago.

### Implication for the Translation Strategy

The three-zone translation strategy from the v0.10.1 brief is validated:
- **Zone 1 (Tools/Product UI):** Full multilingual — confirmed working in practice
- **Zone 2 (Commercial Website):** Full multilingual — confirmed working in practice
- **Zone 3 (Documentation):** English-only for now — correct decision, invest translation effort in Zones 1-2

The Translator community engagement plan (GitHub PRs for translation improvements) becomes more urgent — we now have real users in these languages who could contribute corrections.

## 2. Billing Primitives — Next Set of Billable Units

### The Problem

Currently, the only billable unit is **files uploaded** (1 credit = 1 file upload). This was the right MVP, but it creates perverse incentives and leaves revenue on the table as usage patterns diversify.

### What Needs to Be Billable

| Billable Unit | Why | Priority |
|---|---|---|
| **File upload** (existing) | Core action — sender pays to send | ✅ Live |
| **Bandwidth (upload)** | Large files consume more resources. A 300MB upload costs us more than a 1KB upload. Without bandwidth billing, we subsidise heavy users. | P1 |
| **Bandwidth (download)** | Every download costs egress bandwidth. A popular file downloaded 1,000 times costs us significantly — we don't want to become someone else's CDN. | P1 |
| **Number of downloads** | Related to bandwidth but also a usage signal. A file downloaded once vs. 1,000 times represents different value and cost. | P1 |
| **Storage duration (days)** | Files sitting on our servers indefinitely cost money. We need a retention model: files stored for X days included, then per-day charges or auto-deletion. | P1 |
| **Vault creation** | Once vaults are live, creating a vault is a billable event (or a subscription). | P2 |
| **Vault storage** | Vault files are persistent by design (unlike one-off transfers). Storage billing is essential. | P2 |
| **Vault usage** | Active vault operations (uploads, edits, shares) — consumption-based. | P2 |
| **Scanning services** | Per-scan charges when the scanning chain is live (CDR, AV, etc.) | P3 |

### The CDN Problem

Without download billing, someone could upload a large file, share the link publicly, and use SG/Send as a free CDN. We pay the bandwidth; they get free distribution. This needs to be addressed before it becomes a problem.

**Options:**
- Download count limits per file (e.g., 100 downloads included, then charges apply or file is rate-limited)
- Bandwidth caps per token (e.g., 10GB/month included)
- Auto-expiry on files after X days unless the owner pays for extended storage
- Rate limiting on popular files

### Architecture Implication

The billing service (from the v0.10.1 billing automation brief) needs to track more than just upload events. Every API call needs metering:

```
Event: file_uploaded    → increment credit counter (existing)
Event: file_downloaded  → increment download counter (new)
Event: bytes_uploaded   → increment bandwidth counter (new)
Event: bytes_downloaded → increment bandwidth counter (new)
Event: file_stored_day  → increment storage counter (daily cron, new)
Event: vault_created    → increment vault counter (new)
```

The Accountant agent needs visibility into all of these metrics, and the customer-facing dashboard needs to show usage clearly.

## 3. Video Sharing — Double Down on This

### Why Video Is a Killer Use Case

Video sits at the perfect intersection of SG/Send's strengths:

| Factor | Why It Favours SG/Send |
|---|---|
| **Large file** | Videos are big (50MB-2GB+). Most sharing tools choke on this. |
| **Pain to share** | WhatsApp compresses. Email bounces. Cloud drives require download + find viewer. |
| **Encryption adds real value** | Private video (medical, legal, personal, corporate) is genuinely sensitive content. |
| **In-browser playback is the differentiator** | Click link → video plays. Zero friction. No download, no app, no viewer. |
| **Mobile-friendly** | Opening a video link on a phone and having it just play is magic. |

### The "Share Video" UX — A New View Mode

The current SG/Send UX treats every file the same: show the encryption details, show the file metadata, show the download button. This is correct for the general case, but for specific file types, we should create **purpose-built views** where the file IS the UI.

**The principle: 90% of the screen is the content. Encryption is invisible.**

#### Share Video View

```
┌──────────────────────────────────────────────────┐
│ ▶                                                │
│                                                  │
│                                                  │
│              [VIDEO PLAYER]                      │
│              Full screen                         │
│              Auto-rotates on mobile              │
│              Play/pause/scrub controls           │
│                                                  │
│                                                  │
│                                                  │
├──────────────────────────────────────────────────┤
│ 🔐 Encrypted with SG/Send        [Download] [↗] │
└──────────────────────────────────────────────────┘
```

**What changes from the current UX:**
- The left panel (decryption details) is hidden by default
- The video takes 90%+ of the screen
- Mobile rotation works properly (landscape for landscape video)
- Playback controls are prominent
- The encryption badge is subtle — "this is encrypted" is a trust signal, not the main feature
- A "Show details" link expands the technical details for those who want them

#### The Simplified Key Concept

For video sharing (and other "share with friends" use cases), the current encryption key is too long and complex. The human insight: **generate the full cryptographic key from a simpler, human-friendly key** that's easy to say aloud or type.

```
Full key:     a7f2b9c4d8e1f3a5b7c9d2e4f6a8b0c3d5e7f9a1b3c5d7e9f1a3b5c7d9e1f3
Friendly key: purple-sunset-442

User shares:  "Hey, I sent you a video on SG/Send. The key is purple-sunset-442"
```

The friendly key is derived into the full key via KDF. The cryptographic strength comes from the derivation, not from the key length. This is the same model as BIP-39 mnemonic phrases in cryptocurrency wallets.

**This is critical for the WhatsApp/voice use case:** "I'm sending you a video. The code is purple-sunset-442." That's it. The recipient clicks the link, enters the code, video plays.

### Other Purpose-Built Views

The same principle applies to other file types:

| View | UX Focus | What Changes |
|---|---|---|
| **Share Video** | Full-screen video player, auto-rotate on mobile | Encryption details hidden, video dominates |
| **Share PDF** | Full-page PDF reader, page navigation | Feels like a PDF viewer, not a file transfer tool |
| **Share Image** | Full-screen image display, pinch-to-zoom on mobile | Gallery-like experience |
| **Share Markdown** | Rendered document view, clean typography | Feels like a blog post or document viewer |
| **Share Folder/ZIP** | File browser, selective preview | Feels like a file manager |

**The pattern:** The encryption is the reason the file is safe. But the UX should be about the file, not the encryption. The encryption is a feature of the platform, not a feature of the experience.

### Connection to DRU (Daily Recurring Usage)

**DRU is the North Star metric.** Every feature and UX improvement should be evaluated against: "Does this lead to more files being shared and viewed?"

Purpose-built views directly drive DRU because:
- Better viewing experience → recipients are more impressed → they ask "how did you send this?" → new users
- Lower friction → senders use SG/Send for more use cases (not just "when I need encryption" but "when I need to share a large file easily")
- Mobile-optimised views → files shared via WhatsApp/text actually work well → more casual sharing
- Each new viewer opens a new use case → new market segment

## 4. Public Demo Examples on the Website

### The Problem

The website is good at "buy credits" but not at "here's how it works and where it adds value." A visitor can understand what SG/Send does in theory but can't experience it without buying credits first. That's backwards — they need to experience the value before they'll pay.

### The Solution: Public Demo Links

Create a set of **public example transfers** with leaked access tokens embedded on the website. Visitors can experience the full SG/Send workflow — clicking a link, seeing the file, using the viewer — without any purchase.

**Implementation:** Create access tokens specifically for demo content. Embed them as links and iframes on the website. The demo content is carefully curated to showcase the platform's strengths.

## 5. Large Files as a Core Message

### The Positioning Shift

"Use us to easily share large files" should be a primary marketing message, not a technical capability buried in a features list.

**Why this works:**

| Factor | Details |
|---|---|
| **Universal pain point** | Everyone has tried to send a large file and hit a wall (email limits, WhatsApp compression, slow cloud uploads) |
| **Competitive advantage** | SG/Send handles 300MB+ confirmed (and likely much more). In-browser viewing means no download required. |
| **Security is the bonus, not the pitch** | "Send large files easily" gets attention. "...and they're encrypted end-to-end" is the bonus that closes the deal. |
| **Measurable** | We can test and publicise the largest file successfully transferred. "We just handled a 1GB file" is a headline. |

### The Messaging

**Current:** "Your files, your keys, your privacy" — encryption-first messaging.

**Addition:** "Share files of any size. Encrypted, instant, no limits." — utility-first messaging.

Both messages are true. The question is which one leads the conversation for which audience:
- **Security-conscious users:** Lead with encryption, follow with utility
- **General users:** Lead with utility ("share large files easily"), follow with security as the differentiator

### Waiting for the 1GB+ Test

When someone sends a file larger than 1GB, that's a headline. Track the largest file transferred and be ready to publicise it. If no customer does it organically, we should test it ourselves and publish the result.

## 6. New File Viewers = New Markets

### The Insight

Every new language we add opens a new geographic market. **Every new file viewer we add opens a new use-case market.** The pattern is identical:

| Dimension | What We Add | Market Opened |
|---|---|---|
| **Language** | Portuguese translation | Portuguese-speaking users |
| **File viewer** | PDF viewer | "Share PDFs easily" users |
| **File viewer** | Video player | "Share videos easily" users |
| **File viewer** | ZIP browser | "Share folders easily" users |
| **File viewer** | DICOM viewer | Medical imaging sharing |
| **File viewer** | CAD viewer | Engineering file sharing |
| **File viewer** | 3D model viewer | Design/architecture sharing |
| **File viewer** | Code viewer (syntax highlighting) | Developer file sharing |
| **File viewer** | Spreadsheet viewer | Business data sharing |

### Why This Works

Today, sharing a DICOM medical image means: the sender exports it, finds a way to send it (usually a USB drive or a special medical file transfer system), the recipient needs DICOM viewing software installed. With an SG/Send DICOM viewer: upload → share link → view in browser. Encrypted. No special software.

The same applies to any file format that currently requires special software to view: CAD files, 3D models, scientific data formats, audio production files, legal document formats. Each viewer that can be implemented via a JavaScript parser running in the browser eliminates a pain point and opens a niche market.

### Prioritisation

| Viewer | JS Parser Available? | Market Size | Implementation Effort | Priority |
|---|---|---|---|---|
| **PDF** | ✅ pdf.js (Mozilla) | Massive | ✅ Done | ✅ Shipped |
| **Video/Audio** | ✅ Native browser | Massive | ✅ Done | ✅ Shipped |
| **ZIP** | ✅ JSZip | Large | ✅ Done | ✅ Shipped |
| **Images** | ✅ Native browser | Large | ✅ Done | ✅ Shipped |
| **Markdown** | ✅ marked.js, etc. | Medium (developers) | Small | P1 |
| **Code files** | ✅ Prism.js, highlight.js | Medium (developers) | Small | P1 |
| **CSV/Spreadsheet** | ✅ SheetJS, PapaParse | Large (business) | Medium | P1 |
| **DOCX** | ✅ mammoth.js | Large (business) | Medium | P2 |
| **DICOM** | ✅ cornerstone.js | Niche (high value) | Medium | P2 |
| **3D models** | ✅ three.js | Niche (growing) | Medium | P3 |
| **CAD** | Partial (DXF via JS) | Niche (high value) | High | P3 |

### The Compounding Effect

Languages × File viewers = the total addressable market.

17 language variants × 12 file viewers = 204 unique value propositions. "Share DICOM images in Portuguese" is a niche — but it's a niche nobody else is serving, and each additional language or viewer is incremental effort on an existing platform.
