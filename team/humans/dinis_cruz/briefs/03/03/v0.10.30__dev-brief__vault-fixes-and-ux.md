## Fixes and Improvements

### 1. Auth Token Should Only Be Required for Upload

**Current behaviour:** The vault asks for the auth token (access token) at the start, before you can do anything.

**Desired behaviour:** To open and browse a vault, you only need the **Vault Key** (the decryption key). The auth token is only needed when you want to upload/write files. Reading and downloading doesn't require it.

**Why this matters:** It dramatically simplifies the read-only experience. Someone you share a vault with can view everything with just the Vault Key. If they need to add files, then they provide the auth token. This is the difference between "view this vault" and "collaborate on this vault."

**Implementation:**
```
Open vault → prompt for Vault Key only → decrypt settings + tree → browse + preview + download
Upload file → if no auth token → prompt for auth token → proceed with upload
```

### 2. Settings Panel

Add a Settings panel accessible from the vault UI:

**Content:**
- Vault name and description (editable)
- Vault Key display (for sharing — with copy-to-clipboard button)
- Auth token management (enter/change the auth token for uploads)
- Raw JSON viewer: show the actual vault-settings.json and vault-tree.json (decrypted) for debugging
- Vault statistics: file count, total size, folder count, last modified

**The raw JSON viewer is important for debugging:** Being able to see the actual data structure that controls the vault UI helps diagnose issues and eventually enables manual editing of the vault structure.

### 3. Folder Upload Bug

**Bug:** Files uploaded while inside a subfolder don't get created in the correct folder — they go to the root.

**This is the same bug from v0.1.0.** Must be fixed in this iteration. The upload handler needs to use the current folder path when updating the vault tree.

### 4. Drag-and-Drop

**Current state:** Not working in the vault (was listed as a feature but not implemented in this version).

**Required:** Drag-and-drop files onto the vault to upload. Should work:
- In the main file list area
- Into specific folders (drop onto a folder to upload into it)
- With multiple files
- With folders (auto-zip as per the new send.sgraph.ai capability)

### 5. Folder Management

**Add folder:** Clear UI to create a new subfolder (currently possible but UX needs improvement).

**Rename folder:** Click/double-click folder name to rename inline.

**Rename file:** Same pattern — click/double-click to rename.

**Move file:** Drag a file from one folder to another (future — not required for this iteration).

### 6. File Metadata Banner

When a file is selected, show a banner/panel at the top of the right panel:

```
┌──────────────────────────────────────────────────┐
│ 📄 report.pdf                                     │
│ Size: 245 KB | Type: PDF | Uploaded: 2 Mar 2026  │
│ Folder: /documents/                               │
│ [Download] [Delete] [Rename]                      │
└──────────────────────────────────────────────────┘
```

**Future:** This banner will also show name-value pairs (custom metadata about the file). The infrastructure should support arbitrary key-value metadata per file from the start, even if the UI for editing it comes later.

### 8. Inline Editing (Text, Markdown, JSON)

For text-based files, allow editing directly in the vault:
- Click "Edit" on a text/markdown/JSON file
- Edit in a simple text editor (syntax highlighting nice-to-have)
- Save → encrypt → upload as new version → update vault tree

### 9. Local Storage Viewer (Debug Tool)

Add a panel (accessible from Settings or Debug) that shows everything stored in the browser's local storage. This is a development and debugging tool:

- What keys are stored
- What values they hold
- Ability to clear individual keys or all local storage
- Display storage quota usage

### 10. Chrome Sync Exploration (MVP)

**The idea:** Can we store vault keys in Chrome's synced storage so that when a user opens Chrome on another device (logged into the same Google account), their vault keys are already available?

**What to explore:**
- Chrome extensions can use `chrome.storage.sync` which syncs across devices
- Can a web app access this without being an extension?
- Alternative: use a very small Chrome extension that stores vault keys and makes them available to sgraph.ai domains
- What are the security implications? (The vault key gives full read access)

**MVP output:** A proof-of-concept that answers: "Can I open a vault on one device and have the key automatically available on another device via Chrome sync?"

## Implementation Notes

**All features should be individual web components** with minimal coupling. Keep the blast radius small — a bug in the markdown editor shouldn't affect the folder tree.

**Leverage existing code:** Import modules from send.sgraph.ai rather than duplicating. The shared module publishing (DevOps action from the v0.10.23 brief) enables this.
