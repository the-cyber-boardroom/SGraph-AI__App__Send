## The Milestone

On 2 March 2026, SG/Send had its **first 10 paying customers**. All from personal outreach — messages sent to friends, ex-colleagues, and contacts. Every one of them completed the full purchase flow and sent encrypted files.

This is the first real commercial validation of SG/Send.

## What Happened

### The Outreach

The message was Variant A from the outreach brief: "Hey, do me a favour — in exchange for a £5 Amazon voucher, go to the website, buy credits, and try the product. Tell me what you think."

Messages sent via WhatsApp and LinkedIn DM. Some in English, some to Portuguese and Spanish contacts (multilingual test).

### The Purchase Flow

Every customer completed the purchase successfully. Specific feedback:

| Aspect | Result |
|---|---|
| Finding the website | ✅ No issues reported |
| Finding the purchase flow | ✅ People navigated to it |
| Stripe checkout | ✅ Smooth — Apple Pay worked well |
| Payment completion | ✅ All 10 completed |
| Receiving access token | ✅ Manual — within 5-10 minutes of purchase |
| First file transfer | ✅ All customers sent files successfully |
| Large files (300MB+) | ✅ Confirmed working |
| Product not working | ❌ Zero reports of failure |

### The Manual Token Flow

The human manually created tokens via the admin UI using a convention:
```
sg_send___[name]___[random_numbers]
```

Within 5-10 minutes of each Stripe notification, the customer had their token and a personal message: "Thanks! Try dropping some PDFs."

**This personal touch actually worked well.** Customers felt welcomed, not processed. The 5-minute response time felt fast enough. At this scale, manual is fine.

### Early Warning: Bank Fraud Flag

One customer reported: **"The transaction gave a red flag on my bank."**

The transaction on Stripe's side is clean — payment completed, no issues. This is almost certainly because The Cyber Boardroom Ltd is a new merchant with no transaction history. Banks flag unfamiliar merchants, especially for online card-not-present transactions from new companies.

**Why this matters:**
- If one customer reported it, others may have seen the flag and not mentioned it — or worse, abandoned the purchase
- This will resolve over time as the merchant builds transaction history, but in the short term it could suppress conversion
- Apple Pay transactions may be less affected (Apple's own fraud layer provides additional trust signals to the issuing bank)

**Actions:**
- Track how many customers report this (add to the customer tracking spreadsheet)
- If it's widespread, consider adding a note on the checkout page: "You may see a verification prompt from your bank — this is normal for first-time purchases from new vendors"
- Monitor Stripe's Radar dashboard for declined transactions that customers never reported
- This may resolve naturally as Stripe builds a positive transaction history for the merchant account
- Long term: Stripe's fraud score for the merchant improves with successful transactions and low chargeback rates

**Note for Sherpa:** If any customer mentions a bank flag, respond immediately with reassurance: "That's your bank being cautious because we're a new merchant — the transaction is confirmed on our side and your credits are ready." Speed of response here is critical — a customer who sees a fraud flag and doesn't hear back quickly will assume the worst.

## The Folder-to-Zip Case Study

This deserves to be documented as a case study because the sequence is textbook:

| Time | Event |
|---|---|
| T+0 | Customer reports: "You can't drop folders" |
| T+5min | Human asks: "What's the workflow you want?" |
| T+10min | Agreement on solution: folder → auto-zip → send |
| T+15min | Brief sent to Architect + Developer |
| T+20min | Team confirms feasibility: lazy-load zip library |
| T+25min | Implementation plan created, pushed to GitHub |
| T+30min | Customer (developer) reviews the plan — impressed |
| T+45min | Team builds the feature |
| T+60min | Shipped to dev, first round of testing |
| T+75min | Bug fixes applied |
| T+90min | **Live in production** |

**Feature request to production in 90 minutes.** This is what agentic development looks like.

**The reverse insight:** After shipping folder-to-zip upload, the human realised downloading a zip and having to unzip locally is annoying. So the team also built in-browser zip preview — browse zip contents, selectively view files. This "reverse the annoyance" thinking is a pattern to cultivate.

## The Video Sharing Opportunity

The video/audio preview capability opens an underappreciated use case. Compare:

| Method | Send a 300MB video to someone |
|---|---|
| **WhatsApp** | Compression, quality loss, 16MB limit (or 2GB with recent updates but still compressed) |
| **Email** | Attachment limit (25MB Gmail). Forget it. |
| **Dropbox/Google Drive** | Upload, share link, recipient downloads, finds a viewer, opens |
| **SG/Send** | Upload, share link, **plays instantly in browser**. Zero friction. Encrypted. |

The zero-friction playback is the differentiator. This should be highlighted in communications.
