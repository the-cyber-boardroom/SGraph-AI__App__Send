## Executive Summary

SG/Send is a zero-knowledge encrypted file transfer and data room platform. It delivers files securely — encrypted end-to-end, tamper-evident, zero-knowledge server, receiver-verified. But delivering a file securely does not make the file itself safe. A malicious sender can encrypt malware and SG/Send delivers it just as faithfully as a clean document. SG/Send is the secure courier. It does not inspect the package.

This document describes the **File Safety Orchestration Architecture** — a platform-level capability that enables third-party scanning services (CDR engines, antivirus, file reputation, DLP, sandboxing, metadata stripping, and others) to be plugged into the SG/Send transfer flow in a way that preserves zero-knowledge guarantees while providing cryptographically verifiable proof that a file was scanned and what the result was.

> **The Core Insight:** SG/Send's zero-knowledge constraint makes it impossible for the server to scan files. Scanning services are the trusted third parties that can — and whose signed results become cryptographically bound to the file and encrypted exclusively for the receiver. The server stores encrypted blobs it cannot read: the file, the scan results, and the attestation chain.

This architecture delivers three things:

**1. A safe file, delivered safely.** Scanning services rebuild, sanitise, or verify the file. SG/Send then encrypts it and cryptographically binds the scan result to that specific file via hash. The receiver can verify, at the point of opening, that: (a) this file was processed by a named scanning service, and (b) what they are about to open is byte-for-byte what the scanning service produced — nothing was substituted, nothing was injected in transit. Any post-scan tampering breaks the hash binding and is immediately detectable.

**2. A novel commercial model.** The receiver defines which scanning services are required, and the sender pays per scan. This inverts the traditional buyer model for security tooling — procurement is no longer the only path to revenue for scanning vendors. Every receiver who mandates a scanning service generates recurring API revenue automatically, driven by SG/Send's user growth rather than the vendor's sales effort.

**3. A cross-domain architecture that has no equivalent in the market.** The SG/Send scanning chain is the first cross-domain file security architecture in which the crossing infrastructure never sees plaintext — ever. Not transiently, not in a buffer, not in a log. The only entity that sees cleartext is a minimal, isolated pod with a single API endpoint, in its own network, separate from both the crossing infrastructure and the destination. These capabilities map directly to requirements that government, defence, and regulated-industry customers already have.

## The Scanning Chain — Platform Architecture

### What "Making the File Safe" Means

There is no single technology that makes a file safe. "File safety" is a spectrum of complementary techniques, each addressing a different class of risk:

| Service Type | What It Does | Risk Addressed | Examples |
|---|---|---|---|
| **Content Disarm and Reconstruction (CDR)** | Deconstructs a file to its structural elements and rebuilds a known-good version. Strips macros, embedded objects, scripts, polyglot structures. Does not rely on signatures or detection — assumes everything is potentially malicious and rebuilds from specification. | Zero-day exploits, weaponised documents, evasion techniques, polyglot files | OPSWAT MetaDefender, Votiro, Resec, and others |
| **Antivirus / Anti-malware** | Signature-based and heuristic detection of known malware, trojans, ransomware. Fast, well-understood, high coverage for known threats. | Known malware, variants of known families | ClamAV (open source), commercial AV engines, multi-engine scanning |
| **File Reputation** | Hash lookup against databases of known-bad, known-good, and suspicious files. Instant verdict for previously seen files. | Known-bad files, previously reported malware, supply chain compromise | Hash-based reputation services, community threat intelligence feeds |
| **Data Loss Prevention (DLP)** | Scans file content for sensitive data: PII, financial data, classified markers, regulatory-controlled information. Blocks or flags files before they leave the sender's control. | Accidental data leakage, regulatory non-compliance, insider threat | Pattern-based DLP, NLP-based content classification, custom policy engines |
| **Sandbox / Detonation** | Executes the file in an isolated environment and observes behaviour: network connections, file system changes, registry modifications, process spawning. Catches threats that evade static analysis. | Advanced persistent threats, behaviourally malicious files, obfuscated payloads | Dynamic analysis environments, behavioural analysis engines |
| **Metadata Stripping** | Removes hidden metadata: author names, GPS coordinates, revision history, tracked changes, embedded thumbnails, printer paths. | Privacy leakage, unintentional disclosure, forensic traces | ExifTool, custom strippers, CDR engines (as a side effect) |
| **Format Validation** | Confirms the file matches its claimed type. Detects polyglot files (files that are valid as two or more file types simultaneously, used to evade scanners). | Type confusion attacks, polyglot evasion, mislabelled files | Structural validators, magic byte verification |
| **Watermarking / Provenance** | Embeds invisible markers that track document origin and distribution path. Does not prevent misuse but enables attribution. | Document leakage attribution, supply chain tracking | Steganographic watermarking, blockchain-anchored provenance |

**No single service covers all risks.** The architecture must support chaining multiple services in a single flow — CDR + AV + DLP in a single scan pass, with each result independently attested and bound to the file.

### Why SG/Send Is the Right Orchestrator

SG/Send's position is unique: it sits at the point where files move between parties, which is exactly where scanning has the highest impact. The sender has the file. The receiver needs assurance. SG/Send is the transfer layer between them.

The platform already has:
- Zero-knowledge encrypted storage (the server cannot scan files — this is a feature, not a limitation)
- PKI key infrastructure (every party has a key pair)
- Access token management (controls who can send and receive)
- A pod architecture designed for trusted intermediary processing
- Open-source codebase (Apache 2.0) that partners can inspect, audit, and extend

What the scanning chain adds is a pluggable, PKI-attested layer where third-party services process the file in a trust-isolated environment, and the results are cryptographically bound to the file before delivery.

## The Receiver-Defined Scanning Model — A New Commercial Pattern

In the traditional model, the buying organisation decides what scanning is required for inbound files and purchases the scanning tools centrally. This works for enterprises with procurement budgets and security teams, but it has a ceiling: the buyer is always the receiving organisation, procurement is centralised and slow, and there is no per-use component.

SG/Send enables a fundamentally different model:

1. The receiver publishes their delivery requirements in their SG/Send profile or data room: *"All files sent to me must pass CDR and antivirus scanning before delivery."*
2. The sender, before uploading the file, is shown the requirement: *"This recipient requires CDR + AV scanning. Proceed?"*
3. The file is passed to the **SG/Send pod** — the trusted intermediary. The pod is the only component that can see cleartext (by design), is not controlled by the sender, and is not the zero-knowledge server.
4. The pod calls the required scanning services via their APIs, submitting the file. The sender's browser never touches the scanning APIs directly — removing any possibility of a sender substituting a clean file for scanning and uploading a malicious one afterwards.
5. Each scanning service processes the file and returns results (rebuilt file, scan report, verdict, hashes).
6. The pod constructs an attestation — signed with **the pod's own PKI key**: `{ file_hash_in, scanning_services_called, results_per_service, policy_applied, timestamp, file_hash_out }`. This is the pod attesting: *"I submitted this exact file to these services and received these results."*
7. The pod encrypts the rebuilt/scanned file and the signed attestation with the **receiver's public key only**. The sender cannot read either. The SG/Send server stores encrypted blobs it cannot read.
8. The receiver decrypts both, verifies the pod's signature, and confirms that `file_hash_out` matches `sha256(decrypted_file)`. The chain is verified.
9. **The sender pays the scanning cost** — a per-use charge driven by receiver requirements, not a procurement decision.

> **Commercial Implication:** This creates a pull-based demand model. The receiver's requirements automatically generate scanning API calls whenever anyone sends them a file. As SG/Send's user base grows, every active receiver who sets scanning requirements generates recurring API revenue — without any additional sales effort from the scanning vendor.

## Cross-Domain File Security — A Massive Opportunity

### The Core Insight: The Crossing Point Never Sees Plaintext

Traditional cross-domain solutions have a structural weakness: the crossing point must see cleartext to perform scanning, which makes it both a high-value attack target and a trust bottleneck. Every file that crosses the boundary is held in plaintext at the very point most likely to be under attack.

The SG/Send scanning chain breaks this assumption entirely.

**The crossing point is not the trusted pod — and neither is fully trusted.** The crossing point — the network boundary, the gateway, the firewall — never sees plaintext at any stage. The **trusted pod** is a separate, minimal, hardened service sitting in its *own* isolated network: not the untrusted source domain, not the destination domain, and deliberately not co-located with the crossing point infrastructure. The pod's network is separately managed, separately monitored, and has a minimal attack surface by design — one port, one API method: "process file." Keeping the pod's network distinct from the crossing point is intentional: crossing points, by their nature, are processing dangerous content and are therefore themselves not fully trusted. If a crossing point is compromised, it should not have any lateral reach into the pod's environment.

The parties and what each can see:

- **Untrusted sender:** has the file in plaintext, has the pod's public key, has nothing else
- **Crossing infrastructure** (network hops, gateways, firewalls): sees only encrypted blobs in transit — never plaintext at any stage
- **Pod** (in its own isolated network): sees cleartext transiently during scanning only; immediately re-encrypts and discards
- **Final recipient:** the only entity that can open the scanned file; verifies the full provenance chain independently

### The Cloud Account Use Case

This pattern is immediately applicable to cloud-to-cloud data flows, which represent one of the most common and least well-solved cross-domain problems in enterprise security today.

**Scenario:** An organisation has an untrusted AWS account (e.g., a partner's account, a data ingestion pipeline, a third-party processing environment) that needs to deliver files into a trusted AWS account (e.g., a regulated workload, a compliance-controlled zone).

**Today's options are all bad:**
- Share credentials or IAM roles → the untrusted account gets broader access than intended
- Set up a VPN or Direct Connect → expensive, complex, doesn't address file-level trust
- Manual review process → doesn't scale, human bottleneck at the boundary
- Traditional scanning gateway → the gateway holds plaintext, becomes a target

**With SG/Send's scanning chain:**

1. The untrusted account encrypts the file with the pod's public key — a publicly available endpoint, no pre-agreed trust relationship needed
2. The untrusted account sends the encrypted file directly to the pod endpoint
3. The pod — running in its own isolated network, reachable from both accounts but co-located with neither — decrypts, runs the required scanning services per the destination account's policy, re-encrypts the rebuilt file for the final recipient's key, and returns the encrypted pack to the sender
4. The untrusted account forwards the encrypted pack through normal channels to the destination
5. The trusted account's recipient decrypts and verifies the full chain — scanning was applied, the file hash is intact, the pod attested the process

**What this unlocks:**
- The untrusted account needs only the pod's public key and the API endpoint — no credentials, no IAM roles, no VPN
- The trusted account's security policy is enforced without the untrusted account being able to observe or influence it
- The crossing infrastructure is stateless with respect to file content — it forwards encrypted blobs, nothing more
- Every crossing generates a cryptographically verifiable audit record: who sent what, when, what scanning found, what was delivered

## Technical Integration Architecture

### The PKI-Verified Scanning Chain

The integration is built on SG/Send's PKI scanning chain architecture. Every party has a PKI key pair. The scan result is cryptographically bound to the specific file and cannot be forged, replayed, or modified.

A critical design principle runs through the entire flow: **the sender is not trusted.** The hash that binds the scan result to the file must never be computed by the sender — a malicious sender could compute the hash of a clean file and upload a different one. All trust-bearing operations go through the **SG/Send pod**, which is the only component that is both trusted and able to see cleartext.

**Current flow (pod as trusted intermediary):**

| Step | What Happens |
|---|---|
| 1 | Sender selects a file and uploads it to the SG/Send pod. The pod, not the sender's browser, is the trusted intermediary from this point forward. |
| 2 | The pod computes `sha256(file_bytes)`. This hash — computed by a trusted party — is the binding primitive for the entire chain. |
| 3 | The pod calls each required scanning service API (CDR, AV, DLP, etc.) with the cleartext file and the receiver's named policy for each service. |
| 4 | Each service returns its result (rebuilt file, verdict, report, hashes). The pod cross-checks returned hashes against its own computed hash. |
| 5 | The pod constructs and signs an attestation with its own PKI key: `{ file_hash_in, services_called[], results_per_service[], policy_applied, file_hash_out, timestamp }`. This is the pod certifying: "I submitted this exact file to these services and received these results." |
| 6 | The pod encrypts the rebuilt/scanned file and the signed attestation with the receiver's public key only. The sender cannot read either. The SG/Send server stores encrypted blobs it cannot read. |
| 7 | Receiver downloads both blobs, decrypts with their private key, verifies the pod's attestation signature, and confirms `file_hash_out` matches `sha256(decrypted_file)`. Chain verified. |

**Future flow (once scanning services support native result signing):**

The pod intermediary step is a pragmatic solution for now. The cleaner end state is for scanning services to sign API responses directly, removing the need for the pod to act as a trust proxy. There are two paths:

| Path | What the Scanning Partner Adds | What It Enables |
|---|---|---|
| **A — Service signs results with its own private key** | The scanning service adds a stable PKI key pair and signs every API response: `{ file_hash_in, result, file_hash_out, timestamp, service_signature }` | The attestation becomes "the scanning service itself certified this result" — no intermediary required. The receiver verifies the service's signature directly. |
| **B — Service encrypts output with the receiver's public key** | The scanning service takes the receiver's public key as a request parameter and encrypts the output file directly | The rebuilt file leaves the scanning service already encrypted for the intended receiver. No intermediary ever holds the cleartext rebuilt file. Only the receiver can open it. |

Both paths strengthen the chain and reduce trust dependency on the SG/Send pod. Path A is the more immediately valuable — it is a differentiator no scanning vendor currently offers and does not require changes to how SG/Send manages sender identity.

**A note on trust levels:** SG/Send is actively encouraging scanning partners to adopt native result signing. It upgrades the attestation from "SG/Send vouches that this result came from Service X" to "Service X itself cryptographically signed this result" — a materially stronger guarantee for high-assurance environments. SG/Send will work with any partner willing to implement this.

### What the Sender Cannot Do — Attack Surface Analysis

The sender is explicitly untrusted throughout this flow. The pod architecture is specifically designed to remove the sender from any trust-bearing operation.

| Attack | Why It Fails | Who Prevents It |
|---|---|---|
| Submit clean file for scanning, swap in malicious file before upload | The sender never calls scanning services directly. The pod receives the file, computes the hash, calls services, and re-encrypts the output — all without the sender involved. There is no window to swap the file. | SG/Send pod |
| Forge a passing scan result | The pod's attestation is signed with the pod's private key. The sender does not have it. Once scanning services sign natively, the sender does not have the service's private key either. | Pod key pair (now); service key pair (future) |
| Skip scanning entirely | The receiver requires a valid signed attestation. No attestation = file rejected by the receiver's client before opening. | Receiver enforces the requirement client-side |
| Read the scan result to learn what defences are in place | The attestation and report are encrypted with the receiver's public key only. The sender never sees them. | SG/Send pod encrypts for receiver only |
| Replay an old attestation against a new file | The attestation includes `file_hash_in`, `file_hash_out`, and timestamp. The receiver verifies `file_hash_out` matches the actual decrypted file, and checks the timestamp is within an acceptable window. | Receiver verifies hash binding and recency |
| Claim the pod processed the file when it did not | The pod's signature is over a specific `{ file_hash_in, file_hash_out, results[], timestamp }` tuple. Any mismatch between the attested values and the actual file is detectable by the receiver. | Pod attestation structure |

### Integration Tiers

There are multiple paths to integration, building from simple REST API calls to fully embedded scanning engines. Each tier adds capability; all build on the same underlying pod architecture.

#### Tier 1 — REST API Integration (~3–5 weeks per service)

The SG/Send pod calls the scanning service's cloud API on behalf of the sender. This is the fastest path to a working integration and requires nothing from the scanning partner beyond existing API access.

| Item | Spec |
|---|---|
| API endpoint | The scanning service's existing REST API (sync for small files, async for large) |
| Policy mode | Named policy per receiver tier — mapped to the service's native policy model |
| Hash binding | Pod computes `sha256(file_bytes)` before and after scanning; both hashes embedded in attestation |
| Attestation | Signed by pod's private key: `{ file_hash_in, file_hash_out, service_name, service_response, policy, timestamp }` |
| Output | Scanned/rebuilt file + signed attestation, both encrypted with receiver's public key |
| Limitation | Trust is "SG/Send pod vouches for the scanning service's result" — not a direct service signature |

#### Tier 2 — Data Room Policy Enforcement (~6–10 weeks, builds on Tier 1)

Scanning becomes a configurable security policy at the data room level. Every file uploaded to a room with scanning requirements passes through the pod before other members can access it. Fail modes (block or quarantine for admin review) are configurable per room. Scan reports are stored encrypted alongside the file so members can see what scanning found before opening.

| Policy Setting | Description |
|---|---|
| `scanning_required: [cdr, av]` | All files uploaded to this room must pass CDR and antivirus scanning. Uploads that fail are quarantined automatically. |
| `cdr_policy: strict_office` | Named CDR policy variant for Office documents. Macros, DDE, hyperlinks, tracked changes — all stripped. |
| `cdr_policy: pdf_flatten` | PDF CDR with JavaScript, actions, and embedded files removed. AcroForm fields preserved or stripped per room setting. |
| `av_engine: multi` | Multi-engine antivirus scanning (multiple AV engines in parallel for highest detection rate). |
| `dlp_policy: pii_block` | DLP scan blocks files containing PII patterns (email addresses, phone numbers, national ID numbers). |
| `scan_fail_mode: block` | If any required scan fails or returns non-conforming status, file is blocked. Members never see it. |
| `scan_fail_mode: quarantine_review` | Non-conforming files go to the room admin's review queue. Admin can release or reject after manual inspection. |
| `require_scan_report: true` | Scan reports stored (encrypted) alongside the file. Members can see what scanning found before opening. |
| `new_member_strict: true` | Dual authorization for files uploaded by low-trust members. Admin reviews scan report before releasing to other members. |

#### Tier 3 — Embedded Engine Integration (~12–16 weeks, builds on Tier 2)

Scanning engines are integrated directly into the SG/Send pod runtime. This eliminates the network round-trip to cloud APIs, reduces latency to sub-100ms for common file types, and enables offline/air-gapped operation — critical for the `.sgroom` portable data room use case where files travel through supply chains without internet access. This is the deployment model for defence and government environments where cloud API connectivity is not permitted.

#### Tier 4 — Browser-Side Scanning (Future, WASM)

For scanning engines that can be compiled to WebAssembly, scanning can happen entirely in the sender's browser — no pod, no server, no API call. The scanned file never leaves the sender's machine unencrypted. This is the ultimate zero-knowledge model: the file is scanned and rebuilt locally, the results are signed locally (with a client-side key), and only the scanned file is uploaded. This tier is limited to scanning engines that can run in WASM and is primarily applicable to CDR (which is deterministic and doesn't require cloud-based threat intelligence).

## Cross-Domain Technical Architecture

### Encrypting Scan Results for the Crossing Point — Separating Policy from Content

Scan results should be encrypted separately — **for the crossing point's public key**, not the recipient's. This produces a clean separation of responsibilities:

- The **crossing point** can decrypt and verify the scan results — confirm that CDR passed, that AV found no threats, that DLP cleared the file, that the policy was applied correctly — without ever seeing the file content
- The **recipient** receives the rebuilt file but is never asked to make a security decision. The policy enforcement happened at the boundary, verifiably, before the file reached them
- The **file content** and the **scan results** are encrypted under different keys, for different parties, for different purposes

This supports chained crossing points — a file passing through multiple domain boundaries can have scan results re-encrypted for each successive crossing point's key. Each crossing point verifies policy compliance for their boundary without seeing what the previous crossing point saw, and without ever seeing the file content.

### Layered Crossing Points — Defence in Depth and Compromise Detection

In a real-world deployment, there would typically be a small number of highly isolated, heavily protected SG/Send pod deployments — and a much larger number of crossing points at the edges of various networks, which are explicitly *not* fully trusted. This is by design.

Crossing points are boundary infrastructure. They process dangerous content by definition. They are exposed to untrusted networks. Some will be compromised. **The architecture assumes this and plans for it.**

Because every step in the chain is cryptographically attested by the pod, a compromised crossing point cannot silently pass a malicious file. The inner crossing points — or the final recipient's client — will independently verify the pod's attestation. If a file arrives without a valid pod attestation, or with an attestation that doesn't match the file's hash, it is rejected automatically. No human has to notice. The cryptographic chain does the detection.

```
ATTACKER          OUTER EDGE                INNER CROSSING          DESTINATION
                  CROSSING POINT            POINT                   DOMAIN
                  (compromised)
────────────      ──────────────            ──────────────          ───────────

Submit            Outer crossing
malicious ──────► point is                  
file              compromised —
(no valid         passes file    ─────────► Receives encrypted blob
pod               through                   Verifies pod attestation
attestation)      without                         │
                  checking                   Attestation missing
                                             or hash mismatch
                                                   │
                                                   ▼
                                             FILE REJECTED ◄── Automatic,
                                                   │             cryptographic,
                                             ALERT RAISED        no human required
                                                   │
                                             Outer crossing
                                             point flagged
                                             as compromised
                                                                  ✗
                                                             File NEVER
                                                             reaches
                                                             destination
```

This means:

- **Outer crossing points can be compromised without catastrophic consequences** — inner layers catch what they miss
- **Compromise is detected automatically** — not by log analysis after the fact, but by cryptographic verification failing in real time
- **The pod network remains isolated** — a compromised crossing point has no lateral reach into the pod's network, because the pod network is separate and only exposes one API endpoint
- **The number of pod deployments can be small** — a handful of well-managed pod deployments can serve many crossing points across a wide network topology

In practice, this maps well to how government and defence networks are actually structured: a small number of highly accredited, independently audited processing environments, with a much larger number of boundary points managed to a lower standard because their exposure makes full accreditation impractical.

### Multi-Classification Documents in a Single Transfer

This architecture extends naturally to a scenario that is extremely common in government and enterprise environments: a single transfer containing documents at different classification levels, each readable only by the authorised recipient for that classification.

**Example:** a sender packages two documents in a ZIP:

1. A **routing document** — encrypted with the key of the *routing service* inside the destination domain. This document contains metadata: "the sensitive file in this package should be delivered to Person X at Department Y." Only the routing service can read it.
2. A **sensitive document** — encrypted with the key of the *final recipient* (Person X at Department Y). Only that individual — or their authorised pod — can open it.

The routing service receives the ZIP, decrypts the routing document, reads the delivery instruction, and forwards the sensitive document to Person X. It cannot read the sensitive document — it holds the wrong key. Person X decrypts the sensitive document and cannot read the routing metadata — they hold the wrong key.

At no point did any single party have access to both the routing intent and the sensitive content. The network saw only encrypted blobs throughout. This pattern scales to arbitrary classification hierarchies — a single ZIP can contain documents at multiple classification levels, each encrypted for the appropriate key, with a routing manifest encrypted for the delivery service.

### Recipient Privacy — The Receiving Party Is Never Exposed

The untrusted sender never learns who the actual recipient is. The sender encrypts for a public key — a public key is just a number, carrying no identity information. The untrusted domain has no way to determine whether that key belongs to a person, a pod, a team, or an automated pipeline — and no way to link two separate deliveries to the same recipient if the key has changed between them.

Combined with key rotation, this becomes a powerful privacy and counter-surveillance property:

- **Rotate keys hourly:** the untrusted sender cannot correlate deliveries over time
- **Rotate keys per engagement:** a law firm receiving documents from multiple opposing parties uses a different key for each — no party can determine that the same firm received documents from the others
- **Rotate keys per file:** every file has a unique recipient key

### Time-Bound Keys — Audit Trail and Instant Revocation

Rather than the recipient holding a permanent key pair, the recipient **requests a time-bound private key** for each receiving window. This produces two extremely valuable properties simultaneously.

**A powerful audit trail by construction:** every key request is a logged event — who requested a key, for what time window, from which context. You cannot receive a file without requesting a key, and you cannot request a key without generating an audit record. The audit trail is not something that can be turned off or bypassed; it is the mechanism itself.

**Instant, cryptographic revocation:** deleting a private key is not a policy change or an ACL update — it is a mathematical event. Every file encrypted for that key becomes permanently unreadable the moment the private key is deleted, regardless of where copies of the encrypted file exist. A concrete scenario: a regulatory investigation requires that a set of documents become inaccessible as of a specific date and time. Under a key-rotation model, this is a single operation — delete the private keys for the relevant time windows. Every file encrypted in those windows, across every storage system that holds copies, becomes unreadable simultaneously and permanently.

### Ephemeral and Document-Specific Keys — The Highest-Assurance Mode

**Document-specific key pairs:** each file gets a unique key pair generated at the moment of sending. The private key is held in escrow (accessible only to the intended recipient, via a key request that generates its own audit record) or destroyed after first use.

This enables:

- **Provable single-read documents:** the private key is destroyed after the recipient's first decryption. No second read is possible — not by the recipient, not by an attacker, not by anyone.
- **Time-locked documents:** the private key is not released until a specified time. The file can be delivered in advance and become readable only at the agreed moment.
- **Quorum-unlocked documents:** the private key is split across multiple keyholders (Shamir's Secret Sharing). The document becomes readable only when a quorum of keyholders agree to reconstruct it — relevant for board resolutions, legal settlements, regulated disclosures.
- **Cryptographic destruction:** deleting a document-specific private key is the equivalent of shredding every copy simultaneously, regardless of where those copies exist or how many there are.

For locked-down environments — defence, legal, regulated finance — this is not a theoretical capability. It is a compliance and operational requirement that no current file transfer solution provides.

### This Is Already Buildable — SG/Send Has Most of It Today

This workflow, while it sounds architecturally complex, is largely already present in SG/Send's existing open-source codebase:

| Capability Needed | SG/Send Status |
|---|---|
| Browser-based PKI key generation | ✅ In production today |
| Key storage with pluggable backends (memory, S3, filesystem, ZIP) | ✅ MemoryFS abstraction already built |
| Per-file or per-session key generation | ✅ Follows directly from existing key generation |
| Key request workflow with audit logging | 🔧 Architecture designed; implementation milestone |
| Time-bound key expiry | 🔧 Straightforward extension of existing key management |
| Secure key deletion | 🔧 Requires explicit zeroing on deletion; well-understood implementation |
| Pod as trusted key escrow intermediary | 🔧 Pod architecture designed; implementation milestone |
| Encrypted file storage with receiver-only access | ✅ In production today |

The items marked 🔧 are implementation milestones, not architectural unknowns. The hard problems are already solved. **And because SG/Send is open source (Apache 2.0), any partner or customer does not need to wait for SG/Send to prioritise these features.** A capable team could build the time-bound key request workflow on top of SG/Send's existing infrastructure in weeks, not months — and contribute it back to the platform.

## Scanning Service Menu — The Partner Ecosystem

### How Partners Integrate

Every scanning service follows the same integration pattern:

| Step | What Happens |
|---|---|
| 1 | Partner provides API access (cloud endpoint or SDK) |
| 2 | SG/Send builds a pod adapter: a small module that translates between the pod's internal format and the partner's API |
| 3 | The adapter is tested against the partner's sandbox/test environment |
| 4 | The partner is added to the scanning service menu — receivers can select it in their requirements |
| 5 | When a sender uploads a file to a receiver who requires this service, the pod calls the adapter, which calls the partner API |
| 6 | Results are attested, encrypted, and delivered as described in the scanning chain architecture |

**The adapter is typically 200-500 lines of code.** It maps the pod's `{ file_bytes, policy_name }` to the partner's API format and maps the partner's response back to the pod's `{ result, rebuilt_file, hashes }` format. This is straightforward integration work, not architectural innovation.

### Service Tiers — Receiver Choice

Receivers choose from a menu of scanning services, organised by category:

| Category | Services Available | Cost Model | Use Case |
|---|---|---|---|
| **Basic** | Open-source AV (ClamAV), format validation | Free (included in SG/Send) | Default for all transfers |
| **Standard** | Commercial multi-engine AV + file reputation | Per-scan, low cost | Business file transfers |
| **CDR** | CDR engine (rebuild to known-good spec) | Per-scan, medium cost | High-assurance document exchange |
| **Enterprise** | CDR + multi-AV + DLP + sandbox | Per-scan, higher cost | Regulated industries, legal, finance |
| **Government** | CDR + AV + DLP + embedded engine (air-gapped) | Enterprise licence | Defence, classified environments |
| **Custom** | Receiver specifies exact service chain | Per-scan, variable | Specific compliance requirements |

The Basic tier (ClamAV + format validation) can be built into the pod at near-zero cost and provides a baseline level of scanning for every transfer. This establishes the scanning chain as a default capability, not a premium add-on.

### Open-Source First

ClamAV integration is the first scanning service to build. It is:
- Free and open source (no commercial dependency)
- Well-understood API (clamdscan or REST API)
- Sufficient for detecting known malware in most business contexts
- A proof-of-concept for the entire scanning chain architecture
- The baseline that every higher tier builds on

Once ClamAV is working end-to-end in the pod — file in, scan, attestation, encryption, delivery, receiver verification — adding commercial services is adapter work, not architecture work.

## Commercial Structure — How Partners Get Paid

### Option A — API Reseller Model
SG/Send routes all scanning API calls and marks up the per-call cost. Simple to implement, fast to launch, partner revenue from day one. SG/Send takes margin on volume; the partner gets a distribution channel with no direct sales effort required.

### Option B — White-Label Tier Inclusion
Scanning is bundled into SG/Send's paid tiers (e.g., a "Verified Transfer" tier includes CDR + AV scanning). SG/Send pays partners a volume licence; users pay SG/Send a subscription premium. Predictable revenue for both parties; simpler UX where scanning is automatic rather than a user choice.

### Option C — Joint Enterprise GTM
For government and enterprise data room deals, SG/Send and scanning partners co-sell. SG/Send brings zero-knowledge transfer and data room capability; the partner brings scanning accreditation and government relationships. Both appear on the contract. Opens deals neither could win alone.

**Recommendation:** Start with **Option A** (fast, low commitment, proves the model) while building toward **Option C** for the government/defence segment where both brands carry independent weight.

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Partner data retention contradicts zero-knowledge positioning | Medium | High | Contractually require zero retention after scan completion. Alternatively: embedded engine (Tier 3) where no external API call is made. |
| Cloud API latency at p95 creates unacceptable UX on file upload | Medium | Medium | Async API for files > 5MB. Progress indicator during scanning. Fall back to advisory mode (scan in background, deliver immediately) for lower-sensitivity tiers. |
| Native result signing requires partner engineering investment | Medium | High | Phase 1 uses SG/Send's own signing wrapper around the partner's result — the pod signs that it received this result from the service at this timestamp for this file hash. Imperfect but deployable immediately. Native signing is Phase 2+. |
| Scanning requirement creates friction that kills adoption | Low | High | Scanning is optional — sender can proceed without it for lower-sensitivity transfers, with a warning. Only high-sensitivity data rooms mandate it. Basic tier (ClamAV) is free and near-instant. Scanning is a service layer, not a gate. |
| Open-source ClamAV has lower detection rates than commercial engines | Low | Medium | ClamAV is the baseline, not the ceiling. Commercial engines are available for users who need higher assurance. The architecture supports both simultaneously. |
| Multiple scanning services add latency | Medium | Medium | Services can run in parallel in the pod (CDR and AV simultaneously). Async mode for large files. Results are combined into a single attestation regardless of how many services ran. |

## Appendix: SG/Send Architecture Concepts Referenced

| Term | Definition |
|---|---|
| **PKI scanning chain** | Architecture where third-party scanning services produce results that are hash-bound to the specific file, attested by the pod, and encrypted for the receiver only. Sender cannot forge or read results. |
| **Pod** | A minimal, trust-isolated service running in its own isolated network. Has a single API endpoint ("process file"), can see cleartext transiently, calls external scanning services, and re-encrypts output before returning it. Deliberately separate from both the untrusted domain and the crossing point infrastructure. |
| **Zero-knowledge server** | The SG/Send server stores only encrypted blobs and has no access to encryption keys, file content, or scan results. Scanning must happen off-server, in the pod. |
| **Receiver-defined requirements** | Receivers publish what scanning, browser, and environment conditions they require before they will accept a file. Senders must satisfy requirements before the file is released. |
| **Dual authorization** | A two-step delivery flow: sender uploads but does not release; receiver signals readiness with environment report; sender reviews and releases. Scanning is embedded in this flow. |
| **`.sgroom`** | Portable data room format — a ZIP archive containing all room files, configuration, and audit logs. Can travel offline through supply chains and be scanned at each checkpoint. |
| **Trust web** | Reputation graph built through successful file transfers. Scan pass/fail results are a trust signal. Senders whose files consistently fail scanning see their trust score drop. |
| **DRU/MRU/ARU** | Daily/Monthly/Annual Recurring Usage — the North Star metric. Success = number of encrypted messages facilitated, directly or indirectly. |
| **Adapter** | A small module (200-500 lines) that translates between the pod's internal format and a specific scanning service's API. One adapter per scanning partner. |
