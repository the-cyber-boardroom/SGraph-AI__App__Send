## Public Vaults and Long-Lived Files

The time has come to start sharing vaults and folders publicly, using the gallery and folder views. This means introducing the concept of **long-lived vaults**: files and folders that persist, are tracked, and are referenced from the website.

### The Vault Registry

Create a page on the website (or a JSON/YAML file in the website repo) that lists the public vaults:

```yaml
public_vaults:
  - id: "abc123"
    token: "gleam-flint-6001"
    description: "SG/Send v0.2.x screenshots (historical)"
    created: "2026-03-27"
    type: "gallery"

  - id: "def456"
    token: "ocean-spark-4200"
    description: "Portugal trip photos"
    created: "2026-03-27"
    type: "gallery"

  - id: "ghi789"
    token: "river-stone-3300"
    description: "Product walkthrough videos"
    created: "2026-03-27"
    type: "folder"
```

This registry serves multiple purposes:
- Tracks which vaults should be preserved (not cleaned up by expiry)
- Provides the data for a "Public Galleries" page on the website
- Allows monitoring (are the public vaults still accessible?)
- Eventually moves vaults between S3 buckets or environments as needed

### Exposing Transfer ID from Simple Token

A practical need: when managing public vaults, the simple token (e.g. "gleam-flint-6001") is what gets shared, but the transfer ID is what appears in logs and monitoring. Add a utility (UI or CLI) that resolves a simple token to its transfer ID, so public vaults can be tracked in the logging infrastructure.

## The Try-Now Workflow

Currently, a new visitor to sgraph.ai must either pay for credits or join the Early Access Programme. There is no way to just try it. This needs to change.

### The Shared Token Approach

Create a token that is publicly available (shared on the website, on LinkedIn, in posts). This token has a limited budget (credits that are topped up daily). Anyone can use it to try the platform.

The workflow:

1. User visits sgraph.ai
2. User does not have a token
3. The page offers three options: **Enter an access token**, **Buy credits**, or **Try it free**
4. "Try it free" provides a shared evaluation token
5. The token has a soft budget (e.g. 10 uploads per day)
6. Client-side tracking via localStorage shows "You have X free tries remaining"
7. When the budget is exhausted, prompt to buy credits

### Why This Is Safe Now

The threat landscape is low. No attacks detected. No token abuse. The AppSec team can confirm this. With the monitoring and visibility infrastructure now in place (CloudFront logs, Grafana dashboards, the QA smoke tests), any abuse will be detected quickly.

The risk of sharing tokens publicly is outweighed by the growth benefit of letting anyone try the product immediately.

### Hard-Coded Sherpa Logic (Not Inline LLM)

The try-now flow should be a hard-coded state machine, not an LLM making decisions. The Sherpa-like guidance ("Hello! Would you like a free token to try SG/Send?") is pre-written copy, not generated. The budget logic is simple code:

```javascript
const tries = parseInt(localStorage.getItem('sg_free_tries') || '0');
if (tries < 10) {
  // Show the shared token
  localStorage.setItem('sg_free_tries', String(tries + 1));
} else {
  // Prompt to buy credits
}
```

This is the "LLM creates the code, code runs without LLM" principle from the dynamic credit allocation brief.

## UI Simplification

### Consolidate Steps

The current six-step upload flow (Upload, Delivery, Share mode, Confirm, Encrypt and Upload, Done) has redundancy. The delivery mode step and the share mode step could be combined into one "How do you want to share?" step. This reduces the flow to five or even four steps for most use cases.

Map out the consolidation. Do not implement yet (the v0.3.0 flow works). But design the simplified version for v0.3.1.

### The Try-Now Component

Build the token entry/acquisition page as a separate Web Component:

- Develop in isolation on tools.sgraph.ai
- Three states: enter existing token, buy credits, get free trial token
- Include the soft budget tracking (localStorage)
- Test with the QA infrastructure before integrating into the main flow

This follows the component-first principle: build it as a tool, test it, then integrate.
