## 2. What SG/Send Actually Is (Post v0.3.0)

### The one-liner
**Zero-knowledge encrypted file sharing with rich content viewing.**

### The expanded pitch
SG/Send encrypts files in your browser before upload using AES-256-GCM. The server only stores encrypted ciphertext — it never sees your data, your filenames, or your decryption key. Recipients get a link or a friendly token (like `royal-stamp-1474`) that lets them decrypt and view files directly in the browser.

### What makes it different from every other encrypted file sharing tool
Most encrypted file sharing tools are "upload → download." The recipient gets a blob and has to figure out what to do with it. SG/Send is different:

1. **You don't just download files — you experience them.** Markdown renders with full typography. PDFs open in a built-in viewer. Images display in a gallery. JSON shows with structure. Code renders with formatting. All in the browser, all zero-knowledge.

2. **You can browse entire folders.** Drop a folder, and the recipient gets a full file explorer — tree navigation, tabbed multi-pane layout, inline preview of every file type side by side. It's like having a desktop file browser inside the browser.

3. **Every transfer is transparent.** The recipient can inspect the zip structure, see the auto-generated metadata, verify the encryption, and check what the server stored vs what it never saw. Zero-knowledge isn't just claimed — it's verifiable.

4. **Sharing is human-friendly.** Instead of a 64-character hex string, you get `coral-equal-1774` — a token you can say over the phone, type from memory, or share in a message.

## 3. Target Audiences — Who Visits sgraph.ai

### Audience A: The Recipient (highest volume)
**Who they are:** Someone who received a link via email, chat, or verbally. They've never heard of SG/Send.
**What they need:** To understand what just happened (why they're on this site), trust that it's safe, and potentially become a sender themselves.
**Where they land:** Not on sgraph.ai — they land on send.sgraph.ai download pages. But sgraph.ai is one click away (the "sgraph.ai" link in the SG/Send header). The main website must support this conversion: recipient → curious visitor → new sender.

### Audience B: The Evaluator (decision-maker)
**Who they are:** CISO, IT manager, security-conscious professional evaluating whether to use SG/Send for their team or organisation.
**What they need:** Technical credibility (AES-256-GCM, Web Crypto API, zero-knowledge architecture), compliance positioning (GDPR, zero cookies, no tracking), and a clear explanation of the trust model.
**Where they land:** Directly on sgraph.ai, probably from a search or referral.

### Audience C: The Investor / Partner
**Who they are:** Someone evaluating SGraph as a company and SG/Send as a product.
**What they need:** Traction evidence, differentiation from competitors, market positioning, and a vision for where this goes.
**Where they land:** Directly on sgraph.ai, probably from a pitch deck link or meeting follow-up.

### Audience D: The Power User
**Who they are:** Someone who already uses SG/Send and wants to understand advanced features, check the latest version, or share the product with others.
**What they need:** Feature documentation, version history, and sharable pages they can send to colleagues.
**Where they land:** Directly on sgraph.ai or from send.sgraph.ai.

## 4. The Features to Showcase — Grounded in What Actually Works

Every feature listed below has been tested and verified in the v0.3.0 review. The descriptions include what the user actually sees, so the design team can create accurate representations.

### 4.1 Zero-Knowledge Encryption

**What it is:** Files are encrypted in the browser using AES-256-GCM (Web Crypto API) before upload. The decryption key never leaves the sender's device. The server stores only encrypted ciphertext.

**What the user sees:**
- During upload: "Encrypting..." with timing (e.g. "Encrypting: 1ms", "Uploading: 600ms")
- After upload: "What we stored about this transfer" — a transparency panel showing what the server has (file size, encrypted content) vs what it never saw (decryption key, raw IP, plaintext)
- On the download page: "Decrypted in 0.26s" — confirming decryption happened in the browser
- Zero cookies, zero tracking — verifiable by opening DevTools → Application → Cookies

**Key messaging:**
- "Your files are encrypted in your browser before upload"
- "The server never sees your data"
- "Zero cookies. Zero tracking. We cannot read your files."
- "Your files, your keys, your privacy."

**Visual opportunity:** An animation or diagram showing the encryption flow: File → Browser encrypts → Ciphertext uploaded → Server stores only ciphertext → Recipient's browser decrypts → File. Emphasise that the key never travels to the server.

### 4.2 Friendly Tokens

**What it is:** Instead of sharing a URL with a 64-character hash, users can share a human-readable token like `royal-stamp-1474`. The token derives both the transfer ID and the decryption key using SHA-256 and PBKDF2.

**What the user sees:**
- On the Done page: "Simple token: `voice-match-6116`" with a large, prominent display
- "This token derives both the transfer ID and decryption key"
- "Share this verbally or in a message — easy to remember and type"
- "320 words × 320 words × 10,000 = ~1 billion combinations"
- Copy button, QR code, "Open in new tab" to test the recipient experience

**Key messaging:**
- "Share a token, not a link"
- "Easy to say over the phone, type from memory, or share in a message"
- "One token. One billion possible combinations."
- The contrast: `royal-stamp-1474` vs `#88bff7cba6a2/NLEWAl4Q8iGCn2sRJmZh3ugUIWzQGWOqerM_ivY-TxI`

**Visual opportunity:** Side-by-side comparison of a friendly token vs a traditional crypto URL. Show someone speaking a token over a phone call, typing it on a phone, sending it in a text message.

### 4.3 Rich Content Viewing (The Gallery)

**What it is:** When a recipient opens a shared link to a multi-file transfer, they see a gallery grid with type-aware thumbnails — not just a list of filenames. Images show image thumbnails. PDFs show the first page. Markdown shows rendered text preview. Each card has a coloured type badge (JPG blue, PDF red, MD green, etc.).

**What the user sees:**
- Gallery header: filename, "GALLERY" badge, file count, size, "✓ Decrypted"
- Three view modes: compact (many columns), grid (4 columns with names), large (2 columns, detailed)
- Click any card → lightbox with full content: images at full resolution, markdown rendered with typography, PDFs in a built-in viewer
- Navigation between files with arrow keys or click
- Save individual files, save the entire archive, copy link, email, print
- Switch to folder view at any time

**Key messaging:**
- "Don't just download — browse, read, and explore"
- "Every file type rendered beautifully"
- "Gallery. Lightbox. Folder browser. Print-ready output."

**Visual opportunity:** A gallery grid showing a mix of file types — a slide deck, a markdown brief, an infographic, a data file — each with its type badge and thumbnail. This is the product's most visual feature and should be front-and-centre on the website.

### 4.4 The Folder Browser (sg-layout)

**What it is:** For multi-file transfers, recipients get a full file explorer with a tree view on the left, tabbed content panels in the centre and right, and inline rendering of every file type. Files can be opened side by side. It's like Finder or VS Code's explorer — but zero-knowledge, in the browser.

**What the user sees:**
- Left panel: folder tree with expand/collapse, file count badges, file type icons
- Centre/right panels: opened files in tabs, each rendered inline (markdown, images, PDFs, JSON, code, text)
- sg-layout: drag to resize panes, open multiple files simultaneously, compare side by side
- Tab bar with Share, Info, and file tabs
- Action buttons: Save Locally, Copy Link, Email, Print, Gallery View switcher

**Key messaging:**
- "A file browser in the browser"
- "Open, compare, and explore — without downloading"
- "See everything the sender shared, exactly as they organised it"

**Visual opportunity:** A screenshot of the folder browser with 3-4 files open side by side — a markdown document, a PDF, an image, and a JSON file. Show the tree on the left with folders expanded. This demonstrates the product's depth.

### 4.5 Print-Ready Output (SgPrint)

**What it is:** Any markdown file can be printed as a beautiful, branded A4 document. The output includes: SG/Send logo header with teal accent line, filename, properly styled typography (headings, lists, tables, code blocks, blockquotes), and footer.

**What the user sees:**
- Click the print icon → a new tab opens with a print preview
- Toolbar: "SG/Send | Print preview for **filename.md** | Print / Save PDF | Close"
- A4 page preview with professional layout: DM Sans font, proper headings, styled tables with borders and header backgrounds, coloured code blocks, blockquotes with left border
- Print or Save as PDF directly from the preview

**Key messaging:**
- "From encrypted markdown to professional PDF in one click"
- "Branded, beautiful, print-ready"

**Visual opportunity:** Before/after — raw markdown text on one side, the SgPrint A4 output on the other. The quality difference is striking.

### 4.6 Three Share Modes

**What it is:** After encrypting, the sender chooses how to share:
1. **Simple token** — a friendly word-word-number token. Easiest to share, single token derives everything.
2. **Combined link** — one URL with the decryption key embedded. Simplest for the recipient — one click to decrypt.
3. **Link + key separate** — the link and decryption key travel through different channels (e.g. link via email, key via chat). More secure — neither piece works alone.

**What the user sees:**
- Step 3 (Share mode): three cards with icons, descriptions, and "Best for:" hints
- Done page adapts to the chosen mode: Simple token shows the token prominently with QR code; Combined link shows the full URL; Link + key separate shows two sections ("1. Send this link" / "2. Send this key separately")

**Key messaging:**
- "Choose your security level"
- "From verbal tokens to split-channel delivery"
- "Simple for friends, secure for confidential documents"

**Visual opportunity:** The three cards from Step 3, or the three Done page variants side by side showing how each mode presents the sharing information differently.

### 4.7 Upload Wizard (6-Step Flow)

**What it is:** A guided wizard that takes the user from file selection to sharing in 6 steps: Upload → Delivery → Share mode → Confirm → Encrypt & Upload → Done.

**What the user sees:**
- Step indicator showing progress through all 6 steps
- Drag-and-drop or browse for files/folders
- File/Text mode toggle (encrypt text notes, not just files)
- Delivery mode selection (Gallery / Folder view / Download)
- Share mode selection (Simple token / Combined link / Link + key separate)
- Confirmation page with summary of all choices
- Encryption progress with rotating privacy/feature messages
- Done page with QR code, copy buttons, email, open in new tab

**Key messaging:**
- "Drop. Encrypt. Share."
- "Six steps from file to secure link"

**Visual opportunity:** The step indicator showing the flow, or an animated walkthrough of the wizard. The confirmation page (Step 4) is a particularly clean summary of the entire flow.

### 4.8 Multi-Language Support

**What it is:** The UI supports multiple languages. Currently the header and some elements translate (PT-PT was tested), with the framework in place for full localisation.

**What the user sees:**
- Language selector in the header (flag icon + language name)
- Header tagline translates (e.g. "Os seus ficheiros, as suas chaves, a sua privacidade")
- URL structure supports locales (`/en-gb/`, `/pt-pt/`, etc.)

**Key messaging:**
- "Available in 17 languages"
- "Same zero-knowledge encryption everywhere"

### 4.9 Transparency Panel

**What it is:** After upload, a panel shows exactly what the server stored — and crucially, what it didn't. This is the trust mechanism.

**What the user sees:**
```
What we stored about this transfer
Stored by the server
  File size 73 B
  Stored encrypted (only you can read it)
  File name Encrypted (only you can read it)
  File content Encrypted (only you can read it)
Never sent to the server
  Decryption key NOT stored (only you have it)
  Raw IP address NOT stored
That's everything. Nothing else is captured.
```

**Key messaging:**
- "We show you exactly what we know — and what we don't"
- "Verifiable privacy, not just promises"

**Visual opportunity:** This panel as a clean, styled component — two columns, "Stored" vs "Never sent", with icons. This is unique to SG/Send and should be highlighted.

## 6. Design Direction — What the Website Should Feel Like

### Brand identity to carry forward
- **Dark theme** with teal (#4ECDC4) accent — this is already established in the product
- **DM Sans** for body text, clean and modern
- **The lightning bolt icon** (⚡) as the SG/Send logo mark
- **"Your files, your keys, your privacy"** as tagline

### Design principles
1. **Show the product, don't describe it.** Screenshots and interactive demos > paragraphs of text. The product is visual — the gallery, the folder browser, the lightbox — and should sell itself.
2. **Security without austerity.** Most security products look cold and corporate. SG/Send's dark theme with teal accents is warm and modern. The website should continue this — trustworthy but not intimidating.
3. **Progressive disclosure.** Homepage: the story in 30 seconds. Features page: the details. Security page: the technical depth. Don't front-load everything.
4. **The recipient journey is paramount.** A large percentage of website visitors will be people who just received a file and clicked "sgraph.ai" in the header. The website needs to immediately make them feel confident about what just happened to their file, AND make them want to send their own.

### Responsive / mobile considerations
- The product works on mobile (Web Components, responsive CSS)
- The website must work on mobile too — many recipients will open links on phones
- QR codes on the Done page are specifically for phone-to-phone sharing

## 7. The Demo Flow — How to Show the Product

The website should include an interactive or video demo. Based on the review session, the most compelling demo flow is:

### Quick demo (30 seconds)
1. Drop a file → show it encrypting → show the friendly token
2. Open the link → show the recipient seeing the file in the viewer
3. Punchline: "The server never saw your file."

### Full demo (2 minutes)
1. Drop a folder with mixed content (markdown, images, PDFs)
2. Watch the wizard: delivery (gallery mode), share (simple token), confirm, encrypt
3. Note the thumbnail generation, the timing breakdown, the transparency panel
4. Open the gallery: show the type badges, click through the lightbox
5. Switch to folder view: show the tree, open multiple files, use sg-layout
6. Print a markdown file: show the SgPrint output
7. Inspect the _gallery folder: show the manifest, metadata, thumbnails
8. Punchline: "All of this — encrypted end-to-end. The server stored only ciphertext."

### The "conversion moment" demo
1. Show a recipient opening a link for the first time
2. They see the loading page (which should have privacy messaging, per D-009)
3. The file decrypts in their browser
4. They browse the content in the gallery or folder view
5. They click "Send another file" → now they're a sender
6. Punchline: "Every recipient is a potential sender."

## 9. Technical Notes for Dev/Architect Review

### The website stack
The sgraph.ai website should be:
- **Static** — HTML/CSS/JS, deployable to S3/CloudFront (same pattern as send.sgraph.ai)
- **Fast** — no framework overhead, minimal JS
- **Responsive** — mobile-first
- **Multi-language ready** — same i18n framework as send.sgraph.ai

### Links between sgraph.ai and send.sgraph.ai
- sgraph.ai "Send a file" CTA → send.sgraph.ai/en-gb/
- send.sgraph.ai "sgraph.ai" header link → sgraph.ai
- The two sites should feel like the same brand — same dark theme, same fonts, same teal accent
- Consider whether sgraph.ai and send.sgraph.ai should eventually merge, or remain separate (marketing site vs product)

### SEO considerations
- Pages for each feature (gallery, folder browser, encryption, tokens) for long-tail search
- Security page for "zero-knowledge file sharing", "encrypted file transfer", "AES-256 file sharing"
- Schema markup for software product

### Analytics
- The website can have standard analytics (unlike the product which is zero-cookie)
- Track: homepage → send.sgraph.ai conversion, feature page engagement, security page depth
- Note: "Claude products are ad-free" — no ads on the website

## 10. What NOT to Claim

**Reality document principle:** Only claim what exists. The following are NOT yet shipped:
- One-time secrets / view-limited secrets (planned, not built)
- Full multi-language support (framework exists, most strings still English)
- User accounts / settings page (planned, not built)
- API access (exists but not documented for external use)

**Do NOT describe these as features.** They can be mentioned as "coming soon" or in a roadmap section, clearly labelled as planned.
