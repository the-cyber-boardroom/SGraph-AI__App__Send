## The Urgent Need

The product is live. Users need to be contacted. New features need announcing. Investors need updates. Early Access users need engagement. Paid users need nurturing. This needs to happen today, not next week.

The Sherpa needs its own repo, its own CLI, and its own tools.

## The Principle: API First, But CLI Now

Everything should be API-first (classes and methods that can later be exposed via FastAPI). But deploying a full serverless FastAPI is not the priority. The priority is a CLI that works right now, today, to send emails and manage outreach.

The architecture:

```
Sherpa CLI (immediate)
  │
  ├── Uses classes and methods (reusable)
  │
  └── Later: FastAPI wraps the same classes
        │
        └── Later: Lambda deploys the FastAPI
```

Build the classes first. The CLI invokes them. The FastAPI comes later and wraps the same classes. No code rewrite needed for the upgrade path.

## The Sherpa Repo

Create a dedicated repo for the Sherpa with:

```
sherpa/
  cli/
    sg-sherpa             CLI entry point
    commands/
      email.py            Read, compose, send, follow up
      users.py            User list management, segments

      inbox.py            WorkMail inbox interaction
  services/
    workmail.py           WorkMail API integration (read/send)
    ses.py                SES integration (bulk send, future)

  templates/

    investor-update.html  Investor communication
  data/
    users.json            User list (from Stripe, EAP, manual)
    segments.json         User segments (paid, free, EAP, investor)
    sent.json             Send history (who received what, when)
```

## CLI Commands

### Email Management

```bash
# Read inbox
sg-sherpa inbox list                      # List recent emails
sg-sherpa inbox read <id>                 # Read specific email
sg-sherpa inbox unread                    # List unread

# Compose and send
sg-sherpa email compose --template new-features --to paid-users
sg-sherpa email preview <draft-id>        # Preview before sending
sg-sherpa email send <draft-id>           # Send (after preview/approval)
sg-sherpa email send-all --template new-features --segment paid-users

# Follow up
sg-sherpa email follow-up --segment no-response --days 7
```

### User Management

```bash
# User lists
sg-sherpa users list                      # All users
sg-sherpa users list --segment paid       # Paid users only
sg-sherpa users list --segment eap        # Early Access Programme
sg-sherpa users add <email> --segment eap # Add user
sg-sherpa users import --stripe           # Import from Stripe
sg-sherpa users export                    # Export user list
```

### Outreach Campaigns

```bash
# Campaign management
sg-sherpa campaign create "v0.3.0 Launch" --template new-features --segment all
sg-sherpa campaign preview "v0.3.0 Launch"
sg-sherpa campaign send "v0.3.0 Launch"
sg-sherpa campaign status "v0.3.0 Launch"  # Sent, opened, bounced
```

## Email Infrastructure: WorkMail Now, SES Later

### Phase 1: WorkMail (Now)

Use WorkMail as both inbox and outbox. Replace the n8n integration (which was messy) with direct WorkMail API calls via boto3:

- Read inbox (incoming emails, replies from users)
- Send emails (individual and batch)
- Track sent history

WorkMail gives us a real email address, inbox management, and the ability to receive replies. This is important for user communication (users reply to announcements).

### Phase 2: SES for Bulk (Later)

When volume increases, move bulk sending to SES (cheaper, higher throughput, better delivery tracking). Keep WorkMail as the inbox for receiving replies.

### Phase 3: SNS for Inbound (Exploration)

Explore whether SNS can replace or supplement WorkMail for receiving emails. This would simplify the architecture (SNS triggers a Lambda on inbound email) but needs evaluation.

## Vault Communication

The Sherpa CLI should support vault-based communication:

```bash
# Push outreach materials to a vault
sg-sherpa vault push --campaign "v0.3.0 Launch"

# Pull instructions from a vault (from the human)
sg-sherpa vault pull --inbox
```

This enables the two-way vault communication pattern: the human pushes instructions to a vault, the Sherpa pulls and executes, the Sherpa pushes results back.
