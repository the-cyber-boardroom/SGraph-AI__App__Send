## The Evolution

The briefing system has organically evolved into a project management structure. We have briefs per agent, dev briefs per feature, teams inside those briefs, and issue files from the QA workflow. This memo formalises and expands that into a full agentic project management approach.

## The Principle: All Communication via Patches

For the next website version and all future projects, enforce that **no agent has write access to any repo.** Everything communicates using patches. This means:

- The dev team produces patches for the main repo
- The QA team produces patches for the QA repo
- The design team produces mockups and assets as patches
- The Sherpa produces content as patches
- Each team reads the other teams' repos (read-only clones) to coordinate

The vault is the communication channel between teams. Patches are the language. This is the Dev-to-QA workflow brief extended to all teams.

Let us see if we can make an entire website redesign happen with zero direct repo access by any agent. That would be a powerful demonstration of the workflow.

## Temporal Analysis and Simulation

### Task Dependency Mapping

For every task, map:
- What it depends on (which tasks must complete first)
- What depends on it (which tasks are blocked until this completes)
- Which team owns it
- Estimated effort

This produces a dependency graph. The graph reveals which tasks can run in parallel (no dependencies between them) and which are on the critical path (everything waits for them).

### Simulation: "Assume This Is Done"

A powerful technique used successfully in past projects: simulate task completion.

For each task, ask: "Assume this task is done. Does the next team have everything they need to proceed?" This reveals hidden dependencies and missing handover artifacts before any work begins.

Example:
- "Assume the design mockups are done. Does the dev team have enough detail to implement?"
- "Assume the upload component is refactored. Does the QA team have updated test instructions?"
- "Assume the try-now workflow is built. Does the Sherpa have the copy for the guidance text?"

Every time this simulation has been done, it has revealed gaps that would otherwise only surface during implementation.

### Teams Across Time

Map what each team looks like at different phases:

**Phase 1: Design and Planning**
- Design team: active (mockups, flows, architecture diagrams)
- Content team: active (copy, feature descriptions)
- Dev team: reading briefs, preparing components
- QA team: writing test plans from mockups

**Phase 2: Implementation**
- Design team: reviewing, iterating based on dev feedback
- Dev team: active (building components, producing patches)
- QA team: writing tests alongside dev (Dev-to-QA patch workflow)
- Content team: updating copy based on what is actually built

**Phase 3: Integration and Launch**
- QA team: full regression, smoke tests, sign-off
- Dev team: fixing issues from QA
- Content team: final copy review
- All teams: release documentation from test artifacts

Each phase has different team compositions and handover points. Mapping this in advance prevents the "everyone is blocked waiting for everyone else" problem.

## Component-First Design

Before building pages, map the components needed:

- Create a folder per page (or per use case)
- Each folder contains: wireframes, architecture diagrams, content requirements, component list
- Identify which components already exist and which need building
- Build new components in isolation (on tools.sgraph.ai) before integration

By the time implementation starts, the brief should be so detailed that the dev team has a rock-solid spec. This is the Villager principle: invest in understanding before investing in code.

## Mockups and AI Design Tools

Explore using current AI design tools (Figma AI, v0.dev, or similar) to generate mockups and layout proposals. These are not final designs. They are conversation starters that the Design team, Sherpa, and Architect can react to and iterate on.

The mockups live in vaults (or folders) alongside the briefs. The historical progression (mockup v1, feedback, mockup v2, final) is preserved. Every design decision has provenance.

## Use the Technology to Build the Technology

The website redesign should be a showcase of SG/Send's own capabilities:

- Briefs and mockups shared via vaults
- Design reviews via galleries (upload mockups, share gallery link, get feedback)
- Historical versions preserved as long-lived vaults
- QA documentation auto-generated from tests
- Release notes assembled from issue files

Every step of building the new website demonstrates why someone would use SG/Send.
