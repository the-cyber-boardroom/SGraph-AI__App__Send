## The Short Answer

There is no `create-named-token` endpoint. There is no server-side token API. **Simple tokens are entirely client-side.** The server does not know that simple tokens exist.

A simple token is a human-readable string (like `royal-stamp-1234`) that deterministically derives both the transfer ID and the decryption key using standard cryptographic functions. The derivation happens in the browser (or, for the CLI, in Python). The server only ever sees the derived transfer ID — a 12-character hex string — and has no way to know whether it came from a simple token, a raw URL hash, or any other source.

---

## How Simple Tokens Work

### Token Format

```
word-word-NNNN
```

Two words from a 320-word wordlist, a hyphen, and a 4-digit number. Examples: `royal-stamp-1234`, `coral-equal-1234`, `onset-hazel-2694`, `chain-eagle-0669`.

**Entropy:** 320 × 320 × 10,000 = 1,024,000,000 (~1 billion combinations, ~30 bits of entropy).

### Token → Transfer ID (how the server finds the file)

```
transfer_id = SHA-256(token)[0:12]    // first 12 hex characters
```

Pseudocode:
```python
import hashlib

token = "royal-stamp-1234"
hash_hex = hashlib.sha256(token.encode('utf-8')).hexdigest()
transfer_id = hash_hex[:12]   # e.g. "88bff7cba6a2"
```

The transfer ID is what gets sent to the server's API: `GET /api/transfers/info/{transfer_id}`. The server uses this to look up the encrypted file. The server sees `88bff7cba6a2`, not `royal-stamp-1234`.

### Token → Decryption Key (how the file gets decrypted)

```
key = PBKDF2-SHA256(token, salt="sgraph-send-v1", iterations=600000, keyLength=32)
```

Pseudocode:
```python
import hashlib

token = "royal-stamp-1234"
key = hashlib.pbkdf2_hmac(
    'sha256',
    token.encode('utf-8'),
    b'sgraph-send-v1',    # salt — exact value from friendly-crypto.js
    iterations=600000,     # exact value from friendly-crypto.js
    dklen=32              # 256-bit key for AES-256-GCM
)
```

The resulting 32-byte key is then used as the AES-256-GCM decryption key — the same key that was used to encrypt the file during upload.

**These parameters are confirmed from source code (`friendly-crypto.js`):**
- **Salt:** `sgraph-send-v1` (string, UTF-8 encoded)
- **Iterations:** `600,000`
- **Hash:** `SHA-256`
- **Key length:** 256 bits (32 bytes)
- **Algorithm:** `AES-GCM`

### Token → URL

The token goes in the URL fragment (after `#`), which is never sent to the server by the browser:

```
https://send.sgraph.ai/en-gb/gallery/#royal-stamp-1234
https://send.sgraph.ai/en-gb/browse/#royal-stamp-1234
https://send.sgraph.ai/en-gb/download/#royal-stamp-1234
```

The route (`/gallery/`, `/browse/`, `/download/`) determines the viewing experience. The fragment `#royal-stamp-1234` is processed entirely client-side.

For comparison, the non-token URL format uses the raw transfer ID and base64-encoded key:

```
https://send.sgraph.ai/en-gb/browse/#88bff7cba6a2/NLEWAl4Q8iGCn2sRJmZh3ugUIWzQGWOqerM_ivY-TxI
```

The simple token replaces this entire fragment with something a human can remember and type.

## The Upload Flow (Token Generation)

When a user uploads a file with "Simple token" share mode selected, the browser:

1. **Generates a random token** — picks two random words from the 320-word wordlist + a random 4-digit number → `royal-stamp-1234`
2. **Derives the transfer ID** — `SHA-256("royal-stamp-1234")` → first 12 hex chars → `88bff7cba6a2`
3. **Derives the AES key** — `PBKDF2("royal-stamp-1234")` → 32-byte key
4. **Encrypts the file** — uses the derived AES-256-GCM key to encrypt the file content in the browser
5. **Uploads the ciphertext** — sends the encrypted blob to `POST /api/transfers/` with the derived transfer ID (`88bff7cba6a2`)
6. **Displays the token** — shows `royal-stamp-1234` on the Done page with Copy button and QR code

**The server receives:** transfer ID `88bff7cba6a2` + encrypted ciphertext. It stores these. It has no idea that `royal-stamp-1234` exists.

---

## The Download Flow (Token Resolution)

When a recipient opens `https://send.sgraph.ai/en-gb/browse/#royal-stamp-1234`:

1. **Browser reads the fragment** — `royal-stamp-1234` (never sent to server)
2. **Detects it's a friendly token** — matches pattern `word-word-NNNN`
3. **Derives the transfer ID** — `SHA-256("royal-stamp-1234")` → `88bff7cba6a2`
4. **Derives the AES key** — `PBKDF2("royal-stamp-1234")` → 32-byte key
5. **Fetches the encrypted file** — `GET /api/transfers/info/88bff7cba6a2` → server returns ciphertext
6. **Decrypts in the browser** — uses the derived key to decrypt → original file

**The server sees:** `GET /api/transfers/info/88bff7cba6a2`. It returns the encrypted blob. It never sees the token or the decryption key.

## What the CLI Needs to Implement

### To CREATE a transfer with a simple token:

```python
# 1. Generate or accept a token
token = generate_friendly_token()   # or accept from user: --token royal-stamp-1234

# 2. Derive transfer ID
transfer_id = sha256(token.encode())[:12]  # hex — first 6 bytes = 12 hex chars

# 3. Derive encryption key
key = pbkdf2_hmac('sha256', token.encode(), b'sgraph-send-v1', 600000, 32)

# 4. Encrypt the file with this key (AES-256-GCM)
ciphertext = aes_gcm_encrypt(plaintext, key)

# 5. Upload to the API
POST /api/transfers/ with transfer_id and ciphertext

# 6. Return the token to the user
print(f"Token: {token}")
print(f"URL: https://send.sgraph.ai/en-gb/browse/#{token}")
```

### To DOWNLOAD a transfer using a simple token:

```python
# 1. Accept token from user
token = "royal-stamp-1234"

# 2. Derive transfer ID
transfer_id = sha256(token.encode())[:12]

# 3. Derive decryption key
key = pbkdf2_hmac('sha256', token.encode(), b'sgraph-send-v1', 600000, 32)

# 4. Fetch encrypted file from API
GET /api/transfers/info/{transfer_id}

# 5. Decrypt
plaintext = aes_gcm_decrypt(ciphertext, key)
```

### There is NO server-side token API needed because:

- Token generation is random word selection — no server involved
- Token → transfer ID is a deterministic hash — no server involved
- Token → key is a deterministic KDF — no server involved
- The upload API accepts any transfer ID (it doesn't care how it was generated)
- The download API looks up by transfer ID (it doesn't care that it came from a token)

The server's API surface is unchanged: `POST /api/transfers/` (upload) and `GET /api/transfers/info/{id}` (download). Simple tokens are a client-side abstraction over the same API.

## Security Considerations

1. **Token entropy is ~30 bits.** This is designed for usability (speakable, typeable), not for maximum security. For higher security, users should use the "Combined link" or "Link + key separate" modes which use full 256-bit keys.

2. **PKI is the proper long-term solution.** Simple tokens are a pragmatic usability feature for the current symmetric-key model. The planned PKI integration (already architected — see the PKI briefs in the infographics repo) solves the key exchange problem properly: the sender encrypts to the recipient's public key, the recipient decrypts with their private key. No shared secret needs to travel at all — not as a token, not as a URL fragment, not in an email. PKI eliminates the entire class of "secret in the link" concerns. Simple tokens will remain as a convenience layer for anonymous/ad-hoc sharing where the recipient doesn't have a PKI identity, but for authenticated use cases (agents, teams, organisations), PKI is the correct answer.

3. **The token must never be sent to the server.** The v0.2.14 security fix patched a bug where the UI was accidentally sending `GET /api/transfers/info/royal-stamp-1234` (the raw token as transfer ID) before the overlay code had a chance to derive the real transfer ID. The CLI must ensure it only sends the derived transfer ID, never the raw token.

4. **Brute force resistance.** ~1 billion combinations means an attacker who knows a transfer exists would need to try ~500 million tokens on average. The 600,000 PBKDF2 iterations add significant computational cost per attempt — at ~600K iterations per guess, brute-forcing becomes expensive even with dedicated hardware. For sensitive content, recommend using the full-key modes instead (or PKI when available).

5. **Token pattern detection.** The pattern `word-word-NNNN` is how the client distinguishes a friendly token from a raw `transferId/base64key` fragment. The CLI should use the same regex: `/^[a-z]+-[a-z]+-\d{4}$/`.

## Reference Implementation (JavaScript — authoritative source)

This is the complete `friendly-crypto.js` from the SG/Send UI codebase. The CLI must produce identical outputs for the same input token.

```javascript
const FriendlyCrypto = {

    SALT: 'sgraph-send-v1',
    ITERATIONS: 600000,

    /** Check if a string looks like a friendly token (word-word-NNNN) */
    isFriendlyToken(str) {
        if (!str) return false;
        return /^[a-z]+-[a-z]+-\d{4}$/.test(str);
    },

    /** Derive a deterministic 12-char hex transfer ID from a friendly token.
     *  Uses SHA-256(token) → first 12 hex chars. */
    async deriveTransferId(friendlyToken) {
        const enc = new TextEncoder();
        const hash = await crypto.subtle.digest('SHA-256', enc.encode(friendlyToken));
        const bytes = new Uint8Array(hash);
        let hex = '';
        for (let i = 0; i < 6; i++) {
            hex += bytes[i].toString(16).padStart(2, '0');
        }
        return hex;
    },

    /** Derive an AES-256-GCM key from a friendly token via PBKDF2.
     *  Salt: 'sgraph-send-v1', iterations: 600000, hash: SHA-256. */
    async deriveKey(friendlyToken) {
        const enc = new TextEncoder();
        const material = await crypto.subtle.importKey(
            'raw', enc.encode(friendlyToken), 'PBKDF2', false, ['deriveKey']
        );
        return crypto.subtle.deriveKey(
            { name: 'PBKDF2', salt: enc.encode(this.SALT),
              iterations: this.ITERATIONS, hash: 'SHA-256' },
            material,
            { name: 'AES-GCM', length: 256 },
            true,
            ['encrypt', 'decrypt']
        );
    }
};
```

### Python Equivalent (for the CLI)

```python
import hashlib
import os

# Constants — must match FriendlyCrypto exactly
SALT = b'sgraph-send-v1'
ITERATIONS = 600000
TOKEN_PATTERN = r'^[a-z]+-[a-z]+-\d{4}$'

# The wordlist — must be the same 320 words as friendly-crypto.js
# Copy WORDLIST from the JS source into a Python list
WORDLIST = [...]  # 320 words from friendly-crypto.js

def generate_token() -> str:
    """Generate a random friendly token: word-word-NNNN"""
    w1 = WORDLIST[int.from_bytes(os.urandom(2)) % len(WORDLIST)]
    w2 = WORDLIST[int.from_bytes(os.urandom(2)) % len(WORDLIST)]
    num = int.from_bytes(os.urandom(2)) % 10000
    return f"{w1}-{w2}-{num:04d}"

def is_friendly_token(s: str) -> bool:
    """Check if string matches word-word-NNNN pattern"""
    import re
    return bool(re.match(TOKEN_PATTERN, s))

def derive_transfer_id(token: str) -> str:
    """SHA-256(token) → first 12 hex characters"""
    h = hashlib.sha256(token.encode('utf-8')).hexdigest()
    return h[:12]

def derive_key(token: str) -> bytes:
    """PBKDF2-SHA256(token, salt, 600000 iterations) → 32-byte AES key"""
    return hashlib.pbkdf2_hmac(
        'sha256',
        token.encode('utf-8'),
        SALT,
        ITERATIONS,
        dklen=32
    )

# Verification — run this against the browser to confirm interop:

# print(derive_key(token).hex())    # should match browser's exported key bytes
```

### Interop Test

To verify the CLI and browser produce identical outputs, run this test:

1. Pick a token: `test-token-1234`
2. In the browser console: `await FriendlyCrypto.deriveTransferId('test-token-1234')` → note the 12-char hex
3. In Python: `derive_transfer_id('test-token-1234')` → must produce the same hex

## Wordlist and Custom Tokens

The browser UI wordlist is embedded in `friendly-crypto.js` and contains 320 lowercase English words used for **generating** random tokens. But the wordlist is irrelevant for **resolving** tokens — the derivation functions (SHA-256, PBKDF2) operate on the raw token string, not on wordlist indices.

**This means any string matching `word-word-NNNN` works.** The browser checks the regex pattern, not whether the words are in its list. A CLI-generated token with completely different words resolves identically in the browser. A user-provided custom token works too.

### The CLI should use a different wordlist for generation

When the CLI generates a random token, it should draw from its own 320-word list. This adds entropy — an attacker who doesn't know which client generated the token can't narrow down the wordlist, effectively doubling the search space to ~2 billion for a blind attack.

### The CLI should also accept user-provided tokens

Since the whole point of the token is that it deterministically derives the transfer ID and decryption key, the CLI should let the user provide their own token:

```bash
# Random token from CLI wordlist
sgit send myfile.pdf
# → Token: forge-anvil-3847

# User-provided token
sgit send myfile.pdf --token "budget-report-2026"
# → Token: budget-report-2026

# User can pick any word-word-NNNN they want
sgit send myfile.pdf --token "patient-records-0042"
# → Token: patient-records-0042
```

This is powerful because it lets users create **memorable, meaningful tokens** — `board-meeting-0315` for a March 15th board pack, `audit-draft-0001` for a first audit draft, `project-alpha-2026` for a project delivery. The token is just an input to SHA-256 and PBKDF2 — the crypto doesn't care what the words mean.

**Validation rule:** The CLI should validate that user-provided tokens match the `word-word-NNNN` pattern (lowercase alpha, hyphen, lowercase alpha, hyphen, exactly 4 digits) so that both browser and CLI can detect them consistently. Beyond the pattern check, any words are valid.

### Wordlist implementation guidance

- Choose 320 common, unambiguous, easy-to-type English words that don't overlap with the browser wordlist (for the entropy benefit)
- Keep them all lowercase, ASCII-only, no hyphens within words
- The token format remains `word-word-NNNN`
- Different wordlists for generation; identical derivation logic for resolution
