## 1. The Big Picture

SG/Send is not a standalone file sharing product. It's the **first product** in a platform that is building the cryptographic infrastructure for human and AI collaboration. The platform has five layers, each building on the one before:

1. **SG/Send** — Zero-knowledge encrypted file sharing (shipping now, v0.3.0)
2. **SGit** — Encrypted, PKI-signed Git. The foundational engine that powers everything below. Has its own site ([sgit.ai](https://sgit.ai)), PyPI package (`pip install sgit-ai`), and full CLI. This is real, working software.
3. **SG-Vault** — Zero-knowledge version control built on SGit — the browser-based UI for vault operations
4. **Data Rooms** — On-demand encrypted workspaces for collaboration
5. **PKI Infrastructure** — Machine and human identity, chain of trust, signed messages
6. **Agentic Communication Plane** — Secure agent-to-agent and human-to-agent messaging

**Important:** SGit is not a future product — it's the working engine underneath SG-Vault and the sg-send-cli toolchain. It has a PyPI package, a GitHub organisation, a designed website with terminal demos, and a full CLI (init, clone, commit, push, pull, branch, merge, keygen, sign, verify, encrypt, decrypt, inspect). The website should reflect that SG/Send and SGit are both live products, with the rest of the platform building on top.

## 2. SGit-AI: The Encrypted Git Engine

**What it is:** A lightweight, PKI-signed and encrypted Git. Not a full Git reimplementation — a convention and set of light implementations using content-addressed storage, tree structures, commits, branches, merges, and remotes. All content is encrypted. The server stores opaque blobs. File names, folder structure, and content are all encrypted.

**The one-line pitch:** "Git, but encrypted and signed by default. Runs anywhere. No SSH."

**Status:** Live. PyPI package published (`pip install sgit-ai`). Own GitHub organisation. Website designed and specced at [sgit.ai](https://sgit.ai). Full CLI with vault, PKI, and inspection commands. This is the technology that powers SG-Vault and the sg-send-cli toolchain.

**Why it matters:**
- **No SSH required.** Pure Python and browser-based JavaScript. Runs in Lambda functions, CI pipelines, browser tabs — anywhere Python or JS runs. Git in Lambda has always been painful because Lambda doesn't have SSH. SGit doesn't need it.
- **PKI-signed by default.** Every commit is signed by the branch key. Provenance is mandatory, not optional. Stronger than Git's optional GPG signing.
- **Encrypted by default.** All content encrypted with AES-256-GCM. Server stores only ciphertext.
- **Agentic-first.** The primary audience is people building agentic workflows: "Give your AI agent its own identity, not your credentials. Agents commit to their own branches, signed with their own keys. No SSH, no Git credentials, no overpowered tokens."

**CLI commands (shipped):**
- Vault: `init`, `clone`, `commit`, `push`, `pull`, `branch`, `merge`
- PKI: `keygen`, `sign`, `verify`, `encrypt`, `decrypt`, `contacts`
- Inspection: `log`, `fsck`, `inspect`, `cat-object`, `inspect-stats`

**Crypto pipeline:** `Vault Key → PBKDF2 (600k iterations) → Master Key (256-bit) → HKDF (SHA-256) → Per-File Key → AES-256-GCM → Ciphertext`

**Web Crypto interop:** The same crypto operations work identically in Python (CLI) and JavaScript (browser). This is what enables SG-Vault to be the browser UI for SGit vaults — the keys derived in the CLI produce identical ciphertext to keys derived in the browser.

## 3. SG-Vault: Zero-Knowledge Version Control

**What it is:** The browser-based UI for SGit vaults. Git-like version control with integrated encryption, PKI signing, and provenance tracking — powered by SGit's Python CLI on the backend and Web Crypto API in the browser. Every commit is cryptographically signed. Every file is encrypted. The server stores only ciphertext. SGit provides the engine; SG-Vault provides the browser experience.

## 12. How to Use This for the Website

**For the homepage:** The platform story should appear as a "Where we're going" section below the current SG/Send features. Show the 6-layer stack (Send → SGit → Vault → Data Rooms → PKI → Agentic Plane) as a visual roadmap. SG/Send and SGit are "live now", the rest is "coming". Keep it brief — one line per layer.

**For the ecosystem navigation:** The sgraph.ai website should link to both send.sgraph.ai and sgit.ai as live products, plus vault.sgraph.ai, tools.sgraph.ai, and workspace.sgraph.ai as the broader ecosystem. The shared header component (`<sg-site-header>`) already supports this pattern.

**For a dedicated "Platform" or "Vision" page:** Use the infographics and briefs linked above as source material. The Designer agent can pull visual inspiration from the infographics. The Sherpa agent can use the markdown briefs as content sources.

**For the security page:** The PKI, trust spectrum, and file security pipeline documents add significant depth to the security story.

**For vertical landing pages (future):** The ambassador briefs for accountants, investors, musicians, and supply chain are ready-made content for industry-specific pages.

**Critical principle:** Only claim what's shipped. SG/Send v0.3.0 is live. SGit-AI is live on PyPI with a full CLI. SG-Vault, Data Rooms, and the agentic plane exist as architecture and briefs — they should be positioned as "our roadmap" or "where we're going", not as current features.
