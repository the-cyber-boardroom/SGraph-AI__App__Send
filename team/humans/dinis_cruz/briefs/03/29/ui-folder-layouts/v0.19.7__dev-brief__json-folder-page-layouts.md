# Folder Page Layouts: JSON-Driven Components That Claude Can Assemble

**version** v0.19.7
**date** 29 Mar 2026
**from** Human (project lead)
**to** Developer (Claude session building the layouts), Architect
**type** Dev brief

---

## The Problem

We now have markdown-based mini websites inside folders: pages with links, images, navigation. Claude creates these and they work. But markdown is not a great user experience. You cannot point to things easily. The navigation is hardcoded links at the top. The layout is linear (top to bottom). It looks like a document, not a page.

We can do much better. The components exist (Web Components, Shadow DOM, SG Layout). What is missing is a way for Claude to describe a rich page layout that the Browse view renders.

## The Idea: JSON Layout Files

Instead of Claude writing markdown pages, Claude writes a JSON file that describes the layout using a set of available components. The Browse view detects this JSON file and renders it as a rich page.

### How It Works

1. A set of Web Components are available in the Browse environment (Hero, Title, Image, Text, Slides, PDF, Section, BulletPoints, Gallery, etc.)
2. Claude writes a JSON file (e.g. `_page.json`) that describes the page as a sequence of components with their data
3. When the Browse view opens a folder that contains `_page.json`, it renders the JSON layout instead of the default file listing
4. Each folder can have its own `_page.json`, creating a navigable multi-page site

### Example JSON Layout

```json
{
  "title": "Security Assessment: Key Findings",
  "components": [
    {
      "type": "hero",
      "props": {
        "title": "Q1 2026 Security Assessment",
        "subtitle": "Key findings and recommendations",
        "image": "hero-banner.png",
        "color": "#1a1a2e"
      }
    },
    {
      "type": "section",
      "props": { "title": "Executive Summary" },
      "children": [
        {
          "type": "markdown",
          "props": { "file": "executive-summary.md" }
        }
      ]
    },
    {
      "type": "section",
      "props": { "title": "Infographic" },
      "children": [
        {
          "type": "image",
          "props": {
            "file": "infographic.png",
            "caption": "Visual summary of findings",
            "width": "100%"
          }
        }
      ]
    },
    {
      "type": "section",
      "props": { "title": "Detailed Findings" },
      "children": [
        {
          "type": "slides",
          "props": {
            "images": ["slide-01.png", "slide-02.png", "slide-03.png"],
            "captions": ["Risk landscape", "Top vulnerabilities", "Recommendations"]
          }
        }
      ]
    },
    {
      "type": "section",
      "props": { "title": "Full Report" },
      "children": [
        {
          "type": "pdf",
          "props": { "file": "full-report.pdf" }
        }
      ]
    }
  ],
  "navigation": [
    { "label": "Summary", "anchor": "executive-summary" },
    { "label": "Infographic", "anchor": "infographic" },
    { "label": "Findings", "anchor": "detailed-findings" },
    { "label": "Full Report", "anchor": "full-report" }
  ]
}
```

## The Component Library

### Available Components to Start With

| Component | Props | What It Renders |
|---|---|---|
| `hero` | title, subtitle, image, color | Full-width banner with text overlay |
| `title` | text, level (h1/h2/h3) | Styled heading |
| `markdown` | file (path to .md) or text (inline) | Rendered markdown (with image and link support) |
| `image` | file, caption, width | Image with optional caption |
| `slides` | images (array), captions (array) | Slideshow with navigation arrows |
| `pdf` | file | Embedded PDF viewer |
| `section` | title, children (array of components) | Container with heading, groups child components |
| `bullet-points` | items (array of strings) | Styled bullet list |
| `gallery` | images (array), columns | Grid of image thumbnails with lightbox |
| `text` | content | Plain styled text block |
| `csv` | file | Table rendered from CSV data |
| `navigation` | items (array of label + anchor) | Fixed navigation bar or sidebar |

These are all Web Components with Shadow DOM. They are self-contained, styled, and render independently.

### Fractal: Folders as Pages

If every folder can have a `_page.json`, the folder structure becomes a website:

```
root/
  _page.json              → renders as the homepage
  executive-summary/
    _page.json            → renders as the summary page
    summary.md
    chart.png
  findings/
    _page.json            → renders as the findings page
    slides/
      slide-01.png
      slide-02.png
  infographic/
    _page.json            → renders as the infographic page
    infographic.png
    article.md
```

Clicking a folder in the tree navigation opens its `_page.json` layout. The folder structure IS the site structure. No routing needed. No build step. Just JSON files and assets.

## The Workflow for Claude

### Step 1: Design the Perfect UI (ASCII Art)

Before writing any JSON, Claude should design the ideal page layout in ASCII art:

```
┌─────────────────────────────────────────┐
│            HERO: Title + Image          │
├─────────────────────────────────────────┤
│  [Summary] [Infographic] [Findings]     │  ← navigation
├─────────────────────────────────────────┤
│  Executive Summary                      │
│  ┌─────────────────────────────────┐    │
│  │  Markdown rendered from .md     │    │
│  └─────────────────────────────────┘    │
├─────────────────────────────────────────┤
│  Infographic                            │
│  ┌─────────────────────────────────┐    │
│  │  Full-width infographic image   │    │
│  └─────────────────────────────────┘    │
├─────────────────────────────────────────┤
│  Key Findings (Slides)                  │
│  ┌──────┐  ┌──────┐  ┌──────┐          │
│  │ Sl 1 │  │ Sl 2 │  │ Sl 3 │  ← / →  │
│  └──────┘  └──────┘  └──────┘          │
└─────────────────────────────────────────┘
```

This grounds the design before any code is written.

### Step 2: Write the JSON

Translate the ASCII layout into the JSON component format. Reference files in the folder by relative path.

### Step 3: Test in Browse View

Push the JSON and assets to the vault. Open the share link. The Browse view detects `_page.json` and renders the layout. Iterate.

## Why JSON, Not HTML

Claude could write HTML directly. But JSON is:

- **Safer:** No script injection risk (JSON is data, not code)
- **Constrainable:** Claude can only use the components we provide
- **Versionable:** JSON diffs are clean and meaningful
- **LLM-friendly:** Easy for Claude to generate and modify programmatically
- **Theme-able:** The components handle styling, the JSON handles structure. Changing the theme does not require changing the JSON.

## Connection to Existing Architecture

This is the same manifest-driven pattern appearing again:
- IFD v2: manifest controls which files load
- Focused use-case UIs: workflow config controls which steps show
- Folder narrative: `_config.json` controls section ordering
- Page layouts: `_page.json` controls visual rendering

JSON manifests controlling behaviour is the unifying pattern.
