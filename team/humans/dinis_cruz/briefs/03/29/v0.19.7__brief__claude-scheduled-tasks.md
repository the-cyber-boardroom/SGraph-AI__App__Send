## Another Milestone

Claude now has scheduled tasks (Claude Code sessions that run on a schedule). This is a significant capability that changes how we operate. Two schedules are already running:

### Schedule 1: The Librarian (Daily, 9:00 AM)

The Librarian agent fires up on the main repo every morning, processing files, indexing changes, cataloguing updates, and maintaining the knowledge base.

This solves a drift problem. Previously, the Librarian ran manually every couple of days. Between runs, context accumulated, briefs piled up, and the index fell behind. The regularity was never scalable. Now it is automatic: every morning at 9:00, the Librarian processes everything that changed since the last run.

### Schedule 2: QA Smoke Test (Daily, 9:00 AM)

A mini QA session starts every morning, leveraging the browser automation environment we already built. It checks the live website, runs Playwright-based smoke tests, and confirms the site is functioning correctly.

This works because all the hard problems were already solved:
- Chromium installation in the Claude environment (figured out during QA development)
- Proxy handling inside Claude (already dealt with)
- The QA test suite (already written and stable)

The scheduled task just runs what already works. The infrastructure investment pays off immediately.

## Why This Is a Wardley Maps Moment

Claude's scheduled tasks are a textbook ILC (Innovate, Leverage, Commoditise) example. All they did was allow starting a Claude Code session on a schedule. But because we already have a rich Claude Code environment (repo access, browser automation, vault communication, CLI tools), the scheduled session inherits all of that. The capability multiplies.

## Vault-Based Communication Between Schedules

The key architectural pattern: **use vaults to communicate between scheduled sessions.**

Each scheduled session does not need repo write access or extensive permissions. It just needs a vault key. The vault is the communication channel:

- The Librarian writes its daily index to a vault
- The QA session writes its smoke test results to a vault
- The morning briefing session reads from both vaults and assembles a daily status
- The human pulls from the vault to see what happened overnight

Environment variables pass the vault key to the scheduled session. The session clones the vault, reads or writes, and exits. Minimal permissions. Maximum communication.

## The Temporal Layer

This introduces time as a first-class element of the agentic workflow. Previously, everything happened when the human triggered it. Now, things happen on a schedule, overnight, while the human is not watching.

This means:
- **The Librarian keeps up without being asked.** Every morning, the index is current.
- **The QA team verifies without being asked.** Every morning, the website is confirmed working.
- **The daily briefing assembles itself.** The human opens their laptop to a ready status report.
- **Drift is eliminated.** Regular, automatic tasks prevent the slow accumulation of unprocessed work.

### Vault as the Temporal Communication Layer

Multiple scheduled sessions running in parallel need coordination. The vault handles this:

- Session A writes to vault at 9:00 AM
- Session B reads from vault at 9:30 AM
- No shared file system needed
- No database needed
- Just encrypted vaults that sessions push and pull

This also stress-tests the vault's concurrent access model (multiple sessions committing and pushing). Any issues that emerge here improve the vault for everyone.

## Mapping the Commit Flow

With multiple scheduled sessions committing to the repo, we need to understand and test:

- Multiple agents committing in parallel
- SGit handling concurrent pushes
- Merge conflicts between scheduled sessions
- The commit history remaining clean and attributable (which agent made which commit)

This is a real-world test of the multi-user vault workflow described in the branch model spec. The scheduled sessions are the first "always-on" agents that operate independently.
