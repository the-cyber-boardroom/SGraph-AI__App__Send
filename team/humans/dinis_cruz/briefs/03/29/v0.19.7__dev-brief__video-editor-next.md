## Where We Are

The first version of the video editor is working. Client-side WebAssembly using FFmpeg tools. Already has: frame extraction, cropping, and basic trimming. All client-side, no server processing. This is a strong foundation.

Now: expand it into a proper lightweight video editing suite that drives usage, drives token revenue (both LLM and SG/Send credits), and serves our own content creation needs.

## The Concrete Use Case

An MRI video of a knee. The edits needed:

- Crop the frame (remove surrounding UI/borders)
- Extract the audio
- Add captions (explaining what the viewer is seeing)
- Trim the start and end
- Mask sensitive areas (patient information visible in corners)
- The mask position changes because the camera moved during recording

This is a real, practical example that also demonstrates the doctor/medical use case from the earlier brief.

## Features to Add

### Multiple Timelines

The core of any video editor. A timeline panel at the bottom showing:

- Video track(s): the main video, plus overlay videos or image sequences
- Audio track(s): extracted audio, voiceover, background music
- Caption track: text overlays with timing, positioning, and styling
- Mask track: regions to blur or block, with keyframes for position changes

Drag and drop assets onto timelines. Reorder. Trim by dragging edges. Standard video editing interaction patterns.

### Asset Registry

A panel showing available assets for the current project:

- Video files (from local storage, from SG/Send transfers, or dropped from desktop)
- Images (for overlays, thumbnails, watermarks)
- Audio files (for voiceover, music)

This could use the simple token system: reference an SG/Send transfer by its friendly token and pull assets from it. The asset registry becomes a bridge between SG/Send's file sharing and the video editor.

### Captions

- Add text at specific timestamps
- Configure: duration, position (top/bottom/custom), font size, background, animation
- Preview captions in real time on the video
- Import/export as SRT or VTT files

### Masking

- Draw a region on a video frame to mask (blur, black out, pixelate)
- Keyframe the mask position (for when the camera moves)
- The mask moves smoothly between keyframes
- Essential for medical/legal content where sensitive information is visible

### Preview

- Real-time preview of the timeline (play the composed video with all tracks, captions, and masks)
- Scrub through the timeline
- Take screenshots at any point

### Crop and Transform

- Crop the video frame
- Rotate
- Scale
- These are per-clip transforms, stackable

## Architecture: Transformations as JSON

Every edit operation is captured as a JSON transformation:

```json
{
  "type": "crop",
  "params": { "x": 10, "y": 20, "width": 640, "height": 480 },
  "applied_at": "2026-03-29T10:30:00Z"
}
```

```json
{
  "type": "caption",
  "params": {
    "text": "Left knee, medial view",
    "start_time": 2.5,
    "end_time": 6.0,
    "position": "bottom-center",
    "style": "white-on-black"
  }
}
```

```json
{
  "type": "mask",
  "params": {
    "keyframes": [
      { "time": 0.0, "x": 500, "y": 10, "width": 200, "height": 50 },
      { "time": 3.0, "x": 510, "y": 15, "width": 200, "height": 50 }
    ],
    "mode": "blur"
  }
}
```

The video file itself is never modified until export. The project is a JSON file listing all transformations. This gives us:

- **Undo/redo:** Remove or reorder transformations
- **Non-destructive editing:** The original video is untouched
- **Replay:** Apply the same transformations to a different video
- **Remixing:** Share the transformation JSON, someone else applies it to their version
- **LLM-operable:** An LLM can read and write these JSON transformations without seeing the full video

### Consolidation

Multiple small transforms can be consolidated: if the user moved a mask 10 times while positioning it, consolidate into one final position. Store the full history for undo, but the "committed" version is consolidated.

## LLM Integration

The JSON transformation format is the meta-language that LLMs can operate on.

### Screenshot-Based Analysis

Take screenshots at regular intervals (every 1-2 seconds). Feed these to an LLM:

- "What is shown in this video?" (content description)
- "Where is the sensitive information I should mask?" (automatic mask suggestion)
- "Generate captions for each segment" (automatic captioning)
- "This is a medical scan. Create a professional narration" (content creation)

The LLM sees screenshots, not the full video. This is practical (video files are too large for LLM context) and efficient (a few screenshots convey the key information).

### LLM-Driven Editing

An LLM could produce transformations directly:

```
User: "Add a caption at the bottom saying 'Left knee MRI' for the first 5 seconds"
LLM: { "type": "caption", "params": { "text": "Left knee MRI", "start_time": 0, "end_time": 5, "position": "bottom-center" } }
```

The LLM does not need video access. It writes JSON. The editor applies it. The user previews and adjusts.

## Local Storage

### The Problem

Every page refresh loses the loaded video. Re-uploading a 100MB video file every time is painful during development and editing.

### The Solution

Use browser local storage (IndexedDB for large files, localStorage for metadata) to persist:

- The video file(s) currently being edited
- The project JSON (all transformations)
- Editor state (timeline position, zoom level, selected track)

Build this as a standalone Web Component (`sg-local-storage` or similar) that provides:

- A viewer showing what is stored locally (files, sizes, dates)
- Upload/download from local storage
- Capacity management (show how much space is used/available)
- Clear/manage stored files

This component is reusable across all tools, not just the video editor.

## Revenue and Usage

Every video editing session that uses LLM features consumes:

- LLM tokens (for captioning, analysis, mask suggestions): billable at 25% markup
- SG/Send credits (if the video or assets come from transfers): standard credit usage

The video editor as a free tool drives traffic. The LLM-powered features drive revenue.
