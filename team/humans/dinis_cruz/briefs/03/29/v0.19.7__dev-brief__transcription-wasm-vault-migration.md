## Three Related Topics

All driven by the same theme: client-side capabilities that work offline, powered by WebAssembly, with data moving between environments via vaults.

## Component 1: Audio Transcription (Whisper in the Browser)

### What It Does

A Web Component that transcribes audio to text, entirely client-side, using Whisper or an equivalent open source speech-to-text model running via WebAssembly. No server. No API calls. Works offline.

### Two-Pass Architecture

| Pass | Model | Speed | Quality | Use Case |
|---|---|---|---|---|
| Real-time pass | Smaller/faster model | Near real-time | Good enough to follow along | Live transcription as user speaks or plays audio |
| Clean-up pass | Larger/better model | Slower (batch) | High accuracy | Polished transcript after recording completes |

The real-time pass shows text appearing in the browser as the audio plays. The user can see the transcription building. The clean-up pass runs after and produces a higher-quality version with:

- Better punctuation and formatting
- Speaker diarisation (if the model supports it)
- Timestamps (per sentence or per word)

### Timestamp Support

The transcript should include timestamps so that:

- Clicking a timestamp jumps to that point in the audio
- The transcript can be used as navigation for long recordings
- Timestamps can be exported as SRT/VTT subtitles (connects to the video editor)

### Research Required

- Which Whisper variants have WASM builds? (whisper.cpp compiled to WASM is the likely candidate)
- What is the model size for the real-time vs clean-up pass?
- Can we do streaming transcription (process audio as it arrives) or only batch (process after recording)?
- What languages are supported out of the box?

## Component 2: WASM Loader (sg-wasm)

### The Pattern

We are now using WebAssembly in multiple places:

- FFmpeg for video editing
- Whisper for audio transcription
- Potentially other tools (image processing, PDF generation, encryption)

Each WASM module has the same lifecycle: download the binary, cache it, instantiate it, run it. This should be a shared component, not reimplemented per tool.

### What sg-wasm Does

```javascript
const whisper = await SGWasm.load('whisper-base', {
  cache: true,           // Store in IndexedDB after first download
  storage: 'local',      // Use local storage for caching
  onProgress: (p) => ... // Download progress callback
});

const result = await whisper.run(audioData, { language: 'en' });
```

The component handles:

| Responsibility | What It Does |
|---|---|
| Download management | Fetch WASM binary from CDN or local source |
| Caching | Store in IndexedDB or local storage after first download |
| Cache validation | Check if cached version matches expected hash |
| Offline support | Use cached version when offline |
| Instantiation | Load the WASM module, set up memory, initialise |
| Progress reporting | Download progress, initialisation progress |
| Storage viewer | Show what is cached locally, sizes, dates (reuse sg-local-storage) |

### Offline-First

The key property: once a WASM module has been downloaded once, it works offline forever (until the cache is cleared). This is critical for the plane journey use case and any offline work. Download the models while online, use them offline.

The sg-local-storage component (from the video editor brief) provides the UI for managing cached WASM modules: see what is stored, how much space it uses, clear specific modules.

## Component 3: Vault Migration (Move Vaults Between Servers)

### The Need

How to move a vault from one server to another. Use cases:

- Local vault (on laptop) → push to production server (sgraph.ai)
- Production vault → pull to local for offline work
- Vault on one server → replicate to another server (backup, migration)
- Keep two copies in sync (local and remote)

### How It Should Work

SGit already has the concept of remotes (like Git remotes). The vault migration is:

```bash
# Add a second remote
sgit-ai remote add production https://send.sgraph.ai

# Push vault to the new remote
sgit-ai push production

# Pull changes from the remote
sgit-ai pull production

# Sync (push local changes, pull remote changes)
sgit-ai sync production
```

This may already mostly work with the existing remote support. The question is whether the sync operation (which is not pushing new commits, just replicating the vault data) needs additional handling.

### Offline Workflow

The target workflow for the plane journey:

1. Before departure: `sgit-ai pull` all vaults needed for offline work
2. During flight: work on vaults locally, make changes, commit
3. After landing: `sgit-ai push` all changes back to the server

The vaults are the offline work environment. Everything needed (files, history, branches) is in the local vault. No internet required during work. Sync when connectivity returns.

### Research Needed

- Does `sgit-ai push` to a new remote already work?
- What happens with the encryption keys when migrating between servers? (The keys are vault-specific, not server-specific, so this should be transparent)
- How does sync handle conflicts when both local and remote have changes?
- Can we add a `--dry-run` flag to show what would be synced before doing it?
