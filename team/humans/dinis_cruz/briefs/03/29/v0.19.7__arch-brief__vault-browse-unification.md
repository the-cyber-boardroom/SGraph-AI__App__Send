# Vault-Browse Unification: Same UI, Different Data Source

**version** v0.19.7
**date** 29 Mar 2026
**from** Human (project lead)
**to** Architect (lead), Developer
**type** Architecture brief

---

## The Insight

The vault website (vault.sgraph.ai) and the Browse view in SG/Send (send.sgraph.ai/browse) do essentially the same thing: display a tree of files with a preview panel. The only difference is where the tree data comes from:

| | Browse View | Vault View |
|---|---|---|
| Data source | Zip file (downloaded, extracted client-side) | Vault API (tree fetched from server, files fetched on demand) |
| Speed | Needs to download entire zip first | Faster initial load (only fetches tree structure, not file contents) |
| Editing | Read-only (currently) | Read-write (vault supports commits) |
| Version control | No | Yes (vault has commits, branches) |
| Code | In SG/Send repo, `send-browse.js` | In vault repo, separate codebase |

The vault website has been trying to replicate the Browse view's functionality but with its own code. This creates divergence: features added to Browse do not appear in the vault view, and vice versa. The vault codebase got complex trying to add read-write capabilities on top.

## The Refactoring

**Strip the vault website down. Import the Browse module directly. The only code the vault website adds is the data source adapter.**

```
Browse Module (shared)
  │
  ├── Tree navigation component
  ├── Preview panel (markdown, PDF, image, video, CSV, HTML iframe)
  ├── Clickable markdown links
  ├── Inline image rendering
  ├── CSV table rendering
  ├── SG Layout integration
  │
  ├── Data Source: Zip (for send.sgraph.ai/browse)
  │     └── Downloads zip, extracts, builds tree from zip contents
  │
  └── Data Source: Vault API (for vault.sgraph.ai)
        └── Calls vault API, builds tree from API responses
        └── Fetches file contents on demand (not all at once)
        └── Supports write operations (edit, commit)
```

The Browse module is literally the same JavaScript file loaded by both sites. The data source is injected. The vault site adds the vault-specific adapter (API calls instead of zip extraction) and any vault-specific features (edit, commit, branch switching).

## Fractal Vaults

The vault view should support opening vaults inside vaults:

- A parent vault contains references to child vaults
- Clicking a child vault opens it in the same Browse interface (nested)
- The SG Layout already supports fractal nesting (panels within panels)

Use case: the human's master vault contains all their project vaults. Opening the master vault shows a tree of vaults. Clicking one opens that vault's tree. This is how you manage multiple vaults from one place.

## What the Vault View Gains from Unification

All the features recently added to Browse immediately become available in the vault view:

- Clickable markdown hyperlinks
- Inline markdown images
- CSV rendered as tables
- HTML files in iframes
- The folder manifest (`_config.json` for section ordering, start file)
- Any future Browse improvements

No code duplication. One module, two data sources.

## What Browse Gains from Vault Capabilities

In the future, the vault data source could be offered as an option in the Browse view:

- "Open this folder as a vault" (upgrade from read-only zip to read-write vault)
- Version history visible in the Browse view
- Diff between versions
- Collaborative editing (multiple people viewing and modifying the same folder)

This is not for immediate implementation. But the unified architecture makes it possible without rewriting anything.

## Performance Advantage

The vault data source is actually faster for initial load than the zip source:

- Zip: download entire zip (could be large), extract client-side, build tree
- Vault: fetch tree structure via API (small JSON), render immediately, fetch individual files on demand

For a folder with 100 files, the vault view shows the tree instantly and loads file contents as the user clicks. The zip view waits until the entire zip is downloaded before showing anything.
