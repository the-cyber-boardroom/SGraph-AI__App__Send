# IFD in Practice: Surgical Overrides with Version-Stamped Filenames

**Version:** v1.3.0  
**Date:** 29 March 2026  
**Builds on:** v1.1.0 (IFD core methodology), v1.2.1 (IFD intro and how-to-use)  
**Context:** Practical guidance for applying IFD surgical overrides in static-site repos that use a build-time version layering system (`build_ifd_latest.py` or equivalent). Proven in the __Tools repo and informed by SG/Send's v0.3.1 production overlay pattern.  

---

## Why This Document Exists

The original IFD guides (v1.1.0 and v1.2.1) describe surgical overrides where minor versions contain only their deltas — a 15-line prototype override instead of copying a 500-line file. This is the correct approach.

But these guides assume a **loading model** where HTML pages explicitly load each version's files in order via `<script>` tags:

```html
<script src="../v0.1.0/components/config-panel/config-panel.js"></script>
<script src="../v0.1.3/components/config-panel/config-panel.js"></script>
<script src="../v0.1.5/components/config-panel/config-panel.js"></script>
```

In production, static-site repos (like __Tools) use a **build-time layering model** where all IFD version folders are merged into a single `_latest/` directory. Files from later versions overwrite files from earlier versions **with the same name**.

This creates a collision: if v0.1.0 and v0.1.3 both contain `config-panel.js`, the v0.1.3 override (15 lines) destroys the v0.1.0 base (500 lines) in the merged output.

**Version-stamped filenames solve this.** The base keeps its canonical name. Overrides include the version in their filename. Both survive the merge.

---

## The Filename Convention

### Base Files — NO Version Stamp

The initial version's files use their canonical names:

```
v0.1.0/
  sg-tree.js              ← full base class
  sg-tree-events.js       ← frozen event constants
  sg-tree.css             ← theme defaults
```

This is the name that consumers import. It never changes within the same major version.

### Surgical Overrides — Version-Stamped with Double Dash

Override files include the full semver after a double dash (`--`):

```
v0.1.1/
  sg-tree--v0.1.1.js       ← surgical: editor mode (~80 lines)

v0.1.2/
  sg-tree--v0.1.2.js       ← surgical: virtual scrolling (~60 lines)

v0.1.3/
  sg-tree--v0.1.3.js       ← surgical: multi-select (~50 lines)
```

### Why Double Dash (`--`)

The double dash clearly separates the component name from the version suffix:

| Filename | What it is |
|---|---|
| `sg-tree.js` | Base class (canonical name) |
| `sg-tree-events.js` | Related file (single dash = part of the name) |
| `sg-tree--v0.1.1.js` | Surgical override (double dash = version stamp) |
| `sg-tree--v0.1.2.js` | Surgical override (double dash = version stamp) |

The `--` is unambiguous. `sg-tree-events.js` is clearly a different file, not a version override.

### Why Full Semver in the Filename

The version stamp (`v0.1.1`) maps directly to the IFD folder name:

```
v0.1.1/sg-tree--v0.1.1.js
^^^^^^           ^^^^^^
folder           filename
```

No ambiguity about which version introduced which override.

## The SgComponent Base Class Pattern

The __Tools repo has a proven base component pattern in `components/base/v1/v1.0/v1.0.0/sg-component.js`. Components that extend `SgComponent` get Shadow DOM, resource loading (fetches sibling `.html` template + `.css`), tracked event listeners, and a lifecycle:

```javascript
import { SgComponent } from '/components/base/v1/v1.0/v1.0.0/sg-component.js';

class SgKeyGenerator extends SgComponent {
    static jsUrl = import.meta.url;

    get resourceName() { return 'sg-key-generator'; }

    get sharedCssPaths() {
        return ['/components/tokens/v1/v1.0/v1.0.0/sg-tokens.css'];
    }

    bindElements() {
        this._button = this.$('.generate-btn');
    }

    setupEventListeners() {
        this.addTrackedListener(this._button, 'click', this._onGenerate);
    }

    async onReady() {
        await this._generateNewKey();
    }
}

customElements.define('sg-key-generator', SgKeyGenerator);
window.SgKeyGenerator = SgKeyGenerator;  // expose for surgical overrides
```

**The lifecycle is:**

```
connectedCallback()          ← SgComponent handles this
  └→ loadResources()         ← fetches .html + .css, injects into Shadow DOM
  └→ bindElements()          ← subclass caches shadow DOM references
  └→ setupEventListeners()   ← subclass binds event handlers
  └→ onReady()               ← subclass runs init logic
  └→ emit('component-ready') ← signals ready to external listeners
```

**All of these are prototype method calls via `this.`** — which means surgical overrides are picked up at call time.

## connectedCallback and Surgical Overrides — Timing

**Key question:** When a surgical override patches a method like `onReady()` or `_renderRow()`, will `connectedCallback` call the patched version?

**Yes — as long as the override script loads BEFORE the element is connected to the DOM.**

### Why This Works

JavaScript prototype lookup is **dynamic**. When `connectedCallback()` runs `this.onReady()`, the engine looks up `onReady` on the prototype chain **at call time**, not at class definition time. If v0.1.1 patches `SgTree.prototype.onReady` before the `<sg-tree>` element is inserted into the DOM, the patched version runs.

### The Loading Order Guarantee

```html
<!-- 1. Load base class (defines SgTree, calls customElements.define) -->
<script type="module" src="/core/sg-tree/sg-tree.js"></script>

<!-- 2. Load surgical overrides (patch SgTree.prototype) -->
<script type="module" src="/core/sg-tree/sg-tree--v0.1.1.js"></script>
<script type="module" src="/core/sg-tree/sg-tree--v0.1.2.js"></script>

<!-- 3. THEN the element exists in the DOM (connectedCallback fires) -->
<sg-tree id="my-tree"></sg-tree>
```

ES modules are **deferred by spec** — they execute after the HTML parser finishes. So by the time `connectedCallback` fires on `<sg-tree>`, all module scripts (base + overrides) have already executed. The prototype has all patches applied.

**Even if the element is already in the HTML source** (not dynamically created), modules still run before `connectedCallback` because:
1. HTML parser encounters `<script type="module">` tags → schedules them
2. HTML parser encounters `<sg-tree>` → creates element but defers connection
3. All modules execute in source order
4. Custom element upgrades run → `connectedCallback` fires with patched prototype

### What About Dynamically Created Elements?

If JavaScript creates an element AFTER all modules have loaded:

```javascript
const tree = document.createElement('sg-tree');
document.body.appendChild(tree);  // connectedCallback fires with patched prototype
```

This always works because the prototype was patched before this code runs.

## Version Chains: Patches, Rewrites, and Coexistence

Version-stamped files support three scenarios within the same minor version family:

### Scenario 1: Patch Chain

Each override patches the previous state:

```
_latest/
  sg-tree.js              ← base (v0.1.0)
  sg-tree--v0.1.10.js     ← patches sg-tree.js
  sg-tree--v0.1.11.js     ← patches sg-tree--v0.1.10.js (cumulative)
```

Consumer loads all three:
```html
<script type="module" src="sg-tree.js"></script>
<script type="module" src="sg-tree--v0.1.10.js"></script>
<script type="module" src="sg-tree--v0.1.11.js"></script>
```

### Scenario 2: Complete Rewrite Mid-Chain

A version-stamped file can be a **complete rewrite** that replaces the base entirely:

```
_latest/
  sg-tree.js              ← original base (v0.1.0) — still present for older tools
  sg-tree--v0.1.12.js     ← COMPLETE REWRITE (new full class, calls define() with guard)
  sg-tree--v0.1.13.js     ← patches sg-tree--v0.1.12.js
```

**Tools using the old base:**
```html
<script type="module" src="sg-tree.js"></script>
<script type="module" src="sg-tree--v0.1.10.js"></script>
```

**Tools using the rewrite:**
```html
<script type="module" src="sg-tree--v0.1.12.js"></script>
<script type="module" src="sg-tree--v0.1.13.js"></script>
```

Both coexist in `_latest/` because the filenames are different. The HTML page decides which chain to load.

**A rewrite file that replaces the base should guard `define()`:**

```javascript
// sg-tree--v0.1.12.js — complete rewrite
export class SgTree extends HTMLElement {
    // ... complete new implementation
}

// Guard: only define if not already registered (supports both standalone and overlay use)
if (!customElements.get('sg-tree')) {
    customElements.define('sg-tree', SgTree);
}
window.SgTree = SgTree;
```

### Scenario 3: Different Tools Pin to Different Versions

Because the HTML page controls which files to load, different tools can use different points in the chain:

| Tool | Loads | Gets |
|---|---|---|
| Vault Browser | `sg-tree.js` | Base features only |
| File Manager | `sg-tree.js` + `sg-tree--v0.1.1.js` | Base + editor mode |
| Workspace | `sg-tree--v0.1.12.js` + `sg-tree--v0.1.13.js` | Complete rewrite + latest patches |

This is a **major strength** of the version-stamped approach — tools upgrade at their own pace.

## Major Version Consolidation

Each major version (`v{release}.{major}.0`, e.g. v0.2.0) is **100% independent**:

1. All v0.1.x surgical overrides are merged into a single self-contained `sg-tree.js`
2. The consolidated file is placed in `v0.2.0/sg-tree.js`
3. It does NOT depend on any v0.1.x file
4. ALL tools in the suite are updated and verified to work with v0.2.0
5. This is the compatibility checkpoint — everything must work together

```
v0.2.0/
  sg-tree.js              ← complete, self-contained (all v0.1.x features merged)
  sg-tree-events.js       ← complete with all added events
  sg-tree.css             ← complete merged styles
```

After v0.2.0 consolidation, consumers load a single file:
```html
<script type="module" src="/core/sg-tree/v0/v0.2/v0.2.0/sg-tree.js"></script>
```

### Version Pinning via static.sgraph.ai

For cases where a consumer needs a specific version (not latest), ALL versions are hosted on `static.sgraph.ai`:

```html
<!-- Pin to a specific IFD version (never changes) -->
<script type="module"
    src="https://static.sgraph.ai/core/sg-tree/v0/v0.1/v0.1.0/sg-tree.js"></script>
```

This is the escape hatch. The `_latest/` layering is for the tools.sgraph.ai site itself. External consumers can always pin to an exact version.

## How It Works End-to-End

### 1. Developer Creates a Surgical Override

The developer adds a new file in the override version's folder:

```javascript
// v0.1.3/sg-tree--v0.1.3.js
// Surgical override: adds multi-select (Shift+click, Ctrl+click)

const { SgTree } = window;

// Wrap _onNodeClick to add modifier key handling
const _originalOnNodeClick = SgTree.prototype._onNodeClick;
SgTree.prototype._onNodeClick = function(e, nodeId) {
    if (e.shiftKey) {
        this._selectRange(nodeId);
        return;
    }
    if (e.ctrlKey || e.metaKey) {
        this._toggleSelection(nodeId);
        return;
    }
    _originalOnNodeClick.call(this, e, nodeId);
};

// Add new methods (don't exist on base — no wrapping needed)
SgTree.prototype._selectRange = function(toId) { /* ... */ };
SgTree.prototype._toggleSelection = function(id) { /* ... */ };
SgTree.prototype.getSelectedIds = function() { return [...this._selectedIds]; };
```

### 2. Build System Layers All Versions

`build_ifd_latest.py` (or `run-locally.sh` for local dev) copies files from each version folder in order:

```
v0.1.0/sg-tree.js           → _latest/sg-tree.js           (base)
v0.1.0/sg-tree-events.js    → _latest/sg-tree-events.js    (events)
v0.1.0/sg-tree.css           → _latest/sg-tree.css          (theme)
v0.1.1/sg-tree--v0.1.1.js   → _latest/sg-tree--v0.1.1.js   (unique name — preserved)
v0.1.2/sg-tree--v0.1.2.js   → _latest/sg-tree--v0.1.2.js   (unique name — preserved)
v0.1.3/sg-tree--v0.1.3.js   → _latest/sg-tree--v0.1.3.js   (unique name — preserved)
```

All files survive. The base is never overwritten by an override.

### 3. Consumers Load Base + Overrides

```html
<script type="module" src="/core/sg-tree/sg-tree.js"></script>
<script type="module" src="/core/sg-tree/sg-tree--v0.1.1.js"></script>
<script type="module" src="/core/sg-tree/sg-tree--v0.1.2.js"></script>
<script type="module" src="/core/sg-tree/sg-tree--v0.1.3.js"></script>
```

### 4. At Consolidation (v0.2.0), Everything Merges

```
v0.2.0/
  sg-tree.js              ← complete self-contained class (all overrides merged in)
  sg-tree-events.js       ← complete with any added events
  sg-tree.css
```

v0.2.0's `sg-tree.js` overwrites v0.1.0's `sg-tree.js` via normal IFD layering. The version-stamped override files from v0.1.1–v0.1.6 are still in `_latest/` but are no longer loaded by consumers (they load only v0.2.0).

## Why Prototype Patching Works with ES Modules

A common misconception: "ES modules isolate their scope, so you can't modify a class defined in another module." This is **partially wrong**.

### What ES Module Isolation Actually Prevents

- A variable defined in module A is NOT automatically visible in module B
- `export class SgTree` does NOT put `SgTree` on the global `window` object

### What ES Module Isolation Does NOT Prevent

- Explicitly writing to `window`: `window.SgTree = SgTree` works fine
- Reading from `window`: `const { SgTree } = window` works fine
- Modifying a prototype: `SgTree.prototype.method = ...` affects all instances
- `customElements.define()` uses the class reference — prototype changes after `define()` are reflected in all elements

### The One Constraint

`customElements.define('sg-tree', SgTree)` can only be called **once** per tag name. The base class (v0.1.0) calls it. Surgical overrides should NOT call it again. A complete rewrite (v0.1.12) should guard with `if (!customElements.get('sg-tree'))`.

## Comparison with SG/Send's v0.3.1 Pattern

SG/Send v0.3.1 uses the same prototype override approach in production:

```
v0.3.0/_common/js/send-browse.js          ← base (non-module script)
v0.3.1/_common/js/send-browse-v031.js     ← surgical override
```

The override wraps methods using the IIFE pattern:

```javascript
SendBrowse.prototype._renderFolderNode = (function(original) {
    return function(container, folder, depth) {
        const basename = file.name.includes('/')
            ? file.name.split('/').pop()
            : file.name;
        return original.call(this, container, folder, depth);
    };
})(SendBrowse.prototype._renderFolderNode);
```

**Key differences from our convention:**

| Aspect | SG/Send | __Tools |
|---|---|---|
| Naming | `send-browse-v031.js` | `sg-tree--v0.1.1.js` |
| Separator | Single dash | Double dash (`--`) |
| Version format | Compact (`v031`) | Full semver (`v0.1.1`) |
| Module type | Plain `<script>` | `<script type="module">` |
| Class access | Already on `window` | Explicit `window.SgTree = SgTree` |

Both work. We prefer the double-dash + full-semver convention because it's more explicit and maps directly to the IFD folder name.

## Rules Summary

| Rule | Description |
|---|---|
| **Base files: no stamp** | `sg-tree.js`, `sg-tree-events.js`, `sg-tree.css` — canonical names |
| **Overrides: always stamped** | `sg-tree--v0.1.1.js` — double dash + full semver |
| **Stamp = folder name** | `v0.1.1/sg-tree--v0.1.1.js` — no ambiguity |
| **One `define()` call** | Only the base (v0.1.0) registers the custom element |
| **Rewrites guard `define()`** | `if (!customElements.get('sg-tree'))` before re-registration |
| **Expose on `window`** | Base writes `window.ClassName = ClassName` for overrides to access |
| **Wrap, don't replace** | Use IIFE pattern to preserve original: `(function(orig) { ... })(proto.method)` |
| **New methods: just add** | Methods that don't exist on the base don't need wrapping |
| **Consolidation overwrites** | v0.2.0's `sg-tree.js` replaces v0.1.0's via normal IFD layering |
| **CSS overrides: same pattern** | `sg-tree--v0.1.3.css` for CSS-only surgical changes |
| **HTML page is the composition root** | The page decides which scripts to load and in what order |

## When to Use Each Pattern

| Scenario | Approach | Example |
|---|---|---|
| Fix a bug in one method | Surgical override | `sg-tree--v0.1.3.js` (patches `_renderRow`) |
| Add new feature (new + modify existing methods) | Surgical override | `sg-tree--v0.1.4.js` (adds drag, wraps pointer handlers) |
| Add an entirely new file | Normal IFD | `v0.1.5/new-feature.js` (unique name — no stamp needed) |
| Replace a CSS value | Surgical CSS override | `sg-tree--v0.1.2.css` |
| Complete rewrite of a component | Version-stamped rewrite | `sg-tree--v0.1.12.js` (full class, guards `define()`) |
| Update the landing page | Full replacement | `v0.1.5/en-gb/index.html` (overwrites earlier version) |
| Consolidate for major release | Major version | `v0.2.0/sg-tree.js` (all features merged, self-contained) |

## Practical Example: Full Version Chain

```
_latest/
  sg-tree.js                    # v0.1.0 — base class (800 lines)
  sg-tree-events.js             # v0.1.0 — frozen event constants
  sg-tree.css                   # v0.1.0 — theme defaults
  sg-tree--v0.1.1.js            # 80 lines  — patches: adds editor mode
  sg-tree--v0.1.2.js            # 60 lines  — patches: adds virtual scrolling
  sg-tree--v0.1.10.js           # 40 lines  — patches: performance fix
  sg-tree--v0.1.11.js           # 30 lines  — patches v0.1.10: additional fix
  sg-tree--v0.1.12.js           # 900 lines — COMPLETE REWRITE (new base, guards define())
  sg-tree--v0.1.13.js           # 50 lines  — patches v0.1.12: minor addition
```

**Tool A** (uses original base + editor):
```html
<script type="module" src="sg-tree.js"></script>
<script type="module" src="sg-tree--v0.1.1.js"></script>
```

**Tool B** (uses rewrite + latest):
```html
<script type="module" src="sg-tree--v0.1.12.js"></script>
<script type="module" src="sg-tree--v0.1.13.js"></script>
```

**Both coexist.** Different tools, different version chains, same `_latest/` directory.

**At v0.2.0 consolidation:** Everything is merged into one `sg-tree.js`. All tools are updated to use it. Verified working together. Shipped.

## Applicability Beyond sg-tree

This pattern applies to ANY component that needs surgical IFD overrides:

- `sg-layout--v0.1.5.js` — add closable/fixedSize as a surgical overlay
- `sg-image-editor--v1.0.1.js` — add rotation as a surgical overlay
- `sg-video-splitter--v1.0.1.js` — add visual timeline as a surgical overlay

The convention is universal: `{component-name}--v{version}.js` for surgical overrides, `{component-name}.js` for the base or consolidated version.
