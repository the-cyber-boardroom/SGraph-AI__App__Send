## The Problem

Claude's artifact system does not support importing external scripts. This means the Web Components we have been building (on tools.sgraph.ai, in the IFD structure) cannot be used inside Claude's UI development environment. You cannot `import` sg-layout, sg-tree, or any of the existing components.

But the need is real: develop new components interactively, using an LLM, while being able to reference and reuse the component library we already have.

Additionally, in a standard LLM chat, the conversation history grows exponentially. Every message adds to the context window. Eventually you hit the limit, lose context, or pay for tokens that carry stale information.

## The Idea: Managed Context, One-Shot Execution

Instead of a growing conversation, build a development environment where:

1. **The context is explicit and managed.** Not "everything said in this conversation" but specific, curated blocks of information that are assembled into a single prompt.

2. **Every LLM call is one-shot.** The full context is assembled and sent in one request. The response comes back. The context is then updated based on the result. The next call gets the updated context.

3. **The context is structured as components:**

| Context Block | What It Contains | When It Changes |
|---|---|---|
| Project background | What the project is, the stack, the architecture | Rarely |
| Component library | List of available components, their APIs, how to use them | When components are added |
| Current component | The component being developed (code, state, spec) | Every iteration |
| Memory | Accumulated decisions, preferences, patterns from this session | After each LLM interaction |
| Task | What to do right now ("add a collapse feature to the tree view") | Every interaction |

4. **Two LLM execution modes:**

**Mode 1: Build.** The LLM receives the full context + the task. It produces code (the component, the test, the HTML). The code is rendered in an iframe for visual feedback.

**Mode 2: Update memory.** The LLM receives the current memory + the latest interaction result. It produces an updated memory (pruned, consolidated, focused). This memory feeds into the next Build call.

This prevents the exponential growth problem. The memory is actively managed, not passively accumulated. Old decisions that are no longer relevant are pruned. The context stays focused and within token limits.

## The Development UI

A tool (on tools.sgraph.ai or standalone) with:

### Left Panel: Context Editor

- Editable text blocks for each context component (project, library, current code, memory, task)
- Toggle blocks on/off (do not send the full project background every time if it is not needed)
- Token count per block (see how much context you are using)
- "Condense memory" button (triggers Mode 2: LLM prunes the memory)

### Center Panel: Code Editor

- Write HTML, CSS, JavaScript
- Syntax highlighting
- The current component code

### Right Panel: Live Preview

- An iframe rendering the current component
- Auto-refreshes when code changes
- Can load external scripts (the tools.sgraph.ai components, CDN libraries)

### Bottom Panel: LLM Interaction

- Task input ("add keyboard navigation to the tree view")
- Submit button (assembles context + task, sends one-shot to LLM)
- Response area (the LLM's output: code changes, explanations)
- "Apply" button (updates the code editor with the LLM's changes)

## How It Works in Practice

1. Load the development environment
2. Set the context: "I am building a folder viewer component using sg-layout and sg-tree"
3. Load the component library context (APIs for sg-layout, sg-tree)
4. Write initial code (or let the LLM generate a starting point)
5. See it in the iframe
6. Type a task: "Add expand/collapse to the tree nodes"
7. The environment assembles: project context + component library + current code + memory + task
8. Sends one-shot to the LLM (OpenRouter, direct API, or local Ollama)
9. LLM returns updated code
10. Apply to the code editor, iframe refreshes
11. Memory is updated: "added expand/collapse, uses aria-expanded attribute"
12. Repeat

Each iteration is independent. The context is managed. The history does not grow unboundedly. The LLM always receives a focused, curated prompt.

## Connection to Existing Work

### Workspace v1

The first version of the workspace already had LLM integration with Ollama and OpenRouter. The context management concept was partially there. This brief expands it with explicit context blocks, the two-mode execution, and the managed memory pattern.

### sg-llm Component

The one-shot LLM component from the client-side LLM tools brief is the execution layer. This environment is a UI built on top of sg-llm.

### IFD Components

The whole point: use this environment to develop and test the IFD-versioned components (sg-tree, sg-transfer, sg-portfolio, etc.) interactively, with LLM assistance, with live preview.

## Why Not Just Use Claude Code?

Claude Code is excellent for working on a Git repo. But:

- It cannot render a live preview of a Web Component
- Its context is the conversation history (grows, cannot be pruned)
- It has repo access but not a visual development environment
- Importing custom scripts is not supported in artifact mode

This tool fills a different niche: visual, interactive Web Component development with managed LLM context. Claude Code is for backend work, refactoring, and repo-level changes. This tool is for component design and experimentation.
