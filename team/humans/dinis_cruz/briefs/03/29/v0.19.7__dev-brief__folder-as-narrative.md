# Folder as Narrative: Custom Viewers, Portfolio Layouts, and Component Configuration

**version** v0.19.7
**date** 29 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Architect, Designer
**type** Dev brief

## The Realisation

The folder view is becoming as powerful as the gallery, maybe more so. Because with a folder, you can create a narrative. You can structure sub-folders, order files, add a "start here" markdown. Yesterday, a shared folder had five sub-folders: source materials, created documents, five individual output folders (each with an infographic and a slide deck), and a security review. Powerful content. But already overwhelming: where does the recipient start?

The gallery is great for visual browsing. The folder is great for structured storytelling. Both need to evolve.

## Problem 1: Where Do You Start?

A folder with 30 files across 5 sub-folders needs guidance. A markdown file called "start-here.md" works but is crude. We can do better.

### The Folder Manifest

If a folder contains a special file (e.g. `_config.json` or `_manifest.json`), the folder view reads it and uses it to customise the presentation:

```json
{
  "title": "Security Review Pack",
  "description": "Materials from the Q1 security assessment",
  "start_file": "00-executive-summary.md",
  "layout": "narrative",
  "sections": [
    { "folder": "source-materials", "label": "Source Documents", "collapsed": true },
    { "folder": "findings", "label": "Key Findings", "collapsed": false },
    { "folder": "infographics", "label": "Visual Summaries", "featured": true },
    { "folder": "security-review", "label": "Security Assessment" }
  ]
}
```

The folder view reads this and renders accordingly: title at the top, description, then the sections in the specified order with the specified collapse states. The "start_file" opens automatically. The "featured" section gets prominence.

This is the same pattern as the IFD manifest: a JSON file that controls how things are presented. The content is the files. The manifest is the presentation layer.

## Problem 2: Opening Files in the Right Component

Currently: images open in the image viewer, videos open in the video player, markdown opens in the markdown renderer. These are sensible defaults.

But what if:
- A video should open in the video editor (with trimming, captioning)
- An image should open in an image redaction tool (for sensitive content)
- A set of files should open in a portfolio layout (headshot + bio + showreel together)
- A markdown file should render with inline images from the same folder

### Per-File Component Override

The folder manifest can specify which component to use for specific files:

```json
{
  "file_overrides": {
    "knee-mri.mp4": { "component": "sg-video-editor", "mode": "view" },
    "headshot.jpg": { "component": "sg-image-viewer", "mode": "lightbox" },
    "portfolio.json": { "component": "sg-portfolio", "mode": "view" }
  },
  "default_components": {
    "video": "sg-video-player",
    "image": "sg-image-viewer",
    "markdown": "sg-markdown-viewer",
    "pdf": "sg-pdf-viewer"
  }
}
```

When the user clicks a file, the folder view checks: is there an override for this file? If yes, open in that component. If no, use the default for the file type.

## Problem 3: The Portfolio Layout

The portfolio use case (from the earlier brief) needs a specific layout: headshot at the top, name and bio, then a grid of videos and images below. This is not a generic folder view. It is a custom component that reads specific files from the folder.

### The Portfolio Component (sg-portfolio)

A Web Component that expects:

```json
{
  "component": "sg-portfolio",
  "config": {
    "headshot": "headshot.jpg",
    "title": "Jane Smith",
    "subtitle": "Actor / Singer",
    "bio": "bio.md",
    "sections": [
      { "label": "Showreel", "files": ["showreel.mp4"] },
      { "label": "Headshots", "files": ["headshot-1.jpg", "headshot-2.jpg", "headshot-3.jpg"] },
      { "label": "Credits", "files": ["credits.pdf"] }
    ]
  }
}
```

The portfolio component reads this config, loads the referenced files from the folder, and renders a professional portfolio page. The recipient sees a polished presentation, not a file listing.

### How It Gets Created

Two options:

1. **Manual:** The sender creates the `_portfolio.json` config file and adds it to the folder before sharing. The folder view detects it and renders the portfolio layout.

2. **LLM-assisted:** The sender drops files into the folder. An LLM analyses the contents ("I see a headshot, a video, a PDF of credits, and a text bio") and generates the config file. The sender reviews and adjusts.

## Problem 4: Markdown with Inline Images

A markdown file that references images in the same folder should render those images inline:

```markdown
# Executive Summary

Here is the overview of our findings:

![Architecture diagram](./diagrams/architecture.png)

The key risk areas are highlighted below:

![Risk matrix](./diagrams/risk-matrix.png)
```

The markdown viewer needs to resolve relative image paths against the folder contents. The images are in the same encrypted folder. The viewer decrypts and renders them inline.

This turns a markdown file from a text document into a rich document with embedded visuals. Combined with the print-to-PDF feature, this produces professional reports directly from the folder contents.

## The Component Architecture

Every viewer (video player, image viewer, markdown renderer, portfolio, editor) should be a standalone Web Component with:

- **View mode:** Read-only presentation (the default for recipients)
- **Edit mode:** Allows modifications (for the sender during preparation, or for recipients with edit permissions)
- **Config mode:** Allows customising the layout and settings

The folder manifest decides which component and which mode to use for each file. The components are loaded from tools.sgraph.ai (or vendored locally). Adding a new viewer is: build the component, register it in the component registry, reference it in a folder manifest.

## Implementation: Start Small

1. **Folder manifest:** Implement `_config.json` support in the folder view. Start with title, description, start_file, and section ordering.
2. **Markdown with images:** Make relative image paths work in the markdown viewer.
3. **Portfolio component:** Build as a standalone tool. Test with the artist use case.
4. **Per-file component override:** Allow the manifest to specify which component opens which file.

Each step is independently valuable. The manifest alone (with section ordering and a start file) dramatically improves the experience of receiving a complex folder.
