# From Voice Memo to Published Article in One Workflow

I think I have finally cracked the content creation workflow I have been chasing for months. Let me describe what just happened, because it is genuinely remarkable.

## The Old Workflow (Painful)

My content creation process has been roughly this:

1. Record a voice memo (5-10 minutes, walking, thinking out loud)
2. Get it transcribed by Otter.ai
3. Give the transcript to NotebookLM
4. NotebookLM creates infographics and slide decks (which are often excellent)
5. Give the voice memo to Claude, which creates a written document
6. Sometimes iterate on the document with Claude
7. Post the infographic or a slide to LinkedIn

The problem: I was posting the infographic but missing the narrative. The documents Claude created were often quite good, sometimes really good. The infographics NotebookLM created were sometimes excellent. But combining them into a proper article with inline images, context, and a narrative arc was heavy lifting. I wanted Claude to help me write articles that included the infographics and slides, but getting the images into Claude and getting a formatted article out was painful.

I also discovered that the better the document I give to NotebookLM, the better the infographics it creates. So feeding Claude's cleaned-up document back to NotebookLM (instead of the raw voice memo) produces significantly better visual materials. There is a feedback loop: voice memo produces document, document produces better infographics, better infographics could produce a better article, if only I could assemble them.

## The New Workflow (This Morning)

Here is what I did this morning, and it worked spectacularly.

**Step 1: Prepare the materials.** I had my folder with the markdown document (Claude's written version of the voice memo), the infographic from NotebookLM, and the slide deck from NotebookLM. Three files in a folder.

**Step 2: Drop into Claude with vault skills.** I dropped the folder into a Claude session that already has my SGit vault skills loaded. Claude created a new encrypted vault, uploaded the files, and was ready to work.

**Step 3: Extract visual assets.** I asked Claude to extract screenshots from every slide in the slide deck (one image per slide) and to crop the infographic into individual sections. Claude did this and organised everything into clean folders.

**Step 4: Write the articles.** I asked Claude to write an article for the infographic (introduction, text, cropped image, text, cropped image, flowing through the infographic's narrative) and a separate article for the slide deck (introduction, text, slide screenshot, text, slide screenshot, building the argument slide by slide).

Claude did both. The output was markdown files with inline image references.

**Step 5: Push to vault and share.** Claude pushed everything to the vault: the articles, the extracted images, the original materials, all organised in folders. Claude also gave me the SG/Send share link.

**Step 6: View it.** I opened the share link in a new browser window. The Browse view rendered the markdown articles with the inline images. Clickable navigation between sections. The article looked like a proper web page, not a file dump.

**Step 7: Print or publish.** The markdown-to-PDF print feature produced a clean PDF of the article with all images inline. For LinkedIn, the article is ready to copy (or I can use the PDF as a visual attachment).

The entire workflow, from "I have a folder with three files" to "I have a formatted article viewable as a web page and printable as a PDF," took one Claude session. Claude never had access to any Git repo. Just a vault.

## Why This Matters

### Every Piece Solved a Problem I Had Been Fighting

The voice memo workflow was always good at capturing ideas. NotebookLM was always good at creating visual materials. Claude was always good at writing clean documents. The missing piece was assembly: combining the text and the images into a finished article. That assembly required shuffling files between tools, manually placing images, fighting with formatting.

Now: drop the folder into Claude, ask Claude to extract images and write the article, Claude pushes to a vault, I open the share link. Done.

### The Vault Is the Collaboration Layer

Claude and I collaborated through an encrypted vault. Claude created it. Claude pushed files to it. I viewed the results via a share link. At no point did Claude have access to my Git repos, my file system, or anything beyond the vault.

This is the same pattern I use for code development (patches in vaults), for the accountant pack (documents in vaults), and for sharing with partners (materials in vaults). The vault is becoming the universal collaboration layer between humans and AI.

### The Browse View Is Becoming a Publishing Platform

The SG/Send Browse view now renders markdown with inline images, clickable links, and CSV tables. It supports printing to PDF. This means a folder shared via SG/Send is not just a file dump. It is a readable, navigable document collection. Almost a mini website.

For content creation, this changes the game. The output of the Claude session is not a file I download and then upload somewhere else. The output is immediately viewable, shareable, and printable via a link.

### The Feedback Loop Closes

The workflow now has a complete feedback loop:

```
Voice memo
  → Otter transcription
    → Claude document (better than transcript)
      → NotebookLM infographics (better because the document is better)
        → Drop folder into Claude (document + infographics + slides)
          → Claude extracts images, writes articles
            → Vault + share link
              → Browse view (readable web page)
                → Print to PDF or copy to LinkedIn
```

Each step improves on the previous one. The voice memo is raw. The transcript is structured. The Claude document is polished. The NotebookLM infographics are visual. The final article combines all of them. And the whole thing is viewable as a web page and printable as a PDF from a single link.

## What I Want Next

Can I copy and paste the rendered article directly into a LinkedIn article editor? The formatting, the images, everything. If that works, the workflow goes from voice memo to published LinkedIn article without ever leaving the tools I already use.

Even if it does not work perfectly, the PDF fallback is solid. A well-formatted PDF with inline infographics and slide screenshots is a publishable artifact.

This workflow is worth documenting on the website. It is a genuine, daily-use workflow that demonstrates the technology doing something useful that was previously painful. Not hypothetical. Not a demo. The way I actually create content now.
