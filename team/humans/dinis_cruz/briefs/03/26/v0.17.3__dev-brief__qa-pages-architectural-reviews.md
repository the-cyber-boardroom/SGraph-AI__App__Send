## Clarifying the Vision

The QA pages at qa.send.sgraph.ai are documentation and test evidence. But they should be more than that. They should be **architectural reviews that someone with technical expertise can read to understand and verify what the code does.**

The key concept: there is a one-to-one relationship between a test and a page. That page should contain everything needed to review the functionality, on one page, without clicking away to GitHub or opening an IDE. Print it out, read it with a pen. That is the bar.

## What Each QA Page Should Contain

### The User Story

Start with the human context: "As a user, I want to share a file with someone and have them download it." This grounds the page. Before the code, before the screenshots, the reader knows what they are looking at and why it matters.

Connect to the semantic graphs and ontologies: which part of the application does this test cover? Where in the flow graph does this user journey sit?

### The Code (Actual Code, Not Links)

The page should contain the actual test code, inline, syntax highlighted. Not a link to GitHub. The actual code. When reviewing, the reader should not need to switch context to a different tool.

Additionally: the application code that the test exercises. If the test covers the upload workflow, include the relevant sections of the upload component code. The reviewer sees both what the test checks and what the code does, on the same page.

### The Screenshots (With Code Annotations)

Each screenshot is paired with the line(s) of code that produced it. The reader sees: "this action was performed" (the code) and "this is what happened" (the screenshot). The connection between action and result is explicit.

### The Blast Radius

This is the new concept. When a code change requires this test to be updated, the page should show the blast radius:

- Which code files changed
- Which tests were affected
- How many screenshots changed
- How many pages on the QA site needed updating

If you make a change and only one test needs updating, the blast radius is small. If you make a change and 15 tests need new screenshots and 8 pages need rewriting, that tells you something important about the coupling and impact of that change.

Over time, this creates a measurable record of change cost. The Accountant role (when introduced) can use this to quantify the cost of features, bugs, and refactorings in terms of blast radius.

### The Semantic Context

Where does this test sit in the ontology? Which entities does it exercise? Which state machine transitions does it verify? This connects the individual test page to the broader system understanding.

## How This Changes the Review Process

Today: developer makes a change, reviewer reads the diff on GitHub, maybe runs the code locally.

With enriched QA pages: developer makes a change, the CI generates updated QA pages, the reviewer reads the QA page which contains the user story, the test code, the application code, the screenshots (before and after), and the blast radius. One page. Everything needed.

**This is the pen-and-paper code review.** The reviewer does not need an IDE. They need one document that captures the full context of what changed and what it looks like.

## Implementation: Start with Two Examples

Do not try to enrich all pages at once. Pick two test pages and make them exemplary:

1. **Combined link flow** (upload + download in one test, multi-step, good complexity)
2. **Friendly token flow** (involves PBKDF2 derivation, good for showing application code)

For each:
- Add the user story at the top
- Embed the full test code (syntax highlighted)
- Embed the relevant application code sections
- Annotate each screenshot with the code line that triggered it
- Add semantic context (which flow graph states are covered)

Once these two are excellent, the pattern can be applied to all pages.

## Adding Video

### Browser Recording

Add a toggle to the Playwright execution: `record_video=True`. When enabled, Playwright records the browser session as a video file. This captures the full user journey as motion, not just static screenshots.

For now, save videos locally. Later, publish to the QA site or a vault.

### Video from Screenshots

An alternative (or complement): stitch the screenshots into a video with captions. Each screenshot becomes a frame. The caption explains the action. This produces a narrated walkthrough from the test artifacts that already exist.

Tools to explore: ffmpeg for stitching, adding text overlays for captions. LLMs could generate the caption text from the test code.

### The Movie Script Analogy

Each QA test page is a movie script:
- The user story is the plot
- The code is the stage directions
- The screenshots are the storyboard frames
- The video is the final cut

This framing is useful for the Sherpa and Ambassador: the same materials that verify the product also demonstrate the product. A video generated from a QA test is simultaneously a regression test artifact and a product demo.

### Future: LLM-Enhanced Materials

Once video generation works, explore:
- NotebookLM-style audio explanations of test pages
- Infographics generated from page schemas and flow graphs
- Animated diagrams showing state transitions during the test
- Narrated walkthroughs using LLM-generated voiceover

These are future enhancements, not immediate priorities. But the architecture should not prevent them.
