## Architect Review

### Overall Assessment

The brief is architecturally sound. The three-layer model (schemas → flow graphs → ontology) is the right decomposition. The pragmatic path ("start from QA, zoom out") is correct — Layer 1 is done, Layer 2 is next, Layer 3 follows naturally.

The key architectural insight: **the state machine definitions should be the single source of truth that both the tests and the application reference.** Currently the application has its own state machine (in `send-upload.js` and `send-download.js`) and the tests have their own implicit copy (in `wait_for_upload_state` calls). These are two independent descriptions of the same thing. When they diverge, bugs happen silently.

### Layer 2 Design — State Machine as Type_Safe

Following Type_Safe rules strictly (Reference Guide v0.5.7):

**Rule 5: Schemas are pure data — no methods.** The state machine definitions are schemas. Validation, Mermaid generation, and diff logic belong on separate helper/utility classes.

**Rule 3: No mutable defaults.** `List[State_Transition]` auto-converts to `Type_Safe__List` at construction — no `= []` needed.

**Rule 4: Ban raw primitives.** State names, triggers, and guards all get domain-specific Safe_Str subclasses. These are one-line definitions — create them if they don't exist in osbot-utils. The Safe_Str__Id base (alphanumerics + `-` `_`, max 128 chars) fits all SG/Send state names.

```python
from typing                                                                   import List
from osbot_utils.type_safe.Type_Safe                                          import Type_Safe
from osbot_utils.type_safe.primitives.domains.identifiers.safe_str.Safe_Str__Id import Safe_Str__Id

# Domain-specific Safe_Str types — create these in sg_send_qa/state_machines/primitives/
class Safe_Str__State_Name   (Safe_Str__Id): pass     # e.g. 'idle', 'confirming', 'complete'
class Safe_Str__Trigger_Name (Safe_Str__Id): pass     # e.g. 'file_selected', 'click_next'
class Safe_Str__Guard_Expr   (Safe_Str__Id): pass     # e.g. 'file_input_has_file'
class Safe_Str__Security_Tag (Safe_Str__Id): pass     # e.g. 'plaintext_in_memory', 'key_in_url'
class Safe_Str__Machine_Name (Safe_Str__Id): pass     # e.g. 'upload_wizard', 'download_flow'

# Pure data — no methods (Rule 5), no raw primitives (Rule 4)
class State_Transition(Type_Safe):
    from_state  : Safe_Str__State_Name                 # auto-inits to ''
    to_state    : Safe_Str__State_Name
    trigger     : Safe_Str__Trigger_Name               # 'file_selected', 'click_next', 'auto_advance'
    guard       : Safe_Str__Guard_Expr                 # 'file_input_has_file', 'delivery_mode_selected'
    data_out    : Safe_Str__Id                         # what data is produced at this transition
    security    : Safe_Str__Security_Tag               # 'plaintext_in_memory', 'key_in_url'

# Pure data — typed collections, no methods (Rules 3, 4, 5)
class State_Machine__Upload(Type_Safe):
    name        : Safe_Str__Machine_Name = 'upload_wizard'   # auto-converts str to Safe_Str__Machine_Name
    states      : List[Safe_Str__State_Name]                 # auto-converts to Type_Safe__List
    transitions : List[State_Transition]                     # auto-converts to Type_Safe__List

# Logic in a separate class (Rule 5)
class State_Machine__Utils(Type_Safe):

    def validate_transition(self, machine, from_state: str, to_state: str) -> bool:
        return any(t.from_state == from_state and t.to_state == to_state
                   for t in machine.transitions)

    def to_mermaid(self, machine) -> str:
        lines = ['stateDiagram-v2']
        for t in machine.transitions:
            label = t.trigger
            if t.guard:
                label += f' [{t.guard}]'
            lines.append(f'    {t.from_state} --> {t.to_state}: {label}')
        return '\n'.join(lines)

    def missing_transitions(self, machine, observed_transitions) -> list:
        expected = {(t.from_state, t.to_state) for t in machine.transitions}
        observed = {(t.from_state, t.to_state) for t in observed_transitions}
        return [t for t in machine.transitions if (t.from_state, t.to_state) not in observed]
```

**Serialisation is free.** Type_Safe's `.json()` recursively serialises all attributes, including nested `Type_Safe` objects in typed lists. `.from_json()` restores everything with full type information:

```python
upload_sm = State_Machine__Upload(
    states      = ['idle', 'file-ready', 'choosing-delivery', 'choosing-share',
                   'confirming', 'encrypting', 'uploading', 'completing', 'complete'],
    transitions = [
        State_Transition(from_state='idle',       to_state='file-ready',        trigger='file_selected'),
        State_Transition(from_state='file-ready',  to_state='choosing-delivery', trigger='auto_advance'),
        ...
    ]
)

json_data = upload_sm.json()                              # recursive, handles List[State_Transition]
restored  = State_Machine__Upload.from_json(json_data)    # full type restoration
assert type(restored.transitions[0]) is State_Transition  # preserved
```

No manual `to_json()`. No `setup()`. No `__init__` override. Type_Safe handles all of it.

**Why not Enum for states?** Enums are rigid. The SG/Send UI's state names are strings set via JS (`this._state = 'confirming'`). Using strings in Python matches the runtime truth. If we introduce an Enum, every new state requires a Python code change and redeploy. With strings, the state machine definition can be updated from a JSON file or discovered at runtime.

### Layer 2 Design — Where Do State Machines Live?

```
sg_send_qa/state_machines/
├── __init__.py
├── State_Transition.py                 — pure data class (shared by all machines)
├── State_Machine__Upload.py            — upload wizard: states + transitions (pure data)
├── State_Machine__Download.py          — download flow: states + transitions (pure data)
├── State_Machine__Gallery.py           — gallery flow: states + transitions (pure data)
├── State_Machine__Utils.py             — logic: validate, to_mermaid, diff (separate from data)
├── State_Machine__Snapshot.py          — observed transitions per test run (pure data)
└── schemas/                            — JSON exports (generated via .json(), committed)
    ├── upload.json
    ├── download.json
    └── gallery.json
```

The JSON files are generated from the Python classes (`.json()` — no custom serialiser), committed to the repo, and consumed by the Jekyll site for Mermaid rendering. The Python classes are the source of truth. The JSON is a build artifact.

### Version Comparison — Architecture

The graph diff concept is powerful but needs a concrete data model. Following Type_Safe rules — pure data, typed collections, no raw primitives:

```python
from typing                                              import List
from osbot_utils.type_safe.Type_Safe                     import Type_Safe
from osbot_utils.type_safe.primitives.domains.identifiers.safe_str.Safe_Str__Version import Safe_Str__Version
from osbot_utils.type_safe.primitives.domains.identifiers.Timestamp_Now              import Timestamp_Now
from sg_send_qa.state_machines.primitives                import Safe_Str__Machine_Name, Safe_Str__State_Name
from sg_send_qa.state_machines.State_Transition          import State_Transition

class State_Machine__Snapshot(Type_Safe):                    # pure data — observed transitions per run
    version                : Safe_Str__Version               # e.g. 'v0.3.0'
    timestamp              : Timestamp_Now                   # UTC milliseconds
    state_machine          : Safe_Str__Machine_Name          # e.g. 'upload'
    states_found           : List[Safe_Str__State_Name]      # states actually observed during test run
    transitions_observed   : List[State_Transition]          # (from, to, trigger) actually seen
    transitions_expected   : List[State_Transition]          # from the defined state machine
    missing                : List[State_Transition]          # expected but not observed
    unexpected             : List[State_Transition]          # observed but not expected
```

Serialisation is automatic: `snapshot.json()` produces the full nested structure. `State_Machine__Snapshot.from_json(data)` restores it with all `State_Transition` objects properly typed.

After a test run, produce one snapshot per state machine. Compare snapshots between versions. `missing` = regression. `unexpected` = undocumented behaviour (new feature or bug).

### Download State Machine

The download flow is more complex — it has branching:

```
loading → entry                        (no hash in URL — manual entry form)
loading → ready                        (hash present, transfer fetched)
loading → error                        (hash present, fetch failed)

entry → ready                          (valid transfer ID submitted)
entry → error                          (invalid transfer ID)
entry → entry                          (bogus token, form stays)

ready → decrypting                     (key entered + decrypt clicked)
ready → [browse|gallery|viewer]        (auto-route based on content type)

decrypting → complete                  (decryption succeeded)
decrypting → error                     (decryption failed)
```

The auto-routing from `ready` to `browse|gallery|viewer` is the interesting branching — the download component detects the content type and redirects. Same pure-data pattern:

## QA Lead Review

### How This Changes Testing

Currently, tests assert against individual states:
```python
assert self.sg_send.upload_state() == 'complete'
```

With a formalised state machine, tests can also assert against transitions:
```python
# After every action, validate the transition was expected
state_before = self.sg_send.upload_state()
self.sg_send.upload__click_next()
state_after = self.sg_send.upload_state()
assert upload_sm.validate_transition(state_before, state_after)
```

This turns every test into a graph edge validator. A test that passes confirms an edge. A test that fails means the expected edge doesn't exist (bug) or an unexpected edge was taken (bug or model update needed).

### The Bug as Graph Anomaly — Concrete Example

We already have one: `bug__separate_key_decrypt_content_not_visible.py`

**Expected graph edge:**
```
decrypting → complete (content visible in body)
```

**Actual behaviour:**
```
decrypting → complete (content in shadow DOM, not visible via inner_text("body"))
```

The state transition happened (`decrypting` → `complete`), but the **data output** was wrong — `content_text` should be populated but isn't. This is a graph anomaly at the data layer, not the state layer. The state machine said "complete" and it IS complete — but the schema snapshot (`extract__download_page().content_text == ""`) reveals the data expectation was violated.

This means the graph anomaly detection has two levels:
1. **State anomaly:** wrong state reached (or no state reached). Caught by `validate_transition()`.
2. **Data anomaly:** correct state reached, but schema snapshot doesn't match expectations. Caught by asserting against `Schema__*Page` fields.

Both are graph anomalies. Both should be reportable as such.

### Integration with QA API

The QA API (Task 2 from the dev brief) returns `page_model` in every response. If we add `transitions_observed` to the response, every API call produces a partial graph:

```json
{
    "status": "pass",
    "page_model": { "state": "complete", ... },
    "transitions_observed": [
        { "from": "idle", "to": "file-ready", "trigger": "file_selected" },
        { "from": "file-ready", "to": "choosing-delivery", "trigger": "auto_advance" },
        ...
    ]
}
```

Aggregate these across all API calls in a test run → you get the full observed graph for that version. Diff against the defined graph → missing edges = untested transitions, unexpected edges = bugs or new features.

### Test Coverage as Graph Coverage

Currently test coverage is measured by file/line. With state machines, coverage becomes:

```
Upload wizard: 8/8 transitions covered (100%)
Download flow: 6/10 transitions covered (60%)
  Missing: entry→ready, ready→gallery, ready→viewer, decrypting→error
Gallery flow: 0/5 transitions covered (0%) — no gallery state machine tests yet
```

This is more meaningful than line coverage — it answers "which user paths are verified?"

The QA site can render this as a Mermaid diagram with covered transitions in green and uncovered in red.

## AppSec Review

### Threat Modelling from the Graph — Concrete Application

The brief says "the graph IS the architecture diagram, the threat model IS the annotated graph." Here's how this applies to SG/Send specifically:

**Data sensitivity at each state:**

| State | Sensitive data present | Where |
|---|---|---|
| `idle` | None | — |
| `file-ready` | Plaintext file content | Browser memory (JS) |
| `encrypting` | Plaintext + key | Browser memory (JS) |
| `uploading` | Ciphertext + transfer ID | Network (HTTPS) + browser |
| `complete` | Transfer ID + key (in URL) | Browser URL bar + localStorage |
| `decrypting` | Ciphertext + key | Browser memory (JS) |
| `complete` (download) | Plaintext content | Browser DOM |

**Threat: what if a transition is skipped?**

```
file-ready → uploading (SKIP encrypting)
```
If the encryption step is bypassed, plaintext goes to the server. The zero-knowledge model is violated. The test `p0__zero_knowledge` exists specifically to check this — it verifies the server never sees plaintext. But it checks via the API, not via the state machine. With a formalised graph, the test can also verify: "was the `encrypting` state entered before `uploading`?"

**Threat: what if the key leaks via the URL?**

The `complete` state has the key in the URL hash (`#tid/key`). The friendly token mode exists specifically to avoid this — the key never appears in the URL. The state machine for token mode should show:

```
complete → key_in_url: false       (token mode)
complete → key_in_url: true        (combined mode)
```

Test `p0__friendly_token` already checks this: "after token resolution, the URL should NOT contain the key." This is a data annotation on the state, not a transition. The schema supports it — `Schema__Download_Page.transfer_id` is populated from the URL hash, and the test can verify the key portion is absent.

**Recommendation:** The `security` field already exists on `State_Transition` (added in the Architect review above). Use it to annotate upload and download flows:

```python
State_Transition(from_state='file-ready', to_state='encrypting',
                 trigger='click_encrypt_upload',
                 security='plaintext_in_memory'),
State_Transition(from_state='encrypting', to_state='uploading',
                 trigger='auto_advance',
                 security='ciphertext_only'),
State_Transition(from_state='completing', to_state='complete',
                 trigger='auto_advance',
                 security='key_in_url'),              # combined mode only
```

This makes the threat model part of the graph definition, not a separate document. The `security` field is a plain string — Rule 5 compliant (pure data, no methods). Security analysis logic (e.g. "find all transitions where `security == 'key_in_url'`") lives on `State_Machine__Utils`.

## Key Insight: The State Machine Is Already There

The most important observation from this review: **we are not designing state machines from scratch.** The SG/Send UI already HAS state machines (`send-upload._state`, `send-download.state`). The tests already USE those state machines (`wait_for_upload_state`, `wait_for_download_state`). The only thing missing is a formal definition that both the tests and the documentation can reference.

The work is extraction and formalisation, not invention. Every state name, every transition, every guard condition exists in either the UI JavaScript or the test Python. The task is to pull them into Type_Safe pure-data classes (schemas are data, logic is separate — Rule 5), let `.json()` / `.from_json()` handle serialisation automatically (no manual `to_json()`), and generate the artifacts (JSON, Mermaid, coverage maps) from the utility class that operates on the data.

## Type_Safe Best Practices for This Work

This section exists so the CCW implementation session has the rules in one place without needing the full Type_Safe reference guides.

### Rule 1: Always inherit from Type_Safe

Every class in `sg_send_qa/state_machines/` must inherit from `Type_Safe`. No plain classes. This gives automatic `__init__`, `__setattr__` enforcement, `.json()`, `.from_json()`, context manager (`with`), and `print()`.

### Rule 2: Type-annotate every attribute

No bare attributes. Every field has a type annotation. Type_Safe auto-initialises each to its type's default (Safe_Str → `''`, Safe_UInt → `0`, List[X] → `[]`).

### Rule 3: No mutable defaults — ever

```python
# ✗ WRONG — mutable default shared across instances
transitions : list = []

# ✓ CORRECT — Type_Safe auto-inits an empty Type_Safe__List per instance
transitions : List[State_Transition]
```

### Rule 4: Ban raw primitives

Never use `str`, `int`, `float`, `list`, `dict` in a Type_Safe class. Use `Safe_Str__*`, `Safe_UInt`, `Safe_Float__*`, `List[X]`, `Dict[K, V]` respectively.

For this project, create domain-specific subclasses in `sg_send_qa/state_machines/primitives.py`:

```python
from osbot_utils.type_safe.primitives.domains.identifiers.safe_str.Safe_Str__Id import Safe_Str__Id

class Safe_Str__State_Name   (Safe_Str__Id): pass     # 'idle', 'confirming', 'complete'
class Safe_Str__Trigger_Name (Safe_Str__Id): pass     # 'file_selected', 'click_next'
class Safe_Str__Guard_Expr   (Safe_Str__Id): pass     # 'file_input_has_file'
class Safe_Str__Security_Tag (Safe_Str__Id): pass     # 'plaintext_in_memory', 'key_in_url'
class Safe_Str__Machine_Name (Safe_Str__Id): pass     # 'upload_wizard', 'download_flow'
```

`Safe_Str__Id` base allows alphanumerics + `-` `_`, max 128 chars — fits all SG/Send state names. If a value needs different validation (e.g. dots, colons), use a different base (`Safe_Str__Key`, `Safe_Str__Label`).

### Rule 5: Schemas are pure data — no methods

State machine definitions (`State_Transition`, `State_Machine__Upload`, `State_Machine__Snapshot`) are data-only. No `to_mermaid()`, `validate_transition()`, or `to_json()` methods on them.

Logic lives on a separate class: `State_Machine__Utils`.

```python
# ✗ WRONG — method on a schema
class State_Machine__Upload(Type_Safe):
    ...
    def to_mermaid(self) -> str: ...          # NO — move to State_Machine__Utils

# ✓ CORRECT — logic in separate class
class State_Machine__Utils(Type_Safe):
    def to_mermaid(self, machine) -> str: ... # YES — operates on data
```

### Rule 6: No `__init__` override, no `setup()` method

Type_Safe handles all construction via constructor kwargs. Don't populate fields in a method — pass them at construction time:

```python
# ✗ WRONG — setup method populates fields
class State_Machine__Upload(Type_Safe):
    transitions : List[State_Transition]

    def setup(self):
        self.transitions = [State_Transition(...), ...]

# ✓ CORRECT — pass via constructor, Type_Safe assigns automatically
upload_sm = State_Machine__Upload(
    transitions = [State_Transition(from_state='idle', to_state='file-ready', trigger='file_selected')]
)
```

### Rule 7: Serialisation is free

`Type_Safe.json()` recursively serialises all attributes, including nested Type_Safe objects in typed lists. `Type_Safe.from_json(data)` restores everything with full types. Never write manual `to_json()` or `from_json()`.

```python
json_data = upload_sm.json()                              # handles List[State_Transition] recursively
restored  = State_Machine__Upload.from_json(json_data)    # full type restoration
assert type(restored.transitions[0]) is State_Transition  # preserved
```

### Rule 8: Never shadow reserved names

Don't use `json`, `obj`, `reset`, `print`, `from_json`, `serialize_to_dict`, `update_from_kwargs`, `__locals__`, `__kwargs__` as attribute names.

### Rule 9: Use `json_save_file` / `json_load_file` not raw `json` module

```python
from osbot_utils.utils.Json import json_save_file, json_load_file

json_save_file(upload_sm.json(), 'path/to/upload.json')     # safe, returns None on error
data = json_load_file('path/to/upload.json')                 # safe, returns {} on error
```

## Implementation Strategy: Start Small, Get It Right

**Do NOT model the entire application at once.** Start with two concrete use cases, build the full stack (Python classes → JSON → Mermaid → QA site page → tests using the model), validate the approach, then expand.

### First use case: Upload Wizard (happy path)

**Why first:** it's the simplest linear flow, all states are confirmed, all transitions exist in test code, and we have screenshots of every step on the QA site already.

Deliverables for this use case:
1. `primitives.py` — all Safe_Str subclasses
2. `State_Transition.py` — pure data class
3. `State_Machine__Upload.py` — pure data: 9 states, 9 transitions (linear happy path only)
4. `State_Machine__Utils.py` — `validate_transition()`, `to_mermaid()`
5. Unit tests: `.json()` round-trip, `validate_transition()` true/false, `to_mermaid()` output
6. JSON export: `sg_send_qa__site/_data/state_machines/upload.json`
7. QA site page with Mermaid diagram (see below)
8. One integration test that calls `State_Machine__Utils.validate_transition()` after every `upload__*()` action

**Stop here. Review. Get feedback. Then proceed to use case 2.**

### Second use case: Download Flow (branching)

**Why second:** it introduces branching (`ready` → `browse|gallery|viewer`), error states, and multiple entry points (`loading` → `entry` or `loading` → `ready`). This validates that the Type_Safe model handles non-linear flows.

Same deliverable stack as above, plus:
- Branch guards (`content_type_is_folder`, `content_type_is_single_file`)
- Error transitions (`decrypting` → `error`)
- Self-transitions (`entry` → `entry` on bogus token)

### What NOT to do yet

- Don't model the gallery state machine until upload + download are proven
- Don't build the ontology (Layer 3) until state machines are published on the QA site
- Don't build the snapshot diff tool until we have two versions to compare
- Don't optimise the Mermaid rendering until we have content to render
