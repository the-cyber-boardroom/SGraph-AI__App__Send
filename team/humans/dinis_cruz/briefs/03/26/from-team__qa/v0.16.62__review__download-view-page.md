## Overview

The download/view pages are what the recipient sees when they open a shared link. Depending on the route and content type, the user sees different layouts:
- `/browse/` — two-column viewer (folder tree + preview) for zips, or single-file viewer
- `/gallery/` — thumbnail grid for image-heavy zips (covered in gallery-lightbox review)
- `/download/` — auto-download on decrypt
- `/view/` — single file preview

This document covers issues found on the single-file view and the browse/download pages.

---

## Issue D-001: Single text file view — dark background instead of white

**What the reviewer said:** "use of black background, should be white like how we show the other text files"

**What's visible in the screenshot:** After opening the link for a text message (`message-2026-03-25T15-23-14.txt`), the two-column view shows:
- Left panel: filename (wrapping across two lines), "text 73 B" badge, upload date, Save Locally button, Copy Link / Email / Print buttons, download link with Copy button, "Decrypted in 0.26s" indicator, "Send another file" link
- Right panel: the text content "this is a secrert" displayed in white text on a **dark background**

The dark background makes the text hard to read and doesn't match how text files are shown elsewhere. Text/markdown/code content should be rendered on a white/light background for readability — the same way it appears in the print preview and in v0.2.x.

## Issue D-002: Single-file view lost brand/privacy statements from v0.2.x — REGRESSION

**What the reviewer said:** "from v0.2.x to v0.3.0 we lost quite a lot of important info and 'brand/privacy statements' that existed on the decryption page"

**Details:** The v0.2.x single-file view included additional context that gave users confidence in the security of the transfer:

The v0.3.0 view (shown in the screenshot) is functional but minimal — filename, type badge, size, date, action buttons, and the content. The security/brand messaging that built trust with the recipient is missing.

**Note:** This is related to the already-known "two-column single-file layout" regression (brief exists at `team/roles/dev/reviews/03/22/v0.16.45__brief__restore-two-column-single-file-layout.md`). ## Issue D-004: Text message filename should include SG/Send branding

**What the reviewer said:** "looking at the file name, I think we could put our branding in there :) something like sg-send__text__{date/time}"

**What's visible in pic1:** The downloaded file is saved as `message-2026-03-25T15-23-14.txt`. When opened locally, the browser shows the full path `/Users/diniscruz/Downloads/message-2026-03-25T15-23-14.txt`.

**Suggested naming:** `sg-send__text__2026-03-25T15-23-14.txt` — this brands every downloaded text file with the SG/Send name, making it immediately identifiable in the user's Downloads folder and reinforcing the brand every time they see the file.

## Issue D-005: No way to copy the text content — only the link

**What the reviewer said:** "I have a way to copy the link, but don't have a way to copy the value :)"

**What's visible in pic2:** The left panel has "Copy Link" button and "Download link" with a Copy button — both copy the URL. But there is no "Copy text" or "Copy content" button to copy the actual decrypted text content to clipboard.

**The problem:** For a text message, the most common action after viewing it would be to copy the text itself — not the link. The user has to manually select the text in the right panel and copy it. A "Copy text" button (or even making the text panel easily selectable with a copy button) would be a natural addition, especially for the "secret text" use case.

## Issue D-006: Email link from download page uses different text and opens in same tab

**What the reviewer said:** "email link on this download page (pic3) uses different text, and is opened in the same tab (i.e. it should be opened in a new tab)"

**What's visible in pic3:** Clicking the email (✉) button on the download page opens Gmail compose with:
- Subject: "Shared file via SG/Send"
- Body: "Shared file via SG/Send" (repeated), then the raw URL

**Two problems:**
1. **Different text from upload Done page email:** The Done page (U-019) uses "Secure file for you" / "I've sent you an encrypted file via SG/Send." But the download page email uses "Shared file via SG/Send" with just the URL. The two email templates are inconsistent — different subjects, different body text, different structure.
2. **Opens in same tab:** The `mailto:` link opens in the current tab, navigating away from the SG/Send download view. If the user clicks Back, they may lose context or need to re-decrypt. It should open the email compose in a new tab so the download view is preserved.

## Issue D-008: Print view for single text file needs improvement

**What the reviewer said:** "pic1 is the print page (which we need to improve)"

**What's visible in pic1:** Clicking the print button (🖨) on the single-file text view triggers the browser's native print dialog. The print preview shows:
- The entire SG/Send page chrome printed as-is: header with logo/tagline/language selector, filename, type badge, size, upload date, "Save Locally" button, "Copy Link" button, email/print icons, download link with Copy button, "Decrypted in 0.26s" badge, "Send another file" link
- The actual text content ("this is a secrert") appears at the bottom in a dark box
- The browser says "2 pages" — so even a tiny text message generates a two-page print

**The problem:** The print output includes all the UI chrome (buttons, links, badges) which is meaningless on paper. It should either:
- Use the SgPrint component (which already produces beautiful branded A4 output with just the content + SG/Send header), or
- Have proper `@media print` CSS that hides the left panel chrome and shows only the content in a readable format

The gallery lightbox already has a print button that could use SgPrint — this single-file view should do the same.

## Issue D-009: Loading/decryption page needs better UX — "Loading transfer info..." is bare

**What the reviewer said:** "pic2 is the page I see when I open the page https://main.send.sgraph.ai/en-gb/browse/#royal-stamp-1474 which has a weird message. This is also a bigger piece of work, since here we should be showing the user more details on what is happening and also those rotating messages about the SG/Send features/privacy"

**What's visible in pic2:** When opening a shared link, the user sees:
- SG/Send header
- A card with just the text "Loading transfer info..." in teal
- Footer with "Powered by SGraph / v0.16.62 · UI v0.3.0"
- Lots of empty space

**The problem:** This is the first thing a recipient sees when they click a shared link — it's their first impression of SG/Send. Currently it's a nearly empty page with a vague "Loading transfer info..." message. The user doesn't know:
- What SG/Send is
- Why their file is secure
- What's actually happening (resolving token? downloading encrypted data? decrypting?)
- How long it will take

**Suggested improvements:**
1. **Show what's happening:** Break the loading into visible stages — "Resolving token...", "Downloading encrypted data...", "Decrypting in your browser..." — so the user understands the zero-knowledge process
2. **Rotating feature/privacy messages:** While loading, show rotating messages about SG/Send's features and privacy: "Your file was encrypted before it left the sender's browser", "We never see your decryption key", "Zero cookies, zero tracking", "AES-256-GCM encryption — the same standard used by banks". This educates the recipient about the product while they wait.
3. **Brand presence:** This is a marketing moment — every recipient is a potential new user. The loading page should make them want to use SG/Send themselves.
4. **Progress indicator:** A progress bar or spinner with stage labels, not just static text.

**Note:** This ties into U-018 (Lambda cold start) — the loading page may be visible for several seconds on cold starts, making it even more important that this time is used productively.

## Issue D-007: Text workflow will be refactored — future note (NOT BLOCKING)

**What the reviewer said:** "all this 'text' workflow will be massively refactored and improved once we start implementing the ideas of 'one time only secret' and 'secret that can only be seen x times'"

**Details:** The current text upload/download flow is a basic implementation. Future plans include:
- **One-time secret:** text that can only be viewed once, then is permanently deleted
- **View-limited secret:** text that can only be viewed N times
- These features will require significant refactoring of both the upload flow (new options in Share mode) and the download/view flow (view counting, deletion after view, expiry messaging)

**Status:** NOT BLOCKING. Future planning note for the text workflow evolution.

## CRITICAL NOTE: All download pages are the first impression for new users

**What the reviewer said:** "add an important note to all these 'Download' pages, which is that this is the page that new users will see for the first time, so not only we need to make it really clear what is going on (and what to do), but this is our opportunity to convert a 'user' into a 'customer'"

**This applies to ALL download-side pages:**
- D-009: Loading/decryption page ("Loading transfer info...")
- D-001/D-002: Single-file view (text on dark background, missing brand/privacy statements)
- D-008: Print view
- D-010: Gallery view for text
- D-011: Download route auto-download confirmation

**The fundamental insight:** Every person who receives a shared file via SG/Send is seeing the product for the first time. They didn't choose to come here — someone sent them a link. This is SG/Send's primary acquisition funnel:

1. **Clarity:** The recipient needs to immediately understand what's happening — "Someone sent you an encrypted file. It was decrypted in your browser. The server never saw your data."
2. **Trust:** The privacy/zero-knowledge messaging builds confidence in the security of the transfer
3. **Conversion:** Every download page should have a clear, compelling call to action — "Send your own encrypted file" with a direct link to the upload page. This turns recipients into senders.

Currently, the download pages are functional but do almost nothing to educate, build trust, or convert. This is a significant missed opportunity.

**This should be treated as a cross-cutting concern** — not individual page fixes, but a unified "recipient experience" design pass that touches all download-side pages consistently.
