## Category 1: REGRESSIONS

### REG-001: Markdown lightbox missing rich typography

**Severity:** P1
**View:** Gallery → Lightbox
**Route:** `/en-gb/gallery/#<token>` (click on .md file in gallery)

**Expected (v0.2.x behaviour):** Markdown rendered in lightbox with full typography — styled headings (h1 with border-bottom), colored code (`#d63384`), styled pre blocks with background + border, blue links, blockquote with left border, proper table styling, box-shadow on container, generous padding, 70vw width.

**Actual (v0.3.0 behaviour):** White box with browser-default typography. No heading styles, no code colors, no table styling, no box-shadow. Container doesn't fill viewport width (only max-width: 800px, no explicit width).

**v0.2.x code reference:**
- File: `v0.2.3/_common/js/components/send-download/send-download-v027.js`
- Method: `injectStyles()` IIFE, lines 593–623
- What it did: Injected ~20 lines of CSS for `.v027-lightbox__md` with full descendant typography
- Note: No subsequent overlay modified these styles — this was the final state

**v0.3.0 code location:**
- File: `v0.3.0/_common/js/components/send-download/send-gallery.css`, lines 358–367
- What's missing: `.sg-lightbox__doc` has only basic container properties — zero descendant typography rules

**QA test (demonstrates the bug):**
- Navigate to `/en-gb/gallery/#<friendly-token>` with a zip containing a .md file
- Click on the markdown thumbnail to open lightbox
- ASSERT (current, buggy): markdown renders with browser-default typography in a narrow white box
- EXPECTED (after fix): styled headings, colored code, table borders, box-shadow, 70vw width

**Suggested fix approach:** Port v0.2.7 descendant typography rules into `send-gallery.css` under `.sg-lightbox__doc`. Add missing container properties (width: 70vw, box-shadow, padding: 2.5rem 3rem).

### REG-002: Lightbox missing Print and Share action buttons

**Severity:** P2
**View:** Gallery → Lightbox
**Route:** `/en-gb/gallery/#<token>`

**Expected (v0.2.x behaviour):** Lightbox header had three action buttons: "Save locally", Print (🖨), and Share (copy link). Print called `window.print()`. Share copied URL to clipboard with "✓ Copied!" feedback.

**Actual (v0.3.0 behaviour):** Lightbox footer only has counter, Present (PDF-only), and Save. No Print or Share buttons inside the lightbox. The gallery header has these buttons, but they're hidden behind the lightbox dark overlay.

**v0.2.x code reference:**
- File: `v0.2.3/_common/js/components/send-download/send-download-v027.js`
- Method: `_v025_openLightbox()`, lines 436–439 (HTML), lines 464–472 (event handlers)
- What it did: Print button → `window.print()`, Share button → `navigator.clipboard.writeText()` with visual feedback
- Note: v0.2.10 later upgraded the print button for markdown to use SgPrint (A4 print preview)

**v0.3.0 code location:**
- File: `v0.3.0/_common/js/components/send-download/send-gallery.js`, `_buildLightbox()` method (lines 198–217)
- What's missing: No Print or Share buttons in the lightbox HTML

**QA test:**
- Open lightbox on any file in gallery view
- ASSERT (current, buggy): only Save, Present (PDF), and counter visible in lightbox
- EXPECTED (after fix): Save, Print, Share buttons in lightbox header/footer

**Suggested fix approach:** Add Print and Share buttons to `_buildLightbox()`. Wire Print to `window.print()` (or SgPrint for markdown). Wire Share to clipboard copy with feedback.

### REG-003: File/Text mode toggle styling mismatch

**Severity:** P2
**View:** Upload — Step 1
**Route:** `/en-gb/`

**Expected (v0.2.x behaviour):** File/Text toggle used `.btn .btn-sm` — compact, borderless inactive state (`border: none`, `padding: 0.375rem 0.75rem`, `font-size: 0.8rem`). Looked like lightweight tab pills.

**Actual (v0.3.0 behaviour):** `.mode-toggle__btn` has visible border on inactive state (`border: 1px solid rgba(78, 205, 196, 0.15)`), bigger padding (`0.5rem 1rem`), larger font (`0.875rem`). Looks like two separate chunky buttons instead of a tab strip.

**v0.2.x code reference:**
- File: `v0.2.0/_common/js/components/send-upload/send-upload.js`, `renderModeTabs()` lines 98–107
- Styling: `v0.2.0/_common/css/shared-components.css` — `.btn` (borderless) + `.btn-sm` (0.375rem 0.75rem, 0.8rem)
- Note: No overlay ever modified this — unchanged from v0.2.0 base

**v0.3.0 code location:**
- File: `v0.3.0/_common/js/components/send-upload/upload-step-select/upload-step-select.css`, lines 325–350
- What's different: `.mode-toggle__btn` has border, bigger padding, bigger font

**QA test:**
- Navigate to `/en-gb/` upload page
- ASSERT (current): File/Text buttons have visible borders, look like standalone buttons
- EXPECTED (after fix): Borderless inactive state, compact sizing, tab-like appearance

**Suggested fix approach:** Remove border from `.mode-toggle__btn` base state, reduce padding and font-size to match v0.2.0's `.btn-sm`.

## Category 2: BUGS

**Severity:** P2
**View:** Upload — Step 1
**Route:** `/en-gb/`

**Description:** The File/Text toggle sits orphaned between the step indicator and the drop zone. There's too much visual gap — the toggle appears disconnected from the content it controls. The card's `padding: 24px` plus the step-select component's `margin-bottom: 1rem` on `.mode-toggle` create excessive separation.

**QA test:**
- Navigate to `/en-gb/` upload page

**Suggested fix approach:** Reduce separation — either move toggle closer to drop zone, tuck it inside the drop zone border, or reduce the margin.

## Category 3: FEATURE REQUESTS

### FEAT-001: Gallery lightbox markdown should match SgPrint quality

**Priority:** P3
**View:** Gallery → Lightbox (markdown files)

**Description:** The print preview (via SgPrint) renders markdown beautifully — branded header with SG/Send logo + filename, DM Sans / JetBrains Mono fonts, proper table rendering, clean heading hierarchy. The lightbox should aspire to this quality bar. This goes beyond what v0.2.7 had — it's a new quality bar.

**Notes:** SgPrint already exposes `PRINT_STYLES` as a public property. Could potentially reuse a subset of those styles for the lightbox doc container. Defer until after v0.2.x parity is achieved, unless it can be done quickly alongside REG-001.

### FEAT-002: Upload page should fit in single viewport

**Priority:** P2
**View:** Upload — Step 1
**Route:** `/en-gb/`

**Description:** The upload page doesn't fit in one screen — requires scrolling to see test files and footer. The page layout is stacked elements in `<body>` (header → main → footer) with no flex/viewport-lock. `body` has `min-height: 100vh` but no `display: flex; flex-direction: column`, so footer just flows below everything.

**Suggested approach:** Use `display: flex; flex-direction: column; height: 100vh` on body, `flex: 1` on main, sticky/always-visible footer. The footer could host additional UX elements.

---

### FEAT-003: Footer styling improvements

**Priority:** P2
**View:** Upload page footer
**Route:** `/en-gb/`

**Description:** Multiple issues with the footer:
1. **Version too faint** — `.sg-footer__version` has `opacity: 0.6` nested inside parent `.sg-footer` with `opacity: 0.7`, making it effectively ~0.42 opacity. Very hard to read.
2. **Takes two lines** — version string (`v0.16.62 · UI v0.3.0`) is `display: block` forcing it to a second line below "Powered by SGraph". Should be single-line.
3. **Too much padding** — `padding: 2rem 1.5rem 1.5rem` is generous, wastes vertical space.
4. **Version number is critical info** — it's how bugs get reported and builds get identified. Should be clearly readable.

**Suggested approach:** Put everything on one line (inline version after "Powered by SGraph"), increase version visibility (drop nested opacity, use at least 0.6 combined), reduce padding.

---

### FEAT-004: Test Files section needs rethink

**Priority:** P2
**View:** Upload page — Test Files area
**Route:** `/en-gb/`

**Description:** Multiple issues:
1. **Misleading text** — "Drag a file into the upload zone, or click to download" — the actual UX is just clicking the button, not dragging. "To test folder upload, drag any folder from your desktop" is also misleading since you just click.
2. **Folders not distinguishable** — "4 test images" and "Test folder" buttons look identical to file buttons. No visual differentiation for folders/collections.
3. **Quality and scope** — The test files are critical for demos. Current set is minimal (test-text.txt, test-data.json, etc.). Needs richer, more representative content.
4. **Layout** — The whole section needs a visual refresh to better communicate what each button does.

**Notes:** This is VERY important for demos. Defer detailed design until after parity, but flag as high priority for the demo readiness pass.

## Release Criteria Candidates

Features that should NEVER break — candidates for permanent regression tests:

1. Encryption: downloaded data is always AES-256-GCM ciphertext, never plaintext
2. Friendly tokens resolve correctly (word-word-NNNN → transfer ID + key)
3. Gallery lightbox renders all supported types (image, markdown, PDF, code)
4. Markdown in lightbox has styled typography (not browser defaults)
5. File/Text mode toggle works and switches input mode
6. Upload wizard completes all 6 steps successfully
7. Version number visible in footer

## Recommendations

1. **REG-001 (markdown lightbox typography) is the highest-impact regression** — it's immediately visible when anyone clicks a markdown file in gallery view. Quick fix: port ~20 lines of CSS from v0.2.7.
