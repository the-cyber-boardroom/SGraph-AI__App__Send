## Overview

The upload page is the main landing page of SG/Send. It contains:
- SG/Send header with branding, tagline, sgraph.ai link, and language selector
- Upload card with 6-step wizard (step indicator at top)
- File/Text mode toggle

## Issue U-001: File/Text toggle looks weird and breaks page flow

**What the reviewer said:** "this part looks weird, since those buttons are kinda by themselves and it breaks the flow of the page"

**What's visible in the screenshots:** The File and Text buttons sit between the step indicator ("Upload → Delivery → Share mode → Confirm → Encrypt & Upload → Done / Step 1 of 6") and the dashed drop zone box. There is significant empty space above and below the toggle. The buttons appear disconnected — they're not visually grouped with either the step indicator above or the drop zone below.

**Additional context from earlier review (localhost testing):** The reviewer also noted the buttons "look differently" compared to v0.2.x. In the current v0.3.0:
- Both buttons have visible rounded borders (inactive "Text" has a subtle grey border outline)
- The active "File" button is solid teal/green with dark text
- They are relatively large/chunky buttons
- They sit left-aligned with nothing else on their row

In v0.2.x, the same toggle used smaller, more compact buttons where the inactive state had no border at all — they looked like lightweight tab pills rather than two standalone boxed buttons.

**The combined effect:** The toggle feels orphaned. It's too prominent for a secondary control, and the gap between it and the drop zone makes the page feel like it has disconnected sections rather than a cohesive flow.

## Issue U-004: Test Files section needs a rethink

**What the reviewer said:**
- "the text 'Drag a file into the upload zone, or click to download.' and 'To test folder upload, drag any folder from your desktop into the upload zone.' is wrong since what happens is that the user just needs to click on those buttons (which is a nicer UI)"
- "the couple test 'folders' are not easily identified as folders"
- "this whole 'test files/folder' (which is VERY important) needs a rethink and layout improvement"
- "we also need to improve the quality and scope of these example files, since this is how a lot of demos will be done"

**What's visible in the screenshots:** The Test Files section shows:

Row 1: `test-text.txt 508 B` | `test-data.json 460 B` | `test-page.html 1.9 KB` | `test-image.png 522 B` | `test-photo-2.png 10.4 KB`

Row 2: `test-document.pdf 933 B` | `test-readme.md 1.0 KB` | `4 test images click to load` | `Test folder all 7 files`

Each item is a button/chip with a small file-type icon on the left, the filename, and the size. All buttons have the same visual style — the same dashed border, same layout. The two collection/folder items ("4 test images" and "Test folder") are not visually distinct from the individual file buttons.

**Specific problems:**
1. **Misleading instructions:** The text says "Drag a file into the upload zone, or click to download" but the actual UX is that clicking a test file button loads it into the upload wizard directly. The drag instruction is confusing and outdated.
2. **Folder text is also misleading:** "To test folder upload, drag any folder from your desktop into the upload zone" — same issue, you just click the "Test folder" button.
3. **Folders don't look like folders:** "4 test images" and "Test folder" have the same visual treatment as individual files. There's no visual cue (different border, background, size, icon emphasis) to indicate these are collections.
4. **Content quality:** The test files are small and generic (508 B text file, 460 B JSON, 933 B PDF). For demos, these need to be richer — a real-looking document, a meaningful image, representative content that showcases what SG/Send can do.
5. **Layout:** The section is a simple flex-wrap grid of chips. For something this important to the demo flow, it deserves a more thoughtful layout — perhaps grouped by type (files vs folders), with better visual hierarchy.

**Why this matters:** This is how a lot of demos will be done. A potential customer or investor will click a test file, watch it encrypt and upload, then view the result. The test files section is the starting point of that experience — it needs to look polished and contain compelling content.

**What the reviewer said:** "when I click on 'Show current token' (pic1) I get pic2, which still has the access token masked, where I need to click one more button to see it"

## Issue U-022: Done page needs security context for sharing methods

**What the reviewer said:** "on this simple view we need to capture somewhere (on this page or in a section that the user can click to see/expand), that mentions the security issues with the simple token, the email (all data is on URL) and the copy link (secret is in link)"

**Details:** The Done page shows sharing options (Simple token, Full link, QR code, Email link, Open in new tab) but provides no security context about the tradeoffs of each method. Users should understand:

- **Simple token** — the friendly token (`royal-stamp-1234`) derives both the transfer ID and decryption key. Anyone who knows this token can decrypt the file.
- **Email link** — the full decryption key is in the URL. If the email is intercepted or the recipient's email is compromised, the file can be decrypted. Email is not encrypted in transit by default.
- **Copy link** — the secret/key is embedded in the link. Anyone who has the link can decrypt the file. Sharing the link via insecure channels (chat, SMS) exposes the key.

This doesn't need to be in-your-face — it could be a collapsible "Security notes" section or an info icon that expands to show the tradeoffs. But it should be available on this page so users can make informed decisions about how they share.

## Issue U-018: Lambda cold start causes visible hang on first encrypt — needs API warm-up on page load (ARCHITECTURE)

**What the reviewer said:** "another important architecture issue to capture is what is happening with that first 'encrypt' request, which hangs for a bit, which is caused by a current known bug in the lambda (which has an expensive cold start). What is interesting is that as long as the user has a token in its browser local storage, and the entire UI is loaded from S3, this is the first time we actually have a call to the API. So what we need to do is that on all pages load, we make an async call to the backend which will create a hot lambda function, but will also allow us to detect if there are major issues with the API (we have a health API that we can use here)"

**Context:** The SG/Send architecture is:
- UI: static files served from S3 (no server rendering)
- Auth: access token stored in browser localStorage
- API: Lambda functions behind API Gateway

This means when a user lands on the upload page, the browser loads HTML/CSS/JS from S3 and checks localStorage for a token. Zero API calls happen. The **first** API call only occurs when the user clicks "Encrypt & Upload" on Step 5 — at which point the Lambda may be cold, causing a noticeable delay during what should feel like a fast operation.

**The visible effect:** During "Creating transfer" (shown in the screenshot), there's a hang before the progress bar moves. Reading (8ms) and Encrypting (115ms) are fast, but the transfer creation call hits a cold Lambda.

**Proposed solution:** On every page load, make an async background call to the health API endpoint. This serves two purposes:
1. **Warms the Lambda** — by the time the user completes Steps 1–4 and hits Encrypt & Upload, the Lambda is hot and responds fast
2. **Detects API issues early** — if the health check fails, the UI can show a warning before the user goes through the entire wizard only to fail at Step 5

This should be a fire-and-forget call — it should not block the UI or affect page load performance.

**Status:** NOT BLOCKING for v0.3.0 release, but important for production quality. The cold start is a known Lambda issue.

## Issue U-016: Text mode — Delivery step shows irrelevant options (Gallery, File viewer)

**What the reviewer said:** "the delivery page (pic1) is weird here, since two of the options are a bit weird (the folder and gallery). The download kind of makes sense, but I think in this case we could skip this section and go straight to share mode"

**What's visible in the screenshot:** After entering text ("message-2026-03-25T15-23-14.txt / Text · TXT · 17 B") and advancing to Step 2 (Delivery), the user sees the standard three delivery options:

1. **File viewer mode** — "Recipient reads/views the file directly with inline preview. Best for: documents, reports, code files" ★ DEFAULT
2. **Gallery mode** — "Recipient views the file in a gallery-style layout with metadata. Best for: photo sets, documents, mixed files"
3. **Download mode** — "Recipient gets a file to save to their device. Best for: large archives, backups, binary files"

**The problem:** For a plain text message, Gallery mode and File viewer mode don't make much sense. The user typed a short text message — showing it in a "gallery-style layout with metadata" or as a "file with inline preview" is over-engineered for what is essentially a note/message.

**Suggested approach:** For text-mode uploads, skip the Delivery step entirely and go straight to Share mode (Step 3). The sensible default for a text message is File viewer mode (so the recipient can read it). The user doesn't need to choose between gallery/viewer/download for a text snippet. This would also simplify the text workflow: type → share mode → confirm → encrypt & upload → done (4 steps instead of 6).

**Additional note:** In the screenshot, the "File viewer mode" card shows "★ DEFAULT" text but is not visually selected — it has the same border/background as the other two unselected cards. If the Delivery step is kept (rather than skipped for text mode), the default option should be pre-selected with a distinct visual state (e.g. teal border, highlighted background). However, this is moot if text mode skips Delivery entirely as suggested above.

## Issue U-015: Browser Back button navigates away from SG/Send instead of going to previous step — ARCHITECTURE

**What the reviewer said:** "if I use the browser 'back' functionality (for example using the back button), I actually go back to the previous page I loaded and NOT the previous phase. This last one needs to be captured differently, since what it means is that we need to start capturing the state of where we are, and support a SPA (Single Page Application) like architecture, where as we move on each step, we change the URL so that the user can use it to go back and make changes"

**Details:** The upload wizard is a multi-step flow within a single page. The browser has no knowledge of the wizard steps — from the browser's perspective, the URL never changed. So pressing the browser Back button doesn't go to the previous wizard step; it navigates to whatever page the user was on before SG/Send entirely.

**This is a fundamental architecture issue, not a quick fix.** The solution requires:
1. Using the History API (`pushState`/`replaceState`) to update the URL as the user moves through wizard steps (e.g. `/en-gb/?step=2`, or `/en-gb/upload/delivery`)
2. Listening for `popstate` events to handle browser Back/Forward
3. Preserving wizard state (entered text, selected file, chosen options) so it survives navigation
4. This effectively makes the upload page a SPA (Single Page Application)

**Status:** NOT BLOCKING for v0.3.0 release. Needs a full architecture planning phase. But it would solve U-014 (text loss on back) as a side effect, since state would be properly preserved.

## Issue U-009: Text mode UI needs improvement — button style, Next button, and workflow

**What the reviewer said (multiple points):**

1. "when I click on the 'Text' button, I get the UI in pic1 which also needs improvement, since the button is using our previous mode (for example it would be better to have that big 'Next' button here)"
2. "I think we should already have the 'Next' button visible (which will show up in the next step), since that will make the top menu consistent and not move"
3. "it would provide us a nice way to handle this text mode, since the Next button would be disabled, and enabled as soon as some text (maybe 5 chars) is there"
4. "it would also be good to have a shortcut to submit, maybe Cmd+Enter?"

**What's visible in the screenshot:** After clicking "Text", the UI shows:
- Step indicator: "Upload → Delivery → Share mode → Confirm → Encrypt & Upload → Done / Step 1 of 6" (no Next button in the header row — it only appears in Step 2+)
- File/Text toggle with "Text" now active (teal)
- A large textarea with placeholder "Type or paste text to encrypt and share..."
- Below the textarea: "0 characters" on the left, "Encrypt & Send" button (teal, small) on the right
- Below that: "Your text is encrypted in your browser before upload"
- Trust badge: "Zero cookies · Zero tracking · We cannot read your text"

**Problems identified:**

**a) "Encrypt & Send" button doesn't match the rest of the wizard.** In File mode, there is no action button on Step 1 — you just drop/select a file and the wizard advances. In Step 2+, the action button is the big "Next →" in the top-right of the header row. But in Text mode, the action is a small "Encrypt & Send" button below the textarea. This breaks the visual pattern of the wizard.

**b) The Next button should be visible from Step 1.** Currently the Next button only appears in Step 2 (Delivery). If it were always visible in the header row — disabled/greyed out on Step 1 until a file is selected or text is entered — the header layout would be consistent across all steps. **c) For Text mode specifically:** The Next button could start disabled and enable once the user has typed enough text (e.g. 5+ characters). This replaces the need for the separate "Encrypt & Send" button, keeping the workflow consistent with File mode: enter content → Next → Delivery → Share mode → etc.

**d) Missing keyboard shortcut.** There's no way to submit via keyboard. Cmd+Enter (Mac) / Ctrl+Enter (Windows) would be a natural shortcut for "I'm done typing, proceed." This is a standard pattern in text input UIs (Slack, GitHub comments, etc.).

## Issue U-006: Dropping files outside the drop zone opens them in a new browser tab

**What the reviewer said:** "the drop feature works quite nicely, but it will be better if we can drop anywhere on the page (since I had a couple times where I dropped a file outside that area and that just opens the file in a new tab, which is not a good experience)"

**What's visible in the screenshots:** The drop zone has a clear dashed teal border and shows a nice hover state when dragging over it (the border becomes bright teal/cyan). However, the droppable area is only the dashed rectangle inside the upload card — not the full page.

**The problem:** If you're dragging a file and release it slightly outside the dashed zone (on the card background, the step indicator area, the header, or anywhere else on the page), the browser's default behaviour kicks in: it opens the file directly in the tab. This navigates away from SG/Send entirely, losing the user's context. This is especially easy to do because the drop zone is a relatively small target compared to the full page.

**Expected behaviour:** The entire page should act as a drop target during Step 1. Dragging a file anywhere over the page should trigger the upload flow. At minimum, the page should prevent the browser's default drag-and-drop behaviour (`e.preventDefault()` on `dragover`/`drop` at the document level) so that missing the drop zone doesn't navigate away. Ideally, the whole page shows a visual cue when a file is being dragged over it.

**Note:** The existing drop-zone hover state (bright teal dashed border) looks nice. The page-wide drop should probably show a full-page overlay or highlight, then funnel into the same upload flow.
