## test-page.html (1.9 KB)

**View result (pic6):** HTML displayed as raw source code on dark background. No syntax highlighting — HTML tags, attributes, CSS properties, and values are all the same colour.

### Issue FT-002: HTML viewer should have syntax highlighting

**Details:** The HTML file is shown as raw source rather than either:
- **Rendered HTML** (showing the actual page as it would appear in a browser), or
- **Syntax-highlighted source** (with colour coding for tags, attributes, CSS, etc.)

Currently it's just plain text dump of the source. Same light background issue as D-001.

**The reviewer noted:** "color coding and CSS would be good here"

**Two possible approaches:**
1. **Render the HTML** — show the page as it would look in a browser (sandboxed iframe). This is the more useful view for the recipient.
2. **Syntax highlight the source** — if showing source, at minimum colour code HTML tags, attributes, CSS properties/values, and JS.

Ideally offer both: rendered view as default with a "View source" toggle.

### Issue FT-002b: HTML rendering must be safe (XSS prevention) + tracking pixel warning

**What the reviewer said:** "add a note to try to render safely the HTML (in a way that prevents XSS) since for example see pic1 for what that HTML looks like. Note: we should add a comment somewhere (and eventually try to detect it), that these type of HTML pages could allow tracking and identification of who/where the user is opening the file (via tracking pixels or HTML/JS/CSS resources loaded from that page)"

**What's visible in pic1:** The downloaded HTML file opened locally shows a well-formatted page: "SG/Send Test Page" heading in teal, "Zero-Knowledge Encryption" badge, a table of encryption properties (AES-256-GCM, 256 bits, Web Crypto API, Zero ciphertext only), privacy text, and SG/Send footer tagline. It's a compelling demo page.

**Security considerations for rendering HTML in the viewer:**

1. **XSS prevention:** If we render HTML in the viewer, it MUST be sandboxed to prevent script execution. Options:
   - `<iframe sandbox="allow-same-origin">` (no `allow-scripts`)
   - `srcdoc` attribute with CSP headers
   - DOMPurify sanitization before rendering

2. **Tracking pixel / resource loading warning:** Even without JavaScript, HTML can include:
   - `<img src="https://tracker.example.com/pixel.gif">` — tracking pixels that reveal the recipient's IP, location, and when they opened the file
   - `<link href="https://...">` — CSS resources that phone home
   - `<style>@import url('https://...')</style>` — CSS imports
   
   **The viewer should:**
   - Default to NOT loading external resources (block all network requests from rendered HTML)
   - Show a warning: "This HTML file may contain external resources that could track when and where you view it"
   - Optionally allow the user to "Load external resources" with informed consent
   - Eventually: detect and flag HTML files that contain external resource references

3. **Dual view approach:** Show syntax-highlighted source by default (safe — no execution). Offer "Render preview" as an opt-in with the security warnings above.

### Issue FT-003: HTML view — viewport not locked, content scrolls past header

**What the reviewer said:** "pic7 shows the problem that we have where we are not locking the screen top and footer (note how we lost the top bit)"

**What's visible in pic7:** After scrolling down in the HTML source view, the SG/Send header and left panel have scrolled off the top of the screen. Only the HTML source code is visible, filling the entire viewport. The user has lost the header, navigation, action buttons (Save, Copy Link, etc.), and all context about what they're looking at.

**The problem:** The right content panel (source code) doesn't have its own scroll container — it scrolls the entire page. The header and left panel should remain fixed/sticky while only the content panel scrolls.

**Note:** This likely affects all file types with long content (text, JSON, code, markdown), not just HTML. This is a cross-cutting layout issue for the single-file viewer.

### Issue FT-004: HTML print view same problems as other file types

**What's visible in pic8:** The print dialog shows the full page chrome (header, buttons, download link, file info) plus the raw HTML source below. Same issue as D-008 — should use SgPrint or proper `@media print` CSS to show only the content.

### Issue FT-005: "Upload and open in new window" power-user shortcut

**What the reviewer said:** "for power users (or regular users) it would be good to have an 'Upload and open in new window' option for users like me, that don't want to click 4 times after dropping a file. We need to find a nice way to put this on the UI, maybe with the mode that Apple does in macOS where pressing Shift changes some of the buttons. For example in some macOS apps the 'Save' menu item becomes 'Save As' — this is a nice power user mode."

**Details:** Currently, uploading a file and previewing it requires: drop file → Next (Delivery) → Next (Share mode) → Confirm → Encrypt & Upload → Open in new tab. That's ~4-5 clicks minimum after the initial drop.

For power users who frequently upload and immediately want to verify/share, a one-click "Upload & Open" option would be valuable. Implementation ideas:
- **Shift+Drop** or **Shift+Click** modifies the behaviour — skips the wizard steps, uses defaults, uploads immediately, and opens the result in a new tab
- Like macOS where holding Shift changes "Save" to "Save As" — holding Shift could change "Next →" to "Upload & Open ↗"
- Could also be a modifier on the test file buttons (Shift+click = upload with defaults and open)

### Issue FT-006: User preferences saved in localStorage for common options and defaults

**What the reviewer said:** "we could also add some options for the user (saved in local storage) with common options and defaults"

**Details:** Frequently-used settings could be persisted in localStorage so users don't have to make the same choices every time:
- Default delivery mode (File viewer / Gallery / Download)
- Default share mode (Simple token / Combined link / Link + key separate)
- Whether to skip certain wizard steps
- Power user mode on/off
- Preferred view mode in gallery (compact / grid / large)

This ties into U-008 (user/settings page) — the settings page would be where users configure these preferences, and localStorage would persist them.

### Issue FT-010: PDF viewer's download button saves file with GUID filename

**What the reviewer said:** "when I click on the download button in the pdf viewer (on the right) I get a GUID"

**What's visible in pic3:** Clicking the download button in the pdf.js toolbar (top-right of the PDF panel) opens a Save dialog with the filename `805cd735-acb4-40c6-9bed-bad25...` — a raw GUID, not the actual filename.

**Pic4 confirms:** Using SG/Send's own "Save Locally" button (left panel) works correctly — the file downloads as `2) v0.16.26__brief__investor-positioning.md (1).pdf` with the correct name. The issue is only with the pdf.js toolbar's built-in download button.

**The problem:** The PDF is loaded into pdf.js via a blob URL, and pdf.js uses the blob URL's identifier as the filename. The actual filename needs to be passed to pdf.js so its download button uses the correct name.

### Issue FT-011: Print button on PDF single-file view is broken

**What the reviewer said:** "our print button (on the left) is broken (pic5 and pic6)"

**What's visible in pic5:** Clicking the print button (🖨) on the left panel for the small PDF shows: all page chrome (header, buttons, download link) printed, plus the PDF content area appears as a mostly dark/blank rectangle with a small white area. The PDF content is not rendered in the print output.

**Pic6:** Same issue for the large PDF — the print preview shows page chrome + a checkerboard/grid pattern where the PDF content should be. The actual PDF pages are not printed.

**The problem:** The print button calls `window.print()` which tries to print the page including the pdf.js iframe/canvas. The PDF content rendered in canvas elements doesn't print correctly via the browser's native print. For PDFs specifically, the print should either:
- Use pdf.js's built-in print functionality (which renders pages to a hidden iframe optimized for printing)
- Or open the PDF blob URL directly and trigger print on that

### Issue FT-012: PDF view missing "Present mode" / fullscreen button — REGRESSION

**What the reviewer said:** "we are also missing a 'present mode / full screen' mode/button, which I believe was in v0.2.x"

**Details:** The v0.2.x viewer had a fullscreen/present mode button for PDFs (added in v0.2.13) that opened the PDF in a dedicated fullscreen viewer — one page at a time with arrow key navigation and a page counter. This was useful for presenting PDFs (e.g. pitch decks like pic2) to an audience.

The v0.3.0 single-file view has no fullscreen/present mode button. The pdf.js toolbar has its own presentation mode, but the SG/Send-branded present mode (which existed in v0.2.x) is missing from the left panel controls.

**Note:** The gallery lightbox does have a "Present" button for PDFs (added in patch 0018), but the single-file browse view does not.

## Large upload with videos (~160 MB) — Gallery crash

### Issue FT-018: Gallery view hangs and crashes on large uploads with videos

**What the reviewer said:** "when uploading multiple folders (in a parent folder) with about 160M, which contained some videos — the upload worked ok — but in the gallery view, the viewer hanged and crashed the page — the folder view was ok"

**Details:**
- Upload of ~160 MB containing multiple folders with videos completed successfully
- Gallery view attempted to render but hung and eventually crashed the browser tab
- Folder/browse view handled the same content without issues

**Root cause likely:** The gallery view tries to generate/load thumbnails for all files at once. For large uploads with videos, this means:
- Attempting to decode video frames for thumbnails (expensive)
- Loading many large files into memory simultaneously
- No lazy loading or virtualization for the gallery grid

The folder view works because it only loads content on-demand (when you click a file in the tree).

### Issue FT-019: Gallery should NOT be the default view for v0.3.0 release

**What the reviewer said:** "this is another reason why for this release we actually shouldn't have the gallery to be the default. It is not as stable and nice as the browser/folder view"

**Recommendation:** For the v0.3.0 release, change the default delivery mode from "Gallery mode" to "Folder view mode" for multi-file uploads. Reasons:

1. **Stability** — Gallery crashes on large uploads with videos (FT-018). Folder view handles all content sizes.
2. **Maturity** — The folder/browse view with sg-layout is the more polished, feature-complete component. It handles all file types, has the tab system, multi-pane layout, and proper on-demand loading.
3. **Consistency** — The folder view provides a consistent experience regardless of content type or size.
4. **Gallery can still be offered** — Keep it as an option (not removed), but don't default to it until the stability and thumbnail resolution issues (FT-007, FT-018) are resolved.

**This should be added to the ship blocker list** — changing the default is a one-line change but has significant impact on the out-of-box experience.

## Multi-folder drop — BUG

### Issue FT-015: Dropping folders (or selecting files+folders together) fails — files-only works

**Initial finding:** Dropping two folders only picks up 2 files and upload fails with "file or directory not found".

**Additional testing confirms:**
- Pic1/2/3: Selecting ALL items (files + folders) from Finder and dropping → same error. Shows "13 files / 22.6 MB" on Delivery step but then fails with the same "A requested file or directory could not be found" error.
- Pic4/5: Selecting ONLY the files (no folders) from the same Finder location → **works perfectly**. Shows "11 files / 22.6 MB / 5 documents, 4 images, 2 others" and uploads successfully to Done page with token `onset-hazel-2694`.

**Root cause narrowed:** The bug is specifically about **folder entries in the drag-and-drop DataTransfer**. When folders are included in the drop, the file traversal fails. Files-only drops work correctly. This is likely a `webkitGetAsEntry()` / `FileSystemDirectoryReader` issue where folder entries go stale before they can be read.
