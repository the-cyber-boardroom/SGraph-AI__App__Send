# Robust Upload and Download Components: Retry, Resume, and Speed Detection

**version** v0.17.3
**date** 26 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Architect, QA
**type** Dev brief

---

## Why This Matters Now

Two things happened that make this urgent:

1. **A user trying to download from WeTransfer** got stuck at 53% on a multi-gigabyte file. No ability to recover. No visibility into what was happening. No pause/resume. Just stuck. This is not a WeTransfer-specific problem. It is the industry standard, and the industry standard is terrible.

2. **A real SG/Send user hit three failures in sequence:** uploading a file from Apple/Google Drive that had not been downloaded locally (file not found), an internet dropout mid-upload (undetected, just showed an error), and a 6GB file that exceeded limits (another error). Each of these is a solvable problem, but all three were handled with generic error messages and no recovery path.

These are the moments when invisible infrastructure becomes visible. Upload and download work perfectly 95% of the time and nobody notices. The other 5% is where trust is built or lost.

## Upload: What Needs to Happen

### Error Detection and Recovery

| Failure Mode | Current Behaviour | Target Behaviour |
|---|---|---|
| File from cloud drive (not local) | "File not found" error | Detect, explain: "This file is on Google Drive. Download it first, then upload." |
| Internet drops mid-upload | Generic error, upload lost | Detect connectivity loss, pause, auto-retry when connection returns |
| File exceeds size limit | Error | Detect before upload starts, show limit clearly, suggest alternatives |
| Browser tab closed mid-upload | Upload lost | Warn before closing ("Upload in progress"), offer resume on return |
| Slow/mobile connection | No indication | Show speed, estimated time remaining, suggest Wi-Fi |

### Chunked Upload with Resume

We already break files into chunks for parallel upload. Extend this:

- Track which chunks have been uploaded successfully
- On failure, resume from the last successful chunk (do not re-upload the entire file)
- On retry, verify existing chunks are intact before continuing
- Show per-chunk progress ("45 of 120 chunks uploaded")

### Speed Measurement

Measure the upload speed in real time:

- Show current speed (e.g. "2.4 MB/s")
- Calculate and show estimated time remaining
- Detect speed changes (switched from Wi-Fi to mobile, speed dropped)
- Use speed measurement to optimize parallel thread count (fast connection = more threads, slow connection = fewer threads to avoid congestion)

### Speed Test Tool

Create a standalone speed test tool (similar to speedtest.net) that measures upload and download speed to the SG/Send infrastructure. This serves as:

- A diagnostic tool ("is my connection fast enough for this file?")
- A marketing tool (people share speed test results)
- A baseline measurement for the QA and monitoring systems

Publish on tools.sgraph.ai.

### Upload Visibility

During upload, show maximum information:

- Current file being uploaded (name, size)
- Overall progress (files completed / total, percentage, bytes transferred)
- Per-file progress (for multi-file uploads)
- Upload speed (real-time)
- Estimated time remaining
- Thread count (and option to adjust: "Using 4 parallel connections")
- Encryption status ("Encrypting..." / "Encrypted, uploading...")

Most of the time, this information scrolls by too fast to read. That is fine. It is there for when the upload is slow or stuck, and the user needs to know what is happening.

## Download: What Needs to Happen

### Same Principles Apply

| Failure Mode | Current Behaviour | Target Behaviour |
|---|---|---|
| Download stalls mid-way | Stuck with no indication | Detect stall, show message, auto-retry |
| Internet drops | Failed download, start over | Pause, resume from where it stopped |
| Large file on slow connection | No progress indication beyond spinner | Show speed, percentage, estimated time |
| Partial download (browser closed) | Start over | Detect existing partial, offer resume |

### Chunked Download with Resume

Mirror the upload chunking:

- Download in parallel chunks
- Track successful chunks
- Resume from last successful chunk on failure
- Show per-chunk progress

### Decryption Progress

For large encrypted files, the decryption step can take noticeable time. Show:

- "Downloading... 78%"
- "Decrypting..."
- "Ready to view"

Do not leave the user staring at a spinner with no indication of what is happening.

## Battle-Hardened Components

These upload and download components should be treated as core primitives:

- **Independently testable:** The upload component works in isolation. The download component works in isolation. Each has its own test suite covering every failure mode.
- **Distributable as a library:** Someone who wants to add encrypted upload/download to their own product should be able to use our component. This is the open source ecosystem play: widespread adoption finds corner cases and makes the platform more robust.
- **Exhaustively tested in the QA environment:** Every failure mode listed above should have a QA test that simulates it (network dropout, speed throttling, file-not-found, oversized file, etc.). The Playwright environment can simulate these conditions.

## Connection to Other Briefs

- **Focused use-case UIs:** The "Share a Video" workflow needs robust large-file upload. The portfolio view needs robust multi-file upload with resume.
- **Browser Automation API:** The QA infrastructure needs to simulate network failures and slow connections to test these recovery paths.
- **Speed test tool:** Joins the tools.sgraph.ai collection alongside the other mini tools.
