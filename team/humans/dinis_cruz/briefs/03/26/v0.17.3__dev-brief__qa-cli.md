## What We Need

A command-line interface for the QA automation that separates triggering from execution. The CLI drives the FastAPI service. The FastAPI service drives Playwright. The CLI can invoke the FastAPI locally, on a Lambda, on an EC2 instance, or anywhere else the service is deployed.

```
CLI (triggers)
  │
  ├── --target local     → calls FastAPI on localhost
  ├── --target lambda    → calls FastAPI on Lambda function URL
  ├── --target ec2       → calls FastAPI on EC2 instance
  │
  └── FastAPI (executes)
        │
        └── Playwright (browser automation)
              │
              └── Target website (dev, main, prod)
```

This separation means the CLI is lightweight (makes HTTP calls) and the execution environment is flexible (runs wherever the FastAPI is deployed).

## CLI Architecture

Take inspiration from the existing SGit CLI (Type_Safe structure, clean command hierarchy). The CLI should use the same architectural patterns.

### Core Commands

```bash
# Health and status
sg-qa health                          # Check FastAPI service is reachable
sg-qa status                          # Current session info, target server

# Single workflow execution
sg-qa run upload-file                 # Run the file upload workflow
sg-qa run download-file               # Run the file download workflow
sg-qa run gallery-view                # Run the gallery view workflow
sg-qa run friendly-token              # Run the friendly token flow
sg-qa run combined-link               # Run the full upload + download

# Run against a specific target
sg-qa run upload-file --target prod   # Run against production
sg-qa run upload-file --target dev    # Run against dev server
sg-qa run upload-file --target local  # Run against local server

# Multiple users / traffic generation
sg-qa traffic --users 5 --workflow upload-file    # 5 parallel users
sg-qa traffic --users 20 --workflow combined-link # 20 parallel round trips
sg-qa traffic --users 10 --mix                    # 10 users, random workflow mix

# Session management
sg-qa session start                   # Start a new session (creates run folder)
sg-qa session stop                    # End session, package artifacts
sg-qa session list                    # List past sessions
sg-qa session view <id>               # View session results

# Data and reporting
sg-qa report <session-id>             # Generate a report from session data
sg-qa report --latest                 # Report from most recent session
sg-qa export <session-id> --vault     # Export session to a vault
sg-qa export <session-id> --s3        # Export session to S3
```

### The FastAPI Endpoint Parameter

The CLI needs a configuration for where the FastAPI service lives:

```bash
sg-qa config set endpoint http://localhost:8000     # Local
sg-qa config set endpoint https://qa-lambda.sgraph.ai  # Lambda
sg-qa config set api-key <key>                       # API key for auth
```

Or pass per-command:

```bash
sg-qa run upload-file --endpoint https://qa-lambda.sgraph.ai --api-key <key>
```

## Sessions and Run Capture

### The Session Concept

Every CLI invocation produces data. A session groups related invocations:

```bash
sg-qa session start
# Session abc123 started

sg-qa run upload-file
sg-qa run download-file
sg-qa run gallery-view

sg-qa session stop
# Session abc123 complete: 3 runs, 0 failures
```

### The Run Folder

Each session produces a folder (conceptually, a vault) containing all artifacts:

```
sessions/
  abc123/
    metadata.json           # Session config, target, timestamps
    runs/
      001-upload-file/
        result.json         # Success/failure, timing, assertions
        metrics.prom        # Prometheus-formatted metrics
        screenshots/        # If captured
        trace.json          # Playwright trace (if enabled)
      002-download-file/
        result.json
        metrics.prom
      003-gallery-view/
        result.json
        metrics.prom
    summary.json            # Aggregated results, pass/fail counts
    report.md               # Human-readable report (auto-generated)
```

This folder can be:
- Saved locally (default)
- Pushed to a vault (`sg-qa export --vault`)
- Uploaded to S3 (`sg-qa export --s3`)
- Sent to Grafana/Prometheus (metrics extracted and pushed)
- Sent to OpenSearch/Elastic (logs and traces shipped)

### The Vault Connection

The run folder IS a vault. The same `sg-send-cli` tooling that handles encrypted file transfer handles QA session data. Push a session to a vault. Share the vault key. Anyone with the key can review the full session: results, screenshots, traces, metrics. This is the same pattern used for code patches and developer communication.

## Traffic Generation

The traffic generation mode runs multiple users in parallel:

```bash
sg-qa traffic --users 20 --workflow combined-link --target prod --duration 5m
```

This:
1. Starts 20 parallel browser sessions (or HTTP sessions)
2. Each runs the combined-link workflow (upload + download)
3. Runs for 5 minutes (recycling users as workflows complete)
4. Collects timing, success/failure, and metrics from all sessions
5. Produces a session folder with aggregated results

For HTTP-only traffic (no browser, faster, cheaper):

```bash
sg-qa traffic --users 100 --mode http --target prod --duration 10m
```

## LLM Analysis (Future)

Add an option to send session results to an LLM for analysis:

```bash
sg-qa analyse <session-id>                    # Analyse with default LLM
sg-qa analyse <session-id> --model claude     # Use Claude
sg-qa analyse <session-id> --model local      # Use local model
sg-qa analyse <session-id> --api-key <key>    # OpenRouter or direct API
```

The LLM receives the session summary, any failure details, and optionally screenshots. It produces an analysis: what failed, likely causes, suggested fixes. This is the one-shot analysis pattern: give the LLM the evidence, get back a diagnosis.

Not a priority for the initial CLI. But the architecture should make it easy to add (the session folder is already structured data that an LLM can consume).

## Three-Layer Architecture

```
Layer 1: CLI (sg-qa)
  - Triggers execution
  - Manages sessions
  - Generates reports
  - Exports data

Layer 2: FastAPI Service
  - Runs on Lambda / EC2 / local
  - Executes Playwright workflows
  - Returns results as JSON

Layer 3: Observability
  - Grafana (dashboards from Prometheus metrics)
  - OpenSearch (searchable logs and traces)
  - Vault (session artifacts for sharing and review)
```

Each layer is independent. The CLI does not know how the FastAPI works. The FastAPI does not know where its results end up. The observability layer consumes whatever data the other layers produce.

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | CLI built with Type_Safe architecture (matching SGit CLI patterns) |
| 2 | `sg-qa run <workflow>` executes a single workflow via FastAPI |
| 3 | `--target` parameter points at local, dev, or prod FastAPI endpoint |
| 4 | `--endpoint` and `--api-key` configuration for flexible deployment |
| 5 | Session management: start, stop, list, view |
| 6 | Run folder captures results, metrics, screenshots, traces per execution |
| 7 | `sg-qa traffic --users N` runs parallel browser or HTTP sessions |
| 8 | Session exportable to vault, S3, or local folder |
| 9 | Prometheus metrics extractable from session data |
| 10 | LLM analysis option noted in architecture (future implementation) |
