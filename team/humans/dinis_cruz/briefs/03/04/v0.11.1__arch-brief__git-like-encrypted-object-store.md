# Architecture Brief: Git-Like Encrypted Object Store for Vaults

**version** v0.11.1  
**date** 4 Mar 2026  
**from** Human (project lead)  
**to** Architect (lead), Developer  
**type** Architecture brief — vault internals, distributed storage, conflict resolution

## The Problem

The current vault has a flat tree structure that creates real problems at scale:

**Overwrite bug:** Two sessions saving files to disk simultaneously create conflicts. There is no merge strategy — last write wins, and content is lost.

**Single tree bottleneck:** The entire tree view is one unit. Every change syncs the whole tree. With nested vaults, workspaces, and multiple users, this creates unnecessary contention and bandwidth waste.

**No history:** There is no commit structure. You cannot see what changed, when, or by whom. You cannot roll back. You cannot branch.

**No conflict resolution:** When two people edit the same vault, there is no structured way to detect or resolve conflicts. The current workaround (refresh and hope) does not scale.

The human's conclusion: **stop inventing solutions to these problems. Git already solved all of them. Copy Git.**

---

## The Principle: When In Doubt, Do What Git Did

Git was designed by some of the best minds in the industry to solve exactly these problems: distributed content, multiple authors, branching, merging, conflict detection, offline operation, and cryptographic integrity. The architecture works. The data structures are proven at massive scale.

The principle is not "build Git" — it is "use Git's data structures and algorithms as the internal architecture of the vault, with an encryption layer on top."

Linus Torvalds looked at SVN and did the exact opposite. We should look at Git and do exactly the same thing, then add encryption.

## The Architecture: Encrypted Content-Addressable Object Store

### Git's Core Model (What We're Copying)

Git stores everything as objects in a content-addressable store. There are four object types:

| Object Type | What It Stores | Identified By |
|---|---|---|
| **Blob** | File content | SHA hash of content |
| **Tree** | Directory listing (pointers to blobs and other trees) | SHA hash of the tree structure |
| **Commit** | Snapshot of the tree + parent commit + metadata | SHA hash of the commit |
| **Tag** | Named reference to a commit | Name → commit hash |

Trees can contain other trees (directories within directories). Commits point to a tree (the state of the entire vault at that moment) and to parent commits (the history). This is how Git supports trees within trees within trees — and how we solve the "single tree bottleneck."

### What We Add: Encryption

Every object in the store is encrypted. The encrypted blob is stored on the server. The hash is of the encrypted content (so the server can do content-addressable storage without knowing what's inside).

```
Git object:     content → SHA(content) → stored by hash
Our object:     content → encrypt(content, key) → SHA(encrypted) → stored by hash
```

The server sees: a content-addressable store of encrypted blobs. It can store them, serve them by hash, and detect duplicates — but it cannot read any of them.

### Clone = Download Everything (Encrypted)

When you clone a vault, you download the entire object store — every blob, every tree, every commit. The full history. All encrypted.

This is the distributed property: once cloned, you have everything. You can work offline. You can browse history. You can create branches. All locally, all encrypted.

**Performance consideration:** For large vaults, this could be a lot of data. Git solves this with packfiles (compressed collections of objects). We can do the same — or we can start with the simple model (one object per file) and optimise later.

### Multi-Key Encryption Within a Single Vault

This is where the architecture becomes more powerful than Git. Because every object is independently encrypted, different objects can be encrypted with different keys:

```
vault/
  object-A  ← encrypted with team key (everyone can read)
  object-B  ← encrypted with team key
  object-C  ← encrypted with Alice's public key (only Alice can read)
  object-D  ← encrypted with Bob's public key (only Bob can read)
```

You can clone the vault and receive all objects — but you can only decrypt the objects for which you have the corresponding key. The rest are opaque blobs you carry around but cannot read.

**Use cases:**
- Shared vault where some files are visible to everyone, others are restricted
- Data room where the auditor can see financial documents but not HR documents
- Multi-tenant vault where each tenant's data is encrypted with their own key

### Commits: Automatic for Now

Git has a staging area (git add) and an explicit commit step. For our MVP:

**Skip staging. Every save is a commit.** When you save a change, the system automatically creates a commit object: captures the current tree state, points to the parent commit, records a timestamp and author (public key).

This gives us history without requiring the user to learn Git workflows. Later, we can add explicit staging, commit messages, and interactive rebase for power users.

### Branches and Pull Requests (Future)

Once we have the commit tree, branches are trivial: a branch is just a named pointer to a commit. Creating a branch is creating a pointer. Switching branches is loading a different tree.

Pull requests follow: "I've made changes on my branch, please review and merge into main." The vault becomes a collaborative editing environment with full version control.

## How This Solves the Current Problems

| Problem | Git Solution |
|---|---|
| **Overwrite bug** | Commits are immutable. Two concurrent saves create two commits. Merge logic combines them. No data loss. |
| **Single tree bottleneck** | Trees contain trees. Only changed subtrees need to be synced. |
| **No history** | Every change is a commit. Full history by construction. |
| **No conflict resolution** | Three-way merge (common ancestor + both changes). Same algorithm Git uses. |
| **No rollback** | Check out any previous commit. History is immutable. |
| **No offline access** | Full clone = full offline capability. |
| **No diffs** | Diff any two commits, any two versions of a file. |

## Trees Within Trees

The current flat tree structure means one tree object holds the entire vault structure. With Git-style trees:

```
Root tree
├── blob: README.md
├── tree: workspace-1/
│   ├── blob: source.html
│   ├── blob: view-1.html
│   └── tree: prompts/
│       ├── blob: step-1.txt
│       └── blob: step-2.txt
├── tree: workspace-2/
│   └── ...
└── tree: shared/
    └── ...
```

Changing a file in workspace-1 only updates that subtree and the root tree. workspace-2's tree is unchanged — same hash, no sync needed. This is how Git handles repositories with millions of files efficiently.

## Signed Content

Every commit can include a cryptographic signature from the author's private key. This provides:

- **Authorship:** Proof of who made the change
- **Integrity:** Proof the content hasn't been tampered with
- **Non-repudiation:** The author cannot deny making the change

Eventually, this can go per-line (each line of a document signed by its author), but per-commit signing is the practical starting point.

---

## Git Compatibility (Future)

If the internal object model matches Git's closely enough, we could potentially:

- **Export to Git:** Convert an encrypted vault into a standard Git repo (decrypting in the process) for use with standard Git tools
- **Import from Git:** Take a Git repo and import it into an encrypted vault
- **Git as source of truth:** Use a Git repo as the backing store, with the vault providing encryption, access control, and web UI on top

This is not a priority for now, but it's worth keeping the object model compatible where possible so the option exists.

---

## Browser-Side Full Clone (Future)

The CLI already downloads the full vault locally. The same could be done in the browser:

- Download and cache the entire encrypted object store using IndexedDB or the Cache API
- Full offline access in the browser
- Full history browsing
- Diffs, branches, merges — all client-side
- Sync when online (push/pull like the CLI)

This eliminates the "always online" requirement for the web workspace and gives browser users the same capabilities as CLI users.

## Implementation Sequence

### Phase 1: Object Store Foundation
- Content-addressable encrypted object store (blob, tree, commit objects)
- Automatic commits on every save (no staging)
- Trees within trees (nested directory support)
- Clone downloads full object store
- CLI updated to use object store internally

### Phase 2: History and Diffs
- Commit log visible in UI and CLI
- Diff between any two commits
- Rollback to any previous commit
- Basic conflict detection (two commits with same parent)

### Phase 3: Branches and Merge
- Named branches (pointers to commits)
- Three-way merge algorithm
- Conflict resolution UI
- Pull request workflow (propose merge, review, accept)

### Phase 4: Advanced
- Per-object multi-key encryption (different keys for different files)
- Signed commits (public key signature)
- Browser-side full clone with offline access
- Git import/export compatibility
- Packfile optimisation for large vaults
