# Dev Brief: Scalable Document Classification via Hash Caching — Workspace Plugins

**version** v0.11.1  
**date** 4 Mar 2026  
**from** Human (project lead)  
**to** Architect (lead), Developer  
**type** Dev brief — workspace plugin architecture, LLM cost optimisation

## The Problem

LLM-based document classification and translation is powerful but slow and expensive. Even a fast LLM call takes 5-30 seconds and consumes tokens. If you're processing the same document repeatedly (because it changed slightly, or because you want a different classification), you're re-processing content that hasn't changed. This doesn't scale.

**The insight:** Most document edits change a small fraction of the total content. If we hash every content unit, cache the LLM results per hash, and only send uncached units to the LLM, we can reduce LLM calls by an order of magnitude or more.

---

## The Architecture

### Step 1: Decompose Document into Hashable Units

Take any document (HTML page, markdown file, website) and decompose it into discrete content units. Each unit gets a hash and metadata:

```json
{
  "units": [
    {
      "id": "unit_001",
      "hash": "sha256_of_content",
      "text": "The actual text content of this paragraph",
      "type": "paragraph",
      "size": 142,
      "created": "2026-03-04T10:00:00Z"
    }
  ]
}
```

The hash is the key primitive. Same content = same hash = cached result available.

**This runs entirely client-side.** Pure JavaScript, no server involvement. This is a workspace plugin.

### Step 2: Check the Cache

Download the results file for this document from the vault. This file contains all previously computed results keyed by content hash — translations, classifications, metadata, which model produced them, and when.

### Step 3: Identify the Delta

Compare current hashes against the cache. Three categories: hash exists in cache (use it, zero cost, instant), hash is new (send to LLM), hash in cache but not in document (content deleted, keep or prune).

### Step 4: Only Process the Delta

Send only new/changed content units to the LLM. For a document where 95% of content hasn't changed, this means 95% fewer LLM calls.

### Step 5: Save Updated Cache

Merge new results into the cache file. Save back to vault (encrypted). Next time, even more content is cached.

## The Pipeline

```
Document (HTML/Markdown/Page)
    |
    v
[Extract] -> Content units with hashes
    |
    v
[Cache Check] -> Download results file from vault
    |
    +-- Cached units -> Use immediately (zero cost)
    |
    +-- Uncached units -> Send to LLM
                             |
                             v
                         [LLM processes only the delta]
                             |
                             v
                         [Merge into cache, save to vault]
    |
    v
[Assemble] -> Full document with all results applied
```

## Translation and Classification: Same Architecture, Different Prompts

The only difference is the prompt. Translation: "Translate to Brazilian Portuguese." Classification: "Classify by topic and sentiment." The cache stores results per operation type. A single content unit can have cached translations in 5 languages AND a classification, all keyed by the same hash. Adding a new operation is just a new prompt.

**Hypothesis:** Even storing all translations/classifications in one large file per site, downloading and decrypting that file is still orders of magnitude faster than any LLM call. A file download from S3 + decrypt is sub-second. Any LLM call is minimum 5 seconds. The cache check is effectively free.

## Workspace Plugins

This engine should be the first **workspace plugin** — a modular, self-contained processing component.

### Plugin Architecture

Each plugin is a folder containing a manifest (name, description, capabilities), processing logic (JavaScript), default prompts/config, and a cache storage location. The workspace loads available plugins and shows them in a selector. Plugins run client-side, read/write to the vault, and can call external APIs (LLM via Open Router).

### First Plugins

| Plugin | What It Does |
|---|---|
| **HTML Transformation** | Current workbench (HTML + prompt to LLM, get transformed HTML) |
| **Content Classification** | Decompose, classify via LLM with hash caching |
| **Translation** | Same as classification with translation prompts |
| **Content Mapping** | Extract hash-based meta-structure (connects to the privacy-preserving content assembly brief) |

---

## Parallel Requests and Model Optimisation

Send uncached units in parallel (Open Router supports concurrent requests). For high-assurance: send same unit to 3-5 models, compare results, flag disagreements. Use caching data to identify which cheaper models produce equivalent results for specific content types — route future requests to cheaper models.
