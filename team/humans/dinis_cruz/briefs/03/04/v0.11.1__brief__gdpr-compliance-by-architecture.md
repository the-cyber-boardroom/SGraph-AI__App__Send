## The Fundamental Problem

Companies processing sensitive health data face a structural contradiction: GDPR requires them to protect personal data, but their entire technology stack is built on the assumption that servers, databases, and administrators can see that data. Every system in the chain — web servers, databases, email, cloud storage, backup systems, analytics — has access to plaintext personal data.

This means GDPR compliance is enforced by policy, not architecture. Companies write privacy policies, sign data processing agreements, conduct impact assessments, train staff, and hope nobody makes a mistake. When they do — and they always do — the result is a data breach, because the data was always accessible to begin with.

**SG/Send inverts this.** The server never sees plaintext. The data is encrypted in the user's browser before it leaves their device. The server stores encrypted blobs it cannot read. GDPR compliance is not a policy promise — it is a mathematical property of the system.

## GDPR Article-by-Article: How SG/Send's Architecture Responds

### Article 5: Principles of Data Processing

**Article 5(1)(f) — Integrity and Confidentiality:** Personal data must be processed with appropriate security, including protection against unauthorised access and accidental loss.

**SG/Send response:** Data is encrypted with AES-256-GCM before it leaves the user's browser. The encryption key never touches the server. Even if the server is fully compromised — database dump, admin access, insider threat — the attacker gets encrypted blobs they cannot decrypt. Confidentiality is not a configuration setting. It is a consequence of the cryptographic design.

**Article 5(1)(c) — Data Minimisation:** Only data that is necessary should be processed.

**SG/Send response:** The server processes zero personal data. It stores encrypted blobs and serves them by ID. It does not know what is inside them. It cannot know. The most minimal processing possible is processing nothing — which is what the server does.

**Article 5(1)(e) — Storage Limitation:** Data should not be kept longer than necessary.

**SG/Send response:** Time-bound encryption keys (on the roadmap) allow data to become cryptographically inaccessible after a defined period — without requiring the server to delete anything. The encrypted blob can remain in storage, but once the key expires, the data is effectively destroyed. This is stronger than deletion: even if backups exist, the data is unreadable.

### Article 6: Lawful Basis for Processing

For health data processors, lawful basis is typically consent (Article 6(1)(a)) or legitimate interest. The challenge is proving consent was given and managing withdrawal.

**SG/Send response:** Consent records can be stored in the vault alongside the data they relate to. The consent record is signed with the user's key (provenance) and timestamped. If the user withdraws consent, their access key is revoked — the data becomes cryptographically inaccessible. The signed consent record remains (proving proper procedure was followed) even after the data itself is no longer accessible.

### Article 9: Special Categories (Health Data)

Health data requires explicit consent and additional safeguards. Processing health data triggers the highest level of GDPR scrutiny.

**SG/Send response:** The additional safeguard is that the processor literally cannot access the data without the user's key. There is no risk of "accidental processing" of health data because processing requires active cryptographic participation from the data subject. The health data exists in the system, but it exists as an opaque encrypted blob until the user explicitly grants access by sharing a decryption key.

### Article 17: Right to Erasure ("Right to Be Forgotten")

The data subject can request deletion of their personal data.

**SG/Send response:** Two mechanisms, both stronger than traditional deletion:

1. **Key revocation:** The data subject revokes the decryption key. The encrypted blob remains in storage but is permanently unreadable. This is cryptographic erasure — stronger than file deletion (which can be recovered from backups, undelete tools, or forensic analysis).

2. **Blob deletion:** The encrypted blob is deleted from storage. Since no other system ever had the plaintext, there are no copies to hunt down across email servers, backup tapes, log files, analytics databases, or employee laptops. The data only ever existed in one place (the encrypted blob), so deleting it actually deletes it.

Compare this with traditional systems where "delete the data" means: delete from the production database, delete from all replicas, delete from all backups, delete from email threads where it was shared, delete from the analytics pipeline, delete from the data warehouse, delete from any exports or reports, contact any third parties who received it. Deletion in traditional systems is a process that takes weeks and is never fully complete.

### Article 20: Right to Data Portability

The data subject has the right to receive their data in a structured, commonly used format.

**SG/Send response:** The vault IS the portable data package. The user's vault contains all their data, encrypted with their key. They can clone it (via CLI), download it, move it to another storage provider, or share it with a new service provider. The data is already structured (files in folders), already portable (standard encryption, open-source decryption tools), and already under the user's control.

No data extraction request needed. No waiting for the company to compile a SAR response. The user already has their data.

### Article 25: Data Protection by Design and Default

Companies must implement technical measures to protect data from the moment of system design, not as an afterthought.

**SG/Send response:** This is the article SG/Send was built for. Zero-knowledge encryption is not a feature added to a system — it is the system. There is no "unencrypted mode." There is no admin panel where someone can view patient records. There is no database query that returns plaintext health data. The protection is not a policy or a configuration — it is the architecture itself.

**By default:** All data is encrypted. There is no opt-in. There is no "premium encryption tier." Every file, every vault, every transfer is encrypted with keys the server never sees.

### Article 28: Data Processor Obligations

When a company processes data on behalf of another (e.g., a health tech company processing patient data for a hospital), the processor must provide sufficient guarantees of security.

**SG/Send response:** The strongest possible guarantee: the processor cannot access the data. A data processing agreement backed by SG/Send's architecture doesn't say "we promise not to look at your data." It says "we are architecturally incapable of looking at your data." This is a qualitatively different guarantee — it does not depend on the trustworthiness of the processor's employees, the effectiveness of their access controls, or the competence of their security team.

### Article 32: Security of Processing

Companies must implement appropriate technical measures including encryption, the ability to ensure ongoing confidentiality, the ability to restore availability, and regular testing.

**SG/Send response:**

| Requirement | SG/Send Implementation |
|---|---|
| Encryption of personal data | AES-256-GCM, client-side, keys never on server |
| Ongoing confidentiality | Zero-knowledge architecture — confidentiality is structural, not operational |
| Ability to restore availability | Vault backup (multiple locations via custody service), Git-like object store with full history |
| Regular testing | Integrity verification (custody service checks vaults are intact and decryptable) |

### Article 33/34: Breach Notification

Companies must notify the supervisory authority within 72 hours of a breach, and notify affected individuals if the breach poses a high risk.

**SG/Send response:** A server compromise is not a personal data breach. If an attacker gains full access to the SG/Send server — database, file storage, admin access, everything — they get encrypted blobs. No personal data is exposed. No notification is required under Article 33(1) because the data was rendered unintelligible through encryption, and the keys were not compromised (they were never on the server).

This is explicitly addressed in GDPR Recital 83: encryption is cited as a measure that can render a breach non-notifiable if the data is unintelligible to anyone not authorised to access it.

**This is perhaps the single most valuable property for healthcare companies.** Data breaches in healthcare are catastrophic — reputational damage, regulatory fines, patient harm. With SG/Send, a server breach is a security incident, not a data breach. The distinction saves organisations from the most damaging consequence of a compromise.

### Article 44-49: International Data Transfers

Transferring personal data outside the EEA requires adequate safeguards.

**SG/Send response:** Encrypted data is not personal data in transit. If the encryption keys remain within the EEA (e.g., stored in a European custody service), the encrypted blobs can be stored or transferred anywhere without triggering transfer restrictions. The ciphertext in a US data centre is not a "transfer of personal data to the US" because the US data centre cannot access the personal data — it only has encrypted blobs.

This is a significant practical advantage: companies can use global cloud infrastructure for storage and CDN without triggering Schrems II concerns, as long as the keys remain in the appropriate jurisdiction.

## Healthcare-Specific Workflows

### Patient Assessment Form → Encrypted Vault

A health tech company provides online health assessments. Today: patient fills the form, the server stores the results, the company becomes a data processor holding sensitive health data with all the GDPR obligations that entails.

With SG/Send: patient fills the form, the browser encrypts the results and stores them in a vault. The company never touches plaintext health data. The company is not a data processor for that data — it is an encrypted storage provider. The patient shares a read key with their clinician. Only the clinician can see the results.

**GDPR implication:** The company has dramatically reduced its regulatory burden. It does not process health data. It does not need to conduct a DPIA for the assessment results. It does not need to appoint a DPO specifically for that processing activity. It does not need to maintain records of processing for that data. The patient is the data controller. The company is just an encrypted pipe.

### Clinical Trial Data Collection

A pharmaceutical company runs a multi-site clinical trial. Today: all participant data flows to a central database. The pharmaceutical company is a data controller for thousands of patients' health records. Massive GDPR exposure.

With SG/Send's multi-vault architecture: each participant has their own vault. The pharmaceutical company cannot access raw data. They receive only anonymised exports that each participant has reviewed and approved. The signed consent records provide an auditable trail. Participant withdrawal is key revocation, not a data deletion project.

**GDPR implication:** The pharmaceutical company's exposure is limited to anonymised, aggregated data. The raw health data never enters their systems. The consent trail is cryptographically verifiable. Participant withdrawal is instant and complete.

### Sharing Medical Records Between Providers

A patient moves from one GP to another. Today: records transferred via NHS systems (centralised, with all the privacy and security concerns of centralised health records) or, worse, printed and hand-carried.

With SG/Send: the patient's vault contains their medical records. The patient shares a read key with the new GP. The transfer is instant, encrypted, and under the patient's control. The old GP's access can be revoked. No centralised system needed. No intermediary sees the data.

## The Competitive Positioning

Traditional health tech companies trying to be GDPR compliant must:

1. Encrypt data at rest (server-side encryption — the server still has the keys)
2. Encrypt data in transit (TLS — protects the pipe, not the data)
3. Implement access controls (admin can override)
4. Maintain audit logs (can be tampered with)
5. Conduct DPIAs for every processing activity
6. Appoint a DPO
7. Sign data processing agreements with every sub-processor
8. Maintain records of processing activities
9. Implement data retention policies
10. Handle SARs within 30 days
11. Handle erasure requests across all systems
12. Notify breaches within 72 hours
13. Train all staff on data protection
14. Hope nobody makes a mistake

A company built on SG/Send:

1. Never has the data in plaintext — most of the above becomes irrelevant or dramatically simplified

This is not an incremental improvement. It is a category change. The company moves from "we process health data and must comply with extensive regulations" to "we provide encrypted storage infrastructure and the health data never enters our systems."

## What SG/Send Does NOT Solve

Honesty matters. SG/Send's architecture does not address every GDPR concern:

**Metadata:** The server knows that a vault exists, when it was accessed, how large the files are, and the IP addresses that accessed it. Metadata is not encrypted. Depending on the context, metadata can be personal data.

**User authentication for key recovery:** If a user loses their key and uses a custody service to recover it, that recovery process involves identity verification — which is itself a processing activity.

**Processing that requires plaintext:** Some health data processing genuinely requires the processor to see the data (e.g., a diagnostic AI that analyses medical images). SG/Send's zero-knowledge model does not apply here. However, the document transformation workspace and the pod architecture offer a path: processing happens in a controlled, auditable environment with minimal exposure.

**Consent for the encryption itself:** The user must understand and consent to their data being encrypted and that key loss means data loss. This is a UX and communication challenge, not a technical one.
