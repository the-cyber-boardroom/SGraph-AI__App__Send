## Opportunity 1: DPO Services on Demand

### What a DPO Needs

DPOs handle sensitive data by definition: privacy impact assessments, data breach notifications, subject access request (SAR) responses, compliance evidence, internal audit reports. All of this is confidential. Currently, DPO advisory firms share this material via email, cloud drives, or specialised (expensive) compliance platforms.

### What SG/Send Already Provides

**The data room is the product.** A DPO advising a company can:

1. Create a data room (folder upload) for each client engagement
2. Organise materials: privacy policies, DPIA templates, breach response plans, SAR records
3. Share the encrypted link with the client — one link, all materials, previewable in browser
4. Update the data room as the engagement progresses (new versions, new documents)
5. All materials encrypted end-to-end — the advisory firm's server never sees client data

This works **today** with zero additional features. The data room that shipped accidentally for the investor review use case is exactly what a DPO needs.

### What the Vault Adds

Once the vault is stable for client use:

- **Persistent data room:** Not a one-shot folder upload, but a living workspace that grows over time
- **Access control:** Read key for the client (view-only), write key for the DPO (edit)
- **Audit trail:** Download logs show when the client accessed materials (from the analytics work)
- **CLI access:** The DPO can manage client vaults from the command line, push documents without opening a browser
- **Workspace:** The DPO can use the LLM transformation studio to process documents (classify, translate, summarise) within the encrypted environment

### Specific DPO Use Cases

| Use Case | How SG/Send Handles It |
|---|---|
| **Privacy Impact Assessment (DPIA)** | DPO creates assessment in vault, shares read-only link with client for review. Comments and revisions stay in the data room. |
| **Subject Access Request (SAR) response** | SAR responses contain personal data. Package the response files in a data room, share with the data subject via encrypted link. Provable delivery, encrypted in transit and at rest. |
| **Data breach notification** | Breach evidence and notification drafts shared via encrypted data room. Only authorised parties can access. Audit trail proves who saw what and when. |
| **Compliance evidence** | Ongoing compliance documentation stored in a vault. Auditors get a read-only link. Evidence is verifiable and tamper-evident (hash binding). |
| **Multi-client management** | Each client gets their own vault. The DPO manages all vaults via CLI. Client data is isolated — one client's vault cannot access another's. |
| **Cross-border data sharing** | GDPR-compliant sharing of personal data across jurisdictions. Zero-knowledge server means the infrastructure provider (SG/Send) is not a data processor of the content — it never sees it. |

### The GDPR Angle

SG/Send's zero-knowledge architecture has a specific GDPR advantage: **the platform operator is not a data processor of the file contents.** The server stores only encrypted blobs it cannot decrypt. This simplifies the data processing agreement and reduces the compliance burden for both the DPO and their clients.

## Opportunity 2: Whistleblower Platform

### The Insight

SG/Send's PKI infrastructure enables a simple, powerful whistleblower reporting system. The key properties:

1. **The company publishes a public key** — this is a public endpoint, available to anyone
2. **The whistleblower encrypts their report in the browser** using the company's public key
3. **The encrypted report is sent through SG/Send** — the server stores only the encrypted blob, cannot read it
4. **Only the company (holding the private key) can decrypt the report**
5. **The whistleblower is anonymous** — no tracking, no account required, no identity linked to the submission

### The Privacy Properties

| Property | How It's Achieved |
|---|---|
| **Whistleblower anonymity** | No account, no login, no tracking. The whistleblower accesses a public URL and submits encrypted data. |
| **Report confidentiality** | Encrypted with the company's public key — only the private key holder can read it. SG/Send server never sees the content. |
| **Traceability (optional)** | The whistleblower can optionally include a public key of their own. This creates a return channel: the company can encrypt a response that only the whistleblower can read. The whistleblower remains anonymous until they choose to reveal themselves by proving they hold the matching private key. |
| **Tamper evidence** | The encrypted report has a hash. If anyone modifies the encrypted blob, the hash changes and decryption fails. The company can verify the report is exactly what was submitted. |
| **Audit trail** | Submission timestamps and download logs prove when the report was submitted and when it was accessed. |

### The "Delayed Identity Reveal" Pattern

This is particularly powerful for whistleblowing:

1. Whistleblower submits anonymously, but includes their public key
2. Company receives the report, investigates
3. Company encrypts a response with the whistleblower's public key: "We've investigated, please provide more details on X"
4. Whistleblower downloads the response (only they can decrypt it), provides more information
5. This back-and-forth continues anonymously — neither party knows the other's identity
6. If the whistleblower wants legal protection or a reward, they reveal their identity by proving they hold the private key that matches the public key in the original submission

The identity reveal is the whistleblower's choice, not the platform's. And the proof is cryptographic — not "I claim I submitted this" but "I can prove I hold the key that was in the original submission."
