## Why This Extension Exists

Every friction point in the current SG/Send product traces back to key management. Users lose passphrases. Keys don't follow users across devices. Vaults need manual key entry every session. The workspace forgets its state when the browser closes. The data room recipient has to manually enter a decryption key.

The Chrome extension solves this by turning key management into a background service: keys are encrypted with a user-chosen passphrase, stored in Chrome Sync (encrypted — useless without the passphrase), and automatically provided to sgraph.ai pages when needed. The SG/Send server never sees the keys. The zero-knowledge model is preserved.

This is the missing piece that makes vaults, workspaces, data rooms, and the full PKI story usable for real daily work.

## Part 1: Core Extension Architecture

### Data Model

One encrypted bundle per user, stored in `chrome.storage.sync`:

```
KeyBundlePlaintext:
  version, createdAt, updatedAt
  identityKeys:
    ed25519 private + public key pair
  rooms:
    [roomId]: roomKey, name, lastUsedAt

KeyBundleCiphertext:
  version, salt, kdfParams (algo, iterations, mem)
  nonce, ciphertext (AEAD with auth tag)
```

Stored under `sgraph_key_bundle_v1` in Chrome Sync. The passphrase and derived masterKey live only in memory — never stored, never transmitted.

### Crypto Primitives

- **Derivation:** passphrase to masterKey via PBKDF2 or Argon2id (WASM) with per-bundle random salt
- **Encryption:** AEAD (AES-256-GCM or XChaCha20-Poly1305) on the JSON-serialised plaintext bundle
- **Identity:** Ed25519 key pair for signing and PKI operations

### Minimum Permissions

```json
{
  "manifest_version": 3,
  "permissions": ["storage"],
  "host_permissions": ["https://send.sgraph.ai/*", "https://*.sgraph.ai/*"],
  "externally_connectable": {
    "matches": ["https://send.sgraph.ai/*", "https://*.sgraph.ai/*"]
  }
}
```

Two permissions only: `storage` for Chrome Sync, and host permissions for sgraph.ai. Add `management` only if implementing the other-extension detection feature (see Part 2). No scary permissions — the install prompt says "Read and change your data on send.sgraph.ai."

### Core Flows

**First-time setup:** User opens options page, chooses a passphrase. Extension generates identity key pair + empty rooms map. Derives masterKey from passphrase + random salt. Encrypts the bundle. Stores in Chrome Sync. Keys in memory only.

**Unlock on new device:** Extension loads encrypted bundle from Chrome Sync. Prompts for passphrase. Derives masterKey, decrypts bundle. Cache decrypted bundle in service worker memory. Optionally store a short-lived unlock token in `chrome.storage.session` to avoid re-prompting within a browser session.

**Page interaction:** The sgraph.ai page communicates with the extension via `chrome.runtime.sendMessage` (using `externally_connectable`) or via content script message relay. The page requests a specific key (e.g., "give me the room key for vault X"). The service worker checks if the bundle is unlocked, returns only the requested key. The page never sees the passphrase, the masterKey, or the full bundle.

**Passphrase change:** Require current passphrase to decrypt. Re-encrypt same plaintext with new passphrase + new salt. Overwrite in Chrome Sync.

**Key rotation:** Generate new identity keys. Update plaintext bundle. Re-encrypt with current masterKey. Save. Webapp handles "active key vs legacy keys."

### Storage Quota

Chrome Sync quota is approximately 100KB. One bundle with identity keys + a few hundred room keys fits easily. If heavy users exceed this: split rooms into multiple sync items with an index, or fall back to `chrome.storage.local` with manual export/import for cross-device.

## Part 2: Security Posture Detection

### What the Extension Can Detect

**Network-level:**
- HTTP vs HTTPS (refuse to release keys over HTTP)
- TLS certificate properties (unexpected CA = possible corporate MITM proxy)
- Response integrity (fetch a known endpoint, compare response hash to expected)
- Unusual latency (traffic routed through inspection proxy)
- Canary header checks (SG/Send includes a specific header, extension verifies it wasn't stripped or modified)

**Browser environment:**
- DevTools open (signals possible inspection)
- Other extensions with broad permissions that could access sgraph.ai pages (via `chrome.management` API)
- DOM tampering: unexpected script tags, iframes, event listeners detected via MutationObserver
- Script integrity: compare loaded sgraph.ai JavaScript hashes against known-good values
- Monkey-patched APIs: detect if another extension's content script has overwritten `fetch`, `XMLHttpRequest`, `crypto.subtle`, or `postMessage` (compare against originals from the isolated world)
- Incognito mode (keys won't persist after close)

**Device signals:**
- Device fingerprint (hardwareConcurrency, deviceMemory, platform) to detect known vs unknown devices
- Registry of previously seen devices maintained in `chrome.storage.local`
- Network type (wifi vs cellular vs ethernet via `navigator.connection`)

**What the Extension CANNOT Detect:**
- OS-level malware, keyloggers, screen recorders (Chrome sandbox prevents visibility)
- VPN or OS-level proxy configuration
- Disk encryption status
- Other browser profiles or other browsers

### Traffic Light Model

**Green (extension icon badge: green):**
- HTTPS with valid, expected certificate
- No unexpected extensions with sgraph.ai access
- Known device
- Script integrity check passes
- No DOM tampering

**Amber (extension icon badge: amber):**
- Unknown device (first time)
- DevTools open
- Extensions with broad permissions detected
- Incognito mode
- Public/shared network indicators

**Red (extension icon badge: red, keys locked, user must explicitly override):**
- HTTP connection
- Script integrity check fails
- DOM injection detected
- Known-compromised extension detected
- Certificate mismatch or unexpected redirect chain
- Monkey-patched browser APIs detected

On red: the extension refuses to release keys until the user acknowledges the risk. The extension becomes a gatekeeper protecting vault keys from compromised environments.

### Device Attestation

Every vault access can include a signed posture report: the extension signs a statement with the identity key containing the current security posture (green/amber/red), device fingerprint, timestamp, and extension version. This creates an audit trail: "vault accessed from known device, green status" vs "vault accessed from unknown device, amber status, DevTools open." Valuable for healthcare compliance and DPO requirements.

## Part 3: Extension vs Other Extensions

### Detecting Other Extensions

With the `management` permission, the extension can enumerate all installed extensions and their permissions. For each extension, check: does it have host permissions that cover sgraph.ai? Does it have `webRequest` or `webRequestBlocking`? Does it have `<all_urls>`? Flag any that could theoretically intercept sgraph.ai traffic.

The extension cannot see other extensions' code, storage, or runtime behaviour. But it can see their declared permissions, which is sufficient for risk assessment.

### Detecting DOM Tampering by Other Extensions

Other extensions' content scripts share the DOM. Our content script can:

1. On load: snapshot critical DOM elements (forms, script tags, iframes)
2. Start a MutationObserver on the document
3. Flag unexpected mutations: new script tags, new iframes, modified form actions, new event listeners on sensitive elements
4. Compare loaded script sources against a whitelist
5. Check for monkey-patched globals (fetch, XMLHttpRequest, crypto.subtle) by comparing against originals in the isolated world

If tampering is detected: escalate to red status, lock keys, alert the user.

## Part 4: Independent API Channel

### Extension as Trusted API Client

The service worker can make `fetch()` calls directly to send.sgraph.ai APIs. These calls:

- Come from the extension context, not any page
- Are not affected by page CSP
- Are not visible to content scripts, page scripts, or other extensions
- Can include a custom authentication header (e.g., `X-SGGraph-Extension: signed_token`) that page JavaScript cannot forge

This creates a **trusted channel** between the extension and the server, independent of and more secure than the page's own API calls.

**Use cases:**
- Vault operations (read metadata, push keys) directly from the extension
- Health check against the server (verify the server is genuine)
- Deposit and retrieve encrypted blobs for cross-device communication (see Part 5)
- Submit device attestation reports

**Server-side:** The API can have extension-only endpoints that verify the signed request header. Requests without valid extension signatures are rejected. This gives the server confidence that the request comes from a genuine, unlocked extension — not from a compromised page.

## Part 5: Page Detection of the Extension

### Three Mechanisms

**1. DOM flag:** Content script sets `document.documentElement.dataset.sgraphExtension = 'active'`. Page checks for this attribute. Simple, fast, but spoofable by other extensions.

**2. Externally connectable handshake:** Page calls `chrome.runtime.sendMessage(EXTENSION_ID, { type: 'healthCheck' })`. Extension responds with version, posture status, device fingerprint. This is a private channel — other extensions cannot intercept it. The page must know the extension ID (public, fixed after Chrome Web Store publication).

**3. Challenge-response:** Page sends a nonce. Extension signs it with the identity key. Page verifies the signature against the known public key. This proves the extension is genuine (not spoofed) and has an unlocked key bundle.

**UI adaptation:** When the page detects the extension, it can:
- Hide manual key entry fields (extension provides keys automatically)
- Show a security badge (green/amber/red from the extension's posture report)
- Enable advanced features that require persistent keys (PKI signing, vault sync, device attestation)
- Show "Install our extension for a better experience" if the extension is not detected

## Part 6: Cross-Device 2FA and Communication

### The Breakthrough Idea

Two browsers, two computers, two extensions — communicating via encrypted messages relayed through SG/Send's API. The server is a relay for encrypted blobs it cannot read. This enables zero-knowledge 2FA and general-purpose cross-device communication.

### 2FA Flow

**Device A (new/untrusted) requests vault access:**
1. Extension on Device A creates a challenge: a short code (e.g., "742") + a unique token
2. Encrypts the challenge with the user's own public key
3. Pushes the encrypted blob to SG/Send API via the extension's independent channel
4. Shows the user: "Waiting for approval from your other device..."

**Device B (existing/trusted) picks up the challenge:**
1. Extension on Device B polls (or listens via WebSocket/SSE) for pending authorisation requests
2. Finds the encrypted challenge blob, decrypts it
3. Shows the user: "A new device is requesting access. Code: 742. Approve or deny?"

**User approves on Device B:**
1. Extension on Device B signs the approval with the identity key
2. Pushes the signed approval back to SG/Send API

**Device A completes:**
1. Extension on Device A picks up the signed approval
2. Verifies the signature
3. Unlocks the vault

**Zero-knowledge property:** The SG/Send server relays encrypted blobs. It sees "blob deposited by session X, picked up by session Y, response from Y, picked up by X." It cannot read any of the blobs. The 2FA handshake is end-to-end encrypted between two instances of the same user's extension.

### General Cross-Device Communication Channel

The 2FA flow is a specific instance of a general pattern: two extensions communicating via encrypted blobs relayed through the SG/Send API. Other use cases:

| Use Case | How It Works |
|---|---|
| **Key sharing** | Send a new vault/room key from one device to another, encrypted for the recipient's extension |
| **Vault invitation** | "I'm adding you to this vault" — send the encrypted vault key to another user's extension |
| **Approval workflows** | "Someone wants to edit this document — approve?" with cryptographic approval |
| **Session migration** | "Continue on my other device" — transfer session state securely |
| **Encrypted clipboard** | Copy on laptop, paste on desktop — encrypted in transit via SG/Send relay |

This turns SG/Send + the extension into a **secure device-to-device communication layer** that works across any devices with the extension installed.

### Notification Mechanism

For the 2FA and cross-device flows to feel instant, Device B needs to know quickly that a request is waiting. Options:

| Mechanism | Latency | Complexity |
|---|---|---|
| **Polling** (extension checks every N seconds) | Seconds to minutes | Low — just periodic fetch calls |
| **Server-Sent Events (SSE)** | Sub-second | Medium — extension opens persistent connection |
| **WebSocket** | Sub-second | Medium — bidirectional, more complex |
| **Push notification** (via Chrome's push API) | Seconds | Higher — requires push subscription setup |

For MVP: polling every 5-10 seconds is sufficient. The approval flow isn't time-critical (the user on Device A is waiting and expecting a short delay). SSE or WebSocket is the upgrade path for real-time feel.

## Part 7: Multi-Browser Support

### Chromium-Based Browsers (Works Out of the Box)

| Browser | Market Share | Sync Mechanism | Extension Support |
|---|---|---|---|
| **Google Chrome** | ~65% | Google account | Native |
| **Microsoft Edge** | ~13% | Microsoft account | Chrome Web Store compatible |
| **Brave** | ~2% | Brave Sync chain | Chrome extensions supported |
| **Opera** | ~2% | Opera account | Chrome extensions supported |
| **Vivaldi** | ~1% | Vivaldi sync | Chrome extensions supported |
| **Arc** | Growing | Chrome profile sync | Chrome extensions supported |

Total Chromium coverage: approximately 85%+ of desktop browsers.

### The Sync Caveat

`chrome.storage.sync` uses the browser's native sync: Google account for Chrome, Microsoft account for Edge, Brave Sync for Brave. The encrypted bundle format is identical, but the sync mechanism is browser-specific. Cross-browser sync (Chrome on one device, Edge on another) would require a fallback — manual export/import, or syncing the encrypted bundle via a vault (which is elegantly recursive).

### Non-Chromium Browsers

**Firefox:** WebExtensions API is close enough for a port. The core APIs (storage, content scripts, background scripts) have Firefox equivalents. `browser.storage.sync` works with Firefox Sync. Moderate effort.

**Safari:** Different extension model (Safari Web Extensions). Higher effort. Lower priority given market share on desktop.

## Part 8: Extension UI at extension.sgraph.ai

### Two UI Surfaces: Popup for Daily Use, Web App for Management

**Extension popup (built-in):** Click the extension icon, get a small panel (~400x600px). Handles quick interactions: unlock/lock, see current security posture (traffic light), approve a 2FA request, see which vault is active. This is bundled with the extension — local HTML, no network request, fully isolated. Same pattern as 1Password: the popup is for daily use.

**Full management UI at extension.sgraph.ai:** A web page on the sgraph.ai domain, purpose-built for key management. The page communicates with the extension via `externally_connectable` — the service worker does all crypto, the page is just the UI. This handles everything the popup is too small for:

- Initial setup wizard (create passphrase, generate identity keys)
- Key bundle management (view all rooms/vaults, add/remove keys, rename)
- Device registry (list known devices, revoke a device, see last access time)
- Security posture dashboard (current status, history of access events per device)
- 2FA approval queue (pending requests from other devices)
- Extension settings (auto-lock timeout, notification preferences)
- Audit log (when were keys accessed, from which device, what posture)
- Export/import (encrypted bundle export for CLI use or cross-browser transfer)

### Architecture

extension.sgraph.ai is another static site in the existing S3 + CloudFront setup — same deployment pattern as send.sgraph.ai and vault.sgraph.ai. The CI/CD path scoping applies: changes to `extension/` only rebuild extension.sgraph.ai.

The page contains zero crypto logic. All sensitive operations happen in the extension's service worker. The page sends requests (`createBundle`, `unlockBundle`, `listRooms`, `exportKey`, `approveDevice`) and receives responses. If the extension is not installed, the page shows an install prompt with a link to the Chrome Web Store listing.

### Navigation Between Surfaces

| Action | Where It Happens |
|---|---|
| Quick unlock / lock | Extension popup |
| See security status at a glance | Extension popup (traffic light icon) |
| Approve a 2FA request | Extension popup (notification badge + approve button) |
| Manage keys and rooms | extension.sgraph.ai |
| View device registry | extension.sgraph.ai |
| Review audit log | extension.sgraph.ai |
| Initial setup | extension.sgraph.ai (linked from popup on first install) |
| Export keys for CLI | extension.sgraph.ai |

## Part 9: Chrome Web Store Distribution

### Publishing Process

**One-time setup (~1 hour):**
- Register a Chrome Web Store developer account ($5 one-time fee)
- Verify identity (Google requires this for new developers)

**Per-release (~30 minutes of active work, then waiting for review):**
- Package the extension as a .zip (manifest.json, service worker, options/popup HTML, content script, crypto library)
- Upload to the Chrome Web Store developer dashboard
- Fill in the listing: name ("SGraph Key Vault"), description, screenshots, category, privacy policy URL
- Declare permissions justification (Google requires an explanation for each permission — `storage` for key sync and host permissions for your own domains are straightforward justifications)
- Submit for review

**Review timeline:**
- First submission: 1-3 business days (Google reviews for policy compliance)
- Updates to approved extensions: typically within 24 hours, often same-day
- Google checks: permissions match stated purpose, no obfuscated code, privacy policy exists, no malware. This extension is clean on all counts.

**Auto-update:** Once published, Chrome auto-updates all installed instances when a new version is pushed and reviewed. Users do nothing. Push an update, Google reviews it, it rolls out to all users within hours.

### The Extension ID

Once published, the extension receives a permanent 32-character ID. This ID is critical — it is used in `externally_connectable` so sgraph.ai pages can communicate with the extension. During development (unpacked/local), the ID changes on every reload. Once published, it is fixed forever.

**Important:** Publish early — even as unlisted — to lock the extension ID. Then build the page integration (`externally_connectable`, `chrome.runtime.sendMessage(EXTENSION_ID, ...)`) against the real, permanent ID. Waiting to publish until the extension is "ready" means all the page-to-extension communication code has to be rewritten when the ID changes.

### Multi-Store Distribution

| Store | Browser | Effort | Notes |
|---|---|---|---|
| **Chrome Web Store** | Chrome, Brave, Opera, Vivaldi, Arc | Primary — one listing covers all Chromium browsers | Brave/Opera/Vivaldi users install directly from Chrome Web Store |
| **Microsoft Edge Add-ons** | Edge | Low — same .zip, different upload dashboard | Edge also allows Chrome Web Store installs, but a native listing builds trust |
| **Firefox Add-ons (AMO)** | Firefox | Medium — port needed (WebExtensions API is similar but not identical) | Phase 5 |
| **Safari Extensions** | Safari | Higher — different extension model | Low priority given desktop market share |

### Practical Sequence

1. Build Phase 1 MVP locally (Chrome "Load unpacked" developer mode)
2. Test across sgraph.ai pages (send, vault, workspace, extension)
3. Publish to Chrome Web Store as unlisted (locks the extension ID, allows testing with real sync)
4. Build page integration against the permanent extension ID
5. Deploy extension.sgraph.ai (management UI)
6. Switch Chrome Web Store listing to public
7. Add "Install our extension" prompt on sgraph.ai pages when extension is not detected
8. Submit to Microsoft Edge Add-ons store (same package, different dashboard)
9. Firefox port (Phase 5)

### Privacy Policy

Google requires a privacy policy URL. The policy is simple and honest:

- The extension stores an encrypted key bundle in Chrome Sync. The passphrase never leaves the device.
- The extension communicates only with sgraph.ai domains. No third-party analytics, no tracking, no telemetry.
- The extension cannot read your browsing history, bookmarks, or data on other websites.
- Source code is open (Apache 2.0) and auditable.

Host this at sgraph.ai/extension/privacy or similar.

## Part 11: DevOps — Testing, Deployment, and Publishing Pipeline

### Why This Needs a Dedicated Pipeline

The extension is code that runs on every user's browser, auto-updates without user action, and handles the most sensitive material in the system (encryption keys). A bad update that breaks the extension or introduces a vulnerability affects every user simultaneously. This is the highest-stakes deployment in the entire SG/Send ecosystem — more consequential than a website bug, because the extension holds keys.

The DevOps team and agents must build extensive, automated testing, deployment, and publishing pipelines that match this level of criticality.

### Testing Pipeline

**Unit tests (run on every commit):**
- Crypto operations: key generation, encryption, decryption, key derivation (test vectors from known standards)
- Bundle serialisation/deserialisation
- Message schema validation (content script to service worker messages)
- Storage operations (mock Chrome APIs)

**Integration tests (run on every PR):**
- Full flow: create bundle, lock, unlock, provide key to page
- Chrome Sync simulation: save on one profile, load on another
- Page-to-extension communication via `externally_connectable`
- Security posture detection (mock various browser states)
- 2FA flow end-to-end

**Browser tests (run on every release candidate):**
- Automated browser tests using Puppeteer or Playwright with the extension loaded
- Test against live sgraph.ai pages (send, vault, workspace, extension management)
- Test on multiple Chromium browsers: Chrome, Edge, Brave
- Test extension install, first-time setup, unlock, key provision, lock
- Test upgrade path: install old version, upgrade to new version, verify bundle integrity

**Security tests (run on every release candidate):**
- Content Security Policy validation
- Verify no keys leak to console, network, or storage in plaintext
- Verify service worker memory is cleared on lock
- Verify `externally_connectable` only responds to declared origins
- Attempt message injection from non-sgraph.ai pages (must fail)
- Attempt DOM tampering detection (inject test scripts, verify detection triggers)

**Corporate extension variant tests (run when building branded extensions):**
- Verify custom branding applied correctly (icons, names, descriptions)
- Verify custom host permissions work
- Verify custom security policies enforced
- Verify the branded extension does not interfere with the official extension (both can be installed simultaneously)


### Publishing Automation

The Chrome Web Store has a publishing API (`chrome-webstore-upload` npm package or direct REST API). The pipeline should automate:

- Uploading the .zip to the Chrome Web Store
- Setting the listing metadata (version number, changelog)
- Submitting for review
- Monitoring review status
- Notifying the team when the update goes live

For corporate branded extensions: the same pipeline runs per-customer, uploading to either the customer's Web Store account (if they want to own the listing) or to our account (if we manage it for them).

The goal: **zero manual steps from code merge to published extension.** The human approval gate is the only manual step, and it is a review, not a build/upload action.

### Rollback Strategy

If a bad update ships:

1. **Immediate:** Publish the previous known-good version as a new version number (Chrome Web Store does not support rollback — you publish forward, but with the old code)
2. **Detection:** Automated smoke tests post-deploy catch regressions within minutes
3. **User impact window:** Chrome checks for extension updates every few hours. Worst case: users are on the bad version for a few hours before the fix auto-updates.
4. **Mitigation:** The extension should have a "safe mode" — if it detects its own state is corrupted (bundle won't decrypt, service worker crashes), it locks all keys and shows a "please update" message rather than failing silently.

### Version Management for Corporate Extensions

All corporate branded extensions share the same codebase version. When we release v1.2.3 of the official extension, all corporate extensions are rebuilt from v1.2.3 and published. Corporate extensions never fall behind the official extension — they are always on the same version with only configuration/branding differences.

Exception: if a corporate customer has requested a custom feature not yet in the official extension, it is feature-flagged. The flag is on in their config, off in the official config. When the feature graduates to the official extension, the flag is removed.

## Implementation Sequence

### Phase 1: Key Vault MVP
- Encrypted bundle create/unlock/lock
- Chrome Sync storage
- Page detection (`externally_connectable` handshake)
- Automatic key provision to sgraph.ai pages
- Extension popup UI (unlock, lock, status)
- Publish to Chrome Web Store as unlisted (lock the extension ID)
- extension.sgraph.ai deployed (setup wizard, key management, device registry)

### Phase 2: Security Posture
- HTTPS / certificate checks
- Script integrity verification
- DOM tampering detection (MutationObserver)
- Known device registry
- Traffic light badge (green/amber/red)
- Key locking on red status

### Phase 3: Independent API Channel
- Service worker fetch to SG/Send API
- Signed request headers (extension authentication)
- Device attestation reports
- Health check endpoint

### Phase 4: Cross-Device Communication
- Encrypted blob relay via SG/Send API
- 2FA approval flow (challenge on Device A, approve on Device B)
- Polling for pending requests (upgrade to SSE later)
- Key sharing between devices
- Session migration

### Phase 5: Distribution, Corporate Extensions, and Advanced
- Chrome Web Store listing switched to public
- "Install extension" prompts on sgraph.ai pages when extension not detected
- Microsoft Edge Add-ons store listing
- Full CI/CD pipeline: lint, test, build, publish (zero manual steps except approval gate)
- Corporate branded extension templating pipeline (`sg-extension create`)
- First corporate branded extension published
- Firefox port
- Other extension detection (management API)
- Monkey-patch detection for browser APIs
- Cross-browser sync fallback (vault-based bundle sync)
- Push notifications for real-time approval requests
