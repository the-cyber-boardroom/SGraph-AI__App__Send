## The Problem: Translations Don't Scale

We have 17 language files for the SG/Send website. Each is a JSON key-value map. When a customer reports a missing translation, the current workflow is:

1. Give the English file to an LLM agent
2. Give the target language file to the same agent
3. Agent reads both files entirely (~thousands of tokens each)
4. Agent finds the missing keys
5. Agent writes the entire target language file back

This is catastrophically token-inefficient. For 17 languages, adding one key means the agent reads and writes ~34 full files. And this is a hash table diff — a Python script can find missing keys in milliseconds with zero tokens.

This is the LLM cost engineering problem from yesterday's article, applied to our own workflow: **use the LLM to write the script, not to do the work.**

---

## The Solution: Engineered Translation Pipeline

### Step 1: Script, Not LLM

A simple script (Python or JS) handles the mechanical work:

```
1. Load en.json (English — the master)
2. Load pt-PT.json (target language)
3. Diff the keys: which English keys are missing from pt-PT?
4. Output: { "missing_key_1": "English text 1", "missing_key_2": "English text 2" }
5. Send ONLY the missing keys to the LLM for translation
6. LLM returns translated values for ONLY those keys
7. Script merges the translations into pt-PT.json
8. Done — the LLM never saw the full files
```

Token cost: from ~20K tokens (reading two full files) to ~500 tokens (translating 3-4 missing keys). A 40x reduction. Multiply by 17 languages.

### Step 2: CLI Tool

This becomes a CLI command:

```bash
sg-translate check --lang pt-PT          # Show missing keys
sg-translate translate --lang pt-PT      # Translate missing keys via LLM
sg-translate translate --all             # Translate missing keys for all languages
sg-translate validate --lang pt-PT       # Check for known bad translations
sg-translate audit                       # Full audit: missing, outdated, duplicates
```

The CLI wraps the diff script + LLM call + merge. It's also the tool that LLM agents use — instead of reading files manually, the Developer agent runs `sg-translate check` and gets a structured report.

### Step 3: Translation Quality Database

Pre-approved translations for common phrases, preventing the "enviado" vs "carregado" problem. If we know that "uploaded" should be "carregado" in pt-PT (not "enviado"), that mapping is stored and enforced.

```json
{
  "en": "Your file has been encrypted and uploaded.",
  "pt-PT": "O seu ficheiro foi encriptado e carregado.",
  "verified": true,
  "verified_by": "human",
  "context": "Success message after file upload without recipient"
}
```

When the LLM translates new text, it checks against the quality database: "Does this phrase reuse any terms we've already verified?" If "uploaded" appears, the LLM uses "carregado" (the verified translation) rather than guessing.

## babel.sgraph.ai — Translation Platform

### The Concept

A standalone website/service that hosts all translations, provides a translation API, and enables community contribution.

**Name options:** babel.sgraph.ai (Babel — the tower of languages), lingua.sgraph.ai, or translations.sgraph.ai. Babel is the strongest metaphor.

### What It Contains

| Feature | Description |
|---|---|
| **Translation registry** | All key-value pairs across all languages, with hashes, verification status, and context |
| **Shared vocabulary** | Pre-approved translations for common phrases. New projects check here first before creating new text. |
| **Translation API** | A service (or static JSON files) that any SG/Send project can consume at runtime |
| **Community contribution UI** | Human translators can review, correct, and add translations. Pull request workflow for changes. |
| **Quality dashboard** | Coverage per language, verified vs unverified, known issues, recently changed |

### The Economics of Shared Vocabulary

If five different text strings all say the same thing with minor variations ("Your file was uploaded", "File uploaded successfully", "Upload complete", "Your file has been uploaded", "File upload successful"), the accountant should flag: these are five separate translation targets across 17 languages = 85 translation units. If we consolidate to two variants, that's 34 translation units — a 60% reduction in ongoing translation cost.

The shared vocabulary enforces this: before creating new text, check if an existing verified translation says the same thing. If it does, use it. If a new variant is genuinely needed, it's allowed — but the cost is visible (new text × N languages).

## Translation Service Web Component

### The Concept

A web component (`<sg-translations>`) that runs as a service inside any sg-layout-hosted UI. Other components don't manage their own translations — they request translations from the service.

### How It Works

```
1. Component needs text → dispatches event: { key: "upload.success", lang: "pt-PT" }
2. <sg-translations> receives the event
3. Checks local cache (localStorage) → if found, returns immediately (synchronous)
4. If not cached, fetches from babel.sgraph.ai → returns when ready (async)
5. If service unavailable, falls back to English
6. Component renders the translated text
```

**Why this is powerful:**
- Components never manage translation files. They just ask for keys.
- The translation service caches aggressively (localStorage, in-memory)
- Adding a new language = loading new translation data, not changing any component code
- A translator UI is trivial: it's just a component that reads the same keys and lets you edit the values
- LLM-driven runtime translation: a local LLM could provide translations on-the-fly for unsupported languages

### Integration with sg-layout

The translation service is a "supporting service" component — it doesn't render visible UI (or renders a small language switcher). It sits inside the sg-layout tree as a service provider. Other panels query it via events. This is the same pattern as the EventBus, the API logger, and the execution history — invisible components that provide services to visible ones.

## Fractal CLIs — Every Project Gets a CLI

### The Principle

Every project, every tool, and eventually every component should have a CLI. The CLI serves two consumers: human developers and LLM agents.

**For humans:** `pip install sg-send-tools && sg-translate check --lang pt-PT`

**For LLM agents:** Instead of reading files and writing code to diff them, the agent runs `sg-translate check` and gets structured output. This is the "use LLMs to not use LLMs" principle applied to tooling — the CLI does the mechanical work, the LLM makes decisions based on the output.

### CLI Architecture

Build on the same class-based pattern from the vault CLI (Python classes with methods, CLI as a thin wrapper):

```python
class SGTranslate:
    def check(self, lang: str) -> dict:
        """Return missing translation keys for a language."""
        
    def translate(self, lang: str, keys: list) -> dict:
        """Translate specific keys using LLM."""
        
    def validate(self, lang: str) -> dict:
        """Check for known bad translations."""
        
    def audit(self) -> dict:
        """Full audit across all languages."""
```

The classes are importable (for use in other Python code or by LLM agents directly). The CLI is a thin wrapper using argparse or the existing CLI framework. No dependency on Typer — keep it consistent with the existing vault CLI pattern.

### Publishing

```bash
pip install sg-send-tools    # Installs: sg-translate, sg-vault, sg-send CLI tools
```

Each tool is a module inside the package. The CLI entry points are registered in `setup.py` / `pyproject.toml`.

## Connection to the Accountant

The Accountant role needs to be more involved in every work stream. Specific responsibilities for translation:

| Metric | What to Track |
|---|---|
| Tokens per translation update | How many tokens does the current workflow use vs the optimised pipeline? |
| Translation coverage | % of keys translated per language. Which languages are falling behind? |
| Duplicate text detection | How many strings say the same thing? What's the consolidation opportunity? |
| Cost per new text addition | Adding one English key = N translations needed. What's the actual cost? |
| Session token usage (Claude Code) | If the Accountant can see token usage per session, it becomes a quality metric: high tokens = poor engineering |

**The Accountant's principle:** Tokens are a bellwether for engineering quality. If a task uses many tokens, it either means the task is genuinely complex or the workflow isn't optimised. The Accountant distinguishes between the two.
