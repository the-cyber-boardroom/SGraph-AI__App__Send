## Part 1: Web Components as the Default Architecture

### The Principle

Every UI element we build from now on starts as a Web Component. Not "write HTML, then extract a component later." Start with the component. The question is always: **"Can this be a component?"** The answer is almost always yes.

The workspace proves this works at scale: 19 web components, event-driven communication via EventBus, each component a vanilla JS `HTMLElement` subclass, no framework, no Shadow DOM, independently versioned. The result: a 390KB application that's composable, maintainable, and reusable.

### What This Means in Practice

**The foundation is already designed.** The `sg-layout` framework (see companion documents: `v0_11_25__arch-brief__sg-layout-framework-architecture.md` and `v0_11_25__dev-brief__sg-layout-implementation-plan.md`) defines the fractal window management layer that all tools and UIs will use. Every panel, every tab, every resizable zone is an `sg-layout` instance hosting Web Components. The framework itself is a Web Component — `<sg-layout>` — and it can nest inside itself infinitely. This is the concrete implementation of the fractal architecture principle.

**Before (current pattern in many places):**
```html
<!-- Inline HTML with embedded JS -->
<div id="file-list">
  <script>
    // 200 lines of file management logic mixed with DOM manipulation
  </script>
</div>
```

**After (component-first pattern):**
```html
<!-- Self-contained web component -->
<sg-file-manager data-mode="edit" data-source="vault"></sg-file-manager>
```

The component encapsulates all its logic, renders into its container, communicates via events, and can be used anywhere — in send.sgraph.ai, vault.sgraph.ai, workspace.sgraph.ai, tools.sgraph.ai, or even in the future desktop app.

### Component Design Rules

These rules align with and extend the principles defined in the `sg-layout` architecture brief (Section 2: Core Design Principles). Components that follow these rules will work seamlessly inside `sg-layout` panels.

1. **Self-contained:** A component works with just its tag in the HTML. No external setup required beyond providing attributes/properties.
2. **Event-driven:** Components communicate via custom events on the EventBus, never by reaching into each other's DOM.
3. **No side effects:** Loading a component renders into its container. Unloading removes everything. No global state pollution.
4. **Themeable:** Components use CSS custom properties (the Aurora theme tokens) for styling. A consuming project can override the theme without modifying the component.
5. **Independently versioned:** Each component has its own version in the tools.sgraph.ai canonical library. Consumers pin to specific versions.
6. **Fractal:** A component can contain other components. The shell contains panels. Panels contain viewers. Viewers contain editors. Same pattern at every level.

### Daily Briefs Should Be Component-Level

The shift: instead of writing feature briefs ("add folder editing to the download page"), write component briefs ("build `<sg-file-manager>` that handles folder viewing, editing, drag-and-drop restructuring, and selective download — then integrate it into the download page, the upload page, the workspace, and the vault").

This ensures reusability from day one. The component is designed for multiple consumers, not retrofitted after being built for one.

### Workspace Migration

The current `workspace-shell.js` (32.5KB) implements its own five-zone CSS Grid layout with custom resize handles. This will be replaced by `sg-layout` — the workspace becomes a consumer of the framework rather than maintaining its own layout system. The sg-layout implementation plan (Section 15: Definition of Done for v0.2.0) explicitly lists the workspace migration as a requirement. Once migrated, the workspace gains for free: drag-to-dock panels, tabbed views, fractal nesting, layout serialisation/restore, and the digital twin.

## Part 2: The File Manager Component — First Component-Level Brief

### Why This Component

The ability to view, edit, restructure, and manage files within a zip/folder is needed by every UI in the ecosystem:

| Consumer | How They Use It |
|---|---|
| **send.sgraph.ai (upload)** | Select files to include/exclude before sending. Detect and warn about hidden files (.git, .env). |
| **send.sgraph.ai (download/view)** | Browse folder contents, preview files, select multiple for download. |
| **vault.sgraph.ai** | Browse vault contents, move files between folders, rename, delete. |
| **workspace.sgraph.ai** | Navigate vault files, open files for transformation, manage pipeline folders. |
| **view.sgraph.ai (viewers)** | Data room viewer, gallery viewer — all need file browsing with selection. |
| **tools.sgraph.ai** | Standalone "zip viewer/editor" tool — drag-drop a zip, browse, edit, re-download. |

One component. Six (or more) consumers. This is the argument for component-first architecture.

### Component: `<sg-file-manager>`

**Tag:** `<sg-file-manager>`

**Host:** Designed to run inside `sg-layout` panels (implements the `sg-layout:register` protocol from the sg-layout architecture brief) but also works standalone. When hosted in `sg-layout`, it receives resize notifications and adapts its layout. When standalone, it renders into its container at whatever size is available.

**Modes:**

| Mode | What It Does |
|---|---|
| `view` | Read-only browsing. Click to preview. Select for download. |
| `edit` | Full editing: rename, move, delete, add files, restructure folders. |
| `select` | Selection mode: checkboxes on each file/folder for batch operations. |
| `upload` | Pre-upload mode: shows what will be sent, allows deselection. Hidden file warnings. |

**Data sources:**

| Source | How It Gets Files |
|---|---|
| `vault` | Reads from an SG/Send vault via sg-vault.js |
| `zip` | Reads from an in-memory zip (JSZip) |
| `local` | Reads from a dropped folder (File API) |
| `url` | Fetches from a URL (for the URL-as-file pattern) |

**Events emitted:**

| Event | When |
|---|---|
| `file-selected` | User clicks a file (for preview/open) |
| `files-selected` | User selects multiple files (checkboxes) |
| `file-renamed` | User renames a file |
| `file-moved` | User moves a file to a different folder |
| `file-deleted` | User deletes a file |
| `file-added` | User adds a new file (drag-drop or file picker) |
| `structure-changed` | Any structural change (rename, move, delete, add) |
| `hidden-files-detected` | Hidden files found in upload (.git, .env, etc.) |

**Events listened to:**

| Event | Action |
|---|---|
| `vault-opened` | Load vault contents |
| `zip-loaded` | Load zip contents |
| `file-dropped` | Add dropped file to the tree |

### Standalone Tool: Zip/Folder Viewer-Editor

The same `<sg-file-manager>` component, wrapped in a minimal HTML page, becomes a standalone tool at `tools.sgraph.ai/file-manager/`:

- Drag-drop a folder or zip file
- Browse contents (tree view with file sizes, types)
- Preview files inline (PDF, images, markdown, code, text)
- Edit structure: rename files, move between folders, delete, add
- Re-download as zip (modified structure)
- No server, no account, no encryption — pure client-side file management

This is useful on its own (people need to inspect and restructure zips without installing tools) and serves as a demo of the component's capabilities.

## Part 3: Cross-Repo Agent Collaboration — Formalising the Pattern

### What Happened

The human was working in the `sgraph_ai__tools` repo session. They asked the agents to:

1. Clone the `SGraph-AI__App__Send` repo (the main send project)
2. Fire up the send team's Librarian and Developer agents (by reading their role definitions)
3. Have the send team agents review the tools team's IFD implementation
4. The send team found the tools team wasn't implementing IFD correctly
5. Return to the tools team agents with the feedback
6. The tools team implemented the corrections

This worked because the repos are well-structured: role definitions in known locations, status updates accessible, breadcrumbs maintained by the Librarian.

### Formalising the Pattern

This should be a repeatable, documented pattern: **"Summon a team from another repo."**

**The steps:**

1. The current session clones another repo (read-only)
2. The session reads the target repo's role definitions and status
3. The session instantiates the requested agents from that repo (Librarian, Developer, Architect, etc.)
4. The summoned agents review, advise, or collaborate
5. The findings are brought back to the current session's team
6. The current team acts on the findings

**Requirements for this to work:**

| Requirement | Why |
|---|---|
| Role definitions in standard locations | The summoning session needs to find them predictably |
| Current status document maintained | The summoned agents need to know the project's current state |
| BRIEF_PACK.md or equivalent | Fast bootstrap for summoned agents |
| Public/accessible repo | The summoning session needs to clone it |

### Skills Packaging

The human identified that guidance documents (how to do IFD properly, how to structure DevOps, how to write type-safe code) should be packaged as **skills** — formal, reusable documents that any agent team can import.

**What should become a skill:**
- IFD (In-Flight Development) methodology
- Web Components architecture patterns
- DevOps CI/CD pipeline structure
- Type-safe coding conventions
- Briefing pack creation and maintenance
- Agent role definition writing
- Cross-repo collaboration protocol

**Where skills live:** A shared skills repository (or a `skills/` folder in the main repo) that any team can reference. When spinning up a new team, bring the relevant skills. This is the same pattern as Claude's `/mnt/skills/` — but for our own team-specific knowledge.

**MCP integration:** As these skills mature, they could be exposed as MCP resources that Claude Code sessions can access natively, rather than requiring explicit file reads.

## Part 4: Desktop App Exploration

### The Need

The human finds it annoying to open markdown files on the computer — it fires up VS Code, Cursor, or other heavy editors. A lightweight app that just opens the SG/Send web UI with local storage access would be useful.

### Options

| Option | What It Is | Pros | Cons |
|---|---|---|---|
| **Electron** | Chromium-based desktop wrapper | Cross-platform, mature ecosystem, local storage access, can access filesystem | Heavy (~150MB+), another dependency |
| **Tauri** | Rust-based lightweight wrapper | Much smaller (~10MB), faster, native OS integration | Newer ecosystem, different webview per OS |
| **Chrome Extension as app** | The Chrome extension we're already building, with a dedicated window/tab | Zero additional install if extension exists, same codebase, syncs via Chrome Sync | Depends on Chrome, limited local filesystem access |
| **PWA (Progressive Web App)** | Install the website as an app from the browser | Zero build step, works today, browser handles updates | Limited local storage, no filesystem access, looks slightly different per OS |

### Recommendation: Start with PWA, Evolve to Electron/Tauri

**Phase 1: PWA.** Add a `manifest.json` and service worker to send.sgraph.ai. Users can "Install" it from Chrome's menu. It runs as a standalone window, has its own icon in the taskbar, and works offline for cached content. This costs almost nothing to implement and gives 80% of the desktop app experience.

**Phase 2: Chrome Extension.** The Chrome extension (already planned) provides local secret storage, cross-device sync, and security posture — this is the real "desktop" capability layer.

**Phase 3: Electron/Tauri (if needed).** Only if PWA + extension doesn't cover the use cases. The main driver would be local filesystem access (running a local vault, syncing with local folders) which neither PWA nor extension can do.

### What the Desktop App Would Do

- Open SG/Send web UI in a dedicated window (not a browser tab)
- **`sg-layout`-powered panel management** — the same fractal layout framework from the web, with drag-to-dock, tabs, and resizable panels working identically in the desktop context
- Local storage of secrets (access tokens, vault keys)
- Quick access to vaults and workspaces
- File type associations: open `.md`, `.pdf`, `.html` files in the SG/Send viewer
- Eventually: local vault storage, local file sync (like the CLI but with a GUI)
- Eventually: local LLM execution (Ollama integration from the workspace already supports this)
