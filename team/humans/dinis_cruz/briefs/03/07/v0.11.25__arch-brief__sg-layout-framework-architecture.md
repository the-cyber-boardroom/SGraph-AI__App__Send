# Architecture Brief: sg-layout — Fractal Window Management Framework

**Version:** v0.11.25
**Date:** 2026-03-06
**Role:** Architect
**Type:** Strategic architecture brief
**Status:** Pre-implementation — awaiting v0.1.0 kickoff

---

## 1. Strategic Intent

`sg-layout` is a standalone, zero-dependency, vanilla JS + Web Components window management framework. Its single responsibility is layout — placing, sizing, tabbing, resizing, and moving panels that host arbitrary HTML content.

It has no opinion about what lives inside its panels. A panel host can be a tool, an app, another `sg-layout` instance, or a plain `<div>`. The framework treats them all identically.

This document covers the architectural decisions, the design principles, the component model, the event architecture, and the fractal composition model. The implementation plan is in the companion Dev brief.

---

## 2. Core Design Principles

### 2.1 Single Responsibility

`sg-layout` does exactly one thing: layout. It does not own:
- Application state
- Routing
- Authentication
- Theme tokens (it *consumes* CSS custom properties but does not define them)
- API communication
- Business logic of any kind

If a feature does not directly relate to "where is this panel and how big is it", it does not belong in `sg-layout`.

### 2.2 Fractal Composition

An `sg-layout` instance is a valid panel host for another `sg-layout` instance. This is not a special case — it is the natural consequence of the portal contract. The outer layout treats the inner layout identically to any other hosted component.

```
sg-layout (root)
├─ panel A: <video-splitter>          ← a tool
├─ panel B: <sg-layout>               ← a nested layout
│            ├─ panel B1: <llm-client>
│            └─ panel B2: <file-hasher>
└─ panel C: <sg-layout>               ← another nested layout
             ├─ panel C1: <sg-layout> ← arbitrarily deep
             └─ panel C2: <my-app>
```

Every node in this tree is self-contained. Every `sg-layout` knows its immediate parent layout (injected at connect time). No node knows its grandparent or deeper ancestors.

### 2.3 Zero Coupling

Hosted components do not import `sg-layout`. They do not reference it by name. The only shared surface between a hosted component and the framework is:

- The event name string `'sg-layout:register'` (imported from `sg-layout-events.js`)
- The `dataset.panelId` attribute (a string, injected by the framework)
- A direct reference to the parent `sg-layout` instance (injected at connect time as `_sgLayoutRef`)

A component that does not use any of these three things still works inside `sg-layout` — it just does not participate in the resize notification system.

### 2.4 Shadow DOM Everywhere

Every framework component (`sg-layout`, and its internal elements) uses Shadow DOM. This provides:

- True CSS isolation — no leakage between framework chrome and hosted content
- Structural enforcement of the portal contract — hosted components cannot query upward through a shadow boundary by accident
- Clean slot-based composition — the framework controls exactly which DOM is projected where

**The theme bridge:** CSS custom properties pierce shadow boundaries by spec. All visual customisation of `sg-layout` is done via custom properties. No class-based overrides from outside are supported or needed.

### 2.5 Synchronous Bootstrap

The entire layout construction, component registration, and initial render sequence completes synchronously, within a single call stack, before any frame is painted. The user never sees a partially constructed layout. Components never render twice.

This is achieved via the synchronous `dispatchEvent` / `CustomEvent` registration protocol described in Section 5.

## 3. The Layout Tree Model

The internal state of an `sg-layout` instance is a **layout tree** — a recursive structure of three node types:

```js
// Row: children arranged horizontally, separated by resize handles
{ type: 'row',    id, children: [node, ...], sizes: [0.5, 0.5] }

// Column: children arranged vertically, separated by resize handles
{ type: 'column', id, children: [node, ...], sizes: [0.6, 0.4] }

// Stack: one or more tabs sharing a single slot, one active at a time
{ type: 'stack',  id, tabs: [tab, ...], activeTab: 0 }

// Tab: a leaf node — references a hosted element
{ type: 'tab', id, title, icon, tag, state, el }
```

`sizes` arrays are normalised fractions that always sum to 1.0. They are recalculated on every resize drag. They are the serialisation format — layout JSON is nothing more than the tree with `el` stripped out.

The DOM is a **projection** of the tree. When the tree changes, the affected subtree re-renders. The tree is the source of truth. The DOM is never queried to understand layout state.

## 4. Component Architecture

### 4.1 Public Surface: One Element

`sg-layout` is the only custom element exposed publicly. All internal elements (panel chrome, tab bar, resize handle, drop zone) live inside `sg-layout`'s Shadow DOM and are never registered as custom elements on `customElements`. They are private implementation detail.

```html
<!-- everything the consumer ever writes -->
<sg-layout id="root" layout='{...}'>
    <video-splitter data-panel-title="Video Splitter"></video-splitter>
    <sg-layout data-panel-title="Tools">
        <llm-client data-panel-title="LLM"></llm-client>
    </sg-layout>
</sg-layout>
```

### 4.2 Internal Structure (inside Shadow DOM)

```
sg-layout (Shadow DOM root)
├─ .sgl-root                          ← top-level flex/grid container
│   ├─ .sgl-row / .sgl-column        ← recursive layout containers
│   │   ├─ .sgl-stack                ← tab container
│   │   │   ├─ .sgl-tab-bar          ← tab strip (private)
│   │   │   └─ .sgl-slot-wrapper     ← wraps <slot>
│   │   │       └─ <slot name="p-{id}"> ← hosted content projected here
│   │   └─ .sgl-resize-handle        ← drag target (private)
│   └─ ...
└─ .sgl-drag-overlay                  ← drop zone overlay (hidden until drag)
```

### 4.3 CSS Custom Properties (the theme contract)

```css
/* Geometry */
--sgl-handle-size: 4px;
--sgl-tab-height: 28px;
--sgl-panel-header-height: 32px;
--sgl-min-panel-size: 80px;

/* Colours */

/* Motion */
--sgl-transition-speed: 150ms;
--sgl-reveal-transition: opacity 120ms ease;
```

No other CSS crosses the shadow boundary.

## 5. The Event Architecture

### 5.1 Three Channels

| Channel | Direction | Mechanism | Use case |
|---|---|---|---|
| **Internal bus** | within one `sg-layout` | `EventBus` instance | resize, tab switch, drag, panel state |
| **Upward bubble** | child → ancestor | `CustomEvent` `composed:true` | registration, cross-boundary drag |
| **Downward injection** | parent → child | direct method call | focus, force-close, state push |

### 5.2 The Registration Protocol (synchronous)

This is the wiring mechanism. It runs synchronously inside `connectedCallback`.

```
hosted component connectedCallback()
│
├─ dispatch 'sg-layout:register' { composed:true, bubbles:true }
│   detail: { respond: (config) => { ... } }   ← synchronous callback
│
│   event bubbles up through shadow boundaries
│
└─ nearest sg-layout shadowRoot listener fires
    │
    ├─ event.stopPropagation()         ← claim it, don't let it go further
    ├─ resolve panel for event.target
    ├─ call event.detail.respond({
    │      panelId,
    │      layoutRef,     ← direct reference to this sg-layout instance
    │      depth,         ← nesting depth (0 = root)
    │      bounds         ← initial { width, height }
    │  })
    └─ return
    
back in hosted component connectedCallback():
    config is now populated (dispatchEvent is synchronous)
    this._panelId   = config.panelId
    this._layoutRef = config.layoutRef
    this._render()  ← first render with full context, correct dimensions
```

### 5.3 The Fractal Registration Case

When an inner `sg-layout` is hosted inside an outer `sg-layout`, both registrations happen — but they are independent events at different times:

1. Outer `sg-layout` connects → builds its tree → appends inner `sg-layout` to a slot
2. Inner `sg-layout` `connectedCallback` fires → dispatches `sg-layout:register`
3. **Outer** layout catches it (it is the nearest `sg-layout` ancestor) — inner layout gets `layoutRef = outerLayout`, `depth = 1`
4. Inner `sg-layout` now builds its own tree → appends its hosted components to slots
5. Each hosted component dispatches `sg-layout:register`
6. **Inner** layout catches it (`stopPropagation` prevents it reaching the outer) — each component gets `layoutRef = innerLayout`, `depth = 2`

Every component sees exactly its immediate parent layout. No component sees a grandparent.

### 5.4 Infinite Loop Detection

The fractal model creates one class of hazard: a component that programmatically adds an `sg-layout` which programmatically adds the same component, producing a registration cycle.

Detection signature — **unambiguous and unique**:

```
'sg-layout:register' fires
→ sg-layout handler fires
→ handler calls addPanel() programmatically
→ addPanel() appends element to slot
→ element.connectedCallback fires
→ element dispatches 'sg-layout:register'
→ [same sg-layout handler fires again, for a child of a panel just created]
```

The differentiator: this second `register` event fires **synchronously inside the first handler** — i.e. the call stack depth increases while a `register` handler is already on the stack.

Detection implementation:

```js
// sg-layout internal state
this._registerDepth = 0;
const MAX_REGISTER_DEPTH = 10;  // generous, legitimate nesting never exceeds 4-5

// in the register handler:
shadowRoot.addEventListener('sg-layout:register', (event) => {
    this._registerDepth++;
    
    if (this._registerDepth > MAX_REGISTER_DEPTH) {
        console.error(`[sg-layout] Circular registration detected at depth 
            ${this._registerDepth}. Panel id: ${event.target.id || event.target.tagName}.
            Breaking cycle.`);
        this._registerDepth--;
        event.stopPropagation();
        return;  // do not call respond() — component degrades gracefully to standalone
    }
    
    // ... normal handling ...
    
    this._registerDepth--;
});
```

Additional guard: each `sg-layout` maintains a `Set` of element references it has already registered. If the same element reference appears in a second `register` event, it is a cycle — log and break.

## 6. The Serialisation Contract

Layout state serialises to a clean JSON tree with no element references:

```json
{
    "type": "row",
    "id": "r1",
    "sizes": [0.5, 0.5],
    "children": [
        {
            "type": "stack",
            "id": "s1",
            "activeTab": 0,
            "tabs": [
                { "id": "t1", "title": "Video Splitter", "tag": "video-splitter", "state": {} }
            ]
        },
        {
            "type": "stack",
            "id": "s2",
            "activeTab": 0,
            "tabs": [
                { "id": "t2", "title": "Tools", "tag": "sg-layout",
                  "state": {
                      "layout": {
                          "type": "column",
                          "children": [...]
                      }
                  }
                }
            ]
        }
    ]
}
```

When an `sg-layout` is a hosted component (fractal case), its inner layout tree serialises into the `state` field of its parent tab. Full depth is preserved. Restore is deterministic.

Hosted components participate via an optional interface:

```js
// component declares these if it wants state persistence
getLayoutState() { return { ...myCurrentState }; }   // called on serialise
setLayoutState(state) { this._restore(state); }       // called on restore
```

If a component does not implement these, its tab is restored with an empty state object — the component starts fresh.

## 7. Tools as Components, Components as Tools

The registration protocol and the fractal model together mean there is no distinction between a "tool" and a "component" from the framework's perspective. The same element works in both contexts:

```html
<!-- As a tool: standalone page on tools.sgraph.ai -->
<body>
    <video-splitter></video-splitter>
</body>

<!-- As a component: inside a panel -->
<sg-layout>
    <video-splitter data-panel-title="Video Splitter"></video-splitter>
</sg-layout>

<!-- As a component inside a nested layout inside a panel -->
<sg-layout>
    <sg-layout data-panel-title="Media Tools">
        <video-splitter data-panel-title="Video Splitter"></video-splitter>
    </sg-layout>
</sg-layout>
```

`video-splitter.connectedCallback` handles all three cases:

```js
connectedCallback() {
    let config = null;
    this.dispatchEvent(new CustomEvent('sg-layout:register', {
        bubbles: true, composed: true,
        detail: { respond: c => { config = c; } }
    }));

    if (config) {
        // inside an sg-layout — use injected dimensions, register for resize
        this._initWithLayout(config);
    } else {
        // standalone — fill viewport, no resize events
        this._initStandalone();
    }
}
```

One component. Two contexts. Zero branching in the framework.

## 8. The Dependency Graph

```
sg-layout-events.js      ← event name constants only. No imports.
       ↑
sg-layout.js             ← imports event names only
       ↑                    no other imports
hosted components        ← import event names only (optional)
                            no import of sg-layout itself
```

`sg-layout.js` has zero runtime dependencies. It does not import any other module. It registers one custom element. It publishes nothing to `window`. The only globals it reads are `document`, `customElements`, `CustomEvent`, and `requestAnimationFrame`.

## 9. Bidirectional Host Compatibility

`sg-layout` works in both directions without special handling:

**Direction 1 — sg-layout inside plain HTML:**
`sg-layout` can be placed inside any HTML element — `<body>`, `<div>`, `<table>`, a third-party component. The registration event fires with `composed:true` and bubbles upward. If no `sg-layout` ancestor responds, the instance treats itself as a root layout and runs standalone. No configuration required.

**Direction 2 — plain HTML inside sg-layout:**
Any HTML element — a `<div>`, a `<canvas>`, a legacy widget with no shadow DOM — works as a hosted component. It simply does not participate in the registration protocol. The panel chrome wraps it, resize handles work, tab switching hides/shows it. It is unaware of the framework.

**The panel = maximised div contract:**
Every panel slot wrapper enforces this CSS so that any hosted element that sets `width: 100%; height: 100%` fills its panel correctly:

```css
.sgl-slot-wrapper {
    display: block;
    width: 100%;
    height: 100%;
    overflow: hidden;
    position: relative;
    box-sizing: border-box;
}
```

This is an explicit acceptance criterion for v0.1.0: *a component that sets itself to 100% width and 100% height, with no other layout awareness, must fill its panel correctly at all times — including after resize, collapse, and tab switch.*

## 10. The Command Bus (sg-commands)

The layout framework's internal event bus handles layout events. A separate companion module — `core/sg-commands/` — handles application-level commands: hide/show, CSS updates, JS method calls, theme changes, data pushes.

`sg-layout` and `sg-commands` are independent modules. Neither depends on the other. A page can use `sg-layout` without `sg-commands`, and vice versa.

### Command addressing

Commands are addressed by `panelId` (point-to-point) or broadcast (all panels):

```js
// Point-to-point — only the panel with this id receives it
window.sgCommands.send('panel-xyz', {
    type: 'css-update',
    payload: { '--accent-color': '#ff0000' }
});

// Broadcast — all panels receive it
window.sgCommands.broadcast({
    type: 'theme-change',
    payload: 'dark'
});

// Path-addressed — panels matching a tree path receive it
window.sgCommands.sendTo('row[0]/stack/**', {
    type: 'hide'
});
```

### Component subscription

A hosted component subscribes in `connectedCallback` after registration resolves:

```js
connectedCallback() {
    // ... registration ...
    window.sgCommands?.subscribe(this._panelId, cmd => this._handleCommand(cmd));
}

disconnectedCallback() {
    window.sgCommands?.unsubscribe(this._panelId);
}
```

### Command types (initial set)

| Type | Payload | Effect |
|---|---|---|
| `show` | — | Make panel visible |
| `hide` | — | Hide panel |
| `focus` | — | Bring tab to front |
| `css-update` | `{ prop: value }` | Set CSS custom properties on the panel element |
| `js-exec` | method name + args | Call a named method on the component |
| `state-push` | any object | Deliver data to the component |
| `theme-change` | theme name | Broadcast theme switch |
| `close` | — | Close the panel |

Components only handle command types they declare support for. Unknown command types are silently ignored. No component can be forced to execute arbitrary JS — `js-exec` calls a named method that the component explicitly exposes in a `sgCommands` object on its class.

## 11. PKI Per Panel — Identity and Encrypted Messaging

Every panel instance gets a cryptographic keypair on registration. This is provided by a companion module `core/sg-panel-identity/`.

### What this solves

1. **Unique unforgeable identity** — two panels hosting the same component tag have different keypairs. You can never accidentally address the wrong one.
2. **Rogue component containment** — a component cannot impersonate another panel's identity. It does not have the private key.
3. **Encrypted point-to-point messaging** — a message encrypted to a panel's public key can only be decrypted by that panel. No routing table, no permission system — the cryptography enforces it structurally.
4. **Web Worker communication** — when panel work moves to a Worker, the keypair moves with it. Messages from the main thread encrypted to the public key can only be decrypted in that Worker.
5. **Persistent cross-session identity** — export the keypair at serialisation time. Restore it at deserialisation time. The panel has the same identity across page loads.

### Key generation

ECDH P-256 is used rather than RSA-OAEP. Reasons: key generation is ~1ms (vs ~20ms for RSA-2048), equivalent security for this use case, smaller key sizes in serialised output.

```js
// sg-panel-identity generates a keypair per panel
const { publicKey, privateKey } = await crypto.subtle.generateKey(
    { name: 'ECDH', namedCurve: 'P-256' },
    true,       // extractable — for export at serialisation time
    ['deriveKey', 'deriveBits']
);
```

Key generation is async. The bootstrap sequence accommodates this:

```
sg-layout connects
→ tree is built and rendered (synchronous)
→ all panels are mounted and registered (synchronous)
→ layout:ready fires
→ sg-panel-identity generates keypairs in parallel (async, Promise.all)
→ each panel receives its keypair via direct injection (_sgPrivateKey, dataset.panelPublicKey)
→ layout:identities-ready fires
→ page is revealed (requestAnimationFrame)
```

The synchronous bootstrap sequence is preserved. Identity assignment happens as a second async phase before the layout is made visible.

### The key registry

Each `sg-layout` instance maintains a public key registry — a `Map<panelId, CryptoKey>`. Any component can read the public key of any panel via:

```js
const pubKey = this._sgLayoutRef.getPublicKey('panel-xyz');
```

Private keys never leave the component. The registry is read-only from outside the framework.

### Relationship to sg-commands

`sg-commands` can optionally use PKI for encrypted point-to-point delivery. If `sg-panel-identity` is loaded, `sgCommands.send()` automatically encrypts payloads to the target panel's public key. If not loaded, commands travel unencrypted. The choice is a deployment decision, not an API change.

## 12. Digital Twin — Complete Serialisation

The layout tree, component states, geometry, identity, and timestamp combine into a single serialisable document — the **digital twin** of the UI at any moment in time.

### The twin document structure

```json
{
    "sgLayoutVersion": "v0.1.0",
    "id": "layout-root-abc",
    "timestamp": 1741305600000,
    "geometry": { "width": 1200, "height": 800 },
    "tree": {
        "type": "row",
        "id": "r1",
        "sizes": [0.5, 0.5],
        "children": [
            {
                "type": "stack",
                "id": "s1",
                "activeTab": 0,
                "tabs": [{
                    "id": "panel-xyz",
                    "tag": "video-splitter",
                    "title": "Video Splitter",
                    "publicKey": "base64-exported-ecdh-public-key",
                    "componentState": { "currentFile": "demo.mp4", "splitMode": "fixed" }
                }]
            },
            {
                "type": "stack",
                "id": "s2",
                "activeTab": 0,
                "tabs": [{
                    "id": "panel-def",
                    "tag": "sg-layout",
                    "title": "Tools",
                    "publicKey": "base64-exported-ecdh-public-key",
                    "componentState": {
                        "sgLayoutVersion": "v0.1.0",
                        "id": "layout-inner-ghi",
                        "timestamp": 1741305600000,
                        "geometry": { "width": 600, "height": 800 },
                        "tree": { "type": "column", "children": [...] }
                    }
                }]
            }
        ]
    }
}
```

### The fractal serialisation rule

Each layer is responsible for serialising exactly its own state. It asks its children for theirs. Nobody knows about grandchildren. An inner `sg-layout`'s full twin document lives in the `componentState` field of its parent tab. The structure is infinitely recursive and always valid JSON.

### Serialisation on demand — the pull protocol

At any moment, any code can ask the entire tree for its current state:

```js
// Synchronous pull — components return their current state immediately
const twin = layout.getTwin();   // full digital twin, all depths

// Async pull — for components whose state requires async capture (e.g. canvas snapshot)
const twin = await layout.getTwinAsync();
```

Components implement whichever protocol they support:

```js
// Sync (preferred)
getLayoutState() { return { mode: this._mode, file: this._currentFile }; }

// Async (for components that need it)
async getLayoutStateAsync() {
    const snapshot = await this._captureScreenshot();
    return { mode: this._mode, snapshot };
}
```

The framework calls `getLayoutStateAsync` if it exists, falls back to `getLayoutState`, falls back to `{}`. No component is required to implement either.

## 13. Mini DSL — Path-Based Tree Addressing

A thin API layer on top of the layout tree enables concise programmatic layout construction and command routing. The path syntax is XPath-inspired, adapted for the row/column/stack/tab tree structure.

### Path syntax

```
row[0]          first child that is a row
column[1]       second child that is a column
stack[0]        first stack
tab[0]          first tab in the nearest stack
*               any single node (wildcard)
**              all descendants (deep wildcard)
panel-xyz       a specific panel by id
```

### Usage

```js
const L = document.querySelector('sg-layout');

// Read
L.at('row[0]/stack[0]').getTabs();
L.at('panel-xyz').getBounds();

// Modify
L.at('row[0]/stack[0]').addTab({ tag: 'llm-client', title: 'LLM' });
L.at('panel-xyz').split('right', { tag: 'file-hasher' });
L.at('row[1]').setSizes([0.3, 0.7]);

// Commands (requires sg-commands)
L.at('panel-xyz').send({ type: 'hide' });
L.at('row[0]/**').broadcast({ type: 'theme-change', payload: 'dark' });
L.at('**').broadcast({ type: 'state-push', payload: { user: 'dinis' } });
```

### Layout construction DSL

Complex layouts can be built programmatically in a readable chain:

```js
SgLayout.build()
    .row([
        SgLayout.stack([
            { tag: 'video-splitter', title: 'Video' }
        ]),
        SgLayout.column([
            SgLayout.stack([{ tag: 'llm-client', title: 'LLM' }]),
            SgLayout.stack([{ tag: 'file-hasher', title: 'Hasher' }]),
        ], [0.6, 0.4])
    ], [0.5, 0.5])
    .mountTo('#root');
```

The builder produces the same JSON tree that `setLayout()` consumes. It is purely a convenience wrapper — no new concepts.

## 14. What sg-layout Is NOT

To keep the scope tight:

- **Not a router.** Navigation between views is the host application's responsibility.
- **Not a state manager.** Component state is each component's responsibility.
- **Not a theme system.** It consumes CSS custom properties. It does not define them.
- **Not an animation library.** One CSS transition for panel reveal. Nothing else.
- **Not a drag-and-drop library.** It implements the specific drag behaviour needed for panel rearrangement only. No general-purpose DnD API.
- **Not a virtual DOM.** The layout tree is a structural model, not a vDOM diffing system. Re-renders are surgical and targeted.
- **Not a command bus.** `sg-commands` is a companion module. `sg-layout` does not own application-level messaging.
- **Not an identity provider.** `sg-panel-identity` is a companion module. `sg-layout` requests identities from it but does not implement crypto.
