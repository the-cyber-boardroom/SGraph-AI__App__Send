## Part 1: The Post-Purchase Gap

### The Problem

The only real friction point in the customer journey is the moment after purchase. The user pays via Stripe and then... nothing. They wait for the human to manually provision a token and send it. Currently that happens within an hour, but the user doesn't know that. There's a dead gap between "I just paid" and "I can use the product."

We're not ready for full automation yet (the email pipeline and CRM folders from yesterday's brief need building first). But there's a simple interim solution.

### The Solution: Redirect to a Temp Token Page

After Stripe checkout completes, redirect the user to a page on send.sgraph.ai that:

1. Thanks them for purchasing
2. Automatically applies a **temporary shared token** to their browser
3. Lets them start using the product immediately

```
Stripe checkout complete
    → Redirect to: send.sgraph.ai/welcome#{encrypted-token-ref}
    → Page JavaScript:
        1. Read the hash fragment (decryption key)
        2. Fetch the encrypted token from SG/Send
        3. Decrypt it client-side
        4. Save to localStorage
        5. Display: "Your temporary token is active. Start sending files now!"
        6. Note: "Your personal token will be provisioned within 24 hours."
```

### The Temp Token Model

The token on the page is a **shared temporary token** — the same one for everyone who buys in a given period. It has limited credits that the human tops up periodically.

**Security model:** Yes, the token could leak (it's on a URL). But at this stage, leaking a shared token with limited credits is an acceptable risk — the human monitors usage and rotates the token when needed. The personal token (provisioned later) is the permanent, secure one.

**Monitoring opportunity:** The page load gives us analytics: how many people hit the post-purchase page? How quickly after purchase? Do they immediately try to send a file? This is conversion funnel data we don't have today.

### The SG/Send Delivery Model

The token is not hardcoded on the page or in GitHub. Instead:

1. The human creates a token and uploads it to SG/Send (encrypted)
2. The page contains only the SG/Send decryption key (in the URL hash fragment, or in the page JS)
3. On page load, the JS fetches the encrypted token from SG/Send and decrypts it client-side
4. The server never sees the plaintext token

This means: rotating the token = uploading a new encrypted token to SG/Send. The page code doesn't change. The decryption key is the same (or updated in the redirect URL in Stripe's settings).

## Part 2: Individual File Storage Architecture

### The Problem

Currently, when a user uploads a zip/folder, the entire thing is encrypted and stored as a single blob. A 1GB zip with 50 files = one 1GB encrypted blob. To view any file, the recipient must download and decrypt the entire 1GB.

### The Solution: Store Files Individually

Instead of one blob, store: (a) a tree structure file (metadata: filenames, sizes, types, hashes, locations) and (b) each file as a separate encrypted blob.

```
Upload: project.zip (50 files, 1GB total)
    → Tree structure: 5KB (filenames, sizes, types, hashes, blob IDs)
    → File 1: 200KB encrypted blob (ID: abc123)
    → File 2: 15MB encrypted blob (ID: def456)
    → ...
    → File 50: 50MB encrypted blob (ID: xyz789)
```

### The Tree Structure Document

```json
{
  "version": "1.0",
  "total_files": 50,
  "total_size": 1073741824,
  "decryption_key_hash": "sha256_of_shared_key",
  "files": [
    {
      "path": "src/index.html",
      "size": 3501,
      "type": "text/html",
      "extension": "html",
      "hash": "sha256_of_plaintext",
      "blob_id": "abc123",
      "public_key": null
    },
    {
      "path": "assets/video.mp4",
      "size": 52428800,
      "type": "video/mp4",
      "extension": "mp4",
      "hash": "sha256_of_plaintext",
      "blob_id": "def456",
      "public_key": null
    }
  ]
}
```

### Why This Is a Killer Feature

**Instant browsing:** Download the tree structure (5KB, instant). See all filenames, sizes, types. Browse the folder without downloading anything.

**On-demand download:** Click a file → download and decrypt just that file. A 200KB HTML file from a 1GB zip downloads in milliseconds, not minutes.

**Three file states in the UI:**

| State | Icon/Visual | Meaning |
|---|---|---|
| **Remote** | Cloud icon, dimmed | File exists on server. Not yet downloaded. |
| **In browser** | Filled icon | Downloaded and decrypted in browser memory. Will be lost on page refresh. |
| **Cached locally** | Filled icon + pin | Stored in localStorage/IndexedDB. Survives page refresh. |

This is the same UX pattern as Google Drive, iCloud, and Dropbox's smart sync: visual distinction between local and cloud files.

**Decouple download from decryption.** Currently: download blob → decrypt blob (single pipeline). With individual files: download tree → show structure → user clicks file → download file blob → decrypt file blob. The user sees something immediately (the tree), then each file loads independently.

### Performance at Scale

For large uploads (1-10GB+), the difference is dramatic:

| Approach | Download to view one file | Download to see file list |
|---|---|---|
| **Current (single blob)** | Download entire 1GB, decrypt entire 1GB | Same — download everything |
| **Individual files** | Download 5KB tree + download that one file | Download 5KB tree (instant) |

### PKI: Per-File Encryption Keys (Future)

The `public_key` field in the tree structure enables a future killer feature: different files in the same zip can have different encryption keys.

**Use case:** A data room with 10 documents. Documents 1-5 are encrypted with a shared key (everyone can see them). Documents 6-10 are encrypted with specific recipients' public keys (only those recipients can decrypt them). The tree structure is visible to everyone (they can see that 10 documents exist), but only authorised recipients can decrypt specific files.

### Limits and Implementation

**File count limit:** 100 files per upload for now (individual storage). Above 100, fall back to single-blob storage. This is pragmatic — 100 individual S3 puts per upload is manageable; 10,000 is not (yet).

**Immutability:** Once uploaded, the blob set is immutable. Adding a file creates a new tree structure that references the existing blobs plus the new one. The original tree is preserved. This is the git model: content is immutable, references are mutable.

**Updating files:** If the uploader has the ID and decryption key, they can upload a new version of a specific file. The tree is updated to point to the new blob. Previous versions can be preserved (version history per file). This needs a security assessment — the ability to update requires authentication, which currently means "you have the decryption key."

**Live upload sharing:** Because the tree is uploaded first and files are uploaded individually, the share link can be active before all files finish uploading. The recipient sees the tree, sees which files are uploaded (and which are still in progress), and can download completed files immediately. Real-time collaboration during upload.

|---|---|
| 1 | Stripe redirects to send.sgraph.ai/welcome after purchase |
| 2 | Welcome page fetches encrypted token from SG/Send and decrypts client-side |
| 3 | Token auto-saved to localStorage — user can send files immediately |
| 4 | "Your personal token will be provisioned within 24 hours" message shown |

| # | Criterion |
|---|---|
| 6 | Zip/folder uploads store tree structure + individual file blobs |
| 7 | Tree structure downloadable independently (instant file list browsing) |
| 8 | Individual files downloadable on demand |
| 9 | Three file states in UI: remote, in browser, cached locally |
| 10 | File count limit: 100 files individually stored, fallback to single blob above |
| 11 | Live upload sharing: link active before all files finish uploading |

| # | Criterion |
