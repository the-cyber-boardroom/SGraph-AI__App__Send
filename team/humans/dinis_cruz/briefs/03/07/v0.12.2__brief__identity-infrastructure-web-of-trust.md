### 2. DNS TXT Records

Publish a public key as a DNS TXT record on dinis.ai and sgraph.ai:

```
sgraph.ai  TXT  "sg-pubkey=base64_encoded_public_key"
sgraph.ai  TXT  "sg-verify=https://sgraph.ai/verify"
```

Anyone can query DNS and get the public key. Messages signed with the corresponding private key are verifiable against this published key.

### 3. GitHub Profile Key

Add the same public key to the GitHub profile and to the repo (`.well-known/sg-identity.json`):

```json
{
  "entity": "SGraph / The Cyber Boardroom Ltd",
  "operator": "Dinis Cruz",
  "public_key": "base64_encoded_public_key",
  "verification": [
    { "type": "dns", "domain": "sgraph.ai" },
    { "type": "web", "url": "https://sgraph.ai/verify" },
    { "type": "github", "profile": "DinisCruz" }
  ],
  "signed": "base64_signature_of_this_document"
}
```

This is a self-signed identity document that cross-references multiple verification points.

## Medium-Term Actions (1-3 Months)

### 4. Agent Identity Framework

Each agent role gets a key pair:

```json
{
  "agent": "sherpa",
  "team": "explorer",
  "public_key": "base64_key",
  "authorised_by": "dinis_cruz_public_key",
  "capabilities": ["send_messages", "review_feedback", "create_briefs"],
  "created": "2026-03-07",
  "signature": "base64_signature_by_authoriser"
}
```

**Chain of trust:** Dinis → signs Agent Identity → Agent acts on behalf of Dinis → Recipient verifies Agent's key → checks it's signed by Dinis's key → trusts the message.

**Cross-team agent communication:** When the tools team's Developer asks the send team's Librarian for a review (as happened yesterday), the request should be signed by the tools Developer's key. The send Librarian verifies the signature against the tools team's registered agents.

### 5. Signed Messages in Customer Communication

When the Sherpa sends a welcome email or a WhatsApp message, include a verification link:

> "This message was sent by the SG/Send Sherpa on behalf of Dinis Cruz. Verify: sgraph.ai/verify/msg/abc123"

The verification link shows: who sent it, when, the signed content hash, and how to verify. This doesn't require the recipient to understand cryptography — they just click a link and see "This message is verified."

### 6. Authentication by Decryption

The principle: **authenticate via mathematics, not via a table in a database.**

If I send you a message encrypted with your public key, only you can decrypt it. The act of decrypting IS the authentication. No passwords, no login pages, no database lookups.

Applied to agent communication: Agent A encrypts a request with Agent B's public key. Agent B can decrypt it. No other agent can. This is authentication, authorisation, and confidentiality in one operation.

## Long-Term Vision

### 7. Web of Trust at Scale

Multiple verification points that collectively make impersonation difficult:

| Verification Point | Strength | Who Controls It |
|---|---|---|
| DNS TXT record | Strong (requires domain access) | Domain owner |
| Company website page | Medium (requires hosting access) | Company |
| GitHub profile/repo | Medium (requires account access) | Individual |
| LinkedIn profile | Weak (no real verification) | Platform |
| Companies House registration | Strong (legal document) | Government |
| Signed messages on multiple platforms | Cumulative | Individual |

Each point adds a strand. An attacker needs to compromise multiple independent systems simultaneously.

### 8. Multilingual and Multicultural Identity

As we communicate with people in different countries, identity verification has cultural dimensions:
- Portugal: NIF (tax ID), Cartão de Cidadão (citizen card)
- UK: Companies House, National Insurance number
- International: Passport, government ID

Each culture has different identity anchors. The web of trust should accommodate all of them without requiring any single one.
