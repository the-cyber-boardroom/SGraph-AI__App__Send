# Brief: sg-layout Multi-Disciplinary Review Commission — Four Documents

**version** v0.12.2  
**date** 7 Mar 2026  
**from** Human (project lead / Conductor)  
**to** Architect, Developer, Designer, Sherpa, AppSec, Librarian, Conductor  
**type** Review commission — multi-perspective analysis of sg-layout before implementation  

---

## Why This Review

The sg-layout architecture brief and implementation plan (see companion documents `v0_11_25__arch-brief__sg-layout-framework-architecture.md` and `v0_11_25__dev-brief__sg-layout-implementation-plan.md`) were written by an LLM agent that did not have the full context of working inside the workspace. The briefs are solid — technically correct, well-structured — but they haven't been stress-tested against reality: the actual layout pain we've experienced, the real consumers that will use this, and the edge cases that only emerge from building on top of something.

This is a core primitive. It must be town-planner quality from the start. It cannot be a Genesis project. It should rarely need to change after v0.2.0. So we invest upfront in getting it right.

---

## Who Reviews and Why

The direct consumer of sg-layout is not the end user — it's every tool, component, and developer (human or LLM) that builds on top of it. The end user experiences it indirectly through better layouts, consistent theming, and productive UI. But the direct customer is the code that hosts inside it.

This means the review needs perspectives beyond just Architect and Developer:

| Reviewer | What They Examine |
|---|---|
| **Architect** | Structural soundness, event model, fractal composition, serialisation model |
| **Developer** | Implementability, DX (developer experience), how easy is it to build on top of this |
| **Designer** | Consumer experience design — how do tools integrate? How do LLMs program against this? Future: customised UIs per user, LLM-driven layout changes at runtime |
| **Sherpa** | Onboarding friction — how quickly can a new developer (or LLM) start using this? What documentation is needed? |
| **AppSec** | Security review of the event model, the registration protocol, the PKI identity system, the command routing |
| **Librarian** | Cross-reference with existing code — what layout code exists today? What will be replaced? |
| **Conductor** | Phasing, prioritisation, resource allocation |

## Four Documents to Produce

### Document 1: Technical Review — Fresh Eyes Analysis

**Lead:** Architect + Developer + AppSec  
**Scope:** Review the sg-layout architecture and implementation plan critically.

Questions to answer:
- Do they agree with the approach? Is it sound?
- Are there any dragons — architectural risks, edge cases, performance concerns?
- Do they agree with the phasing (v0.1.0 through v0.2.0)?
- Is the event model correct (three channels: internal bus, upward bubble, downward injection)?
- Is the synchronous registration protocol robust? What happens in edge cases?
- Security review: is the PKI panel identity model secure? Should private keys be exportable?
- Is the CSS Grid approach viable at scale? (The 32×32 stress test will answer this empirically)

**Format:** Structured review with sections: "Agree", "Concerns", "Suggested changes", "Phasing feedback"

### Document 2: Current State Audit — Where sg-layout Applies

**Lead:** Librarian + Developer  
**Scope:** Audit everything we have today and map where sg-layout would apply.

Questions to answer:
- What layout code exists today in: workspace, vault, send, admin panels, tools?
- Estimate: what percentage of development time is currently spent on layout problems? (Hypothesis: ~20%)
- For each existing UI: how would sg-layout simplify it? Where would it add value?
- Are there any places where sg-layout would make things worse or add unnecessary complexity?
- What is the migration strategy? (The fractal model allows incremental adoption — replace a child component first, then its parent, then the root)
- What problems does sg-layout solve that we've already experienced?
- What problems does it NOT solve?

**Format:** UI-by-UI audit with: current layout approach, pain points, how sg-layout changes it, migration path

### Document 3: Implementation Notes — What the Review Uncovered

**Lead:** Architect + Developer  
**Scope:** Based on Documents 1 and 2, any additional implementation recommendations.

This could be short ("the implementation plan is fine, no changes needed") or substantial ("based on reviewing the vault's layout code, we recommend adding feature X to v0.1.0 because we'll definitely need it").

Questions to answer:
- Any features that should be added to v0.1.0 based on known consumer needs?
- Any simplifications — "this is overengineered for our current scale"?
- Any missing edge cases from real-world usage?
- LLM developer experience: how can we make it maximally easy for an LLM to program layouts? (This is a design question as much as an engineering question)

**Format:** Actionable list of additions, removals, or changes to the implementation plan

### Document 4: Usage Scenarios — "Assume It Exists"

**Lead:** Designer + Developer + Sherpa  
**Scope:** Assume sg-layout is fully implemented. Show how we'd build things with it.

Scenarios to explore:
- **PGP key creation component:** Currently inside the admin panel. How would we rewrite it (a) as an sg-layout hosted component, (b) as a standalone tool?
- **Folder/zip manipulation:** Drag-drop a zip, browse, edit, restructure, re-download. How does this work inside sg-layout?
- **The workspace migration:** The current `workspace-shell.js` (32.5KB) becomes a layout JSON + hosted components. Show the before/after.
- **A tool in the tools.sgraph.ai shell:** How does a tool register, render, and communicate?
- **The 32×32 grid:** The stress test scenario. Show how the builder DSL constructs it.
- **Increasing complexity:** Start simple (two panels side by side), add tabs, add nesting, add drag-to-dock. Show the progression.

**Format:** Code examples (layout JSON + component registration) for each scenario. These become the definition of success for the implementation AND the documentation for future consumers.
