# Vault UI Components and Pack Architecture

**version** v0.13.29
**date** 11 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Architect, Designer
**type** Dev brief

## Context

The Alchemist investor pack milestone (see companion debrief) proved that vaults work as the handover format between AI agents and humans. But the presentation layer is missing. When someone receives a vault, they see raw files. There is no branding, no guided experience, no custom workflow.

The vault needs to become a first-class UI element. Not just a storage backend, but a component that drives the interface.

## The Vault Component

A new Web Component: `<sg-vault>`. This component is the bridge between encrypted storage and UI rendering.

Responsibilities:
- Load vault definition (tree structure, encryption config)
- Handle decrypt-on-demand (request key, decrypt blob, serve to other components)
- Emit events when data is available (file loaded, tree updated)
- Accept write requests from other components (save file, create folder)

The vault component should have no opinions about display. It manages data. Other components request data from it and render it however they need.

### Separation of Concerns

```
<sg-vault>          Data layer. Loads, decrypts, syncs.
<sg-tree-view>      Display layer. Shows file tree. Requests data from vault.
<sg-file-viewer>    Display layer. Renders file content. Requests data from vault.
<sg-pack-theme>     Presentation layer. Custom logos, colours, workflows.
```

The tree view component should have no idea which vault (or how many vaults) it is connected to. It issues a request ("show me the tree for this path") and the vault component responds. Same for saving: the file viewer says "save this content," the vault component encrypts and commits.

This is dependency inversion applied to the UI: display components depend on an abstract data interface, not on vault internals.

## Pack Themes

A pack is a vault with a UI configuration stored inside it. The vault contains:

- The content (files, folders, documents)
- A manifest describing what the pack is (type, target audience, branding)
- UI configuration (which components to load, which visualisers to use, custom logos, colour schemes)
- Workflow definition (what to show first, what actions are available)

Pack types already identified:

| Pack Type | Target | Key Features |
|---|---|---|
| Investor Pack | Investors, VCs | Pitch deck, financials, team bios, data room structure |
| ERM Pack | Relationship management | Contact cards, outreach history, follow-up actions |
| Customer Pack | Early access users, customers | Onboarding guide, access tokens, support contacts |
| Compliance Pack | Auditors, regulators | Policy documents, evidence collection, audit trail |

### Self-Describing Vaults

The vault stores its own UI configuration. When you open a vault, the pack theme component reads the manifest and configures the interface. This means:

- Different vaults render differently based on their content type
- The same viewer component handles all pack types
- New pack types are created by writing a new manifest, not new code
- AI agents can create packs with specific UI configurations

## CRM Through Vaults

Contact management, customer tracking, and outreach history stored in encrypted vaults. Each contact is a file (or folder) in the vault. Interaction history is append-only (commit per interaction). The ERM pack theme provides the UI.

This solves the current problem of managing early access users and customers. Instead of spreadsheets and emails, contacts live in an encrypted vault with structured data, interaction history, and cryptographic audit trail.

## Implementation Priority

1. `<sg-vault>` component (data layer, decrypt-on-demand, event-based API)
2. Pack manifest format (JSON in vault root: `_pack.json`)
3. Investor pack theme (first concrete pack type, already proven useful)
4. ERM pack theme (contact management, outreach tracking)

| 4 | Pack manifest format (`_pack.json`) defined and documented |
