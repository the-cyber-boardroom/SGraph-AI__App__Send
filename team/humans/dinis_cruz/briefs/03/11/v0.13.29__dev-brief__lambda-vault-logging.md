## Context

Lambda function logs are a pain point. CloudWatch/CloudFront logging is unreliable, doesn't scale the way we need, and doesn't give structured capture of invocation data. The vault system now supports commits. The CLI is maturing. The question: can we use vaults to capture Lambda invocation logs with zero-knowledge encryption?

The answer is yes, and the performance characteristics make it viable.

## Headless Vault Mode

A new CLI mode where vaults can be loaded and unloaded without user interaction. The vault exists as a local encrypted blob (a folder of encrypted files). Without the decryption key, the folder is meaningless.

This mode enables:

- Programmatic vault access from services (Lambda, containers, background workers)
- Shared folders where different entities need different decryption keys to access different content
- Fast local read/write against encrypted storage

The key insight: read and write operations are ultimately just S3 calls (get the tree, get/put blobs). Performance should be limited only by S3 latency.

## Lambda Invocation Log Capture

### The Pattern

A Lambda function captures its full invocation context (request, response, timing, errors, debug data) and writes it to an encrypted vault instead of CloudWatch.

### Lambda Identity

Each Lambda function has its own identity (key pair). Log entries are signed by the Lambda function's key, so you know which function produced the entry. Entries can also be encrypted with the SG/Send vault's public key, so only the vault owner can read them.

This is the "encrypt for reader" paradigm applied to infrastructure: the log storage system holds ciphertext it cannot decrypt. Only the intended reader (the ops team, the security team) can open it.

### Two Operating Modes

**Mode 1: Direct vault update (one vault per day)**

The Lambda function loads the vault definition and keys once at boot (cold start). After that, each invocation commits a new entry and pushes. The vault supports commits without merging, so there is no conflict resolution needed. The function takes the current HEAD, commits, and pushes.

Flow per invocation:
1. Capture invocation data (request, response, timing, errors)
2. Commit to local vault (append-only, no merge)
3. Push to remote

Performance question: how fast is commit + push? If it is a single S3 PUT for the new blob plus one S3 PUT for the updated tree, this could be sub-100ms.

**Mode 2: Execution folder (one bundle per invocation)**

Each Lambda invocation creates a self-contained vault bundle with everything from that execution. The bundle is saved to an S3 executions folder. No remote vault interaction during the invocation, just a local encrypt + S3 PUT.

This mode is simpler but produces more objects. Better for high-throughput functions where per-invocation push latency is unacceptable.

### Why Commits Matter

The vault now supports commits, which changes the concurrency model. A Lambda function does not need to merge. It only needs to commit. Take the current HEAD, append a new blob, commit the updated tree, push. If two concurrent invocations both push, the commit history preserves both. No data loss, no conflict resolution.

### Boot Sequence

1. Lambda cold start
2. Load vault definition (tree structure, encryption config) from environment or S3
3. Cache vault state in memory for the lifetime of the execution context
4. Per invocation: commit + push (Mode 1) or bundle + S3 PUT (Mode 2)

The vault definition load only happens once per cold start. Warm invocations skip directly to step 4.

## Connection to Deploy Service

The Deploy Service (v0.13.12 FINAL) already specifies vault-based state storage. Lambda vault logging is the same pattern applied to operational telemetry. The Deploy Service's Router Lambda is the first candidate for this logging approach.

## CLI Changes Required

- `sg-send-cli` needs a headless/programmatic mode (no interactive prompts)
- Support for commit-only operations (no full clone required for append)
- Support for vault bundle creation (Mode 2: single-shot encrypt + package)
- Performance benchmarks for commit + push latency

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | `sg-send-cli` supports headless mode (no interactive prompts, exit codes for automation) |
| 2 | Lambda function can load vault definition once at boot and commit per invocation |
| 3 | Commit + push latency benchmarked (target: sub-200ms for small payloads) |
| 4 | Mode 1 (daily vault) demonstrated with a test Lambda function |
| 5 | Mode 2 (execution bundle) demonstrated with S3 output |
| 6 | Lambda identity: function signs its own log entries with its key pair |
| 7 | Log entries encrypted with reader's public key (not just storage encryption) |
