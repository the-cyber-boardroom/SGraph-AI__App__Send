# Encrypt for the Reader, Not for Storage

**version** v0.13.29
**date** 11 Mar 2026
**from** Human (project lead)
**to** Architect (lead), Developer, AppSec
**type** Architecture brief

---

## The Problem

Most encryption today encrypts for storage. The entity currently holding the data decides the encryption scheme, manages the keys, and controls access. This means every entity in the data pipeline (the application, the database, the log aggregator, the backup system, the crash reporter) becomes a potential source of compromise.

Logs are the canonical example of this failure. Logs sit in a catch-22:

- **Log everything:** full provenance, full debuggability, full audit trail. But secrets, passwords, credit card numbers, customer PII all end up in plaintext on disk. Debug logs written to disk during a crash become the breach vector.
- **Log nothing sensitive:** no provenance, no ability to trace what happened, no forensics capability.

The number of major security incidents caused by secrets appearing in debug logs, crash dumps, or log aggregation pipelines is enormous. The root cause is always the same: the log was encrypted for storage (or not encrypted at all), and too many entities had access.

## The Paradigm Shift

Encrypt for the reader. Encrypt for the entity that is supposed to have access, not the entity that happens to be storing the data.

Instead of "this data is encrypted at rest," the question becomes: "who is this data encrypted FOR?" The encryption target is a specific reader or set of readers, identified by their public keys.

### What This Changes

**Current model (encrypt for storage):**
Every entity that touches the log is a potential source of compromise. Storage system holds the keys. Access control is policy-based (and policies drift, get misconfigured, get bypassed).

**New model (encrypt for reader):**
Data is encrypted with the reader's public key at the point of creation. The storage system holds ciphertext it cannot decrypt. The log aggregator holds ciphertext it cannot decrypt. Only the intended reader (the security team, the on-call engineer, the auditor) can decrypt.

### Multi-Level Encryption

This naturally supports multiple levels of access:

- Encrypt operational metrics with the DevOps team's public key
- Encrypt security events with the AppSec team's public key
- Encrypt PII-containing entries with the compliance team's public key
- Encrypt everything with a master audit key held in escrow

The same log entry can be encrypted multiple times for different readers. Each reader only sees what they're authorised to see, enforced cryptographically, not by access control lists.

## Connection to SG/Send

This is exactly what SG/Send already does for file transfer. The server stores ciphertext. The key lives in the URL fragment (never sent to the server). The server cannot read the data even if subpoenaed or breached.

The vault system extends this: encrypted blobs stored server-side, decryption keys held only by authorised parties. The Lambda logging pattern (see companion dev brief) is the first concrete application of "encrypt for reader" to operational infrastructure.

## Why This Matters

This is not just a technical improvement. It is a different security model. "Encrypt for storage" means the storage provider is trusted. "Encrypt for reader" means the storage provider is untrusted by design. The data is safe even if the storage is compromised, the log aggregator is breached, or the backup system is exposed.

This is the article-worthy concept: the shift from encrypting for where data lives to encrypting for who data is for.
