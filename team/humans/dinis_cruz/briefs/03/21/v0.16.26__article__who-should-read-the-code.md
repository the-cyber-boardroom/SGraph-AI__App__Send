# Who Should Be Reading the Code?

Let me start with a caveat. Before AI agents started writing code, you could argue that 90% of the code running in production was not being read by anyone. Even companies with pull request workflows and code review mandates had a discipline problem. Commits with thousands of lines of changes. Reviews that were rubber stamps. Dependencies pulled in without anyone checking what they contained.

So let us not pretend that "nobody reads the code anymore" is a new problem. It is an old problem, accelerated.

But the question is worth asking properly: in a world where AI agents write significant amounts of code, who should be reading it? And the answer, like most things, depends on which phase the code is in.

## The Explorer Phase: Code as a Brief

In the Explorer phase, code is written fast. It is prototyping. It is experimentation. It is functionality-first, engineering-second. The people writing it might be business owners, non-developers, product people exploring an idea. They might also be experienced developers (like me) who are deliberately not focusing on code quality because the goal is not production code. The goal is learning what works.

In my own workflow, when I am in Explorer mode, I do not really read the code the agents produce. I glance at it. I check if the functionality works. I care about what it does, not how it does it. Because Explorer code is not a product. It is a brief for the next phase. It is a description, in working code, of what we want to build properly.

So who reads Explorer code? Mostly other agents. Agents that do security scans, basic quality checks, dependency audits. These are imperfect reviewers, but they catch the obvious problems early. And that is good enough for this phase, because this code is not going to production. It is going to the Villagers.

## The Villager Phase: Code Gets Read

The transition from Explorer to Villager is where code reading begins in earnest. This is the handover. The Explorer produced working prototypes. The Villager takes that, and either rewrites from scratch or converts it into code that meets engineering standards.

This is where code needs to be read. Not glanced at. Read. Understood. Questioned. Refactored.

The readers at this stage might not be the most senior developers. They might be mid-level engineers, developers who are still building their architectural intuition. And that is fine. This is where they learn. They read the Explorer's output, understand the intent, and rebuild it properly. LLMs assist, but the human reads the code.

This is also where a lot of the real engineering happens:

- Architecture reviews (does this structure make sense at scale?)
- Code reduction (the Explorer over-engineered or duplicated things, simplify)
- Coding practices enforcement (naming, structure, patterns, consistency)
- Refactoring (the Explorer's code works but is tangled, untangle it)
- Visualisation and analysis (use tools to understand the code's shape, dependencies, complexity)

The Villager phase produces code that is cleaner, smaller, better structured, and covered by tests. But it is not yet production-grade.

## The Town Planner Phase: Code Gets Really Read

This is where senior developers, architects, and experienced engineers read the code. And I mean really read it. Print it out and read it with a pen if that is what it takes. Debate it. Argue about it. Question every design decision.

The Town Planners care about things the Villagers did not focus on:

- Performance under load
- Long-term maintainability
- How this code fits into the broader system
- Security at a deep level (not just scans, but architectural security properties)
- What happens when this code runs for two years and the original authors are gone
- Edge cases that only emerge at scale or under adversarial conditions

These are the people with 15, 20, 30 years of experience. They have seen systems fail in ways that junior developers cannot imagine. Their code review is not about style. It is about survival.

## The Feedback Loop: Guidance Flows Back

Here is where it gets interesting. Each phase does not just pass code forward. It passes guidance backward.

When a Villager picks up Explorer code and finds a problem that should have been caught earlier, they do not just fix it. They write guidance: "Explorer agents should check for X before handing off." That guidance goes into the Explorer prompts, the role definitions, the skills. The next Explorer session does not make the same mistake.

When a Town Planner picks up Villager code and finds an architectural issue that should have been resolved in the Villager phase, same thing. They document it. If the issue is something that should always be caught at the Villager level, that is an incident. Why did this reach the Town Planner? What was missing in the Villager's review process? Fix the process, not just the code.

This feedback loop is how the system gets better over time. The Explorers produce better briefs. The Villagers produce cleaner code. The Town Planners spend less time on issues that should have been caught earlier. Each cycle raises the quality floor.

## Environments Match Phases

Each phase pushes to its own environment:

- Explorer code runs in the Explorer environment (experimental, unstable, throw-away)
- Villager code runs in the Villager environment (dev, staging, tested)
- Town Planner code runs in the production-path environment (pre-prod, load-tested, hardened)
- Production is production

This means every phase has its own CI pipeline, its own quality gates, its own definition of "done." The Explorer's "done" is "it works and I learned something." The Town Planner's "done" is "I would bet the company on this code running reliably for the next two years."

## The Junior-to-Senior Path

People worry that if AI writes all the code, there is no path for junior developers to learn. I think the opposite is true.

Junior developers start at the Explorer and Villager phases. They read agent-generated code. They learn by understanding what the agents produced and why. They refactor, simplify, and improve. They write tests. They document. They get feedback from senior reviewers. And through this process, they build the experience and judgment needed to eventually operate at the Town Planner level.

The learning path is not eliminated. It is restructured. Instead of learning by writing simple code from scratch, juniors learn by reading, evaluating, and improving code at scale. The density of learning per hour increases because they are exposed to more code, more patterns, more architectures than they could ever write themselves.

## The Uncomfortable Truth About Abstraction

There is a popular argument that programming always moves toward higher abstractions and that is fine. Nobody writes assembly anymore. Nobody manages memory in most languages. We just keep moving up the stack.

I would argue that this narrative has caused more damage than people admit. Look at the state of modern software. Applications with 2,000 dependencies. CI pipelines that download half the internet. Layers of abstraction where nobody on the team understands what is happening below their layer. We have been abstracting complexity away and then wondering why systems are fragile, bloated, and expensive to maintain.

The answer was never "stop abstracting." The answer was "someone still needs to understand what is underneath." Linus Torvalds reads C code. The best systems have people at every level of the stack, not just at the top.

In the past, having people who could operate at multiple levels of abstraction was expensive and rare. With AI agents handling the volume, it becomes economically viable. The senior developer who used to be too expensive to spend time on low-level code reviews can now review at scale, with agents handling the initial triage.

## The Bottom Line

The answer to "who should read the code" is not "nobody" and not "everybody." It is:

- **Explorers:** agents read it (scans, checks, basic quality). Humans read for functionality, not engineering.
- **Villagers:** mid-level developers read it. They refactor, test, and enforce standards. LLMs assist.
- **Town Planners:** senior developers really read it. They debate, question, and verify at an architectural level.

The more you move toward production, the more carefully the code is read, and the more experienced the readers are. The more you move toward experimentation, the less the code is read for quality and the more it is read for intent.

And every phase writes guidance for the phase before it. The system learns. The code gets better. The path from junior to senior is not "write code from scratch." It is "read, understand, improve, and eventually architect."

This is not slower. It is faster. Because the quality at each phase reduces the rework at the next phase. Because the guidance flowing backward reduces the mistakes flowing forward. Because the environments are separated so that experimental code never accidentally reaches production.

The more we create software, the more we need multiple levels of developers with multiple levels of expertise, operating at multiple phases simultaneously. That is not a problem AI creates. It is a problem AI finally makes economical to solve.
