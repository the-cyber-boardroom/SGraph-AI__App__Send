# SGit Tree Dump, Local-vs-Server Diff, and Read-Structure Key

**date** 21 Mar 2026
**from** Human (project lead)

## The Problem

There is already a bug: two vaults with inconsistent state that cannot push or pull. The merge logic has produced a state where neither side can reconcile. To debug this (and prevent it in the future), we need the ability to dump the full internal state of a vault and compare it against the server.

More broadly, as vault workflows get more complex (branches, merges, multi-user collaboration), we need diagnostic tools that let us see exactly what a vault client thinks it has, what the server thinks it has, and where they diverge.

## Feature 1: Tree Dump (Local)

A command that dumps the complete internal state of a local vault to a JSON file:

```
sgit-ai dump --local > vault-state.json
```

What it contains:

- The traversal path: which file is read first, which file is read next, the full sequence the client follows from the root ref to the leaf blobs
- All refs and what they point to
- All branch metadata (branch IDs, head refs, key IDs)
- All commit objects (parent chain, tree IDs, signatures, timestamps)
- All tree objects (entries, sub-trees, blob references)
- Blob inventory (IDs, sizes, but not content)
- Dangling objects: anything in `bare/data/` that is not referenced by any tree or commit
- The branch index state

This is a complete snapshot of the client's understanding of the vault. No server calls. Just reads the local `bare/` folder and reports everything it finds.

## Feature 2: Tree Dump (Server)

The same dump, but against the server:

```
sgit-ai dump --remote origin > server-state.json
```

This connects to the server, reads the vault's `bare/` structure remotely, and produces the same JSON format. The command does not save anything locally. It does not modify any state. Pure read-only analysis.

This requires the vault decryption key (to decrypt branch metadata, refs, and tree structures). The blobs are listed by ID and size but not decrypted (the dump is structural, not content).

## Feature 3: Diff Local vs Server

```
sgit-ai diff-state vault-state.json server-state.json
```

Or in one step:

```
sgit-ai diff-state --local --remote origin
```

Compares the two dumps and reports:

- Refs that differ (local points to commit A, server points to commit B)
- Objects present locally but missing on server
- Objects present on server but missing locally
- Branch metadata differences
- Dangling objects on either side
- Commit chain divergence (where the histories split)

This is the tool that solves the current bug: dump both sides, diff them, see exactly where the inconsistency is. It is also the tool that prevents future issues: run the diff after every push/pull to verify consistency.

## Feature 4: Read-Structure Key (New Key Type)

Currently, the vault has one key: the vault key that decrypts everything (content + structure). But for analysis and diagnostics, you often do not need to read file contents. You need to read the structure: refs, branches, trees, commit metadata.

Introduce a second key tier:

| Key | What It Decrypts | Use Case |
|---|---|---|
| Vault key (existing) | Everything: structure + file content | Full access (read, write, clone) |
| Structure key (new) | Refs, branches, trees, commits, metadata. Not blob content. | Analysis, diagnostics, monitoring, auditing |

The structure key enables:

- **Diagnostics without content access:** An SRE or QA agent can analyse vault health (dangling objects, ref consistency, branch state) without being able to read the actual files
- **Monitoring:** The SRE dashboard can show vault structure stats (commits, branches, objects) without exposing content
- **Auditing:** An auditor can verify the commit chain and signature integrity without reading the documents
- **Lightweight sharing:** Give someone the structure key so they can see what is in the vault (file names, sizes, history) without reading the content. A table of contents without the book.

### Implementation

The structure key could be derived from the vault key (so anyone with the vault key can always generate the structure key) but not the reverse (having the structure key does not reveal the vault key or the content encryption key).

```
vault_key → HKDF → { read_key (content), structure_key (metadata) }
```

The structure key decrypts tree entries (file names, sizes, sub-tree references) and commit objects (parents, timestamps, messages, signatures) but cannot decrypt blob content.

This is a clean separation that adds a new capability without changing the existing access model. Anyone with the vault key still has full access. The structure key is an additional, more limited credential.

## Connection to Existing Architecture

- **Branch model spec:** The dump and diff tools verify the invariants defined in the spec (named branch never has conflicts, refs point to valid commits, signature chain is intact)
- **SRE agents:** The Town Planner SRE can run vault health checks using the structure key without needing content access
- **Explorer-to-Villager methodology:** The dump/diff tools are diagnostic infrastructure that makes the Villager phase (testing, verification) more effective
- **The current bug:** Immediately solvable with the local dump + server dump + diff workflow

## Acceptance Criteria

| # | Criterion |
|---|---|
| 1 | `sgit-ai dump --local` produces JSON of complete local vault state |
| 2 | Dump includes traversal path (sequence of reads from root to leaves) |
| 3 | Dump identifies dangling objects (not referenced by any tree or commit) |
| 4 | `sgit-ai dump --remote` produces same JSON format from server state |
| 5 | `sgit-ai diff-state` compares two dumps and reports all divergences |
| 6 | Current inconsistent vault bug diagnosed and resolved using dump/diff |
| 7 | Read-structure key derivation specified (from vault key, non-reversible) |
| 8 | Structure key decrypts metadata (trees, commits, refs) but not blob content |
| 9 | Diagnostic dump runnable with structure key only (no content access needed) |
