## The Urgency

We need to be emailing users regularly. Weekly at minimum, more frequently around feature launches. The folder view, gallery with PDF thumbnails, markdown print, video playback: these are features worth announcing. Users with unused credits need encouragement. Early access users need updates.

This is not optional. It is a growth requirement. The workflow needs to be operational now.

## The Email Pipeline: Three Agents, Three Steps

The key architectural decision: the agent that creates the email is not the agent that sends the email. Separation of concerns, reviewability, and security.

### Step 1: Create (Agent 1: Composer)

An agent (or a Claude session) that produces EML files from templates:

- Reads the email template (HTML with templating variables)
- Reads the user list (Stripe customers, early access users)
- Reads per-user profiles (from the vault-based mini CRM)
- Generates personalised EML files (one per recipient)
- Stores EML files in a vault: `outbox/2026-03-21/user-001.eml`

The personalisation is light: a custom paragraph based on the user's profile, their usage (or lack of usage), their market. The LLM generates the paragraph. The template handles the rest.

### Step 2: Review (Agent 2: Reviewer)

A second agent (or the human) reviews the generated emails:

- Opens the vault containing the outbox
- Renders the EML files visually (this becomes the beginning of a mini email client)
- Checks for errors, tone issues, personalisation accuracy
- Approves or flags individual emails
- Moves approved emails from `outbox/` to `approved/`

This review step is critical. No email sends without approval. The reviewer agent can be an LLM checking for quality, or it can be the human scanning the batch.

### Step 3: Send (Agent 3: Sender)

A third agent sends the approved emails:

- Reads from `approved/` in the vault
- Sends each EML via the email service (see infrastructure below)
- Moves sent emails from `approved/` to `sent/`
- Records delivery metadata (timestamp, message ID, any bounce information)

The sender agent only has permission to read from `approved/` and send. It cannot create or modify emails. It cannot access the user list or profiles. Minimum privilege.

## Email Infrastructure: Move from WorkMail to SES

WorkMail worked initially but is messy and hard to manage programmatically. The path:

| Option | Pros | Cons |
|---|---|---|
| AWS SES (Simple Email Service) | Programmatic, scalable, cheap, well-documented API | Send-only (no receiving without extra setup) |
| AWS SNS + SES | Can receive emails (SNS) and send (SES) | More complex setup |
| Keep WorkMail | Can receive and send, has a UI | Messy programmatic access, hard to automate |

**Recommendation:** Move to SES for sending. It is the right tool for programmatic email. If we need to receive emails later (for the mini email client), add SNS or keep WorkMail as a receiver only. But the priority is sending, not receiving.

**Important:** Do not use MCP for this. MCP is not performant enough for batch email operations. Write the sender as direct code (Python calling the SES API via boto3). Clean, fast, predictable.

## Mini CRM in a Vault

The email pipeline needs a user database. Build it as a vault:

```
crm-vault/
  users/
    user-001.json    { name, email, company, stripe_id, credits, last_login, ... }
    user-002.json    ...
  early-access/
    ea-001.json      { name, email, signup_date, notes, ... }
  templates/
    weekly-update.html
    feature-launch.html
    credits-reminder.html
  outbox/
    2026-03-21/
      user-001.eml
      user-002.eml
  approved/
  sent/
    2026-03-14/
      user-001.eml    (with delivery metadata)
```

The vault IS the CRM. No external database. No SaaS tool. The data lives in a vault, encrypted, versioned, accessible by agents.

Populate initially from:
- Stripe customer list (API export)
- Early access programme spreadsheet or list
- Manual additions

## Toward a Mini Email Client

The email pipeline builds the foundation for a broader email client:

1. **Phase 1 (now):** Visualise emails in the vault (render EML files in the browser)
2. **Phase 2:** Create emails in the browser (compose EML, save to vault)
3. **Phase 3:** Send emails from the browser (call SES via the vault pipeline)
4. **Phase 4:** Receive emails (SNS routes inbound to a vault)

This evolves toward the agent-to-agent communication pattern (EML as the message format between agents) and eventually becomes a full encrypted email client built on vault infrastructure.
