# AWS Security Features Research: What Can We Enable Now

**version** v0.16.26
**date** 21 Mar 2026
**from** Human (project lead)
**to** AppSec (lead), DevOps, SRE, Architect
**type** Research brief

## Context

AWS Startup credits ($1,000) are available. Features that were previously skipped due to cost (WAF on CloudFront distributions, GuardDuty, enhanced monitoring) are now viable. The goal: maximum security posture and visibility without introducing heavy databases or complex infrastructure that would not be portable to other cloud providers.

## Research Required

Produce a document that maps out:

1. **What can we enable** across all AWS services we use
2. **What it costs** (monthly estimate, covered by credits)
3. **What changes are needed** (configuration only, or code changes required)
4. **What we get** (what threats are detected, what visibility improves)
5. **How agents can consume it** (can AppSec and SRE agents monitor the output)

## Services to Evaluate

### CloudFront

| Feature | Current State | What It Gives Us |
|---|---|---|
| AWS WAF (Web Application Firewall) | Not enabled (cost was ~$5-20/month per distribution) | Rate limiting, bot detection, SQL injection/XSS blocking, geo-blocking, custom rules |
| CloudFront security headers | Partially configured | HSTS, X-Frame-Options, Content-Security-Policy, X-Content-Type-Options |
| Real-time logs | Not enabled | Per-request logging to Kinesis for real-time analysis |
| Standard access logs | Being enabled (see visibility brief) | Batch logs to S3 for post-hoc analysis |

WAF is the big one. With credits available, enabling WAF on all CloudFront distributions gives us rate limiting, bot management, and geo-blocking out of the box. Map the rule sets available (AWS managed rules, bot control, IP reputation lists).

### Lambda

| Feature | Current State | What It Gives Us |
|---|---|---|
| X-Ray tracing | Being enabled | Request tracing across Lambda, API Gateway, S3 |
| Lambda function URLs (auth) | Some functions public | Review which functions should have IAM auth vs. public access |
| Reserved/provisioned concurrency | Not configured | Protect against Lambda exhaustion attacks |
| Code signing | Not enabled | Verify Lambda code has not been tampered with |

### S3

| Feature | Current State | What It Gives Us |
|---|---|---|
| S3 Block Public Access | Verify enabled | Prevent accidental public exposure of buckets |
| S3 Object Lock | Not enabled | Prevent deletion of critical data (compliance) |
| S3 versioning | Verify state | Recover from accidental or malicious deletions |
| S3 server access logging | Being enabled | Track every read/write |

### Account-Level Security

| Service | What It Does | Priority |
|---|---|---|
| AWS GuardDuty | Threat detection: unusual API calls, compromised credentials, cryptocurrency mining, reconnaissance | High |
| AWS Security Hub | Aggregates findings from GuardDuty, Inspector, Macie, IAM Access Analyzer | High |
| AWS CloudTrail | API call logging for all AWS actions (who did what, when) | High (likely already on) |
| AWS Config | Track configuration changes across all resources | Medium |
| IAM Access Analyzer | Find resources shared with external entities | Medium |
| AWS Inspector | Vulnerability scanning for Lambda functions and EC2 | Medium |
| AWS Macie | Discover and protect sensitive data in S3 | Low (our data is encrypted, Macie cannot read it) |

GuardDuty is the highest-value enablement. It detects compromised credentials, unusual API patterns, and reconnaissance, which are the real threats at this stage.

### Elastic / OpenSearch

If using AWS Managed OpenSearch for log aggregation (CloudFront logs, Lambda logs, application logs):

- What is the minimum viable cluster size?
- Can we use OpenSearch Serverless (no cluster management)?
- How do we feed CloudFront and Lambda logs into it?
- What pre-built dashboards and detection rules are available?
- What is the monthly cost estimate?

## Architecture Constraints

**Keep it light.** No major databases. No persistent clusters if avoidable (prefer serverless options). The infrastructure should be reproducible on other cloud providers. Avoid deep AWS-specific lock-in for core functionality. Use AWS-specific services for monitoring and security (these are naturally provider-specific) but keep the application architecture portable.

**Agent-consumable output.** Every security service we enable should produce output that the AppSec and SRE agents can consume. GuardDuty findings as JSON. WAF logs. CloudTrail events. The agents should be able to clone a vault containing the day's security findings and produce a daily security report.

## Deliverable

A research document with:

- Table of all services evaluated, current state, recommended action, cost estimate
- Prioritised enablement plan (what to turn on first)
- Configuration steps for the top-priority items
- Monthly cost projection against the $1,000 credit budget
- Monitoring integration: how each service feeds into Grafana/Elastic/agent workflows
