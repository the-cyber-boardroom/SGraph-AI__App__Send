# SGit on Top of Git: Patch-Based Agent Collaboration

**version** v0.16.26
**date** 21 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Architect
**type** Dev brief

## Positioning: Not Replacing Git

SGit is not competing against Git. SGit works on top of Git, addressing specific workflows where Git alone falls short. The sweet spot is agentic collaboration: the problem of giving an agent controlled, scoped access to a codebase without handing over full repo credentials.

## The Problem (Repeated Because It Is Real)

Right now, to have an agent work on a Git repo:

- Give it full access to the repo (overpowered, risky)
- Or give it a zip and manually apply changes back (tedious, no provenance)

What we want: the agent works on a specific part of the codebase, produces patches with full provenance, and communicates the changes through a vault. The human reviews and applies. The agent never has write access to the repo.

## The Patch-Based Workflow

### Step by Step

```
1. Human says: "Clone the tools repo, work on the new gallery component"

2. Agent clones the public repo (or gets it from PyPI, or receives a zip)
   - Agent only needs read access to the repo
   - For PyPI packages: source code is included, no Git clone needed
     (though tests may be missing)

3. Agent reads the brief pack from a designated folder:
   humans/dinis-cruz/briefs/03/21/
   - The brief describes what to build
   - Reference files, design specs, acceptance criteria

4. Agent makes changes locally
   - Only in the scoped folder (e.g. src/components/gallery/)
   - Cannot touch other parts of the codebase

5. Agent generates a Git patch:
   git diff > changes.patch
   - The patch captures exactly what changed, nothing more

6. Agent commits to the vault:
   - The brief (what was asked)
   - The documentation (what was decided)
   - The patch (what was changed)
   - Any notes or findings

7. Agent shares the vault key with the human

8. Human pulls the vault:
   - Reads the brief, documentation, notes
   - Reviews the patch
   - Applies: git apply changes.patch
   - Commits to the real repo on the appropriate branch
```

### What the Vault Contains

```
vault/
  brief.md              What was asked
  documentation.md      What was decided, architecture notes
  findings.md           Issues found, questions raised
  changes.patch         The Git patch (the actual code changes)
  metadata.json         Repo, branch, base commit, timestamp
```

This is not just a code delivery mechanism. It is a complete record of the work: what was asked, what was thought, what was built, and the exact diff. Other agents can review the vault before the patch is applied. A QA agent can read the patch and check for issues. An AppSec agent can scan for vulnerabilities. All without any code change having been made to the actual repo.

## Why Patches Are the Right Primitive

Patches are Git's native communication format. They are:

- **Small:** Only the diff, not the full files
- **Portable:** A text file that can be emailed, vaulted, or applied anywhere
- **Reviewable:** Human-readable diff format
- **Composable:** Multiple patches from multiple agents can be stacked
- **Verifiable:** Apply to a known base commit, produces predictable results

Using patches means the vault never needs to contain the full codebase. It only contains the brief and the delta. This is lightweight and secure: even if the vault is compromised, only the patch is exposed, not the entire repo.

## Scoped Agent Access

The agent should only be able to work on a specific part of the codebase. In this workflow:

- The agent clones the full repo (read-only, it is public)
- The brief tells it to work in `src/components/gallery/`
- The agent makes changes only in that folder
- The patch only contains changes to that folder
- If the agent touched files outside the scope, the human sees it in the patch review and rejects

This is convention-enforced for now (the agent is told to stay in scope). In the future, SGit could enforce this cryptographically: the agent's key only grants write access to specific vault paths.

## Vault as Zip in the Repo

An interesting option: commit the vault (as a zip) into the Git repo alongside the code. This means:

- The work record (brief + docs + patch) lives in the same repo as the code
- Other agents or humans can find and review past work by browsing the vault zips
- The vault zip in the repo is encrypted, so only authorised reviewers can read it

This is not required for the workflow to function, but it provides provenance within the repo itself: "here is the encrypted vault that produced this commit."

## Features to Add to SGit

A few additions that make this workflow smoother:

| Feature | What It Does |
|---|---|
| `sgit-ai diff` | Generate a patch from local changes against the base |
| `sgit-ai apply` | Apply a patch from a vault to a local Git repo |
| `sgit-ai vault-from-zip` | Operate on a vault stored as a zip file (no server needed) |
| Vault scoping | Restrict which paths an agent's key can write to |
| Patch verification | Verify a patch applies cleanly to a specified base commit before accepting |

## What This Enables

- Agents work on code without repo write access
- Multiple agents work on different components in parallel, each producing patches in their own vaults
- Patches are reviewed before any code change hits the repo
- Full provenance: every code change has a brief, documentation, and a reviewable patch
- The workflow works with public repos today (no infrastructure changes needed)
- The same workflow works for private repos if the agent receives a zip instead of cloning
