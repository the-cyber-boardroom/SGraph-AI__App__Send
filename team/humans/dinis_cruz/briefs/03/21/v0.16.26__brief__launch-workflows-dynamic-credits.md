## Part 1: Workflow-by-Workflow Documentation

### The Principle

Every documentation page should answer one question: "why would you use SG/Send for this?" Not what the feature is. What it makes better. Each page shows a specific workflow that is currently painful or impossible, and demonstrates how SG/Send solves it.

### Workflows to Document

Each of these is a real, concrete workflow that exists today and is harder than it should be:

**Share a folder and browse it in the browser**
Today: zip the folder, email it or upload to WeTransfer, recipient downloads, unzips, clicks through files. With SG/Send: drop the folder, share the link, recipient browses the folder structure in the browser. No download needed.

**Gallery preview of mixed file types**
Today: send a folder of PDFs, images, and documents. Recipient downloads everything to see what is inside. With SG/Send: thumbnails for every file type (PDF first page, markdown rendered, images resized). Recipient sees everything at a glance without downloading.

**Print a markdown file as PDF**
Today: install Puppeteer, run a headless Chrome, render the markdown, capture as PDF. Or find a VS Code extension that may or may not work. With SG/Send: open the markdown in the viewer, click print. The print-friendly view is already rendered. This sounds trivially simple, but there is genuinely no easy way to do it right now.

**Send a video with a playable link**
Today: upload to YouTube (public), or send via WhatsApp (compressed), or email (size limit), or WeTransfer (recipient must download before playing). With SG/Send: share a link, recipient opens it, video plays in the browser. No download. No account. No compression.

**Share a document with an expiry**
Today: send via email (lives forever in their inbox), or shared drive (have to remember to revoke access). With SG/Send: set expiry to 1/3/7/30 days. File is automatically deleted. No manual cleanup.

**Send something that can only be opened once**
Today: not really possible without specialised tools. With SG/Send: set download limit to 1. After one download, the file is gone.

**Share files with someone in a different language**
Today: the UI is in your language. The recipient sees your language. With SG/Send: the UI renders in the recipient's language automatically. Gallery comments can be pre-translated.

**Collaborate on documents without giving full access**
Today: shared Google Drive (everyone sees everything), or email attachments (no version control). With SG/Send: vault with branches. Each collaborator works on their branch. Merge when ready. PKI per user so different people see different things.

### Format

Each workflow page should be:

- One page, one workflow
- Title: the problem ("How do I share a folder someone can browse?")
- Three sections: "Today" (the pain), "With SG/Send" (the solution), "Try it now" (live demo with public credentials)
- Every page links to the interactive Sherpa demo for that workflow

These pages are marketing material, documentation, and conversion funnels simultaneously. Every "Try it now" link puts the user into the product.

## Part 2: Dynamic Credit Allocation Agent

### The Idea

An agent (or a logic-driven service) that manages the free tier dynamically. Instead of a fixed "5 credits per fingerprint per day," the allocation adjusts based on current conditions.

### Inputs

| Factor | Effect on Credits |
|---|---|
| Current traffic volume | Low traffic → more generous (e.g. 20-50 credits). High traffic → more conservative (e.g. 2-5 credits). |
| User behaviour | Normal usage → standard allocation. Suspicious patterns → reduced or zero credits. |
| Geographic region | Target markets (UK, PT, EU) → generous. Non-target regions → standard or reduced. |
| Time of day | Off-peak → more generous. Peak → standard. |
| Repeat visits | First visit → welcome bonus. Frequent visitor who never converts → reduced. |

### Architecture: State Machine, Not Inline LLM

The credit allocation should not have an LLM making real-time decisions on every request. That is too slow, too expensive, and too unpredictable. Instead:

1. **LLM designs the rules** (offline, in a briefing session): "When traffic is below X, give Y credits. When user fingerprint matches pattern Z, reduce to W."
2. **Rules are codified as a state machine** (JSON config or simple logic in Lambda)
3. **State machine executes on every request** (fast, deterministic, cheap)
4. **LLM reviews the results periodically** (daily: "here is today's allocation data, are the rules working? Should we adjust?")

The LLM is never inline. It designs the rules and reviews the outcomes. The state machine executes. This is the "agentic workflow done safely" principle: the agent codifies decisions, it does not make them in real time on user-facing traffic.

### The Agent

A new agent role, or an extension of the Sherpa: the **Gatekeeper**. Responsibilities:

- Monitor free tier usage patterns
- Adjust credit allocation rules based on traffic and abuse signals
- Report on conversion (how many free-tier users become paying users)
- Flag anomalous usage for AppSec review
- Publish daily allocation stats to the SRE dashboard

### Display to User

When a free-tier user visits, show them clearly:

- "You have X free credits today"
- "Credits refresh daily"
- "Want more? Buy credits for 5 GBP"

Transparent. No surprises. The user knows what they get and how to get more.
