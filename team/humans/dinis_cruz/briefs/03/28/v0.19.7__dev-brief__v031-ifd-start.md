# v0.3.1 IFD Start: Dependency Tracking and Version Compatibility

**date** 28 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Architect, QA, DevOps
**type** Dev brief

## Moving to v0.3.1

The v0.3.0 UI is live and stable. Time to start the next IFD cycle on v0.3.1. This is the test of whether the new IFD v2 architecture (manifest-driven, file-level replacement) actually makes iteration easier than the v0.2.x overlay approach.

**v0.3.0 must not be touched.** It is the stable baseline. All changes happen in v0.3.1. If we need to deploy just v0.3.0 (rollback, separate environment), we need to know exactly what it depends on.

## Dependency Tracking: What v0.3.0 Needs

A critical gap: we do not currently track which backend version v0.3.0 is compatible with. As we make changes to the backend (new methods, changed APIs, new features), the UI and backend can drift out of sync.

### What to Capture

Create a compatibility file (markdown + JSON) in the UI repo:

```json
{
  "ui_version": "0.3.0",
  "backend_compatibility": {
    "min_version": "0.16.38",
    "tested_with": "0.19.7",
    "commit": "abc123def456"
  },
  "dependencies": {
    "sg-layout": "0.1.0",
    "sg-send-cli": "0.16.45"
  },
  "deploy_notes": "Requires Lambda function version >= 0.16.38. CloudFront distribution must serve v0.3.0 from /v0/v0.3/v0.3.0/"
}
```

This is the beginning of an SBOM (Software Bill of Materials) for the UI. If someone needs to deploy v0.3.0 offline, or in an isolated environment, this file tells them exactly what they need.

### Markdown Reference

In addition to the JSON (machine-readable), maintain a markdown file (human-readable):

```markdown
# v0.3.0 Compatibility

## Backend
- Minimum: v0.16.38
- Tested with: v0.19.7
- Commit: abc123def456

## External Dependencies
- sg-layout v0.1.0 (CDN or vendored)
- sg-send-cli v0.16.45

## Deploy Requirements
- Lambda function >= v0.16.38
- CloudFront distribution serving from /v0/v0.3/v0.3.0/
- S3 bucket with v0.3.0 static files
```

## v0.3.1 Development Flow

### The IFD v2 Test

v0.3.1 is where we validate the manifest-driven architecture:

1. Start from v0.3.0 as the base
2. Create the v0.3.1 manifest listing which files are replaced
3. Only the changed files exist in the v0.3.1 directory
4. The loader resolves v0.3.1 manifest against v0.3.0 base
5. Deploy v0.3.1 by updating the manifest reference

If this works smoothly, the IFD v2 architecture is validated. If it has friction points, we learn and adjust before v0.3.2.

### Release Cadence Target

The goal: comfortable releases every couple of days, or every week, to production. Not monthly. Not quarterly. The QA infrastructure (smoke tests, deployment gate, automated screenshots) makes this safe. The IFD architecture makes this manageable.

Every release should be:
- Small (a few changed files, bounded blast radius)
- Tested (QA smoke tests pass on main)
- Documented (release notes auto-generated from issue files and test artifacts)
- Reversible (switch the manifest back to v0.3.0 if needed)

### Version Tag Before Docs

Adjust the CI pipeline so that the version number increment happens before the documentation generation step. This ensures generated QA pages reference the correct version. Currently there may be a mismatch where docs are generated before the version tag is created.
