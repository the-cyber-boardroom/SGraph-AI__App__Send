## Context

The tools team is hitting a good stride (the video editing tool is working). Time to add client-side LLM tools to the toolkit. We already have the code from the workspaces project that integrated with Ollama and OpenRouter. Refactor that into a reusable component and build new tools on top of it.

## Component: Client-Side LLM Integration

### Refactor from Workspaces

The existing workspaces code already has:
- OpenRouter integration (multiple model access via one API)
- Ollama integration (local model execution)
- Chat interface basics

Extract this into a standalone component (`sg-llm` or similar) that any tool can use:

```html
<sg-llm
  provider="openrouter"
  api-key="user-provided"
  model="claude-sonnet-4-20250514"
  mode="one-shot"
  on-response="handleResponse"
>
</sg-llm>
```

### The One-Shot Token Pattern

This is important: do not build multi-turn conversation history. The tools should use a one-shot pattern:

- User provides input (text, a file, a question)
- The LLM processes it in a single request
- The LLM returns a result
- Done. No history. No context window management.

This is simpler, cheaper, and more predictable. Each interaction is independent. The user can run it again with different input. There is no state to manage between calls.

For tools that need context (e.g. "analyse this vault"), the context is assembled per-request from the vault contents, not from conversation history.

### Provider Options

| Provider | How It Works | Use Case |
|---|---|---|
| OpenRouter | API call to OpenRouter, routes to chosen model | Default, works everywhere |
| Direct API | API call to Anthropic/OpenAI directly | Users with their own API keys |
| Ollama | Local model execution, no network | Privacy-sensitive, offline use |
| Bring your own key | User enters their API key in the tool | Try before you buy |

All providers use the same component interface. The tool does not know or care which provider is being used.

## Tool 1: Simple LLM Chat (MVP)

The simplest possible tool: a text input, a submit button, a response area. One-shot. Type a question, get an answer. No history.

This serves as:
- The test bed for the sg-llm component
- A demonstration that client-side LLM calls work
- The foundation for more complex tools

## Tool 2: NanoBanana Infographics Integration

NanoBanana (from NotebookLM) creates infographics from content. This is an underexplored capability that even technically sophisticated users have not fully grasped.

### The Opportunity

If we can replicate the infographic generation workflow inside an SG/Send tool:
- Users upload content (documents, data, notes)
- The tool generates an infographic (via LLM + templating)
- The user downloads or shares the infographic
- Traffic driver: people share infographics, infographics carry the brand

### Exploration Steps

1. Research how NanoBanana generates infographics (API, prompts, templates)
2. Build an MVP that takes text input and produces a visual summary
3. Integrate with the sg-llm component for the LLM processing
4. Output as an image (SVG or PNG) or an HTML page

This does not need to be perfect. An MVP that produces reasonable infographics from text input is enough to test the market response.

## Tool 3: LLM on Vault Contents (Future)

The long-term vision: open a vault in the browser, and use a client-side LLM to analyse, summarise, or transform the contents. Everything happens in the browser. The vault contents are decrypted client-side. The LLM processes them client-side (or via an API call where only the user has the key). The server never sees the unencrypted data.

Examples:
- "Summarise this folder of documents"
- "Find inconsistencies between these reports"
- "Create a cover letter for this set of tax documents" (the accountant use case)
- "Generate a timeline from these medical records" (the doctor use case)

This is the convergence of zero-knowledge encryption and client-side AI. The data stays private. The AI adds value. The server stays blind.

Not for immediate implementation. But the sg-llm component and the one-shot token pattern are the foundation.

## Revenue Angle
