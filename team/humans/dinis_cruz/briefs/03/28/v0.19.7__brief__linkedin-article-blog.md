## Part 1: The Article - Introducing SG/Send

The product is live. The LinkedIn newsletter has a couple of thousand subscribers. It is time to formally introduce SG/Send.

### What the Article Should Cover

Write in my voice (as with all previous articles). The article should:

- Introduce SG/Send: what it is, why it exists, what problem it solves
- Explain the key features (zero-knowledge encryption, gallery view, folder view, friendly tokens, client-side encryption, zero cookies, zero tracking)
- Show the new v0.3.0 UI (screenshots needed, listed below)
- Explain what makes it different from WeTransfer, Dropbox, etc. (not just more secure, but expanding the category: vaults, version control, agentic collaboration)
- Mention the open source angle
- Include a "try it now" call to action with a shared token

### Screenshots Needed

The article needs visual evidence. List of screenshots to create:

1. The upload page (step 1, file drop zone, "Zero cookies, Zero tracking, We cannot read your files")
2. A file being uploaded (progress indicator, encryption status)
3. The share step (friendly token displayed, copy link)
4. The gallery view (thumbnails of mixed file types)
5. The folder/browse view (tree navigation with preview panel)
6. The download page (decryption in progress or completed)
7. A markdown file rendered in the viewer
8. The print-friendly PDF output from markdown

The project lead will provide these screenshots via a vault. The Ambassador or Alchemist can then fine-tune the article in a separate session using the screenshots.

### Workflow for Creating the Article

1. This Claude session writes the first draft (the briefing team has full context of the product)
2. The project lead creates a vault with the required screenshots
3. A new Claude session (Ambassador or Alchemist) receives the draft + vault with screenshots
4. That session fine-tunes the article, placing screenshots, adjusting copy
5. The project lead reviews and publishes

## Part 2: LinkedIn Publishing Workflow Tool

A recurring pain point: getting formatted content with images from a Claude session into a LinkedIn article. This needs a tool.

### The Brief for a Separate Session

Create a mini UI (React component or HTML page) that:

1. Accepts markdown text + images
2. Renders them in a layout that matches LinkedIn's article format
3. Allows the user to copy and paste the rendered output directly into LinkedIn's editor
4. Preserves formatting (headings, bold, images inline, etc.)

Explore whether this can be built as an artifact in Claude's React component system. If so, the workflow becomes: paste the article text, upload the images, arrange them, copy to LinkedIn.

Additionally explore: can Claude Code read content pasted into this UI? This would enable an editing loop where the project lead pastes content, Claude reads it, suggests improvements, the project lead pastes the updated version.

## Part 3: Blog Section on the Website

The website needs a news/blog section. Every article, LinkedIn post, and announcement should be published there as well.

### What Goes in the Blog

- LinkedIn articles (like this one and the 6 already written)
- Product announcements (new features, version releases)
- Case studies (accountant, doctor, artist portfolios)
- Technical deep dives (how encryption works, how IFD works)
- Related LinkedIn posts (any post that mentions SG/Send)

### Encrypted Content Experiment

An interesting experiment: store blog content as encrypted vaults. The blog page contains the vault ID or simple token. The decryption happens client-side when the reader opens the page. The server only stores encrypted blobs.

This means:
- Blog content is zero-knowledge (the server cannot read the blog)
- The same infrastructure used for file sharing serves content delivery
- Performance needs testing (is client-side decryption fast enough for a blog reading experience?)

For now, this is an experiment. If performance is acceptable, it becomes a feature. If not, blog content is stored as regular static files and the experiment informs future architecture.

### Publishing Workflow

Every time content is created for external publishing:

1. Write the content (in a Claude session, with vault collaboration)
2. Publish to the blog section on the website
3. Publish to LinkedIn
4. Store the source materials in a vault (for historical record)

The blog becomes the canonical location. LinkedIn is the distribution channel. The vault is the archive.
