# In-Browser Zip and Folder Editing: The Tree View as an Editor

**version** v0.19.7
**date** 28 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Architect, Designer
**type** Dev brief

---

## The Use Case

A recurring workflow: create a folder or zip, realise something needs changing (a file renamed, a file added, something moved, a description updated), and then having to recreate the entire transfer just to make a small edit.

The real workflow should be: open the folder in the browser, make changes, and either save it back or share it as a new transfer. No round-trip through the local file system. No re-uploading everything.

## What the Tree View Editor Enables

If the sg-tree component (from the core components brief) natively supports editing operations, the folder view becomes an editor:

| Operation | What It Does |
|---|---|
| Rename | Click a file/folder name, type new name |
| Move | Drag a file from one folder to another |
| Delete | Select and delete files or folders |
| Add | Drop new files into the tree (from desktop or paste) |
| Reorder | Drag to change display order within a folder |
| Create folder | Right-click or button to create a new subfolder |

These operations modify the in-memory zip/folder structure. Nothing goes to the server until the user explicitly saves or shares.

## Two Modes

### Mode 1: Edit Before Sharing (Preparation)

The user is building a folder to share. They add files, organise them, rename things, maybe add a README. This is the curation phase. When they are happy, they click Share and the folder is encrypted and uploaded.

This is the workflow from the accountant/doctor use case brief: prepare the materials (possibly with LLM assistance), organise them, then share.

### Mode 2: Edit After Receiving (Collaboration)

Someone receives a shared folder. They open it in the browser. They want to make changes: add their own files, annotate something, reorganise. They make the edits in the browser, click Share, and a new transfer is created with their modifications.

This is lightweight collaboration without a vault. The recipient does not need Git, does not need SGit, does not need any tooling. They just edit in the browser and share again.

The vault version of this (with version control, branches, commits) comes later. The folder editing version is the simple, accessible entry point.

## The Standalone Zip Editor Tool

Build this as a standalone tool on tools.sgraph.ai:

1. Drop a zip file into the tool
2. See the contents as a tree view
3. Rename, move, delete, add files
4. Download the modified zip

No server. No SG/Send account. No encryption. Just a zip file editor in the browser. This is a mini tool that drives traffic to tools.sgraph.ai and demonstrates the tree view component.

The SG/Send integration adds encryption and sharing on top. But the zip editing capability is valuable independently.

## Multi-Tree Operations

If two tree views are displayed side by side (e.g. two different folders, or a folder and a vault), enable drag-and-drop between them:

- Drag a file from folder A to folder B
- Drag a file from a received transfer into a new transfer being prepared
- Drag files from a vault into a folder for export

This is the file manager experience, in the browser, with encryption underneath.

## Connection to the Vault Workflow

The vault CLI already supports export. The tree view editor completes the visual side:

```
Vault (version controlled, branches, commits)
  ↕ export/import
Folder (tree view, editable in browser)
  ↕ share
Transfer (encrypted, shared via link)
```

Users who need simple sharing use folders. Users who need collaboration use vaults. The tree view editor serves both, with the same component.
