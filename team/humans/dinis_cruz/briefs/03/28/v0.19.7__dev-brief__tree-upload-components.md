# Core Components: Tree View and Upload/Download as Standalone, Battle-Hardened Primitives

**version** v0.19.7
**date** 28 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Architect, QA, Designer
**type** Dev brief

## Context

Now that v0.3.0 is live and getting good reviews, two components need the same treatment that sg-layout received: careful architecture, multiple variations, IFD version control, rock-solid testing, and deployment as standalone tools that can be used anywhere.

## Component 1: Tree View (sg-tree)

### What It Is

A generic tree view component that renders hierarchical data: folders and files, organisational charts, dependency graphs, ontology visualisations, or any node-and-edge structure.

The folder view in SG/Send's browse page is a tree view. But it is currently embedded in `send-browse.js`, not a standalone component. It needs extracting, generalising, and hardening.

### Why "Tree View" Not "Folder View"

A folder view is just one presentation of a tree. By building it as a generic tree component:

- Folder view: nodes are folders and files, icons are file types
- Ontology view: nodes are entities, edges are relationships
- Dependency view: nodes are components, edges are dependencies
- Organisation view: nodes are people/roles, edges are reporting lines

Same component, different data, different icons. The tree structure, expand/collapse, selection, navigation, and keyboard shortcuts are shared across all presentations.

### Architecture

Follow the sg-layout pattern:

1. **Map what already exists.** The current tree rendering in `send-browse.js` (lines, folder structure, file selection, expand/collapse). Extract this into a standalone component.

2. **Build as a tools.sgraph.ai component.** Full IFD version control. Isolated development. Own test suite.

3. **Design the API:**

```html
<sg-tree
  data="[{id, label, children, icon, type}]"
  selectable="true"
  expandable="true"
  keyboard="true"
  on-select="handleSelect"
>
</sg-tree>
```

4. **Customisation via properties:**
   - Icon set (file type icons, org chart icons, custom)
   - Selection mode (single, multi, none)
   - Expand/collapse behaviour (click, double-click, arrow keys)
   - Rendering (indentation depth, connector lines, badges)

5. **Keyboard navigation:** Arrow keys for tree navigation, Enter to select, j/k for vim-style movement (the keyboard nav that was missing in v0.3.0 and added back in the refactoring).

### Where It Gets Used

| Application | Data |
|---|---|
| SG/Send browse view | Folder and file tree from uploaded zip |
| Vault explorer | Vault tree structure (commits, branches, blobs) |
| Ontology visualiser | Entity and relationship graph (from the Ontologist) |
| Component dependency view | IFD version tree |
| QA test tree | Test suite hierarchy |

## Component 2: Upload/Download Engine (sg-transfer)

### What It Is

A standalone, battle-hardened upload and download component that handles all the complexity of transferring files over the internet: chunking, parallel threads, resume, retry, speed measurement, encryption, progress reporting.

This was detailed in the robust upload/download brief from 26 March. Now it needs to be built as a proper component.

### Architecture

Two sub-components:

**sg-upload:** Handles file selection, chunking, encryption, parallel upload, progress tracking, error recovery.

**sg-download:** Handles fetching, parallel chunk download, decryption, assembly, progress tracking, error recovery.

Both share:
- A transfer engine (chunk management, parallel threads, retry logic)
- A speed measurement module (real-time bandwidth estimation)
- A progress reporter (events for UI to consume)
- An error handler (categorised errors with recovery strategies)

### Distributable as a Library

This component should be usable outside SG/Send. Someone building an encrypted file transfer into their own product should be able to:

```bash
npm install @sgraph/transfer
```

```javascript
import { SGUpload, SGDownload } from '@sgraph/transfer';

const upload = new SGUpload({
  endpoint: 'https://api.example.com/upload',
  chunkSize: 1024 * 1024,  // 1MB chunks
  parallel: 4,
  encrypt: true,
  onProgress: (p) => console.log(`${p.percent}%`),
  onError: (e) => console.log(e.category, e.recoverable),
});

await upload.send(file);
```

Wide adoption finds corner cases. The more people use the component, the more robust it becomes for SG/Send.

### QA for Both Components

Both sg-tree and sg-transfer need the full QA treatment:

- Page object classes in the Playwright automation
- Schema models for their state
- Screenshot-based documentation on the QA site
- Regression tests for every feature and failure mode
- Component-level QA site (auto-generated documentation from tests, same pattern as the main QA site)

For sg-transfer specifically, the QA environment needs to simulate:
- Network dropouts (Playwright can throttle/disconnect)
- Slow connections (bandwidth throttling)
- Large files (multi-GB test files)
- Concurrent transfers
- Browser tab close/reopen (resume testing)

## Implementation Plan

Both components go through the same path:

1. **Map what exists** (extract from current codebase)
2. **Create the tools.sgraph.ai project** (IFD versioned, isolated)
3. **Build the core** (API design, basic functionality)
4. **Add variations** (tree: folder/ontology/dependency views; transfer: upload/download/resume modes)
5. **Full QA cycle** (page objects, schemas, screenshots, regression tests)
6. **Documentation** (auto-generated from QA, published on the QA site)
7. **Integration** (replace the current embedded code in SG/Send with the standalone component)
