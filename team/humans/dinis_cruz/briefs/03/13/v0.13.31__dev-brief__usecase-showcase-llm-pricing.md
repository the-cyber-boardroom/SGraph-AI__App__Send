## The Page

The main SG/Send website gets a "Try It Now" section. Not a features page. Not a pricing page. A page where the user picks a use case and runs it.

Each use case is a one-shot LLM prompt with a UI wrapper. The user provides input, the LLM processes it, the output is delivered via vault or download. One request, one result. This keeps costs low and demonstrates the one-shot prompt pattern (the same pattern described in the published "One-Shot LLM" article).

## Bring Your Own Token

Two modes on every use case page:

**Mode 1: Use our credits.** User enters their SG/Send access token. Credits are consumed. We pay the LLM provider at wholesale, charge the user at markup. Simple.

**Mode 2: Bring your own token.** User enters their own OpenRouter/OpenAI/Gemini API key. No markup. No SG/Send credits consumed for the LLM call. They still pay credits for any vault/send operations. This mode removes the barrier for users who already have API access and want to try the workflow without buying credits first.

Both modes use the same UI. The token input field just has a toggle: "Use SG/Send credits" or "Use my own API key."

## LLM Pricing

### The Markup

25% markup on LLM costs that flow through SG/Send credits.

The justification: we handle provisioning, key management, model selection, and the encrypted delivery pipeline. 25% is reasonable when individual operations are cheap. If a document analysis costs 12p at wholesale, the user pays 15p. The absolute cost is low enough that the markup does not feel significant.

### Making the Cost Feel Right

The key is showing that most LLM calls are very cheap. Frame it in terms of outcomes, not per-token pricing:

- "With 3 GBP of credits, you can generate approximately 10-20 document analyses"
- "Image generation: 15-50p per image depending on model"
- "Startup review: approximately 30-60p per review"

When the user sees "this review cost 40p" after running it, 25% markup on that is 8p. Nobody notices.

### Updated Pricing Page

The pricing page needs to show both existing and new billable units:

| Category | Unit | Pricing |
|---|---|---|
| File transfer | Upload/download | Existing credit model |
| Storage | Per GB | Existing credit model |
| Vault operations | Pulls, pushes, commits, branches | New (from vault billing brief) |
| LLM operations | Per call, model-dependent | Wholesale + 25% markup |

The "No Surprises" principle applies: all costs visible before the user confirms. The cost preview on every use case page shows exactly what the operation will cost before they click generate.

## One-Shot Prompt Pattern

All use case demos use one-shot prompts. Not conversations. Not multi-turn. One request in, one result out.

This is deliberate:

- **Cost control.** One API call per use case. No runaway conversations.
- **Simplicity.** The user does not need to learn how to prompt. They fill in a form. The prompt is pre-built.
- **Demonstration of the principle.** The "One-Shot LLM" article argued that curated context plus a single well-crafted prompt beats open-ended conversation. Every use case page is a live proof of that argument.
- **Cacheability.** Similar requests can hit prompt caches, further reducing cost.

The one-shot pattern also means the page can show the full prompt to the user if they want to see it ("Show what we sent to the LLM"). Transparency. No magic.
