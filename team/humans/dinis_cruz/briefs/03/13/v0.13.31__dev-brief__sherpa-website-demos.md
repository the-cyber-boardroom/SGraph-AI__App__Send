# Sherpa Website: Interactive Self-Contained Demo Pages

**version** v0.13.31
**date** 13 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Sherpa (lead), Designer, Ambassador
**type** Dev brief

## The Problem

There are users right now who need a page they can be pointed to. Not documentation. Not a "read about it" page. A page where they experience the technology by using it, in the browser, on that page, right there.

The current website explains what SG/Send is. It needs to shift from "here's what it is" to "here's how you use it, try it now."

## The Sherpa Website

Each agent role owns a website:

| Role | Website | Purpose |
|---|---|---|
| Librarian | library.sgraph.ai | Knowledge base, reference documents, 258 markdown files |
| Sherpa | guides.sgraph.ai (or maps/sherpa TBD) | User guides, interactive demos, walkthroughs |

The naming is open. Options: guides, maps, sherpa, help, docs. The key question: what does the Sherpa maintain? The Sherpa is the guide. So the website is the guide. It should feel like someone is walking you through it, not like you are reading a manual.

The website is primarily English with translation support. Pages should be translation-compatible (i18n-ready) so specific pages can have Portuguese or other language versions. Small flag indicators show when content is available in other languages. Full translation comes later when the LLM pipeline (self-hosted Ollama) is wired up.

## Self-Contained Demo Pages

This is the core concept. A demo page is not a description of a feature. It is the feature, running live, embedded in a guided walkthrough.

### The Two-Column Layout

Left column: the explanation. A guided narrative that tells the user what to do, step by step.

Right column: the live demo area. Embedded Web Components, input fields, interactive UI. The user follows the left column and acts on the right column.

### The Flow

```
Step 1:  "Take this vault ID and enter it on the right"
         [user enters ID in the embedded component]

Step 2:  "Now enter the decryption key. Here's one you can use: abc123..."
         [user enters key, vault decrypts in the browser]

Step 3:  "You just opened an encrypted vault. The server never saw the key."
         [vault tree displays, files are browsable]

Step 4:  "Now take this file and paste it into the upload area."
         [user drops a file, it encrypts and uploads]

Step 5:  "You just sent an encrypted file. Click here to see it on the other side."
         [second embedded component shows the received file]
```

Each step builds on the previous one. The user is not reading about encryption. They are encrypting. They are not reading about vaults. They are inside a vault.

### Prerequisites Bar

At the top of each demo page, a clear requirements bar:

| Prerequisite | Status |
|---|---|
| Browser | You have one (you're here) |
| Access token | Needed / Not needed (some demos are free) |
| Vault key | Provided on this page / Bring your own |
| Decryption key | Provided on this page / Bring your own |
| CLI installed | Not needed (browser-only) / Needed for advanced steps |

Some demos leak public decryption keys intentionally. These are demo vaults with demo content. The user can use them to experience the full workflow without signing up. The keys and demo data should be managed from an easy-to-update config (not hardcoded in the HTML).

### Progressive Complexity

Demo pages scale from simple to advanced:

**Level 1: Send a file.** Drop a file, get a link, open the link on another device. No account needed.

**Level 2: Open a vault.** Enter a demo vault ID and key. Browse the tree. Read a file. See that the server stores encrypted blobs.

**Level 3: Create a vault.** Create a new vault in the browser. Add files. Get the key. Share it.

**Level 4: CLI walkthrough.** Side-by-side: the web UI on the left, the CLI equivalent on the right. Show how the same vault can be accessed from both. (If Pyodide works, the CLI runs in the browser too.)

**Level 5: Gallery.** Upload photos, create a public gallery, share the link. See thumbnails, slideshow, download.

**Level 6: LLM workflow.** Upload a document, run an LLM analysis (using provisioned token), get a report in a vault. The investor review workflow as an interactive demo.

## Technical Implementation

Each demo page is a composition of Web Components:

- `<sg-send-upload>` for file upload demos
- `<sg-vault>` for vault interaction
- `<sg-file-viewer>` for displaying decrypted content
- `<sg-tree-view>` for vault tree browsing
- `<sg-key-input>` for entering keys
- `<sg-gallery>` for photo galleries

The demo page is a static HTML page that composes these components with explanatory text. The components do the work. The page tells the story.

If Pyodide (see companion brief) works, the CLI demos can also run in the browser. The demo page embeds a Pyodide terminal alongside the Web Components, showing both interfaces operating on the same vault.

## Connection to Marketing

Every demo page is a marketing page. The user does not read about the product. They use the product. By the time they finish a demo, they have already sent an encrypted file, opened a vault, or created a gallery. The conversion from "demo user" to "paying user" is: "you just did this with a demo key. Want your own?"
