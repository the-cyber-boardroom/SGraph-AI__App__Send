## The Workflow

A user wants to generate images (or use other LLM capabilities) through SG/Send. They need an OpenRouter API token. That token is a secret. The provisioning workflow must create, deliver, and manage that token securely.

### Step by Step

1. **User arrives at the image generation page.** They enter their access token (existing SG/Send credential).

2. **First-time setup: generate a device key pair.** On first use, the browser generates an EC key pair for this device. The public key is sent to the server. The private key stays in the browser (local storage or in-memory). This key pair is tied to this device and this access token.

3. **Server creates an OpenRouter token.** The server calls the OpenRouter API programmatically to create a new token with a spending cap (e.g. 2 GBP, or 50-60% of the user's current credit balance). The token is scoped to the models we offer.

4. **Server encrypts the token with the user's public key.** The OpenRouter token is encrypted with the device public key from step 2. Only this device can decrypt it.

5. **Encrypted token is returned to the user.** The browser decrypts with its private key. The plaintext OpenRouter token is now available in-memory in the browser. It never touches disk unencrypted. It never exists in plaintext on the server after the encryption step.

6. **User makes LLM requests.** The browser calls OpenRouter directly using the decrypted token. SG/Send is not in the request path for the actual LLM call. The browser talks directly to OpenRouter.

7. **Token exhaustion.** When the token's spending cap is reached, the user needs to top up credits. A new token is provisioned with the same flow.

### What the Server Tracks

- How many device public keys have been created for a given access token
- Which OpenRouter tokens are active (for monitoring and revocation)
- Spending per token (via OpenRouter's reporting API)
- Margin per operation (user credit cost minus OpenRouter wholesale cost)

## OpenRouter API Research Needed

Key questions to answer:

- Can we create tokens programmatically via the OpenRouter API?
- Can we set per-token spending caps?
- Can we scope tokens to specific models?
- Can we query token usage/spend in real time?
- What are the rate limits on token creation?

If OpenRouter does not support programmatic token creation with caps, we need an alternative: either a proxy that uses our master token with per-user rate limiting, or a different LLM provider API.

## Model Selection

Offer a small, curated set of models at different price points:

| Tier | Use Case | Example Models | Approximate Cost |
|---|---|---|---|
| Budget | Quick drafts, experiments | Smaller open models via Ollama (self-hosted) | Near zero (infra cost only) |
| Standard | Good quality image generation | GPT-4o, Gemini Flash | 10-30p per image |
| Premium | Best quality, complex prompts | GPT-4o (high quality), Gemini Pro | 50p-1 GBP per image |

The user sees the cost before confirming. No surprises. This follows the existing "No Surprises" billing principle.

## Why This Is a PKI Showcase

This workflow demonstrates the full SPKI value proposition:

- Device key pair generation (PKI primitive)
- Encrypt-for-reader (the token is encrypted for the device, not for storage)
- Zero-knowledge delivery (server encrypts and forgets, only the device can decrypt)
- Key-to-access-token binding (cryptographic link between device identity and account)

Every step uses the same primitives that power vault encryption, document signing, and secure file transfer. The LLM token provisioning is not a separate system. It is the same system applied to a different secret.
