# How My Agentic Workflow Actually Works

People keep asking me how I run 18 AI agents across multiple teams, so let me explain it simply. Because I think the temptation is to overbake this and make it sound more complicated than it is.

## What an Agent Actually Is

An agent is a very focused LLM session. That is it. Whether it is Claude, ChatGPT, or anything else, an agent is an LLM session that has been primed with the context, background information, and materials required to think and act in a specific role.

LLMs have always been good at this. Once you prime them with a particular focus, they act within that frame. But the key insight is not just role-playing. What you are really doing is framing the reference and providing a better prompt.

The power of LLMs is that they can go in any direction. That is also the problem. Even when you say "act like a developer," you have not said much. Which developer? Front end? Back end? Python? Mainframe? An explorer who prototypes fast, or a villager who writes production-grade code with full test coverage? Each of those developers has a different centre of gravity. Each one makes different micro-decisions.

What the agent roles do is provide that guidance. They tell the LLM how to approach problems in this specific context, so it does not spend time figuring out which direction to go or wandering off on tangents. When people create "skills" for Claude, they are doing the same thing: providing pre-built decision frameworks so the LLM can focus on execution instead of orientation.

## What It Looks Like in Practice

Literally, an agent is a folder in a GitHub repo. Each folder contains a clear definition of what the agent is, what their role is, and what their task focus is. When I spin up a session, I point Claude Code at that folder, and it reads the role definition (a `CLAUDE.md` file) and operates within that context.

I have 18 agents across three teams for the main project, plus agents in other repos. But here is the important part: not all 18 are running at the same time. I do not think I have ever had all 18 spinning simultaneously. What actually happens is a small subset (3 to 5 agents) works on a specific task for an hour or two. Think of it as a very short-term squad running a very short-term sprint.

## The Workflow

### Step 1: I Create the Input

I record voice memos throughout the day. I export the transcript and give it to a Claude session (regular Claude web, not the agentic setup). That session structures the messy transcript into a clean, well-formatted document. It reorders things, creates a cohesive argument, and produces something that when I read it, I go "yes, this is what I was talking about."

This is important: I do not give the agents raw materials. I give them high-quality, structured documents. Sometimes those documents are created in one pass from a voice memo. Sometimes they require several rounds of refinement. But the input to the agentic workflow is always a solid brief, not a stream of consciousness.

### Step 2: I Commit to Git

I save the brief into the repo at a specific path: `humans/dinis-cruz/briefs/03/13/` (month and day). I commit it. Git is the communication channel. Everything flows through commits: briefs going in, debriefs coming out, agent outputs, updated documents.

### Step 3: The Librarian Processes It

I go to my Librarian agent session and say "pull from there and process this." The Librarian is responsible for cataloguing, contextualising, and routing. It reads the new brief, adds references, updates existing documents, puts it into context with everything that has come before.

If the brief is purely informational (like this article), not much happens beyond cataloguing. But if the brief contains actionable items (build a new website, design a gallery feature, create an architecture spec), the Librarian creates targeted briefs for the specific agents that need to act on it: the Architect, the Developer, the Sherpa, the Designer, whoever is relevant.

### Step 4: Agents Process in Parallel

The Librarian triggers sub-agents. Each one is a new session (or a session with a clean context) that reads its role definition and processes the brief within its specific frame. The Architect reviews it for architectural implications. The Developer reviews it for implementation details. The Sherpa reviews it for user-facing impact.

Each agent produces an output document. The Librarian collects these into a debriefing, links everything together, and pushes to a branch in Git.

### Step 5: I Review and Execute

I read the debriefing. If it looks right, I say "implement it." That triggers the execution phase, where Claude Code (the coding agent) spins up, reads the briefs, and writes the actual code. Sometimes multiple agents work the implementation in parallel across different parts of the codebase.

## The Conductor

I call my main agent the Conductor. This is the session I use to coordinate everything: routing work, checking status, asking for updates across agents, syncing the project state. It is the orchestration layer, and it is just another Claude session with a specific role definition.

## What Makes This Work

A few things that are not obvious:

**I run this on the web, not locally.** Mainly for security. Cloud-based sessions have a contained execution environment. Each session only has access to one repo and its branches. I have a much better understanding and control of what happens compared to running locally on my laptop.

**Git is the communication layer.** Agents do not talk to each other directly. They communicate through commits, briefs, and debriefs in the repo. This creates an audit trail and makes the entire workflow inspectable.

**The input quality matters enormously.** The difference between a mediocre agent output and a great one is almost always the quality of the brief I give it. A well-structured brief with clear context produces focused, actionable output. A vague brief produces vague output. This is where the voice-memo-to-structured-document pipeline pays for itself.

**Claude Code's February release was the inflection point.** The ability to properly instrument agents, define roles via markdown files, and run sub-processes in contained environments is what made this practical. The concept was possible before, but the tooling was not there.

**It is ridiculously productive.** I can have multiple tabs open, each with a different agent environment. I can say "give me a new Explorer session" or "I want to talk to the Architect of the deploy team" and it spins up with the right context. The time from "I have an idea" to "there is a structured brief, the agents have reviewed it, and implementation has started" is measured in hours, not days.

## What It Is Not

It is not 18 autonomous agents running in the background making decisions. It is not a swarm. It is not self-directing. Every significant decision goes through me. The agents are highly capable assistants with focused expertise, coordinated through a simple workflow of briefs, reviews, and commits. The intelligence is in the role definitions, the quality of the briefs, and the discipline of the workflow. The LLMs provide the execution capacity.
