## Part 1: Three Components to Build in Parallel

Each of these is an independent Web Component. IFD-versioned. Lives in the tools repo. Can be developed by separate agent teams simultaneously. Each should work standalone AND inside sg-layout.

---

### Component 1: `<sg-file-selector>` (Individual File Upload)

**What it does:** The user drops a folder. Instead of zipping everything and uploading one blob, the component shows the file tree with checkboxes. The user selects which files to include. The upload sends the tree structure first, then each file individually, prioritised by size (smallest first).

**The receiver experience:**

```
1. Recipient opens the link
2. Immediately sees the tree structure (filenames, sizes, types)
3. Small files (text, JSON, small images) arrive within seconds
4. Large files stream in the background with progress indicators
5. Each file becomes viewable/downloadable as soon as it arrives
6. The recipient can start working before the upload is complete
```

**Component contract:**

| Attribute | Values | Description |
|---|---|---|
| `mode` | `select`, `view`, `receive` | Select = pre-upload chooser. View = browse uploaded tree. Receive = live incoming files. |
| `source` | `local`, `vault`, `url` | Where files come from |
| `show-hidden` | `true`, `false` | Show/warn about .git, .env, etc. |

**Events emitted:**

| Event | When |
|---|---|
| `files-selected` | User confirms file selection (with tree structure) |
| `tree-uploaded` | Tree structure file has been uploaded |
| `file-uploaded` | Individual file upload complete (per file) |
| `all-uploaded` | All files uploaded |
| `file-available` | (Receive mode) A new file has arrived and is viewable |

**Upload priority:** Files sorted by size ascending. Text files and metadata first, images and documents next, video and large binaries last. The tree structure always goes first.

**Connection to previous briefs:** This implements the "Individual File Storage Architecture" from v0.12.2. The tree structure document, per-file encryption, and three file states (remote/in-browser/cached) are all defined there.

### Component 2: `<sg-key-generator>` (Friendly Key Creator)

**What it does:** A visual, interactive component for creating and understanding encryption keys. The user can pick their own words, see the entropy, and understand how secure their key is.

**The UI:**

```
 ┌─────────────────────────────────────────────┐
 │  Create Your Key                            │
 │                                             │
 │  Word 1: [apple    ▼]  (or type your own)   │
 │  Word 2: [mango    ▼]  (or type your own)   │
 │  Suffix: [56       ] (auto-generated)        │
 │                                             │
 │  Your key: apple-mango-56                   │
 │                                             │
 │  Security: ████████░░  ~4.2 billion combos  │
 │                                             │
 │  [Geek Mode ▼]                              │
 │  Derived AES key: a7f2b9c4d8e1...           │
 │  KDF: PBKDF2 (600,000 iterations)           │
 │  Time to derive: 142ms                      │
 │                                             │
 │  [Copy Key]  [Use This Key]                 │
 └─────────────────────────────────────────────┘
```

**Key design decisions:**

**User-chosen words are allowed.** The user can type any words they want. The key might be "highly-confidential-xyz" or "invoice-acme-march". This is fine because the recipient who receives the key will see the file contents anyway. The key leaking its context is not a threat.

**Visual entropy indicator.** Show the user how many combinations their key has:
- Single digit: 10 combinations
- Single letter: 26 combinations  
- Alphanumeric character: 62 combinations
- Two common words: ~2,000 x 2,000 = 4 million combinations
- Two words + 2-digit number: ~400 million combinations
- Two words + 3-character suffix: ~4 billion combinations

The minimum threshold could be displayed: "Recommended: at least 1 billion combinations" with a visual bar showing where the current key sits.

**Geek mode (expandable).** Shows: the derived AES-256 key (hex), the KDF used (PBKDF2, iteration count), the derivation time in milliseconds. This is educational and builds trust with technical users.

**Performance consideration.** The PBKDF2 derivation must be fast enough for good UX. At 600,000 iterations, test the derivation time across devices. If it's over 500ms on mobile, consider reducing iterations for friendly keys (which have lower entropy expectations than full random keys). Show the derivation time in geek mode so users can see the tradeoff.

**Multilingual words.** The word lists should include words in the user's language. Portuguese users might prefer "sol-praia-42" over "sun-beach-42". The component loads word lists from the translation service.

**Standalone tool.** This component should also exist at tools.sgraph.ai/key-generator/ as a standalone page. It's a marketing tool: people visit to play with key generation, see the entropy visualisation, learn about how encryption keys work. Some of them become SG/Send users.

**Connection to previous briefs:** Implements the "Friendly Decryption Keys" section from the URL simplification brief (v0.11.25). The word-word-number format, the KDF derivation, and the key strength indicator are all defined there.

### Component 3: `<sg-key-input>` (Retrieval / Key Entry)

**What it does:** A focused input for typing a friendly key to retrieve a file. Designed for mobile keyboards.

**The UI (mobile-optimised):**

```
 ┌─────────────────────────────────────┐
 │                                     │
 │  Enter your key to open the file    │
 │                                     │
 │  ┌─────────────────────────────┐    │
 │  │ apple-mango-56              │    │
 │  └─────────────────────────────┘    │
 │                                     │
 │  [Open File]                        │
 │                                     │
 └─────────────────────────────────────┘
```

**Why this matters for mobile:** With friendly keys, the mobile app becomes a retrieval tool. Someone sends you a key via WhatsApp: "apple mango 56". You open the SG/Send app, type three words, tap Open. File decrypts. This reframes the product from "a website where you upload files" to "an app where you send and receive encrypted files."

**Input features:**
- Auto-complete from known word lists (suggest as you type)
- Whitespace/dash tolerance: "apple mango 56", "apple-mango-56", "applemango56" all resolve to the same key
- Visual feedback: key length indicator, format validation
- Paste detection: if user pastes a full URL, extract the key portion automatically
- Token input sanitisation (trim whitespace, strip invisible chars, strip smart quotes) from the debugging brief

**Connection to previous briefs:** Complements `<sg-key-generator>`. Also connects to the token input issues from the customer feedback pack (v0.11.25) where users had problems with invisible characters from WhatsApp copy-paste.

## Part 2: QA and Documentation Pipeline

### The Existing QA Project

There's an existing QA project that already figured out how to use Selenium/Playwright to:
- Launch the website
- Navigate through workflows
- Take screenshots at each step
- Validate UI states

This needs to be revived and integrated into the current workflow.

### What We Need Now

**Automated screenshot pipeline:**

```
1. Playwright launches send.sgraph.ai
2. Navigates through key workflows (upload, receive, key creation, vault)
3. Takes screenshots at each step
4. Screenshots saved to a docs folder (per language, per workflow)
5. Static site generator transforms screenshots + markdown into docs pages
6. Deployed on push (same pattern as the multilingual site generator)
```

**Regression testing:**

```
1. On every PR/push to main
2. Playwright runs through core workflows
3. Upload a test file, receive it, verify contents
4. Check all viewers work (PDF, image, video, text, folder)
5. Check all languages render (at least en-GB and pt-PT)
6. Screenshot comparison: flag visual regressions
```

**Documentation generation:**
Use the screenshots + component descriptions to generate user-facing docs. The static site generator (the IFD transformation pipeline that only runs on push) creates the docs site from markdown + screenshots. This produces documentation that's always current because it's generated from the actual running UI.

## Part 3: Cross-Team Collaboration Workflow

### The Problem

We now have multiple repos: SG/Send (main), tools, desktop, user, admin. Each Claude Code session is tied to one repo. But features span repos: the `<sg-key-generator>` is built in tools but consumed by send and desktop.

### The Pattern: Claude Web for Thinking, Claude Code for Building

```
Phase 1: Thinking (Claude Web)
  - Start a Claude Web session
  - Prompt: "Clone https://github.com/.../SGraph-AI__App__Send
    Read the team roles. I want to discuss [topic] with the Architect."
  - Have an interactive conversation: debate, analyse, explore
  - Output: a brief document (markdown)

Phase 2: Handover
  - Add the brief to the target repo (e.g., tools repo)
  - The brief contains: problem, approach, acceptance criteria, context

Phase 3: Building (Claude Code)
  - Open Claude Code on the target repo
  - "Read the new brief at [path]. Implement it."
  - Claude Code executes: writes code, runs tests, commits
```

### Why This Works

Claude Web is optimised for interactive, multi-turn conversations. It's good at debating options, analysing tradeoffs, and creating structured documents. It's NOT designed for writing code to disk and running tests.

Claude Code is optimised for execution: read a brief, write code, test, commit. It's NOT great for open-ended exploration (it costs more tokens, and the "go figure it out" loop can be wasteful).

The brief is the interface between thinking and building. The same pattern we use between humans and agents, now applied between two different Claude environments.

### Prompts for Each Team

The daily brief should include pre-built prompts for firing up team sessions in Claude Web:

**SG/Send team (main repo):**
```
Clone https://github.com/the-cyber-boardroom/SGraph-AI__App__Send
Read: .claude/CLAUDE.md, .claude/explorer/CLAUDE.md
Read the latest daily brief.
I want to talk to the Architect about [topic].
```

**Tools team:**
```
Clone https://github.com/the-cyber-boardroom/sgraph_ai__tools
Read: .claude/CLAUDE.md
Also clone the SG/Send repo for reference.
I want to discuss a new component: [component name].
```

**Desktop team:**
```
Clone https://github.com/the-cyber-boardroom/SGraph-AI__Desktop
Read: .claude/CLAUDE.md
I want to discuss [feature] for the desktop app.
```
