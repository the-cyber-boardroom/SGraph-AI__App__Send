# Dev Brief: Visual Workflow and Graph Libraries. Mapping Open Source Tools to SG/Send Visualisation Needs

**version** v0.13.5  
**date** 9 Mar 2026  
**from** Human (project lead) + research  
**to** Architect (lead), Developer, Designer  
**type** Research + dev brief. Visual tooling, open source evaluation

## What We Need to Visualise

Before evaluating tools, here's what we actually need to render and edit across the SG/Send ecosystem:

| Visualisation Need | Where It's Used | View/Edit | Complexity |
|---|---|---|---|
| **sg-layout event flow** | Workspace debug panel, library docs | View + Edit | Medium. Nodes = components, edges = events, three channels (internal bus, upward bubble, downward inject) |
| **Digital twin (layout tree)** | Workspace state viewer, sg-layout inspector | View + Edit | Medium. Tree of row/column/stack/tab nodes. Serialisable JSON. |
| **Execution bundle tree** | Workspace history, one-shot LLM bundles | View | Medium. Git-like branching tree. Nodes = bundles, edges = parent relationships, forks = branches. |
| **Agent workflow orchestration** | Agentic team mode, sequential agent calls | View + Edit | High. Nodes = agent roles, edges = data flow between agents, sequential/parallel execution paths. |
| **Encryption pipeline** | Architecture docs, library, marketing | View | Low-Medium. The diagrams from the v0.15.5 architecture explainer as interactive visuals. |
| **File transfer flow** | Debug, docs, support | View | Low. Linear: browser encrypt -> upload -> S3 -> download -> decrypt. |
| **Translation workflow** | babel.sgraph.ai, translation management | View | Low. Linear with branches: English master -> delta detection -> LLM translation -> merge. |
| **Version compatibility graph** | Component registry, sg-layout | View | Medium. Nodes = component versions, edges = compatibility declarations. |
| **Vault tree structure** | Vault browser, individual file storage | View + Edit | Medium. File/folder tree with metadata (size, type, encryption status). |
| **Customer journey / CRM** | Admin, Sherpa dashboard | View | Low-Medium. Timeline/funnel: purchase -> token -> first upload -> ongoing usage. |

## The Evaluation Criteria

Based on SG/Send's architecture principles:

| Criterion | Weight | Why |
|---|---|---|
| **Vanilla JS, no framework dependency** | Critical | Our stack is vanilla JS + Web Components. No React, no Vue, no Angular. |
| **Zero or minimal dependencies** | High | Aligns with our zero-dependency principle (sg-layout has zero deps) |
| **JSON import/export** | Critical | Everything we do is JSON-serialisable. The graph data must be saveable to vault as JSON. |
| **Embeddable in a Web Component** | Critical | Must work inside `<sg-graph-viewer>` hosted in sg-layout |
| **CDN-loadable (single file)** | High | Aligns with IFD and CDN hosting pattern |
| **Edit mode + View mode** | High | Some visualisations are read-only (docs, debug), some are editable (workflow builder) |
| **Themeable (CSS custom properties)** | Medium | Must adopt Aurora theme |
| **Mobile-friendly** | Medium | Must work in the Tauri mobile app WebView |
| **Licence** | Critical | Must be MIT, Apache 2.0, or similarly permissive |

## The Candidates

### Tier 1: Best Fit for SG/Send

#### Drawflow
**The strongest candidate for workflow editing.**

| Attribute | Value |
|---|---|
| **URL** | [github.com/jerosoler/Drawflow](https://github.com/jerosoler/Drawflow) |
| **Dependencies** | Zero. Vanilla JavaScript. |
| **Size** | ~30KB minified |
| **Rendering** | HTML/CSS (DOM-based, not Canvas) |
| **Licence** | MIT |
| **Stars** | ~4.5K |
| **CDN** | Yes (jsDelivr, unpkg) |
| **JSON export/import** | Yes. Full graph serialisable as JSON. |
| **Mobile** | Yes (touch support, pinch zoom) |
| **Edit + View modes** | Yes (`editor.editor_mode = 'edit'` / `'fixed'` / `'view'`) |
| **Custom nodes** | Yes. Nodes are HTML elements. Can register Web Components as nodes. |

**Why Drawflow wins for us:**
- Zero dependencies aligns perfectly with our architecture
- DOM-based rendering means nodes can contain actual Web Components (not just Canvas rectangles)
- A node in the workflow could literally be a live `<sg-key-generator>` or `<sg-file-selector>` component
- JSON export maps directly to vault storage
- Edit/view/fixed modes match our debug-panel-vs-production needs
- Four lines of code to get started
- Custom element support: their docs show how to use Drawflow inside a custom element

**Best for:** Agent workflow orchestration, encryption pipeline visualisation, translation workflow, customer journey, any workflow where nodes represent distinct steps with inputs/outputs.

**Limitation:** Not designed for tree visualisation (it's a flow graph, not a hierarchical tree). Not great for the layout tree or execution bundle tree.

---

#### LiteGraph.js (original, jagenjo fork)
**The strongest candidate for node-based computation graphs.**

| Attribute | Value |
|---|---|
| **URL** | [github.com/jagenjo/litegraph.js](https://github.com/jagenjo/litegraph.js) |
| **Dependencies** | Zero runtime dependencies |
| **Size** | ~150KB minified |
| **Rendering** | HTML5 Canvas |
| **Licence** | MIT |
| **Stars** | ~6K (original), ComfyUI fork is much larger |
| **CDN** | Yes (single file) |
| **JSON export/import** | Yes. Graphs export as JSON. |
| **Mobile** | Limited (Canvas-based, no touch gestures by default) |
| **Edit + View modes** | Yes |
| **Custom nodes** | Yes. Powerful node type system with typed inputs/outputs. |

**Why LiteGraph is interesting for us:**
- Used by ComfyUI (the most popular AI image generation UI), so battle-tested at scale
- The node type system (typed inputs/outputs) maps perfectly to sg-layout's event contract
- Graphs can RUN: nodes can execute code, not just be visual. This means a visual workflow can actually be an executable pipeline.
- Server-side execution via Node.js: the same graph that's built visually can run headless
- "Unreal Blueprints in the browser": the visual programming paradigm for agent orchestration

**Best for:** sg-layout event flow visualisation, digital twin inspection, computation graphs where the visual representation IS the execution model. Future: visual agent orchestration where you wire up agents and the graph runs.

**Limitation:** Canvas-based means nodes can't contain live HTML/Web Components (they're drawn, not DOM elements). Heavier than Drawflow. The ComfyUI fork has diverged significantly and is now embedded in the ComfyUI monorepo.

---

#### Mermaid (for static diagrams in docs/library)
**Already supported. Use for documentation, not for interactive editing.**

| Attribute | Value |
|---|---|
| **URL** | [mermaid.js.org](https://mermaid.js.org) |
| **Dependencies** | d3, dagre |
| **Size** | ~1.5MB (heavy) |
| **Rendering** | SVG |
| **Licence** | MIT |
| **Interactive** | No. Static rendering from text definitions. |

**Best for:** Architecture diagrams in the library, README files, static documentation. The v0.15.5 architecture explainer's ASCII diagrams could be converted to Mermaid for richer rendering.

**Not suitable for:** Anything interactive or editable.

### Tier 2: Worth Evaluating

#### Flowy.js
**Minimal flowchart builder. Even simpler than Drawflow.**

| Attribute | Value |
|---|---|
| **URL** | [github.com/alyssaxuu/flowy](https://github.com/alyssaxuu/flowy) |
| **Dependencies** | Zero. Vanilla JS. |
| **Size** | ~15KB minified |
| **Rendering** | DOM-based |
| **Licence** | MIT |
| **JSON export** | Yes (JS object/JSON) |

**Best for:** Very simple, linear workflows. Drag-and-drop from a sidebar palette. The simplest possible visual workflow builder.

**Limitation:** Less feature-rich than Drawflow. No zoom, no modules, limited node customisation. Might be TOO simple for our needs, but worth considering for the simplest use cases (linear file transfer flow, customer journey).

---

#### D3.js (for custom tree visualisations)
**Low-level. Build exactly what we need.**

| Attribute | Value |
|---|---|
| **URL** | [d3js.org](https://d3js.org) |
| **Dependencies** | None (modular, import only what you need) |
| **Rendering** | SVG/HTML |
| **Licence** | ISC (permissive) |

**Best for:** The execution bundle tree (git-like branching), the layout tree (row/column/stack hierarchy), version compatibility graph. D3's tree and force-directed layouts are purpose-built for these.

**Tradeoff:** D3 is a toolkit, not a ready-made editor. We'd need to build the interactive editing layer ourselves. Good for view-only visualisations, significant work for editable ones.

---

#### JointJS (free tier)
**The most mature diagramming library. Free tier is sufficient for view/basic edit.**

| Attribute | Value |
|---|---|
| **URL** | [jointjs.com](https://jointjs.com) |
| **Dependencies** | None (standalone since v4) |
| **Size** | ~300KB |
| **Rendering** | SVG |
| **Licence** | MPL 2.0 (free tier), commercial for JointJS+ |

**Best for:** If we need production-grade BPMN-style workflow editing. The most polished option.

**Tradeoff:** Heavier than Drawflow or LiteGraph. The free tier lacks some advanced UI components (minimap, toolbar, snaplines). The commercial tier is subscription-based.

### Tier 3: Not Recommended

| Library | Why Not |
|---|---|
| **React Flow** | React dependency. Excellent library but incompatible with our vanilla JS + Web Components stack. |
| **GoJS** | Commercial licence (not free for production). Very capable but not open source. |
| **Syncfusion** | Commercial. Enterprise pricing. Overkill. |
| **bpmn-js** | Excellent but BPMN-specific. We don't need BPMN notation. |
| **dgrm.net** | Interesting (pure JS, no deps) but very early stage and not well documented. |

## The Recommendation: Two Libraries, One Wrapper

### Strategy: Drawflow for workflows, D3 for trees, wrapped in Web Components

| Visualisation Type | Library | Component |
|---|---|---|
| **Workflow editing** (agent orchestration, encryption pipeline, translation flow) | Drawflow | `<sg-workflow-editor>` |
| **Tree viewing** (execution bundles, layout tree, vault structure, version graph) | D3.js (tree layout) | `<sg-tree-viewer>` |
| **Static diagrams** (library docs, architecture) | Mermaid | `<sg-markdown-viewer>` (already planned, renders Mermaid blocks) |
| **Future: computation graphs** (visual agent programming, live event monitoring) | LiteGraph.js | `<sg-graph-engine>` (P3, when we need executable visual graphs) |

### Why Two, Not One

No single library does both workflow editing (horizontal flow with inputs/outputs) and tree visualisation (hierarchical branching) well. Drawflow is perfect for the first, D3 for the second. Both are vanilla JS, both export JSON, both work in a Web Component wrapper.

### The Web Component Wrapper Pattern

Each library gets wrapped in a Web Component that:
- Hosts the library in its container
- Implements the sg-layout registration protocol (works inside panels)
- Accepts data via attributes or events (JSON graph/tree data)
- Emits events on user interaction (node-clicked, edge-added, graph-changed)
- Saves/loads state to/from vault (JSON export)
- Applies the Aurora theme (CSS custom properties mapped to library styles)

```html
<!-- Workflow editor in an sg-layout panel -->
<sg-workflow-editor
  data-source="vault://workflows/agent-pipeline.json"
  mode="edit"
  theme="aurora">
</sg-workflow-editor>

<!-- Tree viewer in another panel -->
<sg-tree-viewer
  data-source="vault://executions/tree.json"
  layout="horizontal"
  mode="view">
</sg-tree-viewer>
```

The consuming code never interacts with Drawflow or D3 directly. It interacts with `<sg-workflow-editor>` and `<sg-tree-viewer>`. If we swap the underlying library later, the component interface doesn't change.

## Concrete Use Cases Mapped to Libraries

| Use Case | Library | How |
|---|---|---|
| **Agent workflow builder** | Drawflow | Nodes = agent roles (Sherpa, Ambassador, etc.). Edges = data flow. User drags agents from a palette, connects them, configures each. Export as JSON bundle. |
| **Execution bundle history** | D3 tree | Git-like branching tree. Nodes = bundles. Click a node to load that bundle into the workspace. Branches show forks. |
| **sg-layout digital twin** | D3 tree | Render the layout tree (row/column/stack) as an interactive tree diagram. Click a node to highlight the corresponding panel. |
| **sg-layout event monitor** | Drawflow (view mode) | Live event flow: nodes = components, edges light up when events fire between them. Real-time debug visualisation. |
| **Encryption pipeline docs** | Mermaid | Convert the v0.15.5 ASCII diagrams to interactive Mermaid flowcharts in the library. |
| **Translation pipeline** | Drawflow (view mode) | English master -> delta detect -> LLM translate -> merge. Each node shows live status. |
| **Version compatibility** | D3 force-directed | Nodes = component versions. Edges = compatibility declarations. Force layout shows clusters of compatible versions. |
| **One-shot prompt builder** | Drawflow | Nodes = context elements (system prompt, files, constructed history, question). Edges = assembly order. The visual IS the prompt construction. |
