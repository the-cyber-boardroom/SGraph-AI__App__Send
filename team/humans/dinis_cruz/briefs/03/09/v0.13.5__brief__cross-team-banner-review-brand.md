# Brief: Cross-Team Component Review. Top Banner Refactoring, Brand Identity, and Consumption Pattern

**version** v0.13.5  
**date** 9 Mar 2026  
**from** Human (project lead / Conductor)  
**to** SG/Send team (Architect, Developer, Designer, Sherpa) reviewing the Tools team's work  
**type** Review commission + brand decision + dev brief  

---

## Context: Two Teams, One Component

The SG/Send agentic team (bootstrapped via repo clone, working on the main send repo) has been asked to review work done by the Tools agentic team (working on the sgraph_ai__tools repo). The Tools team is refactoring the top banner component, which is the header bar that will appear across all SG/Send websites.

This is the first real cross-team component review. The pattern: one team builds the component in the tools repo, the other team reviews it and prepares to consume it.

---

## The Refactoring Principle: Zero Visual Difference First

**This is critical and must be solidified as a principle.**

When refactoring code from one repo to another (send repo to tools repo), or when consolidating from multiple IFD minor versions to one major version, the first iteration must produce **zero visual difference**. Pixel-perfect. No font changes, no spacing shifts, no colour variations.

This is the Wardley Map maturity progression applied to code movement:

| Stage | What Happens |
|---|---|
| **Genesis** (current, in send repo) | Top banner exists, works, but was "glued together" inside the send website |
| **Custom Built** (tools repo, first refactoring) | Same visual output, but cleaner code, better architecture, proper Web Component structure |
| **Product** (tools repo, subsequent versions) | Configurable, themeable, consumable by multiple sites, translation-aware |

The common mistake: teams introduce "improvements" during the refactoring step. A font gets changed. A spacing gets adjusted. A colour gets tweaked. Now you have two slightly different banners and nobody can tell if the difference is intentional or a bug. This creates debugging noise and erodes confidence in the component.

**The rule: refactoring changes the structure, not the output. Clean up inside. Don't touch the surface.**

## What the Review Team Must Do

### Review 1: Visual Fidelity Check (Designer + Developer)

Compare the top banner in the tools repo against the current banner on send.sgraph.ai. They should be visually identical.

Check for:
- Font family, weight, and size
- Colours (exact hex values, not "looks close")
- Spacing and padding (pixel-level)
- Language selector rendering
- Logo placement and sizing
- Responsive behaviour (at different widths)
- Hover states, transitions, active states

**If there are differences:** Flag them. Do not accept the component until they are resolved. The first version in tools must be a byte-for-byte visual match.

### Review 2: Architecture Quality (Architect + Developer)

The component in the tools repo should have *better* architecture than what exists in the send repo. This is the maturity upgrade:

Check for:
- Proper Web Component structure (self-contained, event-driven, no side effects)
- CSS custom properties for all visual values (no hardcoded colours or sizes)
- Translation support built in from the ground up (keys, fallback to English, ready for babel service)
- Clean event contract (what events does the banner emit? language-changed, navigation, etc.)
- No dependencies on the send repo (the component must work standalone)
- IFD versioning in the tools repo (proper folder structure, version pinning)

### Review 3: Consumer Experience (Designer + Sherpa)

Think about the experience of consuming this component from another repo:

- How does the library repo include the top banner? (A single script tag + custom element?)
- How does a new website configure the banner for its context? (Attributes? Slots? Config object?)
- What documentation is needed for a developer (or LLM) to integrate the banner?
- Is the integration a one-liner or does it require understanding internal state?

### Review 4: Reverse Impact on Send Repo (Architect + Developer)

Once the tools repo owns the banner, the send repo must:
- Remove its own banner code (significant code deletion)
- Import the banner from tools (script tag from CDN)
- Pin to a specific version (IFD versioning)
- Verify that the import produces the identical visual result

Map out: what code in the send repo gets deleted? What's the import mechanism? What's the testing strategy to confirm nothing broke?

## Brand Decision: SG/Send Is the Core Brand

**Executive decision:** The core brand is **SG/Send** (not SGraph, not tools.sgraph.ai, not individual sub-brands).

What this means:

| Site | Top Banner Identity | Reasoning |
|---|---|---|
| **send.sgraph.ai** | SG/Send | The core product |
| **vault.sgraph.ai** | SG/Send (with "Vault" subtitle) | Part of the SG/Send family |
| **workspace.sgraph.ai** | SG/Send (with "Workspace" subtitle) | Part of the SG/Send family |
| **tools.sgraph.ai** | SG/Send (with "Tools" subtitle) | Part of the SG/Send family |
| **library.sgraph.ai** | SG/Send (with "Library" subtitle) | Part of the SG/Send family |
| **view.sgraph.ai** | SG/Send (minimal, viewer-focused) | Special identity for recipients |

**The top banner linked sites feel identical.** If a user clicks from send to tools to library, the top banner should not change (except the subtitle). No floating elements, no rearranging, no font shifts. Consistency IS the brand.

**Deeper pages can have their own identity.** The download viewer, the vault file browser, the workspace panels can have specialised headers. But the top-level navigation bar is always SG/Send.

### The Tools Branding Question

Should tools.sgraph.ai have its own identity? **No, not when linked from the top banner.** If a user clicks "Tools" in the top nav, they should feel like they're still in SG/Send, just in a different section. The URL changes but the experience doesn't. This means the tools site should NOT override the top banner.

If a user arrives at tools.sgraph.ai directly (from a search engine, a shared link), they see the SG/Send banner with "Tools" subtitle. They understand immediately that this is part of the SG/Send ecosystem.

## Translation Must Be Core

The tools team's top banner has a translation bug. This must be fixed before the component is shipped.

**Translation requirement for all shared components:**
- Every user-visible string must be a translation key (not hardcoded text)
- The component must support the `<sg-translations>` service (from the babel brief)
- If the translation service is not loaded, fall back to English
- The language selector in the top banner must work across all consuming sites
- When the language changes in the banner, all components on the page must update

This is not optional. Every component in the tools repo must be translation-ready from v0.1.0. Retrofitting translations later is the same mistake as retrofitting tests later.

## Output: What This Review Produces

The SG/Send team produces documents that go into the tools repo so the Tools team can execute:

| Document | Content |
|---|---|
| **Visual Review** | Differences found (if any), required fixes, pixel-level comparison |
| **Architecture Review** | Code quality assessment, suggested improvements, event contract |
| **Consumer Guide** | How to import and configure the banner, one-liner examples, attribute reference |
| **Migration Plan** | What code to delete from send repo, how to import from tools, testing strategy |
| **Brand Guidelines** | SG/Send as core brand, subtitle pattern, consistency requirements |
| **Translation Requirements** | Key naming convention, service integration, fallback behaviour |

These documents become the brief that the Tools agentic team (which has limited context about the send repo) uses to finish the implementation.
