**date** 9 Mar 2026  
**from** Human (project lead)

## Why Now

The library has 258 markdown files, 8 dev packs, 4 incident records, 14 research documents, 12 investor materials, and 12 development guides. It's a rich knowledge base. But it lives inside a GitHub repo where finding anything requires knowing the path. When the human wanted to start a new Claude session to discuss tools architecture, the immediate friction was: "where's that file? Where can I find it?"

The content exists. The discoverability doesn't.

At the same time, we just created a 780-line architecture explainer (`v0_15_5__sg_send_architecture_explainer.md`) with detailed diagrams of the entire encryption pipeline. That document should be easy to find from a website, not buried in a repo folder. Every document like it should be.

---

## What library.sgraph.ai Is

A static website that makes the library's content browsable by humans and consumable by LLMs. Two audiences, one content source, two delivery formats.

**For humans:** Browse by section, search by topic, read rendered markdown with diagrams and code blocks. Navigate to any document in a few clicks. See what's current, what's stale, what's the reading order for a new team member.

**For LLMs:** Structured indexes in JSON and CSV. Raw markdown files accessible at predictable URLs. A site map that an LLM can parse to understand what's available. Machine-readable metadata: document titles, versions, dates, roles, topics, staleness status.

**The principle:** Markdown first. Stored as markdown. Served as HTML (client-side rendering via a lightweight markdown component). But the raw markdown is always available alongside the HTML. The LLM doesn't need to parse HTML; it reads the markdown directly.

## Architecture

### Content Source

The content lives in the GitHub repo (`library/` folder). The website reads from this source. There is no separate CMS, no database, no admin panel. Push to the repo, the website updates.

### Rendering Pipeline

```
library/ (GitHub repo)
    │
    ├── markdown files (source of truth)
    │
    ▼
S3 sync (on push to main)
    │
    ├── *.md files → served as-is (for LLMs)
    ├── *.html files → generated from markdown (for humans)
    ├── index.json → generated site index (for LLMs and search)
    ├── catalogue.json → structured metadata per document
    └── sitemap.xml → standard sitemap
    │
    ▼
CloudFront → library.sgraph.ai
```

### Client-Side Rendering (for humans)

The HTML pages are lightweight shells that load markdown and render it client-side using a markdown component (we already have `MarkdownParser` in the CDN shared libraries). No server-side rendering. No build step that generates hundreds of HTML pages.

```html
<!-- Every page is the same shell -->
<sg-markdown-viewer src="/library/guides/development/ifd/v1.2.1__ifd__intro-and-how-to-use.md">
</sg-markdown-viewer>
```

The `<sg-markdown-viewer>` component:
- Fetches the markdown file
- Renders it to HTML (with syntax highlighting for code blocks, diagram rendering for ASCII art)
- Adds navigation (breadcrumbs, section links, prev/next)
- Supports the Aurora theme (same as workspace, vault, tools)

### Raw Markdown Access (for LLMs)

Every document is available as raw markdown at a predictable URL:

```
Human URL:  library.sgraph.ai/guides/ifd/intro
Raw URL:    library.sgraph.ai/raw/guides/development/ifd/v1.2.1__ifd__intro-and-how-to-use.md
```

LLMs clone the repo OR fetch the raw URLs. Both work. The raw URLs are stable, versioned, and don't require HTML parsing.

## LLM-Optimised Content Delivery

### Machine-Readable Indexes

**`/index.json`** (full site index):
```json
{
  "generated": "2026-03-09T14:00:00Z",
  "version": "v0.13.6",
  "total_documents": 258,
  "sections": [
    {
      "path": "/guides/",
      "title": "Development Guides",
      "documents": [
        {
          "path": "/guides/ifd/intro",
          "raw": "/raw/guides/development/ifd/v1.2.1__ifd__intro-and-how-to-use.md",
          "title": "IFD Methodology: Introduction and How to Use",
          "version": "v1.2.1",
          "date": "2026-02-20",
          "roles": ["developer", "architect"],
          "topics": ["ifd", "methodology", "development"],
          "status": "current"
        }
      ]
    }
  ]
}
```

**`/catalogue.csv`** (spreadsheet-friendly):
```csv
path,raw_url,title,version,date,section,roles,topics,status
/guides/ifd/intro,/raw/guides/.../v1.2.1__ifd__intro.md,IFD Intro,v1.2.1,2026-02-20,guides,"developer,architect","ifd,methodology",current
```

**`/reading-order.json`** (for new agents bootstrapping):
```json
{
  "new_developer": [
    "/guides/ifd/intro",
    "/dependencies/osbot-utils/type-safe",
    "/dependencies/memory-fs/architecture",
    "/specs/readme"
  ],
  "new_security_reviewer": [
    "/architecture/sg-send",
    "/incidents/",
    "/investor/business-plan"
  ]
}
```

### Navigation for LLMs

An LLM consuming the library should be able to:
1. Fetch `/index.json` to understand what exists
2. Fetch `/reading-order.json` to know where to start
3. Fetch specific raw markdown files for the content it needs
4. Never need to parse HTML

This is more efficient than cloning the entire repo (which works but pulls all code too). The library website serves just the knowledge content, structured for consumption.

## Technical Requirements

### Same Patterns as Other SG/Send Sites

- IFD versioned
- Aurora theme (CSS custom properties)
- Web Components (sg-layout for page structure, sg-markdown-viewer for content)
- Static hosting on S3 + CloudFront
- Multilingual support via the babel/translation service (en-GB first, then expand)

### What's NOT Needed

- No server-side rendering (markdown renders client-side)
- No database (content comes from the repo)
- No CMS (edit markdown in the repo, push, site updates)
- No Jekyll, Hugo, or any static site generator framework (we build our own with Web Components)
- No SEO optimisation initially (the markdown files are the primary content, HTML is a convenience view)

### New Components Needed

| Component | What It Does |
|---|---|
| `<sg-markdown-viewer>` | Fetches and renders markdown with syntax highlighting, diagrams, navigation. Extends the existing MarkdownParser. |
| `<sg-library-nav>` | Section navigation, breadcrumbs, search. Reads from index.json. |
| `<sg-document-meta>` | Shows document metadata: version, date, roles, staleness status, raw markdown link. |
| `<sg-prompt-card>` | Displays a copyable team prompt (for the /team/ section). |

## Content Curation: The Librarian's Daily Practice

The library website isn't just a static dump of files. It needs ongoing curation:

**Daily (automated):**
- Regenerate index.json, catalogue.csv, sitemap.xml from the current repo state
- Flag documents where the version is more than 2 versions behind the project version

**Weekly (Librarian agent):**
- Review staleness flags, update status markers
- Check that reading orders are still correct
- Cross-reference new briefs with library content (does a new brief obsolete a library document?)

**On major milestones:**
- Create a new release pack in `/releases/`
- Update the investor materials if the milestone affects the business narrative
- Update the architecture section if the architecture changed

**Budget:** The Accountant should track the token cost of library curation. The daily automated tasks should cost near zero (they're scripts, not LLM calls). The weekly review should be a focused, one-shot LLM call with curated context (what changed this week, what documents are flagged). The curation itself should follow the penny-per-request principle.
