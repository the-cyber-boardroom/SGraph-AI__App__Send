## Part 1: Website Pivot. From "Buy Now" to "Show Me"

### The Problem

The website's first phase worked: when nudged users arrived, they found the Buy button and purchased. But nobody buys without understanding. A technical friend just asked the simplest, most important question: **"Why is this needed?"**

We need to answer that question head-on. The website must shift from "here's a button, buy it" to "here's what it does, here's why it matters, here's how people use it."

### The "Why Is This Needed?" Article

Commission an article (same format as the LLM cost engineering and identity articles) that answers this directly. Two layers:

**Layer 1: Why it's useful right now (features that are shipped).** Even without encryption as the selling point, the UX is already better than email for file sharing. Real customers are already using it: drop a folder, get a link, send via WhatsApp. No attachment size limits, no "your file is too large" bounces, no file sitting in someone's inbox forever. The early adopter who sent their feedback as a ZIP via SG/Send is proof: it was easier than email.

**Layer 2: Why the security architecture matters (what's coming).** Client-side encryption, zero-knowledge, PKI, vaults with governance and control. The positioning: "We can make secure file sharing actually usable." Today, secure file sharing is either unusable (PGP) or not actually secure (WeTransfer's server-side encryption means they CAN see your files). SG/Send is the intersection: genuinely secure AND genuinely usable.

Reference the two recent LinkedIn articles:
- ["One-Shot LLM: Engineering AI Requests from Dollars to Fractions of a Cent"](https://www.linkedin.com/pulse/one-shot-llm-engineering-ai-requests-from-dollars-fractions-cruz-7tupe/)
- ["Trust Is a Spectrum, Not a Switch"](https://www.linkedin.com/pulse/trust-spectrum-switch-dinis-cruz-ng1we/)

These demonstrate the thinking behind the product and establish credibility with a technical audience.

### Website Structure Needed

```

  [Hero: What SG/Send does, in one sentence]
  [Demo: Show it working. Interactive component or video.]
  [Why: "Why is this needed?" section]
```

The "Demo" section is critical. This is where the public content component (Part 2 below) becomes both a feature and a marketing tool.

## Part 2: Public Content Component. `<sg-public-viewer>`

### The Concept

A Web Component that can be embedded on any page to display encrypted content from SG/Send. The page provides the ID and decryption key (or public key). The component fetches, decrypts, and renders the content.

### Two Modes

**Mode 1: Symmetric key (leaked by design).** The page contains both the file ID and the decryption key. The content is technically encrypted on the server, but anyone viewing the page can see it. The point isn't secrecy; it's proof that the encryption pipeline works, and that the content is served via SG/Send.

```html
<sg-public-viewer
  file-id="abc123"
  key="apple-mango-56"
  viewer="pdf">
</sg-public-viewer>
```

**Mode 2: Public key verification.** The content is encrypted with a public key published on the website. The browser uses the public key to decrypt. This proves: the content was encrypted FOR this website, and only this website's published key can decrypt it. A stronger trust signal.

```html
<sg-public-viewer
  file-id="def456"
  public-key-url="https://sgraph.ai/.well-known/sg-identity.json"
  viewer="html">
</sg-public-viewer>
```

### Why This Matters

This component turns the website into a live demo. Instead of saying "we encrypt files," the website shows encrypted content being decrypted in real-time in the browser. The viewer is the proof. Every page on the website that uses `<sg-public-viewer>` is simultaneously a feature, a demo, and a trust signal.

### Use on Multiple Domains for Web of Trust

Publish the same public key on multiple domains the human controls: sgraph.ai, dinis.ai, thecyberboardroom.com, etc. Each domain hosts a page with `<sg-public-viewer>` that decrypts content using the published public key. An attacker would need to compromise ALL domains simultaneously to substitute a fake key. This is the web of trust from the identity brief (v0.12.2), made tangible and visual.

## Part 3: Mixed Public/Private Vaults. `<sg-mixed-vault>`

### The Concept

A vault (or folder) that contains both public and private data. Public files have their decryption keys leaked on the page. Private files require a separate key or PKI access.

### Real-World Use Case 1: Medical Data Sharing

The human has an MRI scan. They want to crowdsource opinions from their network. The workflow:

```
Vault: "knee-mri-march-2026"
  /public/
    overview.md          <- Public: "Here's what happened, here's what I'm asking"
    xray-screenshot.png  <- Public: Low-res screenshot for context
  /private/
    full-mri-scan.dcm    <- Private: Full MRI, needs separate key
    doctor-notes.pdf     <- Private: Doctor's assessment, needs separate key
```

Anyone with the vault link sees the public folder: the overview and the screenshot. To access the private files (full MRI, doctor's notes), they contact the human and get a separate decryption key. The page shows: "2 public files, 2 private files (request access)."

### Real-World Use Case 2: Event Coordination (Poll)

Organising a group event (Pilates class). The vault has:

```
Vault: "pilates-free-class"
  /public/
    details.html         <- Public: Date, time, location, description
    studio-photo.jpg     <- Public: Photo of the studio
  /organiser/
    responses.json       <- Private: Who's coming (organiser access only)
    contact-list.csv     <- Private: Participant contact details
```

The public link shows the event details. The organiser has a separate key for the management files. Participants can vote/respond (future: a simple form component that writes to the vault). This is a micro-app built from vault primitives.

### Component Design

```html
<sg-mixed-vault
  vault-id="knee-mri-march-2026"
  public-key="apple-mango-56"
  viewer="folder">
</sg-mixed-vault>
```

The component shows:
- Public files: rendered immediately (decryption key is on the page)
- Private files: listed by name/type/size but content locked. "Request access" button.
- Visual distinction: public files have an open padlock icon, private files have a closed padlock

## Part 4: Version Compatibility as First-Class Architecture

### The Problem

We already have two incompatible vault versions: the original (every file in a tree) and the new Git-inspired version (content-addressed blobs). As we push more components, version incompatibilities will multiply. We need a way for components to declare their version and compatibility, and for the system to route messages correctly.

### Version Registration in sg-layout

When a component registers with sg-layout, it should declare:

```javascript
this.dispatchEvent(new CustomEvent('sg-layout:register', {
  bubbles: true, composed: true,
  detail: {
    respond: (config) => { ... },
    identity: {
      componentId: 'sg-vault-browser',
      version: '2.0.0',
      compatibleWith: ['1.0.0', '1.1.0', '2.0.0'],
      events: {
        emits: ['vault-opened', 'file-selected'],
        listens: ['vault-open', 'file-open']
      }
    }
  }
}));
```

### What This Enables

**Parallel versions.** Two versions of the vault component can coexist in the same layout. v1.0 listens for `vault-open@1.0` events. v2.0 listens for `vault-open@2.0`. A caller can target a specific version.

**Compatibility shimming.** A v2.0 component can declare compatibility with v1.0. It listens for both v1.0 and v2.0 events, translating the old format internally. This allows gradual migration: consumers can keep calling the v1.0 API while the component has already upgraded.

**Version flipping.** In development or testing, switch between component versions with a config change. "Show me this page with vault v1.0" vs "show me this page with vault v2.0." This is A/B testing at the component level.

**Future: Signed events.** When events include PKI signatures, the version and identity are cryptographically verified. A component can't impersonate another version. The event bus becomes an authenticated message bus.

### The Migration Pattern

```
Phase 1: New component registers as v2.0, compatibleWith: ['1.0']
Phase 2: Consumers gradually update to call v2.0 API
Phase 3: v1.0 compatibility removed from the component
Phase 4: v1.0 component retired
```

This is the same pattern as API versioning in web services, applied to Web Components.

### First Application: Vault Migration

The vault v1.0 (tree-based) and v2.0 (Git-inspired) should be the first test of this pattern. Both register in the same layout. The file browser component targets whichever version the user's vault uses. Eventually, a migration tool converts v1.0 vaults to v2.0 format.
