# Dev Brief: URL Simplification — view.sgraph.ai, Friendly Keys, and Link Architecture

**version** v0.11.25  
**date** 6 Mar 2026  
**from** Human (project lead)  
**to** Architect (lead), Developer  
**type** Dev brief — URL design, key derivation, viewer routing  

---

## The Problem

The current share URLs are a mess. They contain the full transfer ID and the decryption key in the hash fragment, producing URLs that are long, ugly, and impossible to communicate verbally. When a user shares a link via WhatsApp or reads it aloud on a phone call, the URL is a wall of characters.

This matters because the URL IS the product experience for recipients. Every shared link is a product demo. An ugly URL undermines trust. A clean URL builds it.

---

## The Proposed URL Structure

### New Domain: view.sgraph.ai

A dedicated viewing domain, separate from the send domain:

```
Current:   send.sgraph.ai/#/download/abc123def456...#key=789xyz...
Proposed:  view.sgraph.ai/pdf/Kx7mQ#{friendly-key}
```

### URL Pattern

```
view.sgraph.ai/{viewer}/{short-id}#{decryption-key}
```

| Part | Purpose | Example |
|---|---|---|
| `view.sgraph.ai` | Viewing domain (clean, memorable) | |
| `/{viewer}` | Primes the viewer type (optional, can be auto-detected) | `/pdf`, `/video`, `/gallery`, `/folder`, `/secret`, or just `/v` for generic |
| `/{short-id}` | Short, unique identifier for the transfer | `Kx7mQ` (5 chars), `Kx7mQ9bR` (8 chars) |
| `#{decryption-key}` | Decryption key in the hash fragment (never sent to server) | `apple-mango-56` or `aBcDeFgH` |

**Examples:**

```
view.sgraph.ai/pdf/Kx7mQ#apple-mango-56
view.sgraph.ai/video/R8tNp#blue-sunset-442
view.sgraph.ai/folder/mW3xK#crystal-river-88
view.sgraph.ai/secret/Yq9Lm#one-time-key-here
view.sgraph.ai/v/Kx7mQ#aBcDeFgH1jKL
```

### The Viewer Prefix

The viewer prefix (`/pdf`, `/video`, `/gallery`, `/folder`, `/secret`) is optional but valuable:

- **Primes the UI:** The page knows which viewer to load before fetching the file
- **Sets expectations:** The recipient knows they're about to see a PDF, not a mysterious encrypted blob
- **SEO-friendly:** Different viewer paths can have different meta tags
- **Shareable context:** "I'm sending you a video — view.sgraph.ai/video/..." tells the recipient what to expect

If omitted, use `/v/` (generic) and auto-detect the file type after decryption.

## Friendly Decryption Keys

### The Current State

Decryption keys are cryptographically random strings — secure but impossible to communicate verbally or remember.

### The Friendly Key Model

Generate human-friendly keys that derive into full cryptographic keys via a KDF (Key Derivation Function):

```
Friendly key:    apple-mango-56
                      ↓
                 PBKDF2 / Argon2 with known salt
                      ↓
Full AES key:    a7f2b9c4d8e1f3a5b7c9d2e4f6a8b0c3d5e7f9a1b3c5d7e9f1a3b5c7d9e1f3
```

**Formats:**

| Format | Example | Entropy | Use Case |
|---|---|---|---|
| **Word-word-number** | `apple-mango-56` | ~30 bits (2048 words × 2048 words × 100) | Verbal sharing, WhatsApp |
| **Word-word-word** | `crystal-river-sunset` | ~33 bits (2048³) | Higher security verbal |
| **Short alphanumeric** | `Kx7mQ9bR` | ~48 bits | Copy-paste, not verbal |
| **Full random** | `a7f2b9c4d8e1...` | 256 bits | Maximum security (current) |

The user chooses the format at share time. The system generates the friendly key and derives the full AES key from it. The friendly key is what gets shared. The full key is what decrypts.

**Security tradeoff:** A 30-bit friendly key is brute-forceable in theory (~1 billion combinations). But the attacker also needs the transfer ID (which they can only get from the link, not from brute-forcing). And the KDF (PBKDF2 with high iterations) makes each attempt expensive. For casual sharing (photos with friends, non-sensitive files), this is sufficient. For high-security sharing, the user uses the full random key.

## Short IDs: Mapping to Transfer IDs

### The Challenge

The backend generates transfer IDs (currently sequential or random). These are long. We need a short, unique, URL-safe identifier that maps to the full transfer ID.

### Options

**Option A: Server-generated short ID.** When a file is uploaded, the server generates a short ID (5-8 characters, base62 alphanumeric) and stores the mapping: short ID → full transfer ID. The client uses the short ID in the URL.

- Pros: Short, clean, no collisions (server controls uniqueness)
- Cons: Requires a server-side lookup on every view request

**Option B: Deterministic short ID from transfer ID.** The short ID is derived from the full transfer ID using a hash: `shortId = base62(sha256(transferId).slice(0, 5))`.

- Pros: No server-side mapping needed — the server can reconstruct the full ID from the short ID (or vice versa)
- Cons: Collision risk at 5 chars (~916 million combinations in base62 — low risk at our scale but needs monitoring)

**Option C: GUIDs as transfer IDs.** Move to deterministic GUIDs (UUIDv5) for transfer IDs. The GUID can be shortened for display while remaining unique.

- Pros: Deterministic from input data, universally unique, standard format
- Cons: Still 32 hex chars — need shortening for the URL

**Recommendation:** Option A for the MVP (server generates short IDs, stores the mapping). It's the simplest, most reliable, and collision-free. The mapping table is tiny (one row per transfer). The lookup is fast.

### Backward Compatibility

Old URLs (`send.sgraph.ai/#/download/...`) must continue to work. The new view.sgraph.ai URLs are additional, not replacements. Both URL schemes resolve to the same encrypted blob.

## The .gs-vault Folder Convention

### The Idea

Instead of scattered metadata folders (`.bundles/`, `.executions/`, `.shares/`), introduce a single `.gs-vault/` folder per vault that contains all SG/Send-specific metadata:

```
my-vault/
  .gs-vault/
    bundles/           <- LLM execution bundles
    shares/            <- Share links (short ID + access key per shared file)
    executions/        <- Raw execution history
    config.json        <- Vault configuration
  documents/
  images/
  ...
```

The `.gs-vault/` folder is the "system folder" — it contains metadata about vault operations, not user content. Tools and viewers know to look here for share links, bundles, and configuration.

### Share Links in .gs-vault

When a file is shared, the share metadata is saved:

```json
{
  "file": "documents/welcome-email.eml",
  "short_id": "Kx7mQ",
  "transfer_id": "full-transfer-id-here",
  "decryption_key": "apple-mango-56",
  "created": "2026-03-06T10:00:00Z",
  "max_downloads": 5,
  "downloads": 2,
  "viewer": "pdf"
}
```

This creates a record of everything that's been shared, with what settings, and how many times it's been accessed. The human can see: "I shared this file 3 days ago, it's been downloaded twice, 3 downloads remaining."

## Threat Model: User-Controlled IDs and Keys

### What the User Controls Today

- **Transfer ID:** Server-generated (user cannot choose)
- **Decryption key:** Client-generated (user cannot choose — it's random)

### What We'd Like the User to Control

- **Decryption key:** Allow the user to provide a friendly key instead of a random one
- **Transfer ID:** Potentially allow the user to suggest a short ID (vanity URLs)

### Threat Analysis

| Threat | User-controlled decryption key | User-controlled transfer ID |
|---|---|---|
| **Weak key chosen** | Risk: user picks "password123" — brute-forceable | N/A |
| **Key collision** | No risk — key is per-transfer, not global | N/A |
| **ID collision** | N/A | Risk: two users pick the same short ID |
| **ID enumeration** | N/A | Risk: attacker tries common short IDs to find files |
| **Denial of service** | N/A | Risk: attacker claims all short IDs |

**Recommendation:**

- **Decryption key: Allow user control.** The key never goes to the server. A weak key only hurts the user who chose it. Offer friendly key generation as default, with the option to type a custom key. Show a strength indicator.
- **Transfer ID: Server-controlled with optional vanity prefix.** The server generates the short ID but can accept a user-suggested prefix (e.g., `acme-` → `acme-Kx7mQ`). The server ensures uniqueness. Don't allow fully user-controlled IDs (enumeration and collision risks).
