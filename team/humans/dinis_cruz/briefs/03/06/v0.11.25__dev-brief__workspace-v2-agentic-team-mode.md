## Context: What v0.1 Proved

Yesterday's workspace session confirmed the core workflow works powerfully: load HTML, write/generate JavaScript transformations, run in sandbox, iterate with LLM assistance, save results to vault. The CLI sync is already useful — make changes in the browser, `sg-send pull` on the local machine, and the files are there. The resize UI took effort but landed well. Bundle save/load is working.

The workspace is now ready for v0.2: expanding from a single-purpose HTML transformation tool into a multi-mode workbench with attachments, agent workflows, and context management.

## Bug Fixes and UX Improvements (P0)

### Bundle Renaming

Bundles currently get auto-generated titles based on the prompt. The user needs to rename them to meaningful names ("Remove negative articles v2", "Grid layout fix attempt"). This is a simple inline-edit on the bundle title in the execution history list.

### Attachment Panel

Images can already be pasted (clipboard → base64 → multi-modal prompt). But the current UI shows pasted images in a small inline preview. This needs to become a proper panel.

**Add a new horizontal section** in the chat zone (same pattern as the system prompt and user prompt sections): an **Attachments** panel that shows all attached context:

- Pasted images (with preview, remove button)
- Attached files (markdown, text, JSON — loaded from vault)
- Attached URLs (fetched content — see URL-as-File below)
- Clear indication of what will be sent with the next request
- Token count estimate per attachment

This panel is where memory, skills, and context files are managed — not mixed into the prompt text.

## New Mode: Agentic Team Workflow

### The Idea

The workspace currently has one mode: HTML transformation (Source → Script → Result with LLM assistance). The next mode is **Agentic Team** — where the user talks to individual agent roles (Sherpa, Ambassador, Librarian, etc.) with full product context.

### The Use Case

The human manages customers daily. The workflow today:
1. Open a text file on the laptop with message snippets and templates
2. Manually compose an email or LinkedIn message
3. No context about what was sent before, what the customer has done, what version of the product they're using

What it should be:
1. Open the workspace in Agentic Team mode
2. Select the Sherpa role
3. Say: "Here's the email I want to send to [customer]. What do you think?"
4. The Sherpa responds with the full context of: the product's current state, the customer communication history, the role definition, and the skills/memory files
5. Or: "Hey Ambassador, draft a LinkedIn post about the new data room feature"
6. Or even: sequential — "Sherpa, draft the email. Ambassador, review the tone. Librarian, check for accuracy."

### How It Works

The Agentic Team mode swaps the workspace layout:

| HTML Transformation Mode | Agentic Team Mode |
|---|---|
| Source panel (HTML) | Context panel (role definitions, memory files) |
| Script panel (JS) | Conversation panel (current interaction) |
| Result panel (transformed HTML) | Output panel (drafted messages, reviews) |
| Chat panel (LLM with HTML context) | Chat panel (LLM with team role context) |

The key difference: instead of attaching Source HTML + Script + Result, the attachments are **role definitions + memory files + customer history + product context**.

### Attachments as Memory and Skills

In Agentic Team mode, the attachments panel becomes the **context loader**:

- **Role definition:** Load the Sherpa's ROLE.md (from vault or from URL)
- **Memory files:** A folder of markdown files containing: customer list, previous messages sent, product current state, pricing info, feature status
- **Skills:** The specific skill files that guide the agent (e.g., "how to write an onboarding email", "how to respond to a feature request")
- **Customer folder:** Per-customer context (what they bought, when, what messages they received)

The user selects which role and which context files to include. The LLM receives all of it as a one-shot request (no history — the one-shot principle from yesterday's brief still applies). The context IS the memory.

### Folder-as-Environment

Each folder in the vault can define its own mode. When you open a folder:

- If it contains HTML + JS files → HTML Transformation mode activates
- If it contains ROLE.md + memory files → Agentic Team mode activates
- If it contains a `manifest.json` → the manifest defines the mode and layout

This is the plugin model: each folder IS a plugin instance. The workspace reads the folder contents and switches to the appropriate mode.

### Sequential Agent Calls

The UI should support calling multiple agents in sequence:

1. Draft: "Sherpa, write the welcome email for [customer]" → Sherpa drafts
2. Review: "Ambassador, review this email for tone and positioning" → Ambassador reviews
3. Check: "Librarian, verify the feature claims are accurate" → Librarian fact-checks

Each call is a separate one-shot request. The output of step 1 becomes an attachment for step 2. The output of step 2 becomes an attachment for step 3.

**Future:** A visual orchestration UI where you drag agents into a sequence and the workflow runs automatically. For now: manual sequential execution with the ability to copy output from one call into the attachments of the next.

## URL-as-File: Live References in the Vault

### The Concept

A file in the vault can be a **URL reference** instead of static content. When opened, the workspace fetches the URL and displays the content. When refreshed, the latest version is fetched.

```json
{
  "type": "url-reference",
  "url": "https://sgraph.ai/docs/roles/sherpa",
  "fetched_at": "2026-03-06T08:00:00Z",
  "content_hash": "sha256_of_last_fetch"
}
```

### Why This Matters

**Role definitions stay in sync.** Instead of copying ROLE__Sherpa.md into every vault, the vault contains a URL reference to the live version on the docs site (or on GitHub). When the role definition is updated in the repo, a refresh in the workspace picks up the changes.

**Morning sync workflow:** The human arrives in the morning, clicks "Sync Team" — the workspace fetches all URL-referenced files, compares with the previous version (content hash), and shows what changed. The vault now contains a versioned snapshot of what the agents knew at that moment in time.

**Bundle integrity preserved:** When a bundle is saved, the URL-referenced files are saved as content (not as URLs). This means the bundle captures exactly what was used at that moment — even if the URL content changes later. References for live use, snapshots for history.

### Docs Site as Source of Truth

The role definitions and project context should live on a static docs page (e.g., `sgraph.ai/docs/`) that:
- Has CORS support (so the workspace can fetch it)
- Contains the latest version of each role definition
- Contains the latest project state summary
- Is maintained by the Librarian (who already maintains the briefing packs)

This eliminates the fork between: role definitions in the repo, role definitions in the vault, and role definitions in reality. One source, multiple consumers.

## Multilingual Materials Management

### The Problem

The human communicates with customers in multiple languages. The current workflow: write in English, manually translate (or use an LLM), send. No version control. No way to know if the Portuguese version matches the current English version.

### The Solution

For the Agentic Team mode, materials are stored in a structured multilingual format:

```
customer-communications/
  welcome-email/
    en.md          ← Master (English)
    pt-BR.md       ← Brazilian Portuguese
    es-AR.md       ← Argentinian Spanish
    hr.md          ← Croatian
    sync.json      ← Which translations are in sync with the master
```

`sync.json`:
```json
{
  "master": "en.md",
  "master_hash": "sha256_of_current_en_version",
  "translations": {
    "pt-BR.md": { "hash": "sha256", "synced_with_master_hash": "sha256_of_en_at_translation_time", "in_sync": true },
    "es-AR.md": { "hash": "sha256", "synced_with_master_hash": "older_sha256", "in_sync": false },
    "hr.md": { "hash": "sha256", "synced_with_master_hash": "sha256_of_en_at_translation_time", "in_sync": true }
  }
}
```

When the English master is edited, `sync.json` automatically marks all translations as out-of-sync. The UI shows: "Spanish (Argentinian) is out of date — the English master was updated." One click to re-translate via LLM (using the translation hash caching from the v0.11.1 brief — only changed sections are re-translated).

### Connected Bundles

Bundles can reference other bundles across languages. The English draft bundle and the Portuguese translation bundle are linked — changing one flags the other as potentially stale. This is the beginning of cross-bundle relationships.
