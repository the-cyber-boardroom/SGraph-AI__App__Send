Here's an uncomfortable truth: if your team is spending significant money on LLM API calls, it almost certainly means you're not managing context properly. You're not bad engineers. You just haven't built the infrastructure to be efficient ones — and the LLM providers have no incentive to help you fix that.

## The Exponential Growth Problem

Every developer who has tried to build a production LLM application has hit this wall. A conversation starts at 500 tokens. The response adds 1,000. The next request includes the full history: 1,500 in, plus the new prompt. Then 3,000. Then 4,500. Each request is small, but the accumulated context grows exponentially.

This is the fundamental catch-22 of LLM conversations: the only way to maintain context is to send the full history. But sending the full history means every request carries the weight of every previous request. By request number 10, you're sending 50,000 tokens — and most of that context is irrelevant to the current question.

The providers hide this in their consumer products. ChatGPT and Claude optimise behind the scenes — caching, compression, context management. But when you call the API, you're on your own. Every token you send, you pay for. And there's a perverse incentive: the providers make money when you don't optimise. It's not malicious — it's just market economics. Their consumer product is optimised for their cost. Your API usage is optimised for their revenue.

## The Agentic Explosion

The problem gets worse with agentic workflows. The "just figure it out" pattern — give an agent a goal and let it iterate until it works — burns through tokens at an astonishing rate. Each iteration sends the full context plus the new attempt. The agent doesn't know when to stop, because there's always something to improve. Left unchecked, a single agentic task can consume millions of tokens.

And here's what I've learned: the result often works, but it's held together with chewing gum. The tests might be hacked. The code might not be well-engineered. The agent solved the problem by brute force, not by understanding it. When I look at "failed" LLM interactions, most of the time the model didn't hallucinate — it responded accurately to a brief that had gaps. The problem wasn't the model. It was the input.

## The Principle: Every Request Should Be One-Shot

After months of building LLM-powered workflows in production, I've arrived at a principle that goes against the conventional wisdom: **every LLM request should be a one-shot request. No history. No accumulated context.**

This sounds extreme. How do you maintain continuity without history? The answer: you don't maintain continuity through the LLM. You maintain it through your engineering.

Instead of sending the LLM a conversation with 10 previous messages, you send it exactly the context it needs for this specific request — and nothing else. You curate the context. You control what goes in. You strip out everything irrelevant. The LLM gets a focused, minimal prompt and returns a focused, minimal response.

This requires more work upfront. You need to think about what the LLM actually needs to see. You need to build infrastructure to manage and curate context. But the payoff is dramatic: 10x-100x reduction in token usage, faster responses, better results (because the LLM's attention isn't diluted across irrelevant history), and predictable costs.

## Case Study 1: HTML Transformations — Use LLMs to Not Use LLMs

I needed to transform web pages: clean up HTML, remove ads, classify content, translate sections. The naive approach: send the full HTML (30,000-60,000 tokens) to an LLM with instructions, get modified HTML back.

This was slow, expensive, and most models couldn't even do it. I tested across OpenAI, Google, Anthropic, and others. Most models lost content, broke structure, or hallucinated tags when asked to return complete modified HTML. The ones that worked cost $0.15 per request. Ten iterations to get one transformation right: $1.50 for a single page.

The solution: **use the LLM to write the code, not to do the transformation.** Instead of HTML → LLM → HTML, I switched to HTML → LLM → JavaScript → HTML. The LLM writes a transformation script. The script runs locally in the browser against the HTML. The transformation executes instantly, deterministically, at zero token cost after the initial code generation.

Now instead of sending 60K tokens per transformation, I send the HTML structure once (maybe 5K tokens as a skeleton), ask the LLM to write a JavaScript function (maybe 2K tokens back), and that function runs against any page on that site, forever, for free. The LLM is the author of the transformation, not the runtime.

## Case Study 2: Managing 18 Agents on $200/Month

I run a team of 18 AI agents — each with a defined role, responsibilities, and communication style. Explorer, Developer, Architect, Librarian, Sherpa, Ambassador, Journalist, Accountant, and more. They produce briefs, review code, draft communications, analyse architecture, and track costs.

I do this within Claude's $200/month Max plan. Not $2,000. Not $20,000. Two hundred dollars.

How? By treating agent management like project management.

Not every agent runs every day. I create micro-teams daily — three or four agents focused on a specific task. The others aren't invoked. This alone eliminates most of the token waste that "run all agents on everything" approaches produce.

I communicate with agents through briefs — structured markdown documents that contain everything the agent needs. By the time a brief reaches the team, it's already well-processed. The agent doesn't need to "discover" the problem. It just executes. Sometimes there's barely any new code to write because the brief already contains the architecture.

And I watch the costs obsessively. I've added an Accountant role specifically to track externalities — when one agent delegates work to another agent unnecessarily, when an agent produces output that could have been achieved with fewer tokens, when a model is being used for a task that a cheaper model could handle equally well.

The agents that create the most value are often the ones that prevent token waste: the Librarian who maintains context so agents don't have to re-discover it, the Architect who provides structure so the Developer doesn't iterate blindly, and the Accountant who flags when costs are creeping up.

## The Wardley Map of LLM Usage

Think of LLM usage through the lens of Wardley Maps — the evolution from Genesis through Custom Built to Product to Commodity.

Most teams stop at Genesis. They get something working — an agentic workflow that produces results — and they call it done. "It works!" Yes, but it works like a prototype: expensive, fragile, and inefficient.

The Custom Built phase is where you take that working prototype and engineer it properly: manage context, cache results, build abstraction layers, find the minimum viable prompt. This is where the 10x cost reduction happens.

The Product phase is where you hand it to the "town planners" — the people who optimise relentlessly. Hash-based caching (only process content that's changed). Model routing (use the cheapest model that produces acceptable results for each task type). Skeleton prompts (send structure, not content). Pre-computed context (site descriptions instead of raw HTML).

Most teams never get past Genesis because the LLM made it "work" and they moved on. The money they're spending isn't the cost of LLMs. It's the cost of not engineering their LLM usage.

## The Uncomfortable Conclusion

If you're spending a lot of money on LLMs, you have an engineering problem. Not a model problem. Not a provider problem. An engineering problem.

The fix isn't a better model. It isn't a cheaper provider. It's infrastructure: context management, one-shot prompts, caching, abstraction layers, model routing, cost tracking, and the discipline to treat every token as a resource that must be justified.

Build the tools that make your LLM usage efficient. Use LLMs to build those tools. Then use the tools to reduce how much you need LLMs. That's the virtuous cycle. That's how you go from spending thousands on LLM APIs to spending two hundred dollars a month and getting more done.
