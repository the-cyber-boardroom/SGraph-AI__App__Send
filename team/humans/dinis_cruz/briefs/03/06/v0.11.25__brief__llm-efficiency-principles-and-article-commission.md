## The Principles (Team Doctrine)

### Principle 1: One-Shot Requests — No Accumulated History

Every LLM API request should be self-contained. No conversation history. No accumulated messages. The request contains exactly the context needed for this specific task — nothing more.

**Why:** Conversation history grows exponentially. Request #10 carries the token weight of requests #1-9. Most of that context is irrelevant to the current question. One-shot eliminates this entirely.

**How we enforce it:** The workspace INCLUDE checkboxes (Src, Data, Script, Result, Console) make context selection explicit. Nothing is included by default. The user chooses what to send. The token cost is visible before sending.

### Principle 2: Minimum Viable Context

For every request, ask: **what is the smallest amount of context the LLM needs to succeed?** Then send only that.

**Examples from our practice:**
- Instead of sending 94KB of full HTML (18K tokens), send a 5KB structural skeleton (1K tokens)
- Instead of sending the entire script + HTML + result, send only the script and a specific question
- Instead of sending raw content, send a pre-computed site description (150 tokens)
- Instead of sending full translation files, send only the changed segments (hash-based delta)

**The meta-goal:** Use the workspace to discover the minimum viable context for each task type, then encode that discovery as a reusable pattern.

### Principle 3: Use LLMs to Not Use LLMs

The LLM's job is to create the code, the structure, the abstraction layers — not to be the runtime. Once the LLM has written the transformation script, the script runs locally at zero token cost forever.

**The progression:**
1. LLM does everything (expensive, slow, unreliable)
2. LLM writes code that does everything (one-time cost, then free)
3. LLM writes better code that's more efficient (optimisation)
4. No LLM needed at all for this task (the code is self-sufficient)

Each step reduces LLM dependency. The goal is to reach step 4 for as many tasks as possible, using the LLM only for genuinely novel problems.

### Principle 4: Briefs Before Execution

Communicate with agents through structured briefs, not open-ended conversations. By the time a brief reaches the team, it should contain:
- The problem definition
- The constraints
- The approach (or options)
- The acceptance criteria
- The relevant context (references, not full content)

This means the agent doesn't need to "discover" the problem through conversation. It reads the brief and executes. Sometimes there's barely any new code to write because the brief already contains the architecture. This is not bureaucracy — it is project management.

### Principle 5: Not Every Agent Runs Every Day

Having 18 agent roles doesn't mean 18 agents run on every task. Create micro-teams daily: 3-4 agents focused on the specific task. The others are not invoked.

**The anti-pattern:** "All agents process all briefs." This is the equivalent of inviting the entire company to every meeting.

**The pattern:** The human decides which agents are relevant, creates a focused squad, and only those agents consume tokens. The Librarian maintains the briefs so any agent can be activated later with full context — they don't need to have been "in the room" for every previous conversation.

### Principle 6: Track Externalities (Accountant's Job)

Agents will optimise locally but create global inefficiency. An agent might delegate work to three other agents (10K tokens each = 30K total) instead of doing it itself (15K tokens). The Accountant role exists to catch this:

- Track token cost per agent per task
- Flag when delegation increases total cost
- Identify which models are being used for which tasks (is an expensive multi-modal model being used for a text-only task?)
- Report on free models that produce equivalent results for specific task types
- Provide a weekly "cost optimisation" review

### Principle 7: Wardley Map Evolution — Don't Stop at Genesis

Most teams stop when it works. That's Genesis. The evolution path:

| Stage | What Happens | Token Impact |
|---|---|---|
| **Genesis** | LLM does the task end-to-end (agentic, iterative) | Very high — fire and forget, millions of tokens |
| **Custom Built** | Engineer the workflow: manage context, cache results, build abstractions | 10x reduction — structured prompts, one-shot |
| **Product** | Hand to town planners: hash caching, model routing, skeleton prompts, pre-computed context | 100x reduction — LLM only for genuinely novel content |
| **Commodity** | No LLM needed — the generated code/scripts/structures handle it | Zero tokens for this task |

**Our practice:** The HTML transformation workflow has already gone from Genesis (send full HTML to LLM) through Custom Built (LLM writes JS, JS runs locally) to Product (skeleton prompts, hash caching for classification). Some tasks are at Commodity (the generated scripts run without any LLM involvement).

## For the Accountant: Cost Tracking Requirements

The Accountant needs to capture and report on:

| Metric | How to Capture |
|---|---|
| Token cost per agent per session | Workspace already logs this in bundle metadata |
| Token cost per task type (HTML transform vs text draft vs code gen) | Tag bundles by task type |
| Model used per task (was the right model used?) | Bundle metadata includes model |
| Delegation chains (which agent invoked which other agent) | Event bus logs in workspace |
| Free model equivalence testing | A/B test same prompt on paid vs free models, compare quality |
| Weekly cost trend | Aggregate bundle metadata across sessions |

**The goal:** A weekly report that says: "This week we used X tokens across Y sessions. The most expensive task type was Z. We could save A% by switching B tasks from [expensive model] to [cheaper model]. The Librarian's context maintenance saved an estimated C tokens by preventing re-discovery."

## Commission: Companion Article

### What We Need

The Journalist, Historian, and Cartographer should collaborate on a companion article that provides **concrete technical evidence** for the claims in the standalone article.

### Source Material

Go through the development history of the last few months — all the briefs, all the session transcripts, all the decisions. Look for:

**Token usage data:**
- The HTML transformation workflow: how many tokens did the "send full HTML to LLM" approach use vs the "LLM writes JS" approach? The workspace bundle metadata has this.
- The translation system: how many tokens were saved by hash-based caching (only translating changed segments)?
- The agent team sessions: what was the actual token usage per session? How does it compare to what it would have been with full conversation history?

**Cost comparisons:**
- What did specific tasks cost at Genesis stage vs after optimisation?
- Which models were used for which tasks, and what was the cost difference?
- What is the actual monthly spend, and how does it compare to the output volume?

**Architecture decisions that reduced LLM dependency:**
- The JavaScript transformation pipeline (LLM-as-author vs LLM-as-runtime)
- The hash caching system (classification results cached per content hash)
- The skeleton prompt approach (sending structure, not content)
- The one-shot principle (explicit context selection vs accumulated history)
- The briefing pack system (structured briefs vs open-ended conversations)

**Wardley Map examples:**
- Map the evolution of specific workflows from Genesis to their current stage
- Show which workflows have reached Commodity (no LLM needed)
- Identify which workflows are still at Genesis and what the optimisation path looks like

### Format

A technical blog post / case study. 2,000-3,000 words. With actual numbers (token counts, costs, time savings). With before/after comparisons. With architecture diagrams showing the evolution.

The article should be publishable on sgraph.ai/blog alongside the standalone article, and shareable on LinkedIn and developer communities.

### Tone

Evidence-based, not preachy. "Here's what we measured. Here's what we changed. Here's the result." Let the data make the argument. The standalone article makes the claim; the companion article proves it.
