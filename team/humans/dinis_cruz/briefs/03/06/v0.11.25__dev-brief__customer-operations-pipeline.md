## Context: Why Now

We have real customers. Real purchases. Real support interactions. The human currently reacts within the hour to every Stripe purchase — manually creating access tokens, manually composing thank-you messages, manually tracking who received what. This works at 10-15 customers. It does not work at 50 or 100.

But automating too early is dangerous. The manual process has taught us what the workflow actually needs to be. Now is the right time to build — not from theory, but from lived experience.

**Strategic principle:** Every interaction with users should itself leverage SG/Send. We should be the heaviest users of our own platform. We should be creating tonnes of SG/Send usage — both to test the technology and to demonstrate workflows by example.

## Competitive Intel: What Customers Use Today

From a real customer conversation:

| Tool | What They Use It For |
|---|---|
| **1Password** | Internal secrets sharing (team passwords, API keys) |
| **DocuSign** | External document signing and sharing |
| **DocSend** | External document sharing with analytics (who opened, how long they viewed) |
| **1Password** | Some external sharing too (sharing credentials with partners) |

**What this tells us:** The market is fragmented. Customers use 3-4 separate tools for what SG/Send could do in one: share secrets (1Password → SG/Send one-time tokens), share documents with tracking (DocSend → SG/Send data rooms with analytics), and share securely with external parties (DocuSign → SG/Send with PKI). Each tool is a potential displacement target.

**DocSend in particular** is a direct competitor for the data room use case. DocSend's core value: share a document, get analytics on who opened it and how long they viewed each page. SG/Send already has the sharing part. The analytics (download count, timestamps) are being built. The viewing analytics (per-page time) are a future feature. Positioning: "Like DocSend, but encrypted end-to-end and pay-per-use instead of $10/month."

## The Email Template Workflow

### The Insight

The workspace already has the four-element transformation model: Source HTML + Data + Script → Output HTML. This is exactly what email template rendering needs:

| Workspace Element | Email Use |
|---|---|
| **Source HTML** | Email template ("Thanks for purchasing, {name}!") |
| **Data (JSON)** | Customer data (name, email, access token, purchase amount, language) |
| **Script (JS)** | Template rendering logic (replace placeholders, apply translations, format dates) |
| **Output HTML** | The final email, ready to send |

The email pipeline is not a new feature — it is a use case of the existing workspace, with an email-specific viewer/plugin on top.

### The Flow

```
1. Stripe webhook fires → "Customer X purchased Y"
2. Customer data extracted: name, email, amount, product, language
3. Data saved as JSON in customer's vault folder
4. Email template loaded (from vault — multilingual variants available)
5. Transformation script runs: template + data → final email HTML
6. Email saved as .eml file in customer's folder (creation decoupled from sending)
7. Human reviews the email (or it auto-sends if template is approved)
8. Email sent via Sherpa inbox / SG WorkMail
9. Send record saved to customer's folder (audit trail)
```

### Why Decouple Creation from Sending

The email creation step (template + data → HTML) should be separate from the email sending step. This is important for:

- **Review:** The human can check the email before it goes out
- **Prompt injection safety:** The LLM is not creating and sending an email in one step — the email is a static file that gets reviewed
- **Outbox pattern:** Emails queue in an "outbox" folder. A separate process sends what's in the outbox. This is safer and more auditable than real-time sending.
- **Multilingual:** The same template + different translation data = emails in multiple languages. All created and reviewable before sending.

### Dynamic Content from Agent Roles

Each email can include content from multiple agents:

- **Sherpa:** Personalised onboarding guidance ("Try dropping a PDF first")
- **Journalist:** Latest feature update ("We just added folder sharing!")
- **Ambassador:** Positioning message ("Here's how SG/Send compares to WeTransfer")

These content blocks are loaded as additional data files. The transformation script assembles them. This means the email content evolves daily without changing the template — the agents just update their data files.

## Per-Customer CRM Folders

### The Structure

Each customer gets a folder in the vault:

```
customers/
  customer-abc123/
    profile.json           <- Name, email, purchase date, plan, language
    access-tokens/
      token-001.json       <- Token value, created date, credits, status
    communications/
      2026-03-01_welcome.eml
      2026-03-03_feature-update.eml
      2026-03-06_nudge.eml
    usage/
      usage-log.json       <- What they've uploaded, downloaded, when
    notes/
      notes.md             <- Human's notes about this customer
```

**All encrypted.** Even admin-side, customer data is stored in the vault. We practice what we preach — customer PII never exists in plaintext on the server.

### What This Enables

- **Communication history:** See every message sent to every customer. Never send a duplicate.
- **Usage tracking:** See what the customer has done (uploads, downloads, credits remaining). Know who to nudge and who's already active.
- **Notes:** The human's context about each customer. "Mentioned they use 1Password and DocSend" — captured, not forgotten.
- **Agent context:** When the Sherpa drafts an email for a customer, it can read that customer's folder and personalise: "You've used 3 of your 500 credits — here are some things to try."

## One-Time Tokens and N-Time Access

### One-Time Password Links

When sharing an access token with a customer, the token should be delivered as a **one-time viewable link** — not as plaintext in an email body.

**The flow:**

1. Access token is encrypted and uploaded to SG/Send
2. A link is generated: `view.sgraph.ai/secret/{id}#{key}`
3. The link is included in the welcome email
4. Customer clicks the link → sees the access token → link is consumed
5. After viewing, the encrypted blob is deleted (or marked as consumed)
6. If someone intercepts the email and clicks the link later → it's already consumed, they see nothing

### N-Time Access

One-time is the most extreme case. More generally, files should support N-time access:

| N | Use Case |
|---|---|
| 1 | One-time secret sharing (access tokens, passwords) |
| 5 | Default for most shared files (sender + a few recipients) |
| N (custom) | User specifies the limit |
| Unlimited | No limit (current default) |

**Implementation:** The download counter (already being built per the billing brief) tracks downloads per file. When the count reaches N, the file is either deleted or returns a "this link has expired" message. The sender sets N at upload time.

### Password Manager Integration

When the recipient views a one-time token, the page should make it easy to save to a password manager:

- **Copy button** (already exists)
- **"Save to Password Manager" hint:** A well-structured page with `autocomplete` attributes that password manager browser extensions can detect and offer to save
- **Future:** Explicit integrations with 1Password, LastPass, Google Passwords (via their browser APIs or deep links)

This is particularly relevant given the customer intel — they already use 1Password. Making SG/Send token delivery integrate with 1Password is a direct value proposition.

## Email Viewer (EML Support)

### The Concept

Add an email viewer to the file viewers family (alongside PDF, video, image, markdown, zip). An `.eml` file renders as a formatted email — from, to, subject, date, body, attachments.

**Why this matters:**

- Customer emails saved as .eml files in the CRM folders are viewable in the workspace and in the browser
- The email template workflow outputs .eml files (not just HTML)
- Users can share email files via SG/Send and recipients see a properly rendered email, not raw source
- Connects to the data room use case: a data room containing correspondence (legal, compliance, HR) renders each email properly

### Implementation

EML is a well-defined RFC 2822 format. A JavaScript parser can extract headers (From, To, Subject, Date) and body (text/html multipart). The viewer renders the email with a standard email chrome (header bar with from/to/subject, body below). Attachments within the EML can be extracted and previewed using existing file viewers.

## Multi-Document Delivery

### The Concept

Currently, a shared link delivers one encrypted blob (which can be a zip/folder). The next step: deliver multiple documents in one link, where different documents can have different security levels.

**Example:** A customer welcome package:

| Document | Source | Security |
|---|---|---|
| Welcome HTML template | Public (from repo/website) | None — it's a public template |
| Customer data (name, token) | Private (per-customer) | Encrypted, one-time |
| Transformation script | Public (standard template renderer) | None — it's public code |

The link passes three IDs. The client loads all three, runs the script (which combines template + data), and renders the final page. The server never sees the assembled result.

**Why this matters:**

- The HTML template can be cached (it's the same for every customer) — only the customer data changes
- The template can be in a public repo. The customer data is encrypted. The client assembles them.
- This is more efficient than encrypting the entire assembled email for each customer
- Connects to the privacy-preserving content assembly brief (v0.11.1): page skeleton + content fragments assembled client-side

### The Data Injection Pattern

Already implemented in the workspace: a JavaScript object `__sg_send_data__` injected into the DOM before the script runs. The script reads from this object to get customer data, translations, and other dynamic content.

This same pattern works for multi-document delivery: the link loads the HTML, injects the data object, runs the script, and the page renders with personalised content — all client-side.
