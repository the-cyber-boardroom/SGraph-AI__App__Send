## Part 1a: Milestone — Users Sending 1GB Files, Limit Increase to 5GB

A customer just uploaded a ~1GB file through the browser. It completed in 30.86 seconds (read: 2.8s, encrypt: 14.2s, prepare: 0.9s, send: 11.8s, finalise: 1.2s). This is the largest confirmed real-user upload to date.

**Performance breakdown:**

| Step | Time | % of Total | Notes |
|---|---|---|---|
| Reading file | 2,826ms | 9% | Browser reading from external disk |
| Encrypting | 14,178ms | 46% | AES-256-GCM via Web Crypto API — the bottleneck |
| Preparing transfer | 873ms | 3% | Minimal |
| Sending (upload) | 11,774ms | 38% | Network transfer to S3 |
| Finalising | 1,205ms | 4% | Server-side confirmation |

Encryption at 46% of total time is the dominant cost. Worth benchmarking across devices (older phones, low-spec laptops) to understand the range. On modern hardware, ~70MB/s encryption throughput is reasonable for Web Crypto.

### Decision: Increase Upload Limit to 5GB

**Effective immediately.** The current limit is conservative. Real users are already at 1GB with no issues. S3 billing shows no significant cost impact. At our current user volume, even heavy usage at 5GB per upload is well within budget.

**Why 5GB:**
- Covers the vast majority of consumer use cases (video files, photo folders, project archives)
- Matches the user feedback price point (£5 = 5GB at £1/GB — "buy 5GB and use it all in one upload" should be possible)
- Still within browser memory limits for the encryption step (5GB is manageable on modern devices with 8GB+ RAM)
- Competitive: WeTransfer free is 2GB (we're 2.5x that on paid), WeTransfer Ultimate is 200GB (we'll get there later)

**What to monitor:**
- S3 storage costs as users send larger files
- CloudFront egress costs when large files are downloaded multiple times
- Browser memory pressure on lower-spec devices at 5GB
- Encryption time at 5GB (extrapolating: ~70 seconds for encryption alone — needs a progress indicator)

**Action for Developer (P0):** Update the upload limit to 5GB. Ensure the progress UI handles long encryption times gracefully (the existing progress bars shown in the customer's screenshot are good — just need to work at 5GB scale).

## Part 2: Token Bug — "sg-send__[redacted]__8ab9f" Fails, "owasp" Works

### The Incident

A Portuguese user (Customer PT-2) received their access token via WhatsApp: `sg-send__[redacted]__8ab9f`. They entered it on send.sgraph.ai/pt-pt. Error: "Token não encontrado" (Token not found). The human sent a simpler token: `owasp`. That worked immediately. The user is on Chrome.

### Why This Matters

The human tested `sg-send__[redacted]__8ab9f` on their own laptop and it worked. Same token, different machine, different result. This is a client-side issue — something about the user's browser environment, the pt-PT locale page, or the way the token was entered.

### Possible Causes

| Hypothesis | Why It Could Happen |
|---|---|
| **Invisible characters from WhatsApp** | WhatsApp sometimes adds zero-width spaces or changes quotes when copying text. If the user copied the token from WhatsApp, invisible characters may have been added. |
| **Smart quotes** | The user mentioned trying "com e sem aspas" (with and without quotes). If WhatsApp converted straight quotes to curly quotes, and Customer PT-2 included them, the token would fail. |
| **Locale-specific character handling** | The pt-PT page might handle underscores or hyphens differently in the token input. The double underscore (`__`) is unusual and could be parsed incorrectly by some input sanitisation. |
| **localStorage encoding issue** | The token is stored in localStorage. If the pt-PT page uses a different localStorage key or encoding than the main page, the token might not be found. |
| **Copy-paste whitespace** | Leading or trailing spaces copied from WhatsApp. The token input should trim whitespace. |

### Investigation Needed

This bug is impossible to debug without seeing the user's browser state. The human had no way to help beyond "try this other token." This is the problem the debugging infrastructure below addresses.

## Part 4: Debugging Infrastructure — Making Token Issues Diagnosable

### The Problem We Just Experienced

A customer had a token that didn't work. The human had no way to diagnose it remotely. The only options were: try a different token (which worked) and move on. The root cause is unknown. This will happen again.

### What We Need

#### 1. Token via URL

Allow setting the access token via the URL hash fragment:

```
https://send.sgraph.ai/pt-pt#access-token=sg-send__[redacted]__8ab9f
```

The page reads the hash fragment, sets the token in localStorage, and removes it from the URL. This eliminates copy-paste issues entirely — the user clicks a link and the token is set.

**Security:** The hash fragment is never sent to the server. Same model as the decryption key. The token is in the URL momentarily, then stored in localStorage and cleared from the URL bar.

#### 2. Token via Encrypted SG/Send Link

Create a special "set token" viewer: the access token is encrypted and uploaded to SG/Send. The link opens a page that:
1. Decrypts the token
2. Saves it to localStorage
3. Shows a confirmation: "Your access token has been set. You can now upload files."
4. Redirects to the send page

This is the one-time token delivery from the customer operations brief, applied to access tokens specifically.

#### 3. Debug Mode

A diagnostic page at `send.sgraph.ai/debug` (or activated by a URL parameter `?debug=true`) that:

**Captures and displays:**
- Current localStorage contents (token present? what value? any invisible characters?)
- Browser information (Chrome version, OS, locale)
- Network status (can the browser reach the API? what response codes?)
- JavaScript errors (any console errors during token validation?)
- Token validation result (why exactly did it fail? "not found" vs "invalid format" vs "network error")

**Provides guidance:**
- Step-by-step instructions with screenshots for opening Chrome DevTools
- "Copy diagnostics" button that packages all debug info into a single text block the user can paste back to the human via WhatsApp
- Clear explanation of what each piece of information means

**Visual error details:**
- Instead of just "Token não encontrado", show: "Token 'sg-send__bru...' was not found. This could be because: (a) the token was entered incorrectly, (b) there are invisible characters in the token, (c) the token hasn't been activated yet. Here's what we detected: [show the raw bytes of what was entered]."

#### 4. Token Input Sanitisation

Regardless of debug mode, the token input should:
- Trim leading/trailing whitespace
- Strip zero-width spaces and other Unicode invisible characters
- Strip curly/smart quotes (replace with straight quotes, or just remove quotes entirely)
- Show the token length and a character preview ("24 characters entered") so the user can verify
- If the token contains unexpected characters, show a warning: "Your token contains unusual characters. Did you copy it correctly?"

#### 5. Browser-Based Sherpa (Future)

When the LLM-on-browser capability is ready (via the sg-llm.js module from tools.sgraph.ai), add a support chat on the debug page:

- A Sherpa agent running in the browser (using the user's own API key, or a limited free allocation)
- The Sherpa can see the diagnostic data (localStorage, browser info, errors)
- The user describes their problem in natural language: "I entered my token but it says not found"
- The Sherpa diagnoses: "I can see your token has 26 characters but the expected length is 24. There appear to be two extra invisible characters at the end. Let me clean that up for you."
- The Sherpa fixes it: removes the invisible characters, sets the clean token, confirms it works

This is the ultimate troubleshooting experience: the user talks to an agent that can actually see their browser state and fix the problem. Zero-knowledge is preserved because the Sherpa runs in the browser — no data goes to any server.

### Developer Secrets Management — Use Case from Customer

The same customer (who uses 1Password + DocuSign + DocSend) described a specific workflow:

> "Claude post link to $some_key in Slack for other person/Claude with an invalidation routine once consumed probably"
> "I think these features will be big with Claude users. Service registries and secrets management"

**What they're describing:** An AI agent (Claude Code) generates a secret or API key, uploads it to SG/Send as a one-time link, posts the link in Slack, and the recipient (human or another Claude/LLM session) opens it — at which point the secret is automatically invalidated.

**Why this matters enormously:**

This is not a human-to-human file sharing use case. This is **programmatic secrets management for AI agent workflows.** The customer sees SG/Send as infrastructure for:

| Use Case | How It Works with SG/Send |
|---|---|
| **Agent-to-agent secret sharing** | Claude Code generates a key → uploads as one-time link → posts in Slack → another Claude session consumes it → link invalidated |
| **Service registry secrets** | Deployment pipeline creates credentials → stored as one-time SG/Send links → services consume on first boot → credentials invalidated |
| **Developer key rotation** | New API key generated → shared via one-time link in Slack/Teams → old key revoked → only the person who opened the link has the new key |
| **CI/CD secrets injection** | Build pipeline needs a secret → fetches from SG/Send one-time link → secret is consumed and invalidated → never persists in logs |

**The competitive framing:** This customer sees SG/Send competing not just with WeTransfer (file sharing) and DocSend (document sharing) but with **1Password** and **HashiCorp Vault** for secrets management. The differentiator: one-time consumption + automatic invalidation + zero-knowledge + API-accessible.

**What we need to enable this:**
- One-time / N-time download links (already in this brief as P1)
- API for programmatic upload and link generation (the CLI already does this — needs to be exposed as a proper API)
- Webhook or callback on consumption ("this link was just opened" → trigger next step in workflow)
- Claude Code / MCP integration: make SG/Send available as a tool that Claude Code can call natively

**Action for Ambassador:** This is a positioning goldmine. "Encrypted secrets management for AI agents" is a category that barely exists yet. If Claude Code users can natively share secrets via SG/Send one-time links, that's a distribution channel of potentially millions of developers. Add to the competitive positioning research and the website use cases.

The customer found a critical translation error that changes user comprehension:

| Language | Message Shown | What It Means | Correct? |
|---|---|---|---|
| **en-GB** | "Your file has been encrypted and **uploaded**." | File stored on server, ready to share | ✅ Correct |
| **pt-PT** | "O seu ficheiro foi encriptado e **enviado**." | File has been encrypted and **sent** | ❌ Wrong — "enviado" means sent to someone |

The correct Portuguese is "encriptado e **carregado**" (encrypted and uploaded) or "encriptado e **armazenado**" (encrypted and stored). "Enviado" implies the file was delivered to a recipient, which is not what happened — especially in the "Store only" mode where there is no recipient.

**This is exactly the kind of subtle translation error that AI translations produce.** The English word "uploaded" was translated to a Portuguese word that is technically a valid translation in some contexts but carries completely different implications for the user. The user's mental model shifts from "my file is stored and I control who gets it" to "my file was sent somewhere."

**Action for Developer + Librarian (P0):** Fix this specific translation immediately. Then audit all status messages across all language variants for similar "close but wrong" translations — especially verbs that imply action was taken on behalf of the user (sent, shared, delivered) vs verbs that describe what the user did (uploaded, stored, encrypted).

### Context-Aware Success Messages

The customer's additional feedback: he uploaded a ~1GB file without specifying a recipient, and the success message said "encrypted and sent" — causing confusion about what to do next and what happened to the file.

**The success message should adapt based on what actually happened:**

| Scenario | Message |
|---|---|
| File uploaded, no recipient | "Your file has been encrypted and stored. Share the link below when you're ready." |
| File uploaded, recipient email specified | "Your file has been encrypted and sent to [email]." |
| File uploaded, store-only mode | "Your file has been encrypted. It's stored securely — only someone with the key can access it." |

This is a one-liner change in the i18n strings, but it requires the message template to accept a variable for the scenario. The performance stats display below the message (reading file: 2826ms, encrypting: 14178ms, preparing transfer: 873ms, sending: 11774ms, finalising: 1205ms) is excellent and should be kept — it builds trust by showing exactly what happened.
