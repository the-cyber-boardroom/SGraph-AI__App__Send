# Brief: Interactive Agent Panel — Clone-and-Talk Product Experience

**version** v0.11.17  
**date** 6 Mar 2026  
**from** Human (project lead)  
**to** Ambassador (lead), Sherpa, Librarian, Developer, Designer  
**type** Strategy + dev brief — user onboarding, agent experience, website integration, skill definition  

---

## What Happened

The human shared a prompt with a friend. The prompt told an LLM (Claude Code) to clone the SGraph Send repo, read the team role definitions, and then respond as a multi-agent panel — Sherpa, Ambassador, and Librarian — each in character, responding to a non-technical first-time user named [redacted].

**It worked brilliantly.** The responses were:

- **Sherpa** welcomed the user, set the scene (zero-knowledge encryption explained without jargon), acknowledged the friction ("when you say 'technically heavy language,' that's not casual feedback — that's a friction event with a location, a persona, and an impact"), and handed off to the other agents.
- **Ambassador** diagnosed the problem ("the landing page is currently at Genesis stage on our Wardley Map — it was built to demonstrate the capability, not to acquire users"), identified the ownership ("I drive the positioning brief, the Journalist writes the copy, the Sherpa validates it against users like you"), and named the fix needed ("That page needs to answer: 'This keeps your files safe in a way WeTransfer cannot'").
- **Librarian** kept the team honest ("the landing page is explicitly flagged as Genesis stage in the Explorer team priorities — not a finished product"), confirmed what exists and what doesn't ("a brief at v0.3.2 names the landing page as a design priority... what does NOT exist: any completed copy, user-tested messaging, or Journalist review"), and asked for specifics ("If you're willing to log exactly which phrases stopped you, that becomes the Sherpa's first friction report").

This is not a gimmick. This is the agent team architecture working in production, with a real external user, generating real product feedback, through a cloned repo that contains all the role definitions and context.

## How It Works

### The Workflow

```
1. User receives a prompt (via WhatsApp, email, website, etc.)
2. User pastes the prompt into Claude Code (or any LLM with repo access)
3. The LLM clones the SGraph Send repo (public, read-only)
4. The LLM reads the role definitions:
   - .claude/CLAUDE.md (project context)
   - .claude/explorer/CLAUDE.md (explorer team context)
   - team/roles/sherpa/ROLE__Sherpa.md (Sherpa role definition)
   - team/roles/ambassador/ROLE.md (Ambassador role definition)
   - team/roles/librarian/reviews/ (Librarian's latest review)
5. The LLM now has full context: what SG/Send is, what the team roles are,
   what's been built, what's been decided, what the priorities are
6. The LLM responds as a multi-agent panel: each agent labelled,
   responding in character, with different perspectives
7. The user has a conversation with the team
```

### Why It Works

**The repo IS the context.** The role definitions, the briefs, the architecture decisions, the priorities — they're all in the repo. Cloning the repo gives the LLM everything it needs to respond authentically. No separate knowledge base. No fine-tuning. No custom model. Just structured markdown files in a git repo.

**The role definitions constrain the responses.** The Sherpa knows it owns onboarding and friction. The Ambassador knows it owns positioning and growth. The Librarian knows it tracks what exists and what doesn't. These constraints produce responses that are genuinely different from each other — not three versions of the same generic answer.

**The user's message is the only variable.** Everything else (repo content, role definitions, team context) is pre-loaded. The user just talks. The agents respond from their roles. The conversation feels natural because the LLM is doing what it's good at (responding in character with rich context) and the repo provides the character and context.

### What Makes This Different from a Chatbot

This is not a chatbot. A chatbot has pre-written responses and decision trees. This is:

- **Multi-perspective:** Three agents with different viewpoints respond to the same input
- **Context-rich:** The agents know the actual product architecture, the actual briefs, the actual priorities — because they read them from the repo
- **Honest:** The Librarian saying "the landing page language hasn't been revisited since Genesis stage" is a real fact from the real repo, not a marketing response
- **Actionable:** The Sherpa asking "what specifically tripped you up?" is a real friction data collection mechanism, not a pleasantry

## The Product Opportunities

### 1. Website Integration: "Talk to the Team"

Put a button on sgraph.ai: **"Talk to Our Team"** or **"Ask the SG/Send Team"**

The button links to a page (or opens a panel) that:
- Provides the prompt pre-filled
- Links to Claude Code (or other LLM) with instructions
- Or, if we build it: runs the agent panel inline on the page (using the Anthropic API with the repo context pre-loaded)

**For non-technical users:** "Have questions about SG/Send? Talk to our team. They'll explain it in plain language and help you get started."

**For technical users:** "Want to understand the architecture? Clone our repo and ask the team anything."

**For potential contributors:** "Want to contribute? The team will orient you to the codebase and priorities."

### 2. Onboarding Channel: First-Time User Welcome

When a new user buys credits, the Sherpa sends them a link: "Welcome! If you have any questions, talk to the team: [link to agent panel]." The user asks "how do I share a folder?" and gets an answer from the Sherpa that knows exactly what folder sharing is, how it works, and what the current limitations are — because it read the repo.

### 3. Friction Data Collection

Every conversation with the agent panel is potential product feedback. When the user says "the language feels technically heavy," that's a friction data point. The Sherpa knows to capture it. If we can collect these conversations (with user consent), they become the raw material for:
- Landing page rewrites (what language confuses people?)
- Feature prioritisation (what do people ask about most?)
- Documentation gaps (what questions do the agents struggle to answer?)

### 4. Investor and Partner Demos

Share the prompt with an investor: "Clone our repo and ask the team about our architecture." The investor gets a multi-perspective briefing from agents that know the actual codebase. This is more impressive and more authentic than a slide deck.

### 5. Reusable Pattern for Any Project

This pattern is not specific to SG/Send. Any project with well-structured role definitions and context documents in a repo can create an agent panel. This could itself be a tool/template we share — "How to create a multi-agent team that anyone can talk to."

## The Prompt Template

### Current Working Version

```
You are the SGraph Send Explorer Team — a group of AI agents built around
a real product.

Clone this repo (read-only):
https://github.com/the-cyber-boardroom/SGraph-AI__App__Send

Then read these files in order:
1. .claude/CLAUDE.md
2. .claude/explorer/CLAUDE.md
3. team/roles/sherpa/ROLE__Sherpa.md
4. team/roles/ambassador/ROLE.md  (if it exists)
5. team/roles/librarian/reviews/ (latest file)

You will now run as a multi-agent panel. The active agents are:

- **Sherpa** — guides users through the product, owns onboarding and
  friction
- **Ambassador** — owns growth, user acquisition, and how we talk about
  SG/Send to the world
- **Librarian** — knows everything that exists (and doesn't), keeps the
  team honest

When I ask a question or share an observation, the most relevant agent
responds — and the others can jump in if they have something useful to
add. Each response should be clearly labelled with the agent name.

I'm [NAME]. [BRIEF CONTEXT ABOUT THE USER]. [FIRST QUESTION OR
OBSERVATION].

Start by having the Sherpa welcome me and set the scene, then open it up
to the panel.
```

### Variations for Different Audiences

**Non-technical first-time user:**
```
I'm [Name]. I'm not a developer. I've just tried SGraph Send for the
first time and I have thoughts.
```

**Technical evaluator:**
```
I'm [Name]. I'm a senior developer evaluating SG/Send for my
organisation. I want to understand the architecture and security model.
```

**Potential contributor:**
```
I'm [Name]. I'm interested in contributing to the project. What should
I work on first?
```

**Investor:**
```
I'm [Name]. I'm considering investing. Walk me through the product,
the market, and the roadmap.
```

Each variation triggers different agent behaviours because the Sherpa adapts its welcome, the Ambassador adjusts its positioning, and the Librarian surfaces different context.

## For the Librarian: Supporting This Workflow

### What the Librarian Must Ensure

The agent panel quality depends entirely on the repo content. If the role definitions are stale, the agents give stale answers. If the briefs are missing, the agents can't answer questions about recent features. The Librarian is responsible for:

**1. Role definitions are current.** Every role file (Sherpa, Ambassador, Librarian, and any others) must reflect the current product state. If the product has evolved past what the role file describes, the agent will give outdated answers.

**2. Briefs are in the repo.** The agent panel is only as good as the context in the repo. Key briefs (data room shipped, billing models, Chrome extension, GDPR compliance, tools.sgraph.ai) should be referenced or summarised in a location the agent can find.

**3. A "current state" summary exists.** A single file (e.g., `team/CURRENT_STATE.md`) that captures: what's live, what's being built, what the priorities are, what the known gaps are. This is the first thing agents should read after role definitions. The Librarian maintains this.

**4. The "guest" experience is tested.** Before sharing the prompt with new audiences, the Librarian (or any team member) should run the prompt themselves and verify: Do the agents respond correctly? Is the context current? Are there gaps? This is a QA step for the agent panel.

**5. Friction reports are captured.** When users share feedback through the agent panel, that feedback should flow back into the repo: as friction reports, as brief updates, as role definition refinements. The panel is a feedback loop, not a one-way channel.

### The "Guest" Brief Connection

This connects to the "guest users" concept from the user activation brief (v0.11.08). The agent panel IS a guest experience — the user interacts with the product team without creating an account, without paying, without installing anything. The prompt is the guest pass. The repo is the product tour. The agents are the guides.

The Librarian should create a `team/guests/` folder in the repo for:
- The canonical prompt template (with variants)
- Example conversations (redacted if needed)
- Friction reports collected from agent panel sessions
- Instructions for the team on how to share and when to update

## Skill.md for Agent Panel Creation

### Location: `/mnt/skills/user/agent-panel/SKILL.md`

This skill enables any Claude session to create a multi-agent panel from a repo's role definitions.

```markdown
# Skill: Agent Panel from Repo

## When to Use
Use this skill when the user wants to create an interactive multi-agent
panel where AI agents respond in character based on role definitions
stored in a git repository.

## Prerequisites
- A git repo URL (public or accessible)
- Role definition files in the repo (markdown files defining each
  agent's name, responsibilities, knowledge, and communication style)
- Project context files (what the product is, what's been built,
  current priorities)

## How It Works

1. Clone the repo (read-only)
2. Read role definition files to understand each agent
3. Read project context files to understand the product
4. Present as a multi-agent panel where each agent responds from
   their role's perspective

## Creating the Prompt

The prompt has four parts:

### Part 1: Team Identity
"You are the [Project Name] [Team Name] — a group of AI agents built
around a real product."

### Part 2: Repo and Context Loading
"Clone this repo: [URL]
Then read these files in order:
1. [project context file]
2. [team context file]
3. [role definition files — one per agent]"

### Part 3: Agent Definitions
"You will now run as a multi-agent panel. The active agents are:
- **[Agent 1]** — [one-line responsibility]
- **[Agent 2]** — [one-line responsibility]
- **[Agent 3]** — [one-line responsibility]"

### Part 4: Interaction Rules
"When I ask a question, the most relevant agent responds. Others jump
in if they have something useful to add. Each response is labelled
with the agent name."

### Part 5: User Introduction
"I'm [Name]. [Context]. [First question]."
"Start by having [lead agent] welcome me and set the scene."

## Best Practices

- **3-5 agents maximum.** More than 5 creates confusion.
- **Distinct responsibilities.** Each agent should have a clearly
  different perspective (e.g., user experience vs. technical
  architecture vs. business strategy).
- **Honest agents.** The most powerful moment in the first test session was
  the Librarian saying "this is a known gap that hasn't been
  addressed yet." Agents that are honest about limitations build
  more trust than agents that spin everything positively.
- **Role files should include what the agent DOESN'T know.** This
  prevents agents from making up answers.
- **Keep the repo current.** The agents are only as good as the
  context in the repo. Stale docs = stale answers.
- **Test the prompt before sharing.** Run it yourself first.
  Verify agents respond correctly.

## Variations

- **Technical audience:** Include architecture docs in the reading
  list. Activate the Architect agent.
- **Business audience:** Include market briefs. Activate the
  Ambassador agent.
- **Contributor onboarding:** Include the contribution guide and
  open issues. Activate the Librarian as lead.
- **User support:** Include the user guide and known issues.
  Activate the Sherpa as lead.
```
