## The Big Idea

Translators are both a community we need and a market we serve.

**As a community:** We have 17 language variants that need cultural validation. AI-generated translations are a starting point, not an endpoint. Native speakers who review and fix translations make the product better, drive GitHub forks and PRs, and become advocates.

**As a market:** Translators are a perfect SG/Send use case. They receive files constantly — documents, contracts, manuscripts. They deal with credentials and access tokens from clients. They need data rooms for project files. They're a high-frequency file transfer market that aligns exactly with our product.

**The intersection:** By building tools that help translators contribute to our project, we're simultaneously building relationships with a market that needs our product. The translator who fixes our Spanish translation today sends their first encrypted file tomorrow.

### Strategic Context: Translators and GenAI

The Translator role definition (v0.4.3, 16 Feb 2026) lays out the full vision. The key argument: **the Translator role's job is NOT to replace human translators — it's to make them dramatically more productive.** Instead of translating from scratch, the human reviewer receives a 90%-done AI translation and focuses on the 10% that requires native intuition — tone, cultural fit, naturalness.

The document also articulates the **non-English market hypothesis:** there is likely a bigger market for secure file sharing in non-English-speaking countries than in English-speaking ones. English-speaking markets are saturated (WeTransfer, Dropbox, etc.), while non-English markets are underserved — especially for products that feel culturally native, not translated from English as an afterthought. Every language we add is a market we can credibly enter.

## The Three-Phase Vision (from Translator Role Definition)

The translation tool and translator community are Phase 1-2 of a three-phase evolution:

**Phase 1: Internal Translation (now)** — Translate all user-facing content. Establish human reviewer workflow. Build translation memory and glossary. Use SG/Send for reviewer workflows (dogfooding).

**Phase 2: Scale and Market Entry (near-term)** — Add many more languages. Community-driven translation (users contribute corrections, best contributors become official reviewers). Native content production (some content created directly in target languages, not translated from English). The non-English market strategy becomes the growth strategy.

**Phase 3: Customer-Facing Translation Product (future)** — Translation as a user-facing feature: recipients receive files/text in their preferred language, translated automatically. Client-side LLM translation for privacy. The Translator owns a P&L: revenue from translation services vs LLM API costs. The internal pipeline becomes the foundation for a commercial product.

And beyond Phase 3, the **corporate translation profile** opportunity: multinational companies don't have one language and one culture — they have dozens. Each subsidiary, each acquired company retains its own brand voice. SG/Send can translate content not just into languages but into specific corporate identities (language + cultural context + brand voice + terminology + compliance context). Per-entity translation profiles are a recurring revenue stream with margins that improve over time as profiles mature.

The translation visualisation tool (this brief) is the visible starting point for all of this.

## Part 1: Translation Visualisation Tool

### What It Does

A client-side tool at `sgraph.ai/tools/translations/` that visualises the translation JSON files used by the sgraph.ai website. It's both a useful utility for contributors and a demonstration of what static client-side tools can do.

### Features

#### Culture Variant Diff View

The most interesting feature: compare two variants of the same language and show ONLY the differences. For example, ES-ES vs ES-MX, PT-PT vs PT-BR, EN-GB vs EN-US:

```
┌─────────────────────┬──────────────────────┬──────────────────────┐
│ Key                 │ PT-PT 🇵🇹             │ PT-BR 🇧🇷             │
├─────────────────────┼──────────────────────┼──────────────────────┤
│ nav.pricing         │ Preços               │ Preços               │  ← same (hidden)
│ hero.upload         │ ficheiro             │ arquivo              │  ← DIFFERENT ✏️
│ cta.get_started     │ Começar              │ Começar              │  ← same (hidden)
│ pricing.per_file    │ por ficheiro         │ por arquivo          │  ← DIFFERENT ✏️
└─────────────────────┴──────────────────────┴──────────────────────┘

Showing 2 differences out of 847 keys (99.8% identical)
```

**Key UX:** Don't list the 845 keys that are the same. Show only the differences. Allow sorting by: number of character changes, alphabetical, section/page.

This is genuinely interesting content — what's the actual difference between Portuguese from Portugal and Portuguese from Brazil in a software context? Between Spanish from Spain and Spanish from Argentina? Between English UK and English US? The diff view makes it visible and educational.

#### Translation Glossary (Built-In)

The Translator role definition includes a starter glossary that the tool should display and make searchable:

| English | PT-BR 🇧🇷 | PT-PT 🇵🇹 | Notes |
|---|---|---|---|
| File | Arquivo | Ficheiro | Classic BR vs PT difference |
| Encrypt | Criptografar | Encriptar | Different verbs |
| Upload | Upload / Enviar | Upload / Carregar | |
| Download | Download / Baixar | Download / Transferir | |
| Token | Token | Token | Keep as-is in tech context |
| Access code | Código de acesso | Código de acesso | |
| Drag file here | Arraste o arquivo para aqui | Arraste o ficheiro para aqui | Already live in current UI |
| Early Access | Acesso Antecipado | Acesso Antecipado | |

This glossary grows with every human review. The tool should surface it — when a contributor is reviewing translations, they can see the approved term for their language variant. This ensures consistency across all content.

**Future:** As corrections accumulate, they build a translation memory — a database of approved translations for recurring terms. The Translator role definition describes this as the classic Computer-Assisted Translation (CAT) workflow powered by LLMs instead of traditional translation memory systems.

#### Quality Indicator

Each translation should have a quality badge:

| Badge | Meaning |
|---|---|
| 🤖 AI | Machine-translated, not human-reviewed |
| 👁️ Reviewed | Human has reviewed and approved |
| ✏️ Edited | Human has made corrections |
| 🌍 Native | Translated or validated by a native speaker |

Currently all translations are 🤖 AI. The goal is to move them towards 🌍 Native through community contributions.

### Technical Implementation

The translation JSON files are already in the GitHub repo (they power the website). The tool:

1. Loads the JSON files directly (either bundled into the tool's build or fetched from the repo via GitHub raw URLs)
2. Parses and compares them client-side
3. Renders the comparison UI
4. Links each key back to the source file in GitHub for editing

**All client-side. No server. No tracking.** Same principles as the IP tool and CF log viewer — the data stays in your browser.

### Ship the Translation JSON Files

The voice memo notes that in the current build, the translation JS files may not be shipped as static assets. They need to be:

1. Available as static files on the deployed site (so the tool can load them)
2. Available in the GitHub repo (so contributors can find and edit them)
3. Linked from the tool to their GitHub location (so "Edit this translation" opens the right file)

**Action for Developer:** Ensure the i18n JSON files are:
- Shipped in the static site build
- Available at a predictable URL (e.g., `sgraph.ai/i18n/en-gb.json`, `sgraph.ai/i18n/es-es.json`)
- Linked to their GitHub source file from the translation tool

## Part 2: Translator Contribution Workflow

### The User Journey for a Translator

```
1. Translator discovers sgraph.ai (via outreach, LinkedIn, search)
   
2. Notices "Help us improve translations" CTA
   → Links to sgraph.ai/tools/translations/

3. Opens the translation tool
   → Sees the side-by-side comparison for their language
   → Notices an AI translation that sounds unnatural

4. Clicks "Edit this translation on GitHub" 
   → Opens the JSON file in GitHub's web editor
   → Makes the fix
   → Submits a Pull Request

5. PR is reviewed and merged
   → Next site deployment includes the fix
   → Translator is credited (contributors page)
```

### Measure of Success

**Can we get a single external person to submit a translation PR?**

That's the bar. If one person goes through the flow — discovers the tool, sees an issue, edits on GitHub, submits a PR — the entire pipeline is validated. Forks, PRs, contributor engagement, community building — it all starts with one.

### Making GitHub Contribution Easy

GitHub's web UI allows editing files directly in the browser. For each translation key in the tool, provide a link:

```
[Edit on GitHub →] 
→ https://github.com/{org}/{repo}/edit/main/i18n/es-es.json
```

If GitHub supports deep-linking to a specific line, link to the exact line. If not, link to the file and let the contributor search for the key.

**Future improvement:** If GitHub's API allows it, build a "suggest edit" feature directly in the tool that creates a PR via the GitHub API. But for v1, a link to the file in GitHub is sufficient.

### Contributor Recognition

Create a contributors page or section that credits everyone who submits translation improvements:

```
🌍 Translation Contributors

ES-ES 🇪🇸  Maria G. (47 corrections)
PT-BR 🇧🇷  João S. (23 corrections)  
DE-DE 🇩🇪  Anna M. (12 corrections)
...
```

This gives translators public credit and creates a visible community.

A visitor who generates SSH keys on our site, checks their IP without tracking, and reviews translations is a visitor who trusts our security claims when they send their first encrypted file.

## Part 3.5: Dogfooding — Use SG/Send for Translation Review

The Translator role definition specifies that the translation review workflow should use SG/Send itself:

1. Translator produces a translation (AI-generated draft)
2. The translation is packaged as a structured JSON file (original text + proposed translation per key)
3. **Sent to the human reviewer via SG/Send** — encrypted, tracked, with a unique access token
4. The reviewer opens it, reviews translations, makes corrections
5. The corrected version is sent back via SG/Send
6. Corrections feed back into the translation memory

This is dogfooding at its best — every translation review generates real SG/Send usage, tests the product end-to-end, and demonstrates the workflow to people (translators) who are also a target market.

**Future (when interactive encrypted forms ship):** The reviewer doesn't edit a JSON file — they see a dynamically generated UI where they can approve, reject, or edit each translation line directly in the browser. JSON in, JSON out, with a beautiful interface. This is described in the Explorer brief for the data room vision.

## Part 3.6: Audience Translation

The Translator role definition identifies a second dimension beyond language translation: **audience translation** — transforming the same information for different personas.

Example: "We deployed a new Lambda function that reduces API response time by 40%."

| Audience | Translation |
|---|---|
| **Developer** | "The /api/upload endpoint now uses a streaming Lambda response, reducing p95 latency from 850ms to 510ms." |
| **Business stakeholder** | "File uploads are now 40% faster. Users will experience noticeably snappier performance." |
| **End user** | "We've made sending files faster." |
| **Investor** | "Platform performance improved by 40%, directly impacting user retention metrics." |

The translation tool could eventually support this: show the same content translated not just across languages but across audience levels. For the website, this means the technical architecture page speaks differently to a CISO than to a developer — same information, different depth and framing.

This is not for Phase 1, but the tool architecture should be aware that "translation" eventually means both language AND audience.

## Part 4: Translators as a Market

### Why Translators Are a Perfect SG/Send User

| Need | How SG/Send Serves It |
|---|---|
| **Receive files from clients** | Clients send documents via SG/Send — encrypted, no account needed |
| **Share translated files back** | Translator sends completed work via SG/Send |
| **Manage project credentials** | API keys, platform logins, client credentials — all via one-time secret links |
| **Organise project files** | Data rooms (when live) for per-project file management |
| **Work across borders** | Multilingual interface in their language, GDPR-compliant for EU clients |
| **Handle sensitive content** | Legal documents, medical records, financial reports — zero-knowledge encryption |

### Outreach Strategy to Translator Communities

| Channel | Approach |
|---|---|
| **ProZ.com** | Professional translator community — post about the project, invite contributions |
| **TranslatorsCafe** | Similar community — share the contribution opportunity |
| **LinkedIn** | Search for translators in target languages, personalised outreach |
| **Translation subreddits** | r/TranslationStudies, r/translator — share the tool and invite feedback |
| **University language departments** | Translation students — offer as a portfolio contribution opportunity |

**The pitch to translators:** "We're building open-source encrypted file transfer — the kind of tool you'd use with your clients. We've AI-translated our site into 17 variants and we'd love native speakers to help us get the cultural nuances right. Here's a tool to see the translations side-by-side and submit corrections via GitHub."

### Language-Specific LLM Research

**Action for Developer/Translator:** The role definition identifies an important research task — the major LLMs (Claude, GPT) are excellent but English-centric. There may be models specifically trained on Portuguese, Spanish, German, or other language corpora that produce more natural first-draft translations.

Research questions:
- Which models are natively trained in target languages (not just fine-tuned from English)?
- Are they available via API? (OpenRouter, HuggingFace Inference API, Replicate, Together AI, Fireworks?)
- Are translations more natural than Claude/GPT for the target language? (Side-by-side comparison with human reviewers)
- Can we call them from our existing infrastructure?

If better models exist and are accessible via API, they should be integrated into the translation pipeline for initial drafts, with Claude used for consistency checking.
