# Dev Brief: Replace LaunchList with Internal Early Access Signup

**Date:** 2026-03-01
**Status:** Draft
**Priority:** High — blocks Challenge campaign launch
**Triggered by:** UX brief `v0.10.4__sg-send-homepage-ux-brief`

## Problem

The Early Access signup flow currently redirects users to `getlaunchlist.com` after they submit their name and email. This breaks the SG/Send brand experience — the user is ejected to a bright green third-party page with a different logo, referral mechanics they didn't ask for, and their email visible in the URL. For a zero-knowledge encryption product, this is a trust violation at the worst possible moment: right after the user demonstrated trust by giving us their details.

## What We Need

A simple API endpoint on `send.sgraph.ai` that replaces LaunchList entirely. When a user submits the Early Access form, we handle it ourselves.

## Specification

**Endpoint:** `POST /api/early-access/signup` (or equivalent path that fits the current routing structure)

**Request body:**
```json
{
  "name": "string",
  "email": "string"
}
```

**What the endpoint does:**

1. Basic validation (name not empty, email looks valid)
2. Sends a notification email to **two recipients**:
   - `dinis.cruz@owasp.org` (or whichever address Dinis wants — confirm before implementation)
   - `sherpa.explorer@sgraph.ai`
3. The email should contain the signup details: name, email, timestamp, and the locale/language the user signed up from (if detectable from the page URL or headers)
4. Returns a success response

**Email content — keep it simple:**
```
Subject: [SG/Send] New Early Access Signup: {name}

New Early Access signup received.

Name:      {name}
Email:     {email}
Signed up: {timestamp UTC}
Locale:    {locale if available}
```

**What the frontend does after a successful response:**

Instead of redirecting to LaunchList, show an inline confirmation within the SG/Send UI — same dark theme, same branding. Something like:

```
✓ You're in.

We'll send your access token to {email} within 48 hours.

While you wait, you can:
  → Read how SG/Send works (link to docs)
  → View the source code on GitHub (link to repo)
  → Buy credits now and start immediately (link to Stripe)
```

The confirmation stays on `sgraph.ai` — no redirect, no third-party domain, no brand break.

## What We Do NOT Need

- A database or persistence layer for signups (the email notification is the record for now — we can add a proper signup store later)
- Waitlist position numbers or queue mechanics (LaunchList's main feature, which we don't actually use meaningfully)
- Referral tracking or social sharing (LaunchList features we're explicitly removing)
- Rate limiting beyond basic sanity (we can add this later if needed; the Challenge campaign will send ~50-100 people max)

## Infrastructure Notes

- The existing `send.sgraph.ai` backend runs on Lambda — this endpoint should fit within the existing deployment
- For sending email: use SES if already configured, or any available email sending capability in the current stack. If nothing is set up, a simple SES integration is the path of least resistance given we're already on AWS
- If SES isn't viable quickly, even a webhook to an n8n workflow that sends the email would work as an interim solution

## Acceptance Criteria

1. User submits Early Access form → stays on sgraph.ai (no redirect)
2. Notification email arrives at both addresses within 60 seconds
3. User sees branded confirmation with next steps
4. No user data is exposed in URLs at any point
5. LaunchList integration is removed entirely

## Timeline

This should be implemented before the Challenge campaign launches. It's a small endpoint — the scope is intentionally minimal to keep it shippable in a single session.
