# Brief: Translation Scaling Strategy — Lessons, Architecture, and Content Zones

**version** v0.10.1  
**date** 1 Mar 2026  
**from** Human (project lead)  
**to** Developer (lead), Architect, Sherpa, Ambassador, Translator  
**type** Architecture + strategy brief — translation infrastructure  

---

## What Happened

The multilingual website launch was a success — 12 languages, 17 country variants, live from day one. But the translation process itself revealed scaling problems that need to be addressed before we add more content or more languages.

### Problem 1: Claude Code Agent Paralysis

When implementing new pages that needed translation, Claude Code spun up 8 translator agents in parallel. Nearly all of them hung. The root cause: each agent was loading the full translation JSON file (which is large — all keys, all text, all languages), then attempting to translate every single string. This creates:

- Massive context window consumption per agent
- Agents competing for the same resources
- Sessions running out of context window and becoming unresponsive
- Unable to reconnect to hung sessions

**Lesson:** "Translate everything in this JSON" doesn't scale. The translation task needs to be decomposed into smaller units.

### Problem 2: Context Window Exhaustion

Several sessions ran out of context window entirely. When working on a UI that has a multilingual element, the agent loads multiple JSON files (one per language variant) plus the code being edited. With 17 variants, this fills the context window before the actual work begins.

**Lesson:** The development workflow cannot require loading all language variants simultaneously. Agents should work on one language at a time, or on a small subset of keys.

### Problem 3: Claude Code Usage Limits

The human hit Claude Code usage limits for the first time ever during the translation session. The volume of translation work (17 variants × all pages) consumed the daily allocation.

**Lesson:** Translation is expensive in AI compute terms. Translating N pages × M languages scales multiplicatively. We need to be deliberate about what gets translated and when.

## The New Strategy: Three Content Zones

All content falls into one of three zones, each with a different translation policy:

### Zone 1: Tools and Product UI — Always Multilingual

**What:** The interactive tools and product interfaces that users directly interact with.

- send.sgraph.ai (the Send tool)
- SSH key generator
- Translation visualisation tool
- Future: vault, data room, other tools
- UI elements: buttons, labels, navigation, tooltips, error messages, status messages

**Translation policy:** Full multilingual support in all 17+ variants. These are the touchpoints where language matters most — a user interacting with a tool needs it in their language.

**Admin interfaces are excluded** — admin is English only. That's fine.

**Why this zone scales:** Tool UI text is relatively small and stable. Once translated, it rarely changes. The JSON files are manageable in size.

### Zone 2: Commercial Website — Multilingual

**What:** The main sgraph.ai website — the pages that drive awareness, trust, and conversion.

- Homepage / landing page
- Pricing page
- Product overview
- Architecture overview
- Team page
- About page
- News headlines and summaries

**Translation policy:** Full multilingual support in all 17+ variants. These pages are what a visitor sees first. A Portuguese visitor seeing a Portuguese website is qualitatively different from seeing an English website with a translate button.

**Why this zone scales:** The commercial website is a bounded set of pages. New pages are added infrequently. The content is marketing-oriented (shorter, punchier) rather than documentation (longer, more detailed).

### Zone 3: Documentation — English First, Translate on Demand

**What:** Detailed explanations, guides, technical documentation, deep-dive articles.

- User guides
- API documentation
- Architecture deep-dives
- How-to articles
- Educational content ("What is PKI?", "Symmetric vs asymmetric encryption")
- Blog posts and detailed news articles

**Translation policy:** English (EN-GB) primary. Translations added based on:
1. Community contributions (someone volunteers to translate)
2. Customer demand (a significant user base in a specific language)
3. Strategic priority (entering a new market)

**Why this zone is different:** Documentation is high-volume, changes frequently, and is text-heavy. Translating every doc page into 17 variants is prohibitively expensive in AI compute and human review time. Better to have excellent English docs than mediocre translations of everything.

**Location:** The docs site (docs.sgraph.ai or sgraph.ai/docs/) is a separate deployment from the main site. This separation is important for the development workflow — agents working on docs don't need to load translation JSONs.

## Cross-Zone Language Navigation

### The Problem: Language Transitions

When a user is browsing the Portuguese version of sgraph.ai and clicks a link to the docs site (which is English only), they hit a jarring language transition. Similarly, if auto-detect redirects someone to PT-PT but they actually prefer English, they need an obvious escape hatch.

### Solution: Visual Language Indicators

**Principle:** Every link that crosses a language boundary should have a visual indicator.

**Outbound from multilingual page to English-only content:**
```
📖 Documentation 🇬🇧
```
The flag icon signals: "this link takes you to English content." The user isn't surprised when they arrive at an English page.

**Inbound from English-only page to multilingual content:**
```
🌐 Available in: 🇵🇹 🇧🇷 🇩🇪 🇪🇸 🇫🇷 ...
```
A banner or sidebar indicating which languages are available for the current content, or for the parent site.

**Auto-detect override:**
The current auto-detect feature redirects users to their browser's locale. But some users (e.g., an English speaker in Portugal) don't want the local language. The language picker must be:
- Always visible and easy to find (it already is — top-right corner)
- Persistent (once you switch, it sticks via a cookie or URL parameter)
- Clear about what it does (shows flag + language name in the target language)

**Action for Designer/Developer:** Research how other multilingual sites handle cross-language navigation (Wikipedia, EU institutional sites, Stripe's documentation). Adopt best practices rather than inventing from scratch.

## Development Workflow: Keeping Translation Out of the Critical Path

### The Problem

When a developer adds a new page or changes UI text, the current workflow requires updating all 17 translation JSON files. This:
- Fills context windows with translation data
- Slows down development (8 agents hanging)
- Creates merge conflicts (multiple agents editing the same JSON files)
- Costs significant AI compute

### The New Workflow

```
Developer creates/edits page
  ↓
English text added directly in code (or EN-GB JSON only)
  ↓
Page ships in English
  ↓
Translation task created (async, separate from development)
  ↓
Translator agent(s) process ONE language at a time
  ↓
Human review per language (if available)
  ↓
Translated version deployed
```

**Key change:** Translation is decoupled from development. A new page ships in English immediately. Translations follow asynchronously. The developer never needs to load 17 JSON files.

### Decomposed Translation Tasks

Instead of "translate this entire JSON file," break the work into:

```
Task: translate keys hero.title, hero.subtitle, hero.cta 
      from EN-GB to PT-PT
Input: 3 keys + their English values
Output: 3 keys + their Portuguese values
Context needed: minimal
```

Each task is small, self-contained, and can be processed by an agent without loading the full translation file. Multiple small tasks can run in parallel without competing for context.

## The Translation Archive

### The Naming Question

The human asks: what do we call the repository where translations live, where people work on translations, and where translated content accumulates?

Options considered:

| Name | Pros | Cons |
|---|---|---|
| **Tower of Babel** | Evocative, everyone knows it | The biblical story ends in failure and confusion — mixed connotations |
| **Library** | Clear, professional | Generic, doesn't convey the translation-specific nature |
| **Lexicon** | Language-focused, scholarly | More about words/definitions than translations |
| **Rosetta** | Famous translation artefact (Rosetta Stone), positive connotations, concise | Might be confused with Rosetta Stone (the language learning company) |
| **Lingua** | Latin for "language/tongue," clean, short | Less immediately recognisable |
| **i18n** | Industry standard abbreviation for internationalisation | Technical, not evocative |
| **Translations** | Simple, obvious, no ambiguity | Boring |

**Recommendation: Rosetta.**

"The Rosetta" or "rosetta/" as a directory name works well:
- Immediately conveys "translations" to anyone who knows the Rosetta Stone
- Positive connotations (unlocking understanding across languages)
- Short enough for paths: `rosetta/en-gb.json`, `rosetta/pt-br.json`
- Works as a tool name: "Check the Rosetta for the Portuguese translation"
- Aligns with the mission: making content accessible across languages and cultures

```
rosetta/
├── locales/
│   ├── en-gb.json        ← Base language
│   ├── en-us.json
│   ├── pt-pt.json
│   ├── pt-br.json
│   ├── es-es.json
│   ├── ...
│   └── tlh.json          ← Klingon
├── glossary/
│   ├── pt-br.json        ← Approved terms for Brazilian Portuguese
│   ├── pt-pt.json
│   └── ...
├── memory/
│   ├── pt-br.json        ← Translation memory (corrections over time)
│   └── ...
└── status.json            ← Coverage, quality, last updated per locale
```

### Preserving Existing Translations

The content already translated should not be lost. It's a valuable archive — even if the pages move to the docs site, the translations represent significant work.

**Action:** Move translated content from the main site repo to the Rosetta archive. The main site's JSON files remain slim (Zone 1 + Zone 2 keys only). The docs site pulls from the Rosetta archive for any content that has been translated.

### Semantic Graph (Future)

The human notes that eventually the translations will be in a semantic graph, enabling interesting cross-referencing: "show me every place where we say 'encrypt' and how it's translated in each language." This is a future capability but the data structure should support it — each translation key should be addressable and queryable.

## Cost Management

### The Multiplication Problem

Translation cost = (number of pages) × (number of languages) × (cost per translation)

With 17 variants, every new page costs 17x in translation compute. This is fine for Zone 1 (small, stable) and Zone 2 (bounded), but unsustainable for Zone 3 (high-volume, frequently changing).

### Strategies to Reduce Cost

| Strategy | Impact |
|---|---|
| **Decomposed tasks** (translate 3 keys, not 800) | Reduces context window waste, prevents agent paralysis |
| **Incremental translation** (only translate changed keys) | Avoids re-translating stable content |
| **Priority languages** (translate to top 5 first, others on demand) | Reduces the multiplier from 17x to 5x for new content |
| **Community contributions** (native speakers fix translations) | Zero AI compute cost, higher quality |
| **Translation memory** (reuse approved translations for recurring patterns) | Reduces volume over time |
| **Language-specific LLMs** (from Translator role brief) | Potentially higher quality per call, reducing human review iterations |
| **Zone 3 stays English** until demand proves otherwise | Eliminates the largest source of translation volume |
