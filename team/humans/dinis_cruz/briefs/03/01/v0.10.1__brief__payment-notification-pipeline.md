# Brief: Payment Notification Pipeline and Multi-Channel Alerts

**version** v0.10.1  
**date** 1 Mar 2026  
**from** Human (project lead)  
**to** Developer (lead), Architect, Sherpa  
**type** Dev brief — notification infrastructure + customer feedback  

---

## The Problem

Stripe no longer sends email notifications for new payments by default. Since the payment link is now live, we need to know immediately when someone pays. A £5 payment sitting unnoticed means a customer waiting for their access token — that's a terrible first experience.

This is actually a great problem to have: it means we need to build notification infrastructure that will serve us well beyond just Stripe.

---

## What Should Happen When Someone Pays

```
Customer completes Stripe checkout (£5)
  │
  ▼
Stripe fires webhook → FastAPI endpoint (admin API)
  │
  ├──► Email to Sherpa (sherpa@sgraph.ai)
  │    "New payment received: £5 from {email}. Create access token."
  │
  ├──► Email to human's personal email
  │    Same content — human monitors this more closely
  │
  ├──► SMS to human's phone
  │    "💰 SG/Send: New £5 payment from {email}"
  │
  └──► WhatsApp message to human's phone
       "New SG/Send payment: £5 from {email}. Token needed."
```

**All four channels for now.** The human needs to see this immediately so they can manually create the access token and email it to the customer. Redundancy is intentional — if one channel is missed, another catches it.

## Technical Implementation

### Step 1: Stripe Webhook → FastAPI

Add a webhook endpoint to the admin API:

```
POST /admin/webhooks/stripe
Authorization: Stripe-Signature header (verify with webhook signing secret)
```

Stripe sends a `checkout.session.completed` event. The endpoint extracts:
- Customer email
- Amount paid
- Payment ID
- Timestamp

**Authentication:** Stripe webhooks use a signing secret to verify the payload. The endpoint MUST verify the signature — don't just trust any POST to this URL.

**Action for Developer:** Add the webhook endpoint to the existing FastAPI admin API. Use the Stripe webhook signing secret (stored in environment/secrets, not in code).

### Step 2: Email Notifications

Two emails on each payment:

| Recipient | Address | Purpose |
|---|---|---|
| Sherpa | sherpa@sgraph.ai | Official record, Sherpa can action it |
| Human | (personal email) | Human monitors this more closely for fast response |

Use the SG WorkMail service (from v0.7.6 brief) or the existing n8n email workflow — whichever is available first.

**Email content:**
```
Subject: 💰 New SG/Send Payment — £5.00 from {customer_email}

A new payment has been received:

Amount: £5.00
Customer: {customer_email}
Payment ID: {stripe_payment_id}
Time: {timestamp}

Action needed: Create an access token for this customer 
and send it to {customer_email}.

Admin panel: https://admin.send.sgraph.ai/tokens/create
```

### Step 3: SMS Notification

Short, immediate, phone-buzzing:

```
💰 SG/Send: £5 payment from {customer_email}. Create token.
```

**Options for SMS delivery:**

| Service | Pros | Cons |
|---|---|---|
| **AWS SNS** | Already in our AWS account, simple API, cheap | Another AWS dependency |
| **Twilio** | Industry standard, reliable, good API | New vendor, new account |
| **n8n SMS node** | Already have n8n | Adds n8n dependency for critical path |
| **WhatsApp Business API** | Covers both WhatsApp and could replace SMS | More complex setup |

**Recommendation:** Start with AWS SNS for SMS (we're already in AWS, it's one API call). Add WhatsApp as a separate channel.

### Step 4: WhatsApp Notification

This is both a practical notification channel and an exploration of a bigger idea: communicating with Sherpa and the agents via WhatsApp.

**For now (MVP):** One-way notification — payment happens, human receives WhatsApp message.

**Options for WhatsApp delivery:**

| Service | Pros | Cons |
|---|---|---|
| **WhatsApp Business API** (via Meta) | Official, reliable, supports rich messages | Complex setup, approval process, costs |
| **Twilio WhatsApp** | Easier API than direct Meta, familiar | New vendor |
| **n8n WhatsApp node** | Already have n8n | May be limited |
| **WhatsApp via browser automation** | No vendor needed | Fragile, not production-grade |

**Recommendation:** Research which is fastest to get a message delivered to the human's phone. Twilio WhatsApp sandbox might be the quickest path for testing.

## Future: Two-Way Agent Communication via WhatsApp

The human noted a bigger vision: communicating with Sherpa and the agents through WhatsApp. This is a significant capability:

```
Human (WhatsApp) → "Create a token for user@example.com, 500 credits"
  ↓
WhatsApp Business API → FastAPI webhook
  ↓
Sherpa processes the request → creates token via admin API
  ↓
Sherpa → WhatsApp: "Done. Token {token} created for user@example.com. 
                     Sent via email. 500 credits, expires {date}."
```

And eventually:
```
Human (WhatsApp) → "What payments came in today?"
Sherpa (WhatsApp) → "3 payments totalling £15. 2 tokens delivered, 
                      1 pending. Want me to create the last one?"
Human (WhatsApp) → "Yes"
Sherpa (WhatsApp) → "Done. All 3 customers have tokens."
```

This is not for this sprint — but the webhook infrastructure we're building now (Stripe → FastAPI → multi-channel notification) is the foundation. The same FastAPI service that receives Stripe webhooks can receive WhatsApp messages.

**Eventually: communication via SG/Send itself.** The human noted this would be "really interesting" — when the messaging features are live, the agent communication channel becomes the product itself.

## Customer Feedback: Portugal Contact

### The Conversation

A friend building a B2B startup in Portugal (coordinating suppliers and clients) reviewed sgraph.ai and provided feedback:

### What He Liked

| Feedback | Implication |
|---|---|
| **Portuguese translations worked well** | The multilingual investment is paying off — first external validation |
| **Interested in the messaging/inbox concept** | The "fractal email" architecture (encrypted inbox, consolidated communications) resonated strongly |
| **Interested in data rooms** | File organisation per project/client — centralising documents, emails, credentials in one place |
| **Immediately identified PKI as the real challenge** | Sophisticated technical understanding — "sending the public key is the real challenge." Validates our roadmap priority |

### His Pain Points (Our Opportunities)

**1. Centralised communications:** When coordinating suppliers and clients across multiple platforms, messages get scattered across email, WhatsApp, Slack, etc. A single place that consolidates project communications — with the files, the emails, everything together — would be very valuable.

**2. Notification fatigue:** When something happens, where do you get notified? Email is official but slow. SMS is immediate but gets lost. WhatsApp is where people actually are but mixes personal and business. The ideal is a single place you go to that has everything organised by project/client.

**3. File + message organisation:** The pain isn't just sending files securely — it's organising them alongside the related communications. A data room that has the files AND the email threads AND the messages, all in one encrypted space.

### How This Maps to Our Roadmap

| His Need | Our Feature | Status |
|---|---|---|
| Centralised communications | Data room messaging / encrypted inbox | Planned (v0.7.4 fractal email brief) |
| Secure file sharing | SG/Send core | ✅ Live |
| PKI / public key exchange | PKI mode | Planned (next after symmetric) |
| Data rooms | Data vault | Planned |
| Send + receive email from within | WorkMail integration in data rooms | Planned (v0.7.6 WorkMail brief) |
| Email classification | AppSec pre-read / classification | Planned (v0.7.4 prompt injection brief) |
| Encrypted email | Signed/encrypted email from data rooms | Planned |

**Key insight:** He didn't care about the encryption for its own sake. He cared about the centralisation and organisation. The encryption is table stakes — the value proposition is "one place for everything related to this project/client, and it happens to be encrypted." This is an important framing for the Ambassador's messaging.

### Action Items from This Feedback

1. **Ambassador:** Add "B2B coordinator" as a use case persona (someone coordinating suppliers and clients, needs centralised project communications)
2. **Advocate:** The "data room as project hub" framing (files + messages + emails in one encrypted space) should be prominent in marketing
3. **Developer:** PKI mode remains the highest-priority feature after current sprint — external validation confirms this
