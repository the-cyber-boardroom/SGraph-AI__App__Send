## The Service: FastAPI QA API

The browser automation capabilities need to be exposed as a proper API service. Build a FastAPI service (using the same technology stack as the existing backend) that provides:

- An endpoint for every action (click, navigate, type, screenshot)
- An endpoint for every page (upload page, download page, gallery, browse, welcome)
- Compound endpoints for workflows (upload a file end-to-end, download and verify, gallery round-trip)
- Each endpoint returns artifacts: screenshots, Playwright traces, folder listings, Prometheus metrics

The API can be pointed at any server:
- Local dev server (for development and Explorer testing)
- Dev server (dev.send.sgraph.ai)
- Production server (send.sgraph.ai)
- Any other instance

Protected by an API key for now. This single API serves QA testing, synthetic traffic generation, smoke testing, documentation screenshots, and performance testing. One API, many uses.

## Principles for Browser Automation (Learned from Experience)

The Explorer phase of the QA automation produced a lot of useful test cases and screenshots. But it also reproduced the classic browser automation mistakes: brittle tests, excessive browser restarts, hardcoded waits, tight coupling to DOM structure. These are the same problems that made Selenium projects into white elephants for years.

The following principles come from significant past experience with browser automation and must be applied in the Villager phase of this QA infrastructure.

### Principle 1: It Is All About the API

The test itself should be the last and least important thing written. The test should be completely abstracted from the browser. A test should read like a description of functionality:

- "Upload a file, verify it appears in the gallery"
- "Enter a friendly token, verify decryption succeeds"
- "Switch from gallery to folder view, verify all files visible"

Not:

- "Find the element with class .upload-zone, simulate dragenter event, wait 500ms, check if element with id #progress is visible..."

This means the investment goes into the API layers (browser level, page level, component level, workflow level). The tests become trivial once the APIs are solid. The API tests are critical and need to be stable. But they stabilise quickly because the API itself does not change often.

### Principle 2: Minimise Browser and Page Creation

A common code smell in browser automation: starting a new browser for every test suite, opening a new page for every test. This is slow and wasteful.

Better approach:
- Start one browser for the entire test run
- Navigate to the page you need (do not open a new page unless you must)
- Maintain state between related tests
- Only create a new page when testing requires a clean state

If your website is so fragile that you need a fresh browser for every test, fix the website. The fragility is the bug, not the test setup.

### Principle 3: Fix Problems in the Application, Not the Tests

When the test infrastructure hits a problem (element not findable, state not accessible, timing issue), the first question should be: "Can I fix this in the application?"

Example: we needed to check if localStorage had been set (for the access token). Instead of writing complex Playwright JavaScript evaluation, we added a small QA debug page to the website that exposes this information via a simple HTML interface. Problem solved at the source.

This principle means the application gets better as the tests get written. Every testing difficulty that gets fixed in the application is also a usability improvement, a debugging aid, or an observability enhancement.

### Principle 4: No Delays, Almost Ever

Hardcoded waits (`await page.waitForTimeout(2000)`) are always a code smell. They signal that something about the test infrastructure or the application is not right.

Most operations should be synchronous or event-driven:
- Wait for a specific element to appear (not a timer)
- Wait for a network request to complete (not a guess)
- Wait for a state change (not a hope)

The test environment has access to the full Chrome debugger, the page object, network events, console output, and DOM mutations. Use these to wait for the right thing, not an arbitrary duration.

Create simple wrappers that convert async Playwright patterns into sequential, sync-feeling calls. The goal: tests read top-to-bottom without callback nesting or promise chains cluttering the logic.

### Principle 5: Set Up State Directly, Do Not Replay Entire Workflows

To test the download page, you do not need to run the upload workflow first. You can:
- Pre-populate test transfers via the backend API
- Create test files during server setup
- Set localStorage values via the QA debug page
- Call the API directly to create the state needed

This is not cheating. It is efficient testing. The end-to-end workflow tests verify the full chain. The component tests verify individual components in isolation. The component tests should set up their state directly, not by replaying every prior step.

Having direct access to the backend (in memory, callable via FastAPI and directly via Python objects) makes this trivial. Set up exactly the state you need for the test. No more, no less.

### Principle 6: Screenshots Document, They Do Not Drive

A workflow that was already emerging and needed correction: describing bugs to Claude using screenshots and text descriptions, then asking Claude to fix them. This is backwards.

The correct workflow: every bug becomes a test. The test produces screenshots. The screenshots are documentation. The dev agent reads the test (not a screenshot), understands the expected vs actual behaviour, and produces a fix.

Screenshots are artifacts of tests, not replacements for tests. Claude can read screenshots (which is powerful for understanding visual issues), but the source of truth is the test script, not the screenshot.

### Principle 7: Interactive Development, Then Automation

During the Explorer phase of test development, the critical capability is interactive iteration: execute a step, observe the result, execute the next step, observe. The page maintains its state throughout. You build the test by interacting with a live browser.

This is why the local Playwright setup (browser open, page loaded, state preserved between commands) is so valuable. You do not write the test in one go and hope it works. You build it step by step, with the browser showing you what is happening at each point.

Once the test works interactively, wrap it in a function. That function becomes a permanent automated test. The Explorer produced it. The Villager made it reliable.

## The Setup We Have

The test environment access is extraordinary:

| Layer | Access Type | What It Gives Us |
|---|---|---|
| SG/Send API | In-memory Python object | Direct method calls, no HTTP overhead, full internal state |
| FastAPI service | HTTP on localhost | Same API as production, realistic request/response |
| Static website | File system + HTTP | Modify pages live, serve locally, no deployment needed |
| Chrome (Playwright) | Chrome DevTools Protocol | Page object, DOM, console, network, localStorage, cookies |

Four layers of access, all local, all in the same process. This means there is no legitimate excuse for brittle tests or hardcoded waits. If the test needs something, the infrastructure can provide it.

## From Tests to Releases

The long-term flow:

1. Bug or feature is identified (by human, QA agent, or monitoring)
2. QA writes a test (with screenshots, with bug/feature ID)
3. Test is published to the QA website (Jekyll, auto-built)
4. Dev agent reads the test, implements the fix (as a patch)
5. Fix is applied, test is updated to regression mode
6. Release notes are generated from the test/bug/feature IDs
7. Every release links to every fix, with screenshots, with tests

The release documentation writes itself. Every bug has a test. Every fix has a before/after screenshot. Every release is a list of resolved test IDs with links to the evidence.
