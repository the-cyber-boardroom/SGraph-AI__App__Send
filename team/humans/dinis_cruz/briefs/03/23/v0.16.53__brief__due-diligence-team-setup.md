# Due Diligence Report: Team Setup and Execution Brief

**version** v0.16.53
**date** 23 Mar 2026
**from** Human (project lead)
**to** Conductor (this session)
**type** Team setup brief

---

## Your Mission

You are the Conductor for a due diligence research team. Your job is to set up the team, coordinate the workflow, and produce a comprehensive due diligence report on a target company.

## Step 1: Set Up the Team

Create the following folder structure and role definitions. Each agent gets a folder containing their role definition, identity, tasks, objectives, and how they work.

```
due-diligence-team/
  _team/
    objectives.md          Team objectives, target company, scope
    workflow.md             The execution workflow (see below)
  conductor/
    role.md                Coordinates all agents, manages workflow, quality gates
  grc-analyst/
    role.md                Governance, Risk, and Compliance analysis
    tasks.md               Research tasks, frameworks to apply, output format
  dpo/
    role.md                Data Protection Officer perspective
    tasks.md               Privacy assessment, data handling review, GDPR analysis
  appsec/
    role.md                Application Security specialist
    tasks.md               Technical security review, vulnerability assessment, attack surface
  architect/
    role.md                Technical architecture review
    tasks.md               Infrastructure analysis, scalability, technical debt assessment
  developer/
    role.md                Builds the report website and any tooling needed
    tasks.md               Website creation, data visualisation, report packaging
  librarian/
    role.md                Catalogues all findings, cross-references, maintains the index
    tasks.md               Index all research, link findings across agents, produce bibliography
```

### Role Definitions to Create

For each agent, the role.md should contain:

- **Identity:** Who they are, their expertise, their perspective
- **Objective:** What they are trying to achieve in this engagement
- **Approach:** How they work, what frameworks they apply, what they look for
- **Output format:** What their deliverables look like (reports, findings, recommendations)
- **Interactions:** Which other agents they collaborate with and how

Create these role definitions now. They should be specific to due diligence work, UK-based, English language. The roles should reflect real-world expertise:

- **GRC Analyst:** Thinks in terms of ISO 27001, NIST CSF, SOC 2, corporate governance, risk registers, regulatory compliance. Looks at public filings, certifications, regulatory actions, news, leadership track record.
- **DPO:** Thinks in terms of UK GDPR, Data Protection Act 2018, ICO guidance, privacy policies, data processing agreements, international transfers, cookie compliance, breach history.
- **AppSec:** Thinks in terms of OWASP, CVEs, public-facing attack surface, SSL/TLS configuration, security headers, exposed services, dependency vulnerabilities, leaked credentials, GitHub security posture.
- **Architect:** Thinks in terms of technology stack, infrastructure choices, scalability indicators, technical blog posts, job listings (reveal tech stack), API design, open source contributions.
- **Developer:** Builds the output. Creates a static website presenting the findings. Produces visualisations, summaries, and the final report package.
- **Librarian:** Indexes everything. Cross-references findings between agents. Maintains the master document list. Produces the executive summary by synthesising all agent outputs.

## Step 2: Capture the Target

Once the team is set up, ask me for the target company. I will provide:

- Company name
- Website URL
- Any specific areas of concern or focus
- The audience for the report (investor, board, partner, internal)

If a simple UI or structured form would help capture this information, the Developer and Architect can create one. Otherwise, a structured conversation works.

## Step 3: Execute the Research (Parallel)

Three agents research in parallel:

| Agent | What They Research | Sources |
|---|---|---|
| GRC Analyst | Corporate governance, risk posture, compliance certifications, regulatory history, leadership, financial health signals | Company website, Companies House, regulatory bodies, news, LinkedIn |
| DPO | Privacy policy analysis, cookie compliance, data processing transparency, breach history, international transfer mechanisms | Privacy policy, cookie banner, ICO records, GDPR compliance indicators |
| AppSec | External attack surface, SSL/TLS, security headers, exposed services, GitHub repos (if public), dependency health, leaked credentials | SSL Labs, SecurityHeaders.com, Shodan, GitHub, public vulnerability databases |

The Architect also researches the technology stack from public signals (job listings, blog posts, open source repos, API documentation).

Each agent produces a structured findings document in their folder.

## Step 4: Librarian Indexes and Cross-References

Once the research agents have produced their findings:

1. Librarian reads all findings documents
2. Creates a master index linking findings across agents
3. Identifies cross-cutting themes (e.g. "multiple agents flagged inadequate security headers")
4. Produces an executive summary synthesising all findings
5. Flags contradictions or gaps (e.g. "GRC found SOC 2 certification but AppSec found critical security header issues")

## Step 5: Review and Scoring

All agents review the compiled findings. The GRC Analyst applies a scoring framework:

- Overall risk rating (Low / Medium / High / Critical)
- Per-category scores (Governance, Privacy, Security, Technical)
- Key findings (top 5 risks, top 5 strengths)
- Recommendations (prioritised actions the target company should take)

## Step 6: Build the Report Website

The Developer and Architect create a static website presenting:

- Executive summary (one page, the Librarian's synthesis)
- Per-category detailed findings (one section per agent)
- Risk scores and visualisations
- Recommendations
- Source references and methodology

The website should be clean, professional, and suitable for sharing with the intended audience. It is a static site (HTML, CSS, JavaScript, no backend). It can be packaged as a vault or a folder for sharing via SG/Send.

## Communication

For this initial run, all work happens inside this Claude session. Each agent is a role you adopt as the Conductor directs.

When the team produces outputs that need to be shared with me or with external parties, we can use an SG/Send vault:

- Push findings to a vault using `sg-send-cli`
- I pull from the vault on my side
- The final report website can be packaged in a vault for delivery to the recipient

But start with everything in-session. We move to vaults when we need to share externally.

## Constraints

- UK-based analysis (UK GDPR, Companies House, ICO, FCA where relevant)
- English language
- All research from public sources only (no access to internal systems)
- Findings must cite sources
- The report should be honest and balanced (strengths and weaknesses, not just a hit piece)

## Begin

Start by creating the folder structure and all role definitions. Once the team is set up, ask me for the target company.
