## What Was Fixed

### CI Pipeline — 3 Root Causes Resolved

**1. Chrome `--no-sandbox` on Linux**

GH Actions Ubuntu disables unprivileged user namespaces. Chrome crashes on startup with `FATAL: No usable sandbox!`. OSBot-Playwright's `Playwright_Process` doesn't handle this.

Fix: `SG_Send__Playwright_Process` subclass overrides `start_process()` to add `--no-sandbox` on Linux. macOS delegates to parent (no issue).

```
GH Actions stderr (before fix):
  FATAL:content/browser/zygote_host/zygote_host_impl_linux.cc:128
  No usable sandbox!

After fix: Chrome starts cleanly with --no-sandbox
```

**2. `qa-setup.html` 404**

The lightweight QA page for localStorage was referenced by tests but never committed to the UI package. Worked locally only because persistent browser cached it.

Fix: committed `qa-setup.html` to `sgraph_ai_app_send__ui__user/v0/v0.3/v0.3.0/_common/` in the main SG/Send repo. `build_ui_serve_dir()` now copies it automatically.

**3. Port 10070 collision in CI**

All browser tests defaulted to the fixed debug port (10070). In CI, multiple test classes run sequentially in the same pytest session — second class finds port occupied.

Fix: `QA_Browser.chrome()` now uses `port=None` (random) when `headless=True`, fixed port only when `headless=False` (debug mode).

```python
port = self.port if not self.headless else None     # fixed for debug, random for CI
```

### CI Pipeline — Timeouts Added

Three layers of protection against hung processes (Chrome zombies after failed `setUpClass`):

| Layer | Setting | Catches |
|---|---|---|
| Job | `timeout-minutes: 10` | Entire job including install |
| Step | `timeout-minutes: 5` | Just the pytest run |
| Test | `--timeout=30` / `--timeout=60` | Individual stuck test |

## What Was Built

### Harness State Persistence

New classes: `Harness_State__Persistence` and `Schema__Harness_State`

**The problem:** every test run started servers on random ports. The browser's `build-info.js` pointed at the old API port. localStorage was on a different origin. Running a single test step-by-step in debug mode gave "Failed to fetch" because the API server had moved.

**The solution:** borrowed the pattern from OSBot-Playwright's `Playwright_Process` — save state to a JSON file at a well-known temp path, reload on next run.

```
/tmp/sg_send_qa_harness/harness_state.json
{
    "api_port"        : 54321,
    "ui_port"         : 63960,
    "ui_build_folder" : "/tmp/sg_send_qa_ui_build_v0.2.18",
    "ui_version"      : "v0.2.18",
    "access_token"    : "a1b2c3...",
    "chrome_port"     : 10070
}
```

**The behavior:**

```
Debug mode (headless=False):

  First run:
    _load_saved_state()   → None
    _start_api_server()   → random port 54321
    _build_ui()           → build to /tmp/sg_send_qa_ui_build_v0.2.18 (91ms)
    _start_ui_server()    → random port 63960
    _save_state()         → write JSON

  Second run:
    _load_saved_state()   → {api_port: 54321, ui_port: 63960, ...}
    _start_api_server()   → starts on port 54321 (SAME)
    _build_ui()           → version match + folder exists → SKIP (0ms)
    _start_ui_server()    → starts on port 63960 (SAME)
    _save_state()         → overwrite (same data)

  Result: localStorage persists. build-info.js still valid. set_access_token() → no-op.

CI mode (headless=True):
    _load_saved_state()   → None (skipped)
    Everything random, everything fresh, no persistence.
```

**Stale process detection:** `ports_in_use()` checks if saved ports are still occupied — warns if a previous test crashed without teardown.

### Upload Wizard End-to-End

The sequential test pattern proven through Shadow DOM:

```
test__01__upload__set_file            → page__root, set file via Shadow DOM
test__02__upload__click_next          → advance to choosing-share
test__03__upload__select_share_mode   → select "token" mode, advance to confirming
test__04__upload__click_encrypt       → encrypt & upload, wait for complete
test__05__upload__get_simple_token    → extract friendly token from Shadow DOM
```

**Shadow DOM discovery:** Playwright's `locator("upload-step-done #simple-token")` didn't reliably pierce the shadow boundary for text extraction. The fix was manual JS through `shadowRoot.querySelector`:

```python
js = """
var node = document.querySelector("upload-step-done")
                   .shadowRoot.querySelector("#simple-token")
result = node ? node.innerText : null
result
"""
result = self.invoke__javascript(js)
```

This is documented with a TODO — needs a generic helper for shadow DOM text extraction. The `locator` approach works for interaction (click, fill) but not always for reading text content from shadow roots.

**The friendly token format was verified:** `len(simple_token.split("-")) == 3` — three parts, word-word-NNNN format.

## What's Next

### Immediate

1. **Download round-trip** — upload via friendly token, then resolve token → browse → verify decrypted content. Tests 06-08 in the upload class are commented out pending this.

2. **Shadow DOM text extraction helper** — generic method for `document.querySelector(outer).shadowRoot.querySelector(inner).innerText`. Needed because Playwright `locator` doesn't reliably extract text from shadow roots.

3. **CI pipeline: `ci-pipeline__dev.yml` and `ci-pipeline__main.yml`** — still pass `target_url` which was removed from the base workflow. Need to remove.

### Medium-term

4. **Duration capture schema** — the harness has a TODO for `Schema__Harness_Durations` to track setup/teardown timing. The `capture_duration` import is already there.

5. **`set_access_token` XSS note** — `storage__set_token` now uses raw `localStorage.setItem('sgraph-send-token', '{token}')` which is vulnerable to injection if the token contains quotes. Should use parameterised JS evaluation. Marked with `# security-vuln` comment.

6. **Persist combined link / friendly token between tests** — the upload test extracts a friendly token but doesn't persist it for use by download tests. Need a class-level or harness-level state mechanism.
