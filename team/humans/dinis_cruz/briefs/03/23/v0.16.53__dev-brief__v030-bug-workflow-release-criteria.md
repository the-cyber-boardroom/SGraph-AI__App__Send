## Context

The v0.3.0 IFD refactoring is at 95% parity with v0.2.x. A thorough testing pass is finding bugs. Each bug needs to be classified, documented, tested, and fixed in a disciplined way. This brief defines the classification, the QA workflow, and the release criteria.

## Bug Classification: Three Categories

Every issue found in v0.3.0 falls into one of three categories. The classification determines priority and approach.

### Category 1: Refactoring Regressions

Something that was working in v0.2.x and is broken in v0.3.0. The v0.2.x code had a fix or a feature that was lost during the consolidation.

**Example:** When you drop a folder, the old folder configuration UI appears (options for excluding empty folders, dot files, encryption mechanism selection). This was removed in a later v0.2.x overlay because we picked good defaults. But the v0.3.0 rewrite pulled from an earlier version of that code, so the removed UI came back.

This is the hardest category to catch. The bug exists because somewhere in the v0.2.x chain (say v0.2.7), a feature was added. Then in v0.2.9, it was removed or replaced. The v0.3.0 rewrite might have included the v0.2.7 version but missed the v0.2.9 removal. The overlay chain makes this easy to miss.

**Priority:** Highest. These must be fixed before v0.3.0 ships. The goal is parity with v0.2.13 (the last stable release).

**Approach:** For each regression, identify which v0.2.x overlay had the fix. Show the specific file and code that implemented the correct behaviour. This gives the developer a clear reference for the fix.

### Category 2: Existing Bugs (Pre-Refactoring)

Bugs that exist in both v0.2.x and v0.3.0. The refactoring did not introduce them and did not fix them. They need fixing, but they are not blockers for v0.3.0 shipping (since v0.2.x shipped with them).

**Priority:** Medium. Fix for v0.3.0 if straightforward. Otherwise, track as v0.3.1 items.

### Category 3: Feature Requests

New functionality that did not exist in v0.2.x. Not a bug. Not a regression. A new thing to build.

**Priority:** Low for the v0.3.0 release. Track for v0.3.x Explorer phase.

## The QA Test-First Workflow

For every bug in categories 1 and 2, the workflow is:

### 1. Write a test that passes WITH the bug

This is critical and counterintuitive. The test does not fail when the bug is present. The test succeeds and captures the bug as the current state.

The test:
- Navigates to the page or component
- Performs the action that triggers the bug
- Takes a screenshot showing the bug
- Asserts that the screenshot was captured (the test passes)

This gives us:
- A reproducible demonstration of the bug
- A screenshot that documents what the bug looks like
- A test script that anyone (human or agent) can run to see the bug

### 2. Fix the bug

The developer makes the code change. For Category 1 regressions, the reference is the specific v0.2.x file and code that had the correct behaviour.

### 3. Update the test to fail WITH the bug

After the fix, update the test so that it would fail if the bug reappeared. The test now:
- Navigates to the page or component
- Performs the same action
- Takes a screenshot showing the correct behaviour
- Asserts that the correct behaviour is present (not the buggy behaviour)

This becomes a permanent regression test. If the bug ever returns (in a future refactoring, a new overlay, a merge conflict), this test catches it.

### 4. Commit the test alongside the fix

The fix and the updated test ship together. The test is the proof that the fix works and the guarantee that the bug will not return.

## What the Claude Session Needs to Do

For each bug discovered:

1. **Describe the bug clearly** (what happens, what should happen)
2. **Classify it** (regression, existing bug, or feature request)
3. **For regressions:** Trace back to the v0.2.x overlay that had the working code. Identify the specific file and the specific code. If the Claude Code session cannot fully trace it, make a reference note and the Claude Web session will continue the analysis.
4. **Write the QA test** (Playwright script that reproduces the bug with screenshot)
5. **Propose the fix** (as a patch, referencing the v0.2.x code where applicable)
6. **Update the QA test** (so it becomes a regression test)

## Additional Testing to Add

### Viewport and Responsive Testing

Several bugs have been spotted around different screen sizes and landscape mode. The Playwright Browser Automation API should include viewport control:

```javascript
await sgQA.setViewport(375, 812)   // iPhone X
await sgQA.setViewport(768, 1024)  // iPad
await sgQA.setViewport(1920, 1080) // Desktop
await sgQA.setViewport(1080, 1920) // Desktop portrait/rotated
```

Run the core workflows at each viewport size. Take screenshots. Catch responsive layout bugs automatically.

### Performance Baseline

While running the QA tests, capture basic performance metrics:
- Time to first render
- Time to complete upload workflow
- Time to decrypt and display a download

These become baseline numbers. If a future change makes things noticeably slower, the test catches it.

### Backend Testing Options

Currently the QA stack runs the backend in memory (memoryFS). For some tests, it may be worth running the backend on disk to:
- Test larger file uploads
- Test concurrent operations
- More closely simulate production behaviour

Note this for future consideration. The in-memory backend is sufficient for the v0.3.0 release testing.

## Release Criteria for v0.3.0

### Must Have (Ship Blockers)

- All Category 1 regressions fixed
- QA regression tests written for each fix
- Core workflows pass end-to-end: single file upload, folder upload, gallery view, folder view, download, friendly token resolution
- No broken routes (all routes from v0.2.x work in v0.3.0)
- Encryption round-trip verified (upload, encrypt, download, decrypt, compare)

### Should Have (Fix Before Production, Can Ship to Dev)

- Category 2 bugs with straightforward fixes
- Responsive layout working at common viewport sizes
- Print functionality working for markdown and gallery

### Future (v0.3.x)

- Category 3 feature requests
- Performance baseline established
- Full viewport regression suite
- Backend-on-disk test option

### Non-Negotiable Invariants

A separate document should list the features that must never break under any circumstances:

- Files are always encrypted before upload (never cleartext to server)
- Downloaded data always matches uploaded data after decryption
- The server never receives unencrypted content
- Friendly token resolution produces the correct decryption key
- Access gate blocks access without a valid token

These invariants get permanent, always-running tests. If any of these tests ever fail, it is a critical incident. No release ships with a failing invariant test.
