## What Just Happened

Another major milestone. As part of the v0.3.0 IFD refactoring, we reached the point where we needed to verify that the refactored code actually behaves the same as the original. The question "do we know what good looks like?" drove us to build something that goes well beyond simple testing.

We now have Playwright browser automation running against a fully local stack:

1. Clone the main repo
2. Start the backend in memory (leveraging memoryFS, all web services running locally)
3. Serve the static website locally
4. Playwright launches a browser, navigates to pages, executes workflows, takes screenshots
5. Screenshots are committed to the QA repo
6. The Jekyll documentation site is rebuilt with the updated screenshots

This works in both CI pipeline and Claude Code. The entire stack is self-contained. No external dependencies. No live server required. The open source architecture and the static website design made this possible: you only need two things (the backend and the static files), both of which are available from the repo.

### Confirmed: This Works Inside Claude Web

In a separate session, we confirmed that Claude Web (regular Claude, not Code) can start a local server, launch Playwright, navigate to pages, take screenshots, and share those screenshots. This means the full QA workflow (start stack, run tests, capture screenshots, share results) runs entirely inside a Claude conversation. No CI pipeline needed. No local machine needed. An agent can independently verify the state of the UI and share visual proof.

The first test run achieved 5 out of 7 tests passing (including the critical end-to-end test). The 2 failures are assertion text matching issues, not functional failures: the UI uses Web Components with Shadow DOM, so `page.text_content("body")` does not capture rendered text inside Shadow DOM. The screenshots visually confirm everything works. The fix is to assert on screenshot existence or use `page.locator("send-upload").is_visible()` instead. The architecture is proven. The assertions need a Shadow DOM-aware approach.

## Why This Is Powerful

The screenshots are not just test artifacts. They are documentation. Every screenshot needed for user guides, marketing pages, Sherpa demos, and release notes is generated automatically from the same test infrastructure. When the UI changes, the documentation updates automatically on the next test run.

Screenshot diffing catches visual regressions: if a screenshot changes significantly (beyond minor pixel variations), the new screenshot is committed. The diff is visible in the commit history. Visual changes are tracked the same way as code changes.

## The Browser Automation API

The raw Playwright scripts work, but they need abstraction. Build a two-layer API:

### Layer 1: Component-Level Primitives

Low-level actions that map to the website's components and functionality:

```javascript
// Navigation
await sgQA.goto('/en-gb/')
await sgQA.goto('/en-gb/download/#id/key')

// Upload actions
await sgQA.upload.dropFile('test-photo.png')
await sgQA.upload.dropFolder('/path/to/folder')
await sgQA.upload.selectDeliveryMode('link_with_key')
await sgQA.upload.clickSend()

// Download actions
await sgQA.download.enterToken('gleam-flint-6001')
await sgQA.download.switchToGallery()
await sgQA.download.switchToFolder()
await sgQA.download.clickFile('report.pdf')

// General
await sgQA.screenshot('step-name')
await sgQA.waitForEncryption()
await sgQA.getTransferLink()
```

This layer tests individual components in isolation. Each command maps to a specific UI element or action. It is also the layer that catches regressions: if a button moves, a component breaks, or a workflow step changes, these primitives fail.

### Layer 2: Workflow Commands

High-level commands that execute entire user journeys:

```javascript
// Full upload workflow
const link = await sgQA.workflow.uploadFile('test-photo.png', {
  deliveryMode: 'link_with_key',
  expiry: 3,
  screenshots: true
})

// Full download workflow
await sgQA.workflow.downloadFromLink(link, {
  screenshots: true
})

// Gallery workflow
const galleryLink = await sgQA.workflow.uploadGallery('/path/to/images/', {
  screenshots: true
})
await sgQA.workflow.viewGallery(galleryLink)

// End-to-end round trip
await sgQA.workflow.roundTrip('test-document.pdf', {
  screenshotSteps: ['upload', 'confirm', 'done', 'download', 'view']
})
```

One command, one complete workflow. Screenshots captured at key steps automatically. These are the commands used for smoke tests, documentation generation, and synthetic traffic.

## Three Uses for the Same API

### 1. QA and Regression Testing

Run in CI on every push. Execute all workflows against a local stack. Compare screenshots. Flag regressions. This is the primary use case and the reason the API was built.

The API needs two execution modes:

- **Functional mode:** Execute the workflow as fast as possible. Verify each step succeeds. No screenshots (faster).
- **Screenshot mode:** Execute the workflow with pauses at configured points. Capture screenshots. Compare against baseline.

Some workflows need setup (state) before they can run. For example, testing the gallery view requires a file to have been uploaded first. The API should support chaining: "run the upload workflow up to 'done' as fast as possible, then switch to screenshot mode for the download workflow."

### 2. Automated Documentation

Every screenshot in the documentation is produced by the API. When the UI changes (new theme, new layout, new feature), re-run the screenshot suite. The documentation updates automatically. No manual screenshot-taking. No stale screenshots showing an old UI.

The screenshots commit to the QA repo. The Jekyll site builds from the repo. The documentation is always current.

### 3. Synthetic Traffic Generation

The same API that simulates one user can simulate hundreds. Run the workflow commands in parallel to generate realistic traffic patterns:

- Performance testing: how does the server handle 100 concurrent uploads?
- Cost measurement: what does a traffic spike cost in Lambda invocations and S3 operations?
- Smoke testing: run a round-trip workflow every 10 minutes against production to confirm the service is up
- Monitoring: output the test results as Prometheus metrics, visualise in Grafana

The synthetic traffic is realistic because it uses the same browser automation that tests real workflows. It is not fake HTTP requests. It is actual browser sessions executing actual user journeys.

## Prometheus Output

Test results and synthetic traffic metrics should be output in Prometheus exposition format:

```
# HELP sg_upload_duration_seconds Time to complete upload workflow
# TYPE sg_upload_duration_seconds histogram
sg_upload_duration_seconds_bucket{workflow="single_file",le="5.0"} 48
sg_upload_duration_seconds_bucket{workflow="single_file",le="10.0"} 95
sg_upload_duration_seconds_count{workflow="single_file"} 100

# HELP sg_workflow_success_total Successful workflow completions
# TYPE sg_workflow_success_total counter
sg_workflow_success_total{workflow="upload"} 847
sg_workflow_success_total{workflow="download"} 812
sg_workflow_success_total{workflow="gallery"} 234
```

This feeds directly into the Grafana monitoring stack (see companion Grafana brief). The same dashboard that shows production metrics also shows synthetic test results. A drop in synthetic workflow success rate is an early warning that something is broken, before real users notice.
