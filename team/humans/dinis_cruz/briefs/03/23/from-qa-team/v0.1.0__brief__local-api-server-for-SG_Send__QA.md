## Context

The SG_Send__QA test suite uses Playwright against a `TEST_TARGET_URL` (defaults to `https://send.sgraph.ai`). This brief explains how to run a **real, local SG/Send server** so you can test the UI without hitting production.

---

## Answer: Missing Dependencies

**Assume `sgraph-ai-app-send` will be available at runtime.** Do not write graceful skips — write the fixtures to use the real local server directly.

The package `sgraph-ai-app-send` provides `setup__send_user_lambda__test_server()` which spins up the **full SG/Send User Lambda** — the exact same FastAPI app that runs in production — as a real HTTP server on a random port. The only difference is storage: in-memory (`Storage_FS__Memory`) instead of S3.

- Starts in ~100ms
- Real TCP on `127.0.0.1:{random_port}`
- All vault endpoints: write, read, batch, list, delete, `write-if-match` (CAS)
- No S3, no AWS credentials, no external dependencies
- Context manager — clean slate per test session, auto-cleanup on exit

## Fixture Pattern for SG_Send__QA

Your existing `tests/conftest.py` gets `target_url` from `TEST_TARGET_URL` env var. To wire up a local server, add a session-scoped fixture that starts the server and overrides that URL:

```python
import pytest

from sgraph_ai_app_send.lambda__user.testing.Send__User_Lambda__Test_Server import setup__send_user_lambda__test_server


@pytest.fixture(scope='session')
def send_server():
    """Start a real SG/Send User Lambda on a random port (in-memory storage).

    Returns test_objs with:
        .server_url     — e.g. http://127.0.0.1:54321
        .access_token   — pre-generated random GUID
        .write_key      — pre-generated random GUID
    """
    with setup__send_user_lambda__test_server() as test_objs:
        yield test_objs


@pytest.fixture(scope='session')
def target_url(send_server):
    """Override the default target_url to point at the local server."""
    return send_server.server_url
```

This replaces the existing `target_url` fixture that reads from `TEST_TARGET_URL` env var. Playwright tests hit the local server instead of production — no code changes needed in the test files themselves.

## Authentication

| Header | Value | Required For |
|--------|-------|-------------|
| `x-sgraph-access-token` | `send_server.access_token` | Write, delete, batch |
| `x-sgraph-vault-write-key` | `send_server.write_key` | Write, delete, batch |

Read and list endpoints require **no authentication** (zero-knowledge model).

If your Playwright tests need to set up vault state before testing the UI, use the `send_server` fixture directly:

```python
import requests
import base64

def seed_vault(send_server, vault_id, file_id, data):
    """Write test data to the local server before Playwright tests."""
    headers = {'x-sgraph-access-token'   : send_server.access_token,
               'x-sgraph-vault-write-key': send_server.write_key,
               'content-type'            : 'application/octet-stream'}
    r = requests.put(f'{send_server.server_url}/api/vault/write/{vault_id}/{file_id}',
                     data=data, headers=headers)
    assert r.status_code == 200
```

## Available Endpoints

All production User Lambda endpoints are available:

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| PUT | `/api/vault/write/{vault_id}/{file_id}` | Yes | Write a single blob |
| GET | `/api/vault/read/{vault_id}/{file_id}` | No | Read a single blob (binary) |
| GET | `/api/vault/read-base64/{vault_id}/{file_id}` | No | Read as base64 JSON |
| DELETE | `/api/vault/delete/{vault_id}/{file_id}` | Yes | Delete a blob |
| POST | `/api/vault/batch/{vault_id}` | Yes | Batch operations (incl. `write-if-match` CAS) |
| GET | `/api/vault/list/{vault_id}?prefix=` | No | List files by prefix |
| GET | `/info/health` | No | Health check |

## How It Works Internally

```
Playwright (Chromium)
    │
    │  browses http://127.0.0.1:{port}/...
    │
    ▼
┌──────────────────────────────────────────────────┐
│  uvicorn (background thread, random port)         │
│                                                   │
│  Fast_API__SGraph__App__Send__User                │
│  (identical to production Lambda)                 │
│                                                   │
│  Routes:                                          │
│    Routes__Transfers                              │
│    Routes__Presigned                              │
│    Routes__Vault__Pointer  ← batch, list, CAS     │
│                                                   │
│  Storage:                                         │
│    Storage_FS__Memory  ← in-memory, no S3         │
└──────────────────────────────────────────────────┘
```

Each `with setup__send_user_lambda__test_server()` block starts with a clean slate. When the context manager exits, all data disappears.

## Troubleshooting

| Symptom | Cause | Fix |
|---------|-------|-----|
| `ModuleNotFoundError: sgraph_ai_app_send` | Package not installed | `pip install sgraph-ai-app-send` |
| `403 on write/batch` | Wrong access token or write key | Use `send_server.access_token` and `send_server.write_key` |
| `400 Missing write key` | Header `x-sgraph-vault-write-key` not set | Add the header to your request |
| Vault write returns 403 after first write | Vault already has a different write_key | Each vault_id gets one write_key on first write. Use a unique vault_id per test. |
| Slashed file_ids don't work with individual PUT | URL path captures only to first slash | Use the batch endpoint for `bare/` paths (file_ids with slashes) |
