## 1. Executive Summary

This document maps AWS Managed Grafana, AWS Managed Prometheus, and AWS OpenSearch against our existing SGraph Send infrastructure to determine what each service can replace, what each adds, and where the gaps are.

**The short version:**

| Service | What It Replaces In Our Stack | What It Adds |
|---|---|---|
| **Managed Grafana** | The admin UI `metrics-dashboard` component + our custom `Routes__Metrics` + `Service__Metrics__Cache` | Real dashboarding, alerting, multi-data-source correlation, shareable dashboards, no custom UI maintenance |
| **Managed Prometheus** | Our `CloudWatch__Client` + `Service__Metrics__Collector` (the entire metrics collection pipeline) | PromQL queries, long-term metrics storage, recording rules, alerting rules, standard exporters ecosystem |
| **OpenSearch** | Our `CloudFront__Logs__Collector` (the tab-separated-file parser reading from S3) | Full-text log search, anomaly detection, log correlation with metrics, dashboards for access patterns, retention management |

All three are already available as native data sources in Managed Grafana, meaning Grafana becomes the single pane of glass that unifies metrics (from Prometheus), logs (from OpenSearch), and AWS-native metrics (from CloudWatch).

## 4. AWS Managed Prometheus — Integration Map

### 4.1 The Challenge: Lambda Is Not a Traditional Prometheus Target

Standard Prometheus works by scraping `/metrics` endpoints on long-running services. Our Lambdas are ephemeral — they spin up, handle a request, and shut down. There is no persistent `/metrics` endpoint to scrape.

**Three paths to get metrics into Managed Prometheus:**

| Path | How It Works | Effort | Best For |
|---|---|---|---|
| **Path A: CloudWatch Metric Streams → Firehose → Prometheus** | CloudWatch streams metrics via Firehose to a Lambda that pushes to AMP's `remote_write` endpoint | Medium (CDK/CFN setup) | All existing CloudWatch metrics (Lambda, S3, CloudFront) — zero application code changes |
| **Path B: Lambda Telemetry API extension** | A Lambda layer receives per-invocation telemetry and pushes to AMP's `remote_write` endpoint | High (extension development) | Fine-grained per-invocation metrics, custom labels |
| **Path C: Application-level `remote_write` in Lambda code** | Our Lambda code pushes custom metrics directly to AMP's `remote_write` API on each invocation | Low (add HTTP calls to existing code) | Custom business metrics (transfer counts, token usage, vault operations) |

### 4.2 Recommended Approach: Path A + Path C

**Path A** gets all infrastructure metrics (Lambda, S3, CloudFront) into Prometheus with zero application code changes. This is a pure infrastructure setup: CloudWatch Metric Stream → Firehose → transformer Lambda → AMP `remote_write`. AWS provides a CDK template for this exact pattern.

**Path C** adds custom business metrics. In our existing service code, we add lightweight metric pushes:

```python
# Conceptual — what the code change looks like
# In Transfer__Service.create_transfer():
self.metrics.increment('sgraph_transfers_created_total',
                       labels={'stage': self.stage})

# In Routes__Transfers.upload__transfer_id():
self.metrics.observe('sgraph_transfer_upload_bytes',
                     value=len(payload),
                     labels={'stage': self.stage})
```

The metrics client would batch these and push to AMP's `remote_write` API. For a Lambda function handling ~100 requests/minute, this adds negligible overhead.

### 4.3 What Custom Metrics We'd Push

| Metric Name | Type | Labels | Source |
|---|---|---|---|
| `sgraph_transfers_created_total` | Counter | stage, token_name | Transfer__Service |
| `sgraph_transfers_upload_bytes` | Histogram | stage | Routes__Transfers |
| `sgraph_transfers_download_total` | Counter | stage | Routes__Transfers |
| `sgraph_vault_writes_total` | Counter | stage | Routes__Vault__Pointer |
| `sgraph_vault_reads_total` | Counter | stage | Routes__Vault__Pointer |
| `sgraph_token_validations_total` | Counter | stage, result (valid/invalid/expired) | Routes__Transfers |
| `sgraph_presigned_uploads_total` | Counter | stage, mode (direct/multipart) | Routes__Presigned |
| `sgraph_early_access_signups_total` | Counter | stage | Routes__Early_Access |
| `sgraph_room_joins_total` | Counter | stage | Routes__Join |

### 4.4 PromQL Queries We'd Use in Grafana

```promql
# Transfer rate over time
rate(sgraph_transfers_created_total[5m])

# Upload size distribution (p50, p95, p99)
histogram_quantile(0.95, rate(sgraph_transfers_upload_bytes_bucket[5m]))

# Error rate by Lambda function
rate(aws_lambda_errors_sum[5m]) / rate(aws_lambda_invocations_sum[5m])

# Token validation failure rate
rate(sgraph_token_validations_total{result="invalid"}[5m])
  / rate(sgraph_token_validations_total[5m])

# Vault read/write ratio
rate(sgraph_vault_reads_total[1h]) / rate(sgraph_vault_writes_total[1h])
```

## 5. AWS OpenSearch — Gap Filler

### 5.1 What Grafana + Prometheus Cannot Do

Prometheus is a **metrics** store (numeric time series). It cannot store or search **log data**. Grafana can visualise logs but needs a log-capable backend. This is the gap OpenSearch fills.

| Capability | CloudWatch | Prometheus | OpenSearch |
|---|---|---|---|
| Numeric time-series metrics | Yes | Yes (better) | No |
| Aggregate metrics (p50, p95) | Yes | Yes (better) | No |
| Log storage and search | CloudWatch Logs Insights | No | Yes |
| Full-text search across logs | Limited (Logs Insights) | No | Yes (Lucene/PPL) |
| Anomaly detection on logs | No | No | Yes (Random Cut Forest ML) |
| Log correlation with metrics | Manual | No | Yes (via Grafana) |
| Access pattern analysis | No | No | Yes (aggregations, geo, user-agent) |
| Retention management | CloudWatch retention policies | AMP retention (150 days default) | ISM policies (automated index lifecycle) |
| Cost at scale | Expensive for high-volume queries | Cheap for metrics | Cheap for log storage, expensive for compute |

### 5.2 What We'd Send to OpenSearch

**Source 1: CloudFront Access Logs (replacing our current S3 + parser)**

Currently, Firehose delivers CloudFront real-time logs to S3 as GZIP tab-separated files. Our `CloudFront__Logs__Collector` reads these files, parses them, and produces an in-memory summary. This is fragile and non-queryable.

**Target architecture:** Add OpenSearch as a **second Firehose destination** (or replace S3). Firehose natively supports OpenSearch as a destination — it auto-creates indices, handles batching, and retries on failure.

```
CloudFront Real-Time Logs
        │
        ▼
  Kinesis Data Stream
        │
        ▼
  Firehose Delivery Stream
        ├──► S3 (archive — keep for compliance)
        └──► OpenSearch Service (searchable index)
                 │
                 ▼
          Grafana → OpenSearch data source
                 │
                 ▼
          Dashboards: top paths, status codes,
          edge locations, cache hit rates,
          anomaly detection
```

This eliminates our `CloudFront__Logs__Collector` entirely (another ~150 lines of code including the raw boto3 violation).

**Source 2: Lambda Application Logs (future)**

Lambda functions emit logs to CloudWatch Logs. These can be streamed to OpenSearch via a CloudWatch Logs subscription filter → Firehose → OpenSearch pipeline. This would give us full-text search across application logs (Python print statements, error tracebacks, request/response logging).

This is a Phase 2 item — CloudFront logs first, application logs later.

### 5.3 OpenSearch Anomaly Detection

OpenSearch includes a built-in anomaly detection engine based on the Random Cut Forest (RCF) algorithm. This is where OpenSearch goes beyond what Prometheus can do.

| Detection | What It Finds | Index Source |
|---|---|---|
| Traffic spike by country | Unusual request volume from a specific country code | CloudFront logs (`x-edge-location`) |
| Path scanning | Rapid sequential requests to many different paths from one edge location | CloudFront logs (`cs-uri-stem`) |
| Error rate anomaly | Sudden increase in 4xx or 5xx responses | CloudFront logs (`sc-status`) |
| Cache miss anomaly | Sudden drop in cache hit rate | CloudFront logs (`x-edge-result-type`) |
| User-agent anomaly | New or unusual user-agent patterns (potential bot/scraper) | CloudFront logs (`cs-user-agent`) |

These detectors run continuously and generate alerts when anomalies are detected, which can be surfaced in Grafana via the OpenSearch data source.

### 5.4 OpenSearch Index Design

```
Index pattern: cloudfront-logs-{YYYY.MM.dd}

Mapping:
{
  "timestamp"           : "date"        (epoch seconds → date)
  "sc_status"           : "keyword"     (HTTP status code)
  "sc_status_group"     : "keyword"     (2xx, 3xx, 4xx, 5xx — derived)
  "sc_bytes"            : "long"        (response size)
  "cs_method"           : "keyword"     (GET, POST, etc.)
  "cs_uri_stem"         : "keyword"     (URL path — keyword for exact match)
  "cs_uri_stem_text"    : "text"        (URL path — text for full-text search)
  "cs_protocol_version" : "keyword"     (HTTP/1.1, HTTP/2, HTTP/3)
  "sc_content_type"     : "keyword"     (response MIME type)
  "cs_user_agent"       : "text"        (browser user agent — searchable)
  "x_edge_location"     : "keyword"     (CloudFront POP)
  "x_edge_result_type"  : "keyword"     (Hit, Miss, Error)
  "time_taken"          : "float"       (total serve time in seconds)
  "time_to_first_byte"  : "float"       (TTFB from origin)
  "ssl_protocol"        : "keyword"     (TLS version)
  "cs_host"             : "keyword"     (Host header — which domain)
}

ISM (Index State Management) policy:
  - Hot:    0-7 days   (full replicas, fast queries)
  - Warm:   7-30 days  (reduced replicas)
  - Delete: 30+ days   (auto-purge)
```

**Privacy note:** Our CloudFront real-time log configuration already excludes `c-ip` (client IP). This means the OpenSearch index contains no PII — consistent with our zero-knowledge design principle.

## 9. What Each Service Cannot Do (Honest Gaps)

### 9.1 Managed Grafana Limitations

| Limitation | Impact | Mitigation |
|---|---|---|
| No browser direct access to data sources | All queries go through Grafana backend — cannot query OpenSearch from browser JS | Not an issue for dashboarding; an issue if we wanted to embed Grafana panels in our admin UI |
| Single VPC connection per workspace | If OpenSearch is in a VPC, Grafana can only connect to one VPC | Use public OpenSearch domain with fine-grained access control, or VPC peering |
| Dashboard-as-code is JSON, not Python | Our stack is Python-first; Grafana dashboards are JSON configs | Accept this — dashboards are configuration, not application code |

### 9.2 Managed Prometheus Limitations

| Limitation | Impact | Mitigation |
|---|---|---|
| Cannot scrape Lambda functions directly | Lambda is ephemeral — no persistent `/metrics` endpoint | Use `remote_write` from Lambda code (Path C) or CloudWatch Metric Streams (Path A) |
| Metrics only — no logs or traces | Cannot replace CloudWatch Logs or X-Ray | Use OpenSearch for logs, X-Ray or Tempo for traces (future) |
| 150-day default retention | Historical analysis beyond 5 months requires export | Sufficient for our current needs; export to S3 if needed |
| Cost scales with active series count | High-cardinality labels (e.g., per-transfer-id) would be expensive | Use low-cardinality labels only (stage, function_name, token_name) |

### 9.3 OpenSearch Limitations

| Limitation | Impact | Mitigation |
|---|---|---|
| Not a metrics store | Cannot efficiently do PromQL-style time-series queries | Use Prometheus for metrics, OpenSearch for logs only |
| Compute cost scales with query complexity | Complex aggregations on large indices can be slow/expensive | ISM policy to age out old data; use date-range filters in queries |
| Requires index management | Indices grow indefinitely without ISM policies | Configure ISM from day one (§5.4) |
| No built-in PII detection | If our CF log config changes to include c-ip, OpenSearch would store it | Our CF log config already excludes c-ip; add a Firehose transform Lambda as a safety net |

### 9.4 Gaps None of the Three Services Fill

| Gap | What's Needed | Candidate Solution |
|---|---|---|
| **Distributed tracing** | End-to-end request tracing across User Lambda → Admin Lambda | AWS X-Ray (already integrated with Lambda natively) or Grafana Tempo |
| **Synthetic monitoring** | Proactive health checks from external locations | CloudWatch Synthetics (canaries) or Grafana synthetic monitoring |
| **Real user monitoring (RUM)** | Browser-side performance metrics (page load, JS errors) | CloudWatch RUM or custom telemetry in our Web Components |
| **Cost monitoring** | AWS spend tracking and forecasting | CloudWatch Billing metrics → Grafana, or AWS Cost Explorer |
| **Deployment tracking** | Correlating metric changes with deployments | Grafana annotations via CI/CD (push annotation on deploy) |

## 10. Decision Matrix

| Question | Recommendation |
|---|---|
| Start with which service? | **Grafana first** — zero code changes, immediate dashboarding and alerting |
| CloudWatch or Prometheus for infrastructure metrics? | **Both** — CloudWatch direct in Grafana for immediate value, Prometheus via Metric Streams for PromQL power |
| Where do CloudFront logs go? | **OpenSearch** — replaces our fragile S3 parser, adds full-text search and anomaly detection |
| Do we need all three services? | **Yes** — each fills a distinct role: Grafana (visualisation + alerting), Prometheus (metrics store + PromQL), OpenSearch (log store + search + anomaly detection) |
