## 1. Why IFD v2 Exists

IFD v1 proved that overlay-based development works. The v0.2.x series delivered 17 overlays on the upload component alone, with remarkably few side effects. The Web Component isolation and prototype mutation pattern gave genuine independence between versions.

But v1 hit three walls at scale:

**Wall 1: Overlay chains become unreadable.** 17 overlays loading sequentially, each patching methods on the previous. Claude Code could not refactor the resulting 7,221-line monolith — two sessions timed out. The layering was too deep to reason about in one pass, for both LLMs and humans.

**Wall 2: Consolidation is undefined.** IFD v1 says "merge surgical overrides into complete, self-contained files" but gives no specification for how. The v0.3.0 refactoring had to invent a modular architecture (7 shared modules + 6 Shadow DOM steps) from scratch. That is an architectural decision that was not anticipated by the methodology.

**Wall 3: There is no manifest.** The `index.html` is the manifest — 17 `<script>` tags in exact order. Every version change requires an HTML edit. Deployment copies files on top of each other. There is no machine-readable record of what version uses what files.

IFD v2 addresses all three by changing the fundamental unit of change from **methods** to **files**, and introducing a **manifest** as the single source of truth for composition.

## 2. Core Principles

### 2.1 Replace Files, Not Methods

IFD v1 core pattern:
```javascript
// v0.2.5 — patches one method on the v0.2.0 class
SendUpload.prototype.handleDrop = function() { /* new behaviour */ };
```

IFD v2 core pattern:
```
v0.3.1/manifest.json declares: upload-engine.js replaces upload-engine.js from v0.3.0
v0.3.1/upload-engine.js is the complete new file
```

The v2 file completely replaces the v1 file. No patching. No chains. No implicit dependencies between override layers.

**This changes the incentive structure.** In v1, you were rewarded for making small changes inside large files. In v2, you are rewarded for making small files. The smaller the file, the smaller the blast radius when it is replaced. This naturally drives toward more components, more modules, cleaner separation — exactly the architecture that v0.3.0 had to invent manually.

### 2.2 Everything Is Versioned

Every artifact has its own IFD version tree:
- Components (send-upload, send-download, send-access-gate)
- Shared modules (crypto, i18n, api-client, design tokens)
- The loader itself
- The deploy script
- Pages (index, download, gallery, welcome)

Nothing is unversioned. Nothing is "just shared."

### 2.3 The Manifest Is the Source of Truth

A machine-readable JSON manifest declares what files a version contains, what it depends on, and what changed from the previous version. The loader reads the manifest. The deploy script reads the manifest. The SBOM is generated from the manifest. No implicit knowledge.

### 2.4 Offline-First

The resolved file set must work with zero network access. All files, all dependencies, all assets are present locally. The `latest/` production folder is a complete, self-contained application. No CDN calls. No external fonts. No runtime dependency resolution.

## 3. The Two-Layer Dependency Model

Inspired by Poetry (Python), Cargo (Rust), and npm (package-lock.json), IFD v2 uses a two-layer dependency model:

### 3.1 Component Manifest (`manifest.json`)

Each component version declares its files and compatible dependency ranges:

```json
{
  "component": "send-upload",
  "version": "0.3.1",
  "description": "Upload workflow — 6-step encrypted file transfer",
  "files": {
    "js": [
      "upload-constants.js",
      "upload-crypto.js",
      "upload-file-utils.js",
      "upload-thumbnails.js",
      "upload-folder.js",
      "upload-engine.js",
      "send-upload.js",
      "upload-step-select/upload-step-select.js",
      "upload-step-delivery/upload-step-delivery.js",
      "upload-step-share/upload-step-share.js",
      "upload-step-confirm/upload-step-confirm.js",
      "upload-step-progress/upload-step-progress.js",
      "upload-step-done/upload-step-done.js"
    ],
    "css": [
      "send-upload.css",
      "upload-step-select/upload-step-select.css",
      "upload-step-delivery/upload-step-delivery.css",
      "upload-step-share/upload-step-share.css",
      "upload-step-confirm/upload-step-confirm.css",
      "upload-step-progress/upload-step-progress.css",
      "upload-step-done/upload-step-done.css"
    ],
    "html": [
      "upload-step-select/upload-step-select.html",
      "upload-step-delivery/upload-step-delivery.html",
      "upload-step-share/upload-step-share.html",
      "upload-step-confirm/upload-step-confirm.html",
      "upload-step-progress/upload-step-progress.html",
      "upload-step-done/upload-step-done.html"
    ]
  },
  "dependencies": {
    "send-step-indicator": ">=0.1.0",
    "send-access-gate": ">=0.2.0",
    "core/crypto": ">=0.1.0",
    "core/api-client": ">=0.1.0",
    "core/i18n": ">=0.1.0"
  },
  "replaces_from_previous": [
    "upload-engine.js"
  ]
}
```

**The `dependencies` field uses compatible-version ranges** — it says "I work with any version of `send-step-indicator` at 0.1.0 or above." This is the `pyproject.toml` equivalent. It declares intent, not resolution.

### 3.2 Release Lock File (`manifest.lock`)

At the page/release level, a lock file resolves all ranges to exact versions:

```json

  "generated": "2026-03-22T16:00:00Z",
  "generator": "sg-manifest-resolve v0.1.0",
  "resolved": {
    "send-upload": {
      "version": "0.3.1",
      "path": "components/send-upload/v0/v0.3/v0.3.1/",
      "integrity": "sha256-abc123..."
    },
    "send-download": {
      "version": "0.1.2",
      "path": "components/send-download/v0/v0.1/v0.1.2/",
      "integrity": "sha256-def456..."
    },
    "send-access-gate": {

      "version": "0.1.0",

      "integrity": "sha256-stu901..."

    "core/i18n": {

      "path": "core/i18n/v0/v0.1/v0.1.0/",
      "integrity": "sha256-vwx234..."
    },
    "core/design-tokens": {
      "version": "0.1.0",
      "path": "core/design-tokens/v0/v0.1/v0.1.0/",
      "integrity": "sha256-yza567..."
    },
    "core/loader": {
      "version": "0.1.0",
      "path": "core/loader/v0/v0.1/v0.1.0/",

    }
  }

```

**Key properties of the lock file:**

- **Exact versions only.** No ranges. Every component pinned to one version.
- **Paths are relative.** Resolved from the lock file location. Works offline, works on any host.
- **Integrity hashes.** SHA-256 of the manifest.json content for each component. Detects tampering. Enables offline verification.
- **The lock file IS the SBOM.** It is a complete, machine-readable inventory of every component, exact version, and file path in the application. No separate SBOM generation needed — the lock file already satisfies the requirement.
- **The lock file IS the `latest/` build recipe.** The deploy script reads the lock file, copies all resolved files into `latest/`. If the lock file is correct, `latest/` is correct. No other logic needed.

### 3.3 The Poetry Analogy

| Poetry | IFD v2 | Purpose |
|--------|--------|---------|
| `pyproject.toml` | Component `manifest.json` | Declare files + compatible dependency ranges |
| `poetry.lock` | Release `manifest.lock` | Resolve all ranges to exact versions |
| `poetry install` | `sg-manifest resolve` | Read manifests, resolve ranges, write lock file |
| `poetry build` | `sg-manifest build` | Read lock file, copy resolved files to `latest/` |
| Package on PyPI | Component in IFD tree | Versioned, self-contained, with declared dependencies |

### 3.4 SBOM Generation

The lock file already contains the full dependency graph. To produce a standard SBOM (CycloneDX or SPDX), a simple transform reads the lock file and emits the standard format:

```bash
sg-manifest sbom --format cyclonedx --input manifest.lock --output sbom.json
```

No additional scanning or analysis needed. The lock file is the single source of truth. Every component listed, every version pinned, every file path known. This is a direct consequence of getting the manifest right — the SBOM falls out for free.

## 5. The Loader

### 5.1 What It Does

The loader is a small, versioned component in `core/loader/`. Its responsibilities:

1. Read the release `manifest.lock`
2. Resolve all component paths from the lock file
3. Load CSS files first (link tags, order from lock file)
4. Load JS files in dependency order (script tags)
5. Report loading errors (missing files, integrity mismatches)

### 5.2 The Loader Is Versioned

The loader is versioned like everything else. It has its own `manifest.json`. It appears in the lock file. The only thing special about the loader is that it must be loaded first — the page HTML has exactly one hardcoded script tag:

```html
<!DOCTYPE html>
<html lang="en-gb">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>SG/Send — Encrypted File Transfer</title>
</head>
<body>
  <send-upload></send-upload>

  <script src="loader.js"></script>
  <script>
    SGLoader.init({
      lock: './manifest.lock',
      version: localStorage.getItem('sg-send-version') || null
    });
  </script>
</body>
</html>
```

**In `latest/`, `loader.js` is just a copy** of the resolved loader version from the lock file. One hardcoded reference. Everything else is manifest-driven.

### 5.3 Loader Boot Sequence

```
1. SGLoader.init() called
2. Fetch manifest.lock (or use embedded lock if offline)
3. Parse resolved component list
4. Compute load order (dependencies first, then dependents)
5. Inject <link> tags for all CSS files
6. Inject <script> tags for all JS files, in order
7. When all scripts loaded, dispatch 'sg-ready' event
8. If any file fails to load, dispatch 'sg-load-error' with details
```

### 5.4 Loader Failure Mode

If the loader cannot read the lock file or a file fails to load:
- Log the error to console with full details
- Dispatch `sg-load-error` CustomEvent on `document`
- Display a minimal fallback message in the page body (no dependency on any component)
- The loader must never throw an unhandled exception

### 5.5 Offline Support

For fully offline deployments (no internet access), two options:

**Option A: Embedded lock.** The build script inlines the lock file contents into `loader.js` as a JSON constant. No fetch needed. The loader reads from its own embedded data.

**Option B: Lock file alongside.** The `latest/` folder includes `manifest.lock` as a static file. The loader fetches it via relative path. Works offline because all files are already local.

Both work. Option A is more resilient (single file). Option B is more inspectable (lock file is a standalone artifact you can read). **Recommendation: Option B for dev, Option A for production builds.**

## 6. Version Selection and localStorage

### 6.1 The Mechanism

The loader accepts an optional version override from localStorage:

```javascript
const version = localStorage.getItem('sg-send-version');
```

If set, the loader loads the lock file for that version instead of the default. This enables:
- Live version switching in the browser (change one value, reload)
- A/B testing (different users on different versions)
- Developer debugging (load any version without redeploying)

### 6.2 The Official Release

There is always an "official" release version. This is the version served when localStorage has no override. In production, the `latest/` folder is built from this version's lock file.

When a new version goes to production:
1. The lock file for the new version is built
2. `latest/` is rebuilt from the new lock file
3. All users without a localStorage override automatically get the new version
4. Users with a stale localStorage override continue on their pinned version

### 6.3 Edge Cases and Guardrails

| Edge case | Guardrail |
|-----------|-----------|
| Stale localStorage pins user to old version forever | **Max-age:** Loader checks a `sg-send-version-set-at` timestamp. If older than 7 days, clear the override and use official. |
| localStorage points to a version that no longer exists on server | **Fallback:** If lock file fetch for the overridden version returns 404, clear localStorage and fall back to official. Log a warning. |
| User sets a nonsense value | **Validation:** Loader validates the version string format before attempting to use it. Reject anything that doesn't match `\d+\.\d+\.\d+`. |
| Multiple tabs with different versions | **Not a problem.** Each tab loads independently. localStorage is shared but the version is read once at page load. |
| Admin wants to force all users to new version | **Server override:** A header or meta tag `<meta name="sg-send-force-version" content="0.3.2">` in the HTML. Loader checks this first, overrides localStorage if present. This is the "everyone moves now" lever. |

### 6.4 The Version UI (Future — PROPOSED)

A small debug panel (accessible via keyboard shortcut or URL param) showing:
- Current loaded version
- Official version
- Available versions (from a versions index)
- Toggle to switch versions

This is not needed for launch. It is useful for testing and debugging.

## 7. The `latest/` Build

### 7.1 What It Is

`latest/` is a flat directory containing all files needed to serve the application. It is built entirely from the lock file. No other input.

### 7.2 Build Process

```
Input:  manifest.lock
Output: latest/

Algorithm:
  1. Read manifest.lock
  2. For each resolved component:
     a. Read its manifest.json to get file list
     b. Copy all files to latest/ preserving component-relative paths
        e.g. components/send-upload/v0/v0.3/v0.3.1/upload-engine.js
             → latest/components/send-upload/upload-engine.js
  3. Copy the loader to latest/loader.js
  4. Copy the lock file to latest/manifest.lock
  5. Copy page HTML files to latest/ (e.g. latest/en-gb/index.html)
  6. Generate integrity manifest (hash of every file in latest/)
  7. Done
```

### 7.3 Properties

- **Deterministic.** Same lock file always produces the same `latest/`. No ambient state.
- **Offline-complete.** Everything needed is in `latest/`. No external fetches.
- **Inspectable.** The lock file is included in `latest/`. You can always see exactly what was resolved.
- **Reproducible.** Given the lock file and the component tree, anyone can rebuild `latest/` and get an identical result. The integrity hashes verify this.

### 7.4 Comparison with Current Deploy

| Aspect | Current (v1) | IFD v2 |
|--------|-------------|--------|
| Build input | Implicit: list of v0.2.x directories | Explicit: `manifest.lock` |
| File selection | Convention: later version directories win | Manifest: lock file lists exact files |
| Conflict resolution | Last-write-wins (overlay model) | No conflicts: lock file resolves to one version per component |
| Deploy script complexity | 669 lines, overlay logic, i18n generation, multi-step | Read lock file, copy files, done |
| Rollback | Re-sync a previous release folder | Point to previous lock file, rebuild `latest/` |
| Audit trail | Check S3 file timestamps | Read the lock file (it IS the audit trail) |
| Offline guarantee | Not explicitly designed for | Built in: `latest/` is self-contained |

## 8. Deploy Script v2

The current `deploy_static_site.py` (669 lines) implements the v1 overlay model. IFD v2 replaces this with a simpler script that is itself IFD-versioned.

### 8.1 Core Operations

```
sg-manifest resolve    Read all component manifests, resolve ranges, write manifest.lock
sg-manifest build      Read manifest.lock, produce latest/
sg-manifest deploy     Sync latest/ to S3 + CloudFront invalidation
sg-manifest sbom       Generate SBOM from manifest.lock
sg-manifest validate   Check all lock file references exist and integrity hashes match
```

### 8.2 The Deploy Script Is Versioned

The deploy script lives in:

```
scripts/
  sg-manifest/
    v0/
      v0.1/
        v0.1.0/
          manifest.json
          sg_manifest.py
          resolve.py
          build.py
          deploy.py
          sbom.py
          validate.py
```

This means the deploy script itself follows IFD. A new version of the deploy script does not automatically affect older deployments. The CI workflow pins to a specific deploy script version.

### 8.3 CI Workflow Changes

The `.github/workflows/deploy-ui-user.yml` simplifies:

```yaml
steps:
  - name: Resolve dependencies
    run: python scripts/sg-manifest/v0/v0.1/v0.1.0/sg_manifest.py resolve

  - name: Validate lock file
    run: python scripts/sg-manifest/v0/v0.1/v0.1.0/sg_manifest.py validate

  - name: Build latest
    run: python scripts/sg-manifest/v0/v0.1/v0.1.0/sg_manifest.py build

  - name: Deploy
    run: python scripts/sg-manifest/v0/v0.1/v0.1.0/sg_manifest.py deploy --env ${{ env.DEPLOY_ENV }}
```

The i18n locale generation integrates into the build step: the build script generates locale pages from the resolved English templates, just as it does today, but driven by the manifest rather than by scanning directories.

## 9. Migration Path

### 9.1 Constraints

- v0.3.0 is the working UI. Do not break it during migration.
- No need for v0.2.x backwards compatibility (no live deployments).
- Incremental refactoring: each step produces a working system.
- v0.3.0 is the canary: if it stops working, something is wrong.

### 9.2 Phase Sequence

**Phase 1: Manifest schema + tooling (no code changes to UI)**

1. Define the `manifest.json` schema (component manifest)
2. Define the `manifest.lock` schema (release lock file)
3. Build `sg-manifest resolve` (reads manifests, writes lock file)
4. Build `sg-manifest validate` (checks lock file integrity)
5. Build `sg-manifest build` (produces `latest/` from lock file)

**Acceptance:** `sg-manifest build` can produce a `latest/` folder from a hand-written lock file that contains exactly the same files as the current v0.3.0.

**Phase 2: Add manifests to v0.3.0 (additive, no restructuring)**

1. Write `manifest.json` for each existing v0.3.0 component (send-upload, send-download, send-access-gate, send-step-indicator, send-test-files)
2. Write `manifest.json` for each core module (crypto, api-client, i18n, design-tokens, icons)
3. Write `manifest.json` for the upload page
4. Write a top-level `manifest.lock` that resolves to the current v0.3.0 files
5. Verify: `sg-manifest build` from this lock file produces a working `latest/`

**Acceptance:** The manifest-built `latest/` is byte-identical to the manually-assembled v0.3.0. The canary still works.

**Phase 3: Build the loader**

1. Create `core/loader/v0/v0.1/v0.1.0/` with `manifest.json` and `loader.js`
2. Build the loader: reads lock file, injects script/link tags, dispatches events
3. Create a new `index.html` that uses only the loader (no hardcoded script tags except the loader itself)
4. Test: loader-driven page loads and functions identically to the hardcoded-script-tag page

**Acceptance:** Side-by-side: loader-driven page and current page are functionally identical. The canary still works.

**Phase 4: Restructure into component IFD trees**

1. Move component files from `_common/js/components/{name}/` into `components/{name}/v0/v0.{major}/v0.{major}.0/`
2. Move core files from `_common/js/` and `_common/css/` into `core/{name}/v0/v0.1/v0.1.0/`
3. Update manifests to reflect new paths
4. Re-resolve lock file
5. Rebuild `latest/`
6. Verify the canary

**Acceptance:** All files now live in the IFD v2 tree structure. Lock file resolves correctly. `latest/` works.

**Phase 5: Deploy script migration**

1. Build `sg-manifest deploy` (reads lock file → builds latest → syncs to S3 → invalidates CloudFront)
2. Update CI workflow to use manifest-driven deploy
3. Test on dev environment
4. Retire the v1 overlay deploy logic

**Acceptance:** CI deploys v0.3.0 via the manifest pipeline. The canary still works on dev.send.sgraph.ai.

**Phase 6: First IFD v2 minor version**

1. Make a small change to one component (e.g. fix a string in send-upload)
2. Create `components/send-upload/v0/v0.3/v0.3.1/` with only the changed file + manifest
3. Update the top-level manifest ranges
4. Re-resolve lock file
5. Build + deploy
6. Verify: only the changed component updated, everything else untouched

**Acceptance:** The IFD v2 cycle works end-to-end: change one file, update manifest, resolve, build, deploy.

## 10. Relationship to Explorer/Villager Cycle

The v0.2.x → v0.3.0 transition was an Explorer-to-Villager handover. IFD v2 makes this cycle explicit and tooling-supported:

**Explorer phase (minor versions):**
- Add a new file to a component: create v0.3.2 with the new file + updated manifest
- Modify an existing file: create v0.3.3 with the replacement file + updated manifest
- Add a new component: create `components/new-thing/v0/v0.1/v0.1.0/` with manifest
- Each minor version is small, focused, independently deployable

**Villager phase (major versions):**
- Consolidate: create v0.4.0 that contains all files from the v0.3.x series, merged and cleaned
- The lock file diff between v0.3.x and v0.4.0 is the exact record of what changed
- No feature loss possible: the manifest for v0.3.x lists every file; the manifest for v0.4.0 must account for each one

**The manifest solves the "consolidation is lossy" problem.** In v1, the gap analysis had to manually compare overlays to find missing features. In v2, the manifests are machine-readable: a diff between the v0.3.x manifests and the v0.4.0 manifest immediately shows any file that was present in v0.3.x but absent in v0.4.0. The tooling catches what human review missed.

## 11. How IFD v2 Addresses Known Problems

| Problem | IFD v1 Behaviour | IFD v2 Solution |
|---------|------------------|-----------------|
| 17 overlays patching methods in sequence | Prototype mutation chain, implicit load order | File replacement, explicit manifest |
| 7,221-line monolith after consolidation | No incentive for small files | Small-file incentive (replace whole files, not methods) |
| `index.html` changes on every release | Script tags are the manifest | Loader reads manifest; HTML is static |
| `[object Object]` API mismatch bug | No component contracts | Versioned dependencies with integrity hashes |
| Claude Code cannot reason about full overlay chain | Deep layering exceeds context | Small files, each fully comprehensible in one pass |
| Consolidation missed 16 features | No machine-readable feature inventory | Manifest diff catches missing files |
| Deploy script has overlay-specific logic (669 lines) | Built for overlay model | Read lock file → copy files → deploy |
| Unversioned `_common/` causes accidental breakage | Shared code not versioned | Everything versioned, explicit dependencies |
| No SBOM | No machine-readable dependency list | Lock file IS the SBOM |
| No offline guarantee | External font references still exist | Offline-first: `latest/` is self-contained |
| Patch 0003 failed (generated against wrong base) | No tooling for patch generation | Manifest provides the exact file list for any version |
| v0.2.9 became a no-op (logic moved to v0.2.6) | Overlays reference each other informally | Each version is self-contained; no cross-version references |

## 12. Open Questions

| # | Question | Options | Recommendation |
|---|----------|---------|----------------|
| 1 | Should `manifest.json` use JSON or YAML? | JSON is universal and parseable everywhere. YAML is more readable. | **JSON.** No build tool needed. Parseable in browser, Python, CLI. |
| 2 | Should the loader support hot-reload (change manifest, reload components without full page refresh)? | Useful for dev, complex to implement. | **Not for v0.1.0.** Full page reload on version change. Add hot-reload as a future enhancement. |
| 3 | Should component manifests declare their Web Component tag names? | Would enable the loader to verify registration. | **Yes.** Add `"elements": ["send-upload"]` to manifest.json. |
| 4 | Should sub-components (upload-step-select, etc.) be independent components with their own IFD trees? | More granular versioning. More manifests to maintain. | **Not initially.** Keep them as files within the parent component. Promote to independent components if they start being reused across pages. |
| 5 | How should the i18n locale generation integrate? | Part of the build step. Part of the loader. Pre-built. | **Part of the build step.** `sg-manifest build` generates locale variants from the English templates, just as today's CI does. |
| 6 | Should vendor libraries (JSZip, pdf.js, qr-code) be versioned components? | Gives explicit control. Adds manifest overhead. | **Yes.** Wrap each in a component with a manifest. Makes the SBOM complete and vendor updates explicit. |
