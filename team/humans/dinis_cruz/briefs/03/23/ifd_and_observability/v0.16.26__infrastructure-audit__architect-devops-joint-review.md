# Infrastructure Audit — SGraph Send v0.16.26
**Joint Review: Architect + DevOps**
Date: 19 March 2026 · Region: eu-west-2 (primary), us-east-1 (CloudFront metrics)

---

## 1. Executive Summary

This is a code-verified audit of the SGraph Send infrastructure as of v0.16.26. Every claim below was confirmed by reading source code, CI workflows, deployment scripts, and configuration files — not briefs or documentation.

**What exists:** A well-structured two-Lambda architecture behind CloudFront, with S3 as the storage backend, Kinesis Data Firehose for real-time log delivery, CloudWatch for metrics collection, and a Python-first CI/CD pipeline via GitHub Actions. The IFD overlay deployment model for static UIs is mature and auto-discovering.

**What's good:** Storage abstraction via Memory-FS works cleanly across dev/test/prod. The CI pipeline is reliable (tests → tag → parallel Lambda deploy). The metrics pipeline has proper Type_Safe schemas, a stub mode for testing, and a caching layer. The static site deploy script is production-grade (per-extension cache control, version archiving, CloudFront invalidation).

**What needs attention:** Two raw `boto3` usages violate the stack rules. All infrastructure is Console-configured (no IaC). The S3 request metrics filter is assumed, not verified. The CloudFront routing rules that split `/api/*` from `/*` are invisible to code review. The Firehose pipeline configuration is undocumented. Several proposed monitoring improvements (Grafana, Prometheus, SRE agents) exist only in briefs.

## 4. Findings

### 4.1 Stack Violations (HIGH)

**Finding V-1: Raw `boto3` in `CloudFront__Logs__Collector.setup()`**

```python
def setup(self):
    import boto3
    self._s3_client = boto3.client('s3', region_name=str(self.region))
```

The stack rule (`.claude/CLAUDE.md` §Stack, Rule: "All AWS calls go through `osbot-aws`, never `boto3` directly") is violated. The S3 reads for CloudFront log files should go through an osbot-aws S3 client.

**Finding V-2: Raw `boto3` in `CloudWatch__Client.setup()`**

```python
def setup(self):
    import boto3
    self._boto3_client    = boto3.client('cloudwatch', region_name=str(self.region))
    self._boto3_client_cf = boto3.client('cloudwatch', region_name='us-east-1')
```

Same violation. The CloudWatch client creates two raw boto3 clients. This should use osbot-aws's CloudWatch wrapper (if one exists) or be documented as a known exception with a justification.

**Architect recommendation:** Both violations are in the observability pipeline, which was built quickly for the admin dashboard. The fix is straightforward — check if `osbot-aws` has CloudWatch and S3 clients. If yes, swap. If `osbot-aws` lacks a CloudWatch client (likely), either add one to osbot-aws or document this as a justified exception (CloudWatch support may not exist in osbot-aws).

### 4.2 Infrastructure-as-Code Gaps (MEDIUM)

**Finding I-1: No IaC for any AWS resource**

The following are all Console-configured with no CloudFormation, Terraform, or CDK:

| Resource | Impact If Lost |
|---|---|
| CloudFront distributions (×3) | All public access to the product stops |
| CloudFront behaviors (routing rules) | API traffic stops reaching Lambda |
| CloudFront edge function | Website navigation breaks (no index.html rewrite) |
| Kinesis Data Firehose delivery stream | Real-time logs stop flowing |
| CloudFront real-time log configuration | Firehose stops receiving data |
| S3 buckets (×4) | All data and assets gone |
| S3 request metrics filters | Admin dashboard S3 metrics return empty |
| Lambda URL Function configurations | Direct HTTPS endpoints disappear |

**DevOps recommendation:** This is acceptable for the Explorer team phase (Genesis → Custom-Built). But before the Villager team takes this to production, the minimum IaC coverage should be: CloudFront distributions with behavior rules, the Firehose delivery stream, and S3 bucket policies. The Lambda deployment is already code-driven via `Deploy__Service`.

**Finding I-2: S3 request metrics filter assumed but not verified**

`Service__Metrics__Collector` uses `s3_filter_id = 'all-requests'` as a dimension for S3 request metrics. If this filter doesn't exist on the S3 buckets (it must be manually created in the S3 Console under "Metrics" → "Request metrics"), the S3 metrics endpoints return empty data with no error.

### 4.3 Observability Gaps (MEDIUM)

**Finding O-1: CloudFront log field order is hardcoded**

`CF_LOG_FIELDS` in `CloudFront__Logs__Collector` lists 27 fields in a specific order. This must match the field selection in the CloudFront real-time log configuration in the Console. If someone adds or reorders fields in the Console, log parsing silently produces garbage.

**DevOps recommendation:** Add a validation step: the first line of a log file (or a separate manifest) could be compared against `CF_LOG_FIELDS`. Alternatively, document the exact Console configuration and include a screenshot in the repo.

**Finding O-2: No alerting**

The metrics pipeline collects data and evaluates health thresholds, but there's no alerting mechanism. A 5xx spike or a Lambda throttle only becomes visible when someone opens the admin dashboard.

**Finding O-3: CloudFront metrics require us-east-1**

The `CloudWatch__Client` correctly creates a separate us-east-1 client for CloudFront metrics. This is correct (CloudFront is a global service, metrics are only in us-east-1). However, this means the Admin Lambda in eu-west-2 makes cross-region API calls to us-east-1 on every metrics refresh. The caching layer in `Service__Metrics__Cache` mitigates this, but a cold start or forced refresh incurs the latency.

### 4.4 Deployment Concerns (LOW)

**Finding D-1: Runtime upgrade during deployment test**

The deploy test sequence is: create Lambda (Python 3.12) → upgrade runtime to 3.13 → invoke. The package is built for 3.12 but runs on 3.13. This works today because the dependencies have no 3.12-specific native code, but it's fragile. The correct fix is to build the Lambda package targeting 3.13 from the start.

**Finding D-2: Parallel Lambda deployment with no dependency ordering**

The admin and user Lambda deployments run in parallel (`needs: [check-aws-credentials, increment-tag]`). If the user Lambda depends on the admin Lambda being updated first (e.g., new token validation endpoint), a race condition could leave the user Lambda calling an old admin Lambda. In practice, the user Lambda only calls admin for token lookup, which hasn't changed, so this is currently safe.

**Finding D-3: No rollback mechanism**

There's no automated rollback. If a Lambda deployment breaks, the fix is to re-run the pipeline on the previous commit. For static sites, the versioned archive in S3 (`releases/{version}/`) allows manual rollback by re-syncing a previous version to `latest/`, but there's no script or workflow for this.

### 4.5 Type_Safe Compliance (LOW)

**Finding T-1: `health_status: list` in Schema__Metrics__Snapshot**

```python
class Schema__Metrics__Snapshot(Type_Safe):
    health_status : list                     # untyped — should be List[Schema__Health__Status]
```

This is the metrics pipeline's own schema violating the Type_Safe rule against untyped collections. It should be `List__Health__Statuses` with a proper collection subclass.

**Finding T-2: `_s3_client` and `_boto3_client` as untyped instance attributes**

Both `CloudFront__Logs__Collector` and `CloudWatch__Client` store boto3 clients as `self._s3_client` / `self._boto3_client` — set in `setup()`, not declared as class attributes. These are invisible to Type_Safe.
