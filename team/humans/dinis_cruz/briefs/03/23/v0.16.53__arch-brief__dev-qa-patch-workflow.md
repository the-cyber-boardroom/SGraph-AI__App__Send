# Dev-to-QA Workflow: Patches as Bidirectional Communication

**version** v0.16.53
**date** 23 Mar 2026
**from** Human (project lead)
**to** Developer (lead), QA (lead), Architect, AppSec, All teams
**type** Architecture brief

## The Problem We Have Always Had

Development and QA have a communication problem. It has existed for decades and it looks like this:

The developer makes a change. The change breaks a QA test. The QA team gets frustrated. The developer does not understand why the test broke. The test was written by someone else, at a different time, with different context. Documentation is sparse. The change was thrown over the fence.

Over time, this creates a perverse incentive: QA teams write tests that reward not being broken. Tests become brittle on purpose (because brittle tests break, and broken tests mean work). Or worse, test coverage stays shallow (because shallow tests break less often). The system optimises for silence, not for quality.

The result: brittle tests, adversarial relationships, and a QA process that punishes thoroughness. The more tests you write, the more things break when code changes. So you write fewer tests. Which means less coverage. Which means more bugs in production.

## The Insight: Breaking Tests Is Good

But the correct attitude is the opposite: **worry when a code change does not break a test, not when it does.**

A code change that does not break any test is either:

- A refactoring (functionality unchanged, tests should not break, this is fine)
- A gap in test coverage (functionality changed but no test noticed, this is an incident)

A code change that adds a feature or fixes a bug SHOULD break tests. That is the tests doing their job. If a developer refactors and tests do break, that means the refactoring changed behaviour (which it should not have). The signal is in the breakage pattern, not in the breakage itself.

The problem was never that tests broke. The problem was that the developer who broke the tests was not the one responsible for updating them, and the communication between "I changed this" and "you need to update that" was terrible.

## What Changes Now

Three things have converged:

1. **Multiple agents in the same team.** The developer agent and the QA agent are not separate teams throwing work over a fence. They are roles in the same project, reading the same briefs, working toward the same objectives.

2. **Patches as communication.** Both sides communicate through patches, not tickets, not Jira descriptions, not Slack messages. Actual code changes, reviewable diffs, applicable patches.

3. **Screenshots from Playwright.** The ability to take screenshots from within Claude and CI means both sides can show, not just describe, what is happening.

## The Workflow: Developer to QA

When a developer agent makes a code change:

### 1. Developer runs the existing tests

Every feature or component should have instructions (in its README or a companion file) explaining:
- How to run unit tests (hitting the code directly)
- How to run integration tests (component in context)
- How to run end-to-end tests (full workflow via Playwright)

The developer runs all of these before and after the change. If tests break, the developer understands why (because they just made the change).

### 2. Developer updates the tests

This is the key shift. The developer who makes the change is responsible for updating the tests that the change breaks. Not the QA team. The developer.

The workflow:
1. Make the code change
2. Run the tests, observe which break
3. Clone the QA repo (read-only)
4. Update the tests to reflect the new behaviour
5. Verify the updated tests pass
6. Produce a patch for the QA repo

### 3. Developer communicates via patches

The developer's own repo now contains:
- The code change (committed to their branch)
- A patch for the QA repo (updating the tests)
- A patch for the documentation (updating any affected docs)
- The brief that drove the change
- Screenshots showing the before and after

This is not "I made a change, please update the tests." This is "I made a change, here are the test updates I think are needed, here are screenshots of what it looks like." The developer communicates with code, not with descriptions.

### 4. QA reviews the developer's patches

The QA agent (or team) reads the developer's repo:
- Reviews the code change
- Reviews the proposed test patches
- Decides: accept as-is, improve, or rewrite

The QA team has the context the developer does not: the bigger picture of all tests, all workflows, all the places this component is used. The developer wrote a test for the immediate change. The QA team asks: "is this comprehensive enough? Did they test every variation? Where else is this component used?"

The QA team can:
- Accept the patch and apply it
- Modify the patch (add more test cases, improve assertions)
- Reject the patch and write their own tests (with an explanation of why)

The air gap is preserved. The developer proposes. The QA team decides. Neither has write access to the other's repo.

## The Workflow: QA to Developer

It goes both ways. When the QA team finds a bug or needs a capability:

### QA communicates via patches too

The QA agent:
1. Clones the developer's repo (read-only)
2. Writes a suggested patch: "here is the code change I think would fix this" or "here is the utility method I need for testing"
3. Creates a patch and commits it to the QA repo alongside the bug report
4. Includes a screenshot of the failing behaviour (from browser automation)

These are not patches that go into production. They are recommendations. The developer reads them and goes: "I understand what you need. Here is how I will implement it." The QA team communicated with code, not with a ticket description. The developer received something they can read, diff, and reason about.

### Screenshots as communication

With the Playwright screenshot capability, both sides can show what they see:
- QA sends screenshots of the bug ("this is what it looks like when it breaks")
- Developer sends screenshots of the fix ("this is what it looks like after my change")
- Both use the same Browser Automation API, so the screenshots are reproducible

### The Communication Flow

```
Developer repo                          QA repo
  │                                       │
  │ Code change + brief                   │
  │ + patches for QA tests                │
  │ + patches for docs                    │
  │ + screenshots                         │
  │──────────────────────────────────────►│
  │                                       │ QA reviews proposed test patches
  │                                       │ QA accepts, improves, or extends
  │                                       │ QA updates test suite
  │                                       │
  │                                       │ Bug found + suggested code patch
  │                                       │ + screenshot of failure
  │◄──────────────────────────────────────│
  │ Developer reads QA's suggestion       │
  │ Developer implements fix              │
  │ Developer sends updated patches       │
  │──────────────────────────────────────►│
```

Both directions use patches. Both directions use read-only clones. Neither side has write access to the other's repo. The patches are recommendations that the receiving side evaluates and decides whether to accept.

## Why This Works with Agents

In a traditional team, this level of bidirectional communication would be too expensive. Writing test patches for every code change? Writing suggested fixes for every bug report? The overhead would kill velocity.

With agents, the overhead approaches zero. The developer agent already understands the code change (it just made it). Writing the corresponding test patch is incremental work. The QA agent already understands the test suite. Evaluating a proposed test change is fast. The screenshots are generated automatically by the browser automation API.

The agents communicate through artifacts (patches, screenshots, briefs) not through meetings, tickets, or Slack messages. The artifacts are precise, reviewable, and version-controlled.

## Why This Solves the Historical Problems

| Old Problem | How This Solves It |
|---|---|
| QA frustrated when tests break | Tests are expected to break on feature changes. The developer updates them. |
| Developer does not understand why tests broke | The developer runs the tests and sees the breakage directly. |
| Thrown over the fence | Both sides communicate via patches in the same project. |
| QA writes tests that reward not breaking | QA focuses on comprehensiveness and coverage, not on stability. |
| No documentation of what changed | Every change has a brief, patches, and screenshots. |
| Separate teams, separate context | Same agentic team, different roles, shared objectives. |
| Neither side has write access to the other | Read-only clones + patches preserve the air gap structurally. |

## The QA Signs Off Release

This is important and empowering for the developer. The QA team owns the release gate. A feature goes to production when QA signs it off. All tests pass. The developer can stop worrying about "did I break production?" because if QA signs off, it is safe.

The developer can make bold changes, break things, experiment, because they know the QA gate will catch problems before production. The developer stops caring about "will this break production?" and starts focusing on "does this solve the problem?" The QA process is the safety net that allows speed.

## Incidents and Gaps

Two situations that trigger incidents:

1. **A code change that changes behaviour but no test breaks.** This means the test coverage has a gap. Write the missing test. Investigate why the gap existed. Update the guidance for the Explorer/Villager phases.

2. **A test that blocks deployment but the failure is a false positive (or fails without a corresponding code change).** The test infrastructure has a problem. Fix the test infrastructure. This is as important as fixing production code.

Both are incidents, not normal workflow. They are signals that the system needs to improve, and the improvement is documented and fed back into the process. Every gap is visible. Every gap has an owner. The system self-corrects.

## Connection to Other Briefs

- **Browser Automation API:** The QA primitives and workflow commands are what both sides use to write and run tests. Dev uses it to show what changed. QA uses it to verify. Documentation uses it for screenshots.
- **SGit patch workflow:** The same patch-based communication pattern, applied to Dev-QA instead of agent-to-Git. The bidirectional patches between repos are the same patch workflow used for agent-to-human communication via vaults.
- **Explorer-to-Villager methodology:** The Villager phase (where we are now) is where this Dev-QA workflow becomes essential. The Explorer builds fast. The Villager's QA process catches what the Explorer missed.
- **"Who Should Read the Code" article:** The QA agent reads the code at the Villager and Town Planner level. The developer reads at the Explorer level. Different depth, same codebase, complementary perspectives.
