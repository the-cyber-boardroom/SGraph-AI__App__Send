**Version:** v0.2.17

## Executive Summary

In a single session we built the foundation layer of the Browser Automation API described in the v0.16.53 dev brief. The result is a fully self-contained test harness that starts two local servers (API + UI), opens a persistent Chromium browser via CDP, and runs Shadow DOM-aware browser tests against the live v0.3.0 UI — with **0.35s setup** and **0.17s teardown**.

This replaces the previous v030 test infrastructure which used raw Playwright fixtures, subprocess-managed HTTP servers, and polling loops — taking 6-7 seconds to start and producing 17 test failures due to Shadow DOM blindness.

The new architecture is built on three existing OSBot libraries (`osbot-utils`, `osbot-playwright`, `osbot-fast-api`) and introduces four new classes that form Layer 1 of the Browser Automation API.

## What Was Built

### New Classes (4 files)

| Class | Location | Purpose |
|---|---|---|
| `SG_Send__Playwright_Browser__Chrome` | `sg_send_qa/browser/for__osbot_playwright/` | Chromium launcher that skips OSBot-Playwright's install path and uses the Playwright-installed binary directly |
| `QA_Browser` | `sg_send_qa/browser/` | Type_Safe wrapper — persistent browser on fixed CDP port, lazy init, survives across test runs |
| `SG_Send__Browser__Pages` | `sg_send_qa/browser/` | Layer 1 API — page navigation, access gate, upload wizard (Shadow DOM aware), download, state queries, workflow commands |
| `test_SG_Send__Browser__Pages` (+ 5 variants) | `tests/integration/browser/` | Integration tests split by concern: navigation, state, access gate, upload, download, workflow |

### Key Design Decisions

**1. CDP (Chrome DevTools Protocol) over Playwright's built-in launch**

The previous approach used `sync_playwright().chromium.launch()` which creates and destroys a browser per test session. The new approach uses `Playwright_Browser__Chrome` which launches Chrome as an independent subprocess on a fixed debug port (10070) and connects via CDP. The browser process persists across test runs — reconnection is instant.

**2. Temp_Web_Server over subprocess.Popen**

The previous UI server was started via `subprocess.Popen([sys.executable, "-m", "http.server", ...])` with a polling loop (`while time.time() < deadline: try: httpx.get(...)`). The new approach uses `Temp_Web_Server` from osbot-utils — auto-assigned port, context manager lifecycle, zero polling.

**3. Shadow DOM-aware selectors from the start**

The previous tests used `page.locator("#file-input")` which cannot pierce Shadow DOM. The new `SG_Send__Browser__Pages` class uses Playwright's shadow-piercing selectors: `page.locator("upload-step-select #file-input")` — component tag name followed by element ID. Every method in the upload wizard section documents which DOM boundary it crosses.

**4. `inner_text()` over `text_content()`**

The previous tests used `page.text_content("body")` which includes text from `<script>` tags, causing false "error" detection from inline JavaScript. The new `visible_text()` method uses `page.inner_text("body")` which returns only visible rendered text.

## Shadow DOM Fix

The core reason the v030 tests had 9 failures was the v0.3.0 upload refactoring which decomposed `send-upload` into Shadow DOM sub-components. Here is the component tree and how selectors changed:

```
v0.2.x (monolith — everything in light DOM):
┌─────────────────────────────────────────┐
│ <send-upload>                           │
│   <input id="file-input">  ◀── VISIBLE │  page.locator("#file-input") ✓
│   <button id="upload-next-btn">         │  page.locator("#upload-next-btn") ✓
│   <div data-mode="combined">            │  page.locator('[data-mode="combined"]') ✓
│   <input readonly>  (download link)     │  page.locator("input[readonly]") ✓
└─────────────────────────────────────────┘

v0.3.0 (decomposed — sub-components use Shadow DOM):
┌─────────────────────────────────────────────────────────────┐
│ <send-upload>  (light DOM orchestrator)                      │
│   ┌───────────────────────────────────────────────────────┐ │
│   │ <upload-step-select>  (Shadow DOM)                    │ │
│   │   #shadow-root                                        │ │
│   │     <input id="file-input">  ◀── HIDDEN from light   │ │
│   │     <div id="drop-zone">                              │ │
│   └───────────────────────────────────────────────────────┘ │
│   <button id="upload-next-btn">  ◀── STILL in light DOM    │
│   ┌───────────────────────────────────────────────────────┐ │
│   │ <upload-step-share>  (Shadow DOM)                     │ │
│   │   #shadow-root                                        │ │
│   │     <div data-mode="combined">  ◀── HIDDEN            │ │
│   └───────────────────────────────────────────────────────┘ │
│   ┌───────────────────────────────────────────────────────┐ │
│   │ <upload-step-done>  (Shadow DOM)                      │ │
│   │   #shadow-root                                        │ │
│   │     <div id="combined-link">  ◀── HIDDEN              │ │
│   │     <div id="simple-token">   ◀── HIDDEN              │ │
│   └───────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘

Selector migration:
  BEFORE (broken):                    AFTER (works):
  page.locator("#file-input")    →    page.locator("upload-step-select #file-input")
  page.locator('[data-mode="…"]')→    page.locator('upload-step-share [data-mode="…"]')
  page.locator("input[readonly]")→    page.locator("upload-step-done #combined-link")
  page.text_content("body")     →    page.inner_text("body")
```

Playwright auto-pierces open Shadow DOM when selectors include the host element tag name. The `SG_Send__Browser__Pages` class encodes this knowledge in every method — test authors never need to think about Shadow DOM boundaries.

## The Core Insight: The Browser as a Stateful Debugging Workbench

The persistent browser is not primarily a performance optimisation. **It is a fundamentally different way to develop and debug browser tests.**

### The Problem With Disposable Browsers

Traditional Playwright testing treats the browser as a disposable fixture:

```
┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│  Launch   │────▶│  Run     │────▶│  Assert  │────▶│ Destroy  │
│  browser  │     │  test    │     │  (fail)  │     │ browser  │
│  (~2s)    │     │  (~5s)   │     │          │     │  (~1s)   │
└──────────┘     └──────────┘     └──────────┘     └──────────┘
                                       │
                                       ▼
                              Fix code, re-run
                              from the beginning
                              (another 8 seconds)
```

When a test fails at step 5 of a 6-step wizard, you:
1. Read the error message
2. Guess what went wrong
3. Change the test code
4. Re-run — wait 8 seconds for browser launch + replay steps 1-4
5. Hit a different failure at step 5
6. Repeat

There is no way to inspect the live page. No way to try selectors. No way to check Shadow DOM. The browser is gone.

### The Persistent Browser Changes Everything

With CDP persistence, the browser is a **live workbench** that holds application state between test runs:

```
┌──────────────────────────────────────────────────────────────────┐
│                    Chromium (CDP port 10070)                      │
│                                                                  │
│  State preserved between runs:                                   │
│  ├─ Current page URL and DOM                                     │
│  ├─ localStorage (access tokens, preferences)                    │
│  ├─ Shadow DOM component trees (fully rendered)                  │
│  ├─ JavaScript console (errors, warnings)                        │
│  └─ Network state (cookies, cache)                               │
│                                                                  │
│  DevTools attachable at any time:                                │
│  ├─ Elements panel → inspect Shadow DOM live                     │
│  ├─ Console → test selectors interactively                       │
│  ├─ Network → see API calls the UI is making                     │
│  └─ Application → inspect localStorage directly                  │
└───────────────────────────────────┬──────────────────────────────┘
                                    │ CDP (always connected)
                                    │
┌───────────────────────────────────┴──────────────────────────────┐
│                         IDE (PyCharm)                             │
│                                                                  │
│  Run test_upload_step_4  ──▶  35ms  ──▶  Browser at step 4      │
│  │                                       │                       │
│  │  Inspect browser:                     │  Page shows confirm   │
│  │  - What does Shadow DOM look like?    │  step with file info  │
│  │  - What selector hits the button?     │  and "Encrypt" button │
│  │  - What's in localStorage?            │                       │
│  │  - Is the API endpoint correct?       │                       │
│  │                                       │                       │
│  ▼  Fix selector, re-run same test       │                       │
│  Run test_upload_step_4  ──▶  35ms  ──▶  Browser updates        │
│                                          (no replay of steps 1-3)│
└──────────────────────────────────────────────────────────────────┘
```

The development loop becomes:

1. Run a test that builds up state (navigate, enter token, upload file, reach step 4)
2. Test completes in **35ms** — browser is still open at step 4
3. **Open DevTools on the live page** — inspect Shadow DOM, try CSS selectors in the console, check localStorage, watch network requests
4. Find the right selector or identify the bug
5. Update the test code
6. Re-run **just that one test** — 35ms, browser picks up from the same state
7. Iterate until green

No 8-second replay. No blind guessing. No lost state.

### Why This Matters for SG/Send Specifically

SG/Send's v0.3.0 UI is built from **Shadow DOM Web Components with async resource loading**. Each component (`upload-step-select`, `upload-step-share`, `upload-step-done`) has its own shadow root, fetches its own CSS and HTML templates over HTTP, and renders asynchronously. Getting a test to work requires:

- Knowing which component hosts which element
- Knowing when async loading completes
- Knowing the exact selector that pierces the right shadow boundary

With a disposable browser, you cannot see any of this. With a persistent browser, you open Elements panel, expand the shadow roots, and **see exactly where every element lives**. The 9 Shadow DOM failures in the v030 test suite would have been trivially debuggable with this setup — you would see that `#file-input` lives inside `upload-step-select`'s shadow root within seconds.

### The Compound Effect: State-Heavy Test Scenarios

Some test scenarios require significant setup before the interesting assertion:

```
Upload workflow test (step 6 — extract download link):
  Step 1: Navigate to /en-gb/              (~80ms)
  Step 2: Enter access token               (~50ms)
  Step 3: Select file                      (~100ms)
  Step 4: Choose delivery mode             (~50ms)
  Step 5: Select share mode                (~50ms)
  Step 6: Click "Encrypt & Upload"         (~2000ms — crypto + network)
  ─────────────────────────────────────────────────
  ASSERT: download link appears in done step
```

With a disposable browser, every re-run of the assertion costs **~2.4 seconds** of state replay. With persistent state, you reach step 6 once, and then iterate on the assertion at **35ms per attempt** — the encrypted transfer is already complete, the done step is rendered, the download link is in the Shadow DOM. You just need to find the right selector.

This is the real reason for persistent CDP browsers. Performance is the side effect. **Interactive, stateful debugging of complex UI workflows** is the feature.

## Different Tools for Different Phases

This session is also a clear example of how the Explorer / Villager / Town Planner methodology applies to testing infrastructure itself — not just to product features.

The persistent browser is an **Explorer tool**. It solves an Explorer problem. It should not become the default everywhere.

### The Pattern

```
                    Wardley Evolution Axis
                    ─────────────────────────────────────────▶

    Genesis          Custom-Built         Product            Commodity
    (Explorer)       (Villager)           (Town Planner)

    ┌──────────┐     ┌──────────┐        ┌──────────┐
    │PERSISTENT│     │ EITHER   │        │DISPOSABLE│
    │ BROWSER  │     │          │        │ BROWSER  │
    │          │     │ Persist  │        │          │
    │ headless │     │ for debug│        │ headless │
    │ = False  │     │ headless │        │ = True   │
    │          │     │ in CI    │        │          │
    │ CDP port │     │          │        │ No CDP   │
    │ fixed    │     │          │        │          │
    │          │     │          │        │ Isolated │
    │ State    │     │          │        │ per test │
    │ survives │     │          │        │          │
    │ between  │     │          │        │ Zero     │
    │ runs     │     │          │        │ state    │
    │          │     │          │        │ leakage  │
    │ DevTools │     │          │        │          │
    │ attached │     │          │        │ Maximum  │
    └──────────┘     └──────────┘        │ parallel │
                                         └──────────┘
```

### Explorer Phase: Persistent Browser is Essential

The Explorer is discovering what works. The UI is changing rapidly. Shadow DOM boundaries are being drawn. New components are being created. The question is not "does this test pass?" — it is "what does this page even look like now, and how do I reach that element?"

In this phase:
- **Visible browser** — you need to see what the UI renders
- **Persistent state** — you need to build up to step 5 once, then iterate on the step 5 assertion 20 times
- **DevTools attached** — you need to inspect Shadow DOM, try selectors, watch network calls
- **Fast feedback loop** — 35ms re-runs, not 8-second replays
- **Shared browser across tests** — intentional, because you are exploring state transitions

The persistent browser is not a shortcut. It is the right tool. Without it, the 9 Shadow DOM failures would have taken hours of blind iteration to fix. With it, you open Elements panel, see that `#file-input` lives inside `upload-step-select`'s shadow root, write the piercing selector, and move on.

### Villager Phase: Transitional

The Villager stabilises what the Explorer built. Tests are known to work. The selectors are proven. The question shifts from "what does good look like?" to "does it still look good after this change?"

In this phase:
- **Headless in CI** — the Villager runs tests in the pipeline, no human watching
- **Persistent locally** — developers debugging a failing CI test still benefit from the persistent browser
- **Screenshot diffing** — visual regression detection replaces interactive inspection
- **Known selectors** — the Shadow DOM piercing selectors are already written and proven

The persistent browser is useful but not essential. The Villager might keep it for local debugging but run headless disposable browsers in CI.

### Town Planner Phase: Disposable Browsers Only

The Town Planner transmutes testing into a commodity. Tests are tight, optimised, focused, and efficient. They run in parallel across multiple workers. They are the gate for production deployment.

In this phase:
- **Headless always** — no human in the loop
- **Disposable per test** — zero state leakage between tests, maximum isolation
- **No fixed CDP port** — random ports, parallel workers, no conflicts
- **No persistent state** — every test starts clean, every test is independent
- **Maximum parallelism** — 10 browsers, 10 workers, 10x throughput
- **Prometheus metrics** — test duration and pass rate are operational data, not debugging aids

A persistent browser would be **harmful** here. Shared state between tests means test A's localStorage leaks into test B. Fixed ports mean no parallelism. A visible browser means wasted resources on a headless CI runner.

### The Key Lesson

The three phases do not use the same tools in different amounts. They use **fundamentally different tools** because they solve fundamentally different problems:

| Phase | Problem | Tool | Browser Mode |
|---|---|---|---|
| **Explorer** | "What does this look like? How do I reach that element?" | Persistent visible browser + DevTools | `headless=False`, fixed CDP port, state preserved |
| **Villager** | "Does this still work after the change?" | Headless CI + persistent local debug | `headless=True` in CI, `headless=False` for debugging |
| **Town Planner** | "Can we deploy to production?" | Disposable headless, parallel, isolated | `headless=True`, random port, zero state, 10x parallel |

The Browser Automation API supports all three modes through `QA_Browser(headless=True/False, port=FIXED/random)`. The same `SG_Send__Browser__Pages` methods work in all modes — only the browser lifecycle changes. This is by design: the Layer 1 primitives and Layer 2 workflows are phase-agnostic. The browser configuration is phase-specific.

## Secondary Insight: Performance

The persistent browser also cascades into a 20x setup improvement (0.35s vs 6-7s), 37x faster per-page navigation (80ms vs 3s), and 6x faster teardown (0.17s vs 1s). Combined with `Temp_Web_Server` (eliminating polling loops) and `Fast_API_Server` (eliminating subprocess overhead), browser tests feel like unit tests. But this is a consequence of the architectural decision, not the reason for it.
