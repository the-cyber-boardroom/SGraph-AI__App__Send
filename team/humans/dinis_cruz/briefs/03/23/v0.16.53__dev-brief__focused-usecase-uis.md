# Simple, Focused Use-Case UIs: One Click, Not Ten Options

**version** v0.16.53
**date** 23 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Designer, Architect, Ambassador
**type** Dev brief

---

## The Emerging Picture

A pattern is clarifying. SG/Send competes on usability through a large number of small, well-executed features that compound. The v0.3.0 IFD refactoring and the new manifest architecture make it possible to push this further.

The contradiction: the main send workflow needs to keep gaining features and capabilities (more options, more delivery modes, more customisation). But every option added is complexity the user sees. The power user wants all the options. The first-time user wants one click.

## The Solution: Two Layers

### Layer 1: The Full Workflow (Evolving)

The main send UI (the six-step flow: Upload, Delivery, Share mode, Confirm, Encrypt and Upload, Done) continues to evolve. It gains more options, more delivery modes, more power. This is the "super duper" version for users who want full control.

### Layer 2: Focused Use-Case UIs (Pre-Baked)

Separate, simplified UIs where most options are pre-baked. The user does not choose a delivery mode or a share method. Those decisions are already made for the use case.

Examples:

**Share a Video**
1. User goes to "Share a Video" page
2. Drops the video file
3. Maybe sees a preview (plays it to confirm it is the right file)
4. Maybe adds a short message for the recipient
5. Clicks Send
6. Gets a link

Two actions: drop and send. Everything else (encryption settings, delivery mode, share format, expiry) is pre-configured for the "share a video" use case.

**Share Photos (Gallery)**
1. User drops images
2. Sees the gallery preview
3. Clicks Send
4. Gets a gallery link

**Send a Document Securely**
1. User drops a PDF or document
2. Clicks Send
3. Gets a link with separate decryption key (higher security default for documents)

**Quick File Transfer**
1. User drops any file
2. Link is generated immediately (auto-send, no confirmation step)

Each of these is a one-click or two-click workflow. The underlying infrastructure is identical to the full send flow. The difference is which options are exposed and which are pre-decided.

## Why the IFD v2 Manifest Makes This Easy

With the manifest-driven architecture, creating a new use-case UI is:

1. Write a manifest file that specifies which components to load and with which configuration
2. Possibly write a small workflow configuration (which steps to show, which to skip, what defaults to apply)
3. Deploy

No new components needed. No new JavaScript. The same upload component, the same encryption engine, the same delivery options, just wired differently. The manifest decides what the user sees.

### The Workflow Configuration File

A JSON file (or a section of the manifest) that defines the state machine for a specific use case:

```json
{
  "use_case": "share-video",
  "steps": ["upload", "preview", "done"],
  "skip_steps": ["delivery", "share_mode", "confirm"],
  "defaults": {
    "delivery_mode": "link_with_key",
    "expiry_days": 7,
    "max_downloads": null,
    "show_preview": true,
    "allow_message": true
  }
}
```

The upload component reads this configuration and adjusts its behaviour: skip the delivery step, skip the share mode step, use the defaults, go straight from upload to preview to done. Same component. Different experience.

This means creating a new use-case UI is not a development task. It is a configuration task. The Ambassador or the Sherpa can define a new use case by writing a JSON file. The Developer does not need to touch any code.

## Eager Upload (Future Enhancement)

An interesting optimisation captured for later: start uploading the file as soon as the user drops it, before they click Send. The encryption and upload happen in the background while the user adds a message or confirms. If they cancel, delete from the server. If they confirm, the upload is already done (or nearly done).

For large files, this could save 10-20 seconds of perceived wait time. The user thinks they are adding a message. The file is already uploading.

Not for the initial implementation. But the architecture should not prevent it. The upload engine already runs independently of the UI steps. Triggering it earlier is a timing change, not a structural change.

## How This Connects to Everything Else

**Sherpa demos.** Each focused UI is also a demo page. "Share a Video" is both a product page and a live walkthrough. The user experiences the product by using it.

**Marketing workflows.** The "Why Would You Use It?" pages from the launch brief map directly to focused UIs. Each page shows the pain, shows the solution, and lets you try it with one click.

**Industry verticals.** The musician vertical, the Cyber Boardroom, the investor pack: each is a focused UI with a specific manifest and workflow configuration. The theming layer (`<sg-theme>`) changes the look. The manifest changes the behaviour.

**Component pipeline.** Every focused UI is a composition of existing components via manifest. No new code per use case. The component pipeline produces the parts. The manifest assembles them.

## The Competitive Advantage

Most competitors have one UI for everything. You upload a file the same way whether it is a video, a photo gallery, a legal document, or a song demo. The UI does not adapt.

With manifest-driven focused UIs, SG/Send can have dozens of tailored experiences, each optimised for a specific workflow, each deployable as a configuration change. The development cost per new use case approaches zero. The perceived value per use case is high because the experience feels purpose-built.

This is the compound effect of good architecture: the v0.3.0 refactoring made the components modular, the IFD v2 manifest made them configurable, and the focused UIs make them accessible.
