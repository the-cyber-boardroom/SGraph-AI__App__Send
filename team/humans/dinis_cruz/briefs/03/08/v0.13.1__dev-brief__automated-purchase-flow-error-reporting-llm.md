## Milestone: First Fully Automated Purchase-to-Transfer

A user went through the complete flow with zero human intervention:

```
1. Visited send.sgraph.ai
2. Purchased credits via Stripe
3. Redirected to the welcome page
4. Temporary token auto-applied to their browser  ← First time this worked end-to-end
5. Uploaded a file successfully
6. Transfer completed
```

**This is the full loop.** No manual token provisioning. No WhatsApp messages. No waiting. The user committed their time, was in the zone, and could immediately start using the product. This validates the temp token architecture from today's earlier brief.

**Why this matters for usage:** The human's hypothesis is that the 10-20 operations pattern (customers try a few things then stop) is partly because of the previous manual provisioning gap. By the time the admin sent the token (even within 15 minutes), the user had context-switched to something else. The slotted moment — the moment they were motivated to try — was lost. Auto-provisioning preserves that moment.

## Bug: JavaScript Error on Second Transfer

### What Happened

The user's first transfer worked. The second (or third) transfer failed with a JavaScript error. The user could not send more files after the first success.

### Root Cause (Found and Fixed)

The user found the base64 string from the Stripe checkout flow (which is the **decryption key** for the temp token SG/Send transfer) and base64-decoded it, then tried to use the decoded value as an **access token**.

This is a credential confusion problem:

| What the User Saw | What It Actually Is | What They Thought It Was |
|---|---|---|
| Base64 string from Stripe redirect | Decryption key for the SG/Send transfer containing the temp token | An access token to paste into the token field |

The user was technically savvy (they knew to base64-decode a string), but the UX didn't make it clear that the auto-provisioning had already handled everything — no manual token entry was needed.

### Fix Applied

The immediate bug is fixed. But the underlying UX problem remains: the post-purchase page shows or exposes credential-like strings that users can find and misuse.

### UX Actions

| # | Action | Priority |
|---|---|---|
| 1 | Post-purchase welcome page should clearly state: "Your token has been set up automatically. You're ready to go — no need to enter anything." | P0 |
| 2 | Hide or minimise any visible base64/credential strings on the welcome page | P0 |
| 3 | If a user enters an invalid token (wrong format, decoded decryption key, etc.), show a helpful error: "This doesn't look like an access token. Your token was set up automatically when you purchased — you should be ready to go." | P1 |
| 4 | Differentiate visually between "token active" and "no token" states so users can see they're already authenticated | P1 |

## Error Reporting Infrastructure

### The Problem

When the user hit the JavaScript error, nothing was reported back to us. The human only found out because the user mentioned it. There's no error callback, no telemetry, no way to know that a user experienced a failure.

### What We Need

#### Anonymous Error Reporting

When a JavaScript error occurs in the browser, automatically report it:

```json
{
  "type": "error",
  "timestamp": "2026-03-08T14:32:00Z",
  "error": "TypeError: Cannot read property 'x' of undefined",
  "stack": "at handleUpload (send.js:142)...",
  "page": "/en-gb",
  "action": "second_transfer",
  "browser": "Chrome 124",
  "anonymous_id": "sha256_of_session_id"
}
```

**What we DON'T send:** No personal data. No file names. No token values. No IP addresses. Just the error, the stack trace, the page, and the action that triggered it.

**Implementation options:**

| Option | How It Works | Effort |
|---|---|---|
| **CSP report-uri** | Content Security Policy has a built-in error reporting mechanism. Add `report-uri` to CSP headers. Browser automatically sends violation reports. | Low — server-side config change |
| **window.onerror + fetch** | Global error handler catches all uncaught errors, sends a POST to our error endpoint | Low — a few lines of JS |
| **Custom error service** | A dedicated error reporting endpoint (Lambda) that receives, stores, and aggregates errors | Medium — new Lambda + storage |
| **Third-party (Sentry, etc.)** | Use an existing error reporting service | Low effort, but adds a third-party dependency and potential privacy concern |

**Recommendation:** Start with `window.onerror` + a simple Lambda endpoint. Store errors in S3 as JSON files. Review manually for now. Add aggregation later.

#### User-Initiated Bug Reports

In addition to automatic error reporting, give users a way to submit a report:

- **"Report a problem" button** on the error page (or always visible in footer)
- Opens a form: "What happened?" (free text) + "Include diagnostic data?" (checkbox — opts in to sending browser info, localStorage state, etc.)
- Optionally: include contact info (email) so we can follow up
- The report is sent via SG/Send itself (encrypted, of course) — dogfooding

#### Anonymous Follow-Up (Future)

Once we have messaging, a powerful concept: send a message back to the anonymous error reporter.

The error report includes an anonymous session ID (hash). We can create an SG/Send one-time message addressed to that session ID. If the user returns to the site with the same session, they see: "We fixed the bug you reported. Thanks!" — without ever knowing who they are.

This is the zero-knowledge support model: fix the bug, notify the reporter, never learn their identity.

## One-Shot LLM Execution on the Desktop

### The Concept

The desktop becomes the environment for constructing and executing one-shot LLM prompts — not by running models locally, but by building the perfect prompt and sending it to an API.

**Why not run models locally?** Local model execution (Claude Desktop, Cursor, etc.) is hard to contain. Controls are soft. The model can access the filesystem, make network calls, and potentially do things it shouldn't. Prompt injection is "on steroids" when the model runs with local system access.

**The one-shot model is safer:** The LLM only sees what we explicitly include in the prompt. We control the entire reality: the system prompt, the context files, the history (which we construct, not accumulate), the data, and the question. The LLM has no access to anything outside the prompt. It's a sandbox by construction.

### The Desktop Advantage

The desktop adds capabilities the web workspace doesn't have:

| Capability | Web Workspace | Desktop |
|---|---|---|
| Construct prompts from vault files | ✅ | ✅ |
| Attach screenshots | ✅ (paste) | ✅ (paste + screenshot tool) |
| Access local files as context | ❌ | ✅ (filesystem access) |
| Store execution bundles in vault | ✅ | ✅ |
| Run against multiple models in parallel | ✅ | ✅ + background execution |
| Use vault secrets as API keys | Manual | Automatic (vault integration) |
| Access data from other WebViews | ❌ | ✅ (JS injection extracts data from open sites) |

The last point is powerful: the desktop can extract data from an open GitHub page or AWS console (via JS injection) and include it as context in an LLM prompt. "Summarise the last 10 commits from the GitHub page I have open" — the desktop reads the page, extracts the data, constructs the prompt, sends it to the API.

### Prompt Construction as Engineering

The bundle model from the workspace brief (v0.11.10) applies directly:

```
Bundle = {
  system_prompt: "...",
  context_files: [ vault://role-definition.md, vault://project-status.md ],
  constructed_history: [
    { role: "user", content: "Here's the code structure..." },
    { role: "assistant", content: "I see a tree/blob model..." }
  ],
  data: { ... extracted from WebViews ... },
  question: "What's the most efficient way to..."
}
```

Each element is deliberately chosen. Nothing accumulates automatically. The constructed history is fake (hand-crafted user/assistant pairs that prime the LLM's understanding) — the multi-prompt composition technique from the v0.11.10 token optimisation brief.

### The Wardley Map of Prompt Engineering

The evolution from Genesis to Commodity, applied to a specific LLM task:

| Stage | What the Prompt Looks Like | Token Cost |
|---|---|---|
| **Genesis** | Massive context, full file contents, long conversation history, vague question | 50K+ tokens |
| **Custom Built** | Curated context, skeleton files, one-shot, specific question | 10K tokens |
| **Product** | Compressed context, meta-language, structured schema in/out, pre-computed summaries | 2-3K tokens |
| **Commodity** | No LLM needed — the task is now a script/function that runs deterministically | 0 tokens |

The desktop workspace is the environment for doing this evolution: experiment at Genesis, refine to Custom, compress to Product, codify to Commodity. Each stage is saved as a bundle in the vault.

### Saving Results to Vault

Every LLM execution on the desktop saves its bundle (prompt + response + metadata) to the user's personal vault. This creates:

- A history of what was asked, what context was provided, what the response was
- A record of files sent and received via SG/Send (the user can see "I sent 15 files this week")
- Token cost tracking per execution (the Accountant's data)
- Replayability: load a previous bundle, modify one element, re-execute
