## Context: Why the Desktop Changes Everything

On the web, the capabilities are scattered: send.sgraph.ai for transfers, vault.sgraph.ai for storage, workspace.sgraph.ai for transformations, tools.sgraph.ai for utilities. Each has its own tab, its own context, its own localStorage. The user hops between them.

The desktop app brings these together into one window, one context, one identity. And with two additional milestones from yesterday — the CLI now uses a git-like tree/blob structure, and the in-memory storage server runs locally — the desktop becomes the place where the full vision crystallises.

**The usage signal is clear:** customers who bought tokens are mostly doing 10-20 operations — drop one file, look around, think "this is cool," and stop. They don't have a daily reason to come back. The desktop app, with personal vaults and persistent state, creates that daily recurrence: "open SGraph Desktop, check my vault, send a file, manage my teams."

---

## Milestone: CLI with Git-Like Structure

The CLI now implements the tree/blob architecture from the v0.11.1 architecture brief. Files are stored as content-addressed blobs, folder structure as tree objects, and changes as commit-like operations. This means:

- `sg-send pull` fetches only changed blobs (delta sync)
- `sg-send push` uploads only new/modified files
- Version history is implicit in the tree structure
- Conflicts are detectable (two trees modified the same file)

This was designed on 4 March and is now running. The git-like structure underpins everything below — personal vaults, team vaults, and local storage all use this model.

## Milestone: Local In-Memory Storage Running

The in-memory version of the storage service runs locally. This proves the architecture can work offline: the desktop app could run the full storage stack on the user's machine, no server needed. Not shipping this as a feature yet — keeping it in the back pocket until we add visualisation for the local store and cache. But it validates the decentralised architecture.

## Part 1: Personal Vaults — user.sgraph.ai

### The Concept

Every user gets a personal vault. This is their private, encrypted space for: configuration, tokens, saved links, team memberships, and personal files. It's the user's home.

**URL:** `user.sgraph.ai` (or `user.dev.sgraph.ai` in development)

**Access:** Via a master password that the user creates. This is NOT the access token — it's a separate credential that the user controls entirely.

### Why Not Use the Access Token as the Vault Key?

The access token is something the human (admin) creates and gives to the user. This means the admin knows the token and could derive the vault key — breaking zero-knowledge. The whole point of a personal vault is that only the user can access it.

**Solution: Master password.** The user creates their own master password during personalisation setup. This password derives the vault encryption key (via PBKDF2, same as existing vaults). The admin never sees the master password. If the user loses it, they lose access to their personal vault — same model as any password manager.

### What Lives in the Personal Vault

```
user-vault/
  profile/
    settings.json          <- Language, theme, notification preferences
    personalisation.json   <- Display name, avatar, preferences
  tokens/
    token-001.json         <- Access token (encrypted), credits, status
    token-002.json         <- Another token
  teams/
    team-alpha/
      team-config.json     <- Team name, team vault access code, role
    team-beta/
      team-config.json
  saved-links/
    shared-files.json      <- Links to files the user has shared
    received-files.json    <- Links to files received
  keys/
    keypair-001.json       <- User's PKI key pair (encrypted with master password)
```

### The Onboarding Flow

```
1. User purchases credits → receives access token
2. User can immediately use SG/Send with just the token (anonymous mode)
3. Optional: User creates a master password → personal vault created
4. Token is stored in the personal vault (encrypted)
5. Future logins: master password → vault opens → tokens, settings, teams loaded
6. Desktop app: master password saved in OS keychain or password manager
```

**Anonymous mode remains.** Users who don't want personalisation just use their token directly. No master password, no vault, no persistence. The personal vault is opt-in, not required.

### Password Manager Integration

The master password should be easy to save in existing password managers:

- **Browser autofill:** Structure the login form with standard `autocomplete` attributes so Chrome, Firefox, Safari offer to save the credentials
- **1Password / LastPass / Bitwarden:** The login page should trigger password manager detection
- **Google Password Manager:** Same — standard form attributes
- **Desktop app (Tauri):** Can integrate with the OS keychain (macOS Keychain, Windows Credential Manager) to store the master password securely

**Research needed:** What APIs do password managers expose? Can we trigger "save this credential" programmatically? Can we deep-link to "save in 1Password"?

## Part 2: Team Vaults

### The Concept

A team vault is a shared vault where multiple users collaborate. The team creator sets it up, invites members, and everyone accesses the same encrypted space.

**The key insight:** There's no architectural difference between a personal vault and a team vault. Both are vaults. Both use the same encryption model. Both are stored on the same infrastructure. The difference is social: a personal vault has one key holder; a team vault has multiple.

### How Teams Work

```
1. User A creates a team → generates a team vault with a shared access code
2. User A saves the team config in their personal vault:
   { "team_name": "Acme Corp", "team_vault_id": "xyz", "team_access_code": "..." }
3. User A invites User B → sends them the team access code (via SG/Send, of course)
4. User B enters the team access code → team vault opens
5. User B saves the team config in their OWN personal vault
6. Both users can now access the team vault independently
```

**User A's personal vault:**
```
user-vault/
  teams/
    acme-corp/
      team-config.json    <- { vault_id, access_code, role: "admin" }
    side-project/
      team-config.json    <- { vault_id, access_code, role: "member" }
```

### Multi-Team Membership

A user can belong to multiple teams. Their personal vault stores the access details for each. The desktop app shows all teams in the sidebar. Switching teams = opening a different vault.

### Team Tokens and Billing

Teams could have their own access tokens and credit pools — separate from individual users. This enables: "The company buys 50GB of credits. Team members use them from a shared pool." This connects to the billing models brief (v0.11.08) — specifically the "third-party pays" and "split billing" models.

## Part 3: Desktop as Multi-WebView Workspace

### The Problem

The human has a daily workflow that involves multiple web tabs: several GitHub repos, Claude Code sessions, SG/Send, vault, workspace, admin panel. Switching between them is painful. Information is scattered across tabs.

### The Tauri Multi-WebView Opportunity

Tauri can create multiple WebView instances within the same app window. This means:

```
┌─────────────────────────────────────────────────┐
│ SGraph Desktop                                  │
├──────────┬──────────┬──────────┬───────────────┤
│ GitHub   │ GitHub   │ SG/Send  │ Vault         │
│ Repo 1   │ Repo 2   │          │               │
│          │          │          │               │
├──────────┴──────────┼──────────┤               │
│ Claude Code         │ Workspace│               │
│                     │          │               │
└─────────────────────┴──────────┴───────────────┘
```

Four, five, six WebViews in a grid — each showing a different site, all visible simultaneously. This is the IDE-like experience: everything you need, visible at once, in one app.

**Key question to explore:** Can Tauri embed multiple WebViews? The `X-Frame-Options` and CSP headers on sites like GitHub may prevent iframe embedding. Tauri's WebView might bypass this (since it's not an iframe — it's a native WebView). This needs testing.

**If multi-WebView works:** The desktop app becomes a tiling window manager for web applications — specifically tuned for the SG/Send ecosystem but capable of embedding any web page. This is the sg-layout vision applied at the OS level.

**If multi-WebView doesn't work (frame restrictions):** Fall back to tabbed navigation (which already works) with a quick-switcher. Still valuable, just not as revolutionary.

### Link Management

The desktop app becomes the place to manage frequently used links:

- GitHub repos (pinned, quick-open)
- Claude Code sessions
- SG/Send admin panels
- Vault access shortcuts
- Team-specific bookmarks

These are stored in the personal vault — so they persist across devices and survive app reinstalls.

## Part 4: Components First — Everything is a Tool

### The Principle

Every UI concept in this brief should be built as a Web Component in the tools repo:

| Component | What It Does | Where It's Used |
|---|---|---|
| `<sg-token-manager>` | View, add, remove, switch access tokens | Desktop, user.sgraph.ai, send.sgraph.ai |
| `<sg-user-profile>` | Personalisation: name, avatar, language, theme | Desktop, user.sgraph.ai |
| `<sg-team-manager>` | Create teams, invite members, switch teams | Desktop, user.sgraph.ai |
| `<sg-vault-browser>` | Browse vault contents (personal or team) | Desktop, vault.sgraph.ai, workspace |
| `<sg-secrets-manager>` | Store and retrieve encrypted secrets | Desktop, Chrome extension |
| `<sg-link-manager>` | Save, organise, quick-open frequently used links | Desktop |
| `<sg-master-password>` | Master password entry, vault key derivation | Desktop, user.sgraph.ai |

### Genesis in the Consumer Repo, Mature in Tools

Components start in whatever repo needs them (user.sgraph.ai, desktop app). Once they reach a level of maturity and are needed by multiple consumers, they migrate to the tools repo as canonical components. The individual repos get smaller over time as they increasingly compose from tools.

This is the Wardley Map evolution applied to components: Genesis (built in the consumer repo) → Custom (works but not reusable) → Product (migrated to tools, used by multiple consumers) → Commodity (stable, rarely changes, deeply relied upon).
