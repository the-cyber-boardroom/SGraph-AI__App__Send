# Bug Debrief — Corrupted Token / ISO-8859-1 Header Error

**Version:** v0.12.6  
**Date:** 2026-03-08  
**Severity:** High (blocks all subsequent uploads for affected user)  
**Status:** Confirmed, fix required  

---

## Summary

A user on a second browser attempted to enter their access token manually after not having it written down. Unable to locate the token, they decoded the AES-256-GCM key from their download URL and submitted it as the token. The token input accepted the value without complaint. The decoded binary string contained a non-ISO-8859-1 unicode character (`ɸ`, U+0278), which caused the browser's fetch API to hard-reject every subsequent request with:

```
Failed to execute 'fetch' on 'Window': Failed to read the 'headers' property
from 'RequestInit': String contains non ISO-8859-1 code point.
```

The product entered a silently broken state — no recovery prompt, no explanation, every upload attempt failed. The user had no path forward without external help.

---

## Root Cause Chain

### Step 1 — User switched browsers

The access token is stored in `localStorage` under key `sgraph-send-token` (see `api-client.js`). localStorage is scoped per browser per origin. The user sent one file successfully in Browser A, then opened Browser B to send another file. Browser B had no token. The access gate showed the token entry form.

### Step 2 — User attempted to recover token from download URL

The user reasoned (incorrectly but logically) that the token might be embedded in the download URL of their first transfer. The URL structure is:

```
https://send.sgraph.ai/en-gb/download/#<transfer_id>/<base64_aes_key>
```

They took the base64 string after the `/`, decoded it, and submitted the result as their token. What they decoded was the raw AES-256-GCM encryption key — 32 bytes of binary data, base64-encoded. Decoded, this produces arbitrary binary including bytes outside the printable ASCII range.

### Step 3 — Token format validation is absent

In `send-access-gate.js`, the `submitToken()` function:

```javascript
async submitToken() {
    const token = (input.value || '').trim().toLowerCase();
    if (!token) { error.classList.remove('hidden'); return; }
    // ...
    const result = await ApiClient.checkToken(token);   // uses encodeURIComponent
    // if check endpoint falls through (catch), token is accepted anyway
    ApiClient.setAccessToken(token);  // stored to localStorage unconditionally
    this.showContent();
}
```

There is no format validation before storage. Any non-empty string passes.

### Step 4 — checkToken silently passes

`ApiClient.checkToken()` uses `encodeURIComponent()` on the token before inserting it into the URL path. This percent-encodes the unicode character, so the HTTP request to `/api/transfers/check-token/...` succeeds. The server-side fallback (when admin service is unavailable, or token not found) returns `{ valid: true }` in dev mode. In production, a 404 returns `{ valid: false, reason: 'not_found' }` — but `encodeURIComponent` may cause it to match nothing cleanly.

Regardless, the `catch` block in `submitToken()` silently falls through on any exception:

```javascript
} catch (e) {
    // Token check endpoint unavailable — fall through
}
```

This means even if `checkToken` throws, the token gets saved.

### Step 5 — Corrupted token written to localStorage

`ApiClient.setAccessToken(token)` writes the raw decoded binary string — including `ɸ` (U+0278) — to localStorage. This is now the active token for all future requests.

### Step 6 — Every upload attempt fails with a browser exception

`_authHeaders()` reads the token from localStorage and places it directly into the `x-sgraph-access-token` HTTP header:

```javascript
_authHeaders() {
    const token = this.getAccessToken();
    return token ? { 'x-sgraph-access-token': token } : {};
}
```

HTTP headers are ISO-8859-1. The browser's fetch API validates header values before sending. `ɸ` is U+0278, outside ISO-8859-1, so fetch throws synchronously:

```
Failed to execute 'fetch' on 'Window': Failed to read the 'headers' property
from 'RequestInit': String contains non ISO-8859-1 code point.
```

This exception is caught by `send-upload.js`'s catch block:

```javascript
} catch (err) {
    if (err.message === 'ACCESS_TOKEN_INVALID') {
        document.dispatchEvent(new CustomEvent('access-token-invalid'));
        return;
    }
    this.errorMessage = err.message || this.t('upload.error.upload_failed');
    this.state = 'error'; this.render(); this.setupEventListeners();
}
```

The error message shown to the user is the raw browser exception string — technically accurate, completely meaningless to a non-specialist. The corrupted token remains in localStorage. The user is stuck.

## Required Fixes

### Fix 1 — Token format validation on input (PRIMARY)

**Location:** `send-access-gate.js`, `submitToken()`, before `ApiClient.setAccessToken(token)`

Add a format check immediately after the empty-check:

```javascript
async submitToken() {
    const input  = this.querySelector('#access-token-input');
    const error  = this.querySelector('#access-token-error');
    const submit = this.querySelector('#access-token-submit');
    const token  = (input.value || '').trim().toLowerCase();

    if (!token) {
        error.textContent = this.t('access_gate.empty');
        error.classList.remove('hidden');
        return;
    }

    // NEW: validate token format before doing anything else
    if (!/^[a-z0-9_-]+$/.test(token)) {
        error.textContent = this.t('access_gate.invalid_format');
        error.classList.remove('hidden');
        return;
    }

    // ... rest of submitToken unchanged
}
```

This rejects any token containing unicode, spaces, base64 padding (`=`), or other characters that cannot appear in a valid token. It fires before any network call and before localStorage is touched.

**New i18n key needed:**

```javascript
'access_gate.invalid_format': "That doesn't look like a valid token. Tokens look like: sg-send__name__xxxxx"
```

---

### Fix 2 — Sanitise token before header use (DEFENCE IN DEPTH)

**Location:** `api-client.js`, `_authHeaders()`

Even if an invalid token is somehow in localStorage (corrupted, manually edited, migrated from an old format), sanitise before inserting into a header:

```javascript
_authHeaders() {
    const token = this.getAccessToken();
    if (!token) return {};
    // Reject any token that would cause a fetch header violation
    if (!/^[a-z0-9_-]+$/i.test(token)) {
        this.clearAccessToken();
        document.dispatchEvent(new CustomEvent('access-token-invalid'));
        return {};
    }
    return { 'x-sgraph-access-token': token };
},
```

This auto-clears a corrupted token and triggers the `access-token-invalid` event, which the access gate already listens for and handles by re-showing the token entry form. A user in Pedro's state would have seen the token form again with a prompt to re-enter, rather than an opaque crash.

---

### Fix 3 — Catch the ISO-8859-1 error explicitly in upload (UX)

**Location:** `send-upload.js`, catch block in `_uploadViaDirect` (and `_uploadViaPresigned`)

The current catch block only intercepts `ACCESS_TOKEN_INVALID`. Add detection for the header encoding error:

```javascript
} catch (err) {
    this._setBeforeUnload(false);
    if (err.message === 'ACCESS_TOKEN_INVALID') {
        document.dispatchEvent(new CustomEvent('access-token-invalid'));
        return;
    }
    // NEW: detect corrupted token in localStorage causing header rejection
    if (err.message && err.message.includes('ISO-8859-1')) {
        ApiClient.clearAccessToken();
        document.dispatchEvent(new CustomEvent('access-token-invalid'));
        return;
    }
    this.errorMessage = err.message || this.t('upload.error.upload_failed');
    this.state = 'error'; this.render(); this.setupEventListeners();
}
```

---

### Fix 4 — Copy: explain localStorage scope on token entry form (UX)

**Location:** `send-access-gate.js`, token entry form HTML template

Add a single line of explanatory copy below the token input field:

```html
<p class="hint">${this.t('access_gate.browser_scope_hint')}</p>
```

**New i18n key:**

```javascript
'access_gate.browser_scope_hint': "Your token is saved in this browser only. If you switch browsers or devices, you'll need to enter it again."
```

This is the single change that would have prevented the entire incident.

## Secondary Observation — checkToken fallthrough on exception

In `submitToken()`, the catch block on `ApiClient.checkToken()` silently falls through:

```javascript
} catch (e) {
    // Token check endpoint unavailable — fall through
}
```

This was intended as a resilience measure (don't block the user if the check endpoint is down). However, it means a token that passes format validation but gets a network error during check will be accepted and stored. This is acceptable behaviour — but the comment should be explicit:

```javascript
} catch (e) {
    // checkToken unavailable (network error or service down) — accept token optimistically.
    // Format validation above already ensured it is structurally valid.
    // If the token is actually invalid, the next upload attempt will return 401.
}
```

## What Does NOT Need Changing

- The zero-knowledge encryption architecture is unaffected. The AES key in the URL fragment is working exactly as designed — it never touches the server.
- The `checkToken` endpoint behaviour on the server side is correct.
- The localStorage storage mechanism for the token is correct — the bug is in what value is allowed to be stored there, not in the storage mechanism itself.
- The `encodeURIComponent` in `checkToken` is correct and should stay.

---

## User Impact

One user entered a fully broken state with no self-recovery path. The fix required manual token replacement and a direct conversation to diagnose. With Fix 1 alone, the entire incident would have been a three-second rejection with a clear format hint. With Fix 4 alone, the incident would likely not have occurred (user would have understood the browser-scope limitation immediately).

Priority recommendation: Fix 1 and Fix 4 are the highest value changes — implement together. Fix 2 and Fix 3 are defence-in-depth and should follow in the same session.
