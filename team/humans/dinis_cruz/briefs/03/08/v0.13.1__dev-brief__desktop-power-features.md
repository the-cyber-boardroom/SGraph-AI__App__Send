## The Opportunity

The SG Desktop app is 2MB. It loads in seconds. It already runs the full SG/Send UI. But the real opportunity isn't wrapping the website — it's doing things that a website **cannot do**: opening multiple sites side by side, injecting JavaScript into pages, automating logins, inspecting traffic, and building a personalised command centre for a developer's daily workflow.

The principle: **keep the desktop as lean as possible.** No dual codebases. Prefer web (sg-layout, web components) for everything that can be done in a browser. Use the desktop only for capabilities that genuinely require it: multi-WebView, OS-level integration, JavaScript injection into third-party sites, local secrets for automation.

## The Daily Pain: Tab Hell

A typical development day currently requires opening:

| Site | Why | Frequency |
|---|---|---|
| Claude (multiple windows) | One per agent team (3-5 windows) | Constantly open |
| GitHub | Repos, releases, commits, actions | Multiple times/hour |
| AWS Console | Billing (must watch closely), CloudWatch logs, S3, Lambda | Multiple times/day |
| WorkMail | Sherpa inbox, customer emails | Several times/day |
| Gmail (work) | Main account communications | Several times/day |
| Gmail (personal) | Personal account | Daily |
| send.sgraph.ai | Testing, admin | Multiple times/day |
| vault.sgraph.ai | File management | Multiple times/day |
| workspace.sgraph.ai | Transformations | As needed |

Chrome's tab groups help but don't solve the problem: tabs are still hidden behind each other, switching requires clicking, and there's no way to see multiple sites at once without manually arranging windows.

## Feature 1: Multi-Site Dashboard (Quick Launch + Side-by-Side)

### Phase 1: Quick Launch Buttons

The left sidebar already has icons. Add configurable quick-launch buttons that open sites inside the app:

```
[Send icon]    → send.sgraph.ai
[Vault icon]   → vault.sgraph.ai  
[Tools icon]   → tools.sgraph.ai
[Settings]     → settings panel
[+]            → add custom site: GitHub, AWS, Gmail, Claude, etc.
```

Even just cycling between sites in the same WebView (like tabs but with one-click access) is a massive improvement over browser tab management. The site list is stored in the personal vault — so it syncs across devices.

### Phase 2: Side-by-Side WebViews

Open multiple Tauri WebViews in a grid layout within the same window:

```
┌──────────────────┬──────────────────┐
│ AWS Billing      │ GitHub Actions   │
│                  │                  │
├──────────────────┼──────────────────┤
│ Claude (Team 1)  │ SG/Send          │
│                  │                  │
└──────────────────┴──────────────────┘
```

On the iMac's large screen: 4-6 sites visible simultaneously. One glance shows: billing status, CI/CD status, customer activity, and current work. A "refresh all" button updates every pane.

**Layout presets:** Save named layouts ("Development", "Monitoring", "Customer Support") that restore a specific arrangement of sites. Switch between presets with one click.

### Phase 3: Background Polling

Some sites don't need to be visible — they just need to be checked. The desktop app can poll in the background:

- WorkMail: check for new emails every 5 minutes, show badge count
- GitHub: check for new commits/PRs, show notification
- AWS: check billing threshold, alert if approaching limit

This runs as a headless WebView (no display needed). Extract the relevant data via JavaScript injection and surface it in a compact notification bar.

## Feature 2: JavaScript Injection into Third-Party Pages

### The Capability

Tauri's WebView allows injecting JavaScript into loaded pages. This unlocks:

| Use Case | What It Does |
|---|---|
| **Remove clutter** | Strip left menus, sidebars, ads, unnecessary UI from sites like Claude, GitHub, AWS Console |
| **Extract data** | Pull specific data points from a page (billing totals, commit counts, email counts) without seeing the full page |
| **Automate forms** | Fill login forms, click buttons, navigate to specific pages |
| **Custom styling** | Inject CSS to theme third-party sites to match the SG Desktop aesthetic |
| **Content scripts** | Run custom scripts on specific sites (like a Chrome extension's content scripts, but more powerful) |

**Example: Clean Claude UI.** Open Claude in a WebView, inject a script that hides the left conversation list, maximising the chat area. The script is saved in the personal vault and auto-applied every time Claude opens in the desktop.

**Example: AWS Billing at a Glance.** Open AWS billing in a hidden WebView, inject a script that extracts the current month's total, and display just the number in the desktop's status bar: "AWS: $47.23 MTD".

### Per-Site Scripts

Each site in the quick-launch list can have associated scripts:

```json
{
  "name": "AWS Console",
  "url": "https://console.aws.amazon.com",
  "scripts": [
    { "name": "Extract billing total", "run": "on_load", "code": "..." },
    { "name": "Remove sidebar", "run": "on_load", "code": "..." }
  ],
  "credentials": "vault://user-vault/keys/aws-console"
}
```

Scripts are stored in the personal vault (encrypted). They're essentially personal Chrome extension content scripts — but managed through the vault, not through the browser's extension system.

## Feature 3: Login Automation

### The Concept

The personal vault stores credentials (encrypted with the master password). The desktop app can automate login sequences:

1. Open a site in a WebView
2. Detect the login form (by known selectors or by heuristic)
3. Retrieve credentials from the vault
4. Fill the form fields via JavaScript injection
5. Submit

This is what 1Password and LastPass do via browser extensions — but we control the WebView, so we can do it without an extension.

### Security Model

- Credentials are encrypted in the vault (master password required to unlock)
- Auto-fill only works in the desktop app (not on the web — web version doesn't have WebView access)
- Each site's credentials are a separate vault entry
- The user explicitly grants auto-login per site (not automatic for all sites)
- Session cookies persist in the Tauri WebView between launches (so login is only needed once per session/week)

### What This Replaces

Instead of: open Chrome → find the right tab → click on 1Password extension → select the credential → fill → submit — it's: click the AWS button in the sidebar → desktop opens AWS, auto-logs in, shows the billing page. One click.

## Feature 4: Traffic Inspection (Explore)

### The Question

Can Tauri intercept or inspect HTTP traffic from its WebViews? If yes, this unlocks:

- **API monitoring:** See what requests a site makes, what data it sends/receives
- **Data extraction:** Capture API responses (JSON) rather than scraping HTML
- **Security analysis:** Inspect what third-party scripts are loading on a page
- **Performance monitoring:** Track request timing, payload sizes

### What to Investigate

| Question | Impact |
|---|---|
| Can Tauri intercept WebView HTTP requests? | If yes: API-level data extraction instead of JS injection |
| Can Tauri modify WebView requests/responses? | If yes: can inject headers, modify responses, proxy traffic |
| Is this OS-specific? | macOS WebKit vs Windows WebView2 may have different capabilities |
| What about HTTPS? | Can Tauri see decrypted traffic from its own WebViews? (Probably yes, since it controls the WebView) |

This is exploratory — we don't need this for the MVP. But if traffic inspection is available, it transforms the desktop from a "better browser" into a full development/monitoring workstation.

### The Concept

A dedicated view in the desktop app for monitoring and managing the SG/Send infrastructure:

```
┌─────────────────────────────────────────────────┐
│ SG Desktop — DevOps View                        │
├─────────────┬───────────────┬───────────────────┤
│ AWS Status  │ GitHub CI/CD  │ Alerts            │
│ S3: ✅      │ Send: ✅ pass │ AWS: $47 MTD      │
│ Lambda: ✅  │ Vault: ⚠️ 1   │ Disk: 62% used    │
│ CF: ✅      │ Tools: ✅ pass│ Errors: 0 (24h)   │
│ SES: ⚠️     │ Desktop: ✅   │                   │
├─────────────┴───────────────┼───────────────────┤
│ CloudWatch Logs (live tail) │ Chat with Agent   │
│ [streaming log entries...]  │ "Hey DevOps, what │
│                             │  happened with    │
```

This combines: AWS service status (from the billing/console WebView), GitHub Actions status (from the GitHub WebView), and an agent chat panel (for asking the DevOps agent about issues).

Eventually this could be its own application variant — "SG Desktop: DevOps Edition" — but for now it's a view/layout preset within the main desktop app.

## Feature 6: Desktop in CI/CD (GitHub Actions)

### The Question

Can the Tauri desktop app run in a GitHub Actions container? This would enable:

- **Automated UI testing:** Launch the app in CI, run through test scenarios, capture screenshots
- **Automated screenshots for docs/marketing:** Boot the app, take screenshots of specific views
- **Headless monitoring:** Run the app in the background, poll sites, extract data, generate reports

### What to Investigate

| Question | Notes |
|---|---|
| Does Tauri run headless? | May need a virtual display (Xvfb on Linux) |
| Can GitHub Actions containers run GUI apps? | Yes, with Xvfb or similar |
| Can we script the app from outside? | Tauri supports CLI arguments and IPC |
| Can we capture screenshots programmatically? | Tauri has window screenshot capabilities |

If this works, the desktop app becomes not just a user tool but an automation platform — running in CI to test, monitor, and generate assets.

## Design Principle: Web First, Desktop for Exceptions

Every capability should work on the web wherever possible. The desktop adds value only for things the web cannot do:

| Capability | Web | Desktop Only |
|---|---|---|
| SG/Send file transfer | ✅ | |
| Vault management | ✅ | |
| Workspace transformations | ✅ | |
| Multi-site side-by-side | ❌ | ✅ |
| JavaScript injection into third-party sites | ❌ | ✅ |
| Login automation with vault credentials | ❌ | ✅ |
| Traffic inspection | ❌ | ✅ |
| Background polling / notifications | ❌ | ✅ |
| OS keychain integration | ❌ | ✅ |
| Local storage server | ❌ | ✅ |

The desktop is an amplifier for the web, not a replacement.
