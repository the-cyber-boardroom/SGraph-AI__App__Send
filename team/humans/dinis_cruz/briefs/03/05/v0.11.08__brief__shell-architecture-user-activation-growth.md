## Part 1: The Shell — IDE-Like Panel Manager Component

### The Problem

Every UI we build ends up as a combination of panels: left sidebar, main content, right panel, bottom chat, tabbed views. The workspace has it. The vault has it. The admin panel has it. Each project implements its own panel layout from scratch, with its own resize logic, its own panel state management. This complexity is being pushed down to individual tools, where it doesn't belong.

### The Solution: A Shell Component

Build a single, reusable **shell component** — an IDE-like multi-panel window manager — that becomes the container every tool and application runs inside.

**What it provides:**
- Multiple resizable panels (split horizontal, split vertical)
- Drag-and-drop panel reordering (move a panel from left to right)
- Tab support within panels (multiple tools in one panel, tabbed)
- Panel minimise/maximise/close
- Responsive layout (panels stack on mobile, split on desktop)
- Layout persistence (remember panel arrangement across sessions)
- Event broadcasting system (panels communicate via events, not direct references)

**The key principle: tools don't know they're inside the shell.** A tool receives a div and maximises within it. The tool doesn't control its own size — the shell does. The tool doesn't know what's in the panel next to it. The tool just renders into its container and listens for events.

### Event-Driven Communication

Inspired by the Windows message passing model: there's a broadcasting channel, and individual windows listen and react.

```javascript
// Shell broadcasts an event
shell.broadcast('file-selected', { fileId: 'abc', name: 'document.pdf' });

// Any panel/tool that cares, listens
shell.on('file-selected', (event) => {
  // React: load the file in the preview panel
});
```

Tools communicate through the shell's event bus, not by reaching into each other's DOM. This is the same isolation principle as the web component model — blast radius is contained.

### Fractal Nesting

The shell supports nesting: a tool inside a panel can itself contain a shell with sub-panels. A left menu component could have its own internal panels. A workspace tool could have a nested shell with source/transformation/output panels.

```
Shell (outer)
├── Panel: Left sidebar (menu component)
├── Panel: Main content
│   └── Shell (nested)
│       ├── Panel: Source HTML
│       └── Panel: Transformed HTML
├── Panel: Right sidebar
│   └── Shell (nested)
│       ├── Tab: API Log
│       ├── Tab: Events
│       └── Tab: Messages
└── Panel: Bottom (chat/prompt)
```

The same shell component powers every level of nesting. Fractal.

### Component Manifest

Every tool declares its requirements in a manifest:

```json
{
  "name": "video-splitter",
  "version": "1.0.0",
  "panels": [
    { "id": "input", "label": "Video Input", "position": "left" },
    { "id": "output", "label": "Clips", "position": "right" },
    { "id": "controls", "label": "Segment Controls", "position": "bottom" }
  ],
  "dependencies": {
    "sg-video": "latest",
    "upload-dropzone": "v1.0.0"
  },
  "events": {
    "listens": ["file-selected"],
    "emits": ["split-complete", "clip-ready"]
  }
}
```

The shell reads the manifest, creates the panels, loads the dependencies, and wires the events. The tool just provides the rendering logic for each panel.

### No Side Effects

Components must be clean: loading a component renders into its container and listens for events. Unloading a component removes its DOM and event listeners. No global state left behind. No DOM pollution outside the container. This means you can swap component versions at runtime — unload v1.0, load v1.1 — without reloading the page.

This enables the powerful capability of **in-browser version switching**: a dropdown in a panel header lets you choose which version of the component to load. Useful for development ("does the new version of file-preview break anything?") and for users ("I prefer the old layout").

## Part 2: Version Pinning Strategy

### The Lock File Pattern

Inspired by `poetry.lock` (Python) and `package-lock.json` (NPM): during development, tools use `latest` for all dependencies. When merging to main (production), all versions are pinned to their current resolved versions.

```
Development (dev branch):
  import { encrypt } from '/core/crypto/latest/crypto.js';
  import { UploadDropzone } from '/components/upload-dropzone/latest/upload-dropzone.js';

Production (main branch):  
  import { encrypt } from '/core/crypto/v1.2.3/crypto.js';
  import { UploadDropzone } from '/components/upload-dropzone/v1.0.1/upload-dropzone.js';
```

**The merge step** resolves all `latest` references to their current pinned versions. This could be a build script that rewrites import URLs, or a manifest that maps `latest` → specific version at deploy time.

**Why this matters:** In dev, you always test against the newest components (catch incompatibilities early). In production, nothing changes unless you explicitly upgrade (stability). Same principle as lock files, but applied to CDN-served ES module imports.

### Visualising Component Versions

The shell should have a developer mode that shows, for every loaded component, which version is running:

```
[file-preview v1.2.0] [upload-dropzone v1.0.1] [api-logger v0.9.0]
```

In developer mode, each component header shows its version and a dropdown to switch versions. This is a powerful debugging and testing tool — if something breaks, switch one component to a previous version and see if it's the cause.

## Part 3: User Activation — Nudging Strategy

### The Problem

We have paying customers but not enough usage. Users buy credits, test once, and don't come back. Credits sit unused. We need to understand what drives repeated usage and what features trigger engagement.

### The Sherpa's Job: Nudge Users to Use the Platform

**Per-user tracking folders:** Create a folder per customer on the admin side. Track: which messages we've sent, when we sent them, what they responded to, what features they've used, how many credits consumed, how many remaining. All data encrypted — even admin-side, use encrypted storage (we should practice what we preach).

**Nudging sequence:**

| Day | Message | Goal |
|---|---|---|
| 0 | Welcome + thank you + "try dropping a PDF" | First use |
| 2 | "Did you know you can drop a folder? It creates a data room." | Second feature discovery |
| 5 | "Share a file with a friend — they can view it in the browser without installing anything" | First share (activates DRU) |
| 7 | "You have X credits remaining. Here are 3 things to try this week." | Re-engagement |
| 14 | "We just added [new feature]. Try it with your remaining credits." | Feature announcement |

**Encourage "abuse":** We WANT users to burn through credits fast. We need to understand: what happens when credits run out? Does the user top up? Do they churn? What's the tipping point? We can't learn this if users have 490 of 500 credits sitting unused.

**What we need to observe:**
- Which features drive the most uploads?
- Which features drive the most downloads (by recipients)?
- What file sizes are typical?
- How many recipients per share?
- Do users come back without nudging?
- What's the credit exhaustion to top-up conversion rate?

### Credits: Cover Both Uploads AND Downloads

**Change:** Credits should cover both uploads and downloads. Currently credits are upload-only. But if we're tracking downloads and want to understand download patterns, making downloads consume credits (from the sender's balance) gives us better data and protects against bandwidth abuse.

**The math stays generous:** Even if credits cover both directions, at the current pricing (£5 = 500 file operations or 5GB), a typical one-to-one share (upload + download = 2 operations or 2x file size) means £5 still covers hundreds of shares.

**Or switch to bandwidth-based:** £1 per GB of total activity (upload + download). A 100MB file shared with one person = 200MB total = £0.20. A 3.5GB file shared with one person = 7GB total = £7. Transparent and predictable.

### Encourage Social Sharing

**Phase 1 (now):** Nudge users to share tokens with friends and family. "Send a file to a friend — they'll be impressed." Every share is a product demo.

**Phase 2 (with monitoring):** Encourage users to post on LinkedIn. "Share your multilingual SG/Send link on social media." Give them something shareable — a public data room, a portfolio, a demo. Each post is free marketing.

**Phase 3 (with abuse protection):** The human should be creating and sharing daily tokens publicly — demonstrating the product, building content, attracting organic users. This requires confidence in monitoring and rate limiting.

## Part 4: Guest Users

### The Concept

Currently, recipients are anonymous — anyone with the link and key can download. There's no concept of "who downloaded this." The guest user model adds a lightweight identity layer without requiring account creation.

**How it works:**

1. When a recipient visits an SG/Send link, the system generates a **guest fingerprint** from: IP address, browser fingerprint, and/or a generated hash
2. The recipient sees: "You are Guest User [short hash]. You have 5 free downloads."
3. The guest fingerprint is shown to the user — transparent, not hidden
4. Free allowance (e.g., 5 downloads or 1GB) is tied to the guest fingerprint
5. After the free allowance: "Want more? Create an account or buy credits."

**Why this works:**
- No anonymous abuse (each "guest" has a trackable identity, even if it's just a fingerprint)
- Free tier without account creation (like guest checkout in e-commerce)
- Conversion funnel: free guest → registered user → paying customer
- Bot protection: harder to generate thousands of unique guest fingerprints than to hit an anonymous endpoint

**Same model as shopping cart guest checkout:** The user doesn't need to create an account to complete the action. But we know enough about them to limit abuse and to nudge them toward account creation.

## Part 5: Multi-Region Deployment

### The Vision

Deploy SG/Send to multiple AWS regions (and eventually multiple cloud providers) so that users are served from infrastructure in their geographic area.

```
User from Argentina  → AWS São Paulo    → send.sgraph.ai (South America edge)
User from Portugal   → AWS Ireland      → send.sgraph.ai (Europe edge)
User from Japan      → AWS Tokyo        → send.sgraph.ai (Asia Pacific edge)
```

### Why This Matters

**Performance:** Files are stored and served from the nearest region. Upload and download speeds improve dramatically for users far from the current single region.

**Data sovereignty:** "Your files are stored in AWS Ireland" or "Your files are stored in AWS São Paulo." This matters for GDPR (EU data stays in EU), for enterprise customers with data residency requirements, and for the general trust story.

**Resilience:** If one region goes down, others continue serving. True multi-region availability.

**Marketing:** "Your data is in your country." This is a powerful message, especially in markets where US data hosting is viewed with suspicion.

### The Connection to Guest Users

Guest user fingerprints include geographic signal (IP-based region). This means: "You're coming from Brazil, so your files are stored in AWS São Paulo." The guest user sees where their data lives. Transparency builds trust.

### Multi-Cloud (Future)

Eventually: AWS + Azure + GCP. "Your files happen to be stored on Azure in Brazil" or "GCP in Europe." Each provider and region is a deployment target. The user (or their organisation) can choose where their data lives. This connects to the custody services brief — jurisdiction-specific data storage.

### Implementation Approach

Start with CloudFront (already global CDN). The static assets (JS, HTML, CSS) are already served globally. The next step is regional storage — files uploaded in Europe are stored in Europe, files uploaded in South America are stored in South America. The API routes based on the user's geographic location.

This is a P3 item — the infrastructure is the same S3 + CloudFront pattern we already use, just replicated across regions. The engineering challenge is routing and synchronisation, not architecture.
