## The Problem: People Lose Keys

Encryption has a fundamental usability problem: if you lose the key, you lose everything. That is exactly how it should be — it means the encryption is real. But users are terrible at key management. They lose passphrases, wipe browsers, switch devices, forget which key goes with which vault.

As adoption grows and users have dozens or hundreds of vaults, this becomes critical. A single lost key doesn't just lose one file — it loses everything in that vault. The more successful the platform becomes, the bigger this problem gets.

---

## The Idea: Vault Custody Service

A trusted service — engineered like a bank — that stores private keys, verifies vault integrity, ensures data remains accessible, and provides resilient backup.

**This is not SG/Send.** SG/Send is zero-knowledge by design — the server never has the keys. The custody service is the opposite: it explicitly has the keys, by design, because its entire purpose is to be the trusted party that ensures you never lose access.

### Wardley Map Position

This is firmly in the "utility" zone of the Wardley Map. Not an explorer, not custom-built, not even a product in the startup sense. This must be:

- Six sigma reliability
- Nine nines availability
- Physically secured
- Properly engineered
- Auditable
- Compliant with financial-grade regulation

This is the infrastructure that makes the entire ecosystem trustworthy at scale. It has to be done by specialists, done properly, done with the seriousness of a bank vault.

**The good news:** It has far fewer moving parts than the creative, rapidly-evolving product layer. It is a constrained, well-defined problem: store keys securely, verify data integrity, provide backup and recovery. The engineering challenge is operational excellence, not innovation.

## What the Custody Service Does

### 1. Key Storage (Escrow)

The user deposits their private keys with the custody service. The service stores them in hardware security modules (HSMs) or equivalent high-security infrastructure. If the user loses their key, they can recover it from the custody service (after identity verification).

Think of it as a safety deposit box for cryptographic keys.

### 2. Vault Integrity Verification

The custody service periodically verifies that vaults are intact: the encrypted blobs exist, they can be decrypted with the stored keys, the content is not corrupted. This is a health check — "is your data still there and still readable?"

This catches problems early: storage corruption, provider outages, accidental deletion.

### 3. Multi-Location Resilient Backup

For critical data, the custody service stores encrypted copies in multiple locations:

- Primary cloud storage (current SG/Send backend)
- Secondary cloud storage (different provider, different region)
- Offline / air-gapped physical storage (tape, cold storage, secure facility)

Three copies, three locations, three failure domains. The user's data survives any single provider failure, region outage, or even a complete cloud provider shutdown.

### 4. Anonymisation Services

Because the custody service can decrypt vault contents (it holds the keys), it can provide value-added services:

- **Anonymise and re-encrypt:** Decrypt a dataset, strip PII, re-encrypt with a different key for sharing
- **Aggregate across vaults:** Combine anonymised data from multiple vaults for research (e.g., clinical trials from the healthcare brief)
- **Validate and attest:** Verify that vault contents meet certain criteria, sign an attestation without revealing the contents

This is the "pod" concept from the file safety orchestration brief — a trusted intermediary that can see plaintext but operates under strict controls.

### 5. Access Control Management

The custody service can manage who has access to which vaults — a centralised access control layer for users who want it. This is adjacent to what the Solid project (Tim Berners-Lee) attempted: user-controlled data pods where the user decides who can access what.

The difference: PKI on top makes it cryptographically enforced, not just policy-enforced. Access is granted by sharing a key, not by setting a permission flag that an admin could override.

## The Market

### Data Sovereignty

Countries and regions increasingly require data to be stored within their jurisdiction. A custody service in a specific jurisdiction provides:

- Guaranteed data residency
- Local legal jurisdiction
- Compliance with local data protection laws
- Physical proximity (performance)

A European custody service stores European citizens' vault keys and backup data within the EU. A UK service does the same for UK data. The encrypted blobs can be stored anywhere (they're encrypted, jurisdiction doesn't matter for ciphertext), but the keys — and therefore the ability to access the data — are jurisdictionally controlled.

### Long-Term Data Preservation

Will your data still be accessible in 3, 5, 10, 20 years? Cloud providers change pricing, deprecate services, go bankrupt. The custody service provides:

- Format migration (if encryption standards change, re-encrypt with new algorithms)
- Storage provider migration (move data between providers as needed)
- Key rotation (update keys on a schedule, old keys retired securely)
- Continuity guarantee (the custody service's entire business model is "your data is still there")

### Enterprise and Regulated Industries

Banks, hospitals, law firms, government agencies — any organisation that must retain data for regulatory periods and prove it hasn't been tampered with. The custody service provides the audit trail, the integrity verification, and the guaranteed availability that regulators require.

## Architecture: The Trust Model

```
+-------------------+     +-------------------+     +-------------------+
|                   |     |                   |     |                   |
|   SG/Send         |     |   Custody         |     |   User's          |
|   (Zero-Knowledge)|     |   Service         |     |   Devices         |
|                   |     |   (Trusted)       |     |                   |
|   Stores:         |     |   Stores:         |     |   Stores:         |
|   Encrypted blobs |     |   Private keys    |     |   Private keys    |
|   Cannot decrypt  |     |   Can decrypt     |     |   Can decrypt     |
|                   |     |   Backup copies   |     |   Working copies  |
|                   |     |   Integrity logs  |     |                   |
+-------------------+     +-------------------+     +-------------------+
```

**SG/Send** remains zero-knowledge. Its security model is unchanged.

**The Custody Service** is a separate entity with a different trust model: it is explicitly trusted with keys. Its security comes from operational controls (HSMs, physical security, audits, insurance) rather than from zero-knowledge architecture.

**The User** can choose: zero-knowledge only (manage your own keys, accept the risk of loss) or custody-assisted (deposit keys with a trusted party, gain recovery and backup at the cost of trusting that party).

Both modes coexist. The custody service is opt-in. Users who want pure zero-knowledge never interact with it.
