**version** v0.11.1  
**date** 4 Mar 2026  
**from** Human (project lead)  
**to** Developer (lead), Architect  
**type** Dev brief — workspace evolution, transformation pipeline, plugin architecture  

---

## The Problem with the Current Approach

The current HTML workbench sends the entire HTML page to an LLM with a transformation prompt. The LLM returns the modified HTML. This works — but it has serious limitations:

**Token inefficiency:** Sending 30-60K tokens of HTML content to an LLM and asking it to return the whole thing modified is wildly inefficient. The LLM is being used as a runtime to execute what is fundamentally a code transformation.

**Most models can't do it well.** Tested across multiple providers: most models — including expensive ones from OpenAI, Google, and others — struggle significantly with "take this HTML, modify it, return the complete result." They lose content, break structure, hallucinate tags. Ironically, the two that worked acceptably were Anthropic's Claude Sonnet 4.6 (or 4.7) and Haiku, which handled it cleanly.

**It doesn't scale.** Every transformation re-sends the full HTML. A pipeline of 5 transformations sends the full page 5 times. And if the page hasn't changed much between runs, you're re-processing identical content at full token cost every time.

---

## The Paradigm Shift: Data → Code → Data

Instead of: **HTML → LLM → HTML** (the LLM IS the transformation)

Move to: **HTML → JavaScript → HTML** (the LLM WRITES the transformation)

Or more generally: **Data → Code → Data**

The LLM's job changes from being the runtime to being the author. The LLM writes the JavaScript transformation code. That code runs locally in the browser against the HTML. The transformation executes instantly, deterministically, at zero token cost after the initial code generation.

This is the fundamental principle: **use GenAI and LLMs to help create the code that does the transformations. The actual transformations are all done with code.**

## The Transformation Pipeline

A real-world example: take a website, clean it up, classify its content, and reassemble a filtered version.

### Step 1: Compress / Clean the HTML

Take the raw HTML and strip it down. Remove ads, tracking scripts, analytics, unnecessary headers, bloated CSS, callback functions. Reduce the page to its essential content without changing the visual elements.

This is itself a JavaScript transformation: parse the DOM, remove elements matching selectors (ad containers, tracking pixels, script tags for analytics), strip inline styles that aren't needed, collapse redundant wrappers.

**Why this matters for the pipeline:** Every subsequent step processes less data. If the raw page is 200KB and the cleaned version is 40KB, every later transformation is 5x faster and cheaper.

**Interesting question:** How much of an HTML page can you remove without changing the visual experience? This is effectively HTML compression — strip everything the user doesn't see or want.

### Step 2: Tag and Extract Content

Write JavaScript that identifies and tags specific content elements on the page. These are site-specific DOM selectors — hardcoded for this particular website, not generic.

For example, on a tech news page:
- Each article has a specific CSS class or structure
- Headlines are in a predictable element
- Body text follows a pattern
- Dates, authors, categories are in known locations

The JavaScript tags each element with a unique ID and extracts the content into a structured format:

```json
{
  "elements": [
    {
      "id": "article_001",
      "hash": "sha256_of_content",
      "selector": ".article-card:nth-child(1)",
      "headline": "New Serverless Framework Released",
      "body": "The full text of the article...",
      "date": "2026-03-04",
      "category": "serverless"
    }
  ]
}
```

**Critical:** Leave placeholders in the HTML where content was extracted. These are the reconnection points — when the page is reassembled later, the classified/transformed content goes back into these slots.

### Step 3: Classify the Extracted Content

This is the only step that may need an LLM — and only for content not already cached (see the hash caching brief). The extracted JSON is sent for classification: topic, sentiment, relevance to user criteria.

But the classifier doesn't have to be an LLM. It could be AWS Comprehend, a local language model, a rule-based classifier, or even a human reviewer. The architecture is pluggable — the classification step is an interface, not an implementation.

And crucially: **classification only happens once per unique content hash.** After the first classification, the result is cached. Second time the same content appears, classification is instant (zero cost).

### Step 4: Filter and Transform Based on Classification

Another JavaScript transformation: take the classified data and the placeholder-tagged HTML, and decide what stays and what goes. Multiple modes:

- **Remove and leave gaps:** Filtered content is removed, empty space remains (simple)
- **Remove and collapse:** Filtered content is removed, page reflows to fill gaps (medium)
- **Remove and rebuild:** Filtered content is removed, page is restructured with new layout (complex — LLM writes this JS)

### Step 5: Reassemble the Page

Final JavaScript transformation: take the filtered/transformed content and put the page back together. This produces the final HTML — a clean, classified, filtered version of the original page.

### The Full Pipeline

```
Raw HTML (200KB)
    |
    v
[Step 1: Clean/Compress]     <- JavaScript (written by LLM, runs locally)
    |
    v
Clean HTML (40KB)
    |
    v
[Step 2: Tag & Extract]      <- JavaScript (site-specific selectors)
    |
    v
Tagged HTML + Extracted JSON
    |
    v
[Step 3: Classify]            <- LLM / classifier (only for new content, cached)
    |
    v
Classified JSON
    |
    v
[Step 4: Filter]              <- JavaScript (apply user criteria)
    |
    v
Filtered content
    |
    v
[Step 5: Reassemble]         <- JavaScript (rebuild page)
    |
    v
Final HTML (customised, clean, filtered)
```

**LLM involvement:** Steps 1, 2, 4, and 5 are pure JavaScript — zero LLM cost at runtime. Step 3 uses an LLM only for uncached content. In steady state (same site, content mostly unchanged), the entire pipeline runs with zero LLM calls.

## Site-Specific by Design

These transformations are explicitly site-specific. They don't need to work on every website. They are hardcoded DOM selectors, hardcoded extraction logic, hardcoded reassembly templates — all written for one specific site.

**This is a feature, not a limitation.** Generic web scraping/transformation is an unsolved problem. Site-specific transformation is a solved problem — it's just tedious to write the code. The LLM eliminates the tedium: describe what you want, the LLM writes the site-specific JavaScript.

Over time, patterns emerge. Common transformations (remove ads, extract article text, classify by topic) become reusable plugins. But the starting point is: get it working for this one site, perfectly.

## The JavaScript Execution Environment

Each transformation step needs a safe execution environment:

**Option A: iframe with loaded DOM.** Load the HTML into an iframe, run the JavaScript against the iframe's document. The iframe provides a real DOM — selectors work, layout is computed, CSS applies.

**Option B: Virtual DOM / DOMParser.** Parse the HTML into a DOM tree without rendering. Faster (no layout computation), but CSS selectors that depend on computed styles won't work.

**Option C: Hybrid.** Use DOMParser for structural transformations (tag extraction, content removal) and iframe for visual transformations (where you need to verify the visual result).

For the workbench UI: the left panel shows the input (source HTML or previous step's output), the right panel shows the output (the result of running the JavaScript transformation), and both have the Rendered toggle to see the actual visual result in an iframe.

## Folder Structure and Code-Behind Pattern

### The Problem

A single transformation pipeline generates many files: the source HTML, the cleaned version, the extraction script, the extracted JSON, the classification results, the filter script, the reassembly script, the final HTML, plus all the prompts used to generate the JavaScript. Showing all of these in the file browser is overwhelming.

### The Code-Behind Pattern

Inspired by ASP.NET's code-behind model (where `page.aspx` had a hidden `page.aspx.cs` that the IDE showed as a nested child), the workspace should present a simplified view:

**What the user sees in the file browser:**
```
my-website/
  source.html
  step-1-clean.html
  step-2-extract.json
  step-3-classify.json
  step-4-final.html
```

**What actually exists on disk (in the vault):**
```
my-website/
  source.html
  step-1-clean.html
  step-1-clean.js              <- hidden: the transformation script
  step-1-clean.prompt.md       <- hidden: the prompt used to generate the script
  step-2-extract.json
  step-2-extract.js            <- hidden: the extraction script
  step-2-extract.prompt.md     <- hidden: the prompt
  step-3-classify.json
  step-3-classify.config.json  <- hidden: classification criteria
  step-4-final.html
  step-4-final.js              <- hidden: the reassembly script
  pipeline.json                <- hidden: pipeline definition (step order, connections)
```

The "code-behind" files (.js, .prompt.md, .config.json) are accessible when you need them (click to expand, or switch to "developer view"), but hidden by default so the user focuses on the data flow — inputs and outputs — not the plumbing.

### Plugin as Folder Convention

Each transformation pipeline is a folder. The folder contains a `pipeline.json` that defines the steps, their order, and their input/output connections. The plugin system knows how to read this manifest and present the appropriate UI.

When a folder contains a `pipeline.json`, the workspace switches to pipeline mode: showing steps as blocks, visualising the flow, letting the user run steps individually or in sequence.

## UI for Pipeline Development

### The Development Experience

This is fundamentally a development tool — a tool to create the JavaScript that will execute the transformations. The UI should support this workflow:

1. **Describe the transformation in natural language.** "Take this HTML page and remove all ad containers and tracking scripts."
2. **LLM generates the JavaScript.** The code appears in an editor panel.
3. **Run the JavaScript against the source HTML.** See the result side-by-side.
4. **Iterate.** Edit the prompt or the code directly. Re-run. Compare.
5. **Save as a pipeline step.** The code, prompt, and output are saved to the vault.
6. **Chain steps.** Connect the output of one step to the input of the next.

### Pipeline Visualisation (Future)

A visual pipeline editor — blocks connected by arrows, similar to n8n or other workflow orchestration tools. Each block is a step. Click a block to edit its code/prompt. Run the pipeline end-to-end or step-by-step.

This is P2/P3 — the file-based pipeline with sequential execution is the MVP.

## Dual Folder Views: Vault Structure vs Context View

### The Problem

The vault file browser shows the raw folder structure — every file, every code-behind file, every prompt, every intermediate output. When working on a specific page transformation, most of those files are noise. The user cares about: source, transformation script, result, and their prompt variations. Not the 15 other files in the vault.

### Two Panels

**Panel 1: Vault folder view.** The actual folder structure stored in the vault. Shows everything. This is the "developer view" — useful for seeing what's really there, debugging, managing files across pages.

**Panel 2: Context-specific view.** A purpose-built panel for the current page/pipeline being worked on. Shows only:
- Source (button — click to load into the editor)
- Transformation script (button — click to edit the JS)
- Result (button — click to see the transformed output)
- Prompt variations (list — click to load a specific prompt)
- Multiple versions of each (if saved)

The context view is what the user works with 90% of the time. The vault view is available when needed (toggle, or side-by-side). Eventually the vault view can be hidden by default, with the context view as the primary navigation.

This is the same principle as the code-behind pattern but applied to navigation, not just file visibility.

## Prompt Composer: Choose What Context to Include

### The Problem

The current prompt workflow sends the HTML to the LLM. But in the three-element model (source + transformation script + result), the user might want to send different combinations of context depending on what they're asking the LLM to do.

### What the User Should Be Able to Choose

When composing a prompt, the user selects which artefacts to include as attachments/context:

| Attachment | When You'd Include It |
|---|---|
| **Nothing** | "What's a good CSS grid layout for a news page?" — general question, no context needed |
| **Source HTML** | "Write JavaScript to extract all article headlines from this page" — LLM needs to see the HTML structure |
| **Transformation script** | "This script has a bug, the selector doesn't match — fix it" — LLM needs to see the code |
| **Result HTML** | "The output is missing the footer — what went wrong?" — LLM needs to see what was produced |
| **Source + Result** | "Compare these and tell me what's different" — diff analysis |
| **Source + Script + Result** | "Here's the full pipeline — why is the table formatting broken?" — full debugging context |

### UI

The prompt composer (bottom panel) should have checkboxes or toggles next to each artefact: "Include: [ ] Source [ ] Script [ ] Result". The selected artefacts are appended to the user's prompt as context when sent to the LLM. The system prompt explains what each attachment is.

This keeps token usage efficient — you only send what the LLM needs for this particular question.

## Execution History: Save Every LLM Call

### The Principle

Every LLM interaction is saved as an input/output pair. Not just the final "accepted" step — every attempt, every iteration, every failed experiment. This creates an audit trail the user can scrub through.

### What Gets Saved Per Execution

```
execution-001/
  input.md        <- The prompt as sent (system prompt + user prompt + attachments)
  output.md       <- The raw LLM response
  output.js       <- Extracted code (if the response contained code)
  output.html     <- Extracted HTML (if the response contained HTML)
  metadata.json   <- Timestamp, model used, token count, latency, success/fail
```

### Why This Matters

- **Rollback:** "The prompt I used three attempts ago worked better — let me go back to that"
- **Comparison:** "Why did Sonnet produce better code than GPT for this transformation?"
- **Cost tracking:** Each execution records token usage — the user can see how much each attempt cost
- **Learning:** Review what prompts worked and which didn't — builds intuition for prompt engineering
- **Debugging:** When a transformation produces unexpected results, the full chain of LLM interactions is inspectable

### Storage

Execution history can grow large. Two approaches:
- **Keep all locally** (in the vault) — storage is cheap, history is valuable
- **Keep recent, archive old** — last N executions in the active folder, older ones archived to a subfolder or compressed

The archive/download feature should let the user export the entire pipeline — source, all steps, all prompts, all execution history — as a single zip for backup or sharing.

## Performance: Rapid Save/Encrypt Cycles

### The Concern

The transformation workflow generates lots of saves: save the prompt, save the LLM response, save the extracted code, save the transformed output, save the execution metadata. Each save involves encryption + upload to the vault. If each save takes 500ms, five saves per execution means 2.5 seconds of overhead just for persistence.

### What Needs to Be Measured

- Time to encrypt a typical file (1KB prompt, 50KB HTML, 5KB JS) using AES-256-GCM
- Time to upload the encrypted blob to the server
- Total round-trip per save operation
- Cumulative overhead for a typical transformation cycle (5-10 saves)

### Optimisation Strategies

- **Batch saves:** Instead of saving each file individually, batch multiple files into a single encrypted blob and upload once. Save the prompt + response + extracted code + metadata as one operation, not four.
- **Async saves:** Save in the background while the user continues working. Show an "unsaved" indicator, but don't block the UI on persistence.
- **Debounce:** If the user is rapidly iterating (edit prompt, run, edit again, run again), don't save every intermediate state — save after a pause, or on explicit "save" action.
- **Local-first:** Write to local storage (or in-memory) immediately, sync to the vault asynchronously. The user never waits for the server.

### Connection to Git-Like Object Store

The object store architecture (from today's companion brief) is relevant here: content-addressable storage means identical content is stored once. If the source HTML doesn't change between executions, it's the same blob — no re-upload needed. Only the new/changed files (prompt, response, metadata) need to be written.

## Lazy Vault Loading: Only Load What's Needed

### The Problem

Loading the entire vault tree structure on page open doesn't scale. A vault with 50 page folders, each with 10+ files (source, scripts, prompts, versions, execution history), could have hundreds of files. Loading all of them to display one page's transformation workspace is wasteful.

### The Approach

The main vault tree only holds pointers (references) to page folders. When the user opens a specific page folder, only that folder's contents are loaded and decrypted. Other folders remain as collapsed references until opened.

```
Vault tree (always loaded, lightweight):
  sg-send-website/     <- pointer, not loaded
  acme-homepage/       <- pointer, not loaded
  client-portal/       <- pointer, LOADED (user is working here)
```

This is the same principle as Git's tree objects: the root tree points to subtrees, and subtrees are only resolved when you need them. It also connects to the "trees within trees" architecture from the Git-like object store brief.

### Archive and Download

When a pipeline is complete, the user should be able to archive the entire folder — source, all steps, all prompts, all execution history, all versions — as a single downloadable package. This serves as a backup, a shareable deliverable (send the pipeline to a colleague), or an archival record. Once archived, the folder can be collapsed back to a pointer in the vault tree, freeing up the active workspace.

## Beyond HTML: Data → Code → Data

The pattern generalises beyond HTML:

| Source Data | Transformation Code | Output Data |
|---|---|---|
| HTML page | JavaScript (DOM manipulation) | Modified HTML |
| HTML page | JavaScript (content extraction) | JSON / CSV |
| Markdown document | JavaScript (text processing) | Classified JSON |
| JSON data | JavaScript (data transformation) | Different JSON structure |
| CSV data | JavaScript (analysis) | Summary / report |
| Any text | JavaScript (parsing + transformation) | Any structured output |

The workbench is not an HTML tool. It is a **data transformation tool** where the LLM writes the transformation code and the code executes locally. HTML is the first and most visual use case, but the architecture works for any data-to-data transformation.
