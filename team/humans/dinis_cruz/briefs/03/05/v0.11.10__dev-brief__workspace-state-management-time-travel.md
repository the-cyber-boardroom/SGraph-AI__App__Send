# Dev Brief: Workspace State Management — Execution Bundles, Time Travel, and Forking

**version** v0.11.10  
**date** 5 Mar 2026  
**from** Human (project lead)  
**to** Developer (lead), Architect  
**type** Dev brief — workspace evolution, state persistence, execution history  

---

## Context: What We Have Now

The workspace (v0.1.0, live at workspace.sgraph.ai) is already a powerful LLM development environment with 19 web components. The current execution flow:

1. Load an HTML source from the vault into the SOURCE panel
2. Write or load a JavaScript transformation in the SCRIPT panel
3. Click Run — the script executes in a sandboxed iframe against the HTML
4. See the result in the RESULT panel (rendered or code view)
5. In the CHAT panel, compose an LLM prompt with selectable context (Src, Data, Script, Result, Console checkboxes)
6. Paste screenshots for multi-modal context (already working — clipboard images as base64)
7. LLM responds with code or analysis — response appears in LLM OUTPUT
8. Copy the LLM's code to the Script panel, run it, iterate

The `ExecutionHistory` component already saves every LLM interaction as JSON in the vault's `/executions` folder (prompt, system prompt, response, model, tokens, latency, attachments). But this history is append-only and flat — there's no concept of forking, no time travel, no ability to reload a previous state and branch from it.

---

## The Problem

The current workflow is linear: each request follows the previous one, and there's no structured way to go back. If request #7 produced a worse result than request #4, the user has to manually reconstruct request #4's state (reload the source, reload the script from that point, remember what context they included). There's no "undo to that point and try a different approach."

Also, every piece of state is scattered: the source HTML is a file in the vault, the script is another file, the LLM prompt is in the chat panel, the included context is determined by checkboxes, the screenshots are pasted and ephemeral. There's no single snapshot that captures "everything that was true at the moment I made this request."

---

## The Solution: Execution Bundles

### What Is a Bundle?

An **execution bundle** is a complete snapshot of the workspace state at the moment an LLM request is made (or at any point the user chooses to save). It captures everything needed to reproduce that exact request and result.

### What a Bundle Contains

```json
{
  "id": "exec-2026-03-05T17-08-42-001",
  "timestamp": "2026-03-05T17:08:42.001Z",
  "parent_id": "exec-2026-03-05T17-05-10-001",

  "config": {
    "model": "anthropic/claude-3.7-sonnet",
    "streaming": true,
    "gen_js_mode": false
  },

  "context": {
    "source": { "ref": "vault://test-vault-1/html-pages/theregister/index.html" },
    "script": { "ref": "vault://test-vault-1/html-pages/theregister/transform.js" },
    "data": null,
    "result": { "ref": "vault://test-vault-1/html-pages/theregister/removed__new_mode.html" },
    "console": "Removed 31 negative articles. 71 articles remaining.\nRedistributed 2 articles from nearly empty rows.",
    "screenshots": [
      { "type": "image/png", "data": "base64...", "description": "Screenshot of layout with white gaps" }
    ]
  },

  "prompts": {
    "system": "you are an experienced web developer and architect who is helping to create HTML and JS transformations...",
    "user": "Hi the transformation script is doing a great job at removing articles with negative rating, but has you can see in the attached image, the layout is not being very well preserved, can you improve the script...",
    "include_flags": { "src": true, "data": false, "script": true, "result": true, "console": false }
  },

  "response": {
    "text": "// Function to check if an article has negative sentiment...",
    "tokens_in": 43300,
    "tokens_out": 1800,
    "cost": 0.1567,
    "duration_ms": 28700,
    "speed_tps": 62.3,
    "finish": "stop"
  }
}
```

### Key Design Decisions

**References, not copies.** The bundle stores `vault://` references to source HTML, scripts, and results — not the full content. If the source HTML is 94KB (like the Register's index.html in the screenshot), we don't duplicate it in every bundle. We just point to it. The content is loaded on demand when the bundle is opened.

Exception: console output, screenshots, and the user's prompt text are stored inline (they're small and ephemeral — not files in the vault).

**Parent ID creates the tree.** Every bundle points to its parent (the bundle that was active when this request was made). This creates a tree structure — exactly like git commits, or like Claude/ChatGPT conversation forks. Request #4 is the parent of #5, #6, #7. But if you go back to #4 and make a different request, the new bundle's parent is #4, creating a fork.

```
exec-001
  └── exec-002
       └── exec-003
            ├── exec-004 (original path)
            │    └── exec-005
            └── exec-006 (forked from 003 — tried different approach)
                 └── exec-007
```

**Save is explicit.** Not every LLM request automatically becomes a saved bundle. The user chooses when to save (a "Save State" button). This keeps the history clean — only meaningful checkpoints, not every typo correction. However, all LLM interactions continue to be logged by the existing `ExecutionHistory` component as they are today (append-only log). The bundles are a curated, forkable layer on top of the raw log.

## Time Travel

### Loading a Previous Bundle

The user browses the execution history (list or tree view), selects a bundle, and the workspace reconstructs that state:

1. Load the source HTML from the vault reference
2. Load the script from the vault reference
3. Load the result from the vault reference
4. Restore the console output
5. Restore the prompt text and system prompt
6. Restore the include flags (which checkboxes were active)
7. Restore the screenshots
8. Show the LLM response

The workspace is now in the exact state it was at that point in time. The user can review what they did, or modify something and execute again (which creates a fork).

### Forking

When the user loads a previous bundle and makes a new request, the new bundle's `parent_id` points to the loaded bundle, not to the most recent one. This creates a branch:

- "I was on a good track at request #4, but #5-#7 went wrong. Let me go back to #4 and try something different."
- Load bundle #4 → modify the prompt → execute → new bundle #8 is created with parent=#4
- The tree now shows two branches from #4: the original path (#5→#6→#7) and the new path (#8→...)

### Visualisation (Future)

A tree/graph view showing the execution history as a branching structure — similar to how Claude and ChatGPT show conversation forks. Click any node to load that state. This is P2 — for the MVP, a flat list with parent references is sufficient.

## Storage

### Where Bundles Live

Bundles are stored in the vault under a dedicated folder per workspace/page:

```
vault/
  html-pages/
    theregister/
      index-original.html
      transform.js
      removed__new_mode.html
      .executions/                    <- Bundle storage
        exec-2026-03-05T17-05-10.json
        exec-2026-03-05T17-08-42.json
        exec-2026-03-05T17-12-15.json
        tree.json                     <- Index: parent-child relationships
```

The `tree.json` file is a lightweight index that maps bundle IDs to parent IDs, so the tree structure can be loaded without opening every bundle file.

### Performance

Bundle JSON files are small (1-3KB each — mostly references and metadata, not content). Loading the tree index is a single file read. Loading a specific bundle's full state means resolving the vault references (loading the source HTML, script, result) which would happen anyway when the user works on that page.

## Acceptance Criteria

### Phase 1: Execution Bundles
| # | Criterion |
|---|---|
| 1 | Bundle JSON schema defined with all fields (config, context refs, prompts, response, parent_id) |
| 2 | "Save State" button in the workspace captures current state as a bundle |
| 3 | Bundle stored in vault under `.executions/` folder for the current page |
| 4 | Context references use `vault://` pointers, not content copies |
| 5 | Screenshots and console output stored inline in the bundle |
| 6 | Parent ID links to the previously active bundle (or null for first) |

### Phase 2: Time Travel
| # | Criterion |
|---|---|
| 7 | Execution history list shows saved bundles with timestamp, model, prompt preview |
| 8 | Clicking a bundle loads its full state into the workspace (source, script, result, prompt, screenshots) |
| 9 | Modifying and executing from a loaded bundle creates a fork (new bundle with parent = loaded bundle) |
| 10 | Tree index (tree.json) maintained for parent-child relationships |

### Phase 3: Visualisation
| # | Criterion |
|---|---|
| 11 | Tree/graph view of execution history showing branches and forks |
| 12 | Click any node to load that state |
| 13 | Visual indicator of which branch is currently active |
