## Context

We now have a working vault, web-based workspaces, and a CLI that can clone, pull, and push from vaults. The foundation is solid. This brief is about making the workspace into a proper **HTML Workbench** — a tool for doing iterative LLM-driven transformations on HTML pages.

## The HTML Workbench Model

### Two Types of Prompts

**System Prompt (top-level):** Defines the rules of engagement. Persistent across all transformations. Sets the LLM's role:

> "You are an HTML transformation agent. You will receive HTML source code and a transformation instruction. Reply with ONLY the transformed HTML. Do not include markdown backtick fences. Do not include any explanation or commentary. Start your response with `<!DOCTYPE html>`."

This is effectively the system prompt in the LLM API call.

**Transformation Prompts (individual):** The specific edit instructions, executed one at a time:

- "Remove all inline styles"
- "Change the title to 'ABC'"
- "Replace the second paragraph with 'This is a surgical test'"
- "Convert all tables to responsive CSS grid layouts"
- "Add dark mode support"
- "Translate all visible text to Portuguese"

Each transformation prompt is a text file stored in the vault. The user selects one and executes it against the current HTML.

### Prompts as Files in the Vault

Transformation prompts should be **editable text files** stored in the vault, not hardcoded in the UI. This means:

- Prompts are `.txt` or `.md` files in a prompts folder
- The user can edit them in the workspace text editor
- The user can add new prompts by creating new files
- The user can organise prompts into subfolders (e.g., `prompts/cleanup/`, `prompts/styling/`, `prompts/translation/`)
- The workspace UI shows a prompt selector — a dropdown or list populated from the prompts folder
- Select a prompt, click execute, the LLM runs the transformation

This makes the workspace a **reusable transformation pipeline**. Define a set of prompts once, apply them to any HTML page.

## Folder-Per-Page Model: Source + Views

### The Structure

Each HTML page being worked on lives in its own folder in the vault. Inside that folder:

```
my-website/
  source.html          <- The original HTML (never modified by transformations)
  view-1.html          <- First transformation output
  view-2.html          <- Second transformation (built on view-1, or fresh from source)
  view-3.html          <- Third transformation
  view-dark-mode.html  <- Named view (renamed from a numbered view)
  view-responsive.html <- Another named view
```

### The Workflow

1. **Load:** Open a folder. The workspace loads `source.html` into the left panel (source preview).
2. **Transform:** Select a prompt, execute against the source (or against a specific view). The LLM returns transformed HTML.
3. **Save as view:** The output is saved as `view-1.html`. The right panel shows the rendered preview.
4. **Iterate:** Run another transformation. Save as `view-2.html`.
5. **Rename and branch:** Rename `view-2.html` to `view-dark-mode.html`. Now it has a meaningful name. Continue transforming from there — saving creates a new view.
6. **Compare:** Click any view in the file list to load it into the preview pane. Click source to go back to the original. Flip between views to compare transformations.

### Save Behaviour

- **Save** writes the current transformation output to the current view filename
- **Rename + Save** writes to a new filename (the renamed one), creating a new view. The old view is preserved.
- The naming convention is `view-{name}.html` where name can be a number or a descriptive string

## LLM Transformation Pipeline (Future)

The individual prompt execution is the MVP. The next step is chaining:

**Step-by-step pipeline:** Define an ordered sequence of prompts (step-1.txt, step-2.txt, step-3.txt). Execute the pipeline: source -> step 1 -> output 1 -> step 2 -> output 2 -> step 3 -> final output. Each intermediate result is saved as a view. The user can inspect any step's output.

This turns the workspace into an ETL-like pipeline for HTML: Load, Transform (multiple steps), Save. Define the pipeline once, apply it to any website.

For now, the one-at-a-time execution is sufficient. The pipeline is the obvious evolution once the manual workflow is solid.

## API Panel: Expand Request and Response Detail

### Current State

The API tab (right panel) already shows a timestamped log of all HTTP calls — both POST requests to `openrouter.ai/api/v1/chat/completions` and GET requests to `send.sgraph.ai/api/vault/read/...`. Each entry shows the timestamp, HTTP status (200), method (POST/GET), URL, and latency in milliseconds. This is useful but not enough for debugging transformations.

### What is Needed

Clicking on any API log entry should expand to show the full request and response detail:

**Request detail:**
- Full URL
- Headers (model, content-type, authorization — mask the API key)
- Body: the full messages array (system prompt + user message with HTML + transformation prompt)

**Response detail:**
- Status code + status text
- Response body: the full LLM response (before HTML extraction — useful for debugging when extraction fails)
- Token usage: prompt tokens, completion tokens, total (Open Router returns this)
- Model used (Open Router may route to a specific model variant)
- Latency breakdown if available

**Open Router awareness:** The display should parse Open Router's response format and surface the metadata it provides (model ID, token counts, generation info). This makes the API panel useful as a cost and performance monitoring tool, not just a log.

### Implementation Note

There is already better code for request/response viewing elsewhere in the codebase that was not copied over when building the workspace. The Developer or Librarian should locate this existing implementation and port it rather than rebuilding from scratch.
