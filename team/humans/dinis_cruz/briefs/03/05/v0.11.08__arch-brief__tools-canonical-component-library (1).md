## The Problem: Redundancy Across Projects

Looking at the current codebase:

**send.sgraph.ai** (`v0.2/v0.2.0/_common/js/`) contains:
- Core JS: `api-client.js`, `crypto.js`, `file-type-detect.js`, `i18n.js`, `markdown-parser.js`
- Components: `send-access-gate`, `send-download`, `send-footer`, `send-header`, `send-locale`, `send-transparency`, `send-upload`
- Plus: `base/`, `vendor/`

**vault.sgraph.ai** (`sgraph_ai_app_send__ui__vault/v0/v0.1.1/_common/js/`) contains:
- Components: `vault-app`, `vault-browser`, `vault-entry`, `vault-file-preview`, `vault-file-properties`, `vault-share`, `vault-shell`, `vault-tree-view`, `vault-upload`, `vault-upload-dropzone`
- Plus: `base/`, `lib/`, `services/`, `shared/`, `api-logger`, `events-viewer`, `messages-panel`, `storage-viewer`

There is almost certainly duplicated logic between these two projects: crypto operations, API client code, file upload handling, file type detection, header/footer UI, internationalisation. A bug fix in one requires a manual port to the other. They drift apart over time.

---

## The Inversion: tools.sgraph.ai Becomes the Source

Instead of tools.sgraph.ai consuming components from send and vault, **invert the dependency**: tools.sgraph.ai becomes the canonical component library, and send.sgraph.ai, vault.sgraph.ai, the workspace, the Chrome extension, and all future projects import from it.

```
BEFORE (current — duplicated, divergent):

send.sgraph.ai          vault.sgraph.ai         workspace
  _common/js/              _common/js/             _common/js/
    crypto.js                crypto.js  (copy!)       crypto.js  (copy!)
    api-client.js            api-client.js (copy!)    api-client.js (copy!)
    send-upload/             vault-upload/ (similar!)  ...
    ...                      ...

AFTER (proposed — single source of truth):

tools.sgraph.ai  (THE canonical source)
  core/
    crypto/            <- One crypto.js, used by everyone
    api-client/        <- One API client, used by everyone
    i18n/              <- One i18n system, used by everyone
  components/
    upload-dropzone/   <- One upload component, styled per-project via CSS
    file-preview/      <- One file preview, used by send and vault
    header/            <- One header, themed per-project
  tools/
    video-splitter/    <- Standalone tool (imports from core/ and components/)
    ssh-keygen/        <- Standalone tool
    llm-client/        <- Standalone tool

send.sgraph.ai         vault.sgraph.ai         workspace
  imports from           imports from             imports from
  tools.sgraph.ai       tools.sgraph.ai          tools.sgraph.ai
  + send-specific UI    + vault-specific UI      + workspace-specific UI
```

**Every project becomes a "mashup"** — a composition of core libraries, shared components, and project-specific UI pages. The project-specific code is the thin layer on top. The heavy lifting (crypto, API, file handling, upload, preview) lives in the shared library.

### The Three Tiers Explained

**Core Libraries (`core/`):** Pure JavaScript modules with zero UI. They do one thing: crypto, API calls, file detection, LLM access, video processing, etc. No DOM manipulation, no CSS, no HTML templates. Any project can import them. These are the foundation that everything else builds on.

**Components (`components/`):** Reusable UI elements that combine a core library with a visual representation. An upload dropzone, a file preview panel, a tree view, an API logger. Each component is a JS file (which may import from core/) plus a CSS file for default styling. Components are themeable — the consuming project can override the CSS to match its own look.

**Tools (`tools/`):** Standalone single-page apps that compose core libraries and components into a complete user experience. A tool is a thin HTML page that imports what it needs and wires it together. Tools are what users interact with on tools.sgraph.ai. They are also examples of how to compose the library.

### How a Tool Is Built

The video splitter tool (`tools/video-splitter/index.html`):

```html
<!DOCTYPE html>
<html>
<head>
  <link rel="stylesheet" href="/components/upload-dropzone/latest/upload-dropzone.css">
  <link rel="stylesheet" href="/components/header/latest/header.css">
  <link rel="stylesheet" href="/components/footer/latest/footer.css">
  <link rel="stylesheet" href="./video-splitter.css">  <!-- tool-specific styles -->
</head>
<body>
  <script type="module">
    import { SgHeader } from '/components/header/latest/header.js';
    import { SgFooter } from '/components/footer/latest/footer.js';
    import { UploadDropzone } from '/components/upload-dropzone/latest/upload-dropzone.js';
    import { splitVideo } from '/core/video/latest/sg-video.js';

    // Wire it all together — tool-specific logic only
  </script>
</body>
</html>
```

### How send.sgraph.ai Imports

send.sgraph.ai stops bundling its own crypto, API client, and upload components. Instead:

```html
<script type="module">
  import { encrypt, decrypt } from 'https://tools.sgraph.ai/core/crypto/v1.2.0/crypto.js';
  import { ApiClient } from 'https://tools.sgraph.ai/core/api-client/v1.0.0/api-client.js';
  import { UploadDropzone } from 'https://tools.sgraph.ai/components/upload-dropzone/v1.0.0/upload-dropzone.js';
  import { FilePreview } from 'https://tools.sgraph.ai/components/file-preview/v1.0.0/file-preview.js';
</script>
```

Note: send.sgraph.ai pins to specific versions (not `latest`). This means a new version of crypto.js can be published and tested before send.sgraph.ai adopts it. The upgrade is an explicit decision: change `v1.2.0` to `v1.3.0` in the imports.

The vault does the same. The workspace does the same. The Chrome extension does the same. One source, many consumers.

## Versioning and IFD Per Module

### Every Module Has Its Own Version

Each core library and each component is independently versioned. A change to `crypto.js` bumps `core/crypto/` from v1.0.0 to v1.1.0. This does not affect `core/api-client/` or `components/upload-dropzone/` — they stay at their current versions.

### Folder-Based Versioning (No Package Manager)

Versions are folders on S3. The `latest` folder is a copy of (or redirect to) the most recent stable version. This is the versioning model already in use across the SGraph projects — proven and understood.

```
core/crypto/
  v1.0.0/crypto.js      <- First release
  v1.1.0/crypto.js      <- Bug fix
  v1.2.0/crypto.js      <- New feature
  latest/crypto.js       <- Copy of v1.2.0 (or S3 redirect)
```

### Pinned vs Latest

| Consumer | Version Strategy | Why |
|---|---|---|
| **Production projects** (send, vault) | Pin to specific version | Stability. Upgrade is a deliberate, tested change. |
| **Tools on tools.sgraph.ai** | Use `latest` | Tools are our own pages — we control them, we can fix quickly. |
| **Development / staging** | Use `latest` | Always testing against the newest version. |
| **Third-party imports** | Pin to specific version | They need stability. Breaking changes would break their apps. |

### IFD Pipeline Per Module

Each module has its own CI/CD trigger:

```yaml
# When core/crypto/ changes:
on:
  push:
    paths:
      - 'core/crypto/**'
# -> Run crypto unit tests
# -> Deploy to core/crypto/v{new_version}/
# -> Update core/crypto/latest/
# -> Do NOT rebuild or redeploy anything else
```

Changes to a specific module only trigger that module's pipeline. This is the same principle as the CI/CD scoping from the data room enhancements brief, applied at the module level.

## Migration Path

### Phase 1: Set Up the Structure

1. Create the `sgraph_ai__tools` repo with the three-tier folder structure
2. Copy (not move yet) the existing shared modules from send and vault into the appropriate `core/` and `components/` locations
3. Deploy tools.sgraph.ai with the initial modules
4. Verify all modules are importable via CDN URLs

### Phase 2: Tools Import from tools.sgraph.ai

5. Build the MVP tools (video splitter, LLM client, audio transcription) importing from `core/` and `components/`
6. Verify the composition model works: a tool is a thin HTML page that imports and wires modules

### Phase 3: send.sgraph.ai Migrates

7. Update send.sgraph.ai to import core libraries from tools.sgraph.ai instead of its local `_common/js/` copies
8. Start with one module (e.g., crypto.js) — pin to a specific version
9. Verify send.sgraph.ai works with the remote import
10. Migrate remaining modules one by one
11. Once all shared code is imported from tools.sgraph.ai, delete the local copies from send's `_common/js/`

### Phase 4: vault.sgraph.ai Migrates

12. Same process as send — migrate imports one module at a time
13. Identify vault-specific components that should stay in the vault repo vs move to `components/`
14. Components that are reusable (tree-view, file-preview, upload-dropzone) move to tools.sgraph.ai
15. Components that are vault-only (vault-shell, vault-app) stay in the vault repo but import core from tools.sgraph.ai

### Phase 5: Full Ecosystem

16. Workspace imports from tools.sgraph.ai
17. Chrome extension imports from tools.sgraph.ai (for the crypto module at minimum)
18. All `_common/js/` local copies are eliminated
19. Single source of truth achieved

## Performance: CDN-Served Imports

### Why This Works

All `*.sgraph.ai` sites are served from S3 + CloudFront. CloudFront is a CDN. Importing a module from `tools.sgraph.ai/core/crypto/v1.2.0/crypto.js` means:

- First load: fetched from S3 via CloudFront edge location (~50-100ms)
- Subsequent loads: served from CloudFront cache (~5-20ms)
- Browser cache: after first visit, the pinned version URL is cached by the browser indefinitely (the URL never changes for a pinned version, so it can be cached aggressively)

### Cache Headers

For pinned versions (e.g., `/core/crypto/v1.2.0/crypto.js`): set `Cache-Control: public, max-age=31536000, immutable`. The content at this URL will never change (it's a specific version). Cache forever.

For `latest` (e.g., `/core/crypto/latest/crypto.js`): set `Cache-Control: public, max-age=300` (5 minutes). This allows `latest` to update while still being cached for reasonable periods.

### Cross-Origin

All `*.sgraph.ai` subdomains share the same parent domain. If CORS becomes an issue, set `Access-Control-Allow-Origin: *.sgraph.ai` on the tools.sgraph.ai CloudFront distribution. For third-party imports, either allow `*` (public CDN model) or restrict to approved domains.

## Component Portability

### Future: Components as Their Own Repos

The architecture is designed so that any core library or component can be extracted into its own repo/project in the future if it grows large enough or needs its own team:

```
Today:     tools.sgraph.ai/core/crypto/     (folder in the tools repo)
Future:    crypto.sgraph.ai                  (its own repo, its own team)
           OR: npm package @sgraph/crypto    (published to npm)
```

The consuming projects don't change their import URLs — the S3 path stays the same regardless of whether the source is a folder in a monorepo or a separate repo that deploys to the same S3 path.

This is the power of CDN-served imports: the consumer doesn't know or care where the source lives. They just import from a URL. The URL is the interface. What's behind it is an implementation detail.
