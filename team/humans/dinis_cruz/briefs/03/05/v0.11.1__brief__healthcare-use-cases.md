## The Core Advantage in Healthcare

Healthcare workflows involve sensitive personal data at every step: assessment forms, medical records, clinical trial data, research results. The current state is broken — PDFs downloaded to unencrypted disks, results emailed to inboxes, form data stored on servers where admins and operators can see it.

SG/Send's distributed, zero-knowledge architecture solves a whole category of GDPR and data logistics problems immediately: the server never sees plaintext, the data is encrypted at the point of creation (in the user's browser), and access is controlled by cryptographic keys, not by trust in the server operator.

## Use Case 1: Online Assessment Form → Encrypted Vault

### The Current Workflow (Broken)

1. User fills an online form (health assessment, cybersecurity questionnaire, any sensitive survey)
2. Server processes the form and generates a result (PDF report, score, assessment)
3. User downloads the PDF → saved to their unencrypted disk
4. Or: user receives the result by email → sits in their inbox, visible to email provider, IT admins, etc.

Both outcomes are insecure. The PDF on disk has no access control. The email is plaintext in transit and at rest.

### The SG/Send Workflow

1. User fills the form in their browser
2. The browser generates the assessment/report (client-side, or received from the server)
3. **Instead of downloading or emailing:** the browser imports the sg-send.js library and creates a vault tied to this browser session
4. The report is encrypted and uploaded to the vault — the user's browser holds the only key
5. The user now has an encrypted vault containing their assessment

**Key property:** At no point does the server (or the form operator) need to store the user's result in plaintext. The user's browser does the encryption. The vault is tied to that browser on that device.

### Sharing with a Clinician

The user wants to share the assessment with their doctor — and only their doctor. Not the server admin, not the form operator, not the IT team.

The flow:
1. User gives the clinician a **read-only key** for the vault (or for a specific file in the vault)
2. The read key can be targeted — encrypted with the clinician's public key, so only they can use it
3. The clinician opens the vault, sees the assessment, cannot modify it
4. The user retains the write key — they control the vault, they can revoke access

**The server never sees the content.** The form operator never sees the result. Only the user and their chosen clinician have access. This is GDPR by architecture, not by policy.

### Anonymous Assessments

For anonymous online assessments (e.g., mental health screening, substance abuse questionnaires):

- The server does not know who the user is (no login required)
- The assessment runs client-side
- A passphrase is generated and shown to the user (or sent via WhatsApp/email)
- The result is encrypted into a vault
- Only the passphrase holder can access it

Even though the answers may be sensitive, the server never sees them, and the user is never identified. The vault exists as an encrypted blob — the user's passphrase is the only way in.

## Use Case 2: Clinical Trials — Multi-Vault Architecture

### The Problem

Clinical trials collect sensitive data from many participants. The data needs to be:
- **Private:** Each participant's data visible only to authorised researchers
- **Anonymisable:** Participants can control what gets shared
- **Provable:** Researchers need to trust the data is real, not fabricated
- **Revocable:** Participants can withdraw consent (GDPR Article 7)

Current solutions require trusting the trial operator with all data. If the operator is compromised, all participant data is exposed.

### The Multi-Vault Pattern

Each participant gets their own vault. The trial organiser creates the vault structure, but only the participant holds the write key.

```
Trial Organiser
├── Participant Vault A  (write key: Participant A only)
│   ├── raw-data.json         ← Full responses (only Participant A can see)
│   ├── anonymised-data.json  ← Anonymised version (shared with researchers)
│   └── consent-record.json   ← What the participant agreed to share
├── Participant Vault B  (write key: Participant B only)
│   ├── raw-data.json
│   ├── anonymised-data.json
│   └── consent-record.json
└── Research Vault  (write key: Trial Organiser)
    ├── aggregated-anonymised-data.json  ← Combined anonymised data
    └── trial-metadata.json
```

**How it works:**

1. Participant fills the form → data goes into their personal vault (encrypted, only they can read the raw version)
2. Participant reviews an anonymised version → confirms what they're comfortable sharing → that version is saved separately
3. The anonymised version is shared with the research vault using a read key
4. The trial organiser can see anonymised data from all participants, but never the raw data
5. If a participant withdraws consent, they revoke the read key → their anonymised data becomes inaccessible to the research vault

**The organiser can decrypt all vaults' edit keys** (because they created them), but the architecture makes it possible to set up vaults where even the organiser cannot access raw data — only the anonymised exports the participant explicitly approved.

### User-Controlled Anonymisation

The participant reviews what will be shared before it leaves their vault. This is the same pattern from the interview transcription use case (v0.10.1 brief): the user checks and approves what gets shared. The difference is that here, the checking happens inside an encrypted vault, and the sharing uses cryptographic access control rather than trust.

### Connected Vaults Without Visible Connectivity

Multiple vaults can be logically connected (all part of the same trial) without the connection being visible from any single vault. The trial organiser knows which vaults belong to the trial. Each participant only knows their own vault. An external observer sees unrelated encrypted blobs.

## Use Case 3: Data Provenance and Signed Attestation

### The Problem with Research Data

Most research data is, to put it bluntly, unverifiable. There is no cryptographic proof that the data came from real participants, that it wasn't fabricated, or that it hasn't been tampered with after collection. "Trust us" is the current standard.

### Provenance Through Signing

The vault architecture enables cryptographic provenance:

**At the point of creation:** When a participant submits data, it can be signed with their key (even a temporary, anonymous key). This creates a cryptographic record: "this data was created by the holder of key X at time T."

**Third-party attestation:** The same pattern used for file safety (the scanning pod signs the result of a scan) can be applied here. A trusted third party — a clinic, a hospital, a regulatory body — can countersign the data:

```
Participant submits data
    ↓
Data signed with participant's key (provenance)
    ↓
Data sent to attestation service (encrypted for their key)
    ↓
Attestation service validates (e.g., confirms the participant exists, the data format is valid)
    ↓
Attestation service countersigns
    ↓
Doubly-signed data stored in vault
```

**The result:** Data that carries cryptographic proof of origin AND third-party validation. Not just "we collected this" but "this person submitted it, and this clinic confirmed it."

### Revocable Access with Provenance

When a participant revokes access, the data becomes inaccessible — but the provenance records (the signatures) remain. This means: "we can prove data was collected and validated, even though we can no longer access the data itself." This is important for regulatory compliance — the trial can prove it followed proper procedures even after participants withdraw.

## The Broader Pattern: Distributed Data Logistics

These healthcare use cases illustrate a pattern that applies far beyond healthcare:

| Problem | How SG/Send Solves It |
|---|---|
| Sensitive data created in the browser | Encrypt at point of creation, never send plaintext to server |
| User needs to share with specific person | PKI: encrypt for recipient's public key |
| User needs to control access | Read/write key separation, revocable keys |
| Data needs to be verifiable | Cryptographic signatures at creation and attestation |
| Multiple parties need partial access | Multi-key encryption within a vault |
| Regulatory compliance (GDPR) | Zero-knowledge by architecture, consent records in vault, revocable access |
| Data collected from many sources | Multi-vault pattern: one vault per participant, aggregation via shared keys |

**GDPR solved by architecture, not by policy.** The server cannot access the data, so there is no data breach if the server is compromised. There is no "data processor" in the GDPR sense — the server is just an encrypted blob store. The user is the data controller. Access is cryptographic, not administrative.
