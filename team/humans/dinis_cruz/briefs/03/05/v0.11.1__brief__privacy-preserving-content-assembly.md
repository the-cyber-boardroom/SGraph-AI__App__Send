# Brief: Privacy-Preserving Content Assembly — Client-Side Page Customisation

**version** v0.11.1  
**date** 4 Mar 2026  
**from** Human (project lead)  
**to** Architect (lead), Developer  
**type** Architecture brief — content delivery, privacy, PKI

## The Idea

How can a user customise the content they receive from a website while maintaining anonymity — so that neither the server nor any intermediary can determine what specific content a particular user chose to see?

This builds directly on SG/Send's existing PKI primitives and introduces a pattern where the page is assembled client-side from encrypted fragments, with the server unable to correlate which fragments any particular browser session received.

---

## The Primitives We Already Have

| Primitive | Status | Role in This Pattern |
|---|---|---|
| Browser-based PKI key generation | ✅ In production | Client creates a key pair on page load — "this is who I am" |
| Server public key download | ✅ In production | Client downloads the server's public key — all traffic to the server is encrypted for that key |
| Client public key sent to server | ✅ In production | Server receives client's public key — responses can be encrypted exclusively for that client |
| Zero-knowledge encrypted storage | ✅ In production | Content fragments stored encrypted, server cannot read them |

This goes above SSL. SSL protects the transport. PKI protects the content itself — even if the transport is compromised, only the intended recipient can read the payload.

## The Workflow

### Step 1: Server Sends the Page Structure (No Content)

The server sends down a version of the web page where all content has been replaced with **hash placeholders**. The page has structure (HTML skeleton, layout, navigation) but no actual text, images, or data.

```html
<!-- What the server sends -->
<article>
  <h2 data-hash="a7f2b9c4"><!-- placeholder --></h2>
  <p data-hash="d8e1f3a5"><!-- placeholder --></p>
  <p data-hash="b7c9d2e4"><!-- placeholder --></p>
</article>
<article>
  <h2 data-hash="f6a8b0c3"><!-- placeholder --></h2>
  <p data-hash="d5e7f9a1"><!-- placeholder --></p>
</article>
```

At this point, the client has a page skeleton with hash IDs for every content block — but no actual content. The client knows what slots exist, but not what goes in them.

This is conceptually identical to how the translation system already works: English content stored in HTML, translations loaded from JSON files keyed by hash, swapped in at render time.

### Step 2: Client Sends Preferences (Encrypted)

The client sends a request — encrypted with the server's public key — specifying what content they want:

```json
{
  "criteria": {
    "topics": ["cybersecurity", "privacy"],
    "exclude": ["marketing", "sponsored"],
    "sentiment": "objective_only",
    "language": "pt-BR"
  }
}
```

Or simpler: "I only want articles about topic X" or "exclude negative news" or "translate to Portuguese."

### Step 3: Server Returns Matching Content Fragments (Encrypted for Client)

The server evaluates the criteria against its content database and returns only the content fragments that match, encrypted with the client's public key:

```json
{
  "fragments": {
    "a7f2b9c4": "Encrypted(headline text, client_public_key)",
    "d8e1f3a5": "Encrypted(paragraph text, client_public_key)",
    "f6a8b0c3": "Encrypted(another headline, client_public_key)"
  }
}
```

Fragments that don't match the criteria are simply not sent. The client never sees them.

### Step 4: Client Assembles the Page

A small JavaScript script (sent with the page skeleton, or already cached) decrypts the fragments and inserts them into the corresponding hash slots. Content that wasn't returned (because it didn't match criteria) is removed from the DOM. The page renders with only the content the user requested.

## The Privacy Properties

### What the Server Knows
- That a client requested content with certain criteria
- Which fragments it sent to that client's public key

### What the Server Does NOT Know (with the Two-Service Variant)
If the architecture uses two separate services — one that provides the HTML skeleton (with hash slots) and another that provides the content fragments — and the client uses a different public key for each:

- **Service A** (skeleton provider) knows the page structure was requested but not which content was selected
- **Service B** (content provider) knows which fragments were requested but not which page they were for
- **Neither service** can correlate a browser session to a specific set of content on a specific page

Even without the two-service split, the architecture already separates structure from content, which is a meaningful privacy improvement over server-side personalisation.

### What the Client Controls
- Which content to display (via criteria)
- Which content to exclude (never downloaded, never seen)
- The assembly happens locally — no server-side personalisation needed

## Why This Matters

### Server-Side Personalisation Is the Problem

Today, personalised content is assembled on the server. The server knows exactly what each user sees, when they see it, how long they view it. This is the surveillance capitalism model — personalisation requires the server to know everything about the user.

Client-side assembly inverts this: the server provides fragments, the client assembles. The server does not need to know the final page state. The personalisation happens where it should — on the user's device, under the user's control.

### Performance Advantages

Server-side personalisation is expensive: for every request, the server queries a database, applies personalisation rules, renders a custom page, and returns it. This doesn't cache well.

Client-side assembly: the server sends the same skeleton to everyone (cacheable), the content fragments are keyed by hash (cacheable), and the client does the assembly (fast, local). Only the criteria request is unique per user.

### Translation as the First Use Case

This pattern is already partially implemented in the translation system: English content in HTML, translations loaded from JSON keyed by content hash, swapped in client-side. The leap from "swap language" to "swap content based on user preferences" is an incremental step on the same architecture.

## Future Direction: Meta-Structure Websites

The end state is websites that send a **meta-structure** — a description of what the page contains (structure, content hashes, metadata) — and the client assembles it however they want to see it. Different clients can render the same meta-structure differently: different layouts, different content selections, different languages, different accessibility modes.

This is the beginning of agent-assembled web pages: an AI agent receives the meta-structure, applies the user's preferences, fetches the needed fragments, and presents a customised page — without the server ever knowing what the final result looks like.
