# Dev Brief: Token Optimisation and Prompt Engineering — One-Shot Principle, Context Curation, and Future Diff Responses

**version** v0.11.10  
**date** 5 Mar 2026  
**from** Human (project lead)  
**to** Developer (lead), Architect  
**type** Dev brief — LLM cost optimisation, prompt architecture, workspace evolution  

---

## Context: What We've Learned

The workspace is working. The human is successfully using LLMs to write JavaScript that transforms HTML pages. But watching the token usage (visible in the LAST REQUEST stats panel) reveals a problem: requests are expensive because we're sending too much context.

The Register's index.html is 94.9KB. With script and result included, a single request can hit 43K tokens in, 1.8K tokens out, costing ~$0.15 per request. Ten iterations to get a transformation right = $1.50 just for one page. This doesn't scale to processing dozens of pages, and it's avoidable — most of that context is redundant or unnecessary for the specific question being asked.

---

## The One-Shot Principle

### The Rule

**Every LLM request is a one-shot request. No history. No accumulated context.** Each request stands alone. There is no conversation thread where message #5 builds on the context of messages #1-4.

This is a deliberate, principled decision:

**Why people overspend on LLMs:** It's easy to use the "conversation" pattern — each request includes all previous messages. This feels natural (it's how ChatGPT works) but it's catastrophically expensive on API calls. Request #10 in a conversation carries the full weight of requests #1-9. A conversation that starts at 5K tokens is at 50K tokens by the 10th message. Most of that context is irrelevant to the current question.

**Why one-shot is better for our use case:** We're writing transformation code. The LLM needs to see the HTML structure and the current script to make improvements. It does NOT need to see the 8 previous prompts where we asked it to try different approaches. The current state is what matters, not the history of how we got here.

**Why most people don't understand this:** The convenience of conversation mode hides the cost. People don't realise that "continue the conversation" means re-sending everything. The workspace makes this explicit: you choose exactly what to include. Nothing is implicit. Nothing accumulates silently.

### What This Means for the UI

The current INCLUDE checkboxes (Src, Data, Script, Result, Console) already implement this — the user explicitly chooses what context to include. This is the right model. The enhancement is making it even more visible what's being sent and how much it costs:

- Show token count estimate BEFORE sending ("This request will be ~25K tokens / ~$0.08")
- Show which attachments are contributing how many tokens ("Source: 18K tokens, Script: 2K tokens, Result: 15K tokens")
- Make it easy to exclude heavy attachments ("Do you really need the full result? Your question is about the script.")

## Context Curation: Send Less, Get More

### The Problem

Sending the full 94KB source HTML for every request is wasteful. After the first request where the LLM sees the page structure, subsequent requests often need only the script and maybe a structural skeleton of the HTML — not the full content.

### Strategy 1: Skeleton Mode

Instead of sending the full HTML, send a structural skeleton: the DOM tree with tag names and classes, but content replaced with placeholders.

```html
<!-- Full HTML: 94KB, ~18K tokens -->
<article class="story">
  <h4><a href="/2026/03/04/google_chrome_updates/">Google feels the need for security speed...</a></h4>
  <div class="body"><p>Retains eight-weekly Extended Stable releases but warns...</p></div>
</article>

<!-- Skeleton: ~5KB, ~1K tokens -->
<article class="story">
  <h4><a href="...">[headline]</a></h4>
  <div class="body"><p>[body text]</p></div>
</article>
```

The skeleton preserves the structure the LLM needs to write selectors and transformations, but drops the content (which is irrelevant to the code being written). This could reduce source context from 18K tokens to 2-3K tokens.

**Implementation:** A "Skeleton" button on the Source panel that generates a structural view. This skeleton can be saved as a file in the vault and included as context instead of the full HTML. The skeleton is itself a JavaScript transformation — we can use the workspace to build it.

### Strategy 2: Progressive Context Reduction

Use the workspace's own transformation pipeline to reduce context for subsequent LLM requests:

1. **First request:** Send full HTML + prompt → LLM writes initial script
2. **Second request:** Send skeleton + script + error output → LLM fixes the script
3. **Third request:** Send script only + screenshot of result → LLM refines visually
4. **Subsequent:** Send only the script + specific question → LLM makes targeted changes

Each iteration sends less context because the LLM has already encoded its understanding of the page structure into the script. The script IS the compressed representation of the LLM's understanding.

### Strategy 3: Pre-Computed Page Summaries

For a specific website, create a "site description" file — a compact document (500-1000 tokens) that describes the page structure, the key CSS classes, the DOM patterns. This file is included as context instead of the raw HTML.

```markdown
## The Register - Article Listing Page
- Articles are in `<article class="story">` elements
- Each article has: headline in `h4 > a`, body in `.body > p`, author in `.byline`
- Articles are arranged in rows of 4 in a CSS grid (`.article_columns`)
- Categories shown as labels in `.topic-tag`
- Ads are in `.ad-banner`, `.ad-leaderboard`, `.gpt-ad` elements
- Tracking scripts: Google Analytics, Permutive, Lotame
```

This is 150 tokens vs 18,000 tokens. For most transformation questions, this summary plus the current script provides enough context.

### The Self-Improving Loop

The workspace can be used to create the context reduction tools for itself: write a transformation that extracts a skeleton from HTML, write another that generates a site summary, save them as reusable scripts. The system improves its own efficiency.

## Multi-Prompt Composition: Fake Conversations

### The Idea

Although every request is one-shot (no real history), the `messages` array sent to the LLM can contain a **constructed conversation** — multiple system/user/assistant messages that the user hand-crafts to provide better context.

This is not conversation history. This is **prompt engineering via simulated dialogue.**

### Why This Might Work

LLMs are trained on conversations. They respond differently to a single long instruction vs. a conversation that builds context step by step. A "fake conversation" can:

- **Set up context progressively:** System prompt establishes the role → User says "here's the HTML structure" → Assistant says "I understand, it uses a grid layout with article cards" → User says "now write a script to remove negative articles" → This final response is what we actually want
- **Pre-seed understanding:** The "assistant" messages in the fake conversation tell the LLM what it "already knows" — effectively injecting knowledge without the LLM having to discover it
- **Guide the response format:** An assistant message like "Here's the script structure I'll use: [skeleton]" primes the LLM to follow that structure in its actual response

### The Messages Array

```json
{
  "messages": [
    { "role": "system", "content": "You are an expert web developer..." },
    { "role": "user", "content": "Here's the site description: [site summary]" },
    { "role": "assistant", "content": "I understand. The page uses a CSS grid layout with .story articles. Key selectors: .article_columns for rows, article.story for items." },
    { "role": "user", "content": "Here's the current transformation script: [script]" },
    { "role": "assistant", "content": "I see the script removes articles by checking sentiment. It's working but leaving white gaps in the grid." },
    { "role": "user", "content": "Can you fix the grid gaps? Here's a screenshot of the current result." }
  ]
}
```

The first two user/assistant exchanges are fabricated — the user wrote both sides. Only the final user message is the real question. But the LLM sees a conversation where "it" already understood the page structure and the current script, and now needs to address a specific issue.

### Hypothesis

In some cases, this constructed conversation will produce significantly better results than a single massive prompt, because:

- The LLM's attention is focused on the final question (not spread across a 20K token wall of text)
- The "assistant" messages pre-empt misunderstandings ("I see the script removes articles by checking sentiment" — the LLM won't waste tokens re-explaining what the script does)
- The progressive context mimics how a human expert would work through a problem

### UI for Multi-Prompt Composition

The current UI has a system prompt and a user prompt. The enhancement:

- **Add message slots:** The user can add multiple user/assistant message pairs before the final user message
- **Template messages:** Common setups (site description → acknowledgment, script summary → acknowledgment) saved as reusable templates
- **Visual builder:** See the full messages array before sending. Edit any message. Reorder. Remove. The user builds the array like building a playlist.
- **Toggle view:** Switch between "simple mode" (single system + user prompt, current UI) and "composer mode" (full messages array builder)

### Experimentation Plan

This is an experiment. We don't know if fake conversations are better than single prompts for our use cases. The plan:

1. Build the multi-message UI
2. Test the same request with: single prompt vs. fake conversation with site description vs. fake conversation with progressive context
3. Compare: quality of generated code, token usage, cost, number of iterations needed
4. If fake conversations win for specific patterns, save those patterns as templates

## Raw Request Inspector

### The Need

The user needs to see exactly what's being sent to the LLM — the fully assembled `messages` array, including all the context that was auto-appended from the Include checkboxes. This exists partially in the LLM Debug panel (component #18), but needs enhancement.

### What's Needed

A "Request Inspector" view that shows:

1. **The raw request body** — exactly what will be sent to Open Router (JSON, formatted)
2. **Editable** — the user can modify the raw request before sending (add a message, change content, adjust parameters)
3. **Token count per message** — see which messages are consuming the most tokens
4. **Diff from previous request** — highlight what changed between this request and the last one
5. **Send from inspector** — execute the modified raw request directly
6. **Round-trip:** Changes in the inspector reflect back to the UI elements (edit the system prompt in the inspector → it updates in the system prompt panel), and vice versa

This turns the workspace into a proper LLM API development environment — the user can experiment at the prompt level AND at the raw API level.

## Future: Diff-Based Responses (Mapped, Not Implemented)

### The Vision

Currently, when the LLM modifies a 1000-line script, it returns all 1000 lines (or 900 lines with changes). The response tokens are high, the response is slow, and reviewing the changes requires manually diffing old vs new.

**What if the LLM returned only the changes?**

```json
{
  "changes": [
    { "type": "replace", "line": 45, "old": "if (sentiment < 0)", "new": "if (sentiment < -0.3)" },
    { "type": "insert", "after_line": 67, "content": "// Redistribute articles to fill gaps\nredistributeGrid();" },
    { "type": "delete", "line": 120 }
  ]
}
```

10-15 lines instead of 1000. Faster response. Lower cost. And the changes are surgical — reviewable like a git diff before applying.

### Why This Might Work

This is how Claude Code works internally — it uses `str_replace`-style edits rather than regenerating entire files. The LLM already thinks in diffs internally; the question is whether we can reliably prompt it to output diffs in a structured format.

### Why This Is Mapped, Not Implemented

- Reliability is unproven for our use case (code transformation scripts, not just text)
- Applying diffs to JavaScript requires exact line matching (fragile)
- A malformed diff could corrupt the script
- Need to test across multiple models to see which can reliably produce structured diffs

### When to Explore

Once the multi-prompt composition is working and we have good data on token usage patterns. If we find that response tokens are the dominant cost (not input tokens), diff-based responses become a high priority.
