## The Concept: tools.sgraph.ai

A collection of standalone, browser-based utility tools. Each tool is a single-page app — pure HTML + CSS + vanilla JS, no frameworks, no server involvement. Everything runs client-side. Zero data leaves the browser.

### Why This Matters

**Traffic acquisition:** Each tool is a standalone landing page that ranks for specific search queries ("browser video splitter," "online SSH key generator," "browser file converter"). Users discover SG/Send through the tools, not the other way around.

**Trust building:** Every tool demonstrates the same principle as SG/Send — "your data stays in your browser, nothing goes to a server." Users experience zero-knowledge client-side processing before they even know about encrypted file transfer.

**Technology demonstration:** The tools showcase the same browser capabilities that power SG/Send: WebCrypto, WASM, blob handling, client-side file processing. Each tool is proof that serious computation can happen entirely in the browser.

**SEO and content marketing:** Each tool generates organic search traffic. "Free online video splitter no upload" is a high-intent search query. The user lands on tools.sgraph.ai, uses the tool, sees the SG/Send branding, and discovers the platform.

### What We Already Have

|---|---|---|

### Deployment

tools.sgraph.ai is another static site in the existing S3 + CloudFront setup. Same CI/CD pattern as send.sgraph.ai, vault.sgraph.ai, extension.sgraph.ai. Each tool is a folder:

```

  ssh-keygen/
```

Each tool is self-contained: one HTML file (or a small set of HTML + JS + CSS), no build step, no dependencies on other tools. A tool can be developed, tested, and deployed independently.

### Shared Elements Across All Tools

Every tool on tools.sgraph.ai shares:

- **Header:** SGraph branding, link back to sgraph.ai, link to tools directory
- **Footer:** "All processing happens in your browser. No data is sent to any server." + link to SG/Send ("Need to share files securely?")
- **Privacy badge:** A visible indicator that the tool is client-side only
- **Consistent styling:** Same colour scheme and typography as the rest of the SGraph family
- **No tracking / no analytics cookies:** Practice what we preach. Use privacy-respecting analytics only (e.g., Plausible, or just CloudFront access logs)

## Tool: Browser Video Splitter

### What It Does

A single-page web app that lets a user split a video file into multiple clips entirely in the browser. No upload. No server. The video never leaves the user's device.

Uses FFmpeg compiled to WebAssembly (ffmpeg.wasm) to perform the actual splitting. This is the same FFmpeg that professionals use on the command line, running in the browser via WASM.

### User Flow

1. **Select a video:** User picks a local MP4 file via file input (or drag-and-drop)
2. **See video info:** App shows filename, duration, file size, and a preview player
3. **Define segments:** User specifies how to split:
   - **Fixed segment length:** "Split every 20 seconds" — app calculates segments automatically
   - **Custom segments:** User adds individual segments with start and end times (mm:ss or hh:mm:ss format)
4. **Split:** Click "Split" — FFmpeg WASM loads (lazy, only when needed), processes each segment
5. **Download clips:** Each resulting clip appears in a list with: segment number, start/end time, duration, file size, a preview player, and a "Download" button
6. **Done:** User downloads what they need. Nothing was uploaded anywhere.

### Technical Architecture

**Pure client-side. No frameworks.**

```
video-splitter/
  index.html        <- Single file containing HTML + CSS + JS (or split into 3 files)
```

**FFmpeg integration:**

```html
<script type="module">
  import { FFmpeg } from 'https://unpkg.com/@ffmpeg/ffmpeg@0.12.10/dist/esm/index.js';
  import { fetchFile } from 'https://unpkg.com/@ffmpeg/util@0.12.1/dist/esm/index.js';
</script>
```

FFmpeg WASM is loaded from CDN. It is lazy-loaded — only fetched after the user picks a video and clicks Split. This avoids a large upfront download for users who are just browsing.

**Splitting command per segment:**

```
-ss {start} -t {duration} -i input.mp4 -c copy out_N.mp4
```

The `-c copy` flag means no re-encoding — the video is split at keyframe boundaries without quality loss. This is fast (seconds, not minutes) because it's just copying bytes, not transcoding.

**Virtual filesystem:** FFmpeg WASM uses a virtual filesystem in memory. The input file is written into the virtual FS, FFmpeg processes it, and the output files are read back as blobs. The real filesystem is never touched (browser sandbox).

**Output:** Each output clip is converted to a Blob URL via `URL.createObjectURL()`. The download button creates an `<a>` element with the blob URL and triggers a download.

### Time Input Parsing

Accept multiple formats and convert to seconds internally:

| Input | Parsed As |
|---|---|
| `30` | 30 seconds |
| `1:30` | 90 seconds |
| `01:30` | 90 seconds |
| `1:02:30` | 3750 seconds |
| `90s` | 90 seconds |
| `1.5m` | 90 seconds |

### Performance and Limits

- **Lazy-load FFmpeg:** Only download the WASM binary (~30MB) when the user actually clicks Split. Show a "Loading FFmpeg..." indicator.
- **File size limit:** Warn at 200MB, hard limit at 500MB (browser memory constraints). Show "For files larger than 500MB, consider using FFmpeg on your computer."
- **Memory management:** After each segment is processed and the blob URL is created, free the virtual FS entry. Don't hold all input + all outputs in memory simultaneously.
- **Progress indicator:** Show which segment is being processed (e.g., "Processing segment 2 of 5") and a percentage based on segments completed.
- **Web Worker (future):** Move FFmpeg execution to a Web Worker so the UI stays responsive during processing. For MVP, main thread is fine — the UI just freezes briefly during each segment.

### The SG/Send Connection

**"Download All as ZIP"** naturally leads to **"Share this ZIP securely via SG/Send."** The user has just split a video into clips. They probably want to send them to someone. One click: package as ZIP, encrypt, upload to SG/Send, get a shareable link.

This is the conversion funnel: free tool → useful result → "want to share this?" → SG/Send. The tool is the top of the funnel.
