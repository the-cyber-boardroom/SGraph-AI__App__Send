# The Power of Primitives: Why Getting the Foundations Right Makes Everything Else Fast

One of my core principles is that a good idea gets better the more you use it, and a bad idea gets worse the more you use it.

What I have been doing for the last 18 months is, in hindsight, a pretty good example of Wardley Maps applied to software development. I have been constantly productising and commoditising components, and some of them have become surprisingly powerful. The best ones became invisible: primitives so solid that I stopped thinking about them and just built on top of them.

In Wardley Maps terms, I have been running something like an ILC (Innovate, Leverage, Commoditise) cycle with myself and my various projects. The same components flow across multiple products. The same patterns get refined across multiple contexts. And I have even organised my 18+ AI agents into three teams that map directly to the Wardley Maps evolution model: Explorers (prototype fast, discover what works), Villagers (productise, test, harden), and Town Planners (commoditise, optimise, scale). Each team has a different relationship with the code and a different definition of "done."

Not all of these primitives worked out. But enough of them compounded into capabilities I did not plan and could not have predicted.

## What I Mean by Primitives

A primitive is a building block so simple that it becomes invisible. You stop thinking about it and start building on top of it. The encryption pipeline in SG/Send is a primitive: encrypt in the browser, send the ciphertext, key stays with the user. That is it. One idea.

From that single primitive, we went from browser-based file transfer to desktop apps, to mobile apps, to encrypted vaults, to a CLI, to workspaces, to what is now essentially a PKI-signed and encrypted Git. Each step was possible because the previous step did not add complexity. It added capability.

The fact that I was able to build a PKI-signed and encrypted version control system this fast is, for me, the proof that the primitives are right. We are finishing off something that solves multi-user collaboration on documents and content with end-to-end encryption, running in the browser, on the CLI, on mobile, everywhere. That is not a small thing. But it felt small to build because the foundations were already there.

## The Primitives That Compound

**Client-side encryption.** No server-side state for user data. The server stores ciphertext and has no idea what is inside. This is not just a security decision. It is an architecture decision that eliminates a large category of complexity. Fewer access control lists to manage. Less data classification. Minimal key management on the server. Reduced compliance burden for stored data. The server is deliberately minimal, and that simplicity ripples through everything else.

**Open source.** The repos are public. The components are pullable. This sounds like a values statement, but it is a development velocity decision. Claude Code does not currently support good authentication for private submodules. If the repos were private, the agentic workflow would be significantly harder, maybe impractical. Because they are open, agent sessions can pull repos, read code, and produce briefs that span multiple codebases. The agents can consume the information because the information is accessible.

When I needed to spin up a local copy of the main send web service for integration testing during the CLI's Villager phase, it was trivial. Pull the repo, run it. No tokens to configure, no VPN to connect, no access requests to file. These little things feel minor. They are not. They are the degrees of inflection that make the difference between "this works" and "this is too complex to maintain."

**Zero server-side state.** The server does not know what files exist in a vault. It does not know who owns a branch. It does not know which objects are blobs versus trees versus commits. This means most new features are client-side features. The server rarely needs to change. The CLI got branches, commits, and merges, and the server needed very few updates. The branch model was largely implemented without touching the backend. That is the power of a minimal, opaque server.

**Constant refactoring.** I try not to protect past code. I try not to worry about proprietary lock-ins or make decisions based on sunk cost. If a component needs to be restructured, it gets restructured. The Explorer builds fast and messy. The Villager cleans up and tests. Then the Explorer builds again on a better foundation. Each cycle tends to make the next cycle faster because the codebase is cleaner, not more burdened.

**Simple components with high reusability.** Web Components with Shadow DOM. They tend to do one thing well. They compose. They try not to depend on each other's internals. The dependency inversion pattern (tools.sgraph.ai is canonical, other projects import from it) means adding a component to a new context is usually straightforward. Drop it in. It works.

## Why the Agents Thrive on This

The agentic development workflow compounds on these primitives. The agents are effective precisely because the system is simple. There are fewer moving parts. The architecture is understandable from a brief. The codebases are open and readable. The components are independent.

When I ask the Architect agent to design a branch model, it can read the CLI code, the send server code, and the vault spec in one session. When I ask the Developer agent to implement it, the implementation is client-side only. When I ask the QA agent to test it, it can spin up a local server instance in seconds.

The primitives I got right tend to make the agents more effective. The agents being effective lets me build faster. Building faster lets me iterate on the primitives more. It compounds.

## The Encrypted Git Moment

We are essentially finishing a PKI-signed and encrypted Git. Content-addressed storage, tree structures, commits, branches, merges, signed commits, remotes. All encrypted. Running in the browser, on the CLI, on mobile.

That is not a thing I set out to build 18 months ago. It is where the primitives led. Each one was simple. Each one solved the immediate problem. And each one created the foundation for the next thing that I did not know I would need.

This is the commoditisation cycle in practice. The encryption pipeline was commoditised. Vaults were built on it. The CLI was built on vaults. The branch model was built on the CLI. Each layer became a primitive for the next. And because the ILC pattern means the same components flow across projects, each improvement compounds across everything I am building.

The Explorers discover the next capability. The Villagers harden it into a reliable primitive. The Town Planners make sure it scales and integrates. And then the Explorers build on top of it again. The PKI-signed encrypted Git moment is not a destination. It is the current layer. The next layer is already emerging, and it will feel easy to build, because the foundations are already there.

Everything else is vanilla. And that is the point. When the primitives are right, the hard things start to feel approachable.
