## 1. Conductor Summary — Human Feedback Integration

The human's comments sharpened several key decisions. Here is the updated picture:

| Topic | Original Team Position | Human Correction | Updated Direction |
|-------|----------------------|------------------|-------------------|
| Step 2 destination | Single page with card choices | Each option should lead to a **dedicated page** optimised for that use case | Build separate download, preview, zip-browse, folder-view, gallery pages |
| Step 3 backend | Assumed simple-token needs backend | **No backend needed.** The simple token is a client-side PBKDF2-derived key (like the key-generator tool) | Reusable component: `word-key → AES-256 key` via PBKDF2 |
| File size limit | Check presigned path configuration | Already supported via S3 presigned URLs; the 5MB is a **JS-enforced** UI limit only | Remove the JS limit, verify presigned URL has no cap |
| Step 1 buttons | Keep Browse File / Browse Folder | Buttons should exist (mobile needs them) but not dominate the drop zone | Buttons smaller / below the primary "Drop" message |
| Step 3 default | Default to "link with key" (Mode 2) | **Default to simple token** (Mode 1) — friendliest UX | Simple token first, with copy link + change token + email options |
| Upload progress | Just a progress bar | **Use the upload time to show trust-building messages** — privacy policy, tips, agent quotes | Carousel / rotating messages during upload |
| Demo packs | On the website | Also link from the **send page itself**, since users land there directly | Demo section or link on the upload page |
| Power user mode | Mentioned as future | **Store preferences in localStorage** for repeat senders | Design state machine to support "quick send" collapse |

## 2. Agent Responses to Human Feedback

### 🏗️ Architect — Updated Response

**On Step 2 being separate pages:** Agreed. This is cleaner architecturally. Each download experience (direct download, PDF viewer, zip explorer, folder browser, gallery) is already a distinct component with different JS dependencies. Routing to separate pages means:

- No bloated single page loading all viewers
- Each page can have its own IFD versioning (we already do this — the zip explorer is v0.2.2)
- Deep-linkable: `send.sgraph.ai/{lang}/download/{id}`, `send.sgraph.ai/{lang}/view/{id}`, `send.sgraph.ai/{lang}/browse/{id}`, `send.sgraph.ai/{lang}/gallery/{id}`
- Language-prefixed: all new pages follow the existing i18n pattern — language is always the first path segment (e.g. `/en-gb/d/coal-render-5512`)

The upload page's Step 2 becomes a **routing decision** that constructs the correct URL pattern, preserving the current locale.

**Language support requirement (all new pages):** Every new route must be locale-prefixed, consistent with the existing `/{lang}/` pattern. The 17 supported locales apply. The simple-token page, download page, view page, browse page, and gallery page all follow `/{lang}/{page-type}/{id-or-token}`. The upload page itself already lives at `/{lang}/` (e.g. `/en-gb/`). This is non-negotiable — we do not create English-only pages.

**On simple-token being client-side:** This is the right call. The tools repo already has the full pipeline:

```
sg-pbkdf2.js:  "coal-render-5512"  →  PBKDF2 (600K iterations, SHA-256, salt: "sgraph-send-v1")  →  AES-256 key
sg-key-generator.js:  UI for creating word-based keys with entropy display
sg-key-input.js:  UI for entering word-based keys on the download side
```

The conversion is deterministic: same words always produce the same AES key. No server involvement. The reusable component is already built — it just needs integration into the send flow.

**On the 10GB limit:** Verified. The presigned URL initiation at `/presigned/initiate` does not enforce a size cap in the request. The JS limit is in `SendUpload`:

```javascript
static MAX_FILE_SIZE_DIRECT    = 5 * 1024 * 1024;     // 5MB for Lambda direct
static MAX_FILE_SIZE_PRESIGNED = 1 * 1024 * 1024 * 1024; // 1GB presigned
```

We need to:
1. Change `MAX_FILE_SIZE_PRESIGNED` to `10 * 1024 * 1024 * 1024` (10GB)
2. Ensure the capabilities check routes anything >4MB to presigned path (it already does)
3. Update the UI to show "Maximum file size: 10 GB" instead of "5.0 MB"

### 🎨 Designer — Updated Response

**On Step 1 — single focus with mobile-conscious buttons:**

The drop zone should dominate. The Browse buttons are necessary for mobile but should be subordinate — placed below the primary "Drop a file or folder here" message, styled as secondary/ghost buttons. The current design has them centred and prominent with full borders; they should be more like text links with subtle styling.

The File/Text toggle should be removed from Step 1 entirely. Text mode can be a separate entry point or a small link: "Or encrypt text instead →"

**On Step 2 — "Send as..." framing:**

Each option card should show a **mini preview** of what the recipient will see. We have real screenshots now (the v0.16.4 debrief shows the zip explorer working). This is our strongest selling point — show, don't describe:

- "Send as download" → icon of a download button
- "Let them browse the folder" → mini screenshot of the two-column zip explorer
- "Show as photo gallery" → grid of thumbnail placeholders
- "Let them view it" → document preview icon

**On Step 3 — simple token as default:**

The key-generator screenshot (pic1) shows the full geek mode. For the upload flow, we need a **dramatically simplified** version:

- Show the generated key prominently: `coal-render-5512`
- A "shuffle" button to regenerate
- An "edit" option that expands into the word picker (collapsed by default)
- The full link preview below: `send.sgraph.ai/{lang}/d/coal-render-5512`
- Copy / Email / QR actions

The geek mode (derived AES key, KDF details, entropy) stays hidden unless the user actively wants it.

**On upload progress messages:**

This is a fantastic opportunity. The progress bar is dead time — typically 2–30 seconds for most uploads, longer for big files. We should fill it with trust-building content:

- Privacy policy statements (rotating, one at a time)
- "Zero cookies. Zero tracking. Verify: open DevTools → Application → Cookies"
- "Your file is encrypted with AES-256-GCM. The key never leaves your device."
- "Even we can't read what you're uploading. That's the point."
- Agent tips: "🏔️ Sherpa tip: Share the code by voice, the link by text — different channels, maximum security"

### 🏔️ Sherpa — Updated Response

**On "encrypting in your browser" messaging:**

During Step 1, after the file is dropped and encryption begins, the message should be:

> 🔒 Encrypting in your browser...
> Your file never leaves your device unencrypted. Not now, not ever.

After encryption completes:

> ✅ Encrypted and ready
> Your file is locked with AES-256-GCM encryption.
> Only someone with your key can open it.
> **Choose how to share it →**

**On cookies and privacy visibility:**

I'd add a persistent but unobtrusive trust badge on the upload page — something like a small shield icon with "Zero cookies · Zero tracking · Open source" that expands on click to show the six-sentence policy. This is always visible but never in the way.

**On demo packs on the send page:**

Since users land on `send.sgraph.ai` directly (from download links, bookmarks, word of mouth), the send page should have a "Try it first" section. Not the full test files we have now, but real demo packs:

- "📁 See a folder shared with SG/Send →" (opens the zip explorer with a real demo)
- "📄 See a PDF shared with SG/Send →" (opens the PDF viewer)
- "🖼️ See a gallery shared with SG/Send →" (opens the gallery view)

These use publicly-leaked keys so anyone can try the receiving experience without uploading anything.

**On power user / quick mode:**

For v1, just default to previous choices. localStorage stores:
- `sg_preferred_sharing_mode`: 'token' | 'link' | 'split'
- `sg_preferred_delivery`: 'download' | 'browse' | 'view' | 'gallery'
- `sg_last_token_style`: the word pattern they last used

When these exist, Steps 2 and 3 show the previous choice pre-selected with a "Change" link. For true power users who send frequently, we could later add a "Quick send" toggle that collapses to: Drop → Uploading → Here's your link.

## 10. Decisions (Confirmed by Human — 16 March 2026)

| # | Question | Decision | Rationale |
|---|----------|----------|-----------|
| 1 | Should the upload page URL change during steps? | **No.** Single page, state transitions. URL stays as `/{lang}/`. | We don't persist step state; URL changes would break back-button and refresh. |
| 2 | Where should the reusable components live? | **In the Send repo first.** Copied from Tools repo into Send. Once mature, moved back to Tools repo as shared components. | Avoids cross-repo dependency during development. Maturity-gated promotion. |
| 3 | Should `/d/{token}` be a new page or extend existing download page? | **New page.** Route: `/{lang}/d/{token}`. Language-prefixed. Contains `sg-key-input` component → PBKDF2 → redirect to correct viewer. | Clean separation. All new pages are language-prefixed — no English-only routes. |
| 4 | How visible should the "try it first" demo section be? | **Pretty visible.** Below the drop zone, always shown, not collapsed. | Users land on the send page directly and need to see the receiving experience before committing to upload. |
| 5 | Upload progress carousel: timer or progress-based? | **Timer-based (rotate every ~4 seconds) with manual navigation.** User can click through messages manually, which pauses the auto-rotation. | Decouples message pacing from upload speed. Works for both fast (2s) and slow (minutes) uploads. |
