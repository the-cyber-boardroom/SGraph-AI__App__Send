# Upload UX Redesign: Three Steps, Multiple Sharing Modes, and Pricing Update

**version** v0.16.10
**date** 16 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Designer, Sherpa, Ambassador
**type** Dev brief

## The Problem

The current upload page is too complicated. Too much on one screen. Users need a clear, guided flow.

## Three-Step Upload Flow

### Step 1: Upload

The user drops a file, folder, zip, or PDF. The file encrypts in the browser. Nothing has been sent yet. The file is ready.

### Step 2: Choose How to Distribute

Once the file is encrypted in the browser, present the distribution options. The options depend on what was uploaded:

| Uploaded | Available Options |
|---|---|
| Single file | Download link, view link |
| Folder | Download link, folder view link, gallery link (if images) |
| PDF | Download link, view link |
| Zip | Download link, folder view link (if we can preview contents) |
| Images | Download link, gallery link, folder view link |

Only show options that make sense. There is no point offering a "view" link for a binary file that cannot be previewed. There is no point offering a "gallery" link for a folder of spreadsheets.

### Step 3: Upload and Get Share Token

The user confirms. The encrypted file is uploaded to the server. The user receives their sharing credentials.

## Three Sharing Modes

After upload, the user chooses how to share:

**Mode 1: Simple token.** A short, predictable, human-friendly token. Easy to type, easy to share by voice or text message. The recipient enters the token on the site to access the file. This could be generated before upload (predictable) so the user can share it in advance.

**Mode 2: Shareable link with decryption key.** A single URL that contains everything: the file reference and the decryption key (in the URL fragment). One click to access. The recipient does not need to enter anything. This is the "just paste this link" mode.

**Mode 3: Separate key and link.** The download link and the decryption key are provided separately. The user sends the link via one channel and the key via another (e.g. link by email, key by WhatsApp). Higher security: compromising one channel does not compromise the file.

All three modes should be presented clearly after upload, with a brief explanation of when to use each one.

## Pricing Update

### Bandwidth Increase

Change the per-purchase allowance from the current 1GB to:

| What 5 GBP Buys | Amount |
|---|---|
| Maximum single file upload | 10 GB |
| Total bandwidth (upload + download) | 20 GB |

This is a significant increase. The rationale: storage and bandwidth are cheap. The value is in the encryption, the sharing workflow, and the vault features. Limiting bandwidth to 1GB makes the product feel restrictive for the folder sharing use case. 10GB upload means a user can genuinely share large folders, which is the launch message.

### Enforcement

Not enforced on the backend yet. The tracking data will be available (we can measure it) but we are not building the enforcement logic now. The pricing page states the limits. The principle is to be generous and not punish early users. Enforcement comes later when usage patterns are clear.

The "No Surprises" principle still applies: users see what they are buying. They just will not hit an artificial wall at 1GB.

## Demo Packs on the Website

Create public demo packs that demonstrate the different sharing modes:

- A folder with mixed file types (documents, images, PDFs) showing the folder view
- A gallery demo (images only) showing the gallery view
- A single large file showing the download experience
- An LLM-generated report showing the view experience

Each demo pack has a publicly leaked decryption key. Users can try the full experience without signing up.
