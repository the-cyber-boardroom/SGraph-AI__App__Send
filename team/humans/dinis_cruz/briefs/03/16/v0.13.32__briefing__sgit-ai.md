## 1. What sgit Is

**sgit** is Git with integrated trust — a version control system where every
commit is signed, every file is encrypted, and the server never sees your
content. It preserves the workflow developers already know (clone, commit,
push, pull, merge) while adding end-to-end encryption, cryptographic identity,
and verifiable provenance at the architecture level.

The name: **s** can stand for **signed**, **secure**, or **sgraph** — all
correct, all different angles on the same product. The `.ai` domain reflects
how the product is built, not what it puts inline (more on this in Section 5).

**Tagline:** Git with integrated trust.

## 2. Git Interoperability

### Why This Matters

sgit uses the same core primitives as git: blobs, trees, commits, refs,
branches. The data model is the same directed acyclic graph. The difference
is an encryption and signing layer on top. This means the two systems can
talk to each other — the conversion is mechanical, not conceptual.

This is a strategic advantage: sgit doesn't ask developers to abandon git.
It wraps git in a trust layer.

### The Mapping

Every git object has a direct sgit equivalent:

```
  git                               sgit
  ┌────────────────────┐            ┌──────────────────────────────┐
  │ blob               │            │ blob                         │
  │ plaintext content   │──encrypt─►│ ENC(read_key, content)       │
  │                    │◄─decrypt──│                              │
  └────────────────────┘            └──────────────────────────────┘

  ┌────────────────────┐            ┌──────────────────────────────┐
  │ tree               │            │ tree                         │
  │ "README.md" → blob │──encrypt─►│ ENC("README.md") → obj-{hash}│
  │ plaintext names    │  names    │ encrypted names              │
  └────────────────────┘            └──────────────────────────────┘

  ┌────────────────────┐            ┌──────────────────────────────┐
  │ commit             │            │ commit                       │
  │ tree, parent       │──sign────►│ tree, parents[], branch_id   │
  │ author, message    │           │ signature, message           │
  └────────────────────┘            └──────────────────────────────┘

  ┌────────────────────┐            ┌──────────────────────────────┐
  │ ref                │            │ ref                          │
  │ plaintext commit ID│──encrypt─►│ ENC(read_key, commit_obj_id) │
  └────────────────────┘            └──────────────────────────────┘
```

What maps 1:1 with zero transformation:
- The DAG structure (commit parents)
- Tree-per-folder granularity
- Refs pointing to commits
- Branching and merging
- Content-addressed storage (different hash inputs, same principle)

What needs translation but is straightforward:

| Concept | git → sgit | sgit → git |
|---------|-----------|-----------|
| Blob content | encrypt | decrypt |
| Filenames in trees | encrypt | decrypt |
| Commit parents | `parent` → `parents[]` | `parents[]` → `parent` lines |
| Signatures | generate + sign | preserve as git notes or strip |
| Object IDs | recompute from ciphertext | recompute from plaintext |
| Ref values | encrypt | decrypt |
| Author identity | map to branch key | map to git author metadata |

### The Two Tools

**sgit import** (git → sgit)

```
  $ sgit import --from git@github.com:org/repo.git

  1. clone git repo
  2. walk every object:
     - encrypt blobs
     - encrypt filenames in trees, rebuild as sgit tree objects
     - convert commits: parent → parents[], generate branch key, sign
     - recompute all object IDs (from ciphertext)
  3. preserve graph topology exactly (same parents, same structure)
  4. store as sgit vault in bare/
  5. git commits become sgit commits signed by the import branch key
     with metadata: "imported from git (unsigned original)"
```

**sgit export** (sgit → git)

```
  $ sgit export --to git@github.com:org/repo.git

  1. walk every sgit object:
     - decrypt blobs
     - decrypt filenames, rebuild as git tree objects
     - convert commits: parents[] → parent lines, map branch_id to author
     - recompute git object IDs (from plaintext)
  2. preserve graph topology exactly
  3. optionally preserve sgit signatures as git notes
     (refs/sgit/signatures/{commit_id} → signature data)
  4. push to git remote
```

### Round-Trip Fidelity

A round-trip (git → sgit → git) preserves:
- Complete commit history and graph structure
- All file contents
- All directory structure
- Commit messages and timestamps
- Merge topology

A round-trip adds:
- sgit signatures (preserved as git notes if desired)
- Branch key provenance (which device/session performed the import)

A round-trip loses:
- Nothing from the git side
- sgit-specific metadata (branch_id, signatures) becomes git notes
  rather than first-class fields — but the data is preserved

### Hybrid Workflows

The interop enables a powerful pattern: developers use git locally with
their familiar tools, while the organisation maintains a signed, encrypted
copy in sgit.

```
  Developer (git)                    Organisation (sgit)
  ┌──────────────────┐              ┌──────────────────────────┐
  │ git clone        │              │ sgit vault on sgit.ai    │
  │ git commit       │              │ encrypted, signed,       │
  │ git push         │              │ zero-knowledge server    │
  │                  │              │                          │
  │ familiar tools   │              │ audit trail, provenance, │
  │ IDE integration  │              │ compliance record        │
  └────────┬─────────┘              └────────────┬─────────────┘
           │                                     │
           └──────────── sgit sync ──────────────┘
                    (encrypt on push, decrypt on pull)
```

The vault becomes an **encrypted mirror** of a git repo:
- Developers use git locally (no workflow change)
- The vault stores the encrypted, signed version
- CI/CD reads from git (standard tooling)
- Compliance auditors read from sgit (with cryptographic provenance)
- Both are the same content, different trust layers

### sgit as a Git Remote

The ultimate integration: sgit as a git remote backend.

```
  $ git remote add sgit sgit://vault-key@sgit.ai/vault-id
  $ git push sgit main    ← auto-encrypts, signs, pushes to sgit vault
  $ git pull sgit main    ← auto-decrypts, pulls from sgit vault
```

From the developer's perspective, it's just another remote — like pushing
to GitHub and Bitbucket simultaneously. The sgit remote transparently
handles encryption, signing, and zero-knowledge storage.

This is architecturally feasible because sgit's `bare/` maps to four
operations (`read`, `write`, `delete`, `list`) — the same abstraction
that already supports S3, local folders, and zip files as backends.
A git-protocol adapter is one more backend.

## 3. What sgit.ai Is (The Platform)

**sgit.ai** is the hosted platform for sgit vaults — the equivalent of
GitHub for encrypted, cryptographically-signed version control.

```
  git       ←→  GitHub
  sgit      ←→  sgit.ai
```

The platform provides:
- **Vault hosting** — store and sync encrypted vaults
- **Collaboration** — share vault keys, push/pull/merge
- **Branch visibility** — see who's working, what's changed (after decryption)
- **Change packs** — accept encrypted contributions from external systems
  without giving them vault access
- **Batch API** — high-performance sync (1 round trip instead of N)
- **Git interop** — import from and export to git repos

The platform CANNOT:
- Read vault contents (zero-knowledge)
- See file names, branch names, or commit messages
- Provide server-side search or rendered diffs
- Forge commits or tamper with history

What it CAN do is host opaque encrypted blobs and serve them back. Trust
is in the cryptography, not the platform.

## 5. The .ai Angle — Built by AI, No AI Inline

### What .ai Means for sgit

The `.ai` in sgit.ai does NOT mean "we put an LLM in your version control."
There is no AI inline in sgit workflows. No model reads your commits. No
model generates your merges. No model processes your files. The zero-knowledge
guarantee means no entity — human, server, or AI — sees your content unless
they hold the vault key.

What `.ai` DOES mean:

**sgit is developed, optimised, and maintained by an agentic team.**

The SGraph Send project operates with an 18+ agent team — Architect, Dev,
DevOps, AppSec, QA, Sherpa, Ambassador, Librarian, Cartographer, and more.
Each agent owns a specific domain. They review each other's work. They
produce architecture documents, implementation plans, security audits, and
test strategies. The human (project lead) provides direction via briefs;
the agents execute, review, and iterate.

This very architecture was validated using a novel methodology: **LLM-as-simulator**.
Before writing a single line of code, the architecture was executed as a
simulation — an LLM traced through every workflow, predicting every file,
key, object, and API call at each step. Fourteen architectural gaps were
found and resolved through this process. Zero lines of code were written.
The simulation document became the test oracle for implementation.

### The Principle: AI Builds the Tool, AI Doesn't Run the Tool

```
  ┌─────────────────────────────────────────────────────────────────┐
  │                                                                 │
  │  HOW sgit IS BUILT              HOW sgit RUNS                  │
  │                                                                 │
  │  Agentic team of 18+            Pure cryptographic engine       │
  │  AI agents:                     No LLM in the loop:            │
  │                                                                 │
  │  • Architect designs             • AES-GCM encryption          │
  │  • Dev implements                • EC P-256 signing             │
  │  • AppSec audits                 • SHA-256 hashing             │
  │  • QA validates                  • Content-addressed storage   │
  │  • Librarian documents           • Deterministic merge         │
  │  • Simulation validates          • Four-function backend API   │
  │                                                                 │
  │  AI accelerates development     No AI processes your data      │
  │  AI finds architectural gaps    No model sees your content     │
  │  AI reviews security claims     No inference at runtime        │
  │  AI maintains documentation     No probabilistic operations    │
  │                                                                 │
  │  Result: 10x development        Result: 100% deterministic,    │
  │  velocity                       auditable, verifiable          │
  │                                                                 │
  └─────────────────────────────────────────────────────────────────┘
```

This is a deliberate and important distinction:

**Why no AI inline in sgit workflows:**
- Cryptographic operations must be deterministic. An LLM is probabilistic.
  You cannot have a probabilistic merge algorithm.
- Zero-knowledge means no entity reads your content. If an LLM processed
  your files, zero-knowledge is broken.
- Trust is in the math. Introducing an AI model introduces a trust
  dependency on the model provider. sgit's trust model depends on
  AES-GCM, EC P-256, and SHA-256 — not on any AI company's API.
- Auditability. Every sgit operation is deterministic and reproducible.
  Given the same inputs, you get the same outputs. LLMs don't have
  this property.

**Why AI in the development process:**
- Architecture simulation (14 gaps found before coding)
- Agentic code review (AppSec auditing every security claim)
- Documentation that stays current (Librarian tracking what's real vs proposed)
- Velocity — the same team that would take 6 months with human-only
  development ships in weeks with agentic acceleration

The `.ai` domain signals: **this product was built with AI craftsmanship,
but it runs on mathematical trust.** The AI is in the forge, not in the
blade.

### AI Agents as sgit Users (Not Operators)

There is one place where AI and sgit intersect at runtime: **AI agents as
vault participants.** An AI agent can clone a vault, commit changes, and
push — exactly like a human. The agent gets its own branch key pair and
its own first-class cryptographic identity. Its commits are signed. Its
changes are traceable.

But this is the agent using sgit as a tool — the same way a human uses it.
The agent holds a vault key, makes commits, pushes changes. sgit doesn't
know or care that the committer is an AI. The branch key proves provenance
regardless of whether the key holder is human, agent, or CI pipeline.

This was proven in March 2026 when Claude successfully completed a full
vault round-trip: clone, create files, push, with a human pulling and
confirming the results. The zero-knowledge guarantee held end-to-end —
even with an AI participant, the server never saw plaintext.

## 6. The Product Family

```
  sgraph.ai                          the company
  ├── sgit.ai                        Git with integrated trust
  │   ├── sgit CLI                   the tool (clone, commit, push, pull, merge)
  │   ├── sgit.ai platform           hosted vault platform (the "GitHub")
  │   └── sgit ↔ git interop         import/export, git remote backend
  │
  ├── send.sgraph.ai                 zero-knowledge file transfer
  │   (the current SG/Send product — evolves into sgit.ai's storage backend)
  │
  └── SGraph ecosystem               graph-based knowledge infrastructure
```

send.sgraph.ai is the existing product that provides the vault storage API.
sgit.ai is the version control layer built on top of it. The vault pointer
API (`PUT /write`, `GET /read`, `DELETE /delete`, `GET /list`) is the
interface between them. sgit.ai can use send.sgraph.ai as a backend, or
S3, or a local folder, or a zip file. The backend is pluggable.
