## What Happened

A real-world user test. A non-technical user wanted AI-generated image edits. The workflow that emerged:

1. User sent the original photo via SG/Send (first send)
2. User sent the prompt via WhatsApp
3. Human put the prompt into ChatGPT Pro, generated images
4. Human sent generated images back via SG/Send (second send)
5. Multiple rounds of iteration (prompt tweaks, regeneration, re-send)
6. Final images collected into a folder
7. User sent the folder via SG/Send to someone else (third send)

Observations: the user got visibly more comfortable with SG/Send through repeated use. The workflow worked. The friction was in the middle (switching between WhatsApp for prompts, ChatGPT for generation, SG/Send for transfer). That middle part is the opportunity.

## Bug Found: 5MB Upload Failure

A folder that happened to be exactly around 5MB failed to upload. The encrypted payload (base64 + encryption overhead) pushed it past the Lambda function's payload size limit. Adding one more file to the folder pushed it into the chunked upload path, which worked fine.

**Fix:** Validate the encrypted payload size before calling the server. If the payload exceeds 6MB (Lambda limit), automatically route through the chunked upload path. The user should never see this failure. The threshold check should happen client-side before the upload call.

**Priority:** P0. This is a silent failure that makes users think the service is broken.

## LLM Image Generation as a Paid Feature

### The Idea

Add image generation (and eventually other LLM capabilities) directly to SG/Send. The user uploads a photo, writes a prompt, gets a generated image back. All within the same interface. No switching to ChatGPT. No switching to Gemini. One workflow.

The workspace already supports LLM integration via OpenRouter keys. The infrastructure exists. The UI needs to be built as a standalone tool/component that makes it dead simple: prompt in, image out.

### Why This Works as Revenue

The current revenue model: users buy credits, credits cover uploads and downloads. This adds a second revenue stream: markup on LLM API calls.

The economics:

- User pays credits for the LLM operation (image generation, text processing, etc.)
- SG/Send pays OpenRouter/OpenAI/Gemini at wholesale API rates
- The margin is the markup

This is attractive because:

- Many people have not used the latest image generation models (ChatGPT, Gemini). The barrier is having an account, understanding the API, managing keys. SG/Send removes all of that.
- The cost per operation is visible. Users see "this image generation cost 50p" and can decide if it is worth it.
- Different models at different price points. Cheap and fast vs. expensive and high quality. The user chooses.
- The LLM feature gives users a reason to be on the platform beyond file transfer. They come for image generation, they stay for encrypted transfer and vaults.

### The Workflow We Should Build

```
1. User opens image generation tool
2. Uploads a photo (optional) and writes a prompt
3. Chooses a model (GPT-4o, Gemini, Ollama local, etc.)
4. Sees the cost before confirming
5. Image is generated
6. Image appears in their workspace
7. User can send it via SG/Send, add it to a vault, or download it
```

The observed workflow (photo in, prompt via WhatsApp, image back via SG/Send, folder to someone else) becomes: photo in, prompt in the same UI, image out, folder to someone else. All in one place.

## Secure API Key Sharing via Vault

### The Problem

To give a user access to LLM APIs, they need an API key. API keys are secrets. How do you share a secret securely?

### The Solution

Use the vault. Use the user's public key.

The flow:

1. SG/Send creates an OpenRouter API key programmatically (with a spending cap, e.g. 4 GBP)
2. The key is encrypted with the user's public key (or their access token key)
3. The encrypted key is stored in the user's vault (or delivered via SG/Send transfer)
4. Only the user can decrypt it
5. The user's browser uses the key to call the LLM API directly

This is a perfect demonstration of the encrypt-for-reader paradigm. The API key is encrypted for the user, not for storage. SG/Send creates the key and encrypts it. Only the intended user can decrypt and use it. If the vault is compromised, the key is still encrypted.

### Why This Is Powerful

This is not just a feature. It is an end-to-end demonstration of the entire technology stack:

- Public/private key encryption (encrypt the API key for the user)
- Vault storage (store the encrypted key)
- Zero-knowledge (SG/Send does not retain the decrypted key)
- Billing (the API key has a spending cap, tied to the user's credits)
- Real value delivery (the user gets access to LLM capabilities they could not easily access otherwise)

The whole thing is a case study in what SPKI infrastructure makes possible.

## Version Control for People Who Have Never Used Git

### The Observation

The user in this test did what most non-technical users do: collected the final images into a folder and sent it. If there had been a second round of edits, the folder would have been duplicated. A third round would produce `photos_v3_final_FINAL.zip`. This is how the majority of the world manages versions: by renaming files and duplicating folders.

These users have never experienced branches and merges. They have never seen a commit history. They have never had the ability to say "show me what changed" or "go back to the version from Tuesday." Version control either does not exist in their workflow or it is done via never-ending filenames.

### The Opportunity

The vault branch model (see companion architecture brief) gives us Git-like capabilities: branches, commits, merges, history. But the UI does not have to look like Git. It does not have to use Git terminology. It does not have to feel technical.

Imagine a shared folder (a vault) where:

- Multiple people are editing at the same time
- Each person's changes are tracked automatically (their branch, created on clone)
- When someone adds a file, it appears for everyone after they sync
- When two people edit the same file, the system shows both versions side by side and lets them choose
- The full history is visible: who added what, when, what changed between versions
- Any previous version can be restored with one click

For a young user who has only ever used Google Drive or WeTransfer, this is magical. Folders that track their own history. Files that know who changed them. Conflicts that resolve visually instead of silently overwriting.

For a business user who manages projects via email attachments and `report_Q3_final_v2_ACTUALLY_FINAL.docx`, this is transformative. No more "which version is the latest." No more "did you see my changes." No more lost edits.

### What Makes This Spectacular

The audience has no frame of reference. They have never used Git. They have never seen a branch diagram. When they see a visual timeline of their folder's history, with thumbnails of what changed at each step, it feels like the folder is alive. When they see two people's edits merged automatically, it feels like collaboration actually works for the first time.

This is Git's power, delivered through a UI that looks like a file manager, not a terminal. The vault branch model provides the engine. The UI provides the experience. The encryption provides the trust. Together they create something that no consumer product currently offers: real version control for normal people, with zero-knowledge encryption.

### Implementation Notes

The UI should never use the words "branch," "commit," "merge," or "HEAD." Instead:

| Git concept | User-facing language |
|---|---|
| Branch | "Your changes" or "Your workspace" |
| Commit | "Save" or "Checkpoint" |
| Merge | "Sync" or "Combine changes" |
| Conflict | "Two versions exist, pick one" (with visual diff) |
| History/log | "Timeline" or "Version history" |
| Revert | "Restore this version" |

The technical infrastructure is the branch model. The user experience is a folder that remembers everything.
