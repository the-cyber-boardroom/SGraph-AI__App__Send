# Simulation-Driven Architecture: A Case Study in LLM-as-Simulator

**version** v0.13.30
**date** 13 Mar 2026
**author** Architect (Claude) + Human (Dinis Cruz)
**context** SGraph Send CLI — branch model architecture design session

---

## What Just Happened

Over the course of a single session, we designed a complete branch model
architecture for an encrypted vault system — without writing a single line
of production code. We did this by using an LLM (Claude) as a **simulator**
for code that doesn't exist yet, executing the architecture on paper and
tracking every file, key, object, and API call that would result from each
user action.

The process produced 6 revisions of a simulation document, found 14
architectural gaps, resolved 13 of them through conversation, and ended
with a locked specification that is ready for implementation. The architecture
changed significantly from start to finish — not because the original design
was wrong, but because simulation revealed things that static analysis couldn't.

This document explains the methodology, what it found, and why it works.

---

## The Setup

### The Problem

SGraph Send is a zero-knowledge encrypted file vault. The CLI tool (`sg-send-cli`)
already supports basic vault operations: init, clone, push, pull, status. The next
phase adds branches, multi-user collaboration, merge, and conflict resolution —
a significant architectural expansion with many moving parts.

Three architecture briefs had been written by the project lead. The Explorer team
(Architect + Dev) had reviewed them and given a GO verdict with an 8-9 week
timeline. Two companion architecture exploration documents had been drafted
covering the branch model and the vault filesystem design.

The architecture was ready for implementation. Or so we thought.

### The Insight

The project lead proposed something different: instead of handing the architecture
directly to the implementation team, **simulate it first**. The idea:

> "Let's use Claude as the CLI engine. One side issues commands, the other
> responds with the predicted state. We should be able to predict exactly
> what files are created, what keys exist, what objects are in the store —
> at every step. If we can't predict it, the design has a gap."

This is the key insight: **an architecture you can't simulate is an architecture
you don't fully understand.** If you can't walk through `init → commit → push`
and state exactly what happens at each step — every file, every key, every API
call — then there are unstated assumptions that will surface as bugs during
implementation.

## The Methodology

### Step 1: Create an Architecture Flows Document

Before simulating, we created a visual reference document (`arch-flows`) that
mapped every workflow as ASCII art: init, clone, commit, push, pull, merge,
change packs, remotes, zip export. Each flow showed the API calls, the objects
created, and the request budget.

This document served as the **hypothesis** — our best understanding of how
the system should work. The simulation would test it.

### Step 2: Simulate — Predict State at Every Step

The simulation document traced through concrete scenarios:

- **Scenario 1:** Solo workflow (init, add files, commit, push)
- **Scenario 2:** Two-user collaboration (clone, diverge, push, merge)
- **Scenario 3:** Conflict (both edit same file)
- **Scenario 4:** Change pack (keyless creator, no vault access)
- **Scenario 5:** Status and update detection

For each step within each scenario, the simulation predicted:

| What we tracked | Example |
|----------------|---------|
| Every file created on disk | `bare/data/obj-h_blob_rm01` |
| Every file on the server | `bare/refs/ref-a3f8` |
| Every cryptographic key | `key-9b3e` (current public, in vault) |
| Every object in the store | `obj-h_cmt_0002` (commit, signed by br1) |
| Every ref update | `ref-7b2e → obj-h_cmt_0002` |
| Every API call | `1 GET + 1 POST (batch) = 2 round trips` |
| The complete object graph | ASCII art showing commit parents and tree structure |
| Who signed what | `signed: fp_br1_3c8f ✓` on every commit |

The simulation was **executable on paper** — Claude acted as the CLI engine,
processing each command and producing the resulting state, exactly as the code
would if it existed.

### Step 3: Gaps Surface Naturally

When the simulation couldn't predict the next state, that was a gap. When two
parts of the design contradicted each other, that was a gap. When a workflow
required something the architecture hadn't specified, that was a gap.

Gaps weren't found by reviewing the architecture document. They were found by
**trying to execute it**.

### Step 4: Fix Gaps in the Simulation, Not the Architecture Doc

This is the critical methodological choice. When a gap was found, we didn't
go back and edit the architecture exploration documents. We fixed it in the
simulation — by discussing the options, choosing one, and continuing the
simulation with the corrected design.

The simulation became the **living design document** — the architecture
exploration docs were the starting point, but the simulation was where decisions
were actually made and tested.

### Step 5: Loop Until No More Gaps

Each revision of the simulation either found new gaps or resolved existing ones.
The loop:

```
  simulate → find gap → discuss options → decide → update simulation → repeat
```

This produced 6 revisions over the course of one session. Each revision was
tighter than the last.

### Step 6: Lock the Simulation, Then Update the Architecture

Once the simulation was stable (no new gaps found, all existing gaps resolved
or explicitly deferred), we locked it. The simulation document is now the
**source of truth** for how the architecture should behave. The architecture
spec documents will be updated to match the simulation, not the other way around.

## What the Simulation Found: 14 Gaps

### The Original Gaps (v1 — First Simulation)

| # | Gap | How it was found |
|---|-----|-----------------|
| 1 | **Clone checks out empty main** | Simulating Scenario 2 (User B clones). Main was still at init because User A had committed to their branch but never merged. User B would get an empty working directory despite files existing on another branch. |
| 2 | **Change pack auth model** | Simulating Scenario 4 (keyless creator). The creator has no `write_key` because they don't have the vault passphrase. But all API writes require `write_key`. The architecture's most novel feature was blocked by the auth model. |
| 3 | **Who triggers merge into main?** | Simulating the push flow. If nobody explicitly merges into the named branch, it stays behind indefinitely. There was no specified trigger. |

These three gaps existed in the architecture exploration documents. They would
have been discovered during implementation — as bugs, as confused developers,
as "wait, how does this work?" conversations that block progress for days.

### The Simulation-Only Gaps (v2–v6 — Found During Execution)

| # | Gap | How it was found |
|---|-----|-----------------|
| 4 | **Merge-then-fetch ordering** | Simulating User B's push when remote had diverged. The engine merged the clone branch into current BEFORE checking if the remote had moved — creating a throwaway merge commit that was immediately discarded. Reordering to fetch-first-then-merge eliminated the waste. |
| 5 | **Conflicts on named branch** | Simulating the conflict scenario. The original design had conflicts appearing on the named branch, which would block further merges from other users. Moving conflicts to the clone branch kept the named branch clean. |
| 6 | **Batch endpoint spec** | Counting API calls during push. Individual PUTs from client to server were 6+ round trips. A batch endpoint (1 POST, Lambda does direct S3 calls) reduced this to 2 round trips. |
| 7 | **Clone branch ref visibility** | Simulating whether the server should know about clone branches. Yes — for visibility ("who's working on this vault") and for the `--branch-only` push flow. |
| 8 | **Init commit signing** | Tracing the signature chain. Who signs the init commit? The clone key (not the named branch key). The named branch's signature chain starts at the first merge. |
| 9 | **Fast-forward vs merge commit** | Simulating the named branch receiving work. Should it fast-forward or create a merge commit? Always merge commit — preserves the "this was explicitly integrated" boundary. Fast-forward only for local current accepting remote current. |
| 10 | **Branch discoverability** | Simulating clone — how does the client know what branches exist? Needed an encrypted index file. |
| 11 | **merge --abort orphaned commits** | Simulating cancel-after-conflict. The conflict commit becomes unreachable. GC cleans it up. |
| 12 | **--branch-only push semantics** | Emerged from the collaborative conflict resolution flow. Need clear semantics: pushes only the clone branch ref + objects, never touches the named branch. |
| 13 | **Index staleness window** | Reasoning about the branch index. Between GC runs it may not reflect the latest state. Acceptable — eventually consistent by design. |
| 14 | **Object type opacity** | Collapsing `objects/commits/`, `objects/trees/`, `objects/blobs/` into `bare/data/obj-{hash}`. The server can't tell types apart. Client discovers type via `schema` field after decryption. |

### The Design Improvements (Not Bugs — Architecture Enhancements)

Beyond gaps, the simulation drove several design improvements that weren't
bugs but made the architecture significantly better:

**The "current" named branch pattern.** The original briefs described merging
into `main`. The simulation revealed that an intermediary — a local named
branch that tracks the remote — creates a cleaner three-layer model where
the remote is always the authority, the local named branch fast-forwards to
match it, and the clone branch absorbs conflicts.

**Conflicts on clone branch, not named branch.** This emerged from simulating
what happens when User B has a conflict. If the conflict is on the named
branch, no one else can merge until it's resolved. If it's on the clone
branch, the named branch stays clean and other users are unaffected. This
also enabled the "push conflicted branch for collaborative resolution" and
"cancel merge by resetting the ref" workflows — neither of which were in
the original architecture.

**Zero-knowledge branch naming.** Walking through what the server would see
in its file listing revealed that file paths like `refs/heads/current` and
`keys/public/fp_br1.pub` leak branch names and relationships. The rename to
opaque IDs (`ref-a3f8`, `key-9b3e`) with an encrypted index was a direct
result of the simulation forcing us to state exactly what the server sees.

**`bare/` as the vault.** Listing every file in `.sg_vault/` during the
simulation made it visible that `objects/` was the complete vault and
`local/` was just device config. Renaming `objects/` to `bare/` made the
portable-vault concept explicit: `bare/` IS the zip, IS the server content,
IS what you get on `clone --bare`.

**Uint timestamps everywhere.** Seeing ISO datetime strings in pending
filenames during the simulation triggered the correction: all timestamps
use `Timestamp_Now` (uint milliseconds since epoch) — compact, sortable,
and no human-readable temporal metadata leaked to the server.

## Why This Works

### 1. Simulation Forces Completeness

An architecture document can leave things vague. "The system merges the
branches" is a valid sentence in a spec. But in a simulation, you have to
answer: which branch merges into which? What commit is created? Who signs
it? What are the parents? What's the tree_id? What happens to the refs?

Vagueness in the architecture becomes a **runtime error** in the simulation.
The LLM can't proceed without making the decision, which forces the human
to make it.

### 2. The LLM Holds State That Humans Can't

By Scenario 2, Step 2.4, the simulation was tracking: 3 key pairs across
2 devices, 2 branches with diverged histories, 8+ objects with content-addressed
IDs, refs pointing to different commits on different devices, and the merge
resolution of all of them. A human sketching this on a whiteboard would lose
track. The LLM maintains the full state at every step.

### 3. Gaps Compound — Early Discovery Saves Exponential Time

Gap 4 (merge-then-fetch ordering) would have been a subtle bug in
implementation. It would have manifested as "sometimes a push creates a
commit that immediately gets discarded" — hard to reproduce, hard to diagnose,
and the kind of thing that erodes trust in the system. Finding it in
simulation cost 5 minutes of conversation. Finding it in code would have
cost hours or days.

Gap 5 (conflicts on named branch) would have required a **design-level
rewrite** if found during implementation. The conflict flow touches every
part of the merge system. Discovering in simulation that conflicts should
live on the clone branch was a paragraph of conversation. Discovering it
in code would have been a multi-day refactor.

### 4. The Simulation Becomes the Test Spec

The QA predictions table at the end of the simulation document is a concrete,
testable specification: "After init, there should be 10 encrypted files in
bare/ and 0 working directory files." After implementation, QA runs the exact
same scenarios and verifies the counts match. If they don't, either the
implementation diverged or the simulation had an unstated assumption — both
are valuable findings.

### 5. The Human Steers, the LLM Executes

The methodology works because the human provides the design judgment ("conflicts
should live on the clone branch, not the named branch") and the LLM provides
the exhaustive state tracking ("given that decision, here are the exact 28
files that exist on the server after Scenario 2"). Neither could do the other's
job effectively. The human can't track 28 files across 2 devices. The LLM can't
decide which branch should hold conflicts.

## The Workflow, Generalised

This methodology is not specific to vault architecture. It applies to any
system with predictable state transitions:

```
  ┌──────────────────────────────────────────────────────────────────┐
  │                                                                  │
  │  1. WRITE the architecture                                       │
  │     (briefs, exploration docs, design reviews)                   │
  │                                                                  │
  │  2. CREATE a flows document                                      │
  │     (visual reference — ASCII art, sequence diagrams)            │
  │                                                                  │
  │  3. SIMULATE concrete scenarios                                  │
  │     (LLM as the engine, human as the command invoker)            │
  │     predict EVERY state change: files, keys, objects, API calls  │
  │                                                                  │
  │  4. FIND gaps                                                    │
  │     (the simulation can't proceed → the design is incomplete)    │
  │                                                                  │
  │  5. FIX gaps in the simulation                                   │
  │     (not the original architecture doc — that comes later)       │
  │                                                                  │
  │  6. LOOP until stable                                            │
  │     (no new gaps, all existing gaps resolved or deferred)        │
  │                                                                  │
  │  7. LOCK the simulation                                          │
  │     (it is now the source of truth for expected behaviour)       │
  │                                                                  │
  │  8. UPDATE the architecture documents                            │
  │     (to match the simulation, not the other way around)          │
  │                                                                  │
  │  9. HAND OFF to the implementation team                          │
  │     (Architect, Dev, DevOps, AppSec review the updated spec      │
  │      plus the simulation as a test oracle)                       │
  │                                                                  │
  │  10. QA VALIDATES against simulation predictions                 │
  │      (are the file counts right? are the keys where expected?    │
  │       does the object graph match?)                              │
  │                                                                  │
  └──────────────────────────────────────────────────────────────────┘
```

The key principle: **the simulation is the design**. The architecture documents
are the explanation of the design. When they disagree, the simulation wins —
because the simulation was tested against concrete scenarios, and the
architecture document was not.

## What Comes Next

The simulation document is locked. The next steps are:

1. **Update the architecture exploration documents** (`branch-model.md` and
   `vault-filesystem.md`) to reflect everything the simulation settled:
   the `bare/` structure, the "current" named branch, the conflict-on-clone
   flow, zero-knowledge naming, uint timestamps, the batch endpoint.

2. **Hand the updated architecture + simulation to the agentic team:**
   - **Architect** reviews the updated spec for consistency
   - **Dev** creates implementation plans from the simulation steps
   - **DevOps** designs the batch endpoint and server-side S3 integration
   - **AppSec** reviews the zero-knowledge properties and key lifecycle

3. **QA uses the simulation as a test oracle.** The QA predictions table
   provides exact expected state at each step. After implementation, every
   scenario is replayed and validated against the predictions.

4. **The simulation methodology is added to the team's standard practice.**
   Every significant architecture change gets simulated before implementation.
   The cost is one Claude session. The savings are measured in weeks of
   implementation rework avoided.

## Closing Thought

The most expensive line of code is the one you write based on an architecture
that has an unstated assumption. You don't discover the assumption until
integration testing, by which point you've built on top of it. The fix isn't
changing one line — it's rethinking the foundation.

Simulation surfaces unstated assumptions before the first line of code is
written. The LLM doesn't know the architecture is right — it just tries to
execute it. When it can't, you've found the gap. When it can, you've built
confidence. Either way, you've learned something that would have cost
significantly more to learn from code.

Fourteen gaps. Zero lines of code. One session.
