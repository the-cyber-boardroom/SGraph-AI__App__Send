## How to Read This Document

Each step shows:
- **Command** — what the user types
- **Engine** — what the CLI does internally
- **State** — exact files that exist after this step

Conventions:
```
  📁  file on disk (plaintext working copy)
  🔒  encrypted object in .sg_vault/ or on server
  🔑  cryptographic key
  📌  ref (pointer to a commit)
  ☁️   on SG/Send server
  ⚡  new since previous step
```

**Timestamp format:** All timestamps throughout the vault use `Timestamp_Now`
from `osbot-utils` — a uint representing milliseconds since epoch
(e.g. `1710244800000`). Never ISO datetime strings. This is a project-wide
rule, not just for filenames. The uint format is compact, sortable, timezone-free,
and reveals no human-readable temporal information on the server.

## Why `bare/` Not `objects/` (v6 Correction)

Looking at `.sg_vault/`, there are exactly two children:

```
  .sg_vault/
  ├── bare/             ← THE vault
  └── local/            ← YOUR device config
```

Everything inside `bare/` is the vault — branches, refs, data objects, keys,
indexes, pending change packs. It is the complete, self-contained, portable
unit that syncs identically to the server, to a zip, or to another clone.
Nothing else needs to exist inside `.sg_vault/` for the vault to be whole.

`local/` is purely your device's relationship to the vault — your clone
private key, vault key, token, remote config, tracking state. It never syncs.
Another device creates its own `local/` with its own keys and config.

The name `bare/` was chosen because:

- **It IS the bare vault.** When a user does `clone --bare`, what they get is
  exactly the contents of this folder — no plaintext files extracted, no vault
  key on disk, just the encrypted vault.
- **It IS the zip.** `sg-send-cli export` zips this folder. A recipient unzips
  it, creates their own `local/`, and has a working vault.
- **It IS the server.** The server stores exactly what's inside `bare/`.
  After a sync, `bare/` and the server are identical.
- **Symmetry with git.** A git bare repo (`git clone --bare`) contains `.git/`
  contents without a working tree. Same concept — the vault without the checkout.

The invariant: **delete `bare/` → the server is a complete backup. Delete the
server → `bare/` is a complete backup. They are equals.**

## Key Design Correction (v4→v5): Zero-Knowledge Branch Naming + Folder Structure

**Problem in v3:** File paths like `refs/heads/current` and `keys/public/fp_br1.pub`
leak branch names, types, and relationships to the server. The server — or anyone
who intercepts the storage — can see "there's a branch called current" and
"there's a public key for branch fp_br1." This is metadata leakage.

**Principle:** There must be ZERO non-encrypted data in file paths. Branch names
are data — they belong inside encrypted blobs, not in filenames.

**Solution:** Folders group by type (acceptable metadata — the server knows "there
are 3 refs"). Filenames within each folder are opaque IDs. Content is always encrypted.

```
  Folder structure:
    bare/branches/        branch metadata files
    bare/refs/            ref pointers (mutable)
    bare/data/            content-addressed objects (blobs, trees, commits)
    bare/keys/            cryptographic keys
    bare/indexes/         eventually-consistent index files
    bare/pending/         change packs
    local/                never synced (clone private key, config, tracking)

  File naming within folders:
    bare/branches/branch-named-{id}     a named branch
    bare/branches/branch-clone-{id}     a clone branch
    bare/refs/ref-{id}                  a ref (commit pointer)
    bare/data/obj-{hash}                a blob, tree, or commit
    bare/keys/key-{id}                  a public or private key
    bare/indexes/idx-{id}               an index file
    bare/pending/pending-{uint_ts}_{rnd}  a change pack

  The server sees folders + opaque filenames:
    bare/branches/branch-named-a3f8c2
    bare/branches/branch-clone-p4n8v2
    bare/refs/ref-7b2e1d
    bare/data/obj-c9a4f1
    bare/keys/key-9b3e7f
    ...

  The server knows:  types (from folders) and counts
  The server does NOT know: names, relationships, content, which key
                            belongs to which branch, or which objects
                            are blobs vs trees vs commits
```

**The encrypted index file:**

```
  bare/indexes/idx-{branches-index-id} contains (encrypted):
  {
    "branches": {
      "branch-named-x7k2m9": {
        "name": "current",
        "type": "named",
        "head_ref_id": "ref-a3f8c2",
        "public_key_id": "key-9b3e7f",
        "private_key_id": "key-d4a1c8",
        "created_at": 1710244800000
      },
      "branch-clone-p4n8v2": {
        "name": "fp_br1_3c8f",
        "type": "clone",
        "head_ref_id": "ref-7b2e1d",
        "public_key_id": "key-e5f2a9",
        "created_at": 1710244800000
      }
    }
  }

  This index is EVENTUALLY CONSISTENT.
  Guaranteed accurate after GC or explicit index-update.
  May be stale between syncs.
```

## The Three-Layer Model (Unchanged From v3)

```
  REMOTE
  ┌──────────────────────────────────┐
  │  "current" named branch          │  authoritative
  │  (stored as branch-named-{id})   │  other users push here
  └──────────────┬───────────────────┘
                 │
  ── ── ── ── ──│── ── ── ── ── network boundary ── ── ── ──
                 │
  LOCAL          │
  ┌──────────────▼───────────────────┐
  │  "current" named branch          │  fast-forwards to match remote
  │  (local copy, receives merges)   │  NEVER has conflicts
  └──────────────┬───────────────────┘
                 │
  ┌──────────────┴───────────────────┐
  │  your clone work branch          │  commits go here
  │  (conflicts appear HERE)         │  absorbs current via merge
  └──────────────────────────────────┘

  Merge directions:
    remote  ──fast-forward──►  local current
    local current  ──merge──►  clone branch    (conflicts HERE)
    clone branch   ──merge──►  local current   (always clean)
    local current  ──push──►   remote
```

## The Definitive Push Workflow (Unchanged From v3)

```
  1. CHECK: working dir clean?
  2. FETCH remote "current" (by reading the branch index, finding the ref)
  3. Remote ahead? → fast-forward local current
  4. Current ahead of clone? → merge current INTO clone branch
     CONFLICTS? → stop (resolve on clone branch, push again)
  5. .conflict files? → refuse to push
  6. MERGE clone INTO current (always clean, signed by named branch key)
  7. PUSH batch (write-if-match on current's ref)
```

## Unified Storage Layout

**Identical on server, local bare, and zip. Folders group by type. Filenames are opaque.**
**`bare/` IS the vault — portable, syncable, zippable.**

```
  vault/{vault_id}/
  │
  ├── bare/
  │   │
  │   ├── branches/
  │   │   ├── branch-named-{id_1}          🔒 ENC(read_key, {
  │   │   │                                     name: "current",
  │   │   │                                     type: "named",
  │   │   │                                     created_at: 1710244800000,
  │   │   │                                     creator_branch: "branch-clone-{id_2}"
  │   │   │                                   })
  │   │   │
  │   │   └── branch-clone-{id_2}          🔒 ENC(read_key, {
  │   │                                         name: "fp_br1_3c8f",

  │   │   │                                   ↑ current's head ref (mutable)
  │   │   └── ref-{id_b}                    🔒 ENC(read_key, commit_obj_id)

  │   │

  │   │   │                                   (double-enc for locked named branches)
  │   │   └── key-{id_e}                    🔒 ENC(read_key, public_key)
  │   │

  │   │   ├── idx-{branches}                🔒 encrypted branch index
  │   │   │                                   (eventually consistent)
  │   │   ├── idx-{pending_counts}          🔒 { branch_id → count }
  │   │   └── idx-{temporal_hour}           🔒 recent commit_ids
  │   │
  │   └── pending/
  │       └── pending-{uint_ts}_{random}    🔒 encrypted change pack
  │
  └── local/                                  (NEVER synced to server)

### Step 1.1: Init (Local Only)

```
$ sg-send-cli init my-vault
```

**Engine:**

```
  1. generate vault_key = "abc123def456ghi789jkl012:v8x2m4p1"

  2. derive keys (HKDF-SHA256)
     read_key, write_key

  3. generate EC P-256 key pair: "current" named branch
     current_pub, current_priv

  4. generate EC P-256 key pair: this clone's work branch
     br1_pub, br1_priv

  5. assign opaque IDs (random):
     branch IDs:  branch-named-x7k2 (current), branch-clone-p4n8 (clone)
     ref IDs:     ref-a3f8 (current's HEAD), ref-7b2e (clone's HEAD)
     key IDs:     key-9b3e (current pub), key-d4a1 (current priv),
                  key-e5f2 (br1 pub)

  6. create empty root tree
     obj-h_tree_0001 = ENC(read_key, { schema: "tree_v1", entries: [] })

  7. create init commit (signed by clone key)
     obj-h_cmt_0001 = ENC(read_key, {
       schema:     "commit_v1",
       tree_id:    "obj-h_tree_0001",
       parents:    [],
       timestamp:  1710244800000,
       message:    "init",
       branch_id:  "branch-clone-p4n8",
       signature:  SIGN(br1_priv, ...)
     })

  8. write refs (both point to init commit)
     ref-a3f8 = ENC(read_key, "obj-h_cmt_0001")    ← current's HEAD
     ref-7b2e = ENC(read_key, "obj-h_cmt_0001")    ← clone's HEAD

  9. write branch metadata
     branch-named-x7k2 = ENC(read_key, {
       name: "current", type: "named",
       head_ref_id: "ref-a3f8",
       public_key_id: "key-9b3e",
       private_key_id: "key-d4a1",
       created_at: 1710244800000
     })
     branch-clone-p4n8 = ENC(read_key, {
       name: "fp_br1_3c8f", type: "clone",
       head_ref_id: "ref-7b2e",
       public_key_id: "key-e5f2",
       created_at: 1710244800000
     })

  10. write keys
      key-9b3e = ENC(read_key, current_pub)
      key-d4a1 = ENC(read_key, current_priv)     ← anyone with read_key can merge
      key-e5f2 = ENC(read_key, br1_pub)

  11. write branch index
      idx-m8n4 = ENC(read_key, {
        "branch-named-x7k2": { name: "current", head_ref: "ref-a3f8", ... },
        "branch-clone-p4n8": { name: "fp_br1_3c8f", head_ref: "ref-7b2e", ... }
      })
```

**State — LOCAL DISK:**

```
  my-vault/
  ├── .sg_vault/
  │   └── bare/
  │   │   ├── branches/
  │   │   │   ├── branch-named-x7k2       🔒 "current" metadata
  │   │   │   └── branch-clone-p4n8       🔒 clone branch metadata
  │   │   ├── refs/
  │   │   │   ├── ref-a3f8                 🔒 → obj-h_cmt_0001
  │   │   │   └── ref-7b2e                 🔒 → obj-h_cmt_0001
  │   │   ├── data/
  │   │   │   ├── obj-h_tree_0001          🔒 empty root tree
  │   │   │   └── obj-h_cmt_0001           🔒 init commit
  │   │   ├── keys/
  │   │   │   ├── key-9b3e                 🔒 current public key
  │   │   │   ├── key-d4a1                 🔒 current private key
  │   │   │   └── key-e5f2                 🔒 br1 public key
  │   │   └── indexes/
  │   │       └── idx-m8n4                 🔒 branch index
  │   └── local/
  │       ├── branch_key.pem               🔑 br1_priv (NEVER LEAVES DEVICE)
  │       ├── vault_key                    "abc123....:v8x2m4p1"

  │
  (no working copy — empty vault)
  (no remotes.json — no remote)
```

       signed: branch-clone-q9w3           ← clone key
     )
     ref-c5d1 → obj-h_cmt_b_conflict

  5. .conflict files → PUSH BLOCKED
     Named branch (ref-a3f8) is UNTOUCHED. Clean.
```

**User's three options:**

```
  Option A: resolve locally
    edit README.md, rm README.conflict.md, commit, push

  Option B: push conflicted branch for help
    sg-send-cli push origin --branch-only
    → pushes ref-c5d1 + new objects only. ref-a3f8 untouched.
    → another user can check out br2, resolve, push back

  Option C: cancel merge
    sg-send-cli merge --abort
    → ref-c5d1 reset to pre-merge value
    → conflict commit becomes orphaned (GC cleans up)
    → working dir restored to your pre-merge state
```

## Future Capability: Worktrees (Git Worktree Equivalent)

The `bare/` + `local/` architecture naturally supports multiple working copies
from a single vault — the equivalent of `git worktree`. This is NOT in the
initial implementation phases. It will be built when there is customer demand.

**Why it's architecturally ready now:**

The current design already separates the vault (`bare/`) from the working copy
(plaintext files next to `.sg_vault/`). A working clone is just `bare/` + one
checkout. There's nothing tying the checkout to a specific directory — the
connection is implicit (the working copy sits next to `.sg_vault/`).

Worktrees make this explicit: one `bare/`, multiple checkouts at different
paths, each on a different branch.

**What it would look like:**

```
  .sg_vault/
  ├── bare/                    ← single copy of all objects (shared)
  ├── local/                   ← device config (keys, token, remotes)
  └── worktrees/               ← checkout metadata (NOT in bare/, NOT synced)
      ├── worktree-{id_1}.json    🔒 ENC(read_key, {
      │                                path: "/home/user/project",
      │                                branch_ref: "ref-7b2e",
      │                                checked_out_at: 1710244800000,
      │                                head_at_checkout: "obj-h_cmt_0002"
      │                              })
      │
      └── worktree-{id_2}.json    🔒 ENC(read_key, {
                                       path: "/home/user/experiment",
                                       branch_ref: "ref-c5d1",
                                       checked_out_at: 1710246600000,
                                       head_at_checkout: "obj-h_cmt_b001"
                                     })

  /home/user/project/          ← working copy on branch br1
  ├── README.md
  └── configs/EC2.json

  /home/user/experiment/       ← working copy on branch br2 (different content)
  ├── README.md
  └── configs/EC2.json
```

**Key properties:**

- Same object store. No duplication of blobs, trees, or commits.
- Two different branches checked out simultaneously in two directories.
- `worktrees/` lives in `.sg_vault/` alongside `bare/` and `local/` — it's
  device-local state (another device has its own worktrees at different paths).
- Adding a worktree is cheap: just create the metadata file and extract files.
- The three-folder model remains clean:

```
  bare/        = the vault (syncs, portable, is the zip)
  local/       = your device identity (keys, token, remotes)
  worktrees/   = your device's working copies (paths, checkout state)
```

**Not in scope for initial phases.** The first implementation uses the simpler
model: one working copy next to `.sg_vault/`. Worktrees are a natural extension
when multi-branch local workflows become a customer need.
