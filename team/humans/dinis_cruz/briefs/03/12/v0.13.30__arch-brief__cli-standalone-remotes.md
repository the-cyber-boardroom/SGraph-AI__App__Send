# CLI as Standalone Project with Remotes

**date** 12 Mar 2026
**from** Human (project lead)

## The Shift

The CLI is becoming a standalone tool, not just a client for SG/Send. The mental model should invert: the CLI is the vault tool. SG/Send is one storage backend.

Today, the CLI accesses the SG/Send API to manage vaults. But the vault operations (encrypt, decrypt, commit, branch, merge) are all local. The server is just an encrypted blob store. The CLI should work identically against:

- The SG/Send API (current default)
- A local folder
- A zip file
- Another vault (clone from vault to vault)
- Any S3-compatible bucket
- A URL (download a vault bundle)

This means the CLI is a complete encryption and version control tool that happens to have SG/Send as its default remote. Someone using only the CLI and a local folder should have the full feature set: create vaults, add files, commit, branch, merge, encrypt, decrypt, sign, verify.

## Remotes (Git Model)

Adopt Git's remotes concept:

```
sg-send-cli remote add origin https://send.sgraph.ai
sg-send-cli remote add backup /mnt/external/vaults
sg-send-cli remote add s3 s3://my-bucket/vaults
```

Push and pull become remote-aware:

```
sg-send-cli push origin
sg-send-cli pull backup
sg-send-cli clone /path/to/local/vault
sg-send-cli clone https://send.sgraph.ai/api/vault/download/abc123
```

This makes the CLI portable. A user can push to SG/Send for sharing and to a local folder for backup. A Lambda function can push to S3 directly. A CI/CD pipeline can clone from a URL.

## Feasibility Research

Questions to answer:

1. What is the full feature set if someone only has the CLI and no server?
2. Which operations currently depend on the SG/Send API vs. which are pure local?
3. What is the minimum abstraction layer needed to support multiple remote types?
4. Can we factor out the API client into a pluggable backend?

The goal: if someone installs `sg-send-cli` from PyPI and never configures a remote, they can still create encrypted vaults, add files, commit, branch, and merge. All locally. The server is an optional sync target.

## Offline Mode

Full offline support is already close. Validate that these work without any network access:

- Create vault
- Add files
- Commit
- Branch
- Merge
- Encrypt/decrypt
- Sign/verify

Network is only needed for push, pull, and clone-from-remote.

## Email and WhatsApp Distribution

If the CLI can export a vault as a zip, that zip can be sent via any channel: email, WhatsApp, Signal, AirDrop. The recipient installs the CLI, points it at the zip, provides the key. No server needed.

This is a distribution model that bypasses the web entirely. The vault travels as a file. The key travels as a message. The CLI does the rest.
