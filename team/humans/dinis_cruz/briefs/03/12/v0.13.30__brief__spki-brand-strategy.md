## Summary

The domain `spki.ai` has been acquired as the future platform brand. This is not an urgent rebrand. It is a direction to implement organically as the product evolves beyond file transfer toward key-based encrypted infrastructure. No timelines, no migration deadlines. The brand moves when the product is ready.

---

## Why the Current Branding Has Limits

### SG/Send

"SG/Send" was the right name when the product was encrypted file transfer. It described the core action: you send a file. But the product has outgrown the verb.

Current problems with SG/Send:

**The name describes one feature, not the platform.** Vaults are now the centre of gravity. Lambda identity, document signing, pack architecture, headless CLI, encrypt-for-reader are all key-based operations that have nothing to do with "sending." The vault component (`<sg-vault>`) is the core data layer. The Alchemist creates investor packs via vaults. Deploy Service stores state in vaults. Claude communicates through vaults. "Send" is one thing you do with the infrastructure. It is not what the infrastructure is.

**The slash is a technical liability.** "SG/Send" breaks in URLs without encoding. It creates ambiguity: is it "SG Send", "SG/Send", "SGSend", "sgsend"? Every user, every developer, every search engine encounters this friction. The slash looks like a path separator, not a brand element.

**"SG" is opaque.** It stands for SGraph, which itself requires explanation. Two layers of indirection before the user understands the name.

### sgraph.ai

The current domain. "Semantic Graph" is an accurate description of the underlying data model but tells the user nothing about what the product does for them. It is a technical description, not a brand. Nobody searches for "semantic graph encrypted file transfer."

### sgsend

As a single word, "sgsend" is hard to parse visually. Where does "sg" end and "send" start? Native English speakers hesitate. Non-English speakers struggle more. It fails the "spell it over the phone" test.

## Why SPKI

### The Acronym

SPKI originally stood for Simple Public Key Infrastructure (RFC 2693, IETF). The RFC was written in the late 1990s as an alternative to X.509 PKI. Its philosophy: local identity, direct authorisation, no certificate authorities. It never won against X.509 in the enterprise. But its design principles map almost exactly to what we are building now.

For our purposes, the S can stand for Scalable or Secure. Or for nothing. The strongest brand play is to let SPKI be the word, not an expansion. Like AWS is just AWS.

### What SPKI Represents

The product is built on public and private key pairs as the fundamental primitive:

- Public key encrypts. Private key decrypts.
- Public key verifies signatures. Private key signs.
- The key pair is the identity. The key pair is the access control. The key pair is the trust boundary.

Everything we build uses this: file transfer (AES-256-GCM with key in URL fragment), vaults (encrypted blobs, key-holder is the authority), document signing (CLI PKI features), Lambda identity (functions sign their own logs), encrypt-for-reader (encrypt with the reader's public key, not for storage).

PKI (Public Key Infrastructure) is the closest established category. But traditional PKI carries baggage: certificate authorities, X.509 certificates, revocation lists, OCSP responders, enterprise complexity. We are building PKI without the pain. SPKI as an IETF concept was designed to be exactly that.

## The Internationalisation Advantage

This is a significant factor. The product already supports en-GB and pt-PT, with babel planned for translation management. The brand name needs to work across languages.

### Problems with English-verb brands

"Send" is an English verb. In Portuguese it is "Enviar." In German, "Senden." In French, "Envoyer." In Japanese, a completely different word and script. Any brand built on an English verb requires either:

- Leaving the English word untranslated (alienating non-English speakers)
- Translating the brand name per locale (fragmenting the brand)
- Explaining the brand name in every market

None of these are good options.

### Why acronyms travel

Acronyms are language-neutral. IBM, AWS, API, PKI, USB, PDF. Nobody translates them. They are adopted as-is across all languages and scripts. They work in right-to-left scripts. They work in logographic scripts. They work over the phone in any language.

SPKI is four letters. It is the same in English, Portuguese, German, Japanese, Arabic. The UI copy, documentation, and marketing materials get translated. The brand name does not. This means:

- One brand, every market
- Zero brand fragmentation
- The translation effort (babel.spki.ai) focuses on content, not on explaining the name
- Domain and app store listings are consistent globally

### Pronunciation

The one remaining question. Options: "spee-kye," "S-P-K-I," or just "spicky." Any of these can be established with consistent use. The important thing is that none of them collide with existing words in any major language, which avoids the worst i18n naming problem (where your brand name means something unfortunate in another language).

## The Market Positioning Shift

| Under SG/Send | Under SPKI |
|---|---|
| "Encrypted file transfer" | "Key-based encrypted infrastructure" |
| Feature company | Platform company |
| Competes with WeTransfer, Dropbox | Competes with nothing (new category) |
| Investor pitch: "secure Dropbox" | Investor pitch: "PKI for the AI age" |
| 40+ customers using "send" | 40+ customers on a platform that also sends |

"Encrypted file transfer" is a feature. Features get commoditised. "Scalable public key infrastructure with zero-knowledge encryption" is a category. Categories get funded.

The Alchemist investor pack workflow demonstrated this already: the investor pack was created by an AI agent, stored in an encrypted vault, accessed via a key. That is not a file transfer story. That is a key-based infrastructure story.

## Transition Approach

This is explicitly not urgent. The transition happens organically.

**Phase 0 (Now):** Domain acquired. Brand direction documented. No code changes, no migration.

**Phase 1 (When natural):** New documentation, new briefs, and new articles start using "SPKI" alongside "SG/Send." Dual branding. "SPKI (formerly SG/Send)" where needed.

**Phase 2 (When ready):** New sub-domains (vault.spki.ai, send.spki.ai) stand up alongside existing sgraph.ai domains. Redirects in place. Existing URLs continue to work.

**Phase 3 (When complete):** SPKI is the primary brand. SG/Send is a product name for the file transfer feature. sgraph.ai redirects to spki.ai.

No timelines on any phase. The product drives the pace, not a brand migration plan.
