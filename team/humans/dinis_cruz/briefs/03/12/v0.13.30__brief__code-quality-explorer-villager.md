# Code Quality Phases: The Explorer-to-Villager Methodology

**version** v0.13.30
**date** 12 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Architect, QA, AppSec, Designer
**type** Process brief

---

## Context

Projects are reaching maturity. The CLI is the first example: the Explorer team built it fast, organically, iterating in flow state. The code works. The features are there. But the code grew organically and now needs structured quality investment before the next wave of features (branch model, unified vault structure, remotes) lands on top of it.

This is the Explorer-to-Villager transition. The Explorer builds. The Villager improves. Then the Explorer builds again on a solid foundation.

This brief defines the phased methodology. It applies to the CLI now and to every project that reaches this inflection point.

---

## Phase 1: Architectural Review

**Who:** Architect, Developer, QA, AppSec, Sherpa, Designer

**What:** Fresh eyes on everything that exists. This is not about fixing. It is about understanding.

The review covers multiple angles:

| Angle | What to Assess |
|---|---|
| Code quality | Structure, naming, abstractions, duplication, complexity |
| Architecture | Is the code well-structured? Are responsibilities clear? Are boundaries clean? |
| Security | Are there secrets in the bare vault? Is sensitive data leaking into structures? Are there injection points? |
| Design | Is the vault well-designed as a product? Is the CLI well-designed as a developer experience? |
| Business | Does the architecture support the billable units? Does the vault structure support the branch model? |

**Output:** A findings document. Gaps identified. No code changes. No fixes. Just a clear map of what needs attention.

## Phase 2: Test Coverage

**Rule: No functionality changes. No behaviour changes. The only code written is test code and test infrastructure code.**

This phase has one job: build a safety net dense enough that Phase 3 (refactoring) can proceed with confidence.

### What Gets Tested

- Unit tests (individual functions and methods)
- Integration tests (components working together)
- End-to-end tests (full workflows: create vault, add file, commit, push, pull, clone)
- Use case coverage (user journeys, not just code paths)
- CI pipeline integration (all tests run automatically on every commit)

### Target

As close to 100% code coverage as practical, but coverage percentage is secondary to use case coverage. A project can have 100% line coverage and still miss critical workflows. The priority is: can a user do everything they need to do, and does every workflow have a test?

### Bugs as Passing Tests

When a bug is found, write a test that passes with the current (buggy) behaviour. The test documents the bug. The test confirms the bug exists. When the bug is fixed in Phase 3, the test is updated to expect the correct behaviour. This means: the test suite is always green. Bugs are tracked as test assertions, not as separate tickets that get lost.

Same for vulnerabilities: write a test that passes with the current (vulnerable) behaviour. The test documents the vulnerability. Fix it in Phase 3.

### No Mocks. No Patches.

**This is a hard rule.** No mocking. No patching. No faking.

If a test needs the SG/Send server, spin up a real instance. If a test needs a vault, create a real vault. If a test needs a user, create a real user. If a test needs an end-to-end flow, run the real end-to-end flow.

This means investing in test infrastructure:

- Test data generators (create vaults, files, users programmatically)
- Test harnesses (spin up a local send server instance for integration tests)
- Test environments (CI pipeline that boots the full stack)

The test infrastructure is first-class code. It gets the same quality, the same review, the same attention as the product code. The test infrastructure is the foundation that makes everything else safe.

### Why This Matters

The next batch of changes is big: unified vault structure, branch model, PKI per branch, remotes. These changes will touch every part of the codebase. Without comprehensive test coverage, every change is a gamble. With it, every change is verifiable.

## Phase 3: Refactoring

**Rule: All tests must pass after every change. If a test breaks, the change is wrong (or the test found a real problem).**

This phase is about improving the code without changing what it does:

- Move code to better locations (file structure, module boundaries)
- Create classes and abstraction layers
- Eliminate duplication
- Improve naming
- Clean up folder structure
- Separate concerns (encryption logic, storage logic, CLI interface, API client)

Every refactoring step is small. Commit after each step. Run tests after each commit. If tests pass, the refactoring is safe. If tests fail, either the refactoring introduced a bug (fix it) or the tests found a gap in the previous phase (good, fix the test).

### Adversarial Testing

A specific exercise during this phase: try to make code changes that alter the application's behaviour without any test failing.

- Change a default value. Does a test catch it?
- Remove a validation. Does a test catch it?
- Alter an encryption parameter. Does a test catch it?
- Change a state transition. Does a test catch it?

Every "successful" adversarial change (behaviour changed, no test failed) is a gap in test coverage. Write the missing test. Then revert the adversarial change. This is the AppSec team and QA team working together to stress-test the safety net.

The goal: reach a state where any code change that changes behaviour triggers at least one test failure. At that point, refactoring is safe and the codebase is ready for the next wave of features.

---

## Phase 4: Next Wave (Explorer Returns)

With the codebase clean, well-tested, and well-structured, the Explorer team picks it back up. The big features land:

- Unified vault structure (server-side and bare vault merged)
- Branch model (auto-branch on clone, PKI per branch)
- Signed commits
- Remotes
- Merge

Breaking changes are expected and welcome. The difference: every breaking change is caught by a test. Every breaking change is documented by the test that broke. The transition from version N to version N+1 is visible in the test diff.

---

## The Cycle

This is not a one-time process. It repeats:

```
Explorer builds (fast, organic, flow state)
    ↓
Villager reviews + tests + refactors (quality, safety net, structure)
    ↓
Explorer builds again (on solid foundation, with confidence)
    ↓
Villager reviews + tests + refactors (next quality pass)
    ↓
...
```

Each cycle raises the quality floor. Each Explorer phase can be more ambitious because the Villager phase made the foundation stronger.

---

## First Application: CLI Project

The CLI project is the first candidate. The vault mega-memo identified multiple large changes coming (branch model, unified vault structure, remotes). Before those land, the CLI needs:

1. Phase 1: Architectural review of current CLI codebase
2. Phase 2: Test coverage (all vault operations, all CLI commands, all workflows)
3. Phase 3: Refactoring (clean abstractions for the remote/branch/merge additions)
4. Phase 4: Branch model, unified vault, remotes implementation

| 5 | All known bugs captured as passing tests |
