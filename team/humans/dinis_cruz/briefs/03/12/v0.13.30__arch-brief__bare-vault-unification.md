# Bare Vault Structure Unification and Security Review

**version** v0.13.30
**date** 12 Mar 2026
**from** Human (project lead)
**to** Architect (lead), Developer, AppSec
**type** Architecture brief

---

## The Problem

There are currently two different data structures for the same vault:

1. **Server-side structure:** How the vault is stored on the SG/Send backend (S3 blobs, tree references, metadata).
2. **Clone-side (bare) structure:** How the vault is stored locally after a clone operation.

These two representations should be identical. They are not. This creates complexity, potential security gaps, and unnecessary transformation logic.

## Why Unification Matters

If the server-side and bare vault structures are the same, then:

- **Cloning is a zip download.** Clone becomes: download the server-side files as a zip. No transformation. No repackaging. The server computes the zip once and caches it. Subsequent clones are a single cached download, not N API calls.
- **One codebase.** The CLI reads and writes the same format regardless of whether it is operating on a server-hosted vault or a local folder. No format conversion layer.
- **Security review is one review.** Instead of auditing two data structures for leaked sensitive data, there is one structure to review.
- **Push is the reverse of clone.** Upload the changed files. No format translation.

## Security Audit Required

Both representations need a security review. The question: does either structure contain any unencrypted sensitive data?

**What should be present (unencrypted):**
- File IDs (content-addressed hashes)
- File sizes
- Tree structure (paths, directory hierarchy)
- Signature metadata (who signed, hash of signed content)
- Commit metadata (timestamp, message, branch reference)

**What must NOT be present (must be encrypted or absent):**
- File contents (must be encrypted blobs)
- File names (question: are file names in the tree encrypted or plaintext?)
- Any decryption keys or key derivation material
- Any user-identifiable information beyond what is in signatures

This review should produce a single architecture document that maps every field in both structures, flags any discrepancies, and recommends the unified format.

## Vaults Inside Vaults

A vault can contain another vault as a file. The inner vault is zipped and stored as an encrypted blob inside the outer vault. Each vault has its own decryption keys. The outer vault's key decrypts the zip. The inner vault's key decrypts its contents.

This enables nested access control: decrypt the outer vault to see the structure, but you need a separate key for the inner vault's contents. Useful for compartmentalised data where different teams have different access levels.

Implementation: store inner vaults as zip files. The CLI recognises `.vault.zip` (or similar convention) as a nested vault and can operate on it recursively.

## SQLite as Storage Backend

For larger vaults with many blobs, loading a directory of individual files may become slow. SQLite as an alternative local storage backend would:

- Allow indexed access to blobs without loading the full directory listing
- Support faster queries ("which files changed since commit X")
- Reduce filesystem overhead (one file instead of thousands)

This is not urgent. It is a performance optimisation for when vaults grow past the point where directory-of-files is efficient. Note it as a future option.

## Backwards Compatibility

**Not a concern right now.** The vault format is pre-launch. No backwards compatibility debt should be incurred at this stage. Once vaults are officially launched as a feature, add a version field to the vault format so future migrations can be handled. Until then, break freely and fix the structure.
