# Branch Model: PKI Per Branch and Multi-User Collaboration

**date** 12 Mar 2026
**from** Human (project lead)

## The Core Idea

Every clone creates a branch. Every branch has its own key pair. Every commit is always to a branch, never directly to main. Merges are the only way content moves from a branch to main.

This is Git's branching model, but with cryptographic identity baked into every branch. The branch is not just a pointer. It is an identity: a public/private key pair that ties the branch to a specific device, session, or user.

## Why This Solves Multi-User Collaboration

The current vault model has a concurrency problem: if two users push at the same time, one of them overwrites the other. Merge conflicts are messy.

With the branch model:

- User A clones the vault. A branch is auto-created for them (e.g., `branch-a`). Their device's key pair is associated with this branch.
- User B clones the vault. A branch is auto-created for them (e.g., `branch-b`). Different key pair.
- Both users commit freely to their own branches. There is never a conflict within a branch because only one entity writes to it.
- Pushes always succeed because each branch is unique to its creator.
- Merges are explicit. User A merges their branch into main. User B merges their branch into main. Conflicts only arise at merge time, not at commit time.

The key insight: **conflicts are never within a branch. Conflicts are always at merge boundaries.** This separates the "save my work" operation (commit + push, always safe) from the "integrate with others" operation (merge, may conflict).

## PKI Per Branch

Every branch has a public/private key pair:

| Element | Has Key Pair | Purpose |
|---|---|---|
| Vault | Yes | Root identity. Vault-level encryption and signing. |
| Branch | Yes | Branch identity. Ties branch to device/session/user. |
| Commit | Signed by branch key | Provenance. Which branch (and therefore which device/user) made this commit. |

This gives full traceability: every commit is signed, every signature traces to a branch, every branch traces to a device or session. You can answer "who changed this file, when, from which machine" cryptographically.

### Key Storage Question

Where do branch keys live?

| Location | Use Case |
|---|---|
| User's vault | Personal key management. Keys encrypted in the user's own vault. |
| Browser local storage | Browser-based vault editing. Keys persisted in the browser. |
| OS keychain | Desktop/mobile app. Keys managed by the OS. |
| Another vault | Vault-of-keys pattern. A meta-vault containing branch keys for all vaults a user accesses. |

The answer may be "all of the above" depending on context. The CLI should support pluggable key storage backends.

## Signed Commits

Every commit is signed by the branch key. The signature covers:

- The tree hash (state of all files at this commit)
- The parent commit hash (chain integrity)
- The commit message
- The timestamp
- The branch identity (public key)

This means the commit history is a signed chain. Tampering with any commit invalidates all subsequent signatures. This is stronger than Git's optional GPG signing because it is mandatory and built into the vault format, not bolted on.

### Self-Signed vs. Authority-Signed

Two modes:

- **Self-signed:** The branch key signs its own commits. Proves "the same entity that created this branch made this commit." No external authority.
- **Authority-signed:** The commit is also signed by a vault-level or user-level key. Proves "the vault owner approves this commit." Useful for controlled environments where not every branch should be able to merge without approval.

## Auto-Create on Clone

When a user clones a vault:

1. Clone the vault contents (all files, current tree)
2. Auto-create a new branch for this clone
3. Generate a key pair for the branch
4. Associate the branch with the current device/session
5. All subsequent commits go to this branch automatically

The user never thinks about branches for basic workflows. They clone, edit, commit, push. The branch is invisible until they need to merge or until they want to see who else is working on the vault.

## Merge Patterns

**Automatic merge:** When the branch only adds new files (no conflicts), merge into main automatically on push. This covers the majority of use cases (appending log entries, adding new documents, expanding a pack).

**Manual merge:** When the branch modifies existing files that have also been modified on main, flag the conflict. Present the diff. Let the user (or an AI agent) resolve it.

**Merge-only main:** The main branch is never written to directly. All changes come through merges from feature branches. This is the "dev branch" model: main is always a clean integration point.

## Visualisation

This branch model requires visualisation to be understandable. Priority:

- ASCII art in CLI (`sg-send-cli log --graph`)
- Mermaid diagrams (Mermaid has native Git graph support: `gitGraph` directive)
- Web component (`<sg-vault-graph>`) showing branches, commits, merges
- Multi-vault view: multiple vaults side by side in the same UI, showing where data flows

The visualisation should support:

- Branch timeline (who committed when)
- Merge history (which branches have been integrated)
- Conflict indicators (where merges failed or are pending)
- Drag-and-drop between vaults (move files from vault A to vault B)

## The Bigger Picture

This branch model is the warm-up. The same fractal pattern applies at smaller granularity:

- Today: branches at the vault level (files and folders)
- Next: branches at the document level (sections and paragraphs)
- Eventually: branches at the concept level (individual assertions, data points)

The vault becomes a graph database where every node has identity, every change has provenance, and every collaboration has explicit merge boundaries. This is the "lightweight encrypted Git" target. With branches, signed commits, and merge, the gap between "encrypted blob store" and "full version control system" is small.

## How Far Are We?

To claim "lightweight encrypted Git," the vault needs:

- ✅ Content-addressed storage (blob hashes)
- ✅ Tree structure (directory hierarchy)
- ✅ Commits (already shipped)
- ⬜ Branches (this brief)
- ⬜ Merge (this brief)
- ⬜ Signed commits (this brief)
- ⬜ Remotes (see companion CLI brief)
- ⬜ Diff (show what changed between commits)
- ⬜ Log (commit history with graph)

Four features to build. Two of them (branches and merge) are the big ones. The rest are incremental. This is close.
