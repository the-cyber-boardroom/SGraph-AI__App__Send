## Gallery Concept (gallery.sgraph.ai)

A new product surface for sharing collections of files (photos, documents, media) with public or private access.

At its core, a gallery is a vault displayed as a browsable collection. The underlying storage is always encrypted on SG/Send. The access mode determines who can view it:

**Public gallery:** The decryption key is embedded in the URL. Anyone with the link can view the contents. The server stores encrypted blobs. The browser decrypts on load. No password prompt needed.

URL structure: `gallery.sgraph.ai/view/{title-slug}/{decryption-hint}`

Because everything is public, the URL can include a human-readable title. This makes gallery URLs shareable and meaningful, unlike the current hash-based vault URLs.

**Private gallery:** The decryption key is not in the URL. The viewer must enter a passkey or have the private key installed. Same zero-knowledge model as existing SG/Send transfers.

**Key-gated gallery:** A gallery encrypted with a specific public key. Only the holder of the corresponding private key can decrypt. The gallery can be permanently closed by re-encrypting with a new key.

### Architecture

Gallery is a separate site (gallery.sgraph.ai) with its own API, but the data lives in the existing SG/Send encrypted storage. The gallery API is a presentation layer over the vault API. This means:

- No data duplication
- Same encryption pipeline
- Same billing (downloads consume credits)
- Gallery creation is just vault creation with a `_gallery.json` manifest (similar to the `_pack.json` concept from the vault UI components brief)

### Monetisation

Track and charge for:
- Storage (encrypted blobs in S3)
- Downloads (bandwidth per view)
- Gallery count (number of active galleries per user)

The "No Surprises" credit model applies. Gallery views consume credits. Heavy traffic galleries need more credits.

### Immediate Use Case

The human is in Portugal and can take photos. A gallery of location photos, published as a public gallery, is both a personal use case and a live demo of the technology. This becomes content for the Portuguese market.

## Public Key Registry (registry.sgraph.ai or pt.registry.spki.ai)

### The Idea

Create a public directory of public keys for Portugal. Scrape and collect public keys from:

- GitHub profiles of Portuguese developers
- Government websites (if any publish public keys)
- Company websites
- PGP keyservers (for Portuguese email addresses)

For each key, store:

- The public key itself
- Source URL (where it was found)
- Metadata (owner name, organisation, email if public)
- Timestamp (when it was indexed)
- SG/Send signature (we sign the record, providing our attestation of the source)

### Why This Is Powerful

1. **Public service.** A free, browsable directory of public keys. Useful to anyone who wants to communicate securely with a Portuguese entity.
2. **Brand presence.** Every key in the registry has an "Send secure message to this person" button that uses SG/Send to encrypt and deliver.
3. **Viral growth.** People discover their own keys in the registry. They share it. They tell colleagues. The registry becomes a reference point.
4. **Scalable to other countries.** Start with Portugal (small, manageable, personal connection). Expand to other markets.
5. **Content driver.** Every key in the registry is a potential article: "We found that [organisation] publishes a public key. Here's how to use it."

### Trust Model

The registry does not vouch for key ownership. It attests: "we found this public key at this URL on this date." The SG/Send signature on the record proves the registry entry has not been tampered with since indexing. Key verification remains the user's responsibility.
