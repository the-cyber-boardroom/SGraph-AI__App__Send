## Vault Website UX

The vault website needs a simple, direct UI:

- **Open vault** by ID. Paste or type a vault ID, enter key, view contents.
- **Create vault.** One-click vault creation. This should be prominent, not buried. Vault creation is the primary growth action.

Creation is the marketing moment. Every new vault is a new user committed to the platform. The website should make vault creation feel like the main thing you do here, not a secondary feature behind file transfer.

The vault website should move to a separate Git repo. Static pages. Independent deployment. The vault is becoming its own product surface, not a sub-page of the send site.

## Launch the Vault as a Feature

The vault concept needs a formal launch to existing users:

- Announce to all current users (40+) and early access contacts
- Marketing campaign focused on vault creation and use cases
- Position vaults as the core capability, with send as one way to use them
- Demo vaults on the website showing real use cases in action

## Demo Vaults

Create a set of demonstration vaults visible from the website:

| Demo Vault | Shows |
|---|---|
| Investor pack | Subfolders, structured documents, professional presentation |
| Signed documents | Document identity, signature verification |
| Shared workspace | Multi-file collaboration, commit history |
| Photo gallery | Media files, public/private access modes |
| Case study | Real-world encrypted data exchange scenario |

Each demo vault should be interactive: the user can browse the tree, see file metadata, understand the structure. The demo vaults are both marketing assets and functional demonstrations of the technology.

## Billable Units

Vault operations become billable units alongside the existing credit system:

| Unit | Description | Suggested Tier |
|---|---|---|
| Vault count | Number of active vaults | 10 free, 20 for 5 GBP |
| Pulls | Clone/pull operations | Metered |
| Pushes | Push operations | Metered |
| Commits | Individual commits | Metered |
| Branches | Active branches per vault | Metered |
| Storage | Total encrypted blob storage | Per GB |

The pricing model follows the existing "No Surprises" principle: prepaid credits, hard caps, no unexpected bills. Vault operations consume credits just like file transfers do.

## Usage Tracking

Current gap: we don't have visibility into vault usage. Options for tracking:

- S3 server access logs (low effort, captures read/write at the storage level)
- CloudFront logs (if vault API is behind CloudFront)
- Application-level logging in the vault API (most detailed, most effort)

Priority: get any tracking in place first. S3 server logs are the lowest friction starting point. Refine later.

Also: support bare vault download on clone so that repeated clones don't generate unnecessary API requests. Download the full vault once, then work locally.
