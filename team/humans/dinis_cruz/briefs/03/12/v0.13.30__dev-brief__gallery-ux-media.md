# Gallery UX: Media Features, Export, and Presentation

**version** v0.13.30
**date** 12 Mar 2026
**from** Human (project lead)
**to** Developer (lead), Designer, Ambassador
**type** Dev brief

---

## Context

The gallery concept (see companion Portugal strategy brief) needs a detailed UX specification. The gallery is the first product surface that is explicitly designed to look beautiful. File transfer is functional. The vault is structural. The gallery is visual. It has to feel good.

## Core Workflow

1. User drops a folder of photos (or files)
2. User adds a title
3. User chooses: public (link with key) or private (passkey required)
4. Gallery is created, encrypted, stored
5. User gets a shareable link
6. User can screenshot the gallery and share the screenshot as a preview

This should be fast. Drop, title, share. Three actions.

## Media Features

### Thumbnails

Generate thumbnails automatically on upload. For images, create a small preview (200px wide). Thumbnails should be:

- Generated client-side (browser, no server processing of unencrypted images)
- Stored as separate encrypted blobs in the vault (not inline in the tree)
- Loaded first when browsing the gallery (progressive loading: thumbnails appear instantly, full images load on click)

### Slideshow

A built-in slideshow mode. Click play, images cycle with transitions. Simple. No complex configuration. This is a demonstration of what the gallery can do, and it is also a genuine feature for sharing photo sets.

### LLM Integration (Future)

Once the self-hosted Ollama instance is running (LLM Service, v0.13.12):

- Auto-caption images (describe what is in the photo)
- Auto-tag (categorise images by content)
- Search by description ("show me the sunset photos")
- Basic editing suggestions

This is not for the MVP. Note it as the next phase after the gallery ships.

### Basic Video Editing (Future)

The existing plan for basic video editing capabilities applies here. Short clips, trim, stitch. Not a full editor. Enough to assemble a highlights reel from video files in the gallery. Also not MVP.

## Export and Interoperability

The gallery should not be a walled garden. Users need to get their files out.

### Export Options

| Destination | Method | Complexity |
|---|---|---|
| Download all as zip | One-click download, browser decrypts and zips | Low |
| Download individual files | Click to download single file | Low |
| Google Drive | OAuth flow, upload decrypted files to Drive | Medium |
| Google Photos | OAuth flow, upload to Photos library | Medium |
| Dropbox | OAuth flow or manual upload instructions | Medium |
| Email | Attach files (small galleries) or share link | Low |

For the MVP: download all as zip and individual file download. For the OAuth integrations (Google Drive, Photos, Dropbox), start with clear instructions ("here's how to move these files to Google Photos") and build the integrations when demand justifies it.

The principle: always err on the side of what is better for the user. If they want their files somewhere else, make it easy. Don't trap them.

## Visual Design

The gallery has to be beautiful. This is the first SG/Send (or SPKI) surface that a non-technical user might see. It is a marketing asset as much as a product feature.

Requirements:

- Clean grid layout for thumbnails
- Full-screen image viewer on click
- Smooth transitions between images
- Responsive (works on phone, tablet, desktop)
- Minimal chrome (the photos are the content, the UI gets out of the way)
- Gallery title displayed prominently
- Easy screenshot (the gallery should look good when screenshotted and shared on social media or messaging)

## Immediate Test Case

The human is in the Algarve. A gallery of Algarve photos, published as a public gallery, serves as:

- First live gallery on the platform
- Portuguese market content
- Visual proof that the technology works for real use cases
- Social media shareable demo
