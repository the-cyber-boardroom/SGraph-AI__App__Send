# The Cryptographer Who Outsmarted Himself (And Why That's Exactly What We Needed)

**A Journalist's Field Report from the SGraph Send Beta Trenches**
**v0.13.0 — 8 March 2026**

---

## Chapter 1: In Which a Power User Does Something Perfectly Logical and Absolutely Devastating

It started, as the best bug reports always do, with a reasonable idea.

Pedro had successfully sent a file through SGraph Send in Browser A. The upload went smoothly. Encryption happened. A download link appeared. Life was good.

Then he opened Browser B.

The app asked for his access token. Pedro didn't have it written down. It wasn't in this browser. He paused. He thought. And then — with the quiet confidence of someone who has reverse-engineered worse — he thought: *"The download URL. The token must be in there somewhere."*

Reader, it was not.

---

## Chapter 2: In Which an Encryption Key Goes on a Journey It Was Never Meant to Take

The SGraph Send download URL looks something like this:

```
https://send.sgraph.ai/en-gb/download/#<transfer_id>/<base64_encoded_something>
```

That "base64 encoded something" is not the access token. It is the **AES-256-GCM encryption key** — 32 bytes of raw cryptographic entropy, base64-encoded so it can safely live in a URL fragment.

Pedro decoded it.

What came out the other side was a string of binary chaos: unprintable characters, control codes, and — crucially — the Unicode character `ɸ` (U+0278, Latin Small Letter Phi With Stroke). A character that looks perfectly innocent sitting in a linguistics textbook, but which is about to ruin everyone's afternoon.

Pedro pasted this decoded binary into the token input field and hit Enter.

The app said: *"Looks good."*

The app was wrong.

---

## Chapter 3: In Which the Browser Draws a Line in the Sand

Here's what happened next, in slow motion:

1. The UI accepted the binary string without complaint — no format validation
2. `checkToken()` encoded it with `encodeURIComponent()`, which percent-escapes everything exotic, so the network call succeeded
3. The corrupted binary was lovingly saved to `localStorage`
4. Pedro clicked "Upload"
5. The app read the "token" from storage and attempted to set it as an HTTP header
6. The browser's fetch API checked the header value and discovered `ɸ`

And then the browser, enforcing a standard written in 1999, threw this:

> *Failed to execute 'fetch' on 'Window': Failed to read the 'headers' property from 'RequestInit': String contains non ISO-8859-1 code point.*

Every. Single. Upload. This error. Every time. No explanation. No recovery button. No "try entering your token again." Just a cryptic message about a character encoding standard that most developers have to Google.

Pedro was stuck. The product was broken. And the corrupted token sat in `localStorage` like a land mine, waiting to detonate on every page load.

---

## Chapter 4: In Which We Count the Dominoes

Let's appreciate the chain of events that had to occur for this bug to manifest:

1. User switches browsers (localStorage is per-browser — who knew? Well, everyone, but also no one when they're in the moment)
2. User attempts to recover token from download URL (creative! wrong! creative!)
3. User base64-decodes an AES-256-GCM key (technically proficient! catastrophically misguided!)
4. Token input has zero format validation (our fault)
5. Token check endpoint doesn't reject binary strings (also our fault)
6. Corrupted value is stored to localStorage without validation (still our fault)
7. HTTP header construction doesn't validate the value (you see where this is going)
8. Error handler shows a raw browser exception with no recovery path (sorry, Pedro)

**Eight dominoes.** Any one of them not falling would have prevented the incident. All eight fell. That's not bad luck — that's a test scenario that no QA engineer would ever have written, because no QA engineer would ever think to decode an AES key and paste it as an access token.

But Pedro did. Because power users don't read your mental model of how the product works. They have their own.

## Chapter 5: A Brief Taxonomy of Power User Creativity

Pedro's adventure is a textbook example of a phenomenon every software team discovers during beta: **technical users will find paths through your product that you didn't know existed.** Not because they're trying to break things, but because they're trying to *make things work.*

Here are some classic archetypes from the annals of beta testing:

**The Reverse Engineer** (Pedro's genus)
*"I can figure out how this works by looking at what it produces."* They'll decode URLs, inspect network traffic, read your minified JavaScript, and cheerfully paste binary data into text fields. They're not attacking your system. They're complimenting it by assuming it's robust enough to handle their curiosity.

**The Workflow Inventor**
*"Your UI has 5 steps but I only need 2."* They'll bookmark deep links, copy tokens between devices via notes apps, share download URLs through channels you never considered, and compose your API endpoints into shell scripts before you've written the documentation.

**The Edge Case Artist**
*"What happens if I upload a 0-byte file? A 4GB file? A file named `../../../etc/passwd`? A file with emoji in the name? All five simultaneously?"* They don't do this to be difficult. They do it because their brain pattern-matches on boundaries, and boundaries are where bugs live.

**The Concurrency Explorer**
*"What if I open three tabs and upload from all of them at the same time?"* They'll race-condition your backend before you've written your first mutex. They'll find the session conflicts you didn't know you had.

**The Helpful Documenter**
*"Your error message says 'something went wrong' but here's the stack trace, the network log, my OS version, the phase of the moon, and a suggested fix."* These people are a gift. Treasure them.

## Chapter 6: Why MVPs Need Friendlies (Not Just Friends)

There's a specific window in a product's life — after "it works on my machine" and before "it works for 10,000 strangers" — where you need a very particular kind of tester. Not your colleagues, who share your mental model. Not random users, who'll bounce at the first confusion. You need **friendlies**: people who are technically capable, genuinely invested in your success, and willing to tell you what they actually did rather than what they think you want to hear.

Pedro didn't just find a bug. He found a **class of vulnerability** — the entire category of "what happens when non-ASCII data enters the token path." His single incident led to four fixes across three files, a new UX pattern for token recovery, and a defence-in-depth strategy that protects against corrupted state from any source.

One friendly tester. One creative workaround. Four layers of defence we wouldn't have built otherwise.

This is the economics of MVP testing:

| Stage | Tester type | Cost of finding a bug | Typical bug severity |
|-------|-----------|---------------------|---------------------|
| **Dev** | Engineer | Low | Low (you catch the obvious ones) |
| **Friendly beta** | Technical allies | Medium | **High** (they find the creative ones) |
| **Public beta** | Early adopters | High | Medium (they find the common ones) |
| **Production** | Everyone | Very high | Variable (and now it's an incident) |

The Pedro Bug found at the friendly-beta stage cost us an afternoon of engineering. Found in production, it would have been a support ticket, a frustrated customer, a potential churn event, and — because the error was so cryptic — probably a negative review.

## Chapter 7: The One-Sentence Fix That Would Have Prevented Everything

Here's the punchline.

The bug required four fixes to properly defend against. Input validation. Header sanitisation. Error catching. Token auto-recovery. Good engineering. Defence in depth. The works.

But you know what would have prevented the entire incident?

One line of text on the token entry form:

> *"Your token is saved in this browser only. If you switch browsers or devices, you'll need to enter it again."*

That's it. 18 words. If Pedro had seen that hint, he would have known why his token wasn't in Browser B. He wouldn't have gone spelunking in the download URL. The AES key would have stayed base64-encoded. The `ɸ` would have remained happily theoretical.

Eighteen words versus eight falling dominoes.

This is why beta testing with friendlies matters. Not just because they find the bugs, but because they reveal **where a single sentence of explanation would have prevented the bug from being findable in the first place.**

## Epilogue: A Thank You to the Breakers

To Pedro, and to every friendly tester who takes a product for a spin and comes back with something the team never expected: thank you.

You don't just find bugs. You find the gaps between what the builder imagined and what the user actually does. You find the assumptions we didn't know we'd made. You find the error messages we never thought anyone would trigger.

Your curiosity makes the product better. Your willingness to report what happened — honestly, in detail, without embarrassment — makes the fixes possible.

And sometimes, when you base64-decode an AES-256-GCM encryption key and paste it into a text field, you accidentally teach an entire team about ISO-8859-1 header validation.

That's worth more than a thousand automated tests.
