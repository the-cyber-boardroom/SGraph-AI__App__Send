## What We Fixed (4 Layers of Defence)

Here's a technical summary of the changes, since we know you'll want the details:

### Fix 1 — Token Format Validation at Input (Primary)
**File:** `send-access-gate.js`

The token input field now validates against `/^[a-z0-9_-]+$/` before anything else happens. If you paste in a base64-decoded binary string (or anything containing unicode, spaces, or special characters), the UI immediately rejects it with a clear message:

> *"That doesn't look like a valid token. Tokens contain only letters, numbers, hyphens and underscores."*

This single check would have prevented the entire incident.

### Fix 2 — Defence-in-Depth in `_authHeaders()` (Safety Layer)
**File:** `api-client.js`

Even if a corrupted token somehow makes it into `localStorage`, the API client now validates it before constructing HTTP headers. If validation fails, it:
- Clears the corrupted token from `localStorage`
- Fires an `access-token-invalid` event
- Automatically shows the token re-entry form so you can recover

### Fix 3 — Catch ISO-8859-1 Error in Upload (Safety Net)
**File:** `send-upload.js`

As a final catch, the upload handler now explicitly traps the browser's `ISO-8859-1` fetch error. If it fires, the corrupted token is cleared and recovery is triggered — no more raw browser exceptions.

### Fix 4 — Browser-Scope Hint on Token Form (UX Prevention)
**File:** `send-access-gate.js` (template) + `i18n.js`

The token entry form now shows a hint:

> *"Your token is saved in this browser only. If you switch browsers or devices, you'll need to enter it again."*

This addresses the root cause — understanding that the token from Browser A isn't available in Browser B, so there's no reason to try extracting it from a download URL.
