# Observation: Vault UI — Settings Panel, Navigation, and Loading Feedback

**Version:** v0.10.33
**Date:** 2026-03-03
**Source:** Human feedback on vault settings screenshot
**Status:** Design direction — approved by Dinis

---

## Context

The vault settings UI has improved significantly. The current implementation renders configuration (token input, statistics, raw JSON) in what appears to be a modal/overlay dialog. Three design improvements have been identified.

---

## 1. Settings Should Be a Top-Level Panel, Not a Modal

**Current:** Vault settings opens as a modal dialog (overlay with close button).

**Target:** Settings should be a full panel — the same level as the file browser, upload, and other primary views. Users should navigate *to* settings, not *open* settings on top of something else.

**Rationale:** Settings is a destination, not a momentary interruption. It contains configuration (token, endpoint), statistics, and diagnostic info that users may want to read alongside other panels. A modal forces a single-task context and blocks interaction with the rest of the UI.

---

## 2. Add Left-Side Navigation (Like Admin UI)

**Current:** The vault UI has no persistent navigation. Panels are accessed through inline buttons or implicit state.

**Target:** Add a left sidebar navigation matching the admin UI pattern — fixed sidebar with section headers, nav items with icons, and active-state highlighting (left border accent).

This sidebar will link to:
- **Vault** (file browser — the current default view)
- **Settings** (the settings panel, promoted from modal)
- Additional top-level panels/views that will be added soon

**Design reference:** The admin UI's `admin-shell.js` has the established pattern — `<nav class="as-sidebar">` with section groups, icon + label items, and `as-nav-item--active` state.

**Note:** Keep the sidebar minimal to start — just the panels that exist. Don't pre-populate with empty sections. Add items as panels ship.

---

## 3. Design Guideline: Avoid Modal Windows

**Guideline:** Prefer panels and inline UI over modals. Only use modals when the action is:
- **Destructive** and needs explicit confirmation (delete, revoke)
- **Blocking** — the user genuinely cannot proceed without completing this step (key generation, first-time setup)
- **Ephemeral** — a quick input that doesn't need its own panel (rename inline, paste a key)

**Do not use modals for:**
- Settings or configuration (use a panel)
- Viewing data or statistics (use a panel)
- File previews (use a split view or dedicated viewer)
- Multi-step workflows (use a panel with steps)

**Rationale:** Modals break spatial context — the user loses sight of where they were. They create accessibility issues (focus trapping, scroll locking). And they don't compose — you can't have two modals open or compare modal content with the page behind it. Panels preserve context and allow side-by-side work.

---

## 4. Add Visual Loading Indicators for Async Requests

**Problem:** When the user clicks an action (e.g. loading vault, saving settings, fetching files), there is no visual feedback if the response is slow. This is especially noticeable on Lambda cold starts (which can take 1-3 seconds). The user clicks, nothing happens, and they don't know if the click registered.

**Target:** Every user-initiated async action must show immediate visual feedback:

| Pattern | When to Use | Example |
|---|---|---|
| **Button spinner** | Button triggers an API call | Button text changes to spinner + "Loading..." and becomes disabled |
| **Inline skeleton** | Loading a list or panel content | Grey shimmer placeholders where content will appear |
| **Progress bar** | Upload/download with known size | Determinate bar with percentage |
| **Toast/status line** | Background operation completes | "Settings saved" or "Connection failed" appears briefly |

**Minimum requirement:** When a button triggers an API call, it must:
1. **Immediately** show a spinner or change state (within 1 frame — no waiting for the response)
2. **Disable** the button to prevent double-clicks
3. **Restore** the button on success or show an error state on failure

**Cold start mitigation:** If a response takes >500ms, show a subtle message like "Connecting..." to reassure the user that the request is in progress. This is especially important for first-time Lambda invocations.

**Existing pattern:** The admin UI already has `.admin-spinner` (16px CSS spinner) and `.admin-loading` (centered spinner + text). The vault upload component uses status icons (hourglass for active, checkmark for done). Reuse these established patterns.
