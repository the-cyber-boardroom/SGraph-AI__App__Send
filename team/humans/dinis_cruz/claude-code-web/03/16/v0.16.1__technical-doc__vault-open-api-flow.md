# Vault Open — API Call Flow (Technical Reference)

**Version:** v0.16.1
**Date:** 2026-03-16
**Purpose:** Document the exact sequence of API calls made when opening a vault, for CLI dev team debugging.

---

## Problem Observed

Opening vault key `ditg7au7rr0rsw4ont6qlzcw:98a3heec` on `dev.send.sgraph.ai` produces:
- `GET /api/vault/read/98a3heec/112c1dad1a72` → **404 Not Found**
- `GET /api/vault/read/98a3heec/115a772efd9e` → **404 Not Found**
- UI shows: "Vault not found. The link may have expired."

---

## Vault Key Format

```
{passphrase}:{vault_id}
```

Example: `ditg7au7rr0rsw4ont6qlzcw:98a3heec`
- **passphrase:** `ditg7au7rr0rsw4ont6qlzcw`
- **vault_id:** `98a3heec` (8 hex chars, generated at vault creation time)

---

## Step-by-Step Open Flow

### Step 1: Parse Vault Key

**File:** `sg-vault-crypto.js` → `SGVaultCrypto.parseVaultKey()`

```
Input:  "ditg7au7rr0rsw4ont6qlzcw:98a3heec"
Output: { passphrase: "ditg7au7rr0rsw4ont6qlzcw", vaultId: "98a3heec" }
```

Splits on `:` — last segment is `vault_id`, everything before is `passphrase` (passphrase may contain colons).

### Step 2: Derive Keys (PBKDF2 + HMAC)

**File:** `sg-vault-crypto.js` → `SGVaultCrypto.deriveKeys(passphrase, vaultId)`

From the passphrase + vault_id, four values are derived **deterministically**:

| Value | Derivation | Purpose |
|-------|-----------|---------|
| `readKey` | PBKDF2 (600k iterations, salt: `sg-vault-v1:{vault_id}`) → AES-256-GCM CryptoKey | Encrypt/decrypt all vault content |
| `writeKey` | PBKDF2 (600k iterations, salt: `sg-vault-v1:write:{vault_id}`) → hex string | Server-side write authorization (sent as `x-sgraph-vault-write-key` header) |
| `refFileId` | HMAC-SHA256(readKey, `sg-vault-v1:file-id:ref:{vault_id}`) → first 12 hex chars | Deterministic file ID for the HEAD ref pointer |
| `branchIndexFileId` | HMAC-SHA256(readKey, `sg-vault-v1:file-id:branch-index:{vault_id}`) → first 12 hex chars | Deterministic file ID for the branch index |

For this vault key, the derived IDs are:
- `refFileId` = `112c1dad1a72`
- `branchIndexFileId` = `115a772efd9e` (likely — the second 404'd request)

### Step 3: Read HEAD Ref

**File:** `sg-vault-ref-manager.js` → `SGVaultRefManager.readRef(refFileId)`

```
HTTP Request:
  GET https://dev.send.sgraph.ai/api/vault/read/98a3heec/112c1dad1a72
  No auth headers (vault reads are public — zero-knowledge design)

Expected Response (200):
  Binary: AES-256-GCM ciphertext containing JSON: { "commit_id": "obj-xxxxxxxxxxxx" }

Actual Response: 404 → null → "Vault not found: HEAD ref missing"
```

The ref file is a small encrypted blob. When decrypted with `readKey`, it contains a JSON pointer to the latest commit object ID.

### Step 4: Load Commit (skipped — step 3 failed)

If step 3 had succeeded, it would return a `commitId` (e.g., `obj-a1b2c3d4e5f6`), then:

```
GET /api/vault/read/98a3heec/obj-a1b2c3d4e5f6
```

The commit object is encrypted JSON:
```json
{
  "schema": "commit_v1",
  "parents": [],
  "tree_id": "obj-xxxxxxxxxxxx",
  "timestamp_ms": 1710000000000,
  "message": "Initial vault creation",
  "branch_id": "main",
  "signature": null
}
```

### Step 5: Load Tree (skipped — step 3 failed)

Would fetch `tree_id` from the commit:
```
GET /api/vault/read/98a3heec/obj-xxxxxxxxxxxx
```

The tree contains encrypted entries listing all files and their blob IDs.

### Step 6: Load File Blobs (skipped — step 3 failed)

Each file in the tree has a `blob_id`. To read a file:
```
GET /api/vault/read/98a3heec/{blob_id}
```

## Complete API Call Chain for a Successful Open

```
1. GET /api/vault/read/{vault_id}/{refFileId}           → HEAD ref (encrypted commit pointer)
2. GET /api/vault/read/{vault_id}/{commitObjectId}       → Commit object (encrypted JSON)
3. GET /api/vault/read/{vault_id}/{treeObjectId}         → Tree object (encrypted file listing)
4. GET /api/vault/read/{vault_id}/{blobId}               → Per-file blob (N calls, one per file)
```

All calls use the same endpoint pattern: `GET /api/vault/read/{vault_id}/{file_id}`

No authentication headers are required for reads (zero-knowledge design — the data is encrypted, only someone with the passphrase can decrypt it).

## Backend Route & Storage

**Route:** `Routes__Vault__Pointer.py`
```python
@route_path('/read/{vault_id}/{file_id:path}')
def read__vault_id__file_id(self, vault_id, file_id) -> Response:
    payload = self.vault_service.read(vault_id=str(vault_id), file_id=str(file_id))
    if payload is None:
        raise HTTPException(status_code=404, detail='Vault file not found')
    return Response(content=payload, media_type='application/octet-stream')
```

**Storage path:** `Service__Vault__Pointer.py`
```python
def vault_payload_path(self, vault_id, file_id):
    return f'transfers/vault/{vault_id}/{file_id}/payload'
```

So the first read looks for:
```
transfers/vault/98a3heec/112c1dad1a72/payload
```

If this path doesn't exist in the storage backend (Memory-FS), you get 404.

## Write Flow (for context)

When creating a vault, the same file IDs are used for writes:

```
PUT /api/vault/write/{vault_id}/{file_id}
Headers:
  x-sgraph-access-token: {access_token}
  x-sgraph-vault-write-key: {writeKey}
  Content-Type: application/octet-stream
Body: encrypted binary data
```

Write order during vault creation:
1. Store settings blob → `PUT /api/vault/write/98a3heec/obj-{hash1}`
2. Store tree object → `PUT /api/vault/write/98a3heec/obj-{hash2}`
3. Store commit object → `PUT /api/vault/write/98a3heec/obj-{hash3}`
4. Write HEAD ref → `PUT /api/vault/write/98a3heec/112c1dad1a72`

Writes require both the `x-sgraph-access-token` and `x-sgraph-vault-write-key` headers.

---

## Possible Root Causes for 404

1. **Storage backend is in-memory:** If the Lambda running `dev.send.sgraph.ai` restarted since the vault was created, all in-memory data is gone.
2. **Different environment:** Vault was created on `send.sgraph.ai` (prod) but being accessed on `dev.send.sgraph.ai` — each environment has independent storage.
3. **Storage backend mismatch:** The vault data was written to a different storage backend (S3, disk) than what the current deployment reads from.
