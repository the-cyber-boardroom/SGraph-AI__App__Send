## 1. Mission

Build `SGraph-AI__Vault` as a standalone encrypted vault library and CLI tool. The vault is the core data primitive for the entire SPKI platform. It provides: content-addressed encrypted storage, Git-like version control (commits, branches, merge), PKI-based identity (signed commits, document signing), and multi-remote sync.

The vault must work **completely standalone** — no server dependency for local operations. SG/Send (`send.sgraph.ai`) is one remote backend, not a requirement.

## 3. What Needs Building (Priority Order)

### Phase 1: Core Library (Foundation)

**Goal:** A Python library (`sgraph-vault`) that implements the vault data model independently of any HTTP server.

```python
from sgraph_vault import Vault

# Create vault
vault = Vault.create(path="/tmp/my-vault")

# Add files
vault.add("README.md", b"Hello world")
vault.add("docs/spec.md", b"Specification content")

# Commit
vault.commit("Initial commit")

# Encrypt
vault.encrypt(key=b"...")

# Export
vault.export_zip("/tmp/my-vault.zip")
```

**Data model:**
```
Vault/
├── .vault/
│   ├── config.json          # Vault metadata, version, remotes
│   ├── HEAD                 # Current branch reference
│   ├── refs/
│   │   └── heads/
│   │       └── main         # Branch pointer (commit hash)
│   ├── objects/
│   │   ├── blobs/           # Content-addressed encrypted blobs
│   │   ├── trees/           # Tree objects (path → blob mapping)
│   │   └── commits/         # Commit objects (tree + parent + message + signature)
│   └── keys/
│       ├── branch.pub       # Branch public key
│       └── branch.key       # Branch private key (encrypted)
├── README.md                # Working tree (decrypted files)
└── docs/
    └── spec.md
```

**Key design rules:**
- `Type_Safe` from `osbot-utils` for all schemas (never Pydantic)
- Content-addressed storage: blob ID = SHA-256 of encrypted content
- Tree = JSON mapping of `{path: blob_id}`
- Commit = `{tree_hash, parent_hash, message, timestamp, branch_pubkey, signature}`
- All operations work on local filesystem first. Remote sync is separate.

### Phase 2: Branch Model + Signed Commits

**Goal:** Every clone creates a branch. Every commit is signed.

```python
# Clone creates a branch automatically
vault = Vault.clone(remote="https://send.sgraph.ai", vault_id="abc123", key=b"...")
# vault is now on branch "clone-{random}" with its own key pair

# Commits are automatically signed by branch key
vault.add("new-file.md", b"content")
vault.commit("Add new file")  # Signed by branch key

# Push always succeeds (branch is unique)
vault.push()

# Merge is explicit
vault.merge("main")  # May conflict
```

**Branch rules:**
- Clone = auto-create branch + generate key pair
- Push always succeeds (each branch is unique to its creator)
- Merge is the only way content moves to main
- Auto-merge for non-conflicting changes (append-only)
- Conflict detection + reporting for overlapping changes

**Signed commits:**
- Every commit signed by branch private key
- Signature covers: tree hash, parent hash, message, timestamp, branch identity
- Two modes: self-signed (default) and authority-signed (vault owner co-signs)

### Phase 3: Remotes

**Goal:** Git-like remote model. Multiple backends.

```python
vault.remote_add("origin", "https://send.sgraph.ai")
vault.remote_add("backup", "/mnt/external/vaults")
vault.remote_add("s3", "s3://my-bucket/vaults")

vault.push("origin")
vault.pull("backup")
```

**Remote types to implement:**
1. **SG/Send API** — HTTPS REST calls to send.sgraph.ai (exists)
2. **Local folder** — direct filesystem read/write
3. **Zip file** — import/export as distributable archive
4. **S3 bucket** — direct S3 read/write via osbot-aws (never boto3 directly)
5. **URL** — download-only remote (for vault bundles)

**Abstraction layer:**
```python
class Remote:
    def push(self, objects: list[VaultObject]) -> None: ...
    def pull(self, refs: list[str]) -> list[VaultObject]: ...
    def list_refs(self) -> dict[str, str]: ...
```

### Phase 4: CLI Upgrade

**Goal:** `sg-send-cli` becomes `sg-vault` (or vault commands added to existing CLI).


## 4. Architecture Constraints

### Non-Negotiable

| Rule | Why |
|------|-----|
| Server never sees plaintext | Zero-knowledge model. Encryption is client-side only. |
| `Type_Safe` for all schemas | Project standard. Never Pydantic. |
| `osbot-aws` for all AWS calls | Project standard. Never boto3 directly. |
| No mocks in tests | Real implementations, in-memory backends. |
| Content-addressed storage | Blob ID = hash of content. Enables deduplication and integrity. |
| Offline-first | All vault operations work without network. Remote sync is optional. |

### Vault Format

| Field | Encrypted? | Notes |
|-------|-----------|-------|
| Blob content | YES (AES-256-GCM) | Always encrypted. Server stores ciphertext only. |
| File names/paths | Decision needed | Currently in tree as plaintext. Security audit required. |
| Tree structure | NO (metadata) | Maps paths to blob IDs. Visible to server. |
| Commit metadata | NO (metadata) | Timestamp, message, parent, branch. Visible. |
| Signatures | NO (public data) | Branch key signature on commits. Visible. |
| Blob sizes | NO (metadata) | Visible. Could leak information about content type. |

**Open question:** Should file names in the tree be encrypted? This is a security/usability trade-off. Encrypted names = stronger privacy but no server-side search. Plaintext names = server can see structure but not content.

### Encryption

| Algorithm | Use |
|-----------|-----|
| AES-256-GCM | Blob encryption (symmetric, fast) |
| ECDSA P-256 or Ed25519 | Commit signing (branch keys) |
| ECDH or RSA-OAEP | Encrypt-for-reader (asymmetric key exchange) |

**Decision needed:** Ed25519 vs ECDSA P-256 for signing. Ed25519 is faster and simpler but has less Web Crypto API support. ECDSA P-256 works natively in browsers. The Architect review at `team/roles/architect/reviews/03/11/v0.13.19__guidance__cli-pki-algorithm-decisions.md` has analysis.

## 11. Open Decisions (For Human)

| # | Decision | Options | Recommendation |
|---|----------|---------|----------------|
| 1 | Signing algorithm | Ed25519 (fast, simple, less browser support) vs ECDSA P-256 (native Web Crypto) | ECDSA P-256 for browser compat, Ed25519 for CLI/server |
| 2 | File names in tree | Encrypted (stronger privacy) vs plaintext (server can index) | Start plaintext, add encryption option later |
| 3 | Package name | `sgraph-vault` vs `sgraph-ai-vault` vs `spki-vault` | `sgraph-vault` (consistent with existing `sg-send-cli`) |
| 4 | CLI command name | `sg-vault` vs extend `sg-send-cli` | `sg-vault` (standalone, matches repo separation) |
| 5 | Vault format version | Start at v1.0 or v0.1 | v0.1 (pre-launch, breaking changes expected) |
| 6 | Branch key storage | File-based vs OS keychain vs vault-of-keys | File-based first, pluggable interface |
