## Brief 1: Sherpa Inbox Prompt Injection Testing

**Source:** `team/humans/dinis_cruz/briefs/02/27/part-2/v0.7.4__appsec-brief__sherpa-inbox-prompt-injection-testing.md`

### Summary

This AppSec brief defines a structured penetration testing plan for the Sherpa's email processing workflow. It maps the full attack surface across three phases (inbox listing, email reading, and guardrail/classifier agents), provides six categories of adversarial payloads (direct instruction override, data exfiltration, action injection, hidden/encoded payloads, multi-stage injection, and classifier evasion), and specifies a three-phase execution protocol (baseline testing, with-guardrails testing, and live validation). The goal is to answer three questions: do we have a prompt injection problem, what is the attack surface, and what protections exist versus what is missing.

### Key Asks

1. Set up a dedicated test mailbox (sherpa-test@sgraph.ai) wired to the same pipeline as production.
2. Create a payload library with at least 18 adversarial payloads (3 per category, 6 categories).
3. Execute Phase 1 baseline testing (no guardrails) and record structured results.
4. Produce a vulnerability report with severity ratings per finding.
5. Prioritise mitigations (P0/P1/P2) based on results.
6. Execute Phase 2 testing after guardrails are implemented.
7. Execute Phase 3 live validation with 3-5 payloads against the production inbox.
8. Store all results in a security data room accessible to AppSec and GRC.

### Primary Roles

- **AppSec** (lead) -- owns the test plan, executes payloads, writes vulnerability report
- **Developer** -- implements mitigations (input sanitisation, context separation, capability minimisation)
- **Sherpa** -- the system under test
- **GRC** -- reviews findings for compliance implications

## Brief 2: Advocate and Designer in the Loop -- NotebookLM Case Study

**Source:** `team/humans/dinis_cruz/briefs/02/27/part-2/v0.7.4__brief__advocate-designer-in-the-loop-notebooklm-case-study.md`

### Summary

This brief uses three persistent bugs in Google's NotebookLM (garbled text in infographics, an 18-step upload workflow that should be 2-3 steps, and a pointless button rearrangement) to argue that small persistent bugs are not QA problems but pipeline problems. It establishes two mandates for SG/Send: (1) the Advocate and Designer must be involved at every decision point, the same way AppSec is involved at every layer, and (2) every change has a cognitive cost to the user that must be budgeted by an Accountant role. The brief proposes per-user changelogs, a change credit system, and agentic scaling of review through per-bug agent teams.

### Key Asks

4. Implement per-user changelogs that show users what changed between their last version and the current one.
5. Track time-to-fix for user-reported issues as a pipeline health metric.
6. Scale the review process through agentic teams (Advocate + Designer + Dev + QA + Accountant per issue).

## Cross-Brief Analysis

### Common Themes

1. **Pipeline quality over point fixes.** Both briefs argue that individual issues (prompt injection vulnerabilities, persistent UI bugs) are symptoms of pipeline gaps. Brief 1 tests whether the Sherpa pipeline can resist adversarial input. Brief 2 tests whether the development pipeline can respond to user feedback.

2. **Role-at-every-layer pattern.** Brief 1 has AppSec reviewing every point where attacker content meets LLM context. Brief 2 has the Advocate reviewing every point where a change meets the user. Same structural pattern, different axis (security vs. user experience).

3. **Structured result tracking.** Brief 1 proposes a JSON results template for each payload tested. Brief 2 proposes a credit ledger for each change. Both require systematic, machine-readable tracking of outcomes.

4. **Agentic scaling.** Brief 1 implies multi-agent testing (AppSec + Dev + Sherpa). Brief 2 explicitly proposes per-bug agent teams. Both point toward a model where review is scaled through agents, not meetings.

### Shared Dependencies

- Both briefs depend on the **MCP infrastructure** (EXISTS) for agent-to-system interaction.
- Both briefs reference the **agentic email pipeline** (n8n + WorkMail, external to SG/Send) as infrastructure.
- Both briefs build on **v0.7.1 role activations** (Advocate, AppSec threat model).

### Recommended Execution Order

1. **First:** Brief 1 / P0 -- Map the Sherpa email pipeline attack surface. This is the highest-risk item (active security concern) and has no code dependencies.
2. **Second:** Brief 1 / P1 -- Build payload library and execute simulated injection tests (Option C). Fast path to initial security findings.
3. **Third:** Brief 2 / P1 -- Write Advocate and Designer role MDs. These are documentation tasks that establish the process framework for all future work.
4. **Fourth:** Brief 1 / P1 -- Define MCP scope for Sherpa during email reading (read-only). Highest-value mitigation.
5. **Fifth:** Brief 2 / P2 -- Add Advocate review step to Explorer workflow and define Accountant role.
6. **Sixth:** Brief 1 / P2-P3 -- Test mailbox setup, results schema, security data room.
7. **Last:** Brief 2 / P3 -- Time-to-fix metric, agentic per-bug review architecture.
