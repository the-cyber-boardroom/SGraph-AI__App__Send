## Brief 1: Email Sending Workflow Optimisation

**Source:** `team/humans/dinis_cruz/briefs/02/27/part-2/v0.7.4__dev-brief__email-sending-workflow-optimisation.md`
**Addressed to:** Developer (lead), Architect, Sherpa

### Summary

The first 10 Early Access Programme emails were successfully sent through the agentic workflow, but the process exposed three critical inefficiencies: (1) the 14KB HTML template is regenerated by the LLM for every single email instead of being templated once, (2) the send loop accumulates context inside Claude's session, risking context window exhaustion and duplicate sends at scale, and (3) there is no durable state record (send manifest) to track which emails were sent, failed, or never attempted. The brief prescribes a template-then-fill pattern, an external stateless send loop, and a JSON send manifest stored in the data room.

### Key Asks

1. **Template-then-fill pattern** -- Generate HTML email template once per batch with `{{placeholders}}`, then do mechanical string replacement per recipient (no LLM inference in the send loop).
2. **Stateless send loop** -- Move the send loop out of Claude's context window into n8n or a Python script. Claude produces template + personalisation JSON; the loop executes sends mechanically.
3. **Send manifest** -- Before sending, create a JSON manifest in the data room recording the batch (recipients, personalisation, status). Update per-email during send. Enable retry of failed/pending emails without re-running the full workflow.
4. **Context isolation** -- Ensure a 50-email batch uses no more context than a 10-email batch.
