## Brief 1: n8n Platform Assessment and Encrypted Workflow Data

### Summary

The brief provides a hands-on assessment of n8n as an orchestration layer, concluding that n8n should be **contained** — kept for triggering and routing, with all sensitive data handling moved to SG/Send's encrypted infrastructure. The critical finding is that n8n execution logs capture full request/response bodies including decryption keys and email content, violating SG/Send's zero-knowledge principle. The fix: encrypted envelope architecture where n8n only handles opaque reference IDs.

## Brief 2: Email-FS — Email as Native Data Room Format and Agent Communication Primitive

### Summary

The brief proposes treating EML files (RFC 5322 + MIME) as a native data room format, enabling the data room to function as an encrypted email system. A folder of EML files sorted by date becomes an inbox. The further insight: email primitives (to, from, subject, body, attachments, signatures) map naturally to agent-to-agent communication, creating a traceable, verifiable communication channel called "Email-FS" — following the same pattern as Issues-FS (file system as database).

### Dependencies

- **Critical dependency:** The reality doc confirms that **data room in-room file preview does not exist**. The EML viewer requires either (a) building a general preview framework first, or (b) building the EML viewer as the first preview format, establishing the pattern for others. Option (b) is recommended — use EML as the pilot for the preview component architecture.
- Depends on existing **vault file endpoints** (exist) for storage
- Depends on existing **room-view component** (exists) for the file browser shell
- Connects to **n8n Encrypted Envelopes** (Brief 1) — the encrypted envelope concept maps to EML envelope format
- Connects to **Issues-FS** (exists) as the same file-system-as-database pattern
- **Phase 3** depends on **MCP** (exists) but needs new tool definitions

### Conflicts or Concerns

1. **No rule violations detected.** The brief correctly targets vanilla JS Web Components with zero framework dependencies. No Pydantic, no direct boto3.
2. **Data room preview gap is the critical blocker.** The reality doc explicitly says in-room preview does not exist. Building the EML viewer means also building the preview component infrastructure. This should be acknowledged as a larger scope item than "just an EML parser."
3. **HTML sanitisation in EML bodies is a security concern.** EML files can contain arbitrary HTML including scripts, external image loads (tracking pixels), and CSS that could exfiltrate data. The EML renderer MUST sanitise HTML aggressively — strip scripts, strip external resource loads, sandbox iframes. AppSec should review the sanitisation approach before shipping.
4. **Fractal Email-FS adds complexity.** The brief describes data rooms within data rooms each having their own email system. This is architecturally elegant but increases the testing surface significantly. Recommend starting with a single-level Email-FS and deferring the nested/fractal property.
5. **EML parsing in vanilla JS.** Parsing MIME multipart messages in browser JavaScript without a library is non-trivial. Consider whether a lightweight, dependency-free MIME parser needs to be written or if the Web Component can handle a simplified EML subset initially (single-part text/html, simple attachments).

### Shared Theme: The Preview Gap

All three briefs assume data room content can be previewed in-browser. The reality document confirms this does not exist. The EML viewer (Brief 2) is the natural first implementation of the preview infrastructure. Building it correctly (as a pluggable preview component architecture) unblocks:
- EML rendering for Email-FS (Brief 2)
- Board material viewing for Cyber Boardroom (Brief 3)
- Future format support (PDF, CSV, markdown — already mentioned in existing specs)

**Recommendation:** Treat the data room preview component architecture as a cross-cutting deliverable. Design it as an extensible Web Component that dispatches to format-specific renderers. Build EML as the first renderer. This creates the pattern for all future formats.
