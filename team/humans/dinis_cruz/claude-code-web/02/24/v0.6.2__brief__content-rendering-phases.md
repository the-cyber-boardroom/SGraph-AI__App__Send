## What exists today

The download page (`send-download` web component) currently handles decrypted content in two ways:

1. **Text** (`content_type_hint` starts with `text/`): displayed as plain text in a `<pre>` element using `.textContent` (XSS-safe)
2. **Everything else**: auto-downloaded as a file via blob URL — the user never sees the content in-browser

The key decision point is in `send-download.js` at line ~383:
```js
if (this.isTextContent()) {
    this.decryptedText = new TextDecoder().decode(content);
} else {
    this.saveFile(content, this.fileName || 'download');
}
```

The file name is available from the SGMETA envelope metadata after decryption (line ~377). The `content_type_hint` comes from `transferInfo`.

## What to build: In-browser rendering for more file types

After decryption, instead of auto-downloading, render supported file types inline in the browser. The user should still have a "Download" button, but they should **see the content first**.

**Implementation order** (each is a separate IFD minor version):

### Phase 1 — Markdown rendering (quick win)
- Detect `.md` files from SGMETA filename or content_type_hint
- Write a **mini markdown parser** supporting only a safe subset: headings, bold, italic, lists, code blocks (with syntax highlighting if simple), tables, links (display only, not clickable or open in new tab with `rel="noopener"`)
- **No arbitrary HTML passthrough** — the parser only emits known-safe elements
- Render inside a **sandboxed iframe** (`sandbox="allow-same-origin"` + no script) for defence-in-depth
- Two layers of XSS protection: limited parser + sandboxed iframe
- Show a "View raw" toggle so users can switch between rendered and raw markdown
- Keep the existing "Copy text" and "Download as file" buttons

### Phase 2 — Image rendering
- Detect image files from SGMETA filename extension (`.png`, `.jpg`, `.jpeg`, `.gif`, `.webp`, `.svg`) or content_type_hint (`image/*`)
- After decryption, create a blob URL and display the image inline with `<img>` inside a sandboxed iframe
- **SVG special case**: SVGs can contain JavaScript — render SVGs as `<img src="blob:...">` (which disables scripts), never as inline `<svg>` elements
- Show image dimensions and file size
- "Download" button below the image
- Max display width constrained to container (640px), with click-to-expand for full size

### Phase 3 — PDF rendering
- Detect `.pdf` files from SGMETA filename or content_type_hint (`application/pdf`)
- Use the browser's built-in PDF viewer via `<iframe src="blob:...#toolbar=1">` or `<embed>` — avoid bundling PDF.js unless the browser-native approach proves inadequate
- Fallback: if embedded PDF doesn't work (some mobile browsers), show a message with a download button
- "Download" button always visible

### Phase 4 — Code/structured text rendering
- Detect code files from extension (`.js`, `.ts`, `.py`, `.json`, `.yaml`, `.yml`, `.xml`, `.html`, `.css`, `.sh`, `.sql`, `.go`, `.rs`, `.java`, `.c`, `.cpp`, `.rb`, `.php`)
- Display in a `<pre><code>` block with monospace font and line numbers
- Basic syntax highlighting if achievable with a small (~50 line) highlighter, otherwise just monospace display is fine
- JSON: auto-format/pretty-print with `JSON.stringify(parsed, null, 2)`
- Keep "Copy" and "Download" buttons

## Architecture guidance

- **All rendering is client-side** — the server never knows file types (zero-knowledge principle)
- **File type detection happens after decryption** — from the SGMETA metadata `filename` field (extension) or `content_type_hint`
- **IFD methodology** — each phase is a new minor version under `sgraph_ai_app_send__ui__user/v0/v0.1/v0.1.{next}/`
- **Security first** — sandboxed iframes for all rendered content, no inline event handlers, no `eval`, no arbitrary HTML
- **The pattern to modify**: the `startDownload()` method in `send-download.js` — after decryption, instead of the binary text-vs-file branch, add file type detection and route to appropriate renderers
- **Keep the download fallback** — every rendered file type must also have a visible "Download" button. Rendering is a preview, not a replacement for download

## Security constraints (non-negotiable)

1. **No arbitrary HTML rendering** — markdown parser emits only known-safe elements
2. **Sandboxed iframes** for all rendered content — `sandbox="allow-same-origin"` minimum
3. **SVGs rendered as `<img>` only** — never inline `<svg>` (prevents script execution)
4. **No external resources loaded** — all rendering from decrypted blob data only
5. **CSP-compatible** — no inline scripts, no `eval`, no `new Function()`
6. **`.textContent` for any user-provided text** — never `.innerHTML` with user data

## Origin of this feature request

A real user (security researcher) sent a markdown security report via SG/Send. The recipient saw raw markdown — literal `**bold**` text and `# headers`. This was documented in:
