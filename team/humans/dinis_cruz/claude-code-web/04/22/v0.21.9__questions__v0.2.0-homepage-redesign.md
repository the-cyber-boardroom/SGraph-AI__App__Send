## Q1 — Vault blob deletion: what is the accurate statement?

**Context:** The security page (`/en-gb/security/`) currently states:

> "Every transfer has a TTL. When it expires, the ciphertext is deleted from storage. There is no manual retrieval after expiry. Vaults can also be deleted on demand by the vault owner."

Dinis flagged this as **factually wrong** — encrypted blobs are not automatically deleted on expiry.

**Question for Architect / Dev:**
1. What IS the actual deletion behaviour for transfers and vaults? Do they expire passively (access blocked) but blobs remain in storage? Is deletion triggered manually only?
2. What is the accurate one-paragraph statement we can put on the security page?
3. Is this the same for both the transfer model AND the vault model, or do they differ?

**Blocked:** Cannot fix the security page text until this is answered.

## Q3 — i18n approach for v0.2.0: pre-rendered or runtime?

**Context:** The v0.1.0 site (currently live) uses a pre-rendered approach — translations are baked into the HTML at build time per locale, and the `i18n.js` module only handles locale detection and navigation (no runtime JSON fetching). All translations live in `sgraph_ai__website/i18n/en-gb.json` (and equivalents) as a single large file per locale (currently 857 lines for `en-gb.json`).

Dinis asked that:
- All content be in separate i18n translation files (split by type/content, not one massive file)
- OR in Markdown files
- Following the pattern of the current v0.1.0 live site

**Questions for Dev / Librarian:**
1. **Pre-rendered vs runtime:** Should v0.2.0 follow the same pre-rendered approach as v0.1.0 (translations baked into HTML per locale, one HTML file per locale)? Or should it fetch translations at runtime from JSON/Markdown files?
2. **Split strategy:** If splitting i18n files, what is the preferred split? Options:
   - By page: `homepage.en-gb.json`, `vaults.en-gb.json`, `security.en-gb.json`
   - By content type: `nav.json`, `hero.json`, `use-cases.json`
   - By feature: one file per major section
3. **Markdown files:** Which content types should be in Markdown (e.g. privacy sentences, use-case descriptions, long-form copy) vs JSON (e.g. nav labels, button text, short strings)?
4. **Build process:** Is there a build step that compiles translations into pre-rendered HTML? Or does the site rely on runtime JS for translation application? If build step: where is it documented?
5. **`data-i18n` attribute:** The v0.1.0 HTML uses `data-i18n="key.path"` attributes. Should v0.2.0 use the same convention?

**Not blocked** — can proceed with hardcoded English and apply i18n as a second pass, but need the split strategy confirmed before adding new content to avoid large-scale refactoring later.
