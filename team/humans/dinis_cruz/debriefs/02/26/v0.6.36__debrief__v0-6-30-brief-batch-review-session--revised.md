## Baseline: What Exists Today (Code-Verified)

Before reading the brief reviews, the reader should understand the current state of the system. All of the following is verified in code (see the [reality document](../../../../../roles/librarian/reality/v0.6.36__what-exists-today.md) for full details):

- **Data rooms** -- create, list, archive, manage members, generate invites, accept invites, room session tokens, audit trail (backend + UI)

- **Admin UI** -- 15 web components across system, data rooms, PKI, vault, registry, tokens, monitoring, and debug panels
- **User UI** -- upload, download, room join, room view, vault, SSH keygen (6 pages, 8 languages)
- **Infrastructure** -- 2 Lambda functions (User + Admin), 3 stages each (dev/qa/prod), CI/CD via GitHub Actions
- **Email pipeline** -- n8n + WorkMail pipeline exists and has been used (Milestone M-008)

## 1. Executive Summary

Six role agents worked in parallel to review **18 unprocessed briefs** uploaded on 25-26 February 2026 -- the largest single batch of briefs in the project's history. The briefs collectively **propose expanding** SGraph Send from its current state (encrypted file transfer with data rooms, vaults, and PKI) into a broader zero-knowledge collaboration platform by adding ephemeral secrets, password management tools, browser transparency tools, and automated email outreach.

**Important context:** Data rooms, vaults, and PKI infrastructure already exist in code (see Baseline above). The briefs propose **new features** (one-time secrets, password generator, browser storage viewer) and **enhancements** to existing features (data room preview, PKI-friendly URLs, automated key exchange). This debrief covers the review of those proposals.

**What the session produced:**

- A **master index** cataloguing all 18 briefs with thematic grouping, duplicate analysis, cross-references, and a five-tier processing order
- A **product assessment** with priority classification (P1 through Research), four build waves (38-56 sessions estimated), and seven identified risks
- An **architecture review** identifying six encryption contexts, five shared components, four architectural tensions, and eight architecture decisions (four blocking)
- A **security assessment** uncovering five metadata leak vectors, mapping 15 new endpoints and 8 trust boundaries, and proposing 13 security actions across P0-P3
- A **trails and MCP review** mapping five proposed user journeys, threat modelling three priority trails, assessing MCP gaps (seven new tools needed), and drafting a waiting list email
- An **evolution map** plotting 17 proposed new components (C29-C45) on the Wardley Map, estimating 44-74 sessions total, and identifying the infrastructure critical path

**The key message across all roles:** Build one-time secret links first (this feature does not exist yet). Infrastructure decisions (S3/CloudFront, URL architecture) are the critical path. Enforce opaque IDs everywhere to preserve zero-knowledge. Start security review in parallel with everything.

## 3. Key Findings Across Roles

### 3.1 Conductor: Wave 1 Priorities

The Conductor classified all briefs into P1 (Build Now), P2 (Build Next), P3 (Build Later), and Research. Four items are P1 for Wave 1 (this week / next 3-5 sessions):

| P1 Item | Sessions | Rationale | Current Status |
|---------|----------|-----------|----------------|
| **One-time secret link** | 3-4 | Highest standalone user value; unblocks data room key sharing; front-door acquisition feature | **Does not exist** -- no code |
| **Browser storage viewer** | 1-2 | Zero dependencies; demonstrates transparency; "we are different" proof point | **Does not exist** -- no code |
| **Waiting list outreach** | 2-3 | M-008 proved the pipeline; momentum is perishable; users are waiting | **Partially exists** -- n8n + WorkMail email pipeline works; no signup UI or waiting list flow |
| **Security review framework** (start) | 1-2 | Creates the risk register that governs all build decisions; runs in parallel | **Does not exist** as formal framework -- security properties are verified in code but no structured risk register |

**Recommendation:** Ship Wave 1 now. Do not start data room enhancements (Wave 2) until the one-time secret has a working demo. Do not start PKI enhancements (Wave 4) until real users are asking for it.

**Clarification:** Data rooms and PKI already exist and are functional. Wave 2 and Wave 4 refer to **enhancements** to these existing features (preview, versioned editing, friendly URLs), not to building them from scratch.

### 3.2 Architect: Four Blocking Architecture Decisions

The Architect identified eight architecture decisions, four of which are **blocking** (must be resolved before implementing the proposed new features):

| # | Decision | Recommendation |
|---|----------|----------------|
| 1 | **SGMETA Envelope Specification** | Formalise with version byte (`SGMETA\x01`); publish as a specification |
| 2 | **Key Registry Architecture** | Unified browser-side Key Registry with typed entries and context-based resolution |
| 3 | **Component Versioning Strategy** | Two-tier: IFD shell (monolithic) + shared components (independently versioned via manifest) |
| 4 | **PKI URL Naming** | Opaque IDs in all URL paths; human-friendly names only in encrypted metadata |

**Context:** SGMETA envelope already exists in the current client-side code (used by `send-upload` and `send-download` components). The decision is about formalising and versioning it, not creating it from scratch. The Key Registry and Component Versioning decisions are about architectural patterns for the proposed new features.

### 3.3 AppSec: PKI URL Metadata Leak Is HIGH Severity

The AppSec review found that the PKI-friendly links brief **proposes** URLs like `/d/b12/quarterly-report` where the document name is in the URL path -- **visible to the server, logged by CloudFront, and exposed in a breach**. This directly contradicts the one-time secret brief's principle that "a full server-side data leak should expose zero customer data."

**Important:** This is a review of a **proposal**, not existing code. PKI-friendly links do not exist in code. The AppSec review is flagging this before it gets built, which is exactly the right time.

**Five metadata leak vectors identified (mix of existing and proposed):**

| # | Vector | Severity | Status |
|---|--------|----------|--------|
| 1 | PKI URL document names (`/d/b12/quarterly-report`) | **HIGH** | **Proposed only** -- no code exists |
| 2 | Data room names stored as plaintext on server | MEDIUM | **Existing concern** -- room endpoints store room names |
| 3 | Version chain metadata reveals content importance and editor identities | MEDIUM | **Proposed only** -- versioned editing does not exist |
| 4 | Timing correlation between secret creation and data room user creation | LOW | **Proposed only** -- one-time secrets do not exist |
| 5 | URL route type reveals content category (`/s/` vs `/d/` vs `/room/`) | LOW | **Proposed only** -- these URL patterns do not exist |

**P0 Security Actions (before building any of the proposed new features):**
- Enforce opaque IDs in ALL URL paths
- Add rate limiting to all new endpoints
- Require authentication or proof-of-work for anonymous secret creation

### 3.4 Sherpa: Proposed Trail 1 (One-Time Secret) Is the Hub

The Sherpa mapped five **proposed** user trails and discovered that **Trail 1 (One-Time Secret) would be the hub connecting all other trails:**

```
Trail 4: Password Generator (PROPOSED)
  |
  v (share via secret link)
Trail 1: One-Time Secret (PROPOSED)  <---- Trail 2: Data Room Admin (share key via secret)
  |                                                 (rooms EXIST; key sharing enhancement PROPOSED)
  v (output: one-time link)
Trail 3: Data Room User (receive key via one-time link)
  |                      (rooms EXIST; automated key handoff PROPOSED)
  v (browse room, receive files)
Trail 5: PKI File Receiver (import sender key, reply)
         (PKI EXIST; friendly links PROPOSED)
```

**None of these trails are implemented.** They are proposed user journeys that describe how the new features would connect. Building Trail 1 (one-time secret) first would unblock three other proposed trails. This aligns with the Conductor's P1 classification.

**MCP gap analysis:** Seven new MCP tools are proposed to support the new trails:

| Tool | Backend Exists? | MCP Exposure Exists? |
|------|----------------|---------------------|
| `secrets_create` | No -- one-time secrets not built | No |
| `secrets_status` | No -- one-time secrets not built | No |
| `rooms_create` | **Yes** -- `/rooms/create` endpoint works | **No** -- not exposed via MCP |
| `rooms_add_user` | **Yes** -- `/rooms/members-add/{id}` endpoint works | **No** -- not exposed via MCP |
| `rooms_revoke_user` | **Yes** -- `/rooms/members-remove/{id}/{user}` endpoint works | **No** -- not exposed via MCP |
| `rooms_list_users` | **Yes** -- `/rooms/members/{id}` endpoint works | **No** -- not exposed via MCP |
| `rooms_upload` | **Yes** -- `/vault/file` and `/vault/file-chunk` endpoints work | **No** -- not exposed via MCP |

**Key distinction:** For rooms, the backend functionality already exists. The gap is MCP tool exposure, not backend implementation. For secrets, nothing exists at any layer.

### 3.5 Cartographer: 17 Proposed New Components, Infrastructure Is the Critical Path

The Cartographer mapped 17 **proposed** new components (C29-C45), which would more than double the system's mapped component count. Most enter at Genesis stage (Explorer territory). Total estimated effort: **44-74 sessions** across all proposed components.

**Critical path:**
1. **C42: S3/CloudFront Hosting** -- blocks URL Architecture, which blocks all new URL patterns
2. **C41: URL Architecture** -- blocks `/s/` (secrets), `/d/` (PKI), `/tools/` (tools)
3. **C29: One-Time Secret** -- blocks Data Room key sharing enhancement
4. **C32: Data Room Preview** -- blocks Versioned Editing and Minimal Viewers

**Context:** S3 storage already exists for production file storage. The proposed C42 is about S3/CloudFront as a **static hosting layer** for UI assets and new URL patterns, which is a different use from the existing S3 file storage.

**Infrastructure first, features second.** The Cartographer and Architect agree: deploy S3/CloudFront for static hosting, then URL architecture, then build new features on top.

## 5. Key Decisions for the Human

The following decisions emerged from the session and require the human's input before implementation of the **proposed new features** proceeds.

### Decision A: Infrastructure-First or Feature-First?

**The tension:** The Conductor says new features (one-time secret, storage viewer) can be built on the existing Lambda path and migrated later (infrastructure is P2). The Architect and Cartographer say new infrastructure (S3/CloudFront static hosting, URL architecture) should come first to avoid rework and unblock everything cleanly.

**Options:**
1. **Features first** -- ship one-time secret and storage viewer on the current Lambda path this week; migrate to S3/CloudFront in Wave 2
2. **Infrastructure first** -- set up S3/CloudFront static hosting (2-3 sessions), then build new features on the new path
3. **Parallel** -- one track builds infrastructure while another builds features on the current path

**Recommendation from the session:** Option 3 (parallel) if capacity allows. Option 1 if the priority is speed of first user-facing delivery.

### Decision B: Four Blocking Architecture Decisions

The Architect identified four decisions that must be made before implementing the proposed new features (see Section 3.2 above). The human should review and approve or modify these recommendations:

1. **SGMETA envelope** -- formalise with version byte? (SGMETA already exists in client code; this is about standardising it)
2. **Key Registry** -- unified registry or per-feature key management?
3. **Component versioning** -- two-tier (IFD shell + manifest) or monolithic?
4. **PKI URL naming** -- opaque IDs or human-readable names in URLs?

### Decision C: Data Room Metadata Privacy

**The tension:** The existing data room implementation stores room names in the server. The AppSec review flags this as a zero-knowledge concern. The Architect recommends plaintext for MVP with a migration path to encrypted.

**Options:**
1. **Plaintext for MVP** (Architect recommendation) -- accept the current trade-off; data rooms are the usability-first mode
2. **Hashed** (AppSec recommendation) -- server stores SHA-256 of room/user names; client resolves
3. **Encrypted** -- room and user names encrypted with master key; server sees nothing

### Decision D: Chrome Password Sync for Private Keys

**The tension:** The PKI-friendly links brief proposes storing private keys in Chrome's credential manager for cross-device portability. AppSec rates this as HIGH risk (Google becomes a trust dependency). The Cartographer rates it the highest-risk proposed component (Genesis/Research).

**Note:** This feature does not exist in code. PKI key generation exists in both the Admin UI (`pki-keys` component) and User UI (vault). The proposal is about cross-device sync of those keys.

**Options:**
1. **Proceed as research** with prominent user warning (AppSec P1 recommendation)
2. **Defer entirely** until symmetric features are proven and users request PKI portability
3. **Alternative approach** -- investigate dedicated password manager integration instead

### Decision E: Outreach Timing

**The tension:** The Conductor says "ship the M-008 story immediately -- the momentum is perishable." The Sherpa identifies five blocking prerequisites (sending address, template, Getting Started guide, recipient list, human approval).

**Context:** The n8n + WorkMail email pipeline was proven in Milestone M-008 (Claude.ai generated a PDF, encrypted it, uploaded via MCP, sent the link via email). What does not exist: a Getting Started guide page, a waiting list signup UI, and a formal recipient list.

**Options:**
1. **This week** -- resolve the five prerequisites and send first batch
2. **After Wave 1** -- wait until one-time secret and storage viewer are live so the outreach email has more to show
3. **Phased** -- send a teaser email this week ("exciting things coming"), then a feature email after Wave 1
