## 1. Executive Summary

Six role agents worked in parallel to review **18 unprocessed briefs** uploaded on 25-26 February 2026 -- the largest single batch of briefs in the project's history. The briefs collectively describe the transition of SGraph Send from a single-purpose encrypted file transfer tool into a **multi-surface zero-knowledge collaboration platform** with data rooms, ephemeral secrets, password management, browser transparency tools, PKI infrastructure, and agent-operated email workflows.

**What the session produced:**

- A **master index** cataloguing all 18 briefs with thematic grouping, duplicate analysis, cross-references, and a five-tier processing order
- A **product assessment** with priority classification (P1 through Research), four build waves (38-56 sessions estimated), and seven identified risks
- An **architecture review** identifying six encryption contexts, five shared components, four architectural tensions, and eight architecture decisions (four blocking)
- A **security assessment** uncovering five metadata leak vectors, mapping 15 new endpoints and 8 trust boundaries, and proposing 13 security actions across P0-P3
- A **trails and MCP review** mapping five new user journeys, threat modelling three priority trails, assessing MCP gaps (seven new tools needed), and drafting a waiting list email
- An **evolution map** plotting 17 new components (C29-C45) on the Wardley Map, estimating 44-74 sessions total, and identifying the infrastructure critical path

**The key message across all roles:** Build one-time secret links first. Infrastructure (S3/CloudFront, URL architecture) is the critical path. Enforce opaque IDs everywhere to preserve zero-knowledge. Start security review in parallel with everything.

## 3. Key Findings Across Roles

### 3.1 Conductor: Wave 1 Priorities

The Conductor classified all briefs into P1 (Build Now), P2 (Build Next), P3 (Build Later), and Research. Four items are P1 for Wave 1 (this week / next 3-5 sessions):

| P1 Item | Sessions | Rationale |
|---------|----------|-----------|
| **One-time secret link** | 3-4 | Highest standalone user value; unblocks data room key sharing; front-door acquisition feature |
| **Browser storage viewer** | 1-2 | Zero dependencies; demonstrates transparency; "we are different" proof point |
| **Waiting list outreach** | 2-3 | M-008 proved the pipeline; momentum is perishable; users are waiting |
| **Security review framework** (start) | 1-2 | Creates the risk register that governs all build decisions; runs in parallel |

**Recommendation:** Ship Wave 1 now. Do not start data rooms (Wave 2) until the one-time secret has a working demo. Do not start PKI (Wave 4) until real users are asking for it.

### 3.2 Architect: Four Blocking Architecture Decisions

The Architect identified eight architecture decisions, four of which are **blocking** (must be resolved before implementation begins):

| # | Decision | Recommendation |
|---|----------|----------------|
| 1 | **SGMETA Envelope Specification** | Formalise with version byte (`SGMETA\x01`); publish as a specification |
| 2 | **Key Registry Architecture** | Unified browser-side Key Registry with typed entries and context-based resolution |
| 3 | **Component Versioning Strategy** | Two-tier: IFD shell (monolithic) + shared components (independently versioned via manifest) |
| 4 | **PKI URL Naming** | Opaque IDs in all URL paths; human-friendly names only in encrypted metadata |

### 3.3 AppSec: PKI URL Metadata Leak Is HIGH Severity

The AppSec review found that the PKI-friendly links brief proposes URLs like `/d/b12/quarterly-report` where the document name is in the URL path -- **visible to the server, logged by CloudFront, and exposed in a breach**. This directly contradicts the one-time secret brief's principle that "a full server-side data leak should expose zero customer data."

**Five metadata leak vectors identified:**

| # | Vector | Severity |
|---|--------|----------|
| 1 | PKI URL document names (`/d/b12/quarterly-report`) | **HIGH** |
| 2 | Data room names stored as plaintext on server | MEDIUM |
| 3 | Version chain metadata reveals content importance and editor identities | MEDIUM |
| 4 | Timing correlation between secret creation and data room user creation | LOW |
| 5 | URL route type reveals content category (`/s/` vs `/d/` vs `/room/`) | LOW |

**P0 Security Actions (before any implementation):**
- Enforce opaque IDs in ALL URL paths
- Add rate limiting to all new endpoints
- Require authentication or proof-of-work for anonymous secret creation

### 3.4 Sherpa: Trail 1 (One-Time Secret) Is the Hub

The Sherpa mapped five new user trails and discovered that **Trail 1 (One-Time Secret) is the hub connecting all other trails:**

```
Trail 4: Password Generator
  |
  v (share via secret link)
Trail 1: One-Time Secret  <---- Trail 2: Data Room Admin (share key via secret)
  |
  v (output: one-time link)
Trail 3: Data Room User (receive key via one-time link)
  |
  v (browse room, receive files)
Trail 5: PKI File Receiver (import sender key, reply)
```

Building Trail 1 first unblocks three other trails. This aligns with the Conductor's P1 classification of the one-time secret.

**MCP gap analysis:** Seven new MCP tools are needed to support the new trails (`secrets_create`, `secrets_status`, `rooms_create`, `rooms_add_user`, `rooms_revoke_user`, `rooms_list_users`, `rooms_upload`).

### 3.5 Cartographer: 17 New Components, Infrastructure Is the Critical Path

The Cartographer mapped 17 new components (C29-C45), more than doubling the system's mapped component count. Most enter at Genesis stage (Explorer territory). Total estimated effort: **44-74 sessions** across all components.

**Critical path:**
1. **C42: S3/CloudFront Hosting** -- blocks URL Architecture, which blocks all new URL patterns
2. **C41: URL Architecture** -- blocks `/s/` (secrets), `/d/` (PKI), `/tools/` (tools)
3. **C29: One-Time Secret** -- blocks Data Room key sharing
4. **C32: Data Room Preview** -- blocks Versioned Editing and Minimal Viewers

**Infrastructure first, features second.** The Cartographer and Architect agree: deploy S3/CloudFront, then URL architecture, then build features on top.

## 5. Key Decisions for the Human

The following decisions emerged from the session and require the human's input before implementation proceeds.

### Decision A: Infrastructure-First or Feature-First?

**The tension:** The Conductor says features (one-time secret, storage viewer) can be built on the existing Lambda path and migrated later (infrastructure is P2). The Architect and Cartographer say infrastructure (S3/CloudFront, URL architecture) should come first to avoid rework and unblock everything cleanly.

**Options:**
1. **Features first** -- ship one-time secret and storage viewer on the current Lambda path this week; migrate to S3/CloudFront in Wave 2
2. **Infrastructure first** -- set up S3/CloudFront (2-3 sessions), then build features on the new path
3. **Parallel** -- one track builds infrastructure while another builds features on the current path

**Recommendation from the session:** Option 3 (parallel) if capacity allows. Option 1 if the priority is speed of first user-facing delivery.

### Decision B: Four Blocking Architecture Decisions

The Architect identified four decisions that must be made before implementation begins (see Section 3.2 above). The human should review and approve or modify these recommendations:

1. **SGMETA envelope** -- formalise with version byte?
2. **Key Registry** -- unified registry or per-feature key management?
3. **Component versioning** -- two-tier (IFD shell + manifest) or monolithic?
4. **PKI URL naming** -- opaque IDs or human-readable names in URLs?

### Decision C: Data Room Metadata Privacy

**The tension:** The data room brief stores room names and user names in plaintext on the server. The AppSec review flags this as a zero-knowledge violation. The Architect recommends plaintext for MVP with a migration path to encrypted.

**Options:**
1. **Plaintext for MVP** (Architect recommendation) -- accept the trade-off; symmetric data rooms are the usability-first mode
2. **Hashed** (AppSec recommendation) -- server stores SHA-256 of room/user names; client resolves
3. **Encrypted** -- room and user names encrypted with master key; server sees nothing

### Decision D: Chrome Password Sync for Private Keys

**The tension:** The PKI-friendly links brief proposes storing private keys in Chrome's credential manager for cross-device portability. AppSec rates this as HIGH risk (Google becomes a trust dependency). The Cartographer rates it the highest-risk component (Genesis/Research).

**Options:**
1. **Proceed as research** with prominent user warning (AppSec P1 recommendation)
2. **Defer entirely** until symmetric features are proven and users request PKI portability
3. **Alternative approach** -- investigate dedicated password manager integration instead

### Decision E: Outreach Timing

**The tension:** The Conductor says "ship the M-008 story immediately -- the momentum is perishable." The Sherpa identifies five blocking prerequisites (sending address, template, Getting Started guide, recipient list, human approval).

**Options:**
1. **This week** -- resolve the five prerequisites and send first batch
2. **After Wave 1** -- wait until one-time secret and storage viewer are live so the outreach email has more to show
3. **Phased** -- send a teaser email this week ("exciting things coming"), then a feature email after Wave 1
