## 3. Website Specification Summary

### 3.1 Page Structure (from v0.7.1 dev brief)

```
sgraph.ai/
  index.html              Landing page
  agents/
    index.html            Agent directory overview
    sherpa/index.html     Per-agent page (+ public keys)
    ambassador/index.html
    {role}/index.html
    keys.json             Machine-readable key directory
  product/index.html      SG/Send product explainer
  architecture/index.html How it's built (investors/developers)
  contact/index.html      Contact form
```

### 3.2 Landing Page (from v0.7.1 dev brief)

```
[HERO]
  Headline:  "AI agents that build, use, and secure the same platform."
  Subhead:   "SG/Send is a zero-knowledge encrypted file sharing platform,
              built entirely by AI agents working in parallel.
              The same agents that built it use it every day."
  CTA:       [Try SG/Send ->]  [Meet the Agents ->]

[SECTION: The Platform]
  Description of SG/Send; link to send.sgraph.ai

[SECTION: The Agents]
  Grid of agent cards (role name, description, link to agent page)

[SECTION: How It's Built]
  "18 AI agents. One human. Everything in public."
  Links to architecture page and GitHub repo

[FOOTER]
  Links: send.sgraph.ai | GitHub | Contact
  "Email Sherpa" button on every page
```

### 3.3 Agent Directory

Each agent gets a page with:
- Role name, one-sentence description, inbox email
- Ed25519 public key (copy/download)
- PGP public key (copy/download)
- "Send via SG/Send" button (pre-populated with recipient key)

Machine-readable endpoint at `/agents/keys.json` for programmatic discovery.

### 3.4 Contact Integration

Modal on every page footer:
- Message field (required, 10–5000 chars)
- Email field (optional — enables reply)
- Screenshot attachment (optional — signals UX issues)
- Submits to Lambda endpoint -> SES -> `sherpa@sgraph.ai`
- CC's user email if provided

Metrics tracked: total submissions, submissions with email (conversion intent), submissions with screenshot (UX signal), page attribution, reply rate.

### 3.5 Product Segmentation

```
sgraph.ai              <- Brand, project, agent directory, contact
send.sgraph.ai         <- SG/Send product (file transfer, data rooms, tools)
static.send.sgraph.ai  <- Static assets for product (CDN)
docs.send.sgraph.ai    <- Product documentation
team.send.sgraph.ai    <- Internal team site
```

### 3.6 Acceptance Criteria (10 items from v0.7.1 brief)

1. `sgraph.ai` serves a static landing page explaining the project
2. Site built with Jekyll (or agreed alternative), source in `/sgraph-ai-website/`
3. CI pipeline deploys to S3 -> CloudFront on merge to main
4. Agent directory at `/agents/` with one page per active role
5. Each agent page shows Ed25519 and PGP public key with copy/download
6. Machine-readable key directory at `/agents/keys.json`
7. Contact modal on every page: message, optional email, optional screenshot
8. Contact form submits to SES -> `sherpa@sgraph.ai`, CC's user if provided
9. Contact metrics tracked in outreach data room
10. Product segmentation map documented

## 4. Architect Assessment

### 4.1 Decisions Requiring Architect Input

| # | Decision | Options | Architect Recommendation |
|---|----------|---------|-------------------------|
| 1 | **Static site generator** | Jekyll (brief suggests), Hugo, Astro, plain HTML | **Hugo** — single binary, fastest builds, no Ruby dependency, excellent templates |
| 2 | **Repository structure** | Separate repo vs folder in main repo | **Folder in main repo** (`/sgraph-ai-website/`) — enables auto-generation of agent pages from ROLE.md files |
| 3 | **Contact form backend** | Lambda function vs n8n webhook | **Lambda function** — lightweight, integrated with existing AWS infrastructure, testable |
| 4 | **CloudFront deployment** | Shared distribution vs separate distribution | **Separate distribution** — sgraph.ai is pure static (no API), cleaner to manage, separate monitoring |
| 5 | **Agent page generation** | Manual HTML vs auto-generated from ROLE.md | **Auto-generated** — keeps pages in sync with role definitions, build pipeline reads from `team/roles/{role}/` |

### 4.2 Infrastructure Architecture

**Proposed deployment topology:**

```
Distribution 1: send.sgraph.ai (existing)
  Origin: S3 + API Gateway (Lambda)
  Behaviours: /api/*, /transfers/*, /* (static)

Distribution 2: sgraph.ai (NEW)
  Origin: S3 only (no Lambda for page serving)
  Behaviours: /* (all static, 24h cache TTL)
  Contact form: POST /api/contact -> Lambda endpoint
```

**S3 structure:**
```
s3://sgsend-static/
  sgraph-ai/v0.1.0/
    index.html
    css/style.css
    js/contact-form.js
    agents/
      sherpa/index.html
      ambassador/index.html
      keys.json
    assets/logo.svg
```

**Build pipeline:**
```
CI Pipeline:
  1. Build Hugo site from /sgraph-ai-website/
  2. Generate agent pages from /team/roles/{role}/ROLE.md
  3. Build keys.json from role metadata
  4. Upload to s3://sgsend-static/sgraph-ai/
  5. Invalidate CloudFront paths
```

### 4.3 Integration Points

| System | How Website Connects |
|--------|---------------------|
| URL Architecture (v0.6.30) | Same S3/CloudFront deployment pattern; different domain |
| PKI Key Registry | Agent key directory is the public-facing counterpart |
| Sherpa Inbox Workflow | Contact form feeds into Sherpa's n8n inbox workflow |
| MCP Segmentation (v0.7.1) | Website makes platform boundaries visible to users/agents |
| DRU/MRU/ARU (v0.6.14) | Every contact submission is a potential DRU event |

## 5. Ambassador Assessment

### 5.1 Messaging for Different Audiences

**End Users (Non-Technical):**
- "Send anything. Privately."
- "Like WeTransfer, but nobody can see what you're sending — not even us."
- Use the locked-box analogy, avoid "zero-knowledge" in headlines

**Security Professionals / Developers:**
- "Browser-side AES-256-GCM encryption via Web Crypto API"
- "Server never touches plaintext"
- "Open source; verify yourself"

**Investment Firms / Enterprise:**
- "Enterprise security at startup pricing"
- "Secure data rooms your team owns. Zero-knowledge encryption. Costs nothing when idle."
- Competitive differentiation: client-side encryption (not server-side like legacy VDRs), serverless pricing model, zero lock-in

**AI / Agent Community:**
- "19 AI agents build, use, and secure the same platform"
- Public key directory for agent-to-agent encrypted coordination
- "The same agents that built it use it every day"

### 5.2 Brand Identity (from v0.3.5 Design Guidelines)

| Element | Value |
|---------|-------|
| Logo | `SG/Send` — purely typographic, slash is the hero element |
| Slash angle | 75 degrees |
| Primary colour | Teal (#4ECDC4) |
| Background | Dark Navy (#1A1A2E) |
| Font | DM Sans (body/headings), JetBrains Mono (code) |
| Tagline | "Your files, your keys, your privacy" |
| Default theme | Aurora (dark) |
| Enterprise theme | Glacier (light) |

### 5.3 Competitive Positioning (from Ambassador research)

| Dimension | Legacy VDRs | Modern VDRs | SG/Send |
|-----------|-------------|-------------|---------|
| Encryption | Server-side | Server-side | **Client-side** |
| Pricing | GBP 7,500-25,000+ per project | GBP 140-460/month | **GBP 49-999/month; zero when idle** |
| Infrastructure | Vendor-hosted | Vendor-hosted | **Customer owns; self-host option** |
| Setup | Days to weeks | Hours | **Minutes** |
| Audit trail | Proprietary logs | Proprietary | **Git history (immutable)** |
| Lock-in | High | Medium | **Zero** |

- Card front: `send.sgraph.ai` with pre-printed token

## 7. Recommended Implementation Phasing

### Phase 1: MVP (P0 — immediate)

1. Architect: Confirm technology choice (Hugo recommended)
2. Create `/sgraph-ai-website/` folder in repo
3. Build landing page (hero, platform, agents grid, architecture, footer)
4. Build 3 agent pages (Sherpa, Ambassador, Architect) with public keys
5. Generate `keys.json` for those 3 agents
6. Build contact form modal + Lambda backend
7. Deploy to S3/CloudFront (separate distribution for sgraph.ai)
8. CI pipeline: merge to main -> deploy to S3

**Deliverables:** Live sgraph.ai with landing page, 3 agent pages, contact form.

### Phase 2: Complete Agent Directory

1. Auto-generate all 18 agent pages from ROLE.md files
2. Auto-generate keys.json from role metadata
3. Add recent review links to agent pages
4. Create per-agent email addresses (if not already done)

**Deliverables:** Full agent directory, auto-generated, kept in sync with ROLE.md.

### Phase 3: Enhanced Features (P2)

1. Use-case pages (passwords, documents, Claude integration)
2. Marketing images and comparison tables
3. Contact form analytics dashboard
4. Blog/news section for project updates
