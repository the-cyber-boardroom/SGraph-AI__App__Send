## Executive Summary

Only one brief arrived on 16 April: a detailed investigation document from a Claude Web agent session.

### What the Brief Documents

`sgit share` fails consistently with HTTP 503 "DNS cache overflow" when uploading a ~38MB vault (56 files) from a Claude Web container. This is a **reproducible, hard failure** — not a flaky network issue.

| What Works | What Fails |
|------------|------------|
| `sgit push` — individual object uploads | `sgit share` — single large blob via S3 multipart PUT |
| `sgit pull`, `sgit clone`, `sgit commit` | `sgit push` batch mode (SSL EOF, graceful fallback) |
| All vault read/write operations | Share token creation for vaults > ~4MB |

**Root cause (hypothesis):** The Claude Web container's egress proxy rejects large PUT requests to S3 presigned URLs. The "DNS cache overflow" response is non-standard (not an S3 error) — it comes from the proxy, not from AWS. This was confirmed by:
- DNS resolution working (`curl` returns normal 403 from S3)
- Plain-text response body (S3 always returns XML)
- A prior successful `sgit share` when the vault was ~38KB

**Workaround (available now):** `sgit push` + share the vault key out-of-band. The trade-off is that vault keys give write access; share tokens are read-only.
