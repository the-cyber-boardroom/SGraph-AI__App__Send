### The Three Key Findings

1. **Local LLM backend decision:** `mlx-openai-server` is the right choice today for running Gemma 4 locally with an OpenAI-compatible API. LiteRT-LM is not viable for desktop API use (no HTTP server, 4K context cap). Use MLX.

2. **Ollama audio/video is a confirmed blocker:** Ollama has no mechanism to accept audio or video input across any of its API surfaces. `_buildOllamaBody` must fail fast with a clear error when audio/video content parts are present. This prevents a class of silent failures.

## Key Decisions Needing Your Review

### Decision 1: Ollama + Audio/Video — Fail Fast vs Auto-Reroute

When a user has Ollama selected and submits a request with audio/video content, two options:
- **A (recommended):** Fail with a clear error, prompt the user to switch to mlx/OpenAI
- **B:** Auto-reroute to an audio-capable backend silently

The Architect recommends Option A because auto-rerouting changes the user's explicit privacy/cost choice. Do you agree?

### Decision 2: LiteRT-LM — Gating Condition

The Architect proposes not integrating LiteRT-LM until it ships a built-in HTTP server with OpenAI-compatible endpoints. The alternative would be to build a custom FastAPI bridge around the Python SDK now. Given the SDK is day-1 and E4B isn't available in LiteRT format, do you want to defer or build the bridge now?

### Decision 3: MLX Provider as First-Class Option in UI

Should `mlx-openai-server` appear in the provider picker UI alongside OpenAI and Ollama? Or should it be a hidden/developer option? (Relevant because it requires the user to have it running locally.)
