## Key Decisions Needing Your Review

### Decision 1: OD #6 — Upload Branch Strategy
**What it is:** When a vault-backed folder upload fails partway, the vault exists with partial content. What happens?
**Architect recommendation:** Dedicated branch per upload session; merge to main on completion. Failed session = branch not merged.
**Why it matters:** Blocks vault upload beta from ever leaving beta status.

### Decision 2: OD #7–9 — Playwright Session Behaviour
**What they are:** Chrome profile persistence (fresh vs vault-stored), timeout routing (Lambda vs Fargate), concurrent sessions per container.
**Architect recommendation:** (7) Fresh-per-session default; (8) caller specifies target, no auto-routing; (9) one-per-container.
**Why it matters:** These unresolved decisions are blocking the Playwright first-pass implementation.

### Decision 3: OD #11 — Master Vault Key Derivation
**What it is:** PBKDF2/HKDF scheme for deriving vault-specific keys from a master key.
**Who needs to sign off:** AppSec.
**Why it matters:** Blocks the auth MVP's vault key storage implementation.

### Decision 4: OD #13 — Vault Key Storage Option (A vs B)
**What it is:** MVP uses Option A (user attributes). Production path may need Option B (encrypted bundle in SG/Send) for scale.
**Architect recommendation:** Build `<sg-user-data>` as storage-agnostic so Option B can be substituted without changing callers.
**Why it matters:** Affects auth MVP implementation approach.

### Decision 5: OD #14 — Docker Hub Namespace
**What it is:** All container briefs reference `sgraph/` (or `sgraph-ai/` per v4 doc). Neither is confirmed.
**Recommendation:** Register immediately to avoid namespace squatting.
**Why it matters:** Blocks any container publication.
