## What Was Built

This session created a complete Docker container deployment for SGraph Send ŌĆö User Lambda only, no Admin. The container runs the same FastAPI app that powers `send.sgraph.ai` but under uvicorn instead of Mangum, with static UI served directly from FastAPI instead of CloudFront/S3.

### Key Design Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Container scope | **User Lambda only** ŌĆö Admin excluded | Admin panel not ready; simplifies everything |
| Auth for API | **Existing `SGRAPH_SEND__ACCESS_TOKEN` env-var fallback** | Already built into all User routes ŌĆö no new code needed |
| Auth for UI | **Global `Middleware__Check_API_Key` middleware** | Protects ALL routes (API + static files) when token is set |
| Auth login flow | **Cookie via `/auth/set-cookie-form`** | Existing osbot-fast-api pattern ŌĆö user enters token, cookie set, browses freely |
| Static UI serving | **FastAPI `StaticFiles` mount at `/send/*`** | No CloudFront needed in container; negligible perf impact for single-instance |
| Storage default | **Memory-FS** (ephemeral) | Configurable to DISK (Docker volume) or S3 via env vars |
| EC2 provisioning | **User-data bootstrap** (no Packer/AMI) | Following Playwright service pattern ŌĆö simpler, proven |
| CI/CD | **Job in existing `ci-pipeline.yml`** | Runs alongside Lambda deploys, same credentials |

## How the Auth Works

This is the critical design ŌĆö understanding this is essential for future work.

### Two modes, zero code changes to the main app

**Open mode** (default, same as `send.sgraph.ai`):
- No `SGRAPH_SEND__ACCESS_TOKEN` env var set
- No global middleware added
- Per-route token validation still works (if Admin Lambda is configured)
- UI served at `/send/*` with no restrictions

**Authenticated mode** (set `SGRAPH_SEND__ACCESS_TOKEN`):
- `Fast_API__SGraph__Send__Container.should_enable_global_auth()` returns `True`
- `enable_global_auth()` adds `Middleware__Check_API_Key` to the ASGI stack
- Maps `SGRAPH_SEND__ACCESS_TOKEN` ŌåÆ `FAST_API__AUTH__API_KEY__VALUE` env var
- Maps header name `x-sgraph-access-token` ŌåÆ `FAST_API__AUTH__API_KEY__NAME` env var
- **ALL routes blocked** without valid token (header or cookie) ŌĆö including static files
- **Excluded paths:** `/auth/set-cookie-form`, `/auth/set-auth-cookie`, `/docs`, `/openapi.json`, `/config/status`

### Browser login flow

1. User navigates to `http://host:8080/` ŌåÆ gets 401
2. User navigates to `http://host:8080/auth/set-cookie-form` ŌåÆ sees login page (excluded from auth)
3. User enters the access token ŌåÆ clicks "Set Cookie"
4. JavaScript POSTs to `/auth/set-auth-cookie` ŌåÆ sets `httponly` cookie with name `x-sgraph-access-token`
5. Subsequent requests include the cookie ŌåÆ middleware validates ŌåÆ 200
6. All pages (UI + API) now accessible

### API access

```bash
curl -H "x-sgraph-access-token: <token>" http://host:8080/info/health
```

### Why this doesn't affect Lambda deployments

1. Lambda uses `Fast_API__SGraph__App__Send__User` directly ŌĆö never imports `Fast_API__SGraph__Send__Container`
2. The User app sets `enable_api_key = False` (line 48) ŌĆö no global middleware
3. Static UI served by CloudFront/S3, not FastAPI
4. Per-route `check_access_token()` unchanged ŌĆö validates via Admin Lambda or env-var fallback
5. Only `Enum__Storage__Mode` and `Send__Config` were modified ŌĆö existing MEMORY/S3 paths untouched

## How the CI Pipeline Works

The `docker-build-push` job was added to `.github/workflows/ci-pipeline.yml` alongside the existing Lambda deploy jobs:

```
Run Unit Tests ŌåÆ Increment Tag ŌåÆ Check AWS Credentials
                                        Ōåō
                    ŌöīŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌö╝ŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöĆŌöÉ
                    Ōåō                   Ōåō                   Ōåō
            Admin Lambda          User Lambda        Docker ECR Push
              Deploy                Deploy          (Build + Push + Smoke)
```

### Job steps

1. Checkout + update branch (gets latest version tag)
2. Configure AWS credentials (from secrets)
3. Login to ECR (`aws-actions/amazon-ecr-login`)
4. **Auto-create ECR repo** if it doesn't exist (`aws ecr describe-repositories || create-repository`)
5. Read version from `sgraph_ai_app_send/version`
6. Build from `sgraph_ai_app_send__docker/Dockerfile` (context = repo root)
7. Push two tags: `sg-send:latest` + `sg-send:{version}`
8. Smoke test: start container, hit `/info/health`, verify 200

### Multi-account/region ready

- `ECR_REPO_NAME` env var at workflow level (currently `sg-send`)
- ECR registry host derived from AWS credentials (account + region)
- Different secrets per environment (`ci-pipeline__dev.yml`, `ci-pipeline__prod.yml`)
- ECR repo auto-created ŌĆö no manual setup per account

## How the EC2 Provisioner Works

`sgraph_ai_app_send__docker/provision_ec2.py` ŌĆö single-file Typer CLI, modelled on the Playwright service (`SGraph-AI__Service__Playwright` branch `claude/start-explorer-session-OgbPq`).

### Commands

| Command | Description |
|---|---|
| `sg-send-ec2 create` | Launch EC2 instance (AL2023, t3.large, installs Docker, pulls from ECR, runs container) |
| `sg-send-ec2 list` | List all sg-send-ec2 instances |
| `sg-send-ec2 info [target]` | Show details including access token |
| `sg-send-ec2 wait [target]` | Poll until service is healthy |
| `sg-send-ec2 health [target]` | Check service health |
| `sg-send-ec2 logs [target]` | Show setup logs via SSM |
| `sg-send-ec2 connect [target]` | Interactive SSM shell |
| `sg-send-ec2 forward [target]` | Port-forward via SSM (no firewall needed) |
| `sg-send-ec2 exec-cmd [target]` | Run one-shot command via SSM |
| `sg-send-ec2 delete [target]` | Terminate instance |

### Key patterns (from Playwright guide)

- **No SSH** ŌĆö SSM only (`AmazonSSMManagedInstanceCore` policy)
- **ECR pull** ŌĆö `AmazonEC2ContainerRegistryReadOnly` policy, credentials revoked after pull
- **Tag-based metadata** ŌĆö access token stored in `sg:access-token` tag (IAM-gated)
- **Auto-select** ŌĆö when one instance exists, target is optional
- **Random deploy names** ŌĆö `bold-curie`, `swift-tesla` (625 combinations)
- **User-data bootstrap** ŌĆö Docker installed, image pulled, container started with `--restart always`

## DISK Storage Mode

Added `Enum__Storage__Mode.DISK` backed by `memory-fs`'s existing `Storage_FS__Local_Disk` provider.

### Configuration

| Env var | Default | Description |
|---|---|---|
| `SEND__STORAGE_MODE` | `memory` | Set to `disk` for persistent local storage |
| `SEND__DISK_PATH` | `/data` | Directory for file storage |

### Auto-detection

`Send__Config.determine_storage_mode()` checks in order:
1. Explicit `SEND__STORAGE_MODE` env var
2. AWS credentials available ŌåÆ S3
3. `SEND__DISK_PATH` env var set ŌåÆ DISK
4. Default ŌåÆ MEMORY

## Docker Usage

### Quick start (ephemeral)
```bash
docker run -p 8080:8080 <ecr-registry>/sg-send
```

### With auth
```bash
docker run -p 8080:8080 -e SGRAPH_SEND__ACCESS_TOKEN=my-token <ecr-registry>/sg-send
```

### Persistent storage
```bash
docker run -p 8080:8080 -e SEND__STORAGE_MODE=disk -v sg-send-data:/data <ecr-registry>/sg-send
```

### EC2 deployment
```bash
pip install -e .
sg-send-ec2 create --token my-secret-token
sg-send-ec2 wait
sg-send-ec2 health
```

## CI Fixes During This Session

1. **`FileNotFoundError: README.md`** ŌĆö `pyproject.toml` declares `readme = "README.md"` but Dockerfile didn't COPY it. Fix: `COPY pyproject.toml README.md ./`
