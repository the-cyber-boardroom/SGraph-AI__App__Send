## Key Decisions Needing Your Review

### 1. Email-FS Stage 1 Pilot — Which Vault?

The spec is written, the SKILL is ready. Agents can start using Email-FS manually today with `cat`/`mv`/`sgit`. The question is: **which vault do we pilot in?**

Options:
- **This repo's vault** (the SGraph Send project vault) — real workflows, surfaces rough edges immediately
- **A dedicated scratch vault** — lower risk, but delays the "real" adoption signal

Recommend: pilot in a dedicated scratch vault first for one week. If no rough edges, adopt in the project vault.

### 2. Email-FS v0.6 Spec → Library

The four-document set is mature enough to be a stable reference. Consider moving from `briefs/04/29/` to `library/docs/specs/email-fs/` so it's discoverable as a stable reference (not just a dated brief).

### 3. Cross-Vault Forwarding — Design Before Multi-Team Use

The Architect review flags this as OPEN Decision #28. Before Explorer and Villager teams start using Email-FS in their respective vaults, the cross-vault forwarding semantics need to be designed. The spec leaves a forward-compatible stub but no implementation. If agents in different team vaults need to communicate before this is resolved, they'll need a workaround (e.g. a shared multi-team vault).

### 4. SnapStart Stability

The cold-start area had multiple fix commits on 04/29 (platform, env vars, STS call, retry on alias invoke). This area is still settling. The Architect recommends a full cold-start + SnapStart deploy test cycle in CI before the next Villager release.

## What's in Email-FS v0.6 (Quick Reference)

**Email-FS** = file-based agent-to-agent email on top of sgit vaults.

- **Folder structure:** `mail/{agent}/inbox/`, `processed/`, `outbox/`, `purged/`, `state/` + `mail/mailroom/{recipient}/` + `mail/sessions/active/`, `archived/`, `stale/`
- **Message format:** RFC 2822 `.eml`, immutable once written, ULID filenames
- **Five primitives:** SEND / DELIVER / READ / PROCESS / PURGE
- **Session model:** one coherent brief = one session; many check-ins per session; sessions immutable once `end.json` written
- **Core invariant:** only the named agent writes to `mail/{agent}/`; mailroom is producer-consumer (disjoint filenames)
- **Already works:** `.eml` viewer in Browse v0.3.2 renders Email-FS messages as proper email client UI

**PROPOSED — does not exist yet:** CLI (`pip install email-fs`), native inbox/thread/session views in vault UI, MCP tools, SMTP bridge.
