## Part 1 — The Problem: Why Cold Starts Were So Slow

### Original dependency loading (osbot_aws `load_dependencies`)

Each Lambda cold start was calling osbot_aws's `load_dependencies()`, which:
1. Called `boto3.client('sts').get_caller_identity()` **once per package** — 6 STS network calls for 6 packages (~900ms wasted)
2. Did one S3 download + one zip extraction per package (serial)
3. Did not reuse `/tmp` across warm invocations correctly

This produced ~8s cold starts on `main`.

### The additional root cause: pydantic_core was silently broken

`load_dependencies()` used pip platform `manylinux1_x86_64` (glibc 2.5). This is too old for `pydantic_core`, which requires `manylinux2014_x86_64` (glibc 2.17). Because `osbot_aws.Process.run()` does not check pip's exit code, pip silently skipped pydantic_core — it was never installed. The function happened to work because pydantic was not exercised on the critical path, but the import would fail if it was ever reached.

## Part 2 — Phase 1: Combined Dependency Zip

### What we built

**`sgraph_ai_app_send/_for_osbot_aws/Lambda__Dependencies__Builder.py`**

A builder that:
1. Runs `pip install` with correct cross-compilation flags into a temp directory
2. Strips the result (removes runtime packages, `__pycache__`, `.pyc`/`.pyi`, `bin/`, slims `.dist-info`)
3. Zips the result and uploads to S3 as a single file

Key pip flags:
```
--platform manylinux2014_x86_64   # Lambda default (x86_64, glibc 2.17+)
--python-version 3.13
--implementation cp
--abi cp313
--only-binary=:all:               # wheels only — no compilation on CI runner
```

The zip is **content-addressed** via SHA-256 of the sorted package list:
- Name: `sgraph-send-user-{hash12}.zip`
- Upload is skipped if the hash hasn't changed (packages unchanged)
- A single changed package bumps the hash and triggers a fresh upload

**`sgraph_ai_app_send/_for_osbot_aws/Lambda__Dependencies__Loader.py`**

A loader that:
1. Checks if already in `sys.path` (in-process reuse)
2. Checks if `/tmp/lambdas-dependencies-combined/{name}` exists (warm container `/tmp` reuse)
3. Otherwise: **one STS call** → **one S3 download** → **one zip extraction** → `sys.path.insert`

Result: combined zip alone reduced cold start from ~8.1s (main) to ~4.7s (dev). All tests passing at 697.

### Bugs encountered and fixed

**Bug 1: Wrong platform tag**
First attempt used `manylinux2014_aarch64` based on CLAUDE.md saying "arm64". Lambda's actual default is `x86_64` (`Lambda.py` line 31: `self.architecture = 'x86_64'`). Fixed to `manylinux2014_x86_64`.

**Bug 2: `importlib.metadata` needs METADATA file**
The `_strip()` step originally deleted entire `.dist-info` directories. The `mcp` package calls `importlib.metadata.version('mcp')` at import time, which reads `.dist-info/METADATA`. This crashed the Lambda with `PackageNotFoundError`.

Fix: instead of deleting `.dist-info` directories, iterate their contents and delete everything **except** `METADATA`. This preserves `importlib.metadata` compatibility while still removing `RECORD`, `WHEEL`, `INSTALLER`, and `top_level.txt`.

**Zip size journey:**
- Raw pip install: ~38MB
- After removing runtime packages (boto3, botocore, s3transfer, jmespath, dateutil, six, urllib3) + pycache + bin: ~12.5MB
- After restoring METADATA in dist-info: ~12.9MB

## Part 4 — Phase 2: SnapStart

### What SnapStart does

AWS Lambda SnapStart (Python GA: November 2024, Python 3.12+) snapshots the Firecracker microVM state **after** the Init phase completes. On subsequent cold starts, Lambda restores from the snapshot instead of re-running Init.

**Critical constraint:** SnapStart only applies to **published version aliases**, NOT to `$LATEST`. The `$LATEST` Function URL always does a full cold start regardless.

### What we implemented

**`Deploy__Service.enable_snapstart()`** — a deploy-time method that:

1. Enables SnapStart on the function: `configuration_update(SnapStart={'ApplyOn': 'PublishedVersions'})`
2. Waits for the configuration update to complete
3. Publishes a new version: `lf.client().publish_version(FunctionName=lf.name)`
4. Creates or updates a `snapstart` alias pointing to the published version
5. Creates a Function URL for the alias (once only, guarded by `function_url_exists`)
6. Ensures both required permissions exist on the alias URL (idempotent)

**Test coverage** in `test_Deploy__User__Service__base.py`:
```python
def test_5b__enable_snapstart(self):
    result = self.deploy_fast_api.enable_snapstart()
    assert result['version']   != '$LATEST'
    assert result['alias']     == 'snapstart'
    assert result['alias_url'].startswith('https://')

def test_7b__invoke__snapstart_url(self):
    assert self.deploy_fast_api.invoke__snapstart_url('/info/health') == {'status': 'ok'}
```

### The alias URL is permanent

The alias URL never changes between deployments:
```
Deploy v1 → publish "1" → create alias + URL (once)
Deploy v2 → publish "2" → update_alias("snapstart", "2") → same URL
Deploy v3 → publish "3" → update_alias("snapstart", "3") → same URL
```

Update CloudFront (or DNS) once to point to the alias URL. All future deploys are transparent.

### Bug: missing `lambda:InvokeFunction` permission

The alias Function URL returned HTTP 403 Forbidden. The AWS console warning explained it:

> "Your function URL auth type is NONE, but is missing permissions required for public access. Create a resource-based policy that grants **lambda:invokeFunction** and **lambda:invokeFunctionUrl** permissions to all principals."

Our initial `add_permission` call only added `lambda:InvokeFunctionUrl`. The fix adds both, and moves permission setup outside the URL-creation guard so it runs on **every deployment** (idempotent via try/except):

```python
def _ensure_alias_url_permissions(self, lf, fn_name, alias):
    permissions = [
        (f'public-access-{alias}', 'lambda:InvokeFunctionUrl', {'FunctionUrlAuthType': 'NONE'}),
        (f'public-invoke-{alias}',  'lambda:InvokeFunction',   {}),
    ]
    for stmt_id, action, extra in permissions:
        try:
            lf.client().add_permission(FunctionName=fn_name, Qualifier=alias,
                                       StatementId=stmt_id, Action=action,
                                       Principal='*', **extra)
        except Exception:
            pass  # Statement already exists — idempotent
```

A second bug: the original code had `lf.function_url_create_with_public_access.__func__` — accessing the unbound function object without calling it. This was a no-op. Removed.

## Part 5 — Concurrency Probe Test Script

**`tests/qa/test_cold_start__concurrent.py`** — a standalone CLI tool (no pytest, pure stdlib) that sends bursts of concurrent requests and colour-codes response times:

```
Green  ≤300ms   warm / SnapStart restore
Yellow ≤1500ms  SnapStart restore (possible)

```

Usage:
```bash
python tests/qa/test_cold_start__concurrent.py --env both --concurrency 20 --bursts 3
```

The `--env both` comparison mode reports per-environment: cold-start count, cold-start avg, warm p50, and worst case — making it easy to see SnapStart vs classic side by side.

## Part 7 — Guide: Applying This to Another Lambda Function

### Prerequisites
- Python 3.12 or 3.13 runtime
- x86_64 architecture (Lambda default — verify with `Lambda.py`)
- `osbot_fast_api_serverless` deploy base class

### Step 1: Add combined zip support to the Deploy class

```python
from sgraph_ai_app_send._for_osbot_aws.Lambda__Dependencies__Builder import Lambda__Dependencies__Builder

PACKAGES = ['your-package-list']
SERVICE_NAME = 'your-service-name'  # e.g. 'sgraph-send-admin'

def upload_combined_dependencies_to_s3(self, force=False):
    builder = Lambda__Dependencies__Builder(SERVICE_NAME, PACKAGES)
    return builder.upload(force=force)
```

### Step 2: Add combined zip loading to the lambda handler

```python
import os, time
_t_init = time.time()

if os.getenv('AWS_REGION'):
    from sgraph_ai_app_send._for_osbot_aws.Lambda__Dependencies__Loader import load_combined_dependency
    load_combined_dependency('your-service-name', YOUR_PACKAGES)
    print(f'[init] deps loaded — {(time.time()-_t_init)*1000:.0f}ms')
```

### Step 3: Add SnapStart to the Deploy class

Copy `enable_snapstart()` and `_ensure_alias_url_permissions()` from `Deploy__Service` verbatim — they are generic and reusable.

### Step 4: Add deploy tests

Copy `test_5b__enable_snapstart` and `test_7b__invoke__snapstart_url` into the function's deploy test base class.

### Step 5: Deploy sequence

Run these steps in order (they correspond to the numbered tests):
1. `test_2b` — upload combined zip to S3 (only uploads if packages changed)
2. `test_4` — create/update Lambda function
3. `test_5` — update runtime to Python 3.13
4. `test_5b` — enable SnapStart, publish version, create/update alias + URL
5. `test_7b` — verify alias URL responds correctly

### Step 6: Update CloudFront

After `test_5b` prints the alias URL:
- Update the CloudFront distribution origin (or Route 53 CNAME) to the alias URL
- The `$LATEST` Function URL continues to exist but is not used for production traffic
- The alias URL is permanent — do this once only

### Gotchas

| Issue | Cause | Fix |
|-------|-------|-----|
| `pydantic_core` missing | Wrong platform tag (`manylinux1` or `aarch64`) | Use `manylinux2014_x86_64` |
| `PackageNotFoundError` for mcp/fastapi | `_strip()` deleted all `.dist-info` | Keep `METADATA` file inside each `.dist-info` |
| Alias URL returns 403 | Only `lambda:InvokeFunctionUrl` added, missing `lambda:InvokeFunction` | Add both; use `_ensure_alias_url_permissions()` |
| SnapStart has no effect | `dev.send.sgraph.ai` still points to `$LATEST` URL | Update CF to alias URL |
| Permissions not added on redeploy | `add_permission` gated inside URL-creation guard | Move `_ensure_alias_url_permissions` outside the guard |

### What SnapStart cannot help with

- **Warm container reuse** — SnapStart only fires on cold start. Warm invocations are unaffected.
- **`$LATEST`** — SnapStart never applies to `$LATEST`, only to published version aliases.
- **I/O in Init** — if Init opens network connections, SnapStart restores the snapshot state from when they were open. For short-lived clients (boto3, httpx) this is fine. Long-lived connections may need post-restore hooks (rare in FastAPI Lambda patterns).
- **Python < 3.12** — SnapStart for Python requires Python 3.12+.
