## 1. VFS Bridge (HTML viewer)

**Problem:** HTML files displayed in a sandboxed `<iframe src="blob:...">` have no base URL. Any relative `fetch()` call (e.g. `fetch('content/manifest.json')`) fails immediately because the blob URL carries no path context.

**Solution:** Before creating the blob URL, inject a `<script>` into the HTML that overrides:
- `window.fetch()` — intercepts relative URLs, sends them to the parent via `postMessage`
- `HTMLImageElement.prototype.src` setter — intercepts `img.src = 'relative/path'` assignments at the moment they happen, before the browser makes any network request
- `MutationObserver` — backup for `img.setAttribute('src', ...)` calls

The parent window receives each `__sgVfsReq` message, resolves the relative path against the HTML file's vault folder, fetches and decrypts the bytes via `dataSource.getFileBytes()`, then returns an `ArrayBuffer` via `postMessage` with transfer (zero-copy).

This works for both **vault** (`VaultDataSource`) and **zip** (`ZipDataSource`) — the VFS bridge is data-source agnostic.

**Key insight — `HTMLImageElement.prototype.src` setter vs MutationObserver:**  
Presenters create `new Image()` elements and set `img.src` before appending to the DOM. A MutationObserver on `document` misses these. Overriding the prototype setter intercepts the assignment at the point of creation, regardless of DOM state.

**`__sgVfs` flag lifecycle:**  
The flag is set `true` before the async VFS fetch (prevents MutationObserver double-processing while in-flight), then cleared `false` in the `_vfsReq` callback. This allows slide navigation to update the same `<img>` element's `src` multiple times — each relative path is intercepted correctly.

### Security properties
- The iframe is sandboxed (`allow-scripts` only, no `allow-same-origin`)
- The iframe cannot access parent DOM, localStorage, or cookies
- Only messages from that specific iframe's `contentWindow` are handled
- No external URLs are loaded — all assets come from the vault

## 2. Immutable Block Cache

**Problem:** Every vault page load re-fetches all encrypted blocks from `send.sgraph.ai` and re-decrypts them. For a 251 MB vault with 155 files this was ~6 seconds on cold load.

**Solution:** `obj-cas-imm-*` blocks are content-addressed — the object ID is `SHA-256(ciphertext)[:12]`, so the same ID guarantees the same bytes forever. They can be cached permanently.

Cache location: browser Cache API (same origin as the vault page):

```
Cache name:  sg-vault-blocks
Cache key:   https://sgvault/{vaultId}/bare/data/{objectId}
             e.g. https://sgvault/jrinl85l/bare/data/obj-cas-imm-a2ad790bd3f1
```

The `{vaultId}` prefix scopes entries per vault, preventing collisions across vaults. The `-imm-` substring in the object ID is the guard — only immutable blocks are cached; `muw`/`snw` refs (which can change) never pass through this path.

**What is cached:** encrypted ciphertext, exactly what the server stores. Decryption still happens in the browser on each page load (pure CPU, ~ms). The expensive operation — the network round-trip to `send.sgraph.ai` — is eliminated for cached blocks.

**Measured results:**

| Condition | Load time |
|---|---|
| Cold (no cache, cache disabled) | ~6 s |
| Second load (cache populated) | ~1 s |
| Warm cache (browser cache + sg-vault-blocks) | 820 ms |

**Storage management API** (for a future settings UI):

```js
// List all cached blocks
const cache = await caches.open('sg-vault-blocks')
const keys  = await cache.keys()  // → array of Request objects, URL = https://sgvault/{vaultId}/...

// Clear one vault's cache
for (const req of keys) {
    if (req.url.includes('/jrinl85l/')) await cache.delete(req)
}

// Clear all vault blocks
await caches.delete('sg-vault-blocks')
```

## 3. Local Dev Server — CDN Override

**Problem:** `vault__run-locally.sh` served the vault's own `_common/` locally but `index.html` loaded `send-browse--v0.3.2.js` and all shared components from `https://dev.send.sgraph.ai/_common/` (CDN). Local edits to those files had no visible effect.

**Fix:**
1. Copy locale dirs (e.g. `en-gb/`) instead of symlinking — so `index.html` can be patched in place
2. Merge all IFD user UI layers in order on top of vault `_common/`:  
   `v0.3.0/_common` → `v0.3.1/_common` → `v0.3.2/_common`  
   Each layer only contains files changed in that version; later layers override earlier ones
3. Patch all `index.html` files: replace `https://dev.send.sgraph.ai/_common/` → `/_common/`

This exactly replicates the CDN's flattened view. After restarting the server, the browser loads local files — confirmed by the version stamp `[send-browse v0.3.2-vfs-N] loaded OK` appearing in the console.

## 4. Expected 404s (not bugs)

The network tab shows two 404s on every vault open that hit `dev.send.sgraph.ai`:

| Request | Why it 404s | Why it's OK |
|---|---|---|
| `bare/refs/ref-pid-snw-...` | Clone branch ref — created lazily on first web-UI commit; doesn't exist for fresh/read-only vaults | `readRef()` returns `null` on 404; vault falls back to named HEAD |
| `bare/idx/idx-pid-muw-...` | Branch index — only created by `sgit`; not present on older vaults | `readBranchIndex()` returns `null`; vault uses derived `refFileId` |

These are correct zero-knowledge behaviour — the server returns 404 for any file that doesn't exist, with no information leak. The vault handles both gracefully.
