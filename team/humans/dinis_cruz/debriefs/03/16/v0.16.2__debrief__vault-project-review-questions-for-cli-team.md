# Debrief: Questions for CLI Team — Browser Adapting to CLI's bare/ Path Scheme

**Version:** v0.16.2
**Date:** 2026-03-16
**Team:** Explorer

---

## Context

We've confirmed the root cause of the vault path mismatch and agreed the **browser will adapt to the CLI's model** (not the other way round). To implement this, the vault browser team needs answers to the following questions from the CLI team.

---

## Questions

### 1. Ref path convention — deterministic or discovered?

The brief shows `bare/refs/ref-7b2e1d`. Is that ID:
- **A well-known name** like `bare/refs/main` or `bare/refs/HEAD` that the browser can hardcode?
- **A UUID/random** that requires discovery via `GET /api/vault/list/{vault_id}?prefix=bare/refs/`?

If it's random, is there always exactly one ref, or can there be multiple (one per branch)?

### 2. Branch index — same question

`bare/indexes/idx-m8n4r1` — is this a fixed name like `bare/indexes/main`, or does the browser need to list and pick?

### 3. Content objects — same `bare/data/` prefix?

Does the CLI store commit, tree, and blob objects at `bare/data/obj-{hash}`? Or just `obj-{hash}` (flat, no prefix)? The browser currently stores objects as `obj-{sha256[:12]}` at the vault root — do we need to add the `bare/data/` prefix?

### 4. Exact encrypted payload format for refs

The browser currently expects the HEAD ref to decrypt (AES-256-GCM with `readKey`) to:
```json
{ "commit_id": "obj-xxxxxxxxxxxx" }
```

Does the CLI write the same JSON schema? Or does it use a different structure (e.g. `{ "ref": "...", "branch": "main" }`)?

### 5. Commit object schema

The browser expects commits to decrypt to:
```json
{
  "schema": "commit_v1",
  "parents": [],
  "tree_id": "obj-xxxxxxxxxxxx",
  "timestamp_ms": 1710000000000,
  "message": "...",
  "branch_id": "main",
  "signature": null
}
```

Does the CLI use the same `commit_v1` schema, or a different version?

### 6. Tree entry schema

The browser expects tree entries with encrypted fields (`name_enc`, `size_enc`, `content_hash_enc`) and plaintext pointers (`blob_id`, `tree_id`). Does the CLI use the same tree format, or does it structure entries differently?

### 7. Python `Vault__Crypto` mismatch

The Python `Vault__Crypto.derive_keys()` returns `tree_file_id` and `settings_file_id` (domains `sg-vault-v1:file-id:tree:` and `sg-vault-v1:file-id:settings:`). The JS derives `refFileId` and `branchIndexFileId` (domains `sg-vault-v1:file-id:ref:` and `sg-vault-v1:file-id:branch-index:`). Neither matches. Since the CLI doesn't use HMAC-derived IDs anyway, should we remove both and standardise on the `bare/` paths exclusively?

---

## What Happens Next

Once we have these answers, we can update:
- `sg-vault.js`
- `sg-vault-ref-manager.js`
- `sg-vault-crypto.js`

to align with the CLI's model.
