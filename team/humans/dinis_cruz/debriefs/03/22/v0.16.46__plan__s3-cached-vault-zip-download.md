# Plan: S3-Cached Vault Zip Download

**Version:** v0.16.46
**Date:** 2026-03-22
**Team:** Explorer
**Status:** PROPOSED — does not exist yet

---

## Problem

Downloading an entire vault as a zip currently requires the Lambda to:
1. Read every file from storage into memory
2. Build the zip in-memory
3. Return it in the HTTP response body

This hits Lambda's 6 MB response limit, can't handle large vaults, and rebuilds the zip from scratch every time — even if nothing changed.

## Proposed Solution

Store generated zips in S3 with a **content-addressable hash** as the cache key. Return a **presigned download URL** instead of raw bytes.

### Three wins

| Win | Description |
|-----|-------------|
| **No size limit** | Zip lives in S3, downloaded via presigned URL — bypasses Lambda response limit |
| **Cache reuse** | Same content hash → same zip. Repeated downloads are instant (no rebuild) |
| **Future delta path** | Old zips preserved in S3. Future versions can diff file lists and only fetch changed files when rebuilding |

---

## Design

### Content Hash (Cache Key)

```
content_hash = SHA-256( sorted( f"{file_id}:{file_size}" for each file in vault ) )
```

- Deterministic — same files in same vault always produce the same hash
- Changes when any file is added, removed, or modified (size change)
- Computed from `list_files()` + file sizes — no need to read file contents

### S3 Storage Path

```
vault-zips/{vault_id}/{content_hash}.zip
```

- One zip per unique vault state
- Old zips preserved (enables future delta optimization)
- Same Storage_FS abstraction used by all other storage

### Flow

```
Client: GET /api/vault/zip/{vault_id}
        Header: x-sgraph-vault-write-key: <key>

Server:
  1. Verify write-key against vault manifest
  2. List vault files → compute content_hash
  3. Check if vault-zips/{vault_id}/{content_hash}.zip exists in S3
     ├─ YES (cache hit)  → generate presigned GET URL → return
     └─ NO  (cache miss) → read all files, build zip, upload to S3 → presigned URL → return
  4. Return JSON: { url, content_hash, cached, file_count }
```

### Response Format

```json
{
  "status"       : "ok",
  "vault_id"     : "a1b2c3d4",
  "content_hash" : "e3b0c44298...",
  "file_count"   : 42,
  "cached"       : true,
  "download_url" : "https://s3.../vault-zips/a1b2c3d4/e3b0c44298...zip?X-Amz-..."
}
```

In memory mode (dev/test), the zip bytes are returned directly in the response body since there's no S3 for presigned URLs.

## Implementation Plan

### Step 1: `Service__Vault__Zip` (new service)

**File:** `sgraph_ai_app_send/lambda__user/service/Service__Vault__Zip.py`

| Method | Purpose |
|--------|---------|
| `vault_content_hash(vault_id)` | List files + sizes → SHA-256 hash |
| `zip_s3_path(vault_id, content_hash)` | Returns `vault-zips/{vault_id}/{content_hash}.zip` |
| `zip_exists(vault_id, content_hash)` | Check if cached zip exists in storage |
| `build_zip(vault_id)` | Read all files, create zip in-memory, return bytes |
| `get_or_create_zip(vault_id, write_key_hex)` | Full flow: auth → hash → cache check → build if needed → upload → return result |

Dependencies:
- `Service__Vault__Pointer` — for auth verification and file reading
- `Storage_FS` — for zip storage (same backend as everything else)
- `Service__Presigned_Urls` — for generating download URLs (S3 mode)

### Step 2: Route `GET /api/vault/zip/{vault_id}`

**File:** `sgraph_ai_app_send/lambda__user/fast_api/routes/Routes__Vault__Pointer.py` (add to existing routes)

- Requires `x-sgraph-vault-write-key` header
- Returns JSON with presigned URL (S3 mode) or zip bytes (memory mode)
- Registered alongside existing vault pointer routes

### Step 3: Tests

**Service tests:** `tests/unit/lambda__user/service/test_Service__Vault__Zip.py`
- Content hash determinism (same files → same hash)
- Content hash changes when files added/removed/modified
- Cache miss → builds zip → uploads

- Auth failure (wrong write key) → rejected

**Route tests:** `tests/unit/lambda__user/fast_api/routes/test_Routes__Vault__Zip.py`

## Auth Model

**Write-key required.** Bulk export of an entire vault is a privileged operation — same auth level as write/delete. Read-only access (individual file reads) remains unauthenticated per the zero-knowledge model.

---

## Future: Delta Zip Optimization (NOT built now)

The design preserves old zips at `vault-zips/{vault_id}/{old_hash}.zip`. A future version could:

1. Accept optional `previous_hash` parameter
2. Diff the file list between old and new state
3. Download the old zip from S3, add/remove only changed files
4. Upload the new zip

This would make zip regeneration proportional to the **delta**, not the total vault size. The current design is forward-compatible with this — no schema changes needed.

## Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Storage backend | Same `Storage_FS` | Project rule — no direct S3/boto3 calls |
| Cache key | Content hash (file_ids + sizes) | Deterministic, no false positives |
| Auth level | Write-key | Bulk export is privileged |
| Delivery | Presigned URL (S3) / bytes (memory) | No Lambda size limit in production |
| Old zips | Preserved | Enables future delta optimization |
| Zip path | `vault-zips/{vault_id}/{hash}.zip` | Namespaced, no collisions |
