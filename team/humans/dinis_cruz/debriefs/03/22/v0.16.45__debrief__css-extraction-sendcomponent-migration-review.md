## What Was Done

A Claude Code session executed a systematic refactoring of the v0.3.0 UI codebase: extracting inline CSS into separate files, evolving the `SendComponent` base class, and migrating 10 Web Components from `extends HTMLElement` to `extends SendComponent`.

This was reviewed by a separate Claude Web session (Dev agent) that verified all changes against the current dev branch, confirmed no functional regressions, and approved the merge.

---

## The Problem Before This Refactoring

The v0.3.0 codebase had 18+ Web Components that all extended `HTMLElement` directly. Each reinvented resource loading in its own way — 4 distinct patterns scattered across the codebase:

| Pattern | Components | Problem |
|---------|-----------|---------|
| Manual `SendComponentPaths.basePath` lookup | 8 upload-step-* components | 8+ copies of the same availability check |
| `static CSS` string literal in JS | send-browse, send-gallery, send-viewer | CSS mixed into JS — not cacheable, not inspectable in DevTools |
| `style.textContent` injection | send-step-indicator, send-access-gate | Inline CSS blocks embedded in JS |
| Pure `innerHTML` with inline styles | send-header, send-footer, send-locale | No separation at all |

Meanwhile, `SendComponent` — a well-designed base class adapted from MGraph — existed in the codebase but zero components used it.

## What Changed (10 commits, 23 files, +1,644 / -1,479 lines)

### 1. SendComponent evolved (3 small additions)

| Addition | Purpose |
|----------|---------|
| `static useShadow = true` | Opt-out flag for light DOM components (header, footer, locale) |
| `static useTemplate = true` | Opt-out for components that generate HTML dynamically (gallery, browse, viewer) |
| `get renderRoot()` | Returns `shadowRoot` or `this` — single abstraction for both modes |
| `get basePath()` | Centralised path resolution, eliminates 8+ scattered availability checks |
| `_injectHeadCss()` | Light DOM CSS loading via `<link>` in `document.head`, deduplicated with a `Set` |

### 3. Ten components migrated to `extends SendComponent`

| Component | Migration type | What was removed |
|-----------|---------------|-----------------|
| send-viewer | Full (CSS-only, light DOM) | `static CSS` string, boilerplate |
| send-gallery | Full (CSS-only, light DOM) | `static CSS` string, `_injectCss()` |
| send-browse | Full (CSS-only, light DOM) | `static CSS` string, `_injectCss()` |
| send-step-indicator | Full (Shadow DOM) | Inline `style.textContent`, manual CSS loading |
| send-header | Partial (utilities only) | Direct `HTMLElement` extension |
| send-footer | Partial (utilities only) | Direct `HTMLElement` extension |
| send-locale | Partial (utilities + `$`/`$$`) | Direct `HTMLElement` extension, `querySelector` calls |
| upload-step-delivery | Full (Shadow DOM + template) | Manual template loading, basePath resolution, `_esc()` helper |
| upload-step-share | Full (Shadow DOM + template) | Manual template loading, `_resolveBasePath()`, CSS `<link>` creation |

### 4. Alias support added to SendComponentPaths

Components that live in non-standard directories (e.g. `send-viewer` lives in `send-download/`) register via `SendComponentPaths.alias('send-viewer', 'send-download')`. This is called in the HTML page init blocks.

## Why This Matters — IFD v2 Methodology in Practice

This refactoring demonstrates a key principle of IFD v2: **major versions are consolidation checkpoints, and consolidation includes architectural cleanup.**

### Before: v0.3.0 was a functional consolidation

The v0.2.x → v0.3.0 work (patches 0015–0019) made v0.3.0 feature-complete and self-contained. But the internal architecture still carried technical debt: CSS mixed into JS, duplicated boilerplate, an unused base class.

### After: v0.3.0 is a structural consolidation

The CSS extraction + SendComponent migration makes the codebase ready for the next phase of development:

**CSS is cacheable.** Browsers can cache `send-gallery.css` independently of `send-gallery.js`. Previously, changing one CSS property required downloading the entire JS file again.

**CSS is inspectable.** DevTools can show, edit, and hot-reload CSS from `.css` files. Inline CSS in JS template literals was invisible to the CSS inspector.

**New components inherit infrastructure.** Any new component created for v0.3.x can `extend SendComponent` and get Shadow DOM encapsulation, CSS loading, event tracking, and lifecycle hooks for free — instead of copying 40+ lines of boilerplate.

**Separation of concerns is enforced.** The pattern is now: `.js` for behaviour, `.css` for presentation, `.html` for structure (Shadow DOM components). Each is independently editable.

**The base class is a single point of improvement.** Enhancements to `SendComponent` (e.g. theme support, performance instrumentation, error boundaries) automatically flow to all 10+ components that extend it.

### This is the IFD v2 rhythm

1. **Explorer builds features** (v0.2.0 → v0.2.17 overlay chain)
2. **Major version consolidates** (v0.3.0 — functional parity, self-contained)
3. **Architecture improves within the major version** (CSS extraction, base class migration)
4. **Villager hardens** (deployment, CI pipeline, acceptance tests)

Steps 2 and 3 happened in the same day across two parallel Claude sessions — one doing functional parity (Claude Web via vaults), the other doing structural cleanup (Claude Code). Both worked independently, merged cleanly, and produced a codebase that's materially better than what either could achieve alone.
