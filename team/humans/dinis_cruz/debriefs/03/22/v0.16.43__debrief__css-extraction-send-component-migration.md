# Debrief: CSS Extraction & SendComponent Migration Plan

**Version:** v0.16.43
**Date:** 2026-03-22
**Branch:** `claude/refactor-v0.3.0-ifd-qie25`
**Status:** PLAN — implementation about to begin

---

## Context

Following the Shadow DOM sub-component extraction (v0.16.33 debrief), the codebase has 18+ Web Components that all `extend HTMLElement` directly. Each reinvents resource loading (CSS, HTML templates, basePath resolution) in its own way — 4 distinct patterns across the codebase.

Meanwhile, `SendComponent` (`_common/js/base/send-component.js`) already solves this problem cleanly — Shadow DOM encapsulation, parallel CSS/HTML fetch, event tracking, lifecycle hooks, error display — but **zero components use it**.

This session will fix that: extract inline CSS into files, evolve `SendComponent` slightly, and migrate components to extend it.

---

## What Exists Today (Code-Verified)

### SendComponent base class (`_common/js/base/send-component.js`)
- `loadResources()` — fetches shared CSS + component CSS + HTML template in parallel
- `fetchCss(url)` / `fetchHtml(url)` — with graceful degradation
- `bindElements()`, `setupEventListeners()`, `onReady()` — lifecycle hooks
- `emit()`, `addTrackedListener()`, `$()`, `$$()` — event/DOM helpers
- `escapeHtml()`, `formatBytes()`, `copyToClipboard()` — delegates to `SendHelpers`
- `whenReady(timeout)` — promise-based ready gate
- `cleanup()` — tracked listener removal on disconnect

### SendComponentPaths (`_common/js/base/send-component-paths.js`)
- `init(path)` — set basePath at boot
- `resolve(tagName)` — returns `{js, html, css}` paths for a component
- `sharedCss.components()` — shared CSS path

### Current component patterns (all extend HTMLElement directly)

| Pattern | Components | Problem |
|---------|-----------|---------|
| Manual `SendComponentPaths.basePath` lookup | upload-step-select, upload-step-progress, upload-step-confirm, upload-step-share, upload-step-delivery, upload-step-done, send-test-files | 8+ copies of `typeof SendComponentPaths !== 'undefined' && SendComponentPaths.basePath` |
| `static CSS` string literal in JS | send-browse, send-gallery, send-viewer | CSS mixed into JS, not cacheable |
| `style.textContent` injection | send-step-indicator, send-access-gate | Inline CSS blocks in JS |
| Pure `innerHTML` with inline styles | send-header, send-footer, send-locale, send-transparency, send-welcome, send-download | No separation at all |

## Implementation Plan

### Phase 1: Evolve SendComponent (small additions)

Add to `SendComponent`:

1. **`static useShadow = true`** — flag for components that use light DOM (send-header, send-footer). When `false`, renders into `this` instead of `this.shadowRoot`.
2. **CSS-only mode** — `loadResources()` already handles missing CSS gracefully; make it also handle missing HTML gracefully (return empty string instead of throwing). Components like send-browse generate HTML dynamically.
3. **`get basePath()`** — single getter to replace the 8+ copies of the `SendComponentPaths` availability check scattered across components.

### Phase 2: CSS Extraction (Tier 1 — static CSS components)

Extract inline CSS from components that have CSS as string literals in JS:

| Component | Current CSS location | Target file |
|-----------|---------------------|-------------|
| `send-browse` | `static CSS` in JS (~80 lines) | `send-browse/send-browse.css` |
| `send-gallery` | `static CSS` in JS (~60 lines) | `send-gallery/send-gallery.css` |
| `send-viewer` | `static CSS` in JS (~50 lines) | `send-viewer/send-viewer.css` |
| `send-step-indicator` | `style.textContent` in JS (~90 lines) | `send-step-indicator/send-step-indicator.css` |
| `send-access-gate` | Inline in JS (~100 lines) | `send-access-gate/send-access-gate.css` |

### Phase 3: Migrate Tier 1 to `extends SendComponent`

Convert each Tier 1 component from `extends HTMLElement` to `extends SendComponent`, removing all boilerplate resource-loading code.

### Phase 4: CSS Extraction (Tier 2 — upload step components)

These already have `.css` files but still have some inline CSS. Clean up and ensure full separation.

### Phase 5: Migrate remaining components

Convert upload step components and other components to `extends SendComponent`.

---

## Key Design Decisions

1. **Incremental migration** — each commit is independently testable. No big-bang rewrite.
2. **SendComponent evolves, not replaced** — it's already well-designed (from MGraph). We add 3 small features, not a new base class.
3. **CSS extraction first, then migration** — separating concerns before changing inheritance ensures each step is verifiable.
4. **No behaviour changes** — pure refactoring. Components render identically before and after.

---

## Risk Mitigation

- **Shadow DOM vs Light DOM** — the `useShadow` flag handles components that can't use Shadow DOM
- **Dynamic HTML generators** — CSS-only mode means send-browse/send-gallery keep their dynamic `render()` methods
- **basePath resolution** — centralised in `SendComponent.basePath` getter, falls back to `'../_common'`
- **Existing `.html` templates** — upload-step-* components already have `.html` files; they'll work as-is with SendComponent
