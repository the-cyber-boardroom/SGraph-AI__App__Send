# Debrief — P0 Fix: Non-ISO-8859-1 Header Error on Upload

**Version:** v0.12.3
**Date:** 2026-03-08
**Severity:** P0 — Customer-reported upload failure
**Status:** Fix committed and pushed

---

## Executive Summary

A customer reported that after uploading one file successfully, subsequent uploads failed with:

> Failed to execute 'fetch' on 'Window': Failed to read the 'headers' property from 'RequestInit': String contains non ISO-8859-1 code point.

**Root cause:** The `x-sgraph-access-token` HTTP header contained characters outside the ISO-8859-1 range (code points > U+00FF). Browsers enforce this limit on all HTTP header values per the Fetch specification.

**Fix:** Added `_sanitiseToken()` to strip non-printable/non-ASCII characters from the token at both storage time and usage time.

---

## Technical Analysis

### The Error

The browser's `fetch()` API validates all header values before sending a request. If any value contains a character with code point > 255 (outside ISO-8859-1), the request is rejected immediately — it never reaches the network.

### Why the Token Is the Culprit

Every `fetch()` call in the upload flow was audited. The headers are:

| Fetch call | Headers |
|---|---|
| `createTransfer` | `Content-Type: application/json` (static), `x-sgraph-access-token: <token>` |
| `uploadPayload` | `Content-Type: application/octet-stream` (static), `x-sgraph-access-token: <token>` |
| `completeTransfer` | `x-sgraph-access-token: <token>` |

All `Content-Type` values are hardcoded ASCII strings. The **only dynamic header value** is the access token from `localStorage`.

### How Non-ASCII Characters Enter the Token

**Two entry points** were identified:

#### 1. Welcome Page (`send-welcome.js:114`)

```javascript
const tokenString = new TextDecoder().decode(content).trim();
```

If the encrypted token payload has encoding issues (corruption, wrong key, partial decryption), `TextDecoder` can produce strings with arbitrary unicode characters including code points > U+00FF.

#### 2. Copy-Paste in Access Gate (`send-access-gate.js:118`)

```javascript
const token = (input.value || '').trim().toLowerCase();
```

When users copy tokens from rich text sources (formatted emails, Slack, Word documents), invisible unicode characters can be embedded:
- Zero-width spaces (U+200B) — **above ISO-8859-1**
- Zero-width non-breaking spaces (U+FEFF) — **above ISO-8859-1**
- Right-to-left marks (U+200F) — **above ISO-8859-1**
- Smart quotes, em dashes — some above, some below

`.trim()` and `.toLowerCase()` do **not** remove these invisible characters.

### Why Token Validation Didn't Catch It

Both the welcome page and access gate have a dangerous fallthrough pattern:

```javascript
try {
    const result = await ApiClient.checkToken(token);
    if (result.valid === false) { /* show error, return */ }
} catch (e) {
    // Token check endpoint unavailable — fall through   ← SILENT SKIP
}
ApiClient.setAccessToken(token);   // Stored regardless of validation failure
```

Additionally, `checkToken()` sends the token via **URL encoding** (`encodeURIComponent(token)` in the path), which handles any unicode. But the upload endpoints send the same token as a **header value**, which rejects non-ISO-8859-1. So validation can succeed where upload fails.

### Why First Upload Worked, Second Didn't

Most likely scenario: The customer entered a clean ASCII token initially (first upload succeeded), then something overwrote the localStorage token between uploads — either:
- Welcome page activation in another tab (decrypted a corrupted token)
- Re-entry of the token from a different rich-text source with invisible unicode
- Another page/script writing to the same `sgraph-send-token` localStorage key

## Fix Applied

**File:** [`sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/_common/js/api-client.js`](../../../../../sgraph_ai_app_send__ui__user/v0/v0.2/v0.2.0/_common/js/api-client.js)

### Changes

1. **New `_sanitiseToken(token)` method** — strips anything outside printable ASCII (`\x20-\x7E`)

2. **`setAccessToken(token)`** — now sanitises before storing (prevention at source)

3. **`_authHeaders()`** — now sanitises before using as header value (defence in depth, catches tokens stored before the fix)

```javascript
_sanitiseToken(token) {
    if (!token) return token;
    return token.replace(/[^\x20-\x7E]/g, '');
},

setAccessToken(token) {
    localStorage.setItem('sgraph-send-token', this._sanitiseToken(token));
},

_authHeaders() {
    const token = this.getAccessToken();
    if (!token) return {};
    const safe = this._sanitiseToken(token);
    return safe ? { 'x-sgraph-access-token': safe } : {};
},
```

### Why Defence in Depth

- `setAccessToken` catches new tokens at write time
- `_authHeaders` catches tokens already in localStorage from before the fix (existing customers who may have corrupted tokens stored)

## Risk Assessment

| Risk | Mitigation |
|---|---|
| Sanitisation silently modifies a valid token containing non-ASCII chars | Access tokens should be ASCII-only by design. Non-ASCII token names are not supported by the admin system. |
| Existing corrupted tokens in customer localStorage cause silent auth failures | `_authHeaders` sanitises on read, so the cleaned token will be sent. If the server stored the token as ASCII, the cleaned version will match. |
| Regex `[^\x20-\x7E]` is too aggressive (strips extended Latin chars) | Token names should be alphanumeric + hyphens. Extended Latin (U+0080–U+00FF) in token names would still work in headers but is unusual. Keeping to printable ASCII is the safest approach. |

---

## Recommended Follow-ups

1. **Harden checkToken fallthrough** — The `catch (e) { /* fall through */ }` pattern in both `send-welcome.js` and `send-access-gate.js` should be tightened. A transient server error should not silently accept any token.

2. **Server-side token name validation** — When admins create tokens, validate that names are ASCII-only to prevent this class of issue entirely.

3. **User feedback on sanitisation** — If `_sanitiseToken` changes the token (i.e., characters were stripped), consider showing a warning to the user so they know their input was modified.

4. **Add telemetry** — Log (client-side) when token sanitisation strips characters, to understand how often this happens in practice.
