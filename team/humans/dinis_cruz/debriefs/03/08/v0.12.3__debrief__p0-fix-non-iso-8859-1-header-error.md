# Debrief — P0 Fix: Non-ISO-8859-1 Header Error on Upload

**Version:** v0.13.0
**Date:** 2026-03-08
**Severity:** P0 — Customer-reported upload failure (blocks all subsequent uploads)
**Status:** All 4 fixes implemented and pushed
**Source brief:** [v0.12.6 — Corrupted Token ISO-8859-1 Bug](../../../briefs/03/08/v0.12.6__debrief__corrupted-token-iso8859-bug.md)

---

## Executive Summary

A user switched browsers, couldn't find their access token, decoded the AES-256-GCM key from their download URL and submitted it as the token. The decoded binary contained `ɸ` (U+0278), which is outside ISO-8859-1. Every subsequent upload failed with:

> Failed to execute 'fetch' on 'Window': Failed to read the 'headers' property from 'RequestInit': String contains non ISO-8859-1 code point.

The product entered a silently broken state with no recovery path. The user had to be helped manually.

**4 fixes implemented** as prescribed in the source brief — input validation, defence-in-depth header check, upload error catch, and UX hint about browser scope.

---

## Root Cause Chain

1. **User switched browsers** — `localStorage` token not available in Browser B
2. **User decoded AES key from download URL** — mistook it for the access token
3. **No format validation on token input** — `send-access-gate.js` accepted any non-empty string
4. **`checkToken` uses `encodeURIComponent`** — so the unicode passed URL validation but failed header validation
5. **`checkToken` catch block silently falls through** — token saved even on network error
6. **Corrupted token stored to localStorage** — binary string with U+0278
7. **`_authHeaders()` passed raw token to fetch** — browser rejected the header value
8. **Upload error handler showed raw browser exception** — no recovery, no explanation

---

## Fixes Implemented

### Fix 1 — Token format validation at input (PRIMARY)

**File:** `send-access-gate.js` line 121

```javascript
if (!/^[a-z0-9_-]+$/.test(token)) {
    error.textContent = this.t('access_gate.invalid_format');
    error.classList.remove('hidden');
    return;
}
```

Rejects tokens containing unicode, spaces, base64 padding (`=`), or any non-alphanumeric characters before any network call or localStorage write. This single check would have prevented the entire incident.

Also updated the catch block comment to be explicit about the format validation providing the safety guarantee.

### Fix 2 — Validate + clear + fire event in `_authHeaders()` (DEFENCE IN DEPTH)

**File:** `api-client.js` line 34

```javascript
_authHeaders() {
    const token = this.getAccessToken();
    if (!token) return {};
    if (!/^[a-z0-9_-]+$/i.test(token)) {
        this.clearAccessToken();
        document.dispatchEvent(new CustomEvent('access-token-invalid'));
        return {};
    }
    return { 'x-sgraph-access-token': token };
},
```

**Changed from the initial approach:** The earlier fix silently stripped non-ASCII characters. The brief correctly identified that this is wrong — a corrupted token should be **rejected and cleared**, not silently modified. The `access-token-invalid` event triggers the access gate to re-show the token entry form, giving the user a recovery path.

Removed the `_sanitiseToken()` method — no longer needed since we reject rather than strip.

### Fix 3 — Catch ISO-8859-1 error in upload (SAFETY NET)

**File:** `send-upload.js` line 684

```javascript
if (err.message && err.message.includes('ISO-8859-1')) {
    ApiClient.clearAccessToken();
    document.dispatchEvent(new CustomEvent('access-token-invalid'));
    return;
}
```

Even with Fix 1 and Fix 2, if a corrupted token somehow reaches `fetch()`, this catch block clears it and shows the re-entry form instead of displaying a raw browser exception.

### Fix 4 — Browser-scope hint on token entry form (UX PREVENTION)

**File:** `send-access-gate.js` (template), `i18n.js`

Added hint text below the token input:

> Your token is saved in this browser only. If you switch browsers or devices, you'll need to enter it again.

This is the change that would have prevented the root cause — the user would have understood why the token wasn't available in Browser B and wouldn't have tried to recover it from the download URL.

**New i18n keys:**

| Key | Value |
|---|---|
| `access_gate.invalid_format` | "That doesn't look like a valid token. Tokens contain only letters, numbers, hyphens and underscores." |
| `access_gate.browser_scope_hint` | "Your token is saved in this browser only. If you switch browsers or devices, you'll need to enter it again." |

## What Does NOT Need Changing

- Zero-knowledge encryption architecture is unaffected
- `checkToken` server-side behaviour is correct
- `localStorage` storage mechanism is correct (the bug was what was stored, not how)
- `encodeURIComponent` in `checkToken` is correct and stays

---

## Remaining Follow-ups

1. **Welcome page (`send-welcome.js`)** — The `TextDecoder().decode()` path also needs format validation before calling `setAccessToken()`. Same regex check should be applied there.
2. **Server-side token name validation** — When admins create tokens, validate ASCII-only to prevent this class of issue at the source.
