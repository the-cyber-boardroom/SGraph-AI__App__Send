## Key Finding: Simple Token Vault is SHIPPED

The CLI team delivered an implementation debrief for the Simple Token Vault feature. This is live in sgit-ai:

- **Commit:** `fc9ac67` | **Branch:** `claude/new-explorer-team-cHLXU` (sgit-ai repo)
- **Tests:** 1,424 passing (22 new), 0 regressions

**New commands:**
```bash
sgit init                    # auto-generates token: coral-equal-1234
sgit clone coral-equal-1234  # clone from SGit-AI server
sgit clone dawn-haven-6034   # import from SG/Send transfer
sgit share                   # publish HEAD to SG/Send, returns browser URL
sgit share --rotate          # new share token, old URL goes stale
```

**Two-token model:** Edit token (write access, must be kept secret) vs share token (read-only browser view, safe to share publicly).

**Why this matters:** The hallucinated slide deck workflow (`sgit clone vault://coral-equal-1234`) is now real. Human-readable tokens as vault identities bridge CLI usage and browser viewing. The newsletter/airdrop/collaboration workflows described in the debrief are all enabled.

**Reality document correction:** Section 17 previously listed `sgit share` as PROPOSED. It is now SHIPPED. The Simple Token architecture entry in the Platform & Product section has been corrected accordingly.

**One outstanding concern:** There is no warning at `sgit init` saying "Save your edit token. It cannot be recovered." This is a P0 UX issue — without it, users will lose vault access. Should be fixed before PyPI release.

## sg-tree: Architecture Spec Ready

The Architect delivered a full pre-implementation spec for `sg-tree` — a generic tree-view Web Component that will replace three independent implementations in the codebase.

**The problem:** Three trees (send-browse, vault-tree-view, vault-panel) with zero shared code. Same bugs fixed in one, not propagated to others.

**The spec defines:**
- Canonical data format: `{id, label, type, children, icon, meta, badge, disabled}`
- 12 public methods, 17 CSS custom properties, full event architecture
- Adapter pattern: each consumer writes ~20 lines; sg-tree never changes

**Status:** PROPOSED — no code yet. Architecture review approved the spec with minor recommendations (resolve `getSelectedId()` multi-select return type before starting; runtime duplicate-id validation).

**Estimated effort:** v0.1.0 (read-only) in 3–4 days.

## Video Editor: Commissioned but Needs Scoping

The project lead commissioned an expanded video editing suite. The motivating use case is compelling (MRI knee video: crop frame, mask patient info with keyframes, add captions, export). The proposed architecture (JSON transformation model, non-destructive editing, LLM via screenshots) is sound.

**Key architectural insight from the brief:** Every edit is a JSON transformation stored separately from the source file. The LLM never sees the video — it writes JSON transformations that the editor applies. This maintains zero-knowledge for sensitive content.

**Current status in reality doc:** PROPOSED — the brief says "a first version is working" but this is not verified in the reality document. Dev team should inspect tools.sgraph.ai before starting.

**Recommended scope for v0.1.0:** JSON transformation model + `sg-local-storage` component + extend existing crop/trim. Leave timelines, captions, masking, and LLM for v0.2.0–v0.3.0.
