Implemented the 4 server endpoints requested in the CLI team's [large blob support spec](../../../briefs/03/28/from-team__cli/spec--large-blob-support.md). The CLI team can now upload and download vault blobs of any size via S3 presigned URLs, bypassing Lambda payload limits.

## New Endpoints

All endpoints are live on the user Lambda. In S3 storage mode, they produce real presigned URLs. In memory mode (local dev / tests), they return `{ "error": "presigned_not_available" }` — clients should fall back to the normal `/api/vault/write` and `/api/vault/read` paths.

### 1. Initiate Multipart Upload

```
POST /api/vault/presigned/initiate/{vault_id}
```

**Headers:**
```
x-sgraph-access-token:    {access_token}
x-sgraph-vault-write-key: {write_key_hex}
Content-Type:             application/json
```

**Request body:**
```json
{
  "file_id":         "bare/data/{blob_id}",
  "file_size_bytes": 52000000,
  "num_parts":       5
}
```

- `file_id` — same path format used with `/api/vault/write/{vault_id}/{file_id}`
- `num_parts` — set to `0` or omit for auto-calculation (10 MB per part)
- `file_size_bytes` — total size of the encrypted blob

**Response (200):**
```json
{
  "upload_id": "s3-multipart-upload-id",
  "part_urls": [
    { "part_number": 1, "upload_url": "https://s3.amazonaws.com/...?partNumber=1&..." },
    { "part_number": 2, "upload_url": "https://s3.amazonaws.com/...?partNumber=2&..." }
  ],
  "part_size": 10485760
}
```

**Error responses:**
- `400` — presigned not available (memory mode), too many parts
- `403` — write key mismatch

### 2. Upload Parts (Direct to S3 — No Server Endpoint)

For each part, the client does a raw HTTP PUT directly to the presigned URL:

```
PUT {upload_url from part_urls}
Body: raw bytes (chunk of encrypted blob)
```

**Response from S3:**
```
HTTP 200
ETag: "abc123..."
```

Collect the `ETag` header from each part response — needed for the complete call.

**Important:** Presigned URLs expire after **1 hour**. Parts can be uploaded in parallel.

### 3. Complete Multipart Upload

```
POST /api/vault/presigned/complete/{vault_id}
```

**Headers:** same as initiate (access token + write key)

**Request body:**
```json
{
  "file_id":   "bare/data/{blob_id}",
  "upload_id": "s3-multipart-upload-id",
  "parts": [
    { "part_number": 1, "etag": "\"abc123...\"" },
    { "part_number": 2, "etag": "\"def456...\"" }
  ]
}
```

**Response (200):**
```json
{
  "status": "ok",
  "etag":   "\"combined-etag\""
}
```

After this, the blob is readable via the normal `/api/vault/read/{vault_id}/{file_id}` path (for small reads) or via presigned read-url (for large reads).

### 4. Cancel Multipart Upload

```
POST /api/vault/presigned/cancel/{vault_id}
```

**Headers:** same as initiate (access token + write key)

**Request body:**
```json
{
  "upload_id": "s3-multipart-upload-id",
  "file_id":   "bare/data/{blob_id}"
}
```

**Response (200):**
```json
{
  "status": "cancelled"
}
```

Call this on upload failure to clean up orphaned S3 parts. Best-effort — don't fail the overall operation if this errors.

### 5. Presigned Download URL

```
GET /api/vault/presigned/read-url/{vault_id}/{file_id}
```

**No auth required** — same threat model as `/api/vault/read/` (data is encrypted, key never reaches server).

**Response (200):**
```json
{
  "url":        "https://s3.amazonaws.com/...?X-Amz-Expires=3600&...",
  "expires_in": 3600
}
```

The client then does a raw HTTP GET on the URL to fetch the encrypted blob directly from S3.

## Client Integration Guide

### Push Flow (Large Blob Upload)

```python
LARGE_BLOB_THRESHOLD = 4 * 1024 * 1024   # 4 MB (encrypted size)

for blob_id, encrypted_blob in blobs_to_push:
    if len(encrypted_blob) <= LARGE_BLOB_THRESHOLD:
        # Existing path — include in batch or individual write
        vault_api.write(vault_id, f"bare/data/{blob_id}", encrypted_blob)
    else:
        # Large blob — presigned upload
        num_parts = max(1, -(-len(encrypted_blob) // (10 * 1024 * 1024)))

        # 1. Initiate
        result = vault_api.presigned_initiate(vault_id,
            file_id=f"bare/data/{blob_id}",
            file_size_bytes=len(encrypted_blob),
            num_parts=num_parts)

        upload_id = result["upload_id"]
        part_size = result["part_size"]

        # 2. Upload parts (can parallelise)
        etags = []
        for part in result["part_urls"]:
            offset = (part["part_number"] - 1) * part_size
            chunk  = encrypted_blob[offset : offset + part_size]
            # Raw HTTP PUT — no wrapper needed
            req = urllib.request.Request(part["upload_url"], data=chunk, method="PUT")
            with urllib.request.urlopen(req) as resp:
                etags.append({"part_number": part["part_number"],
                              "etag": resp.headers["ETag"]})

        # 3. Complete
        vault_api.presigned_complete(vault_id,
            file_id=f"bare/data/{blob_id}",
            upload_id=upload_id,
            parts=etags)

        # Set large=True in tree entry (client-side schema)
        tree_entry["large"] = True
```

### Pull Flow (Large Blob Download)

```python
for entry in tree_entries:
    file_id = f"bare/data/{entry['blob_id']}"

    if entry.get("large"):
        # Presigned download
        result     = vault_api.presigned_read_url(vault_id, file_id)
        ciphertext = urllib.request.urlopen(result["url"]).read()
    else:
        # Normal path
        ciphertext = vault_api.read(vault_id, file_id)

    plaintext = aes_gcm_decrypt(ciphertext, key)
```

### Capability Detection

Use the existing `/api/presigned/capabilities` endpoint to check if presigned URLs are available:

```json
GET /api/presigned/capabilities
→ { "presigned_upload": true, "multipart_upload": true, "presigned_download": true, ... }
```

If `presigned_upload` is `false` (memory mode), fall back to normal vault write/read for all sizes.

### Error Handling

| Scenario | What happens | Client action |
|---|---|---|
| Memory mode server | `initiate` returns 400 | Fall back to normal write (accept size limit) |
| Wrong write key | `initiate`/`complete`/`cancel` return 403 | Fail — auth error |
| Presigned URL expired | S3 PUT returns 403 | Re-initiate (get fresh URLs) |
| Partial upload failure | Some parts uploaded, some failed | Call `cancel` to clean up, then retry from `initiate` |
| Upload completed but `complete` call fails | Parts exist in S3 but not assembled | Retry `complete` with same `upload_id` + ETags |

### S3 Key Alignment

The presigned upload writes to the **exact same S3 key** that `Service__Vault__Pointer` uses:

```
{s3_prefix}/transfers/vault/{vault_id}/{file_id}/payload
```

This means after a presigned upload completes, the blob is immediately readable via the normal `/api/vault/read/{vault_id}/{file_id}` endpoint — no migration, no special handling.

### Auth Model

| Endpoint | Access token | Write key | Rationale |
|---|---|---|---|
| `initiate` | Required | Required | Writes to vault |
| `complete` | Required | Required | Writes to vault |
| `cancel` | Required | Required | Modifies vault state |
| `read-url` | Not required | Not required | Same as `/api/vault/read/` — data is encrypted |
