## Key Findings

### 1. No Backend Changes Needed (Phase 1)

All 10 vault API endpoints the workspace needs already exist. The workspace is purely a new frontend that calls existing APIs. The only backend change: add `workspace.sgraph.ai` to the admin Lambda's CORS allowlist.

### 2. Reuse Inventory

| Category | Count | Lines of Code |
|----------|-------|--------------|
| **Direct reuse** (copy as-is) | 11 files | ~2,500 lines |
| **Adapt** (copy + modify) | 4 files | ~1,000 lines of changes |
| **Build new** | 5 components | ~2,550 lines |

The EventBus, MessagesService, API Logger, Events Viewer, Messages Panel, vault crypto, vault API client, file type detection, markdown parser, and Aurora theme CSS are all production-tested and ready to drop in.

### 3. LLM Strategy: OpenRouter + Ollama First

Both roles converge: start with OpenRouter (one API key → all major models, explicitly browser-friendly) and Ollama (local, zero-cost, maximum privacy). This avoids the CORS uncertainty of calling Anthropic/OpenAI directly from the browser. Direct provider support is Phase 2 after verifying CORS behaviour.

### 4. Deployment: S3 + CloudFront (Same Pattern as sgraph.ai)

The workspace deploys as a static site to the shared S3 bucket, with its own CloudFront distribution and DNS record. The existing `deploy_static_site.py` script handles it — just a new site name and CloudFront dist ID.

### 5. Scheduling: After Vault Fixes

The Dev flags that the v0.10.32 Sprint 1 (vault fixes — folder upload bug, auth separation, drag-and-drop) should complete first. The workspace depends on a stable vault. Recommended start: next week.
