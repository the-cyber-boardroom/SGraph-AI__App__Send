## Key Findings (Cross-Role Convergence)

### Finding 1: The PBKDF2 Iteration Count Bug

**All three technical roles** (Dev, Architect, AppSec) independently found the same bug: the vault CLI brief's Python code sample uses `iterations=100000` but the JavaScript browser implementation uses `600000`. This 6x discrepancy would cause **silent cross-platform decryption failure** — files encrypted by one implementation would be unreadable by the other.

**Action:** Fix the brief's code sample. Gate all CLI development on a cross-language crypto round-trip test before any feature work.

### Finding 2: Vault Auth Flow Change Needs Careful Thought

The proposal to make auth tokens only required for upload (not browsing) is architecturally sound but has a **security implication**: sharing the vault key grants permanent, irrevocable, undetectable read access. There is no server-side mechanism to revoke a reader.

**Action:** Document the trust model explicitly. Consider whether this is acceptable for the MVP use case (personal vault, single user). For shared vaults, a different approach will be needed.

### Finding 3: Gallery Sharing Is the Highest-Impact Feature

All five roles converge: the encrypted gallery is the best acquisition engine. But the Sherpa flags a **pricing risk** — per-file billing at 50-100 credits per album may feel punitive. The Architect flags a **design fork** — gallery-as-zip (good for sharing) vs. gallery-as-vault-folder (needed for collaboration). The AppSec flags **5 XSS vectors** (SVG scripts, polyglot files, subtitle injection, EXIF injection, filename injection).

**Action:** Build gallery Phase 1 as zip-in-transfer. Design gallery.json with nullable file_id references to enable vault-backed evolution later. Mitigate XSS vectors before exposing to untrusted content. Consider a "gallery pack" pricing model.

### Finding 4: Recipient Experience Is a Data Gap

The Sherpa identifies that **we have zero visibility on the recipient experience**. All 10 customer interactions were observed from the sender side. We don't know: do recipients find the download flow confusing? Does the decryption key entry make sense? Do they successfully open files?

**Action:** Add lightweight analytics to the download page (anonymised, no PII). Ask the first 10 customers to share the link with someone and report back on the recipient's experience.

### Finding 5: Reality Document Is Stale

The Dev review flags that the reality document is at v0.6.36 while we're at v0.10.32 — **93 commits behind**. There is a newer v0.10.13 version but it's still 40+ commits behind. This undermines the reality check process.

**Action:** Librarian should update the reality document to v0.10.32 this sprint, covering vault v0.1.1, zip viewer, folder upload, gallery preview, and all new UI components.

### Finding 6: File Safety Architecture — Impressive but Long-Term

The AppSec review calls the file safety orchestration "the strongest security design in the project." The Architect estimates 30-50 weeks (7-12 months) realistically. All roles agree: acknowledge and defer to Q2+. The immediate value is the architecture document itself as an investor/enterprise conversation piece.

**Action:** No development work this quarter. Use the architecture brief for investor materials and enterprise pre-sales conversations.
