# Debrief — First LLM Call from SG/Workspace

**Version:** v0.10.44
**Date:** 4 March 2026
**Session type:** Explorer

---

## Executive Summary

On 4 March 2026, the SG/Workspace made its first successful LLM call. A document was loaded from an encrypted vault, transformed by an LLM via OpenRouter, and the streamed response rendered in real-time — all in the browser, with zero server involvement in the LLM pipeline.

Both LLM providers were confirmed working on the same day:
- **OpenRouter** (cloud) — first call
- **Ollama** (local, Gemma3:4b) — confirmed shortly after, using `OLLAMA_ORIGINS="https://dev.workspace.sgraph.ai" ollama serve` for CORS

A blocking bug (custom element upgrade order) was fixed in one commit before the successful call.

## Key Decisions Made

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | All LLM calls go browser → provider (no server proxy) | Maintains zero-knowledge architecture |
| 2 | Ollama CORS via `OLLAMA_ORIGINS` env var | User controls access; avoids server proxy |
| 3 | Two stream readers: SSE (OpenRouter) + NDJSON (Ollama) | Native format per provider, clean abstraction |
| 4 | Five-stage pipeline: vault → source → LLM → transform → vault | Every stage client-side, encryption bookends the flow |
| 5 | Guard cross-component calls with `typeof` checks | Web Components don't guarantee sibling upgrade order |

## Bug Fixed

**`conn.getModels is not a function`** — When `workspace-shell` sets innerHTML, `<llm-chat>` fires `connectedCallback` before `<llm-connection>` is upgraded from a bare HTMLElement. Fixed by checking `typeof conn.getModels === 'function'` before calling methods on sibling custom elements.
