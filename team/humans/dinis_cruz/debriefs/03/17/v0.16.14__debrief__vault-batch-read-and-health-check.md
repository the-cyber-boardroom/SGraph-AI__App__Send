## Executive Summary

Two high-priority backend features were implemented to support the next version of the Vault CLI and client: **batch read** (N files in one round trip) and a **vault health check** (existence check + Lambda warm-up). Both are unauthenticated, consistent with the zero-knowledge model — the server only stores encrypted ciphertext.

---

## What Was Delivered

### 1. Batch Read — `POST /api/vault/batch/{vault_id}`

Extended the existing batch endpoint to support `"op": "read"` alongside the existing `write`, `write-if-match`, and `delete` operations.

**Request:**
```json
{
  "operations": [
    { "op": "read", "file_id": "bare/refs/ref-pid-muw-b1d1771f9fb5" },
    { "op": "read", "file_id": "bare/data/obj-cas-imm-4602b76dd1ad" },
    { "op": "read", "file_id": "bare/data/obj-cas-imm-867f9d7b56e1" }
  ]
}
```

**Response:**
```json
{
  "vault_id": "12cpxeq9",
  "results": [
    { "file_id": "bare/refs/ref-pid-muw-b1d1771f9fb5", "status": "ok", "data": "<base64>" },
    { "file_id": "bare/data/obj-cas-imm-4602b76dd1ad", "status": "ok", "data": "<base64>" },
    { "file_id": "bare/data/obj-cas-imm-867f9d7b56e1", "status": "not_found" }
  ]
}
```

**Key design decisions:**
- **Read-only batches skip auth entirely** — if all ops are `read`, no write-key or access token required. Consistent with individual `GET /api/vault/read/` which is also unauthenticated
- **Mixed batches still require full auth** — if any op is `write`/`write-if-match`/`delete`, write-key + access token enforced as before
- **Read ops also work inside authenticated mixed batches** — a push can read + write in one call
- **Non-existent files return `not_found` status** (not an error) — the batch continues

**Performance impact:** Opening a vault with 2 files previously required ~6 HTTP requests (ref + commit + root tree + sub-tree + 2 blobs). Now it's **1 request**.

### 2. Health Check — `GET /api/vault/health/{vault_id}`

New unauthenticated endpoint that checks whether a vault manifest exists.

**Response (exists):** `200 {"status": "ok", "vault_id": "12cpxeq9"}`
**Response (not found):** `404 {"detail": "Vault not found"}`

**Use cases:**
- **Lambda cold-start warm-up** — CLI calls this before push/pull/clone to prime the Lambda
- **Vault existence check** — fast validation before starting a multi-step operation

## What's Next — Lower Priority Items from v0.16.3 Spec

These items from the Architect's spec were reviewed and deferred. They are hardening/optimisation, not blocking for the next vault version:

| Item | Priority | Notes |
|------|----------|-------|
| **Server-side caching by mutability** | Medium | Parse `imm`/`snw`/`muw` from self-describing file IDs. Cache `imm` objects forever, `snw` for 5s, always read `muw` fresh. Useful optimisation once vaults carry many immutable objects |
| **File ID validation regex** | Low | `^[a-zA-Z0-9/_-]{1,128}$` — defence in depth against path traversal. Not strictly necessary since Storage_FS already sandboxes paths |
| **Compare-and-swap enforcement by mutability** | Low | Reject `write-if-match` on `*-imm-*` files, require it on `*-muw-*` files. Clients already follow these rules; server enforcement is belt-and-braces |
| **Auth on list endpoint** | Low | With deterministic addressing, `list` is only needed for admin/debug. Could require auth token. Would be a behaviour change for existing clients |

---

## No Breaking Changes

- The `{file_id:path}` route capture handles both old (`bare/data/obj-xxx`) and new self-describing (`bare/data/obj-cas-imm-xxx`) formats
- Existing batch callers (write/write-if-match/delete) are unaffected — auth is only skipped when **all** ops are reads
- The health check is a new route with no overlap
