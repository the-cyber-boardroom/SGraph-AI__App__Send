## Executive Summary

The standalone vault UI at `static.sgraph.ai` calls the `send.sgraph.ai` API for all transfer operations. That API requires an `x-sgraph-access-token` header, but the vault entry screen had no way for users to provide one — resulting in a 401 "Access token required" error on every create/open attempt.

This session added an **Access Key input field** to the vault entry screen. The user pastes their send.sgraph.ai access token before opening or creating a vault. The token is persisted in `sessionStorage` (survives navigation, cleared when the tab closes) and included in every API request. The fix was tested live — vault creation, file upload, and folder browsing all work with the token in place.

## Key Design Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| D-1 | **Input field on entry screen (not settings/config)** | The token is required for every API call — it must be visible and obvious, not hidden in settings |
| D-2 | **sessionStorage, not localStorage** | Access tokens should not persist across browser sessions. sessionStorage clears when the tab closes — appropriate security posture for bearer tokens |
| D-3 | **Falls back to `data-token` attribute** | Preserves programmatic embedding — if a parent page sets `data-token` on the component, it still works without manual input |
| D-4 | **Validation before API calls** | Shows "Please enter your access key first" and focuses the field if empty — prevents confusing 401 errors |

## Technical Flow

```
User pastes access key
    ↓
sessionStorage.setItem('sg-vault-access-key', key)
    ↓
User clicks Open/Create Vault
    ↓
_requireAccessKey() validates token is present
    ↓
_getSGSend() reads token from input (or data-token fallback)
    ↓
new SGSend({ endpoint, token })
    ↓
All fetch() calls include: x-sgraph-access-token header
    ↓
send.sgraph.ai accepts the request ✓
```
