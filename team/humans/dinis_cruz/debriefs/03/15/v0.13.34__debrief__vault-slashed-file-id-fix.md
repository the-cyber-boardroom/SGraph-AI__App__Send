# Debrief: Vault Endpoint Fix — Slashed file_id Support on Individual Routes

**Version:** v0.13.34
**Date:** 2026-03-15
**Team:** Explorer

---

## Problem

Individual PUT/GET/DELETE vault endpoints (`/api/vault/write/{vault_id}/{file_id}`, `/read/...`, `/read-base64/...`, `/delete/...`) returned 404 for file IDs containing slashes (e.g. `bare/data/obj-abc123`, `bare/refs/heads/main`). Starlette's default `{file_id}` path parameter only captures a single segment — slashes split into extra path segments causing route mismatch.

A second issue: even if routing had worked, `Safe_Str__Id` silently converts `/` to `_`, so `bare/refs/main` would become `bare_refs_main` at the service layer.

## What Changed

| File | Change |
|------|--------|
| [`Routes__Vault__Pointer.py`](../../../../../sgraph_ai_app_send/lambda__user/fast_api/routes/Routes__Vault__Pointer.py) | Added `@route_path` decorator with `{file_id:path}` on write, read, read-base64, delete |
| [`Routes__Vault__Pointer.py`](../../../../../sgraph_ai_app_send/lambda__user/fast_api/routes/Routes__Vault__Pointer.py) | Changed `file_id` parameter type from `Safe_Str__Id` to `str` |
| [`test_Routes__Vault__Pointer.py`](../../../../../tests/unit/lambda__user/fast_api/routes/test_Routes__Vault__Pointer.py) | Added 2 tests: slashed file_id lifecycle + deeply nested (`bare/refs/heads/main`) |

## What Did NOT Change

Batch and list endpoints — they already support slashes (batch uses JSON body, list uses query param prefix).

## Test Results

- 33 route tests pass (31 existing + 2 new)
- 203 full user lambda tests pass

## For the CLI Client

Both URL-encoded (`bare%2Fdata%2Fobj-abc`) and plain slash (`bare/data/obj-abc`) paths now work on all individual endpoints. No client-side changes needed — the server handles it.
