### The Six Major Themes (03/13–03/18)

**Theme 1: Vault Architecture Overhaul (03/17 — the big one)**
The `vault-redesign/` subdirectory under `03/17` contains 11 documents that represent the most significant architectural change since the branch model spec. Key innovations:
- **Self-describing file IDs:** `{type}-{derivation}-{mutability}-{hex_id}` — eliminates the dual-write hack, enables storage-agnostic backends
- **Sub-tree model:** One tree per directory instead of flat tree. Vault open drops from O(files) to 3 requests (ref + commit + root tree)
- **Four-layer security model:** Modes A (full zero-knowledge, current), B (structural visibility), C (signed plaintext), D (pure VCS). The codebase already supports all four — just different client decisions about what to encrypt
- **11-phase CLI implementation plan:** ~300 LOC added, ~370 removed. Phase 5 (sub-tree building) is the critical path. A detailed 841-line implementation guide exists at `v0.16.3__impl-guide__cli-detailed.md`
- **Three-client convergence:** CLI uses PBKDF2 passphrase, browser uses different crypto, user vault UI uses RSA-4096. All three must converge on one model

**Theme 2: PKI & Vault Positioning (03/14)**
Five vault positioning documents (`vault/01-05`) in the 03/14 briefs define SG Vault as "Git with integrated trust". Four PKI operating modes from anonymous (Mode 1, current) through device-signed (Mode 2, next), author-identified (Mode 3), to countersigned (Mode 4, regulated). Phasing: Mode 2 now, Mode 3 when needed, Mode 4 for enterprise/regulated.

**Theme 3: Upload UX Redesign (03/16)**
Messaging pivot from "encrypted transfer" to "share large files and folders". Three-step upload flow (upload → distribution → credentials), three sharing modes (simple token, full link, separate key), 10GB limit. The team response document (`v0.16.10__upload-ux-redesign-team-response.md`) has ASCII mockups, state machines, component inventories, and i18n requirements (26 new strings, 17 locales).

**Theme 4: Revenue & Use Cases (03/13 + 03/16–03/17)**

**Theme 5: Agent Communication & Workflows (03/17)**

## Librarian Operating Guidance — Lessons From This Session

These are the patterns, pitfalls, and techniques I developed while operating as the Explorer Librarian. The next Librarian should internalise these before starting work.

### How Brief Processing Actually Works

**Step 1: Determine scope.** Find the last brief-processing debrief (search `debriefs/` for files with `briefs-processing` or `daily-brief` in the name). Everything after that date is unprocessed. Don't confuse implementation debriefs (which are about code shipped) with brief-processing debriefs (which catalogue and assess briefs).

**Step 2: Read daily briefs first.** Each date has a `daily-brief` file that summarises the day's output. Read these first to understand themes before diving into individual briefs. The daily briefs are at `team/humans/dinis_cruz/briefs/03/DD/v0.X.Y__daily-brief__sgraph-send-DD-mar-2026.md`.

**Step 3: Watch for subdirectories.** Briefs now have subdirectories (e.g. `03/14/vault/`, `03/17/vault-redesign/`, `03/10/cli-team/`, `03/10/tools-team/`). These contain the most architecturally significant documents and are easy to miss if you only `ls` the date folder. Always check for subdirectories.

**Step 4: Use parallel agents for reading.** The briefs are numerous and long. Spawn Explore agents to read batches of documents in parallel. One agent per date, or one per subdirectory. This is much faster than reading them sequentially. In this session I used an Explore agent for the 03/17 vault-redesign subdirectory (11 documents, 841 lines for the longest one) which returned comprehensive summaries.

**Step 5: Produce role reviews, not summaries.** The output is role-specific reviews (Architect, Dev, Designer, AppSec, etc.), not generic summaries. Each review should assess the briefs through that role's lens — the Architect evaluates architectural soundness, the Dev estimates implementation effort, the Designer assesses UX decisions. The master index is the generic catalogue.

**Step 6: Update reality document with PROPOSED items.** Every brief that mentions a new feature, endpoint, component, or capability that doesn't exist in code must be added to the reality document as PROPOSED. Organise by theme, not by date. The reality document is the single source of truth — if it's not listed there, it doesn't exist.

### Pitfalls I Hit

**1. Relative links in debriefs are tricky.** Debriefs live 5 directories deep (`team/humans/dinis_cruz/debriefs/MM/DD/`). Links to role reviews need `../../../../../roles/`. Links to villager/town-planner roles also need `../../../../../villager/` or `../../../../../town-planner/` — NOT `../../../../villager/` (that resolves to `team/humans/dinis_cruz/villager/` which doesn't exist). **Always verify links with `ls` before committing.**

**2. Reality document versioning.** The reality document filename includes the version (`v0.16.26__what-exists-today.md`). When updating it, both update the existing file AND create a copy with the new version number. The CLAUDE.md references a specific version — updating CLAUDE.md to point to the latest is a separate task (and currently stale).

**3. Brief-processing vs. code audit.** These are different activities. Brief processing adds PROPOSED items and produces role reviews. Code auditing runs tests, inspects endpoints, and marks things as EXISTS. Don't conflate them. This session did brief processing only — the EXISTS sections of the reality document are still from the v0.13.34 code audit and are likely incomplete.

**4. Briefs are aspirations, not facts.** A brief describing "per-supplier encryption" does NOT mean per-supplier encryption exists. The Librarian's job is to catalogue what's proposed while maintaining the hard line between PROPOSED and EXISTS. When in doubt, mark it PROPOSED.

**5. Volume management.** 51 briefs in one batch is a lot. The Architect, Dev, and Designer reviews were produced but AppSec, DevOps, Journalist, and Conductor reviews were not. Future sessions should either process briefs more frequently (daily or every 2-3 days) or accept that a single session will produce 3-4 role reviews and flag the rest as gaps.

### Document Structure Patterns

**Master index format:**
- Catalogue by date (table with #, document name, type, key theme)
- Then thematic grouping (cross-cutting themes that span multiple dates)
- Then list of role reviews produced
- This format has been consistent since v0.3.0 and works well

**Role review format:**
- Executive summary (2-3 sentences)
- Sections organised by the role's concerns (Architect: decisions, concerns, open questions; Dev: implementation streams by priority, risks; Designer: assessment of design decisions, concerns, priority deliverables)
- Each section references specific brief documents by name/version
- Verdicts on key decisions (Sound/Concern/Reject)

**Reality document format:**
- Changes since previous version (table)
- Numbered sections (1: Backend endpoints, 2: Frontend UI, etc.)
- Each section has EXISTS (code-verified) and DOES NOT EXIST (proposed) subsections
- PROPOSED items include source brief and date

**Debrief format:**
- Executive summary
- Recommended reading order (numbered, with rationale)
- Deliverables table (with relative links that WORK)
- Theme summaries
- Key decisions needing review
- What's next

### The Brief Taxonomy

Briefs follow naming patterns that signal their type:

| Pattern | Meaning | How to Process |
|---------|---------|---------------|
| `daily-brief` | Day overview, signals, priorities | Read first, use as map |
| `dev-brief` | Implementation spec for a feature | Dev review, add PROPOSED to reality |
| `arch-brief` | Architecture decision/proposal | Architect review, assess soundness |
| `brief` | Strategy/business/ecosystem | Conductor or Ambassador review |
| `article` | Public-facing content for LinkedIn etc | Journalist review |
| `impl-plan` | Detailed implementation guide | Dev review, verify against reality |
| `arch-spec` | Formal architecture specification | Architect review, normative reference |
| `arch-simulation` | Simulated execution trace | Architect review, validate assumptions |
| `arch-addendum` | Extension to existing spec | Architect review, check consistency |
| `arch-note` | Concept/exploration (not for immediate build) | Note in review, don't add as P0 |
| `role-definition` | New role definition | Create ROLE.md files |
| `briefing` | Positioning/strategy document | Multiple roles |
| `guide` | Implementation/usage guide | Dev review |
