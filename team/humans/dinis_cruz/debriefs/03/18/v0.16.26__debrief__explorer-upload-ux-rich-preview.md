## Technical Details for Next Session

### How v0.2.12 Works

The IFD overlay pattern: `send-upload-v0212.js` loads AFTER v0.2.11 and overrides methods via `SendUpload.prototype` mutation. No `customElements.define()` — reuses v0.2.0's registration.

**Core flow:**
1. Intercepts `_buildZipBlob()` (the method that creates the encrypted zip)
2. For each file, calls `generateImageThumbnail()` using Canvas API (200px wide JPEG at 75% quality)
3. Builds a `_preview/` folder inside the zip containing:
   - `_preview/_manifest.json` — gallery config, file index, type breakdown
   - `_preview/thumbnails/file-001.jpg` — one thumbnail per image file
   - `_preview/metadata/file-001.json` — per-file metadata (type, size, dimensions, category)
4. Also generates sender-side gallery preview cards shown in Step 2 (Delivery)

**Key design decisions:**
- SVG files are passed through as-is (already tiny vectors)
- Non-image files get metadata JSON but no thumbnail (Phase 2 will add PDF/MD thumbnails)
- `_v0212_suppressPreview` flag prevents v0.2.11's inline preview from rendering (v0.2.12 replaces it with its own richer preview)
- All processing is client-side — server never sees plaintext (zero-knowledge preserved)

### Gallery Detection Logic (in v0.2.3)

The `detectDeliveryOptions()` function in `send-upload-v023.js` was changed:
- **Before:** Gallery option only shown when ALL files are images (`.every()`)
- **After:** Gallery option shown when ANY file is an image (`.some()`)
- This means a folder with 2 PDFs + 1 image now shows all three delivery options: browse, gallery, zip

1. **Phase 2: PDF thumbnails** — Add pdf.js (vendored, zero CDN) to render page 1 as a canvas thumbnail during zip creation. The `_preview/` folder structure is already built to accommodate this.

4. **Performance testing** — Test with larger folders (20+ files, 50+ MB) to measure thumbnail generation time vs. upload time.
