## Deliverables

### 1. `team/comms/` Structure (New Operating Model)

| Folder | Purpose |
|---|---|
| `changelog/MM/DD/` | What changed — version-tagged, with good/bad failure classification |
| `qa/briefs/MM/DD/` | Briefs TO the QA team (what to test, priorities) |
| `qa/questions/MM/DD/` | Bidirectional Q&A between Explorer and QA |
| `plans/MM/DD/` | Pre-implementation intent documents |

**Why this matters:** The QA agentic team clones this repo to run tests. When tests fail, they need to know instantly whether it's an expected change (update the test) or a regression (file a bug). The changelog provides this classification.

**Context:** The QA site at [qa.send.sgraph.ai](https://qa.send.sgraph.ai) already has CI-driven screenshots with diff detection. The comms structure complements this by providing the *intent* behind changes.

## Key Decisions

1. **`data-ready` on body (not custom event)** — Simplest possible approach. One line per page. QA uses `page.wait_for_selector('body[data-ready]')`. No complex event coordination needed.

2. **`data-testid` (not `data-test` or `data-cy`)** — Standard convention used by React Testing Library, Playwright docs, and most test frameworks. Established as the project-wide pattern going forward.

3. **Additive, non-breaking changes** — All `data-testid` attributes are additive. Existing IDs preserved. No visual or behavioural changes. Zero tests should break from these additions.

4. **`team/comms/` at top level (not under `team/roles/qa/`)** — The changelog is useful to ALL downstream teams (Villager, QA, etc.), not just QA. The `qa/` subfolder is QA-specific, but the top-level is team-wide.
