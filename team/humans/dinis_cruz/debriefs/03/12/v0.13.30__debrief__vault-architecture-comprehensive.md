## Executive Summary

The vault has evolved from a personal encrypted data store (v0.5.15, 22 Feb) into the central architectural primitive of the entire SG/Send platform. In 18 days and 65 documents, the vault has acquired: content-addressed storage, tree structures, commits, CLI access (clone/push/pull/init), AI-native workflows, and a clear path to becoming "lightweight encrypted Git." The SPKI rebrand (v0.13.30) confirms this: vaults are the platform, file transfer is one feature.

This debrief synthesises every vault concept across all briefs into a single document for cross-session reference.

## 1. What Exists Today (Code-Verified)

Source: [Reality document v0.13.22](../../../../../roles/librarian/reality/v0.13.22__what-exists-today.md)

### Vault Backend (Python/FastAPI)
- **Vault Pointer system** — `Routes__Vault__Pointer.py` registered on both User and Admin Lambdas
- **Endpoints:** create vault, upload blob, download blob, get tree, update tree, read-base64 (code-verified with 3 tests)
- **Storage:** Memory-FS (`Storage_FS`) — blobs stored as encrypted ciphertext, server never sees plaintext
- **Commits:** Vault supports commit operations (append-only, no merge)
- **Content-addressed IDs:** Every blob has a hash-based ID

- **5 working commands:** clone, pull, push, status, init
- **Vault lifecycle complete:** create vault from CLI, clone to local, modify, push back
- **First AI-native vault workflow verified:** Claude cloned a vault, created a file, pushed it (M-011)

## 2. The Vault Evolution Timeline

### Phase 1: Personal Data Vault (v0.5.15, 22 Feb)
**Brief:** [PKI-Keyed Personal Data Vault](../../../briefs/02/22/v0.5.15__dev-brief__pki-keyed-personal-data-vault.md)

The original concept: a personal encrypted data store keyed by PKI. Each user has a vault. The vault holds personal data (keys, preferences, settings). Access controlled by public/private key pair.

**Key decisions:**
- Vault built on existing SG/Send encrypted blob storage (no new infrastructure)
- PKI key pair as identity and access control
- Cache Service Data API as the storage backend

### Phase 2: JavaScript Implementation + Admin Integration (v0.10.14–v0.10.28, 2–3 Mar)
**Briefs:**
- [Data Vault — Pure JavaScript](../../../briefs/03/02/v0.10.14__dev-brief__data-vault-pure-javascript.md)

The vault moved from concept to working code. Pure JavaScript implementation using Web Crypto API. Integrated into the admin shell as a first-class component.

**Key decisions:**
- Zero framework dependencies (vanilla JS + Web Components)
- Deterministic vault pointers with read/write key separation
- Vault UI patterns established (tree view, file viewer, key input)

### Phase 3: CLI + Local Sync (v0.10.30, 3 Mar)

**Key decisions:**

**Briefs:**
- [Session Findings: AI-Native Vault Access via MCP](../../../briefs/03/10/cli-team/v0.13.14__session-findings__ai-native-vault-access.md)

### Phase 7: Vault as Infrastructure (v0.13.29, 11 Mar)
**Briefs:**
- [Encrypt for the Reader, Not for Storage](../../../briefs/03/11/v0.13.29__arch-brief__encrypt-for-reader.md)
- [Lambda Vault Logging and Headless CLI Mode](../../../briefs/03/11/v0.13.29__dev-brief__lambda-vault-logging.md)
- [PKI vs URIs: Document Identity](../../../briefs/03/11/v0.13.29__arch-brief__pki-document-identity.md)
- [Alchemist Investor Pack Round-Trip](../../../briefs/03/11/v0.13.29__debrief__alchemist-investor-pack.md)
- [Vault UI Components and Pack Architecture](../../../briefs/03/11/v0.13.29__dev-brief__vault-ui-components.md)

The vault shifts from "encrypted storage" to "infrastructure primitive." Lambda functions write logs to vaults. Documents have cryptographic identity. AI agents create structured vault packs. The UI component model (`<sg-vault>`) separates data from presentation.

**Key concepts introduced:**
- **Encrypt for reader:** Data encrypted with the intended reader's public key, not for the storage system. The storage system holds ciphertext it cannot decrypt. Applied to Lambda logs, audit trails, multi-level access.
- **Lambda vault logging:** Two modes — daily vault (commit per invocation, push) or execution bundle (one vault per invocation, S3 PUT). Lambda identity via key pair. Commit-only pattern (no merge needed).
- **Headless CLI mode:** Programmatic vault access without interactive prompts. Required for Lambda, containers, background workers.
- **Document identity via PKI:** Documents as first-class identity holders. Signed content, verifiable provenance, hash-based references ("clues") instead of URIs.
- **Pack architecture:** Vault + manifest (`_pack.json`) + UI configuration = self-describing, branded presentation. Pack types: investor, ERM, customer, compliance.
- **`<sg-vault>` Web Component:** Data layer that loads, decrypts, syncs. Emits events. Display components (`<sg-tree-view>`, `<sg-file-viewer>`, `<sg-pack-theme>`) consume events without vault coupling.

### Phase 8: Vault as Platform (v0.13.30, 12 Mar)
**Briefs:**
- [Bare Vault Structure Unification](../../../briefs/03/12/v0.13.30__arch-brief__bare-vault-unification.md)
- [Branch Model: PKI Per Branch and Multi-User](../../../briefs/03/12/v0.13.30__arch-brief__branch-model-multi-user.md)
- [CLI as Standalone Project with Remotes](../../../briefs/03/12/v0.13.30__arch-brief__cli-standalone-remotes.md)
- [Vault Access from Claude: Downloadable Link](../../../briefs/03/12/v0.13.30__dev-brief__vault-access-from-claude.md)
- [Vault Website, Billable Units, Demo Vaults](../../../briefs/03/12/v0.13.30__dev-brief__vault-website-billing.md)
- [SPKI Brand Strategy](../../../briefs/03/12/v0.13.30__brief__spki-brand-strategy.md)
- [Portugal Strategy: Gallery + Registry](../../../briefs/03/12/v0.13.30__brief__portugal-strategy-gallery-registry.md)

The vault becomes the platform itself. Everything listed below is PROPOSED — does not exist yet.

**Key concepts introduced:**

- **Bare vault unification:** Server-side and clone-side structures must be identical. Clone = zip download. Push = upload changed files. One format, one security review. Vaults inside vaults (nested access control). SQLite as future local backend for large vaults.

- **Branch model with PKI per branch:** Every clone creates a branch. Every branch has its own key pair. Every commit is signed by the branch key. Push always succeeds (branch is unique). Conflicts only at merge boundaries, never within a branch. Two signing modes: self-signed and authority-signed. Auto-create on clone (invisible to basic users). This is the path to "lightweight encrypted Git."

- **Git-like feature gap to close:**
  - Content-addressed storage — EXISTS
  - Tree structure — EXISTS
  - Commits — EXISTS
  - Branches — PROPOSED
  - Merge — PROPOSED
  - Signed commits — PROPOSED
  - Remotes — PROPOSED
  - Diff — PROPOSED
  - Log — PROPOSED

- **CLI as standalone tool with remotes:** The CLI inverts from "SG/Send client" to "vault tool that has SG/Send as one remote." Supports: local folder, zip file, another vault, S3 bucket, any URL. `sg-send-cli remote add/remove/list`. Full offline mode (create, add, commit, branch, merge, encrypt, decrypt, sign, verify). Vault export as distributable zip for email/WhatsApp/AirDrop.

- **Vault download endpoint for Claude:** `/api/vault/download/{vault_id}` returns full vault as single bundle. Claude uses `web_fetch` to download (bypasses egress proxy). Read-only access without network allowlist. Write-back still requires egress.

- **Vault website + billing:** Separate site (vault.spki.ai). Primary actions: "Open Vault" and "Create Vault." Demo vaults on website. Billable units: vault count, pulls, pushes, commits, branches, storage. Prepaid credits, no surprises.

- **Gallery:** `gallery.sgraph.ai` — vaults displayed as browsable collections. Public (key in URL), private (key required), key-gated (public key encryption). Gallery is a presentation layer over vault API with `_gallery.json` manifest.

- **Public key registry:** Directory of public keys (starting with Portugal). Scrape GitHub, government, PGP keyservers. SG/Send signs each record. "Send secure message" button per entry.

- **SPKI rebrand:** `spki.ai` acquired. Vault becomes core product surface (`vault.spki.ai`). "Send" becomes one feature. Transition is organic, no timeline. Language-neutral acronym for international markets.

## 4. Architecture Principles (Crystallised Across Briefs)

1. **Server never sees plaintext.** Encryption happens client-side. Server stores ciphertext. This is non-negotiable and applies to all vault operations.

2. **Encrypt for the reader, not for storage.** Data is encrypted with the intended reader's public key. The storage system is untrusted by design.

3. **The vault is the universal handover format.** Between AI agents and humans. Between teams. Between services. The vault travels, the key travels separately.

4. **Content-addressed identity.** Blobs identified by hash. Documents identified by signature. Branches identified by key pair. No URIs needed.

5. **Conflicts at merge boundaries only.** Branch model ensures commits always succeed. Conflicts only arise when integrating changes from different branches.

6. **One format, everywhere.** Server-side and clone-side vault structures must be identical. Clone = download. Push = upload.

7. **Offline-first.** All vault operations (create, add, commit, branch, merge, encrypt, decrypt, sign, verify) work without network. Network only needed for sync.

8. **Self-describing.** Vaults carry their own UI configuration (pack manifest, gallery config). The presentation adapts to the content, not the other way around.

## 5. What Your Other Session Needs to Know

If you are building vault compatibility with the latest developments, here are the critical interfaces:

### Current API (Code-Verified)
- `POST /api/vault/create` — create a new vault
- `POST /api/vault/upload/{vault_id}` — upload encrypted blob
- `GET /api/vault/download/{vault_id}/{blob_id}` — download encrypted blob
- `GET /api/vault/tree/{vault_id}` — get vault tree
- `POST /api/vault/tree/{vault_id}` — update vault tree
- `GET /api/vault/read-base64/{vault_id}/{file_id}` — read blob as base64 (MCP-compatible)

### CLI Interface (Code-Verified)
```bash
sg-send-cli clone <vault_url> [--key <key>]
sg-send-cli pull
sg-send-cli push
sg-send-cli status
sg-send-cli init [--server <url>]
```

### Proposed Extensions (Not Yet Built)
```bash
# Branches (PROPOSED)
sg-send-cli branch [name]
sg-send-cli merge <branch>
sg-send-cli log [--graph]

# Remotes (PROPOSED)
sg-send-cli remote add <name> <url>
sg-send-cli remote remove <name>
sg-send-cli remote list

# PKI (PROPOSED)
sg-send-cli sign <file>
sg-send-cli verify <file>

# Distribution (PROPOSED)
sg-send-cli export --zip <output>
```

### Proposed API Extensions (Not Yet Built)
- `GET /api/vault/download/{vault_id}` — full vault as single bundle (for Claude web_fetch)
- Branch operations (create, list, merge)
- Signed commit verification

### Key Compatibility Considerations
1. **Vault format is pre-launch.** No backwards compatibility debt. Break freely, add version field.
2. **Bare vault unification pending.** Server-side and clone-side formats should converge. Build for the unified format.
3. **Branch model not yet implemented.** But the commit model exists. Build on commits, prepare for branches.
4. **Pack manifest (`_pack.json`) not yet defined.** The concept is clear but the schema needs specifying.
5. **Lambda identity not yet implemented.** Key pair generation and log signing are proposed only.
