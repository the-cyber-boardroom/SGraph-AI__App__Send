## Summary

Two commits delivering gallery feature parity with v0.2.6 and implementing Phase 2 of the QA team's screenshot determinism proposal.

## Commit 1: Gallery Parity with v0.2.6

**Files changed:** `send-gallery.js`, `send-gallery.css`, `send-browse.js`

### Features ported from v0.2.6

| Feature | What changed |
|---|---|
| **Auto view mode** | File count determines initial layout: ≤14 → large, 15-30 → grid, 31+ → compact. Was hardcoded to `grid`. |
| **"Decrypted" status** | Green checkmark + "Decrypted" badge in gallery header, matching browse view |
| **Markdown text preview** | Thumbnails show first 200 chars of markdown content (stripped of formatting). Tables are filtered out to avoid raw pipe characters in preview. |

### Markdown rendering fixes

| Issue | Root cause | Fix |
|---|---|---|
| **Tables not rendering in gallery lightbox** (pic1 in brief) | `.sg-lightbox__doc` had no table CSS — only basic color/padding styles | Added full markdown CSS: table borders, `code` highlighting, blockquotes, heading borders, hr, links, images |
| **Gallery lightbox markdown inferior to browse view** (pic2 vs pic3) | Browse view had `.sb-file__markdown` with comprehensive styles; gallery lightbox had none | Ported all styles from browse CSS to lightbox CSS |

### SgPrint integration

- Gallery print button now uses `SgPrint.printMarkdown()` when lightbox shows a markdown file — opens A4 print preview with SG/Send branding
- Added print button to lightbox footer (visible for markdown/text/code files)
- Fixed browse view print — was calling non-existent `SgPrint.print()`, now uses `SgPrint.printHtml()`

## Commit 2: `data-qa-mask` Attributes (QA Proposal Phase 2)

**Brief:** [v0.2.30__proposal__screenshot-determinism-data-qa-mask.md](../../../briefs/03/25/from-qa-team/v0.2.30__proposal__screenshot-determinism-data-qa-mask.md)

**Files changed:** `upload-step-done.js`, `send-download.js`, `send-viewer.js`, `send-browse.js`

### What was added

14 `data-qa-mask` attributes across 4 files, covering every non-deterministic UI element identified in the QA proposal:

| Mask category | Elements | File |
|---|---|---|
| `friendly-token` | `#simple-token` | upload-step-done.js |
| `transfer-url` | `#full-link`, `#combined-link`, `#link-only` | upload-step-done.js |
| `transfer-url` | 2 readonly URL inputs (download mode, single file share) | send-download.js |
| `transfer-url` | `#sv-share-url` | send-viewer.js |
| `transfer-url` | `#sb-share-url` | send-browse.js |
| `encryption-key` | `#decryption-key` | upload-step-done.js |
| `timing` | `.timings__title` ("Completed in 0.04s") | upload-step-done.js |
| `timing` | Decrypt timing ("Decrypted in 0.02s") | send-download.js |
| `timing-bars` | `.timings__rows` (per-stage breakdown) | upload-step-done.js |
| `timestamp` | Upload date (2 locations) | send-download.js |

### Impact assessment

- **User-visible changes:** None. `data-qa-mask` is not styled, not referenced by JS, and not rendered.
- **Runtime cost:** Zero. The attribute exists in the DOM but is never queried by application code.
- **QA team action:** The QA pipeline can now use `page.locator("[data-qa-mask]").all()` as the mask parameter in Playwright's `page.screenshot()`. No JS injection workaround needed.
- **Forward-compatible:** New dynamic elements just need the attribute added — no QA code changes required.

### What the QA team needs to do next

1. **Phase 1 (QA side):** Update `ScreenshotCapture.capture()` to query `[data-qa-mask]` and pass as mask to `page.screenshot(mask=[...], mask_color="#1a2332")`
2. **Phase 3:** Remove the temporary JS injection workaround (if deployed)
3. **Phase 4:** Validate — run CI twice with no code changes, verify 0% deterministic diff
