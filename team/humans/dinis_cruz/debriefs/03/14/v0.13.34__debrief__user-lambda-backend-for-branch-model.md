## Executive Summary

The SG/Send User Lambda backend now supports the three new primitives required by the branch model architecture (v0.13.30 arch spec): **batch write**, **list**, and **write-if-match** (compare-and-swap). Meta.json per-file metadata has been removed — the commit graph supersedes it entirely.

These changes will be deployed to **dev.send.sgraph.ai** first. Production deployment is a Villager-phase activity.

**545 tests pass. 0 failures. No breaking changes to existing endpoints.**

## What's New — API Reference for CLI Team

### 1. Batch Endpoint

```
POST /api/vault/batch/{vault_id}
Headers:
  x-sgraph-access-token: <access_token>
  x-sgraph-vault-write-key: <write_key>
Content-Type: application/json

Body:
{
  "operations": [
    { "op": "write",          "file_id": "bare/data/obj-{hash}", "data": "<base64>" },
    { "op": "write-if-match", "file_id": "bare/refs/ref-{id}",
      "match": "<old_value_base64_or_null>", "data": "<new_value_base64>" },
    { "op": "delete",         "file_id": "bare/data/obj-{hash}" }
  ]
}

Response (200):
{
  "vault_id": "...",
  "results": [
    { "op": "write",          "file_id": "...", "status": "ok" },
    { "op": "write-if-match", "file_id": "...", "status": "ok" },
    { "op": "write-if-match", "file_id": "...", "status": "conflict", "current": "<base64>" }
  ]
}
```

**Semantics:**
- Operations execute **in array order**, best-effort
- Stops on first **conflict** or **error** — subsequent operations are skipped
- Max **100 operations** per batch
- Max **5MB total payload** (Lambda request limit; ~3.75MB actual data after base64)
- Immutable objects (blobs, trees, commits) are idempotent — writing them twice is harmless
- `write-if-match` with `"match": null` succeeds only if the file does **not** exist (first push)

**Use for:** `push`, `clone` (branch registration), any multi-object write

### 2. List Endpoint

```
GET /api/vault/list/{vault_id}?prefix=bare/data/
No authentication required

Response (200):
{
  "vault_id": "...",
  "prefix": "bare/data/",
  "files": ["bare/data/obj-aaa", "bare/data/obj-bbb"]
}
```

**Semantics:**
- Returns all `file_id` values in the vault matching the prefix
- Prefix is a query parameter (supports slashes in `bare/` paths)
- Empty prefix returns all files
- Excludes `manifest.json` (internal)
- No pagination (fine for <1000 files; will add cursor/limit when needed)

**Use for:** GC drain (`prefix=bare/pending/`), branch discovery (`prefix=bare/branches/`), status checks

### 3. Write-If-Match (Compare-and-Swap)

Available as a batch operation (`"op": "write-if-match"`) and as a standalone service method.

**Semantics:**
- Reads current payload, compares raw bytes to `match` value
- If match → writes `data`, returns `{ "status": "ok" }`
- If mismatch → returns `{ "status": "conflict", "current": "<base64>" }`
- `match: null` matches when file does not exist

**Use for:** Ref updates during push (arch spec section 6.3, step 7)

### 4. Existing Endpoints (Unchanged)

```
PUT    /api/vault/write/{vault_id}/{file_id}         AUTH    Write blob
GET    /api/vault/read/{vault_id}/{file_id}          NO AUTH Read blob (binary)
GET    /api/vault/read-base64/{vault_id}/{file_id}   NO AUTH Read blob (base64 JSON)
DELETE /api/vault/delete/{vault_id}/{file_id}         AUTH    Delete blob
```

These continue to work for flat file_ids (no slashes). For `bare/` paths with slashes, use the batch endpoint for writes and the list endpoint for discovery.

## What Changed — Internal

### Meta.json Removed

Every vault file previously stored a `meta.json` sidecar with 7 fields. All were either redundant, dead code, or superseded by the commit graph:

| Field | Why removed |
|-------|------------|
| `file_id`, `vault_id` | Derivable from storage path |
| `mutable`, `status` | Dead — never read by any code path |
| `write_key_hash` | Duplicates vault-level `manifest.json` check |
| `created_at`, `updated_at`, `write_count` | Superseded by commit timestamps and parent chains |

**Impact:** Write operations now make 1 S3 PUT instead of 2 (no meta sidecar). The `write()` response no longer includes `write_count`.

### Manifest.json Simplified

- Now uses `Timestamp_Now` (uint milliseconds) instead of ISO datetime strings
- Remains the sole server-side auth record per vault

## CLI Integration Notes

### Push Workflow (arch spec section 6.3)

The CLI push maps to these API calls:

```
1. GET  /api/vault/read/{vault_id}/{ref_file_id}     ← fetch remote ref
   (or use list + read to discover refs)

2. POST /api/vault/batch/{vault_id}                   ← push delta
   operations: [
     { op: "write", file_id: "bare/data/obj-{blob_hash}",   data: ... },
     { op: "write", file_id: "bare/data/obj-{tree_hash}",   data: ... },
     { op: "write", file_id: "bare/data/obj-{commit_hash}", data: ... },
     { op: "write", file_id: "bare/indexes/idx-{id}",       data: ... },
     { op: "write-if-match",
       file_id: "bare/refs/ref-{id}",
       match: "<old_commit_ref_base64>",
       data:  "<new_commit_ref_base64>" }
   ]
```

**Typical push: 1 GET + 1 POST = 2 round trips.**

If `write-if-match` returns `conflict`, the CLI should re-fetch and re-merge (arch spec step 7 → retry from step 2).

### Slashed File IDs

File IDs like `bare/data/obj-abc123` contain slashes. These work through:
- **Batch endpoint** — file_ids in JSON body, no URL encoding needed
- **List endpoint** — prefix as query parameter
- **Individual read** — the `file_id` URL parameter captures only the first segment before a slash. For reading individual objects by their `bare/` path, use `read-base64` with URL encoding or fetch via a batch read pattern

**Recommendation:** Use batch for all `bare/` writes, individual endpoints for flat file_ids only.

### Race Condition on write-if-match

The CAS is not truly atomic on S3 (no `If-Match` support for updates). The race window is ~50ms (Lambda invocation duration). Two concurrent pushes to the same vault could both succeed if they happen within this window.

**Detection:** The next push will fetch the remote ref, find unexpected divergence, and trigger the merge workflow.

**Recovery guarantees:**
1. Refs always point to valid commit objects
2. Orphaned objects (from a failed push) are harmless — GC prunes them
3. GC and REBUILD INDEXES are idempotent — running them repairs any state
4. The commit graph is append-only — races cannot corrupt existing commits

## Decisions Captured

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Best-effort batch (no DynamoDB) | Immutable objects are idempotent; only ref CAS matters |
| 2 | Remove meta.json entirely | No value in branch model; commit graph provides richer audit |
| 3 | List unauthenticated | Consistent with read; file IDs are opaque; zero-knowledge holds |
| 4 | Accept write-if-match race | ~50ms window; push retries on conflict; GC/REBUILD repairs |
| 5 | Extend existing Routes__Vault__Pointer | One route class for all vault operations |
| 6 | Prefix as query parameter | Supports slashes in bare/ paths without URL path issues |
| 7 | Deploy to dev first | Explorer output; production is Villager territory |
