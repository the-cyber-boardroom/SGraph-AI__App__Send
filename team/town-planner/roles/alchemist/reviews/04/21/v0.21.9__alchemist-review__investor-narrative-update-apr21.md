## 3. The Auth MVP Redesign — Investor Story

### 3.1 The Problem with Cognito

Previous investor materials referenced AWS Cognito as the planned authentication provider. This is now superseded. The Cognito design had three weaknesses:

1. **Creates a server-side user record** — contradicts zero-knowledge positioning
2. **Requires email or phone for signup** — increases friction, reduces conversion
3. **Adds a dependency** — outage of Cognito = outage of login

### 3.2 The New Architecture

The vault key is the credential. The architecture:

```
Vault key (e.g. "apple-river-1234")
         │
         ▼
Browser PasswordCredential API
         │   (iCloud Keychain / Google Password Manager / 1Password)
         ▼
sg-vault-picker component
         │
         ▼
AES-256-GCM vault decryption (client-side)
         │
         ▼
User is "logged in" — no server ever learned the key
```

### 3.3 Why This Is Better for Investors

| Factor | Cognito design | Vault-key design |
|--------|---------------|------------------|
| Server-side user store | Required | None |
| Zero-knowledge guarantee | Partial (login is server-side) | Complete |
| GDPR/HIPAA data scope | User PII in Cognito | No PII anywhere |
| Attack surface | Login server + Cognito | Client-only |
| Third-party dependency | AWS Cognito | Browser API (universal) |
| Enterprise story | Cloud-hosted only | Works on-premise too |

**The investor narrative:** "We eliminated the weakest point in zero-knowledge systems — the authentication layer. Most 'zero-knowledge' products encrypt at rest but authenticate with a standard login server. SGraph Send now has no login server."
