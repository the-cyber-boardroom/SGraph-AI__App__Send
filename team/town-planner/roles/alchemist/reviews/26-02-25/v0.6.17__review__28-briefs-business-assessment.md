## Executive Summary

The 28 briefs delivered on 23-24 February 2026 represent the most significant strategic expansion in the project's history. They transform SGraph Send from an encrypted file transfer product into a **PKI-based trust infrastructure platform** with a credible path to becoming a category-defining company.

From an investment perspective, these briefs accomplish three things that were previously missing:

1. **A metrics framework that investors can model** -- DRU/MRU/ARU replaces revenue-first thinking with ecosystem growth metrics that are defensible and measurable
2. **A partner programme that functions as a distribution strategy** -- reference implementations for 8+ partner types create multiple paths to market

The investor story has shifted from "we built an encrypted file transfer tool" to "we are building the trust layer of the internet, starting with files." This is a materially larger opportunity and requires updated investor materials across the board.

## 1. Investment Narrative Impact

### Before These Briefs

The investment narrative was:

> "SGraph Send is a zero-knowledge encrypted file transfer platform. Files are encrypted in the browser. The server never sees plaintext. It is open source, deployable anywhere, and currently in MVP stage with 111+ tests passing."

This is a product story. It describes what the thing does. Investors hear: small market, niche product, unclear revenue model, open source risk.

### After These Briefs

The investment narrative becomes:

> "SGraph is building the trust infrastructure layer for the internet. We start with encrypted file transfer -- a working product today -- and extend to PKI-verified content provenance, data room portability, and a trust web that grows with every encrypted message in the ecosystem. Our metrics measure ecosystem growth, not just direct revenue. Every SaaS company that embeds our open source encryption library, every cloud provider that adopts our protocol, every email provider that integrates PKI verification -- all of these increase the value of our network. Revenue follows usage, not the other way around."

This is a platform story. It describes a market position. Investors hear: large addressable market, network effects, multiple revenue streams, defensible moat through trust data.

### What Changed

| Dimension | Before | After |
|-----------|--------|-------|
| **Market** | Encrypted file transfer (niche) | Trust infrastructure for the internet (platform) |
| **Revenue model** | Unclear / "we'll figure it out" | Usage-first with 5 revenue streams identified |
| **Competitive moat** | "Better than Tresorit/Wormhole" | Network effects + trust web + interoperability protocol |
| **Growth model** | Direct user acquisition | Direct + indirect (every compatible tool adds value) |
| **Metrics** | Users, files transferred | DRU/MRU/ARU -- ecosystem-wide encrypted messages |
| **TAM argument** | File transfer market ($X billion) | Enterprise security + SaaS compliance + content trust ($XXX billion) |
| **Open source risk** | "How do you compete with free?" | "Every fork grows the network. Competitors are allies." |

### Narrative Risk Assessment

The expanded narrative is more compelling but also more ambitious. Investors will ask:

- **"Can you execute?"** -- The team is one human with AI agents. The briefs are impressive but they describe a vision, not delivered product for the platform layer. The answer: "The MVP proves execution capability. The platform layer uses the same primitives (PKI, encryption, trust signals) at different scales."
- **"Is this too big?"** -- The scope has expanded dramatically. The answer: "Each layer builds on the last. File encryption is working today. Data rooms use the same encryption. Content trust uses the same PKI. The fractal architecture means each step is incremental."
- **"Why you, not AWS?"** -- Brief #18 (Hyperscaler Integration) addresses this directly and turns it into a strength: "We WANT AWS to build this. Every provider that adopts the protocol adds to our network."

## 2. DRU/MRU/ARU Assessment (Brief #27)

### The Metrics Model

Brief #27 introduces **Daily/Monthly/Annual Recurring Usage** as the North Star metrics, replacing the traditional SaaS MRR/ARR framework. This is the single most important brief from an investment perspective because it defines how the company measures success.

### Investor Implications

**Strengths of the DRU/MRU/ARU model:**

| Strength | Why Investors Care |
|----------|-------------------|
| **Ecosystem-wide measurement** | Captures indirect value (partner usage, protocol adoption) -- not just direct product usage |
| **Leading indicator** | Usage growth predicts revenue, not the other way around. Investors can model conversion rates from usage to revenue |
| **Network effect proof** | Rising DRU demonstrates network effects in action. Each new user/partner adds measurable value |
| **Anti-lock-in alignment** | The metric explicitly rewards users who leave the platform but keep encrypting. This resolves the "open source business model" tension |
| **Defensible at scale** | Even if a competitor replicates the code, the usage data, trust web, and scanning history are not replicable. The moat is in the data, not the code |

**Risks and investor objections:**

| Risk | Mitigation |
|------|------------|
| **"Usage is not revenue"** | Revenue streams are identified (PKI management, scanning chain, compliance reporting, consulting). DRU-to-revenue conversion rate becomes the key metric to track |
| **"Indirect usage is unverifiable"** | The brief acknowledges this: indirect usage is estimated from public data (GitHub key growth, AGE downloads). This is a weakness until partner reporting is live |
| **"What is a unit?"** | "Encrypted message" is clear but needs definition: does a 1KB file and a 1GB file count the same? The brief does not address volume-weighted metrics |
| **"How do you benchmark?"** | There is no industry benchmark for DRU. Investors cannot compare to competitors. This is both a strength (first mover) and a weakness (unfamiliar) |

### Fundraising Impact

The DRU/MRU/ARU model changes the fundraising conversation from "what is your MRR?" to "what is your ecosystem growth rate?" This is the conversation Slack, WhatsApp, and Signal had in their early stages -- usage first, monetisation later. The precedents exist and are well-understood by growth-stage investors.

**Recommendation:** Create a DRU dashboard mockup for investor presentations. Show the breakdown (direct vs. CLI vs. library vs. partner-reported) and the estimated indirect usage. Even with placeholder numbers, the framework itself demonstrates strategic thinking.

### Materials Needed

- DRU/MRU/ARU explainer one-pager (for investors who ask "what is your MRR?")
- Conversion model: DRU to revenue projections at 3 scenarios (base, growth, breakout)
- Comparison document: DRU vs. MRR at equivalent companies (Slack DAU, Signal message volume, Let's Encrypt certificates issued)

## 5. Market Positioning

### Where SGraph Send Sits After These Briefs

The competitive landscape has been reframed. Before these briefs, SGraph Send competed with:

- **Tresorit Send** -- encrypted file sharing
- **Wormhole** -- peer-to-peer encrypted transfer
- **OnionShare** -- Tor-based file sharing
- **Firefox Send** (defunct) -- Mozilla's encrypted file transfer

After these briefs, SGraph Send is positioned against a different set of competitors at the platform layer:

| Competitor Category | Competitors | SGraph Differentiation |
|--------------------|-------------|----------------------|
| **Encrypted file transfer** | Tresorit, Wormhole, OnionShare | Zero lock-in. Open source. Deployable anywhere. Data rooms. |
| **Data room platforms** | Intralinks, Datasite, Firmex | Zero-knowledge. Portable (.sgroom). Not SaaS-locked. |
| **PKI infrastructure** | Let's Encrypt (TLS), Keybase (defunct) | Files + content + trust web. Not just certificate issuance. |
| **Content provenance** | C2PA (Adobe/Microsoft), Truepic | PKI-signed paragraphs, not just images. Legal enforcement via CC-Signed. |
| **SaaS file encryption** | Virtru, PreVeil, Boxcryptor | Drop-in JS library. Not a separate product. Embeds into existing SaaS. |
| **Enterprise compliance** | Varonis, Vera, Ionic | Zero-knowledge means no data to breach. GDPR advantage is structural. |

### The Unique Position

No competitor combines:
1. Zero-knowledge encryption (server never sees plaintext)
2. Zero lock-in (works with GPG, AGE, OpenSSL, any compatible tool)
3. Open source (no vendor lock-in objections in procurement)
4. Trust web (reputation builds over time, exportable)
5. Platform primitives (encrypted transport + data vault + trust workflows)
6. Content trust vision (facts, not just files)

This combination creates a category: **Trust Infrastructure.** The nearest precedent is Let's Encrypt -- which commoditised TLS certificates and made HTTPS universal. SGraph is doing for file/content encryption what Let's Encrypt did for transport encryption.

### The Let's Encrypt Comparison

| Let's Encrypt | SGraph |
|--------------|--------|
| Free TLS certificates | Free encryption library |
| Made HTTPS universal | Making file encryption universal |
| Funded by ISRG (non-profit) + sponsors | Revenue from trust services, scanning, compliance |
| 300M+ certificates issued | DRU target: ecosystem-wide encrypted messages |
| Competitors (DigiCert, etc.) coexist | Competitors who adopt the protocol grow the ecosystem |

This comparison is powerful for investor presentations. Let's Encrypt proved that commoditising a security primitive creates massive network value. SGraph applies the same model to file/content encryption.

## 8. Revenue Model Evolution

### Before These Briefs

The revenue model was implicit:

- Open source product
- Potential enterprise features (vaguely defined)
- Possible SaaS offering (unspecified)
- "Revenue will follow" (hand-wave)

This is the answer that kills investor conversations.

### After These Briefs

The revenue model is explicit, multi-stream, and tied to ecosystem growth:

| Revenue Stream | Source Brief | Model | Scaling Driver |
|---------------|-------------|-------|----------------|
| **PKI management service** | #15, #27 | Free tier + paid enterprise (key management, rotation, compliance) | DRU growth -- more users = more keys to manage |
| **Scanning chain integration** | #21 | Per-file fees for CDR, malware scanning, trust verification | File volume -- more transfers = more scans |
| **Compliance reporting** | #15 | Enterprise subscription for audit trails, GDPR dashboards, key rotation logs | Regulatory pressure -- more regulation = more demand |
| **Consulting / integration services** | #15, #21 | Per-engagement fees for SaaS companies integrating the encryption library | Partner adoption -- more partners = more integration work |
| **Data room premium features** | #13 | Enterprise features: fractal signing, version pinning, multi-party workflows | Enterprise adoption -- more data rooms = more premium features |
| **Trust web services** | #12, #27 | Verification, reputation scoring, trust certification | Network size -- more nodes = more valuable trust data |
| **Partner revenue share** | #21 | Revenue share from scanning providers, identity providers, cloud storage | Partner ecosystem -- more partners = more revenue channels |

### The Credit Card Model (Brief #15)

This is the single most investable revenue stream because it is:

1. **Analogous** -- investors understand the credit card tokenisation model. "We do for files what Stripe did for payments" is immediately comprehensible.
2. **Drop-in** -- a JavaScript library that any SaaS company can embed. Low friction, high adoption potential.
3. **Network-effect-driven** -- every SaaS company that adopts the library creates more PKI keys, more trust relationships, more nodes.
4. **GDPR-compliant by design** -- the SaaS company literally cannot access the data. This eliminates an entire category of compliance cost.
5. **Free at the base layer** -- the library is open source. Revenue comes from the services on top (PKI management, scanning, compliance).

### Revenue Timing

```
Phase 1 (Now - 6 months):        Direct product revenue
  - Consulting / integration      (from early partners)
  - Demo environments             (as sales tools, not revenue)

Phase 2 (6 - 18 months):         Platform revenue
  - PKI management service        (enterprise subscriptions)
  - Scanning chain fees           (per-file, via partners)
  - Compliance reporting          (enterprise subscriptions)

Phase 3 (18 - 36 months):        Ecosystem revenue
  - Trust web services            (verification, reputation)
  - Partner revenue share         (scanning, identity, storage)
  - Content trust services        (provenance, fact-checking)
```

## Cross-Brief Synthesis: The Investment Thesis

These 10 briefs, taken together, produce a coherent investment thesis that can be stated in one paragraph:

> SGraph is building the trust infrastructure layer for the internet. We start with a working product -- zero-knowledge encrypted file transfer -- and extend to a platform that any SaaS company can embed (the credit card model for files), any cloud provider can adopt (commoditising the storage layer), and any email/messaging provider can integrate (extending the trust web to billions of users). We measure ecosystem growth (DRU/MRU/ARU), not just direct revenue. Our competitive moat is not the code (which is open source) but the trust web -- the accumulating data about who is trustworthy, which scanning services are reliable, and which content is verified. Revenue follows usage through 7 identified streams. The more the ecosystem grows -- including through competitors who adopt our protocol -- the more valuable our trust data and services become. We are doing for file and content encryption what Let's Encrypt did for transport encryption: making it universal, free at the base layer, and building the premium services on top.

This thesis is fundable. It has:
- A working product (MVP complete, 111+ tests)
- A clear market (enterprise security + SaaS compliance + content trust)
- A metrics framework (DRU/MRU/ARU)
- A revenue model (7 streams, credit card analogy)
- A competitive moat (trust web, network effects)
- A distribution strategy (partners, open source, hyperscaler adoption)
- A vision (trust layer of the internet)
- A team efficiency narrative (one human + AI agents = full team output)
