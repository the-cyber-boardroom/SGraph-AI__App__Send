### P-001: Lambda Performance -- CDN Migration as Operational Efficiency Story

**ID:** P-001
**Title:** CDN Migration for send.sgraph.ai Static Assets (Operational Efficiency Milestone)
**Classification:** P (Product/Platform -- with direct Alchemist interest)
**Priority:** P1 -- High. The performance baseline data makes this a time-sensitive opportunity.

**Description:**

The performance baseline from 28 Feb 2026 is documented: cold start 18.5 seconds, warm 599ms, all 18 requests hitting Lambda including static assets. The projected improvement after CDN migration: cold start drops to ~2-3 seconds, warm drops to ~300ms.

This work stream is the CDN migration of static assets for `send.sgraph.ai`: moving HTML, CSS, JS, and fonts to S3/CloudFront using the same pipeline already built for the `sgraph_ai__website/`. The Python scripts (`deploy_static_site.py`) exist. The CloudFront distribution infrastructure exists. The work is applying the already-built tooling to the product UI.

**Alchemist Rationale:**

The Alchemist cares about this for one reason: a product that loads in 18.5 seconds from a cold start cannot be demonstrated credibly to investors. The investor demo starts with opening the product. If the first impression is an 18-second spinner, the demo has already failed before a word is spoken.

The performance baseline is also a narrative asset. The team measured the problem before fixing it. They have projected the improvement. After the CDN migration, they will remeasure and confirm the improvement. This is operational rigour -- "we define the problem, we fix it, we verify the fix." Investors who ask about technical discipline get a concrete answer.

**Cost impact:** Static assets on CloudFront serve for fractions of a cent per GB. Moving static files off Lambda reduces Lambda invocation costs proportionally. At current traffic levels, this is negligible in absolute terms; at scale, it becomes meaningful gross margin improvement.

**Dependencies:**
- `deploy_static_site.py` (EXISTS)
- `deploy-website.yml` pattern (EXISTS, for website; needs adaptation for send.sgraph.ai UI)
- Google Fonts violation fix in User UI (9 files still violated -- this must be fixed as part of or before CDN migration, or the violation moves to CDN)

**Effort:** Medium (adapting the existing pipeline; fixing 9 Google Fonts violations in UI)

### B-008: Open Source + Commercial Dual Model -- Community and Revenue Framing

**ID:** B-008
**Title:** Formalise the Open Source + Commercial Dual Model for Investor Framing
**Classification:** B (Business -- Revenue Model / Investor Positioning)
**Priority:** P2 -- Medium. Addresses the most common investor objection to open source companies.

**Description:**

The 8-option pricing matrix (B-004) implicitly contains an open source + commercial dual model. This work stream makes it explicit: a one-page document for investors that answers "how do you compete with free?" for an open source product.

The document covers: (1) what is open source (the encryption library, the self-hosted deployment, the CLI), (2) what is commercial (the managed cloud service, the dedicated tenant, the AWS Marketplace listing, the managed deployment), (3) why open source is a distribution advantage rather than a competitive threat (every self-hosted deployment proves the architecture; every fork extends the ecosystem; every CLI user is a potential enterprise customer), and (4) the precedents (HashiCorp, Elastic, MongoDB, Grafana -- all built on this model).

**Investor Rationale:**

"Open source" causes a reflexive concern in investors who have seen open source projects fail to monetise. The concern is legitimate: "If anyone can deploy it for free, who pays?" The answer is well-documented in the industry but must be stated explicitly for each product.

For SGraph Send, the answer is: "The code is free. The operations are not. The more you trust SGraph to operate it, the more you pay. The more you want SGraph out of your infrastructure, the more you use the open source version. Both outcomes grow the ecosystem." This is the HashiCorp model, the Elastic model, the Grafana model. It is a proven investor-grade framing for open source + commercial software.

**Dependencies:**
