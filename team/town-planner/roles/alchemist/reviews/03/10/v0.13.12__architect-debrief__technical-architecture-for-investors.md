## 1. Zero-Knowledge Encryption: The Load-Bearing Principle

### What It Means (In Non-Technical Terms for Investors)

The server literally cannot read your files. Not because the engineers promised to be ethical. Not because it's company policy. But because of how the encryption works.

**The critical difference from competitors:**
- Tresorit, Sync.com, Virtru all store encryption keys on their servers (or key management services they control)
- If a government agency subpoenas Tresorit, Tresorit decrypts the files and hands them over
- SG/Send's server has no keys, no ability to decrypt, and cannot be compelled to do what it physically cannot do

### The Technical Architecture

**Sender Side (Browser):**
1. User selects a file
2. Browser generates a random 256-bit AES key (32 bytes of cryptographically secure random)
3. File is encrypted in the browser using AES-256-GCM (authenticated encryption, tamper-evident)
4. Encrypted blob is uploaded to the server
5. Decryption key is placed in the URL fragment (`#key_in_url`), which browsers never send to servers

**Server Side:**
- Stores encrypted blob (opaque, unreadable)
- Records metadata: transfer ID (UUID), file size, status, download count
- **Never knows:** encryption key, filename, file type, content

**Recipient Side (Browser):**
1. Recipient receives URL with key in fragment
2. Browser fetches encrypted blob using transfer ID (key stays local)
3. Browser decrypts locally using the key from the fragment
4. After successful decryption, `history.replaceState()` removes the key from browser history

### Why This Is a Moat Against Regulation and Competition

1. **Regulatory Defensibility** — The UK Online Safety Bill, EU regulations, and US CLOUD Act all contemplate "service providers who can decrypt." SG/Send cannot. This is not a business decision; it's a cryptographic constraint.

2. **No Single Point of Compromise** — Breaching SG/Send's server yields encrypted blobs without keys. Breaching a browser yields one user's key, affecting only their own files.

3. **Liability Separation** — If a file in SG/Send carries malware, SG/Send is the courier, not the inspector. This aligns with the legal principle that postal services are not liable for package contents.

## 3. Vault: Distributed Encrypted Object Store (Git-Like)

### The Evolution Since March 1

**Milestone M-010 (March 4, 2026):** First CLI vault clone.

### The Architecture: Content-Addressable, Encrypted Objects

Modeled after Git internals:

| Object Type | What It Stores |
|---|---|
| **Blob** | Encrypted file content (SHA256 reference) |
| **Tree** | Directory listing with pointers to blobs/trees (all encrypted) |
| **Commit** | Snapshot of tree + parent commit + author + timestamp (immutable history) |
| **Tag** | Human-readable reference to a commit |

### Why This Matters for Investors

1. **Offline-First Capability** — Once cloned, the vault is a complete encrypted archive on disk. No internet needed.
2. **Audit Trail Built-In** — Every change is a commit with who, when, what, and rollback capability.
3. **Multi-Client Synchronization** — Browser, CLI, Workspace, Desktop, Mobile all operate on the same data.
4. **Enterprise Data Room Foundation** — Shared vaults with per-user encryption keys, immutable access logs.

### Current Status (Code-Verified)

- **Browser UI:** Vault v0.1.1, editable, inline editing, drag-and-drop, settings panel
- **CLI:** `clone`, `status`, `pull` working; `push` and `sync` in progress
- **Encryption Interop:** Web Crypto API ↔ Python cryptography library (cross-language testing)
- **Workspace Integration:** Vault files loadable into LLM workspace

## 4. Secure Pod: Third-Party Scanning Intermediary

### The Problem This Solves (The Scanning Paradox)

- If the server scans files, it must see plaintext → violates zero-knowledge
- If the sender self-certifies, a malicious sender can fake results
- If a third party scans, they see plaintext → how to prevent exfiltration?

### SG/Send's Solution

1. Sender uploads encrypted blob + session key encrypted for pod's public key
2. Pod decrypts, scans (ClamAV, Glasswall, etc.), constructs signed attestation
3. Pod re-encrypts for recipient with attestation included
4. Recipient verifies pod's signature and hash chain — cryptographic proof of scan

**The server never sees plaintext at any step.**

### Commercial Model (Receiver-Defined, Sender-Pays)

- Recipient mandates scanning requirements
- Basic tier (ClamAV): included free
- Premium tiers (CDR, DLP): charged per scan, profit shared with scanning partner
- Network effect: every enterprise recipient generates revenue for every file sent to them

**PROPOSED — pod infrastructure does not exist yet. Architecture is designed and documented.**

### Market Positioning

- **Healthcare (HIPAA):** File analysis without exposure to plaintext PHI
- **Finance (PCI-DSS):** Scanning without exposing financial documents
- **Government:** Cross-domain transfer without plaintext at the crossing point
- **Legal:** Recipient-mandated scanning with cryptographic attestation

## 7. Security Properties That Survive Breach

**If SG/Send's server is breached:**
- Attackers gain transfer metadata (IDs, timestamps, sizes)
- Attackers **do not gain** encryption keys, filenames, or content

**If a pod is compromised:**
- Attackers see plaintext during scanning (intended role)
- Attackers **cannot forge attestations** (hardware-protected signing)

**If a client is compromised:**
- Attackers see that user's files only
- Other users' files remain encrypted with separate keys

**Compartmentalized blast radius** — rare and valuable. Competitor breaches often expose all customer data.
