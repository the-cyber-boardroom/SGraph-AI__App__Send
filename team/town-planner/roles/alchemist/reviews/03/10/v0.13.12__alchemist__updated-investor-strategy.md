## Five Investor Narratives (Updated)

### Narrative 1: The Zero-Knowledge Moat

**The claim:** Our server literally cannot read your files. Not because we promised. Because of how the cryptography works.

**The evidence (code-verified):**
- AES-256-GCM encryption happens in the browser before upload
- Decryption key stays in URL fragment (browsers never send fragments to servers)
- Server stores opaque encrypted blobs — no keys, no filenames, no content
- Independent security review of production CloudFront logs confirmed: even published logs reveal nothing
- IP addresses hashed with daily salt (SHA-256) — no user tracking possible

**The competitive advantage:**
- If we're subpoenaed, we physically cannot decrypt (regulatory defensibility)
- If our server is breached, attackers get encrypted blobs without keys (compartmentalized blast radius)
- Competitors (Tresorit, Sync.com) store keys on their servers — they CAN be compelled to decrypt

**Investor one-liner:** "We can publish our server logs and your data is still safe. Here's the review that proves it."

---

### Narrative 2: Multi-Platform from One Codebase

**The claim:** Web, desktop, and mobile from one codebase with encryption parity across all platforms.

**The evidence (code-verified):**
- Tauri desktop app: 10-15MB (vs Electron's 150MB+), Apple-signed, Gatekeeper approved
- Android APK: built for 3 architectures, tested on real Pixel 6, end-to-end encrypted transfer validated
- Same Web Components UI runs everywhere — no separate mobile team, no separate codebases
- CLI vault operations: clone, status, pull — developers and automation can access encrypted vaults

**Why competitors can't replicate easily:**
- WeTransfer: web-only
- Dropbox: separate mobile apps, different security models per platform
- Signal: separate codebases per platform
- Tresorit: Electron-based desktop (150MB+), separate mobile apps

**Investor one-liner:** "A file encrypted on Android decrypts identically on macOS desktop. Same codebase, same security, every platform."

---

### Narrative 3: AI-Native Document Workspace (Zero-Knowledge Preserved)

**The claim:** LLM-powered document transformation that maintains zero-knowledge architecture.

**The evidence (code-verified):**
- Workspace UI (v0.1.1): five-zone layout, vault file navigator, LLM chat panel
- OpenRouter (cloud) + Ollama (local) LLM providers both working
- Documents load from encrypted vault → transform via LLM in browser → save back encrypted
- Server never sees document content OR LLM conversation
- Cross-browser validated: edit in Safari → transform in Chrome → save to vault → clone via CLI

**Why this matters:**
- Every competitor that adds "AI features" routes documents through their server AND the LLM provider
- SG/Send's architecture keeps everything client-side — two fewer parties see your data
- Users can run Ollama locally: zero API costs, zero data leaves the machine

**Investor one-liner:** "Your documents never leave your browser. Neither does the AI processing."

**The market opportunity (PROPOSED — use cases briefed, not yet shipped):**

| Vertical | Problem | SG/Send Solution | TAM |
|----------|---------|-------------------|-----|
| Healthcare | HIPAA/GDPR: can't share patient data through intermediaries who see plaintext | Form-to-vault encryption, revocable access (consent withdrawal = key revocation) | £500M+ |
| Financial Services | M&A data rooms cost £3-10K/month (Intralinks, Datasite) | Zero-knowledge data rooms at fraction of the cost | £1B+ |
| Legal | Attorney-client privilege requires evidence never in plaintext at intermediary | Client uploads encrypted, attorney gets key, privilege boundary maintained | £300M+ |

- Multi-deployment: Lambda, Docker, EC2, CLI — deploy in UK-only, EU-only, on-premise, air-gapped

**Investor one-liner:** "Every regulated industry needs to share sensitive data across organisations without trusting intermediaries. We're the only platform that solves this with zero-knowledge across web, desktop, mobile, and CLI."
