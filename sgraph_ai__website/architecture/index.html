How SG/Send is Built

A zero-knowledge encrypted file sharing platform built on serverless infrastructure, developed by 18 AI agents working in parallel under human oversight.

Technology Stack

Every layer is chosen for simplicity, security, and zero-dependency deployment.

Layer Technology Purpose
Runtime Python 3.12 / arm64 Application server language
Web Framework FastAPI via osbot-fast-api HTTP routing and request handling
Compute AWS Lambda + Mangum Serverless execution, pay-per-use
Storage Memory-FS (Storage_FS) Pluggable backend: memory, disk, or S3
Encryption Web Crypto API (AES-256-GCM) Client-side encryption in the browser
Type System Type_Safe (osbot-utils) Schema definitions without Pydantic
Frontend Vanilla JS + Web Components Zero framework dependencies (IFD)
Testing pytest, in-memory stack No mocks, no patches, real implementations
CI/CD GitHub Actions Test, tag, deploy pipeline

Security Model

Zero knowledge is not a feature — it is the architecture.

Client-Side Encryption

All encryption and decryption happens in the browser using the Web Crypto API. AES-256-GCM with randomly generated keys. The server never has access to plaintext.

No Plaintext on Server

The backend stores only encrypted ciphertext. Even with full database access, file contents cannot be read without the decryption key.

No File Names Stored

Original file names are never sent to the server. The server assigns random transfer IDs. File metadata stays with the sender.

Hashed IP Addresses

IP addresses are SHA-256 hashed with a daily rotating salt before storage. Stored as ip_hash for rate limiting, never as raw IPs.

All backend data is non-sensitive and non-confidential by design. A complete server compromise reveals nothing about the content of transferred files.

Deployment Targets

One codebase, seven deployment targets grouped into four patterns.

Lambda

Primary deployment. Two Lambda functions (User + Admin) behind Lambda Function URLs. Direct HTTPS endpoints, no API Gateway needed.

Production

Container

Docker, AWS Fargate, and GCP Cloud Run. The same application packaged as a container for environments that prefer containers over serverless.

Docker / Fargate / GCP

Server

EC2 instances and AMI builds. For teams that need full control of the runtime environment or have compliance requirements for dedicated infrastructure.

EC2 / AMI

CLI

Command-line interface for scripted transfers, CI/CD pipelines, and power users who prefer terminal-based workflows.

Terminal

Agentic Development

18 AI agents collaborate under human oversight using a role-based workflow.

Role-Based Teams

Three teams — Explorer (build new), Villager (stabilise), and Town Planner (strategy). Each agent has a defined role: Architect, Dev, QA, AppSec, Librarian, and more.

IFD Methodology

Incremental Feature Development. Small, complete, testable increments. No framework dependencies. Web Components with surgical versioning.

Everything in Public

All code, reviews, decisions, and briefs live in the open repo. Every agent produces versioned documents tracked by the Librarian.
