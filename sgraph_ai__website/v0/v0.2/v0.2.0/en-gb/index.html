What people build with SG/Send

From security research to healthcare workflows to multi-agent pipelines — client-encrypted vaults are the common thread.

Security Research

Share vulnerability research without exposure

Security teams create encrypted vaults containing OWASP findings, CVE analysis, and remediation guides. Each vault is a versioned, client-encrypted package. The recipient gets a share token — no login, no account, no plaintext on any server.

  • Vault encrypted before it leaves the researcher's browser
  • Share token gives read-only access to a point-in-time snapshot
  • Full version history — every revision auditable
Multi-Agent Workflows

Agents collaborate via encrypted vaults

Two agents, one vault. Agent A researches and writes findings to the vault. Agent B reads, analyses, and writes back a report — all without sharing credentials or touching a shared file system. The vault is the communication channel.

  • Each agent gets a vault key scoped to one vault only
  • Push/pull model handles concurrent writes cleanly
  • Full commit history — every agent action auditable
Healthcare

Patient intake that never reaches the server

Encrypted questionnaires collected in the browser, assembled into a vault, and shared with clinicians via token. Deploy your own SG/Send AMI inside your VPC — data never leaves your infrastructure.

  • AES-256-GCM in the browser — server sees ciphertext only

  • No plaintext patient data on any third-party server
Browser Automation

The Playwright AMI runs browser automation sequences and writes all output — screenshots, PDFs, network dumps — directly to a vault. The MITM proxy captures 100% of traffic. Every session is reproducible and auditable.

Our privacy model in six sentences

How SG/Send handles your data

Files are encrypted in your browser using AES-256-GCM before they are uploaded. The encryption key is derived from a passphrase you choose and never transmitted to our servers.

Our server stores only ciphertext — we cannot read your files, and we cannot give them to anyone else because we do not have the keys.

IP addresses are hashed with a daily salt before storage; we cannot reverse the hash to identify you. No file names are stored on the server.

Transfers have a configurable TTL; after expiry, access to the ciphertext is revoked. We collect no analytics, no tracking pixels, and no third-party scripts run on this page.

The server-side code is open source. You can audit exactly what runs on our infrastructure, or deploy your own instance on your own AWS account using our published AMI.

If you have questions about data handling, email privacy@sgraph.ai.
