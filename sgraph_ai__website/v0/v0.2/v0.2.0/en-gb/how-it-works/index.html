Two primitives. One encryption model.

File transfers for one-shot sharing. Vaults for persistent, version-controlled collaboration. Both encrypted in your browser before anything leaves your device.

The core flow

🔐

Encrypt

Your file is encrypted with AES-256-GCM using a key you choose — in your browser, before upload. The server never sees plaintext.

→
☁️

Store

The ciphertext is stored on the server (S3-backed). You receive a transfer ID. No file name, no plaintext, no key — just an encrypted blob.

→
🔓

Decrypt

The recipient pastes the token at send.sgraph.ai, enters the passphrase, and the file decrypts locally. The server is not involved in decryption.

The passphrase (and the key derived from it) travels out-of-band — via chat, email, or voice. The server only ever sees encrypted bytes.

File transfers vs Vaults

Choose the right primitive for your workflow.

Property 📁 File Transfer 🔐 Vault
Best for One-shot: send a file to someone right now Ongoing: a shared encrypted workspace that persists
Versioning None — single snapshot Full commit history (like Git) — every change auditable
Access Share token → one-time download link Vault key (read/write) or share token (read-only snapshot)
Expiry Automatic TTL — deleted after expiry Persistent until deleted by vault owner
Multiple files One file per transfer Unlimited files, folders, branches
Agent use Agents can send/receive via HTTP API Agents use SGit CLI: push, pull, branch, collaborate
Web UI send.sgraph.ai → upload → share token dev.vault.sgraph.ai → browse vault in browser
CLI sgit send <file> sgit init / commit / push / pull

Sending a file

Three ways: web UI, CLI, or HTTP API.

🌐

Web UI

Go to send.sgraph.ai, choose a file, set a passphrase, upload. You receive a transfer ID. Share the ID and passphrase with your recipient separately.

⌨️

SGit CLI

# Install once
pip install sgit-ai

# Send a file
sgit send report.pdf

# → Transfer ID: coral-stamp-5678
# Share ID + passphrase out-of-band
🤖

HTTP API

Agents POST encrypted bytes to /api/transfers/upload and receive a transfer ID. Full API docs in the source repo. No auth required for public transfers.

Working with vaults

A vault is a version-controlled encrypted folder. Think Git, but every object is encrypted before it leaves your machine.

Create and use a vault

SGit CLI
# Create a new vault
sgit init my-project
# → Vault key: mypassphrase:vault-abc123
#   Save this — it's your access key

# Add files, then commit
cd my-project
echo "hello" > notes.md
sgit commit "first notes"
sgit push

# Share a read-only snapshot
sgit share
# → Token: river-cloud-3847
#   Recipient browses at send.sgraph.ai/#river-cloud-3847

Clone and collaborate

On another machine or in another session
# Clone with vault key
sgit clone mypassphrase:vault-abc123

# Pull latest changes
sgit pull

# Check status
sgit status

# Work on a separate branch
sgit branch new analysis
sgit switch analysis
sgit commit "analysis results"
sgit push

Using vaults with AI agents

Vaults are the natural communication channel between agents. Each agent gets a scoped vault key — not your credentials.

Single agent workflow

Pattern: brief → work → share
Human
  │ Create vault with briefing + context
  │ sgit commit "session brief"
  │ sgit push
  ↓
Agent Session
  │ sgit clone mypassphrase:vault-id
  │ Read brief, do work, write outputs
  │ sgit commit "session results"
  │ sgit push
  ↓
Human
  │ sgit pull
  └ Review outputs in vault

Multi-agent collaboration

Pattern: parallel branches → merge
Shared Vault (one vault key)
  │
  ├── Agent A (branch: research)
  │     Browses web, writes findings
  │     sgit push
  │
  ├── Agent B (branch: analysis)
  │     Reads Agent A's output
  │     Writes analysis report
  │     sgit push
  │
  └── Human (branch: main)
        sgit pull (from both branches)
        Review, merge, share token
Key principle: each agent gets a vault key scoped to one vault only. No shared credentials. No access to your Git account. Full audit trail of every commit.
