The server cannot read your files.

Not "will not." Cannot. The encryption key never reaches the server. Here is exactly how it works.

Encryption in four steps

Every step happens in your browser. The server participates only at step 3 — and only sees ciphertext.

1
You choose a file
File stays in browser memory. Nothing is uploaded yet.
2
Browser encrypts it
AES-256-GCM via Web Crypto API. Key derived from your passphrase — never transmitted.
3
Ciphertext uploaded
Only the encrypted blob reaches the server. No plaintext, no file name, no key.
4
Recipient decrypts
The recipient enters the passphrase in their browser. Decryption happens locally. Server not involved.

Security properties

Six properties enforced at the architecture level, not by policy.

🔑
Encryption key never leaves the browser

The AES-256-GCM key is derived from your passphrase using PBKDF2 inside the Web Crypto API. It exists only in browser memory during the operation. It is never serialised, never logged, never sent to the server.

🙈
Server stores only ciphertext

What reaches the server is an opaque blob of encrypted bytes. No file name. No content type. No metadata that reveals what was shared. The server cannot distinguish a PDF from a spreadsheet.

🔒
AES-256-GCM with authentication

GCM (Galois/Counter Mode) provides both confidentiality and integrity authentication. Tampering with the ciphertext causes decryption to fail with an explicit error — the recipient cannot be given corrupted or forged content.

🕵️
IP addresses hashed, not stored

IP addresses are hashed with SHA-256 and a daily rotating salt before any storage. The plaintext IP is discarded. We cannot reverse the hash to identify a sender, and the hash rotates daily so it cannot be used for long-term tracking.

⏱
Automatic expiry

Every transfer has a configurable TTL. When it expires, access is revoked — the token no longer works and the ciphertext cannot be retrieved. There is no manual retrieval after expiry.

📖
Open source — verify what runs

The complete server-side code is public on GitHub. You can read every endpoint, every storage operation, and every place the server touches data. You can also deploy your own instance and not depend on ours at all.

System architecture

Two Lambda functions, S3 storage, CloudFront CDN. All data in transit encrypted by TLS. All data at rest is already ciphertext.

Architecture
┌─────────────────────────────────────────────────────────────────┐
│  SENDER'S BROWSER                                               │
│                                                                 │
│  File → AES-256-GCM encrypt (Web Crypto API) → Ciphertext blob │
│                         ↓                                       │
│                Key (from passphrase) — stays here              │
└──────────────────────────┬──────────────────────────────────────┘
                           │ HTTPS (TLS 1.3)
                           │ Ciphertext only — no key
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│  AWS INFRASTRUCTURE                                             │
│                                                                 │
│  CloudFront CDN → Lambda (User function) → S3 (ciphertext)     │
│                                                                 │
│  Lambda (Admin function) → Transfer tokens, stats              │
│                                                                 │
│  Server sees: encrypted blob, transfer ID, IP hash, TTL        │
│  Server cannot see: plaintext, file name, passphrase, key      │
└──────────────────────────┬──────────────────────────────────────┘
                           │ Share token (out-of-band)
                           │ e.g. email, chat
                           ▼
┌─────────────────────────────────────────────────────────────────┐
│  RECIPIENT'S BROWSER                                            │
│                                                                 │
│  Token → fetch ciphertext → enter passphrase → derive key      │
│       → AES-256-GCM decrypt (Web Crypto API) → Original file   │
└─────────────────────────────────────────────────────────────────┘
