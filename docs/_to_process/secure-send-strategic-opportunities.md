## 3. Proposal 1: Developer Platform â€” CLI Tool + SDK

### 3.1 Vision

Secure Send's zero-knowledge encrypted storage is a **primitive** â€” a building block that other applications, scripts, and workflows need. Right now it's locked behind a web UI. Exposing it as a CLI tool and SDK transforms Secure Send from "a file sharing app" into "zero-knowledge file transfer infrastructure." This is the Stripe analogy: Stripe isn't a payment page â€” it's payment infrastructure. Secure Send shouldn't just be a web page â€” it should be an encrypted transfer layer.

- Works against any Secure Send instance (SaaS or self-hosted)

## 5. Proposal 3: One-Time Secrets Sub-Product

### 5.1 Vision

A focused, standalone micro-product for sharing passwords, API keys, and sensitive text. One link, one view, then permanently deleted. This already exists in SND-7 (text sharing mode) and the retention system (1 download limit). The proposal is to give it a **dedicated landing page, dedicated URL, and dedicated marketing** as a top-of-funnel acquisition channel.

### 5.2 Why This Is High Leverage

- **Viral by nature** â€” every secret shared exposes the receiver to the brand
- **Trivial to build** â€” it's literally the existing file transfer flow with text input, 1-download limit, and 1-day retention pre-selected
- **Massive addressable market** â€” every developer who's ever pasted a password in Slack, every HR person who's emailed credentials, every sysadmin who's shared an API key in plain text
- **Conversion funnel** â€” user discovers one-time secrets â†’ realises they need file sharing too â†’ becomes a Secure Send user
- **SEO goldmine** â€” "share password securely", "one time secret link", "send API key safely" are all high-intent, low-competition search terms

## 15. Proposal 13: Time-Locked Transfers

### 15.1 Vision

A file that can't be decrypted until a specific date/time. The decryption key is either split using a time-lock mechanism or held by a trusted timeserver that releases it at the designated moment.

Use cases: embargoed press releases, sealed bids, scheduled announcements, dead man's switch, will/estate documents.

### 15.2 Implementation Options

| Approach | Complexity | Trust Model |
|----------|-----------|-------------|
| **Server-held key release** | Low | Trust the server to release the key at the right time (weaker) |
| **Threshold secret sharing** | Medium | Split key across multiple parties; quorum releases at time T |
| **Verifiable delay function** | High | Cryptographic puzzle that takes exactly N seconds to solve |
| **Third-party timeserver** | Medium | External service (e.g. timelock.dev) holds and releases |

**Recommendation:** Start with server-held key release (simple, MVP), then explore threshold sharing for higher-trust use cases.
